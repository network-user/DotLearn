# Не паттерном единым: как Python вызывает движок

За шесть концептов вы научились писать сами паттерны: классы символов, квантификаторы, группы, якоря, lookaround, флаги. Но до сих пор в примерах использовались в основном `re.search` и `re.findall`, будто это единственный способ применить паттерн к строке. На практике модуль `re` даёт с десяток функций, и выбор не между «правильной» и «неправильной», а между разными вопросами к тексту: есть ли совпадение где-нибудь, начинается ли строка с него, вся ли строка ему равна, сколько таких совпадений и что с ними делать дальше.

К концу раздела вы будете уверенно выбирать между `search`, `match`, `fullmatch`, `findall` и `finditer`, компилировать паттерн один раз вместо пересборки на каждом вызове, заменять текст через `sub` со строкой и с функцией, а также обходить самую коварную ловушку модуля — раздельные Python-строки и регулярки.

## search, match и fullmatch: три разных вопроса

`re.search(pattern, text)` ищет совпадение в любом месте строки и возвращает первое найденное. `re.match(pattern, text)` тоже начинает с позиции 0, но, в отличие от `fullmatch`, не требует, чтобы совпадение доходило до конца строки — хвост после него может быть каким угодно. `re.fullmatch(pattern, text)` самый строгий: вся строка целиком, от первого символа до последнего, обязана подойти под паттерн.

```python
import re

text = "тел: 89261234567"
print(re.match(r"\d{11}", text))
print(re.search(r"\d{11}", text))
print(re.fullmatch(r"\d{11}", "89261234567"))
```

`re.match` печатает `None`, потому что строка начинается с «тел», а не с цифры. `re.search` находит номер в середине и возвращает объект совпадения. `re.fullmatch`, применённый уже к чистому номеру без префикса, тоже находит совпадение — потому что теперь вся строка ему и равна.

Самая частая ошибка здесь — валидация формата через `re.match`. Если задача «проверить, что вся строка это валидный email», а не «найти email внутри произвольного текста», `re.match` не подходит: он промолчит про мусор после совпадения. `re.match(r"\d{11}", "89261234567abc")` вернёт совпадение, хотя строка явно не «просто номер». Для проверки формата целиком нужен `re.fullmatch`.

## findall и finditer: список строк против потока совпадений

`re.findall` возвращает список — но, как вы видели в концепте про группы, форма элементов списка зависит от числа групп в паттерне: без групп это полные совпадения, с одной группой её захваты, с двумя и больше кортежи. Из-за этого `findall` неудобен, когда, помимо самого текста совпадения, нужна ещё его позиция в строке.

`re.finditer` решает эту проблему: он всегда возвращает итератор объектов `Match`, независимо от числа групп. У каждого объекта есть `.group()`, `.start()`, `.end()` и `.span()`.

```python
import re

text = "Иванов И.И.: 89261234567, Петрова А.С.: 84951234567"
for m in re.finditer(r"\d{11}", text):
    print(m.group(), m.span())
```

Печатается `89261234567 (13, 24)` и `84951234567 (40, 51)`. `findall` с тем же паттерном без групп вернул бы просто `['89261234567', '84951234567']` — текст без позиций. Если позиция не нужна, `findall` короче; если нужна позиция или сам объект совпадения (например, чтобы потом вызвать `.groupdict()`), берите `finditer`.

## re.compile: паттерн один раз, вызовов много

Каждый вызов `re.search(pattern, text)` требует, чтобы Python разобрал строку `pattern` в объект регулярного выражения. Модуль `re` кэширует последние скомпилированные паттерны, поэтому одиночный вызов не катастрофа, но в цикле по тысячам строк с одним и тем же паттерном явная компиляция и читается яснее, и не зависит от размера внутреннего кэша.

```python
import re

PHONE_RE = re.compile(r"\d{11}")

customers = ["89261234567", "не указан", "84951234567"]
valid = [c for c in customers if PHONE_RE.fullmatch(c)]
print(valid)
```

Печатается `['89261234567', '84951234567']`. `PHONE_RE` скомпилирован один раз вне цикла, а внутри вызываются его методы `.fullmatch()`, `.search()`, `.findall()` — те же самые функции, но как методы объекта, без имени паттерна первым аргументом. Это особенно важно вместе с `re.VERBOSE` из прошлого концепта: многострочный паттерн с комментариями почти всегда стоит собирать через `re.compile` один раз на уровне модуля, а не пересобирать на каждый вызов.

## re.sub: замена строкой и замена функцией

`re.sub(pattern, repl, text)` заменяет каждое совпадение на `repl`. Если `repl` строка, в ней можно ссылаться на захваченные группы через `\g<1>` или короткую форму `\1`:

```python
import re

date = "Победа: 09.05.1945"
iso = re.sub(r"(\d\d)\.(\d\d)\.(\d{4})", r"\g<3>-\g<2>-\g<1>", date)
print(iso)
```

Печатается `Победа: 1945-05-09`. Форма `\g<3>` предпочтительнее короткой `\3` по конкретной причине: если сразу после номера группы в строке замены должна идти цифра, `\g<3>` однозначна, а `\30` движок прочитает как «группа номер 30», а не «группа 3, затем символ 0». `\g<3>0` такой проблемы не создаёт.

Когда замена не сводится к перестановке кусков, вместо строки `repl` передают функцию — она получает объект `Match` и возвращает строку замены:

```python
import re

def mask(m):
    digits = m.group()
    return digits[:4] + "*" * (len(digits) - 4)

text = "карта 4276123456781234, код 89261234567"
print(re.sub(r"\d{11,}", mask, text))
```

Печатается `карта 4276************, код 8926*******`, потому что каждое совпадение обрабатывается функцией независимо: первые четыре цифры остаются, остальные заменяются на `*`. Функция даёт доступ ко всему объекту совпадения — `m.start()`, `m.group("name")` и так далее, а не только к тексту, что невозможно при замене строкой.

## re.split: делит строку, но не теряет группы

`re.split(pattern, text)` режет строку по каждому совпадению паттерна и возвращает список кусков между ними. Неочевидная деталь: если в паттерне есть захватывающие группы, их содержимое тоже попадает в результат — между кусками текста, как разделители, которые сами хочется сохранить.

```python
import re

text = "Иванов,Петров;Сидоров"
print(re.split(r"[,;]", text))
print(re.split(r"([,;])", text))
```

Первый вызов печатает `['Иванов', 'Петров', 'Сидоров']` — обычное разбиение без групп. Второй, с группой вокруг класса, печатает `['Иванов', ',', 'Петров', ';', 'Сидоров']`: разделители тоже в списке, каждый на своей позиции. Это полезно, когда после разбиения важно помнить, каким именно символом были разделены соседние куски, и бесполезный сюрприз, когда об этом не думали заранее и код по индексам списка вдруг съезжает.

## Где re в Python удивляет

Самый опасный сюрприз не про сам паттерн, а про Python-строку, в которую он записан. `\b` внутри обычной (не «сырой») строки — это не «граница слова» для движка, а управляющий символ backspace, потому что `\b` входит в стандартные escape-последовательности Python наравне с `\n` и `\t`. Паттерн собирается неправильно ещё до того, как до него доходит модуль `re`:

```python
import re

pattern_bad = "\bкот\b"
pattern_good = r"\bкот\b"
print(re.findall(pattern_bad, "домашний кот спит"))
print(re.findall(pattern_good, "домашний кот спит"))
```

Первый вызов печатает `[]`: строка `pattern_bad` на самом деле состоит из символа backspace, слова «кот» и ещё одного backspace, и такой последовательности в тексте, конечно, нет. Второй печатает `['кот']`, потому что префикс `r` перед строкой отключает интерпретацию `\b` как escape-последовательности и передаёт движку ровно два символа: обратный слеш и букву «b». Правило простое: любой паттерн `re` пишите как raw-строку, `r"..."`, без исключений — даже если сейчас в нём нет `\b`, кто-то добавит его при следующей правке.

Второй сюрприз уже встречался в концепте про группы, но здесь он особенно легко ловится: добавление «безобидной» группы в паттерн меняет форму результата `findall` с списка строк на список кортежей, и код, который ожидал `for phone in re.findall(...)`, вдруг получает кортежи вместо строк.

Третий сюрприз — `re.match` против `re.fullmatch` для проверки формата, разобранный выше: `match` проверяет только начало, не конец, и валидатор на его основе пропускает мусор в хвосте строки.

## Частые ошибки

- **`re.match` вместо `re.fullmatch` в валидации.** `match` не проверяет, что строка закончилась там же, где и совпадение — «89261234567мусор» пройдёт проверку `\d{11}`. Для «строка целиком обязана подойти» нужен `fullmatch`.
- **Паттерн в обычной строке, не в raw.** `\b`, а иногда и другие последовательности, совпадающие с escape-кодами Python, молча превращаются в другие символы ещё до того, как их увидит `re`. Пишите паттерны только как `r"..."`.
- **`\1` в строке замены перед цифрой.** `re.sub(pattern, r"\1" + "0", text)` работает, а `re.sub(pattern, r"\10", text)` — нет, потому что `\10` читается как группа 10. Используйте `\g<1>0`.
- **Пересборка одного и того же паттерна в цикле.** Работает благодаря внутреннему кэшу `re`, но менее явно и менее предсказуемо, чем один `re.compile` на уровне модуля и переиспользование объекта.

<Detail summary="Кэш модуля re: почему re.search(pattern, ...) в цикле не катастрофа, но и не идеал">

Модуль `re` хранит внутри себя кэш последних скомпилированных паттернов (историческая граница — несколько сотен записей), поэтому повторный вызов `re.search("тот же паттерн", text)` обычно не пересобирает объект заново, а достаёт готовый из кэша. Это спасает от катастрофы, если код вызывает функции модуля напрямую внутри цикла, но не от двух более мелких минусов: словарь кэша общий на весь процесс и может вытеснить ваш паттерн чужим при большом разнообразии регулярок, а сам поиск в кэше по строке паттерна чуть дороже, чем прямой вызов метода уже готового объекта. Явный `re.compile` на уровне модуля, вне функций и циклов, снимает оба вопроса и дополнительно документирует: вот тут используется регулярка, и вот с какими флагами.

```python
import re

TAG_RE = re.compile(r"<(\w+)>")
print(TAG_RE.findall("<div><span></span></div>"))
```

Печатается `['div', 'span']` — метод `.findall()` компилированного объекта делает ровно то же самое, что модульная функция `re.findall(TAG_RE.pattern, text)`, но без обращения к кэшу на каждый вызов.

</Detail>
