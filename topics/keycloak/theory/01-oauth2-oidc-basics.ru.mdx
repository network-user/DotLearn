# OAuth2 и OIDC: кто кого пускает и по какому документу

Представьте, что стороннему сервису нужно прочитать ваши фотографии в облаке. Самый прямолинейный способ - отдать ему логин и пароль от облака. Это плохо по очевидной причине: сервис получает всё, навсегда, и вы уже не можете забрать доступ, не сменив пароль. OAuth2 родился ровно из этой проблемы - как отдать ограниченный доступ к ресурсу, не показывая никому свой пароль. Он отвечает на вопрос «что этому приложению разрешено делать от моего имени», то есть решает задачу **авторизации**.

Но у разработчиков был и второй, не менее частый запрос: «дайте мне просто кнопку *войти через Google*, чтобы я знал, кто этот пользователь». Это уже не про доступ к ресурсу, а про личность - **аутентификацию**. OAuth2 для этого не предназначен, и попытки натянуть его на вход приводили к тонким уязвимостям. Поэтому поверх OAuth2 надстроили OpenID Connect (OIDC) - тонкий слой, который добавляет проверяемое удостоверение личности. К концу раздела вы будете чётко различать эти две задачи, знать четыре роли OAuth2 и понимать, чем `access_token` отличается от `id_token`.

## Ментальная модель: четыре роли вокруг одного согласия

В основе OAuth2 - разговор четырёх участников. **Resource Owner** - это вы, владелец данных. **Client** - приложение, которое хочет действовать от вашего имени (мобильный клиент, SPA, backend-сервис). **Authorization Server** - тот, кто спрашивает ваше согласие и выдаёт token; в нашей теме это Keycloak. **Resource Server** - API, которое хранит данные и пускает только по валидному token.

Ключевая идея: client никогда не получает ваш пароль. Он отправляет вас к authorization server, вы там логинитесь и подтверждаете согласие, а client получает обратно не ваши credentials, а **access_token** - строку с ограниченными правами и сроком жизни. С этим token client идёт к resource server, и тот проверяет права по token, а не по личности пользователя.

Запомните этот треугольник: владелец доверяет authorization server, client получает token, resource server верит token. Пароль остаётся между владельцем и authorization server и больше нигде не появляется.

## Scope: ограниченное согласие, а не «всё или ничего»

Согласие в OAuth2 - не выключатель, а набор галочек. **Scope** - это список запрашиваемых разрешений, который client указывает при обращении к authorization server. Пользователю показывают именно эти пункты: «приложение хочет читать ваши фото» - и он соглашается на конкретный объём, а не на полный доступ.

```text
GET /authorize?response_type=code
  &client_id=gallery-app
  &scope=photos:read profile
  &redirect_uri=https://gallery.app/callback
```

Здесь client просит два scope: `photos:read` и `profile`. Выданный access_token будет нести ровно эти права - попытка удалить фото с таким token упрётся в отказ resource server. Scope - это механизм наименьших привилегий: вы выдаёте минимум, нужный для задачи.

<Callout type="info">
Scope отвечает на вопрос «что приложению можно делать», но ничего не говорит о том, *кто* пользователь. Имя, email и факт входа - это уже территория OIDC, к которой мы переходим.
</Callout>

## OIDC: удостоверение личности поверх OAuth2

OAuth2 выдаёт access_token - пропуск к ресурсу. Но в самом access_token, по спецификации OAuth2, нет гарантий о личности пользователя: его формат не стандартизован, и client не должен его «читать», он лишь передаёт его resource server. Когда задача - именно вход, нужен документ, который client может проверить и которому может верить. Это и есть **id_token** из OIDC.

OIDC добавляет к OAuth2 один специальный scope - `openid` - и новый артефакт: **id_token** в формате JWT, подписанный authorization server. В нём лежат **claims** о пользователе: `sub` (стабильный идентификатор), `iss` (кто выдал), `aud` (для кого), время выдачи и истечения, опционально email и имя.

```text
scope=openid profile email
  →  выдаётся id_token (JWT) + access_token
```

Разница в назначении принципиальна: **access_token - для доступа к API, id_token - для самого приложения**, чтобы узнать, кто вошёл. Resource server проверяет access_token; id_token до resource server обычно не доходит вовсе.

Хорошая мнемоника: если вы строите «вход через X» и хотите знать личность - вам нужен OIDC и id_token. Если вы строите доступ к API от имени пользователя - вам достаточно OAuth2 и access_token. Чаще всего в одном потоке вы получаете оба.

## Откуда client знает адреса: discovery-документ

Чтобы не зашивать URL руками, OIDC определяет **discovery endpoint** - один JSON по фиксированному пути, где authorization server публикует все свои адреса и параметры.

```text
GET /.well-known/openid-configuration
```

Ответ содержит `authorization_endpoint`, `token_endpoint`, `jwks_uri` (откуда брать ключи для проверки подписи), список поддерживаемых scope и алгоритмов. Библиотеки читают этот документ один раз и дальше знают, куда слать запросы. Для Keycloak путь будет вида `/realms/<realm>/.well-known/openid-configuration` - что такое realm, разберём в следующем разделе.

## Краевой случай: access_token, который вы не должны разбирать

Вот что удивляет многих: по спецификации OAuth2 **формат access_token не определён**. Он может быть JWT, а может быть непрозрачной случайной строкой (opaque token), за которой стоит запись в базе authorization server. Client не должен пытаться его декодировать или принимать решения по его содержимому - для client это просто «строка, которую надо передать дальше».

```text
access_token = "8xLOxBtZp8...opaque..."   допустимо
access_token = "eyJhbGciOi...JWT..."       тоже допустимо
```

Keycloak по умолчанию выдаёт access_token в формате JWT, и поэтому его *можно* разобрать на resource server. Но привычка «распарсю access_token на фронте и достану email» - это ошибка дизайна: для личности есть id_token, а access_token завтра может стать opaque, и код сломается.

## Типичные ошибки

- **Использовать OAuth2 для входа без OIDC.** Брать access_token как доказательство «пользователь вошёл» опасно: token могли выдать другому client, и без проверки `aud`/`iss` вы примете чужой пропуск за вход. Для аутентификации существует id_token - используйте его.
- **Путать access_token и id_token по назначению.** Отправлять id_token в заголовке `Authorization` к API или, наоборот, разбирать access_token ради email - частая путаница. Правило: access_token едет к API, id_token остаётся у client.
- **Считать scope ролями.** Scope - это согласие на объём доступа в рамках одного запроса, а не постоянные права пользователя в системе. Роли - отдельная модель, и в Keycloak это разные сущности.

<Detail summary="Почему OAuth2 сам по себе не годится для входа">

Исторически разработчики делали «login with X» прямо на OAuth2: получали access_token и считали, что раз токен выдан - пользователь аутентифицирован. Проблема в том, что access_token не привязан к конкретному client на уровне, который client может проверить. Атака «token substitution» выглядела так: вредоносное приложение получало access_token от своего пользователя и подсовывало его честному приложению, выдавая себя за этого пользователя.

OIDC закрывает это тем, что id_token содержит claim `aud` (audience) - идентификатор client, для которого token предназначен, - и подписан так, что client обязан проверить и подпись, и `aud`, и `iss`. Подменить id_token, не зная приватного ключа authorization server, нельзя. Поэтому правильный «вход через X» - всегда OIDC, а не голый OAuth2.

</Detail>
