# Realm, client, and roles: how Keycloak organizes access

In the previous section the authorization server was abstract — "whoever issues tokens". Now it has a name — Keycloak — and internal structure. When you open the Keycloak admin console for the first time, a dozen tabs hit you: realms, clients, users, roles, groups, client scopes. Without a map it is easy to pile everything into one heap and, six months later, end up with a system where nobody can tell who allowed what to whom.

Keycloak brings order with three main entities. **Realm** isolates whole worlds from each other. **Client** describes each application that talks to Keycloak. **Roles** define what a user or service may do. By the end of this section you will confidently answer which realm a user lives in, which client they represent, and where roles in a token come from — and you will not confuse realm roles with client roles.

## Mental model: a realm is a separate universe

**Realm** is an isolated space with its own users, clients, roles, and signing keys for tokens. A user from one realm does not exist in another; a token signed by one realm's keys will not be accepted in another. Typical split is by environment or product: one realm for employees, another for external customers.

```text
realm: corp           realm: customers
  users: alice, bob     users: c-12345, c-98472
  clients: hr-portal    clients: shop-spa
  keys: RS256 (key-A)   keys: RS256 (key-B)
```

The realm boundary is impermeable: `alice` from `corp` cannot sign into `shop-spa` even if she knows the password, because her account does not exist in realm `customers`. This isolation is the first thing you choose when designing a system: everything else lives *inside* a realm.

There is one special realm — `master`. It is created at install time and is for administering Keycloak itself. Do not keep application users in `master`: create separate realms for them so you do not accidentally grant server admin rights.

## Client: each application is its own record

**Client** is an application registration in a realm. Every application that wants tokens from Keycloak needs its own client with a unique `client_id`. The main distinction between clients is whether they can store a secret.

**Confidential client** is an app with a server side (backend) that can safely hold `client_secret`. **Public client** is a SPA or mobile app: its code is visible to the user, a secret cannot be hidden, so it has no secret at all.

```text
client: shop-backend    confidential   secret: ●●●●
client: shop-spa        public         secret: (none)
```

This distinction determines which authentication flow applies. A public client cannot prove itself with a secret, so it relies on PKCE (covered in the next section). A confidential client proves itself with a secret and can use the client credentials flow for machine access.

<Callout type="warning">
Never make a SPA or mobile app a confidential client with an embedded secret. A secret in code the user downloads is not a secret. Such clients must be public and use PKCE.
</Callout>

## Roles: realm roles and client roles

**Role** in Keycloak is a named permission you assign to a user or service. There are two kinds, easy to confuse.

**Realm role** is realm-wide, shared by all clients: e.g. `admin` or `user`. **Client role** is defined inside a specific client and only meaningful there: client `shop-backend` may have role `order:refund`, which means nothing to other applications.

```text
realm roles (shared):     admin, user, auditor
client roles (local):
  shop-backend:  order:refund, order:read
  hr-portal:     employee:edit
```

Choose by scope. If a permission makes sense across the whole realm — make it a realm role. If it is about one application — a client role inside that client. That split keeps roles from becoming a pile of hundreds of global names.

Roles do not land in a token by themselves — a client **mapper** does that. By default Keycloak puts assigned roles in claim `realm_access.roles` (realm roles) and `resource_access.<client>.roles` (client roles). These are the fields the resource server reads when deciding whether to admit a request.

## Groups and composite roles: scaling assignments

Assigning roles one by one to every user does not scale. **Group** is a set of users with a shared role assignment: add someone to group `support` and they get all its roles at once. Groups form a hierarchy; a nested group inherits parent roles.

Separately there is **composite role** — a role that includes other roles. Assigning composite role `manager` may automatically pull `order:read`, `order:refund`, and realm role `user`. Convenient, but dangerous: composite roles easily balloon, and a person gets more than the name suggests.

## Edge case: one user, two clients, different roles

What is not obvious: a user's client roles can differ per client, and the token only includes roles for clients relevant to the request. A user may be `admin` in `hr-portal` and have no roles in `shop-backend`.

```text
user alice:
  resource_access.hr-portal.roles    = ["employee:edit"]
  resource_access.shop-backend.roles = []
```

Practical rule: the resource server must check roles in *its* section `resource_access.<its-client>`, not anywhere else. If backend `shop-backend` admits by role `employee:edit` from `hr-portal`, it opens access to people who were never granted it in the shop context.

## Common mistakes

- **Putting application users in realm `master`.** `master` is for Keycloak administration; a user there may manage the server. Application users live in a separate realm.
- **Making a public app confidential with an embedded secret.** A secret in a JS bundle or mobile binary is trivial to extract. SPAs and mobile clients are public clients with PKCE.
- **Mixing realm roles and client roles by meaning.** Global permission (`admin`) is a realm role; app-specific permission (`order:refund`) is a client role of that app. Otherwise roles sprawl and lose meaning.
- **Reading roles from the wrong token section.** The resource server must look at `resource_access.<its-client>.roles`, not another client's roles, or authorization leaks across applications.

<Detail summary="Service account: the client as a user">

For a confidential client you can enable **service account** — then the client itself becomes a "user without a human". That is for machine access: a background service calls an API in its own name, not as a person.

The service account lives in the same realm, has its own assigned roles (via the Service account roles tab), and gets a token through the client credentials flow covered in the next section. In such a token subject (`sub`) points at the client itself, not a person, and claim `typ` helps the resource server tell machine access from user access. This is the standard way to give a service exactly the rights it needs without inventing a fake "person".

</Detail>
