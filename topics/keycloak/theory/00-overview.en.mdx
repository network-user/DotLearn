Almost every application eventually gets a "sign in" button, and there is almost always a temptation to store usernames and passwords yourself. Then it only gets worse: a mobile client, a background service, and a "sign in with Google" button attach to the same user, and the question "who is this person and what are they allowed to do" spreads across the whole codebase. This topic is about moving login and access control into a separate service that all your applications trust, so you stop building a custom password system in every project. Keycloak plays that role here, with two standards underneath: OAuth2 and OIDC.

**What you'll be able to do.** Tell apart when the question is user identity versus access rights; organize users, applications, and roles in a clear Keycloak structure; pick the right login flow for a browser app and a separate one for machine access; read token contents and validate them without calling the server on every request.

**Route through this topic.** Four sections, each built around a concrete situation:

1. **OAuth2 and OIDC** — why handing your password to a third-party service is a bad idea, and how the standards separate access from identity.
2. **The Keycloak model** — how realm, client, and roles bring order where the admin console dumps a dozen tabs on you.
3. **Login flows** — why a browser app needs redirects with PKCE and a background service needs a secret.
4. **JWT and validation** — how a resource server validates a token locally and why refresh lives alongside it.

**What you need first.** Nothing to install: the sections use diagrams and walkthroughs, with no code to run. It is enough to understand that an application calls an API over the network and that a user can have a session — everything else, from roles to token signatures, we cover along the way.
