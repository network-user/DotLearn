# Потоки получения token: редиректы для людей, секрет для машин

У вас есть client и realm - теперь нужно, чтобы client реально получил token. Способ получения token называется **flow** (поток, grant type), и их несколько не от избытка фантазии: разные клиенты находятся в разных условиях безопасности. Браузерное SPA не может хранить секрет, но может водить пользователя по редиректам. Фоновый сервис, наоборот, секрет хранить умеет, но никакого человека за ним нет, чтобы тот вводил пароль.

Из этого вырастают два главных потока. **Authorization code flow с PKCE** - для входа человека через браузер; это сегодня дефолт для веба и мобильных приложений. **Client credentials flow** - для машинного доступа, когда сервис ходит в API от своего имени. К концу раздела вы будете понимать, зачем нужен промежуточный `code`, как PKCE защищает его перехват и почему один поток требует присутствия человека, а другой - нет.

## Ментальная модель: почему не отдать token сразу

Самый прямой путь - чтобы authorization server после логина сразу отдал token в редиректе обратно client. Так когда-то и работал implicit flow, и именно поэтому от него отказались: token летел через адресную строку браузера, попадал в историю, в логи прокси, в Referer. Перехватить его было слишком легко.

Authorization code flow вводит обходной манёвр. Сначала через браузер возвращается не token, а одноразовый **authorization code** - короткоживущий код, бесполезный сам по себе. Затем client меняет этот code на token прямым запросом backend-to-backend (или прямым POST из SPA), который не проходит через адресную строку. Token таким образом никогда не появляется в браузерной навигации.

Двухшаговость - это не бюрократия, а защита. Code виден в браузере, но без второго шага (обмена) он ничего не даёт. А второй шаг происходит там, куда у злоумышленника нет доступа. Остаётся одна дыра: что если злоумышленник перехватит code и сам выполнит обмен? Её закрывает PKCE.

## PKCE: привязка code к тому, кто его запросил

**PKCE** (Proof Key for Code Exchange) защищает от перехвата authorization code. Идея - доказать, что token меняет тот же client, что и запрашивал code, без участия секрета (которого у public client нет).

Перед стартом client генерирует случайную строку - **code_verifier** - и держит её у себя. Из неё он вычисляет **code_challenge** как `SHA-256` хэш и отправляет в `/authorize` именно challenge, а не verifier.

```text
code_verifier  = "dBjftJeZ4CVP...случайное"
code_challenge = BASE64URL(SHA256(code_verifier))
GET /authorize?...&code_challenge=...&code_challenge_method=S256
```

Когда приходит время менять code на token, client присылает в `/token` уже сам verifier. Keycloak хэширует присланный verifier и сверяет с challenge, который запомнил на первом шаге. Совпало - значит обмен делает тот же client.

```text
POST /token  code=<code>&code_verifier=dBjftJeZ4CVP...
Keycloak: SHA256(verifier) == сохранённый challenge ?  да → token
```

Перехваченный code теперь бесполезен: у злоумышленника нет verifier, а из challenge его не восстановить - это односторонний хэш. PKCE задумывался для мобильных приложений, но сейчас рекомендован для всех authorization code flow, включая confidential clients.

<Callout type="tip">
Всегда используйте `code_challenge_method=S256` (хэширование SHA-256). Метод `plain`, где challenge равен verifier, спецификация допускает только как запасной для платформ без SHA-256 - он не даёт защиты от перехвата.
</Callout>

## Client credentials: token без человека

Иногда никакого пользователя нет: фоновый сервис раз в минуту дёргает чужое API. Здесь нет кого редиректить и некому вводить пароль. Для этого случая есть **client credentials flow**: confidential client получает token, предъявляя свой `client_secret` напрямую, без браузера и без redirect.

```text
POST /realms/corp/protocol/openid-connect/token
  grant_type=client_credentials
  client_id=report-worker
  client_secret=●●●●
```

В ответ приходит access_token, выписанный на сам client (его service account), а не на человека. `id_token` здесь не выдаётся - личности нет, удостоверять нечего. Поток применим только к confidential clients: public client не может предъявить секрет, поэтому client credentials для него недоступен.

Различие между потоками сводится к одному вопросу: есть ли за запросом человек. Есть - authorization code с PKCE, и вы получаете id_token про этого человека. Нет - client credentials, и token представляет сам сервис.

## Краевой случай: redirect_uri должен совпадать точно

Вот что ловит многих на старте: Keycloak принимает code и меняет его на token, только если `redirect_uri` в запросе совпадает с одним из заранее зарегистрированных у client значений. Не «начинается с», не «похож» - совпадает по правилам, заданным в настройках client.

```text
зарегистрировано: https://app.example/callback
пришло:           https://app.example/callback/   ← лишний слеш → отказ
```

Это не придирка, а защита от open redirect: без строгой проверки злоумышленник подставил бы свой `redirect_uri` и увёл code на свой сервер. Поэтому при ошибке `Invalid redirect_uri` первым делом сверяйте строку символ в символ с тем, что прописано в client.

## Типичные ошибки

- **Использовать implicit flow в новых проектах.** Он отдаёт token прямо в редиректе, где его легко перехватить. Современный дефолт для браузера - authorization code с PKCE; implicit считается устаревшим.
- **Опускать PKCE у public client.** Без PKCE перехваченный authorization code можно обменять на token. Для SPA и мобильных приложений PKCE обязателен, метод - `S256`.
- **Пытаться использовать client credentials для входа пользователя.** Этот поток не аутентифицирует человека и не выдаёт id_token - он про сам сервис. Для входа человека нужен authorization code.
- **Слать `client_secret` из браузера.** Секрет, попавший в код фронтенда, скомпрометирован. Если потоку нужен секрет, его место - на backend (confidential client).

<Detail summary="state и nonce: ещё две защитные строки">

Кроме PKCE, authorization code flow обычно несёт два дополнительных одноразовых значения.

**state** - случайная строка, которую client кладёт в запрос `/authorize` и проверяет при возврате. Она защищает от CSRF на этапе логина: если вернувшийся `state` не совпадает с отправленным, ответ не от вашего запроса, и его надо отбросить.

**nonce** - значение для OIDC, которое client кладёт в запрос и затем ожидает увидеть тем же claim `nonce` внутри id_token. Это связывает конкретный id_token с конкретным запросом и не даёт переиспользовать старый перехваченный id_token. PKCE защищает обмен code, а nonce - привязку id_token; они решают разные задачи и работают вместе.

</Detail>
