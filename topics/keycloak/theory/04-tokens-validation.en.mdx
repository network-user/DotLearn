# JWT: a document you verify, not re-query

You got a token, but what is that string and why does the resource server trust it? If every API request made the resource server run to Keycloak and ask "is this token still valid?", you get a bottleneck: one extra network call per request, and the identity provider becomes a single point of failure. JWT solves this differently: the token carries what you need inside and is signed so the resource server checks it *locally*, without calling Keycloak.

That convenience has a price. Because the token is verified offline, it cannot be revoked instantly: it is valid until expiry. So sessions do not live forever and users do not log in every five minutes, there is a pair: access_token (short) and refresh_token (long, for renewal). By the end of this section you will understand JWT structure, the mandatory validation checklist, and why two tokens instead of one.

## Mental model: three parts separated by dots

**JWT** (JSON Web Token) is a string of three dot-separated parts: `header.payload.signature`. The first two are JSON encoded in Base64URL; **anyone can read them**. The third is a signature Keycloak attaches to the first two with its private key.

```text
eyJhbGciOiJSUzI1Ni…  .  eyJzdWIiOiJhbGljZ…  .  Q1f7kQ…
     header                   payload              signature
```

**Header** says which algorithm signed the token (e.g. `RS256`) and which key (`kid`). **Payload** carries claims: who (`sub`), issuer (`iss`), audience (`aud`), expiry (`exp`), plus roles. **Signature** is the result of signing header and payload with Keycloak's private key; you verify it with the public key and cannot forge it without the private key.

Main takeaway from this picture: **payload is not encrypted, it is signed**. Anyone can read email and roles from an intercepted token. What protects you is the inability to change content: any edit breaks the signature. So JWTs always travel over HTTPS and must not carry what must stay secret.

## Validation: what the resource server must check

Receiving a JWT in header `Authorization: Bearer <token>`, the resource server validates it itself. Minimum mandatory checks:

1. **Signature**: with Keycloak's public key (from `jwks_uri` via discovery). Mismatch: forged or corrupted token.
2. **`exp`**: not expired. Expired token is rejected.
3. **`iss`**: issued by the expected issuer (your realm address). Wrong issuer means wrong token.
4. **`aud`**: intended for this resource server. A token issued for another recipient must not be accepted.

```text
Authorization: Bearer eyJhbGci…
checks: signature ok, exp ok, iss == https://kc/realms/corp, aud ok
```

Skipping any check is a hole. Especially treacherous is skipping `aud`: without it the resource server accepts any validly signed token of this realm, even one issued to another application with different rights.

<Callout type="warning">
Not verifying the signature and trusting payload "because it is our Keycloak" is a grave mistake. JWT without signature verification is forged in seconds: change `"role":"user"` to `"role":"admin"` and without signature check the resource server will accept it.
</Callout>

## Keys and kid: how to find what to verify with

Keycloak signs tokens with a private key; the resource server verifies with the public key. Where does it get the public key? From **JWKS** (JSON Web Key Set) published at `jwks_uri`. The set may contain several keys; the token header carries `kid`: the id of the key that signed this token.

```text
header: { "alg": "RS256", "kid": "key-A" }
JWKS:   [ { "kid": "key-A", ... }, { "kid": "key-B", ... } ]
        → verify with key-A
```

This supports **key rotation**: Keycloak introduces a new key; for a while JWKS holds old and new, and the token is verified with the key whose `kid` is listed. The resource server caches JWKS but re-reads the set on an unknown `kid`, otherwise after rotation all tokens would fail validation.

## Access and refresh: short pass and long mandate

Because JWT is verified offline, you cannot revoke it before `exp`. The fix is to make access_token short (minutes) so the window for a stolen token is small. But making users log in every five minutes is impossible. So alongside it you get **refresh_token**: a long-lived token whose only job is to obtain a new access_token when the old one expires.

```text
POST /token  grant_type=refresh_token&refresh_token=<refresh>
  → new access_token (+ usually new refresh_token)
```

Do not send refresh_token to the API; store it carefully and show it only to Keycloak's token endpoint. Unlike access_token, refresh can be revoked: on logout or compromise Keycloak invalidates refresh_token and renewal stops, though the last issued access_token lives until its `exp`.

This pair is the compromise between offline verification convenience and session control. access_token is fast but non-revocable and short; refresh_token is revocable but must not go to the API. Together they give both performance and a way to end a session.

## Edge case: clock skew and `exp`

What surprises in production: a token Keycloak just issued is sometimes rejected as "not yet valid" or "already expired". The cause is not code but clocks. Checking `exp` (and `nbf`, not before) compares token time to the resource server's local clock. If servers drift by ten seconds, borderline tokens flip between accepted and rejected.

```text
exp = 12:00:00, resource server clock = 12:00:07  → "expired" 7 seconds early
```

So validators allow a small **clock skew**: a few seconds slack when comparing time. And on servers handling tokens, NTP-synced time is mandatory: time drift shows up as flaky authorization errors.

## Common mistakes

- **Trusting payload without verifying signature.** Header and payload are openly editable; without signature check anyone can grant themselves role `admin`. Always verify signature.
- **Not checking `aud` and `iss`.** Without `aud` the resource server accepts a token issued to another app; without `iss`, a token from another realm. Both checks are mandatory.
- **Putting secrets in payload.** JWT is signed, not encrypted: all content is readable from an intercepted token. Passwords, keys, and private data in claims are forbidden.
- **Sending refresh_token to the API or storing it carelessly.** refresh_token is a long-lived mandate for new access_tokens; its place is only in exchange with the token endpoint, not in a request to the resource server.

<Detail summary="Offline verification vs introspection">

The resource server has two ways to check an access_token. **Local (offline) verification**: parse the JWT and verify signature with the public key as above. Fast, no network, but no instant revocation: a revoked token counts as valid until `exp`.

**Token introspection** (RFC 7662): ask Keycloak directly via `/introspect` whether the token is active. Sees revocation immediately but adds a network call per check and dependency on Keycloak availability.

The trade-off: most APIs use offline verification with short access_tokens: stolen token window is small and Keycloak is not a bottleneck. Introspection is for cases where instant revocation is critical (e.g. money operations) and you can tolerate the extra call. Often combined: offline on the hot path, introspection on sensitive operations.

</Detail>
