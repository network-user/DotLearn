# useRef - карман, который переживает рендеры

Вы делаете секундомер. Кнопка «старт» запускает `setInterval`, кнопка «стоп» должна его остановить, а для этого нужно сохранить идентификатор таймера между кликами. Обычная переменная не подойдёт: она рождается и умирает вместе с каждым вызовом компонента-функции, а кликов будет много. Положить идентификатор в состояние тоже неправильно - число таймера нигде не показывается на экране, но каждая запись в состояние заставит компонент перерисоваться зря.

Нужен третий вид памяти: значение, которое переживает рендеры, как состояние, но менять его можно тихо, без перерисовки. Ровно это даёт `useRef`. Его часто представляют только как «ссылку на элемент DOM», и это правда лишь наполовину. По сути `useRef` - это карман компонента, куда можно положить что угодно, и оно останется там до размонтирования, никого не потревожив.

## Коробочка с полем current

`useRef(initial)` возвращает один и тот же объект на всех рендерах - обычный объект с единственным полем `current`, куда при первом вызове кладётся начальное значение. React не делает с этим объектом ничего умного: не следит за ним, не сравнивает, не перерисовывает компонент при записи в `current`. Он лишь гарантирует, что между рендерами вы получаете тот же самый объект, а не новый.

```jsx
function Stopwatch() {
  const timerRef = useRef(null);
  function start() {
    timerRef.current = setInterval(tick, 1000);
  }
  function stop() {
    clearInterval(timerRef.current);
  }
  return <div><button onClick={start}>старт</button><button onClick={stop}>стоп</button></div>;
}
```

`start` записывает идентификатор в `timerRef.current`, `stop` читает его оттуда. Между кликами компонент мог перерисоваться сколько угодно раз - объект `timerRef` тот же, значение на месте. Записи в `current` перерисовку не вызывают, и это именно то, что нужно: секундомер тикает, а React не дёргается на служебное число.

## Ref против состояния

Ref и состояние легко перепутать: оба помнят значение между рендерами. Разница в двух вещах, и обе решают, что выбрать.

Первое - перерисовка. Запись в состояние планирует новый рендер, запись в `current` не делает ничего видимого. Значит, если от значения зависит то, что на экране, это состояние. Если значение чисто служебное - идентификатор таймера, ссылка на узел, счётчик срабатываний для отладки, - это ref.

Второе - снимок против последнего значения. Состояние внутри рендера заморожено: `count` всегда равен снимку этого рендера. А `ref.current` всегда возвращает последнее записанное значение, без всякой заморозки. Ref не участвует в модели снимков, поэтому читать его во время рендера бессмысленно и опасно - вы увидите значение, не связанное с тем, что сейчас на экране.

<Callout type="warning">
Не читайте и не пишите `ref.current` во время рендера: тело компонента должно быть чистым, а ref - мутабельная величина вне модели снимков. Работайте с `current` в обработчиках событий и внутри эффектов, где рендер уже завершён.
</Callout>

Мнемоника: состояние - это то, что React показывает; ref - это то, что React запоминает, но не показывает.

## Ссылка на узел DOM

Второе применение вырастает из первого. Иногда нужно обратиться к настоящему элементу страницы: поставить фокус в поле, измерить размер, запустить воспроизведение видео. React сам создаёт и обновляет DOM, но даёт вам доступ к узлу через тот же `useRef`. Передайте ref в атрибут `ref` элемента, и после коммита React запишет в `current` соответствующий узел.

```jsx
function SearchBox() {
  const inputRef = useRef(null);
  useEffect(() => {
    inputRef.current.focus();
  }, []);
  return <input ref={inputRef} placeholder="Поиск по Ozon" />;
}
```

При монтировании React создаёт `<input>`, кладёт его в `inputRef.current`, и эффект ставит фокус. Почему в эффекте, а не в теле? Потому что во время рендера узла ещё нет - он появляется только после коммита. Ref к DOM читают там же, где и любой другой ref: после того, как экран собран.

## Измерить и прокрутить

Ссылка на узел нужна не только для фокуса. Через неё измеряют элемент и управляют прокруткой - тем, что React декларативно не выражает. Классический пример: список сообщений в чате, который после отправки должен прокручиваться к последнему.

```jsx
function Messages({ items }) {
  const bottomRef = useRef(null);
  useEffect(() => {
    bottomRef.current.scrollIntoView({ behavior: 'smooth' });
  }, [items]);
  return (
    <div>
      {items.map((m) => <p key={m.id}>{m.text}</p>)}
      <div ref={bottomRef} />
    </div>
  );
}
```

Пустой `<div>` в конце служит якорем: `bottomRef` указывает на него, а эффект после каждого нового сообщения зовёт `scrollIntoView`. Почему снова в эффекте, а не в обработчике отправки? Потому что на момент обработчика новое сообщение ещё не в DOM - оно появится после того, как React закоммитит обновлённый список. Эффект с зависимостью `[items]` срабатывает именно тогда, когда узлы уже на месте. Тот же приём - ссылка на узел плюс работа с ним в эффекте - лежит в основе измерений (`getBoundingClientRect`), запуска видео и интеграции со сторонними библиотеками, которые ждут готовый элемент DOM.

## Частые ошибки

**Ref вместо состояния для видимых данных.** Кладут счётчик лайков в `ref`, увеличивают `current` и удивляются, что число на экране не меняется. Запись в ref не перерисовывает компонент. Всё, что видно пользователю, живёт в состоянии.

**Чтение current во время рендера.** Строчка вроде `return <p>{countRef.current}</p>` иногда показывает старое значение, иногда новое, в зависимости от того, кто и когда писал в ref. Тело компонента должно зависеть только от пропсов и состояния. Нужно показать - заведите состояние.

**Обращение к `current` до коммита.** Попытка `inputRef.current.focus()` прямо в теле компонента падает: там `current` ещё `null`. Работа с узлом - только после монтирования, в эффекте.

<Detail summary="Почему ref не вызывает перерисовку - и это хорошо">

Могло бы показаться удобным, чтобы запись в ref тоже перерисовывала компонент - тогда не пришлось бы выбирать между ref и состоянием. Но именно отсутствие перерисовки делает ref полезным. Есть значения, которые обязаны существовать, но не должны влиять на картинку: идентификаторы таймеров, предыдущее значение пропса для сравнения, флаг «первый ли это рендер», ссылки на внешние библиотеки. Заведи вы их в состоянии - каждое служебное изменение гоняло бы лишний рендер, а иногда и зацикливало бы эффект, который сам же меняет это состояние. Ref разрывает эту связь: значение живёт, но React про него не думает, пока вы сами не попросите через состояние.
