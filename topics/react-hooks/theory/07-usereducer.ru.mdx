# useReducer - состояние как список переходов

Форма оформления заказа разрослась. Есть адрес, способ доставки, промокод, флаг «нужен чек», список товаров с количеством. Каждое поле - свой `useState`, обработчиков десяток, и логика начинает переплетаться: выбрали самовывоз - обнуляется адрес и меняется цена, ввели промокод - пересчитывается скидка и, может быть, слетает бесплатная доставка. Обновления одного поля тянут за собой другие, обработчики распухают, и уследить, в каком состоянии сейчас форма, всё труднее.

Проблема не в количестве полей, а в том, что логика изменений размазана по компоненту. `useReducer` собирает её в одном месте. Идея древняя и надёжная: вместо того чтобы менять состояние тут и там, вы описываете все возможные переходы одной функцией, а компонент лишь сообщает, что произошло. Разберём, как это устроено и почему для сложного состояния такой подход спокойнее, чем россыпь сеттеров.

## Reducer, действие и dispatch

`useReducer` принимает функцию-редьюсер и начальное состояние, а возвращает пару: текущее состояние и функцию `dispatch`. Редьюсер - это чистая функция `(state, action) => newState`: она получает текущее состояние и объект-действие, описывающий, что случилось, и возвращает новое состояние. Компонент не меняет состояние сам; он вызывает `dispatch(action)`, а React прогоняет редьюсер и берёт то, что тот вернул.

```jsx
function reducer(state, action) {
  switch (action.type) {
    case 'increment':
      return { count: state.count + 1 };
    case 'reset':
      return { count: 0 };
    default:
      return state;
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, { count: 0 });
  return <button onClick={() => dispatch({ type: 'increment' })}>{state.count}</button>;
}
```

Клик отправляет действие `{ type: 'increment' }`. React вызывает `reducer` с текущим состоянием и этим действием, получает новый объект и планирует рендер с ним. Обработчик в компоненте больше не знает, как именно меняется счётчик, - он лишь заявляет о событии. Вся механика «как» переехала в редьюсер.

## Действия описывают что, а не как

Ключевая ценность - в языке действий. Действие говорит, что произошло с точки зрения пользователя: «добавлен товар», «применён промокод», «выбран самовывоз». А как на это отреагировать - решает редьюсер, в одном месте. Когда логика меняется, вы правите редьюсер, а не полтора десятка обработчиков.

```jsx
function cartReducer(state, action) {
  switch (action.type) {
    case 'add':
      return { ...state, items: [...state.items, action.item] };
    case 'applyPromo':
      return { ...state, discount: 0.1 };
    default:
      return state;
  }
}
```

Обратите внимание на `...state`: редьюсер не меняет старое состояние на месте, а собирает новый объект, копируя нетронутые поля. Это обязательное условие - как и в `useState`, React сравнивает ссылки, и мутация старого объекта может не вызвать перерисовку. Редьюсер обязан быть чистым: никаких запросов к сети, таймеров или записи в `ref` внутри него, только вычисление нового состояния из старого и действия.

<Callout type="tip">
`dispatch` стабилен: React гарантирует, что его ссылка не меняется между рендерами. Поэтому его можно смело указывать в массиве зависимостей эффекта и не заворачивать в `useCallback`.
</Callout>

## Действия в очереди складываются

У редьюсера есть приятное свойство, которого не хватало прямой форме `useState`. Несколько `dispatch` в одном обработчике батчатся, но, в отличие от `setCount(count + 1)`, редьюсер каждый раз получает уже обновлённое состояние из очереди, а не замороженный снимок. То есть `dispatch` ведёт себя как функциональный апдейт по своей природе.

```jsx
function handleClick() {
  dispatch({ type: 'increment' });
  dispatch({ type: 'increment' });
}
```

Два действия подряд дадут плюс два, а не плюс один: первый вызов редьюсера превратит 0 в 1, второй получит эту единицу и сделает 2. Думать о снимках и функциональной форме здесь не нужно - редьюсер и так всегда видит свежее состояние очереди.

## useReducer или useState

Оба хука взаимозаменяемы: любой редьюсер можно разложить на несколько `useState`, и наоборот. Выбор - вопрос удобства. `useState` выигрывает для простого независимого состояния: флаг, строка ввода, счётчик. `useReducer` окупается, когда состояние составное и его части меняются вместе, когда переходов много и они нетривиальны, или когда логику обновления хочется вынести из компонента и покрыть тестами - редьюсер тестируется как обычная функция, без всякого React.

## Список дел одним редьюсером

Полноценный пример показывает, зачем всё это. Список задач умеет три вещи: добавить, переключить готовность, удалить. На `useState` это три обработчика, каждый со своей возней над массивом. На редьюсере - одна функция и три коротких действия.

```jsx
function todosReducer(todos, action) {
  switch (action.type) {
    case 'added':
      return [...todos, { id: action.id, text: action.text, done: false }];
    case 'toggled':
      return todos.map((t) => (t.id === action.id ? { ...t, done: !t.done } : t));
    case 'deleted':
      return todos.filter((t) => t.id !== action.id);
    default:
      return todos;
  }
}
```

Каждая ветка возвращает новый массив, не трогая старый: `map` для точечной замены, `filter` для удаления, spread для добавления. Компонент теперь пишет `dispatch({ type: 'toggled', id })` и ничего не знает о том, как устроено переключение. Вся логика списка собрана в одном месте, читается сверху вниз как перечень возможных событий, и тестируется без React: вызвали `todosReducer` со стартовым массивом и действием, сравнили результат. Разрастётся приложение - редьюсер останется одной понятной функцией, а не десятком расползшихся по компоненту сеттеров.

## Частые ошибки

**Мутация состояния в редьюсере.** `state.items.push(action.item); return state;` - и перерисовка не происходит, потому что ссылка на объект та же. Всегда возвращайте новый объект или массив: `[...state.items, action.item]`.

**Побочные эффекты внутри редьюсера.** Запрос к серверу или запись в `localStorage` прямо в редьюсере ломают его чистоту: React может вызвать редьюсер дважды при проверках в разработке, и эффект выполнится лишний раз. Побочные действия - в обработчике или в `useEffect`, редьюсер только считает.

**Забытый `default`.** Если редьюсер не вернёт состояние для неизвестного действия, оно станет `undefined`, и компонент упадёт. Ветка `default: return state` обязательна.

<Detail summary="Почему эта модель старше React">

Схема «состояние плюс список переходов» называется конечным автоматом и известна задолго до фронтенда. Тот же принцип лежит в основе Redux, откуда React и позаимствовал слово reducer: имя пришло от метода массивов `reduce`, который сворачивает последовательность значений в одно, применяя функцию шаг за шагом. Список действий, свёрнутый редьюсером в текущее состояние, - буквально `reduce` по истории событий. Полезное следствие: раз состояние определяется только начальным значением и последовательностью действий, его легко воспроизвести, залогировать и откатить. На этом строятся инструменты отладки с «машиной времени», где можно отмотать состояние назад по списку действий. useReducer - маленькая дверь в эту большую и проверенную идею.

</Detail>
