# Кастомные хуки - вынос логики, а не состояния

В трёх местах приложения повторяется один и тот же узор. Компонент подписывается на ширину окна: заводит состояние, в эффекте вешает слушатель `resize`, в очистке его снимает, при размонтировании прибирается. Пять строк, но они копируются из компонента в компонент, и если завтра понадобится добавить троттлинг, править придётся во всех трёх. Логика с состоянием просится в переиспользование, но обычная функция сюда не годится: из неё нельзя звать `useState` и `useEffect` - правила хуков запрещают.

Выход - кастомный хук. Это ваша собственная функция, которая имеет право вызывать другие хуки, потому что сама является хуком. Она вбирает в себя кусок логики с состоянием и эффектами, а компоненты просто зовут её. Разберём, как такую функцию объявить, что происходит с состоянием при повторных вызовах и где кастомные хуки особенно уместны.

## Функция, которая начинается с use

Кастомный хук - это функция, имя которой начинается с `use` и которая внутри вызывает хуки React. Больше никакой магии: ни регистрации, ни специального синтаксиса. Префикс `use` - не украшение, а сигнал и для линтера, и для читателя: перед вами не обычная функция, к ней применяются правила хуков.

```jsx
function useWindowWidth() {
  const [width, setWidth] = useState(window.innerWidth);
  useEffect(() => {
    const onResize = () => setWidth(window.innerWidth);
    window.addEventListener('resize', onResize);
    return () => window.removeEventListener('resize', onResize);
  }, []);
  return width;
}

function Header() {
  const width = useWindowWidth();
  return <span>{width < 640 ? 'Мобильный' : 'Десктоп'}</span>;
}
```

Вся возня со слушателем спрятана в `useWindowWidth`, а `Header` получает готовое число и забывает про подписку. Возвращать хук может что угодно: одно значение, как здесь, пару `[значение, функция]` в духе `useState`, или объект с несколькими полями. Форму выбирают по удобству вызова.

## Каждый вызов - своё состояние

Здесь кроется тонкость, которая сначала удивляет. Кастомный хук переиспользует логику, но не состояние. Каждый вызов хука заводит собственный, полностью независимый набор состояния и эффектов - ровно так же, как два вызова `useState` в одном компоненте дают две разные ячейки.

```jsx
function useCounter(start) {
  const [n, setN] = useState(start);
  const increment = () => setN((x) => x + 1);
  return [n, increment];
}

function Panel() {
  const [likes, likePlus] = useCounter(0);
  const [reposts, repostPlus] = useCounter(0);
}
```

`likes` и `reposts` не связаны: нажатие «лайк» не трогает счётчик репостов, хотя оба пришли из одного `useCounter`. Хук дал им общий рецепт, но каждая порция состояния своя. То же верно и для разных компонентов: два `Header` на странице, каждый со своим `useWindowWidth`, ведут по отдельному слушателю и по отдельному значению ширины.

Мнемоника: кастомный хук - как рецепт блюда. Рецепт один, но каждый, кто по нему готовит, получает своё блюдо на своей тарелке. Общего котла нет. Если общее состояние всё-таки нужно - скажем, один пользователь на всё приложение, - его поднимают в контекст или менеджер состояния, а хук лишь читает оттуда.

## Где кастомные хуки особенно к месту

Три типовых повода вынести логику в хук. Первый - повторяющаяся подписка на внешнее: размер окна, статус сети, положение прокрутки, значение из `localStorage`. Второй - работа с данными: `useUser(id)` прячет загрузку, состояние ожидания и ошибку за одним вызовом. Третий - производные приёмы над состоянием: `useToggle`, `useDebounce`, `usePrevious` - маленькие удобства, которые приятно иметь под рукой в едином виде.

```jsx
function useToggle(initial = false) {
  const [on, setOn] = useState(initial);
  const toggle = () => setOn((v) => !v);
  return [on, toggle];
}
```

`useToggle` в четыре строки убирает из компонентов повторяющийся код открытия и закрытия модалок, сворачивания панелей, переключения флагов. Компонент пишет `const [open, toggleOpen] = useToggle()` и не думает о механике.

## Хук, синхронизирующий с localStorage

Соберём хук посерьёзнее, объединяющий состояние и эффект. `useLocalStorage` ведёт себя как `useState`, но переживает перезагрузку страницы: значение читается из хранилища при первом рендере и записывается туда при каждом изменении.

```jsx
function useLocalStorage(key, initial) {
  const [value, setValue] = useState(() => {
    const saved = localStorage.getItem(key);
    return saved !== null ? JSON.parse(saved) : initial;
  });
  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);
  return [value, setValue];
}
```

Ленивая инициализация из прошлого урока читает хранилище один раз, а не на каждом рендере. Эффект зависит от `key` и `value`, поэтому запись происходит при любом изменении. Наружу хук отдаёт ту же пару `[value, setValue]`, что и `useState`, - вызывающий код почти не замечает разницы: `const [draft, setDraft] = useLocalStorage('draft', '')`. Внутри спрятаны и чтение, и запись, и разбор JSON. Именно так кастомные хуки набирают силу: они склеивают несколько встроенных хуков в одно понятие предметной области, а компонент оперирует этим понятием, не вникая в механику.

## Частые ошибки

**Имя без префикса use.** Назвали хук `getWindowWidth` вместо `useWindowWidth` - и линтер перестал применять к нему правила хуков, а внутри вы зовёте `useState`. Ошибки всплывут неожиданно и не там. Имя хука всегда начинается с `use`.

**Ожидание общего состояния.** Расчёт на то, что два компонента через один хук увидят одно значение, - частое заблуждение. Хук делится логикой, а не данными. Общее состояние - это контекст или внешний менеджер, а не кастомный хук сам по себе.

**Хук ради одного вызова.** Выносить в хук логику, которая используется в единственном месте и не станет проще, - лишняя абстракция. Кастомный хук окупается повторным использованием или тем, что заметно расчищает раздутый компонент, а не самим фактом выноса.

<Detail summary="Кастомные хуки заменили классовую наследственность">

До хуков переиспользование логики с состоянием решалось тяжёлыми приёмами: миксинами, компонентами высшего порядка, рендер-пропсами. Все они оборачивали компонент в дополнительные слои, и дерево обрастало обёртками, которые ничего не рисовали, а лишь пробрасывали данные, - явление прозвали wrapper hell. Кастомные хуки убрали проблему в корне: логика переиспользуется без единой обёртки в дереве, потому что хук - это просто вызов функции внутри компонента, а не новый узел над ним. Именно ради этого хуки и задумывались: дать способ делиться логикой с состоянием, не искажая структуру компонентов. Умение вынести узор в аккуратный хук - один из главных навыков, отличающих зрелый React-код от нагромождения копипасты.
