# Состояние - это снимок, а не переменная

Начинающий разработчик пишет счётчик лайков и хочет, чтобы двойной клик прибавлял сразу два. Логика кажется железной: вызову `setLikes(likes + 1)` два раза подряд - получу плюс два. Он запускает, кликает и видит прибавку на единицу. Добавляет третий вызов `setLikes(likes + 1)` - всё равно плюс один. Переменную `likes` он не трогает, значение в неё не присваивает, а она будто застыла.

За этой загадкой прячется самая важная идея всего React: состояние ведёт себя не как обычная переменная, которую можно менять по ходу дела, а как снимок, зафиксированный на время одного рендера. Пока вы не поймёте, что `likes` внутри одного рендера неизменно, хуки будут казаться капризными. Как только поймёте - счётчик на два лайка, батчинг и функциональные апдейты станут очевидными следствиями одного правила.

## Каждый рендер видит свой likes

Компонент - это функция, которую React вызывает заново на каждую перерисовку. При вызове `useState` возвращает пару: текущее значение из своей ячейки и функцию-сеттер. Ключевое слово - «текущее на момент этого вызова». React подставляет значение в тот момент, когда рендерит компонент, и дальше на протяжении всего рендера это просто константа. Сеттер не меняет её задним числом; он планирует новый рендер, в котором функция компонента выполнится снова и `useState` вернёт уже другое число.

```jsx
function LikeButton() {
  const [likes, setLikes] = useState(0);
  return <button onClick={() => setLikes(likes + 1)}>{likes}</button>;
}
```

На первом рендере `likes` равно 0 и таким останется до конца рендера, что бы вы ни вызвали. Клик планирует обновление, React перезапускает `LikeButton`, и на втором рендере `likes` уже равно 1 - но это новая, отдельная константа в новом вызове функции. Старый рендер про неё ничего не знает.

Отсюда сразу видно ошибку с двойным лайком. Оба вызова `setLikes(likes + 1)` происходят в одном рендере, где `likes` равно 0. Первый планирует «поставить 1», второй тоже планирует «поставить 1», потому что читает всё то же нулевое `likes`. React применяет оба, но оба говорят одно и то же, и результат - единица.

## Один клик - один рендер

Почему оба вызова случились «в одном рендере»? Из-за батчинга. React не перерисовывает компонент после каждого вызова сеттера. Он собирает все обновления, случившиеся внутри одного обработчика события, в одну очередь и применяет их разом, после чего делает ровно один рендер. Три вызова `setLikes` подряд не породят три перерисовки - будет одна.

Батчинг - это не помеха, а оптимизация: без него интерфейс перерисовывался бы лишний раз на каждое мелкое изменение. В React 18 батчинг работает даже для обновлений внутри промисов и таймеров, а не только в обработчиках событий. Практический вывод один: не рассчитывайте, что значение состояния обновится «прямо сейчас», сразу после строки с сеттером. В пределах текущего кода оно останется прежним до следующего рендера.

## Функциональный апдейт читает свежее значение

Как же всё-таки прибавить два? Нужен способ сказать сеттеру «прибавь к тому, что будет в очереди, а не к тому, что я вижу сейчас». Такой способ есть: передайте в сеттер не новое значение, а функцию. React вызовет её и передаст ей самое свежее состояние из очереди, а не снимок вашего рендера.

```jsx
function LikeButton() {
  const [likes, setLikes] = useState(0);
  function addTwo() {
    setLikes((prev) => prev + 1);
    setLikes((prev) => prev + 1);
  }
  return <button onClick={addTwo}>{likes}</button>;
}
```

Теперь первый вызов получает `prev` равным 0 и кладёт в очередь 1. Второй вызов получает `prev` равным уже 1 - результату предыдущего обновления в очереди - и кладёт 2. Итог рендера - двойка. Разница между `setLikes(likes + 1)` и `setLikes((prev) => prev + 1)` в том, откуда берётся исходное число: из замороженного снимка рендера или из живой очереди обновлений.

<Callout type="tip">
Правило простое: если новое значение состояния вычисляется из старого, используйте функциональную форму. Она устойчива к батчингу, к асинхронным вызовам и к устаревшим замыканиям, о которых пойдёт речь в последнем уроке темы.
</Callout>

## Ленивая инициализация

Аргумент `useState` вычисляется на каждом рендере, но используется только на первом. Обычно это неважно: `useState(0)` дёшев. Но если начальное значение получают тяжёлой операцией - например, читают и разбирают строку из `localStorage`, - расчёт впустую повторяется на каждой перерисовке.

```jsx
const [cart, setCart] = useState(() => JSON.parse(localStorage.getItem('cart') ?? '[]'));
```

Передайте функцию вместо значения, и React вызовет её только один раз, при первом монтировании. На всех последующих рендерах функция игнорируется. Не перепутайте с функциональным апдейтом сеттера: там функция получает предыдущее состояние, здесь - не получает ничего и просто возвращает начальное значение.

## Объекты и массивы меняют копией

Пока в состоянии лежит число или строка, всё просто: передал новое значение, React увидел, что оно другое, и перерисовал. С объектами и массивами появляется подвох. React сравнивает старое и новое состояние поверхностно, по ссылке, через `Object.is`. Измените поле объекта на месте - ссылка не изменится, и React решит, что ничего не произошло.

```jsx
function Profile() {
  const [user, setUser] = useState({ name: 'Анна', city: 'Казань' });
  function moveToMoscow() {
    setUser({ ...user, city: 'Москва' });
  }
  return <button onClick={moveToMoscow}>{user.city}</button>;
}
```

`setUser(user)` после `user.city = 'Москва'` не сработал бы: объект тот же, ссылка та же, перерисовки нет. Поэтому состояние обновляют копией: `{ ...user, city: 'Москва' }` создаёт новый объект, копируя старые поля и подменяя одно. Массивы так же - не `push`, а новый массив: `[...items, next]` для добавления, `items.filter(x => x.id !== id)` для удаления, `items.map(...)` для замены элемента. Правило короткое: состояние трактуют как неизменяемое, а сеттеру всегда отдают свежий объект или массив.

## Частые ошибки

**Чтение состояния сразу после сеттера.** Строка `setCount(count + 1); console.log(count);` печатает старое значение, и это не баг. `count` в этом рендере - константа; новое значение появится в переменной только на следующем рендере. Если значение нужно здесь и сейчас, посчитайте его в обычной переменной: `const next = count + 1`.

**Мутация состояния на месте.** Вызвать `cart.push(item)` и затем `setCart(cart)` - частая ловушка. React сравнивает старую и новую ссылку и, увидев тот же массив, может не перерисовать компонент. Всегда создавайте новое значение: `setCart([...cart, item])`.

**Цепочка `setLikes(likes + 1)` вместо функциональной формы.** Как мы видели, несколько таких вызовов в одном событии схлопываются в одно прибавление. Как только обновление зависит от предыдущего, переходите на `setLikes((prev) => prev + 1)`.

<Detail summary="Почему React вообще замораживает состояние">

Заморозка выглядит как каприз, но именно она делает интерфейс предсказуемым. Раз `likes` внутри рендера - константа, то и весь JSX этого рендера согласован: обработчики, тексты, переданные в дочерние компоненты пропсы видят одно и то же значение. Если бы состояние менялось прямо посреди рендера, две части разметки могли бы отрисоваться с разными числами, и на экране появилась бы каша. Модель «рендер - это чистая функция от пропсов и состояния» держится ровно на том, что состояние на время рендера неизменно. Все странности сеттеров - плата за эту предсказуемость, и плата небольшая, стоит только принять снимок как данность.

</Detail>
