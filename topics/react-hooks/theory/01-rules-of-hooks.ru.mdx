# Почему хук нельзя спрятать в if

Вы пишете форму профиля. Появляется задача: показывать поле «отчество» только для граждан России, а для остальных его скрывать. Рука тянется написать очевидное: если страна равна «Россия», заводим состояние через `useState`, иначе не заводим. Компонент даже запускается и рисует форму. А потом пользователь меняет страну в выпадающем списке, и приложение падает с загадочной ошибкой про «порядок хуков». Ничего не предвещало беды: обычный `if`, обычный `useState`.

Разгадка в том, что хуки только выглядят как обычные вызовы функций. На самом деле они общаются с React через скрытый механизм, у которого есть одно жёсткое требование к порядку. Нарушить его легко и незаметно, поэтому React вынес требование в два коротких правила и приложил линтер, который следит за ними. Разберёмся, что это за механизм, почему `if` вокруг хука его ломает и как жить с ограничением, не чувствуя его как клетку.

## Хуки лежат в списке, а не в переменных

React не знает имён ваших переменных. Когда вы пишете `const [name, setName] = useState('')`, React не видит слова `name`. Он видит только сам факт: «на этом рендере вызвали `useState` первый раз, выдай первую ячейку состояния». Второй `useState` получит вторую ячейку, третий - третью. React хранит ячейки в обычном списке, привязанном к экземпляру компонента, и на каждом рендере проходит этот список строго по порядку вызовов.

Пока набор вызовов от рендера к рендеру одинаковый, всё сходится: первая переменная всегда получает первую ячейку, вторая - вторую. Список - это и есть память компонента между перерисовками. Именно поэтому `useState` умеет «помнить» значение: он не хранит его в вашей переменной, которая умирает вместе с вызовом функции, а достаёт из своей ячейки по номеру.

Теперь понятно, почему `if` вокруг хука опасен. Стоит на одном рендере пропустить вызов, и вся нумерация после него съезжает на единицу.

```jsx
function Profile({ country }) {
  const [name, setName] = useState('');
  if (country === 'Россия') {
    const [patronymic, setPatronymic] = useState('');
  }
  const [age, setAge] = useState(0);
}
```

На рендере со страной «Россия» вызовов три, и `age` берёт третью ячейку. Пользователь меняет страну, `if` не срабатывает, вызовов остаётся два, и `age` внезапно забирает вторую ячейку - ту, где раньше жило отчество. React видит, что список изменил длину и форму, и останавливается с ошибкой, потому что дальше он уже не может гарантировать, какое состояние кому принадлежит.

## Два правила и их причина

Из устройства списка вытекают ровно два правила, которые и называют правилами хуков.

Первое: вызывайте хуки только на верхнем уровне тела компонента или другого хука. Никаких хуков внутри `if`, циклов, `try` или вложенных функций. Это гарантирует, что на каждом рендере хуки вызываются в одном и том же порядке и в одинаковом количестве. Верхний уровень - единственное место, где порядок вызовов не зависит от данных.

Второе: вызывайте хуки только из двух мест - из функциональных компонентов React и из других хуков. Из обычной функции хук звать нельзя, потому что вне рендера у React нет того самого списка ячеек, к которому можно обратиться. Компонент - это функция, которую вызывает сам React в контексте рендера; кастомный хук наследует этот контекст, потому что его вызывают из компонента.

<Callout type="tip">
Имя хука обязано начинаться с `use`. Это не украшение: по префиксу линтер отличает хук от обычной функции и применяет к нему правила. Назовёте свой хук `getUser` вместо `useUser` - линтер перестанет вас страховать.
</Callout>

## Условие снаружи, а не внутри

Ограничение звучит строго, но на практике оно почти не мешает, потому что условную логику всегда можно вынести за пределы хука. Хук вызывается безусловно и всегда, а вот условие переезжает внутрь его аргументов или в разметку.

Вернёмся к отчеству. Состояние заводим всегда, а прячем поле уже в JSX по условию:

```jsx
function Profile({ country }) {
  const [name, setName] = useState('');
  const [patronymic, setPatronymic] = useState('');
  return (
    <form>
      <input value={name} onChange={(e) => setName(e.target.value)} />
      {country === 'Россия' && (
        <input value={patronymic} onChange={(e) => setPatronymic(e.target.value)} />
      )}
    </form>
  );
}
```

Порядок вызовов теперь неизменен: два `useState` на каждом рендере при любой стране. Лишняя ячейка состояния для отчества, которое сейчас скрыто, ничего не стоит. Тот же приём работает и с эффектами: не оборачивайте `useEffect` в `if`, а поместите условие внутрь колбэка - эффект вызовется всегда, а тело отработает, только когда нужно.

## Список, которому нужны свои хуки

Правило «никаких хуков в цикле» рано или поздно упирается в живую задачу: у каждого элемента списка своё состояние - раскрыта ли строка, редактируется ли ячейка. Вызвать `useState` внутри `map` нельзя, число вызовов будет скакать вместе с длиной массива. Выход всегда один и тот же: единицу повторения, которой нужно состояние, выделяют в отдельный компонент.

```jsx
function TodoList({ todos }) {
  return (
    <ul>
      {todos.map((todo) => (
        <TodoItem key={todo.id} todo={todo} />
      ))}
    </ul>
  );
}

function TodoItem({ todo }) {
  const [expanded, setExpanded] = useState(false);
  return (
    <li onClick={() => setExpanded(!expanded)}>{expanded ? todo.fullText : todo.title}</li>
  );
}
```

`TodoList` перебирает массив и не заводит ни одного хука на элемент. Всё состояние переехало в `TodoItem`, где оно снова на верхнем уровне: сколько бы строк ни было в списке, каждый `TodoItem` вызывает `useState` ровно один раз за свой рендер. Это общий рецепт на все случаи «хук нужен в цикле» - не гнуть правило, а выделить повторяющуюся единицу в компонент. Заодно код становится чище: логика одной строки собрана в одном месте, а не размазана по родителю.

## Частые ошибки

**Ранний `return` перед хуком.** Классика: `if (!user) return null;` стоит выше, чем `useState`. При отсутствии пользователя часть хуков пропускается, и на следующем рендере порядок съезжает. Лечится так же, как `if`: сначала все хуки, а `return null` уже после них.

**Хук в колбэке или обработчике.** Соблазн написать `onClick={() => useState(...)}` возникает редко, но `useEffect` внутри `map` по списку - ошибка частая. Внутри цикла количество вызовов зависит от длины массива, а значит меняется. Если состояние нужно каждому элементу списка, вынесите элемент в отдельный компонент - там его хуки снова окажутся на верхнем уровне.

**Хук из обычной вспомогательной функции.** Вы завели `function loadData()` и позвали в ней `useState`. React вызывает `loadData` не в контексте рендера, ячеек нет, вызов падает. Либо переименуйте функцию в хук с префиксом `use` и зовите её из компонента, либо уберите хук из неё и передавайте данные аргументом.

<Detail summary="А если очень надо условно, придумайте компонент">

Иногда кажется, что без условного хука никак: например, подписка на данные нужна только для авторизованного пользователя. Решение - не прятать хук, а прятать компонент. Вынесите логику в отдельный компонент `AuthedPanel`, который безусловно вызывает свои хуки, и рендерите сам компонент по условию: `{user && <AuthedPanel user={user} />}`. Когда `user` пуст, React не монтирует `AuthedPanel` вовсе, и его хуки просто не участвуют в игре - но внутри самого компонента порядок вызовов всегда неизменен. Условие переехало на уровень выше хуков, туда, где ему и место.

</Detail>

Правила хуков выглядят как произвольный запрет, пока не увидишь список ячеек за ними. Как только картинка сложилась, оба правила читаются как одно: не делай ничего, что меняет порядок или число вызовов хуков между рендерами. Всё остальное - следствия, и линтер `eslint-plugin-react-hooks` подскажет, если вы случайно их нарушите.
