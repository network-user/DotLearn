# useContext против бесконечного проброса пропсов

Тема оформления - светлая или тёмная - нужна кнопке глубоко внутри карточки товара. Кнопка сидит в карточке, карточка в списке, список на странице, страница в приложении. Чтобы значение темы добралось до кнопки, приходится продеть его через каждый слой: страница передаёт тему списку, список - карточке, карточка - кнопке. Промежуточным компонентам тема не нужна, они лишь передают её дальше, как ведро на пожаре. Добавится новый уровень - придётся вставлять проброс и в него.

Это и есть prop drilling, проброс пропсов сквозь дерево. Контекст его убирает: он позволяет объявить значение где-то наверху и прочитать его в любом месте поддерева напрямую, минуя промежуточные слои. Разберём, из чего состоит контекст, как `useContext` находит нужное значение и где у этого механизма острые углы.

## Три части одного механизма

Контекст - это связка из трёх вещей. Сначала вы создаёте объект контекста с начальным значением по умолчанию. Затем оборачиваете часть дерева в его `Provider` и передаёте туда актуальное значение. Наконец, любой компонент внутри читает это значение хуком `useContext`.

```jsx
const ThemeContext = createContext('светлая');

function App() {
  const [theme, setTheme] = useState('тёмная');
  return (
    <ThemeContext.Provider value={theme}>
      <ProductList />
    </ThemeContext.Provider>
  );
}

function ThemedButton() {
  const theme = useContext(ThemeContext);
  return <button className={theme}>Купить</button>;
}
```

`ThemedButton` может лежать сколь угодно глубоко внутри `ProductList` - он получит тему напрямую, без единого пропса на промежуточных уровнях. `useContext(ThemeContext)` поднимается вверх по дереву и берёт значение из ближайшего `Provider` этого контекста. Если по пути наверх ни одного `Provider` не нашлось, возвращается значение по умолчанию, переданное в `createContext`.

## Потребители перерисовываются вместе со значением

Контекст - не просто способ передать данные, но и канал обновления. Когда значение в `value` у `Provider` меняется, React перерисовывает все компоненты, которые читают этот контекст через `useContext`, - независимо от того, как глубоко они сидят. Промежуточные компоненты, не читающие контекст, при этом не трогаются. Именно поэтому переключение темы в одном месте мгновенно перекрашивает все кнопки: каждая из них - потребитель, и каждая получает новое значение.

Из этого вытекает и главная осторожность. Если в `value` передавать заново создаваемый объект, все потребители будут перерисовываться на каждом рендере провайдера, даже когда данные фактически не поменялись.

```jsx
<ThemeContext.Provider value={{ theme, setTheme }}>
```

Объект в фигурных скобках рождается новым на каждом рендере `App`, его ссылка всегда другая, и React считает значение изменившимся. Лечится это мемоизацией из прошлого урока: заверните объект в `useMemo` с правильными зависимостями, и ссылка станет стабильной.

<Callout type="warning">
Передавать в `value` объект, собранный прямо в разметке, - частая причина лишних перерисовок всего поддерева. Если значение контекста составное, мемоизируйте его через `useMemo`.
</Callout>

## Контекст любит компанию кастомного хука

Читать контекст напрямую через `useContext` работает, но в реальных проектах его почти всегда заворачивают в кастомный хук. Вместо того чтобы в каждом компоненте импортировать и объект контекста, и `useContext`, объявляют один хук `useTheme` и зовут его.

```jsx
function useTheme() {
  return useContext(ThemeContext);
}
```

Теперь компонент пишет просто `const theme = useTheme()`. Плюс не только в краткости: внутри хука удобно добавить проверку, что компонент находится под нужным провайдером, и бросить понятную ошибку, если нет. О том, как устроены кастомные хуки, - следующий урок; пока достаточно знать, что связка «контекст плюс хук» - стандартный способ раздавать общие данные: тему, текущего пользователя, язык интерфейса.

## Провайдер, хук и явная ошибка

Соберём разрозненные части в один аккуратный модуль - так контекст оформляют в реальных проектах. Провайдер держит состояние и мемоизирует значение, а кастомный хук читает контекст и заодно проверяет, что компонент действительно находится под провайдером.

```jsx
const ThemeContext = createContext(null);

function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('светлая');
  const value = useMemo(() => ({ theme, setTheme }), [theme]);
  return <ThemeContext.Provider value={value}>{children}</ThemeContext.Provider>;
}

function useTheme() {
  const value = useContext(ThemeContext);
  if (value === null) {
    throw new Error('useTheme должен вызываться внутри ThemeProvider');
  }
  return value;
}
```

Компоненту снаружи видны только `ThemeProvider` и `useTheme` - о самом объекте контекста он не знает. `useMemo` не даёт объекту `value` рождаться заново на каждом рендере провайдера, так что потребители не дёргаются зря. А проверка на `null` превращает забытый провайдер из невнятного падения где-то в глубине в понятную ошибку с точным адресом. Эта тройка - провайдер, мемоизированное значение и хук с проверкой - стандартный шаблон, по которому раздают тему, текущего пользователя и язык интерфейса.

## Частые ошибки

**Контекст для всего.** Соблазн сложить в один глобальный контекст всё состояние приложения велик, но вреден: любое изменение перерисовывает всех потребителей, а их становится много. Контекст хорош для редко меняющихся общих данных - темы, локали, авторизованного пользователя. Часто меняющееся состояние формы туда класть не стоит.

**Забытый Provider.** Компонент читает контекст, но выше по дереву нет провайдера, и `useContext` молча возвращает значение по умолчанию. Если по умолчанию стоит `null`, дальше всё падает с невнятной ошибкой. Кастомный хук с явной проверкой ловит это сразу.

**Новый объект в value.** Разобрали выше: несмемоизированный объект в `value` обнуляет всю пользу и превращает контекст в генератор лишних перерисовок.

<Detail summary="Контекст - это не менеджер состояния">

Контекст часто противопоставляют библиотекам вроде Redux или Zustand, но это разные инструменты. Контекст только передаёт значение вниз по дереву и перерисовывает потребителей при его смене - он не хранит состояние, не оптимизирует подписки, не умеет обновлять только тех, кому нужна конкретная часть данных. Само состояние по-прежнему живёт в `useState` или `useReducer` у провайдера, а контекст лишь доставляет его. Для небольшого общего состояния - темы, пользователя - этого достаточно. Для крупного, часто меняющегося состояния с точечными подписками берут специализированный менеджер, потому что контекст перерисует всех потребителей на любое изменение. Понимание этой границы избавляет от попыток построить на голом контексте то, для чего он не создан.

</Detail>
