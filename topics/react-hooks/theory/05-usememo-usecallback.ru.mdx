# Мемоизация: кэш между рендерами и когда он нужен

Компонент - это функция, которую React вызывает заново на каждую перерисовку, а перерисовок бывает много: изменилось состояние, пришёл новый пропс, обновился родитель. Каждый раз тело компонента выполняется целиком: снова считается отфильтрованный список, снова создаётся объект настроек, снова рождается функция-обработчик. Обычно это дёшево и незаметно. Но иногда вычисление тяжёлое, а иногда важна не стоимость, а сам факт: новый объект или функция на каждом рендере ломает оптимизацию где-то ниже по дереву.

Для этих двух случаев есть `useMemo` и `useCallback`. Их репутация двойственная: одни обмазывают ими каждую переменную «на всякий случай», другие боятся и не используют вовсе. Истина посередине, и она про понимание, что именно кэшируется и зачем. Разберёмся с механизмом, а потом честно очертим границу, за которой мемоизация приносит больше вреда, чем пользы.

## useMemo помнит результат

`useMemo(factory, deps)` вычисляет `factory()` на первом рендере, запоминает результат и на следующих рендерах отдаёт его же - пока значения в массиве зависимостей не изменятся. Как только хоть одна зависимость стала другой, React вызывает `factory` заново и запоминает свежий результат.

```jsx
function Report({ rows, currency }) {
  const total = useMemo(() => rows.reduce((sum, r) => sum + r.amount, 0), [rows]);
  return <p>Итого: {total} {currency}</p>;
}
```

Сумма пересчитывается только при смене `rows`. Перерисовался компонент из-за нового `currency` - `total` берётся из кэша, `reduce` не бежит. Массив зависимостей здесь читается точно так же, как у эффекта: перечислите всё реактивное, что использует вычисление.

## useCallback помнит функцию

`useCallback` - это тот же `useMemo`, только он кэширует не результат вызова, а саму функцию. Запись `useCallback(fn, deps)` в точности равна `useMemo(() => fn, deps)`. Зачем кэшировать функцию? Из-за ссылочного равенства. Функция, объявленная в теле компонента, - это каждый раз новый объект: `onClick` с прошлого рендера и `onClick` с этого не равны друг другу, даже если код внутри одинаковый.

```jsx
function Toolbar({ userId }) {
  const handleSave = useCallback(() => save(userId), [userId]);
  return <SaveButton onSave={handleSave} />;
}
```

Пока `userId` не меняется, `handleSave` остаётся тем же самым объектом-функцией от рендера к рендеру. Это важно ровно в двух ситуациях. Первая: `SaveButton` обёрнут в `React.memo` и перерисовывается, только когда его пропсы изменились по ссылке; нестабильный `onSave` сводил бы эту оптимизацию на нет. Вторая: функция попадает в массив зависимостей эффекта - нестабильная ссылка перезапускала бы эффект на каждый рендер.

<Callout type="info">
`useMemo` кэширует значение, `useCallback` - функцию. Обе фиксируют результат ровно до тех пор, пока не изменятся зависимости. Разницы в механизме нет, только в том, что кладётся в кэш.
</Callout>

## Когда мемоизация действительно нужна

Мемоизация не бесплатна. React хранит прошлые зависимости и результат, на каждом рендере сравнивает массивы, а вы платите за это лишним кодом, который надо читать и поддерживать. Поэтому по умолчанию не мемоизируйте ничего. Оборачивайте в `useMemo` или `useCallback`, только когда есть конкретная причина.

Причин ровно три. Первая: вычисление заметно дорогое - сортировка тысяч строк, тяжёлый разбор, - и профайлер подтверждает, что оно тормозит. Вторая: значение или функция передаётся в дочерний компонент, обёрнутый в `React.memo`, и вы хотите сохранить его от лишних перерисовок. Третья: значение используется как зависимость другого хука, и нужна стабильная ссылка, чтобы эффект или мемо не срабатывали впустую.

Если ни одна причина не подходит, мемоизация - это шум. Кэшировать `useMemo(() => a + b, [a, b])` бессмысленно: сложение дешевле, чем сравнение массива зависимостей вокруг него.

## Ссылочное равенство на живом примере

Чтобы почувствовать, зачем нужна стабильная ссылка, посмотрим на `React.memo`. Обёрнутый в него компонент перерисовывается, только когда его пропсы изменились по ссылке. Проблема в том, что функция, объявленная в родителе, - каждый раз новая.

```jsx
const Row = React.memo(function Row({ onSelect }) {
  return <button onClick={onSelect}>выбрать</button>;
});

function List() {
  const [count, setCount] = useState(0);
  const handleSelect = () => console.log('выбрано');
  return <div><Row onSelect={handleSelect} /><button onClick={() => setCount(count + 1)}>{count}</button></div>;
}
```

Каждый клик по счётчику перерисовывает `List`, а вместе с ним рождается новая `handleSelect`. Её ссылка отличается от прошлой, `React.memo` считает пропс изменившимся и перерисовывает `Row` зря - оптимизация не работает. Оберните `handleSelect` в `useCallback(() => console.log('выбрано'), [])`, и ссылка станет стабильной: `Row` перестанет перерисовываться на каждый чужой клик. Вот та самая вторая причина из списка выше, увиденная вживую. Без `React.memo` на `Row` весь этот `useCallback` был бы пустой формальностью - стабильность ссылки некому оценить.

## Частые ошибки

**Мемоизация всего подряд.** Обёртка каждой переменной в `useMemo` не ускоряет, а замедляет: добавляются сравнения и память под кэш, а выигрыша нет, потому что вычисления и так дешёвые. Код становится труднее читать без единой причины.

**Неполный массив зависимостей.** Если вычисление читает `rows` и `taxRate`, а в зависимостях только `[rows]`, кэш вернёт устаревшую сумму после смены ставки. Правило то же, что у эффекта: в массив попадает всё реактивное, что вычисление использует.

**useCallback там, где ссылка никому не важна.** Оборачивать обработчик, который передаётся в обычный `<button>`, бессмысленно: `<button>` не мемоизирован, стабильность ссылки ему безразлична. Польза появляется только рядом с `React.memo` или в зависимостях хука.

<Detail summary="React Compiler и будущее ручной мемоизации">

Ручная мемоизация - во многом починка того, что в идеале должен делать инструмент. React Compiler, выходящий из экспериментальной стадии, автоматически расставляет мемоизацию на этапе сборки, анализируя, какие значения от каких зависят. В проекте с включённым компилятором большая часть `useMemo` и `useCallback` становится ненужной: компилятор сам понимает, что пересчитывать, а что переиспользовать. Но понимать механизм всё равно нужно: чтобы читать существующий код, отлаживать оптимизации и работать там, где компилятор ещё не подключён. Ручная мемоизация никуда не денется в одночасье, а модель «кэш по зависимостям» под капотом остаётся той же.

</Detail>
