# useEffect синхронизирует, а не реагирует

Компонент чата должен подключаться к серверу при открытии комнаты и отключаться при выходе. Первый порыв - написать подключение прямо в теле компонента: вот код рендера, вот вызов `connect()`. Но тело компонента React выполняет когда угодно и сколько угодно раз, иногда просто чтобы прикинуть разметку и выбросить её. Подключаться на каждый такой прогон - значит открыть десяток сокетов там, где нужен один. Нужен способ сказать: «выполни это не во время рендера, а после того, как результат уже на экране, и не забудь прибраться, когда придёт время».

Этим и занимается `useEffect`. Его вечно понимают как «код, который реагирует на изменения», и оттого пишут туда логику, которой там не место. Правильная картинка другая: эффект синхронизирует компонент с чем-то внешним - сервером, таймером, подпиской, DOM-узлом. Разберём, когда эффект срабатывает, как массив зависимостей этим управляет и зачем эффекту функция очистки.

## Эффект живёт после рендера

React делит работу на две фазы. Сначала рендер: компонент-функция выполняется и возвращает разметку, React вычисляет, что показать. Затем коммит: изменения попадают в реальный DOM, браузер рисует пиксели. Эффекты запускаются после коммита, когда пользователь уже видит результат. Поэтому в эффекте безопасно трогать DOM, слать запросы и открывать соединения: экран согласован, ничего не отрисуется наполовину.

Отсюда практическое правило: то, что можно посчитать прямо во время рендера, в эффект не выносят. Отфильтрованный список, форматированная дата, сумма корзины - это обычные переменные в теле компонента. Эффект нужен там, где вы выходите за пределы React: подписка на внешнее событие, ручная работа с DOM, сетевой запрос, таймер.

## Массив зависимостей - это список синхронизаций

Второй аргумент `useEffect` определяет, на каких рендерах эффект нужно запускать заново. React сравнивает элементы массива с их значениями на прошлом рендере и, если хоть один изменился, выполняет эффект снова. Есть три случая, и их стоит держать в голове по отдельности.

```jsx
useEffect(() => {
  console.log('каждый рендер');
});

useEffect(() => {
  console.log('только при монтировании');
}, []);

useEffect(() => {
  console.log('когда меняется roomId');
}, [roomId]);
```

Без второго аргумента эффект бежит после каждого рендера. С пустым массивом - один раз, при монтировании, и больше никогда, потому что сравнивать нечего и «изменений» не бывает. С массивом `[roomId]` эффект срабатывает при монтировании и затем каждый раз, когда `roomId` стал другим. Сравнение поверхностное, через `Object.is`: два разных массива или объекта с одинаковым содержимым React считает разными, и это частый источник лишних запусков.

<Callout type="warning">
Массив зависимостей - не список «того, что хочется отслеживать», а список всех реактивных значений, которые эффект читает: пропсы, состояние, производные от них переменные. Соврать в этом списке - пропустить зависимость - значит получить эффект, который смотрит на устаревшие данные. Об этой ловушке подробно в последнем уроке темы.
</Callout>

## Очистка закрывает то, что эффект открыл

Эффект, который подписывается, обязан уметь отписаться. Для этого он возвращает функцию очистки. React вызывает её в двух моментах: перед тем как запустить эффект заново, и когда компонент размонтируется. Симметрия здесь буквальная: сколько раз эффект подключился, столько раз очистка отключит.

```jsx
function ChatRoom({ roomId }) {
  useEffect(() => {
    const connection = connect(roomId);
    return () => connection.disconnect();
  }, [roomId]);
  return <h1>Комната {roomId}</h1>;
}
```

Проследим, что происходит, когда пользователь переходит из комнаты «general» в «random». При монтировании эффект подключается к «general». Меняется `roomId` - и React сначала вызывает очистку старого эффекта, которая отключает «general», и только потом запускает новый эффект, подключающий «random». При выходе со страницы очистка отключает «random». Ни одного висящего соединения.

Правило-мнемоника: если эффект что-то начал - подписку, таймер, соединение, слушатель события, - он обязан это завершить в очистке.

## Гонка запросов и флаг отмены

Очистка нужна не только соединениям и таймерам. Сетевой запрос в эффекте порождает коварную гонку. Пользователь быстро переключает вкладки: эффект стартует запрос для вкладки A, затем для вкладки B. Ответы приходят по сети в любом порядке. Если ответ A придёт последним, он перезапишет данные B, и на экране окажется содержимое вкладки, которую уже закрыли.

Лечит это та же функция очистки. Заведите локальный флаг и в очистке пометьте эффект как устаревший; результат обрабатывайте, только если флаг не взведён.

```jsx
useEffect(() => {
  let ignore = false;
  fetchUser(userId).then((data) => {
    if (!ignore) {
      setUser(data);
    }
  });
  return () => {
    ignore = true;
  };
}, [userId]);
```

Когда `userId` меняется, React вызывает очистку прошлого эффекта, та ставит `ignore = true`, и запоздавший ответ старого запроса тихо игнорируется. Переменная `ignore` у каждого запуска эффекта своя - это обычное замыкание, и здесь оно работает на вас. Запрос при этом не отменяется по-настоящему (для этого есть `AbortController`), но его результат больше не портит состояние, а это главное.

## Частые ошибки

**Забытая очистка.** `setInterval` без `clearInterval`, `addEventListener` без `removeEventListener`. Симптом - таймеры и обработчики множатся с каждым рендером. Лечится возвратом функции очистки, которая отменяет ровно то, что эффект создал.

**Пустой массив зависимостей «чтобы не мешал».** Соблазн поставить `[]`, когда линтер ругается на пропущенную зависимость, велик и опасен. Эффект перестанет ругаться, но и перестанет реагировать на изменения: подключится к первой комнате и застрянет в ней навсегда. Пустой массив означает «зависимостей действительно нет», а не «замолчи».

**Логика рендера в эффекте.** Пересчёт производных данных через `useEffect` с записью в отдельное состояние - лишний рендер и рассинхрон. Если значение выводится из пропсов и состояния, считайте его прямо в теле компонента, без эффекта.

<Detail summary="Почему эффект не стоит воспринимать как «onChange»">

Модель «эффект реагирует на изменение переменной» подталкивает к антипаттернам: связать два состояния через эффект, синхронизировать пропс в состояние, дёрнуть колбэк родителя при смене значения. Всё это чаще решается без эффекта - вычислением во время рендера или обработчиком события, который и так знает, что произошло. Официальная документация советует спрашивать себя: синхронизирую ли я компонент с внешней системой? Если да - эффект уместен. Если я просто хочу «сделать что-то, когда значение изменилось» в ответ на действие пользователя, этому место в обработчике события, а не в эффекте. Эффектов в здоровом компоненте обычно мало, и почти все они про подписки, таймеры и сеть.

</Detail>
