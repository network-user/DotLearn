# Устаревшее замыкание - главный баг новичка

Счётчик, который сам тикает раз в секунду. Кажется, дел на минуту: заводим состояние, в эффекте ставим `setInterval`, внутри увеличиваем счётчик. Массив зависимостей делаем пустым - интервал ведь нужно поставить один раз при монтировании. Запускаем: число доходит до единицы и застывает. Секунды идут, интервал явно работает, а на экране упрямая единица. Ни ошибки, ни предупреждения - просто счётчик, который считать не хочет.

Это самый частый и самый поучительный баг в React, и корень у него один - устаревшее замыкание. Если урок про замыкания в JavaScript вы уже прошли, половина дела сделана: механизм тот же, просто он встречается с моделью снимков React и даёт неожиданный на первый взгляд результат. Разберём, почему счётчик застывает, как связаны замыкания и массив зависимостей и какими тремя способами это чинится.

## Каждый рендер замыкает своё значение

Вспомним две вещи из предыдущих уроков. Первая: значение состояния - снимок, замороженный на время одного рендера. Вторая: функция уносит с собой окружение, в котором родилась, - это и есть замыкание. Соедините их. Каждый рендер компонента заново создаёт все функции в его теле, и каждая такая функция замыкает переменные именно этого рендера. Обработчик, рождённый на рендере, где `count` равен 3, навсегда помнит тройку, потому что для него `count` - константа из его окружения.

Обычно это незаметно и даже полезно: обработчик клика видит согласованный снимок, всё сходится. Проблема возникает, когда функция переживает свой рендер и вызывается позже, когда состояние уже другое.

```jsx
function Ticker() {
  const [count, setCount] = useState(0);
  useEffect(() => {
    const id = setInterval(() => {
      setCount(count + 1);
    }, 1000);
    return () => clearInterval(id);
  }, []);
  return <h1>{count}</h1>;
}
```

Эффект с `[]` выполняется один раз, при монтировании, в окружении первого рендера, где `count` равен 0. Функция внутри `setInterval` замкнула этот ноль. Каждую секунду она исполняет `setCount(0 + 1)` - всегда `setCount(1)`, потому что её `count` навсегда 0. Состояние становится единицей и больше не растёт: интервал жив, но смотрит в прошлое.

## Массив зависимостей - это обещание честности

Тут и вскрывается настоящая роль массива зависимостей. Это не «список, при смене которого перезапустить эффект» в вакууме - это ваш договор с React: «эффект использует ровно эти реактивные значения и никакие другие». Пустой массив - обещание, что эффект не читает ни пропсов, ни состояния. Наш эффект это обещание нарушил: он читает `count`, но в списке его нет. React поверил на слово, не стал пересоздавать интервал, и замыкание застряло.

Линтер `eslint-plugin-react-hooks` с правилом `exhaustive-deps` именно за этим и следит: он видит, что эффект читает `count`, а в зависимостях его нет, и предупреждает. Соблазн заглушить предупреждение пустым массивом - и есть источник почти всех устаревших замыканий.

<Callout type="warning">
Пустой массив зависимостей означает «эффект действительно ничего реактивного не читает», а не «отстань, линтер». Если эффект читает состояние или пропс, они обязаны быть в списке - или логику надо переписать так, чтобы чтения не было.
</Callout>

## Три способа починить

Первый способ - честный массив зависимостей. Добавьте `count` в список. Теперь при каждом изменении счётчика эффект перезапускается: старый интервал очищается, новый ставится уже в свежем окружении, где `count` актуален. Работает, но интервал пересоздаётся каждую секунду - для таймера это лишнее.

Второй способ, для этого случая лучший, - функциональный апдейт. Уберите чтение `count` вовсе.

```jsx
useEffect(() => {
  const id = setInterval(() => {
    setCount((prev) => prev + 1);
  }, 1000);
  return () => clearInterval(id);
}, []);
```

`setCount((prev) => prev + 1)` не читает `count` из окружения - свежее значение ему приносит React из очереди. Замыкать больше нечего, обещание пустого массива снова честное, а интервал ставится один раз. Правило из урока про `useState` окупается: как только новое значение выводится из старого, переходите на функциональную форму.

Третий способ - `ref` для случаев, когда нужно именно прочитать последнее значение, а не вывести его из предыдущего. Держите свежее значение в `ref.current`, обновляйте его в отдельном эффекте, а в долгоживущей функции читайте `ref.current` - он не замораживается и всегда отдаёт актуальное. Приём выручает с обработчиками, которые нельзя пересоздавать, но применять его стоит с осторожностью: он выводит значение из-под контроля модели снимков.

## Объект в зависимостях и бесконечный цикл

У устаревшего замыкания есть зеркальный близнец - эффект, который перезапускается без остановки. Причина та же ссылочная природа, что и в уроке про мемоизацию.

```jsx
function Search({ query }) {
  const options = { query, limit: 20 };
  useEffect(() => {
    fetchResults(options);
  }, [options]);
}
```

Кажется логичным: эффект читает `options`, значит, `options` в зависимостях. Но `options` - объект, который рождается заново на каждом рендере. Его ссылка всегда новая, React на каждом рендере считает зависимость изменившейся, запускает эффект, эффект вызывает `setState` где-то внутри `fetchResults`, происходит новый рендер, рождается новый `options` - и так по кругу. Линтер здесь не спасёт: формально зависимость указана верно.

Лечится тем же, что и лишние перерисовки. Проще всего зависеть от примитивных полей, а объект собирать внутри эффекта: массив зависимостей `[query]` сравнивает строку по значению, и цикл рвётся. Если объект действительно нужен снаружи целиком, оберните его в `useMemo` с примитивными зависимостями - тогда ссылка станет стабильной. Общее правило: в массив зависимостей стремятся класть примитивы, а не свежесозданные объекты и функции.

## Частые ошибки

**Пустой массив как заглушка линтера.** Разобрали подробно: `[]` при чтении состояния замораживает эффект на начальных значениях. Лечится честными зависимостями, функциональной формой или ref.

**Объект или функция в зависимостях без мемоизации.** Если эффект зависит от `options` - объекта, собранного в теле компонента, - его ссылка новая на каждом рендере, и эффект перезапускается всегда, будто в бесконечном цикле. Либо мемоизируйте объект через `useMemo`, либо перенесите его создание внутрь эффекта, либо зависьте от примитивных полей.

**Чтение свежего состояния из старого обработчика.** Обработчик, сохранённый один раз и вызываемый позже, видит состояние того рендера, где родился. Если нужен доступ к последнему значению, ведите его через `ref` или пересоздавайте обработчик при смене зависимостей.

<Detail summary="Почему React не «просто берёт свежее значение»">

Возникает законный вопрос: зачем эти сложности, почему React не сделает так, чтобы функции всегда видели последнее состояние? Потому что тогда рухнула бы модель снимков, на которой держится вся предсказуемость. Если бы `count` внутри обработчика мог измениться между строками, две части одного рендера могли бы увидеть разные значения, и рассуждать о коде стало бы невозможно. Заморозка `count` на время рендера - не баг, а сознательное решение: она делает рендер чистой функцией от пропсов и состояния. Устаревшее замыкание - обратная сторона той же медали, и правильный ответ на него не «отменить заморозку», а честно объявить зависимости или не читать замороженное значение вовсе. Как только принимаешь снимок как основу, все три способа починки перестают быть трюками и становятся очевидными следствиями.

</Detail>
