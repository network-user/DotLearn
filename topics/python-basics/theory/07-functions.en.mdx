# You've been writing functions since lesson one

Time to let out a secret that's been running through the whole course. In every exercise you got a stub like `def taxi_price(km):` and filled in lines underneath, below that first line with the mysterious word `def`. Touching that line was off-limits, nobody ever explained it, and it obediently worked anyway. Confession: this whole time you've been writing functions. Every exercise you solved was already a function, you were simply filling in a ready-made body while the header patiently waited its turn.

Today the stubs run out. You'll take the header apart piece by piece, learn to declare a function in full, from `def` to `return`, and untangle the biggest beginner confusion of all: how returning a value differs from printing it to the screen. This is the final lesson, so its last task pulls together conditions, lists, loops, and arithmetic from all six lessons before it.

## Why give logic a name

Picture calculating a taxi fare: a 100-ruble base fee plus 25 rubles per kilometer. Computing it once is easy. But a program needs that calculation again and again: for Anna's ride, for Boris's ride, for the monthly report. Copying the same lines everywhere is a bad idea: find a bug in the calculation, and you'll have to fix every copy, and you're guaranteed to miss one.

A function solves the problem differently. You describe the steps once and give them a name, then call that calculation by name as many times as you like, with different numbers. A everyday analogy fits: a coffee machine. You drop in coins and press a button, that's the input. Inside, the machine whirs through the same routine every time, that's the body. A cup rolls out, that's the result. Nobody takes the machine apart again for a second coffee: pressing the button once more is enough.

## Anatomy of a declaration: from def to return

Here's a complete function, together with a call:

```python
def greet(name):
    text = "Привет, " + name + "!"
    return text

message = greet("Анна")
print(message)
```

The first line is called the header. `def` tells Python: "a function declaration follows." Next comes the name `greet`, one you pick yourself, following the same rules as variable names. Parentheses hold the parameters, one here, `name`. A colon closes the header, and everything indented below it is the function's body: the steps that run on each call.

<MarginNote label="Naming">
Function names use lowercase with underscores: `taxi_price`, `send_sms`. A good name answers the question "what does this do".
</MarginNote>

An important detail: the `def` line itself computes nothing. Python reads the declaration, remembers the function under its name, and moves on. The body wakes up only on a call: the name plus parentheses with arguments, `greet("Анна")`.

Two similar-sounding words are easy to mix up here, so let's pin them down. A parameter is the name in the declaration's parentheses: `name`. An argument is the actual value passed at the call: `"Анна"`. Parameters get declared once, arguments change from call to call:

```python
print(greet("Борис"))
print(greet("Вера"))
```

One function, two calls, two different greetings. That's the entire point of functions.

## return hands back a value and exits immediately

`return` does two jobs. First: hand a value back to the call site. Second, less obvious: end the function right away. The moment `return` fires, the function is done, and the rest of its body never runs.

The second job shows itself clearly in branches, familiar from the third lesson:

```python
def transport(rubles):
    if rubles < 300:
        return "автобус"
    return "такси"
```

With less than 300 rubles, the first `return` fires, and the last line never gets a chance. When the condition is false, that branch gets skipped, and the function runs straight into the second `return`. This is called an early exit, and several `return` statements in different branches is a normal, readable style: each branch is responsible for its own result.

## print shows a person, return hands off to the program

Now for the lesson's main trap. Nearly every beginner stumbles on this, so let's go slowly. Here are two functions, both "adding" numbers:

```python
def add_print(a, b):
    print(a + b)

def add_return(a, b):
    return a + b
```

Call each with the arguments 2 and 3, and at first glance there's no difference: a 5 shows up on screen either way. But the difference is fundamental. `print` shows a value to a human looking at the console, and that's it: the program itself never gets the number. `return` hands the value to the program: it can be stored in a variable, compared against another, passed into the next function.

Let's test that in practice. Try saving the "result" of the first function:

```python
result = add_print(2, 3)
print(result)
```

Two lines show up: first 5, then None. The `print` inside the function printed the 5. And `result` ended up holding None, Python's special value meaning "nothing here". A rule with no exceptions: a function that reaches the end of its body without a `return` returns None. Printing inside changes nothing, `print` is not a return.

None's mischief is that it doesn't surface right away. Look:

```python
def double_price(x):
    print(x * 2)

total = double_price(450) + 100
```

The code prints 900 and immediately crashes with `TypeError`: you can't add 100 to None. The symptom "the function seems to print the right thing, but I can't use the result" almost always means one thing: there's a `print` where a `return` should be.

## Optional parameters and keyword arguments

The `delivery_price` header sneaked in something new: `express=False`. That's a parameter with a default value. If the call doesn't pass that argument, Python quietly substitutes the default. That's why `delivery_price(2)` works: weight is required, express shipping isn't.

```python
def greet(name, polite=True):
    if polite:
        return "Здравствуйте, " + name
    return "Привет, " + name

print(greet("Анна"))
print(greet("Борис", polite=False))
```

The second call demonstrates another trick: the argument is passed by name, `polite=False`. Arguments like this are called keyword arguments. The name spares the reader from guessing what a bare `False` in the parentheses would mean, and once there are several defaults, it lets you set only the one you need and skip the rest. One rule applies: in the header, parameters with defaults come after required ones, and Python rejects `def greet(polite=True, name)` before it even runs.

## Variables live inside a function

One last piece remains: the fate of variables created inside a body. Short answer: they live only inside the function, and only for the duration of a single call.

```python
def delivery_price(weight_kg):
    price = 199 + 40 * weight_kg
    return price

total = delivery_price(2)
print(price)
```

The last line crashes with `NameError: name 'price' is not defined`. The variable `price` was born inside the call and vanished along with it. The only thing that made it out is the value `return` carried, and it lives in `total`. The zone where a name exists is called its scope. This is a protection for you, not a limitation: inside a function you can name variables freely, without worrying about clashing with names elsewhere in the program, and every call starts fresh with its own clean set.

## Frequent mistakes

- **`print` instead of `return`.** Everything looks right on screen, yet a check fails, or trying to compute something with the result crashes with `TypeError`. It happens because printed output looks exactly like a result. The fix: a function returns a value, and whoever calls it decides whether to print.
- **Calling without parentheses.** `greet` without parentheses is the function itself as a value, not its result. After `message = greet`, the variable holds a function; the body only runs with parentheses, `greet("Анна")`.
- **Expecting a local variable outside.** Compute `price` inside a function, then try to print it after the call: you get `NameError`. Only `return` carries a value out.
- **Code after `return`.** Lines below a `return` that already fired never run. If code you added to a function "just doesn't work for some reason," check whether it sits behind an early exit.

<Detail summary="A function's name is just an ordinary variable">
The line `def greet(...):` does two things: it creates a function object and binds it to the name `greet`, exactly the way `x = 5` binds a number to the name `x`. That's why you can drop a function into another variable: after `say = greet`, calling `say("Анна")` works exactly like the original. And `print(greet)` without parentheses will honestly show what's sitting in the name: something like `<function greet at 0x7f9c...>`. The idea that "functions are values, just like numbers and strings" will surface again and again in more advanced Python topics, from sorting with a key to decorators.
</Detail>
