# Your first lines of code: variables and data types

Anna is buying books on Ozon: one book costs 450 rubles, she needs three, and delivery adds another 199. A calculator gives you the total and forgets it instantly. Change the price tomorrow, and you retype the whole calculation button by button. A program works differently: it remembers every value under its own name and keeps the calculation itself like a recipe. Change the price, rerun the program, get a fresh total. Naming a value and then working with that name is the first skill any programmer picks up, and it's exactly what this lesson teaches.

Here you'll write your first lines of Python: print to the screen, create variables, learn what separates the number 5 from the text "5", and see why division in Python doesn't quite answer like a calculator. No setup required, and every example runs right on this page, ready for you to edit.

## A program is a list of commands

A Python program is just plain text made of commands. Python reads it top to bottom, line by line, and runs each command in turn: nothing gets skipped, nothing jumps ahead, the order never changes. The one command no lesson can do without is `print()`. It prints to the screen whatever you put inside its parentheses.

```python
print("Привет!")
print(2 + 2)
```

Two commands here. The first prints text: quotes tell Python "output this literally". The second first evaluates the expression `2 + 2`, then prints the result, the number 4. That difference matters: anything in quotes is taken as-is, anything without quotes Python tries to evaluate. Line order matters too: ask Python to print a variable before it exists, and it will honestly tell you it knows no such name.

<MarginNote label="History">Guido van Rossum created the language in 1991 and named it after the comedy troupe Monty Python, not the snake.</MarginNote>

## A variable is a name tag, not a box

Picture a kitchen shelf lined with jars. Each jar holds some grain, and to avoid mixing them up you stick a label on each one: "rice", "buckwheat". The label doesn't hold the grain, it just points at the jar. A Python variable works the same way: the value lives in memory on its own, and the name is a label stuck onto that value.

```python
price = 450
qty = 3
print(price * qty)
```

Read the line `price = 450` right to left: create the value 450 and stick a label named `price` onto it. The `=` sign here isn't the "equals" from school math, it's an assignment command. From then on you can use the name anywhere: on the third line Python sees `price * qty`, follows the labels, finds 450 and 3, multiplies them, and prints 1350.

Many tutorials draw a variable as a box you drop a value into. For Python the label is the more accurate picture: several names can point at the very same value at once. Look at the figure below: after `cost = price`, both names lead to the exact same object, and nothing gets copied anywhere.

While you're working with numbers, the difference between a label and a box stays invisible. But the name-as-label model is how Python actually works under the hood, and it becomes decisive in the lesson on lists and dictionaries, where two labels on one list regularly trip up beginners. Better to get used to the honest picture from the start.

A label can be re-stuck. A repeated assignment peels the name off the old value and sticks it onto the new one:

```python
balance = 1000
balance = 750
print(balance)
```

The program prints 750. A name always leads to the last value assigned to it, and the old link vanishes without a trace: Python keeps no history of relabeling.

## Naming variables

A variable name is built from Latin letters, digits, and underscores, but it can't start with a digit: `price2` is fine, while `2price` crashes the program with a `SyntaxError` before it even runs. Upper and lower case are different: `Price` and `price` are two independent names, and a typo in casing creates a second variable instead of changing the first.

Python convention is snake_case: lowercase words separated by underscores, like `delivery_price` or `user_age`. And the golden rule of a good name: it should answer the question "what's stored here". A week from now, `price_rub` will explain itself, while `p` will leave you guessing what you meant.

## Four data types

Every value in Python has a type. Four are enough to get started:

- `int`, whole numbers: 450, 3, 0.
- `float`, fractional numbers: 3.5, 199.99. The fractional part is separated by a dot, not a comma.
- `str`, strings, meaning any text in quotes: "Anna", "Kazan", "450".
- `bool`, the logical type: exactly two values, True and False.

The `type()` function reveals the type of any value:

```python
print(type(450))
print(type(3.5))
print(type("450"))
print(type(True))
```

Look closely at the third line. The value "450" is in quotes, so its type is str: it's text, not a number. For Python, a jar labeled "honey" and the honey itself are fundamentally different things. Shortly you'll see how mixing them breaks a program. It's worth memorizing these type names for another reason too: they keep showing up in error messages, and recognizing them on sight saves a lot of time.

For now it's enough to know `bool` exists: this type answers yes-or-no questions and takes the spotlight in the lesson on conditions. Strings get their own dedicated lesson, next in line.

## Arithmetic and the first surprise

Python handles numbers like a calculator: `+` adds, `-` subtracts, `*` multiplies, `/` divides. There's exponentiation too: `2 ** 10` gives 1024. No surprises so far. Now run this:

```python
print(6 / 2)
```

The screen shows 3.0, not 3. Division with `/` always returns a float, even when the numbers divide evenly. That's deliberate: the result of a division isn't known in advance, so Python picks one predictable answer type instead of guessing case by case.

When you need a whole-number answer, a different pair of operators steps in. `//` divides and drops the fractional part, `%` returns the remainder. A everyday example: seven candies split between two people. Each person gets `7 // 2`, three candies, and `7 % 2`, one candy, stays in the bowl.

```python
print(7 // 2)
print(7 % 2)
print(7 / 2)
```

Now put it all together: below is a live example built around Anna's order from the start of the lesson. Change `qty` to 2 and rerun, the total recalculates itself. Then swap `/` for `//` on the last line and compare the answers. Finally, add a line with `print(type(total))` and check your intuition about types.

## From string to number and back

Data arrives in your program as text far more often than you'd like. A web form hands you the entered age as a string even if the field held nothing but digits; a price list file stores a price as the text "450". Try adding such a "price" to a number:

```python
print("5" + 5)
```

The program crashes with a `TypeError`. That's Python looking out for you: it doesn't know whether you wanted to glue the text into "55" or add the numbers to get 10, so it refuses to guess. The fix is an explicit type conversion:

```python
age = int("34")
price = float("199.99")
label = str(450)
print(age + 1)
```

`int()` turns a string of digits into a whole number, `float()` into a fractional one, and `str()` works the other way and turns a number into text. After `int("34")` the age behaves like any ordinary number: the program prints 35. You'll dig into string concatenation with `+` and other text tricks in the lesson on strings.

## Common pitfalls

Two cases regularly trip up beginners. First, `int()` doesn't round, it truncates: `int(3.9)` returns 3, not 4. Second, fractional numbers are stored approximately:

```python
print(int(3.9))
print(0.1 + 0.2)
```

The second line prints 0.30000000000000004. Nothing is wrong with your computer: that's how float works in every programming language, from Python to JavaScript. The practical takeaway is simple: count money in whole cents, using `int`, and save `float` for measurements where a tiny error fifteen digits in bothers nobody.

<Detail summary="Why 0.1 + 0.2 doesn't equal 0.3">
Computers store numbers in binary. The fraction 0.1 is infinite in binary, just like 1/3 is infinite in familiar decimal: 0.3333 and on forever. The float type allocates 64 bits to a number, so the infinite "tail" gets cut off, and memory holds the nearest representable number, slightly off from a true tenth. When you add two such approximations, the error creeps into the last digits of the result. For money and other exact quantities, use whole numbers or the special `Decimal` type from the standard library.
</Detail>

## Frequent mistakes

- **Text without quotes.** `name = Анна` crashes with `NameError`: without quotes, Python looks for a variable named Анна instead of the text. Anything meant as text belongs in quotes.
- **String plus number.** `"5" + 5` raises `TypeError`. Convert both values to the same type first: `int("5") + 5` gives 10, and `"5" + str(5)` gives "55".
- **A name starting with a digit.** `2fast = 10` is a `SyntaxError`. Digits are allowed in a name, just not as the first character.
- **Expecting `=` to compare.** The `=` sign assigns. Checking whether two values are equal uses the double sign `==`, covered in detail in the lesson on conditions.
