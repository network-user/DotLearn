# Lists and dictionaries: many values under one name

Picture a program that totals up a shopping cart for an online store. Anna picked out twenty items on Ozon, and each one needs somewhere to live. From previous lessons you know one storage method: variables. Create `item1`, `item2`, `item3`, and keep going up to `item20`? By line five it's already clear the plan is doomed: the code balloons, and there's no way to go through every purchase at once. Worse, you don't even know in advance how many items there will be: the cart fills up while the program runs, and variables can't spring into existence like that.

Python has collections for jobs like this, values that hold other values inside them. This lesson covers the two main ones. A list stores values in order, like a queue or a shelf. A dictionary stores "key and value" pairs, like an address book. By the end you'll assemble a shopping cart, total its cost in a single line, build a phone book, and run into a trap that catches nearly every beginner.

## A list: a shelf with numbered slots

A list is created with square brackets, values inside separated by commas. A useful picture: a shelf with numbered slots. Each slot holds one value, the slot's number is called its index, and, just like with strings from the second lesson, counting starts at zero.

```python
cart = ["хлеб", "молоко", "сыр"]
print(cart)
print(len(cart))
print(cart[0])
```

The familiar `len()` function returns a list's item count: here it's three, however many letters sit inside the words themselves. Indexing works just like with strings: `cart[0]` gives the first value, `cart[2]` the last of the three. The figure above shows why "the third item" and "index 3" aren't the same thing: there's no slot number 3 on this shelf, and `cart[3]` stops the program with `IndexError`.

A list can hold values of any type: strings, numbers, booleans, even other lists. Elements are usually the same type, since that's simpler to work with, but Python doesn't require it. Lists surround you in every app: transaction history in a banking app, a playlist, a delivery queue. Anywhere values are ordered and their count changes, a programmer reaches for a list.

## A list is alive and changes

With strings in the second lesson you learned something unpleasant at first glance: a string can't be changed, every method returns a new one. A list works the opposite way: it's mutable. Three operations cover most tasks. `.append()` adds a value to the end. Assignment by index drops a new value onto an existing slot. `.remove()` finds the first matching value and takes it out.

```python
cart = ["хлеб", "молоко", "сыр"]
cart.append("яблоки")
cart[0] = "батон"
cart.remove("сыр")
print(cart)
```

Walk through the code top to bottom. After `append`, the shelf grew by one slot on the right. The assignment `cart[0] = "батон"` replaced the bread, and the length stayed the same. The `remove` call took out the cheese, and its neighbors slid left without leaving a gap. The print shows `['батон', 'молоко', 'яблоки']`. Notice the key point: all three operations change the list itself. No new object gets created, the same one is modified.

The `in` operator, familiar from strings, checks whether a value sits on the shelf. It returns a boolean, the same kind from the lesson on conditions, so it's handy to drop straight into an `if`.

```python
cart = ["батон", "молоко", "яблоки"]
print("молоко" in cart)
if "сыр" not in cart:
    print("надо купить сыр")
```

With a list of numbers, Python computes a summary without a single extra line: `sum()` adds up every element, `min()` and `max()` find the smallest and largest. ## A dictionary: lookup by name, not by number

A numbered shelf works great as long as order matters. Now for a different task: storing friends' phone numbers. In a list, Boris's number might sit third, and finding it would mean remembering the position. An address book is friendlier: you look things up by name. Python's answer is the dictionary. It stores entries where every value has its own key, and finds the value by that key.

```python
phones = {"Анна": "+7 901 123-45-67", "Борис": "+7 902 555-66-77"}
print(phones["Анна"])
```

A dictionary is created with curly braces. Entries are listed comma-separated, each one a key, then a colon, then a value. Reading looks like a list, only the square brackets hold a key instead of a slot number. Dictionaries show up everywhere data has a natural name: a price by product name, a population by city, a profile field on a government portal.

Adding and overwriting use the exact same construction:

```python
phones = {"Анна": "+7 901 123-45-67"}
phones["Вера"] = "+7 903 777-88-99"
phones["Анна"] = "+7 901 000-00-00"
print(len(phones))
```

The rule is simple. If the key didn't exist, a new entry appears, which is how Vera got added to the book. If the key already existed, the old value is overwritten by the new one, which is how Anna's number changed. A dictionary never holds two entries with the same key, so `len` prints 2, not 3.

What happens when you read a key that doesn't exist? Square brackets stop the program with `KeyError`, exactly the way a list stops it with `IndexError`. When a missing key is a normal situation rather than a bug, reach for the `.get()` method: its second argument is a fallback value, returned instead of an error.

```python
phones = {"Анна": "+7 901 123-45-67"}
print(phones.get("Борис", "номера нет"))
```

The `in` operator on a dictionary checks only the keys. The check `"Анна" in phones` gives `True`, while checking the phone number itself gives `False`, even if that number sits inside the dictionary: `in` never looks at values.

A value can be anything: a string, a number, another dictionary, even a whole list. That's how several values end up living under one key:

```python
orders = {"Анна": ["хлеб", "молоко"], "Борис": ["сыр"]}
orders["Борис"].append("чай")
print(orders["Борис"])
```

## Which to pick: list or dictionary

Both collections hold many values, but they answer different questions, and the choice is almost always obvious from the task.

## Surprise: a copy that never existed

Now for the promised trap. Intuition suggests the line `b = a` builds a second, identical list. Check it:

```python
a = ["хлеб", "молоко"]
b = a
b.append("сыр")
print(a)
```

The print shows `['хлеб', 'молоко', 'сыр']`. The cheese was added through `b`, and somehow `a` changed too. The explanation: a second list never existed. A variable name is a label stuck onto an object, and the line `b = a` simply sticks a second label onto that same list. There's one list and two labels, so the change shows up through either one.

When you genuinely need a copy, ask for one explicitly: `b = a.copy()`. From that point on there are two independent lists. Notice that this surprise is impossible with strings and numbers, and here's why: they can't be changed in place. Two labels on one string are harmless, because no operation will ever change that string. Remember the labeling mechanics: it fires again in the seventh lesson, once you start passing lists into functions.

## Frequent mistakes

- **Counting indexes from one.** `cart[1]` is the second element, not the first. Real-life habit outweighs logic, so this mistake slips past even experienced developers. Check yourself against the first element: it's always `cart[0]`.
- **Assigning the result of append.** The line `cart = cart.append("сыр")` destroys the list: the method changes it in place and returns `None`, and that `None` lands in `cart`. Call `cart.append("сыр")` on its own line, with no assignment.
- **Reading an uncertain key with square brackets.** If a key might not exist, `phones["Вера"]` crashes the program with `KeyError`. For the normal "might not be there" case, use `.get()` with a fallback value.
- **Expecting `b = a` to make a copy.** Both names point at one object, and changes show through either one. If you need an independent copy, write `a.copy()`.

<Detail summary="Why a dictionary finds a key instantly">
A list searches by scanning: to check whether cheese sits in a cart of a thousand items, Python may have to look through the whole thousand in the worst case. A dictionary works a smarter way. From the key it computes a number called a hash, and that number points straight to where the value lives inside the dictionary. Lookup speed doesn't depend on size: ten entries or a million, the answer arrives in the same time. That comes with a restriction: only an immutable value can be a key, a string or a number, for instance. A list can never be a key: it changes, and its hash would change along with it, and the dictionary would lose track of its own entry.
</Detail>
