# Status codes: the language a server uses to tell you what happened

Support writes in: "Users say the site is down." The logs for the last hour show three hundred responses with code 502 and fifty with 404. Those are two completely different diagnoses: 404 means "no such page," a links-or-content problem; 502 means "the proxy never got a response from the app," your backend is down. Whoever can read status codes starts fixing the right thing half an hour sooner.

## Mental model: the first digit is the class

A status code is a three-digit number in the response's start line. You don't need to memorize all of them: the first digit sorts them into five classes, and the class alone tells you most of what matters, as shown in Figure 1.

The line between 4xx and 5xx is a line of responsibility. 4xx means "this request is bad": a typo in the address, missing permissions, an invalid body. The client can fix the request and try again. 5xx means "the request was fine, the server couldn't handle it": no amount of fixing the request helps, the server itself needs fixing.

## The working set: codes you'll meet this week

Success rarely comes in many flavors: `200 OK` means done, here's the result. `201 Created` means a new resource now exists, the normal response to a successful POST. `204 No Content` means done, and there's nothing to report: the typical response to DELETE, with no body at all.

Redirects are more interesting. The server answers with a 3xx code and a `Location` header telling you where to go:

```http
HTTP/1.1 301 Moved Permanently
Location: https://market.yandex.ru/catalog

```

On seeing this, the browser fires off a new request to the address in `Location` automatically. The user only ever sees the final page.

Client errors are the most crowded class: `400 Bad Request` means the body or syntax couldn't be parsed; `401 Unauthorized` means "I don't know who you are"; `403 Forbidden` means "I know who you are, and you can't"; `404 Not Found` means no such resource; `429 Too Many Requests` means slow down.

Server errors: `500 Internal Server Error` means an unhandled exception in the application code; `502 Bad Gateway` and `504 Gateway Timeout` mean the proxy couldn't reach the app or gave up waiting for it; `503 Service Unavailable` means the server is alive but temporarily not serving, usually overload or a deploy in progress.

## Pairs everyone mixes up

**301 vs. 302.** Both mean "resource is elsewhere," the difference is the word "permanently." `301 Moved Permanently` is a move for good: search engines transfer the page's ranking to the new address, and browsers cache the redirect itself aggressively. `302 Found` is temporary: everyone keeps treating the old address as home. Mixing these up hurts: a 301 set in a hurry will stick in browsers' memory long after you change your mind.

**401 vs. 403.** `401` means "I don't know you": the request had no token, or an expired one; the correct client reaction is to send the user to log in. `403` means "I know exactly who you are, and you're not allowed": a regular user knocking on the admin panel. Logging in again won't help. Mnemonic: 401 is about authentication, 403 is about authorization.

**500 vs. 502/503/504.** Look at Figure 2: there's almost always a proxy like nginx sitting between client and application. `500` is born inside the application, go look for a stack trace in the app logs. `502` and `504` are born in the proxy, when the application died or hung. `503` is a deliberate "can't handle this right now" answer, often during a deploy.

## What will surprise you

**200 doesn't guarantee success.** Some APIs return `200 OK` with a JSON body that has an error field buried inside. These "soft errors" slip past monitoring set up to watch status codes, and caches don't know to skip them either. It's bad practice, but common enough that you still have to check the body regardless.

**404 can mean "I'm not telling you."** Services like GitHub return 404 instead of 403 for someone else's private repository: otherwise the mere fact of answering "forbidden" would confirm the resource exists.

**429 comes with instructions.** Alongside `429 Too Many Requests`, servers often send a `Retry-After` header telling you how many seconds to wait. A well-behaved client reads it instead of hammering away with retries.

## Common mistakes

**Returning 200 for everything.** Clients have to parse the body just to find out something failed, caches store broken responses as if they were fine, and monitoring goes blind. The code in the start line exists for machines, use it.

**Reaching for 500 when the client is at fault.** An invalid request body is a 400 or 422, not "we threw an exception." 5xx pages wake up an on-call engineer at night; don't page them over someone else's mistake.

**Ignoring the difference between 301 and 302.** Especially during a domain move: a temporary redirect where a permanent one belongs dilutes search ranking, and a permanent one where a temporary one belongs gets stuck in browser caches for good.

<Detail summary="The teapot that made it to production">
  Status code 418, "I'm a teapot," comes from RFC 2324, an April Fools' RFC about a protocol for brewing tea: a teapot must respond 418 if you ask it to brew coffee. The joke stuck around, plenty of frameworks still support the code, and some services return it for deliberately absurd requests.
