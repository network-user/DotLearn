# Методы: глаголы протокола и их обещания

Пользователь нажал «Оплатить», интернет мигнул, страница зависла. Он нажал ещё раз. Спишутся ли деньги дважды? Ответ зависит не от везения, а от того, каким методом отправлен запрос и выполняет ли сервер обещание этого метода. Методы HTTP - это не просто слова в стартовой строке, а контракты: каждый метод обещает, можно ли его безопасно повторять и меняет ли он что-то на сервере.

## Модель: метод отвечает на вопрос «что сделать с ресурсом»

Путь в запросе называет **ресурс** - вещь, с которой работаем: страницу, товар, заказ. Метод называет **действие** над ней. Пять методов покрывают почти всю практику:

- `GET` - прочитать ресурс. Ничего не менять.
- `POST` - создать новый ресурс или запустить обработку.
- `PUT` - целиком заменить ресурс присланной версией.
- `PATCH` - изменить часть ресурса.
- `DELETE` - удалить ресурс.

Помимо действия каждый метод несёт два свойства-обещания, и на рис. 1 видно, как методы по ним распределяются. **Безопасный** (safe) метод не меняет состояние сервера: сколько ни читай, ничего не сломается. **Идемпотентный** метод можно повторить много раз, и итоговое состояние сервера будет таким же, как после одного раза.

Почему это не теория ради теории: **сети ненадёжны**. Запрос может дойти, а ответ потеряться. Клиент не знает, выполнилась ли операция, и единственное, что он может сделать, - повторить запрос. С идемпотентным методом повтор безвреден. С POST повтор означает второй заказ и второе списание.

## Читаем методы в сыром виде

Запрос на чтение выглядит знакомо: тела нет, вся суть в пути.

```http
GET /orders/1042 HTTP/1.1
Host: api.wildberries.ru
```

Создание заказа - POST с телом. Обратите внимание: путь называет коллекцию, а не конкретный заказ, ведь номера у нового заказа ещё нет.

```http
POST /orders HTTP/1.1
Host: api.wildberries.ru
Content-Type: application/json

{"product_id": 77, "quantity": 2}
```

Полная замена ресурса - PUT по конкретному адресу. Клиент присылает новое состояние целиком, и повтор такого запроса запишет то же самое поверх того же самого:

```http
PUT /orders/1042 HTTP/1.1
Host: api.wildberries.ru
Content-Type: application/json

{"product_id": 77, "quantity": 3, "comment": "позвонить заранее"}
```

Частичное изменение - PATCH: в теле только то, что меняется.

```http
PATCH /orders/1042 HTTP/1.1
Host: api.wildberries.ru
Content-Type: application/json

{"quantity": 5}
```

## PUT против PATCH: замена против правки

Разница между ними - любимый вопрос на собеседованиях, и она проще, чем кажется. PUT говорит: «вот ресурс целиком, положи вместо старого». Поля, которые вы не прислали, исчезнут или сбросятся. PATCH говорит: «вот изменения, примени к тому, что есть». Отсюда и идемпотентность: PUT с одним и тем же телом идемпотентен по определению, а PATCH - смотря какой. Патч «установи quantity равным 5» повторять безопасно, патч «увеличь quantity на 1» - нет.

<MarginNote label="На практике">Многие API используют PATCH только с «установи значение», и тогда он ведёт себя идемпотентно. Но протокол этого не гарантирует - гарантирует только дисциплина авторов сервера.</MarginNote>

## Что вас удивит

**Идемпотентность - про состояние сервера, а не про ответы.** Первый DELETE вернёт 200, повторный - скорее всего 404: ресурса уже нет. Это не нарушение контракта: итоговое состояние сервера одинаковое - ресурс удалён. Разные ответы клиенту контракт не нарушают.

**Обещания держит сервер, а не протокол.** Ничто физически не мешает написать обработчик GET, который удаляет записи. Протокол лишь фиксирует договорённость, на которую полагается вся инфраструктура: браузеры смело повторяют GET при обрыве, прокси кешируют ответы на GET, поисковые роботы ходят по всем ссылкам подряд. История знает случай, когда робот, обходя админку со ссылками «удалить» на GET, вычистил сайт целиком.

**У GET может быть тело, но его никто не ждёт.** Стандарт не запрещает тело в GET, однако серверы и прокси вправе его игнорировать. Параметры чтения передают в строке запроса, а не в теле.

## Частые ошибки

**Мутации через GET.** «Быстренько сделаю ссылку /delete?id=7» - и однажды её дёрнет предзагрузка браузера или робот. Всё, что меняет данные, должно ехать методом с подходящей семантикой.

**Выбор между PUT и PATCH наугад.** Спросите себя: клиент присылает ресурс целиком или только изменения? Целиком - PUT, изменения - PATCH.

**Повтор POST без защиты.** Кнопка «Оплатить» должна блокироваться после клика, а серьёзные API принимают ключ идемпотентности, чтобы распознать повтор. Полагаться на «пользователь не нажмёт дважды» нельзя.

<Detail summary="HEAD, OPTIONS и ключи идемпотентности">
  Кроме большой пятёрки есть служебные методы. HEAD - это GET без тела ответа: заголовки те же, груза нет. Удобно проверить размер файла до скачивания. OPTIONS спрашивает сервер о его возможностях; браузер использует его в preflight-запросах CORS - подробности в теме про CORS. А от дублей POST платёжные API защищаются ключом идемпотентности: клиент генерирует уникальную строку и шлёт её в заголовке вроде Idempotency-Key; получив повтор с тем же ключом, сервер не выполняет операцию заново, а возвращает сохранённый ответ.
</Detail>
