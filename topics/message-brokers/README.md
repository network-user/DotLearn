# Брокеры сообщений: RabbitMQ и Kafka

Тема для backend-разработчиков и архитекторов, которые хотят понять, что такое брокер сообщений, чем отличаются две доминирующие модели - RabbitMQ и Kafka - и какие гарантии доставки реально даёт каждая. Без привязки к конкретному языку: упор на ментальные модели (очередь против лога, push против pull, at-least-once против exactly-once), а не на API клиента. После темы вы сможете выбирать брокер под задачу осознанно и проектировать consumer'ов, которые не теряют и не дублируют данные.

## Концепты

1. **Зачем нужен брокер сообщений** - синхронный вызов против асинхронного сообщения, развязка сервисов, сглаживание пиков, буфер между быстрым producer и медленным consumer.
2. **RabbitMQ: exchange, queue, binding, ack** - модель умного брокера: exchange маршрутизирует, queue хранит, binding связывает; ручное подтверждение и возврат сообщений.
3. **Kafka: лог, партиции, offset, consumer groups** - распределённый append-only лог, партиционирование по ключу, offset как позиция чтения, масштабирование через consumer groups.
4. **Гарантии доставки и идемпотентность** - at-most-once, at-least-once, exactly-once; почему дубликаты неизбежны и как идемпотентный consumer делает повтор безопасным.

## Prerequisites

Нет. Достаточно базового понимания, что такое сервис и сетевой вызов между сервисами.
