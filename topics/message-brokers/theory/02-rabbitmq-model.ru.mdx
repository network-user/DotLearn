# RabbitMQ: умный брокер, который сам решает, куда положить сообщение

В прошлом разделе брокер был чёрным ящиком: producer кладёт, consumer забирает. Но как именно сообщение находит свою очередь? У RabbitMQ на этот вопрос есть подробный ответ, и он устроен не так, как многие ожидают. Producer в RabbitMQ не пишет в очередь напрямую - он вообще не знает, какие очереди существуют. Он отправляет сообщение в exchange и прикладывает к нему ярлык - routing key. А дальше брокер сам, по заранее настроенным правилам, раскладывает сообщение по нужным очередям.

Это и есть модель «умного брокера» (smart broker, dumb consumer): логику маршрутизации держит сервер, а клиенты остаются простыми. К концу раздела вы будете различать четыре кирпича модели AMQP - exchange, queue, binding, routing key - понимать четыре типа exchange и знать, почему без ручного подтверждения (manual ack) RabbitMQ теряет сообщения при падении consumer'а.

## Ментальная модель: почта с сортировкой

Представьте почтовое отделение. Вы не кладёте письмо сразу в чужой почтовый ящик - вы отдаёте его на сортировочный стол (exchange) с адресом на конверте (routing key). Сортировщик смотрит на правила (bindings), решает, в какие ячейки (queues) положить копию письма, и раскладывает. Получатель (consumer) забирает письма только из своей ячейки и понятия не имеет про сортировочный стол.

На схеме сообщение с `key=eu` попадает в `queue.eu` (точное совпадение) и в `queue.all` (привязана по `#`, «всё подряд»), но не в `queue.us`. Ключевое: producer не выбирал очереди - он только пометил конверт `eu`. Решение приняли bindings, настроенные заранее. Поменяете bindings - изменится маршрут, и ни одной строчки в producer'е трогать не придётся.

## Четыре кирпича: exchange, queue, binding, routing key

Разберём роли по одной. **Queue** - именованный буфер, где сообщения лежат и ждут consumer'а; это единственное место, где сообщение реально хранится. **Exchange** - точка приёма от producer'а; сам он ничего не хранит, только маршрутизирует. **Binding** - правило-связка «класть из этого exchange в эту queue при таком условии». **Routing key** - ярлык на сообщении, который binding сверяет со своим условием.

```text
publish(exchange="orders", routing_key="order.eu", body=...)
binding: orders --[order.eu]--> queue.eu
binding: orders --[order.#]--> queue.audit
```

Сообщение с ключом `order.eu` уйдёт в обе очереди: `queue.eu` ловит точный ключ, `queue.audit` ловит всё семейство `order.*` по шаблону `order.#`. Одно сообщение - сколько угодно копий, по числу сработавших bindings. Если ни один binding не совпал, сообщение по умолчанию тихо отбрасывается - это частый источник «куда делось сообщение».

<Callout type="warning">
Сообщение, опубликованное в exchange, у которого нет ни одного подходящего binding, исчезает без ошибки. Producer считает, что отправил успешно. Прежде чем искать баг в consumer'е, проверьте, что binding вообще существует.
</Callout>

## Четыре типа exchange решают «куда»

Тип exchange определяет, как именно binding сверяет routing key. Их четыре, и они покрывают почти все сценарии маршрутизации.

```text
direct  : routing key совпадает с binding key точно   ("order.eu" == "order.eu")
topic   : совпадение по шаблону с * и #               ("order.eu" matches "order.#")
fanout  : routing key игнорируется, копия во все queue
headers : маршрут по заголовкам сообщения, не по ключу
```

`direct` - для точечной адресации: ключ либо равен, либо нет. `topic` - для иерархических ключей вроде `order.eu.priority`, где `*` заменяет одно слово, а `#` - любое их число. `fanout` - чистый broadcast: каждая привязанная очередь получает копию, ключ не смотрится вовсе. `headers` используют редко - когда маршрутизировать удобнее по парам ключ-значение в заголовках, чем по строке-ключу.

## Ack: сообщение удаляется только после подтверждения

Дальше начинается самое важное для надёжности. Consumer забрал сообщение - можно ли сразу стереть его из очереди? Если стереть и consumer тут же упадёт, не доделав работу, сообщение потеряно навсегда. Поэтому в RabbitMQ есть ручное подтверждение: сообщение остаётся в очереди (в состоянии unacked), пока consumer явно не пришлёт `ack`. Только тогда брокер его удаляет.

```text
1. consumer получает сообщение      -> статус: unacked
2. consumer обработал успешно       -> шлёт ack -> брокер удаляет
3. consumer упал до ack             -> брокер возвращает сообщение в очередь
```

Если consumer закрыл соединение, не подтвердив сообщение, RabbitMQ считает его необработанным и отдаёт другому consumer'у (или тому же после перезапуска). Это и есть гарантия at-least-once: сообщение не теряется, но при неудачном падении может прийти повторно. Противоположный режим - автоматический ack (autoack), когда брокер удаляет сообщение в момент доставки, ещё до обработки. Быстрее, но любое падение consumer'а означает потерю.

<Callout type="tip">
По умолчанию включайте manual ack и подтверждайте сообщение только после того, как работа реально сделана и её результат сохранён. autoack оставляйте для потоков, где потерять отдельное сообщение не страшно (например, метрики).
</Callout>

## Краевой случай: prefetch и «жадный» consumer

Вот что удивляет новичков. Без настройки prefetch брокер может отгрузить одному consumer'у сразу сотни неподтверждённых сообщений - тот «расхватал» их, пока второй consumer простаивает. В итоге вместо равномерной нагрузки один воркер завален, другой скучает, а если жадный упадёт, в очередь разом вернётся вся его пачка.

```text
prefetch = 1  : брокер даёт следующее сообщение только после ack предыдущего
prefetch = N  : у consumer'а одновременно до N unacked сообщений
```

`prefetch=1` даёт честное распределение: следующее сообщение уходит к тому, кто только что освободился. Это идеально для долгих неравномерных задач. Для коротких однородных сообщений маленький prefetch создаёт лишние round-trip'ы, и небольшое значение вроде 10-50 даёт пропускную способность выше. Дефолт без ограничения почти всегда неправильный выбор.

## Типичные ошибки

- **Публиковать в очередь напрямую, минуя exchange.** В чистом AMQP producer всегда пишет в exchange. «Прямая отправка в очередь» в клиентах - это сахар: сообщение уходит в безымянный default-exchange с routing key, равным имени очереди. Понимать это важно, иначе topic- и fanout-маршрутизация кажутся магией.
- **Оставлять autoack ради скорости.** autoack удаляет сообщение до обработки. При любом падении consumer'а сообщение исчезает. Скорость, купленная ценой потери данных, - плохая сделка для всего, кроме заведомо необязательных потоков.
- **Забывать про несработавшие bindings.** Опубликованное «в никуда» сообщение пропадает молча. Если consumer ничего не получает, первым делом проверяйте, что binding между exchange и очередью существует и его шаблон совпадает с ключом.

<Detail summary="Что происходит при отказе: dead-letter и возврат">

Когда consumer отвергает сообщение (`nack`/`reject`) без requeue, или у сообщения истёк TTL, или очередь переполнена, сообщение можно не выбрасывать, а перенаправить в dead-letter exchange (DLX) - отдельный exchange, привязанный к «карантинной» очереди. Туда стекаются сообщения, которые не удалось обработать, чтобы их можно было разобрать вручную или повторить позже с задержкой. Без DLX отвергнутое без requeue сообщение просто исчезает. DLX - стандартный способ строить retry с экспоненциальной задержкой: отвергнутое сообщение уходит в очередь с TTL, по истечении которого возвращается на повторную обработку.

</Detail>
