# Kafka: не очередь, а журнал, который помнит порядок

RabbitMQ думает очередями: сообщение пришло, его забрали, его больше нет. Kafka устроена иначе настолько, что переносить интуицию очереди на неё - первая и самая дорогая ошибка. Kafka - это распределённый журнал (log): новые записи только добавляются в конец, ничего не вставляется в середину и не удаляется при чтении. Consumer не «забирает» сообщение - он читает журнал с того места, где остановился, и двигает свою закладку дальше. Прочитанная запись остаётся в логе и доступна следующему читателю.

Эта смена модели - от «очередь, которая пустеет» к «журнал, который растёт» - тянет за собой всё остальное: партиции для масштаба, offset как позицию чтения, consumer groups для параллелизма. К концу раздела вы будете понимать, почему порядок в Kafka гарантирован только внутри партиции, зачем нужен ключ сообщения и почему добавить больше consumer'ов, чем партиций, бессмысленно.

## Ментальная модель: append-only журнал с закладками

Топик в Kafka - это журнал, разбитый на партиции. Каждая партиция - независимая последовательность записей, пронумерованных подряд: 0, 1, 2, 3... Этот номер и есть offset - позиция записи в партиции. Producer дописывает в конец партиции, и offset новой записи всегда на единицу больше предыдущего. Consumer хранит свой offset - закладку, говорящую «я дочитал до этого места». Чтение не удаляет записи; оно лишь сдвигает закладку вперёд.

На схеме две закладки стоят в разных местах одной партиции: consumer A дочитал до offset 1, consumer B - до offset 3. Они не мешают друг другу и не «расходуют» записи - те же самые записи 0..4 доступны обоим. Producer тем временем готовит запись с offset 5 в конец. Запомните этот образ: в Kafka данные не исчезают при чтении, они живут в логе по времени удержания (retention), независимо от того, кто и сколько раз их прочитал.

## Партиции: единица параллелизма и порядка

Одна партиция - это строго упорядоченная последовательность, но и узкое место: её читает в группе только один consumer. Чтобы масштабироваться, топик делят на несколько партиций, и тогда несколько consumer'ов читают их параллельно. Расплата за параллелизм - порядок. Kafka гарантирует порядок записей **только внутри одной партиции**. Между партициями порядка нет.

```text
topic "orders" с 3 партициями:
  P0: [order-7] [order-7] [order-3]
  P1: [order-2] [order-9]
  P2: [order-5] [order-1] [order-5]
```

Внутри P0 события заказа 7 идут строго по порядку их записи. Но событие заказа 2 (в P1) и событие заказа 5 (в P2) между собой не упорядочены - они в разных партициях. Если вам важен порядок событий конкретного заказа, все его события должны попасть в одну партицию. Этим управляет ключ сообщения.

<Callout type="info">
Порядок в Kafka - свойство партиции, а не топика. «Сообщения приходят по порядку» верно только для записей с одинаковым ключом, которые поэтому лежат в одной партиции.
</Callout>

## Ключ сообщения определяет партицию

Producer может задать ключ сообщения. Kafka берёт хеш ключа по модулю числа партиций и так выбирает партицию: одинаковый ключ - всегда одна и та же партиция, а значит, гарантированный порядок для этого ключа.

```text
partition = hash(key) % partition_count
key="order-7" -> всегда P0 -> все события заказа 7 упорядочены
key=null      -> распределение по кругу (round-robin), порядок не гарантирован
```

Выбор ключа - это проектное решение. Ключ `order_id` собирает все события одного заказа в одну партицию и сохраняет их порядок. Ключ `user_id` упорядочивает события одного пользователя. А вот ключ с одним-двумя значениями (например, флаг `is_premium`) сложит почти все сообщения в одну-две партиции и убьёт параллелизм - это перекос (partition skew).

## Consumer groups: параллелизм без дублирования

Consumer group - это набор consumer'ов с общим именем группы, которые делят между собой партиции топика. Kafka раздаёт партиции членам группы так, что **каждую партицию читает ровно один consumer в группе**. Добавили consumer'а - Kafka перераспределит партиции (rebalance) и отдаст ему часть. Это даёт горизонтальное масштабирование: больше партиций и consumer'ов - выше пропускная способность.

```text
topic 3 партиции, группа из 2 consumer'ов:
  C1 читает P0, P1
  C2 читает P2
добавили C3:
  C1 -> P0, C2 -> P1, C3 -> P2  (по одной каждому)
добавили C4:
  C4 простаивает - партиций всего 3
```

Отсюда правило: число партиций - это потолок параллелизма группы. Consumer'ов в группе может быть меньше партиций (тогда кто-то читает несколько), но больше - бессмысленно: лишние будут простаивать без партиций. Если вы упёрлись в пропускную способность и все партиции уже разобраны, добавлять consumer'ов бесполезно - нужно увеличивать число партиций (а это нельзя сделать без последствий для порядка по ключу).

<Callout type="tip">
Закладывайте число партиций с запасом на рост: уменьшить его нельзя, а увеличение перетасует распределение ключей по партициям и нарушит порядок для существующих ключей. Лучше сразу взять, скажем, 12 партиций, чем потом мучительно репартиционировать.
</Callout>

## Краевой случай: одно сообщение - много групп

Вот что ломает интуицию очереди. В RabbitMQ сообщение, отданное consumer'у, для остальных исчезает. В Kafka одну и ту же запись независимо читают сколько угодно групп. Сервис биллинга (группа `billing`) и сервис аналитики (группа `analytics`) оба читают топик `orders` целиком, каждый со своей закладкой, не мешая друг другу.

```text
topic "orders":
  группа "billing"   : своя закладка, видит все записи
  группа "analytics" : своя закладка, видит все те же записи
```

Внутри группы партиция достаётся одному consumer'у (параллелизм без дублей), а между группами - полный broadcast (каждая группа видит весь поток). Это сочетание - очередь внутри группы и подписка между группами - и есть то, что делает Kafka одновременно и очередью задач, и шиной событий. Один топик обслуживает обе модели сразу.

## Типичные ошибки

- **Ждать глобального порядка по топику.** Порядок есть только внутри партиции. Если логика требует строгого порядка событий сущности, кладите их в одну партицию через общий ключ - иначе порядок не гарантируется.
- **Ставить consumer'ов больше, чем партиций.** Лишние consumer'ы в группе простаивают: партиция достаётся ровно одному члену. Параллелизм ограничен числом партиций, а не числом процессов.
- **Выбирать ключ с малым числом значений.** Ключ-флаг или ключ-страна с двумя популярными значениями создаёт перекос: одна партиция переполнена, остальные пусты, и параллельная обработка превращается в последовательную.

<Detail summary="Куда девается старое и кто хранит offset">

Kafka удаляет старые записи не по факту чтения, а по политике retention: по времени (например, 7 дней) или по размеру лога. Пока запись в пределах retention, её можно перечитать, сдвинув offset назад - в этом сила лога как источника правды. Сами offset'ы групп Kafka хранит в служебном топике `__consumer_offsets`: когда consumer коммитит offset, он по сути пишет в Kafka запись «группа G дочитала партицию P до позиции N». Поэтому при перезапуске consumer продолжает с закоммиченной позиции, а не с начала. Отдельный режим - log compaction: вместо удаления по времени Kafka хранит последнее значение для каждого ключа, превращая топик в постоянную таблицу «ключ - актуальное состояние».

</Detail>
