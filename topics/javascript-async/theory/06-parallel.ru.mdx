# Три запроса, которые ждут друг друга без причины

Анна собирает страницу профиля для Ozon-подобного магазина. Перед отрисовкой нужно получить три независимые вещи: данные пользователя, содержимое корзины и список рекомендаций. Ни один из трёх запросов не нуждается в результате другого, поэтому Анна пишет три `await` подряд, один за одним, как в списке дел:

```js
const profile = await fetchProfile(userId);
const cart = await fetchCart(userId);
const recommendations = await fetchRecommendations(userId);
```

Каждый запрос к бэкенду занимает примерно 700 миллисекунд. Анна замеряет время загрузки страницы и получает не 700, а больше двух тысяч миллисекунд. Втрое дольше, хотя ни один запрос не стал медленнее. Причина в том, что `await` ставит асинхронную функцию на паузу до тех пор, пока промис не осядет, и следующая строка даже не начинает выполняться, пока не закончилась предыдущая. Три независимых запроса Анна случайно выстроила в очередь, хотя они прекрасно могли лететь одновременно. К концу главы вы будете запускать независимые промисы параллельно и собирать их результаты через `Promise.all`, `allSettled`, `race` и `any`, выбирая нужный инструмент осознанно, а не наугад.

Секрет параллельного запуска простой: промис создаётся в момент вызова асинхронной функции, а не в момент `await`. Если вызвать все три функции подряд без `await`, все три запроса уйдут на сервер сразу же, и уже потом можно дождаться готовых промисов вместе:

```js
const profilePromise = fetchProfile(userId);
const cartPromise = fetchCart(userId);
const recommendationsPromise = fetchRecommendations(userId);

const [profile, cart, recommendations] = await Promise.all([
  profilePromise,
  cartPromise,
  recommendationsPromise,
]);
```

Три вызова без `await` в первых трёх строках запускают все три запроса почти в одну и ту же миллисекунду. `Promise.all` собирает три промиса в один и ждёт, пока осядут все, а общее время страницы падает до времени самого медленного запроса, около 700 миллисекунд вместо 2100.

<PullQuote>Промис начинает работать в момент вызова функции, а не в момент await: если ждать нечего в порядке, ждать можно параллельно.</PullQuote>

## Promise.all: ждать всех, а падать на первой ошибке

`Promise.all` принимает массив промисов и возвращает один промис. Он осядет, когда осядут все переданные промисы, причём результат придёт массивом в том же порядке, в каком промисы стояли в исходном массиве, независимо от того, какой из них закончился раньше остальных:

```js
function delay(ms, value) {
  return new Promise((resolve) => setTimeout(() => resolve(value), ms));
}

const results = await Promise.all([delay(300, 'a'), delay(100, 'b'), delay(200, 'c')]);
console.log(results);
```

Промис `'b'` осядет первым, через 100 миллисекунд, `'c'` через 200, `'a'` последним, через 300. Но в консоль уйдёт `['a', 'b', 'c']`: массив результатов сохраняет порядок аргументов, а не порядок завершения. Это удобно: результат для корзины всегда лежит на своём месте, даже если сервер рекомендаций ответил быстрее сервера корзины.

Есть и обратная сторона. `Promise.all` работает по принципу «падает на первом же провале»: как только хотя бы один из промисов отклоняется, весь `Promise.all` немедленно отклоняется с той же причиной, даже если остальные промисы ещё выполняются и в итоге успешно завершились бы.

## Promise.allSettled: когда нужны все ответы, даже вперемешку с ошибками

Если сервис рекомендаций у Ozon временно упал, а профиль и корзина загрузились нормально, `Promise.all` выбросит их результаты вместе с ошибкой: одна авария стоит трёх потерянных ответов. Для страницы это плохо: лучше показать профиль и корзину, а вместо рекомендаций нарисовать заглушку. Именно для такого случая существует `Promise.allSettled`. Он тоже ждёт все промисы, но никогда сам не отклоняется. Вместо значения или ошибки каждому промису достаётся объект с полем `status`, равным `'fulfilled'` или `'rejected'`:

```js
const results = await Promise.allSettled([
  fetchProfile(userId),
  fetchCart(userId),
  fetchRecommendations(userId),
]);

for (const result of results) {
  if (result.status === 'fulfilled') {
    console.log(result.value);
  } else {
    console.log('ошибка:', result.reason);
  }
}
```

Успешный промис даёт объект вида `{ status: 'fulfilled', value }`, отклонённый даёт `{ status: 'rejected', reason }`. Проверка `result.status` в цикле разбирает оба случая по отдельности, поэтому падение одного сервиса не отнимает результаты у остальных.

## Promise.race: побеждает тот, кто осел первым, хоть с успехом, хоть с ошибкой

`Promise.race` принимает массив промисов и оседает так же, как первый из них, каким бы этот исход ни был, успехом или ошибкой. Классический пример: гонка настоящего запроса с искусственным таймаутом.

```js
function timeout(ms) {
  return new Promise((_, reject) => setTimeout(() => reject(new Error('таймаут')), ms));
}

const data = await Promise.race([fetchRecommendations(userId), timeout(1000)]);
```

Если сервис рекомендаций отвечает быстрее секунды, `data` получит его результат, а промис таймаута просто останется невостребованным. Если сервис завис дольше секунды, первым осядет `timeout`, он отклонится, и `Promise.race` отклонится вслед за ним с ошибкой «таймаут». Страница получает шанс показать заглушку вместо бесконечного ожидания зависшего сервера.

## Promise.any: побеждает только успех, ошибки терпят до последнего

`Promise.any` внешне похож на `race`, но реагирует только на успех. Он оседает, как только хоть один промис зафиксирует успешный результат, а отклонения остальных до этого момента просто игнорирует. Отклонится `Promise.any` только в одном случае: если отклонились все переданные промисы, и тогда он вернёт `AggregateError` со списком всех причин.

```js
try {
  const response = await Promise.any([
    fetchFromMirror('ru-1'),
    fetchFromMirror('ru-2'),
    fetchFromMirror('ru-3'),
  ]);
  console.log(response);
} catch (error) {
  console.log('все зеркала недоступны:', error.errors);
}
```

Такой код подходит для опроса нескольких равнозначных серверов одного и того же сервиса: важно получить ответ от любого живого зеркала, а какое из них ответило первым, не имеет значения.

## Особенности, которые легко просмотреть

Первая неожиданность: `Promise.all` не отменяет проигравшие промисы после падения на первой ошибке. Промисы, которые уже были запущены, продолжают выполняться в фоне до конца, просто их результат уже некому забрать. Если один из запросов при этом производит побочный эффект, например списывает деньги, он всё равно спишет их, несмотря на то что `Promise.all` уже отклонился где-то раньше в коде.

Вторая неожиданность уже прозвучала выше, но её стоит закрепить: порядок результатов в `Promise.all` и `Promise.allSettled` всегда совпадает с порядком аргументов, а не с порядком завершения. Полагаться на «кто раньше ответил, тот и первый в массиве» нельзя.

Третья неожиданность касается пустых массивов. `Promise.all([])` оседает немедленно с пустым массивом `[]`: ждать нечего, значит ждать не нужно. А вот `Promise.race([])` ведёт себя ровно наоборот: раз оседать не с кем, промис не осядет никогда, и код, ожидающий его через `await`, зависнет навсегда. Пустой массив в `race` почти всегда ошибка программиста, а не осознанное решение.

## Частые ошибки

Первая ошибка: `await` внутри цикла там, где запросы независимы.

```js
const results = [];
for (const id of userIds) {
  const user = await fetchUser(id);
  results.push(user);
}
```

Каждая итерация ждёт предыдущую, поэтому время растёт линейно с числом пользователей, хотя запросы никак друг от друга не зависят. Лечится сбором промисов через `map` и одним `Promise.all` в конце:

```js
const promises = userIds.map((id) => fetchUser(id));
const results = await Promise.all(promises);
```

Вторая ошибка: брать `Promise.all` там, где на самом деле нужен `allSettled`. Если хотя бы один запрос из десяти падает регулярно, скажем сервис рекомендаций, `Promise.all` будет терять результаты девяти успешных запросов из-за одного нестабильного. Признак этой ошибки простой: вы ловите ошибку `Promise.all` в `try/catch` и внутри блока `catch` пытаетесь угадать, какие из промисов всё-таки выполнились. Если такой код появился, пора переходить на `allSettled`.

Третья ошибка: путать `race` и `any`. `race` реагирует на первый осевший промис вообще, включая отклонённые, поэтому одна упавшая раньше остальных попытка испортит результат для всех. `any` создан именно для обратного случая: нужен хотя бы один успех, а сколько попыток провалилось по пути, неважно. Если задача звучит как «дай ответ от любого живого сервера», нужен `any`, а не `race`.

<Detail summary="Почему Promise.all не может отменить проигравшие промисы">
У промиса в JavaScript нет встроенного способа остановить работу, которая его создала. Промис только сообщает о результате, но не управляет породившей его операцией. `Promise.all` умеет перестать ждать и передать управление дальше, но заставить `fetch` прекратить сетевой запрос он не может: для этого существует отдельный механизм, `AbortController`, и он не связан с `Promise.all` напрямую. Поэтому проигравшие промисы просто доигрывают в фоне вхолостую.
</Detail>
