# Код, который расползается вправо с каждым шагом

Вера дорабатывает старый раздел админки маркетплейса, похожего на Ozon: нужно по идентификатору достать пользователя, затем его заказы, затем детали последнего заказа. Библиотека, с которой она работает, написана до того, как в языке появились промисы, и вся асинхронность в ней держится на функциях, которые передают внутрь других функций. Вера дописывает третий шаг, и код в редакторе начинает уезжать вправо: полезная строка с `console.log` прячется в конце длинной лестницы отступов, а перед ней десяток закрывающих скобок подряд. У такого кода есть отдельное название: разработчики в шутку зовут его «пирамидой судьбы». Коллега, который проверяет её изменения, оставляет короткий комментарий в духе «раздели это на функции», и Вера понимает: дело не в одной неудачной строке, а в самом способе писать асинхронный код через колбэки.

Вера не первая, кто упирается в эту лестницу. До появления промисов почти весь асинхронный код в JavaScript строился на одной и той же идее: функция, которую передают внутрь другой функции, чтобы та вызвала её позже, когда закончит свою работу. Именно так вы уже пользовались `setTimeout` в прошлой главе: туда тоже передают функцию, и она срабатывает не сразу, а когда истечёт время. Такую функцию называют колбэком, от английского call back, «перезвонить». Раньше колбэк был единственным инструментом асинхронности в языке, сегодня он чаще прячется под капотом промисов и `async`/`await`, но сама идея никуда не делась.

## Каждый следующий шаг живёт внутри предыдущего

Проблема Веры не в том, что она плохо пишет код. Три шага её сценария зависят друг от друга по-настоящему: нельзя узнать заказы пользователя, пока не узнали, что это за пользователь, и нельзя открыть детали заказа, пока не получили список заказов. Когда каждый следующий шаг стартует только после того, как готов предыдущий, а готовность приходит через колбэк, единственный способ написать код такой: вложить вызов следующего шага прямо внутрь колбэка предыдущего. Три зависимых шага дают три уровня вложенности, а пять шагов дадут пять, и на каждом уровне добавляется свой отступ. Изменить такой код тоже непросто: чтобы вставить новый шаг в середину цепочки, приходится раздвигать отступы во всех вложенных функциях, а лишняя или потерянная закрывающая скобка в конце теряется среди десятка таких же.

Сценарий Веры, если разложить его по шагам, выглядит так:

<Steps>
  <li>Получить пользователя по идентификатору.</li>
  <li>По идентификатору пользователя получить список его заказов.</li>
  <li>По идентификатору последнего заказа получить его детали.</li>
</Steps>

Каждый следующий шаг нельзя запустить раньше предыдущего, и передать управление дальше при колбэках можно только одним способом: вложить следующий вызов внутрь колбэка предыдущего. Ниже мы соберём этот сценарий в код и увидим ту же лестницу уже не на схеме, а на экране.

## Ошибка приезжает первым аргументом

Прежде чем собирать шаги вместе, разберёмся с одной важной привычкой. В экосистеме Node.js колбэки почти никогда не получают только результат: первым аргументом им передают ошибку, а вторым, если ошибки не случилось, сам результат. Такое соглашение называют error-first: сигнатура выглядит как `(err, result) => { ... }`, и правильный колбэк первым делом проверяет `err`, а уже потом читает `result`. Такую сигнатуру можно встретить почти в любой встроенной функции Node.js, от чтения файла до запроса к базе данных. Привычка возникла не из любви к строгим правилам: если у всех асинхронных функций один и тот же порядок аргументов, их проще комбинировать и читать чужой код без документации под рукой.

```js
function loadUser(id, callback) {
  setTimeout(() => {
    if (id <= 0) {
      callback(new Error('нет такого пользователя'));
      return;
    }
    callback(null, { id, name: 'Вера' });
  }, 300);
}
```

Функция `loadUser` моделирует запрос к серверу через `setTimeout`, точно так же как в прошлой главе. Если `id` некорректный, она вызывает колбэк с ошибкой первым аргументом и без второго. Если всё в порядке, первым аргументом идёт `null`, то есть «ошибки нет», а вторым готовый пользователь. Колбэк, который использует такую функцию, обязан заглянуть в `err` раньше, чем трогать `user`:

```js
loadUser(1, (err, user) => {
  if (err) {
    console.log('ошибка:', err.message);
    return;
  }
  console.log(user.name);
});
```

Пока шаг один, проверка `err` совсем не напрягает: одна строка в начале колбэка. Проблема начинается, когда шагов становится несколько и каждый следующий зависит от результата предыдущего.

## Лестница из трёх шагов

Добавим ещё две функции того же вида: они грузят заказы пользователя и детали конкретного заказа, тоже через `setTimeout` и тоже по соглашению error-first.

```js
function loadOrders(userId, callback) {
  setTimeout(() => callback(null, [{ id: 501, total: 1200 }]), 300);
}

function loadOrderDetails(orderId, callback) {
  setTimeout(() => callback(null, { id: orderId, status: 'доставлен' }), 300);
}
```

Теперь соберите три функции в цепочку так, как это делала Вера: следующий шаг вызывается внутри колбэка предыдущего, и на каждом уровне повторяется проверка `err`, потому что ошибка может случиться на любом из трёх шагов.

```js
loadUser(1, (err, user) => {
  if (err) return console.log(err.message);
  loadOrders(user.id, (err, orders) => {
    if (err) return console.log(err.message);
    loadOrderDetails(orders[0].id, (err, details) => {
      if (err) return console.log(err.message);
      console.log('заказ', details.status);
    });
  });
});
```

Вот она, пирамида судьбы с рисунка выше: три содержательные строки и три проверки ошибок расползаются вправо, потому что каждый следующий шаг физически вложен в предыдущий. В реальном модуле Веры шагов было пять, а не три, и последняя содержательная строка съезжала за правый край монитора.

## Управление, которое вы отдаёте чужому коду

Есть менее заметная проблема, которая прячется за самим устройством колбэков. Когда вы вызываете `loadUser` и передаёте туда свою функцию, вы теряете контроль над тем, как её вызовут. Библиотека может вызвать колбэк ровно один раз, как ожидается. А может вызвать его дважды. А может забыть вызвать вовсе, если где-то внутри случится необработанное исключение. Может вызвать его синхронно, прямо во время вызова `loadUser`, а может, как в примерах этой главы, только через `setTimeout`, то есть асинхронно. Со стороны вызывающего кода все эти варианты неотличимы, пока не проявятся на практике.

Есть ещё более коварный вариант той же непредсказуемости. Представьте функцию, которая сначала проверяет кеш и только потом идёт за данными на сервер. Если нужный пользователь уже лежит в кеше, функция может вызвать колбэк сразу же, синхронно, прямо во время вызова `loadUser`. Если кеша нет, она полезет в сеть и вызовет тот же колбэк только через `setTimeout` или ответ сервера, то есть асинхронно. Снаружи оба случая выглядят как один и тот же вызов, но код, который стоит в вызывающей функции сразу после `loadUser`, в первом случае отработает уже после колбэка, а во втором раньше него. Такую особенность в сообществе Node.js в шутку называют Zalgo: функция обещала быть асинхронной, но временами нарушает своё же обещание.

<PullQuote>Передавая колбэк в чужую функцию, вы отдаёте ей право решать, когда и сколько раз вас позвать.</PullQuote>

Такой эффект называют инверсией управления: обычно ваш код сам решает, что и когда вызвать, а здесь это решение отдано чужой функции. Именно непредсказуемость этого решения, а не только разрастающиеся отступы, много лет была главным доводом в пользу более строгого инструмента для асинхронного кода. В большом приложении колбэков такого рода десятки: загрузка данных, повторные попытки после сбоя сети, аналитика, и каждый из них скрыто отдаёт решение о времени вызова чужому коду.

## Частые ошибки

Первая ошибка: пропустить проверку `err` и сразу читать результат.

```js
loadUser(-1, (err, user) => {
  console.log(user.name);
});
```

Так как `id` отрицательный, `loadUser` вызовет колбэк с ошибкой и без второго аргумента, поэтому `user` окажется `undefined`, а обращение к `user.name` уронит программу с исключением уже внутри колбэка. Проверка `err` в первой строке колбэка не декоративная деталь, а единственная защита от такого падения.

<Callout type="warning" title="Пропущенная проверка err молчит до поры">
Если код не проверяет `err`, ошибка не исчезает: она просто ждёт своего часа в виде `undefined` там, где ожидалось значение. Программа падает не в `loadUser`, а несколькими строками ниже, там, где `undefined` впервые попытались использовать как объект.
</Callout>

Вторая ошибка: не замечать, что вложенность стала архитектурной проблемой. Один уровень колбэка внутри колбэка почти не мешает читать код. Но если содержательная строка провалилась на четыре или пять уровней вправо, это уже не вопрос вкуса, а сигнал, что структуру пора менять: разбивать шаги на именованные функции или переходить на другой инструмент, который решает эту проблему по-другому. Практический тест простой: если, чтобы прочитать код одного шага, приходится проскролливать вправо за пределы экрана, вложенность уже стала проблемой, а не особенностью стиля.

Третья ошибка: колбэк, вызванный дважды или не вызванный вовсе. Если внутри `loadUser` в одной из веток забыли вызвать колбэк, для остального кода это будет означать вечно висящий индикатор загрузки: ничего не упадёт, но и дальше ничего не произойдёт. А если колбэк вызовут дважды, например один раз по успеху и ещё раз по ошибке таймаута, любой код внутри него, который меняет состояние, скажем увеличивает счётчик заказов, выполнится дважды и незаметно испортит данные. На практике это может выглядеть как письмо с подтверждением заказа, которое покупатель Ozon получает дважды подряд, хотя оплатил один раз.

<Detail summary="Не всякий колбэк про ошибки">
Соглашение error-first придумано для асинхронных функций, которые могут не выполниться. Синхронные колбэки вроде тех, что передают в `array.map` или `array.filter`, устроены иначе: они получают только элемент массива, без `err` первым аргументом, потому что вызываются сразу и упасть с асинхронной ошибкой не могут. Сигнатура `(err, result)` относится именно к асинхронному стилю Node.js, а не к колбэкам вообще.
</Detail>

<MarginNote label="История">Соглашение error-first сложилось в экосистеме Node.js в конце 2000-х как неформальный стандарт, ещё до того, как в язык добавили промисы в 2015 году.</MarginNote>

Все три проблемы этой главы, расползающуюся вложенность, забытые проверки `err` и непредсказуемость чужого вызова, в JavaScript решает следующий инструмент асинхронности, о котором пойдёт речь в следующей главе. Тот же сценарий Веры с пользователем, заказами и деталями заказа в нём умещается без единой лестницы отступов.
