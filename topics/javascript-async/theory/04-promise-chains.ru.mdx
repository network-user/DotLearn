# Пирамида, которая выпрямляется в цепочку

Анна переписывает вход в личный кабинет магазина вроде Ozon: сначала запрос login с почтой и паролём, из ответа нужен токен, на токен грузится профиль, из профиля берётся id и грузятся заказы, заказы отрисовываются на экране. Четыре асинхронных шага подряд, и Анна честно вкладывает каждый следующий шаг внутрь предыдущего `then`, как будто это просто вложенные условия:

```js
login(email, password).then((token) => {
  loadProfile(token).then((profile) => {
    loadOrders(profile.id).then((orders) => {
      render(orders);
    });
  });
});
```

Код работает, но растёт вправо с каждым новым шагом. Добавить пятый шаг значит вложить ещё один уровень скобок, а обработку ошибки пришлось бы дублировать в каждой вложенной функции отдельно. Это и есть та самая пирамида, о которой предупреждают на любом собеседовании по JavaScript: чем длиннее реальный сценарий (войти, загрузить профиль, загрузить заказы, посчитать бонусы), тем труднее в ней вообще что-то найти глазами. У неё есть простое решение: тот же самый `then` можно ставить не внутрь, а подряд, одним плоским списком.

```js
login(email, password)
  .then((token) => loadProfile(token))
  .then((profile) => loadOrders(profile.id))
  .then((orders) => render(orders));
```

Четыре шага, четыре строки одного уровня вложенности. Чтобы понять, почему такая запись вообще законна, разберёмся, что на самом деле возвращает `then`.

## Мысленная модель: труба из промисов

`then` не просто «навешивает колбэк на промис». Он создаёт и возвращает совершенно новый промис. Именно поэтому на результате `then` можно сразу же вызвать ещё один `then`: вы работаете не с исходным промисом от `login`, а с промисом, который вернул предыдущий `then`. Цепочка `then` после `then` после `then` это на самом деле труба из промисов, соединённых один за другим, где значение течёт от входа к выходу через каждую секцию. Такую трубу можно тянуть сколько угодно шагов: пять, десять `then`, каждый получает то, что вернул предыдущий, и передаёт дальше своё.

На рисунке видно две вещи разом. В обычном ходе событий значение идёт по трубе слева направо: каждый `then` берёт то, что пришло из предыдущей секции, и отдаёт дальше уже своё. А если на каком-то шаге случается ошибка, труба меняет маршрут: оставшиеся `then` она пропускает насквозь, не трогая, и ошибка вылетает в ближайший `catch`. К этому маршруту мы вернёмся чуть ниже, а сначала разберёмся с обычным течением значения.

## Что именно возвращает then в следующий then

Значение, которое вы вернули из колбэка `then`, становится результатом промиса, который `then` отдал наружу. Это значение и придёт аргументом в следующий `then`.

```js
fetchOrderTotal('order-42')
  .then((total) => total * 1.2)
  .then((totalWithTax) => {
    console.log(totalWithTax);
  });
```

Первый `then` получает число `total`, умножает его и возвращает результат обычным `return`. Этот результат не потерялся: он стал значением, с которым разрешился промис первого `then`, и именно оно попало в `totalWithTax` второго `then`. Если бы функция ничего не возвращала, второй `then` получил бы `undefined`, а не итог с налогом. К этой ловушке мы ещё вернёмся в разделе про частые ошибки.

<Callout type="tip" title="then всегда возвращает промис, даже из обычного значения">
Если колбэк then вернул обычное число или строку, then всё равно оборачивает его в новый разрешённый промис. Поэтому следующий then в цепочке получает уже готовое значение и не обязан ничего проверять: работает ли предыдущий шаг с простым return или с настоящим асинхронным запросом, запись `.then().then()` остаётся одинаковой.
</Callout>

## Если вернуть промис, цепочка ждёт его

Самое полезное свойство `then` раскрывается, когда из колбэка возвращают не обычное значение, а сам промис. Так и устроен пример с профилем и заказами из начала главы.

```js
login(email, password)
  .then((token) => loadProfile(token))
  .then((profile) => loadOrders(profile.id))
  .then((orders) => render(orders));
```

`loadProfile(token)` сама по себе асинхронная и возвращает промис. Когда колбэк первого `then` возвращает этот промис, движок не передаёт его следующему `then` как есть. Он дожидается, пока `loadProfile` осядет, разворачивает результат и только тогда запускает следующий `then` с готовым значением `profile`. Это называют выравниванием, или flattening: сколько бы вложенных промисов вы ни возвращали, следующий `then` в цепочке всегда получит готовое значение, а не обёртку промиса вокруг промиса. На этом выравнивании и держится вся последовательность из четырёх асинхронных шагов, уложенная в четыре ровные строки: каждый шаг стартует ровно тогда, когда предыдущий по-настоящему закончился, а не когда до него просто дошла очередь исполнения.

## Ошибка перепрыгивает через then

Теперь вторая часть модели с рисунка выше. Если колбэк `then` бросает исключение через `throw`, либо промис, который он вернул, отклоняется, управление не идёт в следующий `then` по порядку. Оно перепрыгивает через все промежуточные `then` и попадает в ближайший `catch` дальше по цепочке.

```js
login(email, password)
  .then((token) => loadProfile(token))
  .then((profile) => loadOrders(profile.id))
  .then((orders) => render(orders))
  .catch((error) => showError(error));
```

Если `loadProfile` отклонится (например, токен истёк), второй и третий `then` не выполнятся вовсе. Управление сразу окажется в `catch`, который стоит в самом конце. Ровно так же поведёт себя ошибка из `loadOrders` или из `render`: один `catch` в хвосте цепочки ловит ошибку с любого из предыдущих шагов, где бы она ни случилась. Это удобно: не нужно оборачивать в try каждый шаг отдельно, достаточно одного обработчика на всю последовательность.

<PullQuote>Ошибка не проверяет then один за другим, она сразу прыгает к ближайшему catch.</PullQuote>

## Забытый return рвёт цепочку

Самая частая неожиданность связана не с ошибками, а с обычным возвратом значения. Стрелочная функция с фигурными скобками не возвращает результат сама, ей нужен явный `return`. В цепочке промисов эта забывчивость выглядит особенно коварно.

```js
login(email, password)
  .then((token) => {
    loadProfile(token);
  })
  .then((profile) => {
    console.log(profile);
  });
```

Здесь `loadProfile(token)` вызван, но его промис никуда не возвращён. Первый `then` завершится, даже не дождавшись ответа профиля, и вернёт `undefined`. Следующий `then` не станет ждать `loadProfile`, он запустится почти сразу и напечатает `undefined` вместо профиля. Программа не падает и не сообщает об ошибке, она просто продолжает работу с неверными данными, а гонка между реальным ответом сервера и уже напечатанным `undefined` каждый раз может выглядеть по-разному. Чинится это одним словом:

```js
login(email, password)
  .then((token) => {
    return loadProfile(token);
  })
  .then((profile) => {
    console.log(profile);
  });
```

Возврат `loadProfile(token)` отдаёт недостающий промис обратно в цепочку, и второй `then` дожидается настоящего профиля вместо мгновенного `undefined`.

## catch посередине не обрывает цепочку

`catch` не обязан стоять только в самом конце. Если поставить его между шагами, он поймает ошибку предыдущего участка, а цепочка после него продолжится как ни в чём не бывало, потому что `catch` тоже возвращает обычный промис.

```js
loadOrders(userId)
  .catch(() => [])
  .then((orders) => renderOrders(orders));
```

Если `loadOrders` отклонится, `catch` подставит пустой массив вместо падения, и `then` дальше по цепочке получит именно его. Разница между `catch` посередине и `catch` в конце в том, что именно они экранируют: серединный ловит и гасит ошибку только с шагов до себя, а хвостовой перехватывает всё, что не поймали раньше. Такой приём удобен для запасных значений: список заказов не загрузился, но страница всё равно отрисуется, пусть и пустой.

## finally выполняется всегда, но не меняет значение

`finally` ставят, когда какое-то действие нужно сделать в любом случае: скрыть спиннер загрузки, закрыть соединение. Он срабатывает и при успехе, и при ошибке, но не получает передаваемое по цепочке значение и не может его подменить.

```js
loadOrders(userId)
  .then((orders) => renderOrders(orders))
  .catch((error) => showError(error))
  .finally(() => hideSpinner());
```

`hideSpinner` вызовется независимо от того, отработал `then` или сработал `catch`. Но если внутри `finally` написать `return` с каким-то значением, оно никуда не передастся дальше: цепочка продолжает нести то, что пришло в `finally`, будто его и не было, за исключением случая, когда `finally` сам бросает исключение.

## Частые ошибки

Первая ошибка: забыть `return` внутри `then`, когда там вызывается ещё одна асинхронная функция. Цепочка перестаёт ждать нужный шаг и едет дальше с `undefined`. Лечится одним правилом: если внутри `then` есть асинхронный вызов, его результат нужно вернуть, а не просто выполнить и забыть.

Вторая ошибка: снова вкладывать промисы внутрь `then` вместо того, чтобы их возвращать. Стоит один раз забыть про `return` промиса и написать вместо этого ещё один вложенный `.then` прямо внутри колбэка, и пирамида, от которой цепочка должна была избавить, возвращается на то же место, только теперь она спрятана посередине с виду плоского кода.

Третья ошибка: поставить единственный `catch` не там, где нужно, и потерять ошибки поздних шагов. Например, поставить `catch` сразу после первого `then`, а не в конце всей цепочки: ошибки третьего и четвёртого шага пройдут мимо него необработанными и попадут в консоль как необработанное отклонение промиса.

Эти три ошибки объединяет одно неприятное свойство: цепочка не падает с понятным сообщением, она просто начинает вести себя не так, как задумано, а разбираться приходится по логам и догадкам. Читать такую последовательность шагов сверху вниз ещё удобнее с `async` и `await`: это тема следующей главы.

<Detail summary="catch это сокращение для then с двумя аргументами">
Метод then умеет принимать два колбэка: первый для успешного результата, второй для ошибки. Запись `.catch(onError)` это то же самое, что `.then(undefined, onError)`, только короче и читается яснее. Разница на практике проявляется в одном месте: ошибка, брошенная внутри самого обработчика успеха в паре `then(onSuccess, onError)`, не попадёт в соседний onError той же пары, а полетит дальше по цепочке, тогда как отдельный catch после такого then её поймает.
</Detail>
