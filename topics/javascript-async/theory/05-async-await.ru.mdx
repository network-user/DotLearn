# async/await: промисы, которые читаются сверху вниз

Анна дорабатывает оформление заказа на Ozon. Сначала нужно получить профиль покупателя, затем по идентификатору профиля запросить сохранённые адреса, затем создать заказ по первому адресу. Три асинхронных шага она собрала в цепочку `then`:

```js
function checkout(userId) {
  return getUser(userId)
    .then((user) => getAddresses(user.id))
    .then((addresses) => createOrder(addresses[0]))
    .then((order) => order.id)
    .catch((error) => {
      console.error('оформление не удалось', error);
      throw error;
    });
}
```

Код рабочий, но чтение требует усилия: приходится держать в голове, что результат первого `then` подставляется во второй, а обработка ошибки собрана отдельно в самом низу. Борис предлагает переписать те же шаги через `async` и `await`:

```js
async function checkout(userId) {
  try {
    const user = await getUser(userId);
    const addresses = await getAddresses(user.id);
    const order = await createOrder(addresses[0]);
    return order.id;
  } catch (error) {
    console.error('оформление не удалось', error);
    throw error;
  }
}
```

Порядок шагов и итоговое поведение остались теми же самыми. Изменилось только то, как код выглядит на экране: второй вариант читается подряд, строка за строкой, будто перед вами обычный синхронный расчёт, без единого `then`.

Разница особенно заметна не при первом написании кода, а через полгода, когда в оформление заказа нужно добавить ещё один шаг или найти, откуда пришла ошибка. В цепочке `then` для этого приходится читать снизу вверх, восстанавливая, какой колбэк за каким следует и что именно он получает на вход. В варианте с `await` достаточно поставить точку останова на нужной строке: переменные `user`, `addresses` и `order` видны сразу, как в обычной функции, без раскапывания вложенных колбэков.

## async/await это тот же промис, только по-другому записанный

`async` и `await` не заменяют промисы и не создают новый способ асинхронности. Это синтаксис поверх уже знакомого механизма: под капотом всё так же остаются промисы, микрозадачи и событийный цикл из предыдущих глав. `async/await` меняет не порядок исполнения, а форму записи этого порядка.

Есть и менее заметное отличие. В цепочке `then` каждый шаг живёт в своём колбэке с отдельной областью видимости: чтобы передать значение из первого шага в третий, его приходится протаскивать через параметры промежуточных функций или собирать в общий объект. В `async`-функции все переменные объявлены в одном теле функции и видны на всех последующих строках, включая `catch`. Это не новая возможность языка, обычная область видимости функции работает так всегда, но именно в асинхронном коде разница между «переменная видна» и «переменную нужно протаскивать через колбэки» ощущается заметнее всего.

## async function всегда возвращает промис

Первое правило, без которого `async/await` не понять: функция, объявленная со словом `async`, всегда возвращает промис, даже если внутри неё нет ни одного `await`. Значение после `return` не выходит наружу напрямую, оно оборачивается в разрешённый промис:

```js
async function getAnswer() {
  return 42;
}

getAnswer().then((value) => console.log(value));
```

Выводится 42, но только после `then`. Сам вызов `getAnswer()` отдаёт не число, а промис, разрешённый в 42, и получить число без `await` или `then` нельзя:

```js
async function getAnswer() {
  return 42;
}

console.log(getAnswer());
```

Здесь в консоль уйдёт промис, а не число. Точно так же работает и обратная ситуация: `throw` внутри `async`-функции не роняет программу, а превращает возвращаемый промис в отклонённый:

```js
async function loadUser(id) {
  if (!id) {
    throw new Error('нет идентификатора');
  }
  return { id, name: 'Борис' };
}

loadUser().catch((error) => console.log(error.message));
```

Выводится «нет идентификатора». `throw` здесь ведёт себя как `.reject` в промисе, собранном вручную: ошибку можно поймать любым способом, каким ловят отклонение обычного промиса.

Слово `async` ставится не только перед `function`. Оно точно так же работает со стрелочной функцией (`const load = async () => {}`) и с методом класса или объекта (`async loadUser() {}` внутри класса). Правило о промисе на выходе остаётся неизменным для всех трёх форм записи: какую бы из них вы ни выбрали, вызов всё равно вернёт промис, а не готовое значение.

## await разворачивает промис в значение

Второе правило касается того, что происходит внутри `async`-функции. Оператор `await` приостанавливает выполнение функции до тех пор, пока промис справа от него не осядет, и возвращает уже развёрнутое значение, а не сам промис:

```js
async function loadOrderTotal(orderId) {
  const order = await getOrder(orderId);
  return order.total;
}
```

Внутри `loadOrderTotal` переменная `order` это готовый объект заказа, а не промис с этим объектом внутри: разворачивание уже произошло на строке с `await`. Если справа от `await` стоит не промис, а обычное значение, оно просто возвращается как есть, без ожидания:

```js
async function showTotal() {
  const price = await 100;
  console.log(price);
}

showTotal();
```

Выводится 100. `await` не требует промиса как условия работы, он одинаково успешно разворачивает и промис, и уже готовое значение, только в первом случае реально ждёт, а во втором отдаёт значение сразу.

Важно, что именно останавливается на строке с `await`. Это не вся программа и не главный поток целиком, а только тело текущей `async`-функции. Пока `loadOrderTotal` ждёт ответа сети, событийный цикл свободен обрабатывать клики, перерисовывать интерфейс и выполнять другой код, ровно как это описано в главе про событийный цикл. `await` приостанавливает одну конкретную функцию, а не замораживает вкладку.

## Ошибки ловятся обычным try/catch

Раз `await` разворачивает промис в обычное значение, отклонённый промис ловится там же, где ловят любую другую ошибку: в `try/catch`. Дополнительный синтаксис не нужен:

```js
async function payOrder(orderId) {
  try {
    const result = await chargeCard(orderId);
    return result.status;
  } catch (error) {
    return 'ошибка оплаты';
  } finally {
    console.log('попытка завершена');
  }
}
```

Если `chargeCard` вернёт отклонённый промис (например, банк отказал в списании рублей), исполнение прыгнет в `catch`, минуя оставшиеся строки `try`, точно так же, как это происходит с обычным синхронным исключением. `finally` при этом сработает всегда, независимо от того, прошла оплата успешно или нет.

Гранулярность обработки ошибок при этом остаётся такой же гибкой, как в цепочке `then`. Один общий `try/catch` вокруг нескольких `await` ловит ошибку любого из шагов, а если нужно по-разному реагировать на отказ каждого конкретного запроса, можно обернуть в собственный `try/catch` только один `await`, оставив остальные снаружи. Выбор между одним общим блоком и несколькими маленькими это вопрос того, насколько по-разному нужно обрабатывать разные шаги, а не ограничение синтаксиса.

## Шаги идут в том порядке, что и строки

Из первых двух правил складывается главное практическое следствие: последовательность независимых `await` читается как обычный синхронный сценарий, шаг за шагом:

```js
async function loadDashboard(userId) {
  const user = await getUser(userId);
  const orders = await getOrders(user.id);
  const bonus = await getBonusBalance(user.id);
  return { user, orders, bonus };
}
```

Каждая строка ждёт своего промиса, затем передаёт развёрнутое значение дальше, и весь сценарий для личного кабинета на Госуслугах или в интернет-банке Сбера выглядит как обычный расчёт сверху вниз, а не как набор вложенных колбэков.

## await работает только внутри async-функции

Здесь кроется первая неожиданность. `await` нельзя использовать в обычной функции: движок сочтёт это синтаксической ошибкой ещё до запуска кода, потому что `await` имеет смысл только там, где функция умеет приостанавливаться и возвращать промис. Из этого правила есть единственное исключение: в ES-модулях допускается так называемый top-level await, `await` прямо на верхнем уровне файла, вне какой-либо функции. Это узкий частный случай, и для обычных функций правило остаётся строгим: пометили `async`, только тогда внутри разрешён `await`.

Вторая неожиданность уже прозвучала выше, но стоит подчеркнуть отдельно: `async`-функция оборачивает результат в промис всегда, даже когда внутри неё вообще нет `await`. Функция без единой асинхронной операции всё равно возвращает промис, и вызывающему коду придётся развернуть его через `await` или `then`, других способов достать значение не существует.

<Callout type="warning" title="Значение прячется в промисе, даже если ждать было нечего">
Если внутри async-функции нет ни одного await, это не превращает её в синхронную. Возвращаемое значение всё равно уедет внутрь промиса, и снаружи его нужно разворачивать так же, как любой другой асинхронный результат.
</Callout>

## Частые ошибки

Первая ошибка: забытый `await`. Переменная получает не значение, а промис, и дальнейшие вычисления с ней ломаются молча:

```js
async function getPrice(id) {
  return db.findPrice(id);
}

async function getDiscountedPrice(id) {
  const price = getPrice(id);
  return price * 0.9;
}
```

`price` здесь это промис, а не число, потому что вызов `getPrice(id)` не дожидается разворачивания. Умножение промиса на 0.9 не бросит исключение, оно тихо даст `NaN`, и баг всплывёт где-то дальше, там, где к `NaN` меньше всего готовы. Лечится добавлением `await` перед вызовом: `const price = await getPrice(id);`.

Вторая ошибка: `await` в функции без `async`:

```js
function getDiscountedPrice(id) {
  const price = await getPrice(id);
  return price * 0.9;
}
```

Такой код не запустится вовсе: движок откажется его разбирать с синтаксической ошибкой, потому что ключевое слово `await` встретилось вне `async`-функции. Фикс единственный: дописать `async` перед `function`.

Третья ошибка тоньше первых двух: последовательные `await` там, где шаги друг от друга не зависят. В примере с личным кабинетом заказы и бонусный баланс запрашиваются независимо, но код всё равно ждёт заказы, прежде чем начать запрос баланса, хотя оба запроса можно было отправить одновременно. Это не баг в смысле неправильного результата, но потерянное время: два независимых ожидания складываются вместо того, чтобы идти параллельно. Как запускать несколько промисов разом и ждать их вместе, разбирается в следующей главе про `Promise.all`.

<Detail summary="Что происходит с await на уровне событийного цикла">
Строка после await не выполняется мгновенно после разрешения промиса: продолжение функции становится микрозадачей, точно такой же, как колбэк в then. Это тот же механизм, что описан в главе «Микрозадачи и макрозадачи»: async/await не создаёт отдельную очередь, а лишь прячет работу с промисами и микрозадачами за более короткой записью.
</Detail>

## Частые вопросы на практике

Если результат `async`-функции нужно передать дальше без `await`, например в обработчике события, который сам не может быть `async`, помогает обычный `then` или `catch`, ведь `async`-функция остаётся промисом снаружи в любом случае. Смешивать `await` и `then` внутри одной функции можно, но обычно код чище, если выбрать один стиль на весь сценарий.
