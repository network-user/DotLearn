# Три ошибки, которые проходят ревью

Борис отправляет пул-реквест поздно вечером. Код короткий, тесты зелёные, ревьюер бегло смотрит диффы и ставит одобрение. Через два дня саппорт Ozon получает три независимые жалобы: карточка товара иногда показывает пустое имя продавца, поиск по каталогу временами откатывается к результатам пятисекундной давности, а форма оформления заказа один раз просто «съела» заявку без единого сообщения об ошибке. Борис открывает консоль и не находит ни одного красного исключения: программа не падает, она молча делает не то.

В этом и состоит особенность асинхронных ошибок. Синхронный баг обычно кричит: неверный тип, обращение к несуществующему полю, деление логики без проверки, и движок тут же бросает исключение прямо в точке вызова. Асинхронный баг чаще молчит. Он либо тихо подсовывает не то значение, либо решает вопрос порядка не так, как рассчитывал автор кода, либо теряет ошибку где-то в глубине промиса, откуда её никто не поймал. Все три бага у Бориса растут из одного корня: код написан так, будто `await`, порядок ответов и `reject` всегда ведут себя предсказуемо, а на практике это не так.

## Почему асинхронный баг проявляется не там, где спрятан

Ключевая идея, без которой три следующих раздела будут просто списком рецептов, звучит так: вызов асинхронной операции и получение её результата это два разных момента времени, и между ними может произойти что угодно. Может прийти второй вызов той же функции. Может истечь время ожидания у одного запроса раньше другого, хотя ушёл он позже. Может случиться ошибка сети, а обработчика для неё просто не существует. Код, который написан в расчёте на синхронный порядок «позвал, тут же получил», регулярно ошибается именно в проде: там сеть медленнее, пользователь печатает быстрее, а нагрузка выше, чем на ноутбуке разработчика.

Нагляднее всего это видно на поиске по каталогу. Пользователь печатает быстро, и на каждое нажатие клавиши уходит новый запрос к серверу. Запросы летят в сеть в одном порядке, а возвращаются в произвольном: короткий запрос может обогнать длинный, отправленный раньше него.

Запрос «кошка» отправлен вторым, но обслуживается быстрее и рисует результат первым. Следом приходит ответ на «кот», отправленный раньше и висевший в сети дольше, и молча перезаписывает уже нарисованный актуальный результат устаревшим. Пользователь видит на экране результаты поиска по слову, которое он уже стёр. Это и есть гонка, race condition: порядок отправки не совпадает с порядком получения, а код полагается на первое вместо второго.

## Ошибка первая: забытый await

Первый баг Бориса выглядел безобидно. Функция `loadUser` объявлена как `async` и внутри честно дожидается сети:

```js
async function loadUser(id) {
  const response = await fetch('/api/users/' + id);
  return response.json();
}

function renderProfile(id) {
  const user = loadUser(id);
  if (user) {
    console.log(user.name);
  }
}
```

Проблема в `renderProfile`: он вызывает `loadUser` без `await`. Любая `async`-функция при вызове мгновенно возвращает промис, а не итоговое значение, и именно этот промис попадает в переменную `user`. Дальше начинается тихая ложь. Промис это обычный объект, а любой объект в логическом контексте истинен, поэтому проверка `if (user)` срабатывает всегда, даже если сервер вернёт ошибку. Обращение `user.name` тоже не упадёт: у промиса просто нет поля `name`, и наружу уходит `undefined` вместо реального имени. Если бы код пытался что-то посчитать через `user`, например сложить числовое поле, результатом стал бы `NaN`, и снова без единого исключения.

Чинится это одной пропущенной частицей: раз `renderProfile` использует результат асинхронного вызова, ему самому нужно стать `async` и дождаться значения через `await`.

```js
async function renderProfile(id) {
  const user = await loadUser(id);
  if (user) {
    console.log(user.name);
  }
}
```

У забытого `await` есть более коварный родственник: висящий промис, floating promise. Это ситуация, когда асинхронный вызов сделан вообще без `await`, без `then` и без `catch`, а его результат никому не нужен, но за поведением всё равно важно проследить.

```js
async function autosaveDraft(note) {
  writeToServer(note);
  console.log('черновик отправлен');
}
```

Строка `console.log` выполнится сразу, не дожидаясь ответа сервера. Если `writeToServer` завершится отклонением промиса, эту ошибку никто не поймает: ни `try`, обёрнутый вокруг вызова `autosaveDraft`, ни внешний код. Черновик тихо не сохранится, а лог скажет, что всё отправлено.

## Ошибка вторая: гонка устаревших ответов

Вернёмся к гонке ответов с рисунка выше. Наивная реализация поиска обрабатывает каждый ответ ровно так, как он приходит:

```js
async function search(query) {
  const response = await fetch('/api/search?q=' + query);
  const data = await response.json();
  renderResults(data);
}
```

Каждое нажатие клавиши вызывает `search` заново. Если пользователь напечатал «кот» и тут же дописал «кошка», уйдут два запроса. Функция понятия не имеет, что где-то параллельно уже летит более новый запрос, поэтому она всегда рисует то, что получила, независимо от того, устарел ли этот ответ. Итог в точности как на рисунке: медленный ответ на старый запрос приходит последним и переписывает свежий результат.

Лечится это не отменой запросов, а простой меткой актуальности. Заводим счётчик, запоминаем номер запроса на момент отправки и перед отрисовкой сверяем его с текущим:

```js
let latestQuery = 0;

async function search(query) {
  const requestId = ++latestQuery;
  const response = await fetch('/api/search?q=' + query);
  const data = await response.json();
  if (requestId !== latestQuery) return;
  renderResults(data);
}
```

Если к моменту ответа `latestQuery` уже сдвинулся дальше, значит пользователь успел напечатать что-то ещё, и текущий ответ устарел: функция просто выходит, ничего не рисуя. Тот же приём применяют не только к счётчику, но и к отмене запроса через `AbortController`: разница в том, отбрасываем мы устаревший ответ молча или обрываем его на сети, но идея одна, у актуального запроса должен быть способ победить более старый.

## Ошибка третья: необработанный reject

Третий баг Бориса, пропавшая заявка на оформление заказа, тоже не бросал исключений, потому что их некому было бросать:

```js
async function submitOrder(order) {
  const response = await fetch('/api/orders', {
    method: 'POST',
    body: JSON.stringify(order),
  });
  return response.json();
}

submitOrder(order);
```

Если `fetch` отклонится, скажем, из-за обрыва сети, или сервер вернёт ошибку, которую код решает превратить в `throw`, промис функции `submitOrder` окажется отклонён. Но вызов `submitOrder(order)` в конце ничего не ждёт и ни к чему не привязан. Ошибка не падает синхронно, потому что упасть ей уже некуда: движок фиксирует её как unhandled rejection и сообщает об этом отдельным событием, которое почти никто не слушает. Пользователь просто не получает ни подтверждения заказа, ни сообщения об ошибке.

Правило простое: у любой цепочки промисов должен быть явный обработчик отказа, будь то `catch` в конце цепочки или `try/catch` вокруг `await`.

```js
async function submitOrder(order) {
  try {
    const response = await fetch('/api/orders', {
      method: 'POST',
      body: JSON.stringify(order),
    });
    return await response.json();
  } catch (error) {
    showErrorToast('Не удалось оформить заказ');
    throw error;
  }
}
```

Обратите внимание: `catch` здесь не глотает ошибку молча, а показывает пользователю понятное сообщение и пробрасывает её дальше через `throw`, чтобы вызывающий код тоже знал об отказе. Проглоченная без следа ошибка часто ничем не лучше необработанной: пользователь так и не понимает, что заказ не прошёл.

<Callout type="warning" title="try/catch снаружи не спасает от висящего промиса">
Если асинхронный вызов сделан без await внутри другой функции, внешний try/catch вокруг этой функции ошибку из него не поймает: к моменту, когда отклонение случится, обёртка уже давно отработала и вышла. Ловить отказ нужно там же, где сделан асинхронный вызов, а не надеяться на код снаружи.
</Callout>

## На что ещё обратить внимание

Все три ошибки объединяет то, что JavaScript не считает их достаточно серьёзными, чтобы немедленно уронить программу. Промис всегда истинен как объект, поэтому забытый `await` не даёт ни одной подсказки в виде исключения, только тихо неверную логику дальше по коду. Гонка ответов не нарушает ни одного правила языка, оба запроса и оба ответа совершенно легальны, просто порядок оказался не тем, что ожидал автор. А необработанный reject существует в собственном временном окне: событие `unhandledrejection` происходит уже после того, как стек вызовов, который мог бы его поймать, давно опустел.

Из этого следует практическая привычка при чтении и написании асинхронного кода: на каждый `async`-вызов проверяйте, ждёт ли его `await` там, где нужен результат; на каждый обработчик, который может получить несколько ответов подряд, проверяйте способ отличить свежий ответ от устаревшего; и на каждую цепочку промисов проверяйте, есть ли у неё явный `catch`.

<Detail summary="unhandledrejection: последний шанс заметить проблему">
Когда отклонённый промис так и остаётся без catch, среда выполнения генерирует глобальное событие unhandledrejection с самим промисом и причиной отказа. В браузере на него можно подписаться через window, в Node.js через process. Логирование этого события не заменяет нормальную обработку ошибок в коде, но часто оказывается единственным способом заметить проблему вроде истории Бориса: без него отказ прошёл бы совершенно незамеченным, разве что по жалобам пользователей.
</Detail>
