# Номерок, который выдают сразу, а кофе приносят потом

Вера заходит в кофейню на Баумана в Казани и просит капучино. Бариста не может налить кофе мгновенно: кофемашине нужно время. Вместо кофе Вера сразу получает бумажный номерок с цифрой. Номерок не сам кофе, а обещание, что через несколько минут либо принесут капучино, либо выйдут и скажут, что молоко закончилось. Пока номерок у Веры в руке, ничего ещё не решено: кофе не готов и отказа тоже нет. Вера отдаёт номерок Борису и уходит по делам. Борис сам не заказывал ничего, но именно он в итоге заберёт капучино или услышит про закончившееся молоко: важен номерок, а не то, кто стоял в очереди.

Promise в JavaScript устроен так же. Функция, которая начинает асинхронную работу, не может отдать результат мгновенно, поэтому она сразу возвращает объект-заглушку: он ещё не хранит готовое значение, но гарантированно рано или поздно осядет либо в успех, либо в ошибку. Этот объект можно вернуть из функции, передать в другую переменную, отдать другому куску кода, а обработчик результата навесить в любой момент, даже если к тому времени всё уже случилось. Даже потерянный номерок из аналогии имеет смысл: пока Вера не забрала капучино, кто угодно с номерком в руке способен спросить бариста о результате, и ответ будет одним и тем же для всех, кто спросит. Так же ведёт себя и Promise: сколько бы кода ни держало ссылку на один и тот же объект, итог у всех окажется идентичным.

К концу главы вы будете различать три состояния Promise, создавать его через `new Promise` и забирать значение через `then`, `catch` и `finally`.

## Три состояния и дорога с односторонним движением

У Promise ровно три состояния. Пока результат не готов, объект висит в состоянии `pending`, ожидание. Как только асинхронная операция завершилась успехом, объект переходит в `fulfilled`, выполнено, и внутри у него лежит готовое значение. Если операция завершилась с ошибкой, объект переходит в `rejected`, отклонено, и внутри лежит причина ошибки.

Здесь и прячется главное правило, которое стоит запомнить крепче остальных: переход из `pending` возможен только один раз и только в одну сторону. Как только Promise осел в `fulfilled` или `rejected`, говорят, что он settled, определился, и обратно в `pending` он уже не вернётся, а из `fulfilled` в `rejected` не перепрыгнет. Дороги назад нет и переключиться со случившегося успеха на ошибку тоже нельзя.

Обратите внимание, что на рисунке из `pending` выходят только две стрелки и обе указывают вниз. Третьей стрелки, ведущей назад к `pending`, не существует в принципе. Именно эта необратимость делает Promise надёжным: неважно, сколько раз вы посмотрите на уже решённый объект, ответ не изменится.

Похожую трёхступенчатую модель легко узнать в обычном заказе на Ozon. Пока заказ собирают на складе, он висит в статусе «в обработке», это и есть `pending`. Если сборка прошла успешно и курьер забрал коробку, статус меняется на «передан в доставку», аналог `fulfilled`, и обратно в «в обработке» заказ уже не откатится. Если же товара не оказалось на складе, заказ переходит в статус «отменён», аналог `rejected`, и это тоже финал, из которого нет пути назад к ожиданию. Три состояния Promise устроены ровно так же строго: одна развилка, два финала, и оба необратимы.

## Создание Promise: executor запускается сразу

Promise создают через конструктор `new Promise`, которому передают функцию с двумя параметрами, `resolve` и `reject`. Эту функцию называют executor, и важно понимать: executor выполняется синхронно и немедленно, в тот самый момент, когда движок доходит до строки `new Promise`. Асинхронной здесь может быть только работа внутри executor, например обёрнутый `setTimeout`:

```js
function makeCoffee(name) {
  return new Promise((resolve, reject) => {
    console.log('заказ принят: ' + name);
    setTimeout(() => {
      if (name === 'капучино') {
        resolve('капучино готов');
      } else {
        reject(new Error('нет такого напитка'));
      }
    }, 2000);
  });
}
```

Строка `console.log` внутри executor напечатается сразу же при вызове `makeCoffee`, ещё до того, как функция вернёт хоть что-то наружу. А вот вызов `resolve` или `reject` произойдёт только через две секунды, когда сработает таймер. До этого момента возвращённый Promise висит в `pending`. Вызов `resolve('капучино готов')` переводит его в `fulfilled` со значением `'капучино готов'` внутри, а вызов `reject` перевёл бы его в `rejected` с объектом ошибки внутри.

## Получение значения: then, catch и finally

Сам по себе Promise ничего не печатает и никому ничего не сообщает: чтобы забрать значение, к нему подвешивают обработчик через метод `then`. Он принимает функцию, которая получит значение из `fulfilled`, и вызовет её именно тогда, когда Promise осядет:

```js
makeCoffee('капучино').then((result) => {
  console.log(result);
});
```

Строка `капучино готов` появится в консоли через две секунды, ровно тогда, когда исполнится `resolve` внутри executor. Если бы вы вызвали `then` не сразу, а через десять секунд после создания Promise, результат был бы тем же самым: к этому моменту объект уже давно определился в `fulfilled`, и `then` просто заберёт готовое значение.

Ошибку так же ловят не мимоходом, а отдельным методом `catch`, который получает причину из `rejected`:

```js
makeCoffee('латте')
  .then((result) => console.log(result))
  .catch((error) => console.log('ошибка: ' + error.message));
```

Здесь `resolve` не вызовется никогда, зато вызовется `reject(new Error('нет такого напитка'))`, и колбэк `then` попросту пропускается. Управление уходит в `catch`, и в консоль печатается `ошибка: нет такого напитка`. Так реализуется главное удобство Promise по сравнению со старыми колбэками: успех и ошибка разведены по разным методам, а не свалены в один и тот же параметр функции.

Третий метод, `finally`, выполнится в любом случае, независимо от того, чем закончилось дело, успехом или отказом:

```js
makeCoffee('капучино')
  .then((result) => console.log(result))
  .catch((error) => console.log('ошибка: ' + error.message))
  .finally(() => console.log('чек закрыт'));
```

`finally` не получает ни значения, ни причины ошибки: он не для результата, а для действий по завершению, которые нужны в любом случае, например скрыть индикатор загрузки на странице или закрыть соединение. Строка `чек закрыт` появится и при успешном капучино, и при закончившемся молоке.

Иногда готовый Promise нужен без всякого executor и без ожидания, например в тестах или когда значение и так уже под рукой. Для этого есть короткие фабрики `Promise.resolve` и `Promise.reject`, которые сразу создают объект в нужном settled-состоянии:

```js
const already = Promise.resolve('значение уже есть');
const brokenAlready = Promise.reject(new Error('сразу отказ'));

already.then((value) => console.log(value));
brokenAlready.catch((error) => console.log(error.message));
```

Разницы в поведении между таким Promise и Promise, который дошёл до `resolve` через таймер, нет никакой: методы `then` и `catch` работают с ним точно так же. Разница только в том, что `pending` здесь длится нулевое время, объект рождается сразу решённым.

## Оседает один раз, и это буквально

Возьмём executor, который по ошибке вызывает `resolve` дважды:

```js
const oddPromise = new Promise((resolve) => {
  resolve('первое значение');
  resolve('второе значение');
});

oddPromise.then((value) => console.log(value));
```

В консоль попадёт только `первое значение`. Второй вызов `resolve` не бросает исключение и не заменяет уже установленное значение: он просто молча игнорируется, потому что Promise уже settled и повторное решение никого не интересует. Это правило работает и для смешанных вызовов: если сначала вызвать `resolve`, а следом `reject`, второй вызов тоже пропадёт без следа, и Promise так и останется в `fulfilled`.

Вторая неожиданность касается времени, а не состояния. Даже если Promise уже давно `fulfilled` к моменту вызова `then`, колбэк всё равно не выполнится в ту же строку кода. Он встанет в очередь микрозадач и сработает чуть позже, после текущего синхронного кода:

```js
const ready = Promise.resolve('готово');
ready.then((value) => console.log(value));
console.log('это выполнится первым');
```

Печатается сначала `это выполнится первым`, а `готово` только следом. Подробно про очередь микрозадач и почему так происходит рассказано в отдельной главе про микрозадачи, здесь достаточно запомнить факт: колбэк `then` асинхронный всегда, даже у уже решённого Promise. На практике это означает, что код сразу после строки с `then` не может рассчитывать на то, что значение уже доступно: оно появится чуть позже, а весь код, которому это значение нужно, обязан жить внутри самого `then` или дальше по цепочке.

<Callout type="tip" title="pending значит просто не готово">
Не путайте `pending` с ошибкой или зависанием. Это нормальное промежуточное состояние, в котором Promise проводит какое-то время между созданием и решением. Само по себе `pending` ничего плохого не означает.
</Callout>

## Частые ошибки

Первая ошибка: путать создание Promise с его потреблением. Начинающие разработчики оборачивают в `new Promise` то, что и так уже возвращает Promise, например встроенный `fetch`. Получается лишний слой:

```js
function getUserBroken(id) {
  return new Promise((resolve, reject) => {
    fetch('/api/users/' + id)
      .then((response) => resolve(response))
      .catch((error) => reject(error));
  });
}
```

Эта функция работает, но полностью бессмысленна: `fetch` и так возвращает Promise, а лишняя обёртка только добавляет шанс забыть про `reject` в каком-нибудь из веток. Оборачивать в `new Promise` стоит только по-настоящему колбэковый код вроде таймеров или старых API с колбэками. Если функция уже возвращает Promise, к ней сразу применяют `then`, а не заворачивают её ещё раз.

Вторая ошибка: считать executor асинхронным целиком. Executor запускается синхронно и сразу, в момент вызова `new Promise`. Асинхронна не сама функция, а операция внутри неё, вроде таймера или сетевого запроса. Код до вызова `setTimeout` внутри executor выполнится ещё до того, как строка с `new Promise` вообще закончится.

Третья ошибка, самая частая на практике: забыть `catch`. Если Promise перешёл в `rejected`, а обработчика ошибки нигде нет, ошибка не пропадает бесследно, а превращается в необработанное отклонение (unhandled rejection), которое браузер печатает в консоль как предупреждение и которое легко пропустить в реальном приложении. Каждую цепочку, которая может закончиться отказом, стоит завершать методом `catch`.

<Detail summary="Чем Promise лучше обычного колбэка">
До Promise асинхронный результат отдавали через колбэк, который передавали параметром и который сама функция вызывала когда захочет. У такого подхода было неудобство: успех и ошибку часто мешали в один и тот же колбэк, а несколько вложенных асинхронных шагов превращались в лестницу из колбэков внутри колбэков. Promise решает это тем, что результат становится полноценным значением: его можно вернуть, передать дальше, подождать через `then` из любого места кода, а ошибку обработать в одном едином `catch`. Как соединять несколько Promise подряд в цепочку, разберём в следующей главе.
</Detail>
