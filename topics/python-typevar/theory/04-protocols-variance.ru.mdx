# Protocol и вариантность: типизация по форме, а не по родословной

Python всегда любил утиную типизацию: функции всё равно, какого класса объект, лишь бы у него был нужный метод. Но обычная аннотация это игнорирует. Если Вы напишете `def total(items: SupportsRead)`, придётся либо заводить общий базовый класс и заставлять всех от него наследоваться, либо смириться с тем, что в аннотации стоит конкретный класс - и тогда любой посторонний объект с тем же методом анализатор отвергнет, хотя в рантайме он бы прекрасно подошёл. Статическая проверка вступает в конфликт с динамической природой языка.

`Protocol` примиряет их: он описывает тип по **форме** - по набору методов и атрибутов, - не требуя наследования. А когда у вас появляются обобщённые контейнеры, всплывает второй тонкий вопрос: если `Cat` - подтип `Animal`, то является ли `list[Cat]` подтипом `list[Animal]`? Ответ - вариантность - не всегда «да», и непонимание этого порождает целый класс ошибок. К концу раздела Вы будете писать протоколы для структурной типизации и понимать, когда контейнер одного типа можно подставить вместо контейнера другого.

## Ментальная модель: совпадение по форме

Протокол - это объявленный набор требований: «тип подходит, если у него есть метод `read() -> str`». Никакого наследования: класс соответствует протоколу автоматически, как только у него есть всё перечисленное. Это называется структурной типизацией - в противоположность номинальной, где важна родословная (кто от кого наследует).

В этом сила протоколов: функция, принимающая `Readable`, согласится на любой объект с методом `read`, - файл, сетевой ответ, обёртку над базой, заглушку в тесте, - и анализатор это одобрит, ничего не зная об их иерархии. Вы описываете нужную форму один раз, и всё, что в неё укладывается, подходит.

## Объявить протокол

Протокол - это класс, унаследованный от `Protocol`, в котором методы оставлены без тела (`...`). Он задаёт форму, не реализацию.

```python
from typing import Protocol

class Readable(Protocol):
    def read(self) -> str: ...

def load(source: Readable) -> int:
    return len(source.read())
```

Теперь `load` принимает что угодно с методом `read`, возвращающим строку. Класс-кандидат не упоминает `Readable` нигде - он просто имеет нужный метод, и этого достаточно.

```python
class Report:
    def read(self) -> str:
        return "данные отчёта"
```

`load(Report())` проходит проверку: `Report` структурно соответствует `Readable`, хотя про него и не подозревает. Если бы у `Report` не было `read` или тот возвращал бы не строку, анализатор отметил бы несоответствие - на этапе проверки, а не в рантайме.

## Вариантность: когда list[Cat] не подходит вместо list[Animal]

Перейдём ко второй теме. Пусть `Cat` - подтип `Animal`. Кажется логичным, что `list[Cat]` можно подставить туда, где ждут `list[Animal]`. Но это **небезопасно**, и анализатор такую подстановку запрещает.

```python
class Animal: ...
class Cat(Animal): ...
class Dog(Animal): ...

def add_dog(animals: list[Animal]) -> None:
    animals.append(Dog())
```

Если бы `list[Cat]` сошёл за `list[Animal]`, функция `add_dog` дописала бы `Dog` в список котов - и потом тот, кто держит этот список как `list[Cat]`, наткнулся бы на собаку. Изменяемый контейнер, в который можно и читать, и писать, обязан быть **инвариантным**: `list[Cat]` и `list[Animal]` не взаимозаменяемы ни в одну сторону.

Иное дело - контейнер только для чтения. Из него нельзя ничего записать, поэтому опасность из примера не возникает. Такой контейнер **ковариантен**: `Sequence[Cat]` подходит туда, где ждут `Sequence[Animal]`, потому что любой кот - это животное, а добавить чужого в последовательность нельзя. Отсюда практическое правило: если функции достаточно читать из коллекции, аннотируйте параметр как `Sequence[...]` или `Iterable[...]`, а не `list[...]` - так Вы примете больше корректных аргументов.

## Потрогайте структурную типизацию руками

Запустите код: `load` работает и с `Report`, и с любым объектом, у которого есть `read`, - наследование нигде не нужно. Теперь добавьте свой класс с методом `read`, возвращающим строку, передайте его экземпляр в `load` и запустите снова: он подойдёт автоматически. А уберите `read` у одного из классов - в рантайме получите `AttributeError`, и ровно это несоответствие протоколу анализатор отловил бы до запуска.

## Краевой случай: isinstance с протоколом требует @runtime_checkable

Это вас удивит: по умолчанию протокол нельзя использовать в `isinstance`. Протоколы - инструмент статической проверки, и `isinstance(obj, Readable)` без подготовки бросит `TypeError`. Чтобы разрешить проверку в рантайме, протокол помечают декоратором `@runtime_checkable` - но и тогда проверяется лишь **наличие** методов, а не их сигнатуры.

```python
from typing import Protocol, runtime_checkable

@runtime_checkable
class Readable(Protocol):
    def read(self) -> str: ...

class Report:
    def read(self) -> str:
        return "x"

print(isinstance(Report(), Readable))
```

Печатается `True`: декоратор включил проверку, и `Report` прошёл по наличию метода `read`. Но имейте в виду: `runtime_checkable` смотрит только на имена методов. Он подтвердит соответствие, даже если `read` принимает не те аргументы или возвращает не строку, - полную проверку сигнатур делает только статический анализатор.

## Типичные ошибки

- **Заставлять классы наследоваться от протокола.** Смысл `Protocol` в том, что наследование не нужно: соответствие определяется наличием методов. Наследовать от протокола можно для самопроверки, но это не требование.
- **Аннотировать параметр как `list[...]`, когда хватило бы `Sequence[...]`.** `list` инвариантен и примет только точное совпадение; `Sequence` ковариантен и примет `Sequence` подтипа. Если функция лишь читает - берите `Sequence`/`Iterable`.
- **Ждать, что `list[Cat]` подойдёт вместо `list[Animal]`.** Изменяемый список инвариантен: подстановка запрещена в обе стороны, потому что иначе в список одного типа можно было бы дописать чужой объект.
- **Вызывать `isinstance` с обычным протоколом.** Без `@runtime_checkable` это `TypeError`. А с ним проверяется только наличие методов, не их сигнатуры - не полагайтесь на такую проверку как на гарантию типа.

<Detail summary="Почему dict-ключи инвариантны, а возвращаемые значения ковариантны">

Вариантность - не свойство «типа вообще», а свойство **позиции**, в которой параметр используется. Грубое правило: позиции, откуда значение только читают (выход функции, элементы `Sequence`), ковариантны - туда можно подставить подтип. Позиции, куда значение только пишут (вход-приёмник), контравариантны - туда можно подставить надтип. Позиции, где значение и читают, и пишут (элементы изменяемого `list`, ключи `dict`), инвариантны - подходит только точное совпадение.

Поэтому `Callable[[Animal], Cat]` ковариантен по результату и контравариантен по аргументу: функция, возвращающая `Cat`, годится там, где ждут возврат `Animal`, а функция, принимающая `Animal`, годится там, где ждут приём `Cat`. Эта «перевёрнутость» аргументов поначалу сбивает с толку, но следует из того же принципа безопасной подстановки: вызывающий передаст `Cat`, а функция, готовая принять любое `Animal`, с котом справится. Для своих обобщённых классов вариантность по позиции `TypeVar` задают через `covariant=True` / `contravariant=True`.
