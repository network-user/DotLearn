You write a function that takes the first element of a list and want your IDE and type checker to suggest the correct result type. With a list of numbers you get a number; with a list of strings you get a string. But as soon as you make the function generic, the type is "lost": the output becomes a faceless `object` with neither string nor numeric methods, and hints disappear exactly where you need them most. This topic is about describing generic code so it works with any type while still preserving information about which type is inside.

**What you'll learn to do.** Read and write annotations for variables, arguments, and return values; link input and output types through `TypeVar`; build your own generic container classes; narrow generics through `bound` and constraints when a type needs certain operations; describe object shape requirements through `Protocol`; understand variance so you know when one container type can be substituted for another.

**Route through the topic.** Four sections, each with a Python sandbox right in the browser:

1. **Type annotations** - record in the signature what a function expects and returns, and see how an external analyzer catches mismatches before run time.
2. **`TypeVar` and `Generic`** - tie input and output types with one type variable and build your own generic containers.
3. **`bound` and constraints** - narrow the set of allowed types so you can compare, add, and call the methods you need inside a function.
4. **`Protocol` and variance** - type by object shape without inheritance and understand when a `list` of one type can stand in for a `list` of another.

**What you need first.** Nothing to install: Python runs right in the browser. Confidence with classes and inheritance from the OOP topic helps - everything about annotations and generics we'll cover from scratch along the way.
