# Аннотации типов: подсказки, которые читает не только человек

Вы открываете чужую функцию `def process(data):` и сразу спотыкаетесь: что такое `data` - список, словарь, объект с какими-то методами? Чтобы это понять, приходится читать тело целиком, искать вызовы, иногда запускать отладчик. Имя аргумента подсказывает намерение, но ничего не гарантирует: туда можно передать что угодно, и Python промолчит ровно до того момента, пока внутри не упадёт `AttributeError` где-нибудь на третьем уровне вложенности.

Аннотации типов - это способ записать прямо в сигнатуре, что функция ожидает и что возвращает. К концу раздела Вы научитесь читать и писать аннотации для переменных, аргументов и возвращаемых значений, понимать, почему интерпретатор их не проверяет, и пользоваться внешним type checker, который ловит несоответствия ещё до запуска.

## Ментальная модель: аннотация - это контракт, а не проверка

Главное, что нужно усвоить про аннотации в Python: интерпретатор их **не исполняет**. Запись `age: int` не заставляет Python отвергнуть строку - она лишь объявляет намерение. Реальную проверку выполняет отдельный инструмент - статический анализатор (mypy, pyright, тот, что встроен в вашу IDE), - который читает код, не запуская его, и сверяет фактическое использование с объявленным контрактом.

Из этого следует неочевидное: программа с неверными аннотациями всё равно запустится и может отработать без ошибок. Аннотации - это документация, которую умеет читать машина, а не валидатор времени выполнения. Их ценность в том, что несоответствие находится раньше: не в проде на реальных данных, а в редакторе, пока Вы печатаете.

## Базовый синтаксис: переменные, аргументы, возврат

Аннотация переменной - это имя, двоеточие и тип. Аннотация аргумента живёт в той же форме внутри скобок, а тип возвращаемого значения пишется после стрелки `->`.

```python
name: str = "Ана"
age: int = 34

def greet(person: str, times: int) -> str:
    return f"Привет, {person}! " * times
```

Здесь сигнатура сама себя документирует: `greet` принимает строку и целое, возвращает строку. Любой, кто вызовет `greet(34, "Ана")`, перепутав аргументы, получит претензию от анализатора - хотя сам Python такой вызов выполнит и вернёт абсурдный, но не падающий результат.

Функция без возврата аннотируется как `-> None`. Это не то же самое, что отсутствие аннотации: `-> None` явно говорит «ничего полезного не возвращаю», а пустая стрелка означает «тип не указан».

```python
def log(message: str) -> None:
    print(f"[log] {message}")
```

## Составные типы: контейнеры и Optional

Для контейнеров одного имени мало - надо сказать, что внутри. С Python 3.9 встроенные коллекции параметризуются прямо: `list[int]`, `dict[str, int]`, `tuple[int, str]`. Квадратные скобки здесь читаются как «список **из** int», «словарь **из** str в int».

```python
scores: list[int] = [10, 20, 30]
ages: dict[str, int] = {"Ана": 34, "Борис": 29}

def total(values: list[int]) -> int:
    return sum(values)
```

У `dict` два параметра - тип ключа и тип значения, поэтому `dict[str, int]` читается как «ключи-строки, значения-числа». У кортежа фиксированной длины каждый элемент аннотируется отдельно: `tuple[int, str]` - это пара, где первый элемент целый, второй строковый, и анализатор проследит, чтобы порядок не нарушился. Чем точнее параметры контейнера, тем больше ошибок ловится до запуска - например, попытка положить строку в `list[int]` или обратиться к несуществующему третьему элементу `tuple[int, str]`.

Когда значение может быть либо чем-то, либо `None`, это записывают через `| None` (или `Optional[T]` из модуля `typing` в более старом стиле). Запись `str | None` означает «строка или ничего» и заставляет анализатор требовать проверку на `None` перед использованием. Это не косметика: пока тип объявлен как `str | None`, анализатор не даст вызвать строковый метод напрямую и потребует сначала отсеять случай `None` - например, через `if result is not None`.

```python
def find_user(user_id: int) -> str | None:
    table = {1: "Ана", 2: "Борис"}
    return table.get(user_id)
```

<Callout type="tip">
`dict.get` возвращает `None`, если ключа нет, поэтому честная аннотация результата - `str | None`, а не `str`. Анализатор после этого не даст вызвать у результата строковый метод, пока Вы не отсеете `None`.
</Callout>

## Потрогайте типы руками

Аннотации не исчезают бесследно: интерпретатор складывает их в атрибут `__annotations__`, доступный во время выполнения. Запустите код ниже, чтобы увидеть, что туда попало. Затем передайте в `person` число вместо строки - программа всё равно отработает, потому что Python типы не проверяет. Это и есть «контракт без принуждения» наглядно.

## Краевой случай: аннотация не приводит тип

Это вас удивит: присваивание с аннотацией не конвертирует значение. Если Вы напишете `x: int = "5"`, никакого преобразования к числу не произойдёт - в `x` останется строка, а несоответствие заметит только анализатор.

```python
x: int = "5"
print(type(x))
print(x + "0")
```

Печатается `<class 'str'>` и `50`. Переменная `x` объявлена как `int`, но фактически держит строку `"5"`, и `+` сработал как конкатенация строк. Аннотация - это обещание, а не каст; за выполнение обещания отвечаете Вы и анализатор, а не интерпретатор.

## Типичные ошибки

- **Ждать, что Python проверит типы сам.** Не проверит. Без подключённого type checker (mypy, pyright, проверка в IDE) аннотации не дают никаких гарантий - только документацию. Запускайте анализатор в CI, иначе контракт некому соблюдать.
- **Аннотировать `list` вместо `list[int]`.** Голое `list` означает «список чего угодно», и анализатор не сможет поймать, что Вы кладёте туда строки. Указывайте параметр контейнера, когда он известен.
- **Забывать про `None` в возврате.** Функция, которая иногда возвращает значение, а иногда `None` (через `return` без аргумента или `dict.get`), должна объявлять `T | None`. Иначе анализатор поверит, что `None` невозможен, и пропустит ошибку.
- **Путать `-> None` и отсутствие стрелки.** `-> None` - это явный контракт «ничего не возвращаю», а отсутствие аннотации - «тип не указан». Для функций-процедур пишите `-> None` явно.

<Detail summary="Откуда взялся синтаксис аннотаций и почему он необязателен">

Аннотации функций появились в Python 3.0 (PEP 3107) как чисто синтаксическая возможность: язык позволял писать произвольные выражения после двоеточия и стрелки, но не придавал им смысла. Только в 3.5 (PEP 484) их договорились использовать для типов и появился модуль `typing`. Аннотации переменных пришли в 3.6 (PEP 526).

Принципиальное решение авторов: типизация в Python остаётся **постепенной** (gradual typing). Вы можете не аннотировать ничего, аннотировать часть кода или всё - и любой из этих вариантов рабочий. Непроаннотированный код анализатор считает имеющим «динамический» тип и не предъявляет к нему претензий. Это позволяет добавлять типы в существующий проект постепенно, файл за файлом, не переписывая всё сразу.

</Detail>
