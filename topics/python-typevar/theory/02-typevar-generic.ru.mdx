# TypeVar: связать тип входа с типом выхода

Напишите функцию, которая берёт первый элемент списка. С аннотациями из прошлого раздела у Вас два плохих варианта. Можно объявить вход как `list[int]` - но тогда функция работает только со списками целых, а для списка строк придётся писать копию. Можно объявить вход как `list[object]` и возврат как `object` - функция станет универсальной, но анализатор «забудет», что внутри: вызвав её на `list[str]`, Вы получите результат типа `object`, у которого нет строковых методов. Универсальность куплена ценой потери информации о типе.

Не хватает способа сказать: «какой тип лежит в списке на входе, такой же выйдет на выходе» - не называя конкретный тип, но связав вход и выход в одну переменную. Это и делает `TypeVar`. К концу раздела Вы научитесь писать обобщённые функции, в которых тип результата выводится из типа аргумента, и собственные обобщённые классы-контейнеры через `Generic`.

## Ментальная модель: TypeVar - это переменная для типа

Обычная переменная держит значение, которое станет известно при вызове. `TypeVar` держит **тип**, который станет известен на месте вызова. Вы объявляете `T = TypeVar("T")` и используете `T` в сигнатуре несколько раз; анализатор понимает, что во всех этих местах должен оказаться один и тот же тип, и подставляет его, когда видит конкретный вызов.

Ключевая разница с `object`: `object` стирает тип навсегда, а `T` его запоминает и переносит на выход. Когда Вы вызываете `first(["a", "b"])`, анализатор фиксирует `T = str` для этого вызова и обещает, что результат - именно `str`, а не размытый `object`. Имя `T` - всего лишь соглашение; подойдёт любое, но односимвольные заглавные (`T`, `K`, `V`) приняты по традиции.

## Первая обобщённая функция

Соберём ту самую функцию «первый элемент». Объявляем `TypeVar`, используем его дважды - в типе аргумента и в типе возврата.

```python
from typing import TypeVar

T = TypeVar("T")

def first(items: list[T]) -> T:
    return items[0]
```

Одна функция теперь обслуживает любые списки, и тип результата каждый раз точный. `first([1, 2, 3])` для анализатора возвращает `int`, `first(["a", "b"])` - `str`. Никаких копий под каждый тип, и при этом ни грамма потерянной информации.

Связь работает в обе стороны и через несколько аргументов. Функция, которая возвращает один из двух переданных объектов, требует, чтобы оба были одного типа `T`, и обещает вернуть тот же `T`.

```python
def pick(a: T, b: T, use_first: bool) -> T:
    return a if use_first else b
```

Вызвав `pick(1, 2, True)`, Вы получите `int`; `pick("x", "y", False)` даст `str`. А вот `pick(1, "x", True)` анализатор забракует: `T` не может быть одновременно `int` и `str`.

## Обобщённые классы через Generic

`TypeVar` оживляет не только функции. Чтобы написать собственный контейнер, который помнит тип своего содержимого, класс наследуют от `Generic[T]`. Тогда `T` становится параметром класса: его указывают при создании, и все методы получают согласованные типы.

```python
from typing import Generic, TypeVar

T = TypeVar("T")

class Box(Generic[T]):
    def __init__(self, item: T) -> None:
        self._item = item

    def get(self) -> T:
        return self._item
```

`Box[int]` - это коробка, чей `get()` возвращает `int`; `Box[str]` - чей `get()` возвращает `str`. Один класс, параметризуемый типом, ровно как встроенный `list[int]`. Аннотация при создании, `box: Box[int] = Box(42)`, фиксирует параметр, и анализатор проследит, чтобы Вы не положили в эту коробку строку.

<MarginNote label="PEP 695">В Python 3.12 появился синтаксис `def first[T](items: list[T]) -> T:` и `class Box[T]:` без явного `TypeVar`. Он короче, но под капотом - тот же механизм. Старый стиль из этого раздела работает везде и встречается чаще.</MarginNote>

## Потрогайте обобщённость руками

Запустите код: `Box` хранит и отдаёт значение любого типа, а во время выполнения никакой `T` не существует - это видно по тому, что обе коробки одного класса. Затем поменяйте `Box(42)` на `Box([1, 2, 3])` или на `Box("привет")` и запустите снова: контейнер примет что угодно, потому что вся «обобщённость» живёт в аннотациях, а не в рантайме. Анализатор при этом для каждого случая вывел бы свой тип `get()`.

Вторая строка печатает `True`: с точки зрения интерпретатора `Box(42)` и `Box("привет")` - объекты одного и того же класса `Box`. Разница между `Box[int]` и `Box[str]` существует только для анализатора.

## Краевой случай: TypeVar нельзя делить между несвязанными контекстами

Это вас удивит: `T` в одной сигнатуре и `T` в другой - независимы, даже если это буквально один и тот же объект `TypeVar`. Анализатор заново «решает», чему равен `T`, для каждого вызова. Поэтому общий `T = TypeVar("T")` спокойно используют в десятках функций: каждая получает свою подстановку, конфликта нет.

```python
T = TypeVar("T")

def head(items: list[T]) -> T:
    return items[0]

def repeat(value: T, n: int) -> list[T]:
    return [value] * n
```

`head` и `repeat` делят одно имя `T`, но это не связывает их между собой: в `head(["a"])` будет `T = str`, в `repeat(7, 3)` - `T = int`. Связь возникает только внутри одной сигнатуры, между её собственными позициями `T`.

## Типичные ошибки

- **Использовать `object` там, где нужен `TypeVar`.** `def first(items: list[object]) -> object` делает функцию универсальной, но стирает тип результата. Если выход должен повторять тип входа - это `TypeVar`, а не `object`.
- **Забывать наследовать класс от `Generic[T]`.** Просто использовать `T` в методах класса недостаточно - без `class Box(Generic[T])` анализатор не поймёт, что класс параметризуется, и `Box[int]` окажется ошибкой.
- **Ждать `T` в рантайме.** Во время выполнения `Box[int]` и `Box[str]` - один класс, а `T` нигде не хранится. Нельзя «спросить у объекта его `T`»; стирание типов означает, что эта информация до рантайма не доживает.
- **Связывать разные типы одним `T` по ошибке.** Если два аргумента должны быть разных типов, им нужны разные `TypeVar` (`K` и `V`). Один `T` на оба заставит анализатор требовать совпадения типов.

<Detail summary="Стирание типов: почему дженерики Python не похожи на дженерики C++">

В языках вроде C++ шаблоны порождают отдельный машинный код под каждый конкретный тип: `vector<int>` и `vector<string>` - это два разных сгенерированных класса. В Python всё иначе. `Generic` и `TypeVar` существуют исключительно на уровне аннотаций; в рантайме `Box[int]` - это тот же объект класса `Box`, а параметр типа стёрт (type erasure).

Практическое следствие: нельзя написать `if T is int` внутри метода - переменной `T` в рантайме просто нет. Если поведение должно зависеть от типа во время выполнения, его передают как обычный аргумент-значение (например, фабрику или сам класс), а не пытаются достать из параметра дженерика. Аннотации и исполнение - два разных мира, и `TypeVar` целиком живёт в первом.

</Detail>
