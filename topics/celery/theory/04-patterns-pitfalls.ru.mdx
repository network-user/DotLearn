# Когда одной задачи мало

Одна задача отправляет одно письмо - это понятно. Но настоящая работа редко укладывается в одну задачу. Надо скачать файл, затем его обработать, затем сохранить результат - три шага, где каждый следующий зависит от предыдущего. Или: разослать уведомление тысяче пользователей - тысяча независимых задач, после которых надо подвести итог «отправлено столько-то». Писать это вручную через вложенные `.delay()` быстро превращается в кашу: кто кого вызывает, как дождаться всех, что делать с результатами.

Celery даёт готовый язык для сборки задач в структуры - это canvas. К концу раздела вы будете различать `chain` (задачи по цепочке, результат передаётся дальше), `group` (задачи параллельно) и `chord` (группа плюс финальный шаг), знать, когда какой брать, и - не менее важно - узнавать подводные камни, на которых ломаются даже корректные по отдельности задачи: тяжёлые аргументы, общее состояние и переполнение очереди.

## Ментальная модель: chain - последовательно, group - параллельно

Два базовых способа соединить задачи. **Chain** ставит задачи в очередь так, что они выполняются по порядку, и результат каждой передаётся в следующую как аргумент. **Group** запускает набор задач параллельно, независимо друг от друга. Их комбинация под названием **chord** запускает группу и, когда вся группа завершилась, выполняет финальную задачу-callback над собранными результатами.

На схеме видно разницу в стрелках. В chain стрелки идут от задачи к задаче: результат первой - вход второй. В group стрелок между задачами нет вовсе: они не знают друг о друге и бегут параллельно. В chord три параллельные задачи сходятся в один callback, который запустится, только когда завершатся все три. Это и есть три кирпича, из которых собираются почти любые многошаговые сценарии.

<MarginNote label="signature">Кирпич canvas - это signature: задача с уже зафиксированными частью аргументов (`add.s(2)`). Из signature собираются chain, group и chord. Метод `.s()` создаёт signature, не выполняя задачу.</MarginNote>

## Chain: результат течёт по цепочке

Цепочка нужна, когда шаги зависимы: следующий работает над результатом предыдущего. Соберём «скачать → обработать → сохранить».

```python
from celery import chain

workflow = chain(
    download.s(url),
    process.s(),
    store.s(),
)
workflow.delay()
```

`download.s(url)` - signature с зафиксированным аргументом `url`. В цепочке результат `download` подставится первым аргументом в `process`, а результат `process` - в `store`. Поэтому `process` и `store` написаны с `.s()` без первого аргумента: его даст предыдущий шаг. Запускается цепочка одним `.delay()`; Celery сам ставит следующий шаг в очередь, когда предыдущий завершился. Если любой шаг упадёт, дальнейшие по умолчанию не выполнятся - цепочка прервётся.

<Callout type="info">
Есть короткий синтаксис через оператор `|`: `(download.s(url) | process.s() | store.s()).delay()`. Он читается как конвейер слева направо и делает ровно то же, что `chain(...)`.
</Callout>

## Group и chord: много параллельно, потом итог

Группа - для независимой работы над множеством элементов. Разослать письмо каждому из списка - классический случай.

```python
from celery import group

job = group(send_email.s(uid) for uid in user_ids)
job.delay()
```

`group` создаёт по signature на каждого пользователя и ставит их в очередь все сразу. Worker'ы разберут их параллельно, в произвольном порядке, независимо. Группа не ждёт завершения и не собирает результаты в одно место - она просто веер из задач. Если же после рассылки нужно подвести итог («отправлено N писем»), группа оборачивается в chord с финальным callback.

```python
from celery import chord

chord(
    (send_email.s(uid) for uid in user_ids),
    summarize.s(),
).delay()
```

`summarize` получит список результатов всех `send_email` и выполнится ровно один раз, когда отработает вся группа. Chord - единственный из трёх паттернов, который требует result backend: чтобы понять, что группа завершилась, и собрать её результаты, Celery должен где-то их хранить. Chain и group без callback в backend не нуждаются.

## Подводный камень: тяжёлые аргументы

Аргументы задачи сериализуются (по умолчанию в JSON) и едут через брокер. Что вы передаёте - то и кладётся в очередь целиком.

```python
process_image.delay(huge_image_bytes)
process_image.delay(image_id)
```

Первый вызов запихивает в брокер всю картинку: сообщение раздувается, сериализация и пересылка тормозят, а очередь забивается мегабайтами. Второй передаёт только `image_id`, а саму картинку задача загрузит из хранилища по этому id. Правило простое: через очередь передавайте ссылки (идентификаторы, ключи, пути), а не данные. То же касается ORM-объектов - передавайте `user.id`, а не сериализованного `user`, который к моменту выполнения задачи мог уже устареть.

## Краевые случаи

**Это вас удивит: задача может выполниться раньше, чем закоммитилась транзакция.** Вы создаёте пользователя в транзакции БД и тут же зовёте `send_welcome_email.delay(user.id)`. Worker свободен и хватает задачу за миллисекунды - до того, как ваша транзакция закоммитилась. Задача читает базу, не находит пользователя и падает. Лечится постановкой задачи после коммита (в Django - `transaction.on_commit(lambda: ...)`): producer должен ставить задачу, только когда данные для неё уже видимы остальным.

**Общее состояние между задачами не разделяется так, как кажется.** Worker исполняет задачи в пуле процессов. Глобальная переменная или кэш в памяти, заполненный одной задачей, не виден другой, если та попала в другой процесс. Любое состояние, которое должны видеть несколько задач, держите во внешнем хранилище (база, Redis), а не в памяти процесса.

**Переполнение очереди тихо растит задержку.** Если задачи ставятся быстрее, чем worker'ы успевают их разбирать, очередь растёт без ошибок - просто всё выполняется всё позже. Внешне «Celery тормозит», по сути - не хватает пропускной способности. Решается добавлением worker'ов, разделением задач по очередям и приоритетам или ограничением скорости постановки.

## Типичные ошибки

- **Передавать в задачу тяжёлые объекты вместо идентификаторов.** Большие аргументы раздувают сообщения в брокере и замедляют всю очередь. Передавайте id и ключи, грузите данные внутри задачи.
- **Ставить задачу до коммита транзакции.** Worker может схватить задачу раньше, чем producer закоммитит данные, - задача не найдёт того, ради чего её запустили. Ставьте задачу после коммита.
- **Брать chord без result backend.** Chord физически не работает без хранилища результатов: ему негде дождаться группы и собрать её вывод. Либо настройте backend, либо обойдитесь chain и group без финального callback.

<Detail summary="Почему gen-выражение в group иногда коварно, и как Celery собирает результаты chord">

Group и chord часто строят из генераторного выражения: `group(send.s(uid) for uid in user_ids)`. Удобно, но помните, что генератор материализуется в момент построения группы, и весь список signature собирается в памяти producer. Для тысяч элементов это нормально, для миллионов - повод бить работу на пачки (`chunks`) или ставить под-группы, а не одну гигантскую группу: иначе одно сообщение-родитель распухает, как и тяжёлый аргумент.

Сборка результатов chord устроена так: Celery отслеживает, сколько задач группы завершилось, через result backend (счётчик в Redis или периодический опрос в зависимости от backend). Когда счётчик дошёл до размера группы, запускается callback с собранными результатами. Отсюда два следствия. Первое: если хоть одна задача группы упала и не дала результата, chord по умолчанию не запустит callback - вся конструкция повиснет в ожидании, поэтому ошибки внутри группы надо обрабатывать. Второе: на огромных группах опрос backend сам по себе создаёт нагрузку, и тут снова спасает разбиение на пачки.

</Detail>
