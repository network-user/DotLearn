# Задача может выполниться дважды, и вы не должны этого бояться

Worker забрал задачу «списать 100 рублей со счёта», начал её выполнять, успел списать деньги - и в этот момент сервер с worker перезагрузился. Брокер не получил подтверждения, что задача выполнена, и по своим правилам отдаст её другому worker. Тот спишет ещё 100 рублей. Никто не написал бажный код: каждый компонент вёл себя ровно так, как обещано. Просто в распределённой системе «выполнить ровно один раз» - недостижимая по умолчанию гарантия, и проектировать надо исходя из этого.

Эта тема пугает, но на ней стоит всё надёжное использование очередей. К концу раздела вы будете понимать, почему задача в принципе может выполниться больше одного раза, как Celery повторяет упавшие задачи и - главное - что значит сделать задачу идемпотентной, чтобы повторный запуск не ломал данные. Это не про экзотику, а про обычную отправку письма и списание денег.

## Ментальная модель: повтор - это норма, дубль - ваша забота

Celery работает по принципу «хотя бы один раз» (at-least-once). Когда worker берёт задачу, она остаётся в брокере как «выданная, но не подтверждённая». Worker подтверждает (acknowledge) её, только закончив. Если worker умер до подтверждения, брокер считает задачу невыполненной и отдаст её снова. Итог: задача гарантированно выполнится хотя бы раз, но может выполниться и дважды, и трижды.

Картинка показывает суть: брокер честно гарантирует доставку задачи worker, но не может гарантировать, что побочный эффект случится ровно один раз - он не видит, что происходит внутри задачи, только факт ack. Отсюда практический вывод, который определяет всё дальнейшее: раз дубли в принципе возможны, задачу надо писать так, чтобы повторный запуск был безопасен. Это свойство называется идемпотентностью.

<MarginNote label="acknowledge">Ack - сигнал worker брокеру «эту задачу можно забыть, я её обработал». По умолчанию Celery шлёт ack рано (когда задача только взята), но для критичных задач переключают на late ack - подтверждение после выполнения.</MarginNote>

## Повторы при сбоях: retry

Отдельно от переотправки брокером есть явные повторы, которые задача запускает сама, когда упёрлась во временную ошибку (внешний API ответил 503, сеть моргнула). Для этого задаче дают доступ к себе через `bind=True` и вызывают `self.retry`.

```python
@app.task(bind=True, max_retries=3)
def fetch_rate(self, currency):
    try:
        return call_external_api(currency)
    except TimeoutError as exc:
        raise self.retry(exc=exc, countdown=10)
```

`bind=True` делает первым аргументом сам объект задачи (`self`). При таймауте `self.retry(...)` не выполняет задачу заново тут же - он ставит её в очередь снова с задержкой `countdown=10` секунд и поднимает специальное исключение, прерывая текущий заход. `max_retries=3` ограничивает число попыток: после трёх повторов задача окончательно падает. Повторяйте только временные ошибки: если API вернул «нет такой валюты», повтор не поможет и лишь засорит очередь.

<Callout type="tip">
Ставьте задержку между повторами растущей (`countdown` побольше с каждой попыткой) - это exponential backoff. Долбить упавший внешний сервис каждую секунду означает мешать ему встать на ноги. В Celery это включается параметрами `retry_backoff=True` и `retry_jitter=True`.
</Callout>

## Что значит «идемпотентная задача»

Задача идемпотентна, если её повторный запуск с теми же аргументами не меняет результат по сравнению с однократным. Запустили один раз или пять - состояние системы одинаковое. Сравним две версии начисления бонуса.

```python
@app.task
def grant_bonus(user_id):
    user = User.objects.get(id=user_id)
    user.balance += 100
    user.save()
```

Эта задача не идемпотентна: каждый повтор добавляет ещё 100. Два прогона - плюс 200. Теперь сделаем операцию идемпотентной, привязав начисление к уникальному событию.

```python
@app.task
def grant_bonus(user_id, bonus_id):
    _, created = Bonus.objects.get_or_create(
        id=bonus_id, defaults={"user_id": user_id, "amount": 100}
    )
    if created:
        User.objects.filter(id=user_id).update(balance=F("balance") + 100)
```

Теперь `bonus_id` - ключ идемпотентности. `get_or_create` создаёт запись о бонусе только один раз; на повторе `created` будет `False`, и баланс не тронется. Сколько бы раз задача ни выполнилась, бонус начислится ровно однажды. Ключ должен приходить снаружи (его генерирует producer), а не создаваться внутри задачи - иначе каждый повтор сгенерирует новый ключ и защита не сработает.

Первый приём - **уникальный ключ операции** (как `bonus_id` выше): уникальное ограничение в базе физически не даст создать дубль. Второй - **проверка состояния перед действием**: задача «отправить заказ» сначала смотрит, не отправлен ли заказ уже (`if order.status != "sent"`), и только тогда отправляет. Третий - **естественная идемпотентность операции**: `set status = "done"` или `PUT` по конкретному ключу можно повторять сколько угодно, результат один и тот же; в отличие от `balance += 100`, которое накапливает эффект.

Выбирайте приём под операцию. Запись с уникальным ключом - для «создать ровно один раз». Проверка состояния - когда есть явный статус-флаг. Перезапись по ключу - для операций «установить значение», которые по своей форме уже безопасны к повтору.

## Краевые случаи

**Повтор после частичного выполнения.** Самое коварное: задача сделала первый из двух побочных эффектов (списала деньги), упала перед вторым (не начислила товар) и повторилась. Если идемпотентность навешена только на второй шаг, деньги спишутся дважды. Идемпотентным должен быть каждый внешний эффект задачи, а не только последний.

**Это вас удивит: даже «успешная» задача может повториться.** Кажется, что повтор бывает только у упавших задач. Но если worker выполнил задачу полностью и упал ровно перед отправкой ack (или сеть до брокера моргнула в этот миг), брокер не узнает об успехе и переотправит задачу. Поэтому идемпотентность нужна и для задач, которые «никогда не падают» внутри: причина дубля может быть снаружи кода задачи.

**Повтор не вечен, но окно велико.** `max_retries` ограничивает явные `self.retry`, но не переотправки брокером при падении worker. Не закладывайтесь на то, что «после N попыток точно остановится»: проектируйте так, чтобы безопасным был любой повтор, а не первые несколько.

## Типичные ошибки

- **Считать выполнение «ровно один раз» данностью.** Celery даёт «хотя бы один раз». Любая задача с внешними эффектами должна переживать повтор без порчи данных - это не опция, а условие корректности.
- **Повторять не те ошибки.** `self.retry` оправдан для временных сбоев (таймаут, 503, сетевой обрыв). Повторять `ValueError` из-за плохих входных данных бессмысленно: на следующем заходе данные те же, и вы лишь забьёте очередь обречёнными повторами.
- **Генерировать ключ идемпотентности внутри задачи.** Если ключ создаётся в теле задачи (новый `uuid4` на каждом заходе), повтор получит новый ключ, и дедупликация не сработает. Ключ должен прийти в аргументах от producer и быть стабильным между повторами.

<Detail summary="acks_late и почему ранний ack иногда теряет задачи">

По умолчанию worker шлёт ack сразу, как взял задачу из брокера, и только потом начинает её выполнять. Это значит: если worker упадёт во время выполнения, брокер уже считает задачу подтверждённой и переотправлять её не будет - задача потеряется. Для быстрых неважных задач это приемлемо, для критичных - нет.

Параметр `acks_late=True` (на задаче или глобально) переносит ack на момент после завершения. Тогда падение во время выполнения приводит к переотправке - задача не потеряется. Цена этого - именно те дубли, о которых весь раздел: задача, упавшая под конец, выполнится снова. Поэтому `acks_late` и идемпотентность ходят парой: позднее подтверждение даёт «не потеряем», идемпотентность - «переживём дубль». Включать `acks_late` без идемпотентности - значит обменять потерю задач на двойное выполнение, что часто хуже.

</Detail>
