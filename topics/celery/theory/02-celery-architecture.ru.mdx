# Три части, которые часто путают в одну

«Я поставил Celery» - так говорят, как будто Celery это одна программа, которую запускаешь и она работает. На деле работающая система фоновых задач - это три разные движущиеся части, и каждая может быть на своём сервере, отказать независимо и требовать своей настройки. Когда задачи «теряются», «не выполняются» или «выполнились, но я не вижу результата», почти всегда дело в том, что три части перепутали в одну и не понимают, какая за что отвечает.

В прошлом разделе мы увидели общий контур: producer кладёт, consumer берёт, между ними хранилище. Теперь разберём конкретную реализацию Celery. К концу раздела вы будете чётко различать broker (где живут задачи в ожидании), worker (кто их выполняет) и result backend (где лежат результаты), понимать, что из них обязательно, а что нет, и читать ошибки вроде «нет соединения с брокером» как указание на конкретную часть.

## Ментальная модель: где задача, кто её делает, куда кладёт ответ

Запустить Celery - значит запустить и связать три роли. **Broker** - это очередь: хранилище, куда producer кладёт задачи и откуда worker их забирает. Чаще всего это Redis или RabbitMQ. **Worker** - процесс, который подключается к брокеру, забирает задачи и исполняет их код. **Result backend** - отдельное хранилище, куда worker, если его попросили, кладёт результат и статус задачи, чтобы producer мог их потом прочитать.

На схеме видно, что broker и backend - это разные хранилища, даже если технически вы используете для них один и тот же Redis. Поток задачи: producer → broker → worker (выполнение) → backend. Поток результата идёт обратно: producer читает из backend по идентификатору задачи. Broker обязателен всегда - без него задачу некуда положить. Backend опционален: он нужен, только если кому-то нужен результат или статус.

<MarginNote label="Пунктир неспроста">Стрелки к backend на схеме пунктирные: этот путь существует, только если result backend настроен. Без него задача выполнится, но результат нигде не сохранится.</MarginNote>

## Минимальная настройка

Объект `Celery` создаётся с указанием брокера и (опционально) backend. Задача - это обычная функция, помеченная декоратором.

```python
from celery import Celery

app = Celery(
    "myapp",
    broker="redis://localhost:6379/0",
    backend="redis://localhost:6379/1",
)

@app.task
def add(x, y):
    return x + y
```

Здесь `broker` указывает, куда класть и откуда брать задачи, а `backend` - куда складывать результаты. Обратите внимание на `/0` и `/1`: это разные логические базы Redis, чтобы очередь задач и хранилище результатов не мешали друг другу, даже живя в одном Redis. Декоратор `@app.task` превращает `add` в задачу, у которой появляется метод `.delay()`.

## Producer и worker - это разные процессы

Один и тот же файл с задачами импортируется и в веб-приложении (producer), и в worker. Но запускаются они отдельно. Worker поднимается отдельной командой.

```python
result = add.delay(2, 3)
print(result.id)
```

Этот код - сторона producer. Он кладёт задачу `add(2, 3)` в broker и получает объект `AsyncResult` с идентификатором задачи. Сам `add` здесь не выполняется: ни `2 + 3`, ни `return` в этом процессе не происходят. Чтобы задача выполнилась, где-то должен работать worker, запущенный командой `celery -A myapp worker`. Он подключится к тому же брокеру, увидит задачу, выполнит `add(2, 3)` и положит `5` в backend под ключом `result.id`.

<Callout type="warning">
Если worker не запущен, `add.delay(2, 3)` всё равно отработает без ошибки: задача ляжет в broker и будет там ждать. Поэтому «задача не выполняется» часто означает не баг в коде, а просто незапущенный или упавший worker.
</Callout>

## Как достать результат

Producer получил `AsyncResult` сразу, но результата в нём ещё нет - он появится в backend, только когда worker закончит. Чтение результата идёт через тот же объект.

```python
result = add.delay(2, 3)
print(result.ready())
print(result.get(timeout=5))
```

`result.ready()` спрашивает backend, готов ли результат, и возвращает `True` или `False`, не блокируя. `result.get(timeout=5)` блокирует выполнение и ждёт результат до пяти секунд, после чего бросит исключение по таймауту. Важно: если backend не настроен, `result.get()` работать не будет - результату негде храниться. Вызывать `.get()` синхронно прямо в веб-обработчике обычно плохая идея: вы снова блокируете запрос, ровно от чего уходили в прошлом разделе.

## Брокер и backend решают разные задачи

Broker оптимизирован под «принять сообщение и быстро отдать его одному получателю». Backend оптимизирован под «сохранить значение по ключу и отдать по запросу». Это разные модели доступа, поэтому Celery держит их раздельно.

Выбор брокера влияет на гарантии. RabbitMQ - полноценный message broker с подтверждениями и маршрутизацией, исторически основной выбор для Celery. Redis проще в эксплуатации и популярен, но как брокер имеет нюансы с надёжностью при перезагрузке. Для backend Redis удобен скоростью, но результаты в нём по умолчанию живут с TTL и пропадают, а реляционная база хранит их долго, но медленнее.

## Краевые случаи

**Result backend не настроен, а вы зовёте `.get()`.** Задача выполнится, worker отработает, но результат сохранять некуда. Попытка `result.get()` повиснет или бросит ошибку конфигурации. Если результаты не нужны, не зовите `.get()`; если нужны - настройте backend.

**Это вас удивит: хранить каждый результат по умолчанию дорого.** Кажется логичным включить result backend и складывать туда всё. Но Celery по умолчанию пишет в backend статус и результат каждой задачи, и на больших объёмах это превращает backend в постоянно растущую свалку. Для задач, чей результат никому не нужен (отправили письмо - и забыли), backend выключают или ставят `ignore_result=True`, чтобы не платить за хранение того, что никто не прочитает.

**Worker и producer запущены с разным кодом.** Producer ставит задачу по имени-строке (например `myapp.add`). Worker ищет у себя функцию с этим именем. Если вы добавили задачу, передеплоили producer, но забыли перезапустить worker - worker не знает такой задачи и упадёт с `unregistered task`. Код задач должен быть одинаковым с обеих сторон.

## Типичные ошибки

- **Считать broker и backend одним и тем же.** Это разные хранилища с разной ролью. «Соединение с брокером» и «соединение с backend» настраиваются отдельно и отказывают независимо.
- **Звать `result.get()` в HTTP-обработчике.** Это снова блокирует запрос на время выполнения задачи и сводит на нет весь смысл фона. Если результат нужен в ответе - возможно, эту работу не стоило выносить в фон.
- **Забывать перезапустить worker после деплоя.** Worker держит в памяти код задач на момент своего старта. Новый или изменённый код задачи появится у него только после перезапуска.

<Detail summary="Что такое prefork-пул и почему worker - это не один процесс">

Команда `celery -A myapp worker` поднимает не один процесс, а главный процесс-координатор и пул дочерних процессов-исполнителей (по умолчанию prefork - форк по числу ядер). Координатор общается с брокером и раздаёт задачи дочерним процессам, каждый из которых выполняет одну задачу за раз. Поэтому worker с пулом из четырёх процессов выполняет до четырёх задач параллельно.

Это объясняет два практических момента. Во-первых, если задача держит глобальное состояние, помните, что параллельные задачи могут идти в разных процессах и состояние не разделяется так, как вы думаете. Во-вторых, для задач, которые в основном ждут сети (вызовы внешних API), prefork с процессами расточителен по памяти - под такие нагрузки выбирают пулы на основе gevent или eventlet, где один процесс держит много лёгких задач, пока те ждут ввода-вывода. Выбор пула задаётся флагом `--pool`.

</Detail>
