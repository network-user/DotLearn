The user clicks "Sign up" and your handler honestly waits on a third-party SMTP server that takes four seconds to send a confirmation email. The browser spins, the web server worker is tied up on that one request and cannot take the next, and if ten people click the button at once you hit not your code but the speed of someone else's mail server. Sending email, generating PDFs, transcoding video, calling external APIs - all of this is slow and unreliable, while an HTTP request must answer quickly. This topic is about moving slow work out of the request and making it reliable.

**What you will learn to do.** Tell work that can stay in the handler from work that belongs in the background; assemble a working background task system from broker, worker, and backend and understand who does what; design tasks so a retry after failure does not charge money twice; compose tasks into chains and groups and avoid traps that break even individually correct tasks.

**Topic route.** Four sections, each building on the previous:

1. **Why queues** - why long work cannot live inside a request, and how a producer enqueues while a consumer picks up.
2. **Celery architecture** - three parts people often merge into one: broker (where tasks wait), worker (who runs them), and result backend (where the answer lives).
3. **Retries and idempotency** - accept that a task may run twice and learn to write it so a repeat does not corrupt data.
4. **Patterns and pitfalls** - assemble tasks with `chain`, `group`, and `chord` and learn traps: heavy arguments, shared state, queue overflow.

**Prerequisites.** Comfortable Python and a basic web handler that responds to requests. Nothing to install: we explain Redis, RabbitMQ, and Celery as we go, from simple "enqueue - dequeue" to reliable task pipelines.
