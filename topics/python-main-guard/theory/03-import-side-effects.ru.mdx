# Когда забытый guard превращается в бесконечный цикл

Пока модуль импортируют только люди, незащищённый код на верхнем уровне в худшем случае печатает лишнюю строку. Но есть библиотека, которая делает забытый guard по-настоящему опасным - `multiprocessing`. Чтобы запустить новый процесс на Windows и в части режимов на других системах, Python берёт Ваш главный файл и **импортирует его заново** в дочернем процессе. Если код запуска не спрятан под guard, дочерний процесс при импорте снова дойдёт до строки, которая создаёт процессы, - и создаст ещё, и ещё. Вместо параллельной обработки Вы получаете лавину процессов, которая кладёт машину.

К концу раздела Вы будете понимать, что значит «сайд-эффект импорта», почему `multiprocessing` повторно импортирует главный модуль, и почему для программ с процессами guard - это не стиль, а условие работоспособности. Это объединяет всё, что Вы узнали: модуль исполняется при импорте, `__name__` различает запуск и импорт, а guard разводит их - здесь становится видно, зачем это нужно на практике.

## Ментальная модель: сайд-эффект - это действие при импорте

Сайд-эффект импорта - это любое наблюдаемое действие, которое происходит просто потому, что модуль импортировали: печать, запись в файл, сетевой запрос, запуск потока или процесса. Определение функции сайд-эффектом не является - оно лишь создаёт объект, ничего не «делая» во внешнем мире. А вот вызов этой функции на верхнем уровне модуля - уже сайд-эффект. Guard нужен, чтобы опасные сайд-эффекты случались только при запуске, а не при каждом импорте.

## Сначала - безобидный сайд-эффект

Прежде чем `multiprocessing`, посмотрим на сайд-эффект попроще: открытие файла на верхнем уровне модуля.

```python
log = open("audit.log", "a")

def record(event):
    log.write(event + "\n")
```

Файл `audit.log` открывается на запись в момент импорта этого модуля - даже если ни разу не вызвать `record`. Любой `import` оставляет за собой открытый файловый дескриптор. Это сайд-эффект: импорт «сделал» что-то во внешнем мире. Часто это не то, чего Вы хотите, и лечится переносом открытия внутрь функции или под guard.

## Почему `multiprocessing` повторно импортирует модуль

Когда Вы создаёте процесс через `multiprocessing`, дочернему процессу нужно где-то взять код функции, которую он должен исполнить. В методе запуска `spawn` (единственном на Windows и доступном везде) дочерний процесс стартует «с чистого листа» и, чтобы добраться до этой функции, импортирует Ваш главный модуль заново. Именно тут срабатывает всё из первого раздела: импорт исполняет тело файла сверху вниз.

```python
from multiprocessing import Process

def worker():
    print("работаю в отдельном процессе")

Process(target=worker).start()
```

Без guard эта строка `Process(...).start()` стоит на верхнем уровне. В дочернем процессе при повторном импорте она исполнится снова, создаст ещё один процесс, тот импортирует модуль, дойдёт до той же строки - и так по нарастающей. На Windows такой файл способен за секунды наплодить сотни процессов.

## Guard разрывает цепную реакцию

Достаточно убрать запуск под guard. В дочернем процессе главный модуль импортируется, значит его `__name__` равно имени модуля, а не `"__main__"` - и строка запуска пропускается.

```python
from multiprocessing import Process

def worker():
    print("работаю в отдельном процессе")

if __name__ == "__main__":
    Process(target=worker).start()
```

Теперь запуск процесса происходит только в исходном, главном процессе. Дочерний импортирует модуль ради определения `worker`, проходит мимо guard и спокойно делает свою работу. Та же переменная `__name__`, что в первом разделе казалась мелочью, здесь спасает от каскада процессов.

<Callout type="warning">
В документации `multiprocessing` защита главного модуля через `if __name__ == "__main__"` названа обязательной для метода запуска `spawn`. Это не рекомендация по стилю, а требование корректности: без неё программа с процессами может не запуститься или зациклиться на порождении процессов.
</Callout>

## Потрогайте идею руками

Песочница не запускает реальные процессы, но саму логику повторного импорта можно смоделировать. Ниже функция `simulate_import` имитирует, что происходит, когда модуль исполняется как `"__main__"` и как импортированный. Запустите как есть, затем поменяйте значение переменной `current_name` с `"__main__"` на `"worker_module"` и снова запустите: увидите, что во втором случае «запуск» не происходит - ровно так дочерний процесс проходит мимо guard.

<PyDemo
  title="Как guard ведёт себя при разных __name__"
  code={`def spawn_processes():
    return "СОЗДАЮ процессы"

def simulate(current_name):
    if current_name == "__main__":
        return spawn_processes()
    return "импорт: запуск пропущен"

current_name = "__main__"
print(simulate(current_name))
print(simulate("worker_module"))`}
/>

Первый вызов печатает «СОЗДАЮ процессы», второй - «импорт: запуск пропущен». Это и есть поведение реального guard в двух режимах: при `"__main__"` блок запуска работает, при импортированном имени - молчит.

## Сюрприз: на Linux то же самое может «работать» по случайности

Долгое время на Linux метод запуска по умолчанию был `fork`: дочерний процесс получал копию памяти родителя и **не** импортировал главный модуль заново. Поэтому незащищённый код там часто прощался, и программисты привыкали к мысли «у меня всё работает без guard». Но эта удача хрупкая: тот же код на Windows или с явным `set_start_method("spawn")` мгновенно ломается, а в свежих версиях Python метод по умолчанию на ряде платформ смещается в сторону, где повторный импорт происходит. Код, который «работал без guard», - это не корректный код, а код, которому повезло с платформой.

```python
import multiprocessing as mp
print(mp.get_start_method())
```

Эта строка печатает текущий метод запуска - `fork`, `spawn` или `forkserver`. Если когда-нибудь будете отлаживать загадочное поведение процессов на чужой машине, начните с неё: метод запуска объясняет, почему один и тот же код ведёт себя по-разному на разных системах.

## Типичные ошибки

- **Опираться на «у меня и так работает» с Linux/fork.** Отсутствие guard может прощаться на одной платформе и ломаться на другой. Для переносимого кода с процессами guard обязателен.
- **Класть тяжёлую инициализацию вне guard в программе с процессами.** Всё, что на верхнем уровне модуля, исполнится в каждом дочернем процессе при повторном импорте. Дорогую подготовку прячут под guard или внутрь функций.
- **Считать определение функции сайд-эффектом.** Само `def worker(): ...` ничего не запускает - проблема не в определении, а в его вызове на верхнем уровне. Под guard прячут вызов, а не определение.
- **Забывать про guard в скриптах, которые позже станут модулями.** Маленький скрипт без guard однажды кто-то импортирует - и его код запуска выстрелит в чужой программе. Дешевле сразу писать guard.

<Detail summary="fork, spawn и forkserver - чем они отличаются">

`multiprocessing` умеет запускать процессы тремя способами. `fork` копирует процесс целиком вместе с его памятью и импортированными модулями - быстро, но небезопасно при работе с потоками и недоступно на Windows. `spawn` стартует чистый интерпретатор и заново импортирует нужные модули - медленнее, переносимо, и именно он требует guard. `forkserver` - компромисс: отдельный процесс-сервер форкает воркеров из заранее подготовленного состояния. Узнать текущий метод можно через `multiprocessing.get_start_method()`, а задать - через `set_start_method()`. Важное следствие: как только в игре `spawn` или `forkserver`, главный модуль импортируется в дочерних процессах, и без guard код запуска исполнится повторно. Детали - в официальной документации по методам запуска.

</Detail>
