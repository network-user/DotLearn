You wrote a handy file `calc.py`: inside is a useful function, and at the end a couple of lines call it and print the result, so when you run it you immediately see that everything works. Later that function was needed in another script, you wrote `from calc import add`, and in the middle of someone else's program something printed that should not have been there. The same file behaves differently depending on how you use it: run it directly or import it for one function. This topic is about how Python tells these two cases apart and how to separate "code for running" from "code for importing" with one familiar line.

**What you will learn to do.** Understand that any `.py` file is a module that runs in full top to bottom when imported; read the value of `__name__` and use it to tell running a file apart from importing it; write the guard `if __name__ == "__main__"` so run code does not clutter output on import; see the specific problem it fixes and why programs using `multiprocessing` without it can spawn a flood of processes.

**Topic route.** Three sections, each with a live sandbox where you can change and run code right in the browser:

1. **Modules and `__name__`**: why import means running the whole file, and where the variable comes from that holds either `"__main__"` or the module name.
2. **The guard itself** - turn that fact into a practical tool: what code goes inside `if __name__ == "__main__"` and what stays outside.
3. **Import side effects**: how a forgotten guard breaks `multiprocessing` and turns a style question into a requirement for correctness.

**Prerequisites.** Nothing to install: Python runs right in the browser. You only need to write simple functions and understand what import means; we will cover everything else along the way, from the first `print` to multiprocess code.
