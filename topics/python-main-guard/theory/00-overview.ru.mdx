title: "if __name__ == \"__main__\" в Python: обзор темы"
estimatedMinutes: 3
---

Вы написали удобный файл `calc.py`: внутри полезная функция, а в конце пара строк, которые её вызывают и печатают результат - чтобы при запуске сразу видеть, что всё работает. Потом эта функция понадобилась в другом скрипте, Вы написали `from calc import add` - и посреди чужой программы внезапно напечаталось то, чего там быть не должно. Один и тот же файл ведёт себя по-разному, смотря как Вы его задействовали: запустили напрямую или импортировали ради одной функции. Эта тема о том, как Python различает эти два случая и как развести «код для запуска» и «код для импорта» одной знакомой строкой.

**Что Вы научитесь делать.** Понимать, что любой `.py`-файл - это модуль, который при импорте исполняется целиком, сверху донизу; читать значение переменной `__name__` и по нему отличать запуск файла от импорта; писать guard `if __name__ == "__main__"`, чтобы код запуска не мусорил при импорте; видеть, какую конкретную проблему он устраняет и почему в программах с `multiprocessing` без него дело доходит до лавины процессов.

**Маршрут темы.** Три раздела, каждый - с живой песочницей, где код можно изменить и выполнить прямо в браузере:

1. **Модули и `__name__`** - разбираемся, почему импорт - это запуск всего файла, и откуда берётся переменная, хранящая либо `"__main__"`, либо имя модуля.
2. **Сам guard** - превращаем этот факт в рабочий инструмент: какой код кладут внутрь `if __name__ == "__main__"`, а какой оставляют снаружи.
3. **Сайд-эффекты импорта** - смотрим, как забытый guard ломает `multiprocessing` и превращается из вопроса стиля в условие работоспособности.

**Что нужно заранее.** Ничего устанавливать не придётся: Python запускается прямо в браузере. Достаточно уметь писать простые функции и понимать, что такое импорт, - всё остальное разберём по ходу, от первого `print` до многопроцессного кода.
