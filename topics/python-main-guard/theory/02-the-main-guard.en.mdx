# One line that splits a file into "for running" and "for importing"

Imagine a file `calc.py`: inside is a useful function `add`, and at the end a couple of lines call it and print the result — so when you run the file you immediately see that everything works. Convenient while the file lives on its own. But as soon as you write `from calc import add` in another script, those final lines execute during import: in the middle of someone else's program a result prints that should not be there. A file that worked well "on run" starts cluttering on "import". The idiom `if __name__ == "__main__"` exists exactly to separate these two modes.

By the end of this section you will learn to read and write this guard, understand what code goes inside it and what stays outside, and see the specific problem it fixes. From the previous section you already know that `__name__` equals `"__main__"` only when you run the file — now we turn that fact into a practical tool.

## Mental model: one file, two readers

A module has two different "readers". The first is a person or the operating system running the file with `python calc.py`: they need code that does something right now — reads arguments, prints a result, starts the program. The second reader is another module that writes `import calc`: it needs only definitions, functions and classes, without side effects. The guard `if __name__ == "__main__"` is a switch that shows run code to the first reader and hides it from the second.

On the diagram the `add` definition is visible to both. The block under the guard connects only to direct run: on import execution never reaches it because `__name__ == "__main__"` is false. Now let's map this to code.

## Minimal example: before and after

Here is a file without a guard. On import of `add` the line with `print` runs and prints a result where it is not expected.

```python
def add(a, b):
    return a + b

print("2 + 3 =", add(2, 3))
```

Wrap the run code in a guard — and `print` runs only on direct file run, staying silent on import.

```python
def add(a, b):
    return a + b

if __name__ == "__main__":
    print("2 + 3 =", add(2, 3))
```

The `add` definition stayed outside the guard — both readers need it. The call and print moved inside — they are needed only on run. That is the whole trick: definitions outside, actions inside.

## What goes inside the guard and what stays outside

The rule is simple and almost always works: **definitions** (functions, classes, constants) live outside the guard, **actions** (calls, printing, reading command-line arguments, starting the program) go inside. A common form is to move all run logic into a `main()` function and leave a single call under the guard.

```python
def add(a, b):
    return a + b

def main():
    print("2 + 3 =", add(2, 3))

if __name__ == "__main__":
    main()
```

This form reads clearly: everything above the guard is reusable building blocks; the single line under the guard says "this is how the file runs on its own". The `main()` function remains available for import — you can call it from a test.

<Callout type="tip">
If you are unsure where a line belongs: ask whether it should run when someone writes `import your_module`. If not — it belongs inside the guard.
</Callout>

## Try the guard yourself

The sandbox runs code as the main program, so `__name__` equals `"__main__"` and the block under the guard runs. Run as-is — you will see output from `main()`. Now change the condition to `if __name__ == "library":` and run again: the condition becomes false, `main()` is not called, and you see exactly what happens on import — silence instead of output. That is how the module looks to whoever imports it.

## Surprise: the guard does not make import free

It is easy to decide the guard "turns off" the whole file on import. It does not. On import the entire module body runs — only the block under `if __name__ == "__main__"` is skipped because the condition is false. Any code outside the guard — including heavy work: opening a connection, reading a large file, a long loop — still runs on import.

```python
data = [x * x for x in range(1000000)]

def get_squares():
    return data

if __name__ == "__main__":
    print(len(get_squares()))
```

The list of a million elements is computed at module level, so it is built on any `import` of this module, even though `print` under the guard does not run on import. The guard hides only what is inside it; everything outside still executes on import. Expensive initialization should go either under the guard or inside a function called only when needed.

## Common mistakes

- **Typo in `"__main__"`.** The string must be exactly `"__main__"` with two underscores on each side. `"_main_"` or `"__main__ "` with a trailing space is a different string, the condition is always false, and the block never runs.
- **Comparing to your module name.** Writing `if __name__ == "calc":` to "catch import" is unreliable: when imported from a package `__name__` becomes `"pkg.calc"` and the condition fails. With `"__main__"` there are no such surprises — that value is fixed.
- **Putting definitions inside the guard.** If you move a function or class under the guard, on import they will not exist at all, and `from module import func` fails with `ImportError`. Definitions must live outside.
- **Thinking the guard speeds up import by itself.** It only skips its own block; heavy top-level code still runs on import.

<Detail summary="Why this form became the standard, not a flag function">

You could imagine a built-in like `is_main()`, and sometimes beginners write their own wrappers. But the idiom `if __name__ == "__main__"` won because it requires no imports and relies only on a variable the interpreter already sets for every module. It works in any Python without dependencies, reads the same in every project, and adds no imports. The official docs recommend exactly this form and the form with `main()` under the guard as the canonical way to structure a module entry point. Recognizability is value too: anyone reading your code instantly knows where "run" begins.

</Detail>
