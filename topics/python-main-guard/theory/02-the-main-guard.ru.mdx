# Одна строка, которая делит файл на «для запуска» и «для импорта»

Представьте файл `calc.py`: внутри полезная функция `add`, а в конце пара строк, которые её вызывают и печатают результат - чтобы при запуске файла сразу видеть, что всё работает. Удобно, пока файл живёт сам по себе. Но как только Вы напишете `from calc import add` в другом скрипте, эти финальные строки исполнятся прямо во время импорта: посреди чужой программы внезапно напечатается результат, которого там быть не должно. Файл, который хорошо работал «на запуск», начинает мусорить при «импорте». Идиома `if __name__ == "__main__"` существует ровно для того, чтобы развести эти два режима.

К концу раздела Вы научитесь читать и писать этот guard, понимать, какой код кладут внутрь него, а какой оставляют снаружи, и видеть, какую конкретно проблему он устраняет. Из прошлого раздела Вы уже знаете, что `__name__` равно `"__main__"` только при запуске файла - теперь мы сделаем из этого факта рабочий инструмент.

## Ментальная модель: один файл, два читателя

У модуля два разных «читателя». Первый - человек или операционная система, которые запускают файл командой `python calc.py`: им нужен код, который что-то делает прямо сейчас - читает аргументы, печатает результат, запускает программу. Второй читатель - другой модуль, который пишет `import calc`: ему нужны только определения, функции и классы, без побочных действий. Guard `if __name__ == "__main__"` - это переключатель, который показывает первому читателю код запуска и прячет его от второго.

На схеме определение `add` видно обоим. А блок под guard соединён стрелкой только с прямым запуском: при импорте поток до него просто не доходит, потому что условие `__name__ == "__main__"` ложно. Теперь разложим это по коду.

## Минимальный пример: до и после

Вот файл без guard. При импорте `add` строка с `print` исполнится и напечатает результат там, где её не ждут.

```python
def add(a, b):
    return a + b

print("2 + 3 =", add(2, 3))
```

Оборачиваем код запуска в guard - и `print` исполняется только при прямом запуске файла, а при импорте молчит.

```python
def add(a, b):
    return a + b

if __name__ == "__main__":
    print("2 + 3 =", add(2, 3))
```

Определение `add` осталось снаружи guard - оно нужно обоим читателям. Вызов и печать ушли внутрь - они нужны только при запуске. Это и есть весь приём: определения снаружи, действия внутри.

## Что кладут внутрь guard, а что снаружи

Правило простое и почти всегда работает: **определения** (функции, классы, константы) живут снаружи guard, **действия** (вызовы, печать, чтение аргументов командной строки, запуск программы) - внутри. Распространённая форма - вынести всю логику запуска в функцию `main()` и оставить под guard единственный вызов.

```python
def add(a, b):
    return a + b

def main():
    print("2 + 3 =", add(2, 3))

if __name__ == "__main__":
    main()
```

Такая форма читается сразу: всё, что выше guard, - это переиспользуемые кирпичики; единственная строка под guard говорит «а вот так файл запускается сам по себе». Функция `main()` при этом тоже остаётся доступной для импорта - её можно вызвать из теста.

<Callout type="tip">
Если сомневаетесь, куда положить строку: спросите себя, должна ли она исполниться, когда кто-то напишет `import ваш_модуль`. Если нет - ей место внутри guard.
</Callout>

## Потрогайте guard руками

Песочница исполняет код как главную программу, поэтому `__name__` равно `"__main__"` и блок под guard срабатывает. Запустите как есть - увидите вывод из `main()`. Теперь поменяйте условие на `if __name__ == "library":` и запустите снова: условие станет ложным, `main()` не вызовется, и Вы увидите ровно то, что происходит при импорте - тишину вместо вывода. Так выглядит модуль «глазами того, кто его импортировал».

Заметьте: строка `print("Тело модуля исполнилось")` стоит вне guard и печатается всегда - и при запуске, и (в реальном проекте) при импорте. Только вызов `main()` под guard зависит от режима. Это наглядно показывает границу между «исполняется всегда» и «исполняется только при запуске».

## Сюрприз: guard не делает импорт бесплатным

Легко решить, что guard «отключает» весь файл при импорте. Это не так. При импорте исполняется всё тело модуля - просто блок под `if __name__ == "__main__"` пропускается, потому что условие ложно. Любой код вне guard - в том числе тяжёлый: открытие соединения, чтение большого файла, долгий цикл - всё равно выполнится при импорте.

```python
data = [x * x for x in range(1000000)]

def get_squares():
    return data

if __name__ == "__main__":
    print(len(get_squares()))
```

Список из миллиона элементов вычисляется на верхнем уровне, поэтому он построится при любом `import` этого модуля, хотя `print` под guard при импорте не сработает. Guard прячет только то, что внутри него; всё, что снаружи, по-прежнему исполняется при импорте. Дорогую инициализацию стоит либо убирать под guard, либо прятать внутрь функции, которая вызывается только при необходимости.

## Типичные ошибки

- **Опечатка в `"__main__"`.** Строка должна быть ровно `"__main__"` с двумя подчёркиваниями с каждой стороны. `"_main_"` или `"__main__ "` с пробелом - это другая строка, условие всегда будет ложным, и блок не исполнится никогда.
- **Сравнивать с именем своего модуля.** Писать `if __name__ == "calc":`, чтобы «поймать импорт», ненадёжно: при импорте из пакета `__name__` станет `"pkg.calc"`, и условие не сработает. С `"__main__"` таких сюрпризов нет - это значение фиксировано.
- **Класть определения внутрь guard.** Если функцию или класс убрать под guard, при импорте их не будет вовсе, и `from module import func` упадёт с `ImportError`. Определения должны жить снаружи.
- **Думать, что guard ускоряет импорт сам по себе.** Он лишь пропускает свой блок; тяжёлый код на верхнем уровне всё равно исполнится при импорте.

<Detail summary="Почему именно эта форма стала стандартом, а не функция-флаг">

Можно было бы представить себе встроенную функцию вроде `is_main()`, и иногда новички пишут собственные обёртки. Но идиома `if __name__ == "__main__"` выиграла, потому что не требует ничего импортировать и опирается только на переменную, которую интерпретатор и так задаёт каждому модулю. Она работает в любом Python без зависимостей, читается одинаково во всех проектах и не добавляет ни одного импорта. Официальная документация прямо рекомендует именно эту форму и форму с `main()` под guard как канонический способ оформить точку входа модуля. Узнаваемость тут - тоже ценность: любой, кто читает Ваш код, мгновенно понимает, где начинается «запуск».

</Detail>
