# You're always standing somewhere: first steps in the terminal

A coworker sends instructions: "open a terminal, go to the project directory, and run the script." You open the black window, see a blinking cursor, and realize there's nothing here to click. No folder icons, no address bar, no back button. Every typo feels threatening, and worst of all, it's completely unclear where you even ended up.

The good news: three commands are enough to navigate with confidence. `pwd` answers "where am I," `ls` shows what's around you, and `cd` moves you somewhere else. By the end of this lesson you'll get around the filesystem faster than in any graphical file browser, and you'll stop worrying about getting lost. Getting lost here simply isn't possible.

## Terminal and shell: who you're talking to

First, a word about who's on the other end. A terminal is a window that can display text and accept keyboard input. Inside it runs a shell, an interpreter program, usually bash or zsh: you type a line, press Enter, and the shell parses the command and runs it. That's all you need to know about the internals for now. Every command in this lesson is typed after the shell prompt; in the examples it's shown as `$`, which you don't type yourself.

## One tree for the whole system

On Windows, files live on drives like `C:` and `D:`, each with its own tree. On Linux there's a single tree for the whole system. It grows from the root, a directory named `/`, and every file has an address read starting from the root. The path `/home/anna/docs/notes.txt` means: the root contains a directory called `home`, which contains `anna`, which contains `docs`, which contains the file `notes.txt`. This kind of address is called a path.

A handful of top-level directories are worth recognizing on sight. `/home` holds user home directories: for user anna that's `/home/anna`, and almost all of your work will happen inside it. `/etc` holds system configuration, `/var` holds changing data like logs, `/tmp` is a dumping ground for temporary files, and `/usr/bin` holds the actual programs, including the `ls` you're about to run.

## The "you are here" marker

Here's the one idea that keeps the terminal from feeling like chaos: the shell always stands in some directory of the tree. That directory is called the current directory, and every command looks there by default unless told otherwise. In figure 1, the current directory is highlighted: `/home/anna/docs`. If you don't know where you're standing, just ask:

```bash
$ pwd
/home/anna
```

`pwd` prints the absolute path of the current directory. Learn it first: it changes nothing, so it's safe in any situation, and it instantly answers "where am I."

<Callout type="tip" title="The prompt already tells you">
  On many systems the shell prompt already shows the current directory, for example `anna@laptop:~/docs$`. The `~` symbol in it stands for the home directory.
</Callout>

## Look around: ls

`ls` lists the contents of the current directory:

```bash
$ ls
docs  music  report.txt
```

By default, names are arranged in columns. The `-1` flag prints one name per line. The `-l` flag adds detail:

```bash
$ ls -l
drwxr-xr-x 2 anna anna 4096 Jul  8 10:20 docs
-rw-r--r-- 1 anna anna  532 Jul  9 18:03 report.txt
```

For now, skim this output lightly. The first character of the line tells a directory (`d`) from a regular file (`-`), followed by permissions, owner, size in bytes, modification date, and name. A dedicated lesson covers permissions in depth; for now it's enough to tell files from directories.

## Move around: cd

`cd` changes the current directory. A target path can be written two ways, and the difference between them matters a lot. An absolute path starts with `/`, is read from the root, and works from anywhere:

```bash
$ cd /home/anna/docs
$ pwd
/home/anna/docs
```

A relative path doesn't start with `/` and is resolved against the current directory. Standing in `/home/anna`, you can reach that same `docs` far more briefly:

```bash
$ cd docs
```

Relative paths have two special names: `..` means the parent directory, and `.` means the current one. `cd ..` moves up one level, from `/home/anna/docs` back to `/home/anna`. The dots work anywhere in the tree, because every directory contains entries for `.` and `..`.

Three more shortcuts save time every day. The `~` symbol expands to your home directory, so `cd ~/docs` takes you to `/home/anna/docs` from anywhere. `cd` with no arguments also takes you home. And `cd -` returns you to wherever you were before the last move, printing that path as it goes:

```bash
$ pwd
/home/anna
$ cd /var/log
$ cd -
/home/anna
```

The last navigation tool isn't a command, it's a key. Start typing a directory name and hit Tab: the shell finishes it for you. If several names match, a second Tab press lists them all. Autocomplete doesn't just save typing, it also catches typos: if Tab doesn't complete anything, that name probably doesn't exist in the current directory.

## What will surprise you

First surprise: `ls` doesn't show everything. Files whose names start with a dot are considered hidden. That's how configuration files like `.bashrc` hide in your home directory: the file exists, but plain `ls` stays silent about it. The `-a` flag reveals the full picture:

```bash
$ ls -a
.  ..  .bashrc  docs  music  report.txt
```

That also reveals the `.` and `..` entries that live in every directory.

Second surprise: spaces separate arguments. A directory named `my docs` can't be opened with `cd my docs`: the shell will decide you passed two separate names. Wrap the name in quotes, `cd "my docs"`, or let the Tab key handle it: autocomplete escapes the space for you.

Third surprise, and this one catches people coming from Windows: names are case-sensitive. `Report.txt` and `report.txt` are two different files, and they happily coexist in the same directory. If `cd Docs` complains that the directory doesn't exist while `ls` clearly shows `docs`, that's the culprit.

## Common mistakes

Confusing a relative path with an absolute one. `cd docs` only works if `docs` sits in the current directory; from anywhere else you'll get `No such file or directory`. That happens because a relative path is silently glued onto the current directory. The fix: run `pwd` first, then either move to the right place or use an absolute path.

Writing an absolute path where a short relative one would do: standing in `/home/anna`, typing `cd /home/anna/docs` instead of `cd docs`. Not technically wrong, just extra typing and extra chances for a typo. It usually comes from not trusting where you currently are; a glance at `pwd` restores that confidence.

Forgetting quotes around names with spaces, then getting baffling complaints about missing files. The rule is simple: any name with a space always goes in quotes.

Panicking after "getting lost." There's no such thing as "unknown location" in a terminal: `pwd` instantly tells you where you are, and `cd` with no arguments is guaranteed to bring you home. Both commands are safe, so start any investigation with them.

<Detail summary="Why cd is built into the shell instead of living in /usr/bin">
  `ls` and `pwd` are ordinary programs from `/usr/bin`. That trick wouldn't work for `cd`: the current directory belongs to a specific process. A separate program would change its own directory, exit, and the shell would be left standing right where it was. That's why the shell handles directory changes itself, and why `cd`'s documentation lives in the shell's own manual rather than in a separate man page.
</Detail>
