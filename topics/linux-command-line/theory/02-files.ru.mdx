# Создать, скопировать, перенести, удалить: файлы под вашим контролем

У Анны в домашнем каталоге скопилась сотня файлов: свежие отчёты вперемешку с черновиками, рядом `zakazy.csv` за прошлый год. Пора навести порядок: создать каталог `arhiv`, перенести туда старое, ненужное удалить. В графическом проводнике это долгий вечер перетаскивания мышью. В терминале та же уборка занимает пять команд, а с шаблонами имён и того меньше.

В этом уроке вы научитесь создавать каталоги и файлы (`mkdir`, `touch`), просматривать содержимое (`cat`), копировать (`cp`), переносить и переименовывать (`mv`) и удалять (`rm`). Заодно узнаете главную особенность терминала, о которой лучше прочитать в статье, чем узнать из потерянного отчёта: корзины здесь нет.

## Ментальная модель: команды двигают узлы дерева

Из прошлого урока вы знаете, что файловая система это дерево, а путь это адрес узла в нём. Все команды этого урока делают с деревом ровно четыре вещи: добавляют узел (`mkdir`, `touch`), создают копию узла по новому адресу (`cp`), меняют адрес узла (`mv`) и убирают узел (`rm`). ## Создать: mkdir и touch

Каталог создаёт `mkdir`:

```bash
$ mkdir arhiv
```

Если нужна вложенная структура, по умолчанию команда откажется:

```bash
$ mkdir arhiv/2026/iyul
mkdir: cannot create directory 'arhiv/2026/iyul': No such file or directory
```

`mkdir` не достраивает промежуточные каталоги сам: раз `2026` ещё не существует, положить в него `iyul` некуда. Ключ `-p` снимает ограничение: `mkdir -p arhiv/2026/iyul` создаст всю цепочку за один вызов и не станет ругаться, если её часть уже есть.

Пустой файл создаёт `touch`:

```bash
$ touch spisok.txt
```

У `touch` есть особенность: на существующем файле он ничего не стирает, а лишь обновляет время последнего изменения. Исторически команда именно для этого и придумана, а создание пустого файла всего лишь побочный эффект, которым пользуются все.

## Посмотреть: cat

```bash
$ cat zakazy.csv
id,tovar
1,kniga
2,lampa
```

`cat` печатает файл в терминал целиком. Имя происходит от concatenate, склеивать: если передать несколько файлов, `cat` выведет их подряд, одним потоком, в порядке перечисления. Для длинных файлов есть команды поудобнее: `head -n 5 fail.log` покажет первые пять строк, `tail -n 5 fail.log` последние, а `less fail.log` откроет файл в листалке, из которой выходят клавишей `q`. Запомните `less`: однажды он спасёт вас от `cat` на логе размером в гигабайт.

## Скопировать: cp

```bash
$ cp otchet.txt otchet-kopiya.txt
$ cp otchet.txt arhiv/
```

Первая форма создаёт копию под новым именем, вторая кладёт копию с прежним именем в каталог `arhiv`. Копия полностью независима: правки в ней не затрагивают оригинал. С каталогами `cp` по умолчанию не работает:

```bash
$ cp docs backup
cp: -r not specified; omitting directory 'docs'
```

Каталог это целое поддерево, и `cp` требует явного подтверждения, что вы намерены копировать его рекурсивно, со всем содержимым: `cp -r docs backup`.

## Перенести и переименовать: mv

```bash
$ mv otchet.txt arhiv/
$ mv arhiv/otchet.txt arhiv/otchet-2026.txt
```

Первая команда переносит файл в каталог, вторая переименовывает. Для `mv` это одна и та же операция: команда просто меняет адрес узла в дереве, как на рис. 1. Отсюда приятный бонус: `mv` не требует `-r` для каталогов, ведь содержимое никуда не копируется, поддерево целиком получает новый адрес.

## Удалить: rm

```bash
$ rm staryj-chernovik.txt
```

Файл исчез. Каталог `rm` без ключей не тронет:

```bash
$ rm arhiv
rm: cannot remove 'arhiv': Is a directory
```

Как и у `cp`, ключ `-r` включает рекурсивный режим: `rm -r arhiv` удалит каталог со всем содержимым. К нему часто добавляют `-f`, отключающий вопросы и предупреждения; о том, почему связка `rm -rf` заслужила дурную славу, чуть ниже.

## Шаблоны имён: звёздочка

Все команды урока принимают несколько путей сразу, и перечислять их руками не обязательно. Символ `*` в имени означает «любая последовательность символов»:

```bash
$ cp *.txt arhiv/
```

Шелл сам развернёт `*.txt` в список всех файлов текущего каталога с расширением `.txt` и передаст его `cp`. У шаблонов много возможностей, но форма `*.txt` покрывает большинство повседневных задач, и пока её достаточно.

## Что вас удивит

<PullQuote>В терминале нет корзины: rm удаляет сразу и навсегда.</PullQuote>

Главный сюрприз уже прозвучал, но заслуживает отдельного абзаца. `rm` не перемещает файлы в корзину, не показывает диалог «вы уверены?» и не умеет отменять сделанное. Команда отработала, файла нет: ни `Ctrl+Z`, ни истории версий, ни второго шанса.

Второй сюрприз тише и потому опаснее: `cp` и `mv` молча перезаписывают существующую цель. Если `b.txt` уже есть, команда `mv a.txt b.txt` уничтожит его содержимое без единого вопроса. Ключ `-i` заставляет обе команды спрашивать подтверждение перед перезаписью: `mv -i a.txt b.txt`.

Третий вы уже видели: `cp` отказывается копировать каталог без `-r` и честно сообщает об этом. Из трёх неожиданностей это единственная, где команда хотя бы подаёт голос вместо молчаливого разрушения.

## Типичные ошибки

`rm -rf` с опечаткой в пути. Хотели удалить `build`, промахнулись клавишей, удалили соседний каталог: `-f` заглушил все предупреждения, а корзины, куда можно заглянуть, не существует. Лекарство: перед `rm -r` выполните `ls` по тому же пути и убедитесь, что видите именно то, что собираетесь удалить.

`mv otchet.txt arhiv`, когда каталога `arhiv` не существует. `mv` не умеет читать намерения: раз цели-каталога нет, он решает, что вы задумали переименование, и файл просто станет называться `arhiv`. Лекарство: пишите цель с завершающим слэшем, `mv otchet.txt arhiv/`. Если каталога нет, команда честно упадёт с ошибкой вместо тихого переименования.

Забывают `-r` при копировании каталога, читают `omitting directory` и решают, что терминал сломался. Терминал в порядке: `cp` защищается от случайного копирования гигантских поддеревьев и ждёт явного разрешения.

`cat` на огромном файле: терминал захлёбывается тысячами строк, и остановить поток непросто. Причина в том, что `cat` не знает пощады и печатает всё, что ему дали. Для больших файлов берите `less`, а если нужно лишь взглянуть на край файла, `head` или `tail`.

<Detail summary="Куда на самом деле исчезает удалённый файл">
  `rm` не затирает байты на диске: он удаляет запись об имени файла из каталога и помечает занятое место как свободное. Данные физически лежат на диске, пока система не использует это место под что-то новое. Утилиты восстановления существуют, но полагаться на них нельзя: чем дольше машина работала после удаления, тем меньше шансов. Практический вывод скучный и надёжный: резервные копии вместо надежды на восстановление.
</Detail>
