# grep reads what's inside files, find reads their labels

Anna messages you: "we already agreed on a discount for this client last quarter, find where that's written down." In front of you is a directory called `projects`: fifty text files, a dozen subdirectories, and the line you need is buried in there somewhere. Opening files one at a time and hitting Ctrl+F could eat your whole evening, and you'd still risk skipping right past the one file that matters.

An hour later a second task lands: you're sure you saved a file called `orders_2024.csv`, but you don't remember where. Notice that this is a different question. In the first case you're searching for text inside files, and you don't care what those files are named. In the second you're searching for a file by name, and you don't care what's inside it. Linux has a separate command for each question: `grep` for the first, `find` for the second. New users lose more time picking the wrong tool than fighting the syntax.

Here's a picture worth keeping: a directory is like a warehouse of boxes. Every box has a label: file name, type (file or directory), size. Inside the box sits the content: lines of text. `find` walks the aisles and reads only the labels, it never looks inside a box. `grep` does the opposite: it opens every box and reads the lines, mentioning the label only to tell you where it found something.

## grep: what's inside files

A basic call takes a pattern and a file:

```bash
$ grep "discount" report.txt
discount for client: 15%
```

grep read `report.txt` line by line and printed every line where the pattern turned up. No matches, and grep simply stays quiet. Notice it prints the whole line, not just the matched piece.

Case matters by default: "Anna" and "anna" are different strings to grep. The `-i` flag turns off that strictness:

```bash
$ grep -i "anna" list.txt
Anna Petrova
email: anna@company.com
```

The `-n` flag adds a line number, and the output shifts to `number:line` form:

```bash
$ grep -n "total" report.txt
12:total for the quarter: 480
```

The `-c` flag prints a single number instead of lines: how many lines contain the pattern. Lines, not occurrences: two instances of "boston" on one line still count as one.

```bash
$ grep -c "boston" orders.csv
7
```

The `-v` flag flips the condition and shows lines where the pattern is absent. Flags combine freely: `-vc` counts the lines without the pattern.

Up to now grep has worked on a single file. The `-r` flag (recursive) runs it across a directory: grep walks every file in every subdirectory itself. Each matched line gets a path prepended:

```bash
$ grep -rn "TODO" src
src/app.py:14:# TODO: add permission check
src/utils.py:3:# TODO: rename function
```

Remember the three output shapes: plain, it's just the line; with `-n` you get `number:line`; with `-r` you get `path:line`, and with `-rn` both path and number together.

<Callout type="tip" title="Pick the right tool in a second">
Ask yourself what you want back. Lines of text: grep. Paths to files: find.
</Callout>

## find: where the files themselves are

find works the other way around: first where to look, then what conditions to check.

```bash
$ find . -name "orders_2024.csv"
./archive/2024/orders_2024.csv
```

The dot here means the starting directory, i.e. the current one. find descends into every subdirectory at any depth and prints the path of each file whose name matches the pattern. find's pattern isn't "text inside the name," it's a glob: an asterisk means "any characters," and the name has to match in full:

```bash
$ find . -name "*.csv"
./orders.csv
./archive/2024/orders_2024.csv
```

The `-type` condition filters by type: `f` keeps regular files, `d` keeps directories. `-maxdepth` limits how deep find descends. Together:

```bash
$ find . -maxdepth 1 -type d
.
./archive
./src
```

The first line of output surprises everyone: a dot. find checks its condition on the starting directory itself too, and `.` matches the condition "directory," so it honestly shows up in the results. It's not a bug, it's just rarely mentioned.

The `-size` condition checks size: plus means "larger than," minus means "smaller than":

```bash
$ find . -type f -size +10M
./archive/db_dump.sql
```

## Why find's pattern lives in quotes

The command `find . -name *.csv` with no quotes looks harmless, but breaks in the sneakiest way possible. The asterisk is seen first by the shell, not by find: it expands `*.csv` into a list of `.csv` files in the current directory before find even starts. If there are two such files in the current directory, find gets an extra word and refuses to run:

```bash
$ ls -1
report_february.csv
report_january.csv
$ find . -name *.csv
find: paths must precede expression: 'report_january.csv'
```

Worse, if there's exactly one csv in the current directory: the shell substitutes its name, find "works," but only ends up searching the whole tree for that single name. You won't see an error, you'll just get an incomplete result. Quotes turn off the expansion: the shell passes `*.csv` through as-is, and find itself compares the pattern against every name in the tree.

grep has a similar trap, but with spaces. A two-word pattern with no quotes falls apart into a pattern plus a filename:

```bash
$ grep anna petrova list.txt
grep: petrova: No such file or directory
list.txt:anna petrova
```

grep decided the pattern was `anna`, while `petrova` and `list.txt` were two files to search. This also reveals another detail: when there are multiple files, grep prepends the filename to every matched line. Getting into the habit of always quoting your pattern clears up both surprises at once.

And one last trap from the same family: pointing grep at a directory without `-r`:

```bash
$ grep "TODO" src
grep: src: Is a directory
```

grep without `-r` can only read files. Seeing "Is a directory" means: add `-r`, or point at a specific file.

## Common mistakes

**Searching for text with find.** The command `find . -name "prepayment"` stays quiet even though that exact word appears in three files. `-name` compares the pattern against file names, find never opens the content. This mistake grows out of the feeling that "find means find anything." The fix: search text with grep: `grep -r "prepayment" .`.

**An unquoted `-name` pattern.** The command works on one machine and breaks on another, because the result depends on what files happen to be in the current directory. The fix: `-name "*.csv"`, quotes, always.

**Forgetting the starting path.** The call `find -name "*.csv"` usually works fine on Linux: GNU find quietly substitutes a dot. On macOS and other BSD systems the same command fails with an error. Write the starting directory explicitly, `find . -name "*.csv"`, and that habit will save you when you move between systems.

<Detail summary="grep's pattern is actually a regular expression">
grep treats the pattern as a regular expression, it's just usually invisible on plain text. The first surprise usually comes from a dot: in regular expressions it means "any single character," so `grep "1.5" prices.txt` matches "1.5," "135," and "1x5" alike. For now it's enough to know this exists; if you need a literal search with no magic, there's a `-F` flag. Regular expressions themselves are a whole separate topic.
</Detail>

<Detail summary="find can filter by almost anything">
Besides name and type, find has conditions for modification time (`-mtime -7`: changed in the last seven days), owner (`-user anna`), emptiness (`-empty`). And the `-exec` flag runs a command on every file it finds. `-name`, `-type`, and `-maxdepth` are enough to start with; bring in the rest as you need them.
</Detail>
