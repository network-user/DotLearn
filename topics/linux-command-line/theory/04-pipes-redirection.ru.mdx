# Собираем конвейер: потоки, перенаправление, пайпы

Файл `access.log` весит триста мегабайт, и Борису нужен всего один ответ: сколько раз за сегодня встретилась ошибка `404`. Открыть файл в редакторе и листать его руками нереально, а писать отдельную программу ради одного числа избыточно. В терминале эта задача решается тремя маленькими командами, соединёнными в одну строку: одна печатает файл, вторая отбирает нужные строки, третья их считает. Соединение команд друг с другом и есть тема этого урока.

## Три потока у каждой команды

У каждой запущенной команды есть три именованных канала для текста: стандартный ввод (`stdin`, номер 0), стандартный вывод (`stdout`, номер 1) и поток ошибок (`stderr`, номер 2). По умолчанию все три подключены к терминалу: `stdin` читает то, что вы набираете на клавиатуре, `stdout` и `stderr` печатают на экран, хотя выглядят одинаково.

```bash
$ ls no-such-folder
ls: cannot access 'no-such-folder': No such file or directory
```

Сообщение об ошибке ушло не в `stdout`, а в `stderr`: экран этого не показывает, но разница станет заметна, как только вы начнёте перенаправлять потоки по отдельности.

## Перенаправление в файл: `>`, `>>`, `<`

Оператор `>` подменяет `stdout`: вместо экрана вывод команды попадает в указанный файл, полностью заменяя его прежнее содержимое.

```bash
$ ls -1 > listing.txt
$ cat listing.txt
arhiv
deploy.sh
otchet.txt
```

Если файл уже существовал, его содержимое исчезает ещё до того, как команда слева успела что-либо напечатать: шелл открывает файл на запись и обрезает его до нуля байт в момент разбора команды, а не в момент завершения. Когда содержимое нужно дописать, а не заменить, берут `>>`:

```bash
$ echo "деплой завершён" >> log.txt
```

Каждый вызов с `>>` добавляет строку в конец файла, не трогая то, что там уже написано. Есть и обратное направление: `<` подаёт содержимое файла на `stdin` команды вместо клавиатуры.

```bash
$ sort < names.txt
anna
boris
irina
```

## Перенаправление ошибок: `2>`

Поток ошибок перенаправляется тем же способом, но с явным номером `2` перед знаком:

```bash
$ find / -name "*.conf" 2> errors.log
```

Обычные найденные пути по-прежнему летят на экран, а все жалобы вроде `Permission denied` из системных каталогов тихо оседают в `errors.log`. Свести оба потока в один файл можно записью `2>&1`, которая означает «направь поток 2 туда же, куда сейчас направлен поток 1»:

```bash
$ find / -name "*.conf" > all-output.log 2>&1
```

Порядок здесь принципиален. Строка выше сначала переключает `stdout` на файл, а уже потом говорит потоку ошибок следовать за первым, поэтому оба оказываются в `all-output.log`. Если поменять операторы местами, `find / -name "*.conf" 2>&1 > all-output.log`, `stderr` скопирует текущую цель `stdout`, то есть в этот момент ещё терминал, а уже потом `stdout` в одиночку переключится на файл. Итог: ошибки продолжат сыпаться на экран, а в файле окажутся только обычные строки.

<Callout type="tip" title="Как прочитать 2>&1 без путаницы">
Читайте справа налево: «направь то, что справа, туда же, куда сейчас смотрит то, что слева». Ampersand перед единицей означает «это номер потока, а не имя файла с названием 1».
</Callout>

## Конвейер: | соединяет команды

Оператор `|` (пайп) подключает `stdout` одной команды прямо к `stdin` следующей, без промежуточного файла. Команды при этом запускаются как отдельные процессы одновременно, и данные текут между ними потоком, по мере готовности.

```bash
$ cat access.log | grep 404 | wc -l
37
```

Ответ на задачу Бориса теперь одна строка: `cat` печатает файл, `grep 404` пропускает только строки с этим кодом, `wc -l` считает строки на входе. До экрана долетает единственное число, всё промежуточное существует лишь мгновение внутри конвейера. Команд в цепочке может быть сколько угодно, каждая получает на входе результат предыдущей.

<PullQuote>Пайп соединяет только stdout со stdin. Поток ошибок по умолчанию в конвейер не попадает.</PullQuote>

## Что вас удивит

Самый опасный сюрприз: `>` обрезает файл раньше, чем команда успевает его прочитать. Команда `sort names.txt > names.txt`, которая на первый взгляд должна отсортировать файл и сохранить результат туда же, на практике уничтожает исходные данные. Шелл открывает `names.txt` на запись и обнуляет его в момент разбора строки, и только потом `sort` пытается его прочитать, а читать уже нечего. Лекарство простое: сортировать во временный файл и затем переименовать его на место оригинала, либо пользоваться специальным ключом `-o` там, где команда его поддерживает.

Второй сюрприз: пайп передаёт только `stdout`, а сообщения об ошибках всё так же летят на экран мимо конвейера. Если нужно, чтобы следующая команда в цепочке видела и ошибки тоже, их сначала объединяют записью `2>&1` перед знаком `|`.

Третий: `>` и `>>` работают только с файлами, но не годятся для передачи данных между командами. Попытка написать `ls > grep txt` не соединит команды, а всего лишь создаст файл с очень странным именем.

## Типичные ошибки

**`command > file.txt` вместо `>>`.** Хотели дописать строку в лог, а вместо этого стёрли всю его историю: единственный `>` заменяет файл целиком. Если цель именно накопление, всегда `>>`.

**Перепутанный порядок `2>&1` и `>`.** Как показано выше, `2>&1 > file` и `> file 2>&1` дают разный результат. Правило запоминания: сначала направьте `stdout` туда, куда нужно, и только потом просите `stderr` последовать за ним.

**Ждут, что пайп передаст и ошибки.** Команда `build.sh | grep error` пропустит настоящие ошибки сборки мимо `grep`, если они печатаются в `stderr`, а не в `stdout`. Нужно сначала объединить потоки: `build.sh 2>&1 | grep error`.

**Читают вывод сортировки файла в себя же.** Классическая порча данных из раздела выше: `sort data.txt > data.txt`. Через минуту после такой команды `data.txt` пуст, и восстановить его нечем, если не было резервной копии.

<Detail summary="/dev/null: место, куда выводы уходят бесследно">
Иногда вывод не нужен вообще, и `/dev/null` служит специальным файлом-пустотой: всё, что в него записано, немедленно и безвозвратно исчезает. Команда `command > /dev/null 2>&1` полностью заглушает и обычный вывод, и ошибки; такую строку часто встречают в фоновых задачах и планировщиках, где важен только факт выполнения, а не текст на экране.
</Detail>
