# grep читает содержимое файлов, find читает их этикетки

Анна пишет вам в чат: «мы уже согласовывали скидку этому клиенту в прошлом квартале, найди, где это зафиксировано». Перед вами каталог proekty: полсотни текстовых файлов, десяток подкаталогов, и где-то в этой куче лежит нужная строчка. Открывать файлы по одному и жать Ctrl+F можно до вечера, и всё равно останется шанс промахнуться мимо нужного файла.

Через час прилетает вторая задача: вы точно помните, что сохраняли файл zakazy_2024.csv, но не помните куда. Обратите внимание: это другой вопрос. В первом случае вы ищете текст внутри файлов, и вам всё равно, как эти файлы называются. Во втором вы ищете сам файл по имени, и вам всё равно, что у него внутри. В Linux под каждый вопрос есть отдельная команда: grep для первого, find для второго. Больше всего времени новички теряют не на синтаксисе, а на том, что берут не тот инструмент.

Удобная картинка: каталог похож на склад коробок. На каждой коробке есть этикетка: имя файла, тип (файл или каталог), размер. Внутри коробки лежит содержимое: строки текста. find ходит вдоль стеллажей и читает только этикетки, внутрь он не заглядывает никогда. grep действует наоборот: открывает каждую коробку и читает строки, а этикетку упоминает только затем, чтобы сказать, где нашёл.

## grep: что внутри файлов

Базовый вызов состоит из шаблона и файла:

```bash
$ grep "skidka" otchet.txt
skidka dlya klienta: 15%
```

grep прочитал otchet.txt строка за строкой и напечатал каждую строку, где встретился шаблон. Совпадений нет: grep просто молчит. Заметьте: печатается строка целиком, а не только совпавший кусок.

Регистр по умолчанию важен: «Anna» и «anna» для grep разные строки. Ключ `-i` выключает эту строгость:

```bash
$ grep -i "anna" spisok.txt
Anna Petrova
pochta: anna@firma.ru
```

Ключ `-n` добавляет номер строки, и вывод меняет форму на `номер:строка`:

```bash
$ grep -n "itogo" otchet.txt
12:itogo za kvartal: 480
```

Ключ `-c` вместо строк печатает одно число: сколько строк содержат шаблон. Именно строк, а не вхождений: две «moskva» в одной строке дадут единицу, а не двойку.

```bash
$ grep -c "moskva" zakazy.csv
7
```

Ключ `-v` переворачивает условие и показывает строки, где шаблона нет. Ключи свободно комбинируются: `-vc` посчитает строки без шаблона.

До сих пор grep работал с одним файлом. Ключ `-r` (recursive) запускает его по каталогу: grep сам обойдёт все файлы во всех подкаталогах. К каждой найденной строке добавится путь:

```bash
$ grep -rn "TODO" src
src/app.py:14:# TODO: dobavit proverku prav
src/utils.py:3:# TODO: pereimenovat funkciyu
```

Запомните три формы вывода: обычно это просто строка; с `-n` получается `номер:строка`; с `-r` получается `путь:строка`, а с `-rn` путь и номер вместе.

<Callout type="tip" title="Как выбрать инструмент за секунду">
Спросите себя, что вы хотите получить в ответ. Нужны строки текста: grep. Нужны пути к файлам: find.
</Callout>

## find: где сами файлы

У find другой порядок мыслей: сначала откуда искать, потом какие условия проверять.

```bash
$ find . -name "zakazy_2024.csv"
./arhiv/2024/zakazy_2024.csv
```

Точка здесь означает стартовый каталог, то есть текущий. find спускается во все подкаталоги на любую глубину и печатает путь каждого файла, чьё имя совпало с шаблоном. Шаблон у find не «текст внутри имени», а glob: звёздочка означает «любые символы», и имя должно подойти целиком:

```bash
$ find . -name "*.csv"
./zakazy.csv
./arhiv/2024/zakazy_2024.csv
```

Условие `-type` отбирает по типу: `f` оставляет обычные файлы, `d` каталоги. Условие `-maxdepth` ограничивает глубину погружения. Вместе:

```bash
$ find . -maxdepth 1 -type d
.
./arhiv
./src
```

Первая строка вывода удивляет всех: точка. find проверяет условие и на самом стартовом каталоге, а `.` подходит под условие «каталог», поэтому честно попадает в результат. Это не баг, об этом просто редко предупреждают.

Размер проверяет условие `-size`: плюс означает «больше», минус «меньше»:

```bash
$ find . -type f -size +10M
./arhiv/dump_bazy.sql
```

## Почему шаблон find живёт в кавычках

Команда `find . -name *.csv` без кавычек выглядит безобидно, но ломается самым коварным образом. Звёздочку первым видит не find, а шелл: он раскрывает `*.csv` в список csv-файлов текущего каталога ещё до того, как find запустится. Если таких файлов в текущем каталоге два, find получит лишнее слово и откажется работать:

```bash
$ ls -1
otchet_fevral.csv
otchet_yanvar.csv
$ find . -name *.csv
find: paths must precede expression: 'otchet_yanvar.csv'
```

Хуже, когда csv в текущем каталоге ровно один: шелл подставит его имя, find «заработает», но станет искать по всему дереву только это одно имя. Ошибки вы не увидите, а результат окажется неполным. Кавычки выключают раскрытие: шелл передаёт `*.csv` внутрь find как есть, и уже сам find сравнивает шаблон с каждым именем в дереве.

У grep похожая история, только с пробелами. Шаблон из двух слов без кавычек разваливается на шаблон и имя файла:

```bash
$ grep anna petrova spisok.txt
grep: petrova: No such file or directory
spisok.txt:anna petrova
```

grep решил, что шаблон это `anna`, а `petrova` и `spisok.txt` два файла для поиска. Заодно здесь видна ещё одна деталь: когда файлов несколько, grep добавляет к каждой найденной строке имя файла. Привычка всегда брать шаблон в кавычки избавляет от обоих сюрпризов разом.

И последняя ловушка из той же серии: натравить grep на каталог без `-r`:

```bash
$ grep "TODO" src
grep: src: Is a directory
```

grep без `-r` умеет читать только файлы. Увидели «Is a directory»: добавьте `-r` или укажите конкретный файл.

## Типичные ошибки

**Ищут текст find'ом.** Команда `find . -name "predoplata"` молчит, хотя слово точно есть в трёх файлах. Условие `-name` сравнивает шаблон с именами файлов, содержимое find не открывает. Ошибка растёт из ощущения «find значит найти что угодно». Исправление: текст ищет grep: `grep -r "predoplata" .`.

**Шаблон -name без кавычек.** Команда работает на одной машине и ломается на другой, потому что результат зависит от того, какие файлы лежат в текущем каталоге. Исправление: `-name "*.csv"`, кавычки всегда.

**Забывают стартовый путь.** Вызов `find -name "*.csv"` на Linux обычно срабатывает: GNU find молча подставляет точку. На macOS и других BSD-системах та же команда падает с ошибкой. Пишите стартовый каталог явно: `find . -name "*.csv"`, эта привычка спасёт при переходе между системами.

<Detail summary="Шаблон grep это на самом деле регулярное выражение">
grep трактует шаблон как регулярное выражение, просто на обычном тексте это незаметно. Первый сюрприз обычно приносит точка: в регулярных выражениях она означает «любой один символ», поэтому `grep "1.5" ceny.txt` найдёт и «1.5», и «135», и «1x5». Пока достаточно помнить об этом; если нужен буквальный поиск без всякой магии, есть ключ `-F`. Сами регулярные выражения отдельная большая тема.
</Detail>

<Detail summary="find умеет фильтровать почти по чему угодно">
Кроме имени и типа, у find есть условия по времени изменения (`-mtime -7`: менялся за последние семь дней), по владельцу (`-user anna`), по пустоте (`-empty`). А ключ `-exec` выполняет команду над каждым найденным файлом. Для старта хватает `-name`, `-type` и `-maxdepth`, остальное подключайте по мере надобности.
</Detail>
