# Who's running right now: ps, top, and kill

A browser tab froze, the laptop fan is screaming at full speed, and the "close window" button does nothing. Rebooting the whole machine over one frozen tab feels wasteful when fifteen other things are open at the same time. What you need is a way to see everything currently running, spot the culprit, and stop exactly that one thing. That's what this lesson's three commands do: `ps` lists what's running, `top` shows that same list live, and `kill` stops a specific process.

## A process and its PID

Every running program, from your browser down to the `ls` you typed a second ago, exists in the system as a process: a separate entry with its own memory and its own number. That number is called a PID, Process ID, and it's unique among processes running right now. The system tells apart two open windows of the same browser by PID, not by program name.

## ps: a snapshot of running processes

`ps` prints the list of processes at this exact moment, a snapshot, not a live feed. The most common call, `ps aux`, shows processes from every user on the system:

```bash
$ ps aux
USER   PID  %CPU  %MEM  COMMAND
anna   1042  0.1   0.3  bash
anna   2087 97.4   4.1  firefox
anna   2200  0.0   0.2  ps
```

Five columns matter for now out of the whole line. `USER` names the process owner, `PID` its unique number, `%CPU` and `%MEM` the share of processor and memory it's using, and `COMMAND` the name of the running program. The example already points at the overheating culprit: `firefox` at PID 2087 is eating nearly the whole processor.

The list is often longer than the screen, and it's easy to narrow down with the same `grep` you already know from the search lesson:

```bash
$ ps aux | grep firefox
anna   2087 97.4   4.1  firefox
anna   2201  0.0   0.1  grep firefox
```

Don't let the second line throw you: `grep` itself becomes a process and lands in its own list, because it was searching for the word `firefox` among the command-line arguments, and its own argument happened to be the word `firefox`. The line you actually want is, as usual, the first one.

## top: the same list, live

`ps` gives you a frozen frame, `top` opens a continuously updating table of the same data, sorted by processor load. It launches with no arguments:

```bash
$ top
```

The screen fills with rows, the hungriest processes on top, numbers refreshing every few seconds. `top` is an interactive program rather than a command with plain text output: parsing it in scripts is awkward, but for quickly diagnosing "what's eating the processor right now" it's indispensable. Exit `top` with a single key: `q`.

## Stopping a process: kill and signals

Once you've found the culprit's PID, `kill` stops it:

```bash
$ kill 2087
```

The name is misleading: `kill` doesn't necessarily kill a process instantly or brutally. What it actually does is send the process a signal, a short system message, and the process itself decides how to respond. Without specifying a signal number, `kill` sends signal 15, `SIGTERM`: a polite request to exit, which a well-written program has time to honor by saving unsaved data and closing files before it shuts down.

If a process is frozen enough that it doesn't even respond to the request, there's a harsher signal:

```bash
$ kill -9 2087
```

You don't have to hunt for a PID by hand with `ps aux | grep`: the `pkill` command takes a program name directly and finds the matching processes itself.

```bash
$ pkill firefox
```

## Background, pause, interrupt

A long-running command can be sent to the background by adding `&` at the end of the line: the terminal immediately hands back the prompt without waiting for it to finish. `Ctrl+Z` pauses an already-running command, and `Ctrl+C` interrupts it outright by sending the same `SIGTERM`. The `jobs` command lists the current terminal's paused and background jobs, and `fg` brings one of them back to the foreground.

## What will surprise you

`kill` doesn't guarantee an instant stop. Plain signal 15 is a request, not an order: a program can catch it, finish whatever it's doing, and only then exit, while a frozen program might not react at all. The illusion of an instant result shatters right here, and that's when `-9` comes into play.

A process's PID isn't permanent and gets reused. A finished process frees up its number, and sooner or later the system hands that same number to the next program it starts. A PID hardcoded into a script risks pointing at a completely different program one day.

`ps aux | grep firefox` almost always turns up one line more than expected: `grep` itself shows up in the process list, since it too just started and got a PID, and the word you searched for happens to be among its own arguments.

## Common mistakes

**Reaching straight for `kill -9`.** It works, but denies the program any chance to save data or close files cleanly: an interrupted write at that moment can leave a file corrupted. The first step should almost always be plain `kill`, with `-9` reserved for when the process clearly isn't responding.

**Confusing a PID from an old terminal session with the current one.** Time passes between running `ps aux` and calling `kill`, and in that gap the process might have exited on its own, its number already reassigned to something else. If `kill`'s result looks off, it's safer to double-check with a fresh `ps aux | grep`.

**Closing the terminal window and expecting the background process to stop with it.** Sometimes it does, sometimes a background command keeps running after the window closes, depending on shell settings. Relying on closing a window as a way to stop a process isn't reliable; use `kill` explicitly.

<Detail summary="Other signals you'll run into later">
There are more than two signals in the system. `SIGINT`, signal 2, is exactly what `Ctrl+C` sends. `SIGHUP`, signal 1, historically meant the terminal connection was lost, and many server programs today use it as a command to reload configuration without a full restart. The `kill -l` command lists every signal and its number.
</Detail>
