# Откуда шелл знает, где ваш дом и где лежат команды

Вы скачали утилиту `deploy`, положили её в `/opt/tools/bin`, набрали `deploy` и получили в ответ `command not found`. При этом `ls` запускается мгновенно, `cat` тоже, хотя вы понятия не имеете, где физически лежат их файлы. Почему шелл одни программы находит сам, а другие в упор не замечает, даже когда файл точно существует?

Вторая загадка из той же серии. Вчера вы задали в терминале переменную с адресом тестового сервера, а сегодня открыли новый терминал, и переменной нет. Скрипт из соседней вкладки не видел её даже вчера. Обе истории объясняются одним механизмом: переменными окружения. Стоит один раз разобраться, как этот механизм устроен, и подобные проблемы перестанут выглядеть мистикой, которую лечат перезагрузкой.

## Ментальная модель: рюкзак со строками и список каталогов

У каждого процесса в Linux есть окружение: набор именованных строк, который процесс носит с собой, как рюкзак с настройками. Там записан домашний каталог `HOME`, язык интерфейса `LANG`, редактор по умолчанию `EDITOR` и десятки других значений. Программы заглядывают в этот набор за настройками: git открывает редактор из `EDITOR`, менеджер пакетов смотрит на адрес прокси, а сам шелл подставляет `HOME` вместо тильды в путях.

Одна переменная в наборе важнее прочих: `PATH`. Она хранит упорядоченный список каталогов, разделённых двоеточием, например `/usr/local/bin:/usr/bin:/bin`. Когда вы набираете имя команды без пути, шелл не сканирует весь диск. Он идёт по каталогам из `PATH` строго слева направо и, как только находит исполняемый файл с нужным именем, запускает его и прекращает поиск. Вторая опора модели: наследование. Каждый процесс в момент старта получает копию окружения родителя, и дальше эти копии живут независимо друг от друга. Из этого правила вырастает почти всё поведение, которое сбивает новичков с толку, поэтому ниже оно разобрано на живом примере.

## Чтение и создание переменных

Значение отдельной переменной подставляется через знак доллара:

```bash
$ echo $HOME
/home/anna
$ echo $PATH
/usr/local/bin:/usr/bin:/bin
```

Весь набор целиком печатают команды `env` и `printenv`. Вывод длинный, поэтому его удобно фильтровать или спрашивать конкретное имя:

```bash
$ printenv LANG
ru_RU.UTF-8
$ env | grep EDITOR
EDITOR=nano
```

Теперь создание. Присваивание пишется слитно, без пробелов вокруг знака равенства:

```bash
$ CITY=Kazan
$ echo $CITY
Kazan
```

Выглядит как полноценная переменная окружения, но пока это только переменная шелла: она существует внутри текущего процесса bash и никуда не передаётся. Проверить это легко, запустив дочерний процесс:

```bash
$ bash -c 'echo [$CITY]'
[]
$ export CITY
$ bash -c 'echo [$CITY]'
[Kazan]
```

Команда `bash -c` запускает новый шелл и выполняет в нём переданную строку. Пока `CITY` не экспортирована, потомок получает окружение без неё, и подстановка даёт пустоту: между скобками ничего нет. После `export CITY` переменная попадает в окружение, и следующий потомок уже видит значение. Скобки в примере стоят ровно затем, чтобы пустая строка была заметна: без них первая команда напечатала бы невидимую пустую строку, и различие легко проглядеть.

<PullQuote>Каждый процесс получает копию окружения родителя в момент старта. Обратной дороги у изменений нет: ни вверх, ни в уже запущенные процессы.</PullQuote>

Отсюда же следует ответ на загадку про соседнюю вкладку терминала. Две вкладки не родитель и потомок, а два независимых процесса, у каждого своя копия окружения. Переменная, заданная в одной вкладке, физически не может появиться в другой.

## Возвращаем deploy к жизни

Диагноз ставит команда `which`: она показывает, какой файл шелл запустит по данному имени.

```bash
$ which ls
/usr/bin/ls
$ which deploy
$
```

Для `deploy` ответ пуст: ни один каталог из `PATH` не содержит такого файла. Лекарство очевидно из модели: каталог `/opt/tools/bin` нужно добавить в список, сохранив всё, что там уже лежит.

```bash
$ export PATH="$PATH:/opt/tools/bin"
$ which deploy
/opt/tools/bin/deploy
```

Подстановка `$PATH` внутри кавычек разворачивается в старый список, двоеточие отделяет новый каталог. Порядок имеет значение. Допишете в конец, и каталог станет запасным вариантом: шелл заглянет туда, только если не нашёл имя раньше. Допишете в начало, `export PATH="/opt/tools/bin:$PATH"`, и ваши файлы начнут перекрывать одноимённые системные команды. Иногда так делают сознательно, например чтобы подменить версию интерпретатора, но по умолчанию безопаснее дописывать в конец.

<MarginNote label="Почему export">Слово export пришло из семидесятых, из оригинального шелла Борна: переменную «экспортируют» из личного пространства шелла во внешний мир, доступный потомкам.</MarginNote>

## Настройки, которые переживут перезапуск

`export` живёт до конца сессии. Закрыли терминал, открыли новый, и `PATH` снова прежний. Постоянные настройки записывают в файл `~/.bashrc`: bash читает его при старте каждого интерактивного шелла и выполняет всё, что там написано.

```bash
$ echo 'export PATH="$PATH:/opt/tools/bin"' >> ~/.bashrc
$ source ~/.bashrc
```

Первая строка дописывает команду в конец файла, вторая заставляет текущий шелл перечитать его. Без `source` правка подействует только на новые терминалы: уже открытые прочитали файл в момент своего запуска и повторно к нему не возвращаются.

<Callout type="tip" title="Быстрая проверка после правки">
Отредактировали `~/.bashrc`, сразу выполните `source ~/.bashrc` и проверьте результат командой `echo $PATH`. Так опечатка обнаружится немедленно, а не через неделю в новом терминале, когда вы уже забудете, что меняли.
</Callout>

## Это вас удивит

Переменная без `export` не видна даже команде, запущенной из той же строки терминала. Интуиция подсказывает, что «мой терминал» и «команды в нём» делят одну память, но `bash -c 'echo $CITY'` создаёт отдельный процесс с копией окружения, а неэкспортированная переменная в эту копию не попадает. Отсюда классический тупик: в шелле `echo $CITY` печатает значение, а скрипт клянётся, что переменной нет. Оба правы.

<Callout type="warning" title="Каталог в начале PATH перекрывает систему молча">
Если в каталоге, стоящем в `PATH` раньше системных, появляется исполняемый файл с именем существующей команды, шелл начинает запускать его без единого предупреждения. Самодельный скрипт `~/bin/test` или сломанный `python` в начале списка портят чужие скрипты самым загадочным образом. Первый шаг диагностики всегда один: `which имя`, чтобы увидеть, кто на самом деле откликается.
</Callout>

Третья неожиданность: `export` ничего не записывает на диск. Все переменные, выставленные руками, живут в памяти шелла и исчезают при закрытии окна. Если назавтра настройка «слетела», она не сломалась, вы её просто не сохранили в `~/.bashrc`.

## Типичные ошибки

**Перезаписали PATH вместо дополнения.** Команда `export PATH=/opt/tools/bin` без части `$PATH:` заменяет весь список одним каталогом. Немедленно ломается всё: `ls`, `cat` и `grep` отвечают `command not found`, потому что шелл больше не знает, где их искать. Ошибка возникает из ложной аналогии с обычным присваиванием, где старое значение не нужно. Исправление: всегда включайте старый список, `export PATH="$PATH:/opt/tools/bin"`. Если уже сломали, вызовите команду по полному пути, `/usr/bin/ls` работает без всякого `PATH`, либо просто закройте терминал: поломка живёт только в этой сессии.

**Правка ~/.bashrc без перезапуска шелла.** Отредактировали файл, а в открытом терминале ничего не изменилось. Ожидание, что конфиг применяется сам, естественно, но bash читает `~/.bashrc` один раз, при старте. Исправление: выполните `source ~/.bashrc` в каждом уже открытом терминале или откройте новый.

**Пробелы вокруг знака равенства.** Строка `CITY = Kazan` не присваивание. Шелл разбирает её как вызов команды `CITY` с двумя аргументами и отвечает `CITY: command not found`. Причина в том, что пробел для шелла разделитель слов команды, а не украшение синтаксиса. Исправление: писать слитно, `CITY=Kazan`.

<Detail summary="Переменная на один запуск">

Есть третья форма, промежуточная между присваиванием и `export`: значение можно выдать одной конкретной команде, написав присваивание перед ней в той же строке. В свежем терминале:

```bash
$ CITY=Kazan bash -c 'echo [$CITY]'
[Kazan]
$ echo [$CITY]
[]
```

Переменная попала в окружение только этого запуска и исчезла вместе с ним: сам шелл её так и не увидел. Форма удобна для разовых запусков с другим языком, часовым поясом или конфигом, когда захламлять сессию постоянным `export` не хочется.

</Detail>
