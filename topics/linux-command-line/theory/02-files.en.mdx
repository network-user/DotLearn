# Create, copy, move, delete: files under your control

Anna's home directory has piled up with a hundred files: fresh reports mixed in with drafts, and somewhere in there `orders.csv` from last year. Time to clean up: create an `archive` directory, move the old stuff into it, delete what's no longer needed. In a graphical file manager, that's a long evening of dragging things with a mouse. In the terminal, the same cleanup takes five commands, and fewer still once you bring in name patterns.

In this lesson you'll learn to create directories and files (`mkdir`, `touch`), view contents (`cat`), copy (`cp`), move and rename (`mv`), and delete (`rm`). Along the way you'll learn the terminal's single most important quirk, one you'd rather read about here than discover from a report you just lost: there is no trash can.

## Mental model: commands move nodes on the tree

From the last lesson you know the filesystem is a tree, and a path is the address of a node in it. Every command in this lesson does exactly one of four things to that tree: add a node (`mkdir`, `touch`), create a copy of a node at a new address (`cp`), change a node's address (`mv`), or remove a node (`rm`). Figure 1 shows the trickiest of these. After `mv`, the file exists in exactly one place, just at a different address: it's gone from the old one.

## Create: mkdir and touch

`mkdir` creates a directory:

```bash
$ mkdir archive
```

If you need a nested structure, the command refuses by default:

```bash
$ mkdir archive/2026/july
mkdir: cannot create directory 'archive/2026/july': No such file or directory
```

`mkdir` won't build the intermediate directories for you: since `2026` doesn't exist yet, there's nowhere to put `july`. The `-p` flag lifts that restriction: `mkdir -p archive/2026/july` creates the whole chain in one call and won't complain if part of it already exists.

`touch` creates an empty file:

```bash
$ touch list.txt
```

`touch` has a quirk: on an existing file it erases nothing, it just updates the last-modified time. That's historically what the command was built for; creating an empty file is really just a side effect everyone happens to rely on.

## Look: cat

```bash
$ cat orders.csv
id,item
1,book
2,lamp
```

`cat` prints a file to the terminal in full. The name comes from concatenate: pass it several files and `cat` prints them one after another, as a single stream, in the order you listed them. For longer files there are friendlier commands: `head -n 5 fail.log` shows the first five lines, `tail -n 5 fail.log` the last five, and `less fail.log` opens the file in a pager you exit with `q`. Remember `less`: one day it will save you from running `cat` on a gigabyte-sized log.

## Copy: cp

```bash
$ cp report.txt report-copy.txt
$ cp report.txt archive/
```

The first form makes a copy under a new name, the second drops a copy with the same name into the `archive` directory. The copy is fully independent: edits to it don't touch the original. `cp` won't work on directories by default:

```bash
$ cp docs backup
cp: -r not specified; omitting directory 'docs'
```

A directory is an entire subtree, and `cp` requires explicit confirmation that you intend to copy it recursively, with everything inside: `cp -r docs backup`.

## Move and rename: mv

```bash
$ mv report.txt archive/
$ mv archive/report.txt archive/report-2026.txt
```

The first command moves the file into a directory, the second renames it. For `mv` these are literally the same operation: the command just changes a node's address in the tree, as in figure 1. That comes with a nice bonus: `mv` doesn't need `-r` for directories, since nothing is copied, the whole subtree just gets a new address.

## Delete: rm

```bash
$ rm old-draft.txt
```

The file is gone. `rm` won't touch a directory without a flag:

```bash
$ rm archive
rm: cannot remove 'archive': Is a directory
```

Just like with `cp`, the `-r` flag switches on recursive mode: `rm -r archive` deletes the directory and everything in it. It's often paired with `-f`, which suppresses prompts and warnings; more on why the combo `rm -rf` earned its bad reputation below.

## Name patterns: the asterisk

Every command in this lesson accepts several paths at once, and you don't have to type them all by hand. The `*` character in a name means "any sequence of characters":

```bash
$ cp *.txt archive/
```

The shell expands `*.txt` into the list of every `.txt` file in the current directory and hands that list to `cp`. Wildcards can do a lot more, but the `*.txt` form covers most everyday work, and for now that's enough.

## What will surprise you

<PullQuote>There's no trash can in the terminal: rm deletes immediately and permanently.</PullQuote>

The main surprise has already been mentioned, but it deserves its own paragraph. `rm` doesn't move files to a trash can, doesn't show an "are you sure?" dialog, and can't undo what it did. The command runs, the file is gone: no `Ctrl+Z`, no version history, no second chance.

The second surprise is quieter, and more dangerous for it: `cp` and `mv` silently overwrite an existing target. If `b.txt` already exists, `mv a.txt b.txt` destroys its contents without asking a single question. The `-i` flag makes both commands confirm before overwriting: `mv -i a.txt b.txt`.

You've already seen the third: `cp` refuses to copy a directory without `-r` and actually tells you so. Of the three surprises, this is the only one where the command speaks up instead of silently destroying something.

## Common mistakes

`rm -rf` with a typo in the path. You meant to delete `build`, fat-fingered a key, and deleted the directory next to it instead: `-f` silenced every warning, and there's no trash can to check. The fix: run `ls` on that same path before `rm -r` and confirm you're looking at exactly what you're about to delete.

`mv report.txt archive`, when the `archive` directory doesn't exist. `mv` can't read your intentions: since there's no target directory, it decides you meant a rename, and the file simply becomes named `archive`. The fix: write the target with a trailing slash, `mv report.txt archive/`. If the directory doesn't exist, the command now fails loudly instead of silently renaming.

Forgetting `-r` when copying a directory, reading `omitting directory`, and concluding the terminal is broken. The terminal is fine: `cp` is protecting you from accidentally copying enormous subtrees and is waiting for explicit permission.

Running `cat` on a huge file: the terminal chokes on thousands of lines, and stopping the flood isn't easy. The reason is that `cat` shows no mercy and prints everything it's given. For big files reach for `less`, or `head`/`tail` if you just need a peek at one end.

<Detail summary="Where a deleted file actually goes">
  `rm` doesn't wipe the bytes on disk: it removes the file's name entry from the directory and marks that space as free. The data physically stays on disk until the system reuses that space for something else. Recovery tools exist, but you shouldn't count on them: the longer the machine keeps running after the deletion, the slimmer the odds. The boring, reliable takeaway: backups beat hoping for recovery.
</Detail>
