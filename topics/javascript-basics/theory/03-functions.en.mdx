# Functions: A Calculation That Lives in One Place

In the online store Anna is building, the loyalty discount gets calculated in three places: on the product card, in the cart, and in the order confirmation email. The formula is the same everywhere, and Anna honestly copy-pasted it twice. Then the rule changes: marketing asks for fifteen percent instead of ten. Anna fixes the product card and the cart, and forgets about the email. For a week, customers see one price on the site and another in their inbox, and support spends the week fielding complaints. The bug didn't happen because Anna is bad at percentages. It happened because the same calculation existed in three copies.

The cure for stories like this has been around for a long time: write the calculation once, give it a name, and refer to it by name from then on. In JavaScript, a named spot like that is called a function. By the end of the chapter you'll be able to declare functions three different ways, understand exactly what `return` does, and read arrow syntax fluently, since it shows up on every other line of modern code.

## A box: arguments go in, a result comes out

The mental model of a function fits in a single picture. There's a box with a name. The box has inputs: values you pass in when you call it. Inside is a hidden sequence of steps. And there's one output: the value the box hands back through `return`. The figure below shows this route for a discount calculation: the numbers 1000 and 10 go in, 900 comes out.

An important detail of this model: what happens inside the box is invisible from outside. Calling code only knows the name, the inputs, and the output. That's exactly why functions cure copy-paste bugs: when the discount rule changes, you change the contents of one box, and the product card, the cart, and the email all start calculating the new way immediately.

This closedness has a second benefit: you can test a function separately from the rest of the program. Feed it 1000 and 10, expect 900. No need to spin up the whole store to confirm the discount math is right. The more logic you pack into boxes like this, the easier bugs get to find: you check the boxes one at a time instead of untangling the whole program at once.

<PullQuote>From outside, a function only shows its name, its inputs, and its output; everything else is its own business.</PullQuote>

## Function declaration: the classic form

The oldest and most recognizable way to declare a function looks like this:

```js
function priceWithDiscount(price, percent) {
  return price - (price * percent) / 100;
}

const total = priceWithDiscount(1000, 10);
console.log(total);
```

Let's take it apart. The word `function` starts the declaration. Next comes the name `priceWithDiscount`, which is how you'll call the function. In parentheses are the parameters `price` and `percent`: local names that only exist inside the body. The body is wrapped in curly braces, and `return` hands the result back out. The call `priceWithDiscount(1000, 10)` substitutes 1000 for `price`, 10 for `percent`, and 900 gets printed to the console.

Pick a function name as a short answer to "what does it do": `priceWithDiscount`, `deliveryLabel`, `sendEmail`. A good name saves the reader from ever looking inside the body. If you can't come up with a name, that's an early sign the box is trying to do two jobs at once and should be split into two.

Here's a good place to untangle two terms people constantly mix up. A parameter lives in the function's declaration: a name, a placeholder for a future value. An argument lives in the call: a concrete value that lands on a parameter. In the example above, `price` and `percent` are parameters, and 1000 and 10 are arguments.

## return stops a function immediately

`return` has two properties worth learning right away. First: it doesn't just hand back a value, it instantly ends the function. Code after a `return` that fires never runs. This is the basis for a handy early-exit pattern:

```js
function deliveryLabel(total) {
  if (total >= 1000) {
    return 'бесплатная доставка';
  }
  return 'доставка 300 ₽';
}

console.log(deliveryLabel(1500));
console.log(deliveryLabel(400));
```

If the order is expensive, the first `return` fires, and the second never gets reached. If the condition is false, the function calmly moves on and returns the default answer. Code like this reads like a list of rules: the special case first, then the general one.

Second property: a function without a `return` still returns something. That value is `undefined`, the same "no value" from chapter one. JavaScript doesn't treat a missing `return` as an error, so a forgotten `return` doesn't crash the program, it quietly smuggles `undefined` further along, and something else breaks three lines later. We'll come back to this trap at the end of the chapter.

## A function as a value: function expression

In JavaScript, a function behaves like a full-fledged value, on par with a number or a string. You can create one without a name and store it in a variable:

```js
const sayHello = function (name) {
  return 'Привет, ' + name;
};

console.log(sayHello('Анна'));
```

This is called a function expression: an anonymous function is created as a value and assigned to the variable `sayHello`, then called through that variable's name. The practical difference from a declaration shows up in one place: a declaration can be called even above the line where it's written, while an expression is only available after the assignment runs. Why the engine does this is covered in the detail box at the end of the chapter.

## Arrow functions: the same box, shorter to write

In 2015, the ES6 standard added another way to write functions, and today it's the most common one in codebases. An arrow function drops the word `function`:

```js
const priceWithDiscount = (price, percent) => {
  return price - (price * percent) / 100;
};
```

Read it like this: "the parameters `price` and `percent` flow into the body." As long as the body sits in curly braces, an arrow behaves like an ordinary function expression: it needs an explicit `return`, or `undefined` slips out.

<MarginNote label="History">Before ES6, every function in JavaScript was written with the word function. Arrow syntax borrows from the mathematical notation of lambda functions.</MarginNote>

The real savings start when the whole body is a single expression. Then the curly braces and `return` both disappear:

```js
const double = (x) => x * 2;

console.log(double(21));
```

The short form returns the expression's result on its own, without the word `return`. This is called an implicit return. The rule is worth saying twice, because it's exactly where the most common arrow mistake comes from: no curly braces means the expression's result returns automatically; curly braces mean you're on the hook for writing `return` yourself.

Two more small things you'll run into in other people's code. When there's exactly one parameter, the parentheses around it are optional: `x => x * 2` works the same as `(x) => x * 2`, and which style to use is up to each team. When there are no parameters at all, empty parentheses are required: `() => 42`.

## Default parameters

Sometimes you want an argument to be optional. For that, give a parameter a default value right in the declaration's parentheses:

```js
function greet(name = 'гость') {
  return 'Привет, ' + name;
}

console.log(greet('Анна'));
console.log(greet());
```

The first call prints "Привет, Анна." The second has no argument, so `name` gets the value `'гость'`, and it prints "Привет, гость." The default also kicks in when a call explicitly passes `undefined`. `null` and an empty string, though, count as normal passed-in values: the default won't fire for them, and that regularly surprises people.

Defaults pay off the most in functions with settings that rarely change. Shipping cost is almost always the standard rate, except for remote regions: make it a parameter with a default value, and nine calls out of ten stay short while the tenth explicitly passes its own price. The usual order is: required parameters first, optional ones with defaults after them.

## A function can be passed into another function

Because a function behaves like a value, you get a trick that half of JavaScript is built on: you can pass a function into another function as an argument. The receiving side decides when and how many times to call it:

```js
function applyTwice(fn, value) {
  return fn(fn(value));
}

const increment = (x) => x + 1;

console.log(applyTwice(increment, 40));
```

`applyTwice` knows nothing about addition: it receives a function through the parameter `fn` and applies it twice, so 42 lands in the console. A function passed in like this is often called a callback. You can pass any function, not just an arrow: a declaration reaches an argument list by its name just as well, arrows are just shorter, and they've become the go-to choice for callbacks. This same trick powers the array method `map`, which you'll meet in "map, filter, and reduce."

## Trap: curly braces swallowed the result

One arrow expression looks almost like another, yet behaves differently. Test yourself: what does this code print?

```js
const thrice = (x) => {
  x * 3;
};

console.log(thrice(5));
```

It prints `undefined`. The expression `x * 3` honestly gets computed and immediately vanishes: the curly braces switch on "regular body" mode, and only `return` can carry a value out of that. Two fixes: either write `return x * 3;`, or drop the curly braces entirely and go back to the short form.

This trap fires most often during a refactor. There was a short arrow, you needed to add a second step, you wrapped the body in curly braces, added the new line, and forgot the `return` for the old one. The function still looks almost the same, but it stopped returning anything.

<Callout type="warning" title="A forgotten return doesn't crash">
The nastiest part of this bug is that the program keeps running. The `undefined` value drifts further along and breaks a calculation somewhere else entirely. If you see an unexpected `undefined`, check the `return` in your functions first.
</Callout>

## Common mistakes

First mistake: calling without parentheses.

```js
const result = priceWithDiscount;
console.log(typeof result);
```

It prints `function`, not a number. Without parentheses you didn't call the function, you took it as a value and stored the whole thing in a variable. In the console, the missing value shows up as something like `[Function: priceWithDiscount]` where you expected a number. Fix it by adding parentheses with arguments: `priceWithDiscount(1000, 10)`.

The second mistake lives in your head, not in the syntax: confusing a parameter with an argument. Inside the body, beginners sometimes write the calculation using the concrete number 1000 instead of the parameter `price`, and the function forever calculates the discount off one single price. The confusion happens because, at the moment you're writing the function, there are no concrete values yet, and thinking in placeholders feels unnatural. It helps to remember the rule: only parameters exist inside the body, concrete numbers arrive from outside at call time.

Third mistake: code after `return`.

```js
function orderSummary(total) {
  return 'итого: ' + total;
  console.log('заказ собран');
}
```

The line with `console.log` never runs: `return` already stopped the function. Linters flag code like this as unreachable. Do anything that needs to happen before the `return`, and make `return` the last meaningful step of a branch.

These three mistakes share one unpleasant trait: JavaScript almost never reports them right away. The program starts, runs, and produces a strange result somewhere further down the code. So when debugging functions, start with three questions: is the right function being called, is it being called with parentheses, and does every branch return a value through `return`?

<Detail summary="Why a declaration can be called before it's declared">
Before running a script, the JavaScript engine scans through the whole file and registers every function declaration ahead of time. The mechanism is called hoisting. Function expressions and arrows stored in variables aren't hoisted the same way: the engine knows the variable's name, but its value only appears on the line where the assignment runs, so an earlier call throws an error.
</Detail>

<Detail summary="One line about this">
Arrow functions don't have their own `this`, they take it from the surrounding code. As long as you're writing standalone functions, this difference never shows up; it becomes important later, once you reach objects with methods and event handlers.
</Detail>
