# A Hundred Products, One Variable

Picture a shopping cart in an online store. While it holds three products, everything gets by on good faith: you set up variables `price1`, `price2`, `price3`, alongside `name1`, `name2`, `name3`, and compute the order total by adding three numbers by hand. The customer adds a fourth product, and you tack on a fourth pair of variables and patch the formula. On a marketplace like Amazon, a cart can hold a hundred items. You could theoretically set up a hundred pairs of variables, but you'd never be able to loop over them: variable names don't have numbers, there's no way to reach `price37` "by counter."

In this chapter you'll meet the two structures that nearly all JavaScript code rests on. The first stores many values in order and hands them back by number. The second describes a single thing in full: a product has a name, a price, a stock flag. By the end you'll combine them into the go-to working pattern, a list of products, and learn what makes a variable holding an array different from a variable holding a number. Almost every newcomer trips on that difference, so we'll work through it with a picture.

## Arrays: numbered cells

An array stores values in order, in numbered cells, and lives entirely under one name. You create one with square brackets, separating elements with commas:

```js
const list = ['хлеб', 'молоко', 'сыр'];
```

Numbering starts at zero: "хлеб" sits in cell 0, "молоко" in cell 1, "сыр" in cell 2. To pull an element out, write its number in square brackets after the array's name: `list[0]` returns "хлеб," `list[2]` returns "сыр." The `length` property tells you how many elements sit in the array: `list.length` is 3. Which gives you a trick you'll use all the time. Since numbering starts at zero, the last element's index is always one less than the length:

```js
const last = list[list.length - 1];
```

The length is 3, so the last occupied cell is number 2, and `last` gets "сыр." The expression looks clunky, but it's written so often that your hands memorize it fast. Negative indexes, by the way, don't work here: `list[-1]` won't fetch the last element like it does in some other languages, it'll honestly look for a key named "minus one" and find nothing.

<MarginNote label="Why zero-based">An index historically means an offset from the start: the first element sits right at the beginning, an offset of zero. JavaScript inherited this tradition from C along with most of its syntax.</MarginNote>

An array can grow and shrink. `push` adds an element to the end, `pop` removes the last element and returns it:

```js
list.push('кефир');
const taken = list.pop();
```

After `push`, the length becomes 4; after `pop`, it's back to 3, and `taken` holds "кефир." The `push`/`pop` pair covers a surprising number of tasks: undo history, a stack of open screens, accumulating results in a loop.

Two methods help you search for a value. `includes` answers whether a value is in the array, and `indexOf` returns the index of the cell where it was found:

```js
list.includes('молоко');
list.indexOf('сыр');
```

The first call returns `true`, the second returns 2. If the value isn't there, `includes` honestly returns `false`, while `indexOf` returns the number -1, which is easy to mistake for a real index if you forget to check for it.

## Objects: a name instead of a number

Now let's look at a single product. It has a name, a price, a stock flag, and you want to reach them by meaning, not by cell number. That's what an object is for in JavaScript: a set of "key and value" pairs inside curly braces.

```js
const item = { name: 'чай', price: 250 };
```

You read properties two ways. Dot notation works when the key is known ahead of time and written literally in the code. Square brackets are needed when the key lives in a variable:

```js
item.name;
item['price'];
const key = 'price';
item[key];
```

The first two calls return "чай" and 250 and are fully interchangeable. The third is more interesting: the brackets hold not a string, but the variable `key`, which contains a string. Dot notation can't do that. Writing `item.key` would look for a property literally named `key`, not peek inside the variable.

New properties get added with a plain assignment; you don't need to recreate the object for that:

```js
item.inStock = true;
item.seller = { name: 'Чайный дом', city: 'Казань' };
```

The second assignment puts another object inside this one. Nesting like this shows up everywhere: server responses, app settings, user profiles. You reach into the depths with a chain: `item.seller.city` returns "Казань." The full list of an object's keys comes from `Object.keys(item)`: an ordinary array of strings, `['name', 'price', 'inStock', 'seller']`, and every trick from the first half of the chapter applies to it.

## An array of objects: the everyday pattern

Let's combine both structures and return to the cart from the start of the chapter. Each product becomes an object, and the cart becomes an array of those objects:

```js
const products = [
  { name: 'хлеб', price: 45 },
  { name: 'молоко', price: 89 },
  { name: 'сыр', price: 320 },
];
```

Three products or a hundred, the processing code doesn't change. A `for...of` loop iterates the array, handing you elements one at a time with no index bookkeeping:

```js
let total = 0;
for (const product of products) {
  total = total + product.price;
}
```

On each lap, `product` gets the next object, and `product.price` pulls its price. After the loop, `total` holds 454, the cart's total. Swap the addition for a condition, and the same skeleton counts products under a hundred rubles or finds the priciest one. This is exactly the shape data takes in the real world: a list of orders, chat messages, table rows almost always turn out to be an array of objects, so you'll write this pairing with `for...of` literally every day.

## A variable holds a reference, not the object itself

Time to make good on a promise from the intro. When you write `const n = 5`, you can think of the number as sitting right inside the variable. With arrays and objects, the picture is different: the "box" of data lives separately in memory, and the variable only holds a reference to it, something like a sticky note with the warehouse's address.

An unexpected consequence follows: two variables can point at the exact same object.

```js
const item = { name: 'чай', price: 250 };
const copy = item;
```

The name `copy` is misleading: no copy ever appeared. The assignment moved a sticky note with an address, and now both variables lead to the same box. So a change through one variable shows up through the other:

```js
copy.price = 199;
console.log(item.price);
```

It prints 199, even though nobody "touched" `item`. Which variable you look through doesn't matter, since there's only one object.

The same model explains `const`'s behavior, which looks contradictory at first. `const` forbids re-binding a variable to a different value, that is, replacing the reference on the sticky note. It never forbade changing the contents of the box the reference points to:

```js
const cart = ['хлеб'];
cart.push('молоко');
```

This works: `push` changes the array from the inside, and the reference stored in `cart` stays the same. But the assignment `cart = ['сыр']` throws an error, because it tries to put a new reference into a `const`.

## Edge cases worth knowing about now

Reading a cell or key that doesn't exist doesn't crash the program. JavaScript silently returns `undefined`:

```js
const list = ['хлеб', 'молоко', 'сыр'];
const item = { name: 'чай' };
console.log(list[10]);
console.log(item.price);
```

Both lines print `undefined`. This is convenient and dangerous at the same time: a typo in a property name won't raise an error at the point where you read it, it'll "go off" later, once `undefined` lands in some arithmetic and turns the result into `NaN` somewhere else in the program entirely. If you spot `NaN` in an order total, check key names and index bounds first.

The second surprise is about comparison. The `===` operator compares arrays and objects by reference, not by content:

```js
console.log([1] === [1]);
```

It prints `false`. Two separate arrays were created on the left and right, two separate boxes, and matching content doesn't interest the operator at all. You only get `true` when both sides point at the exact same object, like `item` and `copy` above. Comparing content has to be done by hand: lengths first, then elements one by one.

## Common mistakes

**Dot notation instead of brackets when the key sits in a variable.** This mistake is sneaky because nothing crashes:

```js
const field = 'price';
const wrong = item.field;
const right = item[field];
```

`item.field` looks for a property literally named `field`, there isn't one, and `wrong` ends up `undefined`. Simple rule: key written literally in the code, use a dot; key coming from a variable, use brackets.

**An off-by-one at the boundary.** A classic: reaching for `list[list.length]` hoping to get the last element. There's no cell with that number, so you get `undefined`. The last element lives at `list[list.length - 1]`, worth repeating to yourself one more time.

**Mutating a "copy" that turns out to be the original.** You "copied" an array with an assignment, sorted the copy or added elements to it, and then discovered the original data changed too. An assignment copies the reference, not the contents. A real copy has to be made explicitly, covered next.

<Detail summary="How to actually copy an array or object">

Spread syntax makes a shallow copy: three dots before a name spread the old contents into a new box.

```js
const orig = ['хлеб', 'молоко'];
const copied = [...orig];
copied.push('сыр');
```

`orig` stays at two elements, because `copied` points at a new array. The same syntax works for objects with curly braces: `{ ...item }`. The catch is hiding in the word "shallow": only the top level gets copied. If objects live inside the array, the copy gets references to those exact same objects, and a nested product ends up shared between the copy and the original. A deep copy comes from the built-in `structuredClone` function, though you'll need it less often than it seems.

</Detail>
