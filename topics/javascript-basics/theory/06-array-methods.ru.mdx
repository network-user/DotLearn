# Три строки вместо трёх циклов

Представьте типичную задачу интернет-магазина. В корзине покупателя лежит десяток товаров, и перед оформлением заказа нужно сделать три вещи: поднять цены на десять процентов после окончания акции, убрать распроданные позиции и посчитать итоговую сумму. Решение «в лоб» разрастается почти на страницу: три цикла `for`, индексы, временные массивы, `push` после каждой проверки. Код работает, но за скобками и счётчиками уже не видно, что вы вообще собирались сделать с данными.

В JavaScript ту же страницу можно сжать до трёх строк. Методы map, filter и reduce берут на себя весь обход массива, а вам оставляют только суть: как преобразовать элемент, какие элементы оставить и как собрать из них итог. К концу главы вы будете читать и писать такие цепочки так же свободно, как обычный цикл, и поймёте, почему в рабочем коде они встречаются гораздо чаще, чем `for`.

## Ментальная модель: конвейер на складе

Методы массивов устроены так же, как эти станции. Каждому нужна инструкция: что именно сделать с одним элементом. Вы описываете инструкцию обычной функцией и передаёте её методу как аргумент. Функция, которую вы отдаёте чужому коду, чтобы он вызвал её в нужный момент, называется callback. Метод сам пройдёт по массиву и вызовет ваш callback для каждого элемента: вы пишете «что сделать», а метод отвечает за «как обойти».

<MarginNote label="История">Тройка map, filter и reduce пришла в JavaScript из функциональных языков: в Lisp эти операции существовали ещё в 1960-х годах.</MarginNote>

У всей троицы общий контракт, и на нём держится вся глава: исходный массив не меняется. Каждый метод возвращает новое значение: map и filter отдают новый массив, reduce отдаёт одно значение. Именно поэтому методы можно выстраивать в цепочки, как станции вдоль одной ленты.

Чем такой стиль лучше цикла? Цикл описывает процесс: заведите счётчик, идите, пока индекс меньше длины, не забудьте увеличить индекс, положите результат во временный массив. Методы описывают результат: из этих цен получить вот такие строки. Читателю не нужно мысленно исполнять код, чтобы понять замысел, а автору негде ошибиться на единицу в границах индексов. Вдобавок у каждого метода говорящее имя: увидев filter, вы ещё до чтения callback знаете, что дальше по коду поедет укороченная версия массива.

## map: преобразовать каждый элемент

Начнём с массива цен в рублях:

```js
const prices = [100, 250, 400, 90, 1990];
```

Метод map вызывает callback для каждого элемента и складывает результаты вызовов в новый массив:

```js
const doubled = prices.map(p => p * 2);
console.log(doubled);
console.log(prices);
```

В консоли появится `[200, 500, 800, 180, 3980]`, а следом исходный `[100, 250, 400, 90, 1990]` без единого изменения. У map жёсткий контракт: сколько элементов пришло на вход, столько выйдет и на выходе. Пять цен на входе, пять удвоенных на выходе, порядок тот же.

Callback может возвращать что угодно, не только число того же типа. Например, из чисел легко получить строки для витрины:

```js
const labels = prices.map(p => `${p} ₽`);
```

Теперь в `labels` лежит `['100 ₽', '250 ₽', '400 ₽', '90 ₽', '1990 ₽']`. Массив чисел превратился в массив строк, но длина снова совпала с исходной. Когда встречаете map в чужом коде, читайте его так: «каждый элемент превратится во что-то новое, и никто не потеряется».

На самом деле map передаёт в callback не только элемент: вторым аргументом приходит индекс, третьим весь массив. Чаще всего они не нужны, но индекс выручает, когда позиция элемента важна сама по себе:

```js
const numbered = prices.map((p, i) => `${i + 1}. ${p} ₽`);
```

Первая цена получит номер 1, вторая номер 2 и так далее. Объявляйте только те параметры, которые действительно используете: варианты `p => p * 2` и `(p, i) => p * 2` работают одинаково, но первый честнее сообщает читателю, что индекс в деле не участвует.

## filter: оставить только нужное

У filter callback особого сорта: он обязан вернуть `true` или `false`. Такую функцию называют предикатом. Метод оставляет в новом массиве только те элементы, для которых предикат вернул `true`:

```js
const cheap = prices.filter(p => p <= 500);
```

В `cheap` попадут `[100, 250, 400, 90]`: всё, что стоит не дороже 500 рублей. Обратите внимание: filter ничего не преобразует. Элементы попадают в результат ровно в том виде, в каком лежали в исходном массиве, просто некоторые не проходят отбор. Длина результата меньше исходной или равна ей, если предикат пропустил всех.

## reduce: свернуть массив в одно значение

map и filter возвращают массивы. Но итоговая сумма корзины уже не массив, это одно число. Для таких свёрток существует reduce:

```js
const total = prices.reduce((sum, p) => sum + p, 0);
```

Здесь больше движущихся частей, поэтому разберём по порядку. Callback у reduce принимает два главных аргумента: аккумулятор `sum` и текущий элемент `p`. Аккумулятор играет роль блокнота, который передаётся от шага к шагу: что callback вернул на этом элементе, то придёт аккумулятором на следующем. Второй аргумент самого reduce, ноль после запятой, задаёт начальное значение блокнота.

Проследите за суммой шаг за шагом: аккумулятор стартует с 0, после первой цены становится 100, затем 350, 750, 840 и, наконец, 2830. Последнее возвращённое значение и есть результат вызова: в `total` лежит 2830.

Начальное значение выбирают под задачу: 0 для суммы, 1 для произведения, пустая строка для склейки, пустой массив или объект, если вы собираете структуру. Ориентир простой: у аккумулятора должен быть тот же тип, что и у значения, которое вы хотите получить на выходе.

## Цепочки: станции передают ленту друг другу

Раз каждый из методов возвращает новый массив, к результату можно сразу применить следующий метод. Вернёмся к корзине, но теперь с настоящими товарами:

```js
const products = [
  { name: 'Чайник', price: 1990, inStock: true },
  { name: 'Кружка', price: 350, inStock: true },
  { name: 'Ложка', price: 90, inStock: false },
  { name: 'Блокнот', price: 120, inStock: true },
];
```

Допустим, витрине нужны названия товаров в наличии:

```js
const names = products
  .filter(p => p.inStock)
  .map(p => p.name);
```

Цепочка читается сверху вниз, как описание конвейера: сначала filter снимает с ленты «Ложку», затем map превращает оставшиеся объекты в строки. В `names` лежит `['Чайник', 'Кружка', 'Блокнот']`.

А вот и задача из начала главы, целиком:

```js
const totalInStock = products
  .filter(p => p.inStock)
  .map(p => p.price)
  .reduce((sum, p) => sum + p, 0);
```

Три станции, три строки, и в `totalInStock` лежит 2460. Порядок станций имеет значение: filter в начале цепочки сокращает объём работы всем, кто стоит дальше по ленте. Отфильтровать четыре товара, а потом преобразовать три оставшихся дешевле, чем преобразовывать все четыре и выбрасывать лишнее.

## Краевые случаи, которые вас встретят

Начальное значение у reduce формально необязательно. Если его не передать, метод возьмёт первый элемент массива как стартовый аккумулятор и начнёт обход со второго. Звучит удобно ровно до тех пор, пока массив не оказался пустым:

```js
const empty = [];
const sum = empty.reduce((acc, p) => acc + p);
```

Эта строка не вернёт 0 и не вернёт undefined. Она бросит ошибку `TypeError: Reduce of empty array with no initial value`: стартовый аккумулятор взять неоткуда. Пустая корзина в магазине совершенно обычное дело, поэтому запомните правило: всегда передавайте начальное значение. Вызов `empty.reduce((acc, p) => acc + p, 0)` спокойно вернёт 0.

Второй сюрприз прячется в синтаксисе стрелочных функций. Стрелка с телом-выражением возвращает результат сама: `p => p * 2`. Но стоит обернуть тело в фигурные скобки, и оно становится блоком, где `return` нужно писать явно:

```js
const broken = prices.map(p => { p * 2 });
console.log(broken);
```

В консоли окажется `[undefined, undefined, undefined, undefined, undefined]`. Выражение `p * 2` честно вычислилось и тут же пропало, функция без явного `return` вернула undefined, и map послушно сложил пять undefined в новый массив. Длина сохранилась, содержимое исчезло.

## Частые ошибки

**Мутирование исходного массива внутри map.** С числами и строками так не получится, а вот объекты приходят в callback по ссылке: это те же самые объекты, что лежат в исходном массиве. Такой код портит оригинал:

```js
const discounted = products.map(p => {
  p.price = p.price * 0.9;
  return p;
});
```

Скидка применилась, но заодно изменились объекты в самом `products`: старые цены потеряны безвозвратно. Так происходит потому, что map копирует ссылки на объекты, а не сами объекты. Правильный callback возвращает новый объект:

```js
const discounted = products.map(p => ({ ...p, price: p.price * 0.9 }));
```

Обратите внимание на круглые скобки вокруг объекта: без них фигурные скобки прочитались бы как блок из предыдущего раздела, и вы получили бы массив undefined.

**map вместо forEach.** Если возвращаемый массив никуда не сохраняется, map выбран зря:

```js
products.map(p => console.log(p.name));
```

Названия напечатаются, но попутно будет создан и тут же выброшен массив из undefined. Хуже другое: читатель, увидев map, ждёт преобразования данных и потратит время на поиски результата, которого нет. Для побочных действий существует forEach, о нём чуть ниже.

**Забытый return в длинном callback.** Пока callback умещается в одно выражение, стрелка возвращает его сама. Как только появляются промежуточная переменная и фигурные скобки, обязанность вернуть значение ложится на вас:

```js
const titles = products.map(p => {
  const title = `${p.name}: ${p.price} ₽`;
});
```

Снова массив из undefined, знакомый по краевому случаю выше. Лекарство одно: дописать `return title;` последней строкой. Если map или filter вернули не то, что вы ждали, первым делом проверьте, есть ли `return` во всех ветках callback.

<Detail summary="Соседи по семейству: forEach, find, some, every">
  У той же ленты есть и другие станции. forEach вызывает callback для каждого элемента и ничего
  не возвращает: он предназначен для побочных действий вроде вывода в консоль. find возвращает
  первый элемент, для которого предикат вернул `true`, или undefined, если такого нет. some
  отвечает на вопрос «есть ли хотя бы один подходящий элемент», а every проверяет, подходят ли
  все. Каждый из них принимает callback той же формы, что map и filter, так что новая механика
  вам не понадобится.
</Detail>
