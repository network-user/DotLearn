# Функция, которая помнит

Представьте интернет-магазин, который вы собираете на JavaScript. Возле каждого товара стоит кнопка «добавить в корзину», и вам нужно считать нажатия. Первое решение приходит само: завести одну переменную наверху файла и увеличивать её при каждом клике. Пока товар один, всё работает безупречно. Но каталог растёт, товаров уже двадцать, у каждого своя кнопка, и все двадцать пишут в один общий счётчик. Число «7» в нём больше ничего не сообщает: то ли покупатель семь раз выбрал чайник, то ли по разу нажал на семь разных товаров.

Завести двадцать глобальных переменных тоже не выход. Придётся выдумывать двадцать имён, следить, чтобы никакой другой код случайно их не перезаписал, а на сотне товаров схема развалится окончательно. Хочется другого: чтобы у каждой кнопки был личный счётчик, который посторонний код не может ни прочитать, ни испортить. В JavaScript для этого не нужны ни классы, ни библиотеки. Достаточно функций, которые вы уже умеете писать, и двух правил о том, где живут переменные. Разберём эти правила, а затем соберём фабрику счётчиков, которая закрывает задачу с корзиной в шесть строк кода.

## Коробки внутри коробок

Область видимости (scope) отвечает на один вопрос: из каких мест кода переменная доступна. Для `let` и `const` правило короткое: переменная живёт внутри ближайшего блока в фигурных скобках и не существует за его пределами. Блоком считается тело `if`, тело цикла `for`, да и просто пара скобок, поставленная ради порядка.

```js
if (true) {
  let secret = 42;
  console.log(secret);
}
console.log(secret);
```

Первый `console.log` печатает 42, второй роняет программу с ошибкой `ReferenceError: secret is not defined`. Снаружи блока переменной `secret` попросту нет. Она не пустая и не «ещё не создана», её не существует как имени, и обращение к ней считается опечаткой.

Зачем языку такая строгость? Она защищает вас от самих себя. Временные имена вроде `i`, `temp` или `item` нужны на пять строк, и блочная область гарантирует, что через сто строк никто случайно с ними не столкнётся. Чем короче живёт переменная, тем меньше кода способно её сломать и тем проще читать программу: увидели объявление внутри блока, значит, дальше конца блока смотреть не нужно.

Функции устроены так же, только их «коробка» вмещает больше. Тело функции видит собственные переменные, свои параметры и вдобавок всё, что объявлено снаружи: во внешних функциях и на верхнем уровне файла. Взгляд работает строго в одну сторону, изнутри наружу. Снаружи внутрь заглянуть нельзя.

```js
function greet() {
  const name = 'Анна';
  function shout() {
    console.log(name.toUpperCase());
  }
  shout();
}
greet();
```

Вложенная функция `shout` не объявляла `name`, но свободно читает её из окружения `greet` и печатает «АННА». А вот попытка обратиться к `name` после вызова `greet()` закончится той же `ReferenceError`: локальные переменные функции снаружи не видны, сколько бы раз вы её ни вызывали.

Осталось третье правило, самое важное в этой главе. Какие переменные видит функция, решается в момент её объявления, по положению в тексте программы, а не в момент вызова. Где функция родилась, туда она и смотрит.

```js
const rate = 90;
function toRub(usd) {
  return usd * rate;
}

function report() {
  const rate = 100;
  return toRub(10);
}
console.log(report());
```

Печатается 900, а не 1000. Функция `toRub` объявлена на верхнем уровне, поэтому её `rate` берётся с верхнего уровня. Локальная `rate` внутри `report` для неё не существует, хотя вызов произошёл именно оттуда. Этот принцип называют лексической областью видимости: слово «лексическая» намекает, что всё определяется текстом программы. Чтобы понять, какую переменную увидит функция, достаточно прочитать код глазами, запускать ничего не нужно.

<PullQuote>Функция смотрит туда, где родилась, а не туда, откуда её позвали.</PullQuote>

## Рюкзак за плечами функции

Теперь главный трюк. Функцию можно не только вызвать на месте, но и вернуть из другой функции, положить в переменную, передать в `map` или в обработчик кнопки. И когда функция уходит из родного окружения, она забирает это окружение с собой. Функция вместе с запомненным окружением называется замыканием (closure). Удобная картинка: рюкзак. В момент объявления функция складывает в рюкзак все внешние переменные, которыми пользуется, и дальше носит его с собой, куда бы её ни передали.

<MarginNote label="История">Термин closure ввёл Питер Лэндин в 1964 году, а популярным его сделал язык Scheme в 1970-х. JavaScript унаследовал механизм именно оттуда.</MarginNote>

```js
function makeCounter() {
  let count = 0;
  return function () {
    count += 1;
    return count;
  };
}

const clicks = makeCounter();
console.log(clicks());
console.log(clicks());
```

Вызов `makeCounter()` создаёт переменную `count`, рождает внутреннюю функцию и возвращает её наружу. Казалось бы, `makeCounter` отработала и её локальные переменные должны исчезнуть. Но внутренняя функция ссылается на `count`, поэтому JavaScript сохраняет окружение живым: движок не удаляет переменные, пока на них ссылается хотя бы одна функция. Первый вызов `clicks()` печатает 1, второй печатает 2. Переменная переживает вызовы и накапливает состояние. Схема ниже показывает эту пару: большая коробка изображает окружение вызова фабрики, маленькая коробка внутри изображает возвращённую функцию, а стрелка показывает ссылку из её рюкзака на захваченную переменную.

Обратите внимание, чего на схеме нет: ни одной глобальной переменной. Снаружи до `count` не добраться в принципе. В языке нет синтаксиса, который позволил бы прочитать или перезаписать чужую локальную переменную. Единственный способ повлиять на неё даёт сама возвращённая функция. Получилось приватное состояние, то самое, которого не хватало кнопкам из начала главы: счётчик, который невозможно испортить извне.

## Фабрика функций

Функцию, которая настраивает и возвращает другую функцию, называют фабрикой. `makeCounter` уже фабрика, но приём раскрывается нагляднее, когда у фабрики есть параметр.

```js
function makeMultiplier(factor) {
  return function (x) {
    return x * factor;
  };
}

const double = makeMultiplier(2);
const triple = makeMultiplier(3);
console.log(double(7));
console.log(triple(7));
```

`makeMultiplier` сама ничего не умножает. Её работа состоит в сборке: принять `factor`, упаковать его в рюкзак новой функции и отдать эту функцию наружу. `double` печатает 14, `triple` печатает 21. Параметр внешней функции ведёт себя как обычная переменная её окружения, поэтому каждая созданная функция помнит собственное значение `factor`.

Этот приём вы уже применяли, возможно не замечая. Когда вы пишете `prices.map(p => p * rate)`, стрелочная функция читает `rate` из внешнего кода, и это тоже замыкание. Колбэки в `map` и `filter`, обработчики событий, отложенные вызовы постоянно носят с собой переменные из места, где были объявлены. Без замыканий пришлось бы протаскивать каждое значение через аргументы вручную.

## Два счётчика не толкаются

Вернёмся к магазину. Двадцать товаров, каждому нужен личный счётчик. С фабрикой задача решается одной строчкой на товар: вызвали `makeCounter()`, получили независимый счётчик.

```js
const teapots = makeCounter();
const books = makeCounter();
teapots();
teapots();
console.log(teapots());
console.log(books());
```

Печатается 3, затем 1. Каждый вызов фабрики создаёт свежее окружение с собственной переменной `count`, поэтому `teapots` и `books` считают, не мешая друг другу. Рюкзаки не пересекаются, как видно на следующей схеме: две пары коробок, и ни одной общей стрелки между ними.

Двадцать товаров из завязки перестали быть проблемой. Пройдитесь по массиву товаров методом, который вы изучили в прошлой главе, вызовите фабрику для каждого и повесьте полученную функцию на кнопку. Ни одной глобальной переменной, ни одного придуманного вручную имени, ни единого шанса, что чайники засчитаются книгам. На этой же идее держится масса взрослого кода: генераторы идентификаторов, кэши, настройка обработчиков, модули с приватными данными. Везде одна и та же пара из фабрики и рюкзака.

## Живая ссылка, а не фотография

Самое частое заблуждение о замыканиях звучит так: «функция сфотографировала значение переменной в момент объявления». Нет. В рюкзаке лежит не копия значения, а живая ссылка на саму переменную. Если переменная изменится после объявления функции, функция при следующем вызове увидит новое значение.

```js
let discount = 10;
function finalPrice(price) {
  return price - (price * discount) / 100;
}

console.log(finalPrice(1000));
discount = 50;
console.log(finalPrice(1000));
```

Первый вызов печатает 900. Затем магазин объявляет распродажу, `discount` меняется, и тот же самый вызов `finalPrice(1000)` возвращает уже 500. Функция не запомнила «десять процентов», она запомнила саму переменную `discount` и каждый раз читает её заново. Вернитесь к первой схеме и обратите внимание: стрелка ведёт к коробке переменной, а не к числу внутри неё. Число в коробке может смениться сколько угодно раз, стрелка продолжит указывать туда же.

Как правило, это ровно то, что нужно: счётчик из `makeCounter` работает именно потому, что все вызовы смотрят на одну и ту же живую `count`. Но у живой ссылки есть знаменитая тёмная сторона, на которой годами спотыкались даже опытные разработчики.

<Detail summary="Ловушка var в цикле">

До появления `let` переменные объявляли через `var`, а у `var` нет блочной области видимости: на весь цикл существует одна общая переменная.

```js
for (var i = 0; i < 3; i++) {
  setTimeout(function () {
    console.log(i);
  }, 100);
}
```

`setTimeout` откладывает вызов функции на 100 миллисекунд, поэтому все три отложенные функции запускаются уже после того, как цикл закончился и `i` доросла до 3. Все три рюкзака ссылаются на одну и ту же переменную, и в консоль трижды печатается 3.

Замена `var` на `let` чинит программу: печатается 0, 1, 2. С `let` каждая итерация цикла получает собственную свежую переменную `i`, и каждая отложенная функция уносит в рюкзаке свою. Эта ловушка стала одной из главных причин, по которым в современном JavaScript `var` практически не используют.

</Detail>

## Частые ошибки

Три ситуации, в которых новички чаще всего спотыкаются об область видимости.

**Чтение переменной из блока снаружи.** Вы объявили `let result` внутри `if`, а используете после него, и программа падает с `ReferenceError`. Так происходит потому, что блок закончился и унёс переменную с собой. Исправление: объявите переменную до блока, а внутри только присваивайте ей значение.

**Ожидание, что замыкание сделало снимок.** Вы меняете переменную после объявления функции и удивляетесь, что функция видит новое значение. Замыкание держит живую ссылку, снимков оно не делает. Если снимок действительно нужен, передайте значение параметром фабрики, как `factor` в `makeMultiplier`: параметр рождается заново при каждом вызове, и у каждой созданной функции будет собственная переменная.

**Случайная глобальная переменная.** Присваивание имени, которое нигде не объявлено, вроде `total = 100` без `let` или `const`, в нестрогом режиме не вызывает ошибку, а молча создаёт глобальную переменную. Дальше два далёких участка кода начинают незаметно перезаписывать друг друга, и вы возвращаетесь к хаосу из первого абзаца главы. Привычка простая: каждая переменная появляется на свет только со словом `let` или `const`, а строка `'use strict'` в начале файла превращает такую опечатку в честную ошибку.

Замыкания принято считать самой пугающей темой базового JavaScript, но весь механизм сводится к двум правилам этой главы. Переменная живёт в своей коробке, а функция навсегда привязана к месту, где родилась. Всё остальное, от фабрик до приватного состояния, лишь следствия, которые вы теперь умеете выводить сами.
