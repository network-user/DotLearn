# Code That Makes Decisions: Conditions and Loops

Picture a checkout page for an online store that promises a 10 percent discount on orders over 1000. Anna's cart totals 1450, Boris's totals 990. The page has to look at the total and behave differently: show Anna the discounted price, show Boris the regular one. At the moment you write the code, you don't know which case you'll need: the decision gets made at runtime, based on live data.

That evening the same store emails everyone who placed an order that day. There might be three orders, there might be thirty thousand, and nobody is going to write thirty thousand identical lines. Two abilities, choosing a branch and repeating an action, turn a program from a calculator into a system with behavior. By the end of this chapter, you'll teach JavaScript both.

## A fork: one question, two paths

Every decision starts with a question that can only be answered "yes" or "no." Is the total at least 1000? Is the name field filled in? JavaScript reduces a question like that to one of two values, `true` or `false`, and `if` picks a road based on it. Look at the figure below: the check sits at a fork, both roads lead to checkout, but one of them sends the order through with a discount.

Look closely at the bottom block of the diagram: whichever way the fork chooses, the order reaches checkout either way. The condition changes the route, not the destination. This picture outlives any particular syntax: forks work this way in every programming language, only the brackets around them change.

In code, the same fork looks like this:

```js
const total = 1450;
if (total >= 1000) {
  console.log('Скидка 10%');
} else {
  console.log('Без скидки');
}
```

`if` takes a condition in parentheses and a block of code in curly braces. If the condition is true, the first block runs; otherwise the block after `else` runs. `else` itself is optional: without it, a false condition simply does nothing, and the program moves on.

When there are more than two options, chain the branches with `else if`:

```js
if (total >= 5000) {
  console.log('Скидка 15%');
} else if (total >= 1000) {
  console.log('Скидка 10%');
} else {
  console.log('Без скидки');
}
```

Checks run top to bottom, and the first true one wins: for a total of 6200, only the 15 percent branch fires, even though `total >= 1000` is also true. That's why branch order matters: start with the strictest condition, or execution will never reach the generous discount.

## Comparisons: why it's always `===`

Comparison operators live inside conditions. For numbers they look familiar from grade-school math: `>` and `<` are strict, `>=` and `<=` are inclusive, "greater than or equal" and "less than or equal." Equality is where JavaScript gets weird: there are two operators, `==` and `===`, and the difference between them has cost developers many late-night debugging sessions.

A useful detail: a comparison's result behaves like an ordinary value. The expression `total >= 1000` evaluates to `true` or `false` before `if` ever sees it, so you can store the result in a well-named variable and plug it into a condition later. Code reads like connected prose once you do that.

Loose `==` tries to convert values to the same type before comparing. Sounds helpful, behaves treacherously:

```js
console.log(5 == '5');
console.log('' == 0);
console.log(5 === '5');
console.log('' === 0);
```

The first two lines print `true`: the number 5 and the string `'5'` are "sort of equal," and an empty string is "sort of equal" to zero. The second pair prints `false`, because strict `===` looks at types first: a number and a string differ right there, and no conversion ever happens. Now imagine a promo-code check where an empty string from a blank field accidentally matches the number 0 from the database. Bugs like this don't crash loudly, they quietly let the wrong data through, and finding them takes days.

<MarginNote label="History">Brendan Eich put together the first version of JavaScript in ten days in 1995. Loose `==` was meant as a convenience for web forms, where all data arrives as strings.</MarginNote>

Which gives you a non-negotiable rule: compare only with `===` and `!==`. Leave loose `==` and `!=` to old code written before your time.

## truthy and falsy: what `if` does with non-boolean values

A condition can hold more than a comparison; any value works, and the line `if (name)` is perfectly valid syntax. Before checking, JavaScript converts the value to a boolean, essentially asking: does this count as "something"? Values that convert to `false` are called falsy, and there are exactly six of them:

- `false`
- `0`
- `''` (an empty string)
- `null`
- `undefined`
- `NaN` (a special "not a number" that comes out of failed arithmetic, like `Number('hello')`)

Everything else is truthy, meaning it converts to `true`. Learn this list the way you learned times tables: in everyday code you won't run into any other falsy values (outside that list, there's only exotica like BigInt's `0n`). The practical payoff is immediate: `if (name)` reads as "if the name is filled in," because an empty string is falsy.

```js
const name = '';
if (name) {
  console.log('Привет, ' + name);
} else {
  console.log('Представьтесь, пожалуйста');
}
```

For the value `''`, the `else` branch runs, and the program politely asks for a name. You can check any hunch about truthy and falsy in a second: open the browser console and call `Boolean(value)`, and it returns exactly what `if` would see.

## The for loop: counter, condition, step

Now the second ability: repetition. The classic `for` loop keeps all the repeat-control logic in one header line:

```js
for (let i = 1; i <= 3; i += 1) {
  console.log('Отправляю письмо номер ' + i);
}
```

The header has three parts separated by semicolons. The first runs once before the loop starts and sets up a counter: `let i = 1`. The second is checked before every lap: while `i <= 3` is true, the body runs once more. The third fires after every lap and advances the counter: `i += 1` means "increase i by one." The moment the condition first turns false, the loop ends, and execution continues right after its closing brace.

Trace it with your eyes: i is 1, condition true, print; i is 2, print; i is 3, print; i is 4, condition false, stop. Exactly three emails. Getting in the habit of tracing a small example like this saves you from an entire class of bugs, which we'll get to shortly.

With three independently movable header parts, you can play with them. A step of `i += 2` visits only odd numbers, a start of `let i = 10` combined with a step of `i -= 1` counts down like a rocket launch. One construct, any behavior, as long as the continuation condition eventually turns false.

## for...of: iterating over a list of values

More common than "repeat N times" is the task "go through every element." Say you have a list of order totals for the day. We'll cover how lists like this work in detail in chapter five; for now it's enough to know that values are listed inside square brackets, separated by commas. The `for...of` loop hands you elements one at a time, no counter needed:

```js
const totals = [1450, 990, 2100, 540];
for (const total of totals) {
  if (total >= 1000) {
    console.log(total);
  }
}
```

On each lap, `total` gets the next value, and the condition inside decides its fate. The figure below shows this pipeline: four orders go in, the condition lets two of them through.

The pattern "loop on the outside, condition on the inside" shows up in real code constantly: filter orders, count the ones that qualify, find the first matching one. Keep it in your head as a single template.

## while: as long as the condition is true

Sometimes you don't know the number of repeats up front: keep retrying a request until the server answers; keep reducing a debt while it's still above zero. For cases like that there's `while`, whose only piece of control is the condition:

```js
let attempts = 0;
while (attempts < 3) {
  attempts += 1;
}
console.log(attempts);
```

The condition is checked before every lap, and the body is responsible for eventually making it false. Remove the line `attempts += 1`, and the loop becomes infinite: the browser tab freezes, the laptop fan spins up. This isn't a theoretical scare story, it's the single most common accident beginners have with `while`.

Technically you could write the email-sending example from earlier with `while` too; the two loops are interchangeable. But your code's reader appreciates the hint: `for` says "the number of repeats is known," `while` warns "we're looping until something happens." Pick a loop by meaning, not by habit.

## break and continue: control from the inside

Two keywords inside a loop body change its course. `break` immediately ends the entire loop. It's handy once you've found what you were looking for and there's no point scanning further:

```js
const totals = [1450, 990, 2100, 540];
for (const total of totals) {
  if (total >= 2000) {
    console.log('Нашли крупный заказ: ' + total);
    break;
  }
}
```

`continue` is gentler: it cuts short only the current lap and hands control to the next one. The code below skips small orders but never leaves the loop:

```js
for (const total of totals) {
  if (total < 1000) {
    continue;
  }
  console.log(total);
}
```

Easy to remember: `break` breaks the loop entirely, `continue` carries on from the next element.

## Edge case: the string '0' versus the number 0

Back to falsy, and a case that trips up even experienced developers:

```js
if ('0') {
  console.log('строка прошла');
}
if (0) {
  console.log('число прошло');
}
```

Only "строка прошла" prints. The number 0 is one of the six falsy values, while the rule for strings only looks at length: `'0'` has one character, so it's truthy, whatever its text happens to mean. The trap is that data from forms and URLs arrives as strings: a user typed zero, you checked `if (value)`, and a check that looked "empty" at first glance let it right through. The moral: when you mean numbers, compare explicitly. `if (count > 0)` is more honest and reads better than `if (count)`.

## Common mistakes

**A single `=` instead of `===` in a condition.** The line `if (total = 1000)` doesn't compare, it assigns, and the result of the assignment (1000, truthy) always sends execution into the `if` branch. A 990-ruble order gets a discount, plus a corrupted total. Editors flag this with a warning; don't ignore it.

**Off-by-one errors.** The choice between `<` and `<=` decides whether the last lap happens or not. A loop "from 1 through n inclusive" needs `<=`; an absent-minded `<` silently drops the last element. Same remedy as above: trace a small example.

**An infinite `while`.** If the body never moves the condition toward false, the loop never ends. Most often the variable gets forgotten inside a nested `if` branch that doesn't fire on every lap. Before you run the loop, ask yourself one question: which line in the body changes the variable from the condition?

**`for...in` instead of `for...of`.** The similarly spelled `for...in` iterates string keys, not values, so a list of values needs `for...of` specifically.

<Detail summary="Where loose == came from">
  In early versions of the language, `==` was the only equality operator. It converted types because almost all data on nineties web pages arrived from forms as strings, and comparing the string `'5'` to the number 5 without conversion would have annoyed the authors of the earliest scripts. Strict `===` was added later, in the third edition of the ECMAScript standard in 1999, and `==` stayed around for backward compatibility with old code. So the language lives on with two equalities: one convenient for 1997, one correct for today.
</Detail>

<Detail summary="The ternary operator: a one-line fork">
  When both branches only pick a value, five lines of `if/else` collapse into one: `const price = total >= 1000 ? total * 0.9 : total;`. Read it as a question: the condition, then the value for "yes" after the question mark, then the value for "no" after the colon. The trick is great right up until the expression stops fitting in your head; nested ternaries turn into a puzzle, and a plain `if/else` wins again.
