# Three Lines Instead of Three Loops

Picture a typical task for an online store. A customer's cart holds a dozen products, and before checkout you need to do three things: raise prices by ten percent now that a sale has ended, drop the items that are out of stock, and add up the final total. A brute-force solution sprawls across almost a page: three `for` loops, indexes, temporary arrays, a `push` after every check. It works, but somewhere behind all the braces and counters you can no longer see what you actually meant to do with the data.

In JavaScript, that same page shrinks to three lines. The methods map, filter, and reduce take over the whole job of walking the array, leaving you with just the essence: how to transform an element, which elements to keep, and how to fold them into a result. By the end of the chapter you'll read and write chains like these as comfortably as an ordinary loop, and you'll understand why they show up in production code far more often than `for` does.

## Mental model: an assembly line

Picture a conveyor belt in a warehouse. Products from the cart ride along it, and stations stand along the belt. One station pulls off boxes that are out of stock, another swaps price tags or reads off names, a third sits at the very end with a calculator and totals up everything that made it through. No station ever runs up and down the belt itself: the belt carries products past, and a station processes whatever arrives.

Array methods work the same way as these stations. Each one needs an instruction: what exactly to do with one element. You describe the instruction as an ordinary function and pass it to the method as an argument. A function you hand to someone else's code, for it to call at the right moment, is called a callback. The method walks the array itself and calls your callback for every element: you write "what to do," and the method handles "how to walk."

<MarginNote label="History">The map, filter, and reduce trio came into JavaScript from functional languages: these operations already existed in Lisp back in the 1960s.</MarginNote>

The whole trio shares one contract, and the entire chapter rests on it: the source array never changes. Every method returns a new value: map and filter hand back a new array, reduce hands back a single value. That's exactly why you can chain them into a sequence, like stations along one belt.

Why is this style better than a loop? A loop describes a process: set up a counter, keep going while the index is less than the length, don't forget to increment the index, push the result into a temporary array. Methods describe a result: turn these prices into these strings. The reader doesn't have to mentally execute the code to understand the intent, and the author has nowhere to make an off-by-one mistake at the index boundaries. On top of that, each method has a name that tells its own story: see filter, and before you even read the callback you know a shorter version of the array is coming next.

## map: transform every element

Let's start with an array of prices:

```js
const prices = [100, 250, 400, 90, 1990];
```

The map method calls a callback for every element and collects the results into a new array:

```js
const doubled = prices.map(p => p * 2);
console.log(doubled);
console.log(prices);
```

The console shows `[200, 500, 800, 180, 3980]`, followed by the original `[100, 250, 400, 90, 1990]`, untouched. map has a strict contract: however many elements go in, exactly that many come out. Five prices in, five doubled prices out, same order.

A callback can return anything, not just a number of the same kind. From numbers, for example, you can easily get display strings:

```js
const labels = prices.map(p => `${p} ₽`);
```

Now `labels` holds `['100 ₽', '250 ₽', '400 ₽', '90 ₽', '1990 ₽']`. An array of numbers turned into an array of strings, but the length still matches the original. Whenever you see map in someone's code, read it as: "every element turns into something new, and nobody gets lost."

Under the hood, map actually passes the callback more than just the element: the index arrives as the second argument, the whole array as the third. Most of the time you won't need them, but the index comes in handy when an element's position matters on its own:

```js
const numbered = prices.map((p, i) => `${i + 1}. ${p} ₽`);
```

The first price gets number 1, the second gets number 2, and so on. Only declare the parameters you actually use: `p => p * 2` and `(p, i) => p * 2` work identically, but the first is more honest about the index playing no role.

## filter: keep only what you need

filter's callback is a particular kind: it has to return `true` or `false`. A function like that is called a predicate. The method keeps only the elements in the new array for which the predicate returned `true`:

```js
const cheap = prices.filter(p => p <= 500);
```

`cheap` ends up with `[100, 250, 400, 90]`: everything priced at 500 or under. Notice: filter transforms nothing. Elements land in the result exactly as they sat in the source array, some just don't make the cut. The result's length is smaller than the source's, or equal to it if the predicate let everyone through.

## reduce: fold an array into one value

map and filter return arrays. But a cart's final total isn't an array anymore, it's a single number. That's what reduce is for:

```js
const total = prices.reduce((sum, p) => sum + p, 0);
```

There are more moving parts here, so let's take it step by step. reduce's callback takes two main arguments: an accumulator, `sum`, and the current element, `p`. The accumulator acts like a notepad passed from one step to the next: whatever the callback returns at this element is what arrives as the accumulator at the next one. reduce's own second argument, the zero after the comma, sets the notepad's starting value.

Trace the running sum step by step: the accumulator starts at 0, becomes 100 after the first price, then 350, 750, 840, and finally 2830. The last returned value is the call's result: `total` ends up holding 2830.

Pick the starting value to match the task: 0 for a sum, 1 for a product, an empty string for concatenation, an empty array or object if you're building up a structure. Simple rule of thumb: the accumulator should have the same type as the value you want to end up with.

## Chains: stations passing the belt along

Since every one of these methods returns a new array, you can apply the next method straight to the result. Let's go back to the cart, now with real products:

```js
const products = [
  { name: 'Чайник', price: 1990, inStock: true },
  { name: 'Кружка', price: 350, inStock: true },
  { name: 'Ложка', price: 90, inStock: false },
  { name: 'Блокнот', price: 120, inStock: true },
];
```

Say the storefront needs the names of the products in stock:

```js
const names = products
  .filter(p => p.inStock)
  .map(p => p.name);
```

The chain reads top to bottom, like a description of the conveyor: filter pulls "Ложка" off the belt first, then map turns the remaining objects into strings. `names` ends up with `['Чайник', 'Кружка', 'Блокнот']`.

And here's the task from the start of the chapter, in full:

```js
const totalInStock = products
  .filter(p => p.inStock)
  .map(p => p.price)
  .reduce((sum, p) => sum + p, 0);
```

Three stations, three lines, and `totalInStock` ends up holding 2460. Station order matters: filter at the front of the chain cuts the workload for everyone downstream. Filtering four products down and then transforming the three that remain is cheaper than transforming all four and throwing away the extra.

## Edge cases you'll run into

reduce's starting value is technically optional. Skip it, and the method takes the array's first element as the starting accumulator and begins walking from the second. That sounds convenient right up until the array turns out to be empty:

```js
const empty = [];
const sum = empty.reduce((acc, p) => acc + p);
```

This line doesn't return 0, and it doesn't return undefined. It throws `TypeError: Reduce of empty array with no initial value`: there's nowhere to pull a starting accumulator from. An empty cart is a perfectly ordinary thing in a store, so remember the rule: always pass a starting value. The call `empty.reduce((acc, p) => acc + p, 0)` calmly returns 0.

A second surprise hides in arrow function syntax. An arrow with an expression body returns the result on its own: `p => p * 2`. But wrap the body in curly braces, and it becomes a block, where `return` has to be written explicitly:

```js
const broken = prices.map(p => { p * 2 });
console.log(broken);
```

The console shows `[undefined, undefined, undefined, undefined, undefined]`. The expression `p * 2` honestly computed and immediately vanished, the function without an explicit `return` returned undefined, and map dutifully packed five undefineds into a new array. The length survived, the content didn't.

## Common mistakes

**Mutating the source array inside map.** You can't do this with numbers or strings, but objects arrive in the callback by reference: they're the exact same objects sitting in the source array. Code like this corrupts the original:

```js
const discounted = products.map(p => {
  p.price = p.price * 0.9;
  return p;
});
```

The discount applied, but the objects inside `products` itself changed too: the old prices are gone for good. This happens because map copies references to the objects, not the objects themselves. The correct callback returns a new object:

```js
const discounted = products.map(p => ({ ...p, price: p.price * 0.9 }));
```

Notice the parentheses around the object: without them, the curly braces would be read as a block from the previous section, and you'd get an array of undefined.

**map instead of forEach.** If the returned array never gets stored anywhere, map was the wrong pick:

```js
products.map(p => console.log(p.name));
```

Names get printed, but along the way an array of undefined is created and immediately thrown away. Worse, a reader who sees map expects a data transformation and will waste time hunting for a result that isn't there. forEach exists for side effects; more on it below.

**A forgotten return in a longer callback.** As long as the callback fits in a single expression, the arrow returns it on its own. The moment you add an intermediate variable and curly braces, returning a value becomes your job:

```js
const titles = products.map(p => {
  const title = `${p.name}: ${p.price} ₽`;
});
```

Another array of undefined, the same edge case from above. One fix: add `return title;` as the last line. If map or filter didn't give you what you expected, check every branch of the callback for a `return` first.
