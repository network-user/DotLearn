# Values, Types, and let/const

Anna is writing her first script for an online store: add up the price of a kettle and the shipping cost, then show the customer a total. The price is 2999, shipping is 299, and the screen should read 3298. The script runs without a single error and confidently prints 2999299. No red text, no warning: the program is convinced it did everything right.

The catch is that the price came from an input field, and JavaScript treats everything coming out of an input field as text. Plus received two pieces of text and glued them together instead of adding two numbers. To keep surprises like this from breaking your programs, you need to get three things straight: what kinds of values exist in JavaScript, how to check a value's type, and how to name a value correctly. By the end of the chapter you'll read Anna's story like an open book and know exactly how to fix it.

## A name points to a value

Everything a program works with is made of values: the number 2999, the text "Austin", the answer "yes" or "no". Every value has a type, and the type determines what you can do with it: numbers follow arithmetic, text can be joined and sliced, booleans only ever answer "true" or "false".

A variable, meanwhile, is nothing like the "box of data" from old textbooks. It's more accurate to picture it as a name tag with an arrow pointing to a value. In the figure below, names sit on the left, values sit in boxes on the right, and each arrow links a name to its current value. Notice where the type is written: it lives inside the box next to the value, not on the name tag.

To create an arrow like this, declare a variable with one of two keywords: `let` or `const`.

```js
let price = 2999;
const city = 'Казань';
let isPaid = false;
```

`let` creates a name whose arrow you can later redirect to a different value. `const` creates a name with an arrow fixed forever: one assignment at declaration, and that's it.

```js
let total = 2999;
total = total + 299;
console.log(total);
```

The first line creates the name `total` and points its arrow at the number 2999. The second computes the new number 3298 and redirects the arrow to it. The old value didn't "change from the inside": numbers in JavaScript are immutable, the name just started pointing at a different box.

```js
const rate = 0.2;
rate = 0.25;
```

The second line throws `TypeError: Assignment to constant variable`. That's good news: `const` protects you from accidentally overwriting something that shouldn't change. Which gives you a rule most teams follow: declare everything with `const`, and reach for `let` only where reassignment is genuinely needed. Then every `let` in the code reads as an honest warning: this value is going to change.

A word about the names themselves. JavaScript allows almost any name, but your colleagues will thank you for meaningful ones: `totalPrice` instead of `t`, `isPaid` instead of `flag`. Words inside a name are joined camelCase-style: the first word lowercase, every following word capitalized. And watch the case: `totalPrice` and `totalprice` are two different names to the language, and a case typo creates a brand-new variable instead of referencing the old one.

Notice an important consequence of the arrow model: the type belongs to the value, not to the name. JavaScript has no way to declare "a variable of type number" the way strictly typed languages do. A name created with `let` can point at a number today and a string tomorrow. This freedom is called dynamic typing, and it cuts both ways: you write faster, but tracking types becomes your job.

## Five primitives you'll meet every day

Values in JavaScript split into two big categories: primitives and objects. There are five primitive types you'll run into constantly:

- `number`: every number at once, integers like 2999, decimals like 3.14, negatives; there's no separate integer type.
- `string`: text in single or double quotes, for example `'Austin'` or `"kettle"`.
- `boolean`: exactly two values, `true` and `false`.
- `undefined`: "there's no value yet"; a variable declared without an assignment gets this.
- `null`: "there's no value, and that was intentional"; an emptiness the programmer put there on purpose.

Two more primitives, `bigint` for very large integers and `symbol` for unique keys, show up much less often: at the start it's enough to know they exist.

The second category covers everything else: objects. An object bundles several values under one roof: an order in a store has a product name, a price, and a paid flag. Arrays and functions are technically objects too, and we'll get to them in later chapters.

The `typeof` operator tells you a value's type: it takes a value and returns a string naming the type.

```js
console.log(typeof 2999);
console.log(typeof 'Казань');
console.log(typeof true);
console.log(typeof undefined);
```

The code prints `number`, `string`, `boolean`, and `undefined`. Open your browser console with F12, go to the Console tab, and try it yourself: type `typeof 'Ozon'`, then `typeof (2 + 2)`. The console runs JavaScript instantly and forgives every mistake, making it the best playground for the experiments in this chapter.

## const locks the arrow, not the contents

The most common misunderstanding about `const` goes like this: "it's a constant, so nothing inside it can change." For primitives that's true: the number 2999 can't be "edited," only replaced entirely, and replacement is exactly what `const` forbids. But objects tell a different story, worth seeing in a picture.

Let's check it in code:

```js
const order = { item: 'чайник', price: 2999 };
order.price = 2599;
console.log(order.price);
```

The code prints 2599 without throwing anything. `const` locked the arrow from the name `order` to the object, so reassigning `order` entirely won't work. But the object itself stayed an ordinary, mutable value: you can write a new price into the `price` field, add a `delivery` field to the object. The arrow stands still, the box lives its own life. Remember the formula: `const` protects the reference, not the contents.

## Three surprises that catch every newcomer

<MarginNote label="History">
  Brendan Eich put together the first version of JavaScript in ten days in May 1995. Several
  of the language's quirks, including the famous behavior of typeof null, grow straight out of that rush.
</MarginNote>

Now let's return to Anna's script and unpack its surprise alongside two relatives.

```js
console.log('2' + 2);
console.log(2 + 2);
```

The first line prints `22`, the second prints `4`. When at least one operand of `+` is a string, JavaScript converts the other operand to a string and glues them together. This automatic type conversion is called coercion. A fun detail: multiplication behaves the exact opposite way. The expression `'5' * 2` gives the number `10`, because `*` has no string mode and converts the string to a number instead. One operator pulls everything toward strings, the other toward numbers: you can't rely on guessing here, you rely on `typeof` and explicit conversion via `Number()`.

This, by the way, is exactly how Anna's script gets fixed. Convert the value from the input field to a number explicitly, just once, and the plus sign works as addition again:

```js
const rawPrice = '2999';
const price = Number(rawPrice) + 299;
console.log(price);
```

The code prints 3298, just as Anna wanted. Rule for the future: anything that arrives from outside, an input field, a URL, a file, gets converted to the right type first and only then goes into calculations.

The second surprise hides in decimals:

```js
console.log(0.1 + 0.2);
console.log(0.1 + 0.2 === 0.3);
```

It prints `0.30000000000000004`, then `false`. This isn't a JavaScript bug: nearly every language, including Python and Java, stores decimals in binary format, where 0.1 can only be represented approximately. Adding two such approximations gives a third approximation, and it lands slightly above 0.3. For money, the practical takeaway is: store amounts as whole cents, and compare decimals with a tolerance instead of strict equality.

The third surprise earned a place of honor in the language's history:

```js
console.log(typeof null);
```

It prints `object`. Not `null`, which would make sense, but `object`. This is a well-known bug from the first 1995 implementation: a value's type was encoded by an internal tag, objects had a zero tag, and `null`, represented by a zero pointer, ended up in the same category. The standards committee has discussed fixing it more than once and backed off every time: over thirty years, millions of pages of code came to depend on the current behavior. Which gives you a rule: check for `null` with strict comparison, `value === null`, because `typeof` won't do the job.

## Common mistakes

Three mistakes trip up newcomers more than any others, and the arrow model now makes all three easy to explain.

**Reassigning `const`.** The code throws `TypeError: Assignment to constant variable`. This usually happens when a name was declared as a constant and later you decide to accumulate a result in it. Two fixes: either honestly switch the declaration to `let`, or, more often the right call, give the new value a new name.

**Confusing `undefined` and `null`.** Both values signal "nothing here," so newcomers use them interchangeably. The difference is authorship. `undefined` is set by the language itself: a variable without a value, a missing object field, a function without a return. `null` is set by you, when you want to explicitly say "there's nothing here, and I checked." Stick to the convention: `null` is written by a human, `undefined` is left to the language. Then the check `value === null` reads as a meaningful signal, not an accident.

**Declaring without `let` or `const`.** The line `total = 2999` without a keyword doesn't throw an error in old "sloppy" mode: it silently creates a global variable visible from anywhere in the program. Two functions that happen to pick the same name will start overwriting each other's data, and that kind of bug can take hours to find. In strict mode and inside modules, the same code honestly throws `ReferenceError`. The rule has no exceptions: every name is born with `const` or `let`.

<Detail summary="Whatever happened to var">
  Before the ES2015 standard, variables were declared with the word `var`, and it still shows up
  everywhere in older code. `var` has two unpleasant properties. First, it ignores block boundaries:
  a variable declared inside an if is visible outside it too. Second, it lets you declare the same
  name twice, and the second declaration silently overwrites the first. `let` and `const` fixed both
  problems, which is why new code doesn't use `var`. If you see `var` in someone else's project,
  you're looking at either pre-2015 code or an old habit of the author.
</Detail>

<Detail summary="Why 0.1 + 0.2 doesn't equal exactly 0.3">
  Numbers in JavaScript are stored as IEEE 754 double-precision: 64 bits, binary. The fraction 0.1
  is infinite in binary, the way 1/3 is infinite in decimal, so what actually lands in memory is the
  nearest representable number, slightly off from a true tenth. The sum of two such approximations
  produces a third approximation, and it comes out slightly above 0.3. For display, the `toFixed`
  method helps; for money, whole cents are more reliable; and for comparing decimals, use a
  tolerance: `Math.abs(a - b) < 1e-9`.
</Detail>

Before moving on to the exercises, give yourself a one-minute check in the console. Predict the result of each expression, then type it in and compare: `typeof ''`, `typeof 0`, `'10' * '2'`, `'10' + '2'`. If all four predictions matched, your mental model is already working.
