# Data models: how databases are built

You have data: users, orders, messages, likes. You need somewhere to store it. The honest first answer - "in a file" - works right up until two processes try to write at once, you need to find one record among a million, or the power cuts out mid-write and the file is left half-updated. A database is software that takes on three boring but critical jobs: store reliably, find quickly, and change safely.

The catch is that "database" is not one tool but several large families, and they diverge on the most basic question: what shape can data take. This first article does not teach a specific DBMS. It maps the territory and gives you axes for comparing any database you meet later - in this course and at work.

## What is a "data model"

A data model is the shape a database imposes on your data, plus the set of operations it makes cheap. Those two parts are inseparable. The relational model spreads everything across tables and joins them almost for free on a shared key. The document model stores nested structures whole and returns a document by identifier for free, but joining two documents is your problem. The graph model makes a relationship a first-class object and walks it in one step, while in tables the same link is a `JOIN` that gets costlier the deeper you go.

Remember this as the main idea of the course: **no model is cheap for everything**. Each one makes some operations cheap at the price of others becoming expensive or awkward. Choosing a database is choosing which operations in your system are most frequent.

The diagram shows that models did not replace one another cleanly. The relational model beat navigational ones not because it was faster, but because it decoupled queries from physical storage layout. NoSQL rose forty years later not as a "SQL killer" but as an answer to loads a single relational machine no longer scaled to. Today all four families coexist, and mature systems often use several at once.

## Five axes of comparison

To avoid drowning in marketing, it helps to place any database on five axes. Later in the course, each model is a specific point in this space.

**Schema.** A rigid schema (*schema-on-write*) is checked at write time: a relational database will not let you insert a row that puts text in a numeric column. A flexible schema (*schema-on-read*) lets you store almost anything; you figure out structure when reading. Rigidity catches errors early; flexibility does not block rapid shape changes.

**Relationships.** How does the model express that an order belongs to a user? Relational - foreign key and `JOIN` at read time. Document - nesting: orders live inside the user. Graph - an explicit edge. That choice determines which queries are cheap.

**Scaling.** Vertical - buy a bigger machine; simple, but you hit a ceiling and price wall. Horizontal - spread data across many machines (*sharding*) and keep copies (*replication*); cheaper in hardware, but it introduces a whole class of new problems we will return to.

**Consistency.** A strict model (ACID) guarantees that after a successful write all readers see fresh data. Eventual consistency allows copies to diverge for a while and converge later. Strictness is convenient for programmers; eventualness survives better in a distributed system.

**Query model.** Declarative (you describe *what* you want, as in SQL, and the engine decides *how*), key access (return the value for this key - and nothing else), or traversal (walk links from a starting node). This is the interface through which you talk to the database at all.

## The axis that breaks everything: the network

While the database lives on one machine, you can juggle the axes above freely. Once data is spread across nodes, an unpleasant theorem kicks in. If the network between nodes is partitioned - and in a large system it partitions regularly - the database must choose: keep answering, risking stale data (*availability*), or refuse to answer rather than lie (*consistency*). There is no third option during a partition. That is the CAP theorem: of Consistency, Availability, and Partition tolerance, a distributed database during a partition keeps only two.

## Surprise: "schemaless" is a myth

The most common misconception about NoSQL hides in the word itself. NoSQL is not "no SQL" but "not only SQL": not a rejection of SQL, but an admission that the relational model has limits. Many NoSQL databases today even accept SQL-like queries.

The second misconception is that "schemaless" databases exist. There is no schema on the database side. But data still gets read somewhere - in application code - and that code expects a document to have an `email` field as a string, not a number. The schema did not vanish: it moved into the application and became implicit. Sometimes that is convenient (change data shape without migrations); sometimes it is a disaster (three format versions in one collection and nobody remembers which fields are required). "Schemaless" is more honestly called "schema-on-read": the schema exists, it is just checked later and not by the database.

<Callout type="warning" title="NoSQL does not mean faster">
Speed is not a property of the model but a consequence of the access pattern. A key-value store returns a value by key faster than a relational database for that pattern, but a relational database by primary key hits the same note. Once the query gets complex, the advantage can flip. "Faster" without naming the query is marketing, not engineering.
</Callout>

## Logical model versus storage engine

Another confusion worth separating early: the data model (how you *think* about data) and the storage engine (how bytes sit on disk) are different layers. PostgreSQL is relational but can store JSON documents. MySQL lets you pick a storage engine. The same relational model can sit on a B-tree for transactions or an LSM-tree for heavy writes. When people say "this database is faster on writes," they usually mean the engine, not the model. In this course we focus on logical models, but remember there is always a second layer underneath.

## Common mistakes

**Choosing a model by hype.** "Everyone uses MongoDB, so will we" - and a year later the team reimplements in application code what a relational database did in one `JOIN` line. Access pattern first, then model.

**Treating "schemaless" as no schema work.** The schema just became implicit and moved into code. Without discipline that is not freedom but technical debt.

**Confusing model and engine.** "Relational databases are slow" means nothing until you name the engine, indexes, and query.

**Thinking distribution is free.** Spread across nodes - you get CAP, network latency, and consistency in question. One machine, while it suffices, is often smarter than a cluster.

<Detail summary="A bit of history: where these families came from">

The hierarchical model took shape in IBM's IMS in the mid-1960s for the Apollo program. The network model was standardized by CODASYL in the early 1970s. In 1970 Edgar Codd published the paper proposing the relational model and decoupling queries from physical storage - for which he later received the Turing Award. Relational databases dominated for thirty years. The term NoSQL in its modern sense settled around 2009, when web loads and horizontal scaling pushed document, key-value, column, and graph databases onto the stage.

</Detail>
