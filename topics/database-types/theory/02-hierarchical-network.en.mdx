# Hierarchical and network models: databases you walked by hand

Before the relational model, databases were *navigational*. You did not ask "give me all employees in sales" in one phrase - you placed a cursor on the "sales department" record, said "go down to the first subordinate," then "move to the next," and walked the data by hand, record by record, like a paper archive with cross-references. The two main models of that era - hierarchical and network - look museum-piece today, but they are worth understanding for two reasons. First, you see what the relational model actually revolutionized. Second, their ideas never went away: trees and graphs of records surround you now under other names.

## Hierarchy: every record has one parent

The hierarchical model took shape in IBM's IMS in the mid-1960s - built for the Apollo lunar program. Data is a tree: records (segments in IMS terms) linked by parent-child, and each record has exactly one parent. Access is navigational: you stand on a node and move up to the parent, down to a child, or sideways to a sibling.

<HierarchyTreeFigure />

Play with the diagram: pick any record and the path to it lights up - always unique, from root down. That is both the strength and the weakness. The strength is that one-to-many fits perfectly: a company has many departments, a department has many employees, and a tree expresses that naturally. Reading along such a path is fast: descend from the root by pointers.

The weakness appears as soon as you need many-to-many. Imagine a warehouse: one part is used in several products. In a tree a part can have only one parent, so you duplicate the part - one copy under each product.

```text
Product A
  └─ Part "M3 screw"   (copy 1)
Product B
  └─ Part "M3 screw"   (copy 2)
```

Duplication brings *update anomalies*: the screw price changed - and you must find and fix every copy. Miss one and data contradicts itself. That is a fundamental problem, not an implementation oversight: a tree physically cannot express one child with two parents.

## Network: a record with several owners

The network model, standardized by CODASYL in the early 1970s, fixed exactly that pain. It allowed a record to belong to several *sets* - several owner-parents. Data became not a tree but a graph.

<NetworkModelFigure />

Hover over a student record: every set it belongs to highlights. "Ivan" belongs both to set "Group A" and set "SQL Course" - impossible in a pure tree, normal in the network model. A warehouse part now exists once and links to several products. Duplication is gone.

You paid in complexity. Queries stayed navigational: to get all students in a course, the programmer wrote a walk - "take the set owner, go to the first member, to the next, until done." Links between records were baked into physical pointers, so application code was welded to how data lay on disk. Change the database structure and you rewrote queries. A new business question - a new traversal block. There was no declarative "describe what you want, the database figures out how."

## Why they were displaced

In 1970 Edgar Codd proposed the relational model and did what neither hierarchy nor network had: decoupled the logical query from physical storage. In a relational database you write *what* you want; the engine decides *how* to get it and in what order to walk data. Many-to-many stopped being a model problem - it is an ordinary link table. Navigational code tied to storage layout disappeared. That decoupling, not speed, buried navigational databases as the mainstream.

<Callout type="info" title="They did not die - they hid">
The hierarchical model surrounds you: a filesystem directory tree, DNS domain names, an HTML page DOM, JSON document structure - all trees of one parent, many children. LDAP and Active Directory catalogs are hierarchy too. The network, graph idea reappeared in graph databases covered in the last lesson.
</Callout>

## Common misconceptions

**"Hierarchical databases are just obsolete junk."** The model aged as a universal enterprise engine, but as a structure it is optimal where data really is a tree. The filesystem is hierarchical for a reason.

**"The network model is the same as a graph database."** The idea is shared (data is a graph), but the implementation is opposite. CODASYL required manual navigational traversal and rigid set schemas; a modern graph database offers a declarative query language over relationships. Related, not identical.

**"A tree is always simpler, so always better."** A tree is simpler only until the first many-to-many link. After that simplicity turns into duplication and update anomalies, and network or relational models are more honest.

<Detail summary="What is an update anomaly">

An update anomaly is when duplication forces one logical change to require several physical edits, and any missed edit makes data inconsistent. If the price of "M3 screw" lives in two places and you updated only one, the database now claims two different prices for the same part. Relational normalization (next lesson) fights exactly this, aiming for each fact stored in exactly one place.

</Detail>
