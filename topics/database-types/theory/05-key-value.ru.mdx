# Ключ-значение: самая простая база и цена её простоты

Если выкинуть из базы данных всё, что можно, останется хранилище ключ-значение. Это гигантский словарь: вы кладёте значение под ключом и достаёте его по тому же ключу. Никаких таблиц, схем, связей и языка запросов - две операции, `SET` и `GET`, и почти мгновенный доступ. На этой аскезе стоят Redis, Amazon DynamoDB, Memcached и десятки других. Именно простота делает их незаменимыми там, где реляционная база избыточна: кэш, сессии, счётчики, очереди, рейтинги.

## Словарь размером с базу

Ментальная модель проста до предела: представьте `dict` из вашего языка программирования, только живущий отдельным сервером и переживающий перезапуск приложения. Ключ - обычно строка. Значение - что угодно: число, строка, сериализованный JSON, бинарный блоб. Доступ идёт по ключу за один шаг.

Запросите на схеме любой ключ. Он прогоняется через хеш-функцию, та выдаёт номер корзины, и значение находится за одно обращение - это и есть обещанная сложность O(1). Не важно, миллион у вас ключей или сто: попадание по ключу стоит примерно одинаково. Отсюда и главный сценарий - всё, где нужно быстро достать заранее известную вещь по известному адресу.

```text
SET user:1001  "Анна"
GET user:1001  → "Анна"
SET session:ab12  "{token: ...}"  EX 3600
GET session:ab12  → "{token: ...}"
```

Обратите внимание на имена ключей. Раз языка запросов нет, структуру наводят соглашением об именах: `user:1001`, `session:ab12`, `cart:1001`. Двоеточие - не синтаксис базы, а общепринятая привычка делать ключи читаемыми и группируемыми по префиксу.

## Значение непрозрачно

Ключевое ограничение спрятано в слове «значение». Для базы оно - чёрный ящик: она хранит и отдаёт его по ключу, но не заглядывает внутрь. Вы не можете спросить «дай все значения, где `age > 30`» - база не знает, что внутри значения вообще есть `age`. Запрос по содержимому невозможен; единственный вход - ключ.

Это и есть цена простоты. Та самая черта, что даёт скорость и тривиальное горизонтальное масштабирование (ключи легко раскидать по узлам хеш-функцией), отнимает гибкость. Нет `JOIN`, нет фильтра по полю, нет сортировки по значению. Если вам нужно искать данные не по первичному ключу, ключ-значение в чистом виде вам не подходит.

<Callout type="info" title="TTL: данные, которые сами исчезают">
Многие ключ-значение базы умеют ставить значению срок жизни (TTL): через N секунд ключ автоматически удаляется. Для кэша и сессий это идеально - устаревшее само убирается, без фоновой уборки. В реляционной базе такого «самоуничтожения по таймеру» из коробки обычно нет.
</Callout>

## Где это выстреливает

**Кэш.** Результат тяжёлого запроса или вычисления кладут под ключом с TTL. Следующий запрос берёт готовое за O(1) вместо повторного похода в медленную базу. Кэш - причина номер один, по которой Redis стоит рядом почти с любой реляционной базой.

**Сессии.** Токен сессии - естественный ключ, данные сессии - значение, TTL - срок жизни. Быстро, просто, само истекает.

**Счётчики и рейтинги.** Инкремент по ключу атомарен и мгновенен: просмотры страницы, лимиты запросов (rate limiting), лидерборды. Реляционная база с её транзакциями здесь была бы тяжеловесна.

Заметьте закономерность: во всех сценариях вы всегда знаете ключ заранее и читаете значение целиком. Ровно туда, где это так, ключ-значение и метит.

## Сюрприз: «простой» не значит «примитивный»

Redis ломает наивное представление о ключ-значении как о голом словаре. Значением там может быть не только строка, но и список, множество, хеш, отсортированное множество, и для каждого типа есть свои операции - добавить в список, пересечь множества, взять топ из отсортированного набора. При этом модель остаётся ключ-значением: всё по-прежнему адресуется ключом, просто значение бывает структурированным. Поэтому Redis называют не «базой», а «сервером структур данных»: примитив тот же, а выразительность куда богаче голого `GET/SET`.

## Типичные ошибки

**Делать его основной базой для связанных данных.** Как только вам захотелось «достать все заказы пользователя» без явного ключа на каждый - вы упёрлись в стену. Связанные данные - не территория ключ-значения.

**Пытаться искать по значению.** Перебор всех ключей со чтением значений (`SCAN` всего keyspace) - антипаттерн, убивающий смысл модели. Нужен поиск по содержимому - возьмите другую модель или постройте отдельный индекс.

**Забыть про вытеснение и память.** Многие такие базы держат данные в оперативной памяти. Без TTL и политики вытеснения (eviction) ключи копятся, пока память не кончится. Кэш без срока жизни - это утечка памяти на таймере.

**Городить ключи без соглашения.** Без префиксной дисциплины (`user:`, `session:`) keyspace превращается в кашу, где невозможно понять, что и зачем лежит.

<Detail summary="Как ключи распределяются по узлам">

Чтобы масштабироваться горизонтально, ключ-значение база раскидывает ключи по узлам с помощью хеш-функции. Наивный подход «номер узла = hash(key) % N» плох: добавили узел - сменилось N, и почти все ключи переехали. Поэтому используют *консистентное хеширование*: узлы и ключи раскладываются на одном кольце, и добавление узла перемещает лишь малую долю ключей, а не весь keyspace. Это одна из идей, благодаря которым ключ-значение масштабируется почти линейно.

</Detail>
