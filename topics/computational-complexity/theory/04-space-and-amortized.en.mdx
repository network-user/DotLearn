# Not only time: memory and average price

Until now, you have been counting steps - how many operations the algorithm will perform. But any calculation has a second count: how much **memory** it will take in addition to the input data. Sometimes it is memory that becomes the bottleneck: the algorithm runs in a second, but requires gigabytes, which are not available. And there are also operations whose honest price is sometimes tiny, sometimes suddenly large, and it makes no sense to analyze them individually - you need to average them over the series. This section is about both of these subjects: about spatial complexity and about amortized complexity.

By the end of this section, you'll learn to value extra memory the same way you counted time, see the time versus memory tradeoff where the dictionary saves passes at the cost of space, and understand why `list.append` is called `O(1)` even though it sometimes copies the entire list. This closes the picture: the algorithm is characterized not by one number, but by a pair - time and memory.

## Mental model: memory is also a function of n

Space complexity is calculated on the same principle as time complexity: it is a **function of the input size**, but it does not measure steps, but the amount of additional memory. The important word is additional. The entrance itself is already in memory; we are interested in what the algorithm does **beyond** it: temporary lists, dictionaries, a stack of recursive calls.

```python
def squares(numbers):
    result = []
    for x in numbers:
        result.append(x * x)
    return result
```

Here `result` grows with the input: for `n` elements it stores `n` numbers. Additional memory is `O(n)`. But the sum of the same squares of memory requires almost no memory:

```python
def sum_of_squares(numbers):
    total = 0
    for x in numbers:
        total += x * x
    return total
```

`total` is one variable, it does not increase with `n`. Additional memory - `O(1)`, constant. The same input, the same time complexity `O(n)`, but the memory is fundamentally different: one algorithm builds a new list, the other makes do with a single accumulator.

## The recursion stack is also memory

Memory is wasted not only on explicit lists. Each incomplete function call hangs on the call stack and takes up space until it returns. Deep recursion quietly eats `O(depth)` memory, even if it does not create any lists.

```python
def rec_sum(numbers, i=0):
    if i == len(numbers):
        return 0
    return numbers[i] + rec_sum(numbers, i + 1)
```

This function does not start any structures, but by the time of the last call all `n` frames are hanging on the stack at once: each one is waiting for the result of the next one. Memory - `O(n)`. An iterative version with a regular loop would cost `O(1)`. Therefore, deep recursion over a list is a hidden waste of memory and a risk of stack overflow.

<Callout type="warning">
Recursion of depth `n` costs `O(n)` memory on the call stack, even when the function body does not create any objects. If the recursion is linear in depth and easily rewritten into a loop, the loop is almost always more memory efficient.
</Callout>

## Exchange: time for memory and back

Memory and time can often be exchanged for one another. A classic example is checking whether two lists have a common element. Naively, this is a double loop: `O(n²)` time, `O(1)` memory. But if you first add one list into a `set`, the check becomes `O(1)` per element.

```python
def has_common(a, b):
    seen = set(a)
    for x in b:
        if x in seen:
            return True
    return False
```

Now the time is `O(n)`: one pass to build the `set`, one to check. But this comes at a cost in memory: `set` stores `n` elements, the additional memory has grown from `O(1)` to `O(n)`. This is the “time for memory” trade: we wasted space to replace the quadratic search with a linear one. Most often, such an exchange is profitable, but it is not free, and on huge inputs memory may be more important.

## Amortized difficulty: average over the series

Some operations are either cheap or expensive, and it is unfair to analyze them one by one. The most famous example is `list.append`. Usually it is instantaneous: putting an element in a free cell is `O(1)`. But when the allocated space for the list runs out, Python allocates a larger block and **copies the entire list** to a new location - and this is `O(n)`. It turns out that a separate `append` costs either `O(1)` or suddenly `O(n)`.

Amortized complexity answers the question “how much does an operation cost **on average over a long series**.” The list grows with a margin - about one and a half to two times at a time - so expensive copying happens less and less as it grows. If you spread the cost of all copies over all `appends`, there is a constant for each operation. That's why `append` is called amortized `O(1)`: not every call is cheap, but `n` calls cost `O(n)` in total, that is `O(1)` per piece.

## An edge case that will surprise you

Amortized `O(1)` does not mean that every single call is fast. If you are building a list of a million elements and the latency of **each** operation is critical (for example, in a real-time system), the rare `append` will still take `O(n)` at the time of copying - and this pause can violate the deadline. Depreciation promises a good **total** value, but is silent about the peaks.

And the opposite surprise: `list.insert(0, x)` - insertion at the beginning - **not** amortized `O(1)`, but honestly `O(n)` every time, because it shifts all the elements. A call similar to `append` turns out to be fundamentally more expensive. When frequent insertions at the beginning are needed, take `collections.deque`, which has `O(1)` amortization at both ends. Moral: the name of the method does not guarantee anything - the price must be known for each operation separately.

## Common mistakes

- **Forget about input memory versus additional memory.** Space complexity considers what the algorithm runs **over** the input. A function returning a new list of length `n` consumes `O(n)` additional memory; function with one accumulator is `O(1)`.
- **Ignore the recursion stack.** A recursion of depth `n` takes `O(n)` memory on the stack, even without a single object created. This is a common cause of `RecursionError` on large inputs.
- **Consider `insert(0, x)` as cheap as `append`.** `append` is `O(1)` amortized, but inserting at the beginning of the list is `O(n)`: shifts all elements. To have a queue at both ends, you need `deque`.

<Detail summary="Where does the list growth factor come from?">

When space in the list runs out, CPython allocates a new block using a formula approximately “current size plus a quarter” (the exact formula has changed between versions). A growth factor greater than one is the key to amortized `O(1)`: when growing by a fixed number of times, the total cost of all copies forms a geometric progression that converges to `O(n)` rather than `O(n²)`. If the list grew by a constant number of cells, rather than exponentially, the copies would happen too often, and `append` would become `O(n)` amortized. Geometric growth is what makes dynamic array practical.

</Detail>
