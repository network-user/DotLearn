# Сложность измеряется в шагах, а не в секундах

Вы написали две функции, которые делают одно и то же, и хотите понять, какая быстрее. Самый очевидный ход - засечь время: запустить обе на одних данных и сравнить миллисекунды. Но стоит пересесть за другой ноутбук, обновить Python или просто запустить замер ещё раз с открытым браузером в фоне - и числа поплывут. Секунды зависят от частоты процессора, от того, насколько прогрелся кэш, от фазы Луны над дата-центром. Они почти ничего не говорят о самом алгоритме.

Вычислительная сложность отвечает на тот же вопрос - «какая функция быстрее» - но иначе. Вместо секунд она считает **шаги**: сколько элементарных операций выполнит алгоритм в зависимости от размера входа. Это число не зависит от железа, и именно поэтому на нём можно строить осмысленные сравнения. К концу раздела Вы научитесь считать шаги, видеть, как их количество растёт с размером входа, и понимать, почему «больше шагов» иногда оказывается быстрее на практике, а иногда нет.

## Ментальная модель: функция от размера входа

Договоримся обозначать размер входа буквой `n`. Для списка `n` - это его длина, для строки - число символов, для числа - иногда количество его цифр. Сложность алгоритма - это **функция**, которая по `n` возвращает количество шагов. Не одно число, а зависимость: как меняется работа, когда вход растёт.

Возьмём поиск максимума в списке. Чтобы найти наибольший элемент, нужно посмотреть на каждый ровно один раз. Список из 10 элементов - 10 сравнений, из 1000 - 1000 сравнений. Количество шагов равно `n`. График этой зависимости - прямая линия: удвоили вход, удвоили работу.

```python
def find_max(numbers):
    best = numbers[0]
    for x in numbers:
        if x > best:
            best = x
    return best
```

Тело цикла выполняется `n` раз, всё остальное - константа. Мы говорим, что у `find_max` линейная сложность. А теперь сравним рост разных зависимостей на одном графике: как ведёт себя `n` против `n` в квадрате и против `log n`.

Главное на этом графике - не конкретные числа, а форма кривых. При `n = 32` логарифм даёт 5 шагов, линия - 32, квадрат - больше тысячи. Чем дальше вправо, тем сильнее расходятся линии. Именно эта форма роста, а не значение в одной точке, определяет, выдержит ли алгоритм большой вход.

## Считаем шаги: цикл за циклом

Чтобы оценить сложность, не нужно запускать код - достаточно прочитать его и посчитать, сколько раз выполнится каждая строка. Начнём с одного цикла.

```python
def sum_list(numbers):
    total = 0
    for x in numbers:
        total += x
    return total
```

Строка `total = 0` выполняется один раз, `return` - один раз, а `total += x` - ровно `n` раз. Итого `n + 2` шага. Размер входа удваивается - число шагов почти удваивается. Это снова линейный рост.

Теперь вложим один цикл в другой. Каждая итерация внешнего цикла запускает внутренний целиком.

```python
def count_pairs(numbers):
    pairs = 0
    for a in numbers:
        for b in numbers:
            pairs += 1
    return pairs
```

Внешний цикл крутится `n` раз, и на каждой его итерации внутренний делает ещё `n` шагов. Перемножаем: `n × n = n²` выполнений строки `pairs += 1`. Для списка из 10 элементов это 100 шагов, для 100 элементов - уже 10000. Вложенность циклов перемножает их длины - вот откуда берётся квадрат.

<Callout type="tip">
Простое правило для первого приближения: последовательные блоки кода складываются, вложенные - перемножаются. Цикл рядом с циклом - это `n + n`, цикл внутри цикла - это `n × n`.
</Callout>

## Не всё, что выглядит как цикл, линейно

Бывает и наоборот: цикл есть, но шагов гораздо меньше `n`. Посмотрите на двоичный поиск по отсортированному списку. На каждом шаге он выбрасывает половину оставшихся кандидатов.

```python
def binary_search(sorted_numbers, target):
    lo, hi = 0, len(sorted_numbers) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if sorted_numbers[mid] == target:
            return mid
        if sorted_numbers[mid] < target:
            lo = mid + 1
        else:
            hi = mid - 1
    return -1
```

Из 1000 элементов после первого сравнения остаётся 500, потом 250, 125 и так далее. Сколько раз можно делить 1000 пополам, пока не останется один? Около десяти. Это и есть логарифм: число шагов растёт как `log₂ n`. Удвоение входа добавляет ровно один шаг, а не удваивает работу. Поэтому двоичный поиск по миллиону элементов укладывается в 20 сравнений.

## Живая песочница: посчитайте шаги сами

Ниже три функции снабжены счётчиком `steps`, который увеличивается на каждой существенной операции. Запустите код и посмотрите на числа. Затем **поменяйте `n` с 8 на 16** и запустите снова: проследите, как линейный счётчик удвоится, квадратичный вырастет вчетверо, а логарифмический добавит всего один шаг.

<PyDemo
  title="Счётчик шагов для трёх законов роста"
  code={`def linear_steps(n):
    steps = 0
    for _ in range(n):
        steps += 1
    return steps

def quadratic_steps(n):
    steps = 0
    for _ in range(n):
        for _ in range(n):
            steps += 1
    return steps

def log_steps(n):
    steps = 0
    size = n
    while size > 1:
        size //= 2
        steps += 1
    return steps

n = 8
print("linear:", linear_steps(n))
print("quadratic:", quadratic_steps(n))
print("log:", log_steps(n))`}

## Лучший, худший и средний случай

Число шагов не всегда зависит только от `n`. Иногда оно зависит ещё и от того, *какие именно* данные пришли. Линейный поиск элемента в списке - хороший пример.

```python
def index_of(numbers, target):
    for i, x in enumerate(numbers):
        if x == target:
            return i
    return -1
```

Если искомое стоит первым, функция вернётся после одного сравнения - это лучший случай, одна операция независимо от `n`. Если его нет вообще, придётся просмотреть весь список - худший случай, `n` сравнений. Когда говорят «сложность линейного поиска - линейная», имеют в виду **худший случай**: гарантию сверху, которая верна для любых данных. Именно худший случай чаще всего интересен на практике, потому что он отвечает на вопрос «насколько плохо может стать».

<MarginNote label="Почему худший случай">Худший случай даёт гарантию, не зависящую от удачи. Средний случай тоже полезен, но требует знать распределение входных данных, а его обычно нет под рукой.</MarginNote>

## Краевой случай, который вас удивит

Кажется логичным: меньше шагов - быстрее работает. Но на реальных, небольших входах это правило часто врёт. Сравните алгоритм со сложностью `n²` и алгоритм со сложностью `100 × n`. Асимптотически второй лучше - при больших `n` линия обгонит любую константу. Но при `n = 50` квадратичный делает 2500 шагов, а «линейный» с большой константой - 5000. На таких размерах формально худший алгоритм быстрее.

Сложность описывает поведение при **больших** `n`, в пределе. Она намеренно игнорирует константы и поведение на маленьких входах, потому что на дистанции форма роста всё равно победит. Но если ваши входы заведомо малы, замер времени иногда честнее теории. Сложность - инструмент для рассуждений о масштабировании, а не универсальный вердикт «этот код быстрее».

## Типичные ошибки

- **Считать секунды свойством алгоритма.** Время на вашей машине зависит от железа, нагрузки и кэша. Свойство алгоритма - это закон роста числа шагов, а не миллисекунды конкретного запуска.
- **Думать, что любой цикл линеен.** Двоичный поиск содержит `while`, но делает `log n` шагов, а не `n`. Считать надо реальное число итераций, а не сам факт наличия цикла.
- **Путать число итераций с числом строк кода.** Десять строк внутри цикла, выполняемого `n` раз, - это работа порядка `n`, а не 10. Сложность считает повторения, а не длину текста.

<Detail summary="Почему мы вообще игнорируем константы">

В число шагов входят сложение, сравнение, присваивание - и каждая из этих операций на разном железе стоит по-разному. Точное количество тактов процессора посчитать невозможно и бессмысленно: оно меняется от машины к машине. Поэтому теория сложности отбрасывает постоянные множители и младшие слагаемые, оставляя только доминирующий член. `3n² + 5n + 17` и `n²` ведут себя одинаково при больших `n`: квадрат растёт быстрее всего, и именно он определяет судьбу алгоритма. Формальный язык для этого отбрасывания - O-нотация, ей посвящён следующий раздел.

</Detail>
