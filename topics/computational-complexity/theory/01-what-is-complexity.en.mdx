# Difficulty is measured in steps, not seconds

You've written two functions that do the same thing, and you want to know which is faster. The most obvious move is to time it: run both on the same data and compare the milliseconds. But if you switch to another laptop, update Python, or simply run the measurement again with the browser open in the background, the numbers will start to flow. Seconds depend on the processor frequency, how warm the cache is, and the phase of the moon over the data center. They say almost nothing about the algorithm itself.

Computational complexity answers the same question - "which function is faster" - but in a different way. Instead of seconds, it counts **steps**: how many elementary operations the algorithm will perform depending on the size of the input. This number is independent of hardware, which is why meaningful comparisons can be made on it. By the end of this section, you'll be able to count steps, see how the number of steps increases with input size, and understand why "more steps" is sometimes faster in practice and sometimes not.

## Mental model: function of input size

Let's agree to denote the input size with the letter `n`. For a list, `n` is its length, for a string - the number of characters, for a number - sometimes the number of its digits. Algorithm complexity is a **function** that returns the number of steps given by `n`. Not one number, but a dependence: how the work changes when the input grows.

Let's take the search for the maximum in the list. To find the largest element, you need to look at each element exactly once. A list of 10 elements - 10 comparisons, from 1000 - 1000 comparisons. The number of steps is `n`. The graph of this dependence is a straight line: double the input, double the work.

```python
def find_max(numbers):
    best = numbers[0]
    for x in numbers:
        if x > best:
            best = x
    return best
```

The body of the loop is executed `n` times, everything else is constant. We say that `find_max` has linear complexity. Now let’s compare the growth of different dependencies on one graph: how `n` behaves against `n` squared and against `log n`.

The main thing on this graph is not the specific numbers, but the shape of the curves. With `n = 32` the logarithm gives 5 steps, the line - 32, the square - more than a thousand. The further to the right, the more the lines diverge. It is this form of growth, not the value at one point, that determines whether the algorithm can handle a large input.

## Counting steps: cycle by cycle

To estimate the complexity, you don't need to run the code - just read it and count how many times each line will be executed. Let's start with one cycle.

```python
def sum_list(numbers):
    total = 0
    for x in numbers:
        total += x
    return total
```

The line `total = 0` is executed once, `return` is executed once, and `total += x` is executed exactly `n` times. Total `n + 2` steps. The size of the entrance doubles - the number of steps almost doubles. This is again linear growth.

Now let's nest one cycle inside another. Each iteration of the outer loop runs the entire inner loop.

```python
def count_pairs(numbers):
    pairs = 0
    for a in numbers:
        for b in numbers:
            pairs += 1
    return pairs
```

The outer loop rotates `n` times, and at each iteration the inner loop takes `n` more steps. Multiply: `n × n = n²` executions of the line `pairs += 1`. For a list of 10 elements this is 100 steps, for 100 elements it is already 10,000. Nesting of loops multiplies their lengths - this is where the square comes from.

<Callout type="tip">
A simple rule for a first approximation: successive blocks of code are added, nested ones are multiplied. A loop next to a loop is `n + n`, a loop inside a loop is `n × n`.
</Callout>

## Not everything that looks like a loop is linear

It also happens the other way around: there is a cycle, but there are much fewer steps `n`. Look at binary search on a sorted list. At each step, it throws out half of the remaining candidates.

```python
def binary_search(sorted_numbers, target):
    lo, hi = 0, len(sorted_numbers) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if sorted_numbers[mid] == target:
            return mid
        if sorted_numbers[mid] < target:
            lo = mid + 1
        else:
            hi = mid - 1
    return -1
```

Out of 1000 elements, after the first comparison, 500 remain, then 250, 125, and so on. How many times can you divide 1000 in half until there is only one left? About ten. This is the logarithm: the number of steps grows as `log₂ n`. Doubling the input adds exactly one step, rather than doubling the work. Therefore, a binary search over a million elements takes 20 comparisons.

## Live sandbox: count the steps yourself

The three functions below are provided with a `steps` counter, which is incremented with each significant operation. Run the code and look at the numbers. Then **change `n` from 8 to 16** and run again: watch as the linear counter doubles, the quadratic counter quadruples, and the logarithmic counter adds just one step.

<PyDemo
  title="Step counter for the three laws of growth"
  code={`def linear_steps(n):
    steps = 0
    for _ in range(n):
        steps += 1
    return steps

def quadratic_steps(n):
    steps = 0
    for _ in range(n):
        for _ in range(n):
            steps += 1
    return steps

def log_steps(n):
    steps = 0
    size = n
    while size > 1:
        size //= 2
        steps += 1
    return steps

n = 8
print("linear:", linear_steps(n))
print("quadratic:", quadratic_steps(n))
print("log:", log_steps(n))`}
/>

Please note: with `n = 8` the quadratic function already outperforms the linear function by eight times, and with `n = 16` - by sixteen. The gap grows with the input, and no amount of constant optimization will close it.

## Best, worst and average case

The number of steps does not always depend only on `n`. Sometimes it also depends on *exactly* what data was received. Linear searching for an element in a list is a good example.

```python
def index_of(numbers, target):
    for i, x in enumerate(numbers):
        if x == target:
            return i
    return -1
```

If what you are looking for comes first, the function will return after one comparison - this is the best case, one operation regardless of `n`. If there is none at all, you will have to look through the entire list - worst case, `n` comparisons. When they say “the complexity of a linear search is linear,” they mean the **worst case**: a guarantee from above that is true for any data. It is the worst case that is most often interesting in practice, because it answers the question “how bad can it get.”

<MarginNote label="Why worst case">Худший случай даёт гарантию, не зависящую от удачи. Средний случай тоже полезен, но требует знать распределение входных данных, а его обычно нет под рукой.</MarginNote>

## An edge case that will surprise you

It seems logical: fewer steps means faster work. But on real, small inputs this rule often lies. Compare an algorithm with complexity `n²` and an algorithm with complexity `100 × n`. Asymptotically, the second one is better - for large `n` the line will outperform any constant. But with `n = 50` the quadratic one takes 2500 steps, and the “linear” one with a large constant takes 5000. At such sizes, the worse algorithm is formally faster.

Complexity describes the behavior for **large** `n`, in the limit. It deliberately ignores constants and behavior on small inputs, because at a distance the growth form will still win. But if your inputs are obviously small, timing is sometimes more honest than theory. Complexity is a tool for reasoning about scaling, not a universal verdict of “this code is faster.”

## Common mistakes

- **Count seconds as a property of the algorithm.** The time on your machine depends on the hardware, load and cache. The property of the algorithm is the law of growth of the number of steps, and not the milliseconds of a specific run.
- **Assume that any loop is linear.** Binary search contains `while`, but takes `log n` steps rather than `n`. It is necessary to count the actual number of iterations, and not the very fact of the existence of a cycle.
- **Confusing the number of iterations with the number of lines of code.** Ten lines inside a loop executed `n` times is `n` order of work, not 10. Complexity counts repetitions, not length of text.

<Detail summary="Why do we ignore constants at all?">

The steps include addition, comparison, assignment - and each of these operations costs differently on different hardware. It is impossible and meaningless to calculate the exact number of processor cycles: it varies from machine to machine. Therefore, complexity theory discards constant factors and minor terms, leaving only the dominant term. `3n² + 5n + 17` and `n²` behave the same for large `n`: the square grows the fastest, and it is this that determines the fate of the algorithm. The formal language for this discarding is O-notation, which is discussed in the next section.

</Detail>
