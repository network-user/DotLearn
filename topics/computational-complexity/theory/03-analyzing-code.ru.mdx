# Читаем сложность прямо из кода

Теория про шаги и про O красиво звучит, пока перед Вами не оказывается чужая функция на тридцать строк с двумя циклами, рекурсивным вызовом и обращением к словарю внутри. Вопрос «какая у этого сложность» из абстрактного становится практическим: нужно посмотреть на код и назвать класс роста, не запуская его. Это навык, и он раскладывается на несколько устойчивых правил, которые срабатывают почти всегда.

К концу раздела Вы научитесь определять сложность по структуре кода: складывать последовательные блоки, перемножать вложенные циклы, узнавать логарифм по делению задачи пополам, видеть скрытую стоимость операций со списками и строками и оценивать рекурсию через число и размер вызовов. Это та самая работа, которую делают на собеседованиях и в код-ревью, когда спрашивают «а как это масштабируется».

## Ментальная модель: сложить последовательное, перемножить вложенное

Любой код - это дерево из блоков. Два правила покрывают почти все случаи. **Последовательные блоки складываются**: сначала один цикл на `n`, потом другой на `n` - это `n + n = O(n)`. **Вложенные блоки перемножаются**: цикл на `n` внутри цикла на `n` - это `n × n = O(n²)`. После сложения и перемножения остаётся выбросить константы и младшие члены, как в прошлом разделе.

Слева два цикла стоят друг за другом - их вклады складываются. Справа один внутри другого - вклады перемножаются. Глядя на код, Вы по сути обходите это дерево: спускаясь во вложенность, умножаете, идя по соседним блокам, складываете. Дальше разберём случаи, где это правило прячется.

## Вложенность не всегда квадрат

Двойной цикл соблазнительно сразу записать в `O(n²)`, но смотреть надо на реальное число итераций внутреннего цикла, а не на сам факт вложенности. Классическая ловушка - внутренний цикл, который зависит от внешнего.

```python
def upper_triangle(items):
    pairs = 0
    for i in range(len(items)):
        for j in range(i + 1, len(items)):
            pairs += 1
    return pairs
```

Внутренний цикл делает `n-1` шаг на первой итерации, `n-2` на второй, и так до нуля. Сумма `(n-1) + (n-2) + ... + 1` равна `n(n-1)/2` - это примерно `n²/2`. Константа `1/2` выбрасывается, остаётся `O(n²)`. Парадокс в том, что цикл проходит лишь половину пар, но класс сложности всё равно квадратичный: половина от квадрата - это квадрат.

<Callout type="warning">
Не считайте сложность по числу строк или по тому, «насколько глубоко вложено». Считайте, сколько раз реально выполнится самая внутренняя операция. Цикл `range(i, n)` внутри `range(n)` - это сумма, а не произведение `n × n`, но упрощается всё равно до `O(n²)`.
</Callout>

## Логарифм прячется в делении пополам

Если на каждом шаге задача уменьшается **в несколько раз**, а не на единицу, появляется логарифм. Признак - переменная, которую делят (`size //= 2`, `lo, hi` сходятся, индекс прыгает в середину), а не уменьшают на один.

```python
def count_halvings(n):
    steps = 0
    while n > 1:
        n //= 2
        steps += 1
    return steps
```

Из `n` получается `n/2`, потом `n/4`, и так до единицы. Число делений - это `log₂ n`: для миллиона около двадцати. Сравните с циклом `while n > 0: n -= 1`, который делает ровно `n` шагов. Разница между «делим» и «вычитаем» - это пропасть между `log n` и `n`. А если внутри логарифмического цикла стоит ещё линейный проход - получится `O(n log n)`, фирменная сложность хороших сортировок.

## Скрытая стоимость: не всё в Python - это O(1)

Самая частая ошибка анализа - принять дорогую операцию за дешёвую. Многие выражения выглядят как один шаг, а внутри прячут целый проход по данным. Несколько примеров, которые стоит запомнить.

```python
def has_dup_slow(items):
    for x in items:
        if items.count(x) > 1:
            return True
    return False
```

Цикл `for` - это `n` итераций. Но `items.count(x)` внутри сам по себе `O(n)`: он проходит весь список. Итог - `n × n = O(n²)`, хотя видимый цикл только один. То же с `x in some_list` (линейный поиск, `O(n)`), со срезами `items[1:]` (копирование, `O(n)`) и со склейкой строк в цикле через `+=`. А вот `x in some_set` и `x in some_dict` - это `O(1)`: множества и словари ищут по хэшу.

## Живая песочница: два способа найти дубликат

Ниже две функции ищут, есть ли в списке повторяющийся элемент. Первая использует `count` внутри цикла - `O(n²)`. Вторая копит виденное в `set` - `O(n)`. Счётчик `ops` считает существенные операции. Запустите код, затем **поменяйте `n` с 200 на 400** и сравните, во сколько раз вырастет каждый счётчик: квадратичный учетверится, линейный удвоится.

<PyDemo
  title="O(n²) против O(n) на поиске дубликата"
  code={`def has_dup_slow(items):
    ops = 0
    for x in items:
        seen_count = 0
        for y in items:
            ops += 1
            if y == x:
                seen_count += 1
        if seen_count > 1:
            return True, ops
    return False, ops

def has_dup_fast(items):
    ops = 0
    seen = set()
    for x in items:
        ops += 1
        if x in seen:
            return True, ops
        seen.add(x)
    return False, ops

n = 200
data = list(range(n))
print("slow:", has_dup_slow(data)[1], "ops")
print("fast:", has_dup_fast(data)[1], "ops")`}
/>

На уникальных данных обе доходят до конца, но цена прохода различается на порядки: при `n = 200` медленный вариант делает около сорока тысяч операций, быстрый - двести. Удвоение `n` разводит их ещё сильнее. Один и тот же результат, разные классы сложности.

## Рекурсия: число вызовов на работу в каждом

Рекурсию анализируют как «сколько всего вызовов» умножить на «сколько работы в одном вызове». Возьмём наивные числа Фибоначчи.

```python
def fib(n):
    if n < 2:
        return n
    return fib(n - 1) + fib(n - 2)
```

Каждый вызов порождает два новых, дерево вызовов разрастается вширь, и общее их число растёт примерно как `2ⁿ` - экспонента. Внутри каждого вызова работа константна, поэтому итоговая сложность `O(2ⁿ)`: для `n = 40` это уже миллиарды вызовов. Сравните с двоичным поиском, где вызов **один** на уровень, а уровней `log n`: там рекурсия даёт `O(log n)`. Разница - в том, ветвится дерево вызовов или вытягивается в линию.

<MarginNote label="Не считайте вызовы вручную">Для рекурсии с делением задачи (merge sort, быстрая сортировка) сложность выводят через рекуррентные соотношения и master-теорему. Здесь достаточно интуиции «ширина дерева × работа в узле».</MarginNote>

## Краевой случай, который вас удивит

Кажется, что сложность всегда определяется самым глубоким циклом. Но иногда доминирует то, что выглядит безобидно. Функция с одним циклом на `n`, внутри которого вызывается `sorted(small_chunk)` на кусочке постоянного размера, остаётся `O(n)` - сортировка константного куска стоит константу. А функция вообще без явных циклов, но с `sorted(items)` на всём входе, - это сразу `O(n log n)`, потому что стоимость спрятана внутри встроенной функции.

Мораль: смотрите не на форму кода, а на стоимость каждой операции по отдельности. Встроенные функции (`sorted`, `min`, `max`, `sum`, `in` для списка, срезы) несут свою сложность, и её надо учитывать наравне с явными циклами. Самый дорогой вклад побеждает - даже если он спрятан в одну строчку без единого `for`.

## Типичные ошибки

- **Считать вложенность по виду, а не по числу итераций.** Цикл `range(i, n)` внутри `range(n)` проходит половину пар, но это всё равно `O(n²)`; а константный внутренний цикл вложенности не добавляет вовсе.
- **Принимать дорогие операции за `O(1)`.** `x in list`, `list.count(x)`, срезы и склейка строк через `+=` линейны. Спрятанный внутри цикла линейный вызов превращает `O(n)` в `O(n²)`.
- **Путать вычитание и деление в цикле.** `n -= 1` даёт `n` шагов (линейно), `n //= 2` даёт `log n` шагов (логарифмически). Признак логарифма - уменьшение в разы, а не на единицу.

<Detail summary="Амортизация: почему append считается O(1)">

`list.append` иногда вынужден перевыделить память и скопировать весь список - в этот момент операция стоит `O(n)`. Почему же её называют `O(1)`? Потому что такие дорогие моменты случаются редко: список растёт с запасом, и стоимость копирований, размазанная по всем `append`, даёт константу на операцию. Это амортизированная сложность - усреднённая стоимость по длинной серии операций. Подробно разберём её в следующем разделе.
