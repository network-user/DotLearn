# Не только время: память и усреднённая цена

До сих пор Вы считали шаги - сколько операций выполнит алгоритм. Но у любого вычисления есть второй счёт: сколько **памяти** оно займёт сверх входных данных. Иногда именно память становится узким местом: алгоритм укладывается в секунду, но требует гигабайт, которых нет. А ещё есть операции, чья честная цена то крошечная, то внезапно большая, и поодиночке их анализировать бессмысленно - нужно усреднять по серии. Этот раздел про оба этих сюжета: про пространственную сложность и про амортизированную.

К концу раздела Вы научитесь оценивать дополнительную память так же, как считали время, видеть размен «время против памяти», когда словарь экономит проходы ценой места, и понимать, почему `list.append` называют `O(1)`, хотя иногда он копирует весь список. Это закрывает картину: алгоритм характеризуют не одним числом, а парой - временем и памятью.

## Ментальная модель: память - тоже функция от n

Пространственная сложность считается по тому же принципу, что и временная: это **функция от размера входа**, но измеряет она не шаги, а количество дополнительной памяти. Важное слово - дополнительной. Сам вход уже лежит в памяти; нас интересует то, что алгоритм заводит **сверх** него: временные списки, словари, стек рекурсивных вызовов.

```python
def squares(numbers):
    result = []
    for x in numbers:
        result.append(x * x)
    return result
```

Здесь `result` растёт вместе с входом: на `n` элементов он хранит `n` чисел. Дополнительная память - `O(n)`. А вот сумма тех же квадратов памяти почти не требует:

```python
def sum_of_squares(numbers):
    total = 0
    for x in numbers:
        total += x * x
    return total
```

`total` - одна переменная, она не растёт с `n`. Дополнительная память - `O(1)`, константа. Один и тот же вход, одна и та же временная сложность `O(n)`, но память отличается принципиально: один алгоритм строит новый список, другой обходится единственным аккумулятором.

Слева - единственная ячейка, не зависящая от `n`. Справа - список, длина которого повторяет вход. Глядя на код, спрашивайте себя: какая структура данных растёт вместе с входом? Её размер и есть пространственная сложность.

## Стек рекурсии - это тоже память

Память тратится не только на явные списки. Каждый незавершённый вызов функции висит на стеке вызовов и занимает место, пока не вернётся. Глубокая рекурсия незаметно ест `O(глубина)` памяти, даже если не создаёт ни одного списка.

```python
def rec_sum(numbers, i=0):
    if i == len(numbers):
        return 0
    return numbers[i] + rec_sum(numbers, i + 1)
```

Эта функция не заводит ни одной структуры, но к моменту последнего вызова на стеке висят все `n` кадров сразу: каждый ждёт результата следующего. Память - `O(n)`. Итеративная версия с обычным циклом обошлась бы `O(1)`. Поэтому глубокая рекурсия по списку - скрытый расход памяти и риск переполнения стека.

<Callout type="warning">
Рекурсия глубиной `n` стоит `O(n)` памяти на стеке вызовов, даже когда тело функции не создаёт никаких объектов. Если рекурсия линейна по глубине и легко переписывается в цикл, цикл почти всегда экономнее по памяти.
</Callout>

## Размен: время за память и обратно

Память и время часто можно менять одно на другое. Классический пример - проверка, есть ли в двух списках общий элемент. Наивно это двойной цикл: `O(n²)` времени, `O(1)` памяти. Но если сначала сложить один список в `set`, проверка станет `O(1)` на элемент.

```python
def has_common(a, b):
    seen = set(a)
    for x in b:
        if x in seen:
            return True
    return False
```

Теперь время - `O(n)`: один проход на построение `set`, один на проверку. Но за это заплачено памятью: `set` хранит `n` элементов, дополнительная память выросла с `O(1)` до `O(n)`. Это и есть размен «время за память»: мы потратили место, чтобы заменить квадратичный перебор линейным. Чаще всего такой обмен выгоден, но он не бесплатный, и на огромных входах память может оказаться важнее.

## Живая песочница: два решения, разная память

Ниже две функции ищут пересечение двух списков. Первая - двойной цикл без лишней памяти. Вторая - через `set`, быстрее по времени, но с дополнительным хранилищем. Счётчики `ops` и `extra_mem` показывают цену каждого подхода. Запустите код, затем **поменяйте `n` с 100 на 200** и сравните: у медленной версии операции учетверятся при нулевой доппамяти, у быстрой - операции удвоятся, но доппамять тоже удвоится.

<PyDemo
  title="Время против памяти на пересечении списков"
  code={`def common_slow(a, b):
    ops = 0
    result = []
    for x in a:
        for y in b:
            ops += 1
            if x == y:
                result.append(x)

    return ops, 0

def common_fast(a, b):
    ops = 0
    seen = set()
    for x in a:
        ops += 1
        seen.add(x)
    result = []
    for y in b:
        ops += 1
        if y in seen:
            result.append(y)
    return ops, len(seen)

n = 100
a = list(range(n))
b = list(range(n, 2 * n))
ops_slow, mem_slow = common_slow(a, b)
ops_fast, mem_fast = common_fast(a, b)
print("slow: ops =", ops_slow, " extra_mem =", mem_slow)
print("fast: ops =", ops_fast, " extra_mem =", mem_fast)`}

На непересекающихся списках медленная версия делает полный квадрат сравнений и не тратит памяти; быстрая - линейна по времени, но держит `set` из `n` элементов. Удвоение `n` разводит время вчетверо у одной и вдвое у другой, а память быстрой версии растёт линейно. Один и тот же ответ, разная цена по двум осям сразу.

## Амортизированная сложность: усредняем по серии

Некоторые операции стоят то дёшево, то дорого, и анализировать их по одной нечестно. Самый известный пример - `list.append`. Обычно он мгновенный: положить элемент в свободную ячейку - это `O(1)`. Но когда выделенное под список место заканчивается, Python выделяет блок побольше и **копирует весь список** в новое место - а это `O(n)`. Получается, что отдельный `append` стоит то `O(1)`, то вдруг `O(n)`.

Амортизированная сложность отвечает на вопрос «сколько стоит операция **в среднем по длинной серии**». Список растёт с запасом - примерно в полтора-два раза за раз - поэтому дорогие копирования случаются всё реже по мере роста. Если размазать стоимость всех копирований по всем `append`, на каждую операцию приходится константа. Поэтому `append` называют амортизированно `O(1)`: не каждый вызов дёшев, но `n` вызовов суммарно стоят `O(n)`, то есть `O(1)` на штуку.

Пики на графике - это моменты копирования, и они становятся всё реже: между ними помещается всё больше дешёвых вставок. Сумма всех столбцов для `n` операций остаётся порядка `n`, поэтому средняя высота - константа. В этом и смысл амортизации: редкая дорогая операция «оплачивается» множеством предшествующих дешёвых.

<MarginNote label="Не путать со средним случаем">Амортизированная сложность - это гарантия на **любую** серию операций, а не вероятностное среднее. Средний случай зависит от распределения данных; амортизация - от структуры серии и верна всегда.</MarginNote>

## Краевой случай, который вас удивит

Амортизированный `O(1)` не значит, что каждый отдельный вызов быстрый. Если Вы строите список из миллиона элементов и Вам критична задержка **каждой** операции (например, в системе реального времени), редкие `append` всё равно займут `O(n)` в момент копирования - и эта пауза может нарушить дедлайн. Амортизация обещает хорошую **суммарную** стоимость, но молчит о пиках.

И обратный сюрприз: `list.insert(0, x)` - вставка в начало - **не** амортизированно `O(1)`, а честно `O(n)` каждый раз, потому что сдвигает все элементы. Похожий на `append` вызов оказывается принципиально дороже. Когда нужны частые вставки в начало, берут `collections.deque`, у которого оба конца амортизированно `O(1)`. Мораль: имя метода ничего не гарантирует - цену надо знать для каждой операции отдельно.

## Типичные ошибки

- **Забывать про память входа против дополнительной памяти.** Пространственная сложность считает то, что алгоритм заводит **сверх** входа. Функция, возвращающая новый список длины `n`, тратит `O(n)` доппамяти; функция с одним аккумулятором - `O(1)`.
- **Не замечать стек рекурсии.** Рекурсия глубиной `n` занимает `O(n)` памяти на стеке, даже без единого созданного объекта. Это частая причина `RecursionError` на больших входах.
- **Считать `insert(0, x)` таким же дешёвым, как `append`.** `append` амортизированно `O(1)`, но вставка в начало списка - `O(n)`: сдвигаются все элементы. Для очереди с двух концов нужен `deque`.

<Detail summary="Откуда берётся коэффициент роста списка">

Когда место в списке кончается, CPython выделяет новый блок по формуле примерно «текущий размер плюс четверть» (точная формула менялась между версиями). Коэффициент роста больше единицы - ключ к амортизированному `O(1)`: при росте в фиксированное число раз суммарная стоимость всех копирований образует геометрическую прогрессию, которая сходится к `O(n)`, а не к `O(n²)`. Если бы список рос на постоянное число ячеек, а не в разы, копирования случались бы слишком часто, и `append` стал бы амортизированно `O(n)`. Геометрический рост - это то, что делает динамический массив практичным.

</Detail>
