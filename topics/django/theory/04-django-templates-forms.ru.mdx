# Где данные превращаются в HTML, а ввод - в проверенные данные

View собрал данные и теперь должен показать их человеку, а потом принять от него ввод обратно. Две стороны одной медали: на выходе нужно вставить значения в HTML, не дав им сломать вёрстку и не пустив чужой скрипт; на входе - принять то, что человек напечатал в поля, и убедиться, что это вообще похоже на правду. Django закрывает обе стороны: шаблоны рисуют страницу, формы принимают и проверяют ввод.

Соблазн велик - собирать HTML строками прямо во view и проверять поля вручную через россыпь `if`. Так пишут ровно до первой строки от пользователя, в которой окажется `<script>` или пустота там, где ждали число. Шаблонный язык намеренно отделяет вывод от логики, а формы превращают разрозненные проверки в один предсказуемый проход с понятными ошибками. К концу раздела Вы будете понимать язык шаблонов и наследование, разницу между связанной и несвязанной формой, как идёт валидация и зачем в каждой форме нужен CSRF-токен.

## Язык шаблонов: вывод, а не программирование

Шаблон Django - это HTML, в который вкраплены три вещи. Переменные подставляют значение из контекста и пишутся в двойных фигурных скобках. Теги управляют структурой (циклы, условия, наследование) и пишутся как ключевое слово в фигурных скобках со знаками процента. Фильтры преобразуют значение перед выводом и пишутся через вертикальную черту после переменной.

```django
<h1>{{ order.customer }}</h1>
<p>Сумма: {{ order.amount|floatformat:2 }} руб.</p>
{% if order.is_paid %}
  <span>оплачен</span>
{% endif %}
<ul>
{% for item in order.items %}
  <li>{{ item.title }}</li>
{% endfor %}
</ul>
```

Здесь переменная `order.customer` берётся из **контекста** - словаря, который view передаёт шаблону. Запись через точку умна: Django пробует и ключ словаря, и атрибут, и даже вызов метода без аргументов. Фильтр `floatformat:2` округляет до двух знаков. Тег `if` и тег `for` управляют тем, что и сколько раз вывести.

Важное ограничение: язык шаблонов **намеренно беден**. В нём нельзя присвоить переменную как в Python, нельзя вызвать функцию с произвольными аргументами, нельзя написать сложное выражение. Это не недоработка, а граница: вычисления - дело view, шаблон только показывает готовое. Когда Вам «не хватает» возможностей в шаблоне, это почти всегда сигнал перенести логику во view.

## Наследование шаблонов: общий каркас и сменные блоки

Страницы сайта похожи: общая шапка, подвал, меню. Дублировать их в каждом шаблоне - путь к рассинхрону. Django решает это наследованием. Базовый шаблон задаёт каркас и объявляет **блоки** - дырки, которые потомки заполняют своим содержимым.

```django
<!DOCTYPE html>
<html>
<body>
  <header>Магазин</header>
  <main>{% block content %}{% endblock %}</main>
  <footer>2024</footer>
</body>
</html>
```

Дочерний шаблон объявляет, от какого базового он наследуется, и переопределяет нужные блоки. Всё, что вне блоков, он берёт у родителя как есть.

```django
{% extends "base.html" %}

{% block content %}
  <h1>Заказ {{ order.pk }}</h1>
{% endblock %}
```

## Форма: объект, который знает свои поля и умеет себя проверять

Теперь обратная сторона - приём ввода. `Form` - это класс, где каждый атрибут описывает поле ввода с его типом и правилами. Форма умеет три вещи: нарисовать себя в HTML, принять присланные данные и проверить их.

```python
from django import forms

class OrderForm(forms.Form):
    customer = forms.CharField(max_length=100)
    amount = forms.DecimalField(min_value=0)
    comment = forms.CharField(required=False)
```

Когда форма привязана к модели почти один в один, писать поля вручную незачем - есть `ModelForm`. Он строит поля автоматически из модели, остаётся указать модель и список полей.

```python
from django import forms
from .models import Order

class OrderModelForm(forms.ModelForm):
    class Meta:
        model = Order
        fields = ["customer", "amount"]
```

## Связанная и несвязанная форма

Различие, которое путают чаще всего. Форма, созданную **без данных** (`OrderForm()`), - **несвязанная** (unbound): ей нечего проверять, она годится только чтобы нарисовать пустые поля. Форма, созданная **с данными** (`OrderForm(request.POST)`), - **связанная** (bound): в ней лежат присланные значения, её можно валидировать.

Типичный view с формой объединяет оба случая: на GET показывает пустую форму, на POST - связывает её с данными и проверяет.

```python
def create_order(request):
    if request.method == "POST":
        form = OrderForm(request.POST)
        if form.is_valid():
            data = form.cleaned_data
            return HttpResponse("создано", status=201)
    else:
        form = OrderForm()
    return render(request, "order_form.html", {"form": form})
```

## Валидация: is_valid, cleaned_data, clean-методы

Сердце формы - проверка. Вызов `form.is_valid()` запускает валидацию и возвращает `True`/`False`. После него появляются два результата: словарь `form.cleaned_data` с приведёнными к нужным типам и проверенными значениями (но только если валидация прошла) и `form.errors` с сообщениями об ошибках по полям.

Под капотом `is_valid()` выполняет каскад. Сначала каждое поле приводит ввод к своему типу и проверяет базовые правила (`required`, `max_length`, `min_value`). Затем, если у формы есть метод `clean_<имя поля>`, вызывается он - место для проверки конкретного поля по своей логике. Наконец, метод `clean` (без суффикса) видит уже все поля сразу - туда кладут проверки, которым нужно несколько полей.

```python
class OrderForm(forms.Form):
    customer = forms.CharField(max_length=100)
    amount = forms.DecimalField(min_value=0)

    def clean_customer(self):
        name = self.cleaned_data["customer"]
        if name.strip().lower() == "test":
            raise forms.ValidationError("Имя 'test' недопустимо")
        return name
```

Чтобы сообщить о невалидном значении, в clean-методе поднимают `ValidationError` - Django поймает её и положит текст в `form.errors`, а форма станет невалидной. Метод `clean_<поле>` обязан **вернуть** значение (часто из `cleaned_data`), иначе поле пропадёт из очищенных данных.

<Callout type="warning">
Читать `form.cleaned_data` до успешного `is_valid()` нельзя: до проверки очищенных данных просто нет. Сначала всегда `if form.is_valid():`, и только внутри этой ветки - `form.cleaned_data`.
</Callout>

## CSRF-токен: зачем он в каждой форме

Каждая форма, отправляемая методом POST, должна нести **CSRF-токен** - скрытое поле с уникальным значением. Защищает он от атаки «межсайтовая подделка запроса»: чужой сайт может заставить браузер пользователя отправить POST на Ваш сайт (браузер приложит куки автоматически), но не может узнать секретный токен, привязанный к сессии. Django проверяет токен и отвергает POST без него с кодом `403`.

В шаблоне токен вставляют одним тегом внутри формы.

```django
<form method="post">
  {% csrf_token %}
  {{ form.as_p }}
  <button type="submit">Отправить</button>
</form>
```

Здесь же видно, как форма рисует себя: `form.as_p` выводит поля, обёрнутые в абзацы. Есть и другие варианты вывода, но суть одна - HTML полей форма генерирует сама.

## Краевые случаи

**Несвязанную форму нельзя провалидировать осмысленно.** `OrderForm().is_valid()` вернёт `False`, но не потому, что данные плохие, - их просто нет. Проверяйте только связанные формы (созданные с данными).

**`clean_<поле>` без `return` молча теряет значение.** Если метод проверил поле, но забыл вернуть значение, оно исчезнет из `cleaned_data`, даже когда ошибки не было. Метод обязан вернуть очищенное значение.

**`required=False` и пустой ввод - не ошибка.** Поле с `required=False`, оставленное пустым, проходит валидацию; в `cleaned_data` оно попадёт пустой строкой (или `None`). Не путайте «поле необязательное» с «поле невалидное».

## Типичные ошибки

- **Собирать HTML строками во view.** Конкатенация тегов в Python обходит автоэкранирование и открывает XSS: ввод пользователя со скриптом попадёт на страницу как есть. Рисуйте через шаблон - он экранирует подставляемые значения по умолчанию.
- **Логика в шаблоне.** Шаблонный язык беден намеренно. Если в шаблоне хочется присвоения или сложного выражения, перенесите вычисление во view, а в контекст положите готовый результат.
- **Забыть `{% csrf_token %}` в POST-форме.** Тогда Django отвергнет отправку с `403`. Тег обязателен внутри каждой формы, отправляемой методом POST.
- **Читать `cleaned_data` без `is_valid()`.** До успешной проверки очищенных данных нет; обращение к ним вне ветки `if form.is_valid()` - ошибка.

<Detail summary="Почему автоэкранирование шаблонов - это про безопасность">

По умолчанию шаблонизатор Django экранирует подставляемые значения: символы `<`, `>`, `&`, кавычки превращаются в HTML-сущности. Поэтому если пользователь ввёл в имя `<script>alert(1)</script>`, на странице он увидит этот текст буквально, а браузер не выполнит его как скрипт. Это защита от XSS «по умолчанию безопасно»: чтобы вывести значение как сырой HTML, нужно явно попросить (фильтром `safe`), и тогда ответственность за безопасность - на Вас. Ровно поэтому сборка HTML строками во view опасна: она проходит мимо этого механизма.

</Detail>
