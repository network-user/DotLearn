# Как Django узнаёт, кому отдать запрос

Запрос приходит строкой `GET /orders/42/`, и где-то должна состояться встреча: этот адрес обязан найти именно ту функцию, что умеет показать заказ номер 42. Никакой магии тут нет - есть список правил, который Вы сами и написали, и есть механизм, который пробегает этот список сверху вниз и берёт первое подошедшее правило. Понять маршрутизацию - значит понять две вещи: как устроен этот список и что попадает в Вашу функцию, когда правило сработало.

Проблема, которую решает URLconf, - развязать адреса и код. Браузер ничего не знает про Ваши функции; он знает только URL. Вам же удобно думать функциями. Между ними нужен переводчик: таблица «такой-то адрес ведёт к такому-то обработчику», да ещё умеющая выдёргивать из адреса переменные части - идентификатор заказа, слаг статьи. К концу раздела Вы будете уверенно читать `urls.py`, понимать конвертеры путей, различать function-based и class-based views и знать, почему порядок правил в списке решает всё.

## Ментальная модель: URLconf - это список правил, проверяемых по порядку

Маршрутизация в Django - это `urlpatterns`, обычный список. Каждый его элемент - пара «шаблон пути → обработчик». Когда приходит запрос, resolver берёт путь (часть после домена) и идёт по списку **сверху вниз**, примеряя путь к каждому шаблону. Первое совпадение выигрывает: его обработчик вызывается, дальше resolver не смотрит. Если не подошёл ни один - Django отдаёт 404.

```python
from django.urls import path
from . import views

urlpatterns = [
    path("orders/", views.order_list),
    path("orders/<int:pk>/", views.order_detail),
]
```

Главное в этой картинке: список упорядочен, а проверка - не «найти лучшее совпадение», а «взять первое подошедшее». Отсюда практическое правило: более конкретные шаблоны ставят выше более общих. Если шаблон `orders/<int:pk>/` окажется ниже шаблона `orders/<path:rest>/`, который ловит вообще всё после `orders/`, до конкретного дело никогда не дойдёт.

## Конвертеры: как из пути достаются переменные части

В шаблоне `orders/<int:pk>/` угловые скобки - это **захват**. Запись `<int:pk>` значит: «здесь стоит целое число, передай его во view под именем `pk`». Часть до двоеточия - **конвертер** (тип), часть после - имя аргумента.

```python
path("orders/<int:pk>/", views.order_detail)
path("articles/<slug:slug>/", views.article_detail)
path("users/<str:username>/", views.profile)
```

Конвертер не только вытаскивает значение, но и **проверяет** его форму и приводит тип. `<int:pk>` совпадёт только с цифрами и отдаст во view настоящий `int`, а не строку. `<slug:...>` примет буквы, цифры, дефисы и подчёркивания (то, из чего состоят «человекочитаемые» адреса вроде `kak-rabotaet-orm`). `<str:...>` - любая непустая строка без слэша. Есть и `<uuid:...>`, `<path:...>` (последний разрешает слэши и потому ловит остаток пути).

Когда стандартных конвертеров не хватает, есть `re_path` - маршрут на регулярном выражении. Он мощнее, но требовательнее к глазам: захват идёт через именованные группы.

```python
from django.urls import re_path

re_path(r"^orders/(?P<year>[0-9]{4})/$", views.orders_by_year)
```

<Callout type="tip">
Начинайте с `path` и конвертеров - они покрывают почти всё и читаются как обычный текст. Переходите на `re_path` только когда нужно то, чего конвертеры не умеют: фиксированная длина, перечисление вариантов, сложный формат.
</Callout>

## Function-based view: запрос на входе, ответ на выходе

Самый прямой вид обработчика - функция. Её контракт прост: первым аргументом она получает `HttpRequest`, дальше - захваченные из URL части, а вернуть обязана `HttpResponse` (или его подкласс).

```python
from django.http import HttpResponse
from django.shortcuts import get_object_or_404
from .models import Order

def order_detail(request, pk):
    order = get_object_or_404(Order, pk=pk)
    return HttpResponse(f"Заказ {order.pk} на сумму {order.amount}")
```

Объект `request` несёт всё про запрос: `request.method` - метод (`"GET"`, `"POST"`), `request.GET` - параметры из строки запроса (`?page=2`), `request.POST` - данные отправленной формы, `request.user` - текущий пользователь. Заметьте `get_object_or_404`: это удобный помощник, который делает `get(...)` и, если объект не найден, сам поднимает 404 вместо `DoesNotExist` - ровно тот случай из прошлого раздела, когда `get` мог упасть.

Ветвление по методу - частый приём: GET показывает, POST обрабатывает отправку.

```python
def create_order(request):
    if request.method == "POST":
        return HttpResponse("создано", status=201)
    return HttpResponse("форма")
```

## Class-based view: тот же контракт, но методами

Когда логики становится больше, функцию удобно разложить на методы класса. **Class-based view** (CBV) наследуется от `View`; вместо ветвления по `request.method` Вы пишете отдельные методы `get`, `post`, и Django сам зовёт нужный по методу запроса.

```python
from django.views import View
from django.http import HttpResponse

class OrderView(View):
    def get(self, request, pk):
        return HttpResponse(f"показ заказа {pk}")

    def post(self, request, pk):
        return HttpResponse("обновлён", status=200)
```

В `urls.py` класс подключают через `.as_view()` - он возвращает функцию, которую ждёт `urlpatterns`.

```python
path("orders/<int:pk>/", OrderView.as_view())
```

Поверх `View` Django даёт **дженерики** - готовые CBV под типовые задачи. `ListView` показывает список объектов модели, `DetailView` - один объект по первичному ключу. Вы задаёте модель и шаблон, а перебор и выборку дженерик берёт на себя.

```python
from django.views.generic import ListView, DetailView
from .models import Order

class OrderList(ListView):
    model = Order

class OrderDetail(DetailView):
    model = Order
```

## HttpResponse и его родня: коды статусов

View возвращает ответ, и у ответа есть код статуса - число, которым сервер сообщает об исходе. `HttpResponse(...)` по умолчанию ставит `200` (успех). Для API удобен `JsonResponse(...)`: он сериализует словарь в JSON и проставляет нужный заголовок.

```python
from django.http import JsonResponse

def order_json(request, pk):
    return JsonResponse({"pk": pk, "status": "paid"})
```

Коды стоит знать хотя бы в общих чертах: `200` - успех, `201` - создано (после удачного POST), `301`/`302` - редирект (постоянный/временный), `400` - ошибка в запросе клиента, `403` - доступ запрещён, `404` - не найдено, `500` - ошибка на сервере. Когда объекта нет, корректно вернуть `404`, а не `200` с пустым телом - иначе клиент решит, что всё хорошо.

## Краевые случаи

**Порядок правил решает всё.** Поскольку выигрывает первое совпадение, общий шаблон, поставленный выше конкретного, перехватит запрос раньше. Если `orders/new/` стоит **ниже** `orders/<str:pk>/`, то по адресу `/orders/new/` сработает второй (он примет `new` как `pk`), и до страницы создания запрос не дойдёт. Конкретное - выше общего.

**Конвертер `int` не примет отрицательное и не примет буквы.** `<int:pk>` совпадает только с последовательностью цифр. Адрес `/orders/-5/` или `/orders/abc/` под этот шаблон не подойдёт - resolver просто пойдёт искать дальше по списку и, не найдя, отдаст 404. Это удобно: непохожие на id адреса отсекаются ещё до view.

**View обязан вернуть `HttpResponse`.** Если функция отработала и ничего не вернула (вернула `None`), Django поднимет ошибку. Каждая ветка кода во view должна заканчиваться возвратом ответа - включая ту, про которую легко забыть (например, «а что, если метод не GET и не POST»).

## Типичные ошибки

- **Общий маршрут выше конкретного.** Шаблон, ловящий многое (`<str:...>`, `<path:...>`), поставленный раньше точного, перехватывает запросы. Ставьте конкретные пути выше общих.
- **Возвращать данные, а не `HttpResponse`.** `return {"pk": 1}` из view - ошибка: словарь не ответ. Заверните его в `JsonResponse`, а HTML - в `HttpResponse` или `render(...)`.
- **`200` вместо `404` для отсутствующего объекта.** Если заказа нет, отдавать пустую страницу со `200` - значит врать клиенту. Используйте `get_object_or_404`, он поднимет корректный `404`.
- **Путать `request.GET` и `request.POST`.** Параметры из адресной строки (`?q=...`) лежат в `request.GET` при любом методе; тело отправленной формы - в `request.POST`. Искать данные формы в `request.GET` - частая причина «почему ничего не пришло».

<Detail summary="Что на самом деле возвращает as_view()">

`OrderView.as_view()` не отдаёт сам класс и не отдаёт экземпляр - он возвращает **функцию**. Эта функция при каждом запросе создаёт свежий экземпляр view, складывает в него `request` и захваченные аргументы, а затем вызывает метод `dispatch`, который по `request.method` выбирает `get`, `post` и так далее. Поэтому `urlpatterns` всегда видит на месте обработчика «что-то вызываемое как функция», независимо от того, написали Вы функцию или класс. И поэтому состояние одного запроса не протекает в другой: экземпляр у каждого запроса свой.

</Detail>
