# Готовая админка и встроенные пользователи, которых не пришлось писать

Стоит зарегистрировать модель в одном файле - и Django поднимает для неё полноценный интерфейс управления: списки с поиском и фильтрами, формы создания и редактирования, удаление, всё с проверкой прав. Рядом из коробки лежит система пользователей: модель `User`, проверка пароля, вход, выход, понятие «текущий пользователь запроса» и механизм, отделяющий «кто ты» от «что тебе можно». Это и есть «батарейки» Django в самом наглядном виде - функциональность, на которую в другом фреймворке ушли бы дни.

Соблазн новичка - написать своё: свою таблицу пользователей, своё хранение паролей, свою страницу «admin». Почти всегда это хуже готового: пароли легко начать хранить небезопасно, в проверке прав легко ошибиться, а самописная админка отнимает время у настоящих задач. К концу раздела Вы будете понимать, как устроена админка и как её настроить под модель, как работает встроенная аутентификация (вход, выход, текущий пользователь), чем аутентификация отличается от авторизации и как закрыть страницу от посторонних.

## Admin: интерфейс из одной строки регистрации

Чтобы модель появилась в админке, её регистрируют. В простейшем виде - одной строкой.

```python
from django.contrib import admin
from .models import Order

admin.site.register(Order)
```

Этого уже хватает, чтобы по адресу `/admin/` увидеть раздел заказов: список, добавление, редактирование, удаление. Но список по умолчанию беден - одна колонка со строковым представлением объекта. Настраивают админку через класс `ModelAdmin`: в нём объявляют, как показывать модель.

```python
from django.contrib import admin
from .models import Order

@admin.register(Order)
class OrderAdmin(admin.ModelAdmin):
    list_display = ["customer", "amount", "status", "created"]
    search_fields = ["customer"]
    list_filter = ["status"]
```

Три самые ходовые настройки: `list_display` - какие колонки показать в списке (вместо единственной); `search_fields` - по каким полям работает строка поиска сверху; `list_filter` - какие поля дают боковую панель быстрых фильтров. Декоратор `@admin.register(Order)` - короткая запись вместо отдельного вызова `admin.site.register`.

<Callout type="info">
Админка - инструмент для сотрудников (staff), а не публичная часть сайта. Это не замена обычным страницам и формам для пользователей: за ней удобно вести данные команде, но показывать её клиентам не стоит.
</Callout>

## Аутентификация: модель User и вход

Встроенная модель `User` хранит учётную запись: имя пользователя, пароль (всегда в виде хеша, не в открытом виде), почту, флаги. Три важных флага: `is_active` (учётка действует), `is_staff` (пускают в админку), `is_superuser` (можно всё). Суперпользователя заводят командой `python manage.py createsuperuser`.

Вход - это два шага. Сначала **проверка** пары «логин-пароль» функцией `authenticate`: она возвращает объект `User`, если данные верны, и `None`, если нет. Затем **установка сессии** функцией `login`, которая запоминает в сессии, что этот запрос (и последующие) принадлежат данному пользователю.

```python
from django.contrib.auth import authenticate, login, logout

def sign_in(request):
    user = authenticate(
        request,
        username=request.POST["username"],
        password=request.POST["password"],
    )
    if user is not None:
        login(request, user)
        return HttpResponse("вошли")
    return HttpResponse("неверные данные", status=401)
```

Выход - `logout(request)`: он очищает сессию. После входа в каждом следующем запросе доступен `request.user` - текущий пользователь. У него есть свойство `is_authenticated`: `True` для вошедшего пользователя и `False` для анонимного (гостя). Именно это свойство проверяют, чтобы понять, вошёл человек или нет.

<Callout type="warning">
`request.user.is_authenticated` - это **свойство-флаг**, а не вызов проверки логина. Анонимный посетитель тоже имеет `request.user` (специальный `AnonymousUser`), просто у него `is_authenticated` равно `False`. Не путайте «есть ли request.user» (он есть всегда) с «вошёл ли пользователь» (это и есть `is_authenticated`).
</Callout>

## Закрыть страницу: login_required и PermissionRequiredMixin

Чаще всего нужно простое: пустить на страницу только вошедших. Для function-based view есть декоратор `login_required`. Он проверяет `request.user.is_authenticated`, и если пользователь анонимный - перенаправляет на страницу входа, не вызывая view.

```python
from django.contrib.auth.decorators import login_required

@login_required
def my_orders(request):
    return HttpResponse(f"заказы пользователя {request.user}")
```

Для class-based view ту же роль играют миксины. `LoginRequiredMixin` пускает только вошедших; `PermissionRequiredMixin` идёт дальше - требует конкретного **права** (а не просто факта входа).

```python
from django.contrib.auth.mixins import PermissionRequiredMixin
from django.views.generic import ListView
from .models import Order

class OrderList(PermissionRequiredMixin, ListView):
    model = Order
    permission_required = "shop.view_order"
```

Здесь `"shop.view_order"` - это **право** в формате `приложение.действие_модель`. Django автоматически создаёт по четыре права на каждую модель (`add`, `change`, `delete`, `view`). Права удобно раздавать не по одному, а **группами**: группа - именованный набор прав (например, «менеджеры»), пользователя добавляют в группу, и он получает все её права разом. Это масштабируется лучше, чем назначать права каждому лично.

## Сессии: как сервер помнит вошедшего

`login` и `request.user` работают благодаря **сессиям**. Сессия - это серверное хранилище данных, привязанное к конкретному посетителю. При входе Django кладёт в сессию идентификатор пользователя, а в браузер отдаёт куку с ключом сессии. В каждом следующем запросе браузер присылает эту куку, middleware по ключу находит сессию, достаёт идентификатор и наполняет `request.user`. Пароль при этом нигде не «возится» туда-сюда - ходит только непредсказуемый ключ сессии. `logout` удаляет сессию, и кука перестаёт что-либо значить.

## Краевые случаи

**`authenticate` возвращает `None`, а не бросает исключение.** Неверный пароль - это не ошибка уровня исключения, а штатный «не тот». Проверяйте результат на `None` явно; нельзя сразу передавать его в `login`, не убедившись, что пользователь найден.

**Вход не равно «всё можно».** `is_authenticated` лишь говорит, что личность установлена. Доступ к конкретному действию - отдельная проверка права. `login_required` пустит любого вошедшего, даже без нужных прав; чтобы требовать право, нужен `PermissionRequiredMixin` или проверка `request.user.has_perm(...)`.

**`is_staff` и `is_superuser` - разные вещи.** `is_staff` лишь пускает в админку, но видеть там пользователь будет только то, на что у него есть права. `is_superuser` обходит проверки прав целиком. Дать `is_staff` без прав - значит пустить в админку с пустыми руками.

## Типичные ошибки

- **Писать свою таблицу пользователей и хранение паролей.** Это легко сделать небезопасно (пароли в открытом виде, слабое хеширование). Берите встроенную модель `User` и `authenticate`/`login` - там пароли хешируются правильно.
- **Путать аутентификацию и авторизацию.** «Пользователь вошёл» не означает «пользователю всё можно». Вход - это аутентификация; доступ к действию - авторизация через права и группы.
- **Проверять `request.user` на истинность вместо `is_authenticated`.** `request.user` есть всегда (для гостя это `AnonymousUser`, который истинен как объект). Признак входа - именно `request.user.is_authenticated`.
- **Назначать права каждому пользователю вручную.** На паре учёток сойдёт, но быстро становится неуправляемым. Группируйте права в группы и добавляйте пользователей в группы.

<Detail summary="Почему пароль хранят как хеш, а не как текст">

Django никогда не хранит пароль в открытом виде - только его хеш, полученный медленной криптографической функцией с солью. Смысл в том, что из хеша нельзя восстановить пароль, а соль не даёт заранее посчитать таблицу соответствий. При входе Django хеширует введённый пароль тем же способом и сравнивает хеши, а не пароли. Поэтому даже при утечке базы исходные пароли не раскрываются мгновенно, и поэтому `authenticate` сравнивает хеши за Вас - писать это вручную не нужно и опасно. Самописное хранение паролей - классический способ завести дыру в безопасности на ровном месте.

</Detail>
