# Упорядочивание и ПЕРВЫЕ

«Покажи десять самых крупных клиентов» - вопрос, в котором спрятаны две операции. Сначала клиентов нужно выстроить по убыванию выручки, потом отрезать первую десятку. Ни то, ни другое запрос не делает по умолчанию: без явного указания порядок строк в результате не определён, а «первые» без сортировки - это первые попавшиеся, а не лучшие. Два раздела - `УПОРЯДОЧИТЬ ПО` и `ПЕРВЫЕ` - закрывают эту задачу, и работают они только вместе.

К концу урока вы будете сортировать результат по одному и нескольким полям, брать верхушку выборки и понимать, почему `ПЕРВЫЕ` без `УПОРЯДОЧИТЬ ПО` - почти всегда ошибка.

## Порядок не гарантирован сам по себе

Результат запроса - это множество строк, и без явной сортировки система вправе вернуть их в любом порядке: в каком удобнее СУБД, в каком лежат в индексе, в каком быстрее. Сегодня строки идут по возрастанию кода, завтра после переиндексации - иначе. Полагаться на «естественный» порядок нельзя. Как только порядок важен, его задают разделом `УПОРЯДОЧИТЬ ПО`.

```1c
ВЫБРАТЬ
    Клиенты.Наименование КАК Клиент,
    Клиенты.Выручка КАК Выручка
ИЗ
    РегистрНакопления.ПродажиОбороты КАК Клиенты
УПОРЯДОЧИТЬ ПО
    Выручка УБЫВ
```

`УБЫВ` сортирует по убыванию, `ВОЗР` - по возрастанию; без явного указания подразумевается `ВОЗР`. Сортировать можно по нескольким полям: первое поле задаёт основной порядок, второе разрешает ничьи внутри одинаковых значений первого.

```1c
УПОРЯДОЧИТЬ ПО
    Клиенты.Город ВОЗР,
    Выручка УБЫВ
```

Сначала клиенты группируются по городу в алфавитном порядке, а внутри каждого города идут от самого крупного к самому мелкому. `УПОРЯДОЧИТЬ ПО` стоит последним в конвейере, поэтому в нём уже видны псевдонимы из `ВЫБРАТЬ`: `Выручка` здесь - это имя колонки результата.

## ПЕРВЫЕ: отрезать верхушку

`ПЕРВЫЕ N` ставится сразу после `ВЫБРАТЬ` и оставляет только первые `N` строк результата. В связке с сортировкой это и есть «топ-N».

```1c
ВЫБРАТЬ ПЕРВЫЕ 3
    Клиенты.Наименование КАК Клиент,
    Клиенты.Выручка КАК Выручка
ИЗ
    РегистрНакопления.ПродажиОбороты КАК Клиенты
УПОРЯДОЧИТЬ ПО
    Выручка УБЫВ
```

Сначала клиенты выстраиваются по убыванию выручки, потом `ПЕРВЫЕ 3` отрезает тройку лидеров. Порядок этих двух действий - сортировка, затем отсечение - и даёт правильную верхушку.

## Почему ПЕРВЫЕ без УПОРЯДОЧИТЬ - ловушка

Если `ПЕРВЫЕ 3` написать без `УПОРЯДОЧИТЬ ПО`, система честно вернёт три строки - но какие именно, не определено. Это будут первые попавшиеся строки в том порядке, в каком их отдала СУБД, а не три лучших. Запрос выглядит работающим, на маленькой базе даже может случайно давать правдоподобный результат, а на реальных данных молча покажет не тех клиентов.

<Callout type="warning" title="Топ-N - это всегда пара">
`ПЕРВЫЕ` отвечает на вопрос «сколько строк взять», а `УПОРЯДОЧИТЬ ПО` - «какие именно». Без сортировки «первые» не значит «крупнейшие»; это значит «те, что оказались сверху». Если в задаче есть слово «самые» или «топ», в запросе обязаны быть оба раздела.
</Callout>

## Порядок по ссылочным полям

Сортировать можно и по ссылочному полю - например, по номенклатуре. Но упорядочивание по ссылке идёт не по наименованию, а по внутреннему представлению ссылки, и в отчёте это выглядит как случайный порядок. Если нужно упорядочить по названию, сортируйте явно по реквизиту: `УПОРЯДОЧИТЬ ПО Продажи.Номенклатура.Наименование`.

<Callout type="tip" title="АВТОУПОРЯДОЧИВАНИЕ">
Ключевое слово `АВТОУПОРЯДОЧИВАНИЕ` в конце запроса просит платформу упорядочить результат «по-человечески»: ссылочные поля - по их представлению, иерархические справочники - с учётом иерархии. Удобно для быстрых списков, но на больших выборках заметно дороже явной сортировки по конкретному полю, поэтому в нагруженных отчётах его избегают.
</Callout>

## Типичные ошибки

**ПЕРВЫЕ без УПОРЯДОЧИТЬ ПО.** Классика «топа не из тех». Всякий раз, когда берёте верхушку, спрашивайте себя: по какому полю она верхушка? Ответ должен быть в `УПОРЯДОЧИТЬ ПО`.

**Сортировка по ссылке вместо представления.** `УПОРЯДОЧИТЬ ПО Продажи.Номенклатура` даёт порядок по внутренней ссылке, а не по алфавиту. Хотите алфавит - сортируйте по `.Наименование`.

**Ожидать стабильного порядка без сортировки.** «У меня же и так всё по коду» - до первой переиндексации. Порядок без `УПОРЯДОЧИТЬ ПО` не гарантирован никогда.

<Detail summary="ПЕРВЫЕ и ничьи на границе">

`ПЕРВЫЕ N` отрезает ровно `N` строк, даже если на границе есть строки с одинаковым значением сортировки. Если у третьего и четвёртого клиента выручка совпадает, `ПЕРВЫЕ 3` возьмёт кого-то одного из них - какого именно, зависит от вторичного порядка. Когда важно не разрывать группы с равными значениями, добавляют дополнительное поле в `УПОРЯДОЧИТЬ ПО`, чтобы сделать порядок однозначным, либо решают задачу иначе - например, через максимум и отбор. Строгого аналога SQL-ного `WITH TIES` в языке запросов 1С нет.

</Detail>
