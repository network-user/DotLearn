# Параметры запроса и условия

Отчёт «продажи за период» не может знать заранее, какой именно период попросит пользователь. Даты приходят снаружи, во время выполнения. Соблазн - собрать текст запроса строкой, подставив даты через склейку. Так делать нельзя: склейка ломается на кавычках, путается в форматах дат и открывает дорогу подстановке неожиданных значений. Правильный способ передать значение в запрос - параметр. А чтобы условия отбора были точными, язык запросов даёт богатый набор операторов: диапазоны, списки, шаблоны, ветвление.

К концу урока вы будете передавать в запрос параметры, строить условия с `МЕЖДУ`, `В`, `ПОДОБНО`, обрабатывать `NULL` и ветвить значения через `ВЫБОР`.

## Параметр вместо склейки

Параметр в тексте запроса записывается с амперсандом: `&Начало`, `&Склад`, `&Список`. Это именованное место, куда значение подставляется отдельно от текста, уже готовым и типизированным.

```1c
ВЫБРАТЬ
    Продажи.Номенклатура КАК Товар,
    Продажи.Количество КАК Количество
ИЗ
    Документ.РеализацияТоваровУслуг.Товары КАК Продажи
ГДЕ
    Продажи.Ссылка.Дата МЕЖДУ &Начало И &Конец
    И Продажи.Склад = &Склад
```

Значения передаются во встроенном языке до выполнения запроса методом `УстановитьПараметр`:

```1c
Запрос.УстановитьПараметр("Начало", НачалоПериода);
Запрос.УстановитьПараметр("Конец", КонецПериода);
Запрос.УстановитьПараметр("Склад", ВыбранныйСклад);
```

Текст запроса при этом остаётся неизменным - меняются только значения. Это не только безопаснее склейки, но и быстрее: СУБД может переиспользовать уже построенный план для того же текста с другими параметрами.

<Callout type="warning" title="Не собирайте запрос строкой">
Склейка значений в текст запроса (`"... ГДЕ Склад = """ + Склад + """"`) - источник ошибок и уязвимостей. Кавычки в наименовании ломают синтаксис, даты зависят от локали, а произвольное значение может изменить смысл запроса. Всё, что приходит извне, передавайте параметром, а не текстом.
</Callout>

## Диапазоны, списки и шаблоны

`МЕЖДУ ... И ...` проверяет попадание в диапазон, границы включаются. `Дата МЕЖДУ &Начало И &Конец` пройдёт для дат от начала до конца включительно.

`В` проверяет вхождение в список значений. Список удобно передать одним параметром-массивом:

```1c
ГДЕ
    Продажи.Склад В (&СпискоСкладов)
```

Для иерархических справочников есть `В ИЕРАРХИИ`: `Номенклатура В ИЕРАРХИИ (&Группа)` отберёт товары не только самой группы, но и всех её подгрупп на любую глубину. Это частый отбор в отчётах по разделам каталога.

`ПОДОБНО` сравнивает строку с шаблоном: `%` заменяет любое число символов, `_` - ровно один. `Наименование ПОДОБНО "%насос%"` найдёт всё, где встречается слово «насос». Поиск по `ПОДОБНО` с `%` в начале шаблона не использует индекс и на больших справочниках медленный, но для интерактивного поиска по подстроке он незаменим.

## ВЫБОР: ветвление значения

Оператор `ВЫБОР КОГДА ... ТОГДА ... ИНАЧЕ ... КОНЕЦ` - это условное выражение прямо в запросе, аналог тернарного оператора. Он вычисляет значение по условию, не выходя из запроса.

```1c
ВЫБРАТЬ
    Продажи.Номенклатура КАК Товар,
    ВЫБОР
        КОГДА Продажи.Сумма >= 100000 ТОГДА "Крупная"
        КОГДА Продажи.Сумма >= 10000 ТОГДА "Средняя"
        ИНАЧЕ "Мелкая"
    КОНЕЦ КАК КатегорияПродажи
ИЗ
    Документ.РеализацияТоваровУслуг.Товары КАК Продажи
```

Ветки `КОГДА` проверяются сверху вниз, срабатывает первая истинная; `ИНАЧЕ` ловит всё остальное и может отсутствовать (тогда без совпадения получится `NULL`). `ВЫБОР` часто используют, чтобы разложить строки по категориям и потом сгруппировать по вычисленной категории.

## NULL в условиях

`NULL` в языке запросов - это не значение, а «неизвестно», и сравнивать его обычными операторами бесполезно: `NULL = NULL` не истинно, `NULL <> 5` тоже не истинно. Проверяют `NULL` отдельными конструкциями `ЕСТЬ NULL` и `ЕСТЬ НЕ NULL`.

```1c
ГДЕ
    Цены.Цена ЕСТЬ NULL
```

Такое условие отберёт строки, где цена не нашлась при соединении. А чтобы не проверять, а *заменить* `NULL` на конкретное значение, используют функцию `ЕСТЬNULL`:

```1c
ВЫБРАТЬ
    Продажи.Номенклатура КАК Товар,
    ЕСТЬNULL(Цены.Цена, 0) КАК Цена
ИЗ
    Документ.РеализацияТоваровУслуг.Товары КАК Продажи
    ЛЕВОЕ СОЕДИНЕНИЕ РегистрСведений.ЦеныНоменклатуры КАК Цены
    ПО Цены.Номенклатура = Продажи.Номенклатура
```

`ЕСТЬNULL(Цены.Цена, 0)` вернёт цену, если она есть, и ноль, если строки цен не нашлось. Это стандартный способ погасить `NULL` из внешнего соединения, чтобы он не отравлял арифметику и сортировку.

<Callout type="tip" title="ЕСТЬ NULL и ЕСТЬNULL - разные вещи">
`ЕСТЬ NULL` (два слова) - это проверка в условии: истина или ложь. `ЕСТЬNULL(выражение, значение)` (одно слово, функция) - это подстановка: вернуть значение вместо `NULL`. Их легко перепутать по названию, но задачи у них противоположные: одна спрашивает, другая заменяет.
</Callout>

## Типичные ошибки

**Склейка значений в текст запроса.** Даты, наименования, ссылки - всё это передают параметром `&Имя`, а не конкатенацией строк. Склейка ломается на кавычках и локали и небезопасна.

**Сравнивать NULL через равенство.** `ГДЕ Поле = NULL` не отберёт ничего: `NULL` ничему не равен, даже `NULL`. Проверяйте через `ЕСТЬ NULL`.

**Забыть границы у МЕЖДУ.** `МЕЖДУ` включает обе границы. Если конец периода задан как начало дня, продажи после полуночи последнего дня в диапазон не попадут - конец периода задают концом дня (`КонецДня`), иначе теряется целый день данных.

<Detail summary="Почему параметр в В (&Список) быстрее списка литералов">

Передать список одним параметром-массивом (`В (&Склады)`) не только чище, чем перечислять значения в тексте, но и лучше для СУБД. Текст запроса с параметром остаётся одним и тем же независимо от длины и содержимого списка, поэтому построенный план переиспользуется. А запрос, где значения вписаны прямо в текст (`В ("Основной", "Резервный")`), для каждого набора значений - это формально новый текст, под который СУБД может строить план заново. На системе с тысячами разных отборов это заметная разница в нагрузке.

</Detail>
