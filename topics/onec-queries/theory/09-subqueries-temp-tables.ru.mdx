# Вложенные запросы и временные таблицы

Настоящие отчёты редко умещаются в один плоский запрос. «Продажи по складам московского региона» - это уже два вопроса: сначала «какие склады в Москве», потом «сколько по ним продали». Один запрос можно вложить в другой, а можно разбить задачу на этапы, сохраняя промежуточный результат во временную таблицу. Оба приёма решают одну проблему - собрать сложный результат из простых шагов, - но делают это по-разному, и выбор между ними влияет и на читаемость, и на скорость.

К концу урока вы будете подставлять вложенный запрос как источник и как список, собирать пакетные запросы с временными таблицами и понимать, когда какой приём уместнее.

## Вложенный запрос как источник

Запрос можно поставить туда, где ожидается таблица, - прямо в раздел `ИЗ`, в круглых скобках, с обязательным псевдонимом. Внешний запрос работает с результатом внутреннего как с обычной таблицей.

```1c
ВЫБРАТЬ
    Итоги.Склад КАК Склад,
    Итоги.Продано КАК Продано
ИЗ
    (ВЫБРАТЬ
        Продажи.Склад КАК Склад,
        СУММА(Продажи.Количество) КАК Продано
    ИЗ
        Документ.РеализацияТоваровУслуг.Товары КАК Продажи
    СГРУППИРОВАТЬ ПО
        Продажи.Склад) КАК Итоги
ГДЕ
    Итоги.Продано > 100
```

Внутренний запрос сворачивает продажи по складам, внешний отбирает из готовых итогов те, где продано больше ста. Внутренний выполняется первым, его результат становится источником для внешнего.

## Вложенный запрос как список

Второе частое место для подзапроса - оператор `В`. Вместо параметра-списка туда подставляют запрос, возвращающий одну колонку значений.

```1c
ВЫБРАТЬ
    Продажи.Склад КАК Склад,
    Продажи.Количество КАК Количество
ИЗ
    Документ.РеализацияТоваровУслуг.Товары КАК Продажи
ГДЕ
    Продажи.Склад В
        (ВЫБРАТЬ Склады.Ссылка ИЗ Справочник.Склады КАК Склады
         ГДЕ Склады.Город = &Город)
```

Условие читается «склад входит в число складов заданного города». Подзапрос в `В` возвращает список ссылок на московские склады, а внешний отбирает продажи только по ним. Это компактнее, чем соединение, когда из второй таблицы не нужно ни одного поля, а только факт вхождения.

## Пакетный запрос и временные таблицы

Когда шагов больше двух или один промежуточный результат нужен в нескольких местах, вложенность становится нечитаемой. Тогда задачу разбивают на пакет - несколько запросов подряд, разделённых точкой с запятой, где ранние запросы складывают результат во временные таблицы, а поздние их читают.

```1c
ВЫБРАТЬ
    Склады.Ссылка КАК Склад
ПОМЕСТИТЬ ВТ_Склады
ИЗ
    Справочник.Склады КАК Склады
ГДЕ
    Склады.Город = &Город
;
ВЫБРАТЬ
    Продажи.Склад КАК Склад,
    СУММА(Продажи.Количество) КАК Продано
ИЗ
    Документ.РеализацияТоваровУслуг.Товары КАК Продажи
    ВНУТРЕННЕЕ СОЕДИНЕНИЕ ВТ_Склады КАК Склады
    ПО Склады.Склад = Продажи.Склад
СГРУППИРОВАТЬ ПО
    Продажи.Склад
```

Слово `ПОМЕСТИТЬ ВТ_Склады` сохраняет результат первого запроса во временную таблицу. Второй запрос обращается к ней, как к обычной, - по имени `ВТ_Склады`. Точка с запятой разделяет запросы пакета; выполняются они по порядку.

Временные таблицы можно индексировать - `ИНДЕКСИРОВАТЬ ПО Склад` после `ПОМЕСТИТЬ` - чтобы последующие соединения по этому полю шли быстрее. По окончании работы менеджер временных таблиц освобождает их автоматически, но большую таблицу, которая больше не нужна, можно убрать явно запросом `УНИЧТОЖИТЬ ВТ_Склады`.

## Что выбрать

Подзапрос хорош, когда шаг один и результат нужен в одном месте: он держит всю логику в одном тексте. Временная таблица выигрывает, когда промежуточный результат используется несколько раз, когда шагов много или когда его выгодно проиндексировать перед соединением. Ключевое преимущество временной таблицы - она материализуется один раз, а подзапрос в некоторых конструкциях СУБД может пересчитывать многократно.

<Callout type="tip" title="Временные таблицы и виртуальные таблицы регистров">
Частый и полезный приём: сначала во временную таблицу собирают отбор (список номенклатуры, складов, контрагентов), а потом передают её в отбор виртуальной таблицы регистра или соединяют с ней. Так тяжёлый расчёт остатков идёт только по нужным позициям, а не по всему регистру.
</Callout>

## Типичные ошибки

**Подзапрос в ИЗ без псевдонима.** Вложенному запросу в `ИЗ` обязательно нужно имя через `КАК`, иначе к его полям не обратиться и запрос не соберётся.

**Подзапрос в В с несколькими колонками.** В операторе `В` подзапрос должен возвращать ровно одну колонку - список значений для сравнения. Две колонки - логическая ошибка.

**Глубокая вложенность вместо пакета.** Три-четыре уровня вложенных скобок нечитаемы и тяжелы в отладке. Как только вложенность мешает понимать запрос, разбейте его на пакет с временными таблицами - каждый шаг станет отдельным читаемым запросом.

<Detail summary="Почему временная таблица иногда быстрее одного большого запроса">

Оптимизатор СУБД строит план для запроса целиком и на сложных многоэтажных запросах иногда ошибается в оценке: неверно предполагает число строк на промежуточном шаге и выбирает неудачный порядок соединений. Разбивая запрос на пакет, вы материализуете промежуточный результат явно - и дальше СУБД точно знает его размер, потому что таблица уже существует. Плюс проиндексированная временная таблица ускоряет последующие соединения. Это не значит, что пакет всегда быстрее: лишняя материализация тоже стоит времени и памяти. Но на тяжёлых отчётах разбиение на шаги - первый инструмент, когда один большой запрос внезапно оказался медленным.

</Detail>
