# Option и Result вместо null и исключений

Тони Хоар, который в 1965 году придумал null-ссылку, позже назвал её своей «ошибкой на миллиард долларов»: удобная идея десятилетиями ломала программы в самый неожиданный момент, потому что почти любое значение могло внезапно оказаться пустым, а компилятор об этом молчал. В Rust null попросту нет как понятия. Нет здесь и исключений в привычном смысле: функция, которая может не справиться с задачей, возвращает ошибку как обычное значение своего типа, и молча пройти мимо неё не получится, потому что компилятор заставит явно решить её судьбу.

## Два способа сказать «может не получиться»

`Option<T>` описывает значение, которое либо присутствует, либо отсутствует: вариант `Some(T)` хранит настоящие данные, вариант `None` означает пустоту. `Result<T, E>` описывает исход операции, а не просто наличие значения: вариант `Ok(T)` несёт результат успеха, вариант `Err(E)` несёт причину неудачи. Оба типа устроены как обычные перечисления с двумя вариантами, и никакой магии компилятора в их устройстве нет. Магия в другом: получить значение из `Option` или `Result` напрямую нельзя, только через разбор варианта, поэтому забыть про пустоту или про ошибку в Rust заметно труднее, чем в языке с `null` и исключениями.

На рисунке видно, что `Some` и `None` образуют два равноправных исхода одного и того же типа `Option`, и код, который просит значение, обязан пройти через `match` или его аналог, а не просто понадеяться, что там что-то есть.

## Рабочие примеры

Функция, которая ищет первое чётное число в срезе, либо находит его, либо нет:

```rust
fn find_first_even(nums: &[i32]) -> Option<i32> {
    for &n in nums {
        if n % 2 == 0 {
            return Some(n);
        }
    }
    None
}
```

Вызывающий код обязан разобрать оба варианта через `match`:

```rust
let nums = [1, 3, 8, 5];
match find_first_even(&nums) {
    Some(n) => println!("нашли {}", n),
    None => println!("чётных нет"),
}
```

Когда для отсутствия значения достаточно значения по умолчанию, `match` можно заменить методом `unwrap_or`, а сам факт наличия значения проверяют методом `is_some`:

```rust
let n = find_first_even(&nums).unwrap_or(-1);
let found = find_first_even(&nums).is_some();
```

`Result<T, E>` разбирают так же, через `match` с ветками `Ok` и `Err`. Возьмём функцию, которая складывает два числа, представленных строками:

```rust
fn parse_two(a: &str, b: &str) -> Result<i32, std::num::ParseIntError> {
    let x = a.parse::<i32>()?;
    let y = b.parse::<i32>()?;
    Ok(x + y)
}
```

Оператор `?` после `a.parse::<i32>()` делает следующее: если разбор дал `Ok(x)`, из него достаётся `x`, и выполнение идёт дальше; если разбор дал `Err(e)`, функция немедленно возвращает этот `Err(e)` вызывающему, минуя оставшийся код. Без `?` пришлось бы писать `match` на каждой строке разбора: сам оператор избавляет от этого повторения, сохраняя ту же строгость. Вызов:

```rust
match parse_two("4", "abc") {
    Ok(sum) => println!("сумма {}", sum),
    Err(e) => println!("не число: {}", e),
}
```

Вторая строка не разбирается как число, поэтому программа напечатает текст ошибки, а не упадёт.

## Крайние случаи

Методы `unwrap()` и `expect("сообщение")` достают значение из `Some`/`Ok` напрямую, без `match`. Удобно в черновике или в примере, но опасно в рабочем коде: если внутри окажется `None` или `Err`, программа паникует и завершается прямо в этой точке. `expect` отличается от `unwrap` только тем, что паника несёт текст, заданный в аргументе, а не общее сообщение «called `unwrap()` on a `None` value».

Оператор `?` работает только внутри функции, которая сама возвращает совместимый `Result` или `Option`: вызвать его в обычном `main` без подходящей сигнатуры возврата нельзя, компилятор потребует привести типы в соответствие.

Компилятор помечает `Result` атрибутом `#[must_use]`, поэтому если результат вызова, возвращающего `Result`, не сохранить и не использовать, появится предупреждение: игнорирование ошибки в Rust видно уже на этапе сборки, а не молчаливо проходит мимо, как непойманное исключение.

## Типичные ошибки

**Ставить `unwrap()` везде подряд.** Быстро в черновике, но каждый такой вызов означает потенциальную панику в проде вместо обработанной ошибки; для кода, который увидит пользователь, вместо `unwrap` разбирают вариант через `match`, `?` или методы вроде `unwrap_or`.

**Игнорировать возвращённый `Result`.** Предупреждение `#[must_use]` подскажет, но его тоже можно механически проигнорировать; тогда ошибка исчезает бесследно, будто её обработали.

**Путать `Option` и `Result` там, где нужен один из них.** `Option` подходит для «может не быть значения», `Result` подходит для «может пойти не так, и вот почему». Функция чтения файла должна возвращать `Result` с текстом ошибки, а не `Option`, который эту причину теряет.

<Detail summary="Как разворачивается оператор ?">

Выражение `x?`, где `x` имеет тип `Result<T, E>`, компилятор разворачивает примерно в такой код: если `x` равен `Ok(value)`, всё выражение вычисляется в `value`; если `x` равен `Err(e)`, из текущей функции происходит немедленный возврат `Err(From::from(e))`. Именно вызов `From::from` даёт оператору `?` умение конвертировать тип ошибки на лету: если функция объявлена как возвращающая `Result<T, MyError>`, а `parse` вернул `ParseIntError`, оператор `?` сработает при условии, что для `MyError` реализован `From<ParseIntError>`. Для `Option<T>` логика та же, только вместо `Err` ранний возврат происходит при `None`.

</Detail>
