# Обобщённые типы и трейты

Функция, которая берёт из пары наибольший элемент, устроена одинаково что для чисел, что для строк: сравнить два значения и вернуть одно из них. Переписывать это тело заново под каждый тип не хочется, а копия ради копии плодит баги на ровном месте. Rust решает это обобщениями: функция получает параметр-тип, который подставляется при вызове. А там, где телу функции нужно не просто хранить значение произвольного типа, а что-то с ним сделать (вычислить площадь, сравнить, напечатать), в дело вступают трейты: общий язык поведения, который разные типы могут говорить каждый по-своему.

## Параметр-тип и обещание поведения

Обобщение (generic) вводит в сигнатуру параметр-тип, обычно однобуквенный, вроде `T`: он подставляется настоящим типом в момент вызова, и компилятор для каждого использования проверяет всё так же строго, как для конкретного типа. Трейт (trait) описывает набор методов, которые тип обязуется иметь, если заявляет о реализации этого трейта. В отличие от языков с утиной типизацией связь между типом и трейтом не выводится из факта совпадения методов: её пишут явно, блоком `impl Трейт for Тип`. Ограничение в сигнатуре, `T: Trait`, говорит компилятору: подставлять сюда можно только тип, для которого такой `impl` существует, а внутри функции на значении типа `T` разрешено вызывать методы этого трейта.

## Рабочие примеры

Простое обобщение без трейтов, только с ограничением `Copy`, чтобы значение можно было вернуть, не забирая владение у пары:

```rust
fn first<T: Copy>(pair: (T, T)) -> T {
    pair.0
}
```

Эта функция одинаково работает и с числами, и со строковыми срезами:

```rust
let a = first((3, 7));
let b = first(("привет", "мир"));
```

Теперь трейт с одним методом и его реализация для конкретного типа:

```rust
trait Area {
    fn area(&self) -> f64;
}

struct Circle {
    radius: f64,
}

impl Area for Circle {
    fn area(&self) -> f64 {
        std::f64::consts::PI * self.radius * self.radius
    }
}
```

Функция, принимающая любой тип с трейтом `Area`, не знает заранее, какая именно фигура ей достанется:

```rust
fn describe<T: Area>(shape: T) -> f64 {
    shape.area()
}
```

Для готовых трейтов вроде `Debug` или `Clone` реализацию не пишут вручную, а просят компилятор сгенерировать её через `#[derive(...)]`:

```rust
#[derive(Debug, Clone)]
struct Point {
    x: i32,
    y: i32,
}

let p = Point { x: 3, y: 4 };
println!("{:?}", p);
```

Формат `{:?}` печатает значение в отладочном виде и работает только для типов, у которых реализован `Debug`: именно этот трейт и добавляет строка `#[derive(Debug, Clone)]`.

## Крайние случаи

Обобщения в Rust не платят за свою гибкость временем выполнения: компилятор проводит мономорфизацию, то есть генерирует отдельную конкретную версию функции под каждый тип, с которым её реально вызвали. `first::<i32>` и `first::<&str>` после компиляции превращаются в две разные, ничем не связанные функции в бинарнике, и вызов через параметр-тип работает так же быстро, как вызов обычной функции без обобщений.

Вызвать метод на значении типа `T` можно только если этот метод объявлен трейтом из ограничения `T: Trait`; без ограничения компилятор не знает, какие методы вообще существуют у неизвестного пока типа, и откажется собирать код. Перегрузки функций по типу аргументов в Rust нет вообще: у каждого имени функции ровно одна сигнатура, а роль, которую в других языках играет перегрузка, здесь берут на себя трейты и их реализации для разных типов.

## Типичные ошибки

**Забыть ограничение и попытаться вызвать метод на `T`.** `fn print_it<T>(x: T) { println!("{}", x); }` не соберётся: компилятор не знает, реализует ли произвольный `T` форматирование, пока в сигнатуре не появится `T: std::fmt::Display`.

**Путать статическую диспетчеризацию с динамической.** `T: Trait` в сигнатуре запускает generic-путь: компилятор мономорфизирует код и выбирает реализацию на этапе сборки. `dyn Trait` работает иначе, через единственную скомпилированную функцию и выбор реализации в рантайме по таблице указателей. Смешивать эти два способа думать о полиморфизме не стоит: у них разная цена и разные ограничения.

**Ждать перегрузку функций по типу аргумента.** Опыт из C++ или Java подсказывает завести две функции с одинаковым именем и разными параметрами, но в Rust так не выйдет: второе объявление просто конфликтует с первым.

<Detail summary="derive против ручной реализации трейта">

`#[derive(Debug)]` работает только для трейтов из стандартной библиотеки, у которых есть предсказуемая, механическая реализация: `Debug` печатает поля структуры в стандартном виде, `Clone` копирует каждое поле, `PartialEq` сравнивает поля попарно. Компилятор просто генерирует этот код во время сборки, ничего не тратя в рантайме. Как только реализация должна отличаться от очевидной (например, `Debug`, который прячет пароль от печати), derive не подходит, и `impl Debug for Тип` пишут вручную, с собственным телом метода `fmt`.

</Detail>
