# Владение: ядро языка

В C вы сами вызываете `free`, когда память больше не нужна, и сами следите, чтобы не освободить её дважды и не обратиться к уже освобождённой. В Python и Java об этом думает сборщик мусора: он периодически обходит объекты и убирает те, до которых не дотянуться. Rust выбрал третий путь. Здесь нет ни ручного `free`, ни сборщика, который тормозит программу в непредсказуемый момент. Вместо этого компилятор по набору правил сам знает, где заканчивается жизнь каждого значения, и вставляет освобождение памяти за вас. Эти правила и называют *владением* (ownership). Разобравшись в них один раз, вы поймёте, почему Rust отклоняет код, который в других языках собрался бы, - и почему это спасает от целого класса ошибок.

## Три правила, из которых растёт всё остальное

Владение держится на трёх утверждениях, и они простые настолько, что их легко недооценить.

<Steps>
  <li>У каждого значения есть ровно один <em>владелец</em> - переменная, которая за него отвечает.</li>
  <li>Владелец в каждый момент времени только один.</li>
  <li>Когда владелец выходит из области видимости, значение освобождается.</li>
</Steps>

Область видимости - это блок между фигурными скобками. Как только выполнение доходит до закрывающей `}`, все объявленные внутри переменные исчезают, и Rust автоматически освобождает связанные с ними ресурсы. Никакого сборщика: момент освобождения известен заранее, прямо из текста программы.

## Перемещение вместо копирования

Вот где Rust расходится с привычной интуицией. Возьмём строку, которая хранит данные на куче:

```rust
let s1 = String::from("Rust");
let s2 = s1;
```

В Python обе переменные указывали бы на один объект; в C вы бы скопировали указатель и получили две ссылки на один буфер. Rust делает иначе: присваивание *перемещает* владение из `s1` в `s2`. Данные на куче не копируются, но с этого момента `s1` считается недействительной. Правило «владелец только один» не позволяет двум переменным отвечать за один буфер.

Если после перемещения обратиться к `s1`, программа даже не соберётся:

```rust
let s1 = String::from("Rust");
let s2 = s1;
println!("{}", s1);
```

Компилятор ответит: `value borrowed here after move`. Это не каприз, а именно та проверка, которая в C защитила бы вас от повторного освобождения: раз владелец один, память освобождается один раз - в конце жизни `s2`, а не дважды.

## Типы, которые копируются, а не перемещаются

Перемещение касается значений, которые владеют ресурсом на куче. Простые числа целиком лежат на стеке, и копировать их дёшево. Такие типы реализуют трейт `Copy`: при присваивании создаётся честная копия, а оригинал остаётся годным.

```rust
let x = 5;
let y = x;
println!("{} {}", x, y);
```

Здесь напечатается `5 5`: `i32` копируется, `x` никуда не «переехал». `Copy` есть у целых и дробных чисел, `bool`, `char` и кортежей, целиком составленных из таких типов. У `String`, `Vec` и прочих владельцев кучи `Copy` нет - для них присваивание всегда move.

<Callout type="tip" title="Хотите две настоящие копии данных на куче - просите явно">
Если вам действительно нужны два независимых буфера, вызовите `s1.clone()`. Это глубокое копирование: выделяется новая память и содержимое переносится в неё. Rust не делает такого за вашей спиной именно потому, что копия кучи может быть дорогой, и язык хочет, чтобы эта цена была видна в коде.
</Callout>

## Функции тоже забирают владение

Передача значения в функцию подчиняется тем же правилам, что и присваивание. Аргумент-`String` перемещается внутрь функции, и снаружи им больше нельзя пользоваться - если только функция не вернёт его обратно.

```rust
fn shout(s: String) -> String {
    s + "!"
}

fn main() {
    let msg = String::from("Rust");
    let loud = shout(msg);
    println!("{}", loud);
}
```

`msg` уехал в `shout`, там к нему приклеили восклицательный знак и вернули как новое значение, которым теперь владеет `loud`. Программа напечатает `Rust!`. Постоянно гонять значения туда-обратно неудобно, и именно поэтому в следующем уроке появится заимствование - способ дать функции доступ, не отдавая владение.

## Типичные ошибки

**Пользоваться переменной после move.** Самая частая осечка новичка: `let b = a;`, а потом обращение к `a`. Компилятор укажет точное место. Решение - либо клонировать (`a.clone()`), либо перестроить код так, чтобы `a` больше не понадобилась.

**Считать, что всё копируется, как числа.** `let y = x;` для `i32` оставляет `x` живым, а для `String` - нет. Разница в том, есть ли у типа `Copy`. Если сомневаетесь, держите в голове вопрос «лежит ли значение целиком на стеке».

**Возвращать значение только чтобы отдать владение назад.** Это работает, но быстро превращается в лапшу из аргументов и возвратов. Правильный инструмент для «дать посмотреть, не отдавая» - ссылки, о них дальше.

<Detail summary="RAII: почему конец скобки освобождает не только память">

Владение освобождает не только память под строку. Rust связывает конец жизни значения с методом `drop`, и туда можно повесить что угодно: закрытие файла, освобождение сетевого сокета, снятие блокировки. Этот приём пришёл из C++ и называется RAII - «захват ресурса есть инициализация». Пока значение живо, ресурс держится; вышли из области видимости - ресурс отдан, причём в строго обратном порядке создания. Поэтому в Rust почти никогда не пишут «не забыть закрыть»: закрытие происходит само в конце блока, даже если из него вышли по ошибке.

</Detail>
