# Перечисления и сопоставление с образцом

В C `enum` - это просто именованные числа, и на этом его возможности заканчиваются. В Rust `enum` куда мощнее: вариант может держать при себе данные, разные варианты - данные совершенно разной формы, а когда вы захотите разобрать такое значение, язык заставит явно учесть каждый возможный случай. Инструмент для этой проверки - `match`, и он устроен так, что забыть один из вариантов физически не получится: код просто не соберётся.

## Тип-сумма и обязательный разбор

`enum` описывает тип-сумму: значение этого типа - это ровно один из перечисленных вариантов, никогда не смесь и не сразу два одновременно. Это прямая противоположность `struct`, где значение - это все поля сразу, тип-произведение. `match` берёт значение `enum` и раскладывает его по веткам: одна ветка отвечает ровно за один вариант, и компилятор на этапе сборки проверяет, что учтены все варианты без исключения - совсем не так, как `switch` в C-подобных языках, где пропущенный `case` просто молча ничего не делает.

## Простой enum без данных

```rust
enum Direction {
    North,
    South,
    East,
    West,
}
```

Здесь `enum` - это список именованных состояний без данных внутри, почти как в C. Создать значение можно, указав полный путь через `::`:

```rust
let d = Direction::East;
```

## enum с данными в вариантах

Куда интереснее, что каждый вариант может нести свои данные, и у разных вариантов эти данные разной формы:

```rust
enum Msg {
    Quit,
    Move(i32, i32),
    Write(String),
}
```

`Quit` не хранит ничего, `Move` хранит пару координат, `Write` - строку. Один тип `Msg` покрывает три разных формы данных сразу, и это не объединение в стиле C, а безопасная альтернатива ему: Rust всегда знает, какой вариант перед ним в данный момент, и не позволит прочитать данные не того варианта - в отличие от C, где перепутать активное поле `union` легко и это не поймает ни компилятор, ни рантайм.

## match разбирает значение по веткам

```rust
fn describe(msg: Msg) -> String {
    match msg {
        Msg::Quit => String::from("выход"),
        Msg::Move(x, y) => format!("движение к {x}, {y}"),
        Msg::Write(text) => format!("текст: {text}"),
    }
}
```

Каждая ветка вида `Enum::Variant => выражение` разбирает свой случай. У `Move` и `Write` шаблон сразу же связывает данные варианта с именами `x`, `y` и `text` - дальше в теле ветки ими пользуются как обычными переменными. Ветки перечисляют в любом порядке, лишь бы был учтён каждый вариант из объявления `enum`.

<Callout type="warning" title="Пропущенная ветка - это ошибка компиляции, а не бага в проде">
Уберите ветку `Msg::Write` из примера выше - и код не соберётся. Компилятор сообщит `non-exhaustive patterns` и назовёт ровно те варианты, которые вы забыли обработать. Это не придирка компилятора, а его основная работа: не дать коду проскочить мимо случая, о котором забыли.
</Callout>

## `_` перехватывает остальное

Если часть вариантов не важна, есть шаблон `_`, который ловит всё, что не подошло под предыдущие ветки:

```rust
match d {
    Direction::North => println!("вверх"),
    _ => println!("куда-то ещё"),
}
```

`_` закрывает исчерпаемость за вас формально, но ценой того, что новый вариант, добавленный в `enum` позже, тихо провалится именно в эту ветку - компилятор больше не подскажет, что про него забыли, потому что формально всё ещё покрыто.

## `if let` вместо полного `match`

Когда интересен только один вариант, а остальные не нужно даже называть, `match` с единственной содержательной веткой и пустым `_` в конце выглядит громоздко. `if let` делает то же самое короче:

```rust
if let Msg::Write(text) = msg {
    println!("{text}");
}
```

Это ровно тот же шаблон, что и в ветке `match`, только без обязательной исчерпаемости: если вариант не подошёл, код просто идёт дальше, ничего не выполнив.

## Крайние случаи

Требование исчерпаемости - не формальность ради формальности, а главная практическая польза `match`. Добавьте в `enum` новый вариант через полгода после того, как код написан, - и каждый `match` по этому типу, где вариант не учтён, откажется собираться. Ошибка компиляции с точным списком мест, которые нужно обновить, надёжнее, чем баг, замеченный постфактум в проде кем-то из пользователей.

Обратная сторона `_` в том, что он ловит одновременно и то, что вы имели в виду, и то, что забыли. Добавили `Msg::Ping` в `enum` - ветка `_` молча его проглотит, и компилятор не подскажет, что новый вариант стоило бы обработать явно. Поэтому `_` уместен, когда «всё остальное» - это действительно один осмысленный случай, а не когда просто лень перечислить оставшиеся варианты по имени.

## Типичные ошибки

**Неисчерпывающий `match`.** Забыли ветку - компилятор откажется собирать код с `non-exhaustive patterns`. Решение: добавить недостающую ветку или явный `_`, если остаток действительно не важен.

**Забыть полный путь `Enum::Variant`.** Вариант `North` сам по себе не существует как имя - это `Direction::North`. Исключение - когда варианты явно занесены в область видимости через `use Direction::*`, но в обычном коде путь принято писать полностью.

**Городить полный `match` там, где хватило бы `if let`.** Если интересен ровно один вариант, а для остальных не нужно никакого действия, `match` с единственной осмысленной веткой и пустым `_` - лишний шум. `if let` короче и точнее показывает намерение.

<Detail summary="Почему исчерпаемость особенно ценна при рефакторинге">

Представьте, что `Msg` уже используется в десятке мест по всему проекту, и требуется добавить новый вариант - скажем, `Msg::Resize(u32, u32)`. В языке без обязательного разбора пришлось бы вручную искать каждое место, где обрабатываются варианты `Msg`, и надеяться, что ни одно не пропущено - надежда плохая стратегия для больших кодовых баз. В Rust это не вопрос дисциплины программиста: стоит добавить вариант в `enum`, и компилятор сразу перечислит все `match`-выражения, которые не покрывают `Resize`. Забытый случай физически невозможно не заметить - он останавливает сборку задолго до того, как код попадёт куда-либо ещё. Поэтому `enum` вместе с `match` в Rust - это не просто удобный синтаксис, а инструмент безопасного рефакторинга: код меняется, а забытые случаи ловятся до запуска программы, а не после.

</Detail>
