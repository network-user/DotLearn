# Cargo и структура проекта

В C сборка - это ремесло само по себе: вы вызываете `gcc` или `clang` с нужными флагами, пишете Makefile, вручную ищете и линкуете сторонние библиотеки, следите, чтобы на машине коллеги стояли те же версии зависимостей. Один и тот же проект на разных машинах может собираться по-разному - или не собираться вовсе, потому что чего-то не хватает в системе.

В Rust почти всю эту работу берёт на себя одна утилита - `cargo`. Она создаёт структуру нового проекта, компилирует код, тянет и версионирует зависимости, запускает тесты и умеет собирать документацию. `cargo` устанавливается вместе с самим Rust и с первого дня становится инструментом, которым вы пользуетесь каждый день, а не только на старте проекта.

## Cargo как сборщик и менеджер пакетов одновременно

`cargo` совмещает две роли, которые в других языках часто разнесены по разным инструментам. Для C это были бы отдельно make или CMake и отдельно ручной поиск библиотек; для Python - pip отдельно от setuptools. В Rust это один предсказуемый жизненный цикл.

Команда `cargo new hello` создаёт папку `hello` с уже готовой структурой: файл-манифест `Cargo.toml` и файл с кодом `src/main.rs`, в котором лежит заготовка «Hello, world!». Как показано на рисунке, дальше вы пишете код внутри `src`, а `cargo build` превращает его в исполняемый файл. Запускать `cargo build` и сам бинарь отдельно неудобно, поэтому чаще всего во время разработки пользуются `cargo run` - команда компилирует проект, если что-то изменилось, и сразу запускает результат одной командой.

Манифест `Cargo.toml` описывает пакет и его зависимости:

```text
[package]
name = "hello"
version = "0.1.0"
edition = "2021"

[dependencies]
```

Секция `[package]` хранит имя пакета, версию и редакцию языка. Секция `[dependencies]` перечисляет внешние крейты, от которых зависит проект; у только что созданного проекта она пуста. Когда нужна оптимизированная сборка для продакшна, вместо `cargo build` вызывают `cargo build --release` - компилятор тратит больше времени на оптимизации, а результат ложится не в `target/debug`, а в `target/release`.

## Инструменты в комплекте

Помимо сборки, `cargo` закрывает почти весь остальной цикл работы над проектом одним и тем же интерфейсом подкоманд - не нужно ни отдельного тест-раннера, ни отдельного генератора документации.

<Steps>
  <li><code>cargo new hello</code> - создать новый проект с готовой структурой папок и файлов.</li>
  <li><code>cargo check</code> - быстро проверить, что код компилируется, не тратя время на генерацию бинарного файла. Удобно гонять постоянно во время написания кода.</li>
  <li><code>cargo test</code> - собрать и запустить тесты из проекта, включая функции, помеченные атрибутом <code>#[test]</code>.</li>
  <li><code>cargo doc --open</code> - собрать документацию по крейту и его зависимостям и открыть её в браузере.</li>
</Steps>

<Callout type="tip" title="cargo fmt и cargo clippy - тоже часть арсенала">
В арсенал входят и `cargo fmt`, который приводит код к единому стилю, и `cargo clippy` - линтер, подсказывающий более идиоматичные способы написать то же самое. Если их ещё нет, они ставятся командой `rustup component add rustfmt clippy`, а дальше запускаются как обычные подкоманды cargo, без отдельной настройки.
</Callout>

## Крейты и модули

Единица компиляции в Rust называется крейтом (crate) - это либо исполняемый бинарь, либо библиотека, которую можно подключить к другому проекту. Проект, который вы только что создали командой `cargo new hello`, сам по себе уже крейт-бинарь: `cargo` понимает это по файлу `src/main.rs` - именно он служит корнем бинарного крейта. Если вместо приложения вы пишете библиотеку для других проектов, корнем становится `src/lib.rs`, а сама библиотека не запускается напрямую и не требует функции `main`. Внутри одного крейта код раскладывают по модулям (`mod`) - модуль группирует связанные функции, структуры и константы и даёт им общее пространство имён.

По умолчанию всё, что объявлено внутри модуля, видно только внутри него самого. Чтобы имя было доступно снаружи, перед ним пишут `pub`. Обратиться к элементу внутри модуля можно по пути через двойное двоеточие, например `math::square`; если писать путь целиком неудобно каждый раз, его сокращают директивой `use`.

```rust
mod math {
    pub fn square(x: i32) -> i32 {
        x * x
    }
}

fn main() {
    let result = math::square(4);
    println!("{}", result);
}
```

Уберите `pub` перед `fn square`, и строка `math::square(4)` перестанет собираться: компилятор скажет, что функция приватна. Видимость в Rust явная - модуль по умолчанию прячет всё, что в нём объявлено, и открывает наружу только то, что вы пометили сами.

## Что стоит знать заранее

Сборка по умолчанию - debug: она быстрая для компилятора, но код в ней почти не оптимизирован и работает заметно медленнее готового релиза. Флаг `--release` включает набор оптимизаций и кладёт результат в `target/release` - именно эту сборку отдают в продакшн, а не `target/debug`.

Рядом с `Cargo.toml`, где вы перечисляете зависимости напрямую, cargo создаёт файл `Cargo.lock` - в нём зафиксированы точные версии всех зависимостей, включая транзитивные, так что сборка воспроизводится одинаково на любой машине. Папка `target/` со всеми промежуточными и итоговыми файлами сборки в систему контроля версий не попадает - `cargo new` сразу добавляет её в `.gitignore`. Сам `Cargo.lock`, в отличие от `target/`, обычно коммитят вместе с исходниками - так все, кто собирает проект, получают одинаковые версии зависимостей.

Добавить внешнюю библиотеку можно командой `cargo add имя_крейта` - она сама допишет строку в `[dependencies]` и обновит `Cargo.lock`; то же самое можно сделать вручную, отредактировав `Cargo.toml`. Зависимости тянутся с crates.io - главного реестра пакетов для Rust, аналога npm для JavaScript или PyPI для Python.

## Типичные ошибки

**Путать крейт и модуль.** Крейт - единица компиляции и публикации на crates.io, то, что собирается в бинарь или библиотеку целиком. Модуль - внутреннее деление кода внутри одного крейта, просто пространство имён. Один крейт почти всегда содержит несколько модулей.

**Забыть `pub` и не увидеть функцию снаружи модуля.** Самая частая причина ошибки «функция не найдена» или «приватна» - именно пропущенное слово `pub` перед объявлением, а не опечатка в пути.

**Править `Cargo.lock` руками.** Файл управляется автоматически; ручные правки cargo перезапишет при следующей сборке, а расхождение с `Cargo.toml` только запутает версии зависимостей. Чтобы обновить зависимость, используют команду `cargo update`.

<Detail summary="Чем debug-сборка отличается от release">

Разница не только в оптимизациях. `cargo build` без флагов собирает как можно быстрее: компилятор почти не переставляет инструкции, не разворачивает функции инлайн, оставляет включёнными проверки переполнения целых чисел - если в debug-сборке `i32` переполнится, программа паникует, а не молча заворачивается через ноль. `cargo build --release` включает полный набор оптимизаций LLVM, убирает часть проверок и на крупных проектах может собираться по несколько минут вместо секунд. Поэтому во время разработки почти всегда пользуются `cargo run` и `cargo build` без флагов, а `--release` подключают перед бенчмарками, финальным тестированием и публикацией.

</Detail>
