# Один разъём вместо сотни переходников

В разделе [[tool-use]] Вы видели, как модель просит вызвать инструмент. Но за вызовом стоит обвязка: кто-то должен описать инструмент, подключиться к нужному сервису, исполнить вызов и вернуть результат. Если каждое приложение пишет эту обвязку для каждого сервиса заново - для GitHub, для базы, для файловой системы, - получается мешанина несовместимых интеграций: N приложений на M сервисов дают N×M переходников. Model Context Protocol (MCP) превращает это в N+M: единый протокол, по которому любой клиент общается с любым сервером. Один стандартный разъём вместо сотни самодельных переходников.

К концу раздела Вы будете понимать, что такое MCP и зачем нужен стандарт, как устроены клиент/хост и MCP-сервер, что сервер предоставляет (инструменты, ресурсы, промпты), на каком транспорте это работает, и чем MCP отличается от обычного вызова функций из предыдущего раздела.

## Зачем вообще стандарт

Вызов инструментов решает, *что* модель может попросить. MCP решает, *как* приложение подключает источники этих возможностей единообразно. До стандарта каждая связка «приложение - сервис» писалась вручную и по-своему: своя авторизация, свой формат, своя обработка ошибок. Стоило добавить новый сервис - и его надо было прикручивать к каждому приложению отдельно.

Как видно на схеме, ценность стандарта - в сокращении комбинаторики. Написал MCP-сервер для сервиса один раз - и к нему может подключиться любой MCP-совместимый клиент: чат-приложение, IDE, агент. Не нужно повторять интеграцию под каждое приложение. Это та же идея, что USB или HTTP: договорились о разъёме - и всё, что его поддерживает, сразу совместимо.

## Хост, клиент и сервер

В MCP три роли. **Хост** - это приложение, в котором живёт модель (чат-клиент, IDE, агент). Внутри хоста работает **MCP-клиент** - часть, которая говорит на протоколе MCP. А на другом конце - **MCP-сервер**: отдельная программа, которая предоставляет конкретные возможности (доступ к файлам, к GitHub, к базе). Клиент и сервер общаются по протоколу; хост связывает всё это с моделью и пользователем.

Наведитесь на сервер на схеме - и увидите, что он предоставляет. Один хост может держать связь сразу с несколькими серверами, собирая их возможности в общий набор для модели. Сервер при подключении объявляет свои возможности, а клиент их обнаруживает - никакого жёсткого зашивания списка в приложение.

## Что предоставляет сервер: tools, resources, prompts

MCP-сервер выставляет наружу три вида возможностей, и их полезно различать.

- **Tools (инструменты)** - действия, которые модель может вызвать: создать issue, выполнить запрос к базе, записать файл. Это прямой родственник function calling из прошлого раздела - именно их модель зовёт в агентном цикле.
- **Resources (ресурсы)** - данные, которые можно прочитать и подложить в контекст: файл, запись в базе, страница документации. Ресурс не «делает», а «даёт прочитать».
- **Prompts (промпты)** - готовые шаблоны взаимодействия, которые сервер предлагает пользователю: например, «сделай code review этого файла» с уже выстроенной структурой запроса.

<Callout type="info">Различие «tools против resources» стоит запомнить: инструмент совершает действие (и его выбирает модель), ресурс предоставляет данные для чтения (и обычно его подключает пользователь или хост). Промпты - заготовки сценариев. Сервер может предоставлять любую комбинацию этих трёх.</Callout>

## Транспорт: JSON-RPC поверх stdio или HTTP

Сообщения между клиентом и сервером идут в формате JSON-RPC - простого протокола «запрос с именем метода и параметрами → ответ». Сам этот обмен может ехать по разным транспортам. Два основных: **stdio** - сервер запущен как локальный процесс, общение идёт через его стандартные ввод и вывод (удобно для локальных серверов: файлы, локальные инструменты); и **HTTP** - сервер живёт по сети как удалённый сервис (удобно для облачных интеграций). Логика поверх одна и та же - меняется лишь «труба», по которой бегут JSON-RPC-сообщения.

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "create_issue",
    "arguments": { "title": "Баг в логине", "body": "Падает на пустом пароле" }
  }
}
```

Это типичное JSON-RPC-сообщение: метод `tools/call`, имя инструмента и аргументы. Сервер исполнит вызов и вернёт ответ с тем же `id`. Заметьте, насколько это похоже на вызов инструмента из прошлого раздела - и это не совпадение.

Соберём крошечный «реестр сервера»: по объявленным возможностям ответить, что сервер умеет. Это упрощённая модель того, как клиент обнаруживает возможности сервера при подключении. Запустите, затем поменяйте набор возможностей и посмотрите на вывод.

## Чем MCP отличается от обычного function calling

Здесь легко запутаться, поэтому проговорим явно. Function calling (раздел [[tool-use]]) - это *механизм*: как модель в рамках одного API-вызова просит выполнить функцию. MCP - это *стандарт интеграции*: как приложение подключает внешние источники инструментов и данных так, чтобы они были переносимы между разными приложениями.

То есть они работают вместе. MCP-сервер объявляет инструмент; хост передаёт его модели; модель вызывает его привычным механизмом function calling; результат возвращается через тот же протокол. MCP не заменяет вызов функций - он стандартизирует, откуда эти функции берутся.

## Когда поведение удивит

Стандарт не делает интеграцию безопасной автоматически. MCP-сервер - это код, которому Вы даёте доступ к своим данным и действиям; подключив непроверенный сервер, Вы пускаете его в свой контекст. Доверие к серверу - отдельный вопрос, который протокол сам не решает.

Второй момент: MCP не «учит» модель новому. Сервер лишь предоставляет возможности; решает, звать ли инструмент и как распорядиться ресурсом, по-прежнему модель - со всеми оговорками из раздела [[tool-use]] про ошибочные и галлюцинированные вызовы.

## Типичные заблуждения

- **«MCP - это замена function calling».** Нет: это разные слои. MCP-сервер предоставляет инструменты, а модель вызывает их обычным function calling; они работают вместе.
- **«MCP-сервер - это сама модель».** Сервер не содержит модель; он предоставляет возможности (tools/resources/prompts). Модель живёт в хосте, на другом конце протокола.
- **«Подключил MCP-сервер - значит, безопасно».** Стандарт унифицирует подключение, но не гарантирует доверие: сервер получает доступ к Вашим данным и действиям, и проверять его надо самому.

<Detail summary="Почему resources и tools разводят по разным сторонам">

На первый взгляд можно было бы свести всё к инструментам: «прочитать файл» - тоже действие. Но различие смысловое и важное для управления. Инструмент модель выбирает сама в ходе рассуждения - это активное действие с потенциальным эффектом (создать issue, удалить запись). Ресурс же обычно подключает пользователь или хост заранее: «вот этот документ - в контекст». Разделение даёт контроль: действия можно гейтить и подтверждать (как в разделе [[tool-use]]), а данные - явно выбирать, что подложить, не отдавая модели право самой решать, что прочитать. Промпты добавляют третью ось - переиспользуемые сценарии, которые пользователь запускает осознанно. Три категории - это не педантизм, а разные модели управления доступом и инициативой.

</Detail>
