# Как модель выходит за пределы своего текста

Языковая модель умеет одно - продолжать текст. Она не знает сегодняшней погоды, не видит Вашу базу данных, не может отправить письмо и даже не считает в столбик надёжно. Всё, что появилось после даты отсечки данных (раздел [[training-llms]]), для неё не существует. И тем не менее современные ассистенты бронируют столики, ищут в интернете и запускают код. Секрет в том, что модель не делает это сама - она просит сделать. Механизм, который превращает «продолжатель текста» в систему, способную действовать, называется вызовом инструментов (tool use, function calling).

К концу раздела Вы будете понимать, что такое инструмент и как его описывают JSON-схемой, как модель решает вызвать инструмент и что она возвращает вместо ответа, что такое структурированный вывод, и как работает агентный цикл «модель → вызов инструмента → результат → ответ».

## Модель не зовёт инструмент - она просит его позвать

Ключевая идея, которую легко понять неправильно: модель сама ничего не выполняет. Она по-прежнему только генерирует текст. Вызов инструмента - это особый, структурированный кусок этого текста: вместо обычного ответа модель выдаёт «хочу вызвать инструмент `X` с такими аргументами». Дальше этот запрос исполняет Ваш код (или платформа), а результат подаёт модели обратно.

## Инструмент описывают JSON-схемой

Чтобы модель знала, что ей доступно, каждый инструмент описывают структурно: имя, человекочитаемое описание (зачем он и когда его звать) и схема аргументов в формате JSON Schema - какие параметры, каких типов, какие обязательны. Описание - не формальность: именно по нему модель решает, подходит ли инструмент под запрос. Расплывчатое описание ведёт к тому, что инструмент зовут не вовремя или не зовут, когда надо.

```json
{
  "name": "get_weather",
  "description": "Узнать текущую погоду в городе. Вызывать, когда пользователь спрашивает про погоду.",
  "input_schema": {
    "type": "object",
    "properties": {
      "city": { "type": "string", "description": "Название города" }
    },
    "required": ["city"]
  }
}
```

Схема задаёт контракт: модель обязана вернуть аргументы в этой форме. Поэтому `city` придёт строкой, а не числом, и поле будет называться именно `city`. Хорошее описание плюс точная схема - половина успеха: модель и поймёт, когда звать инструмент, и заполнит аргументы правильно.

## Как модель решает вызвать инструмент

Когда инструменты объявлены, на каждый запрос у модели есть выбор: ответить текстом сразу или запросить вызов. Решение она принимает по смыслу запроса и по описаниям инструментов. «Сколько будет 2+2» - ответит сама; «какая погода в Париже» - при наличии `get_weather` запросит вызов, потому что погода ей неизвестна.

<Callout type="tip">Сила инструментов - в их описаниях. Будьте прескриптивны: пишите не только что инструмент делает, но и когда его звать («вызывать, когда нужен курс валют или свежая новость»). Точный триггер в описании заметно повышает шанс, что модель позовёт инструмент в нужный момент.</Callout>

Важно: вызов - это запрос, а не гарантия. Модель может ошибиться - позвать не тот инструмент, придумать аргумент или решить ответить сама там, где стоило позвать. Поэтому результаты инструментов и сам факт вызова контролирует Ваш код, а не слепо доверяет модели.

## Структурированный вывод: тот же механизм для формата

У вызова инструментов есть близкий родственник - структурированный вывод (structured output). Идея та же: заставить модель выдать данные в заданной JSON-схеме, а не свободным текстом. Это удобно, когда ответ нужно разобрать программно: извлечь поля, положить в базу, передать дальше. Вместо «надеюсь, модель вернёт валидный JSON» Вы задаёте схему, и ответ ей соответствует.

```json
{
  "type": "object",
  "properties": {
    "sentiment": { "type": "string", "enum": ["позитив", "негатив", "нейтрально"] },
    "confidence": { "type": "number" }
  },
  "required": ["sentiment", "confidence"]
}
```

С такой схемой ответ на «оцени тональность отзыва» придёт не абзацем прозы, а объектом с полями `sentiment` и `confidence`, готовым к разбору. Тот же приём с JSON-схемой работает и для аргументов инструмента, и для финального ответа - это две стороны одной идеи: ограничить форму вывода.

## Потрогайте руками

Соберём ядро диспетчера инструментов: по имени инструмента и аргументам вернуть результат. Это та часть, которую исполняет Ваш код после того, как модель попросила вызов. Запустите, затем поменяйте запрошенный инструмент и посмотрите, как меняется результат.

## Агентный цикл: вызовы один за другим

Один вызов - частный случай. В реальных задачах модель проходит цикл несколько раз: запросила поиск, получила результаты, на их основе запросила ещё один инструмент, и только потом ответила. Это и есть агентный цикл из схемы выше: модель → вызов → результат → (снова модель) → … → ответ. Цикл крутится, пока модель не решит, что собрала достаточно, и не выдаст финальный текст вместо очередного вызова. Так из простого «продолжателя токенов» вырастает система, способная на многошаговые действия.

## Когда поведение удивит

Инструменты не делают модель всеведущей и безошибочной. Если описание расплывчато, модель зовёт инструмент невпопад; если результат инструмента кривой - кривым будет и ответ. Качество агента определяется не только моделью, но и тем, насколько аккуратно описаны инструменты и обработаны их результаты.

Второй сюрприз - модель может «галлюцинировать» вызов: придумать имя инструмента, которого нет, или аргумент не из схемы. Поэтому Ваш код обязан проверять и имя, и аргументы перед исполнением, а на ошибку отвечать модели понятным сообщением, чтобы она поправилась на следующем шаге.

## Типичные заблуждения

- **«Модель сама выполняет инструмент».** Нет: она лишь выдаёт структурированный запрос на вызов; исполняет его Ваш код, и он же возвращает результат.
- **«Описание инструмента - формальность».** Именно по описанию модель решает, звать ли инструмент; расплывчатое описание - главная причина вызовов невпопад.
- **«Раз есть инструмент, модель всегда его позовёт правильно».** Вызов - это запрос, а не гарантия; модель может ошибиться, поэтому результаты и факт вызова контролирует код.

<Detail summary="Чем вызов инструмента отличается от обычного ответа на уровне протокола">

Под капотом ответ модели - это последовательность блоков. Обычно блок один: текст. При вызове инструмента модель вместо (или помимо) текстового блока выдаёт особый блок «tool use» с именем инструмента и аргументами-объектом, а её ход помечается признаком «остановился, чтобы вызвать инструмент». Ваш код видит этот признак, исполняет вызов и присылает обратно блок «tool result», привязанный к тому же вызову по идентификатору. Модель продолжает с того места, уже зная результат. Поэтому диалог с инструментами - это не «магия», а строгий обмен типизированными блоками: текст, запрос вызова, результат вызова - по очереди.

</Detail>
