# Внимание: каждый токен спрашивает у остальных, что важно

В прошлом разделе мы сказали: self-attention - единственное место, где токены обмениваются информацией. Теперь заглянем внутрь. Представьте предложение «банк закрыли, потому что он обанкротился». Чтобы понять, что значит «он», токен «он» должен посмотреть назад и найти «банк». Внимание - это механизм, который позволяет каждому токену самому решить, на какие другие токены посмотреть и насколько сильно, а затем собрать с них информацию. Никто не задаёт связи вручную - модель учится им сама.

К концу раздела Вы будете понимать, что такое query, key и value, как из них считается оценка совместимости (scaled dot-product), зачем эту оценку делят на корень из размерности, и как softmax превращает оценки в веса внимания, которые суммируются в единицу.

## Три роли: query, key, value

Из эмбеддинга каждого токена модель получает три вектора - умножением на три обучаемые матрицы. У каждого своя роль, и удобнее всего думать о них как о поиске в библиотеке:

- **Query (запрос)** - то, что ищет текущий токен: «мне нужно существительное мужского рода слева».
- **Key (ключ)** - то, чем «рекламирует себя» каждый токен: «я существительное, мужской род, я банк».
- **Value (значение)** - то, что токен отдаст, если его выберут: собственно полезное содержимое.

Токен сравнивает свой query с key каждого другого токена. Чем сильнее они совпадают, тем больше внимания он уделит, и тем большую долю value этого токена заберёт в свой результат.

На схеме query токена «он» сильнее всего совпал с key токена «банк» (сплошная стрелка), поэтому именно value «банка» внесёт основной вклад в результат внимания для «он». Так токен «подтягивает» к себе нужный контекст.

## Оценка совместимости - это скалярное произведение

Как измерить, насколько query совпал с key? Скалярным произведением. Чем более сонаправлены query и key, тем больше произведение, тем выше оценка (score). Для одного query и набора keys получается по числу на каждый key.

```python
def dot(a, b):
    return sum(x * y for x, y in zip(a, b))

q = [1, 0]
keys = [[1, 0], [0, 1], [1, 1]]
scores = [dot(q, k) for k in keys]
print(scores)
```

Печатается `[1, 0, 1]`: query `[1, 0]` лучше всего совпал с первым и третьим ключами и совсем не совпал со вторым. Это сырые оценки - они ещё не вероятности и не обязаны быть положительными.

## Зачем делить на корень из размерности

Есть тонкость. Если векторы длинные (а в моделях их размерность - десятки и сотни), скалярные произведения становятся большими по модулю. Большие числа на входе softmax дают почти «одна единица, остальные нули» - распределение становится слишком резким, и градиенты при обучении затухают. Лекарство - поделить каждую оценку на корень из размерности ключа. Отсюда название: scaled dot-product attention.

<Callout type="info">Деление на корень из размерности - не косметика. Без него при большой размерности softmax почти всегда насыщается в один токен, и модель теряет способность мягко распределять внимание между несколькими токенами.</Callout>

## softmax превращает оценки в веса

Масштабированные оценки прогоняют через знакомый softmax (раздел [[what-is-llm]]). На выходе - веса внимания: неотрицательные числа, суммирующиеся в единицу. Каждый вес говорит, какую долю value соответствующего токена забрать. Результат внимания для токена - это взвешенная сумма всех value с этими весами.

На тепловой карте каждая строка - это распределение внимания одного токена-запроса по всем токенам-ключам. Наведите курсор на ячейку и прочитайте конкретный вес «запрос → ключ». Обратите внимание: внимание здесь причинно-следственное (causal), поэтому токен не смотрит вправо, на будущие токены, - правый верхний угол пуст.

## Потрогайте руками

Соберём scaled dot-product attention целиком для одного query: посчитаем оценки, поделим на корень из размерности, прогоним через softmax. Запустите код, затем поменяйте `q` на `[0, 1]` и посмотрите, как сместятся веса - теперь query будет совпадать со вторым ключом, а не с первым.

<PyDemo
  title="Веса внимания для одного query"
  code={`import math

def attention_weights(q, keys):
    d = len(q)
    scores = [sum(a * b for a, b in zip(q, k)) / math.sqrt(d) for k in keys]
    m = max(scores)
    exps = [math.exp(s - m) for s in scores]
    total = sum(exps)
    return [e / total for e in exps]

q = [1, 0]
keys = [[1, 0], [0, 1]]
print([round(w, 3) for w in attention_weights(q, keys)])`}
/>

Выводится `[0.67, 0.33]`: query сильнее совпал с первым ключом, поэтому его вес выше, но второй ключ не обнулился. Веса в сумме дают единицу - это и есть распределение внимания. Умножив каждый вес на соответствующий value и сложив, модель получит итог внимания для этого токена.

## Когда поведение удивит

Кажется, что у токена одно внимание. На деле в каждом блоке несколько голов внимания (multi-head attention), работающих параллельно с разными матрицами query/key/value. Одна голова может следить за синтаксисом, другая - за дальними связями, третья - за чем-то, чему у нас нет названия. Их результаты объединяются. Это даёт модели смотреть на контекст с нескольких «точек зрения» одновременно.

Второй сюрприз - стоимость. Чтобы посчитать внимание, каждый токен сравнивается с каждым, поэтому объём вычислений растёт примерно как квадрат длины контекста. Именно отсюда дороговизна очень длинных запросов из раздела [[embeddings-context]], и именно поэтому так много инженерных усилий уходит на ускорение внимания.

## Типичные заблуждения

- **«Query, key и value - это разные токены».** Это три разных вектора одного и того же токена, полученные тремя матрицами; путать их роли - частая ошибка.
- **«Веса внимания можно читать как объяснение решения модели».** Веса показывают, откуда токен собрал информацию, но это не полное и не всегда честное объяснение того, «почему» модель ответила так; интерпретировать их буквально опасно.
- **«Деление на корень из размерности можно опустить».** Без него softmax насыщается, обучение портится; это часть определения механизма, а не опция.

<Detail summary="Почему именно корень из размерности">

Если компоненты query и key независимы и имеют единичную дисперсию, дисперсия их скалярного произведения растёт линейно с размерностью d, а стандартное отклонение - как корень из d. Деление на корень из d возвращает оценкам стабильный масштаб независимо от размерности, чтобы softmax не уходил в насыщение. Это ровно тот множитель, что выписан в исходной статье про трансформер.

</Detail>
