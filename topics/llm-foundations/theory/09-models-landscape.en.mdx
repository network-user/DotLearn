# How not to drown in the model zoo

By 2026 choosing a large language model is no longer about which one is best. Dozens exist, they ship in families, and each tier has a profile: one cheap and fast, another with a million-token context, a third strongest at reasoning but pricier. Asking for the best model is usually the wrong question - the right one is which model fits this task and this budget. Learn the comparison axes and selection stops being guesswork.

By the end of this section you will navigate major 2026 families, understand open vs proprietary weights, know key axes - size, context length, multimodality, price, latency - and pick a model for a concrete job.

## Families, not lone models

Models ship in families under one name: several tiers from light and cheap to heavy and capable. That lets you pick the right level per task without learning a new API each time.

- **Claude** (Anthropic) - three characteristic tiers: Haiku (fast and cheap), Sonnet (balance), Opus (most capable); by 2026 an even stronger tier sits above.
- **GPT** (OpenAI) - one of the best-known families, compact to flagship tiers.
- **Gemini** (Google) - strong focus on multimodality and very long context.
- **Llama** (Meta) - flagship of open weights: download and run locally.
- **Mistral** - European family emphasizing efficient open models.

<Callout type="info">Names and version numbers go stale in months. Remember structure: family → tiers inside → axes they differ on. That outlives any release.</Callout>

## Open weights vs proprietary

The first fork is weight access. A proprietary model lives behind an API: you send requests to someone else's server and pay per token but never receive weights. An open-weights model is published: download, run on your hardware, fine-tune, work offline.

The choice is priorities: maximum capability and acceptable off-prem data - proprietary API; privacy, control, offline - open weights. Remember from [[fine-tuning-alignment]]: open weights enable custom fine-tuning but shift safety responsibility to whoever runs them.

## Axes models differ on

Within any family, tiers differ on measurable axes. Keep them as a selection checklist.

**Size (parameter count).** Roughly tracks capability and cost: more parameters usually means smarter but pricier and slower. Family tiers (Haiku → Sonnet → Opus) are a ladder on this axis.

**Context length.** How many tokens fit in one request (see [[embeddings-context]]). In 2026 top models reach hundreds of thousands to a million tokens - a whole codebase or thick document. Light tiers usually offer less.

**Multimodality.** Whether the model accepts more than text: images, audio, video. Most 2026 flagships take text and images; audio and video support varies by family.

**Price and latency.** Billing is per token - input (prompt) and output (reply), usually per million. Latency is time to first useful output. Light models are cheaper and faster; flagships cost more and run slower.

## Try it yourself

Reduce selection to a simple rule: estimate request cost from input and output token rates. Run the code, then change output token count and rates to feel how replies cost more than prompts.

<PyDemo
  title="Request cost from token rates"
  code={`def request_cost(in_tokens, out_tokens, in_price, out_price):
    return (in_tokens / 1_000_000) * in_price + (out_tokens / 1_000_000) * out_price

light = request_cost(2000, 500, 1.0, 5.0)
flagship = request_cost(2000, 500, 5.0, 25.0)
print(round(light, 6))
print(round(flagship, 6))`}
/>

This prints `0.0045` and `0.0225`: the same request on a flagship costs five times more than on a light model. One call is pennies; millions of production calls make tier choice a budget choice.

## How to pick a model for a task

Fold the axes into a simple algorithm. Start with the cheapest tier that works and escalate only when needed.

<Steps>
<li>Estimate task difficulty. Classification, extraction, short replies - light tier. Multi-step reasoning, hard code, agent loops - mid or flagship.</li>
<li>Check context needs. Whole codebase or long document - model with a large window.</li>
<li>Check inputs. Images, audio, video - multimodal model.</li>
<li>Weigh price and latency at your call volume. At scale tier differences in cost and speed become decisive.</li>
<li>Decide on privacy and control. Data cannot leave your perimeter or you need offline - consider open weights.</li>
</Steps>

## When intuition misleads

Always taking the most powerful model for headroom feels safe. On simple tasks a flagship rarely improves answers noticeably but always costs more and runs slower. At volume that is real money and latency without quality gain.

A second subtlety: benchmark leader is not automatically best for you. Benchmarks average many tasks; on your specific job a cheaper model may match a flagship. Test on your examples, not someone else's leaderboard.

## Common misconceptions

- **There is one best model.** There is a set of trade-offs: intelligence vs price, context vs speed. Best is always task- and budget-specific.
- **Open models are always worse.** They often trail closed frontier leaders slightly but suffice for many jobs while offering control and privacy APIs lack.
- **More context is always better.** Large windows cost more without making the model smarter; if the doc fits a small window, paying for a million tokens is waste.

<Detail summary="Why providers offer tiers instead of one model">

One strongest model for everything sounds simple. But classifying a review and refactoring a codebase need different intelligence, speed, and price. Running both on a flagship is wasteful when the first job is fine on a light tier for pennies and milliseconds. A tier ladder lets the market pick a point on the quality-cost curve per task; smart systems often mix tiers - cheap model for easy work, flagship for hard work.

</Detail>
