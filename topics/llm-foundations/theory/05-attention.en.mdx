# Attention: every token asks the others what matters

In the last section we said self-attention is where tokens exchange information. Now look inside. Take "the bank closed because it went bankrupt." To resolve "it," that token must look back and find "bank." Attention lets each token decide which other tokens to attend to and how strongly, then gather information from them. No links are hand-written - the model learns them.

By the end of this section you will understand query, key, and value, how scaled dot-product compatibility is computed, why scores are divided by sqrt(dimension), and how softmax turns scores into attention weights that sum to one.

## Three roles: query, key, value

From each token embedding the model forms three vectors by multiplying by three learned matrices. Think of a library search:

- **Query** - what the current token is looking for: "I need a masculine noun to the left."
- **Key** - how each token advertises itself: "I am a noun, masculine, I am bank."
- **Value** - what the token contributes if selected: the actual content.

The token compares its query to every other token key. Stronger match means more attention and a larger share of that token value in the result.

## Compatibility score is a dot product

How do you measure query-key match? Dot product. More aligned vectors yield a larger product and a higher score. One query against many keys gives one score per key.

```python
def dot(a, b):
    return sum(x * y for x, y in zip(a, b))

q = [1, 0]
keys = [[1, 0], [0, 1], [1, 1]]
scores = [dot(q, k) for k in keys]
print(scores)
```

This prints `[1, 0, 1]`: query `[1, 0]` aligns best with the first and third keys, not the second. These are raw scores - not yet probabilities and not necessarily positive.

## Why divide by sqrt(dimension)

There is a subtlety. Long vectors (models use tens or hundreds of dimensions) produce large dot products. Large inputs to softmax become almost one-hot - the distribution too sharp and gradients vanish in training. The fix is to divide each score by sqrt(key dimension). Hence scaled dot-product attention.

<Callout type="info">Dividing by sqrt(dimension) is not cosmetic. Without it at large dimension softmax almost always saturates on one token and the model loses soft attention across several tokens.</Callout>

## softmax turns scores into weights

Scaled scores pass through familiar softmax (see [[what-is-llm]]). Output: attention weights - nonnegative numbers summing to one. Each weight says what fraction of the corresponding value to take. The attention result for a token is the weighted sum of all values.

## Try it yourself

Build scaled dot-product attention for one query: compute scores, divide by sqrt(dimension), apply softmax. Run the code, then set `q` to `[0, 1]` and watch weights shift toward the second key.

<PyDemo
  title="Attention weights for one query"
  code={`import math

def attention_weights(q, keys):
    d = len(q)
    scores = [sum(a * b for a, b in zip(q, k)) / math.sqrt(d) for k in keys]
    m = max(scores)
    exps = [math.exp(s - m) for s in scores]
    total = sum(exps)
    return [e / total for e in exps]

q = [1, 0]
keys = [[1, 0], [0, 1]]
print([round(w, 3) for w in attention_weights(q, keys)])`}
/>

Output is `[0.67, 0.33]`: the query aligned more with the first key, so its weight is higher, but the second is not zero. Weights sum to one - the attention distribution. Multiply each weight by its value and sum to get the attention output for that token.

## When behavior surprises you

It seems like a token has one attention pattern. In each block there are several attention heads running in parallel with different Q/K/V matrices. One head may track syntax, another long-range links, another something we cannot name. Their outputs are combined - several views on context at once.

A second surprise is cost. Every token compares to every token, so compute grows roughly with the square of context length. That drives the cost of very long prompts from [[embeddings-context]] and motivates much attention acceleration research.

## Common misconceptions

- **Query, key, and value are different tokens.** They are three vectors of the same token from three matrices; confusing their roles is a common mistake.
- **Attention weights fully explain model decisions.** Weights show where information was gathered, not a complete or always honest why; literal interpretation is risky.
- **You can skip dividing by sqrt(dimension).** Without scaling softmax saturates and training suffers; it is part of the definition, not optional.

<Detail summary="Why sqrt(dimension) specifically">

If query and key components are independent with unit variance, variance of their dot product grows linearly with dimension d, and standard deviation like sqrt(d). Dividing by sqrt(d) keeps score scale stable across dimensions so softmax does not saturate. That is the factor from the original transformer paper.

</Detail>
