# Абстракция - это обещание, а не реализация

Представьте, что Вы пишете отчёт, который должен куда-то сохранять результат: в файл, в базу, в облако. Если внутри функции отчёта Вы прямо открываете файл, отчёт намертво связан с файловой системой. Захотите сохранять в S3 - придётся переписывать саму функцию отчёта, хотя считать она умеет ровно то же самое. Проблема не в коде сохранения, а в том, что отчёт *знает слишком много* о том, кто его получатель. Ему достаточно одного: «есть кто-то, кому я могу сказать `save(данные)`». Кто именно и как сохраняет - не его забота.

Вот это «есть кто-то, кто умеет `save`» и есть абстракция: контракт без реализации. В этом разделе Вы научитесь описывать такие контракты двумя способами, встроенными в Python - через `abc.ABC` (явное наследование) и через `typing.Protocol` (структурная типизация), - и поймёте, когда уместен каждый. Это фундамент, на котором стоит весь dependency injection: нельзя внедрить зависимость, пока Вы не описали, *какой формы* она должна быть.

## Ментальная модель: розетка и вилка

Абстракция работает как электрическая розетка. Розетка не знает и не хочет знать, что Вы в неё включите - чайник, ноутбук или зарядку. Она задаёт *форму контакта*: два штырька на определённом расстоянии под определённым напряжением. Любое устройство, чьи штырьки совпадают с этой формой, работает. Код, который «питается» от розетки, написан один раз и не меняется, когда Вы меняете чайник на ноутбук.

В коде форму контакта задаёт интерфейс: набор методов с известными именами и сигнатурами. Реализация - это конкретное устройство, которое подходит под форму. `Report` зависит от формы `Storage`, а `FileStorage`, `S3Storage` и `MemoryStorage` под эту форму подходят. Дальше весь вопрос в том, как в Python *записать* эту форму. Языки вроде Java делают это через `interface`. В Python инструментов два, и они устроены по-разному.

## Способ первый: abc.ABC и явный контракт

`abc.ABC` (abstract base class) описывает контракт как базовый класс, от которого реализации обязаны наследоваться. Метод, помеченный `@abstractmethod`, объявлен, но не реализован: подкласс *обязан* его переопределить, иначе Python не даст создать экземпляр.

```python
from abc import ABC, abstractmethod

class Storage(ABC):
    @abstractmethod
    def save(self, data: str) -> None:
        ...
```

`Storage` здесь не хранилище, а контракт хранилища: «у тебя должен быть метод `save(data)`». Сам он бесполезен для работы, и Python это подчёркивает - попытка `Storage()` сразу падает. Контракт существует, чтобы под него подстроились другие.

```python
class FileStorage(Storage):
    def save(self, data: str) -> None:
        self.last = data

fs = FileStorage()
fs.save("отчёт")
print(fs.last)
```

Печатается `отчёт`. `FileStorage` унаследовался от `Storage` и реализовал `save`, поэтому контракт выполнен и экземпляр создаётся. Связь *явная*: в скобках написано `(Storage)`, и любой, кто читает код, видит обещание.

<Callout type="info">
Абстрактный класс может содержать и обычные методы с реализацией - они достаются подклассам по наследству. Абстрактным делается только то, что подкласс *обязан* доопределить. Так общий код живёт в базе, а различия - в подклассах.
</Callout>

## Что произойдёт, если забыть метод

Главная польза `@abstractmethod` в том, что забыть реализацию нельзя - Python ловит ошибку в момент создания объекта, а не где-то глубоко при первом вызове.

```python
from abc import ABC, abstractmethod

class Storage(ABC):
    @abstractmethod
    def save(self, data: str) -> None: ...

class Broken(Storage):
    pass

Broken()
```

Падает с `TypeError: Can't instantiate abstract class Broken with abstract method save`. Подкласс заявил, что он `Storage`, но контракт не выполнил - и Python отказывается выдавать заведомо неисправный объект. Это ранняя, понятная ошибка вместо загадочного `AttributeError` через десять вызовов.

## Способ второй: Protocol и структурная типизация

`abc.ABC` требует явного наследования: реализация должна *знать* про контракт и написать `(Storage)` в скобках. Иногда это неудобно или невозможно - например, объект пришёл из чужой библиотеки, которую Вы не можете заставить наследоваться от вашего класса. Здесь выручает `typing.Protocol`: он проверяет не родословную, а *форму* - есть ли у объекта нужные методы, независимо от того, от кого он наследуется.

```python
from typing import Protocol

class Storage(Protocol):
    def save(self, data: str) -> None: ...

class S3Storage:
    def save(self, data: str) -> None:
        self.bucket = data
```

`S3Storage` нигде не упоминает `Storage` и ни от чего не наследуется, но *структурно* подходит под протокол: у него есть `save(data) -> None`. Для статического анализатора (mypy, pyright) и для аннотаций типов `S3Storage` - это `Storage`, потому что форма совпадает. Это и называют «утиной типизацией с проверкой типов»: если крякает как нужный интерфейс, значит, это он.

Прозаически разница такая: `ABC` спрашивает «ты мой потомок?», а `Protocol` спрашивает «ты умеешь то, что мне нужно?». Второй вопрос либеральнее и часто ближе к духу Python.

## Потрогайте контракты руками

Запустите код. Обе реализации - `FileStorage` через наследование и `S3Storage` без всякого наследования - спокойно ложатся в функцию `make_backup`, которая знает только про метод `save`. Теперь поэкспериментируйте: добавьте третий класс `MemoryStorage` со своим `save`, который, скажем, складывает данные в список, и передайте его в `make_backup`. Вы увидите, что `make_backup` не пришлось менять ни на строчку - в этом весь смысл.

<PyDemo
  title="Один потребитель, разные реализации"
  code={`from abc import ABC, abstractmethod

class Storage(ABC):
    @abstractmethod
    def save(self, data: str) -> None: ...

class FileStorage(Storage):
    def save(self, data):
        print("в файл:", data)

class S3Storage:
    def save(self, data):
        print("в S3:", data)

    storage.save(data)

make_backup(FileStorage(), "отчёт-1")
make_backup(S3Storage(), "отчёт-2")`}

Обратите внимание: `make_backup` принимает `storage` без всякой аннотации и работает с обоими классами. В рантайме Python просто вызывает `.save`, и ему всё равно, наследник это `Storage` или нет, - лишь бы метод существовал. Контракт здесь работает на уровне дисциплины и подсказок типов, а не как жёсткая проверка при каждом вызове.

## register заставляет isinstance врать

Можно зарегистрировать класс как «виртуального наследника» ABC без всякого наследования - через `Storage.register(SomeClass)`. После этого `isinstance(obj, Storage)` вернёт `True`, хотя `SomeClass` ничего не наследовал и контракт *не проверялся*.

```python
from abc import ABC, abstractmethod

class Storage(ABC):
    @abstractmethod
    def save(self, data: str) -> None: ...

class Sneaky:
    pass

Storage.register(Sneaky)
print(isinstance(Sneaky(), Storage))
```

Печатается `True` - несмотря на то, что у `Sneaky` нет метода `save` вообще. `register` даёт обещание за класс, но не заставляет его выполнить контракт. Поэтому `isinstance` против ABC - слабая гарантия: он говорит «этот класс объявил себя совместимым», а не «этот класс точно умеет всё нужное». Полагаться на сам вызов метода надёжнее, чем на проверку типа.

## Типичные ошибки

- **Создавать экземпляр абстрактного класса.** `Storage()` падает с `TypeError`. Абстрактный класс - это контракт, у него нет полезного поведения; работают только конкретные подклассы.
- **Думать, что `Protocol` проверяет форму в рантайме.** По умолчанию `Protocol` - инструмент статической проверки. `isinstance(obj, MyProtocol)` упадёт, пока Вы не пометите протокол декоратором `@runtime_checkable`, и даже тогда проверится лишь наличие методов, а не их сигнатуры.
- **Заводить абстракцию ради абстракции.** Если у контракта вечно будет ровно одна реализация, интерфейс - лишний слой. Абстракция окупается, когда реализаций две и больше или когда одну из них надо подменять в тестах (об этом - в концепте про тестирование).
- **Класть бизнес-логику в абстрактный метод.** `@abstractmethod` объявляет форму, а не поведение. Общую логику выносите в обычные методы базового класса, абстрактными оставляйте только точки расширения.

<Detail summary="Почему ABC ловит ошибку именно при создании экземпляра">

Механика спрятана в метаклассе `ABCMeta`. Когда Вы определяете класс с `@abstractmethod`, метакласс собирает имена всех нереализованных абстрактных методов в служебный атрибут `__abstractmethods__`. При попытке `Cls()` конструктор `object.__new__` смотрит на этот набор: если он непустой, создание блокируется с `TypeError`.

Подкласс «закрывает» абстрактный метод, переопределяя его, - тогда имя уходит из `__abstractmethods__` потомка. Когда набор опустел, экземпляр создаётся. Поэтому ошибка возникает не при объявлении класса `Broken` (он определяется без проблем) и не при первом вызове `save`, а ровно в точке `Broken()` - самый ранний момент, когда Python может гарантировать, что объект исправен.

</Detail>
