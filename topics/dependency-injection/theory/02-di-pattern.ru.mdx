# Кто создаёт зависимость - тот ею и связан

Класс `OrderService` оформляет заказы и для этого пишет письма клиенту. Самое естественное - создать почтовый клиент прямо внутри: `self.mailer = SmtpMailer("smtp.example.com")`. Код работает. Но теперь `OrderService` намертво приклеен к SMTP: чтобы прогнать его в тесте, придётся либо реально слать письма, либо лезть внутрь и менять конструктор. Поменять SMTP на отправку через очередь - снова правка `OrderService`, хотя оформление заказов не изменилось ни на йоту. Класс взял на себя лишнюю ответственность: он не только оформляет заказы, но и решает, *откуда взять* почтовый клиент.

Dependency injection отбирает у класса эту вторую ответственность. Зависимость не создаётся внутри - она *передаётся снаружи*, через конструктор или аргумент. В этом разделе Вы разберёте, как этот сдвиг устроен, почему его называют инверсией управления, и научитесь отличать класс, который держит свои зависимости в кулаке, от класса, который их принимает.

## Ментальная модель: создание против использования

У любой зависимости есть два момента жизни: её *создают* и её *используют*. Когда оба момента живут внутри одного класса, класс несвободен - он привязан к конкретному способу создания. DI разносит эти моменты: создание уходит наружу, к вызывающей стороне, а классу остаётся только использование.

Слева `OrderService` сам породил `SmtpMailer` и теперь от него неотделим. Справа `OrderService` получает уже готовый объект через конструктор и знает о нём только то, что у него есть нужный метод. Кто именно пришёл - `SmtpMailer`, заглушка или клиент очереди - решает вызывающая сторона. Класс перестал управлять созданием своих зависимостей; управление *инвертировалось* и ушло наверх.

## Сначала - как делать не надо

Чтобы прочувствовать пользу DI, посмотрите на тесно связанный вариант. `OrderService` создаёт почтовый клиент сам.

```python
class SmtpMailer:
    def send(self, to, text):
        print(f"SMTP -> {to}: {text}")

class OrderService:
    def __init__(self):
        self.mailer = SmtpMailer()

    def place(self, customer):
        self.mailer.send(customer, "Заказ принят")
```

Здесь `OrderService.__init__` жёстко вызывает `SmtpMailer()`. Никаким способом снаружи нельзя сказать «а пошли через другой канал» или «в тесте не шли вообще». Чтобы это изменить, придётся редактировать сам класс - а значит, рисковать сломать его логику оформления заказов ради дел, к оформлению не относящихся.

## Теперь - инъекция через конструктор

Заберём `new` наружу. `OrderService` больше не упоминает `SmtpMailer`; он принимает любой объект с методом `send`.

```python
class OrderService:
    def __init__(self, mailer):
        self.mailer = mailer

    def place(self, customer):
        self.mailer.send(customer, "Заказ принят")

service = OrderService(SmtpMailer())
service.place("ana@example.com")
```

Печатается `SMTP -> ana@example.com: Заказ принят`. Разница в одной строке конструктора, но последствия большие: теперь *вызывающий* решает, какой mailer подсунуть. Это и есть инъекция через конструктор - самый частый вид DI. Зависимость стала видимой в сигнатуре: достаточно прочитать `__init__`, чтобы понять, что классу нужно для работы.

<Callout type="tip">
Эвристика «спрятанной зависимости»: если внутри класса встречается создание объекта через `SomeClass(...)`, спросите, должен ли этот объект приходить снаружи. Объекты-значения (даты, координаты) создавать внутри нормально. Сервисы, клиенты, репозитории, соединения - кандидаты на инъекцию.
</Callout>

## Контракт делает инъекцию осмысленной

Инъекция раскрывает силу вместе с абстракцией из прошлого концепта. `OrderService` зависит не от `SmtpMailer`, а от *контракта* `Mailer`. Тогда любая реализация контракта подставляется без правок сервиса.

```python
from typing import Protocol

class Mailer(Protocol):
    def send(self, to: str, text: str) -> None: ...

class QueueMailer:
    def send(self, to, text):
        print(f"queue <- {to}: {text}")

service = OrderService(QueueMailer())
service.place("boris@example.com")
```

Печатается `queue <- boris@example.com: Заказ принят`. `OrderService` не изменился ни на строку, а канал доставки сменился полностью. Класс зависит от формы (`send(to, text)`), а конкретное устройство выбирает тот, кто собирает приложение. Так абстракция и инъекция работают в паре: абстракция описывает форму, инъекция доставляет конкретную реализацию.

## Инверсия управления одним предложением

«Инверсия управления» (inversion of control) звучит абстрактно, но смысл простой: раньше класс сам *управлял* тем, какие конкретные объекты использовать, - теперь это управление инвертировано, отдано наружу. Класс декларирует «мне нужен mailer», а *какой именно* - решает уровень выше. Голливудский принцип: «не звоните нам, мы позвоним вам» - класс не идёт за зависимостями сам, их ему приносят.

<PullQuote>Класс перестаёт спрашивать «где мне взять зависимость» и начинает говорить «дайте мне зависимость».</PullQuote>

DI - это конкретная техника (передать объект через конструктор или аргумент), а инверсия управления - принцип, который она реализует. В больших фреймворках управление берёт на себя DI-контейнер, который сам создаёт объекты и раздаёт их по зависимостям; FastAPI делает это через `Depends`, и об этом - следующий концепт.

## Потрогайте инъекцию руками

Запустите код: один и тот же `Notifier` сначала пишет в консоль, потом «отправляет SMS» - разница только в том, какой объект передан в конструктор. Попробуйте дописать третий класс `SilentChannel`, чей `send` ничего не делает (`pass`), передать его в `Notifier` и убедиться, что `notify` отрабатывает без ошибок и без вывода. Это ровно тот приём, которым в тестах зависимость подменяют на заглушку.

<PyDemo
  title="Одна логика, разные каналы"
  code={`class ConsoleChannel:
    def send(self, text):
        print("console:", text)

class SmsChannel:
    def send(self, text):
        print("sms:", text)

class Notifier:
    def __init__(self, channel):
        self.channel = channel

    def notify(self, text):
        self.channel.send(text)

Notifier(ConsoleChannel()).notify("привет")
Notifier(SmsChannel()).notify("привет")`}
/>

Заметьте, что `Notifier` ничего не знает про каналы заранее: он принимает `channel` и зовёт `send`. Добавить новый канал - значит написать новый класс с `send`, а `Notifier` не трогать. Логика и способ доставки разъехались, и каждый меняется независимо.

## Конструктор на семь аргументов - диагноз, а не неизбежность

DI без контракта легко выродить в передачу слишком многого. Соблазн «раз уж передаём, передадим заодно конфиг, логгер, базу, кэш» приводит к конструктору на десять аргументов. Это не делает код гибче - это перекладывает сложность на вызывающего, который теперь обязан собрать десяток объектов в правильном порядке.

```python
class Report:
    def __init__(self, db, cache, logger, mailer, config, clock, metrics):
        self.db = db
        self.cache = cache
        self.logger = logger
        self.mailer = mailer
        ...
```

Семь зависимостей в одном классе - почти всегда сигнал, что класс делает слишком много (нарушает single responsibility). Удивительная часть: лечит это не «менее строгий DI», а *разбиение класса*. Когда обязанностей становится по одной на класс, зависимостей у каждого - одна-две, и инъекция снова читается легко. DI не отменяет хороший дизайн, он его обнажает: раздутый список зависимостей - это диагностика, а не неизбежность.

## Типичные ошибки

- **Создавать зависимость внутри и «как бы инжектить» рядом.** Если конструктор принимает `mailer=None`, а потом делает `self.mailer = mailer or SmtpMailer()`, жёсткая связь вернулась через чёрный ход: дефолт снова привязывает класс к конкретному типу. Делайте зависимость обязательным аргументом или собирайте дефолты на уровне сборки приложения, а не внутри сервиса.
- **Инжектить конкретный класс вместо контракта.** Если в коде везде аннотировано `mailer: SmtpMailer`, подменить реализацию по-прежнему мешает тип. Зависьте от `Protocol`/ABC, тогда подойдёт любая реализация.
- **Путать DI с глобальными переменными.** Брать зависимость из глобального singleton - не инъекция: связь по-прежнему спрятана внутри и не видна в сигнатуре. Инъекция в том и состоит, что зависимость явно входит снаружи.
- **Инжектить всё подряд.** Значения-данные (число, строка, дата) и короткоживущие объекты создавать внутри нормально. Инъекция нужна для *сервисов*, которые имеют разные реализации или которые надо подменять.

<Detail summary="Три формы инъекции: конструктор, сеттер, метод">

Инъекция через конструктор (передать в `__init__`) - основная: объект приходит на свет уже укомплектованным и не существует в недонастроенном состоянии. Её и стоит выбирать по умолчанию.

Инъекция через сеттер (отдельный метод `set_mailer`) допускает смену зависимости после создания и нужна редко - например, когда зависимость по своей природе опциональна или меняется в течение жизни объекта. Платой идёт промежуточное состояние, где зависимость ещё не установлена.

Инъекция через аргумент метода (передать зависимость прямо в `place(customer, mailer)`) уместна, когда зависимость нужна одному методу и меняется от вызова к вызову. Именно эту форму, по сути, и использует `Depends` в обработчиках FastAPI - зависимость приходит как параметр функции на каждый запрос.

</Detail>
