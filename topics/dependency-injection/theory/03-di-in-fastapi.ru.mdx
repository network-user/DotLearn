# Depends: фреймворк собирает зависимости за вас

В прошлом концепте зависимости собирали вручную: `OrderService(SmtpMailer())`. Пока классов три, это терпимо. Но веб-приложение - это десятки обработчиков, каждому нужны соединение с базой, текущий пользователь, настройки, и часто одно зависит от другого: «пользователь» вычисляется из «токена», а «токен» - из заголовка запроса. Собирать эту цепочку руками в начале каждого обработчика - утомительно и легко ошибиться. FastAPI берёт сборку на себя: Вы *объявляете*, что обработчику нужно, а фреймворк сам вычисляет зависимости и подставляет их.

Делает он это через `Depends`. В этом разделе Вы разберёте, как `Depends` объявляет зависимость параметром функции, как FastAPI строит и кэширует граф зависимостей на каждый запрос, и как переопределить зависимость для тестов. FastAPI в pyodide не установлен, поэтому механику мы будем воспроизводить на чистом Python - но это та же самая механика, что внутри фреймворка.

## Ментальная модель: зависимость как параметр

Обычный аргумент функции FastAPI достаёт из запроса: из пути, из query-строки, из тела. `Depends(f)` говорит фреймворку другое: «прежде чем вызвать мой обработчик, вызови функцию `f`, а её результат передай мне как этот параметр». Зависимость перестаёт быть данными из запроса и становится *результатом другой функции*, которую фреймворк выполнит за Вас.

Обработчику `list_users` нужна база. Он не открывает соединение сам - он объявляет параметр `db = Depends(get_db)`. FastAPI видит это объявление, сначала вычисляет `get_db()` (которому, в свою очередь, могут понадобиться настройки), и только потом зовёт `list_users`, подставив готовый `db`. Управление снова инвертировано: обработчик декларирует потребность, а сборкой занимается фреймворк. Это тот же DI из прошлого концепта, только инъектор - не Вы, а FastAPI.

## Как это выглядит в коде FastAPI

Зависимость - это обычная функция. `Depends(get_db)` в сигнатуре обработчика связывает параметр с её результатом.

```python
from fastapi import Depends, FastAPI

app = FastAPI()

def get_db():
    return {"users": ["ana", "boris"]}

@app.get("/users")
def list_users(db = Depends(get_db)):
    return db["users"]
```

Когда приходит запрос на `/users`, FastAPI вызывает `get_db()`, получает словарь, передаёт его в `list_users` как `db` и возвращает `db["users"]`. Вы нигде не написали `get_db()` сами - фреймворк сделал это за Вас, потому что увидел `Depends(get_db)`. Обработчик зависит от *результата* `get_db`, а не от того, как именно база устроена.

## Зависимости вкладываются друг в друга

Главная сила `Depends` - в композиции. Зависимость сама может зависеть от другой зависимости, и FastAPI разрешит всю цепочку.

```python
def get_token(authorization: str = ""):
    return authorization.removeprefix("Bearer ")

def get_user(token: str = Depends(get_token)):
    return {"id": token, "name": "user-" + token}

@app.get("/me")
def me(user = Depends(get_user)):
    return user
```

Здесь `me` зависит от `get_user`, а `get_user` - от `get_token`. FastAPI выстраивает порядок сам: сначала `get_token` (он берёт заголовок из запроса), потом `get_user` (получает токен), потом `me` (получает пользователя). Вам не нужно писать `get_user(get_token(...))` - граф разворачивается автоматически. Каждая зависимость отвечает за один маленький шаг, а композицию собирает фреймворк.

<Callout type="info">
Зависимостью может быть и класс: `Depends(Settings)` вызовет `Settings()` и передаст экземпляр. Если у объекта есть `__call__`, фреймворк позовёт его как функцию. Так удобно инжектить настройки или сервисы с состоянием.
</Callout>

## Кэширование в пределах запроса

Если одну и ту же зависимость требуют сразу несколько мест в графе, FastAPI по умолчанию вычисляет её *один раз на запрос* и переиспользует результат. Иначе `get_db` в примере с пятью зависимыми открыл бы пять соединений - расточительно и опасно.

```python
def get_db():
    print("открыли соединение")
    return object()

@app.get("/report")
def report(a = Depends(get_db), b = Depends(get_db)):
    return a is b
```

Вернётся `True`, а «открыли соединение» напечатается *один раз*, хотя `get_db` указан дважды: `a` и `b` - это один и тот же объект. FastAPI кэширует результат внутри одного запроса по функции-зависимости. Если кэширование не нужно (например, Вам нужны два независимых объекта), его отключают через `Depends(get_db, use_cache=False)`.

## Потрогайте механику руками

FastAPI здесь не запустить, поэтому ниже - крошечный «движок» на чистом Python, который воспроизводит суть `Depends`: маркер `Depends` хранит функцию, а `solve` рекурсивно вычисляет зависимости и кэширует результаты в пределах одного «запроса». Запустите код и посмотрите, как `get_db` вызывается единожды, несмотря на две ссылки. Затем поменяйте `me` так, чтобы он зависел ещё и от `get_db` напрямую, и убедитесь, что соединение всё равно откроется один раз.

<PyDemo
  title="Мини-Depends на чистом Python"
  code={`class Depends:
    def __init__(self, func):
        self.func = func

def solve(func, cache):
    if func in cache:
        return cache[func]
    kwargs = {}
    for name, default in getattr(func, "deps", {}).items():
        kwargs[name] = solve(default.func, cache)
    result = func(**kwargs)
    cache[func] = result
    return result

def get_db():
    print("открыли соединение")
    return {"users": ["ana", "boris"]}

def me(a=Depends(get_db), b=Depends(get_db)):
    return a is b

me.deps = {"a": Depends(get_db), "b": Depends(get_db)}

cache = {}
print(solve(me, cache))`}
/>

Этот движок - грубое упрощение (настоящий FastAPI читает зависимости из аннотаций и обрабатывает данные запроса), но ключевая идея та же: рекурсивный обход графа функций с кэшем на запрос. Когда Вы поймёте, что `Depends` - это просто отложенный вызов функции, разрешаемый фреймворком, магия исчезает.

## Зависимость с yield работает как контекст

Зависимость с `yield` ведёт себя не как обычная функция, а как контекст: код *до* `yield` выполняется на входе, код *после* - после ответа, даже если обработчик упал с ошибкой. Это штатный способ открыть ресурс и гарантированно его закрыть.

```python
def get_db():
    db = {"conn": "open"}
    try:
        yield db
    finally:
        db["conn"] = "closed"
```

FastAPI передаёт в обработчик то, что отдано через `yield` (открытую базу), а строки после `finally` выполняет, когда запрос завершён, - закрывает соединение независимо от исхода. Удивительная деталь: в обработчик попадает *yielded-значение*, а не функция-генератор, и закрытие происходит не сразу после `yield`, а в конце жизненного цикла запроса. Так одна зависимость аккуратно управляет ресурсом от открытия до закрытия, и обработчику об этом думать не нужно.

## Типичные ошибки

- **Вызывать зависимость самому: `Depends(get_db())`.** В `Depends` передают *саму функцию* `get_db`, а не результат её вызова. С круглыми скобками Вы вычислите `get_db` один раз при старте приложения и зафиксируете единственный результат на все запросы - почти всегда не то, что нужно.
- **Делать тяжёлую работу прямо в теле обработчика, минуя `Depends`.** Если открыть соединение или проверить токен внутри обработчика, это нельзя переиспользовать и нельзя подменить в тесте. Выносите такие шаги в зависимости - тогда их можно композировать и переопределять.
- **Полагаться на кэш между запросами.** Кэш зависимости живёт один запрос. Состояние, которое должно переживать запросы (пул соединений, конфиг), создавайте на уровне приложения, а не в обычной зависимости.
- **Забывать `finally` в зависимости с `yield`.** Без `finally` ресурс не закроется, если обработчик бросит исключение. Освобождение ресурса всегда в `finally`.

<Detail summary="dependency_overrides: переопределение зависимости целиком">

У объекта `app` есть словарь `app.dependency_overrides`. Положив в него пару «исходная зависимость → подменная», Вы заставляете FastAPI везде вместо первой вызывать вторую - не трогая ни обработчики, ни сами зависимости.

```python
def get_db():
    return real_database

def fake_db():
    return {"users": []}

app.dependency_overrides[get_db] = fake_db
```

После этой строки каждый обработчик, объявивший `Depends(get_db)`, получит результат `fake_db`. Это штатный механизм тестирования FastAPI: в тесте базу подменяют на in-memory заглушку одной строкой, а после теста словарь очищают. По сути это та же инверсия управления - решение, *какую* реализацию подставить, принято снаружи, на уровне сборки тестового приложения.
