You wrote a service that places orders and sends the customer an email, and you create the mail client right inside the service. Everything works fine until the day you need to run the service in a test without sending a hundred real emails, or swap SMTP for a queue-based sender. It turns out that changing one detail forces you to rewrite a class that has nothing to do with that detail. This topic is about untying those knots: making code depend on the *shape* of a dependency, not its concrete implementation, and making it possible to swap the implementation from outside without touching the code itself.

**What you will learn to do.** Describe a dependency contract with `abc.ABC` and `typing.Protocol` and choose between them; pass dependencies into a class from outside instead of creating them inside; declare handler dependencies with `Depends` in FastAPI and understand how the framework assembles their graph; write tests where a real dependency is replaced with a fake and verify behavior, not internals.

**Topic route.** Four sections, each with a sandbox where code runs right in the browser:

1. **Abstractions**: describe the *shape* of a dependency as a contract without implementation, so code does not know too much about what stands behind it.
2. **The DI pattern**: take away a class's responsibility for creating a dependency and pass it in from outside; understand why this is called inversion of control.
3. **`Depends` in FastAPI**; delegate dependency assembly to the framework: declare what the handler needs, and it builds the graph itself.
4. **Testing through substitution**: plug in a fake dependency and verify that the service called it correctly.

**What you need beforehand.** Confidence with Python classes and inheritance helps (the whole topic rests on it), so review `python-oop` if those words sound unfamiliar. Nothing to install: Python runs in the browser, and we will cover everything else as we go.
