# Подзапросы: запрос внутри запроса

Вам уже встречались вопросы, на которые одним `WHERE` не ответить, потому что порог сравнения сам должен прийти из данных. «Кто получает больше средней зарплаты?» - чтобы написать условие, сначала нужно вычислить среднюю, а это отдельный запрос. «У кого из сотрудников нет ни одного заказа?» - сначала нужен список тех, у кого заказы есть. Каждый раз вопрос распадается на два: сначала посчитать что-то по таблице, потом отфильтровать по этому результату.

Подзапрос склеивает оба шага в один. Это обычный `SELECT`, вложенный в скобки внутрь другого запроса: его результат подставляется туда, где стоят скобки, как будто Вы заранее посчитали число или список и вписали его руками. К концу раздела Вы будете сравнивать строку со средним и максимумом по таблице, отбирать строки по списку из другого запроса, строить производные таблицы во `FROM` и распознавать два честных капкана: скалярный подзапрос, который вдруг вернул несколько строк, и `NOT IN`, который из-за одного NULL молча возвращает пустоту.

## Изнутри наружу: ментальная модель

Запрос с подзапросом читается снаружи внутрь, а выполняется наоборот - изнутри наружу. Сначала движок целиком вычисляет внутренний `SELECT`, получает из него значение (или список значений), подставляет это значение на место скобок и только потом выполняет внешний запрос так, будто никакого вложения и не было. Внутренний запрос для внешнего - это просто константа, которую посчитали чуть раньше.

Эта картинка - ключ ко всему разделу. Если когда-нибудь подзапрос покажется загадкой, мысленно выполните его отдельно, запишите результат на бумажке и вставьте это значение во внешний запрос. Загадка исчезнет: останется обычный запрос с конкретным числом или списком в условии.

## Скалярный подзапрос: одно число из данных

Скалярный подзапрос возвращает ровно одну строку и одну колонку - то есть единственное значение. Его можно поставить везде, где синтаксис ждёт число: в правую часть сравнения, в `SELECT`, в выражение. Чаще всего он отвечает на вопрос «как эта строка соотносится с агрегатом по всей таблице».

Работаем с таблицей сотрудников. Запустите запрос: внутренний `SELECT AVG(salary)` посчитает среднюю зарплату, а внешний оставит тех, кто получает строго больше.

<SideSql
  title="Кто получает больше средней зарплаты"
  fixture={`
    CREATE TABLE employee (
      id INTEGER PRIMARY KEY,
      name TEXT NOT NULL,
      department TEXT NOT NULL,
      salary INTEGER NOT NULL
    );
    INSERT INTO employee VALUES
      (1, 'Ada',   'eng',     9000),
      (2, 'Boris', 'eng',     6000),
      (3, 'Vera',  'sales',   5000),
      (4, 'Grace', 'sales',   7000),
      (5, 'Igor',  'support', 4000);
  `}
  query={`
    SELECT name, salary
    FROM employee
    WHERE salary > (SELECT AVG(salary) FROM employee)
    ORDER BY salary DESC;
  `}
/>

Средняя по пяти зарплатам - `(9000+6000+5000+7000+4000)/5 = 6200`. Условие превращается в `salary > 6200`, и в ответе остаются Ada (9000) и Grace (7000). Borisа с его 6000 средняя обходит ровно на 200 - сравнение строгое, и он не проходит.

Тот же приём работает с `MAX` и `MIN`. Чтобы найти сотрудника с самой высокой зарплатой, сравните каждую строку с максимумом по таблице:

```sql
SELECT name, salary
FROM employee
WHERE salary = (SELECT MAX(salary) FROM employee);
```

Скобки с `MAX(salary)` дают `9000`, условие становится `salary = 9000`, в ответе одна Ada. Заметьте: подзапрос обращается к той же таблице `employee`, и это нормально - внутренний и внешний `SELECT` работают с ней независимо.

## Список значений: подзапрос в IN

Скалярный подзапрос годится, когда из данных нужно одно число. Но часто нужен целый список: «оставь строки, чей ключ есть среди вот этих». Для этого подзапрос ставят справа от `IN` - и он возвращает уже не одно значение, а колонку значений.

Добавим таблицу заказов и найдём сотрудников, оформивших хотя бы один заказ. Внутренний запрос собирает множество `employee_id` из `orders`, внешний оставляет сотрудников, чей `id` в это множество попал.

<SideSql
  title="Сотрудники, у которых есть заказы"
  fixture={`
    CREATE TABLE employee (
      id INTEGER PRIMARY KEY,
      name TEXT NOT NULL
    );
    INSERT INTO employee VALUES
      (1, 'Ada'), (2, 'Boris'), (3, 'Vera'), (4, 'Grace');

    CREATE TABLE orders (
      id INTEGER PRIMARY KEY,
      employee_id INTEGER,
      amount INTEGER NOT NULL
    );
    INSERT INTO orders VALUES
      (1, 1, 500),
      (2, 1, 300),
      (3, 3, 900);
  `}
  query={`
    SELECT name
    FROM employee
    WHERE id IN (SELECT employee_id FROM orders)
    ORDER BY name;
  `}
/>

Внутренний запрос возвращает `1, 1, 3` - повторы внутри `IN` роли не играют. Внешний оставляет Ada (id 1) и Vera (id 3); Boris и Grace в списке заказчиков не значатся. Попробуйте сами выполнить только внутренний запрос: сотрите внешнюю часть и оставьте `SELECT employee_id FROM orders;`. Вы увидите ровно тот список, который дальше подставляется в `IN` - та самая «бумажка» из ментальной модели.

<SideSql
  title="Выполни сначала внутренний запрос отдельно"
  editable
  live
  fixture={`
    CREATE TABLE employee (
      id INTEGER PRIMARY KEY,
      name TEXT NOT NULL
    );
    INSERT INTO employee VALUES
      (1, 'Ada'), (2, 'Boris'), (3, 'Vera'), (4, 'Grace');

    CREATE TABLE orders (
      id INTEGER PRIMARY KEY,
      employee_id INTEGER,
      amount INTEGER NOT NULL
    );
    INSERT INTO orders VALUES
      (1, 1, 500),
      (2, 1, 300),
      (3, 3, 900);
  `}
  query={`
    SELECT employee_id FROM orders;
  `}
/>

Поменяйте запрос на полный, с внешним `SELECT ... WHERE id IN (...)`, и сравните: множество в `IN` ровно то, что Вы только что видели отдельным запросом.

## Производная таблица: подзапрос во FROM

Подзапрос умеет стоять не только в `WHERE`, но и на месте таблицы во `FROM`. Тогда его результат - временная таблица, существующая только на время запроса. Её называют производной таблицей (derived table), и работают с ней как с обычной: выбирают колонки, фильтруют, соединяют.

Пригодится это, когда нужно отфильтровать по результату агрегации. Допустим, мы посчитали суммарную выручку по отделам и хотим оставить только отделы с выручкой больше 1000. Сначала во внутреннем запросе считаем суммы, затем во внешнем фильтруем готовые суммы.

<SideSql
  title="Отделы с выручкой больше 1000"
  fixture={`
    CREATE TABLE sale (
      id INTEGER PRIMARY KEY,
      department TEXT NOT NULL,
      amount INTEGER NOT NULL
    );
    INSERT INTO sale VALUES
      (1, 'eng',   400),
      (2, 'eng',   800),
      (3, 'sales', 300),
      (4, 'sales', 200),
      (5, 'support', 1500);
  `}
  query={`
    SELECT department, total
    FROM (
      SELECT department, SUM(amount) AS total
      FROM sale
      GROUP BY department
    ) AS dept_totals
    WHERE total > 1000
    ORDER BY total DESC;
  `}
/>

Внутренний запрос сворачивает продажи в три строки: eng - 1200, sales - 500, support - 1500. Внешний `WHERE total > 1000` оставляет eng и support. То же самое можно было сделать через `HAVING`, но производная таблица выручает там, где `HAVING` не дотягивается: например, когда нужно отфильтровать по агрегату от агрегата или соединить готовую сводку с другой таблицей.

<Callout type="warning" title="Производной таблице обязательно нужен алиас">
В большинстве движков (включая PostgreSQL и MySQL) подзапрос во `FROM` обязан иметь имя: `... ) AS dept_totals`. Без алиаса запрос не разберётся, как обращаться к этой таблице, и упадёт с синтаксической ошибкой. SQLite мягче и алиас не требует, но привычку давать имя стоит выработать сразу - код станет переносимым и читаемым.
</Callout>

## Коррелированный подзапрос: внутренний смотрит наружу

Во всех примерах выше внутренний запрос был самодостаточным: его можно было выполнить отдельно. Но подзапрос умеет ссылаться на колонки внешнего запроса - тогда он называется коррелированным. Платой за это становится повторное выполнение: такой подзапрос пересчитывается заново для каждой строки внешней таблицы.

Честный пример: оставить сотрудников, чья зарплата выше средней по их собственному отделу. Средняя у инженеров одна, у продаж другая, поэтому одного числа на всю таблицу мало - порог зависит от строки.

<SideSql
  title="Зарплата выше средней по своему отделу"
  fixture={`
    CREATE TABLE employee (
      id INTEGER PRIMARY KEY,
      name TEXT NOT NULL,
      department TEXT NOT NULL,
      salary INTEGER NOT NULL
    );
    INSERT INTO employee VALUES
      (1, 'Ada',   'eng',   9000),
      (2, 'Boris', 'eng',   6000),
      (3, 'Vera',  'eng',   3000),
      (4, 'Grace', 'sales', 7000),
      (5, 'Igor',  'sales', 5000);
  `}
  query={`
    SELECT name, department, salary
    FROM employee AS e
    WHERE salary > (
      SELECT AVG(salary)
      FROM employee
      WHERE department = e.department
    )
    ORDER BY name;
  `}
/>

Обратите внимание на `e.department` внутри подзапроса: он ссылается на отдел текущей строки внешней таблицы. Для строки Ada подзапрос считает среднюю по eng - `(9000+6000+3000)/3 = 6000`, и `9000 > 6000` проходит. Для Borisа порог тот же 6000, а `6000 > 6000` - нет. У sales средняя `(7000+5000)/2 = 6000`: Grace с 7000 проходит, Igor с 5000 - нет. В ответе Ada и Grace.

<Callout type="info" title="Цена коррелированного подзапроса">
Внутренний запрос здесь выполняется не один раз, а по разу на каждую строку `employee` - пять раз для пяти сотрудников. На маленькой таблице это незаметно, на миллионах строк - ощутимо. Часто ту же задачу решает оконная функция или предварительная агрегация в производной таблице. Коррелированный подзапрос - честный инструмент, но держите в голове, что он стоит дороже обычного.
</Callout>

## Капкан NOT IN и NULL

Здесь Вас ждёт сюрприз, который ловит даже опытных. Логичный способ найти сотрудников без заказов - перевернуть прошлый пример через `NOT IN`. Но если в списке заказчиков затешется хоть один NULL, запрос вернёт пустоту - не строки без совпадения, а вообще ничего.

<SideSql
  title="NOT IN и единственный NULL"
  editable
  live
  fixture={`
    CREATE TABLE employee (
      id INTEGER PRIMARY KEY,
      name TEXT NOT NULL
    );
    INSERT INTO employee VALUES
      (1, 'Ada'), (2, 'Boris'), (3, 'Vera'), (4, 'Grace');

    CREATE TABLE orders (
      id INTEGER PRIMARY KEY,
      employee_id INTEGER,
      amount INTEGER NOT NULL
    );
    INSERT INTO orders VALUES
      (1, 1, 500),
      (2, 3, 900),
      (3, NULL, 250);
  `}
  query={`
    SELECT name
    FROM employee
    WHERE id NOT IN (SELECT employee_id FROM orders);
  `}
/>

Ожидаешь Borisа и Grace - у них заказов нет. Получаешь пустую таблицу. Причина в трёхзначной логике SQL. Подзапрос вернул `1, 3, NULL`. Условие `id NOT IN (1, 3, NULL)` раскрывается в `id <> 1 AND id <> 3 AND id <> NULL`. Сравнение `id <> NULL` не истинно и не ложно - оно `UNKNOWN`, и `AND` с `UNKNOWN` уже никогда не станет полностью истинным. Поэтому ни одна строка не проходит.

Лекарство - отсечь NULL прямо в подзапросе: добавьте `WHERE employee_id IS NOT NULL`, и Boris с Grace вернутся. Поменяйте запрос в песочнице выше и убедитесь. Ещё надёжнее переписать условие через `NOT EXISTS`, который к NULL равнодушен:

```sql
SELECT name
FROM employee e
WHERE NOT EXISTS (
  SELECT 1 FROM orders o WHERE o.employee_id = e.id
);
```

`NOT EXISTS` спрашивает «не нашлось ли ни одной совпадающей строки», и NULL в `employee_id` просто не совпадёт с `e.id` - ловушки нет.

## Когда JOIN яснее подзапроса

Подзапрос с `IN` и `JOIN` нередко решают одну задачу. Сравните два способа найти сотрудников с заказами:

<Compare
  leftTitle="Подзапрос с IN"
  rightTitle="JOIN"
  left="SELECT name FROM employee WHERE id IN (SELECT employee_id FROM orders). Читается как вопрос «есть ли id в этом списке?». Не размножает строки: каждый сотрудник встретится один раз."
  right="SELECT DISTINCT e.name FROM employee e JOIN orders o ON o.employee_id = e.id. Соединяет строки и даёт доступ к колонкам orders, но без DISTINCT размножит сотрудника по числу его заказов."
  verdict="Нужна только проверка наличия - берите IN или EXISTS, они выражают намерение прямее. Нужны колонки из второй таблицы - берите JOIN."
/>

Разница не косметическая. `JOIN` соединяет строки и потому может размножить сотрудника: у Ada два заказа - в результате `JOIN` без `DISTINCT` она появится дважды. Подзапрос с `IN` отвечает на вопрос «да или нет» и каждого сотрудника покажет ровно один раз. Если из второй таблицы нужны колонки (сумма заказа, дата) - без `JOIN` не обойтись. Если нужно лишь отобрать строки по факту наличия - подзапрос выражает намерение чище.

## Типичные ошибки

- **Скалярный подзапрос вернул несколько строк.** `WHERE salary = (SELECT salary FROM employee WHERE department = 'eng')` упадёт или поведёт себя непредсказуемо, если в отделе eng больше одного человека: на месте одного значения движок получил список. Лекарство - либо сделать подзапрос действительно скалярным (`MAX`, `MIN`, `AVG`, или `LIMIT 1`), либо сменить `=` на `IN`. Ошибка случается потому, что справа от `=` синтаксис ждёт одно значение, а данные дали несколько.
- **NOT IN с NULL в подзапросе.** Один NULL в результате подзапроса превращает весь `NOT IN` в пустоту - см. раздел выше. Либо добавьте `IS NOT NULL` внутрь подзапроса, либо перепишите через `NOT EXISTS`. С обычным `IN` (без `NOT`) NULL такой беды не делает.
- **Забытый алиас у производной таблицы.** Подзапрос во `FROM` без имени (`FROM (SELECT ...) WHERE ...`) во многих движках - синтаксическая ошибка. Всегда давайте ему имя через `AS`, даже если SQLite готов простить его отсутствие.

<Detail summary="Подзапрос против JOIN: что выберет планировщик">
Можно подумать, что коррелированный подзапрос всегда медленнее эквивалентного `JOIN`, потому что выполняется по разу на строку. На практике современные планировщики (PostgreSQL, недавние версии SQLite) нередко переписывают `IN`-подзапросы и даже коррелированные формы в полу-соединение (semi-join) с тем же планом, что и у явного `JOIN`. То есть форма запроса и фактический план выполнения - не одно и то же. Поэтому выбирайте подзапрос или `JOIN` исходя из читаемости, а к плану обращайтесь через `EXPLAIN QUERY PLAN`, только когда упёрлись в производительность.
</Detail>
