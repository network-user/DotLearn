# SELECT: reading exactly what you need from a table

The `users` table holds five thousand rows, and you need three of them: adult users from Poland who have an email on file. Pulling everything down and filtering in application code is slow and wasteful; the database can hand you exactly the slice you want on its own. That work is done by the pair `SELECT` and `WHERE` - two words you will type more often than any others across an entire career working with data.

By the end of this article you will pick out the columns you need, select rows by compound conditions, search by pattern with `LIKE`, drop duplicates with `DISTINCT` - and understand why comparing against `NULL` quietly returns "nothing" and how to get around it.

## Two cuts through one table

A table is a rectangle: rows run down, columns run across. Any read query does just two things to that rectangle: it throws away the rows you don't want and keeps the columns you do. Everything else in `SELECT` is detail on top of those two moves.

<MarginNote label="History">The SEQUEL language appeared at IBM in 1974. The words SELECT, FROM, and WHERE have not changed since.</MarginNote>

Take a concrete query: `SELECT name, age FROM users WHERE age >= 18`. The condition `WHERE age >= 18` walks the rows top to bottom and keeps only the adults - that is the horizontal cut. The list `name, age` walks the columns and keeps only the name and age of each surviving row - that is the vertical cut. The two operations are independent: you can filter rows and return every column, you can return one column of every row, and most often you do both at once.

These cuts have established names - you'll meet them in the docs and in other people's code. Keeping rows (the horizontal cut) is called *selection*; keeping columns (the vertical cut) is called *projection*; and the condition by which `WHERE` decides a row's fate is, in the jargon, a *predicate*. There's no need to memorize the words: what matters is that underneath them sit just "keep the rows I want" and "keep the columns I want".

In the figure the condition `age >= 18` lets Anna and Eva through and stops Bob and Ian. Notice: a row either passes whole or is discarded whole. `WHERE` never lets a row through "partially" - it answers only the question "is this row alive or not".

The second key fact: the order of execution does not match the order of writing. You write `SELECT` as the first word, but the engine runs it last.

First `FROM` names the data source, then `WHERE` discards the unwanted rows, and only after that does `SELECT` project the survivors onto the columns you want. A practical consequence follows from this order: you cannot reference column aliases declared in `SELECT` from inside `WHERE` - at the moment of filtering they do not exist yet.

## Building a query step by step

Every example in this article works against one table:

```sql
CREATE TABLE users (id INTEGER, name TEXT, age INTEGER, country TEXT, email TEXT);
INSERT INTO users VALUES
  (1, 'Anna', 25, 'PL', 'anna@example.com'),
  (2, 'Bob', 17, 'US', NULL),
  (3, 'Eva', 30, 'PL', 'eva@example.com'),
  (4, 'Ian', 14, 'LV', 'ian@test.org'),
  (5, 'Olga', 41, 'LV', NULL);
```

The minimal query names the columns and the table:

```sql
SELECT name, age FROM users;
```

The asterisk returns all columns at once:

```sql
SELECT * FROM users;
```

For quick experiments `*` is convenient, but in application code list the columns explicitly: a table's schema changes over time, and a query with a star will one day drag a column nobody expected into the application.

Filtering is added with a single condition. The usual comparison operators are available: `=`, `<>`, `<`, `<=`, `>`, `>=`.

```sql
SELECT name FROM users WHERE country <> 'PL';
```

The query returns Bob, Ian, and Olga - everyone whose country is not Poland. Conditions are combined with `AND`, `OR`, and `NOT`:

```sql
SELECT name, age, country FROM users
WHERE age >= 18 AND (country = 'PL' OR country = 'LV');
```

Here the parentheses are not decoration. `AND` binds more tightly than `OR`, so without the parentheses the condition would read as "adult Poles, plus anyone at all from Latvia" - and fourteen-year-old Ian would slip into the result. Parentheses are free: add them at the slightest doubt about precedence.

When the exact value is unknown, you need a pattern search. The `LIKE` operator understands two special characters: `%` matches any sequence of characters (including the empty one), `_` matches exactly one character.

```sql
SELECT name, email FROM users WHERE email LIKE '%@example.com';
```

The query finds Anna and Eva: their addresses end in `@example.com`. The pattern `'a%'` would find addresses starting with `a`, and `'____@%'` would find addresses with exactly four characters before the `@`.

The last tool of this section is `DISTINCT`. It removes duplicate rows from the result:

```sql
SELECT DISTINCT country FROM users;
```

Instead of five rows you get three: `PL`, `US`, `LV`. Importantly, `DISTINCT` applies to the whole result row, not to a single column: `SELECT DISTINCT country, age` removes only fully matching "country + age" pairs.

## NULL: logic with three answers

In ordinary boolean logic a comparison returns true or false. In SQL there are three answers: `TRUE`, `FALSE`, and `NULL` - "unknown". Any comparison with `NULL` yields `NULL`: both `email = NULL`, and `email <> NULL`, and even `NULL = NULL`. The logic is honest: `NULL` means "there is no value" or "the value is unknown", and comparing two unknowns is like asking whether two numbers nobody named are equal.

`WHERE` lets a row through only on `TRUE`. `NULL` is not `TRUE`, so a row with `NULL` in the compared column will pass neither the condition nor its negation.

<SideSql
  title="The NULL trap"
  editable
  live
  fixture={`CREATE TABLE users (id INTEGER, name TEXT, age INTEGER, country TEXT, email TEXT);
INSERT INTO users VALUES
  (1, 'Anna', 25, 'PL', 'anna@example.com'),
  (2, 'Bob', 17, 'US', NULL),
  (3, 'Eva', 30, 'PL', 'eva@example.com'),
  (4, 'Ian', 14, 'LV', 'ian@test.org'),
  (5, 'Olga', 41, 'LV', NULL);`}
  query="SELECT name FROM users WHERE email = NULL;"
/>

Zero rows - even though Bob and Olga really do have no email. Replace `= NULL` with `IS NULL` and run again: both rows appear. For the opposite check there is `IS NOT NULL`.

<Callout type="warning" title="NULL equals nothing">
Not even itself: the expression NULL = NULL returns NULL, not TRUE. The absence of a value is checked only with the IS NULL and IS NOT NULL operators.
</Callout>

The second surprise is more modest, but it matters for application code: an empty result is not an error. The query `SELECT * FROM users WHERE age > 100` on our table returns zero rows and finishes successfully. "There are no such rows" is a complete answer to the question asked, and the application must be ready for an empty list just as for a non-empty one.

<PullQuote>An empty result is an answer, not an error.</PullQuote>

The third surprise concerns case. In SQLite the `=` operator is case-sensitive: `'anna' = 'Anna'` is false. But `LIKE` ignores case by default for Latin letters: `'Anna' LIKE 'anna'` is true. The same literal in `=` and in `LIKE` can find a different number of rows. In other database systems the behavior differs: in PostgreSQL `LIKE` is case-sensitive, and for a case-insensitive search there is a separate operator `ILIKE`. Before relying on case, check how your engine behaves.

<Detail summary="Truth table for three-valued logic">

| Expression | Result |
| --- | --- |
| `TRUE AND NULL` | `NULL` |
| `FALSE AND NULL` | `FALSE` |
| `TRUE OR NULL` | `TRUE` |
| `FALSE OR NULL` | `NULL` |
| `NOT NULL` | `NULL` |

The pattern: if the result is determined by the known operand (`FALSE AND` anything is false, `TRUE OR` anything is true), the unknown does not get in the way. But if the answer depends on the unknown operand, the result stays unknown.

</Detail>

## Common mistakes

**`==` instead of `=`.** In SQL equality is tested with a single `=` sign: there is no assignment inside a query, so the second sign is not needed. SQLite, being lenient, accepts `==` too, but most database systems answer with a syntax error. It is a reflex from programming languages; retrain yourself to `=` right away.

**Mixed-up quotes.** Single quotes are for string values: `'Anna'`. Double quotes are for table and column names: `"name"`. The query `WHERE name = "Anna"` fails in a strict database system with the error "column Anna does not exist". SQLite is sneakier: when it cannot find such a column, it silently treats `"Anna"` as a string, and the query "works" right up until you move it to another database.

**Comparing against `NULL` with `=`.** The symptom: a condition "does not find" rows that are definitely in the table. The cause is the three-valued logic from the previous section; the cure is to switch to `IS NULL` or `IS NOT NULL`.

**`WHERE` with aggregates.** Sooner or later you will write `WHERE COUNT(*) > 5` and get an error. `WHERE` runs before grouping and sees individual rows, not group totals; to filter groups there is `HAVING`, and it comes up in the section on `GROUP BY`. For now it is enough to remember: `WHERE` is about rows, `HAVING` is about groups.
