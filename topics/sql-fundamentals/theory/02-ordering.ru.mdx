# Порядок появляется только по запросу: ORDER BY и LIMIT

На локальной базе запрос `SELECT * FROM tracks` неделями возвращает строки ровно в том порядке, в каком Вы их вставляли. Интерфейс выглядит аккуратно, список «как будто отсортирован», и рука не поднимается писать лишнее слово в запросе. Потом данных становится больше, движок выбирает другой план выполнения или перестраивает страницы хранения - и список перемешивается. Код не менялся; изменилось то, на что Вы молча полагались.

В этой главе Вы научитесь управлять порядком явно: сортировать по одному и нескольким ключам в обе стороны, отрезать верхушку списка через `LIMIT`, листать результат через `OFFSET`, сортировать по выражениям и алиасам, а заодно предскажете, куда упадут `NULL` и что произойдёт со строками-близнецами. К концу фраза «покажи десять самых популярных» будет переводиться в SQL без раздумий.

## Таблица - это множество

Реляционная таблица не хранит обещаний о порядке. Строки в ней - множество: у множества есть состав, но нет «первого» и «последнего» элемента. Когда Вы пишете `SELECT` без `ORDER BY`, движок возвращает строки так, как ему дешевле прямо сейчас: в порядке физического хранения, в порядке индекса, выбранного для `WHERE`, или как-то ещё. Если вывод выглядит отсортированным - это побочный эффект текущего плана выполнения, и обновление движка, `VACUUM` или новый индекс вправе его поменять.

<PullQuote>Таблица - это множество; порядок существует только в результате запроса, и только если Вы его попросили.</PullQuote>

Отсюда мысленная модель: `ORDER BY` - последний содержательный шаг конвейера. Сначала `FROM` и `WHERE` собирают строки, затем `SELECT` формирует колонки, и только готовый результат выстраивается по ключу сортировки. `LIMIT` режет ещё позже - уже отсортированный список. Посмотрите на рисунок: слева строки лежат в произвольном порядке хранения, стрелка `ORDER BY plays DESC` выстраивает их по убыванию прослушиваний, а пунктирная линия отсекает всё, что не попало в тройку.

## Данные главы

Все примеры работают с одной таблицей - мини-каталогом музыкального сервиса:

```sql
CREATE TABLE tracks (
  id INTEGER PRIMARY KEY,
  title TEXT NOT NULL,
  artist TEXT NOT NULL,
  plays INTEGER NOT NULL,
  rating REAL
);
INSERT INTO tracks (id, title, artist, plays, rating) VALUES
  (1, 'Drive', 'Neon Tide', 412, 4.6),
  (2, 'Planes', 'Neon Tide', 268, NULL),
  (3, 'Static', 'Glass Harbor', 597, 4.1),
  (4, 'Ember', 'Glass Harbor', 597, 4.9),
  (5, 'Orbit', 'Satellite Era', 173, NULL),
  (6, 'Glow', 'Satellite Era', 341, 3.8);
```

Запомните две детали, они выстрелят позже: у «Static» и «Ember» одинаковое число прослушиваний, а у «Planes» и «Orbit» нет рейтинга.

## Один ключ: ASC и DESC

Простейшая форма - один ключ сортировки:

```sql
SELECT title, plays FROM tracks ORDER BY plays;
```

Направление по умолчанию - по возрастанию, поэтому первым идёт «Orbit» со 173 прослушиваниями. `ASC` можно написать явно, но обычно его опускают. Чтобы перевернуть список, добавьте `DESC` после ключа:

```sql
SELECT title, plays FROM tracks ORDER BY plays DESC;
```

<MarginNote label="Коллации">Текст SQLite по умолчанию сравнивает побайтово: все заглавные латинские буквы меньше строчных, поэтому «Zoe» встанет раньше «anna». Регистронезависимую сортировку даёт COLLATE NOCASE.</MarginNote>

Ключом может быть любая колонка, не только числовая. Числа сравниваются по величине, текст - по правилам коллации, даты в формате ISO 8601 сортируются как строки, и это одна из причин хранить их именно так.

## Несколько ключей

Один ключ часто не определяет порядок полностью. «Отсортируй по исполнителю, а внутри исполнителя - по популярности» - это два ключа через запятую:

```sql
SELECT artist, title, plays
FROM tracks
ORDER BY artist ASC, plays DESC;
```

Второй ключ просыпается только тогда, когда первый не смог различить строки: внутри «Neon Tide» трек «Drive» (412 прослушиваний) встанет раньше «Planes» (268). Направление задаётся для каждого ключа отдельно - `artist` по возрастанию, `plays` по убыванию - и ключи не влияют друг на друга. Ключей может быть сколько угодно, но каждый следующий работает во всё более узкой зоне: только там, где все предыдущие совпали.

## LIMIT и OFFSET: срез готового списка

`LIMIT` ограничивает число строк в ответе, `OFFSET` пропускает первые несколько. Вместе они дают классическую пагинацию:

```sql
SELECT title, plays
FROM tracks
ORDER BY plays DESC
LIMIT 2 OFFSET 2;
```

Это «вторая страница» списка по две строки: запрос пропустит двух лидеров и вернёт «Drive» и «Glow». Срез выполняется после сортировки, поэтому формула страницы с номером N - `OFFSET (N - 1) * размер`. Обратите внимание на сам порядок рассуждений: сначала Вы фиксируете полный порядок списка, и только потом решаете, какое окно из него показать. Запрос делает ровно то же самое, и если порядок не зафиксирован, окно вырезается из случайной перестановки.

<Callout type="warning" title="LIMIT без ORDER BY">
LIMIT без ORDER BY возвращает N произвольных строк. На тестовой базе они выглядят осмысленно, на проде - как повезёт. Если в запросе есть LIMIT, в нём почти всегда должен быть и ORDER BY.
</Callout>

## Топ-N: самый частый паттерн

Сочетание `ORDER BY ... DESC LIMIT N` встречается так часто, что заслужило собственное имя - топ-N. Десять самых продаваемых товаров, пять последних заказов, три самых популярных трека - один и тот же скелет запроса. Запустите его вживую:

А теперь удалите `ORDER BY`, оставив `LIMIT 3`: запрос продолжит работать, но какие именно три строки он вернёт - уже не Ваше решение.

## Выражения и алиасы

Ключом сортировки может быть не только колонка, но и выражение. Например, рейтинговая формула, смешивающая прослушивания и оценку:

```sql
SELECT title, plays + 100 * coalesce(rating, 0) AS score
FROM tracks
ORDER BY score DESC;
```

`ORDER BY` видит алиасы из `SELECT`, поэтому `score` можно использовать сразу; то же выражение можно написать и прямо в `ORDER BY`, не вынося в список колонок. Заметьте: «Ember» обгоняет «Static» - при равных прослушиваниях исход решает рейтинг. SQLite позволяет сортировать и по номеру колонки (`ORDER BY 2`), но такая запись ломается при первом же изменении списка `SELECT` - избегайте её.

## Куда падает NULL

У «Planes» и «Orbit» рейтинга нет. Куда их поставит `ORDER BY rating`? SQLite считает `NULL` меньше любого значения: при сортировке по возрастанию неизвестные рейтинги всплывают в начало списка, при `DESC` - опускаются в конец. Это выбор движка, а не стандарта: PostgreSQL по умолчанию поступает ровно наоборот.

Два трека без рейтинга стоят первыми. Допишите `NULLS LAST` после ключа - `ORDER BY rating NULLS LAST` - и они уедут в конец, не меняя порядок остальных. Затем поменяйте направление на `DESC` и уберите модификатор: `NULL` окажутся внизу сами. `NULLS FIRST` и `NULLS LAST` доступны в SQLite начиная с версии 3.30 и задаются для каждого ключа отдельно.

## Ничьи: выглядит стабильно, но не гарантировано

«Static» и «Ember» набрали по 597 прослушиваний. `ORDER BY plays DESC` не говорит, кто из них первый: взаимный порядок строк с равным ключом не определён, и движок вправе выбрать любой. Хуже того, выбор может месяцами выглядеть стабильным - пока не изменится план запроса, и Вы будете искать «багу», которой нет в коде.

Лекарство - хвостовой ключ, различающий любые две строки; обычно это первичный ключ: `ORDER BY plays DESC, id`. Он ничего не меняет там, где основной ключ уже различил строки, и аккуратно разрешает все ничьи. Особенно это важно в паре с `LIMIT` и `OFFSET`: если граница страницы рассекает ничью, одна и та же строка может попасть на две страницы подряд, а её соседка - не попасть никуда. Правило простое: как только порядок становится частью контракта с пользователем, последний ключ сортировки должен быть уникальным.

## Типичные ошибки

**Полагаться на порядок вставки.** На маленькой базе порядок хранения совпадает с порядком `INSERT`, и привычка закрепляется. Исправление: явный `ORDER BY` по колонке, которая выражает нужный порядок, например по дате создания. Порядок - часть требований, и его место в запросе, а не в надеждах.

**`LIMIT` без `ORDER BY`.** «Дай три строки» без «в каком порядке» - это произвольный срез множества. Возникает потому, что на тестовых данных любой срез выглядит правдоподобно. Исправление: каждое `LIMIT` сопровождайте сортировкой, даже если «и так понятно».

**Пагинация через `OFFSET` по живым данным.** Пока пользователь читает первую страницу, кто-то вставил строку - и вторая страница начинается со сдвигом: одна строка показана дважды, другая пропала. Вдобавок `OFFSET 100000` заставляет движок вычислить и выбросить сто тысяч строк. Для коротких списков это терпимо; для больших - смотрите блок ниже.

<Detail summary="Почему OFFSET медленный и что такое keyset-пагинация">
OFFSET не умеет перепрыгивать: движок обязан построить и отбросить все пропускаемые строки, поэтому стоимость страницы растёт с её номером. Keyset-пагинация запоминает ключ последней показанной строки и запрашивает следующую страницу условием, а не пропуском:

```sql
SELECT title, plays, id
FROM tracks
WHERE (plays, id) < (341, 6)
ORDER BY plays DESC, id DESC
LIMIT 2;
```

При наличии индекса по (plays, id) чтение начинается сразу с нужного места, а вставки между страницами не сдвигают результат. Цена - нельзя прыгнуть на «страницу 50», только листать подряд.
</Detail>

<Detail summary="Когда сортировки не происходит вовсе">
Если по ключу сортировки есть индекс, движок может читать строки сразу в нужном порядке и не сортировать ничего. В SQLite это видно через EXPLAIN QUERY PLAN: строка «USE TEMP B-TREE FOR ORDER BY» означает настоящую сортировку, её отсутствие - чтение по индексу. Для топ-N с LIMIT SQLite к тому же не сортирует весь результат целиком, а удерживает только N лучших строк.
</Detail>
