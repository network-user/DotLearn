# GROUP BY: одна строка на группу

Вы уже умеете сворачивать таблицу в одно число: `COUNT(*)` считает строки, `SUM` складывает значения. Но настоящие вопросы к данным почти никогда не звучат как «сколько всего». Они звучат как «сколько у каждого»: сколько книг написал каждый автор, какая выручка в каждом месяце, сколько заказов сделал каждый клиент. Слово «каждый» - сигнал, что одного числа мало. Нужна сводная таблица: по строке на автора, на месяц, на клиента.

Запускать отдельный запрос на каждого автора - тупиковый путь: авторов может быть тысяча, а завтра появится тысяча первый. `GROUP BY` решает задачу одним запросом: разбивает строки на группы и считает агрегат внутри каждой. К концу раздела Вы будете отвечать на «каждый»-вопросы одной командой, отбирать готовые группы через `HAVING` и понимать, почему рядом с агрегатом можно вывести далеко не любую колонку.

## Корзины: ментальная модель

Представьте строки таблицы как карточки, у каждой на полях написано значение группировочной колонки. `GROUP BY author` раскладывает карточки по корзинам: одна корзина на каждое уникальное значение `author`. Дальше агрегат работает не с таблицей целиком, а с каждой корзиной отдельно: `COUNT(*)` пересчитывает карточки внутри корзины, `SUM(pages)` складывает их страницы. В результат уходит ровно одна строка на корзину.

Нажмите «Сгруппировать» на рисунке ниже и проследите за любой карточкой: она оказывается ровно в одной корзине, а у корзины появляется собственный итог.

Важное следствие: после свёртки исходные строки исчезают из виду. Результат запроса с `GROUP BY` - таблица о группах, а не о строках. У группы «Ada» нет «своего» заголовка книги: заголовков три, и в одной выходной строке им всем места нет.

Кстати, агрегат для группировки не обязателен. `SELECT author FROM book GROUP BY author` вернёт список уникальных авторов - по одной строке на корзину, без итогов. Это то же самое, что `SELECT DISTINCT author FROM book`, и полезный способ проверить интуицию: группировка сама по себе лишь раскладывает строки, считать она ничего не обязана.

## Правило одной колонки

Отсюда главное правило группировки: каждая колонка в `SELECT` обязана либо входить в `GROUP BY`, либо быть обёрнутой в агрегат (`COUNT`, `SUM`, `AVG`, `MIN`, `MAX`). Запрос `SELECT author, title, COUNT(*) FROM book GROUP BY author` некорректен: для Ada в группе три разных `title`, и движку неоткуда узнать, какой из них Вы имели в виду.

<MarginNote label="Другие движки">PostgreSQL отклонит такой запрос с ошибкой. MySQL ведёт себя как SQLite, пока включён режим совместимости, и как PostgreSQL - в строгом режиме ONLY_FULL_GROUP_BY.</MarginNote>

SQLite поступает с этим правилом по-своему, и это честный footgun, о котором стоит знать с первого дня. Он не выдаст ошибку: «голая» колонка вне `GROUP BY` молча получит значение из произвольной строки группы. Запрос выглядит работающим, тесты на маленькой таблице даже могут проходить, а потом порядок строк в хранилище меняется - и «работающий» отчёт начинает показывать другие заголовки.

<Callout type="warning" title="Голые колонки в SQLite">
Не сгруппированная и не агрегированная колонка в SQLite возвращает значение из произвольной строки группы. Единственный случай, когда на это можно опереться: если в запросе есть `MIN()` или `MAX()`, голые колонки берутся из той строки, где достигается минимум или максимум.
</Callout>

## Конвейер: где стоит HAVING

Чтобы понять `HAVING`, нужно увидеть порядок, в котором запрос выполняется логически. Вы пишете `SELECT` первым словом, но движок начинает с `FROM` и заканчивает `ORDER BY`. Группировка встраивается в середину этого конвейера.

Эта схема - ключ ко всей теме. `WHERE` стоит до группировки: он фильтрует отдельные строки и ничего не знает об агрегатах, потому что на его этапе ни групп, ни `COUNT` ещё не существует. `HAVING` стоит после: он фильтрует готовые корзины и свободно ссылается на `COUNT(*)`, `SUM(pages)` и любые другие агрегаты. Один взгляд на конвейер снимает половину вопросов о группировке.

<PullQuote>До GROUP BY запрос рассуждает о строках, после - только о группах.</PullQuote>

## Считаем по группам

Дальше работаем с каталогом из шести книг трёх авторов. Запустите запрос и посмотрите, как шесть строк превращаются в три:

<SideSql
  title="Книги по авторам"
  fixture={`
    CREATE TABLE book (
      id INTEGER PRIMARY KEY,
      title TEXT NOT NULL,
      author TEXT NOT NULL,
      pages INTEGER NOT NULL
    );
    INSERT INTO book VALUES
      (1, 'Compilers',           'Ada',   620),
      (2, 'Algorithms',          'Ada',   480),
      (3, 'Notes on math',       'Ada',    90),
      (4, 'Operating systems',   'Linus', 700),
      (5, 'Kernel quickstart',   'Linus', 120),
      (6, 'COBOL essentials',    'Grace', 220);
  `}
  query={`
    SELECT author, COUNT(*) AS books, SUM(pages) AS total_pages
    FROM book
    GROUP BY author
    ORDER BY author;
  `}
/>

Три карточки Ada сложились в одну строку: 3 книги, 1190 страниц. У Linus две книги на 820 страниц, у Grace - одна на 220. Обе выходные колонки подчиняются правилу: `author` есть в `GROUP BY`, остальное обёрнуто в агрегаты.

Обратите внимание на `ORDER BY` в конце. По конвейеру сортировка выполняется последней, когда свёртка уже произошла, поэтому сортировать можно и по агрегату: `ORDER BY total_pages DESC` выстроит авторов от самого плодовитого к самому скромному. Алиас из `SELECT` здесь уже виден - в отличие от `WHERE`, который выполняется задолго до того, как алиасы появятся на свет.

## HAVING в деле

Теперь оставим только «крупных» авторов - тех, у кого суммарный объём не меньше 500 страниц. Условие сформулировано про сумму по корзине, значит, это работа для `HAVING`. Запрос ниже редактируемый и перезапускается на лету, а диаграмма под таблицей строится прямо из результата.

<SideSql
  title="Авторы с объёмом от 500 страниц"
  editable
  live
  chart={{ type: 'bar', x: 'author', y: 'total_pages' }}
  fixture={`
    CREATE TABLE book (
      id INTEGER PRIMARY KEY,
      title TEXT NOT NULL,
      author TEXT NOT NULL,
      pages INTEGER NOT NULL
    );
    INSERT INTO book VALUES
      (1, 'Compilers',           'Ada',   620),
      (2, 'Algorithms',          'Ada',   480),
      (3, 'Notes on math',       'Ada',    90),
      (4, 'Operating systems',   'Linus', 700),
      (5, 'Kernel quickstart',   'Linus', 120),
      (6, 'COBOL essentials',    'Grace', 220);
  `}
  query={`
    SELECT author, SUM(pages) AS total_pages
    FROM book
    GROUP BY author
    HAVING SUM(pages) >= 500
    ORDER BY total_pages DESC;
  `}
/>

Поэкспериментируйте с порогом в `HAVING`. Поднимите его до 1000 - останется одна Ada с её 1190 страницами. Опустите до 200 - в сводку вернётся Grace. Столбики перестраиваются вместе с результатом: видно, как `HAVING` буквально отрезает корзины, не дотянувшие до порога.

`WHERE` и `HAVING` прекрасно работают в паре. Посчитаем, у кого из авторов есть хотя бы две длинные книги:

```sql
SELECT author, COUNT(*) AS long_books
FROM book
WHERE pages >= 400
GROUP BY author
HAVING COUNT(*) >= 2;
```

Проследите по конвейеру. `WHERE` оставляет три длинные книги: 620 и 480 страниц у Ada, 700 у Linus. `GROUP BY` раскладывает их в две корзины: у Ada две карточки, у Linus одна. `HAVING` оставляет только корзину Ada. Один и тот же `COUNT(*)` здесь считает уже отфильтрованные строки - длинные книги, а не все.

<Compare
  leftTitle="WHERE"
  rightTitle="HAVING"
  left="Фильтрует отдельные строки до группировки. Агрегаты недоступны: групп ещё нет."
  right="Фильтрует готовые группы после свёртки. Видит COUNT, SUM, AVG и другие агрегаты."
  verdict="Если предикат можно записать без агрегата - это работа для WHERE: он срабатывает раньше и оставляет группировке меньше строк."
/>

Формально SQLite позволит написать в `HAVING` и условие без агрегата, например `HAVING author <> 'Grace'`. Запрос отработает, но это дурной стиль: такое условие фильтрует по значению, известному ещё до свёртки, и ему место в `WHERE`, где оно отсеет строки раньше и дешевле. Держитесь простого правила: `HAVING` - только для предикатов с агрегатами.

Агрегат в `HAVING` не обязан появляться в `SELECT`. Здесь средний объём считается только ради фильтра:

```sql
SELECT author
FROM book
GROUP BY author
HAVING AVG(pages) > 300;
```

В ответе останутся Ada (в среднем около 397 страниц на книгу) и Linus (410), а сама средняя величина в вывод не попадёт.

## Несколько колонок и выражения

Группировать можно сразу по нескольким колонкам - тогда корзина создаётся для каждой уникальной комбинации значений:

```sql
SELECT seller, product, SUM(qty) AS sold
FROM sale
GROUP BY seller, product;
```

Если Ann продаёт чай и кофе, а Bob - только кофе, получится три корзины: `(Ann, tea)`, `(Ann, coffee)` и `(Bob, coffee)`. Правило одной колонки распространяется на обе: и `seller`, и `product` перечислены в `GROUP BY`, поэтому их можно выводить как есть.

Ключом группировки может быть и выражение. Классический пример - сводка по месяцам, когда в таблице хранится полная дата:

```sql
SELECT substr(paid_at, 1, 7) AS month, SUM(amount) AS total
FROM payment
GROUP BY substr(paid_at, 1, 7)
ORDER BY month;
```

Группа определяется вычисленным значением вроде `2024-01`, а не сырым содержимым колонки. SQLite разрешает сослаться в `GROUP BY` и на алиас `month` - получится короче и читабельнее.

## NULL получает собственную корзину

В `WHERE` NULL ведёт себя отстранённо: `NULL = NULL` не истинно, и строка с NULL не проходит ни один фильтр сравнения. Логично ожидать, что и группироваться NULL откажется. Сюрприз: `GROUP BY` поступает наоборот. Все NULL считаются «одинаковыми» и складываются в одну общую корзину.

<SideSql
  title="Игроки без команды образуют группу"
  fixture={`
    CREATE TABLE player (
      id INTEGER PRIMARY KEY,
      name TEXT NOT NULL,
      team TEXT
    );
    INSERT INTO player VALUES
      (1, 'Mira',  'red'),
      (2, 'Oleg',  NULL),
      (3, 'Pavel', 'red'),
      (4, 'Rita',  NULL),
      (5, 'Sana',  'blue');
  `}
  query={`
    SELECT team, COUNT(*) AS members
    FROM player
    GROUP BY team
    ORDER BY team;
  `}
/>

В выводе три строки: группа с пустым `team`, blue и red. NULL-группа стоит первой, потому что при сортировке по возрастанию SQLite ставит NULL раньше любых значений. Если игроки без команды в сводке не нужны, отбросьте их до группировки: `WHERE team IS NOT NULL`.

Ещё одна тонкость про пустой вход: если после `WHERE` не осталось ни одной строки, запрос с `GROUP BY` вернёт ноль строк - нет карточек, нет и корзин. Тот же агрегатный запрос без `GROUP BY` вернул бы одну строку с `COUNT(*) = 0`.

## Типичные ошибки

- **Фильтр по агрегату в WHERE.** `WHERE COUNT(*) > 1` падает с ошибкой `misuse of aggregate`: на этапе `WHERE` групп ещё не существует, считать нечего. Перенесите условие в `HAVING`. Ошибка случается потому, что запрос читается сверху вниз, а выполняется по конвейеру.
- **Колонка в SELECT без места в GROUP BY.** SQLite молча подставит значение из произвольной строки группы, и баг всплывёт не сразу. Либо добавьте колонку в `GROUP BY`, либо оберните её в агрегат.
- **Считать не то.** `COUNT(*)` считает строки группы, `COUNT(col)` - не-NULL значения колонки, `COUNT(DISTINCT col)` - уникальные не-NULL значения. «Сколько заказов у клиента» и «сколько разных товаров он покупал» - это разные формы `COUNT`; выбирайте форму под вопрос, а не наугад.

<Detail summary="Как движок строит группы">
Два классических способа сгруппировать строки: отсортировать их по ключу группировки, чтобы члены каждой группы оказались подряд, или построить hash-таблицу «ключ группы → промежуточный агрегат». SQLite идёт по первому пути: если по группировочной колонке есть индекс, строки уже идут в нужном порядке, иначе движок сортирует их во временной структуре. Практический вывод: `GROUP BY` по индексированной колонке заметно дешевле на больших таблицах.
</Detail>
