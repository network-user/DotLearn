# JOIN: собираем данные из нескольких таблиц

В живой базе данных ответ почти на любой вопрос разбросан по нескольким таблицам. Заказ хранит не имя клиента, а его номер. Строка продажи знает только id книги, а книга - только id автора. Спросите «какие книги продались на этой неделе и кто их написал» - и ответ придётся собирать из трёх таблиц, как из конструктора.

Инструмент сборки называется `JOIN`. К концу раздела Вы будете соединять таблицы через INNER и LEFT JOIN, понимать, откуда в результате берутся NULL и «лишние» строки-дубли, и узнаете главную ловушку новичка: безобидный с виду фильтр, который незаметно превращает LEFT JOIN обратно в INNER.

## Зачем данные вообще разрезаны

Представьте библиотечную базу из одной таблицы, где у каждой книги прямо в строке записано имя автора. Пока записей мало, всё работает. Потом автор меняет фамилию - и Вам нужно обновить пятьдесят строк. Одну Вы пропустите, в другой опечатаетесь, и в базе заведутся два «разных» автора с почти одинаковыми именами. Одна и та же информация, записанная много раз, рано или поздно расходится сама с собой.

Поэтому реляционные базы хранят каждый факт один раз. Имя автора живёт в таблице `author`, в единственной строке. Книга имя не копирует - она хранит ссылку: колонку `author_id` со значением ключа из `author`. Такое разрезание данных на таблицы называют нормализацией.

<MarginNote label="Термин">Колонку, значения которой ссылаются на ключ другой таблицы, называют внешним ключом (foreign key). `book.author_id` - внешний ключ к `author.id`.</MarginNote>

У этой дисциплины есть цена: чтобы показать книгу вместе с именем автора, разрезанные таблицы нужно соединить обратно. Именно это делает JOIN. Хранение оптимизировано под изменения - каждый факт меняется в одном месте; чтение собирает факты заново при каждом запросе. Этот размен почти всегда выгоден: запросов с JOIN в типичном приложении тысячи, а расползшиеся данные не починить ни одним запросом.

## Мысленная модель: все пары, потом фильтр

Самый честный способ думать о JOIN такой: движок берёт каждую строку левой таблицы и прикладывает к ней каждую строку правой. Три автора и три книги дают девять пар - это декартово произведение. Затем условие `ON` проходит по парам и оставляет только те, где ключи совпали. Реальные движки считают хитрее, но результат всегда такой, как будто произошло именно это: все пары, потом фильтр.

В `ON` чаще всего стоит равенство «внешний ключ = первичный ключ», но это не требование языка: годится любое логическое условие над колонками обеих сторон. Пара «строка слева плюс строка справа» проходит, если условие для неё истинно, - вот и вся механика.

Покрутите схему ниже. Слева авторы, справа книги; цветная точка - значение ключа соединения. Переключайте тип JOIN и следите за счётчиком пар внизу: INNER оставляет только совпавшие пары, LEFT добирает строки левой таблицы, оставшиеся без пары, RIGHT - правой.

## INNER JOIN: только совпавшие пары

`INNER JOIN ... ON` оставляет пары, где условие истинно. Всё, что пару не нашло, молча исчезает.

<SideSql
  title="INNER JOIN"
  fixture={`CREATE TABLE author (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL
);
INSERT INTO author VALUES (1, 'Ada'), (2, 'Linus'), (3, 'Grace');

CREATE TABLE book (
  id INTEGER PRIMARY KEY,
  title TEXT NOT NULL,
  author_id INTEGER REFERENCES author(id)
);
INSERT INTO book VALUES
  (1, 'Analytical engine', 1),
  (2, 'Kernel notes', 2),
  (3, 'Loose pages', NULL);`}
  query={`SELECT a.name, b.title
FROM author a
INNER JOIN book b ON b.author_id = a.id
ORDER BY a.name;`}
/>

Две детали синтаксиса. Во-первых, `author a` и `book b` - алиасы: короткие имена, которыми Вы ссылаетесь на таблицы в остальной части запроса. Во-вторых, `ON b.author_id = a.id` - правило склейки: пара выживает, когда ссылка в книге совпала с ключом автора.

В результате нет ни Грейс, ни «Loose pages». INNER JOIN не сообщает о потерях - он просто возвращает меньше строк, чем Вы, возможно, ожидали.

## LEFT JOIN: левая сторона не теряется

Часто терять строки нельзя. «Покажи всех авторов и их книги» означает: всех, включая тех, у кого книг нет. Для этого есть `LEFT JOIN`: каждая строка левой таблицы попадает в результат обязательно, а если пары справа не нашлось, колонки правой стороны заполняются NULL.

<SideSql
  title="LEFT JOIN"
  fixture={`CREATE TABLE author (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL
);
INSERT INTO author VALUES (1, 'Ada'), (2, 'Linus'), (3, 'Grace');

CREATE TABLE book (
  id INTEGER PRIMARY KEY,
  title TEXT NOT NULL,
  author_id INTEGER REFERENCES author(id)
);
INSERT INTO book VALUES
  (1, 'Analytical engine', 1),
  (2, 'Kernel notes', 2),
  (3, 'Loose pages', NULL);`}
  query={`SELECT a.name, b.title
FROM author a
LEFT JOIN book b ON b.author_id = a.id
ORDER BY a.name;`}
/>

Грейс вернулась - с NULL вместо названия. Правило выбора стороны простое: слева ставьте таблицу, строки которой нельзя терять. Вопрос «все авторы и их книги» делает левой таблицей `author`; вопрос «все книги и их авторы» - `book`.

Существует и `RIGHT JOIN` - зеркальный вариант, сохраняющий правую сторону. На практике он встречается редко: тот же результат даёт LEFT JOIN с переставленными таблицами, а читать запрос, где «выживающая» таблица стоит слева, проще. Выбирая между ними, перестраивайте запрос под LEFT.

## Дубликаты ключа умножают строки

Общее правило: если значение ключа встречается m раз слева и n раз справа, в результате будет m × n строк с этим значением. Коварным это становится рядом с агрегатами: посчитайте `SUM` по соединённой таблице - и сумма «удвоится», потому что строки размножились. Если после JOIN строк больше, чем Вы ожидали, первым делом ищите дубликаты ключа.

## NULL в ключе и анти-JOIN

NULL не равен ничему, даже другому NULL, - Вы уже сталкивались с этим в фильтрах. Поэтому строка с NULL в ключе соединения не найдёт пару никогда: условие `b.author_id = a.id` для неё не истинно ни с одной строкой. «Loose pages» не присоединится ни к одному автору.

У этого свойства есть полезное применение - анти-JOIN, то есть «строки слева, у которых нет пары справа»:

```sql
SELECT a.name
FROM author a
LEFT JOIN book b ON b.author_id = a.id
WHERE b.id IS NULL;
```

LEFT JOIN сохранил всех авторов, а NULL в `b.id` остался только у тех, кому пары не нашлось. Фильтр `IS NULL` оставляет именно их: авторов без книг. Для проверки берите колонку, которая в правой таблице не бывает NULL сама по себе, - первичный ключ подходит идеально. Если проверять колонку, где NULL встречается и в обычных строках, анти-JOIN захватит лишнее.

Крайний случай той же логики - пустая правая таблица. INNER JOIN с ней вернёт ноль строк: пар не из чего составить. LEFT JOIN вернёт все строки левой таблицы, и у каждой правая сторона будет NULL. Это поведение часто удивляет, но оно прямо следует из определения: левая сторона не теряется никогда.

## Ловушка: WHERE превращает LEFT JOIN в INNER

Теперь главный подвох. Посмотрите на логический порядок выполнения запроса:

      { label: "ORDER BY" },
    ]}
  />

`ON` работает во время соединения, `WHERE` - после, по уже склеенным строкам. Допустим, Вы хотите «всех авторов и рядом их книги на букву K, если такие есть». Напрашивается:

```sql
SELECT a.name, b.title
FROM author a
LEFT JOIN book b ON b.author_id = a.id
WHERE b.title LIKE 'K%';
```

У авторов без подходящих книг `b.title` равен NULL, а сравнение NULL с шаблоном не истинно - и WHERE выбрасывает их строки. Остаётся один Линус: LEFT JOIN формально отработал, но результат неотличим от INNER. Правильное место для условия по правой таблице - внутри `ON`:

```sql
SELECT a.name, b.title
FROM author a
LEFT JOIN book b
  ON b.author_id = a.id AND b.title LIKE 'K%';
```

Теперь условие участвует в подборе пары: книги не на K просто не считаются совпадением, и каждый автор остаётся в результате - кто с книгой, кто с NULL. Проверьте обе версии в песочнице выше. Единственный фильтр по правой таблице, которому место в WHERE после LEFT JOIN, - это `IS NULL` из анти-JOIN: он специально ловит несовпавших.

## Три таблицы и встреча с самим собой

JOIN не ограничен двумя таблицами - соединения выстраиваются в цепочку, каждое со своим `ON`:

```sql
SELECT a.name, b.title, s.price
FROM sale s
INNER JOIN book b ON b.id = s.book_id
INNER JOIN author a ON a.id = b.author_id
ORDER BY s.price DESC;
```

Читайте сверху вниз: к продажам присоединили книги, к получившемуся результату - авторов. Каждое `ON` связывает новую таблицу с тем, что уже собрано. Стройте такие запросы инкрементально: напишите первое соединение, выполните `SELECT *`, убедитесь, что число строк осталось разумным, и только потом добавляйте следующую таблицу. Если собрать цепочку из четырёх JOIN сразу и получить странный результат, искать виноватого придётся во всех звеньях одновременно.

Алиасы здесь перестают быть удобством и становятся необходимостью, когда одна таблица встречается в запросе дважды. Классический пример - таблица сотрудников, где `manager_id` ссылается на ту же самую таблицу:

```sql
SELECT e.name, m.name AS manager
FROM employee e
LEFT JOIN employee m ON m.id = e.manager_id;
```

Это self-join: `e` и `m` - две независимые «копии» одной таблицы, и без алиасов движок не смог бы их различить. Приём тот же JOIN, просто обе стороны - одна таблица; подробнее Вы встретитесь с ним в более сложных запросах.

## Типичные ошибки

Три ошибки ниже дают почти весь поток вопросов про JOIN на форумах. Каждая возникает не от незнания синтаксиса, а от размытой мысленной модели - и каждая лечится возвратом к картинке «все пары, потом фильтр».

- Забыть `ON`. SQLite молча выполнит перекрёстное соединение и вернёт декартово произведение. Если результат внезапно огромен, проверьте, у каждого ли JOIN есть условие.
- Фильтровать правую таблицу LEFT JOIN в `WHERE`. Сравнение с NULL не истинно, несовпавшие строки исчезают, и соединение ведёт себя как INNER. Условие по правой стороне переносите в `ON`.
- Неоднозначные имена колонок. Если `id` есть в обеих таблицах, `SELECT id` упадёт с ошибкой «ambiguous column name». Уточняйте всегда: `a.id`, `b.id`.

<Detail summary="Как движок выполняет JOIN на самом деле">
Декартово произведение - это модель результата, а не алгоритм. Простейшая реальная стратегия - вложенный цикл: для каждой строки левой таблицы пробежать правую. Если по ключу правой таблицы есть индекс, внутренний проход превращается в быстрый поиск, и соединение миллионных таблиц перестаёт быть страшным. SQLite выполняет JOIN именно как вложенные циклы с индексами; более крупные движки добавляют hash join (построить хеш-таблицу по меньшей стороне) и merge join (слить две отсортированные стороны). На результат стратегия не влияет - только на скорость.
</Detail>
