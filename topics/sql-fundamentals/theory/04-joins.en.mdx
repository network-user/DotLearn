# JOIN: putting data from several tables back together

In a real database, the answer to almost any question is scattered across several tables. An order stores the customer's number, not their name. A sale row knows only the book's id, and the book knows only the author's id. Ask "which books sold this week and who wrote them" - and you have to assemble the answer from three tables, like a kit.

The assembly tool is called `JOIN`. By the end of this section you will combine tables with INNER and LEFT JOIN, understand where NULLs and "extra" duplicate rows in the result come from, and learn the classic beginner trap: an innocent-looking filter that quietly turns a LEFT JOIN back into an INNER one.

## Why the data is cut into pieces in the first place

Imagine a library database made of a single table, where every book row carries the author's name. While there are few records, everything works. Then an author changes their last name - and you have to update fifty rows. You miss one, you mistype another, and the database now holds two "different" authors with almost identical names. The same information written down many times eventually disagrees with itself.

That is why relational databases store every fact once. The author's name lives in the `author` table, in a single row. A book does not copy the name - it stores a reference: an `author_id` column holding a key value from `author`. Cutting data into tables like this is called normalization.

<MarginNote label="Term">A column whose values reference another table's key is called a foreign key. `book.author_id` is a foreign key to `author.id`.</MarginNote>

This discipline has a price: to show a book together with its author's name, the separated tables have to be glued back together. That is exactly what JOIN does. Storage is optimized for change - every fact is updated in one place; reading reassembles the facts on every query. The trade is almost always worth it: a typical application runs thousands of JOIN queries, while data that has drifted apart cannot be repaired by any query.

## The mental model: all pairs, then a filter

The most honest way to think about JOIN is this: the engine takes every row of the left table and lays every row of the right table next to it. Three authors and three books make nine pairs - the Cartesian product. Then the `ON` condition walks the pairs and keeps only those where the keys matched. Real engines compute this more cleverly, but the result is always as if exactly that happened: all pairs, then a filter.

Most of the time `ON` holds an equality of the form "foreign key = primary key", but the language does not require that: any boolean condition over columns of both sides will do. A pair "left row plus right row" survives if the condition is true for it - that is the whole mechanism.

## INNER JOIN: matched pairs only

`INNER JOIN ... ON` keeps the pairs where the condition is true. Anything that found no match silently disappears.

<SideSql
  title="INNER JOIN"
  fixture={`CREATE TABLE author (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL
);
INSERT INTO author VALUES (1, 'Ada'), (2, 'Linus'), (3, 'Grace');

CREATE TABLE book (
  id INTEGER PRIMARY KEY,
  title TEXT NOT NULL,
  author_id INTEGER REFERENCES author(id)
);
INSERT INTO book VALUES
  (1, 'Analytical engine', 1),
  (2, 'Kernel notes', 2),
  (3, 'Loose pages', NULL);`}
  query={`SELECT a.name, b.title
FROM author a
INNER JOIN book b ON b.author_id = a.id
ORDER BY a.name;`}
/>

Two pieces of syntax to notice. First, `author a` and `book b` are aliases: short names you use to refer to the tables in the rest of the query. Second, `ON b.author_id = a.id` is the gluing rule: a pair survives when the reference in the book matches the author's key.

The result contains neither Grace nor "Loose pages". INNER JOIN does not report losses - it simply returns fewer rows than you might have expected.

## LEFT JOIN: the left side never gets lost

Often you cannot afford to lose rows. "Show all authors and their books" means all of them, including those who have no books. That is what `LEFT JOIN` is for: every row of the left table makes it into the result no matter what, and when no match is found on the right, the right-side columns are filled with NULL.

<SideSql
  title="LEFT JOIN"
  fixture={`CREATE TABLE author (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL
);
INSERT INTO author VALUES (1, 'Ada'), (2, 'Linus'), (3, 'Grace');

CREATE TABLE book (
  id INTEGER PRIMARY KEY,
  title TEXT NOT NULL,
  author_id INTEGER REFERENCES author(id)
);
INSERT INTO book VALUES
  (1, 'Analytical engine', 1),
  (2, 'Kernel notes', 2),
  (3, 'Loose pages', NULL);`}
  query={`SELECT a.name, b.title
FROM author a
LEFT JOIN book b ON b.author_id = a.id
ORDER BY a.name;`}
/>

Grace is back - with NULL instead of a title. The side-picking rule is simple: put the table whose rows must not be lost on the left. "All authors and their books" makes `author` the left table; "all books and their authors" makes it `book`.

There is also `RIGHT JOIN` - the mirror image that preserves the right side. In practice it is rare: a LEFT JOIN with the tables swapped gives the same result, and a query where the "surviving" table sits on the left is easier to read. When choosing between the two, restructure the query to use LEFT.

The last experiment shows what happens when `ON` is forgotten: every row on the left joins with every row on the right. On toy tables that is 12 rows; on two million-row tables it is a trillion. The database will not warn you: from its point of view you asked for exactly that Cartesian product.

Take one habit away from this sandbox: the result's row count is your first diagnostic signal. Estimate in advance how many rows you expect and compare with what came back. Fewer than expected - INNER dropped unmatched rows; more - duplicate keys multiplied rows or a join condition went missing.

## Duplicate keys multiply rows

In the sandbox Ada appeared twice - once per book of hers. That is not a bug but the definition of JOIN: a left-table row is repeated for every matching right-table row.

The general rule: if a key value occurs m times on the left and n times on the right, the result holds m × n rows with that value. This gets treacherous next to aggregates: compute a `SUM` over a joined table and the total "doubles", because the rows have multiplied. Whenever a JOIN returns more rows than you expected, look for duplicate keys first.

## NULL in the join key, and the anti-join

NULL equals nothing, not even another NULL - you have already run into this with filters. So a row with NULL in its join key never finds a match: the condition `b.author_id = a.id` is not true for it against any row. "Loose pages" will not attach to any author.

This property has a useful application - the anti-join, meaning "left-side rows that have no match on the right":

```sql
SELECT a.name
FROM author a
LEFT JOIN book b ON b.author_id = a.id
WHERE b.id IS NULL;
```

LEFT JOIN preserved all the authors, and NULL in `b.id` remained only for those who found no match. The `IS NULL` filter keeps exactly them: authors without books. For the check, pick a column that is never NULL in the right table on its own - a primary key is ideal. If you test a column where NULL also appears in ordinary rows, the anti-join will catch too much.

The extreme case of the same logic is an empty right table. An INNER JOIN against it returns zero rows: there is nothing to build pairs from. A LEFT JOIN returns every left-table row, each with NULL on the right side. This behavior often surprises people, but it follows directly from the definition: the left side is never lost.

## The trap: WHERE turns a LEFT JOIN into an INNER one

Now the main catch. Look at the logical execution order of the query:

      { label: "SELECT" },
      { label: "ORDER BY" },
    ]}
  />
</Figure>

`ON` works during the join; `WHERE` works after it, on the already-glued rows. Say you want "all authors, and next to each their books starting with K, if any". The obvious attempt is:

```sql
SELECT a.name, b.title
FROM author a
LEFT JOIN book b ON b.author_id = a.id
WHERE b.title LIKE 'K%';
```

For authors with no qualifying books `b.title` is NULL, and comparing NULL against a pattern is not true - so WHERE throws their rows away. Only Linus remains: the LEFT JOIN formally ran, but the result is indistinguishable from INNER. The right place for a condition on the right table is inside `ON`:

```sql
SELECT a.name, b.title
FROM author a
LEFT JOIN book b
  ON b.author_id = a.id AND b.title LIKE 'K%';
```

Now the condition takes part in matchmaking: books not starting with K simply do not count as matches, and every author stays in the result - some with a book, some with NULL. Verify both versions in the sandbox above. The only right-table filter that belongs in WHERE after a LEFT JOIN is the anti-join's `IS NULL`: it is designed to catch the unmatched.

## Three tables, and meeting yourself

JOIN is not limited to two tables - joins chain, each with its own `ON`:

```sql
SELECT a.name, b.title, s.price
FROM sale s
INNER JOIN book b ON b.id = s.book_id
INNER JOIN author a ON a.id = b.author_id
ORDER BY s.price DESC;
```

Read it top to bottom: books were attached to the sales, then authors to that result. Each `ON` ties the new table to what has been assembled so far. Build such queries incrementally: write the first join, run a `SELECT *`, make sure the row count stays sane, and only then attach the next table. Assemble a four-JOIN chain in one go and get a strange result, and you will be hunting the culprit in every link at once.

Aliases stop being a convenience and become a necessity when the same table appears in a query twice. The classic example is an employees table where `manager_id` references the very same table:

```sql
SELECT e.name, m.name AS manager
FROM employee e
LEFT JOIN employee m ON m.id = e.manager_id;
```

This is a self-join: `e` and `m` are two independent "copies" of one table, and without aliases the engine could not tell them apart. The technique is the same JOIN, only both sides are one table; you will meet it again in more advanced queries.

## Common mistakes

The three mistakes below account for nearly the entire stream of JOIN questions on forums. Each comes not from missing syntax but from a fuzzy mental model - and each is cured by going back to the "all pairs, then a filter" picture.

- Forgetting `ON`. SQLite silently performs a cross join and returns the Cartesian product. If the result is suddenly huge, check that every JOIN has its condition.
- Filtering the right table of a LEFT JOIN in `WHERE`. The NULL comparison is not true, unmatched rows vanish, and the join behaves like INNER. Move right-side conditions into `ON`.
- Ambiguous column names. If `id` exists in both tables, `SELECT id` fails with "ambiguous column name". Always qualify: `a.id`, `b.id`.

<Detail summary="How the engine actually executes a JOIN">
The Cartesian product is a model of the result, not an algorithm. The simplest real strategy is a nested loop: for every left-table row, scan the right table. If the right table's key has an index, the inner scan becomes a fast lookup, and joining million-row tables stops being scary. SQLite executes joins exactly as nested loops with indexes; bigger engines add hash joins (build a hash table over the smaller side) and merge joins (zip two sorted sides together). The strategy never changes the result - only the speed.
</Detail>
