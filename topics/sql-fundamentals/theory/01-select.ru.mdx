# SELECT: читаем из таблицы ровно то, что нужно

В таблице `users` лежат пять тысяч строк, а вам нужны три: совершеннолетние пользователи из Польши, у которых указан email. Выкачивать всё и фильтровать в коде приложения - медленно и расточительно; база данных умеет отдавать ровно нужный срез сама. Эту работу выполняет связка `SELECT` и `WHERE` - два слова, которые вы будете набирать чаще любых других за всю карьеру работы с данными.

К концу статьи вы будете выбирать нужные колонки, отбирать строки по составным условиям, искать по шаблону через `LIKE`, убирать дубликаты через `DISTINCT` - и понимать, почему сравнение с `NULL` молча возвращает «ничего» и как это обойти.

## Два разреза одной таблицы

Таблица - это прямоугольник: строки идут вниз, колонки - вправо. Любой запрос на чтение делает с этим прямоугольником всего две вещи: выбрасывает ненужные строки и оставляет нужные колонки. Всё остальное в `SELECT` - детали этих двух движений.

<MarginNote label="История">Язык SEQUEL появился в IBM в 1974 году. Слова SELECT, FROM и WHERE с тех пор не изменились.</MarginNote>

Возьмём конкретный запрос: `SELECT name, age FROM users WHERE age >= 18`. Условие `WHERE age >= 18` идёт по строкам сверху вниз и оставляет только совершеннолетних - это разрез по горизонтали. Список `name, age` идёт по колонкам и оставляет от каждой выжившей строки только имя и возраст - это разрез по вертикали. Две операции независимы: можно отфильтровать строки и вернуть все колонки, можно вернуть одну колонку всех строк, а чаще всего вы делаете и то и другое сразу.

У этих разрезов есть устоявшиеся названия - они встретятся вам в документации и в чужом коде. Отбор строк по горизонтали называют *селекцией*, отбор колонок по вертикали - *проекцией*, а само условие, по которому `WHERE` решает судьбу строки, на жаргоне зовут *предикатом*. Заучивать слова необязательно: важнее, что под ними прячется всего лишь «оставить нужные строки» и «оставить нужные колонки».

На рисунке условие `age >= 18` пропускает Anna и Eva и останавливает Bob и Ian. Заметьте: строка либо проходит целиком, либо отбрасывается целиком. `WHERE` никогда не пропускает строку «частично» - он отвечает только на вопрос «жива эта строка или нет».

Второй ключевой факт: порядок выполнения не совпадает с порядком записи. Вы пишете `SELECT` первым словом, но движок выполняет его последним.

Сначала `FROM` определяет источник данных, затем `WHERE` отбрасывает лишние строки, и только после этого `SELECT` проецирует выживших на нужные колонки. Из этого порядка вытекает практичное следствие: в `WHERE` нельзя ссылаться на псевдонимы колонок, объявленные в `SELECT`, - в момент фильтрации их ещё не существует.

## Собираем запрос по шагам

Все примеры этой статьи работают с одной таблицей:

```sql
CREATE TABLE users (id INTEGER, name TEXT, age INTEGER, country TEXT, email TEXT);
INSERT INTO users VALUES
  (1, 'Anna', 25, 'PL', 'anna@example.com'),
  (2, 'Bob', 17, 'US', NULL),
  (3, 'Eva', 30, 'PL', 'eva@example.com'),
  (4, 'Ian', 14, 'LV', 'ian@test.org'),
  (5, 'Olga', 41, 'LV', NULL);
```

Минимальный запрос называет колонки и таблицу:

```sql
SELECT name, age FROM users;
```

Звёздочка возвращает все колонки сразу:

```sql
SELECT * FROM users;
```

Для быстрых экспериментов `*` удобна, но в коде приложения перечисляйте колонки явно: схема таблицы со временем меняется, и запрос со звёздочкой однажды притащит в приложение колонку, которую никто не ждал.

Фильтрация добавляется одним условием. Доступны привычные операторы сравнения: `=`, `<>`, `<`, `<=`, `>`, `>=`.

```sql
SELECT name FROM users WHERE country <> 'PL';
```

Запрос вернёт Bob, Ian и Olga - всех, чья страна не Польша. Условия комбинируются через `AND`, `OR` и `NOT`:

```sql
SELECT name, age, country FROM users
WHERE age >= 18 AND (country = 'PL' OR country = 'LV');
```

Здесь скобки не для красоты. `AND` связывает сильнее, чем `OR`, поэтому без скобок условие читалось бы как «совершеннолетние поляки, а ещё кто угодно из Латвии» - и в результат попал бы четырнадцатилетний Ian. Скобки бесплатны: расставляйте их при малейшем сомнении в приоритетах.

Когда точное значение неизвестно, нужен поиск по шаблону. Оператор `LIKE` понимает два спецсимвола: `%` заменяет любую последовательность символов (включая пустую), `_` - ровно один символ.

```sql
SELECT name, email FROM users WHERE email LIKE '%@example.com';
```

Запрос найдёт Anna и Eva: их адреса заканчиваются на `@example.com`. Шаблон `'a%'` нашёл бы адреса, начинающиеся с `a`, а `'____@%'` - адреса, где до `@` ровно четыре символа.

Последний инструмент этого раздела - `DISTINCT`. Он убирает повторяющиеся строки из результата:

```sql
SELECT DISTINCT country FROM users;
```

Вместо пяти строк вернутся три: `PL`, `US`, `LV`. Важно, что `DISTINCT` применяется ко всей строке результата, а не к отдельной колонке: `SELECT DISTINCT country, age` уберёт только полностью совпадающие пары «страна + возраст».

## NULL: логика с тремя ответами

В привычной булевой логике сравнение возвращает истину или ложь. В SQL ответов три: `TRUE`, `FALSE` и `NULL` - «неизвестно». Любое сравнение с `NULL` даёт `NULL`: и `email = NULL`, и `email <> NULL`, и даже `NULL = NULL`. Логика честная: `NULL` означает «значения нет» или «значение неизвестно», а сравнивать два неизвестных - всё равно что спрашивать, равны ли два числа, которые никто не называл.

`WHERE` пропускает строку только при `TRUE`. `NULL` - не `TRUE`, поэтому строка с `NULL` в сравниваемой колонке не пройдёт ни условие, ни его отрицание.

<SideSql
  title="Ловушка NULL"
  editable

  fixture={`CREATE TABLE users (id INTEGER, name TEXT, age INTEGER, country TEXT, email TEXT);
INSERT INTO users VALUES
  (1, 'Anna', 25, 'PL', 'anna@example.com'),

  query="SELECT name FROM users WHERE email = NULL;"
/>

Ноль строк - хотя у Bob и Olga email действительно отсутствует. Замените `= NULL` на `IS NULL` и запустите снова: обе строки появятся. Для обратной проверки существует `IS NOT NULL`.

<Callout type="warning" title="NULL не равен ничему">
Даже самому себе: выражение NULL = NULL возвращает NULL, а не TRUE. Отсутствие значения проверяется только операторами IS NULL и IS NOT NULL.
</Callout>

Второй сюрприз скромнее, но важен для кода приложения: пустой результат - это не ошибка. Запрос `SELECT * FROM users WHERE age > 100` на нашей таблице вернёт ноль строк и завершится успешно. «Таких строк нет» - полноценный ответ на заданный вопрос, и приложение обязано быть готово к пустому списку так же, как к непустому.

<PullQuote>Пустой результат - это ответ, а не ошибка.</PullQuote>

Третий сюрприз касается регистра. В SQLite оператор `=` чувствителен к регистру: `'anna' = 'Anna'` ложно. А вот `LIKE` для латинских букв по умолчанию регистр игнорирует: `'Anna' LIKE 'anna'` истинно. Один и тот же литерал в `=` и в `LIKE` может найти разное число строк. В других СУБД поведение иное: в PostgreSQL `LIKE` чувствителен к регистру, а для нечувствительного поиска есть отдельный оператор `ILIKE`. Прежде чем полагаться на регистр, проверьте, как ведёт себя ваш движок.

<Detail summary="Таблица истинности трёхзначной логики">

| Выражение | Результат |
| --- | --- |
| `TRUE AND NULL` | `NULL` |
| `FALSE AND NULL` | `FALSE` |
| `TRUE OR NULL` | `TRUE` |
| `FALSE OR NULL` | `NULL` |
| `NOT NULL` | `NULL` |

Закономерность: если результат определяется известным операндом (`FALSE AND` что угодно ложно, `TRUE OR` что угодно истинно), неизвестность не мешает. Если же ответ зависит от неизвестного операнда, результат остаётся неизвестным.

</Detail>

## Типичные ошибки

**`==` вместо `=`.** В SQL равенство проверяется одиночным знаком `=`: присваивания внутри запроса не бывает, поэтому второй знак не нужен. SQLite из снисходительности принимает и `==`, но большинство СУБД ответят синтаксической ошибкой. Это рефлекс из языков программирования; переучивайтесь на `=` сразу.

**Перепутанные кавычки.** Одинарные кавычки - для строковых значений: `'Anna'`. Двойные - для имён таблиц и колонок: `"name"`. Запрос `WHERE name = "Anna"` в строгой СУБД упадёт с ошибкой «колонка Anna не существует». SQLite коварнее: не найдя такую колонку, он молча трактует `"Anna"` как строку, и запрос «работает» ровно до переноса в другую базу.

**Сравнение с `NULL` через `=`.** Симптом: условие «не находит» строки, которые точно есть в таблице. Причина - трёхзначная логика из предыдущего раздела; лечится заменой на `IS NULL` или `IS NOT NULL`.

**`WHERE` с агрегатами.** Рано или поздно вы напишете `WHERE COUNT(*) > 5` и получите ошибку. `WHERE` выполняется до группировки и видит отдельные строки, а не итоги по группам; для фильтрации групп существует `HAVING`, и о нём пойдёт речь в разделе про `GROUP BY`. Пока достаточно запомнить: `WHERE` - про строки, `HAVING` - про группы.
