# Хендлер просит то, что ему нужно, а не добывает сам

Хендлеру, который выдаёт список заказов, нужно соединение с базой данных. Самый прямой способ - открыть это соединение прямо внутри функции: создать клиента, сходить в базу, вернуть результат. Беда в том, что теперь каждый хендлер сам отвечает за создание ресурса, а значит знает строку подключения, повторяет код инициализации и тащит за собой всё дерево настроек. Поменялся способ создания соединения - правьте десять функций. А в тестах подменить настоящую базу на фейк вообще негде: хендлер сам её создаёт внутри себя.

Внедрение зависимостей (dependency injection, DI) переворачивает это. Хендлер не добывает соединение - он объявляет в сигнатуре, что ему нужен параметр с таким-то именем, а фреймворк сам его создаёт и передаёт. В Litestar это делается через `Provide`: Вы регистрируете, какая функция отвечает за параметр, и фреймворк вызывает её перед хендлером, подставляя результат. К концу раздела Вы будете понимать, как `Provide` связывает имя параметра с фабрикой, на каких слоях объявляют зависимости и как нижний слой переопределяет верхний, чем sync-зависимость отличается от async, когда кэширование `use_cache` помогает, а когда тихо ломает логику, и как одна зависимость может зависеть от другой.

## Ментальная модель: связывание по имени параметра

DI в Litestar строится вокруг одного соответствия: **имя ключа в словаре зависимостей == имя параметра хендлера**. Вы регистрируете словарь `dependencies={"db": Provide(get_db)}`, и любой хендлер (на этом слое или ниже), у которого в сигнатуре есть параметр `db`, получит результат вызова `get_db()`. Связь идёт по совпадению имён, не по типам.

```python
from litestar import Litestar, get
from litestar.di import Provide

async def get_db() -> str:
    return "соединение с БД"

@get("/orders")
async def list_orders(db: str) -> dict:
    return {"db": db}

app = Litestar(route_handlers=[list_orders], dependencies={"db": Provide(get_db)})
```

`Provide(get_db)` оборачивает фабрику. Ключ `"db"` в словаре зависимостей совпадает с именем параметра `db` в хендлере - поэтому фреймворк вызывает `get_db()` и кладёт результат в `db`. Хендлер не знает, как именно создаётся соединение; он просто получает готовое.

<Callout type="info">
Сравнение с FastAPI: там зависимость подставляется через `Depends(get_db)` прямо в значении параметра по умолчанию (`db = Depends(get_db)`), и связь идёт через сам объект `Depends`. В Litestar связь идёт по имени: фабрика регистрируется в словаре `dependencies` под ключом, а параметр хендлера ловит её по совпадению этого ключа со своим именем.
</Callout>

## Зависимости живут на слоях

Самое важное в DI Litestar - зависимости объявляются на любом из четырёх слоёв иерархии (приложение, роутер, контроллер, хендлер), ровно как префиксы пути из раздела о маршрутизации. Зависимость, объявленная на слое, доступна этому слою и всем вложенным.

Из слоистости следует практический навык: общую для всего приложения зависимость (пул БД, конфиг) кладут на `Litestar`, а специфичную для группы эндпоинтов (репозиторий пользователей) - на их контроллер. Чтобы понять, откуда хендлер берёт зависимость, поднимайтесь по слоям - как при чтении пути.

## Нижний слой переопределяет верхний

Если одно и то же имя зависимости объявлено на нескольких слоях, побеждает ближайший к хендлеру - нижний переопределяет верхний. Это позволяет задать зависимость по умолчанию на приложении и подменить её для конкретного контроллера или хендлера.

```python
from litestar import Litestar, Controller, get
from litestar.di import Provide

async def default_repo() -> str:
    return "обычный репозиторий"

async def special_repo() -> str:
    return "специальный репозиторий"

class AdminController(Controller):
    path = "/admin"
    dependencies = {"repo": Provide(special_repo)}

    @get("/")
    async def index(self, repo: str) -> dict:
        return {"repo": repo}

app = Litestar(
    route_handlers=[AdminController],
    dependencies={"repo": Provide(default_repo)},
)
```

На приложении `repo` указывает на `default_repo`, но `AdminController` переопределяет тот же ключ `repo` своим `special_repo`. Хендлер `index` внутри контроллера получит `"специальный репозиторий"` - сработал нижний слой. Хендлер в любом другом месте приложения, без переопределения, получил бы `default_repo`.

<Callout type="warning">
Переопределение работает по совпадению **имени ключа**. Если на контроллере объявить зависимость под чуть другим именем (`repository` вместо `repo`), переопределения не произойдёт - это будет вторая, независимая зависимость, а хендлер с параметром `repo` по-прежнему получит версию с верхнего слоя. Рассинхрон имён - источник «почему не подменилось».
</Callout>

## Sync и async зависимости

Фабрика зависимости может быть и `async def`, и обычной `def`. Async-фабрику фреймворк дожидается (`await`), sync-фабрику выполняет напрямую (а блокирующую - выносит в пул потоков, как и хендлеры). Выбор - по природе работы: async для I/O (поход в базу, HTTP-запрос), sync для чистого вычисления или чтения уже готового значения.

```python
async def get_session() -> str:
    return await open_db_session()

def get_settings() -> dict:
    return {"page_size": 20}
```

`get_session` асинхронна, потому что внутри настоящее ожидание I/O. `get_settings` синхронна - она просто отдаёт готовый словарь, `await` ей не нужен. Объявлять зависимость `async` «для единообразия», если внутри нет ни одного `await`, смысла не имеет - как и с хендлерами.

## Кэширование: use_cache и его опасность

По умолчанию фабрика зависимости вызывается заново на каждый запрос. Флаг `use_cache=True` меняет это: результат фабрики кэшируется и переиспользуется. Это экономит работу для дорогих, но неизменных значений.

```python
from litestar.di import Provide

Provide(load_config, use_cache=True)
```

Кэширование уместно для значений, которые действительно одинаковы между запросами: разобранный конфиг, справочник, подключение к пулу. Но оно опасно для всего, что зависит от конкретного запроса или должно быть свежим: если закэшировать «текущего пользователя» или «открытую транзакцию», все запросы начнут видеть значение из первого вызова - один пользователь получит данные другого, транзакции перепутаются. Кэш зависимости - это «вычисли один раз на всё приложение», а не «один раз на запрос».

<PullQuote>
use_cache говорит «это значение одинаково для всех запросов». Если это неправда - Вы только что показали данные одного пользователя всем остальным.
</PullQuote>

## Вложенные зависимости

Зависимость может сама зависеть от другой: её фабрика объявляет в своей сигнатуре параметр с именем другой зависимости, и фреймворк разрешает цепочку. Так строят слои: «сессия» зависит от «движка», «репозиторий» - от «сессии».

```python
from litestar import Litestar, get
from litestar.di import Provide

async def get_engine() -> str:
    return "engine"

async def get_session(engine: str) -> str:
    return f"session({engine})"

async def get_repo(session: str) -> str:
    return f"repo[{session}]"

@get("/users")
async def list_users(repo: str) -> dict:
    return {"repo": repo}

app = Litestar(
    route_handlers=[list_users],
    dependencies={
        "engine": Provide(get_engine),
        "session": Provide(get_session),
        "repo": Provide(get_repo),
    },
)
```

Хендлеру нужен только `repo`. Но `get_repo` объявляет параметр `session`, а `get_session` - параметр `engine`. Фреймворк разрешает цепочку снизу вверх: создаёт `engine`, передаёт его в `get_session`, результат передаёт в `get_repo`, и уже готовый `repo` отдаёт хендлеру. Параметр одной фабрики, совпадающий по имени с ключом другой зависимости, - это и есть объявление «я завишу от неё».

## Краевые случаи

**Переопределение не «дополняет», а полностью заменяет.** Нижний слой не комбинирует свою фабрику с верхней - он целиком вытесняет её для своего поддерева. Если на контроллере переопределить `repo`, верхняя фабрика для его хендлеров просто не вызывается; вернуться к ней на этом поддереве уже нельзя.

**`use_cache` кэширует на всё приложение, а не на запрос.** Распространённое заблуждение - что кэш живёт в пределах одного запроса. Нет: закэшированное значение переживает запрос и отдаётся всем последующим. Для «один раз за запрос» нужен не кэш, а просто обычная (некэшируемая) зависимость - в рамках одного запроса повторно к ней не обратятся без нужды, а между запросами она пересоздаётся.

**Имя параметра - это контракт.** Поскольку связывание идёт по имени, переименование параметра хендлера (`db` → `database`) тихо разрывает связь с зависимостью `db`: фреймворк больше не находит ключ под новым именем. Никакой тип не спасёт - сопоставление имён, а не типов.

## Типичные ошибки

- **Создают ресурс внутри хендлера вместо зависимости.** Открывать соединение прямо в теле хендлера - значит дублировать инициализацию и лишать себя возможности подменить ресурс в тестах. Ресурс должен приходить через `Provide`, тогда тест подставит фейковую фабрику.
- **Объявляют зависимость не на том слое.** Зависимость, нужную только одному контроллеру, незачем класть на всё приложение (засоряет глобальное пространство имён); а общую для всех - незачем дублировать на каждом контроллере. Слой выбирают по области видимости зависимости.
- **Кэшируют запросозависимое значение.** `use_cache=True` на «текущем пользователе», «сессии», «времени запроса» - источник утечки данных между запросами. Кэшируйте только то, что одинаково для всех.
- **Рассинхронизируют имя ключа и имя параметра.** Зависимость зарегистрирована как `"user_repo"`, а параметр назван `repo` - связи нет, фреймворк не подставит значение. Имя ключа и имя параметра обязаны совпадать.

<Detail summary="Почему слоистый DI и как он соотносится с тестируемостью">

Слоистый DI - это применение того же принципа, что и слоистая маршрутизация: факт о группе эндпоинтов записывается в одном месте, на нужном уровне. Общая зависимость на приложении не повторяется в каждом хендлере; специфичная на контроллере не загрязняет остальное приложение. Но настоящая выгода DI - в тестировании. Когда хендлер сам создаёт соединение с базой, протестировать его в изоляции нельзя: тест неизбежно поднимет настоящую базу. Когда соединение приходит через `Provide`, тест собирает мини-приложение и регистрирует под тем же ключом фабрику, возвращающую фейк, - и хендлер, ничего не зная, работает с подделкой. Именно поэтому DI и явная регистрация хендлеров (из первого раздела) - две стороны одной идеи: код, который объявляет свои потребности декларативно, а не добывает их сам, легко собрать в любой конфигурации, в том числе тестовой. Цена - косвенность: чтобы понять, что реально придёт в `db`, нужно найти, на каком слое и какой фабрикой эта зависимость объявлена, и не перепутать с переопределением ниже.

</Detail>
