# Один объект внутри, разные его лица снаружи

Представьте таблицу пользователей: `id`, `name`, `email`, `password_hash`, `created_at`. Этот объект живёт внутри приложения целиком - с ним работает бизнес-логика, его кладут в базу. Но наружу, в JSON ответа, он не должен выходить как есть: хэш пароля клиенту видеть незачем, а `created_at` сервер проставляет сам, и принимать его в теле запроса на создание - значит дать клиенту переписать момент регистрации. Получается, что одна и та же модель данных должна показываться по-разному в зависимости от того, читают её, создают или обновляют. Если решать это в лоб - руками выкидывать поля в каждом хендлере, - код быстро зарастает однотипными `del data["password_hash"]`, и стоит добавить одно поле, как про него забывают в трёх местах.

Litestar отвечает на это двумя разными механизмами, и важно их не путать. Первый - модель данных и автоматический разбор тела запроса: Вы объявляете тип, фреймворк сам парсит JSON, валидирует и кладёт готовый объект в параметр хендлера. Второй - DTO (Data Transfer Object): отдельный слой, который определяет, какие поля модели включить или исключить на входе и на выходе. К концу раздела Вы будете понимать, как типизированный параметр `data` превращает тело запроса в объект, зачем нужны DTO поверх готовой модели и как настроить их через include/exclude, чтобы внутренняя структура данных не диктовала внешний JSON.

## Ментальная модель: модель - это форма, DTO - это фильтр

Модель данных описывает полную форму объекта: все его поля и их типы. В Litestar по умолчанию её объявляют как `msgspec.Struct` - быстрый аналог датакласса с валидацией. Pydantic-модели тоже работают, если они привычнее. Модель отвечает на вопрос «какие поля вообще есть у объекта».

DTO отвечает на другой вопрос - «какие из этих полей пропустить через границу приложения в этом конкретном направлении». DTO не описывает новую структуру; он берёт существующую модель и накладывает на неё фильтр: вот эти поля включить, вот эти исключить, вот это переименовать. На вход и на выход фильтры могут быть разными.

Из этой картины следует главное разделение труда: модель - про то, что объект из себя представляет; DTO - про то, как он показывается на границе. Одна модель может иметь несколько DTO для разных операций, и менять их можно независимо.

## Типизированные параметры хендлера

Прежде чем брать тело запроса, вспомним, что Litestar уже умеет определять источник параметра по сигнатуре, как мы видели в маршрутизации: упомянут в пути - path-параметр, примитив не из пути - query. Тело запроса работает по тому же принципу, но с особым именем.

```python
from msgspec import Struct
from litestar import post

class CreateUser(Struct):
    name: str
    email: str

@post("/users")
async def create_user(data: CreateUser) -> CreateUser:
    return data
```

Параметр с именем `data` и типом-моделью - это сигнал «возьми тело запроса, разбери его как `CreateUser`, провалидируй и положи сюда готовый объект». Никакого ручного `request.json()` не нужно: к моменту входа в тело хендлера `data` - уже типизированный объект `CreateUser` с проверенными полями. Аннотация возвращаемого типа `-> CreateUser` говорит, как сериализовать ответ обратно в JSON.

<Callout type="info">
Имя `data` - это соглашение Litestar для тела запроса, как `self` для метода. Параметр именно с этим именем фреймворк связывает с телом; меняя тип у `data`, Вы меняете, во что разбирается тело. Это аналог `body: Model` в сигнатуре хендлера FastAPI, только имя фиксированное.
</Callout>

## msgspec.Struct как модель по умолчанию

Модель объявляется наследованием от `msgspec.Struct`. Поля - это аннотированные атрибуты класса; значение по умолчанию делает поле необязательным.

```python
from msgspec import Struct

class Product(Struct):
    title: str
    price: float
    in_stock: bool = True
```

`title` и `price` обязательны, `in_stock` - нет (по умолчанию `True`). По форме это похоже на датакласс, но msgspec ещё и валидирует типы при разборе JSON и кодирует объект обратно очень быстро - в этом его смысл по сравнению с обычным dataclass. Если в проекте уже принят Pydantic, можно объявлять модели как `BaseModel`: Litestar поддерживает и его, выбор за командой. Менять умолчание на Pydantic стоит осознанно - ради его экосистемы валидаторов, а не «на всякий случай».

## DTO: зачем отдельный слой

Казалось бы, для разных представлений хватит и нескольких моделей - `UserIn`, `UserOut`, как часто делают в FastAPI. DTO выигрывает там, где модель одна и большая, а представлений много: вместо того чтобы копировать поля в три почти одинаковых класса (и ловить рассинхрон при правке), Вы держите одну модель и навешиваете на неё фильтры.

```python
from dataclasses import dataclass
from litestar import get
from litestar.dto import DataclassDTO, DTOConfig

@dataclass
class User:
    id: int
    name: str
    email: str
    password_hash: str

class UserReadDTO(DataclassDTO[User]):
    config = DTOConfig(exclude={"password_hash"})

@get("/users/{user_id:int}", return_dto=UserReadDTO)
async def get_user(user_id: int) -> User:
    return User(id=user_id, name="Анна", email="anna@example.ru", password_hash="secret")
```

Хендлер возвращает полный объект `User` со всеми полями, включая `password_hash`. Но `return_dto=UserReadDTO` с `exclude={"password_hash"}` отфильтровывает это поле на выходе: в JSON попадут `id`, `name`, `email`, а хэш - нет. Бизнес-логика работает с полным объектом, граница приложения видит урезанный. `DataclassDTO` - DTO-фабрика для dataclass-моделей; для `msgspec.Struct` есть `MsgspecDTO`, для Pydantic - `PydanticDTO`.

### include против exclude и rename

`DTOConfig` принимает несколько полей фильтрации. `exclude` перечисляет поля, которые убрать; `include` - наоборот, оставить только перечисленные (всё остальное убрать); `rename_fields` меняет внешние имена полей, не трогая внутренние.

```python
class UserPublicDTO(DataclassDTO[User]):
    config = DTOConfig(
        include={"id", "name"},
        rename_fields={"id": "userId"},
    )
```

Здесь наружу выйдут только `id` и `name`, причём `id` в JSON будет называться `userId`. Внутри объекта поле по-прежнему `id` - переименование живёт только на границе. Выбирайте `include`, когда наружу идёт малая часть большой модели (перечислить разрешённое короче и безопаснее: добавите поле в модель - оно не утечёт само), и `exclude`, когда скрыть нужно одно-два поля из многих.

<Callout type="warning">
`include` и `exclude` - взаимоисключающие способы описать один и тот же фильтр. Перечисляйте поля одним из них, а не обоими сразу: одновременное использование - противоречивая конфигурация, и её стоит избегать.
</Callout>

## Вход и выход - разные DTO

Сила механизма в том, что на приём тела и на отдачу ответа можно повесить разные фильтры. На входе исключают поля, которые клиент не должен задавать (серверные `id`, `created_at`); на выходе - которые не должен видеть (`password_hash`).

```python
class UserCreateDTO(DataclassDTO[User]):
    config = DTOConfig(exclude={"id", "password_hash"})

class UserReadDTO(DataclassDTO[User]):
    config = DTOConfig(exclude={"password_hash"})

@post("/users", dto=UserCreateDTO, return_dto=UserReadDTO)
async def create_user(data: User) -> User:
    return data
```

`dto=UserCreateDTO` управляет разбором тела: клиент не может прислать `id` или `password_hash`, эти поля во входном DTO исключены. `return_dto=UserReadDTO` управляет ответом: `password_hash` не уйдёт обратно, но `id` (проставленный сервером) - уйдёт. Одна модель `User`, два направления, два фильтра.

## DTOData: получить разобранные данные, не собирая объект

Иногда хендлеру нужны не готовый объект модели, а именно те поля, что прислал клиент, - например при частичном обновлении (PATCH), где клиент шлёт только меняющиеся поля. Для этого тип параметра оборачивают в `DTOData`.

```python
from litestar import patch
from litestar.dto import DataclassDTO, DTOData, DTOConfig

class UserPatchDTO(DataclassDTO[User]):
    config = DTOConfig(exclude={"id", "password_hash"}, partial=True)

@patch("/users/{user_id:int}", dto=UserPatchDTO)
async def update_user(user_id: int, data: DTOData[User]) -> dict:
    changes = data.as_builtins()
    return {"user_id": user_id, "changed_fields": list(changes.keys())}
```

`DTOData[User]` даёт доступ к присланным данным как есть, не требуя, чтобы клиент прислал все обязательные поля модели. `partial=True` в конфиге делает все поля необязательными - именно то, что нужно для PATCH. `data.as_builtins()` возвращает разобранные поля словарём, и хендлер сам решает, как применить их к существующей записи. Без `DTOData` фреймворк попытался бы собрать полный `User` и потребовал бы все обязательные поля.

## Ошибки валидации: 422

Если тело запроса не проходит проверку - отсутствует обязательное поле, тип не тот, нарушено ограничение, - Litestar не вызывает хендлер и сам возвращает ответ `400 Bad Request` для большинства ошибок клиента, а для семантически некорректного тела - `422 Unprocessable Entity` с описанием проблемного поля. Хендлеру не нужно ничего проверять руками: до него доходит только валидный объект, иначе клиент получает структурированную ошибку с указанием, что именно не так.

## Краевые случаи

**msgspec строже Pydantic к приведению типов.** Как отмечалось в обзоре, msgspec по умолчанию не станет молча превращать строку `"42"` в число для поля `int`. Тело `{"price": "10.5"}` для поля `price: float` вызовет ошибку валидации, а не тихое приведение. Тех, кто привык к лояльному Pydantic v1, это удивляет: то, что там «просто работало», здесь честно падает с 422.

**Исключённое во входном DTO поле не «обнуляется», а отвергается.** Если входной DTO исключает `id`, а клиент всё равно прислал `id` в теле, поле не молча игнорируется по-тихому везде - в строгой конфигурации лишнее поле вызывает ошибку. Не полагайтесь на то, что «лишнее просто отбросится»; проектируйте входной контракт явно.

**`include`/`exclude` работают по именам полей, а не по их наличию.** Опечатка в имени поля внутри `exclude={"pasword_hash"}` не вызовет ошибку «такого поля нет» - фильтр просто ничего не исключит, и хэш утечёт в ответ. Имена полей в DTO-конфиге - это «магические строки», и их легко рассинхронизировать с моделью при переименовании.

## Типичные ошибки

- **Кладут тело не в `data`.** Параметр с произвольным именем (`payload: CreateUser`) не будет связан с телом запроса так, как `data`. Имя `data` - соглашение Litestar для тела; другое имя фреймворк истолкует иначе.
- **Путают `dto` и `return_dto`.** `dto` фильтрует **вход** (разбор тела), `return_dto` - **выход** (сериализация ответа). Повесив фильтр скрытия пароля на `dto` вместо `return_dto`, Вы защитите вход, но пароль всё равно уйдёт в ответе. Направление выбирается осознанно.
- **Скрывают `password_hash` исключением его из модели.** Если убрать поле из самой модели, с ним не сможет работать и бизнес-логика (проверка пароля). Поле должно жить в модели; прятать его - задача `return_dto`, а не урезание модели.
- **Принимают серверные поля на входе.** Без входного DTO, исключающего `id`/`created_at`, клиент может прислать эти поля и переписать то, что должен ставить сервер. Серверные поля исключают во входном DTO.

<Detail summary="Почему DTO быстрее ad-hoc фильтрации руками">

Соблазн велик: зачем DTO, если можно в хендлере сделать `d = asdict(user); del d["password_hash"]; return d`. На одном эндпоинте это и правда короче. Проблема в трёх вещах. Во-первых, ручная фильтрация повторяется в каждом хендлере, который отдаёт пользователя, - и при добавлении секретного поля его забывают вычистить где-нибудь в одном из десяти мест, что и есть классическая утечка. DTO задаёт фильтр один раз и применяет ко всем хендлерам, где указан. Во-вторых, Litestar генерирует OpenAPI-схему из DTO: документация автоматически показывает урезанную форму, а ручной `del` документация не видит - схема врёт. В-третьих, DTO работают и на входе: ручной фильтр входа пришлось бы писать отдельно и валидировать самому. DTO - это декларативный, документируемый и двунаправленный способ сказать «вот контракт границы», тогда как ручная фильтрация - императивный и легко рассинхронизируемый. Цена DTO - лишний уровень абстракции и знание его API; она окупается ровно тогда, когда представлений модели больше одного.

</Detail>
