# Как из четырёх кусочков пути собирается один адрес

Откройте крупное Litestar-приложение, и Вы не увидите длинного списка хендлеров с полными путями вроде `/api/v1/users/{user_id}/orders`. Вместо этого путь окажется разрезан на части и распределён по слоям: префикс `/api/v1` на роутере, `/users` на контроллере, `/{user_id}/orders` на самом методе. Чтобы понять, какой URL вызовет конкретный метод, нужно мысленно склеить все эти куски сверху вниз. Если этого не делать, легко промахнуться: повесить хендлер на `/orders` и удивляться, почему он отвечает на `/api/v1/users/orders`, а не на том адресе, который Вы видите рядом с методом.

Эта склейка - сердце маршрутизации в Litestar. К концу раздела Вы будете уметь по объявлению хендлера и слоям над ним вычислить итоговый путь, типизировать path- и query-параметры так, чтобы фреймворк сам приводил их к нужному типу, и выбирать между группировкой через `Router` и через класс-контроллер.

## Ментальная модель: путь склеивается по слоям

Каждый слой иерархии - приложение, роутер, контроллер, хендлер - может объявить свой кусок пути. Итоговый путь хендлера - это конкатенация кусков всех слоёв над ним, сверху вниз, в порядке вложенности. Роутер с `path="/api"`, внутри него контроллер с `path="/users"`, внутри него метод `@get("/{user_id:int}")` дают адрес `/api/users/{user_id}`.

Из этой модели сразу следует практический навык: чтобы найти, на какой URL отвечает метод, не смотрите только на его декоратор - поднимитесь по слоям и склейте префиксы. И наоборот: добавив префикс на роутере, Вы сдвигаете все вложенные адреса разом, не трогая хендлеры.

## Хендлеры под каждый метод

В Litestar для каждого HTTP-метода свой декоратор-хендлер: `@get`, `@post`, `@put`, `@delete`, `@patch`. Это отличается от FastAPI, где метод - это имя метода у объекта (`@app.get`, `@app.post`). Импортируются они из `litestar`.

```python
from litestar import get, post, delete

@get("/items")
async def list_items() -> list[dict]:
    return [{"id": 1}]

@post("/items")
async def create_item() -> dict:
    return {"created": True}

@delete("/items/{item_id:int}")
async def delete_item(item_id: int) -> None:
    ...
```

Каждый декоратор привязывает функцию к паре «метод + путь». Имена декораторов отдельные, но идея та же, что и везде: один путь может обслуживаться разными методами через разные хендлеры.

## Path-параметр с типом прямо в шаблоне

Главное синтаксическое отличие от FastAPI - тип path-параметра указывается **в самом шаблоне пути**, а не только в аннотации функции. Шаблон `/{item_id:int}` говорит роутеру: этот сегмент - целое число.

```python
@get("/items/{item_id:int}")
async def read_item(item_id: int) -> dict:
    return {"item_id": item_id}
```

Запрос `GET /items/42` даст `item_id == 42` как целое. Указание `:int` в шаблоне выполняет двойную роль: роутер использует его, чтобы маршрут вообще совпал (на `/items/abc` он не сработает - сегмент не число), а затем приводит значение к типу. Поддерживаются `int`, `str`, `float`, `uuid`, `path` и другие. Аннотация `item_id: int` в сигнатуре согласуется с этим типом, и по ней работает автодокументация.

<Callout type="info">
В FastAPI тип path-параметра живёт только в аннотации функции (`item_id: int`), а в шаблоне стоит голое `{item_id}`. В Litestar тип переносится в шаблон: `{item_id:int}`. Имя при этом по-прежнему связывает сегмент пути с параметром функции.
</Callout>

## Query-параметр: по умолчанию и через Parameter

Если параметр функции не упомянут в пути и имеет примитивный тип, Litestar берёт его из query string - как и FastAPI.

```python
@get("/items")
async def list_items(limit: int = 10, offset: int = 0) -> dict:
    return {"limit": limit, "offset": offset}
```

`GET /items?limit=5` даст `limit == 5`, `offset == 0` (из дефолта). Когда нужно настроить query-параметр тоньше - задать границы, описание для документации, другое внешнее имя, - используют `Parameter`.

```python
from typing import Annotated
from litestar import get
from litestar.params import Parameter

@get("/items")
async def list_items(
    limit: Annotated[int, Parameter(ge=1, le=100)] = 10,
) -> dict:
    return {"limit": limit}
```

`Parameter(ge=1, le=100)` ограничивает `limit` диапазоном от 1 до 100; запрос `?limit=500` получит ошибку валидации. `Annotated[int, Parameter(...)]` - это способ навесить на тип дополнительные правила, не теряя сам тип. Идея как у `Query(...)` в FastAPI, синтаксис через `Annotated`.

## Группировка: Router и Controller

Два способа собрать связанные хендлеры под общий префикс. Первый - `Router`: контейнер с префиксом пути и списком хендлеров.

```python
from litestar import Litestar, Router, get

@get("/")
async def list_users() -> list:
    return []

users_router = Router(path="/users", route_handlers=[list_users])
app = Litestar(route_handlers=[users_router])
```

Хендлер `list_users` с путём `/` под роутером с `path="/users"` даёт итоговый `/users`. Второй способ - **класс-контроллер**: класс с атрибутом `path`, чьи методы-хендлеры наследуют этот префикс.

```python
from litestar import Controller, get, post

class UserController(Controller):
    path = "/users"

    @get("/")
    async def list_users(self) -> list:
        return []

    @post("/")
    async def create_user(self) -> dict:
        return {"created": True}
```

Оба метода живут под `/users`: `list_users` отвечает на `GET /users`, `create_user` - на `POST /users`. Контроллер удобен, когда хендлеры разделяют общий префикс и, как увидим в следующих разделах, общие зависимости или guards. Это близкая аналогия классов-вью в других фреймворках; в FastAPI прямого эквивалента нет - там группируют через `APIRouter`.

## Краевые случаи

**Двойной слэш и пропущенный слэш.** Litestar нормализует склейку путей, но мысленно держите слэши под контролем: контроллер с `path="/users"` и метод `@get("")` дают `/users`, а `@get("/")` - тоже `/users` (хвостовой слэш нормализуется). Сюрпризы чаще возникают, когда кусок пути случайно записан без ведущего слэша.

**`:int` в шаблоне меняет совпадение маршрута, а не только тип.** Без `:int` сегмент совпадёт с любой строкой, и `int`-приведение произойдёт уже по аннотации. С `:int` роутер вообще не сопоставит маршрут для нечислового сегмента - запрос уйдёт в 404, а не в ошибку валидации. Это разные исходы для клиента, и выбор типа в шаблоне на них влияет.

**Порядок специфичности маршрутов.** Если есть и `/items/{item_id:int}`, и `/items/new`, статический сегмент `new` не должен случайно попасть в параметр. Litestar разрешает такие конфликты по специфичности, но осознанно проектируйте пути, чтобы статические и параметрические сегменты не перекрывались двусмысленно.

## Типичные ошибки

- **Читают путь только по декоратору хендлера.** Метод помечен `@get("/orders")`, но он внутри контроллера с `path="/users/{user_id:int}"`, и реальный адрес - `/users/{user_id}/orders`. Всегда склеивайте слои сверху вниз, прежде чем делать вывод об URL.
- **Пишут тип в аннотацию, но забывают в шаблоне.** `@get("/items/{item_id}")` с `item_id: int` тоже сработает (приведение по аннотации), но привычнее и точнее для Litestar писать тип в шаблоне: `{item_id:int}`. Несогласованность шаблона и аннотации - источник путаницы при чтении.
- **Дублируют префикс на каждом методе.** Если у пяти методов путь начинается с `/users`, не повторяйте `/users` в каждом декораторе - вынесите его в `path` контроллера или роутера. Дублирование рассинхронизируется при первой же правке.

<Detail summary="Зачем разделять путь по слоям, если можно писать целиком">

Полный путь в каждом декораторе кажется проще: видно сразу весь адрес. Но он плохо масштабируется. Версионирование API (`/api/v1` → `/api/v2`) при полных путях требует править каждый хендлер; при префиксе на роутере - одну строку. Перенос группы эндпоинтов под другой базовый путь - то же самое: меняется один слой, а не десятки хендлеров. Общие для группы настройки (зависимости, guards, обработчики исключений, теги для документации) тоже естественно живут на слое, а не копируются в каждый метод. Слоистая маршрутизация - это применение принципа «не повторяйся» к адресам и настройкам: каждый факт о группе эндпоинтов записан в одном месте. Цена - необходимость держать иерархию в голове при чтении, поэтому осмысленные имена слоёв и неглубокая вложенность тут важнее, чем где-либо.

</Detail>
