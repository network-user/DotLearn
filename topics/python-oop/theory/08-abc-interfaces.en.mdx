# ABC and abstract methods: a contract that fails early

You are building a notification system: every channel (email, SMS, push) must be able to `send`. As long as the agreement rests on good faith, someone will add a channel without `send`, the object will happily be created, and the failure will strike deep in production when the first message reaches that channel. Abstract base classes (ABCs) turn a spoken agreement into a checked one: a class with an unfilled obligation cannot even be created. The error surfaces at the boundary, at the moment of `Channel()`, not somewhere downstream. By the end of this section you will know how to declare such a contract, make not only methods but also properties abstract, plug in third-party types via `register`, take ready-made interfaces from `collections.abc`, and choose deliberately between an ABC and a `Protocol`.

## A contract checked when the object is created

An abstract base class describes what a subclass must be able to do while leaving the implementation empty. A method marked `@abstractmethod` is a hole in the contract. As long as even one such hole is open, Python refuses to create an instance and raises `TypeError`, listing the unfilled names. The check happens once, at instantiation, and catches a forgotten implementation immediately instead of on the first call to the method.

## `abc.ABC` and `@abstractmethod`

You declare the contract by inheriting from `abc.ABC` and marking methods with the `@abstractmethod` decorator:

```python
from abc import ABC, abstractmethod

class Notifier(ABC):
    @abstractmethod
    def send(self, message):
        ...
```

An attempt at `Notifier()` now fails with `TypeError: Can't instantiate abstract class Notifier with abstract method send`. The class exists and is fit to be a parent, but it cannot be a standalone object. A subclass comes to life the moment it closes every hole:

```python
class Email(Notifier):
    def send(self, message):
        return f"email: {message}"
```

`Email("hi")` is created and works. One subtlety decides everything - `@abstractmethod` on its own forbids nothing. The enforcement lives in the `ABCMeta` metaclass, and inheriting from `ABC` is what supplies it. You can hang the decorator on a method of an ordinary class, but instantiation will pass quietly and the contract turns out decorative.

## Try it: storage with a required interface

The base `Storage` requires `save` and `load`. The concrete `InMemory` implements them, while a bare `Storage()` cannot be created:

Remove the `load` method from `InMemory` and run again: now `InMemory()` itself refuses to be created, because one hole in the contract stayed open. Put the method back and try adding a third abstract method to `Storage`.

## Not only methods can be abstract

The contract can require a property or a class method. Decorators stack, and the order matters: `@abstractmethod` must sit closest to the function, at the bottom.

```python
class Config(ABC):
    @property
    @abstractmethod
    def name(self):
        ...

    @classmethod
    @abstractmethod
    def from_file(cls, path):
        ...
```

A subclass must supply both the `name` property and the `from_file` class method. The wrong decorator order (`@abstractmethod` on top) quietly breaks abstractness: the class stops requiring an implementation, and the forgotten method slips into production again.

## `register` and virtual subclasses

Sometimes a type is already written and you cannot change its hierarchy, yet by meaning it fits the contract. `ABC.register` declares it a virtual subclass:

```python
class JsonFile:
    def save(self, key, value): ...
    def load(self, key): ...

Storage.register(JsonFile)

isinstance(JsonFile(), Storage)   # True
issubclass(JsonFile, Storage)     # True
```

After `register`, the `isinstance` and `issubclass` checks return `True`, even though `JsonFile` does not inherit `Storage`. The price of the flexibility is honest - no methods are inherited (you get no mixins), and there is no contract check at all. You can register even a class without `save` and `load`, and `isinstance` will still lie that it is a `Storage`. This is exactly how the standard library teaches built-in `list` and `dict` to answer to abstract interfaces without rewriting them.

## `collections.abc`: ready-made interfaces from the standard library

You rarely need to write your own ABC - for collections it is all assembled in `collections.abc`. There you find `Iterable`, `Iterator`, `Sequence`, `Mapping`, `Set`, and more. Inheriting from them is useful for two reasons at once, and the second matters more than the first.

```python
from collections.abc import Mapping

class FrozenDict(Mapping):
    def __init__(self, data):
        self._data = dict(data)

    def __getitem__(self, key):
        return self._data[key]

    def __iter__(self):
        return iter(self._data)

    def __len__(self):
        return len(self._data)
```

You implemented three methods and got a full mapping. `Mapping` gives you the mixin methods `keys`, `values`, `items`, `get`, `__contains__`, `__eq__` for free, built on top of your three. That is the first payoff. The second - `isinstance(FrozenDict(...), Mapping)` is now `True`, and your type honestly passes checks that expect a mapping. Inheriting from `collections.abc` gives you both free behavior and a place in the type system in one move.

## ABC vs `Protocol`: nominal and structural typing

An ABC defines **nominal** typing: a type counts as a subtype only if it explicitly inherits the base class or is registered via `register`. Kinship is declared by name. `typing.Protocol` defines **structural** typing: a type fits if it has the right shape (those methods and attributes), with no inheritance at all. It is formalized duck typing, and static analyzers check it, not the runtime.

## Edge cases

The decorator without the metaclass is useless - `@abstractmethod` on a method of a class that does not inherit `ABC` and does not set `metaclass=ABCMeta` does not block instantiation. It is `ABCMeta` that provides abstractness.

`register` checks nothing. It merely adds a type to the list of virtual subclasses, so `isinstance` after registration can hand out a false promise. You cannot rely on it as a guarantee that methods are present.

An abstract method may have a body. A subclass must still override it, but it can call the base's shared logic via `super().method(...)`.

## Common mistakes

- **Forgot to inherit `ABC`.** A lone `@abstractmethod` without `ABCMeta` does not forbid creating an object. Inherit `abc.ABC` or set `metaclass=ABCMeta`.
- **Wrong decorator order.** For an abstract property or class method, `@abstractmethod` goes closest to the function (at the bottom), otherwise abstractness is lost.
- **Mistook `register` for inheritance.** A virtual subclass gets no methods from the base and is not checked against the contract. If you need mixins or a guarantee of implementation, you need ordinary inheritance.
