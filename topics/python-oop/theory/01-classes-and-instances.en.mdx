# A class is a blueprint, an instance is a thing built from it

You are writing a program about visit counters. There are three of them: on the home page, in the cart, on the checkout page. You could declare three variables and three sets of functions that increment them, but they will drift apart at the very first edit. A class solves this differently: you describe once *what a counter is* - what data it holds and what it can do - and then stamp out as many separate counters as you like, each with its own state.

By the end of this section you will be able to define a class, understand who `self` is and why it is passed explicitly, tell an instance attribute from a class attribute, explain when two names point to one object and when to two different ones, and avoid three traps that almost every beginner stumbles on.

## One blueprint, many instances

A class is a description, not a thing. By itself it counts nothing, just as a house blueprint by itself houses no one. The thing appears when you *create an instance* - a concrete object built from the blueprint and holding its own values.

```python
class Counter:
    def __init__(self, start=0):
        self.value = start

    def tick(self):
        self.value += 1
```

The `class` keyword introduces a new type; by convention class names are written in `PascalCase`. Inside there are two methods. `__init__` is the constructor: it runs once at the moment an instance is created and hands it its starting attributes. `tick` is an ordinary method that changes the state of an already existing object.

An instance is created by calling the class as if it were a function. The arguments go to `__init__`.

```python
c = Counter(10)
c.tick()
c.tick()
print(c.value)
```

This prints `12`. And the key property becomes visible once there are several instances: each one has its own attributes, and state does not leak between them.

```python
a, b = Counter(), Counter()
a.tick()
print(a.value, b.value)
```

Here the output is `1 0`: `a.tick()` touched only `a`. One class - any number of independent objects. Watch this live: press "create an instance" and see separate counters come out of a single blueprint, each with its own address and its own `value`.

## self is the instance itself, passed explicitly

A beginner's most frequent question: where does `self` come from in every method, and why is it not passed at the call site? The answer is that `c.tick()` is shorthand. Python expands it into `Counter.tick(c)`: the method lives on the class, and the instance before the dot is substituted as the first argument. So `self` is not a magic word but an ordinary parameter into which Python puts "the object the method was called on".

```python
c = Counter(10)
c.tick()
Counter.tick(c)
print(c.value)
```

Both forms do the same thing; this prints `12`. The name `self` is merely a convention; technically any name could stand in its place, but you should not change it: all Python code is read with `self` in mind. This also explains why `self.value` inside `tick` and `c.value` outside are the same attribute: both reach the same object, just under different names.

<Callout type="info">The expression `c.value` is not "the variable value inside c". It means "on the object that `c` looks at, find the attribute `value`". If tomorrow a name `d` looks at the same object, then `d.value` will read exactly the same thing.</Callout>

## Two names - one object or two objects?

Since `self` and `c` are just names pointing at an object, a question arises: when do we really have two different counters, and when two names for one? The assignment `d = c` does not create a copy - it hangs a second label on the same object.

```python
c = Counter(10)
d = c
d.tick()
print(c.value, c is d)
```

This prints `11 True`. `d.tick()` incremented the single object, and `c` saw it, because `c` and `d` are two names for one thing. The `is` operator asks exactly this: "is this literally the same object?". Compare with genuinely two objects:

```python
e = Counter(10)
f = Counter(10)
print(e.value == f.value, e is f)
```

Here it is `True False`: the values match, but these are different objects with different addresses in memory. The figure shows the difference - the names `c` and `d` converge on one cell, while `e` and `f` point to different ones.

Keep the difference in mind: `is` is about object identity, `==` is about value equality. For counters without special configuration, `==` will also reduce to identity, and that trap is covered below.

## Instance attributes and class attributes

Attributes set through `self.x = ...` inside `__init__` belong to the instance: each object has its own `value`. But an attribute can also be declared at the class level - then there is one for all.

```python
class Pet:
    species = "unknown"

    def __init__(self, name):
        self.name = name
```

`species` is written in the class body, so `Pet.species` is one value visible from any instance. `name` is set in `__init__`, so every pet has its own. Class attributes are handy for shared constants and default values: there is no need to duplicate them in every object.

```python
rex = Pet("Rex")
murka = Pet("Murka")
print(rex.name, murka.name, rex.species)
```

This prints `Rex Murka unknown`: the names differ, the species is shared. When you read `rex.species`, Python does not find `species` on the object itself and goes up to look for it on the class - that is how a shared attribute becomes accessible through any instance.

## Methods are just functions living on the class

It is worth nailing down: a method is not "built into" every object. There is one of it, it lives on the class, and through the dot on an instance it is called with `self` substituted. Attributes can also be created outside `__init__` - the assignment `self.x = ...` in any method adds an attribute to the object at the moment of execution.

```python
class Profile:
    def __init__(self, name):
        self.name = name

    def activate(self):
        self.active = True
```

After `__init__` the profile has only `name`. The `active` attribute appears only after `activate()` is called - before that, accessing it raises `AttributeError`. That is why it is good form to declare all attributes in `__init__`, even with a value of `None`: then the object has a predictable shape from birth.

## When the behavior surprises you

A class attribute shared by all is a great tool as long as the value is immutable. With a mutable one (list, dict) it turns into a trap: all instances share the same object, and an edit through one of them is visible to all the rest.

```python
class Cart:
    items = []

    def add(self, product):
        self.items.append(product)

a = Cart()
b = Cart()
a.add("apple")
print(b.items)
```

This prints `['apple']`, even though nothing was put into `b`. The list `items` was created once at the class level; `self.items.append(...)` does not create a new attribute on the instance but *changes* that very shared list. The cure is to declare mutable state in `__init__`: move `self.items = []` into the constructor, and each cart will get its own list.

<Callout type="warning">A mutable class attribute (`items = []`, `cache = {}`) is almost always a bug. Immutable default values at the class level are safe; mutable ones go in `__init__`.</Callout>

The second surprise is comparison. Without its own `__eq__`, the `==` operator for your objects reduces to `is`, that is, to identity.

```python
print(Counter(5) == Counter(5))
```

This prints `False`: the values are the same, but the objects are different, and Python does not assume equality "by content" out of the box. How to teach a class a meaningful `==` is covered in the [[dunder-methods]] section.

## Common mistakes

- **Forgetting `self` in a method signature.** If you write `def tick():`, then when calling `c.tick()` Python still substitutes the instance as the first argument - and you get `TypeError: tick() takes 0 positional arguments but 1 was given`. The first parameter of any instance method is `self`.
- **A mutable default argument.** `def __init__(self, items=[])` creates a list once when the function is defined, and it becomes shared across all calls where the argument is omitted - the same effect as a mutable class attribute. Write `items=None`, and inside `self.items = items or []`.
- **Confusing a class with an instance.** `c = Counter` (without parentheses) assigns the class itself to `c`, not an object built from it; `c.tick()` then fails, since there is nothing to be `self`. Calling a class always uses parentheses: `Counter()`.

<Detail summary="Where attributes physically live">

For most objects, attributes are stored in a `__dict__` dictionary. An instance's `__dict__` contains only what was assigned through `self.x = ...`; a class's contains methods and class attributes.

```python
c = Counter(10)
print(c.__dict__)
print('tick' in Counter.__dict__)
```

This prints `{'value': 10}` and `True`: the value lives on the instance, and the `tick` method on the class. When you write `c.tick`, Python does not find `tick` in `c.__dict__`, goes up to the class, finds the function there and binds it to `c`. This same "instance -> class" climb explains why a class attribute is visible through any instance.

</Detail>
