Imagine a project that already has three visit counters, a balance per user, and orders, carts and payments scattered nearby. As long as all of that lives in loose variables and functions, every edit spreads across dozens of places: you fix one check, forget the second, and on Friday evening the tests fail with a negative balance. The object-oriented approach gathers data and behaviour into one thing - an object that owns its own state and guards it so nothing can break it from outside.

**What you will be able to do.** Describe a type once and stamp out any number of independent objects; reuse behaviour through inheritance instead of copying code; teach your own classes to work with `+`, `==`, `print` and `len`; hide internal state behind a contract with checks; replace boilerplate with a single `@dataclass` line; and assemble complex objects from small replaceable parts instead of a tower of subclasses.

**The route through this topic.** Eight sections, each with a sandbox where you change the code and run it right in the browser. The last two concepts are advanced: they dig deeper than the rest and assume you already have the basic five down.

1. **Classes and instances** - describe what a counter is once, then create as many separate counters as you want, each with its own state.
2. **Inheritance** - declare `Cat` a special case of `Animal` and get its behaviour without copying a single line.
3. **Dunder methods** - teach your `Money` to add with `+` and print readably instead of `<Money object at 0x...>`.
4. **Encapsulation** - funnel every write to a balance through one checked point so a negative value becomes impossible.
5. **Dataclasses** - replace the routine `__init__`, `__repr__` and `__eq__` with the `@dataclass` decorator.
6. **Composition** - drop the same engine into both a car and a boat, building behaviour from parts instead of a hierarchy.
7. **Descriptors** - write a value check once as a descriptor field and see the shared mechanism behind `property`, methods, and `cached_property`.
8. **ABC and abstract methods** - describe a contract that cannot be broken silently: a class with an unfilled method simply cannot be created.

**What you need first.** Nothing to install: Python runs right in the browser. You only need to write simple functions and work with lists and dictionaries - everything about classes is covered along the way, moving from a single object to how objects interact.
