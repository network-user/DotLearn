# Дескрипторы: как один объект управляет доступом к атрибуту

В классе профиля три поля с одинаковой проверкой: возраст от 0 до 150, рейтинг от 0 до 100, скидка от 0 до 100. Через `property` каждое поле требует пары «геттер плюс сеттер», и в каждом сеттере вы переписываете одну и ту же проверку диапазона. Три поля, три почти одинаковых куска кода, и стоит завести четвёртое поле в другом классе, как копирование начинается заново. Дескрипторы позволяют описать проверку один раз, как самостоятельный объект-поле, и переиспользовать его где угодно. К концу раздела вы увидите, что `property` и обычный метод устроены на одном механизме, научитесь читать порядок поиска атрибутов и напишете валидируемое поле, которое подключается одной строкой.

## Обращение к атрибуту не сводится к чтению из словаря

Кажется, будто `user.age` просто достаёт значение из `user.__dict__["age"]`. На самом деле за точкой стоит вызов: интерпретатор исполняет `type(user).__getattribute__(user, "age")`, и уже этот метод решает, откуда взять значение. Первым делом он смотрит не в словарь экземпляра, а в классы по цепочке наследования (MRO). Если там нашёлся объект с методом `__get__`, значит перед нами дескриптор, и доступ идёт через него, а не через словарь.

Дескриптор - это объект, который лежит атрибутом класса и определяет хотя бы один из методов `__get__`, `__set__`, `__delete__`. Пока такой объект отвечает за имя, каждое чтение и запись этого имени на экземпляре проходит через его методы. Именно так поле получает возможность проверять, логировать или вычислять значение вместо простого хранения.

## Протокол: `__get__`, `__set__` и `__set_name__`

Соберём поле по шагам. Минимальный дескриптор отвечает на чтение и запись:

```python
class Field:
    def __get__(self, obj, objtype=None):
        return obj.__dict__[self.name]

    def __set__(self, obj, value):
        obj.__dict__[self.name] = value
```

`__get__` получает сам экземпляр `obj` и его тип, `__set__` получает экземпляр и присваиваемое значение. Значение мы храним в словаре самого экземпляра, а не в дескрипторе. Это принципиально: дескриптор - один на весь класс, экземпляров много, и если сложить данные в дескриптор, они станут общими для всех.

Откуда взялось `self.name`? Его подставляет третий метод протокола. Когда интерпретатор создаёт класс, он для каждого дескриптора вызывает `__set_name__` и сообщает, под каким именем тот записан:

```python
class Field:
    def __set_name__(self, owner, name):
        self.name = name
```

Теперь `Field`, присвоенный атрибуту `age`, сам узнает строку `"age"` и положит значение в `obj.__dict__["age"]`, без единой захардкоженной строки. Запись напрямую в `obj.__dict__` не запускает протокол заново: это операция со словарём, а не обращение к атрибуту, поэтому рекурсии здесь нет. А поскольку наш `Field` определяет `__set__`, он остаётся главным при чтении, и приватная ячейка в словаре целиком под его контролем.

## data-дескриптор против non-data и порядок поиска

Дескрипторы делятся на два вида, и разница решает, кто победит в конфликте со словарём экземпляра.

Non-data дескриптор определяет только `__get__`. Он уступает словарю экземпляра: если в `obj.__dict__` есть ключ с тем же именем, вернётся значение из словаря, а `__get__` даже не вызовется. Data-дескриптор определяет `__set__` или `__delete__`, пусть даже с телом-заглушкой. Он имеет приоритет над словарём экземпляра: чтение всегда идёт через `__get__`, что бы ни лежало в `obj.__dict__`.

Полный порядок поиска `obj.attr` такой: сначала data-дескриптор из класса, затем словарь экземпляра, затем non-data дескриптор или обычный атрибут класса. Эта иерархия - не академическая деталь. Она объясняет разом поведение `property`, обычных методов и кэшей.

## `property` это data-дескриптор

`property` реализует и `__get__`, и `__set__`, и `__delete__`, а значит, это data-дескриптор. Отсюда его надёжность: сеттер нельзя обойти, записав что-нибудь напрямую в `obj.__dict__` под тем же именем, потому что чтение всё равно пройдёт через `__get__`. Когда вы пишете `@property`, вы создаёте частный случай дескриптора, где `__get__` вызывает вашу функцию-геттер. Всё, что вы узнали про поля, применимо к `property` дословно.

## Функция это non-data дескриптор, поэтому рождается bound method

Обычная функция в теле класса тоже дескриптор: у типа `function` есть `__get__` и нет `__set__`, то есть перед нами non-data дескриптор. Когда вы пишете `user.greet`, доступ идёт через `function.__get__(user, User)`, и возвращается не сама функция, а связанный метод (bound method), у которого первый аргумент `self` уже подставлен как `user`. Обращение к тому же имени через класс, `User.greet`, вызывает `__get__` с `obj=None` и отдаёт обычную функцию.

```python
class User:
    def greet(self):
        return "hi"

u = User()
type(u.greet)                    # method
type(User.greet)                 # function
u.greet.__func__ is User.greet   # True
```

Bound method создаётся заново на каждое обращение, поэтому `u.greet is u.greet` даёт `False`, хотя внутри у них одна и та же функция. То, что метод «знает свой `self`», - не магия строчки `def`, а прямое следствие протокола дескриптора.

## `functools.cached_property`: считаем один раз

`cached_property` опирается ровно на асимметрию data и non-data. Это non-data дескриптор: у него есть только `__get__`. При первом чтении он вычисляет значение, кладёт его в `obj.__dict__` под тем же именем и возвращает. При втором чтении срабатывает порядок поиска: non-data дескриптор уступает словарю экземпляра, поэтому `__get__` больше не вызывается, а значение берётся прямо из словаря. Вычисление происходит один раз, дальше идёт обычный быстрый доступ.

```python
from functools import cached_property

class Dataset:
    def __init__(self, rows):
        self.rows = rows

    @cached_property
    def total(self):
        return sum(self.rows)
```

Плата за скорость известна заранее. Экземпляру нужен `__dict__` (с `__slots__` без словаря `cached_property` не сработает), а имя поля в словаре должно оставаться свободным. Если вычисленное значение способно устареть, кэш придётся сбрасывать вручную через `del obj.total`.

## Краевые случаи

Дескриптор работает, только когда лежит атрибутом класса. Присвоить `Field()` внутрь `__init__` через `self.age = Field()` бессмысленно: `__get__` и `__set__` срабатывают лишь для атрибутов, найденных на типе, а не в словаре экземпляра.

<Callout type="warning" title="Ловушка общего состояния">
Хранить значение в самом дескрипторе (`self.value = value`) означает сделать поле общим на весь класс: дескриптор один, и второй экземпляр затрёт значение первого. Данные всегда кладите в `obj.__dict__`, ключом от `__set_name__`.
</Callout>

Ещё одна ловушка - бесконечная рекурсия. Если внутри `__get__` написать `getattr(obj, self.name)` с публичным именем, вы снова разбудите дескриптор, и он позовёт сам себя. Обращайтесь напрямую к `obj.__dict__`, минуя протокол.

## Типичные ошибки

- **Состояние в дескрипторе вместо экземпляра.** `self.value = value` делает поле общим на весь класс. Храните данные в `obj.__dict__[self.name]`, тогда у каждого экземпляра своё значение.
- **Дескриптор на экземпляре.** `self.age = Field()` в конструкторе не включает протокол. Дескриптор обязан быть атрибутом класса.
- **Захардкоженное имя хранилища.** Фиксированная строка вместо `__set_name__` сломается, стоит переименовать поле или завести второй такой же атрибут. Пусть имя приходит из `__set_name__`.
