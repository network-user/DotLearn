# Наследование: поиск по цепочке, а не копирование кода

Вы написали класс `Dog`. Потом понадобился `Cat`, и половина кода совпала: конструктор с именем, метод `eat`, метод `sleep`. Скопировать класс целиком - значит навсегда чинить каждую ошибку дважды. Наследование решает эту проблему иначе: один класс объявляется частным случаем другого и получает доступ ко всему его поведению, не копируя ни строчки.

К концу раздела Вы сможете объявить подкласс, переопределить метод, расширить родительский конструктор через `super().__init__`, предсказать, чья версия метода выполнится в любой точке иерархии, и распознать ситуации, в которых наследовать не нужно вовсе.

## Что на самом деле делает class Dog(Animal)

Запись `class Dog(Animal)` читается как «Dog - это Animal»: отношение **is-a**. Важнее то, чего при этом не происходит. Python не копирует методы `Animal` внутрь `Dog` - он лишь запоминает ссылку на родителя. Вся работа откладывается до момента обращения к атрибуту.

```python
class Animal:
    def __init__(self, name):
        self.name = name

    def speak(self):
        return f"{self.name} издаёт звук"

class Dog(Animal):
    pass

d = Dog("Rex")
print(d.speak())
```

Программа печатает `Rex издаёт звук`. Откуда у пустого класса взялся `speak`? Когда Вы пишете `d.speak()`, Python идёт по цепочке классов: сначала `Dog`, затем `Animal`, затем `object` - и берёт первое совпадение. Проследите маршрут на схеме: в `Dog` метода нет, поиск продолжается, в `Animal` находится - и на этом останавливается.

`object` - последнее звено любой цепочки: от него неявно наследуется каждый класс в Python. Если поиск дошёл до конца и ничего не нашёл, Вы получаете `AttributeError`.

<MarginNote label="object">Даже `class A: pass` наследуется от `object` - оттуда приходят `__repr__`, `__eq__` и другие методы по умолчанию.</MarginNote>

То, что это поиск в момент вызова, а не копирование в момент создания класса, легко доказать: добавьте метод в `Animal` после того, как `Dog` уже определён и экземпляр уже создан.

```python
def eat(self):
    return f"{self.name} ест"

Animal.eat = eat
print(d.eat())
```

Печатается `Rex ест`. Никто ничего не пересобирал: просто очередной поиск по цепочке нашёл новое имя в `Animal`.

<PullQuote>Наследование - это не копирование кода, а маршрут, по которому ищется имя.</PullQuote>

## Переопределение: подкласс отвечает первым

Раз поиск идёт от подкласса вверх, подкласс может перехватить любой метод - достаточно определить одноимённый у себя. Это называется переопределением (override).

```python
class Dog(Animal):
    def speak(self):
        return f"{self.name} лает"

print(Dog("Rex").speak())
```

Теперь поиск `speak` останавливается уже в `Dog`, и печатается `Rex лает`. Версия из `Animal` никуда не делась - до неё просто не доходит очередь.

## super(): расширить, а не заменить

Часто нужно не заменить родительское поведение, а достроить его. Для этого есть `super()`: он возвращает прокси-объект, поиск через который начинается не с класса экземпляра, а со следующего звена цепочки.

```python
class LoudDog(Dog):
    def speak(self):
        return super().speak().upper()

print(LoudDog("Rex").speak())
```

`super().speak()` пропускает `LoudDog`, находит `speak` в `Dog`, получает `Rex лает` и переводит строку в верхний регистр: печатается `REX ЛАЕТ`. В этом и состоит разница между «заменить» и «расширить»: метод-замена игнорирует родителя, метод-расширение вызывает `super()` и дополняет результат.

Современная форма вызова - пустые скобки: `super().speak()`, а не `super(self).speak()` и не `Dog.speak(self)`. Жёсткое имя родителя в коде метода сломается при перестройке иерархии.

## Самый важный частный случай: super().__init__

`__init__` - обычный метод, и на него действуют те же правила поиска. Отсюда главный подводный камень наследования: как только подкласс определяет собственный `__init__`, родительский перестаёт вызываться автоматически.

```python
class Robot(Animal):
    def __init__(self, serial):
        self.serial = serial

r = Robot("R2-D2")
print(r.speak())
```

Этот код падает с `AttributeError: 'Robot' object has no attribute 'name'`. Конструктор `Animal`, который создавал `self.name`, не выполнился ни разу: поиск `__init__` нашёл версию в `Robot` и остановился. Атрибут не «сломался» - он никогда не существовал.

Лекарство - явно передать работу родителю первой строкой конструктора.

```python
class Robot(Animal):
    def __init__(self, name, serial):
        super().__init__(name)
        self.serial = serial
```

Теперь `Animal.__init__` создаёт `name`, после чего `Robot` добавляет своё. Правило простое: переопределили `__init__` - вызовите `super().__init__(...)` с теми аргументами, которые ждёт родитель.

## isinstance и issubclass

Отношение is-a можно проверять во время выполнения. `isinstance(obj, Cls)` отвечает, является ли объект экземпляром класса или любого его подкласса; `issubclass(Sub, Base)` - входит ли `Base` в цепочку наследования `Sub`.

```python
d = Dog("Rex")
print(isinstance(d, Dog), isinstance(d, Animal))
print(issubclass(Dog, Animal), issubclass(Animal, Dog))
```

Первая строка печатает `True True`: экземпляр подкласса - полноценный экземпляр базового класса. Вторая печатает `True False`: наследование направлено в одну сторону. Предпочитайте `isinstance` сравнению `type(d) == Dog` - точное сравнение типов ломает главное обещание наследования: подкласс можно подставить везде, где ждут родителя.

На практике обе проверки чаще всего встречаются на границах системы: функция принимает «что-нибудь животное» и хочет убедиться, что ей не передали число, или фреймворк решает, как обращаться с объектом, в зависимости от его места в иерархии. Внутри собственного кода частые `isinstance` - повод насторожиться: возможно, ветвление по типу стоит заменить переопределённым методом, и пусть поиск по цепочке сам выберет нужную ветку.

## Сюрприз: родительский код вызывает ваш метод

Поиск всегда начинается с класса экземпляра - даже когда вызов написан внутри родителя. Это удивляет почти каждого, кто видит впервые.

```python
class Animal:
    def __init__(self, name):
        self.name = name

    def speak(self):
        return f"{self.name} издаёт звук"

    def introduce(self):
        return f"Это {self.name}. {self.speak()}"

class Dog(Animal):
    def speak(self):
        return f"{self.name} лает"

print(Dog("Rex").introduce())
```

`introduce` определён в `Animal` и в `Dog` не переопределён, но печатается `Это Rex. Rex лает`. Внутри `introduce` стоит `self.speak()`, а `self` здесь - экземпляр `Dog`, поэтому поиск стартует с `Dog` и находит переопределённую версию. Родительский код вызвал дочерний метод, хотя `Dog` ещё не существовал, когда писали `Animal`.

Эта же механика накладывает обязательство: переопределяя метод, сохраняйте его контракт - те же аргументы, совместимый результат. Код родителя рассчитывает, что `self.speak()` вернёт строку, и подкласс не вправе вернуть список.

## Типичные ошибки

- **Забыть `super().__init__(...)`.** Симптом - `AttributeError` на атрибуте, который задаётся в родительском конструкторе. Причина в том, что переопределённый `__init__` полностью заслоняет родительский; почините первой строкой `super().__init__(...)`.
- **Переопределить `__init__` с несовместимой сигнатурой.** Если `Animal("Rex")` работает, а `Dog("Rex")` падает с `TypeError`, потому что `Dog.__init__` ждёт другие аргументы, экземпляры перестают быть взаимозаменяемыми. Добавляйте новые параметры после родительских и передавайте родительскую часть наверх.
- **Растить глубокие иерархии.** Каждый уровень - ещё одно место, где метод может быть перехвачен, и ещё один класс, который нужно прочитать, чтобы понять поведение. Два уровня читаются легко, пять - превращают каждый вызов в расследование. Если уровни множатся, пересмотрите модель.

<Detail summary="Несколько родителей, ромб и настоящий MRO">

Python разрешает перечислить несколько базовых классов: `class D(B, C)`. Тогда «цепочка» превращается в линеаризованный список - **MRO** (method resolution order), который строится алгоритмом C3 и доступен через `D.__mro__`.

```python
class A:
    def hello(self):
        return "A"

class B(A):
    pass

class C(A):
    def hello(self):
        return "C"

class D(B, C):
    pass

print(D().hello())
print([cls.__name__ for cls in D.__mro__])
```

Печатается `C` и `['D', 'B', 'C', 'A', 'object']`. Это ромбовидная иерархия: `B` и `C` оба наследуют `A`. Наивный поиск «в глубину» заглянул бы в `A` раньше, чем в `C`, и вернул бы `"A"`. C3 гарантирует другое: каждый класс встречается в списке один раз, потомки всегда стоят раньше предков, а порядок baseclass-ов в скобках сохраняется.

`super()` тоже ходит по MRO, а не «к родителю»: он передаёт управление следующему классу в списке, кем бы тот ни был. Именно поэтому в иерархиях с несколькими родителями каждый `__init__` должен вызывать `super().__init__()` - цепочка обойдёт всех ровно по одному разу. Подробности - в [официальном разборе C3](https://docs.python.org/3/howto/mro.html); для повседневного кода достаточно одиночного наследования, где MRO совпадает с очевидной цепочкой.

</Detail>

## Когда наследовать не стоит

Наследование - самая жёсткая связь между двумя классами: подкласс видит каждое изменение родителя и навсегда обещает быть его частным случаем. Прежде чем писать скобки после имени класса, проговорите отношение вслух. «Собака - это животное» звучит честно. «Заказ - это список товаров» уже нет: у заказа *есть* список товаров, а ещё адрес и статус. Отношение «у X есть Y» - сигнал хранить `Y` атрибутом, то есть использовать композицию; ей посвящён раздел [[composition]].

Быстрый тест на ошибочную иерархию: если подкласс вынужден глушить родительский метод заглушкой или `raise NotImplementedError`, модель неверна - родитель пообещал то, чего частный случай дать не может.
