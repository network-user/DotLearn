# Dataclasses - класс как хранилище данных

Большая часть классов в реальном коде - это не хитрая логика, а просто аккуратная упаковка данных: точка с координатами, строка конфигурации, запись о пользователе, позиция в корзине. Для каждого такого класса вы пишете одно и то же: `__init__`, который раскладывает аргументы по атрибутам, `__repr__`, чтобы объект читался в логах, и `__eq__`, чтобы два одинаковых объекта считались равными. Это десятки строк рутины, в которых легко ошибиться и которые ничего не говорят о вашей задаче. К концу этой статьи вы будете заменять всю эту рутину одной строкой `@dataclass` и понимать, что именно она генерирует, где у неё ловушки и когда от неё лучше отказаться.

## Что генерирует декоратор

Декоратор `@dataclass` смотрит на аннотированные поля класса и синтезирует за вас три метода: `__init__`, `__repr__` и `__eq__`. Поля - это переменные класса с аннотацией типа; именно аннотация делает имя полем. Сравните написанный руками класс и его dataclass-эквивалент: справа исчезает весь шаблон, а поведение остаётся тем же.

<Compare

  leftTitle="Вручную"
  rightTitle="@dataclass"
  left={`class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __repr__(self):
        return f"Point(x={self.x}, y={self.y})"

    def __eq__(self, other):
        if not isinstance(other, Point):
            return NotImplemented
        return (self.x, self.y) == (other.x, other.y)

p = Point(3, 4)
q = Point(3, 4)
print(p)
print(p == q)`}
  right={`from dataclasses import dataclass

@dataclass
class Point:
    x: float
    y: float

p = Point(3, 4)
q = Point(3, 4)
print(p)
print(p == q)`}
  verdict="Оба печатают Point(x=3, y=4) и True. Справа двадцать строк рутины сжаты до пяти, и форма данных видна сразу."
/>

Обратите внимание: `__repr__` формирует строку строго как `Point(x=3, y=4)` - имя класса, затем поля в скобках в порядке объявления. `__eq__` сравнивает экземпляры **по значению**: два разных объекта с одинаковыми `x` и `y` равны. Без dataclass `Point(3, 4) == Point(3, 4)` дал бы `False`, потому что по умолчанию Python сравнивает объекты по идентичности.

Сколько кода экономит декоратор, хорошо видно, если посчитать строки на каждый сгенерированный метод. Ручная версия выше тратит примерно по семь строк на `__init__`, `__repr__` и `__eq__`; добавьте упорядочивание - и список dunder-методов растёт ещё.

## Значения по умолчанию и ловушка изменяемых полей

Полю можно задать значение по умолчанию прямо в объявлении - оно работает как аргумент функции со значением по умолчанию: `quantity: int = 1`. Поля без умолчания должны идти раньше полей с умолчанием, ровно как у обычной функции.

Опасность начинается с **изменяемых** значений по умолчанию - списков, словарей, множеств. Написать `items: list = []` нельзя: dataclass это даже не разрешит и бросит `ValueError`. Причина - классическая ловушка Python: значение по умолчанию вычисляется один раз при определении класса, поэтому все экземпляры делили бы один и тот же список. Решение - `field(default_factory=...)`: фабрика вызывается заново при создании каждого экземпляра.

```python
from dataclasses import dataclass, field

@dataclass
class Cart:
    user_id: int
    items: list = field(default_factory=list)
```

Здесь полезно убедиться руками, что списки действительно независимы. Запустите код ниже, затем поменяйте `field(default_factory=list)` на `field(default_factory=dict)` (а `append` - на присваивание по ключу) и проверьте, что и словари не делятся между экземплярами.

<PyDemo
  title="Каждому экземпляру - свой список"
  code={`from dataclasses import dataclass, field

@dataclass
class Cart:
    user_id: int
    items: list = field(default_factory=list)

a = Cart(1)
b = Cart(2)
a.items.append("книга")
print(a.items)
print(b.items)`}
/>

Вы увидите `['книга']` и `[]`: список `a` пополнился, список `b` остался пустым, потому что у каждого экземпляра своя свежая фабрика.

## frozen, order и проверки в __post_init__

У декоратора есть параметры, включающие дополнительное поведение.

`@dataclass(frozen=True)` запрещает менять поля после создания: попытка присвоить новое значение бросит `FrozenInstanceError`. Заодно неизменяемый dataclass становится **хэшируемым** - его можно класть в `set` и использовать ключом `dict`.

<Detail summary="Почему frozen нужен для хэшируемости">
Хэш объекта обязан оставаться постоянным всё время, пока объект живёт в множестве или словаре. Если объект изменяемый и при этом сравнивается по значению, его хэш мог бы поменяться вслед за полями - и объект «потерялся» бы в своей корзине хэш-таблицы. Поэтому обычный `@dataclass` (изменяемый, с `__eq__`) выставляет `__hash__ = None`, делая экземпляры нехэшируемыми. `frozen=True` снимает противоречие: поля менять нельзя, значит хэш стабилен, и Python синтезирует `__hash__`.
</Detail>

`@dataclass(order=True)` добавляет методы сравнения `<`, `<=`, `>`, `>=`. Сравнение идёт лексикографически по кортежу полей в порядке объявления - как сравнение кортежей. Это сразу даёт сортировку: список dataclass-объектов можно передать в `sorted()` без `key=`.

Для проверки инвариантов есть метод `__post_init__`: сгенерированный `__init__` вызывает его в самом конце, уже разложив аргументы по атрибутам. Здесь удобно валидировать данные или вычислять производные поля.

```python
from dataclasses import dataclass

@dataclass
class Temperature:
    celsius: float

    def __post_init__(self):
        if self.celsius < -273.15:
            raise ValueError("ниже абсолютного нуля")
```

<Callout type="tip" title="__post_init__ видит уже готовый объект">
К моменту вызова `__post_init__` все поля уже присвоены, поэтому вы обращаетесь к ним через `self`. Это правильное место для валидации, которая не сводится к простому значению по умолчанию.
</Callout>

## Тонкая настройка полей через field()

`field()` управляет не только умолчанием. Тем же вызовом можно убрать поле из сгенерированных методов, не убирая его из класса:

- `field(repr=False)` - поле не попадёт в `__repr__` (удобно для длинных или секретных значений вроде токена).
- `field(compare=False)` - поле не участвует в `__eq__` и сравнении (например, кэш или временная метка, которые не определяют «равенство» объекта).
- `field(default=...)` - то же самое умолчание, что и присваивание, но в одном ряду с остальными настройками поля.

```python
from dataclasses import dataclass, field

@dataclass
class Session:
    user_id: int
    token: str = field(repr=False)
    created_at: float = field(default=0.0, compare=False)
```

Здесь `__repr__` не покажет `token`, а две сессии с одинаковым `user_id` и `token`, но разным временем создания, всё равно будут равны. Так dataclass остаётся честным описанием данных, но вы решаете, какие поля видны в логах и что значит «одинаковый».

## Когда dataclass не подходит

Dataclass хорош, пока класс - это в основном данные. Откажитесь от него, если:

- В конструкторе тяжёлые инварианты или сложная сборка, которые не выражаются полями и `__post_init__`.
- Класс - это в первую очередь поведение: много методов, за которыми формы данных почти не видно.
- Нужен полный контроль над `__init__` или вы строите изменяемый объект, который нельзя сравнивать по значению.

## Частые ошибки

- **Изменяемое умолчание без `field`.** `items: list = []` бросит `ValueError` при объявлении класса. Используйте `field(default_factory=list)` - своя фабрика на каждый экземпляр.
- **Забыли аннотацию типа.** `count = 0` без `: int` - это **не поле**, а обычная переменная класса: dataclass её проигнорирует, в `__init__` её не будет, в `__repr__` тоже. Поле создаёт именно аннотация.
- **Сравнение с обычным классом.** Сгенерированный `__eq__` сначала проверяет, что у второго объекта тот же класс. Dataclass `Point` никогда не будет равен экземпляру обычного класса с теми же `x` и `y` - сравнение даёт `False`.

Проследите по шагам, как сравнение видит несовпадение типов:
