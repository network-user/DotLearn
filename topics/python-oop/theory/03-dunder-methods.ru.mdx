# Dunder-методы: как ваш класс начинает понимать `+`, `len` и `==`

Вы написали класс `Money` и хотите складывать суммы оператором `+`, печатать их через `print`, сравнивать на равенство и проверять в условии `if`. Но `Money(100) + Money(50)` падает с `TypeError`, `print(m)` выдаёт `<Money object at 0x7f...>`, а `Money(0) == Money(0)` возвращает `False`. Встроенный синтаксис не знает ничего про ваш тип - и не обязан. Вместо этого он спрашивает сам объект, умеет ли тот складываться, печататься и сравниваться.

Этот разговор ведётся через dunder-методы (от *double underscore*, двойное подчёркивание): `__add__`, `__str__`, `__eq__` и десятки других. К концу раздела вы будете читать `a + b` как `a.__add__(b)`, отличать `__repr__` от `__str__` и понимать, какой из них запасной, аккуратно реализовывать `__eq__`, не сломав хэширование, и писать арифметику, которая возвращает новый объект, а не портит старый.

## Протокол: Python вызывает методы за вас

Главная мысль - короткая: **операторы и встроенные функции это синтаксический сахар над вызовами методов**. Когда вы пишете `a + b`, интерпретатор не выполняет никакого «сложения» сам. Он берёт левый операнд и вызывает `a.__add__(b)`. Когда вы пишете `len(x)`, вызывается `x.__len__()`. `str(x)` разворачивается в `x.__str__()`. Имя метода фиксировано протоколом, тело - ваше.

Из этого следует практический вывод: чтобы ваш тип поддерживал какую-то операцию, не нужно учить Python новому синтаксису - достаточно реализовать соответствующий метод. Хотите складывать - определите `__add__`. Хотите длину - определите `__len__`. Тип сам объявляет, что он умеет, перечисляя dunder-методы.

<MarginNote label="Откуда имена">Имена dunder-методов не магия: интерпретатор знает их заранее. `a + b` всегда ищет именно `__add__`, поэтому переименовать метод нельзя - протокол фиксирован.</MarginNote>

## Отображение: `__repr__` и `__str__`, и кто из них запасной

Сначала разберёмся, почему `print(m)` выдаёт нечитаемое `<Money object at 0x...>`. По умолчанию объект наследует `__repr__` от `object`, и тот печатает имя класса с адресом в памяти. Чтобы получить осмысленный вывод, нужно определить свой метод отображения. Их два, и роли у них разные.

`__repr__` отвечает за вывод в REPL и за `repr(x)`. Его задача - быть однозначным для разработчика: по строке должно быть понятно, что это за объект, а в идеале её можно скопировать обратно в код и получить такой же объект.

`__str__` отвечает за `str(x)` и `print(x)`. Его задача - читаемость для конечного пользователя: меньше технических деталей, больше человеческого текста.

```python
class Money:
    def __init__(self, cents):
        self.cents = cents

    def __repr__(self):
        return f"Money({self.cents})"

    def __str__(self):
        return f"{self.cents / 100:.2f}"
```

Теперь `repr(Money(1050))` даёт `Money(1050)` - строку, восстанавливающую объект, - а `str(Money(1050))` даёт `10.50`, привычный пользователю формат.

Ключевая асимметрия: **`__repr__` - запасной вариант, `__str__` - нет**. Если определить только `__repr__`, то `str(x)` и `print(x)` подхватят его автоматически. Если же определить только `__str__`, REPL и `repr(x)` продолжат показывать `<Money object at 0x...>`, потому что `__str__` не используется как замена `__repr__`. Поэтому практическое правило: если пишете лишь один метод отображения, пишите `__repr__` - он покрывает оба случая.

<Callout type="tip">Минимум для отладки - всегда `__repr__`. Логи, трассировки, отладчик и `repr()` внутри коллекций (`print([m1, m2])` зовёт `__repr__` элементов) - всё опирается на него.</Callout>

## Равенство: `__eq__`, и почему `is` это не `==`

По умолчанию `a == b` для ваших объектов сравнивает идентичность: `True` только если это буквально один и тот же объект в памяти. Два разных `Money(100)` окажутся не равны, хотя представляют одну сумму. Чтобы заработало сравнение по значению, переопределите `__eq__`.

```python
class Money:
    def __init__(self, cents):
        self.cents = cents

    def __eq__(self, other):
        if not isinstance(other, Money):
            return NotImplemented
        return self.cents == other.cents
```

Здесь важны оба `is`/`==`-нюанса. Оператор `==` по умолчанию делает то же, что `is` (сравнение идентичности), пока вы не определите `__eq__`. `is` же всегда проверяет идентичность и переопределить его нельзя - он спрашивает «это один и тот же объект?», а не «равны ли значения?». Поэтому `Money(100) is Money(100)` навсегда `False` (объекты разные), а `Money(100) == Money(100)` после нашего `__eq__` станет `True`.

Обратите внимание на `return NotImplemented`, когда другая сторона - не `Money`. Это не то же самое, что `return False`. `NotImplemented` означает «я не знаю, как сравниться с этим типом» - и тогда Python даёт шанс правой стороне через `other.__eq__(self)`. Если вернуть `False` напрямую, вы заявите неравенство от имени обоих, лишив правый операнд права голоса.

<Detail summary="NotImplemented против NotImplementedError - разные вещи">

`NotImplemented` - это **значение-синглтон**, которое возвращают из dunder-методов сравнения и арифметики, чтобы сказать «попробуй другую сторону». Его именно `return`, не `raise`.

`NotImplementedError` - это **исключение**, которое `raise`-ят в абстрактном методе базового класса, чтобы заставить подкласс его переопределить.

```python
def __add__(self, other):
    return NotImplemented

def area(self):
    raise NotImplementedError("подкласс обязан реализовать area")
```

Перепутать легко, а последствия разные: вернёте `NotImplementedError` (как значение) из `__eq__` - и сравнение «успешно» вернёт объект исключения, который истинный, так что всё будет считаться равным. Всегда `return NotImplemented`.

</Detail>

<Detail summary="Определили __eq__ - не забудьте про __hash__">

Как только вы пишете `__eq__`, Python **молча выставляет `__hash__ = None`**. Объект становится нехэшируемым: его нельзя положить в `set` или использовать ключом словаря - вы получите `TypeError: unhashable type`. Логика жёсткая: равные объекты обязаны иметь равный хэш, и Python не угадывает, как хэшировать ваш тип, раз вы переопределили равенство.

Два выхода. Если объект неизменяемый, определите `__hash__` согласованно с `__eq__` - по тем же полям:

```python
def __hash__(self):
    return hash(self.cents)
```

Если объект изменяемый, оставьте его нехэшируемым: это честно, потому что хэш не должен меняться за время жизни ключа.

</Detail>

## Истинность: `__len__`, `__bool__` и цепочка проверки

Когда вы пишете `if x:`, Python должен решить, истинный объект или ложный. Он идёт по цепочке: сначала ищет `__bool__` и берёт его результат; если `__bool__` нет, ищет `__len__` и считает объект ложным при длине `0` и истинным при любой ненулевой; если нет ни того, ни другого, объект истинный всегда.

```python
class Cart:
    def __init__(self, items):
        self._items = list(items)

    def __len__(self):
        return len(self._items)

empty = Cart([])
full = Cart([1, 2])
print(bool(empty), bool(full))
```

Печатается `False True`. Мы не писали `__bool__`, но `__len__` подхватил истинность: пустая корзина ложна, потому что её длина `0`. Это удобно - одна реализация `__len__` даёт и `len(cart)`, и осмысленное поведение в `if cart:`. Если же логика истинности не совпадает с длиной (скажем, объект всегда «есть»), определите `__bool__` явно, и он перебьёт `__len__`.

## Арифметика: `__add__` возвращает новый объект

Теперь к оператору `+`. Реализуем сложение денег - и сразу зафиксируем дисциплину, которая отличает корректную арифметику от опасной.

```python
class Money:
    def __init__(self, cents):
        self.cents = cents

    def __add__(self, other):
        if not isinstance(other, Money):
            return NotImplemented
        return Money(self.cents + other.cents)
```

Метод **создаёт и возвращает новый `Money`**, а не меняет `self`. Это и есть дисциплина неизменяемости: `a + b` не должен трогать ни `a`, ни `b`. Сравните с ошибочной версией, где соблазн «сэкономить» приводит к беде:

```python
def __add__(self, other):
    self.cents += other.cents
    return self
```

Здесь `a + b` молча портит `a`. Хуже того, `total = a + a` удвоит `a`, а потом `total` и `a` окажутся одним объектом. Встроенные типы так себя не ведут: `2 + 3` не превращает `2` в `5`. Ваш тип не должен быть исключением - возвращайте новый объект.

<PullQuote>`a + b` не имеет права менять ни `a`, ни `b`. Сложение рождает новое значение, а не переписывает старое.</PullQuote>

## Краевые случаи и сюрпризы

Самый коварный сюрприз - забытый `__hash__` после `__eq__`: код, который раньше клал объекты в `set`, внезапно падает с `TypeError`. Разобран в детали выше; запомните как связку «`__eq__` определил - про `__hash__` подумал».

Второй - смешанные типы в арифметике. `Money(100) + 5` сработает только если вы это предусмотрели; иначе `__add__` вернёт `NotImplemented`, Python попробует отражённую операцию `(5).__radd__(money)`... не найдёт способа сложить `int` с `Money` и кинет `TypeError`. Это правильно: лучше явный отказ, чем тихое неверное сложение центов с непонятно чем.

Третий - `__eq__`, возвращающий не `bool`. Если ваш метод вернёт, например, список или число, выражение `a == b` отдаст этот странный объект, и `if a == b:` будет проверять его истинность, а не равенство. Возвращайте `bool` или `NotImplemented`, и ничего больше.

## Типичные ошибки

- **`__eq__` возвращает что попало.** Вернули из `__eq__` строку, число или результат вычитания координат - и `==` перестаёт быть булевым. Возвращайте только `True`/`False` или `NotImplemented` для чужого типа.
- **`__add__` мутирует `self`.** Соблазн написать `self.cents += other.cents; return self` ломает неизменяемость: `a + b` портит `a`, а псевдонимы расходятся с интуицией. Всегда возвращайте свежесозданный объект.
- **Неясный `__repr__`.** `__repr__`, возвращающий `"объект денег"` или вообще пользовательский текст, бесполезен в логах и отладчике. Делайте его однозначным и по возможности восстановимым в вызов конструктора: `Money(1050)`, а не `10.50 руб`.
