# ABC и абстрактные методы: контракт, который падает рано

Вы строите систему уведомлений: каждый канал (email, SMS, push) обязан уметь `send`. Пока договор держится на честном слове, кто-нибудь добавит канал без `send`, объект спокойно создастся, и падение случится глубоко в проде, когда до канала дойдёт первое сообщение. Абстрактные базовые классы (ABC) превращают устный договор в проверяемый: класс с незаполненной обязанностью нельзя даже создать. Ошибка выскакивает на границе, в момент `Channel()`, а не где-то ниже по потоку. К концу раздела вы научитесь описывать такой контракт, делать абстрактными не только методы, но и свойства, подключать чужие типы через `register`, брать готовые интерфейсы из `collections.abc` и осознанно выбирать между ABC и `Protocol`.

## Контракт, который проверяется при создании объекта

Абстрактный базовый класс описывает, что подкласс обязан уметь, но сам оставляет реализацию пустой. Метод, помеченный `@abstractmethod`, - это дыра в контракте. Пока хоть одна такая дыра не закрыта, Python отказывается создавать экземпляр и поднимает `TypeError`, перечисляя незаполненные имена. Проверка происходит один раз, при инстанцировании, и ловит забытую реализацию мгновенно, а не при первом вызове метода.

## `abc.ABC` и `@abstractmethod`

Контракт объявляют наследованием от `abc.ABC` и пометкой методов декоратором `@abstractmethod`:

```python
from abc import ABC, abstractmethod

class Notifier(ABC):
    @abstractmethod
    def send(self, message):
        ...
```

Попытка `Notifier()` теперь падает с `TypeError: Can't instantiate abstract class Notifier with abstract method send`. Класс существует и годится в родители, но самостоятельным объектом быть не может. Подкласс оживает, как только закроет все дыры:

```python
class Email(Notifier):
    def send(self, message):
        return f"email: {message}"
```

`Email("привет")` уже создаётся и работает. Одна тонкость решает всё - сам по себе `@abstractmethod` ничего не запрещает. Магия запрета живёт в метаклассе `ABCMeta`, а его подключает наследование от `ABC`. Повесить декоратор на метод обычного класса можно, но инстанцирование пройдёт спокойно, и контракт окажется декоративным.

## Попробуйте: хранилище с обязательным интерфейсом

Базовый `Storage` требует `save` и `load`. Конкретный `InMemory` их реализует, а голый `Storage()` создать нельзя:

class Storage(ABC):
    @abstractmethod
    def save(self, key, value):
        ...

    @abstractmethod
    def load(self, key):
        ...

    def __init__(self):
        self._data = {}

    def save(self, key, value):
        self._data[key] = value

    def load(self, key):
        return self._data[key]

store = InMemory()
store.save("город", "Казань")
print(store.load("город"))

try:
    Storage()
except TypeError as error:
    print("Storage напрямую:", error)`}

Уберите из `InMemory` метод `load` и запустите снова: теперь уже `InMemory()` откажется создаваться, потому что одна дыра контракта осталась открытой. Верните метод и попробуйте добавить в `Storage` третий абстрактный метод.

## Абстрактными бывают не только методы

Контракт умеет требовать свойство или метод класса. Декораторы складывают стопкой, и порядок важен: `@abstractmethod` должен стоять ближе всего к функции, снизу.

```python
class Config(ABC):
    @property
    @abstractmethod
    def name(self):
        ...

    @classmethod
    @abstractmethod
    def from_file(cls, path):
        ...
```

Подкласс обязан предоставить и свойство `name`, и классовый метод `from_file`. Перепутанный порядок декораторов (`@abstractmethod` сверху) тихо ломает абстрактность: класс перестанет требовать реализацию, и забытый метод снова прорвётся в прод.

## `register` и виртуальные подклассы

Иногда тип уже написан и менять его иерархию нельзя, но по смыслу он подходит под контракт. `ABC.register` объявляет его виртуальным подклассом:

```python
class JsonFile:
    def save(self, key, value): ...
    def load(self, key): ...

Storage.register(JsonFile)

isinstance(JsonFile(), Storage)   # True
issubclass(JsonFile, Storage)     # True
```

После `register` проверки `isinstance` и `issubclass` дают `True`, хотя `JsonFile` не наследует `Storage`. Плата за гибкость - честная: методы не наследуются (миксинов вы не получите), и никакой проверки контракта нет. Зарегистрировать можно даже класс без `save` и `load`, и `isinstance` всё равно соврёт, что это `Storage`. Именно так стандартная библиотека приучает встроенные `list` и `dict` откликаться на абстрактные интерфейсы, не переписывая их.

## `collections.abc`: готовые интерфейсы стандартной библиотеки

Свои ABC писать нужно редко - для коллекций всё уже собрано в `collections.abc`. Там лежат `Iterable`, `Iterator`, `Sequence`, `Mapping`, `Set` и другие. Наследование от них полезно по двум причинам сразу, и вторая важнее первой.

```python
from collections.abc import Mapping

class FrozenDict(Mapping):
    def __init__(self, data):
        self._data = dict(data)

    def __getitem__(self, key):
        return self._data[key]

    def __iter__(self):
        return iter(self._data)

    def __len__(self):
        return len(self._data)
```

Вы реализовали три метода, а получили полноценный маппинг. `Mapping` отдаёт даром методы-миксины `keys`, `values`, `items`, `get`, `__contains__`, `__eq__`, собранные поверх ваших трёх. Это первая выгода. Вторая - `isinstance(FrozenDict(...), Mapping)` теперь `True`, и ваш тип честно проходит проверки, ожидающие маппинг. Наследование от `collections.abc` разом даёт и бесплатное поведение, и место в системе типов.

## ABC против `Protocol`: номинальная и структурная типизация

ABC задаёт **номинальную** типизацию: тип считается подтипом, только если он явно наследует базовый класс или зарегистрирован через `register`. Родство объявляется по имени. `typing.Protocol` задаёт **структурную** типизацию: тип подходит, если у него есть нужная форма (те самые методы и атрибуты), безо всякого наследования. Это формализованная утиная типизация, и проверяют её статические анализаторы, а не рантайм.

## Краевые случаи

Декоратор без метакласса бесполезен - `@abstractmethod` на методе класса, который не наследует `ABC` и не задаёт `metaclass=ABCMeta`, не мешает инстанцированию. Абстрактность обеспечивает именно `ABCMeta`.

`register` ничего не проверяет. Он лишь заносит тип в список виртуальных подклассов, поэтому `isinstance` после регистрации способен дать ложное обещание. Полагаться на него как на гарантию наличия методов нельзя.

У абстрактного метода может быть тело. Подкласс всё равно обязан его переопределить, но может позвать общую логику базы через `super().method(...)`.

## Типичные ошибки

- **Забыли унаследовать `ABC`.** Один `@abstractmethod` без `ABCMeta` не запрещает создание объекта. Наследуйте `abc.ABC` или задайте `metaclass=ABCMeta`.
- **Перепутан порядок декораторов.** Для абстрактного свойства или метода класса `@abstractmethod` ставят ближе к функции (снизу), иначе абстрактность теряется.
- **`register` приняли за наследование.** Виртуальный подкласс не получает методы базы и не проверяется на контракт. Нужны миксины или гарантия реализации, значит нужно обычное наследование.
