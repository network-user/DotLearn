# Composition over inheritance: assemble, don't inherit

Imagine designing a car in code. It is tempting to build a tower of classes: `Vehicle` → `Car` → `ElectricCar` → `SportElectricCar`. Then a month later a new requirement arrives: put the same motor in a boat. The tower does not help - a boat is not a car. By the end of this lesson you will know how to assemble behavior from small, swappable parts, and why experienced developers reach for composition before inheritance.

## Two kinds of relationship: is-a and has-a

Inheritance expresses an **is-a** relationship: a `Dog` is an `Animal`. Composition expresses **has-a**: a `Car` has an `Engine`. These are not just different words - they are different degrees of coupling.

A subclass sees everything in its parent and is bound to it forever. Any method added to the base class instantly appears on every descendant - sometimes where it does not belong. Composition keeps parts at arm's length: the host holds a reference to a part and talks to it through the part's own methods. A part can be swapped without touching the host.

## Where inheritance starts to crack

The classic symptom is a parent method some descendants cannot honor:

```python
class Vehicle:
    def drive(self): ...
    def refuel(self): ...

class ElectricCar(Vehicle):
    def refuel(self):
        raise NotImplementedError("electric cars don't take fuel")
```

`NotImplementedError` in a subclass is a smell. The base class promised a capability the subclass has to revoke. That means `refuel` does not belong to every "vehicle", and the hierarchy is wrong: you stretched is-a too far.

<Callout type="warning" title="Warning sign">
If a subclass disables or stubs out a parent method, the model is almost certainly wrong. It breaks the substitution principle: a descendant must work everywhere its ancestor works.
</Callout>

## The same task with composition

Let's assemble a car from a part responsible for motion. The host does not care what is inside, as long as the part has a `start()` method:

<PyDemo
  title="Has-a, not is-a"
  code={`class CombustionEngine:
    def start(self):
        return "vroom"

class ElectricMotor:
    def start(self):
        return "whirr"

class Car:
    def __init__(self, propulsion):
        self.propulsion = propulsion

    def drive(self):
        return f"car says {self.propulsion.start()}"

print(Car(CombustionEngine()).drive())
print(Car(ElectricMotor()).drive())`}
/>

The part is passed into the constructor and stored on `self`. This is **dependency injection** in miniature: the host does not create the part itself, it receives a ready one from outside. That is why a test can hand in any object with a `start()` method - even a stub. Try adding a third motor class and passing it to `Car`: the host needs no changes at all.

`Car` relies not on a concrete type but on the presence of a `start()` method. That is **duck typing** combined with composition: "if it quacks like a duck, it will do."

## Delegation: the host forwards calls

When the host simply forwards a call to its part, that is called **delegation**. A `Stack` built on a list is the textbook example: the stack *has* a list but *is not* a list, so only the operations you want are exposed.

```python
class Stack:
    def __init__(self):
        self._items = []

    def push(self, x):
        self._items.append(x)

    def pop(self):
        return self._items.pop()

    def size(self):
        return len(self._items)
```

If `Stack` inherited from `list`, `insert`, `sort`, and indexing would all leak out - and the "top only" invariant would break. Composition lets you expose exactly the interface you intend to support.

Delegation is almost never "bare": the host usually adds something of its own to the forwarded call. Consider a player that *has* a playback queue. It forwards the choice of the next track to the queue, but along the way keeps its own play counter - behavior the queue itself does not have:

```python
class Queue:
    def __init__(self, tracks):
        self._tracks = list(tracks)
        self._pos = 0

    def next(self):
        track = self._tracks[self._pos]
        self._pos = (self._pos + 1) % len(self._tracks)
        return track

class Player:
    def __init__(self, queue):
        self.queue = queue
        self.played = 0

    def play(self):
        self.played += 1
        return self.queue.next()
```

`Player` does not inherit `Queue` and knows nothing about its internal index - it only calls `next()` and adds its own bookkeeping. The queue can be swapped for a shuffled one or an endless radio station, and `Player` will not notice. That is the power of composition: each object owns one thing, and the host assembles their behavior while adding its own.

## Edge cases and traps

Composition is not free. The most common problem is **reaching through a part**: `car.propulsion.tank.level`. Every dot is extra knowledge about another object's internals. The rule is simple (it is called the Law of Demeter): talk to your friends, not to friends of friends. If the host needs the fuel level, let the part expose it via a `remaining()` method and have the host forward the call.

The second case is when inheritance really does fit. If the relationship is a genuine is-a and the parent's interface suits the child fully, inheritance is shorter and clearer. Composition is a default, not a dogma.

## Common mistakes

- **Inheriting to reuse a couple of methods.** If you need one method of another class, extract it into a function or a part - do not drag the whole class into your ancestry.
- **A god-object host.** If the host knows too much about all its parts and does their work itself, you have only renamed a big class. Delegate, do not duplicate.
- **`a.b.c.d` chains.** A long chain of accesses signals scattered responsibilities. Add forwarding methods on the intermediate objects.

<Detail summary="Deep dive: `__getattr__` as auto-delegation">

Python lets you forward any unknown attribute to a part via `__getattr__`:

```python
class Wrapper:
    def __init__(self, inner):
        self._inner = inner

    def __getattr__(self, name):
        return getattr(self._inner, name)
```

Now `Wrapper(x).anything` goes to `x`. Tempting, but dangerous: you lose control over which interface is exposed, you break IDE autocomplete, and you may accidentally forward methods that violate an invariant. `__getattr__` is great for transparent proxies (logging, lazy loading), but for ordinary composition explicit forwarding methods are more honest.

</Detail>
