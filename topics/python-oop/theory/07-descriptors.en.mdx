# Descriptors: how one object governs attribute access

A user profile class has three fields with the same check: age from 0 to 150, rating from 0 to 100, discount from 0 to 100. With `property` each field needs a getter plus a setter pair, and in every setter you rewrite the same range check. Three fields, three near-identical chunks of code, and the moment a fourth field appears in another class the copying starts again. Descriptors let you describe the check once, as a standalone field object, and reuse it anywhere. By the end of this section you will see that `property` and an ordinary method run on the same mechanism, you will be able to read the attribute lookup order, and you will write a validated field that plugs in with a single line.

## Attribute access is not a dictionary read

It looks as if `user.age` simply pulls a value out of `user.__dict__["age"]`. In reality a call hides behind the dot: the interpreter runs `type(user).__getattribute__(user, "age")`, and that method decides where the value comes from. First it looks not in the instance dictionary but in the classes along the inheritance chain (the MRO). If it finds an object there with a `__get__` method, that object is a descriptor, and access goes through it rather than through the dictionary.

A descriptor is an object that lives as a class attribute and defines at least one of `__get__`, `__set__`, `__delete__`. As long as such an object owns a name, every read and write of that name on an instance passes through its methods. That is exactly how a field gains the ability to validate, log, or compute a value instead of just storing it.

## The protocol: `__get__`, `__set__`, and `__set_name__`

Let's build a field step by step. A minimal descriptor answers reads and writes:

```python
class Field:
    def __get__(self, obj, objtype=None):
        return obj.__dict__[self.name]

    def __set__(self, obj, value):
        obj.__dict__[self.name] = value
```

`__get__` receives the instance `obj` and its type; `__set__` receives the instance and the value being assigned. We store the value in the instance's own dictionary, not in the descriptor. This is essential: the descriptor is one per class - putting the data into it would make one value shared across every instance.

Where did `self.name` come from? The third protocol method supplies it. When the interpreter builds the class, it calls `__set_name__` for every descriptor and tells it under which name it was stored:

```python
class Field:
    def __set_name__(self, owner, name):
        self.name = name
```

Now a `Field` assigned to the `age` attribute learns the string `"age"` itself and stores the value in `obj.__dict__["age"]`, with no hard-coded string anywhere. Writing straight into `obj.__dict__` does not re-trigger the protocol: that is a dictionary operation, not attribute access, so there is no recursion. And because our `Field` defines `__set__`, it stays authoritative on reads, so the private slot in the dictionary is entirely under its control.

## Data descriptor vs non-data, and the lookup order

Descriptors come in two kinds, and the difference decides who wins a conflict with the instance dictionary.

A non-data descriptor defines only `__get__`. It yields to the instance dictionary: if `obj.__dict__` has a key with the same name, the dictionary value is returned and `__get__` is never even called. A data descriptor defines `__set__` or `__delete__`, even with a stub body. It takes precedence over the instance dictionary: a read always goes through `__get__`, whatever sits in `obj.__dict__`.

The full lookup order for `obj.attr` is this: first a class data descriptor, then the instance dictionary, then a non-data descriptor or a plain class attribute. This hierarchy is not an academic detail - it explains the behavior of `property`, ordinary methods, and caches all at once.

## `property` is a data descriptor

`property` implements `__get__`, `__set__`, and `__delete__`, which makes it a data descriptor. Hence its reliability: the setter cannot be bypassed by writing something straight into `obj.__dict__` under the same name, because the read still goes through `__get__`. When you write `@property`, you create a special case of a descriptor where `__get__` calls your getter function. Everything you learned about fields applies to `property` word for word.

## A function is a non-data descriptor, so a bound method is born

An ordinary function in a class body is a descriptor too: the `function` type has `__get__` and no `__set__`, so it is a non-data descriptor. When you write `user.greet`, access goes through `function.__get__(user, User)`, and it returns not the function itself but a bound method whose first argument `self` is already filled in as `user`. Accessing the same name through the class, `User.greet`, calls `__get__` with `obj=None` and returns the plain function.

```python
class User:
    def greet(self):
        return "hi"

u = User()
type(u.greet)                    # method
type(User.greet)                 # function
u.greet.__func__ is User.greet   # True
```

A bound method is created afresh on every access, so `u.greet is u.greet` is `False`, even though they wrap the same function inside. A method knowing its `self` is not magic in the `def` line - it is a direct consequence of the descriptor protocol.

## `functools.cached_property`: compute once

`cached_property` leans on exactly the data/non-data asymmetry. It is a non-data descriptor: it has only `__get__`. On the first read it computes the value, stores it in `obj.__dict__` under the same name, and returns it. On the second read the lookup order kicks in: a non-data descriptor yields to the instance dictionary, so `__get__` is no longer called and the value is taken straight from the dictionary. The computation happens once, and after that access is ordinary and fast.

```python
from functools import cached_property

class Dataset:
    def __init__(self, rows):
        self.rows = rows

    @cached_property
    def total(self):
        return sum(self.rows)
```

The price of the speed is known up front. The instance needs a `__dict__` (with `__slots__` and no dictionary `cached_property` will not work), and the field name in the dictionary must stay free. If the computed value can go stale, you have to clear the cache by hand with `del obj.total`.

## Edge cases

A descriptor works only when it lives as a class attribute. Assigning `Field()` inside `__init__` via `self.age = Field()` is pointless: `__get__` and `__set__` fire only for attributes found on the type, not in the instance dictionary.

<Callout type="warning" title="The shared-state trap">
Storing the value in the descriptor itself (`self.value = value`) makes the field shared across the whole class: there is one descriptor, so a second instance overwrites the first one's value. Always put the data in `obj.__dict__`, keyed by the name from `__set_name__`.
</Callout>

Another trap is infinite recursion. If inside `__get__` you write `getattr(obj, self.name)` with the public name, you wake the descriptor again and it calls itself. Reach into `obj.__dict__` directly, bypassing the protocol.

## Common mistakes

- **State in the descriptor instead of the instance.** `self.value = value` makes the field shared across the class. Store the data in `obj.__dict__[self.name]` so each instance has its own value.
- **A descriptor on the instance.** `self.age = Field()` in the constructor does not engage the protocol. A descriptor must be a class attribute.
- **A hard-coded storage name.** A fixed string instead of `__set_name__` breaks the moment you rename the field or add a second attribute like it. Let the name come from `__set_name__`.
