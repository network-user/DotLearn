# Композиция против наследования: собираем, а не наследуем

Представьте, что вы проектируете автомобиль в коде. Заманчиво построить башню классов: `Транспорт` → `Автомобиль` → `ЭлектроАвтомобиль` → `СпортЭлектроКар`. Но через месяц приходит требование: тот же мотор поставить в катер. Башня не поможет - катер это не автомобиль. К концу этого урока вы научитесь собирать поведение из небольших заменяемых деталей и будете понимать, почему опытные разработчики тянутся к композиции раньше, чем к наследованию.

## Две формы связи: is-a и has-a

Наследование выражает отношение **is-a** («является»): `Собака` - это `Животное`. Композиция выражает **has-a** («содержит»): у `Автомобиля` есть `Мотор`. Это не просто разные слова - это разная жёсткость связи.

Наследник видит всё, что есть в родителе, и привязан к нему навсегда. Любой метод, добавленный в базовый класс, мгновенно появляется у всех потомков - иногда там, где он не нужен. Композиция же держит детали на расстоянии вытянутой руки: хост хранит ссылку на часть и обращается к ней через её собственные методы. Часть можно подменить, не трогая хост.

## Где наследование начинает трещать

Классический симптом - метод родителя, который часть потомков не может выполнить:

```python
class Транспорт:
    def ехать(self): ...
    def заправиться(self): ...

class ЭлектроКар(Транспорт):
    def заправиться(self):
        raise NotImplementedError("электромобиль не заправляют бензином")
```

`NotImplementedError` в потомке - это запах. Базовый класс пообещал возможность, которую наследник вынужден отзывать. Значит, `заправиться` не относится ко всем «транспортам», и иерархия выбрана неверно: вы растянули is-a слишком далеко.

<Callout type="warning" title="Сигнал тревоги">
Если подкласс отключает или подменяет метод родителя заглушкой, модель почти наверняка неправильная. Это нарушение принципа подстановки: потомок должен работать везде, где работает предок.
</Callout>

## Та же задача через композицию

Соберём автомобиль из детали, отвечающей за движение. Хосту неважно, что именно внутри, лишь бы у детали был метод `start()`:

<PyDemo
  title="Has-a, а не is-a"
  code={`class ДвигательВнутреннегоСгорания:
    def start(self):
        return "врум"

class ЭлектроМотор:
    def start(self):
        return "вжжж"

class Автомобиль:
    def __init__(self, привод):
        self.привод = привод

    def ехать(self):
        return f"машина говорит {self.привод.start()}"

print(Автомобиль(ДвигательВнутреннегоСгорания()).ехать())
print(Автомобиль(ЭлектроМотор()).ехать())`}
/>

Деталь передаётся в конструктор и сохраняется на `self`. Это **внедрение зависимости** в миниатюре: хост не создаёт деталь сам, а получает готовую снаружи. Поэтому в тесте можно подсунуть любой объект с методом `start()` - даже заглушку. Попробуйте дописать третий класс мотора и передать его в `Автомобиль`: хост не придётся менять вовсе.

`Автомобиль` опирается не на конкретный тип, а на наличие метода `start()`. Это **утиная типизация** в связке с композицией: «если крякает как утка - сгодится».

## Делегирование: хост перенаправляет вызовы

Когда хост просто пересылает вызов своей детали, это называют **делегированием**. `Стек`, построенный на списке, - хрестоматийный пример: стек *содержит* список, но не *является* списком, поэтому наружу торчат только нужные операции.

```python
class Стек:
    def __init__(self):
        self._items = []

    def push(self, x):
        self._items.append(x)

    def pop(self):
        return self._items.pop()

    def size(self):
        return len(self._items)
```

Если бы `Стек` наследовался от `list`, наружу утекли бы `insert`, `sort`, индексация - и инвариант «только с вершины» сломался бы. Композиция позволяет открыть ровно тот интерфейс, который вы хотите поддерживать.

Делегирование почти никогда не бывает «голым»: хост обычно добавляет к перенаправленному вызову что-то своё. Рассмотрим плеер, который *содержит* очередь воспроизведения. Он пересылает в неё выбор следующего трека, но попутно ведёт собственный счётчик прослушиваний - поведение, которого у самой очереди нет:

```python
class Очередь:
    def __init__(self, треки):
        self._треки = list(треки)
        self._поз = 0

    def следующий(self):
        трек = self._треки[self._поз]
        self._поз = (self._поз + 1) % len(self._треки)
        return трек

class Плеер:
    def __init__(self, очередь):
        self.очередь = очередь
        self.сыграно = 0

    def играть(self):
        self.сыграно += 1
        return self.очередь.следующий()
```

`Плеер` не наследует `Очередь` и ничего не знает о её внутреннем индексе - он лишь вызывает `следующий()` и добавляет свой учёт. Очередь можно подменить на перемешанную или бесконечную радиостанцию, и `Плеер` этого не заметит. Это и есть сила композиции: каждый объект отвечает за одну вещь, а хост собирает их поведение, дополняя своим.

## Краевые случаи и ловушки

Композиция не бесплатна. Самая частая беда - **дотягивание сквозь деталь**: `car.привод.бак.уровень`. Каждая точка - это лишнее знание о внутренностях чужого объекта. Правило простое (его называют законом Деметры): разговаривайте с друзьями, а не с друзьями друзей. Если хосту нужен уровень топлива, пусть деталь сама отдаёт его методом `остаток()`, а хост перенаправит вызов.

Второй случай - когда наследование действительно уместно. Если отношение - честное is-a и интерфейс родителя подходит потомку целиком, наследование короче и яснее. Композиция не догма, а инструмент по умолчанию.

## Типичные ошибки

- **Наследование ради переиспользования пары методов.** Если нужен один метод чужого класса, вынесите его в отдельную функцию или деталь, а не тяните весь класс в предки.
- **Бог-объект как хост.** Если хост знает обо всех своих частях слишком много и сам делает их работу, вы лишь переименовали большой класс. Делегируйте, а не дублируйте.
- **Цепочки `a.b.c.d`.** Длинная цепочка обращений - сигнал, что обязанности размазаны. Добавьте методы-перенаправители на промежуточные объекты.

<Detail summary="Глубокое погружение: `__getattr__` как авто-делегирование">

Python позволяет переслать любой неизвестный атрибут детали через `__getattr__`:

```python
class Обёртка:
    def __init__(self, внутр):
        self._внутр = внутр

    def __getattr__(self, имя):
        return getattr(self._внутр, имя)
```

Теперь `Обёртка(x).что_угодно` уходит в `x`. Соблазнительно, но опасно: вы теряете контроль над тем, какой интерфейс открыт, ломаете автодополнение в IDE и можете нечаянно пробросить методы, нарушающие инвариант. `__getattr__` хорош для прозрачных прокси (логирование, ленивая загрузка), но для обычной композиции явные методы-перенаправители честнее.

</Detail>
