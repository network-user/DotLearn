# Инкапсуляция: договор вместо замка

В пятницу вечером падают тесты: у одного из пользователей баланс счёта равен -3400, хотя ни один сценарий приложения не позволяет уйти в минус. Поиск по коду находит одиннадцать мест, где кто-то пишет в `acc.balance` напрямую, и в одном из них перед записью забыли проверку. Виновата не строка, которая записала минус, - виновата конструкция, в которой записать минус мог кто угодно.

В Java или C++ вы бы пометили поле словом `private`, и компилятор отрезал бы все одиннадцать мест разом. В Python такого ключевого слова нет: видимость здесь - договорённость, выраженная именами, плюс один механизм переименования и один механизм перехвата доступа. К концу статьи вы будете понимать, что на самом деле означает ведущее подчёркивание, во что превращается атрибут с двумя подчёркиваниями и как `property` делает из обычного атрибута контролируемую точку записи с проверкой.

## Инвариант хочет жить в одном месте

Класс полезен не тем, что складывает данные и функции рядом, а тем, что даёт обещания о своём состоянии: баланс неотрицателен, дата окончания позже даты начала, кэш согласован с источником. Такие обещания называются инвариантами. Пока в атрибут пишет любая часть программы, инвариант приходится охранять везде - и однажды кто-то забудет. Инкапсуляция сводит все точки записи к нескольким методам, и проверка живёт в одном месте.

Python подходит к задаче без замков: язык не запрещает доступ, а размечает его именами. Имя без подчёркивания - публичный интерфейс. Одно ведущее подчёркивание - «внутреннее, не полагайтесь на меня». Два - «внутреннее настолько, что Python его переименует». Попробуйте обратиться к каждому из трёх атрибутов снаружи класса и посмотрите, чем уровни отличаются на практике:

## Одно подчёркивание: пометка для людей

Конвенция выглядит так: атрибут, который снаружи трогать не стоит, получает ведущее подчёркивание.

```python
class Account:
    def __init__(self, owner, balance):
        self.owner = owner
        self._balance = balance

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("amount must be positive")
        self._balance += amount
```

<MarginNote label="PEP 8">PEP 8 называет одно ведущее подчёркивание «слабым индикатором внутреннего использования» - этого индикатора хватает для большинства кода.</MarginNote>

`_balance` доступен снаружи точно так же, как `owner`: выражение `acc._balance` вернёт значение без единого предупреждения. Подчёркивание адресовано людям и инструментам: IDE не предложит `_balance` в автодополнении за пределами класса, линтер подсветит обращение, а `from module import *` пропустит имена с подчёркиванием. Это и есть договор: автор класса говорит «я могу переименовать или удалить это поле в любой момент», читатель обещает на него не опираться. Нарушить договор можно, но нарушение видно в diff и легко ловится на ревью.

## Два подчёркивания: переименование, а не замок

С двумя ведущими подчёркиваниями в игру вступает механизм языка - name mangling, искажение имён. Внутри тела класса каждый идентификатор вида `__name` (без двух подчёркиваний на конце) переписывается в `_ClassName__name`. Это происходит при компиляции тела класса, до выполнения какой-либо строки.

```python
class Account:
    def __init__(self, owner, balance):
        self.owner = owner
        self.__balance = balance
```

Когда `__init__` выполняется, в объект записывается атрибут не с именем `__balance`, а с именем `_Account__balance`. Посмотрите на содержимое экземпляра на рисунке: атрибута `__balance` там просто нет.

Отсюда следует и поведение снаружи. Обращение `acc.__balance` за пределами класса падает с `AttributeError` не потому, что Python проверил права доступа, а потому, что атрибута с таким именем не существует: вне тела класса переименование не действует, и интерпретатор ищет буквально `__balance`. Зато `acc._Account__balance` работает - это настоящее имя атрибута, и оно ничем не защищено: его можно прочитать и даже перезаписать.

<PullQuote>Два подчёркивания - это не замок на двери, а табличка с другим именем на ней.</PullQuote>

Проверьте оба пути сами:

<PyDemo

    def __init__(self, owner, balance):
        self.owner = owner
        self.__balance = balance

print(vars(acc))

Запустите код: `vars(acc)` показывает реальные имена атрибутов экземпляра. Теперь замените последнюю строку на `print(acc.__balance)` и убедитесь, что получите `AttributeError`. Затем третий эксперимент: допишите перед `print` строку `acc._Account__balance = -100` - запись пройдёт без возражений. Никакой защиты, только переименование.

Зачем тогда механизм существует? Для защиты от случайных коллизий имён при наследовании - подробности в глубоком разборе ниже. Если вы не пишете базовый класс для чужих подклассов, два подчёркивания почти никогда не нужны: одного достаточно.

## property: настоящий контрольный пункт

Конвенции размечают границу, но не охраняют её. Когда инвариант действительно важен, нужен перехват записи - и для этого в Python есть `property`. Снаружи атрибут выглядит как обычный, но каждое чтение проходит через геттер, а каждое присваивание - через сеттер.

<PyDemo
  title="Сеттер с проверкой"
  code={`class Account:
    def __init__(self, owner, balance):
        self.owner = owner
        self.balance = balance

    @property
    def balance(self):
        return self._balance

    @balance.setter
    def balance(self, value):
        if value < 0:
            raise ValueError("balance cannot be negative")
        self._balance = value

acc = Account("Ada", 100)
acc.balance -= 30
print(acc.balance)`}
/>

Обратите внимание на `__init__`: он присваивает `self.balance = balance`, то есть проходит через тот же сеттер, что и внешний код. Проверка написана один раз и срабатывает и при создании объекта, и при каждом следующем присваивании. Попробуйте: замените `-= 30` на `-= 200` и запустите - сеттер отклонит отрицательный баланс. Потом создайте счёт сразу с минусом, `Account("Ada", -5)`, и проверка сработает и здесь.

Реальное значение лежит в `self._balance` - обычном атрибуте с одним подчёркиванием. Конвенция и `property` работают в паре: публичное имя `balance` стало шлюзом, внутреннее `_balance` хранит данные.

Сеттер при этом не обязателен. `property` без сеттера - это атрибут только для чтения, и заодно удобный способ объявить вычисляемое значение:

```python
class Card:
    def __init__(self, number):
        self._number = number

    @property
    def masked(self):
        return "**** " + self._number[-4:]
```

`Card("1234 5678 9012 4242").masked` вернёт `'**** 4242'`, а попытка присвоить `card.masked = "x"` упадёт с `AttributeError: property 'masked' of 'Card' object has no setter`. Значение не хранится в объекте вовсе: оно вычисляется при каждом чтении и не может разойтись с `_number`.

## Когда обычного атрибута достаточно

Главная причина, по которой в Python не пишут геттеры и сеттеры впрок: переход от обычного атрибута к `property` не меняет вызывающий код. В Java поле, однажды опубликованное напрямую, уже не обернуть проверкой без правки всех вызовов, поэтому там методы доступа пишут заранее. В Python `acc.balance` выглядит одинаково и для атрибута, и для `property` - шлюз можно добавить в тот день, когда он понадобится.

<Callout type="tip" title="Не пишите геттеры впрок">
Пара методов get_x и set_x без логики - привычка, перенесённая из других языков. В Python это шум: обычный атрибут превращается в property позже, без правки вызывающего кода.
</Callout>

Правило по умолчанию простое. Начинайте с публичного атрибута: большая часть состояния публична. Добавляйте `_`, когда поле - деталь реализации. Добавляйте `property`, когда появляется проверка, вычисление или поле только для чтения. И берите `__`, только когда пишете класс, который будут наследовать незнакомые вам люди.

## Что вас удивит

У переименования есть точные границы, и на них легко наткнуться.

Первое: имена с двумя подчёркиваниями с обеих сторон не искажаются. Поэтому `__init__`, `__repr__` и остальные dunder-имена работают как обычно - механизм затрагивает только имена вида `__name`.

Второе: переименование привязано к классу, в теле которого написан код, а не к типу объекта. Если метод класса `Reporter` обратится к `account.__balance`, компилятор перепишет это в `account._Reporter__balance` - и поиск провалится, ведь атрибут называется `_Account__balance`. Чужие «приватные» атрибуты из другого класса недоступны даже по исходному имени.

Третье, самое коварное: проверка в `__init__` без `property` не охраняет ничего, кроме момента создания.

```python
class Account:
    def __init__(self, balance):
        if balance < 0:
            raise ValueError("negative balance")
        self.balance = balance

acc = Account(100)
acc.balance = -3400
```

Конструктор отклонит отрицательный баланс, а присваивание строкой ниже молча его запишет. Это в точности баг из начала статьи: проверка есть, но стоит она не на той двери. Валидация состояния, которое может меняться после создания, живёт в сеттере `property`, а `__init__` присваивает через него.

## Типичные ошибки

**Считать `__` безопасностью.** Симптом: в атрибуте с двумя подчёркиваниями хранят секрет и верят, что снаружи его не достать. Почему случается: аналогия с `private` из других языков. На деле `__pin` лежит в объекте под именем `_Account__pin`, читается и перезаписывается любым кодом. Mangling защищает от коллизий имён, а не от чтения; для чувствительных данных нужны другие средства, а не имена атрибутов.

**`property` без сеттера и попытка присвоить.** Симптом: `AttributeError: property ... has no setter` (в старых версиях Python - `can't set attribute`) на безобидном с виду присваивании. Почему случается: `@property` определяет только путь чтения, путь записи появляется лишь с `@name.setter`. Лечится одним из двух способов: добавить сеттер либо признать поле только для чтения и менять внутреннее `_name` из методов класса.

**Валидация только в `__init__`.** Симптом: объект создаётся корректным, а ломается позже, при обычном присваивании. Почему случается: кажется, что конструктор - единственная точка входа данных, хотя присваивание атрибуту доступно всегда. Лечится переносом проверки в сеттер `property`; конструктор присваивает через публичное имя и автоматически проходит ту же проверку.

<Detail summary="Как mangling взаимодействует с наследованием">

Настоящее назначение механизма видно на двух классах:

```python
class Base:
    def __init__(self):
        self.__id = "base"

    def base_id(self):
        return self.__id

class Child(Base):
    def __init__(self):
        super().__init__()
        self.__id = "child"

c = Child()
print(vars(c))
```

Вывод: `{'_Base__id': 'base', '_Child__id': 'child'}`. Каждый класс получил собственный атрибут: код внутри `Base` скомпилирован с заменой `__id` на `_Base__id`, код внутри `Child` - с заменой на `_Child__id`. Поэтому `c.base_id()` по-прежнему возвращает `'base'`: метод родителя физически не может случайно прочитать поле потомка, и наоборот.

Ради этого mangling и придуман. Базовый класс, который наследуют десятки незнакомых подклассов, защищает свои внутренности от случайного совпадения имён: подкласс заводит свой `__id`, ничего не ломая. Обратная сторона: если подклассу законно нужно поле родителя, придётся писать `self._Base__id` с именем чужого класса посреди кода - верный признак, что полю стоило дать одно подчёркивание и сделать его частью протокола для наследников.

Замена текстовая и тотальная: любой идентификатор вида `__name` в теле класса, включая имена методов и обращения к чужим объектам, проходит через неё.

</Detail>
