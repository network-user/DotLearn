# Хеширование в базах данных и распределённых системах

Запрос `WHERE user_id = 500123` к таблице на сто миллионов строк возвращается мгновенно - база не читает все строки. Сервис с миллионами пользователей раскидывает их по десяти серверам так, чтобы нагрузка легла ровно. Кеш помнит, на какой из сотни машин лежит нужная картинка, не спрашивая каждую. Под всеми тремя задачами - одно и то же: хеширование превращает ключ в **местоположение**, будь то корзина на диске, номер сервера или точка на кольце.

В этом разделе мы посмотрим, как хеширование работает за пределами одной программы: хеш-индексы и хеш-соединения внутри СУБД, шардинг по хешу между серверами и консистентное хеширование, которое спасает распределённые системы от тотальной перетасовки при добавлении узла. Это прямое продолжение хеш-таблиц - те же идеи, но на масштабе диска и сети.

## Хеш-индекс: мгновенный поиск по равенству

Чтобы не сканировать таблицу, СУБД строит индекс. Хеш-индекс - это хеш-таблица поверх колонки: `hash(значение)` ведёт к списку указателей на строки с этим значением. Поиск по равенству (`=`) становится O(1): посчитали хеш, прыгнули в корзину, прочитали указатели.

Но у хеш-индекса есть жёсткое ограничение: он умеет только равенство. Запрос `WHERE age > 30` или `ORDER BY name` он не ускорит - хеш не сохраняет порядок, соседние значения раскиданы по всей таблице. Для диапазонов и сортировки нужен B-дерево-индекс, который хранит ключи упорядоченно. Поэтому СУБД держит оба вида: хеш-индекс - для точечных поисков, B-дерево - для диапазонов.

## Хеш-соединение: как джойнят большие таблицы

Соединить две таблицы по ключу «в лоб» - для каждой строки одной перебрать все строки другой - это O(n·m), смертельно медленно на больших данных. Хеш-соединение (hash join) делает иначе: по меньшей таблице строится хеш-таблица (фаза build), затем большая таблица проходится один раз, и каждая её строка ищет совпадения в хеш-таблице за O(1) (фаза probe). Итог - O(n + m) вместо произведения.

## Шардинг: hash % N и его беда

Когда данные не влезают на один сервер, их **шардируют** - раскладывают по нескольким машинам. Простейшее правило: `номер_сервера = hash(ключ) % N`. Оно равномерно размазывает нагрузку - при хорошем хеше каждый из N серверов получает примерно одинаковую долю.

У этого правила есть катастрофическая беда: стоит изменить N (добавить или убрать сервер), как остаток `% N` меняется почти у **всех** ключей, и почти все данные надо переносить между серверами. Запустите код и убедитесь сами - при переходе с 4 на 5 серверов переезжает подавляющее большинство ключей. Поменяйте `5` на `8` - картина та же.

<PyDemo
  title="hash % N: сколько ключей переедет при добавлении сервера"
  code={`def shard(key, n):
    return key % n

keys = list(range(1000))
moved = sum(1 for k in keys if shard(k, 4) != shard(k, 5))
print("переехало", moved, "из", len(keys), "ключей")`}
/>

Для базы это часы простоя и переливания терабайтов. Нужен способ менять число серверов, трогая лишь малую долю ключей. Этот способ - консистентное хеширование.

## Консистентное хеширование: кольцо вместо остатка

Идея: и серверы (узлы), и ключи раскладывают на одном **кольце** хеш-значений. Ключ принадлежит первому узлу по часовой стрелке от своей позиции. Теперь добавление узла затрагивает только дугу между ним и предыдущим узлом - переезжает лишь часть ключей этой дуги, а не вся система. Покрутите визуализацию: добавляйте и убирайте узлы и следите за числом переехавших ключей внизу.

Сравните с `hash % N`, где сменилось бы почти всё. Именно поэтому консистентное хеширование лежит в основе Cassandra, DynamoDB, распределённых кешей (клиенты memcached) и CDN: узлы приходят и уходят, а перетасовка остаётся локальной. На практике каждый физический узел представляют десятками виртуальных точек на кольце - так нагрузка распределяется ровнее.

## Краевые случаи

- **Перекос (skew).** Если ключ распределён неравномерно (например, шардинг по городу, где половина трафика - Москва), одна партиция станет «горячей». Лечится выбором более равномерного ключа шардирования или подмешиванием.
- **Диапазонные запросы по хеш-ключу.** Шардинг по хешу убивает диапазоны: соседние по значению ключи лежат на разных серверах. Если нужны диапазоны - шардируют по диапазону, а не по хешу.
- **Bloom-фильтр даёт ложные «да».** Вероятностная проверка принадлежности (см. ниже) никогда не ошибается с ответом «точно нет», но изредка говорит «возможно, есть» там, где элемента нет.

## Типичные ошибки

- **Хеш-индекс для диапазонов.** `WHERE price BETWEEN 100 AND 200` хеш-индекс не ускорит - он умеет только равенство. Для диапазонов нужен B-дерево.
- **Шардинг через hash % N в растущей системе.** Добавление сервера перетасует почти все данные. Для изменяемого числа узлов берут консистентное хеширование.
- **Игнорировать перекос ключа.** Равномерность нагрузки - свойство ключа, а не магии хеша. Плохой ключ шардирования создаёт горячие партиции даже при идеальной хеш-функции.

<Detail summary="Bloom-фильтр: членство почти даром">

Bloom-фильтр отвечает на вопрос «есть ли элемент в множестве», тратя в разы меньше памяти, чем само множество, - ценой редких ложных срабатываний. Это битовый массив и k независимых хеш-функций. При добавлении элемента k хешей задают k позиций, которые ставят в единицу. При проверке смотрят те же k позиций: если хоть одна - ноль, элемента **точно нет**; если все единицы - он **возможно есть** (биты могли проставить другие элементы). Ответа «точно есть» фильтр не даёт, зато «точно нет» - всегда честен. СУБД используют его, чтобы не лезть на диск за заведомо отсутствующими ключами; так работает, например, проверка в LSM-деревьях.
