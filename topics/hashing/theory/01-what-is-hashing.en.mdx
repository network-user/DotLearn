# Hashing: a short fingerprint for data of any size

Picture a paper phone book with a million subscribers. To find a name you flip pages, narrowing the range - dozens of comparisons per lookup. Now picture every name already carrying a cell number: "Petrov" lives in cell 412,879, "Sidorova" in cell 12,044. Searching stops being page-flipping: you compute the cell number straight from the name and go there directly. That idea - turning data into a number that tells you where to look - is hashing.

By the end of this section you will understand what a hash function is, what problem it solves, why its result is called a "fingerprint", where the idea came from in the 1950s, and why a hash is fundamentally not unique. This is the foundation: hash tables, Python's `dict`, database indexes, and file integrity checks all grow from it.

## Mental model: a deterministic meat grinder

A hash function is a deterministic procedure that takes input of **any** length (a number, a string, a gigabyte file) and produces output of a **fixed** size. The same input always gives the same output; different inputs almost always give different outputs. That number is the **hash**, or fingerprint: like a fingerprint, it is short but firmly tied to its owner.

## The most naive hash

To make the idea concrete, let us build the simplest hash function by hand: sum the character codes of a string and take the remainder modulo the table size. It is a bad hash function (we will soon see why), but it shows the essence - deterministic compression into a number.

```python
def naive_hash(key, size):
    total = 0
    for char in key:
        total += ord(char)
    return total % size
```

For a phone book of 1000 cells, `naive_hash("Anna", 1000)` always returns the same number. Put a record in that cell on save, look in the same cell on search. No flipping.

## Touch the hash

Run the code and look at the fingerprints of a few keys. Then change `size` from 1000 to 10 and run again: with few cells, different keys start to share a number - the first hint of collisions, which we will return to.

Notice two things. First, the two identical keys `"Anna"` gave the same number - that is **determinism**, and without it hashing is useless: you would store in one cell and search in another. Second, the fingerprint is not the key itself: from the number you cannot reconstruct the string it came from. A hash compresses data lossily and is not meant to be reversed.

## A surprise baked into the nature of a hash

Since input can be any length while output is always fixed, there are always **more** inputs than possible outputs. So different inputs with the same hash must exist. This is not a flaw of a particular function but a law: distribute infinitely many letters among finitely many mailboxes, and some box gets several letters. Such a clash is a **collision**, and getting rid of collisions entirely is impossible in principle.

<Callout type="info">
The pigeonhole principle: if there are more objects than boxes, at least one box holds more than one object. For hashing this means collisions are inevitable; the only question is how rare they are and what we do with them.
</Callout>

A good hash function does not try to repeal this law - it only spreads inputs across outputs as evenly as possible, so collisions are rare and unpredictable. That is the subject of the next section.

## Where the idea came from

<MarginNote label="History">The word hash comes from the French hacher, "to chop". A hash function literally chops data into mince you can no longer reassemble.</MarginNote>

Hashing was invented not for cryptography but for speed. In 1953 the IBM engineer Hans Peter Luhn proposed, in an internal memo, sorting records into "buckets" based on a number computed from the key. In 1956 Arnold Dumey described the division method (`hash = key % table_size`) - the very `% size` operation you just saw. Memory was then expensive and slow, and "find a record by key quickly" was everywhere: card files, symbol tables in compilers, indexes. Hashing turned out to be the answer that outlived all those eras and still works today - in `dict`, in databases, in file storage.

## What hashing does NOT do

- **It is not encryption.** A cipher is reversible: with the key the original text comes back. A hash is irreversible - it loses information on purpose. Passwords are hashed precisely because you cannot get them back.
- **It is not compression.** An archiver shrinks data while keeping the ability to restore it. A hash never returns to the original data.
- **A hash is not unique.** Because of collisions, two different inputs may give one fingerprint. Matching hashes is a reason to compare the data itself, not a reason to call it identical.

## Common misconceptions

- **"A hash is a unique id."** No: collisions are built into the nature of hashing. A hash narrows the set of candidates, but the final equality check is still done on the data itself.
- **"You can reconstruct data from a hash."** You cannot: the function loses information. Anyone "decrypting" a hash is really trying inputs and comparing their hashes - that is not inverting the function.
- **"A hash preserves order."** No: nearby keys (`100` and `101`) usually give completely different hashes. Hashing answers "where does exactly this key live", not "what comes next in order".

<Detail summary="Why a fixed output size at all">

A fixed size is not a whim but a condition for usefulness. If a fingerprint could be any length, you could not use it as a cell index in a fixed-size array, as an index key, or as a column in a table. A fixed length (32 bits, 64 bits, 256 bits) turns the hash into a convenient "coin" of one denomination: it fits into cells, compares in a single step, stores in a database column. All the later machinery - hash tables, indexes, integrity checks - relies on exactly this property.

</Detail>
