# Хеширование в Python: почему список нельзя сделать ключом

Попробуйте выполнить `{[1, 2]: "значение"}` - Python ответит `TypeError: unhashable type: 'list'`. А кортеж `{(1, 2): "значение"}` отлично работает. Почему один контейнер годится в ключи словаря, а другой - нет? Ответ - в контракте хеширования, который пронизывает весь язык: `dict`, `set`, `frozenset` устроены на хеш-таблицах, и любой объект-ключ обязан соблюдать правила, иначе таблица сломается.

В этом разделе мы разберём, что делает объект **хешируемым** (hashable), как связаны методы `__hash__` и `__eq__`, почему изменяемые объекты запрещены в ключах и зачем Python подмешивает в хеши строк случайность при каждом запуске. К концу Вы сможете сделать собственный класс пригодным для ключей словаря - и не наступить на классические грабли.

## Мысленная модель: контракт `__hash__` и `__eq__`

Объект хешируемый, если у него есть метод `__hash__`, возвращающий целое число, и метод `__eq__`, проверяющий равенство. Эти два метода связаны жёстким контрактом:

> Если два объекта равны (`a == b`), то их хеши обязаны совпадать (`hash(a) == hash(b)`).

Обратное не требуется: разные объекты могут случайно иметь одинаковый хеш (это коллизия, и таблица с ней справится). Но равные объекты с разными хешами - катастрофа: словарь положит ключ в одну корзину, а искать будет в другой, и значение «потеряется».

## Что хешируемо, а что нет

Правило простое: **неизменяемые** встроенные типы хешируемы, изменяемые - нет.

```python
hash(42)          # ок: int
hash("Москва")    # ок: str
hash((1, 2, 3))   # ок: tuple из хешируемых
hash([1, 2, 3])   # TypeError: list изменяем
hash({1: 2})      # TypeError: dict изменяем
```

Логика прямая: ключ кладут в корзину по его хешу. Если потом объект **изменить**, его хеш поменяется, и он окажется «не в той» корзине - словарь больше его не найдёт. Запрещая изменяемые объекты в ключах, Python заранее защищает таблицу от такого рассинхрона. Поэтому кортеж (неизменяемый) - годный ключ, а список - нет; `frozenset` хешируем, а обычный `set` - нет.

## Сделаем свой класс хешируемым

Запустите код: класс `City` определяет `__eq__` и `__hash__` через кортеж своих полей, поэтому два разных объекта с одинаковыми данными считаются одним ключом. Поменяйте `__hash__`, чтобы он возвращал константу (например, `return 0`), и запустите снова: словарь продолжит работать корректно, но все ключи свалятся в одну корзину - это легальный, но катастрофически медленный хеш.

<PyDemo
  title="Свой класс как ключ словаря"
  code={`class City:
    def __init__(self, name, region):
        self.name = name
        self.region = region

    def __eq__(self, other):
        return (isinstance(other, City)
                and (self.name, self.region) == (other.name, other.region))

    def __hash__(self):
        return hash((self.name, self.region))

population = {}
population[City("Казань", "Татарстан")] = 1_300_000
print(population[City("Казань", "Татарстан")])`}
/>

Заметьте: мы кладём значение по одному объекту `City`, а читаем по **другому** объекту с теми же полями - и находим. Так работает контракт: равные объекты дают равные хеши, и словарь видит их как один ключ. Без `__hash__` (но с `__eq__`) класс стал бы нехешируемым, а без `__eq__` два «одинаковых» города считались бы разными ключами.

## Случайность в хешах строк

Запустите `python -c "print(hash('a'))"` дважды в терминале - получите разные числа. Это не баг: начиная с версии 3.3, Python подмешивает в хеш строк и байтов случайное «зерно», своё на каждый запуск процесса (переменная `PYTHONHASHSEED`). Цель - защита от атаки **HashDoS**: зная хеш-функцию, злоумышленник мог бы прислать тысячи строк-ключей, которые все падают в одну корзину, и превратить O(1)-словарь в O(n²)-тормоз. Случайное зерно делает такой подбор невозможным.

<Callout type="warning">
Из-за рандомизации `hash()` строки **нельзя** сохранять между запусками или использовать как стабильный идентификатор: завтра то же слово даст другое число. Для воспроизводимого хеша берите `hashlib` (например, `sha256`) - он детерминирован.
</Callout>

## Краевые случаи

- **`hash(-1)`.** В CPython значение `-1` зарезервировано как признак ошибки в C-коде, поэтому `hash(-1)` возвращает `-2`. Мелочь, но показывает, что хеш не обязан совпадать с самим числом.
- **Порядок в словаре.** С версии 3.7 `dict` гарантирует порядок **вставки**, а не порядок хешей. Не полагайтесь на то, что ключи идут «как легли в корзины».
- **Изменённый ключ.** Если положить в ключи изменяемый объект (через хитрый `__hash__`) и потом его изменить, значение пропадёт из словаря - классический способ выстрелить себе в ногу.

## Типичные ошибки

- **`__eq__` без `__hash__`.** Если переопределить только `__eq__`, Python делает класс **нехешируемым** (`__hash__` становится `None`) - именно чтобы вы не нарушили контракт случайно. Переопределяете равенство - переопределяйте и хеш.
- **Изменяемый объект в ключах.** Список, множество, словарь нельзя класть в ключи. Нужен неизменяемый аналог: кортеж вместо списка, `frozenset` вместо `set`.
- **Доверять `hash()` строки между запусками.** Рандомизация ломает любые попытки кешировать или сравнивать строковые хеши между процессами. Для этого есть `hashlib`.

<Detail summary="Как dict устроен внутри">

`dict` в CPython - это хеш-таблица с открытой адресацией. При коллизии он не идёт линейно к соседней ячейке, а вычисляет следующую позицию по формуле возмущения (perturbation): на каждом шаге к индексу подмешиваются старшие биты исходного хеша, поэтому пробные цепочки разных ключей расходятся, и кластеризация подавляется. С версии 3.6 словарь ещё и «компактный»: пары хранятся в плотном массиве в порядке вставки, а хеш-таблица держит лишь индексы в этот массив - отсюда и гарантированный порядок вставки, и экономия памяти. `set` устроен похоже, но без значений.

</Detail>
