Key lookup in a dictionary barely depends on its size: finding a value in a dict of ten elements and one of ten million takes roughly the same time. A list cannot do that: to find an element it scans everything. The secret is that a dict does not store keys in a row; it spreads them across slots by computing a slot address from the key itself. That structure is a hash table, and understanding how it works explains all `dict` and `set` behavior: why keys must be hashable, why a dict sometimes "stutters" on insert, and why since 3.7 it guaranteed insertion order.

By the end of this section you will understand how a hash function turns a key into a slot address, what a collision is and how open addressing resolves it, how the "compact" dict of Python 3.6+ differs from a naive hash table and how that relates to insertion order, and what makes a dict grow.

## Hash table: an address from a key

The idea of a hash table is simple. Memory is allocated as an array of slots (called buckets). To store a key-value pair, Python takes the key's hash with `hash()`, turns the large hash number into a slot index (roughly, remainder modulo table size) and puts the pair there. To find a key it does the same and lands in the right slot immediately, no scan. That is where average constant-time lookup comes from.

```python
print(hash("apple"))
print(hash(42))
```

`hash()` returns an integer for any hashable object. All immutable types are hashable (`int`, `str`, `tuple` of hashables, `frozenset`), which is why a `list` cannot be a dict key or `set` element: a mutable object would change its hash and the table would "lose" it.

<Callout type="warning">A dict key must be hashable and immutable. `d = {[1, 2]: "x"}` raises `TypeError: unhashable type: 'list'`. For a composite key use `tuple`, which is hashable.</Callout>

## Collisions and open addressing

Two different keys can map to the same index: that is a collision. It is inevitable: infinitely many keys, finitely many slots. Python resolves collisions with open addressing: if the target slot is occupied by another key, the interpreter tries the next free slot by a deterministic rule (probing) until it finds an empty one. Lookup repeats the same probe sequence until it finds the key or an empty slot.

Insert keys and watch probing on collision and resize when fill reaches ~2/3 live:

<HashTableViz />

```python
d = {}
d["a"] = 1
d["b"] = 2
print("a" in d, "z" in d)
```

This prints `True False`. Checking `"a" in d` computes the bucket and goes there; if something other than `"a"` sits there, probing continues. `"z" in d` reaches an empty slot and returns `False`. The fuller the table, the longer probe chains, so it is not allowed to fill to the brim.

## Load factor and resize

When the table is about two-thirds full, probes become too long and lookup slows. The ratio "occupied / total slots" is the load factor. When it is exceeded the dict resizes: it creates a larger table and rehashes all existing keys into it. One insert can therefore be noticeably more expensive than others, but on average (amortized) insert stays fast.

```python
d = {}
for i in range(1000):
    d[i] = i * i
print(len(d))
```

This prints `1000`. As the dict grew it quietly resized and rehashed several times, invisible from outside, which is why a random single insert can "jerk".

## Try it yourself

Run the code: the dict kept insertion order `c, a, b`, not sorted keys. Change the assignment order and confirm iteration always follows insertion order. Then overwrite an existing key (`d["a"] = 99`): its position in order does not change, only the value updates.

## Compact dict and insertion order

Before 3.6 a dict's order was effectively random, determined by hashes. Starting in 3.6 CPython moved to a "compact" dict, and from 3.7 insertion order became a language guarantee, not just an implementation detail.

The compact dict trick splits storage in two. There is a dense `entries` array where key-value pairs are appended strictly in insertion order with no holes. And a sparse index table: buckets hold not the pairs themselves but small indices into `entries`. The hash table still gives fast lookup, but data sits compactly in insertion order.

```python
order = {}
for word in ["zebra", "apple", "mango"]:
    order[word] = len(word)
print(list(order))
```

This prints `['zebra', 'apple', 'mango']`: insertion order, not alphabetical or hash order. A side benefit of the compact layout is memory savings: only the light index table stays sparse; heavy entries are packed tightly.

## set: the same hash table without values

A `set` uses the same hash table but stores only keys, no attached values. Hence its properties: membership (`x in s`) in average constant time, elements must be hashable, duplicates are impossible (the same hash leads to the same slot). Important difference from dict: `set` gives no order guarantee, since it has no compact insertion-order layout.

```python
s = {3, 1, 2, 3, 1}
print(len(s), 2 in s)
```

This prints `3 True`: duplicate `3` and `1` collapsed because they lead to slots already occupied by those values.

## Common mistakes

- **Using a mutable object as a key.** `list` and `dict` are unhashable; only immutable objects can be keys. For a composite key use `tuple`.
- **Expecting a dict to sort keys.** Dict order is insertion order, not sorting. For sorted traversal use `sorted(d)`.
- **Relying on `set` order.** Unlike `dict`, a set guarantees no order; do not write code that depends on iteration order of a `set`.

<Detail summary="Why len() and in are so fast, and why hash is cached">

For strings CPython computes the hash once and caches it inside the string object, so repeated `hash(s)` for the same string is not recomputed. That makes repeated lookups by string key especially cheap. And `len(d)` is instant because the dict keeps a separate element count and does not walk them. Checking `x in d` is computing a bucket and a short probe chain, average O(1) regardless of dict size. All of this is CPython implementation detail; the language standard promises operation complexity, not internal layout.

</Detail>
