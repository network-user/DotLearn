Когда последнее имя перестаёт указывать на объект, объект больше никому не нужен, и память под ним стоит освободить. В языках с ручным управлением памятью это делает программист, в Java и Go - трассирующий сборщик, запускаемый периодически. CPython выбрал третий путь: он считает ссылки на каждый объект и освобождает объект в тот же миг, когда счётчик падает до нуля. Это быстро и предсказуемо, но у подхода есть слепое пятно - циклы, - и ради них существует второй, дополнительный механизм.

К концу раздела Вы будете понимать, что увеличивает и уменьшает счётчик ссылок, почему `sys.getrefcount` всегда показывает на единицу больше, чем Вы ожидаете, как возникает циклическая ссылка, которую подсчёт не может убрать, и зачем нужны модуль `gc` с поколениями и слабые ссылки `weakref`.

## Reference counting: счётчик внутри объекта

Каждый объект в CPython хранит внутри себя число - количество ссылок на него. Создали ссылку - счётчик `+1`, ссылка исчезла - счётчик `-1`. Как только счётчик достигает нуля, объект немедленно уничтожается, а память возвращается. Это важная черта CPython: деструктор (`__del__`) у объекта без циклов вызывается сразу, детерминированно, а не «когда-нибудь потом».

Счётчик увеличивают: присваивание имени, добавление объекта в список или словарь, передача в функцию как аргумент. Уменьшают: выход имени из области видимости, `del имя`, переназначение имени на другой объект, удаление из контейнера.

```python
import sys
data = []
print(sys.getrefcount(data) - 1)
alias = data
print(sys.getrefcount(data) - 1)
```

Первое число `1`, второе `2`: имя `alias` добавило одну ссылку. Мы вычитаем единицу, потому что у `getrefcount` есть особенность.

<Callout type="info">`sys.getrefcount(x)` всегда показывает на единицу больше «настоящего» числа ссылок: сам аргумент `x` функции - это ещё одна временная ссылка на объект, существующая на время вызова. Поэтому в коде выше мы пишем `- 1`. Конкретные абсолютные значения зависят от реализации; полагайтесь на разницу до и после, а не на само число.</Callout>

Пунктирная стрелка от `b` - та, что вот-вот исчезнет. После `del b` счётчик станет `2`, объект жив. Уберите ещё `a` и `c` - счётчик дойдёт до нуля, и список будет уничтожен в тот же момент.

```python
import sys
x = object()
y = x
z = x
print(sys.getrefcount(x) - 1)
del y
print(sys.getrefcount(x) - 1)
```

Печатается `3`, затем `2`: три имени (`x`, `y`, `z`) дают три ссылки, `del y` убирает одну. Заметьте: `del y` удаляет имя, а не объект. Объект исчезнет, только когда исчезнет последняя ссылка.

## Слепое пятно: циклические ссылки

Подсчёт ссылок ломается на циклах. Если два объекта ссылаются друг на друга, их счётчики никогда не дойдут до нуля сами по себе - даже когда снаружи на них больше никто не смотрит.

```python
a = {}
b = {}
a["other"] = b
b["other"] = a
del a
del b
```

После `del a` и `del b` ни одно имя не указывает на эти словари, но они держат друг друга: счётчик каждого равен `1`. Для подсчёта ссылок они «живы», хотя программе недостижимы - это утечка памяти.

## Сборщик мусора и поколения

Ровно ради таких циклов CPython держит второй механизм - трассирующий сборщик мусора в модуле `gc`. Он периодически обходит объекты-контейнеры, находит группы, достижимые только изнутри себя (недостижимые циклы), и уничтожает их. Подсчёт ссылок убирает подавляющее большинство мусора мгновенно, а `gc` подчищает циклы.

Сборщик работает по поколениям (generations) на гипотезе: большинство объектов умирает молодыми. Новые объекты попадают в поколение 0, которое проверяется часто. Пережившие сборку переходят в поколение 1, затем в поколение 2 - их проверяют всё реже, потому что долгоживущий объект вряд ли вдруг станет мусором. Это снижает накладные расходы.

```python
import gc
collected = gc.collect()
print(type(collected))
```

`gc.collect()` запускает полную сборку вручную и возвращает число найденных недостижимых объектов (`int`). Обычно трогать его не нужно - сборщик работает сам, - но при отладке утечек это полезно.

## weakref: ссылка, не удерживающая объект

Иногда нужно сослаться на объект, но не мешать ему умереть - например, в кэше. Обычная ссылка увеличивает счётчик и держит объект живым. Слабая ссылка (`weakref`) указывает на объект, не увеличивая счётчик: как только исчезнут все обычные ссылки, объект уничтожается, а слабая ссылка «протухает».

```python
import weakref

class Node:
    pass

n = Node()
ref = weakref.ref(n)
print(ref() is n)
del n
print(ref() is None)
```

Печатается `True`, затем `True`: пока `n` жив, `ref()` возвращает его; после `del n` обычных ссылок не осталось, объект уничтожен, и `ref()` отдаёт `None`. Слабые ссылки также разрывают циклы: если одно из звеньев цикла слабое, подсчёт ссылок сам соберёт остальное без участия `gc`.

## Типичные ошибки

- **Считать абсолютное значение `getrefcount` значимым.** Оно всегда на единицу выше из-за аргумента самой функции и зависит от внутренних оптимизаций интерпретатора. Смотрите на изменение, а не на число.
- **Думать, что `del x` уничтожает объект.** `del x` убирает имя `x` и уменьшает счётчик на единицу. Объект уничтожается, только когда счётчик дошёл до нуля.
- **Полагаться на немедленный вызов `__del__` для объектов в циклах.** Объект в недостижимом цикле уничтожит сборщик `gc` когда-нибудь, а не сразу - поэтому критичные ресурсы (файлы, сокеты) освобождайте явно через `with` или `close()`, не через `__del__`.

<Detail summary="Почему GIL и подсчёт ссылок связаны">

Счётчик ссылок меняется на каждое присваивание и удаление - это происходит постоянно и должно быть атомарным, иначе два потока повредят счётчик и память. В CPython это гарантирует глобальная блокировка интерпретатора (GIL): в каждый момент байткод исполняет только один поток, поэтому операции со счётчиком не требуют дорогой синхронизации на каждое изменение. Это одна из исторических причин существования GIL - и одна из причин, почему убрать его сложно. Деталь специфична для CPython; реализации без подсчёта ссылок (Jython на JVM) GIL не имеют.

</Detail>
