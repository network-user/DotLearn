Гибкость Python не бесплатна. Возможность в любой момент добавить объекту новый атрибут означает, что у каждого экземпляра есть личный словарь. Возможность списка расти означает, что под него выделено больше памяти, чем он сейчас занимает. Чаще всего эти издержки незаметны - но стоит создать миллион мелких объектов или прочитать гигабайт чисел, как раскладка данных в памяти становится решающей. CPython даёт несколько рычагов, чтобы её ужать: `__slots__` вместо словаря, кеширование мелких объектов, продуманный рост списка и специализированные контейнеры из `collections`.

К концу раздела Вы будете понимать, на что тратится память обычного объекта и как `__slots__` её срезает, почему `is` иногда неожиданно возвращает `True`, почему `append` в среднем дёшев, несмотря на переезды списка, и какие структуры из `collections` решают типичные задачи лучше «голых» dict и list.

## __slots__: убираем личный словарь экземпляра

По умолчанию атрибуты экземпляра хранятся в его `__dict__` - обычном словаре, привязанном к объекту. Это удобно: можно навесить любой атрибут когда угодно. Но словарь - это хеш-таблица со своим заголовком и запасом бакетов, и на каждый мелкий объект он добавляет ощутимый перерасход. Если объявить в классе `__slots__`, Python заменит словарь компактным набором фиксированных ячеек под заранее перечисленные имена.

```python
class PlainPoint:
    def __init__(self, x, y):
        self.x = x
        self.y = y

class SlotPoint:
    __slots__ = ("x", "y")
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = SlotPoint(1, 2)
print(hasattr(p, "__dict__"))
```

Печатается `False`: у `SlotPoint` нет `__dict__`, его атрибуты лежат в фиксированных слотах. Платой за экономию становится потеря гибкости - присвоить `p.z = 3` уже нельзя, будет `AttributeError`, потому что складывать незаявленный атрибут больше некуда.

<Callout type="warning">
`__slots__` фиксирует набор атрибутов и убирает `__dict__`. Не используйте его «на всякий случай»: он мешает динамически навешивать атрибуты, конфликтует с некоторыми паттернами множественного наследования и почти не даёт выигрыша на немногочисленных объектах. Это инструмент для массовых мелких экземпляров, а не стиль по умолчанию.
</Callout>

## Кеширование мелких объектов: почему is иногда True

CPython заранее создаёт и переиспользует объекты для маленьких целых чисел в диапазоне от -5 до 256. Любое `10` в программе - это один и тот же объект, поэтому `a = 10; b = 10` дают два имени, указывающих в одно место.

```python
a = 10
b = 10
print(a is b)

x = 1000
y = 1000
print(x is y)
```

Первая строка печатает `True`: `10` кешировано. Вторая в большинстве случаев печатает `False` - `1000` вне кеша, и это два разных объекта с одинаковым значением. Отсюда железное правило: числа и вообще значения сравнивают через `==`, а `is` оставляют для проверки идентичности и для синглтонов вроде `None`.

<Callout type="tip">
`is` отвечает на вопрос «это буквально один и тот же объект?», а `==` - «значения равны?». Используйте `is` только с `None`, `True`, `False` и осознанными проверками идентичности. Полагаться на кеширование мелких целых в логике нельзя - это деталь реализации CPython.
</Callout>

Похожий приём - интернирование строк: короткие строки, похожие на идентификаторы, CPython тоже хранит в единственном экземпляре, чтобы сравнение и использование как ключей было быстрее. Но, как и с целыми, это оптимизация под капотом, а не контракт, на который стоит опираться.

## Рост списка: запас ёмкости и амортизированный append

Список хранит элементы в непрерывном блоке памяти. Если бы каждый `append` выделял блок ровно по размеру, добавление элемента всякий раз копировало бы весь список - это O(n) на операцию. CPython избегает этого, выделяя память с запасом: ёмкость (capacity) списка обычно больше его длины. Пока запас есть, `append` просто кладёт элемент в свободную ячейку за O(1). Когда запас исчерпан, список увеличивает ёмкость кратно и переезжает один раз - и снова надолго.

Именно поэтому отдельная вставка изредка заметно дольше остальных (попала на переезд), но в среднем (амортизированно) `append` остаётся O(1). У неизменяемого `tuple` запаса ёмкости нет вовсе: он фиксирован, занимает ровно нужное и потому компактнее списка - но «добавление» к нему всегда создаёт новый объект.

## collections: специализированные хранилища

Стандартный модуль `collections` даёт контейнеры, которые под капотом устроены эффективнее, чем ручные конструкции на `dict` и `list`.

```python
from collections import deque, Counter, defaultdict

queue = deque([1, 2, 3])
queue.appendleft(0)

freq = Counter("banana")
groups = defaultdict(list)
groups["fruit"].append("banana")

print(list(queue))
print(freq["a"])
print(groups["fruit"])
```

Печатается `[0, 1, 2, 3]`, `3` и `['banana']`. Каждый контейнер закрывает свою боль:

- **`deque`** - двусторонняя очередь на блоках связанной памяти. Вставка и удаление с обоих концов за O(1), тогда как у списка `insert(0, x)` сдвигает все элементы и стоит O(n).
- **`Counter`** - словарь-счётчик: подсчёт частот без ручной проверки «есть ли уже ключ».
- **`defaultdict`** - словарь с фабрикой значений по умолчанию: обращение к новому ключу само создаёт пустой список/число, избавляя от `if key not in d`.

## Типичные ошибки

- **Вешать `__slots__` на каждый класс.** На немногочисленных объектах выигрыш близок к нулю, а гибкость теряется. Применяйте там, где экземпляров миллионы.
- **Полагаться на `is` для чисел и строк.** Кеширование мелких целых и интернирование строк - детали реализации. Для сравнения значений всегда `==`.
- **Делать `list.insert(0, x)` в цикле.** Каждая вставка в начало сдвигает весь список - O(n). Для очереди берите `deque`.
- **Ручной подсчёт частот через `if key in d`.** `Counter` и `defaultdict` короче, быстрее и без забытых веток инициализации.

<Detail summary="Почему __slots__ не всегда уменьшает память, если есть наследник без слотов">

`__slots__` убирает `__dict__` только если он убран по всей цепочке наследования. Стоит хотя бы одному классу в иерархии не объявить `__slots__` - и у экземпляров снова появится словарь, потому что наследник «вернул» гибкость. Поэтому ради экономии слоты объявляют у всех классов цепочки, а пустой `__slots__ = ()` у промежуточного класса говорит «я тоже не добавляю атрибутов». Детали - в [документации по __slots__](https://docs.python.org/3/reference/datamodel.html#slots).

</Detail>
