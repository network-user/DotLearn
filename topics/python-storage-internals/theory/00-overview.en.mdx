You wrote `b = a`, changed something in `b`, and suddenly `a` changed too — even though you never touched it. Or a production process eats gigabytes of memory where your back-of-the-envelope math said megabytes should suffice. Surprises like these stop feeling like magic once you see what happens one level below the syntax: where data actually lives, who references it, when memory is freed, and why one dictionary takes more space than it seems. This topic is about that layer, usually hidden behind Python's convenience.

**What you'll be able to do.** Tell a name from an object and understand why two lists suddenly became one; explain when CPython frees memory and where that mechanism stumbles; read `dict` and `set` behavior through the hash table, not guesswork; and deliberately shrink program memory by choosing the right data layout.

**The route through this topic.** Four sections, each with a sandbox where you can run code right in the browser:

1. **Objects and references** — a variable is a label on an object, not a box, and why that breaks intuition about assignment.
2. **Reference counting** — how CPython counts references and frees objects immediately, and where the cycle collector steps in.
3. **dict and set internals** — the hash table that makes key lookup almost independent of size.
4. **Memory layout optimizations** — cut overhead with `__slots__`, small-object caches, and containers from `collections`.

**What you need first.** Nothing to install: Python runs right in the browser. Comfort with Python classes and objects helps — we cover memory and internals along the way.
