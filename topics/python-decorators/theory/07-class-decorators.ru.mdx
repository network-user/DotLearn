До сих пор Вы декорировали функции функциями. Но `@` работает шире, чем кажется. Им можно украсить не только функцию, но и класс - так устроены `@dataclass` и `@functools.total_ordering`. И наоборот: декоратором может служить не только функция, но и класс - если у него есть `__call__`. В этом разделе мы разберём обе симметрии и поймём, когда удобнее класс-декоратор, а когда функция.

К концу раздела Вы сможете написать декоратор, который принимает класс и дополняет его, и декоратор-класс, который хранит состояние в полях экземпляра вместо замыкания.

## @ над классом получает класс и возвращает класс

Когда `@` стоит над `class`, он работает по тому же правилу: `@register` над `class User` означает `User = register(User)`. Декоратор получает объект-класс, может его осмотреть или дополнить и должен вернуть класс (обычно тот же самый). Это удобно, чтобы добавить классу методы, зарегистрировать его в реестре или провалидировать.

```python
def add_repr(cls):
    def __repr__(self):
        return f"{cls.__name__}({self.__dict__})"
    cls.__repr__ = __repr__
    return cls

@add_repr
class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

print(repr(Point(1, 2)))
```

Печатается `Point({'x': 1, 'y': 2})`. Декоратор `add_repr` получил класс `Point`, привязал к нему новый метод `__repr__` и вернул тот же класс. Снаружи `Point` остался `Point`, но обзавёлся человекочитаемым представлением. Обратите внимание: декоратор класса не оборачивает его в другой объект, а чаще всего мутирует и возвращает исходный класс.

## Реестр классов одной строкой

Частый практический приём - собрать все классы определённого вида в реестр, не перечисляя их вручную. Декоратор записывает класс в общий словарь прямо в момент определения.

```python
PLUGINS = {}

def plugin(cls):
    PLUGINS[cls.__name__] = cls
    return cls

@plugin
class Exporter:
    pass

@plugin
class Importer:
    pass

print(sorted(PLUGINS))
```

Печатается `['Exporter', 'Importer']`. Каждый `@plugin` при определении класса добавил его в `PLUGINS`. Теперь реестр заполняется сам собой: добавили новый класс с `@plugin` - он автоматически попал в словарь, никакого ручного списка.

## Класс как декоратор: состояние в полях, а не в замыкании

Теперь обратная симметрия. Декоратор - это «то, что вызывают на функции и что потом вызывают вместо неё». Функция - не единственный вызываемый объект: экземпляр класса с методом `__call__` тоже вызывается. Поэтому декоратором может быть класс: его `__init__` принимает функцию, а `__call__` играет роль обёртки.

```python
class CountCalls:
    def __init__(self, func):
        self.func = func
        self.count = 0

    def __call__(self, *args, **kwargs):
        self.count += 1
        return self.func(*args, **kwargs)

@CountCalls
def hello():
    return "привет"

hello()
hello()
print(hello.count)
```

Печатается `2`. Разберём. `@CountCalls` означает `hello = CountCalls(hello)` - имя `hello` теперь указывает на экземпляр класса. При каждом вызове `hello()` срабатывает `__call__`: увеличивает `self.count` и зовёт исходную функцию. Состояние (`count`) живёт в поле экземпляра, а не в захваченной переменной замыкания. Бонус: счётчик доступен снаружи как `hello.count` - у функции-декоратора такого естественного места для публичного состояния нет.

## Когда поведение удивит

Класс-декоратор без аргументов прост, но класс-декоратор-фабрика с аргументами устроен иначе, чем кажется по аналогии с функциями. Если у класса есть `__init__(self, *deco_args)`, то `@Repeat(3)` сначала создаёт экземпляр `Repeat(3)`, а уже этот экземпляр применяется к функции - значит, роль «декоратора» играет `__call__`, и он принимает функцию, а не аргументы вызова.

```python
class Repeat:
    def __init__(self, times):
        self.times = times

    def __call__(self, func):
        def wrapper(*args, **kwargs):
            result = None
            for _ in range(self.times):
                result = func(*args, **kwargs)
            return result
        return wrapper

@Repeat(3)
def ping():
    print("ping")

ping()
```

Печатается `ping` трижды. Здесь `Repeat(3)` - экземпляр, `__call__(func)` - средний уровень (принимает функцию), а `wrapper` - внутренний. То есть класс-фабрика смещает роли на уровень: `__call__` принимает уже не аргументы вызова, а саму функцию. Перепутать это с обычным класс-декоратором (где `__call__` - это обёртка) - частая ошибка.

## Типичные ошибки

- **Забыть вернуть класс.** Декоратор класса, который дополняет `cls`, но не делает `return cls`, превратит имя класса в `None`. Всегда возвращайте класс.
- **Потерять `__name__`/метаданные у функции при класс-декораторе.** `@CountCalls` заменяет функцию экземпляром, у которого нет `__name__` функции. Если метаданные важны, копируйте их через `functools.update_wrapper(self, func)` в `__init__`.
- **Спутать `__call__` обычного класс-декоратора и класс-фабрики.** Без аргументов `__call__` - это обёртка (принимает аргументы вызова). С аргументами `__call__` принимает функцию, а обёртку возвращает он.

<Detail summary="Почему @dataclass - это декоратор класса">

Стандартный `@dataclass` - ровно такой декоратор: он получает класс, читает его аннотации полей и генерирует за Вас `__init__`, `__repr__`, `__eq__` и при желании другие методы, после чего возвращает тот же класс. Это показывает мощь приёма: декоратор класса может сгенерировать целый набор методов на основе того, что объявлено в теле. `functools.total_ordering` работает похоже - дописывает недостающие операторы сравнения по одному, который Вы определили. Оба не оборачивают класс в другой объект, а обогащают исходный и возвращают его.

</Detail>
