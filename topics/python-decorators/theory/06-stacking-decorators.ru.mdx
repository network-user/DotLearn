На реальную функцию часто вешают сразу несколько декораторов: проверить права, заглянуть в кэш, записать в лог. Они стоят друг над другом, и порядок строк внезапно начинает значить. Поменяете местами `@cache` и `@log` - и логироваться будут только промахи кэша, а не все вызовы. В этом разделе мы разберём, в каком порядке декораторы применяются, в каком - выполняются, и почему это не одно и то же.

К концу раздела Вы будете уверенно читать стопку из нескольких `@`, предсказывать порядок их срабатывания и понимать, почему стек декораторов разворачивается, как стопка тарелок: последний навешенный - первый снаружи.

## Применяются снизу вверх

Несколько декораторов над одной функцией применяются снизу вверх - от ближайшего к `def` к самому верхнему. Запись

```python
@outer
@inner
def func():
    ...
```

эквивалентна `func = outer(inner(func))`. Сначала к функции применяется `inner` (он ближе), результат заворачивается в `outer`. Поэтому самый верхний декоратор оказывается самым внешним слоем, а нижний - самым близким к исходной функции.

Итог - луковица из обёрток: `outer(inner(func))`. Снаружи `outer`, под ним `inner`, в сердцевине - исходная `func`.

## Выполняются снаружи внутрь, а возвращаются изнутри наружу

Применение - это про сборку слоёв при определении функции. Выполнение - про то, что происходит при вызове. И тут порядок обратный по ощущению, но логичный: вызов входит через самый внешний слой и проваливается внутрь, а результат всплывает обратно. Это в точности поведение стека вызовов: кадры кладутся сверху, пока не достигнут дна, потом снимаются в обратном порядке.

<CallStackViz frames={['@outer', '@inner']} innerLabel="func()" />

Посмотрите код с «до» и «после» в каждом слое:

```python
def outer(func):
    def wrapper(*args, **kwargs):
        print("outer до")
        result = func(*args, **kwargs)
        print("outer после")
        return result
    return wrapper

def inner(func):
    def wrapper(*args, **kwargs):
        print("inner до")
        result = func(*args, **kwargs)
        print("inner после")
        return result
    return wrapper
```

```python
@outer
@inner
def task():
    print("тело")

task()
```

Печатается:

```
outer до
inner до
тело
inner после
outer после
```

Прочитайте вывод как спуск и подъём. Вызов входит в `outer` (внешний слой) - печатается `outer до`. `outer` зовёт то, что под ним, то есть `inner` - `inner до`. `inner` зовёт настоящую функцию - `тело`. Дальше подъём: возвращается `inner` - `inner после`, возвращается `outer` - `outer после`. Симметрия «до» сверху вниз, «после» снизу вверх - подпись стека.

## Потрогайте руками

Запустите код: два декоратора, `bold` и `italic`, оборачивают результат в разметку. Посмотрите, в каком порядке легли теги. Затем поменяйте `@bold` и `@italic` местами и убедитесь, что внешний тег - всегда верхний декоратор: порядок строк управляет порядком вложенности.

## Почему порядок меняет смысл, а не только вид

С тегами разница косметическая, но с настоящими декораторами она меняет логику. Сравните `@log` над `@cache` и наоборот.

```python
@log
@cache
def fetch(key):
    ...
```

Здесь `log` - внешний, `cache` - внутренний. Вызов сначала входит в `log` (логируется каждый вызов), потом в `cache` (проверяет кэш). Логируются все обращения. А если поменять местами - `cache` снаружи, `log` внутри, - то на попадании в кэш `cache` вернёт ответ сам и до `log` дело не дойдёт: залогируются только промахи. Один и тот же набор декораторов, переставленный местами, даёт разное наблюдаемое поведение.

<Callout type="tip">Кладите «сквозные» декораторы, которые должны видеть каждый вызов (лог, метрики, проверка прав), выше кэширующих и фильтрующих, которые могут оборвать цепочку и вернуть результат досрочно.</Callout>

## Когда поведение удивит

Если в стеке нет `functools.wraps`, метаданные теряет каждый слой, и наружу всплывает имя самой внешней обёртки. Но есть и более тонкий сюрприз: декораторы применяются при определении функции, поэтому код самих декораторов (вне их обёрток) выполняется сразу при импорте, причём снизу вверх.

```python
def announce(name):
    def decorator(func):
        print("навешиваю", name)
        return func
    return decorator

@announce("верхний")
@announce("нижний")
def f():
    pass
```

Печатается при определении:

```
навешиваю нижний
навешиваю верхний
```

Сначала срабатывает нижний `@announce("нижний")` (он ближе к `def`), потом верхний. Это та же логика «снизу вверх» при применении - просто здесь декораторы что-то печатают прямо в момент навешивания, и порядок становится виден.

## Типичные ошибки

- **Считать, что верхний декоратор выполняется последним.** Верхний применяется последним (становится внешним слоем), но при вызове выполняется первым - вход идёт снаружи внутрь.
- **Путать порядок применения и выполнения.** Применение: снизу вверх. Выполнение «до»: сверху вниз. Выполнение «после»: снизу вверх. Это три разных направления, и их легко смешать.
- **Ставить обрывающий декоратор (кэш) над логом.** Тогда лог не увидит обслуженные кэшем вызовы. Решайте по смыслу, кто должен видеть каждый вызов.

<Detail summary="Стек декораторов и стек вызовов - это одно и то же">

Когда декораторы вложены, при вызове образуется самый настоящий стек кадров: `wrapper` от `outer` вызывает `wrapper` от `inner`, тот вызывает `func`. Каждый вызов кладёт кадр на стек интерпретатора. Поэтому глубоко вложенные декораторы заметны в трейсбэке: при ошибке внутри `func` Вы увидите цепочку `wrapper -> wrapper -> func`. Именно поэтому `functools.wraps` так ценен - без него все кадры в трейсбэке называются `wrapper`, и понять, чей это слой, нельзя. LIFO-разворачивание стека (последним вошёл - первым вышел) - ровно то, что показывает симметрия «до/после» в выводе.

</Detail>
