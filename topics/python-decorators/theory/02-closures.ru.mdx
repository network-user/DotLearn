Вы вернули функцию из функции в прошлом разделе и, возможно, не заметили странности. Внутренняя функция использовала переменную, которая жила во внешней. Но внешняя к моменту вызова внутренней уже завершилась - её локальные переменные должны были исчезнуть. Тем не менее внутренняя функция продолжает их видеть. Этот фокус называется замыканием (closure), и без него декоратор не смог бы запомнить, какую функцию он оборачивает.

К концу раздела Вы научитесь создавать фабрики функций, у которых «зашита» настройка, понимать, что замыкание захватывает переменную, а не её значение в момент создания, чинить классическую ловушку «все лямбды в цикле дают одно число» и менять захваченное состояние через `nonlocal`.

## Внутренняя функция помнит, где родилась

Когда `def` создаёт функцию внутри другой функции, новая функция запоминает не значения, а ссылку на область видимости, в которой её определили. Эта связка «функция плюс захваченные внешние переменные» и есть замыкание. Переменные, которые внутренняя функция берёт снаружи, называют свободными (free variables).

```python
def make_adder(n):
    def adder(x):
        return x + n
    return adder

add5 = make_adder(5)
add10 = make_adder(10)
print(add5(100), add10(100))
```

Печатается `105 110`. Каждый вызов `make_adder` создаёт свежую внутреннюю функцию и свою ячейку для `n`. `add5` держит ячейку со значением `5`, `add10` - со значением `10`, и они не мешают друг другу, хотя обе родились из одного `def adder`.

Свободные переменные видны и явно: у функции есть атрибут `__closure__`, а имена захваченных переменных лежат в `adder.__code__.co_freevars`. Это не магия и не глобальные переменные - это связанные с конкретным объектом-функцией ячейки.

## Замыкание захватывает переменную, а не значение

Вот ключевая тонкость, на которой спотыкаются почти все. Внутренняя функция держит ссылку на саму переменную внешней области, а не на снимок её значения в момент создания. Если значение переменной изменится после того, как замыкание создано, замыкание увидит новое значение. Это называют поздним связыванием (late binding): имя разрешается в момент вызова, а не в момент определения.

```python
def make():
    value = 1
    def show():
        return value
    value = 99
    return show

print(make()())
```

Печатается `99`, а не `1`. К моменту, когда `show` действительно вызвали, `value` во внешней области уже равно `99`. `show` смотрит на текущее содержимое ячейки, а не на то, что было при её определении.

## Потрогайте классическую ловушку руками

Позднее связывание превращается в баг в цикле. Кажется, что каждая лямбда «запомнит» своё `i`, но все они захватывают одну и ту же переменную цикла, и к моменту вызова она равна последнему значению. Запустите код - вместо ожидаемых `0 1 2` получится `2 2 2`. Затем почините: добавьте лямбде параметр со значением по умолчанию `lambda i=i: i`, который снимает копию `i` в момент создания лямбды, и убедитесь, что вывод стал `0 1 2`.

    funcs.append(lambda: i)

Почему `default`-параметр чинит ловушку: значение аргумента по умолчанию вычисляется в момент создания функции, поэтому `i=i` берёт текущее `i` и кладёт его в собственный параметр лямбды - это уже снимок, а не ссылка на общую переменную.

## nonlocal: менять захваченную переменную

По умолчанию внутренняя функция может читать свободную переменную, но не присваивать ей. Любое присваивание `count = count + 1` внутри `inner` Python считает созданием новой локальной переменной `count`, и тогда чтение `count` справа падает с `UnboundLocalError`. Чтобы присваивать именно внешней переменной, её объявляют через `nonlocal`.

```python
def make_counter():
    count = 0
    def tick():
        nonlocal count
        count += 1
        return count
    return tick

c = make_counter()
print(c(), c(), c())
```

Печатается `1 2 3`. `nonlocal count` говорит: «`count` - не моя локальная, а из ближайшей внешней функции; присваивай туда». Без этой строки `count += 1` упал бы, потому что Python увидел бы присваивание и счёл `count` локальной, а справа её ещё нет.

<Callout type="warning">`nonlocal` нужен только для присваивания свободной переменной. Если внутри Вы лишь читаете её (как в `make_adder`), `nonlocal` не требуется и будет лишним.</Callout>

Этот счётчик на замыкании - в миниатюре то, что делает декоратор: оборачивающая функция держит в свободной переменной либо саму обёрнутую функцию, либо накопленное состояние (кэш, число вызовов), и меняет его через `nonlocal`.

## Когда поведение удивит

Замыкание держит ссылку, поэтому состояние переживает выход из внешней функции и сохраняется между вызовами внутренней. Для счётчика это и нужно, но иногда удивляет: два разных замыкания не делят состояние, а одно и то же - делит.

```python
def make_counter():
    count = 0
    def tick():
        nonlocal count
        count += 1
        return count
    return tick

a = make_counter()
b = make_counter()
print(a(), a(), b())
```

Печатается `1 2 1`. `a` и `b` - результаты двух разных вызовов фабрики, у каждого своя ячейка `count`, поэтому `b()` стартует с нуля и не видит того, что насчитал `a`. А вот если бы Вы написали `b = a`, оба имени смотрели бы на одно замыкание и делили один счётчик.

## Типичные ошибки

- **Захват переменной цикла.** `[lambda: i for i in range(3)]` даёт три лямбды, читающие одну переменную `i`; после цикла все вернут `2`. Снимайте копию через параметр по умолчанию: `lambda i=i: i`.
- **Забытый `nonlocal`.** Присваивание свободной переменной без `nonlocal` создаёт новую локальную, и чтение до присваивания падает с `UnboundLocalError`. Объявляйте `nonlocal` перед изменением.
- **Ожидание снимка значения.** Замыкание видит текущее значение переменной в момент вызова, а не на момент определения. Если нужно зафиксировать значение, передайте его параметром или аргументом по умолчанию.

<Detail summary="Чем замыкание отличается от глобальной переменной">

И замыкание, и глобальная переменная позволяют внутренней функции «помнить» что-то между вызовами. Разница - в изоляции. Глобальная переменная одна на всю программу: любой код может её прочитать и испортить. Захваченная замыканием ячейка привязана к конкретному объекту-функции и недоступна по имени снаружи - к ней нельзя обратиться из другого места модуля. Поэтому счётчик на замыкании безопаснее счётчика на `global`: два независимых счётчика не конфликтуют, и никто посторонний не сбросит их состояние.

```python
def make_counter():
    count = 0
    def tick():
        nonlocal count
        count += 1
        return count
    return tick

print(make_counter().__code__.co_freevars)
```

Печатается `('count',)`: имя захваченной переменной хранится в самой функции, а не в глобальном пространстве.
