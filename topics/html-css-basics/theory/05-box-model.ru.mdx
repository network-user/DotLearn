# Ширина 300 пикселей оказалась 342

Классическая первая встреча с CSS: вы задаёте блоку `width: 300px`, добавляете рамку и внутренний отступ для красоты, и сетка из трёх таких блоков вдруг перестаёт помещаться в ряд, хотя арифметика вроде бы сходилась. Дело не в баге браузера. Дело в том, что «ширина элемента» в CSS изначально означает не то, что кажется на глаз, а конкретный один из четырёх вложенных слоёв, из которых состоит любой блок.

## Четыре слоя одной коробки

Каждый элемент на странице браузер представляет как прямоугольную коробку из четырёх концентрических слоёв, вложенных друг в друга наружу от содержимого.

`content` — область, где рисуется текст или вложенные элементы; именно её размеры задают свойства `width` и `height` по умолчанию. `padding` — внутренний отступ между содержимым и рамкой, закрашивается фоном элемента так же, как и содержимое. `border` — сама рамка, толщина которой прибавляется поверх padding. `margin` — внешний отступ от соседних элементов, всегда прозрачный: фон элемента на margin не заходит, это чистый воздух между коробками.

```css
.card {
  width: 300px;
  padding: 20px;
  border: 1px solid gray;
  margin: 16px;
}
```

## Откуда взялись лишние пиксели

По умолчанию (`box-sizing: content-box`, значение спецификации) свойство `width` задаёт ширину именно слоя `content`, а padding и border добавляются к ней сверху, увеличивая итоговую видимую ширину коробки. Для блока выше: 300px content + 20px padding слева + 20px padding справа + 1px border слева + 1px border справа = 342px итоговой ширины, которую блок реально займёт на странице. Отсюда и «пропавшие» 42 пикселя из вступления — они никуда не пропадали, они всегда были снаружи заданных 300px, просто по умолчанию не считаются в них.

```css
.card {
  box-sizing: border-box;
  width: 300px; /* теперь это итоговая ширина, border и padding входят внутрь */
  padding: 20px;
  border: 1px solid gray;
}
```

`box-sizing: border-box` меняет саму единицу измерения `width`: теперь она означает итоговую видимую ширину коробки вместе с padding и border, а браузер сам уменьшает область содержимого, чтобы всё уместилось. Ширина 300px остаётся 300px независимо от толщины рамки и отступов. Из-за этого удобства современные CSS-сбросы почти всегда включают правило `*, *::before, *::after { box-sizing: border-box; }` в самом начале файла стилей, чтобы избавить весь проект от подобной арифметики раз и навсегда.

<MarginNote label="Наследуемость">
box-sizing не наследуется браузерными стилями по умолчанию, поэтому глобальный сброс пишут через универсальный селектор `*`, а не просто на `body` в расчёте на наследование.
</MarginNote>

## Margin умеет схлопываться, padding — никогда

Есть асимметрия, которая удивляет чаще прочего: вертикальные (верхний и нижний) внешние отступы двух соседних блочных элементов не складываются, а схлопываются в один, равный большему из двух.

```css
.article-title { margin-bottom: 40px; }
.article-lead   { margin-top: 20px; }
```

Между заголовком и следующим за ним абзацем окажется не 60px (40+20), а ровно 40px — больший из двух margin поглощает меньший. Правило действует только для margin и только вертикально в обычном потоке документа; padding никогда не схлопывается, каждый его пиксель складывается с соседним честно. Схлопывание — не баг, а сознательное решение авторов CSS: без него вложенные абзацы с одинаковым `margin` создавали бы неровные, визуально «прыгающие» расстояния там, где автор ожидал единый ритм.

<Callout type="warning">
Схлопывание работает только в обычном потоке блочных элементов. Flex- и grid-контейнеры (следующие два раздела) margin своих детей не схлопывают вовсе — там расстояния между элементами устроены иначе.
</Callout>

## display меняет саму модель коробки

Слои content/padding/border/margin универсальны, но то, как коробка ведёт себя рядом с соседями, определяет свойство `display`.

```css
span { display: inline; }        /* по умолчанию у строчных тегов */
div  { display: block; }         /* по умолчанию у блочных тегов */
.badge { display: inline-block; } /* гибрид: течёт в строке, но принимает размеры */
```

Блочный элемент (`display: block`) всегда занимает всю доступную ширину родителя и начинается с новой строки; `width`, `height` и вертикальные `margin` работают на нём в полной мере. Строчный элемент (`display: inline`) течёт внутри текста, как обычное слово, но игнорирует `width`, `height` и вертикальные `margin` — они физически не могут повлиять на строку текста, в которую он встроен. `inline-block` — гибрид: элемент течёт в строке рядом с текстом и соседями, но при этом уважает `width`, `height` и все стороны `margin`, как блочный. У `img` display по умолчанию `inline`, однако размеры он всё равно принимает: это заменяемый (replaced) элемент, его габариты приходят вместе с самой картинкой. `inline-block` же обычно задают вручную кнопкам, бейджам и другим элементам, которым нужно такое же поведение.

## Частые ошибки

- **Задавать ширину и padding вместе без box-sizing: border-box.** Коробка становится шире расчётной, сетка «съезжает» на строку раньше ожидаемого.
- **Пытаться задать `width` строчному элементу.** `span { width: 200px; }` без смены `display` не сработает: у строчных элементов ширина определяется содержимым, а не свойством.
- **Удивляться «пропавшему» вертикальному отступу между блоками.** Это почти всегда схлопывание margin, а не ошибка в коде: расстояние равно большему из двух соседних значений, а не их сумме.
- **Забывать, что margin не красится фоном.** Если кажется, что «фон соседнего блока не доходит до края», — скорее всего, это прозрачный margin, а не проблема с background.

<Detail summary="Отрицательные и процентные margin">
Margin принимает отрицательные значения — элемент сдвигается в сторону соседей или перекрывает их, приём иногда используют для нарочитого наложения элементов друг на друга. Проценты в margin (как и в padding) вычисляются от ширины ближайшего блочного родителя — причём это верно даже для margin-top и margin-bottom, что часто удивляет: вертикальный отступ в процентах зависит от горизонтальной ширины контейнера, а не от его высоты.
</Detail>
