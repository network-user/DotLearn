# When Flexbox Stops Being Enough

Flexbox handles a row or a column beautifully, one direction at a time. But a full page layout, a header on top, a sidebar on the left, content on the right, a footer at the bottom, is already a grid of rows and columns at once, not a single line of items. You can try assembling a layout like that from nested flex containers, but every level of nesting adds its own layer of fragile width agreements. Grid exists for exactly this case: it lays out elements across rows and columns simultaneously, in one declaration on the parent.

## The Grid Is Defined on the Parent, Not the Children

```css
.page {
  display: grid;
  grid-template-columns: 200px 1fr;
  grid-template-rows: auto 1fr auto;
  gap: 16px;
}
```

`display: grid` turns on a grid on the container. `grid-template-columns` and `grid-template-rows` describe its structure: how many columns and rows, and how big each one is. In the example above there are two columns (a sidebar exactly 200px wide, everything else in the second column) and three rows (height based on content, all remaining space, height based on content again, a typical template for a header, main area, and footer). The container's children, direct descendants of `.page`, automatically fill the grid's cells in order, without a single line of CSS on the children themselves.

## The fr Unit: A Share of Free Space

`fr` (fraction) is a unit invented specifically for grid: it doesn't mean pixels or percent, it means a share of whatever free space is left after the fixed-width columns are accounted for.

```css
.layout { grid-template-columns: 1fr 2fr 1fr; }
```

The three columns split the space in a 1:2:1 ratio, the middle one gets exactly twice as much room as each of the outer ones, and the calculation doesn't need to know the container's exact pixel width, unlike percentages, which you'd have to recompute the moment you add any padding. `fr` mixes freely with fixed values, as in the `200px 1fr` example above: the browser first hands out space to the fixed columns, then splits whatever's left among the `fr` columns in proportion to their count.

## gap Instead of margin Hacks

Just like in flexbox, `gap` in grid sets the distance between cells in both directions at once, with no manual margins on each element and no compensating for the grid's edge items.

```css
.gallery {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: 12px;
}
```

`repeat(3, 1fr)` is shorthand for "three equal columns of 1fr each," equivalent to `1fr 1fr 1fr`; for a grid of ten identical columns, the convenience becomes a lot more obvious.

## Manual Placement: grid-column and grid-row

Grid children don't have to occupy exactly one cell in sequence, you can explicitly stretch them across several rows or columns using grid line numbers.

```css
.hero {
  grid-column: 1 / 3;
  grid-row: 1 / 2;
}
```

Grid lines are numbered starting from one, from the left (or top) edge; the notation `1 / 3` means "from line one to line three," meaning the element spans two columns. Placement like this is handy for one or two special elements in a layout (a wide banner above a card grid), but reading line numbers for an entire layout gets unwieldy fast, which is why grid offers a more visual tool for that.

## Named Areas: grid-template-areas

```css
.page {
  display: grid;
  grid-template-columns: 200px 1fr;
  grid-template-areas:
    "sidebar header"
    "sidebar main"
    "sidebar footer";
}
.sidebar { grid-area: sidebar; }
.header  { grid-area: header; }
.main    { grid-area: main; }
.footer  { grid-area: footer; }
```

`grid-template-areas` draws the layout literally in text: each quoted line is a row of the grid, and each word inside it is the name of the area that should fill that cell. The repeated name `sidebar` across three lines in a row tells the browser to stretch the sidebar across all three rows at once. Each element then gets its own name through `grid-area`, and the mapping to the CSS layout becomes so visual it reads almost like an ASCII sketch of the page itself, a rare thing in CSS, where you usually have to hold the layout in your head.

<Callout type="warning">
Every row in `grid-template-areas` must contain the same number of words, and the rectangle that one area name occupies has to be solid, no holes and no L-shapes. Otherwise the browser treats the whole declaration as invalid and ignores it entirely.
</Callout>

## Fluid Grids Without Media Queries: auto-fit and minmax

```css
.cards {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(220px, 1fr));
  gap: 16px;
}
```

This line is one of grid's most useful "recipes": a card grid that figures out on its own how many columns fit, without a single `@media` query. `minmax(220px, 1fr)` sets the bounds for each column: no narrower than 220px, no wider than its equal share. `auto-fit` makes the browser fit as many columns of that width as the container allows, stretching the last row if there end up being fewer columns than available space. Narrow the browser window and the column count drops automatically, cards staying no narrower than 220px; widen it and columns get added on their own.

<MarginNote label="auto-fit vs auto-fill">
The similar-looking `auto-fill` value differs subtly: it reserves space for empty columns whenever there aren't enough real elements to fill the full width, so existing cards don't stretch into the freed-up space. For an adaptive card grid, you almost always want `auto-fit`.
</MarginNote>

## Common Mistakes

- **Writing grid-template-areas with a different number of columns per row.** The whole rule becomes invalid, and the browser falls back to default grid behavior as if no areas were ever declared.
- **Confusing fr with percentages.** `fr` divides only the free space left after fixed columns and gaps are subtracted, not the container's entire width, in the grid `200px 1fr`, the second column doesn't get 100%, it gets "whatever's left after 200px."
- **Using grid where flexbox would do, or the other way around.** A single axis (a row of buttons, a menu) is a flexbox job; a grid of rows and columns at once (a page layout, a gallery) is a grid job. Mixing approaches isn't wrong by itself, but reaching for the wrong tool makes the code harder than it needs to be.
- **Forgetting that items with no explicit grid-area just fill cells in order.** If your HTML markup isn't in the order you expect visually, a grid with no explicit placement will lay things out differently than you intended.

<Detail summary="The implicit grid: grid-auto-rows and grid-auto-flow">
If there are more elements than cells in an explicitly defined grid, grid doesn't "break," it silently builds extra rows following the rules of the implicit grid. `grid-auto-rows` sets the height of those extra rows (it adapts to their content by default), and `grid-auto-flow: dense` makes the placement algorithm backfill earlier empty cells whenever a later item fits there, instead of leaving holes in the grid for the sake of strict order.
</Detail>
