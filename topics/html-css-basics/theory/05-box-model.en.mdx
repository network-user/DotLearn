# A 300-Pixel Width That Turned Into 342

The classic first encounter with CSS: you set a block to `width: 300px`, add a border and some padding for style, and a row of three such blocks suddenly stops fitting on one line, even though the math seemed to add up. It's not a browser bug. It's that "element width" in CSS never meant what it looks like at a glance, it means one specific layer out of the four nested ones that make up every box.

## Four Layers of One Box

The browser represents every element on a page as a rectangular box made of four concentric layers, nested outward from the content.

`content` is the area where text or nested elements get drawn; its dimensions are exactly what `width` and `height` set by default. `padding` is the inner gap between the content and the border, and it gets painted by the element's background just like the content does. `border` is the frame itself, whose thickness adds on top of the padding. `margin` is the outer gap from neighboring elements, and it's always transparent: an element's background never reaches into its margin, that's pure empty air between boxes.

```css
.card {
  width: 300px;
  padding: 20px;
  border: 1px solid gray;
  margin: 16px;
}
```

## Where Those Extra Pixels Came From

By default (`box-sizing: content-box`, the spec's own default), the `width` property sets the size of the `content` layer specifically, and padding and border get added on top of it, growing the box's total visible width. For the block above: 300px content plus 20px padding on the left plus 20px padding on the right plus 1px border on the left plus 1px border on the right equals 342px of total width, the space the block actually occupies on the page. Those "missing" 42 pixels from the introduction never went anywhere, they were always sitting outside the declared 300px, they just don't count toward it by default.

```css
.card {
  box-sizing: border-box;
  width: 300px; /* now this is the final width; border and padding get absorbed into it */
  padding: 20px;
  border: 1px solid gray;
}
```

`box-sizing: border-box` changes what unit `width` measures in the first place: now it means the box's final visible width including padding and border, and the browser shrinks the content area on its own to make everything fit. A width of 300px stays 300px no matter how thick the border or padding get. Because of this convenience, modern CSS resets almost always open with `*, *::before, *::after { box-sizing: border-box; }` at the very top of the stylesheet, to spare the whole project from this kind of arithmetic for good.

<MarginNote label="Inheritance">
box-sizing doesn't inherit under the browser's default styles, which is why a global reset targets the universal selector `*` rather than just `body` hoping inheritance will carry it down.
</MarginNote>

## Margins Can Collapse, Padding Never Does

There's an asymmetry that catches people off guard more than most: the vertical (top and bottom) margins of two adjacent block-level elements don't add together, they collapse into one, equal to the larger of the two.

```css
.article-title { margin-bottom: 40px; }
.article-lead   { margin-top: 20px; }
```

Between a heading and the paragraph that follows it, you'll find not 60px (40 plus 20) but exactly 40px, the larger of the two margins absorbs the smaller one. The rule applies only to margins, and only vertically in the normal document flow; padding never collapses, every pixel of it adds up with its neighbor honestly. Collapsing isn't a bug, it's a deliberate choice by CSS's authors: without it, nested paragraphs with identical `margin` values would create uneven, visually "jumpy" gaps wherever the author expected a consistent rhythm.

<Callout type="warning">
Collapsing only happens in the normal flow of block elements. Flex and grid containers (the next two sections) never collapse their children's margins at all, spacing between elements works differently there.
</Callout>

## display Changes the Box Model Itself

The content/padding/border/margin layers are universal, but `display` decides how a box behaves next to its neighbors.

```css
span { display: inline; }        /* the default for inline tags */
div  { display: block; }         /* the default for block tags */
.badge { display: inline-block; } /* a hybrid: flows in line, but accepts sizing */
```

A block element (`display: block`) always takes up its parent's full available width and starts on a new line; `width`, `height`, and vertical `margin` all work on it in full. An inline element (`display: inline`) flows inside text like an ordinary word, but ignores `width`, `height`, and vertical `margin`, they physically can't affect the line of text it's embedded in. `inline-block` is a hybrid: the element flows in line next to text and neighbors, but still respects `width`, `height`, and every side of `margin`, like a block element does. `img` defaults to `display: inline`, yet it still takes on real dimensions: it's a replaced element, its size comes bundled with the image itself. `inline-block` is usually set by hand on buttons, badges, and other elements that need that same behavior.

## Common Mistakes

- **Setting width and padding together without box-sizing: border-box.** The box ends up wider than planned, and a grid wraps to a new row sooner than expected.
- **Trying to set width on an inline element.** `span { width: 200px; }` does nothing without changing `display`: inline elements size themselves from their content, not from the `width` property.
- **Being puzzled by a "missing" vertical gap between blocks.** It's almost always margin collapsing, not a bug: the gap equals the larger of the two neighboring values, not their sum.
- **Forgetting that margin never gets painted.** If it looks like "the neighboring block's background doesn't reach the edge," it's probably just transparent margin, not a background issue.

<Detail summary="Negative and percentage margins">
Margin accepts negative values, which pulls an element toward its neighbors or lets it overlap them, a trick sometimes used to deliberately stack elements on top of each other. Percentages in margin (like in padding) are computed from the nearest block-level parent's width, and that holds true even for `margin-top` and `margin-bottom`, which often surprises people: a vertical percentage gap depends on the container's horizontal width, not its height.
</Detail>
