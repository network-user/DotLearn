# Every Page Starts With the Same Skeleton

Open any big e-commerce site and press Ctrl+U to view its page source. The browser dumps raw code: tens of thousands of lines mixing markup, styles, scripts, and analytics snippets. Finding anything meaningful in that stream feels hopeless. Now try the same on Wikipedia, on a government website, on a one-page student portfolio. The first lines always match: a `<!DOCTYPE html>` declaration, the root `html` tag, and inside it exactly two sections, `head` and `body`. Thousands of unrelated pages rest on the same skeleton.

This skeleton is what this section covers. By the end you'll be able to write a correct document shell from scratch, explain what each line does, and predict what breaks if a line gets removed or reordered. Along the way you'll also see why the browser silently forgives some mistakes while others turn readable text into garbled mojibake.

## The Browser Reads Top to Bottom and Builds a Tree

A file with an .html extension is just plain text. Until a browser opens it, no "page" exists yet. Once it receives bytes over the network, the browser runs them through a pipeline: decode bytes into characters, split the character stream into tokens, assemble nodes from tokens, and link them into a tree. That tree is called the DOM, the Document Object Model. Only after the tree is built does the browser compute layout and paint pixels.

The pipeline above leads to two practical conclusions. First: the order of lines in a document is not decoration, it's assembly instructions. The browser processes what it encounters the moment it encounters it and cannot look ahead. Second: a document is a tree by nature. Tags nest inside each other like folders inside folders, and each one must fully close inside its parent. Writing `<b><i>text</b></i>` breaks that nesting, and the browser has to repair it for you.

## Four Parts of the Skeleton

A minimal, correct document looks like this:

```html
<!DOCTYPE html>
<html lang="ru">
  <head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>Список покупок</title>
    <link rel="stylesheet" href="styles.css">
  </head>
  <body>
    <h1>Список покупок</h1>
    <p>Молоко, хлеб, сыр.</p>
  </body>
</html>
```

The roles are assigned strictly. The doctype declaration turns on the correct parsing mode. The `html` tag serves as the tree's root and carries the `lang` attribute. The `head` section holds housekeeping information: encoding, tab title, links to stylesheets. The `body` section holds everything the user sees and touches. You can't swap sections around, and the browser will silently insert any you omit, but the result may not match what you expected.

## Why You Need a Doctype

In the late nineties browsers interpreted CSS differently, and developers hacked their pages around the quirks of specific programs. By the time standards settled down, millions of old pages had already been built around those quirks, and the web couldn't just throw them away. Browsers ended up with two modes: a standards mode for new pages and a quirks mode for old ones, where the old oddities are deliberately reproduced. The most famous one concerns width: in quirks mode, borders and padding get subtracted from an element's declared width instead of added to it, and a tidy grid falls apart.

The doctype became the switch between modes. A page that starts with `<!DOCTYPE html>` is parsed by modern rules. A page without it gets a museum of nineties bugs. This line doesn't download anything, doesn't validate anything, and doesn't declare a language version: it's simply a flag that says "give me standards mode, please." That's why it has to be the very first line of every document you write.

## Inside head, Order Matters

The `head` section holds everything that describes the page without appearing on screen. Four residents show up in almost every document, and their order isn't arbitrary.

`<meta charset="utf-8">` comes first. Recall the pipeline: decoding bytes into characters sits at the very start of it, before tokens and before the tree. Until the browser meets the encoding declaration, it decodes by guesswork, relying on settings and heuristics. Plain English text is easy to guess correctly, but anything outside it, Cyrillic, Chinese, accented Latin, turns into familiar mojibake once read with the wrong table. The standard requires the declaration to land within the document's first 1024 bytes; the safest bet is making it the very first line inside `head`.

Second usually comes `<meta name="viewport" content="width=device-width, initial-scale=1">`. Without it, a mobile browser pretends the phone screen is 980 pixels wide, renders the page at that virtual width, and then shrinks the result down to fit the real screen. Text turns illegibly small, and the user has to pinch-zoom just to read it. The `width=device-width` value tells the browser to use the device's actual width.

<MarginNote label="History">
Those virtual 980 pixels were invented for the first iPhone in 2007: back then the entire web was built for wide monitors, and the phone had to display pages "from a distance."
</MarginNote>

The third resident, `<title>`, sets the tab title. It doubles as the bookmark name and the headline in search results, which is why "Untitled Document" looks especially awkward here.

Finally, `<link rel="stylesheet" href="styles.css">` loads the stylesheet. The link to your styles lives in `head` because the browser needs to know the styling rules before the first paint. If styles arrive later, the user gets a flash of unstyled content that then jumps and repaints.

## Scripts: defer vs. the End of body

The `script` tag historically behaved rudely: on encountering one, the browser stopped parsing, downloaded the file, ran it, and only then kept reading. A script at the top of `head` froze a blank page for the entire load. Hence the old trick: put scripts at the very end of `body` so the markup had time to build and render first.

The modern approach solves the same problem better:

```html
<head>
  <script defer src="app.js"></script>
</head>
```

The `defer` attribute changes behavior in two places: the file downloads in parallel with markup parsing, and it runs only after the DOM is built. Multiple deferred scripts execute in the order they were included. Compared to a script tucked at the end of `body`, the win is that the browser learns about the file from the first lines of the document and starts downloading right away, instead of waiting until the whole page has been parsed.

## Three Ways to Load CSS

Styles reach a page through three routes. The `style` attribute hangs declarations directly on a tag:

```html
<p style="color: crimson">Осталось две штуки</p>
```

A `style` tag inside `head` keeps rules inside the document itself:

```html
<head>
  <style>
    p { color: crimson; }
  </style>
</head>
```

The third route you've already seen in the skeleton: a `link` to an external file. In practice this one wins, and not out of taste. The browser downloads an external file once and caches it: moving to another page on the same site costs zero bytes for reloading styles. One file serves thousands of pages, and a single edit changes the whole site at once. The `style` attribute is left for one-off values computed by JavaScript, and the `style` tag comes in handy for a couple of rules on a page that doesn't deserve its own file, an HTML email, for instance.

## The Browser Silently Fixes Your HTML

Now for an honest surprise. The HTML parser never crashes with an error, no matter what: recovery rules for every possible mistake are written into the standard. Look at this fragment:

```html
<p>Молоко
<p>Хлеб
<table>
  <tr><td>Сыр</td></tr>
</table>
```

Not a single `p` is closed, yet the page "works": on meeting a new `p`, the parser closes the previous one for you. The table is even more interesting: in the DOM, the `tr` row ends up wrapped in a `tbody` that never appears in the source at all. Open a file like this in DevTools and compare the tree to the source: they don't match.

Because of this helpfulness, "it works" doesn't mean "it's correct." The selector `table > tr` will match nothing, because the browser inserted a `tbody` between `table` and `tr`. A script that expects one structure gets a different one. Auto-repair removes the symptom but hides the disease, which is why markup deserves a run through a validator instead of a judgment based on how it looks in the browser.

The skeleton's second quiet participant, the `lang` attribute on the `html` tag, doesn't touch a single pixel, and people forget about it. That's a mistake: a screen reader picks its speech synthesizer language from it, the browser decides whether to offer a translation, and hyphenation algorithms break words according to the right language's rules. A Russian page marked `lang="en"` sounds, to a screen reader, like Russian text read aloud by an English robot.

## Common Mistakes

- **Forgetting meta viewport.** The page looks fine on a monitor and microscopic on a phone: the browser rendered it at a virtual 980 pixels and shrank it down. Check your layout in DevTools' mobile mode from day one.
- **Declaring the encoding late, or not at all.** Text turns into mojibake for some users, while "it works on my machine" holds true for others: the outcome depends on the browser's settings. Keep `<meta charset="utf-8">` as the first line of `head`.
- **Putting a block element inside a p.** A paragraph can't contain a `div`: the parser closes the `p` right before it, and the tree drifts apart from your CSS. Only inline elements like `a`, `strong`, or `span` belong inside a paragraph.
- **Placing a script without defer in head.** Parsing stalls until the script downloads and runs, and the user stares at a blank screen. Add defer, or explain out loud why this particular script doesn't need it.

<Detail summary="What else quirks mode changes">
Besides the box-width formula, quirks mode changes dozens of small things: tables stop inheriting font from their parent, vertical alignment of images inside cells works differently, height percentages get interpreted differently. The full list of differences fills separate documents maintained by browser teams. The practical takeaway is just one: you don't need to memorize any of these quirks, you just need to always start a document with a doctype. You can check the current mode from the console: `document.compatMode` returns CSS1Compat in standards mode and BackCompat in quirks mode.
</Detail>
