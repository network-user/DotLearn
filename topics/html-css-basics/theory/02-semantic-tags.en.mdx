# Why the Browser Needs to Know It's Looking at a Menu, Not a Paragraph

Open the markup of any old student HTML project and you'll see a familiar picture: the whole page is nothing but nested `div`s. The header is a `div`, the menu is a `div` with more `div`s inside, the footer is another `div`. That page will work: the browser will honestly paint pixels exactly where you told it to. But ask it "where's the main site menu?" and there's nobody to answer. To the browser, every `div` looks the same, an anonymous box with no meaning.

Semantic tags solve exactly this problem. They look and behave like a `div` (a full-width block element, nothing changes visually) but carry the name of a role: this is the header, this is the navigation, this is the main content. By the end of this section you'll place them as confidently as you once placed `div`s, and you'll understand who actually needs that name.

## The Page Skeleton Gets Names

Take a typical site structure and swap the wrappers for semantic tags:

```html
<body>
  <header>
    <h1>Магазин «Утро»</h1>
    <nav>
      <a href="/catalog">Каталог</a>
      <a href="/cart">Корзина</a>
    </nav>
  </header>

  <main>
    <article>
      <h2>Кофемашина Delta 200</h2>
      <p>Компактная модель для дома...</p>
    </article>
  </main>

  <aside>
    <h2>Похожие товары</h2>
  </aside>

  <footer>
    <p>© 2026 Магазин «Утро»</p>
  </footer>
</body>
```

Each tag owns its own patch of the page: `header` introduces a section or the page as a whole, `nav` marks a block of navigation links, `main` is the page's single main content area, `aside` holds material that isn't essential to understanding the main content (ads, "related reading" links), and `footer` closes things out. Visually nothing changed: all of them are block-level by default, just like `div`. What changed is meaning, and it's now available not just to you reading the code, but to programs too.

## Who Actually Needs the Names, If the Pixels Look the Same

The first audience is assistive technology. A blind user listens to a page through a screen reader instead of looking at it. Screen-reading software can build a list of page "landmarks" from semantic tags: the user presses a key and jumps straight to `nav`, skipping the header and ads that a sighted user's eyes skip in a fraction of a second anyway. Markup made only of `div`s can't build that list: as far as the program is concerned, there's nothing there but one long page of text it has to read start to finish.

The second audience is search crawlers. When indexing a page, a search engine reads the semantics to figure out that the content inside `article` and `main` is probably the whole reason the page exists, while `aside` and `footer` are just supporting material. The third audience is the developer, six months from now: `grep "<main"` finds the page's content entry point in a second, `grep "<div"` finds nothing useful at all.

<MarginNote label="Try it">
Browser extensions like an "accessibility inspector" show a page's landmark tree. Open one on a div-only site and the tree comes back empty, even if the site looks flawless.
</MarginNote>

## article vs section: A Content Unit and a Thematic Block

These two tags get confused the most, because both "slice" a page into meaningful chunks, just by different rules. `article` is a self-contained piece that makes sense outside the page it lives on: a blog post, a product card, a comment. The test is simple: if you could rip this chunk out of the page and publish it on its own (in an RSS feed, say) without losing meaning, it's an `article`.

`section` is a thematic grouping of content that usually needs a heading and doesn't claim to stand on its own. The "Reviews" block on a product page is a `section`: it's meaningless detached from the product it belongs to.

```html
<article>
  <h2>Кофемашина Delta 200</h2>
  <p>Компактная модель...</p>

  <section>
    <h3>Отзывы покупателей</h3>
    <p>«Пользуюсь два месяца, довольна» — Анна</p>
  </section>
</article>
```

The whole product card is a self-contained unit fit for its own page or an RSS feed, so it's an `article`. The reviews only make sense next to the product they're about, so they're a `section`, not a second `article`.

## Headings Build a Document Outline, Not a Font Size

The six heading levels, `h1` through `h6`, often get used as a ready-made font-size scale: need bigger text, grab an `h2`. That's using the tool for the wrong job. Headings form a document outline, the same kind you'd see in a Word document's table of contents: `h1` is the page title, `h2` marks the major sections inside it, `h3` marks subsections inside an `h2`, and so on. A screen reader builds navigation from exactly this outline and announces the level out loud: "heading level two, Customer Reviews."

Skip a level (jump straight from `h1` to `h4` because you liked the size), and the outline breaks: the program can no longer tell whether something is a subsection or not. Text size has nothing to do with any of this, you change that through the CSS `font-size` property, not by picking a different tag number.

## aside and figure: Side Content and Captioned Media

`aside` marks material related to the main content but not required to understand it: a callout with an interesting fact, a "related reading" block, an ad widget on the side. A screen reader announces it as "complementary content," and the user can skip the whole block with one command without losing the main thread.

```html
<figure>
  <img src="delta-200.jpg" alt="Кофемашина Delta 200 на кухонном столе">
  <figcaption>Кофемашина Delta 200, вид спереди</figcaption>
</figure>
```

`figure` bundles a piece of media (an image, a chart, a code sample) together with its `figcaption`; the connection between them is explicit and programmatic, not "the caption is just the paragraph sitting below it, hopefully readers will guess."

## When div and span Are Still the Right Choice

Semantics doesn't retire `div` entirely. Sometimes you need a wrapper purely for styling or a JS hook, and no tag fits the meaning, a grid container for product cards, say, which doesn't "mean" anything on its own. In that case `div` is an honest choice, not laziness. Same story with `span`: the inline version of `div`, useful for wrapping a chunk of text for styling or an attribute when no inline semantic tag (`strong`, `em`, `time`, `code`) fits the meaning.

<Callout type="warning">
Don't turn `nav` into a tag for "any list of links." Footer links to "Terms of Use" and "Privacy Policy" aren't a site's primary navigation; a plain `footer` with a list of links is fine without `nav`. Save `nav` for genuinely significant navigation blocks: the main menu, breadcrumbs, an article's table of contents.
</Callout>

## Common Mistakes

- **`div` instead of `main`.** A screen reader can't offer a "skip to main content" command, so the user has to sit through the header and menu again on every single page of the site.
- **More than one `main` on a page.** The tag is meant to be singular: it's the one main content area, not "another big block."
- **Skipping heading levels for the sake of font size.** The document outline breaks even though the text itself looks fine. Change size through CSS instead.
- **A `section` with no heading.** The spec expects a section to have its own heading; a section without one is usually just a `div` in disguise.

<Detail summary="The ARIA roles tags assign implicitly">
Every semantic tag carries a built-in (implicit) ARIA role: `nav` is equivalent to `role="navigation"`, `main` to `role="main"`, `aside` to `role="complementary"`, and a top-level `footer` to `role="contentinfo"`. You don't need to write these roles by hand, the tag already provides them. ARIA rule number one puts it plainly: "No ARIA is better than Bad ARIA." Don't override a role that a semantic tag already assigns correctly.
</Detail>
