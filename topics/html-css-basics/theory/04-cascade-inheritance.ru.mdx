# Почему CSS вообще так называется

Буква «C» в CSS означает Cascading — каскадный. Название не для красоты: оно описывает алгоритм, которым браузер разрешает конфликты, когда несколько правил претендуют на одно и то же свойство одного и того же элемента. В прошлом разделе вы разобрали одну ступень этого алгоритма, специфичность. Но специфичность — только вторая ступень из трёх, и правило, побеждающее по специфичности, всё равно может проиграть на более ранней ступени. К концу раздела вы будете видеть всю цепочку целиком, а не только её середину.

## Три ступени каскада по порядку

Столкнувшись с конфликтом, браузер проверяет критерии строго по очереди и останавливается на первом, который различает соперников.

Первая ступень, происхождение и важность, сравнивает не селекторы, а источник правила. Стили браузера по умолчанию (user-agent) стоят ниже стилей вашего сайта (author), поэтому обычная ссылка синяя лишь до тех пор, пока вы не напишете свой цвет. Внутри одного происхождения объявление с `!important` перепрыгивает выше обычных объявлений того же источника. Только если после этой ступени остались равные кандидаты, в дело идёт специфичность, разобранная в прошлом разделе. И только если равна специфичность, решает порядок в коде: правило, встретившееся позже в разобранных браузером стилях, побеждает правило, встретившееся раньше.

```css
p { color: navy; }      /* специфичность 0·0·1 */
p { color: seagreen; }  /* специфичность та же, но ниже в файле */
```

Оба правила бьют по одному и тому же свойству `color` одного тега `p` с одинаковой специфичностью 0·0·1. Ступень «происхождение» их не различает — оба из вашего файла, оба обычные. Ступень «специфичность» тоже не различает — тождественна. Решает порядок: `seagreen` написан позже, значит именно он красит текст.

## Наследование: не все свойства передаются детям

Отдельно от каскада (который решает, чьё правило применить к элементу) существует наследование — механизм, которым значение свойства переходит от родителя к потомку, если для потомка не нашлось собственного правила вовсе. Наследуются не все свойства, а выборочно, и в основе выбора здравый смысл авторов CSS.

```html
<div style="color: darkslategray; font-family: sans-serif; border: 1px solid red;">
  <p>Текст абзаца.</p>
</div>
```

Абзац внутри `div` получит тёмно-серый цвет текста и тот же шрифт, хотя ни одно правило не целится в `p` напрямую — `color` и `font-family` наследуются. Красную рамку абзац не получит: `border` не наследуется, иначе каждый вложенный элемент в разметке обрастал бы рамками своих предков, что почти никогда не нужно.

Общее правило, которое избавляет от заучивания списков: свойства текста наследуются (`color`, `font-*`, `line-height`, `text-align`, `letter-spacing`), свойства коробки и раскладки — нет (`margin`, `padding`, `border`, `width`, `background`, `display`). Логика простая: текстовое оформление разумно течёт вниз по документу целиком, а вот геометрию каждый элемент почти всегда должен считать заново, иначе рамка одного блока рекурсивно расползалась бы на весь его внутренний контент.

## Явное управление: inherit, initial, unset, revert

Четыре ключевых слова позволяют управлять наследованием и каскадом там, где поведение по умолчанию не подходит.

```css
.card a {
  color: inherit;   /* взять текущее значение color у родителя, даже если свойство обычно не наследуется */
}
.reset-box {
  border: initial;  /* вернуть значение по умолчанию из спецификации CSS (обычно none) */
}
.widget * {
  all: unset;       /* обнулить все свойства: unset = inherit там, где наследуется, initial — где нет */
}
```

`inherit` заставляет свойство унаследовать значение родителя, даже если по умолчанию оно этого не делает — частый приём для ссылок внутри цветных карточек, которым нужен не стандартный синий, а цвет текста карточки. `initial` откатывает к значению по умолчанию из спецификации CSS, которое не всегда совпадает со стилями браузера. `unset` действует как `inherit` для наследуемых свойств и как `initial` для ненаследуемых — удобно для «сбросить и не думать, какое это свойство». `revert` откатывает к стилю предыдущего источника в каскаде (обычно к стилям браузера), полезен, когда сброс общего файла стилей случайно погасил разметку системных элементов вроде `button`.

## !important — аварийный тормоз, а не инструмент на каждый день

`!important` перепрыгивает специфичность и порядок кода целиком: объявление с этим флагом побеждает почти любое обычное объявление того же происхождения, независимо от того, насколько «слабый» у него селектор.

```css
.button { color: blue !important; }
#header .button.primary.large { color: red; } /* всё равно проиграет */
```

Несмотря на внушительный селектор, второе правило проиграет первому исключительно из-за флага. Возникает соблазн чинить любой «непослушный» стиль через `!important`, но у приёма есть цена: следующий разработчик, которому понадобится перебить именно это правило, вынужден добавить свой `!important`, и конфликты специфичности превращаются в конфликты важности, где побеждает либо порядок кода, либо чистая случайность. Легитимные случаи редки: точечный override стилей сторонней библиотеки, до селекторов которой вы не можете дотянуться, или utility-класс вроде `.hidden { display: none !important; }`, которому по замыслу положено перебивать всё.

<Callout type="warning">
Стиль в атрибуте `style="..."` формально имеет специфичность выше любого обычного селектора, но проигрывает `!important` из подключённого файла. Единственное, что бьёт `!important` из файла — свой `!important` в самом атрибуте `style`, что уже граница здравого смысла для повседневной вёрстки.
</Callout>

## Частые ошибки

- **Путать каскад со специфичностью.** Специфичность — только один шаг каскада; правило с более «тяжёлым» селектором всё равно проиграет, если конкурент отмечен `!important` или пришёл из более позднего происхождения.
- **Ожидать наследования от геометрии.** `width: 300px` на родителе не передаётся детям; каждый блок по умолчанию считает свою ширину исходя из ширины родителя, а не наследует конкретное число.
- **Сбрасывать через `all: unset` без понимания побочных эффектов.** Свойство обнуляет вообще всё, включая `display`, из-за чего элемент может визуально исчезнуть из раскладки.
- **Разгонять войну `!important`.** Каждое новое исключение получает свой `!important`, пока весь файл стилей не превращается в список аварийных перебивок без внятного порядка.

<Detail summary="Слои каскада: @layer">
Современный CSS добавил ступень до специфичности — именованные слои `@layer`. Правила из более позднего объявленного слоя бьют правила из более раннего слоя целиком, независимо от специфичности внутри них; так библиотека компонентов может объявить свой слой ниже слоя утилит проекта, и обычный класс `.text-center` без единого лишнего id уверенно перебивает стили библиотеки. Слои решают именно ту проблему, ради которой раньше злоупотребляли `!important`.
</Detail>
