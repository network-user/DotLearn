# Why CSS Is Called "Cascading" in the First Place

The letter "C" in CSS stands for Cascading. The name isn't decorative, it describes the algorithm the browser uses to resolve conflicts when several rules claim the same property on the same element. In the previous section you covered one stage of that algorithm, specificity. But specificity is only the second of three stages, and a rule that wins on specificity can still lose at an earlier stage. By the end of this section you'll see the whole chain, not just its middle.

## The Cascade's Three Stages, in Order

Faced with a conflict, the browser checks criteria strictly in sequence and stops at the first one that tells the contenders apart.

The first stage, origin and importance, compares not selectors but where a rule came from. Default browser styles (user-agent) rank below your site's own styles (author), which is why a plain link stays blue only until you write your own color for it. Within a single origin, a declaration marked `!important` jumps above ordinary declarations from that same source. Only if candidates are still tied after this stage does specificity, covered in the last section, come into play. And only if specificity ties too does source order decide: a rule that appears later in the browser's parsed styles beats one that appeared earlier.

```css
p { color: navy; }      /* specificity 0·0·1 */
p { color: seagreen; }  /* same specificity, but lower in the file */
```

Both rules target the same `color` property on the same `p` tag with identical specificity, 0·0·1. The origin stage doesn't distinguish them, both come from your file, both are ordinary. The specificity stage doesn't distinguish them either, it's identical. Source order decides: `seagreen` is written later, so that's what colors the text.

## Inheritance: Not Every Property Passes to Children

Separate from the cascade (which decides whose rule applies to an element), there's inheritance, the mechanism by which a property's value passes from parent to child whenever the child has no rule of its own at all. Not every property inherits, only a select few do, and the selection follows the plain common sense of CSS's authors.

```html
<div style="color: darkslategray; font-family: sans-serif; border: 1px solid red;">
  <p>Текст абзаца.</p>
</div>
```

The paragraph inside the `div` picks up the dark slate gray text color and the same font, even though no rule targets `p` directly: `color` and `font-family` inherit. It won't pick up the red border: `border` doesn't inherit, otherwise every nested element in the markup would grow its own copy of every ancestor's border, which is almost never what anyone wants.

Here's the general rule that saves you from memorizing lists: text properties inherit (`color`, `font-*`, `line-height`, `text-align`, `letter-spacing`), box and layout properties don't (`margin`, `padding`, `border`, `width`, `background`, `display`). The logic is simple: text styling reasonably flows down through an entire document, while geometry almost always has to be recalculated by each element on its own, otherwise one block's border would recursively spread across all of its inner content.

## Explicit Control: inherit, initial, unset, revert

Four keywords let you steer inheritance and the cascade wherever the default behavior doesn't fit.

```css
.card a {
  color: inherit;   /* take the parent's current color value, even though the property doesn't normally inherit */
}
.reset-box {
  border: initial;  /* revert to the CSS specification's default value (usually none) */
}
.widget * {
  all: unset;       /* zero out every property: unset = inherit where it inherits, initial where it doesn't */
}
```

`inherit` forces a property to take its parent's value, even if it doesn't normally inherit, a common trick for links inside colored cards that need the card's text color instead of the standard link blue. `initial` rolls a property back to the CSS specification's default, which doesn't always match the browser's own default styling. `unset` behaves like `inherit` for inheritable properties and like `initial` for non-inheritable ones, handy for "reset this and don't think about which kind of property it is." `revert` rolls back to the previous origin's style in the cascade (usually the browser default), useful when a blanket stylesheet reset accidentally strips a native element like `button` of its usual look.

## !important Is an Emergency Brake, Not a Daily Tool

`!important` leapfrogs specificity and source order entirely: a declaration carrying this flag beats nearly any ordinary declaration from the same origin, no matter how "weak" its own selector is.

```css
.button { color: blue !important; }
#header .button.primary.large { color: red; } /* still loses */
```

Despite its imposing selector, the second rule still loses to the first, purely because of the flag. It's tempting to fix every "misbehaving" style with `!important`, but the trick has a cost: the next developer who needs to override this exact rule has to add their own `!important`, and specificity conflicts turn into importance conflicts, where the winner comes down to source order or plain luck. Legitimate uses are rare: a targeted override of a third-party library's styles whose selectors you can't reach, or a utility class like `.hidden { display: none !important; }` that's designed to beat everything by nature.

<Callout type="warning">
A style in the `style="..."` attribute formally has higher specificity than any ordinary selector, but it still loses to `!important` from a linked stylesheet. The only thing that beats a file's `!important` is another `!important` right inside that `style` attribute, which is already well past the point of common sense for everyday layout work.
</Callout>

## Common Mistakes

- **Confusing the cascade with specificity.** Specificity is only one step of the cascade; a rule with a "heavier" selector still loses if a competitor carries `!important` or comes from a later origin.
- **Expecting geometry to inherit.** `width: 300px` on a parent doesn't pass to its children; every block computes its own width from its parent's width by default instead of inheriting a fixed number.
- **Resetting with `all: unset` without understanding the side effects.** The property zeroes out absolutely everything, including `display`, which can make an element vanish from the layout entirely.
- **Escalating an `!important` war.** Every new exception gets its own `!important`, until the whole stylesheet turns into a list of emergency overrides with no coherent order left.

<Detail summary="Cascade layers: @layer">
Modern CSS added a stage ahead of specificity: named `@layer` layers. Rules from a layer declared later beat rules from an earlier layer entirely, regardless of specificity within them; a component library can declare its layer below a project's utility layer, so an ordinary class like `.text-center`, with not a single extra id, confidently overrides the library's styles. Layers solve exactly the problem that used to get papered over with `!important`.
</Detail>
