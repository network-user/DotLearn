# Who Overrode Your Style: Selectors and the Three Counters

You wrote a rule for a button, saved the file, refreshed the page. The button didn't change. You open DevTools, find the element, and there it is: your property is listed, but crossed out, and one line above sits someone else's rule from a completely different corner of the project. The style was written, applied, and immediately beaten by someone you never even invited.

The winner isn't whichever rule was written last, and it isn't whichever one is "more precise" in spirit. The browser settles these conflicts with arithmetic: every selector has a computable weight, its specificity, and the property goes to the rule with the highest weight. The good news: you can compute that weight in your head in a couple of seconds. By the end of this article you'll be able to name the winner before you even open DevTools, and you'll understand why a single class outweighs eleven tags put together.

## A Selector Answers the Question "Who"

A CSS rule has two parts. Inside the curly braces sit the declarations: what exactly to change. To their left sits the selector: which elements to apply it to. There aren't many basic kinds of selectors, and you've almost certainly met most of them already.

```css
h1 { font-size: 32px; }
.card { border-radius: 12px; }
#menu { position: sticky; }
[disabled] { opacity: 0.5; }
a:hover { text-decoration: underline; }
p::first-line { font-variant: small-caps; }
```

A tag selector picks every element of one type: a rule for `h1` will color every level-one heading on the page. A class selector starts with a dot and catches elements that carry that word in their `class` attribute; the same class can sit on a hundred elements, and one element can carry several classes, which is why classes remain the main workhorse of styling. The `#menu` selector finds the element with `id="menu"`; an id must be unique on the page, so a selector like this always targets exactly one element. Square brackets check an attribute: `[disabled]` catches every disabled form element regardless of tag, while `[href^="/docs"]` catches only links whose address starts with `/docs`.

Two kinds of "pseudo" live on their own. A pseudo-class is written with a single colon and describes a state of a real element: `:hover` under the cursor, `:focus` when keyboard-focused, `:first-child` as the first among its parent's children. A pseudo-element is written with two colons and draws something that isn't actually in the markup: `::before` inserts a virtual node before the content, and `::first-line` wraps a paragraph's first line, wherever it happens to end at the current window width.

## Combinators Chain Selectors Together

Precision shows up once simple selectors get chained together. There are four connectors, and two of them trip up beginners more than the rest.

```css
nav a { color: inherit; }
nav > a { padding: 8px 12px; }
h2 + p { margin-top: 4px; }
h2 ~ p { color: gray; }
```

A space is the descendant combinator: `nav a` finds a link anywhere inside `nav`, no matter how deep, even five levels of nested lists down. The `>` arrow is stricter, the child combinator: `nav > a` only picks links that sit directly inside `nav`, with no wrapper in between. Plus picks the very next sibling: `h2 + p` matches exactly one paragraph, the one sitting right after the heading. Tilde is broader: `h2 ~ p` picks every paragraph after the heading that shares its parent.

<MarginNote label="How the browser reads it">The browser matches a selector right to left: for `nav a` it first grabs every link and only then checks whether a `nav` sits above it. The rightmost part of a selector is always the important one.</MarginNote>

A comma isn't a combinator at all, it's grouping: `h1, h2, h3` applies one set of declarations to three independent selectors. Each member of the group lives its own life, and its weight is computed separately.

## Three Counters, Not a Sum of Points

Now for the main event. When two rules both claim the same property on the same element, the browser compares the specificity of their selectors. Here's how it's computed: walk through the selector and fill three counters. First: how many id selectors it has. Second: how many classes, attributes, and pseudo-classes. Third: how many tags and pseudo-elements. You end up with a triple in the form a·b·c, and that's not a single number, it's a tuple of three independent digits.

<PullQuote>Specificity isn't a sum of points, it's a tuple: the leading digit decides everything.</PullQuote>

The lower counters only come into play when the leading ones tie. And if all three tie completely, specificity is powerless, and the next cascade rule kicks in: whichever declaration sits later in the code wins. The next concept is devoted entirely to the cascade; for now it's enough to remember the order of checks.

## Four Practice Examples

Try estimating each selector's weight yourself, then check your work against the breakdown below.

```css
ul li a.external { color: seagreen; }
#header .logo { height: 40px; }
a[target="_blank"]:hover { outline: 1px dotted; }
main article > p::first-line { font-weight: 600; }
```

First: three tags and one class, so 0·1·3. Second: one id and one class, 1·1·0. Third: the `a` tag, the `[target="_blank"]` attribute, and the `:hover` pseudo-class, together 0·2·1. Fourth: three tags and the `::first-line` pseudo-element, 0·0·4; the `>` combinator doesn't count toward anything. Now rank them by descending strength: 1·1·0 leads, followed by 0·2·1, then 0·1·3, with 0·0·4 trailing. Notice that the selector with the longest text turned out weakest, because every one of its parts landed in the lowest digit.

## What Weighs Zero, and What You Can't Beat

The universal selector `*` and every combinator weigh exactly zero: the chain `* > *` is still 0·0·0, and `nav > a` weighs exactly as much as `nav a`. A combinator changes which elements get matched, not the selector's weight.

The `:not()` pseudo-class is trickier. It doesn't add anything by itself, but its argument counts as usual: `p:not(.intro)` weighs 0·1·1, a tag plus a class inside the parentheses. Forget that, and it's easy to be surprised when a "negating" selector suddenly outweighs a plain neighbor.

The section's most useful surprise: digits never overflow into each other. A selector made of eleven tags weighs 0·0·11 and still loses to a single class at 0·1·0. No matter how much you stack up in the lowest counter, none of it carries over into a higher one. Which means "outmuscling" a class by chaining on more tags is impossible in principle: the selector grows longer while the higher digits stay put.

<Callout type="warning">
A style written in the `style` attribute doesn't even enter the selector competition: an inline declaration beats any specificity. The only thing that can override it from a stylesheet is `!important`. That's one reason inline styles are considered bad practice: they switch off CSS's normal mechanisms.
</Callout>

## Common Mistakes

- **Fixing a losing selector with `!important`.** It works instantly, which is exactly why it's dangerous: the next exception will need its own `!important`, and rules start shouting over each other. The right move: check DevTools to see who won, then either weaken the winner or bump your own weight by the smallest step, adding one class.
- **Id selectors in stylesheets.** A weight of 1·0·0 can't be outweighed by any number of classes, and every exception to "styled by id" needs an even heavier selector. That's how a specificity war starts, one where the code only gets heavier. Leave ids to anchors and JavaScript, and style with classes.
- **Confusing `nav a` with `nav > a`.** A space doesn't look like an operator, so the difference slips by. If the links in a menu are wrapped in `li`, the selector `nav > a` matches nothing, and the rule silently does nothing. Read a space as "somewhere inside" and an arrow as "directly inside."

<Detail summary=":is(), :where(), and the weight of modern selectors">
The functional selectors `:is()` and `:where()` group alternatives: `:is(article, aside) p` is shorter than two comma-separated selectors. They compute weight differently. `:is()` takes the specificity of its heaviest argument, even if the actual match was the lightest one. `:where()` always evaluates to zero, which makes it handy for library base styles: rules like that are easy to override with a single class, without starting a specificity war. Modern CSS resets rely on that same zero-weight trick.
</Detail>
