# Rebase replays your commits on a new base

You worked a week on branch `feature` while `main` moved forward — others merged their changes there. Your branch now "lags behind the base". Merge fixes that with a merge commit, but history becomes branchy. Rebase offers another path: take your commits and replay them as if you had branched from *fresh* `main`. The result is straight, linear history where `feature` work sits neatly on top of current `main`.

By the end of this section you will understand how rebase differs from merge in outcome, when linear history is worth it, and why there is an iron rule: do not rebase what you already shared.

## Merge keeps the fork; rebase straightens it

Both merge and rebase reconcile diverging branches. The difference is what history they leave.

Merge creates a merge commit with two parents: the fork and meeting point stay visible forever. Rebase **rewrites** your commits: it finds the common ancestor, takes commits on `feature` after it, and reapplies them one by one on top of `main`'s tip. Old commits are replaced with new ones — same contents, different parents and identifiers.

The key word is "replay". Rebase does not move commits like files; it takes *changes* from each commit and applies them anew on the new base, creating new commits. After rebase your commits have different identifiers: technically they are different commits, though with the same contents.

## Performing a rebase

Stand on the branch you want to move and say what to rebase *onto*.

```bash
git switch feature
git rebase main
```

Git replays commits from `feature` on top of `main`'s tip. If edits do not overlap, it passes silently and `feature` becomes a direct continuation of `main`. After that you can merge `main` into `feature` or the other way — merge will be fast-forward because history is already linear.

Try it in the terminal. `feature` and `main` diverged: each has its own commit after the common ancestor. Stand on `feature` (`git switch feature`) and run `git rebase main`. The `feature` commit moves to `main`'s tip and history becomes three commits in one line. The check goal is three commits with both files present.

## When behavior surprises you

Rebase can conflict like merge — but conflict is resolved for *each replayed commit in turn*. If three commits on `feature` touch disputed lines, Git may stop three times. At each stop you fix the file, `git add`, and continue with `git rebase --continue`. Lost — `git rebase --abort` returns the branch to before rebase, losing nothing. That is normal rebase rhythm, not a sign something broke.

<Callout type="info">On rebase conflict Git stops at each problematic commit. Resolve one at a time: edit → git add → git rebase --continue. Changed your mind — git rebase --abort returns everything as before.</Callout>

## The golden rule: do not rebase published work

Rebase rewrites commits — creates new ones instead of old. While commits live only with you, that is harmless. But if you already pushed them and others built on them, rebase turns them into "different" commits while old ones remain with colleagues. Histories diverge and sync becomes a mess of duplicates.

Hence the golden rule: **rebase only commits you have not shared yet**. Your local branch before first push — fine. A shared branch others watch — no, use merge there. Same logic that separated `reset` (rewrites history) from `revert` (does not) in [[undoing-changes]].

<Callout type="warning">Do not rebase commits already pushed to a shared remote and visible to others. Rebase replaces them with new ones and others' repositories desync. For published history use merge.</Callout>

## Common mistakes

- **Rebasing a shared branch.** Rebasing `main` the team uses — everyone's histories "split". Rebase is for local, not-yet-shared work.
- **Thinking rebase is "safer" than merge.** It is not safer, it is *different*. Merge honestly documents the fork; rebase trades linearity for rewriting. Choice is team history style ([[workflows]]).
- **Abandoning rebase mid-conflict.** An incomplete rebase leaves the branch in limbo. Finish with `--continue` or cancel explicitly with `--abort`.

<Detail summary="Why linear history matters to some teams">

Linear history (without merge commits) reads as a simple list: "did A, then B, then C". Tools like `git bisect` hunting the commit that introduced a bug run easier on it, and review is simpler — each commit applies to a predictable state. The cost is losing information that B and C were actually done in parallel with something else. Teams valuing a readable line rebase features before merge; teams valuing an exact chronicle keep merge commits. Both positions are defensible — a decision about what matters in your project, not right vs wrong.

</Detail>
