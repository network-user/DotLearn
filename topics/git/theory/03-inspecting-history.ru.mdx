# Три команды-фонарика: что изменилось, что было, в чём разница

Контроль версий полезен ровно настолько, насколько Вы умеете в нём *смотреть*. Сломалось что-то - надо понять, когда оно работало. Перед коммитом - проверить, что именно уйдёт в снимок. После недели работы - вспомнить, что вообще происходило. Для всех трёх задач есть свои фонарики.

К концу раздела Вы будете уверенно читать `git status` (что в каком состоянии прямо сейчас), `git log` (какая история накопилась) и `git diff` (чем версии отличаются построчно), и понимать, почему `git diff` без аргументов показывает не то, что Вы ожидали.

## status: моментальный снимок «здесь и сейчас»

`git status` отвечает на вопрос «в каком состоянии мой рабочий каталог прямо сейчас». Он не показывает историю - только текущую расстановку файлов по трём областям, которые Вы уже знаете: что выставлено на сцену, что изменено, но не выставлено, что вообще не отслеживается.

```bash
git status
```

Этой командой стоит начинать и заканчивать почти любое действие. Перед коммитом - убедиться, что на сцене ровно то, что нужно. После - проверить, что дерево чистое. Запутались в середине rebase или merge - `git status` подскажет, что Git ждёт от Вас дальше.

## log: лента истории

`git log` разворачивает историю - список коммитов от самого свежего к самому старому, с автором, датой и сообщением. Полная форма подробна; на практике чаще нужен сжатый вид.

```bash
git log --oneline
```

Флаг `--oneline` печатает по строке на коммит: короткий хеш и сообщение. Так за секунду видно форму истории - сколько коммитов, какие сообщения, где Вы сейчас.

## diff: построчная разница

`git status` говорит, *какие* файлы изменились, но не *как*. На вопрос «как именно» отвечает `git diff` - он показывает построчные различия. И вот здесь кроется главная тонкость: без аргументов `git diff` сравнивает **рабочее дерево с индексом**, то есть показывает только то, что Вы ещё *не* выставили на сцену.

```bash
git diff
```

Если Вы уже сделали `git add`, то изменения ушли на сцену, и `git diff` без аргументов покажет пусто - хотя файл явно менялся. Это сбивает с толку каждого новичка. Чтобы увидеть то, что *уже* на сцене (и войдёт в коммит), нужен флаг `--staged`.

```bash
git diff --staged
```

Логика стройная, если держать в голове три области: `git diff` смотрит «дерево против индекса», а `git diff --staged` - «индекс против последнего коммита». Вместе они показывают всю разницу между «что на диске» и «что в истории».

## Когда поведение удивит

Самая частая ловушка уже названа: «я изменил файл, а `git diff` пустой». Почти всегда причина в том, что Вы успели сделать `git add`, и правка переехала на сцену. `git diff` показывает разницу «дерево против индекса», а раз дерево и индекс совпали - разницы нет. Запомните пару: для несценированного - `git diff`, для сценированного - `git diff --staged`.

<Callout type="warning">Пустой `git diff` после правки обычно означает, что Вы уже сделали `git add`. Смотрите сценированные изменения через `git diff --staged`.</Callout>

## Типичные ошибки

- **Считать, что `git log` показывает изменения.** `git log` показывает список коммитов с сообщениями, а не их содержимое. Чтобы увидеть, что менялось в коммите, нужен diff (например, `git log -p` или `git show`).
- **Искать свежие правки в `git diff` после `git add`.** Они на сцене - смотрите `git diff --staged`. Без `--staged` Вы видите лишь то, что ещё *не* добавлено.
- **Полагаться на `git status` как на историю.** `status` - это «здесь и сейчас». Что было вчера, покажет только `git log`.

<Detail summary="--oneline и другие способы посмотреть форму истории">

`git log --oneline` - самый быстрый обзор. Чтобы увидеть ветвление, добавляют `--graph`: Git рисует ASCII-граф слияний и расхождений слева от сообщений. `git log --stat` показывает, сколько строк в каких файлах поменял каждый коммит, не вываливая полный diff. А `git show <commit>` разворачивает один конкретный коммит вместе с его построчными изменениями. Эти формы - разные линзы на одну и ту же цепочку снимков.

</Detail>
