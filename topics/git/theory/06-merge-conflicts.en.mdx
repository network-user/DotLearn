# A conflict is a question from Git, not a breakdown

The word "conflict" sounds alarming, but it is not an error and not data damage. A conflict is Git honestly saying: "two branches changed the same lines differently, and I do not know which version you want. You decide." Git stops and passes the choice to a human, because the machine has no right to decide.

By the end of this section you will understand *why* a conflict arises, read the conflict markers Git inserts into a file, and walk the full resolution cycle: edit, add to index, commit.

## Why a conflict arises at all

Recall three-way merge: Git compares the common ancestor with both tips. For each file it looks at who changed which lines. If only one side touched a line, the choice is obvious. If both sides changed **the same line differently**, there is no obvious answer. That is a conflict.

Remember the boundary: conflict is always *overlap of incompatible edits to one line*. Different files, different lines in one file, and the same edit from both sides: Git merges all of that without you. Conflict is a narrow case, but it requires a human.

## Conflict markers

When `git merge` hits a conflict, Git does not crash. It stops the merge, leaves it incomplete, and **writes both versions into the conflicted file**, separated by markers:

```text
<<<<<<< HEAD
port=3000
=======
port=9090
>>>>>>> feature
```

Read them like this. Between `<<<<<<< HEAD` and `=======` is the version from your current branch ("ours"). Between `=======` and `>>>>>>> feature` is the version from the branch being merged ("theirs"). Git is saying: "here are two variants of this line, keep the one you need". Your job is to turn this block into one correct version, **removing the markers** and the extra variant.

## Full resolution cycle

Conflict resolution is always the same three steps, whatever the conflict looks like.

<Steps>
<li><b>Edit the file.</b> Open the conflicted file, find the markers, choose the correct content (one version, both, or a completely new variant) and delete all three marker lines.</li>
<li><b>Add to the index.</b> <code>git add file</code> on the resolved file signals Git "conflict settled, here is the final version".</li>
<li><b>Commit.</b> <code>git commit</code> completes the merge with a merge commit. Git supplies a message like "Merge branch..."; you can keep or refine it.</li>
</Steps>

In the live terminal below two branches changed a line in `config.txt` differently. First stand on `main` and run `git merge feature`: Git reports a conflict and writes markers (view contents with `cat config.txt`). Now resolve: overwrite the file with the final value, for example `echo port=8080 > config.txt`, then `git add config.txt` and `git commit -m "resolve"`. The graph shows the completed merge.

## When behavior surprises you

An incomplete merge is a state you cannot simply walk away from as if nothing happened. Until the conflict is resolved and committed, the repository hangs mid-merge. Two common surprises. First: committing without removing markers saves them into the code, so `<<<<<<< HEAD` rides into history as ordinary text. Always verify no markers remain. Second: if you changed your mind about merging, `git merge --abort` cancels the merge and returns to the state before it.

<Callout type="warning">Before committing a resolved conflict make sure the file contains no lines with `<<<<<<<`, `=======`, or `>>>>>>>`. Otherwise markers enter history as ordinary text.</Callout>

## Common mistakes

- **Leaving markers in the file.** The most common trouble: you picked a version but forgot to erase `=======` and neighboring lines. Code then fails to compile and the diff shows mysterious "arrows". Delete all three markers.
- **Committing without adding the file.** After editing the file is still marked conflicted until you `git add`. That is the "settled" signal. Without it `git commit` complains about unresolved conflict.
- **Panicking and recreating the repository.** Conflict is normal. If you are lost, `git merge --abort` returns everything as before the merge; no commits are lost.

<Detail summary="Why ours and theirs depend on where you stand">

In the markers `HEAD` is always the current branch (the one where you started merge), and `feature` (or another name) is the incoming branch. If you stood on `feature` and merged `main`, the sides would swap: `<<<<<<< HEAD` would point at `feature`'s version. So "ours/theirs" is not "correct/incorrect" but "current/incoming". With rebase, by the way, these roles invert relative to intuition because rebase replays your commits on top of others, which is the topic of [[rebase]].

</Detail>
