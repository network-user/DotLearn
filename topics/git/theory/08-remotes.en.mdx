# A remote repository is a copy of history on another machine

Until now everything you did lived on one machine. But a team works together, code lives on a server (GitHub, GitLab), and laptops sometimes break. A remote repository is a full copy of history somewhere else that your local repository exchanges commits with. Git is distributed: every participant has full history, and a remote is the shared sync point.

By the end of this section you will understand what a remote and `origin` are, distinguish three exchange operations — `fetch`, `pull`, `push` — and know why `fetch` is safer than `pull` and why `push` is sometimes rejected.

## origin — the default name for the "main" remote

A remote is simply a named address of another repository. `git remote add <name> <url>` creates such an entry: from then on you reach the remote by short name, not a long URL.

```bash
git remote add origin https://github.com/user/project.git
```

By convention the main remote is called `origin` — not magic, just habit Git uses in hints. When you clone someone else's repository, Git automatically creates remote `origin` pointing where you cloned from.

## Three exchange operations: fetch, pull, push

Between local repository and remote there are exactly three directions of work — do not confuse them.

**`git fetch`** downloads commits from the remote but **does not touch your working branches**. It updates so-called remote-tracking branches (`origin/main` and similar) — local snapshots of what is on the server. After `fetch` you can see what is new, compare with your work, and only then decide whether to merge. The safest operation: it changes nothing in your working tree.

```bash
git fetch origin
```

**`git pull`** is `fetch` plus immediate merge: download and pour remote changes into your current branch. Convenient but less predictable: merge happens at once, and if it conflicts you land in conflict resolution before you looked around. Essentially `git pull` = `git fetch` + `git merge origin/<branch>`.

```bash
git pull origin main
```

**`git push`** sends your local commits to the remote, updating the branch there. That is how you share work with the team.

```bash
git push origin main
```

Try it in the terminal. The repository has one commit. Add remote `origin` with `git remote add origin https://example.com/repo.git`, then send branch `main`: `git push origin main`. The check goal is the commit in place and the tree still clean.

## When behavior surprises you

The most common surprise is a rejected `push`. If someone sent commits to the remote that you do not have yet, Git refuses your `push`: otherwise others' work would be overwritten. The message sounds like "updates were rejected because the remote contains work that you do not have locally". The fix — fetch others' commits first (`git pull` or `git fetch` + merge), reconcile, then `push`. Git deliberately blocks overwriting history you do not own.

<Callout type="warning">If push is rejected with "updates were rejected", the remote has commits you lack. Fetch them (pull/fetch), reconcile, then push again. Do not use push --force to "force through" — that erases others' work.</Callout>

## Common mistakes

- **Treating `git pull` as harmless.** `pull` immediately merges others' changes into your branch and can pull you into conflict without warning. When you want to inspect what arrived first, `git fetch` then deliberate merge is safer.
- **`push --force` to bypass rejection.** Force-push overwrites commits on the remote that you lack — others' work. Use it rarely and only on personal branches, knowing the consequences.
- **Confusing remote name and branch name.** In `git push origin main` the first word is the remote name (`origin`), the second is the branch (`main`). Swapping them is a typical "remote not found" cause.

<Detail summary="Remote-tracking branches: origin/main is not main">

After `git fetch` Git updates branches like `origin/main` — called remote-tracking — representing "state of branch main on origin at last fetch". That is not the same as your local `main`: you do not edit `origin/main` directly; it only reflects the server. When you `git pull`, Git first updates `origin/main` via fetch, then merges it into your local `main`. Understanding this pair — local branch vs its remote-tracking snapshot — clears most sync confusion: you always see how far your branch is ahead or behind the server.

</Detail>
