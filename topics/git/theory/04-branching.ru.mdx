# Ветка - это движущийся указатель на коммит

Вы пишете новую фичу, и на середине прилетает срочный баг в продакшене. Менять незаконченный код страшно - всё перепутается. Хочется отложить фичу в сторону, спокойно починить баг в чистой версии, а потом вернуться. Ровно для этого нужны ветки: они дают параллельные линии работы в одном репозитории, между которыми Вы переключаетесь мгновенно.

К концу раздела Вы поймёте, почему ветка в Git - это не копия файлов, а всего лишь дешёвый указатель; что такое `HEAD` и как он движется; и сможете создавать ветки, переключаться между ними и понимать, где именно Вы сейчас находитесь.

## Ветка - это ярлык, а не папка

В тяжёлых системах контроля версий «создать ветку» означало скопировать весь проект в новый каталог - дорого и медленно. В Git ветка - это просто **указатель** (имя, ссылающееся на один коммит). Создать ветку - значит записать одно имя; это мгновенно и почти ничего не весит.

Каждый коммит знает своего родителя, поэтому история - связанный список снимков. Ветка лишь помечает один коммит в этом списке как «вершину линии работы». Когда Вы делаете новый коммит на ветке, указатель ветки автоматически сдвигается вперёд, на свежий коммит.

## HEAD - указатель «где я сейчас»

Раз веток несколько, Git нужно знать, на какой из них Вы стоите. За это отвечает `HEAD` - специальный указатель «текущая позиция». Обычно `HEAD` указывает на ветку (например, на `main`), а ветка - на коммит. Поэтому, когда Вы коммитите, Git двигает ту ветку, на которую смотрит `HEAD`.

Создать ветку - команда `git branch <имя>`. Она лишь ставит новый указатель на текущий коммит, но **не** переключается на него: `HEAD` остаётся там же.

```bash
git branch feature
```

Чтобы перейти на ветку, есть две команды. Старая универсальная - `git checkout <имя>`. Новая, более узкая и понятная, - `git switch <имя>`: она занимается только переключением веток и не делает десяти других вещей, как `checkout`.

```bash
git switch feature
```

Часто нужно создать ветку и сразу на неё перейти. Для этого есть короткие формы: `git switch -c <имя>` или `git checkout -b <имя>` - обе создают ветку и переводят на неё `HEAD` одной командой.

```bash
git switch -c hotfix
```

После этого `HEAD` указывает на `hotfix`, и новые коммиты пойдут именно туда, не трогая `main`.

## Переключение наводит рабочее дерево на снимок ветки

Когда Вы переключаетесь на ветку, Git подменяет содержимое рабочего дерева снимком того коммита, на который ветка указывает. Файлы на диске меняются так, будто Вы вернулись в тот момент истории. Поэтому работа на разных ветках действительно изолирована: коммиты в `feature` не видны на `main`, пока Вы их не сольёте.

Потрогайте это в терминале. Создайте ветку `feature` командой `git switch -c feature`, добавьте файл и закоммитьте. Затем `git switch main` и `git status` - нового файла нет, он остался на `feature`. Вернитесь на `feature` - файл снова на месте. Цель встроенной проверки - оказаться на ветке `feature` с двумя коммитами в истории.

## Когда поведение удивит

`git branch feature` создаёт ветку, но `HEAD` не двигает - Вы остаётесь там, где были. Новичок ожидает, что окажется «внутри» новой ветки, делает коммит и удивляется, что он ушёл в `main`. Разделение намеренно: `branch` создаёт, `switch`/`checkout` переключает. Если хотите и то и другое разом - используйте `git switch -c`.

<Callout type="warning">`git branch feature` только создаёт указатель, но оставляет Вас на текущей ветке. Чтобы перейти на новую ветку сразу, нужен `git switch -c feature` (или `git checkout -b feature`).</Callout>

Второй сюрприз - детачнутый HEAD. Если перейти не на ветку, а прямо на коммит (`git checkout <хеш>`), `HEAD` начнёт указывать на коммит напрямую, минуя ветку. Это состояние «detached HEAD»: коммиты, сделанные здесь, не принадлежат ни одной ветке и легко теряются. Подробнее об этом - в разделе [[tags-reflog]].

## Типичные ошибки

- **Думать, что `git branch` переключает.** Он только создаёт указатель. Коммит после `git branch feature` (без перехода) уйдёт в старую ветку.
- **Делать коммиты в detached HEAD по незнанию.** Если `git status` пишет «HEAD detached», Вы не на ветке. Прежде чем коммитить, создайте ветку: `git switch -c <имя>`.
- **Путать `git switch` и `git switch -c`.** Без `-c` Git ждёт *существующую* ветку и ругается, если её нет. С `-c` он создаёт новую. Перепутали - получите «invalid reference».

<Detail summary="Почему git switch появился рядом с git checkout">

Команда `git checkout` исторически перегружена: она и переключает ветки, и восстанавливает файлы, и детачит HEAD на коммит. Из-за этого одна опечатка могла вместо переключения ветки затереть Ваши правки в файле. В 2019 году (Git 2.23) команду разделили: `git switch` отвечает только за переход между ветками, `git restore` - только за восстановление файлов. `checkout` остался для совместимости, но для новых привычек чище использовать узкие команды - меньше шансов случайно сделать не то.

</Detail>
