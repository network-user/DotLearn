# Three flashlight commands: what changed, what happened, what is the difference

Version control is only as useful as your ability to *look* inside it. Something broke, and you need to know when it worked. Before a commit, check exactly what will enter the snapshot. After a week of work, recall what happened at all. Three tasks, three flashlights.

By the end of this section you will confidently read `git status` (what is in which state right now), `git log` (what history has accumulated), and `git diff` (how versions differ line by line), and understand why `git diff` with no arguments shows something other than what you expected.

## status: an instant snapshot of "here and now"

`git status` answers "what state is my working directory in right now". It does not show history, only the current placement of files across the three areas you already know: what is on stage, what is modified but not staged, what is not tracked at all.

```bash
git status
```

Start and end almost any action with this command. Before committing, make sure exactly what you need is on stage. Afterward, check that the tree is clean. If you get lost in the middle of a rebase or merge, `git status` hints what Git expects from you next.

## log: the history tape

`git log` unfolds history: a list of commits from newest to oldest, with author, date, and message. The full form is verbose; in practice you often want a compact view.

```bash
git log --oneline
```

The `--oneline` flag prints one line per commit: short hash and message. In a second you see the shape of history: how many commits, which messages, where you are now.

History runs right to left in time, but `git log` prints top to bottom from fresh to old: first comes `HEAD`, the commit you are on, then its parent, and so on to the first commit, which has no parent.

## diff: line-by-line difference

`git status` says *which* files changed, not *how*. For "how exactly" use `git diff`, which shows line-by-line differences. Here lies the main subtlety: with no arguments `git diff` compares **the working tree with the index**, that is, only what you have *not* yet placed on stage.

```bash
git diff
```

If you already ran `git add`, the changes moved to stage, so `git diff` with no arguments shows empty even though the file clearly changed. That confuses every beginner. To see what is *already* on stage (and will enter the commit), use the `--staged` flag.

```bash
git diff --staged
```

The logic is clean if you keep three areas in mind: `git diff` looks at "tree vs index", and `git diff --staged` looks at "index vs last commit". Together they show the full gap between "what is on disk" and "what is in history".

Experiment in the terminal. The repository has one commit and a modified file. Run `git diff` to see the edit. Then run `git add notes.txt` and `git diff` again: now it comes back empty. Finally run `git diff --staged`, and the edit reappears. Build history from several commits and look at `git log --oneline`.

## When behavior surprises you

The most common trap is already named: "I changed a file but `git diff` is empty". Almost always you had time to run `git add`, and the edit moved to stage. `git diff` shows the difference "tree vs index", and once tree and index match there is no difference. Remember the pair: for unstaged changes use `git diff`, for staged changes use `git diff --staged`.

<Callout type="warning">An empty `git diff` after an edit usually means you already ran `git add`. View staged changes with `git diff --staged`.</Callout>

## Common mistakes

- **Thinking `git log` shows changes.** `git log` shows a list of commits with messages, not their contents. To see what changed in a commit, use diff (for example `git log -p` or `git show`).
- **Looking for fresh edits in `git diff` after `git add`.** They are on stage: use `git diff --staged`. Without `--staged` you only see what is *not* yet added.
- **Relying on `git status` as history.** `status` is "here and now". What happened yesterday is shown only by `git log`.

<Detail summary="--oneline and other ways to see the shape of history">

`git log --oneline` is the fastest overview. To see branching, add `--graph`: Git draws an ASCII graph of merges and divergences to the left of messages. `git log --stat` shows how many lines changed in which files per commit without dumping the full diff. And `git show <commit>` unfolds one specific commit together with its line-by-line changes. These forms are different lenses on the same chain of snapshots.

</Detail>
