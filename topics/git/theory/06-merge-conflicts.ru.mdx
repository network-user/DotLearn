# Конфликт - это вопрос Git к Вам, а не поломка

Слово «конфликт» звучит тревожно, но это не ошибка и не повреждение данных. Конфликт - честное признание Git: «две ветки изменили одни и те же строки по-разному, и я не знаю, какую версию Вы хотите. Решите Вы». Git останавливается и передаёт выбор человеку - там, где машина решать не вправе.

К концу раздела Вы будете понимать, *почему* возникает конфликт, читать маркеры конфликта, которые Git вставляет в файл, и проходить полный цикл разрешения: отредактировать, добавить в индекс, закоммитить.

## Почему конфликт вообще возникает

Вспомните трёхстороннее слияние: Git сравнивает общего предка с двумя вершинами. Для каждого файла он смотрит, кто менял какие строки. Если одну строку тронула только одна сторона - выбор очевиден. Если обе стороны изменили **одну и ту же строку по-разному** - очевидного ответа нет. Вот это и есть конфликт.

Запомните границу: конфликт - это всегда *пересечение несовместимых правок одной строки*. Разные файлы, разные строки одного файла, одинаковая правка с обеих сторон - всё это Git сольёт без Вашего участия. Конфликт - узкий случай, но именно он требует человека.

## Маркеры конфликта

Когда `git merge` натыкается на конфликт, Git не падает. Он останавливает слияние, оставляет merge незавершённым и **вписывает в конфликтный файл обе версии**, разделив их маркерами:

```text
<<<<<<< HEAD
port=3000
=======
port=9090
>>>>>>> feature
```

Читаются они так. Между `<<<<<<< HEAD` и `=======` - версия Вашей текущей ветки («наша»). Между `=======` и `>>>>>>> feature` - версия вливаемой ветки («их»). Git как бы говорит: «вот два варианта этой строки, оставь тот, что нужен». Ваша задача - превратить этот блок в одну правильную версию, **удалив маркеры** и лишний вариант.

## Полный цикл разрешения

Разрешение конфликта - всегда одни и те же три шага, как бы ни выглядел сам конфликт.

<Steps>
<li><b>Отредактируйте файл.</b> Откройте конфликтный файл, найдите маркеры, выберите правильное содержимое (одну из версий, обе сразу или совершенно новый вариант) и удалите все три строки-маркера.</li>
<li><b>Добавьте в индекс.</b> <code>git add file</code> по разрешённому файлу - это сигнал Git «конфликт улажен, вот итоговая версия».</li>
<li><b>Закоммитьте.</b> <code>git commit</code> завершает слияние merge-коммитом. Git сам подставит сообщение вроде «Merge branch...», его можно оставить или уточнить.</li>
</Steps>

В живом терминале ниже две ветки изменили строку в `config.txt` по-разному. Сначала встаньте на `main` и выполните `git merge feature` - Git сообщит о конфликте и впишет маркеры (посмотрите содержимое через `cat config.txt`). Теперь разрешите: перезапишите файл итоговым значением, например `echo port=8080 > config.txt`, затем `git add config.txt` и `git commit -m "resolve"`. Граф покажет завершённое слияние.

## Когда поведение удивит

Незавершённый merge - это состояние, из которого нельзя просто «уйти», сделав вид, что ничего не было. Пока конфликт не разрешён и не закоммичен, репозиторий висит в середине слияния. Два частых сюрприза. Первый: попытка закоммитить, не убрав маркеры, сохранит их прямо в коде - и `<<<<<<< HEAD` поедет в историю как обычный текст. Всегда проверяйте, что маркеров не осталось. Второй: если Вы передумали сливать вовсе, merge можно отменить и вернуться в исходное состояние командой `git merge --abort`.

<Callout type="warning">Перед коммитом разрешённого конфликта убедитесь, что в файле не осталось ни одной строки с `<<<<<<<`, `=======` или `>>>>>>>`. Иначе маркеры попадут в историю как обычный текст.</Callout>

## Типичные ошибки

- **Оставить маркеры в файле.** Самая частая беда: выбрали версию, но забыли стереть `=======` и соседние строки. Код потом не компилируется, а в diff видны загадочные «стрелочки». Удаляйте все три маркера.
- **Закоммитить, не добавив файл.** После правки файл всё ещё помечен как конфликтный, пока Вы не сделаете `git add`. Это и есть сигнал «улажено». Без него `git commit` будет ругаться на неразрешённый конфликт.
- **Паниковать и пересоздавать репозиторий.** Конфликт - штатная ситуация. Если запутались, `git merge --abort` вернёт всё как было до слияния, и можно попробовать заново, никакие коммиты при этом не теряются.

<Detail summary="Почему «наша» и «их» зависят от того, где Вы стоите">

В маркерах `HEAD` - это всегда текущая ветка (та, на которой Вы запустили merge), а `feature` (или другое имя) - вливаемая. Если бы Вы стояли на `feature` и сливали `main`, стороны поменялись бы местами: `<<<<<<< HEAD` указывал бы на версию `feature`. Поэтому «наша/их» - не про «правильную/неправильную», а про «текущую/входящую». При rebase, к слову, эти роли инвертированы относительно интуиции, потому что rebase переигрывает Ваши коммиты поверх чужих, - но это уже тема раздела [[rebase]].

</Detail>
