# The team chooses rules, not commands

You know individual commands: branch, commit, merge, rebase, push. But in a five-person team another question arises, and it is not "how to merge" but "when we branch, how we name branches, how we merge into main, who reviews and when". That is a workflow: agreements that turn parallel-work chaos into a predictable flow. The commands are the same; several established schools differ in how to apply them.

By the end of this section you will understand four common approaches (feature branches, pull requests, gitflow, and trunk-based) and choose consciously for team size and release cadence.

## Feature branches: the basic brick

The most fundamental agreement: **nobody commits directly to `main`**. Each task gets a new branch off `main`. You work in isolation, `main` stays stable, and you merge the finished feature back.

```bash
git switch -c feature/login-form
```

Branch names usually carry meaning: `feature/...`, `fix/...`, `chore/...`. When work is ready the branch merges into `main` (via merge or rebase) and is deleted. This is not a separate "process" but the foundation for all others: pull requests, gitflow, and trunk-based are different layers on "work in branches, protect main".

## Pull requests: merge through review

On top of feature branches teams almost always run **pull requests** (GitHub) or merge requests (GitLab). Not a Git command but a platform process: you push a branch, open a request "please merge my branch into main", and before that happens colleagues *review*: read changes, comment, run tests. Only after approval does the branch merge.

A pull request turns merge from a private act into a team event: code gets another pair of eyes, discussion stays in history, and `main` is protected from direct pushes. For most teams this is the de facto standard because review catches errors before they hit the shared branch.

## Gitflow and trunk-based: two poles

Two schools on opposite sides of the spectrum, disagreeing on how many long-lived branches to keep.

**Gitflow** was popular in the era of rare numbered releases: its `release` and `hotfix` branches fit "ship version 2.3, support 2.2". For web services deploying to production several times a day that structure is heavy.

**Trunk-based** answers continuous delivery: the shorter a branch lives, the less it diverges from `main` and the rarer painful big merges. The price is a strong automated test culture and feature flags (so unfinished work can merge but stay hidden from users). Most modern product teams lean toward this pole.

Try the basic feature-branch cycle by hand in the terminal: create branch `feature/note`, add a file, commit, return to `main` and merge. That skeleton is what other processes hang review and naming rules on.

## When behavior surprises you

Beginners often seek the "one correct" workflow. It does not exist. Gitflow is not "obsolete", just ill-suited to continuous delivery; trunk-based is not "better" but needs infrastructure a small team may lack. Process choice is a tradeoff between structure and speed for release cadence, team size, and test maturity. A good process is one the team actually follows, not the trendiest.

<Callout type="info">There is no single correct workflow. Gitflow fits versioned releases; trunk-based fits continuous delivery. A good process is one the team follows consistently.</Callout>

## Common mistakes

- **Committing directly to `main` on a team project.** That breaks any workflow: no review, `main` destabilizes. Open a branch per task.
- **Keeping a branch alive for weeks.** The longer a branch stays unmerged, the more it diverges from `main` and the more painful the final merge. Split work, merge more often.
- **Adopting gitflow on a tiny project "for respectability".** Five branches for two developers is overhead without benefit. Process complexity should match team complexity.

<Detail summary="Where process truth lives: agreements, not Git">

Git does not know or impose your workflow: to Git `develop`, `feature/x`, and `main` are the same kind of branch, and pull request does not exist at Git level (it is a GitHub/GitLab layer). The whole process rests on team agreements and tools around Git: branch protection on the platform, CI running tests on every pull request, name templates. So two projects on the same Git can have completely different processes. Studying someone else's repo, look not only at commands but at `CONTRIBUTING.md` and branch settings, where the real rules live.

</Detail>
