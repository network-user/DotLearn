# Индекс - это сцена, на которую Вы выкладываете будущий коммит

Вы уже знаете, что коммит - снимок проиндексированного состояния. Теперь разберёмся с самой частой точкой непонимания: зачем между «я поправил файл» и «я сохранил это в истории» стоит индекс, и как им пользоваться так, чтобы каждый коммит был аккуратной, законченной единицей работы.

К концу раздела Вы сможете уверенно стадировать (выкладывать в индекс) отдельные файлы, читать `git status` как карту состояния, собирать один коммит из части изменений, исправлять промах командой `git restore --staged` и понимать, почему `git add` и `git commit` - это два разных шага, а не один.

## Staging как репетиция коммита

Английское staging area буквально - «сцена». Представьте, что коммит - это спектакль, который Вы вот-вот покажете истории. Индекс - сцена, на которую Вы заранее выставляете ровно тех актёров (изменения файлов), что должны выйти на этот спектакль. Пока идёт репетиция, на сцену можно добавлять и убирать актёров. `git commit` - это поднятие занавеса: всё, что стоит на сцене, фиксируется как единый снимок.

Ключевое следствие: на сцену попадает не файл, а **состояние файла на момент `git add`**. Поэтому два шага - `add` и `commit` - разделены намеренно. Между ними Вы можете передумать, посмотреть, что собрали, добавить ещё или убрать лишнее.

## git status - карта трёх состояний

`git status` показывает, в какой из областей сейчас находятся Ваши изменения. Это первая команда, которую стоит набирать всякий раз, когда непонятно, что происходит.

```bash
git status
```

Вывод делит файлы на группы. «Changes to be committed» - то, что уже на сцене (в индексе) и войдёт в коммит. «Changes not staged for commit» - изменённые файлы, которых на сцене ещё нет. «Untracked files» - новые файлы, про которые Git вообще ничего не знает, пока Вы их не добавите. Один и тот же файл может оказаться сразу в двух группах: часть его изменений выставлена, часть - нет.

```bash
git add app.py
git status
git commit -m "добавил обработчик"
```

После `git add app.py` файл переезжает в «Changes to be committed». После `git commit` рабочее дерево становится чистым: `git status` сообщит «nothing to commit, working tree clean».

## Собираем коммит из части изменений

Сила индекса в том, что Вы выбираете состав коммита. Допустим, за вечер Вы поправили два файла: `parser.py` (новая фича) и `README.md` (мелкая опечатка). Логически это две независимые правки - значит, и два коммита.

```bash
git add parser.py
git commit -m "поддержка вложенных тегов"
git add README.md
git commit -m "исправил опечатку в примере"
```

Первый `commit` запечатал только `parser.py`, потому что только он стоял на сцене. `README.md` остался изменённым в рабочем дереве и попал во второй коммит. История читается как осмысленный список решений, а не «правки за вторник».

Потрогайте это в терминале. В репозитории два изменённых файла. Выложите на сцену только `feature.py`, закоммитьте его, проверьте `git status` - увидите, что `notes.txt` всё ещё ждёт. Затем закоммитьте и его. Попробуйте намеренно `git add .` вместо одного файла и сравните, как изменится состав первого коммита.

## Когда поведение удивит

Вы выставили файл на сцену, а потом передумали - хотите убрать его из коммита, но сохранить правки. Интуиция подсказывает `git rm`, но это удалит файл. Правильный инструмент - `git restore --staged`: он снимает файл со сцены (возвращает индекс к версии из последнего коммита), не трогая Ваши изменения в рабочем дереве.

```bash
git add secret.txt
git restore --staged secret.txt
```

После этого `secret.txt` снова «not staged»: правки на месте, но в следующий коммит файл не пойдёт, пока Вы его опять не добавите. Это полная противоположность `git rm`, который убирает файл и из индекса, и с диска.

<Callout type="warning">`git restore --staged file` снимает файл со сцены, сохраняя правки. `git rm file` удаляет файл из индекса и из рабочего дерева. Не путайте: первое отменяет стадирование, второе уничтожает файл.</Callout>

## Типичные ошибки

- **Считать `git add` и `git commit` одной командой.** Это два шага: `add` выкладывает на сцену, `commit` поднимает занавес. Сделать `git commit` сразу после правки нового файла - значит получить «nothing to commit», потому что на сцене пусто.
- **Коммитить пустым или бессмысленным сообщением.** `git commit -m "fix"` через месяц не скажет Вам ничего. Сообщение - это объяснение «зачем», а не «что» (что видно из diff). Пишите «исправил падение при пустом вводе», а не «fix».
- **Забыть про новые файлы при `git add` по имени.** Если стадируете файлы поимённо, легко забыть только что созданный. `git status` перед коммитом покажет всё, что осталось в «untracked».

<Detail summary="git commit -am: соблазн и его граница">

Флаг `git commit -am "сообщение"` объединяет два шага: стадирует изменения и сразу коммитит. Удобно, но с важной оговоркой - он берёт только **уже отслеживаемые** файлы (те, что были в прошлом коммите). Новый, ещё не добавленный файл `-a` проигнорирует: его всё равно нужно сначала `git add`. Поэтому `-am` хорош для быстрой правки существующих файлов, но не заменяет осознанного стадирования, когда коммит надо собрать из части изменений.

</Detail>
