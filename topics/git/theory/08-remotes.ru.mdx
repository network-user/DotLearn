# Удалённый репозиторий - это копия истории на другой машине

Пока всё, что Вы делали, жило на одной машине. Но команда работает вместе, код хранят на сервере (GitHub, GitLab), а ноутбук иногда ломается. Удалённый репозиторий (remote) - это полноценная копия истории где-то ещё, с которой Ваш локальный репозиторий обменивается коммитами. Git распределённый: у каждого участника полная история, а remote - общая точка синхронизации.

К концу раздела Вы будете понимать, что такое remote и `origin`, различать три операции обмена - `fetch`, `pull`, `push` - и знать, почему `fetch` безопаснее `pull`, а `push` иногда отвергается.

## origin - имя по умолчанию для «главного» удалённого

Remote - это просто именованный адрес другого репозитория. Команда `git remote add <имя> <url>` заводит такую запись: с этого момента к удалённому репозиторию можно обращаться по короткому имени, а не по длинному URL.

```bash
git remote add origin https://github.com/user/project.git
```

По соглашению основной remote называют `origin` - это не магическое слово, а просто привычное имя, которое Git подставляет в подсказках. Когда Вы клонируете чужой репозиторий, Git автоматически заводит remote `origin`, указывающий туда, откуда Вы клонировали.

## Три операции обмена: fetch, pull, push

Между локальным репозиторием и remote есть ровно три направления работы, и важно их не путать.

**`git fetch`** скачивает коммиты с remote, но **не трогает Ваши рабочие ветки**. Он обновляет так называемые remote-tracking ветки (`origin/main` и подобные) - локальные слепки того, что лежит на сервере. После `fetch` Вы можете посмотреть, что нового, сравнить со своей работой и только потом решить, сливать ли. Это самая безопасная операция: она ничего не меняет в Вашем рабочем дереве.

```bash
git fetch origin
```

**`git pull`** - это `fetch` плюс немедленное слияние: скачать и сразу влить изменения с remote в Вашу текущую ветку. Удобно, но менее предсказуемо: слияние происходит сразу, и если оно конфликтует, Вы оказываетесь в разрешении конфликта, не успев осмотреться. По сути `git pull` = `git fetch` + `git merge origin/<ветка>`.

```bash
git pull origin main
```

**`git push`** отправляет Ваши локальные коммиты на remote, обновляя там ветку. Это способ поделиться работой с командой.

```bash
git push origin main
```

Попробуйте в терминале. В репозитории один коммит. Добавьте remote `origin` командой `git remote add origin https://example.com/repo.git`, затем отправьте ветку `main`: `git push origin main`. Цель проверки - чтобы коммит был на месте, а дерево осталось чистым.

## Когда поведение удивит

Самый частый сюрприз - отвергнутый `push`. Если кто-то отправил на remote коммиты, которых у Вас ещё нет, Git откажется принимать Ваш `push`: иначе чужая работа была бы затёрта. Сообщение звучит как «updates were rejected because the remote contains work that you do not have locally». Лекарство - сначала забрать чужие коммиты (`git pull` или `git fetch` + слияние), уладить расхождение, и только потом `push`. Git намеренно не даёт перезаписать историю, которой Вы не владеете.

<Callout type="warning">Если push отвергнут с «updates were rejected», на remote есть коммиты, которых у Вас нет. Сначала заберите их (pull/fetch), сведите со своими, затем повторите push. Не используйте push --force, чтобы «продавить» - так Вы затрёте чужую работу.</Callout>

## Типичные ошибки

- **Считать `git pull` безобидным.** `pull` сразу сливает чужие изменения в Вашу ветку и может втянуть Вас в конфликт без предупреждения. Когда хочется сперва посмотреть, что прилетело, безопаснее `git fetch` и затем осознанное слияние.
- **`push --force` для обхода отказа.** Force-push затирает на remote коммиты, которых у Вас нет, - то есть чужую работу. К нему прибегают редко и только на личных ветках, понимая последствия.
- **Путать имя remote и имя ветки.** В `git push origin main` первое слово - имя remote (`origin`), второе - имя ветки (`main`). Перепутать их местами - типичная причина «remote not found».

<Detail summary="Remote-tracking ветки: origin/main - это не main">

После `git fetch` Git обновляет ветки вида `origin/main` - они называются remote-tracking и представляют «состояние ветки main на origin на момент последнего fetch». Это не то же самое, что Ваша локальная `main`: `origin/main` Вы напрямую не редактируете, она лишь отражает сервер. Когда Вы делаете `git pull`, Git сначала обновляет `origin/main` через fetch, а потом сливает её в Вашу локальную `main`. Понимание этой пары - «локальная ветка против её remote-tracking слепка» - снимает большую часть путаницы при синхронизации: Вы всегда видите, насколько Ваша ветка ушла вперёд или отстала от сервера.

</Detail>
