# Tags mark important commits; reflog remembers where you were

Two tools rarely needed daily but indispensable at the right moment. A tag is a permanent label on a commit: "version 1.0 here", "client release here". Reflog is a personal journal of `HEAD` moves: it remembers every position you were on, even ones seemingly "lost" after `reset --hard`. Together they give anchor points in history and insurance against your own mistakes.

By the end of this section you will create tags, understand lightweight vs annotated tags, read reflog, and see why a "lost" commit is usually recoverable.

## Tags: named bookmarks on commits

A branch moves forward with each commit; a tag does not. A tag pins a name to a *specific* commit and stays there forever. Tags mark versions: `v1.0.0`, `v2.3.1`, bookmarks to return to exact release state.

```bash
git tag v1.0.0
```

That line creates a **lightweight** tag: simply a name pointing at the current commit. By default the tag sits on `HEAD`, but you can name any commit explicitly:

```bash
git tag release-1 HEAD~2
```

There is a second kind, the **annotated** tag: `git tag -a v1.0.0 -m "release"`. It stores not only the name but author, date, and message: a full object in the repository. Annotated tags are preferred for public releases because they carry "who and when released" metadata. Lightweight tags are simpler, "just a bookmark" for personal notes.

## reflog: the journal that remembers everything

The most common beginner fear: "I ran `reset --hard` and lost a commit". Good news: almost always the commit is not lost, just no branch points at it anymore. Git does not delete such commits immediately, and `reflog` remembers where `HEAD` has been.

```bash
git reflog
```

`git reflog` shows recent `HEAD` positions: every commit, branch switch, reset, merge leaves an entry here. It is a local journal of *your* moves (not on remote, not for colleagues). Among entries you see the commit you "lost", and knowing it, returning is easy.

## Insurance before a dangerous operation

Reflog is a life ring after the fact, but safer to lay straw *before*. Before a risky operation (same `reset --hard`, complex rebase) mark the current commit: tag or branch. Whatever happens, a name still points at the state you need and return is trivial.

```bash
git tag backup
git reset --hard HEAD~1
```

Even if `reset` moved `main` back, tag `backup` still points at the commit that was the tip. Restore is one command: `git reset --hard backup` returns the branch and working tree.

Try it in the terminal. The repository has two commits; the second is "important work". Tag `backup` on the current commit, then `git reset --hard HEAD~1`, and the tip moves back. Verify tag `backup` still points at the lost work. The check goal is the tag exists and `HEAD` has one commit.

## When behavior surprises you

Tags by default **do not go to remote** on ordinary `git push`. Surprise: you tagged `v1.0.0`, pushed the branch, and the tag is missing on the server. Tags must be sent separately (`git push origin v1.0.0` or `git push --tags`). Logic: tags are intentional marks and Git does not scatter them automatically.

<Callout type="warning">Ordinary git push does not send tags. To put a tag on remote send it explicitly: `git push origin <tag>` or git push --tags.</Callout>

Second point about reflog: it is *local and temporary*. Entries live a limited time (weeks by default) and do not exist on another machine. Reflog saves you from a fresh mistake but is not eternal archive, so do not expect to find month-old moves there.

## Common mistakes

- **Treating a tag like a moving branch.** A tag is pinned to a commit permanently. New commits do not move it: that is the point, it marks fixed state.
- **Panicking after `reset --hard`.** Before calling a commit lost, check `git reflog`: likely it is there and you can return with `git reset --hard <hash>`.
- **Forgetting to push a tag.** You tagged a version locally but the team does not see it because ordinary push skips tags. Send tags explicitly.

<Detail summary="Why reflog sees what log does not">

`git log` walks parent links from the branch tip: it shows only commits *reachable* from current state. After `reset --hard HEAD~1` the undone commit becomes unreachable: no branch points at it and `log` will not show it. But the commit object still sits in the repository and `reflog` holds an entry "HEAD was here". So reflog is not "project history" but "your HEAD history": a move journal that catches even what fell out of reachable history. When reflog retention expires and nothing references the commit, Git's garbage collector removes it, and then it is truly gone.

</Detail>
