# Park unfinished work and move one commit

Two common snags with targeted tools. First: you are mid-edit and must switch urgently to a clean branch, but you do not want to commit a half-done state. Second: one branch has a single commit you need, a ready bugfix, say, and you must move it to another branch without dragging everything else. For the first there is `git stash`; for the second, `git cherry-pick`.

By the end of this section you will confidently hide uncommitted work in stash and bring it back, and move individual commits between branches via cherry-pick, understanding how that differs from merge.

## stash: a pocket for unfinished work

`git stash` takes all uncommitted changes (from working tree and stage), hides them in separate storage, and returns the working tree to the last commit state, clean. As if you put a draft in a desk drawer to clear the desk.

```bash
git stash
```

After that `git status` shows a clean tree: you can switch branches calmly, fix urgent work, commit. When you return, pull the parked work back:

```bash
git stash pop
```

`pop` extracts the latest hidden entry and applies it to the current working tree, removing it from storage. Your draft is back and you continue where you stopped.

Stash is a stack (you can stash multiple times); `git stash list` shows it. Most often the pattern is simple: stash, do urgent work, pop.

## cherry-pick: moving one commit

Sometimes you need exactly one commit from another branch, not the whole branch. Classic case: on `feature` you accidentally made an important bugfix and need it on `main` right now without waiting for the whole feature to merge. `git cherry-pick <commit>` takes changes from that commit and applies them as a new commit on the current branch.

```bash
git switch main
git cherry-pick a1b2c3d
```

Git creates on `main` a new commit with the same contents as the chosen one, but a separate commit with its own identifier, like rebase. Other commits on `feature` stay untouched.

## When behavior surprises you

`git stash pop` can conflict. If you stashed an edit then changed the same lines on the branch (or switched to a branch where they differ), `pop` collides stashed and current state and asks you to resolve, like merge. On successful `pop` the stash entry is removed; on conflict it stays until you settle everything and remove it manually. So if the working tree after `pop` matters, remember stash is a temporary pocket, not reliable archive storage.

<Callout type="warning">stash is temporary storage, not a backup. git stash pop can conflict, and long-forgotten stack entries are easy to confuse. Pop stashed work soon after you stash it.</Callout>

## Common mistakes

- **Using cherry-pick where merge is needed.** Cherry-pick suits *one or two* commits. Dragging a dozen creates duplicate commits in two branches. For a whole branch use merge or rebase.
- **Forgetting about stash.** Stashed an edit, switched, got absorbed and forgot `pop`. A week later `git stash list` shows a mysterious entry. The pocket is for short pauses.
- **Expecting cherry-pick to move a commit's dependencies.** It moves only the chosen commit's changes. If that commit relied on a neighbor not on the target branch, you may get conflict.

<Detail summary="cherry-pick and rebase are close relatives">

Both cherry-pick and rebase do the same in essence: take a commit's changes and reapply them on a new base via three-way merge. Rebase is roughly a series of cherry-picks in a row: "move all my commits onto that tip". Cherry-pick is the point version: "move this one". So they share properties: the new commit gets a new identifier, and overlapping edits can conflict, resolved like merge. Understanding one, you understand both.

</Detail>
