# Слияние сводит две линии работы в одну

Вы закончили фичу на ветке `feature` и хотите вернуть её в `main`. Ветки дали Вам изоляцию; теперь нужна обратная операция - свести наработки воедино. Это и есть merge: Git берёт коммиты одной ветки и вливает их в другую так, чтобы итог содержал работу обеих.

К концу раздела Вы будете различать два сценария слияния - быстрый fast-forward и настоящий трёхсторонний merge с отдельным коммитом, - понимать, когда происходит каждый, и осознанно выбирать между ними через флаг `--no-ff`.

## Два сценария: fast-forward и трёхсторонний merge

Что именно сделает `git merge feature`, зависит от того, расходились ли ветки.

**Fast-forward** случается, когда основная ветка не двигалась с момента, как Вы ответвились. История осталась линейной: коммиты `feature` просто продолжают `main`. Тогда «слияние» - это сдвинуть указатель `main` вперёд, на вершину `feature`. Новый коммит не создаётся; merge-коммита нет, потому что и сливать-то нечего - линия одна.

**Трёхсторонний merge** (3-way) нужен, когда обе ветки получили свои коммиты после точки расхождения. Линии разошлись, и просто сдвинуть указатель нельзя - часть работы потерялась бы. Git находит общего предка (merge base) и сводит три снимка: предка, вершину `main` и вершину `feature`. Результат он запечатывает в особый **merge-коммит** - единственный коммит с двумя родителями.

Merge-коммит с двумя родителями - это узел, где история перестаёт быть прямой линией и становится графом. Именно по таким коммитам потом видно, где две линии работы встретились.

## Делаем слияние

Порядок один и тот же: встаньте на ветку, *в которую* вливаете, и слейте ту, *из которой* берёте.

```bash
git switch main
git merge feature
```

Git сам решит, fast-forward это или 3-way, и сообщит результат. Для fast-forward он напишет, что перемотал ветку вперёд; для 3-way - что создал merge-коммит «recursive»-стратегией.

После успешного слияния работа `feature` целиком в `main`. Сама ветка `feature` никуда не девается - это по-прежнему указатель на свой последний коммит; её можно удалить за ненадобностью (`git branch -d feature`) или продолжать на ней работать.

Попробуйте в терминале. На `feature` уже есть коммит, которого нет на `main`, а `main` с тех пор не двигался - значит, будет fast-forward. Встаньте на `main` и выполните `git merge feature`. Проверка ждёт, что `feature` окажется влит в `main`.

## Когда поведение удивит

Если ветки правили *разные* файлы (или разные части одного), Git сольёт их автоматически - даже трёхсторонний merge пройдёт без Вашего участия. Сюрприз для новичка в том, что «слияние» не значит «конфликт»: конфликт возникает только когда обе ветки изменили *одни и те же* строки несовместимо. Пока правки не пересекаются, Git справляется сам. Разбору конфликтов посвящён следующий раздел [[merge-conflicts]].

<Callout type="info">Слияние двух веток, которые трогали разные файлы, проходит автоматически и без конфликтов. Конфликт - это только про несовместимые правки одних и тех же строк.</Callout>

Второй момент - `--no-ff`. Иногда fast-forward нежелателен: Вы хотите, чтобы в истории остался явный merge-коммит «здесь влилась фича», даже если технически возможна перемотка. Флаг `git merge --no-ff feature` заставляет Git создать merge-коммит принудительно. Это вопрос стиля истории, к которому мы вернёмся в разделе [[workflows]].

## Типичные ошибки

- **Сливать, стоя не на той ветке.** `git merge feature` вливает `feature` в *текущую* ветку. Если Вы случайно на `feature`, то вольёте `main` в `feature` - не то, что хотели. Перед merge проверьте `git status`: на какой Вы ветке.
- **Ждать merge-коммит после fast-forward.** При перемотке отдельного коммита не создаётся - это нормально, история осталась линейной. Если merge-коммит нужен обязательно, используйте `--no-ff`.
- **Считать, что merge переносит ветку.** Слияние копирует *работу* `feature` в `main`, но сам указатель `feature` остаётся на месте. Удалить ветку после слияния - отдельное действие.

<Detail summary="Как Git выбирает merge base">

Общий предок (merge base) - это последний коммит, который есть в истории обеих веток. Git находит его, поднимаясь по родительским ссылкам от обеих вершин, пока линии не сойдутся. Имея три точки - базу, «нашу» вершину и «их» вершину, - Git для каждого файла сравнивает, кто и что менял относительно базы. Если файл тронула только одна сторона, берётся её версия. Если обе - и одинаково, берётся общий результат. Если обе и по-разному в одних строках - это конфликт, который Git отдаёт Вам. Эта же трёхсторонняя логика, как Вы увидите дальше, лежит в основе rebase и cherry-pick.

</Detail>
