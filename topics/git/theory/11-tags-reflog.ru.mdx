# Теги отмечают важные коммиты, reflog помнит, где Вы были

Два инструмента, которые редко нужны ежедневно, но незаменимы в нужный момент. Тег - это постоянная метка на коммите: «здесь версия 1.0», «здесь релиз для клиента». А reflog - персональный журнал перемещений `HEAD`: он помнит каждую позицию, на которой Вы были, даже ту, что вроде бы «потеряна» после `reset --hard`. Вместе они дают точки опоры в истории и страховку от собственных ошибок.

К концу раздела Вы будете создавать теги, понимать разницу между лёгкими и аннотированными тегами, читать reflog и осознавать, почему «потерянный» коммит чаще всего можно вернуть.

## Теги: именованные закладки на коммитах

Ветка двигается вперёд с каждым коммитом, а тег - нет. Тег прибивает имя к *конкретному* коммиту и остаётся на нём навсегда. Поэтому теги используют для версий: `v1.0.0`, `v2.3.1` - это закладки, по которым легко вернуться к точному состоянию релиза.

```bash
git tag v1.0.0
```

Эта строка ставит **лёгкий** (lightweight) тег - просто имя, указывающее на текущий коммит. По умолчанию тег садится на `HEAD`, но можно указать любой коммит явно:

```bash
git tag release-1 HEAD~2
```

Есть и второй вид - **аннотированный** (annotated) тег: `git tag -a v1.0.0 -m "релиз"`. Он хранит не только имя, но и автора, дату и сообщение - полноценный объект в репозитории. Аннотированные теги предпочтительны для публичных релизов, потому что несут метаданные «кто и когда выпустил». Лёгкий тег проще - это «просто закладка» для личных пометок.

## reflog: журнал, который помнит всё

Самый частый страх новичка - «я сделал `reset --hard` и потерял коммит». Хорошая новость: почти всегда коммит не потерян, просто на него больше не указывает ни одна ветка. Git не удаляет такие коммиты сразу, а `reflog` помнит, где `HEAD` побывал.

```bash
git reflog
```

`git reflog` показывает список последних позиций `HEAD`: каждый коммит, переключение ветки, reset, merge оставляют здесь запись. Это локальный журнал именно Ваших перемещений (его нет на remote и нет у коллег). Среди этих записей виден и тот коммит, который Вы «потеряли», - а зная его, легко на него вернуться.

## Страховка перед опасной операцией

reflog - спасательный круг постфактум, но надёжнее подстелить соломку *заранее*. Перед рискованной операцией (тот же `reset --hard`, сложный rebase) поставьте на текущий коммит метку - тег или ветку. Тогда, что бы ни случилось, у Вас останется имя, указывающее на нужное состояние, и вернуться будет тривиально.

```bash
git tag backup
git reset --hard HEAD~1
```

Даже если `reset` увёл `main` назад, тег `backup` по-прежнему указывает на коммит, который был вершиной. Восстановить - вопрос одной команды: `git reset --hard backup` вернёт ветку обратно вместе с рабочим деревом.

Попробуйте в терминале. В репозитории два коммита, второй - «важная работа». Поставьте тег `backup` на текущий коммит, затем выполните `git reset --hard HEAD~1` - вершина уедет назад. Проверьте, что тег `backup` всё ещё указывает на потерянную работу. Цель проверки - чтобы тег существовал, а на `HEAD` остался один коммит.

## Когда поведение удивит

Теги по умолчанию **не уходят на remote** при обычном `git push`. Это сюрприз: Вы поставили `v1.0.0`, запушили ветку, а на сервере тега нет. Теги нужно отправлять отдельно (`git push origin v1.0.0` или `git push --tags`). Логика в том, что теги - намеренная пометка, и Git не разбрасывает их автоматически.

<Callout type="warning">Обычный git push не отправляет теги. Чтобы тег появился на remote, отправьте его явно: `git push origin <тег>` или git push --tags.</Callout>

Второй момент про reflog: он *локальный и временный*. Записи живут ограниченное время (по умолчанию недели), и на другой машине их нет. reflog спасёт Вас от только что сделанной ошибки, но это не вечный архив - не рассчитывайте найти там перемещения месячной давности.

## Типичные ошибки

- **Считать тег движущимся, как ветка.** Тег прибит к коммиту намертво. Новые коммиты его не двигают - в этом и смысл: он отмечает зафиксированное состояние.
- **Паниковать после `reset --hard`.** Прежде чем считать коммит потерянным, загляните в `git reflog`: скорее всего, он там, и на него можно вернуться через `git reset --hard <хеш>`.
- **Забыть запушить тег.** Поставили версию локально, а команда её не видит, потому что обычный push теги не берёт. Отправляйте теги явно.

<Detail summary="Почему reflog видит то, чего не видит log">

`git log` идёт по родительским ссылкам от вершины ветки: он показывает только коммиты, *достижимые* из текущего состояния. После `reset --hard HEAD~1` отменённый коммит становится недостижимым - ни одна ветка на него не указывает, и `log` его не покажет. Но сам объект-коммит ещё лежит в репозитории, и `reflog` хранит запись «HEAD был здесь». Поэтому reflog - это не «история проекта», а «история Вашего HEAD»: журнал перемещений, который ловит даже то, что выпало из достижимой истории. Когда срок хранения reflog истечёт и на коммит по-прежнему ничто не ссылается, сборщик мусора Git его уберёт - вот тогда он действительно исчезнет.

</Detail>
