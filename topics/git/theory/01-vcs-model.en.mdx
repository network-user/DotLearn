# Git stores snapshots, not a list of patches

Imagine a project you have been working on for a week. On Friday something breaks, and you know for sure that everything worked on Wednesday. Without version control, you have one option: a folder called `project-final`, then `project-final-2`, then `project-final-really-last`. A month later, nobody, including you, will remember how they differed or which change was dangerous to touch.

Git solves this differently. It remembers the state of the entire project at moments you choose and lets you return to any of them, compare two moments, and see who changed what and why. By the end of this section you will understand what a commit is, how a "snapshot" differs from a "list of patches", and which three areas a file passes through before it enters history.

## A snapshot of the whole project, not a delta of one file

Many older version control systems stored history as a list of differences: "on line 12 of `app.py`, word X was replaced with Y." To get the project state at a given moment, such a system applied a base version and all deltas in sequence.

Git is fundamentally different. Each commit is a **snapshot** of the entire file tree at the moment you save. Git is as if photographing the whole folder and filing the photo in history under a unique name. If a file did not change since the last commit, Git does not copy it again: it references the already stored version, so snapshots do not bloat the repository. But the model you reason in is "photographs of states", not "a journal of patches".

An important consequence follows immediately. A commit is not a "change" but a fixed state. Returning to a commit means restoring the entire snapshot, not "reverting one line". And when Git compares two commits (diff), it computes the difference on the fly by subtracting one snapshot from another: it shows you the differences, but stores the states.

## Three areas: working tree, index, repository

On its way into history, a file passes through three areas, and understanding this trio clears up most beginner confusion.

The **working tree** is the ordinary files on disk that you edit. This is where the draft lives: everything you typed in the editor but have not yet saved with Git.

The **index** (also called the staging area) is an intermediate zone. With `git add` you move here exactly the changes you want in the next commit. The index answers the question "what will go into the snapshot".

The **repository** is `.git`, where the entire commit history lives. With `git commit` you turn the index contents into a new permanent snapshot.

Why the middle zone, rather than committing straight from disk? The index gives you control over *what exactly* enters the commit. You fixed three files, but logically those are two different tasks, so you can add only the first task's files to the index, commit them with a meaningful message, then add the rest and make a second commit. History stays clean instead of "fixed everything on Tuesday evening".

## Creating a repository and the first snapshot

Enough theory. Create a repository by hand. The command `git init` creates an empty repository: a hidden `.git` folder appears and the default branch is set. From there it follows the diagram: `git add` sends a file to the index, `git commit` seals the snapshot.

```bash
git init
git add hello.txt
git commit -m "первый коммит"
```

The `-m` flag sets the commit message: a short description of what was recorded and why. Without a message Git will not accept the commit: history without labels is useless.

Try it yourself in the live terminal below. The file `hello.txt` is already in the working tree. Run `git status` to see it as "untracked", then run `git add hello.txt` and `git status` again to watch the file move to the index. Commit it and watch the first node appear in the graph. Change the commit message to your own.

## When behavior surprises you

The most common beginner surprise: you added a file to the index with `git add`, then edited it again, and the commit got the **old** version. That is correct. `git add` copies into the index a *snapshot of the file at the moment of the command*, not a "subscription to the file". If you change the file after `git add`, the change stays in the working tree and the index does not know about it. So `git status` can show the same file in two states at once: part ("staged") ready to commit, part ("not staged") not yet. The fix is simple: run `git add` again before committing.

<Callout type="warning">`git add` records the file contents at the moment you run it. If you edit the file after `git add`, run `git add` again. Otherwise the commit will get the previous version.</Callout>

## Common mistakes

- **Thinking of a commit as "saving a file".** A commit saves a snapshot of *all* indexed state, not one open file. Put three files in the index, and all three enter the commit.
- **Committing without `git add`.** A newly created file is "untracked" and will not enter a commit until you add it explicitly. `git commit` without a prior `git add` for a new file gives "nothing to commit".
- **Confusing the working tree with history.** Delete a file in the editor: it disappears only from the working tree. In past commits it is still there and easy to restore. History is immutable until you deliberately rewrite it.

<Detail summary="What physically lives in .git">

The `.git` folder contains the entire repository database: objects (tree snapshots, file contents, commits), references (`refs`) to branches and tags, and the `HEAD` pointer to the current position. Each object is addressed by a hash of its contents, so identical content is stored once and any data corruption is detected immediately by a hash mismatch. When you run `git commit`, Git creates a commit object that references a tree snapshot and a parent commit, which is how history builds into a chain.

</Detail>
