# Rebase переигрывает Ваши коммиты на новом основании

Вы неделю работали на ветке `feature`, а тем временем `main` ушла вперёд - другие влили туда свои изменения. Теперь Ваша ветка «отстала от основания». Слияние решит это merge-коммитом, но история станет ветвистой. Rebase предлагает другое: взять Ваши коммиты и переиграть их так, будто Вы ответвились от *свежей* `main`. Результат - прямая, линейная история, где работа `feature` аккуратно лежит поверх актуального `main`.

К концу раздела Вы будете понимать, чем rebase отличается от merge по результату, когда линейная история того стоит, и почему есть железное правило: не перебазировать то, чем уже поделились.

## Merge сохраняет развилку, rebase её выпрямляет

И merge, и rebase сводят расходящиеся ветки. Разница - в том, какую историю они оставляют.

Merge создаёт merge-коммит с двумя родителями: развилка и место встречи навсегда видны в истории. Rebase же **переписывает** Ваши коммиты: он находит общего предка, берёт коммиты `feature` после него и заново применяет их один за другим поверх вершины `main`. Старые коммиты заменяются новыми - с тем же содержимым, но другими родителями и другими идентификаторами.

Ключевое слово - «переигрывает». Rebase не двигает коммиты как файлы; он берёт *изменения* каждого коммита и применяет их заново на новом основании, создавая новые коммиты. Поэтому после rebase у Ваших коммитов другие идентификаторы: технически это другие коммиты, хоть и с тем же содержимым.

## Делаем rebase

Встаньте на ветку, которую хотите перенести, и укажите, *на что* её перебазировать.

```bash
git switch feature
git rebase main
```

Git проигрывает коммиты `feature` поверх вершины `main`. Если правки не пересекаются, всё проходит молча, и `feature` оказывается прямым продолжением `main`. После этого `main` можно слить в `feature` или наоборот - слияние будет fast-forward, потому что история уже линейна.

Попробуйте в терминале. `feature` и `main` разошлись: на каждой по своему коммиту после общего предка. Встаньте на `feature` (`git switch feature`) и выполните `git rebase main`. Коммит `feature` переедет на вершину `main`, и в истории станет три коммита в одну линию. Цель проверки - три коммита и оба файла на месте.

## Когда поведение удивит

Rebase может конфликтовать так же, как merge, - но конфликт разрешается для *каждого переигрываемого коммита по очереди*. Если три коммита `feature` трогают спорные строки, Git может остановиться три раза. На каждой остановке Вы правите файл, делаете `git add` и продолжаете командой `git rebase --continue`. Запутались - `git rebase --abort` возвращает ветку в состояние до rebase, ничего не теряя. Это нормальный ритм работы с rebase, а не признак, что что-то сломалось.

<Callout type="info">При конфликте rebase останавливается на каждом проблемном коммите. Разрешайте по одному: правка → git add → git rebase --continue. Передумали - git rebase --abort вернёт всё как было.</Callout>

## Золотое правило: не перебазируй опубликованное

Rebase переписывает коммиты - создаёт новые вместо старых. Пока коммиты живут только у Вас, это безвредно. Но если Вы уже отправили их на remote и кто-то на них основался, rebase превращает их в «других» коммитов, а старые остаются у коллег. Истории расходятся, и синхронизация превращается в кашу из дубликатов.

Отсюда золотое правило: **перебазируйте только те коммиты, которыми ещё не поделились**. Своя локальная ветка до первого push - можно. Общая ветка, на которую смотрят другие, - нельзя, там слияние. Это та же логика, что отделяла `reset` (переписывает историю) от `revert` (не переписывает) в разделе [[undoing-changes]].

<Callout type="warning">Не перебазируйте коммиты, которые уже отправлены на общий remote и доступны другим. Rebase заменяет их новыми, и чужие репозитории рассинхронизируются. Для опубликованной истории используйте merge.</Callout>

## Типичные ошибки

- **Rebase общей ветки.** Перебазировали `main`, которой пользуется команда, - и у всех «разъехались» истории. Rebase - инструмент для локальной, ещё не расшаренной работы.
- **Считать, что rebase «безопаснее» merge.** Он не безопаснее, он *другой*. Merge честно документирует развилку; rebase даёт линейность ценой переписывания. Выбор - вопрос принятого в команде стиля истории (раздел [[workflows]]).
- **Бросать rebase в середине при конфликте.** Если оставить незавершённый rebase, ветка висит в промежуточном состоянии. Доводите до конца через `--continue` или явно отменяйте через `--abort`.

<Detail summary="Почему линейная история кому-то важна">

Линейная история (без merge-коммитов) читается как простой список: «сделали А, потом Б, потом В». По ней удобнее бежать инструментами вроде `git bisect`, ищущими коммит, где появился баг, и проще ревьюить - каждый коммит применяется к предсказуемому состоянию. Цена - потеря информации о том, что Б и В на самом деле делались параллельно с чем-то ещё. Команды, ценящие читаемость линии, перебазируют фичи перед вливанием; команды, ценящие точную летопись, оставляют merge-коммиты. Обе позиции защитимы - это решение о том, что важнее в Вашем проекте, а не вопрос «правильно/неправильно».

</Detail>
