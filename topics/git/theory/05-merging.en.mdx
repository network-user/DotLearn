# Merge brings two lines of work into one

You finished a feature on branch `feature` and want it back in `main`. Branches gave you isolation; now you need the reverse: combine the work. That is merge: Git takes commits from one branch and pours them into another so the result contains work from both.

By the end of this section you will distinguish two merge scenarios (fast-forward and a real three-way merge with its own commit), understand when each happens, and choose consciously between them with the `--no-ff` flag.

## Two scenarios: fast-forward and three-way merge

What `git merge feature` does depends on whether the branches diverged.

**Fast-forward** happens when the main branch did not move since you branched off. History stayed linear: commits on `feature` simply continue `main`. Then "merge" is moving the `main` pointer forward to the tip of `feature`. No new commit is created; there is no merge commit because there is nothing to merge: it is one line.

**Three-way merge** is needed when both branches got their own commits after the divergence point. The lines split and you cannot simply move a pointer, because work would be lost. Git finds the common ancestor (merge base) and reconciles three snapshots: the base, the tip of `main`, and the tip of `feature`. It seals the result in a special **merge commit**: the only commit with two parents.

A merge commit with two parents is a node where history stops being a straight line and becomes a graph. These commits show where two lines of work met.

## Performing a merge

The order is always the same: stand on the branch you merge *into*, and merge the one you take *from*.

```bash
git switch main
git merge feature
```

Git decides fast-forward or 3-way and reports the result. For fast-forward it says the branch was fast-forwarded; for 3-way, that it created a merge commit with the "recursive" strategy.

After a successful merge all work from `feature` is in `main`. Branch `feature` itself remains, still a pointer to its last commit; you can delete it if unused (`git branch -d feature`) or keep working on it.

Try it in the terminal. `feature` already has a commit not on `main`, and `main` has not moved since, so it will be fast-forward. Stand on `main` and run `git merge feature`. The check expects `feature` merged into `main`.

## When behavior surprises you

If branches edited *different* files (or different parts of one file), Git merges automatically: even a three-way merge completes without you. The surprise for beginners is that "merge" does not mean "conflict": conflict only arises when both branches changed *the same* lines incompatibly. While edits do not overlap, Git handles it alone. Conflicts are covered in [[merge-conflicts]].

<Callout type="info">Merging two branches that touched different files completes automatically with no conflict. Conflict is only about incompatible edits to the same lines.</Callout>

The second point is `--no-ff`. Sometimes fast-forward is undesirable: you want an explicit merge commit in history marking "feature landed here", even when a fast-forward is possible. `git merge --no-ff feature` forces Git to create a merge commit. That is a history style question we return to in [[workflows]].

## Common mistakes

- **Merging while on the wrong branch.** `git merge feature` merges `feature` into the *current* branch. If you are on `feature` by accident, you merge `main` into `feature`, which is not what you wanted. Before merge check `git status`: which branch are you on?
- **Expecting a merge commit after fast-forward.** On fast-forward no separate commit is created. That is normal because history stayed linear. If you need a merge commit, use `--no-ff`.
- **Thinking merge moves the branch.** Merge copies *work* from `feature` into `main`, but pointer `feature` stays put. Deleting the branch after merge is a separate action.

<Detail summary="How Git chooses the merge base">

The common ancestor (merge base) is the last commit present in both branches' history. Git finds it by walking parent links from both tips until the lines meet. With three points (base, "our" tip, "their" tip), Git compares per file who changed what relative to the base. If only one side touched a file, its version wins. If both changed the same way, the shared result wins. If both changed the same lines differently, that is a conflict Git hands to you. The same three-way logic, as you will see later, underlies rebase and cherry-pick.

</Detail>
