# Git хранит снимки, а не список правок

Представьте проект, над которым Вы работаете неделю. В пятницу что-то сломалось, и Вы точно знаете, что в среду всё работало. Без системы контроля версий у Вас один вариант: папка `project-final`, рядом `project-final-2`, потом `project-final-точно-последний`. Через месяц никто, включая Вас, не вспомнит, чем они отличались и какую правку было опасно трогать.

Git решает это иначе. Он запоминает состояние всего проекта в выбранные Вами моменты и позволяет в любой из них вернуться, сравнить два момента, увидеть, кто и зачем сделал каждое изменение. К концу раздела Вы будете понимать, что такое коммит, чем «снимок» отличается от «списка правок» и через какие три области проходит файл, прежде чем попасть в историю.

## Снимок целого проекта, а не дельта одного файла

Многие старые системы контроля версий хранили историю как список различий: «в строке 12 файла `app.py` слово X заменили на Y». Чтобы получить состояние проекта на нужный момент, такая система складывала базовую версию и все дельты по очереди.

Git устроен принципиально иначе. Каждый коммит - это **снимок** (snapshot) всего дерева файлов на момент сохранения. Git как будто фотографирует всю папку целиком и подшивает фотографию в историю под уникальным именем. Если файл с прошлого коммита не менялся, Git не копирует его заново, а ссылается на уже сохранённую версию, поэтому снимки не раздувают репозиторий. Но модель, в которой Вы рассуждаете, - именно «фотографии состояний», а не «журнал правок».

Из этой модели сразу следует важное. Коммит - не «изменение», а зафиксированное состояние. Вернуться к коммиту значит восстановить весь снимок, а не «откатить одну строку». А сравнение двух коммитов (diff) Git вычисляет на лету, вычитая один снимок из другого, - различия он показывает Вам, но хранит состояния.

## Три области: рабочее дерево, индекс, репозиторий

Файл на пути в историю проходит через три области, и понимание этой троицы снимает большую часть путаницы новичка.

**Рабочее дерево** (working tree) - это обычные файлы на диске, которые Вы редактируете. Здесь живёт черновик: всё, что Вы напечатали в редакторе, но ещё никуда не сохранили средствами Git.

**Индекс** (index, он же staging area) - промежуточная площадка. Командой `git add` Вы переносите сюда именно те изменения, которые хотите положить в следующий коммит. Индекс отвечает на вопрос «что войдёт в снимок».

**Репозиторий** (repository) - это `.git`, где лежит вся история коммитов. Командой `git commit` Вы превращаете содержимое индекса в новый постоянный снимок.

Зачем нужна средняя площадка, почему нельзя коммитить прямо с диска? Индекс даёт контроль над тем, *что именно* попадёт в коммит. Вы поправили три файла, но логически это две разные задачи - можно добавить в индекс только файлы первой задачи, закоммитить их с осмысленным сообщением, потом добавить остальные и сделать второй коммит. История получается чистой, а не «правил всё подряд за вечер».

## Создаём репозиторий и первый снимок

Хватит теории - заведите репозиторий руками. Команда `git init` создаёт пустой репозиторий: появляется скрытая папка `.git` и ветка по умолчанию. Дальше всё как на схеме: `git add` отправляет файл в индекс, `git commit` запечатывает снимок.

```bash
git init
git add hello.txt
git commit -m "первый коммит"
```

Флаг `-m` задаёт сообщение коммита - короткое описание того, что и зачем зафиксировано. Без сообщения Git коммит не примет: история без подписей бесполезна.

Попробуйте сами в живом терминале ниже. Файл `hello.txt` уже лежит в рабочем дереве. Выполните `git status`, чтобы увидеть его как «untracked», затем `git add hello.txt` и `git status` снова - файл переедет в индекс. Закоммитьте его и посмотрите, как в графе появляется первый узел. Поменяйте сообщение коммита на своё.

## Когда поведение удивит

Самый частый сюрприз новичка: добавил файл в индекс через `git add`, потом ещё раз его поправил - и в коммит ушла **старая** версия. Так и должно быть. `git add` копирует в индекс *снимок файла на момент команды*, а не «подписку на файл». Изменили файл после `git add` - изменение осталось в рабочем дереве, индекс про него не знает. Поэтому `git status` покажет один и тот же файл сразу в двух состояниях: часть («staged») готова к коммиту, часть («not staged») - нет. Лекарство простое: повторите `git add` перед коммитом.

<Callout type="warning">`git add` фиксирует содержимое файла на момент вызова. Если правите файл после `git add`, повторите `git add`, иначе в коммит уйдёт прежняя версия.</Callout>

## Типичные ошибки

- **Думать о коммите как о «сохранении файла».** Коммит сохраняет снимок *всего* проиндексированного состояния, а не один открытый файл. Положили в индекс три файла - все три войдут в коммит.
- **Коммитить без `git add`.** Только что созданный файл Git считает «untracked» и в коммит не возьмёт, пока Вы его явно не добавите. `git commit` без предварительного `git add` для нового файла даст «nothing to commit».
- **Путать рабочее дерево и историю.** Удалили файл в редакторе - он исчез только из рабочего дерева. В прошлых коммитах он по-прежнему есть, и его легко вернуть. История неизменна, пока Вы её намеренно не перепишете.

<Detail summary="Что физически лежит в .git">

Папка `.git` содержит всю базу данных репозитория: объекты (снимки деревьев, содержимое файлов, коммиты), ссылки (`refs`) на ветки и теги, а также указатель `HEAD` на текущую позицию. Каждый объект адресуется хешем своего содержимого, поэтому одинаковое содержимое хранится один раз, а любое повреждение данных немедленно обнаруживается по несовпадению хеша. Когда Вы делаете `git commit`, Git создаёт объект-коммит, который ссылается на снимок дерева и на родительский коммит, - так история и выстраивается в цепочку.

</Detail>
