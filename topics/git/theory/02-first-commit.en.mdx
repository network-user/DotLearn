# The index is the stage where you set up the next commit

You already know that a commit is a snapshot of the indexed state. Now let's tackle the most common point of confusion: why there is an index between "I edited a file" and "I saved that in history", and how to use it so each commit is a neat, finished unit of work.

By the end of this section you will confidently stage individual files, read `git status` as a map of state, build one commit from part of your changes, fix mistakes with `git restore --staged`, and understand why `git add` and `git commit` are two separate steps, not one.

## Staging as a rehearsal for the commit

The English term staging area literally means "the stage". Imagine the commit is a performance you are about to show to history. The index is the stage where you place exactly the actors (file changes) that should appear in that performance. While rehearsal is underway, you can add and remove actors from the stage. `git commit` is raising the curtain: everything on stage is fixed as a single snapshot.

The key consequence: what lands on stage is not a file, but **the state of the file at the moment of `git add`**. That is why the two steps (`add` and `commit`) are deliberately separate. Between them you can change your mind, review what you assembled, add more, or remove extras.

## git status: a map of three states

`git status` shows which area your changes are in right now. It is the first command to run whenever you are unsure what is going on.

```bash
git status
```

The output groups files. "Changes to be committed": already on stage (in the index) and will enter the commit. "Changes not staged for commit": modified files not yet on stage. "Untracked files": new files Git knows nothing about until you add them. The same file can appear in two groups at once: part of its changes staged, part not.

```bash
git add app.py
git status
git commit -m "добавил обработчик"
```

After `git add app.py` the file moves to "Changes to be committed". After `git commit` the working tree is clean: `git status` reports "nothing to commit, working tree clean".

## Building a commit from part of your changes

The power of the index is that you choose what goes into the commit. Suppose you edited two files in an evening: `parser.py` (a new feature) and `README.md` (a small typo). Logically those are two independent fixes, so two commits.

```bash
git add parser.py
git commit -m "поддержка вложенных тегов"
git add README.md
git commit -m "исправил опечатку в примере"
```

The first `commit` sealed only `parser.py`, because only it was on stage. `README.md` stayed modified in the working tree and went into the second commit. History reads as a meaningful list of decisions, not "Tuesday evening edits".

Try it in the terminal. The repository has two modified files. Place only `feature.py` on stage, commit it, and check `git status`: you will see that `notes.txt` is still waiting. Then commit it too. Try deliberately using `git add .` instead of one file and compare how the first commit's contents change.

## When behavior surprises you

You placed a file on stage, then changed your mind: you want it out of the commit but keep the edits. Instinct says `git rm`, but that deletes the file. The right tool is `git restore --staged`: it removes the file from stage (returns the index to the version from the last commit) without touching your changes in the working tree.

```bash
git add secret.txt
git restore --staged secret.txt
```

After that `secret.txt` is "not staged" again: the edits remain, but the file will not enter the next commit until you add it again. This is the opposite of `git rm`, which removes the file from both the index and disk.

<Callout type="warning">`git restore --staged file` removes a file from stage while keeping edits. `git rm file` removes the file from the index and working tree. Do not confuse them: the first undoes staging, the second destroys the file.</Callout>

## Common mistakes

- **Treating `git add` and `git commit` as one command.** They are two steps: `add` places on stage, `commit` raises the curtain. Running `git commit` right after editing a new file means "nothing to commit", because the stage is empty.
- **Committing with an empty or meaningless message.** `git commit -m "fix"` tells you nothing a month later. The message explains *why*, not *what* (that is visible in the diff). Write "fixed crash on empty input", not "fix".
- **Forgetting new files when adding by name.** If you stage files one by one, it is easy to forget a file you just created. `git status` before commit shows everything left in "untracked".

<Detail summary="git commit -am: temptation and its limit">

The flag `git commit -am "message"` combines two steps: stages changes and commits immediately. Convenient, but with an important caveat: it only takes **already tracked** files (those that were in the previous commit). A new file not yet added is ignored by `-a`: you still need `git add` first. So `-am` is good for a quick fix to existing files, but does not replace deliberate staging when you need to build a commit from part of your changes.

</Detail>
