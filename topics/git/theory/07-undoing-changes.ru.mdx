# «Отменить» в Git - это четыре разных вопроса

Новичок ищет одну кнопку «отменить». В Git её нет, и это к лучшему: «отменить» может значить совсем разные вещи. Выбросить правку в файле, пока она не закоммичена? Снять файл со сцены? Сдвинуть ветку на коммит назад? Создать новый коммит, отменяющий старый, не переписывая историю? Это четыре операции с четырьмя командами, и путать их - значит однажды потерять работу.

К концу раздела Вы будете уверенно выбирать между `git restore`, тремя режимами `git reset` и `git revert`, понимая для каждого, что именно он трогает - рабочее дерево, индекс или историю - и какие из них безопасны, а какие необратимы.

## Карта: что именно трогает каждая команда

Все операции отката можно разложить по одной таблице: какую из трёх областей - рабочее дерево, индекс, историю - команда меняет. Держите эту карту в голове, и выбор команды станет очевидным.

## restore: откат на уровне файла

`git restore <file>` отвечает на вопрос «выбросить мои правки в этом файле, вернуть как было». Он берёт файл из индекса (или из последнего коммита) и перезаписывает им рабочее дерево. Незакоммиченные изменения этого файла теряются - это и есть смысл команды.

```bash
git restore config.txt
```

Если файл был только изменён (не добавлен в индекс), `git restore` вернёт его к версии последнего коммита. Отдельная форма `git restore --staged <file>`, которую Вы видели в разделе [[first-commit]], не трогает содержимое файла - она лишь снимает его со сцены. Две формы, две разные задачи: одна откатывает текст файла, другая отменяет стадирование.

## reset: сдвиг ветки и три уровня глубины

`git reset <commit>` переносит указатель текущей ветки на другой коммит. Чаще всего его используют, чтобы «отменить последний коммит» - сдвинуть ветку на `HEAD~1`. Но что станет с изменениями отменённого коммита, зависит от режима.

**`--soft`** двигает только указатель ветки. Содержимое отменённого коммита остаётся в индексе, готовое к новому коммиту. Это идеальный способ «пересобрать последний коммит»: откатили, поправили состав, закоммитили заново.

**`--mixed`** (режим по умолчанию) двигает ветку *и* сбрасывает индекс. Изменения отменённого коммита остаются в рабочем дереве, но уже не на сцене - их нужно заново добавить. Полезно, когда хочется переразложить правки по коммитам.

**`--hard`** двигает ветку, сбрасывает индекс *и* переписывает рабочее дерево под целевой коммит. Все изменения отменённого коммита и любые незакоммиченные правки **исчезают**. Это единственный режим, который реально теряет данные.

```bash
git reset --soft HEAD~1
git reset --mixed HEAD~1
git reset --hard HEAD~1
```

Поэкспериментируйте. В репозитории два коммита; второй добавил строку в `a.txt`. Сделайте `git reset --soft HEAD~1` и `git status` - коммитов снова один, но правка ждёт на сцене. Цель проверки - один коммит в истории и файл `a.txt`, выставленный на сцену.

## revert: отмена без переписывания истории

`git reset` переписывает историю - сдвигает ветку, и старый коммит выпадает из неё. Это опасно для коммитов, которыми Вы уже поделились: чужие репозитории всё ещё ссылаются на «отменённый» коммит. Для публичной истории есть безопасная альтернатива - `git revert`.

`git revert <commit>` не удаляет коммит. Он создаёт **новый** коммит, который вносит обратные изменения - как если бы Вы вручную откатили те правки и зафиксировали откат. История растёт вперёд, ничего не переписывается, и каждый, у кого есть репозиторий, спокойно получит этот новый коммит.

```bash
git revert HEAD
```

После этого в истории на один коммит больше: исходный «плохой» коммит на месте, а следом - коммит «Revert ...», возвращающий состояние. Это золотой стандарт отмены того, что уже ушло в общую ветку.

## Когда поведение удивит

Главный источник боли - `git reset --hard`. Он не спрашивает подтверждения и стирает незакоммиченные правки безвозвратно: их нет ни в одном коммите, значит, восстанавливать нечего. Привычка - не запускать `--hard`, пока не убедились через `git status`, что в рабочем дереве нет ничего ценного. Отменённые этой командой *коммиты* какое-то время можно найти через `reflog` (раздел [[tags-reflog]]), а вот незакоммиченные правки - нельзя.

<Callout type="warning">git reset --hard стирает незакоммиченные изменения без возможности восстановления. Перед запуском проверьте git status: всё ценное должно быть либо в коммите, либо вне зоны сброса.</Callout>

## Типичные ошибки

- **Брать `git reset --hard` там, где хватило бы `--soft` или `restore`.** Хотели пересобрать последний коммит - используйте `--soft`, он сохранит правки. `--hard` нужен редко и только осознанно.
- **`reset` на уже опубликованных коммитах.** Если коммит ушёл в общую ветку, `reset` рассинхронизирует Вашу историю с чужими. Для общей истории отменяйте через `revert`.
- **Путать `git restore file` и `git restore --staged file`.** Первая выбрасывает правки в файле, вторая лишь снимает его со сцены, не трогая содержимое. Перепутали - либо потеряете правку, либо не добьётесь нужного эффекта.

<Detail summary="Почему --mixed - режим по умолчанию">

Когда Вы пишете `git reset HEAD~1` без флага, Git применяет `--mixed`. Это разумная середина: ветка откатывается, изменения отменённого коммита сохраняются в рабочем дереве (Вы их не теряете), но снимаются со сцены, чтобы Вы пересобрали коммит осознанно. `--soft` оставил бы их прямо на сцене (удобно, если состав коммита Вас устраивал), `--hard` выбросил бы совсем. По умолчанию Git выбирает наименее разрушительный из «двигающих ветку» вариантов, который при этом заставляет Вас заново подумать о составе коммита.

</Detail>
