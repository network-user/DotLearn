# Отложить незаконченное и перенести один коммит

Два частых затруднения, которые решаются точечными инструментами. Первое: Вы на середине правки, и срочно нужно переключиться на чистую ветку, но коммитить полуфабрикат не хочется. Второе: в одной ветке есть один нужный коммит - например, готовый багфикс, - и его надо перенести в другую ветку, не таща за собой всё остальное. Для первого есть `git stash`, для второго - `git cherry-pick`.

К концу раздела Вы будете уверенно прятать незакоммиченную работу в stash и возвращать её, а также переносить отдельные коммиты между ветками через cherry-pick, понимая, чем это отличается от merge.

## stash: карман для незаконченной работы

`git stash` забирает все незакоммиченные изменения (и из рабочего дерева, и со сцены), прячет их в отдельное хранилище и возвращает рабочее дерево к состоянию последнего коммита - чистым. Будто Вы убрали черновик в ящик стола, чтобы освободить рабочую поверхность.

```bash
git stash
```

После этого `git status` покажет чистое дерево: можно спокойно переключиться на другую ветку, починить срочное, закоммитить. Когда вернётесь, достаньте отложенное обратно:

```bash
git stash pop
```

`pop` извлекает последнюю спрятанную запись и накладывает её на текущее рабочее дерево, удаляя из хранилища. Ваш черновик снова на месте, и Вы продолжаете с того, на чём остановились.

Stash - это стопка (можно отложить несколько раз), `git stash list` показывает её содержимое. Но чаще всего паттерн простой: спрятал, сделал срочное, достал.

## cherry-pick: перенос одного коммита

Иногда нужен ровно один коммит из чужой ветки, а не вся она. Классический случай: на ветке `feature` Вы случайно сделали важный багфикс, и его надо вернуть в `main` прямо сейчас, не дожидаясь, пока вся фича будет готова к слиянию. `git cherry-pick <коммит>` берёт изменения указанного коммита и применяет их как новый коммит на текущей ветке.

```bash
git switch main
git cherry-pick a1b2c3d
```

Git создаёт на `main` новый коммит с тем же содержимым, что у выбранного, - но это отдельный коммит со своим идентификатором, как при rebase. Остальные коммиты `feature` остаются нетронутыми.

На рисунке из ветки с коммитами `X → fix → Y` на `main` переносится только `fix`, превращаясь в новый коммит `fix'`. Соседи `X` и `Y` остаются на своей ветке.

Поэкспериментируйте в терминале. На `feature` есть коммит с багфиксом `fix.txt`. Встаньте на `main` (`git switch main`) и перенесите его: `git cherry-pick feature`. На `main` появится копия багфикса, а сама `feature` не изменится. Цель проверки - чтобы `fix.txt` оказался отслеживаемым и Вы остались на `main`.

## Когда поведение удивит

`git stash pop` может конфликтовать. Если Вы спрятали правку, а потом изменили те же строки на ветке (или переключились на ветку, где они другие), при `pop` Git столкнёт спрятанное с текущим и попросит разрешить конфликт - ровно как при merge. Спрятанная запись при удачном `pop` удаляется из хранилища; при конфликте она остаётся, пока Вы не уладите всё и не уберёте её вручную. Поэтому если рабочее дерево после `pop` важно, не теряйте из виду, что stash - временный карман, а не надёжное архивное хранилище.

<Callout type="warning">stash - временное хранилище, а не бэкап. git stash pop может конфликтовать, а длинные «забытые» записи в стопке легко перепутать. Доставайте отложенное вскоре после того, как спрятали.</Callout>

## Типичные ошибки

- **Использовать cherry-pick там, где нужен merge.** Cherry-pick хорош для *одного-двух* коммитов. Тащить им десяток - утомительно и создаёт дубликаты коммитов в двух ветках. Для целой ветки используйте merge или rebase.
- **Забыть про stash.** Спрятали правку, переключились, увлеклись и забыли сделать `pop`. Через неделю `git stash list` покажет загадочную запись. Карман - для коротких пауз.
- **Ждать, что cherry-pick перенесёт «зависимости» коммита.** Он переносит только изменения выбранного коммита. Если тот опирался на правки соседнего коммита, которого нет на целевой ветке, возможен конфликт.

<Detail summary="cherry-pick и rebase - близкие родственники">

И cherry-pick, и rebase делают одно и то же по сути: берут изменения коммита и заново применяют их на новом основании трёхсторонним слиянием. Rebase - это, грубо говоря, серия cherry-pick подряд: «перенеси все мои коммиты поверх той вершины». Cherry-pick - точечная версия: «перенеси вот этот один». Поэтому у них общие свойства: новый коммит получает новый идентификатор, а при пересечении правок возможен конфликт, разрешаемый так же, как при merge. Понимая одно, Вы понимаете оба.

</Detail>
