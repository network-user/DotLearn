Sooner or later every codebase reaches the question: "what did this look like yesterday?" A file gets rewritten, something breaks, and the old version is gone - unless someone saved it. Git is that someone: it keeps the history of a project, remembers every meaningful change, and lets you return to any point, compare versions, and run several lines of work in parallel without overwriting anyone else's effort.

**What you'll be able to do.** Save snapshots of a project as meaningful commits and read its history; create branches and merge them back; resolve merge conflicts without panic; undo mistakes safely; work with remote repositories (push, pull, fetch); rewrite history with rebase where it fits; and pick a workflow that suits your team.

**The route through this topic.** Twelve sections, each with a trainer where commands run in a real repository right in the browser:

1. **The version-control model** - why Git exists and how it thinks in snapshots, not edits.
2. **Your first commit** - `add`, `commit`, and the idea of the staging area.
3. **History** - `status`, `log`, `diff`: how to read what happened.
4. **Branches** - parallel lines of work and cheap switching between them.
5. **Merging** - bringing branches back together.
6. **Merge conflicts** - what to do when Git can't decide on its own.
7. **Undoing changes** - `reset`, `revert`, `checkout`, and when to use which.
8. **Remotes** - syncing with GitHub and teammates.
9. **Rebase** - a linear history and its price.
10. **Stash and cherry-pick** - park your work and move a single commit.
11. **Tags and reflog** - release markers and Git's safety net.
12. **Workflows** - how it all comes together in team scenarios.

**What you need first.** Nothing to install - the repository runs right on the page. Basic comfort with files and the command line is enough; everything Git-specific we'll cover step by step, from your first commit to confident rebasing.
