# A branch is a moving pointer to a commit

You are building a new feature and halfway through an urgent production bug arrives. Changing unfinished code is scary because everything gets tangled. You want to set the feature aside, fix the bug calmly in a clean version, then return. That is exactly what branches are for: parallel lines of work in one repository that you switch between instantly.

By the end of this section you will understand why a branch in Git is not a copy of files but a cheap pointer; what `HEAD` is and how it moves; and you will be able to create branches, switch between them, and know exactly where you are.

## A branch is a label, not a folder

In heavyweight version control systems "create a branch" meant copying the whole project into a new directory, which was expensive and slow. In Git a branch is simply a **pointer** (a name referring to one commit). Creating a branch means writing one name; it is instant and weighs almost nothing.

Each commit knows its parent, so history is a linked list of snapshots. A branch merely marks one commit in that list as the "tip of a line of work". When you make a new commit on a branch, the branch pointer automatically moves forward to the fresh commit.

## HEAD: the "where am I" pointer

With several branches, Git needs to know which one you are on. `HEAD` handles that: a special pointer for "current position". Usually `HEAD` points to a branch (for example `main`), and the branch points to a commit. So when you commit, Git moves the branch `HEAD` looks at.

Creating a branch is `git branch <name>`. It only sets a new pointer on the current commit but **does not** switch to it: `HEAD` stays where it was.

```bash
git branch feature
```

To move to a branch there are two commands. The old universal one is `git checkout <name>`. The newer, narrower, clearer one is `git switch <name>`: it only switches branches and does not do ten other things like `checkout`.

```bash
git switch feature
```

Often you need to create a branch and switch immediately. Short forms exist: `git switch -c <name>` or `git checkout -b <name>`, both of which create a branch and move `HEAD` in one command.

```bash
git switch -c hotfix
```

After that `HEAD` points at `hotfix`, and new commits go there without touching `main`.

## Switching aligns the working tree with the branch snapshot

When you switch to a branch, Git replaces the working tree contents with the snapshot of the commit the branch points to. Files on disk change as if you returned to that moment in history. Work on different branches is truly isolated: commits on `feature` are not visible on `main` until you merge them.

Try it in the terminal. Create branch `feature` with `git switch -c feature`, add a file and commit. Then `git switch main` and `git status` show that the new file is gone, because it stayed on `feature`. Switch back to `feature`, and the file is there again. The built-in goal is to end on branch `feature` with two commits in history.

## When behavior surprises you

`git branch feature` creates a branch but does not move `HEAD`, so you stay where you were. A beginner expects to be "inside" the new branch, commits, and is surprised it went to `main`. The split is deliberate: `branch` creates, `switch`/`checkout` switches. If you want both at once, use `git switch -c`.

<Callout type="warning">`git branch feature` only creates a pointer but leaves you on the current branch. To switch to the new branch immediately use `git switch -c feature` (or `git checkout -b feature`).</Callout>

The second surprise is detached HEAD. If you check out a commit directly (`git checkout <hash>`), `HEAD` points at the commit itself, bypassing a branch. That is "detached HEAD": commits made here belong to no branch and are easily lost. More on that in [[tags-reflog]].

## Common mistakes

- **Thinking `git branch` switches.** It only creates a pointer. A commit after `git branch feature` (without switching) goes to the old branch.
- **Committing in detached HEAD unknowingly.** If `git status` says "HEAD detached", you are not on a branch. Before committing, create one: `git switch -c <name>`.
- **Confusing `git switch` and `git switch -c`.** Without `-c` Git expects an *existing* branch and errors if it is missing. With `-c` it creates a new one. Mix them up and you get "invalid reference".

<Detail summary="Why git switch appeared alongside git checkout">

Historically `git checkout` was overloaded: it switched branches, restored files, and detached HEAD onto a commit. One typo could wipe your file edits instead of switching branches. In 2019 (Git 2.23) the command was split: `git switch` only moves between branches, `git restore` only restores files. `checkout` remains for compatibility, but for new habits the narrow commands are cleaner, because there is less chance of doing the wrong thing by accident.

</Detail>
