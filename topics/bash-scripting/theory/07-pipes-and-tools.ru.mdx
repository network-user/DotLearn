# Маленькие команды становятся сильными, когда их соединяют в конвейер

Вера разбирает лог веб-сервера и хочет узнать, сколько разных IP-адресов заходило на сайт за день. По отдельности инструменты для этого простые: одна команда вырезает нужный столбец, другая сортирует, третья убирает повторы, четвёртая считает строки. Ни одна из них не умеет решить задачу целиком, но соединённые в цепочку они дают ответ одной строкой.

Философия bash именно в этом: много маленьких команд, каждая делает одно дело, а соединяет их конвейер. Чтобы пользоваться этим свободно, нужно понять три механизма: пайп `|`, перенаправление потоков (`>`, `>>`, `2>&1`) и подстановку команд `$( ... )`.

## Пайп соединяет вывод одной команды со входом другой

У каждой команды есть стандартный вывод, куда она печатает результат, и стандартный вход, откуда читает данные. Пайп `|` берёт стандартный вывод левой команды и подаёт его на стандартный вход правой, не создавая промежуточных файлов.

```bash
$ echo "Москва Казань Тверь" | wc -w
3
```

`echo` печатает строку из трёх слов, а `wc -w` вместо своего обычного входа получает эту строку через пайп и считает в ней слова. Ни echo, ни wc не знают друг о друге: их связал `|`.

## Перенаправление уводит поток в файл, а не на экран

Пайп соединяет команды между собой. Перенаправление связывает поток с файлом. Оператор `>` записывает стандартный вывод в файл, полностью стирая прежнее содержимое, а `>>` дописывает в конец.

```bash
$ echo "первая строка" > out.txt
$ echo "вторая строка" >> out.txt
$ cat out.txt
первая строка
вторая строка
```

Первая команда создала файл и положила в него строку. Вторая дописала, потому что стоит `>>`. Если бы вместо `>>` снова стоял `>`, файл был бы перезаписан и первая строка исчезла бы.

## Потоки вывода и ошибок разделены

У команды два независимых выходных потока: стандартный вывод (номер 1) для обычного результата и стандартный поток ошибок (номер 2) для сообщений об ошибках. По умолчанию оба идут на экран, но перенаправляются по отдельности. Запись `2>&1` означает «направь поток ошибок туда, куда сейчас указывает стандартный вывод».

```bash
$ ls file.txt missing.txt > out.txt 2>&1
```

Здесь стандартный вывод уже уведён в `out.txt`, поэтому `2>&1` отправляет ошибки в тот же файл. Порядок важен: в записи `2>&1 > out.txt` ошибки привязались бы к экрану раньше, чем вывод ушёл в файл, и в файл попал бы только обычный вывод. Чтобы просто выбросить ненужный вывод, его направляют в `/dev/null`.

## Подстановка команд превращает вывод в значение

Пайп передаёт данные другой команде, а подстановка `$( ... )` превращает вывод команды в текст прямо в строке скрипта. Чаще всего результат кладут в переменную.

```bash
$ today=$(date +%Y-%m-%d)
$ echo "Отчёт за $today"
Отчёт за 2026-07-13
```

`$(date +%Y-%m-%d)` выполняет `date`, забирает её вывод и подставляет как строку в присваивание. Тот же приём собирает цепочку в переменную: `count=$(grep -c ERROR app.log)` положит число строк с ошибками в `count`.

## Что вас удивит

Первый сюрприз в том, что `>` срабатывает ещё до запуска команды. Запись `sort file.txt > file.txt` не сортирует файл на месте, а обнуляет его: shell открывает `file.txt` на запись и стирает содержимое раньше, чем `sort` успевает его прочитать. Результатом будет пустой файл. Для сортировки на месте пишут во временный файл или используют `sort -o`.

Второй сюрприз про `uniq`: он убирает только соседние одинаковые строки, а не все повторы по файлу. Поэтому в связке почти всегда стоит `sort | uniq`: сначала сортировка ставит одинаковые строки рядом, и только потом `uniq` их схлопывает. Без предварительного `sort` разбросанные по файлу дубликаты останутся.

## Типичные ошибки

**cmd > file, когда file это же вход.** `sort data.txt > data.txt` уничтожает файл, потому что перенаправление обнуляет его до запуска команды. Пишите в другой файл или в `sort -o data.txt data.txt`.

**uniq без sort.** `uniq` схлопывает только соседние повторы. Чтобы убрать все дубликаты, сначала ставят `sort`, а затем `uniq`.

**Перепутать | и >.** `command | file.txt` пытается запустить `file.txt` как команду и обычно падает с «command not found». В файл уводят через `>`, а на вход другой команды через `|`.

**Неверный порядок 2>&1.** `2>&1 > file` не отправит ошибки в файл: к моменту `2>&1` стандартный вывод ещё указывает на экран. Правильный порядок это `> file 2>&1`.

<Detail summary="Почему переменная, изменённая внутри пайпа, не сохраняется">
Каждая команда в конвейере запускается в отдельном подпроцессе. Поэтому `echo "x" | read var; echo "$var"` во многих оболочках напечатает пустую строку: `read` изменил `var` в подпроцессе, который завершился, а внешняя переменная осталась нетронутой. Когда результат конвейера нужен в переменной, его берут через подстановку команд, например `var=$(echo "x")`, а не через пайп в `read`.
</Detail>
