# Строка, которая спасает скрипт от самого себя

Анна писала скрипт ночной ротации бэкапов уже год, и он ни разу не подводил. Внутри была строчка `rm -rf "$backup_dir/"`, которая перед созданием нового архива подчищала временную папку. Потом сосед по команде переименовал переменную окружения в конфиге, а строчку в скрипте поправить забыл. `$backup_dir` тихо стала пустой, кавычки честно сложили пустую строку и слэш, и команда превратилась в `rm -rf "/"`. Скрипт запустился по расписанию в три часа ночи, и только алерт мониторинга дискового пространства поднял Анну с постели раньше, чем диск опустел до конца.

Никакой экзотики в этой истории нет: опечатка в имени переменной, ни одна команда bash не вывела ни строчки об ошибке, скрипт просто исполнил то, что было написано. Защититься от целого класса таких историй можно одной строкой в начале файла: `set -euo pipefail`. Разберём, что именно ловит каждая из трёх опций и почему привычка ставить эту строку окупается уже на первом сэкономленном инциденте.

## Три опции, три вида прощения, которые отключаются

По умолчанию bash прощает почти всё. Команда упала с ошибкой: неважно, скрипт бежит дальше. Обратились к переменной, которую забыли объявить: bash молча подставит пустую строку. Пайп из нескольких команд, средняя из которых упала: код возврата всей цепочки возьмётся из последней команды, будто ничего не случилось. Строка `set -euo pipefail` включает сразу три опции, каждая из которых отключает одно из этих прощений:

- `-e` (errexit): если очередная команда завершается ненулевым кодом, скрипт немедленно останавливается вместо перехода к следующей строке.
- `-u` (nounset): обращение к необъявленной переменной становится ошибкой, а не пустой строкой. Именно эта опция поймала бы опечатку из истории Анны.
- `-o pipefail`: код возврата пайпа берётся из первой упавшей команды в цепочке, а не только из последней.

На рисунке виден один и тот же сценарий деплоя в двух режимах. Без `set -e` неудачный `tar` печатает сообщение об ошибке и тонет среди остальных строк вывода, а скрипт как ни в чём не бывало доходит до удаления старых бэкапов. С `set -e` первая же ошибка обрывает выполнение, и опасный шаг просто не наступает.

## `-e`: скрипт умирает на первой ошибке

```bash
$ cat deploy.sh
#!/usr/bin/env bash
set -e

echo "собираем архив..."
tar -czf backup.tar.gz /no/such/dir
echo "копируем на сервер..."
```

```bash
$ bash deploy.sh
собираем архив...
tar: /no/such/dir: Cannot stat: No such file or directory
```

Строка «копируем на сервер...» не печатается вовсе: `tar` вернул ненулевой код, и `set -e` тут же завершил скрипт. Без этой опции скрипт напечатал бы обе строки и продолжил бы работать так, будто архив создан.

## `-u`: опечатка в имени переменной становится ошибкой

```bash
$ cat rotate.sh
#!/usr/bin/env bash
set -u

backup_dir="/srv/backups"
echo "чищу $bakup_dir"
```

```bash
$ bash rotate.sh
rotate.sh: line 5: bakup_dir: unbound variable
```

Опечатка `bakup_dir` вместо `backup_dir` без `set -u` тихо превратилась бы в пустую строку, и следующая за ней команда вроде `rm -rf "$bakup_dir/"` разнесла бы корень диска в точности как в истории Анны. С `set -u` bash останавливается прямо на обращении к необъявленной переменной, до того как пустота попадёт в опасную команду.

## `-o pipefail`: код пайпа перестаёт врать

По умолчанию код возврата пайпа берётся только из последней команды в цепочке, что бы ни случилось с предыдущими:

```bash
$ false | true
$ echo $?
0
```

`false` честно вернула единицу, но она стоит не последней, а `true` в конце вернула ноль, и именно этот ноль стал кодом всего пайпа. Скрипт, который проверяет `$?` после такого пайпа, решит, что всё прошло успешно, хотя первая команда упала. `pipefail` меняет это правило:

```bash
$ set -o pipefail
$ false | true
$ echo $?
1
```

Теперь код пайпа берётся из первой упавшей команды в цепочке, и он равен единице, хотя последняя команда всё так же завершилась успешно. Для скриптов, которые собирают код возврата пайпа для проверки ошибок, разница между этими двумя числами и есть разница между «заметили проблему» и «пропустили её». На практике это часто выглядит как `команда_сборки | tee build.log`: без `pipefail` провал самой сборки останется незамеченным, потому что `tee` почти всегда завершается успешно, и именно её код возврата увидит скрипт.

## Кавычки вокруг переменных обязательны

Вернёмся к рецепту из истории Анны: опасность была не только в опечатке, но и в том, что `rm -rf "$backup_dir/"` вообще способен превратиться в `rm -rf "/"`, если переменная пуста. Кавычки здесь не защита от этого сценария, они защищают от другой беды: словоделения и раскрытия шаблонов имён файлов.

```bash
$ dir="Мои документы"
$ ls $dir
ls: cannot access 'Мои': No such file or directory
ls: cannot access 'документы': No such file or directory
```

Без кавычек bash разбивает значение `$dir` на отдельные слова по пробелам и передаёт `ls` два разных аргумента, ни один из которых не существует. То же самое случилось бы, если бы в значении переменной оказалась звёздочка: без кавычек bash попытался бы раскрыть её как маску имён файлов.

```bash
$ ls "$dir"
Мои документы:
отчёт.txt
```

Раскрытие шаблонов работает так же незаметно, только в обратную сторону: значение с точки зрения переменной остаётся одной строкой, а команда получает список файлов.

```bash
$ pattern="*.tmp"
$ rm $pattern
```

Если в текущем каталоге найдётся хоть один файл с расширением `.tmp`, незакавыченная `$pattern` раскроется в список всех таких файлов, и `rm` удалит их все разом, хотя переменная хранила один шаблон, а не список. Закавыченная запись `rm "$pattern"` вернула бы ожидаемую ошибку «файл `*.tmp` не найден», потому что звёздочка внутри кавычек так и остаётся буквальным символом, а не маской. Закавыченная переменная всегда передаётся дальше одним целым аргументом, даже если внутри пробелы, звёздочки или пустая строка. Привычка `"$var"` вместо `$var` должна быть автоматической, а не решением по ситуации.

## `trap`: гарантированная уборка

Опции выше останавливают скрипт при первой проблеме, но резкая остановка сама по себе может оставить мусор: недописанный временный файл, незакрытое соединение. Команда `trap` регистрирует код, который выполнится при выходе из скрипта, независимо от причины выхода.

```bash
#!/usr/bin/env bash
set -euo pipefail

tmp=$(mktemp)
trap 'rm -f "$tmp"' EXIT

echo "данные" > "$tmp"
cat "$tmp"
```

Команда `trap 'rm -f "$tmp"' EXIT` выполнится и если скрипт дошёл до конца успешно, и если его оборвал `set -e` на середине. Временный файл будет удалён в любом случае, и это ровно та уборка, которую сложно гарантировать одними только проверками кодов возврата.

## Что вас удивит

`set -e` не срабатывает там, где код возврата команды проверяется намеренно: внутри условия `if`, внутри `while`, и по обе стороны от `&&` и `||`. Это не баг, а осознанное решение: если бы `set -e` обрывал скрипт прямо на проверке условия, никакое ветвление по коду ошибки было бы невозможно написать.

```bash
set -e
if grep "ERROR" log.txt; then
  echo "нашли ошибки в логе"
fi
echo "скрипт продолжается"
```

Даже если `grep` не найдёт ни одной строки с `ERROR` и вернёт единицу, скрипт не остановится: код возврата здесь проверяет `if`, и `set -e` сознательно отступает в сторону. Ровно то же самое происходит внутри `while условие; do ... done`, а также в `команда && следующая` и `команда || следующая`: код возврата используется, а не считается сигналом к аварийному завершению.

Второй сюрприз касается пайпов, и именно он объясняет, зачем `pipefail` нужен рядом с `-e`, а не вместо неё. Сама по себе `-e` проверяет только итоговый код возврата пайпа, а без `pipefail` этот код берётся из последней команды в цепочке. Значит, под одним `set -e` команда `false | true` скрипт не остановит: с точки зрения `-e` пайп завершился нулём, хотя первая команда в нём упала. Только вместе `set -e` и `set -o pipefail` действительно останавливают скрипт на любой упавшей команде внутри пайпа; по отдельности каждая из опций закрывает лишь часть картины.

## Типичные ошибки

**Полагают, что `set -e` ловит абсолютно всё.** Ошибка внутри `if`, `while`, `&&` или `||` не остановит скрипт, потому что там код возврата проверяется намеренно. Если нужно остановить скрипт именно на такой команде, проверяйте её результат отдельной строкой после условия.

**Забывают, что `pipefail` не включён по умолчанию.** Скрипт читает `$?` сразу после пайпа и верит, что ноль означает полный успех, хотя это может быть кодом только последней команды в цепочке. Без `set -o pipefail` такая проверка обманчива.

**Не закавычивают переменные с путями.** `rm -rf $dir/` при пустой или начинающейся с пробела `$dir` ведёт себя непредсказуемо: словоделение может превратить один аргумент в несколько неожиданных. Кавычки `"$dir/"` не решают проблему пустой переменной, но убирают отдельный класс сюрпризов.

**Ставят `set -euo pipefail` и считают дело сделанным.** Защитная строка ловит целый класс ошибок, но не заменяет проверку критичных операций руками: перед `rm -rf` в переменной с путём разумно дополнительно убедиться, что она не пуста, например конструкцией вида `: "${dir:?путь не задан}"`.

**Регистрируют `trap` слишком поздно.** Если временный файл создаётся раньше, чем объявлен `trap 'уборка' EXIT`, а следующая же команда падает под `set -e`, скрипт завершится раньше, чем уборка вообще будет зарегистрирована, и мусор останется на диске. `trap` стоит объявлять сразу после создания ресурса, который он должен убирать, а не в конце файла.

<Detail summary="Откуда взялось выражение «bash strict mode»">
Сочетание `set -euo pipefail` в начале скрипта иногда называют «строгим режимом bash» (bash strict mode). Термин не встроен в сам bash, это неофициальное соглашение, которое распространилось в основном благодаря статье разработчика Aaron Maxwell в 2014 году. Там же к трём опциям иногда добавляют четвёртую строку, `IFS=$'\n\t'`, которая сужает список разделителей слов до перевода строки и табуляции, убирая пробел из числа символов, по которым bash режет незакавыченные значения. Это уже более специфичная мера: она полезна при построчной обработке файлов с именами, содержащими пробелы, но не является частью базовой защиты и способна неожиданно повлиять на код, который полагался на разделение по пробелу. По этой же причине опытные авторы скриптов добавляют `IFS=$'\n\t'` далеко не всегда: три опции из `set -euo pipefail` почти никогда не ломают рабочий код, а вот смена `IFS` меняет поведение словоделения глобально для всего файла и требует более внимательного тестирования.
</Detail>
