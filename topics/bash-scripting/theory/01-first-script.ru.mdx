# Из истории команд рождается скрипт

Каждое утро Борис делает одно и то же: архивирует папку с отчётами, копирует архив на второй диск и пишет строчку в лог. Три команды, набранные вручную, каждый раз в одном и том же порядке. Рано или поздно палец промахивается, архив уезжает не туда, а лог остаётся пустым. Напрашивается очевидное: сложить эти три команды в файл и запускать файл целиком. Такой файл и называется скриптом, а bash его послушно исполнит строку за строкой.

Но если просто сохранить команды в `backup.sh` и попробовать запустить, терминал ответит холодным `Permission denied`. Пустой на первый взгляд отказ скрывает два разных механизма, которые должны сойтись, чтобы текстовый файл стал исполняемой программой. Разберём оба, и `Permission denied` больше не застанет вас врасплох.

## Скрипт это текстовый файл плюс два разрешения

Сам по себе скрипт не содержит никакой магии: это обычный текстовый файл, в каждой строке которого записана команда. Чтобы система согласилась запустить его как программу, нужны две вещи. Первая: строка `shebang` в самом начале файла, которая сообщает ядру, каким интерпретатором читать остальные строки. Вторая: бит исполнения в правах файла, который разрешает его запуск. Ни имя, ни расширение `.sh` роли не играют: решают именно эти два условия.

## Создаём и запускаем

Начнём с самого файла. Пусть он печатает одну строку, чтобы было видно, что скрипт вообще отработал:

```bash
$ cat backup.sh
#!/usr/bin/env bash
echo "бэкап готов"
```

Первый способ запуска самый прощающий: назвать интерпретатор явно и передать ему файл как аргумент.

```bash
$ bash backup.sh
бэкап готов
```

Здесь bash сам открывает файл и читает его строки. Биту исполнения взяться неоткуда не нужно, и даже строка shebang в этом режиме не важна: интерпретатор вы уже указали руками. Второй способ выглядит короче и именно его вы встретите в чужих инструкциях:

```bash
$ ./backup.sh
bash: ./backup.sh: Permission denied
```

Отказ закономерен: файл ещё не помечен как исполняемый. Бит исполнения ставит команда `chmod` из темы про права доступа:

```bash
$ chmod +x backup.sh
$ ./backup.sh
бэкап готов
```

Теперь запуск проходит. При вызове `./backup.sh` система сначала проверяет бит исполнения, затем читает shebang и по нему понимает, что файл нужно скормить bash.

<Callout type="tip" title="Зачем перед именем стоит ./">
Голое `backup.sh` терминал искал бы в каталогах из переменной `PATH`, а текущий каталог туда обычно не входит, и вы получите `command not found`. Префикс `./` явно говорит: «файл лежит здесь, в текущем каталоге».
</Callout>

## Строка shebang: какой интерпретатор звать

Shebang это первые два символа файла, решётка и восклицательный знак, за которыми следует путь к интерпретатору. Две записи встречаются чаще всего:

```bash
#!/bin/bash
#!/usr/bin/env bash
```

Форма `#!/bin/bash` жёстко указывает путь. Она короче, но ломается там, где bash установлен в другом месте, а так бывает на части систем. Форма `#!/usr/bin/env bash` просит утилиту `env` найти bash по переменной `PATH`, поэтому переносится между машинами лучше и считается предпочтительной. Отдельно стоит запись `#!/bin/sh`: она зовёт не bash, а системную POSIX-оболочку, где часть удобных возможностей bash недоступна.

## Что вас удивит

Строка shebang обязана быть самой первой в файле, буквально с первого байта. Стоит вставить перед ней пустую строку или пробел, и ядро перестанет её распознавать: `#!` в третьей строке для него просто комментарий.

Второй сюрприз прячется в разнице между двумя способами запуска. Команда `bash backup.sh` игнорирует shebang полностью: интерпретатор задан явно, и содержимое первой строки уже не влияет ни на что. Поэтому файл с shebang `#!/usr/bin/env python3`, запущенный как `bash script.sh`, ядро отдаст на растерзание bash, а не Python, и посыплются странные ошибки. При запуске `./script.sh` всё наоборот: shebang решает всё.

<Callout type="warning" title="Скрипт из Windows и невидимый \r">
Файл, сохранённый в Windows с переводами строк CRLF, несёт на конце shebang невидимый символ возврата каретки. Ядро ищет интерпретатор по имени `bash\r`, не находит и отвечает загадочным `bad interpreter: No such file or directory`. Лечится переводом файла в формат LF, например утилитой `dos2unix`.
</Callout>

## Типичные ошибки

**Забыли `chmod +x`.** Запуск через `./script.sh` встречает `Permission denied`. Проверьте права командой `ls -l`: в первой колонке у исполняемого файла должны стоять буквы `x`. Нет их, добавьте бит исполнения.

**Запуск без `./`.** Команда `backup.sh` отвечает `command not found`, хотя файл рядом. Причина в `PATH`: текущего каталога в нём нет. Пишите `./backup.sh`.

**Shebang не в первой строке.** Пустая строка или комментарий над `#!` превращают shebang в обычный текст. При запуске через `./` система не понимает, кем исполнять файл. Поднимите shebang на самый верх.

<Detail summary="А что если бита исполнения нет, но очень надо запустить прямо сейчас">
Пока права не исправлены, скрипт всегда можно запустить через явный интерпретатор: `bash backup.sh` или `sh backup.sh`. Этот путь не требует бита исполнения и удобен, когда файл лежит на разделе, смонтированном без права на исполнение, или когда вы просто хотите быстро проверить чужой скрипт, не меняя его атрибутов.
</Detail>
