# if спрашивает не про true и false, а про код возврата команды

Скрипт install.sh у Веры копирует файл конфигурации, накатывает миграции и перезапускает сервис: три команды подряд, одна за другой, ровно так, как исполняются любые команды в скрипте без всяких проверок. Всё шло гладко, пока однажды конфиг не успели положить на сервер вовремя. Команда `cp` упала с ошибкой «No such file or directory», а следующая за ней `systemctl restart` отработала как ни в чём не бывало, подхватив пустые настройки. Скрипт ни разу не остановился спросить себя, получилось ли предыдущее действие: он просто шёл по строкам сверху вниз.

Чтобы скрипт умел проверять состояние системы и выбирать один из путей, нужен `if`. Синтаксис у него простой, но за ним прячется идея, без которой `if` выглядит магией: он не вычисляет булево выражение вроде «истина» или «ложь» в математическом смысле. Он запускает обычную команду и смотрит на её код возврата.

## if слушает код возврата, а не переменную

Каждая команда в Linux, завершаясь, оставляет после себя число: код возврата, exit code. Ноль означает успех, любое другое значение означает ту или иную неудачу. `if` берёт команду, запускает её и реагирует только на это число: вернула команда ноль, исполняется ветка `then`; вернула что-то другое, управление уходит к `else`, если он есть, или пропускает блок целиком.

```bash
if команда
then
  echo "команда вернула ноль"
fi
```

Самый частый пример выглядит так:

```bash
$ if [ -f config.yaml ]; then echo "конфиг на месте"; fi
конфиг на месте
```

Тут прячется первая неожиданность для новичков: квадратная скобка `[` в этой строке не пунктуация языка, а имя настоящей команды, синонимичной команде `test`. У неё, как у любой команды, есть код возврата, и `if` реагирует именно на него:

```bash
$ test -f config.yaml; echo "$?"
0
$ [ -f config.yaml ]; echo "$?"
0
```

Раз `[` это отдельная команда, ей, как и любой команде, нужны пробелы между аргументами. Запись `[-f config.yaml]` шелл прочитает как попытку вызвать команду с именем `[-f`, такой команды нет, и вы получите «command not found». А закрывающая `]` на самом деле тоже отдельный аргумент той же команды, и перед ней тоже обязателен пробел.

## Есть более современный вариант: [[ ]]

Bash предлагает второй вариант той же проверки: `[[ ]]`. Внешне похоже, но это не отдельная команда, а встроенная конструкция самого языка, и ведёт она себя иначе в нескольких важных местах.

## Операторы: числа сравнивают не так, как строки

И `[`, и `[[` понимают два разных набора операторов, и путать их нельзя. Числа сравнивают мнемоникой: `-eq` равно, `-ne` не равно, `-gt` больше, `-ge` больше или равно, `-lt` меньше, `-le` меньше или равно. Строки сравнивают привычными `=` и `!=`, плюс две проверки на пустоту: `-z` истина для пустой строки, `-n` истина для непустой.

```bash
$ age=20
$ if [ "$age" -ge 18 ]; then echo "можно"; fi
можно
```

```bash
$ role="admin"
$ if [ "$role" = "admin" ]; then echo "доступ разрешён"; fi
доступ разрешён
```

Отдельная группа операторов проверяет файлы: `-f` истина, если путь существует и это обычный файл, `-d` то же самое для каталога, а `-e` самый широкий из трёх: истина, если по этому пути вообще что-то есть, файл, каталог или ссылка.

```bash
$ if [ -d /var/log ]; then echo "каталог есть"; fi
каталог есть
```

Один `if` редко покрывает всю логику целиком. `elif` добавляет ещё одну проверку, если предыдущая не сработала, а `else` ловит всё, что не подошло ни под одно условие:

```bash
score=75
if [ "$score" -ge 90 ]; then
  echo "отлично"
elif [ "$score" -ge 60 ]; then
  echo "хорошо"
else
  echo "плохо"
fi
```

bash проверяет условия по порядку сверху вниз и останавливается на первом, которое вернуло ноль. До `else` дело доходит только тогда, когда ни одно из предыдущих условий не подошло.

## Что вас удивит

Первый сюрприз ждёт тех, кто привык не закавычивать переменные. Пусть `$path` внезапно окажется пустой строкой. Запись `[ $path = "config.yaml" ]` после подстановки превращается в `[ = "config.yaml" ]`: слева от `=` не остаётся ни одного слова, и `test` отвечает синтаксической ошибкой вместо честного «не равно». Именно поэтому в паре с `[` держат привычку писать `"$path"` в кавычках. `[[ $path = "config.yaml" ]]` того же сюрприза не боится: bash разбирает содержимое `[[ ]]` иначе и не раскладывает пустую переменную в ничто.

Второй сюрприз про числа с ведущим нулём. Пусть `x="05"`. Проверка `[ "$x" -eq 5 ]` возвращает истину: `-eq` приводит обе стороны к числам, а числа 05 и 5 это одно и то же значение. Но проверка `[ "$x" = "5" ]` возвращает ложь: `=` сравнивает строки посимвольно, а строки «05» и «5» разной длины и совпадают не полностью. Один и тот же `x` даёт два разных ответа: выбор оператора решает, что вообще сравнивается, число или текст.

## Типичные ошибки

**Пробел забыли рядом со скобкой.** `[$a -eq 1]` шелл ищет команду с именем `[$a`, не находит и отвечает «command not found». `[` это обычная команда, ей нужны пробелы вокруг каждого аргумента: `[ "$a" -eq 1 ]`.

**Переменную сравнили без кавычек.** `[ $status = ok ]` годами работает, пока однажды `$status` не окажется пустым, и синтаксис разваливается. Привычка писать `"$status"` в кавычках закрывает вопрос раз и навсегда.

**Число сравнили строковым оператором.** `[ "$count" = 10 ]` не сработает для значения `010`, хотя по смыслу это то же число: `=` смотрит на символы, а не на величину. Для чисел нужен `-eq`, а не `=`.

**Забыли закрыть блок словом fi.** Пропущенный `fi` в конце `if` обрывает интерпретатор ошибкой «unexpected end of file»: bash до последней строки скрипта ждёт закрывающее слово и не находит его.

<Detail summary="Когда цепочка elif разрастается, на помощь приходит case">
Пять и больше веток `elif` подряд читаются тяжело. Bash даёт для этого случая отдельную конструкцию `case`, которая сравнивает одно значение сразу с несколькими шаблонами и получается заметно компактнее длинной лестницы `elif`. Это отдельная тема, но полезно знать заранее, что длинная цепочка условий не единственный вариант.
</Detail>
