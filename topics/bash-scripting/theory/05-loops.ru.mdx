# Тело цикла делает то, что раньше делал copy-paste

Анна рассылает еженедельный отчёт по трём городам, Москве, Казани и Твери, и для каждого копирует один и тот же блок команд, меняя в нём только название города. Три копии ещё терпимо читать и поддерживать. Но список городов растёт до пятнадцати, а потом приходит требование забирать имена не из головы, а из файла, который каждую неделю меняется. Копипаст в этот момент превращается в источник ошибок: где-то забыли поменять имя, где-то вставили лишний пробел, а где-то оставили старую строку от прошлой недели.

Цикл решает эту задачу ровно: один блок команд, тело цикла, исполняется снова и снова, каждый раз с новым значением. bash даёт под это три конструкции: `for` перебирает уже готовый список, `while` повторяет тело, пока условие истинно, `until` делает то же самое, но пока условие ложно. Освоите одну хорошо, две другие дадутся почти бесплатно.

## Цикл берёт следующий элемент из источника, пока источник не кончится

За разными синтаксисами `for`, `while` и `until` прячется одна и та же механика: взять следующий элемент из некоторого источника, выполнить с ним тело цикла, повторить, остановиться, когда источник исчерпан. Для `for` источник это список слов после `in`. Для чтения файла построчно источником становится сам файл, а следующий элемент это очередная строка.

На рисунке видно, что происходит на каждом шаге: `read` забирает одну строку из файла, тело цикла её обрабатывает, а затем `read` тянется за следующей. Как только строки в файле кончаются, `read` возвращает ненулевой код возврата, и цикл останавливается сам, без отдельного счётчика.

## for перебирает готовый список

```bash
$ for city in Москва Казань Тверь
> do
>   echo "город: $city"
> done
город: Москва
город: Казань
город: Тверь
```

`for` берёт список слов после `in`, кладёт по одному в переменную `city` и на каждое значение исполняет тело между `do` и `done`. Список кончился, цикл завершается сам: отдельное условие для остановки не нужно, в отличие от `while`.

Числа перебираются точно так же, просто список состоит из чисел:

```bash
$ for n in 1 2 3
> do
>   echo "шаг $n"
> done
шаг 1
шаг 2
шаг 3
```

## while считает, пока условие живо; until наоборот

`while` проверяет условие перед каждым повторением тела, точно так же, как `if` проверяет его один раз. Пока условие в `[ ]` возвращает ноль, тело исполняется снова.

```bash
$ i=1
$ while [ "$i" -le 3 ]
> do
>   echo "$i"
>   i=$((i+1))
> done
1
2
3
```

Явное увеличение счётчика через `i=$((i+1))` здесь обязательно: `while` не двигает переменную самостоятельно, в отличие от `for` со списком. Забудете эту строку, условие `"$i" -le 3` останется истинным навсегда, и цикл не остановится сам.

`until` устроен как зеркало `while`: он повторяет тело, пока условие ложно, и останавливается, как только оно становится истинным. Формулировка «делай, пока не случилось» иногда читается естественнее, чем «делай, пока условие истинно», особенно в ожидании внешнего события вроде поднятия сервиса.

## Чтение файла построчно: главный практический приём

Самое частое применение цикла в реальных скриптах, обработка файла построчно, легко испортить одной интуитивно понятной, но неправильной записью:

```bash
$ for line in $(cat names.txt)
> do
>   echo "имя: $line"
> done
```

Такая запись сначала подставляет всё содержимое файла как один текст через `$(cat names.txt)`, а затем `for` раскладывает этот текст на слова по правилам словоделения: разделителями считаются пробелы, табуляции и переносы строк, все вперемешку. Если в файле есть строка «Анна Иванова», она развалится на два отдельных значения `line`, хотя в файле это одна строка с одним именем.

Правильный способ, показанный на рисунке выше, читает файл именно построчно:

```bash
$ while read -r line
> do
>   echo "имя: $line"
> done < names.txt
```

Тут `read -r line` за одну итерацию забирает из файла ровно одну строку целиком, включая пробелы внутри неё, и кладёт её в `line`. Оператор `< names.txt` в конце перенаправляет содержимое файла на вход всей конструкции `while`, а не отдельной команде внутри неё.

## Что вас удивит

Первый сюрприз в том, что `for line in $(cat file)` часто «работает» на простых файлах и ломается незаметно только тогда, когда в строке появляется пробел или файл содержит пустые строки. Пустая строка при словоделении просто исчезает, и её как будто никогда не было. Ошибка потому и живёт годами в чужих скриптах: на тестовых данных без пробелов она не проявляется вовсе.

Второй сюрприз про ключ `-r` у `read`. Без него `read` сама интерпретирует обратный слэш как экранирующий символ и съедает символы после него, поэтому строка с путём вроде `C:\Users\anna\notes.txt`, часто встречающаяся в списках, скопированных с Windows-машины, придёт в `line` искалеченной. `-r` выключает это поведение и передаёт строку как есть, поэтому в паре с `read` его пишут почти всегда.

## Типичные ошибки

**for line in $(cat file) вместо while read.** Работает, пока в строках нет пробелов, и внезапно ломается на первом же имени из двух слов. Причина: словоделение видит только слова, а не строки. Замените на `while read -r line; do ...; done < file`.

**Забыли увеличить счётчик.** `while [ "$i" -le 3 ]; do echo "$i"; done` без строки `i=$((i+1))` не остановится никогда: условие остаётся истинным вечно, а тело печатает одно и то же значение бесконечно.

**Забыли -r у read.** `while read line; do ...; done < file` без `-r` ломает строки с обратным слэшем, потому что `read` интерпретирует его как экранирующий символ и отбрасывает следующий за ним символ.

**Забыли закрыть цикл словом done.** Пропущенный `done` обрывает интерпретатор ошибкой «unexpected end of file»: bash до последней строки скрипта ждёт закрывающее слово для каждого открытого `for`, `while` или `until`.

<Detail summary="until это просто while наоборот">
Любой `until` можно переписать через `while`, просто инвертировав условие, и наоборот: `until [ "$i" -gt 3 ]; do ...; done` ведёт себя ровно так же, как `while [ "$i" -le 3 ]; do ...; done`. Выбор между ними чисто стилистический: берите тот вариант, чьё условие читается по-русски естественнее в конкретном месте скрипта.
</Detail>
