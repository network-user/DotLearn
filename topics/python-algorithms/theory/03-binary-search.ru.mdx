# Бинарный поиск: половина отсекается за один шаг

Когда вы ищете слово в бумажном словаре, вы не листаете его с первой страницы. Вы открываете примерно середину, смотрите, в какую половину попадает слово, и отбрасываете вторую половину целиком. Потом повторяете - и за несколько прыжков добираетесь до нужной страницы среди тысяч. Это и есть бинарный поиск: на каждом шаге зона поиска уменьшается вдвое. Линейный перебор миллиона элементов - это до миллиона сравнений; бинарный поиск находит элемент за два десятка. Но у этой скорости есть жёсткое условие, и весь раздел во многом про него.

К концу раздела вы сможете написать бинарный поиск без классических ошибок с границами, объяснить, почему его сложность O(log n), и понять, какое предусловие делает его вообще возможным.

## Предусловие: данные должны быть отсортированы

Бинарный поиск работает, только если список отсортирован. Вся его логика держится на одном умозаключении: «если средний элемент меньше искомого, то искомого нет во всей левой половине». Это умозаключение верно лишь тогда, когда слева гарантированно лежат меньшие значения, а справа - большие. На неотсортированных данных вывод ломается, и алгоритм пропустит элемент, который на самом деле есть.

<Callout type="warning">
Бинарный поиск на неотсортированном списке не «работает медленнее» - он работает неправильно, молча возвращая «не найдено» для существующих элементов. Сортировка - не оптимизация, а обязательное предусловие.
</Callout>

## Механика: три границы и середина

Алгоритм держит зону поиска двумя индексами: `lo` (левая граница) и `hi` (правая). В начале это весь список: `lo = 0`, `hi = len(a) - 1`. На каждом шаге берётся середина `mid = (lo + hi) // 2` и сравнивается с искомым:

- если `a[mid]` равен искомому - нашли, возвращаем `mid`;
- если `a[mid]` меньше искомого - искомое правее, сдвигаем `lo = mid + 1`;
- если `a[mid]` больше искомого - искомое левее, сдвигаем `hi = mid - 1`.

Зона `[lo, hi]` сжимается, пока не схлопнется. Если `lo` перешагнул `hi`, зона пуста - элемента нет.

```python
def binary_search(a, target):
    lo, hi = 0, len(a) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if a[mid] == target:
            return mid
        elif a[mid] < target:
            lo = mid + 1
        else:
            hi = mid - 1
    return -1
```

## Почему O(log n)

Каждый шаг отбрасывает половину оставшихся элементов. Из `n` остаётся `n/2`, потом `n/4`, и так далее до одного. Число делений `n` пополам до единицы - это и есть `log₂ n`. Для миллиона элементов это около 20 шагов, для миллиарда - около 30. Сравните с линейным поиском, которому в худшем случае нужно `n` сравнений.

## Потрогайте границы руками

Самый коварный класс ошибок в бинарном поиске - неверные границы: `<` вместо `<=`, `mid` вместо `mid + 1`, бесконечный цикл. Запустите код, затем поменяйте `while lo <= hi` на `while lo < hi` и поищите элемент, стоящий ровно в одиночной зоне (например, наибольший) - увидите, как он «теряется». Это лучший способ прочувствовать, почему граница именно `<=`.

## Граничные случаи

- **Пустой список.** `hi = -1`, цикл `lo <= hi` (`0 <= -1`) не выполнится ни разу - сразу `-1`. Корректно.
- **Элемент на краю.** Наибольший элемент стоит в одиночной зоне на последнем шаге; именно его теряют, если ошибиться в `<=` или в обновлении границ.
- **Элемента нет.** Зона схлопывается, `lo` перешагивает `hi`, возвращается `-1`. Важно, что границы двигаются на `mid ± 1`, а не остаются на `mid`, - иначе цикл зациклится.
- **Дубликаты.** Базовый бинарный поиск вернёт индекс какого-то из равных элементов, не обязательно первого. Поиск самого левого вхождения - отдельная задача (см. deep dive).

## Типичные ошибки

- **Забыть про предусловие сортировки.** На неотсортированных данных алгоритм возвращает неверный результат, а не просто медленный. Сортируйте или убедитесь, что вход уже упорядочен.
- **Обновлять границу как `lo = mid` вместо `lo = mid + 1`.** Если `mid` уже проверен и не подошёл, оставлять его в зоне нельзя - это приводит к бесконечному циклу, когда зона перестаёт сужаться.
- **Писать `while lo < hi` вместо `<=`.** Со строгим `<` одиночная зона `lo == hi` не проверяется, и элемент в ней теряется.
- **Сортировать ради единственного поиска.** Сортировка стоит O(n log n) - дороже, чем один линейный поиск за O(n). Бинарный поиск окупается, когда по одному отсортированному списку ищут много раз.

<Detail summary="Готовый бинарный поиск в стандартной библиотеке: модуль bisect">

На практике свой бинарный поиск пишут редко - в стандартной библиотеке есть модуль `bisect`. `bisect.bisect_left(a, x)` возвращает индекс самого левого места, куда можно вставить `x`, сохранив сортировку; это же и индекс первого вхождения `x`, если он есть. `bisect.insort(a, x)` вставляет элемент, сохраняя порядок. Эти функции написаны и проверены за вас и решают в том числе задачу «первое вхождение среди дубликатов». Понимать механику всё равно важно - но в боевом коде берите `bisect`. Детали - в [документации bisect](https://docs.python.org/3/library/bisect.html).

</Detail>
