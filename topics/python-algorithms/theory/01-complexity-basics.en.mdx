# How to compare two algorithms without running them

You've written a function that looks up a name in your contacts list. It works instantly on hundreds of contacts. Then the list grows to a million - and the page hangs for a second. The question is not “whether your computer is fast”, but *how the operating time increases as the data grows*. One algorithm will take a million steps on a million elements, another – only twenty. The difference is not in the hardware, but in the method of counting itself.

By the end of this section, you will be able to evaluate an algorithm by how its time and memory grow with input size, read O(1), O(log n), O(n), O(n²) records, and choose between two solutions the one that won't fall apart on big data.

## Count steps, not seconds

Seconds are deceptive: they depend on the processor, language, and machine load. Therefore, algorithms are compared not by time, but by the number of elementary operations - comparisons, assignments, calls to a cell - depending on the size of the input `n`. This dependence is written as a function of `n` and coarsened to the main term. The notation `O(f(n))` reads “of the order of f from n” and says: for large `n` the number of operations grows no faster than `f(n)`, up to a constant factor.

The coarseness is intentional. If one algorithm does `3n + 7` operations and the other does `100n`, both are considered O(n): as `n` grows, the linear form beats any constants. Constants are important in practice, but the first question is always about the *growth class*, not the factor.

## Four classes that occur most often

Most algorithms in this topic fit into four growth classes. Let's sort them from cheapest to most expensive.

**O(1) - constant time.** The number of operations does not depend on `n`. Accessing a list element by index, checking `len(xs)`, reading the first element - all this is O(1): a million elements or ten, the work is the same.

```python
def first(xs):
    return xs[0]
```

**O(n) - linear time.** Work is proportional to the size of the input. To find the maximum, you have to look at each element exactly once - double the input, double the work.

```python
def maximum(xs):
    best = xs[0]
    for x in xs:
        if x > best:
            best = x
    return best
```

**O(log n) is logarithmic time.** Each step discards *half* of the remaining data. On a million elements this is about twenty steps, on a billion - about thirty. This is how binary search works, which we will come to in the third section.

**O(n²) - quadratic time.** For each element, you go through all the others - a nested loop over the same data. On a thousand elements this is already a million operations. Bubble sort from the second section is exactly this.

```python
def has_duplicate(xs):
    for i in range(len(xs)):
        for j in range(i + 1, len(xs)):
            if xs[i] == xs[j]:
                return True
    return False
```

## Feel the difference with your hands

Run the code: it counts the number of times the inner loop body is executed for the linear and quadratic algorithm on the same `n`. Change `n` from 10 to 50 and watch the linear counter increase by a factor of 5, and the quadratic counter increase by a factor of 25. This is the difference between O(n) and O(n²), expressed in steps.

## Memory is considered the same

The algorithm has not only time complexity, but also **spatial** complexity - how much additional memory it requires above the input itself. Bubble sort rearranges elements directly in the original list and introduces only a couple of variables - this is O(1) additional memory. And if the algorithm created a copy of the entire list for each element, it would be O(n²) in memory. Time and memory are two independent axes: it can be fast, but voracious in memory, and vice versa.

## Worst, average and best case

The same function can work differently on different inputs. A linear search for a number in a list will find it in the first step if you're lucky (**best case**, O(1)), and will go through the entire list if there is no number at all (**worst case**, O(n)). When they say “algorithm complexity”, by default they mean the worst case - a guarantee that will not fail on bad data.

<Callout type="info">Big-O описывает верхнюю границу роста. Запись O(n) для поиска значит «не дольше, чем линейно», даже если на удачном входе всё закончится на первом шаге.</Callout>

## When the assessment deceives

Big-O ignores constants - and sometimes this is misleading. An O(n) algorithm with a huge constant can, on real, small data, lose to an O(n²) algorithm with a tiny one. A classic example: on a list of five elements, the difference between O(n log n) and O(n²) is imperceptible, but the overhead of a complex algorithm is noticeable.

```python
small = [3, 1, 2]
print(sorted(small))
```

For three elements, any sorting algorithm will work instantly - the choice of growth class begins to play a role only when `n` is large. Big-O answers the question “what will happen as `n` grows”, not “what is faster on specific five numbers”.

## Common mistakes

- **Add complexities instead of taking the maximum.** Two consecutive loops on `n` are O(n) + O(n) = O(n), not O(2n) as a separate class: the constant factor is discarded. But nested loops are multiplied: a loop through `n` inside a loop through `n` gives O(n²).
- **Confusing “fast on my laptop” with “good complexity.”** On a hundred elements, even O(n²) is instantaneous. The complexity appears in the growth of `n`, and not in one measurement.
- **Forget about the worst case.** “Usually finds it right away” is not a guarantee. If the algorithm degrades to O(n²) on a bad input, it is this number that determines its behavior under load.

<Detail summary="Why logarithm to base 2?">

When each step discards half the data, the number of steps is how many times `n` can be divided by 2 before one element remains. This is `log₂(n)`. For a million it is approximately 20, for a billion it is approximately 30.

```python
import math
print(math.log2(1_000_000))
```

Approximately `19.93` is printed. In big-O notation, the base of the logarithm is not written: changing the base is multiplying by a constant, and big-O discards constants, so O(log n) without specifying the base.

</Detail>
