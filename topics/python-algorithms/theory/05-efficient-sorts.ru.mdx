# Сортировки за O(n log n): merge и quick

Пузырьковая сортировка и сортировка вставками устроены просто, но упираются в стену. На списке из миллиона элементов их `O(n²)` разворачивается примерно в триллион операций, и это уже не доли секунды, а минуты ожидания. Умные сортировки берут ту же задачу и решают её за `O(n log n)`: тот же миллион элементов обходится примерно в двадцать миллионов операций, то есть почти в пятьдесят тысяч раз дешевле. Разница не в удачной реализации и не в константе, а в самом классе роста.

К концу раздела вы поймёте идею «разделяй и властвуй», научитесь сливать два отсортированных списка и разбивать список вокруг pivot, объясните, почему quicksort вырождается в `O(n²)` на уже упорядоченном входе, и разберётесь, почему стандартная сортировка Python собрана как гибрид. А заодно увидите, где старые квадратичные сортировки всё ещё оказываются лучшим выбором.

## Почему `O(n log n)` остаётся пределом

Любая сортировка, которая опирается только на сравнения элементов, вынуждена различать все возможные исходные порядки. Для списка из `n` различных элементов таких порядков `n!` штук. Каждое сравнение отвечает «да» или «нет» и в лучшем случае делит множество ещё возможных порядков пополам, поэтому чтобы сузить `n!` вариантов до одного, сравнений нужно хотя бы `log2(n!)`. По формуле Стирлинга эта величина растёт как `n log n`. Отсюда нижняя граница: ни один алгоритм, сортирующий сравнениями, не может гарантированно работать быстрее `Ω(n log n)`. Merge sort упирается в эту границу всегда, quicksort достаёт её в среднем. Откуда вообще берутся классы роста и почему константы отбрасывают, разобрано в разделе [[complexity-basics]]; более широкий взгляд собран в отдельной теме «Вычислительная сложность».

## Разделяй и властвуй

Обе быстрые сортировки строятся на одном приёме: разбить задачу на части поменьше, решить каждую, потом собрать ответ. Список из `n` элементов сортировать трудно, а список из одного элемента отсортирован сам собой. Merge sort доводит эту мысль до конца. Он делит список пополам, пополам делит каждую половину и так до одиночных элементов, а затем склеивает отсортированные куски обратно.

Деление доходит до дна за `log n` уровней: на каждом уровне список режется надвое, а вдвое сокращать длину можно ровно `log2 n` раз. Обратный подъём с той же глубиной, и на каждом уровне слияние трогает все `n` элементов один раз. `log n` уровней по `n` работы дают `O(n log n)`.

## Merge sort: механика слияния

Всё держится на одном шаге: слить две уже отсортированные последовательности в одну отсортированную. Держите по указателю на начало каждой из них и на каждом шаге забирайте меньший из двух передних элементов. Тот список, откуда взяли, сдвигает свой указатель; второй ждёт. Когда один список кончился, остаток второго дописывается целиком, потому что он и так отсортирован.

```python
def merge(left, right):
    result = []
    i = j = 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i])
            i += 1
        else:
            result.append(right[j])
            j += 1
    result.extend(left[i:])
    result.extend(right[j:])
    return result
```

Условие `left[i] <= right[j]` берёт нестрогое «меньше или равно» не случайно. При равных элементах оно предпочитает тот, что пришёл из `left`, то есть из левой (более ранней) половины. Так слияние сохраняет исходный относительный порядок равных ключей, и merge sort получается устойчивым. Замена на строгое `<` эту устойчивость сломала бы. Сама рекурсия поверх слияния коротка:

```python
def merge_sort(xs):
    if len(xs) <= 1:
        return list(xs)
    mid = len(xs) // 2
    left = merge_sort(xs[:mid])
    right = merge_sort(xs[mid:])
    return merge(left, right)
```

Цена устойчивости и простоты: merge sort не сортирует на месте. Слияние собирает новый список `result`, поэтому на каждом уровне живёт `O(n)` дополнительной памяти, а всего расход по памяти `O(n)`. Это его слабое место рядом с quicksort.

## Quicksort: разбиение вокруг pivot

Quicksort идёт от обратного. Merge sort делит вслепую пополам, а всю работу делает при слиянии. Quicksort вкладывает работу в деление, а сборка ему не нужна вовсе. Он выбирает опорный элемент, pivot, и одним проходом раскидывает список так, чтобы слева от pivot оказались элементы не больше него, справа не меньше. Этот шаг называется partition. После него pivot стоит на своём финальном месте, а две половины сортируются той же процедурой независимо друг от друга.

Схема Ломуто (Lomuto) делает partition в одном цикле. Указатель `i` отмечает границу зоны «уже меньших или равных pivot». Проходя список, каждый элемент не больше pivot меняем местами с границей и двигаем её вправо; в конце ставим pivot на границу обменом.

```python
def quicksort(xs):
    a = list(xs)

    def sort(lo, hi):
        if lo >= hi:
            return
        pivot = a[hi]
        i = lo
        for j in range(lo, hi):
            if a[j] <= pivot:
                a[i], a[j] = a[j], a[i]
                i += 1
        a[i], a[hi] = a[hi], a[i]
        sort(lo, i - 1)
        sort(i + 1, hi)

    sort(0, len(a) - 1)
    return a
```

Partition переставляет элементы прямо в списке, а рекурсия сортирует непересекающиеся зоны, поэтому quicksort работает на месте. Дополнительная память уходит только на стек рекурсии глубиной `O(log n)` при удачных разбиениях. За скорость и экономию памяти платят двумя вещами: quicksort неустойчив (обмены через весь список легко перекидывают равные элементы друг через друга) и уязвим к плохому выбору pivot.

## Худший случай: отсортированный вход

Выбор pivot решает всё. Если pivot каждый раз оказывается почти минимумом или почти максимумом, partition отрезает не половину, а один элемент. Тогда глубина рекурсии становится `n`, а суммарная работа `O(n²)`, как у пузырька. Именно это происходит с наивным «pivot равен последнему элементу» на уже отсортированном входе: последний элемент там всегда максимум, каждая зона ужимается на один, и «быстрая» сортировка внезапно оказывается медленной.

Запустите и сравните числа: на перемешанном входе сравнений заметно меньше, чем на упорядоченном, хотя элементов поровну. Лекарство известно: брать pivot случайно или как медиану трёх (первый, средний, последний). Тогда специально подобранного «плохого» входа для алгоритма не существует, и `O(n²)` остаётся крайне маловероятной невезучей случайностью, а не гарантией на отсортированных данных.

<Callout type="warning">Худший случай quicksort никуда не девается: он лишь становится маловероятным при случайном pivot. Merge sort честнее: его `O(n log n)` гарантирован при любом входе, ценой `O(n)` памяти.</Callout>

## Timsort: почему Python не пишет quicksort

Вызов `sorted()` или `list.sort()` в CPython запускает не merge и не quick, а Timsort, гибрид сортировки слиянием и вставками. Придуман он был Тимом Петерсом под реальные данные, а реальные данные редко бывают случайными: в них часто попадаются уже упорядоченные куски.

Timsort сначала находит в списке естественные упорядоченные подпоследовательности, которые называют runs (по-русски прогоны). Короткие runs он достраивает сортировкой вставками, потому что на маленьких почти упорядоченных отрезках вставки быстрее любой рекурсии. Затем runs сливаются между собой, как в merge sort. Когда сливают два очень разных по длине run, Timsort включает режим galloping: вместо поэлементного сравнения он бинарным поиском перепрыгивает через длинные участки, где один run заведомо весь меньше. Итог собирает лучшее из двух миров: устойчивость и гарантию `O(n log n)` в худшем случае от слияния, а на почти упорядоченных данных ускорение до `O(n)` от вставок и прогонов.

<Detail summary="Почему гибрид, а не чистый quicksort">

У quicksort два свойства, неприемлемых для стандартной библиотеки. Он неустойчив, а Python обещает устойчивую сортировку (на ней держится идиома сортировки по нескольким ключам подряд). И у него есть вход, дающий `O(n²)`, что открывает возможность намеренно замедлить программу подобранными данными. Timsort устойчив, гарантирует `O(n log n)` и вдобавок выигрывает на типичных частично упорядоченных данных. Поэтому CPython, V8 (JavaScript) и стандартная библиотека Java для объектов используют именно его семейство.

</Detail>

## Когда пузырёк и вставки всё ещё уместны

`O(n log n)` побеждает `O(n²)` только асимптотически, при больших `n`. У быстрых сортировок выше накладные расходы: рекурсия, выделение памяти, обмены через весь список. На коротком списке из десятка элементов эти расходы съедают выигрыш, и простая сортировка вставками обгоняет merge и quick за счёт крохотной константы. Именно поэтому Timsort достраивает короткие runs вставками, а не рекурсией.

Вторая ниша: почти упорядоченные данные. Сортировка вставками на таком входе делает `O(n)` работы, ведь каждый элемент уже почти на месте и сдвигается на пару позиций. Merge sort честно проделает свои `O(n log n)` независимо от порядка, а вставки воспользуются тем, что вход хорош.

## Типичные ошибки

- **Строгое `<` в слиянии.** Условие `left[i] < right[j]` при равных элементах предпочтёт правую половину и переставит равные ключи. Merge sort перестанет быть устойчивым. Берите `left[i] <= right[j]`.
- **Считать merge sort сортировкой на месте.** Слияние собирает новый список, поэтому нужна `O(n)` дополнительной памяти. Если память критична, а устойчивость не нужна, это аргумент за quicksort.
- **Наивный pivot на отсортированных данных.** Pivot равный первому или последнему элементу превращает quicksort в `O(n²)` ровно на упорядоченном входе, который на практике встречается чаще случайного. Берите случайный pivot или медиану трёх.
- **Ожидать от quicksort устойчивости.** Он переставляет равные элементы через весь список. Когда устойчивость нужна, берите merge sort или Timsort (`sorted`).

<Detail summary="Откуда берётся нижняя граница Ω(n log n)">

Представьте работу сортировки как дерево решений: каждый внутренний узел это сравнение двух элементов, каждый лист это один из возможных исходных порядков. Чтобы алгоритм всегда давал верный ответ, у дерева должно быть не меньше `n!` листьев (по числу перестановок). Двоичное дерево с `n!` листьями имеет высоту не меньше `log2(n!)`, а высота это и есть число сравнений в худшем случае. Поскольку `log2(n!) ≈ n log2 n`, никакая сортировка сравнениями не обгонит `n log n` гарантированно. Обойти границу можно, только отказавшись от сравнений: сортировки подсчётом и поразрядная работают за `O(n)`, но требуют, чтобы ключи были целыми числами из ограниченного диапазона.
