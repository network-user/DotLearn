# Binary search: half is cut off in one step

When you look up a word in a paper dictionary, you don't turn to the first page. You open roughly the middle, see which half the word goes into, and discard the other half entirely. Then you repeat - and in a few jumps you get to the desired page among thousands. This is a binary search: at each step, the search area is halved. Linear enumeration of a million elements means up to a million comparisons; binary search finds an element within two tens. But this speed has a strict condition, and the entire section is largely about it.

By the end of this section, you will be able to write a binary search without the classic bounds errors, explain why its complexity is O(log n), and understand what precondition makes it possible at all.

## Precondition: data must be sorted

Binary search only works if the list is sorted. His entire logic rests on one conclusion: “if the middle element is less than the desired one, then the desired one is not in the entire left half.” This conclusion is true only when smaller values ​​are guaranteed to lie on the left and larger values ​​on the right. On unsorted data, the output breaks and the algorithm will skip an element that actually exists.

<Callout type="warning">
Binary search on an unsorted list doesn't "run slower" - it doesn't work correctly, silently returning "not found" for existing elements. Sorting is not an optimization, but a mandatory precondition.
</Callout>

## Mechanics: three borders and a middle

The algorithm keeps the search area by two indexes: `lo` (left border) and `hi` (right). At the beginning this is the whole list: `lo = 0`, `hi = len(a) - 1`. At each step, the middle `mid = (lo + hi) // 2` is taken and compared with the desired one:

- if `a[mid]` is equal to the searched one - found, return `mid`;
- if `a[mid]` is less than the desired one, the desired one is to the right, shift `lo = mid + 1`;
- if `a[mid]` is greater than the desired one - the desired one is to the left, shift `hi = mid - 1`.

The `[lo, hi]` zone shrinks until it collapses. If `lo` has stepped over `hi`, the zone is empty - there is no element.

```python
def binary_search(a, target):
    lo, hi = 0, len(a) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if a[mid] == target:
            return mid
        elif a[mid] < target:
            lo = mid + 1
        else:
            hi = mid - 1
    return -1
```

Start searching for the number 7 in the list `[1, 3, 5, 7, 9, 11]` mentally: `lo=0, hi=5, mid=2`, `a[2]=5 < 7`, then `lo=3`. Then `mid=4`, `a[4]=9 > 7`, which means `hi=3`. Now `lo=hi=3`, `mid=3`, `a[3]=7` - found, return 3. Three comparisons instead of six: the zone was divided in half.

## Why O(log n)

Each step discards half of the remaining elements. From `n` there remains `n/2`, then `n/4`, and so on until one. The number of halvings of `n` to one is `log₂ n`. For a million elements this is about 20 steps, for a billion it is about 30. Compare with linear search, which in the worst case needs `n` comparisons.

## Touch the boundaries with your hands

The most insidious class of errors in binary search are invalid boundaries: `<` instead of `<=`, `mid` instead of `mid + 1`, infinite loop. Run the code, then change `while lo <= hi` to `while lo < hi` and look for an element that is exactly in the single zone (for example, the largest one) - you will see how it gets “lost”. This is the best way to get a feel for why the border is `<=`.

## Edge cases

- **Empty list.** `hi = -1`, the loop `lo <= hi` (`0 <= -1`) will not be executed even once - immediately `-1`. Correctly.
- **Element on the edge.** The largest element is in the single zone at the last step; it is precisely this that is lost if there is a mistake in `<=` or in updating boundaries.
- **There is no element.** The zone collapses, `lo` steps over `hi`, `-1` returns. It is important that the borders move by `mid ± 1` and not stay on `mid` - otherwise the loop will go into a loop.
- **Duplicates.** A basic binary search will return the index of some of the equal elements, not necessarily the first. Finding the leftmost occurrence is a separate task (see deep dive).

## Common mistakes

- **Forget about the sorting precondition.** On unsorted data, the algorithm returns an incorrect result, not just a slow one. Sort or make sure the input is already sorted.
- **Update the border as `lo = mid` instead of `lo = mid + 1`.** If `mid` has already been checked and does not match, you cannot leave it in the zone - this leads to an infinite loop when the zone stops shrinking.
- **Write `while lo < hi` instead of `<=`.** With strict `<` the single zone `lo == hi` is not checked and the element in it is lost.
- **Sort for a single search.** Sorting costs O(n log n) - more expensive than a single linear search at O(n). Binary search pays off when the same sorted list is searched many times.

<Detail summary="Ready-made binary search in the standard library: the bisect module">

In practice, you rarely write your own binary search - the standard library has the `bisect` module. `bisect.bisect_left(a, x)` returns the index of the leftmost place where `x` can be inserted, preserving the sort; this is also the index of the first occurrence of `x`, if there is one. `bisect.insort(a, x)` inserts an element, preserving order. These functions are written and tested for you and solve, among other things, the “first occurrence among duplicates” problem. Understanding the mechanics is still important - but in combat code, use `bisect`. Details are in [bisect documentation](https://docs.python.org/3/library/bisect.html).

</Detail>
