# O(n log n) sorts: merge and quick

Bubble sort and insertion sort are simple, but they hit a wall. On a list of a million elements their `O(n²)` unfolds into roughly a trillion operations, and that is no longer a fraction of a second but minutes of waiting. Smart sorts take the same task and solve it in `O(n log n)`: the same million elements costs about twenty million operations, nearly fifty thousand times cheaper. The difference is not a lucky implementation or a smaller constant, but the growth class itself.

By the end of this section you will understand the divide-and-conquer idea, know how to merge two sorted lists and how to partition a list around a pivot, be able to explain why quicksort degrades to `O(n²)` on already ordered input, and understand why Python's standard sort is built as a hybrid. You will also see where the old quadratic sorts still turn out to be the better choice.

## Why `O(n log n)` is the limit

Any sort that relies only on comparing elements has to tell apart every possible starting order. For a list of `n` distinct elements there are `n!` such orders. Each comparison answers yes or no and at best splits the set of still-possible orders in half, so to narrow `n!` variants down to one you need at least `log2(n!)` comparisons. By Stirling's formula that quantity grows as `n log n`. Hence the lower bound: no comparison-based sort can be guaranteed to run faster than `Ω(n log n)`. Merge sort reaches this bound always; quicksort reaches it on average. Where growth classes come from and why constants are dropped is covered in the [[complexity-basics]] section; a broader treatment lives in the separate Computational complexity topic.

## Divide and conquer

Both fast sorts rest on one trick: break the task into smaller parts, solve each, then assemble the answer. A list of `n` elements is hard to sort, but a list of one element is already sorted. Merge sort takes this thought to its conclusion. It splits the list in half, splits each half in half, and so on down to single elements, then glues the sorted pieces back together.

The split reaches the bottom in `log n` levels: at each level the list is cut in two, and you can halve a length only about `log2 n` times. The climb back up has the same depth, and at every level the merge touches all `n` elements once. `log n` levels times `n` work gives `O(n log n)`.

## Merge sort: the merge step

Everything hinges on one step: merge two already sorted sequences into one sorted sequence. Keep a pointer at the start of each and, at every step, take the smaller of the two front elements. The list you took from advances its pointer; the other waits. When one list runs out, the rest of the other is appended whole, because it is already sorted.

```python
def merge(left, right):
    result = []
    i = j = 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i])
            i += 1
        else:
            result.append(right[j])
            j += 1
    result.extend(left[i:])
    result.extend(right[j:])
    return result
```

The condition `left[i] <= right[j]` takes the non-strict "less than or equal" on purpose. On equal elements it prefers the one from `left`, that is, from the earlier (left) half. So the merge preserves the original relative order of equal keys, and merge sort comes out stable. Switching to a strict `<` would break that stability. The recursion on top of the merge is short:

```python
def merge_sort(xs):
    if len(xs) <= 1:
        return list(xs)
    mid = len(xs) // 2
    left = merge_sort(xs[:mid])
    right = merge_sort(xs[mid:])
    return merge(left, right)
```

The price of stability and simplicity: merge sort does not sort in place. The merge builds a fresh `result` list, so each level holds `O(n)` extra memory, and the total memory cost is `O(n)`. That is its weak point next to quicksort.

## Quicksort: partition around a pivot

Quicksort works the other way around. Merge sort splits blindly in half and does all the work during the merge. Quicksort puts the work into the split and needs no assembly at all. It picks a pivot element and, in a single pass, scatters the list so that everything to the left of the pivot is no greater than it and everything to the right is no smaller. This step is called partition. After it the pivot sits in its final place, and the two halves are sorted by the same procedure, independently of each other.

The Lomuto scheme does partition in one loop. A pointer `i` marks the boundary of the "already less than or equal to pivot" zone. Walking the list, every element no greater than the pivot is swapped with the boundary and the boundary moves right; at the end the pivot is put on the boundary by one swap.

```python
def quicksort(xs):
    a = list(xs)

    def sort(lo, hi):
        if lo >= hi:
            return
        pivot = a[hi]
        i = lo
        for j in range(lo, hi):
            if a[j] <= pivot:
                a[i], a[j] = a[j], a[i]
                i += 1
        a[i], a[hi] = a[hi], a[i]
        sort(lo, i - 1)
        sort(i + 1, hi)

    sort(0, len(a) - 1)
    return a
```

Partition rearranges elements right inside the list, and the recursion sorts non-overlapping zones, so quicksort works in place. The only extra memory goes to the recursion stack, `O(log n)` deep on good splits. You pay for the speed and the memory savings with two things: quicksort is not stable (swaps across the whole list easily leap equal elements past each other) and it is vulnerable to a bad pivot choice.

## The worst case: sorted input

The pivot choice decides everything. If the pivot is almost the minimum or almost the maximum every time, partition peels off one element instead of half. Then the recursion depth becomes `n`, the total work is `O(n²)`, same as bubble sort. This is exactly what happens with a naive "pivot is the last element" on already sorted input: the last element there is always the maximum, each zone shrinks by one, and the "quick" sort is suddenly slow.

Run it and compare the numbers: on shuffled input there are noticeably fewer comparisons than on ordered input, though the element count is the same. The cure is known: pick the pivot at random or as the median of three (first, middle, last). Then no specially crafted "bad" input exists for the algorithm, and `O(n²)` stays an unlikely stroke of bad luck rather than a guarantee on sorted data.

<Callout type="warning">Quicksort's worst case never disappears: it merely becomes unlikely with a random pivot. Merge sort is more honest: its `O(n log n)` is guaranteed on any input, at the cost of `O(n)` memory.</Callout>

## Timsort: why Python does not ship quicksort

A call to `sorted()` or `list.sort()` in CPython runs neither merge nor quick, but Timsort, a hybrid of merge sort and insertion sort. It was designed by Tim Peters for real data, and real data is rarely random: it often contains already ordered stretches.

Timsort first finds the natural ordered subsequences in the list, called runs. Short runs it extends with insertion sort, because on small nearly ordered stretches insertion beats any recursion. Then the runs are merged into each other, as in merge sort. When two very differently sized runs are merged, Timsort switches on galloping mode: instead of comparing element by element it uses binary search to jump over long stretches where one run is known to be entirely smaller. The result gathers the best of both worlds: stability and the guaranteed `O(n log n)` worst case from merging, plus a speed-up to `O(n)` on nearly ordered data from insertion and runs.

<Detail summary="Why a hybrid and not plain quicksort">

Quicksort has two properties unacceptable for a standard library. It is not stable, while Python promises a stable sort (the idiom of sorting by several keys in a row depends on it). And it has an input that produces `O(n²)`, which opens the door to deliberately slowing a program down with crafted data. Timsort is stable, guarantees `O(n log n)`, and on top of that wins on typical partially ordered data. That is why CPython, V8 (JavaScript), and the Java standard library for objects all use its family.

</Detail>

## When bubble and insertion still fit

`O(n log n)` beats `O(n²)` only asymptotically, at large `n`. The fast sorts carry higher overhead: recursion, memory allocation, swaps across the whole list. On a short list of a dozen elements this overhead eats the gain, and plain insertion sort outruns merge and quick thanks to a tiny constant. That is exactly why Timsort extends short runs with insertion, not recursion.

The second niche: nearly ordered data. Insertion sort on such input does `O(n)` work, since each element is already almost in place and shifts by a couple of positions. Merge sort honestly does its `O(n log n)` regardless of order, while insertion takes advantage of the input being good.

## Common mistakes

- **Strict `<` in the merge.** The condition `left[i] < right[j]` prefers the right half on equal elements and rearranges equal keys. Merge sort stops being stable. Use `left[i] <= right[j]`.
- **Treating merge sort as in-place.** The merge builds a new list, so it needs `O(n)` extra memory. If memory is critical and stability is not, that is an argument for quicksort.
- **A naive pivot on sorted data.** A pivot equal to the first or last element turns quicksort into `O(n²)` on exactly the ordered input that in practice appears more often than random. Use a random pivot or the median of three.
- **Expecting stability from quicksort.** It moves equal elements across the whole list. When you need stability, use merge sort or Timsort (`sorted`).

<Detail summary="Where the Ω(n log n) lower bound comes from">

Picture the work of a sort as a decision tree: every internal node is a comparison of two elements, every leaf is one of the possible starting orders. For the algorithm to always give the right answer, the tree must have at least `n!` leaves (one per permutation). A binary tree with `n!` leaves has height at least `log2(n!)`, and the height is the number of comparisons in the worst case. Since `log2(n!) ≈ n log2 n`, no comparison sort beats `n log n` in a guaranteed way. You can get around the bound only by giving up comparisons: counting sort and radix sort run in `O(n)`, but require the keys to be integers from a bounded range.

</Detail>
