# Как сравнить два алгоритма, не запуская их

Вы написали функцию, которая ищет имя в списке контактов. На сотне контактов она работает мгновенно. Потом список вырастает до миллиона - и страница висит секунду. Вопрос не в том, «быстрый ли у вас компьютер», а в том, *как растёт время работы при росте данных*. Один алгоритм на миллионе элементов сделает миллион шагов, другой - всего двадцать. Разница не в железе, а в самом способе считать.

К концу раздела вы научитесь оценивать алгоритм по тому, как его время и память растут с размером входа, читать записи вида O(1), O(log n), O(n), O(n²) и выбирать из двух решений то, что не развалится на больших данных.

## Считайте шаги, а не секунды

Секунды обманчивы: они зависят от процессора, языка, нагрузки на машину. Поэтому алгоритмы сравнивают не по времени, а по числу элементарных операций - сравнений, присваиваний, обращений к ячейке - в зависимости от размера входа `n`. Эту зависимость записывают как функцию от `n` и огрубляют до главного слагаемого. Запись `O(f(n))` читается «порядка f от n» и говорит: при больших `n` число операций растёт не быстрее, чем `f(n)`, с точностью до постоянного множителя.

Огрубление намеренное. Если один алгоритм делает `3n + 7` операций, а другой `100n`, оба считаются O(n): при росте `n` линейная форма побеждает любые константы. Константы важны на практике, но первый вопрос всегда о *классе роста*, а не о множителе.

## Четыре класса, которые встречаются чаще всего

Большинство алгоритмов в этой теме укладываются в четыре класса роста. Разберём их от самого дешёвого к самому дорогому.

**O(1) - константное время.** Число операций не зависит от `n`. Обращение к элементу списка по индексу, проверка `len(xs)`, чтение первого элемента - всё это O(1): миллион элементов или десять, работа одна и та же.

```python
def first(xs):
    return xs[0]
```

**O(n) - линейное время.** Работа пропорциональна размеру входа. Чтобы найти максимум, придётся посмотреть на каждый элемент ровно один раз - удвоили вход, удвоили работу.

```python
def maximum(xs):
    best = xs[0]
    for x in xs:
        if x > best:
            best = x
    return best
```

**O(log n) - логарифмическое время.** Каждый шаг отбрасывает *половину* оставшихся данных. На миллионе элементов это около двадцати шагов, на миллиарде - около тридцати. Так работает бинарный поиск, к которому мы придём в третьем разделе.

**O(n²) - квадратичное время.** Для каждого элемента вы проходите по всем остальным - вложенный цикл по тем же данным. На тысяче элементов это уже миллион операций. Пузырьковая сортировка из второго раздела именно такая.

```python
def has_duplicate(xs):
    for i in range(len(xs)):
        for j in range(i + 1, len(xs)):
            if xs[i] == xs[j]:
                return True
    return False
```

## Память считают так же

У алгоритма есть не только временная сложность, но и **пространственная** - сколько дополнительной памяти он просит сверх самого входа. Пузырьковая сортировка переставляет элементы прямо в исходном списке и заводит лишь пару переменных - это O(1) дополнительной памяти. А если бы алгоритм для каждого элемента создавал копию всего списка, это была бы O(n²) по памяти. Время и память - две независимые оси: бывает быстро, но прожорливо по памяти, и наоборот.

## Худший, средний и лучший случай

Одна и та же функция может работать по-разному на разных входах. Линейный поиск числа в списке найдёт его на первом шаге, если повезёт (**лучший случай**, O(1)), и пройдёт весь список, если числа нет вовсе (**худший случай**, O(n)). Когда говорят «сложность алгоритма», по умолчанию имеют в виду худший случай - гарантию, которая не подведёт на плохих данных.

<Callout type="info">Big-O описывает верхнюю границу роста. Запись O(n) для поиска значит «не дольше, чем линейно», даже если на удачном входе всё закончится на первом шаге.</Callout>

## Когда оценка обманет

Big-O игнорирует константы - и иногда это вводит в заблуждение. Алгоритм O(n) с огромной константой может на реальных, небольших данных проигрывать алгоритму O(n²) с крошечной. Классический пример: на списке из пяти элементов разница между O(n log n) и O(n²) незаметна, а накладные расходы сложного алгоритма - заметны.

```python
small = [3, 1, 2]
print(sorted(small))
```

Для трёх элементов любой алгоритм сортировки отработает мгновенно - выбор класса роста начинает играть роль только когда `n` велик. Big-O отвечает на вопрос «что будет при росте `n`», а не «что быстрее на конкретных пяти числах».

## Типичные ошибки

- **Складывать сложности вместо взятия максимума.** Два последовательных цикла по `n` - это O(n) + O(n) = O(n), а не O(2n) как отдельный класс: константный множитель отбрасывают. А вот вложенные циклы перемножаются: цикл по `n` внутри цикла по `n` даёт O(n²).
- **Путать «быстро на моём ноутбуке» с «хорошая сложность».** На сотне элементов даже O(n²) мгновенна. Сложность проявляется на росте `n`, а не на одном замере.
- **Забывать про худший случай.** «Обычно находит сразу» - не гарантия. Если на плохом входе алгоритм деградирует до O(n²), именно это число и определяет его поведение под нагрузкой.

<Detail summary="Почему именно логарифм по основанию 2">

Когда каждый шаг отбрасывает половину данных, число шагов - это сколько раз `n` можно поделить на 2, прежде чем останется один элемент. Это и есть `log₂(n)`. Для миллиона это примерно 20, для миллиарда - примерно 30.

```python
import math
print(math.log2(1_000_000))
```

Печатается примерно `19.93`. В нотации big-O основание логарифма не пишут: смена основания - это умножение на константу, а константы big-O отбрасывает, поэтому O(log n) без уточнения основания.

</Detail>
