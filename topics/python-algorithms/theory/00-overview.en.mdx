Your function searches for a contact in the list and works instantly - as long as there are a hundred names in the list. The list grows to a million, and the same page begins to freeze for a second with each search. The hardware has not changed, the code has not changed, but the user is waiting. The point is not whether your computer is slow: the point is how the chosen method of processing data behaves when there is a lot of data. This topic is about how to understand in advance, without running the code, which solution will survive growth and which will fall apart - and how to write the very techniques on which almost all search and ordering rests.

**What you will learn to do.** Evaluate an algorithm by how its time and memory grow with input size; read records of the form O(1), O(log n), O(n), O(n²) and choose the survivable solution from two; write sorting from scratch and explain its complexity; find an element in sorted data in tens of steps instead of a million; prove to yourself that the cycle really always works, and not just with proven examples.

**Theme route.** Four sections, each with a sandbox, where the code can be immediately changed and executed directly in the browser:

1. **Complexity** - we learn to compare two algorithms by the increase in the number of steps, and not by seconds on a specific machine.
2. **Bubble sort** - we write the simplest sort, swapping neighbors, and figure out why it is O(n²).
3. **Binary search** - we find an element by cutting off half the range at each step, and find out its strict precondition.
4. **Invariants and choice** - we formulate the cycle invariant and through it we consciously select an algorithm for the task.

**What you need beforehand** No need to install anything: Python runs right in your browser on this page. A basic familiarity with Python lists and loops is enough - everything else, from O notation to reasoning about correctness, will be covered along the way, moving from simple to complex.
