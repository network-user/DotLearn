# Invariants: why the algorithm works at all

It's easy to write a loop that seems to sort or search. Proving to yourself that it does this *always*, and not just on the three examples you've tested, is what separates code that works from code that will silently return nonsense one day. The tool for such a proof is a loop invariant: a statement that is true before each iteration. Understanding invariants turns “it seems to work” into “it works, and I know why,” and at the same time helps you choose the right algorithm for a specific task - the latter is the skill for which you learn sorting and searching.

By the end of the section, you will learn to formulate a loop invariant, reason about the correctness of an algorithm using it, and consciously choose between linear and binary search.

## What is a loop invariant

An invariant is a property that remains true throughout the loop: it is true before the first iteration and remains true after each iteration. If, in addition, after exiting the loop, the invariant together with the exit condition gives the desired result, the algorithm is correct. It sounds abstract, so let’s take a quick example: bubble sorting.

> Invariant of the outer loop of bubble sort: after `i` passes, the last `i` elements of the list are in their final places and are the largest.

Let's check three parts. **Start:** before the first pass `i = 0`, “the last 0 elements are in place” - truly trivial. **Save:** Each pass pushes the next largest element to the end of the unsorted zone, increasing the sorted tail by one. **Completion:** after `n - 1` passes there are `n - 1` elements in place, which means the last one (the only one remaining) too. The list is sorted. We haven't tested any specific list, but we have proven correctness for all of them.

<Callout type="tip">
An invariant is a bridge from “works in my examples” to “works always.” If you can't state your loop invariant, chances are you don't fully understand why it's correct.
</Callout>

## Binary search invariant

In binary search, the invariant explains why discarding half is safe.

> Invariant: if the desired element is in the list at all, then it is in the current zone `[lo, hi]`.

Before the start, the zone is the entire list, so the invariant is true. Each step discards only the half where the element obviously does not exist (sorting guarantees this), so what you are looking for, if it exists, remains in a narrowed zone - the invariant is preserved. When the zone becomes empty (`lo > hi`), the invariant says: “if there was an element, it would be here, but here is empty” - which means there is no element. Returning `-1` is correct. The same invariant explains why the boundary should move by `mid ± 1`: the checked `mid` is definitely not the one being sought, and leaving it in the zone means violating “we discard only where the element is not present”.

## Linear or binary: an informed choice

Binary search is faster, but requires sorted data. Linear is slower, but works on any list and does not assume anything. The choice depends on what you have on hand and how many times you search.

The key pitfall is to sort for the sake of a single search. Sorting costs `O(n log n)`, and linear search alone costs only `O(n)`. Sorting and then searching binary once is more expensive than simply going through the list linearly. Sorting pays off only when there are a lot of searches: once sorted for `O(n log n)`, then each of thousands of searches is cheap `O(log n)`.

## Touch the selection with your hands

Compare two approaches on one list. Linear search finds an element by brute force; binary requires the list to be sorted in advance. Swap `target` for the missing element and see how both honestly return "not found", but at a different cost.

<PyDemo
  title="Linear vs binary on one list"
  code={`def linear_search(a, target):
    for i, x in enumerate(a):
        if x == target:
            return i
    return -1

def binary_search(a, target):
    lo, hi = 0, len(a) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if a[mid] == target:
            return mid
        elif a[mid] < target:
            lo = mid + 1
        else:
            hi = mid - 1
    return -1

data = sorted([9, 1, 7, 3, 11, 5])
print(data)
print(linear_search(data, 7))
print(binary_search(data, 7))`}
/>

## Common mistakes

- **Do not formulate an invariant and check it “by eye”.** Three successful examples do not prove correctness. State a property that is true at each iteration - and check start, save, end.
- **Sort for the sake of one search.** `O(n log n)` sorting is more expensive than one linear `O(n)`. Binary search is only beneficial for many searches on a single list.
- **Apply binary search to data that is “almost sorted.”** “Almost” is not enough: one violation of the order can lead the search to the wrong half. Either the list is sorted or binary search is not applicable to it.
- **Consider linear search always bad.** For short or unsorted data, as well as for one-time searches, it is simpler, more reliable and often faster in practice.

<Detail summary="Correctness and completeness are two different issues">

An invariant proves that *if* the loop ends, the result will be correct - this is correctness. Separately, it is necessary to prove that the cycle will end at all - this is completeness. They are looking for a “variant” for it: a decreasing non-negative value. In binary search, this option is the size of the zone `hi - lo`: it strictly decreases at each step (because the boundaries move by `mid ± 1`) and cannot become negative indefinitely - the loop must stop. This is why the error `lo = mid` instead of `lo = mid + 1` is dangerous: it breaks not correctness, but completeness - the zone stops decreasing, and the loop goes in cycles.

</Detail>
