# Assignment that has a value

Ordinary `=` in Python returns nothing. You write `n = len(data)`, and that line simply binds the name `n` to a number: the line itself is not a value, and you cannot put it inside `if` or pass it to a function. So when you need to compute something, store the result in a variable, and check it right away, you end up with two lines: assign first, compare second. The `:=` operator, called the "walrus" in the community because it looks a bit like walrus eyes and tusks, merges those two steps into one expression.

By the end of this section you will understand how `:=` differs from `=`, what value it returns, where the syntax requires parentheses, and why it was designed as an expression rather than another form of ordinary assignment.

## Mental model: assignment that returns what was assigned

The core idea in one sentence: `name := expr` evaluates `expr`, binds the result to the name `name`, and **returns that same result** as the value of the whole expression. Ordinary `name = expr` does the first two steps, but not the third: an assignment line has no value and cannot be used as part of a larger expression.

Because of that difference, `=` and `:=` live in different places in a program. `=` is a *statement*: it occupies a whole line. `:=` is an *expression*: it embeds inside another expression, in an `if` condition, in a function argument, in `while`.

Look at the bottom line in the figure: `(n := len(data))` not only stores the length in `n`, it also "passes" that number outward. That is why you can put it directly in `if`, and the check sees the same value that just landed in the variable.

## First walkthrough: one computation instead of two lines

The classic walrus motivation is "compute, remember, check." Without it you get two lines where the name `value` repeats.

```python
value = compute()
if value > 10:
    print(value)
```

With `:=` the assignment moves inside the condition. The variable `value` remains available in the `if` body and below: the walrus does not create a temporary scope; it binds the name in the surrounding scope.

```python
if (value := compute()) > 10:
    print(value)
```

Both versions do exactly the same thing. The difference is that in the second, `compute()` and the result check sit together, and the name `value` appears once on the left of `:=`. That is the typical benefit: fewer repeats, computation and its check not split across lines.

<MarginNote label='Where "walrus" comes from'>The official name from PEP 572 is "assignment expression." The nickname "walrus operator" stuck because `:=`, if you squint, looks like walrus eyes and tusks.</MarginNote>

## Parentheses: where they are required

Most of the time you need to wrap `:=` in parentheses. This is not style but grammar: the language deliberately makes the walrus visually obvious and distinct from ordinary `=`. In the condition above, parentheses around `(value := compute())` are required: without them Python will not parse the line.

```python
total = (subtotal := 100) + (tax := 20)
print(subtotal, tax, total)
```

Prints `100 20 120`. Here `subtotal` and `tax` get their values right inside the arithmetic expression, each wrapped in its own parentheses. Without parentheses the expression will not compile. Remember a rough rule: if you are unsure whether parentheses are needed around `:=`, add them: they are almost never wrong.

<Callout type="warning">
`:=` cannot be used as a standalone statement: the line `x := 5` is a syntax error. For ordinary assignment on its own line, use `=`. The walrus only makes sense where its returned value is needed.
</Callout>

## Try the operator yourself

Run the code as-is: the walrus assigns `n` the string length and immediately returns that number into the condition. Then change the text in `text` to something short (for example `"hi"`) and run again: the condition becomes false and the `else` branch prints something different. Notice that `n` is available after `if` too: the walrus binds the name in the ordinary scope, not hidden inside the condition.

## Edge case that will surprise you: `:=` inside an f-string

You can put a walrus in unexpected places, for example right inside an f-string. The expression evaluates, the name binds, and the result is substituted into the string.

```python
import math

print(f"root: {(r := math.sqrt(2)):.3f}")
print(r)
```

Prints `root: 1.414` and then `1.4142135623730951`. The variable `r` survives the f-string and is available on the next line. Technically this works, but it reads poorly: binding a variable inside a format string is almost always a bad idea. Useful to know you *can* do it to understand other people's code, and almost never worth doing yourself.

## Another edge case: walrus and operator precedence

`:=` binds more weakly than almost all operators. So `x := a + b` assigns `x` the result of `a + b`, not just `a`. But without parentheses such an expression will not be accepted outside allowed positions anyway, so in practice you write `(x := a + b)` and precedence inside the parentheses is unambiguous.

```python
data = [3, 1, 4, 1, 5]
if (biggest := max(data)) > 4:
    print("max", biggest, "is greater than four")
```

Prints `max 5 is greater than four`. First `max(data)` is fully evaluated, then the result lands in `biggest`, and only then is it compared to `4`. Parentheses make the order clear and satisfy the grammar.

## Common mistakes

- **Confusing `:=` and `=` by meaning.** `=` is a statement with no value: a whole line. `:=` is an expression with a value that embeds inside another expression. If you do not need the returned value, use ordinary `=`.
- **Forgetting parentheses.** `if value := compute() > 10:` will not parse the way you expect, or will not parse at all. Wrap the walrus: `if (value := compute()) > 10:`.
- **Writing `:=` on its own line.** `x := 5` on its own line is a syntax error. The walrus is justified only when its value is passed somewhere: into a condition, an argument, or `while`.
- **Expecting a new scope.** The walrus does not hide the variable locally: the name stays in the surrounding scope and is available after the block. That is not a bug, but easy to forget.

<Detail summary="Why the walrus was added: the story of PEP 572">

PEP 572 was debated for a long time and heatedly: the argument over it was one reason Guido van Rossum stepped back as BDFL in 2018. The argument for: Python had long lacked a way to assign and use a value in one expression, which made code around `while`, stream reading, and filtering comprehensions verbose. The argument against: the new operator looks too much like `=` and encourages hard-to-read code.

The compromise is exactly the strict syntax you saw above: the walrus cannot stand alone as a statement, it requires parentheses in many positions, and the language deliberately blocks using it where ordinary `=` is clearer. The operator appeared in Python 3.8 (2019). Read the full motivation in [PEP 572](https://peps.python.org/pep-0572/).

</Detail>
