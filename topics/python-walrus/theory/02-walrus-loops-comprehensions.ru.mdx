# Где walrus реально окупается

Присваивание со значением выглядит как мелкое удобство, пока вы не наткнётесь на цикл чтения. Раньше, чтобы читать из файла, очереди или сети по кусочкам, приходилось писать «бесконечный» `while True` с проверкой и выходом в середине тела - читаешь блок, проверяешь, не пуст ли он, и только потом обрабатываешь. Получалась рваная конструкция, где условие выхода спрятано не в заголовке цикла, а где-то внутри. Walrus возвращает условие выхода туда, где ему место, - прямо в `while`.

К концу раздела Вы научитесь переписывать «цикл чтения до сигнального значения» через `while (chunk := read())`, использовать `:=` внутри `comprehension`, чтобы не вычислять одно и то же дважды, и распознавать, в каких именно местах comprehension walrus безопасен, а где - источник тонких ошибок с областью видимости.

## Ментальная модель: условие и его данные в одной точке

Цикл `while` проверяет условие *перед* каждой итерацией. Проблема классического цикла чтения в том, что данные, по которым решается «продолжать или нет», добываются внутри тела, а проверяются - в заголовке. Эти две точки разнесены, и между ними обычно стоит дублирующее «первое чтение» до цикла.

Walrus стягивает чтение и проверку в одну точку - в условие `while`. На каждой итерации сначала выполняется `chunk := read()` (читаем и запоминаем), затем результат проверяется как условие цикла. Тело получает уже готовый `chunk`.

## Разбор: цикл чтения до сигнального значения

Сравните две формы одного цикла, который читает строки, пока не встретит пустую. Сначала - без walrus: чтение приходится дублировать (одно до цикла, одно в конце тела) или городить `while True` с `break`.

```python
line = source.readline()
while line != "":
    process(line)
    line = source.readline()
```

Чтение `source.readline()` встречается дважды, и легко забыть обновить `line` в конце тела - получится бесконечный цикл. С walrus чтение происходит ровно один раз, прямо в условии.

```python
while (line := source.readline()) != "":
    process(line)
```

Тело сократилось до сути, а условие выхода («пока строка не пуста») снова читается в заголовке цикла. Это самый частый и самый бесспорный сценарий применения walrus.

<MarginNote label="Сигнальное значение">«Сентинел» (sentinel) - специальное значение, означающее «данные закончились»: пустая строка для `readline`, пустые байты для `read`, `None` от `input` через обёртку. Walrus особенно удобен с сентинелами.</MarginNote>

## Потрогайте цикл руками

Запустите код: цикл откусывает по два символа от строки `data`, пока она не опустеет. Присваивание `chunk := data[:2]` и проверка «не пусто ли» стоят в одном `while`. Поменяйте размер куска (вместо `data[:2]` возьмите `data[:3]`) или текст в `data` и запустите снова - проследите, как меняется число итераций, при том что условие выхода вы не трогали.

## Walrus в comprehension: не считать дважды

Второе сильное место walrus - `comprehensions`. Часто в comprehension нужно и отфильтровать элементы по результату дорогой функции, и положить этот же результат в итог. Без walrus функцию приходится звать дважды: один раз в условии `if`, второй - в выражении слева.

```python
def transform(x):
    return x * x - 1

result = [transform(x) for x in range(6) if transform(x) > 0]
print(result)
```

Печатается `[3, 8, 15, 24]`, но `transform` вызвана по два раза на каждый `x`. Walrus позволяет вычислить её один раз, запомнить в переменную и использовать и в условии, и в результате.

```python
def transform(x):
    return x * x - 1

result = [y for x in range(6) if (y := transform(x)) > 0]
print(result)
```

Печатается тот же `[3, 8, 15, 24]`, но `transform` вызвана ровно один раз на элемент. Обратите внимание на порядок: `if (y := transform(x))` стоит в части `if`, а в выражении слева используется уже готовый `y`. Walrus здесь экономит не только вызовы, но и убирает риск, что два вызова разойдутся (например, если функция со временем станет недетерминированной).

## Краевой случай, который вас удивит: переменная переживает comprehension

В Python 3 обычная переменная цикла внутри comprehension живёт в собственной невидимой области - после `[x for x in range(3)]` имя `x` снаружи не появляется. А вот имя, связанное через walrus внутри comprehension, **протекает в окружающую функцию** и остаётся доступным после неё. Это сознательное решение PEP 572, а не случайность.

```python
values = [y for x in range(4) if (y := x * 10) < 25]
print(values)
print(y)
```

Печатается `[0, 10, 20]` и затем `30`. Список оборвался на `25`, но `y` хранит последнее присвоенное значение - `30`, то самое, на котором условие стало ложным (`30 < 25` - ложь). Переменная `x`, напротив, после comprehension недоступна. Эта асимметрия удивляет почти всех: walrus-имя «вытекает», а имя счётчика - нет.

<Callout type="warning">
Утечка walrus-имени из comprehension - не баг, но и не повод на неё полагаться. Если вам после comprehension нужна переменная, надёжнее присвоить её явной строкой. Полагаться на «последнее значение из comprehension» - хрупко: оно зависит от того, где именно цикл оборвался.
</Callout>

## Типичные ошибки

- **Ставить `:=` в выражение-результат comprehension, а проверять в `if` голую функцию.** Тогда экономии нет: walrus должен стоять в `if`, а готовая переменная - использоваться слева. Порядок важен.
- **Оставлять старый `while True` с `break`, когда подходит `while (x := ...)`.** Если условие выхода - это «пока чтение не вернуло сентинел», walrus выражает его прямо в заголовке и читается лучше, чем разрыв в середине тела.
- **Полагаться на утёкшую из comprehension переменную как на документированный контракт.** Имя действительно остаётся, но опираться на его последнее значение легко по ошибке: оно зависит от точки обрыва. Нужна переменная после - присвойте её отдельно.
- **Звать дорогую функцию дважды по привычке.** Если в comprehension результат функции нужен и для фильтра, и для значения, walrus убирает второй вызов - это его прямое назначение.

<Detail summary="Почему walrus-имя протекает, а счётчик - нет">

В Python 3 у comprehension есть собственная неявная функция-обёртка - именно поэтому переменная цикла (`x` в `[... for x in ...]`) не загрязняет окружающую область: она локальна для этой скрытой функции. Walrus спроектировали иначе: его имя намеренно связывается в **ближайшей охватывающей обычной** области (функции или модуле), перепрыгивая через неявную обёртку comprehension. PEP 572 сделал так специально - чтобы walrus в comprehension мог, например, накапливать результат во внешней переменной. Цена решения - та самая асимметрия, где `x` локальна, а walrus-имя нет. Деталь описана в разделе «Scope of the target» PEP 572.

</Detail>
