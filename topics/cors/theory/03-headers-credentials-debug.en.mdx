# How to read a CORS error instead of guessing

"Access to fetch at ... has been blocked by CORS policy" — then someone copies the first config from a forum, sets `Access-Control-Allow-Origin: *`, hits a credentials error, and changes something else at random. That loop ends once you see that a CORS response has several headers, each answering one question, and the console text names the exact missing header.

This section covers `Access-Control-*` response headers, credentials (cookies and auth headers) and their special rule with `*`, and reading the console message as a diagnosis pointing at a specific config line.

## Mental model: each header answers one question

Server-side CORS is response headers; each closes one browser question. Not a magic config blob — know which header is about what.

`Access-Control-Allow-Origin` — who gets the response. `Allow-Methods` and `Allow-Headers` — preflight permissions. `Allow-Credentials` — cookies and credentialed requests. `Expose-Headers` — which response headers JavaScript may read. Know the mapping and debugging becomes a checklist, not guessing.

## Allow-Origin: who may read the response

The main header. Either a specific origin or `*`. The browser compares to request `Origin`; mismatch → script does not get the body.

```text
Access-Control-Allow-Origin: https://app.example.com
```

You cannot list multiple origins comma-separated — one origin or `*`. Multi-domain servers keep an allowlist, read request `Origin`, and if allowed **reflect** it back as the single value.

<Callout type="warning">
Reflect `Origin` only after allowlist check. Blindly echoing any `Origin` with `Allow-Credentials: true` effectively opens credentialed responses to anyone — a common misconfiguration bug.
</Callout>

## Credentials: cookies and the star rule

By default `fetch` to a foreign origin **does not send cookies** even if the server sets `Set-Cookie`. For cookies and auth headers cross-origin you need both sides:

```js
fetch("https://api.example.com/me", {
  credentials: "include",
})
```

Client asks with `credentials: "include"`. Server must respond:

```text
Access-Control-Allow-Credentials: true
```

And when credentials are involved, `Access-Control-Allow-Origin: *` is forbidden. The browser requires a **specific** origin. "Give a private cookie response to anyone" is nonsense, so with credentials the server must name the recipient.

All three must align: client `credentials: "include"`, server `Allow-Credentials: true`, and specific (not `*`) `Allow-Origin`.

## Expose-Headers: what JavaScript sees

Even when the response is readable, JS by default sees only a few "safe" response headers. Custom headers like `X-Total-Count` return `null` from `response.headers.get(...)` until the server lists them:

```text
Access-Control-Expose-Headers: X-Total-Count, X-Request-Id
```

"Visible in Network but `get` returns null" is not a fetch bug — missing `Expose-Headers`.

## Debugging: read the error as diagnosis

Console text almost always names the missing header:

Debug steps:

<Steps>
  <li>Open Network. If <code>OPTIONS</code> precedes the request — likely preflight: check its <code>Allow-Methods</code> and <code>Allow-Headers</code>.</li>
  <li>Read console text — it names the header: missing <code>Allow-Origin</code>, origin mismatch, <code>{'*'}</code> vs credentials, disallowed method or header.</li>
  <li>Compare request <code>Origin</code> to response <code>Allow-Origin</code> character by character — scheme, port, trailing slash.</li>
  <li>With <code>credentials: include</code> — ensure <code>Allow-Origin</code> is not <code>{'*'}</code> and <code>Allow-Credentials: true</code> is present.</li>
  <li>CORS is fixed on the **server** response, not client request headers (except asking for credentials).</li>
</Steps>

## Edge case that surprises people

A network error like `502 Bad Gateway` from a foreign origin also shows as CORS in the console — the error response lacked CORS headers. The browser hides the real status; your `fetch` sees a rejected promise without details. Sometimes "CORS error" means "server died and sent no CORS headers", not misconfigured CORS. Check server health first.

## Common mistakes

- **`Allow-Origin: *` with credentials.** Incompatible. Use specific origin + `Allow-Credentials: true`.
- **Comma-separated origins in `Allow-Origin`.** One origin or `*`. Use allowlist + reflect.
- **Fixing CORS on the client.** Permission comes only in response headers.
- **Forgetting `Expose-Headers` for custom response headers.**
- **Treating 502/504 as bad CORS config.** Dead server → no CORS headers → masked CORS error.

<Detail summary="Why CORS is fixed on the server">

Newcomers think "error in browser → fix browser code". But CORS is how the **data owner server** allows or denies reading its response. The browser only enforces; no `fetch` flag (except requesting credentials) can force access. If a public API sends no CORS headers, the browser cannot reach it directly — use your backend proxy. Local dev proxies do the same: accept on your origin, forward server-side.

</Detail>
