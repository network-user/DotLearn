# Как прочитать ошибку CORS вместо того, чтобы гадать

«Access to fetch at ... has been blocked by CORS policy» - и дальше человек идёт на форум, копирует первый попавшийся конфиг, ставит `Access-Control-Allow-Origin: *`, ловит новую ошибку про credentials, ставит ещё что-то наугад. Эта петля кончается, как только вы понимаете, что у CORS-ответа есть несколько разных заголовков, каждый отвечает за свой вопрос, а текст ошибки в консоли прямо называет, какого именно заголовка не хватило.

В этом разделе мы разберём заголовки `Access-Control-*`, которые сервер шлёт в ответе, отдельно поговорим про credentials (cookie и заголовки авторизации) и про их особое правило с `*`, а затем научимся читать сообщение в консоли как диагноз, указывающий на конкретную строку конфигурации.

## Ментальная модель: каждый заголовок отвечает на свой вопрос

CORS со стороны сервера - это набор ответных заголовков, и каждый закрывает ровно один вопрос браузера. Не нужно держать в голове «магический конфиг»: нужно понять, какой заголовок про что.

`Access-Control-Allow-Origin` - кому отдавать ответ. `Allow-Methods` и `Allow-Headers` - что разрешено в preflight. `Allow-Credentials` - можно ли слать cookie и читать ответ с ними. `Expose-Headers` - какие заголовки ответа ваш JavaScript вообще увидит. Запомнив, какой заголовок про что, вы превращаете отладку из угадывания в проверку по списку.

## Allow-Origin: кому отдавать ответ

Главный заголовок. Сервер кладёт в него либо конкретный origin, либо `*` (всем). Браузер сравнивает его со значением `Origin` запроса и, если не совпало, не отдаёт тело скрипту.

```text
Access-Control-Allow-Origin: https://app.example.com
```

Тонкость: в `Allow-Origin` нельзя перечислить несколько origin через запятую - это либо ровно один origin, либо `*`. Поэтому серверы, которым нужно обслуживать несколько доменов, держат список разрешённых origin у себя, читают заголовок `Origin` входящего запроса, проверяют его по списку и, если он допустим, **отражают** его обратно одним значением. Снаружи это выглядит так, будто сервер «подстраивает» `Allow-Origin` под каждого клиента.

<Callout type="warning">
Отражать `Origin` обратно нужно только после проверки по белому списку. Слепо возвращать в `Allow-Origin` любой пришедший `Origin` вместе с `Allow-Credentials: true` - это фактически открыть доступ к ответам с cookie кому угодно. Это распространённая уязвимость в самописных CORS-настройках.
</Callout>

## Credentials: cookie и особое правило со звёздочкой

По умолчанию `fetch` к чужому origin **не шлёт cookie и не отдаёт их из ответа** - даже если сервер вернул `Set-Cookie`. Чтобы cookie и заголовки авторизации ездили с кросс-доменным запросом, нужны две вещи одновременно: клиент должен попросить, сервер - разрешить.

```js
fetch("https://api.example.com/me", {
  credentials: "include",
})
```

`credentials: "include"` со стороны клиента включает отправку cookie. Со стороны сервера нужен ответный заголовок:

```text
Access-Control-Allow-Credentials: true
```

И тут вступает в силу правило, на котором спотыкаются чаще всего. Когда запрос идёт с credentials, `Access-Control-Allow-Origin: *` запрещён. Браузер требует **конкретный** origin, а не звёздочку. Логика прямая: «отдать ответ с приватными cookie кому угодно» - бессмыслица с точки зрения безопасности, поэтому при credentials сервер обязан назвать поимённо, кому именно можно.

Если все три условия не сойдутся - клиент с `credentials: "include"`, сервер с `Allow-Credentials: true` и конкретным (не `*`) `Allow-Origin`, - cookie либо не уедут, либо ответ не дойдёт до скрипта.

## Expose-Headers: что увидит ваш JavaScript

Ещё одна неочевидность. Даже когда ответ успешно прочитан, ваш JavaScript по умолчанию видит лишь несколько «безопасных» заголовков ответа (`Content-Type`, `Cache-Control` и ещё пара). Кастомные заголовки вроде `X-Total-Count` или `X-Request-Id` для `response.headers.get(...)` вернут `null`, пока сервер явно не разрешит их читать:

```text
Access-Control-Expose-Headers: X-Total-Count, X-Request-Id
```

Поэтому ситуация «в панели Network заголовок виден, а `response.headers.get("X-Total-Count")` даёт `null`» - это не баг fetch, а отсутствие `Expose-Headers`. Браузер заголовок получил, но скрипту его не показывает.

Алгоритм отладки сводится к нескольким шагам, и почти всегда сервер исправляется быстрее, чем гуглится:

<Steps>
  <li>Откройте вкладку Network и найдите запрос. Если перед ним есть <code>OPTIONS</code> - проблема, скорее всего, в preflight: смотрите его ответные заголовки <code>Allow-Methods</code> и <code>Allow-Headers</code>.</li>
  <li>Прочитайте текст ошибки в консоли - он называет конкретный заголовок: отсутствует <code>Allow-Origin</code>, не совпал origin, конфликт <code>\*</code> с credentials, не разрешён метод или заголовок.</li>
  <li>Сравните <code>Origin</code> запроса со значением <code>Allow-Origin</code> ответа символ в символ: легко ошибиться на схеме (http против https), порте или хвостовом слэше.</li>
  <li>Если используете <code>credentials: "include"</code> - убедитесь, что <code>Allow-Origin</code> не <code>\*</code>, а конкретный origin, и присутствует <code>Allow-Credentials: true</code>.</li>
  <li>Помните: чинит CORS сервер, а не клиент. Менять заголовки <strong>запроса</strong> в попытке обойти ошибку бесполезно - разрешение даёт сервер в заголовках <strong>ответа</strong>.</li>
</Steps>

## Краевой случай, который удивляет

Сетевая ошибка вроде `502 Bad Gateway` от чужого origin тоже выглядит в консоли как CORS-ошибка - потому что ответ с ошибкой пришёл без CORS-заголовков. Браузер не отдаёт его телу скрипта, а вместе с ним не показывает и реальный статус: для кросс-доменного запроса без `Allow-Origin` ваш `fetch` видит не «502», а просто отклонённый промис без деталей. Поэтому «CORS-ошибка» иногда означает «сервер упал и потому не прислал CORS-заголовки», а не «CORS настроен неверно». Проверять стоит и саму доступность сервера, а не только его CORS-конфиг.

## Типичные ошибки

- **`Allow-Origin: *` вместе с credentials.** Несовместимо по стандарту. С `credentials: "include"` сервер обязан вернуть конкретный origin и `Allow-Credentials: true`, иначе браузер отбросит ответ.
- **Перечислять несколько origin через запятую в `Allow-Origin`.** Заголовок принимает либо один origin, либо `*`. Для нескольких доменов проверяйте `Origin` по белому списку и отражайте подходящий одним значением.
- **Чинить CORS со стороны клиента.** Заголовки запроса разрешения не дают. Менять нужно ответ сервера. Клиент может лишь попросить credentials, всё остальное решает сервер.
- **Забыть `Expose-Headers` для кастомных заголовков ответа.** Без него `response.headers.get("X-...")` вернёт `null`, хотя в Network заголовок виден.
- **Принять `502`/`504` за неверный CORS-конфиг.** Упавший сервер не присылает CORS-заголовки, и ошибка маскируется под CORS. Сначала проверьте, что сервер вообще отвечает.

<Detail summary="Почему отладку CORS делают на сервере, а не на клиенте">

Интуиция новичка - «ошибка в браузере, значит чинить надо в браузерном коде». Но CORS - это механизм, которым **сервер-владелец данных** разрешает или запрещает чтение своего ответа. Браузер лишь исполнитель политики: он не может сам решить отдать вам чужой ответ, как бы вы ни настраивали `fetch`. Поэтому ни один заголовок запроса, ни один флаг `fetch` (кроме `credentials`, который лишь просит слать cookie) не «откроет» CORS - разрешение физически приходит только в ответе сервера. Отсюда практическое следствие: если чужой публичный API не отдаёт CORS-заголовки, из браузера к нему не достучаться напрямую вообще, и обходят это через свой backend-прокси, который ходит к API уже не из браузера и потому под same-origin policy не попадает. На локальной разработке ту же роль играет dev-proxy сборщика: он принимает запрос на ваш origin и переадресует на чужой API на стороне сервера.

</Detail>
