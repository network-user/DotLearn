# Why your JavaScript cannot go anywhere it wants

You open a tab with online banking, leave it open, and read news in another tab. Foreign JavaScript runs on the news page: ads, counters, widgets. What stops that code from quietly calling your bank, reading the balance response, and leaking it? You are logged in; bank cookies ride with the request automatically. If any script could read responses from any site where you are logged in, the web would be unusable for anything worth more than a meme.

Same-origin policy closes that hole: a baseline security rule in every browser. By the end of this section you will know what the browser counts as the "same origin" versus foreign, which operations the rule limits and which it does not, and why CORS is not "turning security off" but a controlled exception.

## Mental model: origin is a triple

Origin is not "domain" or "site" in everyday speech. The browser defines it strictly as three URL parts: **scheme** (`http` or `https`), **host** (`example.com`), and **port** (`443`, `80`, `8080`). Two URLs share an origin if and only if all three match. Path, query, and `#` fragment do not count.

Take `https://shop.example.com` (port 443 implied for https). Then `https://shop.example.com/products` is the same origin: path does not matter. But `http://shop.example.com` is foreign (different scheme). `https://api.example.com` is foreign (different host). `https://shop.example.com:8443` is foreign (different port). The browser ignores that one company owns all these addresses; it compares three fields, and any mismatch makes origins different.

<MarginNote label="origin vs site">There is a related but broader notion called "site": it ignores port and collapses subdomains to registrable domain (`example.com`). Some cookie mechanisms (`SameSite`) use site. Same-origin policy uses origin (the triple), not site. Do not confuse them.</MarginNote>

## What same-origin policy actually limits

The nuance almost everyone trips on: same-origin policy does **not** forbid **sending** a cross-origin request. The browser will send it. What the policy forbids is **reading the response** from JavaScript when the response came from another origin and the server did not explicitly allow reading. The request went out, the server handled it, the response returned, but your script gets an error instead of the body.

Why block reading but not sending? Many legitimate web features send cross-domain requests without needing to read the response. `<img src="https://cdn.other.com/logo.png">` sends a GET to a foreign host by design. A form with `action` on another domain sends POST. Those channels were historically open. What had to close was foreign scripts programmatically pulling private responses using your cookies.

## What is NOT covered

Some cross-domain operations are intentionally allowed because they do not let a script read foreign response bytes:

```html
<img src="https://other.com/pic.png">
<link rel="stylesheet" href="https://other.com/app.css">
<script src="https://other.com/lib.js"></script>
```

Images, styles, and `<script src>` load from any origin and apply. You still cannot read their raw bytes from JavaScript (for images this is "tainted canvas"). Load and apply: yes; read as data: no. `<script>` runs in your context, which is why it works; JSONP once rode that loophole before CORS.

<Callout type="info">
Remember the split: "the browser loaded and applied a resource" versus "my code read the response as data": two different permissions. Same-origin policy cuts the second; the first often stays open.
</Callout>

## What this rule protects

Back to the opening scenario. Banking tab open, session cookies alive. You visit `evil.com`; its script runs `fetch("https://bank.com/balance")`. The browser attaches bank cookies (default for simple requests), the request goes out, the bank responds with balance. Same-origin policy does not give `evil.com` the body: request origin (`evil.com`) ≠ response origin (`bank.com`), and the bank sent no permission. The script errors; balance stays private.

Note what the policy **does not** do. It does not stop the request from reaching the server and executing. So same-origin policy alone does not stop CSRF (state-changing requests like "transfer money"): you need CSRF tokens, `SameSite` cookies, etc. The policy protects response confidentiality, not server-side side effects. Keep that boundary in mind.

## Edge case that surprises people

"One domain - one origin" is false. Subdomains are different origins. Script on `https://app.example.com` cannot read a response from `https://api.example.com` by default, even under the same company. Different host → foreign origin → CORS, same as a totally unrelated site.

```text
https://example.com        baseline origin
https://www.example.com    foreign: www host
https://api.example.com    foreign: subdomain
http://example.com         foreign: http not https
https://example.com:8080   foreign: non-default port
```

Each lower line is a separate origin from the first. Frontend on one subdomain and API on another almost always needs CORS, which surprises newcomers: "but it is our server". To the browser these are two sources.

## Common mistakes

- **"Same-origin policy blocked my request."** It did not block sending: the server may have run your POST. It blocked reading the response in JS. A "failed" POST from CORS may already have executed on the server: important for non-idempotent actions.
- **Treating origin as domain.** Origin is scheme + host + port. `http` vs `https`, `:3000` vs `:8080`, subdomains: all different origins.
- **Expecting the policy to stop CSRF.** It hides responses but not execution. Protect state-changing requests separately.
- **Calling CORS a security hole.** CORS does not disable same-origin policy; it lets the server explicitly allow a specific origin to read its response.

<Detail summary="Where same-origin policy came from">

The rule appeared in Netscape Navigator in 1995 at the dawn of in-browser scripts and became the foundation of web security. Initially it governed DOM access between frames. As XMLHttpRequest, fetch, canvas, and localStorage arrived, each was tied to the same origin logic. Today same-origin policy is a family of rules (network, storage, DOM, cookies) united by origin. CORS, standardized around 2014, is the controlled way to relax the network part where needed.

</Detail>
