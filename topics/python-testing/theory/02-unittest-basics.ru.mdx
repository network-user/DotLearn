# Тесты, собранные в классы по правилам стандартной библиотеки

Голый `assert` из предыдущего раздела отлично объясняет идею проверки, но когда тестов становится сотня, всплывают неудобные вопросы. Где взять отчёт о том, сколько прошло и сколько упало? Как проверить, что функция возбуждает нужное исключение, не оборачивая каждый вызов в `try/except` вручную? Куда деть код подготовки, который повторяется в десяти тестах подряд? На все эти вопросы у Python есть встроенный ответ - модуль `unittest`, входящий в стандартную библиотеку без установки.

`unittest` - это первый тест-фреймворк, который вы встретите в любом проекте на Python, потому что он есть всегда. Он родом из мира xUnit (того же семейства, что JUnit в Java), и потому тесты в нём - это методы класса. К концу раздела вы сможете написать `TestCase`, выбрать правильный assert-метод под проверку, аккуратно подготовить и убрать окружение через `setUp`/`tearDown` и запустить всё одной командой.

## Ментальная модель: тест - это метод класса-наследника TestCase

В `unittest` вы группируете тесты в класс, который наследуется от `unittest.TestCase`. Каждый метод, чьё имя начинается с `test`, фреймворк считает отдельным тестом и запускает изолированно. Методы, названные иначе, тестами не считаются - это вспомогательный код.

```python
import unittest


class TestShipping(unittest.TestCase):
    def test_free_above_3000(self):
        self.assertEqual(shipping_cost(3500), 0)

    def test_paid_below_3000(self):
        self.assertEqual(shipping_cost(1500), 500)
```

Проверки вы делаете не голым `assert`, а методами `self.assert...`. Причина в обратной связи: `self.assertEqual(a, b)` при провале покажет оба значения - и что получили, и что ожидали, - а простой `assert a == b` скажет лишь «ложь». Именно ради информативных сообщений xUnit-фреймворки завели десятки специализированных assert-методов.

## Самые нужные assert-методы

Их много, но в ежедневной работе хватает горстки. `assertEqual(a, b)` проверяет равенство и при провале печатает оба операнда. `assertTrue(x)` и `assertFalse(x)` проверяют истинность значения. `assertIn(item, container)` проверяет вхождение - удобно для списков, строк и словарей.

```python
self.assertEqual(total, 100)
self.assertTrue(user.is_active)
self.assertFalse(cart.is_empty())
self.assertIn("Анна", names)
```

Два метода заслуживают отдельного внимания, потому что у них есть тонкость. Первый - `assertAlmostEqual(a, b)`: он сравнивает числа с плавающей точкой не на строгое равенство, а с округлением (по умолчанию до 7 знаков после запятой). Это спасает от вечной боли float-арифметики, где `0.1 + 0.2` не равно `0.3`.

<Callout type="warning">
Никогда не сравнивайте float через `assertEqual`: `0.1 + 0.2 == 0.3` ложно, потому что число хранится приближённо. Для дробных результатов берите `assertAlmostEqual`, который сравнивает с точностью до заданного знака.
</Callout>

Второй - `assertRaises`. Чтобы проверить, что код возбуждает исключение, его используют как контекстный менеджер: блок внутри `with` обязан возбудить указанный класс ошибки, иначе тест падает.

```python
def test_negative_total_rejected(self):
    with self.assertRaises(ValueError):
        shipping_cost(-100)
```

Если внутри `with` исключение не возникло (или возникло другого типа), тест провалится. Это куда чище, чем ручной `try/except` с флажком «дошли ли мы до строки, где должно было упасть».

## setUp и tearDown: подготовка вокруг каждого теста

Часто несколько тестов начинаются с одной и той же подготовки: создать объект, наполнить его данными. Дублировать это в каждом методе утомительно и хрупко. Для этого есть метод `setUp` - фреймворк вызывает его **перед каждым** test-методом, а `tearDown` - **после каждого**, даже если тест упал.

```python
class TestCart(unittest.TestCase):
    def setUp(self):
        self.cart = Cart()
        self.cart.add("книга", price=500)

    def test_total(self):
        self.assertEqual(self.cart.total(), 500)

    def test_add_increases_count(self):
        self.cart.add("ручка", price=50)
        self.assertEqual(self.cart.count(), 2)
```

Ключевая деталь: `setUp` отрабатывает заново для каждого теста, поэтому `self.cart` во втором методе - свежий объект, а не тот, что остался после первого. Это и есть изоляция: тесты не делят состояние и не зависят от порядка запуска.

Иногда подготовка дорогая - поднять соединение, загрузить большой файл - и повторять её перед каждым тестом расточительно. Тогда берут `setUpClass` и `tearDownClass`: они выполняются **один раз на весь класс**, до первого теста и после последнего. Это методы класса, поэтому их помечают `@classmethod`.

## Запуск и автопоиск тестов

Запускать тесты вручную не нужно - у модуля есть встроенный раннер. Команда `python -m unittest` включает **discovery**: интерпретатор сам обходит каталог, находит файлы вида `test_*.py`, внутри них - классы-наследники `TestCase`, а в них - методы `test_*`, и запускает всё это. Можно сузить запуск до одного класса или метода через путь через точку:

```text
python -m unittest                       все найденные тесты
python -m unittest test_cart             только файл/модуль test_cart
python -m unittest test_cart.TestCart    один класс
python -m unittest -v                    подробный вывод по каждому тесту
```

В отчёте каждая точка - пройденный тест, `F` - проваленный (failure, не сошёлся assert), `E` - ошибка (упало неожиданное исключение). Разница между `F` и `E` важна: failure означает, что код работает, но не так, как ожидает тест; error - что код вообще сломался по пути.

## Пропуск тестов

Иногда тест нужно временно отключить - фича в разработке, баг ещё не починен, проверка не имеет смысла на текущей платформе. Удалять или комментировать тест плохо: он молча исчезает из отчёта. Вместо этого его помечают декоратором `@unittest.skip("причина")`, и в отчёте он отображается как пропущенный (`s`), а причина видна.

```python
@unittest.skip("ждём фикса бага PAY-142")
def test_refund_partial(self):
    ...
```

Есть и условный вариант - `@unittest.skipIf(условие, "причина")`, - который пропускает тест только когда условие истинно (например, на старой версии Python). Пропущенный тест честно виден в отчёте, в отличие от закомментированного.

## Краевой случай: tearDown срабатывает даже после падения

Здесь прячется приятный сюрприз. Если test-метод упал на assert или выбросил исключение, `tearDown` всё равно будет вызван. Это сделано намеренно: уборку (закрыть файл, откатить транзакцию, удалить временную папку) нужно выполнить независимо от исхода теста, иначе мусор от упавшего теста потечёт в следующий и испортит его. А вот если упадёт сам `setUp`, то test-метод не запустится вообще - подготовка не удалась, проверять нечего, - но `tearDown` в этом случае уже не вызывается, потому что и готовить-то было нечего.

## Типичные ошибки

- **Метод теста назван без префикса `test`.** Назвали `check_total` вместо `test_total` - и раннер молча его игнорирует, тест не запускается и не попадает в отчёт. Зелёный прогон создаёт ложное спокойствие. Фикс: имя каждого теста начинается с `test`.
- **Сравнение float через `assertEqual`.** `self.assertEqual(0.1 + 0.2, 0.3)` падает из-за двоичного представления дробей. Фикс: для дробных чисел используйте `assertAlmostEqual`.
- **Подготовка в `__init__` вместо `setUp`.** Кажется логичным завести данные в конструкторе класса, но `unittest` создаёт отдельный экземпляр под каждый тест и не гарантирует поведение, на которое вы рассчитываете. Фикс: вся подготовка живёт в `setUp`, уборка - в `tearDown`.

<Detail summary="assertEqual умеет больше, чем кажется">

`assertEqual` не просто зовёт `==`. Для распространённых типов он подключает специализированные «диффы»: при сравнении двух списков покажет, какие элементы разошлись; для словарей - какие ключи; для многострочных строк выдаст построчный diff с `+`/`-`. Поэтому при сравнении коллекций предпочитайте `assertEqual(got, expected)` ручному перебору в цикле - сообщение о провале будет на порядок понятнее. Эти диффы регистрируются через `addTypeEqualityFunc`, и для своих типов их тоже можно настроить, но в большинстве проектов хватает встроенных.

</Detail>
