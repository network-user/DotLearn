# A test is a question the code answers for itself

You just tweaked the discount function in an Ozon-style store: added a rule that says "orders over 3000 rubles ship free." A two-line change. But that function is also called from the cart, from order history, and from confirmation emails, and you don't remember every place it's used. To make sure nothing broke, you'd have to open every screen by hand, enter data, and eyeball the result. By the third screen your attention drifts, on the fifth you miss a typo, and the bug surfaces in front of a user a week later.

An automated test takes that work off your plate. It's a small program that runs your code with a specific input and checks that the result is exactly what you intended. Running a thousand such checks takes a fraction of a second. By the end of this section you'll understand why tests pay for themselves, how the testing pyramid is shaped, and what three steps any meaningful test breaks down into.

## Mental model: three reasons tests pay off

A test isn't a goal in itself or a box to check. It earns its keep for three concrete reasons, and those are more useful to remember than any formal definition.

The first is **fast feedback**. When you change code, you want to know whether it works in seconds, not after clicking through the whole app by hand. A test suite answers that question instantly, without requiring your attention.

The second is **protection against regressions**. A regression is when a new change breaks something that used to work. A test written against the old behavior fails the instant you accidentally break it, and shows you exactly where. Without tests, the user finds the regression for you.

The third is **tests as a specification**. A well-named test like `test_free_shipping_over_3000_rubles` is an executable description of what the code is supposed to do. Documentation goes stale silently; a test that drifts from the code fails and forces you to update it.

## The pyramid: how many of each

Tests come in different scales, and the pyramid tells you the proportions. At the base sit **unit tests**: they check one small unit (a function, a method, a class) in isolation from everything else. There are many of them, they run in milliseconds, and they point precisely at what broke.

In the middle are **integration tests**: they check that several parts work together - a function and a real database, your code and an HTTP client. They're slower and cover more ground, but a failure no longer points as precisely at the culprit.

At the top sit **end-to-end (e2e) tests**: they run an entire scenario the way a live user would, through the real interface. They give you the most confidence that the system works as a product, but they're the slowest and most brittle - a failure might come from a shifted layout or a network timeout rather than an actual logic bug.

The point of the ratio is simple: the higher up a test sits, the more expensive it is to write and maintain, and the longer it takes to run. Lean on the cheap, fast base, and keep the slow e2e tests rare - reserved for critical end-to-end flows like "place and pay for an order."

## What a test is made of: the AAA pattern

Any meaningful test breaks down into three steps, a structure called **Arrange-Act-Assert**, or Given-When-Then in behavior-driven style.

```python
def test_доставка_бесплатна_от_3000():
    cart = Cart(items_total=3500)

    shipping = cart.shipping_cost()

    assert shipping == 0
```

The first block is **Arrange**: you set up the data and environment the check needs. Here it's a cart with a total of 3500. The second is **Act**: you perform exactly one action, the one you're testing - calling `shipping_cost()`. The third is **Assert**: you compare the result against what you expected. `assert` is an instruction that says "I claim this condition is true"; if it's false, the test fails.

Keep the three steps visually separate (even just a blank line between them), because anyone reading the test should instantly see what's given, what's done, and what's checked. A test where the phases blur together is painful to read.

<Callout type="info">
`assert condition` is a built-in Python statement: if `condition` is true, nothing happens; if it's false, it raises `AssertionError`, and the test counts as failed. The entire checking mechanism is built on it.
</Callout>

## Signs of a good test

Not every test is useful. A bad test is worse than no test: it creates a false sense of safety or flashes red for no reason. A good unit test rests on four properties.

It's **fast**: milliseconds, not seconds. A suite that takes ten minutes to run stops getting run, and it dies.

It's **isolated**: it doesn't depend on other tests or on the outside world. Run order shouldn't affect the outcome. If `test_a` leaves a record in a shared list and `test_b` relies on it, both become unreliable.

It's **deterministic**: given the same code, it produces the same verdict every time. A test that's sometimes green, sometimes red with no code changes (called flaky) is useless - people stop trusting it.

It has **one reason to fail**: it checks one thing, and its name tells you which. When it fails, you know immediately what broke, without reading through a dozen assertions.

## Edge case: the test that's always green

Here's an unpleasant surprise. A test that never fails doesn't prove the code works; it might simply not be checking anything. The classic trap is a forgotten `assert`:

```python
def test_skidka():
    cart = Cart(items_total=3500)
    cart.shipping_cost() == 0
```

This test is **always green**, even if `shipping_cost()` returns 500. The line `cart.shipping_cost() == 0` computes a boolean `True` or `False` and immediately discards it - there's no `assert` in front of it, so nothing can fail. To trust a test, verify it can actually turn red: temporarily break the code and confirm the test catches the breakage. A test that has never failed is suspicious.

## Common mistakes

- **A test with no assert.** You called the function but forgot to check the result. The test is green but verifies nothing. Fix: every test ends with at least one `assert` on the expected result, and gets checked for its ability to fail.
- **Checking implementation instead of behavior.** A test that pokes at private fields and counts calls to internal methods breaks on any refactor, even when behavior hasn't changed. Fix: check the observable result ("shipping is zero"), not how it was produced.
- **Depending on test order.** One test mutates global state, another relies on it, and the suite breaks the moment order changes. Fix: every test sets up its own environment and leaves no trace behind.

<Detail summary="TDD: the test before the code">

Test-Driven Development is a practice where you write the test before the implementation, in a "red-green-refactor" cycle. First you write a failing test for behavior that doesn't exist yet (red): it states what the code is supposed to do. Then you write the minimum code to make it pass (green). Then you calmly improve the code, leaning on the green test as a safety net (refactor). The upside is that a test in the red phase is proven capable of failing - you watched it fail - and it pins down the requirement before the implementation clouds your judgment. TDD isn't required to write good tests, but it instills a great habit: start from "what should happen," not "how do I code this."

</Detail>
