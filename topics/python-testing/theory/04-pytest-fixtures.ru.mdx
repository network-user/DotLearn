# Окружение, которое тест запрашивает по имени

Десять тестов вокруг корзины начинаются одинаково: создать пустую корзину, положить туда пару товаров, подключить тестовую базу. В `unittest` это вешали на `setUp`, и подготовка применялась ко всем тестам класса разом - хочешь не хочешь. Но что, если одному тесту нужна корзина, второму - база, третьему - и то и другое, а четвёртому вообще ничего? `setUp` тут грубоват: он один на класс и не умеет выдавать каждому тесту ровно то, что тот просит.

Фикстуры pytest решают это изящно. Фикстура - это именованная заготовка окружения, а тест **запрашивает** её, просто указав имя в своих аргументах. pytest видит запрос, готовит заготовку, передаёт результат - и убирает за собой, когда тест закончился. К концу раздела вы научитесь объявлять фикстуры, управлять тем, как часто они отрабатывают через `scope`, делиться ими между файлами через `conftest.py`, писать фикстуры с уборкой через `yield`, а также фикстуры-фабрики и параметризованные фикстуры.

## Ментальная модель: тест просит фикстуру по имени аргумента

Фикстуру объявляют функцией с декоратором `@pytest.fixture`. Её имя - это и есть «название» заготовки. Чтобы получить её в тесте, добавьте аргумент с тем же именем: pytest сопоставит имя аргумента с фикстурой, выполнит её и подставит возвращённое значение.

```python
import pytest


@pytest.fixture
def cart():
    c = Cart()
    c.add("книга", price=500)
    return c


def test_total(cart):
    assert cart.total() == 500
```

Это инъекция зависимостей в чистом виде: тест не создаёт `Cart` сам и не знает, как тот собран, - он лишь объявляет «мне нужен `cart`», а pytest доставляет готовый объект. Тест, которому корзина не нужна, просто не упоминает её в аргументах и не платит за её подготовку.

## yield-фикстуры: setup и teardown в одной функции

Возврат через `return` хорош, когда убирать нечего. Но если фикстура открыла соединение или создала временный файл, его нужно закрыть после теста. Для этого пишут фикстуру через `yield`: всё **до** `yield` - это setup, отданное значение - то, что получит тест, а всё **после** `yield` - teardown, который pytest выполнит, когда тест отработает.

```python
@pytest.fixture
def db():
    connection = connect("test.db")
    yield connection
    connection.close()
```

Тест получает `connection`, работает с ним, а строка `connection.close()` отрабатывает автоматически после теста. Это аккуратнее `setUp`/`tearDown` тем, что setup и teardown одного ресурса лежат рядом в одной функции, а не разнесены по двум методам.

## scope: сколько раз фикстура отрабатывает

По умолчанию фикстура выполняется заново **для каждого теста**, который её просит. Это безопасно (полная изоляция), но иногда дорого: незачем поднимать соединение к базе перед каждым из ста тестов. Параметр `scope` управляет тем, как часто фикстура готовится и как долго живёт между тестами:

```python
@pytest.fixture(scope="module")
def db():
    connection = connect("test.db")
    yield connection
    connection.close()
```

- `function` (по умолчанию) - новый экземпляр на каждый тест;
- `class` - один на класс тестов;
- `module` - один на весь файл;
- `session` - один на весь прогон pytest.

Чем шире scope, тем меньше раз отрабатывает setup, но тем больше тесты делят общий объект - а значит, тем выше риск, что один тест испортит состояние для другого. Дорогие и только читаемые ресурсы (соединение, конфиг) можно расширять; изменяемые лучше держать на `function`, чтобы каждый тест получал чистую копию.

<Callout type="warning">
Широкий scope (`module`, `session`) экономит время, но ломает изоляцию, если тест меняет общий объект. Расширяйте scope только для того, что тесты читают, но не мутируют, иначе вернётся зависимость от порядка тестов, ради борьбы с которой всё и затевалось.
</Callout>

## conftest.py: общие фикстуры без импорта

Фикстуру, нужную многим файлам, не дублируют и не импортируют руками. Её кладут в специальный файл `conftest.py`, и pytest делает её доступной всем тестам в этом каталоге и вложенных - автоматически, без `import`.

```python
@pytest.fixture
def cart():
    c = Cart()
    c.add("книга", price=500)
    return c
```

Положите это в `tests/conftest.py`, и любой тест в `tests/` может попросить `cart`, не зная, где фикстура определена. Это механизм, через который команды делятся общим тестовым окружением: фикстуры для базы, клиента приложения, тестовых данных живут в `conftest.py` и доступны всему дереву под ним.

## Фикстуры-фабрики и параметризованные фикстуры

Иногда тесту нужен не один готовый объект, а возможность создавать их с разными параметрами прямо внутри теста. Тогда фикстура возвращает **функцию** - это «фабрика»:

```python
@pytest.fixture
def make_user():
    def _make(name, age=30):
        return User(name=name, age=age)
    return _make


def test_two_users(make_user):
    anna = make_user("Анна", age=25)
    boris = make_user("Борис")
    assert anna.age != boris.age
```

Тест получает `make_user` и зовёт его сколько нужно и с какими угодно аргументами. Заметьте: тест просит фабрику, но вызывает её сам - фикстура лишь даёт инструмент.

Другой приём - **параметризованная фикстура**: указав `params=[...]`, вы заставляете pytest прогнать каждый тест, который её просит, по разу на каждое значение. Доступ к текущему значению - через `request.param`.

```python
@pytest.fixture(params=["sqlite", "postgres"])
def db_engine(request):
    return make_engine(request.param)
```

Любой тест с аргументом `db_engine` теперь выполнится дважды - на sqlite и на postgres. Отличие от `parametrize`: параметризованная фикстура размножает **все** тесты, которые её используют, а не один помеченный.

## Встроенные фикстуры: tmp_path, monkeypatch, capsys

pytest приносит готовые фикстуры, которые не надо объявлять - достаточно попросить по имени. `tmp_path` даёт уникальный временный каталог (объект `Path`), который pytest сам создаст и удалит, - идеально для тестов, пишущих файлы. `monkeypatch` временно подменяет атрибуты, переменные окружения и элементы словарей, откатывая всё после теста (про него подробно в следующем разделе). `capsys` перехватывает то, что тест напечатал в `stdout`/`stderr`, чтобы это можно было проверить.

```python
def test_writes_report(tmp_path):
    target = tmp_path / "report.txt"
    save_report(target, data="итог")
    assert target.read_text() == "итог"
```

Здесь `tmp_path` - чистый каталог под этот тест, и убирать его вручную не нужно.

## Краевой случай: фикстура падает в setup

Здесь поджидает тонкость. Если фикстура упала **до** `yield` (на этапе setup - например, не открылось соединение), сам тест помечается не как failure, а как **error**: проверять было нечего, окружение не собралось. При этом teardown-часть этой фикстуры (код после `yield`) не выполнится - ведь до `yield` дело не дошло, ресурс не создан, закрывать нечего. А вот фикстуры, которые **успели** отработать setup до этой, свой teardown выполнят. Поэтому ресурс открывают как можно ближе к `yield`: чем меньше кода между захватом ресурса и `yield`, тем меньше шанс упасть с уже захваченным, но не освобождённым ресурсом.

## Типичные ошибки

- **Имя аргумента не совпало с именем фикстуры.** Фикстура называется `cart`, а в тесте написали `def test_x(card)` - pytest не найдёт фикстуру `card` и выдаст ошибку «fixture not found». Фикс: имя аргумента обязано в точности совпадать с именем фикстуры.
- **Уборка через `return` вместо `yield`.** Написали `return connection`, а закрытие повесить негде - teardown не выполнится, ресурсы потекут. Фикс: как только появляется teardown, переводите фикстуру на `yield`.
- **Слишком широкий scope для изменяемого объекта.** Поставили `scope="session"` корзине, которую тесты наполняют, - и тесты начали видеть чужие товары. Фикс: изменяемое окружение держите на `function`, расширяйте scope только для read-only ресурсов.

<Detail summary="Фикстуры зависят от фикстур">

Фикстура сама может попросить другую фикстуру - просто указав её в своих аргументах. pytest выстраивает граф зависимостей и готовит их в нужном порядке. Например, фикстура `cart` может зависеть от фикстуры `db`: `def cart(db): ...`. Тогда запрос `cart` в тесте автоматически подтянет и `db`. Это позволяет собирать сложное окружение из маленьких переиспользуемых кусочков, и каждый отвечает за свой ресурс и свою уборку. Порядок teardown - обратный порядку setup (как стек): что создано последним, закрывается первым, что логично для зависимых ресурсов.

</Detail>
