# Обычный assert, который сам объясняет, что пошло не так

Откройте чужой проект на Python, и с почти полной уверенностью тесты в нём написаны на pytest, а не на стандартном `unittest`. Причина не в магии, а в одном решении, которое снимает целый класс церемоний: pytest позволяет писать проверки обычным оператором `assert`, но при провале показывает значения операндов так же подробно, как `assertEqual`. Вам больше не нужно помнить десяток assert-методов и выбирать правильный под каждую проверку - вы пишете `assert x == y`, а фреймворк сам разбирает выражение и печатает, чему равнялись `x` и `y`.

Это называется **assert introspection** (интроспекция assert): pytest на этапе импорта переписывает байт-код ваших тестов, встраивая в каждый `assert` сбор промежуточных значений. Поэтому сообщение о провале информативно, хотя в коде - обычный встроенный оператор. К концу раздела вы будете писать тесты-функции без классов, запускать нужное подмножество по имени и маркеру, размножать один тест на десятки наборов данных через параметризацию и помечать тесты как пропущенные или ожидаемо падающие.

## Ментальная модель: тест - это просто функция с assert

В pytest тест - это функция, имя которой начинается с `test_`. Никакого класса, никакого наследования, никаких `self.assert...`. Внутри - обычный `assert`.

```python
def test_free_above_3000():
    assert shipping_cost(3500) == 0


def test_paid_below_3000():
    assert shipping_cost(1500) == 500
```

Когда `assert shipping_cost(1500) == 0` провалится, pytest не ограничится словом «ложь». Он напечатает примерно `assert 500 == 0` и даже разложит сложные выражения - покажет, что было слева, что справа, какие элементы списка разошлись. Эту работу делает та самая интроспекция: значения собираются автоматически, без вашего участия.

## Запуск и выбор тестов

Команда `pytest` сама находит файлы `test_*.py`, внутри них функции `test_*` и запускает их - discovery работает похоже на `unittest`. Но самое полезное в ежедневной работе - выбирать подмножество, не гоняя весь набор.

Каждый тест адресуется своим **node id** - путём вида `файл::функция`. По нему запускают ровно один тест. Флаг `-k` принимает выражение по подстроке в имени, `-m` - выбирает тесты с заданным маркером.

```text
pytest                              все тесты
pytest tests/test_cart.py::test_total   один конкретный тест
pytest -k "total and not refund"    тесты, в имени которых есть total, но нет refund
pytest -m slow                      только тесты с маркером @pytest.mark.slow
```

Выражение в `-k` - это не регулярка, а булева логика по подстрокам имени: можно `and`, `or`, `not`. Это удобно, когда вы дробите код и хотите крутить только тесты вокруг затронутой функции, а не ждать весь прогон.

## Параметризация: один тест, много наборов

Часто один и тот же тест нужно прогнать на разных входах: проверить скидку и на 3500, и на 3000, и на 1500. Копировать функцию ради смены одного числа - плохо. Декоратор `@pytest.mark.parametrize` размножает тест: вы перечисляете наборы аргументов, и pytest порождает по отдельному тест-кейсу на каждый.

```python
import pytest


@pytest.mark.parametrize("total, expected", [
    (3500, 0),
    (3000, 0),
    (1500, 500),
])
def test_shipping(total, expected):
    assert shipping_cost(total) == expected
```

Это **не** один тест с циклом внутри, а три независимых теста. Если набор `(1500, 500)` упадёт, остальные два всё равно отработают и попадут в отчёт как пройденные - вы сразу видите, какой именно вход сломался, а не «тест упал где-то внутри». В отчёте кейсы различаются по значениям параметров; при желании им дают читаемые имена через `ids=[...]`.

<Callout type="tip">
Параметризация - главный приём против дублирования в тестах. Вместо пяти почти одинаковых функций - одна с пятью наборами данных. Добавить новый случай = дописать одну строку в список, а не копировать функцию целиком.
</Callout>

## Маркеры: метки на тестах

Маркер - это ярлык, который вы вешаете на тест декоратором `@pytest.mark.<имя>`. Часть маркеров встроена и меняет поведение, часть вы заводите сами просто для группировки.

`@pytest.mark.skip(reason="...")` безусловно пропускает тест. `@pytest.mark.skipif(условие, reason="...")` пропускает только когда условие истинно - например, на неподходящей версии Python или ОС.

```python
@pytest.mark.skipif(sys.version_info < (3, 11), reason="нужен 3.11+")
def test_new_syntax():
    ...
```

Особый случай - `@pytest.mark.xfail`: он помечает тест как **ожидаемо падающий**. Известный, ещё не починенный баг оборачивают в `xfail` - тогда его падение не красит весь прогон в красный (в отчёте это `xfail`), но если тест вдруг **пройдёт**, pytest отметит его как `xpass` и обратит на это внимание: возможно, баг уже починили и метку пора снять.

Свои маркеры (например `@pytest.mark.slow` для медленных тестов) стоит зарегистрировать в конфиге, иначе pytest выдаст предупреждение о незнакомом маркере. После регистрации ими удобно фильтровать запуск: `pytest -m "not slow"` прогонит всё, кроме медленного.

## Проверка исключений: pytest.raises

Аналог `assertRaises` в pytest - контекстный менеджер `pytest.raises`. Блок внутри `with` обязан возбудить указанное исключение.

```python
def test_negative_rejected():
    with pytest.raises(ValueError):
        shipping_cost(-100)
```

Часто хочется проверить ещё и текст ошибки. Для этого у `pytest.raises` есть аргумент `match` (регулярное выражение по сообщению) и доступ к самому исключению через `as`:

```python
def test_message():
    with pytest.raises(ValueError, match="отрицательн"):
        shipping_cost(-100)
```

## Краевой случай: одна и та же ссылка в наборах параметризации

Здесь прячется ловушка, которая удивляет даже опытных. Если в наборы параметризации передать **изменяемый** объект как один общий экземпляр, кейсы начнут делить его между собой:

```python
shared = []


@pytest.mark.parametrize("data", [shared, shared])
def test_append(data):
    data.append(1)
    assert len(data) == 1
```

Первый кейс сделает список `[1]` и пройдёт. Второй получит **тот же** список (это одна и та же ссылка `shared`), добавит ещё единицу, увидит длину `2` - и упадёт. Параметризация не копирует объекты, она передаёт ровно то, что вы вписали в список. Лекарство - класть в наборы свежие литералы (`[1]`, `[1]` как два разных списка) или строить объект внутри теста, а не снаружи.

## Типичные ошибки

- **Класс с тестами назвали не `Test...`** При желании сгруппировать тесты в класс pytest подберёт его, только если имя начинается с `Test` и у класса нет `__init__`. Иначе тесты внутри молча не запустятся. Фикс: либо имя `TestCart`, либо вообще функции без класса.
- **Параметры перепутаны местами.** В `parametrize("total, expected", ...)` порядок имён обязан совпадать с порядком значений в кортежах. Перепутали - и тест проверяет не то. Фикс: держите имена и кортежи в одном порядке, при сомнении дайте `ids`.
- **`@pytest.mark.skip` забыт после фикса.** Тест отключили на время, баг починили, а метку снять забыли - проверка молча не работает месяцами. Фикс: для известного бага предпочитайте `xfail`: когда баг починят, тест станет `xpass` и сам напомнит снять метку.

<Detail summary="Почему pytest видит значения внутри assert">

Стандартный оператор `assert x == y` на уровне байт-кода просто проверяет результат `x == y` и при ложности возбуждает `AssertionError` без подробностей - операнды к этому моменту уже потеряны. pytest решает это перезаписью (assertion rewriting): при импорте тестового модуля он перехватывает загрузку, разбирает AST каждого `assert` и подменяет его на код, который сначала вычисляет и сохраняет подвыражения, а затем, если проверка ложна, собирает из них подробное сообщение. Поэтому интроспекция работает только в файлах, которые pytest импортирует сам как тесты (или в явно зарегистрированных через `register_assert_rewrite` модулях-помощниках); обычный `assert` в коде приложения её не получает.

</Detail>
