# One rule from which all of Clean Architecture follows

In the previous section we separated business rules and infrastructure onto different levels. But a gap remained: the service that places an order still has to call saving to the database somewhere. If it directly imports a repository class that imports the PostgreSQL driver, the dependency arrow again points from business logic into technology. You changed the database: you change the import in the service. Layers exist, but dependency still flows the wrong way.

Clean Architecture solves this with one sentence: the dependency rule. By the end of this section you will understand why all dependencies must point inward toward business rules, how dependency inversion physically flips the arrow from the database toward the domain, and how a layer boundary differs from simply splitting into modules. This is the heart of the whole topic.

## Mental model: concentric rings and one arrow

Clean Architecture draws the system as nested rings. At the center sit Entities, the most stable business rules; around them, Use Cases; further out, interface adapters (controllers, presenters, gateways); and on the outside, frameworks and drivers: web, database, external services. Exactly one rule ties the rings together.

> Source code dependencies always point inward, toward higher-level policies. Nothing in an inner circle knows anything about an outer one.

The arrow in the diagram is the direction of dependencies in source code, not the direction of data flow at runtime. Data during execution naturally moves both ways: a request arrives from outside, a response leaves outside. But `import` always points toward the center. A use case imports an entity; an entity never imports a use case. A controller knows about the use case; the use case does not know about the controller.

## Naive code breaks the rule

Back to placing an order. The natural first sketch: the service creates the repository itself and calls it.

```python
from infrastructure.sql_order_repo import SqlOrderRepo

class PlaceOrderUseCase:
    def execute(self, order):
        repo = SqlOrderRepo()
        repo.save(order)
```

Here the use case (inner ring) imports `SqlOrderRepo` from infrastructure (outer ring). The dependency arrow points outward, a direct violation of the rule. The name `Sql` leaked into the center of the system: switching the database to Mongo forces edits in the business-scenario file.

## Dependency inversion flips the arrow

To flip the dependency without flipping data flow, the use case must depend not on a concrete repository but on an **abstraction** it declares itself. Declare the interface next to the use case, in the inner ring.

```python
from abc import ABC, abstractmethod

class OrderRepository(ABC):
    @abstractmethod
    def save(self, order) -> None: ...
```

Now the use case depends only on this interface and knows nothing about SQL.

```python
class PlaceOrderUseCase:
    def __init__(self, repo: OrderRepository):
        self._repo = repo

    def execute(self, order):
        self._repo.save(order)
```

The concrete implementation lives in infrastructure and **implements** the interface from the center.

```python
class SqlOrderRepo(OrderRepository):
    def save(self, order) -> None:
        ...
```

The arrow flipped: now `SqlOrderRepo` (outer ring) depends on `OrderRepository` (inner ring), not the other way around. This is dependency inversion (the D in SOLID). The use case dictates the contract; infrastructure adapts.

## The boundary is the interface data crosses

The line between rings is called the boundary. Data crosses it only in simple, framework-independent forms: primitives, dataclasses, plain structures. You cannot drag through an object that carries knowledge of the outer ring (for example an ORM model that hits the database when you access a field, or a framework HTTP request object).

```python
@dataclass(frozen=True)
class OrderDTO:
    id: str
    total: float
```

The use case accepts and returns such simple structures. The controller outside translates the HTTP request into `OrderDTO`, and the presenter translates the use case result into JSON. The boundary stays clean: no outer-ring detail crosses inward.

<Callout type="warning">
If an object you pass inward through the boundary hits the database when you access a field (lazy ORM loading), the boundary is leaky: the business scenario unknowingly depends on the database. Pass "dead" data structures through the boundary, not live ORM objects.
</Callout>

## This will surprise you: the interface belongs to whoever calls it

Intuition says the repository interface is part of the data layer because it is about storage. In Clean Architecture it is the opposite: the `OrderRepository` interface belongs to the use case that calls it, not the implementation that fulfills it. So the interface file sits in the inner ring, next to business logic, and `SqlOrderRepo` sits outside.

This is the inversion mechanism: the contract owner is the consumer, not the supplier. When the contract belongs to the center, the center dictates what it needs ("be able to save an order"), and infrastructure must provide it by any means: SQL, file, test stub. The `import` arrow runs from supplier to consumer, that is inward, as the rule requires.

## Common mistakes

- **Putting the repository interface in the infrastructure layer.** Then the use case imports it from the outer ring and the arrow points outward again. The interface is part of the inner layer; the implementation is part of the outer.
- **Dragging ORM models through the boundary.** If the use case works with SQLAlchemy objects, it depends on the ORM. Translate data into simple dataclasses at the boundary itself.
- **Confusing data flow direction with dependency direction.** Data at runtime flows outward to the response; that is fine. The rule limits only `import` in source code; it always points inward.

<Detail summary="Why this is called inversion, not just abstraction">

"Just add an interface" is not inversion yet. Dependency Inversion Principle is that both high-level and low-level modules depend on an abstraction, and that abstraction is defined on the high-level side. Normal control flow naturally runs from high level to low: the use case calls the repository. Without inversion, the code dependency would go the same way. Inversion flips exactly the dependency in source code while leaving control flow as it was. Hence "inversion": we reversed the dependency relative to control flow.

</Detail>
