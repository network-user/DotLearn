# От круговых диаграмм к настоящему дереву каталогов

Теория красива на схемах, но в какой-то момент нужно решить приземлённые вопросы: какие папки завести, что куда импортировать, где собрать всё вместе и не превратится ли это в гору абстракций ради абстракций. Python добавляет своих особенностей: тут не принято плодить интерфейс на каждый класс, а утиная типизация позволяет жить вообще без `abc`. Значит, переносить C#-овые рецепты буквально не стоит - нужна прагматичная адаптация.

К концу раздела Вы будете уметь раскладывать слои по пакетам так, чтобы направление импортов соблюдало правило зависимостей, собирать зависимости в одной точке композиции (composition root) и понимать, где в Python уместны протоколы вместо абстрактных классов, а где слои - избыточная роскошь. Это превращает всю тему в работающий шаблон проекта.

## Ментальная модель: пакеты как кольца

Кольца Clean Architecture проще всего отразить пакетами. Внутренние пакеты не импортируют внешние - это и есть правило зависимостей, выраженное в `import`.

```text
src/
  domain/        сущности и value-объекты
  application/   use cases + порты (интерфейсы)
  infrastructure/ адаптеры: SQL-репозитории, клиенты API
  interface/     контроллеры FastAPI, CLI
  main.py        composition root
```

`domain` не импортирует ничего из проекта. `application` импортирует `domain`, но не `infrastructure`. `infrastructure` и `interface` импортируют `application` и `domain`, но не друг друга. `main.py` импортирует всё и связывает.

## Порт - это Protocol, не обязательно ABC

В Python интерфейс не обязан быть абстрактным классом. Тип `Protocol` из `typing` задаёт структурный контракт: реализацией считается любой класс с подходящими методами, без явного наследования. Это ближе к духу утиной типизации и убирает связанность даже на уровне наследования.

```python
from typing import Protocol

class OrderRepository(Protocol):
    def save(self, order: Order) -> None: ...
    def by_id(self, order_id: str) -> Order | None: ...
```

Use case аннотирует зависимость этим протоколом. Реализация в `infrastructure` не наследует `OrderRepository` - ей достаточно иметь методы `save` и `by_id` с совместимыми сигнатурами, и mypy признает её подходящей.

```python
class SqlOrderRepository:
    def save(self, order: Order) -> None: ...
    def by_id(self, order_id: str) -> Order | None: ...
```

`SqlOrderRepository` нигде не упоминает `OrderRepository`, но структурно ему соответствует. Связанность минимальна: инфраструктура не зависит даже от имени порта. Когда же вы хотите, чтобы наследование было обязательным (например, чтобы получить ошибку при отсутствии метода в рантайме), используйте `abc.ABC` - оба подхода уместны.

## Composition root: единственное место, где всё знакомо

Где-то конкретные классы всё-таки нужно соединить: создать репозиторий, передать его в use case, передать use case в контроллер. Это место называется composition root - точка композиции. Её держат одну на приложение, обычно в `main.py` или в фабрике. Только здесь код видит сразу и абстракции, и реализации.

```python
def build_place_order(session) -> PlaceOrderUseCase:
    repo = SqlOrderRepository(session)
    notifier = EmailNotifier(smtp_config)
    return PlaceOrderUseCase(repo=repo, notifier=notifier)
```

Внутри use case, домена, контроллера никаких `SqlOrderRepository(...)` нет - они получают готовые зависимости через конструктор. Вся «грязная» сборка собрана в одной функции. Поменять SMTP на заглушку для теста - значит передать другую реализацию в этой одной точке, не трогая бизнес-код.

<Callout type="tip">
Признак здоровой архитектуры: конкретные имена инфраструктурных классов (`SqlOrderRepository`, `EmailNotifier`) встречаются в проекте ровно в двух местах - в файле, где класс определён, и в composition root, где его создают. Если они всплывают в use cases, инверсия не доведена до конца.
</Callout>

## Это вас удивит: слои бывают вредной роскошью

Соблазн после изучения темы - разложить по пяти кольцам даже скрипт на двести строк, который читает CSV и пишет в другой CSV. Это ошибка обратного знака. Слои окупаются, когда есть нетривиальная бизнес-логика и реальная вероятность смены технологий или серьёзных тестов. Для скрипта-однодневки три интерфейса и composition root - чистый оверхед: больше кода, больше файлов, медленнее чтение, и ноль выгоды, потому что менять там нечего.

Архитектура - это инвестиция, которая окупается изменениями. Нет изменений - нет окупаемости. Здравый подход: начинать просто, выделять слой ровно тогда, когда смешение действительно начало мешать (тест требует базы, второй способ доставки замаячил на горизонте). Преждевременное расслоение так же вредно, как преждевременная оптимизация.

## Типичные ошибки

- **Импорт из внешнего пакета во внутренний.** Самое частое нарушение - `from infrastructure import ...` внутри `application`. Ловится линтером на импортах (например, `import-linter`) в CI, не полагайтесь на дисциплину вручную.
- **Размазанная композиция.** Если `SqlOrderRepository()` создаётся прямо в нескольких use cases, точки композиции больше нет, и подменить реализацию централизованно нельзя. Сводите создание в один composition root.
- **Расслоение ради расслоения.** Пять колец вокруг CRUD без бизнес-логики добавляют церемоний и ничего не защищают. Сначала боль от смешения, потом слой - не наоборот.

<Detail summary="Чем проверять направление зависимостей автоматически">

Полагаться на ревью в вопросе импортов ненадёжно: запрещённый `import` легко проскакивает. Инструмент `import-linter` позволяет описать контракты вроде «пакет domain не должен импортировать application, infrastructure, interface» и проверять их в CI. Тогда нарушение правила зависимостей становится падающим тестом, а не замечанием на ревью, которое заметят через полгода. Это превращает архитектурный принцип в исполняемую проверку - ровно так же, как тип превращает соглашение в проверку компилятором.

</Detail>
