Marketing asks to change a discount rule — seemingly ten minutes of work. You open the function and find an SQL query, a call to the payment API, an email send, and somewhere in between the formula itself. To verify the new formula you have to bring up the database, connect the payment gateway, and configure mail. A small change runs into half the project's infrastructure. This topic is about how to lay out code so a business rule can be changed and tested separately from how it is delivered to the world.

**What you will learn to do.** See different levels of responsibility in code and spread them across layers; invert dependencies so the database and framework depend on the domain, not the other way around; read and apply the vocabulary of ports, adapters, and rings used when designing backends; and finally turn diagrams into a real Python project directory tree without unnecessary abstractions.

**Topic route.** Four sections, from pain in the code to a working template:

1. **Layers** — why mixed "everything in one pile" code is impossible to change and how separation by responsibility level restores control.
2. **The dependency rule** — one sentence from which all of Clean Architecture follows: every arrow points inward, toward business rules.
3. **Onion and hexagonal** — three names for one idea and the differences in emphasis that easily confuse people in interviews.
4. **Applying it in Python** — how to lay out layers in packages, wire everything at the composition root, and avoid drowning in abstractions for their own sake.

**What you need beforehand.** Confidence in Python OOP helps: classes, inheritance, and why interfaces exist (the `python-oop` topic). You will not need to run anything — we work through examples on diagrams and code fragments, and everything else is introduced along the way.
