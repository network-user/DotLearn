# Одно правило, из которого выводится вся Clean Architecture

В прошлом разделе мы разнесли бизнес-правило и инфраструктуру по разным уровням. Но осталась незакрытая дыра: сервис, который оформляет заказ, всё равно где-то должен вызвать сохранение в базу. Если он напрямую импортирует класс репозитория, который импортирует драйвер PostgreSQL, то стрелка зависимости снова указывает из бизнес-логики в технологию. Поменяли базу - поменяли импорт в сервисе. Слои есть, а зависимость по-прежнему течёт не туда.

Clean Architecture решает это одним предложением - правилом зависимостей. К концу раздела Вы будете понимать, почему все зависимости должны указывать внутрь, к бизнес-правилам, как инверсия зависимостей (Dependency Inversion) физически разворачивает стрелку от базы к домену, и чем граница слоя отличается от простого деления на модули. Это сердцевина всей темы.

## Ментальная модель: концентрические круги и одна стрелка

Clean Architecture рисует систему как набор вложенных колец. В центре - сущности (Entities), самые стабильные бизнес-правила. Вокруг - сценарии использования (Use Cases). Дальше - адаптеры интерфейсов (контроллеры, презентеры, шлюзы). Снаружи - фреймворки и драйверы: веб, база, внешние сервисы. И ровно одно правило связывает кольца воедино.

> Зависимости в исходном коде всегда указывают внутрь, к более высоким политикам. Ничто во внутреннем круге не знает ничего о внешнем.

Стрелка на рисунке - это направление зависимостей в исходном коде, а не направление потока данных. Данные во время выполнения, конечно, ходят в обе стороны: запрос приходит снаружи, ответ уходит наружу. Но `import` всегда направлен к центру. Use case импортирует сущность; сущность никогда не импортирует use case. Контроллер знает про use case; use case не знает про контроллер.

## Наивный код нарушает правило

Вернёмся к оформлению заказа. Естественный первый набросок: сервис сам создаёт репозиторий и зовёт его.

```python
from infrastructure.sql_order_repo import SqlOrderRepo

class PlaceOrderUseCase:
    def execute(self, order):
        repo = SqlOrderRepo()
        repo.save(order)
```

Здесь use case (внутреннее кольцо) импортирует `SqlOrderRepo` из инфраструктуры (внешнее кольцо). Стрелка зависимости указывает наружу - прямое нарушение правила. Имя `Sql` просочилось в центр системы: смена базы на Mongo заставит править файл с бизнес-сценарием.

## Инверсия зависимостей разворачивает стрелку

Чтобы развернуть зависимость, не разворачивая поток данных, use case должен зависеть не от конкретного репозитория, а от **абстракции**, которую он же и объявляет. Объявим интерфейс рядом с use case, во внутреннем кольце.

```python
from abc import ABC, abstractmethod

class OrderRepository(ABC):
    @abstractmethod
    def save(self, order) -> None: ...
```

Теперь use case зависит только от этого интерфейса и ничего не знает о SQL.

```python
class PlaceOrderUseCase:
    def __init__(self, repo: OrderRepository):
        self._repo = repo

    def execute(self, order):
        self._repo.save(order)
```

А конкретная реализация живёт в инфраструктуре и **реализует** интерфейс из центра.

```python
class SqlOrderRepo(OrderRepository):
    def save(self, order) -> None:
        ...
```

Стрелка развернулась: теперь `SqlOrderRepo` (внешнее кольцо) зависит от `OrderRepository` (внутреннее кольцо), а не наоборот. Это и есть инверсия зависимостей - буква D в SOLID. Use case диктует контракт; инфраструктура подстраивается.

## Граница - это интерфейс, который пересекает поток данных

Линия между кольцами называется границей (boundary). Через границу данные проходят только в простых, не зависящих от фреймворка формах: примитивы, dataclass, обычные структуры. Нельзя протащить через границу объект, который тащит за собой знание о внешнем кольце - например, ORM-модель, которая при доступе к полю лезет в базу, или объект HTTP-запроса фреймворка.

```python
@dataclass(frozen=True)
class OrderDTO:
    id: str
    total: float
```

Use case принимает и возвращает такие простые структуры. Контроллер снаружи сам переводит HTTP-запрос в `OrderDTO`, а презентер - результат use case в JSON. Граница остаётся чистой: ни одна деталь внешнего кольца не пересекает её внутрь.

<Callout type="warning">
Если объект, который вы передаёте внутрь через границу, при обращении к полю делает запрос в базу (ленивая загрузка ORM), значит граница дырявая: бизнес-сценарий незаметно зависит от базы. Передавайте через границу «мёртвые» структуры данных, а не живые объекты ORM.
</Callout>

## Это вас удивит: интерфейс принадлежит тому, кто его вызывает

Интуиция подсказывает, что интерфейс репозитория - это часть слоя данных, ведь он про хранение. В Clean Architecture - наоборот: интерфейс `OrderRepository` принадлежит use case, который его вызывает, а не реализации, которая его исполняет. Поэтому файл с интерфейсом лежит во внутреннем кольце, рядом с бизнес-логикой, а класс `SqlOrderRepo` - снаружи.

Это и есть механизм инверсии: владелец контракта - потребитель, а не поставщик. Когда контракт принадлежит центру, центр диктует, что ему нужно («умей сохранять заказ»), а инфраструктура обязана это предоставить любым способом - SQL, файл, заглушка в тесте. Стрелка `import` идёт от поставщика к потребителю, то есть внутрь, как и требует правило.

## Типичные ошибки

- **Складывать интерфейс репозитория в слой инфраструктуры.** Тогда use case импортирует его из внешнего кольца, и стрелка снова смотрит наружу. Интерфейс - часть внутреннего слоя; реализация - часть внешнего.
- **Протаскивать ORM-модели через границу.** Если use case оперирует объектами SQLAlchemy, он зависит от ORM. Переводите данные в простые dataclass на самой границе.
- **Путать направление потока данных с направлением зависимостей.** Данные во время выполнения текут наружу к ответу; это нормально. Правило ограничивает только `import` в исходном коде - он всегда внутрь.

<Detail summary="Почему это называется инверсией, а не просто абстракцией">

«Просто завести интерфейс» ещё не инверсия. Инверсия (Dependency Inversion Principle) - о том, что и высокоуровневый, и низкоуровневый модуль зависят от абстракции, и эта абстракция определена на стороне высокоуровневого. Обычный поток управления естественно идёт от высокого уровня к низкому: use case зовёт репозиторий. Без инверсии зависимость в коде шла бы туда же. Инверсия разворачивает именно зависимость в исходном коде, оставляя поток управления как был. Поэтому слово «inversion»: мы перевернули зависимость относительно потока управления.

</Detail>
