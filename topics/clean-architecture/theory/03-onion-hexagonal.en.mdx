# Three names, one idea, and important differences in emphasis

If you already absorbed the dependency rule, your first encounter with onion and hexagonal architecture may legitimately irritate you: it is the same thing, only the circles are a different shape. Partly that is true; all three grew from one principle: domain in the center, dependencies inward. But they appeared independently, from different authors, and each emphasizes something different: hexagonal thinks in ports and adapters, onion in concentric layers around the domain, and Clean Architecture generalizes both. Understanding the differences helps: in interviews and team debates these words are used as if they were different, and confusion about emphasis leads to bad decisions.

By the end of this section you will understand the ports-and-adapters model from hexagonal architecture, the ring structure of onion architecture with domain services, and how all three converge on one dependency diagram. That gives you the vocabulary used when designing backends.

## Mental model: domain in the center, technology on the edges

All three architectures share one core: the domain sits in the center and depends on nothing external; everything tied to concrete technology is pushed to the edges and depends on the center. The difference is which metaphor each chooses for the "edges".

## Hexagonal: ports and adapters

Alistair Cockburn proposed hexagonal architecture (also called "ports and adapters") in 2005 to remove the asymmetry of "input on top, database on bottom". In his model the application is a hexagon that talks to the world only through **ports**, and each port connects to the outside world through an **adapter**.

A port is an interface declared by the application. An adapter is a concrete implementation of that interface tied to a technology. Ports split into two kinds by direction. A driving port (inbound) is how the world drives the application: for example the use case interface that an HTTP controller calls. A driven port (outbound) is how the application drives the world: for example the repository interface that a PostgreSQL adapter implements.

```python
class OrderNotifier(ABC):
    @abstractmethod
    def order_placed(self, order_id: str) -> None: ...

class EmailNotifier(OrderNotifier):
    def order_placed(self, order_id: str) -> None:
        ...
```

`OrderNotifier` is a driven port declared by the application. `EmailNotifier` is an adapter. Tomorrow you need Slack instead of email; you write `SlackNotifier` without touching the application. The hexagon (not a rectangle) is chosen on purpose: the shape has no "top" or "bottom", all sides are equal, and no port is privileged.

## Onion: concentric layers with domain services

Jeffrey Palermo described onion architecture in 2008. Its emphasis is layers around the domain. At the center sits the domain model (entities, value objects); the next ring holds domain services, business logic that does not fit in a single entity; further out are application services (use case orchestration); and on the outside is everything else: UI, infrastructure, tests.

Palermo explicitly introduces domain services, which pure hexagonal wording does not. A domain service is a domain operation that naturally belongs to no single entity, for example transferring money between two accounts.

```python
class TransferService:
    def transfer(self, source, target, amount):
        source.withdraw(amount)
        target.deposit(amount)
```

`TransferService` lives in the domain ring: it works with domain objects and knows nothing about the database or DBMS transactions (that is the outer ring's job). The key rule of onion architecture matches the dependency rule word for word: outer rings depend on inner rings; inner rings know nothing about outer ones.

## This will surprise you: the difference is not the shape but the vocabulary

The most common mistake is treating these as three different architectures you must choose between. At the dependency-diagram level they are isomorphic: domain in the center, dependencies inward. They differ in vocabulary and what they emphasize.

So "we have hexagonal, not onion" is more often a terminology debate than a code debate. It is more useful to ask concretely: where are the boundaries, who declares interfaces, what sits in the center. Answers to those questions match across all three.

## Common mistakes

- **Treating the shape as meaningful.** Six sides of a hexagon do not mean "six ports". It is only a metaphor for no privileged side. You can have as many ports as you need.
- **Putting infrastructure in a domain service.** If `TransferService` opens a DB transaction itself, it stopped being domain and moved to the outer ring. Transactions are the application or infrastructure layer's job.
- **Choosing between architectures as incompatible.** They do not compete. You can say "ports and adapters" and still have onion-style domain services: it is the same design.

<Detail summary="Where does DDD fit?">

Onion architecture is historically tied to Domain-Driven Design: domain model, domain services, value objects (that is DDD vocabulary). Hexagonal architecture is neutral to DDD: it is about application boundaries, not what fills the center. In practice they are often combined: hexagonal ports set boundaries, and the center is organized by DDD. Clean Architecture is compatible with DDD too but does not require it. Architecture sets dependency direction; DDD is how you model the domain inside the center.

</Detail>
