Ваш сервис вызывает модель, чтобы извлечь из письма имя, сумму и дату, а потом записать их в базу. На тесте всё работает. В проде каждое двадцатое письмо ломает пайплайн: модель то добавила вежливое «Конечно, вот данные:» перед JSON, то завернула объект в ```` ```json ````, то выдала сумму строкой «1 480 руб.» вместо числа. Код, который делает `JSON.parse`, падает, и письмо теряется. Проблема не в извлечении - модель всё поняла правильно. Проблема в том, что ответ нельзя надёжно разобрать машиной.

К концу раздела Вы будете уметь получать вывод, который парсится с первого раза: задавать схему ответа, выбирать разделители, которые модель не спутает с данными, и пользоваться режимом структурного вывода современных API вместо того, чтобы вылавливать JSON регулярками.

## Зачем вообще структурный вывод

Свободный текст хорош для человека и плох для программы. Как только ответ модели становится входом другого кода - базы, фронтенда, следующего шага пайплайна, - его форма должна быть предсказуемой до символа. Структурный вывод - это договор: Вы заранее объявляете поля и их типы, а модель обязуется вернуть ровно эту структуру и ничего сверху.

Заметьте, что в правой части нет ни вступления, ни единиц измерения внутри числа, ни обёртки. Сумма - число, валюта - отдельное поле строкой. Именно такую форму ждёт `JSON.parse`, и именно её договор и гарантирует.

## Сначала разделители и явная инструкция

Минимальный уровень надёжности достигается без всякого специального API: чётко скажите, что вернуть, и где это начинается и кончается. Две техники работают вместе.

Первая - **разделители**. Оберните и входные данные, и желаемый вывод в теги, чтобы модель не перепутала, что обрабатывать, а что возвращать. У Claude хорошо работают XML-подобные теги.

```text
Извлеки поля из письма внутри <письмо>...</письмо>.
Верни результат внутри <json>...</json> и ничего вне этих тегов.

<письмо>
Здравствуйте! Оплатите 1480 руб. до 5 мая.
</письмо>
```

Вторая - **запрет на обёртку**. Модели любят быть вежливыми и предварять JSON фразой или заворачивать его в markdown-блок. Явно попросите: «верни только JSON, без markdown, без пояснений». Это убирает самый частый источник ошибок парсинга.

<Callout type="tip">
Дешёвый трюк - префилл начала ответа. Если API позволяет задать первые символы ответа модели, начните его с `{`. Модель вынуждена продолжать с открытой скобки, и вступительная болтовня становится невозможной. У Claude это делается через предзаполнение роли assistant.
</Callout>

## Схема надёжнее уговоров

Инструкция «верни JSON с полями amount и currency» работает, но держится на доброй воле модели: она *может* добавить поле, переименовать ключ или вернуть сумму строкой. Современные API убирают эту неопределённость режимом структурного вывода: Вы передаёте JSON Schema, и провайдер гарантирует, что ответ ей соответствует - валидный JSON, нужные ключи, правильные типы.

```json
{
  "type": "object",
  "properties": {
    "amount": { "type": "number" },
    "currency": { "type": "string" }
  },
  "required": ["amount", "currency"],
  "additionalProperties": false
}
```

С такой схемой `amount` гарантированно число, оба поля гарантированно присутствуют, а лишних ключей не будет из-за `additionalProperties: false`. У OpenAI это режим Structured Outputs со `strict: true`; у Anthropic эквивалентный результат получают, объявив инструмент (tool) со схемой `input_schema` и заставив модель его вызвать - аргументы вызова и есть Ваш типизированный объект. В обоих случаях гарантию даёт провайдер, а не вежливость модели.

## Краевые случаи

Схема гарантирует форму, но не смысл. Поле `amount` будет числом - но если в письме суммы нет, модель всё равно обязана вернуть число и впишет туда выдуманное или ноль. Чтобы отличать «нет данных» от «ноль», добавьте в схему явную возможность пустоты: сделайте поле необязательным или допустите `null`, и попросите ставить `null`, когда значения нет.

Вложенность и единицы - ещё одна ловушка. «Сумма 1480 руб.» легко превратить в `{"amount": 1480, "currency": "RUB"}`, но если схема требует одно поле-строку, модель вернёт `"1480 руб."`, и Вы снова парсите единицы вручную. Проектируйте схему так, чтобы число и единица жили в разных полях.

Очень большие или глубоко вложенные схемы модель соблюдает хуже и медленнее. Если объект распух, надёжнее разбить извлечение на несколько запросов с простыми схемами, чем требовать один гигантский объект.

Отдельная ловушка подстерегает при потоковой выдаче (streaming). Если Вы показываете ответ по мере генерации, в любой момент времени у Вас на руках лишь начало объекта - например, `{"amount": 1480, "curr` - и обычный `JSON.parse` на нём падает. Это не ошибка модели, а нормальное промежуточное состояние: полный валидный JSON появится только на последнем токене. Поэтому при стриминге либо ждут конца ответа перед разбором, либо используют инкрементальный парсер, умеющий читать незавершённый JSON. Гарантия схемы относится к финальному результату, а не к каждому промежуточному кадру.

## Типичные ошибки

- **Парсить весь ответ как JSON, не запретив обёртку.** Вежливое вступление или ```` ```json ```` ломают `JSON.parse`. Запретите markdown и пояснения или префиллом начните ответ с `{`.
- **Полагаться на инструкцию там, где есть режим схемы.** Если API умеет structured outputs, схема надёжнее уговора - используйте её.
- **Не предусмотреть «нет данных».** Без `null`/необязательного поля модель выдумает значение. Заложите явную пустоту в схему.
- **Складывать число и единицу в одно поле.** `"1480 руб."` придётся разбирать вручную. Разделите величину и единицу на разные поля.

<Detail summary="Чем structured outputs отличается от tool calling">

Технически это две стороны одной медали. Вызов инструмента (tool / function calling) заставляет модель вернуть аргументы по `input_schema` - то есть строго типизированный объект; именно так у Anthropic получают гарантированную структуру. Режим structured outputs у OpenAI напрямую привязывает к ответу `response_format` с JSON Schema и `strict: true`. В обоих случаях под капотом ограничивается генерация: на каждом шаге допускаются только токены, ведущие к валидному по схеме результату (constrained decoding). Поэтому гарантия формы - не «модель постаралась», а «другие токены физически не могли быть сгенерированы». Содержание полей это не проверяет - схема отвечает за форму, а не за правду в значениях.

</Detail>
