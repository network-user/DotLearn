# Объекты, ссылки, менеджеры и реквизиты

Здесь встроенный язык перестаёт быть «ещё одним процедурным языком» и становится языком платформы 1С. Данные в базе это не строки таблиц, а справочники, документы и регистры, и работают с ними через три сущности: менеджер, ссылку и объект. Спутать ссылку с объектом это классическая ошибка новичка, из-за которой то ничего не записывается, то каждая мелочь тормозит систему. Разберём, кто из них за что отвечает.

## Менеджер, объект, ссылка

Менеджер это точка входа к целому виду данных. `Справочники.Контрагенты` это менеджер справочника контрагентов: через него создают новые элементы, ищут существующие, получают пустые ссылки. Менеджер не хранит данные конкретной записи, он управляет видом в целом.

Ссылка это лёгкий указатель на конкретную запись в базе. Её удобно хранить, передавать, класть в реквизиты других объектов. Но ссылка предназначена в первую очередь для чтения: обратиться к реквизиту через ссылку можно, а вот изменить нельзя.

Объект это «толстая» версия записи: все реквизиты, все табличные части, и его можно менять и записывать. Объект тяжелее ссылки, поэтому его получают только когда действительно нужно что-то изменить.

## Чтение через ссылку

Если запись менять не нужно, а нужно лишь узнать значение реквизита, работают со ссылкой напрямую через точку.

```bsl
Сообщить(СсылкаКонтрагента.Наименование);
Сообщить(СсылкаКонтрагента.ИНН);
```

Важно понимать, что физически данные реквизита лежат не в самой ссылке, а в базе. Каждое обращение `Ссылка.Реквизит` это чтение из базы. Для одной-двух точек это незаметно, но в цикле по тысячам строк превращается в тысячи запросов (об этом ниже).

## Изменение через объект

Чтобы изменить запись, из ссылки получают объект методом `ПолучитьОбъект`, меняют его реквизиты и вызывают `Записать`.

```bsl
Объект = СсылкаКонтрагента.ПолучитьОбъект();
Объект.Наименование = "ООО Ромашка плюс";
Объект.Записать();
```

Попытка написать `СсылкаКонтрагента.Наименование = "..."` завершится ошибкой: ссылка неизменяема. Менять можно только объект.

Новую запись создают через менеджер методом `СоздатьЭлемент` (для справочника) или `СоздатьДокумент` (для документа). После заполнения и записи у объекта появляется ссылка на сохранённую запись.

```bsl
НовыйКонтрагент = Справочники.Контрагенты.СоздатьЭлемент();
НовыйКонтрагент.Наименование = "ООО Ромашка";
НовыйКонтрагент.Записать();

Ссылка = НовыйКонтрагент.Ссылка;
```

## Пустые ссылки и заполненность

Реквизит-ссылка может быть незаполненным. Пустую ссылку нужного типа даёт менеджер: `Справочники.Контрагенты.ПустаяСсылка()`. Проверять заполненность лучше не сравнением с пустой ссылкой вручную, а функцией `ЗначениеЗаполнено`, которая корректно распознаёт пустую ссылку любого типа.

```bsl
Если ЗначениеЗаполнено(Заказ.Контрагент) Тогда
    Сообщить(Заказ.Контрагент.Наименование);
КонецЕсли;
```

## Типичные ошибки

**Писать в реквизит ссылки.** `Ссылка.Реквизит = Значение` вызывает ошибку. Сначала `ПолучитьОбъект`, потом изменение, потом `Записать`.

**Забыть Записать.** Изменение реквизитов объекта в памяти без вызова `Записать` не попадёт в базу. Правки просто пропадут по завершении процедуры.

**Дёргать реквизиты через ссылку в цикле.** Обращение `Ссылка.Реквизит` внутри цикла по большому набору строк генерирует запрос к базе на каждой итерации. На тысячах строк это резко замедляет систему. Нужные данные получают одним запросом заранее.

**Сравнивать ссылку с Неопределено вместо ЗначениеЗаполнено.** Незаполненная ссылка это не `Неопределено`, а пустая ссылка своего типа. Универсальная проверка это `ЗначениеЗаполнено`.

<Callout type="warning" title="Коллекции и объекты присваиваются по ссылке">
Присваивание `Б = А`, где `А` это структура, массив или объект, не копирует данные, а даёт второе имя тому же значению в памяти. Изменение через `Б` видно и в `А`. Для независимой копии коллекции используют `Скопировать()`. Числа, строки и даты копируются по значению, а вот составные значения передаются по ссылке.
</Callout>

<Detail summary="Предопределённые элементы и поиск">

Часть элементов справочников и все значения перечислений задают в конфигураторе как предопределённые: к ним обращаются по имени прямо через менеджер, например `Перечисления.СтатусыЗаказов.Выполнен`. Искать элементы по данным помогают методы менеджера `НайтиПоКоду`, `НайтиПоНаименованию`, `НайтиПоРеквизиту`: все они возвращают ссылку на найденный элемент или пустую ссылку, если совпадений нет. Для массовых операций вместо перебора менеджером обычно берут запрос: он и быстрее, и нагляднее.

</Detail>
