# Three parts every message passes through

In the first section you called `logging.warning(...)` on the module directly and it worked like magic. But `basicConfig` assembled a small pipeline from ready-made parts behind the scenes, and while those parts are invisible, any non-standard request is confusing: "how do I write to file and console at once?", "why is the date printed in one place but not another?", "where did the word `root` come from?". The answer is always the same: a message passes through a chain of three objects, each responsible for one step.

By the end of this section you will assemble that pipeline by hand: create a named logger, attach a handler, set a formatter on the handler - and understand who does what. After that `logging` stops being incantations and becomes a constructor you build for your task.

## Mental model: a record travels a three-station pipeline

When you call `logger.info("hello")`, a `LogRecord` is born with text, level, time, and context. Then it travels the pipeline. **Logger** is the entry point: it decides whether to admit the record by level and passes it to its handlers. **Handler** is the destination: it decides *where* the record goes (console, file, network). **Formatter** is presentation: it turns the record into a string by template. One logger can have several handlers, each with its own formatter.

## Logger: named entry point

Instead of nameless `root`, create a named logger. The name usually matches the module via `logging.getLogger(__name__)`.

```python
import logging

logger = logging.getLogger("shop.orders")
print(logger.name)
print(logger is logging.getLogger("shop.orders"))
```

Prints `shop.orders` and `True`. The second matters: `getLogger` with the same name always returns **the same** object. Loggers are a global registry by name, not new objects per call. Configure in one place, get configured elsewhere - normal.

## Handler: where the record goes

A named logger prints nothing by itself - it needs a handler. The most common is `StreamHandler`, writing to a stream (default `stderr`).

```python
import logging
import sys

logger = logging.getLogger("shop.orders")
logger.setLevel(logging.INFO)
logger.addHandler(logging.StreamHandler(sys.stdout))

logger.info("order 42 placed")
```

Prints `order 42 placed`. We explicitly created a handler on `sys.stdout` and attached it. Without `addHandler` there is nowhere to write. Note the bare line without time or level - formatting is not set yet.

<Callout type="tip">
`logging.FileHandler("app.log")` writes to a file, `RotatingFileHandler` rotates by size. All handlers share the same interface, so once you know `StreamHandler` you can attach any handler.
</Callout>

## Formatter: how a record becomes a string

To give the line shape, set a formatter on the handler with a template referencing record fields via `%(name)s`.

```python
import logging
import sys

logger = logging.getLogger("shop.pay")
logger.setLevel(logging.INFO)
handler = logging.StreamHandler(sys.stdout)
handler.setFormatter(logging.Formatter("%(levelname)s [%(name)s] %(message)s"))
logger.addHandler(handler)

logger.info("payment accepted")
```

Prints `INFO [shop.pay] payment accepted`. The formatter inserted level and logger name around the text. Many fields exist: `%(asctime)s` for time, `%(funcName)s` for function name, `%(lineno)d` for line number. Formatter lives on the handler, not the logger - so the same record can look different in console and file.

## One logger - several destinations

The pipeline shines when one logger has several handlers, each formatting differently. This sandbox builds exactly that. Run the code: one `error` message arrives in two streams in two formats. Try raising the second handler's level to `logging.ERROR` via `h2.setLevel(...)` and add `logger.warning("verbose only")` - the warning should hit only the verbose handler.

<PyDemo

import sys

logger = logging.getLogger("demo.multi")
logger.setLevel(logging.INFO)
logger.handlers.clear()

h2 = logging.StreamHandler(sys.stdout)

logger.addHandler(h2)

logger.error("database unavailable")`}

`logger.handlers.clear()` at the start is for the sandbox: without it repeat runs stack handlers and messages print multiple times. That leads to the most common beginner mistake below.

## Edge case: duplicates from repeat addHandler

A logger keeps handlers in a list without deduplication. If setup code runs twice, the record prints twice.

```python
import logging
import sys

logger = logging.getLogger("dup")
logger.setLevel(logging.INFO)
logger.addHandler(logging.StreamHandler(sys.stdout))
logger.addHandler(logging.StreamHandler(sys.stdout))

logger.info("how many times")
```

Prints **two** lines `how many times`. Repeat `addHandler` **accumulates**, it does not replace: two identical handlers on `stdout` give two outputs. Configure logging once at startup, not per function call, and clear `logger.handlers` before adding if needed.

## Common mistakes

- **Attaching a handler on every request/function.** Handler setup is startup work. Doing it inside a frequently called handler stacks handlers and multiplies logs. Configure once at application init.
- **Setting formatter on the logger.** Loggers have no `setFormatter` - formatting lives on the handler. The logger does not know which stream a record will use; format depends on destination.
- **Expecting output from a bare logger.** `getLogger("x").info(...)` with no handler on the logger or ancestors is silent (except lastResort for `WARNING+`). No handler means nowhere to write.

<Detail summary="Why split three objects when one could do it all">

Separation gives independent configuration axes. Filtering level (`logger`) changes per subsystem: module `shop.pay` noisy at DEBUG, rest quiet at INFO. Destination (`handler`) changes per environment: rotated file in production, console locally. Appearance (`formatter`) changes per target: human-readable in console, JSON in a collection file. One object would force those three decisions together and make reuse across environments impossible. The price of flexibility is three objects instead of one - excessive until you need a second axis.

</Detail>
