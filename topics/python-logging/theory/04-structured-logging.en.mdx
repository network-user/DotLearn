# When logs number in the millions, text stops being readable

While the app runs on your machine, a log is lines you skim by eye. In production it is different: millions of records per day, read not by humans but by a collection system that indexes, filters, and builds dashboards. Ask that system "show all requests for user 42 slower than one second" and you discover that `user 42 waited 1300ms` forces you to extract numbers with regular expressions. Free text suits humans and frustrates machines.

Structured logs flip the priority: a record becomes named fields (`user_id=42`, `duration_ms=1300`) that the collection system indexes as-is. By the end of this section you will attach arbitrary fields via `extra`, emit machine-readable JSON, and configure everything with one dictionary through `dictConfig` instead of scattered calls. This is the final step from "logging works" to "logging is production-ready".

## Mental model: a record is fields; text is just one of them

Free-form logs glue everything into one string and context dissolves in text. A structured log keeps message and context **separate**: `message` is the human phrase, and beside it fields like `user_id`, `request_id`, `duration_ms`. The formatter decides presentation: humans see a line, the collection system sees JSON where each field stays a key.

The win: when fields are separate, "find by user_id" is trivial; from free text you scrape the same value with parsing. Structure is a pre-answered question to future log queries.

## extra: attach fields to a record

Arbitrary fields attach via `extra` - a dict whose keys become record attributes available in the formatter template.

```python
import logging
import sys

logger = logging.getLogger("api")
logger.setLevel(logging.INFO)
handler = logging.StreamHandler(sys.stdout)
handler.setFormatter(logging.Formatter("%(message)s user=%(user_id)s"))
logger.addHandler(handler)

logger.info("request handled", extra={"user_id": 42})
```

Prints `request handled user=42`. Key `user_id` from `extra` became a record field and the formatter substituted it at `%(user_id)s`. Context travels separately from message text - and can be rendered as a line or JSON.

<Callout type="warning">
If the formatter references `%(user_id)s` but `extra` omits that key, formatting raises `KeyError`. Guarantee keys on every call or supply defaults via a filter.
</Callout>

## Machine-readable output: JSON records

A custom formatter can return JSON where each field is a separate key. Subclass `logging.Formatter` and override `format`.

```python
import logging
import sys
import json

class JsonFormatter(logging.Formatter):
    def format(self, record):
        return json.dumps({
            "level": record.levelname,
            "msg": record.getMessage(),
            "user_id": getattr(record, "user_id", None),
        })

logger = logging.getLogger("api.json")
logger.setLevel(logging.INFO)
h = logging.StreamHandler(sys.stdout)
h.setFormatter(JsonFormatter())
logger.addHandler(h)
logger.info("ok", extra={"user_id": 7})
```

Prints `{"level": "INFO", "msg": "ok", "user_id": 7}`. Valid JSON: the collection system parses without regexp, and `user_id` lands in the index. In production you usually use a library, but the mechanism is the same.

## Configuration in one dict: dictConfig

When settings multiply, scattered `addHandler`/`setLevel` calls become soup. `logging.config.dictConfig` accepts one dict describing formatters, handlers, and loggers entirely.

```python
import logging
import sys
from logging.config import dictConfig

dictConfig({
    "version": 1,
    "formatters": {"plain": {"format": "%(levelname)s %(message)s"}},
    "handlers": {"console": {"class": "logging.StreamHandler",
                              "formatter": "plain", "stream": sys.stdout}},
    "root": {"level": "INFO", "handlers": ["console"]},
})
logging.getLogger("app").info("configured by dict")
```

Prints `INFO configured by dict`. The whole pipeline from earlier sections - formatter, handler, root level - assembled declaratively in one call. Key `"version": 1` is required: schema format version, not your app version. Keep such a dict beside config and vary it per environment without code changes.

## Edge case: reserved names in extra

It is tempting to put `message` or `args` in `extra`. Those names are already taken by internal record attributes; overwriting them raises `KeyError`.

```python
import logging

logger = logging.getLogger("reserved")
logger.addHandler(logging.NullHandler())
try:
    logger.info("hello", extra={"message": "overwrite builtin"})
except KeyError as error:
    print("KeyError:", error)
```

Prints `KeyError: 'Attempt to overwrite ...'`. Not every key works: `extra` protects service fields (`message`, `asctime`, `levelname`, `args`, and others). Use your own names - `user_id`, `request_id`, `duration_ms` - and collisions do not happen.

## Common mistakes

- **Putting variables in message text instead of `extra`.** `logger.info(f"user {uid} ok")` hides data in the string again. Write `logger.info("ok", extra={"user_id": uid})` so the collection system gets a field, not text to parse.
- **Forgetting `"version": 1` in `dictConfig`.** Without it the call fails - required schema version, fixed at 1.
- **Reconfiguring with `basicConfig` after startup and wondering why nothing changed.** `basicConfig` runs once; repeat without `force=True` is ignored. For live changes use `force=True` or `dictConfig`.

<Detail summary="basicConfig versus dictConfig: when to use which">

`basicConfig` is quick start: one line, one root handler, sensible defaults. Its ceiling is simple scenarios with console and one format. `dictConfig` is declarative and complete: several formatters, several handlers, per-subsystem levels and propagation - all in one dict you can externalize and vary per environment without code edits. Practical rule: scripts and prototypes - `basicConfig`; apps heading to production with multiple environments - `dictConfig` (or `fileConfig` for INI fans). Avoid half-measures: mixing manual `addHandler` with `dictConfig` is a path to duplication and surprises.

</Detail>
