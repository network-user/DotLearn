A payment succeeded on your machine, `processed order 42` flashed in the terminal, and the code shipped to the server. A week later the client writes: money was charged but the order is missing. You open the server - the terminal is empty, the process restarted long ago, and that evening is gone. Diagnostics existed only until they were needed. Logging is the habit of leaving a trail that outlives the process and later tells you honestly what happened, on which server, and for which user.

**What you will learn to do.** Replace one-off `print` calls with controlled output and severity levels; build your own pipeline from logger, handler, and formatter to write to console and file at once; understand whether a message will reach output and whether it will duplicate; turn logs from free text into named fields that a log collection system can index; configure all of this with one dictionary instead of scattered calls.

**Topic route.** Four sections, each with a sandbox where Python runs right in the browser:

1. **`print` vs log** - why terminal output disappears with the process and what `logging` adds.
2. **Logger, Handler, Formatter** - three objects every message passes through on the way to its destination.
3. **Levels and propagation** - how importance filters and tree traversal decide what is visible.
4. **Structured logs** - records as fields like `user_id`, `duration_ms`, ready for production and machine reading.

**What you need beforehand.** Nothing to install: Python runs on the page. You should be comfortable writing simple functions and calling them - everything about `logging` we will cover step by step, moving from a single `print` to a configured pipeline.
