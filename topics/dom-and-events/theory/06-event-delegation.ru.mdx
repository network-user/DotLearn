# Один обработчик вместо тысячи

В ленте товаров пятьсот карточек, и на каждой кнопка «В избранное». Прямолинейное решение - повесить обработчик клика на каждую кнопку - работает, но расточительно: пятьсот подписок, пятьсот функций в памяти. Хуже другое: подгрузили ещё сто карточек скроллом - и на их кнопках обработчиков нет, ведь в момент подписки этих кнопок не существовало. Приходится вешать обработчики заново на каждую новую пачку. Есть приём, который снимает обе проблемы разом и держится на всплытии из предыдущих глав. Он называется делегированием.

К концу главы вы будете ловить события от множества элементов одним обработчиком на их общем предке, определять настоящий источник клика через `event.target` и `closest`, и понимать, почему делегирование само собой работает для элементов, добавленных уже после подписки.

## Идея: слушает предок, а решает target

Раз клик по кнопке всплывает через всех её предков, обработчик не обязан висеть на самой кнопке. Его можно повесить один раз на общий контейнер - список, - и он услышит клики по любой кнопке внутри. А чтобы понять, по какой именно кнопке кликнули, обработчик заглядывает в `event.target` - тот самый глубокий элемент, где событие случилось.

Вот базовый каркас делегирования. Обработчик один, висит на списке, а внутри разбирается, откуда пришёл клик:

```js
list.addEventListener('click', (event) => {
  const card = event.target.closest('.card');
  if (!card) {
    return;
  }
  console.log('клик по карточке', card.dataset.id);
});
```

Пятьсот карточек - один обработчик, одна функция в памяти. И когда скролл догрузит ещё сто карточек, они окажутся внутри того же списка, их клики всплывут к тому же обработчику, и всё заработает без единой новой подписки.

Экономия тут не только в строчках кода. Пятьсот отдельных обработчиков - это пятьсот функций, которые браузер держит в памяти и привязывает к пятистам элементам; при удалении элементов их ещё надо не забыть отписать, иначе течёт память. Делегирование снимает всю эту бухгалтерию: одна подписка живёт на контейнере сколько угодно, а элементы внутри свободно появляются и исчезают.

## closest поднимается от target вверх

В примере есть ключевая деталь - `event.target.closest('.card')`. Зачем она? Проблема в том, что `target` - это самый глубокий элемент, а клик мог прийтись не по самой карточке, а по картинке или тексту внутри неё. Тогда `target` - это `img` или `span`, а не `.card`. Метод `closest` решает это: он идёт от `target` вверх по предкам и возвращает первый элемент, подходящий под селектор, - или `null`, если такого предка нет.

Так клик по любой точке внутри карточки - по картинке, по названию, по цене - через `closest('.card')` всё равно приводит к самой карточке. А проверка `if (!card) return` отсеивает клики мимо карточек: щёлкнули по пустому месту списка - `closest` вернул `null`, и обработчик молча выходит.

<Callout type="tip" title="closest включает сам элемент">
closest начинает проверку с самого target, а не с его родителя. Если по селектору подходит уже сам target, closest вернёт его же. Поэтому клик прямо по элементу с нужным классом тоже находится - отдельный случай городить не нужно.
</Callout>

Иногда вместо `closest` в чужом коде встречается `event.target.matches('.card')`. Разница в том, что `matches` проверяет только сам `target` и возвращает `true` или `false`, а `closest` ещё и поднимается по предкам и возвращает найденный элемент. Для делегирования почти всегда нужен именно `closest`: клик приходит по вложенному узлу, и надо добраться до карточки, а не убедиться, что кликнули ровно по ней.

## Приоритет: сначала частное действие, потом общее

Часто внутри одного контейнера живут разные действия. В карточке есть и кнопка «удалить», и остальное тело: клик по кнопке должен удалять карточку, клик по телу - открывать её. Один обработчик разруливает оба случая, и порядок проверок здесь важен.

```js
list.addEventListener('click', (event) => {
  if (event.target.closest('.delete-btn')) {
    console.log('удаляем карточку');
    return;
  }
  if (event.target.closest('.card')) {
    console.log('открываем карточку');
  }
});
```

Кнопка удаления лежит внутри карточки, поэтому у клика по ней в цепочке предков есть и `.delete-btn`, и `.card`. Если сначала проверить `.card`, любой клик, включая клик по кнопке, посчитается за «открыть». Поэтому более частное и глубокое действие проверяют первым, а общее - вторым. Это прямое следствие того, что `closest` идёт снизу вверх: ближайший к клику предок важнее дальнего.

## Данные храните на элементе, а не в замыкании

Обработчик один на всех, поэтому «какой товар кликнули» он должен узнавать из самого элемента, а не из внешней переменной. Здесь и пригождаются `data-` атрибуты из третьей главы: id товара кладут в разметку, а обработчик читает его из `dataset` найденной карточки.

```js
list.addEventListener('click', (event) => {
  const card = event.target.closest('.card');
  if (!card) return;
  const id = card.dataset.id;
  addToFavorites(id);
});
```

Так один обработчик обслуживает любое число карточек: вся разница между ними - в их `data-id`, а логика общая. Это и есть причина, по которой делегирование хорошо масштабируется.

## Собираем всё вместе: список задач

Соберём делегирование целиком на списке задач, где у каждой задачи есть кнопка «удалить». Один обработчик на списке разбирает, куда пришёлся клик, и действует:

```js
list.addEventListener('click', (event) => {
  const deleteBtn = event.target.closest('.delete');
  if (deleteBtn) {
    deleteBtn.closest('.task').remove();
    return;
  }
  const task = event.target.closest('.task');
  if (task) {
    task.classList.toggle('done');
  }
});
```

Клик по кнопке удаления находит её через `closest('.delete')`, поднимается от неё к самой задаче и удаляет узел. Клик по остальному телу задачи кнопку не находит, зато находит `.task` и переключает отметку «сделано». Один обработчик, две ветки, любое число задач - и добавленные позже работают сразу, ведь их клики всплывают сюда же. Порядок веток важен: удаление проверяется первым, иначе клик по кнопке ушёл бы в переключение отметки.

<MarginNote label="Граница">Не вешайте делегированный обработчик на document, если хватает списка: обработчик на корне дёргается на каждый клик страницы и зря гоняет closest.</MarginNote>

## Частые ошибки

Первая ошибка - забыть про `closest` и сравнивать сам `target`. Проверка `if (event.target.classList.contains('card'))` сработает, только если кликнули ровно по карточке, но не по картинке внутри неё. Клики по вложенным элементам она пропустит, и карточка «не нажимается» в половине точек.

Вторая ошибка - неправильный порядок проверок, когда действий несколько. Проверили общий контейнер раньше вложенной кнопки - и кнопка перестаёт работать как отдельное действие. Идите от частного к общему.

Третья ошибка - вешать делегированный обработчик слишком высоко, на `document`, когда хватило бы списка. Технически это работает, но обработчик на `document` будет получать вообще все клики страницы и на каждом дёргать `closest`. Вешайте на ближайший разумный общий контейнер, а не на корень.

Приятная особенность делегирования в том, что оно не требует ничего нового поверх уже пройденного. Оно целиком собрано из трёх кирпичей предыдущих глав: события всплывают снизу вверх, поэтому предок слышит потомков; у события есть настоящий источник, и метод подъёма по предкам умеет добраться до нужного узла; а данные элемента лежат прямо на нём, в атрибутах. Сложите эти три факта, и получится приём, который экономит память, переживает добавление и удаление элементов и обходится одной подпиской там, где наивный подход просил бы сотни. В этом и красота хороших базовых механик: их не нужно расширять ради практической пользы, достаточно увидеть, как они складываются.

Стоит держать в уме и обратную сторону. Делегирование добавляет один слой непрямоты: обработчик висит не там, где происходит действие, и коллеге, который читает код впервые, придётся сделать мысленный шаг от контейнера к реальной цели. Пока элементов много и они меняются, эта цена оправдана с лихвой. Но для единственной кнопки, которая никуда не денется, прямой обработчик проще и честнее, и городить ради неё делегирование не нужно. Как и почти всё в инженерии, делегирование - это обмен: вы платите небольшой добавкой к сложности чтения и выигрываете в памяти, живучести и числе подписок. Пока на другой чаше весов много однотипных, меняющихся элементов, обмен выгоден. Как только там остаётся один статичный элемент, весы качаются в обратную сторону. Умение видеть этот баланс, а не применять приём везде подряд, и отличает зрелый код от механически модного.

<Detail summary="Когда делегирование не подходит">
Делегирование опирается на всплытие, поэтому не работает для событий, которые не всплывают. Классические примеры - focus и blur (у них есть всплывающие двойники focusin и focusout), а также load и mouseenter. Ещё делегирование не даст выигрыша, если элемент ровно один: одному элементу - один прямой обработчик, без посредника. Приём хорош именно для множества однотипных элементов внутри общего контейнера, особенно когда их состав меняется во время работы страницы.
</Detail>
