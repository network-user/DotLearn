# Событий слишком много, а браузеру ещё рисовать

Поле поиска на сайте объявлений подсказывает города по мере ввода. Пользователь печатает «Новосиб», и событие `input` срабатывает на каждую букву - семь раз за секунду. Если на каждое событие лететь на сервер за подсказками, получится семь запросов там, где хватило бы одного. С прокруткой та же беда: `scroll` стреляет десятки раз в секунду, и тяжёлый обработчик на каждом кадре превращает плавную ленту в рывки. Частые события - это ресурс, которым надо распоряжаться, а не тратить весь до дна.

К концу главы вы будете гасить лишние срабатывания двумя классическими приёмами - debounce и throttle, - связывать части приложения кастомными событиями и понимать, почему одни изменения DOM дёшевы, а другие заставляют браузер мучительно пересчитывать всю страницу.

## debounce: дождись паузы

Идея debounce проста: не реагируй на каждое событие, дождись, пока они прекратятся. Пользователь печатает - таймер сбрасывается на каждой букве; как только человек остановился на треть секунды, срабатывает единственная реакция. Семь нажатий - один запрос, посланный уже по готовой строке.

Механика держится на паре `setTimeout` и `clearTimeout`: каждое новое событие отменяет прошлый отложенный вызов и ставит новый. Выживает только последний - тот, после которого паузы хватило.

```js
const debounce = (fn, wait) => {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), wait);
  };
};

const search = debounce((text) => sendRequest(text), 300);
input.addEventListener('input', (event) => search(event.target.value));
```

debounce берут там, где важен только финальный результат серии: поиск по мере ввода, автосохранение черновика, пересчёт после изменения размера окна. Промежуточные события можно смело выбросить.

У debounce есть два варианта поведения. Чаще используют «задний фронт»: реакция после паузы, как в примере выше. Реже - «передний фронт»: сработать сразу на первом событии, а последующие в серии гасить. Первый вариант отвечает интуиции «дождись, пока допечатают», поэтому в поиске по вводу берут именно его.

## throttle: не чаще, чем раз в интервал

throttle решает другую задачу: события идут потоком, и реагировать надо регулярно, но не на каждое. Приём пропускает первое событие, а дальше молчит, пока не пройдёт заданный интервал, - и только тогда снова пускает одно. Поток из сотни событий превращается в ровный ритм: одно срабатывание в интервал.

Разница с debounce принципиальная. debounce ждёт тишины и может не сработать вообще, пока события сыплются без остановки. throttle срабатывает регулярно во время потока. Поэтому для прокрутки, где надо обновлять индикатор позиции прямо в процессе, берут throttle, а не debounce.

Наивная реализация throttle помнит время последнего срабатывания и пропускает событие, только если интервал уже прошёл:

```js
const throttle = (fn, interval) => {
  let last = 0;
  return (...args) => {
    const now = Date.now();
    if (now - last >= interval) {
      last = now;
      fn(...args);
    }
  };
};

const onScroll = throttle(() => updateProgress(), 100);
window.addEventListener('scroll', onScroll);
```

Первое событие срабатывает сразу, ведь `last` равен нулю, а дальше проходят только те, между которыми набежал целый интервал. Индикатор прокрутки обновляется ровно, не чаще десяти раз в секунду, сколько бы событий `scroll` ни прилетело.

## Кастомные события: свой сигнал по имени

События бывают не только браузерные. Вы можете придумать своё - например, «товар добавлен в корзину» - и разослать его. Это способ развязать части приложения: корзина кричит «во мне прибавилось», а кто угодно - счётчик в шапке, всплывающая подсказка - слушает этот крик, не зная про корзину напрямую.

```js
const event = new CustomEvent('cart:add', { detail: { id: 42, price: 450 } });
cart.dispatchEvent(event);

cart.addEventListener('cart:add', (event) => {
  console.log('добавлен товар', event.detail.id);
});
```

Своё событие создают через `CustomEvent`, передавая полезные данные в поле `detail`. Рассылают методом `dispatchEvent`, а слушают тем же `addEventListener`, что и обычные события. Имя выбирают с префиксом-пространством вроде `cart:add`, чтобы не столкнуться с встроенными.

Слушателей может быть сколько угодно, и они ничего не знают друг о друге. Счётчик в шапке и всплывающая подсказка подписываются на то же событие независимо:

```js
header.addEventListener('cart:add', () => refreshCounter());
toast.addEventListener('cart:add', (event) => showToast(event.detail.price));
```

Корзине не нужно знать про счётчик и подсказку, она лишь рассылает сигнал. Добавить нового слушателя или убрать старого можно, не трогая код корзины. Так события становятся клеем, который держит модули вместе, но не сцепляет их намертво.

## Почему один append дёшев, а тысяча в цикле - нет

Осталось понять, почему изменения DOM бывают дорогими. Когда JavaScript меняет дерево - добавляет узел, переключает класс, - браузер не сразу рисует результат. Он проходит конвейер: пересчитывает, какие стили теперь применимы, затем заново вычисляет раскладку (где и какого размера каждый элемент) - это и есть reflow, - потом закрашивает пиксели и склеивает слои.

Reflow - самый дорогой этап, потому что размер и положение одного элемента влияют на соседей: раздулся один блок - поехала вся страница. Браузер умён и старается копить изменения, чтобы сделать один reflow вместо многих. Но эту оптимизацию легко сломать.

Ломает её чтение геометрии. Свойства вроде `offsetHeight`, `offsetWidth` и метод `getBoundingClientRect` должны вернуть точное, актуальное значение. Если к этому моменту накопились несделанные изменения, браузер вынужден немедленно провести reflow, чтобы ответить правильно. А теперь представьте цикл, где на каждой итерации вы что-то меняете, а потом сразу читаете `offsetHeight`: каждое чтение запускает reflow, и цикл на тысячу элементов делает тысячу пересчётов. Это называют layout thrashing.

<Callout type="warning" title="Не чередуйте запись и чтение в цикле">
Изменение DOM, за которым сразу идёт чтение геометрии (offsetHeight, getBoundingClientRect), заставляет браузер сделать reflow немедленно. В цикле это превращается в сотни пересчётов. Сначала все чтения, потом все записи - тогда reflow будет один.
</Callout>

Лечится это разделением фаз: сначала прочитайте всё, что нужно, из DOM, потом одним заходом внесите все изменения. На практике плохой вариант читает и пишет вперемешку:

```js
for (const box of boxes) {
  box.style.width = box.offsetWidth + 10 + 'px';
}
```

Каждая итерация пишет ширину, а следующая тут же читает `offsetWidth` и вынуждает reflow. Хороший вариант сперва собирает все размеры, потом применяет их все:

```js
const widths = boxes.map((box) => box.offsetWidth);
boxes.forEach((box, i) => {
  box.style.width = widths[i] + 10 + 'px';
});
```

Теперь все чтения идут до всех записей, и браузер обходится одним пересчётом. И вставку множества узлов делайте пакетно - через `DocumentFragment` из второй главы или собирая узлы в родителе вне дерева, - чтобы страница пересчиталась один раз, а не на каждой вставке.

## Частые ошибки

Первая ошибка - перепутать debounce и throttle. На прокрутке debounce сработает только когда пользователь остановится, и индикатор позиции будет отставать весь скролл. На поиске throttle пошлёт запрос посреди слова. Сначала решите, нужен вам финал серии или регулярные отметки.

Вторая ошибка - создавать debounce заново на каждое событие. Если обернуть функцию в `debounce` прямо внутри обработчика, каждый раз родится новый таймер со своим счётчиком, и гашение не сработает. Оборачивайте один раз, снаружи, а внутри обработчика зовите уже обёрнутую функцию.

Третья ошибка - трогать раскладку в цикле. Меняете стиль и тут же читаете `offsetHeight` на каждой итерации - получаете layout thrashing и подвисания. Разделяйте чтение и запись.

У всех приёмов этой главы общий мотив: событий и изменений много, а времени у браузера на кадр мало, поэтому лишнюю работу надо не делать вовсе, а не делать быстро. debounce и throttle выбрасывают избыточные срабатывания, так и не доводя их до тяжёлой работы. Разделение чтения и записи не ускоряет reflow, а сокращает число пересчётов. Пакетная вставка превращает сотню перерисовок в одну. Ни один из этих приёмов не про то, чтобы код бежал быстрее; все они про то, чтобы ненужная работа просто не начиналась. Это, пожалуй, самый переносимый навык производительности: он одинаково полезен и в вёрстке, и в бэкенде, и в любой системе, которая захлёбывается не от сложности одной операции, а от их количества.

Кастомные события стоят в этом ряду чуть особняком, но служат той же цели здорового приложения. Они не про скорость, а про то, чтобы части кода не сцеплялись намертво. Когда корзина просто объявляет «во мне прибавилось», а слушатели сами решают, что с этим делать, каждый кусок можно менять, не боясь задеть остальные. Приложение, которое легко менять, в итоге и работает лучше: у вас остаётся силы на то, чтобы вовремя убрать лишнюю работу, а не распутывать, кто кого вызывает.

<Detail summary="requestAnimationFrame и правильный момент для изменений">
Когда изменения DOM связаны с анимацией или прокруткой, их стоит вносить не когда попало, а прямо перед следующей отрисовкой кадра. Для этого есть requestAnimationFrame: он вызывает переданную функцию перед ближайшим кадром, синхронно с циклом отрисовки браузера. Обработчик scroll, который через throttle обновляет позицию, часто пишут так, чтобы сама запись в DOM происходила внутри requestAnimationFrame - тогда браузер делает работу в свой ритм и не дёргает раскладку лишний раз.
</Detail>
