# Списки и ключи

Каталог товаров, лента сообщений, список задач - данные почти всегда приходят массивом, а показать их нужно как набор одинаковых по форме элементов. Писать сто карточек руками невозможно, да и число их заранее неизвестно. React решает это красиво: вы берёте массив данных и превращаете его в массив элементов методом `map`, знакомым по JavaScript. Разметка списка получается из данных, а не пишется вручную.

Есть только одна деталь, без которой React начнёт ругаться в консоли и, что хуже, путать строки при обновлениях, - ключи. В этом уроке разберём, как выводить списки и почему у каждого элемента должен быть свой `key`.

## map превращает данные в разметку

Внутри фигурных скобок JSX можно поставить массив элементов, и React выведет их подряд. А массив элементов удобнее всего получить из массива данных методом `map`: он проходит по каждому элементу и возвращает для него кусочек JSX.

```jsx
function ProductList({ products }) {
  return (
    <ul>
      {products.map((product) => (
        <li key={product.id}>{product.name}</li>
      ))}
    </ul>
  );
}
```

Если в `products` лежат три товара, `map` вернёт три элемента `<li>`, и React выведет их внутри `<ul>`. Данные меняются - меняется и список: добавился товар в массив, появилась строка на экране. Ту же связку вы будете писать постоянно, ведь почти любой экран - это список чего-нибудь.

## Зачем нужен key

Каждому элементу списка React просит дать `key` - строку или число, уникальное среди соседей. Ключ отвечает на вопрос, который встаёт при каждом обновлении: какой элемент нового списка соответствует какому элементу старого. По ключу React понимает, что «строка с ключом 7» - та же самая, что была, даже если она сдвинулась, и её не нужно пересоздавать, а нужно лишь подвинуть.

Без ключей React вынужден сопоставлять элементы по их позиции в списке, и на любых вставках, удалениях и перестановках это приводит к лишней работе и к багам с сохранённым состоянием строк. Поэтому ключ берут из данных - из устойчивого идентификатора, который не меняется от рендера к рендеру. Чаще всего это поле `id` из базы.

<Callout type="warning" title="key - не проп">
Ключ выглядит как атрибут, но пропсом он не становится: внутри компонента `product.key` вы не прочитаете. Это служебная подсказка для самого React. Если то же значение нужно и внутри, передайте его отдельным пропом: `<Row id={product.id} key={product.id} />`.
</Callout>

## Почему индекс - плохой ключ

Соблазн велик: раз `map` даёт второй аргумент - индекс, почему бы не взять `key={index}`? Пока список неизменен, это работает. Но стоит вставить элемент в начало, удалить из середины или отсортировать - и индексы «съезжают»: строка, которая была третьей, станет второй, её ключ сменится с 2 на 1, и React решит, что это другой элемент.

Последствия коварны, потому что видны не сразу. Классический пример: список задач с чекбоксами. Вы отметили вторую задачу, затем удалили первую. С индексными ключами галочка «прилипает» к позиции, а не к задаче, и после удаления отмеченной окажется совсем не та строка. Данные при этом правильные - врёт именно внутреннее состояние элементов, привязанное к съехавшим ключам.

Вывод: индекс годится в ключ, только если список статичен - не переупорядочивается, не растёт и не сокращается. Как только он может измениться, ключ должен приходить из самих данных.

## Краевые случаи

Ключи обязаны быть уникальными лишь среди прямых соседей - в разных списках ключ 1 может встречаться сколько угодно раз. Если в данных нет готового идентификатора, его придумывают заранее, при создании элемента (например, генератором id), а не на лету в рендере: ключ, посчитанный случайно на каждом рендере, сменится каждый раз и лишит React возможности сопоставлять строки. И ставить `key` нужно на самый внешний элемент, который возвращает `map`, а не на что-то внутри него.

## Типичные ошибки

**Забыть key.** React выведет список, но предупредит в консоли и будет сопоставлять строки хуже. Ключ ставят всегда.

**Индекс как ключ в динамическом списке.** Работает, пока ничего не двигается; ломает состояние строк при вставках, удалениях и сортировке. Берите ключ из данных.

**Случайный ключ в рендере.** `key={Math.random()}` меняется каждый рендер, поэтому React каждый раз пересоздаёт все строки - медленно и со сбросом их состояния.

**Читать key как проп.** Внутри компонента ключа нет. Нужно это же значение - передайте его отдельным пропом.

<Detail summary="Что именно ломается при плохих ключах">

Дело в том, что React связывает с каждым элементом списка его внутреннее состояние: значение поля ввода, раскрытость деталей, фокус. Сопоставляются старый и новый элементы по ключу. Если ключ устойчив, состояние остаётся с той же логической строкой, куда бы она ни переехала. Если ключ - индекс, состояние привязывается к позиции: удалили строку сверху - и всё, что ниже, «унаследует» чужие значения полей. Снаружи данные выглядят верными, а формы ведут себя необъяснимо. Поэтому правильный ключ - это не педантизм, а защита от целого класса плавающих багов.

</Detail>
