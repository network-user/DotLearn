# Состояние и useState

Пропсы приходят сверху и меняться внутри компонента не могут. Но интерфейс живой: счётчик лайков растёт, чекбокс переключается, поле ввода наполняется буквами. Этим данным нужно где-то жить - в памяти самого компонента, которая переживает перерисовки и, меняясь, заставляет экран обновиться. Такая память называется состоянием (state), а заводят её хуком `useState`.

Это ваш первый хук, и в этом уроке мы разберём его обзорно: как объявить состояние, как его менять и почему менять его нужно только через специальный сеттер. Глубокое погружение в хуки - `useEffect`, зависимости, кастомные хуки - ждёт в соседней теме `react-hooks`; здесь наша цель - научиться делать компонент интерактивным.

## useState возвращает пару: значение и сеттер

Состояние объявляют одной строкой. Хук `useState` принимает начальное значение и возвращает массив из двух элементов: текущее значение и функцию, которой это значение меняют. Пару тут же разбирают деструктуризацией:

```jsx
import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(count + 1)}>
      Лайков: {count}
    </button>
  );
}
```

Читается это так: «заведи состояние с начальным значением 0; текущее значение назови `count`, а функцию для его смены - `setCount`». Имя сеттеру по традиции дают из `set` и имени переменной: `count` и `setCount`, `name` и `setName`. При клике вызывается `setCount(count + 1)`, и вот тут начинается магия React.

## Сеттер делает две вещи

Вызов `setCount(1)` не просто кладёт единицу в переменную. Он делает два дела: сохраняет новое значение в памяти, которую React держит для этого компонента, и планирует повторный рендер. На следующем рендере функция `Counter` запустится заново, `useState` вернёт уже сохранённое значение (единицу вместо нуля), и разметка соберётся с новым числом. Так замыкается цикл: событие меняет состояние, состояние вызывает рендер, рендер показывает свежие данные.

Начальное значение из `useState(0)` React использует только один раз - при самом первом рендере. Дальше он хранит своё, актуальное значение и на аргумент больше не смотрит. Это регулярно удивляет: передать в `useState` свежий проп и ждать, что состояние «подхватит» его при следующем рендере, не выйдет - начальное значение читается лишь однажды.

## Менять - только через сеттер

Вот главное правило состояния: **новое значение задают сеттером, а не присваиванием**. Строчка `count = count + 1` бесполезна: она поменяет локальную переменную, но React об этом не узнает и перерисовку не запустит - экран останется прежним. Только сеттер сообщает React, что данные изменились.

С объектами и массивами то же самое, но ловушка коварнее. Их нельзя менять на месте - нужно передать сеттеру новый объект или массив:

```jsx
const [items, setItems] = useState(['хлеб']);

function add() {
  setItems([...items, 'молоко']);
}
```

Запись `items.push('молоко')` изменила бы массив внутри, но не создала бы новый - React сравнивает ссылку и, увидев ту же ссылку, решит, что ничего не поменялось, и не перерисует. Поэтому массив пересобирают спред-синтаксисом `[...items, новый]`, а объект - `{ ...obj, поле: значение }`. Замена вместо мутации - привычка, которую стоит выработать сразу.

<Callout type="warning" title="Состояние обновляется не мгновенно">
Сразу после `setCount(count + 1)` переменная `count` в этой же функции ещё старая. Сеттер не переписывает `count` на месте - он готовит значение для следующего рендера. Внутри текущего вызова вы по-прежнему видите старое число. Почему так и что делать, если нужно несколько раз подряд поменять состояние, разбирает следующий урок про рендер.
</Callout>

## Краевые случаи

Состояний у компонента может быть сколько угодно - просто зовите `useState` несколько раз, по одному на каждую независимую величину:

```jsx
const [name, setName] = useState('');
const [age, setAge] = useState(0);
const [agree, setAgree] = useState(false);
```

React различает их по порядку вызова, и поэтому у хуков есть железное правило: их зовут всегда на верхнем уровне компонента и всегда в одном и том же порядке. Нельзя прятать `useState` внутрь `if` или цикла - иначе на разных рендерах порядок собьётся и React перепутает состояния. Подробнее об этом правиле - в теме про хуки.

## Типичные ошибки

**Присваивание вместо сеттера.** `count = count + 1` не перерисует экран. Меняет состояние только `setCount`.

**Мутация объекта или массива.** `items.push(x)` или `obj.done = true` меняют данные на месте, ссылка остаётся прежней, и React не видит изменения. Передавайте сеттеру новый массив или объект.

**Ожидать новое значение сразу.** После вызова сеттера в текущем рендере состояние ещё старое. Свежее значение придёт со следующим рендером.

**Дублировать в состоянии то, что можно вычислить.** Полное имя из имени и фамилии не нужно держать отдельным состоянием - посчитайте его при рендере. Лишнее состояние легко рассинхронизировать.

<Detail summary="Откуда React знает, где чьё состояние">

Внутри `useState` нет ни имени переменной, ни ключа - откуда же React понимает, что этот вызов относится к `count`, а тот к `name`? Ответ прост и немного пугает: по порядку. React хранит состояния компонента списком и на каждом рендере раздаёт их вызовам `useState` строго по очереди - первый вызов получает первое, второй второе. Именно поэтому порядок вызовов обязан быть одинаковым от рендера к рендеру, а хуки нельзя вызывать условно. Это цена простоты синтаксиса: никаких ключей писать не нужно, но и порядок ломать нельзя.

</Detail>
