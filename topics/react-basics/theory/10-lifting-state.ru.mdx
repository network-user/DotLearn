# Подъём состояния и однонаправленный поток

Представьте два компонента-соседа: поле поиска и список результатов. Пользователь печатает в поле - список должен мгновенно отфильтроваться. Проблема в том, что поле и список - разные компоненты, и состояние поиска не может лежать в обоих сразу: заведёте по своей копии - и они разъедутся, покажут разное. Нужен один владелец истины. Приём, которым его находят, называется подъёмом состояния (lifting state up), и он же вводит главный организующий принцип React - однонаправленный поток данных.

В этом уроке вы научитесь решать, где должно жить общее состояние, и увидите, как данные едут вниз пропами, а события поднимаются наверх коллбэками, образуя аккуратный круг.

## Проблема: у кого хранить общее состояние

Пропсы едут только вниз, а менять их ребёнок не вправе. Значит, если двум компонентам нужно одно и то же изменяемое значение, положить его в один из них нельзя - второй не дотянется. Остаётся единственный вариант: поднять состояние в их ближайшего общего родителя, того, кто содержит обоих. Он становится единственным источником истины, а дети получают значение сверху.

Пусть родитель хранит текст запроса. Полю поиска он отдаёт этот текст и функцию, которой текст меняют. Списку - тот же текст, чтобы фильтровать. Оба ребёнка смотрят на одно значение, и рассинхрону взяться неоткуда.

```jsx
function SearchPage() {
  const [query, setQuery] = useState('');
  return (
    <div>
      <SearchInput value={query} onChange={setQuery} />
      <ResultsList query={query} />
    </div>
  );
}
```

## Данные вниз, события вверх

Теперь ключевой узор. Родитель передаёт вниз две вещи: само значение (`value={query}`) и функцию для его изменения (`onChange={setQuery}`). Ребёнок значение читает и показывает, а когда пользователь что-то ввёл, не меняет проп сам - он зовёт переданную функцию и тем сообщает родителю: «поменяй, пожалуйста, на вот это».

```jsx
function SearchInput({ value, onChange }) {
  return (
    <input value={value} onChange={(e) => onChange(e.target.value)} />
  );
}
```

`SearchInput` не владеет запросом - он лишь окно к нему. Ввод превращается в вызов `onChange`, тот идёт наверх, родитель обновляет своё состояние сеттером, происходит рендер, и свежий `query` спускается обратно в оба ребёнка. Данные текут вниз, события - вверх.

## Почему поток именно однонаправленный

Однонаправленность - не ограничение ради ограничения, а то, что делает большое приложение поддающимся пониманию. Раз данные всегда текут в одну сторону и меняет их только владелец, у любого значения на экране есть единственный адрес, где оно живёт. Видите неверный текст в списке - идёте вверх, к его владельцу, и там ищете причину. Не нужно гадать, кто из десятка детей втихаря переписал общий стейт: переписать его некому, дети умеют лишь вежливо просить через коллбэк.

<PullQuote>Данные текут вниз, события поднимаются вверх, а менять их вправе только владелец - в этом весь React.</PullQuote>

## Где именно поднимать: не выше, чем нужно

Поднимать состояние стоит ровно в ближайшего общего родителя обоих компонентов - не выше. Если засунуть каждое состояние в корневой `App`, он распухнет и станет перегонять пропсы через кучу промежуточных слоёв, которым эти данные не нужны. Правило: найдите двух (или больше) потребителей значения и поместите состояние в их наименьшего общего предка. Тогда пропсам не придётся ехать через лишние этажи.

## Типичные ошибки

**Дублировать состояние в обоих детях.** Две копии неизбежно разъедутся. Общее значение - один источник истины в общем родителе, а детям спускают его пропом.

**Поднимать слишком высоко.** Состояние в корне, нужное лишь двум соседним листьям, порождает длинные цепочки прокидывания пропсов. Держите его у ближайшего общего предка.

**Пытаться менять проп у ребёнка.** Ребёнок не владеет значением и переписать его не может. Он вызывает коллбэк, а меняет состояние владелец.

**Забыть передать коллбэк вниз.** Если родитель отдал значение, но не дал функцию для его смены, ребёнок сможет только показывать данные, но не запрашивать их изменение.

<Detail summary="А если пропсы едут через много этажей">

Иногда общий предок оказывается высоко, и значение приходится прокидывать пропами через несколько промежуточных компонентов, которым оно не нужно, - это называют prop drilling. Пока этажей два-три, это нормально и честно. Когда их становится много, на помощь приходят инструменты вроде контекста (`useContext`) или внешних хранилищ состояния, которые дают глубоко вложенному ребёнку доступ к значению напрямую, минуя промежуточные слои. Это тема за пределами основ; важно, что подъём состояния остаётся отправной точкой, а контекст решает уже его крайний случай.

</Detail>
