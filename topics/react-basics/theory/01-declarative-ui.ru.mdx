# Декларативный UI: вы описываете результат, а не шаги

Представьте кнопку «Нравится» под постом. Без всякой библиотеки вы напишете так: найдёте элемент через `document.querySelector`, прочитаете текущее число, прибавите единицу, запишете обратно в `textContent`, потом переключите класс, чтобы сердечко покраснело. Пока состояние одно, это терпимо. Но добавьте загрузку, ошибку сети, состояние «уже поставлено» и «свой пост лайкать нельзя» - и код превращается в клубок команд, где на каждый чих надо вспомнить, какую именно строчку DOM подправить и не забыть ни одного угла экрана.

React переворачивает задачу. Вместо того чтобы отдавать браузеру команды по шагам, вы пишете функцию, которая по текущему состоянию возвращает описание того, как экран должен выглядеть. Вы перестаёте дёргать DOM руками и начинаете описывать цель. В этом уроке разберём, что такое декларативный подход, почему формула «интерфейс - это функция состояния» экономит вам нервы и где у неё своя цена.

## Императивно против декларативно

Императивный код отвечает на вопрос «как»: перечисляет шаги, которые меняют экран. Декларативный отвечает на вопрос «что»: описывает результат для данного состояния, а как его достичь - забота движка. Разница та же, что между «поверни руль на десять градусов, потом прибавь газу» и «доставь меня на Тверскую, 7».

```js
const button = document.querySelector('#like');
let count = 0;
button.addEventListener('click', () => {
  count = count + 1;
  button.textContent = 'Нравится · ' + count;
  button.classList.add('active');
});
```

Каждую видимую деталь здесь приходится воспроизводить вручную, шаг за шагом. Теперь то же самое на React:

```jsx
function LikeButton({ count, liked }) {
  return (
    <button className={liked ? 'active' : ''}>
      Нравится · {count}
    </button>
  );
}
```

Вы не трогаете DOM. Вы описываете: при таком `count` и таком `liked` кнопка выглядит вот так. Меняется состояние - React сам пересчитывает разметку и приводит экран к новому описанию.

## Формула: интерфейс - это функция состояния

Главная мысль React умещается в короткую запись: `UI = f(состояние)`. Компонент - это функция `f`, которая берёт данные и возвращает разметку. Один и тот же вход всегда даёт один и тот же экран, поэтому, глядя на состояние, вы точно знаете, что увидит пользователь, и наоборот.

Отсюда следует правило, к которому мы будем возвращаться весь курс: **чтобы изменить то, что на экране, вы меняете состояние, а не DOM**. Нет строчек вида «найди элемент и поправь текст». Есть новое значение состояния и функция, которая по нему заново рисует картинку.

## Что происходит при изменении

Когда состояние меняется, React заново вызывает вашу функцию и получает свежее описание экрана. Наивно было бы стереть старую разметку и вставить новую целиком - это медленно и сбрасывало бы фокус в полях ввода. Поэтому React сравнивает новое описание со старым и меняет в реальном DOM только то, что реально отличается. Этот процесс называется согласованием (reconciliation), и ему посвящён отдельный урок про рендер. Пока достаточно интуиции: вы отдаёте полную картину, React аккуратно доводит экран до неё минимальными правками.

<Callout type="tip" title="Почему это удобно">
Раз экран всегда выводится из состояния, у вас не бывает рассинхрона «данные обновились, а кусок интерфейса показывает старое». Забыть обновить угол невозможно: он пересчитывается вместе со всем остальным. Целый класс багов просто исчезает.
</Callout>

## Где у модели цена

Декларативность не бесплатна. Первое: вы описываете весь экран заново на каждое изменение, и это поначалу кажется расточительным. На практике React гасит расходы согласованием, но думать о производительности рендера всё же иногда придётся - об этом в уроке про рендер.

Второе: не всё живёт внутри модели «состояние - разметка». Фокус курсора, прокрутка, измерение размеров элемента, интеграция со сторонней библиотекой на чистом DOM - это побочные эффекты, для которых есть отдельный инструмент (`useEffect`) в соседней теме про хуки. Здесь важно запомнить, что такие выходы за пределы модели существуют и что их держат в стороне от основного потока «состояние рисует разметку».

## Типичные ошибки

**Дёргать DOM руками внутри React.** Соблазн написать `document.querySelector` и подправить элемент напрямую велик, но так вы воюете с моделью: на следующем рендере React перерисует всё из состояния и затрёт вашу правку. Нужно поменять экран - меняйте состояние.

**Хранить в состоянии то, что можно вычислить.** Если полное имя всегда складывается из имени и фамилии, не держите его третьим полем состояния: посчитайте прямо в разметке. Лишнее состояние - это лишний источник рассинхрона.

**Считать, что «перерисовать» значит «собрать страницу заново».** Повторный вызов вашей функции - это не пересборка всего DOM. React сравнивает и правит точечно. Об этом легко забыть и начать «оптимизировать» то, что и так дёшево.

<Detail summary="Немного истории: откуда взялась идея">

React появился в Facebook в 2013 году, и главной его ставкой была именно декларативность плюс идея виртуального DOM - лёгкого описания разметки в памяти, которое дёшево пересоздавать и сравнивать. До этого фронтенд жил в мире ручных правок DOM и двусторонних связываний, где источник истины размывался между данными и экраном. Формула «интерфейс выводится из состояния в одну сторону» оказалась достаточно простой, чтобы на ней выросла целая экосистема.
