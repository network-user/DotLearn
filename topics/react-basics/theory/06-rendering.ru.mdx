# Рендер и реактивность: как React обновляет экран

Вы уже знаете, что смена состояния перерисовывает компонент. Но слово «перерисовывает» скрывает целый механизм, и без него легко испугаться: неужели React на каждый клик собирает страницу заново? Нет. В этом уроке мы вскроем, что именно происходит между вызовом сеттера и обновлённым пикселем на экране, и разберём поведение, которое чаще всего сбивает с толку новичков, - объединение обновлений в пакеты.

Понимание этого механизма превращает React из магии в предсказуемый инструмент: вы перестаёте гадать, почему счётчик прибавил единицу вместо трёх, и начинаете видеть причину.

## Рендер - это вызов вашей функции

Отрендерить компонент значит вызвать его функцию и получить JSX - описание того, как экран должен выглядеть сейчас. Первый рендер случается при появлении компонента на странице. Каждый следующий React запускает сам, когда меняется состояние или приходят новые пропсы. Рендер - это не рисование на экране, а вычисление описания; рисованием займётся отдельный шаг.

Собрав новое описание, React не стирает старую разметку и не строит DOM с нуля - это было бы медленно и сбрасывало бы фокус и прокрутку. Вместо этого он сравнивает новое дерево элементов с предыдущим, находит различия и вносит в реальный DOM только их. Этот шаг называется согласованием (reconciliation).

Отсюда важная мысль: повторный вызов вашей функции - дёшев. Он создаёт лёгкие объекты в памяти, а не трогает DOM. Дорогие операции с реальными узлами React выполняет только там, где нашёл разницу. Поэтому «перерисовка» в React почти всегда означает куда меньше работы, чем звучит.

## Реактивность: экран следует за состоянием

Реактивность - это и есть автоматическая связь «состояние изменилось - экран обновился». Вам не нужно вручную звать перерисовку или искать, какие узлы поправить: вы меняете состояние сеттером, а React сам проводит всю цепочку до DOM. Один источник истины - состояние; экран всегда его отражение.

Из этого следует правило чистоты: во время рендера компонент должен только вычислять разметку по текущим данным и ничего не менять снаружи. Никаких запросов к серверу, таймеров или правок чужих переменных прямо в теле функции - для таких побочных эффектов есть `useEffect` из темы про хуки. Рендер, вызванный дважды, обязан давать один и тот же результат.

## Обновления объединяются в пакеты

Теперь самое коварное. React не перерисовывает компонент на каждый вызов сеттера по отдельности. Внутри одного обработчика события он собирает все вызовы сеттеров и выполняет один общий рендер в конце - это называется пакетным обновлением (batching). Приём экономит работу, но порождает знаменитую ловушку:

```jsx
function handleClick() {
  setCount(count + 1);
  setCount(count + 1);
  setCount(count + 1);
}
```

Кажется, что счётчик прибавит три. На деле - один. Пока идёт текущий рендер, переменная `count` заморожена на своём значении, скажем 0. Все три вызова читают один и тот же ноль и трижды просят React установить `0 + 1`, то есть единицу. React объединяет их и рисует один раз со значением 1.

Чтобы прибавить именно три, сеттеру передают не новое значение, а функцию-апдейтер. Она получает последнее актуальное значение и возвращает следующее:

```jsx
function handleClick() {
  setCount((c) => c + 1);
  setCount((c) => c + 1);
  setCount((c) => c + 1);
}
```

Теперь каждая функция получает результат предыдущей: 0 даёт 1, 1 даёт 2, 2 даёт 3. Правило простое: если новое состояние зависит от старого, используйте форму с функцией.

<Callout type="tip" title="Когда какая форма">
Передавайте значение (`setCount(0)`), когда новое состояние не зависит от старого. Передавайте функцию (`setCount(c => c + 1)`), когда оно вычисляется из предыдущего, особенно если сеттер вызывается несколько раз подряд или внутри асинхронного кода.
</Callout>

## Краевые случаи

React старается не рисовать лишнего, но и не гарантирует, что рендер случится строго один. Считать количество рендеров или полагаться на их точный момент - плохая идея; полагайтесь на то, что экран в итоге отразит актуальное состояние. И помните: рендерится не только тот компонент, чьё состояние изменилось, но и его дети - это нормально и обычно дёшево благодаря согласованию.

## Типичные ошибки

**Ждать, что каждый сеттер вызовет свой рендер.** Внутри обработчика вызовы объединяются в один рендер. Не стройте логику на числе перерисовок.

**Читать состояние сразу после сеттера.** В текущем рендере значение ещё старое. Если следующее состояние зависит от предыдущего, берите форму с функцией-апдейтером.

**Побочные эффекты в теле рендера.** Запрос к серверу или таймер прямо в функции компонента ломает чистоту и повторяемость рендера. Их место - в `useEffect` (тема про хуки).

**Считать перерисовку дорогой по умолчанию.** Повторный вызов функции создаёт лёгкие объекты; дорогие правки DOM делаются только для различий. Оптимизировать стоит измеренную проблему, а не воображаемую.

<Detail summary="Виртуальный DOM в двух словах">

Лёгкое дерево объектов, которое возвращает ваш компонент, часто называют виртуальным DOM. Идея в том, что пересоздавать и сравнивать такие объекты в памяти на порядки дешевле, чем трогать реальные узлы браузера. React держит две версии этого дерева - до и после рендера, - за один проход находит различия и переносит в настоящий DOM минимальный набор правок. Это не магия и не всегда абсолютный оптимум, но для подавляющего большинства интерфейсов быстрее и надёжнее ручных манипуляций.

</Detail>
