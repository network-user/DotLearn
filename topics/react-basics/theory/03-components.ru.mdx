# Функциональные компоненты

Интерфейс большого приложения не пишут одной простынёй разметки. Его собирают из кирпичей: шапка, карточка товара, кнопка, форма. В React такой кирпич называется компонентом, и устроен он до смешного просто - это обычная функция, которая возвращает JSX. Никакого особого синтаксиса, никакого класса: если функция вернула разметку, это уже компонент, и React умеет его показывать.

В этом уроке вы научитесь объявлять компоненты, собирать из них экран как из деталей конструктора и не попадаться на единственное строгое правило React - про заглавную букву в имени.

## Компонент - это функция, вернувшая разметку

Вот компонент целиком. Функция без аргументов возвращает JSX - и всё, кирпич готов:

```jsx
function Welcome() {
  return <h1>Добро пожаловать в Ozon</h1>;
}
```

Пользуются им не как обычной функцией. Вы не пишете `Welcome()` руками - вместо этого вставляете имя как тег в другую разметку, и React сам вызовет функцию, когда будет рисовать экран:

```jsx
function App() {
  return (
    <main>
      <Welcome />
      <p>Сегодня хорошая погода для покупок.</p>
    </main>
  );
}
```

React встретит `<Welcome />`, вызовет функцию `Welcome`, возьмёт её JSX и подставит на это место. Так тег вашего собственного компонента живёт в разметке наравне со встроенными `<main>` и `<p>`.

## Композиция: из кирпичей собирают стену

Тот же компонент можно поставить в разметку сколько угодно раз - и каждая копия отрисуется независимо. Именно так один компонент `ProductCard` превращается в сотню карточек каталога. Чем данные разных карточек отличаются, решают пропсы, и это тема следующего урока; пока важно увидеть саму идею повторного использования.

## Единственное строгое правило: имя с заглавной буквы

React различает ваш компонент и обычный HTML-тег ровно по одному признаку - по первой букве имени. Имя с заглавной буквы React считает компонентом и вызывает вашу функцию. Имя со строчной буквы React считает встроенным тегом браузера и просто вставляет как есть.

```jsx
function Message() {
  return <p>Заказ оформлен</p>;
}
```

Запись `<Message />` покажет ваш абзац. А `<message />` React примет за неизвестный HTML-тег `message`, ничего не нарисует и в режиме разработки предупредит в консоли. Отсюда правило без исключений: компоненты называют с заглавной буквы (`ProductCard`, `Header`), и объявляют их на верхнем уровне модуля, а не внутри другой функции.

<Callout type="warning" title="Компонент внутри компонента - ловушка">
Заманчиво объявить один компонент прямо внутри другого, рядом с местом использования. Так делать не стоит: при каждом рендере внешнего компонента вложенный будет создаваться заново как новая функция, и React посчитает его новым типом узла - со сбросом состояния и лишней перерисовкой. Компоненты объявляют рядом, а не вкладывают в тело друг друга.
</Callout>

## Краевые случаи

Компонент подчиняется тем же правилам, что и любой JSX. Возврат обязан иметь один корень: если разметка состоит из нескольких соседних тегов, оберните их во фрагмент `<>...</>`. А если компонент в каком-то состоянии не должен рисовать ничего, он возвращает `null` - это разрешено и означает «пустое место»:

```jsx
function Warning({ show }) {
  if (!show) {
    return null;
  }
  return <p>Внимание: мало на складе</p>;
}
```

Здесь `return null` до основной разметки - обычный приём: он отсекает случай, когда рисовать нечего, и делает остальной код проще. Логику до `return` писать можно и нужно; нельзя лишь ставить инструкции внутрь самих фигурных скобок JSX.

## Типичные ошибки

**Имя со строчной буквы.** `<userCard />` React примет за HTML-тег и не нарисует ваш компонент. Первое, что стоит проверить, если «компонент не появился», - заглавную букву в имени.

**Вызывать компонент как функцию.** Соблазн написать `{Welcome()}` вместо `<Welcome />` велик, и часто это даже «работает». Но для React это уже не отдельный узел дерева, а просто вставленный результат, и такое различие позже ломает состояние и оптимизации. Пользуйтесь тегом: `<Welcome />`.

**Несколько корней в возврате.** Два тега подряд без обёртки - ошибка сборки, как и в любом JSX. Оборачивайте во фрагмент.

<Detail summary="Почему функция, а не класс">

Исторически компоненты писали классами, наследуя `React.Component`, а состояние и жизненный цикл держали в методах. С 2019 года, после появления хуков, функциональные компоненты умеют всё то же самое, но читаются проще: одна функция вместо класса с методами и `this`. Сегодня новый код почти всегда пишут функциями, а классовые компоненты встречаются в старых проектах. В этом курсе мы говорим только о функциональных.

</Detail>
