# Turn meaning into a vector you can measure

A network works only with numbers, but the world is full of non-numbers: words, products, users, images. An **embedding** represents such an object as a vector of numbers so that vector closeness reflects semantic closeness. "Cat" and "dog" end up nearby; "cat" and "bank" far apart. That turns vague "similar in meaning" into precise geometry you can compute. Search, recommendations, and all neural text processing rest on embeddings. In this section you learn what an embedding is, measure closeness of two vectors with the cosine of the angle between them, and implement that measure in Python.

## Meaning as a point in space

An embedding places objects in a multidimensional space where geometric neighbors mean semantic neighbors. Each word is a point; closer points mean closer meanings. Coordinates are not set by hand: the network learns them during training so words appearing in similar contexts land near each other. A striking property of good embeddings is that relations are encoded: the classic "king - man + woman ≈ queen" shows that directions in space carry meaning (for example, the direction "gender").

## Cosine similarity: angle, not distance

How do you measure closeness of two vectors? Ordinary distance works, but for embeddings **direction** matters more than length. So use **cosine similarity**, the cosine of the angle between vectors:

```
cos(a, b) = (a · b) / (|a| * |b|)
```

The numerator is the dot product; the denominator is the product of lengths. The result lies in -1 to 1: `1`: vectors point the same way (maximally similar), `0`: perpendicular (unrelated), `-1`: opposite. Dividing by lengths removes scale: a long and short vector in the same direction count as equally similar.

```python
from math import sqrt

def dot(a, b):
    return sum(x * y for x, y in zip(a, b))

def norm(v):
    return sqrt(sum(x * x for x in v))

def cosine(a, b):
    return dot(a, b) / (norm(a) * norm(b))

print(round(cosine([1, 0], [1, 0]), 4))
print(round(cosine([1, 0], [0, 1]), 4))
print(round(cosine([1, 0], [-1, 0]), 4))
```

This prints `1.0`, `0.0`, `-1.0`: same direction, perpendicular, opposite. Three anchor points on the scale.

**Semantic search** rests on this measure. A query becomes an embedding; cosine similarity is computed against every document embedding; the closest by angle are returned, finding by meaning, not exact word match: query "how to start a car in winter" can surface a document about "cold engine start" with almost no shared words. The same idea underlies recommendations (similar products) and RAG systems that fetch relevant context for a language model. To avoid comparing a query against millions of vectors by brute force, approximate nearest-neighbor indexes (ANN) find nearly exact answers orders of magnitude faster.

That shows cosine captures angle, not length.

## When behavior surprises you

Cosine is undefined for a zero vector: length is zero and division breaks. Before computing, check both vectors are nonzero (or add a tiny epsilon to the denominator for stability). A zero embedding is usually a bad sign: a missing or unprocessed object.

A second surprise: collinear vectors of different length. `[1, 1]` and `[100, 100]` point the same way and cosine is exactly `1.0`, though Euclidean distance is large. That is not a bug but the point of the measure: cosine intentionally ignores length and looks only at direction. If magnitude matters for the task, use another measure (for example, Euclidean distance).

## Common mistakes

- **Skipping normalization (division by lengths).** Raw dot product `a · b` grows with vector length, so alone it is not a similarity measure: a long vector "wins" over a short one from scale alone. Cosine divides by lengths to remove that.
- **Confusing the range.** Cosine lives in [-1, 1], not [0, 1]. A negative value is not an error: it means opposite direction. If you expect only non-negative values, check that your vectors allow it.
- **Comparing embeddings from different models.** Coordinates are meaningful only inside one space. A word vector from one model and one from another are incomparable: similarity between them is a meaningless number.

<Detail summary="Where the network gets embeddings">

Embeddings are not hand-designed: they are learned. In the simple case this is a lookup table: each vocabulary item has a row-vector, and those rows are trainable parameters moved by gradient descent along with other weights. Training indirectly demands similar objects get similar vectors: in word2vec the network predicts a word from neighbors, and words from similar contexts naturally cluster. In large language models a word embedding is only the input layer; context refines the representation at each attention layer, so the same word in different sentences gets different final vectors (contextual embeddings).
