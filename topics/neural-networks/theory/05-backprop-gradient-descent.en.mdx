# How the network learns which way to nudge each weight

You have a network that computes an answer (forward pass) and a number that says how wrong it was (loss). What remains is the main question: how do you adjust millions of weights so loss drops? Brute force is pointless: infinitely many options. Two ideas work in tandem. **Gradient descent** says which direction to move weights: where loss decreases. **Backpropagation** efficiently computes that direction for all weights in one backward pass through the network. In this section you understand both and perform one weight update by hand.

## Descending the loss slope

Think of loss as a function of the weights: change a weight and loss changes. You get a landscape, a "loss terrain", and the goal is to reach its valley. At each point the slope has a direction: the **gradient**. It points toward the steepest **increase** in loss; to decrease loss you step in the **opposite** direction. That is gradient descent: on each step shift weights against the gradient.

Step size is set by **learning rate**: the multiplier in front of the gradient. Too small: descent crawls forever. Too large: steps overshoot the valley and the ball bounces without settling. Tuning learning rate is one of the main practical decisions in training.

The update formula for one weight:

```
w = w - learning_rate * gradient
```

The minus is essential: gradient points uphill, you want downhill. On a two-dimensional terrain the descent path shows as a line running toward the center of the valley.

## One update step by hand

Take one weight `w = 0.5`, learning rate `0.1`, and loss gradient with respect to this weight `2.0` (that is what backpropagation computes). Take a step:

```python
w = 0.5
learning_rate = 0.1
gradient = 2.0

w = w - learning_rate * gradient
print(round(w, 4))
```

This prints `0.3`: a positive gradient means "increasing w raises loss", so the step pulls the weight down to `0.3`. With a negative gradient the weight would grow instead. The update always moves the weight against the sign of the gradient.

## Chain rule: the heart of backpropagation

Where does the gradient of loss with respect to a weight buried deep in the network come from? Loss depends on weights not directly but through a chain: weight → neuron sum → activation → next layer → ... → loss. The **chain rule** from calculus says how to multiply local derivatives along that chain to get the total. Backpropagation applies it efficiently: it walks from loss back toward the input, at each layer multiplying the incoming gradient from above by the layer's local derivative and passing the result downward. One backward pass yields gradients for all weights.

The key idea is that backprop does not invent new math: it carefully reuses intermediate values from the forward pass and multiplies derivatives along the chain. Without it you would compute the derivative for each weight separately, which is infeasible for millions of weights.

## Estimating the gradient numerically

The chain rule gives the exact derivative, but you can approximate it numerically to build intuition: gradient is "how much loss changes if you nudge the weight slightly". Take a small shift `h`, see how the function changes, and divide:

```python
def numeric_gradient(f, x, h=1e-5):
    return (f(x + h) - f(x - h)) / (2 * h)

print(round(numeric_gradient(lambda x: x ** 2, 3.0), 4))
```

For `f(x) = x^2` at 3 the derivative is `2*3 = 6`, and the numeric estimate gives `6.0`. This trick (central difference) is not how gradients are computed in practice (backprop is faster and exact), but it shows that gradient is sensitivity of output to input.

## Try it yourself

Run the code: it performs one gradient descent step and prints the new weight. Increase learning rate to `1.0` and watch one step overshoot the minimum; then make it tiny (`0.001`) and confirm the weight barely moves. That tradeoff, fast but risky versus safe but slow, becomes visible.

## When behavior surprises you

A large learning rate does not just slow training, it can **diverge**: loss does not fall but rises step by step because each step overshoots the minimum further. If loss goes to infinity or becomes `NaN` during training, the first suspect is learning rate that is too large.

A second surprise: gradient descent finds a **local** minimum nearest the start, not necessarily the global one. A complex loss landscape has many valleys. In practice this is rarely a problem: good local minima in large networks give solid quality, and randomness in data and initialization helps avoid bad ones.

## Common mistakes

- **Wrong sign in the update.** `w = w + lr * gradient` pushes uphill and loss grows. The step is always against the gradient: `w = w - lr * gradient`.
- **One learning rate for every task.** A step that works for one problem destroys another. Learning rate is tuned, and often decreased over training (a schedule).
- **Treating backprop as a separate training algorithm.** Backprop only computes gradients; gradient descent (or variants like Adam) takes the step. Two different roles working together.

<Detail summary="Why practice uses batches (SGD) instead of the full dataset">

Honest gradient descent would compute the gradient over the entire dataset every step, which is expensive and slow. **Stochastic** gradient descent (SGD) estimates the gradient from a small random batch of examples. The estimate is noisy but correct on average, and steps per second are orders of magnitude higher. The noise even helps: it can escape bad narrow minima. Modern optimizers (Adam, AdamW) add adaptive per-weight step sizes and momentum on top of SGD, making training more stable and faster.

</Detail>
