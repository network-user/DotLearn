# Превратить смысл в вектор, чтобы его можно было измерить

Сеть умеет работать только с числами, а мир полон не-чисел: слов, товаров, пользователей, картинок. **Эмбеддинг** (embedding) - это представление такого объекта вектором чисел так, чтобы близость векторов отражала близость смысла. «Кошка» и «собака» оказываются рядом, а «кошка» и «банк» - далеко. Это превращает расплывчатое «похоже по смыслу» в точную геометрию, которую можно посчитать. На эмбеддингах держатся поиск, рекомендации и вся обработка текста в нейросетях. В этом разделе Вы поймёте, что такое эмбеддинг, как измерить близость двух векторов косинусом угла между ними, и напишете эту меру на Python.

## Смысл как точка в пространстве

Эмбеддинг укладывает объекты в многомерное пространство так, что геометрическое соседство означает смысловое. Каждое слово - точка; чем ближе точки, тем ближе значения. Эти координаты не задают вручную - сеть выучивает их во время обучения так, чтобы слова, встречающиеся в похожих контекстах, оказались рядом. На рисунке слова сами собой собираются в смысловые кластеры - наведите курсор на любое и посмотрите, кто его соседи.

Размерность реальных эмбеддингов - сотни или тысячи чисел (нарисовать такое нельзя, поэтому на картинке всего два измерения). Удивительное свойство хороших эмбеддингов - в них кодируются и отношения: классический пример «король - мужчина + женщина ≈ королева» показывает, что направления в пространстве несут смысл (например, направление «пол»).

## Косинусная близость - угол, а не расстояние

Как измерить близость двух векторов? Можно взять обычное расстояние, но у эмбеддингов важнее **направление**, а не длина. Поэтому берут **косинусную близость** - косинус угла между векторами:

```
cos(a, b) = (a · b) / (|a| * |b|)
```

В числителе - скалярное произведение векторов, в знаменателе - произведение их длин. Результат лежит в диапазоне от -1 до 1: `1` - векторы смотрят в одну сторону (максимально похожи), `0` - перпендикулярны (не связаны), `-1` - противоположны. Деление на длины убирает влияние масштаба: важен только угол, поэтому длинный и короткий вектор одного направления считаются одинаково похожими.

```python
from math import sqrt

def dot(a, b):
    return sum(x * y for x, y in zip(a, b))

def norm(v):
    return sqrt(sum(x * x for x in v))

def cosine(a, b):
    return dot(a, b) / (norm(a) * norm(b))

print(round(cosine([1, 0], [1, 0]), 4))
print(round(cosine([1, 0], [0, 1]), 4))
print(round(cosine([1, 0], [-1, 0]), 4))
```

Печатается `1.0`, `0.0`, `-1.0`: одинаковое направление, перпендикуляр, противоположность. Это и есть три опорные точки шкалы.

На этой мере держится **семантический поиск**. Запрос превращают в эмбеддинг, считают косинусную близость с эмбеддингами всех документов и возвращают ближайшие по углу - так находят по смыслу, а не по точному совпадению слов: запрос «как завести машину зимой» подтянет документ про «холодный запуск двигателя», хотя общих слов почти нет. Та же идея лежит в основе рекомендаций (близкие товары) и в RAG-системах, которые подбирают релевантный контекст для языковой модели. Чтобы не сравнивать запрос с миллионами векторов перебором, используют специальные индексы приближённого поиска ближайших соседей (ANN) - они находят почти точный ответ на порядки быстрее.

## Когда поведение удивит

Косинус не определён для нулевого вектора: его длина равна нулю, и деление ломается. Поэтому перед расчётом проверяют, что оба вектора ненулевые (или добавляют к знаменателю крошечное число для устойчивости). Нулевой эмбеддинг - вообще плохой знак, обычно это пропущенный или необработанный объект.

Второй сюрприз - сонаправленные векторы разной длины. `[1, 1]` и `[100, 100]` смотрят в одну сторону, и косинус между ними равен ровно `1.0`, хотя по обычному расстоянию они далеко. Это не баг, а суть меры: косинус намеренно игнорирует длину и смотрит только на направление. Если для задачи важна и величина, берут другую меру (например, евклидово расстояние).

## Типичные ошибки

- **Забыть нормировку (деление на длины).** Голое скалярное произведение `a · b` растёт с длиной векторов, поэтому само по себе не мера близости: длинный вектор «победит» короткий просто из-за масштаба. Косинус делит на длины именно чтобы это убрать.
- **Перепутать диапазоны.** Косинус живёт в [-1, 1], а не в [0, 1]. Отрицательное значение - не ошибка, а противоположное направление. Если ждёте только неотрицательных значений, проверьте, что векторы это допускают.
- **Сравнивать эмбеддинги из разных моделей.** Координаты осмысленны только внутри одного пространства. Вектор слова из одной модели и вектор из другой несопоставимы: близость между ними - бессмысленное число.

<Detail summary="Откуда сеть берёт эмбеддинги">

Эмбеддинги не задают вручную - их учат. В простом случае это таблица: каждому объекту словаря соответствует строка-вектор, и эти строки - обучаемые параметры, которые градиентный спуск двигает вместе с остальными весами. Цель обучения косвенно требует, чтобы похожие объекты получали похожие векторы: например, в word2vec сеть учится предсказывать слово по соседям, и слова из похожих контекстов естественно сближаются. В больших языковых моделях эмбеддинг слова - лишь входной слой; дальше контекст уточняет представление на каждом слое внимания, поэтому одно и то же слово в разных предложениях получает разные итоговые векторы (контекстные эмбеддинги).
