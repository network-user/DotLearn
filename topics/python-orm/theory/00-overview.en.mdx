You wrote an SQL query and got back a Python tuple like `(1, 'Anna', 30)`. Then the routine starts: unpack it field by field, build an `INSERT` string, remember escaping, close the connection. Fine for a couple of queries, painful in a real app with hundreds of lines of manual translation between Python objects and table rows. An ORM takes over that translation: you describe which class maps to which table and work with ordinary objects while the library generates SQL. But the magic is expensive for anyone who does not understand what happens underneath — so here you work with the database by hand first, and only then let the ORM handle the translation.

**What you'll be able to do.** Run queries from Python directly and substitute data safely through parameters; describe tables as classes in SQLAlchemy 2.x style; build queries with `select()` and walk relationships like object attributes; see exactly which SQL hits the database and catch expensive mistakes like N+1.

**The route through this topic.** Four sections, each with a sandbox where code runs right in the browser:

1. **Raw SQL** — open a connection and cursor, run queries by hand, and pass data through parameters, not f-strings.
2. **ORM mental model** — map what corresponds to what and declare your first SQLAlchemy 2.x model.
3. **Queries and relationships** — build selections with `select()` and turn foreign keys into `.user` and `.orders` attributes.
4. **The N+1 problem** — recognize how an innocent loop spawns a thousand queries and fix it with eager loading.

**What you need first.** Nothing to install: Python runs right in the browser. Solid basics from the SQL Fundamentals topic help — `SELECT`, `JOIN`, filtering; everything else, including SQLAlchemy itself, we cover along the way.
