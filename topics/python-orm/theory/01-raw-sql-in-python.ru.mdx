# Прежде чем звать ORM, научитесь говорить с базой напрямую

Вы уже умеете писать SQL. Теперь нужно выполнить этот SQL из Python: открыть соединение, отправить запрос, забрать строки обратно в программу. Стандартная библиотека Python даёт для этого `sqlite3` - и важно пройти этот слой руками, прежде чем прятать его за ORM. Тот, кто понимает, что ORM в конечном счёте дёргает курсор и шлёт строки текста, отлаживает проблемы за минуты, а не за часы.

К концу раздела Вы сможете открыть соединение и курсор, выполнить запрос и забрать результат, безопасно подставить пользовательские данные через параметры (и объяснить, почему f-строка в запросе - это дыра в безопасности), а также управлять транзакцией через `commit` и `rollback`, не забывая закрывать ресурсы контекстным менеджером.

## Соединение, курсор, результат - конвейер из трёх звеньев

Любая работа с базой в Python идёт по одному и тому же пути. Сначала Вы открываете *соединение* (`Connection`) - канал к файлу или серверу базы. Затем получаете от него *курсор* (`Cursor`) - объект, через который отправляются запросы и читаются ответы. Курсор выполняет SQL методом `execute`, а строки результата Вы забираете методами `fetchone`, `fetchall` или просто перебирая курсор в цикле.

Минимальный сеанс целиком умещается в несколько строк. Создаём таблицу, кладём пару строк, читаем их обратно:

```python
import sqlite3

conn = sqlite3.connect(":memory:")
cur = conn.cursor()
cur.execute("CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT, age INTEGER)")
cur.execute("INSERT INTO users (name, age) VALUES ('Анна', 30)")
cur.execute("INSERT INTO users (name, age) VALUES ('Борис', 17)")
rows = cur.execute("SELECT name, age FROM users").fetchall()
print(rows)
```

Печатается `[('Анна', 30), ('Борис', 17)]`. Каждая строка результата - это кортеж, а `fetchall` собирает их в список. Строка `:memory:` означает базу, живущую только в оперативной памяти: идеально для примеров и тестов, потому что после завершения программы от неё не остаётся файла.

`fetchone` возвращает следующую одну строку (или `None`, когда строки кончились), `fetchall` - сразу все оставшиеся. Перебор курсора в цикле читает строки по одной, не загружая весь результат в память - так обрабатывают большие выборки.

```python
for row in cur.execute("SELECT name FROM users"):
    print(row[0])
```

## Никогда не клейте запрос f-строкой

Теперь самое важное в этом разделе. Допустим, имя пользователя приходит снаружи - из формы, из аргумента командной строки - и Вы хотите найти его в базе. Соблазнительно собрать запрос строкой:

```python
name = "Анна"
cur.execute(f"SELECT * FROM users WHERE name = '{name}'")
```

Это работает ровно до того момента, пока `name` не станет враждебным. Подставьте сюда `name = "'; DROP TABLE users; --"`, и f-строка соберёт два запроса: безобидный `SELECT` и команду на удаление таблицы. База послушно выполнит оба. Это и есть **SQL-инъекция**: данные, которые должны были стать значением, прорвались и стали кодом.

<Callout type="warning" title="Главное правило работы с базой">
Пользовательские данные никогда не попадают в текст запроса напрямую. Текст SQL пишете Вы; значения передаются отдельным каналом - через параметры. Граница между «код запроса» и «данные» не должна зависеть от содержимого данных.
</Callout>

Правильный способ - оставить в SQL *плейсхолдер* и передать значение вторым аргументом `execute`. В `sqlite3` плейсхолдер - знак вопроса `?`:

```python
name = "'; DROP TABLE users; --"
rows = cur.execute("SELECT * FROM users WHERE name = ?", (name,)).fetchall()
print(rows)
```

Печатается `[]`: база честно ищет пользователя с таким диким именем, не находит и возвращает пустой список. Таблица цела. Драйвер передал значение отдельно от текста запроса, поэтому никакие кавычки и точки с запятой внутри `name` уже не меняют структуру SQL - они остаются просто символами искомой строки.

На рисунке видно, как параметризация разводит два потока: текст запроса и значения идут к базе разными каналами и встречаются только внутри, где значение уже не может стать кодом.

Параметры решают и вторую проблему - экранирование. Имя `O'Brien` с апострофом внутри сломало бы f-строку (апостроф закрыл бы строковый литерал SQL), а через параметр оно передаётся без единой заботы об экранировании. Именованные плейсхолдеры (`:name`) тоже поддерживаются и читаются яснее, когда параметров много: `execute("... WHERE name = :name", {"name": name})`.

## Транзакция: всё или ничего

`execute` с `INSERT` или `UPDATE` меняет данные, но изменение пока живёт только внутри *транзакции* - незавершённого набора правок. Пока Вы не вызвали `conn.commit()`, эти правки не зафиксированы: другой процесс их не увидит, а сбой откатит их целиком. `commit` объявляет «всё прошло, фиксируй», `rollback` - «что-то пошло не так, отмени всё с прошлого commit».

Это даёт атомарность: связанные изменения либо применяются вместе, либо не применяются вовсе. Классический пример - перевод денег между счетами: списание и зачисление должны произойти оба или ни один.

```python
try:
    cur.execute("UPDATE accounts SET balance = balance - 100 WHERE id = 1")
    cur.execute("UPDATE accounts SET balance = balance + 100 WHERE id = 2")
    conn.commit()
except Exception:
    conn.rollback()
    raise
```

Если второй `UPDATE` упадёт, `rollback` вернёт и первый: деньги не исчезнут со счёта 1, не появившись на счёте 2. Без транзакций между двумя `execute` могло бы остаться окно, где сто рублей нигде не существуют.

## Контекстный менеджер закроет ресурсы за Вас

Соединение - это ресурс: его нужно закрывать, иначе файл базы останется заблокированным, а незафиксированные правки потеряются. Ручной `conn.close()` легко пропустить, если между открытием и закрытием выскочит исключение. Контекстный менеджер `with` снимает эту заботу.

```python
with sqlite3.connect("app.db") as conn:
    cur = conn.cursor()
    cur.execute("INSERT INTO users (name, age) VALUES (?, ?)", ("Вера", 29))
```

Здесь есть тонкость, на которой спотыкаются многие. `with sqlite3.connect(...)` управляет *транзакцией*, а не закрытием соединения: при выходе без исключения он делает `commit`, при исключении - `rollback`. Само соединение этим блоком не закрывается. Чтобы гарантированно закрыть и его, оборачивают соединение ещё и в `contextlib.closing` или используют пул соединений - этим, кстати, и займётся за Вас SQLAlchemy в следующих разделах.

## Когда поведение удивит

Без `commit` Ваши `INSERT` существуют только внутри текущего соединения и исчезнут при его закрытии, если оно произошло вне `with`-блока. Программа отработала, ошибок нет, а в файле базы пусто - потому что транзакцию никто не зафиксировал.

```python
conn = sqlite3.connect("app.db")
conn.execute("INSERT INTO users (name) VALUES ('Глеб')")
conn.close()
```

После этого `Глеб` в базе не появится: соединение закрылось, не дойдя до `commit`, и незавершённая транзакция откатилась. Лекарство - либо явный `conn.commit()` перед закрытием, либо `with`-блок, который зафиксирует за Вас.

## Типичные ошибки

- **Склейка запроса f-строкой или `+`.** Любой запрос с пользовательскими данными в тексте - потенциальная инъекция. Всегда `?`/`:name` и значения отдельным аргументом, даже если «данные точно безопасные»: безопасность данных не должна быть условием корректности.
- **Передать один параметр не кортежем.** `execute("... = ?", name)` упадёт: второй аргумент должен быть последовательностью. Для одного значения пишите `(name,)` - именно с запятой, иначе это не кортеж, а просто скобки.
- **Забыть `commit`.** `INSERT`/`UPDATE`/`DELETE` без последующего `commit` (и вне `with`-блока) не сохраняются. Запрос «отработал», но данные не записаны.

<Detail summary="Почему именно ? и :name, а не подстановка драйвером в текст">

Может показаться, что драйвер просто аккуратно экранирует значение и вставляет его в текст за Вас. На многих базах происходит иначе и надёжнее: запрос с плейсхолдерами отправляется на сервер один раз как *подготовленный* (prepared) шаблон, а значения передаются отдельным сообщением. Сервер компилирует план запроса по шаблону, где на месте параметра стоит «дырка», и значение физически не может повлиять на структуру плана.

Побочный бонус - производительность: один и тот же шаблон с разными значениями сервер может переиспользовать, не разбирая SQL заново. В `sqlite3` это видно через `executemany`, который прогоняет один запрос по списку наборов параметров:

```python
cur.executemany(
    "INSERT INTO users (name, age) VALUES (?, ?)",
    [("Анна", 30), ("Борис", 17), ("Вера", 29)],
)
```

</Detail>
