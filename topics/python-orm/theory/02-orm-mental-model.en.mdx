# ORM is a translator between rows and objects

When you write raw SQL, two incompatible worlds live in your program. In Python — objects with attributes and methods; in the database — table rows and tuples of values. Every query you translate by hand: unpack the tuple `(1, 'Анна', 30)` into fields, build an `INSERT` from attributes. ORM (Object-Relational Mapping) takes over that translation. You describe which class maps to which table, then work with objects — the ORM generates SQL and turns result rows into instances.

By the end of this section you will understand exactly what ORM maps to what, how to declare a model in modern SQLAlchemy 2.x style, and what hides behind two key mechanisms — identity map and unit of work — without which ORM behavior feels like magic.

## What maps to what

ORM mapping works on three levels, and it helps to keep them separate in your head:

- **Class ↔ table.** Class `User` corresponds to table `users`.
- **Instance ↔ row.** Object `User(name="Анна")` corresponds to one row in that table.
- **Attribute ↔ column.** Field `user.name` corresponds to column `name`.

## Declaring a model in SQLAlchemy 2.x

Modern SQLAlchemy (2.x) describes models through typed annotations. The base class inherits from `DeclarativeBase`, and columns are declared with `Mapped[...]` and `mapped_column(...)`. The type annotation sets both the Python type and (usually) the column type and whether the field is required.

```python
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column

class Base(DeclarativeBase):
    pass

class User(Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str]
    age: Mapped[int]
```

`__tablename__` sets the table name. `Mapped[int]` for `id` plus `primary_key=True` declare the primary key. `Mapped[str]` with no extra options is a required (NOT NULL) text column; `Mapped[str | None]` would make it optional. This declaration is enough for SQLAlchemy to both create the table and translate rows to `User` objects and back.

<Callout type="info">
The `Mapped[...]` / `mapped_column(...)` style is SQLAlchemy 2.x. In older code you will see `Column(Integer, primary_key=True)` without type annotations. Both work, but the new style gives type checking and reads more clearly — write new models with it.
</Callout>

## Identity map: one row — one object in a session

The first mechanism that explains ORM "magic" is the identity map. All work happens inside a *session* (`Session`). The session guarantees: within one session, each row (by primary key) maps to exactly one Python object. If you request user with `id=1` twice, you get the same object, not two copies.

```python
u1 = session.get(User, 1)
u2 = session.get(User, 1)
print(u1 is u2)
```

This prints `True`: the second request did not create a new object, it returned the one already loaded from the identity map. This matters beyond saving memory: thanks to the identity map, changing `u1.name` is automatically visible through `u2` — it is one object. Without an identity map, two copies would diverge and it would be unclear whose changes reach the database.

## Unit of work: changes accumulate and leave in a batch

The second mechanism is unit of work. The session does not send SQL on every attribute assignment. Instead it *tracks* changes: which objects are new (`add`), which are modified (an attribute changed), which are deleted. Accumulated changes become SQL all at once — on `flush` (which the session usually does automatically before a query) and are committed on `commit`.

```python
user = User(name="Вера", age=29)
session.add(user)
user.age = 30
session.commit()
```

Here, until `commit` (more precisely, until flush), the database is untouched: `add` only marked the object as new, assigning `age = 30` only updated the field in memory. On `commit` the session generates one `INSERT` with the final value `age=30`. You describe *what* should become true; *when and with which queries* to achieve it is unit of work's job.

## Feel the mapping by hand

You can see the row ↔ object idea even without an ORM — on bare `sqlite3`. Here we manually do what the ORM automates: turn a row tuple into an object with attributes. Run it and notice how much manual code goes into translation that the ORM hides behind `session.get`.

<PyDemo
  title="Manual row-to-object translation (what the ORM does)"
  code={`import sqlite3

class User:
    def __init__(self, id, name, age):
        self.id = id
        self.name = name
        self.age = age
    def __repr__(self):
        return f"User({self.id}, {self.name!r}, {self.age})"

conn = sqlite3.connect(":memory:")
conn.execute("CREATE TABLE users (id INTEGER, name TEXT, age INTEGER)")
conn.execute("INSERT INTO users VALUES (1, 'Анна', 30)")

row = conn.execute("SELECT id, name, age FROM users WHERE id = 1").fetchone()
user = User(*row)
print(user)
print(user.name)`}
/>

## Common mistakes

- **Thinking an attribute assignment writes to the database immediately.** It does not: unit of work accumulates changes and sends SQL on flush/commit. If you forget `commit`, changes are not saved — just like with raw SQL.
- **Creating models in the old style without need.** `Column(...)` without annotations works, but new code uses `Mapped[...]` / `mapped_column(...)` for typing and readability.
- **Being surprised that two queries returned one object.** That is the identity map, and it is correct: within a session a row is represented by one object, so `u1 is u2`.
- **Working with objects outside a session and expecting them to stay in sync.** Change tracking lives in the session; an object detached from it stops being "live" (more on that in the pitfalls section).

<Detail summary="Why ORM is called a leaky abstraction">

ORM hides SQL but does not eliminate it. Every operation eventually becomes a query, and not knowing which SQL your code produced is the source of the trickiest performance problems (the classic example is N+1, covered in its own section). Experienced developers keep ORM SQL logging enabled (`echo=True` on `create_engine`) and periodically check which queries it generates. The abstraction is convenient only as long as you understand what is underneath. Raw SQL knowledge from the previous section is not a completed stage — it is the foundation for using ORM well.

</Detail>
