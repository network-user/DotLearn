# Before you reach for an ORM, learn to talk to the database directly

You already know how to write SQL. Now you need to run it from Python: open a connection, send a query, pull rows back into your program. The standard library gives you `sqlite3` for this. Working through this layer by hand before hiding it behind an ORM matters. Developers who understand that an ORM ultimately drives a cursor and sends text strings debug problems in minutes, not hours.

By the end of this section you will be able to open a connection and cursor, run a query and fetch results, substitute user data safely through parameters (and explain why an f-string in a query is a security hole), and manage a transaction with `commit` and `rollback` without forgetting to close resources with a context manager.

## Connection, cursor, result — a three-step pipeline

Every database operation in Python follows the same path. First you open a *connection* (`Connection`) — a channel to a database file or server. Then you get a *cursor* (`Cursor`) from it — the object through which queries are sent and answers read. The cursor runs SQL with `execute`, and you fetch result rows with `fetchone`, `fetchall`, or by iterating the cursor in a loop.

A minimal session fits in a few lines. Create a table, insert a couple of rows, read them back:

```python
import sqlite3

conn = sqlite3.connect(":memory:")
cur = conn.cursor()
cur.execute("CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT, age INTEGER)")
cur.execute("INSERT INTO users (name, age) VALUES ('Анна', 30)")
cur.execute("INSERT INTO users (name, age) VALUES ('Борис', 17)")
rows = cur.execute("SELECT name, age FROM users").fetchall()
print(rows)
```

This prints `[('Анна', 30), ('Борис', 17)]`. Each result row is a tuple, and `fetchall` collects them into a list. The `:memory:` string means a database that lives only in RAM — ideal for examples and tests, because no file remains after the program ends.

`fetchone` returns the next single row (or `None` when rows are exhausted); `fetchall` returns all remaining rows at once. Iterating the cursor in a loop reads rows one at a time without loading the entire result into memory — how large result sets are handled.

```python
for row in cur.execute("SELECT name FROM users"):
    print(row[0])
```

## Never build a query with an f-string

Here is the most important point in this section. Suppose a user name comes from outside — a form, a command-line argument — and you want to find it in the database. It is tempting to assemble the query as a string:

```python
name = "Анна"
cur.execute(f"SELECT * FROM users WHERE name = '{name}'")
```

This works until `name` turns hostile. Substitute `name = "'; DROP TABLE users; --"` and the f-string assembles two queries: a harmless `SELECT` and a command to drop the table. The database obediently runs both. That is **SQL injection**: data that should have been a value breaks through and becomes code.

<Callout type="warning" title="The golden rule of database work">
User data never goes directly into query text. You write the SQL; values travel through a separate channel — parameters. The boundary between "query code" and "data" must not depend on what the data contains.
</Callout>

The correct approach is to leave a *placeholder* in the SQL and pass the value as the second argument to `execute`. In `sqlite3` the placeholder is a question mark `?`:

```python
name = "'; DROP TABLE users; --"
rows = cur.execute("SELECT * FROM users WHERE name = ?", (name,)).fetchall()
print(rows)
```

This prints `[]`: the database honestly searches for a user with that wild name, finds none, and returns an empty list. The table is intact. The driver sent the value separately from the query text, so no quotes or semicolons inside `name` can change the SQL structure — they remain just characters in the searched string.

Parameters also solve a second problem — escaping. The name `O'Brien` with an apostrophe would break an f-string (the apostrophe would close the SQL string literal), but through a parameter it passes without any escaping concern. Named placeholders (`:name`) are supported too and read more clearly when there are many parameters: `execute("... WHERE name = :name", {"name": name})`.

## Transactions: all or nothing

`execute` with `INSERT` or `UPDATE` changes data, but the change lives only inside a *transaction* — an unfinished set of modifications. Until you call `conn.commit()`, those changes are not persisted: other processes cannot see them, and a crash rolls them back entirely. `commit` means "all good, persist"; `rollback` means "something went wrong, undo everything since the last commit".

This gives atomicity: related changes either apply together or not at all. The classic example is transferring money between accounts: debit and credit must both happen or neither.

```python
try:
    cur.execute("UPDATE accounts SET balance = balance - 100 WHERE id = 1")
    cur.execute("UPDATE accounts SET balance = balance + 100 WHERE id = 2")
    conn.commit()
except Exception:
    conn.rollback()
    raise
```

If the second `UPDATE` fails, `rollback` undoes the first too: the money does not vanish from account 1 without appearing on account 2. Without transactions there could be a window between two `execute` calls where the hundred dollars exist nowhere.

## A context manager closes resources for you

A connection is a resource: it must be closed, otherwise the database file stays locked and uncommitted changes are lost. A manual `conn.close()` is easy to skip if an exception fires between open and close. The `with` context manager handles that.

```python
with sqlite3.connect("app.db") as conn:
    cur = conn.cursor()
    cur.execute("INSERT INTO users (name, age) VALUES (?, ?)", ("Вера", 29))
```

Here is a subtlety many people trip on. `with sqlite3.connect(...)` manages the *transaction*, not closing the connection: on exit without an exception it commits; on exception it rolls back. The connection itself is not closed by this block. To guarantee closing, wrap the connection in `contextlib.closing` or use a connection pool — which, incidentally, SQLAlchemy will handle for you in later sections.

## When behavior surprises you

Without `commit`, your `INSERT` statements exist only inside the current connection and disappear when it closes if that happens outside a `with` block. The program finishes with no errors, but the database file is empty — because nobody committed the transaction.

```python
conn = sqlite3.connect("app.db")
conn.execute("INSERT INTO users (name) VALUES ('Глеб')")
conn.close()
```

After this, `Глеб` will not appear in the database: the connection closed without reaching `commit`, and the unfinished transaction rolled back. The fix is either an explicit `conn.commit()` before closing, or a `with` block that commits for you.

## Common mistakes

- **Gluing a query with an f-string or `+`.** Any query with user data in the text is a potential injection. Always use `?`/`:name` and pass values as a separate argument, even when "the data is surely safe": correctness must not depend on data being safe.
- **Passing a single parameter without a tuple.** `execute("... = ?", name)` fails: the second argument must be a sequence. For one value write `(name,)` — the comma matters, otherwise it is not a tuple, just parentheses.
- **Forgetting `commit`.** `INSERT`/`UPDATE`/`DELETE` without a subsequent `commit` (and outside a `with` block) are not saved. The query "succeeded", but the data was not written.

<Detail summary="Why ? and :name, not driver-side text substitution">

It may look like the driver simply escapes the value and inserts it into the text for you. On many databases something more reliable happens: a query with placeholders is sent to the server once as a *prepared* template, and values travel in a separate message. The server compiles a query plan from the template, where a parameter slot is a "hole", and the value physically cannot affect the plan structure.

A side benefit is performance: the same template with different values can be reused without re-parsing SQL. In `sqlite3` this shows up through `executemany`, which runs one query over a list of parameter sets:

```python
cur.executemany(
    "INSERT INTO users (name, age) VALUES (?, ?)",
    [("Анна", 30), ("Борис", 17), ("Вера", 29)],
)
```

</Detail>
