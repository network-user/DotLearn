# N+1: как одна строка кода рождает тысячу запросов

Самая дорогая ошибка в работе с ORM почти невидима в коде. Цикл `for user in users: print(user.orders)` выглядит безобидно - но если связь `orders` ленивая, каждый виток цикла тайком отправляет в базу отдельный запрос. На сотне пользователей это сто один запрос вместо двух. Программа работает, тесты на трёх записях зелёные, а на проде страница открывается пять секунд. Это проблема N+1, и узнавать её надо в лицо, потому что компилятор о ней не предупредит.

К концу раздела Вы научитесь распознавать N+1 по логам SQL, чинить его жадной загрузкой, понимать жизненный цикл `Session` (и почему объект иногда «отваливается» с `DetachedInstanceError`), и трезво решать, когда задача проще решается сырым SQL, чем через ORM.

## Откуда берётся N+1

Вспомним ленивую загрузку из прошлого раздела. Связь не грузится, пока к ней не обратятся. Теперь поставим обращение в цикл:

```python
users = session.scalars(select(User)).all()
for user in users:
    print(user.name, len(user.orders))
```

Первая строка - **один** запрос: `SELECT * FROM users`. Но `user.orders` внутри цикла ленив, поэтому на **каждого** из N пользователей выполняется свой `SELECT * FROM orders WHERE user_id = ?`. Итог: `1 + N` запросов. Имя «N+1» именно отсюда. Сам по себе каждый запрос быстрый, но сетевые задержки и накладные расходы складываются, и на больших N страница умирает.

## Как распознать: смотрите на SQL

N+1 не виден в Python-коде - он виден в логах SQL. Включите эхо запросов у движка, и лавина одинаковых `SELECT ... WHERE user_id = ?` сразу бросится в глаза.

```python
from sqlalchemy import create_engine

engine = create_engine("sqlite:///app.db", echo=True)
```

С `echo=True` SQLAlchemy печатает каждый отправленный запрос. Правило простое: если в логе один и тот же по форме запрос повторяется десятки раз, отличаясь только значением параметра, - это N+1. Тот же приём работал и с raw SQL из первого раздела, и именно поэтому понимание SQL под ORM так важно: проблему видно только на уровне запросов.

<Callout type="tip">
Привыкайте включать `echo=True` в разработке и поглядывать, какой SQL рождает Ваш код. Большинство проблем производительности ORM - это «неожиданно много запросов», и единственный способ их заметить - увидеть запросы.
</Callout>

## Чем чинить: жадная загрузка

Лекарство - заранее сказать ORM, что связь понадобится, чтобы он догрузил её оптом, а не по одному.

```python
from sqlalchemy.orm import selectinload, joinedload

stmt = select(User).options(selectinload(User.orders))
users = session.scalars(stmt).all()
for user in users:
    print(user.name, len(user.orders))
```

Теперь запросов всего два: один за пользователями, второй (`selectinload`) - за всеми их заказами разом, через `WHERE user_id IN (...)`. У жадной загрузки два основных варианта:

- **`selectinload`** - отдельный запрос для связанных объектов с условием `IN`. Хорош для коллекций (один-ко-многим): не раздувает результат и не плодит дубликаты.
- **`joinedload`** - подтягивает связь тем же запросом через `JOIN`. Хорош для связи «к одному» (`order.user`), но на коллекциях размножает строки родителя по числу детей.

## Жизненный цикл Session и DetachedInstanceError

Второй класс подводных камней связан с тем, что объекты ORM живут не сами по себе, а внутри сессии. Сессия - это граница транзакции и «единица работы»: она отслеживает изменения и держит соединение. Объект, загруженный сессией, привязан к ней. Когда сессия закрыта (или сделала `commit` с настройкой expire), объект становится *detached* - оторванным. Обращение к незагруженной ленивой связи у такого объекта падает.

```python
with Session(engine) as session:
    user = session.scalars(select(User)).first()
print(user.orders)
```

Эта строка после `with` нередко падает с `DetachedInstanceError`: сессия закрыта, а `user.orders` лениво и пытается сходить в базу через мёртвое соединение. Лекарств несколько: либо обращаться к связям внутри `with`, пока сессия жива; либо заранее загрузить нужное жадно (`selectinload`); либо настроить сессию так, чтобы не «протухали» атрибуты после commit (`expire_on_commit=False`). Главное - помнить: время жизни объекта ограничено временем жизни сессии.

<Callout type="warning">
Не возвращайте «живые» ORM-объекты за пределы сессии в расчёте лениво дочитать их связи. Либо загрузите всё нужное внутри сессии, либо отдавайте наружу простые структуры (словари, dataclass), а не привязанные к сессии модели.
</Callout>

## Когда сырой SQL лучше ORM

ORM великолепен для типичной работы «достань объекты, измени, сохрани». Но он не серебряная пуля. Есть задачи, где сырой SQL из первого раздела проще, быстрее и честнее:

- **Тяжёлая аналитика и отчёты.** Многоуровневые агрегации, оконные функции, хитрые `GROUP BY` выражаются на SQL прямо, а через ORM превращаются в борьбу с абстракцией.
- **Массовые операции.** `UPDATE ... WHERE` по миллиону строк одним запросом несравнимо дешевле, чем загрузить миллион объектов, изменить и сохранить по одному.
- **Тонкая настройка производительности.** Когда нужен конкретный план запроса или специфичная для СУБД конструкция, прямой SQL даёт полный контроль.

SQLAlchemy этого не запрещает: тот же `session.execute(text("..."))` выполнит сырой SQL внутри той же транзакции. Зрелый подход - не «ORM против SQL», а ORM для объектной работы и сырой SQL там, где он уместнее.

<PyDemo
  title="Считаем запросы: N+1 против оптимизированного доступа"
  code={`class FakeDB:
    def __init__(self):
        self.queries = 0
    def all_users(self):
        self.queries += 1
        return [1, 2, 3, 4, 5]
    def orders_of(self, user_id):
        self.queries += 1
        return [user_id * 10]
    def orders_of_many(self, user_ids):
        self.queries += 1
        return {uid: [uid * 10] for uid in user_ids}

naive = FakeDB()
users = naive.all_users()
for uid in users:
    naive.orders_of(uid)
print("N+1:", naive.queries)

eager = FakeDB()
users = eager.all_users()
eager.orders_of_many(users)
print("eager:", eager.queries)`}
/>

Запустите: наивный доступ печатает `N+1: 6` (1 + 5), жадный - `eager: 2`. Меняйте число пользователей и смотрите, как ленивый счётчик растёт линейно, а жадный остаётся равным двум.

## Типичные ошибки

- **Обращаться к ленивой связи в цикле.** Главный источник N+1. Если связь нужна у многих объектов - грузите её жадно через `selectinload`/`joinedload`.
- **Не смотреть в логи SQL.** N+1 и лишние запросы невидимы в Python. Без `echo=True` (или профайлера запросов) Вы их не заметите, пока не станет медленно.
- **Использовать `joinedload` для больших коллекций.** Он размножает строки родителя по числу детей; для «один-ко-многим» берите `selectinload`.
- **Возвращать ORM-объекты из закрытой сессии.** Ленивое обращение к связи после закрытия сессии даёт `DetachedInstanceError`. Загружайте нужное внутри сессии или отдавайте простые структуры.
- **Тащить всё через ORM.** Для массовых `UPDATE`/`DELETE` и тяжёлой аналитики сырой SQL проще и на порядки дешевле, чем загрузка и сохранение объектов по одному.

<Detail summary="Почему N+1 так легко не заметить на этапе разработки">

N+1 коварен своей незаметностью на малых данных. В тестах и при локальной разработке таблицы крошечные: 1 + 5 запросов отрабатывают за миллисекунды, и проблема не проявляется. Она просыпается на проде, где N - это тысячи строк, а база за сетью с задержкой в миллисекунды на запрос. 1 + 5 превращается в 1 + 5000, и страница, мгновенная в разработке, грузится секундами. Именно поэтому профилируют запросы заранее и в код-ревью обращают внимание на обращения к связям внутри циклов. Дешевле заметить N+1 глазами на ревью, чем ловить его по жалобам на медленный прод.

</Detail>
