# Стек и очередь: один и тот же список, две разные дисциплины

Кнопка «назад» в браузере всегда открывает последнюю посещённую страницу, а не первую. Принтер в офисе печатает документы в том порядке, в каком их поставили в очередь, и не может «перепрыгнуть» через уже стоящие задания ради только что отправленного. Обе системы хранят последовательность элементов, но правило, кто уйдёт следующим, у них противоположное. Это правило целиком и есть разница между стеком и очередью: никакой новой структуры данных под капотом не требуется, требуется лишь дисциплина, с какого конца забирать элементы.

## Стек: последним пришёл, первым ушёл

Стек (stack) отдаёт элементы в порядке, обратном добавлению: последний добавленный уходит первым. Такую дисциплину называют LIFO, last in, first out. Стопка тарелок это ровно она: тарелку кладут сверху и снимают тоже сверху, до нижней доберутся, только когда снимут все, что лежат выше.

В Python отдельного класса Stack нет, потому что он не нужен: обычный `list` уже умеет всё необходимое на своём хвосте. `append` кладёт элемент наверх, `pop()` без аргумента снимает верхний, `items[-1]` подсматривает верхний, не снимая. Все три операции работают с концом списка, где уже выделен запас ёмкости, поэтому каждая стоит O(1) амортизированно, эта механика разобрана в предыдущем разделе про динамический массив.

```python
stack = []
stack.append(1)
stack.append(2)
stack.append(3)
print(stack[-1])
print(stack.pop())
print(stack)
```

<PullQuote>Стек не новая структура данных. Это список, у которого договорились трогать только один конец.</PullQuote>

## Классическая задача: сбалансированные скобки

Стек отвечает на вопрос «что было открыто последним и ещё не закрыто», и это ровно то, что нужно для проверки скобочных выражений в коде, JSON или математических формулах. Каждая открывающая скобка идёт в стек, каждая закрывающая обязана снять с вершины именно ту, что ей парна.

<PyStepper
  title="Проверка скобок: push на открывающей, pop на закрывающей"
  code={`def is_valid(s):
    pairs = {")": "(", "]": "[", "}": "{"}
    stack = []
    for ch in s:
        if ch in "([{":
            stack.append(ch)
        elif ch in ")]}":
            if not stack or stack.pop() != pairs.get(ch, ch):
                return False
    return not stack

print(is_valid("([{}])"))`}
/>

Прошагайте трассировку и обратите внимание на два способа провалить проверку: закрывающая скобка встретилась при пустом стеке (нечего закрывать), либо сняла с вершины символ, который ей не пара. Оба случая возвращают `False` сразу, не дожидаясь конца строки.

## Очередь: первым пришёл, первым ушёл

У очереди (queue) правило обратное: первым уходит тот, кто пришёл первым, FIFO, first in, first out. Она устроена как живая очередь в магазине: новый покупатель встаёт в конец, обслуживают того, кто стоит в начале.

Здесь начинается ловушка. Список Python дёшев только на своём хвосте. `append` на конец стоит O(1), а вот `pop(0)`, снятие с начала, стоит O(n): каждый вызов сдвигает все оставшиеся указатели на одну ячейку влево, ровно как `insert(0, x)` из раздела про динамический массив. На небольших очередях это незаметно, но очередь задач, растущая до тысяч элементов, на `list.pop(0)` превращается в скрытый O(n²).

<Callout type="warning" title="list как очередь: ловушка производительности">
`queue.append(x)` дешёвый, а `queue.pop(0)` дорогой. Работа с обоими концами списка требует структуры, у которой оба конца дешёвые, а не только хвост.
</Callout>

## collections.deque: очередь без компромиссов

Модуль `collections` даёт `deque` (double-ended queue, произносится «дек»), структуру, у которой O(1) амортизированно работают все четыре операции по краям: `append`, `appendleft`, `pop`, `popleft`. Внутри `deque` устроен как двусвязный список блоков фиксированного размера, а не как один непрерывный массив: добавление с любого конца просто пишет в свой блок и не трогает противоположный. За это `deque` расплачивается доступом по индексу, `dq[500000]` идёт по цепочке блоков и стоит O(n), тогда как у `list` такой доступ O(1).

```python
from collections import deque

queue = deque()
queue.append("a")
queue.append("b")
queue.append("c")
print(queue.popleft())
print(queue.popleft())
print(queue)
```

## deque(maxlen=...): кольцевой буфер бесплатно

У `deque` есть параметр `maxlen`. Когда он задан, структура превращается в буфер фиксированного размера: добавление элемента сверх лимита автоматически выталкивает элемент с противоположного конца. Это ровно то, что нужно для «последних N логов», «истории последних действий», скользящего окна метрик.

<PyDemo
  title="deque(maxlen=3): новый элемент вытесняет старый"
  code={`from collections import deque

history = deque(maxlen=3)
for page in ["main", "cart", "checkout", "payment", "done"]:
    history.append(page)
    print(list(history))`}
/>

Поменяйте `maxlen` на 5 и посмотрите, как быстро буфер перестаёт вытеснять страницы: пока элементов меньше лимита, `deque` ведёт себя как обычная очередь без потерь.

## Три применения стека, которые встречаются каждый день

Стек редко называют по имени, но использует его почти любая программа:

- **Стек вызовов.** Каждый вызов функции кладёт кадр (frame) со своими локальными переменными и адресом возврата. Функция завершилась, кадр снимается, управление возвращается туда, куда указывал верхний кадр. Рекурсия работает ровно на этом механизме, и `RecursionError` в CPython, это переполнение именно стека вызовов.
- **Отмена действия (undo).** Редактор кладёт в стек каждое изменение. `Ctrl+Z` снимает верхнее и откатывает его, `Ctrl+Y` может класть отменённое действие в отдельный стек redo.
- **Обход в глубину (DFS).** В прошлой теме про графы DFS шёл через рекурсию или явный стек: снять вершину сверху, обработать, положить её непосещённых соседей. Замените в коде BFS `queue.popleft()` на `stack.pop()`, и обход волнами превратится в обход вглубь.

## Краевые случаи

- **pop из пустой структуры.** И `list.pop()`, и `deque.popleft()` бросают `IndexError` на пустой коллекции. Код, снимающий элементы в цикле, обязан явно проверять `if stack:` или `while queue:`, а не полагаться на исключение как на штатную остановку.
- **queue.pop(0) молча работает, но медленно.** Ошибка не бросает исключение и не ломает результат, программа просто становится квадратичной на больших объёмах. Профилировщик покажет разбухшее время в невинной с виду строке.
- **deque(maxlen=n) при переполнении не предупреждает.** Вытесненный элемент исчезает без следа. Если старые данные нужно сохранить, прежде чем деку их вытеснит, скопируйте их отдельно до `append`.

## Частые ошибки

- **Использовать `list.pop(0)` для очереди.** Каждый вызов двигает весь остаток списка. Исправление: `collections.deque` и `popleft()`.
- **Путать `stack.pop()` и `queue.popleft()`.** Одна буква решает, какой конец списка вы читаете, а с ним и всю логику алгоритма. Перед тем как писать код, явно проговорите: LIFO или FIFO нужен в этой задаче.
- **Проверять «стек пуст» через `len(stack) == 0` вместо `not stack`.** Работает одинаково, но вторая форма это принятая идиома Python и читается быстрее.
- **Забыть, что `deque` не даёт O(1) доступ по индексу.** Код вида `dq[len(dq) // 2]` в горячем цикле незаметно расставляет O(n) там, где ожидался O(1).

<Detail summary="Как deque устроен внутри: блоки, а не один массив">

CPython реализует `deque` как двусвязный список блоков фиксированного размера (в текущих версиях по 64 указателя на блок). Добавление с любого края почти всегда пишет в уже выделенный блок на этом краю, лишь изредка выделяя новый блок и приклеивая его к цепочке, поэтому `append` и `appendleft` в среднем O(1). Между блоками нет общей непрерывной памяти, поэтому доступ `dq[i]` вычисляет, в каком блоке лежит нужный индекс, и идёт по цепочке блоков, O(n) в худшем случае. Компромисс продуман осознанно: `deque` жертвует индексным доступом ради одинаково дешёвых операций на обоих концах.

</Detail>
