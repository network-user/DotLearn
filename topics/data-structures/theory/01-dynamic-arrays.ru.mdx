# Список Python: массив, который притворяется резиновым

Возьмите список из десяти элементов и список из ста миллионов. Попросите у каждого элемент с индексом семь. Время ответа совпадёт: Python не перебирает элементы в поисках нужного, он сразу знает, где тот лежит. Теперь второй опыт: замените в цикле добавление в конец списка на вставку в начало. Скрипт, который прожёвывал лог за пару секунд, повиснет на долгие минуты, хотя объём данных не изменился ни на байт.

Обе истории объясняются одной картинкой: тем, как список раскладывает элементы в памяти. Пока этой картинки нет в голове, стоимость операций выглядит лотереей. С ней каждая строчка из таблиц сложности превращается в предсказуемое следствие.

## Ментальная модель: полка ячеек и арифметика адресов

Представьте полку с пронумерованными ячейками одинаковой ширины, стоящими вплотную, без зазоров. Чтобы добраться до ячейки номер `i`, не нужно идти вдоль полки: адрес вычисляется одним умножением и одним сложением, начало плюс номер, умноженный на ширину ячейки. Процессор просто прыгает по готовому адресу. Отсюда доступ по индексу за O(1), одинаково быстрый на любой длине.

Список Python устроен ровно так, с одной поправкой: в ячейках лежат не сами значения, а указатели на объекты. Каждая ячейка занимает 8 байт на 64-битной системе, поэтому в одном списке спокойно уживаются число, строка и словарь: с точки зрения массива всё это адреса одинакового размера. Сами объекты разбросаны по куче, непрерывен только массив указателей.

На рисунке видна и вторая ключевая деталь: ячеек выделено больше, чем занято. Число занятых хранится в поле размера, его и возвращает `len`. Всё выделенное называют ёмкостью, capacity. Запас в хвосте и делает массив «динамическим»: очередной `append` записывает указатель в свободную ячейку, увеличивает размер, и на этом всё.

<PullQuote>Список Python не хранит ваши объекты. Он хранит адреса, по которым их можно найти.</PullQuote>

## Стоимость операций выводится из картинки

Чтение и замена по индексу: O(1), чистая арифметика адресов. Добавление в конец: почти всегда O(1), запись в свободную ячейку из запаса. А вот всё, что трогает начало или середину, обязано сохранить массив непрерывным. Вставка в нулевую позицию сдвигает каждый из n указателей на одну ячейку вправо, удаление из начала сдвигает их влево. Обе операции стоят O(n) независимо от того, насколько «маленький» элемент вы вставляете.

```python
a = [10, 20, 30]
a.insert(0, 99)
print(a)
a.pop(0)
print(a)
```

Здесь оба вызова отработали мгновенно, потому что элементов три. Но на миллионном списке в цикле счёт пойдёт на минуты: каждый `insert(0, ...)` перекладывает весь массив заново.

Поиск по значению тоже не бесплатный. Массив указателей никак не упорядочен по содержимому, поэтому оператор `in` идёт по ячейкам подряд, разыменовывает указатели и сравнивает объекты, пока не встретит совпадение или конец: O(n).

```python
names = ["Аня", "Борис", "Вера"]
print("Вера" in names)
print("Глеб" in names)
```

Для редких проверок это нормально. Если проверки принадлежности идут тысячами, нужен `set` с его хеш-таблицей, но это отдельная тема.

## Рост capacity: почему append почти всегда дёшев

Рано или поздно запас кончается: размер догоняет ёмкость. Тогда список выделяет новый блок побольше, копирует в него все указатели и отпускает старый. Это честная O(n)-операция. Почему же `append` считается дешёвым?

Секрет в размере нового блока. Ёмкость растёт мультипликативно, с запасом на будущее. В учебной модели она удваивается: 4, 8, 16, 32, 64. Перевыделения случаются на пятом, девятом, семнадцатом, тридцать третьем добавлении, и дальше всё реже. Чтобы дорасти до миллиона элементов, хватает двух десятков переездов. Суммарно на n добавлений приходится порядка n скопированных указателей, в среднем константа на операцию. Такую усреднённую цену называют амортизированной: `append` стоит O(1) амортизированно.

CPython на практике осторожнее учебной модели: он растит ёмкость примерно в 1.125 раза, добавляя восьмую часть текущего размера и небольшую константу. Принцип не меняется: прибавка пропорциональна размеру, поэтому чем длиннее список, тем реже он переезжает. Для рассуждений о сложности смело держите в голове удвоение, выводы совпадают.

<Detail summary="Что именно делает CPython при переполнении">

В исходниках CPython (файл `listobject.c`) новая ёмкость вычисляется по формуле, эквивалентной `size + size // 8 + 6`, с округлением до кратного четырём. Для коротких списков заметна константная прибавка, для длинных работает множитель примерно 1.125. Там же живёт обратная механика: если после удалений размер падает ниже половины ёмкости, массив ужимается. Поле ёмкости называется `allocated`, массив указателей `ob_item`, размер `ob_size`: все три видны в структуре `PyListObject`.

</Detail>

## Алиасинг: два имени одного массива

Присваивание в Python никогда не копирует данные. Строка `b = a` не создаёт второй список, она вешает второй ярлык на тот же массив указателей. Любое изменение через одно имя видно через другое, потому что объект один.

```python
a = [1, 2, 3]
b = a
b.append(4)
print(a)
print(a is b)
```

Настоящую копию просят явно: `list(a)`, срез `a[:]` или `copy.copy(a)`. Все три строят новый массив указателей той же длины, и дальше списки живут независимо.

```python
a = [1, 2, 3, 4]
c = list(a)
c.append(5)
print(a)
print(c)
```

<Callout type="warning" title="Копия поверхностная">
Копируются указатели, а не объекты. Если внутри списка лежат вложенные списки, копия и оригинал продолжают разделять их содержимое: `c[0].append(9)` изменит и `a[0]`. Полную независимость даёт `copy.deepcopy`.
</Callout>

## Краевые случаи

- **Литерал выделяется впритык.** Список из литерала `[1, 2, 3]` или из `list(range(n))` создаётся без запаса: длина известна заранее, и CPython не резервирует хвост. Первый же `append` к такому списку вызывает перевыделение.
- **`a += [x]` и `a = a + [x]` ведут себя по-разному.** Первый вариант дописывает в существующий массив, и все алиасы видят изменение. Второй строит новый список и перевешивает имя, а алиасы остаются со старым содержимым.
- **`pop()` без аргумента ничего не сдвигает.** Удаление с конца просто уменьшает размер, ячейка уходит обратно в запас. Симметрия обманчива: `pop()` стоит O(1), `pop(0)` стоит O(n).

Разница между `+=` и `+` в живом виде:

```python
a = [1, 2]
b = a
a += [3]
print(b)
a = a + [4]
print(a)
print(b)
```

Вторая операция отвязала имя `a` от общего списка, и `b` перестал видеть новые элементы. С учётом алиасинга такие «синонимы» дают разное поведение всей программы.

## Частые ошибки

- **Ожидать, что `b = a` делает копию.** Потом данные «сами» портятся в другом конце программы. Исправление: `list(a)`, `a[:]` или `copy.copy(a)`, для вложенных структур `copy.deepcopy(a)`.
- **Гонять `insert(0, x)` или `pop(0)` в цикле.** Каждый вызов перекладывает весь массив, и цикл превращается в O(n²). Исправление: `collections.deque` с дешёвыми операциями с обоих концов, о ней следующий концепт.
- **Проверять `x in big_list` в горячем цикле.** Каждая проверка сканирует список целиком. Исправление: один раз построить `set` и проверять принадлежность в нём.
- **Размножать вложенные списки умножением.** Выражение `[[0] * 3] * 2` копирует указатель на одну и ту же внутреннюю строку, и правка `grid[0][0]` «портит» обе строки сразу. Исправление: генератор списка.

```python
bad = [[0] * 3] * 2
bad[0][0] = 9
print(bad)
good = [[0] * 3 for _ in range(2)]
good[0][0] = 9
print(good)
```
