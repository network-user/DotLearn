# Куча: структура для вопроса «кто сейчас самый маленький»

Приёмный покой в больнице не обслуживает пациентов в порядке прихода: пациента с остановкой сердца примут раньше того, кто пришёл на час раньше с растяжением. Планировщик операционной системы работает так же: у каждого процесса приоритет, и следующим на процессор ставится самый приоритетный, а не самый старый в очереди. Обеим системам не нужен полный отсортированный список всех ожидающих, им нужен быстрый ответ на один-единственный вопрос: кто сейчас самый срочный. Структуру, заточенную ровно под этот вопрос, называют кучей (heap).

## Мысленная модель: почти сбалансированное дерево с одним правилом

Куча, как и BST из предыдущего раздела, это дерево. Но правило другое и заметно слабее: у BST при каждом сравнении слева меньше, справа больше, у кучи только одно требование, значение родителя не больше значений обоих потомков (для min-heap, кучи с минимумом наверху). Про порядок между двумя детьми одного родителя куча не говорит ничего.

Слабое правило, только родитель не больше детей, стоит слабым не просто так: его дёшево поддерживать. BST после вставки иногда приходится перестраивать целыми поддеревьями ради баланса, куче достаточно локально «всплыть» или «утонуть» новому элементу, и об этом чуть ниже.

<Callout type="info" title="Куча не сортирует, она только знает минимум">
Из формы кучи следует ровно один факт: минимум лежит в корне. Про порядок остальных элементов куча не гарантирует ничего, `heap[1]` не обязан быть вторым по величине.
</Callout>

## Массив вместо дерева: та же экономия, что и у динамического массива

Дерево на рисунке рисуют кружочками и стрелками, но в памяти его хранят без единой ссылки, обычным списком. Узлы кучи нумеруют по уровням слева направо и кладут в массив в этом порядке. Тогда связи между родителем и детьми превращаются в арифметику индексов: у узла с индексом `i` потомки лежат по индексам `2*i + 1` и `2*i + 2`, а родитель по индексу `(i - 1) // 2`.

Никаких узлов-объектов и ссылок `left`/`right`, как в BST, куче они не нужны: обычный список Python и формулы индексов дают то же дерево бесплатно, без накладных расходов на объекты-обёртки.

## heapq: минимальная куча из стандартной библиотеки

В Python куча живёт в модуле `heapq` и представляет собой не отдельный класс, а набор функций, которые поддерживают свойство кучи на обычном списке. `heappush` добавляет элемент и сохраняет свойство кучи, `heappop` снимает и возвращает минимум, тоже сохраняя свойство. `heap[0]` в любой момент, это минимум без снятия, O(1), потому что минимум по определению кучи всегда в корне, то есть на нулевом индексе массива.

<PyStepper
  title="heappush поднимает новый элемент наверх (sift up)"
  code={`import heapq

heap = []
for value in [5, 1, 8, 2, 9]:
    heapq.heappush(heap, value)
    print(value, "->", heap)`}
/>

Прошагайте вставку и обратите внимание: массив после каждого `heappush` не отсортирован целиком, но `heap[0]` всегда остаётся минимумом из уже вставленных чисел. Новый элемент кладут в конец массива, а затем «просеивают» вверх (sift up): сравнивают с родителем и меняют местами, пока не займёт правильное место или не упрётся в корень. Путь просеивания идёт вдоль одной ветки дерева, поэтому `heappush` стоит O(log n), логарифм от высоты кучи.

`heappop` работает зеркально: снимает корень как результат, переносит на его место последний элемент массива и просеивает его вниз (sift down), меняя местами с меньшим из детей, пока свойство кучи не восстановится. Тоже O(log n).

<PyDemo
  title="heappop снимает минимум за минимумом"
  code={`import heapq

heap = [5, 1, 8, 2, 9]
heapq.heapify(heap)
print("после heapify:", heap)

while heap:
    print(heapq.heappop(heap))`}
/>

`heapq.heapify` строит свойство кучи на уже существующем списке за одну функцию. Интуиция подсказывает O(n log n), n просеиваний по log n каждое, но фактическая стоимость ниже, O(n): просеивание вниз, с которого начинается `heapify`, дёшево для узлов у самого низа дерева, а таких узлов, у которых просеивать почти нечего, большинство. Точное доказательство выходит за рамки собеседования джуна, достаточно помнить сам факт: `heapify` дешевле, чем n раз подряд вызвать `heappush`.

## Куда её применяют: top-k, планировщик, Дейкстра

- **k наибольших или наименьших значений.** `heapq.nlargest(k, items)` и `heapq.nsmallest(k, items)` решают задачу top-k без полной сортировки всего набора. Альтернативная ручная реализация держит min-heap размера k: каждый новый элемент крупнее текущего минимума кучи вытесняет его, `heapq.heapreplace`.
- **Приоритетная очередь задач.** Задачи кладут в кучу парами `(приоритет, задача)`, снимают через `heappop`, и наружу всегда выходит самая приоритетная задача, а не самая старая по времени постановки.
- **Алгоритм Дейкстры.** В теме про графы уже упоминалось: BFS находит кратчайший путь только в невзвешенном графе, а на взвешенном рёбрами графе ту же роль очереди играет куча, вершины всплывают в порядке накопленной стоимости пути, а не в порядке номера рёбер.

## Max-heap через минус: трюк, а не отдельная структура

`heapq` реализует только min-heap, кучу с минимумом наверху. Когда нужен максимум наверху, элементы кладут в кучу с обратным знаком: `heapq.heappush(heap, -value)`, а при чтении знак возвращают обратно, `-heap[0]`. Куча по-прежнему держит наверху «минимум», просто из отрицательных чисел, а минимум отрицательного числа соответствует максимуму исходного.

```python
import heapq

max_heap = []
for value in [5, 1, 8, 2, 9]:
    heapq.heappush(max_heap, -value)

print(-max_heap[0])
print(-heapq.heappop(max_heap))
```

## Краевые случаи

- **Пустая куча.** `heapq.heappop([])` бросает `IndexError`. Код, который снимает элементы в цикле, обязан проверять `while heap:`, а не рассчитывать, что куча сама скажет, когда закончится.
- **Куча не хранит отсортированный массив.** Только `heap[0]` гарантированно минимум. `heap[1]` и `heap[2]` не обязаны идти по возрастанию, вывести отсортированный список можно только последовательными `heappop`, и это разрушает кучу.
- **Кортежи с одинаковым первым элементом.** Если в кучу кладут `(приоритет, объект)` и у двух элементов приоритет совпал, Python сравнивает вторые элементы кортежа, чтобы разрешить порядок. Если объекты несравнимы, например два словаря, сравнение упадёт с `TypeError` прямо во время `heappush`, хотя раньше, при разных приоритетах, тот же код работал без ошибок.
- **heapify изменяет список на месте.** Функция возвращает `None`, а не новый список. Код вида `heap = heapq.heapify(items)` перезаписывает `heap` значением `None`, правильно вызывать `heapq.heapify(items)` отдельной строкой и дальше работать с `items`.

## Частые ошибки

- **Забыть про знак при реализации max-heap.** `heapq.heappush(heap, value)` вместо `heapq.heappush(heap, -value)` тихо превращает предполагаемый max-heap в min-heap, и наверх вместо самого срочного всплывает самый незначительный элемент.
- **Класть в кучу кортежи с несравнимым вторым полем.** `heapq.heappush(heap, (priority, task_object))` работает, пока приоритеты не совпадут, а затем падает с `TypeError: '<' not supported`. Исправление: добавить в кортеж счётчик, растущий с каждой вставкой, `(priority, counter, task_object)`, он всегда разный и берёт сравнение на себя раньше, чем Python дойдёт до объекта.
- **Ожидать отсортированный вывод от чтения массива кучи напрямую.** `print(heap)` покажет валидную кучу, но не отсортированный список. Для сортированного порядка нужен цикл `heappop` или отдельный `sorted(...)`.
- **Вызывать heapify(items) как heap = heapify(items).** Функция мутирует список на месте и возвращает `None`; после такого присваивания дальнейшие обращения к `heap` рушат код с ошибками про `NoneType`.

<Detail summary="Почему heapify стоит O(n), а не O(n log n)">

`heapify` запускает просеивание вниз (sift down) для каждого не-листового узла, начиная с самого нижнего уровня и поднимаясь к корню. У кучи из n элементов примерно половина узлов, это листья: просеивать в них нечего. Ещё четверть узлов находится на предпоследнем уровне и просеивается максимум на один шаг вниз. Чем ближе узел к корню, тем дальше ему в теории просеиваться, но таких узлов экспоненциально меньше. Сумма по всем уровням сходится к линейной величине, O(n), а не к n log n, которое получилось бы, если бы каждый узел просеивался на полную высоту дерева.

</Detail>
