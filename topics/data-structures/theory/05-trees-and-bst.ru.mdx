# Деревья и BST: иерархия, в которой поиск не обязан быть линейным

Файловый менеджер показывает папки и вложенные в них папки, и ни одна папка не лежит одновременно в двух родителях. HTML-страница это дерево тегов, где `body` содержит `div`, а `div` содержит `p`. Организационная структура компании: у каждого сотрудника один руководитель, у руководителя несколько подчинённых. Все три картинки объединяет одна фигура: иерархия, где у каждого элемента есть один родитель и произвольное число потомков, а сама иерархия не образует циклов. Эту фигуру называют деревом, и на ней держится один из самых частых вопросов собеседований: как искать в отсортированных данных быстрее, чем перебором, но без накладных расходов на пересортировку при каждой вставке.

## Мысленная модель: узлы и ссылки на потомков

Дерево собирают из узлов, как и связный список из прошлой темы. Разница в числе исходящих ссылок: у узла связного списка одна ссылка `next`, у узла дерева их может быть несколько, по одной на каждого потомка. В двоичном дереве потомков ровно два, их различают: `left` и `right`. Единственный узел без родителя называют корнем (root), узлы без потомков, листьями (leaf), а число рёбер на пути от корня до самого дальнего листа, высотой дерева.

```python
class TreeNode:
    def __init__(self, value, left=None, right=None):
        self.value = value
        self.left = left
        self.right = right

root = TreeNode(5, TreeNode(3), TreeNode(8))
print(root.value)
print(root.left.value)
print(root.right.value)
```

## Двоичное дерево поиска: порядок, зашитый в форму

Голое двоичное дерево ничего не гарантирует про порядок значений. Двоичное дерево поиска (BST, binary search tree) добавляет одно правило и держится на нём целиком: для любого узла все значения в его левом поддереве меньше значения узла, а все значения в правом поддереве больше или равны. Правило рекурсивное, оно верно не только для корня, но и для каждого узла внутри дерева.

Ищем число 4. Начинаем с корня, 5: искомое меньше, уходим налево, к 3. Искомое больше 3, уходим направо, к 4. Значения совпали, поиск завершён за два шага, хотя в дереве семь узлов. На каждом шаге правило BST отбрасывает целое поддерево из рассмотрения, ровно как в бинарном поиске по отсортированному массиву из соседней темы про сложность алгоритмов.

## Поиск, вставка, обход: одна и та же рекурсия

Поиск, вставка и последующий обход опираются на одну и ту же идею: сравнить искомое значение с текущим узлом и решить, налево или направо. Реализация вставки укладывается в несколько строк:

<PyStepper
  title="Вставка в BST: рекурсивный спуск по правилу left/right"
  code={`class TreeNode:
    def __init__(self, value, left=None, right=None):
        self.value = value
        self.left = left
        self.right = right

def insert(node, value):
    if node is None:
        return TreeNode(value)
    if value < node.value:
        node.left = insert(node.left, value)
    else:
        node.right = insert(node.right, value)
    return node

root = None
for v in [5, 3, 8, 1, 4]:
    root = insert(root, v)
print(root.value, root.left.value, root.right.value)`}
/>

Базовый случай рекурсии, `node is None`, срабатывает ровно там, где заканчивается подходящее поддерево: новый узел становится листом. Возврат `node` на каждом уровне переподвешивает изменённое поддерево обратно к родителю, поэтому дерево остаётся связным после каждой вставки.

Обход, который восстанавливает значения по возрастанию, называют inorder: сначала рекурсивно обойти левое поддерево, затем посетить сам узел, затем рекурсивно обойти правое. Для BST это не совпадение, а прямое следствие правила «левое меньше, правое больше», применённого на каждом уровне рекурсии.

<PyDemo
  title="Inorder-обход BST всегда возвращает отсортированный порядок"
  code={`class TreeNode:
    def __init__(self, value, left=None, right=None):
        self.value = value
        self.left = left
        self.right = right

def insert(node, value):
    if node is None:
        return TreeNode(value)
    if value < node.value:
        node.left = insert(node.left, value)
    else:
        node.right = insert(node.right, value)
    return node

def inorder(node, out):
    if node is None:
        return
    inorder(node.left, out)
    out.append(node.value)
    inorder(node.right, out)

root = None
for v in [5, 3, 8, 1, 4, 7, 9]:
    root = insert(root, v)

result = []
inorder(root, result)
print(result)`}
/>

Вставляйте значения в любом порядке, хоть `[9, 1, 8, 2, 7]`, результат `inorder` всегда получится отсортированным. Порядок вставки меняет форму дерева, но не содержимое отсортированного вывода.

## Сложность: всё зависит от формы

Поиск, вставка и удаление в BST стоят O(h), где h, высота дерева: на каждом уровне отбрасывается ровно одно поддерево, и число уровней ограничивает число сравнений. Вопрос в том, чему равно h. На сбалансированном дереве, где обе стороны каждого узла растут примерно поровну, высота равна O(log n): n элементов укладываются в логарифмическое число уровней, ровно как в бинарном поиске.

Проверить деградацию легко: вставьте в дерево значения `1, 2, 3, 4, 5` по возрастанию, и каждое новое число уйдёт направо от предыдущего, потому что оно больше всех уже вставленных. Получится не дерево, а цепочка, и поиск числа 5 пройдёт через все пять узлов, O(n), хотя формально это всё ещё BST.

<MarginNote label="самобалансировка">AVL-дерево и красно-чёрное дерево держат высоту O(log n) при любом порядке вставок за счёт локальных перестроений (поворотов) после каждой операции. Реализация выходит за рамки этого раздела, для собеседования джуна достаточно знать: гарантия существует и стоит она в поворотах узлов.</MarginNote>

## Когда BST, а когда хеш-таблица

Обе структуры ищут значение быстрее, чем полным перебором, но решают разные задачи. Хеш-таблица из прошлой темы не хранит порядок и не умеет отвечать на вопрос «дай все значения от 10 до 20» иначе, чем полным перебором. BST хранит порядок в самой форме дерева, поэтому диапазонный запрос, ближайшее меньшее значение, k-е по счёту значение решаются обходом части дерева, а не всего набора данных.

## Краевые случаи

- **Пустое дерево.** Корень равен `None`. Поиск и обход обязаны пережить это с первого шага: рекурсия с базовым случаем `if node is None: return` останавливается сразу, не читая `.value` у несуществующего узла.
- **Дубликаты значений.** Правило BST в этом разделе отправляет значения, равные текущему узлу, направо (`else` в `insert`). Это соглашение, а не универсальный закон: другая кодовая база может запрещать дубликаты вовсе или отправлять их налево, важно придерживаться одного соглашения во всём коде.
- **Дерево из одного узла.** Он одновременно корень и лист. Поиск и вставка отрабатывают за одно сравнение, высота дерева равна нулю.
- **Отсортированные входные данные.** Вставка по возрастанию или убыванию превращает BST в цепочку, разобрано выше. На собеседовании стоит проговорить это ограничение явно, оно и есть повод для следующего вопроса про самобалансировку.

## Частые ошибки

- **Забыть базовый случай рекурсии.** Без `if node is None: return ...` обход или поиск на пустом поддереве упадёт с `AttributeError: 'NoneType' object has no attribute 'value'`.
- **Не переприсвоить результат рекурсивного вызова.** Строка `insert(node.left, value)` без `node.left = ...` слева ничего не меняет в дереве: новый узел создаётся и тут же теряется, потому что ссылку на него никто не сохранил.
- **Спутать `left < node.value` c `left <= node.value` без явного соглашения о дубликатах.** Несогласованное правило в разных частях кода расщепляет дерево на два разных представления одного и того же набора данных, и поиск начинает промахиваться мимо существующих значений.
- **Ждать O(log n) от несбалансированного дерева.** Асимптотика O(log n) верна для сбалансированного случая. На неудачном порядке вставки настоящая стоимость операций O(n), и об этом стоит явно предупреждать в оценке сложности.

<Detail summary="Обход в ширину дерева тоже нужен, и для него берут deque">

Помимо рекурсивных inorder, preorder и postorder существует обход по уровням, level-order: сначала корень, затем все узлы на глубине 1, затем все узлы на глубине 2. Устроен он как BFS графа из соседней темы: очередь на `collections.deque`, куда сначала кладут корень, а затем на каждом шаге снимают узел через `popleft()` и кладут его непустых потомков `left` и `right`. Та же дисциплина очереди, тот же `popleft()`, только вместо соседей по списку смежности здесь два поля узла.
