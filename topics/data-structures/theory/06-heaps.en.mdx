# Heaps: the structure for answering "who's smallest right now"

A hospital emergency room doesn't treat patients in arrival order: a patient in cardiac arrest gets seen before someone who arrived an hour earlier with a sprain. An OS scheduler works the same way: every process has a priority, and the next one on the CPU is the highest-priority one, not the oldest in line. Neither system needs a fully sorted list of everyone waiting; each needs a fast answer to a single question: who's most urgent right now. The structure built exactly for that question is called a heap.

## Mental model: an almost-balanced tree with one rule

A heap, like the BST from the previous section, is a tree. But the rule is different and noticeably weaker: a BST compares left-smaller, right-bigger at every node; a heap has only one requirement, a parent's value is no greater than either of its children's values (for a min-heap, with the minimum on top). A heap says nothing at all about the order between two children of the same parent.

Slack rule, only the parent no bigger than its children, is deliberately weak: it's cheap to maintain. A BST sometimes has to rebalance whole subtrees after an insertion; a heap only needs to locally "bubble up" or "sink down" the new element, covered just below.

<Callout type="info" title="A heap doesn't sort, it just knows the minimum">
The heap's shape guarantees exactly one fact: the minimum sits at the root. It makes no guarantee whatsoever about the order of the remaining elements; `heap[1]` doesn't have to be the second smallest.
</Callout>

## An array instead of a tree: the same savings as a dynamic array

The tree in the figure is drawn with circles and arrows, but in memory it's stored with no references at all, as an ordinary list. Heap nodes are numbered level by level, left to right, and placed into the array in that order. That turns the parent-child links into index arithmetic: the node at index `i` has children at indices `2*i + 1` and `2*i + 2`, and its parent at index `(i - 1) // 2`.

No node objects and no `left`/`right` references like in a BST, a heap doesn't need them: a plain Python list plus index formulas gives you the same tree for free, with no overhead from wrapper objects.

## heapq: the standard library's minimum heap

In Python a heap lives in the `heapq` module, and it isn't a separate class but a set of functions that maintain the heap property on an ordinary list. `heappush` adds an element while keeping the heap property; `heappop` removes and returns the minimum, also preserving the property. `heap[0]` at any moment gives the minimum without removing it, O(1), because by definition the heap's minimum always sits at the root, meaning index zero of the array.

<PyStepper
  title="heappush bubbles the new element upward (sift up)"
  code={`import heapq

heap = []
for value in [5, 1, 8, 2, 9]:
    heapq.heappush(heap, value)
    print(value, "->", heap)`}
/>

Step through the insertion and notice: the array after every `heappush` isn't fully sorted, but `heap[0]` always stays the minimum of everything inserted so far. The new element is placed at the end of the array and then "sifted" upward (sift up): compared with its parent and swapped until it lands in the right spot or hits the root. The sift path runs along a single branch of the tree, which is why `heappush` costs O(log n), the logarithm of the heap's height.

`heappop` works as a mirror image: it takes the root as the result, moves the last element of the array into its place, and sifts it downward (sift down), swapping with the smaller of its children until the heap property is restored. Also O(log n).

<PyDemo
  title="heappop removes one minimum after another"
  code={`import heapq

heap = [5, 1, 8, 2, 9]
heapq.heapify(heap)
print("after heapify:", heap)

while heap:
    print(heapq.heappop(heap))`}
/>

`heapq.heapify` builds the heap property on an existing list in a single call. Intuition suggests O(n log n), n sifts at O(log n) each, but the actual cost is lower, O(n): the sift-down that `heapify` performs is cheap for nodes near the bottom of the tree, and most nodes have almost nothing to sift. The exact proof is beyond the scope of a junior interview; it's enough to remember the fact itself: `heapify` is cheaper than calling `heappush` n times in a row.

## Where it's used: top-k, scheduling, Dijkstra

- **The k largest or smallest values.** `heapq.nlargest(k, items)` and `heapq.nsmallest(k, items)` solve the top-k problem without fully sorting the whole set. A hand-rolled alternative keeps a min-heap of size k: any new element bigger than the heap's current minimum evicts it, via `heapq.heapreplace`.
- **A priority task queue.** Tasks go into the heap as pairs `(priority, task)`, come out via `heappop`, and what comes out is always the highest-priority task, not the one queued longest ago.
- **Dijkstra's algorithm.** As mentioned in the graphs topic: BFS finds the shortest path only in an unweighted graph, while on a graph with weighted edges a heap plays the same role as the queue, vertices surface in order of accumulated path cost rather than in order of edge count.

## A max-heap via negation: a trick, not a separate structure

`heapq` implements only a min-heap, with the minimum on top. When you need the maximum on top, you insert elements with the sign flipped: `heapq.heappush(heap, -value)`, and flip the sign back on read, `-heap[0]`. The heap still keeps the "minimum" on top, just among negative numbers, and the minimum of a negated number corresponds to the maximum of the original.

```python
import heapq

max_heap = []
for value in [5, 1, 8, 2, 9]:
    heapq.heappush(max_heap, -value)

print(-max_heap[0])
print(-heapq.heappop(max_heap))
```

## Edge cases

- **An empty heap.** `heapq.heappop([])` raises `IndexError`. Code that removes elements in a loop must check `while heap:` rather than expect the heap to announce when it's empty.
- **A heap doesn't store a sorted array.** Only `heap[0]` is guaranteed to be the minimum. `heap[1]` and `heap[2]` don't have to be in ascending order; producing a sorted list is only possible through successive `heappop` calls, and that consumes the heap.
- **Tuples with a matching first element.** If you push `(priority, object)` pairs and two elements happen to share a priority, Python compares the tuples' second elements to break the tie. If the objects aren't comparable, say two dicts, the comparison fails with `TypeError` right inside `heappush`, even though the same code ran fine earlier when priorities differed.
- **heapify mutates the list in place.** The function returns `None`, not a new list. Code like `heap = heapq.heapify(items)` overwrites `heap` with `None`; the right way is to call `heapq.heapify(items)` on its own line and keep working with `items`.

## Common mistakes

- **Forgetting the sign flip when implementing a max-heap.** `heapq.heappush(heap, value)` instead of `heapq.heappush(heap, -value)` silently turns the intended max-heap into a min-heap, so the least urgent element surfaces instead of the most urgent one.
- **Pushing tuples with a non-comparable second field.** `heapq.heappush(heap, (priority, task_object))` works fine until priorities collide, then fails with `TypeError: '<' not supported`. Fix: add a counter to the tuple that increases with every insertion, `(priority, counter, task_object)`; it's always unique and absorbs the comparison before Python ever reaches the object.
- **Expecting sorted output from reading the heap array directly.** `print(heap)` shows a valid heap, but not a sorted list. Sorted order requires either a `heappop` loop or a separate `sorted(...)` call.
- **Calling heapify(items) as `heap = heapify(items)`.** The function mutates the list in place and returns `None`; after that assignment, any further use of `heap` breaks the code with `NoneType` errors.

<Detail summary="Why heapify costs O(n), not O(n log n)">

`heapify` runs a sift-down for every non-leaf node, starting at the bottom level and working up toward the root. In a heap of n elements, roughly half the nodes are leaves: there's nothing to sift in them. Another quarter of the nodes sit one level above and sift down at most one step. The closer a node is to the root, the further it could in theory sift, but there are exponentially fewer such nodes. The sum across all levels converges to a linear quantity, O(n), rather than the n log n you'd get if every node sifted the full height of the tree.

</Detail>
