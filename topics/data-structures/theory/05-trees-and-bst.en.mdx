# Trees and BSTs: a hierarchy where search doesn't have to be linear

A file manager shows folders and folders nested inside them, and no folder ever lives under two parents at once. An HTML page is a tree of tags, where `body` contains `div`, and `div` contains `p`. A company's org chart: every employee has one manager, and a manager has several reports. All three pictures share one shape: a hierarchy where each element has exactly one parent and any number of children, and the hierarchy itself never forms a cycle. That shape is called a tree, and it underlies one of the most common interview questions: how to search sorted data faster than a linear scan, without paying to re-sort on every insertion.

## Mental model: nodes and references to children

A tree is assembled from nodes, just like the linked list from the previous topic. The difference is in the number of outgoing references: a linked-list node has one, `next`; a tree node can have several, one per child. In a binary tree there are exactly two children, distinguished as `left` and `right`. The single node with no parent is called the root, nodes with no children are leaves, and the number of edges on the path from the root to the farthest leaf is the tree's height.

```python
class TreeNode:
    def __init__(self, value, left=None, right=None):
        self.value = value
        self.left = left
        self.right = right

root = TreeNode(5, TreeNode(3), TreeNode(8))
print(root.value)
print(root.left.value)
print(root.right.value)
```

## Binary search tree: order baked into the shape

A plain binary tree guarantees nothing about the order of values. A binary search tree (BST) adds one rule and rests entirely on it: for any node, every value in its left subtree is smaller than the node's value, and every value in its right subtree is greater than or equal to it. The rule is recursive: it holds not just at the root but at every node inside the tree.

We search for 4. Start at the root, 5: the target is smaller, so we go left, to 3. The target is bigger than 3, so we go right, to 4. The values match, search done in two steps, even though the tree has seven nodes. At every step the BST rule discards an entire subtree from consideration, exactly like binary search over a sorted array from the neighboring algorithmic complexity topic.

## Search, insertion, traversal: the same recursion

Search, insertion, and later traversal all lean on the same idea: compare the target value against the current node and decide, left or right. The implementation of insertion fits in a few lines:

<PyStepper
  title="Inserting into a BST: recursive descent by the left/right rule"
  code={`class TreeNode:
    def __init__(self, value, left=None, right=None):
        self.value = value
        self.left = left
        self.right = right

def insert(node, value):
    if node is None:
        return TreeNode(value)
    if value < node.value:
        node.left = insert(node.left, value)
    else:
        node.right = insert(node.right, value)
    return node

root = None
for v in [5, 3, 8, 1, 4]:
    root = insert(root, v)
print(root.value, root.left.value, root.right.value)`}
/>

The base case of the recursion, `node is None`, fires exactly where the fitting subtree runs out: the new node becomes a leaf. Returning `node` at every level reattaches the modified subtree back to its parent, which is why the tree stays connected after every insertion.

The traversal that recovers values in ascending order is called inorder: recursively walk the left subtree, visit the node itself, then recursively walk the right subtree. For a BST this isn't a coincidence, it's a direct consequence of the "left smaller, right bigger" rule applied at every level of the recursion.

<PyDemo
  title="Inorder traversal of a BST always returns sorted order"
  code={`class TreeNode:
    def __init__(self, value, left=None, right=None):
        self.value = value
        self.left = left
        self.right = right

def insert(node, value):
    if node is None:
        return TreeNode(value)
    if value < node.value:
        node.left = insert(node.left, value)
    else:
        node.right = insert(node.right, value)
    return node

def inorder(node, out):
    if node is None:
        return
    inorder(node.left, out)
    out.append(node.value)
    inorder(node.right, out)

root = None
for v in [5, 3, 8, 1, 4, 7, 9]:
    root = insert(root, v)

result = []
inorder(root, result)
print(result)`}
/>

Insert values in any order at all, even `[9, 1, 8, 2, 7]`, and the inorder result always comes out sorted. Insertion order changes the shape of the tree, but not the content of the sorted output.

## Complexity: it all depends on shape

Search, insertion, and deletion in a BST cost O(h), where h is the tree's height: each level discards exactly one subtree, and the number of levels bounds the number of comparisons. The question is what h equals. In a balanced tree, where both sides of every node grow roughly evenly, the height is O(log n): n elements fit into a logarithmic number of levels, exactly like binary search.

<Compare
  leftTitle="Balanced tree"
  rightTitle="Degenerate tree"
  left="Height grows like log n. Inserting numbers in shuffled order usually produces this shape: roughly equal numbers of descendants end up on the left and right of every node."
  right="Height grows like n. Inserting already-sorted data turns a BST into a chain where every node has only a right child, and search degrades to walking a list."
  verdict="A plain BST doesn't guarantee balance by itself. Guaranteeing O(log n) regardless of insertion order is the job of self-balancing trees, AVL and red-black, which restructure part of the tree on every insertion."
/>

Degradation is easy to see: insert `1, 2, 3, 4, 5` into a tree in increasing order, and every new number goes to the right of the previous one, because it's bigger than everything already inserted. What you get isn't a tree but a chain, and searching for 5 walks through all five nodes, O(n), even though it's formally still a BST.

<MarginNote label="self-balancing">AVL trees and red-black trees keep height at O(log n) regardless of insertion order through local restructurings (rotations) after every operation. The implementation is beyond this section's scope; for a junior interview it's enough to know that the guarantee exists and it comes from rotating nodes.</MarginNote>

## When to use a BST, and when a hash table

Both structures find a value faster than a full scan, but they solve different problems. The hash table from the previous topic keeps no order and can't answer "give me all values from 10 to 20" other than by scanning everything. A BST stores order right in its shape, so a range query, the nearest smaller value, or the k-th value are all solved by walking part of the tree rather than the whole dataset.

<Compare

  rightTitle="Balanced BST"

  right="Search, insertion, deletion O(log n), slower than a hash table. Order is built in: inorder traversal, range queries, and value-adjacent elements come almost for free."
  verdict="If you just need fast lookup by key, reach for dict or set. If you need order, ranges, the k-th value, a structure that keeps itself sorted, a BST earns its keep."

## Edge cases

- **An empty tree.** The root is `None`. Search and traversal must survive that from the first step: recursion with the base case `if node is None: return` stops immediately, without reading `.value` on a nonexistent node.
- **Duplicate values.** The BST rule in this section sends values equal to the current node to the right (the `else` branch in `insert`). That's a convention, not a universal law: another codebase might forbid duplicates entirely or send them left; what matters is sticking to one convention throughout the code.
- **A tree with a single node.** It is both root and leaf at once. Search and insertion finish in a single comparison, and the tree's height is zero.
- **Sorted input.** Inserting in ascending or descending order turns a BST into a chain, covered above. In an interview it's worth naming this limitation out loud; it's exactly the setup for the follow-up question about self-balancing.

## Common mistakes

- **Forgetting the recursion's base case.** Without `if node is None: return ...`, traversal or search on an empty subtree crashes with `AttributeError: 'NoneType' object has no attribute 'value'`.
- **Not reassigning the result of the recursive call.** The line `insert(node.left, value)` without `node.left = ...` on the left changes nothing in the tree: the new node gets created and immediately lost, because nobody saved a reference to it.
- **Mixing up `left < node.value` with `left <= node.value` without an explicit duplicate convention.** An inconsistent rule across different parts of the code splits the tree into two different representations of the same dataset, and search starts missing values that are actually there.
- **Expecting O(log n) from an unbalanced tree.** The O(log n) asymptotics hold for the balanced case. With an unlucky insertion order, the real cost of operations is O(n), and that's worth flagging explicitly in any complexity estimate.
