# Stack vs queue: the same list, two different disciplines

The browser's back button always reopens the page you visited last, not the first one. An office printer prints documents in the order they were queued and cannot "jump ahead" of jobs already waiting just because something new arrived. Both systems keep a sequence of items, but the rule for who leaves next is exactly opposite between them. That rule is the entire difference between a stack and a queue: no new data structure is needed under the hood, only discipline about which end you take elements from.

## Stack: last in, first out

A stack hands back elements in the reverse order they were added: the last one in is the first one out. This discipline is called LIFO, last in, first out. A stack of plates is exactly that: you place a plate on top and take one off the top too; you only reach the bottom plate once everything above it is gone.

Python has no separate Stack class, because it does not need one: an ordinary `list` already does everything you need at its tail. `append` places an element on top, `pop()` with no argument removes the top one, `items[-1]` peeks at the top without removing it. All three operations work at the end of the list, where capacity is already reserved, so each one costs O(1) amortized - the mechanics behind that were covered in the previous section on dynamic arrays.

```python
stack = []
stack.append(1)
stack.append(2)
stack.append(3)
print(stack[-1])
print(stack.pop())
print(stack)
```

<PullQuote>A stack is not a new data structure. It's a list where everyone has agreed to touch only one end.</PullQuote>

## The classic problem: balanced brackets

A stack answers the question "what was opened last and hasn't been closed yet," and that is exactly what you need to validate bracket expressions in code, JSON, or math formulas. Every opening bracket goes onto the stack; every closing bracket must pop exactly the bracket it pairs with off the top.

<PyStepper
  title="Bracket validation: push on open, pop on close"
  code={`def is_valid(s):
    pairs = {")": "(", "]": "[", "}": "{"}
    stack = []
    for ch in s:
        if ch in "([{":
            stack.append(ch)
        elif ch in ")]}":
            if not stack or stack.pop() != pairs.get(ch, ch):
                return False
    return not stack

print(is_valid("([{}])"))`}
/>

Step through the trace and notice the two ways the check can fail: a closing bracket shows up while the stack is empty (nothing to close), or it pops a character off the top that is not its pair. Both cases return `False` immediately, without waiting for the end of the string.

## Queue: first in, first out

A queue follows the opposite rule: the first one in is the first one out, FIFO, first in, first out. It works like a line at a store: a new customer joins at the back, and the person at the front gets served.

Here's where the trap begins. A Python list is cheap only at its tail. `append` at the end costs O(1), but `pop(0)`, removing from the front, costs O(n): every call shifts all remaining pointers one cell to the left, exactly like `insert(0, x)` from the dynamic array section. On small queues this goes unnoticed, but a task queue that grows to thousands of items turns `list.pop(0)` into a hidden O(n²).

<Callout type="warning" title="list as a queue: a performance trap">
`queue.append(x)` is cheap, but `queue.pop(0)` is expensive. Working with both ends of a sequence calls for a structure where both ends are cheap, not just the tail.
</Callout>

## collections.deque: a queue with no compromises

The `collections` module provides `deque` (double-ended queue, pronounced "deck"), a structure where all four edge operations run O(1) amortized: `append`, `appendleft`, `pop`, `popleft`. Internally a `deque` is built as a doubly linked list of fixed-size blocks rather than one contiguous array: adding at either end just writes into that end's own block and never touches the other. The price `deque` pays for this is indexed access: `dq[500000]` walks the chain of blocks and costs O(n), while the same access on a `list` is O(1).

```python
from collections import deque

queue = deque()
queue.append("a")
queue.append("b")
queue.append("c")
print(queue.popleft())
print(queue.popleft())
print(queue)
```

## deque(maxlen=...): a ring buffer for free

`deque` takes a `maxlen` parameter. Once set, the structure becomes a fixed-size buffer: adding an element past the limit automatically evicts one from the opposite end. That's exactly what you need for "the last N log lines," "recent action history," or a sliding window of metrics.

<PyDemo
  title="deque(maxlen=3): a new element evicts the old one"
  code={`from collections import deque

history = deque(maxlen=3)
for page in ["main", "cart", "checkout", "payment", "done"]:
    history.append(page)
    print(list(history))`}
/>

Change `maxlen` to 5 and watch how quickly the buffer stops evicting pages: while the element count is below the limit, `deque` behaves like an ordinary queue with no losses.

## Three everyday uses of a stack

A stack is rarely called by name, but almost every program uses one:

- **The call stack.** Every function call pushes a frame holding its local variables and return address. When the function finishes, the frame is popped and control returns to wherever the top frame pointed. Recursion runs on exactly this mechanism, and CPython's `RecursionError` is nothing but an overflow of the call stack.
- **Undo.** An editor pushes every change onto a stack. `Ctrl+Z` pops the top one and rolls it back; `Ctrl+Y` can push the undone action onto a separate redo stack.
- **Depth-first search (DFS).** In the graphs topic, DFS ran through recursion or an explicit stack: pop the top vertex, process it, push its unvisited neighbors. Swap `queue.popleft()` for `stack.pop()` in BFS code, and a wave-by-wave traversal turns into a dive-deep one.

## Edge cases

- **pop from an empty structure.** Both `list.pop()` and `deque.popleft()` raise `IndexError` on an empty collection. Code that removes elements in a loop must explicitly check `if stack:` or `while queue:` rather than rely on the exception as a normal stopping condition.
- **queue.pop(0) works silently but slowly.** The mistake raises no exception and doesn't break the result; the program simply turns quadratic at scale. A profiler will show bloated time in an innocent-looking line.
- **deque(maxlen=n) doesn't warn on overflow.** The evicted element vanishes without a trace. If you need to keep old data before the deque evicts it, copy it out separately before the `append`.

## Common mistakes

- **Using `list.pop(0)` for a queue.** Every call moves the entire rest of the list. Fix: `collections.deque` and `popleft()`.
- **Mixing up `stack.pop()` and `queue.popleft()`.** One method decides which end of the sequence you're reading, and with it the whole logic of the algorithm. Before writing the code, spell out explicitly: does this problem need LIFO or FIFO?
- **Checking "stack is empty" with `len(stack) == 0` instead of `not stack`.** Both work the same, but the second form is the accepted Python idiom and reads faster.
- **Forgetting that `deque` doesn't give O(1) indexed access.** Code like `dq[len(dq) // 2]` in a hot loop quietly plants an O(n) where an O(1) was expected.

<Detail summary="How deque is built internally: blocks, not one array">

CPython implements `deque` as a doubly linked list of fixed-size blocks (64 pointers per block in current versions). Adding at either edge almost always writes into a block already allocated at that edge, only rarely allocating a new block and linking it into the chain, which is why `append` and `appendleft` average O(1). There is no shared contiguous memory between blocks, so `dq[i]` has to work out which block holds the target index and walk the chain, O(n) in the worst case. The trade-off is deliberate: `deque` sacrifices indexed access for equally cheap operations at both ends.

</Detail>
