# Связный список: структура, которую пишут на собеседованиях, а не в продакшене

Вставьте элемент в начало питоньего списка из миллиона элементов, и интерпретатор молча сдвинет весь миллион на одну позицию вправо. Каждый такой `insert(0, x)` стоит O(n), и очередь, собранная на `list`, начинает тормозить ровно из-за этого: снять первый элемент так же дорого, как переписать все остальные. При этом существует структура, где вставка в начало обходится в пару присваиваний, хоть элементов миллион, хоть миллиард, и никто никуда не сдвигается.

Парадокс в том, что за годы коммерческой разработки на Python вы, скорее всего, ни разу не напишете эту структуру руками: в прикладном коде её ниши давно заняли `list` и `collections.deque`. А на собеседовании связный список всплывёт почти наверняка, и задача «разверните список» входит в любой шорт-лист вопросов. Разберёмся, как он устроен, что у него дёшево, что дорого и почему интервьюеры не готовы с ним расстаться.

## Мысленная модель: цепочка узлов

Связный список собирают из узлов. Узел хранит два поля: полезное значение и ссылку `next` на следующий узел. У последнего узла в `next` лежит `None`, это знак конца цепочки. Снаружи структуру держит единственная переменная, голова списка (`head`), указывающая на первый узел. Потеряете голову, и сборщик мусора утащит всю цепочку.

Ключевое отличие от массива спрятано в памяти. Массив занимает непрерывный блок, поэтому адрес элемента с индексом i вычисляется арифметикой, и `a[500000]` читается за один шаг. Узлы связного списка разбросаны по куче как попало, и вычислить адрес десятого узла нельзя в принципе: единственный маршрут к нему ведёт от головы по девяти ссылкам. Отсюда два зеркальных следствия:

- доступ по индексу стоит O(n): к i-му узлу идут i шагов по ссылкам;
- вставка и удаление рядом с известным узлом стоят O(1): переписывается пара ссылок, хвост не сдвигается.

Узел на Python занимает три строки, и класс LinkedList не обязателен: головы и договорённости про `next` уже достаточно, чтобы структура жила.

```python
class Node:
    def __init__(self, value, next=None):
        self.value = value
        self.next = next

head = Node(17, Node(42, Node(8)))
print(head.value)
print(head.next.next.value)
print(head.next.next.next is None)
```

Выражение `head.next.next.value` читается как маршрут: от головы два прыжка по ссылкам, затем значение. Так выглядит вся работа со связным списком: не индексы, а прыжки.

## Обход: единственный маршрут

Стандартный обход укладывается в три идеи: завести бегунок `node = head`, крутить цикл, пока `node is not None`, и в конце каждой итерации прыгать дальше через `node = node.next`. Никакого `range(len(...))`: длину списка без обхода даже не узнать. Пройдите трассировку по шагам и последите, как бегунок перескакивает с узла на узел, а сумма накапливается.

<PyStepper title="Обход списка: node прыгает по ссылкам next" code={`class Node:
    def __init__(self, value, next=None):
        self.value = value
        self.next = next

head = Node(17, Node(42, Node(8)))
node = head
total = 0
while node is not None:
    total += node.value
    node = node.next
print(total)`} />

Обход стоит O(n), и это честная цена любого вопроса к списку: длина, поиск, сумма, печать. Тут же прячется неудобная правда: массив за те же O(n) обходится ощутимо быстрее, потому что его элементы лежат в памяти подряд, а узлы списка заставляют процессор прыгать по куче мимо кеша. Асимптотика одинаковая, константа различается в разы.

## Вставка за O(1) и цена этого фокуса

Пусть в руках есть ссылка на узел `spot`, и сразу после него нужен новый узел. Хватит двух присваиваний:

```python
new = Node(99)
new.next = spot.next
spot.next = new
```

Порядок принципиален. Сначала новичок цепляется за продолжение цепочки, и только потом `spot` перенацеливается на него. Выполните присваивания наоборот, и хвост утечёт: `spot.next` уже перезаписан, а старую ссылку никто не сохранил. Удаление симметрично: `spot.next = spot.next.next` выкидывает следующий узел из цепочки, а память приберёт сборщик мусора.

<Callout type="warning">
O(1) действует, только когда узел уже в руках. Если место вставки сначала ищут обходом, операция целиком стоит O(n). Дешёвые вставки окупаются там, где ссылки на узлы хранятся снаружи: именно так делает functools.lru_cache, о нём ниже.
</Callout>

## Разворот: задача, по которой калибруют кандидатов

Классика жанра: развернуть односвязный список за один проход и без дополнительной памяти. Идея держится на трёх указателях. `prev` смотрит на уже развёрнутую часть, `node` на текущий узел, а `nxt` спасает продолжение до того, как ссылка `next` будет перекинута назад. Запустите код и посмотрите на список до и после разворота. Затем поменяйте местами строки `nxt = node.next` и `node.next = prev` внутри цикла и посмотрите, что останется от списка: хвост теряется ровно из-за порядка присваиваний.

<PyDemo title="Разворот списка тремя указателями" code={`class Node:
    def __init__(self, value, next=None):
        self.value = value
        self.next = next

def build(values):
    head = None
    for value in reversed(values):
        head = Node(value, head)
    return head

def show(head):
    parts = []
    node = head
    while node is not None:
        parts.append(str(node.value))
        node = node.next
    print(" -> ".join(parts) if parts else "пусто")

def reverse(head):
    prev = None
    node = head
    while node is not None:
        nxt = node.next
        node.next = prev
        prev = node
        node = nxt
    return prev

head = build([1, 2, 3, 4])
show(head)
show(reverse(head))`} />

Обратите внимание на возврат. После цикла `node` дошёл до `None`, а `prev` остановился на бывшем хвосте: он и стал новой головой, поэтому функция возвращает `prev`. Вернуть `head` было бы ошибкой: после разворота старая голова превратилась в хвост, и обход от неё покажет один элемент.

Почему эту задачу так любят интервьюеры. Она за пять минут показывает, умеет ли человек обращаться со ссылками: удержать инвариант цикла, не потерять хвост, пережить пустой список, вернуть правильную голову. Ни фреймворки, ни знание библиотек здесь не спасают. Тот же навык нужен в реальной работе при отладке графов объектов, круговых ссылок и структур вида «родитель и дети».

## Односвязный, двусвязный и где это всё в Python

У двусвязного списка узел хранит две ссылки, `next` и `prev`. Это даёт обход в обе стороны и главный бонус: узел можно удалить, держа ссылку только на него самого, без поиска предыдущего. Расплата скромная, но постоянная: лишняя ссылка в каждом узле и вдвое больше возни при перестановках.

В стандартной библиотеке связные списки живут не там, где их ищут новички:

- `collections.deque` устроена как двусвязный список блоков по 64 элемента: компромисс между дешёвыми операциями на обоих концах и дружбой с кешем процессора;
- `functools.lru_cache` держит кольцевой двусвязный список, чтобы за O(1) переносить только что использованную запись в конец и выгонять самую старую;
- цепочки в хеш-таблицах собираются из таких же узлов со ссылками: учебные реализации словаря раскладывают столкнувшиеся ключи именно в связные списки.

<MarginNote label="deque">Блочное устройство объясняет, почему deque не даёт O(1) доступа по индексу: внутри всё равно прыжки по блокам.</MarginNote>

А вот руками связный список в прикладном питоньем коде почти не пишут, и об этом стоит говорить на собеседовании прямо. Каждый узел обходится в полноценный объект с заголовком в десятки байт, ссылки убивают кеш-локальность, и `list` с `deque` выигрывают почти в любом сценарии. Спрашивают его не как рабочий инструмент, а как полигон: на нём видно, как кандидат думает о ссылках и порядке операций.

## Краевые случаи

- **Пустой список.** Голова равна `None`, и корректный обход обязан пережить это с первого шага: цикл `while node is not None` просто не запустится. Разворот пустого списка возвращает `None`, и это правильный ответ, а не ошибка.
- **Один узел.** Разворот ничего не меняет, но код обязан вернуть тот же узел, а не упасть при чтении `next`.
- **Цикл в списке.** Самый коварный случай: если чья-то ссылка `next` замкнулась на уже пройденный узел, обход не упадёт с исключением, а молча зависнет навсегда, ведь условие `node is not None` больше никогда не сработает. Программа не сообщает об ошибке, она просто перестаёт отвечать. Проверка списка на цикл выросла в отдельную классическую задачу, о ней в Detail ниже.

## Частые ошибки

- **Потерять хвост при вставке.** `spot.next = new` выполняется до того, как `new.next` получил старое продолжение. Исправление: сначала `new.next = spot.next`, потом `spot.next = new`.
- **Забыть шаг `node = node.next`.** Цикл `while node is not None` без шага вечно крутится на первом узле. Исправление: шаг ставится последней строкой тела цикла, и глаз должен искать его там автоматически.
- **Вернуть старую голову после разворота.** `head` теперь указывает на хвост, обход от него покажет один элемент. Исправление: возвращать `prev`.
- **Сравнивать узел вместо значения.** Условие `if node == target` сравнивает объект-узел с числом и молча даёт `False`. Исправление: сравнивайте `node.value == target`.
