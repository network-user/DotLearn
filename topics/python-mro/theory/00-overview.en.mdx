You added another mixin to the project, overrode a method, ran the code - and `super()` suddenly called a class you did not expect. Or Python refused to create the class and raised a `TypeError` about an impossible inheritance order. With single inheritance there are no such surprises: one parent, an obvious chain. But as soon as a class gets a second base, the simple rule "go up to the parent" stops working, and behavior starts depending on the order in which you listed the bases. One mechanism stands behind all of these cases - the order in which Python searches for attributes and methods. Understand it, and multiple inheritance stops being a lottery.

**What you will learn to do.** Read `__mro__` for any class and predict which method wins when names conflict; derive the resolution order on paper faster than you can print it; apply `super()` correctly in cooperative chains and pass arguments through them; design mixins and diamond hierarchies so they compose predictably instead of failing with an error.

**Topic route.** Four sections, each with a sandbox where you can edit and run code right in the browser:

1. **What MRO is** - how Python flattens the ancestor graph into one flat list and searches attributes by walking it left to right.
2. **C3 linearization** - the algorithm that builds that list, and how to derive the order by hand with the merge operation.
3. **`super()`** - why it is not "the parent" but the next class in MRO, and how that lets classes cooperatively share one operation.
4. **Diamond and mixins** - diamond inheritance without double calls, and why the order of base classes in the declaration is part of the contract.

**What you need beforehand.** Nothing to install: Python runs in the browser. You should be comfortable with ordinary inheritance and classes from the "OOP in Python" topic - everything about multiple inheritance and MRO we will cover here step by step.
