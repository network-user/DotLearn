# MRO: список, по которому Python ищет атрибут

Вы пишете `obj.method()`, и где-то находится нужное определение - но где именно? При одиночном наследовании ответ очевиден: поднимаемся от класса экземпляра к родителю, потом к его родителю. Стоит появиться второму базовому классу, и «вверх по цепочке» перестаёт быть однозначным маршрутом: куда идти из класса с двумя родителями - в левого, в правого, в их общего предка? Python отвечает на этот вопрос заранее, один раз на класс, и записывает ответ в виде плоского упорядоченного списка - method resolution order, MRO.

К концу раздела Вы научитесь читать `__mro__` любого класса, предсказывать по нему, чей атрибут выиграет при конфликте имён, и понимать, почему поиск атрибута - это не обход дерева наследования, а проход по заранее вычисленному списку.

## Ментальная модель: один список вместо дерева

Иерархия классов в голове рисуется деревом (или графом) со стрелками к родителям. Но Python для поиска атрибутов деревом не пользуется. В момент создания класса он сворачивает весь граф предков в **один линейный список** и кладёт его в атрибут `__mro__`. Любое обращение `obj.attr` - это проход по этому списку слева направо до первого класса, в `__dict__` которого есть `attr`.

```python
class Animal:
    def speak(self):
        return "звук"

class Dog(Animal):
    def speak(self):
        return "гав"

print([c.__name__ for c in Dog.__mro__])
```

Печатается `['Dog', 'Animal', 'object']`. Это и есть маршрут поиска: сначала `Dog`, потом `Animal`, в самом конце - `object`, общий предок всех классов. Когда Вы вызываете `Dog().speak()`, Python идёт по этому списку и останавливается на `Dog`, потому что `speak` нашёлся уже там.

Ключевая мысль: список фиксируется один раз, когда класс определяется. Поиск во время выполнения не «думает», куда свернуть, - он просто перебирает готовый список. Вся сложность спрятана в том, *как* этот список строится; ей посвящён следующий раздел.

<MarginNote label="__mro__ и mro()">`Dog.__mro__` - это кортеж, доступный только для чтения. Метод `Dog.mro()` возвращает тот же порядок списком; его можно переопределить в метаклассе, но в обычном коде это почти никогда не нужно.</MarginNote>

## Два способа посмотреть на порядок

У MRO два интерфейса: атрибут `__mro__` (кортеж) и метод `mro()` (список). Содержимое одинаковое.

```python
class A: pass
class B(A): pass
class C(B): pass

print(C.__mro__ == tuple(C.mro()))
print([c.__name__ for c in C.mro()])
```

Печатается `True` и `['C', 'B', 'A', 'object']`. Линейная цепочка `C → B → A → object` - вырожденный случай, где MRO совпадает с тем самым «вверх по родителям». Интересное начинается, когда родителей несколько.

## Конфликт имён: список решает спор

Когда один и тот же метод определён в нескольких предках, выигрывает тот, кто стоит в `__mro__` раньше. Никакой магии «базовый класс главнее» нет - есть только позиция в списке.

```python
class Base:
    def role(self):
        return "base"

class Mixin:
    def role(self):
        return "mixin"

class Widget(Mixin, Base):
    pass

print(Widget().role())
print([c.__name__ for c in Widget.__mro__])
```

Печатается `mixin`, потому что MRO - это `['Widget', 'Mixin', 'Base', 'object']`: `Mixin` стоит левее `Base`, и его `role` находится первым. Поменяйте порядок на `Widget(Base, Mixin)` - и победит уже `Base`. Порядок базовых классов в скобках напрямую управляет тем, чьё поведение перекроет чьё.

<Callout type="tip">
Когда не уверены, чей метод вызовется в множественном наследовании, не гадайте по дереву - распечатайте `Cls.__mro__` и прочитайте список слева направо. Первое совпадение и есть ответ.
</Callout>

## Потрогайте список руками

Запустите код как есть: при `Widget(Mixin, Base)` метод `role` берётся из `Mixin`. Теперь поменяйте порядок родителей местами - `class Widget(Base, Mixin)` - и запустите снова. Вывод `role()` сменится, и `__mro__` тоже: следите, как перестановка двух имён в скобках переставляет два класса в списке.

## Поиск останавливается на первом совпадении

MRO определяет не только победителя при конфликте, но и факт остановки. Как только класс с нужным атрибутом найден, оставшаяся часть списка не просматривается вовсе.

```python
class A:
    x = "из A"

class B:
    x = "из B"
    y = "только в B"

class C(A, B):
    pass

print(C.x, C.y)
```

Печатается `из A только в B`. Атрибут `x` есть и в `A`, и в `B`, но `A` стоит в MRO раньше - берётся его версия. Атрибут `y` есть только в `B`, поэтому поиск проходит мимо `A` и доходит до `B`. Один и тот же список обслуживает оба обращения, просто останавливается в разных точках.

## Сюрприз: object всегда последний, и это не случайно

Может показаться, что `object` в конце списка - формальность. Но именно его положение в самом конце делает MRO согласованным: любой ваш метод, определённый где угодно выше, гарантированно перекроет реализацию по умолчанию из `object`. Если бы `object` мог оказаться в середине, его `__init__` или `__eq__` иногда выигрывали бы у ваших - и поведение зависело бы от формы иерархии.

```python
class A:
    def __repr__(self):
        return "A-объект"

class B(A):
    pass

print(repr(B()))
print(B.__mro__[-1].__name__)
```

Печатается `A-объект` и `object`. `__repr__` из `object` существует всегда, но стоит последним, поэтому ваша версия из `A` находится раньше. Это инвариант, который C3 обязан сохранять для любой допустимой иерархии, - и причина, по которой некоторые иерархии Python вообще откажется строить.

## Типичные ошибки

- **Считать, что «базовый класс важнее производного».** В MRO всё ровно наоборот: производный класс всегда стоит раньше своих предков, поэтому переопределение работает. Важна позиция в списке, а не «уровень» в дереве.
- **Гадать порядок при множественном наследовании по картинке дерева.** Дерево не показывает, как C3 переставит классы. Единственный надёжный способ - прочитать `Cls.__mro__`.
- **Путать `type(obj)` и MRO.** `type(obj)` - это только класс экземпляра, первый элемент списка. Поиск атрибута видит весь `__mro__`, а не один класс.

<Detail summary="Откуда у MRO ноги растут: классы старого и нового стиля">

В Python 2 сосуществовали классы «старого стиля» (не наследующие `object`) и «нового стиля» (наследующие `object` явно). У старого стиля MRO строился наивным обходом в глубину слева направо и в ромбовидных иерархиях давал неинтуитивный порядок, ломавший переопределения. Новый стиль принёс C3-линеаризацию.

В Python 3 классов старого стиля больше нет: каждый класс неявно наследует `object` и получает корректный C3-MRO. Поэтому всё, что Вы читаете здесь, относится к любому современному коду без оговорок. Подробности перехода - в [официальном разборе MRO](https://docs.python.org/3/howto/mro.html).

</Detail>
