# Ромб, миксины и почему порядок баз решает всё

Ромбовидное наследование - та самая конфигурация, ради которой вообще придумали C3. Четыре класса складываются в фигуру ромба: общий предок наверху, два независимых класса по бокам, и потомок, наследующий обоим. Наивные языки на такой фигуре спотыкаются - либо вызывают общего предка дважды, либо вынуждают программиста вручную разрешать конфликт. Python проходит ромб ровно один раз и в предсказуемом порядке - но только если Вы понимаете, как MRO собирает классы и почему порядок базовых классов в объявлении не косметика, а часть контракта.

К концу раздела Вы научитесь читать ромб по его MRO, проектировать миксины так, чтобы они складывались без сюрпризов, и распознавать ситуацию, когда Python отказывается строить MRO и бросает `TypeError`.

## Ромб: один предок, два пути, один проход

Классический ромб выглядит так: `Base` наверху, `Left` и `Right` наследуют `Base`, а `Both` наследует обоим.

```python
class Base:
    def __init__(self):
        print("base")

class Left(Base):
    def __init__(self):
        print("left")
        super().__init__()

class Right(Base):
    def __init__(self):
        print("right")
        super().__init__()

class Both(Left, Right):
    def __init__(self):
        print("both")
        super().__init__()

Both()
print([c.__name__ for c in Both.__mro__])
```

MRO у `Both` - это `[Both, Left, Right, Base, object]`. Поскольку каждый класс зовёт `super().__init__()`, вызов `Both()` проходит по списку ровно по разу: печатается `both`, `left`, `right`, `base`. Обратите внимание на главное: `Base` выполнился **один раз**, хотя к нему ведут два пути - через `Left` и через `Right`. Это и есть то, что ломается в наивных схемах: там `Base.__init__` выполнился бы дважды.

Почему `Base` не задвоился? Инвариант C3: предок всегда стоит в MRO **позже** всех своих потомков. У `Base` два потомка - `Left` и `Right`, - поэтому он откладывается до тех пор, пока оба не встанут в список. Когда поток `super()` доходит до `Base`, он там уже последний содержательный класс перед `object`, и пройти через него можно только один раз.

## Почему наивный обход ломает ромб

Если бы Python обходил иерархию простым поиском в глубину слева направо, MRO `Both` получился бы `[Both, Left, Base, Right, object]` - `Base` встал бы сразу после `Left`, **до** `Right`. Последствия два, и оба плохие.

Во-первых, `Base` оказался бы раньше `Right`, своего потомка, - нарушение порядка «потомок перед предком». Во-вторых, кооперативная цепочка оборвалась бы: `super()` из `Left` ушёл бы прямо в `Base`, и `Right` вообще не выполнился бы. C3 ставит `Base` после обоих, и эстафета проходит через `Right`. Подробный разбор алгоритма был в разделе про C3-линеаризацию - здесь важно следствие: предсказуемый ромб возможен только с правильной линеаризацией.

## Миксины: маленькие классы, добавляющие одну способность

Миксин - это класс, не предназначенный для самостоятельного использования. Он добавляет одну узкую способность классу, к которому подмешан, и опирается на кооперативный `super()`, чтобы встроиться в чужую цепочку методов. Миксины - главная практическая причина, по которой в Python вообще используют множественное наследование.

```python
class LoggingMixin:
    def save(self):
        print(f"saving {self.__class__.__name__}")
        return super().save()

class TimestampMixin:
    def save(self):
        self.updated = True
        return super().save()

class Model:
    def save(self):
        return "stored"

class User(LoggingMixin, TimestampMixin, Model):
    pass

u = User()
print(u.save(), u.updated)
```

MRO `User` - `[User, LoggingMixin, TimestampMixin, Model, object]`. Вызов `u.save()` проходит логирование, потом проставление таймстампа, потом реальное сохранение в `Model`. Печатается `stored True` (и строка лога). Каждый миксин сделал свою узкую часть и передал эстафету дальше через `super()`.

<Callout type="tip" title="Правило порядка миксинов">
Миксины ставят **слева** от базового класса: `class User(Mixin1, Mixin2, Base)`. Тогда в MRO они окажутся раньше базы и смогут перехватить вызов, добавить своё поведение и через super() пробросить его в базу. Поставите миксин справа от базы - база перехватит вызов первой, и миксин не сработает.
</Callout>

## Потрогайте порядок руками

Запустите ромб. Так видно, что порядок баз - это не стиль, а то, кто кого перехватывает.

## Когда Python отказывается строить MRO

Не всякая комбинация баз линеаризуема. Если требования двух базовых классов к порядку противоречат друг другу, C3 не найдёт ни одного «хорошего» кандидата и Python бросит `TypeError` прямо в момент объявления класса - не при вызове, а при создании.

```python
class A:
    pass

class B(A):
    pass

class X(A, B):
    pass
```

Эта программа падает с `TypeError: Cannot create a consistent method resolution order (MRO) for bases A, B`. Причина: `B` наследует `A`, поэтому `B` обязан стоять в MRO **раньше** `A` (потомок перед предком). Но в объявлении `X(A, B)` Вы перечислили `A` раньше `B`, требуя обратного порядка. Два требования несовместимы, линеаризации нет. Лечится перестановкой: `class X(B, A)` корректно даёт `[X, B, A, object]`.

<Callout type="warning">
Ошибка MRO возникает в момент `class ...:`, а не при использовании. Если код падает «на импорте» с упоминанием consistent method resolution order - ищите базовый класс, перечисленный раньше своего же потомка.
</Callout>

## Типичные ошибки

- **Ставить миксин справа от базового класса.** Тогда база окажется в MRO раньше миксина, перехватит вызов и (если не зовёт super) оборвёт цепочку - миксин не выполнится. Миксины идут слева.
- **Ждать, что общий предок ромба выполнится по разу «сам собой».** Это верно, только если все промежуточные классы зовут `super()`. Жёсткий вызов `Base.__init__(self)` в обход MRO в ромбе приводит к двойному выполнению `Base`.
- **Пытаться «исправить» TypeError про MRO повтором класса в базах.** Перечислять класс и его потомка вместе в любом порядке, нарушающем «потомок раньше предка», бесполезно - правьте порядок, а не дублируйте.
- **Делать миксин зависимым от конкретной базы.** Хороший миксин не знает, к кому подмешан, и общается с остальной цепочкой только через `super()` и документированный контракт метода.

<Detail summary="Как проверить линеаризуемость, не запуская весь модуль">

Если сомневаетесь, что комбинация баз корректна, не обязательно ждать импорта. Создайте класс динамически через `type("X", (A, B), {})` в защищённом `try/except TypeError` - так Вы локализуете именно проблему MRO, не таща за собой остальной модуль. А для готового класса всегда доступен `Cls.__mro__` (кортеж) или `Cls.mro()` (список) - это и есть итоговый порядок, по которому пойдёт любой `super()`.

</Detail>
