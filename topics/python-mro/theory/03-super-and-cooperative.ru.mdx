# super() - это не «родитель», а следующий по MRO

Почти каждый, кто учил наследование, носит в голове ложное определение: «`super()` вызывает метод родительского класса». Для одиночного наследования эта ложь безвредна - там родитель и есть следующий по цепочке. Но стоит появиться второму базовому классу, как `super()` начинает прыгать в классы, которые формально вообще не родители вызывающего, - и интуиция «super - это родитель» приводит к багам, которые невозможно объяснить, не зная правды.

Правда такая: `super()` возвращает прокси, который продолжает поиск со **следующего класса в MRO того объекта, на котором вызван метод**, - а не в статической иерархии класса, где написан `super()`. К концу раздела Вы поймёте, как это позволяет нескольким классам кооперативно поделить одну операцию, как правильно пробрасывать `**kwargs` по такой цепочке и какие ошибки совершают, путая super с родителем.

## Ментальная модель: super - это «следующий», а не «вверх»

Когда внутри метода класса `X` Вы пишете `super().method()`, Python делает следующее: берёт MRO объекта `self`, находит в нём `X`, и продолжает поиск `method` начиная с класса, идущего **сразу после** `X`. Слово «после» относится к списку MRO конкретного объекта - а он зависит от настоящего класса `self`, а не от `X`.

```python
class A:
    def greet(self):
        return "A"

class B(A):
    def greet(self):
        return "B->" + super().greet()

class C(A):
    def greet(self):
        return "C->" + super().greet()

class D(B, C):
    def greet(self):
        return "D->" + super().greet()

print(D().greet())
print([c.__name__ for c in D.__mro__])
```

MRO у `D` - это `[D, B, C, A, object]`. Вызов `D().greet()` идёт по нему: `D` зовёт `super()` → `B`, `B` зовёт `super()` → **`C`** (а не `A`, хотя `C` не родитель `B`!), `C` зовёт `super()` → `A`. Печатается `D->B->C->A` и `['D', 'B', 'C', 'A', 'object']`. Вот он, главный сюрприз: `super()` внутри `B` ушёл в `C`, потому что в MRO объекта `D` именно `C` стоит после `B`.

## Кооперативное наследование: каждый делает свою часть

Эта механика - не курьёз, а инструмент. Если *каждый* класс в иерархии делает свою часть работы и затем зовёт `super()`, то один вызов на листе иерархии пройдёт через все классы ровно по одному разу, в порядке MRO. Это называется **кооперативным множественным наследованием**: классы не знают друг о друге, но честно передают эстафету дальше.

```python
class Save:
    def save(self):
        return ["base"]

class Timestamped(Save):
    def save(self):
        return ["timestamp"] + super().save()

class Audited(Save):
    def save(self):
        return ["audit"] + super().save()

class Record(Timestamped, Audited):
    pass

print(Record().save())
```

MRO у `Record` - `[Record, Timestamped, Audited, Save, object]`. `Record` не переопределяет `save`, поэтому стартует версия `Timestamped`: она добавляет `"timestamp"` и зовёт `super().save()`, которая по MRO попадает в `Audited` (не в `Save`!), та добавляет `"audit"` и зовёт `super().save()`, попадая уже в `Save`. Печатается `['timestamp', 'audit', 'base']`. Три независимых класса собрали результат вместе, и ни один не знал заранее, кто стоит после него.

<PullQuote>Кооперативная иерархия работает, только если каждое звено зовёт super(). Один пропущенный вызов обрывает эстафету.</PullQuote>

## Проброс **kwargs: общий контракт конструкторов

Кооперативность особенно важна в `__init__`, потому что разные классы ждут разные аргументы. Приём такой: каждый класс забирает свои именованные аргументы из `**kwargs`, а остальное передаёт дальше через `super().__init__(**kwargs)`. Последним стоит `object.__init__`, который не ждёт аргументов, - к нему `kwargs` должен прийти пустым.

```python
class Base:
    def __init__(self, **kwargs):
        super().__init__(**kwargs)

class Named(Base):
    def __init__(self, name, **kwargs):
        super().__init__(**kwargs)
        self.name = name

class Aged(Base):
    def __init__(self, age, **kwargs):
        super().__init__(**kwargs)
        self.age = age

class Person(Named, Aged):
    pass

p = Person(name="Ana", age=30)
print(p.name, p.age)
```

Печатается `Ana 30`. `Person(name="Ana", age=30)` запускает `Named.__init__`: он забирает `name`, отдаёт `{'age': 30}` дальше; `Aged.__init__` забирает `age`, отдаёт `{}` дальше; `Base.__init__` получает пустой `kwargs` и зовёт `object.__init__()`. Каждый класс взял ровно своё, и ничей аргумент не потерялся.

## Потрогайте эстафету руками

Запустите кооперативную иерархию. Затем уберите `super().save()` из `Audited` (пусть она возвращает просто `["audit"]`) и запустите снова: эстафета оборвётся на `Audited`, и `Save.save()` уже не выполнится - результат станет короче. Так наглядно видно, что `super()` - это не вызов конкретного класса, а передача эстафеты следующему по MRO.

## Сюрприз: super(B, self) ходит по MRO self, а не B

Двухаргументная форма `super(B, self)` явно говорит: «продолжи поиск после `B` в MRO объекта `self`». Слово «после» относится к MRO *настоящего* класса `self`, а не к иерархии `B`. Если `self` - экземпляр `D(B, C)`, то после `B` в его MRO стоит `C`, и `super(B, self).m()` уйдёт именно в `C`.

```python
class A:
    def m(self):
        return "A"

class B(A):
    def m(self):
        return "B"

class C(A):
    def m(self):
        return "C"

class D(B, C):
    pass

d = D()
print(super(B, d).m())
```

Печатается `C`. Хотя `B` напрямую наследует только `A`, `super(B, d)` смотрит в MRO объекта `d` - `[D, B, C, A, object]` - и берёт класс после `B`, то есть `C`. Это финальное доказательство, что super завязан на MRO объекта, а не на статических родителях класса, где он написан.

## Типичные ошибки

- **Думать «super - это родитель» и удивляться чужому классу.** В множественном наследовании super идёт к следующему по MRO, и это часто «sibling»-класс, а не предок. Лекарство - всегда сверяться с `type(self).__mro__`, а не с `__bases__` класса.
- **Пропустить `super()` в середине кооперативной цепочки.** Один класс не передал эстафету - и все классы после него по MRO молча не выполнились. Симптом: часть инициализации не произошла, ошибки нет. Правило: в кооперативной иерархии каждый метод зовёт super.
- **Жёстко вписать имя родителя: `Base.__init__(self, ...)`.** Это вызывает конкретный класс в обход MRO и в ромбе приводит к двойному вызову общего предка либо к пропуску sibling-классов. Используйте `super()`, а не имя класса.
- **Не принимать и не пробрасывать `**kwargs` в кооперативном `__init__`.** Если класс ждёт фиксированные аргументы и не пробрасывает остаток, чужие именованные аргументы упрутся в него с `TypeError`. Принимайте `**kwargs` и передавайте дальше.

<Detail summary="Почему super() без аргументов вообще работает">

Пустой `super()` - синтаксический сахар Python 3. Компилятор замечает обращение к `super` внутри метода и неявно прокидывает в него два скрытых значения: класс, в теле которого написан метод (через ячейку `__class__`), и первый позиционный аргумент (`self`). Поэтому `super()` эквивалентен `super(__class__, self)`, но не ломается при переименовании класса и не повторяет имя дважды. Именно из-за ячейки `__class__` `super()` нельзя вызвать вне тела метода - там нет нужного контекста. Детали - в [документации super()](https://docs.python.org/3/library/functions.html#super).

</Detail>
