# C3: как из графа предков получается один список

В прошлом разделе MRO появился как готовый список, который Python кладёт в `__mro__`. Но откуда берётся именно этот порядок? Почему `Widget(Mixin, Base)` даёт `[Widget, Mixin, Base, object]`, а не, скажем, `[Widget, Base, Mixin, object]`? За порядком стоит конкретный алгоритм - **C3-линеаризация**. Он не эвристика и не «как получится»: это детерминированная процедура с тремя гарантиями, и понимание её механики - разница между «MRO как чёрный ящик» и «я могу вывести MRO на бумаге быстрее, чем напечатаю `print`».

К концу раздела Вы сможете вручную линеаризовать любую иерархию операцией merge, объяснить понятия head и tail, и понять, почему наивный обход в глубину из старого Python давал неправильный ответ там, где C3 даёт правильный.

## Три обещания, которые держит C3

C3 строит такой порядок, что выполняются сразу три условия:

1. **Класс раньше своих предков.** Если `B` наследует `A`, то `B` стоит в списке раньше `A`.
2. **Сохранение порядка родителей.** Если класс объявлен как `class D(B, C)`, то `B` стоит раньше `C` (этот порядок называют *local precedence order*).
3. **Монотонность.** Порядок, который C3 даёт для класса, согласуется с порядками всех его предков: ни один потомок не «переворачивает» линеаризацию родителя.

Если ни один список, удовлетворяющий всем трём условиям, не существует, Python отказывается строить класс - но об этом в разделе про ромб. Сейчас важно: C3 - это не «обойти и записать», а «слить несколько упорядоченных списков, не нарушив ни одного из них».

## Операция merge: head, tail и правило хорошего кандидата

Линеаризация `L[C]` класса `C` с родителями `B1, B2, ..., Bn` определяется так:

```text
L[C] = C + merge(L[B1], L[B2], ..., L[Bn], [B1, B2, ..., Bn])
```

То есть: сам класс `C`, а за ним - результат слияния линеаризаций всех родителей плюс список самих родителей. Базовый случай - `L[object] = [object]`.

Вся работа в операции `merge`. Она берёт несколько списков и сливает их в один по шагам. На каждом шаге:

- **head** - первый элемент списка; **tail** - всё остальное.
- Берём head первого непустого списка. Это **хороший кандидат**, если он не появляется в tail (хвосте) *ни одного* из сливаемых списков.
- Если кандидат хороший - дописываем его в результат и вычёркиваем из всех списков. Начинаем шаг заново.
- Если head первого списка плохой - переходим к head следующего списка и проверяем его.
- Если все списки пусты - merge закончен. Если ни один head не оказался хорошим, а списки не пусты - линеаризация невозможна.

«Кандидат в tail другого списка» означает: где-то есть класс, который обязан идти *перед* кандидатом. Поставить кандидата сейчас - нарушить чей-то порядок. Поэтому такой head пропускается.

## Ручной разбор: линеаризуем ромб

Возьмём классический ромб и пройдём merge до конца руками. Объявления:

```python
class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass
```

Сначала линеаризации, которые мы уже знаем (одиночное наследование - тривиально):

```text
L[A] = [A, object]
L[B] = [B, A, object]
L[C] = [C, A, object]
```

Теперь `D`:

```text
L[D] = D + merge(L[B], L[C], [B, C])
     = D + merge([B, A, object], [C, A, object], [B, C])
```

Идём по шагам. Кандидат - head первого списка.

<Steps>
  <li>head = <code>B</code>. Есть ли <code>B</code> в хвосте какого-нибудь списка? В <code>[A, object]</code> нет, в <code>[C, A, object]</code> нет, в <code>[C]</code> нет. Хороший. Берём <code>B</code>, вычёркиваем отовсюду. Результат: <code>[B]</code>.</li>
  <li>Остались <code>[A, object]</code>, <code>[C, A, object]</code>, <code>[C]</code>. head = <code>A</code>. Но <code>A</code> в хвосте второго списка <code>[C, <strong>A</strong>, object]</code>. Плохой. Пробуем head следующего списка.</li>
  <li>head второго списка = <code>C</code>. Есть ли <code>C</code> в чьём-то хвосте? Нет. Хороший. Берём <code>C</code>. Результат: <code>[B, C]</code>.</li>
  <li>Остались <code>[A, object]</code>, <code>[A, object]</code>, <code>[]</code>. head = <code>A</code>, в хвостах его больше нет. Хороший. Результат: <code>[B, C, A]</code>.</li>
  <li>Остались <code>[object]</code>, <code>[object]</code>. head = <code>object</code>, хороший. Результат: <code>[B, C, A, object]</code>. Списки пусты - merge завершён.</li>
</Steps>

Итог: `L[D] = [D, B, C, A, object]`. Проверим, что Python согласен.

```python
class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

print([c.__name__ for c in D.__mro__])
```

Печатается `['D', 'B', 'C', 'A', 'object']` - ровно то, что мы получили вручную. Обратите внимание на ключевой момент шага 2: `A` не встал сразу после `B`, хотя в `L[B]` он шёл вторым. C3 отложил `A` до тех пор, пока не выстроились *все* его потомки (`B` и `C`), - в этом и состоит сохранение инварианта «класс раньше предков».

## Почему наивный DFS ломался

До C3 (и в классах старого стиля Python 2) MRO строился наивным обходом в глубину слева направо: спускаемся к первому родителю до упора, потом ко второму. Для ромба `D(B, C)` такой обход дал бы `D, B, A, object, C, A, object`, после удаления повторов слева - `[D, B, A, C, object]`.

Видите беду? `A` оказался *раньше* `C`, хотя `C` - потомок `A`. Если `A` и `C` оба определяют `method`, наивный порядок вызовет версию из `A`, проигнорировав более специализированную версию `C`. Переопределение в `C` тихо перестало работать. Именно эту ошибку и устраняет C3: его второй проход проверки хвостов не даёт предку встать раньше потомка.

<Callout type="warning">
Наивный DFS нарушает базовый инвариант «потомок раньше предка». C3 его сохраняет ценой более сложного слияния. Поэтому в Python 3, где остался только C3, переопределения в ромбе работают предсказуемо.
</Callout>

## Типичные ошибки в ручной линеаризации

- **Проверять кандидата только в «своём» списке.** Хороший кандидат не должен быть в хвосте *ни одного* из сливаемых списков, включая список самих родителей `[B1, ..., Bn]`. Забыли проверить один список - получили неверный порядок.
- **Считать head в хвосте, если он там стоит первым.** Кандидат - это сам head; «в хвосте» означает на любой позиции, кроме первой, в любом *другом* списке. Первая позиция (тоже head) конфликтом не считается.
- **Останавливаться на первом плохом head, объявляя ошибку.** Плохой head первого списка - не повод сдаваться: нужно попробовать head следующего списка. Ошибка объявляется, только когда *все* head плохие, а списки не пусты.

<Detail summary="Откуда взялось название C3">

«C3» - не версия и не кодовое имя Python. Так алгоритм назвали авторы статьи о линеаризации в языке Dylan: три «C» соответствуют трём свойствам, которые он гарантирует одновременно (согласованность с порядком предков, согласованность с локальным порядком родителей и монотонность). Python заимствовал этот алгоритм для классов нового стиля начиная с версии 2.3. Канонический разбор с доказательствами - официальный документ [The Python 2.3 Method Resolution Order](https://docs.python.org/3/howto/mro.html).
