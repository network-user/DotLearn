# Партиционирование: одна таблица, нарезанная на управляемые куски

Таблица `events` растёт третий год и добралась до двух миллиардов строк. Запросы по свежим данным ещё терпимы, но всё вокруг начало болеть: индекс распух, `VACUUM` не успевает, а самое неприятное происходит ночью, когда чистильщик пытается удалить данные старше года. Один большой `DELETE` блокирует таблицу, генерирует горы мусорных версий строк и растягивается на часы. При этом логически задача простая: свежее нужно, старое нет. Партиционирование ровно про это. Оно разрезает одну логическую таблицу на несколько физических кусков внутри того же узла, и вдруг «удалить старое» превращается из ночного кошмара в мгновенную операцию.

## Одна логическая таблица, много физических партиций

Партиционирование это разбиение одной таблицы на партиции по значению ключа партиционирования. Для приложения таблица остаётся одной: те же `INSERT`, те же `SELECT`, тот же внешний вид. Под капотом СУБД держит несколько физических кусков и по значению ключа знает, в какой из них класть строку и в каком искать. Важно не путать это с шардированием: все партиции живут на одном узле. Партиционирование не добавляет мощности железа, оно наводит порядок внутри одного сервера.

## Три способа резать: range, list и hash

Ключ партиционирования режут одним из трёх правил, и выбор диктуется тем, как вы обычно спрашиваете данные.

Партиционирование по диапазону (range) кладёт в партицию непрерывный интервал упорядоченного ключа: одна партиция на месяц по дате, интервалы id, ценовые полосы. Это самый частый выбор для данных с временем, потому что запросы к таким данным почти всегда содержат диапазон дат.

Партиционирование по списку (list) назначает партицию по явному набору значений: партиция для региона `RU`, отдельная для `EU`, отдельная для остальных. Годится, когда ключ это небольшой набор категорий с понятной семантикой.

Партиционирование по хешу (hash) кладёт строку в одну из `N` партиций по `hash(ключ) % N`. Естественного диапазона нет, зато нагрузка размазывается ровно. Это выбор, когда вам нужна просто равномерная нарезка большой таблицы без привязки к времени или категории.

## Partition pruning: откуда берётся ускорение

Главная выгода партиционирования по скорости называется partition pruning (отсечение партиций). Когда в запросе есть условие по ключу партиционирования, планировщик заранее понимает, какие партиции вообще могут содержать нужные строки, и остальные даже не открывает. Запрос `WHERE created_at >= '2026-06-01'` к таблице, нарезанной по месяцам, прочитает одну июньскую партицию вместо всех тридцати шести. Читается меньше данных, кеш попадает точнее, план проще.

У этого свойства есть обратная сторона, про которую забывают. Отсечение работает, только если фильтр по ключу партиционирования вообще есть в запросе. Запрос `WHERE user_id = 42` к таблице, нарезанной по дате, не сможет ничего отсечь: нужный пользователь мог наследить в любом месяце, поэтому планировщик пройдёт все партиции. Если такие запросы у вас основные, партиционирование по дате не ускорит их, а лёгкий накладной расход на много партиций даже слегка замедлит. Ключ партиционирования выбирают под форму частых запросов, а не наугад.

## Партиции и индексы решают разные задачи

Их легко перепутать, потому что оба ускоряют запросы, но работают они по-разному и не заменяют друг друга. Индекс это вторичная структура, которая для селективного условия быстро находит нужные строки во всей таблице. Партиция это физический раздел данных по ключу. Внутри партиций вы по-прежнему строите индексы: партиционирование не отменяет их, а дополняет. Селективный поиск `WHERE email = ...` ускоряет индекс; отсечение по дате и дешёвое управление жизненным циклом даёт партиционирование.

## Время и жизненный цикл: где партиции незаменимы

Самая сильная причина партиционировать это не скорость чтения, а управление жизненным циклом данных. Вернёмся к ночному `DELETE` из завязки. В обычной таблице удаление данных за старый месяц читает и помечает миллионы строк, держит блокировки, плодит мусорные версии, а потом за ними ещё бежит `VACUUM`. В таблице, нарезанной по месяцам, удаление старого месяца это отсоединение или удаление целой партиции: операция над метаданными, почти мгновенная и без нагрузки на остальные данные. Хотите не удалять, а увезти в холодное хранилище, отсоедините партицию и выгрузите её отдельно. Ротация данных «держим год, старое в архив» на партициях по времени становится тривиальной, и ради одного этого партиционирование часто и затевают.

## Декларативное партиционирование в Postgres

В PostgreSQL это описывается прямо в схеме и называется декларативным партиционированием. Родительскую таблицу объявляют партиционированной по ключу (например, по диапазону `created_at`), а затем создают дочерние партиции под конкретные интервалы: под каждый месяц свою. Приложение пишет и читает родительскую таблицу как обычную, а планировщик сам направляет строки в нужную партицию и отсекает лишние при чтении. Новый месяц это создание новой дочерней партиции; выход месяца из окна хранения это отсоединение старой. Часто заводят автоматику, которая заранее создаёт партиции на несколько периодов вперёд, чтобы вставка никогда не упёрлась в отсутствие подходящей партиции.

## Краевые случаи

- **Запрос без ключа партиционирования.** Отсекать нечего, планировщик идёт по всем партициям. На многих партициях это даже чуть дороже обычной таблицы. Форма частых запросов должна включать ключ партиционирования, иначе выигрыша нет.
- **Слишком много партиций.** Тысячи мелких партиций раздувают время планирования и потребление памяти планировщиком. Гранулярность выбирают разумную: обычно месяц или неделя, а не час.
- **Глобальная уникальность.** Уникальность, не включающая ключ партиционирования, тяжела: СУБД проверяет её отдельно в каждой партиции. Уникальный ключ удобно строить так, чтобы он содержал ключ партиционирования.
- **Перекос между партициями.** Промо-месяц может распухнуть в разы. На одном узле это не катастрофа, как при шардировании, но запросы по этому периоду медленнее, а обслуживание тяжелее.

## Частые ошибки

- **Партиционировать не по тому ключу.** Если частые запросы фильтруют по `user_id`, а таблица нарезана по дате, отсечение не сработает. Ключ выбирают под запросы.
- **Ждать от партиций магии индекса.** Партиции не ускоряют точечный селективный поиск сами по себе. Внутри партиций всё равно нужны индексы.
- **Удалять старое обычным DELETE в партиционированной таблице.** Смысл партиций по времени в том, чтобы отсоединять и дропать партицию целиком, а не гонять построчное удаление.
- **Дробить слишком мелко.** Партиции по часам на годы истории это десятки тысяч кусков и медленный планировщик. Начните с месяца.
