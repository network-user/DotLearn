# Three strategies everyone mixes up

A single database on a single server carries a surprising amount of load. But eventually a day arrives when requests start queueing, the nightly backup no longer finishes by morning, and the disk sits at ninety percent full. The first instinct, a bigger server, works only until the next ceiling. From there the choice turns architectural, and three words surface that people constantly blur together in conversations and interviews: replication, partitioning, sharding.

Keeping them apart matters because they solve three different problems and cost differently. Replication keeps the same data on several nodes: it is about fault tolerance and read scale. Partitioning cuts a large table into manageable chunks inside a single node: it is about operability and query speed against part of the data. Sharding spreads different data across different nodes: it is about volume and write scale, and it is also the most expensive of the three to run.

The three strategies are not mutually exclusive; they stack in layers. A large production system is typically sharded across several nodes, each shard is partitioned by time, and every shard carries replicas for failover and for offloading reporting reads. The rollout order is almost always the same too: teams squeeze out vertical growth and replicas first, then partition the heaviest tables, and only turn to sharding once they hit a wall on write throughput or on how much a single node can hold.

| Strategy | What it copies or splits | What it buys you | Main pain point |
| --- | --- | --- | --- |
| Replication | same data on many nodes | reads, fault tolerance | replica lag, stale reads |
| Partitioning | one table into chunks, one node | manageability, speed on part of the data | doesn't help once a single node runs out of room |
| Sharding | different data on different nodes | writes, volume | cross-shard queries, resharding |

This topic follows exactly that order. We start by locating the ceiling of a single server and seeing why read and write paths get split apart first. Then, one at a time, we cover replication, partitioning, and sharding, unpack consistent hashing as a separate technique against painful data reshuffling, and finish with a scenario-based walkthrough: which load profile calls for which strategy, and in what order to bring it in.
