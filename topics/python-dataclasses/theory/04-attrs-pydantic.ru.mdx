# Где данным нужна не структура, а проверка

Обычный dataclass честно раскладывает аргументы по полям и на этом останавливается. Он вам полностью доверяет: `Box("5")` для поля `size: int` создаст объект, в котором `size` останется строкой `"5"`, и никто не возразит. Внутри вашего кода, где вы контролируете входные данные, это ровно то, что нужно, и лишняя проверка только тормозила бы.

Проблема на границе. Тело HTTP-запроса, строка конфигурации, сообщение из очереди приходят от того, кому нельзя верить: типы не те, полей не хватает, числа приезжают строками. Тут структуры мало, нужна валидация. За неё отвечают две библиотеки за пределами стандартной: `attrs` и `pydantic`. Этот раздел проводит границу между «доверять» и «проверять» и показывает, какой инструмент где уместен. Код `attrs` и `pydantic` здесь только для чтения: в песочнице этих библиотек нет, запускать их не нужно.

## Спектр от структуры к валидации

Три инструмента выстраиваются в линию по тому, сколько работы они берут на себя. Стандартный dataclass даёт структуру и ничего не проверяет. `attrs` добавляет к структуре опциональные валидаторы и конвертеры. `pydantic` идёт дальше всех: он валидирует и приводит типы, и создан именно для приёма недоверенных данных.

## dataclass доверяет: увидьте это своими глазами

Прежде чем звать внешние библиотеки, стоит прочувствовать, чего именно не делает стандартный dataclass. Аннотация типа для него это подсказка человеку и анализатору, но не проверка в рантайме. Запустите код: поле `size` объявлено как `int`, а внутри спокойно лежит строка. Поменяйте `"5"` на число `5` и посмотрите, как вторая строка вывода превратится из `55` в `10`.

<PyDemo
  title="dataclass не приводит и не проверяет типы"
  code={`from dataclasses import dataclass

@dataclass
class Box:
    size: int

b = Box("5")
print(b)
print(b.size + b.size)
print(type(b.size).__name__)`}
/>

Вывод `Box(size='5')`, затем `55`, затем `str`: аннотация `int` не помешала положить строку, и `"5" + "5"` дало конкатенацию, а не сумму. Это не баг dataclass, а его контракт: структура без проверок. Как только данные приходят извне, такой контракт становится опасным.

## attrs: структура плюс валидаторы и конвертеры

`attrs` это предшественник dataclass и его более гибкий родственник. Синтаксис похож, но у полей есть два рычага, которых нет в стандартной библиотеке. **Конвертер** преобразует значение перед сохранением. **Валидатор** проверяет значение и бросает ошибку, если оно недопустимо.

```python
from attrs import define, field
from attrs.validators import ge

@define
class Account:
    name: str
    price: int = field(converter=int)
    balance: int = field(validator=ge(0))
```

Здесь `converter=int` превратит `"5"` в `5` при создании, а `validator=ge(0)` не даст завести отрицательный баланс, бросив ошибку. `attrs` берут, когда нужна тонкая настройка полей и производительность, но не хочется тащить всю модель валидации pydantic.

## pydantic: проверка и коэрция на границе

`pydantic` создан для одного: принять данные из внешнего мира и либо превратить их в проверенный объект, либо внятно отказать. Модель наследуется от `BaseModel`, а разбор входа делает `model_validate`.

```python
from pydantic import BaseModel

class User(BaseModel):
    id: int
    name: str

user = User.model_validate({"id": "1", "name": "Анна"})
```

Строка `"1"` в поле `id: int` не вызовет ошибку: pydantic приведёт её к числу `1` (коэрция). А вот значение, которое привести нельзя, вызовет `ValidationError` с описанием, что именно не так и в каком поле. Обратная операция это `model_dump`, отдающая словарь для сериализации.

```python
user.id
user.model_dump()
```

Первое вернёт `1` (уже `int`), второе `{"id": 1, "name": "Анна"}`. Это две границы одной модели: `model_validate` на входе, `model_dump` на выходе.

<Callout type="info">
pydantic v2 переписан с ядром на Rust, поэтому валидация быстрая, но не бесплатная. Когда коэрция нежелательна и типы должны совпадать точно, включают strict-режим, и тогда `"1"` в поле `int` уже будет ошибкой, а не молчаливым приведением.
</Callout>

## Правило: dataclass внутри, pydantic на границе

Философия двух миров умещается в одну фразу: **dataclass доверяет, pydantic проверяет**. Отсюда практическое правило размещения. На границе ввода-вывода, где данные недоверенные, ставьте pydantic: он проверит форму и типы один раз, при входе. Дальше, в глубине домена, где данные уже проверены, достаточно dataclass: платить за повторную валидацию на каждом шаге незачем.

Эта связка особенно заметна во фреймворках вроде FastAPI. Там модели запроса и ответа это pydantic BaseModel: фреймворк сам валидирует тело запроса на входе и сериализует ответ на выходе. А доменные объекты, которые живут между обработчиком и базой, спокойно остаются обычными dataclass, потому что до них доходят уже проверенные данные.

## Мост между мирами: pydantic поверх dataclass

Иногда хочется знакомый синтаксис dataclass, но с проверкой на входе. Для этого у pydantic есть отдельный декоратор `pydantic.dataclasses.dataclass`: он выглядит и ведёт себя как стандартный dataclass (работают `fields()`, `asdict`, тот же способ объявления полей), но добавляет валидацию при создании. Это способ навесить проверки на существующую доменную модель, не переписывая её в `BaseModel`. Обратная сторона в том, что это уже не стандартный `dataclasses.dataclass`: сторонний код, завязанный на точный тип из стандартной библиотеки, может такую модель не распознать.

```python
from pydantic.dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int
```

Такой `Point(x="1", y=2)` уже приведёт `"1"` к числу или бросит `ValidationError`, тогда как стандартный dataclass промолчал бы.

Сама `ValidationError` несёт не строку, а структурированный список: у каждого пункта есть путь к полю, тип ошибки и сообщение, доступные через `.errors()`. Именно поэтому pydantic удобен на границе HTTP: из этого списка легко собрать точный ответ о том, какое поле не прошло проверку. Платить за это приходится тактами: даже с ядром на Rust валидация не бесплатна, и это ещё один довод проверять данные один раз на границе, а не на каждом внутреннем объекте.

## Частые ошибки

- **pydantic везде.** Валидировать каждый внутренний объект на каждом шаге это лишние такты. Проверьте один раз на границе, дальше доверяйте типам.
- **dataclass на границе.** Принимать недоверенный JSON в обычный dataclass опасно: он молча примет строку вместо числа и отсутствующую вложенность. На входе нужна валидация.
- **Путать API pydantic v1 и v2.** В v2 методы называются `model_validate` и `model_dump`; старые `parse_obj` и `dict` из v1 в новом коде не используют.
- **Ждать проверки от аннотаций.** Ни dataclass, ни attrs без явного валидатора не проверяют типы в рантайме. Аннотация это контракт для человека и анализатора, а не гарантия при запуске.

<Detail summary="Почему dataclass так похож на attrs">
Это не совпадение: dataclass вырос из `attrs`. Когда идея «сгенерируй мне __init__ и __repr__ по полям» доказала пользу в стороннем `attrs`, её перенесли в стандартную библиотеку как PEP 557. Стандартный dataclass намеренно взял только базовую часть (структуру и генерацию методов) и оставил валидаторы, конвертеры и прочую гибкость снаружи. Поэтому `attrs` жив: он покрывает то, что dataclass принципиально не делает, сохраняя знакомый синтаксис.
</Detail>
