# Что декоратор @dataclass дописывает в ваш класс

Вы уже ставили `@dataclass` над классом и получали бесплатные `__init__`, `__repr__` и `__eq__`. Пока всё работает, устройство декоратора можно не знать. Проблемы начинаются, когда поле молча пропадает из конструктора, `__hash__` внезапно оказывается `None`, а аннотация `ClassVar` протекает туда, где её быть не должно. В такие моменты нужно видеть машину под капотом, а не гадать.

Этот раздел вскрывает декоратор. Вы узнаете, что именно он собирает и генерирует, как заглянуть в результат через `fields()` и `__dataclass_fields__`, разберёте полную поверхность функции `field()`, все флаги самого декоратора и два спецобъявления, которые ведут себя не как поля: `InitVar` и `ClassVar`. Это уровень, на котором вопрос «а что здесь происходит» перестаёт вас останавливать.

## Декоратор это генератор кода, а не обёртка времени выполнения

Ключевая идея: `@dataclass` работает один раз, в момент создания класса, и больше в вызовы не вмешивается. Он проходит по аннотированным переменным класса, строит для каждой дескриптор `Field`, складывает их в словарь `__dataclass_fields__`, а затем синтезирует методы как обычный Python-код и исполняет его. После этого перед вами остаётся простой класс с честными методами, будто вы написали их руками.

Из этого следует практичный вывод. Поле определяет именно аннотация типа: `x: int` создаёт поле, а `count = 0` без аннотации остаётся обычной переменной класса и в сгенерированные методы не попадает. Аннотация это единственный сигнал, по которому декоратор понимает, что перед ним поле.

## fields() и __dataclass_fields__: как посмотреть на результат

Собранные поля доступны для интроспекции. Публичный вход это функция `dataclasses.fields()`: она возвращает кортеж объектов `Field` в порядке объявления, у каждого есть `.name`, `.type`, `.default`, `.default_factory`, `.repr`, `.compare` и `.metadata`.

```python
from dataclasses import dataclass, field, fields

@dataclass
class Account:
    login: str
    balance: int = 0

for f in fields(Account):
    print(f.name, f.type, f.default)
```

Тот же словарь лежит в атрибуте `__dataclass_fields__`, но `fields()` предпочтительнее: он отдаёт стабильный кортеж и отсеивает служебные записи. Помните различие: `fields()` возвращает кортеж, `__dataclass_fields__` это словарь по именам. Проверить, что перед вами dataclass, помогает `dataclasses.is_dataclass(obj)`.

## Полная поверхность field()

Присваивание умолчания прямо в объявлении покрывает простой случай. Всё остальное настраивает функция `field()`. Она не задаёт значение сама, а описывает поведение поля:

- `default` задаёт то же умолчание, что и присваивание, но в одном ряду с прочими настройками.
- `default_factory` вызывает фабрику при создании каждого экземпляра. Это единственный правильный способ дать полю изменяемое умолчание (список, словарь, множество).
- `repr=False` убирает поле из `__repr__`. Удобно для длинных или секретных значений вроде токена.
- `compare=False` исключает поле из `__eq__` и сравнения. Так ведут себя кэш или временная метка, которые не определяют равенство объекта.
- `kw_only=True` делает поле только-именованным в конструкторе.
- `metadata` принимает словарь произвольных данных для сторонних инструментов. Сам dataclass его не читает.

```python
from dataclasses import dataclass, field

@dataclass
class Account:
    login: str
    password: str = field(repr=False)
    balance: int = 0
    _cache: dict = field(default_factory=dict, compare=False, repr=False)
    weight_kg: float = field(default=0.0, metadata={"unit": "kg"})
```

Здесь `__repr__` не покажет `password` и `_cache`, две учётки с одинаковыми `login` и `balance` окажутся равными несмотря на разный `_cache`, а `metadata` донесёт до сериализатора, что вес измеряется в килограммах. Метаданные доступны только для чтения: `fields(Account)[3].metadata` вернёт `MappingProxyType`, изменить его нельзя.

Загляните в поля вживую. Запустите код, затем поменяйте у любого поля `compare=True` на `compare=False` и снова запустите: увидите, как меняется колонка `compare`.

## Флаги декоратора

Скобки после `@dataclass` включают и выключают целые методы:

- `init` (по умолчанию `True`): генерировать ли `__init__`.
- `repr`, `eq` (по умолчанию `True`): генерировать ли соответствующие методы. Отключите `repr`, если пишете свой.
- `order` (по умолчанию `False`): добавить `<`, `<=`, `>`, `>=`. Сравнение идёт лексикографически по кортежу полей.
- `frozen` (по умолчанию `False`): запретить присваивание полям после создания.
- `unsafe_hash` (по умолчанию `False`): принудительно синтезировать `__hash__`. Название предупреждает: вы берёте на себя ответственность за стабильность хэша.
- `slots` (по умолчанию `False`): сгенерировать `__slots__` вместо `__dict__`.
- `kw_only` (по умолчанию `False`): сделать все поля только-именованными.
- `match_args` (по умолчанию `True`): сгенерировать `__match_args__` для конструкций `match`.

Поведение `frozen`, `slots`, наследования и `kw_only` заслуживает отдельного разбора, ему посвящён следующий раздел. Здесь важно запомнить сам набор рычагов.

## Два объявления, которые не поля: ClassVar и InitVar

Аннотация обычно создаёт поле, но два маркёра из этого правила выпадают.

`typing.ClassVar` помечает атрибут как принадлежащий классу, а не экземпляру. Декоратор его распознаёт и полностью исключает: такого имени не будет ни в `__init__`, ни в `fields()`, ни в `__eq__`. Это способ хранить общую для всех экземпляров константу рядом с полями, не превращая её в поле.

`dataclasses.InitVar` описывает псевдополе: значение приходит в `__init__` и передаётся в `__post_init__`, но не сохраняется как атрибут и в `fields()` не попадает. Через него удобно прокидывать входные данные, нужные только на этапе инициализации.

```python
from dataclasses import dataclass, InitVar, field
from typing import ClassVar

@dataclass
class User:
    name: str
    version: ClassVar[int] = 2
    seed: InitVar[int] = 0
    token: str = field(init=False, default="")

    def __post_init__(self, seed: int):
        self.token = f"{self.name}-{seed}"
```

Здесь `version` общий для всех и в конструктор не просится, `seed` доходит до `__post_init__` первым аргументом после `self`, но атрибутом не становится, а `token` вычисляется внутри и в `__init__` не принимается благодаря `init=False`.

<Callout type="warning">
Аргументы `InitVar` приходят в `__post_init__` в порядке объявления. Если забыть параметр в сигнатуре `__post_init__`, Python бросит `TypeError` о лишнем позиционном аргументе. Число и порядок `InitVar` и параметров `__post_init__` должны совпадать.
</Callout>

## Ловушка изменяемого умолчания, коротко

Одну граблю стоит держать в голове даже при поверхностной работе с dataclass. Строка `items: list = []` не пройдёт: декоратор бросит `ValueError` уже при определении класса. Причина классическая для Python: умолчание вычисляется один раз, и все экземпляры делили бы один список. Лекарство ровно одно, `field(default_factory=list)`: фабрика создаёт свежий объект каждому экземпляру. Подробный разбор этой ловушки есть во вводном разделе про dataclass в курсе ООП, здесь достаточно помнить рецепт.

## Частые ошибки

- **Ждать `ClassVar` в конструкторе.** Поле с аннотацией `ClassVar` в `__init__` не попадает и в `fields()` не появляется. Если значение должно приходить в конструктор, `ClassVar` вам не подходит.
- **Путать `fields()` и `__dataclass_fields__`.** Первое отдаёт кортеж и это публичный API, второе это внутренний словарь. Для перебора берите `fields()`.
- **Менять `metadata`.** Метаданные поля доступны только для чтения через `MappingProxyType`. Присваивать в них нельзя, задавайте словарь сразу в `field(metadata=...)`.
- **Забыть аргумент `InitVar` в `__post_init__`.** Тогда либо значение не дойдёт, либо конструктор упадёт на несовпадении сигнатуры.

<Detail summary="Как декоратор на самом деле синтезирует методы">
Внутри `dataclasses` собирает текст функции как обычную строку и исполняет её через вспомогательный `exec`. Например, для `__init__` он строит строку вроде `def __init__(self, x, y=0): self.x = x; self.y = y` и компилирует её в настоящую функцию, привязанную к классу. Поэтому у полей есть строгий порядок, а умолчания обязаны идти после обязательных полей: это правила обычной сигнатуры функции, а не отдельная выдумка dataclass. Значения умолчаний при этом прячутся в замыкание генератора, чтобы каждое поле ссылалось на свой объект.
</Detail>
