# Как выбрать форму данных между слоями

Вы передаёте словарь `{"host": ..., "port": ...}` через пять функций. Где-то в глубине кто-то пишет `config["prot"]` вместо `"port"`, получает `KeyError` в лучшем случае или `None` в худшем, и вы час ищете, откуда взялась дыра. Форма этих данных нигде не записана: какие ключи, какие типы, что обязательно. Она живёт только в вашей голове и разъезжается по мере роста кода.

Моделирование данных это про то, чтобы сделать форму явной и проверяемой там, где сейчас голый словарь. В этом разделе собраны рабочие паттерны: DTO между слоями, config-объект вместо словаря настроек, value object для величин, а также обратная сторона, когда обычный класс уместнее dataclass, и почему сырой dict через всё приложение это антипаттерн.

## Модель данных это явный контракт

Разница между сырым словарём и dataclass не в синтаксисе, а в том, записан ли контракт. У словаря форма неявная: какие ключи есть, разработчик выясняет, читая код, который его наполняет. У dataclass форма это объявление класса: поля, типы, умолчания видны сразу, а опечатка в имени поля ловится сразу, как `TypeError` при создании или `AttributeError` при доступе, а не всплывает через три слоя.

## Config-объект вместо словаря настроек

Настройки это первый кандидат на замену словаря. У конфигурации фиксированный набор полей, у большинства из них есть разумные умолчания, и обращаются к ним из многих мест. dataclass собирает всё это в одном объявлении.

```python
from dataclasses import dataclass

@dataclass
class ServerConfig:
    host: str = "localhost"
    port: int = 8080
    debug: bool = False
```

Теперь `ServerConfig(port=443)` создаст конфиг с проверяемой формой: неизвестный ключ вызовет `TypeError` сразу, умолчания лежат в одном месте, а автодополнение в редакторе знает все поля. Словарь `{"port": 443}` не даёт ничего из этого.

## Value object: равенство по значению

Некоторые сущности определяются не идентичностью, а значением. Сто рублей это сто рублей независимо от того, какой это объект в памяти; две одинаковые координаты это одна и та же точка. Такие величины моделируют value object: неизменяемый объект, равный другому, если совпадают все поля. В Python это `frozen` dataclass, у которого бесплатно есть и сравнение по значению, и хэшируемость.

Запустите код: три платежа, из них два одинаковы по значению, поэтому множество схлопывает их в один. Проверка `in` тоже идёт по значению, а не по ссылке. Добавьте `Money(200, "RUB")` в список и посмотрите, как изменится длина.

<PyDemo
  title="Value object: равенство и дедупликация по значению"
  code={`from dataclasses import dataclass

@dataclass(frozen=True)
class Money:
    amount: int
    currency: str

payments = [Money(100, "RUB"), Money(100, "RUB"), Money(100, "USD")]
unique = set(payments)
print(len(unique))
print(Money(100, "RUB") in unique)`}
/>

Множество оставит два элемента, а проверка вхождения вернёт `True`: для value object два объекта с одинаковыми полями это один и тот же объект по смыслу.

## DTO: узкий контракт между слоями

Доменный объект часто содержит больше, чем нужно показывать наружу. У `User` есть `password_hash`, но обработчик HTTP не должен его сериализовать в ответ. DTO (объект передачи данных) это узкая форма, которую один слой отдаёт другому, с только нужными полями.

```python
from dataclasses import dataclass

@dataclass
class User:
    id: int
    name: str
    password_hash: str

def to_public_dto(user: User) -> dict:
    return {"id": user.id, "name": user.name}
```

`to_public_dto` намеренно собирает новый словарь, а не отдаёт `asdict(user)`: разворот всего объекта протащил бы `password_hash` наружу. Явный DTO это граница, на которой вы решаете, что уходит в другой слой. Как эти границы между слоями выстраиваются целиком, подробно разобрано в теме про clean architecture; здесь важно, что переход между слоями это хорошее место для явной, суженной формы данных.

## Когда обычный класс лучше dataclass

Моделирование не значит «всё в dataclass». dataclass силён, пока класс это в основном данные. Обратный случай, когда стоит вернуться к обычному классу:

- класс это в первую очередь поведение: много методов и логики, а полей-данных почти нет, и `@dataclass` только шумит;
- в конструкторе тяжёлая сборка или инварианты, которые не выражаются полями и `__post_init__`;
- нужен полный контроль над `__init__`, или объект изменяемый и сравнивать его по значению нельзя.

Не всякую пачку данных нужно моделировать вообще. Одноразовая пара значений внутри одной функции, которая никуда не передаётся, спокойно живёт кортежем: заводить под неё класс это лишний код.

## Сделайте недопустимые состояния невыразимыми

Сильнейшая форма моделирования это подобрать типы так, чтобы неверные данные нельзя было даже собрать. Поле статуса, объявленное как `str`, примет любую опечатку: `"aktive"`, `"ACTIVE"`, пустую строку. То же поле, объявленное `Enum`, принимает только перечисленные значения, а всё остальное подсветит анализатор ещё до запуска.

```python
from dataclasses import dataclass
from enum import Enum

class Status(Enum):
    ACTIVE = "active"
    BANNED = "banned"

@dataclass(frozen=True)
class Account:
    id: int
    status: Status
```

Для закрытого набора значений берите `Enum` вместо голой строки, для доменной величины отдельный value object вместо безымянного кортежа. Идея та же, что и с pydantic на границе из прошлого раздела: разобрать данные один раз в точную форму, а дальше доверять типу, а не проверять его снова и снова. Чем больше запрещает сам тип, тем меньше остаётся ручных проверок в коде и тем меньше багов доживает до рантайма.

## Частые ошибки

- **Сырой dict через всё приложение.** Форма неявная, опечатки всплывают поздно, поля и типы нигде не зафиксированы. Как только словарь переживает больше одной функции, дайте ему имя: dataclass, NamedTuple или, на границе, pydantic.
- **Протащить лишнее поле в DTO.** `asdict(user)` в ответ вынесет наружу и `password_hash`. Собирайте DTO явно из нужных полей.
- **Изменяемый value object.** Если величина сравнивается по значению, она обязана быть неизменяемой: иначе после мутации её хэш разъедется с корзиной в множестве. Делайте value object `frozen`.
- **Моделировать всё подряд.** dataclass под каждую двухэлементную мелочь это переусложнение. Для локальной одноразовой пары хватит кортежа.

<Detail summary="DTO и доменный объект это разные вещи">
Доменный объект живёт внутри приложения и несёт бизнес-смысл со всеми полями и инвариантами. DTO существует ради передачи: он подстроен под конкретную границу (ответ API, сообщение в очередь) и содержит ровно то, что этой границе нужно. Их часто путают и пытаются гонять один и тот же объект и как доменный, и как транспортный, отсюда и утечки лишних полей, и завязка домена на форму чужого протокола. Разделение стоит недорого (одна функция-конвертер), а развязывает домен и внешний контракт: домен меняется по своим причинам, транспорт по своим.
</Detail>
