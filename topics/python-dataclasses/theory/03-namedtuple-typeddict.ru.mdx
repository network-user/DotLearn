# Когда данным не нужен полноценный класс

Не каждой пачке данных нужен класс. Иногда функция возвращает пару значений, и хочется читать их по имени, а не по индексу `[0]` и `[1]`, но заводить ради этого dataclass кажется тяжеловесным. Иногда вы принимаете чужой JSON, который уже приходит словарём, и нужно лишь описать его форму для проверки типов, ничего не оборачивая. Для этих двух ниш в стандартной библиотеке есть `typing.NamedTuple` и `typing.TypedDict`.

Разобравшись, чем они отличаются от dataclass и друг от друга, вы перестанете тащить полноценный класс туда, где хватило бы именованного кортежа, и научитесь типизировать словари, не превращая их в объекты. Выбор между этими тремя формами это частый вопрос на собеседовании.

## Одна помечает кортеж, другая помечает словарь

`NamedTuple` порождает настоящий кортеж, у полей которого появились имена. Он неизменяем, итерируем, распаковывается, индексируется и сравнивается как обычный кортеж. Разница только в том, что к элементам можно обратиться по имени.

`TypedDict` не порождает никакого объекта. Это аннотация формы для обычного словаря: во время выполнения перед вами остаётся тот же `dict`, а описание полей существует лишь для проверяющего типы. Накладных расходов ноль, потому что нового типа в рантайме не появляется.

## NamedTuple: кортеж, который читается по имени

Классовый синтаксис `typing.NamedTuple` выглядит почти как dataclass: аннотированные поля, при желании со значениями по умолчанию. Но результат это подкласс `tuple`.

```python
from typing import NamedTuple

class Point(NamedTuple):
    x: int
    y: int = 0
```

У получившегося типа есть служебные атрибуты кортежа с именами: `Point._fields` вернёт `('x', 'y')`, а `_replace` отдаст копию с изменённым полем (сам объект неизменяем). Раз это кортеж, работает и распаковка, и индексация, и любые кортежные операции.

Потрогайте это вживую. Запустите код, затем добавьте третье поле `z: int = 0` и посмотрите, как распаковка и `sum` подхватят его без единой правки в остальном коде.

## TypedDict: форма словаря без обёртки

`TypedDict` описывает, какие ключи и каких типов ожидаются в словаре. Во время выполнения это по-прежнему обычный `dict`, поэтому создаётся он словарным литералом, а не конструктором.

```python
from typing import TypedDict

class Movie(TypedDict):
    title: str
    year: int

film: Movie = {"title": "Солярис", "year": 1972}
```

По умолчанию все ключи обязательны. Ослабить это можно двумя способами. Флаг `total=False` делает необязательными сразу все ключи. Точечно управлять помогает `NotRequired` (и парный `Required`): им помечают отдельные ключи, оставляя остальные как есть.

```python
from typing import TypedDict, NotRequired

class Config(TypedDict):
    host: str
    port: NotRequired[int]
```

`TypedDict` создан ровно для границы с внешним миром: тело HTTP-запроса, строка из очереди, ответ чужого API приходят словарями, и `TypedDict` даёт им проверяемую форму, не заставляя оборачивать каждый в объект.

## Что вас удивит

Раз `NamedTuple` это кортеж, он сравнивается как кортеж, **по значению и без оглядки на тип**. Поэтому `Point(1, 2) == (1, 2)` истинно, и `Point(1, 2)` из одного модуля равен `Rgb`-подобному именованному кортежу с теми же числами. Для dataclass это было бы `False` (там `__eq__` сначала проверяет класс), а для `NamedTuple` структурное равенство кортежей часто оказывается неожиданностью.

`TypedDict` не проверяет типы во время выполнения. Он существует только для статического анализатора. В рантайме `Movie` это `dict`, `isinstance(film, dict)` истинно, а вот `isinstance(film, Movie)` вообще запрещён. Никто не помешает положить в поле `year` строку: проверку сделает mypy или pyright при анализе кода, но не Python при запуске.

<Callout type="warning">
Ни `NamedTuple`, ни `TypedDict` не валидируют данные в рантайме. `NamedTuple` спокойно примет `Point("a", "b")`, а `TypedDict` это просто словарь. Нужна проверка при получении данных извне, обращайтесь к pydantic, о нём следующий раздел.
</Callout>

## Как выбрать между тремя формами

| Форма      | Что это в рантайме         | Когда брать                                                        |
| ---------- | -------------------------- | ----------------------------------------------------------------- |
| NamedTuple | подкласс `tuple`           | лёгкий неизменяемый набор значений, часто возврат из функции       |
| TypedDict  | обычный `dict`             | типизировать форму словаря на границе с JSON, без обёртки          |
| dataclass  | обычный класс с `__dict__` | методы, `__post_init__`, `frozen`, изменяемость, богатое поведение |

Правило короткое: возвращаете несколько значений и хотите неизменности, берите `NamedTuple`; типизируете чужой словарь, берите `TypedDict`; всё, что сложнее (поведение, валидация в `__post_init__`, настройка полей), это территория dataclass.

## Цена и границы NamedTuple

Раз `NamedTuple` это кортеж, у экземпляра нет `__dict__`: он весит столько же, сколько обычный кортеж, и заметно меньше, чем dataclass с полноценным словарём атрибутов. На миллионах мелких записей (строки из базы, точки, координаты) это ощутимая экономия памяти и честная причина предпочесть `NamedTuple`, когда запись только читается. Доступ по имени идёт через дескриптор поля, поэтому он почти так же быстр, как индексация по числу. Ещё одно следствие кортежной природы: `NamedTuple` хэшируется по значению сразу, без `frozen`, поэтому годится ключом словаря или элементом множества из коробки, тогда как обычный изменяемый dataclass для этого приходится явно замораживать флагом `frozen=True`, иначе у него остаётся `__hash__ = None`.

У кортежной природы есть и границы. Подкласс `NamedTuple` может добавить методы, но не может добавить новые поля: набор полей фиксируется базовым объявлением. Все поля публичны и позиционны, приватного состояния и скрытых инвариантов у кортежа нет. Как только запись обрастает поведением, изменяемостью или проверками, кортеж начинает мешать, и форму переносят на dataclass.

## Частые ошибки

- **Ждать проверку типов от `TypedDict` в рантайме.** Её нет. `TypedDict` помогает анализатору, но при запуске это словарь без гарантий.
- **Сравнивать `NamedTuple` и забыть, что это кортеж.** Именованный кортеж равен обычному кортежу с теми же значениями. Если равенство должно учитывать тип, вам нужен dataclass.
- **Пытаться изменить `NamedTuple` на месте.** Он неизменяем: присваивание полю запрещено, для «изменения» используйте `_replace`, получая новый кортеж.
- **Брать `NamedTuple` для объекта с поведением.** Как только появляются методы и логика, кортеж мешает: переходите на dataclass.

<Detail summary="Чем typing.NamedTuple отличается от collections.namedtuple">
Старый `collections.namedtuple` задаёт поля строкой имён и не знает про аннотации типов. `typing.NamedTuple` даёт тот же кортеж, но с классовым синтаксисом, аннотациями и значениями по умолчанию, читаемыми анализатором. Под капотом оба порождают подкласс `tuple` с одинаковыми служебными методами (`_fields`, `_replace`, `_asdict`); разница в удобстве объявления и в поддержке типов. Для нового кода берут `typing.NamedTuple`.
</Detail>
