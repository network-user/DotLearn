# frozen, slots и наследование: где dataclass показывает характер

Простой `@dataclass` покрывает случай «класс это в основном данные». Но как только вы делаете значение по-настоящему неизменяемым, экономите память на миллионах объектов или выстраиваете иерархию, всплывают острые углы. Вы пишете нормализацию в `__post_init__` неизменяемого класса и получаете `FrozenInstanceError`. Добавляете `slots=True` и ломаете наследование. Наследуете dataclass с полем-умолчанием и внезапно не можете добавить обязательное поле в потомке.

Здесь разобраны четыре механизма, которые превращают контейнер данных в инструмент моделирования: `frozen`, `slots`, наследование полей и семейство `replace`/`asdict`/`astuple`. Каждый из них экономит вам код, и у каждого есть цена, которую стоит знать заранее.

## frozen ставит стену на присваивание, а не морозит объект

`@dataclass(frozen=True)` не делает объект глубоко неизменяемым. Он лишь заменяет `__setattr__` и `__delattr__` на версии, которые бросают `FrozenInstanceError` при любой попытке присвоить или удалить атрибут. Внутри объекта всё остаётся как было: если поле хранит список, этот список по-прежнему можно пополнять. Заморожена ссылка, а не то, на что она указывает.

Отсюда единственный законный способ задать поле внутри `__post_init__` неизменяемого класса: обойти собственный `__setattr__` через `object.__setattr__`. Это ровно тот вызов, которым пользуется и сгенерированный `__init__`, когда класс заморожен.

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class Percent:
    value: float

    def __post_init__(self):
        clamped = min(100.0, max(0.0, self.value))
        object.__setattr__(self, "value", round(clamped, 1))
```

Прямое `self.value = ...` здесь бросило бы `FrozenInstanceError`; `object.__setattr__` записывает значение в обход стены.

## frozen, eq и __hash__: три расклада

Хэшируемость dataclass зависит от пары флагов `eq` и `frozen`, и раскладов ровно три. Понимание этой таблицы снимает большинство вопросов «почему мой объект вдруг нехэшируемый».

/>

Логика простая. Изменяемый объект, который сравнивается по значению, опасно класть в множество: поменяются поля, поменяется хэш, и объект «потеряется» в своей корзине. Поэтому такой dataclass получает `__hash__ = None`. `frozen=True` снимает противоречие: поля неизменны, хэш стабилен, и Python его синтезирует.

Проверьте расклад руками. Запустите код, затем уберите `frozen=True` и снова запустите: строка с `hash()` упадёт с `TypeError`, потому что объект стал нехэшируемым.

print(a, b)

## slots=True: меньше памяти ценой гибкости

`@dataclass(slots=True)` генерирует `__slots__` из имён полей и убирает у экземпляров словарь `__dict__`. На больших количествах объектов это заметная экономия памяти и чуть более быстрый доступ к атрибутам: значения лежат в фиксированных ячейках, а не в хэш-таблице.

Плата за это гибкость:

- новый атрибут, не объявленный полем, присвоить нельзя: `obj.extra = 1` бросит `AttributeError`;
- слабые ссылки (`weakref`) требуют отдельного слота `__weakref__`, иначе не работают;
- множественное наследование от нескольких slotted-классов конфликтует по раскладке слотов;
- декоратор пересоздаёт класс (нельзя добавить `__slots__` к существующему), поэтому ссылки на исходный объект класса, снятые до `@dataclass`, укажут на другой класс.

`slots=True` берут осознанно, под профиль «много мелких объектов, память дороже гибкости». Для конфигурации в единственном экземпляре выигрыша нет.

## Наследование: порядок полей и проблема дефолтов

При наследовании dataclass собирает поля от базовых классов к производным: сначала идут поля базы в порядке объявления, затем поля потомка. Это делает сигнатуру `__init__` предсказуемой.

Но тот же порядок рождает классическую ловушку. Если у поля базы есть умолчание, а в потомке вы добавляете обязательное поле, получается «обязательный аргумент после аргумента с умолчанием», чего сигнатура функции не допускает:

```python
from dataclasses import dataclass

@dataclass
class Base:
    created_at: str = "2024"

@dataclass
class Event(Base):
    name: str
```

Этот код бросит `TypeError: non-default argument 'name' follows default argument` уже при определении `Event`. Спасает `kw_only`: поле, объявленное только-именованным, не участвует в позиционном порядке, поэтому конфликт исчезает.

```python
from dataclasses import dataclass, field

@dataclass
class Event(Base):
    name: str = field(kw_only=True)
```

Теперь `Event(name="click")` создаётся штатно, `created_at` берёт умолчание из базы. Тот же эффект даёт `@dataclass(kw_only=True)` на всём классе.

## frozen нельзя смешивать в иерархии

Заморозку выбирают не для одного класса, а для всей ветки наследования. Если `@dataclass(frozen=True)` наследует от обычного dataclass, Python бросит `TypeError: cannot inherit frozen dataclass from a non-frozen one`. Обратное сочетание, обычный потомок замороженной базы, тоже запрещено с симметричной ошибкой.

```python
from dataclasses import dataclass

@dataclass
class Base:
    x: int

@dataclass(frozen=True)
class Sub(Base):
    y: int
```

Причина в согласованности контракта. Замороженный класс держит `__setattr__`, который бросает ошибку, и синтезированный `__hash__`, который рассчитывает на неизменность полей. Обычный потомок мог бы менять поля в обход этого, и хэш перестал бы быть надёжным. Поэтому решение о `frozen` принимают один раз на всю иерархию, а не на отдельный класс внутри неё. Если базовый класс задуман как основа для неизменяемых потомков, замораживают уже его.

## replace, asdict и astuple: копии, а не ссылки

`dataclasses.replace(obj, **changes)` не мутирует объект, а строит новый: он берёт текущие значения полей, накладывает изменения и вызывает `__init__` заново. Значит, `__post_init__` тоже отрабатывает снова, а поля с `init=False` передавать в `replace` нельзя (будет `ValueError`), они пересчитываются внутри. Для неизменяемых объектов `replace` это штатный способ «изменить» значение: получить копию с правкой.

`asdict(obj)` и `astuple(obj)` разворачивают dataclass в словарь и кортеж, причём **рекурсивно**: вложенные dataclass, списки, словари и кортежи копируются вглубь.

```python
from dataclasses import dataclass, asdict

@dataclass
class Line:
    sku: str
    qty: int

@dataclass
class Order:
    id: int
    lines: list

asdict(Order(1, [Line("a", 2), Line("b", 3)]))
```

Результат это `{'id': 1, 'lines': [{'sku': 'a', 'qty': 2}, {'sku': 'b', 'qty': 3}]}`: новые словари, новые списки, глубокая копия. В горячем цикле это не бесплатно: `asdict` большого дерева выполняет полный обход и копирование, а не отдаёт дешёвое представление.

## Частые ошибки

- **`self.x = ...` в `__post_init__` frozen-класса.** Прямое присваивание бросит `FrozenInstanceError`. Записывайте через `object.__setattr__(self, "x", value)`.
- **Ждать, что `frozen` заморозит вложенный список.** frozen неглубокий: `obj.items.append(...)` сработает, если `items` изменяемый. Для настоящей неизменности храните кортежи, а не списки.
- **Обязательное поле в потомке после дефолтного поля базы.** Это `TypeError` при определении класса. Пометьте поле `kw_only=True` или сделайте потомок целиком `kw_only`.
- **`asdict` в горячем пути.** Это глубокая копия. Если нужна лишь пара полей, обращайтесь к ним напрямую, без разворота всего объекта.

<Detail summary="Почему replace вызывает __init__ и __post_init__ заново">
`replace` намеренно не копирует объект байт в байт. Он собирает словарь аргументов из текущих полей, подменяет указанные и вызывает `type(obj)(**kwargs)`. Такой путь гарантирует, что новый объект пройдёт ту же валидацию и нормализацию, что и любой другой: если `__post_init__` обрезает диапазон или пересчитывает производное поле, копия получит те же инварианты. Обратная сторона: поля с `init=False` в аргументы `__init__` не входят, поэтому передать их в `replace` нельзя, они вычисляются заново.
</Detail>
