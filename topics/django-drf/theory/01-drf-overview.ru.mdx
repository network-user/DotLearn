# Почему голый Django устаёт раздавать JSON

Сервису каталога Ozon нужно отдавать товары не браузеру, а мобильному приложению - значит, не HTML, а JSON. На голом Django Вы пишете view, которая руками достаёт данные, руками сериализует их в `JsonResponse`, руками читает тело запроса через `json.loads(request.body)`, руками проверяет, что пришло, и руками решает, какой код вернуть при ошибке. Одна view - терпимо. Двадцать endpoint'ов, и половина файла - это повторяющийся разбор тела, одинаковые `try/except` вокруг `json.loads` и развесистые проверки полей, скопированные из соседней view.

Проблема не в том, что Django чего-то не умеет. Проблема в том, что JSON-API - это один и тот же набор шагов на каждом endpoint: распарсить вход независимо от формата, проверить данные, превратить объекты в примитивы, выбрать формат ответа, проставить статус. Django REST Framework вытаскивает эти шаги в переиспользуемые слои. К концу раздела Вы будете понимать, что даёт `APIView` поверх обычной view, почему `request.data` единообразен для JSON и форм, и как `Response` сам выбирает формат ответа, не заставляя Вас вызывать `json.dumps`.

## Ментальная модель: DRF - это слой между HTTP и вашей логикой

Обычная Django-view получает «сырой» `HttpRequest` и обязана вернуть `HttpResponse` с уже готовой строкой-телом. Всё, что между ними - разбор, валидация, сериализация - ваша забота. DRF вставляет между HTTP и вашим кодом тонкий слой: запрос приходит уже разобранным в `request.data`, а ответ Вы отдаёте питоновским словарём, который рендерер сам превратит в нужный формат.

Главное в этой модели: Вы перестаёте трогать байты. Вход - это `request.data`, словарь, собранный из тела независимо от того, прислал клиент JSON или форму. Выход - это `Response(data)`, где `data` - обычная питоновская структура, а уже формат (JSON, browsable HTML) выбирается фреймворком по заголовку `Accept`. Между ними ваша логика говорит на языке объектов, а не строк.

`APIView` - это класс-наследник джанговской `View`, который оборачивает входящий запрос в `Request` DRF, а на выходе ждёт `Response`. Метод называется по HTTP-глаголу: `get`, `post`, `put`, `patch`, `delete`. Внутри Вы уже не думаете о парсинге и рендеринге - только о логике.

## request.data: один вход для JSON и форм

В голом Django, чтобы прочитать JSON-тело, Вы пишете `json.loads(request.body)`, а чтобы прочитать данные формы - лезете в `request.POST`. Это два разных места для двух форматов. DRF прячет это за `request.data`.

```python
class CreateItem(APIView):
    def post(self, request):
        name = request.data["name"]
        return Response({"created": name})
```

Если клиент прислал `{"name": "шапка"}` с заголовком `Content-Type: application/json`, `request.data` будет `{"name": "шапка"}`. Если тот же клиент прислал HTML-форму с полем `name=шапка`, `request.data` будет ровно таким же словарём. Какой parser разобрал тело, ваш код не волнует - вход единообразен. Рядом живёт `request.query_params` - это аккуратное имя для query string (в Django это был `request.GET`, что путало: query string приходит и в POST-запросах).

## Response и content negotiation

`Response` принимает питоновские данные и откладывает выбор формата до последнего момента. Какой именно рендерер сработает, решает content negotiation - переговоры по заголовку `Accept`.

```python
from rest_framework.response import Response
from rest_framework import status

class Ping(APIView):
    def get(self, request):
        return Response({"pong": True}, status=status.HTTP_200_OK)
```

Если клиент прислал `Accept: application/json`, тот же `Response` отрендерится в JSON. Если запрос пришёл из браузера с `Accept: text/html`, сработает browsable-рендерер и вернёт человекочитаемую HTML-страницу с этим же ответом, формой для отправки запроса и кликабельными ссылками. Один и тот же код обслуживает и машину, и человека - это и есть browsable API, фирменная фича DRF для отладки руками.

<Callout type="tip">
Модуль `rest_framework.status` хранит именованные константы статусов: `status.HTTP_201_CREATED`, `status.HTTP_404_NOT_FOUND`. Они читаются лучше голых чисел: `201` легко перепутать с `200`, а `HTTP_201_CREATED` - нет.
</Callout>

## Краевые случаи

Несколько мест, где DRF ведёт себя не так, как ожидает новичок из голого Django.

**`request.data` нельзя читать как поток дважды теми же глазами.** DRF разбирает тело один раз и кэширует результат, поэтому многократное обращение к `request.data` безопасно - но обращаться к `request.body` после этого уже поздно, поток прочитан. Выберите один уровень: либо `request.data`, либо сырой `body`, не оба.

**`Response` без явного статуса - это всегда 200.** Если Вы создаёте ресурс через `POST` и забыли `status=status.HTTP_201_CREATED`, клиент получит `200 OK` вместо `201 Created`. Семантически это враньё: ресурс создан, а код говорит «просто ок». Статус - часть контракта, его задают осознанно.

**Browsable API виден только в DEBUG-режиме мышления.** Он включён по умолчанию и удобен в разработке, но на проде его обычно отключают или закрывают - это лишняя поверхность и лишний рендеринг. Удобство отладки не должно протекать в боевой ответ.

<Callout type="warning">
`APIView` не делает за Вас валидацию входа. `request.data["name"]` бросит `KeyError`, если поля нет. Проверка данных - работа сериализатора из следующего раздела, а не самой view. `APIView` лишь даёт удобный вход и выход.
</Callout>

## Типичные ошибки

- **Возврат `JsonResponse` из `APIView`.** Внутри DRF-view нужно возвращать `Response`, а не джанговский `JsonResponse`. С `JsonResponse` content negotiation и browsable API не сработают - Вы вернётесь к ручному рендерингу, ради ухода от которого и взяли DRF.
- **Чтение тела через `json.loads(request.body)` внутри `APIView`.** Это игнорирует parser и ломается на формах. `request.data` уже сделал разбор - используйте его.
- **Голые числа статусов.** `return Response(data, status=204)` работает, но `status.HTTP_204_NO_CONTENT` самодокументируется и страхует от опечатки в числе.
- **Ожидание, что `APIView` сам проверит данные.** Он не проверит. Без сериализатора `request.data` - это просто словарь, каким его прислал клиент, со всеми возможными пропусками и лишними полями.

<Detail summary="Откуда у DRF свой Request, если у Django уже есть HttpRequest">

DRF не подменяет джанговский `HttpRequest`, а оборачивает его. Класс `rest_framework.request.Request` хранит исходный `HttpRequest` внутри и добавляет поверх него ленивые свойства: `request.data` запускает подходящий parser при первом обращении и кэширует результат, `request.query_params` - это просто более честное имя для `request.GET`. Выбор parser'а зависит от заголовка `Content-Type` и списка `parser_classes` у view (по умолчанию JSON, форма и multipart). Симметрично на выходе `Response` хранит несериализованные `data` до самого конца обработки: рендерер выбирается по заголовку `Accept` уже в финальном middleware-слое, поэтому одна и та же view может отдать и JSON, и HTML, ничего не меняя в коде. Этот отложенный рендеринг - причина, по которой Вы возвращаете объекты, а не строки.

</Detail>
