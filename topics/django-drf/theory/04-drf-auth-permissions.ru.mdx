# Два вопроса, которые путают даже опытные

К endpoint'у личного кабинета на Госуслугах приходит запрос. Прежде чем отдать данные, сервер должен ответить на два вопроса, и порядок между ними не случаен. Первый: «кто это вообще?» - установить личность отправителя по токену или сессии. Второй, и только потом: «а этому конкретному ему можно?» - можно ли данному пользователю выполнить это действие над этим объектом. Смешать вопросы легко, и тогда рождаются классические баги: чужой профиль открывается, потому что забыли спросить «можно ли», или анонимный запрос получает `403` вместо честного «представьтесь».

DRF держит эти два вопроса в двух разных слоях, и слои всегда идут в одном порядке. Сначала **аутентификация** (authentication) выясняет, кто отправитель, и кладёт результат в `request.user`. Затем **права** (permissions) смотрят на этого `request.user` и решают, пропустить запрос или отказать. К концу раздела Вы будете уверенно различать классы аутентификации и классы прав, понимать, почему `401` и `403` - это про разные вопросы, и где живёт проверка доступа к конкретному объекту.

## Ментальная модель: проходная с двумя постами

Представьте проходную с конвейером. Запрос въезжает и проходит посты по очереди. Первый пост - аутентификация: он не пускает и не выгоняет, он лишь устанавливает личность, прикалывает к запросу бейдж `request.user`. Анонима он тоже пропускает дальше - просто с бейджем «гость» (`AnonymousUser`). Второй пост - права: он смотрит на бейдж и решает, пускать ли к этой двери. Третий, по желанию, - throttling, считающий, не слишком ли часто этот посетитель ходит. Только пройдя все посты, запрос доходит до view.

Главное в этой модели - разделение труда. Аутентификация **не решает** про доступ, она только устанавливает личность; права **не устанавливают** личность, они только судят уже установленную. Поэтому анонимный запрос - это нормальное состояние после первого поста, а не ошибка: ошибкой (или не ошибкой) его сделает уже пост прав.

## Аутентификация: кто отправитель

Класс аутентификации читает что-то из запроса - заголовок, cookie - и по этому устанавливает `request.user` и `request.auth`. DRF пробует классы из списка по очереди; первый, кто опознал отправителя, побеждает. Если не опознал никто, `request.user` становится `AnonymousUser`.

- **SessionAuthentication** - на сессионной cookie Django. Подходит, когда фронт и API на одном домене (тот же браузер, что и для обычных страниц). Требует CSRF-защиты на небезопасных методах.
- **TokenAuthentication** - на статичном токене в заголовке `Authorization: Token <ключ>`. Токен выдаётся один раз и живёт, пока его не отозвать. Просто, но токен не истекает сам.
- **JWT через SimpleJWT** - на самоподтверждаемом access-токене (`Authorization: Bearer <jwt>`), который несёт срок жизни внутри себя и не требует похода в базу за каждым запросом. Пара access/refresh: короткий access и длинный refresh для обновления.

```python
class Profile(APIView):
    authentication_classes = [TokenAuthentication]

    def get(self, request):
        return Response({"user": request.user.username})
```

После того как класс отработал, `request.user` указывает на конкретного пользователя. Заметьте: на этом шаге ещё никто не спросил, можно ли ему сюда. Аутентификация лишь представила гостя.

## Права: кому что можно

Класс прав получает `request` (а значит, и установленный `request.user`) и возвращает `True`/`False`. Их два метода - на двух уровнях.

- **`has_permission(self, request, view)`** - проверка на уровне всего endpoint'а, до выбора объекта. «Аутентифицирован ли вообще?», «у пользователя роль staff?». Срабатывает для любого запроса к view.
- **`has_object_permission(self, request, view, obj)`** - проверка на уровне конкретного объекта. «Этот заказ принадлежит этому пользователю?». Вызывается, когда view достаёт один объект через `get_object()`.

Встроенные классы покрывают частые случаи: `AllowAny` (пускать всех), `IsAuthenticated` (только опознанных), `IsAdminUser` (только `is_staff`), `IsAuthenticatedOrReadOnly` (читать всем, писать опознанным).

```python
from rest_framework.permissions import IsAuthenticated

class Profile(APIView):
    permission_classes = [IsAuthenticated]

    def get(self, request):
        return Response({"user": request.user.username})
```

Кастомный класс наследует `BasePermission` и переопределяет нужный метод.

```python
from rest_framework.permissions import BasePermission

class IsOwner(BasePermission):
    def has_object_permission(self, request, view, obj):
        return obj.owner_id == request.user.id
```

`IsOwner` пускает к объекту только его владельца. Заметьте: эта проверка - на уровне объекта, поэтому ей нужен `obj`, и срабатывает она лишь когда view достаёт конкретную запись.

## Порядок: сначала «кто», потом «можно ли»

Порядок не настраиваемый и не случайный. Аутентификация всегда раньше прав, потому что права судят `request.user`, а его ещё надо установить. Если бы права шли первыми, им было бы нечего судить.

<Steps>
  <li>Классы аутентификации по очереди пытаются опознать отправителя; устанавливается <code>request.user</code> (возможно, <code>AnonymousUser</code>).</li>
  <li>Для каждого класса прав вызывается <code>has_permission(request, view)</code>; хоть один <code>False</code> - отказ.</li>
  <li>Если view достаёт объект через <code>get_object()</code>, для него вызывается <code>has_object_permission(request, view, obj)</code>.</li>
  <li>Только пройдя всё это, запрос попадает в тело действия (<code>get</code>, <code>retrieve</code> и т.д.).</li>
</Steps>

<PullQuote>
Аутентификация отвечает «кто ты», права - «что тебе можно»; перепутать порядок значит судить того, кого ещё не опознал.
</PullQuote>

## 401 против 403

Два кода ошибки - про два разных вопроса, и DRF выбирает между ними осознанно.

DRF выбирает код умно: если запрос анонимный и есть схема аутентификации, обещающая залогинить (есть `WWW-Authenticate`), он отдаёт `401` - «попробуй представиться». Если пользователь уже опознан, но права сказали `False`, отдаёт `403` - «представляться поздно, тебе просто нельзя».

## Краевые случаи

**Пустой `permission_classes` - это не «запретить всем».** Пустой список означает «никаких проверок прав» - то есть пускать всех, как `AllowAny`. Чтобы закрыть endpoint, нужен явный `IsAuthenticated`, а не пустой список. Молчание здесь читается как «разрешено».

**`has_object_permission` не вызывается для списков.** Он срабатывает только когда view дёргает `get_object()` за одной записью. В `list` объектов много, и DRF не зовёт объектную проверку для каждого - фильтровать выдачу по правам владельца нужно через queryset (`get_queryset`), а не надеяться на `has_object_permission`.

**`IsAuthenticated` ничего не знает о владении.** Он пускает любого залогиненного - и владельца заказа, и постороннего пользователя. «Только свой объект» - это уже `has_object_permission`, отдельная проверка. Аутентифицирован не значит уполномочен на этот объект.

<Callout type="warning">
Surprise: проверки прав на уровне объекта легко обойти, если доставать запись вручную, минуя `get_object()`. Если во view написать `Order.objects.get(pk=pk)` напрямую, `has_object_permission` **не вызовется** - его дёргает именно `get_object()`. Объектные права работают только тогда, когда объект получают штатным путём.
</Callout>

## Типичные ошибки

- **Расчёт на `IsAuthenticated` для защиты чужих объектов.** Он проверяет лишь факт логина, не владение. Доступ «только к своему» - это `has_object_permission` или фильтрация queryset'а, иначе любой залогиненный увидит чужое.
- **Путаница `401` и `403` в своей логике.** Не возвращайте `403` анониму - это говорит «тебе нельзя», хотя честный ответ «представься» (`401`). И не возвращайте `401` опознанному - логин он уже прошёл.
- **Объектная фильтрация списков через `has_object_permission`.** Для `list` он не вызывается. Прячьте чужие записи в `get_queryset()`, отдавая только `request.user`-овские.
- **Пустой `permission_classes` в надежде «закрыть».** Пустой список открывает endpoint всем. Закрытие - это явный класс прав, а не отсутствие классов.

<Detail summary="Где DRF берёт классы по умолчанию и как их переопределить точечно">

И `authentication_classes`, и `permission_classes` имеют глобальные значения по умолчанию в `REST_FRAMEWORK` settings (`DEFAULT_AUTHENTICATION_CLASSES`, `DEFAULT_PERMISSION_CLASSES`). Если на view ничего не указано, берутся глобальные. Указанный на view список **полностью заменяет** глобальный для этой view, а не дополняет его - поэтому, переопределяя права на одном endpoint'е, легко случайно убрать базовую аутентификацию, если забыть, что списки независимы. Хорошая практика - задать безопасный глобальный дефолт (`IsAuthenticated`) и точечно ослаблять его на публичных endpoint'ах через `AllowAny`, а не наоборот: дефолт «закрыто», исключения «открыто». Тогда забытый endpoint остаётся закрытым, а не открытым. Метод `get_permissions()` позволяет вернуть разные классы для разных действий одного ViewSet - например, `list` открыть всем, а `create` закрыть.

</Detail>
