# Один список и двадцать лишних запросов

Откройте кабинет продавца на Ozon и посмотрите на список последних заказов: двадцать строк, в каждой имя покупателя и число позиций в корзине. Список отдаёт ViewSet, а сериализатор для каждого заказа лезет за именем покупателя и пересчитывает позиции. Кажется, что база отвечает одним аккуратным запросом. На деле их двадцать один: один за самим списком заказов и ещё по одному на каждый заказ, чтобы догрузить покупателя.

Пока строк двадцать, лишние запросы теряются в шуме. Когда заказов станет двести, страница пошлёт больше двухсот запросов и начнёт заметно подтормаживать, а время ответа будет расти линейно вместе с числом строк. Это классическая проблема N+1. В DRF она возникает особенно легко: сериализатор охотно ходит по связям объекта и не спрашивает, во сколько запросов это обойдётся. К концу раздела вы научитесь замечать лишний запрос там, где его не видно в коде, и убирать его двумя инструментами ORM.

## Ментальная модель: один список и россыпь точечных запросов

QuerySet в Django ленивый: пока вы не начали перебирать результат, к базе никто не обращается. Первый запрос забирает список заказов одной выборкой. Но связанные объекты в эту выборку не попали, ведь про покупателя и позиции заранее никто не просил. Как только сериализатор берётся за очередную строку и читает `order.customer`, ORM обнаруживает, что данных под рукой нет, и молча выполняет отдельный точечный запрос за этим покупателем. Строк N, значит и добавочных запросов N. Итого один запрос за списком плюс N за связями, отсюда и название.

Сериализатор не делает ничего злонамеренного: он честно читает атрибут, а ленивая связь честно подгружает данные по требованию. Беда в том, что требование повторяется в цикле по каждой строке. Если сериализатор трогает две разные связи, точечных запросов станет вдвое больше.

## Где N+1 прячется именно в DRF

В сериализаторе лишний запрос почти всегда спрятан за безобидным на вид полем. Опасны четыре места:

- **Вложенный сериализатор** `items = ItemSerializer(many=True)`: чтобы отдать позиции каждого заказа, DRF переберёт `order.items`, и это отдельный запрос на каждый заказ.
- **`SerializerMethodField`**, чей метод трогает связь: вызов `obj.customer.name` или `obj.items.count()` внутри `get_*` повторяется для каждой строки.
- **`StringRelatedField` и `PrimaryKeyRelatedField`** при обходе связи: чтобы показать строковое представление или собрать список id, DRF идёт по связанным объектам.
- **Пагинация не спасает.** Она лишь режет выборку до одной страницы, но на этой странице N+1 остаётся в полном объёме: двадцать строк дадут свои двадцать добавочных запросов.

```python
class OrderSerializer(serializers.ModelSerializer):
    customer_name = serializers.CharField(source="customer.username")
    item_count = serializers.SerializerMethodField()

    class Meta:
        model = Order
        fields = ["id", "customer_name", "item_count"]

    def get_item_count(self, obj):
        return obj.items.count()
```

Здесь на одну строку приходится два обращения к базе: одно за `customer`, другое на `obj.items.count()`. Двадцать заказов превращаются в 1 + 20 + 20 запросов. Поле выглядит невинно, а цена спрятана в его реализации.

## Два инструмента: select_related и prefetch_related

ORM умеет загрузить связи заранее, одним заходом. Инструментов два, и они не взаимозаменяемы: выбор диктует тип связи.

Правило простое: спросите себя, сколько объектов на другом конце связи. Объект один? JOIN дешёвый, берите `select_related`. Объектов много? JOIN размножил бы строки заказа, поэтому Django делает отдельный запрос и склеивает результат в памяти, это `prefetch_related`.

## Куда класть оптимизацию

Предзагрузке не место в сериализаторе. Сериализатор описывает форму данных, а не способ достать их из базы. Правильное место это `queryset` (или метод `get_queryset()`) у ViewSet, там, где рождается выборка.

```python
class OrderViewSet(viewsets.ModelViewSet):
    queryset = Order.objects.select_related("customer").prefetch_related("items")
    serializer_class = OrderSerializer
```

Если выборка зависит от запроса или пользователя, тот же вызов уходит в `get_queryset()`:

```python
def get_queryset(self):
    return (
        Order.objects
        .select_related("customer")
        .prefetch_related("items")
        .filter(seller=self.request.user)
    )
```

Сам сериализатор при этом не меняется: он по-прежнему читает `order.customer` и перебирает `order.items`, только теперь данные уже лежат в памяти и точечных запросов не будет.

## Тонкая настройка: Prefetch, аннотации, only

Иногда нужна не вся связь, а её отфильтрованный или отсортированный срез. Для этого есть класс `Prefetch`, который принимает готовый `queryset` и умеет складывать результат в отдельный атрибут через `to_attr`:

```python
from django.db.models import Prefetch

queryset = Order.objects.prefetch_related(
    Prefetch(
        "items",
        queryset=Item.objects.filter(price__gt=1000),
        to_attr="expensive_items",
    )
)
```

Теперь у каждого заказа появится список `order.expensive_items`, собранный тем же вторым запросом, без похода в базу на каждую строку.

<Callout type="warning">
Предзагрузка кэширует ровно тот набор, который вы попросили. Если внутри метода сериализатора снова позвать `obj.items.filter(...)` или `obj.items.order_by(...)`, Django посчитает это новым запросом и сходит в базу заново для каждой строки. N+1 вернётся, хотя `items` уже предзагружены. Пользуйтесь предзагруженным списком как есть, а фильтрацию и сортировку задавайте заранее через `Prefetch`.
</Callout>

Счёт по связи это отдельная ловушка. `obj.items.count()` в методе сериализатора шлёт `SELECT COUNT(*)` на каждую строку. Замените его на аннотацию, которая посчитает всё одним запросом вместе со списком:

```python
from django.db.models import Count

queryset = Order.objects.annotate(item_count=Count("items"))
```

Тогда `item_count` приедет как обычное поле, и `SerializerMethodField` вам вообще не понадобится. По той же логике `only("id", "title")` и `defer("body")` сокращают набор колонок, а вложенная предзагрузка `select_related("customer__city")` достаёт покупателя и его город за один JOIN на любую глубину связей «к одному».

## Как поймать N+1

Проблему не видно глазами, её надо измерять числом запросов.

- **`assertNumQueries(n)`** в тестах фиксирует ожидаемое число запросов: если кто-то добавит поле со связью, тест упадёт на превышении. Это лучший способ не допустить регресс.
- **`connection.queries`** в шелле показывает список выполненных запросов после обращения к endpoint (при `DEBUG=True`).
- **django-silk** и **Django Debug Toolbar** показывают запросы каждого ответа в браузере, включая повторяющиеся, и сразу подсвечивают дубли.

Чтению плана и профилированию медленных запросов посвящена отдельная тема про профилирование запросов в Django; там разобрано глубже, как читать вывод этих инструментов и что делать дальше. Здесь важно запомнить одно: пока вы не считаете запросы, вы не знаете, есть ли у вас N+1.

## Типичные ошибки

- **Оптимизация в сериализаторе.** Вызов `select_related` внутри `SerializerMethodField` бесполезен: выборка уже случилась во view, сериализатор работает с готовыми объектами. Предзагрузку задают у queryset.
- **select_related для связи «ко многим».** `select_related("items")` для обратного ForeignKey упадёт с ошибкой: JOIN размножил бы строки заказа. Такие связи только через `prefetch_related`.
- **`.filter()` или `.count()` по связи в цикле или методе.** Любой новый запрос по связи идёт мимо кэша предзагрузки и воскрешает N+1. Фильтр задают через `Prefetch`, счёт через `Count`.
- **Проверка на десяти строках.** С горсткой тестовых данных N+1 незаметен. Меряйте число запросов тестом с `assertNumQueries`, а не на глаз.

<Detail summary="Почему select_related падает на связи «ко многим», а prefetch_related нет">

`select_related` строит один SQL-запрос с JOIN. Для связи «к одному» это безопасно: у заказа ровно один покупатель, строка результата не размножается. Для связи «ко многим» JOIN дал бы по строке заказа на каждую позицию, и одна запись расползлась бы на десятки дубликатов; Django на такую связь `select_related` просто не пускает и бросает ошибку. `prefetch_related` обходит проблему иначе: сначала первый запрос за заказами, затем второй запрос за всеми их позициями одним `WHERE order_id IN (...)`, и только потом Python раскладывает позиции по заказам в памяти. Два запроса вместо N+1 и без всякого размножения строк. Поэтому «к одному» это JOIN, а «ко многим» это второй запрос со склейкой.

</Detail>
