# When an index silently stops working

You've set up a perfect index on the `email` column, and the query still crawls. You dig into the code and find the culprit: `WHERE lower(email) = 'anna@mail.ru'`. Someone wrapped the column in `lower()` just in case, to make the match case-insensitive, and that alone killed the index. The database indexed `email`, not `lower(email)`, and a value computed on the fly is useless for a sorted lookup. There's no error, no warning; the query runs and returns the right answer. It's just slow, a full scan.

This is the sneakiest class of index problems: the index exists, looks like a good fit, and quietly goes unused. By the end of this chapter you'll instantly recognize the conditions that neutralize an index: a function or arithmetic over a column, a `LIKE` with a leading wildcard, an `OR` across different columns, implicit type coercion, low selectivity. And you'll know how to rewrite each one so the index works again.

## An index wants a bare column

Keep one mental model in mind and almost every case becomes obvious. An index stores a column's values exactly as they are, laid out in order. A sorted lookup only works when you compare the column itself to a value. The moment something computed from the column (a function, arithmetic, a substring) appears on the comparison side, the database has to evaluate that expression for every single row, and evaluating something for every row is exactly what a full scan is. Such a condition has a name: non-sargable.

Why can't the database peek inside a function and still use the index? Because in the general case it has no way of knowing whether the function preserves value order. Lowercasing reorders strings unpredictably from the engine's point of view: in the original index, `Anna` and `anna` sit far apart, but after `lower()` they collapse into the same value. The planner won't try to reason about what an arbitrary function does to order, so it honestly falls back to a scan. There are only two ways to get the index back: remove the function from the condition, or index the function's result directly, ahead of time.

## LIKE: a prefix works, a leading wildcard doesn't

With strings, this model gives a neat consequence. An index stores strings sorted alphabetically, which means sorted by their beginning. That's why `email LIKE 'anna%'` uses the index beautifully: the known prefix `anna` defines a contiguous range, and every string starting with `anna` sits together. But `email LIKE '%mail.ru'` can't use the index: a leading wildcard means the string can start with anything, and you can't build a range around an unknown beginning.

The practical rule: searching by the start of a string is indexable, searching by a substring or the end is not. If you need to search by the end of a string (say, by email domain), that's a signal your data model deserves a fix: add a separate domain column, use a full-text index, or store the string reversed and search by prefix on that instead. The same logic covers searching for a substring anywhere in the middle: the moment the target can start at any character, a sorted index is useless, and you need a different tool, such as a full-text or trigram index.

## OR across different columns

The condition `WHERE user_id = 5 OR status = 'paid'` stumps the engine: one index covers one column and can't check two unrelated conditions in a single pass. This usually ends in a full scan, even if both columns are individually indexed. The fix is to split it into two branches with `UNION`:

```sql
SELECT * FROM orders WHERE user_id = 5
UNION
SELECT * FROM orders WHERE status = 'paid';
```

Now each branch is a separate, single-column query, and each one uses its own index, while `UNION` merges the rows and, as a side effect, removes duplicates (rows that match both conditions). What looked like one non-sargable condition has become two sargable ones.

## Implicit coercion and low selectivity

Two quiet traps to close with. The first is implicit type coercion. If the `phone` column is declared `TEXT` and you write `WHERE phone = 74951234567` (a number), the database inserts a conversion to compare text with a number, and that conversion can knock the index out of use. Compare a column to a value of its own type: `WHERE phone = '74951234567'`. SQLite's flexible typing makes this worse: a text column stores `'0042'` and `'42'` as different strings, and comparing against a number produces a surprising result even before any index is involved. An explicit string literal clears up both the ambiguity and the lost index at once.

The second is low selectivity. When a condition on an indexed column selects a large share of the table, the planner deliberately ignores the index. Reading half the rows through an index, jumping from reference to reference, costs more than reading that same half straight through. To make that call correctly, the planner needs fresh statistics about how values are distributed, and the `ANALYZE` command is what refreshes them. On a table that changed drastically after a bulk load, stale statistics can push the planner toward a bad plan, and a one-off `ANALYZE` often fixes the query without adding a single new index.

## Common mistakes

**Wrapping a column in a function out of habit.** `date(created_at) = '2024-02-15'` looks tidy but kills the index. The same filter written as a half-open range, `created_at >= '2024-02-15' AND created_at < '2024-02-16'`, leaves the column bare.

**Comparing a TEXT column to a number.** Implicit coercion quietly cancels the index. Always compare against a string literal when the column is text.

**Expecting a miracle from an index on a low-selectivity column.** An index on gender, a three-value status, or an `is_active` flag is rarely used: the condition selects too much. Before creating an index, estimate the fraction of rows the condition actually filters out.

**A leading wildcard in a substring search.** `LIKE '%Smith%'` looks like a convenient universal search, but on a large table it guarantees a full scan. That's because the match can start at any character, so the index's ordering doesn't narrow the candidates at all.

<Detail summary="Expression indexes and partial indexes">

Sometimes a function over a column is genuinely needed in a query, and there's no way to rewrite around it. In that case, you index the expression itself rather than the column. Case-insensitive email search is the classic example: `CREATE INDEX idx_users_email_lower ON users(lower(email))`, and now `WHERE lower(email) = ?` uses the index, because the index holds exactly `lower(email)`. The second tool is a partial index with a condition: `CREATE INDEX idx_orders_pending ON orders(created_at) WHERE status = 'pending'`. It indexes only the rows with `status = 'pending'`, so it stays small and cheap to maintain, which is ideal when hot queries always filter on one rare status value. Both techniques exist in SQLite and in Postgres.

</Detail>
