# B-trees under the hood: why searching a million rows takes three reads

In the previous chapter, the index found the right rows without ever looking at the rest. Now it's time to answer honestly how. The naive guess goes like this: since the index values are sorted, the database just does a binary search over them. For a million rows that's around twenty comparisons. Fast, but the guess hides a problem: the data lives on disk, not in an in-memory array, and every binary-search jump to a random spot is a separate disk read. Twenty random reads for a single lookup is already expensive, and a million rows is a modest size for a database.

A real index is cleverer than that. It finds the same million rows in three or four disk accesses, not twenty. The secret is a structure called a B-tree: a balanced tree where each node isn't a single value but an entire page holding hundreds of keys. By the end of this chapter you'll understand why such a tree's height is a logarithm with a huge base, why it's equally good at point lookups and range scans, why an `ORDER BY` on an indexed column needs no separate sort, and why a hash index, for all its speed, can't replace a B-tree.

## A tree of pages, not numbers

A database doesn't read or write the disk one value at a time; it works in fixed-size pages (4096 bytes by default in SQLite). That page is its unit of exchange: even to fetch a single number, it pulls an entire page off disk. A B-tree is built around that fact. Every node in the tree is one page, and a page holds not one key but hundreds: sorted keys plus references to the pages one level down.

The search goes top to bottom. The topmost page (the root) holds a handful of separator values and tells you which lower page to descend into. That page, in turn, sends you a level lower still. After a few hops you land on the bottommost page (a leaf), which holds the key you're after and a reference to the table row.

## Height is a logarithm with a large base

The key metric for a B-tree is fanout: how many references fit on one page. If a page holds around two hundred keys, each level you descend narrows the search space not by half, as in binary search, but by a factor of two hundred. That's where the height comes from: it equals the logarithm of the row count, with the fanout as the base.

Do the math on your fingers. One level above the leaves covers 200 pages, two levels is 200 squared, around forty thousand, three levels is eight million. So a tree three or four pages tall can address millions of rows. The non-obvious consequence: height grows agonizingly slowly.

The chart above makes the main point: raising the height by one level requires multiplying the row count by hundreds. A table with a million rows and one with a billion differ in point-lookup cost by only one or two extra reads. That's exactly why indexes scale: they barely notice data growth.

## One mechanism for both points and ranges

You've already seen point lookups: descending from root to leaf for a specific key. But a B-tree has a second ability that hash indexes lack: ranges. The tree's leaves aren't scattered; they're linked into a sorted chain. To run `amount BETWEEN 500 AND 800`, the engine descends to the first leaf holding a value at or above 500, then simply walks the leaf chain in order, handing back rows until the value passes 800. Not a single wasted leaf.

That same sortedness gives you `ORDER BY` for free. If a query asks for rows in the order of an indexed column, the engine doesn't need to collect every row and sort it: it just reads the leaf chain, which is already in ascending key order. You can spot this in the plan by the absence of `USE TEMP B-TREE FOR ORDER BY`, a line that only shows up when the order had to be produced by a separate sort.

Try it live. Below is an `orders` table with fifteen hundred rows and an index on `amount`. The query takes a range and asks for order on that same column. Run it and read the plan: you'll see `SEARCH ... USING INDEX` and no temporary sort. Now delete the `CREATE INDEX` line and run it again: the plan switches to `SCAN`, and `ORDER BY` picks up a `USE TEMP B-TREE`.

## Why not hash

If a hash table finds a value in one step on average, and a B-tree needs a logarithm's worth of steps, why do indexes default to trees? Because a hash table trades order for speed. It scatters keys into buckets by hash value, so keys that are neighbors in value end up at opposite ends of the structure. A hash index handles point equality beautifully, but ranges, sorting, and prefix search are simply impossible for it: it has no notion of "the next key."

The practical takeaway is simple: the universal choice is a B-tree, which is exactly why a plain `CREATE INDEX` builds a tree. Hash only makes sense where queries are pure equality and the column never appears in a sort or a range.

## Common mistakes

**Treating an index lookup as instant.** A dictionary lookup in memory is `O(1)`, while descending a B-tree is `O(log n)` with a disk access at every level. The difference between a million rows and a billion is small, but it's real, and on a very hot path you can sometimes feel it.

**Expecting a hash index to always be faster.** It's only faster on strict equality, and only when the data is already in memory. The very first query with a range or a sort will run into its inability to produce order, forcing you back to a tree.

**Assuming ORDER BY always means a separate sort.** If the requested order matches a suitable index's column, there's no sort at all: rows arrive already ordered. A `USE TEMP B-TREE FOR ORDER BY` showing up in the plan is a signal that no index backs that sort.

<Detail summary="B-trees, B+trees, and why leaves are linked into a list">

Strictly speaking, relational databases use a variant called a B+tree. The difference: all the real data (keys with row references) lives only in the leaves, while internal pages hold nothing but navigation separators. This has two effects. First, internal pages are more compact, so more separators fit, fanout is higher, and the tree ends up even shorter. Second, the leaves are linked into a sorted list, so a range scan after descending is just walking the chain with no trips back to the root. That chain is exactly what makes both `BETWEEN` and a sort-free `ORDER BY` possible. A classic B-tree would store data in internal nodes too, and a range scan would cost more.

</Detail>
