# Когда async ускоряет сервис, а когда нет

FastAPI позволяет писать обработчики двумя способами: обычным `def` и асинхронным `async def`. Соблазн прост - «async быстрее, пишу всё через `async def`». Но это полуправда, которая на проде оборачивается зависшим сервисом. Асинхронность ускоряет не вычисления, а ожидание; и если внутри `async def`-обработчика поставить блокирующий вызов, можно сделать хуже, чем если бы вы вообще не трогали `async`.

Чтобы выбирать между `def` и `async def` осознанно, нужна модель того, что происходит под капотом: что такое event loop, чем «ожидание» отличается от «работы», и почему один поток может обслуживать тысячи одновременных запросов, пока они чего-то ждут. К концу раздела Вы будете уметь по природе обработчика - считает он или ждёт - выбрать правильную форму и распознать вызов, который заблокирует весь сервер.

## Ментальная модель: один повар, много кастрюль

Представьте кухню с одним поваром. Если он варит суп и просто стоит, глядя на кастрюлю, пока она кипит, - все остальные заказы ждут. Если же он, поставив суп на огонь, идёт резать салат для следующего заказа и возвращается к супу, когда тот готов, - за то же время он обслужит больше гостей. Повар тут один (один поток), а «поставил на огонь и отошёл» - это `await`.

Event loop - это цикл, который выполняет одну корутину, пока она не дойдёт до `await` на операции ожидания (запрос к базе, обращение к другому сервису по сети). В точке `await` корутина говорит «я жду, займись другими» - и loop переключается на следующую готовую задачу. Когда ожидаемое завершилось, loop вернётся и продолжит с того же места.

Из этой модели следует главное правило. Async выигрывает, когда обработчик **много ждёт** ввода-вывода: пока он ждёт, loop делает другую полезную работу. Async ничего не даёт и даже вредит, когда обработчик **много считает**: вычисление не содержит точек `await`, поэтому loop застревает на нём и не может переключиться - все остальные запросы стоят в очереди.

## `def` против `async def` в FastAPI

FastAPI умно обходится с обоими видами обработчиков, и это снимает часть опасности.

```python
@app.get("/async")
async def handler_async():
    data = await fetch_from_db()
    return data

@app.get("/sync")
def handler_sync():
    data = blocking_db_call()
    return data
```

`async def`-обработчик loop выполняет прямо в основном потоке: на каждом `await` он уступает управление. `def`-обработчик (обычный, синхронный) FastAPI запускает в отдельном пуле потоков, чтобы его блокирующий код не заморозил loop. Поэтому синхронный обработчик с блокирующим вызовом - безопасный выбор: он не мешает остальным. А вот блокирующий вызов внутри `async def` - беда: он выполняется в самом потоке loop и замораживает всё.

<Callout type="warning">
Правило выбора: если внутри есть `await` на настоящих асинхронных I/O-операциях - пишите `async def`. Если используете обычные блокирующие библиотеки (синхронный драйвер БД, `requests`, `time.sleep`) - пишите `def` и пусть FastAPI вынесет его в пул потоков. Не смешивайте: блокирующий вызов в `async def` - худший из вариантов.
</Callout>

## Главная ловушка: блокировка внутри async

Покажем разницу на самом наглядном случае - паузе.

```python
import asyncio, time

@app.get("/good")
async def good():
    await asyncio.sleep(1)
    return {"ok": True}

@app.get("/bad")
async def bad():
    time.sleep(1)
    return {"ok": True}
```

`await asyncio.sleep(1)` - это «поставил на огонь и отошёл»: на эту секунду loop свободен и обслуживает другие запросы. `time.sleep(1)` внутри `async def` - это «стоит и смотрит на кастрюлю»: целую секунду loop заблокирован в потоке корутины, и ни один другой запрос не двигается. Сто одновременных запросов к `/bad` выполнятся последовательно, по секунде каждый; сто запросов к `/good` уложатся примерно в ту же одну секунду суммарно.

Тот же эффект даёт любая тяжёлая чисто вычислительная задача в `async def`: разбор гигантского файла, перемножение матриц, хеширование пароля медленным алгоритмом. У этих операций нет точек `await`, поэтому loop на них застревает. Их место - в синхронном `def`-обработчике (пул потоков) либо в вынесенном фоновом исполнителе.

## Краевые случаи

**`async def` без единого `await` бессмыслен, но не вреден.** Если в асинхронном обработчике нет ни одного `await`, он отработает как обычная функция в потоке loop, просто без переключений. Вреда нет, пока внутри нет блокировки; но и пользы от `async` тоже ноль - честнее объявить его `def`.

**Параллелизм против конкурентности.** Один event loop в одном потоке даёт **конкурентность** (запросы перекрываются по времени), но не **параллелизм** (одновременное исполнение на разных ядрах). Чтобы задействовать несколько ядер, запускают несколько процессов-воркеров (например, через Uvicorn/Gunicorn). Async ускоряет I/O-bound нагрузку; CPU-bound нагрузку масштабируют процессами, а не корутинами.

**Забытый `await` возвращает корутину, а не результат.** Вызов `fetch_from_db()` без `await` вернёт объект корутины, а не данные. Это не ошибка на месте, а тихий баг: дальше по коду вы оперируете «обещанием», а не значением. Линтеры обычно ловят такое, но привычка проверять `await` у каждого асинхронного вызова надёжнее.

## Типичные ошибки

- **Делают обработчик `async def` «для скорости» и зовут внутри синхронную библиотеку.** `requests.get` или синхронный драйвер БД внутри `async def` блокируют loop. Либо берите асинхронный аналог и `await`, либо оставляйте обработчик синхронным `def`.
- **Кладут тяжёлое вычисление в `async def`.** CPU-bound код без `await` замораживает все остальные запросы на время счёта. Выносите его в `def`-обработчик или в отдельный исполнитель.
- **Забывают `await`.** Без него вызов корутины не выполняется, а возвращается сам объект корутины. Симптом - данные, которые «почему-то» оказались странным объектом, или предупреждение «coroutine was never awaited».

<Detail summary="Почему один поток - это не медленно для I/O">

Веб-сервис чаще ждёт, чем считает: он ждёт ответа базы, ответа платёжного шлюза, чтения с диска. В традиционной модели «поток на запрос» каждый ждущий запрос держит поток, который ничего не делает, но занимает память (стек) и нагружает планировщик переключениями контекста. Тысяча ждущих запросов - тысяча почти простаивающих потоков. Event loop ломает эту связь: ожидание не держит поток, оно лишь регистрирует «разбуди меня, когда придёт ответ» и освобождает loop. Поэтому один поток с loop обслуживает тысячи одновременных I/O-bound соединений на скромной памяти. Цена - дисциплина: ни одна операция в потоке loop не должна блокировать его надолго, иначе рушится весь выигрыш.

</Detail>
