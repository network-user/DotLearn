# Модель, которая проверяет данные за вас

Без проверки входных данных любой обработчик превращается в минное поле: клиент пришлёт `price` строкой, забудет обязательное поле, передаст `email` без собаки - и упадёте Вы, где-нибудь глубоко внутри бизнес-логики, с невнятным `TypeError`. Классический ответ - писать руками гору `if`-ов в начале функции: проверить наличие ключа, проверить тип, привести, вернуть `400` при любом расхождении. Эта гора растёт с каждым полем и живёт отдельно от описания того, как данные должны выглядеть.

Pydantic переворачивает подход: Вы один раз описываете, как выглядят корректные данные, в виде класса с типовыми полями, а проверку и приведение Pydantic делает сам. К концу раздела Вы будете уметь объявить модель, задать ограничения на поля, добавить собственную проверку и предсказать, что именно вернёт FastAPI, когда вход не пройдёт валидацию.

## Ментальная модель: класс как форма данных

Pydantic-модель - это класс, наследник `BaseModel`, где каждое поле объявлено с типовой аннотацией. Аннотация - не подсказка для IDE, а исполняемое правило. Когда FastAPI создаёт экземпляр модели из входного JSON, Pydantic проходит по полям и для каждого делает одно и то же: берёт значение, проверяет и приводит к объявленному типу, а при невозможности - копит ошибку.

```python
from pydantic import BaseModel

class User(BaseModel):
    name: str
    age: int
    is_active: bool = True
```

Поле `name` обязательно и должно быть строкой. `age` обязательно и приводится к `int`. `is_active` имеет значение по умолчанию, поэтому необязательно. Из JSON `{"name": "Ann", "age": "30"}` получится `User(name='Ann', age=30, is_active=True)`: строка `"30"` приведена к числу, `is_active` подставлен из дефолта.

Ключевое следствие: внутри обработчика Вы работаете с объектом, чьи поля уже гарантированно нужного типа. Ни одной ручной проверки. Если данные не прошли - обработчик не вызвался, а клиент получил `422` со списком всех проблем сразу, а не первой попавшейся.

## Ограничения на поля через Field

Тип задаёт грубую форму. Тонкие границы - длину строки, диапазон числа - задаёт `Field`.

```python
from pydantic import BaseModel, Field

class Product(BaseModel):
    name: str = Field(min_length=1, max_length=50)
    price: float = Field(gt=0)
    quantity: int = Field(ge=0, le=1000)
```

`gt=0` требует цену строго больше нуля, `ge=0` для количества - больше или равно. JSON с `"price": -5` теперь даст `422` с понятным сообщением, что значение должно быть больше нуля. Вы описали правило один раз, рядом с полем, и оно работает на каждом запросе.

<Callout type="tip">
`Field` различает `gt`/`ge` и `lt`/`le` так же, как математика: `gt` (greater than) - строго больше, `ge` (greater or equal) - больше или равно. Перепутать границу - частая причина того, что нолик проходит туда, где не должен.
</Callout>

## Вложенные модели

Поле модели может быть другой моделью. Pydantic рекурсивно проверит и его.

```python
class Address(BaseModel):
    city: str
    zip_code: str

class Customer(BaseModel):
    name: str
    address: Address
```

JSON `{"name": "Ann", "address": {"city": "Москва", "zip_code": "101000"}}` соберётся в `Customer`, где `customer.address` - полноценный объект `Address`. Если во вложенном объекте не хватит `city`, ошибка укажет точное место: `address.city`. Глубина вложенности не ограничена, и проверка остаётся одинаково подробной на любом уровне.

## Своя проверка через валидатор

Когда правило не выражается через тип и `Field` - например, «пароль и его подтверждение должны совпадать», - подключают валидатор: метод, помеченный декоратором, который Pydantic вызывает в ходе проверки.

```python
from pydantic import BaseModel, field_validator

class Signup(BaseModel):
    username: str

    @field_validator("username")
    @classmethod
    def no_spaces(cls, v: str) -> str:
        if " " in v:
            raise ValueError("в имени не должно быть пробелов")
        return v
```

Валидатор получает уже приведённое к типу значение, проверяет своё условие и либо возвращает значение (возможно, изменённое - например, `v.strip().lower()`), либо бросает `ValueError`. Брошенный `ValueError` Pydantic превратит в обычную ошибку валидации, и клиент увидит её в том же `422`-ответе рядом с остальными.

## Ответ - тоже модель: response_model

Модели описывают не только вход, но и выход. Параметр `response_model` декоратора фильтрует ответ по схеме: в JSON попадут только поля модели, остальное отсечётся.

```python
class UserOut(BaseModel):
    id: int
    name: str

@app.get("/users/{user_id}", response_model=UserOut)
def get_user(user_id: int):
    return {"id": user_id, "name": "Ann", "password": "secret"}
```

Несмотря на то что обработчик вернул и `password`, клиент получит только `{"id": ..., "name": ...}`. `response_model` - это страховка от случайной утечки полей: то, чего нет в выходной модели, наружу не уйдёт, даже если затесалось в возвращаемый словарь.

## Краевые случаи

**Лишние поля по умолчанию молча отбрасываются.** JSON с полем `nickname`, которого нет в модели, не вызовет ошибку - Pydantic просто проигнорирует `nickname`. Если хотите запрещать неизвестные поля, это настраивается отдельно (`model_config` с `extra="forbid"`), но из коробки лишнее просто теряется. Это удивляет тех, кто ждёт `422` на каждое незнакомое поле.

**`str` принимает почти всё, что строкой не является, осторожно.** В строгом режиме Pydantic не станет молча превращать число `42` в `"42"` для поля `str`, но в режиме по умолчанию (lax) поведение зависит от версии и настроек. Не полагайтесь на молчаливое приведение там, где тип важен, - объявляйте его точно.

**Значение по умолчанию вычисляется один раз.** Если дефолт поля - изменяемый объект, для него нужен `Field(default_factory=list)`, а не `= []`, иначе все экземпляры разделят один список. Это та же ловушка, что и с мутабельными аргументами функций по умолчанию в Python.

## Типичные ошибки

- **Делают руками то, что делает модель.** Цепочка `if "name" not in data: ...` внутри обработчика дублирует работу Pydantic. Если поле описано в модели как обязательное, проверка уже произошла до вызова функции.
- **Путают `gt` и `ge` на границе.** Поле `price: float = Field(ge=0)` пропустит ноль, а `gt=0` - нет. Для цены обычно нужен `gt`, для остатка на складе - `ge`. Граничное значение всплывает на проде, а не в наспех собранных тестах.
- **Возвращают из обработчика лишние поля без `response_model`.** Без выходной модели наружу уйдёт весь словарь, включая внутренние или чувствительные поля. `response_model` - явный контракт выхода.
