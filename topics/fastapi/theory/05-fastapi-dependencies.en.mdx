# The same piece of logic in every handler

Five handlers, and at the start of each one - the same dozen lines: pull the token from the header, open a database session, read the pagination parameters, check that the user is not banned. Copied five times. Change the token-checking rule, and you have to fix five places without missing any. This is exactly the pain that frameworks invent clever mechanisms for, and the price of relief is usually magic you can no longer debug.

FastAPI solves this differently - without magic. The repeated chunk is extracted into a plain function, and the handler declares in its signature that it needs that function by wrapping it in `Depends`. From there the framework calls the dependency function before the handler, substitutes its result into the parameter - and does this for every request. By the end of this section you will read a signature with `Depends` as fluently as ordinary parameters, and understand the order in which dependencies are resolved before control reaches the handler body.

## Mental model: "give me whatever this function returns"

A dependency in FastAPI is just a callable (most often a function) whose result the framework substitutes into a handler parameter. The line `current_user: User = Depends(get_current_user)` reads literally: "before calling me, call `get_current_user`, and put whatever it returns into `current_user`".

The key word is inversion. The handler does not call `get_current_user`; the framework calls it on the handler's behalf and brings back a ready result. The handler does not know how `User` is assembled: from a token, from a header, from the database. It only declares the need. This is the same trick as path/query/body from the first section - you declare what you need, and FastAPI fetches it from the right place. Now the "right place" is simply the result of another function.

The main thing in this model: dependencies form a graph that FastAPI walks bottom-up. First the ones with no dependencies of their own are computed, their results are passed to those that rely on them, and so on up to the handler. If a dependency at any level raises `HTTPException`, the walk stops and the handler is not called at all - exactly as with a validation error.

## The simplest dependency: an extracted function

Let's start with the most common case - shared query parameters. Pagination is needed by ten endpoints; let's describe it once.

```python
from fastapi import Depends, FastAPI

app = FastAPI()

def pagination(skip: int = 0, limit: int = 10):
    return {"skip": skip, "limit": limit}

@app.get("/items/")
def list_items(page: dict = Depends(pagination)):
    return page
```

The function `pagination` itself declares query parameters `skip` and `limit` - FastAPI parses them out of the request exactly as if they stood in the handler. The request `GET /items/?skip=20&limit=5` arrives in `pagination`, which returns a dict, and it ends up in `page`. The handler is clean: all the pagination fuss is hidden behind a single name. Add this dependency to another endpoint and it instantly gets the same `skip`/`limit` without a single copied line.

<Callout type="info">
A dependency is not a "magic object" but an ordinary function. You can call it directly in a test, passing arguments by hand, and check its logic in isolation from HTTP.
</Callout>

## Sub-dependencies: a dependency depends on a dependency

Dependencies nest inside one another. A dependency function may itself declare a parameter with `Depends` - and FastAPI resolves the whole chain.

```python
def get_token(authorization: str = ""):
    return authorization.removeprefix("Bearer ")

def get_current_user(token: str = Depends(get_token)):
    return {"user": token or "anonymous"}

@app.get("/me")
def me(user: dict = Depends(get_current_user)):
    return user
```

To obtain `user`, FastAPI must first obtain `token`, and for that - read the header. The walk goes from leaves to root: first `get_token` is computed, its result is passed to `get_current_user`, which hands `user` to the handler. The handler `me` sees only the final result - it has no idea there are two levels beneath it.

## Caching within a request

A subtlety that saves work: if the same dependency appears in the graph several times within one request, FastAPI calls it only once and reuses the result.

```python
def get_db():
    return {"conn": "session-42"}

def repo_users(db: dict = Depends(get_db)):
    return ("users", db["conn"])

def repo_orders(db: dict = Depends(get_db)):
    return ("orders", db["conn"])

@app.get("/dashboard")
def dashboard(u=Depends(repo_users), o=Depends(repo_orders)):
    return {"users": u, "orders": o}
```

`get_db` is needed by both `repo_users` and `repo_orders`. But within one request it runs exactly once, and both dependencies get the same session `session-42`. This matters for the database: otherwise one request would open a connection per use. You can disable the cache via `Depends(get_db, use_cache=False)`, but by default reuse is what you want.

## Dependencies with yield: setup before, teardown after

The most powerful trick is a dependency that uses `yield` instead of `return`. The code before `yield` is the setup, run before the handler. The value handed over via `yield` is substituted into the parameter. And the code after `yield` is the teardown, run after the handler has already returned its response.

```python
def get_session():
    session = open_session()
    try:
        yield session
    finally:
        session.close()

@app.get("/users/{user_id}")
def read_user(user_id: int, session=Depends(get_session)):
    return session.fetch(user_id)
```

The canonical example is a database session. Before `yield` the session is opened, via `yield` it is handed to the handler, after `yield` it is reliably closed, even if the handler raised an exception (the `finally` is responsible for that). The handler uses the session without worrying about closing it: the resource's lifecycle is tied to the request's lifecycle.

## Dependencies for a side effect: `dependencies=[...]`

Sometimes a dependency returns nothing to the handler - it exists only for a check. The classic is "let admins only through". The result of such a dependency is of no interest to the handler; what matters is the fact: if the check fails, it raises `HTTPException`, and the handler is never reached.

For this, the dependency is hung not in the signature but in `dependencies=[...]` - at the level of the endpoint, the router, or the whole application.

```python
def verify_admin(role: str = "guest"):
    if role != "admin":
        raise HTTPException(status_code=403, detail="admins only")

@app.delete("/items/{item_id}", dependencies=[Depends(verify_admin)])
def delete_item(item_id: int):
    return {"deleted": item_id}
```

`verify_admin` returns no value and is not mentioned in the parameters of `delete_item` - it is merely a guard. The same list can be hung on an `APIRouter`, and the check applies to all its routes, or on `FastAPI(dependencies=[...])` - and it covers the whole application. One declarative list instead of a permission check copied into every handler.

## Classes as parameterizable dependencies

A dependency function is fixed: its behavior is set once and for all. If you need a dependency you can configure, a class plays the role of the dependency. FastAPI parses `__init__` just like a function signature, and the instance itself becomes the result.

```python
class Paginator:
    def __init__(self, max_limit: int = 100):
        self.max_limit = max_limit

    def __call__(self, skip: int = 0, limit: int = 10):
        return {"skip": skip, "limit": min(limit, self.max_limit)}

short = Paginator(max_limit=20)

@app.get("/items/")
def list_items(page: dict = Depends(short)):
    return page
```

Here `Paginator(max_limit=20)` creates a configured instance, and `Depends(short)` uses it as a dependency: FastAPI calls `short(skip, limit)` and applies the limit from the configuration. One class - a family of dependencies with different settings. This is handy when similar endpoints have different bounds.

## Edge cases

A few places where the "function before the handler" model springs a surprise.

**Teardown in a yield dependency runs after the response, not inside it.** Code after `yield` fires when the handler has already returned a value. So an exception raised in teardown will not turn into a tidy response to the handler - it is already gone. Closing a resource in `finally` is reliable, but do not try to change an already-sent response from teardown.

**The order of dependencies in `dependencies=[...]` is respected.** The list runs left to right. If the first dependency checks authentication and the second checks permissions, put them in exactly that order: it is pointless to check permissions before establishing who is in front of us.

**Caching is within one request, not between requests.** Reuse of a dependency's result lives for exactly one request. The next request calls `get_db` anew. Do not confuse this cache with a long-lived application-level cache - a dependency does not keep state between requests by itself.

<Callout type="warning">
If you write `Depends(get_current_user())` with parentheses, you pass the framework an already-called result rather than the function itself. FastAPI expects a callable and will call it itself - pass `Depends(get_current_user)` without parentheses.
</Callout>

## Common mistakes

- **Called the dependency inside `Depends`.** `Depends(get_db())` executes `get_db` once at startup and slips its result to the framework as the "dependency". You need `Depends(get_db)` - the function object that FastAPI calls on every request.
- **Expecting a result from a guard dependency.** A dependency in `dependencies=[...]` puts nothing into the handler. If you need its result in the body, declare it as a parameter via `= Depends(...)`, not in the list of side-effect checks.
- **Closing a resource with `return` instead of `yield`.** A dependency with `return` cannot run teardown - after `return` the function is dead. To close the session after the response you need `yield` with `finally`, otherwise the connection leaks.

<Detail summary="Why yield rather than a context manager directly">

Under the hood FastAPI wraps a yield dependency in an async context manager. Once the dependency graph is resolved and the handler is called, the framework holds the generator "paused" at the `yield` while the request is processed. After the response is built, it resumes the generator - control returns to the line right after `yield`, the `finally` runs, and the generator finishes. This is exactly why one yield dependency can wrap both opening/closing a transaction and acquiring/releasing a lock: setup and teardown end up on opposite sides of the single `yield`, and the framework guarantees the second half runs. Several yield dependencies in one request are closed in the reverse order of opening - like nested `with` blocks.

</Detail>
