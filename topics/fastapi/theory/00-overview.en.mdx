You need to send an HTTP service externally: the client sends a request to `/items/42?q=header` with JSON in the body, and in response it expects correct data and a clear status. You can manually parse the URL, get query parameters, check each body field and put together a response - and drown in a mountain of `ifs' even before the business logic. FastAPI takes care of this routine: you describe what the input looks like with the usual function signature and types, and the framework decides where to get the value from, checks it and generates a response. This topic is about how exactly he makes such decisions - so that you stop guessing and start predicting.

**What you will learn to do.** Using one handler signature, predict which URL will call it and what will be in each parameter; describe input data with a model that checks it for you; consciously choose between `def` and `async def`, without bogging down the server with a blocking call; return the correct HTTP status and build a common layer of logic around each request.

**Theme route.** Four sections, each building on the previous one:

1. **Routes** - based on the function signature, FastAPI decides what to take from the path, what from the query string, and what from the request body.
2. **Pydantic** - you describe the form of the correct data once with a class, and the framework does the checking and type casting itself.
3. **async** - we figure out when `async def` speeds up the service, and when a blocking call inside it hangs the entire server.
4. **Errors and middleware** - return honest statuses instead of `200` for everything and wrap each request with a common layer of logic.

**What you need beforehand** Be confident enough to read regular Python: functions, classes, type annotations. There is no need to install anything to read - we analyze the examples by code. We introduce concepts about HTTP, event loop and validation along the way, from a simple handler to a composite service.
