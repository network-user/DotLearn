# Откуда FastAPI берёт параметры обработчика

Вы пишете функцию `def get_item(item_id: int, q: str = None)`, привязываете её к адресу декоратором, запускаете сервер - и вдруг по адресу `/items/42?q=шапка` вызывается именно эта функция, причём `item_id` уже равен числу `42`, а не строке `"42"`. Никто явно не разбирал URL, не доставал query string, не приводил типы. Откуда фреймворк узнал, что `item_id` нужно взять из пути, `q` из строки запроса, и где он вообще решил, что `item_id` - это `int`?

Ответ - в сигнатуре функции. FastAPI читает имена и аннотации параметров обработчика и по набору простых правил решает, откуда взять каждое значение: из шаблона пути, из query string или из тела запроса. К концу раздела Вы будете понимать эти правила настолько, чтобы по одной сигнатуре предсказать, какой URL вызовет обработчик, что окажется в каждом параметре и что фреймворк сделает с некорректным вводом.

## Ментальная модель: сигнатура - это контракт входа

Обработчик в FastAPI - это обычная функция, привязанная к паре «HTTP-метод + шаблон пути» декоратором вроде `@app.get("/items/{item_id}")`. Когда приходит запрос, фреймворк смотрит на параметры функции и для каждого решает источник по трём правилам, которые применяются именно в таком порядке.

Имя параметра совпало с фигурной скобкой в пути - это **path-параметр**. Тип параметра - примитив (`int`, `str`, `float`, `bool`), а в пути его нет - это **query-параметр**. Тип - Pydantic-модель (или объявлен как `Body`) - это **тело запроса**, тот самый JSON в `POST`. Один обработчик может смешивать все три источника в одной сигнатуре.

Главное в этой модели: Вы не парсите запрос вручную. Вы объявляете, что нужно функции, а FastAPI достаёт это из правильного места и приводит к объявленному типу. Запрос - это сырые байты, сигнатура - контракт, фреймворк - посредник между ними.

## Path-параметр: часть адреса

Начнём с самого простого источника. Кусок пути в фигурных скобках попадает в одноимённый параметр функции.

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/items/{item_id}")
def read_item(item_id: int):
    return {"item_id": item_id}
```

Запрос `GET /items/42` вызовет `read_item` с `item_id == 42` - целым числом, а не строкой `"42"`. Аннотация `int` здесь не украшение: FastAPI по ней конвертирует значение из URL. Если прислать `/items/abc`, конвертация в `int` провалится, и клиент получит ответ `422 Unprocessable Entity` с описанием, что `item_id` должен быть целым. Обработчик при этом не вызовется вовсе.

<Callout type="info">
Имя параметра в функции обязано совпадать с именем в фигурных скобках пути. `{item_id}` в пути и `item_id` в сигнатуре - это связка по имени, а не по позиции.
</Callout>

## Query-параметр: хвост после вопроса

Если параметр функции не упомянут в пути и имеет примитивный тип, FastAPI ищет его в query string - части URL после `?`.

```python
@app.get("/items/")
def list_items(skip: int = 0, limit: int = 10):
    return {"skip": skip, "limit": limit}
```

Запрос `GET /items/?skip=20&limit=5` даст `skip == 20`, `limit == 5`. Значение по умолчанию делает параметр необязательным: `GET /items/` без query вернёт `skip=0, limit=10`. Уберите `= 0`, и `skip` станет обязательным - запрос без него получит `422`. Значение по умолчанию - это и есть граница между «обязательно» и «нет».

Типы query-параметров тоже приводятся. `bool`-параметр понимает `true`, `1`, `yes`, `on` как истину; `?active=1` даст `active == True`. Это удобно, но и источник сюрпризов - о них ниже.

## Body: JSON в теле запроса

Когда параметр - это Pydantic-модель, FastAPI читает тело запроса как JSON и собирает из него объект модели.

```python
from pydantic import BaseModel

class Item(BaseModel):
    name: str
    price: float

@app.post("/items/")
def create_item(item: Item):
    return {"name": item.name, "total": item.price}
```

`POST /items/` с телом `{"name": "шапка", "price": 9.9}` придёт в обработчик уже как объект `Item`, где `item.name == "шапка"`. Если в JSON нет поля `price` или оно не приводится к `float`, FastAPI вернёт `422` с указанием, какое именно поле невалидно. Детальный разбор Pydantic-моделей - в следующем разделе; здесь важно лишь то, что модель в сигнатуре переключает источник на тело запроса.

## Три источника в одной сигнатуре

Самое полезное - комбинировать. Эта сигнатура читается как «возьми `item_id` из пути, `q` из query, тело собери в `Item`».

```python
@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item, q: str = None):
    return {"item_id": item_id, "q": q, "name": item.name}
```

`PUT /items/42?q=зима` с JSON-телом `{"name": "шапка", "price": 9.9}` распределится так: `item_id == 42` (путь), `q == "зима"` (query), `item` - объект модели (тело). Порядок параметров в функции значения не имеет - источник определяется правилами, а не позицией.

## Краевые случаи

Несколько мест, где интуиция подводит.

**`bool` понимает строки слишком вольно.** Query-параметр `flag: bool` примет `?flag=1`, `?flag=true`, `?flag=on` как `True`, но `?flag=2` или `?flag=да` даст `422`. А вот `?flag=False` (с заглавной) и `?flag=0` корректно дают `False`. Если ждёте от клиента произвольную строку, не объявляйте её `bool`.

**Слэш в path-параметре по умолчанию ломает маршрут.** Параметр `{file_path}` со значением `docs/intro.txt` не совпадёт с `/files/{file_path}`: слэш считается границей сегмента пути. Чтобы принять путь со слэшами, нужен конвертер `{file_path:path}` в шаблоне.

**Тело у `GET` игнорируется.** Можно объявить модель в `GET`-обработчике, но по соглашению HTTP тело в `GET` не передаётся, и клиенты его не шлют. Body-параметры - это про `POST`, `PUT`, `PATCH`.

<Callout type="warning">
`422` - это не ошибка вашего кода, а корректный ответ на невалидный ввод. Если при тестах вы видите `422` вместо ожидаемого результата, проверьте не обработчик, а форму запроса: тип path-параметра, наличие обязательных query, структуру JSON-тела.
</Callout>

## Типичные ошибки

- **Имя query-параметра не совпало с тем, что шлёт клиент.** Параметр `limit`, а в URL `?max=5` - FastAPI не найдёт `max`, а `limit` останется дефолтным (или даст `422`, если обязателен). Источник по имени, регистр и написание важны.
- **Забыли, что path-параметр без аннотации остаётся строкой.** `def read(item_id):` без `: int` положит в `item_id` строку `"42"`, и сравнение `item_id == 42` будет ложным. Аннотация - это и есть инструкция на приведение типа.
- **Ждут `200` при кривом вводе.** Невалидный path или отсутствующий обязательный query дают `422` до вызова функции. Тело обработчика в этот момент ещё не выполнялось, так что искать баг внутри него бесполезно.

<Detail summary="Откуда берётся 422 и почему именно он">

FastAPI строит вокруг каждого обработчика слой валидации на базе Pydantic. Любое значение - из пути, query или тела - сначала проходит проверку и приведение типа. Если хоть одно значение не проходит, фреймворк не вызывает обработчик, а сразу формирует ответ `422 Unprocessable Entity` с телом, где перечислены все ошибки: расположение (`loc`), тип ошибки и сообщение. Код `422` выбран намеренно: `400 Bad Request` слишком общий, а `422` из стандарта WebDAV точно означает «синтаксис запроса верный, но содержимое не проходит проверку». Эта же машинерия генерирует интерактивную документацию по адресу `/docs` - схемы там выводятся из тех же аннотаций.

</Detail>
