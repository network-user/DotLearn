# Как сервер узнаёт, кто к нему пришёл

HTTP не помнит вас. Каждый запрос приходит сам по себе, без памяти о предыдущем: сервер не знает, тот ли это пользователь, что заходил минуту назад, или кто-то совсем другой. И всё же приложение должно отличать Анну от Бориса, пускать одного в его профиль и не пускать в чужой. Раз состояния нет, каждый запрос должен сам нести доказательство, кто его отправил. Это доказательство и есть токен - короткая строка, которую клиент прикладывает к запросу, а сервер проверяет.

Сложить это в работающую схему - значит ответить на три вопроса. Как пользователь получает токен, не пересылая пароль в каждом запросе? Как сервер по токену понимает, кто перед ним? И чем «я не знаю, кто ты» отличается от «я знаю, кто ты, но сюда нельзя»? К концу раздела Вы будете видеть весь путь - от логина до защищённого эндпоинта - как одну цепочку, в которой `Depends` и токен играют те же роли, что и в прошлом разделе.

## Ментальная модель: пропуск, а не пароль на каждом шаге

Представьте проходную. Один раз вы показываете паспорт (логин и пароль) и получаете пропуск (токен). Дальше на каждую дверь вы предъявляете не паспорт, а пропуск - его быстро проверяет любой турникет, не запрашивая отдел кадров. Пароль участвует один раз; всё остальное время по зданию вас водит пропуск.

В вебе это поток OAuth2 с Bearer-токеном. Клиент один раз отправляет логин и пароль на эндпоинт `/token`, получает в ответ токен и хранит его. Дальше в каждый защищённый запрос он добавляет заголовок `Authorization: Bearer <токен>`. Сервер на каждый такой запрос достаёт токен из заголовка, проверяет его и по нему узнаёт пользователя - не трогая пароль.

Главное в этой модели: пароль появляется ровно один раз, при выдаче токена. Дальше вся защита держится на токене - его проверке и на том, что из него можно достать. Сервер на каждый запрос проходит один и тот же путь: вынуть токен, проверить, опознать пользователя.

## Откуда берётся токен в заголовке: OAuth2PasswordBearer

FastAPI не угадывает, где лежит токен, - вы объявляете это зависимостью. `OAuth2PasswordBearer` - это вызываемый объект, который умеет одно: достать токен из заголовка `Authorization: Bearer ...`. Его кладут в `Depends`, и в параметр эндпоинта приезжает уже сама строка токена.

```python
from fastapi import Depends, FastAPI
from fastapi.security import OAuth2PasswordBearer

app = FastAPI()
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="token")

@app.get("/items/")
def read_items(token: str = Depends(oauth2_scheme)):
    return {"token": token}
```

Аргумент `tokenUrl="token"` - это не магия, а указатель: где находится эндпоинт логина, на котором токен выдают. Если в запросе нет заголовка `Authorization`, `oauth2_scheme` сам бросит `HTTPException(401)`, и обработчик не вызовется. Так извлечение токена становится переиспользуемой зависимостью - ровно тот приём из прошлого раздела.

## Хеширование пароля: почему его нельзя хранить как есть

Прежде чем выдать токен, сервер должен проверить пароль. Но хранить пароли в открытом виде нельзя: утечёт база - утекут все пароли. Поэтому хранят не пароль, а его хеш - результат одностороннего преобразования. По хешу восстановить пароль нельзя; при логине сервер хеширует присланный пароль и сравнивает хеши.

```python
from passlib.context import CryptContext

pwd = CryptContext(schemes=["bcrypt"], deprecated="auto")

stored_hash = pwd.hash("plaintext-password")

def check(password: str) -> bool:
    return pwd.verify(password, stored_hash)
```

`pwd.hash` берёт пароль и возвращает хеш для хранения; `pwd.verify` проверяет, что присланный пароль даёт тот же хеш. Алгоритм вроде `bcrypt` сделан намеренно медленным и солит каждый хеш, чтобы перебор был дорогим. Деталей реализации знать не нужно - важна идея: в базе лежит хеш, пароль в открытом виде не хранится нигде.

<Callout type="warning">
Никогда не храните и не логируйте пароли в открытом виде и не придумывайте своё хеширование. Пароль превращается в хеш проверенной библиотекой и только в таком виде попадает в хранилище.
</Callout>

## Что внутри токена: выпуск и проверка JWT

Чаще всего токен - это JWT (JSON Web Token). Внутри он несёт «claims» - набор полей: кто пользователь (`sub`), когда токен истекает (`exp`), иногда права. Всё это подписано секретным ключом сервера. Подпись не прячет содержимое (claims читаются), а гарантирует, что их не подменили: меняешь хоть байт - подпись не сходится.

```python
from jose import jwt

SECRET = load_secret()

def issue(username: str) -> str:
    payload = {"sub": username, "exp": expiry()}
    return jwt.encode(payload, SECRET, algorithm="HS256")

def read(token: str) -> str:
    data = jwt.decode(token, SECRET, algorithms=["HS256"])
    return data["sub"]
```

При логине сервер кладёт в payload `sub` и `exp`, подписывает ключом и отдаёт строку. При запросе он `jwt.decode` проверяет подпись и срок: если подпись неверна или `exp` в прошлом, `decode` бросит исключение, и токен отвергается. Никакой сессии на сервере не нужно - вся нужная информация лежит в самом токене, защищённая подписью.

## Зависимость get_current_user: токен превращается в пользователя

Теперь соберём всё в одну зависимость. `get_current_user` опирается на `oauth2_scheme` (достать токен), читает из токена `sub`, находит пользователя - и возвращает его. Эндпоинту достаётся уже готовый объект пользователя, а не возня с токеном.

```python
from fastapi import Depends, HTTPException

def get_current_user(token: str = Depends(oauth2_scheme)):
    try:
        username = read(token)
    except Exception:
        raise HTTPException(status_code=401, detail="неверный токен")
    user = users.get(username)
    if user is None:
        raise HTTPException(status_code=401, detail="нет такого пользователя")
    return user

@app.get("/me")
def me(user: dict = Depends(get_current_user)):
    return user
```

Это та же иерархия зависимостей из прошлого раздела: `get_current_user` зависит от `oauth2_scheme`, эндпоинт зависит от `get_current_user`. Любой защищённый эндпоинт теперь добавляет один параметр `= Depends(get_current_user)` - и получает текущего пользователя, не зная, как именно он опознан. Если токена нет, неверен или истёк, до тела эндпоинта дело не доходит: ответ `401`.

## Scopes: права внутри токена

Аутентификация отвечает «кто ты». Авторизация - «что тебе можно». В OAuth2 для второго есть scopes - помеченные права, например `items:read` и `items:write`. Их кладут в токен при выдаче, а защищённый эндпоинт требует нужный scope. Если у пользователя его нет, эндпоинт отвечает `403` - токен валиден, но прав не хватает.

## 401 против 403: два разных «нет»

Два статуса легко перепутать, а разница принципиальна.

`401 Unauthorized` - «я не знаю, кто ты». Токена нет, он испорчен или истёк - сервер не смог опознать отправителя. По смыслу это «представься заново».

`403 Forbidden` - «я знаю, кто ты, но сюда нельзя». Токен валиден, пользователь опознан, но у него нет прав на это действие. Повторный логин не поможет - не хватает именно прав.

Порядок проверок прямо вытекает из этого: сначала аутентификация (есть ли валидный токен - иначе `401`), потом авторизация (хватает ли прав - иначе `403`). Проверять права раньше, чем выяснили, кто перед нами, бессмысленно.

## Краевые случаи

Несколько мест, где интуиция подводит.

**Подпись JWT не шифрует payload.** Claims внутри токена читаются кем угодно - они лишь закодированы, не зашифрованы. Подпись защищает от подмены, но не от чтения. Не кладите в JWT секреты вроде пароля или номера карты - туда идут идентификатор и срок, не более.

**Истёкший токен - это `401`, а не `403`.** Просроченный токен сервер не смог принять как доказательство личности, значит отправитель не опознан - `401`. `403` уместен только когда личность установлена, но прав не хватает. Спутать их - значит сбить клиента с толку: он не поймёт, логиниться заново или просить доступ.

**Скрытие ресурса иногда возвращают как `404`, а не `403`.** Чтобы не подтверждать само существование чужого ресурса, его порой прячут под `404` вместо `403`. Это осознанный выбор приватности, а не противоречие правилу: дефолт - `403`, но `404` уместен, когда сам факт существования ресурса не должен утекать.

<Callout type="info">
Самопроверка: «надо ли клиенту залогиниться заново?» Да - это `401`. Нет, он уже опознан, ему просто сюда нельзя - это `403`.
</Callout>

## Типичные ошибки

- **Возвращают `403` на отсутствующий или истёкший токен.** Если отправитель не опознан, корректен `401`. `403` говорит «ты опознан, но тебе нельзя» - для анонима это неверный сигнал.
- **Кладут чувствительные данные в payload JWT.** Payload читается без ключа. Пароль, токен карты, приватные заметки в нём окажутся на виду у любого, кто перехватит токен. Внутрь - только идентификатор и метаданные вроде срока.
- **Проверяют права раньше аутентификации.** Сначала надо знать, кто перед нами (валиден ли токен), и лишь затем - хватает ли ему прав. Обратный порядок и нелогичен, и даёт неверный статус.

<Detail summary="Почему 401 нередко сопровождается заголовком WWW-Authenticate">

По спецификации HTTP ответ `401` должен содержать заголовок `WWW-Authenticate`, сообщающий клиенту, какая схема аутентификации ожидается, - например `Bearer`. Это часть контракта: сервер не просто отказывает, а подсказывает, как представиться правильно. FastAPI при использовании `OAuth2PasswordBearer` выставляет такой заголовок автоматически, когда зависимость отвергает запрос без токена. У `403` такого заголовка нет, и это логично: проблема не в способе аутентификации (он прошёл), а в нехватке прав - предлагать «представься иначе» бессмысленно. Эта деталь - ещё один признак, по которому `401` и `403` различаются на уровне самого протокола, а не только по нашему соглашению.

</Detail>
