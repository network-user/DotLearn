# A model that checks the data for you

Without checking the input data, any handler turns into a minefield: the client will send `price` as a string, forget a required field, send `email` without a dog - and you will fall, somewhere deep inside the business logic, with an incomprehensible `TypeError`. The classic answer is to write a bunch of `if's by hand at the beginning of the function: check for the presence of a key, check the type, cast, return `400` for any discrepancy. This mountain grows with each field and lives separately from the description of how the data should look.

Pydantic reverses the approach: You once describe what the correct data looks like in the form of a class with standard fields, and Pydantic does the checking and casting itself. By the end of this section, you will be able to declare a model, set field constraints, add your own validation, and predict what FastAPI will return when the input fails validation.

## Mental model: class as a form of data

A Pydantic model is a class derived from `BaseModel`, where each field is declared with a standard annotation. An annotation is not a hint for the IDE, but an executable rule. When FastAPI instantiates a model from the input JSON, Pydantic goes through the fields and does the same thing for each: takes the value, checks and results in the declared type, and if not possible, generates an error.

```python
from pydantic import BaseModel

class User(BaseModel):
    name: str
    age: int
    is_active: bool = True
```

The `name` field is required and must be a string. `age` is required and casts to `int`. `is_active` has a default value, so is optional. From JSON `{"name": "Ann", "age": "30"}` we get `User(name='Ann', age=30, is_active=True)`: the string `"30"` is reduced to a number, `is_active` is substituted from the default.

The key consequence: inside the handler, you are working with an object whose fields are already guaranteed to be of the desired type. No manual checks. If the data did not go through, the handler was not called, and the client received `422` with a list of all problems at once, and not the first one that came up.

## Restrictions on fields via Field

The type specifies the rough shape. Thin boundaries - line length, number range - are set by `Field`.

```python
from pydantic import BaseModel, Field

class Product(BaseModel):
    name: str = Field(min_length=1, max_length=50)
    price: float = Field(gt=0)
    quantity: int = Field(ge=0, le=1000)
```

`gt=0` requires price strictly greater than zero, `ge=0` for quantity - greater than or equal to. JSON with `"price": -5` will now give `422` with a clear message that the value must be greater than zero. You described the rule once, next to the field, and it works on every request.

<Callout type="tip">
`Field` distinguishes between `gt`/`ge` and `lt`/`le` just like mathematics: `gt` (greater than) - strictly greater, `ge` (greater or equal) - greater than or equal. Mixing up the boundary is a common reason for the zero going where it shouldn't.
</Callout>

## Nested models

The model field may be another model. Pydantic will check it recursively as well.

```python
class Address(BaseModel):
    city: str
    zip_code: str

class Customer(BaseModel):
    name: str
    address: Address
```

JSON `{"name": "Ann", "address": {"city": "Moscow", "zip_code": "101000"}}` will be collected in `Customer`, where `customer.address` is a full-fledged `Address` object. If the nested object is missing `city`, the error will indicate the exact location: `address.city`. The depth of nesting is not limited, and the check remains equally detailed at any level.

## Custom verification via validator

When a rule is not expressed in terms of a type and a `Field` - for example, "the password and its confirmation must match" - a validator is included: a method marked with a decorator that Pydantic calls during validation.

```python
from pydantic import BaseModel, field_validator

class Signup(BaseModel):
    username: str

    @field_validator("username")
    @classmethod
    def no_spaces(cls, v: str) -> str:
        if " " in v:
            raise ValueError("в имени не должно быть пробелов")
        return v
```

The validator receives a value already cast to the type, checks its condition and either returns the value (possibly modified - for example, `v.strip().lower()`), or throws `ValueError`. Pydantic will turn a thrown `ValueError` into a regular validation error, and the client will see it in the same `422` response next to the others.

## The response is also a model: response_model

Models describe not only the input, but also the output. The `response_model` parameter of the decorator filters the response according to the scheme: only the model fields will be included in the JSON, the rest will be cut off.

```python
class UserOut(BaseModel):
    id: int
    name: str

@app.get("/users/{user_id}", response_model=UserOut)
def get_user(user_id: int):
    return {"id": user_id, "name": "Ann", "password": "secret"}
```

Even though the handler also returned `password`, the client will only receive `{"id": ..., "name": ...}`. `response_model` is insurance against accidental leakage of fields: what is not in the output model will not escape, even if it is included in the returned dictionary.

## Edge cases

**Excess fields are silently discarded by default.** JSON with a `nickname` field that is not in the model will not raise an error - Pydantic will simply ignore the `nickname`. If you want to prohibit unknown fields, this can be configured separately (`model_config` with `extra="forbid"`), but out of the box the extra stuff is simply lost. This surprises those who expect `422` for every unfamiliar field.

**`str` accepts almost anything that is not a string, with caution.** In strict mode, Pydantic will not silently turn the number `42` into `"42"` for the `str` field, but in default mode (lax) the behavior depends on the version and settings. Don't rely on silent casting where the type is important - declare it precisely.

**The default value is evaluated once.** If the default field is a mutable object, it needs `Field(default_factory=list)` rather than `= []`, otherwise all instances will share the same list. This is the same pitfall as with Python's default mutable function arguments.

## Common mistakes

- **They do with their hands what the model does.** The `if "name" not in data: ...` chain inside the handler duplicates the work of Pydantic. If a field is described as required in the model, validation has already occurred before the function is called.
- **Confuse `gt` and `ge` at the border.** The `price: float = Field(ge=0)` field will skip a zero, but `gt=0` will not. For the price you usually need `gt`, for the balance in the warehouse - `ge`. The limit value appears in production, and not in hastily assembled tests.
- **Return extra fields from the handler without `response_model`.** Without an output model, the entire dictionary, including internal or sensitive fields, will go outside. `response_model` is an explicit exit contract.

<Detail summary="Pydantic v1 and v2: what has changed in the names">

The theme uses the Pydantic v2 API - this is what modern FastAPI installs. If you see in old examples `@validator` instead of `@field_validator`, `class Config` instead of `model_config`, `.dict()` instead of `.model_dump()` - this is Pydantic v1. The idea is the same, the names and some defaults have changed (in particular, v2 is stricter about type casting). For new code, stick to v2 names; When reading someone else's code, keep in mind that both sets mean approximately the same thing.
