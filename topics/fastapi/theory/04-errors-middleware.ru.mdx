# Правильный статус и общий слой вокруг каждого запроса

Обработчик нашёл товар - вернул его, и клиент видит `200`. А если товара нет? Вернуть `200` с пустым телом - значит соврать: для клиента «успешно, вот ничего» и «не найдено» неразличимы, и его код не сможет правильно отреагировать. HTTP-статус - это не косметика, а часть контракта: по нему клиент решает, повторять запрос, показать ошибку пользователю или уйти на страницу логина. Запутанные статусы превращают понятный API в гадание.

Вторая половина раздела - о коде, который должен выполняться вокруг каждого запроса, независимо от обработчика: залогировать, замерить время, добавить заголовок, проверить токен. Дублировать это в каждой функции бессмысленно. Middleware - это обёртка над всем конвейером, через которую проходит любой запрос и любой ответ. К концу раздела Вы будете уметь возвращать корректные статусы и ошибки и понимать, где в жизненном цикле запроса работает middleware и чем он отличается от обработки исключений.

## Ментальная модель: запрос в кольцах

Представьте запрос, проходящий сквозь несколько концентрических колец на пути к обработчику и обратно. Внешнее кольцо - middleware - видит запрос первым и ответ последним. Дальше идёт обработчик в центре. Если внутри что-то бросает `HTTPException`, FastAPI перехватывает его и формирует HTTP-ответ с нужным статусом, который точно так же поднимается наружу через кольца middleware.

Модель «в кольцах» объясняет два разных механизма. **Статус и ошибки** - это про то, что отдаёт обработчик в центре. **Middleware** - это про общее кольцо, которое одинаково оборачивает любой обработчик. Оба нужны для аккуратного API, но решают разные задачи.

## Статус по умолчанию и его смена

Без указаний успешный обработчик возвращает `200 OK`. Для создания ресурса честнее `201 Created` - его задают в декораторе.

```python
from fastapi import FastAPI, status

app = FastAPI()

@app.post("/items/", status_code=status.HTTP_201_CREATED)
def create_item(name: str):
    return {"name": name}
```

`status_code=201` означает «ресурс создан». Модуль `status` даёт читаемые имена вместо магических чисел: `status.HTTP_201_CREATED` понятнее, чем `201`, и не даёт опечататься в коде. Это статус «счастливого пути» - его FastAPI поставит, когда обработчик отработал без исключений.

## Ошибка - это исключение HTTPException

Когда обработчик не может выполнить запрос - ресурса нет, прав нет, - он бросает `HTTPException` с нужным статусом. FastAPI перехватит его и сформирует корректный ответ-ошибку.

```python
from fastapi import HTTPException

items = {1: "шапка"}

@app.get("/items/{item_id}")
def read_item(item_id: int):
    if item_id not in items:
        raise HTTPException(status_code=404, detail="товар не найден")
    return {"name": items[item_id]}
```

`GET /items/99` поднимет `HTTPException(404)`, и клиент получит `404 Not Found` с телом `{"detail": "товар не найден"}`. Важно: это `raise`, а не `return`. Исключение прерывает обработчик в точке броска - код после него не выполнится, что и нужно: незачем продолжать, если ресурса нет.

Какой статус выбрать - вопрос семантики HTTP, а не вкуса. `400` - запрос синтаксически плох. `401` - вы не аутентифицированы. `403` - аутентифицированы, но прав нет. `404` - ресурса нет. `409` - конфликт (например, дубликат). `422` - тело синтаксически верно, но не прошло валидацию (этот FastAPI ставит сам). `500` - сломалось на сервере. Подбирайте статус под смысл исхода, и клиентский код станет проще.

## Middleware: код вокруг каждого запроса

Middleware - функция, которая получает запрос, передаёт его дальше по кольцам через `call_next` и получает обратно ответ. До `call_next` Вы видите входящий запрос, после - исходящий ответ.

```python
import time
from fastapi import Request

@app.middleware("http")
async def add_timing(request: Request, call_next):
    start = time.perf_counter()
    response = await call_next(request)
    elapsed = time.perf_counter() - start
    response.headers["X-Process-Time"] = str(elapsed)
    return response
```

Этот middleware замеряет время обработки и кладёт его в заголовок ответа - для любого маршрута, без правок в обработчиках. До `await call_next(request)` выполняется код «на входе», после - код «на выходе». Так в одно место выносят сквозную логику: логирование, метрики, проверку заголовков, добавление CORS. Обработчики при этом ничего о middleware не знают.

<Callout type="info">
Middleware выполняется на каждый запрос в порядке регистрации на входе и в обратном порядке на выходе - как вложенные кольца. Не кладите в middleware тяжёлую блокирующую работу: она удлинит абсолютно каждый запрос, а не только тот, которому нужна.
</Callout>

## Краевые случаи

**`return HTTPException` вместо `raise` ничего не сделает.** Если по ошибке написать `return HTTPException(404)`, исключение не поднимется - FastAPI попытается сериализовать сам объект исключения как тело ответа, и клиент получит странный `200`. Ошибки отдаются только через `raise`.

**Middleware видит даже ответы-ошибки.** Ответ, собранный из `HTTPException` или из `422`-валидации, всё равно проходит через middleware на выходе. Поэтому логирующий middleware зафиксирует и неуспешные запросы - это удобно, но помните, что в `response.status_code` там будет `4xx`/`5xx`, а не только `200`.

**Заголовок к ошибке через `HTTPException` тоже можно.** У `HTTPException` есть параметр `headers` - например, `raise HTTPException(401, headers={"WWW-Authenticate": "Bearer"})`. Ошибочный ответ - полноценный HTTP-ответ, и заголовки в нём так же легитимны, как в успешном.

## Типичные ошибки

- **Возвращают `200` на «не найдено».** Пустое тело с `200` неотличимо для клиента от успеха. Бросайте `404` через `HTTPException` - клиентский код тогда сможет ветвиться по статусу.
- **Путают `401` и `403`.** `401` - «вы не представились» (нет или невалиден токен), `403` - «представились, но сюда нельзя». Это разные исходы, и клиент реагирует на них по-разному: на `401` ведёт на логин, на `403` показывает «доступ запрещён».
- **Кладут в middleware блокирующий или тяжёлый код.** Поскольку middleware - на пути каждого запроса, его задержка прибавляется ко всем. Логируйте дёшево, тяжёлое выносите за пределы горячего пути.

<Detail summary="HTTPException - это не сбой, а штатный ответ">

Слово «exception» сбивает с толку: кажется, что `HTTPException` - это про аварию. На деле это штатный механизм вернуть не-`2xx`-ответ из любой точки обработчика, не таща признак ошибки сквозь все промежуточные вызовы. Бросив `HTTPException(404)` глубоко во вложенной функции, вы немедленно прерываете цепочку и отдаёте клиенту `404` - не нужно возвращать `None` наверх и проверять его на каждом уровне. Для ошибок, которые повторяются по всему приложению (например, своя доменная «не найдено»), удобно завести собственный класс исключения и зарегистрировать для него обработчик через `@app.exception_handler` - тогда из обработчиков вы бросаете доменное исключение, а перевод его в нужный HTTP-статус живёт в одном месте.

</Detail>
