# Один и тот же кусок логики в каждом обработчике

Пять обработчиков, и в начале каждого - один и тот же десяток строк: достать токен из заголовка, открыть сессию к базе, прочитать параметры пагинации, проверить, что пользователь не забанен. Скопировано пять раз. Меняется правило проверки токена - и нужно править пять мест, не забыв ни одного. Это та самая боль, ради которой в фреймворках появляются хитрые механизмы, и обычно ценой за избавление становится магия, которую потом не отладить.

FastAPI решает это иначе - без магии. Повторяющийся кусок выносится в обычную функцию, а обработчик объявляет в своей сигнатуре, что эта функция ему нужна, обернув её в `Depends`. Дальше фреймворк сам вызовет функцию-зависимость до обработчика, подставит её результат в параметр - и так для каждого запроса. К концу раздела Вы будете читать сигнатуру с `Depends` так же бегло, как обычные параметры, и понимать порядок, в котором зависимости разрешаются перед тем, как управление дойдёт до тела обработчика.

## Ментальная модель: «дай мне то, что вернёт эта функция»

Зависимость в FastAPI - это просто вызываемый объект (чаще всего функция), результат которого фреймворк подставляет в параметр обработчика. Запись `current_user: User = Depends(get_current_user)` читается дословно: «прежде чем вызвать меня, вызови `get_current_user`, а то, что она вернёт, положи в `current_user`».

Ключевое слово - инверсия. Не обработчик зовёт `get_current_user`, а фреймворк зовёт её за него и приносит готовый результат. Обработчик не знает, как именно собирается `User`: из токена, из заголовка, из базы. Он лишь объявляет потребность. Это тот же приём, что и с path/query/body из первого раздела - Вы декларируете, что нужно, а FastAPI достаёт это из правильного места. Просто теперь «правильное место» - результат другой функции.

Главное в этой модели: зависимости образуют граф, который FastAPI обходит снизу вверх. Сначала вычисляются те, у кого нет своих зависимостей, их результаты передаются тем, кто на них опирается, и так до самого обработчика. Если на любом уровне зависимость бросит `HTTPException`, обход прерывается и обработчик не вызывается вовсе - ровно как при ошибке валидации.

## Простейшая зависимость: вынесенная функция

Начнём с самого частого случая - общие query-параметры. Пагинация нужна десяти эндпоинтам; опишем её один раз.

```python
from fastapi import Depends, FastAPI

app = FastAPI()

def pagination(skip: int = 0, limit: int = 10):
    return {"skip": skip, "limit": limit}

@app.get("/items/")
def list_items(page: dict = Depends(pagination)):
    return page
```

Функция `pagination` сама объявляет query-параметры `skip` и `limit` - FastAPI разберёт их из запроса точно так же, как если бы они стояли в обработчике. Запрос `GET /items/?skip=20&limit=5` придёт в `pagination`, та вернёт словарь, и он окажется в `page`. Обработчик чист: вся возня с пагинацией спрятана за одним именем. Добавьте эту зависимость в другой эндпоинт - и он мгновенно получит те же `skip`/`limit` без единой скопированной строки.

<Callout type="info">
Зависимость - не «магический объект», а обычная функция. Её можно вызвать в тесте напрямую, передав аргументы руками, и проверить логику в изоляции от HTTP.
</Callout>

## Под-зависимости: зависимость зависит от зависимости

Зависимости вкладываются друг в друга. Функция-зависимость сама может объявить параметр с `Depends` - и FastAPI разрешит всю цепочку.

```python
def get_token(authorization: str = ""):
    return authorization.removeprefix("Bearer ")

def get_current_user(token: str = Depends(get_token)):
    return {"user": token or "anonymous"}

@app.get("/me")
def me(user: dict = Depends(get_current_user)):
    return user
```

Чтобы получить `user`, FastAPI должен сначала получить `token`, а для этого - прочитать заголовок. Обход идёт от листьев к корню: сперва вычисляется `get_token`, его результат передаётся в `get_current_user`, а та отдаёт `user` обработчику. Обработчик `me` видит только конечный результат - он не подозревает, что под ним два уровня.

## Кэширование в пределах запроса

Тонкость, экономящая работу: если одна и та же зависимость встречается в графе несколько раз в рамках одного запроса, FastAPI вызовет её только один раз и переиспользует результат.

```python
def get_db():
    return {"conn": "session-42"}

def repo_users(db: dict = Depends(get_db)):
    return ("users", db["conn"])

def repo_orders(db: dict = Depends(get_db)):
    return ("orders", db["conn"])

@app.get("/dashboard")
def dashboard(u=Depends(repo_users), o=Depends(repo_orders)):
    return {"users": u, "orders": o}
```

`get_db` нужна и `repo_users`, и `repo_orders`. Но за один запрос она выполнится ровно раз, и обе зависимости получат одну и ту же сессию `session-42`. Это важно для базы: иначе один запрос открывал бы по соединению на каждое использование. Отключить кэш можно через `Depends(get_db, use_cache=False)`, но по умолчанию переиспользование - то, что нужно.

## Зависимости с yield: setup до, teardown после

Самый мощный приём - зависимость не с `return`, а с `yield`. Код до `yield` - это подготовка (setup), он выполняется до обработчика. Значение, отданное через `yield`, подставляется в параметр. А код после `yield` - это очистка (teardown), и он выполняется уже после того, как обработчик вернул ответ.

```python
def get_session():
    session = open_session()
    try:
        yield session
    finally:
        session.close()

@app.get("/users/{user_id}")
def read_user(user_id: int, session=Depends(get_session)):
    return session.fetch(user_id)
```

Канонический пример - сессия базы данных. До `yield` сессия открывается, через `yield` отдаётся обработчику, после `yield` - гарантированно закрывается, даже если обработчик бросил исключение (за это отвечает `finally`). Обработчик пользуется сессией, не заботясь о её закрытии: жизненный цикл ресурса привязан к жизненному циклу запроса.

## Зависимости ради побочного эффекта: `dependencies=[...]`

Иногда зависимость ничего не возвращает обработчику - она нужна только ради проверки. Классика - «пускать только админов». Результат такой зависимости обработчику не интересен, важен сам факт: если проверка не прошла, она бросает `HTTPException`, и до обработчика дело не доходит.

Для этого зависимость вешают не в сигнатуру, а в `dependencies=[...]` - на уровне эндпоинта, роутера или всего приложения.

```python
def verify_admin(role: str = "guest"):
    if role != "admin":
        raise HTTPException(status_code=403, detail="только для админов")

@app.delete("/items/{item_id}", dependencies=[Depends(verify_admin)])
def delete_item(item_id: int):
    return {"deleted": item_id}
```

`verify_admin` не возвращает значения и не упомянута в параметрах `delete_item` - она лишь страж. Тот же список можно повесить на `APIRouter`, и проверка применится ко всем его маршрутам, или на `FastAPI(dependencies=[...])` - и она охватит всё приложение. Один декларативный список вместо проверки прав, скопированной в каждый обработчик.

## Классы как параметризуемые зависимости

Функция-зависимость фиксирована: её поведение задано раз и навсегда. Если нужна зависимость, которую можно настроить, в роли зависимости выступает класс. FastAPI разберёт `__init__` так же, как сигнатуру функции, а сам экземпляр станет результатом.

```python
class Paginator:
    def __init__(self, max_limit: int = 100):
        self.max_limit = max_limit

    def __call__(self, skip: int = 0, limit: int = 10):
        return {"skip": skip, "limit": min(limit, self.max_limit)}

short = Paginator(max_limit=20)

@app.get("/items/")
def list_items(page: dict = Depends(short)):
    return page
```

Здесь `Paginator(max_limit=20)` создаёт настроенный экземпляр, а `Depends(short)` использует его как зависимость: FastAPI вызовет `short(skip, limit)` и применит лимит из конфигурации. Один класс - семейство зависимостей с разными настройками. Это удобно, когда у похожих эндпоинтов разные пределы.

## Краевые случаи

Несколько мест, где модель «функция до обработчика» преподносит сюрприз.

**Teardown в yield-зависимости выполняется после ответа, а не в его теле.** Код после `yield` срабатывает, когда обработчик уже вернул значение. Поэтому исключение, брошенное в teardown, не превратится в аккуратный ответ обработчику - тот уже ушёл. Закрытие ресурса в `finally` надёжно, но не пытайтесь из teardown менять уже отправленный ответ.

**Порядок зависимостей в `dependencies=[...]` соблюдается.** Список выполняется слева направо. Если первая зависимость проверяет аутентификацию, а вторая - права, ставьте их именно в этом порядке: бессмысленно проверять права раньше, чем выяснили, кто перед нами.

**Кэширование - в пределах одного запроса, не между запросами.** Переиспользование результата зависимости живёт ровно один запрос. Следующий запрос вызовет `get_db` заново. Не путайте этот кэш с долгоживущим кэшем уровня приложения - зависимость не хранит состояние между запросами сама по себе.

<Callout type="warning">
Если поставить `Depends(get_current_user())` со скобками, Вы передадите фреймворку уже вызванный результат, а не саму функцию. FastAPI ждёт вызываемый объект и вызовет его сам - передавайте `Depends(get_current_user)` без скобок.
</Callout>

## Типичные ошибки

- **Вызвали зависимость в `Depends`.** `Depends(get_db())` исполняет `get_db` один раз при старте и подсовывает фреймворку её результат как «зависимость». Нужно `Depends(get_db)` - объект функции, который FastAPI вызовет на каждый запрос.
- **Ждут результат от зависимости-стража.** Зависимость в `dependencies=[...]` ничего не кладёт в обработчик. Если её результат нужен в теле, объявляйте её параметром через `= Depends(...)`, а не в списке побочных проверок.
- **Закрывают ресурс через `return` вместо `yield`.** Зависимость с `return` не умеет выполнять teardown - после `return` функция мертва. Чтобы закрыть сессию после ответа, нужен `yield` с `finally`, иначе соединение утечёт.

<Detail summary="Почему yield, а не контекстный менеджер напрямую">

Под капотом FastAPI оборачивает yield-зависимость в асинхронный контекстный менеджер. Когда граф зависимостей разрешён и обработчик вызван, фреймворк держит генератор «на паузе» в точке `yield`, пока запрос обрабатывается. После того как ответ сформирован, он возобновляет генератор - управление возвращается в строку сразу после `yield`, отрабатывает `finally`, и генератор завершается. Именно поэтому одной yield-зависимостью можно обернуть и открытие/закрытие транзакции, и захват/освобождение блокировки: setup и teardown оказываются по разные стороны от единственного `yield`, а фреймворк гарантирует, что вторая половина выполнится. Несколько yield-зависимостей в одном запросе закрываются в порядке, обратном открытию, - как вложенные `with`.

</Detail>
