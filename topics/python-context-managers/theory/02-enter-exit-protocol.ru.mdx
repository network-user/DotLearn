# Что делает объект контекст-менеджером

В прошлом разделе `with` казался магией: он откуда-то знал, как захватить файл и как его закрыть. Магии нет - есть договор. Любой объект, у которого определены два метода с правильными именами, может встать после `with`, и язык будет вызывать эти методы в нужные моменты. Файл, блокировка, соединение с базой - все они контекст-менеджеры просто потому, что соблюдают один и тот же протокол.

К концу раздела Вы напишете собственный контекст-менеджер с нуля, будете точно знать, что и в каком порядке получает метод выхода, и поймёте тонкий, но важный момент: как именно объект может подавить исключение - и почему делать это случайно опасно.

## Ментальная модель: два метода, которые вызывает with

Контекст-менеджер - это объект с двумя методами. `__enter__` вызывается при входе в блок и возвращает то, что попадёт в переменную после `as`. `__exit__` вызывается при выходе и получает информацию о том, как блок завершился. Оператор `with` - это, по сути, обёртка, которая дёргает эти два метода в правильные моменты и в правильном порядке.

Когда Вы пишете `with cm as x`, Python делает следующее: вызывает `cm.__enter__()`, присваивает его результат имени `x`, выполняет тело, а затем при любом исходе вызывает `cm.__exit__(...)`. Всё, что нужно для управления ресурсом, помещается в эти два метода.

## Минимальный контекст-менеджер

Соберём простейший контекст-менеджер, который ничего не захватывает, а только сообщает о входе и выходе. Этого достаточно, чтобы увидеть протокол целиком.

```python
class Greeter:
    def __enter__(self):
        print("вход")
        return "привет"

    def __exit__(self, exc_type, exc, tb):
        print("выход")
        return False
```

`__enter__` печатает «вход» и возвращает строку. Эта строка - не сам объект `Greeter`, а именно то, что вернул метод. `__exit__` печатает «выход» и возвращает `False`. Зачем `False` - разберём чуть ниже; пока примите, что это означает «исключение не подавлять».

## Что попадает в переменную после as

Распространённое заблуждение: «после `as` всегда лежит сам контекст-менеджер». На самом деле там лежит результат `__enter__`, а это решает автор класса.

```python
with Greeter() as value:
    print(value)
```

Печатается `вход`, потом `привет`, потом `выход`. В `value` оказалась строка `"привет"`, потому что её вернул `__enter__`, - а не объект `Greeter()`. У файлов `__enter__` возвращает сам файл, поэтому там кажется, что `as` даёт «тот же объект»; это просто такое решение в реализации файла, а не общее правило.

Различие неочевидно, пока его не увидишь в коде. Сравните два мысленных варианта: контекст-менеджер, который захватывает соединение и возвращает в `as` курсор для работы, и контекст-менеджер таймера, которому возвращать наружу нечего. В первом случае `__enter__` отдаёт полезный объект, во втором - не отдаёт ничего, и `as` просто опускают. Автор класса сам решает, что именно станет «ручкой» для работы внутри блока, и эта свобода - часть протокола, а не его побочный эффект.

<Callout type="tip">
Если `__enter__` вообще ничего не возвращает (нет `return`), то после `as` окажется `None`. Поэтому контекст-менеджеры, которыми пользуются «ради эффекта» (например, временно меняют настройку), часто пишут вообще без `as` - возвращать нечего.
</Callout>

## Что получает __exit__

Метод выхода всегда вызывается с тремя аргументами. Если блок завершился нормально, все три равны `None`. Если в блоке возникло исключение, в них попадает информация о нём: тип, сам объект-исключение и traceback.

```python
class Watcher:
    def __enter__(self):
        return self

    def __exit__(self, exc_type, exc, tb):
        print("тип:", exc_type)
        return False
```

Вызовите `with Watcher(): pass` - напечатается `тип: None`, потому что исключения не было. Вызовите `with Watcher(): raise ValueError("x")` - напечатается `тип: <class 'ValueError'>`. По этим аргументам `__exit__` может вести себя по-разному: например, при ошибке откатить транзакцию, а при успехе - зафиксировать.

## Потрогайте руками: как with видит исключение

## Это вас удивит: return из __exit__ может проглотить исключение

Вот неочевидная часть протокола. Значение, которое возвращает `__exit__`, решает судьбу исключения. Вернёте «ложное» значение (`False`, `None`, ничего) - исключение полетит дальше, как обычно. Вернёте «истинное» (`True`) - Python сочтёт исключение обработанным и **подавит** его: блок завершится тихо, будто ошибки не было.

```python
class Swallow:
    def __enter__(self):
        return self

    def __exit__(self, exc_type, exc, tb):
        return True

with Swallow():
    raise ValueError("исчезну без следа")
print("программа продолжается")
```

Печатается только `программа продолжается`. `ValueError` возникло, но `__exit__` вернул `True`, и Python проглотил исключение. Это мощный, но опасный механизм: случайный `return True` в `__exit__` превращает контекст-менеджер в чёрную дыру, которая молча съедает любые ошибки внутри блока. Поэтому по умолчанию в `__exit__` либо явно пишут `return False`, либо вообще не пишут `return` - и подавляют исключение только осознанно и точечно.

<MarginNote label="Почему именно три аргумента">Тройка `(exc_type, exc, tb)` - это ровно то, что возвращает `sys.exc_info()`. Так исторически устроена система исключений в Python, и `__exit__` получает её в том же виде.</MarginNote>

## Типичные ошибки

- **Возвращать `True` из `__exit__` «на всякий случай».** Это подавляет все исключения в блоке, включая те, что Вы не ожидали. По умолчанию возвращайте `False` (или ничего) и подавляйте лишь конкретный тип ошибки осознанно.
- **Забыть `return self` в `__enter__`, когда ресурс нужен внутри блока.** Без `return` после `as` окажется `None`, и `with cm as x: x.do()` упадёт с `AttributeError`. Если объект нужен в теле - возвращайте его явно.
- **Делать освобождение в `__enter__`.** Захват - в `__enter__`, освобождение - в `__exit__`. Если перепутать, ресурс закроется до того, как Вы им воспользуетесь.
- **Игнорировать аргументы `__exit__`.** Транзакции, повторные попытки, логирование ошибок - всё это решается по `exc_type`. Метод, который не смотрит на свои аргументы, не умеет различать успех и провал.

<Detail summary="Полная семантика возвращаемого значения __exit__">

Правило строже, чем «`True` подавляет». Python проверяет результат `__exit__` на истинность по обычным правилам: `True`, ненулевое число, непустая строка - всё это подавит исключение. `False`, `None`, `0`, пустая строка - пропустят его дальше. Поэтому метод, который случайно заканчивается на `print(...)` без явного `return`, безопасен (`print` возвращает `None`), а метод, заканчивающийся на выражении вроде `return some_dict`, может неожиданно подавить ошибку, если словарь непустой.

Ещё тонкость: если `__exit__` сам возбудит исключение, оно заменит исходное - и про исходную ошибку Вы не узнаете. Поэтому код освобождения должен быть максимально надёжным и сам по себе не падать.

</Detail>
