# Где контекст-менеджеры решают настоящие задачи

Файл в `with` - учебный пример, на котором удобно объяснять протокол. Но настоящая ценность контекст-менеджеров проявляется там, где «забыть освободить» означает не утечку дескриптора, а зависший процесс или разорванную целостность данных. Захваченная и не отпущенная блокировка вешает все потоки, ждущие её. Транзакция, которую не зафиксировали и не откатили, оставляет базу в подвешенном состоянии. Эти ошибки дорого ловятся и легко предотвращаются одним и тем же приёмом.

В этом разделе разберём три паттерна, которые встречаются в продакшене постоянно: управление блокировками, транзакции «всё-или-ничего» и временная подмена состояния с гарантированным восстановлением. Все три - вариации одной идеи: парная операция, где вторую половину нельзя пропустить.

## Ментальная модель: симметрия захвата и отката

У каждого из этих паттернов есть «прямое» действие на входе и «обратное» на выходе. Захватить lock - отпустить lock. Начать транзакцию - зафиксировать или откатить. Подменить настройку - вернуть прежнюю. Контекст-менеджер делает эту пару неразрывной: обратное действие привязано к выходу из блока и выполнится, даже если в середине всё рухнуло.

## Блокировки: lock сам себя отпускает

В многопоточном коде `threading.Lock` защищает общий ресурс. Захватить его можно вручную через `acquire()` и `release()`, но `release()` легко пропустить при исключении - и тогда блокировка останется захваченной навсегда. `Lock` сам является контекст-менеджером: `with lock` захватывает его на входе и гарантированно отпускает на выходе.

```python
import threading

lock = threading.Lock()
shared_total = 0

def add(n):
    global shared_total
    with lock:
        shared_total += n
```

Внутри `with lock` доступ к `shared_total` защищён, и неважно, завершится блок нормально или исключением - `release()` вызовется в любом случае. Ручной вариант с `acquire()`/`finally release()` делает то же самое, но `with` короче и не оставляет шанса забыть про освобождение.

<Callout type="warning">
Захваченный и не отпущенный lock - один из худших классов багов: программа не падает, а тихо зависает, когда другой поток вечно ждёт освобождения. `with` устраняет этот риск, привязывая `release()` к выходу из блока.
</Callout>

## Транзакции: фиксируем при успехе, откатываем при ошибке

Транзакция должна примениться целиком или не примениться вовсе. Контекст-менеджер выражает это правило напрямую: на выходе он смотрит, было ли исключение, и решает - `commit` или `rollback`. Здесь впервые по-настоящему важны аргументы `__exit__`.

```python
class Transaction:
    def __init__(self, db):
        self.db = db

    def __enter__(self):
        self.db.begin()
        return self.db

    def __exit__(self, exc_type, exc, tb):
        if exc_type is None:
            self.db.commit()
        else:
            self.db.rollback()
        return False
```

Если блок отработал без ошибок, `exc_type` равен `None` - фиксируем изменения. Если внутри возникло исключение, `exc_type` не пустой - откатываем всё к состоянию до `begin()`. Метод возвращает `False`, то есть исключение после отката всё равно полетит дальше: откатить транзакцию и скрыть причину - разные вещи, и причину прятать не надо.

## Потрогайте руками: commit или rollback

Запустите код с игрушечной «базой», которая просто хранит число. В первом блоке исключения нет - изменения фиксируются. Во втором блоке намеренно происходит ошибка - и контекст-менеджер откатывает изменение, возвращая значение к исходному. Поменяйте `raise RuntimeError(...)` на `pass` во втором блоке и запустите снова: теперь зафиксируются оба изменения.

## Временная подмена состояния

Третий паттерн - временно что-то изменить и гарантированно вернуть как было. Сменили текущую директорию, подменили настройку, увеличили лимит - и на выходе откатили. Через `@contextmanager` это пишется особенно чисто.

```python
from contextlib import contextmanager

settings = {"debug": False}

@contextmanager
def temporary(key, value):
    saved = settings[key]
    settings[key] = value
    try:
        yield
    finally:
        settings[key] = saved
```

До `yield` запоминаем старое значение и ставим новое. После `yield`, в `finally`, возвращаем сохранённое - при любом исходе блока. Внутри `with temporary("debug", True)` флаг включён, а сразу после выхода он снова таков, каким был, даже если внутри блока вылетело исключение.

<MarginNote label="Готовое из стандартной библиотеки">Многие такие подмены уже реализованы: `contextlib.chdir` временно меняет рабочую директорию, `contextlib.redirect_stdout` перенаправляет вывод, `unittest.mock.patch` подменяет атрибут на время теста. Изобретать их с нуля обычно не нужно.</MarginNote>

## Это вас удивит: ресурс, открытый в __init__, не защищён

Соблазнительно открыть ресурс прямо в конструкторе контекст-менеджера, а в `__enter__` просто вернуть `self`. Это работает, пока создание объекта и вход в `with` идут рядом. Но если объект создан, а до `with` дело по какой-то причине не дошло - случилось исключение между созданием и `with`, - ресурс уже захвачен, а `__exit__` никто не вызовет.

```python
class Risky:
    def __init__(self, path):
        self.f = open(path)

    def __enter__(self):
        return self.f

    def __exit__(self, exc_type, exc, tb):
        self.f.close()
        return False
```

Если между `cm = Risky("a.txt")` и `with cm:` произойдёт ошибка, файл из `__init__` останется открытым: `__exit__` срабатывает только на выходе из блока `with`, а в блок мы так и не вошли. Надёжный приём - захватывать ресурс в `__enter__`, а не в `__init__`: тогда захват и гарантированное освобождение всегда идут парой. Конструктор пусть только запоминает параметры.

## Типичные ошибки

- **Захватывать ресурс в `__init__`.** Освобождение привязано к выходу из `with`, а захват в конструкторе случается раньше входа. Между ними может пройти исключение, и ресурс утечёт. Захват - в `__enter__`.
- **Подавлять исключение в транзакции.** `__exit__` транзакции должен откатить и вернуть `False`, чтобы причина ошибки дошла до вызывающего кода. `return True` скроет, почему откат вообще случился.
- **Ручной `acquire`/`release` для lock.** Между ними может вылететь исключение, и `release` не выполнится - поток зависнет. `with lock` решает это автоматически.
- **Забыть `finally` при временной подмене.** Без `finally` исключение в блоке оставит подменённое значение навсегда. Восстановление состояния всегда живёт в `finally` (или в `__exit__`).

<Detail summary="Вложенные контексты и порядок отката">

Когда паттерны комбинируются - например, транзакция внутри блокировки, - порядок освобождения становится важен. `with lock, Transaction(db) as conn:` сначала захватит `lock`, потом откроет транзакцию. На выходе всё пойдёт в обратном порядке: сначала закроется транзакция (commit или rollback), и только потом отпустится lock. Это правильный порядок: транзакцию нужно завершить, пока ресурс ещё защищён блокировкой, иначе другой поток может вклиниться между завершением транзакции и снятием lock.

Тот же принцип «обратного порядка» работает и в `ExitStack`, и в `with a, b, c`: что захвачено последним, освобождается первым. Поэтому, выстраивая несколько контекстов, ставьте «внешний» ресурс первым - он закроется последним и подстрахует всё, что внутри.

</Detail>
