An order list page takes one second, then two, and nothing in the code looks suspicious: fetch orders, loop, show each customer's name. The ORM hides SQL so well that `order.customer.name` looks like an ordinary in-memory field access - but behind it may be a separate database round-trip per row. This topic is about seeing hidden queries, measuring them by count, and removing extras so the page does not slow down as data grows.

**What you will learn to do.** Recognize the N+1 pattern from code shape before running a profiler; count real queries via `connection.queries` and reset the counter where you need a pinpoint measurement; lock expected query count with `assertNumQueries` tests that catch regression on CI; capture a full HTTP request profile with django-silk and read its tables; turn diagnosis into fixes with `select_related` and `prefetch_related`.

**Topic route.** Four sections, from first symptom to verified fix:

1. **N+1** - why one loop over a hundred orders becomes 101 queries and how to spot it in code.
2. **Counting** - read executed SQL from `connection.queries` and lock the count in a test.
3. **django-silk** - install profiler middleware that captures all queries for one HTTP request with timings.
4. **Reading and fixing** - from a profile with dozens of identical SELECTs, pick `select_related` or `prefetch_related` and verify with the same counter.

**What you need beforehand.** Comfort with Django ORM from the `python-orm` topic: model relations, querysets, lazy loading. No standalone server required - we work through code and numbers, and profiling tools are introduced along the way.
