# Сделать невидимые запросы видимыми

В прошлом разделе мы выяснили, что число запросов важнее времени: оно детерминировано и сразу показывает, растёт ли нагрузка вместе с данными. Осталось научиться это число получать. Django хранит список выполненных запросов прямо в объекте соединения, и достать его - вопрос одной строки. А чтобы регрессия не вернулась через месяц, это же число можно зафиксировать в тесте: `assertNumQueries(3)` упадёт, как только кто-то добавит скрытый поход в базу.

К концу раздела Вы научитесь считать запросы в любом куске кода через `connection.queries`, обнулять счётчик там, где нужно мерить точечно, и закреплять ожидаемое число запросов тестом, который ловит N+1 ещё на CI. Это два инструмента из стандартной поставки Django - никаких зависимостей.

## Ментальная модель: соединение ведёт журнал запросов

У каждого database-соединения Django есть атрибут `connection.queries` - список словарей, по одному на каждый выполненный SQL. В каждом словаре лежит сам SQL (`sql`) и время выполнения (`time`). Список накапливается за всё время жизни соединения в текущем потоке: каждый новый запрос дописывается в конец.

Длина этого списка и есть число запросов. Чтобы посчитать запросы конкретного куска кода, запоминают длину до и берут разницу после - или, что чаще, обнуляют журнал, выполняют код и смотрят `len(connection.queries)`.

<Callout type="warning">
`connection.queries` заполняется только когда `DEBUG = True`. В продакшене с `DEBUG = False` журнал не ведётся - это сделано, чтобы не тратить память на хранение SQL под нагрузкой. Считайте запросы в dev-окружении или в тестах, а не на боевом сервере.
</Callout>

## Считаем запросы напрямую

Самый прямой способ - посмотреть длину журнала до и после интересующего кода. `reset_queries()` очищает журнал, чтобы старые запросы не мешали счёту.

```python
from django.db import connection, reset_queries

reset_queries()
orders = list(Order.objects.all())
for order in orders:
    _ = order.customer.name
print(len(connection.queries))
```

Если в базе сто заказов, напечатается `101`: один запрос за списком и сто за клиентами. Поменяйте `Order.objects.all()` на `Order.objects.select_related("customer")` - и число станет `1`: связанные клиенты подтянутся тем же запросом через JOIN. Журнал делает скрытый N+1 видимым числом, на которое можно ориентироваться.

Обратите внимание на `list(...)` вокруг queryset: без него запрос ушёл бы лениво позже, и в момент `reset_queries()` он мог бы ещё не выполниться. Оборачивая queryset в `list`, Вы заставляете его выполниться здесь и сейчас, между обнулением и замером, - тогда счёт честно отражает именно этот участок кода, а не отложенный запрос, всплывший позже. Это общий приём при ручном замере: материализуйте queryset там, где хотите его посчитать.

## connection.queries показывает не только счёт, но и сам SQL

Журнал хранит и текст каждого запроса, поэтому по нему видно не только сколько, но и *что именно* ушло в базу. Это помогает отличить N+1 (много почти одинаковых запросов) от одного тяжёлого запроса.

```python
reset_queries()
list(Order.objects.all()[:3])
for q in connection.queries:
    print(q["sql"][:40], q["time"])
```

В выводе будет одна строка с началом `SELECT` и временем в секундах строкой. Когда таких строк десятки и они различаются только значением в `WHERE id = ?`, это характерная подпись N+1: один и тот же запрос, повторённый с разными ключами. Глядя на тексты, Вы видите не абстрактное «много запросов», а конкретный повторяющийся SELECT.

Этот разбор по текстам отличает N+1 от других причин медлительности. Один тяжёлый запрос с большим `time` и сложным планом лечится совсем иначе - индексом или переписыванием SQL, а не предзагрузкой связей. Журнал показывает оба случая: при N+1 Вы увидите много коротких одинаковых строк, при тяжёлом запросе - одну длинную с заметным временем. Не посмотрев на тексты, эти две проблемы легко перепутать и потратить время не на то лекарство.

<MarginNote label="time - это строка">Поле `time` в `connection.queries` - строка вроде `"0.001"`, а не число. Чтобы суммировать времена, приведите к `float`. Это частый сюрприз при попытке сложить тайминги.</MarginNote>

## assertNumQueries: зафиксировать число в тесте

Считать вручную полезно при отладке, но завтра кто-то добавит во вью обращение к новой связи, и N+1 вернётся незамеченным. Чтобы этого не случилось, Django даёт `assertNumQueries(n)` - контекстный менеджер, который проверяет, что внутри блока выполнилось ровно `n` запросов.

```python
from django.test import TestCase

class OrderQueryTests(TestCase):
    def test_list_uses_two_queries(self):
        Customer.objects.create(name="Ada")
        with self.assertNumQueries(1):
            list(Order.objects.select_related("customer"))
```

Если запросов окажется не один, тест упадёт и в сообщении покажет фактическое число и список выполненных SQL. Так ожидание «этот эндпоинт делает не больше одного запроса» становится исполняемым контрактом: добавил кто-то скрытый поход в базу - CI краснеет.

<Callout type="tip">
Пишите `assertNumQueries` на эндпоинты, где число запросов должно быть константой независимо от данных. Тогда тест ловит именно регрессию в N+1: добавили строку с ленивым доступом - число запросов выросло - тест упал.
</Callout>

## Краевой случай: число запросов зависит от данных в фикстуре

`assertNumQueries(1)` пройдёт на пустой базе даже для наивного N+1-кода: нет заказов - нет цикла - нет дополнительных запросов. Это удивляет: тест зелёный, а в проде N+1. Поэтому фикстура теста должна содержать хотя бы две связанные строки - тогда наивный код сделает минимум три запроса, и `assertNumQueries(1)` его поймает.

```python
def test_no_n_plus_one(self):
    c = Customer.objects.create(name="Ada")
    Order.objects.create(customer=c, total=10)
    Order.objects.create(customer=c, total=20)
    with self.assertNumQueries(1):
        for o in Order.objects.select_related("customer"):
            _ = o.customer.name
```

Два заказа - минимум, при котором наивный вариант (`Order.objects.all()`) дал бы три запроса вместо одного. На одном заказе разница между «1» и «1+1=2» слишком мала, чтобы уверенно ловить регрессию; на двух паттерн N+1 уже виден как рост числа запросов.

## Типичные ошибки

- **Считать запросы при `DEBUG = False`.** Журнал `connection.queries` тогда пуст, и `len(...)` всегда вернёт ноль. В тестах Django сам обеспечивает учёт запросов для `assertNumQueries`, но при ручном чтении журнала следите, чтобы `DEBUG` был `True`.
- **Забыть `reset_queries()` перед замером.** Журнал копит запросы с начала жизни соединения. Если не обнулить, в счёт попадут запросы из подготовки данных, и число будет завышенным и непонятным.
- **Жёстко зашить число запросов на пустой фикстуре.** `assertNumQueries(1)` на базе без связанных строк не отличит наивный код от оптимизированного. Кладите в фикстуру минимум две строки, чтобы N+1 проявился.

<Detail summary="CaptureQueriesContext - когда нужен и счёт, и сам SQL">

`assertNumQueries` проверяет число, но не отдаёт тексты запросов для разбора. Когда в тесте нужно и посчитать, и заглянуть в SQL, используют `CaptureQueriesContext`:

```python
from django.test.utils import CaptureQueriesContext
from django.db import connection

with CaptureQueriesContext(connection) as ctx:
    list(Order.objects.select_related("customer"))
print(len(ctx.captured_queries))
print(ctx.captured_queries[0]["sql"])
```

`captured_queries` - тот же формат, что `connection.queries`, но собранный только за время блока, без ручного `reset_queries()`. Это удобно, когда нужно проверить не только сколько запросов, но и что внутри них есть JOIN, а не подзапрос на каждую строку.

</Detail>
