# Когда ручной счётчик перестаёт справляться

`connection.queries` и `assertNumQueries` отлично работают, когда вы знаете, *где* смотреть: обернули подозрительный кусок, посчитали, зафиксировали тестом. Но реальная медленная страница не приходит с ярлыком «N+1 вот тут». Она проходит через middleware, вью, несколько сервисов, сериализатор и шаблон, и запросы рождаются по всему этому пути. Оборачивать каждый участок руками - утомительно. Нужен инструмент, который перехватит все запросы за время обработки HTTP-запроса целиком и покажет их списком с таймингами и стек-трейсами. Это и делает django-silk.

К концу раздела Вы поймёте, как silk встраивается в Django через middleware, что именно он записывает на каждый HTTP-запрос, и как читать его интерфейс, чтобы за минуту найти эндпоинт, который делает сто запросов вместо двух. Самих исправлений тут пока не будет - сначала разберёмся, как снять и прочитать профиль.

## Ментальная модель: middleware на входе и выходе запроса

silk - это middleware, которое садится в конвейер обработки HTTP-запроса. Django пропускает каждый входящий запрос через цепочку middleware сверху вниз, отдаёт его вью, а ответ прогоняет через ту же цепочку снизу вверх. silk встаёт в этой цепочке и засекает время на входе, перехватывает каждый SQL, выполненный за время обработки, и фиксирует время на выходе. Всё это он складывает в свои таблицы в базе - по записи на каждый HTTP-запрос.

Из этой картинки следует важное: silk видит *все* запросы за HTTP-запрос, включая те, что родились в шаблоне и сериализаторе, - именно туда, где `connection.queries` пришлось бы оборачивать руками. Профиль строится сам, без правок в коде вью.

## Установка и подключение

silk ставится как пакет и подключается тремя правками в настройках: добавить приложение, вставить middleware и применить миграции для его таблиц.

```python
INSTALLED_APPS = [
    "silk",
]

MIDDLEWARE = [
    "silk.middleware.SilkyMiddleware",
]
```

После этого нужно прогнать `python manage.py migrate` - silk создаёт свои таблицы для хранения профилей - и подключить его URL-ы, чтобы открыть интерфейс. UI живёт по адресу `/silk/`.

```python
urlpatterns = [
    path("silk/", include("silk.urls", namespace="silk")),
]
```

<Callout type="warning">
Позиция middleware в списке имеет значение: silk должен стоять достаточно высоко, чтобы оборачивать вью, но учитывать порядок с middleware аутентификации и gzip. И главное - silk сам пишет в базу на каждый запрос, поэтому в продакшене его держат выключенным или строго ограниченным: под нагрузкой он сам становится источником запросов.
</Callout>

## Что silk записывает на каждый запрос

Для каждого HTTP-запроса silk сохраняет запись с метаданными и привязанным к ней списком SQL-запросов. Ключевые поля видны прямо в списке запросов в UI.

| Поле | Что показывает |
|---|---|
| Path | URL и метод запроса |
| Time | сколько всего заняла обработка |
| Queries | число SQL-запросов за этот HTTP-запрос |
| Time spent on queries | сумма времени, проведённого в базе |

Колонка **Queries** - это и есть ваш счётчик запросов, но автоматический и для всего пути сразу. Эндпоинт со значением `101` в этой колонке - кандидат на N+1 ещё до того, как вы открыли его детали. А столбец времени в базе показывает, сколько из общего времени ушло именно на SQL, а не на Python.

## Читаем профиль: от списка к одному запросу

Открыв конкретный запрос в UI, вы видите вкладку с полным списком выполненных SQL, для каждого - текст, время и стек-трейс с точкой в вашем коде, откуда запрос ушёл. Подпись N+1 здесь видна глазами: длинный список почти идентичных SELECT, различающихся только значением в `WHERE`.

```text
SELECT ... FROM orders                          1.2 ms
SELECT ... FROM customers WHERE id = 1          0.4 ms
SELECT ... FROM customers WHERE id = 2          0.4 ms
SELECT ... FROM customers WHERE id = 3          0.4 ms
... ещё 97 таких же
```

Когда вы видите такой список, диагноз поставлен: один SELECT за списком и сотня одинаковых за связью. Стек-трейс у каждого из повторяющихся запросов указывает на одну и ту же строку - то место в шаблоне или сериализаторе, где происходит ленивый доступ к связи. Так silk не только считает, но и приводит вас к точке, которую надо чинить.

<Callout type="tip">
Сортируйте список запросов silk по колонке Queries по убыванию. Сверху окажутся эндпоинты с самым большим числом запросов - почти всегда это и есть ваши N+1. Время при этом может быть не самым большим: сто быстрых запросов суммарно медленнее одного среднего, но каждый по отдельности кажется быстрым.
</Callout>

## Краевой случай: silk искажает собственные измерения

silk записывает профиль в ту же базу, что и приложение, на каждый запрос. Это значит, что включённый silk сам добавляет запросы и время - и абсолютные миллисекунды в его профиле всегда немного завышены по сравнению с продом без silk. Это удивляет новичков: «я замерил, оптимизировал, а в проде всё равно другие цифры». Доверяйте silk в относительном: *число* запросов приложения он считает точно (свои служебные он не приписывает вашему вью), а абсолютное время используйте для сравнения «до и после», а не как абсолютную истину.

Кроме того, под реальной нагрузкой silk можно настроить на сэмплирование - писать профиль не для каждого запроса, а для доли, - иначе его собственная запись в базу становится узким местом. Профайлер, который меняет поведение системы, - это всегда компромисс; silk не исключение.

## Типичные ошибки

- **Оставить silk включённым в продакшене на 100% запросов.** Он пишет в базу на каждый запрос и сам становится источником нагрузки. Для прода используйте сэмплирование или держите его только в dev/staging.
- **Читать абсолютное время silk как истину.** Накладные расходы самого профайлера завышают миллисекунды. Сравнивайте «до и после» в одном и том же окружении, а не цифры silk с продом.
- **Смотреть только на общее время, игнорируя колонку Queries.** N+1 проявляется именно как большое число запросов; суммарное время может казаться терпимым, пока данных мало. Число запросов - более ранний и надёжный сигнал, чем секунды.

<Detail summary="Чем silk отличается от django-debug-toolbar">

Обе утилиты показывают SQL-запросы Django, но решают разные задачи. debug-toolbar - это панель, прибитая к HTML-странице: она показывает запросы *текущей* отрисованной страницы прямо в браузере, удобна при ручной отладке вью, отдающего HTML. silk хранит историю запросов в базе и показывает её отдельным интерфейсом, поэтому годится для API без HTML-страниц и для разбора запросов, прошедших раньше, в том числе с других машин. Грубо: toolbar - «что прямо сейчас на этой странице», silk - «журнал всех запросов, который можно листать и сортировать». Для профилирования REST-API silk обычно удобнее, потому что у API нет страницы, к которой можно прибить панель.

</Detail>
