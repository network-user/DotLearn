title: "Профилирование запросов: query counter и Silk: обзор темы"
estimatedMinutes: 3
---

Страница со списком заказов грузится секунду, потом две, а в коде ничего подозрительного: достали заказы, прошлись циклом, у каждого показали имя клиента. ORM прячет SQL так удобно, что строчка `order.customer.name` выглядит как обычное обращение к полю в памяти - а на деле за ней может стоять отдельный поход в базу на каждую строку. Эта тема о том, как увидеть скрытые запросы, измерить их числом и убрать лишние, чтобы страница не замедлялась вместе с ростом данных.

**Что вы научитесь делать.** Распознавать паттерн N+1 прямо по форме кода, ещё до запуска профайлера; считать реальные запросы через `connection.queries` и обнулять счётчик там, где нужно мерить точечно; закреплять ожидаемое число запросов тестом `assertNumQueries`, который ловит регрессию на CI; снимать полный профиль HTTP-запроса через django-silk и читать его таблицы; превращать диагноз в исправление методами `select_related` и `prefetch_related`.

**Маршрут темы.** Четыре раздела, от первого симптома до проверенного исправления:

1. **N+1** - почему один цикл по ста заказам превращается в сто один запрос и как заметить это по коду.
2. **Подсчёт** - достаём список выполненных SQL из `connection.queries` и фиксируем их число тестом.
3. **django-silk** - ставим профайлер-middleware, который перехватывает все запросы за один HTTP-запрос и показывает их с таймингами.
4. **Чтение и починка** - по профилю с десятками одинаковых SELECT выбираем `select_related` или `prefetch_related` и проверяем результат тем же счётчиком.

**Что нужно заранее.** Пригодится уверенность в Django ORM из темы «python-orm»: связи между моделями, queryset, ленивая загрузка. Самостоятельный сервер тут не нужен - примеры разбираем на коде и числах, а все инструменты профилирования вводятся по ходу.
