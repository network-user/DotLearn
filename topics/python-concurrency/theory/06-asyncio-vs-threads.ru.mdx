# Тысячи соединений в одном потоке: event loop против потоков

Потоки честно ускоряют сетевую работу, но у них есть потолок, и упираются в него быстрее, чем кажется. Каждый поток это поток ОС со своим стеком в несколько мегабайт и местом в очереди вытесняющего планировщика ядра. Пока потоков десятки, всё хорошо. Но веб-сервер, чат или парсер, которым надо держать десять тысяч одновременных соединений, попросят десять тысяч потоков, а это уже гигабайты памяти под стеки и планировщик, задыхающийся от переключений между спящими нитями. Инструмент, задуманный ради конкурентности, сам становится её ограничителем.

asyncio решает ту же IO-задачу с другого конца: тысячи одновременных операций живут в одном-единственном потоке, и ядро между ними ничего не вытесняет. В этом разделе разберём, чем кооперативная многозадачность отличается от вытесняющей, как один event loop крутит тысячи корутин, почему в однопоточном asyncio всё равно нужен свой замок, как отдать блокирующую работу в пул через `run_in_executor` и как выбрать между `threading`, `asyncio` и `multiprocessing`. Это соберёт воедино всё, что мы разобрали про GIL, гонки и примитивы синхронизации.

## Одна нить и добровольные уступки

Потоки и корутины решают одну задачу (заниматься другим делом, пока текущее чего-то ждёт), но передают управление противоположными способами. Поток вытесняется ядром: планировщик отбирает у него процессор между любыми двумя инструкциями, поток об этом не подозревает и разрешения не даёт. Корутина переключается кооперативно: она сама, в явно отмеченной точке `await`, сообщает event loop «мне сейчас нечего делать, я жду сеть, займись другими». Между двумя соседними `await` корутина выполняется целиком, и никто не прервёт её на середине.

Это различие меняет способ рассуждать о коде. В разделе про гонки мы боялись, что поток вытеснят посреди `counter += 1`; в asyncio точки, где вас могут прервать, известны заранее и видны прямо в исходнике, это ровно все `await`. Код без `await` внутри исполняется от начала до конца непрерывно. За предсказуемость приходится платить дисциплиной: если ни одна корутина добровольно не уступит управление, остальные не сдвинутся с места.

## Event loop: кто кого запускает

Сердце asyncio это event loop, цикл в одном потоке, который ведёт список готовых к работе задач. Он берёт одну готовую корутину, исполняет её до ближайшего `await` на чём-то ещё не готовом (не пришёл ответ сети, не истёк таймер), паркует её и переходит к следующей готовой. Когда ожидаемое событие случается, парконая задача снова становится готовой, и loop к ней вернётся. Никакого параллелизма на ядрах здесь нет: в каждый момент реально исполняется ровно одна корутина.

На практике это выглядит так. Функция с `async def` это корутинная функция; её вызов не запускает тело, а возвращает объект корутины, который начнёт работать, только когда его дождутся через `await` или отдадут loop как задачу. `asyncio.run` поднимает loop и крутит его до завершения переданной корутины.

```python
import asyncio

async def fetch(name):
    print(f"старт {name}")
    await asyncio.sleep(1)
    print(f"готово {name}")
    return name

async def main():
    return await asyncio.gather(fetch("A"), fetch("B"), fetch("C"))

print(asyncio.run(main()))
```

`asyncio.gather` запускает три корутины конкурентно. Все три печатают «старт», затем каждая доходит до `await asyncio.sleep(1)` и паркуется, отдавая поток соседям. Через секунду все три просыпаются и печатают «готово». Три секунды ожидания уложились в одну, и всё это в одном потоке, без единого потока ОС на задачу. Ключевая деталь тут `await asyncio.sleep`, а не `time.sleep`: только асинхронный сон возвращает управление loop, обычный подвесил бы весь поток.

<MarginNote label="История">Синтаксис `async`/`await` появился в Python 3.5 (PEP 492, 2015). До него корутины писали на генераторах с `yield from`, и читалось это заметно хуже.</MarginNote>

## Зачем замок, если поток всего один

Раз поток один и корутину не вытесняют посреди операции, легко решить, что гонок из раздела про потоки тут быть не может и синхронизация не нужна. Это верно ровно наполовину. Составную операцию без единого `await` внутри действительно никто не разорвёт. Но стоит внутри критической секции появиться `await`, и в этой точке loop вправе переключиться на другую задачу, а та увидит промежуточное, ещё не согласованное состояние.

```python
balance = 100

async def withdraw(amount):
    global balance
    if balance >= amount:
        await log_to_db(amount)
        balance -= amount
```

Между проверкой `balance >= amount` и самим списанием стоит `await`, и на нём loop может отдать управление второй задаче. Та проверит тот же баланс (первая ещё ничего не списала), тоже пройдёт условие, и обе спишут деньги. Это снова проверка-затем-действие из раздела про гонки, только точка переключения не «где угодно», а ровно на `await`. Инвариант нужно защищать и здесь, для этого есть `asyncio.Lock`: его `async with` не пускает две задачи в секцию одновременно. Отличие от потокового `Lock` в том, что `asyncio.Lock` при занятости не блокирует поток, а паркует задачу и возвращает управление loop, чтобы тот занялся другими.

## Когда одна корутина вешает весь сервер

Единственный поток это и сила, и ахиллесова пята asyncio. Если корутина вызовет блокирующую операцию, обычный `time.sleep`, синхронный `requests.get`, тяжёлый вычислительный цикл на чистом Python, она не дойдёт до `await` и не вернёт управление loop. А раз loop крутится в том же потоке, замрут разом все его задачи, все тысячи соединений. Один неосторожный синхронный вызов подвешивает весь сервер, и снаружи это выглядит как необъяснимая пауза, в которой не отвечает вообще ничего.

<Callout type="warning">
Всё, что делит event loop, должно быть неблокирующим. Синхронный вызов внутри корутины останавливает не одну задачу, а весь loop целиком.
</Callout>

Лечится это двумя способами. Если есть асинхронная версия библиотеки (`aiohttp` вместо `requests`, `asyncio.sleep` вместо `time.sleep`), берут её: она устроена так, чтобы уступать управление на `await`. Если асинхронной версии нет, а вызов блокирующий и неизбежный, его уносят в отдельный поток через `run_in_executor`, чтобы loop остался свободным.

## Как выбрать: threading, asyncio или multiprocessing

Три инструмента закрывают три разные ниши, и выбор диктует природа нагрузки, а не мода. Процессорную нагрузку не спасут ни потоки, ни asyncio: и те, и другие упираются в GIL и работают в итоге на одном ядре. Для тысяч соединений asyncio легче потоков, потому что не платит за поток ОС на каждую задачу. Для умеренного IO с обычными синхронными библиотеками проще потоки: не нужно переписывать код на `async`.

| Инструмент | Когда брать | Почему работает |
| --- | --- | --- |
| `threading` | IO-bound, десятки-сотни задач, библиотеки синхронные | на IO поток отпускает GIL и ждёт параллельно, код менять не нужно |
| `asyncio` | IO-bound, тысячи задач, есть async-библиотеки | одна нить держит тысячи соединений без веса потоков ОС |
| `multiprocessing` | CPU-bound вычисления | у каждого процесса свой GIL, настоящий параллелизм на ядрах |

Таблица не про то, что один инструмент лучше другого, а про совпадение инструмента с задачей. Спрашивайте себя: чего не хватает, процессора или уходит время на ожидание сети? И есть ли у нужных библиотек async-версии? Ответы почти всегда однозначно указывают на одну из трёх строк.

## Типичные ошибки

- **Блокирующий вызов внутри корутины.** `time.sleep`, синхронный `requests`, тяжёлый цикл не доходят до `await` и замораживают весь loop, а с ним все задачи. Фикс: async-библиотеки или вынос блокирующего в `run_in_executor`.
- **Считать, что раз поток один, синхронизация не нужна.** `await` внутри секции это точка переключения, и проверка-затем-действие через `await` это полноценная гонка. Фикс: `asyncio.Lock` через `async with` вокруг инварианта, который пересекает `await`.
- **Забыть `await`.** Вызов корутинной функции без `await` возвращает объект корутины и не исполняет тело; Python предупреждает «coroutine was never awaited». Фикс: дождитесь корутину через `await` или заведите задачу через `asyncio.create_task`.
- **Гонять CPU-bound через asyncio.** Тяжёлый счёт не отдаёт loop и не ускоряется, а вдобавок подвешивает остальные задачи. Фикс: `multiprocessing` или `run_in_executor` с `ProcessPoolExecutor`.

<Detail summary="run_in_executor: подружить асинхронный код с блокирующим">

Когда блокирующий вызов неизбежен (старая библиотека без async-версии), event loop умеет отдать его в пул, не подвешивая себя. `loop.run_in_executor(None, blocking_fn, arg)` выполняет `blocking_fn` в отдельном потоке из общего `ThreadPoolExecutor` (первый аргумент `None` означает пул по умолчанию) и возвращает awaitable. Корутина спокойно `await`'ит результат, а loop тем временем крутит другие задачи.

```python
import asyncio, requests

async def fetch(url):
    loop = asyncio.get_running_loop()
    return await loop.run_in_executor(None, requests.get, url)
```

Так блокирующий `requests.get` уезжает в поток, а loop остаётся свободным. Для CPU-bound вместо потока передают `ProcessPoolExecutor`: тогда тяжёлый счёт уходит в отдельный процесс со своим GIL, и event loop не стоит. Это мост между двумя мирами: асинхронное ядро приложения и синхронные библиотеки уживаются, потому что блокирующее всегда уносится с event loop в пул.

</Detail>
