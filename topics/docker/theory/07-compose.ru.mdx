# docker compose: многоконтейнерное приложение

В прошлом уроке, чтобы поднять приложение с базой, пришлось выполнить пять команд по порядку: создать сеть, запустить базу с томом и паролем, запустить приложение в той же сети, не забыть флаги. Стоит ошибиться в одной, и связка не работает. А теперь представьте, что то же самое должен повторить каждый коллега, на каждой машине, и помнить точный порядок и все флаги. Так рождается стопка инструкций в README, которую все ненавидят.

`docker compose` убирает эту боль. Вы описываете всё приложение, его сервисы, сети, тома и настройки, в одном текстовом файле, а поднимаете одной командой. Файл лежит рядом с кодом, читается как документация и одинаково работает у всех. К концу урока вы будете поднимать многоконтейнерное приложение командой `docker compose up` и понимать, что именно она делает за вас.

## Один файл вместо стопки команд

Compose читает файл `compose.yaml` (историческое имя `docker-compose.yml` тоже работает). В нём вы декларативно перечисляете сервисы. Сервис это описание контейнера: из какого образа его поднять или как собрать, какие порты открыть, какие переменные передать, что смонтировать. Вот то самое приложение с базой целиком:

```yaml
services:
  web:
    build: .
    ports:
      - "8080:80"
    environment:
      - APP_ENV=production
    depends_on:
      - db
  db:
    image: postgres:16
    environment:
      - POSTGRES_PASSWORD=secret
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

Читается это почти как русский текст. Сервис `web` собирается из `Dockerfile` в текущем каталоге (`build: .`), публикует порт `8080:80`, получает переменную окружения и зависит от `db`. Сервис `db` поднимается из готового образа `postgres:16`, получает пароль и монтирует именованный том `pgdata` для данных. Внизу том `pgdata` объявлен один раз, чтобы compose его создал.

## Одна команда поднимает всё

Дальше вся работа сводится к нескольким командам, которые запускают из каталога с файлом. Поднять приложение целиком:

```bash
$ docker compose up -d
```

Одна команда создаёт сеть проекта, том `pgdata`, поднимает `db`, затем `web`, кладёт оба контейнера в общую сеть и связывает по именам сервисов. Ровно то, что в прошлом уроке было пятью командами. Посмотреть, что запущено, почитать логи и остановить всё так же просто:

```bash
$ docker compose ps
$ docker compose logs -f
$ docker compose down
```

Команда `down` останавливает и удаляет контейнеры проекта вместе с созданной сетью. Важная тонкость: именованные тома она по умолчанию не трогает, поэтому данные базы переживают `down` и появляются снова при следующем `up`. Убрать и тома можно явным флагом.

<Callout type="info" title="Имя сервиса это имя в сети">
Compose сам создаёт для проекта пользовательскую сеть, поэтому сервисы находят друг друга по именам из файла. `web` обращается к базе по адресу `db:5432`, где `db` это ровно имя сервиса в `compose.yaml`. Ничего дополнительно настраивать не нужно.
</Callout>

## Что вас удивит

Первое: `depends_on` управляет порядком старта, но не ждёт готовности. Compose запустит `db` раньше `web`, но не станет проверять, успела ли база принять подключения. Контейнер базы может уже работать, а сам PostgreSQL внутри ещё подниматься. Поэтому приложение должно уметь повторять попытку подключения, а не падать с первой. Настоящую проверку готовности задают через `healthcheck` и условие в `depends_on`.

Второе: тома переживают `down`. Многих это удивляет в обе стороны. С одной стороны, `docker compose down` не сотрёт вашу базу, и это хорошо. С другой, если вы хотели чистый старт, придётся убрать тома явно, иначе новый `up` поднимет базу со старыми данными.

Третье: `build` и `image` это разные способы получить образ сервиса. `build` собирает образ из вашего `Dockerfile` прямо здесь, `image` берёт готовый из реестра. Своё приложение обычно собирают через `build`, а базы, кеши и очереди берут готовыми через `image`.

## Типичные ошибки

Ждут, что `depends_on` дождётся готовности базы, и удивляются, что приложение падает на старте с ошибкой подключения. Причина в том, что `depends_on` следит только за порядком запуска. Лекарство это `healthcheck` плюс устойчивое к повторам подключение в приложении.

Запускают `docker compose up` не из того каталога и получают ошибку про отсутствующий файл. Compose ищет `compose.yaml` в текущем каталоге. Команды compose выполняют там, где лежит файл.

Ждут чистую базу после `docker compose down` и не понимают, откуда старые данные. Именованный том пережил `down`. Для полного сброса тома убирают явным флагом.

Дублируют в файле то, что уже делает compose: вручную прописывают сеть для связи сервисов, хотя compose создаёт её сам. Сервисы одного проекта и так в общей сети и видны друг другу по именам.

<Detail summary="Compose и Dockerfile: разные задачи">

Их легко перепутать, но роли не пересекаются. `Dockerfile` описывает, как собрать один образ: слой за слоем, от базового образа до команды запуска. `compose.yaml` описывает, как запустить приложение целиком: какие сервисы поднять, как их связать, какие тома и сети создать. Часто они работают в паре: `compose.yaml` для сервиса `web` указывает `build: .`, и compose при `up` сначала соберёт образ по вашему `Dockerfile`, а потом поднимет из него контейнер. Один файл про сборку образа, другой про оркестрацию контейнеров.

</Detail>
