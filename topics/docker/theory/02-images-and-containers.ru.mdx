# Образы и контейнеры

Вы скачали `nginx` командой `docker pull nginx` и запустили его дважды. Теперь на машине два работающих веб-сервера, но образ по-прежнему один. Вы зашли в первый контейнер, поправили файл конфигурации, перезапустили, и правка на месте. Удалили контейнер, создали заново из того же образа, и правка исчезла, будто её не было. Пока эти слова «образ» и «контейнер» звучат как синонимы, происходящее выглядит капризом. Как только вы разведёте их в голове, всё встанет на места.

Разница ровно та же, что между рецептом и приготовленным блюдом или между классом и объектом. Образ это неизменяемый шаблон: зафиксированный набор файлов и настроек, из которого можно создавать сколько угодно одинаковых экземпляров. Контейнер это запущенный экземпляр образа со своим изменяемым состоянием. Один образ порождает много контейнеров, как один рецепт порождает много обедов. К концу урока вы будете уверенно отвечать на вопрос, почему правки в контейнере пропадают, а образ при этом не меняется.

## Образ это стопка слоёв только для чтения

Образ собирается по рецепту (о нём следующий урок) и состоит из слоёв, каждый из которых фиксирует одно изменение файловой системы: положили базовую систему, установили пакет, скопировали конфиг. Все эти слои доступны только для чтения. После сборки образ неизменяем, у него есть содержимое, есть уникальный идентификатор по хешу содержимого, и переписать его на месте нельзя.

Когда вы запускаете контейнер, Docker берёт слои образа как есть и добавляет сверху ещё один, тонкий и записываемый. Всё, что процесс внутри создаёт или меняет, попадает именно в этот верхний слой. Слои образа снизу остаются нетронутыми, поэтому из одного образа можно запустить хоть десять контейнеров, и каждый получит собственный записываемый слой, ничего не зная о соседях.

Отсюда сразу следует ответ на загадку из начала урока. Правка конфигурации легла в записываемый слой контейнера. Пока контейнер жив, правка на месте. Удалили контейнер, вместе с ним исчез и его верхний слой, а слои образа никогда не менялись. Создали контейнер заново, он снова начинается с чистого записываемого слоя поверх того же неизменного образа.

## Посмотреть на то и другое

У образов и контейнеров разные команды просмотра, и путать их не стоит. Образы на машине показывает `docker images`:

```bash
$ docker images
REPOSITORY   TAG       IMAGE ID       SIZE
nginx        latest    a1b2c3d4e5f6   187MB
python       3.12      0f1e2d3c4b5a   1.02GB
```

Здесь каждая строка это шаблон, из которого пока, возможно, никто ничего не запускал. Контейнеры показывает `docker ps`:

```bash
$ docker ps
CONTAINER ID   IMAGE     STATUS         NAMES
7a8b9c0d1e2f   nginx     Up 3 minutes   web-1
3c4d5e6f7a8b   nginx     Up 3 minutes   web-2
```

Обратите внимание: два контейнера, `web-1` и `web-2`, запущены из одного и того же образа `nginx`. Это и есть «один шаблон, много экземпляров» в чистом виде. У образа `IMAGE ID`, у контейнера `CONTAINER ID`, и это разные идентификаторы разных сущностей.

<Callout type="tip" title="Мнемоника">
`docker images` перечисляет то, из чего можно запуститься. `docker ps` перечисляет то, что уже запущено. Слово «images» про шаблоны, `ps` про процессы.
</Callout>

## Что вас удивит

Первое: изменения в контейнере по умолчанию никуда не сохраняются вне его. Отредактировали файл, положили загруженные пользователем картинки, накопили строки в локальной базе, всё это лежит в записываемом слое и умирает вместе с контейнером. Это не баг, а прямое следствие того, что контейнеры одноразовые. Чтобы данные пережили пересоздание, их выносят в тома, и об этом отдельный урок.

Второе: два контейнера из одного образа полностью независимы. Упал `web-1`, `web-2` даже не заметит. Записали файл в один, в другом его нет. Общее у них только неизменное дно, слои образа, а записываемые слои у каждого свои.

Третье: образ можно получить из работающего контейнера командой `docker commit`, которая «запекает» его текущее состояние в новый слой. На практике так почти не делают: такой образ непрозрачен, его нельзя пересобрать и невозможно повторить. Нормальный способ описать образ рецептом, файлом `Dockerfile`, чтобы сборка была воспроизводимой.

## Типичные ошибки

Правят файлы прямо в работающем контейнере и ждут, что изменения где-то сохранятся. После пересоздания всё пропадает, и это удивляет. Причина в записываемом слое: он часть контейнера, а не образа. Что должно быть в образе, кладут в `Dockerfile` и пересобирают; что должно пережить контейнер, выносят в том.

Путают `docker images` и `docker ps` и делают неверный вывод. Пустой `docker ps` не значит, что образов нет, он значит, что сейчас ничего не запущено. Образы при этом спокойно лежат на диске и видны через `docker images`.

Считают, что «удалить образ» и «удалить контейнер» одно и то же. Это разные операции над разными сущностями: `docker rm` убирает контейнер, `docker rmi` убирает образ. Причём образ, из которого запущен хотя бы один контейнер, просто так не удалить, Docker не даст выдернуть основание из-под работающего экземпляра.

<Detail summary="Copy-on-write: почему запуск контейнера почти бесплатный">

Записываемый слой работает по принципу copy-on-write, копирование при записи. Пока контейнер только читает файл из образа, он читает его прямо из общего слоя, никаких копий. Копия делается лишь в момент, когда контейнер решает файл изменить: тогда файл копируется в верхний записываемый слой, и правка идёт уже в копию. Поэтому запустить сотый контейнер из образа почти ничего не стоит по месту: заводится лишь пустой записываемый слой, а гигабайты образа остаются одни на всех.

</Detail>
