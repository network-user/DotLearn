# Основные команды: build, run, ps, exec, logs, stop, rm

Девяносто процентов ежедневной работы с Docker это одна и та же короткая петля. Вы собираете образ из рецепта, запускаете из него контейнер, смотрите список запущенного, при необходимости заглядываете внутрь и читаете логи, а затем останавливаете и удаляете контейнер. Семь команд, и они закрывают почти все задачи новичка. Остальное это варианты и флаги.

В прошлых уроках вы разобрались, что такое образ, контейнер и слои. Теперь научимся ими управлять руками. К концу урока вы будете уверенно проводить контейнер через весь жизненный цикл, от сборки до удаления, и понимать, в каком он состоянии на каждом шаге.

## Жизненный цикл контейнера

Контейнер это не «включён или выключен», у него несколько состояний, и команды переводят его из одного в другое. Он рождается созданным, переходит в работающий, когда стартует его главный процесс, становится остановленным, когда процесс завершился, и исчезает, когда его удаляют. Держите эту цепочку в голове: почти каждая команда урока это переход по ней.

## Собрать и запустить

Сборка превращает `Dockerfile` в образ. Флаг `-t` даёт образу имя (тег), а точка в конце указывает каталог сборки:

```bash
$ docker build -t hello .
```

Запуск создаёт из образа контейнер и стартует его главный процесс:

```bash
$ docker run hello
Привет из контейнера
```

По умолчанию `run` держит контейнер на переднем плане: вы видите его вывод, а терминал занят, пока контейнер работает. Для служб это неудобно, поэтому их запускают в фоне флагом `-d` (detached). Заодно полезно дать контейнеру имя, чтобы потом обращаться к нему не по случайному идентификатору:

```bash
$ docker run -d --name web nginx
7a8b9c0d1e2f...
```

Docker печатает длинный идентификатор контейнера и возвращает терминал. Ещё один частый флаг, `--rm`, велит удалить контейнер сразу после остановки, чтобы за собой не убирать вручную.

## Посмотреть, заглянуть, прочитать

`docker ps` показывает работающие контейнеры. С флагом `-a` он показывает все, включая остановленные:

```bash
$ docker ps
CONTAINER ID   IMAGE     STATUS         NAMES
7a8b9c0d1e2f   nginx     Up 2 minutes   web
```

Когда контейнер ведёт себя не так, есть два инструмента. `docker logs` печатает то, что процесс вывел в стандартный вывод и поток ошибок:

```bash
$ docker logs web
```

А `docker exec` запускает ещё одну команду внутри уже работающего контейнера. Чаще всего так открывают оболочку, чтобы осмотреться изнутри. Флаги `-it` дают интерактивный терминал:

```bash
$ docker exec -it web sh
```

<Callout type="warning" title="exec не меняет образ">
Всё, что вы наделаете внутри через `docker exec`, ляжет в записываемый слой контейнера и исчезнет при его удалении. Это инструмент для осмотра и отладки, а не способ настроить образ. Настройки живут в `Dockerfile`.
</Callout>

## Остановить и удалить

`docker stop` вежливо просит главный процесс завершиться: посылает сигнал `SIGTERM` и, если через несколько секунд процесс жив, добивает его `SIGKILL`. Контейнер переходит в состояние «остановлен», но с диска не исчезает:

```bash
$ docker stop web
```

Остановленный контейнер всё ещё занимает место и виден в `docker ps -a`. Чтобы убрать его совсем, есть `docker rm`:

```bash
$ docker rm web
```

Работающий контейнер так просто не удалить: Docker не даст выдернуть основание из-под живого процесса и попросит сначала остановить его. Если нужно снести на месте, добавляют `-f` (force), который сначала останавливает, потом удаляет. Образы удаляют отдельной командой `docker rmi`, ведь образ и контейнер это разные сущности.

## Что вас удивит

Первое: контейнер живёт ровно столько, сколько работает его главный процесс. Запустили `docker run` с командой, которая напечатала строку и завершилась, и контейнер тут же перешёл в «остановлен». Это не сбой: закончился процесс, закончился и контейнер. Поэтому веб-серверы внутри не «висят в фоне сами по себе», их главный процесс не завершается по своей воле.

Второе: `docker stop` не мгновенный. Он даёт процессу время на аккуратное завершение (по умолчанию около десяти секунд) и только потом убивает жёстко. Если процесс игнорирует `SIGTERM`, вы заметите паузу перед остановкой. Мгновенный и грубый вариант это `docker kill`, он сразу шлёт `SIGKILL`.

Третье: имя контейнера должно быть уникальным. Попытка запустить второй контейнер с именем `web`, пока первый существует (пусть даже остановлен), упрётся в ошибку про занятое имя. Либо удалите старый, либо дайте новому другое имя.

## Типичные ошибки

Запускают службу без `-d` и удивляются, что терминал завис. Он не завис, он занят контейнером на переднем плане. Для служб нужен фоновый режим `-d`, а вернуть терминал у уже запущенного переднего контейнера помогает сочетание, отвязывающее его без остановки.

Пытаются удалить работающий контейнер обычным `docker rm` и получают отказ. Причина в защите от удаления живого. Либо `docker stop`, потом `docker rm`, либо сразу `docker rm -f`.

Настраивают контейнер через `docker exec` и ждут, что настройка сохранится в образе. Она остаётся в записываемом слое конкретного контейнера и пропадает при пересоздании. Настройка это `Dockerfile` и пересборка.

Путают остановку и удаление. `docker stop` лишь замораживает контейнер, он остаётся на диске и всплывает в `docker ps -a`. Чтобы освободить и место, и имя, нужен `docker rm`.

<Detail summary="Передний план, фон и что делает --rm">

Без флагов `docker run` работает на переднем плане и привязывает ваш терминал к потокам контейнера. Флаг `-d` отвязывает контейнер сразу, и он уходит в фон. Позже подключиться к его выводу можно через `docker logs`, а зайти внутрь через `docker exec`.

Флаг `--rm` удобен для разовых запусков: контейнер удаляется автоматически, как только его процесс завершится, и не оседает мусором в `docker ps -a`. Для служб его обычно не ставят, ведь остановленный контейнер иногда нужно осмотреть или запустить снова. А чтобы контейнер сам поднимался после падения или перезагрузки хоста, задают политику перезапуска флагом `--restart`.

</Detail>
