# Тома и сети Docker

Вы подняли базу данных в контейнере, накопили в ней данные за неделю, а потом решили обновить версию. Обновление образа требует пересоздать контейнер, вы удаляете старый, поднимаете новый и обнаруживаете пустую базу. Неделя данных исчезла вместе с записываемым слоем. Отдельная беда рядом: ваше приложение в одном контейнере не может найти базу в другом, ведь адрес контейнера меняется при каждом запуске.

Две задачи, два решения. Тома отвечают за то, чтобы данные пережили контейнер. Сети за то, чтобы контейнеры находили друг друга. Оба механизма встроены в Docker и просты в использовании, но оба неочевидны, пока не увидишь картинку. К концу урока вы будете хранить данные так, чтобы их нельзя было потерять при пересоздании, и связывать контейнеры по понятным именам.

## Том: хранилище, которое переживает контейнер

Записываемый слой контейнера, как вы помните, умирает вместе с ним. Том решает эту проблему в лоб: это хранилище, которое живёт отдельно от контейнера и монтируется внутрь по нужному пути. Контейнер пишет в путь как обычно, но данные ложатся в том снаружи и остаются на месте, даже когда контейнер удалён.

Томов два вида, и разница между ними важна. Именованный том (named volume) создаётся и хранится самим Docker в его служебной области. Вы обращаетесь к нему по имени и не думаете, где он физически лежит:

```bash
$ docker run -d -v pgdata:/var/lib/postgresql/data postgres
```

Здесь `pgdata` это имя тома, а `/var/lib/postgresql/data` путь внутри контейнера, куда том смонтирован. Удалите контейнер, том `pgdata` останется, и следующий контейнер, смонтировав его по тому же пути, увидит все данные на месте.

Второй вид это bind mount, монтирование каталога хоста. Слева указывают путь на хосте, справа путь в контейнере:

```bash
$ docker run -d -v /home/anna/app:/app myapp
```

Теперь каталог `/home/anna/app` с вашей машины виден внутри как `/app`. Правите файл на хосте, и контейнер сразу видит правку. Именованный том берут для данных, которые ведёт само приложение, например базы. Bind mount берут для разработки, когда хочется, чтобы контейнер подхватывал ваш код без пересборки.

<Callout type="tip" title="Порядок в -v тот же, что в -p">
В паре `-v ИСТОЧНИК:ЦЕЛЬ` слева хранилище (имя тома или путь хоста), справа путь внутри контейнера. Перепутаете стороны, и монтирование сработает не туда.
</Callout>

## Сеть: контейнеры находят друг друга по имени

Теперь вторая задача. Приложению нужно достучаться до базы, но обе живут в отдельных контейнерах со своими сетевыми namespace. Использовать IP-адрес бесполезно: он назначается при запуске и меняется при пересоздании. Решение это пользовательская сеть.

Стоит поместить контейнеры в одну пользовательскую сеть, и Docker поднимает для них внутренний DNS: контейнер становится доступен соседям по своему имени. Создаём сеть и запускаем в ней оба контейнера:

```bash
$ docker network create appnet
$ docker run -d --network appnet --name db postgres
$ docker run -d --network appnet --name web myapp
```

Теперь `web` обращается к базе просто по имени `db`, например по адресу `db:5432`. Docker сам переведёт имя `db` в актуальный адрес контейнера базы, сколько бы раз его ни пересоздавали.

Заметьте: чтобы `web` достучался до `db`, публиковать порт базы через `-p` не нужно. Флаг `-p` открывает порт наружу, на хост. А внутри одной сети контейнеры общаются напрямую, по внутренним портам. Порт публикуют только для того, что должно быть доступно снаружи, обычно это веб-сервис, а не база за ним.

## Что вас удивит

Первое: у стандартной сети `bridge`, куда контейнеры попадают без указания `--network`, нет разрешения имён. Обратиться к соседу по имени там нельзя, только по адресу. Именно поэтому связывать контейнеры принято через свою, пользовательскую сеть: в ней DNS по именам работает из коробки.

Второе: bind mount и именованный том ведут себя по-разному при удалении. Данные bind mount лежат на хосте, и удаление контейнера их не трогает, они просто остаются в каталоге. Именованный том тоже переживает контейнер, но он в ведении Docker, и убрать его нужно явной командой `docker volume rm`. Забытые тома копятся и занимают место.

Третье: смонтировать том можно и поверх непустого каталога образа. Тогда содержимое тома «затеняет» то, что лежало в образе по этому пути. Это удобно для данных, но может неприятно удивить, если случайно смонтировать пустой том поверх нужных файлов.

## Типичные ошибки

Держат важные данные в контейнере без тома и теряют их при первом же пересоздании. Причина в записываемом слое, который живёт не дольше контейнера. Всё ценное монтируют в том.

Прописывают в приложении IP-адрес другого контейнера и удивляются, что после пересоздания связь рвётся. Адрес меняется, имя нет. Контейнеры кладут в общую пользовательскую сеть и обращаются по имени.

Ждут разрешения имён в стандартной сети `bridge` и не понимают, почему `db` не находится. По имени соседи видны только в пользовательской сети. Достаточно создать свою сеть и запустить контейнеры в ней.

Публикуют наружу порт базы через `-p`, чтобы к ней ходило приложение из другого контейнера. Это лишнее и небезопасно: внутри общей сети база и так доступна по внутреннему порту. Наружу публикуют только то, к чему действительно обращаются с хоста.

<Detail summary="Том, bind mount и tmpfs; драйверы сети">

У монтирования есть три вида. Именованный том (volume) в ведении Docker, лучший выбор для данных приложения. Bind mount привязывает конкретный каталог хоста, удобен для разработки и для конфигов. Tmpfs держит данные только в оперативной памяти и исчезает вместе с контейнером, его берут для временных или чувствительных данных, которым нельзя оседать на диск.

У сетей тоже несколько драйверов. `bridge` это сеть по умолчанию для контейнеров на одном хосте, именно её пользовательский вариант даёт DNS по именам. `host` убирает сетевую изоляцию, и контейнер делит сетевой стек с хостом напрямую. `none` полностью отключает контейнеру сеть. Для связки контейнеров на одной машине почти всегда берут пользовательский `bridge`.

</Detail>
