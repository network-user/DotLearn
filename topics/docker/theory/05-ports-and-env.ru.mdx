# Порты и переменные окружения

Вы запустили `docker run -d nginx`, открыли в браузере `localhost:80` и увидели пустоту. Сервер внутри контейнера работает, логи чистые, всё как надо, но снаружи до него не достучаться. Это не поломка, а прямое следствие изоляции: у контейнера собственный сетевой мир, и его порты по умолчанию не видны с хоста. Чтобы снаружи попасть внутрь, порт нужно явно опубликовать.

Порты и переменные окружения это две ручки, которыми контейнер соединяют с внешним миром и настраивают. Порт отвечает на вопрос «как до сервиса достучаться», переменная окружения на вопрос «с какими настройками он запустится». Оба механизма простые, но у обоих есть неочевидные грабли, на которые наступают все. К концу урока вы будете уверенно публиковать порты и передавать конфигурацию, не зашивая её в образ.

## У контейнера свои порты

Помните network namespace из первого урока? Благодаря ему у контейнера собственный сетевой интерфейс и собственный набор портов. `nginx` внутри слушает порт `80`, но это его личный порт `80`, а не порт вашего хоста. Пока вы не соединили одно с другим, они живут в разных мирах.

Соединяет их публикация порта флагом `-p` в формате `ХОСТ:КОНТЕЙНЕР`. Запись `-p 8080:80` означает: трафик, пришедший на порт `8080` хоста, направляй в порт `80` контейнера.

```bash
$ docker run -d -p 8080:80 nginx
```

Теперь `localhost:8080` в браузере доедет до `nginx` внутри контейнера. Рис. 1 показывает этот мостик.

Порядок в паре важен и его легко перепутать. Слева всегда порт хоста, справа порт контейнера. Хостовый порт можно выбрать любой свободный: `-p 3000:80` тоже приведёт к `nginx`, просто открывать нужно будет `localhost:3000`. А вот порт контейнера диктует само приложение: раз `nginx` слушает `80`, справа должно стоять `80`.

Отдельно стоит инструкция `EXPOSE` в `Dockerfile`. Её часто принимают за публикацию порта, но она ничего не открывает. `EXPOSE 80` это лишь пометка в образе, документация «я слушаю порт 80». Реально пробрасывает порт только флаг `-p` при запуске.

## Настройка через переменные окружения

Один и тот же образ должен работать в разных условиях: на ноутбуке разработчика, на тестовом сервере, в бою. Отличаются они настройками: адрес базы, режим отладки, секретный ключ. Зашивать настройки в образ нельзя, иначе для каждой среды пришлось бы пересобирать. Поэтому конфигурацию передают снаружи, через переменные окружения.

При запуске переменную задают флагом `-e ИМЯ=значение`:

```bash
$ docker run -e APP_ENV=production -e DEBUG=0 myapp
```

Значение по умолчанию можно запечь в образ инструкцией `ENV` в `Dockerfile`:

```dockerfile
ENV APP_ENV=development
```

Правило приоритета простое: значение, переданное при запуске через `-e`, перекрывает значение из `ENV`. Так образ несёт разумные умолчания, а конкретная среда переопределяет то, что ей нужно. Когда переменных много, их выносят в файл и подключают флагом `--env-file`.

<Callout type="warning" title="Секреты не место в ENV">
Значение из `ENV` попадает в образ и видно в его истории любому, у кого есть образ. Пароли, токены и ключи через `ENV` не передают: их отдают при запуске через `-e` или `--env-file`, а лучше через отдельные механизмы секретов. Образ должен быть безопасно публикуемым.
</Callout>

## Что вас удивит

Первое: без `-p` сервис недоступен, даже если внутри всё работает идеально. Изоляция сети это поведение по умолчанию, а не сбой. Первое, что стоит проверить, когда «сервер не отвечает», это опубликован ли порт.

Второе: хостовый порт можно занять только раз. Если `-p 8080:80` уже держит один контейнер, второй с тем же `-p 8080:80` не запустится: порт `8080` на хосте занят. Дайте второму другой хостовый порт, например `-p 8081:80`. При этом два контейнера могут спокойно слушать один и тот же внутренний порт `80`, ведь внутренние порты у них свои.

Третье: переменные окружения читаются процессом при старте. Поменяли `-e` и хотите, чтобы контейнер это увидел? Придётся пересоздать контейнер: на лету окружение живого процесса Docker не меняет.

## Типичные ошибки

Забывают `-p` и удивляются, что сервис не отвечает, хотя логи чистые. Причина в сетевой изоляции: внутренний порт по умолчанию наружу не выведен. Лекарство одно, опубликовать порт флагом `-p ХОСТ:КОНТЕЙНЕР`.

Путают порядок в паре и пишут `-p 80:8080`, ожидая открыть `nginx` на `localhost:8080`. Получают либо ошибку, либо тишину, потому что Docker шлёт трафик с хостового `80` в контейнерный `8080`, где никто не слушает. Слева хост, справа контейнер, всегда.

Принимают `EXPOSE` за публикацию и не понимают, почему после сборки порт всё ещё закрыт. `EXPOSE` только документирует, открывает порт `-p` при запуске.

Зашивают пароль в `ENV` и публикуют образ. Секрет уезжает в реестр вместе с образом и виден в истории. Секреты передают при запуске, а не запекают в образ.

<Detail summary="Варианты -p и цепочка приоритета переменных">

У флага `-p` есть варианты. Можно привязать публикацию к конкретному адресу хоста: `-p 127.0.0.1:8080:80` откроет порт только на локальном интерфейсе, недоступном извне машины. А флаг `-P` (заглавная) опубликует все порты из `EXPOSE` на случайные свободные порты хоста, узнать которые можно через `docker ps`.

Для переменных окружения приоритет выстраивается в цепочку от слабого к сильному: сначала значения из `ENV` в образе, затем значения из `--env-file`, и наконец `-e` в командной строке, которое перекрывает всё. Благодаря этому образ несёт умолчания, файл задаёт настройки среды, а точечное `-e` правит отдельное значение под конкретный запуск.

</Detail>
