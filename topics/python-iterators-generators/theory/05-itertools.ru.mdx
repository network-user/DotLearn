# Готовые кирпичи для обхода вместо самописных циклов

Вы снова и снова пишете одни и те же циклы: склеить два списка в один проход, взять первые пять элементов у чего-то бесконечного, сгруппировать подряд идущие одинаковые записи, перебрать все пары из двух наборов. Модуль `itertools` собирает эти операции в набор маленьких ленивых деталей, написанных на C и стыкующихся друг с другом как конструктор. Читаются они как названная операция, работают быстро и почти без памяти. Но у двух из них есть ловушки, на которые попадается каждый ровно один раз.

К концу раздела вы научитесь склеивать и нарезать потоки через `chain` и `islice`, ограничивать бесконечные `count`, `cycle`, `repeat`, перебирать комбинации через `product`, группировать данные через `groupby` и обходить парой стороной ловушки `groupby` и `tee`.

## itertools стыкует ленивые кирпичи

Каждая функция `itertools` принимает итераторы и возвращает итератор. Пока потребитель не потянет результат, ни один элемент не вычисляется. Из этих кирпичей собирают конвейер: комбинатор соединяет источники, срезатель берёт нужный кусок, а потребитель в конце всё материализует.

## Склеить и нарезать: chain, islice и бесконечные источники

`chain` проходит несколько итераторов подряд, как один. `islice` берёт срез итератора по правилам `range`, но без индексации и на любом, даже бесконечном источнике.

```python
from itertools import chain, islice, count

print(list(chain([1, 2], [3, 4], [5])))
print(list(islice(count(10), 3)))
```

Печатается `[1, 2, 3, 4, 5]` и `[10, 11, 12]`. `chain` склеил три списка без промежуточной копии, а `islice(count(10), 3)` отрезал три элемента у бесконечного счётчика `count`, стартующего с десяти.

Тройка `count`, `cycle`, `repeat` порождает бесконечные потоки: `count` считает вверх, `cycle` бесконечно повторяет последовательность по кругу, `repeat` повторяет одно значение. Их всегда ограничивают, обычно через `islice` или `takewhile`, иначе обход не завершится никогда.

```python
from itertools import cycle, repeat

print(list(islice(cycle("АБ"), 5)))
print(list(repeat("x", 3)))
```

Печатается `['А', 'Б', 'А', 'Б', 'А']` и `['x', 'x', 'x']`. У `repeat` есть удобная конечная форма со вторым аргументом, а вот `cycle` и `count` без внешнего ограничителя бесконечны.

## product: перебрать все комбинации

`product` даёт декартово произведение, то есть все сочетания элементов из нескольких наборов. Он заменяет вложенные циклы одним вызовом.

```python
from itertools import product

sizes = ["S", "M"]
colors = ["красный", "синий"]
print(list(product(sizes, colors)))
```

Печатается `[('S', 'красный'), ('S', 'синий'), ('M', 'красный'), ('M', 'синий')]`. Правый набор прокручивается быстрее левого, ровно как во вложенных циклах, где `colors` был бы внутренним.

## groupby: группирует только подряд идущие ключи

`groupby` собирает соседние элементы с одинаковым ключом в группы. Ключевое слово здесь «соседние»: он сравнивает только соседей и начинает новую группу, как только ключ сменился. Поэтому на неотсортированных данных один и тот же ключ рассыпается на несколько групп. Запустите пример и уберите `sorted`, чтобы увидеть, как ключ `1` разбивается на две группы вместо одной.

<PyDemo
  title="groupby до и после сортировки"
  code={`from itertools import groupby

data = [1, 1, 2, 3, 3, 3, 1]
raw = [(k, len(list(g))) for k, g in groupby(data)]
print("как есть:", raw)

ordered = [(k, len(list(g))) for k, g in groupby(sorted(data))]
print("после sorted:", ordered)`}
/>

На неотсортированном `data` получаются группы `(1, 2), (2, 1), (3, 3), (1, 1)`: единица встречается дважды, потому что её вхождения не идут подряд. После `sorted` группы становятся `(1, 3), (2, 1), (3, 3)`, и каждый ключ попадает в результат один раз. Отсюда правило: хотите сгруппировать по всему набору, сначала отсортируйте по тому же ключу, что передаёте в `groupby`.

## Где itertools удивляет

Группа внутри `groupby` это подытератор, живой только пока вы не сдвинули внешний цикл. `groupby` переиспользует один буфер, поэтому попытка сохранить группы на потом даёт пустоту или чужие элементы.

```python
from itertools import groupby

groups = list(groupby(sorted("aabbbc")))
print([(k, list(g)) for k, g in groups])
```

Печатается `[('a', []), ('b', []), ('c', [])]`, а не ожидаемые группы. К моменту печати внешний обход `groupby` уже дошёл до конца и опустошил все группы, даже последнюю. Чтобы сохранить содержимое, материализуйте его сразу внутри цикла: `[(k, list(g)) for k, g in groupby(...)]`.

Вторая ловушка это `tee`, который раздваивает один итератор на несколько. Пока ветки идут вровень, всё дёшево. Но если одна ветка убежала далеко вперёд, `tee` вынужден буферизовать всё, что между ними, и память растёт. И ещё: после `tee` исходный итератор трогать нельзя, иначе ветки потеряют элементы.

```python
from itertools import tee

source = iter([1, 2, 3, 4])
a, b = tee(source, 2)
print(list(a))
print(list(b))
```

Печатается `[1, 2, 3, 4]` дважды: обе ветки увидели полный поток, потому что исходный `source` мы больше не трогали. Тронь мы `next(source)` между строками, одно значение пропало бы из обеих веток.

Третья тонкость касается `zip`. Обычный `zip` молча останавливается на самой короткой последовательности, и лишние элементы теряются без предупреждения. Флаг `strict=True` (Python 3.10 и новее) превращает рассинхрон длин в явную ошибку.

```python
print(list(zip([1, 2, 3], ["a", "b"])))
```

Печатается `[(1, 'a'), (2, 'b')]`: тройка потерялась беззвучно. С `zip([1, 2, 3], ["a", "b"], strict=True)` вы вместо тихой потери получили бы `ValueError`, что чаще именно то, что нужно.

## Частые ошибки

Ниже промахи, которые встречаются чаще всего.

- **`groupby` без сортировки.** Он группирует только соседей. Хотите группы по всему набору, отсортируйте по тому же ключу заранее, иначе один ключ разъедется на несколько групп.
- **Сохранить группы `groupby` на потом.** Подытератор группы живёт до следующего шага внешнего цикла. Нужны элементы дальше, оберните группу в `list` прямо в теле цикла.
- **Трогать исходный итератор после `tee`.** Ветки `tee` читают через общий буфер. Обращение к оригиналу после `tee` крадёт элементы у веток.
- **Бесконечный обход `count` и `cycle`.** Без `islice` или `takewhile` эти источники не кончаются, и обход зависает. Всегда ставьте ограничитель.

<Detail summary="Почему itertools обычно быстрее самописного цикла">

Функции `itertools` реализованы на C и работают лениво: они не строят промежуточных списков и крутят внутренний цикл без возврата в байткод Python на каждый элемент. Там, где ваш цикл на каждом шаге выполняет несколько инструкций интерпретатора, `chain` или `islice` делают ту же работу в скомпилированном коде.

Официальная документация приводит и раздел «recipes»: короткие функции вроде `take`, `pairwise` (до версии 3.10) или `flatten`, собранные из базовых кирпичей. Их полезно прочитать целиком: это готовый словарь идиом обхода, которые иначе пришлось бы изобретать заново.

```python
from itertools import islice

def take(n, iterable):
    return list(islice(iterable, n))

print(take(3, range(100)))
```

Печатается `[0, 1, 2]`: `take` из рецептов берёт первые `n` элементов у любого итератора, включая бесконечный.
