# Как один `for` умеет обходить списки, файлы и словари

Вы пишете `for x in ...` поверх списка, ключей словаря, строк файла, символов строки, объекта `range` и десятков других несвязанных типов. Один и тот же оператор, а под ним объекты, у которых нет ни общего базового класса, ни общего представления в памяти. Держится этот фокус на крошечном контракте из двух методов. Разберётесь в нём один раз и перестанете видеть в `for` магию: научитесь обходить свои объекты, поймёте, почему одни вещи обходятся сколько угодно раз, а другие ровно один, и станете читать `itertools` и генераторы без удивления.

К концу раздела вы сможете вручную разложить любой `for` на вызовы `iter()` и `next()`, отличить итерируемое от итератора, написать класс, по которому можно пройтись циклом, и заранее сказать, когда второй проход не даст ничего.

## Две роли: кто отдаёт итератор и кто отдаёт элементы

В обходе участвуют две разные роли, и путать их дорого. **Итерируемое** (iterable) знает метод `__iter__`, который по запросу выдаёт свежий итератор. **Итератор** (iterator) знает метод `__next__`, который на каждый вызов отдаёт очередной элемент, а когда элементы кончились, возбуждает `StopIteration`. Ещё у итератора есть `__iter__`, и он возвращает сам объект (`self`), поэтому итератор можно подставить в `for` напрямую.

Разделение ролей объясняет привычную асимметрию: список можно обходить в цикле снова и снова, потому что каждый `for` берёт у него новый итератор через `__iter__`. А сам итератор одноразовый: он хранит позицию внутри себя и, дойдя до конца, назад не отматывается.

## Что делает `for` на самом деле

Цикл `for value in data:` Python исполняет примерно так, как написано ниже. Ровно эту разборку полезно держать в голове, когда поведение цикла кажется странным.

```python
iterator = iter(data)
while True:
    try:
        value = next(iterator)
    except StopIteration:
        break
    # тело цикла
    print(value)
```

`iter(data)` вызывает `data.__iter__()` и получает итератор. Дальше `next(iterator)` вызывает `iterator.__next__()`, пока тот не возбудит `StopIteration`; это исключение служит не ошибкой, а обычным сигналом «поток закончился», и `for` молча его гасит. Отсюда важное следствие: если ваш код случайно проглотит `StopIteration` внутри тела, цикл может оборваться там, где вы не ждали.

## Обходим руками: iter() и next()

Запустите код и подёргайте итератор вручную. Второй аргумент `next` задаёт значение по умолчанию: вместо `StopIteration` вернётся оно. Поменяйте список на строку `"абв"` и убедитесь, что символы отдаются по одному; затем допишите лишний `next(it)` в конце и поймайте исключение.

<PyDemo

print(next(it, "поток пуст"))`}

Функции `iter()` и `next()` просто оборачивают дандер-методы: `iter(x)` зовёт `x.__iter__()`, `next(x)` зовёт `x.__next__()`. Форма `next(it, default)` ловит `StopIteration` за вас и подставляет запасное значение, что удобно, когда исчерпание итератора ожидаемо, а не служит поводом падать.

## Свой итератор: класс, по которому можно пройтись

Чтобы объект заработал в `for`, дайте ему `__iter__` и `__next__`. Ниже обратный счётчик. Метод `__iter__` возвращает `self`, значит объект сам себе итератор; `__next__` отдаёт числа от старта до единицы, а на нуле возбуждает `StopIteration`.

<PyDemo
  title="Обратный счётчик как итератор"
  code={`class Countdown:
    def __init__(self, start):
        self.current = start

    def __iter__(self):
        return self

    def __next__(self):
        if self.current <= 0:
            raise StopIteration
        value = self.current
        self.current -= 1
        return value

print(list(Countdown(4)))
print(sum(Countdown(4)))`}
/>

Обратите внимание, что `list()` и `sum()` тоже работают через тот же протокол: оба внутри зовут `iter()` и крутят `next()` до `StopIteration`. Поэтому один написанный контракт открывает объекту доступ сразу ко всему, что умеет обходить: распаковке, `in`, `min`/`max`, `sorted`, генераторным выражениям.

<Callout type="warning">
Раз `__iter__` у `Countdown` возвращает `self`, экземпляр одноразовый: после `sum(c)` тот же `c` уже исчерпан. Если нужен объект, который переобходится, пусть `__iter__` возвращает новый итератор, а состояние счётчика живёт в нём, а не в самом контейнере.
</Callout>

## Где это удивляет

- **Список нельзя дёргать через `next()`.** `next([1, 2, 3])` падает с `TypeError`: у списка есть `__iter__`, но нет `__next__`. Обходить его можно, а дёргать напрямую нельзя. Сначала возьмите итератор: `next(iter([1, 2, 3]))`.
- **`iter()` от списка каждый раз новый, от итератора тот же.** `iter(data)` на списке отдаёт свежий независимый итератор при каждом вызове, поэтому два цикла по списку не мешают друг другу. А `iter(it)` на итераторе возвращает его же (`self`), так что «скопировать» позицию так не выйдет.
- **Одноразовость встроенных итераторов.** `zip`, `map`, `enumerate`, `filter`, `reversed` и генераторы возвращают итераторы, а не списки. Пройтись по результату `zip(...)` дважды не получится: первый проход выберет всё, второй увидит пустоту.

## Частые ошибки

- **Обойти результат `zip`/`map` дважды.** Классическая ловушка: сохранили `pairs = zip(a, b)`, посчитали по нему сумму, а потом хотите распечатать, и там пусто. Если данные нужны повторно, зафиксируйте их списком: `pairs = list(zip(a, b))`.
- **Вернуть из `__iter__` что-то кроме итератора.** Если `__iter__` возвращает не объект с `__next__`, цикл упадёт с `TypeError`. У класса-итератора `__iter__` почти всегда это просто `return self`.
- **Забыть возбудить `StopIteration`.** Итератор без условия остановки крутится вечно; `for` по нему не завершится никогда. Всегда должна быть ветка, поднимающая `StopIteration`.
- **Поймать `StopIteration` шире, чем нужно.** Широкий `except StopIteration` вокруг вызова, который сам обходит вложенный итератор, может проглотить чужой сигнал конца и оборвать не тот цикл.

<Detail summary="Старый протокол через __getitem__ и двухаргументный iter()">

До появления `__iter__` (PEP 234) объект считался итерируемым, если у него был `__getitem__`, принимающий целые индексы с нуля. Python и сегодня умеет это как запасной путь: если `__iter__` нет, `iter()` строит итератор, который дёргает `obj[0]`, `obj[1]`, ... пока не поймает `IndexError`. Поэтому старые классы-последовательности обходятся в `for` без единого дандера итерации.

Есть и редкая двухаргументная форма `iter(callable, sentinel)`: она делает итератор, который зовёт `callable()` на каждом шаге и останавливается, когда результат равен `sentinel`. Идиома `for block in iter(lambda: f.read(4096), ""):` читает файл кусками, пока чтение не вернёт пустую строку.

</Detail>
