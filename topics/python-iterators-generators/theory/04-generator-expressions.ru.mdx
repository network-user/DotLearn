# Один символ между списком в памяти и ленивым конвейером

Напишите `sum([x * x for x in range(10_000_000)])`, и Python честно построит список из десяти миллионов чисел, сложит их и тут же выбросит список целиком. Уберите квадратные скобки, оставив круглые, и та же сумма посчитается почти без памяти: `sum(x * x for x in range(10_000_000))`. Разница в один символ отделяет материализованный список от ленивого конвейера, по которому элементы текут по одному. Такая круглая форма называется генераторным выражением.

К концу раздела вы научитесь отличать генераторное выражение от списочного comprehension по памяти и по времени, собирать конвейеры из нескольких ленивых стадий, передавать выражение в `sum` и `any` без лишних скобок и понимать, какой кусок выражения вычисляется сразу, а какой откладывается.

## Генераторное выражение это ленивый список

Синтаксис повторяет comprehension, но круглые скобки вместо квадратных меняют суть. Списочный comprehension вычисляет и складывает все элементы в память сразу. Генераторное выражение не вычисляет ничего заранее: оно возвращает генератор, который выдаёт элементы по одному и лишь когда их просят.

Сравните две записи. Списочная версия строит промежуточный список, генераторная обходится без него:

```python
nums = [1, 2, 3, 4]
as_list = [x * x for x in nums]
as_gen = (x * x for x in nums)
print(as_list)
print(as_gen)
```

Печатается `[1, 4, 9, 16]` и что-то вроде `<generator object <genexpr> at 0x...>`. Список уже посчитан целиком, а генератор пока не тронул ни одного элемента: он лишь готов их выдать. Обойти его можно так же, как список, но только один раз.

## Свёртка без промежуточного списка

Главная выгода проявляется, когда результат сразу сворачивается функцией вроде `sum`, `max`, `any` или `all`. Если генераторное выражение это единственный аргумент функции, внешние круглые скобки самой функции служат и скобками выражения, дублировать их не нужно:

```python
nums = [1, 2, 3, 4, 5]
print(sum(x * x for x in nums))
print(any(x > 4 for x in nums))
```

Печатается `55` и `True`. Ни списка квадратов, ни списка сравнений в памяти не возникает: `sum` и `any` тянут элементы по одному. У `any` есть и второй бонус, короткое замыкание: как только нашёлся `True`, выражение перестаёт вычислять остаток.

## Конвейер из нескольких ленивых стадий

Стадии можно ставить друг за другом, и пока никто не запросил результат, данные не текут. Запустите пример: список строк проходит фильтр, преобразование в число и свёртку суммой, и всё это по одному элементу. Поменяйте порог во второй стадии с `n >= 10` на `n >= 20` и посмотрите, как из суммы уйдут числа поменьше.

<PyDemo
  title="Конвейер: строки в сумму"
  code={`lines = ["10", "", "20", "30", "", "5"]
numbers = (int(x) for x in lines if x)
big = (n for n in numbers if n >= 10)
print(sum(big))`}
/>

Каждая стадия это генератор поверх предыдущего. `numbers` тянет из `lines`, `big` тянет из `numbers`, а `sum` тянет из `big`. Ни один промежуточный список не создаётся, что и позволяет обрабатывать файлы, которые целиком в память не влезли бы. Именно так удобно читать большой лог: строка за строкой, не загружая его весь.

## Что вычисляется сразу, а что откладывается

Здесь прячется тонкость, которую любят на собеседованиях. Выражение для источника самого внешнего `for` вычисляется немедленно, в момент создания генератора. Всё остальное, включая фильтры и преобразование, откладывается до обхода.

```python
data = [1, 2, 3]
gen = (x * 10 for x in data)
data = [7, 8]
print(list(gen))
```

Печатается `[10, 20, 30]`, а не `[70, 80]`. При создании `gen` выражение `data` уже вычислилось и генератор захватил тот, первый список. Переприсваивание имени `data` новому списку на генератор не влияет, потому что он держит ссылку на исходный объект, а не на имя. А вот если бы вы не переприсвоили `data`, а мутировали его через `append` до обхода, генератор увидел бы изменения, ведь ленивый обход дошёл бы до них позже.

## Частые ошибки

Ниже промахи, которые сводят на нет выгоду генераторных выражений.

- **Лишний `list()` вокруг выражения.** Запись `sum(list(x for x in data))` сначала строит список, теряя всю экономию памяти. Если результат сразу сворачивается, скобки `list` не нужны.
- **Повторный обход исчерпанного выражения.** Генераторное выражение одноразовое. Посчитали по нему сумму, затем хотите максимум, а там пусто. Нужны два прохода, значит создайте два выражения или один список.
- **Ожидание длины и индексов.** У генераторного выражения нет `len()` и обращения по индексу: это поток, а не последовательность. Нужна длина, придётся материализовать список.
- **Промежуточные списки в конвейере.** Собирая пайплайн, легко машинально обернуть каждую стадию в список. Тогда ленивость теряется, и в памяти оказываются все стадии сразу. Держите стадии генераторами до финальной свёртки.

<Detail summary="Когда список всё-таки быстрее генераторного выражения">

Ленивость не бесплатна. На каждый элемент генератор приходится размораживать и снова замораживать, и этот межкадровый переход стоит чуть дороже, чем добавление в список инструкцией `LIST_APPEND`. Поэтому на небольших данных, которые надо целиком обойти сразу, списочный comprehension нередко оказывается быстрее по времени.

Правило выбора простое. Генераторное выражение берут, когда важна память (большие или бесконечные данные), когда результат сразу сворачивается (`sum`, `any`, `max`), или когда пригодится короткое замыкание и до конца данных вы можете не дойти. Список берут, когда данные небольшие, нужен второй проход, длина или индексация.

```python
nums = range(1000)
print(next(x for x in nums if x * x > 100))
```

Печатается `11`: `next` поверх генераторного выражения нашёл первый подходящий элемент и остановился, не проверяя остальные 989 чисел. Списочная версия проверила бы все.

</Detail>
