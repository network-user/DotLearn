Детектор предсказал рамку, разметчик нарисовал эталонную. Они почти совпали, но не пиксель в пиксель. Хорошее это предсказание или промах? «На глаз похоже» - не метрика; обучению и оценке нужна одна цифра, которая растёт, когда рамки совпадают, и падает, когда расходятся. Эта цифра - IoU, и без неё не работает ни сопоставление предсказаний с эталоном, ни подавление лишних рамок, ни подсчёт mAP.

К концу раздела Вы будете свободно переводить рамку между форматами `xywh` и `xyxy`, считать площадь пересечения двух прямоугольников через перекрытие по осям, выводить IoU как отношение пересечения к объединению и аккуратно обрабатывать случай, когда рамки не пересекаются вовсе.

## Два формата одной рамки

Прямоугольник можно задать двумя способами, и оба встречаются в коде детекторов на каждом шагу.

- **`xyxy`** - два угла: `(x1, y1, x2, y2)`, левый верхний и правый нижний. Удобен для пересечений: пересечение задаётся максимумами левых краёв и минимумами правых.
- **`xywh`** - центр и размер: `(cx, cy, w, h)`. Удобен для предсказания сетью: модель выдаёт смещение центра и размеры.

Переход между ними - чистая арифметика. Из центра в углы: `x1 = cx - w/2`, `y1 = cy - h/2`, `x2 = cx + w/2`, `y2 = cy + h/2`. Обратно: `cx = (x1 + x2)/2`, `w = x2 - x1`. Для пересечений мы всюду работаем в `xyxy`, потому что так формула пересечения прямее.

```python
def xywh_to_xyxy(cx, cy, w, h):
    return (cx - w / 2, cy - h / 2, cx + w / 2, cy + h / 2)

print(xywh_to_xyxy(10, 10, 4, 6))
```

Печатается `(8.0, 7.0, 12.0, 13.0)`: центр `(10, 10)` со сторонами `4` и `6` даёт углы `(8, 7)` и `(12, 13)`.

## IoU: пересечение, делённое на объединение

IoU (Intersection over Union) - это отношение площади пересечения двух рамок к площади их объединения. Если рамки совпадают, пересечение равно объединению и IoU = 1. Если не касаются - пересечение нулевое и IoU = 0. Всё промежуточное - между нулём и единицей, и чем больше перекрытие, тем ближе к единице.

Подвигайте предсказанную рамку на рисунке. Когда она наезжает на эталон, закрашенное пересечение растёт, и IoU ползёт вверх; когда уходит - падает. Обратите внимание: при половинном перекрытии IoU далеко не 0.5. Это важная интуиция - IoU «строже» наивного ожидания, потому что в знаменателе стоит объединение, которое всегда не меньше пересечения.

## Площадь пересечения через перекрытие по осям

Пересечение двух прямоугольников в `xyxy` - тоже прямоугольник (или пустота). Его левый край - это правый из двух левых краёв, то есть `max(ax1, bx1)`. Правый край - левый из двух правых, `min(ax2, bx2)`. Так же по вертикали. Ширина пересечения по оси - это перекрытие, и оно может оказаться отрицательным, если рамки разошлись.

```python
def overlap(a1, a2, b1, b2):
    lo = max(a1, b1)
    hi = min(a2, b2)
    return max(0, hi - lo)

print(overlap(0, 4, 2, 6), overlap(0, 2, 5, 7))
```

Печатается `2 0`: первые отрезки `[0,4]` и `[2,6]` перекрываются на `2`, вторые `[0,2]` и `[5,7]` не пересекаются, и `max(0, ...)` обрезает отрицательную ширину до нуля. Этот `max(0, ...)` - сердце корректной реализации: без него непересекающиеся рамки дадут ложную положительную площадь.

Площадь пересечения - произведение перекрытий по двум осям. Площадь объединения - сумма площадей минус пересечение (иначе общую часть посчитали бы дважды).

```python
def iou(box_a, box_b):
    ax1, ay1, ax2, ay2 = box_a
    bx1, by1, bx2, by2 = box_b
    iw = max(0, min(ax2, bx2) - max(ax1, bx1))
    ih = max(0, min(ay2, by2) - max(ay1, by1))
    inter = iw * ih
    union = (ax2 - ax1) * (ay2 - ay1) + (bx2 - bx1) * (by2 - by1) - inter
    return inter / union if union > 0 else 0.0

print(iou((0, 0, 4, 4), (2, 2, 6, 6)))
```

Печатается `0.142857...`: пересечение `(2,2)-(4,4)` имеет площадь `4`, площади рамок по `16`, объединение `16 + 16 - 4 = 28`, и `4/28 = 1/7`. Половина высоты и половина ширины перекрытия - а IoU всего около `0.14`: ещё одно напоминание, что объединение в знаменателе делает метрику строгой.

## Где это удивит

Непересекающиеся рамки без `max(0, ...)` дают катастрофу. Если рамки разошлись по обеим осям, обе «ширины перекрытия» отрицательны, их произведение - **положительное**, и без обрезки Вы получите выдуманную площадь пересечения. Метрика выдаст положительный IoU там, где рамки даже не касаются. Всегда обрезайте перекрытие по каждой оси до нуля до перемножения.

Второй сюрприз - вложенные рамки. Если маленькая рамка целиком внутри большой, пересечение равно площади маленькой, а объединение - площади большой. IoU маленькой-в-большой равен отношению площадей и может быть далёк от единицы, хотя «маленькая попала точно внутрь».

## Типичные ошибки

- **Перемножить отрицательные перекрытия.** Без `max(0, ...)` по каждой оси два отрицательных числа дают положительную «площадь». Обрезайте до нуля **до** умножения, а не после.
- **Забыть вычесть пересечение из объединения.** `union = area_a + area_b` завышает знаменатель: общая часть учтена дважды. Правильно - `area_a + area_b - inter`.
- **Считать IoU в разных форматах.** Подставить `xywh`-рамку в формулу для `xyxy` - значит интерпретировать `w` как `x2`. Приведите обе рамки к одному формату до расчёта.
- **Делить на ноль.** Если обе рамки вырождены (нулевая площадь), объединение нулевое. Возвращайте `0.0` при `union == 0`, а не падайте с ошибкой.

<Detail summary="GIoU, DIoU, CIoU - почему обычного IoU мало для обучения">

У обычного IoU есть слепое пятно: для двух непересекающихся рамок он всегда ровно ноль, независимо от того, разошлись они на пиксель или на полкадра. Для метрики качества это нормально, но как сигнал для градиентного спуска - плохо: нулевой IoU не подсказывает, в какую сторону двигать рамку. Поэтому в обучение добавляют варианты, которые остаются информативными и при нулевом перекрытии: GIoU учитывает минимальную охватывающую обе рамки коробку, DIoU - расстояние между центрами, CIoU - ещё и согласованность пропорций. Сама метрика оценки при этом остаётся обычным IoU.
