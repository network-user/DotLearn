# Горутины, каналы и select

Конкурентность в большинстве языков - библиотека поверх языка: потоки, пулы, мьютексы, обёрнутые в классы. В Go она встроена в сам язык, и это, пожалуй, его главная причина существования. Запустить функцию параллельно стоит одного слова - `go`. Горутины настолько дёшевы, что их держат тысячами там, где обычные потоки задушили бы систему. А чтобы эти тысячи не дрались за общие данные, Go предлагает не замки, а каналы - трубы, по которым горутины передают друг другу значения. Девиз языка: «не общайтесь, разделяя память; разделяйте память, общаясь».

## Горутина: функция, запущенная вбок

Поставьте `go` перед вызовом - и функция начнёт выполняться параллельно, а ваш код пойдёт дальше, не дожидаясь её.

```go
go sendEmail(user)
fmt.Println("письмо поставлено в очередь")
```

`sendEmail` работает сама по себе, а печать происходит сразу, не дожидаясь отправки. Горутина - это не поток операционной системы: их планирует сам рантайм Go, укладывая тысячи горутин на горстку системных потоков. Создать сто тысяч горутин - нормально, сто тысяч потоков - нет.

Есть одна тонкость, о которую спотыкаются все новички: функция `main` - тоже горутина, и когда она завершается, программа гаснет вместе со всеми ещё живыми горутинами. Запущенная, но не дождавшаяся своего часа горутина просто не успеет ничего сделать.

## Канал: труба между горутинами

Канал (`chan`) - это типизированная труба: в один конец значение кладут оператором `ch <- v`, из другого достают `v := <-ch`.

```go
ch := make(chan int)
go func() {
    ch <- 5
}()
fmt.Println(<-ch)
```

Программа напечатает `5`. И тут кроется вторая сила каналов: обычный (небуферизованный) канал *синхронизирует* горутины. Отправка `ch <- 5` блокируется, пока кто-то не начнёт читать, а чтение `<-ch` ждёт, пока кто-то не пришлёт. Поэтому `main` здесь не завершится раньше времени - строка `<-ch` держит её, пока горутина не отправит значение. Канал одновременно передаёт данные и согласует момент.

Каналу можно задать буфер: `make(chan int, 3)` примет до трёх значений без ожидающего читателя. Отправка блокируется, лишь когда буфер полон.

## select: ждать сразу несколько каналов

Когда горутина работает не с одним каналом, а с несколькими, на помощь приходит `select`. Он ждёт сразу несколько операций и выполняет ту ветку, чей канал готов первым.

```go
select {
case msg := <-chA:
    fmt.Println("из A:", msg)
case msg := <-chB:
    fmt.Println("из B:", msg)
default:
    fmt.Println("пока ничего")
}
```

Если готовы оба канала, `select` выбирает один случайно. Ветка `default`, если она есть, срабатывает мгновенно, когда не готов ни один канал, - так делают неблокирующее чтение. Без `default` `select` терпеливо ждёт первого готового канала.

<Callout type="warning" title="Дедлок: все спят и никто не разбудит">
Если main читает из небуферизованного канала, в который никто не пишет, программа зависнет навсегда - рантайм это распознаёт и падает с «fatal error: all goroutines are asleep - deadlock». Частая причина - забыли поставить go перед функцией-отправителем, и отправка так и не запустилась. Дедлок - не редкий экзотический сбой, а первое, что вы увидите, перепутав отправку и приём.
</Callout>

## Как дождаться нескольких горутин

Если вы запустили несколько горутин и хотите дождаться их всех, канал можно заменить на `sync.WaitGroup` - счётчик незавершённых задач:

```go
var wg sync.WaitGroup
for i := 0; i < 3; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        doWork()
    }()
}
wg.Wait()
```

`Add` увеличивает счётчик, `Done` (обычно через `defer`) уменьшает, а `Wait` блокирует, пока счётчик не обнулится. Это идиоматичный способ сказать «подожди, пока все закончат», когда результат складывать в канал не нужно.

## Типичные ошибки

**Забыть, что `main` не ждёт горутины.** Программа завершится, как только вернётся `main`, оборвав всё незаконченное. Дождитесь горутин через канал или `WaitGroup`.

**Читать из канала, в который никто не пишет.** Небуферизованный канал без отправителя вешает программу в дедлок.

**Считать, что горутины печатают по порядку.** Планировщик волен запускать их как угодно; порядок вывода независимых горутин не определён. Нужен порядок - введите его явно через каналы.

**Делить переменную между горутинами без синхронизации.** Две горутины, пишущие в одну переменную, порождают гонку данных. Передавайте значение через канал вместо общей переменной.

<Detail summary="Почему горутины дешевле потоков">

Поток операционной системы резервирует под стек мегабайт-другой памяти и переключается через ядро - это дорого, поэтому потоков заводят десятки, максимум сотни. Горутина стартует со стеком всего в пару килобайт, который растёт по мере надобности, а переключает их планировщик Go в пространстве пользователя, без похода в ядро. Рантайм мультиплексирует тысячи горутин на небольшое число системных потоков (их количество задаёт `GOMAXPROCS`). Отсюда и возможность держать сотни тысяч горутин: каждая почти ничего не стоит, пока чего-то ждёт.

</Detail>
