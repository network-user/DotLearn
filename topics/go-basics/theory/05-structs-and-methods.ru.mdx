# Структуры и методы

В Go нет классов. Для человека из мира ООП это звучит тревожно: а как же объекты, поля, поведение? Оказывается, всё это прекрасно собирается из двух простых кубиков. Структура (`struct`) группирует поля в один тип - это данные. Метод - это функция, привязанная к типу, - это поведение. Вместе они дают почти всё, ради чего в других языках заводят классы, но без наследования, конструкторов по умолчанию и прочего багажа. Разберём, как объявить структуру, повесить на неё метод и почему выбор между приёмником по значению и приёмником-указателем - не мелочь.

## Структура: набор полей под одним именем

Структура описывает, из каких полей состоит значение вашего типа.

```go
type User struct {
    Name string
    Age  int
}
```

Создать значение можно, перечислив поля по имени. Так надёжнее всего - порядок и число полей потом можно менять, не ломая вызовы:

```go
u := User{Name: "Анна", Age: 30}
```

К полям обращаются через точку: `u.Name` вернёт `"Анна"`, а `u.Age = 31` изменит возраст. Если объявить структуру без значений, `var u User`, все поля получат нулевые значения своих типов: `Name` станет пустой строкой, `Age` - нулём. Отдельного «конструктора», который надо звать, здесь нет.

## Метод: функция, привязанная к типу

Метод отличается от обычной функции одним - у него есть *приёмник*, указанный в скобках перед именем. Приёмник говорит, к какому типу привязан метод.

```go
func (u User) Greet() string {
    return "Привет, " + u.Name
}
```

Теперь у любого значения `User` есть метод `Greet`:

```go
u := User{Name: "Анна"}
fmt.Println(u.Greet())
```

Внутри метода приёмник `u` работает как обычный параметр: через него читают поля структуры. По сути метод - это синтаксически удобная функция, которой первым аргументом неявно передают само значение.

## Приёмник по значению против приёмника-указателя

Вот здесь кроется различие, которое ловит почти каждого новичка. Приёмник по значению (`u User`) получает *копию* структуры. Изменения внутри метода не видны снаружи - вы правите копию, а оригинал остаётся прежним. Приёмник-указатель (`u *User`) получает адрес оригинала, и через него структуру можно менять.

```go
func (c Counter) IncValue() { c.n++ }
func (c *Counter) IncPtr()  { c.n++ }
```

`IncValue` увеличивает поле копии - после вызова оригинал не изменится. `IncPtr` увеличивает поле оригинала. Правило на каждый день: если метод должен *менять* структуру, приёмник обязан быть указателем. Читающим методам достаточно значения, но для единообразия внутри одного типа приёмники обычно делают одного вида.

<Callout type="tip" title="Go сам берёт адрес - и это скрывает разницу">
Вызывая `c.IncPtr()` на обычной переменной c, вы не пишете `(&c).IncPtr()` - Go подставляет взятие адреса автоматически. Из-за этого удобства разница между IncValue и IncPtr на месте вызова незаметна: оба зовутся как `c.Inc()`. Понять, меняет ли метод структуру, можно только заглянув в его сигнатуру - есть ли там звёздочка.
</Callout>

## Типичные ошибки

**Ждать, что метод с приёмником-значением что-то изменит.** Он работает с копией. Счётчик, «увеличенный» методом с приёмником по значению, останется на месте - нужен приёмник-указатель.

**Создавать структуру позиционно и ломаться при добавлении поля.** `User{"Анна", 30}` завязан на порядок полей. Добавили поле в середину - и все такие литералы поехали. Инициализация по именам `User{Name: ..., Age: ...}` от этого защищена.

**Искать наследование.** Его нет. Повторное использование в Go достигают вложением структур (embedding) и интерфейсами, а не иерархией классов.

<Detail summary="Встраивание: композиция вместо наследования">

Структуру можно вложить в другую, указав тип без имени поля. Тогда поля и методы вложенного типа становятся доступны напрямую у внешнего:

```go
type Animal struct {
    Name string
}

type Dog struct {
    Animal
    Breed string
}

d := Dog{Animal: Animal{Name: "Рекс"}, Breed: "лайка"}
fmt.Println(d.Name)
```

`d.Name` работает, хотя `Name` объявлен в `Animal`: встроенный тип «поднимает» свои поля наверх. Это похоже на наследование, но это композиция - `Dog` содержит `Animal`, а не «является» им. Го намеренно подталкивает вас собирать поведение из частей, а не выстраивать деревья классов.

</Detail>
