# Интерфейсы и утиная типизация

Интерфейсы - то место, где Go расходится с Java и C# сильнее всего. Там тип обязан во всеуслышание объявить: «я реализую интерфейс Comparable». В Go никто ничего не объявляет. Если у типа есть нужные методы, он подходит под интерфейс - молча, автоматически, без единого ключевого слова связи. Это и называют утиной типизацией: если нечто крякает как утка, значит, для наших целей это утка. Такой подход развязывает руки: вы можете написать интерфейс уже после того, как типы созданы, и даже для типов из чужого пакета, который вы не в силах изменить.

## Интерфейс - это список методов

Интерфейс описывает *поведение*: набор методов, которые тип должен иметь. Не поля, не данные - только методы.

```go
type Shape interface {
    Area() float64
}
```

Всё, что имеет метод `Area() float64`, является `Shape`. Тип не пишет «я Shape» - соответствие проверяет компилятор по факту наличия метода.

```go
type Square struct{ side float64 }

func (s Square) Area() float64 { return s.side * s.side }
```

`Square` нигде не упоминает `Shape`, но удовлетворяет ему, потому что у него есть `Area() float64`. Теперь `Square` можно передать всюду, где ждут `Shape`.

Функция, принимающая интерфейс, работает с любым подходящим типом, не зная его конкретики:

```go
func describe(sh Shape) {
    fmt.Println("площадь:", sh.Area())
}
```

Передайте туда `Square`, `Circle` или что угодно с методом `Area` - `describe` вызовет нужную реализацию. Это и есть полиморфизм в Go.

## Пустой интерфейс: контейнер для чего угодно

Интерфейс без методов, `interface{}` (в современном Go у него есть псевдоним `any`), удовлетворяется вообще любым типом - ведь требований ноль. Он держит значение любого вида и нужен там, где тип заранее неизвестен, например у `fmt.Println`, принимающего любые аргументы.

```go
var x any = 42
x = "теперь строка"
```

Расплата за универсальность - вы теряете статическую типизацию: чтобы что-то сделать со значением внутри `any`, придётся сначала выяснить его настоящий тип.

## Type assertion: достать конкретный тип обратно

Извлечь исходный тип из интерфейсного значения помогает *type assertion*. Безопасная форма возвращает значение и признак успеха:

```go
var x any = "привет"
s, ok := x.(string)
```

`s` станет `"привет"`, `ok` - `true`. Если бы внутри лежало не строка, `ok` был бы `false`, а `s` - пустой строкой, и паники бы не случилось. Когда вариантов несколько, вместо цепочки проверок пишут *type switch*:

```go
switch v := x.(type) {
case int:
    fmt.Println("число", v)
case string:
    fmt.Println("строка", v)
default:
    fmt.Println("другой тип")
}
```

<Callout type="warning" title="Одиночная форма assertion паникует при промахе">
Форма s := x.(string) без второго результата удобна, но опасна: если внутри интерфейса лежит не строка, программа не вернёт false, а аварийно завершится паникой. В коде, где вы не уверены в типе, всегда используйте форму с ok - она превращает промах в обычную проверку вместо падения.
</Callout>

## Типичные ошибки

**Класть в интерфейс поля.** Интерфейс описывает только методы. Общие данные выражают структурами, а не интерфейсами.

**Проектировать огромные интерфейсы.** В Go ценят маленькие интерфейсы в один-два метода. Чем меньше метода требует интерфейс, тем больше типов ему удовлетворяет и тем гибче код.

**Злоупотреблять `any`.** Пустой интерфейс выключает проверки типов. Каждый `any` - это отложенная на время выполнения ошибка, которую компилятор больше не ловит. Берите его только когда тип действительно заранее неизвестен.

<Detail summary="Интерфейсное значение - это пара (тип, значение)">

Внутри переменная-интерфейс хранит две вещи: конкретный тип, который сейчас в ней лежит, и само значение. Отсюда знаменитая тонкость: интерфейс, содержащий указатель со значением `nil`, сам *не равен* `nil` - ведь тип-то в нём записан. Проверка `if err != nil` может неожиданно оказаться истинной, если в интерфейс `error` положили типизированный `nil`-указатель. Это редкий, но болезненный баг; пока достаточно помнить, что «пустой» и «содержит nil-указатель» - для интерфейса разные состояния.

</Detail>
