# Обработка ошибок

В Python или Java ошибку бросают вверх исключением, и она летит по стеку, пока кто-нибудь её не поймает. В Go так не делают. Здесь ошибка - это обычное значение, которое функция возвращает наравне с результатом, а вызвавший обязан на него посмотреть. Отсюда та самая идиома, которую вы увидите в любом Go-коде буквально через строчку: `if err != nil`. Кому-то она кажется многословной, но у неё есть свойство, которого нет у исключений: по коду сразу видно, где может пойти что-то не так, потому что обработка стоит прямо на месте вызова, а не прячется в далёком `catch`.

## error - это интерфейс

`error` - встроенный интерфейс с единственным методом:

```go
type error interface {
    Error() string
}
```

Всё, что умеет вернуть текст ошибки методом `Error()`, является ошибкой. Функция, которая может не справиться, возвращает ошибку последним значением. По соглашению `nil` в этом месте означает «всё хорошо».

## Идиома if err != nil

Разберём типичную функцию, которая может не справиться:

```go
func divide(a, b int) (int, error) {
    if b == 0 {
        return 0, errors.New("деление на ноль")
    }
    return a / b, nil
}
```

При делении на ноль она возвращает нулевой результат и ошибку, созданную `errors.New`. В нормальном случае - результат и `nil`. Вызывающий проверяет ошибку сразу:

```go
result, err := divide(10, 0)
if err != nil {
    fmt.Println("не получилось:", err)
    return
}
fmt.Println(result)
```

Пока `err` не проверена, использовать `result` нельзя - он бессмыслен, если функция не справилась. Проверка стоит вплотную к вызову, и в этом вся идея: путь ошибки виден прямо здесь, а не прячется где-то выше.

## Создание и оборачивание ошибок

Простую ошибку создают через `errors.New`. Когда нужно добавить контекст, но сохранить исходную ошибку внутри, используют `fmt.Errorf` с глаголом `%w` (*wrap*):

```go
data, err := readFile(path)
if err != nil {
    return fmt.Errorf("загрузка конфига: %w", err)
}
```

Такая обёртка добавляет к сообщению фразу «загрузка конфига», но не теряет оригинал: позже его можно достать через `errors.Is` или `errors.As`. Так на ошибке накапливается «маршрут» - откуда она пришла и через какие слои прошла, - и по логу видно всю цепочку.

<Callout type="info" title="Errorf с %w сохраняет ошибку, %v - только текст">
Глагол %w встраивает исходную ошибку внутрь новой так, что errors.Is и errors.As смогут её распознать. Глагол %v лишь подставит её текст как строку, потеряв возможность позже сопоставить ошибку с известной. Оборачиваете, чтобы сохранить причину, - берите %w.
</Callout>

## panic и recover: для по-настоящему исключительного

У Go всё же есть механизм, похожий на исключения, - `panic` и `recover`. Но применяют его не для обычных ошибок вроде «файл не найден», а для ситуаций, из которых нормально продолжать нельзя: выход за границы слайса, разыменование `nil`, нарушение инварианта. `panic` разворачивает стек, выполняя по пути все `defer`, а `recover`, вызванный внутри отложенной функции, может перехватить панику и вернуть выполнение в норму.

```go
defer func() {
    if r := recover(); r != nil {
        fmt.Println("восстановились после:", r)
    }
}()
```

Правило простое: ожидаемые проблемы - через возвращаемую ошибку, непредвиденные катастрофы - через панику. Использовать панику как обычный поток управления в Go считается дурным тоном.

## Типичные ошибки

**Игнорировать возвращённую ошибку.** `result, _ := divide(...)` выбрасывает ошибку в пустой идентификатор. Иногда это осознанно, но чаще - спрятанный баг.

**Оборачивать через `%v` вместо `%w`.** С `%v` вы теряете исходную ошибку и больше не сможете сопоставить её через `errors.Is`.

**Пробрасывать панику вместо ошибки.** `panic` на «пользователь ввёл не то» - перебор. Обычные, ожидаемые проблемы возвращают ошибкой.

**Использовать `result` до проверки `err`.** Пока ошибка не проверена, результат может быть бессмысленным нулевым значением.

<Detail summary="Сигнальные ошибки и errors.Is">

Иногда вызвавшему важно отличить конкретную ошибку - например, «запись не найдена» - от любой другой. Для этого заводят *сигнальную* (sentinel) ошибку как переменную пакета:

```go
var ErrNotFound = errors.New("не найдено")
```

Функция возвращает именно её, а вызвавший сравнивает через `errors.Is(err, ErrNotFound)`. Почему не обычное `err == ErrNotFound`? Потому что ошибку могли обернуть по дороге через `%w`, и прямое сравнение промахнётся. `errors.Is` разворачивает всю цепочку обёрток и находит исходную ошибку внутри.

</Detail>
