# Generics: типы с параметром

Функция «вернуть первый элемент массива» одинаково работает и с числами, и со строками, и с объектами. Как её типизировать? Написать `number[]` - и она откажется работать со строками. Написать `any[]` - и она заработает со всем, но потеряет тип: результат станет `any`, и редактор перестанет знать, что вы достали из массива чисел именно число. Обе крайности плохи. Нужен способ сказать «функция работает с массивом любого типа, но результат имеет ровно тот же тип, что и элементы». Это и делают generics - типы с параметром.

Generic - это тип, у которого часть оставлена переменной. Как функция принимает значение аргументом, так generic принимает тип параметром. Разберёмся, как объявлять обобщённые функции и типы, как TypeScript сам подставляет параметр по аргументам и как ограничить его, чтобы внутри можно было хоть что-то делать со значением.

## Параметр типа

Обобщённую функцию объявляют с параметром типа в угловых скобках перед списком аргументов:

```ts
function first<T>(arr: T[]): T {
  return arr[0];
}
```

`T` здесь - параметр типа, обычная переменная, только для типов. Читается так: `first` работает с массивом каких-то элементов типа `T` и возвращает значение того же типа `T`. Связь между входом и выходом теперь записана в типе: что за тип пришёл в массиве, тот и вернётся.

Самое приятное - параметр обычно не нужно указывать при вызове. TypeScript выводит `T` из аргумента:

```ts
const n = first([10, 20, 30]);
const s = first(['Москва', 'Казань']);
```

В первом вызове `T` вывелся как `number`, поэтому `n` имеет тип `number`. Во втором `T` стал `string`, и `s` - строка. Одна функция, а результат каждый раз точного типа. Именно этого не смог бы дать `any[]`: там результат всегда был бы `any`.

## Обобщённые типы

Параметр типа бывает не только у функций, но и у интерфейсов и псевдонимов. Так описывают контейнеры - структуры, которые хранят значение любого типа:

```ts
interface Box<T> {
  value: T;
}

const numberBox: Box<number> = { value: 2999 };
const cityBox: Box<string> = { value: 'Казань' };
```

`Box<T>` - шаблон: `Box<number>` требует `value` числом, `Box<string>` - строкой. В отличие от вызова функции, здесь параметр указывают явно в угловых скобках. Именно так устроены встроенные `Array<T>`, `Promise<T>`, `Map<K, V>`: это обобщённые типы, у которых вы каждый раз подставляете свой параметр. `Promise<string>` - обещание строки, `Map<string, number>` - словарь со строковыми ключами и числовыми значениями.

Параметров может быть несколько:

```ts
function pair<K, V>(key: K, value: V): [K, V] {
  return [key, value];
}
```

`pair('Казань', 2999)` вернёт кортеж `[string, number]`: `K` вывелся строкой, `V` числом.

## Ограничения: extends

У чистого `T` есть предел: внутри функции про него ничего не известно, поэтому и сделать с ним почти ничего нельзя. Нельзя обратиться к `value.length` - вдруг `T` окажется числом, у которого длины нет. Чтобы позволить операции, параметр ограничивают через `extends`:

```ts
function longest<T extends { length: number }>(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}
```

`T extends { length: number }` означает «`T` - любой тип, у которого есть поле `length` числом». Теперь `a.length` разрешён: TypeScript гарантирует наличие поля. При этом сам `T` остаётся конкретным: `longest('Москва', 'Казань')` вернёт `string`, `longest([1, 2], [3])` вернёт массив. Ограничение расширяет то, что можно делать внутри, не теряя точности результата.

<Callout type="tip" title="Ограничение не превращает результат в тип ограничения">
Легко подумать, что longest вернёт `{ length: number }`. Нет: возвращается T - конкретный тип аргумента. Ограничение через extends - это требование к параметру, а не замена его типа. Строку вы получите строкой, массив массивом, а поле length лишь разблокировало доступ внутри функции.
</Callout>

## Крайние случаи

Иногда вывод `T` даёт более широкий тип, чем ожидалось. Для `first([1, 2, 3])` вывелся `number` - хорошо. Но для смешанного `first([1, 'два'])` параметр `T` станет `number | string`, и результат - union, который придётся сужать. Вывод честно берёт самый узкий тип, покрывающий все аргументы; если аргументы разнородны, `T` расширяется.

Второй момент - однобуквенные имена вроде `T` это лишь традиция, а не требование. `T` для «type», `K` для «key», `V` для «value», `E` для «element» - конвенция, которую все читают с ходу. Для сложных обобщений понятное имя вроде `TItem` читается лучше загадочной буквы. Компилятору всё равно, важна читаемость.

## Частые ошибки

**Брать `any` там, где нужен generic.** `any` выключает проверку и теряет связь входа с выходом. Generic сохраняет тип насквозь: массив чисел отдаёт число. Если функция должна работать с разными типами, но помнить, с каким именно, - это generic, а не `any`.

**Работать с неограниченным `T`, будто про него всё известно.** У чистого `T` нет ни полей, ни методов. Хотите обращаться к `length` или `id` - объявите это через `extends`.

**Указывать `T` вручную, когда он выводится.** `first<number>([1, 2])` избыточно: `T` и так выведется из аргумента. Явное указание нужно лишь там, где вывести неоткуда.

**Плодить лишние параметры типа.** Если параметр типа встречается в сигнатуре ровно один раз, он, скорее всего, не нужен и его стоит заменить конкретным типом. Каждый `T` должен связывать хотя бы два места.

<Detail summary="Почему это называется «обобщённым программированием»">
Идея не нова и старше TypeScript на десятилетия: обобщённые типы (generics) появились в языках вроде Ada и ML, а в мейнстрим их принесли шаблоны C++ и дженерики Java и C#. Суть везде одна - написать алгоритм один раз для «любого типа», не жертвуя проверкой типов и не копируя код под каждый конкретный случай. Контейнеры (список, словарь, дерево), функции над коллекциями (map, filter), обёртки вроде Promise - всё это естественно обобщается по типу элемента. TypeScript добавил к классической идее мощный вывод: параметр почти всегда подставляется сам, поэтому в повседневном коде дженерики выглядят почти незаметно - вы пользуетесь Array и Promise, не думая, что это они и есть.
</Detail>
