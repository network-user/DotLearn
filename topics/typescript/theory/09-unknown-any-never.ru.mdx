# unknown, any и never

Три типа стоят особняком от `number` и `string`: `any`, `unknown` и `never`. Они описывают не конкретные значения, а крайние положения самой системы типов. `any` - «отключить проверку здесь». `unknown` - «сюда влезет что угодно, но пользоваться без проверки нельзя». `never` - «сюда не влезет ничего, значения этого типа не существует». Звучит абстрактно, но именно эта тройка отделяет крепкий TypeScript-код от дырявого: неверный выбор между `any` и `unknown` тихо открывает дыру, а понимание `never` даёт проверку на полноту, которую компилятор делает за вас.

Разберёмся, чем `unknown` безопаснее `any`, почему `any` стоит считать красным флагом и откуда берётся тип, у которого нет ни одного значения.

## any: выключатель проверки

`any` означает «любой тип, и перестань проверять». Значение типа `any` можно использовать как угодно: вызывать любые методы, обращаться к любым полям, присваивать куда угодно.

```ts
let data: any = 'Казань';
data.toFixed(2);
data.foo.bar.baz;
```

Ни одна из этих строк не вызовет ошибки типа, хотя `toFixed` у строки нет, а `data.foo` - обращение к несуществующему полю. `any` выключил проверку целиком, и все гарантии, ради которых берут TypeScript, здесь испарились. Опасность `any` ещё и заразна: результат операции над `any` тоже `any`, поэтому одна дыра расползается по коду, который её касается.

Иногда `any` неизбежен - при постепенной миграции старого JavaScript или на границе с совсем нетипизированной библиотекой. Но каждый `any` стоит воспринимать как временную заплату и красный флаг, а не как рабочий инструмент.

## unknown: безопасный родственник any

`unknown` - это «сюда можно положить значение любого типа», как и `any`. Разница в другом конце: со значением `unknown` ничего нельзя сделать, пока вы не докажете компилятору его тип.

```ts
let data: unknown = 'Казань';
data.toUpperCase();
```

Вторая строка - ошибка: «Object is of type unknown». TypeScript не даёт вызвать метод, пока не знает, что `data` - строка. Чтобы работать со значением, его сужают - теми же приёмами, что и union:

```ts
if (typeof data === 'string') {
  data.toUpperCase();
}
```

Внутри проверки `data` сужен до `string`, и метод разрешён. `unknown` заставляет вас проверить тип перед использованием - ровно то, чего не делает `any`. Поэтому `unknown` - правильный тип для всего, что приходит извне: ответа `fetch`, результата `JSON.parse`, данных из хранилища. Вы принимаете «что угодно», но обязаны разобрать это, прежде чем доверять.

## never: тип без значений

`never` - тип, у которого нет ни одного значения. На первый взгляд бессмыслица: зачем тип, куда нельзя ничего положить. Но он естественно возникает в двух ситуациях.

Первая - функция, которая никогда не возвращает управление нормально: всегда бросает исключение или крутит бесконечный цикл. Ей нечего вернуть, и это честно выражает `never`:

```ts
function fail(message: string): never {
  throw new Error(message);
}
```

Вторая, куда более полезная, - проверка на полноту при разборе union. Когда все варианты разобраны, в ветку-иначе «доходит» пустой тип `never`. Присвоив его специальной переменной, вы просите компилятор следить за полнотой:

```ts
type Payment = { kind: 'card' } | { kind: 'cash' };

function pay(p: Payment): string {
  if (p.kind === 'card') return 'карта';
  if (p.kind === 'cash') return 'наличные';
  const rest: never = p;
  return rest;
}
```

Пока разобраны оба варианта, в `rest` попадает `never`, и всё компилируется. Добавят третий вариант `{ kind: 'crypto' }`, а обработку забудут - в `rest` попытается попасть этот вариант, но в `never` он не влезет, и TypeScript подсветит ошибку. Компилятор напоминает дописать ветку. Это тот же приём, что упоминался в главе про union, теперь с именем для типа в его основе.

<Callout type="tip" title="Почему never, а не пустой объект">
never - «низ» системы типов: он совместим с любым типом, потому что множество его значений пусто, а пустое множество вкладывается в любое. Поэтому never можно вернуть из функции, ожидающей string: раз значения не существует, противоречия не возникает. Это делает never идеальным маркером «сюда исполнение не дойдёт».
</Callout>

## Крайние случаи

Тонкость про `any` и совместимость: `any` совместим в обе стороны - его можно присвоить чему угодно и в него присвоить что угодно. Из-за этого `any` иногда проскакивает мимо проверок незаметно: функция, случайно вернувшая `any`, отравляет типы вызывающего кода, а редактор молчит. Строгий режим включает предупреждения на неявный `any`, и это одна из главных причин его включать.

С `unknown` есть обратная приятная тонкость: его нельзя присвоить в переменную конкретного типа без сужения. `const s: string = someUnknown` - ошибка. Это и есть защита: `unknown` не даёт «протечь» в типизированный код неразобранным.

## Частые ошибки

**Ставить `any`, чтобы «убрать красную волну».** Ошибка исчезнет, но вместе с проверкой. Почти всегда правильный ответ - `unknown` с последующим сужением или точный тип, а не `any`.

**Типизировать внешние данные как `any`.** Ответ сети - классический кандидат в `unknown`: примите что угодно, но разберите перед использованием. `any` здесь просто отложит ошибку до рантайма.

**Пытаться создать значение типа `never`.** Его не существует. `never` появляется сам - как результат полного сужения или тип не возвращающей функции; присвоить в него можно только в целях проверки на полноту.

**Забывать, что `any` заразен.** Одно значение `any` в цепочке вычислений делает `any` весь результат. Локализуйте его и сузьте как можно раньше.

<Detail summary="unknown, any и «верх/низ» системы типов">
В теории типов у системы есть верхний тип, куда вкладываются все остальные, и нижний, который вкладывается во все. unknown - верхний: любое значение является unknown, поэтому в переменную unknown можно положить что угодно, но обратно достать - только после проверки. never - нижний: он вкладывается в любой тип, поэтому значение never подходит куда угодно, вот только взять это значение неоткуда. any стоит вне этой стройной картины: он одновременно ведёт себя и как верх, и как низ, жертвуя ради удобства именно той строгостью, которую верх и низ обеспечивают. Поэтому unknown и never - «честные» края системы, а any - дыра в ней, оставленная нарочно для совместимости со старым кодом.
</Detail>
