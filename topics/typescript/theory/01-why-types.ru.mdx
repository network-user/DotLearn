# Зачем статические типы

Борис пишет корзину для интернет-магазина. Функция считает итог: складывает цену товара и стоимость доставки. На его машине всё работает: 2999 плюс 299 даёт 3298. Через неделю на сайт приходит первый настоящий заказ, и покупатель видит в итоге `299900`. Цена пришла из API строкой, доставка тоже, плюс склеил два текста вместо сложения. Ошибки не было ни при написании кода, ни при запуске тестов на «правильных» числах: JavaScript узнал о проблеме только на живом покупателе.

TypeScript ловит ровно такие ошибки за секунды до того, как код вообще запустится. Это не новый язык, а JavaScript плюс слой аннотаций о типах: вы помечаете, что `price` это число, а `apiResponse` строка, и специальная программа проверяет, что вы нигде не смешали одно с другим. К концу главы вы будете понимать, что именно TypeScript проверяет, когда он молчит, а когда кричит, и почему в готовом браузере от всех этих типов не остаётся ни следа.

## Проверка до запуска, а не во время

Обычный JavaScript узнаёт тип значения только в момент выполнения строки: пока код не дошёл до сложения, никто не знает, что там окажется текст. TypeScript добавляет отдельный этап - проверку типов - который происходит **до** запуска, прямо в редакторе и при сборке. Компилятор `tsc` читает ваши аннотации, прослеживает, какие значения куда текут, и подсвечивает несоответствия красным. Только после успешной проверки типы стираются, и на выходе получается обычный JavaScript, который и уезжает в браузер.

Из этой картинки следует главный факт о TypeScript, который экономит часы недоумения: типы существуют только на этапе разработки. Аннотация `: number` не превращается в проверку во время выполнения. Если строка всё-таки просочилась в число обходным путём, в готовом браузере её никто не остановит. TypeScript страхует вас, пока вы пишете код, а не пока пользователь им пользуется.

## Один и тот же код, две версии

Посмотрим на функцию Бориса дважды. Сначала так, как её пишут на голом JavaScript:

```js
function total(price, delivery) {
  return price + delivery;
}
total('2999', 299);
```

Вызов `total('2999', 299)` вернёт строку `'2999299'`, и ни редактор, ни браузер слова не скажут. Теперь та же функция с аннотациями типов:

```ts
function total(price: number, delivery: number): number {
  return price + delivery;
}
total('2999', 299);
```

Запись `price: number` читается «параметр `price` имеет тип число». Последняя строка теперь подсвечивается красной волной ещё в редакторе: TypeScript сообщает `Argument of type 'string' is not assignable to parameter of type 'number'`. Ошибка найдена там, где её дёшево чинить, а не на покупателе. При этом собранный `.js`-файл выглядит так же, как в первом примере: аннотации `: number` из него исчезают.

<Callout type="info" title="TypeScript не запускает ваш код">
Проверка типов - это анализ текста программы, а не её выполнение. TypeScript не вызывает `total`, не ходит в сеть и не считает сумму. Он лишь доказывает, что при любых входных данных выбранных типов сложение будет складывать числа. Поэтому проверка мгновенна даже для кода, который в реальности работает минутами.
</Callout>

## Что вы получаете, кроме отлова ошибок

Ошибки типов - самая заметная выгода, но не единственная. Раз редактор знает тип каждого значения, он подсказывает доступные поля и методы: набрали `order.` и сразу видите `price`, `items`, `status`, а не гадаете по памяти. Переименование поля в одном месте безопасно расходится по всему проекту, потому что инструмент знает, где это поле используется. И сами типы работают как документация, которая не может устареть: сигнатура `function total(price: number, delivery: number): number` говорит о функции больше, чем абзац комментариев, и всегда синхронна с кодом.

## Чего типы не делают

Честный список ограничений важнее списка достоинств. TypeScript **не** ускоряет ваш код: собранный JavaScript выполняется ровно с той же скоростью. Он **не** заменяет тесты: тип гарантирует, что в `total` придут числа, но не что вы сложили их, а не перемножили. Он **не** проверяет данные из внешнего мира на лету: ответ сервера TypeScript считает тем типом, который вы ему пообещали, и если сервер вернул другое, узнаете вы об этом только в рантайме. Граница между «мир типов» и «реальный мир» проходит по вводу данных, и охранять её на этой границе придётся вам самим.

## Частые заблуждения

**«Раз есть типы, тесты не нужны».** Типы и тесты ловят разные ошибки. Тип не даст перепутать число со строкой; тест проверит, что бизнес-логика верна. Одно не отменяет другое.

**«Аннотации замедляют программу».** В собранном коде аннотаций нет вообще, так что замедлять нечему. Дольше идёт только сборка, и то на секунды.

**«TypeScript проверит ответ от API».** Не проверит. Для внешних данных тип - это ваше обещание компилятору, а не факт. На границе с сетью и хранилищем данные нужно проверять обычным кодом, и уже проверенные пускать в типизированную часть.

<Detail summary="Откуда взялся TypeScript">
Язык выпустила Microsoft в 2012 году; ведущим разработчиком стал Андерс Хейлсберг, автор C# и Turbo Pascal. Задача формулировалась узко: дать большим командам, которые пишут на JavaScript, инструменты, привычные по строго типизированным языкам, не заставляя переписывать существующий код. Отсюда ключевое решение - любой корректный JavaScript является корректным TypeScript. Вы можете переименовать `.js` в `.ts` и добавлять типы постепенно, файл за файлом, не останавливая проект.
</Detail>
