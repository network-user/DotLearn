# Объекты, интерфейсы и type

Половина данных в реальном приложении - это объекты: заказ с товаром и ценой, пользователь с именем и email, ответ сервера с полями и вложенными полями. Пока такой объект не описан типом, редактор ничего о нём не знает: не подскажет поля, не заметит опечатку `oreder.prise`, не остановит вас, когда вы забудете обязательное поле. Описать форму объекта - значит превратить его из безымянного мешка значений в предсказуемую структуру, которую проверяет компилятор.

TypeScript даёт три способа описать форму объекта: аннотацию прямо на месте, именованный `interface` и псевдоним `type`. Разберёмся, когда что удобнее, что такое необязательные и только-читаемые поля и почему опечатка в имени поля перестаёт быть ошибкой рантайма.

## Форма объекта как тип

Простейший способ - описать поля прямо в аннотации:

```ts
let order: { item: string; price: number };
order = { item: 'чайник', price: 2999 };
```

Тип `{ item: string; price: number }` требует ровно два поля нужных типов. Забыли `price` - ошибка. Написали `prise` вместо `price` - ошибка. Положили в `price` строку - ошибка. Редактор теперь знает форму объекта и после `order.` предложит `item` и `price`, а не заставит вспоминать имена по памяти.

Встроенная аннотация хороша для одноразового объекта, но описывать одну и ту же форму в десяти местах утомительно. Для повторного использования форму выносят в именованный тип.

## interface: имя для формы

`interface` даёт форме имя, на которое потом ссылаются:

```ts
interface Order {
  item: string;
  price: number;
}

const first: Order = { item: 'чайник', price: 2999 };
```

Теперь `Order` можно использовать где угодно: как тип переменной, как тип параметра функции, как тип элемента массива. Одно изменение формы в объявлении `interface` разойдётся по всем местам, где она применяется.

Интерфейсы расширяют друг друга через `extends`, что удобно для родственных форм:

```ts
interface Product {
  item: string;
  price: number;
}

interface DiscountedProduct extends Product {
  oldPrice: number;
}
```

`DiscountedProduct` получает все поля `Product` плюс своё `oldPrice`. Дублировать `item` и `price` не нужно.

## type: псевдоним для любого типа

`type` создаёт псевдоним - второе имя для типа. Для объекта он выглядит почти как `interface`:

```ts
type Order = {
  item: string;
  price: number;
};
```

Разница в охвате. `interface` описывает только формы объектов, а `type` умеет давать имя вообще любому типу: union из глав дальше, кортежу, функции, примитиву. Для описания объекта оба взаимозаменяемы; выбор чаще диктует соглашение команды.

## Необязательные и только-читаемые поля

Не каждое поле обязано присутствовать. Знак `?` после имени делает поле необязательным:

```ts
interface User {
  name: string;
  phone?: string;
}

const anna: User = { name: 'Анна' };
```

Объект без `phone` теперь корректен. Но у медали есть обратная сторона: тип поля `phone` становится `string | undefined`, и обратиться к нему как к гарантированной строке уже нельзя - сначала нужно проверить, что оно есть. Необязательность честно проникает в тип.

Модификатор `readonly` запрещает запись в поле после создания объекта:

```ts
interface Account {
  readonly id: number;
  balance: number;
}
```

Присвоить `account.id = 2` теперь ошибка проверки типов: идентификатор задаётся один раз при создании и дальше только читается. Как и всё в TypeScript, `readonly` действует только на этапе типов и исчезает при сборке - в рантайме поле остаётся обычным.

## Крайние случаи

Самый неожиданный для новичков момент - структурная типизация. TypeScript сравнивает типы по форме, а не по имени. Если объект имеет все поля, которых требует `Order`, он подходит под `Order`, даже если его создавали без всякого упоминания этого интерфейса.

```ts
interface Point {
  x: number;
  y: number;
}

const marker = { x: 10, y: 20, label: 'старт' };
const p: Point = marker;
```

`marker` подходит под `Point`, потому что имеет `x` и `y` нужных типов; лишнее поле `label` совместимости не мешает. Языки вроде Java так бы не сделали - там важно имя типа. TypeScript смотрит только на форму, и это удобно: вам не нужно заранее объявлять, что объект «реализует» интерфейс.

<Callout type="warning" title="Лишние поля при литерале - всё же ошибка">
У структурной типизации есть одно исключение. Если вы присваиваете переменной типа Point не отдельную переменную, а прямо объектный литерал с лишним полем, TypeScript ругается: `const p: Point = { x: 1, y: 2, label: 'старт' }` даст ошибку про неизвестное свойство label. Это защита от опечаток в имени поля прямо на месте. Через промежуточную переменную, как в примере выше, проверка мягче.
</Callout>

## Частые ошибки

**Считать, что имя типа важно.** Оно не важно: совпадение идёт по форме. Два разных интерфейса с одинаковыми полями взаимозаменяемы.

**Забыть, что `?` добавляет `undefined`.** Необязательное поле нельзя использовать как гарантированное. `user.phone.length` - ошибка, пока вы не проверили, что `phone` вообще есть.

**Ждать защиты `readonly` в рантайме.** `readonly` останавливает только компилятор. После сборки поле обычное, и присвоить в него из чистого JavaScript ничто не мешает.

**Тащить `interface` туда, где нужен union.** Форму объекта опишут оба, но union допустимых значений умеет только `type`. Для «либо то, либо это» берите `type`.

<Detail summary="Почему один interface можно объявить дважды">
Уникальная способность interface - слияние объявлений. Два `interface User` в одной области видимости не конфликтуют, а складываются в один тип со всеми полями обоих. Это редко нужно в своём коде, но незаменимо для расширения чужих типов: так дополняют, например, глобальные типы браузера или сторонней библиотеки, не трогая её исходники. У `type` такой способности нет: второе объявление с тем же именем - ошибка. Если вам нужно расширять тип извне, это аргумент за interface.
</Detail>
