# Union-типы и narrowing

Поле ввода на форме заказа принимает либо число артикула, либо строку с названием товара. Функция статуса возвращает `'new'`, `'paid'` или `'shipped'` - и ничего кроме. Настройка языка интерфейса - это ровно `'ru'` или `'en'`. Такие значения не описываются одним базовым типом: они честно бывают одним из нескольких. TypeScript выражает это union-типом - типом-«или», который перечисляет допустимые варианты через вертикальную черту.

Но union приносит и обязанность. Если значение бывает и строкой, и числом, к нему нельзя обращаться как к гарантированной строке: у числа нет метода `toUpperCase`. Прежде чем работать с конкретным вариантом, его нужно выделить из union - это называется narrowing, сужение. В этой главе увидим, как объявлять union, как TypeScript сужает тип внутри проверок и как одна пометка-тег превращает набор объектов в удобный для разбора союз.

## Тип-«или»

Union записывают через `|`, перечисляя варианты:

```ts
let articleId: string | number;
articleId = 2999;
articleId = 'A-14';
```

Оба присваивания корректны: `string | number` допускает и то, и другое. Особенно полезны union из литеральных типов - они задают точный набор допустимых значений:

```ts
type Lang = 'ru' | 'en';
let lang: Lang = 'ru';
lang = 'py';
```

Последняя строка - ошибка: `'py'` не входит в `'ru' | 'en'`. Так union из литералов ловит опечатки и «невозможные» значения ещё в редакторе, работая как компактная замена перечислениям. Именно ради таких наборов `const` в прошлых главах сужал строки до литералов.

## Narrowing: сузить тип проверкой

Внутри union-значения TypeScript не даёт вызвать метод, которого нет у всех вариантов. Решение - проверка, после которой тип сужается до одного варианта. Самый частый инструмент - `typeof`:

```ts
function describe(x: string | number): string {
  if (typeof x === 'number') {
    return 'артикул ' + x.toFixed(0);
  }
  return x.toUpperCase();
}
```

Внутри ветки `if` TypeScript знает: раз `typeof x === 'number'`, значит здесь `x` - это `number`, и `toFixed` доступен. В ветке после `if` остаётся единственный вариант `string`, поэтому `toUpperCase` тоже разрешён. Компилятор проследил за логикой проверки и сузил тип в каждой ветке автоматически.

Способов сузить несколько, и каждый под свой случай:

- `typeof x === 'string'` - для примитивов.
- `if (x)` - отсекает `null`, `undefined`, `0` и пустую строку по правдивости.
- `'field' in obj` - проверяет наличие поля у объекта.
- `x instanceof Date` - для экземпляров классов.

## Discriminated union: союз с ярлыком

Отдельного упоминания заслуживает приём, на котором держится добрая половина хорошего TypeScript-кода. Когда union состоит из объектов, к каждому добавляют общее поле-ярлык с литеральным типом - дискриминант. По нему TypeScript и различает варианты.

```ts
type Payment =
  | { kind: 'card'; last4: string }
  | { kind: 'cash'; received: number };

function describe(p: Payment): string {
  if (p.kind === 'card') {
    return 'карта ' + p.last4;
  }
  return 'наличные ' + p.received;
}
```

Проверка `p.kind === 'card'` сужает `p` до варианта с полем `last4`; в другой ветке доступно `received`. Обратиться к `p.received` в первой ветке нельзя - у варианта `'card'` его нет, и TypeScript это видит. Ярлык `kind` превращает разбор союза в простой и безопасный `if` по одному полю.

<Callout type="tip" title="Проверка на полноту">
Если позже к Payment добавят третий вариант, а разобрать его забудут, это можно поймать типом. Присвойте значение в ветке-иначе переменной типа never: пока все варианты разобраны, туда попадает never и всё в порядке; появился неучтённый вариант - в never он не влезет, и TypeScript подсветит ошибку. Так компилятор напоминает дописать обработку. Тип never разбирается в главе про unknown, any и never.
</Callout>

## Крайние случаи

Коварная мелочь - сужение по правдивости и число ноль. Проверка `if (count)` отсекает не только `undefined`, но и `0`, потому что ноль в JavaScript ложен. Для `count: number | undefined`, где ноль - валидное значение, это баг: заказ с нулём товаров молча выпадет из обработки. В таких случаях проверяйте явно: `if (count !== undefined)`.

Второй капкан - `typeof null`. Как и в чистом JavaScript, `typeof null` возвращает `'object'`, а не `'null'`. Поэтому отделить `null` через `typeof` не выйдет; для него используют прямое сравнение `x === null`.

## Частые ошибки

**Обращаться к union без сужения.** `x.toUpperCase()` на `string | number` - ошибка: у числа нет такого метода. Сначала сузьте тип проверкой, потом вызывайте метод варианта.

**Сужать число через `if (x)`.** Ноль ложен, поэтому валидный ноль потеряется. Для чисел, где ноль допустим, сравнивайте с `undefined` или `null` явно.

**Забыть дискриминант.** Без общего поля-ярлыка разбирать union объектов приходится по косвенным признакам, и TypeScript сужает хуже. Добавьте `kind` с литеральным типом - это стандарт.

**Ждать сужения после присваивания в другую переменную.** TypeScript отслеживает сужение по конкретному имени. Скопировали значение в новую переменную - сужение по старой на неё не переносится.

<Detail summary="Почему union безопаснее, чем один тип «на всё»">
Заманчиво описать «значение бывает разным» типом any и не думать о вариантах. Но any выключает проверки целиком: язык больше не помнит, что там могло быть, и пропустит любую ошибку. Union, наоборот, помнит полный список вариантов и заставляет обработать каждый. Это разница между «я не знаю, что здесь» и «здесь ровно одно из трёх, и вот как отличить». Хороший TypeScript-код почти не содержит any, зато полон точных union: чем уже тип, тем больше ошибок ловится и тем точнее подсказки редактора.
</Detail>
