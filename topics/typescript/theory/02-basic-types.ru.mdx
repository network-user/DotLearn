# Базовые типы и вывод типов

Начинающие часто думают, что TypeScript - это язык, где к каждой переменной нужно приписывать тип. Открываешь чужой проект, а там `const name = 'Анна'` без единого двоеточия, и всё компилируется. Секрет в том, что TypeScript почти всегда сам догадывается о типе по значению справа. Аннотации нужны не везде, а в нескольких конкретных местах, и умение отличать эти места от остальных - половина комфортной работы с языком.

В этой главе разберёмся с базовым набором типов, которыми вы будете пользоваться каждый день, и с выводом типов - механизмом, который избавляет от лишних аннотаций. Заодно увидим, почему `const` и `let` дают разные типы для одного и того же значения.

## Базовый словарь типов

Имена базовых типов в TypeScript совпадают с тем, что возвращает `typeof` в JavaScript, только строчными буквами. Каждое значение получает один из них.

- `number` - любое число, целое или дробное: `2999`, `3.14`, `-1`.
- `string` - текст: `'Казань'`, `"чайник"`.
- `boolean` - `true` или `false`.
- `null` и `undefined` - две разновидности пустоты.

Аннотация ставится через двоеточие после имени:

```ts
let price: number = 2999;
let city: string = 'Казань';
let isPaid: boolean = false;
```

Запись `price: number` фиксирует: сюда можно класть только числа. Попытка `price = 'дорого'` подсветится ошибкой ещё в редакторе, потому что строка не подходит под тип `number`.

## TypeScript додумывает тип сам

Приглядитесь к примеру выше: тип и так очевиден из значения справа. Раз `2999` - число, то и `price` число, писать `: number` необязательно. TypeScript выводит тип автоматически, и это называют выводом типов (type inference).

```ts
let price = 2999;
let city = 'Казань';
```

Здесь `price` всё равно получил тип `number`, а `city` - тип `string`, хотя ни одной аннотации нет. Наведите курсор на имя в редакторе, и всплывёт выведенный тип. Правило вырабатывается быстро: если значение присваивается сразу, тип обычно можно не писать - он лишь повторил бы очевидное.

## Где аннотации всё-таки нужны

Раз вывод так хорош, зачем вообще аннотации? Есть три места, где TypeScript не может или не должен догадываться сам.

Первое - параметры функций. У функции нет «значения справа», из которого виден тип аргумента, поэтому параметры аннотируют почти всегда:

```ts
function discount(price: number, percent: number): number {
  return price - (price * percent) / 100;
}
```

Второе - объявление без начального значения. Когда переменную создают пустой, выводить не из чего:

```ts
let selectedCity: string;
selectedCity = 'Казань';
```

Третье - когда выведенный тип шире, чем вам нужно, и вы хотите сузить его намеренно. Об этом отдельная глава про union-типы.

<Callout type="tip" title="Практическое правило">
Аннотируйте границы: параметры функций и то, что возвращается наружу. Внутри функции, где значение присваивается сразу, доверяйте выводу - лишние аннотации только зашумляют код и расходятся с реальностью при рефакторинге.
</Callout>

## const и let выводят разные типы

Одно и то же значение `const` и `let` типизируют по-разному, и это логичное следствие их природы.

```ts
const city = 'Казань';
let region = 'Татарстан';
```

У `region` выведется тип `string`: раз `let` разрешает переприсваивание, TypeScript оставляет место под любую строку. А у `city` выведется не `string`, а литеральный тип `'Казань'` - ровно эта строка и никакая другая. Причина проста: `const` переприсвоить нельзя, значит значение навсегда останется именно `'Казань'`, и TypeScript фиксирует это в типе. Такое поведение называют сужением до литерала, и позже оно окажется очень полезным для union-типов, где важно перечислить набор допустимых строк.

## Крайние случаи

Пустое объявление без аннотации - ловушка. В нестрогом режиме `let value;` получает тип `any`, то есть «любой тип», и проверка по этой переменной фактически отключается. Это одна из причин, по которой строгий режим (о нём последняя глава) требует аннотировать такие объявления явно.

Второй сюрприз - `null` и `undefined`. Без строгого режима они молча совместимы с любым типом: строке можно присвоить `null`, и TypeScript промолчит. В строгом режиме `null` и `undefined` становятся отдельными типами, и вот тогда язык начинает по-настоящему защищать вас от обращения к пустому значению.

## Частые ошибки

**Аннотировать всё подряд.** `const price: number = 2999` не ошибка, но шум: тип и так очевиден. Избыток аннотаций мешает читать код и устаревает при правках. Аннотируйте границы, остальное доверьте выводу.

**Забыть аннотировать параметры.** Без строгого режима параметр без типа тихо становится `any`, и функция перестаёт проверяться. Параметры - главное место, где аннотация обязательна.

**Ждать от вывода чудес на пустом объявлении.** `let x;` без значения и без типа выводить не из чего, отсюда `any`. Либо присваивайте сразу, либо ставьте аннотацию.

**Путать `number` со строкой цифр.** `'2999'` - это `string`, а не `number`, сколько бы цифр в нём ни было. Тип определяется кавычками, а не содержимым.

<Detail summary="Что такое литеральные типы и зачем они">
Тип может быть не только «любая строка», но и «ровно эта строка». `'Казань'` как тип допускает единственное значение - строку `Казань`. Сами по себе такие типы кажутся бесполезными: зачем переменная, куда можно положить лишь одно значение. Но в союзе с union-типами они дают точные наборы: тип `'ru' | 'en'` описывает язык интерфейса, где допустимы только два конкретных значения, и любая опечатка вроде `'py'` тут же становится ошибкой. Именно ради этого `const` сужает строку до литерала: чтобы такие значения естественно попадали в union.
</Detail>
