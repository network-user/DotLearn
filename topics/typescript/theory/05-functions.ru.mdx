# Функции и сигнатуры

Функция - это договор: подай ей такие-то аргументы, получи такой-то результат. На голом JavaScript этот договор нигде не записан, и нарушить его легко: передал строку вместо числа, забыл аргумент, вернул `undefined` там, где ждали объект. TypeScript записывает договор в сигнатуре функции - в типах её параметров и результата - и следит, чтобы обе стороны его соблюдали: и тот, кто функцию пишет, и тот, кто её вызывает.

Разберёмся, как аннотировать параметры и возвращаемое значение, когда возвращаемый тип стоит писать явно, а когда довериться выводу, и как обозначить необязательные аргументы, значения по умолчанию и переменное число параметров.

## Сигнатура: типы на входе и выходе

Параметры аннотируют как обычные переменные, а тип результата ставят после списка параметров:

```ts
function discount(price: number, percent: number): number {
  return price - (price * percent) / 100;
}
```

Часть `(price: number, percent: number): number` - это сигнатура. Она проверяется с двух сторон. При вызове TypeScript следит, что аргументы подходят по типам и что их ровно столько, сколько параметров: `discount(2999)` - ошибка, не хватает второго аргумента. Внутри функции он следит, что вы действительно возвращаете `number`: если из какой-то ветки вернуть строку, будет ошибка.

## Возвращаемый тип: писать или выводить

Тип результата TypeScript обычно выводит сам из `return`, поэтому `: number` в конце можно опустить, и функция всё равно останется типизированной. Но у явной аннотации есть ценное свойство: она проверяет тело функции на соответствие обещанию сразу, а не только по месту вызова.

```ts
function toPrice(value: number): string {
  return value;
}
```

Здесь аннотация `: string` ловит ошибку прямо в теле: вы обещали вернуть строку, а вернули число. Без неё TypeScript вывел бы тип результата как `number`, и ошибка всплыла бы дальше - там, где результат используют как строку. Отсюда практика: для функций, которые вы отдаёте наружу, возвращаемый тип стоит писать явно - он фиксирует контракт и ловит расхождение в самом близком месте.

## Необязательные параметры и значения по умолчанию

Не каждый аргумент обязателен. Знак `?` делает параметр необязательным, а его тип получает `undefined`:

```ts
function greet(name: string, title?: string): string {
  return title ? title + ' ' + name : name;
}
```

`greet('Анна')` теперь корректно. Внутри функции `title` имеет тип `string | undefined`, поэтому использовать его без проверки нельзя - здесь это делает тернарный оператор.

Часто вместо «может отсутствовать» удобнее «есть значение по умолчанию». Значение по умолчанию задаётся присваиванием прямо в параметре, и тип тогда выводится из него:

```ts
function greet(name: string, greeting = 'Привет'): string {
  return greeting + ', ' + name;
}
```

Тип `greeting` вывелся как `string`, аннотация не нужна. Разница с необязательным параметром важная: у параметра со значением по умолчанию внутри функции тип уже без `undefined` - на месте пропущенного аргумента подставится строка `'Привет'`, а не пустота.

## Переменное число аргументов

Когда аргументов заранее неизвестно сколько, их собирают в массив rest-параметром. Тип ставят на элемент массива:

```ts
function sum(...nums: number[]): number {
  return nums.reduce((acc, n) => acc + n, 0);
}
```

Запись `...nums: number[]` означает «сколько угодно чисел». Внутри `nums` - обычный массив `number[]`. Вызов `sum(1, 2, 3)` пройдёт, а `sum(1, '2')` - ошибка: строка не число.

## Функция как значение и её тип

В JavaScript функцию можно передать в другую функцию или сохранить в переменную. У такого значения тоже есть тип - тип функции, записанный стрелкой:

```ts
type Formatter = (value: number) => string;

const rub: Formatter = (value) => value + ' руб';
```

Тип `(value: number) => string` описывает «функцию, которая принимает число и возвращает строку». Обратите внимание: у `rub` параметр `value` не аннотирован - его тип TypeScript взял из `Formatter`. Это называют контекстной типизацией: когда тип функции известен из окружения, типы её параметров выводятся, и повторять их не нужно. Так типизируют колбэки для `map`, обработчики событий и любые функции, передаваемые аргументом.

<Callout type="warning" title="void не значит «нельзя вернуть»">
Возвращаемый тип void означает «результат не используется», а не «функция обязана ничего не вернуть». Колбэк с типом () =&gt; void можно реализовать функцией, которая на самом деле что-то возвращает - её результат просто проигнорируют. Это сделано нарочно: массиву не важно, что вернёт колбэк в forEach, поэтому под () =&gt; void подходит и функция, возвращающая число.
</Callout>

## Частые ошибки

**Перепутать необязательный параметр и значение по умолчанию.** Необязательный `title?: string` внутри имеет тип со `undefined` и требует проверки. Параметр по умолчанию `greeting = 'Привет'` - уже гарантированную строку. Выбирайте по тому, нужна вам пустота внутри или нет.

**Ставить необязательный параметр перед обязательным.** `function f(a?: number, b: number)` - ошибка: после необязательного параметра обязательных быть не может, иначе вызов стал бы неоднозначным.

**Не аннотировать возвращаемый тип у публичной функции.** Без него ошибку «вернул не то» вы увидите не в теле функции, а далеко от него, у места использования. Явный возвращаемый тип ловит расхождение сразу.

**Аннотировать параметры колбэка, когда они и так выводятся.** В `nums.map((n) => n * 2)` тип `n` уже известен из массива. Лишняя аннотация только зашумляет.

<Detail summary="Перегрузки: одна функция - несколько сигнатур">
Иногда функция ведёт себя по-разному в зависимости от типов аргументов: приняла строку - вернула строку, приняла число - вернула число. Такое описывают перегрузками - несколькими сигнатурами перед одной реализацией. Каждая сигнатура задаёт допустимую комбинацию входа и выхода, а тело реализует их все сразу, обычно через проверки внутри. Перегрузки нужны нечасто: в большинстве случаев union-тип или generic из следующей главы выражают ту же идею короче и без дублирования. Беритесь за перегрузки, только когда связь между конкретным входом и конкретным выходом нельзя выразить одним параметром типа.
</Detail>
