# Строгий tsconfig

Можно писать на TypeScript месяцами и не поймать ни одной настоящей ошибки типов - если проверка настроена вполсилы. По умолчанию, ради совместимости со старым JavaScript, TypeScript многое прощает: пропущенный тип параметра тихо становится `any`, а `null` свободно течёт в любую переменную. В таком режиме язык подсказывает поля, но почти не защищает. Один флаг в настройках включает настоящую строгость, ту самую, ради которой TypeScript и берут.

Этот флаг живёт в `tsconfig.json` - файле настроек компилятора, который лежит в корне проекта. Разберёмся, что он включает, почему `strict` стоит включать с первого дня нового проекта и что делать со старым кодом, который под строгостью вдруг покраснел.

## tsconfig.json и флаг strict

`tsconfig.json` описывает, как компилятор обрабатывает проект: какие файлы включать, в какую версию JavaScript собирать и, главное для нас, насколько строго проверять типы. Минимальный строгий конфиг умещается в несколько строк:

```json
{
  "compilerOptions": {
    "strict": true,
    "target": "ES2020"
  }
}
```

`"strict": true` - это не один флаг, а выключатель целого семейства проверок разом. Включая его, вы активируете сразу все строгие правила, и это правильное значение по умолчанию для любого нового проекта. Два правила из семейства важно понимать в лицо.

## noImplicitAny: запрет тихого any

Без строгого режима параметр без аннотации молча получает тип `any`, и функция перестаёт проверяться - вы видели это в главе про базовые типы. Флаг `noImplicitAny` запрещает такой неявный `any`:

```ts
function total(price, delivery) {
  return price + delivery;
}
```

С включённым `noImplicitAny` эти параметры подсветятся ошибкой: «Parameter implicitly has an any type». Компилятор требует явно указать тип - либо честный `number`, либо, если тип и правда любой, честный `any`, написанный руками. Разница между «забыл тип» и «намеренно поставил `any`» становится видимой, а не тонет в молчании.

## strictNullChecks: null больше не всеобщий

Это самая ценная из строгих проверок. Без неё `null` и `undefined` совместимы с любым типом: переменной `string` можно присвоить `null`, и компилятор промолчит - пока в рантайме не случится классическое «Cannot read property of null».

```ts
function firstLetter(name: string): string {
  return name[0].toUpperCase();
}
```

С `strictNullChecks` тип `string` больше не включает `null`. Если значение действительно бывает пустым, это придётся написать явно - `string | null` - и тогда компилятор **потребует** проверить на `null` перед обращением к значению:

```ts
function firstLetter(name: string | null): string {
  if (name === null) {
    return '?';
  }
  return name[0].toUpperCase();
}
```

Так `strictNullChecks` превращает целый класс рантайм-падений в ошибки на этапе типов. Именно эта проверка окупает переход на строгий режим быстрее всего.

## Крайние случаи

Строгость не меняет собранный код. `tsconfig.json` управляет только проверкой; на то, какой JavaScript получится после сборки, флаг `strict` не влияет вообще. Строгий и нестрогий проект с одинаковым кодом дают одинаковый рантайм - разной будет лишь строгость проверки на этапе разработки.

Отдельный флаг `target` в конфиге определяет версию JavaScript на выходе: `"ES2020"` разрешит современный синтаксис, а более старое значение заставит компилятор переписать новые конструкции под старые движки. Это соседняя настройка, не связанная со строгостью типов, но её часто задают в том же `compilerOptions`.

## Частые ошибки

**Откладывать `strict` на потом.** На новом проекте строгость почти бесплатна: кода мало, чинить нечего. На разросшемся - это сотни ошибок разом. Включайте `strict` с первого дня.

**Глушить ошибки строгости через `any`.** Когда `strictNullChecks` требует проверить `null`, соблазн приписать `any` и забыть. Это возвращает ровно ту дыру, от которой строгость защищает. Проверьте значение, а не отключайте проверку.

**Отключать отдельные строгие флаги, не понимая цены.** `"strictNullChecks": false` внутри строгого конфига выглядит безобидно, но выключает самую полезную проверку. Прежде чем гасить флаг, разберитесь, что именно он ловил.

**Считать, что `strict` влияет на скорость или размер сборки.** Не влияет. Это исключительно про то, сколько ошибок поймает компилятор, а не про итоговый JavaScript.

<Detail summary="Как включают strict в существующем проекте">
Включить strict в большом нестрогом проекте разом - значит получить стену ошибок. На практике переходят постепенно. Один путь - включить strict глобально, а файлы, которые ещё не готовы, временно пометить послаблением, разгребая их по одному. Другой, более аккуратный, - включать строгие флаги семейства по очереди: сперва noImplicitAny, разобрать все неявные any, затем strictNullChecks, пройтись по обращениям к возможному null. Так стена ошибок дробится на управляемые порции. Ключевая мысль: strict - это цель, к которой мигрируют, а не рубильник, который обязательно щёлкать сразу на живом коде. Но для нового проекта верно обратное - там его включают первым же коммитом, пока чинить нечего.
</Detail>
