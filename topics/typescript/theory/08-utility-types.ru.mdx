# Утилити-типы: Partial, Pick, Omit, Record

Вы описали интерфейс `User` с полями `id`, `name`, `email`, `age`. Дальше по коду нужны его вариации: форма обновления, где все поля необязательны; краткая карточка только с `id` и `name`; тип без служебного `id` для создания нового пользователя. Соблазн - объявить три новых интерфейса руками. Но это дублирование: добавите в `User` поле, и придётся не забыть про все копии. TypeScript предлагает другое - выводить производные типы из исходного одной строкой, чтобы они автоматически следовали за оригиналом.

Инструменты для этого называют утилити-типами. Это встроенные обобщённые типы, которые принимают ваш тип параметром и возвращают его преобразованную версию. Четыре из них покрывают большинство ежедневных задач: `Partial`, `Pick`, `Omit` и `Record`. Все они - обычные дженерики из прошлой главы, только уже написанные за вас.

## Partial: сделать все поля необязательными

`Partial<T>` берёт тип и возвращает такой же, но со всеми полями необязательными:

```ts
interface User {
  id: number;
  name: string;
  email: string;
}

function update(user: User, patch: Partial<User>): User {
  return { ...user, ...patch };
}
```

Тип `Partial<User>` - это `{ id?: number; name?: string; email?: string }`. Функция `update` принимает пользователя и «заплатку», где можно указать только меняющиеся поля: `update(anna, { email: 'new@mail.ru' })` корректно, остальные поля трогать не обязательно. Опишите `patch` как `User` - и пришлось бы передавать все поля целиком; `Partial` снимает это требование, оставаясь связанным с `User`.

## Pick и Omit: выбрать или отбросить поля

`Pick<T, K>` строит тип из выбранных полей исходного. Список полей передают union из литеральных строк:

```ts
type UserPreview = Pick<User, 'id' | 'name'>;
```

`UserPreview` - это `{ id: number; name: string }`, ровно два поля из `User`. Тип для карточки в списке, где email и прочее не нужны.

`Omit<T, K>` - зеркальная операция: берёт всё, кроме перечисленных полей:

```ts
type NewUser = Omit<User, 'id'>;
```

`NewUser` - это `User` без `id`: `{ name: string; email: string }`. Естественный тип для создания записи, у которой идентификатор ещё не выдан базой. Выбирайте между ними по тому, что короче перечислить: оставить два поля из десяти - `Pick`; убрать одно из десяти - `Omit`. И `Pick`, и `Omit` следуют за исходным типом: добавите поле в `User`, и `Omit<User, 'id'>` автоматически его получит.

## Record: словарь с типизированными ключами и значениями

`Record<K, V>` описывает объект-словарь: тип ключей `K`, тип значений `V`.

```ts
type PriceList = Record<string, number>;

const prices: PriceList = {
  чайник: 2999,
  зонт: 899,
};
```

`Record<string, number>` - объект с любыми строковыми ключами и числовыми значениями. Ключи можно и ограничить конкретным набором через union литералов:

```ts
type Lang = 'ru' | 'en';
type Labels = Record<Lang, string>;
```

`Record<Lang, string>` требует объект ровно с ключами `ru` и `en`, оба со строковыми значениями. Забыли один язык - ошибка. Так `Record` из литерального union превращается в проверяемую таблицу соответствий.

## Крайние случаи

Главное, что нужно помнить: утилити-типы существуют только на этапе типов. `Partial`, `Pick`, `Omit`, `Record` не создают ни объектов, ни функций - они лишь описывают форму. В собранном JavaScript от них не остаётся ничего, как и от всех прочих типов. Строка `type NewUser = Omit<User, 'id'>` не выполняет никакой работы в рантайме; она инструктирует только компилятор.

Вторая тонкость - `Partial` необязателен вглубь на один уровень. `Partial<User>` делает необязательными поля самого `User`, но если внутри лежит вложенный объект, его собственные поля остаются обязательными. Для рекурсивной необязательности нужен отдельный, более сложный тип - встроенный `Partial` работает поверхностно.

<Callout type="tip" title="Утилити-типы комбинируются">
Ничто не мешает вкладывать одно преобразование в другое. `Partial<Omit<User, 'id'>>` - это «пользователь без id, и все оставшиеся поля необязательны»: удобная форма частичного обновления, которая заодно запрещает трогать идентификатор. Утилити-типы - обычные дженерики, поэтому свободно вкладываются друг в друга.
</Callout>

## Частые ошибки

**Дублировать производные типы руками.** Отдельный интерфейс «то же, но без id» разойдётся с оригиналом при первой правке. `Omit<User, 'id'>` останется синхронным сам.

**Передавать в `Pick` и `Omit` несуществующее поле.** `Pick<User, 'phone'>`, когда в `User` нет `phone`, - ошибка. Второй параметр обязан быть реальными ключами типа, и это как раз хорошо: опечатку в имени поля поймает компилятор.

**Путать `Pick` и `Omit`.** `Pick` перечисляет, что оставить; `Omit` - что убрать. Перепутаете - получите ровно дополнение к нужному типу.

**Ждать от `Record` защиты значений в рантайме.** `Record<Lang, string>` требует оба ключа на этапе типов, но если данные пришли из сети, реальный объект может не соответствовать - типы его не проверяют. Внешние данные валидируйте кодом.

<Detail summary="Как утилити-типы устроены внутри">
Ни один из этих типов не встроен на уровне компилятора магией - все они написаны на самом TypeScript и лежат в стандартной библиотеке типов. Partial, например, определён через отображённый тип (mapped type): «пройди по всем ключам T и сделай каждый необязательным». Pick перебирает переданные ключи и собирает из них новый объект. Это тот же язык типов, что доступен и вам: при желании можно написать собственные утилити-типы такой же силы. Знать их внутреннее устройство для повседневной работы не обязательно, но полезно понимать, что за именами Partial и Pick стоит не встроенная команда, а обычный обобщённый тип, который можно прочитать и повторить.
</Detail>
