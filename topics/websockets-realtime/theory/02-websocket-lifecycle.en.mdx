# Four states, four events, and one ordering trap

You don't write the protocol from the last section by hand in the browser. You create a `WebSocket` object, subscribe to events, and call `send`. But behind that simple facade hides a finite state machine: the connection moves through strictly defined states, and trying to send data in the wrong state fails silently or throws. Most "it sometimes doesn't arrive" bugs in realtime code come from ignoring which state the connection is in.

By the end of this section you'll read and write client WebSocket code confidently: know the four `readyState` values, understand when and in what order `open`, `message`, `error`, and `close` fire, tell a clean shutdown from a drop, and see how ping/pong keep the connection alive underneath while you handle business logic on top.

## Mental model: the connection is a finite state machine

The `WebSocket` object has a `readyState` property that takes exactly four values. `CONNECTING (0)` - handshake in progress, not ready yet. `OPEN (1)` - channel established, you can send and receive. `CLOSING (2)` - shutdown started, exchange winding down. `CLOSED (3)` - done, channel dead. Transitions are one-way: there's no path back from `CLOSED` to `OPEN`; you need a new object.

These constants - `WebSocket.OPEN` and so on - are available as static fields. In practice you compare `socket.readyState === WebSocket.OPEN` before sending, because sending in any other state is the most common source of bugs.

## Creation and the open event

The constructor starts the handshake immediately, but it's asynchronous: while the lines after `new WebSocket(...)` run, the connection is still in `CONNECTING`. Readiness is signaled by the `open` event.

```js
const socket = new WebSocket("wss://example.com/chat");

socket.addEventListener("open", () => {
  socket.send("hello");
});
```

Inside the `open` handler, `readyState === OPEN` is guaranteed, so `send` is safe. Here's trap number one: the temptation to call `socket.send(...)` right after the constructor, on the next line. At that moment the connection is almost certainly still in `CONNECTING`, and the browser throws `InvalidStateError`. Any send belongs inside `open` or after checking `readyState`.

<Callout type="warning">
`new WebSocket(url)` does not open the connection by the time the constructor returns. Calling `send` on the next line throws `InvalidStateError`. Send only from the `open` handler or after checking `readyState === WebSocket.OPEN`.
</Callout>

## Receiving data: the message event

Every incoming data frame becomes a `message` event. The payload is in `event.data`. For text frames it's a string; for binary frames it's a `Blob` or `ArrayBuffer` (depends on the `binaryType` property).

```js
socket.addEventListener("message", (event) => {
  const payload = JSON.parse(event.data);
  render(payload);
});
```

Important: WebSocket delivers exactly what the other side sent, in order - the protocol over TCP preserves ordering. But it knows nothing about meaning. `event.data` is always a raw string or bytes; JSON parsing, routing by event type, schema validation - all your code. Socket.IO automates that manual work in the next section.

## Closing: clean and not so clean

Either side can close with `close()`. That moves `readyState` to `CLOSING`, the sides exchange close control frames, and `CLOSED` arrives with a `close` event. The event has a `code` (numeric reason) and `wasClean` - a flag that separates a proper shutdown from a drop.

```js
socket.addEventListener("close", (event) => {
  if (event.wasClean) {
    console.log("closed cleanly, code", event.code);
  } else {
    reconnectLater();
  }
});
```

`wasClean === true` means both sides exchanged close frames - a polite goodbye. `wasClean === false` means the connection broke: network lost, server crashed, Wi-Fi dropped. That difference matters for reconnect logic: after a clean close you usually don't reconnect; after a drop you do, ideally with backoff.

## Surprise: `error` almost never tells you what happened

The `error` event feels like the right place for diagnostics. In practice the spec deliberately hides details: the error object in the browser carries no code or message - for security, so a page can't scan someone else's network by probing different URLs. After `error`, `close` always follows, and **`close.code`** carries the real reason. Build diagnostics around `close`; use `error` only as a signal that something went wrong.

```js
socket.addEventListener("error", () => {
  console.warn("WebSocket error, waiting for close with code");
});
```

Close codes are standardized: `1000` - normal, `1001` - going away (tab closed), `1006` - abnormal drop without a close frame (you see this on network loss), `1011` - server internal error. `1006` is especially telling: it can't be sent in a close frame; the browser sets it when the connection dies silently.

## Ping, pong, and why the connection "stays alive"

A TCP connection can die quietly so neither side knows: packets stop arriving but `close` never fires. The protocol provides ping and pong control frames. One side sends ping, the other must reply with pong; if there's no reply within a timeout, the connection is considered dead and closes.

In the browser API you can't send ping/pong directly - the browser and server handle them. But knowing they exist matters: they're what detect "stuck" connections, not your code. The application layer (Socket.IO, for example) often adds its own heartbeat on top so it doesn't depend on browser settings.

<Callout type="tip">
Don't try to detect "is the connection alive?" by sending empty `send` calls. Ping/pong at the protocol level and heartbeat at the library level exist for that. Your job is to react correctly to `close` with code `1006`, not invent your own pulse.
</Callout>

## Common mistakes

- **Calling `send` right after the constructor.** The connection is still in `CONNECTING`; send throws `InvalidStateError`. Send from `open` or after checking `readyState`.
- **Diagnosing failures from the `error` event.** It's intentionally empty. The reason is in `close.code`; build handling around `close`, not `error`.
- **Not distinguishing `wasClean`.** Reconnecting after a clean `close` (`wasClean === true`) means knocking on a door you closed yourself. Reconnect after a drop, ideally with exponential backoff.
- **Sending in `CLOSING`/`CLOSED`.** In a long-lived app, check `readyState` before every send, or some messages vanish silently.

<Detail summary="Buffering: bufferedAmount and backpressure">

`send` has a hidden side: data doesn't always hit the network instantly. If you send faster than the channel allows, data piles up in an internal buffer whose size is visible in `socket.bufferedAmount`. On a slow link or when streaming, that buffer grows and consumes memory. Serious apps check `bufferedAmount` before sending large volumes and throttle manually, implementing backpressure - the browser won't stop the sender for you. For chat that's fine; for telemetry streams or files it's critical.

</Detail>
