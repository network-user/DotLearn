# Когда одного сервера перестаёт хватать

Один процесс на одном сервере держит тысячи WebSocket-соединений и счастливо рассылает им сообщения. Пока всё в одной памяти, broadcast тривиален: пробежал по списку открытых сокетов, каждому отправил. Сложности начинаются ровно в тот момент, когда серверов становится два. Пользователь Аня сидит на сервере №1, Боря - на сервере №2, оба в одном чате. Аня пишет сообщение, оно приходит на сервер №1 - а как до него дотянется сервер №2, в памяти которого про Аню вообще ничего нет?

Этот раздел - про паттерны, которые делают realtime живым: broadcast (послать многим), presence (кто сейчас онлайн), и горизонтальное масштабирование (как заставить несколько серверов вести себя как один). К концу вы будете понимать, почему наивный broadcast ломается при масштабировании, зачем нужен внешний слой вроде Redis, и какие подводные камни ждут в presence и при балансировке соединений.

## Broadcast: послать всем, кроме отправителя

Базовый паттерн realtime - разослать событие группе. У Socket.IO для этого три варианта адресации, и их легко перепутать. `io.emit` шлёт **всем** подключённым. `socket.broadcast.emit` - всем, **кроме** самого отправителя (то самое «не показывай мне моё же сообщение дважды»). `io.to(room).emit` - только участникам комнаты.

```js
io.on("connection", (socket) => {
  socket.on("chat:message", (msg) => {
    socket.broadcast.to(msg.room).emit("chat:message", msg);
  });
});
```

Здесь `socket.broadcast.to(room)` доставляет сообщение всем в комнате, исключая отправителя: его собственный клиент уже отрисовал сообщение оптимистично, и эхо ему не нужно. Различие между «всем» и «всем кроме меня» кажется мелочью, но именно на нём ломаются дублирующиеся сообщения в чатах.

## Presence: кто сейчас онлайн

Presence - индикатор «в сети», список участников, зелёная точка у аватара. Соблазн прост: на `connection` добавить пользователя в множество онлайн, на `disconnect` - убрать. Работает - до первого обрыва сети. Если у пользователя оборвался Wi-Fi, событие `disconnect` придёт не сразу, а через таймаут heartbeat; всё это время он числится онлайн, хотя его уже нет. А если у пользователя две вкладки, `disconnect` одной не означает, что человек ушёл.

Поэтому надёжный presence считают не по «есть ли соединение», а по счётчику соединений на пользователя: вкладка подключилась - инкремент, отключилась - декремент, онлайн - пока счётчик больше нуля.

```js
io.on("connection", (socket) => {
  const userId = socket.handshake.auth.userId;
  presence.increment(userId);
  socket.on("disconnect", () => presence.decrement(userId));
});
```

Этот счётчик переживает множественные вкладки и реконнекты корректно: человек считается онлайн, пока хоть одно его соединение живо. А чтобы счётчик не «протёк» при падении сервера (соединения умерли, а `disconnect` не сработал), его держат во внешнем хранилище с TTL, а не в памяти процесса.

<Callout type="warning">
Presence «по факту соединения» врёт при обрывах и нескольких вкладках: `disconnect` после потери сети опаздывает на таймаут heartbeat. Считайте онлайн по счётчику соединений на пользователя и храните его с TTL, чтобы упавший сервер не оставил «вечно онлайн» призраков.
</Callout>

## Масштабирование: почему наивный broadcast ломается

Вернёмся к Ане и Боре на разных серверах. `io.to(room).emit` на сервере №1 пробегает по сокетам **этого** процесса - про соединения сервера №2 он ничего не знает. В итоге Боря сообщения Ани не увидит. Это фундаментальное ограничение: in-memory список соединений виден только своему процессу, а комната, размазанная по нескольким серверам, наполовину невидима каждому из них.

Решение - вынести рассылку за пределы процесса. Серверы подключают общий **adapter** (для Socket.IO - Redis adapter): когда сервер №1 делает `io.to(room).emit`, adapter публикует событие в Redis, а все остальные серверы на него подписаны и доставляют его своим локальным сокетам в этой комнате. Redis здесь работает как шина publish/subscribe: один опубликовал, все доставили.

```js
const { createAdapter } = require("@socket.io/redis-adapter");
io.adapter(createAdapter(pubClient, subClient));
```

После подключения adapter ваш прикладной код **не меняется**: вы по-прежнему пишете `io.to(room).emit`, просто теперь рассылка дотягивается до всех серверов кластера. Вся магия спрятана в adapter; для приложения кластер снова выглядит как один сервер.

## Sticky sessions: ещё одна ловушка масштабирования

Есть второй подводный камень, отдельный от broadcast. Socket.IO начинает соединение с long-polling и затем апгрейдит до WebSocket - а это несколько HTTP-запросов, которые **обязаны** попасть на один и тот же сервер. Если балансировщик раскидает их по разным процессам по round-robin, рукопожатие Engine.IO развалится: один сервер начнёт сессию, другой про неё не знает.

Лечится это sticky sessions: балансировщик должен привязывать клиента к одному backend на всё время соединения (обычно по cookie или IP-hash). Redis adapter решает проблему broadcast между серверами, но **не** заменяет sticky sessions - это две независимые настройки, и забыть про вторую легко.

<Callout type="info">
Redis adapter и sticky sessions решают разные задачи. Adapter доставляет broadcast между серверами кластера. Sticky sessions гарантируют, что все запросы одной сессии Engine.IO (polling + апгрейд) попадут на один сервер. Для горизонтального масштабирования Socket.IO нужны обе настройки.
</Callout>

## Сюрприз: масштаб упирается не в соединения, а в fan-out

Кажется, что предел масштабирования - число открытых соединений. На деле первым обычно упирается **fan-out**: сколько исходящих сообщений порождает одно входящее. В чате на 10 000 человек одно сообщение - это 10 000 отправок. При шумном чате на тысячу сообщений в секунду это уже десятки миллионов исходящих в секунду, и сеть с CPU кончаются раньше, чем лимит дескрипторов. Поэтому большие realtime-системы дробят аудиторию на комнаты поменьше, агрегируют события (не каждый «печатает...», а раз в N секунд) и иногда сознательно роняют часть событий, которые не критичны. Масштаб realtime - это в первую очередь управление исходящим трафиком, а не приём соединений.

## Типичные ошибки

- **Слать `io.emit` там, где нужна комната.** `io.emit` бьёт по всем подключённым на сервере; в многокомнатном приложении это утечка сообщений между чатами и лишний трафик. Адресуйте `io.to(room)`.
- **Presence по факту `disconnect`.** Обрывы и вкладки ломают наивный счётчик. Считайте по числу живых соединений на пользователя и храните с TTL.
- **Думать, что Redis adapter заменяет sticky sessions.** Это разные проблемы: adapter - про broadcast между серверами, sticky - про то, чтобы сессия не размазалась по серверам при апгрейде.
- **Игнорировать fan-out при оценке нагрузки.** Считать только соединения - значит проспать настоящий предел: исходящий трафик растёт как произведение размера комнаты на частоту сообщений.

<Detail summary="Доставка при разрыве: история сообщений и идемпотентность">

WebSocket и Socket.IO дают доставку «пока соединение живо», но при обрыве сообщения, отправленные в окно простоя, до клиента не дойдут (буфер emit спасает лишь исходящие самого клиента, не входящие). Для чатов это решают не транспортом, а прикладным слоем: каждому сообщению дают возрастающий id или временную метку, и после реконнекта клиент запрашивает «всё, что было после последнего id, который я видел» обычным HTTP-запросом к истории. Транспорт realtime отвечает за «здесь и сейчас», а надёжная история - за «что я пропустил». Та же идея с id даёт идемпотентность: повторно доставленное буфером сообщение клиент отбросит по уже виденному id, и дубликат не появится.

</Detail>
