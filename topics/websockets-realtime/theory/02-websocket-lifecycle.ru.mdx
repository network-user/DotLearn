# Четыре состояния, четыре события и одна ловушка с порядком кода

Протокол из прошлого раздела вы в браузере не пишете руками. Вы создаёте объект `WebSocket`, подписываетесь на события и зовёте `send`. Но за этим простым фасадом прячется конечный автомат: соединение проходит через строго определённые состояния, и попытка отправить данные не в том состоянии тихо проваливается или бросает исключение. Большая часть «он у меня иногда не доходит» в realtime-коде - это игнорирование того, в каком состоянии сейчас находится соединение.

К концу раздела вы будете уверенно читать и писать клиентский WebSocket-код: знать четыре значения `readyState`, понимать, в каком порядке и при каких условиях срабатывают события `open`, `message`, `error` и `close`, отличать чистое закрытие от обрыва, и видеть, как ping/pong снизу поддерживают соединение живым, пока вы заняты бизнес-логикой сверху.

## Ментальная модель: соединение - это конечный автомат

У объекта `WebSocket` есть свойство `readyState`, и оно принимает ровно четыре значения. `CONNECTING (0)` - идёт handshake, соединение ещё не готово. `OPEN (1)` - канал установлен, можно слать и получать. `CLOSING (2)` - запущено закрытие, обмен сворачивается. `CLOSED (3)` - всё, канал мёртв. Переходы между ними однонаправленны: назад из `CLOSED` в `OPEN` дороги нет, нужно создавать новый объект.

Эти константы - `WebSocket.OPEN` и так далее - доступны как статические поля. На практике вы сравниваете `socket.readyState === WebSocket.OPEN` перед отправкой, потому что отправка в любом другом состоянии - источник самых частых багов.

## Создание и событие open

Конструктор запускает handshake немедленно, но он асинхронный: пока строки кода после `new WebSocket(...)` выполняются, соединение ещё в `CONNECTING`. Готовность сигнализирует событие `open`.

```js
const socket = new WebSocket("wss://example.com/chat");

socket.addEventListener("open", () => {
  socket.send("привет");
});
```

Внутри обработчика `open` гарантируется `readyState === OPEN`, поэтому `send` безопасен. Здесь же прячется ловушка номер один: соблазн позвать `socket.send(...)` сразу после конструктора, в следующей строке. В этот момент соединение почти наверняка ещё в `CONNECTING`, и браузер бросит `InvalidStateError`. Любая отправка должна жить внутри `open` или после проверки `readyState`.

<Callout type="warning">
`new WebSocket(url)` не открывает соединение к моменту возврата из конструктора. Вызов `send` строкой ниже падает с `InvalidStateError`. Отправляйте только из обработчика `open` либо предварительно проверяйте `readyState === WebSocket.OPEN`.
</Callout>

## Приём данных: событие message

Каждый входящий фрейм с данными превращается в событие `message`. Полезная нагрузка лежит в `event.data`. Для текстовых фреймов это строка, для бинарных - `Blob` или `ArrayBuffer` (зависит от свойства `binaryType`).

```js
socket.addEventListener("message", (event) => {
  const payload = JSON.parse(event.data);
  render(payload);
});
```

Важно: WebSocket доставляет именно те сообщения, которые отправила другая сторона, в том же порядке - протокол поверх TCP сохраняет очерёдность. Но он ничего не знает про их смысл. `event.data` - это всегда сырая строка или байты; разбор JSON, маршрутизация по типу события, схема - всё это ваш код. Эту ручную работу как раз и автоматизирует Socket.IO в следующем разделе.

## Закрытие: чистое и не очень

Закрыть соединение можно с любой стороны вызовом `close()`. Это переводит `readyState` в `CLOSING`, стороны обмениваются управляющими close-фреймами, и наступает `CLOSED` с событием `close`. У события есть `code` (числовая причина) и `wasClean` - флаг, который отличает корректное завершение от обрыва.

```js
socket.addEventListener("close", (event) => {
  if (event.wasClean) {
    console.log("закрыто штатно, код", event.code);
  } else {
    reconnectLater();
  }
});
```

`wasClean === true` означает, что обе стороны успели обменяться close-фреймами - это вежливое прощание. `wasClean === false` - соединение оборвалось: пропала сеть, упал сервер, вырубился Wi-Fi. Разница принципиальна для логики переподключения: после штатного закрытия лезть обратно обычно не нужно, после обрыва - нужно, желательно с нарастающей паузой.

## Сюрприз: «error» почти никогда не говорит вам, что случилось

Событие `error` интуитивно хочется использовать для диагностики. На деле спецификация намеренно скрывает детали: объект ошибки в браузере не несёт ни кода, ни сообщения - из соображений безопасности, чтобы со страницы нельзя было сканировать чужую сеть по реакции на разные URL. После `error` всегда следует `close`, и вот **именно `close.code`** несёт реальную причину. Поэтому диагностику стройте вокруг `close`, а `error` используйте лишь как сигнал «что-то пошло не так».

```js
socket.addEventListener("error", () => {
  console.warn("WebSocket error, ждём close с кодом");
});
```

Коды закрытия стандартизированы: `1000` - штатно, `1001` - сторона уходит (вкладка закрылась), `1006` - аномальный обрыв без close-фрейма (его вы увидите при потере сети), `1011` - внутренняя ошибка сервера. `1006` особенно показателен: его нельзя отправить в close-фрейме, браузер выставляет его сам, когда соединение умерло молча.

## Ping, pong и почему соединение «живёт» само

TCP-соединение может тихо умереть так, что обе стороны об этом не знают: пакеты просто перестают доходить, но `close` не приходит. Чтобы это ловить, протокол даёт управляющие фреймы ping и pong. Одна сторона шлёт ping, другая обязана ответить pong; если ответа нет за таймаут - соединение считается мёртвым и закрывается.

В браузерном API ping/pong вам недоступны напрямую - ими управляет сам браузер и сервер. Но знать о них важно: именно они, а не ваш код, обнаруживают «зависшие» соединения. Прикладной уровень (тот же Socket.IO) часто добавляет ещё и свой heartbeat поверх, чтобы не зависеть от настроек браузера.

<Callout type="tip">
Не пытайтесь определять «живо ли соединение» отправкой пустых `send`. Для этого есть ping/pong на уровне протокола и heartbeat на уровне библиотеки. Ваша задача - корректно реагировать на `close` с кодом `1006`, а не изобретать свой пульс.
</Callout>

## Типичные ошибки

- **Звать `send` сразу после конструктора.** Соединение ещё в `CONNECTING`, отправка падает с `InvalidStateError`. Шлите из `open` или после проверки `readyState`.
- **Диагностировать сбои по событию `error`.** Оно намеренно пустое. Причина - в `close.code`; стройте обработку вокруг `close`, а не `error`.
- **Не различать `wasClean`.** Переподключаться после штатного `close` (`wasClean === true`) - значит ломиться в дверь, которую сами закрыли. Реконнект нужен после обрыва, и желательно с экспоненциальной задержкой.
- **Слать данные в `CLOSING`/`CLOSED`.** Перед каждой отправкой в долгоживущем приложении проверяйте `readyState`, иначе часть сообщений молча потеряется.

<Detail summary="Буферизация: bufferedAmount и обратное давление">

У `send` есть скрытая сторона: данные не всегда уходят в сеть мгновенно. Если вы шлёте быстрее, чем позволяет канал, они копятся во внутреннем буфере, и его размер виден в `socket.bufferedAmount`. На медленной сети или при потоковой отправке этот буфер растёт, потребляя память. Серьёзные приложения проверяют `bufferedAmount` перед отправкой больших объёмов и притормаживают, реализуя обратное давление (backpressure) вручную, - браузер сам отправителя не остановит. Для чата это неважно, для стрима телеметрии или файлов - критично.

</Detail>
