# Why a whole protocol grew on top of ready-made WebSocket

Bare WebSocket gives you a pipe where bytes flow. Everything else you write yourself: parse JSON, invent an "event type" field, manually reconnect after drops, check whether a message arrived, and hope corporate proxies don't block the upgrade. By the third chat that's tedious. Socket.IO is a library that takes that routine work and leaves you with two clear operations: `emit` (send a named event) and `on` (subscribe to it). In the browser you install it as `socket.io-client`; on the server, as the `python-socketio` package.

Don't confuse the layers. Socket.IO is **not** WebSocket and not even "WebSocket with sugar": it's its own protocol that in the good case rides on WebSocket, and in the bad case on HTTP long-polling, switching between them transparently. By the end of this section you'll understand what Socket.IO adds over WebSocket (events, rooms, auto-reconnect, acknowledgements, fallback transport), how to stand up a `python-socketio` server under uvicorn, and why a Socket.IO client can't connect to a bare WebSocket server.

## Mental model: a layer above transport

Think of Socket.IO as two layers. The lower one - Engine.IO - handles transport: establishes the connection, chooses between WebSocket and long-polling, keeps heartbeat, reconnects. The upper one - Socket.IO itself - provides application semantics: named events, rooms, namespaces, delivery acknowledgements. Your code talks only to the upper layer; the lower one quietly decides which pipe everything rides on.

Because of this layering, Socket.IO has its own packet format on top of WebSocket frames. So `new WebSocket("wss://.../socket.io/")` in the browser won't connect to a Socket.IO server: it doesn't speak Engine.IO. You need the `socket.io-client`, and client and server versions must be compatible - and the protocol doesn't depend on the server's language: `python-socketio` speaks to the same browser client as a Node implementation.

## Named events instead of parsing by type

With bare WebSocket you get one `message` event and parse what's inside yourself. Socket.IO flips that: you send an event with a name, and the handler on the other side subscribes to the same name. In the browser it looks like this - the client stays JavaScript:

```js
socket.emit("chat:message", { text: "hello", room: "general" });

socket.on("chat:message", (msg) => {
  appendToChat(msg.text);
});
```

The name `chat:message` is an arbitrary string - your contract. Payload serializes automatically: objects, arrays, strings, numbers, even binary data - no manual JSON encoding. The server follows the same name-based contract, but the code is Python, via `python-socketio`:

```python
import socketio
import uvicorn

sio = socketio.AsyncServer(async_mode="asgi", cors_allowed_origins="*")
app = socketio.ASGIApp(sio)

@sio.on("chat:message")
async def chat_message(sid, msg):
    await sio.emit("chat:message", msg)

if __name__ == "__main__":
    uvicorn.run(app, host="0.0.0.0", port=8000)
```

`socketio.AsyncServer` is the Socket.IO server itself; `socketio.ASGIApp` wraps it into an ASGI app that uvicorn runs (you can also hand `app` to the CLI: `uvicorn server:app`). The `@sio.on("chat:message")` decorator binds a handler to an event by name, and `await sio.emit(...)` broadcasts the event. The key difference from the browser client: there's no socket object on the server. The handler receives `sid` - a string session id - and anything you do to a specific connection you do by passing `sid` to server methods. For the system events `connect` and `disconnect` the `@sio.event` decorator is handier: it takes the event name straight from the function name.

## Acknowledgements: did the message arrive

Bare WebSocket doesn't confirm delivery of a single message - you only know the channel is open. Socket.IO adds acknowledgement: the sender passes a callback, and the receiving side returns a value into it. The browser client passes the callback as the last `emit` argument:

```js
socket.emit("order:create", { id: 42 }, (response) => {
  console.log("server confirmed:", response.status);
});
```

On a `python-socketio` server the callback is not passed into the handler. Instead the handler simply **returns** a value, and Socket.IO delivers it into the client's callback:

```python
@sio.on("order:create")
async def order_create(sid, order):
    saved = await orders.save(order)
    return {"status": "ok", "id": saved["id"]}
```

Whatever the handler returned lands in `response` on the client. That turns emit into something like request-response over a persistent connection - useful when the sender needs to wait for a result, not fire blindly.

<Callout type="info">
Acknowledgement confirms that the handler on the other side ran and returned a value, not that the event changed the database or reached a third system. It's a "received and processed" signal, not a distributed transaction. For exactly-once guarantees you need idempotency at the application level.
</Callout>

## Rooms: addressing a group

The most common realtime task is sending a message not to everyone and not to one person, but to a group: chat participants, channel subscribers, players at a table. Socket.IO handles this with rooms. A room is a named set of connections on the server; the server adds a connection to it, then broadcasts into it.

```python
@sio.event
async def connect(sid, environ, auth):
    await sio.enter_room(sid, "room:general")
    await sio.emit("chat:message", {"text": "hello everyone"}, room="room:general")
```

`await sio.enter_room(sid, "room:general")` adds the connection to a room; `await sio.emit(..., room="room:general")` delivers the event to all members, and the inverse is `sio.leave_room(sid, room)`. Rooms live only on the server - the browser client doesn't know about them and has no room API. One `sid` can be in many rooms, and that's the main addressing tool we'll return to in the patterns section. A useful detail: every connection is already in a room named after its own `sid`, so "send to one client" is just an emit to the room named after that `sid` (or `to=sid`).

## Surprise: auto-reconnect can silently swallow your messages

Socket.IO prides itself on automatic reconnection: network drops - the client reconnects with increasing delay, and your code barely notices. But there's a nuance that catches many people. While disconnected, the client's `emit` calls **buffer by default** and flush after reconnect. Convenient - but if during downtime you sent a bunch of "increment counter" commands, they all apply at once after reconnect and state diverges.

Worse: a reconnected client gets a **new** `sid`, and its previous room membership on the server doesn't restore itself - rooms were tied to the old `sid`. The server must re-join rooms in the `connect` handler by calling `sio.enter_room` again. Treating reconnect as fully transparent means debugging "why did room messages stop after network loss" someday.

<Callout type="warning">
After reconnect the client has a new `sid`, and server-side rooms from the old connection aren't inherited. Restore room membership in the server `connect` handler (`sio.enter_room`), and make critical emits idempotent - the buffer may deliver them again.
</Callout>

## Fallback transport: when WebSocket isn't allowed

Engine.IO starts with HTTP long-polling and then, if the environment allows, upgrades to WebSocket. That's the insurance against proxies and firewalls from the first section: even where `Upgrade` is blocked, long-polling passes as ordinary HTTP and the app keeps working, just with more latency. Transparent fallback is one of the main reasons teams pick Socket.IO over bare WebSocket in production with diverse clients.

## Common mistakes

- **Connecting `new WebSocket` to a Socket.IO server (or vice versa).** Different protocols. Use `socket.io-client` with a Socket.IO server and compatible versions; a bare WebSocket server won't understand the Socket.IO client.
- **Treating acknowledgement as a business guarantee.** Ack means "the handler ran and returned a value," not "money charged." Build guarantees with application-level idempotency.
- **Relying on the emit buffer during outages.** Buffered events deliver after reconnect - possibly duplicating effects. Don't send non-idempotent commands that way.
- **Forgetting to restore rooms after reconnect.** A new `sid` doesn't remember old rooms; put the client back in its rooms on the server via `sio.enter_room` in the `connect` handler.

<Detail summary="Namespaces: another addressing level">

Besides rooms, Socket.IO has namespaces - a way to split one physical connection into logical channels with different handlers: `/chat`, `/admin`, `/notifications`. The client connects to a namespace via `io("/chat")` in the browser, and on the server handlers take a `namespace="/chat"` argument (for example `@sio.on("chat:message", namespace="/chat")`); events in one namespace don't cross into another, though they share one transport. The difference from rooms: a namespace splits by functionality type (defined in application code); a room is a dynamic grouping of connections inside a namespace (created and changed at runtime). Most apps get by with the default `/` namespace and rooms; namespaces appear when you need to isolate whole subsystems with different authorization.

</Detail>
