# When one server is no longer enough

One process on one server holds thousands of WebSocket connections and happily broadcasts to them. While everything lives in one memory space, broadcast is trivial: walk the list of open sockets, send to each. Complexity starts the moment you have two servers. User Anna is on server #1, Boris on server #2, both in the same chat. Anna sends a message; it lands on server #1 - how does server #2 reach it when Anna's connection isn't in its memory at all?

This section covers the patterns that keep realtime alive: broadcast (send to many), presence (who's online now), and horizontal scaling (making several servers behave as one). By the end you'll understand why naive broadcast breaks when you scale, why an external layer like Redis is needed, and what pitfalls wait in presence and connection load balancing.

## Broadcast: send to everyone except the sender

The basic realtime pattern is delivering an event to a group. Socket.IO offers three addressing options, and they're easy to mix up. `io.emit` sends to **everyone** connected. `socket.broadcast.emit` sends to everyone **except** the sender (the "don't show me my own message twice" case). `io.to(room).emit` sends only to room members.

```js
io.on("connection", (socket) => {
  socket.on("chat:message", (msg) => {
    socket.broadcast.to(msg.room).emit("chat:message", msg);
  });
});
```

Here `socket.broadcast.to(room)` delivers to everyone in the room except the sender: their client already rendered the message optimistically, so echo isn't needed. The difference between "everyone" and "everyone but me" seems small, but duplicate messages in chats break on exactly that line.

## Presence: who's online now

Presence is the "online" indicator, participant list, green dot on an avatar. The naive approach: on `connection` add the user to an online set, on `disconnect` remove them. That works - until the first network drop. If Wi-Fi dies, `disconnect` arrives only after a heartbeat timeout; until then they're "online" when they're not. And with two tabs open, one tab's `disconnect` doesn't mean the person left.

Reliable presence counts connections per user, not "is there a connection": tab connects - increment, disconnects - decrement, online while the count is above zero.

```js
io.on("connection", (socket) => {
  const userId = socket.handshake.auth.userId;
  presence.increment(userId);
  socket.on("disconnect", () => presence.decrement(userId));
});
```

That counter survives multiple tabs and reconnects correctly: a person is online while at least one connection lives. To keep the counter from leaking when a server crashes (connections died but `disconnect` never ran), store it in external storage with TTL, not in process memory.

<Callout type="warning">
Connection-based presence lies on drops and multiple tabs: `disconnect` after network loss is delayed by the heartbeat timeout. Count online by connections per user with TTL so a crashed server doesn't leave "forever online" ghosts.
</Callout>

## Scaling: why naive broadcast breaks

Back to Anna and Boris on different servers. `io.to(room).emit` on server #1 walks sockets **in that process** - it knows nothing about server #2's connections. Boris never sees Anna's message. That's fundamental: the in-memory connection list is visible only to its process, and a room spread across servers is half invisible to each one.

The fix is to move broadcast outside the process. Servers attach a shared **adapter** (for Socket.IO - Redis adapter): when server #1 does `io.to(room).emit`, the adapter publishes the event to Redis, and every other server subscribed delivers it to its local sockets in that room. Redis acts as a publish/subscribe bus: one publishes, all deliver.

```js
const { createAdapter } = require("@socket.io/redis-adapter");
io.adapter(createAdapter(pubClient, subClient));
```

After attaching the adapter, your application code **doesn't change**: you still write `io.to(room).emit`, but broadcast now reaches the whole cluster. The magic hides in the adapter; to the app the cluster looks like one server again.

## Sticky sessions: another scaling trap

There's a second pitfall, separate from broadcast. Socket.IO starts with long-polling and upgrades to WebSocket - several HTTP requests that **must** hit the same server. If the load balancer round-robins them across processes, the Engine.IO handshake falls apart: one server starts the session, another knows nothing about it.

Sticky sessions fix this: the load balancer binds the client to one backend for the whole connection (usually by cookie or IP hash). Redis adapter solves broadcast across servers but **doesn't** replace sticky sessions - two independent settings, and the second is easy to forget.

<Callout type="info">
Redis adapter and sticky sessions solve different problems. The adapter delivers broadcast across cluster servers. Sticky sessions ensure all requests of one Engine.IO session (polling + upgrade) hit one server. Horizontal Socket.IO scaling needs both.
</Callout>

## Surprise: scale limits are usually fan-out, not connections

It seems the scaling limit is open connection count. In practice **fan-out** hits first: how many outbound messages one inbound message creates. In a 10,000-person chat, one message is 10,000 sends. At 1,000 messages per second that's tens of millions outbound per second, and network and CPU run out before descriptor limits. Large realtime systems split audiences into smaller rooms, aggregate events (not every "typing..." tick, but once every N seconds), and sometimes deliberately drop non-critical events. Realtime scale is mostly about managing outbound traffic, not accepting connections.

## Common mistakes

- **Using `io.emit` where you need a room.** `io.emit` hits everyone on the server; in a multi-room app that leaks messages between chats and wastes traffic. Address with `io.to(room)`.
- **Presence from raw `disconnect`.** Drops and tabs break a naive flag. Count live connections per user and store with TTL.
- **Thinking Redis adapter replaces sticky sessions.** Different problems: adapter is broadcast across servers; sticky is keeping one Engine.IO session on one server during upgrade.
- **Ignoring fan-out when estimating load.** Counting only connections misses the real limit: outbound traffic grows as room size times message rate.

<Detail summary="Delivery after disconnect: message history and idempotency">

WebSocket and Socket.IO deliver "while the connection is alive," but during an outage messages sent in the gap never reach the client (the emit buffer only helps the client's own outbound, not inbound). Chats solve this at the application layer: each message gets a monotonic id or timestamp, and after reconnect the client requests "everything after the last id I saw" via a normal HTTP history request. Realtime transport handles "here and now"; reliable history handles "what I missed." The same id idea gives idempotency: a message redelivered from the buffer is dropped if the client already saw that id, so duplicates don't appear.

</Detail>
