You're building a chat, a stock ticker, or a collaborative editor - and you keep hitting the same wall. Sending your own message to the server is easy with a normal request. But your peer's message has to reach you on its own, right away, without the browser polling the server over and over: "anything new yet?" Classic HTTP can't do that: the server only responds to a request and stays silent the rest of the time. This topic is about opening a persistent two-way channel between browser and server where data flies both ways instantly, and about not drowning in boilerplate when you need that channel in production.

**What you'll be able to do.** Turn a plain HTTP connection into a persistent channel and read what travels over it at the frame level; write client code that doesn't send into a dead connection and survives disconnects cleanly; offload routine work to Socket.IO with its events, rooms, and auto-reconnect; broadcast to groups of users, track who's online, and spread load across multiple servers.

**The route through this topic.** Four sections, from a single connection to a whole cluster:

1. **Handshake and frames** - how a normal HTTP request with an `Upgrade` header turns a TCP connection into a two-way WebSocket channel.
2. **Lifecycle** - four `readyState` values and the `open`, `message`, and `close` events whose ordering is why messages "sometimes don't arrive."
3. **Socket.IO** - why a whole protocol with `emit`, `on`, and fallback transport grew on top of bare WebSocket.
4. **Realtime patterns** - broadcast, presence, and scaling when one server is no longer enough.

**What you need first.** Enough to understand how the browser makes a normal HTTP request and what an event is in JavaScript - we'll cover the rest as we go. Nothing to install: the examples show client and server code you can reproduce locally.
