# Один макет для телефона и монитора

Один и тот же сайт открывают на экране в 360 пикселей и на мониторе в 2560. Верстать две отдельные версии - тупик: их придётся синхронно править вечно. Адаптивная вёрстка решает задачу иначе - один макет, который сам перестраивается под доступную ширину. Инструментов для этого три поколения: медиазапросы, функции вроде `clamp`, убирающие часть точек перелома вовсе, и совсем свежие container queries. Разберём, как они складываются в цельный подход, и почему начинать почти всегда стоит с мобильной версии.

## Mobile-first: от узкого к широкому

Правильная точка старта - самый узкий экран. Базовые стили пишут без всяких условий, под телефон: одна колонка, крупные пальцевые зоны. А затем медиазапросами добавляют усложнения для широких экранов. Обратный подход (сначала десктоп, потом «чинить» под мобильный) почти всегда рождает больше кода и больше переопределений.

```css
.cards {
  display: grid;
  grid-template-columns: 1fr;
  gap: 16px;
}

@media (min-width: 768px) {
  .cards {
    grid-template-columns: repeat(3, 1fr);
  }
}
```

Медиазапрос `@media (min-width: 768px)` включает вложенные правила, только когда ширина вьюпорта не меньше 768 пикселей. До этого порога действует базовая одноколоночная раскладка. Значения-пороги называют точками перелома (breakpoints); привязывать их стоит к тому, где макет реально ломается, а не к моделям конкретных телефонов.

## clamp, min и max: адаптивность без точек перелома

Часто величина должна плавно меняться с шириной экрана, а не прыгать на порогах. Для этого есть функции `min()`, `max()` и `clamp()`, работающие прямо в значениях свойств.

`clamp(min, идеальное, max)` возвращает «идеальное» значение, но не даёт ему выйти за границы. Классика - типографика, растущая с экраном, но в разумных пределах.

```css
.title {
  font-size: clamp(24px, 5vw, 48px);
}
```

Единица `vw` - это один процент ширины вьюпорта. При ширине окна 600 пикселей `5vw` даёт 30 пикселей - это между 24 и 48, поэтому заголовок будет ровно 30. На узком телефоне `5vw` опустился бы ниже 24 - тогда сработает нижняя граница и размер замрёт на 24. На огромном мониторе `5vw` превысил бы 48 - вступит верхняя граница. Одна строка заменяет два-три медиазапроса на размер шрифта.

`min(90%, 1200px)` для ширины контейнера означает «90 процентов, но не больше 1200 пикселей» - удобная замена паре `width`/`max-width`. `max()` работает зеркально, задавая нижнюю границу.

## Медиазапросы умеют больше, чем ширину

Ширина - самый частый, но не единственный признак, на который смотрит `@media`. Запрос реагирует и на ориентацию, и на предпочтения пользователя, и на возможности устройства.

```css
@media (prefers-color-scheme: dark) {
  :root { --bg: #111; }
}

@media (prefers-reduced-motion: reduce) {
  * { animation: none; }
}
```

`prefers-color-scheme` подхватывает системную тёмную тему без единой кнопки в интерфейсе; `prefers-reduced-motion` уважает просьбу пользователя убрать анимации - это важно и для доступности, и для тех, кого укачивает от движения; `hover: none` отличает сенсорный экран от мыши, чтобы не вешать важное действие на наведение, которого на телефоне нет. Эти запросы - не про размер, а про контекст и про человека за экраном, и зрелая адаптивная вёрстка учитывает их наравне с шириной.

## Container queries: реагировать на контейнер, а не на экран

У медиазапросов есть слабое место: они смотрят на ширину всего экрана, а не на место, доступное конкретному компоненту. Одна и та же карточка в узкой боковой колонке и в широкой основной должна выглядеть по-разному - но экран-то один. Container queries решают именно это: компонент реагирует на размер своего контейнера.

```css
.sidebar {
  container-type: inline-size;
}

@container (min-width: 400px) {
  .card {
    grid-template-columns: 120px 1fr;
  }
}
```

Сначала контейнер объявляют через `container-type: inline-size`, разрешая замерять его ширину. Затем `@container` включает правила в зависимости от ширины этого контейнера, а не вьюпорта. Одна и та же карточка автоматически становится горизонтальной там, где ей просторно, и вертикальной там, где тесно, - независимо от размера экрана. Это делает компоненты по-настоящему переносимыми между местами вёрстки.

## Частые ошибки

- **Верстать desktop-first и «чинить» под мобильный.** Ведёт к каскаду переопределений; начинать с узкого экрана проще и чище.
- **Забыть мета-тег вьюпорта.** Без него медиазапросы на телефоне не срабатывают - страница рисуется как уменьшенный десктоп.
- **Ставить точки перелома под конкретные модели устройств.** Экранов слишком много; порог привязывают к тому, где макет ломается, а не к «айфону».
- **Тянуть медиазапрос там, где хватило бы `clamp`.** Плавно меняющиеся размеры (шрифт, отступы, ширина) чаще решаются одной функцией без точек перелома.

<Detail summary="Единицы вьюпорта и коварство vh на мобильных">
Кроме `vw` есть `vh` (процент высоты вьюпорта) и парные `vmin`/`vmax` (процент меньшей/большей стороны). С `vh` на телефонах связана давняя боль: адресная строка мобильного браузера то появляется, то прячется, меняя видимую высоту, из-за чего блок `height: 100vh` то обрезается, то оставляет полосу. Ответ спецификации - новые единицы `svh`, `lvh` и `dvh` (small, large, dynamic viewport height): `dvh` следует за реальной видимой высотой в моменте, `svh` берёт наименьшую (когда панели браузера развёрнуты), `lvh` - наибольшую. Для полноэкранных секций на мобильных сегодня берут `100dvh` вместо `100vh`.
</Detail>
