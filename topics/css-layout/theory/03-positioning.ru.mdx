# Кнопка «закрыть» в углу карточки

Крестик закрытия почти всегда живёт в правом верхнем углу карточки, поверх её содержимого, не сдвигая текст ни на пиксель. Значок «онлайн» на аватарке, шапка, прилипающая к верху экрана при прокрутке, всплывающая подсказка рядом с кнопкой - всё это выход за пределы нормального потока. Свойство `position` даёт пять режимов, и путаница между ними - источник самых упорных багов вёрстки. Разберём каждый и, главное, поймём, относительно чего элемент отсчитывает свои `top` и `left`.

## Пять значений одного свойства

`position: static` - значение по умолчанию: элемент стоит в нормальном потоке, а свойства `top`, `right`, `bottom`, `left` на него не действуют вовсе. Остальные четыре режима выводят элемент из потока в разной степени.

`position: relative` сдвигает элемент относительно того места, где он стоял бы в потоке, на заданные `top`/`left` - но исходное место остаётся занятым, соседи не сдвигаются. Сам по себе relative двигают редко; куда важнее его побочный эффект, к которому мы вернёмся.

## absolute: относительно ближайшего позиционированного предка

`position: absolute` полностью вынимает элемент из потока: место в потоке освобождается, соседи смыкаются, будто элемента там нет. Дальше элемент позиционируется по `top`/`right`/`bottom`/`left` относительно своего containing block - ближайшего предка с `position` отличным от `static`. Если такого предка нет, точкой отсчёта становится корневой блок (по сути вьюпорт при прокрутке в начало).

Отсюда - главный рецепт наложения: делаем карточку `relative`, а крестик внутри - `absolute`, и он отсчитывает координаты от карточки, а не от всей страницы.

```css
.card {
  position: relative;
}
.card__close {
  position: absolute;
  top: 8px;
  right: 8px;
}
```

Вот зачем нужен «бесполезный» `relative`: он делает карточку containing block для абсолютного потомка, ничего не сдвигая. Забыть его - классическая ошибка: крестик уезжает в угол всего экрана, потому что ближайшего позиционированного предка не нашлось.

<Callout type="warning">
absolute отсчитывает координаты не от родителя, а от ближайшего предка с position не static. Если крестик оказался не там, где ждали, - почти всегда забыт position: relative на нужном контейнере.
</Callout>

Если у абсолютного элемента задать сразу противоположные смещения, он не встанет в угол, а растянется между ними. `top: 0; right: 0; bottom: 0; left: 0` (сокращённо `inset: 0`) прижимает элемент ко всем четырём сторонам containing block - готовая подложка поверх карточки или затемнение под модальным окном.

```css
.overlay {
  position: absolute;
  inset: 0;
  background: rgba(0, 0, 0, 0.5);
}
```

Комбинируя частичные смещения, абсолютным позиционированием точно ставят что угодно: значок «онлайн» в правый нижний угол аватара (`bottom: 0; right: 0`), подпись поверх картинки, бейдж в углу карточки. Все эти координаты по-прежнему отсчитываются от ближайшего позиционированного предка, поэтому `position: relative` на контейнере - обязательный спутник любого такого приёма.

## fixed и sticky: привязка к экрану

`position: fixed` привязывает элемент к вьюпорту: он остаётся на месте при прокрутке страницы. Так делают шапки, кнопки «наверх», модальные окна. Containing block для fixed - сам вьюпорт (за редким исключением, о котором в глубоком нырке).

`position: sticky` - гибрид relative и fixed. Пока элемент виден в своей обычной позиции, он ведёт себя как `relative` и едет вместе с содержимым. Как только при прокрутке он доходит до заданного порога (например `top: 0`), он «прилипает» и дальше остаётся на этом пороге, пока его секция-родитель не уедет из виду.

```css
.section-title {
  position: sticky;
  top: 0;
}
```

Две частые причины, по которым sticky «не работает»: не задан порог (`top`, `bottom` и т. п. обязательны - без них прилипать не к чему) или у одного из предков стоит `overflow: hidden`/`auto`, обрезающий область прокрутки, внутри которой sticky живёт.

## z-index и контекст наложения

Когда элементы перекрываются, порядок наложения задаёт `z-index`: больше значение - выше слой. Но `z-index` действует только на позиционированные элементы (не `static`) и - ловушка - работает не глобально, а внутри своего контекста наложения (stacking context).

Новый контекст наложения создаёт не только `position` с `z-index`, но и, неожиданно, `opacity` меньше единицы, `transform`, `filter`, `will-change` и ещё несколько свойств. Внутри такого контекста дети сравниваются по `z-index` только между собой; наружу это число не «просвечивает».

```css
.modal { z-index: 1000; }
.tooltip { z-index: 10; }
```

Отсюда обескураживающий баг: элемент с `z-index: 1000` оказывается ниже элемента с `z-index: 10`, потому что первый заперт в родительском контексте с низким приоритетом. Никакое увеличение числа не поможет - проблема в иерархии контекстов, а не в величине `z-index`.

## Частые ошибки

- **Забыть `position: relative` на контейнере для absolute-потомка.** Крестик или значок уезжает относительно всей страницы вместо карточки. Это причина номер один среди багов позиционирования.
- **Ждать от sticky прилипания без порога.** `position: sticky` без `top`/`bottom` не делает ничего: порог обязателен.
- **Ломать sticky родительским `overflow`.** `overflow: hidden` на любом предке часто отменяет прилипание - ищите его вверх по дереву.
- **Крутить `z-index` всё выше и удивляться.** Если слой не всплывает, дело обычно не в числе, а в том, что элемент заперт в чужом контексте наложения (созданном `transform`, `opacity` и т. п.).

<Detail summary="Почему opacity создаёт контекст наложения">
Логика такая: чтобы применить полупрозрачность к группе элементов, браузер должен сначала отрисовать её как единое целое на отдельном слое, а затем смешать этот слой с фоном. Значит, всё содержимое обязано наложиться друг на друга ещё внутри группы, до смешивания, - иначе полупрозрачность считалась бы попиксельно неправильно. Поэтому `opacity: 0.99` молча создаёт контекст наложения и запирает `z-index` детей внутри. Тот же механизм у `transform` и `filter`: они тоже рендерятся на собственном слое. Знание этого экономит часы: увидев «не всплывающий» элемент, первым делом ищите у предков opacity, transform или filter.
</Detail>
