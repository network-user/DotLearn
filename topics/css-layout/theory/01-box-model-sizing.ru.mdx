# Почему три блока по 33% не помещаются в ряд

Вы делите строку на три равные колонки: каждой задаёте `width: 33.33%`, добавляете скромный `padding: 16px` и рамку в один пиксель для аккуратности. И третья колонка уезжает вниз. Проценты сходятся, а блоки не помещаются. Причина не в округлении и не в баге браузера: слово `width` в CSS по умолчанию означает не то, что вы имеете в виду. Разберёмся, как браузер на самом деле считает итоговый размер коробки, и почему одна строка `box-sizing` избавляет от этой арифметики навсегда.

## Коробка из четырёх слоёв

Любой элемент браузер представляет как прямоугольную коробку из четырёх слоёв, вложенных наружу от содержимого: `content`, `padding`, `border`, `margin`. Первые три слоя красятся фоном элемента, `margin` всегда прозрачен - это воздух между коробками.

Ключевой вопрос вёрстки звучит так: когда вы пишете `width: 300px`, какой из этих слоёв получает 300 пикселей? Ответ зависит от свойства `box-sizing`, и именно оно решает исход задачи из вступления.

## content-box: ширина задаёт только содержимое

Значение по умолчанию - `box-sizing: content-box`. При нём `width` управляет шириной слоя `content`, а `padding` и `border` прибавляются снаружи. Итоговая видимая ширина коробки получается больше заданной.

```css
.card {
  width: 300px;
  padding: 16px;
  border: 1px solid gray;
}
```

Итоговая ширина этой карточки: 300 (content) + 16 + 16 (padding слева и справа) + 1 + 1 (border) = 334 пикселя. Вернёмся к колонкам из вступления: каждая просила 33.33% на content, а сверху получила ещё по 34 пикселя padding и border. Три раза по 34 лишних пикселя переполняют строку, и третья колонка переносится. Блоки честно заняли ровно столько, сколько попросили - просто попросили больше, чем казалось.

## border-box: ширина задаёт всю коробку

`box-sizing: border-box` переопределяет смысл `width`: теперь это итоговая ширина коробки вместе с padding и border. Браузер сам ужимает слой content, чтобы всё уместилось в заданное число.

```css
.card {
  box-sizing: border-box;
  width: 300px;
  padding: 16px;
  border: 1px solid gray;
}
```

Здесь коробка занимает ровно 300 пикселей при любой толщине рамки и отступов, а под содержимое остаётся 300 - 32 - 2 = 266 пикселей. Проценты снова работают предсказуемо: три колонки по 33.33% занимают ровно треть строки каждая, что бы вы ни добавили внутрь. Поэтому первое правило почти любого проекта - глобальный сброс, включающий `border-box` для всех элементов сразу.

```css
*, *::before, *::after {
  box-sizing: border-box;
}
```

<Callout type="tip">
border-box не наследуется браузерными стилями, поэтому сброс пишут через универсальный селектор `*` и псевдоэлементы, а не в расчёте на наследование от `body`. Это одна строка, которая экономит часы отладки «съехавшей» сетки.
</Callout>

## width - это пожелание, а не приказ

Даже с `border-box` заданная ширина не всегда становится итоговой. Поверх `width` действуют ограничители `min-width` и `max-width`, и они сильнее: `min-width` не даёт коробке стать уже порога, `max-width` - шире. Когда значения конфликтуют, `min-width` побеждает `max-width`.

```css
.hero {
  width: 100%;
  max-width: 640px;
  min-width: 320px;
}
```

Такой блок тянется на всю ширину родителя, но никогда не станет шире 640 пикселей на большом экране и не сожмётся уже 320 на маленьком. Это рабочая лошадка адаптивной вёрстки: `max-width` держит строки текста читаемой длины, `min-width` защищает от схлопывания. Браузер вычисляет так: берёт `width`, затем зажимает результат между `min` и `max`.

## Что делать, если содержимое не влезло

Иногда содержимое больше коробки: длинное слово, широкая таблица, картинка. За поведение в этом случае отвечает `overflow`. Значение по умолчанию `visible` - лишнее просто вылезает за границы и рисуется поверх соседей. `hidden` обрезает всё, что не поместилось. `scroll` всегда показывает полосы прокрутки, `auto` - только когда они нужны.

```css
.log {
  max-height: 200px;
  overflow: auto;
}
```

Понимание `overflow` спасает от классической ловушки: «почему у меня появилась горизонтальная прокрутка всей страницы». Обычно виновата одна коробка с фиксированной шириной или неразрывным содержимым, вылезшая за пределы вьюпорта при `overflow: visible`.

## Высота ведёт себя не как ширина

Слои и `box-sizing` для высоты работают так же, как для ширины, но в процентах прячется асимметрия. `width: 50%` почти всегда срабатывает: ширина родителя известна из потока. А `height: 50%` часто «не работает». Процентная высота требует, чтобы у родителя была задана определённая высота, - иначе браузеру не от чего считать процент, и он трактует высоту как `auto`, то есть по содержимому.

```css
html, body { height: 100%; }
.app { height: 100%; }
```

Отсюда вечное «`height: 100%` не растянул блок на весь экран»: у цепочки родителей вплоть до `html` и `body` высота не задана, и проценты рушатся по всей цепочке. Лечений три: задать высоту явно по всей цепочке предков (как выше), перейти на единицы вьюпорта (`100vh` или `100dvh`), либо отдать растяжение по высоте flex- или grid-раскладке, где оно решается выравниванием, а не процентами. Из-за этой возни высоту в вёрстке фиксируют куда реже ширины - чаще ей дают течь по содержимому.

## Частые ошибки

- **Считать `width` итоговой шириной при `content-box`.** Добавили padding или border - и коробка стала шире, чем заданное число. Либо считайте слои вручную, либо переходите на `border-box`.
- **Забыть глобальный сброс `box-sizing`.** Без него проценты и пиксели в сетке ведут себя непредсказуемо, а отладка превращается в подбор значений на глаз.
- **Ставить `width` и `max-width` в пикселях одновременно и жёстко.** `width: 640px` плюс `max-width: 640px` бессмысленно; для адаптивности `width` задают в процентах, а `max-width` - в пикселях.
- **Ловить горизонтальный скролл и винить в нём вьюпорт.** Ищите переполняющую коробку: чаще всего это фиксированная ширина, `min-width` на flex-элементе или длинная неразрывная строка.

<Detail summary="Логические свойства: inline-size вместо width">
Современный CSS предлагает логические свойства: `inline-size` и `block-size` вместо `width` и `height`. Они привязаны не к экрану, а к направлению письма: `inline-size` - размер вдоль строки, `block-size` - поперёк. В привычной горизонтальной раскладке `inline-size` совпадает с `width`, но в вертикальном тексте или в интерфейсе на арабском они меняются местами автоматически. То же касается `padding-inline`, `margin-block` и подобных - вёрстка, написанная в логических свойствах, работает в любом направлении письма без переписывания.
</Detail>
