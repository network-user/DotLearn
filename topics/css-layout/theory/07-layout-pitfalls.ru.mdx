# Пять багов, на которые уходит по вечеру

Отдельные свойства вы уже знаете. Но раскладка ломается не на свойствах, а на их взаимодействии - в местах, где интуиция подсказывает одно, а браузер делает другое. Эти ловушки объединяет одна черта: код выглядит правильно, а результат неверен, и часами непонятно, где ошибка. Соберём пять самых частых, каждую - с симптомом, причиной и лечением, чтобы в следующий раз узнать врага в лицо за минуту.

## Ловушка 1: flex-элемент, который отказывается сжиматься

Симптом: внутри flex-контейнера лежит блок с длинным текстом, широкой таблицей или блоком `pre`, и он вылезает за пределы контейнера, растягивая всю строку и рождая горизонтальную прокрутку. Вы ставите `overflow: hidden`, `max-width: 100%` - не помогает.

Причина коварна: у flex- и grid-элементов по умолчанию `min-width: auto`, а это значит «не сжимайся уже своего содержимого». Длинное неразрывное содержимое задаёт большой минимальный размер, и `flex-shrink` бессилен - он не может опустить элемент ниже этого минимума.

```css
.feed-item {
  min-width: 0;
}
```

Лечение - одна строка: `min-width: 0` (для колонок - `min-height: 0`). Она снимает защиту, и элемент снова может сжиматься, отдавая переполнение внутреннему `overflow`.

## Ловушка 2: пропавший вертикальный отступ

Симптом: между заголовком с `margin-bottom: 40px` и абзацем с `margin-top: 20px` оказывается не 60 пикселей, а ровно 40. Отступ будто «съелся».

Причина - схлопывание margin (margin collapsing): вертикальные внешние отступы двух соседних блоков в нормальном потоке не складываются, а сливаются в один, равный большему из двух. Это осознанное решение авторов CSS ради ровного вертикального ритма, а не баг. Работает только для вертикальных `margin` и только в нормальном потоке.

```css
.title { margin-bottom: 40px; }
.lead  { margin-top: 20px; }
```

Между ними будет 40 пикселей, а не 60. Важное следствие из прошлых уроков: flex- и grid-контейнеры margin своих детей вообще не схлопывают - там расстояния задают через `gap`. Поэтому переезд с потока на flex часто «неожиданно» меняет вертикальные отступы.

## Ловушка 3: sticky, который не липнет

Симптом: `position: sticky` с заданным `top: 0` просто не работает, элемент прокручивается вместе со всем.

Двe обычные причины. Первая - не задан порог: `sticky` без `top` (или `bottom`) прилипать не к чему. Вторая, куда более скрытая - у одного из предков стоит `overflow: hidden` или `overflow: auto`, обрезающий область прокрутки, внутри которой sticky живёт. Лечение - искать `overflow` вверх по дереву и убирать лишний.

<Callout type="warning">
overflow на предке - причина номер один «неработающего» sticky. Часто overflow: hidden добавляют совсем для другой цели (обрезать тень или скролл), не подозревая, что он ломает прилипание глубоко внутри.
</Callout>

## Ловушка 4: z-index, который не поднимает

Симптом: у модального окна `z-index: 9999`, а оно всё равно оказывается под каким-то элементом с `z-index: 5`. Наращивание числа не помогает.

Причина - контекст наложения (stacking context). `z-index` действует не глобально, а только внутри своего контекста. Если модалка заперта в родителе, который сам создал контекст наложения (через `transform`, `opacity` меньше единицы, `filter`), её `z-index` соревнуется лишь с соседями внутри этого родителя, а наружу «просвечивает» только z-index самого родителя. Лечение - не крутить число, а вынести модалку выше по дереву, из-под родителя, создающего контекст (часто - в конец `body`).

## Ловушка 5: центрирование, которое раньше было болью

Симптом (исторический): «как отцентрировать блок и по горизонтали, и по вертикали». До flex это требовало таблиц, отрицательных margin или трансформаций. Сегодня ловушка не в сложности, а в том, что старые рецепты всё ещё копируют из привычки.

```css
.center {
  display: flex;
  justify-content: center;
  align-items: center;
}
```

Три строки центрируют что угодно внутри контейнера по обеим осям. Для одного элемента то же делает `display: grid; place-items: center`. Ловушка - тянуть `position: absolute` с `transform: translate(-50%, -50%)` там, где flex или grid решают задачу проще и без побочных эффектов на поток.

## Бонус: загадочный отступ под картинкой

Симптом: под `img` в блоке появляется несколько пикселей пустоты, которых нет ни в margin, ни в padding. Причина - природа картинки: по умолчанию `img` строчный элемент и сидит на базовой линии текстовой строки. А под базовой линией браузер резервирует место под «хвосты» букв (у p, g, y). Картинке этот запас не нужен, но, будучи строчной, она его занимает.

```css
img {
  display: block;
}
```

Лечение - вывести картинку из строчного потока: `display: block` (или `vertical-align: bottom`, если она должна остаться строчной, например внутри строки текста). Это одна из тех загадок, что съедают полчаса, пока не вспомнишь простую вещь из урока про поток: `img` - строчный элемент, живущий по правилам текста.

## Общий метод отладки

За всеми пятью ловушками - один навык: не менять свойства наугад, а спросить «в какой системе сейчас живёт этот элемент». В нормальном потоке, во flex-контейнере, в новом контексте наложения, в обрезанной области прокрутки. Инструменты разработчика в браузере подсвечивают box model, показывают, какой предок создал containing block, и рисуют границы grid-треков. Минута в инспекторе экономит вечер подбора значений.

<Detail summary="Почему horizontal scroll появляется «сам по себе»">
Горизонтальная полоса прокрутки всей страницы почти всегда значит, что какая-то одна коробка шире вьюпорта. Частые виновники: элемент с фиксированной шириной в пикселях больше экрана; flex- или grid-элемент с `min-width: auto` и длинным содержимым (ловушка 1); отрицательный margin, выталкивающий блок за край; `100vw` при наличии вертикальной полосы прокрутки (vw не вычитает её ширину). Быстрый способ найти виновника - временно задать `* { outline: 1px solid red }` и проскроллить: переполняющая коробка обведётся и выдаст себя.
</Detail>
