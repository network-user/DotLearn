# Резиновая колонка рядом с фиксированной

Панель фильтров слева ровно 280 пикселей, а лента карточек справа занимает всё остальное - и подстраивается, когда окно меняет ширину. Ни одного процента, ни одного медиазапроса: две строки flex делают то, на что раньше уходили расчёты и хаки. Flexbox вы, скорее всего, уже трогали. Здесь мы разберём его до дна: не «какое свойство центрирует», а как именно браузер делит место между элементами - потому что именно эта арифметика отделяет угадывание от понимания.

## Две оси - фундамент всего

`display: flex` на контейнере выстраивает его прямых детей вдоль главной оси (main axis). Перпендикулярна ей поперечная ось (cross axis). По умолчанию главная ось горизонтальна, но `flex-direction: column` разворачивает их: главной становится вертикаль.

`justify-content` распределяет свободное место вдоль главной оси, `align-items` выравнивает элементы поперёк неё. Перепутать их - самая частая ошибка: «центрирую, а съезжает не туда» почти всегда значит, что работает не та ось. Отдельному ребёнку можно переопределить выравнивание через `align-self`.

Значения `justify-content` различают по тому, куда уходит свободное место. `flex-start`, `center` и `flex-end` собирают элементы плотной группой у начала, центра или конца оси. Три распределяющих значения раскидывают промежутки между элементами: `space-between` прижимает крайние к бортам и делит остаток между ними; `space-around` даёт каждому элементу равные поля с обеих сторон, из-за чего промежуток между соседями визуально вдвое больше краевого; `space-evenly` делает все промежутки, включая краевые, буквально одинаковыми.

```css
.nav {
  display: flex;
  justify-content: space-between;
  align-items: center;
}
```

Панель навигации «логотип слева, меню справа» - это `space-between` на одну строку, без единого `margin-left: auto` вручную. У `align-items` набор шире по смыслу: `stretch` (по умолчанию) тянет элементы на всю высоту контейнера, если своя высота не задана; `center`, `flex-start`, `flex-end` выравнивают по центру и краям; а `baseline` ставит элементы так, чтобы совпали базовые линии их текста - незаменимо, когда в ряду соседствуют надписи разного кегля и их нужно «посадить» на одну линию.

## flex - это три числа

Слово «flex» - про гибкость самих элементов. Свойство `flex` - сокращение трёх: `flex-grow`, `flex-shrink`, `flex-basis`.

```css
.sidebar { flex: 0 0 280px; }
.feed    { flex: 1 1 0; }
```

`flex-basis` - стартовый размер элемента вдоль главной оси до распределения места. `flex-grow` - насколько охотно элемент заберёт лишнее свободное место. `flex-shrink` - насколько охотно сожмётся при нехватке. Читается это так: панель `flex: 0 0 280px` не растёт и не сжимается, всегда 280 пикселей; лента `flex: 1 1 0` стартует с нуля и забирает весь остаток. Отсюда та самая резиновая колонка из вступления.

## Как делится свободное место при росте

Вот арифметика, которую стоит уметь считать в уме. Браузер складывает `flex-basis` всех элементов, вычитает сумму из ширины контейнера и получает свободное место. Затем раздаёт его пропорционально `flex-grow`.

Пусть контейнер шириной 800 пикселей содержит два элемента: `.a { flex: 1 1 0 }` и `.b { flex: 3 1 0 }`. Базы нулевые, свободного места - все 800. Сумма grow-долей 1 + 3 = 4, значит одна доля - 200 пикселей. Элемент `.a` получает одну долю (200), `.b` - три (600).

Если базы ненулевые, они учитываются первыми. Контейнер 500 пикселей, `.a { flex: 1 1 200px }`, `.b { flex: 1 1 100px }`: сумма баз 300, свободного места 200, делится поровну (grow 1 и 1) - по 100 каждому. Итог: `.a` = 200 + 100 = 300, `.b` = 100 + 100 = 200.

## Как делится нехватка при сжатии

Сжатие - зеркало роста, но с важным поворотом: доля сжатия взвешивается не только на `flex-shrink`, но и на `flex-basis`. Чем крупнее элемент, тем больше он «виноват» в переполнении и тем сильнее ужимается.

Контейнер 300 пикселей, `.a { flex-shrink: 1; flex-basis: 400px }`, `.b { flex-shrink: 2; flex-basis: 200px }`. Суммарная база 600, переполнение 300. Вес сжатия каждого - это `shrink × basis`: у `.a` это 1 × 400 = 400, у `.b` это 2 × 200 = 400. Веса равны, значит переполнение 300 делится поровну: каждый ужимается на 150. Итог: `.a` = 400 − 150 = 250, `.b` = 200 − 150 = 50. Неочевидно: у `.b` вдвое больший `shrink`, но и вдвое меньшая база - и они уравновесились.

<Callout type="tip">
Практический вывод: при росте важен только flex-grow, при сжатии - произведение flex-shrink на flex-basis. Поэтому крупный элемент с обычным shrink ужимается заметно сильнее мелкого - это чувствуется как «резина разной жёсткости».
</Callout>

## Перенос, gap и порядок

По умолчанию flex-контейнер втискивает всё в одну строку, сжимая элементы. `flex-wrap: wrap` разрешает перенос на новую строку, когда места не хватает, - без него сетка карточек схлопнется до нечитаемой ширины на узком экране. Расстояние между элементами задаёт `gap` - без `margin` на каждой карточке и без возни с лишним отступом у крайнего элемента.

```css
.cards {
  display: flex;
  flex-wrap: wrap;
  gap: 16px;
}
```

Такой контейнер - готовая сетка карточек: они идут в ряд, переносятся при нехватке места и разделены ровными промежутками. Но у неё есть подводный камень, к которому мы вернёмся в уроке о ловушках: карточка с длинным неразрывным содержимым по умолчанию отказывается сжиматься уже своего содержимого (`min-width: auto`) и может вылезти за контейнер. Запомните симптом сейчас - лечится он одной строкой `min-width: 0`.

Порядок элементов можно менять свойством `order` (по умолчанию 0): элемент с меньшим `order` идёт раньше. Полезно для перестановки блоков на разных экранах - например, вынести в мобильной версии кнопку действия наверх, не трогая разметку. Но злоупотреблять опасно: визуальный порядок разъезжается с порядком в разметке, а значит и с порядком фокуса по Tab и озвучивания для скринридеров. Меняйте `order` только там, где это не ломает логическую последовательность содержимого.

## Частые ошибки

- **Путать `justify-content` и `align-items`.** Первое - вдоль главной оси, второе - поперёк. При `flex-direction: column` они меняются местами по направлению.
- **Забыть `flex-wrap: wrap` на сетке карточек.** Без него карточки сжимаются до микроскопической ширины вместо переноса.
- **Ждать, что `flex: 1` на всех даст равную ширину при разном содержимом.** С `flex: 1 1 0` - да, ширины равны; но `flex: 1 1 auto` учитывает содержимое, и элементы получаются разной ширины.
- **Ломать доступность через `order`.** Порядок в разметке остаётся исходным для клавиатуры и скринридеров; менять `order` можно, но осознанно.

<Detail summary="Почему flex: 1 и flex: auto ведут себя по-разному">
`flex: 1` разворачивается в `1 1 0%` - база нулевая, поэтому итоговые размеры определяются только долями grow, и три таких элемента будут строго равны, что бы в них ни лежало. `flex: auto` - это `1 1 auto`: база берётся из содержимого (`width` или естественная ширина), а grow лишь делит остаток сверху. Поэтому `flex: auto` даёт «пропорционально содержимому плюс поровну остаток», а `flex: 1` - «строго поровну». Отдельно стоит `flex: none` (`0 0 auto`) - жёсткий элемент по размеру содержимого, который не растёт и не сжимается.
</Detail>
