# Почему абзацы встают друг под другом, а ссылки - в ряд

Откройте любую страницу без единой строки вашего CSS. Заголовки и абзацы всё равно выстроятся сверху вниз, а ссылки и слова внутри абзаца - слева направо. Это не магия и не случайность: браузер раскладывает элементы по правилам нормального потока, и у каждого тега уже есть значение `display`, которое определяет, как он в этом потоке себя ведёт. Прежде чем брать в руки Flexbox или Grid, стоит понять раскладку, которая работает по умолчанию - потому что и flex, и grid лишь заменяют её внутри одного контейнера.

## Нормальный поток: два направления

Нормальный поток (normal flow) - это способ, которым браузер размещает элементы, пока вы не попросили иного. В нём есть ровно два типа участников. Блочные элементы (`display: block`) укладываются друг под другом сверху вниз, каждый начинает новую строку и по умолчанию занимает всю доступную ширину родителя. Строчные элементы (`display: inline`) текут внутри строки слева направо, как слова в предложении, и переносятся на новую строку только когда текущая заканчивается.

Отсюда и наблюдение из вступления: `p` и `h1` - блочные, поэтому встают колонкой; `a`, `span`, `strong` - строчные, поэтому выстраиваются в ряд внутри текста. Никакого CSS для этого не нужно, всё уже заложено в браузерные стили по умолчанию.

## Строчные элементы игнорируют размеры

Важная асимметрия: строчный элемент нельзя заставить принять `width`, `height` и вертикальные `margin`. Они на него просто не действуют.

```css
span {
  width: 200px;
  height: 100px;
}
```

Этот `span` не станет прямоугольником 200 на 100: его размеры определяет содержимое, а высоту строки - `line-height`. Причина логична - строчный элемент встроен в поток текста, и задать ему собственную высоту значило бы сломать ритм строки, в которую он вписан. Горизонтальные отступы (`margin-left`, `margin-right`, `padding`) на строчном элементе работают, а вертикальные - визуально не раздвигают соседние строки.

## inline-block: гибрид двух миров

Между двумя типами есть третий - `display: inline-block`. Такой элемент течёт в строке рядом с текстом и соседями, как строчный, но при этом уважает `width`, `height` и все стороны `margin`, как блочный.

```css
.badge {
  display: inline-block;
  width: 64px;
  height: 24px;
}
```

Именно это поведение нужно кнопкам, бейджам, тегам: они стоят в ряд по горизонтали, но им можно задать размер и вертикальные отступы. У картинки `img` похожая природа - формально она строчная, но габариты принимает, потому что это заменяемый (replaced) элемент, чей размер приходит вместе с файлом.

<Callout type="info">
`display: none` - отдельный случай: элемент полностью исчезает из потока, будто его нет в разметке, и не занимает места. Это не то же самое, что `visibility: hidden`, при котором элемент невидим, но своё место в потоке сохраняет.
</Callout>

## Ширина по умолчанию: тянется или жмётся

Блочный и строчный элементы по-разному реагируют на отсутствие явной `width`. Блочный при `width: auto` (значение по умолчанию) занимает всю доступную ширину родителя за вычетом собственных margin - поэтому абзац во всю строку, даже если текста в нём три слова. Строчный и `inline-block` при auto жмутся до ширины содержимого.

```css
.container {
  width: 640px;
  margin: 0 auto;
}
```

Из этого следует классический рецепт горизонтального центрирования блока: задать ему конечную `width` и `margin-left: auto; margin-right: auto` (сокращённо `margin: 0 auto`). Авто-margin забирает всё свободное место по горизонтали и делит его поровну слева и справа, ставя блок по центру. Важное условие: приём работает только для блочного элемента с заданной шириной. Без ширины блок и так растянут во всю строку, и центрировать попросту нечего - `margin: 0 auto` в этом случае не делает ничего.

## Контекст форматирования: почему поток «замыкается»

У нормального потока есть тонкость, которая позже объяснит половину странностей вёрстки. Некоторые элементы создают собственный блочный контекст форматирования (block formatting context, BFC) - изолированную область, внутри которой поток живёт по своим правилам и не «протекает» наружу. BFC создаётся, например, у элемента с `overflow` отличным от `visible`, у flex- и grid-контейнеров, у элементов с `display: flow-root`.

```css
.clearfix {
  display: flow-root;
}
```

Внутри BFC вертикальные `margin` не схлопываются с внешними, а плавающие (`float`) потомки не вылезают за его границы. Классический приём: контейнер с плавающими детьми «схлопывается» в нулевую высоту, потому что float выпадает из потока; `display: flow-root` создаёт BFC, и контейнер снова обнимает своих детей. Понимать BFC на старте необязательно наизусть, но полезно знать: это тот механизм, который делает `overflow: hidden` неожиданным лекарством от множества проблем раскладки.

## Flex и grid заменяют поток внутри контейнера

Главное, что стоит унести: `display: flex` и `display: grid` не отменяют нормальный поток на всей странице - они меняют раскладку только прямых детей одного контейнера. Сам контейнер остаётся блочным элементом в потоке родителя. Внутри же его дети перестают подчиняться правилам block/inline и начинают выстраиваться по осям flex или по трекам grid. Поэтому следующие уроки - это не отмена того, что вы сейчас прочитали, а новые режимы, включаемые точечно.

## Частые ошибки

- **Задавать `width` строчному элементу и ждать эффекта.** `span { width: 200px }` без смены `display` не сработает. Нужен `inline-block` или `block`.
- **Путать `display: none` и `visibility: hidden`.** Первый убирает элемент из потока целиком, второй прячет, но место сохраняет. Для сворачивания блока нужен именно `none`.
- **Удивляться «схлопнувшемуся» контейнеру с float внутри.** Плавающие дети выпадают из потока; родителю нужен BFC (`display: flow-root` или `overflow: hidden`), чтобы снова их обнять.
- **Считать, что `display: flex` меняет поведение самого контейнера.** Он меняет раскладку только его прямых детей; в потоке родителя контейнер по-прежнему обычный блок.

<Detail summary="display состоит из двух значений">
Современная спецификация трактует `display` как пару: внешнее поведение (как коробка ведёт себя в потоке родителя - `block` или `inline`) и внутреннее (как раскладываются её дети - `flow`, `flex`, `grid`). Привычное `display: flex` - это сокращение от `display: block flex`: снаружи блок, внутри flex-раскладка. Отсюда и `inline-flex` - flex-контейнер, который сам ведёт себя как строчный элемент в тексте родителя. Двухзначный синтаксис (`display: inline flow-root`) пока применяют редко, но он объясняет, почему у display так много на первый взгляд несвязанных значений.
</Detail>
