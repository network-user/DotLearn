# Как дать клиенту скачать приватный файл, не раздав ключи

Пользователь Ozon заказал документ и хочет скачать PDF с чеком. Файл лежит в приватном bucket - в открытый доступ его выкладывать нельзя, чеки чужими быть не должны. Очевидное решение: пусть браузер запросит файл у вашего бэкенда, бэкенд скачает его из S3 и перешлёт пользователю. Работает - пока чеков десяток в час. На распродаже их десятки тысяч, и весь трафик гигабайтных архивов и видео течёт через ваш сервер: его канал забит перекачкой файлов, которые он даже не обрабатывает, а только проксирует.

Есть способ убрать сервер из пути данных, не открывая bucket всему миру. Бэкенд выдаёт клиенту особую ссылку - **presigned URL**, - по которой тот скачивает файл напрямую из S3, минуя ваш сервер, но строго в рамках того, что разрешила ссылка. К концу раздела вы поймёте, что именно зашито в такую ссылку, чем presigned GET отличается от presigned PUT и presigned POST, как задаётся срок жизни и какой риск вы принимаете, выдавая её.

## Ментальная модель: подпись вместо постоянного доступа

Presigned URL - это обычный HTTPS-адрес объекта, к которому добавлены параметры запроса с криптографической подписью. Подпись вычисляется из ваших ключей доступа, имени bucket, ключа объекта, разрешённой операции и времени истечения. Получив такой URL, S3 проверяет подпись и, если она верна и срок не вышел, выполняет операцию - как будто запрос пришёл от владельца ключей. При этом сами ключи в ссылку не попадают: в ней лежит только подпись, восстановить из которой секрет нельзя.

Главное в этой схеме - стрелка снизу: данные идут напрямую между клиентом и S3. Бэкенд участвует только в коротком шаге подписи (стрелки 1-2), где нет ни байта файла. Сервер раздаёт лёгкие подписи, а тяжёлый трафик берёт на себя S3.

## Presigned GET: временная ссылка на скачивание

Самый частый случай - дать скачать приватный объект. `generate_presigned_url` с операцией `get_object` возвращает строку-URL.

```python
url = client.generate_presigned_url(
    "get_object",
    Params={"Bucket": "ozon-media", "Key": "receipts/42.pdf"},
    ExpiresIn=300,
)
```

Этот URL действует пять минут (`ExpiresIn` - в секундах). Любой, у кого он есть, скачает `receipts/42.pdf` без всякой авторизации - подпись уже всё подтвердила. Через пять минут та же ссылка вернёт ошибку `AccessDenied`: срок вышел, подпись больше не действительна. Бэкенд отдаёт такую ссылку в ответ на запрос «дай скачать чек», и дальше браузер качает файл сам.

<Callout type="info">
Presigned URL работает и для приватного bucket, и при включённом Block Public Access. Это не публикация объекта: доступ даёт подпись на конкретную операцию и конкретный ключ, а не открытость bucket. Объект остаётся приватным для всех, у кого подписанной ссылки нет.
</Callout>

## Presigned PUT: дать загрузить, не пропуская файл через себя

Симметрично решается обратная задача - приём файла от клиента. Если пользователь грузит аватар или видео, проксировать загрузку через бэкенд так же расточительно. Presigned URL для `put_object` разрешает клиенту записать объект напрямую в S3.

```python
url = client.generate_presigned_url(
    "put_object",
    Params={"Bucket": "ozon-media", "Key": "uploads/42/avatar.png"},
    ExpiresIn=600,
)
```

Клиент делает HTTP-`PUT` по этому URL, кладя тело файла, - и объект появляется в bucket под заданным ключом. Бэкенд при этом задаёт ключ заранее (например, по id пользователя), так что клиент не выбирает, куда писать: он пишет ровно туда, куда разрешила подпись.

## Presigned POST: загрузка прямо из HTML-формы

У presigned PUT есть ограничение: им неудобно грузить файл обычной браузерной формой и трудно ограничить размер или тип. Для этого есть `generate_presigned_post` - он возвращает не строку, а пару «URL формы + набор скрытых полей», которые вставляют в HTML-форму с `enctype="multipart/form-data"`.

```python
data = client.generate_presigned_post(
    Bucket="ozon-media",
    Key="uploads/42/avatar.png",
    Fields={"Content-Type": "image/png"},
    Conditions=[["content-length-range", 0, 5_000_000]],
    ExpiresIn=600,
)
```

Возвращённый словарь содержит `url` и `fields` - их браузер отправит вместе с файлом. Ключевое преимущество - **условия** (`Conditions`): здесь подпись разрешает загрузку только если размер тела от 0 до 5 МБ. Клиент физически не сможет залить файл больше: S3 отвергнет запрос, не совпавший с условиями. Так пользователь грузит файл прямо в S3 из формы, а вы заранее ограничили, что именно ему позволено.

## Срок жизни: короче - безопаснее

Параметр `ExpiresIn` задаёт, сколько секунд ссылка действительна. Это центральный рычаг безопасности: чем короче срок, тем меньше окно, в которое утёкшая ссылка опасна. Для скачивания по клику обычно хватает нескольких минут; для фоновой загрузки большого файла берут больше, но в пределах разумного. Бесконечных presigned URL не бывает: верхний предел зависит от способа подписи (для подписи ключами IAM-пользователя - до 7 суток), но закладываться на максимум - плохая привычка.

## Зачем это всё: разгрузка бэкенда

Сведём выгоду воедино. Без presigned URL ваш сервер - посредник в передаче каждого байта: он скачивает из S3 и отдаёт клиенту, или принимает от клиента и заливает в S3. Его процессор, память и канал тратятся на перекачку данных, которые он не обрабатывает. С presigned URL сервер делает лишь дешёвую операцию подписи (никакого тела файла), а тяжёлый трафик S3 берёт на себя - сервис, который для этого и построен. На больших файлах и высоком трафике это разница между «канал сервера - узкое место» и «сервер вообще не в пути данных».

## Сюрприз: ссылка - это и есть доступ, отозвать её нельзя

Здесь прячется главный риск. Presigned URL самодостаточен: всё нужное для доступа лежит прямо в нём. Поэтому любой, кому ссылка попала в руки, имеет ровно тот доступ, что она даёт, - до истечения срока. Утекла в логи прокси, в историю браузера, в пересланное сообщение, в скриншот - и до конца срока по ней можно качать (или писать) объект, не будучи тем, кому её выдали.

Хуже того, отозвать конкретную ссылку нельзя: подпись уже подсчитана и живёт сама по себе, «кнопки отзыва» у неё нет. Единственный способ разом погасить все выданные ссылки - сменить или отключить ключ доступа, которым они подписаны, но это обрушит и все остальные подписи. Поэтому короткий `ExpiresIn` - не перестраховка, а основная защита: он ограничивает время, в течение которого утечка вообще что-то значит.

<Callout type="warning">
Не передавайте presigned URL способами, где они оседают надолго: не кладите их в URL, который попадёт в access-логи прокси, не сохраняйте в общедоступных местах. И не выдавайте срок «на всякий случай побольше» - вы продлеваете окно, в котором утёкшая ссылка работает.
</Callout>

## Типичные ошибки

- **Считать presigned URL публикацией объекта.** Это не «сделать файл публичным»: доступ привязан к одной операции, одному ключу и сроку. Открывать bucket целиком, чтобы «раздать ссылки», - значит выдать куда больше, чем presigned URL и так умеет точечно.
- **Выдавать огромный `ExpiresIn`.** Долгая ссылка удобна, пока не утекла. Срок - это окно атаки; берите минимально достаточный.
- **Путать сторону подписи и сторону использования.** Подписывает ссылку бэкенд (тот, у кого ключи); ходит по ней клиент (у кого ключей нет). Если клиенту приходится «подписывать» что-то самому, схема построена неверно - секрет утёк бы на клиент.

<Detail summary="Что именно лежит в подписи и почему её нельзя подделать">

В современной схеме (AWS Signature Version 4) presigned URL несёт в query-параметрах: идентификатор ключа доступа (`X-Amz-Credential`), время создания (`X-Amz-Date`), срок жизни (`X-Amz-Expires`), список подписанных заголовков и саму подпись (`X-Amz-Signature`). Подпись - это HMAC-SHA256 от канонизированного запроса (метод, путь, параметры, заголовки) на ключе, выведенном из вашего секретного ключа доступа. S3 на своей стороне повторяет тот же расчёт по пришедшему запросу и сравнивает подписи. Совпали - запрос подлинный; не совпали или срок вышел - `AccessDenied`. Подделать подпись без секретного ключа нельзя: в этом и смысл HMAC. А поскольку проверка не требует обращения к базе или предварительной регистрации ссылки, S3 валидирует presigned URL мгновенно и без состояния - оттого их отзыв поштучно и невозможен.

</Detail>
