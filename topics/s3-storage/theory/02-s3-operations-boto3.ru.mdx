# Почему `list_objects_v2` не отдаёт вам все файлы

Вы написали код, который выгружает имена всех объектов из bucket: один вызов `list_objects_v2`, цикл по результату, готово. На тестовом bucket с десятком файлов всё работает. Через месяц в проде там лежит сорок тысяч объектов, а ваш код упорно видит ровно тысячу - всегда одну и ту же тысячу, как будто остальные исчезли. Ничего не исчезло: S3 физически не отдаёт больше тысячи ключей за один вызов, и об этом он честно сообщает в ответе - флагом, который ваш код проигнорировал.

boto3 - официальная Python-библиотека AWS, и работа с S3 через неё - это горстка методов, которые легко выучить за вечер. Но два места в этом API регулярно ставят подножку: листинг, который разбит на страницы, и загрузка больших файлов, которую нельзя делать одним куском. К концу раздела вы будете уверенно класть и читать объекты, дочитывать листинг до конца через пагинацию и понимать, когда обычная загрузка должна уступить место multipart.

## Ментальная модель: client - это карта HTTP-вызовов S3

boto3 даёт два уровня доступа к S3. Низкоуровневый - `client`: каждый его метод соответствует одному HTTP-запросу к S3 API. `put_object` - это `PUT`, `get_object` - `GET`, `list_objects_v2` - `GET` со списком. Имена и параметры повторяют протокол S3 один в один. Высокоуровневый - `resource`: объектно-ориентированная обёртка (`bucket.objects.all()`, `obj.download_file(...)`), которая прячет часть деталей.

```python
import boto3

client = boto3.client("s3")
resource = boto3.resource("s3")
```

Практическое правило простое: для предсказуемости и доступа ко всем параметрам берите `client`; `resource` удобен для коротких скриптов, но в AWS его развитие фактически заморожено, и новый код обычно пишут на `client`. Дальше в разделе - `client`, потому что именно его поведение нужно понимать, чтобы не попасться на пагинации.

## Положить и прочитать объект

Базовая пара операций - запись и чтение объекта по ключу.

```python
client.put_object(Bucket="ozon-media", Key="users/42/note.txt", Body=b"привет")

resp = client.get_object(Bucket="ozon-media", Key="users/42/note.txt")
data = resp["Body"].read()
print(data.decode())
```

`put_object` принимает `Body` - байты, строку или файловый объект - и кладёт их под ключом. `get_object` возвращает словарь, где под ключом `Body` лежит не сами байты, а **StreamingBody** - потоковый объект. Это важная деталь: тело не загружается в память целиком в момент вызова. Вы дочитываете его сами - `.read()` забирает всё, а для большого объекта правильнее читать кусками, чтобы не положить процесс гигабайтным файлом. После чтения поток закрывают.

<Callout type="info">
`resp["Body"]` - это поток, а не готовая строка. Частая ловушка - распечатать `resp["Body"]` и удивиться объекту `StreamingBody` вместо содержимого. Данные появляются только после `.read()` (или `.iter_chunks()` для чтения по частям).
</Callout>

## HEAD: метаданные без тела

Иногда нужно узнать про объект, не скачивая его: существует ли он, какого размера, какого типа. Для этого есть `head_object` - он возвращает только метаданные.

```python
meta = client.head_object(Bucket="ozon-media", Key="users/42/note.txt")
print(meta["ContentLength"], meta["ContentType"])
```

`HEAD` не передаёт тело, поэтому он дёшев и быстр даже для огромного объекта. Его же используют как проверку существования: если объекта нет, `head_object` бросит исключение с кодом `404`. Скачивать гигабайт ради того, чтобы узнать его размер, не нужно.

## Листинг и пагинация: вот где прячется баг из начала

`list_objects_v2` возвращает список объектов bucket, но за один вызов отдаёт **не больше 1000 ключей** - таков жёсткий лимит S3. Если объектов больше, ответ обрезается, и S3 помечает это флагом `IsTruncated: True`, прикладывая `NextContinuationToken` - закладку, с которой нужно запросить следующую страницу.

Параметр `MaxKeys` задаёт размер страницы (по умолчанию и максимум - 1000). Уменьшить его можно, увеличить сверх 1000 - нет. Чтобы получить все ключи, листинг крутят в цикле, передавая токен предыдущей страницы, пока `IsTruncated` не станет ложным.

```python
token = None
keys = []
while True:
    params = {"Bucket": "ozon-media", "Prefix": "users/"}
    if token:
        params["ContinuationToken"] = token
    resp = client.list_objects_v2(**params)
    keys += [o["Key"] for o in resp.get("Contents", [])]
    if not resp["IsTruncated"]:
        break
    token = resp["NextContinuationToken"]
```

Ровно эту логику инкапсулирует **paginator** - встроенный помощник boto3, который сам прокручивает страницы:

```python
paginator = client.get_paginator("list_objects_v2")
for page in paginator.paginate(Bucket="ozon-media", Prefix="users/"):
    for obj in page.get("Contents", []):
        print(obj["Key"])
```

Paginator избавляет от ручной возни с токенами и не теряет страницы. Параметр `Prefix` фильтрует ключи по началу строки (тот самый «вход в папку» из первого раздела), а `Delimiter="/"` группирует общие префиксы в `CommonPrefixes` - так консоль и рисует «папки».

<Callout type="warning">
Пустой bucket или префикс без объектов не вернёт ключ `Contents` вовсе. Поэтому пишут `resp.get("Contents", [])`, а не `resp["Contents"]`: прямое обращение упадёт с `KeyError` на пустом результате. Это вторая по частоте ошибка после игнорирования `IsTruncated`.
</Callout>

## Удаление: по одному и пачкой

Удалить объект - `delete_object` по ключу. Удалить много за один запрос - `delete_objects`, принимающий список ключей (до 1000 за вызов).

```python
client.delete_object(Bucket="ozon-media", Key="users/42/note.txt")

client.delete_objects(
    Bucket="ozon-media",
    Delete={"Objects": [{"Key": "a.txt"}, {"Key": "b.txt"}]},
)
```

Пакетное удаление экономит сетевые хопы: тысяча отдельных `delete_object` - это тысяча запросов, один `delete_objects` со списком из тысячи ключей - один запрос. Удаление несуществующего ключа, к слову, не считается ошибкой - S3 отвечает успехом, как будто объект был и его убрали.

## High-level: upload_file и download_file

Для работы с файлами на диске есть удобная пара поверх client: `upload_file` и `download_file`. Они принимают путь к локальному файлу и сами решают, как его передать.

```python
client.upload_file("/tmp/big.zip", "ozon-media", "backups/big.zip")
client.download_file("ozon-media", "backups/big.zip", "/tmp/restored.zip")
```

Их главное достоинство - не в краткости. Эти методы сами включают **multipart**, когда файл крупный, и грузят его кусками параллельно, с автоматическими повторами упавших частей. Вам не нужно вручную решать, резать файл или нет, - библиотека делает это за вас по порогу размера.

## Multipart upload: большой файл частями

Объект до 5 ГБ можно залить одним `put_object`. Больше - физически нельзя: это верхний предел одной загрузки. Для больших объектов существует multipart upload - файл режут на части (минимум 5 МБ каждая, кроме последней), грузят части по отдельности, а в конце просят S3 собрать их в один объект.

Зачем это нужно, кроме обхода лимита в 5 ГБ. Во-первых, докачка: если часть не долетела, повторяют только её, а не весь файл с нуля. Во-вторых, параллелизм: части грузят одновременно, утилизируя канал. В-третьих, устойчивость к обрывам сети на гигабайтных загрузках. Платить за это приходится сложностью - и одной коварной деталью.

## Сюрприз: брошенный multipart остаётся и стоит денег

Если multipart начали (`CreateMultipartUpload`), залили часть частей, но не вызвали `CompleteMultipartUpload` и не прервали загрузку явно, - залитые части **не исчезают сами**. Они лежат в bucket как незавершённая загрузка, не видны в обычном листинге объектов, но занимают место и тарифицируются как хранение. Прерванный деплой или упавший скрипт могут месяцами копить такие «осколки», раздувая счёт, а владелец bucket даже не видит их среди объектов.

Лечится это lifecycle-правилом «удалять незавершённые multipart-загрузки старше N дней» (о lifecycle - в пятом разделе) и явным `abort_multipart_upload` в обработке ошибок.

## Типичные ошибки

- **Игнорировать `IsTruncated` и читать только первую страницу.** Самая частая ошибка с S3 в Python: код видит ровно 1000 (или `MaxKeys`) объектов и считает, что это все. Всегда дочитывайте листинг через токен или, лучше, через paginator.
- **Считать `resp["Body"]` готовыми данными.** Это поток `StreamingBody`. Содержимое появляется после `.read()`; забыли прочитать - получили объект потока вместо байтов.
- **Слать гигабайты одним `put_object`.** Выше 5 ГБ это просто не сработает, а выше сотен мегабайт - хрупко на нестабильной сети. Для больших файлов берите `upload_file` (он включит multipart сам) или multipart напрямую.

<Detail summary="Почему именно 1000 ключей и 5 ГБ - откуда эти числа">

Оба лимита - инженерный компромисс протокола. Лимит в 1000 ключей на страницу листинга ограничивает размер одного XML-ответа и время его формирования: без потолка листинг bucket с сотней миллионов объектов выдал бы гигантский ответ и заблокировал бы сервер. Пагинация перекладывает обход на клиента маленькими порциями. Лимит в 5 ГБ на одиночный `PUT` ограничивает объект, который должен пройти как единое тело запроса; сам объект при этом может быть огромным (до 5 ТБ) - просто собирать его сверх 5 ГБ положено через multipart, где каждая часть остаётся управляемого размера. Числа удобно держать в голове как «тысяча ключей за листинг, пять гигабайт за один PUT».

</Detail>
