# Веб-шрифты без сдвигов

Вы открываете статью на телефоне в парке, интернет не самый быстрый, и первые секунды заголовок вообще не показывается: место под текст на странице пустое, хотя разметка и CSS давно загружены. Бывает и наоборот: текст появляется сразу, но через мгновение буквы дёргаются и переставляются, будто страница ожила сама по себе. В обоих случаях виноват один и тот же ресурс: веб-шрифт, который браузер ещё не докачал.

То, как именно браузер поступает с текстом, пока веб-шрифт в пути, решает одно CSS-свойство, `font-display`. Разобравшись, как оно устроено и какими числами измеряется цена шрифта, вы сможете выбрать между невидимым текстом, миганием и лишними килобайтами осознанно, а не по умолчанию браузера.

## Невидимый текст против мигания

У браузера по сути два способа обращаться с текстом, пока веб-шрифт качается: спрятать его или показать чем-то другим. Спецификация делит время на периоды: сначала период блокировки, потом период подмены. Значение `font-display` решает, сколько длится первый период и что происходит, если шрифт так и не пришёл вовремя. На схеме ниже видно два крайних случая для одного и того же шрифта.

При `font-display: block` браузер держит текст невидимым на короткий период блокировки, обычно около 3 секунд. Если веб-шрифт успевает за это время, текст сразу выходит нужным начертанием. Если нет, браузер переключается на fallback, а веб-шрифт всё равно подставляется позже, когда он наконец придёт. Пользователь в худшем случае видит пустоту, потом текст, потом ещё раз текст, но уже другой гарнитурой. Такое поведение достаётся почти даром: специально настраивать ничего не нужно, оно и было исторически принятым по умолчанию ещё до того, как свойство `font-display` вообще появилось в спецификации.

При `font-display: swap` периода невидимости нет вовсе: fallback выходит на экран немедленно, а веб-шрифт просто подменяет его собой при готовности. Текст читаем с первого кадра, поэтому swap чаще всего и выбирают для основного контента.

Есть и промежуточные значения. `auto` оставляет решение браузеру, и на практике оно обычно ведёт себя близко к `block`. `fallback` даёт очень короткий период блокировки и ограниченное окно на подмену, после которого браузер оставляет fallback навсегда, даже если шрифт всё же догрузится. `optional` идёт дальше всех: если сеть занята или шрифт не успел прийти почти мгновенно, браузер может вообще не подставить веб-шрифт в этой загрузке, полагаясь на кеш при следующем визите.

## Разбор примеров

Объявление шрифта в CSS выглядит так:

```css
@font-face {
  font-family: 'Inter';
  src: url('/inter.woff2') format('woff2');
  font-display: swap;
}
```

Формат `woff2` представляет собой современный сжатый формат веб-шрифтов, его поддерживают все актуальные браузеры, и он ощутимо легче старых `woff` и `ttf`. Указывать несколько форматов через запятую в `src` сегодня почти не нужно: практически весь трафик получает `woff2` первым же совпадением.

Даже со `swap` шрифт всё равно должен откуда-то взяться, и обычно он стартует поздно: браузер находит `@font-face` только после разбора CSS, значит запрос за шрифтом идёт где-то в конце цепочки HTML, затем CSS, затем шрифт. Критичный шрифт, без которого не отрисуется заголовок, стоит объявить заранее через preload:

```html
<link rel="preload" href="/inter.woff2" as="font" type="font/woff2" crossorigin>
```

Атрибут `as="font"` подсказывает браузеру приоритет запроса, а `crossorigin` обязателен даже для шрифта с того же домена: шрифты по спецификации всегда качаются в анонимном режиме CORS, и без этого атрибута браузер скачает файл вторым, отдельным запросом, будто preload не случался вовсе. Этот приём разрывает цепочку зависимостей так же, как и в [[resource-loading]]: чем раньше стартует запрос за критичным ресурсом, тем меньше он держит первый осмысленный кадр.

Тут есть тонкость, о которую легко споткнуться: preload должен указывать ровно тот файл, который CSS реально запросит, вплоть до пути и формата. Если preload тянет `inter.woff2`, а `@font-face` в итоге просит другой файл или другой диапазон символов, браузер один раз скачает шрифт впустую, а затем ещё раз по-настоящему, и выигрыша в скорости не будет вовсе.

<Callout type="tip" title="Один файл вместо десяти">
Если сайту нужно сразу несколько начертаний одной гарнитуры (обычный текст, полужирный, курсив), необязательно тянуть отдельный файл на каждое. Variable font хранит целый диапазон значений веса в одном файле, и браузер сам интерполирует нужную толщину внутри `@font-face` с записью вроде `font-weight: 100 900`. Для сайта с тремя-четырьмя начертаниями это часто оказывается легче, чем несколько статических файлов вместе взятых, и уж точно означает меньше отдельных запросов.
</Callout>

Вес самого файла шрифта тоже поддаётся управлению. Полный файл шрифта вроде Roboto несёт кириллицу, латиницу и ещё десятки других письменностей, о которых конкретный сайт может даже не знать. Subsetting выбрасывает из файла глифы, которых сайт не использует, и оставляет, например, только кириллицу и латиницу. Экономия обычно измеряется в разах, а не в процентах.

300 КБ полного набора превращаются в 120 КБ subset-версии, то есть подмножество занимает 40% исходного веса. Уточнить диапазон символов внутри одного файла можно и без пересборки всего набора, через дескриптор `unicode-range` в `@font-face`: браузер скачает файл, только если на странице реально встретился символ из указанного диапазона, и это удобно, когда на сайте подключены сразу несколько начертаний одного шрифта.

```css
@font-face {
  font-family: 'Inter';
  src: url('/inter-cyrillic.woff2') format('woff2');
  unicode-range: U+0400-04FF;
}

@font-face {
  font-family: 'Inter';
  src: url('/inter-latin.woff2') format('woff2');
  unicode-range: U+0000-00FF;
}
```

Оба блока делят одно и то же имя `font-family` на два отдельных файла по алфавитам. Если страница целиком на кириллице, латинский файл браузер вообще не запросит: попадание в диапазон проверяется лениво, по факту встреченных на странице символов, а не по самому объявлению в CSS.

## Что вас удивит

Кажется, что `swap` решает всё: невидимого текста больше нет, значит и цена шрифта в глазах пользователя обнулилась. На деле у `swap` есть отложенный побочный эффект. В момент подмены fallback на веб-шрифт две гарнитуры почти никогда не совпадают по метрикам: ширина конкретных букв, межстрочный интервал и общая ширина слова отличаются. Абзац, который секунду назад ровно укладывался в три строки, вдруг растягивается на четыре, а всё, что стояло ниже на странице, дёргается вниз. Разница в средней ширине символа между системным шрифтом вроде Arial и типичным веб-шрифтом достигает нескольких процентов на каждую букву, а на длинном абзаце эти проценты складываются в целую лишнюю строку. Такая подвижка учитывается в Core Web Vitals как самый настоящий сдвиг макета, подробнее в [[core-web-vitals]].

Смягчить это можно двумя способами, и они не исключают друг друга. Первый: подобрать fallback с метриками, близкими к веб-шрифту, через дескрипторы `size-adjust`, `ascent-override` и `descent-override` в `@font-face` для fallback-гарнитуры, чтобы разница в занимаемом месте была почти незаметна. Второй: ускорить приход самого веб-шрифта через preload, чтобы подмена случилась до того, как пользователь успел начать читать, а не посреди чтения.

## Частые ошибки

Оставлять `font-display` без явного значения. Тогда браузер применяет поведение, близкое к `auto`, и по сути ведёт себя как `block`: короткий, но реальный период невидимого текста при каждой холодной загрузке. Исправление: указывать `swap` явно для основного текста, если читаемость важнее точного совпадения с дизайном с первого кадра.

Забывать `crossorigin` на preload шрифта. Без этого атрибута браузер считает предзагруженный файл отдельным ресурсом, и запрос за самим шрифтом стартует заново, будто preload вообще не было. Исправление: всегда ставить `crossorigin` на preload шрифтов, даже если шрифт лежит на том же домене.

Держать в проекте полный файл шрифта со всеми письменностями. Лишние глифы утяжеляют загрузку без всякой пользы, если сайт работает только на русском и английском. Исправление: собирать subset под реальный набор языков сайта и подключать дополнительные начертания только там, где они правда нужны.

Подключать сразу много начертаний одного шрифта на всякий случай, все weight и style подряд. Каждое начертание грузится отдельным файлом и отдельным запросом, а используется из них обычно два-три. Исправление: подключать только реально применяемые сочетания weight и style, а остальные оставлять системному fallback.

<Detail summary="Как боролись с миганием шрифта до font-display">

До появления `font-display` в спецификации оставалось два грубых обходных пути. Первый: JavaScript-библиотеки вроде Font Face Observer подписывались на событие загрузки шрифта и сами переключали класс на `<html>`, добавляя фактически ту же логику swap руками, только с задержкой на выполнение скрипта. Второй: шрифт заранее превращали в data-URI и клали прямо в критичный CSS, чтобы убрать сетевой запрос вовсе, ценой раздутого HTML. `font-display` сделал оба обходных пути ненужными почти везде: браузер решает эту задачу одной строкой в `@font-face`, без единого байта JavaScript.

</Detail>
