# Главный поток: бандл, длинные задачи и reflow

Страница весит 800 КБ JavaScript. Даже по быстрой сети байты придут за доли секунды, но дальше начинается настоящая работа: телефон должен эти 800 КБ распарсить, скомпилировать и выполнить, и всё это в одном потоке, который параллельно рисует страницу и обрабатывает нажатия. Пока поток занят вашим кодом, палец пользователя стучит по неотзывчивому экрану. Размер загрузки описывает лишь половину проблемы; вторая половина в том, сколько работы вы взвалили на главный поток. Этот концепт про два рычага: грузить меньше кода и заставлять поток делать меньше за раз.

## Один поток на всё

У вкладки есть главный поток, и он один. В нём происходит разбор HTML, построение стилей, layout, paint, выполнение вашего JavaScript и обработка кликов, ввода, скролла. Это очередь: пока выполняется одна задача, следующая ждёт. Клик, пришедший в разгар тяжёлой задачи, не обрабатывается сразу, он стоит в очереди до её конца.

Отсюда два независимых способа ускориться, и они дополняют друг друга. Первый: привезти на поток меньше кода, чтобы разбор и компиляция были короче. Второй: разбить работу на мелкие куски, чтобы между ними поток успевал ответить на действия пользователя. Начнём с первого.

## Меньше кода: бандл и code-splitting

Сборщик по умолчанию склеивает весь ваш JavaScript в один бандл. Проблема в том, что на стартовом экране нужна лишь малая его часть, а платит пользователь за всё сразу. Code-splitting разрезает бандл на чанки и грузит только необходимое здесь и сейчас, а остальное подгружает по требованию.

Самый крупный выигрыш даёт разбиение по маршрутам: код страницы настроек не нужен, пока пользователь на главной. В большинстве сборщиков граница чанка задаётся динамическим импортом:

```js
const Settings = lazy(() => import('./Settings'));
```

Такой `import()` возвращает промис, и сборщик выносит `Settings` со всеми его зависимостями в отдельный файл. На схеме выше стартовые 800 КБ превратились в 300 КБ, потому что 500 КБ уехали в ленивые чанки. Пользователь, оставшийся на главной, эти 500 КБ не скачает вовсе.

## Меньше кода: tree-shaking

Второй способ облегчить бандл работает на этапе сборки. Tree-shaking выбрасывает экспортируемый код, который вы никуда не импортировали. Название буквальное: сборщик трясёт дерево зависимостей, и мёртвые ветки опадают.

Условие на схеме важное: tree-shaking работает только при статических `import` и `export` из стандартных модулей (ESM). Если библиотека собрана в старом формате CommonJS или помечена как имеющая побочные эффекты, сборщик не рискнёт удалять её части, и в бандл приедет вся. Отсюда практическое правило: импортируйте точечно (`import { debounce } from ...`), а не целую библиотеку ради одной функции.

## Меньше работы за раз: длинные задачи

Теперь второй рычаг. Задачу дольше 50 мс браузер считает длинной (long task), потому что на это время поток перестаёт отвечать на ввод. Именно длинные задачи портят INP, метрику отзывчивости из концепта [[core-web-vitals]]: пользователь кликнул, а поток занят, и отклик приходит с задержкой.

Пусть тяжёлая обработка занимает 250 мс. Клик, пришедший на сотой миллисекунде, обработается только через 150 мс, как на первой схеме. Лечится это разбиением работы на короткие куски с уступанием потока между ними:

```js
async function processAll(items) {
  for (const chunk of chunks(items, 50)) {
    handle(chunk);
    await scheduler.yield();
  }
}
```

После каждого куска `scheduler.yield()` возвращает управление браузеру, и тот успевает обработать накопившийся ввод, прежде чем продолжить. Если та же работа нарезана на куски по 50 мс, клик посреди куска ждёт в худшем случае до его конца, десятки миллисекунд вместо сотен. Работа не стала меньше, но отзывчивость выросла, потому что у браузера появились окна между кусками.

## Меньше работы за раз: reflow и repaint

У главного потока есть коварная статья расходов, которую легко создать случайно. Когда меняется геометрия (ширина, положение, количество элементов), браузер пересчитывает layout, этот пересчёт называют reflow. После него идёт repaint, перекраска затронутых пикселей. Reflow дороже repaint, потому что задевает соседей: сдвинулся один блок, пересчитать нужно всю раскладку вокруг.

Опасность в том, что чтение некоторых свойств заставляет браузер сделать reflow немедленно и синхронно. Спросив `offsetHeight`, вы требуете актуальный размер, а значит браузер обязан прямо сейчас применить все отложенные изменения макета. Если в цикле чередовать чтение и запись, каждый оборот форсирует свой reflow:

```js
for (const el of items) {
  const h = el.offsetHeight;
  el.style.height = h * 2 + 'px';
}
```

Здесь запись `style.height` помечает layout «грязным», а следующее чтение `offsetHeight` требует его пересчитать. Сто элементов дают сто синхронных reflow за цикл. Явление называют layout thrashing. Лекарство в разделении фаз: сначала прочитать все размеры, потом в отдельном проходе все записать.

```js
const heights = items.map((el) => el.offsetHeight);
items.forEach((el, i) => {
  el.style.height = heights[i] * 2 + 'px';
});
```

Теперь чтения не перемежаются с записями, и браузер обходится одним пересчётом вместо ста.

## Что вас удивит

Самая дешёвая анимация не трогает layout вовсе. Свойства `transform` и `opacity` браузер умеет анимировать на отдельном слое силами композитора, не запуская ни reflow, ни даже repaint основного контента. А вот анимация `width`, `top` или `margin` дёргает layout на каждом кадре и потому дёргается сама. Правило простое: двигайте через `transform`, а не через `left`.

Ещё одна неожиданность ждёт на серверном рендеринге. Сервер прислал готовый HTML, страница выглядит загруженной, но она ещё не отвечает: браузер обязан скачать тот же по смыслу JavaScript и «оживить» разметку (hydration), и всё это ложится на главный поток. Пользователь видит контент и жмёт кнопку, а обработчик ещё не подключён. Красивый первый кадр не означает готовность к взаимодействию.

## Частые ошибки

Один гигантский бандл на всё приложение. Пользователь платит разбором и компиляцией за код всех экранов сразу. Помогает разбиение по маршрутам через динамический `import()`.

Импорт целой библиотеки ради одной функции. Если tree-shaking не сработал (CommonJS, побочные эффекты), в бандл приедут все её килобайты. Импортируйте нужное точечно и проверяйте вес бандла анализатором.

Тяжёлая синхронная работа на старте. Разбор большого JSON или построение списка на тысячи узлов в один присест создаёт длинную задачу и роняет INP. Нарежьте работу и уступайте поток.

Анимация геометрических свойств. `width` и `top` в переходах заставляют layout пересчитываться каждый кадр. Замените их на `transform`, и анимация уедет на композитор.

<Detail summary="Reflow, repaint, composite: три этажа стоимости">

Изменения на странице обходятся по-разному. Смена геометрии запускает полную цепочку: reflow (пересчёт layout), затем repaint (перекраска), затем composite (склейка слоёв). Смена только цвета или фона пропускает reflow: layout не поменялся, нужен лишь repaint плюс composite. А `transform` и `opacity` при наличии отдельного слоя доходят только до composite, самого дешёвого этажа, потому что браузер просто иначе складывает уже нарисованные слои. Поэтому плавные интерфейсы стараются держать анимации на уровне композитора и избегать свойств, которые роняют изменение на нижние, дорогие этажи.

</Detail>
