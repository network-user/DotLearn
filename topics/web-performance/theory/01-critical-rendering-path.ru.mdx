# Критический путь рендеринга

Вы выкатили страницу, открыли её и секунду смотрите на белый экран, а потом весь контент появляется разом. Между моментом, когда сервер прислал первый байт HTML, и моментом, когда браузер закрасил первый пиксель, происходит цепочка обязательных шагов. Эту цепочку называют критическим путём рендеринга. Понять её стоит хотя бы ради одного факта: единственный тег в `<head>` способен отложить показ всей страницы, и виноват в белом экране обычно не сервер, а порядок ресурсов. Разобравшись с путём, вы научитесь видеть, что именно держит первый кадр, и убирать это адресно.

## Из байтов в пиксели

Браузер не показывает HTML напрямую. Он прогоняет его через конвейер, где у каждого шага свой вход и выход. Сначала из потока символов HTML собирается DOM: дерево узлов, отражающее вложенность тегов. Параллельно из CSS строится CSSOM, такое же дерево, но со стилями и их наследованием. Дальше браузер объединяет два дерева в render tree, куда попадают только видимые узлы (элемент с `display: none` в дерево отрисовки не входит). Затем идёт layout, он же reflow: браузер вычисляет геометрию, то есть координаты и размеры каждого узла. И только потом paint закрашивает пиксели, а compositing склеивает слои в итоговую картинку.

Ключевая мысль конвейера видна на схеме: render tree требует и DOM, и CSSOM одновременно. Пока не готово хотя бы одно из двух деревьев, отрисовки не будет. Отсюда растут все правила про порядок и блокировку.

## Почему CSS задерживает первый кадр

Возьмём страницу без единого скрипта:

```html
<head>
  <link rel="stylesheet" href="/main.css">
</head>
<body>
  <h1>Каталог</h1>
</body>
```

Браузер быстро построит DOM из этого HTML, но CSSOM он не получит, пока не скачает и не разберёт `main.css`. А без CSSOM нет render tree, значит нет и отрисовки. Поэтому внешний CSS называют render-blocking: браузер намеренно ждёт стили, чтобы не мигнуть пользователю неоформленной страницей, а потом резко её перекрасить. Вывод практический: критичный CSS должен быть маленьким и приходить рано, иначе он держит весь первый кадр.

Не всякий CSS блокирует отрисовку. Стили с медиазапросом, который сейчас не выполняется, браузер качает с низким приоритетом и отрисовку ими не задерживает:

```html
<link rel="stylesheet" href="/print.css" media="print">
```

Такой файл нужен только при печати, поэтому ждать его для экрана незачем.

## Скрипты встают поперёк разбора

С JavaScript ситуация жёстче. Встретив обычный синхронный `<script src>`, парсер останавливает построение DOM: скрипт может дописать разметку или изменить уже разобранные узлы, поэтому браузер ждёт, пока файл скачается и выполнится. Разбор возобновляется только после этого.

Два атрибута снимают эту блокировку. С `async` скрипт качается в фоне и выполняется сразу по готовности, вклиниваясь в разбор в случайный момент, без гарантии порядка: это для независимых вещей вроде аналитики. С `defer` скрипт тоже качается параллельно, но выполнение откладывается до готового DOM, и порядок тегов сохраняется. Для скриптов, которые зависят от разметки или друг от друга, правильный выбор почти всегда `defer`:

```html
<script src="/app.js" defer></script>
```

## Что вас удивит

CSS умеет блокировать и JavaScript. Синхронный скрипт вправе спросить вычисленные стили элемента через `getComputedStyle`, поэтому браузер не запустит его, пока не построит CSSOM. Если такой скрипт стоит сразу за тяжёлым `<link>`, он молча ждёт весь этот CSS, а вместе с ним ждёт и всё, что идёт следом в разборе. Крохотный скрипт при этом выглядит невиновным, хотя тормозит именно порядок: большой CSS впереди него.

Ещё одна неожиданность касается инлайновых стилей. Кажется, что вынести критичный CSS прямо в `<style>` внутри HTML быстрее, ведь не нужен отдельный запрос. Часто так и есть, и этот приём называют inline critical CSS. Но перелив всех стилей в HTML раздувает документ, который тоже render-blocking, поэтому инлайнят обычно только тот минимум, что нужен первому экрану, а остальное подгружают отдельно.

## Частые ошибки

Тяжёлый CSS в `<head>` без разбивки. Один большой файл стилей держит первый кадр целиком. Помогает вынести критичное для первого экрана вперёд, а необязательное подключить с медиазапросом или подгрузить позже.

Синхронные скрипты в `<head>`. Классическая причина белого экрана: парсер упирается в `<script>` и ждёт. Лечится атрибутом `defer` или переносом скриптов в конец `<body>`.

Вера в то, что картинки задерживают отрисовку. Изображения не входят в render-blocking: браузер построит макет и дорисует картинку, когда она придёт. Задерживают первый кадр CSS и синхронные скрипты, а не `<img>`.

Отрисовка «на глаз» вместо измерения. Кажется, что всё быстро на вашем ноутбуке. Реальный критический путь виден во вкладке Performance и в водопаде загрузки, а не по ощущению.

<Detail summary="Почему разработчики иногда мирятся с FOUC">

Если увести весь CSS из render-blocking (например, грузить его асинхронно), браузер отрисует страницу голым HTML, а потом резко применит стили. Это мигание называют FOUC, flash of unstyled content. Обычно с ним борются, оставляя критичный CSS блокирующим, а некритичный подгружая асинхронно. Но иногда, ради очень раннего показа текста, идут на управляемый компромисс: пользователь видит читаемый, пусть и неоформленный контент на доли секунды раньше. Это осознанный размен между «раньше, но некрасиво» и «позже, но сразу оформлено».

</Detail>
