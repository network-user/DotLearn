# CSS variables: declaration, var(), and theming

## In short

Custom properties are declared as `--token-name: value` and read with `var(--token-name)`. They inherit through the DOM and exist at runtime, unlike preprocessor variables.

## How to answer in an interview

- Syntax: `--color-accent: #3b82f6;` inside a selector, often `:root` for global tokens.
- Usage: `color: var(--color-accent);` with fallback `var(--x, #000)`.
- Themes: override variables on `[data-theme="dark"]` or `.dark` instead of duplicating every rule.
- Variables can animate and feed `calc()` / `clamp()`.
- I honestly add they cannot name selectors or media queries; breakpoints stay plain `@media`.

## How it works

Design tokens at the root:

```css
:root {
  --space-md: 1rem;
  --font-body: system-ui, sans-serif;
  --surface: #ffffff;
  --text: #0f172a;
}

[data-theme="dark"] {
  --surface: #0f172a;
  --text: #e2e8f0;
}

.card {
  font-family: var(--font-body);
  padding: var(--space-md);
  background: var(--surface);
  color: var(--text);
  border-radius: var(--radius, 12px);
}
```

`var(--radius, 12px)` applies a fallback when `--radius` is unset.

Variables **inherit**: children see overrides on ancestors. For local scope:

```css
.metric {
  --bar-color: #22c55e;
}

.metric__bar {
  background: var(--bar-color);
}
```

JS can theme at runtime: `document.documentElement.style.setProperty('--accent', userColor)`.

## Pitfalls

**Confusion with Sass `$vars`.** `$primary` resolves at build time; `--primary` is live in the browser and DevTools.

**Unitless values in calc.** `calc(var(--gap) + 4px)` needs `--gap` to include a unit (`8px`, not `8`).

**Unwanted inheritance.** A missing token reset on a nested block keeps the parent color.

**Deep trees with inline custom props.** Thousands of unique per-element variables add style recalc cost.

**Invalid at computed value.** A bad `var()` cascades; fallback helps only when the property is missing, not when the value is invalid.
