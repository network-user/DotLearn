# BEM и альтернативы: SMACSS, OOCSS, ITCSS

## Суть

BEM делит интерфейс на блоки, элементы (`block__el`) и модификаторы (`block--mod`). Альтернативы (SMACSS, OOCSS, ITCSS) по-разному раскладывают base/layout/module/state/theme, но общая цель одна: предсказуемые имена и изоляция стилей.

## Что ответить на собесе

- BEM: `card`, `card__title`, `card--featured`; без каскада `.card .title`.
- SMACSS: категории Base, Layout, Module, State, Theme; состояния через `.is-active`.
- OOCSS: отделение структуры (width, grid) от «скина» (цвета); композиция утилитарных классов.
- ITCSS: слои от settings/tools до utilities в порядке возрастания специфичности.
- Честно добавляю: в React с CSS Modules BEM-нотация часто избыточна, хеш класса уже изолирует стиль.

## Как это работает

Пример меню в BEM:

```html
<nav class="site-nav">
  <ul class="site-nav__list">
    <li class="site-nav__item site-nav__item--current">
      <a class="site-nav__link" href="/">Главная</a>
    </li>
    <li class="site-nav__item">
      <a class="site-nav__link" href="/topics">Темы</a>
    </li>
  </ul>
</nav>
```

Модификатор `--current` не создаёт новый блок; он меняет состояние элемента.

SMACSS state:

```css
.tabs__panel { display: none; }
.is-open { display: block; }
```

OOCSS разделяет «скелет» и «оболочку»:

```css
.media { display: flex; gap: 1rem; }
.media--stacked { flex-direction: column; }
.skin-primary { background: var(--brand); color: #fff; }
```

ITCSS импортирует слои сверху вниз: variables → mixins → generic → objects → components → utilities. Utilities побеждают в каскаде без `!important`, если слой последний.

## Подводные камни

**Глубокие цепочки BEM.** `block__el__subel` нарушает методологию; второй уровень оформляют отдельным блоком.

**Глобальные is-* без scope.** `.is-hidden` на всём проекте конфликтует между модулями.

**OOCSS-утилиты без дисциплины.** Десятки атомарных классов в JSX ухудшают читаемость.

**ITCSS без layers в CSS.** Порядок `@import` критичен; tree-shaking и бандл могут перемешать файлы.

**Догма вместо контекста.** В design-system на токенах и Modules жёсткий BEM может дублировать то, что решает сборщик.
