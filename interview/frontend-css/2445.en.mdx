# Which selector wins when styles conflict

## In short

The winner follows: importance (`!important`), then selector specificity, then declaration order. Inline beats a class, ID beats a single class, but loses to sheet `!important` in the right conditions.

## How to answer in an interview

- Compare weights (inline, id, class/pseudo/attr, type/pseudo-element) component-wise.
- Equal specificity goes to the later rule.
- Author `!important` beats normal author; user `!important` (accessibility) beats author.
- I do not fix conflicts with longer selectors; I simplify or use layers.
- Honestly: equal weights from different bundles depend on `<link>` order.

## How it works

```html
<p id="badge" class="tag warn">Status</p>
```

```css
.warn { color: #b45309; }
#badge { color: #1d4ed8; }
p { color: #64748b; }
```

`#badge` (0,1,0,1) wins over `.warn` (0,0,1,0).

```ts
const badge = document.querySelector<HTMLElement>("#badge");
badge?.style.setProperty("color", "#7c3aed");
```

Inline overrides the sheet until `!important`:

```css
.tag { color: #0f766e !important; }
```

@layer for controlled order without specificity races:

```css
@layer base, components;

@layer base {
  .tag { color: #64748b; }
}

@layer components {
  .tag { color: #0f172a; }
}
```

## Pitfalls

**Legacy specificity arms race.** Every fix adds `#id .class`.

**!important as default.** Theme toggles become impossible.

**:is() takes the max.** `:is(#a, .b)` weighs like `#a`.

**Third-party widget CSS.** Loads last and breaks brand colors.

**Source order vs specificity confusion.** Juniors add `!important` when order would suffice.
