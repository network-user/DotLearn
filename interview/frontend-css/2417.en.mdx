# CSS length units: px, rem, %, vw, and when to use each

## In short

CSS splits units into absolute (px, cm, pt) and relative (%, em, rem, vw, vh, ch, fr). Relative units tie to parent, root, viewport, or flex/grid context and keep UI scalable.

## How to answer in an interview

- `px` in the web is a CSS pixel (1/96 inch logically); good for borders and shadows.
- `rem` from `font-size` on `html`; I prefer it for typography and design-system spacing.
- `em` from the element's own font size; handy for components that scale with local text.
- `%` from the parent's corresponding axis (height needs a defined parent height).
- I honestly add that `vw`/`vh` on mobile include the address bar; fullscreen blocks often use `dvh`/`svh`.

## How it works

Baseline scale:

```css
html {
  font-size: 100%;
}

body {
  font-size: 1rem;
  line-height: 1.5;
}

h1 {
  font-size: clamp(1.75rem, 2vw + 1rem, 2.5rem);
}

.sidebar {
  width: min(320px, 90vw);
  padding: 1.25rem;
}

.hero {
  min-height: 100dvh;
}
```

`clamp` blends rem and vw: the heading grows with viewport within bounds. `min()` keeps the sidebar on screen.

**Viewport units:** `1vw` is 1% of viewport width; `vmin`/`vmax` use the smaller/larger side. **Dynamic viewport** (`dvh`, `svh`, `lvh`) accounts for mobile browser chrome.

**Grid `fr`** splits free track space, not a parent percentage:

```css
.layout {
  display: grid;
  grid-template-columns: 240px 1fr 20rem;
  gap: 1rem;
}
```

**`ch`** is the width of "0"; useful for monospace inputs sized in characters.

## Pitfalls

**em compounding.** Three nested `font-size: 1.2em` levels yield 1.728× root, not 1.2×.

**% height without parent height.** `height: 50%` often computes to auto if the parent height is auto.

**100vw horizontal scroll.** `100vw` includes scrollbar width and can cause overflow.

**px and accessibility.** Browser zoom scales px; `rem` still syncs better with root font preferences via `html`.

**fr outside grid.** `fr` only works on grid tracks, not plain `width`.
