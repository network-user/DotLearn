# CSS Modules: scoped classes, pros, and limits

## In short

CSS Modules compile local class names to unique identifiers at build time. Importing `styles.title` in TS/React yields strings like `Hero_title_x3k9`, removing global name clashes without runtime CSS-in-JS.

## How to answer in an interview

- Use `*.module.css`; only exported classes are visible to JS.
- Globals via `:global(.reset)` or a separate `global.css`.
- Composition: `composes: btn from './Button.module.css'`.
- TypeScript: `*.module.css.d.ts` generation for autocomplete.
- I honestly add that dynamic classes need known keys in template strings; arbitrary API-driven names will not map.

## How it works

`TopicCard.module.css`:

```css
.root {
  display: grid;
  gap: 0.75rem;
  padding: 1rem;
  border-radius: 12px;
}

.title {
  font-size: 1.125rem;
  font-weight: 600;
}

:global(.dark) .root {
  background: #0f172a;
}
```

`TopicCard.tsx`:

```ts
import styles from "./TopicCard.module.css";

export function TopicCard({ title }: { title: string }) {
  return (
    <article className={styles.root}>
      <h3 className={styles.title}>{title}</h3>
    </article>
  );
}
```

The bundler (Vite CSS modules) renames `.root` uniquely. Unused-class tree-shaking depends on the plugin.

Conditional modifier:

```ts
const variant = done ? styles.done : styles.pending;
return <span className={`${styles.badge} ${variant}`}>...</span>;
```

## Pitfalls

**Bare tag selectors.** `div { }` inside a module scopes oddly; style through classes.

**:global without discipline.** Leaked globals bring name conflicts back.

**Third-party libraries.** Bootstrap classes are not module-scoped; wrap or use a facade.

**composes and chunk order.** Extracted CSS order can shift cascade between builds.

**SSR + HMR.** Server HTML and client class hashes can diverge during hot reload in dev.
