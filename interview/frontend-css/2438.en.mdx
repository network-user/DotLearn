# How to optimize CSS for production

## In short

CSS optimization is smaller transfer (minify, split, purge) and cheaper runtime (fewer heavy selectors, modern layout, critical path). Structure (BEM, tokens, layers) cuts regressions as the team grows.

## How to answer in an interview

- Delivery: one critical bundle, lazy/route CSS elsewhere, brotli/gzip, HTTP/2 multiplex.
- Purge/content scan drops dead Tailwind/legacy classes; I verify dynamic class names.
- PostCSS/autoprefixer only needed prefixes; cssnano on build.
- Architecture: Grid/Flex over float hacks; design tokens via custom properties.
- Honestly: aggressive purge breaks CMS content and Storybook stories if globs are incomplete.

## How it works

Layered imports:

```css
@layer reset, tokens, components, utilities;

@layer tokens {
  :root {
    --space-3: 0.75rem;
    --radius-md: 8px;
  }
}

@layer components {
  .btn {
    padding: var(--space-3) calc(var(--space-3) * 1.5);
    border-radius: var(--radius-md);
  }
}
```

Build pipeline (conceptual):

```ts
const pipeline = [
  "postcss-import",
  "tailwindcss",
  "autoprefixer",
  "cssnano",
] as const;
```

Route-level split in an SPA:

```ts
import("./AnalyticsPanel.module.css");
```

Minified output:

```css
.btn{padding:.75rem 1.125rem;border-radius:8px}
```

## Pitfalls

**Purge without safelist.** Classes from `clsx(`hidden ${flag}`)` vanish in prod.

**@import in runtime CSS.** Blocks parallel loading.

**Utility + !important inflation.** Fast fix, expensive cascade later.

**Duplicate reset per chunk.** Multiple entries ship the same normalize.

**Optimization without metrics.** Lighthouse/CSS coverage shows real dead weight.
