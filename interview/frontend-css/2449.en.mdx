# Why semantic CSS class names matter

## In short

A semantic class describes role (`card__title`, `btn-primary`), not looks (`red-text`). Code reads easier, themes change without renames, and components reuse across pages.

## How to answer in an interview

- Names reflect intent: block/element/modifier, not color/size in the label.
- Design refactors change CSS; HTML class names stay.
- Team consistency: shared glossary (`btn`, `stack`, `surface`).
- Pairs with BEM/ITCSS; utilities (`mt-4`) for spacing, not domain logic.
- Honestly: "semantic" is not HTML semantic elements; it is a CSS naming convention.

## How it works

```html
<article class="product-card">
  <h2 class="product-card__name">CSS course</h2>
  <p class="product-card__meta">12 lessons</p>
  <button class="btn btn--primary" type="button">Start</button>
</article>
```

```css
.product-card {
  display: grid;
  gap: 0.75rem;
  padding: 1rem;
  border-radius: 12px;
  background: var(--surface-raised);
}

.product-card__name {
  font-size: 1.125rem;
  font-weight: 650;
}

.btn--primary {
  background: var(--accent);
  color: var(--accent-ink);
}
```

```ts
const cards = document.querySelectorAll<HTMLElement>(".product-card");
cards.forEach((card) => {
  card.dataset.enhanced = "true";
});
```

Anti-pattern `.blue-16px-bold` breaks when the brand palette changes.

## Pitfalls

**Too abstract.** `.wrapper2` is not semantic even without color.

**BEM plus utility duplication.** `product-card product-card--shadow p-4 p-6` fights spacing.

**i18n string length.** Semantic layout must survive long German copy.

**Global `.title`.** Collisions between feature modules.

**Storybook happy path only.** Semantic states (`is-error`) missing from docs.
