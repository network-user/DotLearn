# What CSS Modules are and why components use them

## In short

CSS Modules are `.module.css` files whose class names become unique at build time and import into JS as an object. Styles stay component-local and avoid global collisions.

## How to answer in an interview

- The bundler hashes class names; in JSX I use `styles.root`, not the string `"root"`.
- Scoping reduces collisions in large teams and monorepos with legacy CSS.
- Global rules go in `:global` or a separate `global.css`.
- Works with React/Vue/Svelte when a loader exists; Vite ships support.
- Honestly: not a runtime framework but a build convention; without a bundler there are no modules.

## How it works

```css
.card {
  display: grid;
  gap: 0.75rem;
  padding: 1rem;
  border-radius: 12px;
  background: #fff;
  box-shadow: 0 1px 2px rgb(15 23 42 / 8%);
}

.title {
  font-size: 1.125rem;
  font-weight: 600;
}
```

```ts
import styles from "./ProductCard.module.css";

interface ProductCardProps {
  name: string;
  price: string;
}

export function ProductCard({ name, price }: ProductCardProps) {
  return (
    <article className={styles.card}>
      <h2 className={styles.title}>{name}</h2>
      <p>{price}</p>
    </article>
  );
}
```

After build a class might become `ProductCard_card_x7k2f`, but the code API stays the same.

TypeScript typing (Vite plugin / d.ts):

```ts
declare module "*.module.css" {
  const classes: Record<string, string>;
  export default classes;
}
```

## Pitfalls

**String class names from props.** `className="card"` will not match module hashes.

**Duplicate names across files are safe, not within one file.** Two `.title` rules in one module.css merge.

**composes and dead code.** Unused classes still ship in that module's CSS chunk.

**CSS variables vs modules.** Put tokens on `:root`; use modules for component layout.

**Snapshot tests.** Hashes change when the file is renamed; class snapshots are brittle.
