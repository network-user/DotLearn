# CSS-in-JS: benefits, drawbacks, and production pain

## In short

CSS-in-JS generates styles from JavaScript (styled-components, Emotion, Stitches). Dynamic colocated styles fit React, but runtime injection hurts SSR, bundle size, DevTools, and cascade predictability.

## How to answer in an interview

- Pros: prop-driven themes, scoped classes without manual BEM, conditional styles beside JSX.
- Cons: runtime cost, SSR FOUC/hydration issues, opaque generated classes, harder static extraction.
- Zero-runtime options (Linaria, Vanilla Extract, Panda) emit CSS at build time.
- Design tokens often need only CSS variables plus Modules.
- I honestly add that new React 19 apps often pick Modules/Tailwind over runtime CSS-in-JS for RSC and perf budgets.

## How it works

Runtime styled component:

```ts
import styled from "styled-components";

type BtnProps = { $primary?: boolean };

export const ActionBtn = styled.button<BtnProps>`
  padding: 0.5rem 1rem;
  border-radius: 8px;
  background: ${({ $primary }) => ($primary ? "#2563eb" : "#e2e8f0")};
  color: ${({ $primary }) => ($primary ? "#fff" : "#0f172a")};
`;
```

Each unique prop combination may inject a new `<style>` tag in the document head.

SSR needs `ServerStyleSheet` / `StyleRegistry` or HTML ships unstyled until hydration.

Compile-time alternative:

```ts
import { css } from "@linaria/core";

export const btn = css`
  padding: 0.5rem 1rem;
`;
```

The class lives in a static CSS chunk; JS does not emit rules at runtime.

## Pitfalls

**Nested selector specificity.** `& svg { }` in styled components can beat global resets unexpectedly.

**Duplicate keyframes.** Each mount may register the same animation name again.

**RSC boundaries.** Client-only styled in server components breaks architecture; leaf nodes need `"use client"`.

**Production debugging.** Hashed classes without source maps; see the dedicated maps question.

**Migrating to static CSS.** Hundreds of `styled()` calls need a visual regression suite to extract safely.
