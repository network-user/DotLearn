# BEM and alternatives: SMACSS, OOCSS, ITCSS

## In short

BEM splits UI into blocks, elements (`block__el`), and modifiers (`block--mod`). Alternatives (SMACSS, OOCSS, ITCSS) arrange base/layout/module/state/theme differently, but the goal is predictable names and style isolation.

## How to answer in an interview

- BEM: `card`, `card__title`, `card--featured`; avoid `.card .title` cascades.
- SMACSS: Base, Layout, Module, State, Theme categories; states like `.is-active`.
- OOCSS: separate structure (width, grid) from skin (colors); compose utility classes.
- ITCSS: layers from settings/tools to utilities in rising specificity order.
- I honestly add that with React CSS Modules, BEM naming is often redundant because the hash already scopes styles.

## How it works

BEM navigation example:

```html
<nav class="site-nav">
  <ul class="site-nav__list">
    <li class="site-nav__item site-nav__item--current">
      <a class="site-nav__link" href="/">Home</a>
    </li>
    <li class="site-nav__item">
      <a class="site-nav__link" href="/topics">Topics</a>
    </li>
  </ul>
</nav>
```

The `--current` modifier changes element state; it does not create a new block.

SMACSS state:

```css
.tabs__panel { display: none; }
.is-open { display: block; }
```

OOCSS splits skeleton and skin:

```css
.media { display: flex; gap: 1rem; }
.media--stacked { flex-direction: column; }
.skin-primary { background: var(--brand); color: #fff; }
```

ITCSS imports layers top to bottom: variables → mixins → generic → objects → components → utilities. Utilities win in the cascade without `!important` when the layer is last.

## Pitfalls

**Deep BEM chains.** `block__el__subel` breaks the rules; model nested UI as separate blocks.

**Global is-* classes.** `.is-hidden` project-wide collides across modules.

**OOCSS utility sprawl.** Many atomic classes in JSX hurt readability.

**ITCSS without layer discipline.** `@import` order matters; bundlers can reorder files.

**Methodology dogma.** Tokenized design systems with Modules may not need strict BEM prefixes.
