# Как оптимизировать CSS в продакшене

## Суть

Оптимизация CSS это и меньший transfer (minify, split, purge), и более дешёвый runtime (меньше специфичных селекторов, современный layout, critical path). Структура (BEM, tokens, layers) снижает регрессии при росте команды.

## Что ответить на собесе

- Delivery: один critical bundle, остальное lazy/route CSS, brotli/gzip, HTTP/2 multiplex.
- Purge/content scan удаляет мёртвые классы Tailwind/legacy; проверяю dynamic class names.
- PostCSS/autoprefixer только нужные prefixes; cssnano на build.
- Архитектура: Grid/Flex вместо float-хаков; design tokens через custom properties.
- Честно: агрессивный purge ломает CMS-контент и Storybook stories, если glob неполный.

## Как это работает

Layered imports:

```css
@layer reset, tokens, components, utilities;

@layer tokens {
  :root {
    --space-3: 0.75rem;
    --radius-md: 8px;
  }
}

@layer components {
  .btn {
    padding: var(--space-3) calc(var(--space-3) * 1.5);
    border-radius: var(--radius-md);
  }
}
```

Build pipeline (conceptual):

```ts
const pipeline = [
  "postcss-import",
  "tailwindcss",
  "autoprefixer",
  "cssnano",
] as const;
```

Route-level split в SPA:

```ts
import("./AnalyticsPanel.module.css");
```

Minify результат:

```css
.btn{padding:.75rem 1.125rem;border-radius:8px}
```

## Подводные камни

**Purge без safelist.** Классы из `clsx(`hidden ${flag}`)` исчезают из prod.

**@import в runtime CSS.** Блокирует параллельную загрузку.

**Utility + !important inflation.** Быстрый фикс, дорогой каскад потом.

**Дубли reset в каждом chunk.** Несколько entry тянут один и тот же normalize.

**Оптимизация без метрик.** Lighthouse/CSS coverage показывает реальный dead weight.
