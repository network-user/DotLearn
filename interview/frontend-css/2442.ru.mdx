# Как считается специфичность CSS-селекторов

## Суть

При конфликте правил браузер сравнивает специфичность селекторов, затем порядок в таблице стилей, затем importance (`!important`). Inline style и `!important` ломают обычную лестницу.

## Что ответить на собесе

- Вес `(inline, id, class/attr/pseudo-class, type/pseudo-element)`; сравнение по компонентам слева направо.
- `#nav .link` побеждает `.link` и `a`, но проигрывает inline `style=""`.
- `:where()` обнуляет вклад обёртки; `:is()` берёт самый тяжёлый аргумент.
- Для поддерживаемости держу классы одного уровня, избегаю `#id` в компонентах.
- Честно: `!important` в utilities (Tailwind) иногда нужен, но превращает каскад в лотерею.

## Как это работает

```html
<header id="site" class="banner">
  <h1 class="headline">Заголовок</h1>
</header>
```

```css
h1 {
  color: #dc2626;
}

.banner .headline {
  color: #15803d;
}

#site .headline {
  color: #1d4ed8;
}
```

Итог: `#site .headline` (0,1,1,1) сильнее `.banner .headline` (0,0,2,1).

```ts
const title = document.querySelector<HTMLElement>(".headline");
if (title) {
  title.style.color = "#7c3aed";
}
```

Inline (1,0,0,0) перекрывает stylesheet до `!important` в author sheet.

`:where()` для снижения веса:

```css
:where(#legacy) .btn {
  color: #64748b;
}
```

## Подводные камни

**Специфичность как API.** Команды «побеждают» друг друга длинными селекторами.

**Смешение ID из старых CMS.** Один `#content` блокирует redesign.

**:not() считает аргумент.** `:not(.a.b)` вклад как два класса.

**!important в keyframes.** Не наследуется, но локально перебивает cascade.

**DevTools не заменяет расчёт.** Computed panel показывает победителя, но не учит проектировать иерархию.
