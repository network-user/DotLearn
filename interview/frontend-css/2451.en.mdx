# Difference between :focus-visible and :focus-within

## In short

`:focus-visible` styles an element when the UA decides focus should be shown (usually keyboard), hiding the ring after a mouse click. `:focus-within` highlights an ancestor while any descendant is focused.

## How to answer in an interview

- `:focus` always applies on focus; `:focus-visible` drops the extra outline after mouse click.
- `:focus-within` on `.field` when an inner input is active, useful for form groups.
- I never remove outline globally; I replace it with a `:focus-visible` style.
- I pair `:focus-within` for card borders, not only input rings.
- Honestly: browsers without `:focus-visible` need a polyfill or `:focus` fallback.

## How it works

```css
.btn {
  border: none;
  border-radius: 8px;
  padding: 0.5rem 1rem;
  background: #2563eb;
  color: #fff;
}

.btn:focus {
  outline: none;
}

.btn:focus-visible {
  outline: 2px solid #fbbf24;
  outline-offset: 2px;
}

.field:focus-within {
  border-color: #2563eb;
  box-shadow: 0 0 0 3px rgb(37 99 235 / 25%);
}
```

```html
<label class="field">
  Email
  <input type="email" autocomplete="email" />
</label>
```

```ts
document.addEventListener("keydown", (event) => {
  if (event.key === "Tab") {
    document.body.classList.add("kbd-nav");
  }
});
```

```css
body:not(.kbd-nav) .btn:focus:not(:focus-visible) {
  outline: none;
}
```

## Pitfalls

**outline:none without replacement.** Breaks keyboard a11y and WCAG.

**focus-within on huge containers.** Whole card flashes when tabbing 20 controls.

**Custom element shadow DOM.** Focus inside shadow may not bubble for light DOM :focus-within.

**Programmatic focus.** `focus()` may not match :focus-visible in some UAs.

**Confusion with :has().** `:has(:focus)` ≠ `:focus-within` for support and semantics.
