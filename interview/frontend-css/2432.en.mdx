# What the browser does when it hits CSS during rendering

## In short

The HTML parser builds the DOM; on blocking CSS it fetches the file, parses rules, builds the CSSOM, then the render tree. Synchronous CSS in `<head>` delays first paint.

## How to answer in an interview

- `<link rel="stylesheet">` in head blocks rendering until load and parse (render-blocking by default).
- Chain: fetch → parse CSS → CSSOM → match with DOM → layout/paint.
- `@import` inside CSS adds sequential round-trips and hurts the critical path.
- I inline or preload critical above-the-fold CSS; defer the rest via media tricks or splitting.
- Honestly: `media="print" onload` speeds FCP but can FOUC without a fallback.

## How it works

Minimal page:

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8" />
    <link rel="stylesheet" href="/assets/base.css" />
    <title>Demo</title>
  </head>
  <body>
    <h1 class="hero">Hello</h1>
  </body>
</html>
```

```css
.hero {
  color: #1e293b;
  font-size: clamp(1.5rem, 2vw + 1rem, 2.25rem);
  line-height: 1.2;
}
```

```ts
const sheet = [...document.styleSheets].find(
  (s) => s.href?.includes("base.css")
);

console.log(sheet?.cssRules.length ?? 0);
```

Preload plus a normal link reduces discovery delay:

```html
<link rel="preload" href="/assets/base.css" as="style" />
<link rel="stylesheet" href="/assets/base.css" />
```

Pipeline: HTML tokenizer → DOM, CSS tokenizer → CSSOM in parallel where possible, merge into render tree, then layout and paint; JS with `document.write` or sync `<script>` without defer can block parsing again.

## Pitfalls

**CSS at end of body.** Users see an unstyled flash even though HTML parsing started earlier.

**One huge file.** All first paint waits even if only 2 KB of rules are critical.

**@import chains.** Each level waits on the previous file.

**Wrong MIME.** `text/plain` may ignore the stylesheet.

**Sync JS before CSS.** Script without defer/async pushes stylesheet discovery down the document.
