# How CSS selector specificity is calculated

## In short

When rules conflict, the browser compares selector specificity, then source order, then importance (`!important`). Inline styles and `!important` break the normal ladder.

## How to answer in an interview

- Weight `(inline, id, class/attr/pseudo-class, type/pseudo-element)`; compare component-wise left to right.
- `#nav .link` beats `.link` and `a`, but loses to inline `style=""`.
- `:where()` zeroes the wrapper contribution; `:is()` takes the heaviest argument.
- For maintainability I keep single-level classes and avoid `#id` in components.
- Honestly: `!important` in utilities (Tailwind) is sometimes needed but turns the cascade into a lottery.

## How it works

```html
<header id="site" class="banner">
  <h1 class="headline">Title</h1>
</header>
```

```css
h1 {
  color: #dc2626;
}

.banner .headline {
  color: #15803d;
}

#site .headline {
  color: #1d4ed8;
}
```

Result: `#site .headline` (0,1,1,1) wins over `.banner .headline` (0,0,2,1).

```ts
const title = document.querySelector<HTMLElement>(".headline");
if (title) {
  title.style.color = "#7c3aed";
}
```

Inline (1,0,0,0) beats author styles until `!important` in a sheet.

`:where()` to lower weight:

```css
:where(#legacy) .btn {
  color: #64748b;
}
```

## Pitfalls

**Specificity as API.** Teams defeat each other with longer selectors.

**Legacy CMS IDs.** One `#content` blocks redesign.

**:not() counts its argument.** `:not(.a.b)` counts as two classes.

**!important in keyframes.** Not inherited but locally overrides cascade.

**DevTools is not design.** Computed shows the winner, not how to architect layers.
