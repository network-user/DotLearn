# What critical CSS is and how it speeds first paint

## In short

Critical CSS is the minimal rule set for above-the-fold without FOUC. Inline it in `<head>`, load the full stylesheet asynchronously so first paint is not blocked.

## How to answer in an interview

- I extract hero, typography, and layout shell styles; defer the rest.
- Inline `<style>` in head + preload/async trick on the full CSS link.
- Tools: critical, penthouse, vite-plugin-critical; validate on real URLs.
- Balance: inline >14KB eats TTFB budget on slow networks.
- Honestly: on SPAs route-level CSS often replaces hand-tuned critical per page.

## How it works

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8" />
    <style>
      body { margin: 0; font-family: system-ui, sans-serif; }
      .hero { padding: 2rem 1rem; max-width: 48rem; margin-inline: auto; }
      .hero h1 { font-size: clamp(1.75rem, 4vw, 2.5rem); line-height: 1.15; }
    </style>
    <link rel="preload" href="/assets/app.css" as="style" />
    <link rel="stylesheet" href="/assets/app.css" media="print" onload="this.media='all'" />
    <noscript><link rel="stylesheet" href="/assets/app.css" /></noscript>
  </head>
  <body>
    <header class="hero"><h1>Fast first screen</h1></header>
  </body>
</html>
```

```ts
const deferred = document.querySelector<HTMLLinkElement>(
  'link[href="/assets/app.css"]'
);

deferred?.addEventListener("load", () => {
  document.documentElement.dataset.styles = "ready";
});
```

Extract pipeline (build):

```ts
const criticalTasks = ["parse-html", "match-selectors", "inline-head"] as const;
```

## Pitfalls

**Critical without route split.** One critical blob for every SPA route bloats HTML.

**FOUC with the onload trick.** Users see font/color shift.

**@font-face inside critical.** Blocks render until the font loads.

**Duplicate rules.** Inline + external with the same selectors complicate cascade.

**Auto extract without safelist.** Dynamic classes drop out of critical output.
