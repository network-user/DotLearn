# Каскад CSS: origin, специфичность и порядок правил

## Суть

Каскад выбирает победившее объявление, когда на один элемент действуют несколько правил. Современный алгоритм учитывает слой (@layer), важность (`!important`), специфичность селектора и порядок в исходнике; inline-стили и `!important` в user styles перебивают авторские в особых случаях.

## Что ответить на собесе

- Сначала сравниваю origin: user agent, user, author; `@layer` задаёт приоритет слоёв до специфичности.
- Специфичность: inline > id > class/attr/pseudo-class > tag; универсальный `*` не добавляет веса.
- При равной специфичности побеждает правило, объявленное позже в том же слое.
- `!important` инвертирует порядок origin внутри уровня, но не отменяет логику слоёв полностью.
- Честно добавляю: злоупотребление `!important` и id-селекторами лечит симптом, а не архитектуру стилей.

## Как это работает

Пример конфликта:

```css
@layer base, components;

@layer base {
  p { color: #334155; }
}

@layer components {
  .note { color: #0ea5e9; }
}

#lead { color: #dc2626; }

p.note#lead {
  color: #16a34a !important;
}
```

```html
<p class="note" id="lead">Абзац</p>
```

Без `!important` победит `#lead` (0,1,0,1) над `.note` (0,0,1,1). С `p.note#lead !important` цвет станет зелёным, пока нет более специфичного `!important` в том же cascade layer.

**Inline** `style="color: …"` даёт специфичность (1,0,0,0) для этого свойства. Наследуемые свойства (`color` на родителе) проигрывают прямому назначению на элементе.

Порядок загрузки `<link>` и `<style>` важен только при прочих равных. `@import` внутри файла откладывает применение импортированных правил.

## Подводные камни

**Специфичность как гонка вооружений.** `#header .nav .link` трудно переопределить без ещё более длинного селектора.

**!important в утилитах.** Tailwind `!` модификатор спасает в крайнем случае, но ломает предсказуемость каскада.

**Псевдоклассы и :where().** `:where(.a)` обнуляет вклад класса в специфичность; удобно для библиотек.

**Наследование vs каскад.** `font-size` на `html` наследуется; конфликт двух классов на одном элементе решает каскад, не наследование.

**Стили Shadow DOM.** Внутри shadow root свой отдельный дерево правил; глобальный CSS не проникает без `::part` или переменных.
