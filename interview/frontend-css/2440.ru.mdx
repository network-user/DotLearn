# Плюсы и минусы вложенных селекторов

## Суть

Селектор вида `.card .title p` привязывает стиль к иерархии DOM. Это повышает точность, но растит специфичность, стоимость match и хрупкость при рефакторинге разметки.

## Что ответить на собесе

- Плюс: стиль не «утекает» на такие же теги вне блока.
- Минус: высокая специфичность мешает переопределению utility-классами.
- В JS `querySelector('.shell .item')` повторяет ту же хрупкость.
- Предпочитаю плоские BEM-классы или `:where()` для снижения веса.
- Честно: nesting в native CSS удобен, но глубина >2 часто признак лишней связности.

## Как это работает

```css
:where(.article) :where(.lead) {
  font-size: 1.125rem;
  line-height: 1.55;
  color: #334155;
}

.article .note strong {
  font-weight: 650;
}
```

`:where()` обнуляет специфичность обёртки.

```ts
const leads = document.querySelectorAll<HTMLElement>(
  ".article .lead"
);

leads.forEach((node) => {
  node.dataset.enhanced = "true";
});
```

Лучше data-hook без глубины:

```ts
const items = document.querySelectorAll<HTMLElement>(
  "[data-article-lead]"
);
```

```html
<p class="lead" data-article-lead>Вступление</p>
```

## Подводные камни

**Лишний descendant combinator.** `.list li span a` ломается при добавлении wrapper.

**Смешение с ID.** `#page .content p` почти невозможно перебить классом.

**Performance myth vs reality.** На огромных DOM дорогие селекторы ощутимы, но чаще проблема в специфичности.

**Sass nesting copy-paste.** Автонesting раздувает CSS без осознанной иерархии.

**Тесты по innerHTML.** Селектор завязан на структуру, не на контракт компонента.
