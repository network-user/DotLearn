# Pros and cons of nested selectors

## In short

A selector like `.card .title p` ties styles to DOM hierarchy. That adds precision but raises specificity, matching cost, and fragility when markup refactors.

## How to answer in an interview

- Pro: styles do not leak to the same tags outside the block.
- Con: high specificity blocks utility overrides.
- In JS, `querySelector('.shell .item')` repeats the same fragility.
- I prefer flat BEM classes or `:where()` to lower weight.
- Honestly: native nesting is handy, but depth >2 often signals unnecessary coupling.

## How it works

```css
:where(.article) :where(.lead) {
  font-size: 1.125rem;
  line-height: 1.55;
  color: #334155;
}

.article .note strong {
  font-weight: 650;
}
```

`:where()` zeroes wrapper specificity.

```ts
const leads = document.querySelectorAll<HTMLElement>(
  ".article .lead"
);

leads.forEach((node) => {
  node.dataset.enhanced = "true";
});
```

Better: shallow data hooks:

```ts
const items = document.querySelectorAll<HTMLElement>(
  "[data-article-lead]"
);
```

```html
<p class="lead" data-article-lead>Intro</p>
```

## Pitfalls

**Extra descendant combinators.** `.list li span a` breaks when a wrapper appears.

**Mixed with IDs.** `#page .content p` is nearly impossible to override with a class.

**Performance myth vs reality.** Costly selectors hurt huge DOMs, but specificity pain is more common.

**Sass nesting copy-paste.** Auto-nesting inflates CSS without intentional hierarchy.

**Tests tied to innerHTML.** Selectors depend on structure, not a component contract.
