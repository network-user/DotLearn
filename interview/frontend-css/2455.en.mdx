# useEffect vs useLayoutEffect: when to use which

## In short

Both hooks run side effects after React commits to the DOM. `useEffect` runs after paint (async); `useLayoutEffect` runs synchronously before paint. Use layout effect when users must not see an intermediate frame after DOM measurements.

## How to answer in an interview

- useEffect: fetch, subscriptions, analytics, most side effects.
- useLayoutEffect: measure DOM, sync scroll position, prevent flicker after render.
- useLayoutEffect blocks paint; do not overuse on mobile.
- SSR: useLayoutEffect warns on server; defer to client or useEffect.
- Honestly: useEffect covers 95% of cases; layout only with proven flicker.

## How it works

```ts
import { useEffect, useLayoutEffect, useRef, useState } from "react";

function TooltipAnchor({ label }: { label: string }) {
  const anchorRef = useRef<HTMLButtonElement>(null);
  const [offset, setOffset] = useState(0);

  useLayoutEffect(() => {
    const node = anchorRef.current;
    if (!node) return;
    const { width } = node.getBoundingClientRect();
    setOffset(Math.max(0, width - 120));
  }, [label]);

  useEffect(() => {
    const node = anchorRef.current;
    if (!node) return;
    const log = () => console.log("clicked", label);
    node.addEventListener("click", log);
    return () => node.removeEventListener("click", log);
  }, [label]);

  return (
    <button ref={anchorRef} type="button" style={{ transform: `translateX(${offset}px)` }}>
      {label}
    </button>
  );
}
```

Timeline: render → commit DOM → useLayoutEffect → browser paint → useEffect.

## Pitfalls

**LayoutEffect for data fetching.** Blocks UI for no gain.

**Missing deps.** Stale measurements after prop changes.

**Double render from setState in layout.** Expected but can cascade.

**Strict Mode double invoke.** LayoutEffect cleanup must be idempotent.

**JS layout measure instead of CSS.** Sometimes CSS `anchor()` avoids JS.
