# What the CSSOM is and where it sits in rendering

## In short

The CSSOM (CSS Object Model) is the style tree the browser builds after parsing CSS. It merges with the DOM into the render tree for layout and paint; `element.style` and CSSOM APIs mutate styles from JS.

## How to answer in an interview

- Order: HTML → DOM, CSS → CSSOM, then render tree (visible nodes with styles).
- The CSSOM holds cascade, specificity, and computed values for selector matching.
- Inline `style` and JS updates hit the CSSOM and can invalidate layout.
- DevTools Styles shows the computed chain from CSSOM + DOM.
- Honestly: reading `offsetWidth` after a style write forces sync layout between JS and CSSOM.

## How it works

```html
<p id="status" class="badge">Online</p>
```

```css
.badge {
  color: #15803d;
  font-size: 0.875rem;
  font-weight: 600;
}
```

```ts
const status = document.querySelector<HTMLElement>("#status");
if (!status) throw new Error("missing status");

status.style.color = "#b45309";
status.style.fontSize = "1rem";

const computed = getComputedStyle(status);
console.log(computed.color, computed.fontSize);
```

`getComputedStyle` reads cascade output from the CSSOM, not inline alone.

## Pitfalls

**document.styleSheets and CORS.** Cross-origin sheet rules are unreadable from JS.

**Many style.* writes in a loop.** Each write can trigger recalc; prefer classList or CSS variables.

**DOM vs CSSOM confusion.** DOM is structure; color and size live in computed styles.

**insertRule at runtime.** Dynamic CSSOM grows; forgotten `<style>` tags in SPAs never leave.

**getComputedStyle in a hot path.** Costly on thousands of elements every frame.
