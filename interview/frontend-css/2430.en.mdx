# Why CSS cannot style a parent from a child

## In short

CSS flows top-down: a rule selects an ancestor and affects descendants, not the reverse. That keeps the cascade predictable, avoids cycles, and keeps style recalculation cheap.

## How to answer in an interview

- The cascade is one-way: "ancestor descendant", not "child parent".
- Reverse links would create specificity cycles and unstable rule order.
- For "highlight the card when inner field fails" I use `:has()`, a JS-driven class, or a data attribute on the parent.
- Flex/Grid cover many alignment cases without styling the parent from inside.
- Honestly: `:has()` is baseline now, but legacy corporate browsers may need a fallback.

## How it works

Classic JS workaround:

```ts
function syncParentHighlight(root: HTMLElement) {
  const invalid = root.querySelector<HTMLInputElement>(
    "input:user-invalid, input:invalid"
  );

  root.classList.toggle("has-field-error", Boolean(invalid));
}

document.querySelectorAll<HTMLElement>(".field-group").forEach((group) => {
  group.addEventListener("input", () => syncParentHighlight(group));
  syncParentHighlight(group);
});
```

```css
.field-group {
  border: 1px solid #cbd5e1;
  border-radius: 8px;
  padding: 0.75rem;
}

.field-group.has-field-error {
  border-color: #dc2626;
  background: color-mix(in srgb, #dc2626 6%, transparent);
}
```

With `:has()` the parent reacts without script:

```css
.field-group:has(input:user-invalid) {
  border-color: #dc2626;
}
```

Layout needs are often solved on the parent flex container:

```css
.stack {
  display: flex;
  flex-direction: column;
  gap: 0.5rem;
}

.stack > .grow {
  flex: 1 1 auto;
}
```

## Pitfalls

**:has() cost.** A broad selector on a large DOM can be pricier than a single class.

**Portals and modals.** A child outside the parent branch breaks `:has()` and inheritance.

**:has() specificity.** Easy to beat utility classes and get sticky styles.

**JS class without debounce.** No need to rescan the whole form on every input.

**Expecting a parent selector.** There is none; only `:has()` with limits or state on the ancestor.
