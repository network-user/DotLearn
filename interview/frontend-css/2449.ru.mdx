# Зачем семантические имена у CSS-классов

## Суть

Семантический класс описывает роль (`card__title`, `btn-primary`), а не внешность (`red-text`). Код проще читать, темы меняются без переименования, компоненты переиспользуются между страницами.

## Что ответить на собесе

- Имена отражают intent: block/element/modifier, не color/size в title.
- Рефакторинг дизайна меняет CSS, HTML class names остаются.
- Согласованность в команде: общий glossary (`btn`, `stack`, `surface`).
- Сочетается с BEM/ITCSS; utility (`mt-4`) для spacing, не для domain logic.
- Честно: «semantic» не путать с HTML semantic tags; это про naming convention CSS.

## Как это работает

```html
<article class="product-card">
  <h2 class="product-card__name">Курс CSS</h2>
  <p class="product-card__meta">12 уроков</p>
  <button class="btn btn--primary" type="button">Начать</button>
</article>
```

```css
.product-card {
  display: grid;
  gap: 0.75rem;
  padding: 1rem;
  border-radius: 12px;
  background: var(--surface-raised);
}

.product-card__name {
  font-size: 1.125rem;
  font-weight: 650;
}

.btn--primary {
  background: var(--accent);
  color: var(--accent-ink);
}
```

```ts
const cards = document.querySelectorAll<HTMLElement>(".product-card");
cards.forEach((card) => {
  card.dataset.enhanced = "true";
});
```

Антипример `.blue-16px-bold` ломается при смене brand palette.

## Подводные камни

**Слишком abstract.** `.wrapper2` не semantic, даже без цвета.

**Дубли BEM и utility.** `product-card product-card--shadow p-4 p-6` конфликт spacing.

**i18n длина текста.** Semantic layout должен переживать длинные немецкие слова.

**Global `.title`.** Коллизия между feature modules.

**Storybook только happy path.** Semantic states (`is-error`) забывают в docs.
