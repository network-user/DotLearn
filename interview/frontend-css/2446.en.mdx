# Which has higher priority: id or class in CSS

## In short

An ID selector outweighs a class: `(0,1,0,0)` vs `(0,0,1,0)`. On one element, `#id` wins over `.class` unless inline or `!important` says otherwise.

## How to answer in an interview

- On `<p id="x" class="y">`, `#x { color }` beats `.y { color }`.
- Multiple classes add up: `.a.b` (0,0,2,0) is still below one `#id`.
- In components I avoid styling via id; one id per page and heavy weight.
- To override utilities I use source order or layers, not a new id.
- Honestly: anchors and a11y need ids, not id-based CSS.

## How it works

```html
<button id="save" class="btn primary" type="button">Save</button>
```

```css
.btn { color: #334155; background: #e2e8f0; }
.primary { background: #2563eb; color: #fff; }
#save { outline-offset: 2px; }
```

Color comes from `.primary`, not `#save`, if `#save` has no color.

```ts
const save = document.querySelector<HTMLButtonElement>("#save");
save?.classList.toggle("primary");
```

Bad pattern:

```css
#save.primary { background: #16a34a; }
```

Better modifier:

```css
.btn.is-success { background: #16a34a; }
```

## Pitfalls

**IDs in CMS content.** Editors add ids that break global styles.

**React/Vue auto ids on inputs.** Collide with CSS `#email` by accident.

**Nav anchor specificity vs utilities.** `#nav .link.active` blocks redesign.

**Duplicate ids in the DOM.** Invalid; querySelector returns the first.

**Hash routing vs id.** URL `#section` is unrelated to CSS id but confuses teams.
