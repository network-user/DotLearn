# CSS-in-JS: плюсы, минусы и типичные боли в проде

## Суть

CSS-in-JS генерирует стили из JavaScript (styled-components, Emotion, Stitches). Динамика и colocation удобны в React, но runtime-инъекция стилей бьёт по SSR, размеру бандла, DevTools и предсказуемости каскада.

## Что ответить на собесе

- Плюсы: темы от props, scoped классы без ручного BEM, условные стили рядом с JSX.
- Минусы: runtime cost, FOUC/гидрация на SSR, нечитаемые сгенерированные классы, сложнее static extract.
- Zero-runtime варианты (Linaria, Vanilla Extract, Panda) компилируют CSS на билде.
- Для design tokens часто достаточно CSS variables + Modules.
- Честно добавляю: новые проекты на React 19 чаще берут Modules/Tailwind, а не runtime CSS-in-JS, из-за RSC и perf budget.

## Как это работает

Runtime styled-component:

```ts
import styled from "styled-components";

type BtnProps = { $primary?: boolean };

export const ActionBtn = styled.button<BtnProps>`
  padding: 0.5rem 1rem;
  border-radius: 8px;
  background: ${({ $primary }) => ($primary ? "#2563eb" : "#e2e8f0")};
  color: ${({ $primary }) => ($primary ? "#fff" : "#0f172a")};
`;
```

При каждом уникальном наборе props библиотека может вставить новый `<style>` в head.

SSR требует `ServerStyleSheet` / `StyleRegistry`, иначе HTML без стилей до гидрации.

Compile-time альтернатива:

```ts
import { css } from "@linaria/core";

export const btn = css`
  padding: 0.5rem 1rem;
`;
```

Класс статичен в CSS-файле бандла; JS не генерирует правила в рантайме.

## Подводные камни

**Специфичность вложенных селекторов.** `& svg { }` в styled легко бьёт глобальные reset неожиданно.

**Дублирование keyframes.** Каждый mount может регистрировать одноимённую анимацию.

**RSC граница.** Client-only styled в server component ломает архитектуру; нужен `"use client"` на листьях.

**Отладка в production.** Хеши классов без source maps; см. отдельный вопрос про maps.

**Миграция на static CSS.** Сотни styled() трудно вытащить без visual regression suite.
