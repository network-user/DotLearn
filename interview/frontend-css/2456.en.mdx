# display:none vs visibility:hidden

## In short

`display:none` removes the element from the render tree and flow: no space, layout reflows. `visibility:hidden` hides painting but the box remains; for a11y both usually exclude the node from the accessibility tree unless ARIA overrides.

## How to answer in an interview

- display:none: tabs, conditional blocks, treat as removed.
- visibility:hidden: reserve space, skeleton swap, sometimes opacity+visibility fades.
- Tab focus must not hit hidden controls; use `hidden`, `inert`, or conditional render.
- visibility:hidden + opacity:0 fades without layout shift.
- Honestly: for screen readers prefer `hidden`, `aria-hidden`, or conditional render over visibility alone.

## How it works

```css
.panel.is-collapsed {
  display: none;
}

.panel.is-loading {
  visibility: hidden;
}

.panel.is-loading::after {
  visibility: visible;
  content: "…";
}
```

```ts
function toggleTab(panel: HTMLElement, show: boolean) {
  panel.style.display = show ? "block" : "none";
  panel.toggleAttribute("hidden", !show);
}

function reserveSlot(node: HTMLElement, visible: boolean) {
  node.style.visibility = visible ? "visible" : "hidden";
}
```

```html
<section hidden id="settings">…</section>
```

The `hidden` attribute maps to display:none in the UA stylesheet.

## Pitfalls

**visibility:hidden still clickable.** Events may fire; add pointer-events:none.

**display toggle layout thrashing.** Mass show/hide in grids is costly.

**SEO and display:none.** Hidden text can be treated as cloaking if abused.

**opacity:0 vs visibility.** opacity leaves focusable elements.

**Animating display.** Not animatable; use grid 0fr tricks or class swaps.
