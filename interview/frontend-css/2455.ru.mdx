# useEffect и useLayoutEffect: когда что брать

## Суть

Оба хука запускают side effects после React commit в DOM. `useEffect` откладывается после paint (async), `useLayoutEffect` срабатывает синхронно до paint. LayoutEffect нужен, когда пользователь не должен видеть промежуточный кадр после измерений DOM.

## Что ответить на собесе

- useEffect: fetch, subscriptions, analytics, большинство side effects.
- useLayoutEffect: measure DOM, sync scroll position, prevent visual flicker после render.
- useLayoutEffect блокирует paint; не злоупотребляю на mobile.
- SSR: useLayoutEffect warning на сервере; defer до client или useEffect.
- Честно: 95% задач закрывает useEffect; layout только при доказанном flicker.

## Как это работает

```ts
import { useEffect, useLayoutEffect, useRef, useState } from "react";

function TooltipAnchor({ label }: { label: string }) {
  const anchorRef = useRef<HTMLButtonElement>(null);
  const [offset, setOffset] = useState(0);

  useLayoutEffect(() => {
    const node = anchorRef.current;
    if (!node) return;
    const { width } = node.getBoundingClientRect();
    setOffset(Math.max(0, width - 120));
  }, [label]);

  useEffect(() => {
    const node = anchorRef.current;
    if (!node) return;
    const log = () => console.log("clicked", label);
    node.addEventListener("click", log);
    return () => node.removeEventListener("click", log);
  }, [label]);

  return (
    <button ref={anchorRef} type="button" style={{ transform: `translateX(${offset}px)` }}>
      {label}
    </button>
  );
}
```

Timeline: render → commit DOM → useLayoutEffect → browser paint → useEffect.

## Подводные камни

**LayoutEffect для data fetching.** Блокирует UI без причины.

**Missing deps.** Stale measurements после prop change.

**Double render from setState in layout.** Ожидаемо, но может cascade.

**Strict Mode double invoke.** LayoutEffect cleanup должен быть idempotent.

**Замена CSS transition на layout measure.** Иногда CSS `anchor()` решает без JS.
