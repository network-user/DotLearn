# Монолит, сервисы или микросервисы: что меняется для тестирования?

## Суть

Монолит упаковывает UI, бизнес-логику и данные в одно развёртывание. Сервисная архитектура делит систему на крупные автономные блоки с API между ними. Микросервисы дробят продукт на мелкие независимые сервисы со своими релизами и масштабированием. Для QA меняется граница интеграционных проверок, объём контрактного тестирования и способ локализовать дефект.

## Что ответить на собесе

- В монолите фокусируюсь на end-to-end и регрессии одного артефакта: один билд, одна версия, проще воспроизвести баг.
- В сервисной и микросервисной схеме добавляю контрактные проверки между сервисами и мониторинг версий: падение может быть на стороне соседа, а не UI.
- Микросервисы дают независимые релизы, но цена для QA: больше стендов, feature flags, хаос при рассинхроне версий.
- Честно скажу: микросервисы не «лучше для тестирования»; они просто сдвигают риски с деплоя монолита на сеть, контракты и observability.

## Как это работает

**Монолит.** Один репозиторий или один deployable unit. Smoke после деплоя покрывает весь продукт. Регрессия часто одним прогоном UI и API. Масштабирование горизонтальное копирует весь монолит, поэтому нагрузочный тест проще интерпретировать: один bottleneck.

**Сервисная архитектура.** Несколько крупных сервисов (каталог, заказы, биллинг) общаются по HTTP/gRPC/очередям. Интеграционные тесты проверяют цепочки: «создать заказ → списать оплату → отправить письмо». Для QA важны стабильные контракты и тестовые двойники медленных зависимостей.

**Микросервисы.** Десятки мелких сервисов, отдельные команды, CI/CD на каждый. Пользовательский сценарий проходит через gateway, несколько API и асинхронные топики. Локализация бага требует trace id, логов и матрицы совместимости версий.

Пример набора проверок для заказа в распределённой системе:

```gherkin
Feature: Оформление заказа через микросервисный контур

  Background:
    Given сервис catalog доступен на v2.4
    And сервис payments доступен на v1.9
    And в Kafka топик order.events пуст

  Scenario: Happy path сквозного сценария
    When клиент создаёт заказ через API gateway
    Then сервис orders возвращает 201 и orderId
    And в топике order.events появляется событие OrderCreated
    And сервис notifications отправляет email в течение 30 секунд

  Scenario: Деградация payments не ломает каталог
    Given payments отвечает 503
    When клиент добавляет товар в корзину
    Then каталог возвращает 200
    And checkout показывает понятную ошибку оплаты
```

На собесе полезно упомянуть client-server: клиент (браузер, мобилка) тестируется отдельно от серверной части, но контракт API связывает обе стороны.

## Подводные камни

- **«Микросервисы = меньше регрессии».** На практике E2E дороже и хрупче; без контрактных тестов команды ловят баги только на стейджинге.
- **Версионный ад.** Сервис A v3 ожидает поле, которое B v2 ещё не шлёт. Нужна явная политика backward compatibility и smoke на пару версий.
- **Тестовые данные между сервисами.** Очистка БД одного сервиса оставляет мусор в другом; нужны orchestrated fixtures или изолированные tenant id.
- **Монолит внутри «микросервисов».** Shared library с бизнес-логикой создаёт скрытый coupling; интеграционный тест зелёный, прод падает из-за рассинхрона lib.
- **Игнор асинхронности.** Событие в Kafka приходит через секунды; flaky тест из-за фиксированного sleep без polling условия.
