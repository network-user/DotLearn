# Monolith, services, or microservices: what changes for testing?

## In short

A monolith packs UI, business logic, and data into one deployment. Service architecture splits the system into large autonomous blocks with APIs between them. Microservices break the product into small independent services with their own releases and scaling. For QA, integration boundaries, contract testing scope, and defect localization all shift.

## How to answer in an interview

- On a monolith I focus on end-to-end flows and regression of a single artifact: one build, one version, easier reproduction.
- With services and microservices I add contract checks between services and version tracking: failures may sit with a neighbor, not the UI.
- Microservices enable independent releases, but QA pays with more environments, feature flags, and chaos when versions drift.
- I am honest: microservices are not "better for testing"; they move risk from monolith deploys to network, contracts, and observability.

## How it works

**Monolith.** One repo or one deployable unit. Post-deploy smoke covers the whole product. Regression is often one UI and API run. Horizontal scaling copies the entire monolith, so load test results are easier to read: one bottleneck.

**Service architecture.** Several large services (catalog, orders, billing) talk over HTTP/gRPC/queues. Integration tests verify chains: "create order → charge payment → send email." QA needs stable contracts and test doubles for slow dependencies.

**Microservices.** Dozens of small services, separate teams, CI/CD per service. A user journey crosses a gateway, multiple APIs, and async topics. Bug triage needs trace ids, logs, and a version compatibility matrix.

Example checks for an order in a distributed setup:

```gherkin
Feature: Checkout through a microservice mesh

  Background:
    Given catalog service is available at v2.4
    And payments service is available at v1.9
    And Kafka topic order.events is empty

  Scenario: End-to-end happy path
    When the client creates an order via API gateway
    Then orders service returns 201 and orderId
    And order.events contains an OrderCreated event
    And notifications service sends email within 30 seconds

  Scenario: Payments degradation does not break catalog
    Given payments returns 503
    When the client adds an item to the cart
    Then catalog returns 200
    And checkout shows a clear payment error
```

On interviews, mention client-server: the client (browser, mobile) is tested apart from the server, but the API contract binds both sides.

## Pitfalls

- **"Microservices = less regression."** In practice E2E is costlier and flakier; without contract tests teams only catch bugs on staging.
- **Version hell.** Service A v3 expects a field service B v2 still omits. You need explicit backward compatibility and smoke across version pairs.
- **Test data across services.** Clearing one service DB leaves garbage in another; use orchestrated fixtures or isolated tenant ids.
- **Monolith inside "microservices."** A shared business library creates hidden coupling; integration tests pass while prod fails on lib drift.
- **Ignoring async.** A Kafka event arrives seconds later; fixed sleeps without polling make flaky tests.
