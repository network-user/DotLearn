# What **enter** and **exit** do and how the with block works

## In short

`__enter__` and `__exit__` are the context manager protocol that serves the `with` construct. `__enter__` is called on entering the block and usually hands out a resource, `__exit__` is called on exit and reliably releases it. The main advantage: `__exit__` runs even if an exception was raised inside the block.

## How to answer in an interview

An object becomes a context manager if it has `__enter__` and `__exit__`. On entering `with`, Python calls `__enter__`, and its result is bound to the name after `as`. On exit - whether normal or due to an exception - Python calls `__exit__(exc_type, exc_value, traceback)`. If there was an exception inside, these three arguments describe it, otherwise they are `None`. This way resource management does not depend on how we left the block, so a file or connection is guaranteed to be closed.

`with` guarantees the resource is released. So the code thinks about the logic, not about closing:

```python
class FileManager:
    def __init__(self, filename, mode):
        self.filename = filename
        self.mode = mode
        self.file = None

    def __enter__(self):
        self.file = open(self.filename, self.mode)
        return self.file

    def __exit__(self, exc_type, exc_value, traceback):
        if self.file:
            self.file.close()

with FileManager('example.txt', 'w') as f:
    f.write('Hello, World!')
```

Step by step:

1. `__enter__` opens the file and returns it - the object is available as `f`.
2. The body of the block runs.
3. `__exit__` closes the file, and this happens on any exit from the block, including after an error.

Typical uses of the protocol: files (open/close), network connections, database transactions (commit/rollback), locks in multithreading (acquire/release).

## Pitfalls

- The return value of `__exit__` controls the exception. If you return a truthy value, the exception from the block is suppressed. Usually you return `None` (that is, falsy), so the error propagates further after cleanup.
- `__enter__` and `with` are not the same thing. The name after `as` receives exactly the result of `__enter__`, not the manager itself. In the example above `f` is the file, not the `FileManager` object.
- Do not silence errors quietly. Catching an exception in `__exit__` should be done deliberately, otherwise bugs slip away in silence.
- For simple cases writing a class is not required: `contextlib.contextmanager` lets you make a manager out of a generator with a single `yield`, where the code before `yield` is `__enter__` and the code after is `__exit__`.
