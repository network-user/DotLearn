# Polymorphism made simple: one call, many behaviors

## In short

Polymorphism lets you call a method with the same name on objects of different classes, and each one responds in its own way. A single interface fronts different types, so the code is easier to extend without rewriting what already works.

## How to answer in an interview

Polymorphism is the ability of code to work with objects of different types through a common interface: you call `obj.speak()`, and the concrete implementation is selected based on the object's actual type. In Python it is dynamic and relies on duck typing — what matters is not which class the object belongs to, but whether it has the needed method. Because of this, a function expecting "something that can speak" works the same with a dog, a cat, or any new class added later.

## How it works

The word is built from the Greek "poly" (many) and "morph" (form) — "many-formedness". There are two kinds:

1. **Static** — resolved before running: method and operator overloading. In Python an operator is configured through dunders, for example `__add__` overrides `+`.
2. **Dynamic** — resolved at runtime through inheritance and method overriding. This is the main kind in Python.

```python
class Animal:
    def speak(self) -> str:
        raise NotImplementedError

class Dog(Animal):
    def speak(self) -> str:
        return "Woof!"

class Cat(Animal):
    def speak(self) -> str:
        return "Meow!"

def animal_sound(animal: Animal) -> None:
    print(animal.speak())

animal_sound(Dog())  # Woof!
animal_sound(Cat())  # Meow!
```

The `animal_sound` function calls `speak()` without knowing the concrete class. Substituting the right version happens at the moment of the call.

Polymorphism underpins the "Strategy" and "Factory Method" patterns and holds up the framework of extensible systems.

## Pitfalls

- In Python, polymorphism works through duck typing: a common base class isn't required, a matching method is enough. Inheritance is just one way to organize it.
- Don't confuse polymorphism with overloading: Python has no true method overloading by signature — redefining a method simply overwrites the previous one.
- If a subclass changes the meaning of a method (a different return type, different side effects), you break Liskov substitution, and polymorphism turns into a source of bugs.
- `isinstance` branching instead of calling a common method is a typical sign that polymorphism was missed somewhere.
