# Why wrap a complex subsystem behind a single Facade class

## In short

A facade is a single simplified entry point into a tangled subsystem. It hides a dozen internal classes behind a couple of clear methods, and the client works with the result without digging into the internals.

## How to answer in an interview

A facade is a structural pattern. I introduce a separate class that aggregates the needed subsystem components and exposes a narrow, convenient interface tailored to the client's scenarios. It doesn't change the internal logic of the components - it only frees the calling side from having to know the order of calls and the dependencies between them. This reduces coupling: the client depends on the facade rather than on a dozen details.

## How it works

The facade holds references to the subsystem objects and, within its methods, orchestrates them in the right order. From the outside you see a single call; inside there's a sequence of steps.

```python
class Light:
    def turn_on(self):
        print("Light is on")

    def turn_off(self):
        print("Light is off")

class Thermostat:
    def set_temperature(self, temperature):
        print(f"Temperature set to {temperature} degrees")

class MusicSystem:
    def play_music(self):
        print("Playing music")

class SmartHomeFacade:
    def __init__(self):
        self.light = Light()
        self.thermostat = Thermostat()
        self.music_system = MusicSystem()

    def start_evening_routine(self):
        self.light.turn_on()
        self.thermostat.set_temperature(22)
        self.music_system.play_music()

smart_home = SmartHomeFacade()
smart_home.start_evening_routine()
```

A single `start_evening_routine` method turns on the light, sets the temperature, and starts the music. The client doesn't need to know the order of the steps or even the `Light`, `Thermostat`, `MusicSystem` classes. Typical use cases: wrapping bulky graphics and networking-protocol libraries, providing a simplified entry into a legacy API, and tidying up an integration point.

## Pitfalls

- A facade doesn't forbid direct access to the subsystem - when needed, the components remain accessible directly; the facade merely offers a convenient default path.
- Don't turn the facade into a "god object": once it starts pulling in business logic and ballooning, it's no longer a facade but a dumping ground.
- A facade simplifies the interface but doesn't reduce the complexity of the subsystem itself - it hides that complexity rather than removing it.
- Don't confuse it with the Adapter: an adapter converts one interface into another compatible one, whereas a facade exposes a new simplified interface on top of many.
