# Зачем передавать зависимости снаружи вместо создания внутри класса?

## Суть

Внедрение зависимостей (Dependency Injection) - это передача объекту его зависимостей извне вместо их конструирования внутри. Связи между компонентами ослабляются, код становится тестируемее и проще в сопровождении.

## Что ответить на собесе

DI - это реализация принципа инверсии управления: класс не создаёт нужные ему объекты сам, а получает их готовыми.

- Зависимости приходят извне - чаще всего через конструктор, реже через сеттер или метод.
- Класс зависит от абстракции, а не от конкретной реализации, поэтому в тестах настоящую зависимость легко подменить моком, а в продакшене - заменить реализацию, не трогая зависимый код.
- Это прямое следствие буквы D в SOLID (Dependency Inversion).

Зависимость передаётся параметром конструктора и сохраняется в поле. Класс пользуется ею, ничего не зная о том, как она была создана.

<PyDemo title="DI: зависимость приходит снаружи" autoRun code={`class Database:
    def connect(self):
        return "Connected to the database"

class Service:
def **init**(self, database: Database):
self.database = database

    def perform_action(self):
        return self.database.connect()

db = Database()
service = Service(db)
print(service.perform_action())`} />

`Service` не вызывает `Database()` внутри себя - объект приходит снаружи. В тесте вместо реальной базы можно подсунуть заглушку с тем же интерфейсом:

```python
class FakeDatabase:
    def connect(self):
        return "fake"


assert Service(FakeDatabase()).perform_action() == "fake"
```

## Подводные камни

- DI - это не обязательно фреймворк. Простая передача аргумента в конструктор уже является внедрением зависимости; контейнеры DI решают лишь задачу автоматической сборки графа объектов.
- Перебор с инъекциями раздувает конструкторы: длинный список зависимостей - сигнал, что класс делает слишком много (нарушение SRP).
- Зависимость должна задаваться через общий интерфейс/протокол, иначе слабая связанность теряется - подменить реализацию станет нельзя.
- Сервис-локатор (объект сам тянет зависимость из глобального реестра) часто путают с DI, но он, наоборот, прячет зависимости и усложняет тестирование.
