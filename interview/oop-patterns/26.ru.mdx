# Инкапсуляция: почему данные прячут за методами

## Суть

Инкапсуляция объединяет данные и работающие с ними методы в одном объекте и закрывает прямой доступ к внутренностям. Менять состояние можно только через предусмотренный интерфейс, поэтому объект остаётся в согласованном состоянии.

## Что ответить на собесе

Инкапсуляция — это сокрытие внутреннего устройства объекта за методами-интерфейсом, чтобы снаружи нельзя было испортить его состояние в обход проверок. В Python нет жёстких модификаторов доступа, как в Java: всё держится на соглашениях. Одно подчёркивание `_attr` — «трогать снаружи не стоит», два подчёркивания `__attr` включают name mangling, и атрибут становится `_ClassName__attr`. Это не делает его по-настоящему приватным, а лишь усложняет случайный доступ и защищает от коллизий имён в наследниках.

Соглашения об именах задают уровни видимости:

- **Публичные** — без префикса, доступны откуда угодно.
- **Защищённые** — одно подчёркивание `_balance`: сигнал «это внутреннее».
- **Приватные** — два подчёркивания `__balance`: Python переименовывает атрибут в `_ClassName__balance` (name mangling).

```python
class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner
        self.__balance = balance

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("amount must be positive")
        self.__balance += amount
        return self.__balance

    def withdraw(self, amount):
        if not 0 < amount <= self.__balance:
            raise ValueError("invalid amount")
        self.__balance -= amount
        return self.__balance

    @property
    def balance(self):
        return self.__balance
```

Снаружи `__balance` напрямую не изменить — только через `deposit`/`withdraw`, которые гарантируют проверки. Для чтения удобно отдать `@property`: интерфейс остаётся как у атрибута, но за ним стоит метод.

## Подводные камни

- `__balance` не настоящий private. Через `account._BankAccount__balance` к нему всё ещё можно добраться — name mangling защищает от случайностей, не от злого умысла.
- Двойное подчёркивание задумано для предотвращения коллизий имён в иерархии наследования, а не как «секьюрити». Для «не трогай снаружи» обычно хватает одного подчёркивания.
- В Python геттеры/сеттеры на каждый атрибут — антипаттерн. Если логики нет, оставляйте публичный атрибут и переходите на `@property` только когда понадобится валидация или вычисление.
- Имя с двумя подчёркиваниями с обеих сторон (`__init__`) к инкапсуляции отношения не имеет — это дандеры, и mangling на них не распространяется.
