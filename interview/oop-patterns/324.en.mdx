# Strategy: swapping an algorithm at runtime through composition

## In short

Strategy hides a family of interchangeable algorithms behind a common interface and lets the context pick and switch between them at runtime. The behavior is defined by a strategy object rather than hardwired into the context itself.

## How to answer in an interview

A behavioral pattern built on composition, not inheritance. The context holds a reference to a strategy object and delegates the work to it through a single method. Any algorithm is a separate strategy with the same interface, so they can be swapped at runtime without touching the context. The main benefit is removing sprawling `if/elif` branches keyed on algorithm type and gaining openness for extension: a new variant means a new class. In Python a strategy is often just a function, since functions are first-class objects.

## How it works

The context knows nothing about the concrete implementation - only about the `Strategy` interface. Swapping via `set_strategy` changes the behavior instantly.

```python
from abc import ABC, abstractmethod

class SortStrategy(ABC):
    @abstractmethod
    def sort(self, data):
        ...

class Ascending(SortStrategy):
    def sort(self, data):
        return sorted(data)

class Descending(SortStrategy):
    def sort(self, data):
        return sorted(data, reverse=True)

class Sorter:
    def __init__(self, strategy: SortStrategy):
        self._strategy = strategy

    def set_strategy(self, strategy: SortStrategy):
        self._strategy = strategy

    def run(self, data):
        return self._strategy.sort(data)

data = [5, 2, 9, 1]
sorter = Sorter(Ascending())
print(sorter.run(data))    # [1, 2, 5, 9]
sorter.set_strategy(Descending())
print(sorter.run(data))    # [9, 5, 2, 1]
```

In Python this is often simplified: the strategy becomes a plain function or a `key` argument, and separate classes are unnecessary.

```python
strategy = lambda d: sorted(d, reverse=True)
print(strategy([5, 2, 9, 1]))   # [9, 5, 2, 1]
```

## Pitfalls

- Strategy and State are structurally close (the context delegates to an object), but their goals differ: Strategy selects an algorithm, while State models a change of states and the transitions between them.
- The number of classes grows; for a couple of simple branches, separate strategies can be over-engineering.
- The client is required to know which strategies exist in order to pick the right one - part of the responsibility leaks outward.
- In Python, do not create classes where a function suffices: extra boilerplate is a common mistake when porting the pattern "the Java way".
