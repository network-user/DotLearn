# Полиморфизм на пальцах: один вызов — много поведений

## Суть

Полиморфизм позволяет дёргать метод с одним и тем же именем у объектов разных классов, и каждый отрабатывает по-своему. Один интерфейс прикрывает разные типы, поэтому код легче расширять, не переписывая то, что уже работает.

## Что ответить на собесе

Полиморфизм — это способность кода работать с объектами разных типов через общий интерфейс: вызываешь `obj.speak()`, а конкретная реализация выбирается по фактическому типу объекта. В Python он динамический и опирается на утиную типизацию — важно не то, какого класса объект, а есть ли у него нужный метод. За счёт этого функция, ожидающая «что-то, умеющее speak», одинаково работает и с собакой, и с кошкой, и с любым новым классом, который добавят позже.

## Как это работает

Слово составлено из греческих «поли» (много) и «морф» (форма) — «многоформенность». Различают две разновидности:

1. **Статический** — разрешается до запуска: перегрузка методов и операторов. В Python оператор настраивают через дандеры, например `__add__` переопределяет `+`.
2. **Динамический** — разрешается во время выполнения через наследование и переопределение методов. Именно он в Python основной.

```python
class Animal:
    def speak(self) -> str:
        raise NotImplementedError

class Dog(Animal):
    def speak(self) -> str:
        return "Woof!"

class Cat(Animal):
    def speak(self) -> str:
        return "Meow!"

def animal_sound(animal: Animal) -> None:
    print(animal.speak())

animal_sound(Dog())  # Woof!
animal_sound(Cat())  # Meow!
```

Функция `animal_sound` зовёт `speak()`, не зная конкретного класса. Подстановка нужной версии происходит в момент вызова.

Полиморфизм лежит в основе паттернов «Стратегия» и «Фабричный метод» и держит каркас расширяемых систем.

## Подводные камни

- В Python полиморфизм работает через утиную типизацию: общий базовый класс необязателен, достаточно совпадающего метода. Наследование — лишь один из способов это организовать.
- Не путайте полиморфизм с перегрузкой: в Python нет настоящей перегрузки методов по сигнатуре — повторное определение метода просто затирает предыдущее.
- Если наследник меняет смысл метода (другой тип возврата, другие побочные эффекты), вы ломаете подстановку Лисков, и полиморфизм превращается в источник багов.
- `isinstance`-ветвления вместо вызова общего метода — типичный признак, что полиморфизм где-то упустили.
