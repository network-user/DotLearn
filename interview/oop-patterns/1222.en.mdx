# One interface, many types: polymorphism in the Python standard library

## In short

Polymorphism is when the same call works with objects of different types. In Python it rests not on a common base class but on the presence of the needed method (duck typing). The classic example is `len()`: it computes the length of a string, a list, and any object with `__len__` the same way.

## How to answer in an interview

Polymorphism gives a single interface for different types. In Python it relies on dynamic typing: what matters is not inheritance but whether the object has the right method. Built-in functions like `len()`, `sorted()`, `iter()` are polymorphic because they delegate the work to dunder methods (`__len__`, `__iter__`, comparisons). The dunder `__str__` is polymorphic too: each class defines its own string representation. Thanks to this you can write generic functions without tying yourself to the argument's concrete class.

## How it works

Polymorphism comes both through inheritance (a subclass's overridden method) and without it - as long as the object has a method with the right name. Built-in functions reach for the protocol:

```python
print(len("Hello"))      # 5
print(len([1, 2, 3, 4])) # 4
print(sorted([3, 1, 2])) # [1, 2, 3]
```

Under the hood, `len(x)` calls `type(x).__len__(x)`, so your class with `__len__` works too. The same goes for `__str__`:

```python
class Dog:
    def __str__(self):
        return "this is a dog"


class Cat:
    def __str__(self):
        return "this is a cat"


print(Dog())  # this is a dog
print(Cat())  # this is a cat
```

A generic function doesn't check the type - it simply counts on the method being there:

```python
class Dog:
    def speak(self):
        return "Woof!"


class Cat:
    def speak(self):
        return "Meow!"


def animal_sound(animal):
    return animal.speak()


print(animal_sound(Dog()))  # Woof!
print(animal_sound(Cat()))  # Meow!
```

`Dog` and `Cat` aren't even linked by a common ancestor - `animal_sound` works with any object that has `speak`.

## Pitfalls

- Duck typing isn't checked in advance: if the object lacks `speak`, the error surfaces only at call time (`AttributeError`).
- `len()` requires precisely `__len__`, which returns a non-negative integer; having `__iter__` won't help here.
- `sorted()` returns a new list, and for a string it gives a list of characters rather than a string - which surprises people.
- Polymorphism through inheritance requires matching method signatures; if a subclass changes the parameters, generic code can break.
