# What the @dataclass decorator gives you and what boilerplate it removes

## In short

`@dataclass` looks at a class's field annotations and writes `__init__`, `__repr__`, and `__eq__` for you. You get a data container class without any hand-written boilerplate.

## How to answer in an interview

`@dataclass` from the `dataclasses` module (since Python 3.7) automatically generates utility methods based on the declared, type-annotated fields. Instead of hand-writing the constructor, comparison, and string representation, I simply list the fields. Through the decorator's flags you can additionally enable ordering (`order=True`), immutability (`frozen=True`), and other methods.

## How it works

Declare fields with annotations; the methods appear on their own:

```python
from dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int
```

`__init__` now accepts `x` and `y`, `__repr__` gives a readable string, and `__eq__` compares by field values:

```python
p1 = Point(1, 2)
p2 = Point(1, 2)
p3 = Point(3, 4)

print(p1)          # Point(x=1, y=2)
print(p1 == p2)    # True  — fields match
print(p1 == p3)    # False
```

Default values are set right in the annotation, and for mutable values you use `field(default_factory=...)`, otherwise all objects would share the same list:

```python
from dataclasses import dataclass, field

@dataclass
class Cart:
    items: list = field(default_factory=list)
```

## Pitfalls

- A mutable default value (`items: list = []`) is forbidden, and Python will raise an error; you need `field(default_factory=list)`.
- By default the object is mutable and not hashable; to put it in a `set` or use it as a `dict` key, add `frozen=True`.
- Fields with defaults must come after fields without defaults, otherwise you'll get a class-definition error.
- `@dataclass` does no deep copying and does not validate types: annotations are hints, not runtime checks.
