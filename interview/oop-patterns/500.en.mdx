# A factory of object families: how Abstract Factory works

## In short

Abstract Factory is a creational pattern that hides the creation of an entire family of related objects behind a common interface. The client receives ready-made products without knowing which concrete classes stand behind the factory.

## How to answer in an interview

The abstract factory defines an interface for creating a group of interrelated products so that client code does not depend on concrete classes. Each concrete factory produces its own consistent set of products (for example, everything for Windows or everything for macOS), while the client works only through abstractions. This makes it easy to swap an entire family at once and protects against mixing incompatible elements.

The pattern is built from four roles. Abstract products define the interfaces (`Button`, `Checkbox`). Concrete products implement them for a platform. The abstract factory declares a creation method for each product, and concrete factories return a consistent set. The client holds a reference to the abstract factory and never sees the implementations.

The pattern fits when the system must not depend on how products are created, when it is configured with one of several families, and when it is important to guarantee that the objects are compatible with each other.

```python
from abc import ABC, abstractmethod

class Button(ABC):
    @abstractmethod
    def paint(self) -> str: ...

class Checkbox(ABC):
    @abstractmethod
    def paint(self) -> str: ...

class WindowsButton(Button):
    def paint(self) -> str:
        return "Windows button"

class WindowsCheckbox(Checkbox):
    def paint(self) -> str:
        return "Windows checkbox"

class MacOSButton(Button):
    def paint(self) -> str:
        return "macOS button"

class MacOSCheckbox(Checkbox):
    def paint(self) -> str:
        return "macOS checkbox"

class GUIFactory(ABC):
    @abstractmethod
    def create_button(self) -> Button: ...

    @abstractmethod
    def create_checkbox(self) -> Checkbox: ...

class WindowsFactory(GUIFactory):
    def create_button(self) -> Button:
        return WindowsButton()

    def create_checkbox(self) -> Checkbox:
        return WindowsCheckbox()

class MacOSFactory(GUIFactory):
    def create_button(self) -> Button:
        return MacOSButton()

    def create_checkbox(self) -> Checkbox:
        return MacOSCheckbox()

def render(factory: GUIFactory) -> None:
    print(factory.create_button().paint())
    print(factory.create_checkbox().paint())

render(WindowsFactory())
render(MacOSFactory())
```

## Pitfalls

- Don't confuse it with Factory Method: that one creates a single product, while Abstract Factory creates an entire family of consistent products.
- Adding a new product to the family is expensive: you have to change the factory interface and every concrete factory. The pattern extends well with new families but poorly with new kinds of products.
- The point of the pattern is lost if concrete factories mix products from different families (for example, a Windows button with a macOS checkbox) - that is exactly what the pattern guards against.
- For one or two products the pattern is often overkill and only bloats the code.
