# Template Method: fix the algorithm's skeleton, hand the steps to subclasses

## In short

Template Method locks the sequence of an algorithm's steps in a base-class method, while making the steps themselves overridable in subclasses. The common logic lives in one place, the differences in the subclasses.

## How to answer in an interview

A behavioral pattern. The base class has a single public skeleton method that calls a set of smaller steps in the right order. Some steps are implemented in the base and shared by everyone, some are declared abstract or as a stub and overridden by subclasses. The client invokes only the skeleton method, while the internal content varies. The pattern removes duplication of the common part and at the same time prevents subclasses from breaking the order of steps: inversion of control in the spirit of "don't call us, we'll call you".

The skeleton method (`prepare_recipe`) sets a strict order. The steps `boil_water` and `pour_in_cup` are the same for everyone, while each subclass must define `brew` and `add_condiments`.

```python
from abc import ABC, abstractmethod

class CaffeineBeverage(ABC):
    def prepare_recipe(self):
        self.boil_water()
        self.brew()
        self.pour_in_cup()
        if self.wants_condiments():
            self.add_condiments()

    def boil_water(self):
        print("Кипятим воду")

    def pour_in_cup(self):
        print("Наливаем в чашку")

    def wants_condiments(self):
        return True

    @abstractmethod
    def brew(self):
        ...

    @abstractmethod
    def add_condiments(self):
        ...

class Tea(CaffeineBeverage):
    def brew(self):
        print("Завариваем чай")

    def add_condiments(self):
        print("Добавляем лимон")

class Coffee(CaffeineBeverage):
    def brew(self):
        print("Проливаем кофе через фильтр")

    def add_condiments(self):
        print("Добавляем сахар и молоко")

Tea().prepare_recipe()
Coffee().prepare_recipe()
```

The `wants_condiments` method is a hook: an optional step with a sensible default that a subclass may, but is not required to, override.

## Pitfalls

- The pattern is built on inheritance, and that rigidly couples the subclass to the base. If there's a lot of variability, Strategy with composition turns out more flexible.
- The skeleton method should be protected from overriding (by meaning it's `final`); in Python this is only a convention, the language doesn't forbid overriding it.
- It's easy to overload the base with abstract steps, forcing subclasses to implement what they don't need. Hook methods with defaults help.
- Don't confuse a hook (optional, with a default implementation) and an abstract step (mandatory to implement).
