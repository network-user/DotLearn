# __new__ vs __init__: who creates the object, and who configures it

## In short

`__new__` is responsible for allocating and returning the instance itself, while `__init__` only fills the already-created object with data. The order is strict: `__new__` first, then `__init__`.

## How to answer in an interview

When you write `MyClass(10)`, the metaclass runs two steps. First, `__new__(cls, ...)` is called — this is a static method that creates and returns the instance; its job is to hand back an object. If the returned object is an instance of this class, `__init__(self, ...)` is then called, which returns nothing (more precisely, it must return `None`) and only configures state through `self`. In 99% of code you override only `__init__`. `__new__` is touched rarely: when subclassing immutable types (`int`, `str`, `tuple`), when implementing a singleton or instance caching, and in metaclasses.

`__new__` receives the class `cls` as its first argument and usually delegates creation to the parent:

```python
class MyClass:
    def __new__(cls, *args, **kwargs):
        instance = super().__new__(cls)
        print("instance created")
        return instance

    def __init__(self, value):
        self.value = value
        print("initialized")

obj = MyClass(10)
# instance created
# initialized
```

If the class is simple, there's no need to override `__new__` — `__init__` is enough:

```python
class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point(1, 2)
print(p.x, p.y)  # 1 2
```

## Pitfalls

- `__init__` must not return anything other than `None`. `return self.value` from `__init__` leads to a `TypeError`.
- `__init__` runs only if `__new__` returned an instance of the same class. Return a foreign object, and `__init__` is silently skipped.
- `__new__` is a static method and receives `cls` as an explicit first argument; forgetting to pass `cls` into `super().__new__(cls)` is a common mistake.
- When subclassing an immutable type, the value is set precisely in `__new__`: `int.__new__(cls, value)`, because in `__init__` it's too late to change it — the object is immutable.
