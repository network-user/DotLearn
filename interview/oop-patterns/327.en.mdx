# State: an object changes behavior by delegating it to the current state

## In short

State turns each state of an object into a separate class and lets it handle the actions. The context does not branch logic with `if/else`; instead it delegates the work to the current state object, which knows how to switch the context to the next state.

## How to answer in an interview

A behavioral pattern. The context holds a reference to a state object and forwards calls to it. All states share a common interface, and the transitions themselves (who follows whom) are defined inside the states. This removes bulky conditionals checking "what mode am I in right now" and makes adding a new state a local change - a new class. Structurally it resembles Strategy, but the meaning is different: State models a finite state machine and manages the transitions itself, whereas Strategy simply swaps the algorithm at the client's will.

## How it works

Each state implements the interface and returns the next state. The context only stores the current one and reassigns it.

```python
class State:
    def insert_coin(self, machine):
        return machine.state

    def select_drink(self, machine):
        return machine.state

    def dispense(self, machine):
        return machine.state

class WaitingForCoin(State):
    def insert_coin(self, machine):
        print("Coin accepted. Select a drink.")
        return SelectingDrink()

class SelectingDrink(State):
    def select_drink(self, machine):
        print("Drink selected. Dispensing.")
        return Dispensing()

class Dispensing(State):
    def dispense(self, machine):
        print("Drink dispensed. Waiting for the next coin.")
        return WaitingForCoin()

class VendingMachine:
    def __init__(self):
        self.state = WaitingForCoin()

    def insert_coin(self):
        self.state = self.state.insert_coin(self)

    def select_drink(self):
        self.state = self.state.select_drink(self)

    def dispense(self):
        self.state = self.state.dispense(self)

machine = VendingMachine()
machine.insert_coin()
machine.select_drink()
machine.dispense()
```

The transition logic is concentrated in the states themselves, and the context stays thin.

## Pitfalls

- Transitions spread across classes give the full picture of the automaton only if you mentally assemble them together - code navigation gets harder.
- States often need access to the context's data; pass the context into the methods or keep a reference to it instead of duplicating the data.
- If there are only two or three states and the transitions are trivial, a plain `enum` plus `match`/`if` may be simpler than the pattern.
- Do not confuse it with Strategy: the structure is the same, but State is about a finite state machine with self-driven transitions, while Strategy is about interchangeable algorithms chosen by the client.
