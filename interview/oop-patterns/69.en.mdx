# What problem does super() solve, and why you shouldn't hardcode the parent's name

## In short

`super()` gives access to a method's implementation higher up the inheritance hierarchy without tying you to a specific parent's name. Relying on the MRO instead of a hard-coded class name makes the hierarchy portable and saves you under multiple inheritance.

## How to answer in an interview

`super()` returns a proxy object that delegates the call to the next class in the current object's MRO, not to the "parent" in the naive sense. The main benefit is decoupling from a specific class name: when you rename or rearrange base classes, the code keeps working. In cooperative multiple inheritance, it's exactly `super()` that guarantees each class in a diamond scheme runs exactly once and the common ancestor isn't called twice.

## How it works

Basic inheritance is trivial — a subclass simply gets its ancestor's methods:

```python
class Animal:
    def speak(self):
        print("Animal speaks")

class Dog(Animal):
    def bark(self):
        print("Dog barks")

dog = Dog()
dog.speak()  # Animal speaks
dog.bark()   # Dog barks
```

It gets interesting with multiple inheritance, where a diamond scheme appears: two classes inherit a common ancestor, and a third inherits both of those classes. If every method calls `super().method()`, Python walks the classes strictly in MRO order:

```python
class A:
    def do_something(self):
        print("A")

class B(A):
    def do_something(self):
        print("B")
        super().do_something()

class C(A):
    def do_something(self):
        print("C")
        super().do_something()

class D(B, C):
    def do_something(self):
        print("D")
        super().do_something()

D().do_something()
```

The output is `D`, `B`, `C`, `A`: the MRO arranges the classes into a linear order (you can inspect it via `D.__mro__`), and `super()` in `B` points not to `A` but to the class that follows `B` in that order, namely `C`. This way the common ancestor `A` is called once, not twice.

## Pitfalls

- `super()` follows the MRO of the **instance**, not the static "class–its base" pair. That's why `super()` inside `B` may lead to `C`, even though `B` doesn't directly inherit from `C`.
- If even one class in a cooperative chain forgets to call `super()`, the following methods simply won't run — the chain breaks silently.
- Method signatures in a cooperative chain must be compatible, otherwise passing arguments through `super()` will break.
- A hard-coded `A.do_something(self)` instead of `super()` breaks cooperativeness: the common ancestor may be called more than once.
