# Who creates the object and who configures it: **new** vs **init**

## In short

`__new__` creates and returns the instance itself: it runs first. `__init__` receives the already-built object and fills it with state. You rarely need to override `__new__`: mostly when subclassing immutable types and in tricks like singletons.

## How to answer in an interview

An object's birth in Python is two steps. First `__new__(cls, ...)`, a static method that allocates and returns the instance, is called. Then Python calls `__init__(self, ...)` on that instance to set its attributes. The key nuance: `__init__` returns nothing (only `None`) and does not create the object, it merely configures it. If `__new__` returns an instance of a different class, `__init__` may not be called for it at all. You should touch `__new__` only when you need to intervene precisely at the moment of creation (for example with immutable `tuple`/`str`, where the value is fixed at creation time).

`__new__` goes first and must return an instance, otherwise `__init__` won't run:

```python
class MyClass:
    def __new__(cls, *args, **kwargs):
        print("1: creating the instance")
        return super().__new__(cls)

    def __init__(self, value):
        print("2: initializing the instance")
        self.value = value

obj = MyClass(10)
```

The print order is the line from `__new__` first, then from `__init__`. For immutable types the value can't be changed after creation, so a `tuple` subclass edits `__new__`:

```python
class Doubled(tuple):
    def __new__(cls, iterable):
        doubled = [x * 2 for x in iterable]
        return super().__new__(cls, doubled)

print(Doubled([1, 2, 3]))  # (2, 4, 6)
```

Here the tuple's contents are set at the creation stage. Doing this in `__init__` would already be too late, since the object is immutable.

## Pitfalls

- `__new__` is a static method and receives `cls` as its first argument, not `self`; the object doesn't exist yet at this moment.
- If `__new__` doesn't return an instance of `cls` (or a subclass of it), `__init__` won't be called, which is a frequent source of "silent" bugs.
- A naive singleton via `__new__` will still call `__init__` on every access to the class, so state can be accidentally overwritten.
- In ordinary classes you don't need to override `__new__`; `__init__` is enough for initialization.
