# The **del** destructor in Python: why it exists and why you can't rely on it

## In short

A destructor is a method the interpreter calls when an object is destroyed; in Python `__del__` plays this role. Formally it's meant for releasing resources, but the moment of its call is determined by the garbage collector, so it isn't a reliable cleanup mechanism.

## How to answer in an interview

`__del__` is called when an object's reference count drops to zero and the garbage collector decides to remove it. The problem is that the moment of this event is non-deterministic: with circular references the object can live until the cyclic collector runs, and on program exit `__del__` may not run at all. So for deterministic resource release (files, sockets, connections), Python uses not the destructor but context managers (`with` and the `__enter__`/`__exit__` protocol) or an explicit `close()`. `__del__` is left as a last-resort safety net, not as the main path.

## How it works

`__del__` fires at the moment the object is destroyed:

```python
class FileHandler:
    def __init__(self, filename):
        self.file = open(filename, 'w')
        print(f"Файл {filename} открыт")

    def write_data(self, data):
        self.file.write(data)

    def __del__(self):
        self.file.close()
        print("Файл закрыт")

handler = FileHandler('example.txt')
handler.write_data('Привет, мир!')
del handler  # decrement the reference count to zero → call __del__
```

Important: `del handler` does not "delete the object" directly; it only removes one reference. The object is destroyed (and `__del__` is called) only when the **last** reference disappears. If the object is held somewhere else, the destructor is postponed.

A reliable alternative is a context manager, where cleanup is guaranteed on exit from the block:

```python
class FileHandler:
    def __init__(self, filename):
        self.file = open(filename, 'w')

    def __enter__(self):
        return self

    def __exit__(self, exc_type, exc, tb):
        self.file.close()
        print("Файл закрыт")

with FileHandler('example.txt') as h:
    h.file.write('Привет, мир!')
# here __exit__ has already run deterministically
```

## Pitfalls

- The moment `__del__` is called is non-deterministic: it's dictated by the garbage collector, not by your code.
- With circular references, objects are collected by the cyclic collector later; on old Python versions cycles with `__del__` might not be collected at all.
- `del x` decrements the reference count rather than calling the destructor directly: with other live references the object survives.
- An exception raised inside `__del__` doesn't propagate normally, and it's merely printed to `stderr` and ignored.
- For guaranteed resource cleanup use `with`/`__exit__` or an explicit `close()`, and `__del__` only as a backup.
