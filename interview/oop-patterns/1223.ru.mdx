# Абстрактные базовые классы: как зафиксировать обязательный интерфейс

## Суть

Абстрактный базовый класс (ABC) - это контракт: он перечисляет методы, которые наследник обязан реализовать, но сам их не реализует. Попытка создать экземпляр класса с нереализованным абстрактным методом падает с ошибкой ещё до запуска основной логики.

## Что ответить на собесе

ABC задаёт интерфейс и гарантирует, что все наследники его выполнят.

- В Python это модуль `abc`: класс наследуют от `ABC`, а обязательные методы помечают декоратором `@abstractmethod`.
- Пока подкласс не реализует все такие методы, его нельзя инстанцировать: ошибка возникает в момент создания объекта, а не где-то глубоко по ходу выполнения.
- ABC полезны в больших проектах и командной работе: видно, что класс обязан реализовать, поведение становится предсказуемым.
- Альтернатива для проверки «есть ли нужные методы» - это `typing.Protocol` и структурная типизация.

## Как это работает

`Animal` объявляет абстрактный `make_sound` и сам по себе неинстанцируем. `Dog` и `Cat` дают свои реализации и работают нормально. А вот пустой наследник упадёт:

```python
class Mute(Animal):
    pass


Mute()  # TypeError: Can't instantiate abstract class Mute
        # with abstract method make_sound
```

## Подводные камни

- Просто унаследоваться от `ABC` мало: контракт включается только декоратором `@abstractmethod` на конкретных методах.
- Сам по себе абстрактный метод может иметь тело, но наследник всё равно обязан его переопределить, иначе экземпляр не создать.
- Ошибка возникает при создании экземпляра, а не при объявлении подкласса: частая путаница на собесе.
- `@abstractmethod` комбинируют с `@property`, `@classmethod`, `@staticmethod`, но порядок важен: `@abstractmethod` идёт самым внутренним декоратором.
- Если нужна только проверка «у объекта есть метод X» без жёсткой иерархии, обычно лучше `typing.Protocol`.
