# Decoding SOLID: the five pillars of maintainable OOP code

## In short

SOLID is a set of five rules for designing classes that keeps code flexible and understandable as a project grows. Each letter stands for a separate idea, and together they neutralize the most common architectural mistakes.

## How to answer in an interview

SOLID is made up of five principles: single responsibility (a class changes for one reason), open/closed (we add behavior by extension, not by editing), Liskov substitution (a subclass works wherever the base class does without breaking the logic), interface segregation (a client does not depend on methods it doesn't need), and dependency inversion (modules depend on abstractions, not on concrete implementations). By following them, we get loosely coupled, easily extensible code.

## How it works

Let's go through each letter separately.

**S — Single Responsibility.** A class has one area of responsibility, which means a single reason to change. If `Report` both collects data and prints it, it changes for two different reasons at once. We split it into `ReportGenerator` and `ReportPrinter` — each handles its own job.

**O — Open/Closed.** An entity is open for extension but closed for modification. We add a new payment type as a separate class with a shared interface, without cracking open already-debugged code.

**L — Liskov Substitution.** An object of a subtype should substitute for the base type without surprises. The classic anti-example is `Penguin` inheriting `Bird.fly()`: a penguin doesn't fly, so such a method has no place in the hierarchy.

**I — Interface Segregation.** Don't force a client to depend on methods it doesn't use. A fat `Machine` with `print`, `scan`, and `fax` is better split into narrow `Printer` and `Scanner`.

**D — Dependency Inversion.** High-level code relies on an abstraction, not on a detail.

```python
from abc import ABC, abstractmethod

class Database(ABC):
    @abstractmethod
    def save(self, order): ...

class PostgresDatabase(Database):
    def save(self, order):
        return f"saved {order} to postgres"

class OrderProcessor:
    def __init__(self, db: Database):
        self._db = db

    def process(self, order):
        return self._db.save(order)
```

`OrderProcessor` knows nothing about the concrete DBMS — you can swap it out without touching its code.

## Pitfalls

- SOLID is a guideline, not dogma. Blindly splitting things up for the sake of "one responsibility" breeds dozens of tiny classes and makes navigation harder.
- Liskov violations often hide in an "is-a" relationship that holds in everyday speech but is false in code (a square as a subclass of a rectangle).
- Dependency inversion gets confused with dependency injection: DIP is about the direction of dependencies pointing at abstractions, while DI is merely a way to pass them in.
- The principles overlap: fixing one easily affects another, so evaluate the design as a whole rather than checking off boxes.
