# Magic methods: how to teach your class to behave like a built-in type

## In short

Dunder methods (from double underscore: two underscores around the name) are hooks that Python calls itself in response to syntax and built-in functions: `+`, `==`, `len()`, printing. By overriding them you embed your object into the language, and it starts behaving like a native type.

## How to answer in an interview

Magic methods are a protocol between your class and the interpreter. When I write `a + b`, Python doesn't "add objects" itself, it calls `a.__add__(b)`; `len(x)` under the hood is `x.__len__()`; `print(x)` reaches for `__str__`. So by implementing the right dunder methods, I make the object compatible with operators, the container protocol, iteration and formatting; no special APIs are needed, purely familiar syntax. This is exactly "Pythonic-ness": code using my objects reads the same as code with lists and numbers.

## How it works

`__init__` configures the object at creation:

```python
class Dog:
    def __init__(self, name):
        self.name = name

print(Dog("Buddy").name)  # Buddy
```

`__str__` and `__repr__` control the string representation: the first for humans (`print`), the second for debugging (`repr`):

```python
class Dog:
    def __init__(self, name):
        self.name = name
    def __str__(self):
        return f"Dog named {self.name}"
    def __repr__(self):
        return f"Dog('{self.name}')"

d = Dog("Buddy")
print(d)        # Dog named Buddy
print(repr(d))  # Dog('Buddy')
```

`__add__` wires up the `+` operator, and `__eq__` enables equality comparison:

```python
class Vector:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)
    def __eq__(self, other):
        return (self.x, self.y) == (other.x, other.y)
    def __repr__(self):
        return f"Vector({self.x}, {self.y})"

print(Vector(1, 2) + Vector(3, 4))   # Vector(4, 6)
print(Vector(1, 2) == Vector(1, 2))  # True
```

And `__len__` embeds the object into `len()` and into "emptiness" checks:

```python
class Bag:
    def __init__(self, items):
        self.items = items
    def __len__(self):
        return len(self.items)

print(len(Bag([1, 2, 3, 4])))  # 4
```

## Pitfalls

- If `__str__` is missing, `print` falls back to `__repr__`; that's why a meaningful `__repr__` is always worth writing.
- By defining `__eq__` you make the object unhashable: Python sets `__hash__` to `None`, and the object can't be put into a `set` or used as a `dict` key until you add your own `__hash__`.
- `__add__` should return a new object rather than mutate `self`, since doing so breaks the expectation that `+` has no side effects.
- If `other` lacks the needed attributes, operator dunder methods will fail; in production you should return `NotImplemented` so Python tries the reflected operation.
