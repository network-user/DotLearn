# Фабричный метод: как отдать создание объектов подклассам

## Суть

Фабричный метод - порождающий паттерн: базовый класс объявляет метод-«фабрику», а решение, объект какого конкретного класса вернуть, принимают подклассы. Так логика, работающая с продуктом, не привязывается к его конкретному типу.

## Что ответить на собесе

Фабричный метод выносит создание объекта в отдельный метод, который переопределяют наследники. Базовый класс пишет код в терминах общего интерфейса продукта и зовёт `factory_method()`, не зная, какой именно подкласс продукта получит. За счёт этого добавление нового типа продукта сводится к новой паре «продукт + создатель» без правки существующего кода - это прямое следование принципу открытости/закрытости.

## Как это работает

Объявляем абстрактный продукт и абстрактного создателя с методом-фабрикой. Конкретные создатели переопределяют фабрику и возвращают конкретный продукт. Вся остальная логика создателя работает через общий интерфейс.

```python
from abc import ABC, abstractmethod

class Transport(ABC):
    @abstractmethod
    def deliver(self) -> str:
        ...

class Car(Transport):
    def deliver(self) -> str:
        return "Доставка автомобилем"

class Bike(Transport):
    def deliver(self) -> str:
        return "Доставка велосипедом"

class Logistics(ABC):
    @abstractmethod
    def create_transport(self) -> Transport:
        ...

    def plan_delivery(self) -> str:
        transport = self.create_transport()
        return transport.deliver()

class RoadLogistics(Logistics):
    def create_transport(self) -> Transport:
        return Car()

class CityLogistics(Logistics):
    def create_transport(self) -> Transport:
        return Bike()

print(RoadLogistics().plan_delivery())  # Доставка автомобилем
print(CityLogistics().plan_delivery())  # Доставка велосипедом
```

Метод `plan_delivery` не упоминает ни `Car`, ни `Bike` - он работает с абстракцией `Transport`. Конкретику задаёт только переопределённый `create_transport`.

Паттерн уместен, когда:

- решение о классе создаваемого объекта логично отдать подклассам;
- систему хотят расширять новыми типами без правки рабочего кода;
- нужно ослабить связанность кода с конкретными классами и работать через интерфейсы.

## Подводные камни

- Не путайте с Абстрактной фабрикой: фабричный метод порождает один продукт через переопределение метода, а абстрактная фабрика - целое семейство связанных продуктов через объект-фабрику.
- Паттерн оправдан, когда вариантов продукта несколько и они меняются. Для одного-двух стабильных типов он добавляет лишнюю иерархию ради иерархии.
- В Python нередко хватает простой функции-фабрики или передачи самого класса как аргумента - полноценная иерархия создателей не всегда нужна.
- Создатель и продукт должны эволюционировать согласованно: общий интерфейс продукта - это контракт, на который опирается логика создателя.
