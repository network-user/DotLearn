# Расшифруй SOLID: пять опор поддерживаемого ООП-кода

## Суть

SOLID — это набор из пяти правил проектирования классов, который удерживает код гибким и понятным при росте проекта. Каждая буква отвечает за отдельную идею, а вместе они гасят самые частые ошибки архитектуры.

## Что ответить на собесе

SOLID складывается из пяти принципов проектирования классов; соблюдая их, получаем слабо связанный и легко расширяемый код.

- **S** — единственная ответственность: класс меняется по одной причине.
- **O** — открытость/закрытость: поведение добавляем расширением, а не правкой.
- **L** — подстановка Лисков: наследник работает там же, где базовый класс, не ломая логику.
- **I** — разделение интерфейсов: клиент не зависит от методов, которые ему не нужны.
- **D** — инверсия зависимостей: модули завязаны на абстракции, а не на конкретные реализации.

## Как это работает

Разберём каждую букву отдельно.

**S — Single Responsibility.** У класса одна зона ответственности, значит и одна причина для изменения. Если `Report` и собирает данные, и печатает их, то меняется по двум поводам сразу. Разносим на `ReportGenerator` и `ReportPrinter` — каждый отвечает за своё.

**O — Open/Closed.** Сущность открыта для расширения, но закрыта для изменения. Новый тип платежа добавляем отдельным классом с общим интерфейсом, не вскрывая уже отлаженный код.

**L — Liskov Substitution.** Объект подтипа должен подставляться вместо базового без сюрпризов. Классический антипример — `Penguin`, наследующий `Bird.fly()`: пингвин не летает, поэтому такой метод в иерархии не место.

**I — Interface Segregation.** Не навязывай клиенту методы, которыми он не пользуется. Толстый `Machine` с `print`, `scan`, `fax` лучше разбить на узкие `Printer` и `Scanner`.

**D — Dependency Inversion.** Высокоуровневый код опирается на абстракцию, а не на деталь.

```python
from abc import ABC, abstractmethod

class Database(ABC):
    @abstractmethod
    def save(self, order): ...

class PostgresDatabase(Database):
    def save(self, order):
        return f"saved {order} to postgres"

class OrderProcessor:
    def __init__(self, db: Database):
        self._db = db

    def process(self, order):
        return self._db.save(order)
```

`OrderProcessor` ничего не знает о конкретной СУБД — подменить её можно, не трогая его код.

## Подводные камни

- SOLID — ориентир, а не догма. Слепое дробление ради «одной ответственности» плодит десятки крошечных классов и усложняет навигацию.
- Нарушение Лисков часто прячется в «is-a», которое верно в быту, но ложно в коде (квадрат как наследник прямоугольника).
- Инверсию зависимостей путают с внедрением зависимостей: DIP — про направление связей на абстракции, DI — лишь способ их передать.
- Принципы пересекаются: исправляя один, легко зацепить другой, поэтому оценивайте дизайн целиком, а не по галочкам.
