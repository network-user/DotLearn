# Singleton: how to keep exactly one instance of a class

## In short

Singleton guarantees that a class will have no more than one instance and provides a single point of access to it. It's used where extra copies are harmful or wasteful: configuration, a logger, a connection pool.

## How to answer in an interview

Singleton is a creational pattern that restricts a class to exactly one instance and opens up global access to it. In Python it's usually done by overriding `__new__`: the first call creates the object and caches it in a class attribute, and subsequent calls return the same object. An important caveat: global state complicates testing and, in a multithreaded environment, requires a lock. Often, instead of the classic Singleton, Python developers use a module (which is already one per process) or dependency injection.

```python
class Singleton:
    _instance = None

    def __new__(cls, *args, **kwargs):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance


a = Singleton()
b = Singleton()
print(a is b)  # True
```

Breakdown:

- `_instance` — a class-level attribute that holds the single object.
- `__new__` is responsible for allocating the object; we intercept it to return the cache.
- `cls._instance is None` — we check whether an instance has already been created; if not, we create one via `super().__new__(cls)`.

An analogy: one shared printer in an office. Everyone prints on it instead of dragging their own around — less chaos and lower cost.

## Pitfalls

- `__new__` will return the same object, but `__init__` will be called on **every** `Singleton()` invocation. If `__init__` contains state initialization, that state will be overwritten — move it behind a flag or into `__new__`.
- In a multithreaded environment, the naive `if _instance is None` check is a race: two threads may create two objects. You need a lock (`threading.Lock`).
- Singleton is global mutable state, so it hampers test isolation and is often considered a "smell". Consider a module singleton or DI as an alternative.
- Inheriting from Singleton behaves non-obviously: all subclasses share the same `_instance` unless you declare it separately in each subclass.
