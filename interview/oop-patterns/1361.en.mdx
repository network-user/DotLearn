# Why pass dependencies in from outside instead of creating them inside the class?

## In short

Dependency Injection is supplying an object's dependencies from the outside instead of constructing them inside. Coupling between components is loosened, and the code becomes easier to test and maintain.

## How to answer in an interview

DI is an implementation of the Inversion of Control principle: a class doesn't create the objects it needs itself, but receives them ready-made - most often through the constructor, less often via a setter or a method. As a result, the class depends on an abstraction rather than a concrete implementation, so in tests the real dependency is easily swapped for a mock, and in production you can replace the implementation without touching the dependent code. This is a direct consequence of the D in SOLID (Dependency Inversion).

The dependency is passed as a constructor parameter and stored in a field. The class uses it without knowing anything about how it was created.

```python
class Database:
    def connect(self):
        return "Connected to the database"


class Service:
    def __init__(self, database: Database):
        self.database = database

    def perform_action(self):
        return self.database.connect()


db = Database()
service = Service(db)
print(service.perform_action())
```

`Service` does not call `Database()` inside itself - the object arrives from outside. In a test, instead of a real database you can substitute a stub with the same interface:

```python
class FakeDatabase:
    def connect(self):
        return "fake"


assert Service(FakeDatabase()).perform_action() == "fake"
```

## Pitfalls

- DI doesn't necessarily mean a framework. Simply passing an argument into the constructor is already dependency injection; DI containers only solve the problem of automatically wiring the object graph.
- Overdoing injection bloats constructors: a long list of dependencies is a signal that the class does too much (an SRP violation).
- The dependency should be specified through a shared interface/protocol, otherwise loose coupling is lost - you won't be able to swap the implementation.
- A service locator (an object that pulls its dependency from a global registry itself) is often confused with DI, but it does the opposite - it hides dependencies and makes testing harder.
