# Encapsulation: why data is hidden behind methods

## In short

Encapsulation bundles data and the methods that work with it into a single object and closes off direct access to its internals. State can only be changed through the intended interface, so the object stays in a consistent state.

## How to answer in an interview

Encapsulation is hiding an object's internal structure behind an interface of methods, so that from the outside you can't corrupt its state by bypassing the checks. Python has no strict access modifiers like Java: everything rests on conventions. A single underscore `_attr` means "you shouldn't touch this from the outside"; two underscores `__attr` enable name mangling, and the attribute becomes `_ClassName__attr`. This doesn't make it truly private; it only makes accidental access harder and protects against name collisions in subclasses.

Naming conventions define visibility levels:

- **Public**: no prefix, accessible from anywhere.
- **Protected**: a single underscore `_balance` signaling that "this is internal".
- **Private**: two underscores `__balance`, which Python renames to `_ClassName__balance` (name mangling).

```python
class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner
        self.__balance = balance

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("amount must be positive")
        self.__balance += amount
        return self.__balance

    def withdraw(self, amount):
        if not 0 < amount <= self.__balance:
            raise ValueError("invalid amount")
        self.__balance -= amount
        return self.__balance

    @property
    def balance(self):
        return self.__balance
```

From the outside, the only way to change `__balance` is through `deposit`/`withdraw`, which guarantee the checks. For reading it's convenient to expose a `@property`: the interface looks like an attribute, but a method stands behind it.

## Pitfalls

- `__balance` isn't truly private. You can still reach it via `account._BankAccount__balance`; name mangling protects against accidents, not against malice.
- The double underscore is meant to prevent name collisions in an inheritance hierarchy, not as "security". A single underscore is usually enough for "don't touch from the outside".
- In Python, getters/setters for every attribute are an anti-pattern. If there's no logic, leave a public attribute and switch to `@property` only when you need validation or computation.
- A name with two underscores on both sides (`__init__`) has nothing to do with encapsulation: those are dunders, and mangling doesn't apply to them.
