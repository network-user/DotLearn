# Factory Method: how to delegate the choice of class to subclasses

## In short

Factory Method is a creational pattern: the superclass declares an interface for creating an object, while the concrete type is chosen by subclasses. The code that needs the object does not know its exact class.

## How to answer in an interview

Instead of calling the constructor directly, the code calls a factory method that returns an object of the required type. Subclasses override this method and substitute the class being created. The benefit is loose coupling and extensibility: a new type is added as a separate subclass, while the consumer stays untouched (hello, Open/Closed principle). It is important not to confuse the "factory method" with a plain "switch factory": the classic pattern moves the choice specifically into an overridable subclass method.

The base class defines an abstract product interface, and the subclasses implement the specifics.

```python
from abc import ABC, abstractmethod

class Transport(ABC):
    @abstractmethod
    def deliver(self):
        ...

class Car(Transport):
    def deliver(self):
        return "Delivering by car"

class Bike(Transport):
    def deliver(self):
        return "Delivering by bike"
```

Next, object creation is hidden behind a method. In the canonical form, creator subclasses override it.

```python
class Logistics(ABC):
    @abstractmethod
    def create_transport(self) -> Transport:
        ...

    def plan_delivery(self):
        transport = self.create_transport()
        return transport.deliver()

class RoadLogistics(Logistics):
    def create_transport(self):
        return Car()

class BikeLogistics(Logistics):
    def create_transport(self):
        return Bike()

print(RoadLogistics().plan_delivery())   # Delivering by car
print(BikeLogistics().plan_delivery())   # Delivering by bike
```

The `plan_delivery` method works with the abstract `Transport` and does not depend on which exact class a subclass creates. To add a truck, a new `Logistics` subclass is enough - we do not touch the existing code.

## Pitfalls

- Do not confuse it with Abstract Factory: the factory method produces a single product, while an abstract factory produces a family of related products.
- A plain `if/elif` inside a single method is a simplified "parameterized factory". The classic Factory Method moves the decision into an overridable subclass method.
- The pattern is justified when the exact class is unknown until runtime or the system must grow through subclasses. For two stable types it may be an unnecessary complication.
- The returned products must share a common interface, otherwise polymorphism breaks down for the consumer.
