# Singleton: как удержать ровно один экземпляр класса

## Суть

Singleton («Одиночка») гарантирует, что у класса будет не больше одного экземпляра, и даёт к нему единую точку доступа. Применяют там, где лишние копии вредны или расточительны: конфигурация, логгер, пул соединений.

## Что ответить на собесе

Singleton — порождающий паттерн, который ограничивает класс ровно одним экземпляром и открывает к нему глобальный доступ.

- В Python его обычно делают через переопределение `__new__`: первый вызов создаёт объект и кэширует его в атрибуте класса, последующие возвращают тот же объект.
- Оговорка: глобальное состояние усложняет тестирование и в многопоточной среде требует блокировки.
- Часто вместо классического Singleton в Python берут модуль (он и так один на процесс) или внедрение зависимостей.

## Как это работает

Класс сам контролирует создание единственного объекта. Реализация через `__new__`:

```python
class Singleton:
    _instance = None

    def __new__(cls, *args, **kwargs):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance


a = Singleton()
b = Singleton()
print(a is b)  # True
```

Разбор:

- `_instance` — атрибут уровня класса, хранит единственный объект.
- `__new__` отвечает за выделение объекта; мы перехватываем его, чтобы вернуть кэш.
- `cls._instance is None` — проверяем, создан ли уже экземпляр; если нет, создаём через `super().__new__(cls)`.

Аналогия: один общий принтер в офисе. Все печатают на нём, а не таскают свой, — меньше хаоса и затрат.

## Подводные камни

- `__new__` вернёт один и тот же объект, но `__init__` будет вызываться при **каждом** обращении `Singleton()`. Если в `__init__` есть инициализация состояния, оно перезатрётся — выносите её под флаг или в `__new__`.
- В многопоточной среде наивная проверка `if _instance is None` — гонка: два потока могут создать два объекта. Нужна блокировка (`threading.Lock`).
- Singleton — это глобальное изменяемое состояние, поэтому он мешает изоляции тестов и часто считается «запахом». Рассмотрите модуль-синглтон или DI как альтернативу.
- Наследование от Singleton ведёт себя неочевидно: все подклассы делят один и тот же `_instance`, если не объявить его в каждом подклассе отдельно.
