# The __call__ method: how to make an instance a callable object

## In short

`__call__` makes an instance of a class callable: under the hood, `obj(...)` turns into `obj.__call__(...)`. This lets an object behave like a function while still remembering its state between calls.

## How to answer in an interview

In Python, callability is simply the presence of a `__call__` method. Every function is callable because it has this method; conversely, an ordinary object becomes callable if you define `__call__` in its class. Python unfolds `obj(5)` into `type(obj).__call__(obj, 5)`. This is handy for function-objects that hold configuration or accumulated state: function factories, decorator classes, configurable predicates, models that you "call" to get a prediction. You can check callability with `callable(obj)`.

## How it works

```python
class Multiplier:
    def __init__(self, factor):
        self.factor = factor

    def __call__(self, value):
        return value * self.factor


double = Multiplier(2)
print(double(5))          # 10
print(double.__call__(5)) # the same
print(callable(double))   # True
```

The constructor fixes `factor`, and `__call__` uses it on every call. The `double` object both stores state (`factor`) and is called like a function - that is what sets it apart from an ordinary function, where state would have to be carried through a closure or globals.

## Pitfalls

- **`__call__` is looked up on the type, not on the instance.** Assigning `obj.__call__ = lambda ...` is not enough: when you call `obj()`, Python looks at `type(obj).__call__`, so the method must be declared in the class.
- **`callable()` only checks for the method's presence, not the correctness of the signature.** An object can be `callable` yet fail when called because of wrong arguments.
- **Do not confuse `__call__` with `__init__`.** `__init__` fires once at creation, `__call__` fires every time you call an already-built object.
- **Do not overuse it.** If an object needs a single operation, a plain function or closure is often clearer; `__call__` is justified when state, configuration, or a unified "call" interface matters.
