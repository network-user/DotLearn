# State: объект меняет поведение, делегируя его текущему состоянию

## Суть

State превращает каждое состояние объекта в отдельный класс и поручает ему обработку действий. Контекст не разветвляет логику через `if/else`, а делегирует её текущему объекту-состоянию, который умеет переключать контекст на следующее состояние.

## Что ответить на собесе

Поведенческий паттерн: контекст хранит ссылку на объект-состояние и переадресует ему вызовы.

- У всех состояний общий интерфейс, а сами переходы (кто за кем следует) задаются внутри состояний.
- Это убирает громоздкие условные операторы, проверяющие "в каком я сейчас режиме", и делает добавление нового состояния локальным изменением - новым классом.
- По структуре похож на Strategy, но смысл другой: State моделирует конечный автомат и сам управляет переходами, а Strategy просто подменяет алгоритм по воле клиента.

## Как это работает

Каждое состояние реализует интерфейс и возвращает следующее состояние. Контекст лишь хранит текущее и переприсваивает его.

def dispense(self, machine):
print("Напиток выдан. Ждём следующую монету.")
return WaitingForCoin()

class VendingMachine:
def **init**(self):
self.state = WaitingForCoin()

    def insert_coin(self):
        self.state = self.state.insert_coin(self)

    def select_drink(self):
        self.state = self.state.select_drink(self)

        self.state = self.state.dispense(self)

machine.select_drink()

## Подводные камни

- Разнесённые по классам переходы дают полную картину автомата только если собрать их вместе мысленно - навигация по коду усложняется.
- Состояния часто нуждаются в доступе к данным контекста; передавайте контекст в методы или ссылку на него, не дублируя данные.
- Если состояний всего два-три и переходы тривиальны, обычный `enum` плюс `match`/`if` могут оказаться проще паттерна.
- Не путайте со Strategy: одинаковая структура, но State про конечный автомат и самостоятельные переходы, Strategy про взаимозаменяемые алгоритмы по выбору клиента.
