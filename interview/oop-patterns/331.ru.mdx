# Observer: как объекты подписываются на изменения чужого состояния

## Суть

Observer - поведенческий паттерн, который строит связь "один источник - много подписчиков". Источник сам рассылает уведомления, когда его состояние меняется, а подписчики при этом ничего не знают друг о друге и слабо связаны с источником.

## Что ответить на собесе

Observer решает задачу: как оповестить набор объектов об изменении состояния другого объекта, не завязывая их жёстко.

- Издатель (Subject) хранит список подписчиков (Observers) и при каждом изменении проходит по списку и дёргает у каждого согласованный метод вроде `update`, а подписчики реагируют сами.
- Источник зависит не от конкретных классов, а от общего интерфейса наблюдателя, поэтому добавлять и убирать получателей можно на лету, не трогая код издателя.
- Классические места применения - GUI, шины событий, слой Model в MVC.

## Как это работает

Участников двое. **Subject** держит состояние и список наблюдателей, умеет подписать (`attach`), отписать (`detach`) и разослать уведомление (`notify`). **Observer** - это контракт с методом `update`, через который приходит сигнал.

<PyDemo title="Observer: рассылка подписчикам" autoRun code={`class Subject:
    def __init__(self):
        self._observers = []
        self._state = None

    def attach(self, observer):
        self._observers.append(observer)

    def detach(self, observer):
        self._observers.remove(observer)

    def notify(self):
        for observer in self._observers:
            observer.update(self._state)

    def set_state(self, state):
        self._state = state
        self.notify()


class Observer:
    def update(self, state):
        raise NotImplementedError


class PrintObserver(Observer):
    def update(self, state):
        print(f"got new state: {state}")


subject = Subject()
subject.attach(PrintObserver())
subject.attach(PrintObserver())
subject.set_state("ready")`} />

Ключевой момент - `set_state` не просто меняет поле, а сразу вызывает `notify`, который перебирает подписчиков и вызывает у каждого `update`. Издатель работает через общий интерфейс `Observer`, а не через конкретные классы наблюдателей, и в этом вся слабая связанность.

## Подводные камни

- **Утечки и "висячие" подписчики.** Если объект подписался и забыл отписаться через `detach`, Subject продолжит держать на него ссылку - сборщик мусора его не освободит. Для долгоживущих издателей стоит держать наблюдателей через `weakref`.
- **Порядок и побочные эффекты.** Наблюдатели вызываются по очереди; если один из них в `update` меняет состояние Subject, можно запустить повторный `notify` и поймать рекурсию или гонку.
- **Исключение в одном `update`** прервёт рассылку остальным. Часто стоит ловить ошибки внутри цикла рассылки.
- **Не путать с pub/sub через брокера.** В классическом Observer издатель знает своих подписчиков напрямую; в pub/sub между ними стоит посредник-канал.
