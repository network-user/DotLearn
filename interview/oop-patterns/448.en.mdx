# How a request travels through a line of handlers (Chain of Responsibility)

## In short

A request moves through a group of handlers arranged in a row. Each link decides for itself: take the task on or pass it further. Meanwhile the sender does not know which handler will actually process the request.

## How to answer in an interview

Chain of Responsibility is a behavioral pattern. I arrange the handlers into a linked list where each one has a reference to the next. The request walks down the chain until someone accepts it. This decouples the sender from the receiver: I can add, remove, or reorder links without touching the client code. Classic uses are a validation pipeline, multi-level logging, and event handling in a UI.

## How it works

The pattern is built from three roles:

1. **Handler** — a base class with a processing method and a `successor` reference to the next link.
2. **Concrete Handler** — a concrete implementation: it either processes the request or delegates further.
3. **Client** — builds the request and starts it from the beginning of the chain.

```python
class Handler:
    def __init__(self, successor=None):
        self.successor = successor

    def handle(self, request):
        if self.successor:
            return self.successor.handle(request)
        return None


class AuthHandler(Handler):
    def handle(self, request):
        if request == "auth":
            return "AuthHandler handled the request"
        return super().handle(request)


class PaymentHandler(Handler):
    def handle(self, request):
        if request == "payment":
            return "PaymentHandler handled the request"
        return super().handle(request)


chain = AuthHandler(PaymentHandler())

print(chain.handle("payment"))  # PaymentHandler handled the request
print(chain.handle("auth"))     # AuthHandler handled the request
print(chain.handle("unknown"))  # None
```

Note the `super().handle(request)` call: the shared pass-it-on code lives in the base class, while concrete links describe only their own share of the work.

## Pitfalls

- **A request can "fall through" the entire chain** and stay unhandled. Decide in advance what to do in that case: return `None`, raise an exception, or have a fallback link at the end.
- **Debugging gets harder**: the request's path is not visible from a single point. It helps to log which link accepted the request.
- **A long chain costs performance**: every miss is an extra call. Don't turn the pattern into a linear search where a dictionary would do.
- **Order of links matters**: the first matching link takes the request, so more specific handlers usually go before general ones.
