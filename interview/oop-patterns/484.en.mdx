# Why the Bridge separates abstraction and implementation into different hierarchies

## In short

The Bridge splits the abstraction and its implementation into two independent hierarchies linked by composition. This defuses the combinatorial explosion of classes when there are many variants of both the abstraction and the implementation, and lets you change them separately.

## How to answer in an interview

The Bridge is a structural pattern. When there are two independent dimensions of variation (say, the shape type and the rendering platform), inheritance would force you to create a class for every pair - the count grows as a product. I move one dimension into an abstraction hierarchy and the other into an implementation hierarchy, and connect them through composition: the abstraction holds a reference to an implementation object and delegates the low-level calls to it. Now both dimensions can be extended independently, without touching each other's code.

## How it works

The abstraction holds a reference to the implementation and works with it through an interface. A concrete abstraction adds high-level logic; a concrete implementation adds low-level details.

```python
class DrawingAPI:
    def draw_circle(self, x, y, radius):
        pass

class WindowsDrawingAPI(DrawingAPI):
    def draw_circle(self, x, y, radius):
        print(f"Drawing circle on Windows at ({x}, {y}) with radius {radius}")

class MacOSDrawingAPI(DrawingAPI):
    def draw_circle(self, x, y, radius):
        print(f"Drawing circle on macOS at ({x}, {y}) with radius {radius}")

class Shape:
    def __init__(self, drawing_api):
        self.drawing_api = drawing_api

    def draw(self):
        pass

class Circle(Shape):
    def __init__(self, x, y, radius, drawing_api):
        super().__init__(drawing_api)
        self.x = x
        self.y = y
        self.radius = radius

    def draw(self):
        self.drawing_api.draw_circle(self.x, self.y, self.radius)

circle1 = Circle(1, 2, 3, WindowsDrawingAPI())
circle2 = Circle(4, 5, 6, MacOSDrawingAPI())

circle1.draw()
circle2.draw()
```

Adding a new shape means one class in the `Shape` hierarchy; adding a new platform means one class in the `DrawingAPI` hierarchy. An everyday analogy: a remote (the abstraction) and a TV (the implementation) evolve separately, and one remote can control different models.

## Pitfalls

- The Bridge is easy to confuse with the Adapter. The Bridge is designed up front to separate two dimensions; the Adapter is bolted on after the fact to reconcile already existing incompatible interfaces.
- The pattern is justified only when there are two genuinely independent axes of variation. If there's just one dimension, the Bridge adds an extra layer of indirection with no payoff.
- The implementation must be passed in from the outside (dependency injection); otherwise the abstraction is hard-wired to a concrete class and the whole point of the separation is lost.
- Don't confuse it with ordinary composition for reuse: the key marker of the Bridge is precisely two parallel hierarchies that are extended independently.
