# Что делают **enter** и **exit** и как работает блок with?

## Суть

`__enter__` и `__exit__` - это протокол контекстного менеджера, который обслуживает конструкцию `with`. `__enter__` вызывается на входе в блок и обычно отдаёт ресурс, `__exit__` - на выходе и гарантированно его освобождает. Главное достоинство: `__exit__` отрабатывает даже если внутри блока вылетело исключение.

## Что ответить на собесе

Объект становится контекстным менеджером, если у него есть `__enter__` и `__exit__`.

- При входе в `with` Python зовёт `__enter__`, а его результат привязывается к имени после `as`.
- При выходе - неважно, штатном или из-за исключения - Python вызывает `__exit__(exc_type, exc_value, traceback)`; если внутри было исключение, эти три аргумента описывают его, иначе они `None`.
- Так управление ресурсом не зависит от того, как мы покинули блок, поэтому файл или соединение точно закроется.

`with` гарантирует освобождение ресурса. Поэтому код думает о логике, а не о закрытии:

```python
class FileManager:
    def __init__(self, filename, mode):
        self.filename = filename
        self.mode = mode
        self.file = None

    def __enter__(self):
        self.file = open(self.filename, self.mode)
        return self.file

    def __exit__(self, exc_type, exc_value, traceback):
        if self.file:
            self.file.close()

with FileManager('example.txt', 'w') as f:
    f.write('Hello, World!')
```

По шагам:

1. `__enter__` открывает файл и возвращает его - объект доступен как `f`.
2. Выполняется тело блока.
3. `__exit__` закрывает файл, и это происходит при любом выходе из блока, в том числе после ошибки.

Типичные применения протокола: файлы (открыть/закрыть), сетевые соединения, транзакции БД (commit/rollback), блокировки в многопоточности (захват/освобождение лока).

## Подводные камни

- Возвращаемое значение `__exit__` управляет исключением. Если вернуть истинное значение, исключение из блока подавляется. Обычно возвращают `None` (то есть ложь), чтобы ошибка пробрасывалась дальше после очистки.
- `__enter__` и `with` - не одно и то же. Имя после `as` получает именно результат `__enter__`, а не сам менеджер. В примере выше `f` - это файл, а не объект `FileManager`.
- Не глушите ошибки молча. Перехватывать исключение в `__exit__` стоит осознанно, иначе баги уходят в тишину.
- Для простых случаев писать класс необязательно: `contextlib.contextmanager` позволяет сделать менеджер из генератора с одним `yield`, где код до `yield` - это `__enter__`, а после - `__exit__`.
