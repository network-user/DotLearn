# Factory Method: how to hand object creation to subclasses

## In short

The Factory Method is a creational pattern: a base class declares a "factory" method, while the decision about which concrete class to return is made by subclasses. This keeps the logic that works with the product from being tied to its concrete type.

## How to answer in an interview

The Factory Method moves object creation into a separate method that descendants override. The base class writes its code in terms of the product's common interface and calls `factory_method()` without knowing which concrete product subclass it will get. As a result, adding a new product type comes down to a new "product + creator" pair without changing existing code - a direct application of the open/closed principle.

## How it works

We declare an abstract product and an abstract creator with a factory method. Concrete creators override the factory and return a concrete product. All the rest of the creator's logic works through the common interface.

```python
from abc import ABC, abstractmethod

class Transport(ABC):
    @abstractmethod
    def deliver(self) -> str:
        ...

class Car(Transport):
    def deliver(self) -> str:
        return "Delivery by car"

class Bike(Transport):
    def deliver(self) -> str:
        return "Delivery by bike"

class Logistics(ABC):
    @abstractmethod
    def create_transport(self) -> Transport:
        ...

    def plan_delivery(self) -> str:
        transport = self.create_transport()
        return transport.deliver()

class RoadLogistics(Logistics):
    def create_transport(self) -> Transport:
        return Car()

class CityLogistics(Logistics):
    def create_transport(self) -> Transport:
        return Bike()

print(RoadLogistics().plan_delivery())  # Delivery by car
print(CityLogistics().plan_delivery())  # Delivery by bike
```

The `plan_delivery` method mentions neither `Car` nor `Bike` - it works with the `Transport` abstraction. The specifics are set only by the overridden `create_transport`.

The pattern fits when:

- it makes sense to delegate the decision about the created object's class to subclasses;
- you want to extend the system with new types without touching working code;
- you need to loosen the code's coupling to concrete classes and work through interfaces.

## Pitfalls

- Don't confuse it with the Abstract Factory: the Factory Method produces a single product through method overriding, while the Abstract Factory produces a whole family of related products through a factory object.
- The pattern is justified when there are several product variants and they change. For one or two stable types it adds an extra hierarchy for the sake of hierarchy.
- In Python a simple factory function or passing the class itself as an argument is often enough - a full creator hierarchy is not always needed.
- The creator and the product must evolve in sync: the product's common interface is the contract the creator's logic relies on.
