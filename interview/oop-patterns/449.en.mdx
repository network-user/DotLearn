# Patterns that manage object interaction (behavioral)

## In short

Behavioral patterns describe how objects distribute responsibility and exchange messages. Their concern is algorithms and the connections between components, not how objects are created or composed.

## How to answer in an interview

This is one of the three groups of GoF patterns, alongside creational and structural ones. Creational patterns handle object creation, structural ones handle their composition, and behavioral ones handle communication and the distribution of responsibilities at runtime. They reduce coupling: objects communicate through abstractions rather than directly. Among the most common, I'd name Observer, Strategy, and Command.

## How it works

Let me walk through three members of the group.

**Observer** — a subject object broadcasts notifications to subscribers when its state changes. Subscribers don't poll the subject; they receive a push.

```python
class Subject:
    def __init__(self):
        self._observers = []

    def attach(self, observer):
        self._observers.append(observer)

    def notify(self, message):
        for observer in self._observers:
            observer.update(message)

class Observer:
    def update(self, message):
        print(f"Received: {message}")

subject = Subject()
subject.attach(Observer())
subject.attach(Observer())
subject.notify("Weather: sunny")
```

**Strategy** — a family of interchangeable algorithms behind a single interface. The context holds a reference to a strategy and swaps it on the fly.

```python
class SortStrategy:
    def sort(self, data):
        raise NotImplementedError

class AscendingSort(SortStrategy):
    def sort(self, data):
        return sorted(data)

class DescendingSort(SortStrategy):
    def sort(self, data):
        return sorted(data, reverse=True)

class Context:
    def __init__(self, strategy):
        self._strategy = strategy

    def set_strategy(self, strategy):
        self._strategy = strategy

    def run(self, data):
        return self._strategy.sort(data)

ctx = Context(AscendingSort())
print(ctx.run([5, 3, 8, 6]))   # [3, 5, 6, 8]
ctx.set_strategy(DescendingSort())
print(ctx.run([5, 3, 8, 6]))   # [8, 6, 5, 3]
```

**Command** — an action is turned into an object. Requests can be queued, logged, and undone.

```python
class Command:
    def execute(self):
        raise NotImplementedError

class CopyCommand(Command):
    def execute(self):
        print("Copying")

class Editor:
    def __init__(self):
        self._history = []

    def run(self, command):
        self._history.append(command)
        command.execute()

Editor().run(CopyCommand())
```

## Pitfalls

- **Don't confuse the pattern groups.** Singleton and Factory are creational, Adapter and Proxy are structural; calling them behavioral is a mistake.
- **Observer easily becomes a memory leak**: the subject holds references to its subscribers. Don't forget to unsubscribe (`detach`).
- **Strategy and State are structurally similar**, but differ in intent: Strategy selects an algorithm, State changes behavior based on internal state.
- **A pattern for the pattern's sake is an antipattern.** Behavioral patterns are justified when the interaction is genuinely complex; for a simple method call they're overkill.
