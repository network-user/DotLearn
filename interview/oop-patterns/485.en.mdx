# How the Adapter reconciles incompatible interfaces without changing their code

## In short

An adapter is a bridge between two incompatible interfaces. It wraps a "foreign" object and translates the calls into the format the client expects, without changing either side.

## How to answer in an interview

The Adapter is a structural pattern. When I have a ready-made class with one interface but the client expects another, I rewrite neither. Instead I create a wrapper class that implements the expected interface and internally holds a reference to the adaptee, translating calls into its methods. This is a lifesaver when the original code can't be touched - a third-party library, legacy code, or a changed external API.

## How it works

The adapter implements the target interface and delegates the work to the wrapped object, renaming and, where needed, transforming the calls.

```python
class OldPrinter:
    def print_text(self, text):
        print(f"Old Printer: {text}")

class NewPrinterInterface:
    def print(self, text):
        pass

class PrinterAdapter(NewPrinterInterface):
    def __init__(self, old_printer):
        self.old_printer = old_printer

    def print(self, text):
        self.old_printer.print_text(text)

old_printer = OldPrinter()
adapter = PrinterAdapter(old_printer)
adapter.print("Hello, World!")
```

`PrinterAdapter` exposes the `print()` method the new code expects and internally calls `print_text()` on the old printer. Now `OldPrinter` can be plugged in wherever a `NewPrinterInterface` is required. An everyday analogy: an adapter between an HDMI console and an analog TV.

## Pitfalls

- Don't confuse it with the Facade: a Facade simplifies access to an entire subsystem with a new interface, whereas an Adapter reconciles one specific incompatible interface with the expected one.
- An adapter only redirects calls - it shouldn't drag business logic into itself. If nontrivial rules appear inside, it's no longer an adapter.
- With complex data transformations, an adapter easily bloats and becomes a home for hidden bugs: keep the translation thin.
- There's the object adapter (via composition, as in the example) and the class adapter (via multiple inheritance). In Python the object adapter is more common - it's more flexible and doesn't depend on whether the adaptee can be subclassed.
