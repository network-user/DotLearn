# Зачем нужен super() и как он ведёт себя при множественном наследовании

## Суть

`super()` даёт доступ к методам предков, не называя их класс явно. При множественном наследовании он обходит иерархию по порядку разрешения методов (MRO), поэтому каждый класс инициализируется ровно один раз и без дублей.

## Что ответить на собесе

`super()` возвращает прокси, который ищет метод не в текущем классе, а дальше по цепочке MRO.

- Это убирает жёсткую привязку к имени родителя - иерархию можно перестраивать без правки вызовов.
- При множественном наследовании `super()` идёт не «к родителю», а к следующему классу в линеаризованном MRO текущего объекта.
- Поэтому цепочка кооперативных `super().__init__()` обходит всех предков по одному разу, а сам порядок строит алгоритм C3-линеаризации.

## Как это работает

`super()` опирается на MRO - список классов, по которому Python ищет атрибуты и методы. Важно: следующий класс определяется по MRO конкретного объекта, а не по статической ссылке «родитель».

```python
class A:
    def __init__(self):
        print("Инициализация A")

class B(A):
    def __init__(self):
        super().__init__()
        print("Инициализация B")

class C(A):
    def __init__(self):
        super().__init__()
        print("Инициализация C")

class D(B, C):
    def __init__(self):
        super().__init__()
        print("Инициализация D")

d = D()
```

Хотя `B.__init__` зовёт `super().__init__()`, для объекта типа `D` это уходит не в `A`, а в `C` - следующий по MRO. Поэтому вывод такой:

```
Инициализация A
Инициализация C
Инициализация B
Инициализация D
```

Порядок задаёт MRO, его видно через `mro()`:

```python
print([cls.__name__ for cls in D.mro()])
# ['D', 'B', 'C', 'A', 'object']
```

Линеаризация гарантирует: `A` инициализируется один раз, несмотря на два пути к нему (через `B` и через `C`).

## Подводные камни

- Распространённая ошибка - думать, что `super()` всегда зовёт «прямого родителя». В ромбовидной иерархии он зовёт следующий класс по MRO объекта, а это может быть «сосед», а не предок.
- Кооперативность требует дисциплины: если хоть один класс в цепочке не вызовет `super().__init__()`, часть инициализации просто не выполнится.
- При множественном наследовании несовместимые сигнатуры `__init__` ломают цепочку. Часто спасают `*args, **kwargs`, аккуратно пробрасываемые дальше.
- Если C3 не может построить непротиворечивый порядок (циклы, конфликт базовых классов), Python бросит `TypeError` ещё при объявлении класса.
