# How to reuse shared state across many objects (the Flyweight pattern)

## In short

Flyweight saves memory when there are very many objects. It splits state into shared (intrinsic) and unique (extrinsic) parts, stores the shared part in a single reusable instance, and passes the unique part in from the outside at call time.

## How to answer in an interview

Flyweight is a structural pattern for cases where thousands or millions of objects differ only in a small part of their data. I extract the immutable shared part (intrinsic state) into one flyweight object and share it across all of them, while the mutable unique part (extrinsic state) is not stored inside but passed as an argument at use time. A factory with a cache guarantees that only one flyweight is created per identical intrinsic state.

## How it works

1. **Split the state** into intrinsic (shared, immutable) and extrinsic (unique to each use).
2. **Build the flyweight**: an object that carries only the intrinsic state.
3. **Pass the extrinsic state** into methods from the outside, so one flyweight serves many calls.
4. **Cache flyweights in a factory**: a repeat request with the same parameters returns the existing object.

```python
class CharacterFlyweight:
    def __init__(self, font, size, color):
        self.font = font
        self.size = size
        self.color = color

    def display(self, position):
        print(f"Char at {position}: {self.font}/{self.size}/{self.color}")

class CharacterFactory:
    _flyweights = {}

    @classmethod
    def get(cls, font, size, color):
        key = (font, size, color)
        if key not in cls._flyweights:
            cls._flyweights[key] = CharacterFlyweight(font, size, color)
        return cls._flyweights[key]

a = CharacterFactory.get("Arial", 12, "black")
b = CharacterFactory.get("Arial", 12, "black")

print(a is b)          # True: the same flyweight
a.display((10, 20))    # position comes from outside
b.display((30, 40))
```

For a pair of identical parameters the factory returned the same object (`a is b` → `True`), while the position, as extrinsic state, came in as the `display` argument.

## Pitfalls

- **Intrinsic state must be immutable.** A flyweight is shared, so mutating it in one place affects everyone using it. Change only the extrinsic state.
- **The savings are illusory if there's a lot of unique state.** When there's almost no "shared" part, the flyweight doesn't pay off and only adds complexity.
- **Code complexity.** You introduce a factory, a cache, and state splitting; that's justified only with a truly large number of objects.
- **The factory's cache is long-lived.** It's a global dictionary; in a long-running process it can grow unchecked if there are many keys.
