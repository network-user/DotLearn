# Порты и адаптеры: как гексагональная архитектура изолирует ядро

## Суть

Гексагональная архитектура (она же порты и адаптеры) ставит между бизнес-логикой и внешним миром слой абстракций. Ядро не знает, откуда приходят данные и куда уходят - оно общается через порты-интерфейсы, а конкретные БД, API и UI подключаются адаптерами. Внешнюю часть можно заменить, не трогая логику.

## Что ответить на собесе

Идея Алистера Кокберна в том, чтобы бизнес-логика не зависела от деталей инфраструктуры.

- Ядро определяет порты - интерфейсы, через которые им пользуются (входные) и через которые оно само обращается наружу (выходные).
- Адаптеры - это реализации портов под конкретную технологию: SQL-репозиторий, REST-контроллер, консоль.
- Зависимости направлены внутрь: адаптеры знают про ядро, ядро про адаптеры - нет.
- За счёт этого логику тестируют без реальной БД (подставив фейковый адаптер), а смену хранилища или интерфейса делают без правок в ядре.

## Как это работает

Поток управления входит снаружи через входной адаптер, проходит через порт в ядро, а наружу (в БД) ядро выходит через выходной порт, реализованный другим адаптером.

Ядро зависит только от абстракции порта. Конкретный адаптер реализует её и подставляется снаружи.

```python
from abc import ABC, abstractmethod

class BookRepository(ABC):
    @abstractmethod
    def add_book(self, book): ...

    @abstractmethod
    def find_book(self, title): ...

class SQLBookRepository(BookRepository):
    def __init__(self, session):
        self._session = session

    def add_book(self, book):
        self._session.add(book)

    def find_book(self, title):
        return self._session.query(title)

class LibraryService:
    def __init__(self, repo: BookRepository):
        self._repo = repo

    def register(self, book):
        self._repo.add_book(book)
```

`LibraryService` - это ядро. Оно принимает `BookRepository` как порт и не знает, SQL это, файл или память. В тестах вместо `SQLBookRepository` подставляют фейк - и логика проверяется изолированно.

## Подводные камни

- Это не самоцель. Для CRUD-приложения из трёх эндпоинтов слой портов и адаптеров - оверинжиниринг и лишние интерфейсы.
- Направление зависимостей легко нарушить: если в ядро протекают типы ORM или HTTP-запроса, изоляция сломана, и весь смысл теряется.
- Порты определяет ядро, а не инфраструктура. Интерфейс должен выражать потребность бизнес-логики, а не повторять API конкретной библиотеки.
- Путают с луковичной/чистой архитектурой. Идея инверсии зависимостей у них общая, но гексагон делает акцент именно на симметрии входных и выходных адаптеров.
