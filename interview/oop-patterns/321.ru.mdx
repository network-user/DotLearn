# Visitor: как добавлять операции, не вскрывая классы объектов

## Суть

Visitor выносит операции над объектами в отдельный класс-посетитель, оставляя сами объекты нетронутыми. Так алгоритм отделяется от данных: новую операцию добавляют, не правя классы элементов.

## Что ответить на собесе

Это поведенческий паттерн с двумя сторонами: иерархия элементов с методом `accept` и иерархия посетителей с методом `visit` под каждый тип элемента.

- Элемент в `accept` вызывает у посетителя нужный `visit`, передавая `self` (это называется двойной диспетчеризацией).
- Добавить новую операцию - значит просто написать ещё один класс-посетитель, а не дописывать каждый класс элемента.
- Платим тем, что новый тип элемента ломает всех существующих посетителей.
- Поэтому Visitor уместен, когда набор типов стабилен, а операции над ними меняются часто.

Механика держится на двойной диспетчеризации: сначала вызов уходит в `accept` конкретного элемента, затем оттуда в конкретный метод посетителя. Какой именно метод сработает, определяется обоими типами разом.

```python
class DocumentElement:
    def accept(self, visitor):
        raise NotImplementedError

class Paragraph(DocumentElement):
    def accept(self, visitor):
        return visitor.visit_paragraph(self)

class Image(DocumentElement):
    def accept(self, visitor):
        return visitor.visit_image(self)

class Visitor:
    def visit_paragraph(self, paragraph):
        raise NotImplementedError

    def visit_image(self, image):
        raise NotImplementedError

class RenderVisitor(Visitor):
    def visit_paragraph(self, paragraph):
        return "render: paragraph"

    def visit_image(self, image):
        return "render: image"

class WordCountVisitor(Visitor):
    def visit_paragraph(self, paragraph):
        return 42

    def visit_image(self, image):
        return 0

elements = [Paragraph(), Image()]
renderer = RenderVisitor()
for element in elements:
    print(element.accept(renderer))
```

Чтобы добавить экспорт в PDF или подсчёт слов, мы создаём ещё один наследник `Visitor` - классы `Paragraph` и `Image` остаются как были.

## Подводные камни

- Главная цена: добавление нового типа элемента требует дописать метод во все классы-посетители. Если иерархия элементов растёт, Visitor становится обузой.
- Двойная диспетчеризация неочевидна: новички забывают возвращать результат из `accept` или вызывают не тот `visit`.
- Посетитель часто лезет в данные элемента, что подталкивает к публичным геттерам и слегка ослабляет инкапсуляцию.
- В Python вместо классического Visitor нередко достаточно `functools.singledispatch` или сопоставления по типу - это решает ту же задачу с меньшим количеством кода.
