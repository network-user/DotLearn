# Why a class needs multiple parents and what you pay for it

## In short

A class in Python is free to inherit from any number of parents at once. This assembles behavior from independent pieces, but in return it makes method lookup non-obvious: when parents overlap in their ancestors, you must strictly define whose implementation wins. Python computes this order with the C3 algorithm and calls it the MRO.

## How to answer in an interview

Multiple inheritance is listing several base classes in the parentheses when declaring a class. The main risk here is the diamond hierarchy: two parents share a common ancestor, and without rules it would be unclear which method to call. Python resolves this deterministically through C3 linearization, which builds a linear list of classes (the MRO). Attribute lookup goes through this list from left to right, and the first one that has the required method wins. You can view the order through `ClassName.mro()`.

## How it works

Parents are listed separated by commas. Lookup goes through the MRO, so on a name conflict the parent that appears earlier in the list wins.

```python
class Parent1:
    def greet(self):
        print("Hello from Parent1")

class Parent2:
    def greet(self):
        print("Hello from Parent2")

class Child(Parent1, Parent2):
    pass

Child().greet()  # Hello from Parent1
```

When the parents share a common ancestor, that very diamond shape appears:

```python
class A:
    def greet(self):
        print("A")

class B(A):
    def greet(self):
        print("B")

class C(A):
    def greet(self):
        print("C")

class D(B, C):
    pass

D().greet()  # B
```

C3 guarantees two properties: an ancestor never comes before a descendant, and the relative order of the parents repeats the order in which they were listed. The result can be seen directly:

```python
print(D.mro())
# [<class '__main__.D'>, <class '__main__.B'>, <class '__main__.C'>, <class '__main__.A'>, <class 'object'>]
```

When the parents do not overlap, multiple inheritance works as assembling abilities from mixins:

```python
class Flyable:
    def fly(self):
        print("Flying")

class Swimmable:
    def swim(self):
        print("Swimming")

class Duck(Flyable, Swimmable):
    pass

duck = Duck()
duck.fly()   # Flying
duck.swim()  # Swimming
```

## Pitfalls

- The order of parents matters: `class Child(Parent1, Parent2)` and `class Child(Parent2, Parent1)` will produce a different MRO and a different result on a name conflict.
- An inconsistent hierarchy won't compile at all: if C3 cannot build a correct linearization, Python raises a `TypeError` right at the class declaration.
- In cooperative hierarchies, call ancestors through `super()`, not by class name. A direct call `Parent.method(self)` breaks the MRO order and may skip some classes or call them twice.
- Multiple inheritance easily turns into a tangled web of dependencies. It is often cleaner to use composition or mixins with narrow responsibilities.
