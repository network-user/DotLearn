# Why super() exists and how it behaves under multiple inheritance

## In short

`super()` gives access to ancestor methods without naming their class explicitly. Under multiple inheritance it walks the hierarchy in method resolution order (MRO), so each class is initialized exactly once and without duplicates.

## How to answer in an interview

`super()` returns a proxy that looks for a method not in the current class but further down the MRO chain. This removes the hard dependency on the parent's name - the hierarchy can be rearranged without changing the calls. Under multiple inheritance the key point is that `super()` goes not "to the parent" but to the next class in the current object's linearized MRO. That's why a chain of cooperative `super().__init__()` calls visits every ancestor exactly once. The order itself is built by the C3 linearization algorithm.

## How it works

`super()` relies on the MRO - the list of classes Python searches for attributes and methods. Importantly: the next class is determined by the MRO of the concrete object, not by a static "parent" reference.

```python
class A:
    def __init__(self):
        print("Initializing A")

class B(A):
    def __init__(self):
        super().__init__()
        print("Initializing B")

class C(A):
    def __init__(self):
        super().__init__()
        print("Initializing C")

class D(B, C):
    def __init__(self):
        super().__init__()
        print("Initializing D")

d = D()
```

Although `B.__init__` calls `super().__init__()`, for an object of type `D` this goes not to `A` but to `C` - the next class in the MRO. So the output is:

```
Initializing A
Initializing C
Initializing B
Initializing D
```

The order is set by the MRO, which is visible through `mro()`:

```python
print([cls.__name__ for cls in D.mro()])
# ['D', 'B', 'C', 'A', 'object']
```

Linearization guarantees that `A` is initialized once, despite two paths reaching it (through `B` and through `C`).

## Pitfalls

- A common mistake is to think `super()` always calls the "direct parent". In a diamond hierarchy it calls the next class in the object's MRO, which may be a "sibling" rather than an ancestor.
- Cooperation requires discipline: if even one class in the chain fails to call `super().__init__()`, part of the initialization simply won't run.
- Under multiple inheritance, incompatible `__init__` signatures break the chain. The fix is often `*args, **kwargs`, carefully passed along.
- If C3 cannot build a consistent order (cycles, conflicting base classes), Python raises `TypeError` right at class declaration.
