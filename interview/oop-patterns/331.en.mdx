# Observer: how objects subscribe to changes in another object's state

## In short

Observer is a behavioral pattern that builds a "one source - many subscribers" relationship. The source itself broadcasts notifications when its state changes, while the subscribers know nothing about each other and are loosely coupled to the source.

## How to answer in an interview

Observer solves the problem: how to notify a set of objects about a change in another object's state without tightly coupling them. There is a publisher (Subject) that keeps a list of subscribers (Observers) and, on every change, walks the list and calls an agreed method like `update` on each one. The subscribers react on their own. The source depends not on concrete classes but on a common observer interface, so receivers can be added and removed on the fly without touching the publisher's code. Classic use cases are GUIs, event buses, and the Model layer in MVC.

## How it works

There are two participants. **Subject** holds the state and the list of observers; it can subscribe (`attach`), unsubscribe (`detach`), and broadcast a notification (`notify`). **Observer** is a contract with an `update` method through which the signal arrives.

```python
class Subject:
    def __init__(self):
        self._observers = []
        self._state = None

    def attach(self, observer):
        self._observers.append(observer)

    def detach(self, observer):
        self._observers.remove(observer)

    def notify(self):
        for observer in self._observers:
            observer.update(self._state)

    def set_state(self, state):
        self._state = state
        self.notify()


class Observer:
    def update(self, state):
        raise NotImplementedError


class PrintObserver(Observer):
    def update(self, state):
        print(f"got new state: {state}")


subject = Subject()
subject.attach(PrintObserver())
subject.attach(PrintObserver())
subject.set_state("ready")
```

The key point is that `set_state` does not just change a field - it immediately calls `notify`, which iterates over the subscribers and calls `update` on each. The publisher works through the common `Observer` interface rather than through concrete observer classes, and that is the whole point of loose coupling.

## Pitfalls

- **Leaks and "dangling" subscribers.** If an object subscribed and forgot to unsubscribe via `detach`, the Subject keeps holding a reference to it - the garbage collector will not free it. For long-lived publishers it is worth holding observers through `weakref`.
- **Order and side effects.** Observers are called one by one; if one of them changes the Subject's state inside `update`, it can trigger a repeated `notify` and run into recursion or a race.
- **An exception in one `update`** will interrupt the broadcast to the rest. It is often worth catching errors inside the broadcast loop.
- **Do not confuse it with pub/sub via a broker.** In classic Observer the publisher knows its subscribers directly; in pub/sub there is a channel mediator between them.
