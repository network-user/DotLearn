# Visitor: how to add operations without cracking open object classes

## In short

Visitor moves operations over objects into a separate visitor class, leaving the objects themselves untouched. This separates the algorithm from the data: you add a new operation without editing the element classes.

## How to answer in an interview

This is a behavioral pattern. It has two sides: a hierarchy of elements with an `accept` method and a hierarchy of visitors with a `visit` method for each element type. In `accept`, the element calls the appropriate `visit` on the visitor, passing `self` (this is called double dispatch). Thanks to this, adding a new operation means simply writing another visitor class rather than amending every element class. The price we pay is that a new element type breaks all existing visitors. So Visitor is appropriate when the set of types is stable but the operations over them change often.

The mechanics rest on double dispatch: first the call goes into the `accept` of the concrete element, then from there into the concrete visitor method. Which method exactly fires is determined by both types at once.

```python
class DocumentElement:
    def accept(self, visitor):
        raise NotImplementedError

class Paragraph(DocumentElement):
    def accept(self, visitor):
        return visitor.visit_paragraph(self)

class Image(DocumentElement):
    def accept(self, visitor):
        return visitor.visit_image(self)

class Visitor:
    def visit_paragraph(self, paragraph):
        raise NotImplementedError

    def visit_image(self, image):
        raise NotImplementedError

class RenderVisitor(Visitor):
    def visit_paragraph(self, paragraph):
        return "render: paragraph"

    def visit_image(self, image):
        return "render: image"

class WordCountVisitor(Visitor):
    def visit_paragraph(self, paragraph):
        return 42

    def visit_image(self, image):
        return 0

elements = [Paragraph(), Image()]
renderer = RenderVisitor()
for element in elements:
    print(element.accept(renderer))
```

To add export to PDF or word counting, we create another `Visitor` subclass — the `Paragraph` and `Image` classes stay as they were.

## Pitfalls

- The main price: adding a new element type requires adding a method to all visitor classes. If the element hierarchy grows, Visitor becomes a burden.
- Double dispatch is non-obvious: newcomers forget to return the result from `accept` or call the wrong `visit`.
- The visitor often reaches into the element's data, which pushes toward public getters and slightly weakens encapsulation.
- In Python, instead of the classic Visitor, `functools.singledispatch` or type-based matching is often enough — it solves the same task with less code.
