# Strategy: подменяем алгоритм во время выполнения через композицию

## Суть

Strategy прячет семейство взаимозаменяемых алгоритмов за общим интерфейсом и позволяет контексту выбирать и менять их в рантайме. Поведение задаётся объектом-стратегией, а не зашито в сам контекст.

## Что ответить на собесе

Поведенческий паттерн на композиции, а не наследовании. Контекст хранит ссылку на объект-стратегию и делегирует ему работу через единый метод. Любой алгоритм - это отдельная стратегия с тем же интерфейсом, поэтому их можно подменять во время выполнения, не трогая контекст. Главная выгода - убираем разрастающиеся `if/elif` по типу алгоритма и получаем открытость для расширения: новый вариант - новый класс. В Python стратегией часто служит просто функция, ведь функции - объекты первого класса.

## Как это работает

Контекст ничего не знает о конкретной реализации - только об интерфейсе `Strategy`. Подмена через `set_strategy` мгновенно меняет поведение.

```python
from abc import ABC, abstractmethod

class SortStrategy(ABC):
    @abstractmethod
    def sort(self, data):
        ...

class Ascending(SortStrategy):
    def sort(self, data):
        return sorted(data)

class Descending(SortStrategy):
    def sort(self, data):
        return sorted(data, reverse=True)

class Sorter:
    def __init__(self, strategy: SortStrategy):
        self._strategy = strategy

    def set_strategy(self, strategy: SortStrategy):
        self._strategy = strategy

    def run(self, data):
        return self._strategy.sort(data)

data = [5, 2, 9, 1]
sorter = Sorter(Ascending())
print(sorter.run(data))    # [1, 2, 5, 9]
sorter.set_strategy(Descending())
print(sorter.run(data))    # [9, 5, 2, 1]
```

В Python это нередко упрощают: стратегией становится обычная функция или `key`-аргумент, и отдельные классы не нужны.

```python
strategy = lambda d: sorted(d, reverse=True)
print(strategy([5, 2, 9, 1]))   # [9, 5, 2, 1]
```

## Подводные камни

- Strategy и State структурно близки (контекст делегирует объекту), но цель разная: Strategy выбирает алгоритм, State моделирует смену состояний и переходы между ними.
- Растёт число классов; для пары простых веток отдельные стратегии могут быть оверинжинирингом.
- Клиент обязан знать, какие стратегии существуют, чтобы выбрать подходящую - часть ответственности утекает наружу.
- В Python не делайте классы там, где хватит функции: лишний бойлерплейт - частая ошибка переноса паттерна "как в Java".
