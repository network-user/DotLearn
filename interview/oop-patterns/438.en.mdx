# How do you decouple a pile of objects that all hold direct references to each other?

## In short

Mediator removes objects' direct references to one another: they communicate through a single central object. Instead of an everyone-with-everyone web of links, you get a star with the mediator at the center.

## How to answer in an interview

Mediator is a behavioral pattern that fights tight coupling. The problem: when N components talk directly, the number of links grows quadratically, and any change drags edits through all the neighbors. The mediator introduces a single object that encapsulates the interaction protocol. Components no longer know about each other - they only know the mediator and send it notifications about events. The mediator decides whom to nudge and how in response. This lowers coupling and makes the interaction logic centralized and testable. The classic analogy is an airport controller: pilots do not talk to each other, everything goes through the tower.

## How it works

Components hold a reference to the mediator and call `notify` on an action. All the "what follows what" logic lives in one place - in the concrete mediator.

```python
class Mediator:
    def notify(self, sender, event):
        raise NotImplementedError


class ConcreteMediator(Mediator):
    def __init__(self):
        self.c1 = None
        self.c2 = None

    def notify(self, sender, event):
        if event == "A":
            self.c2.do_c()
        elif event == "D":
            self.c1.do_b()
            self.c2.do_c()


class BaseComponent:
    def __init__(self, mediator=None):
        self._mediator = mediator


class Component1(BaseComponent):
    def do_a(self):
        print("C1 does A")
        self._mediator.notify(self, "A")

    def do_b(self):
        print("C1 does B")


class Component2(BaseComponent):
    def do_c(self):
        print("C2 does C")

    def do_d(self):
        print("C2 does D")
        self._mediator.notify(self, "D")


mediator = ConcreteMediator()
mediator.c1 = Component1(mediator)
mediator.c2 = Component2(mediator)

mediator.c1.do_a()
mediator.c2.do_d()
```

Where this feels natural: coordinating widgets in a GUI (toggling a checkbox enables a button), chat servers (clients send messages to the server, not directly to each other), systems that reconcile events between independent modules.

## Pitfalls

- The mediator tends to turn into a "god object": all the interaction logic flows into it and it bloats. If the mediator has grown huge, that is a signal to split it up or rethink the boundaries.
- The pattern lowers coupling between components but raises coupling to the mediator itself - the components are useless without it.
- It is easy to confuse with Observer: Observer is a broadcast one-to-many subscription, while Mediator centralizes a bidirectional protocol between specific participants.
- Do not introduce a mediator where there are few links: for two or three objects, direct calls are simpler and read better.
