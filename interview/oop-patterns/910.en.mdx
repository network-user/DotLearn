# Which OOP principles Python supports and how they look in practice

## In short

Python is a full-fledged object-oriented language: it has encapsulation, inheritance, polymorphism, and abstraction, while composition is used as an alternative to deep hierarchies. Dynamic typing and multiple inheritance make these mechanisms especially flexible.

## How to answer in an interview

Name the four classic pillars - encapsulation, inheritance, polymorphism, abstraction - and add composition as a practical technique. In Python polymorphism is largely built on duck typing: what matters is not inheritance but the presence of the required method. Abstraction is defined through the `abc` module, while encapsulation rests on conventions (`_`, `__`) rather than strict hiding.

Encapsulation binds data and methods together in one object. Inheritance builds a new class on top of an existing one. Polymorphism gives a single call for different types. Abstraction hides details behind an interface. Composition assembles an object from other objects.

```python
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self):
        ...

class Circle(Shape):
    def __init__(self, r):
        self.r = r

    def area(self):
        return 3.14 * self.r * self.r

class Square(Shape):
    def __init__(self, side):
        self.side = side

    def area(self):
        return self.side * self.side

shapes = [Circle(1), Square(2)]
for shape in shapes:
    print(shape.area())
```

Here `Shape` defines the abstraction, the subclasses implement `area`, and the single loop is polymorphism: one call, different behavior.

## Pitfalls

- Python has no truly private fields. `__name` is merely mangled through name mangling (`_Class__name`), but access to it is preserved. Encapsulation is a convention, not a prohibition.
- Polymorphism in Python is more often about duck typing: an object does not have to inherit from a common class, it is enough to have the required method.
- Deep inheritance hierarchies are fragile; composition ("has-a") is often preferable to inheritance ("is-a").
- An abstract class with `@abstractmethod` cannot be instantiated until a subclass implements all abstract methods - otherwise you get a `TypeError` when creating the object.
