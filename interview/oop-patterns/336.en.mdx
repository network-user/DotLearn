# The **init** method: what initializes an object and why it is needed

## In short

`__init__` is the object initializer: a method that Python automatically calls right after an instance is created in order to set its attributes' starting values. By the time you get the object, it is already ready to work.

## How to answer in an interview

`__init__` is a special class method that runs when each instance is created. As its first argument it receives the new object itself (`self`), followed by any parameters for configuration. Inside, we attach attributes onto `self`. Strictly speaking, the object is created by `__new__` (it allocates memory and returns the instance), while `__init__` only fills an already-ready object with data and returns nothing. That is why in everyday speech `__init__` is called a constructor, even though construction is the pair `__new__` + `__init__`.

## How it works

When `Car(...)` is called, Python first calls `Car.__new__`, gets the object, then passes it and the arguments into `__init__`. That fills in the attributes:

```python
class Car:
    def __init__(self, make, model, year):
        self.make = make
        self.model = model
        self.year = year

    def display_info(self):
        print(f"{self.year} {self.make} {self.model}")


my_car = Car("Toyota", "Corolla", 2020)
my_car.display_info()
```

Here `__init__` takes `make`, `model`, `year` and writes them into the instance attributes. Without it you would have to set the fields manually after creation, which is verbose and easy to leave the object in an under-configured state.

## Pitfalls

- **`__init__` must not return anything.** Any `return` with a value other than `None` raises a `TypeError`.
- **Mutable default values are a trap.** `def __init__(self, items=[])` creates the list once, when the function is defined, and it will be shared across all instances. The right way: `items=None`, and inside `self.items = items or []`.
- **`__init__` is not the point where the object is created.** The instance itself is born from `__new__`; for immutable types and metaclasses this distinction is essential.
- **When inheriting, do not forget `super().__init__(...)`.** If you overrode `__init__` and did not call the parent's, part of the state will stay uninitialized.
- **Heavy logic (network requests, opening files) is better kept out of `__init__`** - it complicates testing and object creation; for alternative ways of creating objects, classmethod factories are used.
