# Кто создаёт объект, а кто его настраивает: **new** против **init**

## Суть

`__new__` создаёт и возвращает сам экземпляр — он отрабатывает первым. `__init__` получает уже готовый объект и наполняет его состоянием. Переопределять `__new__` приходится редко: в основном при наследовании от неизменяемых типов и в трюках вроде синглтонов.

## Что ответить на собесе

Рождение объекта в Python — это два шага.

- Сначала вызывается `__new__(cls, ...)` — статический метод, который аллоцирует и возвращает экземпляр.
- Затем Python зовёт `__init__(self, ...)` на этом экземпляре, чтобы задать атрибуты; `__init__` ничего не возвращает (только `None`) и не создаёт объект, а лишь конфигурирует.
- Если `__new__` вернёт экземпляр другого класса, `__init__` для него может вообще не вызваться.
- Трогать `__new__` стоит лишь тогда, когда нужно вмешаться именно в момент создания — например, у неизменяемых `tuple`/`str`, где значение фиксируется при создании.

`__new__` идёт первым и обязан вернуть экземпляр, иначе `__init__` не запустится:

<PyStepper title="Порядок __new__ → __init__" code={`class MyClass:
    def __new__(cls, *args, **kwargs):
        print("1: создаём экземпляр")
        return super().__new__(cls)

    def __init__(self, value):
        print("2: инициализируем экземпляр")
        self.value = value

obj = MyClass(10)`} />

Порядок вывода — сначала строка из `__new__`, затем из `__init__`. Для неизменяемых типов значение нельзя поменять после создания, поэтому подкласс `tuple` правит именно `__new__`:

```python
class Doubled(tuple):
    def __new__(cls, iterable):
        doubled = [x * 2 for x in iterable]
        return super().__new__(cls, doubled)

print(Doubled([1, 2, 3]))  # (2, 4, 6)
```

Здесь содержимое кортежа задаётся на этапе создания — в `__init__` это было бы уже поздно, объект иммутабелен.

## Подводные камни

- `__new__` — статический метод и первым аргументом получает `cls`, а не `self`; объекта в этот момент ещё нет.
- Если `__new__` не вернёт экземпляр `cls` (или его подкласса), `__init__` не вызовется — частый источник «молчаливых» багов.
- Наивный синглтон через `__new__` всё равно вызовет `__init__` при каждом обращении к классу, так что состояние можно случайно перезатереть.
- В обычных классах переопределять `__new__` не нужно — для инициализации хватает `__init__`.
