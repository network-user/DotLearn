# Ports and adapters: how hexagonal architecture isolates the core

## In short

Hexagonal architecture (also known as ports and adapters) places a layer of abstractions between the business logic and the outside world. The core has no idea where data comes from or where it goes - it communicates through port interfaces, and concrete databases, APIs and UIs plug in via adapters. The outer part can be swapped without touching the logic.

## How to answer in an interview

Alistair Cockburn's idea is to keep the business logic independent of infrastructure details. The core defines ports - the interfaces through which it is used (inbound) and through which it reaches out (outbound). Adapters are implementations of ports for a specific technology: a SQL repository, a REST controller, a console. Dependencies point inward: adapters know about the core, but the core does not know about adapters. This lets you test the logic without a real database (by substituting a fake adapter), and swap storage or interface without changes to the core.

## How it works

The flow of control enters from the outside through an inbound adapter, passes through a port into the core, and the core reaches outward (to the database) through an outbound port implemented by another adapter.

The core depends only on the port abstraction. A concrete adapter implements it and is supplied from the outside.

```python
from abc import ABC, abstractmethod

class BookRepository(ABC):
    @abstractmethod
    def add_book(self, book): ...

    @abstractmethod
    def find_book(self, title): ...

class SQLBookRepository(BookRepository):
    def __init__(self, session):
        self._session = session

    def add_book(self, book):
        self._session.add(book)

    def find_book(self, title):
        return self._session.query(title)

class LibraryService:
    def __init__(self, repo: BookRepository):
        self._repo = repo

    def register(self, book):
        self._repo.add_book(book)
```

`LibraryService` is the core. It accepts a `BookRepository` as a port and does not know whether it is SQL, a file, or memory. In tests you substitute a fake for `SQLBookRepository`, and the logic is verified in isolation.

## Pitfalls

- This is not an end in itself. For a CRUD app with three endpoints, a ports-and-adapters layer is over-engineering and needless interfaces.
- The direction of dependencies is easy to break: if ORM types or an HTTP request leak into the core, isolation is broken and the whole point is lost.
- Ports are defined by the core, not by the infrastructure. The interface should express the need of the business logic, not mirror the API of a specific library.
- It gets confused with onion/clean architecture. They share the dependency-inversion idea, but the hexagon specifically emphasizes the symmetry of inbound and outbound adapters.
