# Builder: assembling a complex object step by step without a monster constructor

## In short

Builder is a creational pattern for assembling a complex object step by step. It separates the construction process from the final representation, so a single sequence of steps can produce different variants of the object.

## How to answer in an interview

Builder rescues you from the "telescoping constructor" - a long list of optional arguments that is impossible to read. Instead, the object is assembled in steps: individual methods set the parts, and `build()` returns the finished product. The methods often return `self`, which gives a fluent interface (method chaining). Optionally there is a Director - it encapsulates a typical order of steps so the client does not repeat the assembly recipe.

The product is assembled through the builder, and each step configures its own part.

```python
class Car:
    def __init__(self):
        self.engine = None
        self.wheels = None
        self.color = None

    def __str__(self):
        return f"Car({self.engine}, {self.wheels}, {self.color})"

class CarBuilder:
    def __init__(self):
        self.car = Car()

    def set_engine(self, engine):
        self.car.engine = engine
        return self

    def set_wheels(self, wheels):
        self.car.wheels = wheels
        return self

    def set_color(self, color):
        self.car.color = color
        return self

    def build(self):
        return self.car
```

Each setter returns `self`, so the steps line up into a chain.

```python
class Director:
    def __init__(self, builder):
        self.builder = builder

    def construct_sports_car(self):
        return self.builder.set_engine("V8").set_wheels(4).set_color("Red").build()

car = Director(CarBuilder()).construct_sports_car()
print(car)   # Car(V8, 4, Red)
```

The Director knows the "sports car" recipe and applies it to any builder with the same interface. The assembly process itself is decoupled from what the product turns out to be.

## Pitfalls

- Do not confuse it with the Factory Method: a factory returns a finished object in a single call, while Builder assembles it step by step when the configuration is complex.
- The Director is optional. If the order of steps is simple, a fluent builder is enough, and the Director is an extra layer.
- Returning `self` from methods is only needed for chaining; if you forget it, chaining breaks with an `AttributeError`.
- For immutable products, collect the fields in temporary variables and create the object whole in `build()`, rather than mutating a finished one.
- In Python, for simple cases a `dataclass` with defaults or keyword arguments is often enough - do not drag Builder into places where it is redundant.
