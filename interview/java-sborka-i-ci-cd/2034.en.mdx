# Maven lifecycles: clean, default, site

## In short

Maven organizes builds through lifecycles: ordered phases that plugin goals attach to. Three built-in lifecycles: `clean` (remove artifacts), `default` (compile, test, package, install/deploy), and `site` (generate documentation). Running `mvn package` executes every phase of the default lifecycle up to and including `package`.

## How to answer in an interview

- I name the three lifecycles and key default phases: validate → compile → test → package → verify → install → deploy.
- I explain that a phase is not a goal: a plugin can bind multiple goals to one phase.
- `mvn clean install` chains the clean lifecycle and default up to install.
- In CI, `mvn verify` or `package` plus a separate deploy step is usually enough.
- Honestly: the wrong goal on a phase (e.g. skipping tests with `-DskipTests`) defeats the purpose of the lifecycle.

## How it works

The lifecycle defines **order**; plugins supply **work**. The default lifecycle covers the full path from POM validation to publishing the artifact. The clean lifecycle has `pre-clean`, `clean`, and `post-clean`. The site lifecycle generates reports and a static site.

A typical local run:

```bash
mvn clean verify
```

Maven runs the clean lifecycle first, then walks default phases through `verify`: compile `src/main/java`, run `src/test/java`, build a JAR/WAR, and run extra checks (checkstyle, integration tests) if plugins are configured.

A minimal POM for a Java module:

```xml
<project>
  <modelVersion>4.0.0</modelVersion>
  <groupId>com.acme.billing</groupId>
  <artifactId>invoice-service</artifactId>
  <version>2.1.0</version>
  <packaging>jar</packaging>
  <build>
    <plugins>
      <plugin>
        <artifactId>maven-surefire-plugin</artifactId>
        <version>3.2.5</version>
      </plugin>
    </plugins>
  </build>
</project>
```

`mvn package` stops at the `package` phase and does not run `install` or `deploy`. The `deploy` phase needs configured `distributionManagement` and credentials for the remote repository.

## Pitfalls

- **Phases are not skippable mid-chain.** If the build fails at `test`, later default phases do not run.
- **Multiple plugins per phase.** Execution order within a phase comes from configuration, not intuition.
- **clean ≠ rebuild.** `clean` deletes `target/` but does not fix a corrupted local repo cache.
- **site lifecycle is rare in practice.** Many teams generate docs in CI as a separate step, not via `mvn site`.
- **Mixing lifecycles in one command.** `mvn clean deploy` pulls three different phase sets; deploy failures are harder to pinpoint without per-phase logs.
