# Жизненные циклы Maven: clean, default, site

## Суть

Maven организует сборку через lifecycle: упорядоченные фазы, к которым привязываются goals плагинов. Три встроенных цикла: `clean` (очистка артефактов), `default` (компиляция, тесты, упаковка, install/deploy) и `site` (генерация документации). Вызов `mvn package` выполняет все фазы default lifecycle до `package` включительно.

## Что ответить на собесе

- Называю три lifecycle и ключевые фазы default: validate → compile → test → package → verify → install → deploy.
- Объясняю, что фаза не равна goal: плагин может повесить несколько goals на одну фазу.
- `mvn clean install` склеивает clean lifecycle и default до install.
- В CI обычно достаточно `mvn verify` или `package` с отдельным deploy step.
- Честно: неподходящий goal на фазе (например, пропуск тестов через `-DskipTests`) ломает смысл lifecycle.

## Как это работает

Lifecycle задаёт **порядок**, а плагины приносят **работу**. Default lifecycle покрывает весь путь от валидации POM до публикации артефакта. Clean lifecycle содержит `pre-clean`, `clean`, `post-clean`. Site lifecycle генерирует отчёты и статический site.

Типичный локальный прогон:

```bash
mvn clean verify
```

Maven сначала выполнит clean lifecycle, затем пройдёт default фазы до `verify`: скомпилирует `src/main/java`, прогонит `src/test/java`, соберёт JAR/WAR и запустит дополнительные проверки (checkstyle, integration tests), если плагины подключены.

Минимальный POM для Java-модуля:

```xml
<project>
  <modelVersion>4.0.0</modelVersion>
  <groupId>com.acme.billing</groupId>
  <artifactId>invoice-service</artifactId>
  <version>2.1.0</version>
  <packaging>jar</packaging>
  <build>
    <plugins>
      <plugin>
        <artifactId>maven-surefire-plugin</artifactId>
        <version>3.2.5</version>
      </plugin>
    </plugins>
  </build>
</project>
```

Команда `mvn package` дойдёт до фазы `package`, не выполняя `install` и `deploy`. Фаза `deploy` требует настроенного `distributionManagement` и credentials для remote repository.

## Подводные камни

- **Фазы неотменяемы.** Если сборка упала на `test`, последующие фазы default lifecycle не запустятся.
- **Несколько плагинов на фазу.** Порядок execution внутри фазы определяется конфигурацией, не интуицией.
- **clean ≠ rebuild.** `clean` удаляет `target/`, но не чинит corrupted local repo cache.
- **site lifecycle устарел в практике.** Многие команды генерируют docs в CI отдельным шагом, не через `mvn site`.
- **Смешение lifecycles в одной команде.** `mvn clean deploy` тянет три разных набора фаз; ошибку в deploy сложнее локализовать без логов по фазам.
