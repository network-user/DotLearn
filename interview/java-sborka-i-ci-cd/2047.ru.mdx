# Gradle: система сборки и управления зависимостями

## Суть

Gradle это build automation tool: компилирует Java/Kotlin, запускает тесты, собирает jar/aar/apk, резолвит зависимости из Maven/Ivy репозиториев. Конфигурация через Groovy или Kotlin DSL; выполнение через directed acyclic graph задач. Ключевые преимущества: incremental builds, build cache, поддержка multi-project и composite builds.

## Что ответить на собесе

- Wrapper `./gradlew` фиксирует версию Gradle в репозитории.
- `implementation` / `api` / `compileOnly` / `runtimeOnly` аналоги Maven scopes.
- Стандартный lifecycle Java plugin: `compileJava` → `classes` → `test` → `jar` → `build`.
- Android Studio и Kotlin Multiplatform строятся на Gradle.
- Честно: без понимания configuration vs execution phase скрипты становятся хрупкими.

## Как это работает

Минимальный Kotlin DSL проект:

```kotlin
plugins {
    java
}
repositories {
    mavenCentral()
}
dependencies {
    implementation("org.apache.commons:commons-lang3:3.14.0")
    testImplementation("org.junit.jupiter:junit-jupiter:5.10.2")
}
tasks.test { useJUnitPlatform() }
```

Запуск:

```bash
./gradlew build
```

Gradle пройдёт: resolve dependencies → compileJava → processResources → test → jar.

Multi-module `settings.gradle.kts`:

```kotlin
rootProject.name = "payments"
include("api", "service", "worker")
```

Зависимость между модулями:

```kotlin
dependencies {
    implementation(project(":api"))
}
```

Gradle daemon держит JVM тёплой между вызовами; `--no-daemon` на CI иногда обязателен в контейнерах с лимитом памяти.

## Подводные камни

- **Top-level код в build.gradle.** Выполняется на каждый вызов, даже `tasks`.
- **Dynamic versions `1.+`.** Невоспроизводимые CI сборки.
- **Conflicting repositories order.** Первый repo с артефактом побеждает, не всегда Central.
- **IDE sync ≠ CLI.** Расхождение classpath между IntelliJ и `./gradlew test`.
- **Устаревший Groovy DSL без types.** Ошибки типов только в runtime на CI.
