# Maven vs Gradle: когда что выбирать

## Суть

Maven: декларативный XML, convention over configuration, фиксированные lifecycles, зрелая экосистема enterprise. Gradle: программируемые build-скрипты (Kotlin/Groovy DSL), task graph, incremental build и build cache, быстрее на крупных multi-module. Оба резолвят артефакты из Maven Central; выбор чаще про скорость итерации и сложность кастомизации, не про «лучше/хуже».

## Что ответить на собесе

- Maven проще онбордить: стандартная структура, POM наследуется от parent/BOM.
- Gradle выигрывает на больших репо за счёт up-to-date и remote cache.
- Kotlin DSL Gradle даёт type-safe конфигурацию; Maven XML многословен, но предсказуем.
- Spring Initializr по умолчанию Maven или Gradle: оба first-class.
- Честно: миграция Maven → Gradle окупается не на pet-проекте, а при боли от времени сборки.

## Как это работает

Maven POM (минимум):

```xml
<project>
  <parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.3.1</version>
  </parent>
  <artifactId>catalog-api</artifactId>
  <dependencies>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
    <dependency>
      <groupId>org.junit.jupiter</groupId>
      <artifactId>junit-jupiter</artifactId>
      <scope>test</scope>
    </dependency>
  </dependencies>
</project>
```

Gradle Kotlin DSL эквивалент:

```kotlin
plugins {
    java
    id("org.springframework.boot") version "3.3.1"
}
dependencies {
    implementation("org.springframework.boot:spring-boot-starter-web")
    testImplementation("org.junit.jupiter:junit-jupiter:5.10.2")
}
tasks.test { useJUnitPlatform() }
```

Сравнение по практике:

| Критерий | Maven | Gradle |
|----------|-------|--------|
| Конфиг | XML, plugin goals на phases | Tasks, plugins, lazy API |
| Скорость | Полная пересборка чаще | Incremental + cache |
| Кастомизация | Плагины + profiles | Скриптовая логика |
| IDE | Отличная | Отличная (Kotlin DSL) |

В Android и Kotlin multiplatform Gradle де-факто стандарт; в корпоративных Java monorepo часто Maven reactor с CI кэшем `.m2`.

## Подводные камни

- **Два инструмента в одном репо.** Разные lock/resolution правила, двойная поддержка.
- **Gradle Groovy без типов.** Ошибки только в runtime на CI.
- **Maven без enforcer.** Дрейф версий транзитивов незаметен до runtime.
- **Кэш Gradle без очистки.** Stale output после смены plugin версии.
- **XML copy-paste.** Дублирование dependency версий без BOM.
