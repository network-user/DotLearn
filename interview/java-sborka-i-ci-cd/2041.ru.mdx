# Этапы CI/CD: от commit до production

## Суть

CI/CD разбивает доставку ПО на автоматизированные стадии: интеграция (сборка + тесты на каждый commit), доставка (деплой артефакта), эксплуатация (мониторинг, rollback). Для Java-команды это обычно Git → CI build → container image → staging → canary/production → observability feedback loop.

## Что ответить на собесе

- **CI:** compile, unit, integration, static analysis, security scan dependencies.
- **CD:** promote immutable artifact (jar/image digest), не пересобирать на prod.
- **Release:** feature flags, DB migration job, smoke tests после deploy.
- **Operate:** metrics (latency, error rate), logs, tracing, alerting on SLO burn.
- Честно: CD без automated rollback и миграций в отдельном job рискован для stateful сервисов.

Практический пример Java микросервиса:

1. **Build:** `./gradlew bootJar`, тег `git sha`.
2. **Test:** JUnit + Testcontainers Kafka, contract tests с Pact.
3. **Integration (VCS):** merge в `main` после green PR checks.
4. **Deploy:** Helm upgrade с image `payments@sha256:abc...`, readiness probe `/actuator/health`.
5. **Monitor:** Prometheus scrape + Grafana dashboard, alert на 5xx > 1%.

Smoke test после deploy:

```java
@Test
void healthEndpointReturnsUp() {
    var response = RestClient.create(stagingBaseUrl)
        .get()
        .uri("/actuator/health")
        .retrieve()
        .body(String.class);
    assertThat(response).contains("UP");
}
```

Git merge не равен deploy: trunk-based команды деплоят несколько раз в день с одним и тем же pipeline template, меняется только target environment и approval gate.

## Подводные камни

- **Rebuild на prod.** Дрейф зависимостей: вчерашний main ≠ сегодняшний snapshot.
- **Миграции в app startup.** Flyway на всех pod одновременно гоняет гонки.
- **Staging без parity.** Другой Kafka версии скрывает баг до prod.
- **Monitor без deployment marker.** Невозможно связать spike ошибок с release.
- **Manual gate без SLA.** Очередь на approve тормозит hotfix.
