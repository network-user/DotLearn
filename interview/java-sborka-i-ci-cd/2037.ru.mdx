# Scope зависимостей Maven: compile, test, provided и др.

## Суть

Scope управляет тем, на каких этапах сборки и рантайма dependency доступна и попадает ли в итоговый артефакт. `compile` (дефолт) виден везде и упаковывается. `provided` нужен при компиляции, но ждётся от контейнера. `runtime` не участвует в compile main code. `test` изолирован в тестах. `import` только в `dependencyManagement` для BOM.

## Что ответить на собесе

- `compile`: бизнес-библиотеки (Guava, Jackson).
- `provided`: servlet/Jakarta EE API, Lombok (annotation processing).
- `runtime`: JDBC драйвер, реализация SLF4J backend.
- `test`: JUnit, Mockito, AssertJ.
- Честно: `system` scope с локальным путём ломает воспроизводимость CI, избегаю.

## Как это работает

Таблица видимости (упрощённо):

| Scope | Compile main | Compile test | Runtime | В артефакте |
|-------|-------------|--------------|---------|-------------|
| compile | да | да | да | да |
| provided | да | да | нет | нет |
| runtime | нет | да | да | да |
| test | нет | да | нет | нет |

Пример POM для Spring Boot сервиса в embedded Tomcat vs WAR на внешний сервер:

```xml
<dependencies>
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
  </dependency>
  <dependency>
    <groupId>jakarta.servlet</groupId>
    <artifactId>jakarta.servlet-api</artifactId>
    <scope>provided</scope>
  </dependency>
  <dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
    <scope>runtime</scope>
  </dependency>
  <dependency>
    <groupId>org.mockito</groupId>
    <artifactId>mockito-core</artifactId>
    <scope>test</scope>
  </dependency>
</dependencies>
```

BOM через `import`:

```xml
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-dependencies</artifactId>
      <version>3.3.1</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>
```

`import` не добавляет jar, только версии для дочерних модулей.

## Подводные камни

- **test в production.** Забытый `compile` scope на mock-библиотеке раздувает jar.
- **provided на embedded run.** Servlet API как compile тянет конфликт с embedded container.
- **runtime без интерфейса.** Код компилируется против API, реализация только runtime: `NoClassDefFoundError` при неправильном scope.
- **import вне dependencyManagement.** Maven отклонит POM.
- **optional=true путают со scope.** Optional режет transitive propagation, scope режет classpath.
