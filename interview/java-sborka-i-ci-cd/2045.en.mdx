# Gradle pros and cons for Java projects

## In short

Gradle excels at speed (incremental build, build cache, configuration cache), Kotlin/Groovy DSL flexibility, and multi-module graphs. Weak spots: learning curve, heavy configuration-phase logic risk, RAM use on large builds. It is standard for Android and Kotlin; for enterprise Java the choice often depends on monorepo size.

## How to answer in an interview

- Pros: task graph, up-to-date, remote cache in CI, type-safe Kotlin DSL.
- Cons: complex scripts without review, Groovy without compile-time checks.
- Composite builds and version catalogs simplify multi-repo setups.
- `--scan` gives build insights Maven does not ship out of the box.
- Honestly: Gradle pays off with frequent local builds; on a tiny lib Maven is simpler.

## How it works

Typical `build.gradle.kts`:

```kotlin
plugins {
    java
    id("io.spring.dependency-management") version "1.1.6"
}
java.toolchain.languageVersion.set(JavaLanguageVersion.of(21))
tasks.withType<Test> {
    useJUnitPlatform()
    maxParallelForks = Runtime.getRuntime().availableProcessors() / 2
}
```

Incremental compile: `compileJava` is UP-TO-DATE when source and classpath inputs did not change.

Remote cache in `settings.gradle.kts`:

```kotlin
buildCache {
    remote<HttpBuildCache> {
        url = uri("https://cache.acme.io/")
        isPush = providers.gradleProperty("ci").isPresent
    }
}
```

Practical comparison:

| Pro | Con |
|------|-------|
| Fast feedback loop | Script can become a "mini app" |
| Flexible custom tasks | Flaky from shared mutable state |
| Excellent IDE sync | Daemon + cache consume RAM |
| Same tool as Android | Migration from Maven is costly |

## Pitfalls

- **Configuration cache violations.** Reflection in build scripts breaks CI after upgrade.
- **Implicit task dependencies.** Missing declared inputs/outputs: silent wrong builds.
- **Version catalog drift.** `libs.versions.toml` out of sync with parent BOM.
- **Parallel tests + static state.** Green locally, red in CI.
- **Daemon disabled on CI.** Every job cold-starts; Gradle speed advantage fades.
