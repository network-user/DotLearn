# CI/CD stages: from commit to production

## In short

CI/CD splits software delivery into automated stages: integration (build + tests on every commit), delivery (deploy the artifact), and operations (monitoring, rollback). For a Java team this is usually Git → CI build → container image → staging → canary/production → observability feedback loop.

## How to answer in an interview

- **CI:** compile, unit, integration, static analysis, dependency security scan.
- **CD:** promote an immutable artifact (jar/image digest), do not rebuild for prod.
- **Release:** feature flags, DB migration job, smoke tests after deploy.
- **Operate:** metrics (latency, error rate), logs, tracing, alert on SLO burn.
- Honestly: CD without automated rollback and migrations in a separate job is risky for stateful services.

Practical Java microservice example:

1. **Build:** `./gradlew bootJar`, tag `git sha`.
2. **Test:** JUnit + Testcontainers Kafka, Pact contract tests.
3. **Integration (VCS):** merge to `main` after green PR checks.
4. **Deploy:** Helm upgrade with image `payments@sha256:abc...`, readiness probe `/actuator/health`.
5. **Monitor:** Prometheus scrape + Grafana dashboard, alert on 5xx > 1%.

Post-deploy smoke test:

```java
@Test
void healthEndpointReturnsUp() {
    var response = RestClient.create(stagingBaseUrl)
        .get()
        .uri("/actuator/health")
        .retrieve()
        .body(String.class);
    assertThat(response).contains("UP");
}
```

Git merge is not deploy: trunk-based teams deploy several times a day with the same pipeline template; only target environment and approval gates change.

## Pitfalls

- **Rebuild on prod.** Dependency drift: yesterday's main ≠ today's snapshot.
- **Migrations in app startup.** Flyway on all pods at once races.
- **Staging without parity.** Different Kafka version hides bugs until prod.
- **Monitor without deployment marker.** Cannot tie error spikes to a release.
- **Manual gate without SLA.** Approval queue blocks hotfixes.
