# Maven vs Gradle: when to pick which

## In short

Maven: declarative XML, convention over configuration, fixed lifecycles, mature enterprise ecosystem. Gradle: programmable build scripts (Kotlin/Groovy DSL), task graph, incremental build and build cache, faster on large multi-module repos. Both resolve artifacts from Maven Central; the choice is usually iteration speed and customization complexity, not "better/worse."

## How to answer in an interview

- Maven is easier to onboard: standard layout, POM inherits from parent/BOM.
- Gradle wins on large repos via up-to-date checks and remote cache.
- Gradle Kotlin DSL gives type-safe config; Maven XML is verbose but predictable.
- Spring Initializr defaults to Maven or Gradle: both are first-class.
- Honestly: Maven → Gradle migration pays off when build time hurts, not on a pet project.

## How it works

Minimal Maven POM:

```xml
<project>
  <parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.3.1</version>
  </parent>
  <artifactId>catalog-api</artifactId>
  <dependencies>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
    <dependency>
      <groupId>org.junit.jupiter</groupId>
      <artifactId>junit-jupiter</artifactId>
      <scope>test</scope>
    </dependency>
  </dependencies>
</project>
```

Gradle Kotlin DSL equivalent:

```kotlin
plugins {
    java
    id("org.springframework.boot") version "3.3.1"
}
dependencies {
    implementation("org.springframework.boot:spring-boot-starter-web")
    testImplementation("org.junit.jupiter:junit-jupiter:5.10.2")
}
tasks.test { useJUnitPlatform() }
```

Practical comparison:

| Criterion | Maven | Gradle |
|----------|-------|--------|
| Config | XML, plugin goals on phases | Tasks, plugins, lazy API |
| Speed | Full rebuilds more often | Incremental + cache |
| Customization | Plugins + profiles | Script logic |
| IDE | Excellent | Excellent (Kotlin DSL) |

Android and Kotlin multiplatform are Gradle by default; corporate Java monorepos often use a Maven reactor with CI `.m2` cache.

## Pitfalls

- **Two tools in one repo.** Different lock/resolution rules, double maintenance.
- **Gradle Groovy without types.** Errors surface only at CI runtime.
- **Maven without enforcer.** Transitive version drift hides until runtime.
- **Gradle cache without cleanup.** Stale output after plugin version change.
- **XML copy-paste.** Duplicate dependency versions without a BOM.
