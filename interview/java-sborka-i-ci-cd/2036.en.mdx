# Maven dependency vs plugin: different roles

## In short

A dependency lands on the application classpath (compile/runtime/test) and is resolved from a repository. A plugin extends Maven itself: it runs goals on lifecycle phases (compile, test, shade, docker). Dependencies describe **what the code needs**; plugins describe **how to build the project**.

## How to answer in an interview

- Dependencies go in `<dependencies>`, plugins in `<build><plugins>`.
- Jackson on the classpath, `maven-compiler-plugin` at the compile phase.
- A plugin pulls its own transitive dependencies, but they are not visible to application code.
- I pin plugin versions in a parent POM or `pluginManagement`, not on super-POM defaults.
- Honestly: confusion happens when a business library is wrongly declared as a plugin or the reverse.

## How it works

A runtime dependency for REST serialization:

```xml
<dependency>
  <groupId>com.fasterxml.jackson.core</groupId>
  <artifactId>jackson-databind</artifactId>
  <version>2.17.1</version>
</dependency>
```

Code imports `ObjectMapper` directly; the artifact ends up in a fat jar or WAR `WEB-INF/lib`.

A compile plugin targeting Java 21:

```xml
<build>
  <plugins>
    <plugin>
      <groupId>org.apache.maven.plugins</groupId>
      <artifactId>maven-compiler-plugin</artifactId>
      <version>3.13.0</version>
      <configuration>
        <release>21</release>
      </configuration>
    </plugin>
  </plugins>
</build>
```

On `mvn compile`, Maven loads the plugin jar on the build classpath and runs the `compile` goal, but `maven-compiler-plugin` does not become an application dependency.

Gradle analogy: `implementation` vs `plugins { id(...) }`. In Maven a plugin can declare `<dependencies>` inside itself (e.g. SpotBugs with a specific analyzer version); that is not the same as a project dependency.

## Pitfalls

- **Plugin without a version.** The super-POM default lags behind current JDKs.
- **Dependency in the plugin section.** The build fails with ClassNotFound in a goal, not in the app.
- **Provided dependency vs plugin.** Servlet API `provided` for code; `maven-war-plugin` for packaging.
- **Transitive leakage.** Wrong dependency scope bloats the production jar.
- **Duplicate versions.** One library in dependencies and as a plugin dependency (enforcer duplicate rule).
