# Gradle: build system and dependency management

## In short

Gradle is a build automation tool: compiles Java/Kotlin, runs tests, assembles jar/aar/apk, resolves dependencies from Maven/Ivy repositories. Configuration uses Groovy or Kotlin DSL; execution runs a directed acyclic graph of tasks. Key strengths: incremental builds, build cache, multi-project and composite build support.

## How to answer in an interview

- Wrapper `./gradlew` pins the Gradle version in the repo.
- `implementation` / `api` / `compileOnly` / `runtimeOnly` mirror Maven scopes.
- Standard Java plugin flow: `compileJava` → `classes` → `test` → `jar` → `build`.
- Android Studio and Kotlin Multiplatform are built on Gradle.
- Honestly: without configuration vs execution phase knowledge, scripts become fragile.

## How it works

Minimal Kotlin DSL project:

```kotlin
plugins {
    java
}
repositories {
    mavenCentral()
}
dependencies {
    implementation("org.apache.commons:commons-lang3:3.14.0")
    testImplementation("org.junit.jupiter:junit-jupiter:5.10.2")
}
tasks.test { useJUnitPlatform() }
```

Run:

```bash
./gradlew build
```

Gradle runs: resolve dependencies → compileJava → processResources → test → jar.

Multi-module `settings.gradle.kts`:

```kotlin
rootProject.name = "payments"
include("api", "service", "worker")
```

Cross-module dependency:

```kotlin
dependencies {
    implementation(project(":api"))
}
```

The Gradle daemon keeps the JVM warm between invocations; `--no-daemon` is sometimes required in memory-limited CI containers.

## Pitfalls

- **Top-level code in build.gradle.** Runs on every invocation, even `tasks`.
- **Dynamic versions `1.+`.** Non-reproducible CI builds.
- **Conflicting repository order.** First repo with the artifact wins, not always Central.
- **IDE sync ≠ CLI.** Classpath drift between IntelliJ and `./gradlew test`.
- **Legacy Groovy DSL without types.** Type errors surface only at CI runtime.
