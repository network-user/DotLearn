# Maven dependency scopes: compile, test, provided, and more

## In short

Scope controls when a dependency is available during build and runtime and whether it lands in the final artifact. `compile` (default) is visible everywhere and packaged. `provided` is needed at compile time but expected from the container. `runtime` is not on the main compile classpath. `test` stays in tests. `import` is only for BOMs inside `dependencyManagement`.

## How to answer in an interview

- `compile`: business libraries (Guava, Jackson).
- `provided`: servlet/Jakarta EE API, Lombok (annotation processing).
- `runtime`: JDBC driver, SLF4J backend implementation.
- `test`: JUnit, Mockito, AssertJ.
- Honestly: `system` scope with a local path breaks reproducible CI; I avoid it.

## How it works

Visibility table (simplified):

| Scope | Compile main | Compile test | Runtime | In artifact |
|-------|-------------|--------------|---------|-------------|
| compile | yes | yes | yes | yes |
| provided | yes | yes | no | no |
| runtime | no | yes | yes | yes |
| test | no | yes | no | no |

POM example for a Spring Boot service on embedded Tomcat vs WAR on an external server:

```xml
<dependencies>
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
  </dependency>
  <dependency>
    <groupId>jakarta.servlet</groupId>
    <artifactId>jakarta.servlet-api</artifactId>
    <scope>provided</scope>
  </dependency>
  <dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
    <scope>runtime</scope>
  </dependency>
  <dependency>
    <groupId>org.mockito</groupId>
    <artifactId>mockito-core</artifactId>
    <scope>test</scope>
  </dependency>
</dependencies>
```

BOM via `import`:

```xml
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-dependencies</artifactId>
      <version>3.3.1</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>
```

`import` does not add a jar; it only pins versions for child modules.

## Pitfalls

- **test in production.** A mock library left at `compile` scope bloats the jar.
- **provided on embedded run.** Servlet API as compile conflicts with the embedded container.
- **runtime without an interface.** Code compiles against an API but implementation is runtime only: `NoClassDefFoundError` with wrong scope.
- **import outside dependencyManagement.** Maven rejects the POM.
- **optional=true confused with scope.** Optional cuts transitive propagation; scope cuts classpath.
