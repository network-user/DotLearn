# Плюсы и минусы DDD

## Суть

DDD моделирует сложный domain через Ubiquitous Language, bounded contexts, aggregates. Плюсы: alignment с бизнесом, чёткие границы, evolvable architecture. Минусы: overhead на старте, риск over-engineering, нужны domain experts и дисциплина команды.

## Что ответить на собесе

- Применяю где domain rules нетrivial и живут годами.
- Bounded context с explicit integration (ACL, events).
- Aggregates guard invariants, repos per aggregate root.
- Честно: CRUD microservice без rich domain rarely needs full DDD tactical patterns.

## Как это работает

В сервисе заказов выделяю контексты Billing и Fulfillment. Aggregate OrderRoot инкапсулирует статус и line items; смена статуса только через domain methods. Между контекстами integration event `OrderPaid`, не shared DB tables.

```go
type Order struct {
	id     OrderID
	status Status
	lines  []Line
}

func (o *Order) MarkPaid() error {
	if o.status != StatusPending {
		return ErrInvalidTransition
	}
	o.status = StatusPaid
	return nil
}
```

Go не навязывает DDD types, но пакеты `domain/`, `app/`, `infra/` отражают слои.

## Подводные камни

**Anemic domain model.** Structs + service god object без behavior.

**Huge aggregate boundaries.** Contention и merge conflicts.

**Shared kernel everywhere.** Contexts collapse into monolith.

**DDD without events/versioning.** Breaking cross-team contracts.

**Tactical patterns in greenfield MVP.** Slows time to market.
