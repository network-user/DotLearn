# Зачем microservices

## Суть

Microservices address monolith pain: independent deploy per team, scale hot services only, technology heterogeneity per domain, blast radius isolation on failure. Цена: distributed complexity, observability, data consistency, network latency.

## Что ответить на собесе

- Team autonomy и faster release cadence per bounded context.
- Scale image-processor replicas, not whole app.
- Failure in auth should not take down catalog if designed well.
- Честно: small team greenfield often faster as modular monolith first.

## Как это работает

```go
// users service :8081
func main() {
	http.HandleFunc("/users/", handleUsers)
	log.Fatal(http.ListenAndServe(":8081", nil))
}
```

```go
// orders service :8082
func main() {
	http.HandleFunc("/orders/", handleOrders)
	log.Fatal(http.ListenAndServe(":8082", nil))
}
```

Deploy orders v2 без redeploy users. HPA scales orders under sale traffic independently.

## Подводные камни

**Microservices for resume.** Ops cost exceeds benefit.

**Shared database coupling.** Fake services, real monolith.

**No platform investment.** Each team rebuilds logging, auth, CI.

**Latency chains.** p99 sums across hops.

**Incomplete ownership.** «Not my service» on incidents.
