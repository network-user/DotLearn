# Pros and cons of DDD

## In short

DDD models a complex domain with Ubiquitous Language, bounded contexts, and aggregates. Pros: business alignment, clear boundaries, evolvable architecture. Cons: startup overhead, over-engineering risk, needs domain experts and team discipline.

## How to answer in an interview

- Use it where domain rules are non-trivial and long-lived.
- Bounded contexts with explicit integration (ACL, events).
- Aggregates guard invariants; repos per aggregate root.
- Honestly: a CRUD microservice without a rich domain rarely needs full DDD tactical patterns.

## How it works

In an order service I split Billing and Fulfillment contexts. Aggregate OrderRoot encapsulates status and line items; status changes go through domain methods only. Contexts integrate via `OrderPaid` events, not shared DB tables.

```go
type Order struct {
	id     OrderID
	status Status
	lines  []Line
}

func (o *Order) MarkPaid() error {
	if o.status != StatusPending {
		return ErrInvalidTransition
	}
	o.status = StatusPaid
	return nil
}
```

Go does not enforce DDD types, but `domain/`, `app/`, `infra/` packages reflect layers.

## Pitfalls

**Anemic domain model.** Structs plus a god service, no behavior.

**Huge aggregate boundaries.** Contention and merge conflicts.

**Shared kernel everywhere.** Contexts collapse into a monolith.

**DDD without events/versioning.** Breaking cross-team contracts.

**Tactical patterns in a greenfield MVP.** Slows time to market.
