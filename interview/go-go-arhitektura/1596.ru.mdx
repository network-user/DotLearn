# Коммуникация между microservices

## Суть

Sync: HTTP/REST, gRPC over HTTP/2 with protobuf. Async: message broker (Kafka, NATS, RabbitMQ) for decoupling and buffering. Выбор по latency, delivery guarantees, coupling. Go idiomatic: context deadlines on RPC, idempotent consumers on events.

## Что ответить на собесе

- Sync когда нужен immediate response и simple request/response.
- Async для fan-out, peak smoothing, failure isolation.
- gRPC для internal high-throughput; REST for public API.
- Честно: sync chain of 5 services = latency sum; prefer events or aggregate API.

## Как это работает

```go
package main

import (
	"context"
	"fmt"
	"net/http"
	"time"
)

func callBilling(ctx context.Context, baseURL string) error {
	req, _ := http.NewRequestWithContext(ctx, http.MethodGet, baseURL+"/health", nil)
	client := &http.Client{Timeout: 2 * time.Second}
	resp, err := client.Do(req)
	if err != nil {
		return err
	}
	defer resp.Body.Close()
	fmt.Println("status", resp.StatusCode)
	return nil
}

func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
	defer cancel()
	_ = callBilling(ctx, "http://billing:8080")
}
```

Context propagates deadline. Async path: publish `OrderCreated` to topic, inventory consumer processes independently.

## Подводные камни

**No timeout on HTTP client.** Cascading hang.

**Chatty sync microservices.** N+1 network calls.

**At-least-once without idempotency.** Duplicate side effects.

**Shared proto without versioning.** Breaking wire compat.

**Broker as hidden DB.** Event log grows unbounded.
