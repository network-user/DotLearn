# Horizontal vs vertical scaling

## Суть

Vertical scale up: больше CPU/RAM на одном instance, ceiling у железа, SPOF. Horizontal scale out: больше replicas за load balancer, лучше fault tolerance, нужен stateless design или shared state layer. Cloud-native Go сервисы обычно scale out.

## Что ответить на собесе

- Horizontal: k8s HPA, несколько pods, session sticky или external store.
- Vertical: быстрый fix, но limit и downtime resize.
- Stateless handlers + external DB/cache для scale out.
- Честно: комбинирую: bigger nodes + multiple replicas under LB.

## Как это работает

```go
package main

import (
	"fmt"
	"net/http"
	"os"
)

func handler(w http.ResponseWriter, r *http.Request) {
	hostname, _ := os.Hostname()
	fmt.Fprintf(w, "from %s\n", hostname)
}

func main() {
	http.HandleFunc("/", handler)
	_ = http.ListenAndServe(":8080", nil)
}
```

Несколько replicas за LB; каждый отвечает со своим hostname. No in-memory session without sticky or Redis.

## Подводные камни

**Local cache per replica.** Inconsistent reads.

**Vertical only under load spike.** Hits hardware cap suddenly.

**Scale out without DB pool limits.** Connection storm.

**Stateful goroutine maps.** Breaks with >1 instance.

**Ignore cold start on HPA.** Latency during scale events.
