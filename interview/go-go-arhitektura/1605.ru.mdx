# Consistency models

## Суть

Consistency model defines visibility of writes across nodes: strong/linearizable, sequential, causal, read-your-writes, monotonic reads, eventual. Stronger model = higher coordination cost and latency. Pick per use case, not globally.

## Что ответить на собесе

- Strong for money inventory; eventual for social counters.
- Read-your-writes via sticky session or primary read after write.
- Causal needs version vectors or logical clocks.
- Честно: document which endpoints guarantee what; clients depend on it.

## Как это работает

```go
type Replica struct {
	val   string
	ver   int
}

func (r *Replica) Write(v string) { r.val = v; r.ver++ }
func (r *Replica) Read(minVer int) string {
	if r.ver < minVer {
		return ""
	}
	return r.val
}
```

Client stores ver after write; read passes minVer for read-your-writes on any replica that caught up.

## Подводные камни

**Marketing «strong» without proof.** Linearizability tests needed.

**Eventual without convergence bound.** SLA vague.

**Cross-region strong default.** Cost explosion.

**Cache breaks RYW.** Invalidate on write path.

**Monotonic reads violated by random replica pick.**
