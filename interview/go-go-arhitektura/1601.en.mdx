# What you need to build a microservice

## In short

Minimum checklist: bounded context and API contract, HTTP/gRPC server, config/secrets, structured logging, metrics/traces, health/readiness probes, container image, CI/CD, graceful shutdown, auth between services. The service owns its data store.

## How to answer in an interview

- OpenAPI/proto for the contract, semver or compatibility tests.
- Multi-stage Dockerfile, non-root user.
- `/healthz` liveness, `/readyz` checks DB.
- Honestly: a platform team provides a template; without it each service reinvents the wheel.

## How it works

```go
package main

import (
	"context"
	"log"
	"net/http"
	"os"
	"os/signal"
	"time"
)

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("/healthz", func(w http.ResponseWriter, _ *http.Request) {
		w.WriteHeader(http.StatusOK)
	})
	srv := &http.Server{Addr: ":8080", Handler: mux}

	go func() {
		log.Println(srv.ListenAndServe())
	}()

	stop := make(chan os.Signal, 1)
	signal.Notify(stop, os.Interrupt)
	<-stop
	ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
	defer cancel()
	_ = srv.Shutdown(ctx)
}
```

Contract, observability hooks, container, and orchestration manifest complete the service.

## Pitfalls

**No readiness.** Traffic arrives before DB migration.

**Secrets in image.** Use vault/env inject.

**Missing correlation ID.** Cannot trace cross-service.

**Fat container.** Slow deploy, large attack surface.

**Shared library coupling.** Hidden monolith via imports.
