# Monolith migration strategies

## In short

Monolith to services migration is incremental: Strangler Fig (proxy the new path), modular monolith first, API Gateway routing, event-driven extraction. Goal: continuous delivery without a big-bang rewrite. Data ownership moves with the bounded context.

## How to answer in an interview

- Strangler: nginx/gateway routes traffic to the new service by path.
- Carve a vertical slice with minimal cross-module deps.
- Dual-write or CDC until the data split is stable.
- Honestly: microservices do not always win; sometimes a modular monolith is enough.

## How it works

```go
package main

import (
	"net/http"
	"net/http/httputil"
	"net/url"
)

func main() {
	legacy, _ := url.Parse("http://127.0.0.1:8081")
	newSvc, _ := url.Parse("http://127.0.0.1:8082")
	legacyProxy := httputil.NewSingleHostReverseProxy(legacy)
	newProxy := httputil.NewSingleHostReverseProxy(newSvc)

	http.HandleFunc("/billing/", func(w http.ResponseWriter, r *http.Request) {
		newProxy.ServeHTTP(w, r)
	})
	http.HandleFunc("/", legacyProxy.ServeHTTP)
	_ = http.ListenAndServe(":8080", nil)
}
```

The gateway sends `/billing/` to the new service; everything else stays on the monolith. Routes migrate over time.

## Pitfalls

**Distributed monolith.** Services still deploy as one unit.

**Shared DB anti-pattern lingers.** Late schema extraction hurts more later.

**No contract tests at the boundary.** The gateway hides breaking changes.

**Events without schema versioning.** Consumers break on deploy order.

**Big-bang cutover.** No rollback plan.
