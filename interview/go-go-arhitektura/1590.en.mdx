# Clean Architecture in Go

## In short

Clean Architecture (Uncle Bob) puts domain/use cases at the center and adapters plus frameworks outside. Dependencies point inward: infra implements ports defined by the app layer. In Go I use packages `domain`, `usecase`, `adapter/http`, with wiring in `cmd`.

## How to answer in an interview

- Entities and use cases without importing sql/http.
- Interfaces (ports) in usecase; adapters in infra.
- Test use cases with an in-memory repo, no DB.
- Honestly: overkill for a tiny CLI; worth it for long-lived services.

## How it works

```go
type Account struct {
	ID      int
	Balance int64
}

type Ledger interface {
	Load(id int) (*Account, error)
	Save(*Account) error
}

type TransferUC struct {
	store Ledger
}

func (t *TransferUC) Move(from, to int, amount int64) error {
	a, err := t.store.Load(from)
	if err != nil {
		return err
	}
	a.Balance -= amount
	return t.store.Save(a)
}
```

An HTTP handler in `adapter` calls TransferUC; a postgres repo implements Ledger.

## Pitfalls

**Ports in the wrong layer.** Domain importing adapter DTOs.

**God usecase package.** Split by feature vertical slices.

**Duplicate models.** Map DTO ↔ entity explicitly.

**Clean folders, dirty deps.** The go mod graph reveals truth.

**No composition root.** Global singleton repos break testability.
