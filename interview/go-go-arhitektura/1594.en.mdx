# Horizontal vs vertical scaling

## In short

Vertical scale up: more CPU/RAM on one instance, hardware ceiling, single point of failure. Horizontal scale out: more replicas behind a load balancer, better fault tolerance, needs stateless design or a shared state layer. Cloud-native Go services usually scale out.

## How to answer in an interview

- Horizontal: k8s HPA, multiple pods, session sticky or external store.
- Vertical: quick fix, but limits and resize downtime.
- Stateless handlers plus external DB/cache for scale out.
- Honestly: combine bigger nodes with multiple replicas behind an LB.

## How it works

```go
package main

import (
	"fmt"
	"net/http"
	"os"
)

func handler(w http.ResponseWriter, r *http.Request) {
	hostname, _ := os.Hostname()
	fmt.Fprintf(w, "from %s\n", hostname)
}

func main() {
	http.HandleFunc("/", handler)
	_ = http.ListenAndServe(":8080", nil)
}
```

Multiple replicas behind an LB; each responds with its hostname. No in-memory session without sticky routing or Redis.

## Pitfalls

**Local cache per replica.** Inconsistent reads.

**Vertical only under load spikes.** Hits hardware cap suddenly.

**Scale out without DB pool limits.** Connection storm.

**Stateful goroutine maps.** Breaks with more than one instance.

**Ignore cold start on HPA.** Latency during scale events.
