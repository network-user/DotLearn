# Sharding vs replication

## In short

Sharding splits data horizontally by key across nodes (each shard holds a unique subset). Replication copies the same dataset to multiple nodes for read scale and HA. Sharding scales write volume; replication improves read throughput and failover. Often combined: each shard has its own replica set.

## How to answer in an interview

- Sharding: partition by tenant_id hash.
- Replication: primary plus standby with the same data.
- Cross-shard queries are expensive; replica lag on reads.
- Honestly: replicate first, shard when a single node hits limits.

## How it works

```
Sharding:
  user A -> shard 1
  user B -> shard 2

Replication:
  write -> primary -> binlog -> replica (same rows)
```

A Go router picks the shard; connection pool per shard primary/replica.

## Pitfalls

**Shard without replication.** Shard loss means partial data loss.

**Replicate without need.** Ops cost, no write scale.

**Confuse replica with backup.** A replica is not backup if a bad write propagates.

**Global secondary index cross shard.** Needs an extra service.

**Reshard and replicate at once.** Migration nightmare without a plan.
