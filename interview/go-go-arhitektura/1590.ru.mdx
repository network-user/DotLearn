# Clean Architecture в Go

## Суть

Clean Architecture (Uncle Bob) ставит domain/use cases в центр, adapters и frameworks снаружи. Зависимости направлены внутрь: infra implements ports defined by app layer. Go реализую пакетами `domain`, `usecase`, `adapter/http`, wiring в `cmd`.

## Что ответить на собесе

- Entities + use cases без import sql/http.
- Interfaces (ports) в usecase, adapters в infra.
- Тестирую usecase с in-memory repo без DB.
- Честно: для маленького CLI это overkill; для long-lived service pays off.

## Как это работает

```go
type Account struct {
	ID      int
	Balance int64
}

type Ledger interface {
	Load(id int) (*Account, error)
	Save(*Account) error
}

type TransferUC struct {
	store Ledger
}

func (t *TransferUC) Move(from, to int, amount int64) error {
	a, err := t.store.Load(from)
	if err != nil {
		return err
	}
	a.Balance -= amount
	return t.store.Save(a)
}
```

HTTP handler в `adapter` вызывает TransferUC, postgres repo implements Ledger.

## Подводные камни

**Ports in wrong layer.** Domain importing adapter DTOs.

**God usecase package.** Split by feature vertical slice.

**Duplicate models.** Map DTO ↔ entity explicitly.

**Clean folders, dirty deps.** go mod graph reveals truth.

**No composition root.** Global singleton repos break testability.
