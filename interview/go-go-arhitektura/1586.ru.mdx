# Dependency Inversion в Go

## Суть

DIP в Go: high-level пакеты зависят от маленьких interfaces, concrete types inject через constructors. Consumer defines interface (accept interfaces, return structs). Тесты подставляют fakes без DI-фреймворка.

## Что ответить на собесе

- Interface на стороне потребителя, не у postgres driver.
- Constructor `NewService(repo Store)` wiring в main/cmd.
- `go:generate` mock или ручные fakes в `_test`.
- Честно: не каждый тип нуждается в interface; YAGNI до второй реализации.

## Как это работает

```go
package notify

type Messenger interface {
	Deliver(body string) error
}

type Service struct {
	sender Messenger
}

func NewService(m Messenger) *Service {
	return &Service{sender: m}
}

func (s *Service) Alert(msg string) error {
	return s.sender.Deliver(msg)
}
```

```go
type SMSBackend struct{}

func (SMSBackend) Deliver(body string) error { return nil }

func main() {
	svc := notify.NewService(SMSBackend{})
	_ = svc.Alert("deploy done")
}
```

Service знает только Messenger. main выбирает SMSBackend или EmailBackend.

## Подводные камни

**Interface pollution.** One-method interfaces OK, giant interfaces smell.

**Import cycle.** Extract shared contracts package.

**Concrete return from constructor.** Return *Service, not interface, Go idiom.

**Wire in every test manually.** table-driven with fake.

**Reflection DI containers.** Rare in idiomatic Go services.
