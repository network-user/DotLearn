# Consistency models

## In short

A consistency model defines write visibility across nodes: strong/linearizable, sequential, causal, read-your-writes, monotonic reads, eventual. Stronger models mean higher coordination cost and latency. Pick per use case, not globally.

## How to answer in an interview

- Strong for money/inventory; eventual for social counters.
- Read-your-writes via sticky session or primary read after write.
- Causal needs version vectors or logical clocks.
- Honestly: document which endpoints guarantee what; clients depend on it.

## How it works

```go
type Replica struct {
	val   string
	ver   int
}

func (r *Replica) Write(v string) { r.val = v; r.ver++ }
func (r *Replica) Read(minVer int) string {
	if r.ver < minVer {
		return ""
	}
	return r.val
}
```

The client stores ver after write; read passes minVer for read-your-writes on any replica that caught up.

## Pitfalls

**Marketing «strong» without proof.** Need linearizability tests.

**Eventual without convergence bound.** Vague SLA.

**Cross-region strong by default.** Cost explosion.

**Cache breaks RYW.** Invalidate on write path.

**Monotonic reads violated by random replica pick.**
