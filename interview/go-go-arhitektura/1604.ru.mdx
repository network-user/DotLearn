# CAP и PACELC

## Суть

CAP: при network partition (P) distributed system chooses C (linearizable reads/writes) or A (every request gets response, maybe stale). P unavoidable in WAN. PACELC: if Partition then C vs A; Else Latency vs Consistency. Real systems pick per operation (strong read vs eventual).

## Что ответить на собесе

- CAP not «pick two of three always»; partition forces C/A tradeoff.
- PACELC reminds: even without partition, strong consistency costs latency.
- Dynamo/Cassandra AP; etcd/Zookeeper CP under partition.
- Честно: most prod systems tunable consistency levels per query.

## Как это работает

```go
type Store interface {
	Get(ctx context.Context, key string, level Consistency) (string, error)
	Put(ctx context.Context, key, val string, level Consistency) error
}

const (
	Quorum Consistency = iota
	One
	All
)
```

Quorum read/write balances L and C. During partition, quorum may fail writes (CP) or accept divergent copies (AP).

## Подводные камни

**Misapply CAP to single DC.** Partition rare but not zero.

**Ignore EL in design reviews.** Default strong everywhere kills p99.

**Assumption CA systems exist.** They hide P until outage.

**No client handling stale reads.** UX bugs after partition heals.

**Confuse availability with durability.**
