# Что нужно для microservice

## Суть

Минимальный checklist: bounded context и API contract, HTTP/gRPC server, config/secrets, structured logging, metrics/traces, health/readiness probes, container image, CI/CD, graceful shutdown, auth between services. Data store owned by service.

## Что ответить на собесе

- OpenAPI/proto для контракта, semver или compatibility tests.
- Dockerfile multi-stage, non-root user.
- `/healthz` liveness, `/readyz` checks DB.
- Честно: platform team даёт шаблон; без него каждый сервис reinvent wheel.

## Как это работает

```go
package main

import (
	"context"
	"log"
	"net/http"
	"os"
	"os/signal"
	"time"
)

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("/healthz", func(w http.ResponseWriter, _ *http.Request) {
		w.WriteHeader(http.StatusOK)
	})
	srv := &http.Server{Addr: ":8080", Handler: mux}

	go func() {
		log.Println(srv.ListenAndServe())
	}()

	stop := make(chan os.Signal, 1)
	signal.Notify(stop, os.Interrupt)
	<-stop
	ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
	defer cancel()
	_ = srv.Shutdown(ctx)
}
```

Contract, observability hooks, container, orchestration manifest complete the service.

## Подводные камни

**No readiness.** Traffic hits before DB migration.

**Secrets in image.** Use vault/env inject.

**Missing correlation ID.** Cannot trace cross-service.

**Fat container.** Slow deploy, large attack surface.

**Shared library coupling.** Hidden monolith via imports.
