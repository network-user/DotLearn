# What problems microservices solve

## In short

Microservices address monolith pain: independent deploy per team, scale only hot services, technology heterogeneity per domain, blast-radius isolation on failure. Cost: distributed complexity, observability, data consistency, network latency.

## How to answer in an interview

- Team autonomy and faster release cadence per bounded context.
- Scale image-processor replicas, not the whole app.
- Auth failure should not take down catalog if designed well.
- Honestly: a small team greenfield is often faster as a modular monolith first.

## How it works

```go
// users service :8081
func main() {
	http.HandleFunc("/users/", handleUsers)
	log.Fatal(http.ListenAndServe(":8081", nil))
}
```

```go
// orders service :8082
func main() {
	http.HandleFunc("/orders/", handleOrders)
	log.Fatal(http.ListenAndServe(":8082", nil))
}
```

Deploy orders v2 without redeploying users. HPA scales orders under sale traffic independently.

## Pitfalls

**Microservices for the resume.** Ops cost exceeds benefit.

**Shared database coupling.** Fake services, real monolith.

**No platform investment.** Each team rebuilds logging, auth, CI.

**Latency chains.** p99 sums across hops.

**Incomplete ownership.** «Not my service» on incidents.
