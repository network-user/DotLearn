# Dependency Inversion in Go

## In short

DIP in Go: high-level packages depend on small interfaces; concrete types are injected via constructors. The consumer defines the interface (accept interfaces, return structs). Tests swap fakes without a DI framework.

## How to answer in an interview

- Put the interface on the consumer side, not on the postgres driver.
- Wire with `NewService(repo Store)` in main/cmd.
- Use `go:generate` mocks or hand-written fakes in `_test`.
- Honestly: not every type needs an interface; YAGNI until a second implementation.

## How it works

```go
package notify

type Messenger interface {
	Deliver(body string) error
}

type Service struct {
	sender Messenger
}

func NewService(m Messenger) *Service {
	return &Service{sender: m}
}

func (s *Service) Alert(msg string) error {
	return s.sender.Deliver(msg)
}
```

```go
type SMSBackend struct{}

func (SMSBackend) Deliver(body string) error { return nil }

func main() {
	svc := notify.NewService(SMSBackend{})
	_ = svc.Alert("deploy done")
}
```

Service knows only Messenger. main picks SMSBackend or EmailBackend.

## Pitfalls

**Interface pollution.** Small one-method interfaces are fine; huge interfaces smell.

**Import cycle.** Extract a shared contracts package.

**Concrete return from constructor.** Return *Service, not an interface; Go idiom.

**Manual wiring in every test.** Table-driven tests with a fake.

**Reflection DI containers.** Rare in idiomatic Go services.
