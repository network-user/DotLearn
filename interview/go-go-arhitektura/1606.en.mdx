# Consistency patterns in distributed systems

## In short

Practical patterns: quorum R/W, leader-based replication (strong via single writer), multi-version concurrency, CRDT for mergeable state, saga for cross-service atomicity illusion, outbox for reliable publish. Each trades latency, availability, and dev complexity.

## How to answer in an interview

- Leader plus sync replica for strong OLTP.
- Eventual with CRDT for counters/likes.
- Outbox plus CDC for at-least-once events without dual-write bugs.
- Honestly: no silver bullet; mix patterns in one system by bounded context.

## How it works

```go
type Outbox struct {
	ID    int64
	Topic string
	Payload []byte
	Sent  bool
}

func (s *Service) CreateOrder(ctx context.Context, o Order) error {
	return s.db.Tx(ctx, func(tx Tx) error {
		if err := tx.InsertOrder(o); err != nil {
			return err
		}
		return tx.InsertOutbox(Outbox{Topic: "order.created", Payload: o.Bytes()})
	})
}
```

The same transaction persists the order and outbox row; a relay worker publishes then marks Sent.

## Pitfalls

**Saga without idempotency.** Duplicate compensations.

**CRDT where invariants must reject.** Use a leader instead.

**Quorum too small.** Lost writes on partition.

**Outbox relay single-threaded bottleneck.** Shard the outbox.

**Strong everywhere on read path.** Ignores replica value.
