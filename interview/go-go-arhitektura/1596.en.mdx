# Microservice communication patterns

## In short

Sync: HTTP/REST, gRPC over HTTP/2 with protobuf. Async: message broker (Kafka, NATS, RabbitMQ) for decoupling and buffering. Choose by latency, delivery guarantees, and coupling. Go idioms: context deadlines on RPC, idempotent event consumers.

## How to answer in an interview

- Sync when you need an immediate response and simple request/response.
- Async for fan-out, peak smoothing, failure isolation.
- gRPC for internal high throughput; REST for public APIs.
- Honestly: a sync chain of five services sums latency; prefer events or an aggregate API.

## How it works

```go
package main

import (
	"context"
	"fmt"
	"net/http"
	"time"
)

func callBilling(ctx context.Context, baseURL string) error {
	req, _ := http.NewRequestWithContext(ctx, http.MethodGet, baseURL+"/health", nil)
	client := &http.Client{Timeout: 2 * time.Second}
	resp, err := client.Do(req)
	if err != nil {
		return err
	}
	defer resp.Body.Close()
	fmt.Println("status", resp.StatusCode)
	return nil
}

func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
	defer cancel()
	_ = callBilling(ctx, "http://billing:8080")
}
```

Context propagates the deadline. Async path: publish `OrderCreated` to a topic; inventory consumer processes independently.

## Pitfalls

**No timeout on HTTP client.** Cascading hang.

**Chatty sync microservices.** N+1 network calls.

**At-least-once without idempotency.** Duplicate side effects.

**Shared proto without versioning.** Breaking wire compat.

**Broker as hidden DB.** Event log grows unbounded.
