# Primary (Master) в replication

## Суть

Primary (master) принимает все writes, пишет WAL/binlog, streaming replicas apply changes. Reads могут идти на replicas. Failover promotes replica to new primary. Single writer invariant в classic setup.

## Что ответить на собесе

- Writes only on primary; replicas read-only или async apply.
- Replication lag между primary и standby.
- Automatic failover tools: Patroni, orchestrator.
- Честно: термин slave устарел; говорю replica/standby.

## Как это работает

```
App --write--> Primary --binlog--> Replica1
                              \--> Replica2
App --read-----> Replica1 (optional)
```

Primary генерирует sequential log entries; replica IO thread fetch, SQL thread apply.

## Подводные камни

**Split brain two primaries.** Needs quorum/fencing.

**Read stale on replica.** Business logic assumes fresh.

**Failover without connection pool drain.** Apps hit old primary.

**Long transaction blocks replication.** Lag spikes.

**Backup from replica without coordination.** Inconsistent snapshot.
