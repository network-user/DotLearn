# Sharding vs replication

## Суть

Sharding splits data horizontally by key across nodes (each shard unique subset). Replication copies same dataset to multiple nodes for read scale and HA. Sharding scales write volume; replication improves read throughput and failover. Often combined: shards each with replica set.

## Что ответить на собесе

- Sharding: partition by tenant_id hash.
- Replication: primary + standby same data.
- Cross-shard query expensive; replica lag on reads.
- Честно: replicate first, shard when single node limits hit.

## Как это работает

```
Sharding:
  user A -> shard 1
  user B -> shard 2

Replication:
  write -> primary -> binlog -> replica (same rows)
```

Go router picks shard; connection pool per shard primary/replica.

## Подводные камни

**Shard without replication.** Shard loss = partial data loss.

**Replicate without need.** Ops cost, no write scale.

**Confuse partition with backup.** Replica not backup if bad write propagates.

**Global secondary index cross shard.** Extra service needed.

**Reshard + replicate simultaneously.** Migration nightmare without plan.
