# Сетевые протоколы: что QA должен назвать на tech-собесе?

## Суть

Протокол это правила формата и порядка обмена данными на каждом уровне стека. QA чаще всего сталкивается с HTTP/HTTPS, DNS, TCP/UDP, WebSocket, SMTP/IMAP на интеграциях, DHCP в корпоративной сети, SNMP на инфраструктурных проектах. Знание протокола подсказывает, где искать баг и какой инструмент взять.

## Что ответить на собесе

- HTTP/HTTPS: веб и REST API; проверяю методы, коды, заголовки, TLS.
- DNS: имя в IP; dig перед curl при «не резолвится».
- TCP vs UDP: надёжность vs скорость (видео, VoIP, QUIC поверх UDP).
- SMTP/IMAP/POP3: почтовые нотификации продукта; тестовый Mailhog/Mailosaur.
- FTP/SFTP: legacy интеграции файлов; бинарный режим и passive/active порты.
- Честно: перечисляю протоколы связкой «симптом → инструмент», а не абстрактным списком RFC.

## Как это работает

Карта для triage:

| Протокол | Порт (типично) | Симптом сбоя | Проверка QA |
|----------|----------------|--------------|-------------|
| HTTPS | 443/TCP | 401, CORS, SSL error | curl, DevTools |
| DNS | 53/UDP | NXDOMAIN, медленный TTFB | dig |
| WebSocket | 443/wss | disconnect, 400 upgrade | DevTools WS tab |
| SMTP | 25/587/TCP | письмо не пришло | telnet, Mailhog |
| DHCP | 67/68/UDP | нет IP в офисе | ipconfig, не UI баг |

Gherkin интеграционного smoke:

```gherkin
Feature: Ключевые протоколы продукта

  Scenario: REST over HTTPS
    When клиент GET "https://api.example.com/v1/items"
    Then статус 200 и Content-Type application/json

  Scenario: WebSocket push заказа
    Given клиент открывает wss соединение к "/orders/stream"
    When создаётся заказ через REST
    Then в WS приходит event "order.created" с тем же id

  Scenario: Email уведомление через SMTP sink
    When пользователь регистрируется
    Then в тестовом SMTP inbox появляется письмо с subject "Confirm email"
```

SNMP и DHCP реже в ручном web QA, но полезны при тестировании IoT/enterprise VPN сценариев.

## Подводные камни

- **Путают HTTP и TCP.** «TCP 404» формулировка некорректна; 404 это HTTP.
- **Игнор UDP в realtime.** WebRTC проблемы не ловятся curl.
- **SMTP только happy path.** SPF/DKIM на staging другой; письмо в spam не проверяют.
- **FTP active mode за NAT.** Работает локально, падает у клиента; нужен passive.
- **Зубрёжка без связи с продуктом.** Список из 20 протоколов без контекста приложения.
