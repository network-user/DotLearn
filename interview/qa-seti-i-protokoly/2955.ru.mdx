# IP-адрес: как QA использует IPv4, IPv6 и маршрутизацию в багах?

## Суть

Internet Protocol задаёт адресацию и доставку пакетов без гарантии порядка (это добавляет TCP). IPv4 (32 бита, ~4 млрд адресов) и IPv6 (128 бит) идентифицируют интерфейс в сети. QA сталкивается с IP при whitelist ACL, geo по IP, dual-stack DNS, NAT и «работает по IP, не работает по домену».

## Что ответить на собесе

- IPv4 запись dotted decimal (192.0.2.10); частные диапазоны 10/8, 172.16/12, 192.168/16 не маршрутизируются в интернет.
- IPv6 сокращённая запись, часто dual-stack: A и AAAA записи; браузер может предпочесть v6.
- Маршрутизаторы пересылают по longest prefix match; фрагментация при MTU может ломать большие UDP пакеты.
- Честно: «пингую IP» не заменяет проверку SNI/TLS по hostname; сертификат привязан к имени, не к IP.

## Как это работает

Путь пакета:

```
Client 203.0.113.5 -> Router -> ISP -> Target 198.51.100.20:443
```

Gherkin dual-stack и ACL:

```gherkin
Feature: IP адресация и доступ

  Scenario: Домен резолвится в A и AAAA
    When QA dig A и AAAA для "app.example.com"
    Then обе записи существуют на dual-stack стенде
    And curl по https://app.example.com успешен с обеих семей

  Scenario: Whitelist блокирует неразрешённый IP
    Given API разрешает только подсеть 198.51.100.0/24
    When запрос идёт с IP 203.0.113.99
    Then API возвращает 403
    And в логах зафиксирован denied client IP

  Scenario: Прямой доступ по IP без SNI не принимается
    When клиент открывает "https://198.51.100.20/" с Host по IP
    Then TLS handshake падает или возвращается default vhost 404
    And ожидаемое поведение документировано
```

Приватный IP в Docker/k8s (10.x) объясняет, почему сервис недоступен с ноутбука без port-forward.

## Подводные камни

- **Hardcode IPv4 в конфиге клиента.** Ломается при смене CDN anycast.
- **AAAA опубликован, v6 маршрут битый.** Timeout только у части пользователей.
- **NAT hairpin.** Внутри LAN домен не резолвится на public IP.
- **GeoIP по устаревшей базе.** VPN exit в «неожиданной» стране, ложный fraud block.
- **Путаница MAC и IP.** L2 vs L3; ARP проблема не лечится сменой DNS.
