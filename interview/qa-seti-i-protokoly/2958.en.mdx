# IP vs MAC: why should QA tell L3 and L2 addresses apart?

## In short

MAC (48-bit, burned-in or assigned) works inside a broadcast domain: Ethernet/Wi‑Fi frame delivery to a neighbor. IP (logical, DHCP or static) routes between networks over the internet. ARP links "I know the neighbor IP" to "I need the MAC". QA rarely debugs ARP but understands why changing IP does not change MAC and vice versa.

## How to answer in an interview

- MAC is unique per NIC, changes with adapter swap; IP may change every home router reconnect.
- Routers look at IP headers; LAN switches use MAC tables.
- Wi‑Fi captive portals and DHCP reservations bind to MAC on office/guest routers.
- Honestly: in cloud web QA MAC is almost invisible; IP, X-Forwarded-For, and app device id matter more.

## How it works

Sending to 203.0.113.8 from home LAN:

1. PC knows destination is off-subnet → gateway 192.168.1.1.
2. ARP: "who has 192.168.1.1?" → router MAC aa:bb:cc:dd:ee:ff.
3. Ethernet frame: dst MAC router, payload IP packet to 203.0.113.8.
4. Routers forward by IP; on last hop ARP to destination host.

Guest Wi‑Fi and ACL Gherkin:

```gherkin

  Scenario: Guest portal assigns IP via DHCP
    When the device joins guest Wi‑Fi
    Then DHCP assigns IP from pool 10.88.0.0/24
    And client MAC appears in AP admin panel

  Scenario: MAC block after abuse
    Given client MAC "02:11:22:33:44:55" is blocked
    When the same MAC reconnects
    Then captive portal denies access even with a new IP

  Scenario: API rate limit by IP is not MAC
    When two devices share one NAT public IP
    Then rate limit applies to the shared IP
    And QA documents expected office behavior
```

## Pitfalls

- **Random MAC on iOS/Android.** Guest rules "remember MAC" flake after privacy rotation.
- **Confusing ARP and DNS.** "IP won't ping" fixed by DNS change without ARP/neighbor check.
- **VIP/failover IP.** MAC may change on failover, IP stays; brief disconnect.
- **Docker bridge MAC.** Not the host MAC; confusion in local debug.
- **Answer "MAC is IP v2".** Wrong analogy; different OSI layers.
