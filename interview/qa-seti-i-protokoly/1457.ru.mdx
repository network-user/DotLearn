# TCP или UDP: когда какой транспорт выбирают в проде?

## Суть

TCP и UDP работают на транспортном уровне и решают разные задачи доставки данных. TCP строит надёжную сессию с подтверждениями и сохранением порядка пакетов, UDP отправляет датаграммы без гарантий, зато с минимальной задержкой. Для QA это значит: симптомы сетевых дефектов и стратегия проверки зависят от того, какой протокол использует фича.

## Что ответить на собесе

- Разделяю протоколы по контракту доставки: TCP для сценариев, где критичны целостность и порядок (HTTP, API, загрузка файлов), UDP для потоков, где важнее скорость, а потери допустимы (VoIP, игры, DNS).
- На тестировании смотрю не только «ответ пришёл», а поведение при деградации: ретраи и буферизация у TCP, дропы и джиттер у UDP.
- В Charles или DevTools Network сверяю, что приложение ходит ожидаемым транспортом и портом, а не «случайно» через другой стек.
- Честно оговариваю: UDP сам по себе не «быстрее TCP во всём»; выигрыш по latency появляется там, где приложение готово жить без гарантий ОС.

## Как это работает

TCP перед обменом данными проходит трёхшаговое рукопожатие (SYN, SYN-ACK, ACK), после чего каждый сегмент нумеруется. Получатель подтверждает приём, отправитель ретранслирует потерянное, а механизмы flow control и congestion control подстраивают скорость под канал. Поэтому веб, почта и большинство REST API сидят на TCP: потеря байта недопустима.

UDP упаковывает payload в датаграмму с заголовком портов и отправляет без установления сессии. Нет ACK, нет встроенного восстановления порядка: приложение само решает, терпеть ли потери или собирать фрагменты. DNS-запрос, RTP в видеозвонке или игровой тик часто идут так, потому что повторная доставка старого кадра бессмысленна.

С точки зрения ручного QA удобно оформить проверки в Gherkin, привязав их к наблюдаемому поведению клиента:

```gherkin
Feature: Поведение клиента при нестабильной сети

  Scenario: REST-запрос через TCP переживает кратковременный обрыв
    Given включён throttling "High latency" в Charles
    When пользователь отправляет POST /orders
    Then заказ создаётся ровно один раз
    And в логах клиента видны повторные попытки до успеха

  Scenario: UDP-стрим допускает пропуск кадров без зависания UI
    Given симулирована потеря 5% UDP-пакетов
    When пользователь смотрит live-трансляцию
    Then воспроизведение продолжается без полной остановки
    And допускается кратковременная артефактация картинки
```

При регрессии API достаточно curl или Postman: если это HTTPS, транспорт уже TCP. Для UDP-сервисов (QUIC поверх UDP, игровой сервер) нужны специализированные клиенты или прокси с фильтрацией пакетов. В матрице рисков отмечаю: для TCP проверяю таймауты и идемпотентность ретраев, для UDP проверяю деградацию качества, а не «всё или ничего».

## Подводные камни

- **Путаница уровней.** TCP/UDP это транспорт; HTTP, gRPC и WebSocket сидят выше. На собесе путают «REST поверх UDP» с реальным стеком: типичный REST всё равно едет в TCP (или QUIC, но это отдельная история).
- **Ретраи TCP маскируют баги.** Клиент «в итоге получил 200», но пользователь ждал 30 секунд из-за агрессивных ретраев на сервере с 500. Нужны метрики latency, не только статус.
- **UDP без прикладной логики ломает UX.** Если разработчики не заложили буфер или FEC, один потерянный пакет может означать дырку в аудио или рассинхрон состояния в игре.
- **Firewall и NAT.** UDP часто режут корпоративные прокси; TCP 443 проходит почти везде. Сценарий «работает в офисе, не работает у клиента» часто упирается в транспорт.
- **Нагрузочное тестирование сравнивает несопоставимое.** Прогон HTTP и игрового UDP одним инструментом без учёта семантики доставки даёт ложные выводы о «производительности протокола».
