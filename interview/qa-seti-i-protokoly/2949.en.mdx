# OSI vs TCP/IP: how does QA map a failure to a stack layer?

## In short

OSI is a seven-layer teaching model; TCP/IP is the four-layer stack used on the internet. QA uses them as a checklist: DNS/HTTP issues at application, packet loss at transport, bad routing at network, cable/Wi‑Fi at data link/physical. Mapping speeds triage between "site is down" and "one client VPN blocks the port".

## How to answer in an interview

- OSI: L1 physical, L2 data link (MAC, Ethernet), L3 network (IP, routing), L4 transport (TCP/UDP), L5 session, L6 presentation (TLS often discussed here), L7 application (HTTP, DNS).
- TCP/IP: Network Access (L1+L2), Internet (IP), Transport (TCP/UDP), Application (HTTP, SMTP, etc.).
- In practice I treat curl/DNS as L7, telnet/nc to a port as L4, ping/traceroute as L3, link down as L1/L2.
- Honestly: in incidents you rarely need the perfect OSI number; separating app vs network vs client environment matters more.

## How it works

Triage mapping:

| Symptom | Likely layer | QA tool |
|---------|--------------|---------|
| 404 on URL | L7 HTTP | curl, DevTools |
| Connection refused | L4/L7 port closed | nc, telnet |
| Timeout with no response | L3/L4 firewall, routing | traceroute, ping |
| SSL handshake fail | L6/L7 TLS | openssl s_client |
| No link | L1/L2 | different Wi‑Fi, cable |

Gherkin for stand smoke:

```gherkin
Feature: Network stack and API availability

  Scenario: Application layer responds
    When QA sends GET "https://api.example.com/health"
    Then the response status is 200
    And the body contains status "ok"

  Scenario: Transport accepts TCP on 443
    When QA checks TCP to "api.example.com:443"
    Then the connection opens in under 3 seconds

  Scenario: DNS resolves the host
    When QA resolves "api.example.com"
    Then at least one A or AAAA record is returned
```

For "works in office, fails at home" I first rule out L7 (correct host header), then L3 (split tunnel VPN), then L2 (Wi‑Fi MTU).

## Pitfalls

- **Memorizing OSI without tools.** Layer numbers without ping/curl do not reproduce the bug.
- **TLS only on "layer 6".** In TCP/IP TLS is often discussed inside application; arguing the number does not fix the incident.
- **Ignoring MTU/fragmentation.** L3 ping works while large L7 POST fails.
- **Mixing models in one answer.** Confusing L2 MAC with L3 IP in junior triage.
- **DNS happy path only.** Resolver cache hides a broken secondary NS.
