# Network protocols: what should QA name in a technical interview?

## In short

A protocol defines format and exchange rules at each stack layer. QA most often hits HTTP/HTTPS, DNS, TCP/UDP, WebSocket, SMTP/IMAP on integrations, DHCP on corporate networks, SNMP on infrastructure projects. Knowing the protocol points to where the bug lives and which tool to use.

## How to answer in an interview

- HTTP/HTTPS: web and REST API; I check methods, codes, headers, TLS.
- DNS: name to IP; dig before curl when "won't resolve".
- TCP vs UDP: reliability vs speed (video, VoIP, QUIC over UDP).
- SMTP/IMAP/POP3: product email notifications; test Mailhog/Mailosaur.
- FTP/SFTP: legacy file integrations; binary mode and passive/active ports.
- Honestly: I list protocols as "symptom → tool", not an abstract RFC laundry list.

## How it works

Triage map:

| Protocol | Port (typical) | Failure symptom | QA check |
|----------|----------------|-----------------|----------|
| HTTPS | 443/TCP | 401, CORS, SSL error | curl, DevTools |
| DNS | 53/UDP | NXDOMAIN, slow TTFB | dig |
| WebSocket | 443/wss | disconnect, 400 upgrade | DevTools WS tab |
| SMTP | 25/587/TCP | mail missing | telnet, Mailhog |
| DHCP | 67/68/UDP | no IP in office | ipconfig, not a UI bug |

Integration smoke Gherkin:

```gherkin
Feature: Core product protocols

  Scenario: REST over HTTPS
    When the client GET "https://api.example.com/v1/items"
    Then status is 200 and Content-Type is application/json

  Scenario: WebSocket order push
    Given the client opens wss to "/orders/stream"
    When an order is created via REST
    Then WS receives event "order.created" with the same id

  Scenario: Email via SMTP sink
    When the user registers
    Then the test SMTP inbox has mail with subject "Confirm email"
```

SNMP and DHCP are rare in manual web QA but matter for IoT/enterprise VPN scenarios.

## Pitfalls

- **Confusing HTTP and TCP.** "TCP 404" is wrong wording; 404 is HTTP.
- **Ignoring UDP in realtime.** WebRTC issues are not caught by curl.
- **SMTP happy path only.** SPF/DKIM differ on staging; spam folder never checked.
- **FTP active mode behind NAT.** Works locally, fails for client; need passive.
- **Memorized list without product link.** Twenty protocols with no app context.
