# IPv6: what changes for testing web applications?

## In short

IPv6 expands the address to 128 bits (eight hextet groups), easing the shortage of public IPv4. Notation is shortened with `::` once. Dual-stack hosts often have both A and AAAA; the browser uses Happy Eyeballs to pick the faster stack. QA checks the product on v6-only, v4-only, and both.

## How to answer in an interview

- SLAAC and DHCPv6 assign addresses without classic NAT; firewall rules differ from 192.168.x.
- Path `2001:db8::1` in a URL needs square brackets: `https://[2001:db8::1]/`.
- Mobile carriers are already v6-first; "Wi-Fi only" bugs are often missing AAAA on the CDN.
- Honestly: a full v6 lab is rare; at minimum I run curl `-6` and `-4` on dual-stack staging.

## How it works

Comparing stacks on one host:

```
dig A     api.example.com  → 203.0.113.10
dig AAAA  api.example.com  → 2001:db8:85a3::7334
```

Gherkin dual-stack smoke:

```gherkin
Feature: API reachability over IPv4 and IPv6

  Background:
    Given endpoint "/health" returns 200

  Scenario: Request over IPv4
    When QA runs curl -4 https://api.example.com/health
    Then status is 200
    And response time is within SLO

  Scenario: Request over IPv6
    When QA runs curl -6 https://api.example.com/health
    Then status is 200
    And body matches the v4 response schema

  Scenario: Missing AAAA does not break v4 clients
    Given DNS has only an A record
    When the user on a v4-only network opens the site
    Then the page loads without fallback delay
```

On Linux `ip -6 addr` shows link/global scope; in Charles I check traffic does not go to link-local instead of global.

## Pitfalls

- **NAT64/DNS64.** macOS/iOS translate transparently; bug only on cellular.
- **Firewall open on v4, closed on v6.** Half of users "cannot connect".
- **IPv6 literal without brackets in config.** URL parser breaks.
- **Privacy extension rotation.** Correlating logs by IP is harder, not a product bug.
- **MTU and fragmentation.** Large UDP over v6 through a tunnel causes black hole.
