# DNS: как доменное имя превращается в IP-адрес?

## Суть

DNS (Domain Name System) сопоставляет человекочитаемые имена хостов с числовыми IP-адресами, по которым маршрутизаторы доставляют пакеты. Без резолва браузер не знает, куда открыть TCP-соединение. Для QA сбои DNS выглядят как «сайт недоступен» при живом origin и наоборот: UI грузится, API падает из-за другого A-записи.

## Что ответить на собесе

- DNS это распределённый справочник: клиент спрашивает резолвер, тот идёт по цепочке root → TLD → authoritative.
- Ответ кэшируется на TTL в браузере, ОС и у провайдера; stale cache объясняет «у меня работает, у вас нет».
- Типы записей (A, AAAA, CNAME, MX) влияют на тесты: CNAME на CDN меняет IP без деплоя приложения.
- Честно: правлю hosts или stub-резолвер только на тестовом стенде; в проде без согласования с SRE не трогаю.

## Как это работает

Типичный путь запроса `app.example.com`:

```
Браузер → кэш ОС → рекурсивный резолвер ISP
  → root (.)
    → TLD (.com)
      → NS example.com
        → A 203.0.113.10
```

Gherkin для smoke DNS в регрессии:

```gherkin
Feature: Резолв домена перед UI smoke

  Background:
    Given staging host "app.staging.example.com"

  Scenario: A-запись резолвится до открытия логина
    When QA выполняет nslookup для staging host
    Then ответ содержит минимум один IPv4 адрес
    And TTL больше 0

  Scenario: После смены DNS UI подхватывает новый IP в пределах TTL
    Given authoritative A-запись изменена на 198.51.100.5
    And TTL записи 60 секунд
    When QA ждёт 2 * TTL
    And открывает staging host в инкогнито
    Then TCP connect идёт на 198.51.100.5

  Scenario: NXDOMAIN блокирует загрузку с понятной ошибкой
    Given домен "missing.staging.example.com" без записи
    When пользователь открывает URL
    Then браузер показывает ошибку DNS
    And приложение не пытается fallback на дефолтный API
```

В DevTools Network смотрю, что первый запрос уходит уже на IP; в Charles можно подменить ответ резолвера для негативных кейсов.

## Подводные камни

- **Split-horizon DNS.** Внутренний и внешний IP разные; тест с VPN и без даёт разный результат.
- **Короткий TTL + CDN.** Кэш ещё старый, origin уже новый; flaky smoke после релиза.
- **CNAME chain.** Лишний hop увеличивает latency; таймауты в мобильной сети.
- **IPv6 только AAAA.** Клиент без v6 «не видит» сайт, хотя A есть на другом поддомене.
- **DoH/DoT у браузера.** Системный nslookup и Chrome резолвят по-разному.
