# How this is determined in JavaScript

## In short

`this` is set by how a function is called, not where it is declared (except arrows and `bind`). In strict mode a global call yields `undefined`; in a browser without strict it is `window`. Method call `obj.run()` gets `this === obj`. `new Fn()` uses a fresh object as `this`. `call/apply/bind` set `this` explicitly.

## How to answer in an interview

- Lost `this` on `const fn = obj.run; fn()` is fixed with bind or an arrow wrapper.
- In a class field arrow method, `this` is lexical, handy in page-object callbacks.
- In an event listener, a regular function sets `this` to the target element.
- For spies I assert the method ran with the expected context (`toHaveBeenCalledWith`).
- Honestly: double bind cannot be undone without keeping the original; teardown in `afterEach` is mandatory.

## How it works

Object method:

```
const panel = {
    label: "Orders",
    title() {
        return this.label;
    },
};
panel.title();
```

Lost context:

```
const read = panel.title;
read();
```

Restore:

```
const read = panel.title.bind(panel);
```

Context model in pytest:

```python
class Panel:
    def __init__(self, label: str):
        self.label = label

    def title(self) -> str:
        return self.label


def test_method_receives_instance_context():
    panel = Panel("Orders")
    assert panel.title() == "Orders"
```

An arrow inside a method inherits the method’s `this`, not the DOM target.

## Pitfalls

**Callback without bind in setTimeout.** `this` is global/undefined; state asserts fail.

**Destructuring a method.** `const { title } = panel` detaches context.

**Strict vs sloppy global.** Different `this` in helpers; tests flake between node and browser.

**jest.spyOn without restore.** A patched method carries wrong `this` into the next spec.

**Class method as handler.** Use a field arrow or constructor bind.
