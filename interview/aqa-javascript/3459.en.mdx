# null vs 0 in API responses: how to assert in tests

## In short

`null` means an intentional absence of value; `0` is a valid number. In JSON, `"count": 0` differs from a missing key or `"count": null`. In JS, `null == 0` is false, but both are falsy, which confuses guards in client code and tests.

## How to answer in an interview

- For a nullable counter I check `value === null` separately from `value === 0`.
- In schema tests (Zod/OpenAPI) I distinguish optional, nullable, and required number.
- SQL export: a NULL column is not the same as 0 in an aggregate.
- In a UI table, an empty cell may mean 0, a dash, or backend null.
- Honestly: `if (count)` in prod hides the “zero orders” bug; a `toBeTruthy` test stays green too.

## How it works

JS checks:

```
const missing = null;
const zero = 0;
console.log(missing === zero);
console.log(Boolean(missing), Boolean(zero));
```

API contract:

```
{ "items": null }
{ "items": 0 }
```

A pytest response validator:

```python
def normalize_items(payload: dict) -> str:
    if "items" not in payload:
        return "missing_key"
    value = payload["items"]
    if value is None:
        return "null"
    if value == 0:
        return "zero"
    return "other"


def test_null_vs_zero_semantics():
    assert normalize_items({}) == "missing_key"
    assert normalize_items({"items": None}) == "null"
    assert normalize_items({"items": 0}) == "zero"
```

## Pitfalls

**Coalesce `??` vs `||`.** `0 ?? 1` stays 0; `0 || 1` becomes 1.

**Default in destructuring.** `const { n = 0 } = obj` fills 0 for undefined, not for null.

**GraphQL non-null Int.** null in the response breaks the whole query; 0 is valid.

**toEqual without type.** String `"0"` passes business logic with loose comparison.

**SQL aggregates.** SUM of an empty set is NULL; COUNT is 0.
