# Тернарный оператор в JS и читаемость автотестов

## Суть

Тернарный оператор `condition ? a : b` возвращает одно из двух выражений. В тестах его используют для коротких веток, но вложенные тернарники ухудшают diff и review. В Python аналог: `a if cond else b`.

## Что ответить на собесе

- Синтаксис: operand1 truthy check, затем value if true, else value if false.
- Отличие от `if/else` statement: тернарник expression, можно inline в return/assign.
- В page object предпочитаю явный if или helper function для статусов UI.
- В pytest parametrization заменяет тернарник таблицей кейсов.
- Честно добавляю: nested ternary в production коде часто copy-paste в тест без всех веток.

## Как это работает

```
const badge = score >= 70 ? "pass" : "fail";
const label = role === "admin" ? "Admin panel" : "User home";
```

Покрываю обе ветки отдельными тестами:

```python
import pytest


def ui_label(role: str, score: int) -> str:
    badge = "pass" if score >= 70 else "fail"
    home = "Admin panel" if role == "admin" else "User home"
    return f"{home}:{badge}"


@pytest.mark.parametrize(
    "role, score, expected",
    [
        ("admin", 80, "Admin panel:pass"),
        ("user", 50, "User home:fail"),
    ],
)
def test_ui_label_branches(role, score, expected):
    assert ui_label(role, score) == expected
```

Для границы `score == 70` добавляю отдельный кейс boundary.

## Подводные камни

**Вложенный тернарник.** Невозможно быстро понять fail screenshot vs expected.

**Truthy pitfalls.** `"0"` и `[]` ведут себя иначе, чем ожидает автор теста.

**Side effects внутри веток.** Дважды вызывается API при refactor.

**Тернарник вместо guard clause.** Скрытые throw не покрыты.

**Copy expected из minified bundle.** Неверная ветка после минификации.
