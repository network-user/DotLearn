# Event loop and assertion order in JS tests

## In short

The event loop in browser and Node keeps a single JS thread: the call stack runs sync code, Web APIs/libuv handle timers and I/O, callbacks land in queues (macrotask/microtask). When the stack is empty, the loop drains microtasks (Promise) before the next macrotask (`setTimeout`). Flakiness often comes from expecting the wrong order.

## How to answer in an interview

- Sync code always finishes before a zero-delay timer callback... except microtasks in between.
- `setTimeout(fn, 0)` does not run instantly after a `fetch` thenable.
- In tests I wait on a condition (Playwright `expect.poll`), not a fixed sleep.
- In Node API suites I split unit without I/O and integration with real loop timing.
- Honestly: `queueMicrotask` and `Promise.resolve().then` can beat `setTimeout(0)` and break naive log-order asserts.

## How it works

Order demo:

```
console.log("A");
setTimeout(() => console.log("B"), 0);
Promise.resolve().then(() => console.log("C"));
console.log("D");
```

Output: A, D, C, B. The timer went to Web APIs; `then` entered the microtask queue.

In pytest I pin the expected sequence of test-log events:

```python
def test_expected_async_log_order():
    events = ["sync_start", "sync_end", "microtask", "macrotask"]
    assert events.index("microtask") < events.index("macrotask")
    assert events.index("sync_end") < events.index("microtask")
```

## Pitfalls

**Hard sleep 1000ms.** Hides bugs and slows the pipeline.

**Blocking while in a test.** Freezes the tab/process; timers do not tick.

**Awaiting only the first then.** The second microtask has not run yet.

**requestAnimationFrame in headless.** Frames are throttled; paint asserts flake.

**Parallel Jest workers.** Shared timer mocks need `jest.isolateModules` for isolation.
