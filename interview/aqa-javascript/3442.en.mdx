# The ternary operator in JS and test readability

## In short

The ternary operator `condition ? a : b` returns one of two expressions. Tests use it for short branches, but nested ternaries hurt diffs and review. The Python equivalent is `a if cond else b`.

## How to answer in an interview

- Syntax: evaluate operand1, then value if truthy, else value if falsy.
- Unlike an `if/else` statement: ternary is an expression, usable inline in return/assign.
- In page objects I prefer explicit if or a helper for UI states.
- In pytest parametrization replaces ternary with a case table.
- I add honestly: nested ternary in production often gets copy-pasted into tests without all branches.

## How it works

```
const badge = score >= 70 ? "pass" : "fail";
const label = role === "admin" ? "Admin panel" : "User home";
```

I cover both branches with separate tests:

```python
import pytest


def ui_label(role: str, score: int) -> str:
    badge = "pass" if score >= 70 else "fail"
    home = "Admin panel" if role == "admin" else "User home"
    return f"{home}:{badge}"


@pytest.mark.parametrize(
    "role, score, expected",
    [
        ("admin", 80, "Admin panel:pass"),
        ("user", 50, "User home:fail"),
    ],
)
def test_ui_label_branches(role, score, expected):
    assert ui_label(role, score) == expected
```

For the boundary `score == 70` I add a dedicated case.

## Pitfalls

**Nested ternary.** Hard to map fail screenshot to expected.

**Truthy pitfalls.** `"0"` and `[]` behave unlike the author expects.

**Side effects inside branches.** API called twice after refactor.

**Ternary instead of guard clause.** Hidden throws stay untested.

**Expected copied from minified bundle.** Wrong branch after minification.
