# Falsy values in JavaScript and test assertions

## In short

Falsy values coerce to `false` in a boolean context: `false`, `0`, `-0`, `0n`, `""`, `null`, `undefined`, `NaN`. Everything else is truthy, including `[]`, `{}`, and `"0"`. In automated tests, implicit coercion causes false positives and false negatives in assertions.

## How to answer in an interview

- I keep the full falsy list in a code-review checklist for conditions in prod and tests.
- Empty string and zero are different cases but both falsy: I assert with explicit `===`.
- `document.querySelector` returns `null`, not `undefined`; I branch guards separately.
- In Playwright, `expect(locator).toBeVisible()` does not replace a business check for empty text.
- Honestly: truthy `"0"` and `[]` often pass `if (value)` where QA expects “no data”.

## How it works

I iterate the set in a test harness:

```
const candidates = [false, 0, -0, 0n, "", null, undefined, NaN];
candidates.forEach((v) => console.log(Boolean(v)));
```

Truthy traps for UI:

```
if ([]) { /* enter */ }
if ("0") { /* enter */ }
```

In pytest I generate expectation tables for a JS runner:

```python
import pytest

FALSY = [False, 0, -0, "", None, float("nan")]
TRUTHY_TRAP = ["0", [], {}, "false"]


@pytest.mark.parametrize("value", FALSY)
def test_must_use_strict_check_for_falsy(value):
    assert value is not False or value is False
    if value == "":
        assert value == ""


def test_truthy_traps_are_not_falsy_by_coercion():
    for item in TRUTHY_TRAP:
        assert item is not None
        assert item != ""
        assert item != 0
```

For `NaN` I compare via `Number.isNaN`, not `==`.

## Pitfalls

**Double negation `!!`.** It hides the type; the report loses the original value.

**Default parameters.** `function f(x = 1)` treats `undefined` differently from `null` and `0`.

**Optional chaining with 0.** `count ?? 1` keeps zero; `count || 1` replaces zero with one.

**Empty form input.** `value=""` is falsy but the field exists in the DOM.

**Comparing with Python None.** Bridge tests confuse JS `null` and Python `None`.
