# var, let, and const: scope and hoisting

## In short

`var` binds to a function or the global object; hoisting initializes it to `undefined` before the declaration line. `let` and `const` are block-scoped and sit in the temporal dead zone until declared: early access throws ReferenceError. `const` forbids rebinding the reference, but mutating object or array fields is allowed.

## How to answer in an interview

- In new Jest tests I use only `let`/`const`; I keep `var` when auditing legacy suites.
- A loop with `var` and setTimeout is the classic bug; fix with `let` or an IIFE.
- `const` for page object selector maps, `let` for a mutable counter in a loop.
- ESLint `no-var` and `prefer-const` in CI catch regressions.
- Honestly: `const` does not make an object immutable; use `Object.freeze` in fixtures when needed.

## How it works

var hoisting:

```
console.log(tag); // undefined
var tag = "smoke";
```

let TDZ:

```
{
  // console.log(step); ReferenceError
  let step = 1;
}
```

const and mutation:

```
const selectors = { submit: "#send" };
selectors.submit = "#submit-btn"; // ok
// selectors = {} // TypeError
```

Cheatsheet in pytest:

```python
SCOPES = {
    "var": "function or global",
    "let": "block",
    "const": "block, binding frozen",
}


def test_scope_cheatsheet():
    assert SCOPES["let"] == "block"
    assert "frozen" in SCOPES["const"]
```

Loop trap:

```
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0); // 3,3,3
}
for (let j = 0; j < 3; j++) {
  setTimeout(() => console.log(j), 0); // 0,1,2
}
```

## Pitfalls

**var in if without a block pre-ES6.** One variable for the whole function.

**const in for-in.** Syntax error; use let.

**Shadowing a function parameter.** Inner `let x` duplicates the arg and confuses debugging.

**global var in jsdom.** Leaks between tests without isolateModules.

**Re-declaring var.** A second var in the same function is legal and hides intent.
