# Arrow functions without their own this and arguments

## In short

An arrow function lexically inherits `this` and `arguments` from the outer function and creates neither of its own. A regular function gets `this` at call time. In page objects, arrows work well in timer and handler callbacks but not as methods with dynamic `this`.

## How to answer in an interview

- `this` inside `() =>` matches the enclosing scope, not the object left of the dot.
- `arguments` is missing in arrows; I use rest `(...params)`.
- I do not call arrows with `new`; they have no `prototype` for construction.
- In a test fixture class I keep regular functions or class fields as methods.
- Honestly: an arrow as `{ tick: () => this.count }` breaks expected `this` when passed by reference.

## How it works

A regular function problem in a timer:

```
function Counter() {
    this.count = 0;
    setInterval(function () {
        this.count += 1;
    }, 100);
}
```

An arrow keeps Counter `this`:

```
function Counter() {
    this.count = 0;
    setInterval(() => {
        this.count += 1;
    }, 100);
}
```

Rest instead of arguments:

```
const summarize = (...parts) => parts.join("|");
```

I verify binding in pytest with a simple model:

```python
class Counter:
    def __init__(self):
        self.count = 0

    def tick_lexical(self, delta):
        self.count += delta
        return self.count


def test_lexical_method_updates_state():
    c = Counter()
    assert c.tick_lexical(2) == 2
    assert c.tick_lexical(1) == 3
```

## Pitfalls

**Arrow in addEventListener on a class instance.** `this` is not the instance unless bound.

**Method in an object literal.** `this` is lost when destructuring `{ tick }`.

**call/apply/bind are useless.** Arrow `this` does not change.

**Inheriting outer arguments.** A nested arrow reads the parent’s `arguments`, not its own parameters.

**Arrow as a prototype method.** You cannot override `this` for a subclass hook.
