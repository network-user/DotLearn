# JavaScript closures for test factories

## In short

A closure means an inner function keeps access to lexical bindings from an outer function after the outer finishes. Variables live in the closure environment while any reference to the function exists. In AQA this powers data factories, id counters, and private state in page object helpers.

## How to answer in an interview

- `createUserFactory()` returns a function that increments a hidden seq without exposing it.
- In a loop with setTimeout, `let` creates a per-iteration binding and avoids the closure bug.
- Legacy fixtures used the module pattern: IIFE plus a public API return.
- Memory leak: a closure holding a large DOM snapshot must be cleared in afterEach.
- Honestly: closing over a mutable object shared across tests causes order-dependent flakes.

## How it works

Case id counter:

```
function createCaseId() {
  let seq = 0;
  return () => {
    seq += 1;
    return `case-${seq}`;
  };
}

const nextId = createCaseId();
console.log(nextId(), nextId()); // case-1 case-2
```

Private log buffer:

```
function makeLogger() {
  const lines = [];
  return {
    push(msg) {
      lines.push(msg);
    },
    dump() {
      return [...lines];
    },
  };
}
```

Python analogue:

```python
def create_case_id():
    seq = {"n": 0}

    def next_id() -> str:
        seq["n"] += 1
        return f"case-{seq['n']}"

    return next_id


def test_closure_factory():
    gen = create_case_id()
    assert gen() == "case-1"
    assert gen() == "case-2"
```

## Pitfalls

**Closure in class field arrow.** `this` is lexical, unlike prototype methods.

**for var with async.** Every callback sees the final i.

**Debugger closure scope.** DevTools may show stale bindings after hot reload.

**jest.isolateModules.** Two imports of the same factory get different closures; state is not shared.

**Circular closure.** A DOM node holds a handler that holds the node; teardown must be explicit.
