# NaN in JavaScript and correct assertion checks

## In short

`NaN` (Not a Number) has type `number` but is not equal to itself: `NaN === NaN` is false. It appears from invalid math or `Number("abc")`. For checks I use `Number.isNaN(value)`; global `isNaN` coerces types and causes false positives.

## How to answer in an interview

- `0/0` and `parseInt("foo")` yield NaN, not an exception.
- In asserts you cannot `expect(x).toBe(NaN)`; use `Number.isNaN(x)` or a `toBeNaN` matcher where available.
- `isNaN("12px")` is true due to parsing; `Number.isNaN("12px")` is false.
- In API contract tests I distinguish a missing field from numeric NaN.
- Honestly: JSON does not serialize NaN; the client may get `null` instead of expected NaN.

## How it works

NaN sources:

```
const fromMath = 0 / 0;
const fromParse = Number("not-a-number");
```

Checks:

```
console.log(isNaN("abc"));
console.log(Number.isNaN("abc"));
console.log(Number.isNaN(fromMath));
```

In pytest I validate numeric fields after parsing:

```python
import math


def is_invalid_metric(value) -> bool:
    if value is None:
        return False
    if isinstance(value, float):
        return math.isnan(value)
    return False


def test_metric_validation():
    assert is_invalid_metric(float("nan"))
    assert not is_invalid_metric("abc")
    assert not is_invalid_metric(None)
```

For aggregates I filter NaN before `max()`/`avg`, otherwise the whole result is NaN.

## Pitfalls

**Object.is(NaN, NaN).** True, but teams forget `Object.is` in asserts.

**Sorting with NaN.** Comparison is unstable; element order changes.

**Chart assertions.** NaN breaks scale; screenshot diffs lie.

**parseFloat partial success.** `"12abc"` is not NaN while the test expected NaN.

**Mixing null and NaN in DB export.** CSV empty cell vs NaN string.
