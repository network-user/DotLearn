# Event loop и порядок assert в JS-тестах

## Суть

Event loop в браузере и Node держит один поток JS: call stack выполняет синхронный код, Web API/ libuv уводят таймеры и I/O, колбэки попадают в очереди (macrotask/microtask). Когда стек пуст, loop подтягивает microtask (Promise) раньше следующего macrotask (`setTimeout`). Flaky часто из-за неверного ожидания порядка.

## Что ответить на собесе

- Синхронный код всегда завершается до колбэка таймера с нулевой задержкой... кроме microtask между ними.
- `setTimeout(fn, 0)` не гарантирует мгновенный запуск после `fetch` thenable.
- В тестах жду condition (Playwright `expect.poll`), а не фиксированный sleep.
- В Node API suite разделяю unit без I/O и integration с реальным loop timing.
- Честно: `queueMicrotask` и `Promise.resolve().then` могут опередить `setTimeout(0)` и сломать наивный assert порядка логов.

## Как это работает

Демонстрация порядка:

```
console.log("A");
setTimeout(() => console.log("B"), 0);
Promise.resolve().then(() => console.log("C"));
console.log("D");
```

Вывод: A, D, C, B. Таймер ушёл в Web API, then попал в microtask queue.

В pytest фиксирую ожидаемую последовательность событий тестового лога:

```python
def test_expected_async_log_order():
    events = ["sync_start", "sync_end", "microtask", "macrotask"]
    assert events.index("microtask") < events.index("macrotask")
    assert events.index("sync_end") < events.index("microtask")
```

## Подводные камни

**Жёсткий sleep 1000ms.** Маскирует баг и удлиняет пайплайн.

**Блокирующий while в тесте.** Кладёт весь tab/process, таймеры не тикают.

**await только первого then.** Второй microtask ещё не выполнен.

**requestAnimationFrame в headless.** Кадры throttled, assert на paint flaky.

**Параллельные воркеры Jest.** Общий таймер mock не изолирован без `jest.isolateModules`.
