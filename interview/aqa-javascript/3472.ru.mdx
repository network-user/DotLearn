# Замыкания в JavaScript для тестовых фабрик

## Суть

Замыкание: внутренняя функция сохраняет доступ к лексическим binding внешней функции после того, как внешняя завершилась. Переменные живут в closure environment, пока на функцию есть ссылка. В AQA это основа фабрик данных, счётчиков id и приватного state в page object helper.

## Что ответить на собесе

- `createUserFactory()` возвращает функцию, которая инкрементит скрытый seq и не публикует его наружу.
- В loop с setTimeout `let` создаёт binding на итерацию, без closure bug.
- Module pattern в старых fixture: IIFE + return public API.
- Memory leak: closure держит большой DOM snapshot, очищаю в afterEach.
- Честно: замыкание на mutable object shared между тестами даёт order-dependent flake.

## Как это работает

Счётчик кейсов:

```
function createCaseId() {
  let seq = 0;
  return () => {
    seq += 1;
    return `case-${seq}`;
  };
}

const nextId = createCaseId();
console.log(nextId(), nextId()); // case-1 case-2
```

Приватный буфер логов:

```
function makeLogger() {
  const lines = [];
  return {
    push(msg) {
      lines.push(msg);
    },
    dump() {
      return [...lines];
    },
  };
}
```

Python analogue:

```python
def create_case_id():
    seq = {"n": 0}

    def next_id() -> str:
        seq["n"] += 1
        return f"case-{seq['n']}"

    return next_id


def test_closure_factory():
    gen = create_case_id()
    assert gen() == "case-1"
    assert gen() == "case-2"
```

## Подводные камни

**Closure в class field arrow.** `this` лексический, не как у prototype method.

**for var + async.** Все callback видят финальный i.

**Debugger closure scope.** DevTools показывает устаревший binding после hot reload.

**jest.isolateModules.** Два import одного factory дают разные closures, state не shared.

**Circular closure.** DOM node хранит handler, handler хранит node, teardown нужен явно.
