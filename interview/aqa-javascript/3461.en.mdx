# == vs === in JavaScript test assertions

## In short

`==` compares with type coercion (abstract equality); `===` does not (strict). Examples: `0 == false` is true; `0 === false` is false. In AQA I default to strict matchers and `===`, otherwise string `"5"` and number 5 produce a false green.

## How to answer in an interview

- In Jest/Playwright, `toBe` uses `Object.is`, close to `===`.
- `toEqual` is deep compare but leaf types stay strict.
- `==` is allowed only for explicit legacy contracts documented in the test plan.
- Special cases: `NaN`, `+0/-0` via `Object.is`.
- Honestly: `null == undefined` being true breaks a test that expected only null.

## How it works

```
console.log(5 == "5");
console.log(5 === "5");
console.log(null == undefined);
console.log(null === undefined);
```

Correct asserts:

```
expect(response.status).toBe(200);
expect(payload.count).toBe(0);
```

In pytest I mirror strict policy:

```python
def strict_equal(left, right) -> bool:
    return type(left) is type(right) and left == right


def test_strict_rejects_cross_type():
    assert strict_equal(5, 5)
    assert not strict_equal(5, "5")
```

ESLint `eqeqeq` is enabled in CI for prod and test packages.

## Pitfalls

**Object.is vs ===.** `Object.is(NaN, NaN)` is true; `===` is false.

**Arrays with ==.** Reference compare, not content.

**Coercion in template APIs.** Query string `"true"` is truthy in `if (flag)`.

**Snapshot string/number.** Updating snapshots hides type regression.

**== with objects.** Coerces to `[object Object]`; rare but confusing failures.
