# beforeAll and beforeEach in a JS test runner

## In short

`beforeAll` runs once per describe/suite: heavy setup (DB seed, browser launch). `beforeEach` runs before every `it/test`: fresh context, cookie cleanup, new mock handlers. Jest/Vitest/Mocha share the same idea; hook names differ from JUnit but isolation goals match.

## How to answer in an interview

- `beforeAll` for expensive work; `beforeEach` for test independence.
- Async hooks return a Promise or use `async`, otherwise the runner does not wait.
- `afterEach`/`afterAll` pair to remove stubs and close browser contexts.
- Shared mutable state in `beforeAll` only if read-only after init.
- Honestly: a heavy `beforeEach` (full UI login) doubles pipeline time without parallel sharding.

## How it works

Vitest/Jest style:

```
describe("checkout", () => {
    beforeAll(async () => {
        await seedCatalog();
    });

    beforeEach(async ({ page }) => {
        await page.context().clearCookies();
        await page.goto("/cart");
    });

    test("applies coupon", async ({ page }) => {
        await page.getByLabel("Coupon").fill("QA10");
    });
});
```

JUnit analogy for mixed teams:

```
@BeforeAll
static void seedCatalog() { }

@BeforeEach
void resetSession() { }
```

In pytest I mirror scope:

```python
import pytest

_catalog = []


@pytest.fixture(scope="module", autouse=True)
def seed_catalog():
    _catalog.append("item-a")


@pytest.fixture(autouse=True)
def fresh_cart():
    cart = []
    yield cart
    cart.clear()


def test_cart_starts_empty(fresh_cart):
    assert fresh_cart == []
```

## Pitfalls

**beforeAll without teardown.** A hanging port blocks the next job.

**Mutation in beforeAll.** The second test sees dirty state.

**Missing await in beforeEach.** The test starts before `goto`.

**Global beforeEach far from the spec.** Unclear who resets mocks.

**Parallel suite + beforeAll DB.** Deadlock on a shared migration lock.
