# NaN в JavaScript и корректные проверки в assert

## Суть

`NaN` (Not a Number) имеет тип `number`, но не равен сам себе: `NaN === NaN` ложно. Появляется при невалидной математике или `Number("abc")`. Для проверки использую `Number.isNaN(value)`, глобальный `isNaN` приводит типы и даёт ложные срабатывания.

## Что ответить на собесе

- `0/0` и `parseInt("foo")` дают NaN, не exception.
- В assert нельзя `expect(x).toBe(NaN)`, нужен `Number.isNaN(x)` или matcher `toBeNaN` где доступен.
- `isNaN("12px")` true из-за parse, `Number.isNaN("12px")` false.
- В контрактных тестах API поле missing лучше отличать от numeric NaN.
- Честно: JSON не сериализует NaN, клиент может получить `null` вместо ожидаемого NaN.

## Как это работает

Источники NaN:

```
const fromMath = 0 / 0;
const fromParse = Number("not-a-number");
```

Проверки:

```
console.log(isNaN("abc"));
console.log(Number.isNaN("abc"));
console.log(Number.isNaN(fromMath));
```

В pytest валидирую числовые поля после парсинга:

```python
import math


def is_invalid_metric(value) -> bool:
    if value is None:
        return False
    if isinstance(value, float):
        return math.isnan(value)
    return False


def test_metric_validation():
    assert is_invalid_metric(float("nan"))
    assert not is_invalid_metric("abc")
    assert not is_invalid_metric(None)
```

Для агрегатов фильтрую NaN перед `max()`/`avg`, иначе весь расчёт NaN.

## Подводные камни

**Object.is(NaN, NaN).** True, но в assert команда забывает про Object.is.

**Сортировка с NaN.** Сравнение нестабильно, порядок элементов меняется.

**Chart assertions.** NaN рвёт scale, screenshot diff ложный.

**parseFloat частичный успех.** `"12abc"` не NaN, тест ждал NaN.

**Смешение null и NaN в БД export.** CSV пустая ячейка vs строка NaN.
