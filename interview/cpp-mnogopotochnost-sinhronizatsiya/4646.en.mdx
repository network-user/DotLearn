# What mutex types does the C++ standard library provide?

## In short

The standard offers several mutex types for different access patterns: basic `std::mutex`, recursive, timed, shared (many readers / one writer), and combinations. RAII wrappers (`lock_guard`, `unique_lock`, `shared_lock`, `scoped_lock`) avoid manual unlock.

## How to answer in an interview

- `std::mutex` by default for mutual exclusion.
- `std::recursive_mutex` when one thread re-enters the same guard (use design caution).
- `std::timed_mutex` / `std::recursive_timed_mutex` for `try_lock_for` / `try_lock_until`.
- `std::shared_mutex` (C++17) for read/write: `shared_lock` to read, `unique_lock` to write.
- Honestly: I start with a plain `mutex`; shared and recursive only when measurement shows need.

## How it works

```cpp
#include <iostream>
#include <mutex>
#include <shared_mutex>

std::shared_mutex catalogGuard;
int catalogVersion = 0;

void readCatalog() {
    std::shared_lock lock(catalogGuard);
    std::cout << "version " << catalogVersion << '\n';
}

void publishCatalog(int next) {
    std::unique_lock lock(catalogGuard);
    catalogVersion = next;
}

std::recursive_mutex logStack;
void logLayer(int depth) {
    std::lock_guard lock(logStack);
    if (depth > 0) logLayer(depth - 1);
}
```

Multiple `readCatalog` calls run in parallel under `shared_lock`. `publishCatalog` is exclusive. `recursive_mutex` lets `logLayer` re-enter the same mutex.

## Pitfalls

- **shared_mutex overhead.** For tiny critical sections a single `mutex` is simpler.
- **Writer starvation.** Constant readers can delay a writer; fairness depends on the implementation.
- **recursive hides design flaws.** Deep recursion with locks complicates reasoning.
- **timed without backoff.** `try_lock_for` in a tight loop burns CPU.
- **Incompatible lock types.** Classic `condition_variable` pairs with `std::mutex` (not recursive); use `condition_variable_any` for generic locks.
