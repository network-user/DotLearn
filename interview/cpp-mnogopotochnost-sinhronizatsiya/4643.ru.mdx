# Как процессы обмениваются данными?

## Суть

Межпроцессное взаимодействие (IPC) связывает изолированные адресные пространства: pipes, именованные каналы, очереди сообщений, разделяемая память, семафоры ОС, сокеты и файлы. Выбор зависит от объёма данных, модели синхронизации и нужды в сетевой доставке.

## Что ответить на собесе

- Pipe/FIFO: простой поток байтов между родственными или локальными процессами.
- Shared memory: самый быстрый обмен, но нужна отдельная синхронизация (семафор, mutex в shm).
- Message queue: структурированные сообщения с буфером в ядре.
- Сокеты: универсально, в том числе TCP/UDP между хостами.
- Честно: в чистом C++ STL нет IPC; использую POSIX/WinAPI или обёртки (Boost.Interprocess).

## Как это работает

Анонимный pipe после `fork` даёт односторонний канал родитель ↔ ребёнок.

```cpp
#include <cstring>
#include <iostream>
#include <unistd.h>

int main() {
    int fds[2];
    pipe(fds);
    pid_t pid = fork();

    if (pid == 0) {
        close(fds[1]);
        char buf[64]{};
        read(fds[0], buf, sizeof(buf));
        std::cout << "child: " << buf << '\n';
        close(fds[0]);
    } else {
        close(fds[0]);
        const char msg[] = "ping";
        write(fds[1], msg, sizeof(msg));
        close(fds[1]);
    }
    return 0;
}
```

Родитель пишет в `fds[1]`, ребёнок читает из `fds[0]`. Разные процессы не видят обычные указатели друг друга; ядро копирует буфер. Для двустороннего диалога нужны два pipe или сокет.

## Подводные камни

- **Размер pipe.** Запись блокируется, если читатель не успевает; нужен протокол backpressure.
- **Shared memory races.** Без mutex/семафора в сегменте те же data race, что между потоками.
- **Зомби и дескрипторы.** Незакрытые fd после fork утекают и блокируют EOF.
- **Портируемость.** `fork`/`pipe` POSIX; на Windows другие примитивы (named pipe, mailslot).
- **Сериализация.** Сложные объекты не кладут в shm напрямую; нужен формат (protobuf, flatbuffers).
