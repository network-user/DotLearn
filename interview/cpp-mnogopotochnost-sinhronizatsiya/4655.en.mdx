# Pros and cons of multithreading

## In short

Threads parallelize CPU-bound work across cores and overlap I/O with compute, improving throughput and UI responsiveness. The cost is synchronization complexity, races, deadlocks, harder debugging, and context-switch overhead.

## How to answer in an interview

- Pro: speed up independent tasks on multicore CPUs when a mutex is not the bottleneck.
- Pro: background load/network does not block the interactive thread.
- Con: shared mutable state needs discipline (mutex, atomic, message passing).
- Con: not every problem splits; Amdahl caps gains with the sequential fraction.
- Honestly: sometimes processes or async I/O are simpler and safer than "a thread for everything."

## How it works

```cpp
#include <iostream>
#include <thread>

int sumRange(int lo, int hi) {
    int acc = 0;
    for (int i = lo; i < hi; ++i) acc += i;
    return acc;
}

int main() {
    std::thread left(sumRange, 0, 500000);
    std::thread right(sumRange, 500000, 1000000);
    left.join();
    right.join();
    std::cout << "done\n";
}
```

Two threads sum disjoint halves independently. On two cores wall time nears half if there is no shared state. Once both write one counter without sync, races and cache traffic eat the win.

## Pitfalls

- **False parallelism.** More threads than cores on CPU-bound work only hurts.
- **I/O bound.** Threads help, but epoll/IOCP/async may beat hundreds of blocked threads.
- **Testing.** Event order is nondeterministic; flaky tests without synchronized expectations.
- **Memory.** Each thread carries a stack (MBs); thousands of threads are costly.
- **Cancel and shutdown.** Without a stop protocol the process hangs on join forever.
