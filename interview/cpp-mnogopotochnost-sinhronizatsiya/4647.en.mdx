# What memory regions does a process have?

## In short

A process virtual address space is usually split into text (code), data (initialized globals), BSS (zero-initialized globals), heap (dynamic allocation), and per-thread stacks. Threads share the heap and globals but each has its own stack.

## How to answer in an interview

- Stack: locals, call frames, return addresses; freed on scope exit.
- Heap: `new`/`malloc`, lifetime under program control; shared by all threads.
- Static/globals: storage duration until process end; visible to every thread.
- Text/rodata: machine code and constants, typically read-only.
- Honestly: the exact map depends on OS and linker; interviews care more about lifetime and thread safety than raw addresses.

## How it works

```cpp
int globalCounter = 0;

void worker() {
    int localTick = 0;
    static int persistentTick = 0;
    ++localTick;
    ++persistentTick;
    ++globalCounter;
}

int main() {
    int* heapSlot = new int(7);
    worker();
    delete heapSlot;
}
```

`localTick` lives on `worker`'s stack and vanishes on return. `persistentTick` sits in static storage and keeps its value across calls. `globalCounter` is in the process data segment. `heapSlot` points into the heap; skipping `delete` leaks. Any thread calling `worker` shares `globalCounter` and `persistentTick` but gets its own `localTick` on its stack.

## Pitfalls

- **Stack overflow.** Large stack arrays or deep recursion crash the process.
- **Heap contention.** The allocator is global; multithreading hurts without pool allocators.
- **Static init order.** Globals across translation units initialize in unpredictable order.
- **Dangling stack refs.** Returning `T&` to a local is invalid.
- **mmap and stack rlimit.** On Linux `ulimit -s` caps the stack; the heap grows differently.
