# Сколько потоков может жить в одном процессе?

## Суть

Один процесс может содержать множество потоков: они делят адресное пространство, открытые файлы и другие ресурсы процесса, но выполняются независимо под управлением планировщика ОС.

## Что ответить на собесе

- Да, в одном процессе обычно несколько потоков: main thread плюс worker threads.
- Потоки разделяют heap, глобальные/static данные и дескрипторы файлов.
- У каждого потока свой стек и свой program counter.
- В C++ создаю их через `std::thread`, `std::jthread` или thread pool.
- Честно: верхняя граница задаётся ОС и памятью под стеки, а не языком; тысячи потоков на процесс на практике редко оправданы.

## Как это работает

Процесс в ОС получает изолированное виртуальное адресное пространство. Потоки внутри него легче процессов: переключение контекста дешевле, IPC не нужен для общих данных.

```cpp
#include <iostream>
#include <thread>
#include <string>

void emit(const std::string& label, int rounds) {
    for (int i = 0; i < rounds; ++i) {
        std::cout << label << " tick " << i << '\n';
    }
}

int main() {
    std::thread alpha(emit, "alpha", 3);
    std::thread beta(emit, "beta", 3);

    alpha.join();
    beta.join();
    std::cout << "main done\n";
}
```

`main` сам является потоком процесса. Новые `std::thread` получают отдельный стек (типично мегабайты) и начинают выполнение сразу после конструирования. Общие переменные видны всем потокам, поэтому доступ к ним требует синхронизации.

Типичные сценарии: фоновые вычисления в UI-приложении, параллельная обработка запросов в сервере, pipeline стадий в ETL. На многоядерной машине потоки могут реально выполняться параллельно; на одном ядре планировщик делит кванты времени.

## Подводные камни

- **Data race.** Разделяемая память без mutex/atomic даёт неопределённое поведение.
- **Stack size.** Много потоков быстро съедают RAM из‑за стеков; для массового параллелизма лучше thread pool или корутины.
- **GIL и runtime.** В чистом C++ GIL нет, но сторонние библиотеки могут сериализовать работу внутри.
- **Отладка.** Heisenbugs из‑за недетерминированного порядка сложнее воспроизводить.
- **Завершение процесса.** Убийство процесса завершает все его потоки сразу, кооперативный shutdown всё равно нужен для корректного release ресурсов.
