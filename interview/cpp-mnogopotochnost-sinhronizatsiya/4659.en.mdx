# Can one thread belong to multiple processes?

## In short

No. A thread always belongs to exactly one process: it runs in that process address space and cannot share memory with two isolated processes at once.

## How to answer in an interview

- A thread is a scheduling unit inside a process, not across processes.
- Processes are isolated: each has its own virtual address space.
- Cross-process work uses IPC: pipes, sockets, explicitly mapped shared memory.
- In C++, `std::thread` always creates a thread in the current process.
- Honestly: at the OS level a thread ID is unique within a process; you cannot move a thread to another process, only start a new one.

## How it works

A process holds code, data, a descriptor table, and one or more threads. Each thread has its own stack, registers, and instruction pointer but reads the same memory pages as sibling threads.

```cpp
#include <iostream>
#include <thread>

void side_task() {
    std::cout << "side thread runs in the same process\n";
}

int main() {
    std::thread helper(side_task);
    helper.join();
    std::cout << "main thread finished\n";
}
```

`helper` cannot migrate to another process: its stack and TCB are tied to the current PID. Work in another process means creating that process (`fork`/`exec`, `CreateProcess`) or sending a job over IPC.

Threads in one process share the heap and globals, so synchronization is required. Threads in different processes do not see each other memory by default; shared memory needs a separate system API and agreed permissions.

## Pitfalls

- **Terminology mix-up.** A thread in process B reading a file opened by process A does not mean the thread belongs to both; that is IPC or an inherited descriptor after fork.
- **fork and threads.** After `fork` in a multithreaded program, only async-signal-safe calls are safe in the child until `exec`.
- **Shared memory.** A shared segment does not make one thread common; each process has its own threads, just mapping the same RAM region.
- **Handle inheritance.** Descriptors may inherit, but a thread execution context is still per process.
- **Debugging.** Blending in-process threads with cross-process IPC often causes races at API boundaries.
