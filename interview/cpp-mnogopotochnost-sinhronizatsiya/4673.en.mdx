# What is a semaphore and how does it differ from a mutex?

## In short

A semaphore holds a permit counter for a resource: threads wait on `acquire` (P) and signal with `release` (V). A binary semaphore resembles a mutex; a counting semaphore caps N concurrent users of a resource pool.

## How to answer in an interview

- Mutex: one owner; only the owner unlocks.
- Counting semaphore: N slots; any thread may release (producer/consumer).
- C++20: `std::counting_semaphore`, `std::binary_semaphore`.
- Classic uses: bounded buffer, connection pool, parallel job throttle.
- Honestly: use a mutex for exclusive sections; semaphores when limiting parallelism matters more than ownership.

## How it works

Pool of two connections:

```cpp
#include <chrono>
#include <iostream>
#include <semaphore>
#include <thread>

std::counting_semaphore<2> slots{2};

void client(int id) {
    slots.acquire();
    std::cout << "client " << id << " connected\n";
    std::this_thread::sleep_for(std::chrono::milliseconds(400));
    std::cout << "client " << id << " done\n";
    slots.release();
}

int main() {
    std::thread c1(client, 1);
    std::thread c2(client, 2);
    std::thread c3(client, 3);
    c1.join();
    c2.join();
    c3.join();
}
```

The first two clients proceed; the third waits for a `release`. A binary semaphore can signal events, but ownership is weaker than a mutex.

POSIX `sem_t` was used cross-process; C++20 semaphores are mainly intra-process.

## Pitfalls

- **Release without acquire.** Counter can exceed capacity; logic bug.
- **Missing release.** All waiters deadlock.
- **Mutex vs semaphore mix-up.** Clarify ownership in interviews.
- **Priority inversion.** Same as mutex with a long holder.
- **Not a condvar replacement.** Complex condition waits fit `condition_variable` + predicate better.
