# Which call creates a new thread?

## In short

A new thread appears when you construct `std::thread` (or when `std::async` actually runs with `launch::async`). The constructor takes a callable and arguments; execution starts immediately unless you chose a deferred `async` policy.

## How to answer in an interview

- Key call: `std::thread worker(callable, args...);`
- Alternative: `std::jthread worker(callable, args...);` with auto-join in the destructor.
- For a result: `auto f = std::async(std::launch::async, callable, args...);`
- After creation you must `join()` or `detach()` for `std::thread`.
- Honestly: at the OS level it is `pthread_create` / `CreateThread`, but C++ interviews expect `std::thread`.

## How it works

```cpp
#include <iostream>
#include <thread>

void banner() {
    std::cout << "side thread\n";
}

int main() {
    std::thread side(banner);
    side.join();
    std::cout << "main thread\n";
}
```

The line `std::thread side(banner);` creates `side` and runs `banner` on a new thread. `join()` waits for completion; otherwise `side`'s destructor calls `std::terminate`. Print order may vary before join, but both lines appear before `main` exits.

## Pitfalls

- **Constructor is not start().** The thread already runs after the ctor; there is no separate `run()`.
- **Temporary callable.** `std::thread t([]{...});` is fine; passing a reference to a stack lambda without join before scope end is dangerous.
- **thread is move-only.** Accidental copy does not compile.
- **hardware_concurrency.** A hint for scheduling, not a cap on `thread` count.
- **async ≠ thread ctor.** `async` without a policy may not spawn a thread until `get()`.
