# How do you create and start a thread in C++?

## In short

A thread starts when you construct `std::thread` with a callable (function, lambda, functor) and arguments. You must then `join()` (wait) or `detach()`; otherwise destroying `std::thread` calls `std::terminate`.

## How to answer in an interview

- `std::thread worker(task, arg1, arg2);` runs `task` on a new thread with copied/moved arguments.
- I default to `join()` at scope end via RAII or `std::jthread` (C++20).
- I `detach()` only if the thread finishes before process exit and never touches destroyed objects.
- For task pools I prefer `std::async`, a queue plus workers, or an executor, not hundreds of raw threads.
- Honestly: passing references needs `std::ref`; otherwise a wrapper copies and you get a silent bug.

## How it works

```cpp
#include <iostream>
#include <string>
#include <thread>

void greet(std::string name) {
    std::cout << "hi " << name << '\n';
}

int main() {
    std::thread t1(greet, "Ada");
    std::thread t2([] {
        std::cout << "from lambda\n";
    });

    t1.join();
    t2.join();
    std::cout << "main done\n";
    return 0;
}
```

The `std::thread` constructor takes a callable and arguments, packages them, and starts an OS thread. `join` blocks `main` until completion. An empty `[]` lambda capture is safe unless it references dying locals without synchronization.

## Pitfalls

- **terminate on destructor.** Neither join nor detach means UB via `std::terminate`.
- **detach and lifetime.** A background thread must not touch `main`'s stack after it returns.
- **Copying this.** Class method: `std::thread(&Obj::run, &obj)`; a dangling `obj` kills the thread.
- **Exception in the thread.** An escaping exception can end in `std::terminate` on join depending on standard version.
- **Thread explosion.** Thousands of `std::thread` instances pressure stack memory; use a bounded worker pool.
