# Плюсы и минусы многопоточности

## Суть

Потоки позволяют параллелить CPU-bound работу на ядрах и overlap I/O с вычислениями, повышая throughput и отзывчивость UI. Цена: сложность синхронизации, гонки, deadlock, отладка и overhead на переключение контекста.

## Что ответить на собесе

- Плюс: ускорение независимых задач на многоядерных CPU при отсутствии узкого mutex.
- Плюс: фоновая загрузка/сеть не блокирует интерактивный поток.
- Минус: shared mutable state требует дисциплины (mutex, atomic, message passing).
- Минус: не каждая задача делится; Amdahl ограничивает выигрыш последовательной частью.
- Честно: иногда процессы или async I/O проще и безопаснее, чем «поток на всё».

## Как это работает

```cpp
#include <iostream>
#include <thread>
#include <vector>

int sumRange(int lo, int hi) {
    int acc = 0;
    for (int i = lo; i < hi; ++i) acc += i;
    return acc;
}

int main() {
    std::thread left(sumRange, 0, 500000);
    std::thread right(sumRange, 500000, 1000000);
    left.join();
    right.join();
    std::cout << "done\n";
}
```

Два потока считают половины диапазона независимо. На двух ядрах wall time ближе к половине, если нет общего состояния. Как только оба пишут в один счётчик без sync, выигрыш съедают гонки и cache coherency traffic.

## Подводные камни

- **Ложная параллельность.** Больше потоков, чем ядер, на CPU-bound задаче только хуже.
- **I/O bound.** Потоки помогают, но epoll/IOCP/async может быть эффективнее сотен blocked threads.
- **Тестирование.** Порядок событий недетерминирован; flaky tests без синхронизации ожиданий.
- **Память.** Каждый поток тянет стек (МБ); тысячи потоков дороги.
- **Отмена и shutdown.** Без протокола остановки процесс зависает на join вечно.
