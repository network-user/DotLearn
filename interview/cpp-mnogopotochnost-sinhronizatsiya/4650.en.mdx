# What synchronization primitives exist?

## In short

Primitives split into blocking (mutex, semaphore, condition variable), coordination (barrier, latch), and lock-free (atomic, CAS queues). They address mutual exclusion, waiting for conditions, limiting parallelism, and mutex-free exchange.

## How to answer in an interview

- Mutex and RW locks to protect data-structure invariants.
- Condition variables for "wait for an event"; semaphores for "at most N at once."
- Atomic plus memory order for counters, flags, simple lock-free structures.
- Barrier/latch for phased sync in parallel algorithms.
- Honestly: spinlocks and futexes sit closer to OS/runtime; app code usually uses mutex or atomic.

## How it works

Mutex plus condition variable for a bounded queue:

```cpp
#include <condition_variable>
#include <mutex>
#include <queue>

std::mutex m;
std::condition_variable notFull;
std::condition_variable notEmpty;
std::queue<int> q;
const std::size_t cap = 8;

void push(int v) {
    std::unique_lock lock(m);
    notFull.wait(lock, [] { return q.size() < cap; });
    q.push(v);
    notEmpty.notify_one();
}
```

The producer waits on `notFull` until size is below `cap`, pushes, and signals `notEmpty`. The consumer mirrors this for a non-empty queue. A semaphore capped at `cap` gives a similar effect with less code, but CV plus mutex is more flexible for complex conditions.

## Pitfalls

- **Spinlock in user space.** Burns CPU; justified for tiny sections in kernel/drivers, not business logic.
- **Mixing layers.** POSIX semaphores for processes plus `std::mutex` for threads in one design without care.
- **Lock-free complexity.** ABA, reclamation, and testing cost more than a mutex.
- **C++20 barriers.** Missing on some embedded toolchains; plan a fallback.
- **Too many primitives.** Each new sync type raises deadlock risk; simplify the architecture.
