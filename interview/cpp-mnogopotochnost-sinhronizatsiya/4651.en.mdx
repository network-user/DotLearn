# What IPC mechanisms do you know?

## In short

Inter-process communication uses pipes, message queues, shared memory, OS semaphores, sockets, signals, and sometimes files or databases as intermediaries. The choice depends on latency, data volume, trust model, and whether you need the network.

## How to answer in an interview

- Pipe and named pipe (FIFO) for local byte streams.
- Message queue (System V, POSIX mq) for fixed-format messages.
- Shared memory plus a separate sync primitive for high throughput.
- TCP/UDP/Unix domain sockets when you want network or a unified API.
- Honestly: signals are for control plane only (SIGTERM, SIGCHLD), not payload transfer.

## How it works

Unix domain socket for local RPC without the network stack:

```cpp
#include <sys/socket.h>
#include <sys/un.h>
#include <unistd.h>
#include <cstring>
#include <iostream>

int main() {
    int fd = socket(AF_UNIX, SOCK_STREAM, 0);
    sockaddr_un addr{};
    addr.sun_family = AF_UNIX;
    std::strcpy(addr.sun_path, "/tmp/dotlearn.sock");
    bind(fd, reinterpret_cast<sockaddr*>(&addr), sizeof(addr));
    listen(fd, 1);
    int client = accept(fd, nullptr, nullptr);
    char buf[32]{};
    read(client, buf, sizeof(buf));
    std::cout << buf << '\n';
    close(client);
    close(fd);
}
```

The server listens on a filesystem path; the client connects and sends bytes like TCP but stays on-host. For two local processes this is often simpler than shared memory with manual sync.

## Pitfalls

- **Socket path permissions.** World-readable `sun_path` lets foreign processes connect.
- **Message size.** Pipes and sockets buffer; large blobs favor shared memory or files.
- **Windows portability.** `AF_UNIX` arrived late; named pipes were the classic answer.
- **IPC object cleanup.** SysV shm/msg survives crashes without `IPC_RMID`.
- **Versioned serialization.** Different binary versions must agree on message layout.
