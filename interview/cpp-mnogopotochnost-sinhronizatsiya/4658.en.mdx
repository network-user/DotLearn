# How many threads can one process have?

## In short

A single process can host many threads. They share the address space, open files, and other process resources while the OS scheduler runs them independently.

## How to answer in an interview

- Yes, one process usually has several threads: the main thread plus workers.
- Threads share the heap, globals/statics, and file descriptors.
- Each thread has its own stack and program counter.
- In C++ I create them with `std::thread`, `std::jthread`, or a thread pool.
- Honestly: the upper bound is set by the OS and stack memory, not the language; thousands of threads per process are rarely justified in practice.

## How it works

An OS process gets an isolated virtual address space. Threads inside it are lighter than separate processes: context switches cost less and shared data needs no IPC.

```cpp
#include <iostream>
#include <thread>
#include <string>

void emit(const std::string& label, int rounds) {
    for (int i = 0; i < rounds; ++i) {
        std::cout << label << " tick " << i << '\n';
    }
}

int main() {
    std::thread alpha(emit, "alpha", 3);
    std::thread beta(emit, "beta", 3);

    alpha.join();
    beta.join();
    std::cout << "main done\n";
}
```

`main` is itself a thread of the process. New `std::thread` objects get a separate stack (often megabytes) and start running right after construction. Shared variables are visible to every thread, so access needs synchronization.

Typical uses: background work in a UI app, parallel request handling on a server, pipeline stages in ETL. On a multicore machine threads may run truly in parallel; on one core the scheduler time-slices them.

## Pitfalls

- **Data races.** Shared memory without mutex/atomic is undefined behavior.
- **Stack size.** Many threads burn RAM on stacks; massive parallelism favors a thread pool or coroutines.
- **Runtime locks.** Pure C++ has no GIL, but third-party libraries may serialize work internally.
- **Debugging.** Nondeterministic ordering makes heisenbugs hard to reproduce.
- **Process exit.** Killing the process ends all its threads at once; cooperative shutdown is still needed to release resources cleanly.
