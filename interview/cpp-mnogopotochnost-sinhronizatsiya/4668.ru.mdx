# Минимальное число mutex для взаимной блокировки

## Суть

Межпоточный deadlock на mutex требует как минимум двух mutex (или двух ресурсов) и двух потоков с циклическим ожиданием. Один mutex между разными потоками даёт блокировку, но не замкнутый цикл hold-and-wait.

## Что ответить на собесе

- Ответ: минимум 2 mutex + 2 потока для классического сценария.
- Circular wait невозможен, пока ресурс один.
- Self-lock на `std::mutex` это отдельная ошибка, не межпоточный deadlock.
- `std::scoped_lock`/`std::lock` снимают риск при захвате набора mutex.
- Честно: вопрос часто дублирует соседние карточки; важно различать deadlock, starvation и простое ожидание lock.

## Как это работает

Пара потоков и два mutex:

```cpp
#include <mutex>
#include <thread>

std::mutex north;
std::mutex south;

void climb_north_first() {
    north.lock();
    south.lock();
    north.unlock();
    south.unlock();
}

void climb_south_first() {
    south.lock();
    north.lock();
    south.unlock();
    north.unlock();
}
```

При unlucky interleaving каждый держит «свой» первый mutex и ждёт второй: цикл замкнут.

С одним mutex два потока только очередятся:

```cpp
std::mutex ticket;
ticket.lock(); // thread A
// thread B blocks until A unlocks, then proceeds
```

Для N mutex теоретический минимум цикла длины 2, но на практике deadlock часто строят на 3+ ресурсах в больших системах.

## Подводные камни

- **Duplicate interview cards.** Ответ совпадает с «сколько mutex нужно»; акцент здесь на minimum для создания deadlock.
- **Non-mutex resources.** File locks, DB rows, message queues тоже участвуют в цикле.
- **Lock ordering doc.** Без документированной иерархии даже 2 mutex опасны.
- **try_lock loops.** Избегает deadlock, но может голодать.
- **Static analysis.** Clang thread safety analysis и TSan дополняют знание теории.
