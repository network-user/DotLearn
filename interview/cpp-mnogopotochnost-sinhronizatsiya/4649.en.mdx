# What mutex types do you know?

## In short

The C++ standard library offers a mutex family for different scenarios: plain, recursive, timed, shared for read/write, and timed variants. The choice depends on whether you need exclusive access, concurrent reads, or re-entry by the same thread.

## How to answer in an interview

- `std::mutex` for a typical critical section.
- `std::timed_mutex` when you need `try_lock_for` instead of blocking forever.
- `std::recursive_mutex` for re-entrant code (prefer refactoring over abuse).
- `std::shared_mutex` with `shared_lock`/`unique_lock` for read-mostly data.
- Honestly: I also know OS mutexes (pthread, CRITICAL_SECTION), but in modern C++ I start with the STL.

## How it works

```cpp
#include <chrono>
#include <mutex>
#include <shared_mutex>

int cacheValue = 0;
std::shared_mutex cacheGuard;
std::timed_mutex retryGate;

int readCache() {
    std::shared_lock lock(cacheGuard);
    return cacheValue;
}

void writeCache(int v) {
    std::unique_lock lock(cacheGuard);
    cacheValue = v;
}

bool tryWork() {
    return retryGate.try_lock_for(std::chrono::milliseconds(5));
}
```

`readCache` uses `shared_lock` for parallel readers. `writeCache` takes an exclusive `unique_lock`. `retryGate` lets you back off if the lock stays busy longer than 5 ms instead of deadlocking.

## Pitfalls

- **Overlap with 4646.** In interviews I list briefly and immediately give a selection scenario.
- **shared_timed is rare.** Support and behavior differ across libcs; verify the platform.
- **Wrong wrapper.** `shared_lock` on a plain `mutex` does not compile.
- **Priority inversion.** A normal mutex on Linux does not fix PI; pthread attrs or futex policy may be needed.
- **Critical section size.** Mutex type does not help if heavy work runs under the lock.
