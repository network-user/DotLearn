# Какие области памяти есть у процесса?

## Суть

Виртуальное адресное пространство процесса обычно делят на текст (код), данные (инициализированные глобалы), BSS (нулевые глобалы), кучу (динамика) и стеки потоков. Потоки делят кучу и глобалы, но у каждого свой стек.

## Что ответить на собесе

- Стек: локальные переменные, кадры вызовов, адреса возврата; освобождается при выходе из scope.
- Куча: `new`/`malloc`, lifetime под контролем программы; общая для всех потоков.
- Статика/глобалы: storage duration до конца процесса; видны всем потокам.
- Text/rodata: машинный код и константы, обычно read-only.
- Честно: точная карта зависит от ОС и линкера; на собесе важнее lifetime и thread-safety, чем адреса.

## Как это работает

```cpp
int globalCounter = 0;

void worker() {
    int localTick = 0;
    static int persistentTick = 0;
    ++localTick;
    ++persistentTick;
    ++globalCounter;
}

int main() {
    int* heapSlot = new int(7);
    worker();
    delete heapSlot;
}
```

`localTick` живёт в стеке кадра `worker` и исчезает после return. `persistentTick` в статической памяти, значение сохраняется между вызовами. `globalCounter` в сегменте данных процесса. `heapSlot` указывает в кучу; без `delete` утечка. Все потоки, вызывающие `worker`, делят `globalCounter` и `persistentTick`, но имеют отдельные `localTick` в своих стеках.

## Подводные камни

- **Stack overflow.** Большие массивы на стеке или глубокая рекурсия роняют процесс.
- **Heap contention.** Аллокатор глобален; многопоточность бьёт по perf без pool allocator.
- **Static init order.** Глобалы в разных TU инициализируются в непредсказуемом порядке.
- **Висячие ссылки на стек.** Возврат `T&` на локальную из функции недопустим.
- **mmap и stack rlimit.** На Linux `ulimit -s` ограничивает стек; куча растёт иначе.
