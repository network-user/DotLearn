# Pros and cons of mutexes

## In short

A mutex offers simple, understandable mutual access to a critical section. The cost is blocking, deadlock risk, contention, and kernel wakeup overhead when the section is short but competition is high.

## How to answer in an interview

- Pro: easy to reason about invariants; fits most shared state.
- Pro: broad support, RAII wrappers, integration with condition_variable.
- Con: serialization limits scaling on many cores.
- Con: wrong lock order → deadlock; forgotten unlock → hang.
- Honestly: for a hot-path simple counter, `std::atomic` is often better than a mutex.

## How it works

```cpp
#include <atomic>
#include <mutex>

std::mutex ledgerGuard;
int ledger = 0;
std::atomic<int> fastHits{0};

void recordSlow() {
    std::lock_guard lock(ledgerGuard);
    ++ledger;
}

void recordFast() {
    ++fastHits;
}
```

`recordSlow` is safe for compound invariants around `ledger`. `recordFast` updates a counter without kernel blocking. A mutex pays off when multiple fields or non-atomic structures change under one lock; a single counter is cheaper with atomic.

## Pitfalls

- **Coarse lock.** One mutex for the whole service turns parallelism into a queue.
- **Priority inversion.** A low-priority owner stalls a high-priority waiter.
- **Exception safety.** Manual lock/unlock without RAII leaves the mutex held.
- **Readers without shared_mutex.** Many readers under an exclusive mutex do not scale.
- **Debugging.** Timing Heisenbugs; reproduce with stress tests, not unit tests alone.
