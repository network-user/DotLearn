# Minimum mutex count for mutual blocking

## In short

Inter-thread mutex deadlock needs at least two mutexes (or two resources) and two threads in circular wait. One mutex across threads causes blocking, not a closed hold-and-wait cycle.

## How to answer in an interview

- Answer: at least 2 mutexes + 2 threads for the classic case.
- Circular wait is impossible while only one resource exists.
- Self-lock on `std::mutex` is a separate bug, not inter-thread deadlock.
- `std::scoped_lock`/`std::lock` remove risk when acquiring a mutex set.
- Honestly: this often duplicates neighboring cards; distinguish deadlock, starvation, and plain lock waiting.

## How it works

Two threads and two mutexes:

```cpp
#include <mutex>
#include <thread>

std::mutex north;
std::mutex south;

void climb_north_first() {
    north.lock();
    south.lock();
    north.unlock();
    south.unlock();
}

void climb_south_first() {
    south.lock();
    north.lock();
    south.unlock();
    north.unlock();
}
```

With unlucky interleaving each holds its first mutex and waits for the second: cycle closed.

With one mutex two threads just queue:

```cpp
std::mutex ticket;
ticket.lock(); // thread A
// thread B blocks until A unlocks, then proceeds
```

For N mutexes the theoretical minimum cycle length is 2, but production deadlocks often involve 3+ resources.

## Pitfalls

- **Duplicate cards.** Same answer as how many mutexes are needed; focus here on the minimum to create deadlock.
- **Non-mutex resources.** File locks, DB rows, and message queues join cycles too.
- **Lock ordering docs.** Without a documented hierarchy even 2 mutexes are risky.
- **try_lock loops.** Avoid deadlock but may starve.
- **Static analysis.** Clang thread safety analysis and TSan complement theory.
