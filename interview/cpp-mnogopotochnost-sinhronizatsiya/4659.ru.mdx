# Может ли один поток принадлежать нескольким процессам?

## Суть

Нет. Поток всегда принадлежит ровно одному процессу: он выполняется в его адресном пространстве и не может одновременно разделять память двух изолированных процессов.

## Что ответить на собесе

- Поток это единица планирования внутри процесса, не между процессами.
- Процессы изолированы: у каждого своё виртуальное адресное пространство.
- Межпроцессное взаимодействие идёт через IPC: pipe, socket, shared memory segment с явным mapping.
- В C++ `std::thread` всегда создаёт поток в текущем процессе.
- Честно: на уровне ОС thread ID уникален в рамках процесса; «перенести» поток в другой процесс нельзя, только запустить новый.

## Как это работает

Процесс содержит код, данные, таблицу дескрипторов и один или несколько потоков. Каждый поток имеет свой стек, регистры и счётчик инструкций, но читает те же страницы памяти, что и sibling threads.

```cpp
#include <iostream>
#include <thread>

void side_task() {
    std::cout << "side thread runs in the same process\n";
}

int main() {
    std::thread helper(side_task);
    helper.join();
    std::cout << "main thread finished\n";
}
```

`helper` не может «переехать» в другой процесс: его стек и TCB привязаны к текущему PID. Если нужна работа в другом процессе, создают новый процесс (`fork`/`exec`, `CreateProcess`) или отправляют задачу через IPC.

Потоки одного процесса делят heap и глобальные объекты, поэтому синхронизация нужна. Потоки разных процессов по умолчанию не видят память друг друга; shared memory требует отдельного системного API и согласованных прав доступа.

## Подводные камни

- **Путаница терминов.** «Поток процесса Б читает файл процесса А» не значит, что поток принадлежит двум процессам: это IPC или общий дескриптор после fork.
- **fork и threads.** После `fork` в многопоточном процессе безопасно вызывать только async-signal-safe функции в дочернем процессе до `exec`.
- **Shared memory.** Разделяемый сегмент не делает один поток «общим»; у каждого процесса свой набор потоков, просто mapped одна область RAM.
- **Handle inheritance.** Дескрипторы могут наследоваться, но контекст выполнения потока всё равно один на процесс.
- **Отладка.** Смешение моделей «threads inside process» и «processes with IPC» часто приводит к гонкам на границе API.
