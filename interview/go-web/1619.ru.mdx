# Связь HTTP, gRPC и Protobuf

## Суть

Protobuf задаёт контракт данных и сериализацию в компактный бинарный формат. gRPC использует его как IDL и payload, а HTTP/2 как транспорт: мультиплексирование потоков, сжатие заголовков, bidirectional streams. HTTP в классическом REST несёт JSON/XML в текстовом виде поверх HTTP/1.1 или HTTP/2 без единого codegen контракта.

## Что ответить на собесе

- Proto файл → `protoc` → stubs; wire format protobuf, framing gRPC, transport HTTP/2.
- REST: ресурсы + verbs + статус коды; gRPC: методы сервиса + status codes в trailers.
- HTTP/2 общий для браузерного HTTPS и для gRPC, но семантика слоя разная.
- Для публичного API часто REST/JSON; внутри кластера gRPC за производительность и строгий контракт.
- Честно: grpc-web и gateway добавляют прослойку, теряется часть преимуществ чистого gRPC.

## Как это работает

```protobuf
syntax = "proto3";
package orders.v1;

service OrderService {
  rpc PlaceOrder(PlaceOrderRequest) returns (PlaceOrderResponse);
}

message PlaceOrderRequest {
  string item_id = 1;
  int32 quantity = 2;
}
```

Клиентский вызов:

```go
resp, err := client.PlaceOrder(ctx, &pb.PlaceOrderRequest{
	ItemId: "sku-42", Quantity: 2,
})
```

На проводе: HTTP/2 POST к `/orders.v1.OrderService/PlaceOrder`, заголовки `content-type: application/grpc`, тело: compressed protobuf frames. REST эквивалент был бы `POST /orders` с `{"item_id":"sku-42","quantity":2}` и ручной валидацией JSON schema.

## Подводные камни

**Путать HTTP API с gRPC только из-за HTTP/2.** Инструменты мониторинга без gRPC awareness показывают нечитаемые пути.

**JSON в proto через `google.protobuf.Struct`.** Теряется типизация, ради которой брали protobuf.

**L7 LB без поддержки gRPC.** Нужен HTTP/2 end-to-end или proxy с gRPC routing.

**Огромные shared proto repo.** Coupling версий всех сервисов при каждом breaking change.

**Отсутствие versioning в package path.** `v1`/`v2` в import path обязателен для Go module hygiene.
