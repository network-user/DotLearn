# HTTP methods at the core of REST APIs

## In short

REST relies on HTTP verb semantics: GET reads, POST creates, PUT/PATCH updates, DELETE removes. Method idempotency and safety define whether a request can be cached and retried. CRUD on resources is expressed by URI plus method, not by a body with an operation name.

## How to answer in an interview

- GET/HEAD are safe and idempotent; POST is usually not idempotent.
- PUT replaces the whole resource; PATCH is a partial update.
- OPTIONS for CORS preflight and discovery of allowed methods.
- In Go I use `http.NewRequest` with an explicit `Method` and do not use POST for mutations that should be GET.
- Honestly: "REST" in practice is often RPC over POST `/doSomething`; consistency within the API matters.

## How it works

```go
func main() {
	base := "https://api.example.com/items"

	resp, _ := http.Get(base + "/42")
	defer resp.Body.Close()

	createBody := strings.NewReader(`{"title":"Widget"}`)
	resp, _ = http.Post(base, "application/json", createBody)
	defer resp.Body.Close()

	client := &http.Client{}
	req, _ := http.NewRequest(http.MethodPut, base+"/42", strings.NewReader(`{"title":"Widget v2","qty":3}`))
	resp, _ = client.Do(req)
	defer resp.Body.Close()

	req, _ = http.NewRequest(http.MethodDelete, base+"/42", nil)
	resp, _ = client.Do(req)
	defer resp.Body.Close()
}
```

Typical mapping: `POST /items` create, `GET /items/{id}` read, `PUT /items/{id}` full update, `PATCH /items/{id}` partial, `DELETE /items/{id}` remove. Status codes extend semantics: 201 + Location after POST, 204 with no body after DELETE.

## Pitfalls

**GET with side effects.** Caches and prefetch break data.

**POST instead of PUT for idempotent upsert.** Duplicates on network retry.

**PATCH without merge semantics.** Clients overwrite fields with nulls differently.

**Ignore OPTIONS in CORS.** The browser blocks the frontend without a preflight handler.

**Verb in the URL (`/getUser`).** Breaks REST and complicates caching.
