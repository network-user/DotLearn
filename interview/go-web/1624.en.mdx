# What goes in HTTP request headers

## In short

Headers carry request metadata: host, content type, authorization, language, cache, cookies, trace context. They go as `Name: value` pairs after the request line and before the body. Servers and intermediate proxies decide from headers without parsing the payload.

## How to answer in an interview

- `Host` is required in HTTP/1.1; in HTTP/2 the equivalent is `:authority`.
- `Content-Type` and `Content-Length` (or chunked) for the body.
- `Authorization: Bearer`, `Cookie`, `Accept`, `Accept-Encoding`.
- Custom `X-Request-Id`, `traceparent` for observability.
- Honestly: extra headers inflate every request; in HTTP/2 HPACK helps only on a persistent connection.

## How it works

```go
body := strings.NewReader(`{"email":"user@example.com"}`)
req, _ := http.NewRequest(http.MethodPost, "https://api.example.com/v1/signup", body)
req.Header.Set("Content-Type", "application/json")
req.Header.Set("Accept", "application/json")
req.Header.Set("Authorization", "Bearer "+accessToken)
req.Header.Set("Accept-Language", "ru-RU,ru;q=0.9")
req.Header.Set("X-Request-Id", uuid.NewString())

resp, err := http.DefaultClient.Do(req)
```

Raw HTTP/1.1 example:

```http
POST /v1/signup HTTP/1.1
Host: api.example.com
Content-Type: application/json
Content-Length: 32
Authorization: Bearer eyJhbGciOi...
Accept: application/json
```

In Go `req.Header` is `http.Header` (`map[string][]string`); `Add` vs `Set` matters for multi-value headers (`Set-Cookie` on response, `Accept` on request).

## Pitfalls

**Secrets in query string instead of Authorization.** They land in access logs and Referer.

**Manual duplicate Host with HTTP/2.** The client library usually sets it; a double header breaks the request.

**Content-Length does not match body.** Server truncates or waits forever.

**Missing `Content-Type` on POST JSON.** Server cannot guess format.

**PII in custom client User-Agent.** Leaks into CDN logs.
