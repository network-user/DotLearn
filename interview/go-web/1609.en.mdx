# OAuth 2.0 basics

## In short

OAuth 2.0 delegates access without sharing a password: the resource owner authorizes the client; the authorization server issues an access token (and refresh); the resource server validates the token. Flows: authorization code (+PKCE for public clients), client credentials for service-to-service.

## How to answer in an interview

- OAuth is authorization; OpenID Connect adds identity (ID token).
- Authorization code + PKCE for SPA/mobile.
- Scopes limit access; short-lived access token.
- Honestly: OAuth alone is not authentication until OIDC id_token or userinfo.

## How it works

```
User -> Client -> Auth Server (login + consent)
Auth Server -> Client: authorization code
Client -> Auth Server: code + client_secret -> access_token
Client -> Resource Server: Bearer access_token
```

A Go resource server validates JWT signature and `exp`, checks scope claim.

## Pitfalls

**Legacy implicit flow.** Use auth code + PKCE.

**Long-lived access token leak.** Rotate, bind to DPoP/mTLS.

**Confuse client credentials with user context.** Machine tokens only.

**Redirect URI open redirect.** Strict allowlist.

**Store refresh token insecurely.** Encrypted storage required.
