# HTTP-методы в основе REST API

## Суть

REST опирается на семантику HTTP verbs: GET читает, POST создаёт, PUT/PATCH обновляет, DELETE удаляет. Идемпотентность и безопасность методов задают, можно ли кэшировать и повторять запрос. CRUD на ресурсах выражается комбинацией URI и метода, а не телом с именем операции.

## Что ответить на собесе

- GET/HEAD safe и идемпотентны; POST обычно не идемпотентен.
- PUT заменяет ресурс целиком, PATCH частичное обновление.
- OPTIONS для CORS preflight и discovery allowed methods.
- В Go `http.NewRequest` с явным `Method`, не путаю POST с GET для мутаций.
- Честно: «REST» на практике часто RPC поверх POST `/doSomething`; важна консистентность внутри API.

## Как это работает

```go
func main() {
	base := "https://api.example.com/items"

	resp, _ := http.Get(base + "/42")
	defer resp.Body.Close()

	createBody := strings.NewReader(`{"title":"Widget"}`)
	resp, _ = http.Post(base, "application/json", createBody)
	defer resp.Body.Close()

	client := &http.Client{}
	req, _ := http.NewRequest(http.MethodPut, base+"/42", strings.NewReader(`{"title":"Widget v2","qty":3}`))
	resp, _ = client.Do(req)
	defer resp.Body.Close()

	req, _ = http.NewRequest(http.MethodDelete, base+"/42", nil)
	resp, _ = client.Do(req)
	defer resp.Body.Close()
}
```

Типичное сопоставление: `POST /items` create, `GET /items/{id}` read, `PUT /items/{id}` full update, `PATCH /items/{id}` partial, `DELETE /items/{id}` remove. Статус коды дополняют семантику: 201 + Location после POST, 204 без тела после DELETE.

## Подводные камни

**GET с побочным эффектом.** Кэши и prefetch ломают данные.

**POST вместо PUT для идемпотентного upsert.** Дубли при retry сети.

**PATCH без merge semantics.** Клиенты перетирают поля null-ами по разному.

**Игнор OPTIONS в CORS.** Браузер блокирует фронт без preflight handler.

**Глагол в URL (`/getUser`).** Нарушает REST и усложняет кэширование.
