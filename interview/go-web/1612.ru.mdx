# gRPC call types

## Суть

gRPC on HTTP/2 offers four RPC shapes: Unary (1 req, 1 resp), Server streaming (1 req, N resp), Client streaming (N req, 1 resp), Bidirectional streaming (N/N). Proto defines contract; generated stubs implement Send/Recv loops.

## Что ответить на собесе

- Unary like classic RPC/REST single response.
- Server stream for large lists/logs push.
- Client stream for uploads aggregation.
- Честно: bidi for chat/realtime; needs flow control and backpressure handling.

## Как это работает

```protobuf
service Telemetry {
  rpc PushBatch(stream Metric) returns (Ack);
  rpc Subscribe(Filter) returns (stream Event);
}
```

```go
func (s *Server) Subscribe(req *pb.Filter, stream pb.Telemetry_SubscribeServer) error {
	for evt := range s.bus.Subscribe(req.Topic) {
		if err := stream.Send(evt); err != nil {
			return err
		}
	}
	return nil
}
```

Server streaming sends until ctx cancel or EOF.

## Подводные камни

**Unary for huge payloads.** Hit max message size.

**Stream without deadline ctx.** Hung client.

**No grpc keepalive.** LB idle timeout drops silently.

**Mix streaming + retry naive.** Duplicate messages.

**Forget io.EOF on client Recv loop.**
