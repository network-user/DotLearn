# Go data structure closest to JSON

## In short

For dynamic JSON without a fixed schema use `map[string]interface{}` or `map[string]any`. For a known shape prefer a typed `struct` with `json` tags. Nested objects in a map become another `map[string]any`; arrays arrive as `[]interface{}`.

## How to answer in an interview

- `json.Unmarshal` into `map[string]any` when keys are unknown upfront.
- Type assertion `v.(string)` or `json.Number` for numbers without float loss.
- `json.RawMessage` defers parsing of a nested chunk.
- `struct` + tags for a stable API contract.
- Honestly: `map[string]any` loses types and risks runtime panic; in prod I prefer struct or `json.Decoder` into a concrete type.

## How it works

```go
raw := []byte(`{"name":"Ира","scores":[91,88],"meta":{"role":"admin"}}`)

var loose map[string]any
if err := json.Unmarshal(raw, &loose); err != nil {
	log.Fatal(err)
}

name := loose["name"].(string)
scores := loose["scores"].([]any)
meta := loose["meta"].(map[string]any)

type Profile struct {
	Name   string         `json:"name"`
	Scores []int          `json:"scores"`
	Meta   map[string]any `json:"meta"`
}

var typed Profile
json.Unmarshal(raw, &typed)
```

By default `encoding/json` unmarshals numbers into `float64` inside `interface{}`. For int64 from dynamic data use `json.Decoder` with `UseNumber()` and `json.Number.Int64()`.

## Pitfalls

**Direct assert without `ok` check.** Panic on unexpected type from external API.

**Float for large int64 IDs.** Precision loss on the `interface{}` path.

**Nil vs missing key.** In a map I distinguish absence and `null` via comma-ok.

**Nested `[]map[string]any` without schema.** Walk code grows; struct is better.

**Marshal map with non-deterministic iteration.** Flaky tests when comparing JSON strings.
