# Structure of an HTTP message

## In short

An HTTP message is a start line, header block, blank separator line, and optional body. On a request the start line is method, path, version. On a response: version, status code, reason phrase. HTTP/2 packs the same semantics into HEADERS and DATA frames.

## How to answer in an interview

- Request line vs status line.
- Headers `Host`, `Content-Type`, `Content-Length` or `Transfer-Encoding: chunked`.
- Body separated by a blank CRLF line in HTTP/1.1.
- `net/http` parses automatically into `*http.Request` / `ResponseWriter`.
- Honestly: HTTP/2 has no "text message" on the wire, but the model is the same.

## How it works

Request:

```http
POST /api/v1/orders HTTP/1.1
Host: shop.example
Content-Type: application/json
Content-Length: 38
Authorization: Bearer token

{"sku":"A12","qty":2,"currency":"RUB"}
```

Response:

```http
HTTP/1.1 201 Created
Content-Type: application/json
Location: /api/v1/orders/881
Content-Length: 20

{"id":881,"status":"new"}
```

```go
func createOrder(w http.ResponseWriter, r *http.Request) {
	log.Println(r.Method, r.URL.Path, r.Proto)
	log.Println("Content-Type:", r.Header.Get("Content-Type"))
	var payload Order
	json.NewDecoder(r.Body).Decode(&payload)
	w.Header().Set("Location", "/api/v1/orders/881")
	w.WriteHeader(http.StatusCreated)
	json.NewEncoder(w).Encode(map[string]any{"id": 881, "status": "new"})
}
```

## Pitfalls

**Read `r.Body` twice without TeeReader.** Second consumer gets an empty stream.

**Write body before response headers implicitly.** `Write` before `WriteHeader` locks in 200.

**Mix `Content-Length` and chunked.** RFC forbids both at once.

**Parse HTTP by hand.** Smuggling bugs; use stdlib or a vetted parser.

**Ignore `Expect: 100-continue`.** Some clients wait for an interim response.
