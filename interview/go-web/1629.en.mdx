# How HTTP/2 works

## In short

HTTP/2 multiplexes many streams on one TCP connection, encodes frames in binary, and compresses headers with HPACK. It removes HTTP/1.1 head-of-line blocking at the request level while keeping one connection per origin. ALPN in TLS negotiates h2 over HTTPS.

## How to answer in an interview

- Frames: DATA, HEADERS, SETTINGS, WINDOW_UPDATE, RST_STREAM.
- Stream ID for parallel request/response without a queue.
- HPACK table for repeated cookie and auth headers.
- Server push is rarely used; stream priorities depend on implementation.
- Honestly: TCP HOL blocking remains at the packet level; HTTP/3 (QUIC) solves it differently.

## How it works

```go
srv := &http.Server{
	Addr:    ":443",
	Handler: mux,
}
log.Fatal(srv.ListenAndServeTLS("fullchain.pem", "key.pem"))
```

Client and server exchange SETTINGS (window size, max concurrent streams). Requests `GET /static/app.js` and `GET /api/me` go on streams 1 and 3 over one TLS socket. Headers split into HEADERS frames; body into DATA frames with flow control.

Go `net/http` enables HTTP/2 server with TLS; the client tries h2 via ALPN by default.

## Pitfalls

**L7 proxy to backend over HTTP/1.1.** Multiplex gains lost on the last mile.

**Huge cookie on every stream.** HPACK helps but the table overflows.

**Misunderstanding flow control.** A slow consumer blocks a stream, not the whole connection.

**Debug with curl without `--http2`.** Server looks "slow" due to sequential requests.

**Server push without cache policy.** Extra traffic instead of speedup.
