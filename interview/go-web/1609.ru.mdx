# OAuth 2.0 basics

## Суть

OAuth 2.0 delegates access without sharing password: resource owner authorizes client; authorization server issues access token (and refresh); resource server validates token. Flows: authorization code (+PKCE for public clients), client credentials for service-to-service.

## Что ответить на собесе

- OAuth authorization, OpenID Connect adds identity (ID token).
- Authorization code + PKCE for SPA/mobile.
- Scopes limit access; short-lived access token.
- Честно: OAuth not authentication alone until OIDC id_token or userinfo.

## Как это работает

```
User -> Client -> Auth Server (login + consent)
Auth Server -> Client: authorization code
Client -> Auth Server: code + client_secret -> access_token
Client -> Resource Server: Bearer access_token
```

Go resource server validates JWT signature and `exp`, checks scope claim.

## Подводные камни

**Implicit flow legacy.** Use auth code + PKCE.

**Long-lived access token leak.** Rotate, bind to DPoP/mTLS.

**Confuse client credentials with user context.** Machine tokens only.

**Redirect URI open redirect.** Strict allowlist.

**Store refresh token insecurely.** Encrypted storage required.
