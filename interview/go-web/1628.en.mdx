# PUT vs PATCH: what's the difference

## In short

PUT replaces the whole resource: the client sends a full representation; missing fields are zeroed or removed per API rules. PATCH changes only the fields sent; the rest stays. PUT is idempotent; PATCH in general may differ on retry because of merge semantics.

## How to answer in an interview

- PUT `/users/42` with full profile JSON.
- PATCH `/users/42` with `{"email":"new@mail.com"}`.
- RFC 7396 JSON Merge Patch vs JSON Patch (RFC 6902) as formats.
- In Go separate handlers by `r.Method`.
- Honestly: many APIs call partial update PUT out of laziness; documented semantics matter.

## How it works

```go
func userHandler(w http.ResponseWriter, r *http.Request) {
	id := strings.TrimPrefix(r.URL.Path, "/users/")
	switch r.Method {
	case http.MethodPut:
		var body User
		json.NewDecoder(r.Body).Decode(&body)
		replaceUser(id, body)
		w.WriteHeader(http.StatusOK)
	case http.MethodPatch:
		var patch map[string]any
		json.NewDecoder(r.Body).Decode(&patch)
		applyPatch(id, patch)
		w.WriteHeader(http.StatusOK)
	default:
		w.WriteHeader(http.StatusMethodNotAllowed)
	}
}
```

PUT after two identical requests yields the same resource. PATCH with `{"tags":["go"]}` may append or replace tags depending on implementation; that must be fixed in OpenAPI.

## Pitfalls

**PATCH as PUT alias with one field.** Clients overwrite data when the model grows.

**Null in merge patch removes key.** Clients do not expect deletion on `{"nickname":null}`.

**PUT without If-Match ETag.** Lost update under concurrent edits.

**Idempotent retry POST instead of PUT.** Duplicate resources.

**No validation of partial state.** PATCH leaves the model in an invalid intermediate shape.
