# When to choose gRPC

## Суть

gRPC fits internal service mesh: binary protobuf, HTTP/2 multiplex, strong contracts, streaming, codegen in many langs. Prefer over REST JSON when latency, throughput, and schema evolution matter; public browser APIs often stay REST/GraphQL unless grpc-web gateway.

## Что ответить на собесе

- Microservice east-west traffic, polyglot teams.
- Need streaming or strict proto versioning.
- mTLS + interceptors for auth/metadata.
- Честно: REST simpler for debugging curl; gRPC needs tooling investment.

## Как это работает

```go
func main() {
	lis, _ := net.Listen("tcp", ":50051")
	srv := grpc.NewServer(
		grpc.UnaryInterceptor(loggingInterceptor),
	)
	pb.RegisterInventoryServer(srv, &inventoryService{})
	log.Fatal(srv.Serve(lis))
}
```

Client uses generated stub with context timeout. Proto breaking change = major version package `v2`.

## Подводные камни

**Public API gRPC only.** Browser limits without grpc-web.

**Huge proto messages.** Split or use streaming.

**Without reflection/debug tooling.** Hard onboarding.

**Ignore backward compatible field rules.** Wire breaks clients.

**Load balancer L7 not HTTP/2 aware.** Broken streams.
