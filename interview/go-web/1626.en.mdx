# Why Protobuf is faster than JSON

## In short

Protobuf encodes data in a compact binary wire format: field type and number instead of repeating key names. The schema is known upfront; the parser does not build a generic AST like JSON. Fewer bytes on the wire and less CPU on marshal/unmarshal at high QPS.

## How to answer in an interview

- Varint encoding, field tags, no whitespace or quotes.
- `proto.Marshal` / `Unmarshal` vs `json` reflection or codegen.
- JSON is convenient for humans and browsers; protobuf for service-to-service.
- Benchmark on real message sizes, not hello world.
- Honestly: on tiny payloads the gap may drown in network latency; the win grows with field count and traffic.

## How it works

```protobuf
syntax = "proto3";

message Account {
  int64 id = 1;
  string login = 2;
  string email = 3;
}
```

```go
acc := &pb.Account{Id: 99, Login: "dev", Email: "dev@corp.local"}
wire, err := proto.Marshal(acc)

var decoded pb.Account
err = proto.Unmarshal(wire, &decoded)
```

The JSON equivalent `{"id":99,"login":"dev","email":"dev@corp.local"}` carries field names in every message. Protobuf wire stores tag `1` + varint `99`, tag `2` + length-delimited string. Varint numbers use fewer bytes for small values.

## Pitfalls

**Compare protobuf to `json:",string"` and pretty print.** Unfair benchmark.

**Dynamic message without codegen.** Protobuf reflection is slower; part of the win is lost.

**Huge string/bytes fields.** Wire compactness does not shrink the payload itself.

**Human debugging.** You need `grpcurl`, hex dump, or JSON transcoding.

**float/double as text in JSON vs fixed64.** Different size, rarely the deciding factor.
