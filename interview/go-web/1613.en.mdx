# When to choose gRPC

## In short

gRPC fits internal service mesh: binary protobuf, HTTP/2 multiplex, strong contracts, streaming, codegen in many languages. Prefer it over REST JSON when latency, throughput, and schema evolution matter; public browser APIs often stay REST/GraphQL unless a grpc-web gateway exists.

## How to answer in an interview

- Microservice east-west traffic, polyglot teams.
- Need streaming or strict proto versioning.
- mTLS plus interceptors for auth/metadata.
- Honestly: REST is simpler for curl debugging; gRPC needs tooling investment.

## How it works

```go
func main() {
	lis, _ := net.Listen("tcp", ":50051")
	srv := grpc.NewServer(
		grpc.UnaryInterceptor(loggingInterceptor),
	)
	pb.RegisterInventoryServer(srv, &inventoryService{})
	log.Fatal(srv.Serve(lis))
}
```

The client uses a generated stub with context timeout. A breaking proto change means a major version package `v2`.

## Pitfalls

**Public API gRPC only.** Browser limits without grpc-web.

**Huge proto messages.** Split or use streaming.

**Without reflection/debug tooling.** Hard onboarding.

**Ignore backward-compatible field rules.** Wire breaks clients.

**L7 load balancer not HTTP/2 aware.** Broken streams.
