# Индексы: чем платим за ускорение чтения и почему запись становится дороже

## Суть

Индекс — отдельная структура (чаще B-дерево), которая позволяет находить строки без полного сканирования таблицы. Чтение ускоряется, но за это платят диском и замедлением вставок, обновлений и удалений.

## Что ответить на собесе

Индекс — упорядоченная вспомогательная структура по одному или нескольким столбцам, как предметный указатель в книге: вместо перебора всех строк база спускается по дереву за логарифмическое время.

- Выгода видна на фильтрации, сортировке и соединениях по проиндексированным колонкам.
- Обратная сторона: каждый индекс занимает место и при любой записи его нужно поддерживать, поэтому `INSERT`/`UPDATE`/`DELETE` становятся медленнее.
- Вывод: индексируем то, по чему реально часто ищем, а не всё подряд.

## Как это работает

Под капотом у большинства реляционных БД индекс — это B-дерево: сбалансированная структура, где поиск, вставка и удаление идут за O(log n). Есть и хэш-индексы (быстрый поиск по равенству, но не по диапазону).

Создание индекса по столбцу:

```sql
CREATE INDEX idx_employee_name ON employees (name);
```

После этого запрос с фильтром по `name` уже не сканирует всю таблицу:

```sql
SELECT * FROM employees WHERE name = 'John';
```

База спускается по дереву индекса к нужным значениям и достаёт только подходящие строки. Без индекса тот же запрос на большой таблице — это последовательное чтение каждой записи.

Где индекс помогает сильнее всего:

1. Фильтрация в `WHERE` и сортировка в `ORDER BY` по проиндексированному столбцу.
2. Соединения `JOIN` по ключам, на которых стоят индексы.
3. Уникальность через `UNIQUE INDEX`, который заодно гарантирует отсутствие дублей.

## Подводные камни

- **Накладные расходы на запись**: каждый лишний индекс — дополнительная работа при каждой модификации строки. На таблицах с интенсивной записью это ощутимо.
- **Перебор с индексами**: десяток индексов «на всякий случай» раздувает базу и тормозит запись, а планировщик всё равно использует единицы.
- **Индекс может игнорироваться**: функция над столбцом (`WHERE lower(name) = ...`) или ведущий `%` в `LIKE '%foo'` ломают использование обычного индекса.
- **Порядок столбцов в составном индексе важен**: индекс `(a, b)` помогает фильтру по `a` и по `a, b`, но не по одному `b`.
- **Селективность**: индекс по столбцу с двумя значениями (флаг да/нет) почти бесполезен — база всё равно прочитает половину таблицы.
