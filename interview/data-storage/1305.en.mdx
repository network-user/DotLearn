# Database locks: what they are and what role they play

## In short

A lock is a concurrency-control mechanism that prevents multiple transactions from modifying the same data at the same time. This is how a DBMS preserves integrity and guards against anomalies: dirty reads, lost updates, and phantoms.

## How to answer in an interview

I explain it through the problem: when transactions run in parallel, anomalies arise without coordination - dirty reads, non-repeatable reads, lost updates, phantoms. Locks coordinate access. There's a shared (S) lock for reads, which several transactions can hold at once, but nobody can write. And there's an exclusive (X) lock for writes, held by only one transaction while the rest wait. Locks can apply at the row, page, table, or whole-database level: the finer the granularity, the higher the concurrency, but the greater the overhead. The transaction isolation level is precisely what determines which locks are taken and when. The flip side is deadlocks, where two transactions wait on each other's locks.

The anomalies locks protect against:

- **Dirty Read:** a transaction sees another transaction's uncommitted changes; if they get rolled back, the data turns out to be wrong.
- **Lost Update:** two transactions edit the same record, and one edit overwrites the other.
- **Non-repeatable Read:** rereading the same row yields a different result, because it was modified in between.
- **Phantom Read:** a repeated query with a condition returns new rows that weren't there before.

The main types of locks:

- **Exclusive (X):** only one transaction may modify the data; the rest wait for it to release.
- **Shared (S):** several transactions read at once, but writing is blocked.

Example with two transactions over one record:

```sql
-- Transaction 1
BEGIN TRANSACTION;
SELECT * FROM accounts WHERE account_id = 1;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 1;
COMMIT;

-- Transaction 2
BEGIN TRANSACTION;
SELECT * FROM accounts WHERE account_id = 1;
UPDATE accounts SET balance = balance - 50 WHERE account_id = 1;
COMMIT;
```

The `UPDATE` takes an exclusive row lock, and the second transaction waits for the first to `COMMIT` before applying its own change. `COMMIT` releases the locks and hands the data back to the others.

## Pitfalls

- **Deadlock:** two transactions acquire resources in different orders and wait on each other. The DBMS detects the cycle and forcibly rolls one of them back - the code must be able to retry the transaction.
- Too coarse a granularity (locking a table instead of a row) kills concurrency; too fine raises the overhead of tracking locks.
- In PostgreSQL, readers generally don't block writers thanks to MVCC, so the locking model there doesn't match the classic "S/X on every read."
- A long transaction holding a lock builds a queue and tanks throughput - keep transactions short.
- An explicit `SELECT ... FOR UPDATE` takes a row lock up front and helps against lost updates.
