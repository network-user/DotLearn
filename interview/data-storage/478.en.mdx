# Why does a many-to-many relationship require a third table?

## In short

The relational model can't store M2M directly: a "many-to-many" relationship is decomposed into a separate intermediate (join) table. It holds foreign keys pointing to the primary keys of both sides, and each row is a single "left-right" pair.

## How to answer in an interview

A single table row can't hold a variable number of references, so you can't join two "manys" directly. The solution is a junction table with two foreign keys. The classic example: students and courses. The `StudentCourses` table stores `(student_id, course_id)` pairs, and a composite primary key over these two columns guarantees the pair is unique. This way one student links to many courses, and a course to many students, and any link can be added or removed with a single row.

## How it works

Between two tables you create a join table with two FKs. The composite PK `(student_id, course_id)` forbids duplicate pairs, while the foreign keys maintain referential integrity.

```sql
CREATE TABLE Students (
    student_id INT PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE Courses (
    course_id INT PRIMARY KEY,
    title VARCHAR(100)
);

CREATE TABLE StudentCourses (
    student_id INT,
    course_id INT,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES Students(student_id),
    FOREIGN KEY (course_id) REFERENCES Courses(course_id)
);
```

Linking a student to a course means inserting a row into `StudentCourses`; breaking the link means deleting that row. New links don't require changing the structure of the main tables. In Django this is a `ManyToManyField`, under which the ORM creates exactly such a join table automatically.

## Pitfalls

- Without a composite primary key (or a `UNIQUE` on the pair), duplicates of the same pair will creep into the junction table.
- If the relationship has its own attributes (enrollment date, grade), you need an explicit junction model; in Django that's a `through` model, not a bare `ManyToManyField`.
- Queries across M2M almost always pull in a `JOIN`; for frequent selects, indexes on the FK columns are mandatory, otherwise you get degradation on large tables.
- Deleting a record on one side must clean up the junction correctly: think through `ON DELETE` (or `on_delete` in Django), otherwise you'll be left with dangling rows or integrity errors.
