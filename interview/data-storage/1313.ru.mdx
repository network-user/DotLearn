# Как смоделировать отношение «один ко многим» в реляционной БД

## Суть

Отношение «один ко многим» реализуют через внешний ключ на стороне «многие»: дочерняя таблица хранит столбец, ссылающийся на первичный ключ родительской. Одна родительская строка соответствует множеству дочерних, но каждая дочерняя - ровно одному родителю.

## Что ответить на собесе

Я кладу внешний ключ в таблицу со стороны «многие», а не «один». Классика - автор и книги: один автор пишет много книг, у каждой книги один автор. Значит, в таблице `Books` появляется столбец `AuthorID` - внешний ключ на `Authors.AuthorID`. Никакой отдельной связующей таблицы тут не нужно (она требуется только для «многие ко многим»). База через FK следит, чтобы каждая книга ссылалась на существующего автора, а собрать данные вместе помогает `JOIN` по этому полю.

## Как это работает

Берём `Authors` (сторона «один») и `Books` (сторона «многие»). Внешний ключ живёт в `Books`:

```sql
CREATE TABLE Authors (
    AuthorID INT PRIMARY KEY,   -- идентификатор автора
    Name     VARCHAR(100)
);

CREATE TABLE Books (
    BookID   INT PRIMARY KEY,   -- идентификатор книги
    Title    VARCHAR(100),
    AuthorID INT,               -- внешний ключ на Authors
    FOREIGN KEY (AuthorID) REFERENCES Authors(AuthorID)
);
```

Чтобы собрать все книги конкретного автора, соединяем таблицы по `AuthorID`:

```sql
SELECT Books.Title
FROM Books
JOIN Authors ON Books.AuthorID = Authors.AuthorID
WHERE Authors.Name = 'Имя Автора';
```

`JOIN` сшивает книги с авторами по общему полю, а `WHERE` оставляет только нужного автора.

## Подводные камни

- **FK кладут на сторону «многие»**, а не «один». Попытка хранить список ID детей в родительской строке (через запятую или массив без нормализации) - типичная ошибка, ломающая целостность и запросы.
- **Сторону «один» нельзя продублировать в дочерней таблице**: не надо копировать имя автора в каждую книгу - храните только `AuthorID`, иначе данные разойдутся.
- **NULL в FK = «родитель не назначен»**. Если книга обязана иметь автора, добавьте `NOT NULL` на `AuthorID`, иначе появятся книги-сироты.
- **Удаление родителя**: без правила `ON DELETE` база по умолчанию не даст удалить автора, у которого есть книги. Нужно осознанно выбрать `CASCADE`, `SET NULL` или оставить `RESTRICT`.
- **Индекс на FK-столбец** добавляют вручную: в PostgreSQL он не создаётся автоматически, а без него `JOIN` и фильтры по `AuthorID` будут медленными на больших таблицах.
