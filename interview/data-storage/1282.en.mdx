# A query is slow: which route to take to find the cause and speed it up?

## In short

Optimizing slow SQL isn't one trick but a sequence of steps: first look at the execution plan, then fix indexes, simplify the query itself, touch the schema if needed, and refresh statistics. Hardware is the last resort, not the first.

## How to answer in an interview

I start with `EXPLAIN` (or `EXPLAIN ANALYZE`): the plan shows where the database does a full scan, where it loses time on sorting or joining. Next I check indexes on the columns from `WHERE` and `JOIN`, rewrite the query so it reads less data (the needed columns instead of `SELECT *`, joins instead of heavy subqueries). If that isn't enough, I look at the schema and how current the planner's statistics are. Scaling up server resources is a last resort, used when the logic has already been squeezed dry.

The first thing to do is ask the database to show how it intends to execute the query:

```sql
EXPLAIN SELECT name FROM users WHERE email = 'example@example.com';
```

The plan reveals the bottlenecks: a sequential table scan where index access was expected, expensive joins, unnecessary sorts.

If a filter frequently runs on a column, it's worth indexing - then the database finds rows directly instead of scanning the whole table:

```sql
CREATE INDEX idx_email ON users(email);
```

A query can almost always be made cheaper. We replace a subquery with a join and ask only for the needed columns:

```sql
SELECT users.name, orders.amount
FROM users
JOIN orders ON users.id = orders.user_id;
```

Sometimes the structure itself is the slow part. Normalization removes duplication but spawns joins; denormalization adds redundancy for fast reads - the choice depends on the load profile.

The planner relies on statistics about data distribution. If they're stale, it builds a bad plan - that's why they get refreshed:

```sql
ANALYZE users;
```

## Pitfalls

- Extra indexes aren't free: they take up space and slow down every insert and update, because they too must be maintained.
- An index on a column isn't used if a function or type cast is applied to it in `WHERE` (`WHERE lower(email) = ...` breaks an ordinary index on `email`).
- `SELECT *` defeats covering indexes and pulls extra data over the network.
- Upgrading the server before looking at the plan is a common mistake: you pay money for a problem that a single index would have solved.
