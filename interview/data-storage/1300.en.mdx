# Normalization vs. denormalization: where to remove duplicates and where to deliberately bring them back

## In short

Normalization spreads data across related tables so each fact is stored exactly once, which kills duplicates and update anomalies. Denormalization is a deliberate reintroduction of redundancy for the sake of read speed.

## How to answer in an interview

Normalization is a set of schema-design rules that removes data duplication and protects against insert, update, and delete anomalies. It's achieved by splitting into tables and linking them by keys: 1NF requires atomic values, 2NF requires non-key columns to depend on the whole key, 3NF requires no transitive dependencies. Denormalization goes the other way: we deliberately duplicate data to cut down on `JOIN`s and speed up reads. It's justified in analytics and data marts, where reads happen far more often than writes. By default you design normalized, and you denormalize selectively against a proven performance problem.

## How it works

Normal forms build up in tiers:

1. **1NF**: each cell holds one atomic value, with no repeating groups.
2. **2NF**: already 1NF, and every non-key column depends on the entire primary key, not part of it (relevant for composite keys).
3. **3NF**: already 2NF, and there are no transitive dependencies: non-key columns aren't derived from one another.

A simple example. If an orders table holds the customer's name, then with ten orders from one person the name is repeated ten times. Let's move the customer into a separate table and link by key:

```sql
CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT REFERENCES customers(customer_id),
    amount NUMERIC(10, 2)
);
```

Now the customer's name is stored once; a name change is a single update, not a walk over all orders.

Denormalization is appropriate when:

- **Reads dominate**: in analytical stores, redundancy removes joins and speeds up queries.
- **A simple schema matters more than purity**: in small projects, fewer tables means faster development.
- **Active caching and replication**: flat data is easier to distribute across nodes.

## Pitfalls

- **Anomalies under denormalization**: having duplicated the customer's name into orders, on a name change it's easy to get desync if you don't update all copies.
- **Denormalization without measurement**: adding redundancy "just in case" is harmful. First prove that the `JOIN` really is the bottleneck.
- **Over-deep normalization**: a dozen tables for a single entity spawns expensive multi-level `JOIN`s and complicates queries.
- **3NF is not the limit**: there's BCNF and beyond, but for an interview a solid command of 1–3NF is usually enough.
- Normalization protects integrity through constraints (`FOREIGN KEY`); denormalization offloads that protection onto application code.
