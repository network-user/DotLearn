# Медленный SQL-запрос: как поймать виновника и прочитать план выполнения

## Суть

Запрос медленный, когда работает дольше ожидаемого. Чтобы понять причину, смотрят план выполнения через `EXPLAIN`, ищут полные сканирования и отсутствующие индексы, а затем сокращают ввод-вывод и объём возвращаемых данных.

## Что ответить на собесе

Разбор идёт по шагам: найти, прочитать план, понять причину, вылечить.

- **Найти запрос** через мониторинг (`pg_stat_statements` в PostgreSQL, `Performance Schema` в MySQL) или лог медленных запросов.
- **Прочитать план** через `EXPLAIN` (или `EXPLAIN ANALYZE`): какие индексы используются, сколько строк планируется обработать, где идёт полное сканирование таблицы.
- **Типичные причины**: нет индекса на столбцах из `WHERE`/`JOIN`/`ORDER BY`, запрос тянет лишние строки или столбцы, устаревшая статистика сбивает оптимизатор.
- **Лечим**: добавляем подходящий индекс, переписываем запрос (`JOIN` вместо подзапроса, `LIMIT`), обновляем статистику через `ANALYZE`.

## Как это работает

Порядок действий при разборе:

1. **Найти запрос.** Включаем лог медленных запросов с порогом по времени или смотрим агрегированную статистику расширений мониторинга.
2. **Прочитать план.** `EXPLAIN` показывает, что планировщик собирается делать; `EXPLAIN ANALYZE` к тому же реально выполняет запрос и даёт фактическое время.

```sql
EXPLAIN SELECT * FROM orders WHERE customer_id = 123;
```

3. **Найти узкое место.** В плане ищем `Seq Scan` (полное сканирование) там, где ожидался `Index Scan`, и операции с большим числом строк.
4. **Добавить индекс.** Если фильтр идёт по двум столбцам сразу, помогает составной индекс:

```sql
CREATE INDEX idx_customer_order_date ON orders (customer_id, order_date);
```

После этого запрос вида

```sql
SELECT * FROM orders WHERE customer_id = 123 AND order_date > '2023-01-01';
```

пойдёт по индексу вместо перебора всей таблицы.

5. **Обновить статистику.** Оптимизатор опирается на оценки распределения данных; после массовых изменений их освежают:

```sql
ANALYZE orders;
```

## Подводные камни

- **`EXPLAIN` без `ANALYZE` врёт по времени**: он показывает оценку планировщика, а не реальную длительность. Реальные числа даёт `EXPLAIN ANALYZE` (но он выполняет запрос).
- **Индекс не используется**: функция над столбцом или несоответствие типов в условии заставляют планировщик игнорировать индекс.
- **`SELECT *` мешает**: лишние столбцы раздувают объём чтения и мешают index-only сканированию.
- **Устаревшая статистика**: после крупной загрузки данных без `ANALYZE` оптимизатор может выбрать плохой план.
- **Оптимизация не туда**: ускорять стоит запросы, которые реально влияют на нагрузку, а не первый попавшийся медленный, который выполняется раз в сутки.
