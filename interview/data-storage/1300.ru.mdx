# Нормализация против денормализации: где убрать дубли, а где сознательно их вернуть

## Суть

Нормализация раскладывает данные по связанным таблицам так, чтобы каждый факт хранился ровно один раз. Это убивает дубли и аномалии при изменениях. Денормализация: осознанный возврат избыточности ради скорости чтения.

## Что ответить на собесе

Это две противоположные стратегии работы с дублированием; по умолчанию проектируют нормализованно, а денормализуют точечно под доказанную проблему производительности.

- **Нормализация** убирает дублирование и защищает от аномалий вставки, обновления и удаления через разбиение на таблицы и связи по ключам (1NF/2NF/3NF, детали ниже).
- **Денормализация** идёт в обратную сторону: сознательно дублируем данные, чтобы сократить число `JOIN` и ускорить чтение.
- Денормализация оправдана в аналитике и витринах, где читают намного чаще, чем пишут.

## Как это работает

Нормальные формы наращиваются ступенями:

1. **1NF**: каждая ячейка хранит одно атомарное значение, повторяющихся групп нет.
2. **2NF**: уже 1NF, и каждый неключевой столбец зависит от первичного ключа целиком, а не от его части (актуально для составных ключей).
3. **3NF** - уже 2NF, и нет транзитивных зависимостей: неключевые столбцы не выводятся друг из друга.

Простой пример. Если в таблице заказов лежит имя клиента, то при десяти заказах одного человека имя повторяется десять раз. Вынесем клиента в отдельную таблицу и свяжем по ключу:

```sql
CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT REFERENCES customers(customer_id),
    amount NUMERIC(10, 2)
);
```

Теперь имя клиента хранится один раз; смена имени требует одного обновления, а не обхода всех заказов.

Денормализация уместна, когда:

- **Доминирует чтение**: в аналитических хранилищах избыточность убирает соединения и ускоряет выборки.
- **Простая схема важнее чистоты**: в маленьких проектах меньше таблиц, поэтому разработка идёт быстрее.
- **Активные кэш и репликация**: плоские данные проще раздавать по узлам.

## Подводные камни

- **Аномалии при денормализации**: продублировав имя клиента в заказах, при его смене легко получить рассинхрон, если обновить не все копии.
- **Денормализация без замера**: добавлять избыточность «на всякий случай» вредно: сначала нужно доказать, что `JOIN` действительно узкое место.
- **Слишком глубокая нормализация**: десяток таблиц ради одной сущности порождает дорогие многоэтажные `JOIN` и усложняет запросы.
- **3NF - не предел**: бывают BCNF и выше, но на собесе обычно достаточно уверенно владеть 1–3NF.
- Нормализация защищает целостность через ограничения (`FOREIGN KEY`), денормализация эту защиту перекладывает на код приложения.
