# Indexes: what you pay for faster reads and why writes get more expensive

## In short

An index is a separate structure (usually a B-tree) that lets you find rows without scanning the whole table. Reads speed up, but you pay for it with disk space and slower inserts, updates, and deletes.

## How to answer in an interview

An index is an ordered auxiliary structure over one or more columns, working like a book's index: instead of walking every row, the database descends the tree in logarithmic time. The benefit shows up in filtering, sorting, and joins on indexed columns. The flip side: each index takes up space and must be kept up to date on every write, so `INSERT`/`UPDATE`/`DELETE` get slower. The takeaway: index what you actually search by often, not everything in sight.

## How it works

Under the hood, in most relational databases an index is a B-tree: a balanced structure where search, insertion, and deletion run in O(log n). There are also hash indexes (fast equality lookups, but not range lookups).

Creating an index on a column:

```sql
CREATE INDEX idx_employee_name ON employees (name);
```

After this, a query filtering by `name` no longer scans the whole table:

```sql
SELECT * FROM employees WHERE name = 'John';
```

The database descends the index tree to the needed values and fetches only the matching rows. Without an index, the same query on a large table is a sequential read of every record.

Where an index helps most:

1. Filtering in `WHERE` and sorting in `ORDER BY` on an indexed column.
2. `JOIN`s on keys that have indexes.
3. Uniqueness via a `UNIQUE INDEX`, which also guarantees there are no duplicates.

## Pitfalls

- **Write overhead**: every extra index is additional work on every row modification. On write-heavy tables this is noticeable.
- **Too many indexes**: a dozen indexes "just in case" bloat the database and slow down writes, while the planner still uses only a few.
- **An index can be ignored**: a function over a column (`WHERE lower(name) = ...`) or a leading `%` in `LIKE '%foo'` breaks the use of an ordinary index.
- **Column order in a composite index matters**: an index `(a, b)` helps filtering by `a` and by `a, b`, but not by `b` alone.
- **Selectivity**: an index on a column with two values (a yes/no flag) is nearly useless — the database still reads half the table.
