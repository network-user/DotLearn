# Четыре уровня изоляции транзакций и аномалии, которые они отсекают

## Суть

Уровень изоляции определяет, какие промежуточные результаты соседних транзакций видит ваша. Стандарт SQL описывает четыре ступени — от Read Uncommitted до Serializable; каждая следующая отсекает больше аномалий чтения, но стоит дороже по производительности.

## Что ответить на собесе

Изоляция — это про то, какие аномалии параллельных транзакций допустимы. Аномалий три: грязное чтение (видим незафиксированные правки), неповторяющееся чтение (строка меняется между двумя SELECT) и фантомы (в прочитанный диапазон добавляются новые строки). Четыре уровня закрывают их последовательно:

- **Read Uncommitted** — не защищает ни от чего;
- **Read Committed** — убирает грязное чтение;
- **Repeatable Read** — убирает ещё и неповторяющееся;
- **Serializable** — исключает и фантомы, давая поведение как при последовательном выполнении.

Чем строже уровень, тем больше блокировок и откатов, поэтому по умолчанию обычно берут Read Committed и поднимают планку только там, где это действительно нужно.

Разберём ступени снизу вверх:

1. **Read Uncommitted.** Транзакция видит ещё не зафиксированные правки чужих транзакций. Отсюда грязные чтения: если соседняя транзакция откатится, вы останетесь с несуществующими данными. Применяют редко, разве что для черновых отчётов, где мусор не страшен.
2. **Read Committed.** Видны только зафиксированные изменения, грязное чтение исключено. Но между двумя SELECT внутри одной транзакции строка может измениться — это неповторяющееся чтение. Самый ходовой уровень и дефолт во многих СУБД.
3. **Repeatable Read.** Прочитанные строки не меняются до конца транзакции — неповторяющихся чтений нет. Остаются фантомы: в уже прочитанный диапазон могут добавиться новые строки.
4. **Serializable.** Самый строгий уровень: результат такой, будто транзакции выполнялись по очереди. Исключает все три аномалии ценой блокировок, ожиданий и откатов при конфликтах.

Уровень задаётся перед началом транзакции:

```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

BEGIN;

SELECT balance FROM accounts WHERE account_id = 1;

UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;

COMMIT;
```

После `COMMIT` правки становятся видны остальным транзакциям; до этого их видимость и определяется выбранным уровнем.

## Подводные камни

- **Стандарт описывает минимум гарантий, а не реализацию.** Реальные СУБД ведут себя по-разному: в PostgreSQL Repeatable Read основан на снапшотах и уже не допускает фантомов, а Read Uncommitted фактически работает как Read Committed.
- **Repeatable Read в MySQL (InnoDB) и в стандарте — разные вещи.** InnoDB на этом уровне через next-key locks подавляет и фантомы для текущих запросов, что выходит за рамки определения стандарта.
- **Выше изоляция — выше риск deadlock и откатов.** На Serializable транзакции чаще отменяются из-за конфликтов сериализации, и приложение обязано уметь повторять их.
- **Изоляция не равна корректности.** Даже без классических аномалий возможны write skew и lost update; от write skew надёжно защищает только Serializable.
- **Уровень ставят до начала работы транзакции.** Поменять изоляцию посреди уже идущей транзакции нельзя.
