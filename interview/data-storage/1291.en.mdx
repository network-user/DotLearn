# Managing the schema with Alembic: revision, downgrade, heads

## In short

Alembic versions the database schema: `revision` creates a new migration, `downgrade` rolls the database back one step, `heads` shows the current tips of the revision tree. This trio covers creating, rolling back and diagnosing changes.

## How to answer in an interview

Alembic is the standard migration tool for SQLAlchemy. Each migration is stored as a separate file with `upgrade()` and `downgrade()` functions and a reference to its parent revision, forming a directed chain. `revision` creates such a file, `downgrade` applies the reverse changes, and `heads` lists revisions with no descendants. If there are several branches and more than one head appears, they're reconciled with the `merge` command.

`revision` generates a new migration file in the `versions` directory:

```bash
alembic revision -m "add new table"
```

The `-m` flag sets a human-readable name. With `--autogenerate` Alembic itself compares the SQLAlchemy models with the current schema and pre-fills the changes.

`downgrade` runs the `downgrade()` function of the relevant migrations:

```bash
alembic downgrade -1
```

The `-1` argument means rolling back one revision relative to the current one; you can also specify a concrete revision identifier or `base` to roll everything back.

`heads` shows the tips - revisions that no one references as a parent:

```bash
alembic heads
```

Normally there's a single head. Several heads are a sign of branches that diverged after parallel development.

## Pitfalls

- `revision` (without `--autogenerate`) creates an empty migration - you'll have to write the body of `upgrade()`/`downgrade()` by hand. This is a common cause of "migrations that do nothing".
- Autogeneration doesn't see everything: a column type change, server-side defaults and renames are often missed or treated as drop + add. The generated output must be reviewed.
- Multiple `heads` break `upgrade head` with ambiguity - the branches need to be merged via `alembic merge`.
- `downgrade` is fine in a dev environment, but in production a reverse migration often loses data (for example, when dropping a column). A rollback isn't a free undo.
