# EXPLAIN versus EXPLAIN ANALYZE: the plan in theory and the fact by timing

## In short

`EXPLAIN` shows the plan the planner intends to execute, but it doesn't run the query itself - every number in it is an estimate. `EXPLAIN ANALYZE` actually runs the query and fills in the real timing and row counts, so you can compare the forecast with reality.

## How to answer in an interview

"`EXPLAIN` is a forecast: the database prints a tree of operations with estimated cost and row counts, but the query isn't executed, so it's safe and instant. `EXPLAIN ANALYZE` actually runs the query and, next to the estimates, puts the real timing and actual `rows`. The most useful thing is the difference between expected and actual rows: if the planner expected 10 rows but a million arrived, the statistics are stale or the cardinality estimate missed, and bad plans grow from there." It's worth adding that `ANALYZE` really does modify data for `INSERT/UPDATE/DELETE`, so such queries are wrapped in a transaction with a rollback.

## How it works

`EXPLAIN` produces a tree of plan nodes: scan type (Seq Scan, Index Scan), join order, estimated cost `cost=start..end` and the assumed number of rows.

```sql
EXPLAIN
SELECT * FROM orders WHERE customer_id = 123;
```

`EXPLAIN ANALYZE` runs the query and adds `actual time` and real `rows` to each node, plus an overall `Execution Time`.

```sql
EXPLAIN ANALYZE
SELECT * FROM orders WHERE customer_id = 123;
```

Useful output extensions are `BUFFERS` (how many pages were read from cache and from disk) and `FORMAT JSON` for parsing by tools.

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders WHERE customer_id = 123;
```

You read the plan bottom-up and inside-out: the most deeply nested nodes execute first, and their result bubbles up to the parents.

## Pitfalls

- `EXPLAIN ANALYZE` physically executes the query. For `UPDATE`, `DELETE` and `INSERT` that means real changes, so run it inside `BEGIN; ... ROLLBACK;`.
- The estimated `cost` is the planner's arbitrary units, not milliseconds; you can't compare it directly with real time.
- The main problem signal is the divergence between estimated and actual rows. It's cured by refreshing the statistics with `ANALYZE table_name`, not by editing the query.
- The measurement itself takes time: `EXPLAIN ANALYZE` adds instrumentation overhead, so absolute numbers on short queries can be inflated.
