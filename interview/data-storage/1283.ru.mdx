# Зачем базе индекс и какой ценой он ускоряет выборку?

## Суть

Индекс - вспомогательная структура поверх таблицы, которая даёт базе короткий путь к строкам без перебора всех данных. Чтение ускоряется заметно, но за это платят местом на диске и замедлением записи.

## Что ответить на собесе

Индекс похож на предметный указатель в книге: вместо чтения от корки до корки база сразу прыгает на нужные страницы.

- Технически чаще всего это B-дерево по одному или нескольким столбцам, реже хэш-структура.
- Резко ускоряет `SELECT` с фильтром по проиндексированному полю.
- Цена: каждая вставка, обновление и удаление теперь обязаны ещё и обновить индекс, плюс он занимает дополнительное место.
- Поэтому индексы вешают точечно на столбцы, по которым реально часто ищут.

## Как это работает

Без индекса, чтобы достать нужные строки, система проходит таблицу строка за строкой - на больших объёмах это дорого. Индекс хранит значения столбца в структуре, где поиск быстрый: в B-дереве - за логарифмическое время.

Возьмём таблицу `employees` со столбцами `id`, `name`, `department`. Если поиск по имени частый, индексируем `name`:

```sql
CREATE INDEX idx_name ON employees(name);
```

Теперь запрос `SELECT * FROM employees WHERE name = 'Alice'` идёт через индекс и находит строки напрямую, а не сканирует всю таблицу.

Под капотом обычно лежит B-дерево: оно держит значения упорядоченными, поэтому годится и для равенства, и для диапазонов (`>`, `<`, `BETWEEN`), и для сортировки. Хэш-индекс - альтернатива только для точного равенства.

## Подводные камни

- Индекс не бесплатен на запись: чем их больше, тем медленнее вставки и обновления.
- На маленьких таблицах индекс может игнорироваться - планировщику дешевле прочитать всё.
- Низкая селективность убивает пользу: индекс по столбцу с двумя значениями (например, `is_active`) почти бесполезен.
- Индекс работает только когда столбец используется без обёрток - функция или приведение типа в условии его отключают.
