# Первичный и внешний ключ: в чём принципиальная разница

## Суть

Первичный ключ (primary key) однозначно идентифицирует строку внутри своей таблицы - значения уникальны и не бывают NULL. Внешний ключ (foreign key) - это столбец, который ссылается на первичный ключ другой таблицы и тем самым связывает таблицы, поддерживая ссылочную целостность.

## Что ответить на собесе

Primary key отвечает на вопрос «как однозначно опознать эту строку»: он уникален, не допускает NULL, и СУБД строит под него индекс. Foreign key отвечает на вопрос «на какую строку в другой таблице я указываю»: он хранит значение первичного ключа из таблицы-родителя, и база следит, чтобы такое значение там действительно существовало. Грубо: PK - про идентичность записи, FK - про связь между записями. В одной таблице PK ровно один, а FK может быть несколько.

## Как это работает

**Primary key** делает каждую строку отличимой. Значения уникальны, NULL запрещён, под ключ автоматически создаётся индекс - поэтому поиск по нему быстрый.

```sql
CREATE TABLE Users (
    UserID   INT PRIMARY KEY,   -- уникальный идентификатор пользователя
    UserName VARCHAR(100),
    Email    VARCHAR(100)
);
```

**Foreign key** хранит ссылку на первичный ключ другой таблицы. При вставке или изменении БД проверяет, что такое значение есть в таблице-источнике, иначе операция отклоняется.

```sql
CREATE TABLE Orders (
    OrderID   INT PRIMARY KEY,
    OrderDate DATE,
    UserID    INT,
    FOREIGN KEY (UserID) REFERENCES Users(UserID)  -- ссылка на Users
);
```

Здесь `Orders.UserID` указывает на `Users.UserID`. Каждый заказ гарантированно привязан к существующему пользователю.

Ключевая механика:

- **Primary key**: СУБД строит индекс, обеспечивает уникальность и запрещает NULL.
- **Foreign key**: при вставке/обновлении проверяется существование значения в родительской таблице; при удалении родителя срабатывает заданное правило (`RESTRICT`, `CASCADE`, `SET NULL`), что предотвращает появление «осиротевших» строк.

## Подводные камни

- **NULL в FK обычно разрешён**, а в PK - никогда. Внешний ключ со значением NULL означает «связи пока нет» и не нарушает целостность.
- **FK ссылается не на любой столбец**, а на PK или столбец с ограничением уникальности в родительской таблице. На неуникальный столбец сослаться нельзя.
- **`ON DELETE` по умолчанию** часто `RESTRICT/NO ACTION`: удалить пользователя, на которого ссылаются заказы, не дадут. Хотите каскад - указывайте `ON DELETE CASCADE` осознанно, иначе можно случайно снести связанные данные.
- **FK не индексируется автоматически** (в отличие от PK) во многих СУБД, включая PostgreSQL. Join и проверки по внешнему ключу без индекса будут медленными - индекс на FK-столбец добавляют вручную.
- **Составной PK** возможен (несколько столбцов), и тогда FK на него тоже должен быть составным и совпадать по набору столбцов.
