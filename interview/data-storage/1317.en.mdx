# DELETE vs TRUNCATE: what is the practical difference?

## In short

`DELETE` is DML: it removes rows by condition, fires triggers, and logs every row. `TRUNCATE` is DDL: it instantly clears the whole table and resets auto-increment, but it accepts no `WHERE` and does not invoke per-row triggers.

## How to answer in an interview

"I use `DELETE` when I need to delete selectively - with `WHERE` - or when delete triggers must run. It is row-by-row and logged, so it can be rolled back within a transaction, but it is slow on large tables. I use `TRUNCATE` when I need to quickly empty a whole table: it does not delete rows one by one but frees the data pages, also resetting the auto-increment counter. It does not support conditions and does not call per-row triggers. Roughly: targeted and conditional cleanup is `DELETE`, a full table reset is `TRUNCATE`."

## How it works

`DELETE` belongs to DML and removes one or more rows selected via `WHERE`:

```sql
DELETE FROM employees WHERE department = 'Sales';
```

It records every deleted row in the transaction log so the operation can be rolled back, and along the way any attached triggers fire. On large volumes this is noticeably more expensive.

`TRUNCATE` belongs to DDL and clears the whole table:

```sql
TRUNCATE TABLE employees;
```

It does not iterate over rows but frees the data pages and resets the auto-increment counters - so it works faster but loses flexibility: neither `WHERE` nor per-row triggers.

When to choose which:

- `DELETE` - selective deletion by condition and/or when triggers are needed.
- `TRUNCATE` - a fast full reset of the table without per-row logging.

## Pitfalls

- The behavior depends on the DBMS. In PostgreSQL `TRUNCATE` is transactional and can be rolled back, while in MySQL (InnoDB) it implicitly commits the transaction and cannot be rolled back.
- `TRUNCATE` usually requires DDL-level privileges and may conflict with foreign keys: a table referenced by FKs often cannot be truncated without `CASCADE`.
- The auto-increment reset of `TRUNCATE` is not always what you want: new rows will start numbering from scratch, which may conflict with old external references.
- `DELETE` without `WHERE` also clears the table, but slowly and with the log ballooning; for a full cleanup this is almost always worse than `TRUNCATE`.
