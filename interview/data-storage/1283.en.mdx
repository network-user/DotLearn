# Why does a database need an index and at what cost does it speed up reads?

## In short

An index is an auxiliary structure on top of a table that gives the database a shortcut to rows without scanning all the data. Reads speed up noticeably, but you pay for it with disk space and slower writes.

## How to answer in an interview

An index is like the subject index in a book: instead of reading cover to cover, the database jumps straight to the needed pages. Technically it's most often a B-tree over one or more columns, less often a hash structure. It dramatically speeds up a `SELECT` with a filter on the indexed field, but every insert, update, and delete now also has to update the index, plus it takes up additional space. That's why indexes are placed selectively on columns that are actually searched often.

## How it works

Without an index, to fetch the needed rows the system walks the table row by row - on large volumes this is expensive. An index stores the column values in a structure where search is fast: in a B-tree, in logarithmic time.

Take an `employees` table with columns `id`, `name`, `department`. If searching by name is frequent, we index `name`:

```sql
CREATE INDEX idx_name ON employees(name);
```

Now the query `SELECT * FROM employees WHERE name = 'Alice'` goes through the index and finds the rows directly instead of scanning the whole table.

Under the hood there's usually a B-tree: it keeps values ordered, so it works for equality, ranges (`>`, `<`, `BETWEEN`), and sorting. A hash index is an alternative only for exact equality.

## Pitfalls

- An index isn't free on writes: the more of them there are, the slower inserts and updates become.
- On small tables an index may be ignored - it's cheaper for the planner to read everything.
- Low selectivity kills the benefit: an index on a column with two values (for example, `is_active`) is almost useless.
- An index works only when the column is used without wrappers - a function or type cast in the condition disables it.
