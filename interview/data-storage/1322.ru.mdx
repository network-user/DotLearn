# Какой набор SQL-команд даёт контроль над транзакциями?

## Суть

Транзакциями в SQL управляют пять команд: `BEGIN` открывает блок операций, `COMMIT` фиксирует их навсегда, `ROLLBACK` откатывает всё назад, `SAVEPOINT` ставит промежуточную метку для частичного отката, а `SET TRANSACTION` подкручивает параметры вроде уровня изоляции.

## Что ответить на собесе

Всё это вместе обеспечивает атомарность: либо проходят все операции, либо ни одной.

- `BEGIN` (или `START TRANSACTION`) стартует транзакцию, `COMMIT` закрепляет результат и делает его видимым другим сессиям, `ROLLBACK` отменяет все накопленные изменения.
- `SAVEPOINT` с `ROLLBACK TO SAVEPOINT` позволяют откатить только часть работы внутри транзакции, не сбрасывая её целиком.
- `SET TRANSACTION` задаёт уровень изоляции и режим доступа.

## Как это работает

Транзакция - это группа операций, которая выполняется как единое неделимое целое. Это особенно важно в системах с конкурентным доступом, где одни и те же строки правят несколько клиентов одновременно.

Стартуем транзакцию и помечаем границу:

```sql
BEGIN;
```

Фиксируем накопленные изменения - после этого откат уже невозможен:

```sql
COMMIT;
```

Откатываем всё к состоянию до начала транзакции:

```sql
ROLLBACK;
```

Ставим промежуточную метку и при необходимости возвращаемся к ней, не теряя транзакцию целиком:

```sql
SAVEPOINT before_transfer;
ROLLBACK TO SAVEPOINT before_transfer;
```

Настраиваем уровень изоляции для текущей транзакции:

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

Канонический пример - перевод денег между счетами. Списание и зачисление должны пройти вместе или не пройти вовсе:

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 2;
COMMIT;
```

Если на счёте не хватает средств, всю транзакцию отменяем:

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;
SAVEPOINT after_debit;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 2;
-- проверка прошла неуспешно
ROLLBACK TO SAVEPOINT after_debit;
COMMIT;
```

## Подводные камни

- `COMMIT` необратим: после фиксации `ROLLBACK` уже не вернёт старые данные, нужен компенсирующий UPDATE.
- Многие СУБД работают в режиме autocommit - каждый отдельный оператор фиксируется сразу, и без явного `BEGIN` транзакция из нескольких шагов просто не соберётся.
- Ошибка внутри транзакции в PostgreSQL переводит её в "aborted"-состояние: до `ROLLBACK` (или отката до SAVEPOINT) любые следующие команды будут падать.
- `ROLLBACK TO SAVEPOINT` не завершает транзакцию - после него всё равно нужен `COMMIT` или полный `ROLLBACK`.
- Слишком высокий уровень изоляции (`SERIALIZABLE`) повышает риск конфликтов сериализации, и приложение должно уметь повторять упавшую транзакцию.
