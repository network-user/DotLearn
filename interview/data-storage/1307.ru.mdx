# Как построить SELECT, чтобы он не обращался к закрытым полям?

## Суть

Перечисляйте в запросе нужные колонки явно и забудьте про `SELECT *`. Чтобы это было не просто дисциплиной, а гарантией, спрячьте закрытые поля за представлением или функцией, которые физически отдают только разрешённые столбцы.

## Что ответить на собесе

Дисциплина в запросах плюс ограничение на стороне базы.

- Первое правило — не писать `SELECT *`, а перечислять конкретные колонки: запрос не цепляет поля без доступа и тянет меньше данных.
- Полагаться только на дисциплину разработчика рискованно, поэтому доступ закрываю на стороне базы.
- Создаю view с лишь публичными полями и даю права на представление, а не на исходную таблицу.
- Для сложных правил выношу логику в функцию, возвращающую таблицу с разрешённым набором колонок. Тогда любой `SELECT *` по view или функции вернёт только положенное.

## Как это работает

Явный список колонок - база. Запрос обращается ровно к тем полям, которые названы, и не задевает закрытые.

```sql
SELECT name, email FROM employees;
```

Представление фиксирует разрешённый набор полей один раз. Пользователь обращается к view, а не к таблице, и через него закрытые колонки недоступны в принципе.

```sql
CREATE VIEW public_employees AS
SELECT name, email
FROM employees;

SELECT * FROM public_employees;
```

`SELECT *` по представлению безопасен: звёздочка раскрывается в колонки самого view, то есть только в `name` и `email`. Для более гибких правил доступ оборачивают в функцию, возвращающую таблицу.

```sql
CREATE FUNCTION get_public_employee_data()
RETURNS TABLE(name VARCHAR, email VARCHAR) AS $$
BEGIN
    RETURN QUERY
    SELECT name, email FROM employees;
END;
$$ LANGUAGE plpgsql;

SELECT * FROM get_public_employee_data();
```

## Подводные камни

- `SELECT *` опасен не только по доступу: при добавлении колонки в таблицу выборка молча начнёт возвращать новые поля и может сломать клиентский код, рассчитывающий на фиксированный набор.
- View и функция защищают, только если у пользователя нет прямого доступа к базовой таблице. Без `REVOKE` на `employees` ограничение обходится одним запросом к таблице напрямую.
- Функция с `SECURITY DEFINER` выполняется с правами своего владельца - это удобно для контроля доступа, но опасно: ошибки в такой функции дают пользователю чужие привилегии.
- Скрытие поля из выборки не равно шифрованию. Данные по-прежнему лежат в таблице в открытом виде, и тот, у кого есть доступ к ней, их прочитает.
