# QuerySet в Django: чем он является и какие операции на нём держатся?

## Суть

QuerySet представляет собой отложенное описание выборки из базы, а не сами данные. Он собирает SQL по мере применения фильтров и обращается к базе только тогда, когда результат реально понадобился. На нём держатся выборка, фильтрация, сортировка и массовые изменения записей.

## Что ответить на собесе

QuerySet: объект-конструктор запроса, а не сами данные.

- Каждый вызов `filter()`, `exclude()`, `order_by()` возвращает новый QuerySet, не трогая исходный и не ходя в базу; именно в этом заключается ленивость.
- SQL уходит в базу только при итерации, срезе с шагом, `len()`, `list()`, `bool()` или явных `get()`/`first()`.
- Базовый набор: `all()` (всё), `filter()`/`exclude()` (отбор), `get()` (ровно один объект, иначе исключение), `order_by()` (сортировка), `create()` (создать и сохранить), `update()`/`delete()` (массово изменить или удалить без загрузки объектов в Python).
- Результат кешируется внутри QuerySet, так что повторная итерация по тому же объекту базу не дёргает.

## Как это работает

Каждый метод-фильтр строит новый запрос поверх предыдущего, поэтому цепочки читаются слева направо:

```python
published = Entry.objects.filter(status='published')
recent = published.exclude(archived=True).order_by('-published_date')
```

База ещё не тронута: `recent` пока лишь рецепт запроса. Чтение чего угодно из результата запускает SQL:

```python
for entry in recent:
    print(entry.title)
```

Точечная выборка и обработка отсутствия объекта:

```python
try:
    entry = Entry.objects.get(id=1)
except Entry.DoesNotExist:
    entry = None
```

Создание, массовое обновление и удаление без подъёма объектов в память:

```python
Entry.objects.create(title='New', content='...')
Entry.objects.filter(status='draft').update(status='published')
Entry.objects.filter(status='draft').delete()
```

`update()` и `delete()` на QuerySet работают одним SQL-запросом на уровне базы, минуя цикл по объектам в Python, поэтому на больших выборках они заметно быстрее ручного перебора.

## Подводные камни

- **`get()` бросает исключения парами:** `DoesNotExist`, если ничего не нашлось, и `MultipleObjectsReturned`, если совпадений больше одного. Для «может быть, а может нет» берут `filter().first()`.
- **Ленивость двусторонняя:** QuerySet, созданный один раз и переиспользуемый в шаблоне несколько раз, кеширует результат, но новый QuerySet на каждой итерации цикла породит новый запрос, отсюда и проблема N+1.
- **`update()` не вызывает `save()` и сигналы:** хуки модели и `pre_save`/`post_save` при массовом `update()` не срабатывают.
- **Срез с шагом материализует выборку:** `qs[::2]` выполняет запрос, а `qs[:10]` транслируется в `LIMIT` и остаётся ленивым.
- **Изменение QuerySet «на месте» невозможно:** методы возвращают новый объект, присваивание результата обязательно, иначе фильтр потеряется.
