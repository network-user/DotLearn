# Многоколоночный индекс: почему порядок столбцов решает всё?

## Суть

Составной индекс строится сразу по нескольким столбцам и ускоряет запросы, фильтрующие по ним вместе. Главный нюанс - порядок столбцов в индексе: он работает по принципу левого префикса, и от этого зависит, поможет ли индекс конкретному запросу.

## Что ответить на собесе

Составной индекс: упорядоченная структура по нескольким полям, как телефонная книга, отсортированная сначала по фамилии, потом по имени.

- Включается, когда запрос использует ведущие столбцы индекса слева направо (правило левого префикса).
- Индекс `(customer_id, order_date)` отлично отработает фильтр по `customer_id` и по паре `customer_id + order_date`, но почти не поможет при фильтре только по `order_date`.
- Поэтому порядок столбцов подбирают под самые частые запросы, ставя первым самый селективный и наиболее используемый в `WHERE`.

## Как это работает

Пусть есть таблица `orders` со столбцами `customer_id`, `order_date`, `total_amount`, и мы часто фильтруем по паре `customer_id` + `order_date`:

```sql
CREATE INDEX idx_customer_order_date ON orders (customer_id, order_date);
```

- фильтра по `customer_id`;
- фильтра по `customer_id` И `order_date`;
- `ORDER BY customer_id, order_date`.

И почти бесполезен для запроса, где в условии только `order_date`: ведущий столбец `customer_id` пропущен, левый префикс не сходится.

```sql
SELECT total_amount
FROM orders
WHERE customer_id = 42 AND order_date >= '2026-01-01';
```

Такой запрос ложится на индекс идеально.

## Подводные камни

- **Левый префикс**: пропустили ведущий столбец - индекс не задействуется. Порядок не переставляется автоматически.
- **Селективность**: первым обычно ставят самый избирательный и самый частый в фильтрах столбец.
- **Размер и запись**: чем больше столбцов в индексе, тем он толще и тем сильнее замедляет вставку, обновление и удаление.
- **Диапазон в середине**: после столбца с диапазонным условием (`>`, `<`) следующие столбцы индекса для дальнейшего сужения уже не используются.
