# Where does the N+1 query problem come from and how to fix it?

## In short

N+1 is when the main list is fetched with one query, and then for each of its elements another query flies off to fetch related data. As a result, instead of a single round trip to the database you get N+1 of them. It is cured by eager loading via JOIN or with ORM features.

## How to answer in an interview

"The N+1 problem arises from lazy loading of relationships in the ORM. I fetch the list of authors with one query, and then in a loop I access `author.books` - and for each author the ORM quietly makes a separate SQL query. Ten authors turn into eleven queries, a hundred into a hundred and one. I solve this with eager loading: with a single JOIN or a dedicated ORM method (`joinedload`/`selectinload` in SQLAlchemy, `select_related`/`prefetch_related` in Django) I pull the related rows in advance. The main thing is to be able to spot N+1 in the query log rather than by eye."

Suppose we have authors and their books. The naive path:

1. One query - all authors.
2. One query per author for that author's books.

With 10 authors this is `1 + 10 = 11` queries. Each query is a network round trip, parsing, and execution, so the time grows linearly and the database chokes under load.

Eager loading reduces everything to one query via JOIN:

```sql
SELECT authors.id, authors.name, books.title
FROM authors
LEFT JOIN books ON authors.id = books.author_id;
```

In an ORM this is expressed explicitly. Example in SQLAlchemy:

```python
from sqlalchemy.orm import joinedload

authors = (
    session.query(Author)
    .options(joinedload(Author.books))
    .all()
)
```

`joinedload(Author.books)` tells the ORM to pull books with the same query, and accessing `author.books` in a loop no longer triggers new trips to the database.

## Pitfalls

- Not every "single query" is better. `joinedload` over a collection (one-to-many) multiplies the parent rows in the result; sometimes `selectinload` (a separate batched IN query) is more efficient.
- N+1 is easily hidden behind properties and serializers: a DRF serializer that touches a nested relationship creates the same problem invisibly.
- Profile against the real SQL log (`django-debug-toolbar`, `echo=True` in SQLAlchemy) rather than guessing.
- Overdoing eager loading is also bad: pulling data with a JOIN that you do not need is wasted traffic and memory.
