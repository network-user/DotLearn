# When NoSQL beats relational databases, and when it loses

## In short

NoSQL is chosen for a flexible schema, horizontal scaling and speed on huge volumes. The price for this is often weakened transactions and poor tooling for complex queries, so for systems with strict consistency a relational DBMS is usually more appropriate.

## How to answer in an interview

NoSQL is not "a database better than SQL" but a different set of trade-offs. I choose NoSQL when the data is loosely structured or its shape changes often, when I need linear scalability across many nodes, and when the workload is simple key-based reads and writes over terabytes of data. I stay on a relational DBMS when ACID transactions, complex joins and analytics are critical, or when the data model is stable and maps well onto tables. So it's not fashion that decides, but the workload profile and consistency requirements.

## How it works

NoSQL is a family of stores with different data models, not a single technology. The main families:

- **document** (MongoDB) - records as JSON/BSON documents;
- **key-value** (Redis) - a simple dictionary by key;
- **columnar** (Cassandra) - data by column families, tuned for writes;
- **graph** (Neo4j) - nodes and edges, convenient for traversing relationships.

The strengths follow from how these models are built:

1. **Flexible schema.** A document can be extended with a new field without migrating the whole table - the schema lives in the data itself, not in the DDL.
2. **Sharding out of the box.** Data is split across nodes from the start, so growth in capacity and throughput is achieved by adding servers rather than vertically upgrading a single one.
3. **Speed by key.** Dropping joins and denormalizing data make pinpoint reads/writes very fast.

The weaknesses are the flip side of the same decisions:

1. **Weakened transactions.** Some systems work in an eventual-consistency model, where replicas diverge for a while.
2. **Poor complex queries.** Joins and aggregations are either absent or limited, because the data is denormalized.
3. **A zoo of APIs.** There's no single SQL; each system has its own language, and migrating between them is expensive.

An example of working with the MongoDB document database through the `pymongo` driver:

```python
from pymongo import MongoClient

client = MongoClient("localhost", 27017)
db = client["example_database"]
collection = db["example_collection"]

new_user = {
    "name": "John Doe",
    "age": 30,
    "email": "johndoe@example.com",
}
collection.insert_one(new_user)

found = collection.find_one({"name": "John Doe"})
print(found)
```

Here `db["..."]` and `db["..."]["..."]` lazily access the database and collection, `insert_one` stores a plain Python dictionary as a document, and `find_one` searches by a filter dictionary. No predeclared schema - the document itself defines the structure.

## Pitfalls

- **"NoSQL = no transactions" is an outdated myth.** Modern MongoDB supports multi-document ACID transactions, and Redis supports atomicity through scripts/MULTI. Asserting flatly "there are no transactions" is risky; talk about the consistency model of the specific system.
- **A flexible schema doesn't mean "no schema".** The schema simply moves into the application code; format mismatches between documents surface at runtime, not at insert time.
- **Denormalization duplicates data.** Updating a duplicated fact across a dozen documents is your concern, not the database's; it's easy to end up out of sync.
- **Eventual consistency breaks intuition.** Right after a write, a read replica may return the old value. For counters, balances and "read what you wrote" this is a source of subtle bugs.
