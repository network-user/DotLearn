# Alembic: зачем он нужен для управления версиями схемы базы данных

## Суть

Alembic — инструмент версионирования схемы БД для проектов на SQLAlchemy. Он описывает каждое изменение структуры как отдельный шаг (ревизию), который можно накатить и откатить, и держит схему синхронной во всех окружениях.

## Что ответить на собесе

Alembic решает задачу управляемой эволюции схемы.

- Любая правка структуры — новая таблица, переименованный столбец, удалённый индекс — оформляется отдельной ревизией с функциями `upgrade` и `downgrade`.
- Ревизии складываются в линейную (или ветвящуюся) историю и применяются по порядку.
- За счёт этого dev, staging и продакшен приходят к одинаковой структуре одним набором команд, без ручного переноса SQL и без рассинхрона.
- С `--autogenerate` Alembic сам сравнивает модели SQLAlchemy с текущей схемой и набрасывает черновик миграции.

Базовая единица — ревизия. Это Python-файл с уникальным идентификатором, ссылкой на предыдущую ревизию (`down_revision`) и двумя функциями: что сделать при накате и что при откате.

Стартовый набор команд:

```bash
alembic init alembic
```

Создаёт каталог `alembic/` с конфигом и шаблонами миграций.

```bash
alembic revision --autogenerate -m "add users table"
```

Сравнивает модели с базой и генерирует черновик ревизии. Флаг `-m` задаёт человекочитаемое описание.

```bash
alembic upgrade head
```

Применяет все ещё не накатанные ревизии до самой свежей (`head`).

```bash
alembic downgrade -1
```

Откатывает один последний шаг.

Сквозной пример — добавляем таблицу `users`. Сначала описываем модель:

```python
from sqlalchemy import Column, Integer, String
from sqlalchemy.orm import declarative_base

Base = declarative_base()

class User(Base):
    __tablename__ = "users"
    id = Column(Integer, primary_key=True)
    name = Column(String, nullable=False)
```

Затем генерируем ревизию командой `alembic revision --autogenerate -m "add users table"` и накатываем её через `alembic upgrade head`. В таблице `alembic_version` записывается идентификатор текущей ревизии — так Alembic знает, на каком шаге сейчас находится база.

## Подводные камни

- `--autogenerate` видит не всё: смену типа столбца, переименования, изменения в `CHECK` и серверных `default` он часто пропускает или трактует как drop+create. Сгенерированную ревизию обязательно нужно вычитывать руками.
- Функцию `downgrade` легко оставить пустой или некорректной — тогда откат на проде сломается именно тогда, когда он нужен.
- Несколько веток разработки порождают несколько `head`, и Alembic потребует `alembic merge`. Игнорировать это нельзя.
- Миграции данных (не структуры) надо писать аккуратно: тяжёлый `UPDATE` внутри ревизии может надолго заблокировать таблицу на проде.
- Ревизия фиксирует структуру на момент написания: импортировать «живые» модели приложения внутрь миграции — путь к поломке при будущих изменениях кода.
