# Запрос тормозит: каким маршрутом искать причину и ускорять?

## Суть

Оптимизация медленного SQL - это не один приём, а последовательность шагов: сначала смотрим план выполнения, затем чиним индексы, упрощаем сам запрос, при необходимости трогаем схему и обновляем статистику. Железо - последняя инстанция, а не первая.

## Что ответить на собесе

Иду по шагам от плана к железу, а не наоборот.

- Начинаю с `EXPLAIN` (или `EXPLAIN ANALYZE`): план показывает, где база делает full scan, где теряет время на сортировке или соединении.
- Проверяю индексы по столбцам из `WHERE` и `JOIN`.
- Переписываю запрос так, чтобы он читал меньше данных: нужные столбцы вместо `SELECT *`, соединения вместо тяжёлых подзапросов.
- Если мало, смотрю на схему и актуальность статистики планировщика. Наращивание ресурсов сервера — крайняя мера, когда логика уже выжата досуха.

Первым делом просим базу показать, как она собирается выполнять запрос:

```sql
EXPLAIN SELECT name FROM users WHERE email = 'example@example.com';
```

В плане видно узкие места: последовательное сканирование таблицы там, где ожидался индексный доступ, дорогие соединения, лишние сортировки.

Если фильтр часто идёт по столбцу, его стоит проиндексировать - тогда база найдёт строки напрямую, а не переберёт всю таблицу:

```sql
CREATE INDEX idx_email ON users(email);
```

Запрос почти всегда можно сделать дешевле. Заменяем подзапрос соединением и просим только нужные столбцы:

```sql
SELECT users.name, orders.amount
FROM users
JOIN orders ON users.id = orders.user_id;
```

Иногда тормозит сама структура. Нормализация убирает дублирование, но плодит соединения; денормализация добавляет избыточность ради быстрого чтения - выбор зависит от профиля нагрузки.

Планировщик опирается на статистику распределения данных. Если она устарела, он строит плохой план - поэтому её обновляют:

```sql
ANALYZE users;
```

## Подводные камни

- Лишние индексы не бесплатны: они занимают место и замедляют каждую вставку и обновление, потому что их тоже надо поддерживать.
- Индекс по столбцу не используется, если на него навешена функция или приведение типа в `WHERE` (`WHERE lower(email) = ...` ломает обычный индекс по `email`).
- `SELECT *` мешает покрывающим индексам и тянет лишние данные по сети.
- Сначала увеличить сервер, не посмотрев план, - частая ошибка: вы платите деньгами за проблему, которую решал бы один индекс.
