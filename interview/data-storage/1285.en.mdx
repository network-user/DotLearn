# Multi-column index: why does column order decide everything?

## In short

A composite index is built over several columns at once and speeds up queries that filter on them together. The main subtlety is the order of columns in the index: it works on the leftmost-prefix principle, and that determines whether the index helps a particular query.

## How to answer in an interview

A composite index is an ordered structure over several fields, like a phone book sorted first by last name, then by first name. It kicks in when a query uses the index's leading columns left to right: an index `(customer_id, order_date)` works great for a filter on `customer_id` and on the pair `customer_id + order_date`, but is almost no help if you filter only by `order_date`. That's why you choose column order to match the most frequent queries, putting the most selective and most-used-in-`WHERE` column first.

## How it works

Suppose there's an `orders` table with columns `customer_id`, `order_date`, `total_amount`, and we often filter on the pair `customer_id` + `order_date`:

```sql
CREATE INDEX idx_customer_order_date ON orders (customer_id, order_date);
```

The index stores rows ordered first by `customer_id`, and within each `customer_id` by `order_date`. So it works for:

- a filter on `customer_id`;
- a filter on `customer_id` AND `order_date`;
- `ORDER BY customer_id, order_date`.

And it's almost useless for a query whose condition has only `order_date`: the leading column `customer_id` is skipped, the leftmost prefix doesn't match.

```sql
SELECT total_amount
FROM orders
WHERE customer_id = 42 AND order_date >= '2026-01-01';
```

Such a query fits the index perfectly.

## Pitfalls

- **Leftmost prefix**: skip the leading column and the index isn't used. The order isn't rearranged automatically.
- **Selectivity**: the most selective and most filter-frequent column usually goes first.
- **Size and writes**: the more columns in the index, the fatter it is and the more it slows down inserts, updates, and deletes.
- **A range in the middle**: after a column with a range condition (`>`, `<`), the index's subsequent columns are no longer used for further narrowing.
