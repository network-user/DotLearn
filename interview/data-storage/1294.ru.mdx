# Когда NoSQL выигрывает у реляционных БД, а когда проигрывает

## Суть

NoSQL берут ради гибкой схемы, горизонтального масштабирования и скорости на огромных объёмах. Платой за это часто становятся ослабленные транзакции и бедные средства сложных запросов, поэтому для систем со строгой согласованностью реляционные СУБД обычно уместнее.

## Что ответить на собесе

NoSQL — это не «база лучше SQL», а другой набор компромиссов; решает не мода, а профиль нагрузки и требования к согласованности.

- **Беру NoSQL**, когда данные слабо структурированы или их форма часто меняется, нужна линейная масштабируемость на много узлов, а нагрузка — простые чтения и записи по ключу на терабайтах данных.
- **Остаюсь на реляционной СУБД**, когда критичны ACID-транзакции, сложные join'ы и аналитика либо когда модель данных стабильна и хорошо ложится на таблицы.

## Как это работает

NoSQL — это семейство хранилищ с разными моделями данных, а не одна технология. Основные семьи:

- **документные** (MongoDB) — записи как JSON/BSON-документы;
- **ключ-значение** (Redis) — простой словарь по ключу;
- **колоночные** (Cassandra) — данные по семействам колонок, заточены под запись;
- **графовые** (Neo4j) — узлы и связи, удобно для обхода отношений.

Сильные стороны вытекают из устройства этих моделей:

1. **Гибкая схема.** Документ можно расширить новым полем без миграции всей таблицы — схема живёт в самих данных, а не в DDL.
2. **Шардирование из коробки.** Данные изначально разбиваются по узлам, поэтому рост ёмкости и пропускной способности достигается добавлением серверов, а не вертикальным апгрейдом одного.
3. **Скорость по ключу.** Отказ от join'ов и денормализация делают точечные чтения/записи очень быстрыми.

Слабые стороны — обратная сторона тех же решений:

1. **Ослабленные транзакции.** Часть систем работает в модели eventual consistency, где реплики какое-то время расходятся.
2. **Бедные сложные запросы.** Join'ы и агрегации либо отсутствуют, либо ограничены, потому что данные денормализованы.
3. **Зоопарк API.** Единого SQL нет, у каждой системы свой язык, и миграция между ними дорогая.

Пример работы с документной базой MongoDB через драйвер `pymongo`:

```python
from pymongo import MongoClient

client = MongoClient("localhost", 27017)
db = client["example_database"]
collection = db["example_collection"]

new_user = {
    "name": "John Doe",
    "age": 30,
    "email": "johndoe@example.com",
}
collection.insert_one(new_user)

found = collection.find_one({"name": "John Doe"})
print(found)
```

Здесь `db["..."]` и `db["..."]["..."]` лениво обращаются к базе и коллекции, `insert_one` кладёт обычный словарь Python как документ, а `find_one` ищет по фильтру-словарю. Никакой заранее объявленной схемы — структуру задаёт сам документ.

## Подводные камни

- **«NoSQL = нет транзакций» — устаревший миф.** Современная MongoDB поддерживает многодокументные ACID-транзакции, а Redis — атомарность через скрипты/MULTI. Утверждать в лоб «транзакций нет» рискованно, говорите про модель согласованности конкретной системы.
- **Гибкая схема не значит «без схемы».** Схема просто переезжает в код приложения; рассогласование форматов документов всплывает в рантайме, а не на этапе вставки.
- **Денормализация дублирует данные.** Обновление продублированного факта в десятке документов — это ваша забота, а не базы; легко получить рассинхрон.
- **Eventual consistency ломает интуицию.** Сразу после записи реплика-читалка может вернуть старое значение. Для счётчиков, балансов и «прочитал то, что записал» это источник тонких багов.
