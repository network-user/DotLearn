# Блокировки в базе данных: что это и какую роль они играют?

## Суть

Блокировка - это механизм управления конкурентным доступом, который запрещает нескольким транзакциям одновременно менять одни и те же данные. Так СУБД сохраняет целостность и защищает от аномалий: грязного чтения, потерянных обновлений и фантомов.

## Что ответить на собесе

Объясняю через проблему: при параллельной работе без координации возникают аномалии — грязное чтение, неповторяющееся чтение, потерянное обновление, фантомы. Блокировки координируют доступ.

- **Разделяемая (shared, S)** — под чтение: её можно держать нескольким транзакциям сразу, но писать никто не может.
- **Эксклюзивная (exclusive, X)** — под запись: её держит только одна транзакция, остальные ждут.
- Блокировки бывают на уровне строки, страницы, таблицы или всей базы: чем мельче гранулярность, тем выше параллелизм, но больше накладных расходов.
- Уровень изоляции определяет, какие блокировки и когда берутся. Обратная сторона — дедлоки, когда две транзакции ждут блокировки друг друга.

Аномалии, от которых защищают блокировки:

- **Грязное чтение (Dirty Read):** транзакция видит ещё не зафиксированные чужие изменения; если их откатят, данные окажутся неверными.
- **Потерянное обновление (Lost Update):** две транзакции правят одну запись, и одна правка затирается.
- **Неповторяющееся чтение (Non-repeatable Read):** повторное чтение той же строки даёт другой результат, потому что её успели изменить.
- **Фантомное чтение (Phantom Read):** повторный запрос по условию возвращает новые строки, которых не было раньше.

Основные типы блокировок:

- **Эксклюзивная (X):** только одна транзакция может менять данные; остальные ждут снятия.
- **Разделяемая (S):** несколько транзакций читают одновременно, но запись заблокирована.

Пример с двумя транзакциями над одной записью:

```sql
-- Транзакция 1
BEGIN TRANSACTION;
SELECT * FROM accounts WHERE account_id = 1;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 1;
COMMIT;

-- Транзакция 2
BEGIN TRANSACTION;
SELECT * FROM accounts WHERE account_id = 1;
UPDATE accounts SET balance = balance - 50 WHERE account_id = 1;
COMMIT;
```

На `UPDATE` берётся эксклюзивная блокировка строки, и вторая транзакция ждёт `COMMIT` первой, прежде чем применить своё изменение. `COMMIT` снимает блокировки и возвращает данные остальным.

## Подводные камни

- **Дедлок:** две транзакции захватывают ресурсы в разном порядке и ждут друг друга. СУБД выявляет цикл и принудительно откатывает одну из них - код должен уметь повторить транзакцию.
- Слишком грубая гранулярность (блокировка таблицы вместо строки) убивает параллелизм; слишком мелкая повышает накладные расходы на учёт блокировок.
- В PostgreSQL читатели обычно не блокируют писателей за счёт MVCC, поэтому модель блокировок там не совпадает с классическим «S/X на каждое чтение».
- Долгая транзакция, удерживающая блокировку, выстраивает очередь и роняет пропускную способность - держите транзакции короткими.
- Явный `SELECT ... FOR UPDATE` берёт блокировку строки заранее и помогает против потерянных обновлений.
