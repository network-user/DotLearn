# Откуда берётся проблема N+1 запросов и как её гасить?

## Суть

N+1 - это когда основной список достаётся одним запросом, а затем на каждый его элемент летит ещё по запросу за связанными данными. В итоге вместо одного обращения к БД их получается N+1. Лечится жадной загрузкой через JOIN или средствами ORM.

## Что ответить на собесе

Проблема N+1 возникает из-за ленивой загрузки связей в ORM.

- Беру список авторов одним запросом, а потом в цикле обращаюсь к `author.books`: на каждого автора ORM незаметно делает отдельный SQL-запрос.
- Десять авторов превращаются в одиннадцать запросов, а сто - в сто один.
- Решаю жадной загрузкой: одним JOIN-ом или методом ORM (`joinedload`/`selectinload` в SQLAlchemy, `select_related`/`prefetch_related` в Django) подтягиваю связанные строки заранее.
- Главное: уметь увидеть N+1 в логе запросов, а не на глаз.

Пусть есть авторы и их книги. Наивный путь:

1. Один запрос - все авторы.
2. По одному запросу на книги каждого автора.

При 10 авторах это `1 + 10 = 11` запросов. Каждый запрос - это сетевой round-trip, парсинг и выполнение, поэтому время растёт линейно, а БД захлёбывается под нагрузкой.

Жадная загрузка сводит всё к одному запросу через JOIN:

```sql
SELECT authors.id, authors.name, books.title
FROM authors
LEFT JOIN books ON authors.id = books.author_id;
```

В ORM это выражается явно. Пример на SQLAlchemy:

```python
from sqlalchemy.orm import joinedload

authors = (
    session.query(Author)
    .options(joinedload(Author.books))
    .all()
)
```

`joinedload(Author.books)` говорит ORM подтянуть книги тем же запросом, и обращение к `author.books` в цикле больше не порождает новых походов в базу.

## Подводные камни

- Не всякий «один запрос» лучше. `joinedload` по коллекции (один-ко-многим) размножает строки родителя в результате; иногда `selectinload` (отдельный пакетный IN-запрос) эффективнее.
- N+1 легко спрятан за свойствами и сериализаторами: DRF-сериализатор, дёргающий вложенную связь, создаёт ту же проблему незаметно.
- Профилируйте по реальному логу SQL (`django-debug-toolbar`, `echo=True` в SQLAlchemy), а не угадывайте.
- Перегнуть с жадной загрузкой тоже плохо: тянуть JOIN-ом данные, которые не нужны, - лишний трафик и память.
