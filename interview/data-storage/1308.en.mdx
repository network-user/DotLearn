# What problems come up with replication in PostgreSQL?

## In short

Replication provides fault tolerance and read scaling, but in return it brings replica lag, the risk of write conflicts, the struggle for consistency, mandatory monitoring, and extra hardware usage.

## How to answer in an interview

The main problem is replica lag: by default replication is asynchronous, and for some time the replica serves stale data. For "read right after write" scenarios this breaks the logic, and you have to either read from the master or enable synchronous replication - which slows commits down, because the master waits for confirmation. Next: you can't write to a replica, or it'll desync, so it must be strictly read-only. On top of that, replication requires monitoring of lag and stream state, plus extra CPU, memory, and disk per replica. Roughly: you're choosing between data freshness, write speed, and cost.

Replication copies changes from the master to the replicas through the WAL stream (the write-ahead log). The basic setup on the master sets the WAL level, enables archiving and a limit on streams to replicas, after which a base copy is taken for the replica.

```sql
-- On the master:
ALTER SYSTEM SET wal_level = 'replica';
ALTER SYSTEM SET archive_mode = 'on';
ALTER SYSTEM SET archive_command = 'cp %p /path_to_archive/%f';
ALTER SYSTEM SET max_wal_senders = 3;

SELECT pg_reload_conf();
```

```bash
# Take a base copy for the replica (psql has nothing to do with this - it's a command-line utility)
pg_basebackup -h master_host -D /path_to_replica_data -U replication_user -Fp -Xs -P
```

In modern versions (12 and up) the replica is put into standby with a flag file `standby.signal` and the `primary_conninfo` parameter in `postgresql.conf`. The old `recovery.conf` from older guides was removed long ago.

```ini
# postgresql.conf on the replica
primary_conninfo = 'host=master_host port=5432 user=replication_user password=secret'
```

After startup the replica connects to the master and replays WAL, trailing by the lag amount.

## Pitfalls

- An asynchronous replica may by definition serve stale data - the read-your-own-writes pattern requires reading from the master or using synchronous replication for critical transactions.
- Long-running queries on the replica conflict with applying WAL: either the query is canceled (`max_standby_streaming_delay`) or applying the log is stalled. This is the classic recovery conflict.
- An unclosed replication slot is dangerous: if a replica drops off, the master accumulates WAL for its slot and can fill the disk to the brim.
- `recovery.conf` is dead as of PostgreSQL 12 - guides that mention it are outdated; you need `standby.signal` and parameters in the main config.
- Standard physical replication is all-or-nothing: the entire database. If you need selective replication (individual tables), that's logical replication with its own limitations.
