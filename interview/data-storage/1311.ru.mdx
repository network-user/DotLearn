# Где держать кэш запросов: Redis или таблица в PostgreSQL

## Суть

Redis выбирают ради скорости и низкой задержки: данные в памяти, доступ за микросекунды, ключам легко навесить TTL. PostgreSQL как «кэш» имеет смысл, когда нужны переживание перезапуска, транзакционная согласованность с основными данными или сложные выборки поверх закэшированного.

## Что ответить на собесе

Это не вопрос «что лучше», а вопрос требований. Если кэш горячий, эфемерный и должен отдаваться мгновенно - моё решение Redis: in-memory, key-value, встроенный TTL и вытеснение по памяти. Если же «кэш» обязан быть надёжным, читаться в одной транзакции с остальными данными или участвовать в аналитических запросах - я держу его в PostgreSQL, потому что Redis не даёт ACID-гарантий уровня реляционной БД. На практике частый паттерн - оба слоя: PostgreSQL как источник истины, Redis как быстрый кэш перед ним.

## Как это работает

**Redis** - in-memory key-value хранилище. Данные живут в RAM, поэтому чтение на порядки быстрее диска. Ключу задаётся срок жизни, и по истечении он удаляется сам.

```python
import redis

r = redis.Redis(host="localhost", port=6379, db=0)

r.set("user:1000", "John Doe", ex=3600)  # ex=3600 - живёт 1 час
user = r.get("user:1000")
print(user)  # b'John Doe'
```

**PostgreSQL** хранит кэш как обычные строки на диске - он переживает рестарт, поддерживает SQL и ACID:

```sql
CREATE TABLE cache (
    key        TEXT PRIMARY KEY,
    value      TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

INSERT INTO cache (key, value) VALUES ('user:1000', 'John Doe');

SELECT value FROM cache WHERE key = 'user:1000';
```

Сводка выбора:

- **Redis** - горячие, часто читаемые, временные данные; нужна минимальная задержка и автоудаление по TTL.
- **PostgreSQL** - кэш должен переживать перезапуск, быть согласованным с основными данными или участвовать в сложных запросах и отчётах.

## Подводные камни

- **Redis - не источник истины**. По умолчанию это память; при падении без настроенной персистентности (RDB/AOF) свежие данные теряются. Нельзя складывать в Redis то, что нельзя пересоздать.
- **Инвалидация - главная боль кэша**. TTL спасает от устаревания, но если данные меняются раньше срока, нужен явный сброс ключа при записи в источник, иначе кэш отдаёт устаревшее.
- **«Кэш» в PostgreSQL грузит ту же базу**, которую вы пытаетесь разгрузить. Если цель - снять нагрузку с БД, таблица-кэш в той же БД её не снимает.
- **TTL в PostgreSQL руками**. В таблице нет автоудаления по сроку - нужен фоновый процесс или условие по `created_at`, иначе кэш растёт неограниченно.
- **Память Redis конечна**. Без настроенной политики вытеснения (`maxmemory-policy`) запись в полную память приведёт к ошибкам, а не к молчаливому удалению старого.
