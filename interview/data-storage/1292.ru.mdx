# Добавление колонки в нагруженную таблицу без даунтайма

## Суть

Колонку добавляют через `ALTER TABLE ADD COLUMN`, но на горячей таблице главное — не вызвать долгую блокировку. Ключ к безопасности: добавлять колонку без значения по умолчанию (или полагаться на «мгновенный» DEFAULT современных СУБД) и не смешивать это с тяжёлым переписыванием данных.

## Что ответить на собесе

Сама команда тривиальна — `ALTER TABLE orders ADD COLUMN delivery_date DATE`. Опасность в блокировке и в переписывании таблицы. В старых движках `ADD COLUMN ... DEFAULT <значение>` мог переписать каждую строку под исключительной блокировкой и положить запись на таблицу. Поэтому безопасный сценарий: добавить nullable-колонку без DEFAULT (это быстрая операция с метаданными), затем при необходимости пакетно заполнить значения и лишь потом, отдельным шагом, навесить `NOT NULL`/индекс. В PostgreSQL 11+ и MySQL 8 константный DEFAULT уже добавляется мгновенно, но проверять поведение конкретной версии обязательно.

## Как это работает

Базовая операция:

```sql
ALTER TABLE orders ADD COLUMN delivery_date DATE;
```

Без DEFAULT и без `NOT NULL` СУБД просто правит метаданные таблицы — строки на диске не трогаются, блокировка короткая.

Заполнение существующих строк делают пакетами, чтобы не держать длинную транзакцию и не раздувать журнал:

```sql
UPDATE orders SET delivery_date = '1970-01-01'
WHERE delivery_date IS NULL AND id BETWEEN 1 AND 10000;
```

Диапазон сдвигают и повторяют, пока не покроют всю таблицу. Ограничение `NOT NULL` и индексы добавляют последним шагом, когда данные уже на месте (в PostgreSQL — через `CREATE INDEX CONCURRENTLY`).

## Подводные камни

- `ADD COLUMN ... DEFAULT` с волатильным выражением (например, `now()`) всё равно переписывает таблицу — мгновенным остаётся только константный DEFAULT.
- Оборачивание `ALTER TABLE` в транзакцию не делает его безопасным: транзакция держит блокировку до `COMMIT`, и длинная DDL может выстроить очередь из ожидающих запросов.
- Большой `UPDATE` одной командой раздувает WAL/undo и держит долгую блокировку строк — именно поэтому заполняют батчами.
- В PostgreSQL даже короткий `ALTER` требует `ACCESS EXCLUSIVE` блокировки; если её ждёт долгий запрос, новые запросы встанут за ним. Помогает `lock_timeout` и повтор.
- «Запустить ночью» — паллиатив, а не решение: для систем без окна простоя нужен именно подход с метаданными плюс батчи.
