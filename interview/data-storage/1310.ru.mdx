# Пул соединений в SQLAlchemy: что это решает и из чего собрано

## Суть

Пул держит наготове набор уже открытых подключений к базе и выдаёт их по запросу, вместо того чтобы каждый раз заново устанавливать соединение. Это убирает дорогой хендшейк из горячего пути и заодно ограничивает число одновременных коннектов к БД.

## Что ответить на собесе

Установка нового TCP- и аутентификационного соединения с базой стоит миллисекунд и ресурсов на обеих сторонах — пул убирает этот хендшейк из горячего пути.

- При создании движка SQLAlchemy заводит набор переиспользуемых подключений (по умолчанию `QueuePool`).
- Когда коду нужно соединение, он берёт свободное из пула.
- Когда работа закончена и соединение закрывается на уровне приложения, физический коннект не рвётся, а возвращается в пул.
- Параметры `pool_size`, `max_overflow` и `pool_timeout` регулируют базовый размер, допустимый перерасход и время ожидания свободного коннекта.

- **Создание**: `create_engine` поднимает пул. По умолчанию это `QueuePool` поверх очереди готовых соединений.
- **Выдача**: есть свободное - отдаётся оно; нет, но лимит `pool_size + max_overflow` не достигнут - открывается новое; иначе запрос ждёт до `pool_timeout` секунд и при истечении получает ошибку.
- **Возврат**: `connection.close()` в коде не закрывает физический сокет, а кладёт его обратно в пул.
- **Утилизация**: соединения сверх `pool_size` (overflow) после возврата закрываются; долгоживущие можно пересоздавать через `pool_recycle`.

```python
from sqlalchemy import create_engine, text

engine = create_engine(
    "postgresql://user:password@localhost/mydatabase",
    pool_size=5,        # постоянный набор соединений
    max_overflow=10,    # сколько ещё допускается сверх пула
    pool_timeout=30,    # сколько секунд ждать свободный коннект
    pool_recycle=1800,  # пересоздавать соединение старше 30 минут
)

with engine.connect() as connection:
    result = connection.execute(text("SELECT id FROM my_table"))
    for row in result:
        print(row.id)
# выход из with возвращает соединение в пул, а не рвёт его
```

`engine.connect()` берёт коннект из пула, контекстный менеджер гарантирует возврат, а `pool_recycle` спасает от мёртвых соединений, которые БД или файрвол молча закрыли.

## Подводные камни

- **Незакрытые соединения** утекают: если не использовать `with` или не звать `close()`, коннекты не возвращаются и пул быстро исчерпывается, вызывая `TimeoutError` по `pool_timeout`.
- **Слишком большой пул × число воркеров** легко упирается в лимит `max_connections` PostgreSQL. Реальный потолок - это `(pool_size + max_overflow)` умноженное на количество процессов.
- **Форк процесса** (gunicorn, multiprocessing) ломает унаследованные соединения. После форка нужен `engine.dispose()`, иначе несколько процессов делят один сокет.
- **`max_overflow=-1`** снимает потолок overflow - пул будет открывать соединения без ограничения, что обнуляет защиту базы от перегрузки.
- Overflow-соединения недолговечны: после возврата они закрываются, поэтому постоянная работа на overflow означает, что `pool_size` подобран слишком мал.
