# Почему связь многие-ко-многим требует третьей таблицы?

## Суть

Реляционная модель не умеет хранить M2M напрямую: связь «многие ко многим» раскладывается в отдельную промежуточную (join) таблицу. В ней лежат внешние ключи на первичные ключи обеих сторон, и каждая строка — это одна пара «слева-справа».

## Что ответить на собесе

В одной строке таблицы нельзя хранить переменное число ссылок, поэтому два «многих» соединить напрямую невозможно — нужна таблица-связка с двумя внешними ключами.

- Классический пример: студенты и курсы. Таблица `StudentCourses` хранит пары `(student_id, course_id)`.
- Составной первичный ключ из этих двух колонок гарантирует уникальность пары.
- Так один студент связывается с многими курсами, а курс — со многими студентами, и любую связь можно добавить или убрать одной строкой.

## Как это работает

Между двумя таблицами заводят join-таблицу с двумя FK. Составной PK `(student_id, course_id)` запрещает дубли пары, а внешние ключи держат ссылочную целостность.

```sql
CREATE TABLE Students (
    student_id INT PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE Courses (
    course_id INT PRIMARY KEY,
    title VARCHAR(100)
);

CREATE TABLE StudentCourses (
    student_id INT,
    course_id INT,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES Students(student_id),
    FOREIGN KEY (course_id) REFERENCES Courses(course_id)
);
```

Привязать студента к курсу — вставить строку в `StudentCourses`; разорвать связь — удалить эту строку. Новые связи не требуют менять структуру основных таблиц. В Django это поле `ManyToManyField`, под которым ORM создаёт ровно такую join-таблицу автоматически.

## Подводные камни

- Без составного первичного ключа (или `UNIQUE` на паре) в связку проникнут дубликаты одной и той же пары.
- Если у связи есть собственные атрибуты (дата записи, оценка), нужна явная модель-связка; в Django это `through`-модель, а не голый `ManyToManyField`.
- Запросы через M2M почти всегда тянут `JOIN`; для частых выборок индексы на колонках FK обязательны, иначе деградация на больших таблицах.
- Удаление записи на одной стороне должно корректно подчищать связку: продумайте `ON DELETE` (или `on_delete` в Django), иначе останутся висячие строки или ошибки целостности.
