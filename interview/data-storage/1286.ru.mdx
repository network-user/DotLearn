# Какой индекс PostgreSQL выбрать под задачу: B-tree, Hash, GIN или GiST

## Суть

В PostgreSQL индекс подбирают под форму данных и запросов. B-tree закрывает равенство и диапазоны и стоит по умолчанию, Hash работает только на равенство, GIN живёт на составных значениях (массивы, JSONB, полнотекст), а GiST берёт на себя геометрию, диапазоны и приблизительный поиск.

## Что ответить на собесе

Индекс выбираю под форму данных и запросов, а не ставлю наугад:

- **B-tree** — по умолчанию: умеет `=`, сравнения с `<`/`>` и сортировку;
- **Hash** — редко, только когда нужно строгое равенство и ничего больше;
- **GIN** — для массивов, JSONB и полнотекстового поиска: инвертированный индекс быстро отвечает «в каких строках встречается это значение»;
- **GiST** — для геометрии, диапазонных типов и поиска ближайших точек, где обычное дерево не подходит.

## Как это работает

Индекс - это отдельная структура, которая позволяет находить строки без полного прохода по таблице. Каждый тип оптимизирован под свой класс запросов.

**B-tree** хранит ключи в сбалансированном дереве, поэтому одинаково хорош и для точного поиска, и для диапазонов, и для `ORDER BY`.

```sql
CREATE INDEX idx_orders_created ON orders (created_at);
```

**Hash** держит хеш-коды ключей и отвечает только на `=`. Диапазоны и сортировку он не поддерживает, поэтому выигрыш над B-tree обычно невелик.

```sql
CREATE INDEX idx_users_email_hash ON users USING HASH (email);
```

**GIN** (Generalized Inverted Index) для каждого элемента значения хранит список строк, где он встречается. Это идеально для массивов, JSONB и полнотекстового поиска через `tsvector`.

```sql
CREATE INDEX idx_articles_search ON articles USING GIN (to_tsvector('russian', body));
```

**GiST** (Generalized Search Tree) - это каркас, на котором строятся индексы для геометрии, диапазонных типов и иерархий. Через расширение этот же тип используют для поиска ближайших соседей.

```sql
CREATE INDEX idx_places_geom ON places USING GiST (location);
```

## Подводные камни

- Hash-индекс почти всегда проигрывает: он не ускоряет диапазоны и сортировку, поэтому B-tree по той же колонке универсальнее. До PostgreSQL 10 Hash вдобавок не писался в WAL и терялся при сбое.
- GIN отлично читает, но дорого пишет: вставки и обновления заметно тяжелее, потому что приходится трогать много списков. Под нагрузку на запись смотрите на параметр `fastupdate` и при необходимости на GiST как компромисс.
- Индекс не используется, если в условии стоит функция или приведение типа, под которые он не создан. Для регистронезависимого поиска нужен индекс по `lower(col)`, а не по самой колонке.
- Сам по себе индекс не гарантирует, что планировщик его выберет: на маленькой таблице или при выборке большой доли строк выгоднее полный скан.
