# Шардинг базы: как горизонтальное разрезание данных снимает узкие места

## Суть

Шардинг — это разрезание одной большой базы на независимые куски (шарды), каждый из которых живёт на своём узле и хранит только часть строк. Цель — развести нагрузку по серверам, когда один уже не тянет.

## Что ответить на собесе

Шардинг — это горизонтальное масштабирование хранилища: вместо наращивания мощности одного сервера данные делятся по ключу шардирования на несколько независимых баз.

- Каждый шард обслуживает свой диапазон или хэш ключей, поэтому запросы и запись распределяются между узлами.
- Получаем рост пропускной способности и устойчивость: падение одного шарда не кладёт всю систему.
- Платим усложнением: запросы, задевающие несколько шардов, и поддержание консистентности.

## Как это работает

Ключевой выбор — функция распределения строк по шардам. Есть два базовых способа:

- **По диапазону**: например, `user_id` 1–1млн на первый шард, 1млн–2млн на второй. Просто, но легко получить перекос, если новые записи льются в один диапазон.
- **По хэшу**: `hash(user_id) % N` даёт равномерное размещение, но усложняет добавление новых шардов (приходится перераспределять данные).

Отдельно стоит различать горизонтальное (делим строки) и вертикальное (выносим часть столбцов в отдельное хранилище) разбиение. Шардинг обычно про первое.

Минимальный пример — две таблицы под один логический набор пользователей, разнесённые по диапазону id:

```sql
CREATE TABLE users_shard_1 (
    user_id INT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100)
);

CREATE TABLE users_shard_2 (
    user_id INT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100)
);
```

Приложение или прокси-слой по `user_id` решает, в какой шард писать и откуда читать.

## Подводные камни

- **Перекос (hot shard)**: неудачный ключ шардирования сваливает большую часть нагрузки на один шард, и масштабирование не работает.
- **Кросс-шардовые запросы**: `JOIN` и агрегаты по всем шардам дорогие, их приходится собирать на уровне приложения.
- **Ресхардинг**: добавление шарда при хэш-распределении требует перераспределения данных — это болезненная операция, поэтому часто берут consistent hashing.
- **Уникальность и транзакции**: автоинкрементные id и транзакции через несколько шардов перестают работать «бесплатно».
- Шардинг — последнее средство. Сначала исчерпывают индексы, кэш и реплики для чтения.
