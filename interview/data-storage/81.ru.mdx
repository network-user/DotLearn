# Redis в боевых системах: какие задачи он закрывает и почему его выбирают?

## Суть

Redis - это key-value хранилище, которое держит данные в оперативной памяти, поэтому отдаёт их за микросекунды. В реальных системах его ставят перед основной БД как кэш, а также под сессии, очереди задач и онлайн-метрики.

## Что ответить на собесе

Redis беру там, где нужна скорость и низкая задержка, а не долговременная надёжность хранения.

- Кэш горячих данных, чтобы разгрузить PostgreSQL.
- Общее хранилище сессий для горизонтально масштабированного веб-приложения.
- Лёгкие очереди задач на списках; счётчики и rate-limiting на атомарных операциях.
- Сильная сторона — не просто «быстрый кэш», а готовые структуры данных (списки, множества, хеши, sorted set), которые позволяют решать задачу на стороне хранилища, а не тащить всё в приложение.

Все значения лежат в RAM, отсюда и скорость. Ключевое отличие от «тупого» memcached - типизированные структуры:

- **Строки (String)** - текст, числа или бинарные данные; на них же строят атомарные счётчики через `INCR`.
- **Списки (List)** - двусторонняя очередь; добавление и снятие с обоих концов делают из них простую очередь задач.
- **Множества (Set)** - уникальные неупорядоченные элементы; удобно для тегов, уникальных посетителей.
- **Хеши (Hash)** - набор полей внутри одного ключа; хорошо ложатся на объект (профиль пользователя).
- **Sorted Set** - множество, где у элемента есть score; на нём делают рейтинги и таблицы лидеров.

Пример кэширования: кладём результат в Redis и сначала проверяем его, прежде чем идти в основную БД.

```python
import redis

cache = redis.Redis(host="localhost", port=6379, db=0)


def get_user_name(user_id: int) -> str:
    key = f"user:{user_id}:name"
    cached = cache.get(key)
    if cached is not None:
        return cached.decode("utf-8")

    name = load_name_from_db(user_id)
    cache.set(key, name, ex=300)
    return name
```

Здесь `ex=300` задаёт TTL: запись сама исчезнет через 5 минут, и кэш не будет вечно держать устаревшие данные.

## Подводные камни

- **Память не бесконечна.** Redis хранит всё в RAM; без `maxmemory` и политики вытеснения (`allkeys-lru` и др.) он упрётся в лимит и начнёт отдавать ошибки на запись.
- **Это не замена основной БД.** По умолчанию данные могут потеряться при перезапуске; для долговременного хранения нужна настройка персистентности (RDB/AOF), и всё равно это не транзакционная СУБД.
- **Инвалидация кэша.** Забыть TTL или не сбросить ключ при обновлении данных - классический источник «приложение показывает старые значения».
- **Один поток на команды.** Ядро Redis обрабатывает команды в одном потоке, поэтому одна тяжёлая операция (например `KEYS *` на большой базе) блокирует всех остальных клиентов.
