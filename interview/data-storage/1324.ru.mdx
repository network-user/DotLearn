# В чём главные сложности репликации PostgreSQL и что держать в голове?

## Суть

Основная боль репликации - удержать копии в синхронном с мастером состоянии: следить за отставанием реплик, выбирать между скоростью и согласованностью, настраивать автопереключение при сбое и непрерывно мониторить нагрузку, которую репликация добавляет основному серверу.

## Что ответить на собесе

Репликация PostgreSQL копирует данные с мастера на реплики ради отказоустойчивости, масштабирования чтения и доступности.

- Главный компромисс: асинхронный режим (мастер не ждёт подтверждения, реплики могут отставать, но запись быстрая) против синхронного (мастер ждёт подтверждения, данные согласованы, но запись медленнее).
- Ключевые проблемы: отставание реплик (replication lag), согласованность данных, накладные расходы на мастер.
- Встроенного автоматического failover нет, его настраивают через Patroni или pgpool; нужен постоянный мониторинг через `pg_stat_replication`.
- Отдельно стоит упомянуть обновления и миграции: без планирования легко получить простой или рассинхронизацию.

## Как это работает

Репликация в PostgreSQL построена на WAL - журнале упреждающей записи. Мастер передаёт записи WAL на реплики, а те проигрывают их у себя. В асинхронном режиме мастер не дожидается реплик, в синхронном - ждёт подтверждения хотя бы от одной (`synchronous_standby_names`).

Пример настройки потоковой репликации на стороне мастера:

```sql
-- На основном сервере
ALTER SYSTEM SET wal_level = 'replica';
ALTER SYSTEM SET archive_mode = 'on';
ALTER SYSTEM SET archive_command = 'cp %p /var/lib/postgresql/archive/%f';
ALTER SYSTEM SET max_wal_senders = 3;
ALTER SYSTEM SET wal_keep_size = '1GB';
SELECT pg_reload_conf();

-- Роль для подключения реплик
CREATE ROLE replicator WITH REPLICATION LOGIN PASSWORD 'password';
```

На стороне реплики задаётся подключение к мастеру и режим горячего резерва (в современных версиях - через `standby.signal` и `postgresql.conf`):

```ini
# postgresql.conf на реплике
primary_conninfo = 'host=primary_host port=5432 user=replicator password=password'
restore_command = 'cp /var/lib/postgresql/archive/%f %p'
hot_standby = on
```

Что значат ключевые параметры:

- `wal_level = 'replica'` - уровень журналирования WAL, достаточный для репликации.
- `max_wal_senders` - сколько процессов могут рассылать WAL на реплики одновременно.
- `wal_keep_size` - объём WAL, который мастер удерживает для отстающих реплик (в старых версиях - `wal_keep_segments`).
- `primary_conninfo` - строка подключения реплики к мастеру.
- `hot_standby = on` - разрешает читать с реплики, пока она применяет WAL.

Отставание реплики отслеживают через системное представление:

```sql
SELECT client_addr, state, sent_lsn, replay_lsn,
       sent_lsn - replay_lsn AS lag_bytes
FROM pg_stat_replication;
```

## Подводные камни

- В асинхронном режиме после падения мастера несколько последних транзакций могут быть потеряны - реплика отставала. Если потеря недопустима, нужен синхронный режим.
- Синхронная репликация с единственной репликой опасна: если реплика недоступна, мастер блокирует запись, ожидая подтверждения. Держите хотя бы кворум резервов.
- Автоматического failover в PostgreSQL из коробки нет - без Patroni/pgpool переключение делается руками, и это удлиняет простой.
- Долгие запросы на реплике конфликтуют с применением WAL; параметры `max_standby_streaming_delay` и `hot_standby_feedback` регулируют, кто кому уступает.
- `wal_keep_size` слишком мал - отстающая реплика теряет нужные сегменты WAL и выпадает из репликации; спасает слот репликации (`pg_create_physical_replication_slot`).
- Читать с реплики удобно для масштабирования, но из-за лага можно увидеть устаревшие данные - не годится там, где нужна строгая актуальность сразу после записи.
