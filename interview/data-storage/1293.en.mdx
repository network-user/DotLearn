# What a transaction is and why group operations together

## In short

A transaction is a set of operations that the DBMS executes as one indivisible action: either all succeed, or none is applied. This keeps the database consistent even amid errors and concurrent access.

## How to answer in an interview

A transaction combines several changes into a unit of work with ACID guarantees. The classic example is a money transfer: a debit from one account and a credit to another must happen together. If something fails after the debit, `ROLLBACK` undoes everything, and consistency isn't broken. `COMMIT` confirms, `ROLLBACK` cancels, and the start of the transaction sets the boundaries.

## How it works

In code a transaction is a block between the start and `commit`/`rollback`. Using SQLite and Python as an example:

```python
import sqlite3

connection = sqlite3.connect('bank.db')
cursor = connection.cursor()

try:
    cursor.execute('UPDATE accounts SET balance = balance - 100 WHERE account_id = 1')
    cursor.execute('UPDATE accounts SET balance = balance + 100 WHERE account_id = 2')
    connection.commit()
except Exception as error:
    connection.rollback()
    print(f"Transaction failed: {error}")
finally:
    connection.close()
```

Both `UPDATE`s run inside a single transaction. `commit()` commits them all at once; on any exception `rollback()` returns the balances to their initial state. The `sqlite3` driver opens a transaction itself before the first modifying query, so an explicit `BEGIN` isn't needed - it's enough to call `commit` or `rollback` in time.

## Pitfalls

- A forgotten `commit()` means lost changes: when the connection closes, an uncommitted transaction is rolled back.
- A long transaction holds locks and gets in others' way - the boundaries should be made as narrow as possible.
- In `sqlite3` autocommit behavior depends on the `isolation_level` parameter; with the value `None` every query is committed immediately, and the familiar transactional semantics disappear.
- `rollback` reverts changes in the database, but it doesn't undo the side effects of your code (sent emails, modified variables) - the application is responsible for that.
