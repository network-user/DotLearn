# WHERE or HAVING: where in the SQL pipeline rows and groups are filtered out

## In short

`WHERE` runs earlier: it filters out individual rows before they reach grouping and aggregates. `HAVING` kicks in later - after `GROUP BY` - and selects already-formed groups by the values of aggregate functions.

## How to answer in an interview

"The logical order of execution is this: `FROM` → `WHERE` → `GROUP BY` → `HAVING` → `SELECT` → `ORDER BY`. Everything follows from it. `WHERE` filters raw rows before aggregation, so you can't use `COUNT()` or `SUM()` in it - they don't exist yet. `HAVING` fires after grouping and is exactly meant for conditions on aggregates, for example 'only departments with more than ten employees'. The practical rule: filter out everything you can in `WHERE` - that way less data reaches grouping and the query is faster; leave `HAVING` strictly for conditions on aggregates."

`WHERE` discards rows before grouping. Only employees with a salary above 50000 will reach the aggregate.

```sql
SELECT department, COUNT(*)
FROM employees
WHERE salary > 50000
GROUP BY department;
```

`HAVING` filters the result after `GROUP BY`, relying on aggregate values.

```sql
SELECT department, COUNT(*) AS cnt
FROM employees
GROUP BY department
HAVING COUNT(*) > 10;
```

They can be combined: first `WHERE` narrows the set of rows, then grouping, then `HAVING` cuts off groups.

```sql
SELECT department, AVG(salary) AS avg_salary
FROM employees
WHERE active = true
GROUP BY department
HAVING AVG(salary) > 70000;
```

## Pitfalls

- You can't reference aggregate functions in `WHERE` - at this stage there are no groups yet, and an attempt to write `WHERE COUNT(*) > 10` will raise an error.
- A condition on a non-aggregated column mistakenly moved into `HAVING` works, but wastefully: it filters rows only after needless grouping. Such conditions are better kept in `WHERE`.
- In standard SQL, aliases from `SELECT` aren't visible in `WHERE` and, formally, in `HAVING`, because `SELECT` is computed later; PostgreSQL allows an alias only in `GROUP BY` and `ORDER BY`.
- `HAVING` without `GROUP BY` treats the whole selection as a single group - this is valid, but easy to mistake for an error if you don't expect it.
