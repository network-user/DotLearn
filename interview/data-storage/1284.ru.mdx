# Когда хэш-индекс выигрывает, а когда проигрывает B-дереву?

## Суть

Хэш-индекс хорош ровно для одного сценария - мгновенный поиск по точному равенству ключа. Для диапазонов и сортировки он бесполезен, потому что хэш-функция не сохраняет порядок ключей.

## Что ответить на собесе

Хэш-индекс хорош ровно для одного сценария — поиск по точному равенству.

- Беру его, когда запросы — исключительно проверки на равенство: выборка по `user_id`, по номеру заказа, по точному ключу. Хэш-функция превращает ключ в адрес ячейки, доступ в среднем за константу.
- Как только в запросах появляются `>`, `<`, `BETWEEN` или `ORDER BY` — хэш-индекс не поможет, нужно B-дерево, которое хранит ключи упорядоченно.
- На практике в большинстве СУБД дефолт — B-дерево из-за универсальности, а хэш применяют точечно.

1. **Хэш-функция**: при добавлении записи ключ (например, идентификатор) превращается в число-адрес.
2. **Хэш-таблица**: адрес указывает на ячейку, где хранится запись или ссылка на неё.
3. **Поиск**: тот же ключ снова прогоняют через ту же функцию и сразу попадают в нужную ячейку.

В Python ту же идею наглядно показывает словарь - он и есть хэш-таблица:

```python
hash_table = {}

key = "user_id_123"
value = {"name": "Alice", "age": 30}
hash_table[key] = value

search_key = "user_id_123"
if search_key in hash_table:
    result = hash_table[search_key]
    print("Record found:", result)
else:
    print("Record not found")
```

Поиск `search_key in hash_table` - это одна хэш-операция, а не перебор всех ключей.

## Подводные камни

- Диапазонные запросы и сортировка хэш-индексу недоступны: порядок ключей он не хранит.
- При большом числе коллизий (разные ключи дают один адрес) производительность деградирует к линейному поиску внутри корзины.
- Составной поиск по части ключа невозможен - хэш считается от всего ключа целиком.
- В PostgreSQL хэш-индексы исторически были менее надёжными, чем B-tree, поэтому по умолчанию выбирают B-дерево.
