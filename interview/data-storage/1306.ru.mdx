# Как в PostgreSQL сделать отдельную колонку фактически неизменяемой?

## Суть

Отдельной команды «запретить UPDATE одного столбца» в PostgreSQL нет. Эффект собирают из подручных средств: триггер `BEFORE UPDATE` отбивает попытку изменить значение, а представление прячет поле от тех, кому его вообще не положено видеть.

## Что ответить на собесе

Встроенного флага неизменяемости у колонки в Postgres нет, эффект собирают из двух средств:

- **Триггер `BEFORE UPDATE`** сравнивает `NEW.salary` с `OLD.salary` через `IS DISTINCT FROM` и бросает исключение, если значение пытаются поменять, — это закрывает запись.
- **Представление** без этой колонки отдаю пользователю вместо самой таблицы, а прямой доступ к базовой таблице закрываю правами, — это закрывает чтение.
- Вместе они дают «только для чтения» на уровне одного поля.

## Как это работает

Представление - это виртуальная таблица: в неё попадают только перечисленные при создании колонки. Чувствительное поле туда просто не включают, и пользователь, работающий через view, его не видит.

```sql
CREATE TABLE employees (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    salary NUMERIC
);

INSERT INTO employees (name, salary) VALUES ('Alice', 50000), ('Bob', 60000);

CREATE VIEW public_employees AS
SELECT id, name FROM employees;
```

Запрет на правку вешают триггером. Функция на `plpgsql` сверяет старое и новое значение и при расхождении прерывает операцию исключением - транзакция откатывается целиком.

```sql
CREATE OR REPLACE FUNCTION prevent_salary_update()
RETURNS TRIGGER AS $$
BEGIN
    IF NEW.salary IS DISTINCT FROM OLD.salary THEN
        RAISE EXCEPTION 'Updating salary is not allowed';
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER salary_update_trigger
BEFORE UPDATE ON employees
FOR EACH ROW EXECUTE FUNCTION prevent_salary_update();
```

Триггер `BEFORE UPDATE FOR EACH ROW` срабатывает на каждую изменяемую строку до фактической записи, поэтому запрещённое изменение не попадает в таблицу вовсе.

## Подводные камни

- `IS DISTINCT FROM` обязателен вместо `<>`: обычное сравнение даёт `NULL` (то есть «не истина»), если одно из значений `NULL`, и проверка молча пропустит изменение `NULL → значение`.
- Само по себе view не защищает базовую таблицу: если у пользователя есть права на `employees`, он обойдёт представление. Доступ к таблице нужно закрывать через `GRANT`/`REVOKE`.
- Триггер ловит только `UPDATE`. Перезапись через `DELETE` + `INSERT` или `TRUNCATE` он не остановит - при необходимости вешают отдельные триггеры.
- Суперпользователь и владелец таблицы могут отключить триггер (`ALTER TABLE ... DISABLE TRIGGER`), так что это защита от ошибок и обычных пользователей, а не от админа.
