# How an ORM beats hand-written SQL and where its strength lies

## In short

An ORM (Object-Relational Mapping) turns table rows into language objects and methods on those objects into SQL. As a result you write less boilerplate, make fewer query-syntax mistakes, and automatically get protection against SQL injection.

## How to answer in an interview

An ORM is a layer between the application and the relational database that maps classes to tables and instances to rows. The main upsides I'd name: faster development, because you don't hand-write routine CRUD; query parameterization out of the box, which closes off injection; portability across database engines; and a single codebase in the application's language instead of a mix with SQL strings. The downside, which is fair to mention honestly, is that you can lose control over the generated queries and run into N+1 or an inefficient plan, so I still verify complex selects against the real SQL.

## How it works

An ORM builds queries from operations on objects. Below is a model and working with it in SQLAlchemy.

```python
from sqlalchemy import create_engine, Column, Integer, String
from sqlalchemy.orm import declarative_base, sessionmaker

Base = declarative_base()


class User(Base):
    __tablename__ = "users"

    id = Column(Integer, primary_key=True)
    name = Column(String)
    age = Column(Integer)


engine = create_engine("sqlite:///:memory:")
Base.metadata.create_all(engine)

Session = sessionmaker(bind=engine)
session = Session()

session.add(User(name="Alice", age=30))
session.commit()

for user in session.query(User).filter(User.age >= 18):
    print(user.name, user.age)
```

The `User` class mirrors the `users` table, and `session.query(...).filter(...)` is translated into `SELECT ... WHERE age >= 18`. The value `18` goes into the query as a parameter rather than being spliced into the string - that's where the injection protection comes from.

## Pitfalls

- **The N+1 problem.** Accessing related objects in a loop produces a separate query per row. It's cured by eager loading (`select_related`/`prefetch_related` in Django, `joinedload` in SQLAlchemy).
- **The hidden cost of abstraction.** A convenient method can generate a heavy query; for hot spots look at the resulting SQL and drop down to a raw query when needed.
- **Not a silver bullet for analytics.** Complex aggregates and window functions are sometimes simpler and faster to write in plain SQL.
- **Lazy objects outside the session.** After the session is closed, accessing not-yet-loaded fields can throw an exception.
