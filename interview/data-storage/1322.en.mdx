# Which set of SQL commands provides transaction control?

## In short

Transactions in SQL are managed by five commands: `BEGIN` opens a block of operations, `COMMIT` commits them permanently, `ROLLBACK` undoes everything, `SAVEPOINT` sets an intermediate marker for a partial rollback, and `SET TRANSACTION` tweaks parameters such as the isolation level.

## How to answer in an interview

The basic set for transaction control is `BEGIN` (or `START TRANSACTION`), `COMMIT`, and `ROLLBACK`. The first command starts a transaction, the second locks in the result and makes it visible to other sessions, and the third undoes all accumulated changes. For fine-grained control you add `SAVEPOINT` with `ROLLBACK TO SAVEPOINT` - they let you roll back only part of the work inside a transaction without discarding it entirely. And `SET TRANSACTION` sets the isolation level and the access mode. Together these provide atomicity: either all operations go through, or none do.

## How it works

A transaction is a group of operations that executes as a single indivisible unit. This matters especially in systems with concurrent access, where the same rows are edited by several clients at once.

Start a transaction and mark the boundary:

```sql
BEGIN;
```

Commit the accumulated changes - after this a rollback is no longer possible:

```sql
COMMIT;
```

Roll everything back to the state before the transaction began:

```sql
ROLLBACK;
```

Set an intermediate marker and, if needed, return to it without losing the whole transaction:

```sql
SAVEPOINT before_transfer;
ROLLBACK TO SAVEPOINT before_transfer;
```

Configure the isolation level for the current transaction:

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

The canonical example is a money transfer between accounts. The debit and the credit must both go through or neither:

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 2;
COMMIT;
```

If the account has insufficient funds, cancel the whole transaction:

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;
SAVEPOINT after_debit;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 2;
-- the check failed
ROLLBACK TO SAVEPOINT after_debit;
COMMIT;
```

## Pitfalls

- `COMMIT` is irreversible: after committing, `ROLLBACK` will no longer bring back the old data - you need a compensating UPDATE.
- Many DBMSs work in autocommit mode - each individual statement is committed immediately, and without an explicit `BEGIN` a multi-step transaction simply won't form.
- An error inside a transaction in PostgreSQL puts it into an "aborted" state: until a `ROLLBACK` (or a rollback to a SAVEPOINT), any subsequent commands will fail.
- `ROLLBACK TO SAVEPOINT` does not end the transaction - you still need a `COMMIT` or a full `ROLLBACK` afterward.
- Too high an isolation level (`SERIALIZABLE`) raises the risk of serialization conflicts, and the application must be able to retry a failed transaction.
