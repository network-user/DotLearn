# How to model a one-to-many relationship in a relational database

## In short

A one-to-many relationship is implemented through a foreign key on the "many" side: the child table holds a column that references the primary key of the parent. One parent row corresponds to many child rows, but each child belongs to exactly one parent.

## How to answer in an interview

I put the foreign key in the table on the "many" side, not the "one". The classic example is an author and books: one author writes many books, each book has one author. So the `Books` table gets an `AuthorID` column - a foreign key to `Authors.AuthorID`. No separate junction table is needed here (that is only required for "many-to-many"). Through the FK the database makes sure every book references an existing author, and a `JOIN` on this field brings the data back together.

## How it works

Take `Authors` (the "one" side) and `Books` (the "many" side). The foreign key lives in `Books`:

```sql
CREATE TABLE Authors (
    AuthorID INT PRIMARY KEY,   -- author identifier
    Name     VARCHAR(100)
);

CREATE TABLE Books (
    BookID   INT PRIMARY KEY,   -- book identifier
    Title    VARCHAR(100),
    AuthorID INT,               -- foreign key to Authors
    FOREIGN KEY (AuthorID) REFERENCES Authors(AuthorID)
);
```

To gather all books by a specific author, we join the tables on `AuthorID`:

```sql
SELECT Books.Title
FROM Books
JOIN Authors ON Books.AuthorID = Authors.AuthorID
WHERE Authors.Name = 'Author Name';
```

The `JOIN` stitches books to authors on the shared field, and the `WHERE` keeps only the author we want.

## Pitfalls

- **The FK goes on the "many" side**, not the "one". Trying to store a list of child IDs in the parent row (comma-separated or an array without normalization) is a typical mistake that breaks integrity and queries.
- **The "one" side must not be duplicated in the child table**: do not copy the author's name into every book - store only `AuthorID`, otherwise the data will diverge.
- **NULL in an FK = "parent not assigned"**. If a book must have an author, add `NOT NULL` to `AuthorID`, otherwise orphan books will appear.
- **Deleting the parent**: without an `ON DELETE` rule the database by default will not let you delete an author who has books. You need to deliberately choose `CASCADE`, `SET NULL`, or leave `RESTRICT`.
- **An index on the FK column** is added manually: in PostgreSQL it is not created automatically, and without it `JOIN`s and filters by `AuthorID` will be slow on large tables.
