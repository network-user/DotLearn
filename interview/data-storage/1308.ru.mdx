# Какие проблемы вылезают при репликации в PostgreSQL?

## Суть

Репликация даёт отказоустойчивость и масштабирование чтения, но взамен приносит лаг реплики, риск конфликтов при записи, борьбу за согласованность, обязательный мониторинг и лишний расход железа.

## Что ответить на собесе

Главная проблема - отставание реплики: по умолчанию репликация асинхронная, и реплика какое-то время отдаёт устаревшие данные. Для сценариев «прочитал сразу после записи» это ломает логику, и приходится либо читать с мастера, либо включать синхронную репликацию - а та замедляет коммиты, потому что мастер ждёт подтверждения. Дальше: на реплику нельзя писать, иначе будет рассинхрон, поэтому она должна быть строго read-only. Плюс репликация требует мониторинга лага и состояния потоков, а также дополнительного CPU, памяти и диска под каждую реплику. Грубо: выбираешь между свежестью данных, скоростью записи и стоимостью.

Репликация копирует изменения с мастера на реплики через поток WAL (журнал упреждающей записи). Базовая настройка на мастере выставляет уровень WAL, разрешает архивацию и лимит потоков на реплики, после чего снимается базовая копия для реплики.

```sql
-- На мастере:
ALTER SYSTEM SET wal_level = 'replica';
ALTER SYSTEM SET archive_mode = 'on';
ALTER SYSTEM SET archive_command = 'cp %p /path_to_archive/%f';
ALTER SYSTEM SET max_wal_senders = 3;

SELECT pg_reload_conf();
```

```bash
# Снимаем базовую копию для реплики (psql тут ни при чём - это утилита командной строки)
pg_basebackup -h master_host -D /path_to_replica_data -U replication_user -Fp -Xs -P
```

В современных версиях (с 12-й) реплику переводят в standby файлом-флагом `standby.signal` и параметром `primary_conninfo` в `postgresql.conf`. Старый `recovery.conf` из давних инструкций давно убран.

```ini
# postgresql.conf на реплике
primary_conninfo = 'host=master_host port=5432 user=replication_user password=secret'
```

После старта реплика подключается к мастеру и проигрывает WAL, отставая на величину лага.

## Подводные камни

- Асинхронная реплика по определению может отдать устаревшие данные - паттерн read-your-own-writes требует читать с мастера или использовать синхронную репликацию для критичных транзакций.
- Долгие запросы на реплике конфликтуют с применением WAL: либо запрос отменяется (`max_standby_streaming_delay`), либо тормозится применение журнала. Это classic recovery conflict.
- Незакрытый слот репликации опасен: если реплика отвалилась, мастер копит WAL под её слот и может забить диск под завязку.
- `recovery.conf` мёртв с PostgreSQL 12 - инструкции, которые его упоминают, устарели; нужны `standby.signal` и параметры в основном конфиге.
- Стандартная физическая репликация - всё или ничего: всю базу целиком. Если нужна выборочная (отдельные таблицы), это уже логическая репликация со своими ограничениями.
