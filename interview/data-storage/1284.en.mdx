# When does a hash index win, and when does it lose to a B-tree?

## In short

A hash index is good for exactly one scenario - instant lookup by exact key equality. For ranges and sorting it's useless, because the hash function doesn't preserve the order of keys.

## How to answer in an interview

I reach for a hash index when the queries are exclusively equality checks: a lookup by `user_id`, by order number, by an exact key. The hash function turns the key into a cell address, and access comes out as O(1) on average. But as soon as `>`, `<`, `BETWEEN`, or `ORDER BY` appear in the queries, a hash index won't help - here you need a B-tree, which keeps keys ordered. In practice, in most database engines the default is a B-tree precisely because of its versatility, and hash is applied selectively.

1. **Hash function**: when a record is added, the key (for example, an identifier) is turned into a number-address.
2. **Hash table**: the address points to a cell where the record or a reference to it is stored.
3. **Lookup**: the same key is run through the same function again and lands directly in the right cell.

In Python the same idea is clearly shown by a dictionary - it is a hash table:

```python
hash_table = {}

key = "user_id_123"
value = {"name": "Alice", "age": 30}
hash_table[key] = value

search_key = "user_id_123"
if search_key in hash_table:
    result = hash_table[search_key]
    print("Record found:", result)
else:
    print("Record not found")
```

The lookup `search_key in hash_table` is a single hash operation, not a scan over all keys.

## Pitfalls

- Range queries and sorting are unavailable to a hash index: it doesn't store the order of keys.
- With a large number of collisions (different keys yielding the same address), performance degrades toward a linear search within the bucket.
- A composite lookup by part of the key is impossible - the hash is computed over the whole key as a unit.
- In PostgreSQL hash indexes were historically less reliable than B-tree, so a B-tree is chosen by default.
