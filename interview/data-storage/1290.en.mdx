# ACID transaction guarantees: what each letter is responsible for

## In short

ACID is a set of four guarantees that turn disparate operations into a reliable transaction: atomicity, consistency, isolation and durability. Together they protect data from partial changes, races and loss of results on a crash.

## How to answer in an interview

ACID stands for: Atomicity - a transaction is applied in full or not at all; Consistency - after it the database stays in a valid state, not violating constraints; Isolation - concurrent transactions don't see each other's intermediate results; Durability - a committed result survives a reboot and a crash. These properties matter especially under multi-user access and monetary operations, where a partial change is unacceptable.

## How it works

Atomicity is implemented through an undo log: the DBMS remembers how to roll back unfinished edits, so on an error it rolls them all back.

```sql
BEGIN TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 2;
COMMIT;
```

If a crash happens between the two `UPDATE`s, the `COMMIT` won't occur, and the debit is rolled back - money won't vanish into thin air.

Consistency rests on schema constraints: uniqueness, foreign keys, `CHECK`. A transaction that tried to violate them ends with an error.

Isolation is governed by levels. The stricter the level, the fewer anomalies, but the higher the price in performance.

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

`SERIALIZABLE` guarantees a result as if the transactions ran strictly one after another.

Durability is ensured by writing to the log on persistent storage before confirming `COMMIT`: even if the process crashes right after the commit, the changes will be there on recovery.

## Pitfalls

- The default isolation is rarely `SERIALIZABLE` - usually it's `READ COMMITTED`, and then non-repeatable reads and phantoms are possible. The level must be chosen deliberately.
- Durability depends on settings: with asynchronous log commit (`synchronous_commit = off`) a confirmed transaction can be lost on a crash.
- ACID consistency is about schema rules, not business logic: enforcing application-level invariants is still on you.
- DDL commands in some DBMSs (MySQL/InnoDB) aren't rolled back - don't count on `ALTER TABLE` inside a transaction always being atomic.
