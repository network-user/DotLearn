# Primary key vs foreign key: what is the fundamental difference

## In short

A primary key uniquely identifies a row within its own table - values are unique and never NULL. A foreign key is a column that references the primary key of another table and thereby links the tables, enforcing referential integrity.

## How to answer in an interview

A primary key answers the question "how do I uniquely identify this row": it is unique, disallows NULL, and the DBMS builds an index for it. A foreign key answers the question "which row in another table am I pointing to": it stores a primary-key value from the parent table, and the database makes sure that such a value actually exists there. Roughly: a PK is about the identity of a record, an FK is about the relationship between records. A table has exactly one PK, but it can have several FKs.

## How it works

**Primary key** makes every row distinguishable. Values are unique, NULL is forbidden, and an index is created for the key automatically - so lookups by it are fast.

```sql
CREATE TABLE Users (
    UserID   INT PRIMARY KEY,   -- unique user identifier
    UserName VARCHAR(100),
    Email    VARCHAR(100)
);
```

**Foreign key** stores a reference to the primary key of another table. On insert or update the database checks that such a value exists in the source table, otherwise the operation is rejected.

```sql
CREATE TABLE Orders (
    OrderID   INT PRIMARY KEY,
    OrderDate DATE,
    UserID    INT,
    FOREIGN KEY (UserID) REFERENCES Users(UserID)  -- reference to Users
);
```

Here `Orders.UserID` points to `Users.UserID`. Every order is guaranteed to be tied to an existing user.

The key mechanics:

- **Primary key**: the DBMS builds an index, enforces uniqueness, and forbids NULL.
- **Foreign key**: on insert/update the existence of the value in the parent table is checked; on deletion of the parent the configured rule fires (`RESTRICT`, `CASCADE`, `SET NULL`), which prevents "orphaned" rows.

## Pitfalls

- **NULL in an FK is usually allowed**, but in a PK it never is. A foreign key with a NULL value means "no relationship yet" and does not violate integrity.
- **An FK references not just any column**, but a PK or a column with a uniqueness constraint in the parent table. You cannot reference a non-unique column.
- **The default `ON DELETE`** is often `RESTRICT/NO ACTION`: you will not be allowed to delete a user that orders reference. If you want a cascade, specify `ON DELETE CASCADE` deliberately, otherwise you may accidentally wipe out related data.
- **An FK is not indexed automatically** (unlike a PK) in many DBMSs, including PostgreSQL. Joins and foreign-key checks without an index will be slow - an index on the FK column is added manually.
- **A composite PK** is possible (several columns), and then an FK referencing it must also be composite and match the set of columns.
