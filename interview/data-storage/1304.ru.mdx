# Поиск и разбор тормозящего запроса в PostgreSQL: с чего начать?

## Суть

Чтобы найти медленный запрос в PostgreSQL, подключают расширение `pg_stat_statements`: оно накапливает агрегированную статистику по выполненным запросам. Затем сортируют запросы по суммарному времени, а конкретного виновника разбирают через `EXPLAIN` и `EXPLAIN ANALYZE`.

## Что ответить на собесе

Действую в два этапа. Сначала ищу кандидата: включаю `pg_stat_statements` и смотрю, какие запросы дают наибольшее суммарное и среднее время выполнения - это даёт картину по всей нагрузке, а не по одному вызову. Когда виновник найден, перехожу к разбору плана: `EXPLAIN` показывает план без запуска, а `EXPLAIN ANALYZE` реально выполняет запрос и даёт фактическое время и число строк по каждому узлу. В плане первым делом смотрю на `Seq Scan` там, где ожидался индекс, на расхождение оценочных и фактических строк и на дорогие соединения. Дальше - индекс, переписанный запрос или подкрученные настройки.

## Как это работает

### Подключаем pg_stat_statements

Прописываем библиотеку в `postgresql.conf`, чтобы она грузилась при старте сервера:

```plaintext
shared_preload_libraries = 'pg_stat_statements'
```

После перезапуска создаём расширение в нужной базе:

```sql
CREATE EXTENSION pg_stat_statements;
```

Теперь доступна вьюшка со статистикой - сортируем по суммарному времени:

```sql
SELECT
    query,
    calls,
    total_exec_time,
    mean_exec_time
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;
```

Здесь `query` - текст запроса, `calls` - число вызовов, `total_exec_time` - суммарное время, `mean_exec_time` - среднее на один вызов.

### Разбираем план

```sql
EXPLAIN SELECT * FROM orders WHERE order_date > '2023-01-01';

EXPLAIN ANALYZE SELECT * FROM orders WHERE order_date > '2023-01-01';
```

В выводе `EXPLAIN ANALYZE` читаем:

- `Seq Scan on orders` - идёт последовательное сканирование таблицы.
- `Filter: (order_date > '2023-01-01')` - условие фильтрации.
- `actual time=0.012..0.015 rows=5 loops=1` - фактическое время, число строк и количество проходов узла.

## Подводные камни

- В свежих версиях PostgreSQL колонки называются `total_exec_time` и `mean_exec_time`; старое `total_time`/`mean_time` осталось только до версии 13.
- `EXPLAIN ANALYZE` **выполняет** запрос по-настоящему, поэтому для `UPDATE`/`DELETE`/`INSERT` оборачивайте его в транзакцию с `ROLLBACK`, иначе измените данные.
- Разовый `EXPLAIN ANALYZE` обманчив: первый прогон ловит холодный кэш, повтор уже читает из памяти - сравнивайте несколько запусков.
- `Seq Scan` не всегда зло: на маленькой таблице он быстрее индексного доступа, планировщик это учитывает.
- Сильное расхождение оценочных и фактических `rows` намекает на устаревшую статистику - помогает `ANALYZE`.
