# Чем ORM выгоднее ручного SQL и где её сила

## Суть

ORM (Object-Relational Mapping) превращает строки таблиц в объекты языка, а методы над объектами - в SQL. За счёт этого вы пишете меньше шаблонного кода, реже ошибаетесь в синтаксисе запросов и автоматически получаете защиту от SQL-инъекций.

## Что ответить на собесе

ORM - это слой между приложением и реляционной базой, который сопоставляет классы таблицам, а экземпляры - строкам. Главные плюсы, которые я бы назвал: быстрее разработка, потому что не пишешь руками рутинный CRUD; параметризация запросов из коробки, что закрывает инъекции; переносимость между СУБД; и единая кодовая база на языке приложения вместо смеси с SQL-строками. Минус, о котором честно стоит сказать, - можно потерять контроль над генерируемыми запросами и словить N+1 или неэффективный план, поэтому сложные выборки я всё равно проверяю по реальному SQL.

## Как это работает

ORM строит запросы из операций над объектами. Ниже - модель и работа с ней на SQLAlchemy.

```python
from sqlalchemy import create_engine, Column, Integer, String
from sqlalchemy.orm import declarative_base, sessionmaker

Base = declarative_base()


class User(Base):
    __tablename__ = "users"

    id = Column(Integer, primary_key=True)
    name = Column(String)
    age = Column(Integer)


engine = create_engine("sqlite:///:memory:")
Base.metadata.create_all(engine)

Session = sessionmaker(bind=engine)
session = Session()

session.add(User(name="Alice", age=30))
session.commit()

for user in session.query(User).filter(User.age >= 18):
    print(user.name, user.age)
```

Класс `User` отражает таблицу `users`, а `session.query(...).filter(...)` транслируется в `SELECT ... WHERE age >= 18`. Значение `18` уходит в запрос как параметр, а не вклеивается в строку - отсюда защита от инъекций.

## Подводные камни

- **Проблема N+1.** Обращение к связанным объектам в цикле порождает отдельный запрос на каждую строку. Лечится eager-загрузкой (`select_related`/`prefetch_related` в Django, `joinedload` в SQLAlchemy).
- **Скрытая стоимость абстракции.** Удобный метод может сгенерировать тяжёлый запрос; для горячих мест смотрите итоговый SQL и при необходимости опускайтесь до raw-запроса.
- **Не серебряная пуля для аналитики.** Сложные агрегаты и оконные функции иногда проще и быстрее написать на чистом SQL.
- **Ленивые объекты вне сессии.** После закрытия сессии доступ к ещё не загруженным полям может бросить исключение.
