# The four transaction isolation levels and the anomalies they cut off

## In short

The isolation level determines which intermediate results of neighboring transactions yours can see. The SQL standard defines four tiers — from Read Uncommitted to Serializable; each next one cuts off more read anomalies but costs more in performance.

## How to answer in an interview

Isolation is about which anomalies of concurrent transactions are allowed. There are three anomalies: dirty read (seeing uncommitted changes), non-repeatable read (the same row changes between two SELECTs), and phantoms (new rows appear in a range you've already read). The four levels close them off one by one: Read Uncommitted protects against nothing, Read Committed removes dirty reads, Repeatable Read removes non-repeatable reads too, and Serializable also excludes phantoms, giving behavior as if transactions ran one after another. The stricter the level, the more locks and rollbacks, so the default is usually Read Committed, and you raise the bar only where it's truly needed.

Let's go through the tiers from the bottom up:

1. **Read Uncommitted.** A transaction sees other transactions' not-yet-committed changes. Hence dirty reads: if the neighboring transaction rolls back, you're left with data that never existed. Rarely used, except perhaps for draft reports where garbage doesn't matter.
2. **Read Committed.** Only committed changes are visible, so dirty reads are impossible. But between two SELECTs within the same transaction a row can change — that's a non-repeatable read. The most common level and the default in many DBMSs.
3. **Repeatable Read.** Rows you've read don't change until the end of the transaction — no non-repeatable reads. Phantoms remain: new rows can be added to a range you've already read.
4. **Serializable.** The strictest level: the result is as if transactions ran one at a time. It excludes all three anomalies at the cost of locks, waits, and rollbacks on conflicts.

The level is set before the transaction begins:

```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

BEGIN;

SELECT balance FROM accounts WHERE account_id = 1;

UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;

COMMIT;
```

After `COMMIT`, the changes become visible to other transactions; before that, their visibility is governed by the chosen level.

## Pitfalls

- **The standard describes a minimum of guarantees, not an implementation.** Real DBMSs behave differently: in PostgreSQL, Repeatable Read is snapshot-based and already disallows phantoms, while Read Uncommitted effectively behaves like Read Committed.
- **Repeatable Read in MySQL (InnoDB) and in the standard are different things.** At this level InnoDB suppresses phantoms for current queries via next-key locks, which goes beyond the standard's definition.
- **Higher isolation means a higher risk of deadlocks and rollbacks.** At Serializable, transactions are cancelled more often due to serialization conflicts, and the application must be able to retry them.
- **Isolation does not equal correctness.** Even without the classic anomalies, write skew and lost update are possible; only Serializable reliably protects against write skew.
- **The level is set before the transaction starts working.** You can't change isolation in the middle of an already running transaction.
