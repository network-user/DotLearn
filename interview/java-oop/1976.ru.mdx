# Когда interface лучше abstract class?

## Суть

Interface задаёт контракт поведения без привязки к одной реализации и одному базовому классу. Класс может implement несколько interfaces, что даёт гибкую подмену реализаций и dependency injection без наследования shared state.

## Что ответить на собесе

- Формулирую «can-do» capability: `Payable`, `Serializable`, `Closeable`.
- Подчёркиваю polymorphism через interface type в полях и параметрах.
- Упоминаю Java 8+ default/static methods для эволюции API.
- Сравниваю с abstract class: interface когда нет общего mutable state.
- Честно добавляю: interface не заменяет domain model; анemic interface + god impl тоже антипаттерн.

## Как это работает

```java
public interface SettlementGateway {
    void charge(Money amount);
}

public final class CardGateway implements SettlementGateway {
    @Override
    public void charge(Money amount) {
        System.out.println("card charge " + amount);
    }
}

public final class CheckoutService {
    private final SettlementGateway gateway;

    public CheckoutService(SettlementGateway gateway) {
        this.gateway = gateway;
    }

    public void checkout(Money total) {
        gateway.charge(total);
    }
}
```

Клиент `CheckoutService` не знает про `CardGateway`; в тестах подставляют stub. Новый провайдер добавляется новым class + wiring, без правки checkout logic.

## Подводные камни

- **Fat interface.** `GodService` interface заставляет stub-ить десяток методов.
- **Leaky abstraction.** Interface раскрывает infra детали (SQLException в сигнатуре).
- **Default method diamond.** Несколько interfaces с одинаковым default требуют resolve.
- **implements ради mock only.** Иногда достаточно package-private class + test subclass.
- **Versioning.** Добавление abstract method в public interface ломает все implementors без default.
