# Какую проблему решает инкапсуляция?

## Суть

Инкапсуляция прячет внутреннее состояние объекта и открывает наружу только контролируемый API. В Java это модификаторы доступа и методы вместо прямой работы с полями, что защищает инварианты и позволяет менять реализацию без ломки клиентов.

## Что ответить на собесе

- Формулирую цель: «не дать внешнему коду поставить объект в невалидное состояние».
- Показываю private fields + public методы с validation.
- Связываю с maintainability: можно сменить представление balance (int cents vs BigDecimal) при том же public API.
- Упоминаю JavaBeans/getter-setter как слабую форму; лучше методы с намерением (`debit`, `credit`).
- Честно добавляю: record и immutable DTO сознательно ослабляют инкапсуляцию ради сериализации и простоты; это trade-off, не отказ от принципа.

## Как это работает

```java
public final class Wallet {
    private long balanceMinorUnits;

    public Wallet(long openingMinorUnits) {
        if (openingMinorUnits < 0) {
            throw new IllegalArgumentException("opening balance cannot be negative");
        }
        this.balanceMinorUnits = openingMinorUnits;
    }

    public long balanceMinorUnits() {
        return balanceMinorUnits;
    }

    public void topUp(long amountMinorUnits) {
        requirePositive(amountMinorUnits);
        balanceMinorUnits += amountMinorUnits;
    }

    public boolean charge(long amountMinorUnits) {
        requirePositive(amountMinorUnits);
        if (amountMinorUnits > balanceMinorUnits) {
            return false;
        }
        balanceMinorUnits -= amountMinorUnits;
        return true;
    }

    private static void requirePositive(long value) {
        if (value <= 0) {
            throw new IllegalArgumentException("amount must be positive");
        }
    }
}
```

Клиент не может написать `wallet.balanceMinorUnits = -100`. Любое изменение проходит через методы, которые проверяют правила. Внутри можно перейти на `BigDecimal` или audit log, не трогая вызывающий код.

## Подводные камни

- **Getter/setter без логики.** Публичные сеттеры на все поля формально «инкапсуляция», но по сути открытый struct.
- **Leaky abstraction.** Возврат mutable коллекции из getter (`return items;`) ломает инвариант.
- **protected как public lite.** Наследник может нарушить контракт базового класса.
- **Reflection и serialization.** Инкапсуляция на уровне языка не защищает от обхода через reflection без SecurityManager.
- **Over-exposure.** Слишком широкий public API раскрывает детали, которые должны оставаться internal.
