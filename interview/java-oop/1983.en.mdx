# How does composition differ from inheritance?

## In short

Composition is "has-a": an object holds other objects and delegates work to them. Inheritance is "is-a": a subclass extends the parent's contract and implementation. Composition enables swapping parts without rigid hierarchies; inheritance binds types at compile time.

## How to answer in an interview

- State favor composition over inheritance for reusing behavior.
- Show that components can be injected and mocked.
- Explain lifecycle: composition often creates/owns parts (strong composition).
- Compare with the delegation pattern.
- Be honest: inheritance fits true subtype polymorphism, not reuse alone.

## How it works

```java
final class Motor { void spin() { System.out.println("motor spin"); } }
final class AxleSet { void roll() { System.out.println("axles roll"); } }

final class Hauler {
    private final Motor motor;
    private final AxleSet axles;

    Hauler(Motor motor, AxleSet axles) {
        this.motor = motor;
        this.axles = axles;
    }

    void move() {
        motor.spin();
        axles.roll();
    }
}
```

`Hauler` does not extend `Motor`; tests can substitute another motor. An inheritance design `class Hauler extends Motor` mixes unrelated concerns and blocks replacement.

## Pitfalls

- **Composition without injection.** `new Motor()` inside the class is still hard coupling.
- **Leaking internal component.** A getter exposes a mutable part.
- **God container.** One class aggregates everything without boundaries.
- **Inheritance for reuse only.** Empty extends for one helper method.
- **Circular composition.** A holds B holds A; lifecycle and GC get harder.
