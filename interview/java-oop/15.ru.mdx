# Какие столпы лежат в основе ООП?

## Суть

Объектно-ориентированное программирование строится на четырёх идеях: инкапсуляция, наследование, полиморфизм и абстракция. Вместе они помогают собирать данные и поведение в объекты, скрывать детали реализации и переиспользовать код через иерархии типов.

## Что ответить на собесе

- Называю четыре принципа и связываю каждый с практической задачей: защита инвариантов, повторное использование, подмена реализаций, упрощение модели.
- Привожу короткий Java-пример с приватным состоянием и публичным API вместо «голых» полей.
- Объясняю полиморфизм через ссылку на базовый тип и переопределение метода, а не только через ключевое слово `extends`.
- Уточняю, что абстракция в Java выражается через abstract class и interface, и выбор между ними зависит от того, нужен ли общий код или только контракт.
- Честно добавляю: слепое наследование глубоких иерархий часто хуже композиции; на собесе важно показать, что понимаю trade-off.

## Как это работает

**Инкапсуляция** объединяет состояние и операции внутри класса и закрывает поля от прямого доступа. Внешний код работает через методы, которые могут проверять вход и поддерживать инварианты.

```java
public final class LedgerAccount {
    private long balanceCents;

    public LedgerAccount(long openingBalanceCents) {
        if (openingBalanceCents < 0) {
            throw new IllegalArgumentException("opening balance must be non-negative");
        }
        this.balanceCents = openingBalanceCents;
    }

    public void credit(long amountCents) {
        requirePositive(amountCents);
        balanceCents += amountCents;
    }

    public void debit(long amountCents) {
        requirePositive(amountCents);
        if (amountCents > balanceCents) {
            throw new IllegalStateException("insufficient funds");
        }
        balanceCents -= amountCents;
    }

    public long balanceCents() {
        return balanceCents;
    }

    private static void requirePositive(long amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("amount must be positive");
        }
    }
}
```

**Наследование** позволяет специализировать общий тип. Подкласс расширяет или переопределяет поведение родителя, не дублируя базовую логику.

```java
public class DomesticCanine extends Canine {
    @Override
    public String vocalize() {
        return "woof";
    }
}
```

**Полиморфизм** даёт вызывать один и тот же метод у объектов разных классов через общий тип. JVM выбирает реализацию во время выполнения (virtual dispatch).

```java
Canine companion = new DomesticCanine();
System.out.println(companion.vocalize());
```

**Абстракция** отделяет контракт от деталей. Абстрактный класс или интерфейс описывает, *что* делает сущность; конкретные классы решают *как*.

```java
public interface Transport {
    void depart(String destination);
}

public final class RoadVehicle implements Transport {
    @Override
    public void depart(String destination) {
        System.out.println("Driving toward " + destination);
    }
}
```

Вместе эти приёмы помогают проектировать модульные системы: изменения локализуются внутри класса, а клиентский код опирается на стабильные контракты.

## Подводные камни

- **Нарушение инкапсуляции.** Публичные mutable-поля ломают инварианты; даже `protected` часто протекает наружу через наследников.
- **Хрупкая иерархия.** Глубокое наследование связывает подкласс с деталями родителя; рефакторинг базового класса ломает всех потомков.
- **Путаница override/overload.** Перегрузка выбирается на этапе компиляции, переопределение на рантайме; без `@Override` легко случайно скрыть метод.
- **Абстракция ради абстракции.** Лишние интерфейсы без второй реализации усложняют код без выигрыша в тестируемости.
- **Полиморфизм без равенства контрактов.** Разные реализации одного интерфейса должны соблюдать одинаковые ожидания по pre/post conditions, иначе LSP нарушается.
