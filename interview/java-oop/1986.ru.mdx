# Какие правила нужны для immutable-класса?

## Суть

Immutable-класс не меняет состояние после создания. Базовый чеклист: `final` класс (или контролируемое наследование), все поля `private final`, инициализация только в конструкторе, без сеттеров, defensive copy для mutable полей на входе и в геттерах. Такие объекты безопасны для concurrent-кода без locks.

## Что ответить на собесе

- Перечисляю шесть правил: final class, private final fields, ctor init, no setters, deep copy in, copy out.
- Упоминаю `String`, `Integer`, `LocalDate` как эталоны JDK.
- Объясняю, почему наследование опасно: подкласс может добавить mutability.
- Связываю с thread safety: publish immutable reference без синхронизации.
- Честно добавляю: полная immutability дороже по памяти и копированию; для hot path иногда выбирают unmodifiable view.

## Как это работает

```java
public final class RouteSnapshot {
    private final String routeId;
    private final List<String> stops;

    public RouteSnapshot(String routeId, List<String> stops) {
        this.routeId = routeId;
        this.stops = List.copyOf(stops);
    }

    public String routeId() {
        return routeId;
    }

    public List<String> stops() {
        return stops;
    }
}
```

Конструктор копирует входной список через `List.copyOf`, внешний mutable list не протекает внутрь. Геттер возвращает неизменяемый view: вызывающий код не может изменить внутреннее состояние. Поле `routeId` immutable по природе (`String`). Класс `final` блокирует подкласс, который мог бы открыть protected mutator. Любое «обновление» делается созданием нового `RouteSnapshot`, а не мутацией существующего.

## Подводные камни

- **Shallow copy в ctor.** Сохранили ссылку на переданный `Date[]` или `ArrayList`.
- **Утечка через getter.** Возврат internal mutable collection без обёртки.
- **Наследование без final.** Подкласс добавляет mutable field или override с side effect.
- **Lazy init поля.** Double-checked locking без `volatile` ломает immutability guarantee.
- **Serializable mutable graph.** Десериализация восстанавливает изменяемые ссылки внутри «immutable» объекта.
