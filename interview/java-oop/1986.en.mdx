# What rules do you need for an immutable class?

## In short

An immutable class does not change state after construction. The basic checklist: a `final` class (or controlled inheritance), all fields `private final`, initialization only in the constructor, no setters, and defensive copies for mutable fields on input and in getters. Such objects are safe for concurrent code without locks.

## How to answer in an interview

- I list six rules: final class, private final fields, ctor init, no setters, deep copy in, copy out.
- I mention `String`, `Integer`, and `LocalDate` as JDK reference types.
- I explain why inheritance is risky: a subclass can add mutability.
- I connect it to thread safety: you can publish an immutable reference without synchronization.
- I add honestly: full immutability costs more memory and copying; on a hot path I sometimes choose an unmodifiable view instead.

## How it works

```java
public final class RouteSnapshot {
    private final String routeId;
    private final List<String> stops;

    public RouteSnapshot(String routeId, List<String> stops) {
        this.routeId = routeId;
        this.stops = List.copyOf(stops);
    }

    public String routeId() {
        return routeId;
    }

    public List<String> stops() {
        return stops;
    }
}
```

The constructor copies the input list with `List.copyOf`, so an external mutable list cannot leak inside. The getter returns an unmodifiable view: callers cannot change internal state. The `routeId` field is immutable by nature (`String`). A `final` class blocks a subclass that might expose a protected mutator. Any "update" is done by creating a new `RouteSnapshot`, not by mutating the existing one.

## Pitfalls

- **Shallow copy in the ctor.** You keep a reference to the passed `Date[]` or `ArrayList`.
- **Leak through a getter.** Returning an internal mutable collection without wrapping.
- **Inheritance without final.** A subclass adds a mutable field or an override with side effects.
- **Lazy-init field.** Double-checked locking without `volatile` breaks the immutability guarantee.
- **Serializable mutable graph.** Deserialization restores mutable references inside an "immutable" object.
