# What is method overriding in Java?

## In short

Override means a subclass supplies a new implementation of an instance method with the same signature as a supertype method. The JVM picks the version from the object's runtime type (virtual dispatch), which enables runtime polymorphism.

## How to answer in an interview

- Contrast override vs overload: overload is compile-time, override is runtime.
- Requirements: same name and parameters, compatible (covariant) return type, not weaker access.
- Recommend `@Override` for a compile-time check.
- Mention `super.method()` to extend rather than replace behavior.
- Be honest: private/static/final methods do not override; static hiding is a different story.

## How it works

```java
class Alert {
    void signal() {
        System.out.println("generic alert");
    }
}

class SirenAlert extends Alert {
    @Override
    void signal() {
        System.out.println("siren pulse");
    }
}

public final class OverrideDemo {
    public static void main(String[] args) {
        Alert base = new SirenAlert();
        base.signal();
    }
}
```

Static type is `Alert`, runtime type is `SirenAlert`, output is the siren line. Without `@Override`, a signature typo creates an overload and silently breaks polymorphism.

## Pitfalls

- **Accidental overload.** Changing a parameter type thinking it is an override.
- **Weaker access.** protected/public mismatch will not compile.
- **Wrong equals signature.** `equals(OtherType)` does not override `equals(Object)`.
- **Overriding static methods.** Static hiding, not polymorphism; behavior confuses teams.
- **super call order.** Wrong constructor + override order can observe zero state.
