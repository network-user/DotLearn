# What is an object in Java and what does it contain?

## In short

An object is a class instance on the heap: it has state (fields), behavior (methods), and identity (reference or memory address). A class is the blueprint; `new` creates a concrete instance with its own field values.

## How to answer in an interview

- Separate class (type description) from object (runtime instance).
- Name the three parts: state, behavior, identity.
- Show creation through a constructor and use through a reference variable.
- Clarify that two objects with identical fields are not identical unless `equals` says so.
- Be honest: value objects (records, immutable types) intentionally favor value equality over identity.

## How it works

```java
public final class FleetVehicle {
    private String paintColor;
    private final String modelName;
    private final int modelYear;

    public FleetVehicle(String paintColor, String modelName, int modelYear) {
        this.paintColor = paintColor;
        this.modelName = modelName;
        this.modelYear = modelYear;
    }

    public void igniteEngine() {
        System.out.println(modelName + " engine on");
    }

    public void repaint(String paintColor) {
        this.paintColor = paintColor;
    }

    public String describe() {
        return modelYear + " " + modelName + " (" + paintColor + ")";
    }
}

public final class FleetDemo {
    public static void main(String[] args) {
        FleetVehicle unitA = new FleetVehicle("silver", "Courier", 2022);
        unitA.igniteEngine();
        unitA.repaint("navy");
        System.out.println(unitA.describe());
    }
}
```

Variable `unitA` holds a reference to a heap object. Methods receive implicit `this` and operate on that instance's fields. The JVM allocates memory, runs the constructor, and GC later reclaims the object when no reachable references remain.

Objects model domain entities, encapsulate invariants, and let one class spawn many instances with different state.

## Pitfalls

- **Reference vs object confusion.** Assignment copies the reference, not a clone of the object.
- **== vs equals.** `==` compares identity; values need a proper equals/hashCode contract.
- **Leaking this from constructor.** Publishing `this` early exposes a partially constructed object.
- **Mutable shared state.** One object referenced from many owners needs synchronization or immutability.
- **Anemic model.** Objects with only getters/setters and no behavior become DTOs and weaken OOP benefits.
