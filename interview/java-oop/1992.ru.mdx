# Когда выбрать абстрактный класс, а когда интерфейс?

## Суть

Абстрактный класс объединяет общее состояние и частичную реализацию для узкого семейства типов (single inheritance). Интерфейс задаёт контракт capability без привязки к иерархии и допускает multiple implements. Выбор: shared fields/ctor/template method → abstract class; cross-cutting role, plug-in API, несколько несвязанных реализаций → interface.

## Что ответить на собесе

- Abstract class: поля, конструктор, non-abstract helper methods, один extends.
- Interface: контракт, polymorphism, несколько roles на классе, default/static с Java 8.
- Упоминаю template method pattern в abstract base.
- С Java 8 interface с defaults ближе к abstract, но state в интерфейсе по-прежнему ограничен.
- Честно добавляю: в greenfield сервисах чаще composition + interface; deep abstract hierarchies сложно эволюционировать.

## Как это работает

Абстрактный базовый класс с общим состоянием:

```java
abstract class LedgerEntry {
    private final String entryId;
    private final Instant createdAt;

    protected LedgerEntry(String entryId) {
        this.entryId = entryId;
        this.createdAt = Instant.now();
    }

    abstract Money amount();

    final void audit() {
        System.out.println(entryId + " at " + createdAt + " => " + amount());
    }
}
```

Интерфейс как capability для разных иерархий:

```java
interface Exportable {
    byte[] toPayload();
}

final class InvoiceEntry extends LedgerEntry implements Exportable {
    private final Money total;

    InvoiceEntry(String entryId, Money total) {
        super(entryId);
        this.total = total;
    }

    @Override
    Money amount() {
        return total;
    }

    @Override
    public byte[] toPayload() {
        return total.toString().getBytes(StandardCharsets.UTF_8);
    }
}
```

`LedgerEntry` хранит `entryId` и шаблон `audit()`. `Exportable` подключается к любому классу, не только к ledger. `InvoiceEntry` наследует один abstract class и добавляет второй контракт через interface.

## Подводные камни

- **Abstract class как namespace.** Только static helpers без общего state.
- **Interface с десятком defaults.** Фактически abstract base без полей, но с diamond риском.
- **Неверный extends.** Наследование ради одного util method вместо composition.
- **Tight coupling к base.** Protected fields доступны всем потомкам без границ.
- **Смешение роли и иерархии.** `implements Serializable, Cloneable, Comparable` без осмысленного дизайна.
