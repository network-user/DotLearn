# Which methods does every object inherit from Object?

## In short

Class `Object` supplies core operations: equality (`equals`, `hashCode`), string form (`toString`), reflection (`getClass`), cloning (`clone`), finalization (`finalize`, deprecated), and thread coordination (`wait`, `notify`, `notifyAll`). Most types override for domain semantics or keep defaults deliberately.

## How to answer in an interview

- List key methods and why they matter in collections and logging.
- Stress the equals/hashCode contract for `HashMap`/`HashSet`.
- Clarify that `getClass` is final; `clone` is protected and needs `Cloneable`.
- Note `finalize` is deprecated since 9; prefer `Cleaner` or try-with-resources.
- Be honest: for value types I prefer `record`, which generates equals/hashCode/toString.

## How it works

```java
public final class SkuRef {
    private final String code;

    public SkuRef(String code) {
        this.code = code;
    }

    @Override
    public boolean equals(Object other) {
        if (this == other) return true;
        if (!(other instanceof SkuRef that)) return false;
        return code.equals(that.code);
    }

    @Override
    public int hashCode() {
        return code.hashCode();
    }

    @Override
    public String toString() {
        return "SkuRef(" + code + ")";
    }
}
```

`equals` and `hashCode` must agree: equal objects share the same hash. `toString` helps logs and debuggers. `getClass()` returns the runtime class (useful in strict equals). Monitor methods (`wait`/`notify`) use each instance's intrinsic lock.

## Pitfalls

- **equals without hashCode.** Breaks hash-based collections.
- **Subclassing equals.** Symmetry breaks if a subclass adds fields without a getClass guard.
- **Shallow clone.** Mutable fields still alias; prefer deep copy or a copy constructor.
- **toString leaking secrets.** Passwords and tokens must not appear in logs.
- **wait/notify in domain code.** Easy deadlocks; prefer j.u.c utilities.
