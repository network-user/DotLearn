# Runnable or Callable: which fits the task?

## In short

`Runnable` describes a fire-and-forget task with no return value; `Callable<V>` returns a result and may throw checked exceptions. Both run work on another thread or in a pool, but only `Callable` integrates with `Future` to fetch values and handle failures.

## How to answer in an interview

- Compare signatures: `void run()` vs `V call() throws Exception`.
- Explain that `Thread` accepts `Runnable`, while `ExecutorService.submit(Callable)` returns a `Future`.
- Clarify exceptions: checked exceptions cannot escape a plain `Runnable` without wrapping; `Callable` surfaces them through `ExecutionException`.
- Note that modern code often uses lambdas and `ExecutorService` instead of manual `new Thread`.
- Be honest: background logging can stay on `Runnable`; once you need a result or timeout, switch to `Callable`/`CompletableFuture`.

## How it works

`Runnable` fits side effects without a reply:

```java
Runnable auditTrail = () -> System.out.println("audit: user logged in");

Thread worker = new Thread(auditTrail);
worker.start();
```

`Callable` returns a value and pairs with a pool:

```java
import java.util.concurrent.*;

public final class ReportJob {
    public static void main(String[] args) throws Exception {
        ExecutorService pool = Executors.newFixedThreadPool(2);

        Callable<Long> countRows = () -> {
            Thread.sleep(50);
            return 1280L;
        };

        Future<Long> pending = pool.submit(countRows);
        long total = pending.get(2, TimeUnit.SECONDS);

        System.out.println("rows=" + total);
        pool.shutdown();
    }
}
```

`Future.get()` blocks the caller until completion or timeout. Errors from `call()` arrive wrapped in `ExecutionException`. For task chains, `CompletableFuture.supplyAsync` offers Callable-like semantics with composition.

## Pitfalls

- **Lost stack trace in Runnable.** Checked exceptions inside a lambda must be handled locally or diagnostics disappear.
- **Blocking get without timeout.** `future.get()` can hang forever if the task stalls.
- **Shutdown without await.** `shutdown()` does not wait; use `awaitTermination` or try-with-resources for `ExecutorService`.
- **New Thread per Runnable.** Without a pool you pay heavily on context switches.
- **Ignoring InterruptedException.** On cancellation, restore the thread interrupt flag.
