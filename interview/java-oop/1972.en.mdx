# How do you work around Java's ban on multiple inheritance?

## In short

Java forbids extending multiple classes because of the diamond problem, but a class may **implement** several interfaces and **compose** behavior through fields. That gives multiple contracts and code reuse without inheriting multiple implementations.

## How to answer in an interview

- Name two paths: multiple interfaces with default methods, and composition (has-a).
- Explain diamond conflicts: two defaults with the same signature need explicit resolution.
- Show that class inheritance is single; the rest comes from implements.
- Recommend composition over inheritance for shared implementation.
- Be honest: interfaces with defaults do not replace a base class with protected state; use an abstract class or delegation.

## How it works

Interfaces supply multiple types:

```java
interface Printable { void print(); }
interface Auditable { void audit(); }

final class InvoiceDocument implements Printable, Auditable {
    @Override
    public void print() { System.out.println("invoice pdf"); }

    @Override
    public void audit() { System.out.println("invoice audited"); }
}
```

Composition bundles components:

```java
final class Engine { void ignite() { System.out.println("engine on"); } }
final class Chassis { void roll() { System.out.println("wheels move"); } }

final class DeliveryVan {
    private final Engine engine = new Engine();
    private final Chassis chassis = new Chassis();

    void depart() {
        engine.ignite();
        chassis.roll();
    }
}
```

You can mix one abstract base plus several interfaces when you need both shared code and extra contracts.

## Pitfalls

- **Stateful interface defaults.** Defaults cannot hold per-instance fields; mutable state stays in the class.
- **God interface.** Ten interfaces on one class bloats API and tests.
- **Inheritance for reuse only.** Extending for one helper method is better as a utility or composition.
- **Unresolved diamond.** The compiler forces `IfaceA.super.action()`.
- **LSP breaks.** Polymorphic interfaces do not guarantee equal semantics across implementations.
