# Чем composition отличается от inheritance?

## Суть

Composition это «has-a»: объект содержит другие объекты и делегирует им работу. Inheritance это «is-a»: подкласс расширяет контракт и реализацию родителя. Composition даёт гибкую замену частей без жёсткой иерархии; inheritance связывает типы на compile time.

## Что ответить на собесе

- Формулирую favor composition over inheritance для reuse behavior.
- Показываю, что компоненты можно inject и mock.
- Объясняю lifecycle: composition часто создаёт/владеет parts (strong composition).
- Сравниваю с delegation pattern.
- Честно добавляю: inheritance уместен для true subtype polymorphism, не для code reuse only.

## Как это работает

```java
final class Motor { void spin() { System.out.println("motor spin"); } }
final class AxleSet { void roll() { System.out.println("axles roll"); } }

final class Hauler {
    private final Motor motor;
    private final AxleSet axles;

    Hauler(Motor motor, AxleSet axles) {
        this.motor = motor;
        this.axles = axles;
    }

    void move() {
        motor.spin();
        axles.roll();
    }
}
```

`Hauler` не extends `Motor`; можно подставить другой `Motor` в test или swap implementation. Inheritance version `class Hauler extends Motor` смешивает unrelated concerns и ломает замену.

## Подводные камни

- **Composition without injection.** `new Motor()` внутри класса всё ещё жёсткая связь.
- **Leaking internal component.** Getter возвращает mutable part наружу.
- **God object container.** Один class агрегирует всё подряд без границ.
- **Inheritance for reuse only.** Empty extends ради одного helper method.
- **Circular composition.** A holds B holds A; lifecycle и GC сложнее.
