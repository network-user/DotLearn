# Why make a class immutable?

## In short

An immutable object never changes after construction: fields are final (or effectively final), there are no setters, and mutable references never escape. That simplifies reasoning, makes the object thread-safe without locks, and reduces accidental mutation when passing it across layers.

## How to answer in an interview

- List benefits: thread safety, predictability, safe `HashMap` keys, cacheable hashCode.
- Give the recipe: final class, final fields, defensive copy in/out, no setters.
- Mention `String`, boxed numbers, the Java time API, and `record` as examples.
- Explain "changes" through new instances (`withAmount`) instead of mutation.
- Be honest: immutability is not free for large structures; frequent small edits may allocate many objects.

## How it works

```java
public final class Money {
    private final long minorUnits;
    private final String currency;

    public Money(long minorUnits, String currency) {
        if (minorUnits < 0) throw new IllegalArgumentException("negative amount");
        this.minorUnits = minorUnits;
        this.currency = Objects.requireNonNull(currency);
    }

    public long minorUnits() { return minorUnits; }
    public String currency() { return currency; }

    public Money add(Money other) {
        requireSameCurrency(other);
        return new Money(minorUnits + other.minorUnits, currency);
    }

    private void requireSameCurrency(Money other) {
        if (!currency.equals(other.currency)) {
            throw new IllegalArgumentException("currency mismatch");
        }
    }
}
```

`add` returns a new `Money`; the original stays unchanged. You can share it across threads and use it as a map key with proper equals/hashCode.

## Pitfalls

- **Mutable field or leaked array.** A `final Date` or `final List` without `List.copyOf` breaks immutability.
- **Subclass adds mutation.** A non-final class can introduce mutable state.
- **Unsynchronized lazy cache.** A mutable cache field needs safe publication.
- **Serialization proxy.** Deserialization can bypass finals without `readResolve`.
- **Hot-path performance.** Copying large collections often; consider persistent structures or builder snapshots.
