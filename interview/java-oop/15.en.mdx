# What pillars underpin OOP?

## In short

Object-oriented programming rests on four ideas: encapsulation, inheritance, polymorphism, and abstraction. Together they group data and behavior into objects, hide implementation details, and reuse code through type hierarchies.

## How to answer in an interview

- Name all four principles and tie each to a practical goal: protecting invariants, reuse, swapping implementations, and simplifying the model.
- Show a short Java example with private state and a public API instead of exposed fields.
- Explain polymorphism through a base-type reference and method overriding, not only the keyword `extends`.
- Clarify that abstraction in Java shows up as abstract classes and interfaces, and the choice depends on whether shared code or only a contract is needed.
- Be honest: deep inheritance hierarchies are often worse than composition; interviewers want to see you understand the trade-off.

## How it works

**Encapsulation** keeps state and operations inside a class and blocks direct field access. External code goes through methods that can validate input and preserve invariants.

```java
public final class LedgerAccount {
    private long balanceCents;

    public LedgerAccount(long openingBalanceCents) {
        if (openingBalanceCents < 0) {
            throw new IllegalArgumentException("opening balance must be non-negative");
        }
        this.balanceCents = openingBalanceCents;
    }

    public void credit(long amountCents) {
        requirePositive(amountCents);
        balanceCents += amountCents;
    }

    public void debit(long amountCents) {
        requirePositive(amountCents);
        if (amountCents > balanceCents) {
            throw new IllegalStateException("insufficient funds");
        }
        balanceCents -= amountCents;
    }

    public long balanceCents() {
        return balanceCents;
    }

    private static void requirePositive(long amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("amount must be positive");
        }
    }
}
```

**Inheritance** lets you specialize a common type. A subclass extends or overrides parent behavior without duplicating base logic.

```java
public class DomesticCanine extends Canine {
    @Override
    public String vocalize() {
        return "woof";
    }
}
```

**Polymorphism** lets you call the same method on objects of different classes through a shared type. The JVM picks the implementation at runtime (virtual dispatch).

```java
Canine companion = new DomesticCanine();
System.out.println(companion.vocalize());
```

**Abstraction** separates contract from detail. An abstract class or interface describes *what* an entity does; concrete classes decide *how*.

```java
public interface Transport {
    void depart(String destination);
}

public final class RoadVehicle implements Transport {
    @Override
    public void depart(String destination) {
        System.out.println("Driving toward " + destination);
    }
}
```

Used together, these techniques support modular systems: changes stay local inside a class while client code depends on stable contracts.

## Pitfalls

- **Broken encapsulation.** Public mutable fields break invariants; even `protected` often leaks through subclasses.
- **Fragile hierarchy.** Deep inheritance ties subclasses to parent internals; refactoring the base class breaks every descendant.
- **Override vs overload confusion.** Overloading is resolved at compile time, overriding at runtime; without `@Override` you can accidentally hide a method.
- **Abstraction for its own sake.** Extra interfaces with no second implementation add complexity without testability gains.
- **Polymorphism without equal contracts.** Different implementations of one interface must honor the same pre/post conditions or LSP breaks down.
