# When do you choose an abstract class?

## In short

Pick an abstract class when several subtypes need **shared code and state** plus mandatory extension points through abstract methods. It is a single-inheritance base with constructors, fields, and template methods, not a pure contract.

## How to answer in an interview

- Contrast with interface: abstract class when you have shared implementation, not just signatures.
- Name the Template Method pattern as a typical case.
- Note the limit: only one abstract class in extends chain.
- Explain abstract class plus interfaces when you need a base plus extra role types.
- Be honest: if the hierarchy is unstable, composition plus interfaces is often easier to evolve.

## How it works

```java
abstract class Vehicle {
    protected final int maxSpeedKmh;

    protected Vehicle(int maxSpeedKmh) {
        this.maxSpeedKmh = maxSpeedKmh;
    }

    abstract void propel();

    void reportSpeedCap() {
        System.out.println("cap=" + maxSpeedKmh + " km/h");
    }
}

final class CargoTruck extends Vehicle {
    CargoTruck(int maxSpeedKmh) {
        super(maxSpeedKmh);
    }

    @Override
    void propel() {
        System.out.println("diesel torque applied");
    }
}
```

`reportSpeedCap` is inherited by every subtype; each implements `propel` differently. Field `maxSpeedKmh` is initialized through the protected base constructor.

## Pitfalls

- **Abstract class as namespace.** Prefer a utility class for static helpers without inheritance.
- **Fragile base class.** Changing a protected method breaks all subclasses.
- **Single extends slot.** You cannot extend another rich base.
- **Too much concrete logic in the base.** Subclasses cannot reuse without heavy overrides.
- **Testing via anonymous subclass.** Prefer a small test double in the test source set.
