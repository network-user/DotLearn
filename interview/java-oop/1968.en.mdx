# What problem does encapsulation solve?

## In short

Encapsulation hides an object's internal state and exposes only a controlled API. In Java that means access modifiers and methods instead of direct field access, protecting invariants and letting you change implementation without breaking clients.

## How to answer in an interview

- State the goal: "prevent external code from putting the object into an invalid state."
- Show private fields plus public methods with validation.
- Tie to maintainability: you can switch balance representation (int cents vs BigDecimal) while keeping the same public API.
- Mention JavaBeans getters/setters as a weak form; prefer intention-revealing methods like `debit` and `credit`.
- Be honest: records and immutable DTOs deliberately relax encapsulation for serialization and simplicity; that is a trade-off, not abandoning the idea.

## How it works

```java
public final class Wallet {
    private long balanceMinorUnits;

    public Wallet(long openingMinorUnits) {
        if (openingMinorUnits < 0) {
            throw new IllegalArgumentException("opening balance cannot be negative");
        }
        this.balanceMinorUnits = openingMinorUnits;
    }

    public long balanceMinorUnits() {
        return balanceMinorUnits;
    }

    public void topUp(long amountMinorUnits) {
        requirePositive(amountMinorUnits);
        balanceMinorUnits += amountMinorUnits;
    }

    public boolean charge(long amountMinorUnits) {
        requirePositive(amountMinorUnits);
        if (amountMinorUnits > balanceMinorUnits) {
            return false;
        }
        balanceMinorUnits -= amountMinorUnits;
        return true;
    }

    private static void requirePositive(long value) {
        if (value <= 0) {
            throw new IllegalArgumentException("amount must be positive");
        }
    }
}
```

Clients cannot write `wallet.balanceMinorUnits = -100`. Every change flows through methods that enforce rules. Internally you could move to `BigDecimal` or add audit logging without touching callers.

## Pitfalls

- **Getters/setters without logic.** Public setters on every field look like encapsulation but behave like an open struct.
- **Leaky abstraction.** Returning a mutable collection from a getter (`return items;`) breaks invariants.
- **protected as public lite.** Subclasses can violate the base class contract.
- **Reflection and serialization.** Language-level encapsulation does not stop reflection unless you add security controls.
- **Over-exposure.** A too-wide public API reveals details that should stay internal.
