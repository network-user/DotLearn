# Can you override a static method in Java?

## In short

Static methods in Java are not overridden. They bind to the reference type at compile time, not to the runtime type of the object. A subclass may declare a static method with the same signature, but that is hiding, not polymorphism.

## How to answer in an interview

- I clearly separate override (instance, virtual dispatch) from static hiding.
- I explain early binding: a call to `ref.staticMethod()` resolves from the declared type of the reference.
- I show that `@Override` on a static method does not compile.
- I compare it with an instance method on the same hierarchy for contrast.
- I add honestly: hiding is easy to confuse with override; in production I avoid static methods in class hierarchies.

## How it works

```java
class BaseNode {
    static void report() {
        System.out.println("base report");
    }

    void describe() {
        System.out.println("base describe");
    }
}

class LeafNode extends BaseNode {
    static void report() {
        System.out.println("leaf report");
    }

    @Override
    void describe() {
        System.out.println("leaf describe");
    }
}

class StaticDispatchDemo {
    public static void main(String[] args) {
        BaseNode ref = new LeafNode();

        BaseNode.report();
        LeafNode.report();

        ref.report();
        ref.describe();
    }
}
```

Calls through the class (`BaseNode.report()`, `LeafNode.report()`) pick the method of the declared type. `ref.report()` prints `base report` even though the object is a `LeafNode`, because static dispatch does not use dynamic dispatch. `ref.describe()` prints `leaf describe` thanks to override. Polymorphism in Java applies to instance methods; static methods stay class-level utilities.

## Pitfalls

- **False expectation of polymorphism.** `parentRef.staticHelper()` always calls the version from the reference type.
- **Hidden bug during refactoring.** You rename static in the parent, but the child still hides the old name.
- **@Override on static.** The compiler rejects it; that is a sign the model is wrong.
- **Tests through a subclass instance.** Static behavior is not verified by a polymorphic call.
- **Static in a deep hierarchy.** It is hard to track which version a generic API invokes.
