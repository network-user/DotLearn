# В каких случаях выбирают abstract class?

## Суть

Abstract class выбирают, когда нескольким подтипам нужен **общий код и состояние**, плюс обязательные точки расширения через abstract methods. Это single-inheritance база с конструктором, fields и template methods, а не чистый контракт.

## Что ответить на собесе

- Сравниваю с interface: abstract class когда есть shared implementation, не только сигнатуры.
- Называю Template Method pattern как типичный кейс.
- Уточняю ограничение: extends только один abstract class.
- Объясняю, когда abstract class + interfaces вместе (база + дополнительные role interfaces).
- Честно добавляю: если иерархия не стабильна, composition + interface часто проще эволюционировать.

## Как это работает

```java
abstract class Vehicle {
    protected final int maxSpeedKmh;

    protected Vehicle(int maxSpeedKmh) {
        this.maxSpeedKmh = maxSpeedKmh;
    }

    abstract void propel();

    void reportSpeedCap() {
        System.out.println("cap=" + maxSpeedKmh + " km/h");
    }
}

final class CargoTruck extends Vehicle {
    CargoTruck(int maxSpeedKmh) {
        super(maxSpeedKmh);
    }

    @Override
    void propel() {
        System.out.println("diesel torque applied");
    }
}
```

`reportSpeedCap` наследуется всеми подтипами; `propel` каждый реализует по-своему. Поле `maxSpeedKmh` инициализируется через protected constructor базы.

## Подводные камни

- **Abstract class как namespace.** Статические helpers лучше в utility class без наследования.
- **Fragile base class.** Изменение protected метода ломает все subclasses.
- **Нельзя extends + другой rich base.** Единственный слот наследования занят.
- **Слишком много concrete logic в base.** Подклассы не могут переиспользовать без copy-paste override.
- **Тестирование abstract через anonymous class.** Лучше маленький test double subclass в test source set.
