# Как в Java обойти запрет множественного наследования?

## Суть

Java запрещает extends нескольких классов из-за diamond problem, но класс может **implement** несколько interfaces и **компоновать** поведение через поля. Так получают несколько контрактов и переиспользование кода без множественного наследования реализации.

## Что ответить на собесе

- Называю два пути: multiple interfaces + default methods, composition (has-a).
- Объясняю diamond: два default-метода с одной сигнатурой требуют явного resolve.
- Показываю, что наследование класса только одно, остальное через implements.
- Рекомендую «composition over inheritance» для shared implementation.
- Честно добавляю: interface + default не заменяет базовый класс с protected state; для этого abstract class или делегирование.

## Как это работает

Интерфейсы дают несколько типов:

```java
interface Printable { void print(); }
interface Auditable { void audit(); }

final class InvoiceDocument implements Printable, Auditable {
    @Override
    public void print() { System.out.println("invoice pdf"); }

    @Override
    public void audit() { System.out.println("invoice audited"); }
}
```

Композиция комбинирует готовые компоненты:

```java
final class Engine { void ignite() { System.out.println("engine on"); } }
final class Chassis { void roll() { System.out.println("wheels move"); } }

final class DeliveryVan {
    private final Engine engine = new Engine();
    private final Chassis chassis = new Chassis();

    void depart() {
        engine.ignite();
        chassis.roll();
    }
}
```

Можно смешивать: один базовый abstract class + несколько interfaces, если нужен и shared code, и дополнительные контракты.

## Подводные камни

- **Interface с state через default.** Default не держит per-instance fields кроме static; mutable state всё равно в классе.
- **God interface.** Десять interfaces на класс усложняют API и тесты.
- **Наследование ради reuse.** extends ради одного helper-метода лучше заменить utility class или composition.
- **Diamond без resolve.** Компилятор потребует `IfaceA.super.action()`.
- **LSP нарушения.** Полиморфный interface не гарантирует одинаковую семантику реализаций.
