# How do you design an immutable class in practice?

## In short

An immutable class fixes state at construction time and provides no way to change it. A practical implementation uses a `final` class, `private final` fields, a constructor with defensive copying, getters that do not leak mutable references, and no setters. Updates are modeled as creating a new instance (with-methods or a builder).

## How to answer in an interview

- I show the pattern on a value object with primitives and a collection.
- I explain `clone()` on input and `List.copyOf` / `Collections.unmodifiableList` on output.
- I mention Java 16+ `record` for simple immutable DTOs.
- I connect it to concurrency: safe publication and use as a `Map` key.
- I add honestly: immutability does not replace validation; invalid state can be frozen forever in the ctor.

## How it works

```java
public final class GeoPoint {
    private final int latitudeE6;
    private final int longitudeE6;

    public GeoPoint(int latitudeE6, int longitudeE6) {
        if (latitudeE6 < -90_000_000 || latitudeE6 > 90_000_000) {
            throw new IllegalArgumentException("latitude out of range");
        }
        if (longitudeE6 < -180_000_000 || longitudeE6 > 180_000_000) {
            throw new IllegalArgumentException("longitude out of range");
        }
        this.latitudeE6 = latitudeE6;
        this.longitudeE6 = longitudeE6;
    }

    public int latitudeE6() {
        return latitudeE6;
    }

    public int longitudeE6() {
        return longitudeE6;
    }

    public GeoPoint shift(int deltaLatE6, int deltaLonE6) {
        return new GeoPoint(latitudeE6 + deltaLatE6, longitudeE6 + deltaLonE6);
    }
}
```

All fields are set once in the constructor with range checks. `shift` does not mutate `this`; it returns a new `GeoPoint`. External code cannot assign new values to fields: there are no setters and fields are `final`. Collections follow the same rule: copy on input, unmodifiable view on read. `String` and boxed primitives are already immutable, so storing the reference is enough.

## Pitfalls

- **Missing defensive copy.** You pass a `byte[]` into the ctor and return the same array from a getter.
- **With-method mutates this.** An "immutable" class returns `this` after changing a field.
- **Reflection or serialization.** External code can rebuild a mutable graph after freeze.
- **Inheritance breaks the guarantee.** A non-final class with a protected setter in a subclass.
- **Excessive copying.** On large structures, every getter clones the whole graph unnecessarily.
