# Сохранится ли immutability при наследовании?

## Суть

Immutability **не сохраняется автоматически**. Если родитель mutable (есть mutators или non-final fields), подкласс наследует способы изменить состояние. Надёжный immutable type обычно `final` без subclassing; иначе вся цепочка баз должна быть immutable by construction.

## Что ответить на собесе

- Перечисляю условия immutability: final class (или sealed), final fields, no setters, defensive copies.
- Объясняю, что protected mutator в base ломает immutability child даже если child «не добавляет» setters.
- Рекомендую composition вместо extends для расширения immutable value.
- Упоминаю `record`: implicitly final, но mutable компонент (List) всё равно опасен.
- Честно добавляю: наследование immutable иерархий возможно (например sealed branch), но редко и требует дисциплины в base.

## Как это работает

Mutable parent:

```java
class MutableCounter {
    private int value;
    void increment() { value++; }
}

final class ChildCounter extends MutableCounter {}
```

`ChildCounter` остаётся mutable через унаследованный `increment`.

Immutable chain:

```java
public class ImmutableCounter {
    private final int value;
    protected ImmutableCounter(int value) { this.value = value; }
    public int value() { return value; }
}

public final class PositiveCounter extends ImmutableCounter {
    public PositiveCounter(int value) {
        super(value >= 0 ? value : 0);
    }
}
```

Здесь нет mutators; subclass только constrains construction. Без `final` на base кто-то может добавить mutable subclass elsewhere.

## Подводные камни

- **Non-final immutable base.** Новый subclass может ввести mutation.
- **Protected setter в base.** Ломает immutability всей иерархии.
- **Shallow immutability.** Final reference на mutable List в parent.
- **clone/serialization в subclass.** Может обойти immutability guarantees.
- **equals across hierarchy.** Symmetry ломается при смешении base/sub types.
