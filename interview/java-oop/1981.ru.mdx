# Почему код должен зависеть от interface, а не от impl?

## Суть

Зависимость от абстракции (interface) отделяет policy от деталей реализации. Можно подменить `EmailNotifier` на `SmsNotifier`, подставить mock в тестах и менять infra без переписывания business layer. Это практическое выражение Dependency Inversion Principle.

## Что ответить на собесе

- Формулирую DIP: high-level modules не должны зависеть от low-level concrete classes.
- Показываю constructor injection с interface type.
- Связываю с тестируемостью: fake/stub реализует тот же contract.
- Упоминаю Spring/Jakarta CDI: bean wiring по interface.
- Честно добавляю: не каждый класс нуждается в interface; избыточный `IUserService` для единственной impl добавляет шум.

## Как это работает

```java
public interface AlertChannel {
    void deliver(String message);
}

public final class SmsAlertChannel implements AlertChannel {
    @Override
    public void deliver(String message) {
        System.out.println("sms:" + message);
    }
}

public final class IncidentRouter {
    private final AlertChannel channel;

    public IncidentRouter(AlertChannel channel) {
        this.channel = channel;
    }

    public void raise(String msg) {
        channel.deliver(msg);
    }
}
```

`IncidentRouter` не импортирует SMS SDK. В prod DI container передаёт `SmsAlertChannel`, в test `AlertChannel msg -> recorded.add(msg)`.

## Подводные камни

- **Interface per class reflex.** Лишний слой без второй реализации.
- **Leaking impl types.** Cast `(SmsAlertChannel) channel` возвращает coupling.
- **Fat interface.** Подмена impl заставляет реализовать неиспользуемые методы.
- **Framework magic без contract tests.** Подставили wrong bean silently.
- **Primitive obsession вместо interface.** Stringly typed dependencies хуже, чем narrow interface.
