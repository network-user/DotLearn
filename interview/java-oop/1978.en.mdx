# What role does a class play in Java's type system?

## In short

A Java class is both a type name in the type system and a blueprint for objects: fields, methods, constructors, nested types. The compiler checks reference compatibility, inheritance, and overrides; the JVM uses runtime class metadata for dispatch and reflection.

## How to answer in an interview

- Separate compile-time type (`Car car`) from runtime class (`car.getClass()`).
- Explain hierarchy: class extends class, implements interfaces; a reference may be a supertype.
- Mention generics erasure: type parameters erase, but class literals remain.
- Tie to polymorphism: virtual dispatch uses the object's actual class.
- Be honest: a class file is not only classic OOP; records, enums, and sealed classes extend the type system.

## How it works

```java
public class Sedan {
    private String paint;

    public Sedan(String paint) {
        this.paint = paint;
    }

    public String paint() {
        return paint;
    }
}

public final class ElectricSedan extends Sedan {
    private final int kwh;

    public ElectricSedan(String paint, int kwh) {
        super(paint);
        this.kwh = kwh;
    }

    public int kwh() {
        return kwh;
    }
}
```

Variable `Sedan vehicle = new ElectricSedan("white", 75)` has static type `Sedan` and runtime type `ElectricSedan`. Overridden calls go through the vtable. `instanceof ElectricSedan` inspects runtime class. Class loaders read `.class` bytes and register `Class<?>` in metaspace.

## Pitfalls

- **Type vs instance confusion.** The Class object is per class loader; instances are many.
- **Raw types after erasure.** Plain `List` loses compile-time safety.
- **Class.forName vs .class.** Different loaders yield different Class objects for the same FQN.
- **Reflection bypass.** Private methods can be invoked, breaking encapsulation contracts.
- **Sealed hierarchy.** Missing permits blocks new subclasses at compile time.
