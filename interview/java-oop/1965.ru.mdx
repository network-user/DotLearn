# Зачем в Java нужен abstract class?

## Суть

Абстрактный класс задаёт частичную реализацию и обязательные «дыры» для подклассов. Его нельзя инстанцировать напрямую: он служит базой, где общий код живёт в concrete-методах, а вариативное поведение объявлено через abstract-методы.

## Что ответить на собесе

- Отличаю от interface: abstract class может хранить state, иметь конструктор и protected helpers.
- Объясняю, что подкласс обязан реализовать все abstract-методы, иначе он тоже abstract.
- Привожу кейс: общий шаблонный метод + несколько hook-методов в иерархии.
- Сравниваю с interface (Java 8+ default/static): множественное наследование типов vs одна база с кодом.
- Честно добавляю: если нужен только контракт без shared state, interface обычно чище; abstract class оправдан, когда есть реальная общая реализация.

## Как это работает

```java
abstract class ShipmentLabel {
    private final String trackingCode;

    protected ShipmentLabel(String trackingCode) {
        this.trackingCode = trackingCode;
    }

    abstract String carrierName();

    void printLabel() {
        System.out.println("[" + carrierName() + "] " + trackingCode);
    }
}

final class AirShipment extends ShipmentLabel {
    AirShipment(String trackingCode) {
        super(trackingCode);
    }

    @Override
    String carrierName() {
        return "AIR";
    }
}
```

`printLabel` наследуется как готовая реализация; `carrierName` каждый подкласс определяет сам. Ссылку можно держать как `ShipmentLabel label = new AirShipment("X1")`, вызывая полиморфно.

Abstract class часто комбинируют с Template Method: базовый класс описывает алгоритм, подклассы переопределяют отдельные шаги.

## Подводные камни

- **God abstract base.** Слишком много логики в базе делает подклассы хрупкими при любом изменении.
- **Путаница с interface.** Попытка «наследовать поведение» через interface + default там, где нужен shared mutable state.
- **Забытый abstract метод.** Класс-наследник не компилируется, пока не реализует все abstract members.
- **public abstract class с protected конструктором.** Паттерн верный, но без документированного factory подклассы могут быть единственным способом создания.
- **Глубокая иерархия.** Каждый новый уровень abstract усложняет понимание, кто какой метод реально вызывает.
