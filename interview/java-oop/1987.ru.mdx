# Чем отличаются перегрузка и переопределение методов?

## Суть

Overloading (перегрузка) это несколько методов с одним именем и разными параметрами внутри одного класса; выбор делает компилятор по сигнатуре вызова. Overriding (переопределение) это замена реализации унаследованного instance-метода в подклассе при сохранении контракта; dispatch идёт по runtime-типу объекта.

## Что ответить на собесе

- Overloading: compile-time, один класс, разные parameter lists (тип, порядок, количество).
- Overriding: runtime polymorphism, `@Override`, совместимая сигнатура, не для static/final/private.
- Упоминаю return type covariance (Java 5+) при override.
- Показываю, что overload не меняет иерархию, override расширяет поведение subtype.
- Честно добавляю: varargs и autoboxing создают неочевидный overload resolution; в API лучше явные имена.

## Как это работает

Перегрузка в одном классе:

```java
final class PricingService {
    Money quote(int units) {
        return Money.of(units * 10);
    }

    Money quote(int units, double discountRate) {
        return Money.of((int) (units * 10 * (1 - discountRate)));
    }

    Money quote(Money base, int bonusUnits) {
        return base.plus(Money.of(bonusUnits * 5));
    }
}
```

Переопределение в иерархии:

```java
abstract class Notifier {
    abstract void send(String message);
}

final class EmailNotifier extends Notifier {
    @Override
    void send(String message) {
        System.out.println("email: " + message);
    }
}

final class SmsNotifier extends Notifier {
    @Override
    void send(String message) {
        System.out.println("sms: " + message);
    }
}
```

`PricingService.quote(...)` резолвится на этапе компиляции по аргументам. `Notifier ref = new SmsNotifier(); ref.send("hi")` вызовет SMS-реализацию благодаря virtual dispatch. Overload увеличивает выразительность API внутри типа; override реализует полиморфное поведение в subtype chain.

## Подводные камни

- **Путаница overload vs override.** Одинаковое имя в parent и child с разными параметрами это overload, не override.
- **Скрытие static.** Похожая сигнатура у static не даёт runtime polymorphism.
- **Сужение access в override.** `protected` в parent нельзя сделать `private` в child.
- **Исключения шире в override.** Checked exceptions должны совпадать или быть уже.
- **Ambiguous varargs call.** `method(int...)` vs `method(int, int)` при одном аргументе компилятор может выбрать неожиданно.
