# Does immutability survive inheritance?

## In short

Immutability does **not** carry over automatically. If the parent is mutable (mutators or non-final fields), the subclass inherits ways to change state. A reliable immutable type is usually `final` with no subclassing; otherwise the whole base chain must be immutable by construction.

## How to answer in an interview

- List immutability rules: final class (or sealed), final fields, no setters, defensive copies.
- Explain that a protected mutator in the base breaks child immutability even when the child adds no setters.
- Recommend composition over extends to extend immutable values.
- Mention `record`: implicitly final, but a mutable component (List) still breaks guarantees.
- Be honest: immutable inheritance hierarchies are possible (sealed branches) but rare and need strict base discipline.

## How it works

Mutable parent:

```java
class MutableCounter {
    private int value;
    void increment() { value++; }
}

final class ChildCounter extends MutableCounter {}
```

`ChildCounter` stays mutable through inherited `increment`.

Immutable chain:

```java
public class ImmutableCounter {
    private final int value;
    protected ImmutableCounter(int value) { this.value = value; }
    public int value() { return value; }
}

public final class PositiveCounter extends ImmutableCounter {
    public PositiveCounter(int value) {
        super(value >= 0 ? value : 0);
    }
}
```

No mutators; the subclass only constrains construction. Without `final` on the base, someone can add a mutable subclass elsewhere.

## Pitfalls

- **Non-final immutable base.** A new subclass can introduce mutation.
- **Protected setter in base.** Breaks immutability for the whole hierarchy.
- **Shallow immutability.** Final reference to a mutable List in the parent.
- **clone/serialization in subclass.** Can bypass immutability guarantees.
- **equals across hierarchy.** Symmetry breaks when mixing base and sub types.
