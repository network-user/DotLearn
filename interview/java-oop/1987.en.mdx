# What is the difference between overloading and overriding?

## In short

Overloading means several methods with the same name and different parameters inside one class; the compiler picks the call by signature. Overriding replaces the implementation of an inherited instance method in a subclass while keeping the contract; dispatch uses the runtime type of the object.

## How to answer in an interview

- Overloading: compile time, one class, different parameter lists (type, order, count).
- Overriding: runtime polymorphism, `@Override`, compatible signature, not for static/final/private.
- I mention return-type covariance (Java 5+) when overriding.
- I show that overloading does not change the hierarchy, while overriding extends subtype behavior.
- I add honestly: varargs and autoboxing create non-obvious overload resolution; in APIs I prefer explicit names.

## How it works

Overloading in one class:

```java
final class PricingService {
    Money quote(int units) {
        return Money.of(units * 10);
    }

    Money quote(int units, double discountRate) {
        return Money.of((int) (units * 10 * (1 - discountRate)));
    }

    Money quote(Money base, int bonusUnits) {
        return base.plus(Money.of(bonusUnits * 5));
    }
}
```

Overriding in a hierarchy:

```java
abstract class Notifier {
    abstract void send(String message);
}

final class EmailNotifier extends Notifier {
    @Override
    void send(String message) {
        System.out.println("email: " + message);
    }
}

final class SmsNotifier extends Notifier {
    @Override
    void send(String message) {
        System.out.println("sms: " + message);
    }
}
```

`PricingService.quote(...)` resolves at compile time from the arguments. `Notifier ref = new SmsNotifier(); ref.send("hi")` calls the SMS implementation through virtual dispatch. Overloading makes the API more expressive inside one type; overriding implements polymorphic behavior in a subtype chain.

## Pitfalls

- **Confusing overload with override.** The same name in parent and child with different parameters is overload, not override.
- **Hiding static methods.** A similar signature on a static method does not give runtime polymorphism.
- **Narrower access in override.** You cannot make a `protected` parent method `private` in the child.
- **Broader exceptions in override.** Checked exceptions must match or be narrower.
- **Ambiguous varargs call.** `method(int...)` vs `method(int, int)` with one argument may resolve unexpectedly.
