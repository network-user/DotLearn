# Why should code depend on an interface, not an impl?

## In short

Depending on an abstraction (interface) separates policy from implementation details. You can swap `EmailNotifier` for `SmsNotifier`, inject mocks in tests, and change infrastructure without rewriting the business layer. That is Dependency Inversion in practice.

## How to answer in an interview

- State DIP: high-level modules should not depend on low-level concrete classes.
- Show constructor injection with an interface type.
- Tie to testability: fakes/stubs implement the same contract.
- Mention Spring/Jakarta CDI wiring beans by interface.
- Be honest: not every class needs an interface; a redundant `IUserService` for a single impl adds noise.

## How it works

```java
public interface AlertChannel {
    void deliver(String message);
}

public final class SmsAlertChannel implements AlertChannel {
    @Override
    public void deliver(String message) {
        System.out.println("sms:" + message);
    }
}

public final class IncidentRouter {
    private final AlertChannel channel;

    public IncidentRouter(AlertChannel channel) {
        this.channel = channel;
    }

    public void raise(String msg) {
        channel.deliver(msg);
    }
}
```

`IncidentRouter` does not import an SMS SDK. Production DI passes `SmsAlertChannel`; tests pass `AlertChannel msg -> recorded.add(msg)`.

## Pitfalls

- **Interface-per-class reflex.** Extra layer with no second implementation.
- **Leaking impl types.** Casting `(SmsAlertChannel) channel` restores coupling.
- **Fat interface.** Replacement impls must implement unused methods.
- **Framework magic without contract tests.** Wrong bean wiring fails silently.
- **Primitive obsession instead of interface.** Stringly typed dependencies are worse than a narrow interface.
