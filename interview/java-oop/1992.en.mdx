# When do you choose an abstract class vs an interface?

## In short

An abstract class combines shared state and partial implementation for a narrow family of types (single inheritance). An interface defines a capability contract without tying you to one hierarchy and allows multiple implements. Choose an abstract class for shared fields/ctors/template methods; choose an interface for cross-cutting roles, plug-in APIs, or several unrelated implementations.

## How to answer in an interview

- Abstract class: fields, constructor, non-abstract helper methods, one `extends`.
- Interface: contract, polymorphism, several roles on one class, defaults/static since Java 8.
- I mention the template method pattern in an abstract base.
- Since Java 8, interfaces with defaults are closer to abstract classes, but state in interfaces is still limited.
- I add honestly: in greenfield services I more often use composition plus interfaces; deep abstract hierarchies are hard to evolve.

## How it works

An abstract base class with shared state:

```java
abstract class LedgerEntry {
    private final String entryId;
    private final Instant createdAt;

    protected LedgerEntry(String entryId) {
        this.entryId = entryId;
        this.createdAt = Instant.now();
    }

    abstract Money amount();

    final void audit() {
        System.out.println(entryId + " at " + createdAt + " => " + amount());
    }
}
```

An interface as a capability across hierarchies:

```java
interface Exportable {
    byte[] toPayload();
}

final class InvoiceEntry extends LedgerEntry implements Exportable {
    private final Money total;

    InvoiceEntry(String entryId, Money total) {
        super(entryId);
        this.total = total;
    }

    @Override
    Money amount() {
        return total;
    }

    @Override
    public byte[] toPayload() {
        return total.toString().getBytes(StandardCharsets.UTF_8);
    }
}
```

`LedgerEntry` holds `entryId` and the `audit()` template. `Exportable` can attach to any class, not only ledger types. `InvoiceEntry` extends one abstract class and adds a second contract through an interface.

## Pitfalls

- **Abstract class as a namespace.** Only static helpers with no shared state.
- **Interface with many defaults.** Effectively an abstract base without fields, but with diamond risk.
- **Wrong `extends`.** Inheritance for one util method instead of composition.
- **Tight coupling to the base.** Protected fields are visible to all subclasses without boundaries.
- **Mixing role and hierarchy.** `implements Serializable, Cloneable, Comparable` without meaningful design.
