# How does Object.notify() differ from notifyAll()?

## In short

`notify()` wakes **one** arbitrary thread waiting on this object's monitor. `notifyAll()` wakes every waiter, but only one acquires the monitor once the current holder leaves synchronized code. Both require the caller to already hold the object's monitor.

## How to answer in an interview

- Explain wait/notify as a condition variable on the intrinsic lock.
- Clarify that a awakened thread does not run immediately; it competes for the monitor.
- Recommend `notifyAll` by default; use `notify` only when you can prove a single waiter and single condition.
- Tie to fairness: `notify` may starve the wrong thread when selection is unlucky.
- Be honest: production code often prefers `ReentrantLock` + `Condition` for explicit wait queues.

## How it works

```java
final class TicketWindow {
    private int ticketsLeft;

    synchronized boolean tryBuy() {
        while (ticketsLeft == 0) {
            try {
                wait();
            } catch (InterruptedException ex) {
                Thread.currentThread().interrupt();
                return false;
            }
        }
        ticketsLeft--;
        return true;
    }

    synchronized void restock(int amount) {
        ticketsLeft += amount;
        notifyAll();
    }
}
```

After `restock`, waiting buyers wake, recheck `while (ticketsLeft == 0)`, and take tickets one by one. With only `notify`, some threads might stay asleep even though tickets are available.

## Pitfalls

- **notify without changing the condition.** Waiters wake, fail the check, and sleep again; possible livelock.
- **notifyAll thundering herd.** Many threads wake and sleep again; under load prefer `Condition` or `Semaphore`.
- **Call outside synchronized.** Throws `IllegalMonitorStateException`.
- **Missing while on the waiter side.** State may still be invalid right after notify.
- **Multiple conditions on one lock.** `notify` may wake the wrong waiter; split locks or use `Condition`.
