# Какую роль класс играет в type system Java?

## Суть

Класс в Java одновременно описывает тип (имя в системе типов) и blueprint объектов: поля, методы, constructors, nested types. Компилятор проверяет совместимость ссылок, наследование и override; JVM использует runtime class metadata для dispatch и reflection.

## Что ответить на собесе

- Разделяю compile-time type (переменная `Car car`) и runtime class (`car.getClass()`).
- Объясняю иерархию: class extends class, implements interfaces; ссылка может быть supertype.
- Упоминаю generics erasure: параметр типа стирается, но class literal остаётся.
- Связываю с polymorphism: virtual method dispatch по фактическому классу объекта.
- Честно добавляю: class file это не только OOP model; records, enums, sealed classes расширяют систему типов.

## Как это работает

```java
public class Sedan {
    private String paint;

    public Sedan(String paint) {
        this.paint = paint;
    }

    public String paint() {
        return paint;
    }
}

public final class ElectricSedan extends Sedan {
    private final int kwh;

    public ElectricSedan(String paint, int kwh) {
        super(paint);
        this.kwh = kwh;
    }

    public int kwh() {
        return kwh;
    }
}
```

Переменная `Sedan vehicle = new ElectricSedan("white", 75)` имеет static type `Sedan`, runtime type `ElectricSedan`. Вызов override method идёт через vtable. `instanceof ElectricSedan` проверяет runtime class. Class loaders загружают `.class` bytes и регистрируют `Class<?>` объект в metaspace.

## Подводные камни

- **Путаница type vs instance.** Class object singleton на class loader; instances many.
- **Raw types после erasure.** `List` без generic теряет compile-time safety.
- **Class.forName vs .class.** Разные class loaders дают разные Class objects для одного FQN.
- **Reflection bypass.** Можно вызвать private method, ломая encapsulation contract.
- **Sealed hierarchy.** Забытый permits блокирует новый subclass на compile time.
