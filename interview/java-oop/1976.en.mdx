# When is an interface better than an abstract class?

## In short

An interface defines a behavioral contract without tying types to one base class or one shared state bucket. A class may implement several interfaces, enabling flexible substitution and dependency injection without inheritance of mutable fields.

## How to answer in an interview

- Frame it as a "can-do" capability: `Payable`, `Serializable`, `Closeable`.
- Stress polymorphism through interface-typed fields and parameters.
- Mention Java 8+ default/static methods for API evolution.
- Contrast with abstract class: choose interface when there is no shared mutable state.
- Be honest: an interface does not replace a domain model; anemic interfaces plus god implementations are still an anti-pattern.

## How it works

```java
public interface SettlementGateway {
    void charge(Money amount);
}

public final class CardGateway implements SettlementGateway {
    @Override
    public void charge(Money amount) {
        System.out.println("card charge " + amount);
    }
}

public final class CheckoutService {
    private final SettlementGateway gateway;

    public CheckoutService(SettlementGateway gateway) {
        this.gateway = gateway;
    }

    public void checkout(Money total) {
        gateway.charge(total);
    }
}
```

`CheckoutService` does not know about `CardGateway`; tests inject a stub. A new provider is a new class plus wiring, not a change to checkout logic.

## Pitfalls

- **Fat interface.** A god service interface forces stubs for a dozen methods.
- **Leaky abstraction.** The interface exposes infra details (SQLException in signatures).
- **Default method diamond.** Multiple interfaces with the same default need explicit resolution.
- **implements only for mocks.** Sometimes a package-private class plus test subclass is enough.
- **Versioning.** Adding an abstract method to a public interface breaks implementors unless you add a default.
