# Как проверить, что запрос 1С использует индекс

## Суть

Проверка идёт в два слоя: план запроса 1С в конфигураторе или консоли и фактический план SQL в СУБД. Только второй показывает реальный seek или scan по индексу на боевом объёме данных.

## Что ответить на собесе

- Я включаю «Показать план запроса» в конструкторе запросов и смотрю операторы IndexSeek vs TableScan на уровне платформы.
- На собеседовании добавляю обязательный шаг: скопировать SQL из технологического журнала и открыть `SET SHOWPLAN` или «План выполнения» в SSMS/pgAdmin.
- Подчёркиваю, что план на тестовой базе с тысячей строк может показать индекс, а на продакшене оптимизатор выберет скан.
- Честно говорю: параметризованный запрос 1С иногда даёт разные планы при разных значениях параметра.
- После добавления индекса в метаданных проверяю план только после обновления конфигурации базы и обновления статистики.

## Как это работает

В конфигураторе откройте запрос, выполните с флажком плана. Дерево операций покажет, как платформа читает таблицы метаданных: по индексу ссылки, полным сканом табличной части и т.д.

Для клиент-серверной базы включите в технологическом журнале событие `DBMSSQL` или `DBPOSTGRS` с длительностью выше порога. В тексте будет SQL с подстановкой параметров.

```bsl
Запрос = Новый Запрос;
Запрос.Текст =
"ВЫБРАТЬ
|    Остатки.Номенклатура,
|    Остатки.КоличествоОстаток
|ИЗ
|    РегистрНакопления.ТоварыНаСкладах.Остатки(
|        &Граница,
|        Склад = &Склад) КАК Остатки
|ГДЕ
|    Остатки.КоличествоОстаток > 0";
Запрос.УстановитьПараметр("Граница", ТекущаяДатаСеанса());
Запрос.УстановитьПараметр("Склад", СкладФильтра);
Результат = Запрос.Выполнить();
```

Скопированный SQL выполняется в SSMS с «Включить фактический план выполнения». Ищем оператор Index Seek по ожидаемому имени индекса. Index Scan по тому же индексу означает низкую селективность отбора.

В PostgreSQL используйте `EXPLAIN (ANALYZE, BUFFERS)` на том же тексте. Bitmap Index Scan на измерениях регистра подтверждает использование индекса платформы.

Если план показывает Table Scan на регистре движений, проверьте: есть ли отбор по периоду, совпадает ли порядок полей в составном индексе с условием, актуальна ли статистика после массового переноса данных.

## Подводные камни

- **Только план 1С без SQL.** Платформенный план абстрактен и может не совпасть с решением СУБД.
- **План на пустой таблице.** Оптимизатор почти всегда выбирает скан на малых объёмах.
- **Параметры по умолчанию.** Первый запуск после старта сеанса может отличаться от стабильного плана под боевыми значениями.
- **Подсказки в коде.** `OPTION (FORCE ORDER)` маскирует проблему и ломается при обновлении статистики.
- **Игнорирование Key Lookup.** Seek есть, но миллионы lookup к кластеру всё равно дают таймаут отчёта.
