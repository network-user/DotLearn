# Which fields 1C indexes without developer action

## In short

The platform automatically builds indexes for reference identifiers, register keys, fields with uniqueness enabled, and service columns required for integrity. Application attributes without the indexing flag get no extra structures.

## How to answer in an interview

- I list the automatic set: catalog and document `_IDRRef`, register dimensions, period in accumulation registers, uniqueness fields in information registers.
- In an interview I add that a catalog code is indexed only when the developer enables indexing or code uniqueness in metadata.
- I stress the difference between the platform primary key and the business Code field in the application sense.
- I am honest that automatic indexes do not remove the need to analyze heavy reports; they cover basic platform operations.
- When migrating between DBMS engines the automatic index set stays; only physical implementation changes.

## How it works

Every metadata object on deploy gets mandatory columns: reference, version, deletion mark, sometimes code and description. For a reference field the platform always provides fast lookup by binary identifier.

An accumulation register automatically indexes dimension combinations and period because virtual tables `Остатки` and `Обороты` build on those predicates. An information register with a unique dimension set gets a unique index to block duplicate slices.

Checking metadata from code:

```bsl
Мета = Метаданные.Справочники.Контрагенты;
Для Каждого Реквизит Из Мета.Реквизиты Цикл
    Если Реквизит.Индексирование = Индексирование.Индексировать Тогда
        Сообщить("Индексируется: " + Реквизит.Имя);
    КонецЕсли;
КонецЦикла;
Если Мета.ДлинаКода > 0 И Мета.КонтрольУникальности Тогда
    Сообщить("Уникальность кода включена");
КонецЕсли;
```

Fields like tax ID or SKU are not indexed by themselves. The developer enables Index or uniqueness in attribute properties and only then does a nonclustered index appear on database update.

Tabular sections automatically inherit an index on the owner reference; otherwise loading document lines would be a full scan.

## Pitfalls

- **Expecting an index on description.** Substring search on name does not use a B-tree without full-text search.
- **Uniqueness without an index is impossible.** The DBMS cannot enforce uniqueness efficiently without a structure.
- **Removing indexing in production.** Dropping an index on a hot register table blocks posting.
- **Confusing code and reference.** `НайтиПоКоду` uses a code index only with correct catalog settings.
- **Duplicates in information registers.** Without a unique index on dimensions the platform will not prevent logical duplicates at the SQL level.
