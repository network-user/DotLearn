# Clustered и non-clustered индекс в СУБД под 1С: в чём разница

## Суть

Кластерный индекс задаёт физический порядок строк таблицы и на таблице может быть только один. Некластерный хранится отдельно и указывает на строки, их может быть несколько. Платформа 1С создаёт эти структуры при развёртывании метаданных, разработчик управляет ими через индексирование полей.

## Что ответить на собесе

- Я объясняю на примере MS SQL: clustered определяет порядок листовых страниц, nonclustered это дополнительное B-дерево с ключом-указателем.
- На собеседовании говорю, что в таблицах 1С кластером часто выступает ссылочный `_IDRRef` или составной ключ регистра, точная схема зависит от типа объекта.
- Подчёркиваю: диапазонные запросы по дате выигрывают от кластера по периоду, точечный поиск по контрагенту обычно идёт через некластерный индекс.
- Честно уточняю: в PostgreSQL нет прямого аналога clustered index MS SQL, терминология переносится осторожно.
- Прикладной индекс в конфигураторе почти всегда некластерный с точки зрения DBA.

## Как это работает

При обновлении конфигурации базы платформа генерирует DDL. Для таблицы документа кластерный ключ обеспечивает локальность строк одного документа и его табличных частей в смежных страницах при последовательной записи.

Некластерный индекс по полю `Контрагент` позволяет быстро найти ссылки документов по контрагенту, но физический порядок таблицы остаётся по кластерному ключу. План запроса делает seek по некластерному индексу и lookup в кластер для получения остальных колонок.

```bsl
Запрос = Новый Запрос;
Запрос.Текст =
"ВЫБРАТЬ
|    Реализация.Ссылка,
|    Реализация.Дата,
|    Реализация.СуммаДокумента
|ИЗ
|    Документ.РеализацияТоваров КАК Реализация
|ГДЕ
|    Реализация.Контрагент = &Контрагент
|    И Реализация.Дата МЕЖДУ &Начало И &Конец
|УПОРЯДОЧИТЬ ПО
|    Реализация.Дата";
Запрос.УстановитьПараметр("Контрагент", КонтрагентОтбора);
```

Если некластерный индекс покрывает `(Контрагент, Дата)`, оптимизатор может обойтись без сортировки. Если кластер по `Дата`, диапазон по дате читает последовательные страницы с минимумом случайного IO.

Слишком много некластерных индексов на регистре движений замедляет проведение: каждая вставка обновляет все деревья.

## Подводные камни

- **Путаница с уникальным индексом.** Уникальность можно задать и на clustered, и на nonclustered; это разные оси.
- **Широкий некластерный индекс.** Включение всех колонок в covering index раздувает дерево и замедляет запись.
- **Перенос терминов в PostgreSQL.** Там cluster означает `CLUSTER` команду, а не постоянный clustered index.
- **Ручной CREATE INDEX в SSMS.** Индекс вне метаданных 1С исчезнет при пересоздании таблицы из конфигуратора.
- **Ожидание кластера по любому полю.** Только один clustered на таблицу; второй «кластер» невозможен.
