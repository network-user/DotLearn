# Какие поля в базе 1С имеет смысл индексировать

## Суть

Индекс ускоряет поиск и сортировку по выбранным полям, но замедляет запись. В 1С индексируют измерения регистров, реквизиты в отборах запросов и поля соединений, которые реально участвуют в рабочей нагрузке, а не «на будущее».

## Что ответить на собесе

- Я начинаю с анализа запросов: технологический журнал, планы выполнения СУБД, отчёты СКД с тяжёлыми отборами.
- На собеседовании называю кандидатов: период и регистратор в регистрах, организация и контрагент в типовых отборах, код и ИНН справочников при частом поиске.
- Подчёркиваю, что платформа уже создаёт индексы по ссылкам и ключам; прикладные индексы добавляют в метаданных осознанно.
- Честно говорю: лишний индекс на часто меняемом поле бьёт по проведению сильнее, чем ускоряет редкий отчёт.
- Для временных таблиц и внешних источников индексация на стороне 1С не поможет, нужен дизайн на уровне СУБД или витрины.

## Как это работает

Таблицы платформы 1С в СУБД содержат служебные и прикладные колонки. Индекс это отдельная структура, которая хранит упорядоченные значения выбранных полей и ссылки на строки. Когда запрос содержит отбор или сортировку по индексируемому полю, оптимизатор может пройти по индексу вместо полного сканирования.

Приоритеты для прикладного разработчика:

| Зона | Что индексировать | Зачем |
|------|-------------------|-------|
| Регистр накопления | Период, ключевые измерения в отборах остатков | Быстрые виртуальные таблицы `Остатки` |
| Регистр сведений | Ведущее измерение, период действия | Поиск среза на дату |
| Справочник | Код, ИНН, внешний идентификатор | Поиск при загрузке и интеграции |
| Документ | Дата, организация, контрагент | Журналы и списки с отбором |

В конфигураторе индекс задаётся в метаданных объекта. Платформа создаёт его в СУБД при обновлении структуры:

```bsl
Процедура ПроверитьИндексыРегистра() Экспорт
    МетаРегистра = Метаданные.РегистрыСведений.СтатусыЗаказов;
    Для Каждого Индекс Из МетаРегистра.Индексы Цикл
        Сообщить("Индекс: " + Индекс.Имя + ", поля: " + СтрокаИндексируемыхПолей(Индекс));
    КонецЦикла;
КонецПроцедуры

Функция СтрокаИндексируемыхПолей(ИндексМетаданных)
    Имена = Новый Массив;
    Для Каждого Поле Из ИндексМетаданных.Поля Цикл
        Имена.Добавить(Поле.Имя);
    КонецЦикла;
    Возврат СтрСоединить(Имена, ", ");
КонецФункции
```

Составной индекс полезен, когда запрос всегда фильтрует по организации и дате: порядок полей в индексе должен совпадать с наиболее селективным отбором в реальных запросах. Уникальный индекс применяют для естественных ключей в регистрах сведений независимого вида, чтобы СУБД отклонила дубликат на уровне данных.

Индексация не заменяет корректный запрос. Виртуальная таблица без параметров по периоду просканирует миллионы строк даже при наличии индекса по периоду, если план выбирает другой путь.

## Подводные камни

- **Индекс на каждое поле.** Запись документа обновляет все затронутые индексы и дисковый журнал растёт.
- **Неверный порядок полей в составном индексе.** Индекс (Дата, Организация) не поможет запросу только по организации.
- **Индексация вычисляемых представлений.** Отбор по строковому представлению ссылки не использует индекс ссылочного поля.
- **Игнорирование статистики СУБД.** После массовой загрузки план может перестать использовать индекс до обновления статистики.
- **Дублирование платформенных индексов.** Повторное индексирование ссылочного поля не даёт выигрыша и увеличивает стоимость записи.
