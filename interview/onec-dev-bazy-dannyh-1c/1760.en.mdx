# What a database index is in 1C

## In short

An index is a DBMS auxiliary structure over one or more table fields that speeds lookup and sorting. In 1C indexes are defined in metadata and maintained by the platform when objects are written and documents are posted.

## How to answer in an interview

- I compare an index to a table of contents: the DBMS finds rows through it without reading the whole table sequentially.
- In an interview I list pros: less IO, faster JOIN and ORDER BY; cons: disk space and INSERT/UPDATE cost.
- I clarify that most 1C indexes are B-trees on MS SQL or PostgreSQL, not a separate 1C-only structure.
- I am honest that an index does not speed up every query by itself; the optimizer must choose index seek or scan based on statistics.
- I distinguish field indexing in metadata from a composite index on the Indexed tab.

## How it works

When an index is created the DBMS builds a sorted key tree with pointers to table rows. A query with equality or a range on the indexed field walks the tree in logarithmic time relative to table size.

The effect shows up in application code on point lookup:

```bsl
Запрос = Новый Запрос;
Запрос.Текст =
"ВЫБРАТЬ ПЕРВЫЕ 1
|    Товары.Ссылка,
|    Товары.Наименование
|ИЗ
|    Справочник.Номенклатура КАК Товары
|ГДЕ
|    Товары.Артикул = &Артикул";
Запрос.УстановитьПараметр("Артикул", "SKU-9081");
Результат = Запрос.Выполнить();
```

If indexing is enabled on `Артикул`, the platform created the matching structure in `_Reference*`. When a catalog item is saved the index updates together with the main row in one transaction.

A composite index (Warehouse, Item) in a register helps the balances virtual table when both dimensions appear in the filter. An extra index only on the Quantity resource is usually useless because queries rarely filter by amount without dimensions.

Index size grows with field cardinality. Unique SKUs give a compact tree; booleans and two-value statuses often do not pay back maintenance cost.

## Pitfalls

- **Confusing index and primary key.** A 1C reference is already unique; an extra unique index on code is needed only for a business constraint.
- **Index scan instead of seek.** With low selectivity the optimizer reads the whole index, which is not faster than the table.
- **Fragmentation.** On ever-growing registers the index needs DBMS maintenance.
- **Metadata and database out of sync.** A manual SQL index is invisible in the Designer and disappears on rebuild.
- **Index on a computed query field.** A table column is indexed, not a SELECT expression without a persisted computed column in the DBMS.
