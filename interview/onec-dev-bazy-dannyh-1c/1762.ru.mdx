# Где физически живут временные таблицы в файловой и клиент-серверной 1С

## Суть

Временные таблицы запроса `ПОМЕСТИТЬ` создаются там, где выполняется SQL пакета. В клиент-серверном варианте это tempdb (или аналог) на сервере СУБД; в файловом режиме промежуточные данные обрабатываются локальным движком базы на машине, где крутится процесс 1С.

## Что ответить на собесе

- Я разделяю архитектуры: клиент-сервер значит ВТ на SQL-сервере, файловая база значит локальный каталог данных без отдельного tempdb кластера.
- На собеседовании подчёркиваю, что тяжёлый отчёт с ВТ в клиент-сервере нагружает tempdb и диск сервера, а не RAM клиента.
- Уточняю, что тонкий клиент всё равно отправляет запрос на сервер 1С, а SQL выполняется рядом с СУБД.
- Честно говорю: в файловой базе масштабирование по ВТ ограничено одним хостом и блокировками файла `.1CD`.
- Для распределённых филиалов файловый режим с большими ВТ почти всегда проигрывает клиент-серверу.

## Как это работает

Пакет с `ПОМЕСТИТЬ ВТДанные` компилируется в SQL, который СУБД выполняет в своём временном пространстве. В MS SQL это `tempdb`, таблицы именуются с `#` или `##` при прямом SQL; 1С абстрагирует имена, но ресурсы те же.

```bsl
Запрос = Новый Запрос;
Запрос.Текст =
"ВЫБРАТЬ
|    Обороты.Номенклатура,
|    СУММА(Обороты.Количество) КАК Количество
|ПОМЕСТИТЬ ВТОбороты
|ИЗ
|    РегистрНакопления.ТоварыНаСкладах.Обороты(&Начало, &Конец) КАК Обороты
|СГРУППИРОВАТЬ ПО
|    Обороты.Номенклатура";
Результат = Запрос.Выполнить();
```

В клиент-серверной архитектуре сервер 1С открывает соединение к SQL, tempdb растёт на время сеанса запроса, затем освобождается. Несколько пользователей с тяжёлыми отчётами делят один tempdb, что видно администратору по счётчикам задержек IO.

В файловой базе нет выделенного SQL-сервера: движок встроен в процесс 1С, временные структуры создаются внутри локального файла данных или вспомогательных файлов рядом с ним. Предел производительности это диск и CPU рабочей станции или терминального сервера.

Размещение ВТ не путать с таблицей значений в памяти сеанса: `ТаблицаЗначений` не проходит через `ПОМЕСТИТЬ` и живёт в RAM процесса 1С до выгрузки.

## Подводные камни

- **Ожидание ВТ на клиенте.** Даже толстый клиент не хранит `ПОМЕСТИТЬ` локально при подключении к серверной базе.
- **Заполнение tempdb.** Серия отчётов без отборов может исчерпать диск tempdb и остановить весь кластер SQL.
- **Файловая база на сетевом диске.** ВТ и основная база на медленном SMB умножают задержки.
- **Смешение менеджера ВТ.** Повторное использование между пакетами требует одного `МенеджерВременныхТаблиц` на сервере запроса.
- **Мониторинг только CPU 1С.** Узкое место может быть в tempdb, хотя rphost выглядит спокойным.
