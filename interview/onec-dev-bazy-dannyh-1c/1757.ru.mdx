# Зачем в 1С индексируют поля базы данных

## Суть

Индекс сокращает время поиска и сортировки по выбранным колонкам, потому что СУБД обращается к упорядоченной структуре вместо полного перебора таблицы. В крупных информационных базах 1С это разница между секундами ожидания в списке документов и минутами простоя при отчёте.

## Что ответить на собесе

- Я объясняю индекс как компромисс: быстрее чтение по шаблону запроса, дороже каждая запись при проведении.
- На собеседовании привожу три мотива: ускорение отборов WHERE, поддержка ORDER BY без сортировки всего набора, помощь оптимизатору при соединениях.
- Уточняю, что в 1С индексы задаются в метаданных и материализуются в СУБД при обновлении конфигурации.
- Честно говорю: без измерения реальных запросов индекс может не попасть в план и только увеличить размер базы.
- Слежу за балансом: один тяжёлый ночной отчёт не оправдывает десять индексов на полях, которые меняются при каждом документе.

## Как это работает

Когда пользователь открывает журнал продаж с отбором по контрагенту и периоду, платформа генерирует SQL с предикатами по соответствующим колонкам таблицы документа. Если по этим колонкам построен индекс, движок СУБД читает узлы B-дерева (или аналог в конкретной СУБД) и быстро находит диапазон строк.

Без индекса тот же запрос вызывает sequential scan: каждая страница таблицы читается с диска, строки фильтруются в памяти. На таблице движений регистра за несколько лет это превращается в узкое место всего кластера.

Пример запроса, который выигрывает от индекса по коду номенклатуры:

```bsl
Запрос = Новый Запрос;
Запрос.Текст =
"ВЫБРАТЬ
|    Номенклатура.Ссылка,
|    Номенклатура.Наименование
|ИЗ
|    Справочник.Номенклатура КАК Номенклатура
|ГДЕ
|    Номенклатура.Код = &КодПоиска";
Запрос.УстановитьПараметр("КодПоиска", "АРТ-00421");
Выборка = Запрос.Выполнить().Выбрать();
```

При записи документа платформа обновляет не только строку таблицы, но и все индексы, где участвуют изменённые поля. Поэтому индексация полей табличной части с сотнями строк на документ заметно удорожает проведение.

Оптимизатор может игнорировать индекс, если селективность низкая: например, булево поле «ПометкаУдаления» в активной базе почти везде Ложь, и полный скан иногда дешевле случайного чтения по индексу.

## Подводные камни

- **Индекс ради индекса.** Каждый новый индекс увеличивает время UPDATE при проведении и размер резервных копий.
- **Несовпадение с запросом.** Индекс по (Организация, Дата) бесполезен для отбора только по дате без организации в типичных планах MS SQL.
- **Функции в условии.** Отбор `ГОД(Дата) = 2024` не использует индекс по дате; нужен диапазон дат.
- **Устаревшая статистика.** После массового переноса данных оптимизатор выбирает скан, пока не обновлена статистика.
- **Лечение железом без профиля.** Увеличение RAM не заменяет отсутствие индекса на поле, по которому СКД фильтрует каждый отчёт.
