# Для чего в конфигураторе нужна закладка «Индексированные»

## Суть

Закладка «Индексированные» в свойствах объекта метаданных позволяет описать дополнительные индексы таблиц, которые платформа создаст в СУБД. Это точка, где разработчик связывает прикладную модель с физической производительностью запросов.

## Что ответить на собесе

- Я говорю, что закладка относится к объектам с табличным хранением: регистры, реквизиты справочников и документов с флагом индексирования, иногда табличные части.
- На собеседовании перечисляю действия: добавить составной индекс, включить индексирование реквизита, задать уникальность для регистра сведений.
- Подчёркиваю, что изменения вступают в силу после обновления конфигурации базы и могут занять время на больших таблицах.
- Честно уточняю: не все поля показывают одинаковые опции; у ссылочных полей платформа уже поддерживает быстрый поиск по `_IDRRef`.
- Перед добавлением индекса смотрю типовые запросы конфигурации, а не только красивую схему в конфигураторе.

## Как это работает

В конфигураторе откройте, например, регистр сведений «ЦеныНоменклатуры». На закладке «Индексированные» задаются именованные индексы по комбинации измерений и реквизитов. При выгрузке в базу платформа генерирует DDL для MS SQL, PostgreSQL или другого выбранного движка.

Для отдельного реквизита справочника или документа индексирование включается флажком «Индексировать» в свойствах поля. Это создаёт простой индекс по одной колонке без ручного описания составного индекса.

Программный обзор метаданных помогает аудиту перед релизом:

```bsl
Процедура ВывестиИндексыДокумента() Экспорт
    МетаДокумента = Метаданные.Документы.РеализацияТоваров;
    Для Каждого Реквизит Из МетаДокумента.Реквизиты Цикл
        Если Реквизит.Индексирование <> Индексирование.НеИндексировать Тогда
            Сообщить(Реквизит.Имя + ": " + Строка(Реквизит.Индексирование));
        КонецЕсли;
    КонецЦикла;
    Для Каждого Индекс Из МетаДокумента.Индексы Цикл
        Сообщить("Составной индекс " + Индекс.Имя);
    КонецЦикла;
КонецПроцедуры
```

После изменения индексов администратор выполняет «Обновление конфигурации базы данных». На продакшене это планируют в окно обслуживания: построение индекса на многомиллионной таблице движений блокирует или замедляет запись.

Закладка не заменяет тюнинг на стороне СУБД. Она описывает прикладные индексы в терминах метаданных 1С, а физическое имя таблицы остаётся внутренним (`_Reference*`, `_Document*`).

## Подводные камни

- **Индекс на тестовой копии с пустой таблицей.** На боевой базе создание займёт часы и вызовет блокировки.
- **Дублирование индексов.** Составной индекс (Организация, Контрагент) перекрывает отдельный индекс только по организации не всегда; лишние структуры плодят издержки записи.
- **Индексирование строк неограниченной длины.** Длинные строковые поля индексируются с ограничениями движка СУБД.
- **Забытое снятие индекса.** Удалённый отчёт оставляет ненужный индекс, который продолжает обновляться при проведении.
- **Ручные правки в SQL.** Прямое создание индекса в СУБД вне метаданных потеряется при следующем обновлении структуры из конфигуратора.
