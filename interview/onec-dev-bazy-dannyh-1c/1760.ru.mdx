# Что такое индекс в базе данных 1С

## Суть

Индекс это вспомогательная структура СУБД по одному или нескольким полям таблицы, которая ускоряет поиск и сортировку. В 1С индексы описываются в метаданных и обслуживаются платформой при записи объектов и проведении документов.

## Что ответить на собесе

- Я сравниваю индекс с оглавлением: по нему СУБД находит строки, не читая всю таблицу подряд.
- На собеседовании перечисляю плюсы: меньше IO, быстрее JOIN и ORDER BY; минусы: место на диске и стоимость INSERT/UPDATE.
- Уточняю, что большинство индексов в 1С это B-деревья на стороне MS SQL или PostgreSQL, а не отдельная «1С-структура».
- Честно говорю: индекс не ускоряет сам по себе любой запрос, оптимизатор должен выбрать index seek или scan по статистике.
- Различаю индекс поля в метаданных и составной индекс на закладке «Индексированные».

## Как это работает

При создании индекса СУБД строит отсортированное дерево ключей с указателями на строки таблицы. Запрос с равенством или диапазоном по индексируемому полю проходит по дереву за логарифмическое время относительно размера таблицы.

В прикладном коде эффект виден на точечном поиске:

```bsl
Запрос = Новый Запрос;
Запрос.Текст =
"ВЫБРАТЬ ПЕРВЫЕ 1
|    Товары.Ссылка,
|    Товары.Наименование
|ИЗ
|    Справочник.Номенклатура КАК Товары
|ГДЕ
|    Товары.Артикул = &Артикул";
Запрос.УстановитьПараметр("Артикул", "SKU-9081");
Результат = Запрос.Выполнить();
```

Если по полю `Артикул` включено индексирование, платформа создала соответствующую структуру в таблице `_Reference*`. При записи элемента справочника индекс обновляется вместе с основной строкой в одной транзакции.

Составной индекс (Склад, Номенклатура) в регистре помогает виртуальной таблице остатков, когда оба измерения участвуют в отборе. Лишний индекс только по ресурсу «Количество» обычно бесполезен, потому что запросы редко фильтруют по сумме без измерений.

Размер индекса растёт с кардинальностью поля. Уникальные артикулы дают компактное дерево; булевы и статусы с двумя значениями часто не окупают затрат на поддержку.

## Подводные камни

- **Путаница индекса и первичного ключа.** Ссылка 1С уже уникальна; дополнительный уникальный индекс по коду нужен только для бизнес-ограничения.
- **Index scan вместо seek.** При низкой селективности оптимизатор читает весь индекс, что не быстрее таблицы.
- **Фрагментация.** На вечно растущих регистрах индекс требует обслуживания на стороне СУБД.
- **Несинхрон метаданных и базы.** Ручной индекс в SQL не отображается в конфигураторе и исчезнет при пересоздании.
- **Индекс на вычисляемое поле запроса.** Индексируется колонка таблицы, а не выражение в SELECT без persisted computed column в СУБД.
