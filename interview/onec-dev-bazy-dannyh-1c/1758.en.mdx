# What the Indexed tab in the Designer is for

## In short

The Indexed tab in object metadata properties lets you describe extra table indexes the platform will create in the DBMS. This is where the developer ties the application model to physical query performance.

## How to answer in an interview

- I say the tab applies to objects with table storage: registers, catalog and document attributes with the indexing flag, sometimes tabular sections.
- In an interview I list actions: add a composite index, enable attribute indexing, set uniqueness for an information register.
- I stress that changes take effect after a database configuration update and can take time on large tables.
- I honestly note that not all fields expose the same options; reference fields already get fast lookup via `_IDRRef`.
- Before adding an index I review typical configuration queries, not only a neat schema in the Designer.

## How it works

In the Designer open, for example, the information register `ЦеныНоменклатуры`. On the Indexed tab you define named indexes over dimension and attribute combinations. On deploy the platform generates DDL for MS SQL, PostgreSQL, or another chosen engine.

For a single catalog or document attribute, indexing is enabled with the Index checkbox in field properties. That creates a simple single-column index without manually defining a composite one.

A programmatic metadata review helps before release:

```bsl
Процедура ВывестиИндексыДокумента() Экспорт
    МетаДокумента = Метаданные.Документы.РеализацияТоваров;
    Для Каждого Реквизит Из МетаДокумента.Реквизиты Цикл
        Если Реквизит.Индексирование <> Индексирование.НеИндексировать Тогда
            Сообщить(Реквизит.Имя + ": " + Строка(Реквизит.Индексирование));
        КонецЕсли;
    КонецЦикла;
    Для Каждого Индекс Из МетаДокумента.Индексы Цикл
        Сообщить("Составной индекс " + Индекс.Имя);
    КонецЦикла;
КонецПроцедуры
```

After index changes the administrator runs Update Database Configuration. In production this is scheduled in a maintenance window: building an index on a multi-million-row movement table blocks or slows writes.

The tab does not replace DBMS tuning. It describes application indexes in 1C metadata terms while physical table names stay internal (`_Reference*`, `_Document*`).

## Pitfalls

- **Indexing on a test copy with an empty table.** On production creation can take hours and cause blocking.
- **Duplicate indexes.** A composite index (Organization, Counterparty) does not always cover a separate organization-only index; extra structures add write overhead.
- **Indexing unlimited-length strings.** Long string fields are indexed with DBMS engine limits.
- **Forgotten index removal.** A removed report leaves a useless index that still updates on posting.
- **Manual SQL changes.** Creating an index in the DBMS outside metadata is lost on the next structure update from the Designer.
