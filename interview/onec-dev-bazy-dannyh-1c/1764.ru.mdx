# Какие поля 1С индексирует без участия разработчика

## Суть

Платформа автоматически строит индексы для ссылочных идентификаторов, ключей регистров, полей с включённой уникальностью и служебных колонок, необходимых для целостности. Прикладные реквизиты без флага индексирования остаются без дополнительных структур.

## Что ответить на собесе

- Я перечисляю автоматику: `_IDRRef` справочника и документа, измерения регистров, период в регистрах накопления, поля уникальности регистра сведений.
- На собеседовании добавляю, что код справочника индексируется только если разработчик включил индексирование или уникальность кода в метаданных.
- Подчёркиваю разницу между первичным ключом платформы и бизнес-полем «Код» в прикладном смысле.
- Честно говорю: автоматические индексы не снимают необходимость анализа тяжёлых отчётов, они покрывают базовые операции платформы.
- При миграции между СУБД набор автоматических индексов сохраняется, меняется только физическая реализация.

## Как это работает

Каждый объект метаданных при выгрузке в базу получает набор обязательных колонок: ссылка, версия, пометка удаления, иногда код и наименование. Для ссылочного поля платформа всегда обеспечивает быстрый поиск по бинарному идентификатору.

Регистр накопления автоматически индексирует комбинацию измерений и периода, потому что виртуальные таблицы `Остатки` и `Обороты` строятся на этих предикатах. Регистр сведений с уникальным набором измерений получает уникальный индекс для запрета дублей среза.

Проверка метаданных из кода:

```bsl
Мета = Метаданные.Справочники.Контрагенты;
Для Каждого Реквизит Из Мета.Реквизиты Цикл
    Если Реквизит.Индексирование = Индексирование.Индексировать Тогда
        Сообщить("Индексируется: " + Реквизит.Имя);
    КонецЕсли;
КонецЦикла;
Если Мета.ДлинаКода > 0 И Мета.КонтрольУникальности Тогда
    Сообщить("Уникальность кода включена");
КонецЕсли;
```

Поля вроде ИНН или артикула не получают индекс сами по себе. Разработчик включает «Индексировать» или задаёт уникальность в свойствах реквизита, и только тогда при обновлении базы появится некластерный индекс.

Табличные части наследуют индекс по ссылке на владельца автоматически, иначе выборка строк документа стала бы полным сканом.

## Подводные камни

- **Ожидание индекса по наименованию.** Поиск по подстроке наименования не использует B-дерево без полнотекстового поиска.
- **Уникальность без индекса невозможна.** СУБД не проверит уникальность эффективно без структуры.
- **Снятие индексирования на боевой базе.** Удаление индекса на горячей таблице регистра блокирует проведение.
- **Путаница кода и ссылки.** Поиск `НайтиПоКоду` использует индекс кода только при правильных настройках справочника.
- **Дубли в регистре сведений.** Без уникального индекса по измерениям платформа не спасёт от логических дублей на уровне SQL.
