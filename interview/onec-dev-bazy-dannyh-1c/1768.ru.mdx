# Физическая структура таблиц регистра накопления в СУБД

## Суть

Регистр накопления в базе представлен таблицей движений с периодом, видом движения, измерениями, ресурсами и регистратором. Для ускорения остатков платформа может хранить агрегаты итогов; виртуальные таблицы запроса читают движения и при необходимости итоги.

## Что ответить на собесе

- Я описываю одну строку движения как факт изменения ресурса по набору измерений в момент времени.
- На собеседовании называю ключевые колонки: `Period`, `_RecordKind`, измерения, ресурсы, `_RecorderRRef`.
- Подчёркиваю, что остаток это не отдельная «ячейка», а агрегат приходов и расходов по срезу.
- Честно говорю: включённые итоги ускоряют чтение, но требуют пересчёта при перепроведении старых документов.
- Для диагностики смотрю `_AccumReg*` в СУБД и связываю с движениями документа по регистратору.

## Как это работает

При проведении документа каждая строка движения превращается в INSERT в таблицу `_AccumRegN`. Вид движения кодирует приход или расход. Период и момент времени задают порядок в оперативном учёте.

```bsl
Движения.ОстаткиТоваров.Записывать = Истина;
Движение = Движения.ОстаткиТоваров.Добавить();
Движение.ВидДвижения = ВидДвиженияНакопления.Приход;
Движение.Период = ДатаДокумента;
Движение.Склад = СкладПоступления;
Движение.Номенклатура = Строка.Номенклатура;
Движение.Количество = Строка.Количество;
```

Таблица итогов (агрегатов) хранит накопленные суммы по периодам и измерениям, если регистр использует итоги. Тогда виртуальная таблица `Остатки` комбинирует итог до границы и движения после, сокращая объём чтения.

Запрос остатков компилируется в SQL с GROUP BY или JOIN к таблице итогов:

```bsl
Запрос.Текст =
"ВЫБРАТЬ
|    Остатки.Номенклатура,
|    Остатки.КоличествоОстаток
|ИЗ
|    РегистрНакопления.ОстаткиТоваров.Остатки(&Граница, Склад = &Склад) КАК Остатки";
```

Индексы по умолчанию ориентированы на период и измерения. Массовое перепроведение задним числом инвалидирует итоги и запускает пересчёт, что видно как всплеск IO на таблице регистра.

## Подводные камни

- **Путаница с регистром сведений.** Там нет вида движения приход/расход, другая модель ключа.
- **Отключённые итоги на большом регистре.** Остатки всегда считаются по полной истории движений.
- **Перепроведение без понимания агрегатов.** Итоги расходятся с детализацией до регламентного пересчёта.
- **Запрос остатков без границы периода.** План читает весь регистр.
- **Прямое удаление строк в SQL.** Нарушает связь с регистратором и ломает учёт.
