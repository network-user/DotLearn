# Event subscription types in 1C: manager, object, global

## In short

An event subscription in configuration links a metadata event to a handler in a common module. By scope you distinguish manager-level subscriptions (typical operations on the whole class), instance-level ones (object module events), and "broad" subscriptions without a tight tie to one object when a handler centralizes the same event across types.

## How to answer in an interview

- I explain that a subscription is an "EventSubscription" metadata object, not arbitrary form code.
- I name three levels: manager (create, copy, delete via manager), object (BeforeWrite, OnWrite of a specific instance), global scenarios via a shared handler and source filter.
- I stress handlers are almost always server-side, with the subscription's server call flag set.
- I honestly note that in extensions subscriptions are added separately; I verify order against the base product on a test infobase.
- I separate subscriptions from form handlers: forms react to UI, subscriptions to data lifecycle in the infobase.

## How it works

In the designer you create a subscription: source "Document.CustomerOrder", event "BeforeWrite", handler "CommonModule.OrderControl.BeforeWriteOrder".

```bsl
Процедура ПередЗаписьюЗаказа(Источник, Отказ) Экспорт

    Если Источник.СуммаДокумента <= 0 Тогда
        Отказ = Истина;
        ВызватьИсключение "Сумма заказа должна быть больше нуля";
    КонецЕсли;

КонецПроцедуры
```

A "manager" subscription fires on operations the platform runs through the object manager, e.g. programmatic catalog item creation or bulk delete. An "object" subscription catches write, post, or undo post events of a concrete instance.

Global scenarios in SSL-based products often use one subscription with multiple sources or subsystems like "Attachable commands" / scheduled checks so the same rule is not duplicated in every object module. In an extension you can add a subscription to the same event; the platform invokes base and extension handlers per merge rules.

## Pitfalls

**Duplicated logic.** Several subscriptions on one event without coordination produce unpredictable order and double checks.

**Heavy handler.** Queries and loops in `BeforeWrite` block interactive document save for all users.

**Client context.** A subscription does not replace `&НаКлиенте` on a form; calling UI from a server handler needs a separate notification mechanism.

**Extensions and upgrades.** After a base product update the subscription source may change; regression runs are mandatory.

**Cancel vs exception.** Mixing `Отказ = Истина` and `ВызватьИсключение` changes the user-facing error and transaction behavior.
