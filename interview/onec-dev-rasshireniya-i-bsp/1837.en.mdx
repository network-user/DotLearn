# Ways to override procedures and functions in extensions

## In short

In a configuration extension you change base module procedures via platform annotations: `&Before`, `&After`, `&Instead`, and editing adopted bodies. You also add event subscriptions and own common modules without touching vendor code.

## How to answer in an interview

- I list annotations and when to pick each: `&After` to extend, `&Instead` only for full replacement with known risk.
- I show adopting a metadata object or module before override.
- I mention subscriptions when a module cannot be adopted.
- I honestly say `&Instead` without calling base logic is a common post-upgrade regression cause.
- I name checks: cfe diff, posting smoke test, event log.

## How it works

Adopt a document manager module and add a handler after the base procedure:

```bsl
&После("ОбработкаПроведения")
Процедура Расш1_ОбработкаПроведения(Отказ, РежимПроведения)

    ЗарегистрироватьДвижениеВWMS(ЭтотОбъект);

КонецПроцедуры
```

To change a function result use `&Instead` with a platform continuation call or recalc only the needed part:

```bsl
&Вместо("ПолучитьСкидку")
Функция Расш1_ПолучитьСкидку(Партнер, Сумма)

    БазоваяСкидка = ПродолжитьВызов(Партнер, Сумма);
    Возврат БазоваяСкидка + ДополнительнаяСкидкаПартнера(Партнер);

КонецФункции
```

If the module is not adoptable, add an `OnPost` subscription or fill an SSL overridable module. Order: base extension points first, then annotations, subscriptions last.

## Pitfalls

**&Instead without `ContinueCall`.** Base logic silently disappears; quiet accounting bugs appear after upgrade.

**Annotation conflict.** Two extensions with `&Instead` on one procedure need agreed attachment order.

**Unneeded adoption.** Extra adopted forms complicate merge on upgrade.

**Client context.** Annotating a `&AtClient` procedure does not move server calls automatically.

**Test coverage.** Without posting autotests regression is caught only in production.
