# Библиотека стандартных подсистем: состав и роль в проекте

## Суть

Библиотека стандартных подсистем это отдельная конфигурация-библиотека, которую встраивают в прикладное решение. Она даёт инфраструктурные блоки (безопасность, файлы, обмен, печать, обновления) и договорённости по структуре кода, чтобы прикладная команда фокусировалась на учётных алгоритмах.

## Что ответить на собесе

- Я перечисляю ключевые блоки: пользователи, группы доступа, RLS, файловое хранилище, регламентные задания, версионирование объектов.
- Объясняю роль в проекте: единый стандарт для всех продуктов линейки 1С и партнёрских решений на БСП.
- Говорю про жизненный цикл: версия БСП в readme конфигурации, changelog, регламент обновления на продуктиве.
- Честно уточняю: БСП не заменяет знание предметной области; без понимания документов и регистров типовая всё равно «чёрный ящик».
- Упоминаю инструменты: «Описание обработчиков обновления», сравнение объединением, автоматические тесты Vanessa или своих сценариев.

## Как это работает

При старте проекта команда включает нужные подсистемы в дереве метаданных. Пользовательская учётная логика обращается к API БСП:

```bsl
Пользователь = Пользователи.ТекущийПользователь();
Если Не ПравоДоступа("Изменение", Метаданные.Документы.РеализацияТоваров) Тогда
    ВызватьИсключение "Нет прав на изменение реализации";
КонецЕсли;

ДвоичныеДанные = РаботаСФайлами.ДвоичныеДанныеФайла(ПрисоединенныйФайл);
```

Подсистемы связаны: «Присоединенные файлы» используют «Работа с файлами», «Пользователи» питают «Управление доступом». Обновление ИБ выполняет обработчики из модулей `ОбновлениеИнформационнойБазы*` по порядку версий.

В типовой ERP большая часть меню и админских форм приходит из БСП. Партнёр добавляет отраслевые объекты и заполняет переопределяемые модули, не дублируя механизм ролей.

## Подводные камни

**Отключение подсистемы «на глаз».** Удаление подсистемы из состава ломает зависимые объекты при сборке.

**Смешение версий.** БСП 3.1.25 в конфигурации с документацией 3.1.30 даёт расхождение API.

**Прямые SQL в обход RLS.** Запросы без учёта ограничений доступа БСП обходят настроенную модель безопасности.

**Игнор обработчиков обновления.** Пропуск версии при обновлении оставляет битые регистры и пустые константы.

**Обучение команды.** Без onboarding по БСП разработчики пишут параллельные механизмы пользователей и файлов.
