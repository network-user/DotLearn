# Where configuration extensions fit best

## In short

Extensions are meant to customize a base or update-supported configuration without taking it off vendor support: new objects, attributes, forms, subscriptions, and targeted procedure overrides. Vendor core stays untouched, preserving the path to the next release.

## How to answer in an interview

- I list typical cases: local reporting, integration, extra attributes, form tweaks, attachable commands.
- I stress an extension does not replace a full industry configuration when data architecture changes.
- I mention compatibility mode and checking "Adopted objects" after a base upgrade.
- I honestly note critical posting or RLS changes may need a separate approved patch if customer policy allows.
- I split multiple extensions by purpose (integration, UI, compliance) for easier maintenance.

## How it works

An implementation team attaches an extension to "1C:ERP" or "UT": adds a "CRMCode" attribute to counterparty catalog, shows it on the item form, and in an `OnWrite` subscription pushes changes to an external service.

```bsl
&После("ПриЗаписи")
Процедура Расш1_ПриЗаписиКонтрагента(Источник, Отказ)

    Если Источник.ОбменДанными.Загрузка Тогда
        Возврат;
    КонецЕсли;

    ОбменCRM.ЗарегистрироватьИзменениеКонтрагента(Источник.Ссылка);

КонецПроцедуры
```

New objects (report, data processor, information register) are created entirely in the extension and exposed in the customer's subsystem. Base form changes use adopted forms and programmatic elements in `OnCreateAtServer`, not copying the whole form into main configuration.

After a base upgrade you check adopted object conflicts, `&Before` / `&After` / `&Instead` annotations, and subscriptions on a production copy before go-live.

## Pitfalls

**Taking main configuration off support.** Editing base modules kills easy upgrades; extensions exist to avoid that.

**&Instead without calling base.** Full replacement drops vendor logic; `&After` or an explicit base call is usually required.

**Monolithic extension.** One bundle for all integrations complicates rollback and review.

**Roles and RLS.** New objects need roles; missing rights yield "empty" forms for users.

**CI performance.** cf/cfe build and autotests per extension should be part of the release process.
