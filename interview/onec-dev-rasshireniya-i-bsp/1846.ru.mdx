# Как обновить БСП в доработанной конфигурации

## Суть

Обновление БСП в нетиповой или доработанной конфигурации это controlled merge: новая поставка библиотеки загружается в копию, сравнивается с текущей, конфликты в переопределяемых и изменённых модулях разрешаются вручную, затем прогоняются обработчики обновления ИБ и регрессионные сценарии.

## Что ответить на собесе

- Я начинаю с бэкапа cf и dt, отдельной тестовой базы и фиксации текущей версии БСП в readme.
- Использую «Загрузить конфигурацию из файла» / поставку vendor и «Сравнить, объединить с конфигурацией из файла».
- Отдельно смотрю переопределяемые модули, `ОбновлениеИнформационнойБазы*`, изменённые подсистемы.
- Честно уточняю: если правили типовые модули БСП напрямую, merge дороже, чем при чистых переопределяемых точках.
- После merge обязательно «Обновить конфигурацию БД» и сценарии из release notes БСП.

## Как это работает

Команда получает cf БСП 3.1.30 и текущую конфигурацию ERP с заполненными `ПереопределяемыйОбщий` модулями. В пустой копии загружают целевую ERP, затем выполняют сравнение с файлом БСП или объединение через шаблон обновления от вендора продукта.

Конфликт в модуле `ОбновлениеИнформационнойБазы` разрешают в пользу новой версии, перенося свои обработчики в конец цепочки версий. Проектные процедуры в `ПереопределяемыйПечать` оставляют из своей ветки, если сигнатура не изменилась; при изменении API правят вызовы по changelog.

```bsl
Процедура ПриДобавленииОбработчиковОбновления(Обработчики) Экспорт

    // типовой блок после merge
    ОбновлениеИнформационнойБазы.ДобавитьОбработчик(Обработчики, "3.1.30.1", "ОбновитьРолиПользователей");

    // проектный обработчик сохраняем явно
    ОбновлениеИнформационнойБазы.ДобавитьОбработчик(Обработчики, "3.1.30.99", "Проект_МигрироватьСправочникКонтрагентов");

КонецПроцедуры
```

На тестовой базе запускают клиент, платформа выполняет отложенные обработчики. Проверяют печать, обмен, RLS, регламентные задания. Только после sign-off выкатывают cf на продуктив в окно обслуживания.

## Подводные камни

**Merge на проде.** Объединение конфигураций на рабочей базе без копии недопустимо.

**Пропуск обработчиков.** Невыполненный обработчик версии оставляет битые константы и роли.

**Ручные правки ядра БСП.** Каждый релиз библиотеки re-merge вручную; лучше переопределяемые модули.

**Несовместимость платформы.** Новая БСП может требовать минимальную версию 8.3.x.

**Расширения.** После обновления БСП пересобирают и тестируют все cfe отдельно.
