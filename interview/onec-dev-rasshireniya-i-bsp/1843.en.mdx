# When extensions cannot or should not be used

## In short

Extensions fail when you need deep metadata restructuring, removing base objects, changing posting architecture, or working in an already off-support configuration without merge policy. It is also risky to "patch" with many extensions what needs an industry solution or a fork.

## How to answer in an interview

- I name red flags: rewriting half a base module via `&Instead`, conflicting cfe from three integrators, unadoptable required objects.
- I discuss off-support cases: sometimes the customer deliberately goes non-standard and extensions do not restore upgrades.
- I mention performance: critical cost recalc in subscriptions instead of the built-in mechanism.
- I honestly note "cannot" often means "costlier than alternatives", not a platform ban.
- I suggest alternatives: external processors, separate subsystem, industry product, agreed main configuration changes.

## How it works

Typical rejection: customer wants to change accounting register movements and remove a base document. Extensions cannot delete main configuration objects or change base register structure; parallel registers in cfe create double accounting.

Another case: five extensions override one document's `Posting`. After an ERP upgrade the procedure signature changed; three cfe files fail to compile, month-end close stops.

```bsl
&Вместо("ОбработкаПроведения")
Процедура Расш1_ОбработкаПроведения(Отказ, Режим)

    ВыполнитьСобственноеПроведение(ЭтотОбъект);
    // ContinueCall intentionally omitted

КонецПроцедуры
```

Such code is acceptable only with conscious rejection of base posting and full regression. For extra control only, `&After` or a subscription suffices.

Decisions use a matrix: change depth, base upgrade frequency, extension count, CI presence. Beyond threshold the project moves to agreed non-standard branch or consolidates changes in one cfe.

## Pitfalls

**"Extensions at any cost" policy.** Blocks off-support work where it is cheaper to maintain.

**Ignored upgrade merge.** "Worked for a year" does not guarantee compatibility with a new release.

**Duplicated base logic.** Posting copy in cfe drifts from vendor fixes.

**External bypasses.** COM/SQL outside the platform breaks RLS and audit.

**No cfe owner.** Without an owner the extension is disabled first in incidents, losing functionality.
