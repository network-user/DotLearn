# What an event subscription is in 1C configuration

## In short

An event subscription is a metadata object linking an application object event (e.g. a document `BeforeWrite`) to an exported common module procedure. The platform invokes the handler at event time without embedding code in the object module.

## How to answer in an interview

- I describe three parts: source (metadata type), event name, handler in a common module with server call enabled.
- I contrast it with a form handler: subscriptions run on any object write, not only from UI.
- I show parameters `Источник`, `Отказ`, `СтандартнаяОбработка` depending on the event.
- I honestly note subscriptions run synchronously and inherit the write transaction.
- I mention adding subscriptions in extensions to customize base products without taking them off support.

## How it works

In the designer you create a subscription: source "Document.GoodsSale", event "Posting", handler "WarehouseAccounting.SalePostingHandler".

```bsl
Процедура РеализацияОбработкаПроведения(Источник, Отказ, РежимПроведения) Экспорт

    Если РежимПроведения <> РежимПроведенияДокумента.Проведение Тогда
        Возврат;
    КонецЕсли;

    СкладскойУчет.ЗарезервироватьОстатки(Источник);

КонецПроцедуры
```

On document posting the platform runs the object module and attached subscriptions. Multiple subscriptions on one event run in configuration order; cancel in one handler aborts the operation.

Subscriptions suit cross-cutting rules: fill checks, change log registration, exchange queue enqueue. In an extension you add another subscription or extend logic via `&After` in an adopted module when a subscription is not ideal.

## Pitfalls

**Duplication with object module.** The same check in object `BeforeWrite` and in a subscription runs twice.

**Heavy logic.** Long code in a subscription blocks save for all clients.

**Data exchange.** Without `Источник.ОбменДанными.Загрузка` the subscription breaks message load.

**Opaque stack.** A subscription error does not tell the user which handler failed without the event log.

**Base upgrade.** Renamed events or removed sources in a new version silently disable subscriptions until regression testing.
