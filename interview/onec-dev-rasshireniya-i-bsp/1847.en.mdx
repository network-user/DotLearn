# Event subscriptions vs editing the object module

## In short

The object module holds instance lifecycle handlers called directly by the platform. An event subscription lives in a common module and fires for a chosen metadata type without changing the object module. The first path is tighter to the object; the second suits cross-cutting rules and base product extensions.

## How to answer in an interview

- I say the object module knows `ThisObject`; a subscription gets `Source` as a parameter.
- A subscription attaches to a whole document/catalog type; an object module only to one metadata object.
- For base products without taking them off support I prefer a subscription or extension `&After`, not editing the object module in main configuration.
- I honestly note complex posting tied to private object module procedures is duplicated or moved to a shared service in subscriptions.
- I mention call order: object module first, then subscriptions (confirm for your platform version).

## How it works

In document module "Order", `BeforeWrite` checks the tabular section:

```bsl
Процедура ПередЗаписью(Отказ, РежимЗаписи)

    Если Объект.Товары.Количество() = 0 Тогда
        Отказ = Истина;
        Сообщить("Табличная часть пуста");
    КонецЕсли;

КонецПроцедуры
```

A cross-cutting ban on editing closed periods for several documents uses a subscription:

```bsl
Процедура КонтрольЗакрытогоПериодаПередЗаписью(Источник, Отказ, РежимЗаписи, РежимПроведения) Экспорт

    Если Источник.Дата < Константы.ДатаЗапретаРедактирования.Получить() Тогда
        Отказ = Истина;
    КонецЕсли;

КонецПроцедуры
```

One handler serves "Order", "Sale", and "Receipt". The object module stays document-specific. In an extension you adopt the object module and extend with annotations; add a subscription as separate metadata.

## Pitfalls

**Duplication.** The same check in module and subscription runs twice.

**Invisible from module.** New developers read only the object module and miss subscriptions.

**Performance.** Many subscriptions on `BeforeWrite` for all documents add latency.

**Context.** Subscriptions do not see object module locals; you need an explicit API.

**Base upgrade.** Object module edits in main configuration are overwritten; cfe subscriptions survive releases more easily.
