# How to upgrade SSL in a customized configuration

## In short

Upgrading SSL in a non-standard or customized configuration is a controlled merge: load the new library into a copy, compare with current, resolve conflicts in overridable and changed modules manually, then run infobase update handlers and regression scenarios.

## How to answer in an interview

- I start with cf and dt backup, a separate test infobase, and current SSL version recorded in readme.
- I use "Load configuration from file" / vendor delivery and "Compare and merge with configuration from file".
- I review overridable modules, `ОбновлениеИнформационнойБазы*`, changed subsystems separately.
- I honestly note direct edits to SSL base modules make merge costlier than clean override points.
- After merge I always "Update database configuration" and scenarios from SSL release notes.

## How it works

The team gets SSL 3.1.30 cf and current ERP with filled `Overridable` modules. On an empty copy they load target ERP, then compare or merge via the product vendor update template.

Conflict in `ОбновлениеИнформационнойБазы` is resolved toward the new version, moving custom handlers to the end of the version chain. Project procedures in `OverridablePrint` stay from the custom branch if the signature unchanged; when API changes, fix calls per changelog.

```bsl
Процедура ПриДобавленииОбработчиковОбновления(Обработчики) Экспорт

    // base block after merge
    ОбновлениеИнформационнойБазы.ДобавитьОбработчик(Обработчики, "3.1.30.1", "ОбновитьРолиПользователей");

    // keep project handler explicitly
    ОбновлениеИнформационнойБазы.ДобавитьОбработчик(Обработчики, "3.1.30.99", "Проект_МигрироватьСправочникКонтрагентов");

КонецПроцедуры
```

On the test infobase they start the client; the platform runs deferred handlers. They check print, exchange, RLS, scheduled jobs. Only after sign-off they deploy cf to production in a maintenance window.

## Pitfalls

**Merge on production.** Configuration merge on live infobase without a copy is unacceptable.

**Skipped handlers.** A missed version handler leaves broken constants and roles.

**Manual SSL core edits.** Every library release needs manual re-merge; overridable modules are better.

**Platform mismatch.** New SSL may require minimum 8.3.x.

**Extensions.** After SSL upgrade rebuild and test all cfe separately.
