# Standard Subsystems Library: scope and project role

## In short

The Standard Subsystems Library is a separate library configuration embedded into an application solution. It supplies infrastructure blocks (security, files, exchange, print, updates) and code structure conventions so the application team focuses on accounting algorithms.

## How to answer in an interview

- I list core blocks: users, access groups, RLS, file storage, scheduled jobs, object versioning.
- I explain project role: one standard for 1C product line and partner SSL-based solutions.
- I talk lifecycle: SSL version in configuration readme, changelog, production update procedure.
- I honestly note SSL does not replace domain knowledge; without documents and registers the base product is still a black box.
- I mention tools: "Update handler descriptions", three-way merge, Vanessa or custom scenario tests.

## How it works

At project start the team enables required subsystems in metadata. Application logic calls SSL APIs:

```bsl
Пользователь = Пользователи.ТекущийПользователь();
Если Не ПравоДоступа("Изменение", Метаданные.Документы.РеализацияТоваров) Тогда
    ВызватьИсключение "Нет прав на изменение реализации";
КонецЕсли;

ДвоичныеДанные = РаботаСФайлами.ДвоичныеДанныеФайла(ПрисоединенныйФайл);
```

Subsystems interact: "Attached files" uses "File management", "Users" feeds "Access management". Infobase update runs handlers from `ОбновлениеИнформационнойБазы*` modules in version order.

In a base ERP much of the menu and admin UI comes from SSL. A partner adds industry objects and fills overridable modules without duplicating the roles mechanism.

## Pitfalls

**Casual subsystem removal.** Dropping a subsystem from composition breaks dependent objects at build.

**Version mix.** SSL 3.1.25 in configuration with 3.1.30 docs causes API drift.

**Direct SQL bypassing RLS.** Queries ignoring SSL access limits bypass the security model.

**Ignored update handlers.** Skipping a version during upgrade leaves broken registers and empty constants.

**Team onboarding.** Without SSL training developers build parallel user and file mechanisms.
