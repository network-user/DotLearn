# What is a Kafka topic and why do you need it?

## In short

A topic is a named event log split into partitions and replicated across brokers. Producers append; consumers read by offset; retention and compaction are configured per topic.

## How to answer in an interview

- I name by domain: `payments.events`, `inventory.updates`; ACLs and quotas attach to the topic.
- Partitions set parallelism; replication factor sets fault tolerance.
- Retention by time or size; a compacted topic keeps the latest key for a changelog.
- Honestly: thousands of tiny topics increase metadata overhead; I group by bounded context, not per microservice method.

## How it works

Create a topic with three partitions and RF=3:

```bash
kafka-topics.sh --bootstrap-server kafka.internal:9092 \
  --create --topic orders.events \
  --partitions 3 --replication-factor 3 \
  --config retention.ms=604800000 \
  --config min.insync.replicas=2
```

Publish and read:

```bash
kafka-console-producer.sh --bootstrap-server kafka.internal:9092 --topic orders.events
kafka-console-consumer.sh --bootstrap-server kafka.internal:9092 --topic orders.events --from-beginning
```

Describe shows leader/replica ISR. Tiered storage (if enabled) offloads cold segments to S3. MirrorMaker 2 replicates topics across clusters for DR.

## Pitfalls

- **RF higher than broker count.** Topic create fails or stays under-replicated forever.
- **One partition on a hot topic.** Producer throughput hits a single disk append path.
- **Delete topic without `delete.topic.enable`.** Marked for deletion but disk never clears.
- **Compaction without a key.** Tombstones do not work; disk grows forever.
- **Auto-create in production.** A typo creates default RF=1 and a DR gap.
