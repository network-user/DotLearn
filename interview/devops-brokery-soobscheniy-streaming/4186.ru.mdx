# Что такое Apache Kafka в архитектуре данных?

## Суть

Kafka это распределённый commit log: producers append в topics, brokers реплицируют partitions, consumers pull по offset в consumer groups. Платформа для event streaming, decoupling сервисов, CDC и real-time analytics с horizontal scale.

## Что ответить на собесе

- Компоненты: broker, topic/partition, producer, consumer group, coordinator, (KRaft или ZooKeeper legacy).
- Log retention по time/size, не queue with ack-after-consume; replay возможен.
- Throughput за счёт sequential disk I/O и batching; ordering только per partition.
- Честно добавляю: Kafka не замена БД; long-term source of truth держу в OLTP/warehouse, Kafka как transport layer.

## Как это работает

```bash
kafka-topics.sh --bootstrap-server kafka.internal:9092 \
  --create --topic orders --partitions 6 --replication-factor 3

kafka-console-producer.sh --bootstrap-server kafka.internal:9092 --topic orders <<<'{"id":9001}'
kafka-console-consumer.sh --bootstrap-server kafka.internal:9092 --topic orders --from-beginning --max-messages 1

kafka-consumer-groups.sh --bootstrap-server kafka.internal:9092 --describe --group checkout-workers
```

Topic split на 6 partitions позволяет 6 parallel consumers в одной group. RF=3 переживает 2 broker loss. Consumer group coordinator назначает partition ownership; lag мониторю через `describe`.

## Подводные камни

- **Too few partitions.** Hot topic не масштабируется consumer'ами выше partition count.
- **Key skew.** Один hot key забивает single partition.
- **Unbounded retention.** Disk fill; ставлю retention.ms/bytes и tiered storage policy.
- **Exactly-once myth without idempotent consumers.** EOS только при полной цепочке config.
- **KRaft migration debt.** Legacy ZK ops still в старых кластерах; планирую KRaft roadmap.
