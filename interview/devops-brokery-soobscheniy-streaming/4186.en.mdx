# What is Apache Kafka in a data architecture?

## In short

Kafka is a distributed commit log: producers append to topics, brokers replicate partitions, consumers pull by offset in consumer groups. It powers event streaming, service decoupling, CDC, and real-time analytics with horizontal scale.

## How to answer in an interview

- Components: broker, topic/partition, producer, consumer group, coordinator, (KRaft or legacy ZooKeeper).
- Log retention by time/size, not a queue that deletes on consume; replay is possible.
- Throughput from sequential disk I/O and batching; ordering only per partition.
- I add honestly: Kafka is not a database; long-term source of truth stays in OLTP/warehouse, Kafka is the transport layer.

## How it works

```bash
kafka-topics.sh --bootstrap-server kafka.internal:9092 \
  --create --topic orders --partitions 6 --replication-factor 3

kafka-console-producer.sh --bootstrap-server kafka.internal:9092 --topic orders <<<'{"id":9001}'
kafka-console-consumer.sh --bootstrap-server kafka.internal:9092 --topic orders --from-beginning --max-messages 1

kafka-consumer-groups.sh --bootstrap-server kafka.internal:9092 --describe --group checkout-workers
```

Six partitions allow six parallel consumers in one group. RF=3 survives two broker losses. The group coordinator assigns partition ownership; I monitor lag via `describe`.

## Pitfalls

- **Too few partitions.** Hot topics cannot scale consumers beyond partition count.
- **Key skew.** One hot key saturates a single partition.
- **Unbounded retention.** Disk fills; set retention.ms/bytes and tiered storage policy.
- **Exactly-once myth without idempotent consumers.** EOS needs full chain configuration.
- **KRaft migration debt.** Legacy ZK ops remain in old clusters; plan KRaft roadmap.
