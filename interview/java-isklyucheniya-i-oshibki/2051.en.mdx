# Java exception hierarchy: Throwable and subclasses

## In short

Root: `Throwable` (`Error`, `Exception`). `Exception` splits into checked (subclasses of `Exception` except `RuntimeException`) and unchecked (`RuntimeException` and descendants). Checked forces compile-time handling. `Error` reflects fatal platform failures. Knowing the tree matters for catch order and API exception design.

## How to answer in an interview

- `Throwable` → `Error` | `Exception` → `RuntimeException` (unchecked) | other `Exception` (checked).
- Catch from specific to general: `FileNotFoundException` before `IOException`.
- Custom domain exceptions extend `RuntimeException` in the service layer.
- `Throwable.initCause()` links a wrapper to the root cause.
- Honestly: deep custom exception hierarchies rarely pay off; prefer flat types plus a code field.

## How it works

Checked:

```java
public Invoice parseInvoice(Path xmlPath) throws IOException, SAXException {
    var factory = DocumentBuilderFactory.newInstance();
    var builder = factory.newDocumentBuilder();
    var doc = builder.parse(xmlPath.toFile());
    return mapToInvoice(doc);
}
```

The caller must handle or declare.

Unchecked:

```java
public Money requirePositive(Money amount) {
    if (amount.isNegativeOrZero()) {
        throw new IllegalArgumentException("amount must be positive: " + amount);
    }
    return amount;
}
```

Tree (simplified):

```
Throwable
├── Error (OutOfMemoryError, StackOverflowError)
└── Exception
    ├── RuntimeException (NPE, IllegalStateException)
    └── checked (IOException, SQLException)
```

Multi-catch needs unrelated types:

```java
try {
    process();
} catch (IOException | SQLException ex) {
    throw new DataAccessException("storage failed", ex);
}
```

You cannot use `IOException | Exception` because IOException IS-A Exception.

## Pitfalls

- **catch (Exception) before a specific type.** Dead code; compiler error.
- **Checked in Spring @Service.** Often wrapped to unchecked for a cleaner stack.
- **Error in a custom hierarchy.** Confused with real JVM Errors.
- **Empty catch blocks.** Swallow checked and unchecked equally badly.
- **Serialization of old custom exceptions.** Forgot serialVersionUID after field changes.
