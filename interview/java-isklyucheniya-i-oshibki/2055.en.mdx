# Checked vs unchecked: how to tell them apart

## In short

Checked: subclass of `Exception` but not `RuntimeException`. The compiler requires try/catch or `throws`. Unchecked: `RuntimeException` and subclasses (plus `Error`). Practical rule: external recoverable conditions (I/O, DB, parse) are often checked; programming bugs (NPE, illegal state) are unchecked.

## How to answer in an interview

- I check the hierarchy: `instanceof RuntimeException` → unchecked.
- `IOException`, `SQLException`, `ClassNotFoundException` are checked.
- `NullPointerException`, `IllegalArgumentException` are unchecked.
- Custom domain types for REST APIs usually `extend RuntimeException`.
- Honestly: checked vs unchecked in a new API depends on whether the caller can actually recover.

## How it works

Checked requires handling:

```java
void exportReport(Path target) throws IOException {
    var rows = repository.findAll();
    Files.writeString(target, renderCsv(rows));
}
```

Without `throws` or try/catch the project will not compile.

Unchecked need not be declared:

```java
Money debit(Account account, Money amount) {
    if (amount.isNegative()) {
        throw new IllegalArgumentException("amount must be non-negative");
    }
    return account.debit(amount);
}
```

You may catch it, but the compiler does not require it:

```java
try {
    debit(account, Money.of(-1));
} catch (IllegalArgumentException ex) {
    return ValidationError.of(ex.getMessage());
}
```

Quick IDE/javadoc check: if the supertype chain includes `RuntimeException`, it is unchecked.

## Pitfalls

- **Checked in lambda/stream.** Needs a wrapper like `UncheckedIOException`.
- **extends Exception without reason.** Forces every caller to handle; API gets heavy.
- **RuntimeException for I/O.** Callers may forget catch; unhandled 500s.
- **Confusing Error with Exception.** `catch (Exception)` does not catch OOM.
- **@SneakyThrows (Lombok).** Hides checked exceptions; breaks the contract for consumers.
