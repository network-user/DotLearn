# Types of Java exceptions: checked, unchecked, errors

## In short

Java splits `Throwable` into three practical categories: checked exceptions (compiler enforces handling), unchecked exceptions (`RuntimeException`, logic bugs), and errors (`Error`, fatal JVM failures). The mechanism separates happy path from error handling via try/catch/finally and throws. Goal: controlled failure instead of silent crash.

## How to answer in an interview

- Checked: `IOException`, `SQLException`, `TimeoutException` (when not runtime).
- Unchecked: `NPE`, `IllegalStateException`, `ConcurrentModificationException`.
- Errors: `OutOfMemoryError`, `StackOverflowError`, `NoClassDefFoundError`.
- I do not catch errors in production code; I fix unchecked issues with tests and validation.
- Honestly: catching `StackOverflowError` and continuing is pointless; fix the recursion.

## How it works

Checked file read:

```java
public String readToken(Path secrets) throws IOException {
    return Files.readString(secrets).trim();
}
```

Unchecked validation:

```java
public UUID requireUuid(String raw) {
    try {
        return UUID.fromString(raw);
    } catch (IllegalArgumentException ex) {
        throw new BadRequestException("Invalid uuid: " + raw, ex);
    }
}
```

Error scenario (do not handle):

```java
void brokenRecursion() {
    brokenRecursion();
}
```

This ends in `StackOverflowError`. The right fix is the algorithm, not a bigger stack (except as a temporary measure).

Summary:

| Kind | Compiler check | Typical example | Handling |
|-----|----------------|-----------------|-----------|
| Checked | yes | IOException | try/catch or throws |
| Unchecked | no | NullPointerException | fix + optional catch |
| Error | no | OutOfMemoryError | do not catch in app code |

## Pitfalls

- **catch (Throwable) catch-all.** Masks Error and OOM.
- **Confusing Exception vs RuntimeException.** Not every Exception is checked.
- **Custom extends Error.** Extremely rare and confusing.
- **Catching Error in a library.** Hides non-recoverable JVM state.
- **Same handling for I/O and NPE.** Different recovery semantics.
