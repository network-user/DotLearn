# Generics in catch blocks: why they are not allowed

## In short

Java does not allow `catch (MyException<String> ex)` or parameterized catch types. Type erasure removes generic parameters at compile time; the JVM matches catch only on the raw class. Exceptions dispatch by runtime class of the object, not by generic signature. Put typed payload in exception fields, not in a generic catch.

## How to answer in an interview

- Catch works on the class after erasure: `MyException<T>` → `MyException`.
- The compiler rejects `catch (ValidationException<User> e)`.
- For typed context I use fields like `ErrorCode code` or `Class<?> entityType`.
- Multi-catch also uses only raw or concrete nested classes.
- Honestly: generic exception classes rarely help; prefer composition inside.

## How it works

Does not compile:

```java
// catch (ApiException<PaymentError> ex) { }  // compile error
```

Working pattern:

```java
public final class ApiException extends RuntimeException {
    private final ErrorCode code;
    private final String entityId;

    public ApiException(ErrorCode code, String entityId, String message) {
        super(message);
        this.code = code;
        this.entityId = entityId;
    }

    public ErrorCode code() { return code; }
    public String entityId() { return entityId; }
}

void handle(ApiException ex) {
    switch (ex.code()) {
        case PAYMENT_DECLINED -> retryWithBackoff(ex.entityId());
        case VALIDATION_FAILED -> rejectRequest(ex.getMessage());
    }
}
```

Erasure demo:

```java
List<String> names = List.of("a");
List<Integer> ids = List.of(1);
// names.getClass() == ids.getClass() → true, both List.class
```

Likewise `MyException<String>` and `MyException<Integer>` share one runtime class for catch.

## Pitfalls

- **Wildcard catch workaround.** `catch (MyException<?> e)` is also illegal in Java.
- **instanceof with generics (Java 16+).** Pattern matching on type, not generic parameter.
- **Serializing generic exceptions.** Generic fields erase; deserialization surprises.
- **Per-type exception duplication.** `UserNotFoundException`, `OrderNotFoundException` instead of one generic.
- **Confusion with union types.** Java catch does not support unions like some languages.
