# Иерархия исключений Java: Throwable и наследники

## Суть

Корень: `Throwable` (`Error`, `Exception`). `Exception` делится на checked (наследники `Exception`, кроме `RuntimeException`) и unchecked (`RuntimeException` и потомки). Checked заставляет компилятор проверить обработку. `Error` отражает фатальные сбои платформы. Понимание дерева нужно для правильного catch order и проектирования API exceptions.

## Что ответить на собесе

- `Throwable` → `Error` | `Exception` → `RuntimeException` (unchecked) | остальные `Exception` (checked).
- Catch от специфичного к общему: `FileNotFoundException` перед `IOException`.
- Собственные доменные исключения наследую от `RuntimeException` для service layer.
- `Throwable.initCause()` связывает wrapper с root cause.
- Честно: глубокие иерархии custom exceptions редко окупаются, prefer flat + code field.

## Как это работает

Checked:

```java
public Invoice parseInvoice(Path xmlPath) throws IOException, SAXException {
    var factory = DocumentBuilderFactory.newInstance();
    var builder = factory.newDocumentBuilder();
    var doc = builder.parse(xmlPath.toFile());
    return mapToInvoice(doc);
}
```

Caller обязан обработать или пробросить.

Unchecked:

```java
public Money requirePositive(Money amount) {
    if (amount.isNegativeOrZero()) {
        throw new IllegalArgumentException("amount must be positive: " + amount);
    }
    return amount;
}
```

Дерево (упрощённо):

```
Throwable
├── Error (OutOfMemoryError, StackOverflowError)
└── Exception
    ├── RuntimeException (NPE, IllegalStateException)
    └── checked (IOException, SQLException)
```

Multi-catch требует несвязанных типов:

```java
try {
    process();
} catch (IOException | SQLException ex) {
    throw new DataAccessException("storage failed", ex);
}
```

Нельзя `IOException | Exception`, потому что IOException IS-A Exception.

## Подводные камни

- **catch (Exception) перед конкретным.** Мёртвый код, компилятор ошибка.
- **Checked в Spring @Service.** Часто оборачивают в unchecked для чистоты стека.
- **Error в иерархии custom.** Путаница с настоящими JVM Error.
- **Пустые catch блоки.** Проглатывают checked и unchecked одинаково плохо.
- **Serialization старых custom exceptions.** serialVersionUID забыли при изменении полей.
