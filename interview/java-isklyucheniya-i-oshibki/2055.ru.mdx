# Checked vs unchecked: как отличить исключение

## Суть

Checked: наследник `Exception`, но не `RuntimeException`. Компилятор требует try/catch или `throws`. Unchecked: `RuntimeException` и подклассы (плюс `Error`). Практическое правило: внешние recoverable условия (I/O, DB, parse) часто checked; programming bugs (NPE, illegal state) unchecked.

## Что ответить на собесе

- Смотрю иерархию: `instanceof RuntimeException` → unchecked.
- `IOException`, `SQLException`, `ClassNotFoundException` checked.
- `NullPointerException`, `IllegalArgumentException` unchecked.
- Custom domain: для REST API обычно `extends RuntimeException`.
- Честно: спор checked vs unchecked в новом API решаю по тому, может ли caller реально восстановиться.

## Как это работает

Checked требует обработки:

```java
void exportReport(Path target) throws IOException {
    var rows = repository.findAll();
    Files.writeString(target, renderCsv(rows));
}
```

Без `throws` или try/catch проект не скомпилируется.

Unchecked не обязан объявляться:

```java
Money debit(Account account, Money amount) {
    if (amount.isNegative()) {
        throw new IllegalArgumentException("amount must be non-negative");
    }
    return account.debit(amount);
}
```

Можно поймать, но компилятор не заставляет:

```java
try {
    debit(account, Money.of(-1));
} catch (IllegalArgumentException ex) {
    return ValidationError.of(ex.getMessage());
}
```

Быстрая проверка в IDE/javadoc: если supertype chain включает `RuntimeException`, это unchecked.

## Подводные камни

- **Checked в lambda/stream.** Нужен wrapper `UncheckedIOException`.
- **extends Exception без причины.** Заставляет всех callers handle, API тяжелеет.
- **RuntimeException для I/O.** Caller может забыть catch, необработанный 500.
- **Error перепутали с Exception.** `catch (Exception)` не ловит OOM.
- **@SneakyThrows (Lombok).** Скрывает checked, ломает контракт для потребителей.
