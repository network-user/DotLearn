# Error vs Exception in the Throwable hierarchy

## In short

`Throwable` splits into `Error` (JVM/resource failures, usually not caught) and `Exception` (conditions the application can handle). `Exception` further divides into checked (compiler requires try/catch or throws) and unchecked (`RuntimeException` and subclasses). `Error` signals problems outside business logic: memory, stack overflow, internal VM faults.

## How to answer in an interview

- `OutOfMemoryError`, `StackOverflowError`, `VirtualMachineError` are not caught in application code.
- `IOException`, `SQLException` are checked: declare `throws` or handle.
- `NullPointerException`, `IllegalArgumentException` are unchecked: fix the code, do not swallow with a wide catch.
- In API design I prefer unchecked for programming errors, checked for external I/O.
- Honestly: catching `Error` and continuing almost always masks a non-recoverable state.

## How it works

Checked I/O:

```java
public List<String> loadLines(Path configPath) throws IOException {
    return Files.readAllLines(configPath);
}

void bootstrap() {
    try {
        var lines = loadLines(Path.of("app.conf"));
        applyConfig(lines);
    } catch (IOException ex) {
        throw new IllegalStateException("Config unreadable", ex);
    }
}
```

Unchecked programming bug:

```java
int indexOfPositive(int[] values, int target) {
    if (values == null) {
        throw new IllegalArgumentException("values must not be null");
    }
    for (int i = 0; i < values.length; i++) {
        if (values[i] == target) {
            return i;
        }
    }
    throw new NoSuchElementException("target not found");
}
```

`Error` example (not caught in business code):

```java
void leakMemory() {
    List<byte[]> chunks = new ArrayList<>();
    while (true) {
        chunks.add(new byte[10_000_000]);
    }
}
```

This triggers `OutOfMemoryError`; the right response is JVM flags, heap dump analysis, fix the leak, not `catch (OutOfMemoryError)`.

## Pitfalls

- **catch (Exception) swallows RuntimeException.** Hides bugs instead of fail-fast.
- **throws Exception on public API.** Blurs the method contract.
- **Error in a retry loop.** Pointless retries on OOM.
- **Checked in streams/lambdas.** Needs a wrapper or it will not compile.
- **Throwable in catch.** Catches both Error and Exception: anti-pattern in the application layer.
