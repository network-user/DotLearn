# merge или rebase: два способа собрать ветки и чем они расплачиваются

## Суть

`merge` и `rebase` решают одну задачу - перенести наработки из одной ветки в другую, но по-разному. `merge` сохраняет историю как есть и добавляет коммит слияния, а `rebase` переписывает коммиты заново, выпрямляя историю в одну линию.

## Что ответить на собесе

Обе команды переносят изменения между ветками, но дают разную историю. `merge` создаёт отдельный коммит слияния с двумя родителями и оставляет точки ветвления видимыми - вся картина сохраняется. `rebase` снимает коммиты текущей ветки и накладывает их поверх вершины другой, история становится линейной, но хеши коммитов меняются. Поэтому `rebase` нельзя применять к веткам, которые уже опубликованы и видны другим: переписанная история сломает им работу. Линейность удобна для чтения, разветвлённость честнее отражает то, как шла разработка.

`merge` сводит две ветки и фиксирует результат отдельным коммитом. У этого коммита два родителя: один указывает на текущую ветку, второй на присоединяемую.

```bash
git checkout main
git merge feature
```

История сохраняет ответвление:

```
A---B---C---D  (main)
     \     /
      E---F  (feature)
```

`rebase` действует иначе: он снимает изменения текущей ветки и заново применяет их поверх вершины целевой, будто работа велась уже после её последнего коммита. Хеши при этом пересчитываются - на схеме это `E'` и `F'`.

```bash
git checkout feature
git rebase main
```

```
A---B---C---D  (main)
              \
               E'---F'  (feature)
```

Выбор по ситуации: `merge`, когда важна полная прозрачность и трассируемость каждого слияния; `rebase`, когда нужна плоская читаемая история, например перед отправкой работы в общий репозиторий.

## Подводные камни

- Конфликты возможны в обоих случаях, но при `rebase` их иногда приходится решать по нескольку раз - коммиты применяются по одному поверх новой базы.
- `rebase` меняет хеши, поэтому его нельзя применять к публичным веткам: после `git push --force` коллеги получат расхождение истории.
- Коммит слияния от `merge` плодит "развилки" в истории; в крупных проектах их обилие затрудняет чтение `git log`.
- Линейность после `rebase` обманчиво проста: реальный порядок и контекст работы в ней теряется.
