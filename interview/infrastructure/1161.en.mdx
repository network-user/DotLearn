# Why kill exists and why it is not only about killing processes

## In short

Despite the name, `kill` does not "kill" — it delivers a signal to a process by its PID. Termination is only one scenario: the same mechanism is used to restart processes, reload config, and pause them.

## How to answer in an interview

`kill` sends a signal to a process by its identifier (PID). A signal is a number that the process reacts to with a predefined action. The two termination variants used most often: `SIGTERM` (15) — a polite request to shut down, which the process can intercept and clean up gracefully, and `SIGKILL` (9) — a forced removal that can be neither intercepted nor ignored, because the kernel itself handles it. The right order is `SIGTERM` first, and only `SIGKILL` if the process hangs.

## How it works

A signal is addressed to a process by PID. Some signals can be intercepted by a handler in code, but `SIGKILL` and `SIGSTOP` cannot — they are always carried out by the kernel.

A hard removal when a process is unresponsive:

```bash
kill -9 1234
```

- `-9` is `SIGKILL`, instant termination with no chance to clean up.

A graceful termination that lets the process close files and connections:

```bash
kill -15 1234   # or simply: kill 1234
```

- `-15` is `SIGTERM`, the default signal when no number is given.

Reload the configuration without restarting (typical for daemons):

```bash
kill -1 1234
```

- `-1` is `SIGHUP`. Many services (nginx, sshd) pick up a new config on it.

A typical workflow when a service hangs:

```bash
pgrep myserver        # find the PID by name
kill 1234             # politely first (SIGTERM)
kill -9 1234          # if unresponsive — forcibly
```

## Pitfalls

- Reaching for `kill -9` right away is a bad habit: the process has no time to close files, flush buffers, and release locks, so you can end up with corrupted data.
- `SIGKILL` (9) and `SIGSTOP` cannot be intercepted or blocked — which is why they are used to remove hung processes, but their termination handler will not run either.
- `kill -9 -1` or `kill 0` hit an entire process group at once rather than a single process — it is easy to bring down more than intended.
- You cannot remove another user's process without permission: you need the same user or root.
- `kill` sends a signal by PID, while `killall`/`pkill` work by name; confusing them is dangerous, since a name may match several processes.
