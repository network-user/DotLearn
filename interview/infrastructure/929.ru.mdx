# Статическая типизация в CI: как заставить пайплайн проверять типы автоматически?

## Суть

Проверку типов встраивают как отдельный шаг пайплайна, который запускает статический анализатор (для Python - mypy) на каждый push и pull request. Если типы не сходятся, шаг падает, и проблемный код не доезжает до merge.

## Что ответить на собесе

Добавляю в код аннотации типов, а в CI - отдельную job, которая ставит mypy и прогоняет его по проекту. Главная ценность не в самой проверке, а в том, что она обязательна и автоматическая: разработчик не может «забыть» прогнать анализатор локально, потому что красный пайплайн блокирует слияние. Это ловит целый класс ошибок (неверный тип аргумента, None там, где ждали объект) до запуска кода и до code review, экономя время ревьюера.

Аннотации - это контракт, который mypy проверяет на согласованность без выполнения программы:

```python
def add(a: int, b: int) -> int:
    return a + b
```

Дальше заводим job в пайплайне. Для GitHub Actions кладём файл `.github/workflows/mypy.yml`:

```yaml
name: Type Check

on: [push, pull_request]

jobs:
  type-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install mypy
        run: |
          python -m pip install --upgrade pip
          pip install mypy

      - name: Run mypy
        run: mypy src/
```

Разбор по строкам:

- `on: [push, pull_request]` - триггеры: каждый коммит и каждый PR.
- `runs-on: ubuntu-latest` - чистый раннер для воспроизводимости.
- шаг `checkout` забирает код, `setup-python` фиксирует версию интерпретатора.
- `mypy src/` возвращает ненулевой код выхода при ошибке, и job становится красной.

Аналогично для GitLab это добавление stage в `.gitlab-ci.yml` с тем же `mypy src/`.

## Подводные камни

- **Зелёный mypy не значит «нет багов».** Статическая проверка ловит только несоответствия типов, а не логику; она дополняет тесты, а не заменяет их.
- **Внешние библиотеки без аннотаций.** Многие пакеты не типизированы; mypy будет ругаться, пока не поставите stub-пакеты (`types-...`) или не настроите `ignore_missing_imports`.
- **Версия Python.** Поведение проверки зависит от целевой версии; если в CI и локально она разная, результаты разойдутся - фиксируйте её явно.
- **Внедрение на старом коде.** Сразу включить строгий режим на большом нетипизированном проекте больно; обычно начинают с мягких настроек и постепенно ужесточают, иначе пайплайн утонет в ошибках.
