# Logging in Docker: stdout, logging drivers, and centralized collection

## In short

The canonical approach is to write application logs to `stdout`/`stderr`, while the logging driver (`json-file`, `syslog`, `fluentd`, and others) decides how to store them and where to send them. At scale, logs are pulled into centralized systems like ELK or Fluentd.

## How to answer in an interview

An application in a container should not write to files itself - it writes to the standard streams, and Docker captures them. From there everything is determined by the logging driver: by default it's `json-file`, which stores logs on the host as JSON, and you must give it rotation via `max-size`/`max-file`, otherwise the disk fills up. I view logs with `docker logs`, and in production I switch the driver to `fluentd`/`syslog` or collect everything into ELK for centralized search and analysis.

## How it works

The most reliable path is for the application to print to `stdout` and `stderr` with nothing extra to configure: Docker picks up these streams itself and passes them to the logging driver.

The driver decides where the records go. Popular options:

- **json-file** - JSON files on the host (the default);
- **syslog** - to a system log server;
- **journald** - to the `systemd` journal;
- **fluentd** - to Fluentd for further processing;
- **awslogs** - to Amazon CloudWatch.

The driver and its options are set in the compose file. Rotation is configured here too:

```yaml
version: '3.8'
services:
  my_app:
    image: my_app_image
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"
```

`max-size` caps a single file at 10 megabytes, `max-file` keeps at most 3 files - older ones are deleted once exceeded. To view logs:

```bash
docker logs <container_id_or_name>
```

## Pitfalls

- Without rotation, `json-file` writes endlessly and eventually fills up the host's entire disk - `max-size`/`max-file` are mandatory.
- `docker logs` works only with the `json-file`, `journald`, and `local` drivers. With `syslog`/`fluentd`/`awslogs` the command returns an error - the logs already live in an external system.
- Don't write logs to a file inside the container: the data disappears when the container is recreated and won't show up in `docker logs`.
- Writing to `stdout` is synchronous - a very "chatty" application under load can slow down because it blocks on output.
- The global driver is set in `daemon.json`, and it applies to all new containers; it does not affect already-running ones.
