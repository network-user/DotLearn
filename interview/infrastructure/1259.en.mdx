# Virtual filesystems vs device files: the roles of /proc, /sys, and /dev

## In short

`/proc` and `/sys` are virtual filesystems: they take up no disk and serve kernel data on the fly. `/proc` is about processes and system parameters, `/sys` is about devices and drivers. `/dev` stands apart - these are device files through which programs access hardware as ordinary files.

## How to answer in an interview

All three directories look like ordinary folders, but they differ behind the scenes. `/proc` is a virtual filesystem with information about processes (`/proc/<PID>`) and the system (`/proc/cpuinfo`, `/proc/meminfo`), plus kernel parameters tunable on the fly in `/proc/sys`. `/sys` is also a virtual filesystem, but it's a structured interface to devices and drivers through the kernel model (sysfs). And `/dev` is device nodes: block (`/dev/sda`), character (`/dev/tty`), and pseudo-devices (`/dev/null`), through which actual I/O to the hardware goes. In short: `/proc` and `/sys` are for reading and tuning, `/dev` is for accessing the devices themselves.

## How it works

**The /proc directory.** A virtual filesystem whose contents are generated by the kernel on read and don't reside on disk.

- **Processes:** each PID has its own subdirectory. `/proc/1234` holds the data of the process with PID 1234.
- **System state:** `/proc/cpuinfo` is about the CPU, `/proc/meminfo` is about memory.
- **Kernel parameters:** via `/proc/sys` you change settings on the fly (often more convenient through `sysctl`).

```bash
cat /proc/cpuinfo
```

Outputs the processor model, clock speed, and core count.

**The /sys directory.** Also a virtual filesystem (sysfs), but it hierarchically reflects the tree of kernel devices and drivers.

- **Devices:** `/sys/class`, `/sys/block` - data about devices and their drivers.
- **Attributes:** let you read and change parameters, for example enabling a device by writing to an attribute file.

```bash
cat /sys/class/net/eth0/address
```

Shows the MAC address of the `eth0` interface.

**The /dev directory.** Device nodes - an interface to hardware through file semantics.

- **Block:** `/dev/sda` - a disk, accessed in blocks.
- **Character:** `/dev/tty` - a terminal, accessed as a byte stream.
- **Pseudo-devices:** `/dev/null` swallows input, `/dev/zero` yields zeros.

```bash
ls -l /dev/sda
```

The first character of the output, `b` or `c`, immediately tells you whether it's a block or character device.

## Pitfalls

- **`/proc` and `/sys` are not on disk.** Their size in `df` is zero - they are kernel interfaces, not storage. A reboot wipes any changes, so `sysctl` edits are persisted in `/etc/sysctl.conf`.
- **Writing can be dangerous.** These filesystems aren't read-only: writing to `/proc/sys` or `/sys` changes kernel behavior in real time. A wrong value can hang or destabilize the system.
- **`/dev` is dynamic nowadays.** Nodes are created by `udev`/`devtmpfs` as devices are attached, not a static list. Names like `sda` may change between boots - for stability use `/dev/disk/by-uuid`.
- **Role confusion.** `/proc/<PID>` is about processes, not devices; devices live in `/sys` and `/dev`. In interviews this is a frequent stumbling point.
