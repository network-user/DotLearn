# Docker volumes: surviving container removal without losing data

## In short

A volume is data storage managed by Docker that lives separately from the container's filesystem. That is why the data survives restarting and removing the container, and a single volume can be attached to several containers at once.

## How to answer in an interview

A container's writable layer is ephemeral: remove the container and everything it wrote to its filesystem is gone. For data that must outlive the container (databases, uploads, logs), you use a volume. It is Docker-managed storage: its lifecycle is not tied to the container, it can be reattached to a new container after a rebuild, and it can be shared between several. A volume differs from a bind mount in that Docker itself is responsible for where it lives and how it is managed (`docker volume create/ls/rm`), whereas a bind mount is hard-wired to a specific path on the host. For production data, a volume is almost always preferable.

When you create a volume, Docker provisions storage for it in its own area on the host (on Linux, usually under `/var/lib/docker/volumes/`). You mount the volume into a container by specifying a path inside the container, and everything written there lands in the volume.

```bash
# Create a named volume
docker volume create app_data

# Mount it into the container at /app/data
docker run -d \
  --name web \
  -v app_data:/app/data \
  my_image
```

Whatever the container writes to `/app/data` is stored in `app_data` and remains available after `docker rm web`. Start a new container with the same `-v app_data:/app/data` and you get the same data.

It helps to distinguish three mount types:

- **volume** (`-v name:/path`) — managed by Docker, the recommended way to store persistent data.
- **bind mount** (`-v /host/path:/path`) — exposes a specific host directory; convenient for development.
- **tmpfs** — kept in memory, not persisted to disk; for temporary secrets and caches.

## Pitfalls

- **Without a volume, data is ephemeral.** Writes to the container's normal layer are lost on `docker rm`. A database running in a container without a volume is easy to wipe with an accidental rebuild.
- **Anonymous volumes pile up.** `-v /path` without a name creates an anonymous volume on every run; they accumulate as junk. Clean them up with `docker volume prune`.
- **A volume is not removed with the container by default.** You need the `docker rm -v` flag or an explicit `docker volume rm`; otherwise the data keeps occupying space.
- **A bind mount shadows existing content.** Mounting onto a non-empty container directory hides whatever was at that path in the image.
- **A volume is not a backup.** It is storage, not a backup copy; a volume backup strategy must be set up separately.
