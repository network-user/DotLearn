# Docker daemon: who actually runs the containers?

## In short

The Docker daemon (`dockerd`) is a persistent background process on the host that does all the work: it creates, starts, and removes containers, pulls images, and serves the Docker API. Commands from the terminal only ask it to do this.

## How to answer in an interview

Docker is built on a client-server model. When you type `docker run`, the `docker` binary is just a client: it doesn't run the container itself but sends an HTTP request to the `dockerd` daemon over a socket. The daemon is the server that holds the state, talks to image registries, manages networks and volumes, and through the underlying runtime (containerd → runc) actually spins up the container processes. So if the daemon is stopped, no `docker` command will work, even though the client itself is installed and available.

The picture is easier to keep in layers:

- **Docker Client** - the CLI (`docker`). Translates your command into a Docker API call.
- **Docker Daemon (`dockerd`)** - the server. Listens on a socket (`/var/run/docker.sock` or TCP), executes requests, runs the container lifecycle, caches images.
- **Images** - read-only templates from which containers are born.
- **Containers** - running, isolated instances of images.

Let's trace the path of a single command:

```bash
docker run hello-world
```

1. The client builds the request "create and run a container from the `hello-world` image" and sends it to the daemon over the socket.
2. The daemon checks whether the image exists locally. If not, it pulls it from the registry (Docker Hub) and caches it.
3. Through containerd/runc the daemon creates the container, sets up namespaces, cgroups, and the network, and starts the process.
4. The daemon returns the container's output to the client, which prints it to the terminal.

The key idea: the client and the daemon are different processes and can live on different machines. By pointing `DOCKER_HOST` at a remote TCP endpoint, the local CLI manages a daemon on another server.

## Pitfalls

- **`docker` ≠ Docker.** The CLI is useless without a running daemon; the error "Cannot connect to the Docker daemon" means that `dockerd` specifically isn't running, not that the client is missing.
- **The daemon socket means root access.** Whoever can write to `/var/run/docker.sock` effectively has root on the host. Adding a user to the `docker` group should be a deliberate decision.
- **The daemon holds state.** Images, volumes, and networks live in the daemon. Restarting `dockerd` doesn't wipe them, but it stops all managed containers (unless live-restore is configured).
- **The daemon is a single point.** A `dockerd` crash affects all containers at once; in production this is mitigated with an orchestrator and restart policies.
