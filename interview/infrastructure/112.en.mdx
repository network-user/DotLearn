# Git Flow: branching discipline with long-lived and temporary branches

## In short

Git Flow is an agreement on how a team works with branches. It splits branches into permanent (stable code) and temporary (for features, releases, and urgent fixes), assigning each a role and merge rules.

## How to answer in an interview

Git Flow is a branching model proposed by Vincent Driessen. At its core are two long-lived branches: `main` with stable release code and `develop` as the integration branch where finished features converge. On top of them live temporary branches: feature branches grow out of `develop` and return into it; release branches split off from `develop` for final polishing and merge into `main` and `develop`; hotfix branches come from `main` for urgent fixes and also return into both. The model gives a predictable process and fits large teams and scheduled releases well, but for frequent deployments it's too heavy - there, a trunk-based approach is more often chosen.

Permanent branches:

- `main` (or `master`) - stable, release-ready code; all releases ship from here.
- `develop` - the integration branch where all new work is gathered before it lands in `main`.

Temporary branches and their lifecycle:

```bash
git checkout develop
git checkout -b feature/awesome-feature
```

```bash
git checkout develop
git checkout -b release/1.0.0
```

```bash
git checkout main
git checkout -b hotfix/urgent-fix
```

The full cycle looks like this: a developer branches a feature off `develop`, writes code, and merges it back into `develop`. When a set of features is ready, a release branch is created from `develop`, final tests and fixes are run, then it's merged into `main` and `develop` and a version tag is set. If a critical bug surfaces in production, a hotfix is started off `main`, and the fix returns to both `main` and `develop`.

## Pitfalls

- Two permanent branches and tags require discipline: a forgotten merge of a hotfix back into `develop` will silently reintroduce an already-fixed bug.
- For CI/CD with frequent releases, Git Flow is overkill - long branches diverge and accumulate conflicts.
- A release branch must merge into both permanent branches, otherwise `main` and `develop` will drift apart.
- The model is a convention, not a Git command: people uphold it, and Git won't stop a violation of the branching rules in any way.
