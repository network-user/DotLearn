# Linux: which commands show what's currently running on the system?

## In short

For a static snapshot of processes you use `ps` (for example `ps aux`), and for a live, continuously updating monitor you use `top`, or its convenient colored version `htop`. A snapshot answers "what was running at this moment", a monitor answers "what's happening right now".

## How to answer in an interview

If you need a one-off list of all processes with their owners and resource usage - `ps aux`, often paired with `grep` to find a specific process. If you need to watch the load over time (who's eating CPU and memory right now) - `top` or `htop`. The difference is fundamental: `ps` prints a single frame and exits, while `top`/`htop` are interactive monitors that refresh the picture every couple of seconds and let you sort processes or kill the one you want on the spot.

## How it works

The `ps` (process status) command prints a snapshot of processes at the moment it runs:

```bash
ps aux
```

- `a` - processes of all users, not just the current one;
- `u` - a readable format with owner, CPU share, and memory;
- `x` - including processes without a controlling terminal (daemons).

The output has columns USER, PID, %CPU, %MEM, start time, and the command itself. To find a specific process, you combine it with `grep`:

```bash
ps aux | grep nginx
```

The `top` command shows the same thing, but in real time: a system summary at the top (load average, memory, process count) and a list below that keeps updating. Inside: `q` - quit, `k` - send a signal to a process by PID, `M`/`P` - sort by memory/CPU.

`htop` is an improved `top`: a colored interface, navigation by arrow keys and mouse, convenient filtering. It usually has to be installed separately:

```bash
sudo apt-get install htop   # Debian/Ubuntu
sudo dnf install htop       # Fedora/RHEL
```

## Pitfalls

- **`ps aux` is a snapshot, not a monitor.** From a single call you can't judge a load spike: the process might have loaded the CPU a millisecond before or after.
- **`%CPU` in `ps` is an averaged value** over the process's lifetime, not an instantaneous one; for the current load, look at `top`.
- **`htop` isn't available everywhere.** On a bare server or in a minimal container you'll have to install it, and you may not have the rights - then `top` is what's left.
- **Searching with grep catches grep itself.** `ps aux | grep python` often shows a line with the grep process itself; this is confusing when counting.
