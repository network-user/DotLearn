# Docker daemon: кто на самом деле запускает контейнеры?

## Суть

Docker daemon (`dockerd`) — это постоянный фоновый процесс на хосте, который и делает всю работу: создаёт, запускает и удаляет контейнеры, тянет образы и обслуживает Docker API. Команды из терминала только просят его об этом.

## Что ответить на собесе

Docker устроен по схеме клиент-сервер:

- Когда вы набираете `docker run`, бинарь `docker` — это всего лишь клиент: он не запускает контейнер сам, а отправляет HTTP-запрос демону `dockerd` через сокет.
- Демон — это сервер, который держит состояние, общается с реестрами образов, управляет сетями и томами и через нижележащий рантайм (containerd → runc) реально поднимает процессы контейнеров.
- Поэтому, если демон остановлен, ни одна команда `docker` не сработает, хотя сам клиент при этом установлен и доступен.

Картинку проще держать по слоям:

- **Docker Client** — CLI (`docker`). Переводит вашу команду в вызов Docker API.
- **Docker Daemon (`dockerd`)** — сервер. Слушает сокет (`/var/run/docker.sock` или TCP), исполняет запросы, ведёт жизненный цикл контейнеров, кэширует образы.
- **Образы** — шаблоны только для чтения, из которых рождаются контейнеры.
- **Контейнеры** — запущенные изолированные экземпляры образов.

Проследим путь одной команды:

```bash
docker run hello-world
```

1. Клиент формирует запрос «создать и запустить контейнер из образа `hello-world`» и шлёт его демону через сокет.
2. Демон проверяет, есть ли образ локально. Нет — тянет его из реестра (Docker Hub) и кэширует.
3. Демон через containerd/runc создаёт контейнер, настраивает namespaces, cgroups, сеть и запускает процесс.
4. Вывод контейнера демон возвращает клиенту, а тот печатает его в терминал.

Ключевая мысль: клиент и демон — разные процессы и могут жить на разных машинах. Указав `DOCKER_HOST` на удалённый TCP-эндпоинт, локальный CLI управляет демоном на другом сервере.

## Подводные камни

- **`docker` ≠ Docker.** CLI без работающего демона бесполезен; ошибка «Cannot connect to the Docker daemon» означает, что не запущен именно `dockerd`, а не отсутствует клиент.
- **Сокет демона — это root-доступ.** Кто может писать в `/var/run/docker.sock`, фактически имеет root на хосте. Добавлять пользователя в группу `docker` нужно осознанно.
- **Демон держит состояние.** Образы, тома, сети живут в демоне. Перезапуск `dockerd` не стирает их, но останавливает все управляемые контейнеры (если не настроен live-restore).
- **Демон — единая точка.** Падение `dockerd` затрагивает все контейнеры сразу; в проде это закрывают оркестратором и политиками рестарта.
