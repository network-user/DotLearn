# cherry-pick: как перенести один коммит, не таща за собой всю ветку

## Суть

`git cherry-pick` берёт конкретный коммит (или несколько) из одной ветки и накладывает его на текущую. Это точечный перенос правок без слияния всей исходной ветки.

## Что ответить на собесе

`cherry-pick` нужен, когда из чужой ветки требуется ровно одно изменение, а не вся её история:

- Классический случай: в экспериментальной ветке нашли и починили баг, а фикс срочно нужен в `main`.
- Вместо `merge` всей ветки берём только нужный коммит по его хешу - Git создаёт в текущей ветке новый коммит с теми же изменениями, но с новым хешем.
- Минус в том, что при неаккуратном применении в истории появляются дубли одного и того же изменения, а при пересечении правок придётся разрешать конфликт вручную.

При `git cherry-pick <hash>` Git берёт диф выбранного коммита и применяет его поверх текущей ветки, создавая новый коммит. Содержимое то же, но хеш другой - ведь меняется родитель и контекст.

Пусть в `feature` несколько коммитов, а в `main` нужен только один - с исправлением.

```bash
git checkout main
git cherry-pick <commit-hash>
```

Здесь `<commit-hash>` - идентификатор переносимого коммита. После выполнения в `main` появится отдельный коммит с этим исправлением, при этом остальная работа из `feature` останется нетронутой.

Перенести можно и диапазон или несколько хешей подряд:

```bash
git cherry-pick <hash-a> <hash-b>
```

## Подводные камни

- Если изменения коммита пересекаются с тем, что уже есть в целевой ветке, cherry-pick останавливается и просит разрешить конфликт вручную (`git cherry-pick --continue` после правки).
- Результат всегда получает новый хеш, даже при идентичном содержимом. Это значит, что в истории легко наплодить дубликаты одного фикса.
- Если позже исходную ветку всё же сольют целиком, перенесённое изменение может появиться дважды - готовьтесь к лишнему конфликту.
- Когда нужны все изменения ветки, cherry-pick не инструмент: для этого есть `merge` и `rebase`.
