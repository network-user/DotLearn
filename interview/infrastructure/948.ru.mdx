# Сети в Docker: что поднимается само, а что придётся настроить руками

## Суть

Docker сам создаёт сеть `bridge` по умолчанию и подключает к ней контейнеры, если явно не указано иное. Под изоляцию, контроль доступа и удобную связь контейнеров по именам поднимают собственные пользовательские сети.

## Что ответить на собесе

Базовая сеть появляется автоматически, но осмысленную топологию настраиваешь вручную:

- Из коробки сеть уже есть: при установке Docker создаёт несколько сетей, и контейнер без явного `--network` попадает в дефолтную `bridge`.
- Этого хватает для простых случаев — контейнеры на хосте видят друг друга и выходят в интернет.
- Но в реальных проектах я почти всегда создаю свою сеть через `docker network create`: она даёт изоляцию групп контейнеров и, главное, встроенный DNS — контейнеры обращаются друг к другу по имени сервиса, а не по IP.

## Как это работает

Сеть в Docker — это слой, через который контейнеры общаются между собой и с внешним миром, и одновременно граница изоляции.

Без указания сети контейнер уходит в дефолтный `bridge`:

```bash
docker run -d --name web nginx
```

Ключевой нюанс: в дефолтном `bridge` контейнеры видят друг друга только по IP, а в пользовательской сети — ещё и по имени, благодаря встроенному DNS. Поэтому свою сеть создают почти всегда:

```bash
docker network create app_net
docker run -d --name db  --network app_net postgres:16
docker run -d --name web --network app_net nginx
```

Теперь `web` достучится до базы просто по хосту `db` — без знания её IP.

Схема изоляции двух групп сервисов:

```
+----------------------------+      +----------------------------+
|        frontend_net        |      |        backend_net         |
|   [web]  <----->  [proxy]  |      |   [api]  <----->  [db]     |
+----------------------------+      +----------------------------+
```

Контейнеры из `frontend_net` не достучатся до `db`, пока их явно не подключат к `backend_net`.

## Подводные камни

- В дефолтной сети `bridge` нет автоматического DNS по именам контейнеров — связь только по IP, а он меняется при пересоздании. Резолв по имени работает именно в ПОЛЬЗОВАТЕЛЬСКИХ сетях.
- Публикация порта (`-p 8080:80`) и сеть — разные вещи. Подключение к сети даёт связь между контейнерами; чтобы сервис был доступен снаружи хоста, нужен проброс порта.
- Контейнеры из разных пользовательских сетей по умолчанию изолированы друг от друга — это фича, а не баг. Чтобы они общались, контейнер подключают сразу к нескольким сетям.
- В Docker Compose сеть для проекта создаётся автоматически, и все сервисы видят друг друга по именам без ручного `network create` — про это часто забывают и плодят лишние сети вручную.
