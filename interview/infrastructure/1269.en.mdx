# A baseline set of commands for a Linux system administrator

## In short

Linux administration rests on a few groups of commands: privilege escalation (`sudo`), user management (`useradd`, `usermod`), file permissions (`chmod`, `chown`), services (`systemctl`), disk (`df`, `du`), and processes (`ps`, `top`, `kill`).

## How to answer in an interview

I'd group the commands by task rather than memorize a flat list. Privileges - `sudo` for a one-off elevation of rights. Users - `useradd`/`usermod` (for example, `usermod -aG` adds to a group without wiping the existing ones). File permissions - `chmod` and `chown`. Services on modern systems - via `systemctl` (start/stop/status/enable). Disk - `df` for partitions and `du` for specific directories. Processes - `ps`/`top` to view and `kill` to terminate by PID. The main thing in an interview is to show that you understand *why* a command exists, not just its syntax.

## How it works

**Privileges - `sudo`:**

```bash
sudo apt update
```

Temporarily elevates rights for a single command; who can do what via sudo is described in `/etc/sudoers`.

**Users - `useradd`, `usermod`:**

```bash
sudo useradd newuser
sudo usermod -aG sudo newuser
```

The first creates a user, the second adds them to the `sudo` group. The `-aG` flag matters: `-a` (append) appends the group, otherwise `-G` will overwrite the entire group list.

**Permissions - `chmod`, `chown`:**

```bash
chmod 755 script.sh
chown user:group script.sh
```

`755` - rwx for the owner, r-x for the rest. `chown user:group` changes the owner and group at once.

**Services - `systemctl`:**

```bash
sudo systemctl start apache2
sudo systemctl status apache2
sudo systemctl enable apache2
```

`start` runs it now, `status` shows the state, `enable` turns on autostart at boot (these are different things).

**Disk - `df`, `du`:**

```bash
df -h
du -sh /var/log
```

`df -h` - filesystem usage, `du -sh` - the total size of a directory.

**Processes - `ps`, `top`, `kill`:**

```bash
ps aux
top
kill 1234
```

`ps aux` - a snapshot of all processes, `top` - a live view of the load, `kill 1234` sends a signal to the process with PID 1234.

## Pitfalls

- **`usermod -G` without `-a` wipes groups.** Forget `-a` and the user is dropped from all their previous groups. You almost always want `usermod -aG`.
- **`start` != `enable`.** `systemctl start` brings a service up until reboot; for it to start after a reboot you need `enable`. A common confusion.
- **`kill` sends SIGTERM by default, it doesn't "kill".** The process can intercept and ignore it. A hard termination is `kill -9` (SIGKILL), but that's a last resort: it doesn't let resources be closed cleanly.
- **`df` and `du` can disagree.** A deleted but still-open file continues to occupy disk space (visible in `df`, but not in `du`) as long as a process holds it.
- **`sudo` is not a "god mode forever".** The rights apply to a specific command; pipes and redirects run without root (`sudo echo x > /root/f` fails - the redirect is done by the shell).
- **`useradd` vs `adduser`.** The low-level `useradd` by default doesn't create a home directory and doesn't ask for a password; on Debian systems the friendlier wrapper is `adduser`.
