# Кэширование в стеке Grafana: где оно живёт и как влияет на свежесть метрик

## Суть

Сама Grafana метрики не хранит и не кэширует - она лишь рисует то, что приходит от источников вроде Prometheus или InfluxDB. Кэш живёт у источников, либо его добавляют прокси и плагины. Кэширование ускоряет отклик и снимает нагрузку, но значений метрик не меняет - только их свежесть.

## Что ответить на собесе

Grafana - это слой визуализации, а не хранилище. Дашборды строятся по данным, которые она запрашивает у источников (Prometheus, InfluxDB, Graphite). Поэтому базово Grafana ничего не кэширует - каждый рефреш панели идёт в источник. Кэширование появляется в трёх местах: у самих источников (Prometheus держит свежие данные в памяти), на кэширующем прокси перед источником (Nginx, Varnish), и через плагины/Enterprise-функции, складывающие результаты в Redis или Memcached. Важно: кэш влияет только на актуальность - при слишком долгом TTL пользователь увидит устаревшие точки, поэтому время жизни подбирают под требования к свежести.

Кэш снижает число запросов к источникам и ускоряет отклик - критично при больших объёмах и частых обращениях. Где он возникает:

1. **Источники данных.** У подключённых систем свой кэш. Тот же Prometheus держит свежие сэмплы в оперативной памяти.
2. **Прокси-серверы.** Перед источником ставят Nginx или Varnish - они кэшируют HTTP-ответы и разгружают backend.
3. **Плагины и расширения.** Часть плагинов (и query caching в Grafana Enterprise) кладёт результаты запросов в Redis или Memcached.

### Пример: кэширующий Nginx перед Prometheus

```nginx
http {
    proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=my_cache:10m max_size=1g inactive=60m use_temp_path=off;

    server {
        location / {
            proxy_pass http://prometheus-server;
            proxy_cache my_cache;
            proxy_cache_valid 200 1m;
            add_header X-Cache-Status $upstream_cache_status;
        }
    }
}
```

- `proxy_cache_path` - путь, размер и срок хранения кэша.
- `proxy_pass` - пересылка запросов в Prometheus.
- `proxy_cache` - включает кэш для этого location.
- `proxy_cache_valid 200 1m` - ответы с кодом 200 живут в кэше 1 минуту.
- `X-Cache-Status` - заголовок со статусом (HIT, MISS, EXPIRED) для диагностики.

## Подводные камни

- **Кэш не меняет значения, но меняет свежесть.** При длинном TTL пользователь видит устаревшие точки, хотя источник уже обновился. Для алертов и live-дашбордов это опасно.
- **TTL под тип данных.** Минутный кэш уместен для медленно меняющихся графиков, но недопустим там, где важна реакция в реальном времени.
- **Кэш на ключе запроса.** Прокси кэширует по URL/параметрам. Меняющийся диапазон времени (`now-5m`) даёт новые ключи и низкий hit rate - кэширование «горящих» запросов почти бесполезно.
- **Двойное кэширование путает.** Кэш на прокси поверх кэша источника усложняет отладку: непонятно, чьи данные устарели. Заголовок `X-Cache-Status` помогает понять, откуда пришёл ответ.
