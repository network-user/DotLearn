# Poetry as a dependency manager: how it beats the pip + setuptools combo

## In short

Poetry handles dependency management and project packaging in one go. Unlike the pip + setuptools combo, it resolves compatible versions itself, pins them in a lock file, manages the virtual environment, and publishes the package - all through a single `pyproject.toml`.

## How to answer in an interview

Poetry is a tool that unifies dependency management and Python package building into a single workflow. Its main advantages over pip/setuptools: it picks compatible versions itself (a real resolver, not sequential installation), keeps exact versions in `poetry.lock` for reproducibility, automatically creates and activates a virtual environment, and describes the project declaratively in `pyproject.toml` instead of a scattering of `requirements.txt` plus `setup.py`. It also takes building and publishing to PyPI on itself with ready-made commands. The result is less manual fiddling with versions and environments.

## How it works

Poetry solves two problems: dependency management and packaging. What it provides:

1. **Dependency resolution.** The resolver picks mutually compatible versions of the whole tree, so conflicts are caught before installation rather than at runtime.
2. **Reproducibility.** Direct constraints live in `pyproject.toml`, and the exact versions of the whole tree are in `poetry.lock`. The lock file is what gives an identical installation on every machine.
3. **Virtual environments.** Poetry sets them up and activates them itself, isolating the project from system libraries.
4. **Building and publishing.** Preparing a wheel/sdist and uploading to PyPI follow a single standardized flow.

A typical workflow cycle:

```bash
poetry init                 # create pyproject.toml with metadata
poetry add requests         # add a dependency and record it in pyproject.toml
poetry install              # install the whole tree per the lock file into the venv
poetry run python script.py # run the script inside the project environment
```

`poetry add` not only installs the package but also updates `pyproject.toml` and `poetry.lock`, while `poetry install` guarantees everyone gets the same version.

### The edge over pip/setuptools

- **Automatic version resolution** - Poetry finds a compatible set itself, whereas in pip you control this by hand.
- **One config** - `pyproject.toml` instead of separate `requirements.txt` and `setup.py`.
- **Environments out of the box** - no separate virtualenv needed.
- **Built-in publishing** - `poetry build` and `poetry publish` instead of manual packaging.

## Pitfalls

- **Commit the lock file.** Without `poetry.lock` in the repository you lose reproducibility - half the point of the tool is gone. For applications, the lock is mandatory in VCS.
- **`pyproject.toml` vs `poetry.lock`.** The former sets constraints (`^2.0`), the latter pins concrete versions. `poetry update` pulls in newer versions within the constraints and updates the lock; `poetry install` respects the existing lock.
- **venv behavior is configurable.** By default Poetry creates the environment outside the project; in Docker and CI people often enable `virtualenvs.in-project` or `virtualenvs.create false`, otherwise there are path surprises.
- **Not a pip replacement everywhere.** Internally Poetry still pulls packages, but mixing `pip install` and `poetry add` in the same environment is a path to a mismatch between the lock file and the actual state.
