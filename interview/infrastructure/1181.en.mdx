# How to shrink a Docker image: working techniques against bloat

## In short

Image size is reduced on three levels: a lightweight base, fewer layers and less junk in them, and separating build tools from the runtime via a multi-stage build. The more compact the image, the faster it downloads and deploys, and the fewer potential vulnerabilities it carries.

## How to answer in an interview

I start with the base image: instead of `ubuntu`/`debian` I take `alpine` or a slim variant. Next I watch the layers — combining install and cleanup commands into a single `RUN`, so that cache and temporary files do not settle into a separate layer. For compiled languages I use a multi-stage build: in the first stage I build the binary with the full toolchain, in the second I copy only the finished artifact. On top of that I install packages with `--no-install-recommends` and do not drag dev dependencies into the final image.

Each Dockerfile instruction adds a layer, and deleting a file in a later layer does not shrink the image — the data stays in the previous one. That is why install and cleanup are kept in a single `RUN`.

```dockerfile
RUN apt-get update && \
    apt-get install -y --no-install-recommends python3 && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

A lightweight base immediately removes dozens of unnecessary megabytes:

```dockerfile
# before
FROM ubuntu:latest
# after
FROM alpine:latest
```

A multi-stage build separates "where we build" from "what we ship". The toolchain stays in the first stage and does not end up in the final image.

```dockerfile
FROM golang:alpine AS builder
WORKDIR /app
COPY . .
RUN go build -o myapp

FROM alpine
WORKDIR /app
COPY --from=builder /app/myapp .
CMD ["./myapp"]
```

## Pitfalls

- `rm -rf` in a separate `RUN` after installation does not help: the deleted files are already "baked" into the lower layer, and the total weight does not drop. Cleanup must happen in the same `RUN`.
- `apt-get clean` without `rm -rf /var/lib/apt/lists/*` leaves package indexes behind — they take up space too.
- Instruction order affects the cache: copy the dependency file and install it first, then the whole codebase. Otherwise any code edit invalidates the install layer.
- `.dockerignore` is often forgotten — and `.git`, the host's `node_modules`, and local artifacts fly into the image.
- The `latest` tag is unpredictable: it is better to pin a specific base version.
