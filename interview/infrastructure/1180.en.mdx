# CMD versus ENTRYPOINT: who sets the command and who only the arguments?

## In short

Both instructions describe what runs when the container starts, but they react differently to arguments from `docker run`. CMD is easily overwritten in full, while ENTRYPOINT stays as the immutable core of the command.

## How to answer in an interview

CMD is the default behavior: anything I pass to `docker run` replaces it entirely. ENTRYPOINT sets a fixed command, and arguments from `docker run` are merely appended to it. That is why a container with an ENTRYPOINT behaves like a standalone executable. In practice they are combined: ENTRYPOINT is the program itself, CMD is its default arguments, which can be overridden without touching the program.

## How it works

CMD runs only if no custom command was passed at startup. Specify anything in `docker run` and CMD vanishes entirely.

```dockerfile
FROM ubuntu:latest
CMD ["echo", "Hello, World!"]
```

Running with no arguments prints `Hello, World!`. But `docker run <image> ls` replaces the whole line, and only `ls` runs.

ENTRYPOINT sets a base that cannot be thrown away just by passing arguments — arguments are glued to it on the right.

```dockerfile
FROM ubuntu:latest
ENTRYPOINT ["echo"]
```

Now `docker run <image> "Hello, World!"` runs exactly `echo "Hello, World!"`: `echo` is unchanged, and the string is its argument.

The combination brings out both instructions: ENTRYPOINT is the command, CMD is its default arguments.

```dockerfile
FROM ubuntu:latest
ENTRYPOINT ["echo"]
CMD ["Hello, World!"]
```

With no arguments you get `echo "Hello, World!"`. With an argument, `docker run <image> "Goodbye, World!"` replaces CMD and produces `echo "Goodbye, World!"`, while `echo` itself stays put.

## Pitfalls

- The difference is in the form of notation. The exec form `["echo"]` launches the process directly, while the shell form `echo $X` wraps it in `/bin/sh -c`. In the shell form of ENTRYPOINT, an external CMD and arguments from `docker run` are ignored — a common cause of "why don't my arguments reach the program".
- Without the exec form of ENTRYPOINT, the process does not get PID 1 directly, and signals (SIGTERM on stop) may not reach the application.
- If only CMD is set, any argument in `docker run` overwrites it entirely — it is easy to accidentally "lose" the startup command.
- The ENTRYPOINT itself can only be overridden with the explicit `--entrypoint` flag, not with positional arguments.
