# Сетевые драйверы Docker: bridge, host, none, overlay - что выбрать

## Суть

За связь контейнеров между собой и с внешним миром отвечает сетевой драйвер. Четыре базовых режима покрывают почти всё: `bridge` - изолированная сеть на одном хосте, `host` - прямой стек хоста, `none` - сети нет, `overlay` - единая сеть поверх нескольких хостов.

## Что ответить на собесе

Выбор драйвера - это компромисс между изоляцией, производительностью и масштабом:

- По умолчанию контейнер попадает в `bridge` - виртуальный мост на хосте, где контейнеры видят друг друга, а наружу выходят только через проброс портов.
- `host` отдаёт контейнеру сетевой стек хоста без отдельного namespace - быстрее, но без изоляции и без проброса портов.
- `none` полностью отключает сеть для задач, которым она не нужна.
- `overlay` связывает контейнеры с разных машин в одну логическую сеть - это режим для Swarm и кластеров.

**bridge.** Docker поднимает виртуальный мост `docker0` и подключает к нему новый контейнер, выдавая ему адрес из внутренней подсети. Снаружи контейнер недоступен, пока порт не проброшен.

```bash
docker run -d --name web --network bridge nginx
```

**host.** Контейнер делит сетевой namespace с хостом: его порты - это порты хоста. Нет накладных расходов на NAT, но и нет изоляции.

```bash
docker run -d --name web --network host nginx
```

**none.** Сетевой стек поднимается без интерфейсов - только loopback. Контейнер заперт от сети.

```bash
docker run -d --name job --network none nginx
```

**overlay.** Поверх физической сети строится виртуальная L2-сеть, и контейнеры с разных хостов общаются так, будто они в одном сегменте. Базовый режим для Swarm.

```bash
docker network create -d overlay my_overlay
```

## Подводные камни

- На дефолтном `bridge` (без явной пользовательской сети) встроенного DNS по именам контейнеров нет - резолв по имени работает только в созданной вручную сети `docker network create`.
- `host` доступен полноценно на Linux; на Docker Desktop под macOS/Windows контейнеры крутятся в VM, и поведение `host` отличается.
- В режиме `host` флаг `-p`/`--publish` игнорируется, и два контейнера не займут один и тот же порт хоста.
- `overlay` без Swarm требует ручной настройки key-value store; «просто так» на одиночном хосте он не нужен.
- Все контейнеры в одной bridge-сети видят друг друга по умолчанию - для изоляции сервисов их разводят по разным сетям, а не полагаются на firewall внутри.
