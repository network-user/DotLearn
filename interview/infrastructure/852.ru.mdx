# Зачем нужен git push --force и чем он опасен?

## Суть

Force push заставляет удалённую ветку принять вашу локальную историю, даже если она расходится с серверной. Это перезапись истории на сервере, а значит инструмент мощный и легко стирающий чужую работу.

## Что ответить на собесе

Force push (`git push --force`) отправляет локальную ветку на сервер, игнорируя проверку «удалённая история - предок локальной». Ключевые тезисы:

- Нужен после переписывания истории: `rebase`, `commit --amend`, squash или удаления случайно закоммиченного секрета.
- Обычный push в таких случаях отклоняется, потому что ветки разошлись.
- Главная опасность - можно затереть коммиты, которые кто-то другой уже запушил.
- Поэтому на практике почти всегда используют `git push --force-with-lease`: он откажется перезаписывать ветку, если на сервере появилось что-то, чего вы не видели.

## Как это работает

Git отклоняет обычный push, когда удалённая ветка не является прямым предком вашей локальной (non-fast-forward). Флаг `--force` снимает эту защиту и двигает удалённый указатель ветки туда, куда вы скажете.

```bash
# Переписали последние коммиты локально (rebase / amend / squash)
git rebase -i HEAD~3

# Обычный push отклонён: histories diverged
git push                 # ! [rejected] (non-fast-forward)

# Принудительно перезаписываем удалённую ветку
git push --force
```

Безопасная версия проверяет, что удалённая ветка указывает ровно туда, где вы её в последний раз видели (значение из `git fetch`). Если кто-то успел запушить раньше вас, операция не пройдёт:

```bash
git push --force-with-lease
```

## Подводные камни

- `--force` затирает чужие коммиты молча: если коллега запушил, а вы форснули поверх, его работа исчезает из ветки.
- `--force-with-lease` защищает только при свежем `fetch`. Если вы сделали `fetch`, а потом форснули lease - проверка пройдёт, и чужой коммит всё равно может потеряться.
- Никогда не форсьте общие ветки (`main`, релизные), на которых работает команда. Force допустим на личных feature-ветках до ревью.
- Потерянные коммиты ещё какое-то время живут в `git reflog` и доступны через хеш, но рассчитывать на это как на бэкап нельзя.
