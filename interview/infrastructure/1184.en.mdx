# What data model is Git's versioning built on?

## In short

Git is a distributed version control system where history is a chain of snapshots (commits), not a set of diffs. Each commit points to the previous one, so you can return to any version, branch off, and merge changes.

## How to answer in an interview

Git is built on a snapshot model: each commit stores the full tree of the project's state at the moment of recording, plus a reference to the parent commit. Unchanged files are not duplicated - Git reuses the previous object by content, so storage is economical. History is a directed graph of commits; a branch is just a movable pointer to a commit. A merge combines two lines of development, and a conflict arises when the same lines were edited differently and Git cannot pick automatically. Being distributed means every developer has a full copy of the history, not just the current slice.

## How it works

Key entities:

1. **Repository** - the project together with all of its history; it can be local or remote.
2. **Commit** - a snapshot of the file tree plus a reference to the parent and a message.
3. **Branch** - a movable pointer to a commit, a separate line of development.
4. **Merge** - bringing changes from one branch into another.
5. **Conflict** - incompatible edits to the same places, requiring manual resolution.

A typical workflow cycle:

```bash
git init                          # new local repository
git add filename                  # move changes into the index (staging)
git commit -m "Описание правок"   # snapshot -> new commit
git checkout -b new-feature       # branch + switch to it
git checkout main
git merge new-feature             # merge feature into main
```

## Pitfalls

- Git stores snapshots, not diffs. "Git stores only the difference" is a common mistake; it computes the diff on the fly and deduplicates objects by content (hash).
- A branch is not a copy of the files but a pointer (a single hash in `.git/refs`). That's why creating a branch is instant and cheap.
- The staging area (index) is a separate layer between the working tree and the commit; `git add` records exactly the state that will go into the snapshot, not "the whole file forever."
- A conflict is not a Git error but a signal that an automatic merge is impossible: the decision always rests with a human.
