# Зачем нужна kill и почему это не только про убийство процессов

## Суть

Несмотря на название, `kill` не «убивает», а доставляет процессу сигнал по его PID. Завершение — лишь один из сценариев: тем же механизмом процессы перезапускают, перечитывают конфиг и ставят на паузу.

## Что ответить на собесе

`kill` отправляет сигнал процессу по идентификатору (PID). Сигнал — это число, на которое процесс реагирует заранее заданным действием.

- `SIGTERM` (15) — вежливая просьба закрыться, которую процесс может перехватить и аккуратно прибраться.
- `SIGKILL` (9) — принудительное снятие, которое нельзя ни перехватить, ни проигнорировать, потому что его обрабатывает само ядро.
- Правильный порядок — сначала `SIGTERM`, и только если процесс завис, `SIGKILL`.

## Как это работает

Сигнал адресуется процессу по PID. Часть сигналов можно перехватить обработчиком в коде, а `SIGKILL` и `SIGSTOP` — нет, они всегда выполняются ядром.

Жёсткое снятие, когда процесс не отвечает:

```bash
kill -9 1234
```

- `-9` — это `SIGKILL`, мгновенное завершение без шанса прибраться.

Штатное завершение, дающее процессу закрыть файлы и соединения:

```bash
kill -15 1234   # или просто: kill 1234
```

- `-15` — `SIGTERM`, сигнал по умолчанию, если номер не указан.

Перечитать конфигурацию без перезапуска (типично для демонов):

```bash
kill -1 1234
```

- `-1` — `SIGHUP`. Многие сервисы (nginx, sshd) по нему подхватывают новый конфиг.

Типичный рабочий цикл, когда сервис завис:

```bash
pgrep myserver        # узнаём PID по имени
kill 1234             # сначала вежливо (SIGTERM)
kill -9 1234          # если не реагирует — принудительно
```

## Подводные камни

- Сразу слать `kill -9` — плохая привычка: процесс не успевает закрыть файлы, сбросить буферы и снять блокировки, можно поймать повреждённые данные.
- `SIGKILL` (9) и `SIGSTOP` нельзя перехватить или заблокировать — поэтому ими снимают зависшие процессы, но и обработчик завершения у них не отработает.
- `kill -9 -1` или `kill 0` бьют сразу по группе процессов, а не по одному — легко уронить лишнее.
- Снять чужой процесс без прав не выйдет: нужен тот же пользователь или root.
- `kill` шлёт сигнал по PID, а `killall`/`pkill` — по имени; путать их опасно, имя может совпасть у нескольких процессов.
