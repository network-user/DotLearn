# The minimal set of Docker commands you can't do without in practice

## In short

The everyday Docker workflow comes down to a few commands: build an image (`build`), run a container from it (`run`), see what's running (`ps`), step inside (`exec`), and exchange images with a registry (`pull`/`push`).

## How to answer in an interview

I'd break the commands down by lifecycle stages. Building and delivering images: `docker build` builds an image from a Dockerfile, `docker pull` pulls a ready one from a registry, `docker push` sends yours back. Running and managing containers: `docker run` creates and starts a container, `docker ps` shows running ones (with `-a` for all of them), `docker stop`/`docker start` stop and bring them up. Debugging: `docker exec -it <name> bash` opens a shell inside a live container. Cleanup: `docker rm` removes a container, `docker rmi` removes an image.

## How it works

It's important to keep in mind the difference between an image (an immutable template) and a container (a running instance of an image). The commands split into two groups accordingly.

Images:

```bash
docker build -t my-image:latest .   # build an image from the Dockerfile in the current folder
docker pull ubuntu:24.04            # download an image from a registry
docker push my-repo/my-image        # send your image to a registry
docker rmi my-image                 # remove an image
```

Containers:

```bash
docker run -d --name web nginx      # create and start a container in the background
docker ps                           # running containers
docker ps -a                        # all, including stopped
docker stop web && docker start web # stop and start again
docker exec -it web /bin/bash       # interactive shell inside the container
docker rm web                       # remove a container
```

Flags that come up constantly: `-t` (image tag at build time), `-d` (detached, background mode), `-it` (interactive terminal), `--name` (a human-readable name instead of a random hash).

## Pitfalls

- `docker rm` removes a container, `docker rmi` removes an image. They're easy to mix up; a running container won't let you remove its image until it's stopped and removed itself.
- `docker run` each time creates a NEW container from the image, rather than starting an old one. To bring up an existing one, you need `docker start`. The habit of slapping on `run` breeds zombie containers.
- `docker exec` only works with a live container. If the process inside crashed and the container stopped, you won't get in via `exec`; check `docker logs`.
- `-it` without an interactive command (e.g. `bash`) is useless, and `bash` is absent in minimal images like `alpine`; there you use `sh`.
