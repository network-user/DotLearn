# How Docker differs from Docker Compose and where the line between them lies

## In short

Docker runs and isolates a single container, while Docker Compose orchestrates a bundle of several containers from a YAML description. The first is responsible for the unit, the second for their shared life.

## How to answer in an interview

Docker is a containerization engine: it takes an application with all its dependencies and packages it into a container that behaves identically on any machine running Docker. Docker Compose is a tool on top of Docker for multi-container applications: instead of manually starting each container, I describe all services, networks, and volumes in `docker-compose.yml` and bring them up with a single `docker compose up` command. Roughly: Docker is about one container, Compose is about managing a group of them as a single application.

Docker relies on Linux kernel primitives: namespaces (isolation of process visibility, network, filesystem) and cgroups (CPU and memory limits). Thanks to this, dozens of containers share a single OS kernel, and heavy virtualization isn't needed.

A minimal Dockerfile for a Python service:

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

CMD ["python", "app.py"]
```

Compose doesn't replace Docker, it conducts it. It reads a declarative description and in a single pass creates networks, volumes, and brings up containers in the right order:

```yaml
services:
  web:
    image: my-web-app:latest
    ports:
      - '5000:5000'
    depends_on:
      - db
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: example
      POSTGRES_PASSWORD: example
```

Here `web` depends on `db`, so Compose starts the database before the web service, places both on a shared network, and lets them communicate by service names.

## Pitfalls

- `depends_on` only guarantees the container's start order, not the readiness of the service inside. The database may still be initializing when `web` has already started. A healthcheck or application-side retries are needed.
- The `version:` key is deprecated and ignored in modern Compose; don't copy `version: '3'` from old examples.
- Compose is great for local development and small environments, but it's not a production-grade orchestrator: horizontal scaling, self-healing, and rolling updates are jobs for Kubernetes or Swarm, not Compose.
- Containers on the same host share the kernel. This isn't a full-fledged VM: a Linux image won't run on a kernel that doesn't support it without a compatibility layer.
