# Логи в Docker: stdout, драйверы логирования и централизованный сбор

## Суть

Канонический подход - писать логи приложения в `stdout`/`stderr`, а как их хранить и куда отправлять, решает logging driver (`json-file`, `syslog`, `fluentd` и другие). Для масштаба логи стягивают в централизованные системы вроде ELK или Fluentd.

## Что ответить на собесе

Приложение в контейнере не должно само писать в файлы - оно пишет в стандартные потоки, а Docker их перехватывает. Дальше всё определяет драйвер логирования: по умолчанию это `json-file`, который кладёт логи на хост в JSON, и ему обязательно задают ротацию через `max-size`/`max-file`, иначе диск переполнится. Смотрю логи через `docker logs`, а в продакшене переключаю драйвер на `fluentd`/`syslog` или собираю всё в ELK для централизованного поиска и анализа.

## Как это работает

Самый надёжный путь - приложение печатает в `stdout` и `stderr`, ничего дополнительно настраивать не нужно: Docker сам подхватывает эти потоки и передаёт их драйверу логирования.

Драйвер решает, куда уходят записи. Популярные варианты:

- **json-file** - JSON-файлы на хосте (по умолчанию);
- **syslog** - на системный лог-сервер;
- **journald** - в журнал `systemd`;
- **fluentd** - в Fluentd для дальнейшей обработки;
- **awslogs** - в Amazon CloudWatch.

Драйвер и его параметры задают в compose-файле. Здесь же настраивают ротацию:

```yaml
version: '3.8'
services:
  my_app:
    image: my_app_image
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"
```

`max-size` ограничивает один файл 10 мегабайтами, `max-file` держит максимум 3 файла - при превышении старые удаляются. Просмотр логов:

```bash
docker logs <container_id_or_name>
```

## Подводные камни

- Без ротации `json-file` пишет бесконечно и однажды забивает весь диск хоста - `max-size`/`max-file` обязательны.
- `docker logs` работает только с драйверами `json-file`, `journald` и `local`. С `syslog`/`fluentd`/`awslogs` команда вернёт ошибку - логи живут уже во внешней системе.
- Не стоит писать логи в файл внутри контейнера: данные исчезнут при пересоздании и не попадут в `docker logs`.
- Запись в `stdout` синхронна - очень «болтливое» приложение под нагрузкой может тормозить из-за блокировки на выводе.
- Глобальный драйвер задаётся в `daemon.json`, и он применяется ко всем новым контейнерам; на старые уже запущенные он не действует.
