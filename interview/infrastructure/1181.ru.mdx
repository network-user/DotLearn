# Как ужать Docker-образ: рабочие приёмы против раздувания

## Суть

Размер образа сокращают на трёх уровнях: лёгкая база, меньше слоёв и мусора в них, отделение сборочных инструментов от рантайма через multi-stage build. Чем компактнее образ, тем быстрее он качается и разворачивается и тем меньше в нём потенциальных уязвимостей.

## Что ответить на собесе

Сокращаю образ на нескольких уровнях:

- Базовый образ: вместо `ubuntu`/`debian` беру `alpine` или slim-вариант.
- Слои: объединяю команды установки и чистки в один `RUN`, чтобы кеш и временные файлы не оседали отдельным слоем.
- Для компилируемых языков использую multi-stage build: в первом этапе собираю бинарник со всем тулчейном, во второй копирую только готовый артефакт.
- Плюс ставлю пакеты с `--no-install-recommends` и не тащу dev-зависимости в финальный образ.

Каждая инструкция Dockerfile добавляет слой, и удаление файла в следующем слое не уменьшает образ - данные остаются в предыдущем. Поэтому установку и очистку держат в одном `RUN`.

```dockerfile
RUN apt-get update && \
    apt-get install -y --no-install-recommends python3 && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

Лёгкая база сразу убирает десятки лишних мегабайт:

```dockerfile
# было
FROM ubuntu:latest
# стало
FROM alpine:latest
```

Multi-stage build разделяет «где собираем» и «что отдаём». Тулчейн остаётся в первом этапе и в финальный образ не попадает.

```dockerfile
FROM golang:alpine AS builder
WORKDIR /app
COPY . .
RUN go build -o myapp

FROM alpine
WORKDIR /app
COPY --from=builder /app/myapp .
CMD ["./myapp"]
```

## Подводные камни

- `rm -rf` в отдельном `RUN` после установки не помогает: удалённые файлы уже «зашиты» в нижний слой, и общий вес не падает. Чистить надо в том же `RUN`.
- `apt-get clean` без `rm -rf /var/lib/apt/lists/*` оставляет индексы пакетов - они тоже занимают место.
- Порядок инструкций влияет на кеш: сначала копируют файл зависимостей и ставят их, потом весь код. Иначе любая правка кода инвалидирует слой установки.
- `.dockerignore` часто забывают - и в образ улетают `.git`, `node_modules` с хоста, локальные артефакты.
- Тег `latest` непредсказуем: лучше фиксировать конкретную версию базы.
