# На какой модели данных построено версионирование в Git?

## Суть

Git - распределённая система контроля версий, где история - это цепочка снимков (коммитов), а не набор diff'ов. Каждый коммит ссылается на предыдущий, поэтому к любой версии можно вернуться, ветвиться и сливать изменения.

## Что ответить на собесе

В основе Git лежит модель снимков:

- Каждый коммит хранит полное дерево состояния проекта на момент фиксации и ссылку на родительский коммит.
- Неизменившиеся файлы не дублируются - Git переиспользует прежний объект по содержимому, поэтому хранение экономно.
- История - это направленный граф коммитов; ветка - всего лишь подвижный указатель на коммит.
- Слияние объединяет две линии разработки, а конфликт возникает, когда одни и те же строки правились по-разному и Git не может выбрать автоматически.
- Распределённость означает, что у каждого разработчика есть полная копия истории, а не только текущий срез.

## Как это работает

Ключевые сущности:

1. **Репозиторий** - проект вместе со всей историей; бывает локальным и удалённым.
2. **Коммит** - снимок дерева файлов плюс ссылка на родителя и сообщение.
3. **Ветка** - подвижный указатель на коммит, отдельная линия разработки.
4. **Слияние (merge)** - перенос изменений одной ветки в другую.
5. **Конфликт** - несовместимые правки одних и тех же мест, требующие ручного разрешения.

Типичный цикл работы:

```bash
git init                          # новый локальный репозиторий
git add filename                  # перенос изменений в индекс (staging)
git commit -m "Описание правок"   # снимок -> новый коммит
git checkout -b new-feature       # ветка + переключение на неё
git checkout main
git merge new-feature             # вливаем feature в main
```

## Подводные камни

- Git хранит снимки, а не diff'ы. «Git хранит только разницу» - частая ошибка; diff он вычисляет на лету, а объекты дедуплицирует по содержимому (хешу).
- Ветка - это не копия файлов, а указатель (один хеш в `.git/refs`). Поэтому создание ветки мгновенно и дёшево.
- Staging area (индекс) - отдельный слой между рабочим деревом и коммитом; `git add` фиксирует именно то состояние, что попадёт в снимок, а не «файл целиком навсегда».
- Конфликт - не ошибка Git, а сигнал, что автоматическое слияние невозможно: решение всегда за человеком.
