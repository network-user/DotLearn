# Container vs. virtual machine: where does the isolation boundary run?

## In short

The whole difference is the level at which isolation is built. A container shares the host kernel and is isolated by the kernel's own mechanisms, which makes it lightweight. A VM carries its own OS on top of a hypervisor, so the isolation is stronger, but the cost in resources is higher.

## How to answer in an interview

A virtual machine virtualizes the hardware: a hypervisor runs several guest OSes, each with its own kernel. A container virtualizes at the OS level: all containers use the host kernel, and the kernel itself separates them via namespaces and cgroups. Everything else follows from this: a container starts in milliseconds and weighs megabytes, a VM boots in seconds and drags along gigabytes of guest system. On the other hand, a VM gives a stronger security boundary and lets you run an OS with a different kernel (a Windows guest on a Linux host). A container is bound to the host kernel: on a Linux host, only Linux containers run natively.

## How it works

A VM lives on top of a hypervisor. The hypervisor is installed either directly on the hardware (bare-metal, type 1) or on top of a host OS (type 2). It slices CPU, memory, and disk between guests and keeps them isolated. Each guest boots a full kernel.

A container doesn't emulate hardware. The Docker Engine asks the host kernel to create an isolated environment for a process via two mechanisms:

- **namespaces** - isolate what a process sees: PID, network, mount points, hostnames;
- **cgroups** - limit what a process consumes: CPU, memory, I/O.

A container image carries user space (libraries, binaries) but not the kernel - the kernel comes from the host.

An analogy: VMs are separate houses, each with its own foundation and utilities. Containers are rooms in a single building: their own walls and doors, but a shared foundation and engineering systems, which makes them cheaper and more nimble.

## Pitfalls

- **"A container is a lightweight VM" is wrong.** A container has no kernel of its own and doesn't virtualize hardware. It's an isolated host process.
- **The security boundary is weaker.** A vulnerability in the host kernel is potentially accessible to all containers at once. Where strict isolation of untrusted code is needed, a VM (or a micro-VM like gVisor/Kata) is more appropriate.
- **Kernel binding.** Linux containers on Windows/macOS actually run inside a hidden Linux VM - there's no "nativeness" there.
- **It's not either/or.** In practice containers are often run inside VMs: the VM provides tenant isolation, the containers provide density and speed.
