# Zombie versus daemon: two completely different "undead" in Linux

## In short

A zombie is an already-terminated process whose entry lingers in the process table until the parent collects its exit code via `wait()`. A daemon is the opposite: a fully alive background process, detached from the terminal and serving system tasks. The only thing they have in common is "invisibility"; otherwise they are opposites.

## How to answer in an interview

A zombie (defunct) appears when a child process has finished and terminated, but the parent has not yet called `wait()`/`waitpid()` to read its return code. The code itself is no longer executing and the memory is freed; all that remains is a row in the process table with the exit code. A zombie cannot be "killed" with a signal, because there is nothing left to kill; it disappears when the parent calls `wait()`, and if the parent dies first, the zombie is adopted by `init`/`systemd`, which cleans up after it. A daemon is a specially crafted background process: it is detached from the controlling terminal via `setsid()`, and it runs independently of user sessions, serving the network, hardware, or periodic tasks.

**Zombie.** A process becomes a zombie in the gap between its own termination and the parent's `wait()`:

```c
#include <sys/wait.h>
#include <unistd.h>
#include <stdio.h>

int main() {
    pid_t pid = fork();
    if (pid > 0) {
        sleep(10);          // parent sleeps and does NOT call wait(): the child lingers as a zombie
    } else if (pid == 0) {
        printf("Child done\n");
        _exit(0);           // the child has terminated, its entry awaits reaping
    }
    return 0;
}
```

The entries are needed so the parent can learn how the child finished (exit code, whether it was a signal).

**Daemon.** A daemon is created with a double `fork()` and `setsid()` between them, to detach from the terminal forever:

```c
#include <stdlib.h>
#include <unistd.h>
#include <sys/stat.h>

void create_daemon() {
    if (fork() > 0) exit(EXIT_SUCCESS);   // exit the parent
    setsid();                             // new session, detach from the terminal
    if (fork() > 0) exit(EXIT_SUCCESS);   // second fork: do not be a session leader
    umask(0);
    chdir("/");                           // do not keep a partition busy
    close(STDIN_FILENO);
    close(STDOUT_FILENO);
    close(STDERR_FILENO);
}
```

The second `fork()` guarantees the process does not become a session leader and cannot accidentally grab a terminal.

## Pitfalls

- A zombie is not freed by `kill -9`: it no longer exists, so a `wait()` from the parent or the death of the parent itself is required.
- A crowd of zombies is a symptom of a bug in the parent (it does not reap its children), not a standalone problem; their danger is that they clog up the PID table.
- An orphan is not the same as a zombie: an orphan still has running code, its parent has simply died, and it is adopted by `init`.
- Without the second `fork()`, a daemon may remain a session leader and, on opening a terminal, will grab it as its controlling terminal.
- If a daemon does not `chdir("/")`, it keeps busy the filesystem it started from, which then cannot be unmounted.
