# Networking in Docker: what comes up on its own and what you have to set up by hand

## In short

Docker creates a default `bridge` network on its own and attaches containers to it unless told otherwise. For isolation, access control, and convenient communication between containers by name, you spin up your own user-defined networks.

## How to answer in an interview

Out of the box a network already exists: on installation Docker creates several networks, and a container without an explicit `--network` lands in the default `bridge`. That's enough for simple cases - containers on the host can see each other and reach the internet. But in real projects I almost always create my own network via `docker network create`: it provides isolation of container groups and, crucially, built-in DNS - containers address each other by service name rather than by IP. So the answer is: the basic network appears automatically, but a meaningful topology you set up by hand.

## How it works

A network in Docker is the layer through which containers communicate with each other and the outside world, and at the same time it's an isolation boundary.

Without specifying a network, a container goes into the default `bridge`:

```bash
docker run -d --name web nginx
```

The key nuance: in the default `bridge` containers see each other only by IP, while in a user-defined network they also see each other by name, thanks to built-in DNS. That's why people almost always create their own network:

```bash
docker network create app_net
docker run -d --name db  --network app_net postgres:16
docker run -d --name web --network app_net nginx
```

Now `web` reaches the database simply by the host `db` - without knowing its IP.

The isolation scheme for two service groups:

```
+----------------------------+      +----------------------------+
|        frontend_net        |      |        backend_net         |
|   [web]  <----->  [proxy]  |      |   [api]  <----->  [db]     |
+----------------------------+      +----------------------------+
```

Containers in `frontend_net` won't reach `db` until they're explicitly attached to `backend_net`.

## Pitfalls

- The default `bridge` network has no automatic DNS by container name - communication is only by IP, and it changes when a container is recreated. Name resolution works specifically in USER-DEFINED networks.
- Publishing a port (`-p 8080:80`) and the network are different things. Attaching to a network provides communication between containers; for a service to be reachable from outside the host, you need a port mapping.
- Containers in different user-defined networks are isolated from each other by default - that's a feature, not a bug. For them to communicate, you attach a container to several networks at once.
- In Docker Compose a network for the project is created automatically, and all services see each other by name without a manual `network create` - people often forget this and breed extra networks by hand.
