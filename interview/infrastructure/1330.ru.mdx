# Как вы организуете ветвление в git и зачем нужна эта дисциплина?

## Суть

Воркфлоу — это договорённость команды о том, какие ветки существуют, откуда они растут и куда вливаются. Цель одна: держать релизную ветку всегда стабильной, а работу над фичами, релизами и срочными фиксами вести параллельно и не мешая друг другу.

## Что ответить на собесе

Выбор воркфлоу я подгоняю под команду и частоту релизов:

- Классический Git Flow Винсента Дриссена держит долгоживущие ветки `main` и `develop` плюс короткоживущие feature-, release- и hotfix-ветки; он хорош там, где есть версионированные релизы и несколько параллельных потоков работы.
- Если же мы катим в прод по нескольку раз в день, я предпочитаю trunk-based или GitHub Flow: одна основная ветка, короткие feature-ветки и быстрый merge через pull request с обязательным code review и зелёным CI.
- Главный принцип общий: основная ветка всегда деплоится, ветки живут недолго, а вся история остаётся читаемой.
- Сам по себе Git Flow не самоцель — это инструмент под определённый ритм релизов.

В Git Flow две долгоживущие ветки. `main` хранит стабильный, готовый к релизу код, и каждый релиз помечается тегом. `develop` — интеграционная ветка, куда стекаются завершённые фичи перед выпуском.

Feature-ветку заводят от `develop` и вливают обратно в `develop`:

```bash
git switch develop
git switch -c feature/awesome-feature
# работа над функцией, серия коммитов
git switch develop
git merge --no-ff feature/awesome-feature
git branch -d feature/awesome-feature
```

Release-ветку готовят перед выпуском: в ней допиливают мелочи, бампают версию, после чего вливают и в `main` (с тегом), и обратно в `develop`:

```bash
git switch -c release/1.0.0 develop
# финальные правки, обновление версии
git switch main
git merge --no-ff release/1.0.0
git tag -a 1.0.0 -m "Релиз 1.0.0"
git switch develop
git merge --no-ff release/1.0.0
git branch -d release/1.0.0
```

Hotfix-ветку создают прямо от `main`, чтобы быстро закрыть критический баг в проде, и потом вливают её и в `main`, и в `develop`, иначе фикс потеряется в следующем релизе:

```bash
git switch -c hotfix/urgent-fix main
# исправление
git switch main
git merge --no-ff hotfix/urgent-fix
git tag -a 1.0.1 -m "Хотфикс 1.0.1"
git switch develop
git merge --no-ff hotfix/urgent-fix
git branch -d hotfix/urgent-fix
```

Что это даёт команде: чёткое разделение работы над новым и над исправлениями, постоянно стабильный `main`, удобное управление релизами и тегами и прозрачная история версий.

## Подводные камни

- **Долгоживущие ветки.** Чем дольше feature-ветка живёт в отрыве от `develop`, тем больнее merge — растёт риск конфликтов. Дробите задачи и сливайте часто.
- **Забытый back-merge хотфикса.** Если фикс влит только в `main`, но не в `develop`, баг вернётся в следующем релизе.
- **Git Flow для CI/CD-конвейера.** При нескольких деплоях в день модель с `develop` и release-ветками тормозит; trunk-based подходит лучше.
- **`merge` без `--no-ff` для значимых веток.** Fast-forward стирает факт, что был отдельный поток работы, и история теряет структуру.
- **Прямые коммиты в `main`.** Защитите ветку правилами: только через pull request с прохождением CI и ревью.
