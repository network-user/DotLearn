# Checking whether a TCP port is reachable on a remote server

## In short

To find out whether a remote host accepts connections on a given port, you use `nc` (netcat), `telnet`, or `nmap`. All three try to establish a TCP connection: if it comes up, the port is open and something is listening behind it.

## How to answer in an interview

For a one-off check I usually reach for `nc -zv host port` - it doesn't push any data, it just verifies the connection and prints the result, plus it can do UDP. `telnet host port` is a fallback when netcat isn't installed: if the connection establishes, the port is reachable. I use `nmap -p port host` when I need to scan a range of ports or also figure out which service sits behind the port. It's important to distinguish three outcomes: "open" (there's a response), "closed" (RST - the host is there but nobody's listening), and "filtered" (silence - a firewall is dropping traffic).

## How it works

**netcat (`nc`)** - the most universal tool:

```bash
nc -zv 192.168.1.1 80
```

- `-z` - scan mode: check the port without sending data.
- `-v` - verbose output with the result.

**telnet** - the minimal way to bring up a TCP connection:

```bash
telnet 192.168.1.1 80
```

If the connection establishes, the port is reachable; a connection error means it isn't. To exit telnet: `Ctrl+]`, then `quit`.

**nmap** - a scanner that also determines the state and the service:

```bash
nmap -p 80 192.168.1.1
```

- `-p 80` - which port to check (a range is allowed, `-p 1-1000`).

In its output nmap marks the port as `open`, `closed`, or `filtered`, which is more precise than telnet's binary "yes/no".

## Pitfalls

- **These tools check TCP by default.** You can't "knock" on a UDP port this way: silence doesn't mean "closed". For UDP you need `nc -u` or `nmap -sU`, and the result is still ambiguous.
- **`telnet` hangs under filtering.** If a firewall drops packets, telnet will wait out a long timeout instead of giving a clear refusal. `nc -w` with a timeout is more convenient.
- **"Port closed" != "port filtered".** A closed port answers with RST (host alive, no service), a filtered one stays silent (a firewall is dropping it). Only nmap cleanly separates them.
- **The client's local firewall** can also block an outbound connection - the problem isn't always on the server side.
- **Scanning other people's hosts with nmap** may be treated as reconnaissance/an attack. Scan only what you have permission to.
- **An open port != a working service.** The TCP handshake completed, but the application behind the port may respond with errors - that's a separate check at a higher level.
