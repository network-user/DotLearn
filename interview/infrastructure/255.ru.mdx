# Минимальный набор Docker-команд, без которого не обойтись в работе

## Суть

Повседневный цикл работы с Docker сводится к нескольким командам: собрать образ (`build`), запустить из него контейнер (`run`), посмотреть что крутится (`ps`), зайти внутрь (`exec`) и обмениваться образами с реестром (`pull`/`push`).

## Что ответить на собесе

Я бы разбил команды по этапам жизненного цикла. Сборка и доставка образов: `docker build` собирает образ из Dockerfile, `docker pull` тянет готовый из реестра, `docker push` отправляет свой обратно. Запуск и управление контейнерами: `docker run` создаёт и стартует контейнер, `docker ps` показывает запущенные (с `-a` — вообще все), `docker stop`/`docker start` гасят и поднимают. Отладка: `docker exec -it <name> bash` открывает шелл внутри живого контейнера. Уборка: `docker rm` удаляет контейнер, `docker rmi` — образ.

## Как это работает

Важно держать в голове разницу между образом (неизменяемый шаблон) и контейнером (запущенный экземпляр образа). Команды делятся на две группы соответственно.

Образы:

```bash
docker build -t my-image:latest .   # собрать образ из Dockerfile в текущей папке
docker pull ubuntu:24.04            # скачать образ из реестра
docker push my-repo/my-image        # отправить свой образ в реестр
docker rmi my-image                 # удалить образ
```

Контейнеры:

```bash
docker run -d --name web nginx      # создать и запустить контейнер в фоне
docker ps                           # запущенные контейнеры
docker ps -a                        # все, включая остановленные
docker stop web && docker start web # остановить и снова запустить
docker exec -it web /bin/bash       # интерактивный шелл внутри контейнера
docker rm web                       # удалить контейнер
```

Флаги, которые встречаются постоянно: `-t` (тег образа при сборке), `-d` (detached, фоновый режим), `-it` (интерактивный терминал), `--name` (человекочитаемое имя вместо случайного хеша).

## Подводные камни

- `docker rm` удаляет контейнер, `docker rmi` — образ. Их легко перепутать; запущенный контейнер не даст удалить свой образ, пока не остановлен и не удалён сам.
- `docker run` каждый раз создаёт НОВЫЙ контейнер из образа, а не запускает старый. Чтобы поднять уже существующий, нужен `docker start`. Привычка лепить `run` плодит зомби-контейнеры.
- `docker exec` работает только с живым контейнером. Если процесс внутри упал и контейнер остановлен, зайти через `exec` не выйдет — смотрите `docker logs`.
- `-it` без интерактивной команды (например `bash`) бесполезен, а `bash` отсутствует в минимальных образах вроде `alpine` — там используют `sh`.
