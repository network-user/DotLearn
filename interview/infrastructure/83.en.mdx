# cherry-pick: how to move a single commit without dragging the whole branch along

## In short

`git cherry-pick` takes a specific commit (or several) from one branch and applies it onto the current one. It's a pinpoint transfer of changes without merging the entire source branch.

## How to answer in an interview

`cherry-pick` is needed when you require exactly one change from someone else's branch, not its whole history. The classic case: a bug was found and fixed in an experimental branch, and the fix is urgently needed in `main`. Instead of merging the whole branch, we take only the needed commit by its hash - Git creates a new commit in the current branch with the same changes but a new hash. The downside is that careless use leaves duplicates of the same change in history, and when edits overlap you'll have to resolve a conflict manually.

With `git cherry-pick <hash>`, Git takes the diff of the chosen commit and applies it on top of the current branch, creating a new commit. The content is the same, but the hash is different - because the parent and context change.

Suppose `feature` has several commits, but `main` needs only one - with the fix.

```bash
git checkout main
git cherry-pick <commit-hash>
```

Here `<commit-hash>` is the identifier of the commit being moved. After it runs, `main` gets a separate commit with that fix, while the rest of the work from `feature` stays untouched.

You can also move a range or several hashes in a row:

```bash
git cherry-pick <hash-a> <hash-b>
```

## Pitfalls

- If the commit's changes overlap with what's already in the target branch, cherry-pick stops and asks you to resolve the conflict manually (`git cherry-pick --continue` after editing).
- The result always gets a new hash, even with identical content. This means it's easy to breed duplicates of one fix in history.
- If the source branch is later merged in full after all, the moved change may appear twice - be ready for an extra conflict.
- When you need all of a branch's changes, cherry-pick isn't the tool: `merge` and `rebase` exist for that.
