# Git Flow: дисциплина ветвления с долгоживущими и временными ветками

## Суть

Git Flow - это договорённость о том, как команда работает с ветками. Она делит ветки на постоянные (стабильный код) и временные (под фичи, релизы и срочные правки), задавая каждой роль и правила слияния.

## Что ответить на собесе

Git Flow - модель ветвления, предложенная Винсентом Дриссеном:

- В её основе две долгоживущие ветки: `main` со стабильным релизным кодом и `develop` как интеграционная, куда стекаются готовые фичи.
- Временные ветки: feature растут из `develop` и возвращаются в него; release отпочковываются от `develop` для финальной доводки и вливаются в `main` и `develop`.
- hotfix-ветки идут от `main` для срочных фиксов и тоже возвращаются в обе.
- Модель даёт предсказуемый процесс и хорошо ложится на крупные команды и релизы по расписанию, но для частых деплоев тяжеловата - там чаще выбирают trunk-based подход.

Постоянные ветки:

- `main` (или `master`) - стабильный код, готовый к выпуску; все релизы выходят отсюда.
- `develop` - интеграционная ветка, где собираются все новые наработки перед попаданием в `main`.

Временные ветки и их жизненный цикл:

```bash
git checkout develop
git checkout -b feature/awesome-feature
```

```bash
git checkout develop
git checkout -b release/1.0.0
```

```bash
git checkout main
git checkout -b hotfix/urgent-fix
```

Полный цикл выглядит так: разработчик ответвляет feature от `develop`, пишет код, вливает обратно в `develop`. Когда набор фич готов, от `develop` создают release-ветку, прогоняют финальные тесты и правки, затем сливают её в `main` и `develop` и вешают тег версии. Если в проде всплывает критичный баг, от `main` заводят hotfix, и исправление возвращается и в `main`, и в `develop`.

## Подводные камни

- Две постоянные ветки и теги требуют дисциплины: забытый merge hotfix обратно в `develop` тихо вернёт уже исправленный баг.
- Для CI/CD с частыми релизами Git Flow избыточен - длинные ветки расходятся и копят конфликты.
- Release-ветка должна сливаться в обе постоянные ветки, иначе `main` и `develop` разъедутся.
- Модель - соглашение, а не команда Git: её соблюдают люди, и нарушение правил ветвления Git никак не остановит.
