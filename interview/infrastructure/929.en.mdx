# Static typing in CI: how to make the pipeline check types automatically?

## In short

Type checking is wired in as a separate pipeline step that runs a static analyzer (for Python, mypy) on every push and pull request. If the types don't line up, the step fails and the offending code never reaches merge.

## How to answer in an interview

I add type annotations to the code and, in CI, a dedicated job that installs mypy and runs it over the project. The real value isn't the check itself but the fact that it's mandatory and automatic: a developer can't "forget" to run the analyzer locally, because a red pipeline blocks the merge. This catches a whole class of errors (wrong argument type, a None where an object was expected) before the code runs and before code review, saving the reviewer's time.

Annotations are a contract that mypy checks for consistency without executing the program:

```python
def add(a: int, b: int) -> int:
    return a + b
```

Next we add a job to the pipeline. For GitHub Actions we drop in the file `.github/workflows/mypy.yml`:

```yaml
name: Type Check

on: [push, pull_request]

jobs:
  type-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install mypy
        run: |
          python -m pip install --upgrade pip
          pip install mypy

      - name: Run mypy
        run: mypy src/
```

Line by line:

- `on: [push, pull_request]` - triggers: every commit and every PR.
- `runs-on: ubuntu-latest` - a clean runner for reproducibility.
- the `checkout` step pulls the code, `setup-python` pins the interpreter version.
- `mypy src/` returns a non-zero exit code on error, and the job turns red.

For GitLab it's the same idea - adding a stage to `.gitlab-ci.yml` with the same `mypy src/`.

## Pitfalls

- **Green mypy doesn't mean "no bugs".** Static checking catches only type mismatches, not logic; it complements tests, it doesn't replace them.
- **Third-party libraries without annotations.** Many packages aren't typed; mypy will complain until you install stub packages (`types-...`) or configure `ignore_missing_imports`.
- **Python version.** The check's behavior depends on the target version; if it differs between CI and local, the results will diverge - pin it explicitly.
- **Rolling it out on legacy code.** Turning on strict mode at once on a large untyped project is painful; usually you start with lenient settings and tighten gradually, otherwise the pipeline drowns in errors.
