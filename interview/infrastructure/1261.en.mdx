# Caching in the Grafana stack: where it lives and how it affects metric freshness

## In short

Grafana itself neither stores nor caches metrics - it only draws what comes from sources like Prometheus or InfluxDB. The cache lives at the sources, or it's added by proxies and plugins. Caching speeds up responses and reduces load, but it doesn't change metric values - only their freshness.

## How to answer in an interview

Grafana is a visualization layer, not storage. Dashboards are built from data it queries from sources (Prometheus, InfluxDB, Graphite). So by default Grafana caches nothing - every panel refresh goes to the source. Caching appears in three places: at the sources themselves (Prometheus keeps recent data in memory), at a caching proxy in front of the source (Nginx, Varnish), and through plugins/Enterprise features that store results in Redis or Memcached. Important: the cache affects only freshness - with too long a TTL the user sees stale points, so the lifetime is tuned to the freshness requirements.

A cache reduces the number of requests to the sources and speeds up responses - critical at large volumes and frequent access. Where it arises:

1. **Data sources.** Connected systems have their own cache. Prometheus, for instance, keeps recent samples in RAM.
2. **Proxy servers.** Nginx or Varnish are placed in front of the source - they cache HTTP responses and offload the backend.
3. **Plugins and extensions.** Some plugins (and query caching in Grafana Enterprise) store query results in Redis or Memcached.

### Example: a caching Nginx in front of Prometheus

```nginx
http {
    proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=my_cache:10m max_size=1g inactive=60m use_temp_path=off;

    server {
        location / {
            proxy_pass http://prometheus-server;
            proxy_cache my_cache;
            proxy_cache_valid 200 1m;
            add_header X-Cache-Status $upstream_cache_status;
        }
    }
}
```

- `proxy_cache_path` - the path, size, and retention period of the cache.
- `proxy_pass` - forwarding requests to Prometheus.
- `proxy_cache` - enables the cache for this location.
- `proxy_cache_valid 200 1m` - responses with code 200 live in the cache for 1 minute.
- `X-Cache-Status` - a header with the status (HIT, MISS, EXPIRED) for diagnostics.

## Pitfalls

- **The cache doesn't change values, but it does change freshness.** With a long TTL the user sees stale points even though the source has already updated. For alerts and live dashboards this is dangerous.
- **TTL by data type.** A one-minute cache is fine for slowly changing graphs but unacceptable where real-time reaction matters.
- **The cache keys on the request.** A proxy caches by URL/parameters. A shifting time range (`now-5m`) produces new keys and a low hit rate - caching "hot" requests is nearly useless.
- **Double caching confuses things.** A cache on the proxy on top of the source's cache complicates debugging: it's unclear whose data is stale. The `X-Cache-Status` header helps figure out where the response came from.
