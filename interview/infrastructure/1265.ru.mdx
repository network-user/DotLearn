# PID процесса: что за число и какую роль оно играет в ОС?

## Суть

PID (Process Identifier) - уникальное целое число, которое ядро присваивает процессу при создании. Через PID операционная система и пользователь адресуют процесс: следят за ним, читают его параметры и шлют ему сигналы.

## Что ответить на собесе

PID - это идентификатор процесса, выдаваемый ядром в момент его запуска:

- Он уникален среди живых процессов и закреплён за процессом до его завершения.
- Когда процесс умирает, его PID освобождается и со временем может достаться новому - поэтому PID нельзя считать вечной ссылкой.
- Практически PID нужен, чтобы адресовать процесс: посмотреть статус через `ps` или `/proc/<pid>`, послать сигнал через `kill`, ограничить ресурсы.
- В Linux первый процесс (init/systemd) всегда имеет PID 1 и является предком всех остальных.

При создании процесса (через `fork`) ядро выделяет ему свободный PID и заводит структуру с его состоянием. Этот номер живёт до завершения процесса; после выхода запись остаётся в виде «зомби», пока родитель не заберёт код возврата через `wait`, и только тогда PID окончательно освобождается.

Список процессов и их PID в Linux показывает `ps`:

```bash
ps aux
```

- `ps` - вывести информацию о процессах.
- `aux` - показать все процессы всех пользователей с подробностями.

Пример вывода:

```
USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.1  22528  4100 ?        Ss   10:00   0:01 /sbin/init
user      1234  0.1  0.3 123456  7890 pts/0    S+   10:05   0:02 python3 script.py
```

Колонка `PID` - тот самый идентификатор. Чтобы воздействовать на процесс, по PID посылают сигнал:

```bash
kill 1234        # послать SIGTERM процессу 1234 (мягкое завершение)
kill -9 1234     # SIGKILL: принудительно, без шанса на обработку
```

## Подводные камни

- PID переиспользуется. Если процесс завершился, а вы шлёте сигнал по старому PID, можно попасть в чужой процесс, занявший этот номер. Перед `kill` стоит сверить, что это тот процесс (по имени/времени запуска).
- `kill` - это «послать сигнал», а не обязательно «убить». По умолчанию это SIGTERM, который процесс может перехватить и проигнорировать; жёсткое завершение - `kill -9` (SIGKILL), но оно не даёт процессу корректно закрыть ресурсы.
- PID 1 особенный: в обычной системе это init/systemd, а в контейнере им становится главный процесс. Если PID 1 не пересылает сигналы и не собирает зомби, дочерние процессы накапливаются - отсюда минимальные init вроде `tini`.
- Зомби-процесс (`STAT Z`) уже мёртв и не занимает ресурсов, кроме записи в таблице; убить его `kill` нельзя - нужно, чтобы родитель сделал `wait`, либо завершить самого родителя.
- PID процесса и идентификаторы его потоков - разные вещи: у потоков свои TID, видимые в `/proc/<pid>/task`.
