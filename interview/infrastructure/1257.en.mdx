# What container operations tasks does Kubernetes take off your plate

## In short

Kubernetes is a container orchestrator. You declaratively describe the desired state of your application, and the platform brings the cluster to it on its own: it runs the required number of instances, keeps them alive, distributes traffic, and rolls out updates without stopping the service. The core idea is that you don't manage individual containers by hand but a target state that the system continuously maintains.

## How to answer in an interview

Kubernetes covers the routine of operating distributed containerized applications. List point by point what exactly it takes on:

- **Declarative deployment.** You describe the result (a manifest), not a sequence of commands. The platform brings the cluster to that state and holds it there.
- **Scaling.** The number of instances adjusts to load - manually or automatically based on metrics.
- **Self-healing.** A crashed container is restarted, an unhealthy pod is replaced. The failure of a single instance doesn't take the application down.
- **Load balancing.** Incoming traffic is distributed evenly across live instances via a Service.
- **Configuration and secrets.** ConfigMap and Secret separate settings and sensitive data from the image.
- **Zero-downtime updates.** A rolling update replaces pods gradually, checking the readiness of new ones before shutting down the old ones.

One sentence to sum up: Kubernetes turns the manual operation of containers into managing a desired state through a continuous reconciliation loop.

## How it works

At its core is a reconciliation loop: controllers continuously compare the desired state from the manifests with the actual state of the cluster and eliminate the discrepancy. You declared three replicas but only two are alive - the controller brings up the third.

The typical object for a stateless application is a Deployment. It manages a set of pods through a ReplicaSet and describes which image to run and in how many copies.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app-container
          image: my-app-image:latest
          ports:
            - containerPort: 80
```

What the key fields describe:

- **apiVersion / kind** - the API version and object type; here a Deployment, the deployment controller.
- **metadata.name** - the object's name in the cluster.
- **spec.replicas** - the target number of pods running at the same time.
- **spec.selector** - by which labels the Deployment finds its pods.
- **spec.template** - the template each pod is created from: its labels, containers, image, and exposed ports.

This manifest keeps the application at three instances: on failure a pod is restarted, and the number of replicas changes when needed. The link goes through labels: `selector.matchLabels` must match `template.metadata.labels`, otherwise the Deployment won't "see" its pods.

## Pitfalls

- **Kubernetes is not a substitute for architecture.** It won't make an application scalable on its own: if a service stores state locally, replication and pod restarts will break it. A stateless design is a precondition, not a consequence.
- **Labels must match.** A mismatch between the `selector` and the template labels is a common mistake: the Deployment doesn't manage pods that don't fit the selector.
- **`image: latest` is dangerous.** A floating tag breaks reproducibility and rolling updates: different pods may bring up different builds. In production, pin the image version.
- **Self-healing != absence of failures.** Without correct liveness/readiness probes, Kubernetes either keeps an unhealthy pod in traffic or endlessly restarts a working one. Probes need to be configured deliberately.
- **Over-engineering.** For a single small service, a full-blown cluster is excessive complexity. Kubernetes is justified where there are many components, variable load, and fault tolerance matters.
