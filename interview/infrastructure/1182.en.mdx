# Docker network drivers: bridge, host, none, overlay - which to choose

## In short

The network driver is responsible for how containers communicate with each other and with the outside world. Four basic modes cover almost everything: `bridge` - an isolated network on a single host, `host` - the host's stack directly, `none` - no network at all, `overlay` - a single network spanning multiple hosts.

## How to answer in an interview

By default a container lands on the `bridge` - a virtual bridge on the host where containers see each other but reach the outside world only through port publishing. `host` gives the container the host's network stack with no separate namespace - faster, but with no isolation and no port publishing. `none` disables networking entirely for tasks that don't need it. `overlay` ties containers from different machines into one logical network - this is the mode for Swarm and clusters. The choice is a trade-off between isolation, performance, and scale.

**bridge.** Docker brings up a virtual bridge `docker0` and attaches the new container to it, assigning it an address from an internal subnet. The container is unreachable from outside until a port is published.

```bash
docker run -d --name web --network bridge nginx
```

**host.** The container shares the host's network namespace: its ports are the host's ports. There's no NAT overhead, but no isolation either.

```bash
docker run -d --name web --network host nginx
```

**none.** The network stack comes up with no interfaces - only loopback. The container is locked away from the network.

```bash
docker run -d --name job --network none nginx
```

**overlay.** A virtual L2 network is built on top of the physical one, and containers on different hosts talk as if they were in the same segment. The default mode for Swarm.

```bash
docker network create -d overlay my_overlay
```

## Pitfalls

- On the default `bridge` (with no explicit user-defined network) there is no built-in DNS by container name - name resolution works only in a network created manually with `docker network create`.
- `host` is fully available on Linux; on Docker Desktop under macOS/Windows containers run inside a VM, and `host` behaves differently.
- In `host` mode the `-p`/`--publish` flag is ignored, and two containers cannot claim the same host port.
- `overlay` without Swarm requires a manually configured key-value store; you simply don't need it on a single host.
- All containers in the same bridge network see each other by default - to isolate services you split them across separate networks rather than relying on a firewall inside.
