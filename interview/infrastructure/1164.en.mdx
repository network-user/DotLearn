# What systemd and sysfs are responsible for in modern Linux

## In short

systemd is the init and service manager: it brings up processes at boot and manages their lifecycle. sysfs is a virtual filesystem in `/sys` through which the kernel exposes information about devices and parameters as ordinary files.

## How to answer in an interview

systemd and sysfs solve different problems and do not compete. systemd is responsible for starting and managing services: parallel startup with dependency awareness, restarting crashed processes, a single `systemctl` interface. It replaced System V init. sysfs is not about services but about access to the kernel's internals: through files in `/sys` you can read or change parameters of hardware and drivers. Roughly speaking, systemd manages what gets started, and sysfs shows what it runs on.

## How it works

**systemd.** Everything is described by "units": a service, a mount point, a socket, a timer. systemd reads the dependencies between units and starts whatever it can in parallel, speeding up boot. Management goes through `systemctl`:

```bash
sudo systemctl start nginx      # bring up the service
sudo systemctl stop nginx       # stop it
sudo systemctl restart nginx    # restart it
sudo systemctl status nginx     # current state
sudo systemctl enable nginx     # autostart at boot
```

**sysfs.** It is mounted at `/sys` and lays out kernel data across a directory tree, where each node is a specific aspect of the system. Read a file and you get a value; write to a file and you change a parameter:

```bash
# read the current CPU frequency governor
cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_governor

# switch to maximum performance
echo "performance" | sudo tee /sys/devices/system/cpu/cpu0/cpufreq/scaling_governor
```

## Pitfalls

- sysfs does not store data on disk — it is an in-memory kernel interface, and changes via `echo` do not survive a reboot. For persistent settings you need configs (sysctl, udev rules, systemd units).
- Writing to `/sys` requires root: a plain `echo > file` fails with Permission denied, which is why `sudo tee` is used.
- Do not confuse sysfs (`/sys`, about devices) and procfs (`/proc`, about processes) — these are two different virtual filesystems.
- `systemctl start` brings a service up once, but without `enable` it does not start after a reboot.
