# Как pre-commit хуки фильтруют код ещё до коммита?

## Суть

Pre-commit hook - это скрипт, который Git запускает локально перед созданием коммита. Если проверка падает, коммит не создаётся, и плохой код просто не попадает в историю.

## Что ответить на собесе

Pre-commit - это клиентский хук Git, срабатывающий в момент `git commit` до записи коммита. Ключевые тезисы:

- Обычно через него гоняют линтеры, форматтеры, быстрые тесты и сканеры секретов.
- Ненулевой код возврата любого хука отменяет коммит.
- На практике хуки редко пишут руками: ставят фреймворк `pre-commit` с декларативным `.pre-commit-config.yaml`, где перечислены готовые проверки.
- Важный нюанс: это локальная защита, её можно обойти флагом `--no-verify`, поэтому те же проверки дублируют в CI как обязательный барьер.

Хуки лежат в `.git/hooks/`. Когда вы делаете `git commit`, Git вызывает исполняемый файл `pre-commit`. Возврат 0 - коммит создаётся, любое другое значение - отменяется.

```bash
pip install pre-commit
```

Конфигурация описывается декларативно. Пример для `flake8` (проверка PEP 8):

```yaml
repos:
  - repo: https://github.com/pycqa/flake8
    rev: 7.1.1
    hooks:
      - id: flake8
```

После `pre-commit install` фреймворк подменяет `.git/hooks/pre-commit` своим раннером:

```bash
pre-commit install
```

Теперь каждый коммит автоматически прогоняет `flake8` по изменённым файлам; при нарушениях вы получаете список проблем, а коммит блокируется до их исправления.

## Подводные камни

- Хук обходится через `git commit --no-verify` (`-n`), так что он не гарантия, а удобство. Жёсткий барьер делается в CI.
- Pre-commit живёт в `.git/hooks/` и не клонируется вместе с репозиторием. Чтобы хуки были у всех, нужен фреймворк + `pre-commit install` (или общая команда настройки).
- Тяжёлые проверки и полный прогон тестов в pre-commit раздражают: коммит начинает тормозить. Долгие проверки выносят в pre-push или CI.
- По умолчанию многие хуки работают только по staged-файлам, а не по всему проекту - это и плюс по скорости, и источник пропусков.
