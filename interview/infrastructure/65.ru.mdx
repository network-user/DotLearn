# Чем Docker отличается от Docker Compose и где проходит граница между ними

## Суть

Docker запускает и изолирует один контейнер, а Docker Compose оркеструет связку из нескольких контейнеров по описанию в YAML. Первый отвечает за единицу, второй — за их совместную жизнь.

## Что ответить на собесе

Граница проходит между одним контейнером и их группой:

- Docker — это движок контейнеризации: он берёт приложение со всеми зависимостями и упаковывает его в контейнер, который ведёт себя одинаково на любой машине с Docker.
- Docker Compose — инструмент поверх Docker для многоконтейнерных приложений: вместо ручного запуска каждого контейнера я описываю все сервисы, сети и тома в `docker-compose.yml` и поднимаю их одной командой `docker compose up`.
- Грубо: Docker — про один контейнер, Compose — про управление их группой как единым приложением.

Docker опирается на примитивы ядра Linux — namespaces (изоляция видимости процессов, сети, файловой системы) и cgroups (ограничение CPU и памяти). За счёт этого десятки контейнеров делят одно ядро ОС, и тяжёлая виртуализация не нужна.

Минимальный Dockerfile для Python-сервиса:

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

CMD ["python", "app.py"]
```

Compose не заменяет Docker, а дирижирует им. Он читает декларативное описание и за один проход создаёт сети, тома и поднимает контейнеры в нужном порядке:

```yaml
services:
  web:
    image: my-web-app:latest
    ports:
      - '5000:5000'
    depends_on:
      - db
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: example
      POSTGRES_PASSWORD: example
```

Здесь `web` зависит от `db`, поэтому Compose стартует базу раньше веб-сервиса, кладёт оба в общую сеть и даёт им общаться по именам сервисов.

## Подводные камни

- `depends_on` гарантирует только порядок старта контейнера, а не готовность сервиса внутри. База может ещё инициализироваться, когда `web` уже стартовал, — нужен healthcheck или ретраи на стороне приложения.
- Ключ `version:` в современном Compose устарел и игнорируется — не стоит копировать `version: '3'` из старых примеров.
- Compose хорош для локальной разработки и небольших стендов, но это не оркестратор уровня прода: горизонтальное масштабирование, самовосстановление и rolling-обновления — задача Kubernetes или Swarm, а не Compose.
- Контейнеры на одном хосте делят ядро. Это не полноценная виртуалка: образ под Linux не запустится на ядре, которое его не поддерживает без слоя совместимости.
