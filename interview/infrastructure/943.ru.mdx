# Сигналы в Linux: зачем они нужны и какие важно знать наизусть?

## Суть

Сигнал - это асинхронное уведомление процессу о том, что произошло событие; так ядро и процессы общаются между собой. Ключевая пятёрка: SIGINT (Ctrl+C), SIGTERM (вежливый запрос на завершение), SIGKILL (жёсткое убийство), SIGSEGV (обращение к чужой памяти) и SIGCHLD (дочерний процесс сменил состояние).

## Что ответить на собесе

Сигналы - это механизм межпроцессного взаимодействия для коротких уведомлений. Главное, что нужно сказать: большинство сигналов можно перехватить и обработать своим хендлером (например, по SIGTERM аккуратно закрыть соединения и сохранить данные), но два сигнала - SIGKILL (9) и SIGSTOP - перехватить, проигнорировать или заблокировать нельзя, их обрабатывает само ядро. Поэтому правильный порядок остановки сервиса - сначала SIGTERM и дать время на graceful shutdown, и только если не помогло - SIGKILL.

## Как это работает

Отправить сигнал процессу можно системным вызовом `kill` (или одноимённой утилитой в шелле). Простой пример - процесс шлёт сигнал самому себе:

```c
#include <signal.h>
#include <unistd.h>

int main(void) {
    kill(getpid(), SIGTERM);
    return 0;
}
```

`getpid()` возвращает PID текущего процесса, `kill` доставляет ему SIGTERM. Чтобы не просто получить сигнал, а среагировать на него, вешают обработчик:

```c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>

void handle_sigint(int sig) {
    printf("Caught signal %d\n", sig);
}

int main(void) {
    signal(SIGINT, handle_sigint);
    while (1) {
        printf("Running...\n");
        sleep(1);
    }
    return 0;
}
```

Теперь Ctrl+C не убьёт программу, а вызовет `handle_sigint`. Основные сигналы и их номера: SIGINT (2), SIGKILL (9), SIGSEGV (11), SIGTERM (15), SIGCHLD (17).

## Подводные камни

- **SIGKILL и SIGSTOP неперехватываемы.** Любая попытка повесить на них обработчик игнорируется; рассчитывать на «чистку перед смертью» по SIGKILL нельзя.
- **SIGKILL не освобождает ресурсы вашим кодом.** Процесс умирает мгновенно, деструкторы и flush буферов не выполняются - возможна потеря данных, поэтому сначала всегда SIGTERM.
- **`signal()` исторически непортируемая.** В реальном коде предпочитают `sigaction`: у `signal` поведение перезапуска обработчика различается между системами.
- **Игнорирование SIGCHLD рождает зомби.** Если родитель не считывает статус завершившихся детей (`wait`), они остаются как зомби-процессы в таблице.
- **Номера сигналов не универсальны.** Числовые значения вроде 9 и 15 совпадают на распространённых платформах, но привязываться к ним вместо имён - плохая идея на экзотических архитектурах.
