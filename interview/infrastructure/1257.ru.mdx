# Какие задачи эксплуатации контейнеров снимает Kubernetes

## Суть

Kubernetes - это оркестратор контейнеров. Вы декларативно описываете желаемое состояние приложения, а платформа сама приводит к нему кластер: запускает нужное число экземпляров, держит их живыми, распределяет трафик и катит обновления без остановки сервиса. Главная идея - вы управляете не отдельными контейнерами вручную, а целевым состоянием, которое система постоянно поддерживает.

## Что ответить на собесе

Kubernetes закрывает рутину эксплуатации распределённых контейнеризированных приложений. Перечислите по пунктам, что именно он берёт на себя:

- **Развёртывание по декларации.** Вы описываете результат (манифест), а не последовательность команд. Платформа сама приводит кластер к этому состоянию и удерживает его.
- **Масштабирование.** Число экземпляров подстраивается под нагрузку - вручную или автоматически по метрикам.
- **Самовосстановление.** Упавший контейнер перезапускается, нездоровый под заменяется. Падение одного экземпляра не роняет приложение.
- **Балансировка нагрузки.** Входящий трафик равномерно распределяется между живыми экземплярами через Service.
- **Конфигурация и секреты.** ConfigMap и Secret отделяют настройки и чувствительные данные от образа.
- **Обновления без простоя.** Rolling update заменяет поды постепенно, проверяя готовность новых, прежде чем гасить старые.

Одна фраза для итога: Kubernetes превращает ручную эксплуатацию контейнеров в управление желаемым состоянием через постоянный цикл сверки.

## Как это работает

В основе лежит петля согласования (reconciliation loop): контроллеры непрерывно сравнивают желаемое состояние из манифестов с фактическим состоянием кластера и устраняют расхождение. Объявили три реплики, а живых две - контроллер поднимет третью.

Типичный объект для приложения без состояния - Deployment. Он управляет набором подов через ReplicaSet и описывает, какой образ запускать и в скольких копиях.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app-container
          image: my-app-image:latest
          ports:
            - containerPort: 80
```

Что описывают ключевые поля:

- **apiVersion / kind** - версия API и тип объекта; здесь Deployment, контроллер развёртывания.
- **metadata.name** - имя объекта в кластере.
- **spec.replicas** - целевое число одновременно работающих подов.
- **spec.selector** - по каким меткам Deployment находит свои поды.
- **spec.template** - шаблон, по которому создаётся каждый под: его метки, контейнеры, образ и открытые порты.

Этот манифест держит приложение в трёх экземплярах: при сбое под перезапускается, при необходимости число реплик меняется. Связь идёт через метки: `selector.matchLabels` должен совпадать с `template.metadata.labels`, иначе Deployment не "увидит" свои поды.

## Подводные камни

- **Kubernetes - не замена архитектуре.** Он не сделает приложение масштабируемым сам по себе: если сервис хранит состояние локально, репликация и перезапуск подов его сломают. Stateless-дизайн - предусловие, а не следствие.
- **Метки должны совпадать.** Рассинхрон между `selector` и метками шаблона - частая ошибка: Deployment не управляет подами, которые не подходят под селектор.
- **`image: latest` опасен.** Плавающий тег ломает воспроизводимость и rolling update: разные поды могут поднять разные сборки. В проде фиксируйте версию образа.
- **Самовосстановление != отсутствие сбоев.** Без корректных liveness/readiness-проб Kubernetes либо гоняет нездоровый под в трафике, либо бесконечно перезапускает рабочий. Пробы нужно настраивать осознанно.
- **Оверинжиниринг.** Для одного маленького сервиса полноценный кластер - избыточная сложность. Kubernetes оправдан там, где много компонентов, нагрузка переменная и важна отказоустойчивость.
