# Проверка доступности TCP-порта на удалённом сервере

## Суть

Чтобы выяснить, принимает ли удалённый хост соединения на нужном порту, используют `nc` (netcat), `telnet` или `nmap`. Все три пытаются установить TCP-соединение: если оно поднялось - порт открыт и за ним кто-то слушает.

## Что ответить на собесе

Инструмент выбираю под задачу, а в ответе важно различать три исхода:

- `nc -zv host port` - для разовой проверки: не тащит данные, просто проверяет соединение и печатает результат, плюс умеет UDP.
- `telnet host port` - запасной вариант, если netcat не установлен: установилось соединение - порт доступен.
- `nmap -p port host` использую, когда надо просканировать диапазон портов или заодно понять, какой сервис за портом.
- Три исхода: «открыт» (есть ответ), «закрыт» (RST - хост есть, но никто не слушает) и «отфильтрован» (молчание - режет firewall).

## Как это работает

**netcat (`nc`)** - самый универсальный инструмент:

```bash
nc -zv 192.168.1.1 80
```

- `-z` - режим сканирования: проверить порт, не передавая данные.
- `-v` - подробный вывод с результатом.

**telnet** - минимальный способ поднять TCP-соединение:

```bash
telnet 192.168.1.1 80
```

Соединение установилось - порт доступен; ошибка подключения - нет. Выйти из telnet: `Ctrl+]`, затем `quit`.

**nmap** - сканер, который к тому же определяет состояние и сервис:

```bash
nmap -p 80 192.168.1.1
```

- `-p 80` - какой порт проверять (можно диапазон `-p 1-1000`).

nmap в выводе помечает порт как `open`, `closed` или `filtered`, что точнее, чем бинарное «есть/нет» у telnet.

## Подводные камни

- **Эти утилиты проверяют TCP по умолчанию.** UDP-порт так не «постучишь»: молчание не значит «закрыт». Для UDP нужен `nc -u` или `nmap -sU`, и результат всё равно неоднозначен.
- **`telnet` зависает при фильтрации.** Если firewall дропает пакеты, telnet будет долго ждать таймаута вместо понятного отказа. `nc -w` с таймаутом удобнее.
- **«Порт закрыт» != «порт отфильтрован».** Закрытый порт отвечает RST (хост жив, сервиса нет), отфильтрованный молчит (режет firewall). Только nmap чётко их разделяет.
- **Локальный firewall клиента** тоже может блокировать исходящее соединение - проблема не всегда на стороне сервера.
- **nmap-сканирование чужих хостов** может расцениваться как разведка/атака. Сканируйте только то, на что есть право.
- **Открытый порт != рабочий сервис.** TCP-рукопожатие прошло, но приложение за портом может отвечать ошибками - это отдельная проверка уровнем выше.
