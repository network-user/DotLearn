# Что превращает Dockerfile в готовый образ при сборке

## Суть

Образ собирается из `Dockerfile` командой `docker build`: Docker выполняет инструкции файла по порядку, превращая каждую в отдельный слой, и складывает слои в итоговый образ.

## Что ответить на собесе

Сборка идёт в два шага:

- Сначала я описываю рецепт в `Dockerfile`: от какого базового образа отталкиваться, что скопировать, какие зависимости поставить, чем запускать приложение.
- Потом запускаю `docker build -t myimage:latest .`, где `-t` задаёт имя и тег, а `.` — контекст сборки (директория, файлы которой доступны командам вроде `COPY`).
- Docker читает инструкции сверху вниз, на каждую создаёт слой, кэширует неизменившиеся слои и собирает их в финальный образ, из которого потом поднимаются контейнеры.

## Как это работает

Ключевая идея — слоистость и кэш. Каждая инструкция (`FROM`, `COPY`, `RUN`, ...) даёт один read-only слой. При повторной сборке Docker сверяет слои: если инструкция и её входные данные не менялись, берётся кэш, иначе слой пересобирается — и все слои ниже него тоже.

Отсюда правило: то, что меняется реже, ставим выше. Сначала зависимости, потом исходный код:

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

CMD ["python", "app.py"]
```

Здесь `requirements.txt` копируется и зависимости ставятся ДО копирования всего кода. Пока список зависимостей не меняется, тяжёлый слой `pip install` берётся из кэша, даже если правок в коде сколько угодно.

Схематично порядок слоёв:

```
[FROM python] -> [WORKDIR] -> [COPY requirements] -> [RUN pip install] -> [COPY .] -> [CMD]
   базовый        мета          редко меняется         кэшируется         часто меняется
```

Запуск сборки:

```bash
docker build -t myimage:latest .
```

## Подводные камни

- Порядок инструкций решает скорость. Если поставить `COPY . .` ПЕРЕД установкой зависимостей, любая правка кода инвалидирует кэш `pip install`, и зависимости переустанавливаются каждый раз.
- `.` в команде — это контекст сборки, а не «текущая папка для удобства». Docker упаковывает и отправляет демону всё содержимое контекста; без `.dockerignore` туда улетят `.git`, `node_modules`, venv — образ распухает и сборка тормозит.
- Каждый `RUN` — отдельный слой. Несколько `RUN apt-get ...` подряд раздувают образ; их объединяют в одну инструкцию через `&&`, а кэш apt чистят в том же слое.
- `latest` — не «самая свежая навсегда», а просто тег по умолчанию. Он не обновляется сам и в проде вводит в заблуждение — лучше пинить конкретные версии.
