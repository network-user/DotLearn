# Why do you need git push --force and why is it dangerous?

## In short

Force push makes the remote branch accept your local history even if it has diverged from the server's. It rewrites history on the server, which makes it a powerful tool that easily wipes out others' work.

## How to answer in an interview

Force push (`git push --force`) sends a local branch to the server, ignoring the "remote history is an ancestor of the local one" check. You need it after rewriting history: `rebase`, `commit --amend`, squash, or removing an accidentally committed secret. A normal push is rejected in these cases because the branches have diverged. The main danger - you can wipe out commits someone else has already pushed. That's why in practice you almost always use `git push --force-with-lease`: it refuses to overwrite the branch if something you haven't seen has appeared on the server.

## How it works

Git rejects a normal push when the remote branch isn't a direct ancestor of your local one (non-fast-forward). The `--force` flag removes this protection and moves the remote branch pointer wherever you tell it.

```bash
# Rewrote the last commits locally (rebase / amend / squash)
git rebase -i HEAD~3

# Normal push rejected: histories diverged
git push                 # ! [rejected] (non-fast-forward)

# Forcibly overwrite the remote branch
git push --force
```

The safe version checks that the remote branch points exactly where you last saw it (the value from `git fetch`). If someone managed to push before you, the operation won't go through:

```bash
git push --force-with-lease
```

## Pitfalls

- `--force` overwrites others' commits silently: if a colleague pushed and you forced on top, their work disappears from the branch.
- `--force-with-lease` only protects with a fresh `fetch`. If you did a `fetch` and then forced the lease - the check passes, and someone else's commit can still be lost.
- Never force shared branches (`main`, release ones) the team works on. Force is acceptable on personal feature branches before review.
- Lost commits live on for a while in `git reflog` and are reachable by hash, but you can't rely on this as a backup.
