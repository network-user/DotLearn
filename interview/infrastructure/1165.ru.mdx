# Жёсткая и символическая ссылка: чем отличаются на уровне inode

## Суть

Symlink представляет собой отдельный файл, хранящий путь к цели, как ярлык; он ломается, если цель переместить или удалить, и может вести в другую файловую систему. Hardlink - это второе имя того же inode: данные живут, пока есть хотя бы одно имя, но ссылка работает только внутри одной ФС.

## Что ответить на собесе

Разница в том, на что указывает ссылка:

- Символическая ссылка указывает на путь: это маленький файл с записанным внутри маршрутом к цели. Стоит удалить оригинал, и symlink повисает в воздухе (это «битая» ссылка).
- Жёсткая ссылка указывает не на путь, а прямо на inode, то есть на саму структуру с данными файла.
- Поэтому исходное имя и hardlink равноправны: удаление одного из них не трогает данные, пока счётчик ссылок inode не дойдёт до нуля.
- Symlink умеет ссылаться на каталоги и пересекать границы файловых систем, а hardlink не умеет.

Символическая ссылка представляет собой файл-указатель с маршрутом внутри:

```bash
ln -s /path/to/original/file /path/to/symlink
```

`ln -s` создаёт symlink: первый аргумент задаёт цель, а второй - место, где появится сама ссылка. Целевой путь не проверяется на существование, поэтому можно создать ссылку «в никуда».

Жёсткая ссылка добавляет ещё одно имя тому же inode:

```bash
ln /path/to/original/file /path/to/hardlink
```

После этого `ls -i` покажет одинаковый номер inode у обоих имён, а в выводе `ls -l` счётчик ссылок вырастет до 2.

## Подводные камни

- Hardlink нельзя создать на каталог и нельзя протянуть между разными файловыми системами: у них разные пространства inode.
- Удаление цели у symlink не выдаёт ошибки сразу: ссылка остаётся, но при обращении к ней получите «No such file or directory».
- Физическое освобождение места происходит не при удалении файла, а когда обнуляется счётчик жёстких ссылок inode и его никто не держит открытым.
- `cp -L` пойдёт по symlink и скопирует содержимое цели, а `cp -P` скопирует саму ссылку, так что поведение легко перепутать.
