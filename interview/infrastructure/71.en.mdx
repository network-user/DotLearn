# merge or rebase: two ways to combine branches and what each costs

## In short

`merge` and `rebase` solve the same task - bringing work from one branch into another - but in different ways. `merge` preserves history as it is and adds a merge commit, while `rebase` rewrites the commits anew, straightening history into a single line.

## How to answer in an interview

Both commands move changes between branches, but they produce different histories. `merge` creates a separate merge commit with two parents and keeps the branch points visible - the whole picture is preserved. `rebase` lifts the current branch's commits and replays them on top of another branch's tip; history becomes linear, but commit hashes change. That's why `rebase` must not be applied to branches that are already published and visible to others: rewritten history will break their work. Linearity is convenient to read, branchiness more honestly reflects how development actually went.

`merge` brings two branches together and records the result as a separate commit. This commit has two parents: one points to the current branch, the other to the branch being joined.

```bash
git checkout main
git merge feature
```

History keeps the branch:

```
A---B---C---D  (main)
     \     /
      E---F  (feature)
```

`rebase` works differently: it lifts the current branch's changes and reapplies them on top of the target's tip, as if the work had been done after its last commit. Hashes are recalculated in the process - `E'` and `F'` in the diagram.

```bash
git checkout feature
git rebase main
```

```
A---B---C---D  (main)
              \
               E'---F'  (feature)
```

Choose by situation: `merge` when full transparency and traceability of every merge matters; `rebase` when you need a flat, readable history, for example before sending work to a shared repository.

## Pitfalls

- Conflicts are possible in both cases, but with `rebase` you sometimes have to resolve them several times - commits are applied one by one on top of the new base.
- `rebase` changes hashes, so it must not be applied to public branches: after `git push --force`, colleagues will get a divergent history.
- A `merge` commit breeds "forks" in history; in large projects, an abundance of them makes `git log` hard to read.
- The linearity after `rebase` is deceptively simple: the real order and context of the work is lost in it.
