# How do you organize branching in git, and why is this discipline needed?

## In short

A workflow is a team agreement about which branches exist, where they grow from, and where they merge into. The goal is a single one: keep the release branch always stable, while work on features, releases, and urgent fixes proceeds in parallel without getting in each other's way.

## How to answer in an interview

I tailor the choice of workflow to the team and the release cadence. Vincent Driessen's classic Git Flow keeps the long-lived branches `main` and `develop` plus short-lived feature, release, and hotfix branches; it's good where you have versioned releases and several parallel streams of work. But if we ship to production several times a day, I prefer trunk-based or GitHub Flow: one main branch, short feature branches, and a quick merge via pull request with mandatory code review and a green CI. The main principle is the same: the main branch is always deployable, branches are short-lived, and the whole history stays readable. Git Flow by itself isn't an end in itself - it's a tool for a particular release rhythm.

In Git Flow there are two long-lived branches. `main` holds stable, release-ready code, and each release is marked with a tag. `develop` is the integration branch, where finished features collect before a release.

A feature branch is created off `develop` and merged back into `develop`:

```bash
git switch develop
git switch -c feature/awesome-feature
# work on the feature, a series of commits
git switch develop
git merge --no-ff feature/awesome-feature
git branch -d feature/awesome-feature
```

A release branch is prepared before a release: in it you polish the small things, bump the version, and then merge it both into `main` (with a tag) and back into `develop`:

```bash
git switch -c release/1.0.0 develop
# final fixes, version update
git switch main
git merge --no-ff release/1.0.0
git tag -a 1.0.0 -m "Release 1.0.0"
git switch develop
git merge --no-ff release/1.0.0
git branch -d release/1.0.0
```

A hotfix branch is created straight off `main` to quickly close a critical bug in production, and then it's merged both into `main` and into `develop`, otherwise the fix would be lost in the next release:

```bash
git switch -c hotfix/urgent-fix main
# the fix
git switch main
git merge --no-ff hotfix/urgent-fix
git tag -a 1.0.1 -m "Hotfix 1.0.1"
git switch develop
git merge --no-ff hotfix/urgent-fix
git branch -d hotfix/urgent-fix
```

What this gives the team: a clear separation between work on new things and on fixes, a permanently stable `main`, convenient management of releases and tags, and a transparent version history.

## Pitfalls

- **Long-lived branches.** The longer a feature branch lives detached from `develop`, the more painful the merge - the risk of conflicts grows. Break tasks down and merge often.
- **A forgotten back-merge of a hotfix.** If a fix is merged only into `main` but not into `develop`, the bug will come back in the next release.
- **Git Flow for a CI/CD pipeline.** With several deploys a day, a model with `develop` and release branches slows things down; trunk-based fits better.
- **`merge` without `--no-ff` for significant branches.** A fast-forward erases the fact that there was a separate stream of work, and the history loses its structure.
- **Direct commits to `main`.** Protect the branch with rules: only through a pull request that passes CI and review.
