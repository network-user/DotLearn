# Главные веб-уязвимости, о которых говорю на собесе

## Суть

OWASP Top 10 задаёт язык: XSS, injection (SQL/NoSQL/command), CSRF, broken auth, sensitive data exposure, misconfiguration, SSRF и др. На фронте я фокусируюсь на XSS, CSRF, небезопасном хранении токенов, supply-chain скриптов и ошибках CORS/CSP.

## Что ответить на собесе

- XSS закрываю encoding + CSP; injection на клиенте реже, но DOM XSS и template literals опасны.
- CSRF: SameSite cookies, anti-CSRF token, проверка Origin на mutating requests.
- Auth: короткий access TTL, secure/httpOnly refresh, logout = revoke server-side.
- Честно: фронт не спасёт от SQLi; но я не доверяю client-only validation.
- Регулярно смотрю npm audit и SRI для CDN scripts.

## Как это работает

Пример DOM XSS через query string:

```ts
const params = new URLSearchParams(location.search);
const nick = params.get('nick') ?? 'guest';

const label = document.querySelector('#nick-label');
if (label) label.textContent = nick;
```

Уязвимый вариант был бы `label.innerHTML = nick`, что открывает `<img onerror=...>`.

CSRF: браузер сам шлёт cookies на same-site POST. Защита:

```html
<input type="hidden" name="csrf" value="server-generated-token" />
```

Broken auth на SPA: токен в localStorage переживает XSS; session fixation если login не rotation session id (для cookie-based auth).

Sensitive data: не логирую tokens в console/analytics; маскирую PII в error reports.

## Подводные камни

- **Prototype pollution.** `merge({}, userInput)` в utils может отравить Object.prototype.
- **Open redirect.** `?next=` без allowlist ведёт на phishing после login.
- **Clickjacking + CSRF combo.** iframe + auto-submit формы.
- **Dependency confusion.** Postinstall scripts в transitive deps.
- **CORS misread.** `Access-Control-Allow-Origin: *` с credentials запрещён, но dev proxy часто слишком открыт.
