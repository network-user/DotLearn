# CVE: как фронтендер работает с уязвимостями

## Суть

CVE (Common Vulnerabilities and Exposures) это каталог публичных уязвимостей с id вида CVE-YYYY-NNNN. Для фронта это npm/transitive deps, bundler plugins, browser engines. CVSS score и advisory описывают impact; fix version в release notes.

## Что ответить на собесе

- `pnpm audit` / Dependabot в CI; блокирую merge на critical без exception ticket.
- Различаю direct vs transitive; иногда fix через override/resolution.
- CVE в devDependency (vite plugin) vs runtime dependency оцениваю по exposure.
- Честно: not every audit finding is exploitable in our bundle tree-shaking removes dead code.
- SRI + lockfile commit предотвращают supply-chain drift.

## Как это работает

Workflow:

```bash
pnpm audit --audit-level=high
pnpm update lodash --latest
pnpm why vulnerable-package
```

Advisory для `CVE-2024-XXXXX` в `webpack-dev-middleware` может не затрагивать prod build если dev server не exposed. Runtime `CVE` в `jsonwebtoken` alg bypass критичен на API, но frontend must not bundle verify secret.

SBOM (cyclonedx) помогает security team map CVE к deployed artifact. Browser CVE (V8, WebGL) патчатся auto-update Chrome; поддерживаю evergreen baseline in analytics.

## Подводные камни

- **Audit fatigue.** 200 low noise → team ignores; prioritize reachable prod paths.
- **False negatives.** Private registry mirrors lag NVD sync.
- **Breaking major bumps.** Security fix semver major blocks release; need timeboxed upgrade.
- **Transitive deadlock.** Package A depends on vulnerable B; no fix until A updates.
- **Copy-paste CDN.** Unpinned script URL bypasses lockfile entirely.
