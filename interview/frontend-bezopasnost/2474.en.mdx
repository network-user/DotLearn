# XSS attacks: mechanics and client defenses

## In short

XSS injects executable JavaScript into your origin page. The browser cannot tell legitimate bundles from injected snippets. Impact: cookie/token exfiltration (if JS-readable), keylogging, DOM phishing, actions as the user.

## How to answer in an interview

- Default escape in templates; `innerHTML` only after sanitization.
- CSP blocks unknown scripts; Trusted Types block unsafe sinks.
- HttpOnly cookies do not stop same-origin API calls from injected script.
- Honestly: sanitizers are imperfect; allowlists beat blacklists.
- Security regression tests on user-generated HTML fields.

## How it works

Vulnerable comment render:

```ts
type Comment = { author: string; body: string };

function renderUnsafe(list: Comment[]) {
  const root = document.querySelector('#feed')!;
  root.innerHTML = list
    .map((c) => `<p><b>${c.author}</b>: ${c.body}</p>`)
    .join('');
}
```

Safe variant:

```ts
function renderSafe(list: Comment[]) {
  const root = document.querySelector('#feed')!;
  root.replaceChildren(
    ...list.map((c) => {
      const p = document.createElement('p');
      const strong = document.createElement('strong');
      strong.textContent = c.author;
      p.append(strong, document.createTextNode(`: ${c.body}`));
      return p;
    })
  );
}
```

Stored XSS hits every viewer. Reflected XSS targets victims with crafted links. DOM XSS lives entirely on the client and evades edge WAF.

## Pitfalls

- **SSR template literals.** Unescaped `${user}` in HTML strings becomes reflected XSS.
- **Markdown renderers.** Raw HTML pass-through enabled by default in some libs.
- **SVG/MathML.** Namespaces bypass naive stripTags.
- **postMessage handlers.** `eval(event.data)` is a classic footgun.
- **Supply chain.** Compromised npm package counts as first-party for CSP.
