# XSS-атаки: механика и клиентская защита

## Суть

XSS внедряет исполняемый JavaScript в страницу вашего origin. Браузер не отличает легитимный bundle от injected snippet. Последствия: exfiltration cookies/tokens (если доступны JS), keylogging, DOM phishing, actions от имени пользователя.

## Что ответить на собесе

- Default escape в шаблонах; `innerHTML` только после sanitizer.
- CSP блокирует unknown scripts; Trusted Types блокируют unsafe sinks.
- HttpOnly cookies не спасают API calls из injected script (same-origin).
- Честно: sanitizers не perfect; allowlist лучше blacklist.
- Security regression tests на user-generated HTML fields.

## Как это работает

Уязвимый рендер комментариев:

```ts
type Comment = { author: string; body: string };

function renderUnsafe(list: Comment[]) {
  const root = document.querySelector('#feed')!;
  root.innerHTML = list
    .map((c) => `<p><b>${c.author}</b>: ${c.body}</p>`)
    .join('');
}
```

Безопасный вариант:

```ts
function renderSafe(list: Comment[]) {
  const root = document.querySelector('#feed')!;
  root.replaceChildren(
    ...list.map((c) => {
      const p = document.createElement('p');
      const strong = document.createElement('strong');
      strong.textContent = c.author;
      p.append(strong, document.createTextNode(`: ${c.body}`));
      return p;
    })
  );
}
```

Stored XSS масштабируется на всех viewers. Reflected бьёт жертву с crafted link. DOM XSS живёт полностью на клиенте и не виден WAF на edge.

## Подводные камни

- **Template literals in SSR.** Unescaped `${user}` в HTML string на server = reflected XSS.
- **Markdown renderers.** Raw HTML pass-through включён по default в некоторых libs.
- **SVG/MathML.** Namespaces обходят naive stripTags.
- **postMessage handlers.** `eval(event.data)` classic footgun.
- **Supply chain.** Compromised npm package = first-party script for CSP.
