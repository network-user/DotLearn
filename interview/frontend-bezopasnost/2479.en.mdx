# Can JavaScript update cookies

## In short

JS can write cookies via `document.cookie` when they are not HttpOnly. Rewriting the same name+domain+path replaces value or expiry. HttpOnly Secure cookies from the server are invisible to JS by design for session tokens.

## How to answer in an interview

- Preferences (theme, locale) may live in non-HttpOnly cookies or localStorage from JS.
- Session/auth cookies come only from Set-Cookie HttpOnly on the server.
- JS cannot set HttpOnly; the browser ignores that flag from script.
- Honestly: `document.cookie` is awkward; I use cookie helpers or server endpoints for sensitive updates.
- JS can set SameSite and Secure only when creating non-HttpOnly cookies.

## How it works

```ts
function setClientPref(name: string, value: string, days = 30) {
  const expires = new Date(Date.now() + days * 864e5).toUTCString();
  document.cookie = `${encodeURIComponent(name)}=${encodeURIComponent(value)}; expires=${expires}; path=/; SameSite=Lax`;
}

function readClientPref(name: string): string | null {
  const match = document.cookie
    .split('; ')
    .find((row) => row.startsWith(`${encodeURIComponent(name)}=`));
  if (!match) return null;
  return decodeURIComponent(match.split('=')[1] ?? '');
}

setClientPref('theme', 'dark');
```

Server session update:

```ts
await fetch('/api/session/locale', {
  method: 'POST',
  credentials: 'include',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ locale: 'ru' }),
});
```

The browser applies Set-Cookie from the response; JS cannot read HttpOnly values afterward.

## Pitfalls

- **Size limit.** ~4KB per domain cookie jar total.
- **Encoding.** Commas/spaces in values break parsing without encodeURIComponent.
- **Subdomain scope.** JS on `app.example.com` cannot freely set Domain=example.com.
- **GDPR.** Non-essential client cookies need consent in the EU.
- **SSR mismatch.** Theme cookie read on server must match client hydration.
