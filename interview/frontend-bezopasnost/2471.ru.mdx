# Content Security Policy: зачем и как включаю

## Суть

CSP это HTTP-политика whitelist, которая ограничивает источники script/style/img/fetch и embedding. Браузер блокирует нарушения до выполнения вредоносного кода. Это defense in depth против XSS и data injection, не замена sanitization.

## Что ответить на собесе

- Деплою через header на CDN; meta CSP только для static prototypes.
- Стартую report-only + Reporting-Endpoints, потом enforce без `unsafe-inline` scripts.
- Nonce генерирую per-request на SSR и пробрасываю в `<script nonce>`.
- Честно: legacy inline handlers ломаются; планирую миграцию на bundled JS.
- Комбинирую с Trusted Types где поддерживается.

## Как это работает

```http
Content-Security-Policy-Report-Only:
  default-src 'self';
  script-src 'self' 'nonce-r4nd0m' https://cdn.example;
  report-uri /csp-report
```

```html
<script nonce="r4nd0m">
  window.__BOOT__ = { featureFlags: [] };
</script>
```

Браузер сравнивает каждый load/eval с директивами. Inline script без matching nonce/hash блокируется. Violation report помогает найти забытые CDN домены до cutover.

На Vite/SPA часто две политики: строгая на HTML shell, relaxed на dev server. Production: hash для единственного inline boot snippet или полный отказ от inline.

## Подводные камни

- **Third-party widgets.** Карта/чат тянут 20 доменов; без inventory CSP дырявый.
- **WebAssembly.** Нужен `wasm-unsafe-eval` или `'unsafe-eval'` в некоторых bundler setups.
- **JSONP endpoints.** Обход `script-src` через `<script src="/api?callback=evil">`.
- **Browser extensions.** Inject scripts CSP не контролирует.
- **Report noise.** Report-only без sampling забивает logs; фильтрую duplicates.
