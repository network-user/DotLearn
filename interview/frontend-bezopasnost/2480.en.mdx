# HttpOnly: hiding cookies from JavaScript

## In short

The HttpOnly Set-Cookie flag blocks access via `document.cookie` and DOM APIs. The browser still attaches the cookie to matching HTTP requests automatically. That reduces XSS impact for session hijacking.

## How to answer in an interview

- Session, refresh, CSRF double-submit secrets: HttpOnly + Secure + SameSite.
- JS cannot read HttpOnly even for debugging; I use network tab and server logs.
- HttpOnly does not stop CSRF: the browser still sends cookies on forged requests.
- Honestly: XSS can act as the user without reading the cookie (same-origin fetch).
- Pair with short TTL and rotation on privilege changes.

## How it works

Server response:

```http
Set-Cookie: refresh=eyJ...; HttpOnly; Secure; SameSite=Strict; Path=/auth/refresh; Max-Age=604800
```

Client fetch:

```ts
await fetch('/api/profile', { credentials: 'include' });
console.log(document.cookie);
```

`document.cookie` will not list `refresh`. The browser adds the header automatically:

```http
Cookie: refresh=eyJ...
```

Express-like APIs set the flag server-side only:

```ts
response.headers.append(
  'Set-Cookie',
  `session=${sid}; HttpOnly; Secure; SameSite=Lax; Path=/`
);
```

Defense in depth: HttpOnly removes one XSS exfil path; CSP still blocks script injection.

## Pitfalls

- **Bearer in JS vs cookie auth.** Mixed models make HttpOnly irrelevant if token sits in localStorage.
- **Debug frustration.** Devs disable HttpOnly locally; must not ship.
- **Subresource requests.** HttpOnly cookies still send; CSRF tokens needed.
- **Cookie prefixes.** `__Secure-` and `__Host-` add constraints modern browsers enforce.
- **XSS still dangerous.** Attacker script can call `/api/transfer` with credentials without reading the cookie.
