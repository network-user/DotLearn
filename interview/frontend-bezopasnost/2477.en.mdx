# CVE: how frontend engineers handle vulnerabilities

## In short

CVE (Common Vulnerabilities and Exposures) is a catalog of public flaws with ids like CVE-YYYY-NNNN. For frontend work this covers npm/transitive deps, bundler plugins, and browser engines. CVSS scores and advisories describe impact; release notes give fix versions.

## How to answer in an interview

- `pnpm audit` / Dependabot in CI; block merge on critical without exception ticket.
- I separate direct vs transitive fixes; sometimes overrides/resolutions are needed.
- I weigh CVE in devDependency (vite plugin) vs runtime dependency by exposure.
- Honestly: not every audit finding is exploitable; tree-shaking may drop dead code.
- SRI plus committed lockfile prevent supply-chain drift.

## How it works

Workflow:

```bash
pnpm audit --audit-level=high
pnpm update lodash --latest
pnpm why vulnerable-package
```

An advisory for `CVE-2024-XXXXX` in `webpack-dev-middleware` may not hit prod if dev server is not exposed. Runtime `CVE` in `jsonwebtoken` alg bypass is critical on API, but frontend must not bundle verify secrets.

SBOM (cyclonedx) helps security map CVEs to deployed artifacts. Browser CVEs (V8, WebGL) patch via Chrome auto-update; I track evergreen baselines in analytics.

## Pitfalls

- **Audit fatigue.** 200 low findings → ignored; prioritize reachable prod paths.
- **False negatives.** Private registry mirrors lag NVD sync.
- **Breaking major bumps.** Security fix semver major blocks release; timebox upgrades.
- **Transitive deadlock.** Package A depends on vulnerable B until A updates.
- **Copy-paste CDN.** Unpinned script URLs bypass lockfiles entirely.
