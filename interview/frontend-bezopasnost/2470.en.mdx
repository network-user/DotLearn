# CSRF: how it works and frontend defenses

## In short

Cross-Site Request Forgery makes the victim browser send an authenticated request to your site with the user cookies while they are logged in. The attacker page triggers POST/form/img without user consent. Defenses: CSRF tokens, SameSite cookies, Origin/Referer checks.

## How to answer in an interview

- Mutations use POST/PUT/PATCH/DELETE with a custom header (`X-Requested-With` or CSRF header).
- Session cookies use SameSite=Lax/Strict; cross-site embed needs an explicit token.
- Double-submit: token in cookie plus matching header/body field.
- Honestly: pure JWT in Authorization without cookies reduces classic CSRF, but login endpoints stay exposed.
- CORS does not stop CSRF: simple requests skip preflight.

## How it works

Attacker page:

```html
<form action="https://bank.example/transfer" method="POST">
  <input type="hidden" name="amount" value="1000" />
  <input type="hidden" name="to" value="attacker" />
</form>
<script>document.forms[0].submit()</script>
```

SPA client defense:

```ts
const csrf = document.querySelector('meta[name="csrf-token"]')?.getAttribute('content');

await fetch('/api/transfer', {
  method: 'POST',
  credentials: 'include',
  headers: {
    'Content-Type': 'application/json',
    'X-CSRF-Token': csrf ?? '',
  },
  body: JSON.stringify({ amount: 100, to: 'friend' }),
});
```

The server generates a per-session token; the client reads meta or cookie and mirrors it in a header. SameSite=Lax blocks cross-site POST cookie sends in modern browsers but does not replace tokens for defense in depth.

## Pitfalls

- **GET mutations.** `GET /delete?id=1` fires via `<img src>` without JS.
- **Content-Type text/plain.** Simple requests skip CORS preflight; CSRF token still required.
- **Subdomain XSS.** `.example.com` cookies are visible to all subdomains.
- **Mobile WebView.** Custom URL schemes may bypass SameSite.
- **GraphQL POST.** One endpoint, batch mutations; CSRF token on all operations.
