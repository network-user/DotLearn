# HTTPS: asymmetric and symmetric encryption together

## In short

The TLS handshake uses asymmetric crypto (certificates, ECDHE) to authenticate the server and agree on a session key. After that all HTTP, including cookies and API JSON, is encrypted with a symmetric AEAD cipher. Frontends do not implement TLS but must understand mixed content and cert errors.

## How to answer in an interview

- The browser validates cert chains; self-signed fails unless enterprise trust stores allow it.
- After TLS 1.3 the bulk traffic uses AES-128/256-GCM or ChaCha20.
- HSTS and upgrade-insecure-requests prevent downgrade.
- Honestly: TLS terminates at the CDN; origin may be HTTP on a private network.
- Certificate Transparency logs help detect mis-issued certs.

## How it works

Simplified flow:

1. Client ClientHello with cipher suites.
2. Server Certificate + ServerHello; client verifies CA signature and domain match.
3. ECDHE exchange → shared premaster → derive session keys.
4. Application data encrypted symmetrically.

```ts
if (location.protocol !== 'https:') {
  console.warn('insecure context: Web Crypto and Secure cookies limited');
}

const res = await fetch('https://api.example/data', { credentials: 'include' });
```

Mixed content: HTTPS pages loading `http://` scripts break the padlock and may leak Referer. Browsers block active mixed content.

Frontend impact: Secure cookies are ignored on http:// dev except localhost. WebCrypto `subtle` requires secure contexts.

## Pitfalls

- **Corporate MITM roots.** Users still see a padlock; pinning breaks legit inspection.
- **TLS termination misconfig.** HTTP between CDN and origin on public networks.
- **Cert expiry.** Monitoring alerts; auto ACME on infra.
- **Legacy ciphers.** Old Android clients force TLS1.2 downgrade policies.
- **Local dev certs.** mkcert trust is dev-machine only unless imported in CI.
