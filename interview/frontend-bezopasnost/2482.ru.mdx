# HTTPS: асимметричное и симметричное шифрование вместе

## Суть

TLS handshake использует асимметрию (certificates, ECDHE) для аутентификации сервера и согласования session key. Дальше весь HTTP, включая cookies и API JSON, шифруется симметричным AEAD cipher. Фронтендер не реализует TLS, но должен понимать why mixed content и cert errors matter.

## Что ответить на собесе

- Browser validates cert chain; self-signed fails unless enterprise trust store.
- After TLS 1.3 handshake bulk traffic AES-128/256-GCM or ChaCha20.
- HSTS и upgrade-insecure-requests предотвращают downgrade.
- Честно: TLS terminates at CDN; origin may be HTTP behind private network.
- Certificate Transparency logs help detect mis-issued certs.

## Как это работает

Упрощённый flow:

1. Client ClientHello с cipher suites.
2. Server Certificate + ServerHello; client verifies signature CA → domain match.
3. ECDHE exchange → shared premaster → derive session keys.
4. Application data encrypted symmetrically.

```ts
if (location.protocol !== 'https:') {
  console.warn('insecure context: Web Crypto and Secure cookies limited');
}

const res = await fetch('https://api.example/data', { credentials: 'include' });
```

Mixed content: HTTPS page loading `http://` script breaks padlock and may leak Referer. Browsers block active mixed content.

Frontend impact: Secure cookies ignored on http:// dev unless localhost exception. WebCrypto `subtle` restricted to secure contexts.

## Подводные камни

- **Corporate MITM roots.** Users see same padlock; pinning breaks legit inspection.
- **TLS termination misconfig.** HTTP between CDN and origin on public network.
- **Cert expiry.** Monitoring alerts; auto ACME on infra side.
- **Weak cipher legacy.** Old Android clients force TLS1.2 cipher downgrade policies.
- **Local dev certs.** mkcert trust only on dev machine, not CI headless unless imported.
