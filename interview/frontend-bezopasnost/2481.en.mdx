# Symmetric encryption: frontend use cases

## In short

Symmetric algorithms (AES-GCM, ChaCha20-Poly1305) use one shared secret for encrypt/decrypt. They are fast for bulk data. On the frontend they appear in TLS session keys, Web Crypto local encryption, and encrypted localStorage patterns; keys arrive via asymmetric handshakes.

## How to answer in an interview

- TLS encrypts HTTP with a symmetric session key (AES-GCM) after the handshake.
- Web Crypto `subtle.encrypt` with AES-GCM for client-side file encryption when required.
- Never hardcode secrets in bundles; derive via PBKDF2/Argon2 from user password or RSA wrap.
- Honestly: browsers cannot hide keys from XSS; IndexedDB encryption is readable after decrypt.
- Prefer platform APIs (passkeys) over DIY crypto.

## How it works

```ts
async function aesGcmEncrypt(plain: Uint8Array, rawKey: CryptoKey) {
  const iv = crypto.getRandomValues(new Uint8Array(12));
  const cipher = await crypto.subtle.encrypt({ name: 'AES-GCM', iv }, rawKey, plain);
  return { iv, cipher: new Uint8Array(cipher) };
}

async function importAesKey(secretBytes: Uint8Array) {
  return crypto.subtle.importKey('raw', secretBytes, 'AES-GCM', false, ['encrypt', 'decrypt']);
}
```

Key exchange sketch: client generates random AES key, encrypts with server RSA public key from cert, both use AES for bulk traffic. Messaging E2E libs (Signal) layer the same idea.

Symmetric crypto wins on speed; key distribution is the weakness. HTTPS combines asymmetric bootstrap plus a symmetric tunnel.

## Pitfalls

- **Legacy ECB.** Never invent modes; use GCM or ChaCha20-Poly1305.
- **IV reuse.** Same key+IV breaks GCM confidentiality.
- **Key in localStorage.** XSS steals wrapped or raw key material.
- **Math.random.** Use `crypto.getRandomValues` only.
- **Compliance theater.** Client AES without a threat model adds complexity, not security.
