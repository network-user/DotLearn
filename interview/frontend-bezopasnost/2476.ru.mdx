# Аутентификация в веб-приложениях

## Суть

Аутентификация отвечает на вопрос «кто ты?»: проверка identity через пароль, OTP, WebAuthn, SSO. Это предшествует authorization (что тебе разрешено). На фронте собираю credentials, передаю по TLS, получаю session/token; решения о доступе не принимаю на клиенте.

## Что ответить на собесе

- Различаю authentication vs authorization vs identification.
- Пароли только POST на HTTPS; на клиенте не храню plaintext, optional client-side hash только если spec требует.
- MFA/WebAuthn снижает phishing; push OTP хуже FIDO2.
- Честно: «запомнить меня» увеличивает window compromise; refresh rotation обязателен.
- Generic error messages на login («неверные данные»), не «user not found».

## Как это работает

```ts
async function submitLogin(form: { email: string; password: string }) {
  const res = await fetch('/auth/login', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(form),
  });
  if (!res.ok) throw new Error('invalid credentials');
  return res.json() as Promise<{ mfaRequired?: boolean }>;
}

async function verifyTotp(code: string) {
  await fetch('/auth/mfa', {
    method: 'POST',
    credentials: 'include',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code }),
  });
}
```

Факторы: knowledge (password), possession (OTP key), inherence (biometric via WebAuthn). Passwordless passkeys используют public-key crypto; phishing-resistant.

После успеха сервер выдаёт session cookie или tokens. Frontend routing guard проверяет `/me` endpoint, не local flag «isLoggedIn=true» без verify.

## Подводные камни

- **Client-only auth gate.** `if (localStorage.token)` без server verify = fake security.
- **Credential stuffing.** Rate limit и CAPTCHA на login UI coordination с API.
- **OAuth login button.** Social login всё равно нужен account linking policy.
- **Autofill leaks.** Login form в hidden iframe для scraping.
- **Logout everywhere.** Forgotten tabs with valid refresh after password reset.
