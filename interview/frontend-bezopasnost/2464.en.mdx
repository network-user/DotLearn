# Key web vulnerabilities I discuss in interviews

## In short

OWASP Top 10 sets the vocabulary: XSS, injection (SQL/NoSQL/command), CSRF, broken auth, sensitive data exposure, misconfiguration, SSRF, and more. On the frontend I focus on XSS, CSRF, unsafe token storage, supply-chain scripts, and CORS/CSP mistakes.

## How to answer in an interview

- I mitigate XSS with encoding plus CSP; client-side injection is rarer but DOM XSS and template literals still hurt.
- CSRF: SameSite cookies, anti-CSRF tokens, Origin checks on mutating requests.
- Auth: short access TTL, secure/httpOnly refresh, logout means server-side revoke.
- Honestly: the frontend cannot fix SQLi, but I never trust client-only validation.
- I run npm audit and SRI for CDN scripts regularly.

## How it works

DOM XSS via query string example:

```ts
const params = new URLSearchParams(location.search);
const nick = params.get('nick') ?? 'guest';

const label = document.querySelector('#nick-label');
if (label) label.textContent = nick;
```

The vulnerable version would be `label.innerHTML = nick`, enabling `<img onerror=...>`.

CSRF: the browser sends cookies on same-site POST automatically. Defense:

```html
<input type="hidden" name="csrf" value="server-generated-token" />
```

Broken auth on SPAs: tokens in localStorage survive XSS; session fixation if login does not rotate session id (cookie-based auth).

Sensitive data: never log tokens to console/analytics; mask PII in error reports.

## Pitfalls

- **Prototype pollution.** `merge({}, userInput)` in utils can poison Object.prototype.
- **Open redirect.** `?next=` without allowlist enables phishing after login.
- **Clickjacking + CSRF combo.** iframe plus auto-submit forms.
- **Dependency confusion.** Postinstall scripts in transitive deps.
- **CORS misread.** `Access-Control-Allow-Origin: *` with credentials is forbidden, but dev proxies are often too open.
