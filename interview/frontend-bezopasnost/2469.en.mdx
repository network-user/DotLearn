# Access token: the client-side role

## In short

An access token is a short-lived credential for API access after login. The client sends it via `Authorization: Bearer` or as a cookie (BFF). The server validates signature/scopes and returns resources without re-entering the password.

## How to answer in an interview

- I obtain access after OAuth/password grant; TTL is minutes, not days.
- I attach it only to API calls, never log it or put it in URLs.
- On 401 I try refresh once, then redirect to login.
- Honestly: access in JS memory is lost on reload; need refresh/bootstrap flow.
- I read scopes for UI gating, but authorization stays server-side.

## How it works

```ts
async function login(username: string, password: string) {
  const res = await fetch('/auth/token', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ username, password }),
  });
  if (!res.ok) throw new Error('auth failed');
  return (await res.json()) as { accessToken: string; expiresIn: number };
}

async function fetchProfile(accessToken: string) {
  const res = await fetch('/api/me', {
    headers: { Authorization: `Bearer ${accessToken}` },
  });
  if (!res.ok) throw new Error('unauthorized');
  return res.json();
}
```

OAuth 2.0 access tokens may be opaque or JWT. Bearer means any holder may use it; transport must be TLS. Some IdPs roll access on activity.

On the frontend I store expiry timestamp and proactively refresh 30–60 seconds before exp to avoid mid-form failures.

## Pitfalls

- **Bearer leakage.** Token in WebSocket query or `<img src>` leaks via logs.
- **Over-scoped token.** UI hides a button, but APIs must enforce scope.
- **Clock skew.** Client expiry timers without drift buffer cause false 401s.
- **Multiple tabs.** BroadcastChannel syncs logout/invalidate.
- **Third-party APIs.** CORS plus header token works; cross-site cookie token does not.
