# Content Security Policy: purpose and rollout

## In short

CSP is an HTTP whitelist policy that limits script/style/img/fetch sources and embedding. The browser blocks violations before malicious code runs. It is defense in depth against XSS and injection, not a replacement for sanitization.

## How to answer in an interview

- I deploy via CDN headers; meta CSP only for static prototypes.
- I start report-only plus Reporting-Endpoints, then enforce without script `unsafe-inline`.
- I generate per-request nonces on SSR and pass them to `<script nonce>`.
- Honestly: legacy inline handlers break; I plan migration to bundled JS.
- I combine with Trusted Types where supported.

## How it works

```http
Content-Security-Policy-Report-Only:
  default-src 'self';
  script-src 'self' 'nonce-r4nd0m' https://cdn.example;
  report-uri /csp-report
```

```html
<script nonce="r4nd0m">
  window.__BOOT__ = { featureFlags: [] };
</script>
```

The browser matches every load/eval against directives. Inline script without a matching nonce/hash is blocked. Violation reports surface forgotten CDN hosts before enforcement.

On Vite/SPAs I often run strict policy on the HTML shell and relaxed on dev. Production: hash the single inline boot snippet or drop inline entirely.

## Pitfalls

- **Third-party widgets.** Maps/chat pull many domains; CSP is porous without inventory.
- **WebAssembly.** Needs `wasm-unsafe-eval` or `'unsafe-eval'` in some bundler setups.
- **JSONP endpoints.** Bypass `script-src` via `<script src="/api?callback=evil">`.
- **Browser extensions.** Injected scripts ignore CSP.
- **Report noise.** Report-only without sampling floods logs; dedupe duplicates.
