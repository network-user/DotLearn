# Ранжирование в запросах: ПЕРВЫЕ N и УПОРЯДОЧИТЬ ПО

## Суть

`ПЕРВЫЕ N` ограничивает число возвращаемых строк после сортировки. `УПОРЯДОЧИТЬ ПО` задаёт порядок; без индекса по полям сортировки СУБД сортирует весь набор. Для топа по группам используют подзапрос или оконные аналоги через временные таблицы.

## Что ответить на собесе

- Я ставлю `ПЕРВЫЕ` только после осмысленного `УПОРЯДОЧИТЬ ПО`.
- Для «топ 10 по каждому складу» один глобальный ПЕРВЫЕ 10 неверен.
- Сужаю выборку в ГДЕ до сортировки, не полагаюсь на ПЕРВЫЕ как на фильтр.
- На больших таблицах проверяю план: sort vs index seek.
- Честно: в старых платформах нет полноценных window functions как в SQL:2011.

## Как это работает

```bsl
"ВЫБРАТЬ ПЕРВЫЕ 10
|    Продажи.Номенклатура,
|    СУММА(Продажи.Сумма) КАК Сумма
|ИЗ
|    Документ.Реализация.Товары КАК Продажи
|ГДЕ
|    Продажи.Ссылка.Дата >= &НачалоМесяца
|СГРУППИРОВАТЬ ПО
|    Продажи.Номенклатура
|УПОРЯДОЧИТЬ ПО
|    Сумма УБЫВ"
```

## Подводные камни

**ПЕРВЫЕ без ORDER.** Недетерминированный набор.
**Сортировка по вычисляемому полю без индекса.** Полная сортировка миллионов строк.
**Топ внутри группы одним ПЕРВЫЕ.** Неверная бизнес-логика.
**УПОРЯДОЧИТЬ в подзапросе без лимита.** Всё равно тяжёлый sort.
**Путаница с РАЗЛИЧНЫЕ.** Другая операция, другая стоимость.
