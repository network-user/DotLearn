# Временная таблица запроса vs подзапрос в 1С

## Суть

Подзапрос в WHERE или SELECT выполняется в контексте внешнего запроса, часто многократно. `ПОМЕСТИТЬ` материализует промежуточный результат один раз; выгодно при тяжёлом отборе и повторном использовании в нескольких join.

## Что ответить на собесе

- Я выбираю ПОМЕСТИТЬ, если подзапрос используется в нескольких join или дорогой.
- Коррелированный подзапрос в SELECT на каждую строку заменяю join или ВТ.
- Маленький независимый подзапрос в WHERE может быть дешевле ВТ.
- Индексирую ВТ вторым запросом при большом join.
- Честно: лишняя ВТ на весь справочник без отбора ухудшает план.

## Как это работает

```bsl
"ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТКонтрагенты
|ИЗ Справочник.Контрагенты
|ГДЕ Регион = &Регион;
|ВЫБРАТЬ ...
|ИЗ ВТКонтрагенты КАК ВТ
|    ВНУТРЕННЕЕ СОЕДИНЕНИЕ Документ.Заказ КАК З
|    ПО ВТ.Ссылка = З.Контрагент"
```

## Подводные камни

**Коррелированный подзапрос в SELECT.** N выполнений.
**ВТ без отбора.** Копия большой таблицы в tempdb.
**Забытый ИНДЕССИРОВАТЬ.** Медленный следующий шаг.
**Дубли ВТ и подзапроса.** Две стратегии с разной актуальностью.
**ВТ между несвязанными пакетами.** Нужен менеджер временных таблиц.
