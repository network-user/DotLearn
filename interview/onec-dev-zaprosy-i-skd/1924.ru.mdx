# Функция ВЫБОР в запросе 1С и стоимость вычислений

## Суть

`ВЫБОР ... КОГДА ... ИНАЧЕ ... КОНЕЦ` в SELECT вычисляется для каждой строки результата. Длинные ветки с подзапросами в WHEN умножают стоимость; простые сравнения обычно дешевле join к справочнику классификации.

## Что ответить на собесе

- Я держу ВЫБОР по простым полям строки, без вложенных запросов в условиях.
- Для больших классификаций предпочитаю join к регистру или справочнику.
- ВЫБОР в ГДЕ может мешать использованию индекса на поле.
- Стабильные правила выношу в вычисляемое поле представления или регистр.
- Честно: десятки вложенных WHEN хуже поддерживаются, чем таблица соответствий.

## Как это работает

```bsl
"ВЫБРАТЬ
|    ВЫБОР
|        КОГДА Ост.КоличествоОстаток < 0 ТОГДА ""Дефицит""
|        КОГДА Ост.КоличествоОстаток = 0 ТОГДА ""Нет""
|        ИНАЧЕ ""В наличии""
|    КОНЕЦ КАК Статус"
```

Для сложной матрицы статусов join к `РегистрСведений.КлассификацияОстатков`.

## Подводные камни

**Подзапрос в КОГДА.** Выполняется per row.
**ВЫБОР по неиндексируемому выражению в ГДЕ.** Full scan.
**Дублирование CASE в отчётах.** Расхождение правил.
**Смешение типов в ветках.** Ошибка компиляции или неявный cast.
**ВЫБОР вместо нормализации данных.** Бизнес-логика размазана по запросам.
