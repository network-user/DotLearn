# Автофиксация в динамическом списке и производительность

## Суть

Автофиксация заставляет динамический список периодически перечитывать данные с сервера при изменениях в базе. Удобно для мониторинга, но на тяжёлых запросах даёт лишнюю нагрузку и мерцание UI.

## Что ответить на собесе

- Я отключаю автофиксацию на списках с тяжёлым join и без индексов.
- Для оперативных очередей включаю с увеличенным интервалом или по событию обновления.
- Основной рычаг производительности это упрощение запроса списка, не только интервал.
- Сравниваю план запроса динамического списка с отчётным вариантом.
- Честно: автофиксация не заменяет push; при сотнях пользователей нагрузка умножается.

## Как это работает

В свойствах динамического списка: `АвтоФиксация = Истина` и интервал обновления. Запрос списка держу с отбором по периоду и индексируемым полям. При открытии формы устанавливаю параметры `&НачалоПериода` для сужения выборки.

## Подводные камни

**Тяжёлый запрос + автофиксация.** Постоянные full scan на сервере.
**Слишком частый интервал.** CPU и сеть растут линейно с числом форм.
**Нет отбора по организации.** RLS и лишние строки одновременно.
**Скрытые вычисляемые поля в списке.** Пересчёт на каждую строку.
**Ожидание мгновенного обновления.** Пользователи не видят чужие изменения без фиксации или обновления.
