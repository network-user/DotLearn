# Query execution order in the 1C optimizer

## In short

The 1C optimizer translates the query to SQL and hands the plan to the DBMS. Join and filter order may differ from query text: the DBMS reorders by statistics. Virtual tables expand to subqueries with their own cost.

## How to answer in an interview

- I do not assume `FROM` order is execution order.
- I narrow early in `WHERE` and virtual table parameters.
- I analyze plans in SQL Server/PostgreSQL for heavy reports.
- Temp tables help materialize a small filter before join.
- I am honest: hints in 1C are limited; main tools are simpler queries and DBMS indexes.

## How it works

Expensive join after small filter:

```bsl
"ВЫБРАТЬ Номенклатура.Ссылка ПОМЕСТИТЬ ВТНом
|ИЗ Справочник.Номенклатура КАК Номенклатура
|ГДЕ Номенклатура.Артикул В (&СписокАртикулов);
|ВЫБРАТЬ ... ИЗ ВТНом КАК ВТ
|    ВНУТРЕННЕЕ СОЕДИНЕНИЕ РегистрНакопления.Остатки.Остатки(&Дата, ) КАК Ост
|    ПО ВТ.Ссылка = Ост.Номенклатура"
```

## Pitfalls

**Full register before filter.** Optimizer may not push predicate into virtual table.
**Implicit type cast.** Index not used.
**OR instead of UNION.** Worse plan than two branches.
**Functions on field in WHERE.** Scan instead of seek.
**Expecting text order.** Wrong mental model for tuning.
