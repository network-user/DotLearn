# Время жизни фикстуры: как scope в pytest управляет переиспользованием ресурсов

## Суть

`scope` фикстуры задаёт, как часто pytest её создаёт и уничтожает. Чем шире scope, тем реже выполняется setup/teardown и тем больше тестов делят один экземпляр ресурса.

## Что ответить на собесе

Выбор scope - это компромисс между изоляцией и скоростью, и есть четыре основных уровня жизни фикстуры.

- `function` - заново на каждый тест (значение по умолчанию), `class` - один экземпляр на тестовый класс, `module` - один на файл, `session` - один на весь прогон.
- Узкий scope даёт максимальную изоляцию, но платит временем на повторные setup/teardown.
- Широкий scope ускоряет прогон, переиспользуя дорогой ресурс вроде БД или сетевого соединения, но требует, чтобы тесты не портили общее состояние.

## Как это работает

pytest кэширует результат фикстуры в пределах её scope: пока область активна, повторные запросы возвращают тот же объект, а код после `yield` (teardown) выполняется при выходе из области.

```python
import pytest

@pytest.fixture(scope="function")
def resource():
    print("setup")
    yield "db"
    print("teardown")

def test_one(resource):
    assert resource == "db"
```

При `scope="function"` для каждого теста печатается setup и teardown. Меняем на `scope="module"` - и для всех тестов файла setup напечатается один раз:

```python
import pytest

@pytest.fixture(scope="module")
def resource():
    print("setup once per module")
    yield "db"
    print("teardown once per module")

def test_one(resource):
    assert resource == "db"

def test_two(resource):
    assert resource == "db"
```

`session` идёт ещё шире - ресурс поднимается один раз перед первым тестом и закрывается после последнего. Под капотом порядок такой: фикстуры с более широким scope инстанцируются раньше, а узкие могут зависеть от широких, но не наоборот.

## Подводные камни

- Широкий scope делит состояние между тестами. Если один тест мутирует общий ресурс, следующий может упасть или, хуже, незаметно пройти - изоляция теряется.
- `function` стоит по умолчанию неспроста: при сомнениях берите его, расширяйте scope только под доказанную дорогую инициализацию.
- Фикстура узкого scope не может быть зависимостью фикстуры более широкого: `session`-фикстура не вправе требовать `function`-фикстуру.
- Для параметризованных фикстур помните, что число setup/teardown умножается на число параметров, а не только на число тестов.
- `autouse=True` в связке с широким scope легко создаёт скрытые глобальные побочные эффекты - применяйте осторожно.
