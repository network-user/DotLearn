# Integration tests vs. functional tests: what exactly does each one check?

## In short

Integration tests put the seams between modules under the microscope: do the components pass data to each other correctly? Functional tests look from the outside — does the system produce the result that the spec requires, without diving into its internal structure?

## How to answer in an interview

These are two cross-sections with different points of view. An integration test takes several modules already verified separately, connects them, and checks that data isn't lost or distorted at the boundaries — that is, it tests the interaction. A functional test works like a black box: we feed an input, compare the output against the requirement, and how exactly it was computed inside doesn't matter. One answers the question "do the modules get along?", the other "does the product do what was promised to the user?".

## How it works

Integration testing comes as the next level after unit testing. First each module is covered with unit tests, then they are assembled into a combination and the contract at the boundaries is checked: data format, call order, handling of a neighbor's errors.

```python
def get_data():
    return "data"

def process_data(data):
    return data.upper()

def test_integration():
    raw = get_data()
    result = process_data(raw)
    assert result == "DATA"
```

Functional testing operates only on public behavior. We take a function or feature described in the requirements and check a set of inputs and expected outputs.

```python
def add(a, b):
    return a + b

def test_add_functional():
    assert add(2, 3) == 5
    assert add(-1, 1) == 0
    assert add(0, 0) == 0
```

The key difference is in focus: an integration test may know about the internal modules and their interfaces (this is gray-box/white-box), while a functional test deliberately doesn't know about them (black box).

## Pitfalls

- These categories aren't mutually exclusive. A single functional scenario often exercises the integration of several modules along the way — but the goal and the success criterion of the test must be stated explicitly, otherwise when it fails it's unclear what exactly broke.
- "Integration" is often confused with "system." Integration checks specific seams; system testing checks the whole product in a production-like environment.
- If a functional test starts peeking into an object's internal structures, it stops being a black box and becomes fragile: refactoring without changing behavior will break the tests.
- Integration tests are slower and more capricious than unit tests (external dependencies, network, database), so they are kept as a thin layer on top of a broad base of unit tests, not the other way around.
