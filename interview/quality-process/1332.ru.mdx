# В чём смысл правила DRY и где оно начинает вредить?

## Суть

DRY («Don't Repeat Yourself») требует, чтобы каждый фрагмент знания о системе имел единственное авторитетное представление. Дублирование логики оборачивается будущими багами: правку придётся разносить по всем копиям, и одну из них рано или поздно забудут.

## Что ответить на собесе

DRY нацелен на устранение дублирования знания, а не просто похожих строк.

- Если одна и та же бизнес-логика или формула живёт в нескольких местах, при изменении придётся править каждую копию, и легко пропустить одну, из-за чего возникают рассинхрон и ошибки.
- Поэтому повторяющуюся логику выношу в одну функцию, класс или модуль с единственным источником правды.
- Оговорка: два куска, которые сегодня выглядят одинаково, но меняются по разным причинам, объединять не стоит; это ложное DRY, которое позже придётся мучительно расцеплять.
- Здравым ориентиром служит правило трёх: первое повторение терплю, на третьем выношу абстракцию.

## Как это работает

Возьмём расчёт площади. Без DRY логика размазана по двум почти одинаковым функциям:

```python
def calculate_area_of_rectangle(width, height):
    return width * height

def calculate_area_of_square(side):
    return side * side

rectangle_area = calculate_area_of_rectangle(5, 10)
square_area = calculate_area_of_square(5)
```

С DRY общий расчёт собран в одном месте:

```python
def calculate_area(width, height):
    return width * height

rectangle_area = calculate_area(5, 10)
square_area = calculate_area(5, 5)
```

Теперь формула площади описана однажды. Если правило изменится (скажем, добавится валидация размеров), правка будет ровно в одной точке. На уровнях выше DRY работает так же: общие константы и конфиги хранятся в одном источнике, повторяющиеся схемы переиспользуют общие модели, а дублирующиеся куски шаблонов подключаются через включения и компоненты.

## Подводные камни

- **Ложное DRY.** Похожий код ≠ одинаковое знание. Если два фрагмента совпали случайно и эволюционируют по-разному, их объединение создаёт хрупкую связанность.
- **Перевыпрямление.** Погоня за нулём повторений плодит универсальные функции с кучей флагов и `if`-ов; читать их тяжелее, чем исходное дублирование.
- **Связанность через абстракцию.** Вынесли общий модуль, и теперь несвязанные части системы жёстко зависят друг от друга; правка под один сценарий ломает другой.
- **DRY против KISS.** Иногда небольшое дублирование проще и безопаснее, чем хитрая абстракция ради его устранения.
- **Игнор правила трёх.** Абстрагирование на первом же совпадении означает ранний прыжок к обобщению, которое потом не подойдёт.
