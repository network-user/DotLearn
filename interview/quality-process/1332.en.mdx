# What is the point of the DRY rule, and where does it start to hurt?

## In short

DRY ("Don't Repeat Yourself") requires every piece of knowledge about the system to have a single authoritative representation. Duplicated logic is a future bug: a fix has to be spread across all copies, and sooner or later one of them gets forgotten.

## How to answer in an interview

DRY is about removing duplication of knowledge, not just of similar lines. If the same business logic or formula lives in several places, every change forces you to edit each copy, and it's easy to miss one - which leads to drift and bugs. So I move repeated logic into a single function, class, or module with one source of truth. But there's an important caveat: DRY is about knowledge, not about superficial code similarity. Two chunks that look identical today but change for different reasons should not be merged - that's false DRY, which you later have to painfully untangle. A sound guideline is the rule of three: I tolerate the first repetition and extract an abstraction at the third.

## How it works

Take area calculation. Without DRY, the logic is smeared across two nearly identical functions:

```python
def calculate_area_of_rectangle(width, height):
    return width * height

def calculate_area_of_square(side):
    return side * side

rectangle_area = calculate_area_of_rectangle(5, 10)
square_area = calculate_area_of_square(5)
```

With DRY, the shared calculation is gathered in one place:

```python
def calculate_area(width, height):
    return width * height

rectangle_area = calculate_area(5, 10)
square_area = calculate_area(5, 5)
```

Now the area formula is described once. If the rule changes (say, dimension validation is added), the fix lives in exactly one spot. At higher levels DRY works the same way: shared constants and configs in a single source, repeated schemas via reusable models, duplicated template fragments via includes and components.

## Pitfalls

- **False DRY.** Similar code != the same knowledge. If two fragments coincide by accident and evolve differently, merging them creates fragile coupling.
- **Over-abstraction.** Chasing zero repetition breeds universal functions with a pile of flags and `if`s - harder to read than the original duplication.
- **Coupling through abstraction.** You extract a shared module, and now unrelated parts of the system are tightly dependent on each other; a fix for one scenario breaks another.
- **DRY vs. KISS.** Sometimes a little duplication is simpler and safer than a clever abstraction built solely to remove it.
- **Ignoring the rule of three.** Abstracting on the very first match is an early leap to a generalization that later won't fit.
