# Why do teams choose pytest over the standard unittest?

## In short

Pytest gives a more concise syntax, discovers tests on its own, has built-in parametrization and fixtures, and gathers plugins. unittest from the standard library works, but it requires more boilerplate and ceremony around classes.

## How to answer in an interview

The main argument is less ceremony for the same or greater expressiveness. In unittest a test must live as a method inside a class that subclasses `TestCase`, and assertions are made with methods like `assertEqual`. In pytest a plain function and a simple `assert` are enough, and on failure pytest parses the expression itself and shows which values didn't match. On top of that: automatic test discovery, concise parametrization, powerful fixtures with scopes, and a large ecosystem of plugins (coverage, Django, CI). At the same time pytest can also run existing unittest tests, so the migration doesn't have to be abrupt.

## How it works

Comparing the same test shows the difference in code volume:

```python
import unittest

class TestExample(unittest.TestCase):
    def test_addition(self):
        self.assertEqual(1 + 1, 2)
```

```python
def test_addition():
    assert 1 + 1 == 2
```

Pytest collects functions from `test_*.py` and `*_test.py` files itself, so you don't need to register them manually. Parametrization and fixtures are built in:

```python
import pytest

@pytest.mark.parametrize("a, b, expected", [(1, 1, 2), (2, 3, 5), (3, 5, 8)])
def test_addition(a, b, expected):
    assert a + b == expected

@pytest.fixture
def sample_data():
    return {"key": "value"}

def test_sample_data(sample_data):
    assert sample_data["key"] == "value"
```

When an `assert` fails, pytest applies expression introspection and shows detailed diagnostics — you see the specific values, not just the fact of a mismatch.

## Pitfalls

- "Simpler" doesn't mean "always more appropriate." If a project is already entirely on unittest and tightly coupled to its API, rewriting for taste is expensive — and pytest can run unittest tests as they are.
- A bare `assert` in production code is stripped out by the `-O` optimization flag. In pytest tests this is fine, but don't carry the "I check via assert" habit into production logic.
- Fixtures are powerful, but when abused they hide data setup so deep that the test becomes magical. Keep fixture chains short and explicit.
- A rich plugin ecosystem is both a plus and a source of version conflicts. Pin the versions of pytest and plugins, otherwise updating one can break the build.
