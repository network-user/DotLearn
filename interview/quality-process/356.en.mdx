# Swapping code on the fly: why monkey patching matters in tests

## In short

Monkey patching means replacing the behavior of already-written code right at runtime. In tests it cuts the code under test off from the outside world: instead of the real function, your own one is temporarily substituted.

## How to answer in an interview

Monkey patching is the dynamic replacement of an attribute, function, or method at runtime. In tests I use it to isolate a module from the network, the database, or third-party libraries: I substitute a stub that returns a known answer in advance. This makes tests fast, repeatable, and independent of external services. In practice I usually do it through `unittest.mock.patch`, so the substitution is rolled back automatically after the test.

## How it works

In Python, names are references inside a module's namespace, and they can be redefined while the program is running. The substitution lives only in the memory of the current process and lasts for a limited time.

Take a function that calls an external API:

```python
import requests

def get_data_from_api(url):
    response = requests.get(url)
    return response.json()
```

To test it without a real request, temporarily replace `requests.get`:

```python
from unittest.mock import patch

def fake_get(url):
    class FakeResponse:
        def json(self):
            return {"key": "value"}
    return FakeResponse()

with patch("requests.get", fake_get):
    data = get_data_from_api("http://example.com")
    print(data)  # {'key': 'value'}
```

Inside the `with` block the name `requests.get` points to `fake_get`. As soon as the block finishes, the original is put back automatically.

## Pitfalls

- You need to patch where the name is used, not where it is defined. If a module does `from requests import get`, you patch `my_module.get`, not `requests.get`.
- Manual substitution without `patch` is dangerous: if the test fails before the rollback, the dirty state leaks into neighboring tests. A context manager or decorator rolls everything back by itself.
- Over-mocking hides real integration bugs — patch the boundary of the system, not its internal logic.
