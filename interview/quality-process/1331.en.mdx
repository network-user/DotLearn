# What's behind the KISS principle and how does it shape code?

## In short

KISS ("Keep It Simple, Stupid") calls for choosing the simplest solution among those that solve the task. A simple system is easier to read, cheaper to maintain, and calmer to change under new requirements.

## How to answer in an interview

KISS is the stance of not over-complicating without need. I phrase it like this: a solution should be exactly as complex as the task requires, and not a gram more. In practice that means choosing a straightforward algorithm over a clever one, a clear name over an explanatory comment, a flat structure over extra layers of abstraction. The payoff is in the cost of ownership: a new person gets into the project faster, bugs are easier to track down, changes go in without fear of breaking something. It's important not to confuse KISS with primitiveness: the goal isn't "less code at any cost" but minimal complexity while preserving correctness and readability.

## How it works

The principle works at every level. In architecture - we remove extra layers and duplication, and don't introduce an abstraction that currently has a single implementation. In code - we write straightforwardly and avoid "clever" one-liners that save a line but steal half an hour to decipher. In documentation - short and to the point.

Compare a "clever" and a simple version of the same function:

```python
def calculate_area(width, height):
    return width * height

area = calculate_area(5, 10)
print(f"Area: {area}")
```

There's nothing extra here: the name says what's being computed, the body is a single expression, the call is obvious. An alternative with a "flexible" interface (`*args`, parameter dicts, dynamic formula selection) would look smarter, but without a real need that's complexity for nothing - it costs more to maintain than the benefit of imagined universality.

KISS is close to YAGNI ("you aren't gonna need it"): don't build in generalization for scenarios that don't exist yet.

## Pitfalls

- **Simplicity ≠ primitiveness.** Sometimes a task is inherently complex (concurrency, security), and you can't simplify below its intrinsic complexity - you'll get hidden bugs.
- **"Clever" code instead of simple code.** Three-story ternaries and magic tricks shrink the line count but raise reading complexity - that's anti-KISS.
- **Premature abstraction.** A layer "just in case it's useful" is a typical violation of KISS and YAGNI.
- **Simplicity as an excuse for holes.** Simplifying doesn't mean throwing out error handling and edge cases.
- **Subjectivity.** "Simple" for the author and for the reader are different things; the benchmark is clarity for the team, not for yourself.
