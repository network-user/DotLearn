# unittest, pytest и mock: чем эти инструменты отличаются на практике

## Суть

`unittest` — встроенный в Python фреймворк с классами и методами-проверками. `pytest` — внешний инструмент с лаконичным синтаксисом, фикстурами и плагинами. `mock` — не фреймворк запуска, а библиотека для подмены зависимостей заглушками.

## Что ответить на собесе

Это три инструмента из разных слоёв: два раннера и одна библиотека подмены, которые не противопоставляют, а сочетают.

- `pytest` — мой основной раннер: лаконичнее, расширяется плагинами, главное отличие — обычный `assert` вместо набора методов `assertEqual`, плюс мощные фикстуры и параметризация.
- `unittest` — встроенный в стандартную библиотеку, строится на классах-наследниках `TestCase`; беру там, где не хочу тянуть зависимости.
- `mock` решает другую задачу: не запускает тесты, а создаёт фиктивные объекты, чтобы проверять логику в отрыве от БД, сети и сторонних API.

## Как это работает

`unittest` собирает тесты в классы, наследующие `TestCase`, а проверки делает через методы:

```python
import unittest

class TestMath(unittest.TestCase):
    def test_addition(self):
        self.assertEqual(1 + 1, 2)

if __name__ == "__main__":
    unittest.main()
```

`pytest` находит функции с префиксом `test_` и использует обычный `assert`, разбирая выражение под капотом для понятного сообщения об ошибке:

```python
def test_addition():
    assert 1 + 1 == 2
```

Запуск — команда `pytest` в терминале. `mock` решает ортогональную задачу — подмену объектов:

```python
from unittest.mock import Mock

service = Mock()
service.method.return_value = "mocked result"

def test_mock():
    assert service.method() == "mocked result"
```

## Подводные камни

- `unittest` и `pytest` отвечают за запуск тестов, а `mock` — за изоляцию. Это разные слои, их не противопоставляют, а сочетают.
- В `pytest` параметризация (`@pytest.mark.parametrize`) заменяет копипаст одинаковых тестов с разными данными — этим часто пренебрегают.
- `Mock` принимает обращение к любому атрибуту и не падает на опечатке в имени метода. Для защиты от этого берут `create_autospec` или `spec=`, иначе тест зелёный, а вызов несуществующего метода проходит мимо.
