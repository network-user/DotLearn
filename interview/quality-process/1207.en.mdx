# Fixture lifetime: how scope in pytest controls resource reuse

## In short

A fixture's `scope` defines how often pytest creates and tears it down. The wider the scope, the less frequently setup/teardown runs and the more tests share a single instance of the resource.

## How to answer in an interview

A fixture has four main lifetime levels: `function` - recreated for each test (the default), `class` - one instance per test class, `module` - one per file, `session` - one for the whole run. A narrow scope gives maximum isolation but pays in time on repeated setup/teardown. A wide scope speeds up the run by reusing an expensive resource like a DB or a network connection, but it requires that tests don't corrupt the shared state. Choosing a scope is a trade-off between isolation and speed.

## How it works

pytest caches the fixture's result within its scope: while the scope is active, repeat requests return the same object, and the code after `yield` (teardown) runs when the scope exits.

```python
import pytest

@pytest.fixture(scope="function")
def resource():
    print("setup")
    yield "db"
    print("teardown")

def test_one(resource):
    assert resource == "db"
```

With `scope="function"`, setup and teardown are printed for each test. Switch to `scope="module"` - and setup is printed once for all tests in the file:

```python
import pytest

@pytest.fixture(scope="module")
def resource():
    print("setup once per module")
    yield "db"
    print("teardown once per module")

def test_one(resource):
    assert resource == "db"

def test_two(resource):
    assert resource == "db"
```

`session` goes wider still - the resource is brought up once before the first test and closed after the last. Under the hood, the order is: wider-scoped fixtures are instantiated earlier, and narrow ones may depend on wide ones, but not the other way around.

## Pitfalls

- A wide scope shares state between tests. If one test mutates the shared resource, the next may fail or, worse, silently pass - isolation is lost.
- `function` is the default for a reason: when in doubt, take it, and widen the scope only for proven, expensive initialization.
- A narrow-scoped fixture can't be a dependency of a wider one: a `session` fixture may not require a `function` fixture.
- For parametrized fixtures, remember that the number of setup/teardown runs is multiplied by the number of parameters, not just the number of tests.
- `autouse=True` combined with a wide scope easily creates hidden global side effects - use it carefully.
