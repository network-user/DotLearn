# CI, Continuous Delivery, and Continuous Deployment: where is the line?

## In short

CI is responsible for code being merged into a shared branch frequently and built and tested automatically. CD is responsible for changes that pass the checks reaching environments automatically. The difference between the two "CD"s is whether there is a manual approval before production.

## How to answer in an interview

CI (Continuous Integration) is the frequent merging of changes into a shared repository, where every commit triggers an automatic build and tests; the goal is to catch conflicts and breakages early. CD expands two ways. Continuous Delivery means changes are automatically prepared and promoted all the way up to the production environment, but the final push to production is triggered by a human pressing a button. Continuous Deployment goes further: everything that passes the tests deploys itself to production with no manual step. So the line between Delivery and Deployment is the presence of a manual approval before the live environment. CI is about quality; CD is about the speed and reliability of delivery.

## How it works

**Continuous Integration.** A developer commits changes, the CI system (GitHub Actions, GitLab CI, Jenkins) catches the commit, runs the build and tests, and returns the result. A green build means the changes can be merged. This reduces "integration hell," where branch divergences pile up over weeks.

**Continuous Delivery.** A layer on top of CI: a successful build is automatically deployed to staging, and integration/acceptance tests are run. The artifact is always ready to release, but the "to prod" button is pressed by a human.

**Continuous Deployment.** The same pipeline, but without the manual button: it passed the tests, so it goes to production. This requires mature automation, reliable tests, and monitoring, because every change is immediately visible to users.

## Pitfalls

- **Two different "CD"s.** In an interview, clarify which one is meant: Delivery (with manual approval) or Deployment (fully automatic). They are often conflated.
- **CI without frequent commits is not CI.** If branches live for weeks and merge once a month, having a pipeline doesn't make the process "continuous."
- **Continuous Deployment without good monitoring is dangerous.** Since there's no manual step, automated tests, feature flags, and alerts have to catch regressions, otherwise a bug instantly reaches every user.
- **Green tests != a safe deploy.** You also need rollbacks, zero-downtime DB migrations, and rollout strategies (canary, blue-green).
- **CI/CD is about culture, not just tooling.** Jenkins or Actions guarantee nothing on their own without the discipline of small changes and trust in the tests.
