# What a mock object is and why you need it in testing

## In short

A mock is a controllable fake of a real object. It mirrors the external interface of a dependency but doesn't do its actual work, letting you check logic in isolation from the database, the network, and third-party services.

## How to answer in an interview

A mock is a stub object that replaces a real dependency in a test. I give it the return values or exceptions I need and substitute it for the real service. This gives you four things: isolation of the code under test, speed (no waiting on the network and the database), full control over scenarios, and independence from the state of external systems. As a bonus, a mock remembers how it was called — you can verify that the code reached the dependency with the right arguments.

## How it works

In Python, mocks are made through `unittest.mock`. The object is created empty, and the behavior you need is configured through `return_value`:

```python
from unittest.mock import Mock

api = Mock()
api.get_data.return_value = {"key": "value"}

def fetch_data(client):
    return client.get_data()

result = fetch_data(api)
print(result)  # {'key': 'value'}
```

Here `api` pretends to be an API client: the `get_data` method returns the given dictionary, and `fetch_data` is tested without a real request. On top of that, the mock keeps a history of calls, so you can make sure the call happened exactly once and with the expected arguments:

```python
api.get_data.assert_called_once()
```

## Pitfalls

- A plain `Mock` accepts access to any attribute and won't fail on a typo. To make the stub match the real interface, use `spec=` or `create_autospec`.
- `return_value` returns the same value on every call. If you need different answers in sequence, use `side_effect` with a list or an exception.
- A mock checks the contract, not the real integration. If you substitute too much, the test stays green even when the real dependency is broken — so balance mocks with integration tests.
