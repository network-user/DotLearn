# unittest, pytest, and mock: how these tools differ in practice

## In short

`unittest` is Python's built-in framework with classes and assertion methods. `pytest` is an external tool with concise syntax, fixtures, and plugins. `mock` is not a test runner at all but a library for replacing dependencies with stubs.

## How to answer in an interview

My baseline set is this: `pytest` as the main runner, because it is more concise and extends through plugins, and `unittest.mock` for isolation from external systems. I know and use `unittest` where I don't want to pull in dependencies: it ships in the standard library and is built on classes that subclass `TestCase`. The main difference with `pytest` is a plain `assert` instead of a set of `assertEqual` methods, plus powerful fixtures and parametrization. And `mock` handles a different job: it doesn't run tests, it creates fake objects so you can check logic in isolation from the database, the network, and third-party APIs.

## How it works

`unittest` collects tests into classes that subclass `TestCase`, and makes assertions through methods:

```python
import unittest

class TestMath(unittest.TestCase):
    def test_addition(self):
        self.assertEqual(1 + 1, 2)

if __name__ == "__main__":
    unittest.main()
```

`pytest` finds functions with the `test_` prefix and uses a plain `assert`, parsing the expression under the hood for a clear error message:

```python
def test_addition():
    assert 1 + 1 == 2
```

You run it with the `pytest` command in the terminal. `mock` solves an orthogonal task — substituting objects:

```python
from unittest.mock import Mock

service = Mock()
service.method.return_value = "mocked result"

def test_mock():
    assert service.method() == "mocked result"
```

## Pitfalls

- `unittest` and `pytest` are responsible for running tests, while `mock` is responsible for isolation. These are different layers; you don't pit them against each other, you combine them.
- In `pytest`, parametrization (`@pytest.mark.parametrize`) replaces copy-pasting identical tests with different data — this is often neglected.
- `Mock` accepts access to any attribute and won't fail on a typo in a method name. To guard against that, use `create_autospec` or `spec=`, otherwise the test is green while a call to a nonexistent method slips through.
