# Substituting dependencies in tests: what patch from unittest.mock is for

## In short

`unittest.mock.patch` substitutes a mock for the real object for the duration of a test, then restores the original afterward. This lets you cut the code under test off from external dependencies and control their behavior.

## How to answer in an interview

`patch` temporarily replaces an attribute in a given place with a mock and automatically rolls the substitution back when the test finishes. I use it when the code reaches into the outside world - API, DB, files, time - and I want to test the logic, not a third-party service. The key point: you patch where the name is used, not where it's defined. If a function does `import requests; requests.get(...)`, I patch `requests.get`, but if the module says `from requests import get`, I have to patch `my_module.get`. You can apply `patch` as a decorator, as a context manager, or manually via `start()/stop()`.

## How it works

`patch` takes a string path to an object, replaces it with a `MagicMock` for the duration of the test, and restores the original in `finally`. The mock records how it was called, so you can both set a response and verify the call happened.

```python
import requests

def fetch_data(url):
    response = requests.get(url)
    return response.json()
```

A test with no real request:

```python
from unittest.mock import patch
import unittest

class TestFetchData(unittest.TestCase):
    @patch("requests.get")
    def test_fetch_data(self, mock_get):
        mock_get.return_value.json.return_value = {"key": "value"}

        result = fetch_data("http://example.com")

        self.assertEqual(result, {"key": "value"})
        mock_get.assert_called_once_with("http://example.com")

if __name__ == "__main__":
    unittest.main()
```

The decorator passes the mock into the test argument. The `return_value.json.return_value` chain configures the result of `requests.get(...).json()`. The same thing with a context manager, when the substitution is only needed in part of the test:

```python
from unittest.mock import patch

with patch("requests.get") as mock_get:
    mock_get.return_value.json.return_value = {"key": "value"}
    assert fetch_data("http://example.com") == {"key": "value"}
```

## Pitfalls

- The main trap is "patch where it's used". People patch the place of definition, but the module under test has already imported the name into itself, and the mock doesn't take effect.
- Multiple `@patch` decorators are injected into the arguments bottom-up: the decorator closest to the function corresponds to the first parameter.
- `Mock` accepts any call and any attribute and doesn't fail on typos. Use `autospec=True` or `spec=` to make the mock mirror the real object's signature.
- A manual `patcher.start()` without `patcher.stop()` (or without `addCleanup`) leaks into neighboring tests.
- To raise an exception instead of changing the return value, set `mock_get.side_effect = ConnectionError(...)`, not `return_value`.
