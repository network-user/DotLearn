# Почему команды выбирают pytest вместо стандартного unittest?

## Суть

Pytest даёт более лаконичный синтаксис, сам находит тесты, встроенно умеет параметризацию и фикстуры и обрастает плагинами. unittest из стандартной библиотеки работает, но требует больше шаблонного кода и церемоний вокруг классов.

## Что ответить на собесе

Главный аргумент: меньше церемоний при той же или большей выразительности.

- В unittest тест обязан жить методом внутри класса-наследника `TestCase`, а проверки делаются методами вроде `assertEqual`; в pytest достаточно обычной функции и простого `assert`.
- При падении pytest сам разбирает выражение и показывает, какие значения не сошлись.
- Сверху добавляются автопоиск тестов, лаконичная параметризация, мощные фикстуры с областями видимости и большая экосистема плагинов (покрытие, Django, CI).
- При этом pytest умеет запускать и существующие unittest-тесты, так что миграция не обязана быть резкой.

## Как это работает

Сравнение одного и того же теста показывает разницу в объёме кода:

```python
import unittest

class TestExample(unittest.TestCase):
    def test_addition(self):
        self.assertEqual(1 + 1, 2)
```

```python
def test_addition():
    assert 1 + 1 == 2
```

Pytest сам собирает функции из файлов `test_*.py` и `*_test.py`, поэтому регистрировать их вручную не нужно. Параметризация и фикстуры встроены:

```python
import pytest

@pytest.mark.parametrize("a, b, expected", [(1, 1, 2), (2, 3, 5), (3, 5, 8)])
def test_addition(a, b, expected):
    assert a + b == expected

@pytest.fixture
def sample_data():
    return {"key": "value"}

def test_sample_data(sample_data):
    assert sample_data["key"] == "value"
```

При падении `assert` pytest применяет интроспекцию выражения и показывает развёрнутую диагностику: видно конкретные значения, а не просто факт несовпадения.

## Подводные камни

- «Проще» не значит «всегда уместнее». Если проект уже целиком на unittest и плотно завязан на его API, переписывать ради вкуса дорого, ведь pytest как раз умеет запускать unittest-тесты как есть.
- Голый `assert` в production-коде вырезается флагом оптимизации `-O`. В тестах под pytest это нормально, но не переносите привычку «проверяю через assert» в боевую логику.
- Фикстуры мощные, но при злоупотреблении прячут подготовку данных так глубоко, что тест становится магическим. Держите цепочки фикстур короткими и явными.
- Богатая экосистема плагинов - это и плюс, и источник конфликтов версий. Закрепляйте версии pytest и плагинов, иначе обновление одного может уронить сборку.
