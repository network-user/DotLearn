# Что такое mock-объект и зачем он нужен при тестировании

## Суть

Mock — это управляемая подделка реального объекта. Он повторяет внешний интерфейс зависимости, но не выполняет её настоящую работу, давая проверить логику в изоляции от БД, сети и сторонних сервисов.

## Что ответить на собесе

Mock — это объект-заглушка, которым в тесте подменяют реальную зависимость: задаю ему нужные возвращаемые значения или исключения и подставляю вместо настоящего сервиса.

- Изоляция тестируемого участка и независимость от состояния внешних систем.
- Скорость: не ждём сеть и БД.
- Полный контроль над сценариями, включая ошибки и исключения.
- Бонусом mock запоминает, как его вызывали, — можно проверить, что код обратился к зависимости с правильными аргументами.

## Как это работает

В Python моки делают через `unittest.mock`. Объект создаётся пустым, а нужное поведение настраивается через `return_value`:

```python
from unittest.mock import Mock

api = Mock()
api.get_data.return_value = {"key": "value"}

def fetch_data(client):
    return client.get_data()

result = fetch_data(api)
print(result)  # {'key': 'value'}
```

Здесь `api` притворяется клиентом API: метод `get_data` отдаёт заданный словарь, и `fetch_data` проверяется без реального запроса. Дополнительно mock хранит историю обращений, поэтому можно убедиться, что вызов был ровно один и с ожидаемыми аргументами:

```python
api.get_data.assert_called_once()
```

## Подводные камни

- Обычный `Mock` принимает обращение к любому атрибуту и не падает на опечатке. Чтобы заглушка соответствовала настоящему интерфейсу, используют `spec=` или `create_autospec`.
- `return_value` отдаёт одно и то же значение на каждый вызов. Если нужны разные ответы по очереди, берут `side_effect` со списком или исключением.
- Мок проверяет контракт, а не реальную интеграцию. Если подменить слишком много, тест останется зелёным даже при сломанной настоящей зависимости — поэтому моки уравновешивают интеграционными тестами.
