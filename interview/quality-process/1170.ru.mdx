# Параметризация тестов: как один тест прогнать на десятке наборов данных?

## Суть

Параметризация означает запуск одной и той же тестовой функции на разных наборах входов и ожидаемых результатов. В `pytest` за это отвечает декоратор `@pytest.mark.parametrize`, который превращает один тест в семейство независимых проверок без копипасты.

## Что ответить на собесе

Когда нужно проверить функцию на множестве случаев, плохим решением будет наплодить почти одинаковых тестов, отличающихся только числами; параметризация решает это одним декоратором.

- Задаю имена параметров и список наборов значений, а pytest сам разворачивает их в отдельные тест-кейсы.
- Каждый набор запускается изолированно, и в отчёте видно, какой именно случай упал, а не «тест целиком красный».
- Итог: больше покрытия, меньше дублирования и копеечная стоимость добавления нового случая; дописал кортеж и всё.

## Как это работает

Декоратор `@pytest.mark.parametrize` принимает два аргумента: строку с именами параметров через запятую и список наборов значений (обычно кортежей). Pytest для каждого набора создаёт отдельный запуск тестовой функции, подставляя значения в её аргументы.

```python
import pytest

def add(a, b):
    return a + b

@pytest.mark.parametrize("input1, input2, expected", [
    (1, 2, 3),
    (4, 5, 9),
    (10, 20, 30),
])
def test_add(input1, input2, expected):
    assert add(input1, input2) == expected
```

Этот код порождает три независимых теста. Если упадёт только средний случай, два других всё равно отметятся как пройденные, и в выводе будет видно конкретный набор `(4, 5, 9)`.

## Подводные камни

- Имена в строке параметров должны точно совпадать с именами аргументов функции и их порядком в кортежах; рассинхрон молча подставит данные не туда.
- Не прячьте логику в сами параметры. Если для генерации ожидаемого значения нужен код, лучше явные литералы, иначе тест начинает повторять реализацию и теряет смысл.
- Слишком широкий список наборов превращает один баг в стену красных строк. Дробите по смыслу и используйте `pytest.param(..., id="...")`, чтобы случаи читались в отчёте.
- Параметризацию теста (через `mark`) часто путают с параметризацией фикстуры (через `params`). Первое размножает один тест, второе размножает источник данных, который могут разделять несколько тестов.
