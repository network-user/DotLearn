# Fixtures in pytest: how to prepare the test environment without duplication?

## In short

A fixture is a function that prepares data or an environment before a test and, if needed, cleans up after itself afterward. A test receives its result simply by declaring an argument of the same name, while the preparation itself is moved out of the test body and reused by many tests.

## How to answer in an interview

Fixtures solve the problem of repeated setup and cleanup. For example, tests need a clean database before each run — instead of duplicating setup and teardown in every test, I move it into a function with the `@pytest.fixture` decorator. The test declares the fixture name as an argument, pytest calls it itself and substitutes the result. Through `yield` a fixture can also release resources after the test. On top of that: control over how often it is recreated via `scope`, parametrization, and automatic attachment via `autouse`. The result: cleaner tests, no copy-paste, and data preparation living in one place.

## How it works

A basic fixture is declared with a decorator and returns data. The test receives it by naming the fixture in its arguments:

```python
import pytest

@pytest.fixture
def sample_data():
    return {"key": "value"}

def test_example(sample_data):
    assert sample_data["key"] == "value"
```

To clean up after the test, use `yield`: the code before it is preparation, the code after it is cleanup that runs even if the test failed.

```python
@pytest.fixture
def db_connection():
    conn = open_connection()
    yield conn
    conn.close()
```

Additional levers:

```python
@pytest.fixture(params=[1, 2, 3])
def number(request):
    return request.param

@pytest.fixture(scope="module")
def setup_module():
    return "module setup"

@pytest.fixture(autouse=True)
def auto_setup():
    pass
```

`params` multiplies the test across data sets, `scope` (`function`, `class`, `module`, `session`) sets how often the fixture is recreated, and `autouse=True` attaches it to all tests without an explicit argument.

## Pitfalls

- `return` and `yield` are not the same. Returning via `return` gives no cleanup phase; to release resources you need `yield` with code after it.
- `scope` directly affects isolation. A `session` fixture with mutable state can drag "dirt" between tests, and they will start depending on the run order.
- `autouse=True` is convenient but treacherous: the fixture fires silently for all tests, and the cause of an unexpected state is hard to track down later. Use it sparingly.
- Chains of fixtures that depend on one another easily turn into a tangled graph. Keep them flat and explicit so the test reads without digging.
