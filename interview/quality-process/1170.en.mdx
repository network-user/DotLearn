# Parametrizing tests: how to run one test against a dozen data sets?

## In short

Parametrization means running the same test function against different sets of inputs and expected results. In `pytest` this is handled by the `@pytest.mark.parametrize` decorator, which turns one test into a family of independent checks without copy-paste.

## How to answer in an interview

When you need to check a function across many cases, the bad path is to breed near-identical tests that differ only in numbers. Parametrization solves this with a single decorator: I specify the parameter names and a list of value sets, and pytest expands them into separate test cases by itself. Each set runs in isolation, and the report shows exactly which case failed instead of "the whole test is red." This gives more coverage, less duplication, and a negligible cost for adding a new case — append a tuple and you're done.

## How it works

The `@pytest.mark.parametrize` decorator takes two arguments: a string with comma-separated parameter names and a list of value sets (usually tuples). For each set, pytest creates a separate run of the test function, substituting the values into its arguments.

```python
import pytest

def add(a, b):
    return a + b

@pytest.mark.parametrize("input1, input2, expected", [
    (1, 2, 3),
    (4, 5, 9),
    (10, 20, 30),
])
def test_add(input1, input2, expected):
    assert add(input1, input2) == expected
```

This code spawns three independent tests. If only the middle case fails, the other two are still marked as passed, and the output shows the specific set `(4, 5, 9)`.

## Pitfalls

- The names in the parameter string must exactly match the function's argument names and their order in the tuples — a mismatch silently feeds data into the wrong place.
- Don't hide logic inside the parameters themselves. If generating the expected value requires code, prefer explicit literals — otherwise the test starts to repeat the implementation and loses its meaning.
- Too broad a list of sets turns a single bug into a wall of red lines. Split by meaning and use `pytest.param(..., id="...")` so the cases read well in the report.
- Test parametrization (via `mark`) is often confused with fixture parametrization (via `params`). The first multiplies one test, the second multiplies a data source that several tests can share.
