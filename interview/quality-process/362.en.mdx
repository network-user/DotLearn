# Levels of testing: from unit checks to acceptance

## In short

Tests are divided by scope of coverage: unit tests check individual functions, integration tests check a combination of components, system tests check the product as a whole, and acceptance tests check conformance to the customer's requirements. The higher the level, the broader the coverage and the fewer such tests there are.

## How to answer in an interview

I distinguish four levels. Unit tests check an isolated piece of code (a function or a class) and are written by the developers themselves. Integration tests look at how modules fit together. System tests run the whole product under conditions close to production. Acceptance tests confirm that the system covers the user's requirements. In practice this is a pyramid: many fast, cheap unit tests at the base and a few expensive end-to-end checks at the top.

## How it works

**Unit.** Check an individual function or class on different inputs. They catch bugs early and make refactoring cheaper:

```python
def add(a, b):
    return a + b

def test_add():
    assert add(2, 3) == 5
    assert add(-1, 1) == 0
```

**Integration.** Confirm that modules work correctly together. Example: a payments module and a users module, where we check that a user successfully completes a payment.

**System.** Run the whole system through real scenarios. For a web application this is the end-to-end path: registration, login, an operation, logout.

**Acceptance.** Check behavior against the customer's requirements. For example, the requirement "handle 1000 concurrent users" is verified by a dedicated acceptance test.

## Pitfalls

- An inverted pyramid is a common mistake: many slow end-to-end tests and few unit tests. Such a suite runs for a long time and fails fragilely.
- The line between unit and integration tests blurs when a real database or network is dragged into a "unit" test. At this level, external dependencies are replaced with mocks.
- A high unit-test coverage percentage doesn't guarantee that components fit together. Gaps at the seams are caught precisely by integration and system tests.
