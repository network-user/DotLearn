# TDD: what does writing the test before the code mean, and where are its limits?

## In short

TDD (Test-Driven Development) flips the usual order: first you write a failing test that describes the desired behavior, and only then the minimal code that makes it pass. The test becomes a specification, not a belated check.

## How to answer in an interview

TDD is development through the "red - green - refactor" cycle. First I write a test for behavior that doesn't exist yet, and it fails (red). Then I add the minimum code to make the test pass (green). Then I clean up the code without changing behavior, leaning on the test as a safety net (refactor). The upsides: code is thought through for edge cases in advance, regressions are caught instantly, refactoring is safe, and the tests serve as living documentation. The downsides: it's slower at the start, it demands the discipline to stick to the cycle, and it doesn't fit hard-to-test layers like UI. I apply TDD where the logic is clear and lends itself to unit tests, and I don't force it onto parts that are hard to isolate.

## How it works

The three-step cycle repeats for every new grain of behavior.

1. **Red - write a failing test.** There's no code yet, so the test must fail.

```python
def test_addition():
    assert add(2, 3) == 5
```

2. **Green - write the minimum of code** to make the test pass. No extra functionality ahead of time.

```python
def add(a, b):
    return a + b
```

3. **Refactor - improve the code** without changing behavior. The test from step 1 guarantees we haven't broken anything.

It matters that "minimum of code" means literally the minimum: TDD forces you to write exactly as much as is covered by tests and to grow behavior in small steps.

What the approach gives you in practice:

- **Quality** - edge cases are thought through before implementation.
- **Fewer regressions** - automated tests signal a break immediately.
- **Safe refactoring** - you change the code relying on green tests.
- **Documentation** - the test shows how the code should behave.

## Pitfalls

- TDD doesn't guarantee the absence of bugs: tests check what you thought to check. Poorly designed tests give a false sense of safety.
- The "green" step tempts you to immediately write the "right big" code - this breaks the discipline of small steps and loses the point of the cycle.
- Hard-to-test layers (UI, integrations with the outside world) fit pure TDD poorly; they're pushed beyond the isolation boundary.
- TDD isn't the same as "there are tests." You can have high coverage without TDD; the essence of TDD is precisely the order: the test dictates the design of the code.
