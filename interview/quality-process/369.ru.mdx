# Интеграционные тесты против функциональных: что именно проверяет каждый вид?

## Суть

Интеграционные тесты ставят под микроскоп стыки между модулями: правильно ли компоненты передают данные друг другу. Функциональные тесты смотрят снаружи: выдаёт ли система тот результат, который требуют от неё по спецификации, не вникая в её внутреннее устройство.

## Что ответить на собесе

Это два среза с разной точкой зрения на систему.

- Интеграционный тест берёт несколько уже проверенных по отдельности модулей, соединяет их и проверяет, что на границах не теряются и не искажаются данные, то есть тестирует взаимодействие.
- Функциональный тест работает как чёрный ящик: подаём вход, сверяем выход с требованием, а как именно внутри это посчиталось, неважно.
- Коротко: один отвечает на вопрос «модули дружат?», второй отвечает на вопрос «продукт делает то, что обещали пользователю?».

## Как это работает

Интеграционное тестирование идёт следующим уровнем после модульного. Сначала каждый модуль закрыт юнит-тестами, затем их собирают в связку и проверяют контракт на границах: формат данных, порядок вызовов, обработку ошибок соседа.

```python
def get_data():
    return "data"

def process_data(data):
    return data.upper()

def test_integration():
    raw = get_data()
    result = process_data(raw)
    assert result == "DATA"
```

Функциональное тестирование оперирует только публичным поведением. Берём функцию или фичу, описанную в требованиях, и проверяем набор входов и ожидаемых выходов.

```python
def add(a, b):
    return a + b

def test_add_functional():
    assert add(2, 3) == 5
    assert add(-1, 1) == 0
    assert add(0, 0) == 0
```

Ключевая разница в фокусе: интеграционный тест может знать про внутренние модули и их интерфейсы (это серый/белый ящик), а функциональный сознательно про них не знает (чёрный ящик).

## Подводные камни

- Эти категории не взаимоисключающие. Один функциональный сценарий нередко по дороге задействует интеграцию нескольких модулей, но цель и критерий успеха у теста должны быть заявлены явно, иначе при падении непонятно, что именно сломалось.
- «Интеграционный» часто путают с «системным». Интеграция проверяет конкретные стыки, а системное тестирование проверяет продукт целиком в боевом окружении.
- Если функциональный тест начинает заглядывать во внутренние структуры объекта, он перестаёт быть чёрным ящиком и становится хрупким: рефакторинг без смены поведения будет валить тесты.
- Интеграционные тесты медленнее и капризнее юнитов (внешние зависимости, сеть, БД), поэтому их держат тонким слоем поверх широкой базы юнит-тестов, а не наоборот.
