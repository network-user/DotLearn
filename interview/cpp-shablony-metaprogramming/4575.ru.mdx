# Что компилятор генерирует из шаблонной функции для разных типов?

## Суть

Для каждого уникального набора аргументов шаблон инстанцируется в отдельную нетемплейтную функцию с подставленными типами. Одна и та же логика в исходнике превращается в несколько специализированных машинных реализаций на этапе компиляции.

## Что ответить на собесе

- Говорю, что шаблон сам по себе не существует в бинарнике: есть только конкретные инстанцирования.
- Привожу пример `pick_larger<T>` с вызовами для `int` и `double` и объясняю, что получится две функции.
- Упоминаю неявный вывод аргументов: `pick_larger(1, 2)` создаёт версию для `int` без явного `<int>`.
- Отмечаю, что компилятор может инлайнить и оптимизировать каждую версию отдельно.
- Честно добавляю: чрезмерное инстанцирование раздувает бинарник; иногда выношу общую логику в нетемплейтную функцию.

## Как это работает

Шаблонная функция это чертёж. При первом вызове с конкретными типами компилятор подставляет их вместо параметра `T` и генерирует обычную функцию. Повторный вызов с тем же набором типов переиспользует уже созданную версию.

```cpp
template <typename T>
T pick_larger(T lhs, T rhs) {
    return (lhs > rhs) ? lhs : rhs;
}

int main() {
    int ai = 5, bi = 10;
    double ad = 5.5, bd = 10.5;

    int max_i = pick_larger(ai, bi);
    double max_d = pick_larger(ad, bd);
    (void)max_i;
    (void)max_d;
}
```

После компиляции логически появляются две функции:

```cpp
int pick_larger(int lhs, int rhs) {
    return (lhs > rhs) ? lhs : rhs;
}

double pick_larger(double lhs, double rhs) {
    return (lhs > rhs) ? lhs : rhs;
}
```

Имена в объектном файле могут быть mangled, но семантика именно такая. Если вызвать `pick_larger(1, 2.0)` без приведения, компилятор не сможет вывести единый `T` и выдаст ошибку. Это и есть статическая типобезопасность шаблонов.

Так устроены `std::sort`, `std::max`, контейнеры STL: одна декларация в заголовке, множество инстанцирований в клиентском коде. Linker может отбросить неиспользуемые инстанцирования, но в больших проектах стоит следить за дублированием через explicit instantiation или общие нетиповые ядра.

## Подводные камни

- **Раздувание кода.** Десятки типов дают десятки копий одной функции в бинарнике.
- **Ошибки в глубине инстанцирования.** Сообщение компилятора ссылается на строку шаблона, а не на место вызова; нужно читать всю цепочку.
- **Разные типы в одном вызове.** `T` должен быть единым, иначе вывод аргументов ломается.
- **Скрытые инстанцирования.** Дружественные функции и адрес шаблона `&pick_larger<int>` могут потянуть генерацию неожиданных версий.
- **ODR и inline.** Несколько идентичных инстанцирований в разных TU обычно схлопываются, но расхождение тел даёт UB.
