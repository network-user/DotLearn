# Что происходит при инстанцировании шаблона?

## Суть

Инстанцирование шаблона это момент, когда компилятор подставляет конкретные типы или значения в шаблон и генерирует реальную функцию или класс. До вызова или объявления с конкретными аргументами шаблон остаётся лишь чертежом без отдельного бинарного кода.

## Что ответить на собесе

- Определяю термин: implicit instantiation при использовании и explicit через `template void foo<int>();` или `template class vector<int>;`.
- Объясняю, что для каждой уникальной комбинации аргументов появляется своя специализация в объектном файле.
- Упоминаю ODR: определение шаблона должно быть видно в точке инстанцирования (поэтому заголовки).
- Привожу пример с одной шаблонной функцией и двумя типами, получающими разный машинный код.
- Честно добавляю: чрезмерное инстанцирование раздувает бинарник; иногда выносят реализацию в `.cpp` с явным списком типов.

## Как это работает

Шаблон функции или класса описывает семейство сущностей. Когда пишут `clamp(3, 7, 5)` с шаблоном `clamp<T>`, компилятор выводит `T = int` и создаёт функцию `clamp<int>`. Другой вызов с `double` породит `clamp<double>`. Это не копипаста исходника вручную, а генерация на этапе компиляции.

```cpp
#include <iostream>

template <typename T>
T clamp_value(T lo, T hi, T val) {
    if (val < lo) return lo;
    if (val > hi) return hi;
    return val;
}

int main() {
    auto a = clamp_value(0, 10, 15);
    auto b = clamp_value(0.0, 1.0, 0.75);
    std::cout << a << ' ' << b << '\n';
}
```

Строка с `int` и строка с `double` вызывают два разных инстанцирования. В дизассемблере это две отдельные функции, хотя исходник один. Для классов `std::vector<int>` и `std::vector<double>` компилятор строит два независимых типа с разными размерами и методами.

Явное инстанцирование ускоряет сборку: один `.cpp` инстанцирует `Foo<int>`, остальные единицы трансляции линкуются с готовым символом. Без явного инстанцирования каждый TU мог бы сгенерировать копию, и линкер оставит одну по ODR.

## Подводные камни

- **Шаблон в .cpp без explicit.** Другие единицы трансляции получат undefined reference.
- **Раздувание кода.** Десятки типов в горячем шаблоне умножают размер исполняемого файла.
- **Зависимые ошибки.** Ошибка всплывает только при конкретном `T`, не при парсинге шаблона.
- **Линковка weak symbols.** Дубли implicit instantiation обычно схлопываются, но explicit управляет этим явно.
- **Экспорт шаблонов (C++20 modules).** Модель видимости меняется, но суть подстановки типов та же.
