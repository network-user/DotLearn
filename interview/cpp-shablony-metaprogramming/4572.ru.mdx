# Как показать частичную специализацию шаблона на примере?

## Суть

Частичная специализация позволяет задать отдельную реализацию шаблона для части параметров, не фиксируя весь набор. Это удобно, когда для указателей, массивов или конкретных комбинаций типов нужна иная семантика, чем у общего шаблона.

## Что ответить на собесе

- Объясняю, что полная специализация фиксирует все параметры, а частичная оставляет хотя бы один открытым.
- Привожу пример с `Storage<T>` и `Storage<T*>`, где для указателей разыменование скрыто внутри `getValue()`.
- Уточняю, что частичная специализация работает для классов, но не для функций (для них используют перегрузки).
- Отмечаю, что специализация должна быть объявлена после основного шаблона и видна в точке инстанцирования.
- Честно добавляю: для функций с разным поведением по типам чаще выбираю перегрузку или `if constexpr`, а не специализацию.

## Как это работает

Первичный шаблон описывает поведение по умолчанию. Когда компилятор видит набор аргументов, подходящий под частичную специализацию, он выбирает её вместо общего варианта. Для указателей типичный сценарий: хранить адрес, но наружу отдавать значение по адресу.

```cpp
#include <iostream>

template <typename T>
class Holder {
public:
    explicit Holder(T stored) : stored_(stored) {}

    T read() const { return stored_; }

private:
    T stored_;
};

template <typename T>
class Holder<T*> {
public:
    explicit Holder(T* ptr) : ptr_(ptr) {}

    T read() const { return *ptr_; }

private:
    T* ptr_;
};

int main() {
    Holder<int> by_value{42};
    int payload = 100;
    Holder<int*> by_pointer{&payload};

    std::cout << by_value.read() << '\n';
    std::cout << by_pointer.read() << '\n';
}
```

Здесь `Holder<int>` берётся из основного шаблона, а `Holder<int*>` из частичной специализации для `T*`. Интерфейс `read()` одинаковый, но реализация различается. Компилятор подставляет конкретные типы на этапе инстанцирования, поэтому вызовы остаются типобезопасными без ручного `reinterpret_cast`.

Частичные специализации часто встречаются в метапрограммировании: `std::vector<bool>` (прокси-ссылки), трейты для ссылок и массивов, оптимизированные аллокаторы. Главное правило: специализация не расширяет API произвольно, а уточняет контракт для узкого класса типов.

## Подводные камни

- **Специализация функций.** В стандартном C++ частичная специализация функций запрещена; для них пишут отдельные перегрузки.
- **Порядок объявлений.** Специализация до первичного шаблона даёт ошибку компиляции или неожиданный выбор перегрузки.
- **Скрытие в .cpp.** Если определение специализации не видно в единице трансляции, линковка упадёт с undefined reference.
- **Разные контракты.** Слишком разное поведение в специализации ломает ожидания пользователя шаблона; лучше сохранять единый интерфейс.
- **SFINAE vs специализация.** Для ветвления по свойствам типа иногда проще `std::enable_if` или concepts, чем громоздкая матрица специализаций.
