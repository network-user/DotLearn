# Можно ли объявить виртуальный метод шаблоном?

## Суть

Отдельный виртуальный метод не может одновременно быть шаблоном функции: vtable фиксируется на этапе компиляции класса, а шаблон порождает семейство функций только при инстанцировании. Обходной путь: шаблонный класс с обычными виртуальными методами, параметризованными типом класса.

## Что ответить на собесе

- Отвечаю прямо: `virtual template` для метода в одном классе в стандартном C++ нельзя.
- Объясняю конфликт: полиморфизм требует стабильного слота в vtable, шаблон раскрывается позже и даёт разные сигнатуры.
- Показываю паттерн CRTP или шаблонный базовый класс `Base<T>` с `virtual void handle(const T&)`.
- Упоминаю type erasure (`std::function`, `std::any`) как альтернативу runtime-полиморфизму с разными типами.
- Честно добавляю: если нужен один интерфейс на много типов, чаще проектирую невиртуальный шаблонный API или variant, а не ломаю vtable.

## Как это работает

Виртуальный вызов идёт через таблицу указателей на функции, заполненную при компиляции иерархии. Шаблонный метод генерировался бы для каждого `T` отдельно, и компилятор не смог бы зарезервировать один полиморфный слот. Поэтому синтаксис `virtual template <typename U> void foo(U);` отвергается.

```cpp
#include <iostream>

template <typename Payload>
struct Sink {
    virtual ~Sink() = default;
    virtual void accept(const Payload& item) = 0;
};

template <typename Payload>
struct ConsoleSink : Sink<Payload> {
    void accept(const Payload& item) override {
        std::cout << "got: " << item << '\n';
    }
};

int main() {
    ConsoleSink<int> int_sink;
    ConsoleSink<double> double_sink;

    int_sink.accept(7);
    double_sink.accept(3.14);
}
```

Здесь виртуальность есть, но она действует внутри конкретного `Sink<int>` или `Sink<double>`. Это разные классы с разными vtable, а не один полиморфный базовый тип для произвольного `Payload`. Вызов через `Sink<int>*` остаётся полиморфным среди наследников `Sink<int>`, но не между `Sink<int>` и `Sink<double>`.

Для единого runtime-интерфейса над разными типами применяют type erasure: виртуальный базовый класс хранит type-erased payload и делегирует в шаблонную реализацию внутри .cpp. Так устроены многие фасады в стандартной библиотеке.

## Подводные камни

- **Попытка virtual template.** Компилятор отклоняет; не стоит искать обход через макросы.
- **Ложный полиморфизм между инстанцами.** `Sink<int>` и `Sink<double>` не связаны для `dynamic_cast` через общий базовый тип.
- **Шаблонный конструктор не виртуален.** Путаница с «шаблонным классом» и «шаблонным методом» на собесе частая.
- **Размер vtable.** Много виртуальных методов в шаблонном иерархии дублируется на каждый инстанцированный класс.
- **Type erasure цена.** Аллокации и косвенные вызовы; шаблонный статический путь быстрее, но менее гибкий.
