# Why does C++ need templates?

## In short

Templates parameterize functions and classes by types and values so you write one generic implementation instead of copies per type. Type checking and code generation happen at compile time without giving up performance versus hand-written duplicates.

## How to answer in an interview

- I define them as compile-time parameterization with argument substitution on use.
- I list benefits: less duplication, static type safety, and room for metaprogramming.
- I show a function template and a class template together on one container example.
- I contrast with macros: templates get type checking, macros do not.
- I add honestly: template errors are often verbose; in hot APIs teams sometimes keep untyped C interfaces for ABI stability.

## How it works

Template parameters declare holes in a signature. Calling `sort(v)` makes the compiler deduce the vector element type and plug it into `sort<T>`. For class `Buffer<T>`, creating `Buffer<std::string>` emits a separate type whose methods work on `std::string`.

```cpp
#include <iostream>
#include <vector>

template <typename Elem>
class RingBuffer {
    std::vector<Elem> slots;
    std::size_t head = 0;

public:
    explicit RingBuffer(std::size_t cap) : slots(cap) {}

    void push(const Elem& item) {
        slots[head] = item;
        head = (head + 1) % slots.size();
    }

    const Elem& peek() const { return slots[head]; }
};

template <typename Elem>
Elem pick_larger(const Elem& lhs, const Elem& rhs) {
    return (lhs > rhs) ? lhs : rhs;
}

int main() {
    RingBuffer<int> ints(4);
    ints.push(10);
    std::cout << pick_larger(3, 7) << ' ' << ints.peek() << '\n';
}
```

One `pick_larger` implementation serves any type with `>`. `RingBuffer` reuses ring logic for `int`, `double`, or user structs without changing source. The standard library follows the same model: `std::vector`, algorithms, smart pointers.

Templates also support non-type parameters (`template <int N>`, `template <typename T, std::size_t N>`), which underpin `std::array` and many optimizations.

## Pitfalls

- **Error diagnostics.** Messages point inside the template, not the call site.
- **Binary size.** Each instantiation adds code.
- **Header model.** Template definitions usually live in headers or linking breaks.
- **ABI stability.** Templates in public APIs change layout when arguments change; DLL borders often avoid them.
- **Mixing with virtual dispatch.** Templates give static polymorphism, vtables give dynamic; the choice shapes design.
