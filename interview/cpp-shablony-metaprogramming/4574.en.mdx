# Where should template definitions live in a project?

## In short

Template definitions usually belong in headers because the compiler must see the full body in every translation unit where instantiation happens. Splitting a declaration in `.h` and an implementation in `.cpp` breaks the compilation model for templates.

## How to answer in an interview

- I explain that a template is compiled at the point of use, not abstractly on the first `#include`.
- I show the pattern: declaration and definition in one `.h` included via `#include`.
- I mention alternatives to hide code: explicit instantiation in `.cpp` or C++20 modules.
- I note include guards or `#pragma once`; otherwise multiple inclusion causes ODR issues.
- I add honestly: for small inline templates I keep everything in the header; for heavy libraries I consider modules or explicit instantiation.

## How it works

Unlike ordinary functions, the compiler needs the template body when `func<T>(...)` is called so it can generate a concrete version. The classic header plus `.cpp` split leads to link errors: no translation unit generated the symbol.

```cpp
#pragma once

template <typename T>
T pick_greater(T lhs, T rhs) {
    return (lhs > rhs) ? lhs : rhs;
}
```

```cpp
#include <iostream>
#include "pick_greater.h"

int main() {
    std::cout << pick_greater(3, 9) << '\n';
    std::cout << pick_greater(2.5, 1.1) << '\n';
}
```

When `main.cpp` compiles, the compiler sees the definition from the header and creates `pick_greater<int>` and `pick_greater<double>`. If the implementation lived only in `pick_greater.cpp`, `main.cpp` could not instantiate the template.

Large libraries use explicit instantiation: a `.cpp` file contains `template int pick_greater<int>(int, int);` with the definition in the same file. That speeds builds but fixes the set of supported types. C++20 modules export templates without classic textual includes, though tooling remains mixed.

## Pitfalls

- **Template only in .cpp.** The classic undefined reference at link time.
- **Circular includes.** Heavy template headers inflate compile time; move non-type logic into regular functions.
- **Explicit instantiation drift.** The explicit list goes stale when clients start using new types.
- **ODR violations.** Different definitions of the same template in different headers cause subtle bugs.
- **Mixing with inline.** `inline` functions can be split across TUs; templates cannot without explicit instantiation.
