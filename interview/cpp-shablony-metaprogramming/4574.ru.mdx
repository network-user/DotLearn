# Где в проекте размещают определения шаблонов?

## Суть

Определения шаблонов обычно кладут в заголовки, потому что компилятор должен видеть полное тело в каждой единице трансляции, где происходит инстанцирование. Разнесение объявления в `.h` и реализации в `.cpp` для шаблонов ломает модель компиляции.

## Что ответить на собесе

- Объясняю, что шаблон компилируется при использовании, а не при первом `#include` абстрактно.
- Показываю паттерн: объявление и определение в одном `.h`, подключаемом через `#include`.
- Упоминаю альтернативы для сокрытия кода: explicit instantiation в `.cpp` или модуль C++20.
- Отмечаю include guards или `#pragma once`, иначе ODR-проблемы при множественном включении.
- Честно добавляю: для небольших inline-шаблонов держу всё в заголовке; для тяжёлых библиотек рассматриваю modules или явное инстанцирование.

## Как это работает

В отличие от обычных функций, тело шаблона нужно компилятору в момент вызова `func<T>(...)`, чтобы сгенерировать конкретную версию. Поэтому практика «заголовок + .cpp» для шаблонов приводит к ошибкам линковки: символ не был сгенерирован ни в одной TU.

```cpp
#pragma once

template <typename T>
T pick_greater(T lhs, T rhs) {
    return (lhs > rhs) ? lhs : rhs;
}
```

```cpp
#include <iostream>
#include "pick_greater.h"

int main() {
    std::cout << pick_greater(3, 9) << '\n';
    std::cout << pick_greater(2.5, 1.1) << '\n';
}
```

При компиляции `main.cpp` компилятор видит определение из заголовка и создаёт `pick_greater<int>` и `pick_greater<double>`. Если бы реализация лежала только в `pick_greater.cpp`, `main.cpp` не смог бы инстанцировать шаблон.

Для крупных библиотек применяют явное инстанцирование: в `.cpp` пишут `template int pick_greater<int>(int, int);` и оставляют определение там же. Это ускоряет сборку, но фиксирует набор поддерживаемых типов. C++20 modules позволяют экспортировать шаблоны без классического textual include, однако экосистема до сих пор смешанная.

## Подводные камни

- **Шаблон только в .cpp.** Классическая ошибка: undefined reference при линковке.
- **Циклические include.** Тяжёлые шаблонные заголовки раздувают время компиляции; выносите нетиповую логику в обычные функции.
- **Explicit instantiation drift.** Список явных инстанцирований устаревает, когда клиенты начинают использовать новые типы.
- **ODR нарушения.** Разные определения одного шаблона в разных заголовках дают тихие баги.
- **Смешение с inline.** `inline` функции можно разделять по TU, шаблоны без explicit instantiation нельзя.
