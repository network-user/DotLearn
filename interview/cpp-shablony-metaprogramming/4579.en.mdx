# What happens when a template is instantiated?

## In short

Template instantiation is when the compiler substitutes concrete types or values into a template and emits a real function or class. Until a use with specific arguments, the template is only a blueprint with no separate binary entity.

## How to answer in an interview

- I define the term: implicit instantiation on use and explicit via `template void foo<int>();` or `template class vector<int>;`.
- I explain each unique argument set yields its own specialization in the object file.
- I mention ODR: the template definition must be visible at the instantiation point (hence headers).
- I give one template function used with two types producing different machine code.
- I add honestly: heavy instantiation bloats binaries; sometimes implementation moves to a `.cpp` with an explicit type list.

## How it works

A function or class template describes a family of entities. Writing `clamp(3, 7, 5)` with template `clamp<T>` makes the compiler deduce `T = int` and create `clamp<int>`. A `double` call spawns `clamp<double>`. That is compile-time generation, not manual copy-paste.

```cpp
#include <iostream>

template <typename T>
T clamp_value(T lo, T hi, T val) {
    if (val < lo) return lo;
    if (val > hi) return hi;
    return val;
}

int main() {
    auto a = clamp_value(0, 10, 15);
    auto b = clamp_value(0.0, 1.0, 0.75);
    std::cout << a << ' ' << b << '\n';
}
```

The `int` line and the `double` line trigger two instantiations. In assembly those are two functions from one source. For classes, `std::vector<int>` and `std::vector<double>` are independent types with different sizes and methods.

Explicit instantiation speeds builds: one `.cpp` instantiates `Foo<int>` and other translation units link the symbol. Without it, each TU might generate a copy and the linker keeps one under ODR.

## Pitfalls

- **Template in .cpp without explicit.** Other translation units get undefined reference errors.
- **Code bloat.** Many types through a hot template multiply executable size.
- **Dependent errors.** Failures appear only for a specific `T`, not when parsing the template alone.
- **Weak symbol linking.** Duplicate implicit instantiations usually collapse, but explicit instantiation controls this.
- **Template export (C++20 modules).** Visibility rules change, but type substitution remains the same idea.
