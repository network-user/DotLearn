# Why is the typename keyword needed in templates?

## In short

`typename` tells the compiler that a dependent name inside a template refers to a type, not a value or function. Without that marker, the parser cannot disambiguate constructs such as `T::value_type` before the template is instantiated.

## How to answer in an interview

- I say this is about dependent names: identifiers whose meaning is known only after template arguments are substituted.
- I show the classic case: `typedef typename T::value_type Item;` inside a container wrapper.
- I mention the second context: `typename` in a template parameter list (`template <typename U>`).
- I stress that non-dependent names (for example `int` or `std::string`) do not need `typename`.
- I add honestly: C++20 concepts and `requires` simplify many cases, but legacy code still requires `typename`.

## How it works

Before instantiation, the compiler does not know whether `T::something` is a nested type, a static member, or a function. The `typename` keyword removes that ambiguity during parsing and allows the name to be used as a type.

```cpp
template <typename Seq>
class FrontView {
public:
    using Item = typename Seq::value_type;

    explicit FrontView(Seq& seq) : seq_(seq) {}

    Item peek() const {
        return seq_.front();
    }

private:
    Seq& seq_;
};
```

In `using Item = typename Seq::value_type`, the compiler treats `Seq::value_type` as a type. Removing `typename` often yields a parse error ("expected a type"). The same rule applies to return types and function parameters that involve dependent names.

Two-phase lookup splits names into dependent and non-dependent groups. Non-dependent names are checked on the first pass; dependent names are deferred until instantiation. `typename` applies to the second group. C++11 `using` and `auto` do not remove the need to mark dependent types.

## Pitfalls

- **Missing typename.** The most common beginner mistake in template adapters over STL-like interfaces.
- **Confusion with template.** Dependent templates need both keywords, for example `T::template nest<U>`.
- **Value vs type.** `T::count` may be a static member; `typename` there is wrong and produces invalid code.
- **typedef in legacy code.** Older projects still write `typedef typename ...` instead of `using`; semantics match, style differs.
- **Concepts are not universal.** Even with concepts, dependent-name checks inside template bodies remain; do not treat these keywords as obsolete.
