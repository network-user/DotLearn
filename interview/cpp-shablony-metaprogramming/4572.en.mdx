# How do you demonstrate partial template specialization?

## In short

Partial specialization lets you supply a separate template implementation for a subset of parameters without fixing the entire parameter pack. It is useful when pointers, arrays, or specific type combinations need different semantics than the primary template.

## How to answer in an interview

- I explain that full specialization fixes every parameter, while partial specialization leaves at least one parameter open.
- I walk through `Holder<T>` versus `Holder<T*>`, where the pointer version hides dereferencing inside `read()`.
- I note that partial specialization applies to class templates, not function templates (overloads cover that case).
- I mention that the specialization must appear after the primary template and be visible at the instantiation point.
- I add honestly: for functions with type-dependent behavior I usually prefer overloads or `if constexpr` rather than specialization.

## How it works

The primary template defines default behavior. When the compiler sees arguments that match a partial specialization, it picks that version instead of the general one. For pointers, a common pattern is to store an address but expose the pointee value outward.

```cpp
#include <iostream>

template <typename T>
class Holder {
public:
    explicit Holder(T stored) : stored_(stored) {}

    T read() const { return stored_; }

private:
    T stored_;
};

template <typename T>
class Holder<T*> {
public:
    explicit Holder(T* ptr) : ptr_(ptr) {}

    T read() const { return *ptr_; }

private:
    T* ptr_;
};

int main() {
    Holder<int> by_value{42};
    int payload = 100;
    Holder<int*> by_pointer{&payload};

    std::cout << by_value.read() << '\n';
    std::cout << by_pointer.read() << '\n';
}
```

Here `Holder<int>` comes from the primary template and `Holder<int*>` from the partial specialization for `T*`. The `read()` interface stays the same while the implementation differs. The compiler substitutes concrete types during instantiation, so calls remain type-safe without manual `reinterpret_cast`.

Partial specializations show up often in metaprogramming: `std::vector<bool>` (proxy references), traits for references and arrays, tuned allocators. The key rule: a specialization should refine the contract for a narrow type family, not invent an unrelated API.

## Pitfalls

- **Function specialization.** Standard C++ forbids partial specialization of function templates; use separate overloads instead.
- **Declaration order.** A specialization before the primary template triggers compile errors or surprising overload resolution.
- **Hidden in .cpp.** If the specialization definition is not visible in the translation unit, linking fails with undefined reference.
- **Divergent contracts.** Behavior that diverges too much breaks caller expectations; keep a consistent interface.
- **SFINAE vs specialization.** For branching on type properties, `std::enable_if` or concepts are sometimes simpler than a large specialization matrix.
