# What does the compiler generate from a template function for different types?

## In short

For each unique argument set, the template is instantiated into a separate non-template function with substituted types. One piece of source logic becomes several specialized machine-level implementations at compile time.

## How to answer in an interview

- I say the template itself does not exist in the binary; only concrete instantiations do.
- I use `pick_larger<T>` with `int` and `double` calls and explain that two functions are produced.
- I mention implicit argument deduction: `pick_larger(1, 2)` creates an `int` version without explicit `<int>`.
- I note the compiler may inline and optimize each version independently.
- I add honestly: excessive instantiation bloats the binary; sometimes I move shared logic into a non-template function.

## How it works

A template function is a blueprint. On the first call with concrete types, the compiler substitutes them for `T` and emits an ordinary function. A later call with the same type set reuses that version.

```cpp
template <typename T>
T pick_larger(T lhs, T rhs) {
    return (lhs > rhs) ? lhs : rhs;
}

int main() {
    int ai = 5, bi = 10;
    double ad = 5.5, bd = 10.5;

    int max_i = pick_larger(ai, bi);
    double max_d = pick_larger(ad, bd);
    (void)max_i;
    (void)max_d;
}
```

After compilation, the logical result is two functions:

```cpp
int pick_larger(int lhs, int rhs) {
    return (lhs > rhs) ? lhs : rhs;
}

double pick_larger(double lhs, double rhs) {
    return (lhs > rhs) ? lhs : rhs;
}
```

Symbol names in the object file may be mangled, but the semantics match. Calling `pick_larger(1, 2.0)` without conversion fails because the compiler cannot deduce a single `T`. That is the static type safety of templates.

The same model powers `std::sort`, `std::max`, and STL containers: one header declaration, many instantiations in client code. The linker may drop unused instantiations, but large projects should watch duplication via explicit instantiation or shared non-type cores.

## Pitfalls

- **Code bloat.** Dozens of types produce dozens of copies of the same function in the binary.
- **Deep instantiation errors.** Diagnostics point at the template line, not the call site; read the full instantiation stack.
- **Mixed types in one call.** `T` must be unified or argument deduction fails.
- **Hidden instantiations.** Friend functions and taking the address `&pick_larger<int>` can pull in unexpected versions.
- **ODR and inline.** Identical instantiations across TUs usually merge, but divergent bodies cause UB.
