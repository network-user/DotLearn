# Как выполнить запрос через Swagger UI?

## Суть

Swagger UI читает OpenAPI spec и рисует форму для каждого path/method. QA выбирает операцию, заполняет path/query/header/body, нажимает Execute; браузер отправляет реальный HTTP запрос на server URL из spec и показывает status, headers и response body. Это быстрый smoke без Postman, но без сохранения истории и сложных цепочек.

## Что ответить на собесе

- Открываю `/swagger-ui/index.html` или dev portal URL, проверяю server dropdown (dev/stage).
- Раскрываю нужный tag, например Orders, метод GET /v1/orders/{id}.
- Заполняю path param, query, при необходимости Authorize (Bearer).
- Execute → смотрю curl snippet, response code и schema validation errors если включён validator.
- Честно: Try it out на prod с POST/DELETE рискован; ограничиваю read-only методы или mock stand.

## Как это работает

```gherkin
Feature: Execute API call in Swagger UI

  Scenario: Smoke GET profile after Authorize
    Given Swagger UI for staging API with OAuth2 password flow configured
    When tester clicks Authorize and pastes bearer token
    And expands GET /users/me and sets no extra params
    And clicks Execute
    Then response code 200 and body matches User schema example

  Scenario: Required field validation before send
    Given POST /orders requires body field currency
    When tester leaves currency empty and Execute
    Then UI highlights required field and request not sent

  Scenario: Copy curl for bug attachment
    Given failed POST /refunds returns 409
    When tester copies generated curl from Swagger
    Then curl attached to Jira reproduces same 409 for developer
```

Логическая последовательность UI:

```
1. Select server https://api-stage.shop.example
2. Open POST /v2/cart/items
3. Request body JSON { "sku": "WB-991", "qty": 2 }
4. Execute
5. Response 201 + Location header + example CartItem schema
```

Swagger UI генерирует fetch/XHR из spec; параметры `in: path|query|header|cookie` мапятся в форму. Responses section показывает documented codes; фактический body может отличаться от example, QA фиксирует drift.

## Подводные камни

- **CORS блокирует browser call.** UI открыт с другого origin; используй same-origin gateway или Postman.
- **Auth только в UI session.** Токен не сохраняется в spec export; документируй отдельно.
- **Большой file upload через Try it out.** Браузер зависает; для multipart лучше curl/Postman.
- **Неверный server URL в spec.** Execute бьёт localhost у коллеги; проверяй env override.
- **Пример schema ≠ реальный ответ.** Не закрывай баг «works in Swagger» без assert на поля.
