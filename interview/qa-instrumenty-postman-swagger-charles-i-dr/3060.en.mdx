# Dev, stage, and prod environments: why does QA distinguish them?

## In short

**Dev** is unstable: fresh code, mock services, frequent deploys, for developers and early QA smoke. **Stage (staging/preprod)** mirrors prod config, integrations, and anonymized data; regression and sign-off happen here. **Prod** carries live traffic: controlled checks, synthetics, read-only probes; exploratory on real users is forbidden without policy.

## How to answer in an interview

- Dev: fast feature checks, broken builds acceptable, synthetic data.
- Stage: full regression, perf smoke, migration dry-run, release candidate tag.
- Prod: smoke after deploy, monitoring alerts, no destructive tests.
- Different URLs, tokens, feature flags; I keep separate Postman environments.
- Honestly: stage without parity (different SKU catalog) gives false confidence; document gaps.

## How it works

```gherkin
Feature: Dev stage prod for QA

  Scenario: Feature tested on dev then regressed on stage
    Given developer deploys branch to dev cluster
    When QA smoke passes on dev
    And same build promoted to stage with RC tag
    Then full regression runs before prod deploy approval

  Scenario: Prod smoke read-only
    Given prod deploy completed
    When synthetic monitor GET /health and /catalog?limit=1
    Then alerts fire if non-200 within 2 minutes
    And no POST test orders on prod accounts

  Scenario: Wrong environment bug
    Given tester used prod token in Postman collection
    When accidental DELETE on prod resource
    Then incident process triggered environment guardrails added
```

Comparison:

| | Dev | Stage | Prod |
|---|-----|-------|------|
| Data | fake/small | anonymized copy | real users |
| Deploy freq | many/day | per RC | controlled |
| QA scope | feature smoke | full regression | smoke/monitor |

QA artifacts (cases, data) tag target env; TMS Environment field is mandatory when filing bugs.

## Pitfalls

- **Stage without HTTPS parity.** Secure/SameSite cookies behave differently only on prod.
- **Shared staging one team.** Someone else's deploy wipes your test users.
- **Prod-like load on stage.** Shared DB melts; perf on isolated perf env.
- **Feature flag always on in stage.** Prod default off hides flag dependency bugs.
- **Copy prod to stage without masking.** GDPR incident; sanitized subset only.
