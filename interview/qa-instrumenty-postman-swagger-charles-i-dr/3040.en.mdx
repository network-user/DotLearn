# Postman vs Swagger: what is the difference and when to use each?

## In short

Postman is a client for sending HTTP requests, organizing collections, scripts, and CI runs. Swagger (the OpenAPI ecosystem) describes the API contract: schemas, endpoints, examples, and interactive docs. QA validates behavior in Postman; Swagger is the source of truth for fields, status codes, and auth.

## How to answer in an interview

- Postman: manual and semi-automated checks, request chains, pre/post-request scripts, Newman in the pipeline.
- Swagger UI: understand the contract quickly without reading code; Try it out for dev smoke.
- I import OpenAPI yaml/json into Postman so the collection stays aligned with the spec.
- Role split: Swagger documents and generates stubs; Postman executes and asserts.
- Honestly: Swagger UI does not replace negative cases and load tests; edge cases need dedicated collections and data.

## How it works

```gherkin
Feature: Postman vs Swagger in QA workflow

  Scenario: Contract-first smoke from OpenAPI
    Given openapi.yaml imported into Postman workspace
    When QA sends GET /v2/orders/{id} with valid bearer token
    Then response status 200 and body matches schema Order
    And Swagger UI shows same path definition for developers

  Scenario: Negative test not covered by docs
    Given Swagger documents 400 for invalid sku
    When Postman sends POST /cart with sku length 0
    Then API returns 422 with field errors array
    And QA files spec drift ticket if code differs from yaml

  Scenario: Collection run in CI
    Given Newman job with environment staging.json
    When nightly pipeline executes cart_regression collection
    Then JUnit report published and Slack notified on failure
```

OpenAPI fragment (renamed entities):

```yaml
openapi: 3.1.0
info:
  title: Inventory API
  version: 2.4.0
paths:
  /warehouses/{whId}/stock:
    get:
      summary: Fetch stock snapshot
      parameters:
        - name: whId
          in: path
          required: true
          schema:
            type: string
            format: uuid
      responses:
        "200":
          description: Stock rows
```

Postman request equivalent (logical form):

```
GET {{baseUrl}}/warehouses/{{warehouseUuid}}/stock
Authorization: Bearer {{accessToken}}
Tests: pm.response.to.have.status(200)
```

## Pitfalls

- **Stale spec.** Postman collection is green but prod fails: match OpenAPI version to deployment tag.
- **Try it out on prod.** Accidental writes via Swagger UI on live URL; read-only staging only.
- **Secrets in shared collection.** Tokens in variables without secret scope leak on export.
- **Import without cleanup.** Auto-import duplicates folders; agree on a naming convention.
- **Happy-path-only Swagger examples.** QA adds negative and auth failure scenarios in Postman.
