# What is Charles Proxy and what QA tasks is it used for?

## In short

Charles is a desktop HTTP/HTTPS proxy between client and server: capture, decrypt (with root cert), inspect headers/body, Rewrite, Breakpoints, Throttling, Map Local. QA uses it for API/mobile bug triage, security checks (leaked tokens), slow-network simulation, and response mocks without code changes.

## How to answer in an interview

- I set proxy 8888, SSL Proxying for staging domains, cert on device.
- Structure shows waterfall timing, size, status codes.
- Repeat/Edit and Resend for quick retry with a different header.
- Focus filter by host so analytics noise does not drown the signal.
- Honestly: Charles is a desktop license; for headless CI use mitmproxy or a mock server.

## How it works

```gherkin
Feature: Charles Proxy QA use cases

  Scenario: Verify API payload from mobile app
    Given iPhone Wi-Fi proxy to Charles on laptop
    When user submits order in app
    Then POST /orders body contains expected promoCode field
    And no card PAN in cleartext JSON

  Scenario: Slow 3G checkout UX
    Given Throttling preset 3G enabled
    When user pays on unstable network profile
    Then double-tap Pay does not create duplicate charge UI guard

  Scenario: Automate via requests library through Charles
    Given Python job sets proxies localhost:8888
    When integration script calls staging API
    Then Charles logs same traffic for debug alongside pytest output
```

Proxy in automation (renamed host):

```gherkin
Feature: Route automation through Charles

  Scenario: Capture traffic from script
    Given HTTP client proxy http://127.0.0.1:8888
    When script GET https://api.stage.example/health
    Then Charles records request for attachment to failure report
```

Charles vs browser DevTools: DevTools sees only the browser tab; Charles catches native apps, other browsers, curl with system proxy.

## Pitfalls

- **Forgotten proxy on laptop.** All desktop traffic goes through Charles; disable after session.
- **MITM legal/compliance.** Staging/test accounts only; do not intercept prod customer data without policy.
- **Huge session files.** Export and clear between cases.
- **HTTP/2 push noise.** Filter by path; tools decode differently.
- **Pinning blocks decrypt.** Needs debug build; otherwise false "client sends nothing" conclusion.
