# Что такое Charles Proxy и для каких QA-задач он нужен?

## Суть

Charles это desktop HTTP/HTTPS proxy между клиентом и сервером: перехват, decrypt (с root cert), просмотр headers/body, Rewrite, Breakpoints, Throttling, Map Local. QA использует для triage багов API/mobile, проверки security (leaked tokens), симуляции медленной сети и mock ответов без изменения кода.

## Что ответить на собесе

- Ставлю proxy 8888, SSL Proxying для staging domains, cert на device.
- Structure показывает waterfall timing, size, status codes.
- Repeat/Edit and Resend для быстрого retry с другим header.
- Focus filter по host, чтобы не тонуть в analytics noise.
- Честно: Charles desktop license; для CI headless чаще mitmproxy или mock server.

## Как это работает

```gherkin
Feature: Charles Proxy QA use cases

  Scenario: Verify API payload from mobile app
    Given iPhone Wi-Fi proxy to Charles on laptop
    When user submits order in app
    Then POST /orders body contains expected promoCode field
    And no card PAN in cleartext JSON

  Scenario: Slow 3G checkout UX
    Given Throttling preset 3G enabled
    When user pays on unstable network profile
    Then double-tap Pay does not create duplicate charge UI guard

  Scenario: Automate via requests library through Charles
    Given Python job sets proxies localhost:8888
    When integration script calls staging API
    Then Charles logs same traffic for debug alongside pytest output
```

Пример прокси в автотесте (переименованный host):

```gherkin
Feature: Route automation through Charles

  Scenario: Capture traffic from script
    Given HTTP client proxy http://127.0.0.1:8888
    When script GET https://api.stage.example/health
    Then Charles records request for attachment to failure report
```

Charles vs browser DevTools: DevTools видит только browser tab; Charles ловит native app, другие browsers, curl с system proxy.

## Подводные камни

- **Forgotten proxy on laptop.** Весь desktop traffic идёт через Charles; disable after session.
- **MITM legal/compliance.** Только staging/test accounts; не перехватывай prod customer data без policy.
- **Huge session files.** Export и clear между кейсами.
- **HTTP/2 push noise.** Фильтруй по path; не все tools decode одинаково.
- **Pinning blocks decrypt.** Нужен debug build; иначе ложный вывод «клиент не шлёт запрос».
