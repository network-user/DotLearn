# How do you execute a request in Swagger UI?

## In short

Swagger UI reads the OpenAPI spec and renders a form for each path/method. QA picks an operation, fills path/query/header/body, clicks Execute; the browser sends a real HTTP request to the server URL from the spec and shows status, headers, and response body. Fast smoke without Postman, but no saved history or complex chains.

## How to answer in an interview

- I open `/swagger-ui/index.html` or the dev portal URL and check the server dropdown (dev/stage).
- I expand the needed tag, e.g. Orders, method GET /v1/orders/{id}.
- I fill path params, query, and Authorize (Bearer) when required.
- Execute → I review the curl snippet, response code, and schema validation errors if a validator is enabled.
- Honestly: Try it out on prod with POST/DELETE is risky; I limit to read-only methods or a mock stand.

## How it works

```gherkin
Feature: Execute API call in Swagger UI

  Scenario: Smoke GET profile after Authorize
    Given Swagger UI for staging API with OAuth2 password flow configured
    When tester clicks Authorize and pastes bearer token
    And expands GET /users/me and sets no extra params
    And clicks Execute
    Then response code 200 and body matches User schema example

  Scenario: Required field validation before send
    Given POST /orders requires body field currency
    When tester leaves currency empty and Execute
    Then UI highlights required field and request not sent

  Scenario: Copy curl for bug attachment
    Given failed POST /refunds returns 409
    When tester copies generated curl from Swagger
    Then curl attached to Jira reproduces same 409 for developer
```

Logical UI sequence:

```
1. Select server https://api-stage.shop.example
2. Open POST /v2/cart/items
3. Request body JSON { "sku": "WB-991", "qty": 2 }
4. Execute
5. Response 201 + Location header + example CartItem schema
```

Swagger UI builds fetch/XHR from the spec; parameters `in: path|query|header|cookie` map to the form. The Responses section shows documented codes; the actual body may differ from the example, QA records drift.

## Pitfalls

- **CORS blocks browser call.** UI opened from another origin; use same-origin gateway or Postman.
- **Auth only in UI session.** Token not saved in spec export; document separately.
- **Large file upload via Try it out.** Browser hangs; use curl/Postman for multipart.
- **Wrong server URL in spec.** Execute hits localhost for teammates; verify env override.
- **Example schema ≠ real response.** Do not close a bug as "works in Swagger" without field asserts.
