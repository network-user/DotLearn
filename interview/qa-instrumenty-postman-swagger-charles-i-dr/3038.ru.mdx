# Зачем QA-инженеру Git и как им пользоваться?

## Суть

Git это распределённая система контроля версий: фиксирует снимки файлов, ведёт историю и позволяет параллельно работать в ветках. Для QA это способ хранить автотесты и тестовую документацию рядом с кодом продукта, видеть diff перед релизом и синхронизироваться с разработчиками без «ручной» рассылки файлов.

## Что ответить на собесе

- Клонирую репозиторий, создаю feature-ветку под правки в тестах или чек-листах.
- Коммичу атомарно: один логический шаг, понятное сообщение (`fix: stabilize login e2e on staging`).
- Перед merge смотрю diff и CI: какие файлы затронуты, не сломал ли я чужие тесты.
- Pull request проходит ревью так же, как у разработчиков: артефакты, описание, ссылка на баг.
- Честно: Git не заменяет TMS; для статусов багов и регрессионных матриц нужен Jira/YouTrack, а репозиторий держит исполняемые артефакты.

## Как это работает

```gherkin
Feature: QA workflow with Git

  Scenario: Update automated test for new checkout flow
    Given cloned repo qa-automation on branch main
    When tester creates branch fix/checkout-v2-tests
    And edits specs/checkout.spec.ts
    And commits with message "test: cover promo code validation"
    And pushes branch and opens pull request
    Then CI runs smoke suite on pull request
    And reviewer merges after green pipeline

  Scenario: Sync with developer hotfix
    Given developer merged hotfix into main
    When QA pulls latest main into local branch
    And reruns affected regression pack
    Then failing test pinpoints regression before release tag
```

Типовой цикл команд (переименованный репозиторий):

```
git clone git@gitlab.corp.example:shop/qa-suite.git
cd qa-suite
git checkout -b chore/refresh-api-fixtures
git add tests/api/orders/
git commit -m "chore: refresh order fixtures for v3 schema"
git push -u origin chore/refresh-api-fixtures
```

Ключевые сущности: **репозиторий** хранит историю; **commit** фиксирует снимок; **branch** изолирует работу; **merge/rebase** вливает изменения. QA чаще всего работает с удалённым origin (GitHub/GitLab/Bitbucket), но локальная история остаётся полной даже офлайн.

## Подводные камни

- **Коммит секретов.** Токены и `.env` в истории не удаляются «простым revert»; нужен rotate ключей и git filter.
- **Гигантские бинарники в репо.** Видео скринкасты лучше в артефактный storage, в Git только ссылка.
- **Merge без pull.** Конфликты в общих фикстурах ломают ночной прогон всей команды.
- **Force push в shared branch.** Переписывает историю коллег; для QA допустим только в личной ветке до merge.
- **Игнор `.gitignore` для отчётов.** Allure/HTML отчёты в PR засоряют diff; генерируй локально или в CI artifact.
