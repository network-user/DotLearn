# What is the difference between a release and a deploy?

## In short

**Deploy** is technical delivery of an artifact to an environment: image to k8s, static to CDN, migration job. **Release** is a product event: version available to users, with notes, support readiness, monitoring, and rollback plan. One release may include several deploys (canary → full); deploy to staging without user release is routine.

## How to answer in an interview

- Deploy: CI pipeline, helm upgrade, smoke gate, healthcheck.
- Release: go/no-go, changelog, feature flags flip, comms to support.
- QA sign-off on stage often blocks prod deploy but is not public release if dark launch.
- Rollback deploy is fast (previous tag); release rollback may need comms and data fix.
- Honestly: startups conflate terms; I clarify "deploy to prod" vs "release to 100% users."

## How it works

```gherkin
Feature: Release vs deploy

  Scenario: Deploy staging without user release
    Given merge to main triggers deploy staging
    When QA runs regression on staging URL
    Then bugs found before any prod deploy scheduled

  Scenario: Canary deploy as part of release
    Given release 4.3.0 approved
    When deploy sends 5 percent traffic to new pods
    And error budget green for 2 hours
    Then release promotes to 100 percent users

  Scenario: Hotfix deploy not full release
    Given critical patch 4.2.1 deployed prod
    When marketing release notes still show 4.2.0
    Then users receive fix without major release announcement
```

Deploy steps (logical form):

```
git tag v4.3.0-rc2
ci build image → push registry
kubectl rollout staging/api:v4.3.0-rc2
smoke + QA gate
kubectl rollout production/api:v4.3.0-rc2 (canary)
feature flag checkout_v3 enabled for 10%
monitor SLO → full flag → release notes published
```

QA release checklist: regression done, migrations verified, rollback tested on stage, monitoring dashboards ready, support FAQ updated.

## Pitfalls

- **Prod deploy treated as release.** Flag off, users see nothing; bugs internal only.
- **Release notes without QA review.** Promised feature does not match stage.
- **Irreversible DB migration.** Deploy rollback without data plan breaks prod.
- **Friday prod deploy.** Release without on-call owner.
- **Mobile version skew.** Backend released, app store review lag; contract compatibility tests.
