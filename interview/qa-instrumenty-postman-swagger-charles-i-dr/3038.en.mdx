# Why does a QA engineer need Git and how to use it?

## In short

Git is a distributed version control system: it snapshots files, keeps history, and enables parallel work in branches. For QA it means storing automated tests and test docs next to product code, reviewing diffs before release, and staying in sync with developers without emailing files around.

## How to answer in an interview

- I clone the repo and create a feature branch for test or checklist changes.
- I commit in small logical steps with clear messages (`fix: stabilize login e2e on staging`).
- Before merge I review diff and CI: which files changed, whether I broke someone else's tests.
- Pull requests get review like dev code: artifacts, description, link to the bug.
- Honestly: Git does not replace a TMS; Jira/YouTrack still owns bug status and regression matrices, while the repo holds executable artifacts.

## How it works

```gherkin
Feature: QA workflow with Git

  Scenario: Update automated test for new checkout flow
    Given cloned repo qa-automation on branch main
    When tester creates branch fix/checkout-v2-tests
    And edits specs/checkout.spec.ts
    And commits with message "test: cover promo code validation"
    And pushes branch and opens pull request
    Then CI runs smoke suite on pull request
    And reviewer merges after green pipeline

  Scenario: Sync with developer hotfix
    Given developer merged hotfix into main
    When QA pulls latest main into local branch
    And reruns affected regression pack
    Then failing test pinpoints regression before release tag
```

Typical command cycle (renamed repository):

```
git clone git@gitlab.corp.example:shop/qa-suite.git
cd qa-suite
git checkout -b chore/refresh-api-fixtures
git add tests/api/orders/
git commit -m "chore: refresh order fixtures for v3 schema"
git push -u origin chore/refresh-api-fixtures
```

Core entities: a **repository** stores history; a **commit** captures a snapshot; a **branch** isolates work; **merge/rebase** integrates changes. QA usually works against remote origin (GitHub/GitLab/Bitbucket), but full history stays available locally even offline.

## Pitfalls

- **Committing secrets.** Tokens and `.env` in history need key rotation and git filter, not a simple revert.
- **Huge binaries in the repo.** Screen capture videos belong in artifact storage; Git should hold only a link.
- **Merge without pull.** Conflicts in shared fixtures break the team's nightly run.
- **Force push to a shared branch.** Rewrites teammates' history; acceptable for QA only on a personal branch before merge.
- **Ignoring `.gitignore` for reports.** Allure/HTML reports in PRs clutter diffs; generate locally or as CI artifacts.
