# Чем релиз отличается от деплоя?

## Суть

**Deploy (деплой)** техническая доставка артефакта в среду: образ в k8s, static на CDN, migration job. **Release (релиз)** продуктовое событие: версия доступна пользователям, с notes, support readiness, мониторингом и rollback plan. Один релиз может включать несколько деплоев (canary → full); деплой на staging без релиза пользователям это routine.

## Что ответить на собесе

- Deploy: CI pipeline, helm upgrade, smoke gate, healthcheck.
- Release: go/no-go, changelog, feature flags flip, comms to support.
- QA sign-off на stage часто блокирует prod deploy, но не равен public release если dark launch.
- Rollback deploy быстрый (previous tag); rollback release может включать comms и data fix.
- Честно: в стартапах слова путают; уточняю «deploy to prod» vs «release to 100% users».

## Как это работает

```gherkin
Feature: Release vs deploy

  Scenario: Deploy staging without user release
    Given merge to main triggers deploy staging
    When QA runs regression on staging URL
    Then bugs found before any prod deploy scheduled

  Scenario: Canary deploy as part of release
    Given release 4.3.0 approved
    When deploy sends 5 percent traffic to new pods
    And error budget green for 2 hours
    Then release promotes to 100 percent users

  Scenario: Hotfix deploy not full release
    Given critical patch 4.2.1 deployed prod
    When marketing release notes still show 4.2.0
    Then users receive fix without major release announcement
```

Deploy steps (логическая форма):

```
git tag v4.3.0-rc2
ci build image → push registry
kubectl rollout staging/api:v4.3.0-rc2
smoke + QA gate
kubectl rollout production/api:v4.3.0-rc2 (canary)
feature flag checkout_v3 enabled for 10%
monitor SLO → full flag → release notes published
```

QA чеклист релиза: regression done, migrations verified, rollback tested on stage, monitoring dashboards ready, support FAQ updated.

## Подводные камни

- **Deploy prod = release mentally.** Flag off, users не видят feature; баги только internal.
- **Release notes без QA review.** Обещанный функционал не совпадает со stage.
- **DB migration irreversible.** Deploy rollback без plan данных ломает prod.
- **Friday prod deploy.** Release без дежурного on-call.
- **Version skew mobile.** Backend released, app store review lag; contract compatibility tests.
