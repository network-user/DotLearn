# When to pick REST versus gRPC?

## In short

REST builds on HTTP/1.1, resources, and text formats like JSON; gRPC runs on HTTP/2, uses binary Protocol Buffers, and generates strict contracts from `.proto`. REST is easier to debug with curl and integrate with browsers; gRPC wins on performance, type safety, and streaming.

## How to answer in an interview

- Compare transport: HTTP/1.1 vs HTTP/2 with multiplexing and lower per-connection overhead.
- Stress format: human-readable JSON/XML vs compact protobuf.
- Mention codegen: gRPC needs protoc and schema versioning; REST often lives on OpenAPI.
- Name streaming: gRPC supports server/client/bidi streams out of the box; REST usually needs SSE or WebSocket separately.
- Be honest: for public B2B APIs and mobile clients without grpc libraries, REST stays simpler; inside a data center between Java services, gRPC is often the better fit.

## How it works

REST thinks in resources and HTTP verbs:

```http
GET /api/v1/customers/42 HTTP/1.1
Host: shop.example
Accept: application/json
```

The response serializes to JSON; teams describe the contract with OpenAPI or conventions.

gRPC declares the service:

```proto
syntax = "proto3";

service CustomerDirectory {
  rpc FetchCustomer (CustomerQuery) returns (CustomerView);
}

message CustomerQuery {
  int64 customer_id = 1;
}

message CustomerView {
  string full_name = 1;
  string email = 2;
}
```

Client and server generate stubs; the call looks like a normal RPC:

```java
CustomerView view = stub.fetchCustomer(
    CustomerQuery.newBuilder().setCustomerId(42L).build()
);
```

HTTP/2 multiplexes several RPCs on one TCP connection; protobuf shrinks payloads. For real-time streams (logs, quotes) gRPC opens a long-lived stream without polling.

## Pitfalls

- **REST without versioning.** Breaking JSON field changes hit every client; you need `/v2` or tolerant readers.
- **gRPC through public ingress.** Not every LB and WAF handles HTTP/2 and gRPC equally; you may need grpc-web or a sidecar.
- **Debugging protobuf.** Binary traffic is opaque in DevTools; without grpcurl or reflection, diagnosis is harder.
- **Deadlines and retries.** gRPC needs explicit timeouts and idempotency; blind retries on non-idempotent RPCs are risky.
- **Dual contracts.** Maintaining the same domain model in both OpenAPI and proto without codegen leads to drift.
