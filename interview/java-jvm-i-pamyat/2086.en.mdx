# Heap generations and object promotion

## In short

Generational heap splits object memory into Young (Eden + two Survivors) and Old (Tenured or G1 regions). New objects allocate in Eden; minor GC copies survivors to S0/S1 with incremented age; at threshold (`-XX:MaxTenuringThreshold`) objects promote to Old. Metaspace separately holds class metadata, not object heap.

## How to answer in an interview

- Weak generational hypothesis: most objects die young.
- Minor GC: STW or mostly STW evacuate Eden + one survivor.
- Survivor spaces swap From/To roles each cycle.
- Promotion failure → premature Full GC pressure.
- Honestly: G1 uses regions, not classic fixed Eden, but young/old logic is the same.

## How it works

```java
public final class PromotionDemo {

    public static void main(String[] args) {
        var longLived = new byte[512 * 1024];
        for (int i = 0; i < 20_000; i++) {
            var temp = new byte[1024];
            if (i % 5000 == 0) {
                longLived = longLived;
            }
        }
    }
}
```

Lifecycle:

1. `temp` arrays allocate in Eden; die young → collected in minor GC cheaply.
2. `longLived` survives repeated minor GCs; age counter increments each survivor copy.
3. Age ≥ threshold → moved to Old gen; minor GC rarely touches it.
4. Old fill triggers major/mixed collection.

Classic layout:

```
Young:  [ Eden | Survivor0 | Survivor1 ]
Old:    [ Tenured objects................ ]
```

G1 equivalent: Eden regions + survivor regions + old regions; `-XX:G1NewSizePercent` influences young footprint.

## Pitfalls

- **Huge arrays.** Go direct to Old/humongous region, bypassing generational benefit.
- **MaxTenuringThreshold=15 default.** Premature promotion if survivor too small.
- **Survivor overflow.** Early promotion → Old pressure spike.
- **Confusing PermGen answer.** Metaspace is not part of object heap generations.
- **Tuning only -Xmx.** Without GC logs you cannot see promotion rate problems.
