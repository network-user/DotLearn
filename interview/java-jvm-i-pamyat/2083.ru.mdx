# Виды Garbage Collector в HotSpot

## Суть

HotSpot предлагает несколько interchangeable GC implementations, выбираемых JVM flags. Все используют generational или region heap layout и mark reachability от roots, но отличаются parallelization и concurrent phases. JDK 21 default G1; ZGC и Shenandoah для ultra-low pause; Parallel для batch throughput.

## Что ответить на собесе

- Young gen collectors: Serial, Parallel (Stop-the-world copy/evacuate).
- Old gen: Parallel Old, G1 mixed, ZGC/Shenandoah concurrent relocate.
- CMS legacy, removed; не путать с G1 concurrent mark.
- Generational hypothesis drives young frequent / old rare strategy.
- Честно: один «лучший» GC нет; метрики P99 latency + allocation rate решают.

## Как это работает

Generational layout (classic):

```
┌──────── Young ────────┬──────── Old ────────┐
│ Eden │ S0 │ S1        │ Tenured regions    │
└───────────────────────┴─────────────────────┘
```

Minor GC (Parallel example):

1. STW: mark live in Eden+From survivor.
2. Copy survivors to To survivor or promote Old.
3. Clear Eden.

G1 divides heap in equal regions (~1-32MB); tracks garbage per region, collects highest garbage-first in mixed cycles.

Enable examples:

```bash
# Throughput
java -XX:+UseParallelGC -jar etl.jar

# Balanced (default many JDKs)
java -XX:+UseG1GC -jar api.jar

# Low latency
java -XX:+UseZGC -jar realtime.jar
```

JVM prints active collector at startup with `-Xlog:gc*` (Java 9+ unified logging).

## Подводные камни

- **Conflicting flags.** Multiple `-XX:+Use*GC` → last wins or JVM error.
- **Large heap + Parallel.** Full GC pause scales with live set size.
- **Ignoring GC cause logs.** Allocation failure vs metadata GC need different fixes.
- **Assuming young-only tuning fixes Old gen pressure.** Promotion rate matters.
- **Different JDK vendors.** OpenJDK vs Amazon Corretto same flags mostly, verify release notes.
