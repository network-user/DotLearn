# Java Memory Model: видимость и ordering

## Суть

JMM (JSR-133) формально задаёт, когда write одного потока виден read другого, несмотря на CPU cache и compiler reorder. Ключевой инструмент: happens-before edges (monitor enter/exit, volatile read/write, Thread.start/join). Без HB отношения возможны stale reads и surprising reorderings даже для «простых» полей.

## Что ответить на собесе

- JMM решает visibility + instruction ordering, не atomicity compound ops.
- `volatile` даёт HB и visibility; `count++` на volatile всё ещё race.
- `synchronized` release/acquire semantics на monitor.
- `java.util.concurrent` (Atomic*, locks) строится поверх JMM guarantees.
- Честно: double-checked locking без volatile broken; с volatile или enum singleton OK.

## Как это работает

Broken flag (no HB):

```java
public final class BrokenReadyFlag {
    private boolean ready;
    private int payload;

    public void publish(int value) {
        payload = value;
        ready = true;
    }

    public int consume() {
        while (!ready) { Thread.onSpinWait(); }
        return payload;
    }
}
```

Reader может видеть `ready=true` но stale `payload=0` из reorder.

Fixed:

```java
public final class SafeReadyFlag {
    private volatile boolean ready;
    private int payload;

    public void publish(int value) {
        payload = value;
        ready = true;
    }

    public int consume() {
        while (!ready) { Thread.onSpinWait(); }
        return payload;
    }
}
```

`ready=true` HB следующий read `ready` → видимость `payload` write that happened before.

Happens-before sources также: `ConcurrentHashMap` actions, `CountDownLatch`, `BlockingQueue` ops.

## Подводные камни

- **volatile ≠ mutex.** High contention increment still wrong.
- **Final field safe publication.** Immutable object if ref published after ctor complete in same thread without escape.
- **False sharing.** Padding/cache line effects performance, not JMM visibility (orthogonal but confused).
- **Lazy holder without volatile.** Another thread sees partially constructed singleton.
- **Assuming println flushes memory.** IO side effect ≠ HB for arbitrary fields.
