# How many coroutines to run at once: throttling concurrency with asyncio.Semaphore

## In short

To avoid launching thousands of coroutines simultaneously, the number of active tasks is limited with a semaphore. `asyncio.Semaphore(n)` lets through no more than `n` tasks at a time; the rest wait for a slot to free up.

## How to answer in an interview

I take an `asyncio.Semaphore` and set it a limit on simultaneous tasks. Before doing its work, each coroutine acquires the semaphore via `async with sem:` - this occupies one of the `n` slots. When the slots run out, new coroutines asynchronously wait until someone leaves the block and frees up space. I still create all the tasks at once and pass them to `asyncio.gather`, but at most `n` will actually run. This is how you keep the load on an external API under control and stay within request limits.

The semaphore keeps a counter of free slots. `async with sem` decrements it on entry and returns it on exit. While the counter is above zero, a coroutine passes through immediately; at zero, it falls asleep in the waiting queue and wakes up when another coroutine releases a slot. Unlike a blocking thread semaphore, the wait here does not block the event loop - it keeps spinning other tasks.

Limiting the number of simultaneous HTTP requests:

```python
import asyncio
import aiohttp

async def fetch_url(sem, session, url):
    async with sem:
        async with session.get(url) as response:
            return await response.text()

async def main(urls):
    sem = asyncio.Semaphore(3)
    async with aiohttp.ClientSession() as session:
        tasks = [fetch_url(sem, session, url) for url in urls]
        return await asyncio.gather(*tasks)

urls = [
    "http://example.com",
    "http://example.org",
    "http://example.net",
]

results = asyncio.run(main(urls))
print(len(results))
```

Here a semaphore with 3 slots means that even out of a hundred URLs, no more than three requests are in flight at once; the rest wait their turn inside `async with sem`.

## Pitfalls

- **Acquire inside the coroutine, not outside.** `async with sem` should wrap the actual work. If you acquire the semaphore before `gather`, the limit will not work as expected.
- **Create the session once.** Opening a `ClientSession` inside every task is inefficient: it is better to pass a shared session, otherwise the semaphore limits the requests but not the connection overhead.
- **A semaphore limits, it does not balance.** It does not speed up slow tasks or distribute the load - it only prevents you from exceeding the ceiling.
- **Do not confuse it with the number of threads.** This is concurrency within a single thread, not parallelism across cores: a semaphore will not offload heavy CPU-bound code.
- **Choosing the limit.** Too small an `n` underutilizes the network, too large hits API limits or memory.
