# How do you start a batch of coroutines in asyncio with one call and collect their results?

## In short

To run several coroutines and wait for all of them at once, use `asyncio.gather()`. It schedules the passed coroutines concurrently and, once they finish, returns their results as a list in the same order the coroutines were passed.

## How to answer in an interview

"The most common tool is `asyncio.gather(*coros)`. It wraps the coroutines into tasks, hands them to the event loop, and waits until they all complete. The result is a list of values in the order of the arguments, not in the order of completion. If you need more structured control over a group of tasks, Python 3.11+ has `asyncio.TaskGroup`, which is safer in terms of error handling and cancellation."

`asyncio` lets you write concurrent code for I/O-bound tasks: while one coroutine waits for a network or disk response, the event loop switches to another. Important: this is concurrency within a single thread, not parallelism across multiple cores.

`gather` does not start the coroutines "instantly and physically at the same time" - it hands them to the scheduler, which interleaves them at the `await` points. That is why the benefit shows up precisely when waiting on I/O.

```python
import asyncio

async def fetch_data(site):
    print(f"start {site}")
    await asyncio.sleep(2)  # simulate a network request
    print(f"done {site}")
    return f"data:{site}"

async def main():
    sites = ["site1", "site2", "site3"]
    results = await asyncio.gather(*(fetch_data(s) for s in sites))
    print(results)

asyncio.run(main())
```

All three `fetch_data` calls run concurrently, so the total time is about 2 seconds rather than 6. The order in `results` matches the order of `sites`, even if some coroutine finished earlier.

## Pitfalls

- **Order of results != order of completion.** `gather` preserves the order of arguments; do not assume the first result came from the fastest coroutine.
- **The first exception interrupts the wait.** By default `gather` propagates the first raised exception upward, but the remaining tasks are not cancelled automatically and may leave "dangling" warnings. Use `return_exceptions=True` to collect exceptions as ordinary list elements.
- **You must pass coroutines or futures, not the results of calling them in advance.** `gather(fetch_data())` is correct; `gather(fetch_data)` (without the call) is an error.
- **This is not multithreading.** CPU-bound computations inside the coroutines still block the single thread and kill concurrency.
