# Process vs thread: how the units of execution differ

## In short

A process is an isolated instance of a program with its own address space and resources. A thread is a unit of execution inside a process; threads of the same process share its memory, so they communicate easily but are less isolated.

## How to answer in an interview

A process is a running program with its own memory, set of resources, and state; processes are isolated and cannot reach directly into another's memory. A thread lives inside a process and shares its address space with other threads of the same process, so exchanging data between threads is cheap, and creating and switching them is lightweight. The price for this is the lack of isolation: an error in one thread can bring down the whole process, and shared data needs synchronization. In CPython it's important to remember the GIL: threads don't give CPU parallelism, so for compute-bound tasks you reach for processes.

In Python a separate process is created via `multiprocessing` - it has its own memory:

```python
import multiprocessing

def worker():
    print("This is a separate process")

if __name__ == "__main__":
    process = multiprocessing.Process(target=worker)
    process.start()
    process.join()
```

A thread is created via `threading` - it shares memory with the parent process:

```python
import threading

def worker():
    print("This is a separate thread")

if __name__ == "__main__":
    thread = threading.Thread(target=worker)
    thread.start()
    thread.join()
```

The differences along the key axes:

- **Isolation.** Processes are separated, communication goes through IPC. Threads share memory and communicate directly.
- **Cost.** A process is heavy: its own address space. A thread is light: it reuses the process's resources.
- **Scenarios.** Processes - for independent CPU tasks; threads - for I/O tasks over shared data.

## Pitfalls

- In CPython the GIL prevents two threads from executing bytecode simultaneously, so threads speed up I/O-bound, not CPU-bound, work; for CPU you reach for `multiprocessing`.
- Shared memory across threads requires synchronization (`Lock`, `Queue`), otherwise you get data races.
- `multiprocessing` serializes arguments (pickle) and pays for process startup - on small tasks the overhead eats the gain.
- `process.start()` kicks off the work, `process.join()` merely waits for it to finish; mixing up the order means not waiting for completion.
- A crashing thread can take down the whole process; a crashed process doesn't affect its neighbors.
