# The basic asyncio toolkit: run, sleep, create_task, gather, wait - who does what?

## In short

Five functions cover almost every everyday `asyncio` scenario: `run` enters the async world, `sleep` yields control, `create_task` launches a coroutine in the background, `gather` collects the results of a group of coroutines, and `wait` waits for a group of tasks with a flexible stop condition.

## How to answer in an interview

`asyncio.run(coro)` is the entry point: it creates the event loop, runs the coroutine, and closes the loop. `asyncio.sleep(t)` is a non-blocking pause during which the loop switches to other tasks. `create_task(coro)` schedules a coroutine as a `Task` and returns control immediately without waiting for it. `gather(*coros)` runs several coroutines concurrently and returns their results as a list in argument order. `wait(tasks)` waits for a set of already-created tasks and returns two sets - `done` and `pending` - letting you stop on the first completion or on a timeout.

## How it works

`run` is needed exactly once per program - a nested call inside a running loop raises an error.

```python
import asyncio

async def main():
    print("Hello")

asyncio.run(main())
```

`sleep` is the main demonstration "await": while one coroutine sleeps, the loop advances the others.

`create_task` differs from a direct `await`: the task starts in the background right away, and you can wait for it later.

```python
import asyncio

async def work(name, delay):
    await asyncio.sleep(delay)
    return f"{name} done"

async def main():
    task = asyncio.create_task(work("A", 1))
    print("Task scheduled")
    result = await task
    print(result)

asyncio.run(main())
```

`gather` is the most common way to run a group of coroutines and collect the results in argument order rather than completion order:

```python
import asyncio

async def work(name, delay):
    await asyncio.sleep(delay)
    return f"{name} done"

async def main():
    results = await asyncio.gather(work("A", 2), work("B", 1))
    print(results)

asyncio.run(main())
```

`wait` is more flexible: it takes already-created tasks and a `return_when` condition (`ALL_COMPLETED`, `FIRST_COMPLETED`), and the results are pulled manually from the `done` set:

```python
import asyncio

async def work(name, delay):
    await asyncio.sleep(delay)
    return f"{name} done"

async def main():
    tasks = [asyncio.create_task(work("A", 2)),
             asyncio.create_task(work("B", 1))]
    done, pending = await asyncio.wait(tasks, return_when=asyncio.FIRST_COMPLETED)
    for t in done:
        print(t.result())

asyncio.run(main())
```

## Pitfalls

- `gather` preserves argument order in the result, not completion order - a frequent point of confusion in interviews.
- By default, when one coroutine raises, `gather` propagates the exception outward and does not wait for the rest; the `return_exceptions=True` flag changes the behavior - errors are returned as objects in the list.
- `wait` (unlike `gather`) does not unwrap exceptions automatically: they "sleep" inside the task objects until you call `result()`.
- `wait` in recent versions does not accept bare coroutines - you must first wrap them in `create_task`.
- A task created by `create_task` must be awaited somewhere or have a reference kept: otherwise the garbage collector may collect it, and the "task was destroyed" warning is easy to miss.
