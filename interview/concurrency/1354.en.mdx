# Can you get rid of the GIL and how to work around its limitations

## In short

You can't simply "turn off" the GIL in regular CPython: it's built into memory management and guarantees thread safety of the interpreter. Its limitations, however, are worked around with processes, C extensions, or alternative language implementations.

## How to answer in an interview

The GIL is a global lock that means at any moment only one thread executes CPython bytecode. Because of that, threads give no speedup on CPU-bound tasks, but work great on I/O-bound ones, where the GIL is released during input/output. To actually load several cores, I reach for `multiprocessing`: each process has its own interpreter and its own GIL. Heavy computations can be moved into C extensions that compute without holding the lock. There are also GIL-free implementations - Jython, IronPython - and recent CPython has gained an experimental free-threaded mode.

## How it works

While one thread holds the GIL and runs bytecode, the others wait. On I/O system calls the lock is released, which is exactly why multithreading pays off there. For parallel computations you launch separate processes:

```python
from multiprocessing import Process


def worker():
    print("Work in a separate process")


if __name__ == "__main__":
    process = Process(target=worker)
    process.start()
    process.join()
```

Each process gets its own address space and its own GIL, so they execute truly in parallel on different cores.

## Pitfalls

- A typical mistake is to use `threading` for CPU-bound tasks and expect a speedup: because of the GIL there won't be one, you need `multiprocessing` or C extensions.
- On Windows and with `spawn`, the child process re-imports the module, so the launch must be under `if __name__ == "__main__"`, otherwise processes start spawning recursively.
- Processes don't share memory directly: data is passed via serialization (pickle), which adds overhead and requires picklable objects.
- Alternative GIL-free interpreters don't support all C extensions and lag behind CPython in compatibility.
