# Конкурентность и параллелизм - где между ними граница?

## Суть

Конкурентность - про управление множеством задач с перекрывающимися интервалами выполнения (они продвигаются «как бы вместе», но не обязательно в один такт). Параллелизм - про буквально одновременный счёт, для которого нужны несколько ядер или процессоров.

## Что ответить на собесе

Конкурентность - про то, как задачи устроены, параллелизм - про то, как они физически выполняются.

- Конкурентность - структура программы, где несколько задач разделяют ресурс по времени: пока одна ждёт I/O, система переключается на другую. Это можно делать и на одном ядре.
- Параллелизм - исполнение: несколько задач физически считаются в один и тот же момент на разных ядрах.
- В Python `asyncio` и `threading` дают конкурентность (из-за GIL потоки не считают параллельно), а реальный параллелизм по CPU обеспечивает `multiprocessing`.

Конкурентность хорошо описывается через переключение. Один повар на кухне: поставил суп вариться и, не дожидаясь, режет овощи. В один миг он делает одно дело, но обе задачи продвигаются. Так же `asyncio` на `await` переключается между корутинами:

```python
import asyncio

async def task1():
    print("Task 1 started")
    await asyncio.sleep(1)
    print("Task 1 finished")

async def task2():
    print("Task 2 started")
    await asyncio.sleep(1)
    print("Task 2 finished")

async def main():
    await asyncio.gather(task1(), task2())

asyncio.run(main())
```

Параллелизм - это уже два повара: один варит суп, другой режет овощи в тот же момент. В Python это `multiprocessing`, который поднимает отдельные процессы с собственной памятью и своим интерпретатором, обходя GIL:

```python
from multiprocessing import Process
import time

def task(name):
    print(f"{name} started")
    time.sleep(1)
    print(f"{name} finished")

if __name__ == "__main__":
    p1 = Process(target=task, args=("Task 1",))
    p2 = Process(target=task, args=("Task 2",))
    p1.start()
    p2.start()
    p1.join()
    p2.join()
```

## Подводные камни

- Конкурентность не равна параллелизму: конкурентный код на одном ядре переключается, но не считает одновременно.
- Из-за GIL Python-потоки дают конкурентность, но не параллельный счёт чистого Python-кода; они полезны для I/O-bound нагрузки, а не для CPU-bound.
- Для CPU-bound параллелизма берут `multiprocessing` (или native-расширения, отпускающие GIL), а не `threading`.
- `multiprocessing` под Windows и в spawn-режиме требует защиты входной точки через `if __name__ == "__main__"`, иначе можно получить бесконечный рекурсивный запуск процессов.
- Параллелизм добавляет накладные расходы (создание процессов, IPC, сериализация): на мелких задачах он может оказаться медленнее простого последовательного кода.
