# The event loop: who actually runs the coroutines

## In short

The event loop is the dispatcher of an asynchronous program. It runs ready coroutines one after another, tracks the completion of I/O operations, and switches between tasks in a single thread without blocking.

## How to answer in an interview

The event loop is the core of `asyncio`. On their own, `async def` and `await` start nothing: the loop is what actually drives the coroutines. It holds a queue of tasks ready to run, hands control to each in turn up to the next `await`, and at that moment the coroutine hands control back. In parallel the loop uses system mechanisms (`epoll`/`kqueue`/`select`) to watch which sockets and files are ready and wakes up the corresponding coroutines. The result is single-threaded cooperative multitasking: while one task waits on I/O, others run. `asyncio.run(main())` creates the loop, runs the coroutine in it, and closes the loop cleanly.

## How it works

The loop runs on an "event → reaction" scheme: it continuously checks whether any ready tasks or completed operations have appeared and invokes the corresponding coroutine continuations.

```python
import asyncio

async def say_hello():
    print("Hello")
    await asyncio.sleep(1)
    print("World")

async def main():
    await asyncio.gather(say_hello(), say_hello())

asyncio.run(main())
```

Here `asyncio.gather` puts two coroutines to work at the same time. Both print `Hello`, both go into `await asyncio.sleep(1)` and yield control to the loop. A second later the loop wakes them, and both print `World`. The total time is about one second, not two: the waits overlapped because a single loop managed them.

## Pitfalls

- Only one event loop is active per thread. You cannot call `asyncio.run` inside an already running loop (for example, in Jupyter) - it raises `RuntimeError`.
- Any synchronous blocking call inside a coroutine freezes the whole loop: until it returns, the other tasks stall.
- `gather` gives concurrency only with two or more `await` points. Pure CPU computation without `await` still runs sequentially.
- Do not confuse this with OS threads: switching here is cooperative and happens only at `await`, it is not preempted by the scheduler.
