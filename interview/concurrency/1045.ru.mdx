# За счёт чего асинхронный код выигрывает по времени

## Суть

Выигрыш даёт не более быстрый процессор, а отсутствие простоя: пока одна задача ждёт ответа от сети или диска, цикл событий запускает другие. Складываются не времена операций, а их ожидания перекрываются.

## Что ответить на собесе

Ключевая мысль: ускорение работает только на I/O-bound нагрузке.

- В синхронном коде вызов «сеть/файл/БД» блокирует поток: процессор простаивает, ожидая ответа.
- Асинхронная модель строится на двух вещах: неблокирующем вводе-выводе и цикле событий. Операция I/O уходит в фон, корутина на `await` уступает управление, и loop тут же запускает следующую готовую задачу.
- Если запустить много операций конкурентно (через `gather`), их ожидания накладываются друг на друга, и общее время приближается к самой долгой операции, а не к сумме всех.
- На CPU-bound задачах выигрыша нет: там процессор и так занят, уступать нечему.

## Как это работает

В Python это `asyncio`. Корутина имитирует медленный ввод-вывод через `await asyncio.sleep(...)`, отдавая управление вместо блокировки потока.

```python
import asyncio

async def fetch_data(name, delay):
    print(f"{name}: старт")
    await asyncio.sleep(delay)  # неблокирующее ожидание
    print(f"{name}: готово")
    return name

async def main():
    results = await asyncio.gather(
        fetch_data("A", 2),
        fetch_data("B", 2),
        fetch_data("C", 2),
    )
    print(results)

asyncio.run(main())
```

Три задачи по 2 секунды, запущенные через `gather`, завершатся примерно за 2 секунды, а не за 6: все три «спят» одновременно, и их ожидания перекрываются. Если бы их вызвали последовательными `await`, время сложилось бы в 6 секунд. Именно поэтому асинхронность доминирует в веб-серверах (`aiohttp`, `FastAPI`) и сервисах, которые ходят во внешние API и базы данных.

## Подводные камни

- Замена `asyncio.sleep` на `time.sleep` ломает всё: поток блокируется, ожидания не перекрываются, и три задачи снова дают 6 секунд.
- Последовательные `await fetch(); await fetch()` не дают выигрыша: параллелить надо явно через `gather`/`create_task`.
- Для CPU-bound вычислений эффекта нет: нужен `multiprocessing` или вынос расчёта в пул потоков.
- «Быстрее» здесь означает выше пропускную способность и меньше простоя, а не ускорение одной отдельной операции.
