# Сколько корутин запустить разом: дросселирование параллелизма через asyncio.Semaphore

## Суть

Чтобы не запускать тысячи корутин одновременно, число активных задач ограничивают семафором. `asyncio.Semaphore(n)` пропускает не больше `n` задач за раз, остальные ждут освобождения слота.

## Что ответить на собесе

Я беру `asyncio.Semaphore` и задаю ему лимит одновременных задач.

- Каждая корутина перед работой захватывает семафор через `async with sem:` — это занимает один из `n` слотов.
- Когда слоты кончаются, новые корутины асинхронно ждут, пока кто-то выйдет из блока и освободит место.
- Сами задачи я по-прежнему создаю все сразу и передаю в `asyncio.gather`, но реально выполняться будут максимум `n` штук.
- Так держат под контролем нагрузку на внешний API и не выходят за лимиты запросов.

Семафор хранит счётчик свободных слотов. `async with sem` уменьшает его на входе и возвращает обратно на выходе. Пока счётчик больше нуля, корутина проходит сразу; при нуле она засыпает в очереди ожидания и просыпается, когда другая корутина отпустит слот. В отличие от блокирующего потокового семафора, ожидание здесь не блокирует цикл событий — он продолжает крутить другие задачи.

Ограничение числа одновременных HTTP-запросов:

```python
import asyncio
import aiohttp

async def fetch_url(sem, session, url):
    async with sem:
        async with session.get(url) as response:
            return await response.text()

async def main(urls):
    sem = asyncio.Semaphore(3)
    async with aiohttp.ClientSession() as session:
        tasks = [fetch_url(sem, session, url) for url in urls]
        return await asyncio.gather(*tasks)

urls = [
    "http://example.com",
    "http://example.org",
    "http://example.net",
]

results = asyncio.run(main(urls))
print(len(results))
```

Здесь семафор на 3 слота означает, что даже из сотни URL в полёте окажется не больше трёх запросов одновременно; остальные ждут своей очереди внутри `async with sem`.

## Подводные камни

- **Захват внутри корутины, а не снаружи.** `async with sem` должен оборачивать саму работу. Если захватить семафор до `gather`, лимит не сработает так, как ожидается.
- **Сессию создают один раз.** Открывать `ClientSession` внутри каждой задачи неэффективно: лучше передать общую сессию, иначе семафор ограничит запросы, но не накладные расходы на соединения.
- **Семафор не балансирует, а ограничивает.** Он не ускоряет медленные задачи и не распределяет нагрузку — лишь не даёт превысить потолок.
- **Не путать с числом потоков.** Это конкурентность в одном потоке, а не параллелизм на ядрах: тяжёлый CPU-bound код семафор не разгрузит.
- **Подбор лимита.** Слишком маленький `n` недогружает сеть, слишком большой упирается в лимиты API или память.
