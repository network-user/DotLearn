# Cancelling a Task in asyncio: how to correctly stop a running coroutine

## In short

A task in asyncio is stopped by calling `task.cancel()`. This does not kill it instantly: at the nearest await point inside the coroutine, an `asyncio.CancelledError` is raised, which you can intercept to clean up resources.

## How to answer in an interview

To interrupt execution, I call the `cancel()` method on the `Task` object. It only marks the task as being cancelled: the actual cancellation kicks in when the coroutine next reaches an `await`. At that point, `asyncio.CancelledError` is raised inside it. You catch it with `try/except` to release resources, and usually re-raise it via `raise`, otherwise the cancellation will not complete. After `cancel()`, you should `await task` to wait for the actual termination.

## How it works

A `Task` is a wrapper around a coroutine that the event loop schedules. The `cancel()` method only places a cancellation request; the loop delivers `CancelledError` at the moment the task yields control at its next `await`. If the coroutine is stuck in synchronous blocking code without await points, cancellation will not happen until it returns to the loop.

A minimal example of a full cancellation cycle:

```python
import asyncio

async def worker():
    print("Task started")
    try:
        await asyncio.sleep(10)
    except asyncio.CancelledError:
        print("Cleaning up")
        raise
    print("Task finished")

async def main():
    task = asyncio.create_task(worker())
    await asyncio.sleep(1)
    task.cancel()
    try:
        await task
    except asyncio.CancelledError:
        print("Main: task was cancelled")

asyncio.run(main())
```

Here we wrap the coroutine into a task, let it run for a second, then send `cancel()`. The exception surfaces at `await asyncio.sleep(10)`, is handled inside `worker`, re-raised, and reaches `await task` in `main`.

## Pitfalls

- **`cancel()` is not immediate.** Until the coroutine reaches an `await`, cancellation is deferred. A long synchronous stretch will ignore the request.
- **A swallowed `CancelledError`.** If you catch the exception and do not `raise`, the task finishes "normally", and `task.cancelled()` returns `False` - the cancellation effectively fails.
- **Cleanup in `finally`.** It is better to put resource release in `finally` or catch `CancelledError` explicitly, otherwise unclosed connections and files will leak.
- **`await task` is mandatory.** Without waiting after `cancel()`, you will not know whether the task managed to finish, and you may leave dangling coroutines.
