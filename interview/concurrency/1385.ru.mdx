# GIL: какую цену платит Python за потокобезопасность?

## Суть

GIL (Global Interpreter Lock) - глобальная блокировка интерпретатора CPython, из-за которой байт-код в каждый момент исполняет только один поток. Это упрощает написание потокобезопасного кода, но лишает Python-потоки настоящего параллелизма на нескольких ядрах.

## Что ответить на собесе

GIL - это компромисс. Плюс в том, что внутренние структуры интерпретатора и подсчёт ссылок защищены автоматически: разработчику не нужно вручную синхронизировать доступ к объектам, а C-расширения проще писать безопасно. Минус в том, что CPU-bound задачи на потоках не ускоряются - сколько бы ядер ни было, Python-код крутится по очереди. Для параллельных вычислений берут `multiprocessing` или C-расширения, освобождающие GIL, а потоки оставляют для I/O-bound нагрузки, где GIL отпускается на время ожидания ввода-вывода.

## Как это работает

GIL нужен, потому что многие операции в CPython не атомарны, а память управляется через счётчики ссылок, которые сами по себе не рассчитаны на параллельное изменение. Блокировка гарантирует, что в момент исполнения байт-кода работает ровно один поток.

Сильные стороны:

1. Код проще - не нужно вручную защищать каждый общий объект.
2. Гонок данных на уровне объектов Python не возникает, пока работает один поток.
3. C-расширения опираются на GIL и пишутся безопаснее.

Слабые стороны:

1. На CPU-bound задачах несколько ядер не задействуются - есть потолок производительности.
2. Частые переключения потоков добавляют накладные расходы на захват и отпускание GIL.
3. Масштабировать вычисления по ядрам в рамках одного процесса не получается.

Классический пример - общий счётчик из двух потоков:

```python
import threading

def increment(counter):
    for _ in range(1000000):
        counter[0] += 1

counter = [0]
t1 = threading.Thread(target=increment, args=(counter,))
t2 = threading.Thread(target=increment, args=(counter,))
t1.start(); t2.start()
t1.join(); t2.join()
print(counter[0])
```

Здесь операция `counter[0] += 1` - это чтение, прибавление и запись, между которыми GIL может переключить поток. Поэтому итог часто меньше ожидаемых 2 000 000.

## Подводные камни

- GIL не делает составные операции атомарными. `+= 1` над общим объектом всё равно требует `Lock`, иначе обновления теряются - как в примере выше.
- Заблуждение «GIL значит безопасно» опасно: безопасны лишь отдельные байт-код-операции, но не последовательности из них.
- Потоки реально полезны для I/O-bound кода (сеть, диск): там GIL отпускается на время блокирующего вызова, и потоки работают почти параллельно.
- Для CPU-bound параллелизма потоки бесполезны - нужен `multiprocessing`, отдельные процессы или библиотеки, выпускающие GIL.
