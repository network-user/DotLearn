# Какую задачу решает асинхронность в Python

## Суть

Асинхронность убирает простой на операциях ввода-вывода: пока один вызов ждёт ответа от сети, диска или БД, программа в том же потоке занимается другой работой. Итог — выше пропускная способность и отзывчивость без лишних потоков.

## Что ответить на собесе

Асинхронность нужна для I/O-bound нагрузки - там, где время уходит не на вычисления, а на ожидание внешнего ответа.

- Классический синхронный код на таком ожидании просто стоит и держит поток занятым.
- Модель `asyncio` позволяет одному потоку обслуживать тысячи одновременных соединений: на каждом `await` корутина уступает управление циклу событий, и тот переключается на готовую работу.
- Это дешевле потоков (нет накладных расходов на их создание и переключение контекста ОС) и обходит ограничения GIL, потому что узкое место — ожидание, а не процессор.
- Для CPU-bound задач асинхронность бесполезна — там нужны процессы.

## Как это работает

Базовый набор — модуль `asyncio` и ключевые слова `async`/`await`. Корутина помечается `async def`, а `await` ставит её на паузу до завершения ожидаемой операции, отдавая управление циклу.

```python
import asyncio

async def fetch_data():
    print("Начинаем загрузку данных...")
    await asyncio.sleep(2)  # имитация задержки ввода-вывода
    print("Данные загружены!")
    return "Данные"

async def main():
    print("Запуск программы")
    data = await fetch_data()
    print(f"Получены данные: {data}")

asyncio.run(main())
```

Реальный выигрыш виден, когда таких операций много и они идут конкурентно — например, через `asyncio.gather(...)`: общее время приближается к времени самой долгой операции, а не к их сумме. Поэтому асинхронность доминирует в веб-серверах (`FastAPI`, `aiohttp`), асинхронных драйверах БД и сетевых сервисах с множеством одновременных соединений.

## Подводные камни

- Асинхронность ускоряет только I/O-bound код. Тяжёлый расчёт внутри корутины блокирует цикл, и весь сервис «зависает».
- Один синхронный блокирующий вызов (`requests.get`, `time.sleep`, обычный драйвер БД) останавливает весь цикл событий. Внутри `async` нужны асинхронные аналоги.
- Конкурентность ≠ параллелизм: всё крутится в одном потоке. Несколько ядер задействуют через процессы, а не `asyncio`.
- Последовательные `await fetch(); await fetch()` не дают ускорения. Параллелить нужно явно через `gather` или `create_task`.
