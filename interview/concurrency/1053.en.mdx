# Concurrency and parallelism - where is the line between them?

## In short

Concurrency is about managing many tasks with overlapping execution intervals (they progress "as if together", but not necessarily on the same tick). Parallelism is about literally simultaneous computation, which requires multiple cores or processors.

## How to answer in an interview

Concurrency is the structure of a program where several tasks share a resource over time: while one waits on I/O, the system switches to another. This can be done even on a single core. Parallelism is execution: several tasks are physically computed at the very same moment on different cores. In Python, `asyncio` and `threading` provide concurrency (because of the GIL, threads do not compute in parallel), while real CPU parallelism is provided by `multiprocessing`. Short formula: concurrency is about how tasks are structured, parallelism is about how they are physically executed.

Concurrency is well described through switching. One cook in the kitchen: he puts the soup on to simmer and, without waiting, starts chopping vegetables. At any instant he does one thing, but both tasks make progress. Likewise `asyncio` switches between coroutines at `await`:

```python
import asyncio

async def task1():
    print("Task 1 started")
    await asyncio.sleep(1)
    print("Task 1 finished")

async def task2():
    print("Task 2 started")
    await asyncio.sleep(1)
    print("Task 2 finished")

async def main():
    await asyncio.gather(task1(), task2())

asyncio.run(main())
```

Parallelism is already two cooks: one makes the soup, the other chops vegetables at the same moment. In Python this is `multiprocessing`, which spins up separate processes with their own memory and their own interpreter, bypassing the GIL:

```python
from multiprocessing import Process
import time

def task(name):
    print(f"{name} started")
    time.sleep(1)
    print(f"{name} finished")

if __name__ == "__main__":
    p1 = Process(target=task, args=("Task 1",))
    p2 = Process(target=task, args=("Task 2",))
    p1.start()
    p2.start()
    p1.join()
    p2.join()
```

## Pitfalls

- Concurrency is not parallelism: concurrent code on a single core switches but does not compute simultaneously.
- Because of the GIL, Python threads provide concurrency but not parallel computation of pure Python code; they are useful for I/O-bound workloads, not CPU-bound ones.
- For CPU-bound parallelism you use `multiprocessing` (or native extensions that release the GIL), not `threading`.
- `multiprocessing` on Windows and in spawn mode requires guarding the entry point with `if __name__ == "__main__"`, otherwise you can get infinite recursive process spawning.
- Parallelism adds overhead (process creation, IPC, serialization): on small tasks it can end up slower than plain sequential code.
