# Процесс против потока: чем различаются единицы выполнения

## Суть

Процесс - это изолированный экземпляр программы со своим адресным пространством и ресурсами. Поток - единица выполнения внутри процесса; потоки одного процесса делят его память, поэтому общаются легко, но менее изолированы.

## Что ответить на собесе

Процесс - это изолированный экземпляр программы, поток - лёгкая единица выполнения внутри него.

- Процесс - запущенная программа со своей памятью, набором ресурсов и состоянием; процессы изолированы и напрямую в чужую память залезть не могут.
- Поток живёт внутри процесса и разделяет его адресное пространство с другими потоками, поэтому обмен данными дешёвый, а создание и переключение - лёгкие.
- Платой становится отсутствие изоляции: ошибка в одном потоке может повалить весь процесс, и нужна синхронизация общих данных.
- В CPython важно помнить про GIL: потоки не дают параллелизма по CPU, для счётных задач берут процессы.

Отдельный процесс в Python создают через `multiprocessing` - у него своя память:

```python
import multiprocessing

def worker():
    print("Это отдельный процесс")

if __name__ == "__main__":
    process = multiprocessing.Process(target=worker)
    process.start()
    process.join()
```

Поток создают через `threading` - он делит память с родительским процессом:

```python
import threading

def worker():
    print("Это отдельный поток")

if __name__ == "__main__":
    thread = threading.Thread(target=worker)
    thread.start()
    thread.join()
```

Различия по ключевым осям:

- **Изоляция.** Процессы отделены, общение - через IPC. Потоки делят память и общаются напрямую.
- **Цена.** Процесс тяжёлый: своё адресное пространство. Поток лёгкий: переиспользует ресурсы процесса.
- **Сценарии.** Процессы - для независимых CPU-задач; потоки - для I/O-задач над общими данными.

## Подводные камни

- В CPython GIL не даёт двум потокам одновременно исполнять байт-код, поэтому потоки ускоряют I/O-bound, а не CPU-bound нагрузку; для CPU берут `multiprocessing`.
- Общая память у потоков требует синхронизации (`Lock`, `Queue`), иначе - гонки данных.
- `multiprocessing` сериализует аргументы (pickle) и платит за старт процесса - на мелких задачах накладные расходы съедают выигрыш.
- `process.start()` запускает работу, `process.join()` лишь дожидается её конца; перепутать порядок - значит не дождаться завершения.
- Падение потока способно уронить весь процесс; упавший процесс соседей не задевает.
