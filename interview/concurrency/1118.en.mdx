# asyncio, threads, and processes: which approach for which workload?

## In short

Three ways to "do several things at once" solve different problems: `asyncio` is cooperative concurrency on a single thread for I/O-bound workloads; threads are several lines of execution in shared memory, useful for blocking I/O; processes are independent interpreters across different cores for CPU-bound computation.

## How to answer in an interview

The choice is dictated by the workload type. If a task is bottlenecked on waiting for network or disk (I/O-bound) - use `asyncio`: one thread, minimal overhead, thousands of simultaneous connections. If you need to combine blocking I/O or work with old synchronous libraries - `threading` fits, but because of the GIL it will not speed up pure Python computation. If the task is CPU-bound (math, data processing) - you need `multiprocessing`: separate processes bypass the GIL and genuinely compute on several cores, at the cost of a more expensive start and data exchange through serialization.

## How it works

`asyncio` switches coroutines at `await` without creating new threads:

```python
import asyncio

async def fetch_data():
    print("Start fetching")
    await asyncio.sleep(2)
    print("Done fetching")
    return {"data": 123}

async def main():
    data = await fetch_data()
    print(data)

asyncio.run(main())
```

Threads live in the shared memory of one process and share its resources - convenient, but they require synchronization when state is shared:

```python
import threading

def print_numbers():
    for i in range(5):
        print(i)

def print_letters():
    for letter in "abcde":
        print(letter)

t1 = threading.Thread(target=print_numbers)
t2 = threading.Thread(target=print_letters)
t1.start()
t2.start()
t1.join()
t2.join()
```

Processes are isolated: their own memory, their own interpreter, exchange only through explicit channels - but in return real parallelism across cores:

```python
from multiprocessing import Process

def print_numbers():
    for i in range(5):
        print(i)

def print_letters():
    for letter in "abcde":
        print(letter)

if __name__ == "__main__":
    p1 = Process(target=print_numbers)
    p2 = Process(target=print_letters)
    p1.start()
    p2.start()
    p1.join()
    p2.join()
```

## Pitfalls

- The main fork is I/O-bound versus CPU-bound: for pure computation `asyncio` and `threading` are useless - only `multiprocessing` helps.
- The GIL makes threads pointless for speeding up Python computation, but they are still useful for blocking I/O and integration with synchronous code.
- Threads' shared memory is a source of races: shared state needs locks.
- Processes are more expensive than threads: startup, separate memory, and serialization of arguments/results through `pickle` create overhead.
- `asyncio` does not tolerate blocking calls in coroutines: a single synchronous `time.sleep` or heavy computation stalls the whole loop - such work is offloaded to an executor.
