# Why use a hash table and how does HashMap work?

## In short

A hash table stores key-value pairs and offers amortized constant-time insert, lookup, and delete. A key runs through a hash function, the result maps to a bucket index, and collisions are handled with chaining or another slot search strategy.

## How to answer in an interview

- Define a hash map as an array of buckets plus a `hash(key) → index` function.
- Explain collisions: different keys can land in the same bucket; in Java 8+ that bucket is a linked list or a red-black tree when the chain grows long.
- Mention `equals`/`hashCode`: without a consistent contract the map behaves unpredictably.
- Name typical uses: cache, index, frequency counting, deduplication.
- Be honest: with a bad hash function or a collision attack, worst case degrades to O(n); for hot paths I watch load factor and implementation choice.

## How it works

Insert follows three steps: compute the key hash, pick a bucket, store the entry. Lookup uses the same index; if several entries share the bucket, keys are compared with `equals`.

```java
import java.util.HashMap;
import java.util.Map;

public final class InventoryIndex {
    public static void main(String[] args) {
        Map<String, Integer> stockBySku = new HashMap<>();

        stockBySku.put("SKU-1001", 42);
        stockBySku.put("SKU-2044", 7);
        stockBySku.put("SKU-3300", 15);

        int onHand = stockBySku.getOrDefault("SKU-1001", 0);
        System.out.println("SKU-1001 on hand: " + onHand);

        stockBySku.remove("SKU-2044");

        boolean tracked = stockBySku.containsKey("SKU-3300");
        System.out.println("Tracking SKU-3300: " + tracked);
    }
}
```

Java `HashMap` is unordered: iteration order can change after rehash. When load factor crosses the threshold (0.75 by default), the table grows and entries move. Keys should be immutable or at least must not change fields used in `hashCode` while stored in the map.

Use `LinkedHashMap` for insertion-order traversal; use `TreeMap` for sorted keys (O(log n), no classic hash collisions).

## Pitfalls

- **Mutable key.** Changing a field that feeds `hashCode` "loses" the entry in its bucket.
- **Weak hashCode.** A constant or nearly constant hash collapses the map into one bucket and linear search.
- **equals without hashCode.** Breaking the `Object` contract breaks lookup and `containsKey`.
- **Null keys and values.** `HashMap` allows one null key and multiple null values; `ConcurrentHashMap` rejects null.
- **Iterate while mutating.** Structural changes during active iteration throw `ConcurrentModificationException` unless removal goes through the iterator API.
