# Зачем нужна хеш-таблица и как устроен HashMap?

## Суть

Хеш-таблица хранит пары ключ-значение и даёт амортизированно константное время на вставку, поиск и удаление. Ключ прогоняется через hash-функцию, результат берётся по модулю размера корзины, а при коллизиях элементы связываются в цепочку или ищутся другим слотом.

## Что ответить на собесе

- Определяю hash map как массив корзин (buckets) плюс функция `hash(key) → index`.
- Объясняю коллизии: разные ключи могут попасть в один bucket; в Java 8+ это linked list или red-black tree при длинной цепочке.
- Упоминаю `equals`/`hashCode`: без согласованного контракта map ведёт себя непредсказуемо.
- Называю типичные применения: кэш, индекс, подсчёт частот, дедупликация.
- Честно добавляю: worst case при плохой hash-функции или атаке на коллизии деградирует до O(n); для критичных путей смотрю на load factor и выбор реализации.

## Как это работает

Алгоритм вставки в три шага: вычислить hash ключа, выбрать bucket, положить entry. Поиск идёт по тому же индексу; если в bucket несколько элементов, сравниваются ключи через `equals`.

```java
import java.util.HashMap;
import java.util.Map;

public final class InventoryIndex {
    public static void main(String[] args) {
        Map<String, Integer> stockBySku = new HashMap<>();

        stockBySku.put("SKU-1001", 42);
        stockBySku.put("SKU-2044", 7);
        stockBySku.put("SKU-3300", 15);

        int onHand = stockBySku.getOrDefault("SKU-1001", 0);
        System.out.println("SKU-1001 on hand: " + onHand);

        stockBySku.remove("SKU-2044");

        boolean tracked = stockBySku.containsKey("SKU-3300");
        System.out.println("Tracking SKU-3300: " + tracked);
    }
}
```

`HashMap` в Java не упорядочен: порядок обхода может меняться после rehash. Когда load factor превышает порог (по умолчанию 0.75), таблица расширяется и элементы перераспределяются. Ключи должны быть immutable или хотя бы не менять поля, участвующие в `hashCode`, пока лежат в map.

Для упорядоченного обхода берут `LinkedHashMap`; для сортировки по ключу `TreeMap` (O(log n), но без hash-коллизий в классическом виде).

## Подводные камни

- **Mutable ключ.** Изменение поля, входящего в `hashCode`, «теряет» entry в bucket.
- **Слабый hashCode.** Константный или почти константный hash сводит map к одному bucket и линейному поиску.
- **equals без hashCode.** Нарушение контракта `Object` ломает поиск и `containsKey`.
- **Null-ключи и null-значения.** `HashMap` допускает один null key и несколько null values; `ConcurrentHashMap` null не принимает.
- **Итерация во время модификации.** Структурное изменение map при активном iterator даёт `ConcurrentModificationException`, если нет явного `remove` через iterator API.
