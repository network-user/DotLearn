# How many AZs does Multi-AZ RDS cover?

## In short

RDS Multi-AZ keeps a synchronous standby copy in another Availability Zone in the same region: one primary AZ and one standby AZ (two zones). It is not cross-region replica; read scaling uses Read Replicas separately.

## How to answer in an interview

- Automatic failover: the DNS endpoint switches to standby when primary fails.
- Synchronous replica: commit waits on standby (slightly higher latency than single-AZ).
- Backups and maintenance often run from standby, lowering primary load.
- Honestly: Multi-AZ doubles instance compute cost; for dev I often use single-AZ plus a snapshot policy.

## How it works

Enable Multi-AZ via CLI:

```bash
aws rds modify-db-instance \
  --db-instance-identifier payments-primary \
  --multi-az \
  --apply-immediately

aws rds describe-db-instances \
  --db-instance-identifier payments-primary \
  --query 'DBInstances[0].{MultiAZ:MultiAZ,AZ:AvailabilityZone,Secondary:SecondaryAvailabilityZone}'
```

On AZ failure RDS promotes standby in the other zone; the app reconnects to the same endpoint. RPO is near zero for sync replica; RTO is minutes. Cross-region DR: a separate Read Replica in another region with promote.

## Pitfalls

- **Confused with Read Replica.** A replica in the same region does not give Multi-AZ auto failover.
- **Failover during peak.** Connection storm without a pooler (RDS Proxy) knocks over the app.
- **Maintenance reboot.** Multi-AZ reduces downtime but not zero without graceful drain.
- **Only one standby.** Not three AZ copies out of the box.
- **Latency-sensitive writes.** Sync replica in a peer AZ adds ms on commit.
