# Как хранить Terraform state в S3?

## Суть

Remote backend `s3` кладёт `terraform.tfstate` в bucket вместо локального файла: команда видит одно состояние, state не попадает в git. Минимум: bucket с versioning, backend-блок в конфиге, IAM-права на объект. Для командной работы добавляю DynamoDB table для state locking.

## Что ответить на собесе

- Создаю отдельный bucket только под state; versioning обязателен для отката.
- `encrypt=true` и SSE-S3 или KMS; bucket policy запрещает публичный доступ.
- DynamoDB lock table (`LockID` string hash key) блокирует параллельный `apply`.
- `terraform init -migrate-state` переносит локальный state в S3 один раз.
- Честно: без locking два `apply` одновременно коррумпируют state; в CI lock обязателен.

## Как это работает

Подготовка bucket и lock table:

```bash
aws s3api create-bucket \
  --bucket tfstate-myorg-prod \
  --region eu-central-1 \
  --create-bucket-configuration LocationConstraint=eu-central-1

aws s3api put-bucket-versioning \
  --bucket tfstate-myorg-prod \
  --versioning-configuration Status=Enabled

aws dynamodb create-table \
  --table-name tfstate-lock \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

Backend в `backend.tf`:

```hcl
terraform {
  backend "s3" {
    bucket         = "tfstate-myorg-prod"
    key            = "platform/vpc/terraform.tfstate"
    region         = "eu-central-1"
    encrypt        = true
    dynamodb_table = "tfstate-lock"
  }
}
```

Инициализация и миграция:

```bash
terraform init -migrate-state
terraform plan
```

IAM policy для CI role: `s3:GetObject`, `s3:PutObject`, `s3:DeleteObject` на `arn:aws:s3:::tfstate-myorg-prod/platform/*` и `dynamodb:GetItem`, `PutItem`, `DeleteItem` на lock table.

## Подводные камни

- **State в git.** Даже с remote backend случайный commit `.tfstate` утекает секреты.
- **Без versioning.** `apply` с ошибкой перезаписывает state; откат невозможен.
- **Один key на все модули.** Monolith state тормозит plan; делю по stack (`vpc/`, `eks/`).
- **Lock table в другом регионе.** Latency и cross-region billing; держу рядом с bucket.
- **KMS key rotation.** CI role нужен `kms:Decrypt/GenerateDataKey` на ключ bucket.
