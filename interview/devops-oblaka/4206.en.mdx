# EBS vs EFS: block vs shared file storage

## In short

EBS is a block volume for one EC2 instance in one AZ: low latency, gp3/io2 for databases and boot volumes. EFS is multi-AZ NFS: shared read/write mounts on many instances, elastic capacity, higher latency.

## How to answer in an interview

- EBS attaches to one instance (multi-attach only io2 Block Express with limits).
- EFS mount target per AZ; security group plus NFS 2049.
- K8s StatefulSet on EBS via CSI; shared content (media, configs) on EFS.
- Honestly: EFS costs more at sustained high IOPS; hot database data belongs on EBS, not EFS.

## How it works

EBS gp3 volume:

```bash
aws ec2 create-volume --availability-zone eu-central-1a --size 100 --volume-type gp3 \
  --iops 3000 --throughput 125

aws ec2 attach-volume --volume-id vol-0abc --instance-id i-0def --device /dev/xvdf
```

EFS mount on Linux:

```bash
aws efs create-file-system --encrypted --performance-mode generalPurpose \
  --throughput-mode bursting

sudo mount -t nfs4 -o nfsvers=4.1 fs-0123456789.efs.eu-central-1.amazonaws.com:/ /mnt/shared
```

EBS snapshots for backup; EFS backup policy or AWS Backup. Resize EBS online; EFS grows automatically.

## Pitfalls

- **EBS AZ lock.** Cannot attach a volume in another AZ without snapshot copy.
- **EFS latency spikes.** Small random NFS writes hurt the app.
- **Encryption at rest off on old volumes.** I enable on create.
- **EFS burst credits.** Bursting throughput mode exhausts on batch jobs.
- **Full root volume.** Default AMI 8 GiB; I monitor disk, not only CPU.
