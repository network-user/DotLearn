# How do you store Terraform state in S3?

## In short

The remote `s3` backend stores `terraform.tfstate` in a bucket instead of a local file: the team shares one state, and state never lands in git. Minimum setup: a bucket with versioning, a backend block in config, and IAM rights on the object. For team workflows I add a DynamoDB table for state locking.

## How to answer in an interview

- I create a dedicated bucket for state only; versioning is mandatory for rollback.
- `encrypt=true` with SSE-S3 or KMS; bucket policy blocks public access.
- DynamoDB lock table (`LockID` string hash key) blocks concurrent `apply`.
- `terraform init -migrate-state` moves local state to S3 once.
- Honestly: without locking, two simultaneous `apply` runs corrupt state; lock is mandatory in CI.

## How it works

Prepare bucket and lock table:

```bash
aws s3api create-bucket \
  --bucket tfstate-myorg-prod \
  --region eu-central-1 \
  --create-bucket-configuration LocationConstraint=eu-central-1

aws s3api put-bucket-versioning \
  --bucket tfstate-myorg-prod \
  --versioning-configuration Status=Enabled

aws dynamodb create-table \
  --table-name tfstate-lock \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

Backend in `backend.tf`:

```hcl
terraform {
  backend "s3" {
    bucket         = "tfstate-myorg-prod"
    key            = "platform/vpc/terraform.tfstate"
    region         = "eu-central-1"
    encrypt        = true
    dynamodb_table = "tfstate-lock"
  }
}
```

Init and migration:

```bash
terraform init -migrate-state
terraform plan
```

IAM policy for the CI role: `s3:GetObject`, `s3:PutObject`, `s3:DeleteObject` on `arn:aws:s3:::tfstate-myorg-prod/platform/*` and `dynamodb:GetItem`, `PutItem`, `DeleteItem` on the lock table.

## Pitfalls

- **State in git.** Even with a remote backend, an accidental `.tfstate` commit leaks secrets.
- **No versioning.** A failed `apply` overwrites state; rollback is impossible.
- **One key for all modules.** Monolith state slows plan; I split by stack (`vpc/`, `eks/`).
- **Lock table in another region.** Extra latency and cross-region billing; I keep it next to the bucket.
- **KMS key rotation.** The CI role needs `kms:Decrypt/GenerateDataKey` on the bucket key.
