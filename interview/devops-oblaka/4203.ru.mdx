# NAT Gateway и Internet Gateway в AWS: в чём разница?

## Суть

Internet Gateway (IGW) даёт публичной подсети двусторонний интернет: входящий и исходящий трафик на ресурсы с public IP. NAT Gateway в public subnet переводит исходящий трафик private subnet в интернет, входящие init соединения к private ресурсам не проходят.

## Что ответить на собесе

- IGW attach к VPC; route `0.0.0.0/0` → igw в public route table.
- NAT GW в public AZ; private route table `0.0.0.0/0` → nat-xxx.
- ALB в public, app/DB в private за NAT для патчей и API calls наружу.
- Честно: NAT Gateway платный per-hour и per-GB; для dev иногда NAT instance или VPC endpoints вместо полного NAT.

## Как это работает

Типичная схема VPC:

```bash
aws ec2 create-internet-gateway --query InternetGateway.InternetGatewayId --output text
aws ec2 attach-internet-gateway --internet-gateway-id igw-0abc --vpc-id vpc-0def

aws ec2 create-nat-gateway --subnet-id subnet-public1a \
  --allocation-id eipalloc-0aaa --query NatGateway.NatGatewayId --output text

aws ec2 create-route --route-table-id rtb-public \
  --destination-cidr-block 0.0.0.0/0 --gateway-id igw-0abc

aws ec2 create-route --route-table-id rtb-private \
  --destination-cidr-block 0.0.0.0/0 --nat-gateway-id nat-0bbb
```

EC2 в private subnet без public IP ходит в S3 через NAT или через Gateway Endpoint (без NAT charge). Security group на DB всё равно deny ingress из 0.0.0.0/0.

## Подводные камни

- **NAT в одной AZ.** Падение AZ лишает private subnet интернета без second NAT.
- **IGW на private subnet по ошибке.** Инстансы с public IP обнажаются в интернет.
- **Забытый default route.** Private subnet без 0.0.0.0/0 к NAT не тянет образы.
- **NACL vs SG путаница.** Открытый SG перечёркивает «скрытность» NAT.
- **IPv6 другой путь.** Egress-only IGW для v6, не тот же NAT GW.
