# EC2 families C, M, and T: how to choose?

## In short

C (compute optimized) maximizes vCPU per dollar for CPU-bound work. M (general purpose) balances CPU/RAM/network for typical workloads. T (burstable) offers baseline CPU plus credits for low average load with rare spikes.

## How to answer in an interview

- C7i/c6i: batch, transcode, scientific, high RPS without huge RAM.
- M7i/m6i: app servers, mid-size DB, default Kubernetes nodes.
- T3/T4g: dev/stage, bastion, monitoring sidecars; I watch CPUCreditBalance.
- Honestly: T without unlimited mode throttles hard; prod API on T3 without credit monitoring is risky.

## How it works

Check credits on T3:

```bash
aws cloudwatch get-metric-statistics --namespace AWS/EC2 \
  --metric-name CPUCreditBalance --dimensions Name=InstanceId,Value=i-0abc \
  --start-time 2026-07-13T00:00:00Z --end-time 2026-07-13T12:00:00Z \
  --period 3600 --statistics Average
```

Resize family via stop/modify:

```bash
aws ec2 stop-instances --instance-ids i-0abc
aws ec2 modify-instance-attribute --instance-id i-0abc --instance-type m7i.large
aws ec2 start-instances --instance-ids i-0abc
```

Graviton (g suffix) is cheaper with arm-compatible images. Right-sizing: Compute Optimizer or own CPU/RAM p95 metrics.

## Pitfalls

- **T instance unlimited off.** 20% CPU baseline and throttle.
- **C without enough RAM.** OOM killer with the same vCPU count as M.
- **Oversized M.** Paying for idle RAM in a 24/7 fleet.
- **Burstable for sustained load.** Credits do not recover faster than consumption.
- **Wrong EBS throughput.** CPU is not the bottleneck; gp2 IOPS cap is.
