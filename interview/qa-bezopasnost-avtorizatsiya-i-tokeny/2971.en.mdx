# 401 and 403: how does QA separate authn from authz?

## In short

401 Unauthorized means "identity not established": no token, expired session, wrong password. 403 Forbidden means "identity known, insufficient rights". Mixing codes breaks client refresh flow and misleads users. QA checks status, error body, and WWW-Authenticate headers.

## How to answer in an interview

- No Authorization header on a protected route: expect 401, not 404 (security through obscurity is debatable, but 404 on admin hides existence).
- Valid viewer token on admin-only DELETE: expect 403 with INSUFFICIENT_PERMISSION, not 401.
- 401 often triggers re-login or refresh; 403 must not refresh forever.
- Honestly: on public read endpoints anonymous 200 sometimes applies; I fix the contract in OpenAPI, not "how it feels".

## How it works

Typical responses:

```http
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="api"
Content-Type: application/json

{"code":"TOKEN_EXPIRED"}
```

```http
HTTP/1.1 403 Forbidden
Content-Type: application/json

{"code":"FORBIDDEN","role":"viewer"}
```

Gherkin:

```gherkin
Feature: Authorization error codes

  Scenario: Anonymous request to protected API
    When GET /api/v1/me without Authorization
    Then status is 401
    And body contains code UNAUTHENTICATED

  Scenario: Authenticated viewer on admin route
    Given Bearer token role viewer
    When GET /api/v1/admin/metrics
    Then status is 403
    And body does not expose metrics

  Scenario: Expired token is not masked as 403
    Given access token exp in the past
    When GET /api/v1/me
    Then status is 401
    And client runs refresh flow once
```

In UI 403 shows friendly "no access", 401 redirects to login with returnUrl.

## Pitfalls

- **403 instead of 401 on expired token.** Refresh never runs, logout loop.
- **401 for wrong password and missing header.** Different UX, same code.
- **CORS hides 401 in browser.** OK in Postman, "network error" in UI.
- **API gateway vs service.** Different codes on one path through mesh.
- **404 on forbidden object.** IDOR masked; test with two users needed.
