# Where QA finds the token: headers, storage, and DevTools?

## In short

Bearer access usually goes in the `Authorization` header; refresh may live in HttpOnly cookie or secure storage. SPAs often keep access in memory or sessionStorage. QA traces the full path: login response → storage → outbound requests → logout purge.

## How to answer in an interview

- Network tab: pick XHR/fetch, Request Headers → `Authorization: Bearer ...`.
- Application tab: Local/Session Storage, Cookies (HttpOnly visible but not readable from JS).
- After logout no token should remain in storage or memory (I check re-entry without password fails).
- Honestly: I blur tokens in bug screenshots; in Jira only "present/absent", not the value.

## How it works

Typical SPA path:

```
POST /login → Set-Cookie refresh HttpOnly
           → access in memory
GET /api → Authorization: Bearer <access>
```

Gherkin:

```gherkin
Feature: Token lifecycle in the browser

  Scenario: Access sent in Authorization
    Given user logged in
    When GET /api/v1/profile in Network
    Then request headers contain Authorization Bearer
    And token is not duplicated in query string

  Scenario: Refresh not available from console
    Given HttpOnly session cookie
    When QA runs document.cookie in console
    Then refresh cookie is not in the string
    And cookie-backed requests stay authenticated

  Scenario: Logout clears client state
    When user Logout
    Then sessionStorage is empty
    And next GET /api/v1/me without login returns 401
```

Postman Tests `pm.environment.set` only on local machine; not committed to shared workspace.

## Pitfalls

- **Token in WebSocket subprotocol/query.** Easy to miss in Network tab.
- **Service Worker caches request with old Authorization.** Flaky after refresh.
- **Two tabs, logout in one.** Second tab still sends old access.
- **Mobile Keychain not visible to QA without jailbreak.** Use proxy capture.
- **Copy as cURL leaks to Slack.** Sanitize before sharing.
