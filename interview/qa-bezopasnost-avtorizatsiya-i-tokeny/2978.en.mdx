# Testing authentication: a QA engineer checklist

## In short

Authentication confirms identity before issuing a token or session. QA covers positive login, negative credentials, brute-force lockout, MFA, password reset, session timeout, and transport security. Focus on server-side: clients can bypass UI, not backend hash checks.

## How to answer in an interview

- Equivalence classes: empty fields, wrong password, locked account, login case sensitivity, unicode password.
- Session fixation: after login session id changes; cookie flags Secure, HttpOnly, SameSite.
- MFA: OTP window, backup codes, rate limit on SMS resend.
- Honestly: full OWASP ASVS is not every sprint; I prioritize by product risk matrix.

## How it works

Checklist areas:

| Area | Sample tests |
|------|--------------|
| Login | valid/invalid, lockout |
| Logout | invalidate server session |
| Reset | token one-time, TTL |
| MFA | TOTP drift, bypass |

Gherkin auth smoke:

```gherkin
Feature: Basic authentication

  Scenario: Successful login
    Given active user qa@example.com
    When POST /auth/login with correct password
    Then status is 200
    And Set-Cookie session with Secure and HttpOnly

  Scenario: Brute force lockout
    Given limit 5 failed attempts
    When 6th POST with wrong password
    Then status is 429 or 401 account_locked
    And 7th with correct password also rejected until timeout

  Scenario: Password reset one-time link
    Given reset token from email
    When password change succeeds
    And link is reused
    Then status is 400 token_used
```

For API I use Postman collections with negative data; for UI I also check messages avoid user enumeration.

## Pitfalls

- **User enumeration.** Different "user not found" vs "wrong password" messages.
- **Remember-me forever cookie.** Logout does not invalidate server-side.
- **OAuth social UI only.** API login without password still open.
- **Captcha only on 10th attempt.** Script succeeds on 9.
- **Test bypass header on staging.** Left enabled in prod.
