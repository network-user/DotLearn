# Transpilation in frontend builds

## In short

Transpilation rewrites source into different syntax that stays semantically equivalent at the same level (TS → JS, ES2022 → ES2018, JSX → `React.createElement`). Unlike compilation, it does not emit machine code; it changes source text. It serves browser targets and languages above JS (TS, Vue SFC).

## How to answer in an interview

- Babel, TypeScript (`tsc`), esbuild, SWC: different speed/feature tradeoffs.
- `target`/`browserslist` sets syntax transforms; polyfills are separate (`core-js`).
- TS transpileOnly via esbuild in Vite dev; typecheck runs separately with `tsc --noEmit`.
- Honestly: evergreen-only projects cut transpile to minify and save CI minutes.
- Source maps are mandatory or prod errors become unreadable.

## How it works

Modern input:

```ts
export const formatLabel = (name: string) => `Hello, ${name}!`;

export async function loadConfig(url: string) {
  const res = await fetch(url);
  return res.json();
}
```

Downleveled output (conceptual):

```js
export const formatLabel = function (name) {
  return 'Hello, ' + name + '!';
};

export async function loadConfig(url) {
  const res = await fetch(url);
  return res.json();
}
```

Webpack pipeline: `ts-loader` or `babel-loader` in rules → JS output → terser minify. Vite dev: esbuild transform on request (<50ms). Prod Rollup + esbuild minify. JSX:

```tsx
export function Badge({ text }: { text: string }) {
  return <span className="badge">{text}</span>;
}
```

→ `_jsx("span", { className: "badge", children: text })` (automatic runtime).

## Pitfalls

- **Transpile without types.** `esbuild` ignores TS errors; keep a separate CI gate.
- **Polyfill confusion.** Syntax transforms do not add `Promise.allSettled` on IE without core-js.
- **Double transform.** Babel on top of tsc output bloats build time.
- **Helper duplication.** Missing `@babel/runtime` copies `_extends` into every file.
- **Wrong target.** Too low inflates the bundle; too high breaks legacy SLAs.
