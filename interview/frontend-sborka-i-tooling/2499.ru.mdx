# package.json vs package-lock.json

## Суть

`package.json` описывает проект: scripts, metadata и диапазоны версий зависимостей (`^`, `~`). `package-lock.json` (npm) фиксирует exact resolved tree с tarball URL и integrity hash. Вместе дают human intent и reproducible install на CI и у коллег.

## Что ответить на собесе

- Semver range в dependencies: `^1.4.0` допускает minor/patch, не major.
- Lockfile коммитим в app repos; без него `npm install` сегодня и завтра даст разный tree.
- `npm ci` требует lock и чистый node_modules, быстрее и строже чем `npm install` в CI.
- Честно: pnpm использует `pnpm-lock.yaml`, yarn Berry `yarn.lock`; принцип тот же, формат другой.
- Ручное редактирование lock опасно; меняй `package.json`, затем `npm install` и review diff.

## Как это работает

`package.json` fragment:

```json
{
  "name": "catalog-ui",
  "version": "2.1.0",
  "scripts": {
    "dev": "vite",
    "build": "vite build"
  },
  "dependencies": {
    "react": "^18.3.1"
  },
  "devDependencies": {
    "vitest": "^2.0.5"
  }
}
```

Lock entry (упрощённо):

```json
{
  "packages": {
    "node_modules/react": {
      "version": "18.3.1",
      "resolved": "https://registry.npmjs.org/react/-/react-18.3.1.tgz",
      "integrity": "sha512-..."
    }
  }
}
```

Flow: developer adds `express@^4.21`. npm resolves latest matching 4.x, writes exact 4.21.2 into lock including nested `body-parser`. Teammate clone → `npm ci` → identical `node_modules` layout. Security audit (`npm audit`) и Dependabot читают lock для precise CVE matching.

## Подводные камни

- **Lock not committed.** «Works on my machine» из-за transitive major drift.
- **Mixed package managers.** `package-lock` + `pnpm-lock` в одном repo путают команду.
- **Manual lock conflict.** Merge lock без regenerate ломает integrity hashes.
- **Caret surprise.** `^0.2.0` по semver rules не поднимает 0.3 (0.x special).
- **Optional deps platform.** Lock с optional native binary может fail на другой OS без `npm ci --omit=optional`.
