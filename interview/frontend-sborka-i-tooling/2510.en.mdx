# Role of package-lock.json

## In short

`package-lock.json` pins the full resolved dependency tree: exact version, tarball URL, integrity hash, and nested requires. npm creates and updates it on `install`/`update`. Goal: identical `node_modules` on laptop, CI, and prod images for the same lock commit.

## How to answer in an interview

- Commit the lock to VCS; ignore it only for publishable libraries (rare).
- `npm ci` is strict to the lock, wipes `node_modules`, faster in CI than `install`.
- `lockfileVersion` 3 (npm 9+) is a different format; do not mix generators across npm majors.
- Honestly: security patches for transitive deps need `npm audit fix` or overrides, not hand-edited lock lines.
- pnpm/yarn have their own locks; do not run three managers in one repo.

## How it works

After `npm install lodash@^4.17`:

```json
{
  "name": "demo-app",
  "lockfileVersion": 3,
  "packages": {
    "node_modules/lodash": {
      "version": "4.17.21",
      "resolved": "https://registry.npmjs.org/lodash/-/lodash-4.17.21.tgz",
      "integrity": "sha512-v2Tx..."
    }
  }
}
```

CI pipeline:

```bash
npm ci
npm run build
npm test
```

`ci` fails if `package.json` and the lock diverge. Overrides (npm 8.3+):

```json
{
  "overrides": {
    "lodash": "4.17.21"
  }
}
```

force a version across the tree without manually merging thousands of lock lines.

## Pitfalls

- **Deleted lock.** A fresh `npm install` may pull a breaking transitive major overnight.
- **Merge conflicts.** Blind accept-theirs corrupts integrity; regenerate the lock.
- **Global npm update.** Devs on npm 6 vs 10 write different lockfileVersion values.
- **Optional deps.** Locks with `fsevents` on macOS fail Linux CI without `--omit=optional` policy.
- **Phantom drift.** `npm install pkg` without committing the lock breaks reproducibility for the team.
