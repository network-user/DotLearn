# What package.json is for

## In short

`package.json` is the npm/pnpm project manifest: name, version, entry points, scripts, dependency ranges, engines, and exports map. Package managers read it for installs; bundlers and CI read scripts and the `type` field. It is the contract between package authors and consumers (local and registry).

## How to answer in an interview

- `scripts`: `dev`, `build`, `test` standardize commands (`pnpm run build`).
- `dependencies` vs `devDependencies`: runtime vs tooling only.
- `exports` field (Node 16+) controls the public API and subpath imports.
- Honestly: manual version bumps without semver break downstream; use `npm version patch`.
- `workspaces` in a monorepo link packages without publishing each one locally.

## How it works

```json
{
  "name": "@acme/checkout-ui",
  "version": "3.4.1",
  "type": "module",
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "exports": {
    ".": {
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    }
  },
  "scripts": {
    "dev": "vite",
    "build": "tsc -b && vite build",
    "test": "vitest run"
  },
  "dependencies": {
    "react": "^18.3.1"
  },
  "devDependencies": {
    "typescript": "^5.6.2",
    "vite": "^5.4.0"
  },
  "engines": {
    "node": ">=20"
  }
}
```

`pnpm install` resolves ranges and writes a lockfile. npm publish includes only the `files` whitelist (or defaults without dev artifacts). Consumers `import '@acme/checkout-ui'` hit the `exports.import` target.

## Pitfalls

- **Missing engines.** CI on Node 18, local on 22 → subtle native module failures.
- **exports vs main drift.** Older bundlers read only `main` and land on a CJS stub.
- **Script portability.** Unix env vars in scripts break on Windows without cross-env.
- **Private publish.** Forgetting `"private": true` can publish an internal package.
- **Phantom dependencies.** Monorepo hoist lets you import packages not declared in dependencies.
