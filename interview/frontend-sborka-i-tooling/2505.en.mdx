# Babel: transpiling modern JS

## In short

Babel parses modern JS/TS syntax into an AST, runs a plugin pipeline, and emits code for target environments (browserslist). It does not type-check: TypeScript types are stripped separately (`tsc` or `babel-plugin-transform-typescript`). `@babel/preset-env` picks transforms from a supported-features matrix.

## How to answer in an interview

- Plugins are atomic (`@babel/plugin-transform-arrow-functions`); presets group sets.
- `useBuiltIns: 'usage'` with `core-js` injects only needed polyfills per file.
- With Vite/esbuild, Babel is often only for legacy JSX or custom macros, not full transpile.
- Honestly: for evergreen browsers many teams dropped Babel from prod and rely on tsc/swc.
- Source maps: `sourceMaps: true` keeps output tied to originals for debugging.

## How it works

Input (modern):

```ts
const greetVisitor = (name: string) => `Welcome, ${name}!`;

console.log(greetVisitor('Ada'));
```

Babel output (ES5 style):

```js
var greetVisitor = function greetVisitor(name) {
  return 'Welcome, ' + name + '!';
};

console.log(greetVisitor('Ada'));
```

Config:

```json
{
  "presets": [
    [
      "@babel/preset-env",
      {
        "targets": ">0.5%, not dead",
        "modules": false
      }
    ],
    "@babel/preset-typescript"
  ]
}
```

`modules: false` keeps ESM for Rollup/Webpack tree shaking. CLI: `babel src --out-dir dist --extensions ".ts,.tsx"`. Integrates as a `babel-loader` rule or via `@vitejs/plugin-react` with an optional Babel stage for legacy decorators.

## Pitfalls

- **Double transpile.** Babel plus tsc both convert TS → slower CI and confusing maps.
- **Polyfill bloat.** `useBuiltIns: 'entry'` pulls all of core-js if misconfigured.
- **Plugin order.** Some proposals conflict; class properties vs decorators stage mismatch.
- **Runtime helpers.** Missing `@babel/runtime` duplicates helper bytes in every file.
- **False compatibility.** Babel syntax does not guarantee platform APIs (fetch, Promise) without polyfills.
