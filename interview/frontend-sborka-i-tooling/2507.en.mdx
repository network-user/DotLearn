# Vite: dev server and prod build

## In short

In dev, Vite serves sources as native ESM with esbuild pre-bundled deps (one graph for `node_modules`). Each file request is transformed on demand (TS, Vue SFC). Prod build uses Rollup: tree shaking, code split, asset pipeline. HMR updates only the touched module without a full reload.

## How to answer in an interview

- `pnpm dev` starts instantly: no upfront bundle of the whole app graph.
- `@vitejs/plugin-react` / `plugin-vue` wire framework SFC transforms.
- `import.meta.env` and `import.meta.glob` are Vite compile-time features.
- Honestly: very custom Webpack configs with legacy loaders can migrate painfully; greenfield picks Vite.
- Preview: `vite preview` serves built `dist` for smoke checks before deploy.

## How it works

`vite.config.ts`:

```ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  server: { port: 5173 },
  build: {
    target: 'es2020',
    sourcemap: true,
    rollupOptions: {
      output: {
        manualChunks: {
          vendor: ['react', 'react-dom'],
        },
      },
    },
  },
});
```

Dev flow: browser `import App from '/src/App.tsx'` → Vite transpiles TSX → ESM response. Dependency `react` is pre-bundled in `.vite/deps`. HMR websocket sends accept for the changed module.

Prod: Rollup walks `index.html` entry → chunked JS/CSS in `dist/`. Static assets use hashed filenames for cache busting.

## Pitfalls

- **CJS deps.** Legacy packages without ESM can break dev pre-bundle; fix with `optimizeDeps.include`.
- **Env exposure.** Only `VITE_` vars; a forgotten secret prefix still leaks.
- **SSR frameworks.** Nuxt/SvelteKit add their own layer over Vite; do not confuse with vanilla SPA setup.
- **Rollup plugin compat.** Dev-only plugins may fail in prod build without dual config.
- **Base path.** Subpath deploy needs `base: '/app/'` or assets 404.
