# ESM модули в JavaScript

## Суть

ESM (ECMAScript Modules) это стандарт `import`/`export` с compile-time графом зависимостей. Модуль выполняется один раз, exports live bindings для `export const`. Браузер грузит ESM по URL; Node с `"type": "module"` делает то же с file resolution rules.

## Что ответить на собесе

- Named exports: `import { createStore } from './state.js'`.
- Default export один на файл: `import App from './App.js'`.
- Re-export barrel: `export { Button } from './button.js'` (осторожно с tree shaking).
- Честно: CJS interop в Node иногда требует `import pkg from 'x'` vs `import { createRequire }`.
- Top-level await блокирует module evaluation до promise resolve.

## Как это работает

`vector-math.ts`:

```ts
export function dot(ax: number, ay: number, bx: number, by: number) {
  return ax * bx + ay * by;
}

export const ORIGIN = { x: 0, y: 0 };
```

`scene.ts`:

```ts
import { dot, ORIGIN } from './vector-math.js';

const impulse = dot(1, 2, ORIGIN.x, ORIGIN.y);
console.log({ impulse });
```

Browser entry:

```html
<script type="module" src="/src/scene.js"></script>
```

Static imports позволяют bundlerу построить DAG и shake unused exports. Dynamic `import()` создаёт async boundary для code splitting. ESM strict mode: no undeclared globals, `this` undefined at top level.

## Подводные камни

- **Extension in Node.** Relative import без `.js` падает в native ESM (даже из TS compile output).
- **Circular ESM.** Live bindings могут быть TDZ до инициализации partner module.
- **Dual packages.** Library с CJS+ESM entry может duplicate React instance.
- **JSON/CSS import.** Поддержка зависит от runtime/bundler flags (`assert { type: 'json' }`).
- **Barrel bloat.** `export *` из huge index ломает perf и shake.
