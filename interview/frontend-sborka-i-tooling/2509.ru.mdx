# Назначение package.json

## Суть

`package.json` это манифест npm/pnpm проекта: имя, версия, entry points, scripts, dependency ranges, engines, exports map. Package manager читает его для install; bundler и CI читают scripts и `type` field. Это контракт между автором пакета и потребителями (локально и в registry).

## Что ответить на собесе

- `scripts`: `dev`, `build`, `test` стандартизируют команды (`pnpm run build`).
- `dependencies` vs `devDependencies`: runtime vs tooling only.
- `exports` field (Node 16+) контролирует public API и subpath imports.
- Честно: ручное bump версии без semver ломает downstream; используй `npm version patch`.
- `workspaces` в monorepo связывает пакеты без publish каждого локально.

## Как это работает

```json
{
  "name": "@acme/checkout-ui",
  "version": "3.4.1",
  "type": "module",
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "exports": {
    ".": {
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    }
  },
  "scripts": {
    "dev": "vite",
    "build": "tsc -b && vite build",
    "test": "vitest run"
  },
  "dependencies": {
    "react": "^18.3.1"
  },
  "devDependencies": {
    "typescript": "^5.6.2",
    "vite": "^5.4.0"
  },
  "engines": {
    "node": ">=20"
  }
}
```

`pnpm install` резолвит ranges и пишет lockfile. Publish в npm включает только `files` whitelist (или default без dev artifacts). Consumers `import '@acme/checkout-ui'` попадают в `exports.import` target.

## Подводные камни

- **Missing engines.** CI на Node 18, local на 22 → subtle native module failures.
- **exports vs main drift.** Старые bundlers читают только `main`, попадают в CJS stub.
- **Script portability.** Unix env vars в scripts ломаются на Windows без cross-env.
- **Private publish.** Забытый `"private": true` случайно публикует internal пакет.
- **Phantom dependencies.** Hoist из monorepo позволяет import пакета не объявленного в dependencies.
