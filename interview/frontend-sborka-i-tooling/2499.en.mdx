# package.json vs package-lock.json

## In short

`package.json` describes the project: scripts, metadata, and dependency version ranges (`^`, `~`). `package-lock.json` (npm) pins the exact resolved tree with tarball URLs and integrity hashes. Together they provide human intent and reproducible installs on CI and for teammates.

## How to answer in an interview

- Semver ranges in dependencies: `^1.4.0` allows minor/patch, not major.
- Commit the lockfile in app repos; without it, `npm install` today vs tomorrow yields different trees.
- `npm ci` requires the lock and a clean `node_modules`, faster and stricter than `npm install` in CI.
- Honestly: pnpm uses `pnpm-lock.yaml`, Yarn Berry uses `yarn.lock`; same idea, different format.
- Manual lock edits are risky; change `package.json`, then `npm install` and review the diff.

## How it works

`package.json` fragment:

```json
{
  "name": "catalog-ui",
  "version": "2.1.0",
  "scripts": {
    "dev": "vite",
    "build": "vite build"
  },
  "dependencies": {
    "react": "^18.3.1"
  },
  "devDependencies": {
    "vitest": "^2.0.5"
  }
}
```

Lock entry (simplified):

```json
{
  "packages": {
    "node_modules/react": {
      "version": "18.3.1",
      "resolved": "https://registry.npmjs.org/react/-/react-18.3.1.tgz",
      "integrity": "sha512-..."
    }
  }
}
```

Flow: developer adds `express@^4.21`. npm resolves the latest matching 4.x and writes exact 4.21.2 into the lock including nested `body-parser`. Teammate clone → `npm ci` → identical `node_modules` layout. Security audit (`npm audit`) and Dependabot read the lock for precise CVE matching.

## Pitfalls

- **Lock not committed.** "Works on my machine" from transitive major drift.
- **Mixed package managers.** `package-lock` plus `pnpm-lock` in one repo confuses the team.
- **Manual lock conflict.** Merging the lock without regenerate breaks integrity hashes.
- **Caret surprise.** `^0.2.0` by semver rules does not bump to 0.3 (0.x special case).
- **Optional deps platform.** Locks with optional native binaries may fail on another OS without `npm ci --omit=optional`.
