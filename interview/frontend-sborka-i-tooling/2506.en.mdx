# ESM modules in JavaScript

## In short

ESM (ECMAScript Modules) is the standard `import`/`export` system with a compile-time dependency graph. A module runs once; exports are live bindings for `export const`. Browsers load ESM by URL; Node with `"type": "module"` does the same with file resolution rules.

## How to answer in an interview

- Named exports: `import { createStore } from './state.js'`.
- One default export per file: `import App from './App.js'`.
- Re-export barrels: `export { Button } from './button.js'` (watch tree shaking).
- Honestly: CJS interop in Node may need `import pkg from 'x'` vs `import { createRequire }`.
- Top-level await blocks module evaluation until the promise resolves.

## How it works

`vector-math.ts`:

```ts
export function dot(ax: number, ay: number, bx: number, by: number) {
  return ax * bx + ay * by;
}

export const ORIGIN = { x: 0, y: 0 };
```

`scene.ts`:

```ts
import { dot, ORIGIN } from './vector-math.js';

const impulse = dot(1, 2, ORIGIN.x, ORIGIN.y);
console.log({ impulse });
```

Browser entry:

```html
<script type="module" src="/src/scene.js"></script>
```

Static imports let bundlers build a DAG and shake unused exports. Dynamic `import()` creates an async boundary for code splitting. ESM strict mode: no undeclared globals, `this` is undefined at top level.

## Pitfalls

- **Extension in Node.** Relative imports without `.js` fail in native ESM (even from TS compile output).
- **Circular ESM.** Live bindings may sit in TDZ until the partner module initializes.
- **Dual packages.** A library with CJS+ESM entries can duplicate the React instance.
- **JSON/CSS import.** Support depends on runtime/bundler flags (`assert { type: 'json' }`).
- **Barrel bloat.** `export *` from a huge index hurts perf and shaking.
