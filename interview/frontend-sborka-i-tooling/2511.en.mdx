# Profiler in Chrome DevTools

## In short

The Performance/React Profiler in DevTools records execution timelines: JS call stacks, layout, paint, and long tasks. React DevTools Profiler captures component commit phases (render vs layout effects). The goal is to find bottlenecks (extra re-renders, heavy main thread) with evidence, not guesses.

## How to answer in an interview

- Performance panel: Record → reproduce jank → read the Main thread flame chart.
- Long tasks over 50ms block input; look for forced sync layout (`offsetWidth` after a write).
- React Profiler: highlights why a component rendered (props/state/context change).
- Honestly: profile on throttled CPU with a production build; dev HMR skews numbers.
- The Memory tab is separate for leaks; Performance does not replace heap snapshots.

## How it works

Suspect heavy loop:

```ts
export function computeScoreTable(size: number) {
  let acc = 0;
  for (let i = 0; i < size; i += 1) {
    acc += Math.hypot(i, size - i);
  }
  return acc;
}
```

Workflow:
1. DevTools → Performance → enable Web Vitals overlays.
2. Start recording, trigger UI calling `computeScoreTable(1_000_000)`.
3. Stop: yellow Long Task on Main, function in Bottom-Up tab.
4. Fix: offload to a Web Worker or shrink input plus memoization.

React path: Components → Profiler → record → rank by "self duration" → inspect "why did this render". The flamegraph shows parent chains, not only leaves.

## Pitfalls

- **Profiling overhead.** Instrumentation adds noise; compare short frames relatively.
- **Anonymous functions.** Minified names are useless; enable source maps or name inference.
- **Idle misread.** GPU raster may sit on another track; read Rendering + GPU together.
- **Single sample.** One trace is not statistics; repeat 3 times warm and cold.
- **Fix wrong layer.** Memo tuning when the bottleneck is a network waterfall.
