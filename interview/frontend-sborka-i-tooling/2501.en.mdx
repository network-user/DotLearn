# Tree shaking in JS bundles

## In short

Tree shaking drops unused exports from production bundles using the static import graph. It works with ESM (`import { onlyUsed } from 'lib'`) because imports are analyzed before execution. Dead code elimination (Terser/esbuild) further shrinks the remaining AST.

## How to answer in an interview

- Named imports let the bundler discard other exports from the module.
- `import pkg from 'lodash'` pulls the whole package; prefer `lodash-es` or per-method paths.
- `sideEffects: false` in a library package.json allows aggressive shaking.
- Honestly: CommonJS dynamic `require` and barrel files with side effects break shaking.
- Production mode (`mode: 'production'`, `build.minify`) enables shake plus minify pipeline.

## How it works

Library module:

```ts
export function addPair(a: number, b: number) {
  return a + b;
}

export function subtractPair(a: number, b: number) {
  return a - b;
}
```

App entry:

```ts
import { addPair } from './math-helpers';

console.log(addPair(2, 3));
```

Bundler graph: only `addPair` is reachable, so `subtractPair` is dropped from the emitted chunk. Webpack 5, Rollup, and esbuild support module-level shake. Package metadata:

```json
{
  "name": "math-helpers",
  "sideEffects": false,
  "module": "./dist/index.esm.js"
}
```

Terser `usedExports: true` marks unused symbols within a file. Vite uses Rollup for prod; dev serves native ESM without full shake until build.

## Pitfalls

- **Barrel re-export.** `export * from './icons'` pulls hundreds of components for one import.
- **Side effect imports.** `import 'polyfill'` cannot be removed even if it looks unused.
- **eval / dynamic import templates.** `` import(`./locales/${lang}`) `` limits static analysis.
- **False confidence.** Shaking shrinks JS but does not fix heavy assets/images by itself.
- **CJS deps.** Legacy npm packages without an ESM build stay whole in the bundle.
