# script type="module" in the browser

## In short

`<script type="module">` tells the browser to load the script as an ES module: strict mode, its own scope, static `import`/`export`, deferred execution (after HTML parse). Relative imports need full URL paths with extensions. Classic `<script>` without `type` cannot use `import` syntax.

## How to answer in an interview

- Modules defer automatically: they do not block HTML parse like blocking classic scripts without `defer`.
- Top-level `await` is allowed in the module graph.
- CORS: module scripts treat cross-origin loads as CORS resources (correct headers required).
- Honestly: prod almost always uses a bundler (Vite/Webpack); native modules appear in dev and tiny demos.
- `importmap` (`<script type="importmap">`) maps bare specifiers to URLs without a bundler.

## How it works

`index.html`:

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8" />
    <title>ESM demo</title>
  </head>
  <body>
    <script type="module" src="/src/bootstrap.js"></script>
  </body>
</html>
```

`src/math-helpers.js`:

```ts
export function mergeTotals(a: number, b: number) {
  return a + b;
}
```

`src/bootstrap.js`:

```ts
import { mergeTotals } from './math-helpers.js';

console.log(mergeTotals(4, 9));
```

Browser fetch chain: `bootstrap.js` → parse static import → fetch `math-helpers.js` → evaluate in dependency order. Inline module:

```html
<script type="module">
  import { mergeTotals } from './src/math-helpers.js';
</script>
```

## Pitfalls

- **Missing extension.** `./math-helpers` 404s in browser ESM though bundlers forgive it.
- **File protocol.** `file://` breaks module CORS; you need a dev server.
- **Mixed classic + module order.** Classic side-effect scripts before modules can surprise init order.
- **Duplicate loads.** Two entry modules import the same URL: browser cache dedupes, but watch side effects.
- **Workers need type.** Workers require `type: 'module'` in constructor options for `import`.
