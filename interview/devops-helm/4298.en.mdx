# What are the advantages of Helm?

## In short

Helm packages a Kubernetes app into a reusable chart: one source of truth for Deployment, Service, Ingress, ConfigMap, HPA. Versioned release revisions, per-environment values, and dependency management cut YAML copy-paste and speed repeatable deploys. Platform teams use it as the standard way to ship internal charts.

## How to answer in an interview

- DRY: template plus `values-dev.yaml` / `values-prod.yaml` instead of duplicated manifests.
- Revision history and `helm rollback` for controlled recovery.
- Subchart dependencies (`Chart.yaml` dependencies) bundle app + redis + ingress.
- Hooks (pre-install, post-upgrade) for migrations and smoke tests.
- Honestly: without lint/template CI and pre-upgrade diff, Helm only breaks prod faster.

## How it works

Multi-env deploy of one chart:

```bash
helm upgrade --install api ./charts/api \
  -n staging -f values-staging.yaml --wait

helm upgrade --install api ./charts/api \
  -n prod -f values-prod.yaml --atomic --wait --timeout 15m
```

Dependencies and lock file:

```bash
helm dependency update ./charts/api
cat ./charts/api/Chart.lock
```

Rollback after a failed release:

```bash
helm history api -n prod
helm rollback api 22 -n prod --wait
```

Publish chart to OCI:

```bash
helm package ./charts/api
helm push api-1.4.0.tgz oci://registry.internal/helm-charts
```

## Pitfalls

- **Values sprawl.** Dozens of `--set` flags in pipelines are not reproducible; use git-tracked values files.
- **Subchart pin drift.** `helm dependency update` without review pulls breaking subcharts.
- **Non-idempotent hooks.** Re-run post-upgrade hooks duplicate migrations; add guards.
- **Global chart anti-pattern.** One chart for everything with boolean flags becomes unmaintainable.
- **Imperative only.** Helm CLI without GitOps diverges from cluster drift when someone kubectl-edits by hand.
