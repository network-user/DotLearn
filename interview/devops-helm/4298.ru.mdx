# В чём преимущества Helm?

## Суть

Helm упаковывает Kubernetes-приложение в переиспользуемый chart: один источник правды для Deployment, Service, Ingress, ConfigMap, HPA. Версионируемые release revision'ы, values per environment и dependency management снижают copy-paste YAML и ускоряют repeatable deploy. Для платформенных команд это стандартный способ дистрибуции internal charts.

## Что ответить на собесе

- DRY: шаблон + `values-dev.yaml` / `values-prod.yaml` вместо дублирования манифестов.
- Revision history и `helm rollback` для controlled recovery.
- Subchart dependencies (`Chart.yaml` dependencies) собирают app + redis + ingress.
- Hooks (pre-install, post-upgrade) для миграций и smoke test.
- Честно: без lint/template CI и diff перед upgrade Helm только быстрее ломает prod.

## Как это работает

Multi-env deploy одного chart:

```bash
helm upgrade --install api ./charts/api \
  -n staging -f values-staging.yaml --wait

helm upgrade --install api ./charts/api \
  -n prod -f values-prod.yaml --atomic --wait --timeout 15m
```

Зависимости и lock:

```bash
helm dependency update ./charts/api
cat ./charts/api/Chart.lock
```

Откат после failed release:

```bash
helm history api -n prod
helm rollback api 22 -n prod --wait
```

Публикация chart в OCI:

```bash
helm package ./charts/api
helm push api-1.4.0.tgz oci://registry.internal/helm-charts
```

## Подводные камни

- **Values sprawl.** Десятки `--set` в pipeline не воспроизводимы; всё в git tracked values files.
- **Subchart pin drift.** `helm dependency update` без review тянет breaking subchart.
- **Hook без idempotency.** Повторный post-upgrade hook дублирует миграции; нужны guards.
- **Global chart anti-pattern.** Один chart на всё с boolean flags становится неподдерживаемым.
- **Imperative only.** Helm CLI без GitOps расходится с cluster drift; кто-то правит kubectl вручную.
