# What are Helm and a chart?

## In short

Helm is the package manager for Kubernetes: install, upgrade, and remove apps with one command. A chart is a versioned package (templates + values + metadata) Helm renders into manifests and records as a release in a namespace. It sits between hundreds of YAML files and parameterized deploys for dev/stage/prod.

## How to answer in an interview

- Chart = `Chart.yaml` + `values.yaml` + `templates/` (+ optional `charts/` deps).
- `helm install <release> <chart>` creates revision 1 in the cluster.
- `helm upgrade` bumps the revision; `helm uninstall` removes resources (respecting hooks/policy).
- OCI registry or HTTP repo (`helm repo add`) distributes charts.
- Honestly: Helm does not replace a GitOps controller; in prod I often pair charts with Argo CD/Flux sync.

## How it works

Create skeleton and install:

```bash
helm create payments-api
helm lint ./payments-api

helm install payments ./payments-api \
  -n prod --create-namespace \
  -f values-prod.yaml \
  --set image.tag=2.1.0 \
  --wait
```

Chart layout after `helm create`:

```bash
find payments-api -maxdepth 2 -type f | sort
# payments-api/Chart.yaml
# payments-api/values.yaml
# payments-api/templates/deployment.yaml
# payments-api/templates/service.yaml
# payments-api/templates/_helpers.tpl
```

Upgrade and remove:

```bash
helm upgrade payments ./payments-api -n prod -f values-prod.yaml --atomic --wait
helm list -n prod
helm uninstall payments -n prod
```

## Pitfalls

- **Release name collision.** Two installs with the same release name in a namespace overwrite history; names are unique per namespace.
- **Orphan resources.** `helm.sh/resource-policy: keep` leaves PVCs after uninstall.
- **Values merge surprise.** Multiple `-f` files override keys with the last file; easy to miss in CI.
- **Charts from random repos.** Without signature verification you accept supply-chain risk; pin OCI digests.
- **Tiller legacy confusion.** Helm 2 tiller vs Helm 3 client-only; I clarify 3.x in interviews.
