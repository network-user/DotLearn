# How microservices talk to each other: synchronous calls vs queues

## In short

The ways services communicate come down to two axes - synchronous or asynchronous exchange. The synchronous options are REST (over HTTP, simple and universal) and gRPC (over HTTP/2 with Protobuf, fast for high-load scenarios). The asynchronous one is message queues, which separate sending and processing in time and improve resilience to failures.

## How to answer in an interview

"I start picking a communication style with one question: do I need a response right now? If yes - it's a synchronous call. REST over HTTP fits almost everywhere: simple, readable, debuggable from the browser, but verbose and slower. gRPC over HTTP/2 with Protocol Buffers gives low latency and compact messages, with typed contracts from a .proto file - I reach for it for internal high-load calls. If I don't need a response right now, or decoupling the services matters, I take a message queue: the producer drops a message, the consumer picks it up when it's ready. This buys peak buffering, retries, and fault tolerance, but we pay for it with complexity and with the result becoming eventual rather than instant."

Microservices are a set of independent services, each with its own area of responsibility. For them to behave as one system, they need a communication channel. Three working options:

**REST** - a style over HTTP. Every resource is addressed by a URL, actions are expressed with the GET, POST, PUT, DELETE methods. The orders service synchronously calls the users service to fetch the customer's profile:

```python
import urllib.request, json

def fetch_user(user_id):
    url = f"http://users-svc/users/{user_id}"
    with urllib.request.urlopen(url) as resp:
        return json.load(resp)
```

**gRPC** - an RPC framework from Google. The contract is described in a .proto file, from which client and server code is generated, so a remote method is called like a local one. The HTTP/2 transport provides multiplexing and streaming, and Protobuf provides compact binary serialization. This is the profile for internal communication where speed and a strict contract matter.

**Message queue** (RabbitMQ, Kafka) - an asynchronous channel. The producer publishes a message into the queue, the consumer picks it up at its own pace:

```python
import queue

orders = queue.Queue()

def producer(order_id):
    orders.put(order_id)

def consumer():
    while not orders.empty():
        order_id = orders.get()
        print(f"processing order {order_id}")

producer(1)
producer(2)
consumer()
```

## Pitfalls

- A synchronous call creates temporal coupling: if the callee service goes down, the caller goes down too. Without timeouts, retries, and a circuit breaker, one slow service stalls the whole chain.
- gRPC speeds up internal exchange but lives poorly in the browser directly (you need gRPC-Web/a proxy) and is harder to debug than human-readable JSON in REST.
- A queue doesn't cancel errors, it shifts them in time: duplicate messages appear, ordering can break, and you need idempotency and a dead-letter queue.
- Don't confuse REST as a style with "any HTTP API". A GET that mutates data on the server is already a violation of method semantics.
