# Вертикальное и горизонтальное масштабирование: чем отличаются и когда что брать?

## Суть

Когда нагрузка растёт, у системы два пути: усилить один сервер (scale-up) или поставить рядом много серверов и раскидать нагрузку между ними (scale-out). Первый путь проще внедрить, но упирается в потолок железа; второй сложнее, зато почти безграничен и устойчив к отказам.

## Что ответить на собесе

Есть два базовых подхода. Вертикальное масштабирование - это наращивание ресурсов одной машины: добавляем RAM, CPU, диск. Плюс - не нужно трогать архитектуру приложения, минус - есть физический потолок и обычно простой при апгрейде. Горизонтальное масштабирование - это добавление новых узлов и распределение нагрузки между ними через балансировщик. Плюс - масштабируется почти линейно и переживает отказ отдельных узлов, минус - приложение должно быть готово к распределённой работе: stateless-сервисы, общее хранилище состояния, согласованность данных. На практике монолиты тяготеют к вертикали, а распределённые системы и микросервисы - к горизонтали, и часто их комбинируют.

Вертикальное масштабирование (scale-up) меняет одну ось: тот же сервер, но мощнее. Приложение не знает, что под ним стало больше ядер и памяти, поэтому код не меняется. Проблема в том, что цена ресурса растёт нелинейно, а максимум упирается в самое мощное доступное железо.

Горизонтальное масштабирование (scale-out) добавляет узлы и ставит перед ними балансировщик нагрузки, который распределяет запросы. Ключевое требование - сервис должен быть **stateless**: любое состояние (сессии, кэш) выносится во внешнее хранилище (Redis, БД), иначе запросы пользователя начнут попадать на разные узлы и терять контекст.

Простая иллюстрация горизонтального распределения запросов по узлам:

```python
class RoundRobinBalancer:
    def __init__(self, nodes):
        self.nodes = nodes
        self.index = 0

    def next_node(self):
        node = self.nodes[self.index % len(self.nodes)]
        self.index += 1
        return node


balancer = RoundRobinBalancer(["node-1", "node-2", "node-3"])
route = [balancer.next_node() for _ in range(4)]
print(route)
```

Вывод: `['node-1', 'node-2', 'node-3', 'node-1']` - запросы циклически расходятся по узлам.

## Подводные камни

- **Stateful-сервис не масштабируется горизонтально без боли.** Если сессии лежат в памяти процесса, балансировщик с round-robin начнёт разлогинивать пользователей. Выносите состояние наружу или используйте sticky-сессии (но это снижает равномерность).
- **Вертикальный апгрейд - это часто простой.** Замена железа или ресайз инстанса обычно требует перезапуска. Для систем с высокими требованиями к доступности это критично.
- **Горизонталь добавляет проблему согласованности данных.** Несколько узлов и реплик БД - сразу всплывают вопросы из CAP-теоремы и распределённых транзакций.
- **Балансировщик становится единой точкой отказа.** Его самого нужно резервировать, иначе горизонтальное масштабирование сервисов не спасёт.
- **Не всё распределяется.** Тяжёлый монолит с общим состоянием в памяти иногда проще и дешевле масштабировать вертикально, чем переписывать под scale-out.
