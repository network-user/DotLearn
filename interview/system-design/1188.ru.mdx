# Как микросервисы договариваются между собой: синхронные вызовы против очередей

## Суть

Способы межсервисного общения сводятся к двум осям - синхронный или асинхронный обмен. Синхронно работают REST (поверх HTTP, простой и универсальный) и gRPC (поверх HTTP/2 с Protobuf, быстрый для нагруженных сценариев). Асинхронно - очереди сообщений, которые разносят отправку и обработку во времени и повышают устойчивость к сбоям.

## Что ответить на собесе

Выбор способа связи начинаю с вопроса: нужен ли ответ прямо сейчас.

- **REST поверх HTTP** - синхронный вызов, подходит почти везде: просто, читаемо, отлаживается из браузера, но многословно и медленнее.
- **gRPC поверх HTTP/2 с Protocol Buffers** - синхронный вызов с низкой задержкой и компактными сообщениями, типизированные контракты из .proto; беру для внутренних высоконагруженных вызовов.
- **Очередь сообщений** - когда ответ прямо сейчас не нужен или важна развязка сервисов: продюсер кладёт сообщение, консьюмер забирает его, когда готов. Даёт буферизацию пиков, ретраи и отказоустойчивость.
- Цена очереди - сложность и то, что результат становится eventual, а не мгновенным.

Микросервисы - это набор самостоятельных сервисов, каждый со своей зоной ответственности. Чтобы они вели себя как единая система, нужен канал обмена. Три рабочих варианта:

**REST** - стиль поверх HTTP. Каждый ресурс адресуется URL, действия выражаются методами GET, POST, PUT, DELETE. Сервис заказов синхронно дёргает сервис пользователей, чтобы подтянуть профиль клиента:

```python
import urllib.request, json

def fetch_user(user_id):
    url = f"http://users-svc/users/{user_id}"
    with urllib.request.urlopen(url) as resp:
        return json.load(resp)
```

**gRPC** - RPC-фреймворк от Google. Контракт описывается в .proto, из него генерируется код клиента и сервера, поэтому удалённый метод вызывается как локальный. Транспорт HTTP/2 даёт мультиплексирование и стриминг, а Protobuf - компактную бинарную сериализацию. Это профиль для внутреннего общения, где важны скорость и строгий контракт.

**Очередь сообщений** (RabbitMQ, Kafka) - асинхронный канал. Продюсер публикует сообщение в очередь, консьюмер забирает его в своём темпе:

```python
import queue

orders = queue.Queue()

def producer(order_id):
    orders.put(order_id)

def consumer():
    while not orders.empty():
        order_id = orders.get()
        print(f"processing order {order_id}")

producer(1)
producer(2)
consumer()
```

## Подводные камни

- Синхронный вызов создаёт временную связанность: если вызываемый сервис лёг, падает и вызывающий. Без таймаутов, ретраев и circuit breaker один медленный сервис тормозит цепочку.
- gRPC ускоряет внутренний обмен, но плохо живёт в браузере напрямую (нужен gRPC-Web/прокси) и тяжелее отлаживается, чем человекочитаемый JSON в REST.
- Очередь не отменяет ошибки, а сдвигает их во времени: появляются дубликаты сообщений, порядок может нарушаться, нужна идемпотентность и dead-letter очередь.
- Не путайте REST как стиль с «любым HTTP-API». GET, который меняет данные на сервере, - это уже нарушение семантики методов.
