# API Gateway: зачем системе единая точка входа перед микросервисами

## Суть

API Gateway - это входной сервис, через который проходят все клиентские запросы к микросервисам. Он скрывает внутреннее устройство системы за единым адресом и берёт на себя сквозные задачи: маршрутизацию, аутентификацию, агрегацию ответов, кэширование и сбор метрик.

## Что ответить на собесе

API Gateway - единая точка входа перед роем микросервисов: клиент шлёт один запрос на шлюз вместо того, чтобы знать адреса десятков сервисов и собирать данные сам.

- Шлюз маршрутизирует запрос в нужный сервис, проверяет аутентификацию и авторизацию на входе, при необходимости агрегирует ответы нескольких сервисов в один, кэширует частые ответы и пишет метрики с логами.
- Это упрощает клиента, скрывает внутреннюю топологию и выносит сквозную логику - безопасность, троттлинг, мониторинг - в одно место.
- Минус: шлюз становится единой точкой отказа и узким местом, поэтому его держат отказоустойчивым и не утяжеляют бизнес-логикой.

1. **Приём.** Клиент отправляет запрос на единый адрес шлюза.
2. **Аутентификация и авторизация.** Шлюз проверяет токен и право на действие - до сервисов доходят только проверенные запросы.
3. **Маршрутизация.** По пути URL или заголовкам шлюз выбирает целевой сервис. Внутренняя топология остаётся скрытой от клиента.
4. **Агрегация.** Если ответу нужны данные из нескольких сервисов, шлюз дёргает их и собирает единый результат.
5. **Ответ и кэш.** Клиент получает результат; частые ответы можно отдавать из кэша, снижая нагрузку.

```python
ROUTES = {
    "restaurants": "http://restaurants-svc",
    "menus": "http://menus-svc",
    "promos": "http://promos-svc",
}

def resolve(path):
    head = path.strip("/").split("/")[0]
    return ROUTES.get(head, "404")

def aggregate(parts):
    return {p: ROUTES[p] for p in parts if p in ROUTES}

print(resolve("/menus/42"))
print(aggregate(["restaurants", "promos"]))
```

В примере с доставкой еды клиенту нужны рестораны, меню и акции. Без шлюза - три запроса к трём сервисам; со шлюзом - один запрос, а агрегацию делает gateway.

## Подводные камни

- Шлюз - единая точка отказа и потенциальное узкое место. Его держат в нескольких инстансах за балансировщиком, иначе падение шлюза кладёт всю систему.
- Соблазн сложить в gateway бизнес-логику. Это превращает «тонкий» шлюз в новый монолит. В шлюзе - только сквозные задачи: роутинг, auth, агрегация, кэш, метрики.
- Не путайте API Gateway с балансировщиком нагрузки. Балансировщик распределяет трафик между копиями одного сервиса; шлюз маршрутизирует по разным сервисам и работает на уровне приложения (L7).
- Агрегация на шлюзе экономит круги клиенту, но добавляет связанность: шлюз начинает знать про контракты многих сервисов. Для сложных случаев иногда выносят отдельный BFF (backend for frontend).
