# The CAP theorem: why a distributed system can't have it all at once?

## In short

The CAP theorem states that of the three properties of a distributed system - Consistency, Availability, and Partition Tolerance - you can guarantee only two simultaneously. Since network partitions are inevitable in a real distributed system, the choice essentially boils down to one thing: whether to sacrifice consistency for availability or the other way around.

## How to answer in an interview

The CAP theorem (Brewer's theorem, 2000) says a distributed system can't simultaneously provide consistency, availability, and partition tolerance - at most two of the three. The subtlety is that in a distributed system a network partition happens on its own, so P can't be dropped. That means the real choice is between CP and AP. A CP system, during a network split, prefers to stop and not serve potentially stale data (this is what banking systems do, for example). An AP system keeps answering, but the data on the nodes temporarily diverges (this is how social networks and feeds work). The CA option exists only in theory - in a system without network partitions, which is effectively a single node.

## How it works

CAP describes the fundamental constraints that any system spread across multiple nodes runs into.

Three properties:

1. **Consistency.** On any request the system returns either the freshest data or an error - all nodes see the same state at one moment.
2. **Availability.** Every request gets a response without an error, even if some nodes are unavailable.
3. **Partition Tolerance.** The system keeps working under arbitrary breaks in communication between nodes.

The conflict arises precisely at the moment of a network split. The link between nodes has broken - and the system must choose one of two reactions:

- **CP:** refuse to answer (or return an error) until consistency is restored. Availability is temporarily lost.
- **AP:** keep answering on each node with its own data. Availability is preserved, but the nodes are temporarily inconsistent (eventual consistency).
- **CA:** C and A together are possible only in the absence of splits - that is, in practice, in a non-distributed system.

Where you see this: a bank transfer chooses CP - better to refuse the operation than to debit money twice. A social feed chooses AP - the user cares more about seeing at least something than getting an error, and a small freshness lag is non-critical.

## Pitfalls

- **"Two of three" is a simplification.** P in a real distributed system isn't optional: network failures always happen. So the honest choice isn't "any two", but only CP vs AP.
- **CA is almost a myth.** A CA system can be had only where network partitions never occur, which is effectively a single node without distribution.
- **Consistency isn't binary.** Between strong and eventual consistency there are intermediate levels (read-your-writes, monotonic reads); CAP coarsens them.
- **The choice is made for the duration of the split, not forever.** When the network is fine, the system can be both consistent and available. The sacrifice only kicks in at the moment of a partition - the PACELC model refines this.
- **Confusing the C in CAP with the C in ACID.** These are different concepts: in CAP it's about replica consistency, in ACID it's about invariants within a transaction.
