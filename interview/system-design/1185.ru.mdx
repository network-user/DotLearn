# Назовите ходовые архитектурные стили и объясните, какую задачу решает каждый

## Суть

Архитектурный стиль - это типовой способ нарезать систему на части и описать, как эти части общаются. Чаще всего на собесе ждут семейство UI-паттернов (MVC, MVP, MVVM) и набор системных стилей: клиент-сервер, микросервисы, P2P, событийная архитектура. Каждый закрывает свою боль - разделение ответственности, тестируемость или масштабирование.

## Что ответить на собесе

Делю стили на две группы и показываю, какую боль закрывает каждая:

- **UI-паттерны (MVC, MVP, MVVM)** отличаются тем, кто играет роль посредника между данными и экраном - контроллер, presenter или viewmodel с биндингом.
- **Системные стили:** клиент-сервер как база, микросервисы для независимого деплоя и масштабирования, P2P для распределённой нагрузки без центрального узла, событийная архитектура для асинхронной реакции на изменения.
- Выбор стиля диктуется не модой, а тем, какое свойство системы нужно усилить: тестируемость, независимость частей или отказоустойчивость.

## Как это работает

UI-семейство (MVC / MVP / MVVM) объединяет одна идея - вынести данные и бизнес-логику (Model) отдельно от отображения (View). Разница в третьем участнике:

- **MVC**: Controller принимает ввод, меняет Model, выбирает View. View может читать Model напрямую.
- **MVP**: Presenter держит всю логику отображения, View максимально пассивна - её легко подменить заглушкой в тестах.
- **MVVM**: ViewModel выставляет наблюдаемые свойства, а View связывается с ними через двусторонний биндинг. Изменилось поле - экран обновился сам.

Системные стили решают другую задачу - как раскидать функциональность по процессам и машинам:

- **Клиент-сервер**: логика и данные централизованы на сервере, клиент шлёт запросы.
- **Микросервисы**: приложение - набор мелких независимых сервисов (платежи, каталог, пользователи), каждый деплоится и масштабируется сам.
- **P2P**: все узлы равноправны и одновременно клиенты и серверы, центральной точки нет (торренты).
- **Событийная**: компоненты реагируют на события, обмен асинхронный, что удобно для систем реального времени.

```python
class OrderModel:
    def __init__(self):
        self.items = []

    def add(self, item):
        self.items.append(item)

class OrderController:
    def __init__(self, model):
        self.model = model

    def handle_add(self, item):
        self.model.add(item)
        return f"items: {len(self.model.items)}"
```

## Подводные камни

- MVC, MVP и MVVM - это паттерны слоя представления, а микросервисы и клиент-сервер - стили всей системы. Смешивать их в одном списке как «равнозначные» - частая ошибка на собесе.
- MVVM без фреймворка с биндингом (WPF, Vue, SwiftUI) превращается в ручную писанину - сам по себе паттерн биндинг не даёт.
- Микросервисы добавляют сетевые вызовы, распределённые транзакции и сложность эксплуатации. Для маленького проекта монолит почти всегда дешевле.
- «P2P» иногда путают с «client-server с несколькими серверами». Ключевой признак P2P - отсутствие выделенного сервера, узлы симметричны.
