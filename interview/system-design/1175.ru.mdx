# Kafka против RabbitMQ: когда брать лог событий, а когда брокер очередей

## Суть

Kafka и RabbitMQ обе перекладывают сообщения между сервисами, но решают разные задачи. Kafka - это распределённый лог: события дописываются в конец, лежат на диске и доступны для повторного чтения, отсюда высокая пропускная способность и хорошая основа для потоковой обработки. RabbitMQ - это брокер очередей: его сила в гибкой маршрутизации через обменники и в строгих гарантиях доставки конкретного сообщения конкретному потребителю.

Грубое правило: нужен поток событий с возможностью перечитать историю - Kafka; нужна умная доставка задач с подтверждениями и сложным роутингом - RabbitMQ.

## Что ответить на собесе

Сформулируй разницу через модель данных, а не через список фич:

- **Kafka хранит лог.** Сообщение не удаляется после чтения - его читают по смещению (offset), а ретеншн определяется временем или размером. Несколько групп потребителей читают один топик независимо. Это даёт реплей, аналитику и высокий throughput.
- **RabbitMQ доставляет в очередь.** Сообщение живёт, пока его не заберут и не подтвердят (ack), после чего оно исчезает. Маршрут задают exchange (direct, fanout, topic, headers) и привязки очередей.
- **Масштабирование.** В Kafka параллелизм упирается в число партиций топика. В RabbitMQ - в число потребителей на очереди и схему шардирования.
- **Когда что брать.** Kafka - стриминг логов, метрик, event sourcing, аналитика в реальном времени. RabbitMQ - очереди задач, RPC, интеграция систем с тонкой маршрутизацией и приоритетами.

Короткий вывод для интервьюера: "Kafka - это переигрываемый лог под высокую нагрузку, RabbitMQ - брокер с богатым роутингом и доставкой по одному сообщению".

## Как это работает

**Kafka.** Producer пишет в топик, топик разбит на партиции, партиции раскиданы по брокерам кластера - так растёт пропускная способность и доступность. Внутри партиции порядок строгий, между партициями - нет. Потребитель помнит свой offset, поэтому может перечитать поток с любой точки.

```python
from kafka import KafkaProducer

producer = KafkaProducer(bootstrap_servers="localhost:9092")
producer.send("events", b"order-created")
producer.flush()
producer.close()
```

**RabbitMQ.** Producer публикует в exchange, exchange по routing key и привязкам решает, в какие очереди положить сообщение. Потребитель забирает из очереди и отправляет ack; до подтверждения сообщение считается недоставленным.

```python
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters("localhost"))
channel = connection.channel()
channel.queue_declare(queue="tasks")
channel.basic_publish(exchange="", routing_key="tasks", body="resize-image")
connection.close()
```

## Подводные камни

- **"Kafka - это очередь".** Нет, это лог. Сообщение не пропадает после чтения, поэтому модель потребления (offset, группы) принципиально иная.
- **Порядок сообщений.** В Kafka он гарантирован только внутри партиции. Если важен глобальный порядок по ключу - используй один ключ партиционирования, иначе порядок разъедется.
- **Удаление после прочтения.** В RabbitMQ забыли ack - сообщение вернётся в очередь и обработается повторно. Логика потребителя должна быть идемпотентной.
- **Перепутанные сценарии.** Гонять короткоживущие задачи с приоритетами через Kafka неудобно; строить аналитику с реплеем на RabbitMQ - больно. Под нагрузку и реплей берут Kafka, под маршрутизацию и гарантии по сообщению - RabbitMQ.
- **Ретеншн в Kafka.** Данные не вечны: они чистятся по времени или размеру. Рассчитывай, что "перечитать" можно только в пределах окна хранения.
