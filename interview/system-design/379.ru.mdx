# Как выбрать между Django и FastAPI под конкретную задачу

## Суть

Django и FastAPI решают разные задачи, и противопоставлять их по скорости бессмысленно. Django - это монолитный комбайн с ORM, миграциями, аутентификацией и админкой из коробки, заточенный под крупные приложения с серверным HTML. FastAPI - тонкий асинхронный слой поверх Starlette и Pydantic, который силён на типизированных JSON-API. Решает не хайп, а форма задачи: какой объём готовой инфраструктуры нужен и какая нагрузка преобладает.

## Что ответить на собесе

Противопоставлять по скорости бессмысленно: фреймворки решают разные задачи, и выбор диктует форма задачи.

- **Django** - фреймворк по принципу "batteries included": в ядре уже есть ORM, миграции, формы, шаблоны, автогенерируемая админка и подсистема пользователей с правами. Окупается на монолитах с насыщенной доменной логикой и серверной отдачей HTML - фичи собираются быстро, потому что почти всё написано за вас.
- **FastAPI** - микрофреймворк: по умолчанию асинхронный, валидирует и сериализует вход по аннотациям типов через Pydantic и сам публикует OpenAPI-схему. Его территория - быстрые JSON-API, микросервисы, шлюзы.
- **Итог:** Django берут за полноту и темп разработки сложного продукта, FastAPI - за лёгкость, строгую типизацию и нативный async для API.

## Как это работает

### Django: всё в комплекте

Ядро закрывает типовые потребности веб-разработки без сторонних библиотек:

- **ORM** - доступ к БД через Python-классы вместо ручного SQL.
- **Админка** - управляющая панель, сгенерированная по моделям.
- **Аутентификация и права** - готовое управление пользователями и доступом.
- **Шаблоны** - серверная генерация HTML.
- **Маршрутизация** - объявление URL-маршрутов.

```python
from django.db import models


class BlogPost(models.Model):
    title = models.CharField(max_length=100)
    content = models.TextField()
    created_at = models.DateTimeField(auto_now_add=True)
```

### FastAPI: каркас, остальное сами

Стек вы собираете под себя, а фреймворк даёт основу и три опоры:

- **Async** - нативные `async`/`await` и высокая пропускная способность на I/O.
- **Автодокументация** - OpenAPI и Swagger UI без ручной возни.
- **Валидация по типам** - проверка и приведение входа через Pydantic.

```python
from fastapi import FastAPI

app = FastAPI()


@app.get("/")
async def read_root():
    return {"message": "Hello, World"}
```

### Где что уместно

- **Django** - корпоративные порталы, контент-сайты, всё с админкой и серверным рендерингом, где ценят темп сборки фич.
- **FastAPI** - нагруженные REST/JSON-API, микросервисы и шлюзы, где важна латентность и нужен async.

## Подводные камни

- "FastAPI всегда быстрее" - подмена. Выигрыш виден на I/O-bound нагрузке и асинхронном коде; на CPU-bound задачах разрыв стирается, да и Django давно умеет async-вьюхи.
- FastAPI не даёт ORM, миграции и админку - их добирают отдельно (SQLAlchemy, Alembic и подобное), и на большом продукте объём ручной сборки растёт.
- Django-ORM не терпит синхронных вызовов прямо внутри `async def` без обёрток вроде `sync_to_async`; смешивать парадигмы нужно осознанно.
- "Что знает команда" нередко перевешивает технические нюансы: знакомый стек сокращает путь до результата сильнее любого абстрактного бенчмарка.
