# Зачем топик в Kafka режут на партиции и что это даёт системе?

## Суть

Партиция - это отдельная упорядоченная лента записей внутри топика. Нарезка топика на партиции открывает горизонтальное масштабирование, параллельную обработку и репликацию между брокерами, но строгий порядок сообщений гарантируется только внутри одной партиции, а не по топику целиком.

## Что ответить на собесе

Топик в Kafka - это логический поток событий, а партиция - его физическая единица параллелизма.

- Каждая партиция - упорядоченный, дозаписываемый только в конец лог, который целиком живёт на одном брокере и реплицируется на другие.
- Разбивая топик на N партиций, мы получаем N независимых лент: их раскладывают по разным брокерам и читают N потребителями одновременно.
- Цена за это - порядок: он сохраняется внутри партиции, но между партициями глобального порядка нет.
- Поэтому ключ сообщения выбирают так, чтобы события, которым важна последовательность, попадали в одну и ту же партицию.

## Как это работает

При создании топика задаётся число партиций. Продюсер для каждого сообщения вычисляет целевую партицию: если указан ключ, берётся хеш ключа по модулю числа партиций, и одинаковые ключи всегда идут в одну партицию. Если ключа нет - партиция выбирается по балансирующей стратегии.

Внутри партиции каждому сообщению присваивается монотонно растущий offset. Потребитель хранит свой offset и читает строго по порядку. Группа потребителей делит партиции между своими участниками: одну партицию в каждый момент читает ровно один потребитель группы, поэтому число партиций задаёт потолок параллелизма для группы.

```python
from kafka import KafkaProducer

producer = KafkaProducer(bootstrap_servers="localhost:9092")

producer.send("logs", key=b"serverA", value=b"log from A")
producer.send("logs", key=b"serverB", value=b"log from B")
producer.send("logs", key=b"serverA", value=b"another log from A")

producer.flush()
producer.close()
```

Здесь оба сообщения с ключом `serverA` попадут в одну партицию и сохранят взаимный порядок, а `serverB` уйдёт в свою. Партицию мы не задаём вручную - её выбирает Kafka по хешу ключа, и это правильный подход для продакшена.

## Подводные камни

- Глобального порядка по топику нет. Если задача требует строгой последовательности, либо используйте один ключ, либо одну партицию (теряя параллелизм).
- Число партиций можно увеличить, но не уменьшить. Увеличение ломает привязку ключ→партиция: старые ключи начнут хешироваться в другие партиции.
- Партиций больше, чем потребителей в группе, - часть потребителей будет нагружена сильнее. Потребителей больше, чем партиций, - лишние простаивают.
- Слишком много партиций дорого: растёт нагрузка на контроллер, время восстановления и потребление файловых дескрипторов.
- Партиция целиком помещается на брокер: одна гигантская партиция не масштабируется и упирается в диск этого брокера.
