# Событийная архитектура (EDA): как компоненты общаются через события?

## Суть

Event-Driven Architecture - это стиль проектирования, где части системы не дёргают друг друга напрямую, а обмениваются событиями: один компонент публикует факт «что-то произошло», а другие на него реагируют. За счёт этого компоненты слабо связаны, работают асинхронно и масштабируются независимо.

## Что ответить на собесе

EDA - это архитектура, в которой взаимодействие построено вокруг событий, а не прямых вызовов. Событие - это сообщение о значимом изменении состояния: «заказ оформлен», «оплата прошла». Есть три роли: производители (publishers) генерируют события, потребители (consumers) на них подписаны и реагируют, а брокер (Kafka, RabbitMQ, SNS) принимает события от производителей и доставляет потребителям. Главное преимущество - слабая связанность: производитель не знает, кто и сколько потребителей обработает событие, поэтому новых потребителей можно добавлять без правки существующего кода. Платим за это сложностью отладки, необходимостью обеспечивать идемпотентность потребителей и тем, что согласованность становится eventual.

## Как это работает

Событие - это неизменяемый факт о произошедшем: пользователь оформил заказ, обновился профиль, прошла оплата. Компонент публикует событие и идёт дальше, не дожидаясь, пока кто-то его обработает.

Три участника:

1. **Производитель (Event Producer).** Публикует событие, когда что-то случилось. Например, веб-приложение шлёт событие при оформлении заказа.
2. **Потребитель (Event Consumer).** Подписан на интересующие его события и реагирует на них - запускает обработку, шлёт уведомление, обновляет статистику.
3. **Брокер (Event Broker).** Посредник между ними: принимает события, складывает в очередь/лог и доставляет подписчикам, обеспечивая асинхронность и развязку.

Простая модель «издатель-подписчик» в коде:

```python
class EventBus:
    def __init__(self):
        self.subscribers = {}

    def subscribe(self, event_type, handler):
        self.subscribers.setdefault(event_type, []).append(handler)

    def publish(self, event_type, payload):
        for handler in self.subscribers.get(event_type, []):
            handler(payload)


bus = EventBus()
bus.subscribe("order_created", lambda p: print(f"warehouse: reserve {p}"))
bus.subscribe("order_created", lambda p: print(f"notify: confirm {p}"))
bus.publish("order_created", "order-42")
```

Вывод:

```
warehouse: reserve order-42
notify: confirm order-42
```

Производитель (`publish`) ничего не знает о подписчиках - он лишь сообщает факт. Добавить сервис аналитики - это ещё один `subscribe`, без изменения публикующего кода.

Почему это удобно:

- **Асинхронность.** Компоненты не ждут друг друга, события обрабатываются параллельно.
- **Масштабируемость.** Новых потребителей и производителей добавляют, не трогая остальных.
- **Слабая связанность.** Компоненты можно менять и деплоить независимо.
- **Реактивность.** Система быстро откликается на происходящее.

## Подводные камни

- **Eventual consistency, а не мгновенная.** Потребители обрабатывают события с задержкой, поэтому состояние системы какое-то время рассогласовано. Логику нужно строить с учётом этого.
- **Идемпотентность обязательна.** Брокеры обычно гарантируют доставку «как минимум один раз», то есть событие может прийти повторно. Потребитель должен корректно отрабатывать дубликаты.
- **Сложнее отлаживать.** Поток управления размазан: проследить, кто на что среагировал, труднее, чем при прямом вызове. Нужны трассировка и корреляционные id.
- **Порядок событий не гарантирован по умолчанию.** Если порядок важен (сначала «создан», потом «оплачен»), это надо обеспечивать отдельно - например, ключами партиционирования.
- **Брокер - критичный узел.** Он становится точкой отказа и сам требует отказоустойчивости и мониторинга.
