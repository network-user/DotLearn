# Почему сообщение исчезает из очереди только после подтверждения обработки?

## Суть

В надёжных брокерах сообщение удаляется не в момент чтения, а после явного подтверждения от потребителя: `DeleteMessage` в Amazon SQS, `ack` в RabbitMQ. Так гарантируется, что упавший на полпути обработчик не потеряет сообщение, а вернёт его в работу.

## Что ответить на собесе

Удаление по подтверждению нужно ради семантики at-least-once: чтение и удаление разнесены, потребитель сначала забирает сообщение, обрабатывает его и только потом подтверждает успех.

- Если потребитель упадёт до подтверждения, сообщение снова станет доступным и его подхватит другой.
- В SQS прочитанное сообщение прячется на время Visibility Timeout, и если за этот срок не пришёл `DeleteMessage`, оно возвращается в очередь.
- В RabbitMQ при ручном `ack` сообщение держится за потребителем, пока не придёт подтверждение; на разрыв соединения без `ack` брокер вернёт его в очередь.
- Платой за надёжность становится возможность повторной доставки, поэтому обработчик должен быть идемпотентным.

**Amazon SQS** строит логику на таймауте видимости:

1. `ReceiveMessage` читает сообщение и прячет его на время Visibility Timeout.
2. Потребитель выполняет работу, пока сообщение невидимо для остальных.
3. `DeleteMessage` по `ReceiptHandle` убирает сообщение навсегда. Если не успеть до конца таймаута, сообщение снова станет видно другим.

```python
import boto3

sqs = boto3.client('sqs')
queue_url = 'https://sqs.region.amazonaws.com/account-id/queue-name'

response = sqs.receive_message(
    QueueUrl=queue_url,
    MaxNumberOfMessages=1,
    WaitTimeSeconds=10,
)

message = response['Messages'][0]
receipt_handle = message['ReceiptHandle']

# прикладная обработка сообщения

sqs.delete_message(QueueUrl=queue_url, ReceiptHandle=receipt_handle)
```

**RabbitMQ** работает на подтверждениях `ack`:

1. Сообщение доставляется потребителю из очереди.
2. Потребитель выполняет работу.
3. `basic_ack` подтверждает успех, и брокер удаляет сообщение. Без `ack` при обрыве канала сообщение возвращается в очередь.

```python
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()
channel.queue_declare(queue='task_queue', durable=True)

def callback(ch, method, properties, body):
    # прикладная обработка сообщения
    ch.basic_ack(delivery_tag=method.delivery_tag)

channel.basic_consume(queue='task_queue', on_message_callback=callback)
channel.start_consuming()
```

## Подводные камни

- Автоподтверждение (`auto_ack=True` в RabbitMQ) удаляет сообщение в момент доставки: если обработчик упал, сообщение потеряно. Для надёжности нужен ручной `ack`.
- Слишком короткий Visibility Timeout в SQS приводит к двойной обработке: таймаут истёк, пока задача ещё считалась.
- At-least-once допускает дубликаты, поэтому обработчик обязан быть идемпотентным; ровно-однократной доставки эти схемы не дают.
- Если `ack`/`DeleteMessage` не вызвать никогда (баг в коде), сообщение будет крутиться по кругу и заблокирует прогресс. Здесь спасает dead-letter queue после N попыток.
