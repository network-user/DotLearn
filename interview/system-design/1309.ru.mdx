# Что на самом деле утверждает теорема CAP и как её читать?

## Суть

Теорема CAP говорит: при разрыве сети (partition) распределённая система вынуждена выбрать одно из двух - либо согласованность (Consistency), либо доступность (Availability). Получить и то, и другое одновременно во время разделения невозможно.

## Что ответить на собесе

Главная тонкость, на которой часто валятся: «выбери 2 из 3» - упрощение, реальный выбор только между C и A и только в момент разделения.

- Устойчивость к разделению (P) для распределённой системы не опция: сеть рвётся всегда, и система обязана это переживать.
- При разрыве связи можно либо отказывать в ответе ради согласованности (CP), либо продолжать отвечать, рискуя отдать устаревшие данные (AP).
- Примеры CP - HBase, MongoDB в строгом режиме; примеры AP - DynamoDB, Cassandra.
- «CA-системы» в строгом смысле - это про нераспределённый сценарий, для распределённой БД такой вариант нерабочий.

## Как это работает

Три свойства расшифровываются так:

1. **Consistency** - после успешной записи любое чтение видит уже новое значение; все узлы согласованы.
2. **Availability** - любой запрос к работающему узлу получает ответ без ошибки, пусть и не самый свежий.
3. **Partition Tolerance** - система продолжает работать, когда сбой сети разрывает связь между узлами.

Разница проявляется при разделении. Возьмём AP-систему: при разрыве узлы продолжают принимать запросы, но расходятся в данных.

```python
def write_data(node, data):
    node.store(data)
    return "Data written"

def read_data(node):
    return node.retrieve()

node1 = Node()
node2 = Node()

write_data(node1, "value1")
write_data(node2, "value2")

print(read_data(node1))  # value1
print(read_data(node2))  # value2
```

Узлы держат разные значения, но система доступна: согласованностью пожертвовали ради ответа.

## Подводные камни

- «2 из 3» - вредное упрощение. P не выбирают: для распределённой системы выбор стоит только между C и A, и только в момент разделения.
- Вне разделения система может быть и согласованной, и доступной одновременно - CAP диктует компромисс лишь во время сбоя сети.
- C в CAP - это линеаризуемость, а не ACID-Consistency из транзакций. Это разные «C», их часто путают.
- Модель PACELC точнее: она добавляет, что даже без разделения (Else) приходится выбирать между задержкой (Latency) и согласованностью.
- AP не значит «вечно несогласованная»: такие системы обычно eventually consistent - после восстановления сети узлы сходятся к общему значению.
