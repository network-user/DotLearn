# API Gateway: why a system needs a single entry point in front of microservices

## In short

An API Gateway is an entry service through which all client requests to the microservices pass. It hides the system's internal structure behind a single address and takes on cross-cutting concerns: routing, authentication, response aggregation, caching, and metrics collection.

## How to answer in an interview

"An API Gateway is a single entry point in front of a swarm of microservices. Instead of the client knowing the addresses of dozens of services and assembling the data itself, it sends one request to the gateway. The gateway decides which service to route the request to, checks authentication and authorization at the door, aggregates responses from several services into one when needed, caches frequent responses, and writes metrics and logs. This simplifies the client, hides the internal topology, and moves cross-cutting logic - security, throttling, monitoring - into one place. The downside is that the gateway becomes a single point of failure and a bottleneck, so it's kept fault-tolerant and isn't weighed down with business logic."

1. **Reception.** The client sends a request to the gateway's single address.
2. **Authentication and authorization.** The gateway checks the token and the right to perform the action - only verified requests reach the services.
3. **Routing.** Based on the URL path or headers, the gateway picks the target service. The internal topology stays hidden from the client.
4. **Aggregation.** If the response needs data from several services, the gateway calls them and assembles a single result.
5. **Response and cache.** The client gets the result; frequent responses can be served from cache, reducing load.

```python
ROUTES = {
    "restaurants": "http://restaurants-svc",
    "menus": "http://menus-svc",
    "promos": "http://promos-svc",
}

def resolve(path):
    head = path.strip("/").split("/")[0]
    return ROUTES.get(head, "404")

def aggregate(parts):
    return {p: ROUTES[p] for p in parts if p in ROUTES}

print(resolve("/menus/42"))
print(aggregate(["restaurants", "promos"]))
```

In a food-delivery example the client needs restaurants, menus, and promos. Without a gateway - three requests to three services; with a gateway - one request, and the gateway does the aggregation.

## Pitfalls

- The gateway is a single point of failure and a potential bottleneck. It's kept in several instances behind a load balancer, otherwise the gateway going down brings the whole system down.
- The temptation to pile business logic into the gateway. This turns a "thin" gateway into a new monolith. The gateway should hold only cross-cutting concerns: routing, auth, aggregation, cache, metrics.
- Don't confuse an API Gateway with a load balancer. A balancer distributes traffic among copies of one service; a gateway routes across different services and operates at the application layer (L7).
- Aggregation at the gateway saves the client round trips but adds coupling: the gateway starts knowing the contracts of many services. For complex cases a separate BFF (backend for frontend) is sometimes split out.
