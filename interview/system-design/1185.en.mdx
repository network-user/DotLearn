# Name the common architectural styles and explain the problem each one solves

## In short

An architectural style is a standard way to slice a system into parts and describe how those parts communicate. In an interview, what's usually expected is the family of UI patterns (MVC, MVP, MVVM) and a set of system styles: client-server, microservices, P2P, event-driven architecture. Each addresses its own pain point - separation of responsibilities, testability, or scaling.

## How to answer in an interview

"I split them into two groups. The first is patterns for organizing the UI layer: MVC, MVP, and MVVM. They differ in who acts as the mediator between data and screen - a controller, a presenter, or a viewmodel with binding. The second group is system styles: client-server as the baseline, microservices for independent deployment and scaling, P2P for distributing load without a central node, and event-driven architecture for asynchronous reaction to changes. The choice of style is dictated not by fashion but by which property of the system you need to strengthen: testability, independence of parts, or fault tolerance."

## How it works

The UI family (MVC / MVP / MVVM) shares one idea - separate data and business logic (Model) from the presentation (View). The difference is the third participant:

- **MVC**: the Controller takes input, changes the Model, and selects a View. The View may read the Model directly.
- **MVP**: the Presenter holds all the presentation logic, the View is as passive as possible - easy to replace with a stub in tests.
- **MVVM**: the ViewModel exposes observable properties, and the View binds to them through two-way binding. A field changes - the screen updates itself.

System styles solve a different problem - how to distribute functionality across processes and machines:

- **Client-server**: logic and data are centralized on the server, the client sends requests.
- **Microservices**: the application is a set of small independent services (payments, catalog, users), each deployed and scaled on its own.
- **P2P**: all nodes are equal and are simultaneously clients and servers, with no central point (torrents).
- **Event-driven**: components react to events, exchange is asynchronous, which is convenient for real-time systems.

```python
class OrderModel:
    def __init__(self):
        self.items = []

    def add(self, item):
        self.items.append(item)

class OrderController:
    def __init__(self, model):
        self.model = model

    def handle_add(self, item):
        self.model.add(item)
        return f"items: {len(self.model.items)}"
```

## Pitfalls

- MVC, MVP, and MVVM are presentation-layer patterns, while microservices and client-server are whole-system styles. Mixing them in one list as "equivalent" is a common interview mistake.
- MVVM without a framework that has binding (WPF, Vue, SwiftUI) turns into manual boilerplate - the pattern itself does not provide binding.
- Microservices add network calls, distributed transactions, and operational complexity. For a small project a monolith is almost always cheaper.
- "P2P" is sometimes confused with "client-server with multiple servers." The key marker of P2P is the absence of a dedicated server; the nodes are symmetric.
