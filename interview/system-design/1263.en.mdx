# Load balancer: what problem does it solve and which algorithms spread traffic?

## In short

A load balancer is an entry point that distributes incoming requests across a pool of identical servers so that none gets overloaded and the failure of one does not take the service down. It can be implemented in software (Nginx, HAProxy, Envoy) or on dedicated hardware appliances, and the server choice itself is made by an algorithm: round-robin, least connections, IP/consistent hash.

## How to answer in an interview

A load balancer solves three problems at once: availability, scalability, and performance. A single entry point sits in front of the server pool, distributes requests across the backends, and removes failed nodes from rotation via health checks. Thanks to this, the pool scales horizontally by adding nodes, and the failure of one server is invisible to the user. You pick the algorithm based on the load profile: round-robin for uniform short requests, least connections for requests of varying duration, and a hash by key when you need to pin a client to a specific server (sticky sessions).

## How it works

A software load balancer is a process on a server or in a container that accepts a connection and proxies it to the chosen backend. Nginx and HAProxy work as a reverse proxy: they keep a pool of upstream servers, poll their health, and apply the chosen algorithm. Hardware solutions (F5, Citrix ADC) do the same on a dedicated appliance and often take on SSL termination and DDoS protection.

```nginx
http {
    upstream backend {
        least_conn;
        server backend1.example.com max_fails=3 fail_timeout=10s;
        server backend2.example.com max_fails=3 fail_timeout=10s;
    }

    server {
        listen 80;
        location / {
            proxy_pass http://backend;
        }
    }
}
```

- `upstream backend` - the pool of backends across which traffic is shared.
- `least_conn` - pick the server with the fewest active connections.
- `max_fails` / `fail_timeout` - after how many failures, and for how long, a server is excluded from rotation.
- `proxy_pass http://backend` - route the request into the pool.

Main algorithms:

1. **Round Robin** - servers receive requests in turn, evenly.
2. **Least Connections** - the request goes where there are fewer active connections.
3. **IP / consistent hash** - requests from the same client consistently land on the same server.

## Pitfalls

- The load balancer itself becomes a single point of failure. In production you run at least two with failover (keepalived / VRRP, DNS, anycast).
- Round-robin ignores actual load: a long request on "its" server skews the distribution - least connections is more appropriate here.
- IP-based sticky sessions break behind NAT and CGNAT: thousands of clients hide behind a single IP, and they all get glued to one server.
- Without health checks the balancer keeps sending traffic to a downed node. You should check that the application actually works, not just that the port is open.
- L4 (TCP) and L7 (HTTP) balancing are different things: routing by URL, headers, and cookies is only possible at L7.
