# Event-driven architecture (EDA): how components communicate through events?

## In short

Event-Driven Architecture is a design style where parts of the system don't call each other directly but exchange events: one component publishes the fact that "something happened", and others react to it. Thanks to this, components are loosely coupled, work asynchronously, and scale independently.

## How to answer in an interview

EDA is an architecture in which interaction is built around events rather than direct calls. An event is a message about a significant state change: "order placed", "payment went through". There are three roles: producers (publishers) generate events, consumers are subscribed to them and react, and a broker (Kafka, RabbitMQ, SNS) receives events from producers and delivers them to consumers. The main advantage is loose coupling: the producer doesn't know who or how many consumers will process the event, so new consumers can be added without changing existing code. We pay for this with harder debugging, the need to make consumers idempotent, and consistency becoming eventual.

## How it works

An event is an immutable fact about something that happened: a user placed an order, a profile was updated, a payment went through. A component publishes an event and moves on, without waiting for someone to process it.

Three participants:

1. **Producer (Event Producer).** Publishes an event when something happens. For example, a web app sends an event when an order is placed.
2. **Consumer (Event Consumer).** Subscribed to the events it cares about and reacts to them - kicks off processing, sends a notification, updates statistics.
3. **Broker (Event Broker).** The intermediary between them: receives events, stores them in a queue/log, and delivers them to subscribers, providing asynchrony and decoupling.

A simple "publisher-subscriber" model in code:

```python
class EventBus:
    def __init__(self):
        self.subscribers = {}

    def subscribe(self, event_type, handler):
        self.subscribers.setdefault(event_type, []).append(handler)

    def publish(self, event_type, payload):
        for handler in self.subscribers.get(event_type, []):
            handler(payload)


bus = EventBus()
bus.subscribe("order_created", lambda p: print(f"warehouse: reserve {p}"))
bus.subscribe("order_created", lambda p: print(f"notify: confirm {p}"))
bus.publish("order_created", "order-42")
```

Output:

```
warehouse: reserve order-42
notify: confirm order-42
```

The producer (`publish`) knows nothing about the subscribers - it merely announces a fact. Adding an analytics service is one more `subscribe`, with no change to the publishing code.

Why this is convenient:

- **Asynchrony.** Components don't wait for each other; events are processed in parallel.
- **Scalability.** New consumers and producers are added without touching the rest.
- **Loose coupling.** Components can be changed and deployed independently.
- **Reactivity.** The system responds quickly to what's happening.

## Pitfalls

- **Eventual, not instant, consistency.** Consumers process events with a delay, so the system's state is inconsistent for a while. The logic must be built with this in mind.
- **Idempotency is mandatory.** Brokers usually guarantee "at-least-once" delivery, meaning an event may arrive again. The consumer must handle duplicates correctly.
- **Harder to debug.** The control flow is smeared out: tracing who reacted to what is harder than with a direct call. You need tracing and correlation ids.
- **Event ordering isn't guaranteed by default.** If order matters (first "created", then "paid"), it has to be ensured separately - for instance, with partition keys.
- **The broker is a critical node.** It becomes a point of failure and itself requires fault tolerance and monitoring.
