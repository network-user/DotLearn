# MVC vs MVVM: where the line runs and who binds data to the screen

## In short

Both patterns separate business logic from the interface, but they differ in the intermediary between the model and the view. In MVC the controller wires that connection by hand; in MVVM the ViewModel takes it on through two-way binding - the screen updates automatically when the data changes.

## How to answer in an interview

"MVC and MVVM both split an application into three layers, and the Model is the same in both - data and business logic. The difference is who connects the data to the screen. In MVC that's the controller: it takes input, mutates the model, and decides what to show, and the View often pulls data straight from the model. In MVVM a ViewModel appears - it prepares data for display and exposes observable properties, and the View is bound to them. So when the data changes the interface updates itself, with no explicit code. MVC is simpler and a good fit for the web; MVVM wins in rich desktop and mobile interfaces with lots of two-way interaction."

## How it works

The shared part is three layers:

- **Model**: data and business logic, unaware of the UI.
- **View**: what the user sees.
- **Intermediary**: Controller or ViewModel.

In **MVC** the control flow goes like this: the user acts - the controller catches the input - mutates the model - picks and updates the view. The wiring is imperative: the developer writes exactly what to update and when.

In **MVVM** the ViewModel exposes properties and commands. The View binds to them declaratively. A property changes in the ViewModel - the View re-renders; the user enters a value - it flows back into the ViewModel. This two-way flow is the pattern's main feature.

An analogy: in MVC the controller is a manager who manually hands the order to the kitchen and tells the waiter what to bring. In MVVM the ViewModel is a director with a script, and the stage (the View) automatically reflects its directions through the binding "wire".

```python
class ViewModel:
    def __init__(self):
        self._title = ""
        self._observers = []

    def bind(self, callback):
        self._observers.append(callback)

    @property
    def title(self):
        return self._title

    @title.setter
    def title(self, value):
        self._title = value
        for cb in self._observers:
            cb(value)

vm = ViewModel()
rendered = []
vm.bind(lambda v: rendered.append(v))
vm.title = "Hello"
print(rendered)
```

## Pitfalls

- The main mistake is reducing the difference to "MVVM is newer". The difference is substantive: who binds the data and whether there is automatic binding.
- MVVM without a framework that provides reactive binding (WPF, SwiftUI, Vue) degenerates into manually subscribing to changes - the benefit is lost.
- A "fat" View in MVC is a common problem: display logic leaks into the view and testing gets harder. In MVVM a passive View and a testable ViewModel prevent this.
- Don't confuse MVVM with MVP: in MVP the presenter itself pushes data into a passive View, with no binding. In MVVM the connection is declarative and two-way.
