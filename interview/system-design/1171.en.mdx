# Producer and consumer: two roles in asynchronous data exchange

## In short

A producer forms and sends data; a consumer receives and processes it. These are two sides of asynchronous exchange through a message broker (Kafka, RabbitMQ, SQS), working independently of each other.

## How to answer in an interview

Producer and consumer are roles, not necessarily separate services: one component can be both, in different threads. A producer puts a message into a topic or queue and does not know who will read it or when. A consumer is subscribed to a topic and picks up messages, processing them at its own pace. The point of this decoupling is that the sides do not block each other: the producer doesn't wait for processing, and the consumer doesn't wait synchronously for data to appear. The broker between them acts as a buffer, which provides scalability, fault tolerance, and load-peak smoothing.

The producer sends a message to a topic:

```python
from kafka import KafkaProducer

producer = KafkaProducer(bootstrap_servers='localhost:9092')
producer.send('my_topic', b'Hello, World!')
producer.flush()
producer.close()
```

`send` puts the message into the topic, `flush` waits for the buffer to actually be sent, `close` closes the connection.

The consumer subscribes to the topic and reads the stream:

```python
from kafka import KafkaConsumer

consumer = KafkaConsumer('my_topic', bootstrap_servers='localhost:9092')
for message in consumer:
    print(f"Received: {message.value}")
consumer.close()
```

`KafkaConsumer('my_topic', ...)` subscribes to the topic, and the `for message in consumer` loop reads incoming messages as they arrive.

Decoupling through the broker lets you scale the sides independently: add consumers to a group and the load is split among them across partitions.

## Pitfalls

- A producer without `flush`/`close` may lose messages from the buffer if the process exits: `send` is asynchronous and does not by itself guarantee delivery.
- If a group has more consumers than partitions, the extra ones sit idle - read parallelism is limited by the number of partitions.
- "Independence" does not eliminate backpressure: if consumers can't keep up, lag builds up in the broker, and you need to watch consumer lag.
- Order is guaranteed only within a partition, not across the whole topic - if global order matters, that's a design constraint.
