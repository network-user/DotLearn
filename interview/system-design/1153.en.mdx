# Why does a message leave the queue only after processing is acknowledged?

## In short

In reliable brokers a message is removed not at the moment it is read but after an explicit acknowledgment from the consumer: `DeleteMessage` in Amazon SQS, `ack` in RabbitMQ. This guarantees that a handler that fails midway does not lose the message but instead returns it for reprocessing.

## How to answer in an interview

Deletion-on-acknowledgment exists for at-least-once semantics. Reading and deletion are separated: the consumer first takes the message, processes it, and only then confirms success. If it crashes before confirming, the message becomes available again and another consumer picks it up. In SQS a message that has been read is hidden for the duration of the Visibility Timeout, and if no `DeleteMessage` arrives within that window, it returns to the queue. In RabbitMQ, with manual `ack`, the message stays held by the consumer until confirmation arrives; if the connection drops without an `ack`, the broker returns it to the queue. The price of reliability is the possibility of redelivery, so the handler must be idempotent.

**Amazon SQS** builds its logic on the visibility timeout:

1. `ReceiveMessage` reads a message and hides it for the duration of the Visibility Timeout.
2. The consumer does its work while the message is invisible to the others.
3. `DeleteMessage` by `ReceiptHandle` removes the message for good. Miss the timeout window and the message becomes visible to others again.

```python
import boto3

sqs = boto3.client('sqs')
queue_url = 'https://sqs.region.amazonaws.com/account-id/queue-name'

response = sqs.receive_message(
    QueueUrl=queue_url,
    MaxNumberOfMessages=1,
    WaitTimeSeconds=10,
)

message = response['Messages'][0]
receipt_handle = message['ReceiptHandle']

# application-level message processing

sqs.delete_message(QueueUrl=queue_url, ReceiptHandle=receipt_handle)
```

**RabbitMQ** runs on `ack` confirmations:

1. The message is delivered to the consumer from the queue.
2. The consumer does its work.
3. `basic_ack` confirms success, and the broker deletes the message. Without an `ack`, if the channel drops, the message returns to the queue.

```python
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()
channel.queue_declare(queue='task_queue', durable=True)

def callback(ch, method, properties, body):
    # application-level message processing
    ch.basic_ack(delivery_tag=method.delivery_tag)

channel.basic_consume(queue='task_queue', on_message_callback=callback)
channel.start_consuming()
```

## Pitfalls

- Auto-acknowledgment (`auto_ack=True` in RabbitMQ) deletes the message at the moment of delivery: if the handler crashes, the message is lost. For reliability you need a manual `ack`.
- Too short a Visibility Timeout in SQS leads to double processing: the timeout expired while the task was still running.
- At-least-once allows duplicates, so the handler must be idempotent; these schemes do not provide exactly-once delivery.
- If `ack`/`DeleteMessage` is never called (a bug in the code), the message keeps cycling forever and blocks progress. A dead-letter queue after N attempts saves you here.
