# CAP-теорема: почему распределённая система не может получить всё сразу?

## Суть

CAP-теорема утверждает: из трёх свойств распределённой системы - согласованности (Consistency), доступности (Availability) и устойчивости к разделению сети (Partition Tolerance) - одновременно гарантировать можно только два. Раз сетевые разрывы в реальной распределённой системе неизбежны, выбор по сути сводится к одному: жертвовать ли согласованностью ради доступности или наоборот.

## Что ответить на собесе

CAP-теорема (теорема Брюера, 2000 год): из consistency, availability и partition tolerance одновременно гарантируются максимум два из трёх.

- Тонкость в том, что в распределённой системе разделение сети случается само по себе, поэтому P отбрасывать нельзя - реальный выбор только между CP и AP.
- **CP-система** при разрыве сети предпочтёт остановиться и не отдавать потенциально устаревшие данные (так делают, например, банковские системы).
- **AP-система** продолжит отвечать, но данные на узлах временно разойдутся (так работают соцсети и ленты).
- Вариант CA существует только в теории - в системе без сетевых разделений, то есть фактически в одном узле.

## Как это работает

CAP описывает фундаментальные ограничения, с которыми сталкивается любая система, размазанная по нескольким узлам.

Три свойства:

1. **Consistency (согласованность).** На любой запрос система возвращает либо самые свежие данные, либо ошибку - все узлы видят одно и то же состояние в один момент.
2. **Availability (доступность).** Каждый запрос получает ответ без ошибки, даже если часть узлов недоступна.
3. **Partition Tolerance (устойчивость к разделению).** Система продолжает работать при произвольных разрывах связи между узлами.

Конфликт возникает именно в момент сетевого разрыва. Связь между узлами порвалась - и система обязана выбрать одну из двух реакций:

- **CP:** отказаться отвечать (или вернуть ошибку), пока согласованность не восстановлена. Доступность временно теряется.
- **AP:** продолжать отвечать на каждом узле своими данными. Доступность сохраняется, но узлы временно рассогласованы (eventual consistency).
- **CA:** одновременно C и A возможны только при отсутствии разрывов - то есть на практике в нераспределённой системе.

Где это видно: банковский перевод выбирает CP - лучше отказать в операции, чем дважды списать деньги. Лента соцсети выбирает AP - пользователю важнее увидеть хоть что-то, чем получить ошибку, а небольшая задержка свежести некритична.

## Подводные камни

- **«Два из трёх» - упрощение.** P в реальной распределённой системе не опциональна: сетевые сбои случаются всегда. Поэтому честный выбор не «любые два», а только CP против AP.
- **CA - почти миф.** Систему CA можно получить лишь там, где разделения сети не бывает, то есть фактически в одном узле без распределённости.
- **Согласованность не бинарна.** Между строгой и eventual consistency есть промежуточные уровни (read-your-writes, monotonic reads); CAP их огрубляет.
- **Выбор делается на время разрыва, а не навсегда.** Когда сеть в порядке, система может быть и согласованной, и доступной. Жертва наступает только в момент партиции - это уточняет модель PACELC.
- **Путаница C в CAP и C в ACID.** Это разные понятия: в CAP речь про согласованность реплик, в ACID - про инварианты внутри транзакции.
