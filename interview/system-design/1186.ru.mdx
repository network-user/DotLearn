# MVC против MVVM: где проходит граница и кто связывает данные с экраном

## Суть

Оба паттерна отделяют бизнес-логику от интерфейса, но различаются посредником между моделью и представлением. В MVC связь руками ведёт контроллер, в MVVM её берёт на себя ViewModel через двусторонний биндинг - экран обновляется автоматически при изменении данных.

## Что ответить на собесе

Оба паттерна делят приложение на три слоя с одинаковой Model, а разница - в том, кто соединяет данные с экраном:

- **MVC:** контроллер принимает ввод, меняет модель и решает, что показать, причём View нередко тянет данные из модели напрямую.
- **MVVM:** появляется ViewModel - она готовит данные для отображения и выставляет наблюдаемые свойства, а View связана с ними биндингом, поэтому при изменении данных интерфейс обновляется сам, без явного кода.
- **Где что:** MVC проще и хорош для веба, MVVM выигрывает в насыщенных интерфейсах десктопа и мобилок, где много двусторонних взаимодействий.

## Как это работает

Общая часть - три слоя:

- **Model**: данные и бизнес-логика, не знает про UI.
- **View**: то, что видит пользователь.
- **Посредник**: Controller или ViewModel.

В **MVC** поток управления такой: пользователь действует - контроллер ловит ввод - меняет модель - выбирает и обновляет представление. Связь императивная: разработчик сам пишет, что и когда обновить.

В **MVVM** ViewModel выставляет свойства и команды. View декларативно связывается с ними через биндинг. Изменилось свойство в ViewModel - View перерисовалась; пользователь ввёл значение - оно ушло обратно в ViewModel. Этот двусторонний поток - главная фишка паттерна.

Аналогия: в MVC контроллер - менеджер, который вручную передаёт заказ на кухню и говорит официанту, что нести. В MVVM ViewModel - режиссёр со сценарием, а сцена (View) автоматически отражает его указания через «провод» биндинга.

```python
class ViewModel:
    def __init__(self):
        self._title = ""
        self._observers = []

    def bind(self, callback):
        self._observers.append(callback)

    @property
    def title(self):
        return self._title

    @title.setter
    def title(self, value):
        self._title = value
        for cb in self._observers:
            cb(value)

vm = ViewModel()
rendered = []
vm.bind(lambda v: rendered.append(v))
vm.title = "Привет"
print(rendered)
```

## Подводные камни

- Главная ошибка - сводить различие к «MVVM новее». Различие предметное: кто связывает данные и есть ли автоматический биндинг.
- MVVM без фреймворка с реактивным биндингом (WPF, SwiftUI, Vue) превращается в ручную подписку на изменения - выгода теряется.
- «Толстая» View в MVC - частая беда: логика отображения утекает в представление, тесты усложняются. В MVVM этому мешает пассивная View и testable ViewModel.
- Не путайте MVVM с MVP: в MVP presenter сам толкает данные в пассивную View, биндинга нет. В MVVM связь декларативная и двусторонняя.
