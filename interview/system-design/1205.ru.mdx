# Паттерны устойчивости микросервисов: Circuit Breaker, Retry, Fallback и компания

## Суть

В микросервисах отказ одного сервиса не должен ронять всю систему. Для этого есть набор паттернов: Timeout не даёт запросу зависнуть, Retry повторяет временный сбой, Circuit Breaker перестаёт долбиться в уже упавший сервис, Fallback подставляет запасной вариант, а Bulkhead изолирует ресурсы, чтобы проблема не растеклась дальше.

## Что ответить на собесе

Назову пять основных паттернов. **Timeout** ограничивает ожидание ответа, иначе зависший вызов держит ресурсы. **Retry** автоматически повторяет запрос при кратковременном сбое, желательно с экспоненциальной задержкой и джиттером. **Circuit Breaker** отслеживает ошибки и, когда их слишком много, «размыкается»: какое-то время вызовы сразу падают, не доходя до больного сервиса, потом он пробует полузакрыться и проверить восстановление. **Fallback** даёт деградированный ответ при отказе - кэш или заглушку. **Bulkhead** изолирует ресурсы (например, отдельные пулы потоков), чтобы перегрузка одного интегрированного сервиса не съела ресурсы для всех остальных. На практике они комбинируются: Timeout + Retry + Circuit Breaker + Fallback - типичная связка для исходящего вызова.

## Как это работает

### Circuit Breaker (предохранитель)

Самый показательный паттерн. У него три состояния: **CLOSED** (вызовы идут), **OPEN** (вызовы сразу падают, сервис не трогаем), **HALF-OPEN** (пробный вызов, чтобы проверить восстановление). Это спасает от каскадного отказа: упавший сервис не добивают повторными запросами.

```python
import time


class CircuitBreaker:
    def __init__(self, failure_threshold, recovery_timeout):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.failure_count = 0
        self.state = "CLOSED"
        self.last_failure_time = None

    def call(self, func, *args, **kwargs):
        if self.state == "OPEN":
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = "HALF-OPEN"
            else:
                raise RuntimeError("Circuit is open")
        try:
            result = func(*args, **kwargs)
            self.failure_count = 0
            self.state = "CLOSED"
            return result
        except Exception:
            self.failure_count += 1
            self.last_failure_time = time.time()
            if self.failure_count >= self.failure_threshold:
                self.state = "OPEN"
            raise
```

### Retry (повтор)

Повторяет вызов при временном сбое. Важно: повторять стоит только идемпотентные операции и с задержкой, иначе ретраи добивают и без того нагруженный сервис.

```python
import time


def retry(func, retries=3, delay=2):
    for attempt in range(retries):
        try:
            return func()
        except Exception:
            if attempt < retries - 1:
                time.sleep(delay)
            else:
                raise
```

### Fallback (резервный путь)

При отказе основного источника отдаём запасной - кэш, значение по умолчанию, урезанный ответ. Пользователь получает деградированный, но работающий сервис.

```python
def get_data_with_fallback():
    try:
        return get_data_from_service()
    except Exception:
        return get_cached_data()
```

### Bulkhead (переборка)

Делит ресурсы на изолированные секции, как переборки на корабле: затопление одного отсека не топит судно. Например, для каждого внешнего сервиса - свой пул потоков, чтобы его зависание не выело все воркеры.

```python
from concurrent.futures import ThreadPoolExecutor

payments_pool = ThreadPoolExecutor(max_workers=5)
reports_pool = ThreadPoolExecutor(max_workers=5)
```

### Timeout (тайм-аут)

Ограничивает ожидание ответа. Без таймаута зависший вызов держит соединение и поток, а такие зависания копятся и кладут сервис.

```python
import requests


def fetch_data_with_timeout(url, timeout=5):
    try:
        response = requests.get(url, timeout=timeout)
        return response.json()
    except requests.Timeout:
        return None
```

## Подводные камни

- **Retry без задержки и идемпотентности - вредит.** Повтор неидемпотентной операции может списать деньги дважды, а мгновенные ретраи устраивают «шторм запросов» по падающему сервису. Используйте exponential backoff с джиттером.
- **Timeout обязателен под каждым сетевым вызовом.** Самая частая причина каскадных отказов - вызов без таймаута: один медленный сервис исчерпывает пул потоков вызывающего.
- **Circuit Breaker нужно настраивать по метрикам.** Слишком чувствительный порог размыкает предохранитель на случайных всплесках; слишком грубый - не успевает защитить.
- **Fallback не должен молча скрывать проблему.** Отдавая кэш или заглушку, фиксируйте это в метриках/логах, иначе деградация останется незамеченной.
- **Паттерны работают в связке, а не поодиночке.** Timeout + Retry + Circuit Breaker + Fallback дополняют друг друга; один паттерн в отрыве закрывает лишь часть сценариев отказа.
