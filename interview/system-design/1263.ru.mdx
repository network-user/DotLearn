# Балансировщик нагрузки: какую задачу решает и какими алгоритмами раскидывает трафик?

## Суть

Балансировщик нагрузки - это точка входа, которая разводит входящие запросы по пулу одинаковых серверов, чтобы ни один не перегрузился, а отказ одного из них не уронил сервис. Реализуют его программно (Nginx, HAProxy, Envoy) или на отдельных аппаратных устройствах, а сам выбор сервера делается по алгоритму: round-robin, least connections, IP/consistent hash.

## Что ответить на собесе

Балансировщик решает три задачи разом: доступность, масштабируемость и производительность. Перед пулом серверов ставится единая точка входа, которая распределяет запросы по бэкендам и убирает отказавшие узлы из ротации через health-check. Благодаря этому пул горизонтально масштабируется добавлением узлов, а падение одного сервера не видно пользователю. Алгоритм подбирают под профиль нагрузки: round-robin для однородных коротких запросов, least connections для запросов с разной длительностью, hash по ключу - когда нужна привязка клиента к конкретному серверу (sticky-сессии).

## Как это работает

Программный балансировщик - это процесс на сервере или в контейнере, который принимает соединение и проксирует его на выбранный бэкенд. Nginx и HAProxy работают как обратный прокси: держат пул upstream-серверов, опрашивают их здоровье и применяют выбранный алгоритм. Аппаратные решения (F5, Citrix ADC) делают то же на уровне выделенного устройства и часто берут на себя SSL-терминацию и защиту от DDoS.

```nginx
http {
    upstream backend {
        least_conn;
        server backend1.example.com max_fails=3 fail_timeout=10s;
        server backend2.example.com max_fails=3 fail_timeout=10s;
    }

    server {
        listen 80;
        location / {
            proxy_pass http://backend;
        }
    }
}
```

- `upstream backend` - пул бэкендов, между которыми делится трафик.
- `least_conn` - выбирать сервер с наименьшим числом активных соединений.
- `max_fails` / `fail_timeout` - после скольких неудач и на какое время сервер исключается из ротации.
- `proxy_pass http://backend` - направить запрос в пул.

Основные алгоритмы:

1. **Round Robin** - сервера получают запросы по кругу, поровну.
2. **Least Connections** - запрос уходит туда, где меньше активных соединений.
3. **IP / consistent hash** - запросы от одного клиента стабильно попадают на один сервер.

## Подводные камни

- Сам балансировщик становится единой точкой отказа. В проде их ставят минимум два с failover (keepalived / VRRP, DNS, anycast).
- Round-robin не учитывает реальную загрузку: длинный запрос на «своём» сервере перекосит распределение - тут уместнее least connections.
- Sticky-сессии по IP ломаются за NAT и CGNAT: за одним IP скрываются тысячи клиентов, и они все приклеятся к одному серверу.
- Без health-check балансировщик продолжит слать трафик на упавший узел. Проверять надо именно работу приложения, а не только открытый порт.
- Балансировка L4 (TCP) и L7 (HTTP) - разные вещи: маршрутизация по URL, заголовкам и cookie возможна только на L7.
