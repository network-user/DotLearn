# Direct vs topic exchange in RabbitMQ: what is the fundamental routing difference?

## In short

A direct exchange places a message in a queue only on an exact match between the routing key and the binding key. A topic exchange matches the routing key against a pattern, where `*` replaces exactly one word and `#` replaces zero or more words, which allows flexible routing by hierarchical keys.

## How to answer in an interview

In RabbitMQ a producer does not send a message to a queue directly but to an exchange, and the exchange type determines the routing rule. Direct is an exact string match: the routing key must literally equal the queue's binding key, otherwise the message won't land there. Topic is a pattern match over words separated by dots: `*` stands for one word, `#` for any number of them. You use direct for simple routing by fixed categories, and topic when the key encodes several dimensions (for example `region.service.level`) and subscribers want to select slices by any of them.

## How it works

When a queue is bound to an exchange, a binding key is specified. When a message arrives, the exchange matches its routing key against the bindings of its queues by its type's rule and copies the message into every matching queue.

Direct, exact match:

- Queue A is bound by the key `info`, queue B by `error`.
- A message with `info` → only A; with `error` → only B; with `warning` → nowhere.

Topic, pattern match:

- Queue A is bound by `logs.*`, queue B by `logs.#`.
- `logs.info` → both A and B (one word after `logs` fits both).
- `logs.error.critical` → only B (`#` takes two words, while `*` takes exactly one).

```python
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters("localhost"))
channel = connection.channel()

channel.exchange_declare(exchange="direct_logs", exchange_type="direct")
channel.exchange_declare(exchange="topic_logs", exchange_type="topic")

channel.basic_publish(exchange="direct_logs", routing_key="info",
                      body="exact match on 'info'")
channel.basic_publish(exchange="topic_logs", routing_key="logs.info",
                      body="pattern match on 'logs.*'")

connection.close()
```

## Pitfalls

- With no bound queues, or when nothing matches, the message is silently lost (unless `mandatory` / an alternate exchange is configured). Verify that the binding was actually created.
- In topic, `*` is exactly one word, not "any character". `logs.*` won't catch `logs` with no segment, and won't catch `logs.a.b`.
- `#` also absorbs zero words: the binding `logs.#` will match even `logs`. This is often unexpected.
- The dot is a word separator, not part of the name. The key `user.created` is two words, not a single string.
- You can stretch direct toward topic-like behavior with several bindings on one queue, but that quickly turns into an unmaintainable list of keys - for hierarchies, go straight to topic.
- A message that lands in two queues is delivered twice - these are copies, not one message "shared" between them.
