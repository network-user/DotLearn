# What does the CAP theorem actually claim, and how should you read it?

## In short

The CAP theorem states: during a network partition a distributed system is forced to choose one of two things - either consistency or availability. Getting both at the same time during a partition is impossible.

## How to answer in an interview

I'd stress an important subtlety that people often fail on: "pick 2 of 3" is a simplification. Partition tolerance (P) is not optional for a distributed system: the network always breaks, and the system is simply obligated to survive it. So the real choice is between C and A, and it arises precisely at the moment of a partition. If nodes lose connectivity, you can either refuse to answer for the sake of consistency (CP) or keep answering, risking serving stale data (AP). Examples of CP are HBase and MongoDB in strict mode; examples of AP are DynamoDB and Cassandra. "CA systems" in the strict sense are about the non-distributed scenario; for a distributed database that option doesn't work.

## How it works

The three properties decode as follows:

1. **Consistency** - after a successful write, any read sees the new value already; all nodes agree.
2. **Availability** - any request to a working node gets a response without error, even if not the freshest one.
3. **Partition Tolerance** - the system keeps working when a network failure breaks the link between nodes.

The difference shows up during a partition. Take an AP system: on a break, the nodes keep accepting requests but diverge in their data.

```python
def write_data(node, data):
    node.store(data)
    return "Data written"

def read_data(node):
    return node.retrieve()

node1 = Node()
node2 = Node()

write_data(node1, "value1")
write_data(node2, "value2")

print(read_data(node1))  # value1
print(read_data(node2))  # value2
```

The nodes hold different values, but the system is available: consistency was sacrificed for the sake of a response.

## Pitfalls

- "2 of 3" is a harmful simplification. P is not chosen: for a distributed system the choice stands only between C and A, and only at the moment of a partition.
- Outside of a partition the system can be both consistent and available at the same time - CAP dictates a trade-off only during a network failure.
- The C in CAP is linearizability, not the ACID consistency of transactions. These are different "C"s, and they are often confused.
- The PACELC model is more precise: it adds that even without a partition (Else) you have to choose between latency and consistency.
- AP does not mean "forever inconsistent": such systems are usually eventually consistent - after the network recovers, the nodes converge to a common value.
