# Direct и topic exchange в RabbitMQ: в чём принципиальная разница маршрутизации?

## Суть

Direct exchange кладёт сообщение в очередь только при точном совпадении routing key с binding key. Topic exchange сравнивает routing key с шаблоном, где `*` заменяет ровно одно слово, а `#` - ноль или больше слов, и за счёт этого позволяет гибкую маршрутизацию по иерархическим ключам.

## Что ответить на собесе

В RabbitMQ продюсер шлёт сообщение не в очередь напрямую, а в обменник, и тип обменника определяет правило маршрутизации. Direct - это точное совпадение строки: routing key должен буквально равняться binding key очереди, иначе сообщение туда не попадёт. Topic - это шаблонное совпадение по словам, разделённым точками: `*` стоит за одно слово, `#` - за любое их число. Direct берут для простой маршрутизации по фиксированным категориям, topic - когда ключ кодирует несколько измерений (например `region.service.level`) и подписчики хотят выбирать срезы по любому из них.

## Как это работает

У очереди при привязке к обменнику задаётся binding key. Когда приходит сообщение, обменник сопоставляет его routing key с binding-ами своих очередей по правилу своего типа и копирует сообщение в каждую подходящую очередь.

Direct, точное совпадение:

- Очередь A привязана по ключу `info`, очередь B - по `error`.
- Сообщение с `info` → только A; с `error` → только B; с `warning` → никуда.

Topic, совпадение по шаблону:

- Очередь A привязана по `logs.*`, очередь B - по `logs.#`.
- `logs.info` → и A, и B (одно слово после `logs` подходит обоим).
- `logs.error.critical` → только B (`#` берёт два слова, а `*` - ровно одно).

```python
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters("localhost"))
channel = connection.channel()

channel.exchange_declare(exchange="direct_logs", exchange_type="direct")
channel.exchange_declare(exchange="topic_logs", exchange_type="topic")

channel.basic_publish(exchange="direct_logs", routing_key="info",
                      body="точное совпадение по 'info'")
channel.basic_publish(exchange="topic_logs", routing_key="logs.info",
                      body="совпадение по шаблону 'logs.*'")

connection.close()
```

## Подводные камни

- Без привязанных очередей или при отсутствии совпадения сообщение тихо теряется (если не настроены `mandatory` / alternate exchange). Проверяйте, что binding реально создан.
- В topic `*` - это ровно одно слово, не «любой символ». `logs.*` не поймает `logs` без сегмента и не поймает `logs.a.b`.
- `#` поглощает и ноль слов: binding `logs.#` совпадёт даже с `logs`. Это часто неожиданно.
- Точка - разделитель слов, а не часть имени. Ключ `user.created` - два слова, а не одна строка.
- Direct можно «дотянуть» до похожего на topic поведения несколькими binding-ами одной очереди, но это быстро превращается в неподдерживаемый список ключей - для иерархий сразу берите topic.
- Сообщение, попавшее в две очереди, доставляется дважды - это копии, а не одно сообщение «на двоих».
