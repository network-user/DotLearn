# Session context in managed application

## In short

Session context is isolated state for a user's infobase connection: session parameters, temporary variables, security, and client settings. It lives from login to logout; other sessions cannot see it.

## How to answer in an interview

- I use `ПараметрыСеанса` for global flags like current organization or exchange mode.
- In an interview I separate session context from form context and server object write context.
- I stress `УстановитьПараметрыСеанса` in the session module at startup.
- I honestly note storing large tables in session parameters is wrong; it is server RAM per user.
- Background jobs have a different context; rights and parameters are set separately.

## How it works

At session start in `УстановкаПараметровСеанса` set values:

```bsl
Процедура УстановкаПараметровСеанса(ИменаПараметров)
    ПараметрыСеанса.ТекущийПользователь = Пользователи.АвторизованныйПользователь();
    ПараметрыСеанса.ОсновнаяОрганизация = ПолучитьОрганизациюПоУмолчанию();
КонецПроцедуры
```

In application code read without repeat queries:

```bsl
Функция РазрешенПросмотрОтчета()
    Возврат ПравоДоступа("Просмотр", Метаданные.Отчеты.Продажи);
КонецФункции
```

Rights are cached by the platform within the session. Changing organization in the UI updates the parameter and refreshes forms.

## Pitfalls

- **Memory leak.** A large array in a session parameter scales with user count.
- **Stale values.** A catalog change in another session does not refresh the cache automatically.
- **Background jobs.** No user session; `ПараметрыСеанса` may be empty.
- **Web client.** Inactivity timeout ends the session and drops temporary data.
- **Privileged mode.** Expands rights inside a call; does not change session parameters permanently.
