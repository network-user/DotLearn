# Предопределённые обработчики в модуле менеджера

## Суть

Модуль менеджера объекта метаданных содержит предопределённые обработчики событий жизненного цикла: `ПередЗаписью`, `ПриЗаписи`, `ПередУдалением` и аналоги. Платформа вызывает их автоматически при операциях с объектом, без явного вызова из формы.

## Что ответить на собесе

- Я разделяю модуль менеджера (логика всех экземпляров) и модуль объекта (конкретная ссылка).
- На собеседовании показываю `Отказ = Истина` в `ПередЗаписью` для валидации.
- Подчёркиваю параметр `СтандартнаяОбработка` для отключения типового поведения где допустимо.
- Честно уточняю: тяжёлые проверки в `ПередЗаписью` тормозят каждую запись; выношу в фон при необходимости.
- Для форм дублировать ту же проверку на клиенте для UX, но источник истины на сервере.

## Как это работает

```bsl
Процедура ПередЗаписью(ДанныеФормы, Отказ, СтандартнаяОбработка)
    Если ДанныеФормы.Цена < 0 Тогда
        Отказ = Истина;
        Сообщить("Цена не может быть отрицательной");
    КонецЕсли;
КонецПроцедуры
```

При записи элемента справочника из любой формы или программно платформа сначала вызывает `ПередЗаписью`. Если `Отказ` установлен в `Истина`, запись не выполняется. В `ПриЗаписи` обновляют связанные регистры, пишут в журнал. Обработчики менеджера работают в серверном контексте записи объекта.

## Подводные камни

- **Двойная логика.** Копия проверки в форме и менеджере расходится со временем.
- **Побочные эффекты.** Запись другого объекта внутри `ПередЗаписью` может вызвать рекурсию.
- **Групповая запись.** Обмен и загрузка обходят часть обработчиков при специальных режимах.
- **Отказ без сообщения.** Пользователь видит «запись отменена» без причины.
- **Транзакция.** Ошибка в `ПриЗаписи` после фактической записи требует отката вручную.
