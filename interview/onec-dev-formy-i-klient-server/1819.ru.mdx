# Контекст сеанса в управляемом приложении

## Суть

Контекст сеанса это изолированное состояние подключения пользователя к ИБ: параметры сеанса, временные переменные, безопасность, настройки клиента. Живёт от входа до выхода; другие сеансы его не видят.

## Что ответить на собесе

- Я использую `ПараметрыСеанса` для глобальных флагов вроде текущей организации или режима обмена.
- На собеседовании отличаю контекст сеанса от контекста формы и серверного контекста записи объекта.
- Подчёркиваю `УстановитьПараметрыСеанса` в модуле сеанса при старте.
- Честно уточняю: хранить большие таблицы в параметрах сеанса нельзя, это память сервера на каждого пользователя.
- Для фоновых заданий контекст другой; права и параметры задаются отдельно.

## Как это работает

При старте сеанса в `УстановкаПараметровСеанса` задаём значения:

```bsl
Процедура УстановкаПараметровСеанса(ИменаПараметров)
    ПараметрыСеанса.ТекущийПользователь = Пользователи.АвторизованныйПользователь();
    ПараметрыСеанса.ОсновнаяОрганизация = ПолучитьОрганизациюПоУмолчанию();
КонецПроцедуры
```

В прикладном коде читаем без повторных запросов:

```bsl
Функция РазрешенПросмотрОтчета()
    Возврат ПравоДоступа("Просмотр", Метаданные.Отчеты.Продажи);
КонецФункции
```

Права кэшируются платформой в рамках сеанса. Смена организации в интерфейсе обновляет параметр и перечитывает формы.

## Подводные камни

- **Утечка памяти.** Крупный массив в параметре сеанса умножается на число пользователей.
- **Устаревшие значения.** Изменение справочника в другом сеансе не обновляет кэш автоматически.
- **Фоновые задания.** Нет пользовательского сеанса; `ПараметрыСеанса` могут быть пусты.
- **Веб-клиент.** Таймаут неактивности обрывает сеанс, временные данные теряются.
- **Привилегированный режим.** Расширяет права внутри вызова, не меняет параметры сеанса навсегда.
