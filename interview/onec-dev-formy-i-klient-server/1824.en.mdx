# Managed form open lifecycle

## In short

Opening a managed form runs a chain of events: server creation (`ПриСозданииНаСервере`), client rendering, `ПриОткрытии`, user interaction, then `ПередЗакрытием` and `ПриЗакрытии`. Knowing the order avoids extra server hops and reading uninitialized attributes.

## How to answer in an interview

- I initialize data and rights in `ПриСозданииНаСервере`, UI tweaks in client `ПриОткрытии`.
- In an interview I list stages: parameters → server creation → client open → interaction.
- I stress `ПриОткрытии` runs on each show, including some tab switch scenarios.
- I honestly note heavy queries in `ПриСозданииНаСервере` slow every list open.
- For parameterized forms I validate `Параметры` before hitting the infobase.

## How it works

```bsl
&НаСервере
Процедура ПриСозданииНаСервере(Отказ, СтандартнаяОбработка)
    Если Параметры.Свойство("ДокументСсылка") Тогда
        Объект = Параметры.ДокументСсылка.ПолучитьОбъект();
    КонецЕсли;
    УстановитьВидимостьПоПравам();
КонецПроцедуры

&НаКлиенте
Процедура ПриОткрытии(Отказ)
    УстановитьЗаголовокФормы();
    Элементы.ГруппаДополнительно.Видимость = ПоказыватьДополнительно;
КонецПроцедуры

&НаКлиенте
Процедура ПередЗакрытием(Отказ, СтандартнаяОбработка)
    Если Модифицированность И Не ПодтверждениеЗакрытия() Тогда
        Отказ = Истина;
    КонецЕсли;
КонецПроцедуры
```

The server prepares the object and command availability; the client sets title and elements without a round trip. Close checks unsaved changes.

## Pitfalls

- **Queries in OnOpen via server call.** Extra SCALL right after creation.
- **Reject in OnCreateAtServer.** Form will not open without a clear user message.
- **Repeat initialization.** Same catalogs loaded in multiple handlers.
- **Missing parameters.** Opening without a required parameter yields an empty form or crash.
- **Dynamic lists.** Filter setup after open may need `ОбновитьОтображениеДанных`.
