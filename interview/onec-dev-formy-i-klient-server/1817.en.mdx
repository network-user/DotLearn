# Predefined handlers in the manager module

## In short

The metadata object manager module holds predefined lifecycle event handlers: `–Я–µ—А–µ–і–Ч–∞–њ–Є—Б—М—О`, `–Я—А–Є–Ч–∞–њ–Є—Б–Є`, `–Я–µ—А–µ–і–£–і–∞–ї–µ–љ–Є–µ–Љ`, and similar. The platform invokes them automatically on object operations without an explicit call from a form.

## How to answer in an interview

- I separate the manager module (logic for all instances) from the object module (a specific reference).
- In an interview I show `–Ю—В–Ї–∞–Ј = –Ш—Б—В–Є–љ–∞` in `–Я–µ—А–µ–і–Ч–∞–њ–Є—Б—М—О` for validation.
- I stress the `–°—В–∞–љ–і–∞—А—В–љ–∞—П–Ю–±—А–∞–±–Њ—В–Ї–∞` parameter to disable default behavior where allowed.
- I honestly note heavy checks in `–Я–µ—А–µ–і–Ч–∞–њ–Є—Б—М—О` slow every write; move to background when needed.
- For forms I duplicate checks on the client for UX, but the server remains the source of truth.

## How it works


When saving a catalog item from any form or programmatically, the platform first calls `–Я–µ—А–µ–і–Ч–∞–њ–Є—Б—М—О`. If `–Ю—В–Ї–∞–Ј` is true, the write is aborted. In `–Я—А–Є–Ч–∞–њ–Є—Б–Є` update related registers and log events. Manager handlers run in the server context of the object write.

## Pitfalls

- **Duplicated logic.** The same check in form and manager diverges over time.
- **Side effects.** Writing another object inside `–Я–µ—А–µ–і–Ч–∞–њ–Є—Б—М—О` can recurse.
- **Bulk write.** Exchange and load may bypass some handlers in special modes.
- **Reject without message.** Users see "write canceled" with no reason.
- **Transaction.** An error in `–Я—А–Є–Ч–∞–њ–Є—Б–Є` after the actual write needs manual rollback.
