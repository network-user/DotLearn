# Зачем в модуле формы ставят флаг «Сервер»

## Суть

Флаг «Сервер» у процедуры или функции в модуле формы (или общего модуля) указывает платформе: метод исполняется на сервере приложений, а не на рабочем месте пользователя. Это нужно для доступа к данным ИБ, транзакциям и тяжёлым вычислениям без передачи больших массивов на клиент.

## Что ответить на собесе

- Я объясняю, что `&НаСервере` и `&НаСервереБезКонтекста` фиксируют контур исполнения на стороне СУБД и сервера 1С.
- Показываю типовой вызов: клиентский обработчик кнопки вызывает серверную процедуру записи или запроса.
- Говорю про безопасность: бизнес-правила и проверки прав надёжнее выполнять там, где данные уже лежат.
- Честно добавляю: лишние серверные вызовы из каждого `ПриИзменении` поля замедляют форму; баланс важнее абстрактного «всё на сервер».
- Различаю `&НаСервере` (контекст формы доступен) и `&НаСервереБезКонтекста` (легче переиспользовать, но нельзя напрямую трогать реквизиты формы).

## Как это работает

Пользователь нажимает «Заполнить по остаткам» на форме документа. Клиент не может выполнить запрос к регистру, поэтому обработчик делегирует работу серверу:

```bsl
&НаКлиенте
Процедура ЗаполнитьПоОстаткам(Команда)

    ЗаполнитьТабличнуюЧастьНаСервере();
    ОбновитьИтогиНаКлиенте();

КонецПроцедуры

&НаСервере
Процедура ЗаполнитьТабличнуюЧастьНаСервере()

    Запрос = Новый Запрос;
    Запрос.Текст =
        "ВЫБРАТЬ
        |    Остатки.Номенклатура,
        |    Остатки.КоличествоОстаток КАК Количество
        |ИЗ
        |    РегистрНакопления.ТоварыНаСкладах.Остатки(, Склад = &Склад) КАК Остатки";
    Запрос.УстановитьПараметр("Склад", Объект.Склад);

    Объект.Товары.Очистить();
    Выборка = Запрос.Выполнить().Выбрать();
    Пока Выборка.Следующий() Цикл
        Строка = Объект.Товары.Добавить();
        ЗаполнитьЗначенияСвойств(Строка, Выборка);
    КонецЦикла;

КонецПроцедуры
```

На сервере выполняется запрос, заполняется `Объект` формы, клиент получает уже подготовленную модель и обновляет интерфейс. Сетевой обмен ограничен сериализованным состоянием формы, а не полной выборкой построчно.

Для операций без привязки к реквизитам формы использую `&НаСервереБезКонтекста`: например, проверка уникальности кода справочника по переданным параметрам.

## Подводные камни

**Частые серверные вызовы.** Каждый вызов это round-trip; автозаполнение на каждый символ в поле убивает UX.

**Тяжёлая логика в `&НаСервере` с контекстом формы.** Долгая процедура держит контекст и блокирует интерактив; для пакетной обработки лучше фоновое задание.

**Смешение контуров.** Вызов клиентского API из серверной процедуры без явного возврата на клиент даёт ошибку компиляции или runtime.

**Утечка данных.** Возвращать на клиент «на всякий случай» полные таблицы из запроса опасно и для производительности, и для разграничения доступа.

**Расширения.** Серверный метод в расширении может перекрывать типовой; порядок `&Перед` / `&После` и аннотаций нужно документировать в команде.
