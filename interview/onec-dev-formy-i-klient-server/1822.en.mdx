# How to reduce client-server calls in forms

## In short

Each `&НаКлиенте` → `&НаСервере` hop is a network round trip with serialization. Optimization means batch server methods, client cache, no server calls inside loops, and passing only required fields.

## How to answer in an interview

- I look in the technological log for frequent short `SCALL` events: typical chatter.
- In an interview I show one server method returning a structure with multiple results.
- I stress caching catalogs on the client for the session when data rarely changes.
- I honestly note aggressive cache shows stale balances; invalidate on events.
- I validate field format on the client and business rules on the server in one call.

## How it works

Bad: server on every table row.

```bsl
&НаКлиенте
Процедура ПересчитатьСтроки()
    Для Каждого ИдентификаторСтроки Из ИдентификаторыСтрок Цикл
        ОбновитьЦенуСтрокиНаСервере(ИдентификаторСтроки);
    КонецЦикла;
КонецПроцедуры
```

Better: one call with an array.

```bsl
&НаСервере
Процедура ОбновитьЦеныПакетом(ТаблицаСтрок)
    Для Каждого СтрокаТаблицы Из ТаблицаСтрок Цикл
        СтрокаТаблицы.Цена = ПолучитьЦену(СтрокаТаблицы.Номенклатура);
    КонецЦикла;
КонецПроцедуры
```

Client cache:

```bsl
&НаКлиенте
Перем КэшВидовНоменклатуры;

Функция ВидыНоменклатуры()
    Если КэшВидовНоменклатуры = Неопределено Тогда
        КэшВидовНоменклатуры = ЗагрузитьВидыНаСервере();
    КонецЕсли;
    Возврат КэшВидовНоменклатуры;
КонецФункции
```

## Pitfalls

- **False economy.** One huge call with a full document value table blocks the UI longer than chatter.
- **Stale cache.** Users miss fresh balances after posting in another window.
- **Server in `OnChange`.** Every keystroke hits the server; use deferred recalculation.
- **Web client.** Latency is higher than thin client; optimization matters more.
- **No measurement.** Tuning blindly without profiling and the tech log.
