# Why &AtServerNoContext cannot access form data

## In short

A procedure with `&–Э–∞–°–µ—А–≤–µ—А–µ–С–µ–Ј–Ъ–Њ–љ—В–µ–Ї—Б—В–∞` runs on the server outside the context of a specific form and the caller's session state. Therefore `–Ю–±—К–µ–Ї—В`, `–≠–ї–µ–Љ–µ–љ—В—Л`, form attributes, and client variables are unavailable: they exist only in the form's client context or in `&–Э–∞–°–µ—А–≤–µ—А–µ` server context.

## How to answer in an interview

- I pass required values as explicit parameters when calling from the client or `&–Э–∞–°–µ—А–≤–µ—А–µ`.
- In an interview I explain that a context-free call is cheaper and safer for utilities and background logic.
- I stress that accessing `–≠–ї–µ–Љ–µ–љ—В—Л.–Я–Њ–ї–µ.–Ч–љ–∞—З–µ–љ–Є–µ` inside a no-context method fails.
- I honestly note that working with form attributes needs `&–Э–∞–°–µ—А–≤–µ—А–µ`, not no context.
- Common modules with Server and server call flags are similarly isolated but invocation rules differ.

## Pitfalls

- **Hidden form access.** Using a form global from a no-context module yields `–Э–µ–Њ–њ—А–µ–і–µ–ї–µ–љ–Њ` or an error.
- **Oversized parameters.** Passing whole `–Ю–±—К–µ–Ї—В` drags a large payload and negates no-context benefits.
- **Rights.** No-context code still obeys RLS; privileged mode is enabled deliberately.
- **Reuse.** Function moved to a common module still references `–Ю–±—К–µ–Ї—В`.
- **Debugging.** The stack shows a no-context frame without form attribute locals.
