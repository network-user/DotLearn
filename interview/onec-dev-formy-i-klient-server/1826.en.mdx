# Form data source and server binding

## In short

A form data source is an attribute or dynamic list bound to UI elements. The main form object (`–ě–Ī—ä–Ķ–ļ—ā`, `–ó–į–Ņ–ł—Ā—Ć`, `–°–Ņ–ł—Ā–ĺ–ļ`) lives on the server; on open and changes the platform syncs server state to the client view via serialization.

## How to answer in an interview

- I separate an attribute of type `–Ē–ĺ–ļ—É–ľ–Ķ–Ĺ—ā–ě–Ī—ä–Ķ–ļ—ā` from a dynamic list with a custom query.
- In an interview I explain the path: server init ‚Üí snapshot to client ‚Üí write-back on save.
- I stress tabular sections in `–ě–Ī—ä–Ķ–ļ—ā` edit on the client but persist on the server.
- I honestly note extra form attributes inflate every round trip.
- For lists I set server filters in `–ü—Ä–ł–°–ĺ–∑–ī–į–Ĺ–ł–ł–Ě–į–°–Ķ—Ä–≤–Ķ—Ä–Ķ`, not manual full value table loads.

Dynamic list on a list form: source `–°–Ņ–ł—Ā–ĺ–ļ`, query runs on the server, the client gets pages and filters. Filter changes trigger a server refresh of the dataset.

## Pitfalls

- **Desync.** Long editing without refresh while server data already changed.
- **Heavy –ě–Ī—ä–Ķ–ļ—ā.** Tabular section with thousands of rows slows open; use paging or separate forms.
- **Dynamic list without indexes.** Filter on non-indexed field hurts SQL.
- **Temporary attributes.** Forgotten large value tables ship to the client whole.
- **Save without check.** Client sent stale `–ě–Ī—ä–Ķ–ļ—ā`, server overwrote others' changes.
