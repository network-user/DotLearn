# Чрезмерные клиент-серверные переходы: признаки и лечение

## Суть

Чрезмерные клиент-серверные переходы это паттерн, когда логика дробится на множество коротких серверных вызовов из клиентских циклов и обработчиков событий. Интерфейс «подвисает», в ТЖ сотни `SCALL`, сервер тратит время на сериализацию вместо полезной работы.

## Что ответить на собесе

- Я включаю замер в ТЖ и считаю вызовы на одно действие пользователя (открытие формы, пересчёт таблицы).
- На собеседовании привожу антипример: проверка каждой строки отдельным `&НаСервере` в цикле.
- Подчёркиваю перенос валидации формата на клиент, бизнес-проверки пакетом на сервер.
- Честно уточняю: полный отказ от сервера на клиенте нарушает безопасность и RLS.
- После рефакторинга сравниваю время сценария и число `SCALL` до и после.

## Как это работает

Антипаттерн:

```bsl
&НаКлиенте
Процедура ПроверитьВсеСтроки(МассивСтрок)
    Для Каждого СтрокаДанных Из МассивСтрок Цикл
        Если Не СтрокаПрошлаПроверкуНаСервере(СтрокаДанных) Тогда
            Продолжить;
        КонецЕсли;
        ОбработатьСтрокуЛокально(СтрокаДанных);
    КонецЦикла;
КонецПроцедуры
```

Каждая итерация это отдельный round-trip. Исправление: собрать массив, один вызов:

```bsl
&НаСервере
Функция ОтфильтроватьВалидныеСтроки(МассивСтрок)
    Результат = Новый Массив;
    Для Каждого СтрокаДанных Из МассивСтрок Цикл
        Если ПроверитьБизнесПравила(СтрокаДанных) Тогда
            Результат.Добавить(СтрокаДанных);
        КонецЕсли;
    КонецЦикла;
    Возврат Результат;
КонецФункции
```

## Подводные камни

- **Микропакеты.** Объединили вызовы, но всё равно передаёте всю форму целиком каждый раз.
- **События полей.** `ПриИзменении` с сервером на каждый ввод неизбежно даёт лавину вызовов.
- **Скрытые вызовы.** Стандартные команды платформы тоже ходят на сервер; не всё видно в своём коде.
- **Веб и мобильный.** Те же паттерны болезненнее из-за сети.
- **Преждевременная оптимизация.** Рефакторинг без профиля усложняет код без выигрыша.
