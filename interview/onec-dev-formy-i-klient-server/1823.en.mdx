# Excessive client-server transitions: signs and fixes

## In short

Excessive client-server transitions occur when logic is split into many short server calls from client loops and event handlers. The UI stutters, the tech log shows hundreds of `SCALL` entries, and the server spends time on serialization instead of useful work.

## How to answer in an interview

- I enable tech log measurement and count calls per user action (open form, recalc table).
- In an interview I give an anti-pattern: checking each row with a separate `&НаСервере` in a loop.
- I stress moving format validation to the client and business checks in one server batch.
- I honestly note eliminating all server access on the client breaks security and RLS.
- After refactoring I compare scenario time and `SCALL` count before and after.

## How it works

Anti-pattern:

```bsl
&НаКлиенте
Процедура ПроверитьВсеСтроки(МассивСтрок)
    Для Каждого СтрокаДанных Из МассивСтрок Цикл
        Если Не СтрокаПрошлаПроверкуНаСервере(СтрокаДанных) Тогда
            Продолжить;
        КонецЕсли;
        ОбработатьСтрокуЛокально(СтрокаДанных);
    КонецЦикла;
КонецПроцедуры
```

Each iteration is a separate round trip. Fix: collect the array, one call:

```bsl
&НаСервере
Функция ОтфильтроватьВалидныеСтроки(МассивСтрок)
    Результат = Новый Массив;
    Для Каждого СтрокаДанных Из МассивСтрок Цикл
        Если ПроверитьБизнесПравила(СтрокаДанных) Тогда
            Результат.Добавить(СтрокаДанных);
        КонецЕсли;
    КонецЦикла;
    Возврат Результат;
КонецФункции
```

## Pitfalls

- **Micro-batches.** Calls merged but the whole form is still sent every time.
- **Field events.** `OnChange` with server on every keystroke inevitably floods calls.
- **Hidden calls.** Standard platform commands also hit the server; not all are visible in your code.
- **Web and mobile.** Same patterns hurt more due to network latency.
- **Premature optimization.** Refactoring without a profile complicates code with no gain.
