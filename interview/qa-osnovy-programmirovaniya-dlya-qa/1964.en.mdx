# OOP for testers: why should QA care about objects and classes?

## In short

Object-oriented programming builds software from objects: class instances with fields and methods. Four pillars: encapsulation, inheritance, polymorphism, and abstraction. QA does not need to ship frameworks, but OOP literacy helps read code, design Page Objects, and tell a class contract bug from an integration bug.

## How to answer in an interview

- For me OOP organizes the domain: an Order entity encapsulates status and transition rules instead of scattered ifs in the UI.
- Encapsulation explains why tests should not poke internal fields: I verify public behavior like a user would.
- Inheritance and polymorphism warn about hierarchy regressions: a fix in a base class can break every child.
- Honestly: OOP knowledge does not replace domain test design; it speeds up talks with developers and automation.

## How it works

**Encapsulation** hides implementation behind methods. `BankAccount` does not expose balance directly; it changes through `deposit` and `withdraw` with validation. The tester checks "cannot go negative" without knowing how the number is stored.

**Inheritance** reuses common code: `SavingsAccount extends BankAccount` adds interest. Regression risk: changing `withdraw` in the parent breaks savings.

**Polymorphism** calls one interface for different implementations: `Notifier.send()` for Email and SMS. Tests must cover each implementation, not only the base class.

**Abstraction** keeps essentials: a `PaymentGateway` interface without Stripe or PayPal details. Mocks in automation often implement the same contract.

Page Object in automation is direct OOP for QA:

```gherkin
Feature: Login through a Page Object

  Scenario: Successful sign-in hides selector details
    Given the "/login" page is open
    When the user enters valid credentials via LoginPage
    Then DashboardPage shows a greeting
    And internal form CSS selectors do not appear in scenario steps
```

Matching skeleton (idea, not production copy):

```gherkin
# Pseudo Page Object structure (for Gherkin steps)
# LoginPage.enterCredentials(email, password) -> submit()
# DashboardPage.isVisible() -> assert greeting
```

In bug reports I describe the object and its state ("cart after applyPromo returns total=0"), not random coordinate clicks.

## Pitfalls

- **Class vs object confusion.** A class is a template; an object is a concrete instance. Testing "the class" without data is often meaningless.
- **Fragile UI inheritance.** Page Objects inherited to save lines break half the suite when a base locator changes.
- **Polymorphism without a coverage matrix.** One happy path on an interface misses implementations; classic miss when email is green but SMS is not.
- **Breaking encapsulation in tests.** Reflection into private fields fails on refactor without behavior change.
- **OOP for its own sake.** A 2000-line god object hides defect ownership; QA should name which component owns the symptom.
