# Four OOP pillars: why should QA care about encapsulation and polymorphism?

## In short

OOP builds systems from objects with data and behavior. The four basics are encapsulation (hide state behind methods), inheritance (shared base and extension), polymorphism (one interface, many implementations), and abstraction (deal with the entity without extra detail). QA uses this when reading automation code, Page Objects, and when judging product testability.

## How to answer in an interview

- Encapsulation: private fields, meaningful public methods; the bug "bypassed validation by writing the field directly" is caught in model review.
- Inheritance: base `PaymentMethod` with `Card`, `Wallet`; shared checks in the base test class, specifics in children.
- Polymorphism: `process(payment)` does not know the concrete type; in tests I substitute a mock/stub for each implementation.
- Honestly: deep inheritance trees hurt regression; I prefer composition over deep extends, but I still name all four principles.

## How it works

Page Object as UI encapsulation:

```gherkin
Feature: Encapsulation in a Page Object

  Scenario: The test does not touch selectors directly
    Given LoginPage hides locator "#pwd-field"
    When the test calls loginPage.signIn("qa", "secret")
    Then POST /auth is sent without DOM access from the scenario
```

Polymorphism in test doubles:

```gherkin
Feature: Polymorphic notification delivery

  Scenario Outline: Notification channel is swapped without changing the scenario
    Given OrderNotifier uses channel <channel>
    When order "ord-51" is created
    Then the client receives confirmation via <channel>

    Examples:
      | channel   |
      | EmailSink |
      | SmsSink   |
      | PushSink  |
```

Abstraction: test `ShapeArea` checks the `area()` contract, not each shape formula. Inheritance: `AdminUser extends User` adds `grantRole()`; base login smoke is reused.

## Pitfalls

- **Treating everything as "is-a" inheritance.** Logic duplicated in children; a regression in the parent breaks half the suite silently.
- **Breaking encapsulation in tests.** `@VisibleForTesting` fields in prod code; test is green, contract leaks outward.
- **Polymorphism without a contract.** Implementations throw differently; one assert fails across all.
- **Testing only the concrete class.** Alternate DI implementation never covered.
- **OOP jargon with no QA link.** In interviews I tie it to Page Object, fixtures, mock objects.
