# Классы в ООП: что QA проверяет в объектной модели приложения?

## Суть

Класс это blueprint объекта: fields (state) и methods (behavior). ООП даёт encapsulation, inheritance, polymorphism. QA не пишет классы, но понимает их для тестов: разные subtypes через общий interface, override behavior, constructor invariants.

## Что ответить на собесе

- Class vs instance: один `User` template, много `userAlice`, `userBob`.
- Inheritance: `Admin extends User` добавляет permissions; тестирую override методов.
- Polymorphism: payment `Card` vs `Wallet` один interface `pay()`.
- Encapsulation: private fields меняются только через public API.
- Честно: over-mocked unit tests green, integration между классами падает; нужны сквозные кейсы.

## Как это работает

```gherkin
Feature: Polymorphic notification sender

  Scenario: EmailNotification sends SMTP
    Given notifier type EmailNotification
    When send alert "Order shipped"
    Then SMTP mock получил message

  Scenario: SmsNotification sends SMS gateway
    Given notifier type SmsNotification
    When send alert "Order shipped"
    Then SMS provider вызван

  Scenario: Constructor invariant
    Given new Order без items
    Then создание rejected или total равен 0 per domain rule
```

Наследование: base `Animal.speak()` vs `Dog.speak()` override. Composition over inheritance снижает fragile base class; QA фокус на public contract класса.

## Подводные камни

- **Liskov violation.** Subclass ломает ожидания base в UI.
- **God class.** Один Order делает payment, tax, shipping; untestable matrix.
- **Static singleton state.** Test order зависит от порядка execution.
- **Reflection bypass private.** Security assumption false.
- **DTO vs Entity confusion.** API field не маппится на persisted model.
