# Модель ключ-значение: где QA встречает её в HTTP и JSON?

## Суть

Ключ-значение связывает уникальный идентификатор с данными: быстрый lookup без перебора списка. В QA это JSON-тела, query string, HTTP headers, env vars и cookie attributes. Понимание формата помогает строить parametrized tests и ловить encoding bugs.

## Что ответить на собесе

- JSON object: строковые ключи, значения любого типа; валидирую schema.
- Query params: `?filter=active&page=2`; проверяю URL-encoding спецсимволов.
- Headers: `Content-Type: application/json`; case-insensitive имена в HTTP/2.
- Cookie: `session=abc; Path=/; HttpOnly` тоже пары атрибутов.
- Честно: duplicate keys в query (legacy servers) поведение нестандартно; фиксирую фактическое.

## Как это работает

Примеры из одного GET:

```http
GET /search?query=manual+testing&sort=asc HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer eyJ...
```

JSON response:

```json
{
  "total": 12,
  "items": [{ "id": "q-1", "title": "RBAC basics" }]
}
```

Gherkin:

```gherkin
Feature: Key-value в HTTP слоях

  Scenario: Query param передаёт фильтр
    When GET /search?query=RBAC
    Then каждый item в items содержит "RBAC" в title или tags

  Scenario: Header Content-Type влияет на парсинг
    When POST /api/users с телом без Content-Type application/json
    Then сервер возвращает 415 или корректный fallback по контракту

  Scenario: URL-encoding пробела в query
    When GET /search?query=manual%20testing
    Then backend декодирует значение как "manual testing"
```

В DevTools → Network каждая вкладка (Headers, Query, Payload) показывает пары. Postman Collection variables тоже KV store для reuse.

## Подводные камни

- **Plus vs space в query.** `+` и `%20` декодируются по-разному в старых libs.
- **Duplicate JSON keys.** Последний wins; parser-dependent.
- **Header injection через CRLF в value.** Старые прокси уязвимы.
- **Case sensitivity keys в JSON vs headers.** JSON keys case-sensitive.
- **Nested KV путают с flat env.** `DB__HOST` vs `DB.HOST` в config.
