# Data structures: which ones does QA actually see in APIs?

## In short

Core structures: array, list, stack (LIFO), queue (FIFO), set, map/hash map, tree, graph. In manual QA you most often see array, map, and queue semantics (Kafka, job queue), less often explicit tree/graph in payload. Understanding helps design edge cases and load scenarios.

## How to answer in an interview

- Array/list: ordered collection in JSON responses.
- Stack: undo/redo, browser history; LIFO in UI state.
- Queue: async jobs, message broker; I check ordering and DLQ.
- Hash map: JSON object, headers, query params; O(1) lookup.
- Honestly: in interviews it is enough to tie structure to a test scenario, not prove big-O on a whiteboard.

## How it works

| Structure | Principle | QA example |
|-----------|-----------|------------|
| Stack | LIFO | Back button after 3 navigations |
| Queue | FIFO | Order processing jobs 1,2,3 |
| Set | unique | Tag list without duplicates |
| Map | key→value | userId → profile cache |
| Tree | hierarchy | Category tree in menu |

Gherkin:

```gherkin
Feature: Queue semantics in checkout

  Scenario: Jobs processed FIFO
    Given three orders A, B, C created in order
    When single-thread worker
    Then completed status first for A, then B, then C

  Scenario: Stack undo cancels last action
    Given user added item X, then Y
    When Undo is pressed
    Then cart contains only X
```

Graph appears in permission inheritance and social "friends of friends"; I test cycle detection and depth limit.

## Pitfalls

- **Queue without idempotency.** Retry creates duplicate orders.
- **Stack overflow in UI wizard.** 50 steps back breaks state.
- **Set implemented as list.** Duplicate permissions in token.
- **Tree reorder race.** Drag-drop concurrent edit.
- **Hash collision rarely tested.** Only synthetic load on hash map cache.
