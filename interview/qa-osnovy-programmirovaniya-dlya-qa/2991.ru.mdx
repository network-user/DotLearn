# Циклы for и while: как QA применяет их в data-driven тестах?

## Суть

Цикл for перебирает известную коллекцию или диапазон итераций. Цикл while повторяет блок, пока условие истинно, когда число шагов заранее неизвестно. В QA это основа parametrized tests, polling wait и brute-force boundary checks.

## Что ответить на собесе

- for each row in CSV: login, action, expected status.
- while elapsed < timeout: poll job status until completed.
- Избегаю infinite while без sleep и max iterations.
- В Gherkin Scenario Outline заменяет ручной for для readable cases.
- Честно: copy-paste 20 одинаковых steps хуже одного loop в script; выбираю по аудитории отчёта.

## Как это работает

```gherkin
Feature: Parametrized boundary checks

  Scenario Outline: Invalid age rejected
    When POST /api/users с age <age>
    Then статус <status>

    Examples:
      | age | status |
      | -1  | 400    |
      | 0   | 400    |
      | 150 | 400    |

Feature: Polling async job

  Scenario: while job pending
    Given POST /api/export создан jobId
    When QA опрашивает GET /api/jobs/{id} каждые 2s до 60s
    Then финальный status completed
```

for по индексу полезен, когда нужен previous/next element. while для retry flaky network с backoff cap.

## Подводные камни

- **Busy loop без delay.** DDoS собственного staging API.
- **Off-by-one в range.** Пропущен boundary value 0 или max.
- **Mutating collection while iterating.** Skip every second element silently.
- **Break forgot in while.** Hang CI runner.
- **Scenario Outline 500 rows.** Report unreadable; split by domain.
