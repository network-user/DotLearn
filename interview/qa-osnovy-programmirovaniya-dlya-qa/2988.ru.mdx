# Объекты внутри JSON-массива: как QA это валидирует?

## Суть

JSON-массив `[...]` может содержать примитивы, вложенные массивы и объекты `{...}`. Типичный API pattern: `{ "items": [ { "id": 1 }, { "id": 2 } ] }`. QA проверяет schema каждого элемента, порядок, пустой массив и граничные типы.

## Что ответить на собесе

- Да, массив объектов стандарт для list endpoints.
- Валидирую JSON Schema: `items` type object, required fields per element.
- Проверяю `Array.isArray`, длину, уникальность id внутри collection.
- Пустой массив `[]` vs отсутствие ключа разное поведение UI.
- Честно: heterogenous array `[1, "x", {}]` редок в public API; если есть, фиксирую контракт.

## Как это работает

```json
[
  { "sku": "A-01", "qty": 3 },
  { "sku": "B-02", "qty": 1 }
]
```

Gherkin:

```gherkin
Feature: Массив объектов в ответе

  Scenario: Каждый элемент соответствует schema
    When GET /api/products
    Then response.items is array
    And each element has keys sku, qty, price
    And each element.sku is non-empty string

  Scenario: Поиск объекта по полю в массиве
    Given items содержит объект с name "Alice"
    When QA фильтрует items where name equals "Alice"
    Then найден ровно один элемент

  Scenario: Пустой массив не null
    When GET /api/products?category=empty
    Then items equals []
    And items is not null
```

В Postman Tests: `pm.expect(json.items.every(o => o.id)).to.be.ok)`.

## Подводные камни

- **items как object вместо array.** `{ "items": { "0": {...} } }` ломает клиентов.
- **Null element внутри array.** `[ null, {...} ]` редко описан в schema.
- **Duplicate ids в list.** Pagination total не сходится.
- **Huge array без paging.** Performance и timeout не тестируют.
- **Strict order assumption.** Backend sort changed, UI regression.
