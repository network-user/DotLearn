# Функция и метод: зачем QA читает сигнатуры в коде?

## Суть

Функция это именованный блок логики с параметрами и return value, вызываемый независимо. Метод привязан к объекту/классу и может менять его state через `this`/`self`. QA использует это для white-box: какие аргументы валидируются, side effects, idempotency endpoint handler vs domain service.

## Что ответить на собесе

- Function: `calculateDiscount(price, pct)` pure, без hidden state.
- Method: `cart.addItem(sku)` меняет internal list объекта Cart.
- REST handler часто thin: parse → call service method → map response.
- Читаю swagger + исходник, чтобы найти validation до service layer.
- Честно: без доступа к коду тестирую black-box по контракту; с кодом ищу early return branches.

## Как это работает

```gherkin
Feature: Поведение методов корзины

  Scenario: addItem увеличивает count
    Given пустая корзина
    When вызван метод addItem sku "BOOK-1" qty 2
    Then getTotalItems возвращает 2

  Scenario: standalone function не мутирует input
    Given price 100 и discount 10
    When calculateFinal(price, discount) вызвана
    Then price остаётся 100
    And return value равен 90

  Scenario: static util vs instance method
    Given два Cart instance
    When addItem только в cartA
    Then cartB остаётся пустой
```

Constructor (`__init__`) инициализирует defaults; getter/setter контролируют encapsulation. В API тестах «метод» часто соответствует одному HTTP verb + path.

## Подводные камни

- **Static mutable state.** Function хранит cache между вызовами, flaky tests.
- **Method with side effect без return.** QA не видит failure в response body.
- **Overloaded signatures.** Wrong param type silently coerced in JS.
- **Private method bypass через reflection.** Security test miss.
- **Same name different modules.** Import wrong helper in microservice.
