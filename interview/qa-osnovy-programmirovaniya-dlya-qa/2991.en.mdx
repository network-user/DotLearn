# For and while loops: how does QA use them in data-driven tests?

## In short

A for loop iterates a known collection or iteration range. A while loop repeats a block while a condition holds when step count is unknown upfront. In QA this underpins parametrized tests, polling wait, and brute-force boundary checks.

## How to answer in an interview

- for each row in CSV: login, action, expected status.
- while elapsed < timeout: poll job status until completed.
- I avoid infinite while without sleep and max iterations.
- Gherkin Scenario Outline replaces manual for for readable cases.
- Honestly: copy-paste 20 identical steps is worse than one loop in a script; I choose by report audience.

## How it works

```gherkin
Feature: Parametrized boundary checks

  Scenario Outline: Invalid age rejected
    When POST /api/users with age <age>
    Then status <status>

    Examples:
      | age | status |
      | -1  | 400    |
      | 0   | 400    |
      | 150 | 400    |

Feature: Polling async job

  Scenario: while job pending
    Given POST /api/export created jobId
    When QA polls GET /api/jobs/{id} every 2s up to 60s
    Then final status is completed
```

for by index helps when previous/next element matters. while for retry on flaky network with backoff cap.

## Pitfalls

- **Busy loop without delay.** DDoS your own staging API.
- **Off-by-one in range.** Missed boundary value 0 or max.
- **Mutating collection while iterating.** Skip every second element silently.
- **Forgot break in while.** Hang CI runner.
- **Scenario Outline 500 rows.** Report unreadable; split by domain.
