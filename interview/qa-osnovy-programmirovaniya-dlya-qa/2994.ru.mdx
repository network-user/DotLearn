# Массив vs объект в JSON: как QA не путает контракт?

## Суть

Массив `[]` это упорядоченный список с numeric index доступом. Объект `{}` это набор пар ключ-значение с string keys. В REST list endpoints возвращают array; single resource и envelope metadata object. Путаница типов ломает парсинг на клиенте.

## Что ответить на собесе

- Array: `[1,2,3]`, `items[0].id`, length, order matters.
- Object: `{ "id": 1, "name": "Ann" }`, dot/bracket access by key.
- Array of objects для коллекций; nested object для complex fields.
- JSON Schema `type: array` vs `type: object` в CI contract tests.
- Честно: JS typeof array === "object"; проверяю Array.isArray, не typeof alone.

## Как это работает

```gherkin
Feature: Array vs object в контракте

  Scenario: List endpoint returns array at root or items key
    When GET /api/users
    Then body.items is array
    And typeof body.items is not plain object map

  Scenario: Single resource is object
    When GET /api/users/u-1
    Then body is object with id field
    And body is not array

  Scenario: Wrong type rejected on input
    When POST /api/bulk body {"ids": {"a": 1}}
    Then статус 400 because ids must be array
```

Массив однотипных id `[1,2,3]` vs object map `{ "1": true, "2": false }` для flags. OpenAPI `oneOf` документирует union если допустимо.

## Подводные камни

- **Object with numeric keys mimics array.** `{ "0": "a", "1": "b" }` без length.
- **Empty object instead of empty array.** UI `.map` crash.
- **Array where keyed lookup expected.** O(n) scan hides perf bug.
- **Swagger says array, backend returns object on error.** Parser throws.
- **Tuple vs array not documented.** Fixed length 2 array treated as list.
