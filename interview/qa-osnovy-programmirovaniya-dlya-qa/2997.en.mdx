# OOP inheritance: what regressions does QA look for?

## In short

Inheritance lets a child class receive parent fields and methods and override them. It reduces duplication but couples hierarchies: changing the base class affects all descendants. QA checks override behavior, base vs derived contract, and regression when refactoring the parent.

## How to answer in an interview

- `class Admin extends User` inherits login, adds `manageUsers()`.
- Override: `Dog.speak()` instead of `Animal.speak()`; I test polymorphic call.
- Base test class with shared `setUp()` for descendant suite in automation.
- Honestly: deep inheritance tree is worse than composition; bugs often come from grandparent change.

## How it works

```gherkin
Feature: Inheritance in pricing rules

  Scenario: Base customer standard price
    Given order type StandardCustomer
    When calculateTotal 1000
    Then discount is 0

  Scenario: VIP overrides parent discount
    Given order type VipCustomer extends StandardCustomer
    When calculateTotal 1000
    Then discount is 10 percent

  Scenario: Parent change breaks child
    Given parent ShippingCalculator added fuel surcharge
    When ExpressShipping inherits without override
    Then total includes surcharge per new base behavior
```

On parent refactor QA runs smoke on all known subtypes. Interface inheritance (`implements Payable`) vs class inheritance carry different risks.

## Pitfalls

- **Fragile base class.** Protected field rename breaks child.
- **Override forgot super().** Child skips parent validation.
- **Diamond problem multiple inheritance.** Ambiguous method in mixins.
- **Test only leaf class.** Middle abstract class untested.
- **Breaking LSP.** Child throws where parent promised not to.
