# List, set, and map: when to use each in testing?

## In short

A list (array) holds ordered elements with index access. A set holds unique elements without order. A map (dictionary) links unique keys to values. QA sees them in JSON API responses, fixtures, and collection assertions.

## How to answer in an interview

- List: order matters (history events, queue items); index starts at zero.
- Set: uniqueness needed (tag ids, permission names) and set operations.
- Map: lookup by key (userId → profile, header → value).
- In assertions I compare structure type, not only string representation.
- Honestly: a JSON object is technically unordered; for order-sensitive UI I check array, not map.

## How it works

```gherkin
Feature: Data structures in API responses

  Scenario: List preserves element order
    Given GET /api/feed returns JSON array
    When QA reads items[0].id and items[1].id
    Then order matches publishedAt desc sort

  Scenario: Set of tag ids without duplicates
    Given POST /api/posts with tags ["qa", "qa", "api"]
    Then response tags contain exactly 2 unique values

  Scenario: Map keyed by id
    Given GET /api/users/map returns object keyed by userId
    When key "u-42" is present
    Then value contains email field
    And key "u-99" is absent or null per contract
```

Lists support append/remove by index; value search is O(n). Sets drop duplicates on insert; `in` and union/intersection beat manual dedupe in a list. Maps give O(1) lookup; keys are immutable (string, number).

In Postman/JavaScript: `Array.isArray(data)`, `new Set(arr).size === arr.length`, `Object.keys(obj).includes('id')`.

## Pitfalls

- **Comparing list vs set via JSON.stringify.** `[1,2]` and `{1,2}` in JS set look different.
- **Sparse array.** `[,,,1]` length 4, QA assumes three elements.
- **Dict key type coercion.** `"42"` and `42` are different keys in some languages.
- **Mutability shared reference.** Changed list in one step, broke another.
- **Ordered set illusion in JSON.** Object keys do not guarantee order in old parsers.
