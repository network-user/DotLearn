# JSON array: syntax and common API testing mistakes

## In short

A JSON array uses square brackets, comma-separated elements: `["a", 1, true, null, {}]`. Indexing starts at zero when parsed in code. QA checks trailing comma rejection, nested arrays, and schema minItems/maxItems.

## How to answer in an interview

- Syntax: `[ elem, elem ]`; empty array `[]` is valid.
- Elements of any JSON type, including objects and nested arrays.
- Nested array in object: `{ "tags": ["qa", "api"] }`.
- Invalid JSON: trailing comma, single quotes; array of objects still needs quoted keys.
- Honestly: Postman prettify hides invalid JSON; I send raw body and watch 400 parser error.

## How it works

```json
{
  "results": [
    { "id": "r1", "score": 98 },
    { "id": "r2", "score": 87 }
  ],
  "meta": { "page": 1 }
}
```

Gherkin:

```gherkin
Feature: JSON array syntax

  Scenario: Valid empty array
    When POST /api/clear body {"ids": []}
    Then status is 200
    And affected count is 0

  Scenario: Trailing comma invalid
    When raw body [1, 2, ]
    Then parser rejects before request reaches business logic

  Scenario: Index access after parse
    Given response array length 3
    Then element at index 0 exists
    And index 3 out of range handled per client spec
```

RFC 8259: array grammar `[ value *("," value) ]`. JavaScript `JSON.parse` is strict; Python `json.loads` likewise.

## Pitfalls

- **Trailing comma from JS export.** API 400 in production, worked in browser console.
- **Mixed quote styles.** Invalid strict JSON.
- **Array serialized as string.** `"[1,2]"` double encoding.
- **maxItems not enforced.** DoS huge payload.
- **Order assumed stable without sort param.** Flaky list comparison.
