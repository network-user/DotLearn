# Function vs method: why does QA read signatures in code?

## In short

A function is a named logic block with parameters and return value, called independently. A method belongs to an object/class and may change its state via `this`/`self`. QA uses this for white-box: which arguments are validated, side effects, idempotency of endpoint handler vs domain service.

## How to answer in an interview

- Function: `calculateDiscount(price, pct)` pure, no hidden state.
- Method: `cart.addItem(sku)` changes internal list of Cart object.
- REST handler is often thin: parse → call service method → map response.
- I read swagger plus source to find validation before service layer.
- Honestly: without code I test black-box per contract; with code I hunt early return branches.

## How it works

```gherkin
Feature: Cart method behavior

  Scenario: addItem increases count
    Given empty cart
    When method addItem sku "BOOK-1" qty 2 is called
    Then getTotalItems returns 2

  Scenario: standalone function does not mutate input
    Given price 100 and discount 10
    When calculateFinal(price, discount) is called
    Then price stays 100
    And return value equals 90

  Scenario: static util vs instance method
    Given two Cart instances
    When addItem only on cartA
    Then cartB stays empty
```

Constructor (`__init__`) sets defaults; getter/setter control encapsulation. In API tests a "method" often maps to one HTTP verb + path.

## Pitfalls

- **Static mutable state.** Function holds cache between calls, flaky tests.
- **Method with side effect and no return.** QA misses failure in response body.
- **Overloaded signatures.** Wrong param type silently coerced in JS.
- **Private method bypass via reflection.** Security test miss.
- **Same name different modules.** Import wrong helper in microservice.
