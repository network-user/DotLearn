# if/else: как QA покрывает ветвления в acceptance-тестах?

## Суть

Условие if/else выбирает путь выполнения по boolean expression. Для QA каждая значимая ветка это отдельный тест-кейс: true path, false path, else-if chain, guard clauses. Decision coverage важнее количества happy-path сценариев.

## Что ответить на собесе

- Минимум два кейса на каждое условие: граница true и false.
- Таблица решений для combo условий (role AND balance AND feature flag).
- В automation: assert expected branch via response code/message.
- Ternary и early return эквивалентны if/else, ищу все exit points.
- Честно: 100% branch coverage не цель manual QA; приоритизирую risk и частоту использования.

## Как это работает

```gherkin
Feature: Скидка по сумме заказа

  Scenario: if total >= 1000 применяется premium discount
    Given корзина total 1200
    When checkout
    Then discountPercent равен 15

  Scenario: else ветка без premium
    Given корзина total 800
    When checkout
    Then discountPercent равен 0

  Scenario: else-if tier medium
    Given корзина total 500
    When checkout
    Then discountPercent равен 5
```

В коде review ищу `if (user.isVip())` без negative test для non-Vip. Nested if повышает cyclomatic complexity; matrix test снижает пропуск веток.

## Подводные камни

- **Missing else.** Null/undefined попадает в необработанный path → 500.
- **Truthy coercion.** `"0"` и `0` разные в JS if.
- **Duplicate condition branches.** Dead code маскирует bug.
- **Feature flag only in if.** Flag off → untested else forever.
- **UI hides else branch.** API still reachable.
