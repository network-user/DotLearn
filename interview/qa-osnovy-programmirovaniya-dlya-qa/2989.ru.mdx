# Структуры данных: что из них реально встречается QA в API?

## Суть

Базовые структуры: массив, список, стек (LIFO), очередь (FIFO), множество, словарь/hash map, дерево, граф. В manual QA чаще всего видны array, map и queue semantics (Kafka, job queue), реже явные tree/graph в payload. Понимание помогает проектировать edge cases и нагрузочные сценарии.

## Что ответить на собесе

- Array/list: ordered collection в JSON responses.
- Stack: undo/redo, browser history; LIFO в UI state.
- Queue: async jobs, message broker; проверяю ordering и DLQ.
- Hash map: JSON object, headers, query params; O(1) lookup.
- Честно: на собесе достаточно связать структуру с тест-сценарием, не доказывать big-O на доске.

## Как это работает

| Структура | Принцип | QA пример |
|-----------|---------|-----------|
| Stack | LIFO | Back button после 3 navigation |
| Queue | FIFO | Order processing jobs 1,2,3 |
| Set | unique | Tag list без дублей |
| Map | key→value | userId → profile cache |
| Tree | hierarchy | Category tree в menu |

Gherkin:

```gherkin
Feature: Queue semantics в checkout

  Scenario: Jobs обрабатываются FIFO
    Given три заказа A, B, C созданы по порядку
    When worker один поток
    Then статус completed сначала у A, затем B, затем C

  Scenario: Stack undo отменяет последнее действие
    Given пользователь добавил item X, затем Y
    When нажата Undo
    Then корзина содержит только X
```

Graph встречается в permission inheritance и social graph «friends of friends»; тестирую cycle detection и depth limit.

## Подводные камни

- **Queue без idempotency.** Retry создаёт duplicate orders.
- **Stack overflow в UI wizard.** 50 steps back ломает state.
- **Set implemented as list.** Duplicate permissions в token.
- **Tree reorder race.** Drag-drop concurrent edit.
- **Hash collision rarely tested.** Only synthetic load on hash map cache.
