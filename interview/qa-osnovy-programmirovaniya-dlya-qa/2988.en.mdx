# Objects inside a JSON array: how does QA validate them?

## In short

A JSON array `[...]` may hold primitives, nested arrays, and objects `{...}`. Typical API pattern: `{ "items": [ { "id": 1 }, { "id": 2 } ] }`. QA checks each element schema, order, empty array, and boundary types.

## How to answer in an interview

- Yes, array of objects is standard for list endpoints.
- I validate JSON Schema: `items` type object, required fields per element.
- I check `Array.isArray`, length, unique ids inside collection.
- Empty array `[]` vs missing key is different UI behavior.
- Honestly: heterogeneous array `[1, "x", {}]` is rare in public API; if present, I record the contract.

## How it works

```json
[
  { "sku": "A-01", "qty": 3 },
  { "sku": "B-02", "qty": 1 }
]
```

Gherkin:

```gherkin
Feature: Array of objects in response

  Scenario: Each element matches schema
    When GET /api/products
    Then response.items is array
    And each element has keys sku, qty, price
    And each element.sku is non-empty string

  Scenario: Find object by field in array
    Given items contains object with name "Alice"
    When QA filters items where name equals "Alice"
    Then exactly one element is found

  Scenario: Empty array not null
    When GET /api/products?category=empty
    Then items equals []
    And items is not null
```

In Postman Tests: `pm.expect(json.items.every(o => o.id)).to.be.ok)`.

## Pitfalls

- **items as object instead of array.** `{ "items": { "0": {...} } }` breaks clients.
- **Null element inside array.** `[ null, {...} ]` rarely in schema.
- **Duplicate ids in list.** Pagination total mismatch.
- **Huge array without paging.** Performance and timeout not tested.
- **Strict order assumption.** Backend sort changed, UI regression.
