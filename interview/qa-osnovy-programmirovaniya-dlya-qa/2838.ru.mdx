# Четыре столпа ООП: зачем QA понимает инкапсуляцию и полиморфизм?

## Суть

ООП строит систему из объектов с данными и поведением. Четыре базовых принципа: инкапсуляция (скрытие состояния за методами), наследование (общая база и расширение), полиморфизм (один интерфейс, разные реализации), абстракция (работа с сущностью без лишних деталей). QA использует это при чтении кода автотестов, Page Object и при оценке testability продукта.

## Что ответить на собесе

- Инкапсуляция: поля private, наружу только осмысленные методы; баг «обошли валидацию прямой записью в поле» ловится ревью модели.
- Наследование: базовый `PaymentMethod` и наследники `Card`, `Wallet`; общие проверки в базовом классе тестов, специфика в дочерних.
- Полиморфизм: вызов `process(payment)` не знает конкретный тип; в тестах подставляю mock/stub каждой реализации.
- Честно: глубокие иерархии наследования усложняют регресс; на собесе ценю композицию над глубоким extends, но принципы называю все четыре.

## Как это работает

Page Object как инкапсуляция UI:

```gherkin
Feature: Инкапсуляция в Page Object

  Scenario: Тест не трогает селекторы напрямую
    Given страница LoginPage скрывает локатор "#pwd-field"
    When тест вызывает loginPage.signIn("qa", "secret")
    Then отправляется POST /auth без обращения к DOM из сценария
```

Полиморфизм в тестовых двойниках:

```gherkin
Feature: Полиморфная отправка уведомлений

  Scenario Outline: Канал уведомлений подменяется без смены сценария
    Given сервис OrderNotifier использует канал <channel>
    When создаётся заказ с id "ord-51"
    Then клиент получает подтверждение через <channel>

    Examples:
      | channel   |
      | EmailSink |
      | SmsSink   |
      | PushSink  |
```

Абстракция: тест `ShapeArea` проверяет контракт `area()`, не формулы каждой фигуры. Наследование: `AdminUser extends User` добавляет `grantRole()`, smoke базового логина переиспользуется.

## Подводные камни

- **Путаница наследования с «is-a» everywhere.** Дублирование логики в дочерних классах; регрессия в родителе ломает половину suite неочевидно.
- **Ломать инкапсуляцию в тестах.** `@VisibleForTesting` поля в прод-коде; тест зелёный, контракт наружу утёк.
- **Полиморфизм без контракта.** Разные реализации по-разному бросают исключения; один assert на всех не проходит.
- **Тестирование только конкретного класса.** Пропуск alternate implementation из DI контейнера.
- **ООП термины без связи с QA.** На собесе связываю с Page Object, fixtures, mock objects.
