# Список, множество и словарь: когда что использовать в тестах?

## Суть

Список (массив) хранит упорядоченные элементы с доступом по индексу. Множество хранит уникальные элементы без порядка. Словарь (map) связывает уникальные ключи со значениями. QA встречает их в JSON-ответах API, фикстурах и проверках коллекций.

## Что ответить на собесе

- Список: порядок важен (history events, queue items); индекс с нуля.
- Множество: нужна уникальность (tag ids, permission names) и set-операции.
- Словарь: lookup по ключу (userId → profile, header → value).
- В assertions сравниваю тип структуры, не только строковое представление.
- Честно: JSON object technically unordered; для order-sensitive UI проверяю array, не map.

## Как это работает

```gherkin
Feature: Структуры данных в API-ответах

  Scenario: Список сохраняет порядок элементов
    Given GET /api/feed возвращает JSON array
    When QA читает items[0].id и items[1].id
    Then порядок совпадает с сортировкой publishedAt desc

  Scenario: Множество tag ids без дублей
    Given POST /api/posts с tags ["qa", "qa", "api"]
    Then в ответе tags содержит ровно 2 уникальных значения

  Scenario: Словарь маппит id на объект
    Given GET /api/users/map возвращает object keyed by userId
    When ключ "u-42" присутствует
    Then значение содержит поле email
    And ключ "u-99" отсутствует или null по контракту
```

Списки поддерживают append/remove по индексу; поиск по значению O(n). Множества отсекают дубликаты при insert; `in` и union/intersection быстрее, чем ручной dedupe в list. Словари дают O(1) lookup по ключу; ключи immutable (string, number).

В Postman/JavaScript: `Array.isArray(data)`, `new Set(arr).size === arr.length`, `Object.keys(obj).includes('id')`.

## Подводные камни

- **Сравнение list vs set по JSON.stringify.** `[1,2]` и `{1,2}` в JS set выглядят иначе.
- **Sparse array.** `[,,,1]` length 4, QA думает три элемента.
- **Dict key type coercion.** `"42"` и `42` разные ключи в некоторых языках.
- **Mutability shared reference.** Изменил list в одном step, сломал другой.
- **Ordered set illusion в JSON.** Object keys не гарантируют порядок в старых парсерах.
