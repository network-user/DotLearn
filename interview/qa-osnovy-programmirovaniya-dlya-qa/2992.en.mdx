# if/else: how does QA cover branches in acceptance tests?

## In short

An if/else condition picks an execution path from a boolean expression. For QA each meaningful branch is a separate test case: true path, false path, else-if chain, guard clauses. Decision coverage beats counting happy-path scenarios.

## How to answer in an interview

- At least two cases per condition: true boundary and false boundary.
- Decision table for combo conditions (role AND balance AND feature flag).
- In automation: assert expected branch via response code/message.
- Ternary and early return equal if/else; I find all exit points.
- Honestly: 100% branch coverage is not manual QA goal; I prioritize risk and usage frequency.

## How it works

```gherkin
Feature: Discount by order total

  Scenario: if total >= 1000 premium discount applies
    Given cart total 1200
    When checkout
    Then discountPercent equals 15

  Scenario: else branch without premium
    Given cart total 800
    When checkout
    Then discountPercent equals 0

  Scenario: else-if medium tier
    Given cart total 500
    When checkout
    Then discountPercent equals 5
```

In code review I look for `if (user.isVip())` without negative test for non-Vip. Nested if raises cyclomatic complexity; matrix test reduces missed branches.

## Pitfalls

- **Missing else.** Null/undefined hits unhandled path → 500.
- **Truthy coercion.** `"0"` and `0` differ in JS if.
- **Duplicate condition branches.** Dead code hides bug.
- **Feature flag only in if.** Flag off → else untested forever.
- **UI hides else branch.** API still reachable.
