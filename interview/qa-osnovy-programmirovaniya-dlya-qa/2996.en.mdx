# OOP classes: what does QA verify in an app's object model?

## In short

A class is an object blueprint: fields (state) and methods (behavior). OOP provides encapsulation, inheritance, polymorphism. QA does not write classes but understands them for tests: different subtypes via common interface, override behavior, constructor invariants.

## How to answer in an interview

- Class vs instance: one `User` template, many `userAlice`, `userBob`.
- Inheritance: `Admin extends User` adds permissions; I test method overrides.
- Polymorphism: payment `Card` vs `Wallet` share interface `pay()`.
- Encapsulation: private fields change only through public API.
- Honestly: over-mocked unit tests green, integration between classes fails; need end-to-end cases.

## How it works

```gherkin
Feature: Polymorphic notification sender

  Scenario: EmailNotification sends SMTP
    Given notifier type EmailNotification
    When send alert "Order shipped"
    Then SMTP mock received message

  Scenario: SmsNotification sends SMS gateway
    Given notifier type SmsNotification
    When send alert "Order shipped"
    Then SMS provider invoked

  Scenario: Constructor invariant
    Given new Order without items
    Then creation rejected or total equals 0 per domain rule
```

Inheritance: base `Animal.speak()` vs `Dog.speak()` override. Composition over inheritance reduces fragile base class; QA focuses on class public contract.

## Pitfalls

- **Liskov violation.** Subclass breaks base expectations in UI.
- **God class.** One Order does payment, tax, shipping; untestable matrix.
- **Static singleton state.** Test order depends on execution order.
- **Reflection bypass private.** Security assumption false.
- **DTO vs Entity confusion.** API field does not map to persisted model.
