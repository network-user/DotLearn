# Наследование в ООП: какие регрессии ищет QA?

## Суть

Наследование позволяет child class получать fields и methods parent class и переопределять их. Это сокращает дублирование, но связывает иерархии: изменение base class влияет на всех наследников. QA проверяет override behavior, contract base vs derived и regression при refactor parent.

## Что ответить на собесе

- `class Admin extends User` наследует login, добавляет `manageUsers()`.
- Override: `Dog.speak()` вместо `Animal.speak()`; тестирую polymorphic call.
- Base test class с `setUp()` общим для suite наследников в automation.
- Честно: deep inheritance tree хуже composition; баг часто в grandparent change.

## Как это работает

```gherkin
Feature: Наследование pricing rules

  Scenario: Base customer standard price
    Given order type StandardCustomer
    When calculateTotal 1000
    Then discount 0

  Scenario: VIP override parent discount
    Given order type VipCustomer extends StandardCustomer
    When calculateTotal 1000
    Then discount 10 percent

  Scenario: Parent change breaks child
    Given parent ShippingCalculator добавил fuel surcharge
    When ExpressShipping наследует без override
    Then total включает surcharge per new base behavior
```

При refactor parent QA гоняет smoke на всех known subtypes. Interface inheritance (`implements Payable`) vs class inheritance разные риски.

## Подводные камни

- **Fragile base class.** Protected field rename ломает child.
- **Override forgot super().** Child skip validation parent.
- **Diamond problem multiple inheritance.** Ambiguous method in mixins.
- **Test only leaf class.** Middle abstract class untested.
- **Breaking LSP.** Child throws where parent promised not to.
