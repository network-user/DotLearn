# Variables and data types: what does QA check in API payloads?

## In short

A variable holds a named value; a type limits allowed operations and serialization. In API QA I validate int/float/string/bool/array/object/null, coercion rules, and overflow. Wrong type often yields 400 or silent truncation.

## How to answer in an interview

- int vs float: `"price": 10` vs `10.5`; schema type number/integer.
- string vs number id: `"42"` ≠ `42` in strict JSON schema.
- bool only true/false, not `"true"` string (if contract is strict).
- null vs omitted field differs in PATCH behavior.
- Honestly: JavaScript client coerces softly; mobile strict; I test both consumers when known.

## How it works

```gherkin
Feature: Type validation on registration

  Scenario: age int in valid range
    When POST /api/users body {"age": 25}
    Then status is 201

  Scenario: age string rejected
    When POST /api/users body {"age": "25"}
    Then status is 400
    And error field age

  Scenario: float where integer expected
    When POST /api/items body {"qty": 1.5}
    Then status is 400 or rounding per spec

  Scenario: boolean flag
    When POST /api/users body {"newsletter": true}
    Then persisted value is boolean not string
```

Variables in test script: `const baseUrl`, `let authToken` refresh between steps. Typed languages catch mismatch at compile; dynamic at runtime.

## Pitfalls

- **IEEE float rounding.** 0.1 + 0.2 ≠ 0.3 in assertions.
- **Big int as string in JSON.** JavaScript precision loss.
- **Empty string vs null.** Required field validation gap.
- **Implicit enum from int.** Magic numbers without docs.
- **Date as string without timezone.** Flaky midnight tests.
