# Array vs object in JSON: how does QA avoid contract confusion?

## In short

An array `[]` is an ordered list with numeric index access. An object `{}` is key-value pairs with string keys. REST list endpoints return array; single resource and envelope metadata use object. Type confusion breaks client parsing.

## How to answer in an interview

- Array: `[1,2,3]`, `items[0].id`, length, order matters.
- Object: `{ "id": 1, "name": "Ann" }`, dot/bracket access by key.
- Array of objects for collections; nested object for complex fields.
- JSON Schema `type: array` vs `type: object` in CI contract tests.
- Honestly: JS typeof array === "object"; I check Array.isArray, not typeof alone.

## How it works

```gherkin
Feature: Array vs object in contract

  Scenario: List endpoint returns array at root or items key
    When GET /api/users
    Then body.items is array
    And typeof body.items is not plain object map

  Scenario: Single resource is object
    When GET /api/users/u-1
    Then body is object with id field
    And body is not array

  Scenario: Wrong type rejected on input
    When POST /api/bulk body {"ids": {"a": 1}}
    Then status is 400 because ids must be array
```

Array of homogeneous ids `[1,2,3]` vs object map `{ "1": true, "2": false }` for flags. OpenAPI `oneOf` documents union if allowed.

## Pitfalls

- **Object with numeric keys mimics array.** `{ "0": "a", "1": "b" }` without length.
- **Empty object instead of empty array.** UI `.map` crash.
- **Array where keyed lookup expected.** O(n) scan hides perf bug.
- **Swagger says array, backend returns object on error.** Parser throws.
- **Tuple vs array not documented.** Fixed length 2 array treated as list.
