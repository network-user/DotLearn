# ООП глазами тестировщика: зачем QA понимать объекты и классы?

## Суть

Объектно-ориентированное программирование строит программу из объектов: экземпляров классов с полями и методами. Четыре опоры: инкапсуляция, наследование, полиморфизм и абстракция. QA не обязан писать фреймворки, но понимание ООП помогает читать код, проектировать Page Object и видеть, где баг в контракте класса, а где в интеграции.

## Что ответить на собесе

- ООП для меня это способ организовать домен: сущность «Заказ» инкапсулирует статус и правила смены статуса, а не размазанные if по UI.
- Инкапсуляция объясняет, почему тест не должен лезть во внутренние поля: проверяю публичное поведение, как пользователь.
- Наследование и полиморфизм предупреждают о регрессии в иерархиях: фикс в базовом классе может сломать все наследники.
- Честно: знание ООП не заменяет доменное тест-дизайн; оно ускоряет диалог с разработкой и автоматизацию.

## Как это работает

**Инкапсуляция** прячет детали реализации за методами. Класс `BankAccount` не отдаёт баланс напрямую, а меняет его через `deposit` и `withdraw` с валидацией. Тестировщик проверяет сценарии «нельзя уйти в минус», не зная, как хранится число в памяти.

**Наследование** переиспользует общий код: `SavingsAccount extends BankAccount` добавляет процент. Регрессия: изменили `withdraw` в родителе, сломали сберегательный счёт.

**Полиморфизм** позволяет вызывать один интерфейс для разных реализаций: `Notifier.send()` для Email и SMS. В тестах важно покрыть каждую реализацию, а не только базовый класс.

**Абстракция** выделяет существенное: интерфейс `PaymentGateway` без привязки к Stripe или PayPal. Моки в автотестах часто реализуют тот же контракт.

Page Object в автотестах это прямое применение ООП для QA:

```gherkin
Feature: Логин через Page Object

  Scenario: Успешный вход скрывает детали селекторов
    Given открыта страница "/login"
    When пользователь вводит валидные credentials через LoginPage
    Then DashboardPage отображает приветствие
    And внутренние CSS-селекторы формы не фигурируют в шагах сценария
```

Соответствующий каркас (идея, не копия прод-кода):

```gherkin
# Псевдо-структура Page Object (описание для Gherkin-шагов)
# LoginPage.enterCredentials(email, password) -> submit()
# DashboardPage.isVisible() -> assert greeting
```

На ревью багрепорта смотрю: воспроизведение описывает объект и его состояние («корзина после applyPromo возвращает total=0»), а не случайный клик по координатам.

## Подводные камни

- **Путаница класса и объекта.** Класс это шаблон, объект конкретный экземпляр. Тест «на класс» без данных часто бессмысленен.
- **Хрупкие тесты через наследование UI.** Когда Page Object наследуют друг от друга ради экономии строк, правка базового локатора валит половину suite.
- **Полиморфизм без матрицы покрытия.** Один happy path на интерфейсе не проверяет все реализации; классический пропуск SMS-канала при зелёном email.
- **Нарушение инкапсуляции в автотестах.** Прямой доступ к private полям через reflection ломается при рефакторинге без смены поведения.
- **ООП ради ООП.** God object на 2000 строк затрудняет изоляцию дефекта; QA должен уметь указать, какой компонент ответственен за симптом.
