# Переменные и типы данных: что QA проверяет в API payload?

## Суть

Переменная хранит именованное значение, тип ограничивает допустимые операции и serialization. В API QA валидирует int/float/string/bool/array/object/null, coercion rules и overflow. Неверный тип часто даёт 400 или silent truncation.

## Что ответить на собесе

- int vs float: `"price": 10` vs `10.5`; schema type number/integer.
- string vs number id: `"42"` ≠ `42` в strict JSON schema.
- bool только true/false, не `"true"` строкой (если контракт strict).
- null vs omitted field разное поведение PATCH.
- Честно: JavaScript клиент мягко coerces; mobile strict; тестирую оба consumer если знаю.

## Как это работает

```gherkin
Feature: Type validation на registration

  Scenario: age int в допустимом диапазоне
    When POST /api/users body {"age": 25}
    Then статус 201

  Scenario: age string rejected
    When POST /api/users body {"age": "25"}
    Then статус 400
    And error field age

  Scenario: float там где ожидается integer
    When POST /api/items body {"qty": 1.5}
    Then статус 400 или округление по spec

  Scenario: boolean flag
    When POST /api/users body {"newsletter": true}
    Then persisted value boolean not string
```

Переменные в test script: `const baseUrl`, `let authToken` refresh между steps. Typed languages ловят mismatch на compile; dynamic на runtime.

## Подводные камни

- **IEEE float rounding.** 0.1 + 0.2 ≠ 0.3 в assertions.
- **Big int as string in JSON.** JavaScript precision loss.
- **Empty string vs null.** Required field validation gap.
- **Implicit enum from int.** Magic numbers без docs.
- **Date as string без timezone.** Flaky midnight tests.
