# OOP object: how does QA model state and behavior in tests?

## In short

An object is a class instance with its own state (attributes) and behavior (methods). Two objects of the same class are independent: changing `cartA` does not change `cartB`. QA checks lifecycle: create, mutate, persist, reload identity.

## How to answer in an interview

- Object = data + operations bound together; identity by id/reference.
- Encapsulation: change state only through public methods, not raw fields.
- In API an object often maps to JSON resource `{ "id", "status", ... }`.
- Fixture builds object graph: User + Address + Orders.
- Honestly: anemic domain (only getters/setters) I test as DTO mapping.

## How it works

```gherkin
Feature: Independent cart objects

  Scenario: Two users separate carts
    Given userA cart object empty
    And userB cart object empty
    When userA addItem SKU-1
    Then userA cart total 1
    And userB cart total 0

  Scenario: Object identity after save reload
    Given order object id ORD-7 status draft
    When save and reload from API GET /orders/ORD-7
    Then status draft
    And id ORD-7 unchanged

  Scenario: Method changes internal state
    When account.deposit 100
    Then account.balance 100 without exposing private field
```

Object graph in UI: component instance holds props/state; QA via DevTools React does not confuse two modal instances.

## Pitfalls

- **Shared mutable singleton.** "Two objects" alias same instance.
- **Identity by value.** Reload new id, UI thinks same object.
- **Shallow copy.** Nested object mutated via clone.
- **Serialization loses behavior.** JSON roundtrip only data.
- **Equals without hash contract.** Set dedupe wrong in backend cache.
