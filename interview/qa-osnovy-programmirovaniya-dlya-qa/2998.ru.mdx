# Объект в ООП: как QA моделирует state и behavior в тестах?

## Суть

Объект это экземпляр класса с собственным state (attributes) и behavior (methods). Два объекта одного класса независимы: изменение `cartA` не меняет `cartB`. QA проверяет lifecycle: create, mutate, persist, reload identity.

## Что ответить на собесе

- Object = data + operations bound together; identity по id/reference.
- Encapsulation: меняю state только через public methods, не raw fields.
- В API object часто map на JSON resource `{ "id", "status", ... }`.
- Fixture создаёт object graph: User + Address + Orders.
- Честно: anemic domain (only getters/setters) тестирую как DTO mapping.

## Как это работает

```gherkin
Feature: Независимые объекты корзины

  Scenario: Два пользователя разные carts
    Given userA cart object empty
    And userB cart object empty
    When userA addItem SKU-1
    Then userA cart total 1
    And userB cart total 0

  Scenario: Object identity после save reload
    Given order object id ORD-7 status draft
    When save and reload from API GET /orders/ORD-7
    Then status draft
    And id ORD-7 unchanged

  Scenario: Method changes internal state
    When account.deposit 100
    Then account.balance 100 without exposing private field
```

Object graph в UI: Component instance holds props/state; QA через DevTools React не путает два modal instances.

## Подводные камни

- **Shared mutable singleton.** «Два объекта» alias same instance.
- **Identity by value.** Reload new id, UI думает тот же object.
- **Shallow copy.** Nested object mutated через clone.
- **Serialization loses behavior.** JSON roundtrip only data.
- **Equals without hash contract.** Set dedupe wrong in backend cache.
