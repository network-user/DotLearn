# Как QA убеждается, что сообщение в очереди не потерялось?

## Суть

Потеря сообщения это разрыв между publish ack и подтверждённой обработкой consumer. QA строит сквозной тест: уникальный message id от producer до финальной записи в БД или внешнем эффекте. Используются durable queue, publisher confirms, consumer manual ack, мониторинг unacked и DLQ, сверка счётчиков publish vs processed.

## Что ответить на собесе

- После publish проверяю broker confirm или Kafka offset ack.
- До ack consumer сверяю side effect: заказ, webhook, email log.
- При сбое брокера durable + persistent message переживают рестарт.
- Сравниваю метрики: messages published minus messages processed ≈ depth.
- Честно: at-most-once конфигурация сознательно допускает потерю ради скорости; это надо знать из контракта.

## Как это работает

```gherkin
Feature: Message loss prevention check

  Scenario: End to end trace id
    Given producer sets header trace msg-901
    When event flows through broker to consumer
    Then billing table row contains trace msg-901
    And Kibana shows single processing span

  Scenario: Broker restart survival
    Given queue durable delivery_mode persistent
    When RabbitMQ service restarted during backlog
    Then after up consumer processes all without gap in sequence

  Scenario: Publish without consumer
    Given no active consumer 10 minutes
    When depth metric equals publish count
    Then no silent drop start consumer and verify drain
```

RabbitMQ: `delivery_mode=2`, `publisher confirms`. Kafka: `acks=all`, min ISR. QA чеклист в release: zero unexplained gap in reconciliation job.

## Подводные камни

- **Ack до commit транзакции.** Сообщение «обработано», данных нет.
- **Non-durable queue на тесте.** Ложная потеря при деплое.
- **Только UI проверка.** Backend event потерян незаметно.
- **Игнор prefetch unacked.** Сообщения висят, кажутся lost.
- **Нет reconcile batch job.** Дыра видна только через сутки.
