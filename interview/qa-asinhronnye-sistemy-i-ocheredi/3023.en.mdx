# Kafka vs RabbitMQ: what is the difference for a QA engineer?

## In short

Kafka is a distributed commit log with long retention and replay; strong for stream analytics and throughput. RabbitMQ is a classic AMQP broker with flexible exchange routing and task-queue semantics. QA shifts expectations: in Kafka many consumer groups can read the same event; in Rabbit after ack the message leaves the queue.

## How to answer in an interview

- Kafka: topic/partition, offset, consumer group, retention, key order within a partition.
- RabbitMQ: exchange, binding, routing key, queue, manual ack, DLX.
- Kafka for event streaming and audit trail; Rabbit for job queues and RPC-over-mq patterns.
- I test lag and replay in Kafka; in Rabbit depth, redelivery, prefetch.
- Honestly: stack choice is not “which is better” but which delivery contract the product needs.

## How it works

```gherkin
Feature: Broker comparison for QA

  Scenario: Kafka multiple consumers same event
    Given topic user-clicks retention 7 days
    When AnalyticsGroup and FraudGroup both consume
    Then each group receives full stream independently

  Scenario: Rabbit work queue one consumer
    Given queue render-jobs single active consumer
    When job acked after processing
    Then message removed from queue not visible to others

  Scenario: Routing complexity
    Given Rabbit topic exchange orders.region.*
    When message key orders.region.eu
    Then only eu-fulfillment queue receives
```

On broker migration QA rewrites checklists: offset commit vs manual ack, idempotency on replay.

## Pitfalls

- **Same test cases for both.** Delivery semantics differ.
- **Kafka as delete-on-read queue.** Expect messages to vanish after read.
- **Rabbit without durable.** Loss on restart looks like an app bug.
- **Prefetch and fair dispatch ignored.** Uneven consumer load.
- **Kafka replay on prod without sandbox.** Duplicates in downstream.
