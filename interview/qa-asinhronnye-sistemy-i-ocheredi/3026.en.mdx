# Why do systems use message queues from a QA perspective?

## In short

A queue buffers work between fast request intake and slow processing. It enables async behavior, load balancing across workers, resilience when consumers fail, and horizontal scale. QA verifies that order spikes still get fast API responses while background jobs complete without loss within SLA.

## How to answer in an interview

- The user gets 202 Accepted; email sends from a worker later.
- Multiple consumers share one queue: round-robin or competing consumers.
- When a worker stops, messages remain with a durable queue.
- Metrics: depth, age of oldest message, processing rate.
- Honestly: without prefetch limits one heavy job blocks the whole queue.

## How it works

```gherkin
Feature: Queue purpose validation

  Scenario: Fast API response under load
    Given 500 checkout requests per minute
    When API enqueues PaymentJob only
    Then HTTP response under 300 ms
    And all jobs processed within 10 minutes

  Scenario: Worker scale out
    Given queue depth 2000
    When add second consumer instance
    Then drain rate doubles depth decreases

  Scenario: Consumer crash recovery
    Given worker dies mid job without ack
    When worker restarts
    Then same job redelivered exactly once business effect if idempotent
```

Order flow: `enqueue(orderId)` → workers `process_order` → DB status. QA polls until `COMPLETED`.

## Pitfalls

- **No max queue length.** Broker OOM on burst.
- **Single worker test for prod cluster.** SLA mismatch.
- **Priority queue ignored.** VIP orders wait behind bulk import.
- **Sync fallback untested.** Queue bypass breaks under load.
- **Dead letter not checked.** Stuck jobs invisible to users.
