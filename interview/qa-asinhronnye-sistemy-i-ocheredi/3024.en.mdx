# Why should QA understand what Apache Kafka is used for in a product?

## In short

Kafka ingests event streams from many sources, stores them as an ordered log, and feeds multiple subscribers in real time. In products it acts as a domain event bus, microservice integration, DB CDC, analytics, and monitoring. QA verifies events arrive, survive broker failures, and process within acceptable lag.

## How to answer in an interview

- Producers write to topics; brokers replicate partitions across the cluster.
- Multiple consumer groups read one stream for billing, CRM, and DWH.
- Retention enables replay after a bug or new consumer deploy.
- I test e-commerce: click, cart, payment as an event chain in a topic.
- Honestly: Kafka is overkill for a simple task queue of ten jobs per hour.

## How it works

```gherkin
Feature: Kafka product usage

  Scenario: Real-time analytics pipeline
    Given web shop publishes UserViewedProduct to topic catalog.events
    When recommendation service consumes within 5 seconds
    Then related products block updates on product page

  Scenario: Broker failover
    Given replication factor 3 one broker down
    When producer acks all replicas
    Then QA sees no message loss in consumer output

  Scenario: Replay after bug fix
    Given consumer bug skipped 2 hours of events
    When reset offset to start of window
    Then downstream warehouse reconciles missing rows
```

Parts: producers (API, connectors), brokers, ZooKeeper/KRaft metadata, consumers. QA artifacts: offset lag dashboard, sample event JSON, expected side effect per topic.

## Pitfalls

- **One subscriber only.** One consumer OK, DWH empty.
- **Schema registry ignored.** Invalid Avro breaks the chain.
- **Retention too short.** Replay impossible for RCA.
- **Producer load only.** Consumer bottleneck in prod.
- **Log compaction vs full journal.** Older events unavailable.
