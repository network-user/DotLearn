# Consumer Group в Kafka: зачем QA смотрит на распределение партиций?

## Суть

Consumer Group объединяет несколько инстансов консьюмера, которые совместно читают топик: каждая партиция в группе закреплена только за одним активным консьюмером, поэтому сообщение не обрабатывается дважды внутри группы. При падении инстанса брокер перераспределяет партиции (rebalance). Для тестирования это точка контроля масштаба, отказоустойчивости и отсутствия дублей.

## Что ответить на собесе

- Group id задаёт «логического подписчика»: все инстансы с одним GROUP_ID делят партиции; другая группа читает те же сообщения независимо (broadcast между группами).
- Число консьюмеров в группе не должно превышать число партиций топика: лишние простаивают.
- При rebalance возможны пауза обработки и повторное чтение; проверяю идempotентность обработчика и commit offset.
- Честно: auto-commit удобен в демо, в проде QA смотрит на ручной commit после успешной бизнес-операции, иначе теряем или дублируем эффект.

## Как это работает

Топик `orders-created` имеет 4 партиции. Группа `fulfillment-svc` запускает 2 процесса:

| Консьюмер | Партиции |
|-----------|----------|
| worker-a | 0, 1 |
| worker-b | 2, 3 |

Если `worker-a` падает, координатор группы назначает 0 и 1 на `worker-b` до поднятия нового инстанса. Offset хранится per partition per group: прогресс «где остановились» не общий для всего топика.

Gherkin для регрессии отказоустойчивости:

```gherkin
Feature: Consumer group и rebalance

  Scenario: Два консьюмера делят четыре партиции без дублей в группе
    Given топик "orders-created" с 4 партициями
    And два инстанса консьюмера с GROUP_ID "fulfillment-svc"
    When продюсер публикует 100 сообщений с уникальными ключами
    Then каждое сообщение обработано ровно одним инстансом группы
    And суммарный lag группы стремится к 0

  Scenario: Падение инстанса не останавливает обработку навсегда
    Given worker-a обрабатывает партиции 0 и 1
    When worker-a принудительно останавливается
    Then в течение N секунд партиции 0 и 1 назначены другому живому консьюмеру
    And необработанные сообщения из 0 и 1 eventually дочитываются
```

Ключевые настройки в тестовом стенде (псевдоконфиг):

```
bootstrap.servers=broker:9092
group.id=fulfillment-svc
enable.auto.commit=false
auto.offset.reset=earliest
```

После успешной обработки консьюмер вызывает commit sync; при ошибке сообщение остаётся для retry или уходит в DLQ по политике сервиса.

## Подводные камни

- **Больше консьюмеров, чем партиций.** Часть процессов idle; нагрузка не растёт, команда думает, что масштабировалась.
- **Rebalance во время деплоя.** Всплеск дублей или пропусков, если handler не идемпотентен.
- **Разные group id на одном сервисе.** Два деплоя с опечаткой в GROUP_ID удваивают нагрузку на downstream.
- **Sticky assignor vs range.** После rebalance порядок владения партициями меняется; flaky тесты на «какой pod обработал».
- **Измерение lag только по сумме.** Одна «горячая» партиция тормозит срез пользователей с тем же ключом; нужен lag per partition.
