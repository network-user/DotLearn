# What is a message queue in integration testing?

## In short

A message queue is a FIFO (or priority) buffer between producer and consumer in a distributed system. It provides temporal decoupling, peak smoothing, and delivery guarantees with durable settings. QA treats the queue as a black box with depth, latency, and DLQ metrics, not just “another API”.

## How to answer in an interview

- The sender enqueues a job; the receiver pulls when ready.
- FIFO by default; priority queues for urgent work.
- Rabbit queues, Kafka partition logs, and SQS standard queues differ; tests differ too.
- I verify: published, visible in broker UI, processed exactly once by effect.
- Honestly: a queue does not speed processing; it smooths and shifts work in time.

## How it works

```gherkin
Feature: Message queue integration test

  Scenario: Order pipeline FIFO
    Given publish Order-100 then Order-101 to queue fulfillment
    When single consumer active
    Then process order 100 before 101

  Scenario: Backlog during outage
    Given consumer stopped
    When 50 orders enqueued
    Then depth 50 no message dropped durable true
    And after consumer start depth returns 0

  Scenario: Cross team contract
    Given JSON schema version 2 in queue payload
    When producer sends version 1 fields
    Then consumer rejects to DLQ QA blocks release
```

```gherkin
Feature: Publish to order queue

  Scenario: Fire order job
    Given queue order_pipeline declared durable
    When basic_publish body Order 5567
    Then management UI shows ready 1
```

## Pitfalls

- **Queue without depth monitoring.** Overflow hidden until user timeouts.
- **Publish-only test.** Consumer never verified.
- **Non-durable on CI.** Messages vanish on container restart.
- **Queue confused with cache.** Kafka reread is not queue pop.
- **Poison message ignored.** One bad record blocks FIFO.
