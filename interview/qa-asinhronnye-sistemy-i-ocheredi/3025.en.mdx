# Partitioning in Kafka and databases: why should QA care?

## In short

A partition is a logical data split for parallelism and manageability. In Kafka partitions drive parallel consumers and key-based order. In databases table partitions speed date-range queries and simplify archival. QA checks that order and integrity rules hold after splits and that test data lands in the expected partition.

## How to answer in an interview

- Kafka: more partitions raise throughput but need a meaningful message key.
- DB range partition: a March query reads only partition p202403.
- Hash partition spreads load without a range filter.
- On partition migration QA compares row counts before and after split.
- Honestly: a bad Kafka key creates a hot partition and skewed lag on one consumer.

## How it works

```gherkin
Feature: Partitioning awareness

  Scenario: Kafka hot key detection
    Given topic payments 12 partitions
    When 90 percent messages use key default
    Then partition 0 lag high others idle QA files perf defect

  Scenario: DB range prune old data
    Given sales table partitioned by year
    When archive job drops partition p2019
    Then queries for 2019 fail fast not full table scan

  Scenario: Order per customer in Kafka
    Given key customerId in publish
    When events PAY and REFUND for customer 88
    Then both land same partition processed in offset order
```

SQL table partition example (renamed fields):

```sql
CREATE TABLE audit_events (
  evt_id BIGINT,
  evt_day DATE,
  payload JSON
) PARTITION BY RANGE (YEAR(evt_day)) (
  PARTITION y2023 VALUES LESS THAN (2024),
  PARTITION y2024 VALUES LESS THAN (2025)
);
```

## Pitfalls

- **Random partition key.** Related event order breaks.
- **Too many partitions.** Kafka coordination overhead.
- **Test on non-partitioned staging DB.** Prod perf bug hidden.
- **Confuse Kafka partition with consumer instance.** Not 1:1 without rebalance.
- **Forget partition pruning in EXPLAIN.** False sense of slow queries.
