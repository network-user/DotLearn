# Producer vs consumer: what is the difference for QA?

## In short

A producer creates and sends an event to a broker; a consumer subscribes, reads, and runs business logic. These are different roles, often different services and deploys. A bug may sit on one side only: publish with no subscriber or consumer crash on a valid message. QA tests the contract between them and independent failure of each side.

## How to answer in an interview

- Producer owns correct payload, key, headers, idempotency key.
- Consumer owns ack/nack, retry, side effects, DLQ.
- One topic, several Kafka consumer groups: each group is independent.
- I break the consumer on staging: queue grows, producer still accepts API traffic.
- Honestly: fixing only the consumer may not need republish if retention allows replay.

## How it works

```gherkin
Feature: Producer vs consumer responsibility

  Scenario: Producer publishes valid event
    Given billing API emits InvoiceIssued to topic invoices
    When schema validation passes on publish
    Then broker acks producer
    And consumer not required for API 201 response

  Scenario: Consumer failure isolation
    Given consumer deployment broken
    When producer keeps sending
    Then lag grows no producer 500 unless configured fail-fast

  Scenario: Invalid payload caught at consumer
    Given producer bug sends negative amount
    When consumer validates
    Then nack to DLQ producer still got broker ack QA files two-team bug
```

```gherkin
Feature: Kafka roles smoke

  Scenario: Round trip
    Given KafkaProducer to topic smoke.ping
    When send payload ping-44
    And KafkaConsumer group qa-smoke reads
    Then stdout shows received ping-44
```

## Pitfalls

- **Producer REST only.** Consumer fails silently.
- **Same broker login for both roles.** Unclear who did not ack.
- **Consumer without contract test.** Producer field change breaks prod.
- **Expect sync reply from producer.** Async semantics violated in expectations.
- **Replay forgotten after consumer fix.** Older messages stay unprocessed.
