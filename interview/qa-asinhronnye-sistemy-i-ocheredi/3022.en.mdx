# What is a message broker and why does QA need to understand it?

## In short

A broker sits between producers and consumers: it accepts, stores, routes, and acknowledges messages. It decouples services in time and buffers spikes or consumer outages. QA tests system behavior through the broker, not only APIs: delivery, retry, DLQ, persistence after restart.

## How to answer in an interview

- The sender does not know the receiver address; routing goes through exchange/topic and bindings.
- When a consumer is down, messages stay in a queue or partition.
- Examples: RabbitMQ, Kafka, SQS, NATS; choice affects test guarantees.
- I verify durable queues, ack mode, TTL, max retry.
- Honestly: a broker does not fix idempotency bugs; it only delivers.

## How it works

```gherkin
Feature: Message broker role

  Scenario: Decouple order API from email service
    Given OrderAPI publishes NewOrder to broker
    When EmailService down 15 minutes
    Then messages queue depth grows
    And after EmailService up all emails sent once each

  Scenario: Routing via exchange
    Given fanout exchange order.events
    When order 501 published
    Then billing queue and analytics queue both receive copy

  Scenario: Persistence after broker restart
    Given durable queue flag true
    When broker process restarted
    Then unpublished messages still available to consumer
```

E-commerce: order → broker → fulfillment, notifications, analytics in parallel. QA validates every subscribed queue.

## Pitfalls

- **REST-only tests between services.** Async path skipped.
- **Transient queue on staging.** Messages vanish after restart; false bug.
- **No depth monitoring.** Overflow unnoticed until prod.
- **Confuse broker with worker.** Queue grows from slow consumers, not the broker.
- **Poison message policy ignored.** Whole queue blocks.
