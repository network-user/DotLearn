# Партиции Kafka: что проверяет тестировщик в распределённом топике?

## Суть

Партиция это упорядочный журнал записей внутри топика: каждая запись имеет monotonic offset. Топик делится на партиции, чтобы распределить данные по брокерам и дать параллельное чтение. Порядок гарантирован только внутри одной партиции; ключ сообщения определяет, в какую партицию попадёт запись.

## Что ответить на собесе

- Число партиций задаёт верхнюю грань параллелизма consumer group: больше партиций, больше консьюмеров может работать одновременно.
- С ключом все события одной сущности (userId, orderId) идут в одну партицию и сохраняют порядок; без ключа распределение round-robin и порядок между сообщениями одного пользователя не гарантирован.
- Реплики партиций (leader/follower) влияют на отказоустойчивость: при падении брокера leader переизбирается, QA смотрит на недоступность и потерю in-sync реплик.
- Честно: увеличить партиции задним числом можно, уменьшить без миграции нельзя; это архитектурное решение, не «подкрутка в тесте».

## Как это работает

Топик `checkout-events` с тремя партициями на кластере из двух брокеров:

```
Topic checkout-events
  partition 0 (leader broker-1, replicas 1,2)
  partition 1 (leader broker-2, replicas 2,1)
  partition 2 (leader broker-1, replicas 1,2)
```

Продюсер с ключом `order-9001` всегда попадает в одну и ту же партицию (hash по ключу). Консьюмеры группы `billing` читают offset независимо по каждой партиции.

Gherkin для проверки порядка и маршрутизации:

```gherkin
Feature: Партиции и ключ сообщения

  Scenario: События с одним ключом сохраняют порядок
    Given топик "checkout-events" с 3 партициями
    When продюсер отправляет три записи с key "order-9001" values "created", "paid", "shipped"
    Then консьюмер читает values в порядке "created", "paid", "shipped"
    And все три записи имеют один partition id

  Scenario: Записи без ключа распределяются по партициям
    Given топик "checkout-events" с 3 партициями
    When продюсер отправляет 30 записей без key
    Then сообщения распределены минимум по двум партициям
    And суммарный offset lag по группе не растёт бесконечно
```

Для нагрузочного теста фиксирую: число партиций, replication factor, min.insync.replicas и поведение при acks=all когда ISR уменьшается.

## Подводные камни

- **Ожидание глобального порядка в топике.** Между партициями порядок случаен; баг «оплата раньше создания» возможен без ключа.
- **Слишком мало партиций на проде.** Lag растёт, горизонтальное масштабирование консьюмеров бесполезно.
- **Hot partition.** Один горячий ключ забивает одну партицию; средний lag по топику выглядит нормально.
- **Тест на одной партиции локально.** На кластере rebalance и leader election дают другие тайминги.
- **Retention vs тестовые данные.** Старые offset «исчезают» после retention; регрессия «прочитать с начала» требует отдельного топика или reset offsets.
