# Как устроена Kafka: producer, topic, partition, consumer для QA?

## Суть

Producer пишет записи в topic; topic делится на partition для параллелизма. Каждая запись получает offset внутри partition. Consumer group распределяет partition между инстансами: одна partition в момент времени читает один consumer в группе. QA проверяет доставку, порядок по ключу, lag и поведение при rebalance.

## Что ответить на собесе

- Producer выбирает partition по key hash или round-robin.
- Topic это лог, не очередь с удалением после read: retention по времени/размеру.
- Consumer commit offset после обработки; при сбое чтение продолжается с последнего commit.
- Метрики: consumer lag, under-replicated partitions.
- Честно: без partition key порядок между связанными событиями не гарантирован.

## Подводные камни

- **Путаю topic и consumer group.** Разные группы читают один topic независимо.
- **auto commit до обработки.** Потеря при crash между read и business logic.
- **Один partition на high load.** Bottleneck при scale producers.
- **Тест без key.** Порядок случайный, баг «не воспроизводится».
- **Игнор compaction topic.** Последнее значение ключа, не полная история.
