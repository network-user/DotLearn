# What is a Kafka topic and how does QA test an event stream?

## In short

A topic is a named Kafka record stream split into partitions with independent offsets. Many producers write to one topic; many consumer groups read independently without “taking” messages from each other. QA maps business domains to topics and checks retention, ACLs, schema, and subscriber end-to-end effects.

## How to answer in an interview

- Topic names reflect domains: `user.signups`, `inventory.adjustments`.
- Partition count sets max parallel consumers in one group.
- Retention hours/bytes defines whether old events can be replayed.
- Each group keeps its own committed offset per partition.
- Honestly: a generic `events` topic makes testing chaos without contracts.

## How it works

```gherkin
Feature: Kafka topic testing

  Scenario: Orders topic end to end
    Given producer writes to topic shop.orders
    When WarehouseGroup consumes
    Then stock table decremented for SKU in payload

  Scenario: Multiple groups same topic
    Given AnalyticsGroup and BillingGroup subscribe shop.orders
    When one order event published
    Then both groups receive independent copy

  Scenario: Retention boundary
    Given retention 24 hours
    When event older than 24 hours
    Then replay test cannot fetch QA documents limitation
```

```gherkin
Feature: Topic smoke publish read

  Scenario: Round trip orders topic
    Given send shop.orders body New order 7781
    When consumer in group qa-smoke polls
    Then received body matches published
```

## Pitfalls

- **One partition on high volume topic.** Lag grows linearly.
- **Consumer test without partition count.** Wrong parallelism expectations.
- **Compacted topic as event log.** Change history is lost.
- **Staging ACL typo.** Publish 403 looks like app bug.
- **No schema version in topic contract.** Silent consumer skip.
