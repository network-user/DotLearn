# Как программа выходит в ядро через системный вызов?

## Суть

Системный вызов (syscall) переводит выполнение из пользовательского режима в привилегированный код ядра, когда процессу нужен доступ к файлам, сети, памяти или устройствам. Ядро выполняет операцию и возвращает управление с кодом результата.

## Что ответить на собесе

- Объясняю syscall как контролируемый шлюз к ресурсам ОС: приложение не трогает железо напрямую, а просит ядро сделать это за него.
- Называю типичные примеры: `read`/`write`, `open`, сокетные операции, `mmap`, управление процессами.
- Уточняю, что переход в kernel mode дороже обычного вызова функции, поэтому в горячих путях стараюсь минимизировать количество syscalls.
- Честно добавляю: в Go большинство I/O уже обёрнуто в runtime и стандартную библиотеку, на собесе редко пишут `syscall` руками, но модель переключения режимов важна для понимания блокировок и планировщика.

## Как это работает

Пользовательский код работает в ограниченном адресном пространстве. Когда нужна операция, которую может выполнить только ядро, процесс кладёт номер syscall и аргументы в регистры (или через таблицу на некоторых архитектурах) и выполняет специальную инструкцию. CPU переключает контекст, ядро валидирует параметры, выполняет запрос и возвращает errno или дескриптор.

В Go прямой пакет `syscall`/`syscall.Syscall` сегодня считается низкоуровневым; для файлов и сети обычно используют `os` и `net`, которые внутри всё равно приводят к системным вызовам. Пример низкоуровневого чтения через дескриптор:

```go
package main

import (
	"fmt"
	"os"
	"syscall"
)

func main() {
	handle, err := os.Open("notes.txt")
	if err != nil {
		fmt.Println("open failed:", err)
		return
	}
	defer handle.Close()

	buf := make([]byte, 64)
	n, err := syscall.Read(int(handle.Fd()), buf)
	if err != nil {
		fmt.Println("read failed:", err)
		return
	}
	fmt.Printf("bytes=%d payload=%q\n", n, buf[:n])
}
```

Здесь `handle.Fd()` отдаёт целочисленный дескриптор, а `syscall.Read` уходит в ядро. Ошибки приходят как Go-ошибки, но семантика та же, что у POSIX `read(2)`.

## Подводные камни

- **Повторное использование буфера.** После частичного чтения нужно сдвигать срез, иначе легко перезаписать уже принятые байты или прочитать меньше, чем ожидали.
- **Блокирующие syscalls.** Долгий `read` на сокете блокирует поток ОС; в Go runtime может отцепить `M` от горутины, но нагрузка на планировщик остаётся.
- **Устаревший API.** Пакет `syscall` в stdlib заморожен; для нового кода смотрят `golang.org/x/sys/unix` или высокоуровневые обёртки.
- **Безопасность дескрипторов.** Передача произвольного int как fd без проверки владения файлом опасна в библиотечном коде.
- **Ошибки errno.** Не каждая ошибка фатальна: `EINTR` после сигнала иногда требует повторить вызов, а не сразу падать.
