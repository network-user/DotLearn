# Куда возвращается поток ОС после системного вызова?

## Суть

Когда syscall завершается, поток операционной системы снова попадает в планировщик Go. Тот выбирает, какую горутину возобновить: чаще всего ту, что инициировала вызов, но при необходимости переключается на другую готовую задачу.

## Что ответить на собесе

- Объясняю цепочку: горутина уходит в ядро, поток `M` освобождается или отцепляется, после возврата из ядра runtime снова планирует работу на этом `M`.
- Подчёркиваю, что во время блокирующего syscall другие горутины на том же потоке могут выполняться, если runtime успел перепривязать `M`.
- Уточняю разницу между коротким и долгим syscall: при долгих вызовах runtime может поднять дополнительный поток, чтобы не простаивали остальные горутины.
- Честно добавляю: точные детали зависят от версии Go и ОС, на собесе достаточно модели G-M-P и идеи «поток возвращается в scheduler, а не напрямую в горутину».

## Как это работает

Горутина сама по себе не равна потоку ОС. Планировщик Go (модель G-M-P) мультиплексирует тысячи `G` на небольшое число `M`, каждый из которых привязан к системному потоку. Сетевой `http.Get`, чтение файла или `read` на сокете в итоге приводят к syscall, и пока ядро ждёт I/O, соответствующая горутина не продвигается.

На время syscall runtime может отпустить `M`: поток уходит в ядро, а планировщик перекладывает другие горутины на свободные `M` или будит новый поток через `handoff`, если все заняты блокирующими вызовами. После возврата из ядра `M` снова входит в цикл `schedule`: ищет runnable-горутину в локальной очереди `P`, глобальной очереди или у других `P` (work stealing).

Если горутина, стартовавшая syscall, уже в состоянии runnable, она часто получает CPU первой. Если она ещё ждёт данных, планировщик берёт любую готовую `G`. Так достигается перекрытие сетевых задержек без блокировки всего процесса.

```go
package main

import (
	"fmt"
	"io"
	"net/http"
	"sync"
)

func pullPage(client *http.Client, url string, wg *sync.WaitGroup) {
	defer wg.Done()
	resp, err := client.Get(url)
	if err != nil {
		fmt.Println("request failed:", url, err)
		return
	}
	defer resp.Body.Close()
	body, _ := io.ReadAll(resp.Body)
	fmt.Printf("%s bytes=%d\n", url, len(body))
}

func main() {
	var wg sync.WaitGroup
	client := &http.Client{}
	urls := []string{
		"https://example.com",
		"https://example.org",
		"https://example.net",
	}
	for _, u := range urls {
		wg.Add(1)
		go pullPage(client, u, &wg)
	}
	wg.Wait()
}
```

Здесь каждый `client.Get` уходит в сетевой syscall. Пока один запрос ждёт ответа, runtime переключает `M` на другие горутины из пула. После завершения syscall поток не «прилипает» к одной `G` навсегда, а снова проходит через общий цикл планирования.

## Подводные камни

- **Путаница поток и горутина.** Кандидаты часто говорят, что «поток возвращается в горутину»; корректнее: поток возвращается в runtime, который выбирает следующую `G`.
- **Долгие syscalls.** Синхронный дисковый I/O или DNS без netpoller может надолго занять `M` и снизить параллелизм, если `GOMAXPROCS` мал.
- **Синхронизация через Sleep.** В учебных примерах `time.Sleep` маскирует work stealing; в проде нужны `WaitGroup`, контекст или errgroup.
- **Сетевой poller.** Для сокетов Go предпочитает неблокирующие fd и epoll/kqueue; модель «освободить M» дополняется, а не заменяется netpoller.
- **Версии runtime.** Детали handoff и очередей менялись между релизами; на глубокий разбор ссылаюсь на GODEBUG=schedtrace только если интервьюер просит.
