# What kinds of profiling exist for JVM applications?

## In short

Profiling finds bottlenecks along three axes: CPU (hot methods and sampling), memory (allocations, retained heap, leaks), and I/O (disk, network, JDBC latency). In Java you use async-profiler, JFR, VisualVM, or YourKit; the choice depends on the symptom: high CPU, GC pauses, or a slow endpoint.

## How to answer in an interview

- CPU profiling: flame graph, self time vs total time, inlined frames.
- Memory: heap histogram, dominator tree, allocation profiling.
- I/O: blocked threads, socket read/write wait, JDBC pool saturation.
- I separate production-safe (JFR, low-overhead sampling) from deep dive (heap dump).
- I add honestly: profiling without baseline metrics gives false priorities; I look at p95 latency and GC logs first.

## How it works

A CPU hotspot through a JFR event (simplified):

```java
final class ReportBuilder {
    String buildMonthlySummary(List<LedgerRow> rows) {
        return rows.stream()
            .filter(row -> row.amountCents() > 0)
            .map(this::formatRow)
            .collect(Collectors.joining("\n"));
    }

    private String formatRow(LedgerRow row) {
        return row.accountId() + ":" + row.amountCents();
    }
}
```

A flame graph shows `formatRow` and `Collectors.joining` eating CPU on a large `rows` list. A memory profiler reveals intermediate `String` objects from `map` pressuring the young gen. An I/O trace shows BLOCKED threads on `socketRead0` with a slow downstream. The three profiling types answer different questions and are often combined in one incident.

## Pitfalls

- **Profiling in dev on small data.** The production hotspot does not reproduce.
- **Instrumenting profiler in production.** Bytecode injection distorts timing.
- **Ignoring JVM overhead.** You optimize Java code while the issue is GC tuning.
- **Confusing allocation with retention.** High allocation is not a leak.
- **One snapshot without load.** An idle dump is useless for a capacity issue.
