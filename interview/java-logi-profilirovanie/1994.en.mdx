# How do you choose a log level in Java?

## In short

SLF4J levels (TRACE, DEBUG, INFO, WARN, ERROR) set event priority. A threshold on the logger or appender cuts noise: production is usually INFO or WARN, DEBUG only during an investigation. The right level balances diagnostics and storage volume.

## How to answer in an interview

- TRACE: step-by-step flow, almost never in production.
- DEBUG: dev/stage diagnostics, variable values, branch decisions.
- INFO: normal business events (job start, successful payment).
- WARN: degradation, retry, deprecated path, but the service is alive.
- ERROR: operation failure, exception, needs action; honestly, not every catch deserves ERROR.

## How it works

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

final class PayoutJob {
    private static final Logger log = LoggerFactory.getLogger(PayoutJob.class);

    void run(String payoutId, int amountCents) {
        log.info("payout started payoutId={} amountCents={}", payoutId, amountCents);

        if (amountCents <= 0) {
            log.warn("skipped invalid amount payoutId={} amountCents={}", payoutId, amountCents);
            return;
        }

        log.debug("reserving ledger slot payoutId={}", payoutId);
        try {
            captureFunds(payoutId, amountCents);
            log.info("payout completed payoutId={}", payoutId);
        } catch (PaymentDeclinedException ex) {
            log.error("payout failed payoutId={} reason={}", payoutId, ex.getCode(), ex);
        }
    }

    private void captureFunds(String payoutId, int amountCents) {
        log.trace("enter captureFunds payoutId={}", payoutId);
    }
}
```

`logback.xml` can keep root at INFO while `com.acme.billing` is temporarily DEBUG. Parameterized messages (`{}`) avoid building strings when the level is disabled. ERROR with a throwable keeps the stack trace for RCA. WARN records an anomaly without killing the process.

## Pitfalls

- **ERROR on an expected failure.** 404 or validation errors as ERROR wake on-call.
- **INFO spam.** Logging every row in a batch means millions of lines per day.
- **DEBUG in production without TTL.** You forget to restore the threshold and disk/index growth explodes.
- **TRACE on a hot path.** Synchronous writes hurt latency.
- **Mixing levels and alerts.** Alerting on WARN without filters causes alert fatigue.
