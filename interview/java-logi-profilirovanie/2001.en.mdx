# What is the ELK stack?

## In short

ELK is three roles in an observability pipeline: Elasticsearch stores and indexes events, Logstash (or Beats) collects and transforms them, Kibana visualizes and searches. Together they turn scattered file logs into a centralized searchable store with dashboards and alerts.

## How to answer in an interview

- Elasticsearch: inverted index, shards/replicas, ILM, Query DSL.
- Logstash: input/filter/output pipeline; grok, mutate, date parse.
- Kibana: Discover, Lens, dashboards, Watcher alerts.
- Modern variant: Elastic Beats instead of heavy Logstash on the edge.
- I add honestly: managed OpenSearch/Grafana Loki compete; ELK wins on mature full-text search and aggregations.

## How it works

A minimal pipeline for a Java service:

```plaintext
input {
  beats {
    port => 5044
  }
}

filter {
  json {
    source => "message"
  }
  date {
    match => ["timestamp", "ISO8601"]
    target => "@timestamp"
  }
}

output {
  elasticsearch {
    hosts => ["https://es-cluster:9200"]
    index => "orders-%{+YYYY.MM.dd}"
    user => "logstash_writer"
    password => "${LOGSTASH_ES_PASSWORD}"
  }
}
```

Filebeat on the pod sends JSON stdout to the Logstash beats input. The filter parses timestamp and fields like `level` and `traceId`. Elasticsearch stores the document in the daily index `orders-2026.07.13`. Kibana builds an ERROR rate line chart and a top `logger` table. During an incident, an engineer filters by `traceId` and `level:ERROR` in Discover without SSH grep.

## Pitfalls

- **Grok on an unstable format.** The parser breaks when the log pattern changes.
- **One giant shard index.** The hot node is overloaded.
- **No ILM.** Disk fills up and the cluster goes read-only.
- **Logstash single-thread bottleneck.** You need a persistent queue or Beats direct to ES.
- **Kibana access without RBAC.** Everyone sees production PII in logs.
