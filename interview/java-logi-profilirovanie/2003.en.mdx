# Which JVM profilers do you use in practice?

## In short

For Java applications, profiling uses built-in and external tools: JFR/JMC and async-profiler for low-overhead CPU, VisualVM/MAT for heap, JProfiler/YourKit for deep analysis in dev. The choice depends on the environment: production-safe sampling vs attach in staging with allocation profiling.

## How to answer in an interview

- async-profiler: flame graphs, wall/alloc modes, low overhead.
- JFR (Java Flight Recorder): events in production, analysis in JMC.
- VisualVM/jcmd: quick local snapshot, thread/heap dump.
- MAT (Eclipse Memory Analyzer): dominator tree, leak suspects.
- I add honestly: commercial YourKit/JProfiler are handy in the IDE, but in production I more often use JFR plus Grafana Pyroscope/Parca.

## How it works

Running async-profiler in CPU mode (bash on the host with the JVM):

```bash
./profiler.sh -d 60 -f /tmp/flame.html 8421
```

60 seconds of sampling PID 8421, output flame.html. Hot frames like `java.util.regex.Pattern` or `com.acme.PricingService.quote` show up as wide bars. For memory:

```bash
jcmd 8421 VM.native_memory summary
jcmd 8421 GC.heap_dump /tmp/8421.hprof
```

Open the hprof in MAT: the dominator tree shows `ConcurrentHashMap` retaining 40% of the heap from a cache without TTL. Enable JFR with flags like `-XX:StartFlightRecording=...` and review GC, socket, and method profiling offline in JMC without a permanent agent attach.

## Pitfalls

- **Instrumenting profiler in production.** It distorts latency and throughput.
- **Profiling a cold JVM.** JIT has not warmed hot methods yet.
- **Flame graph without symbols.** Obfuscated or missing debug symbols.
- **Heap dump as the only signal.** Without allocation profiling you miss churn rate.
- **Attach to the wrong PID in k8s.** Sidecar and app PIDs get confused in one pod.
