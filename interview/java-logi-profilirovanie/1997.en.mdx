# Why use the ELK stack in microservices?

## In short

In microservices, logs are spread across dozens of pods and services. ELK (Elasticsearch, Logstash/Beats, Kibana) centralizes collection, indexes JSON events, and provides search and dashboards. That enables correlation by traceId, fast RCA, and one operational view without SSH to every instance.

## How to answer in an interview

- Logstash/Beats/Filebeat: ship logs from nodes and containers.
- Elasticsearch: full-text and field queries, retention via index lifecycle.
- Kibana: Discover, dashboards, alerts.
- I mention replacing Logstash with Fluent Bit + ES in cloud-native setups.
- I add honestly: ELK without structured logs and trace id devolves into text grep; investing in schema pays off.

## How it works

Filebeat ships JSON into Elasticsearch:

```yaml
filebeat.inputs:
  - type: container
    paths:
      - /var/lib/docker/containers/*/*.log
    processors:
      - decode_json_fields:
          fields: ["message"]
          target: "app"

output.elasticsearch:
  hosts: ["https://es.internal:9200"]
  index: "billing-%{+yyyy.MM.dd}"

setup.kibana:
  host: "https://kibana.internal:5601"
```

A Kibana query `traceId:7f3c2a1b9e004d12` shows the payment, ledger, and notification chain across services. A daily index simplifies ILM: hot on SSD, warm delete after 30d. Alerts on `level:ERROR` and rising WARN rate catch degradation before user complaints.

## Pitfalls

- **Unstructured multiline stack traces.** Parsers break event boundaries.
- **One giant index.** Slow search and expensive reindex.
- **Missing timezone normalization.** Events from regions appear out of order.
- **PII in the index.** GDPR risk and leaks with broad Kibana access.
- **Logstash as a SPOF.** Without a queue, backpressure drops events during spikes.
