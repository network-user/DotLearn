# RLS on server and client: how it applies

## In short

RLS applies on server read operations and queries in user context. The client does not bypass RLS unless it calls privileged server code. Thick and web clients alike rely on server checks.

## How to answer in an interview

- I run queries and object reads on server with `&НаСервере` without privileges.
- `УстановитьПривилегированныйРежим` only for service ops with logging.
- I do not cache client-side data the user cannot see under RLS.
- COM and external connections inherit user session or explicit service account.
- I am honest: "thin client is safer" is wrong; security is server context and rights.

## How it works

Client procedure calls server method; query runs as current infobase user. Privileged common module drops RLS for the whole call chain inside that module.

## Pitfalls

**Privileged mode on form.** Leak of all list rows.
**Value table to client.** Bypass UI hiding.
**Background job as user without RLS awareness.** Empty mailed reports.
**Service user with full rights.** Integration sees everything.
**UI-only check.** API and schedules without RLS equivalent.
