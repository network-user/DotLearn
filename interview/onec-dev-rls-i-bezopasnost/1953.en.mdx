# Order of applying RLS filters in a query

## In short

Metadata object rights apply first, then data access profile limits, then conditions from the user query text. The platform ANDs templates within one profile and merges profiles by platform rules.

## How to answer in an interview

- I do not assume `WHERE` order in source text is RLS order.
- User list filter stacks on limits, does not replace them.
- With multiple profiles I inspect final SQL in the tech log.
- Privileged mode disables RLS injection.
- I am honest: you cannot "bypass" RLS in a report query without privileges.

## How it works

Final query = source text + limit fragments per FROM table. Parameters like `&ДоступныеОрганизации` bind from the session.

## Pitfalls

**Expecting RLS at the end.** Optimizer plan differs.
**OR in limit template.** Unpredictable wider selection.
**Privileged module without need.** Data leak.
**Different limits on joined tables.** One table visible, other empty.
**Debug without tech log.** Final SQL invisible.
