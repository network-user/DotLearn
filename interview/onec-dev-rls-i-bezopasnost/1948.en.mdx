# Typical problems when using RLS

## In short

RLS (record-level security) adds extra conditions to queries and read rights. Typical issues: performance drop, empty lists for users, bypass via privileged mode, temp tables without constraint awareness.

## How to answer in an interview

- I test reports as an RLS user, not administrator.
- I split complex limits into an information register of allowed companies.
- Privileged mode only in narrow server procedures with audit.
- Dynamic list with heavy RLS slows more than a plain query.
- I am honest: RLS does not replace roles; it is a second layer on metadata rights.

## How it works

Limits are set in access profiles and `ДляА`, `ДляБ` templates. The platform appends conditions to query text on `Выполнить()` in user context.

## Pitfalls

**Admin-only test.** Production bug "cannot see documents".
**RLS + OR on many fields.** Query plan degrades.
**Privileged mode on form.** Full bypass from client is invalid.
**Data exchange ignoring RLS.** Foreign rows ship out.
**Duplicate limits in code and RLS.** Logic drift.
