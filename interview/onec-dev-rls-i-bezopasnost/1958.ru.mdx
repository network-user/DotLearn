# RLS на сервере и на клиенте: особенности применения

## Суть

RLS применяется при серверных операциях чтения и запросов в контексте пользователя. Клиент не обходит RLS, если не вызывает привилегированный серверный код. Толстый клиент и веб-клиент одинаково зависят от серверной проверки.

## Что ответить на собесе

- Я выполняю запросы и чтение объектов на сервере с `&НаСервере` без привилегий.
- `УстановитьПривилегированныйРежим` только для служебных операций с журналированием.
- Не кэширую на клиенте данные, недоступные пользователю по RLS.
- COM и внешние соединения наследуют сеанс пользователя или сервисного аккаунта явно.
- Честно: «тонкий клиент безопаснее» неверно; безопасность в серверном контексте и правах.

## Как это работает

Клиентская процедура вызывает серверный метод; запрос выполняется от имени текущего пользователя ИБ. Привилегированный общий модуль снимает RLS для всей цепочки вызова внутри модуля.

## Подводные камни

**Привилегированный режим в форме.** Утечка всех строк списка.
**Передача таблицы значений на клиент.** Обход сокрытия в UI.
**Фоновое задание под пользователем без понимания RLS.** Пустые отчёты по почте.
**Сервисный пользователь с полными правами.** Интеграция видит всё.
**Проверка только в UI.** API и регламенты без RLS-эквивалента.
