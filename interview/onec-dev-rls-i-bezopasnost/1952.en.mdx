# How the allowed record set is built under RLS

## In short

The platform computes allowed records as the intersection of role rights, access profiles, and limit templates for the user. The result is embedded in queries and single-reference read checks.

## How to answer in an interview

- I explain the chain: user → access groups → profiles → per-type limits.
- For lists the limit is in SQL; for a single reference a separate read check.
- Data access groups define allowed companies/warehouses.
- Profile change may need re-login or session refresh for full effect.
- I am honest: two profiles with different limits merge by platform rules, not arbitrary "strongest wins".

## How it works

Profile template `ДляА`: `Где Организация В (&ДоступныеОрганизации)`. The platform binds the parameter from user settings on each document query.

## Pitfalls

**Rights without RLS.** User sees metadata but not rows.
**RLS without object role.** Empty list with formally open access.
**Stale access group cache.** User sees old set until re-login.
**Test under full rights.** False confidence in report.
**Duplicate limit in report query.** Double filter and confusion.
