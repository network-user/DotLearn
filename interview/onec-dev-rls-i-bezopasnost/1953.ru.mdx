# Порядок наложения фильтров RLS в запросе

## Суть

Сначала применяются права на объект метаданных, затем ограничения профилей доступа к данным, затем условия из текста запроса пользователя. Платформа объединяет шаблоны ограничений логическим И для одного профиля и по правилам объединения профилей.

## Что ответить на собесе

- Я не полагаюсь на порядок `ГДЕ` в исходном тексте как на порядок RLS.
- Пользовательский отбор в списке добавляется поверх ограничений, не заменяет их.
- При нескольких профилях изучаю итоговый SQL в ТЖ.
- Привилегированный режим отключает наложение RLS.
- Честно: «обойти» RLS условием в запросе отчёта нельзя без привилегий.

## Как это работает

Итоговый запрос = исходный текст + фрагменты ограничений для каждой таблицы в FROM. Параметры `&ДоступныеОрганизации` подставляются из сеанса.

## Подводные камни

**Ожидание что RLS в конце.** План оптимизатора другой.
**OR в шаблоне ограничения.** Непредсказуемое расширение выборки.
**Привилегированный модуль без необходимости.** Утечка данных.
**Разные ограничения на join таблицы.** Одна таблица видна, другая пустая.
**Отладка без ТЖ.** Не видно итоговый SQL.
