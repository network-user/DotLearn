# Как формируется набор разрешённых записей при RLS

## Суть

Платформа вычисляет разрешённые записи как пересечение прав роли, профилей доступа и шаблонов ограничений для пользователя. Результат встраивается в запросы и проверки чтения объекта по ссылке.

## Что ответить на собесе

- Я объясняю цепочку: пользователь → группы доступа → профили → ограничения по типам.
- Для списков ограничение в SQL; для единичной ссылки отдельная проверка чтения.
- Группы доступа к данным задают доступные организации/склады.
- Изменение профиля требует перезахода или обновления сеанса для полного эффекта.
- Честно: два профиля с разными ограничениями объединяются по правилам платформы, не «сильнейший побеждает» наугад.

## Как это работает

Шаблон `ДляА` в профиле: `Где Организация В (&ДоступныеОрганизации)`. Платформа подставляет параметр из настроек пользователя при каждом запросе к документу.

## Подводные камни

**Права без RLS.** Пользователь видит метаданные, но не строки.
**RLS без роли на объект.** Пустой список при формально открытом доступе.
**Кэш старых групп доступа.** Пользователь видит устаревший набор до релогина.
**Тест под полными правами.** Ложная уверенность в отчёте.
**Дублирование ограничения в запросе отчёта.** Двойной фильтр и путаница.
