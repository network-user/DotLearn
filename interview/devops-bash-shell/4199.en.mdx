# How do you write a one-line if in bash?

## In short

A one-line `if` in bash separates `then`, `else`, and `fi` with semicolons: `if condition; then cmd1; else cmd2; fi`. For short guard checks in CI and ad-hoc shell it is a compact alternative to a multi-line block.

## How to answer in an interview

- Classic: `if [ -f file ]; then echo ok; else echo miss; fi`.
- With `[[`, strings and regex are easier: `if [[ -n "$VAR" ]]; then ...; fi`.
- Ternary style via `&&`/`||`: `[[ -d dir ]] && cd dir || exit 1`.
- Honestly: in production scripts a long one-line if hurts review; I expand to a block when a branch has more than one command.

## How it works

Check kubeconfig before apply:

```bash
if [[ -f "$HOME/.kube/config" ]]; then kubectl get nodes; else echo "no kubeconfig" >&2; exit 1; fi
```

Check the previous command's exit code:

```bash
kubectl rollout status deploy/api -n prod; if [[ $? -ne 0 ]]; then logger -t deploy "rollout failed"; exit 1; fi
```

Short guard without else:

```bash
[[ -w /var/log ]] || { echo "log dir not writable" >&2; exit 1; }
```

GitHub Actions and SSH one-liners save escaping; I quote variables inside `[ ]`.

## Pitfalls

- **Empty variable in `[`.** `[ $VAR = x ]` throws a syntax error; quotes or `[[` are required.
- **Missing `fi`.** The shell parses the rest of the line as garbage.
- **`then` after newline without `;`.** In a one-liner `;` is required before `then`.
- **Mixing `[` and `[[` syntax.** `==` in `[` is not POSIX everywhere.
- **Unreadable else with nested if.** Early return in a function is cleaner.
