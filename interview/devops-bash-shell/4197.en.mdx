# What is Bash and where does it fit in DevOps?

## In short

Bash (Bourne Again SHell) is a CLI shell and scripting language on GNU/Linux. It runs commands, pipelines, variables, functions, and conditionals; the de facto glue for CI, deploy hooks, and server troubleshooting.

## How to answer in an interview

- Interactive shell on a jump host and non-interactive scripts in pipelines (`#!/usr/bin/env bash`).
- Automation: backup, rollout, health checks, parsing `kubectl`/`aws` output.
- POSIX sh compatibility is incomplete; in portable scripts I avoid bashisms or require bash explicitly.
- Honestly: I move complex logic to Python/Go; bash stays for CLI orchestration and quick glue jobs.

## How it works

Nightly service health check:

```bash
#!/usr/bin/env bash
set -euo pipefail

services=(api worker scheduler)
fail=0

for svc in "${services[@]}"; do
  if curl -fsS "https://internal/health/$svc" >/dev/null; then
    echo "ok $svc"
  else
    echo "fail $svc" >&2
    fail=1
  fi
done

exit "$fail"
```

Interactive: history expansion, `Ctrl+R`, job control (`fg`, `bg`). Non-interactive mode cares about quoting and word splitting. I check `bash --version`; on macOS default sh is often old bash 3.x, on Linux usually 5.x.

## Pitfalls

- **Unquoted `$@`.** Breaks paths with spaces.
- **`[[` vs `[` on old sh.** Script fails on Alpine without bash in the shebang.
- **Injection via `eval`.** User input in `eval "$cmd"` enables RCE.
- **Globbing without nullglob.** `for f in *.log` with no files runs the literal `*.log` once.
- **Script without shebang.** Running `sh script.sh` instead of bash changes array semantics.
