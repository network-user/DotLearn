# How do you chain multiple bash commands on one line?

## In short

On one line you join commands with `;` (always continue), `&&` (continue on success), `||` (continue on failure), and `&` (background). Parentheses `()` and braces `{ }` group for pipelines and subshells.

## How to answer in an interview

- `cmd1; cmd2` runs both regardless of exit code.
- `cmd1 && cmd2` is the typical guard before a destructive step.
- `cmd1 || cmd2` is fallback or alert on failure.
- Honestly: long one-liners in CI are hard to read and debug; in scripts I split lines with `set -e`.

## How it works

Create a directory and cd only on success:

```bash
mkdir -p /opt/releases/v2.4 && cd /opt/releases/v2.4
```

Fallback when the primary mirror is down:

```bash
curl -fsS https://mirror-a/pkg.tgz -o pkg.tgz || \
  curl -fsS https://mirror-b/pkg.tgz -o pkg.tgz
```

Group for a pipeline:

```bash
( kubectl get pods -n prod | grep -v Running ) || echo "all pods running"
```

Background job with wait:

```bash
long_task.sh & wait $!
```

Makefiles and GitHub Actions use the same operators; remember `set -e` in a script is not always equivalent to manual `&&` in an interactive session.

## Pitfalls

- **`mkdir && cd` without `-p`.** The second command skips, but the first may have changed the FS.
- **`||` swallows exit code.** A `deploy || true` pipeline hides failure in monitoring.
- **Subshell `( )`.** cd variables are not visible to the parent.
- **`&&` and `||` precedence.** Parentheses are required or logic differs from intent.
- **Background without wait.** CI exits before the background task finishes.
