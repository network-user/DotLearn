# Что такое Bash и где он в DevOps?

## Суть

Bash (Bourne Again SHell) это CLI-оболочка и язык сценариев в GNU/Linux. Запускает команды, пайплайны, переменные, функции и условия; де-факто glue для CI, deploy hooks и troubleshooting на серверах.

## Что ответить на собесе

- Интерактивный shell на jump host и non-interactive скрипты в pipeline (`#!/usr/bin/env bash`).
- Автоматизация: backup, rollout, health-check, парсинг `kubectl`/`aws` output.
- POSIX sh совместимость не полная; в portable скриптах избегаю bashisms или явно требую bash.
- Честно: сложную логику переношу в Python/Go; bash оставляю для оркестрации CLI и быстрых glue job.

## Как это работает

Ночной health-check сервисов:

```bash
#!/usr/bin/env bash
set -euo pipefail

services=(api worker scheduler)
fail=0

for svc in "${services[@]}"; do
  if curl -fsS "https://internal/health/$svc" >/dev/null; then
    echo "ok $svc"
  else
    echo "fail $svc" >&2
    fail=1
  fi
done

exit "$fail"
```

Интерактивно: history expansion, `Ctrl+R`, job control (`fg`, `bg`). В non-interactive режиме важны quoting и word splitting. Версию проверяю `bash --version`; на macOS default sh часто старый bash 3.x, в Linux обычно 5.x.

## Подводные камни

- **Unquoted `$@`.** Ломает пути с пробелами.
- **`[[` vs `[` на старом sh.** Скрипт падает на Alpine без bash в shebang.
- **Инъекция через `eval`.** Пользовательский input в `eval "$cmd"` даёт RCE.
- **Глобbing nullglob.** `for f in *.log` без файлов один раз прогоняет литерал `*.log`.
- **Скрипт без shebang.** Запуск `sh script.sh` вместо bash меняет семантику массивов.
