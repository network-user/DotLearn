# HTTPS: из чего складывается защита для QA веб и API?

## Суть

HTTPS это HTTP поверх TLS: канал шифруется, сервер аутентифицируется сертификатом, целостность данных проверяется MAC в рамках записи TLS. Для тестировщика это значит: нет mixed content, валидный сертификат, корректный redirect HTTP→HTTPS, чувствительные cookie только с Secure, плюс проверка, что трафик в DevTools/Charles действительно зашифрован.

## Что ответить на собесе

- TLS даёт три свойства: конфиденциальность (шифрование), аутентификацию сервера (цепочка сертификатов до доверенного CA), целостность (подделка пакета ломает расшифровку).
- Рукопожатие: ClientHello → серверный сертификат → обмен ключами → симметричное шифрование сессии; дальше обычные HTTP-запросы внутри туннеля.
- На регрессе смотрю HSTS, срок действия сертификата, SAN на нужный домен, отсутствие self-signed в prod и что формы логина не уходят по http://.
- Честно: HTTPS не спасает от XSS и утечки в логах приложения; это транспорт, не замена secure coding.

## Как это работает

Упрощённый поток TLS 1.2+:

1. Клиент предлагает cipher suites и SNI (имя хоста).
2. Сервер отдаёт цепочку сертификатов; клиент проверяет подпись CA, срок, имя домена.
3. Стороны согласуют сеансовый ключ (ECDHE) и шифруют Application Data.

Gherkin для smoke веб-приложения:

```gherkin
Feature: HTTPS и транспортная безопасность

  Scenario: Главная доступна только по HTTPS
    When браузер открывает "http://shop.example.com/"
    Then происходит редирект 301 или 302 на "https://shop.example.com/"
    And финальный ответ имеет код 200

  Scenario: Сертификат валиден для домена
    When клиент выполняет GET "https://shop.example.com/health"
    Then TLS handshake успешен
    And subject alternative name содержит "shop.example.com"
    And срок действия сертификата не истёк

  Scenario: Cookie сессии помечена Secure
    Given пользователь успешно логинится
    When браузер сохраняет session cookie
    Then флаг Secure установлен
    And cookie не отправляется на http:// ресурсы
```

В Charles/Fiddler QA проверяет, что тело login/password не видно в plaintext до расшифровки на клиенте (или использует доверенный корневой сертификат прокси только на тестовом стенде). Для API автотестов `verify=true` в клиенте ловит MITM и просроченный cert.

## Подводные камни

- **Mixed content.** Страница https грузит скрипт по http; браузер блокирует или ослабляет защиту.
- **Сертификат на apex без www.** Пользователь на www получает ERR_CERT_COMMON_NAME_INVALID.
- **Истёкший cert на staging.** Команда отключает verify в Postman и тащит привычку в CI.
- **HSTS без preload на поддоменах.** Первый визит на http:// ещё уязвим до редиректа.
- **TLS termination на балансировщике.** Между LB и app может идти plain HTTP; утечка внутри периметра, если сеть не сегментирована.
