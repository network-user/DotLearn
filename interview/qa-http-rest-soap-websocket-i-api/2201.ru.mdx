# REST и Kafka API: когда синхронный HTTP, а когда поток событий?

## Суть

REST API строит обмен по схеме «запрос ответ» поверх HTTP: клиент обращается к ресурсу по URL и получает немедленный результат. Kafka API работает с потоками сообщений в модели pub/sub: продюсер пишет в топик, консьюмеры читают асинхронно. Для QA это два разных набора рисков: контракт HTTP vs порядок, offset и доставка событий.

## Что ответить на собесе

- REST проверяю через CRUD, коды статуса, схему JSON и идемпотентность PUT/DELETE; Kafka через топики, партиции, consumer group и lag.
- REST подходит, когда пользователю нужен мгновенный ответ (форма, каталог); Kafka когда события можно обработать позже (логи, аналитика, интеграции).
- В гибриде часто REST принимает команду, а Kafka несёт факт выполнения; тестирую сквозной сценарий и согласованность данных.
- Честно: «Kafka API» на собесе обычно означает клиентские API продюсера/консьюмера, а не REST-шлюз к брокеру; уточняю, что именно тестируем.

## Как это работает

**REST.** Клиент отправляет HTTP-запрос к ресурсу `/users`, сервер отвечает телом и статусом. Состояние сессии не хранится на сервере между вызовами: каждый запрос самодостаточен. Кэширование возможно по заголовкам. Типичный smoke для QA:

```gherkin
Feature: REST каталог пользователей

  Scenario: GET возвращает список
    When клиент выполняет GET "/users" с Accept application/json
    Then код ответа 200
    And тело содержит массив users

  Scenario: POST создаёт ресурс
    When клиент выполняет POST "/users" с валидным JSON
    Then код ответа 201
    And заголовок Location указывает на новый id
```

**Kafka.** Продюсер публикует записи в топик `user-events`; брокеры хранят их в партициях с offset. Консьюмеры в одной group делят партиции: каждая партиция в момент времени обрабатывается одним инстансом. Сценарий для регрессии интеграции:

```gherkin
Feature: Публикация события активности

  Scenario: Событие login попадает в топик и читается консьюмером
    Given топик "user-events" с 3 партициями
    And консьюмер в группе "analytics-workers" подписан на топик
    When продюсер отправляет запись key "acct-7" value "login"
    Then консьюмер получает сообщение с тем же value
    And offset коммитится после успешной обработки
    And повторная отправка с тем же ключом попадает в ту же партицию
```

Сравнение для тест-плана:

| Критерий | REST | Kafka |
|----------|------|-------|
| Модель | синхронный запрос | асинхронный поток |
| Ошибка видна | сразу в HTTP-коде | через lag, DLQ, retry |
| Масштаб | горизонтально за балансировщиком | партиции + consumer group |
| Типичный баг | неверный статус/схема | дубликаты, потеря порядка по ключу |

## Подводные камни

- **Сравнение «что быстрее».** Kafka выигрывает на объёме событий, REST проще отлаживать в Charles/Postman; смешивать метрики некорректно.
- **REST поверх Kafka без контракта.** HTTP 200 при ошибке записи в топик маскирует сбой; нужны проверки offset и dead-letter.
- **Игнор порядка в партиции.** События одного пользователя без ключа могут прийти не по порядку; регрессия «заказ до оплаты» ломается.
- **Тест только happy path REST.** Асинхронный downstream не покрыт: заказ создан, но письмо из консьюмера не ушло.
- **Путаница Kafka REST Proxy с REST API продукта.** Разные эндпоинты, auth и семантика; чек-лист надо разделять.
