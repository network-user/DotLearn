# REST API: как тестировщику проверять ресурсы и HTTP-методы?

## Суть

REST (Representational State Transfer) это архитектурный стиль для веб-API: ресурсы идентифицируются URL, операции выражаются HTTP-методами, обмен идёт без сохранения состояния на сервере между запросами. Для QA REST даёт предсказуемую матрицу проверок: CRUD, коды ответа, заголовки, идempotентность и контракт JSON.

## Что ответить на собесе

- REST для меня не протокол, а набор ограничений: stateless, единый интерфейс, ресурсы как существительные в URL, глаголы через GET/POST/PUT/PATCH/DELETE.
- Тестирую контракт: статус, тело, схему полей, заголовки Cache-Control и Content-Type, плюс негативные кейсы (401, 404, 422).
- PUT и DELETE проверяю на идемпотентность: повтор запроса не должен плодить дубликаты или менять результат непредсказуемо.
- Честно: «RESTful» на слайдах часто нарушает HATEOAS и версионирование; я сверяю фактическую спецификацию OpenAPI/Swagger, а не buzzword.

## Как это работает

Клиент и сервер разделены: UI не знает, где хранятся данные. Каждый запрос несёт контекст (токен, idempotency-key), сервер не держит сессию бизнес-состояния между вызовами. Ответы можно кэшировать по заголовкам. Многослойность (CDN, gateway, app) прозрачна для клиента, если контракт стабилен.

Типичная модель библиотеки книг:

| Метод | Путь | Смысл |
|-------|------|-------|
| GET | /books | список |
| GET | /books/{id} | одна книга |
| POST | /books | создать |
| PUT | /books/{id} | полная замена |
| PATCH | /books/{id} | частичное обновление |
| DELETE | /books/{id} | удалить |

Gherkin для API-регрессии в Postman/Newman или REST-assured:

```gherkin
Feature: REST контракт каталога книг

  Scenario: GET списка книг
    When клиент отправляет GET "/books" с заголовком Accept application/json
    Then статус ответа 200
    And тело содержит массив items
    And каждый элемент имеет поля id, title, author

  Scenario: POST создаёт ресурс и возвращает Location
    Given валидный JSON книги без id
    When клиент отправляет POST "/books"
    Then статус ответа 201
    And заголовок Location указывает на созданный /books/{id}
    And повторный GET по Location возвращает те же данные

  Scenario: DELETE идемпотентен
    Given существует книга с id "42"
    When клиент дважды отправляет DELETE "/books/42"
    Then первый ответ 204
    And второй ответ 404 или 204 без побочных эффектов
```

Негативные проверки: невалидный JSON, лишние поля (если strict schema), границы пагинации, rate limit 429. Для auth добавляю сценарии без токена и с просроченным JWT.

## Подводные камни

- **POST вместо GET для чтения.** Нарушает кэширование и семантику; тесты через Postman «работают», CDN и мониторинг ломаются.
- **200 OK с ошибкой в теле.** `{ "success": false }` без 4xx/5xx маскирует сбои в алертах и клиентских ретраях.
- **Неверная идемпотентность POST.** Двойной клик «Оплатить» создаёт два заказа без idempotency-key.
- **Версионирование в URL vs заголовке.** `/v1/books` и `/v2/books` требуют отдельной регрессии; смешение ломает мобильных клиентов.
- **Игнор Content-Type.** Сервер отдаёт HTML ошибки nginx вместо JSON; парсер клиента падает неочевидно.
