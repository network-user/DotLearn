# When should a base destructor be virtual?

## In short

A virtual destructor is required if a derived object may be deleted through `Base*` or stored as `unique_ptr<Base>`. Otherwise only ~Base runs and derived resources leak.

## How to answer in an interview

- Interface/base with virtual methods implies virtual ~Base (often `= default`).
- Value types deleted by concrete type or final classes: virtual is optional.
- Rule: polymorphic ownership through a base pointer needs a virtual dtor.
- Protected non-virtual ~Base appears in some abstract factory patterns.
- Honestly: if you only `delete Derived*`, virtual is unnecessary; the issue is upcast delete.

## How it works

```cpp
struct Port {
    virtual ~Port() = default;
    virtual void send() = 0;
};

struct TcpPort : Port {
    ~TcpPort() override { /* close socket */ }
    void send() override {}
};

void shutdown(Port* p) {
    delete p;
}
```

Without virtual ~Port:

```cpp
struct PortBad {
    ~PortBad() {}
};

struct TcpPortBad : PortBad {
    ~TcpPortBad() { /* skipped on PortBad* delete */ }
};
```

`shutdown` on `PortBad*` leaks Tcp state.

## Pitfalls

- **std::unique_ptr incomplete type.** Virtual ~ must be visible where delete happens.
- **Stack polymorphic objects.** Virtual dtor less critical; slicing still bad on copy.
- **Multiple inheritance diamond.** Virtual bases affect destruction order.
- **= default in header.** Inline virtual ~ is fine for polymorphic roots.
- **Non-owning Base*.** Observer pointers without delete do not need virtual ~ for delete safety.
