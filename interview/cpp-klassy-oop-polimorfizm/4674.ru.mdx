# Зачем помечать деструктор базового класса virtual?

## Суть

`virtual ~Base()` гарантирует, что при `delete base_ptr` вызовется цепочка деструкторов от dynamic type до base. Без virtual освобождается только subobject базового класса, ресурсы derived остаются leaked.

## Что ответить на собесе

- Правило: если класс имеет хотя бы один virtual method или удаляют через `Base*`, деструктор base virtual.
- Порядок: ~Derived, затем ~Base.
- Работает через vtable pointer в объекте.
- С smart pointers (`unique_ptr<Base>`) то же правило.
- Честно: если иерархия не polymorphic и delete только по concrete type, virtual не обязателен; для interface base almost always virtual.

## Как это работает

Без virtual:

```cpp
struct Host {
    ~Host() { /* base cleanup */ }
};

struct Worker : Host {
    ~Worker() { /* never runs on base delete */ }
};

void leak() {
    Host* slot = new Worker();
    delete slot;
}
```

С virtual:

```cpp
struct Host {
    virtual ~Host() = default;
};

struct Worker : Host {
    ~Worker() override { /* derived cleanup */ }
};

void ok() {
    Host* slot = new Worker();
    delete slot;
}
```

Вывод: сначала `Worker`, потом `Host`. RAII members derived уничтожаются в ~Worker до base subobject.

`= default` virtual destructor в interface достаточен, если derived сам управляет ресурсами.

## Подводные камни

- **Partial destroy.** Non-virtual delete on derived-only resources через base pointer.
- **Exception in destructor.** Деструкторы не должны бросать; virtual chain усложняет debug.
- **Pure virtual base.** Abstract base всё равно needs virtual ~ (можно defaulted).
- **delete this antipattern.** Ещё опаснее без virtual hierarchy discipline.
- **Object slicing.** Virtual dtor не лечит slicing при copy by value.
