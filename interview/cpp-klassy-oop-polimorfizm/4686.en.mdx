# Why write override on a virtual method?

## In short

`override` documents intent and enables a compile-time check: the derived method must match a virtual base signature. Typos and const mismatches become compile errors instead of silent overloads.

## How to answer in an interview

- It shows this is an override, not a new overload.
- The compiler catches `draw()` vs `draw() const`.
- With `final` you block further overrides.
- It does not replace `virtual` in the base; it applies on the derived side.
- Honestly: cards 4679 and 4686 overlap; one safety-net explanation is enough in an interview.

## How it works

```cpp
struct Device {
    virtual void ping() const;
};

struct Sensor : Device {
    void ping() const override { /* ... */ }
};

void check(const Device& d) {
    d.ping();
}
```

Compile failure:

```cpp
struct Broken : Device {
    void ping() override; // error: base has const, derived omitted const
};
```

With final:

```cpp
struct FixedSensor : Device {
    void ping() const override final {}
};
```

## Pitfalls

- **Missing virtual in base.** override fails compile; fix the base or drop override.
- **Overload sets.** Need `using Base::foo;` before overriding a specific overload.
- **Return type drift.** Covariant rules are narrow; mismatch fails override.
- **Hiding without override.** Without the keyword the compiler accepts an unrelated method silently.
- **Maintenance.** Base signature changes break all overrides loudly (a feature).
