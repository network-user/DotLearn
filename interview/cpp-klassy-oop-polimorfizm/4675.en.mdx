# Why mark a constructor explicit?

## In short

`explicit` disables implicit converting constructors and copy-list initialization from a single argument. The compiler will not synthesize a temporary where you expected a different type.

## How to answer in an interview

- Blocks silent conversion in function calls and assignment.
- Especially important for single-argument ctors: `explicit Foo(int)`.
- C++11+: `explicit operator bool()` controls implicit bool conversions too.
- Improves API clarity: create objects intentionally with `Foo{42}` or `Foo(42)`.
- Honestly: for math types like `Complex(double)` implicit can be convenient; for resource handles use explicit almost always.

## How it works

Without explicit:

```cpp
struct Meter {
    Meter(int value) : value_(value) {}
    int value_;
};

void scan(Meter m) {}

void oops() {
    scan(3); // implicitly Meter(3)
}
```

With explicit:

```cpp
struct Meter {
    explicit Meter(int value) : value_(value) {}
    int value_;
};

void scan(Meter m) {}

void strict() {
    // scan(3); // error
    scan(Meter{3});
}
```

Explicit also affects brace initialization:

```cpp
Meter m = 5;   // OK without explicit
// Meter m2 = 5; // error with explicit
Meter m3{5};     // OK explicit form
```

For converting operators:

```cpp
struct Handle {
    explicit operator bool() const { return ok_; }
    bool ok_;
};
```

## Pitfalls

- **Missing explicit on handles.** `Socket(int fd)` silently accepts any int.
- **initializer_list ctor.** Separate rules; explicit on it blocks `T x = {a,b}`.
- **Copy ctor rarely explicit.** Usually breaks value semantics; choose deliberately.
- **Proxy references.** Types like `std::vector<bool>` proxies favor explicit conversions.
- **Legacy API.** Removing implicit conversion can be a breaking change when modernizing.
