# In what order are destructors called?

## In short

Destructors run in reverse construction order: the last local created is destroyed first; class members in reverse declaration order; inheritance from most derived down to bases.

## How to answer in an interview

- Stack locals: LIFO by scope.
- Members: reverse declaration order in the class body.
- Derived destructors, then bases.
- Static/globals at program exit in reverse initialization order (implementation-defined phases).
- Honestly: unions and trivial members follow the same rules; a trivial destructor may emit no call.

## How it works

```cpp
#include <iostream>
#include <string>

struct Tag {
    explicit Tag(std::string name) : name_(std::move(name)) {
        std::cout << "ctor " << name_ << '\n';
    }
    ~Tag() { std::cout << "dtor " << name_ << '\n'; }
    std::string name_;
};

int main() {
    Tag first("first");
    Tag second("second");
}
```

Dtor output: second, first.

Members:

```cpp
struct Bundle {
    Tag a{"member-a"};
    Tag b{"member-b"};
};
```

~Bundle: member-b, member-a, then ~Bundle.

Inheritance:

```cpp
struct Base { ~Base(); };
struct Derived : Base { ~Derived(); };
```

delete Derived: ~Derived, ~Base.

## Pitfalls

- **Virtual bases.** Most derived drives virtual base subobject destruction once.
- **Exception in dtor.** Other dtors still run for destroyed parts; throw leads to std::terminate.
- **Placement new.** You must call dtors explicitly; order is yours.
- **Thread-local static.** Destroyed on thread exit, not necessarily global exit order.
- **std::exit.** Skips destructors for locals unless atexit handlers run.
