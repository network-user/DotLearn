# Зачем конструктор помечают explicit?

## Суть

`explicit` запрещает неявные converting constructors и copy-list initialization из одного аргумента. Компилятор не подставит временный объект там, где вы ожидали другой тип.

## Что ответить на собесе

- Блокирует silent conversion в function call и assignment.
- Особенно важно для single-argument ctor: `explicit Foo(int)`.
- C++11+: `explicit operator bool()` тоже контролирует неявные bool conversions.
- Улучшает API clarity: объект создают намеренно `Foo{42}` или `Foo(42)`.
- Честно: для mathematical types вроде `Complex(double)` иногда implicit удобен; для resource handles almost always explicit.

## Как это работает

Без explicit:

```cpp
struct Meter {
    Meter(int value) : value_(value) {}
    int value_;
};

void scan(Meter m) {}

void oops() {
    scan(3); // неявно Meter(3)
}
```

С explicit:

```cpp
struct Meter {
    explicit Meter(int value) : value_(value) {}
    int value_;
};

void scan(Meter m) {}

void strict() {
    // scan(3); // ошибка
    scan(Meter{3});
}
```

Также explicit влияет на brace initialization:

```cpp
Meter m = 5;   // без explicit OK
// Meter m2 = 5; // с explicit ошибка
Meter m3{5};     // OK явно
```

Для converting operators:

```cpp
struct Handle {
    explicit operator bool() const { return ok_; }
    bool ok_;
};
```

## Подводные камни

- **Забыли explicit на handle types.** `Socket(int fd)` silently принимает любой int.
- **initializer_list ctor.** Отдельные правила; explicit на нём блокирует `T x = {a,b}`.
- **Copy ctor rarely explicit.** Обычно explicit copy ctor ломает value semantics; нужен осознанно.
- **std::vector<bool> style proxies.** Proxy references любят explicit conversions.
- **Legacy API.** Удаление implicit может быть breaking change при модернизации.
