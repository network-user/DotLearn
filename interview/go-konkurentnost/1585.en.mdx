# Buffered vs unbuffered channels

## In short

Unbuffered (cap 0): synchronous rendezvous; the sender waits for the receiver. Buffered (cap N): async up to N elements, decoupling timing. Both are thread-safe via the runtime; use `<-chan` / `chan<-` for API contracts.

## How to answer in an interview

- Unbuffered: a handoff sync point.
- Buffered: a queue between goroutines; backpressure when full.
- Nil, closed, and directional types follow separate rules.
- Honestly: buffer size is an empirical latency vs throughput tradeoff.

## How it works

```go
package main

import "fmt"

func main() {
	handshake := make(chan string)
	pipeline := make(chan string, 3)

	go func() {
		handshake <- "sync"
		pipeline <- "queued"
	}()

	fmt.Println(<-handshake)
	fmt.Println(<-pipeline)
}
```

Handshake blocks until main reads. Pipeline send may complete before recv if cap allows.

## Pitfalls

**Buffer hides a missing consumer.** Latent deadlock.

**Unbuffered fan-in without a coordinator.** Circular wait risk.

**Copy by value on send.** Large structs are costly.

**Same channel with RO/RW types.** Compile-time safety only.

**One-element buffer «fixes» a race.** Masks a design issue.
