# What knobs control the Go scheduler?

## In short

The main runtime knob is `GOMAXPROCS` (env or `runtime.GOMAXPROCS`). For observability: `GODEBUG=schedtrace`, `scheddetail`, execution tracer. `GOGC` affects GC pauses and indirectly scheduling; `GOTRACEBACK` is for panic stacks only, not the scheduler.

## How to answer in an interview

- GOMAXPROCS = number of P = max goroutines executing on CPU in parallel.
- GODEBUG schedtrace prints runq sizes, threads, idle procs every N ms.
- In code: `runtime.GOMAXPROCS`, `debug.SetGCPercent`, trace.Start.
- Honestly: no fine-grained scheduler tuning from user code; only these env vars and profiling.

## How it works

```go
package main

import (
	"fmt"
	"runtime"
	"runtime/debug"
)

func main() {
	prev := runtime.GOMAXPROCS(2)
	fmt.Println("gomaxprocs was:", prev, "now:", runtime.GOMAXPROCS(0))
	debug.SetGCPercent(100)
	fmt.Println("num cpu:", runtime.NumCPU())
}
```

Env before start:

```
GOMAXPROCS=4 GODEBUG=schedtrace=1000 ./app
```

schedtrace on stderr shows `# Threads`, `# IdleThreads`, `# Goroutines` snapshots.

## Pitfalls

**Confusing GOGC with scheduler.** GOGC is GC cadence, not P count.

**GOTRACEBACK for perf.** Does not affect scheduling, only panic output.

**schedtrace in prod.** Verbose stderr flood; use only in short debug windows.

**GOMAXPROCS >> cores in k8s.** Throttling and latency without automaxprocs.

**LockOSThread is not env.** It is an API call, not an environment variable.
