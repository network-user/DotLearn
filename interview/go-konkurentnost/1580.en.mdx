# Can you control select case order

## In short

Select does not guarantee case order when multiple operations are ready: the runtime picks pseudo-randomly among ready branches. Manual priority requires logic outside select, not the order cases are written.

## How to answer in an interview

- One ready case: it runs.
- Several ready: fair pseudo-random, not top-to-bottom.
- `default` is non-blocking when nothing is ready.
- Honestly: strict priority needs a separate dispatcher, not select alone.

## How it works

```go
package main

import (
	"fmt"
	"time"
)

func main() {
	fast := make(chan string, 1)
	slow := make(chan string, 1)
	fast <- "A"
	slow <- "B"

	select {
	case v := <-slow:
		fmt.Println("slow:", v)
	case v := <-fast:
		fmt.Println("fast:", v)
	default:
		fmt.Println("none ready")
	}

	time.Sleep(10 * time.Millisecond)
}
```

Both channels are ready: which case runs is pseudo-random. Source order is not priority.

## Pitfalls

**Assuming the first case wins.** Flaky tests.

**Select in a tight loop without default.** CPU spin when only one branch could run.

**Nil chan in select.** Never ready, always skipped.

**Send and recv ready together.** Also random among ready ops.

**Priority via nested select.** Works but verbose.
