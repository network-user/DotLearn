# Где goroutine хранит локальные данные?

## Суть

У каждой goroutine свой стек, стартующий с малого размера (порядка килобайт) и растущий по требованию. Локальные переменные и call frames живут на стеке; если компилятор не может доказать bounded lifetime, значение escape в heap через allocator.

## Что ответить на собесе

- Stack growth копирует стек в больший сегмент; указатели в heap обновляются runtime.
- Большие объекты и замыкания, убегающие из функции, аллоцируются в heap, не на стеке goroutine.
- `new` и литералы `&T{}` не означают автоматически heap; решает escape analysis.
- Честно уточняю: бесконечная рекурсия или огромный stack frame всё равно упирается в лимит памяти процесса.

## Как это работает

```go
package main

import "fmt"

func leaf(n int) int {
	buf := [4]int{n, n + 1, n + 2, n + 3}
	return buf[0] + buf[3]
}

func main() {
	go func() {
		fmt.Println(leaf(10))
	}()
}
```

Массив `buf` скорее останется на стеке `leaf`, пока не escape. Если вернуть указатель на элемент `buf`, компилятор перенесёт allocation в heap.

## Подводные камни

- **Escape через interface.** Присвоение локальной переменной `any` часто уводит в heap.
- **Замыкание на loop variable.** Указатель на меняющуюся переменную цикла escape и гонки.
- **Stack shrink.** Стек может сжиматься; не храню сырые указатели в стек вне Go правил.
- **Cgo stack.** Отдельные ограничения; переход границы дорогой.
- **Профилирование.** `go build -gcflags=-m` показывает escape; без него угадывание про heap неточно.
