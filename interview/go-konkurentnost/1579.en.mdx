# Using range over a channel

## In short

`for v := range ch` reads until `close(ch)`. After close, remaining values are drained, then the loop exits. Without close, range blocks forever. The sender should close, not the consumer.

## How to answer in an interview

- Range is cleaner than manual `v, ok := <-ch`.
- Close signals EOF; `ok=false` on the last zero value once drained.
- Send after close panics.
- Honestly: range is wrong if the writer never closes by design.

## How it works

```go
package main

import "fmt"

func emit(out chan<- int) {
	for i := 1; i <= 3; i++ {
		out <- i
	}
	close(out)
}

func main() {
	stream := make(chan int)
	go emit(stream)
	for n := range stream {
		fmt.Println(n)
	}
}
```

The producer closes after the last send. The consumer range gets 1, 2, 3 and exits without an ok check.

## Pitfalls

**Forgot close.** Range deadlocks.

**Double close.** Panic.

**Close from receive-only side.** Design smell; typing can prevent misuse.

**Buffered leftovers.** Close allows draining without new sends.

**Break from range without draining.** Goroutine leak if the writer blocks on send.
