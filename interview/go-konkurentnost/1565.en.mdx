# Why create child contexts?

## In short

A child ctx inherits cancel/deadline/value chain from the parent but can add its own timeout, cancel func, or value. Canceling the parent cancels all descendants. It is a lifecycle tree for request to subtasks without global state.

## How to answer in an interview

- WithCancel: manual shutdown of a subtree.
- WithTimeout/WithDeadline: sub-operation budget inside a request.
- WithValue: request-scoped metadata (trace id), used sparingly.
- Honestly: I do not store optional params in ctx; only cancel/deadline plus a few well-known keys.

## How it works

```go
package main

import (
	"context"
	"fmt"
	"time"
)

func handle(ctx context.Context) {
	subCtx, cancel := context.WithTimeout(ctx, 400*time.Millisecond)
	defer cancel()
	if err := subWork(subCtx); err != nil {
		fmt.Println("sub:", err)
	}
}

func subWork(ctx context.Context) error {
	select {
	case <-time.After(900 * time.Millisecond):
		return nil
	case <-ctx.Done():
		return ctx.Err()
	}
}

func main() {
	root, stop := context.WithCancel(context.Background())
	defer stop()
	handle(root)
}
```

subCtx inherits stop from root but also expires at 400ms. Root cancel still cancels sub even before timeout.

## Pitfalls

**Orphan child without defer cancel.** Timer leak like the WithTimeout rule.

**WithValue abuse.** Untyped keys, mutable values, hidden dependencies.

**Detach ctx from parent.** context.WithoutCancel (Go 1.21+) is rare; know semantics.

**Pass Background mid-chain.** Loses cancel propagation.

**Deep tree cancel storm.** Many children: ensure idempotent cleanup.
