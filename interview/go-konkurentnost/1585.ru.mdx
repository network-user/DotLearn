# Buffered vs unbuffered channels

## Суть

Unbuffered (cap 0): synchronous rendezvous, sender waits receiver. Buffered (cap N): async до N элементов, decouples timing. Оба thread-safe через runtime; direction `<-chan` / `chan<-` для API contracts.

## Что ответить на собесе

- Unbuffered: handoff sync point.
- Buffered: queue между goroutines, backpressure при full.
- Nil, closed, directional types: отдельные правила.
- Честно: выбор buffer size эмпирика под latency vs throughput.

## Как это работает

```go
package main

import "fmt"

func main() {
	handshake := make(chan string)
	pipeline := make(chan string, 3)

	go func() {
		handshake <- "sync"
		pipeline <- "queued"
	}()

	fmt.Println(<-handshake)
	fmt.Println(<-pipeline)
}
```

Handshake blocks until main reads. Pipeline send может complete до recv если cap позволяет.

## Подводные камни

**Buffer hides missing consumer.** Latent deadlock.

**Unbuffered for fan-in without coordinator.** Risk circular wait.

**Copy by value on send.** Large struct expensive.

**Same chan RO/RW types.** Compile-time safety only.

**1-element buffer «fixes» race.** Masks design issue.
