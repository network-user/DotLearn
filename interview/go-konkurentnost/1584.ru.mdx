# Send в nil channel

## Суть

Операции на nil chan block forever: send и recv never proceed. Select с nil case never ready. Это не panic, а permanent block. Частая ошибка: uninitialized var или reset в nil вместо close.

## Что ответить на собесе

- `var ch chan T` then `ch <- x`: deadlock class block.
- Nil в select игнорируется как never ready.
- Отличие от closed: closed recv OK, nil recv block.
- Честно: nil chan иногда используют намеренно disable branch в select.

## Как это работает

```go
package main

import (
	"fmt"
	"time"
)

func main() {
	var disabled chan int
	enabled := make(chan int, 1)

	go func() {
		fmt.Println("blocked send starting")
		disabled <- 1
		fmt.Println("unreachable")
	}()

	select {
	case enabled <- 42:
		fmt.Println("sent to enabled")
	case disabled <- 99:
		fmt.Println("never")
	case <-time.After(100 * time.Millisecond):
		fmt.Println("timeout, nil branch skipped")
	}
}
```

Send на nil park goroutine. Select пропускает nil case, другие branches работают.

## Подводные камни

**Forgot make.** Classic init bug.

**Set ch=nil to «close».** Wrong; use close or separate done chan.

**WaitGroup + nil send deadlock.** main exits, leak G.

**Interface holding nil chan.** Typed nil vs untyped nil in struct.

**Reflect on nil chan.** Still blocks on send.
