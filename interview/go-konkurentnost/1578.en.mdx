# Is channel buffer size required in make

## In short

The second `make` argument is optional. Without it cap=0: an unbuffered channel with rendezvous sync. With a positive cap: buffered sends do not block while len < cap.

## How to answer in an interview

- `make(chan T)` and `make(chan T, 0)` are equivalent: synchronous handoff.
- Buffer > 0 decouples producer and consumer for N elements.
- A nil chan without make blocks forever (a separate case).
- Honestly: unbuffered is easier to reason about; buffered helps throughput and burst smoothing.

## How it works

```go
package main

import "fmt"

func main() {
	syncCh := make(chan string)
	bufCh := make(chan string, 2)

	go func() { syncCh <- "ping" }()
	fmt.Println(<-syncCh)

	bufCh <- "a"
	bufCh <- "b"
	fmt.Println(<-bufCh, <-bufCh)
}
```

Unbuffered: sender and receiver meet on one operation. Buffered: two sends without a receiver goroutine until cap is exhausted.

## Pitfalls

**Confusing zero-value chan with make.** `var c chan int` is nil, not unbuffered.

**Buffer=1 «just in case».** Can hide a design deadlock.

**Range without close on unbuffered.** Reader waits forever.

**Huge buffer + slow consumer.** OOM from queued elements.

**Directional chan types in APIs.** `<-chan` vs `chan<-` does not change buffer semantics.
