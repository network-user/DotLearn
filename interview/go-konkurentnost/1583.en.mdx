# Sending on a closed channel

## In short

Send after `close(ch)` panics: `send on closed channel`. Recv from a closed channel is legal: zero value plus `ok=false` after drain. Only the sender side should close; closing twice also panics.

## How to answer in an interview

- Write on closed: panic unless recovered.
- Read closed: drain then (zero, false).
- Close means «no more data».
- Honestly: use sync.Once or a dedicated closer goroutine for broadcast shutdown.

## How it works

```go
package main

import "fmt"

func main() {
	inbox := make(chan int, 1)
	inbox <- 42
	close(inbox)

	v, ok := <-inbox
	fmt.Println(v, ok)

	defer func() {
		if r := recover(); r != nil {
			fmt.Println("panic:", r)
		}
	}()
	inbox <- 1
}
```

First recv succeeds. Send after close panics; recover if needed.

## Pitfalls

**Concurrent close + send.** Race; panic timing is nondeterministic.

**Close from consumer side.** Protocol violation, subtle bugs.

**Range after partial drain + send attempt.** Writer must stop before close.

**Buffered: send before close OK, after panic.**

**Select send on closed.** Same panic as a direct send.
