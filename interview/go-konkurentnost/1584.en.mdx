# Sending on a nil channel

## In short

Operations on a nil channel block forever: send and recv never proceed. Select treats a nil case as never ready. No panic, just a permanent block. Common mistake: uninitialized var or resetting to nil instead of close.

## How to answer in an interview

- `var ch chan T` then `ch <- x`: deadlock-class block.
- Nil in select is ignored as never ready.
- Unlike closed: closed recv is OK; nil recv blocks.
- Honestly: nil chan is sometimes used to disable a select branch on purpose.

## How it works

```go
package main

import (
	"fmt"
	"time"
)

func main() {
	var disabled chan int
	enabled := make(chan int, 1)

	go func() {
		fmt.Println("blocked send starting")
		disabled <- 1
		fmt.Println("unreachable")
	}()

	select {
	case enabled <- 42:
		fmt.Println("sent to enabled")
	case disabled <- 99:
		fmt.Println("never")
	case <-time.After(100 * time.Millisecond):
		fmt.Println("timeout, nil branch skipped")
	}
}
```

Send on nil parks the goroutine. Select skips the nil case; other branches still run.

## Pitfalls

**Forgot make.** Classic init bug.

**Set ch=nil to «close».** Wrong; use close or a separate done channel.

**WaitGroup + nil send deadlock.** main exits, G leak.

**Interface holding nil chan.** Typed nil vs untyped nil in struct.

**Reflect on nil chan.** Still blocks on send.
