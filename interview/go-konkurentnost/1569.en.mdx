# How do you send 5 HTTP requests in parallel?

## In short

Fan-out: one goroutine per URL, shared `http.Client`, `WaitGroup` or errgroup for sync. Pass request-scoped ctx with timeout; collect results via channel or mutex-backed slice. Client connection pool reuses TCP.

## How to answer in an interview

- errgroup.WithContext: first error cancels siblings.
- Reuse http.Client, do not NewClient per request.
- Limit concurrency with a sem when URLs >> 5.
- Honestly: "5 parallel" is fine with bare goroutines; production adds timeout and max idle conns.

## How it works

```go
package main

import (
	"context"
	"fmt"
	"net/http"
	"sync"
	"time"
)

func fetchAll(ctx context.Context, urls []string) []int {
	client := &http.Client{Timeout: 3 * time.Second}
	out := make([]int, len(urls))
	var wg sync.WaitGroup
	for i, url := range urls {
		wg.Add(1)
		go func(idx int, u string) {
			defer wg.Done()
			req, err := http.NewRequestWithContext(ctx, http.MethodGet, u, nil)
			if err != nil {
				return
			}
			resp, err := client.Do(req)
			if err != nil {
				return
			}
			resp.Body.Close()
			out[idx] = resp.StatusCode
		}(i, url)
	}
	wg.Wait()
	return out
}

func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
	defer cancel()
	urls := []string{
		"https://example.com",
		"https://example.org",
		"https://example.net",
		"https://example.edu",
		"https://example.gov",
	}
	fmt.Println(fetchAll(ctx, urls))
}
```

Five goroutines, one client, indexed slice avoids append races.

## Pitfalls

**No ctx on request.** Hangs past shutdown.

**Shared resp without Close.** FD leak.

**Race on result slice append.** Preallocate by index or use a channel.

**Unbounded parallel to one host.** Rate limit or sem; respect the server.

**Default client no timeout.** Stuck forever on bad network.
