# What is a goroutine in Go?

## In short

A goroutine is a lightweight unit of execution started with the `go` keyword. The Go scheduler multiplexes thousands of goroutines onto OS threads (M:N), switching on I/O blocks and syscalls.

## How to answer in an interview

- Start with `go worker()`; stacks begin small and grow as needed.
- The main goroutine exits the process without waiting for background work unless you sync (WaitGroup, channel, context).
- Goroutines are cheaper than OS threads; typical services use worker pools, not one thread per request.
- I honestly add: a leaked goroutine blocked forever on a channel is as bad as a thread leak.

## How it works

```go
package main

import (
	"fmt"
	"sync"
)

func worker(id int, wg *sync.WaitGroup) {
	defer wg.Done()
	fmt.Println("start", id)
}

func main() {
	var wg sync.WaitGroup
	for i := 1; i <= 3; i++ {
		wg.Add(1)
		go worker(i, &wg)
	}
	wg.Wait()
	fmt.Println("done")
}
```

The runtime GMP model: G (goroutine) runs on M (OS thread) with P (processor) queues. When blocked on a channel or network syscall, G yields M so another G can run.

## Pitfalls

- **Main exits early.** Bare `go` work is cut off when main returns.
- **Closure loop capture.** A classic bug before Go 1.22; I still watch closure bindings.
- **GOMAXPROCS.** Affects CPU-bound parallelism, not how many goroutines you can create.
- **Panic in a goroutine.** Kills only that G without recover; does not always crash the process.
- **Too many Gs.** Millions are possible, but scheduler and stack memory still have limits.
