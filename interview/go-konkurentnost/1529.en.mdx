# Who should close a channel and how?

## In short

Close only with `close(ch)` and only by the sender (producer) when no more values will be sent. Receivers drain what remains; after that `v, ok := <-ch` yields the zero value and `ok=false`. Send on a closed channel panics.

## How to answer in an interview

- `for v := range ch` ends after close and reading all values.
- Double close panics; receive on a nil channel blocks forever.
- Close signals "no more data", not necessarily an error.
- I honestly add: with multiple producers I coordinate a single closer via WaitGroup or a dedicated coordinator goroutine.

## How it works

```go
package main

import "fmt"

func producer(out chan<- int) {
	for i := 0; i < 3; i++ {
		out <- i
	}
	close(out)
}

func main() {
	ch := make(chan int)
	go producer(ch)

	for v := range ch {
		fmt.Println(v)
	}
}
```

After `close`, a buffered channel delivers queued values, then `ok=false`. Multiple receivers can range one channel, but one consumer is the usual pattern.

## Pitfalls

- **Consumer must not close.** Receivers should not close while senders may still write.
- **Send after close.** Panics; use `select` with `ctx.Done()` or a separate done signal.
- **Checking closed before send.** No safe API; ownership rules instead.
- **Zero after close.** Zero value is indistinguishable from a real zero without a protocol.
- **WaitGroup + close race.** Close only after all producers finished sending.
