# Why use mutable in a const method?

## In short

`mutable` marks a field that may change even inside `const` member functions. It splits logical constness (the client-facing object contract) from physical constness (bytes in memory). Common cases: lazy caches, mutexes, profiling counters.

## How to answer in an interview

- I explain logical vs physical const.
- I give examples: memoization, `std::mutex` in a const method, hit counters.
- I stress `mutable` does not make the object thread-safe by itself.
- I note abuse breaks the "const means no side effects" expectation.
- I add honestly: in C++17+ sometimes external cache or explicit `atomic` is clearer.

## How it works

The compiler lifts const restrictions on `mutable` members inside const member functions. Other fields stay read-only for that method.

```cpp
#include <iostream>
#include <string>

class Label {
    std::string text;
    mutable std::size_t hash_cache = 0;
    mutable bool hash_ready = false;

public:
    explicit Label(std::string value) : text(std::move(value)) {}

    const std::string& value() const { return text; }

    std::size_t cached_hash() const {
        if (!hash_ready) {
            hash_cache = std::hash<std::string>{}(text);
            hash_ready = true;
        }
        return hash_cache;
    }
};

int main() {
    const Label tag("sensor");
    std::cout << tag.cached_hash() << '\n';
}
```

Callers see `cached_hash` as a const operation even though the cache updates internally.

## Pitfalls

- **Hidden mutation.** A const method with mutable can surprise API readers.
- **Threads.** Lazy cache without a mutex races.
- **Cache invalidation.** When `text` changes, non-const methods must reset `hash_ready`.
- **Not a sync replacement.** `mutable mutex` helps only with correct locking.
- **Const correctness.** Too much mutable often signals a class split problem.
