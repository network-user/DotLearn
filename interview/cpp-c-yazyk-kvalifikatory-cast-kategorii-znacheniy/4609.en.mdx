# How do you overload prefix operator++?

## In short

Prefix `++obj` is implemented as `T& operator++()` with no parameters: mutate state first, then return `*this`. The postfix form differs by a dummy `int` and returns a copy from before the increment.

## How to answer in an interview

- Prefix signature: `Counter& operator++()`; postfix: `Counter operator++(int)`.
- Prefix is cheaper for iterators: no extra temporary copy.
- STL-style iterators usually implement both so `*it++` matches built-in behavior.
- const-correctness: increment is almost always a non-const member.
- Honestly: if the type is not semiregular, postfix `++` can be expensive or undesirable.

## How it works

The compiler distinguishes overloads by the dummy `int`. The prefix operator updates `value` and returns a reference to the current object, so chains like `++++c` are possible (though avoided in production).

```cpp
#include <iostream>

class TickCounter {
    int ticks;

public:
    explicit TickCounter(int start) : ticks(start) {}

    TickCounter& operator++() {
        ++ticks;
        return *this;
    }

    TickCounter operator++(int) {
        TickCounter snapshot(ticks);
        ++ticks;
        return snapshot;
    }

    int read() const { return ticks; }
};

int main() {
    TickCounter counter(5);
    ++counter;
    std::cout << counter.read() << '\n';
    return 0;
}
```

`operator++()` bumps `ticks` and yields a reference; `++counter` matches built-in increment reference semantics.

## Pitfalls

- **Prefix vs postfix confusion.** One extra `int` in the signature changes return type and cost.
- **Returning by value from prefix.** `Counter operator++()` instead of `Counter&` breaks expectations and copies extra.
- **Double mutation in postfix.** Snapshot first, then increment.
- **Iterator requirements.** InputIterator needs prefix; RandomAccess often needs both.
- **Proxy iterators.** Some containers return proxies; postfix may be deleted or restricted.
