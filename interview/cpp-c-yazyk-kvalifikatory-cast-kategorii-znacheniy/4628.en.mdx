# When should you use static_cast?

## In short

Use `static_cast` for conversions the compiler can validate statically: numeric conversions, void*↔T*, hierarchy up/downcasts without runtime checks, explicit enum↔int, and safe user-defined conversion chains without reinterpret.

## How to answer in an interview

- Numbers: `double`→`int`, enum class↔underlying type.
- Hierarchy: upcast is often implicit; downcast with `static_cast` when logic guarantees the type.
- void* from C APIs back to `T*` after a known allocation site.
- Not for polymorphic downcasts without a guarantee; use `dynamic_cast`.
- Honestly: `static_cast` downcast is faster than RTTI, but a wrong type is UB.

## How it works

Compile-time checks legality; runtime does not validate downcasts.

```cpp
#include <iostream>

struct Base { virtual ~Base() = default; };
struct Derived : Base { int tag = 7; };

int main() {
    const double ratio = 2.9;
    int whole = static_cast<int>(ratio);

    Derived concrete;
    Base* poly = &concrete;
    Derived* back = static_cast<Derived*>(poly);

    std::cout << whole << ' ' << back->tag << '\n';
    return 0;
}
```

The upcast/downcast is fine because `poly` really points to `Derived`. Without that guarantee use `dynamic_cast`.

## Pitfalls

- **Narrowing surprises.** Fraction loss, modular overflow for unsigned.
- **Unchecked downcast.** Silent UB on wrong dynamic type.
- **void* round trip.** Only if lifetime and alignment are preserved.
- **const removal.** Use `const_cast` for const, not `static_cast`.
- **Overloading ambiguity.** Casts may pick an unexpected overload set.
