# На что влияет ключевое слово inline?

## Суть

`inline` в C++ это в первую очередь ODR/linkage rule: одинаковые определения функции или переменной можно включать из заголовка в несколько TU без multiply-defined symbol. Подсказка компилятору встроить тело вторична и может быть проигнорирована.

## Что ответить на собесе

- Главное: разрешение multiple identical definitions при линковке (header-only helpers).
- Inlining решает optimizer; `inline` не гарантирует отсутствие call instruction.
- Member functions, defined inside class, implicitly inline.
- `inline` variables (C++17) для header-only constants.
- Честно: раздувание кода от агрессивного inline бьёт по I-cache; меряю профилем.

## Как это работает

Линкер схлопывает weak ODR-inline definitions. Компилятор может подставить тело в call site, если heuristics выгодны.

```cpp
inline int combine(int left, int right) {
    return left + right;
}

inline constexpr int scale(int v) {
    return v * v;
}

int demo() {
    return combine(scale(3), scale(4));
}
```

`combine` и `scale` могут жить в `.h` и включаться в несколько `.cpp` без linker error. Вызов `combine` может исчезнуть после inlining, но это не обязательно.

## Подводные камни

- **Разные тела в TU.** Identical requirement нарушен: UB или ODR violation.
- **inline на больших функциях.** Рост бинарника и miss rate кэша.
- **inline vs static в .cpp.** `static` функция локальна для TU; `inline` экспортируется с merge.
- **Отладка и breakpoints.** Inlined code сложнее степать в debugger.
- **LTO может inline и без keyword.** Современные toolchain часто игнорируют hint для hot/cold решений.
