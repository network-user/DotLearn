# Гарантирует ли inline подстановку в месте вызова?

## Суть

Нет. `inline` это подсказка компилятору и одновременно ODR-механизм для определений в заголовках. Решение о inlining принимает оптимизатор с учётом размера функции, `-O` уровня, профиля вызовов. Рекурсия и большие тела почти никогда не встраиваются.

## Что ответить на собесе

- Отвечаю «не гарантирует»; компилятор может inline и без ключевого слова.
- Второй смысл `inline`: разрешить multiple definition в TU через weak linkage.
- Называю факторы: размер, виртуальность, взять адрес функции, LTO.
- Упоминаю `__attribute__((always_inline))` / `[[gnu::always_inline]]` как сильнее, но тоже не абсолют.
- Честно добавляю: микрооптимизация через `inline` в .cpp редко даёт выигрыш; профилирую перед раздуванием кода.

## Как это работает

На этапе оптимизации LLVM/GCC оценивает cost model. Маленькая `add` может раскрыться в `a + b` даже без `inline`. Адрес функции (`&add`) часто блокирует inlining в этом TU.

```cpp
#include <iostream>

inline int add(int lhs, int rhs) {
    return lhs + rhs;
}

int use_add(int a, int b) {
    return add(a, b);
}

int main() {
    std::cout << use_add(5, 3) << '\n';
}
```

С `-O2` вызов часто исчезает, но стандарт не обещает этого. `inline` в заголовке позволяет включать определение в несколько .cpp без linker error.

## Подводные камни

- **Code bloat.** Агрессивный inline раздувает бинарник и бьёт по I-cache.
- **Отладка.** Inlined код сложнее степать в debugger.
- **ABI и адрес.** `static` function pointer на `inline` может оставить out-of-line copy.
- **Рекурсия.** Невозможна true inline.
- **Путаница с macro.** `inline` функция типобезопасна, макрос нет.
