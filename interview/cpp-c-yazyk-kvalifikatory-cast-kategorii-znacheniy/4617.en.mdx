# Why use lambdas when regular functions exist?

## In short

A lambda is a local functor: logic sits next to the call site, needs no separate namespace name, and can capture its environment. A free function cannot see locals; for STL callbacks without lambdas you would use globals or state structs.

## How to answer in an interview

- Locality: comparators/predicates inline in `std::sort`, `std::find_if`.
- Capture `[threshold]`, `[&ctx]` without boilerplate structs with `operator()`.
- Generic lambdas (auto parameters) since C++14 reduce template noise.
- Regular functions remain better for reuse across TUs and public APIs.
- Honestly: "you cannot do without them" is wrong; lambdas are ergonomics, not a capability functions + structs lack.

## How it works

The compiler generates a unique closure type; STL templates accept it like any callable.

```cpp
#include <algorithm>
#include <iostream>
#include <vector>

int main() {
    std::vector<int> scores{11, 4, 9, 15, 7};
    int pass_mark = 10;

    std::sort(scores.begin(), scores.end(), [pass_mark](int lhs, int rhs) {
        const bool lhs_pass = lhs >= pass_mark;
        const bool rhs_pass = rhs >= pass_mark;
        if (lhs_pass != rhs_pass) return lhs_pass > rhs_pass;
        return lhs > rhs;
    });

    for (int s : scores) {
        std::cout << s << ' ';
    }
    return 0;
}
```

`pass_mark` is captured by value; the comparator sees the threshold without globals or a friend struct. The same effect needs a noisier `struct Sorter { int pass_mark; bool operator()(int,int) const; };`.

## Pitfalls

- **Stateful lambda in async.** Capture by reference plus posting to a thread = dangling.
- **Bloat in hot paths.** Large bodies belong in named functions for tests and reuse.
- **Not a function pointer with capture.** `[](int x){}` works; `[n](int x){}` does not.
- **Recursive lambda.** Needs `std::function` or a Y-combinator trick; a named function is simpler.
- **Over-local duplication.** Identical lambdas copied instead of a shared helper.
