# Why is const needed in C++?

## In short

`const` enforces immutability at compile time for variables, pointers, references, and methods. It cuts accidental mutation risk, makes intent explicit, and enables optimizations because the compiler knows a value will not change.

## How to answer in an interview

- I mark data `const` when it must not change after initialization.
- I distinguish `const T*`, `T* const`, and `const T* const` using right-to-left reading.
- `const` methods do not change the object's logical state; const overloads expose different APIs.
- I pass `const T&` to avoid copies and prevent the caller from mutating the argument.
- Honestly: `const` does not stop mutation through a non-const alias or `const_cast`; it is a contract, not a runtime fence.

## How it works

You can apply the qualifier to an object, pointer, reference, or method.

```cpp
const int limit = 100;

const int* ro = &limit;
int value = 5;
int* const pinned = &value;

const int& alias = limit;

class Sensor {
public:
    int read() const { return reading_; }
    void calibrate(int v) { reading_ = v; }
private:
    int reading_ = 0;
};
```

`const int* ro` forbids changing `*ro`, but `ro` itself can be reassigned. `int* const pinned` fixes the address while the pointed-to value may change. A `const` method like `read()` cannot modify non-`mutable` members. Violations fail at compile time instead of silently breaking at runtime.

## Pitfalls

- **const order.** `const int*` and `int const*` are equivalent; `int* const` is different: the pointer itself is const.
- **Returning const by value.** Pointless for primitives; for heavy types it can hide moves from temporaries.
- **mutable.** Allows changing a field inside a `const` method; overuse breaks immutability claims.
- **const_cast.** Removing const is valid only if the original object was not const; otherwise UB.
- **constexpr tie-in.** `constexpr` is stronger: it demands compile-time evaluation where applicable; do not conflate them in an interview.
