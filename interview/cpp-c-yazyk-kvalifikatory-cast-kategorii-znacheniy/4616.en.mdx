# What does the inline keyword affect?

## In short

In C++ `inline` is primarily an ODR/linkage rule: identical definitions of a function or variable may live in a header included by multiple TUs without multiply-defined symbols. The inlining hint is secondary and may be ignored.

## How to answer in an interview

- Main effect: allow multiple identical definitions at link time (header-only helpers).
- Inlining is decided by the optimizer; `inline` does not guarantee no call instruction.
- Member functions defined inside a class are implicitly inline.
- `inline` variables (C++17) for header-only constants.
- Honestly: aggressive inlining bloats code and hurts I-cache; I measure with a profiler.

## How it works

The linker merges weak ODR-inline definitions. The compiler may embed the body at the call site when heuristics favor it.

```cpp
inline int combine(int left, int right) {
    return left + right;
}

inline constexpr int scale(int v) {
    return v * v;
}

int demo() {
    return combine(scale(3), scale(4));
}
```

`combine` and `scale` can live in a `.h` included by several `.cpp` files without linker errors. The `combine` call may disappear after inlining, but that is not guaranteed.

## Pitfalls

- **Different bodies across TUs.** The identical requirement breaks: UB or ODR violation.
- **inline on large functions.** Binary growth and cache miss rate increase.
- **inline vs static in .cpp.** `static` functions are TU-local; `inline` exports with merge.
- **Debugging and breakpoints.** Inlined code is harder to step in a debugger.
- **LTO may inline without the keyword.** Modern toolchains often ignore the hint for hot/cold decisions.
