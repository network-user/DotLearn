# Does inline guarantee inlining at the call site?

## In short

No. `inline` is a compiler hint and an ODR tool for definitions in headers. The optimizer decides inlining from function size, `-O` level, and call profile. Recursion and large bodies almost never inline.

## How to answer in an interview

- I answer "not guaranteed"; the compiler may inline without the keyword.
- Second meaning of `inline`: allow multiple definitions across TUs via weak linkage.
- I name factors: size, virtual dispatch, taking the function address, LTO.
- I mention `__attribute__((always_inline))` / `[[gnu::always_inline]]` as stronger but still not absolute.
- I add honestly: sprinkling `inline` in .cpp rarely helps; profile before bloating code.

## How it works

During optimization LLVM/GCC uses a cost model. A tiny `add` may become `a + b` even without `inline`. A function address (`&add`) often blocks inlining in that TU.

```cpp
#include <iostream>

inline int add(int lhs, int rhs) {
    return lhs + rhs;
}

int use_add(int a, int b) {
    return add(a, b);
}

int main() {
    std::cout << use_add(5, 3) << '\n';
}
```

With `-O2` the call often disappears, but the standard does not promise it. Header `inline` lets several .cpp files include the definition without linker errors.

## Pitfalls

- **Code bloat.** Aggressive inline grows the binary and hurts I-cache.
- **Debugging.** Inlined code is harder to step through.
- **ABI and address.** A function pointer to `inline` may keep an out-of-line copy.
- **Recursion.** True inline is impossible.
- **Macro confusion.** `inline` functions are type-safe; macros are not.
