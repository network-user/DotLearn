# How do pointers and references differ?

## In short

A pointer stores an address and is its own variable: it may be `nullptr`, reassigned, and used in pointer arithmetic. A reference is an alias to an existing object: it must be initialized, cannot be empty, and cannot be rebound to another object afterward.

## How to answer in an interview

- I use references for required arguments and returns without copies; pointers when the object may be absent.
- A pointer makes optionality (`nullptr`) and ownership (raw vs smart) explicit.
- A reference needs no dereference syntax; less noise in function APIs.
- For arrays and buffer iteration I need a pointer or iterator, not reference arithmetic.
- Honestly: "a reference is always valid" holds only while the referent lives; a dangling reference is as dangerous as a dangling pointer.

## How it works

```cpp
int payload = 42;
int* addr = &payload;
int& alias = payload;

addr = nullptr;
// int& orphan; // error: reference without initialization

void bump(int& x) { ++x; }
void maybe(int* x) {
    if (x) { ++(*x); }
}

int data[] = {10, 20, 30};
int* walk = data;
++walk;
```

`addr` can be cleared and pointed elsewhere. `alias` always denotes the same `payload` while it exists. `bump` mutates the original without a copy. `maybe` accepts missing values. `walk` advances through the array; references offer no such arithmetic.

## Pitfalls

- **Dangling aliases.** A reference to a local or temporary is invalid after the scope ends.
- **nullptr vs reference.** You cannot pass "no value" through a reference without `std::optional<std::reference_wrapper>` or a pointer.
- **ref/ptr overloads.** Pairs like `void f(int&)` and `void f(int*)` confuse calls with literals.
- **T&& vs T&.** An rvalue reference is a separate mechanism; do not mix it with an lvalue reference in your answer.
- **Size and representation.** A reference is often implemented as a pointer underneath, but language semantics differ; `sizeof` on a reference is the object's size.
