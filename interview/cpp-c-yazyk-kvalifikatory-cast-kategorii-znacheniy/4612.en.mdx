# What ways of type conversion exist in C++?

## In short

Besides the four named casts, C++ still has implicit conversions and C-style `(T)expr`. Explicit casts are for narrow or risky conversions; named operators limit the allowed conversion class and document intent.

## How to answer in an interview

- I list `static_cast`, `dynamic_cast`, `const_cast`, `reinterpret_cast`, and when each fits.
- I add: the compiler performs implicit promotion/conversion; explicit casts opt out of some checks deliberately.
- I avoid C-style casts in modern C++ because they merge all forms.
- For polymorphic downcasts I prefer `dynamic_cast` or virtual dispatch over blind `static_cast`.
- Honestly: `std::bit_cast` (C++20) replaces `reinterpret_cast` for type punning when types are trivially copyable.

## How it works

`static_cast` covers numeric narrowing and hierarchy upcasts. `dynamic_cast` checks RTTI. `const_cast` changes cv only. `reinterpret_cast` reinterprets bits and almost always needs expert review.

```cpp
#include <bit>
#include <cstdint>
#include <iostream>

float to_float(std::uint32_t bits) {
    return std::bit_cast<float>(bits);
}

int main() {
    const double ratio = 3.99;
    int rounded = static_cast<int>(ratio);

    std::uint32_t raw = 0x40490fdb;
    float pi_approx = to_float(raw);

    std::cout << rounded << ' ' << pi_approx << '\n';
    return 0;
}
```

Implicit `double`→`int` in assignment may be allowed, but explicit `static_cast` shows intent to drop the fraction. `bit_cast` copies bits without breaking strict aliasing.

## Pitfalls

- **Mixing styles.** C-style casts hide which cast actually applied.
- **Implicit narrowing.** Without a cast the compiler may warn or fail with `-Werror`.
- **reinterpret vs bit_cast.** The latter is safer for trivially copyable types; the former for arbitrary pointers.
- **dynamic_cast on references.** Failure throws `std::bad_cast`, not `nullptr`.
- **const_cast for legacy APIs.** Stripping const and writing to read-only memory is UB.
