# Example of using std::move

## In short

Typical pattern: pass a heavy object into a function or container as an rvalue via `std::move` so move construction/assignment runs and the buffer is not copied. After the move the source stays valid-but-empty.

## How to answer in an interview

- `process(std::move(vec))` or `container.push_back(std::move(item))`.
- Parameter `T&&` plus move inside accepts rvalues; lvalues need explicit `std::move`.
- Returning a local big object often needs no move: RVO elides copy/move.
- `std::move` does not magically speed up types without move operations.
- Honestly: redundant `std::move` on a local return can block NRVO.

## How it works

A `std::vector` move constructor transfers pointer/size/capacity; the donor is cleared.

```cpp
#include <iostream>
#include <utility>
#include <vector>

void ingest(std::vector<int>&& payload) {
    std::cout << "received size=" << payload.size() << '\n';
}

int main() {
    std::vector<int> buffer{1, 2, 3, 4, 5};
    ingest(std::move(buffer));
    std::cout << "donor size=" << buffer.size() << '\n';
    return 0;
}
```

After the call `buffer.size()` is often 0: the heap array moved into `payload`. Reading `buffer` elements is forbidden; destruction is allowed.

## Pitfalls

- **Move from const.** Copies instead of moves, perf surprise.
- **std::move on return.** Can prevent copy elision in C++17.
- **Moved-from std::string.** Not guaranteed empty, only valid.
- **Concurrent move.** Two threads moving one object is a data race.
- **Small buffer optimization.** `std::string` move may still be a cheap copy under SSO.
