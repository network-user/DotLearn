# How does static affect local variables?

## In short

A `static` local lives until program end (static storage duration), initializes once on first pass, and keeps its value across function calls. Scope stays local: the name is visible only inside the function.

## How to answer in an interview

- Function-local static: persistent state without global name pollution.
- C++11 thread-safe one-time init (magic statics) for function-local statics.
- Use for counters, memoization, function-local lazy singletons.
- Not thread-safe for read-modify-write without a mutex on shared mutable state.
- Honestly: hidden global state complicates tests; inject dependencies when possible.

## How it works

Storage sits in the static segment; visibility remains block-scoped.

```cpp
#include <iostream>

int visits() {
    static int tally = 0;
    ++tally;
    return tally;
}

int main() {
    std::cout << visits() << '\n';
    std::cout << visits() << '\n';
    std::cout << visits() << '\n';
    return 0;
}
```

Prints 1, 2, 3: `tally` is not destroyed between calls. Initialization `= 0` runs once.

## Pitfalls

- **Initialization order.** Static locals init on first use; cross-function order can surprise in complex graphs.
- **Data races.** Concurrent increment without synchronization is UB.
- **Testing reset.** Static state persists between tests unless the process reloads.
- **Destruction.** Static locals destroy at exit in reverse order; IO may already be shut down.
- **Reentrancy.** Recursive first call during init is rare but needs care.
