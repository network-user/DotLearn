# What is std::move for?

## In short

`std::move` does not move bytes itself: it casts an expression to an rvalue reference so move construction or move assignment runs instead of a copy. After a move the source stays valid but unspecified; you usually only assign or destroy it.

## How to answer in an interview

- I explain: cast to rvalue, enable move semantics.
- I show it on `std::vector`, `std::string`, unique resources.
- I stress `std::move` on const objects falls back to copy (move is deleted).
- I mention NRVO/RVO: explicit move on return is not always needed.
- I add honestly: premature `std::move` can block NRVO and slow code down.

## How it works

The move constructor steals a pointer/buffer from the source and clears it. `std::move(x)` lets the compiler pick that overload.

```cpp
#include <iostream>
#include <utility>
#include <vector>

struct Buffer {
    std::vector<int> data;
    explicit Buffer(std::size_t n) : data(n) {}
    Buffer(Buffer&& other) noexcept : data(std::move(other.data)) {}
    Buffer& operator=(Buffer&& other) noexcept {
        data = std::move(other.data);
        return *this;
    }
};

int main() {
    Buffer a(1000);
    Buffer b = std::move(a);
    std::vector<Buffer> stash;
    stash.push_back(std::move(b));
    std::cout << stash[0].data.size() << '\n';
}
```

After the move `a.data` is empty, but `a` can still be destroyed or reassigned.

## Pitfalls

- **Moving const.** `std::move(const T&)` is not a real move.
- **Use after move.** Only assign or destroy is safe.
- **NRVO.** `return std::move(local)` can disable RVO.
- **Small types.** Moving an `int` is not faster than copy.
- **Self move-assignment.** Rare, but guard with `if (this != &other)`.
