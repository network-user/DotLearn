# Зачем в C++ нужен const?

## Суть

`const` фиксирует неизменяемость на уровне компилятора: для переменных, указателей, ссылок и методов. Это снижает риск случайных мутаций, делает намерения явными и открывает оптимизации, потому что компилятор знает, что значение не меняется.

## Что ответить на собесе

- Объявляю `const` там, где данные не должны меняться после инициализации.
- Различаю `const T*`, `T* const` и `const T* const` по правилу чтения справа налево.
- `const`-методы не меняют логическое состояние объекта; перегрузки по const дают разный API.
- Передаю по `const T&`, чтобы не копировать и не дать вызывающему менять аргумент.
- Честно: `const` не защищает от мутаций через неконстантный alias или `const_cast`; это контракт, а не runtime-ограждение.

## Как это работает

Квалификатор можно навесить на объект, указатель, ссылку или метод.

```cpp
const int limit = 100;

const int* ro = &limit;
int value = 5;
int* const pinned = &value;

const int& alias = limit;

class Sensor {
public:
    int read() const { return reading_; }
    void calibrate(int v) { reading_ = v; }
private:
    int reading_ = 0;
};
```

`const int* ro` запрещает менять `*ro`, но сам `ro` можно переназначить. `int* const pinned` фиксирует адрес, значение по нему менять можно. `const`-метод `read()` не может менять не-`mutable` поля. Нарушение даёт ошибку компиляции, а не тихий баг в рантайме.

## Подводные камни

- **Порядок const.** `const int*` и `int const*` эквивалентны; `int* const` другое: const сам указатель.
- **Возврат const по значению.** Для примитивов бессмысленно; для тяжёлых типов иногда скрывает move у временного.
- **mutable.** Позволяет менять поле внутри `const`-метода; злоупотребление ломает «неизменяемость».
- **const_cast.** Снимает const только если исходный объект не был объявлен const; иначе UB.
- **Связка с constexpr.** `constexpr` сильнее: требует compile-time там, где применимо; путать их на собесе не стоит.
