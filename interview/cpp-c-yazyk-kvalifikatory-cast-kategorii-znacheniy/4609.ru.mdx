# Как перегрузить префиксный operator++?

## Суть

Префиксный `++obj` реализуют методом `T& operator++()` без параметров: сначала меняем состояние, затем возвращаем `*this`. Постфиксный вариант отличается фиктивным `int` и возвращает копию до инкремента.

## Что ответить на собесе

- Сигнатура префикса: `Counter& operator++()`; постфикса: `Counter operator++(int)`.
- Префикс дешевле для итераторов: нет лишней копии временного объекта.
- Для итераторов STL обычно реализуют оба, чтобы `*it++` работал как у встроенных типов.
- `const`-correctness: инкремент почти всегда non-const member.
- Честно: если тип не semiregular, постфиксный `++` может быть дорогим или вовсе нежелателен.

## Как это работает

Компилятор различает перегрузки по наличию dummy `int`. Префиксный оператор изменяет `value` и возвращает ссылку на текущий объект, поэтому цепочки вроде `++++c` возможны (хотя в проде их избегают).

```cpp
#include <iostream>

class TickCounter {
    int ticks;

public:
    explicit TickCounter(int start) : ticks(start) {}

    TickCounter& operator++() {
        ++ticks;
        return *this;
    }

    TickCounter operator++(int) {
        TickCounter snapshot(ticks);
        ++ticks;
        return snapshot;
    }

    int read() const { return ticks; }
};

int main() {
    TickCounter counter(5);
    ++counter;
    std::cout << counter.read() << '\n';
    return 0;
}
```

`operator++()` увеличивает `ticks` и отдаёт ссылку; вызов `++counter` эквивалентен встроенному инкременту по семантике ссылки.

## Подводные камни

- **Путаница префикс/постфикс.** Один лишний `int` в сигнатуре меняет возвращаемый тип и стоимость.
- **Возврат по значению в префиксе.** `Counter operator++()` вместо `Counter&` ломает ожидания и копирует лишний раз.
- **Не изменять состояние в постфиксе дважды.** Сначала snapshot, потом increment.
- **Iterator requirements.** Для InputIterator достаточно префикса; RandomAccess часто требует оба.
- **Proxy iterators.** Некоторые контейнеры возвращают proxy; постфикс может быть удалён или ограничен.
