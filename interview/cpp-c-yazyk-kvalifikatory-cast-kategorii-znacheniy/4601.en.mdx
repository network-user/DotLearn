# How do static_cast and dynamic_cast differ?

## In short

`static_cast` performs compiler-checked conversions without a runtime type check: numerics, explicit pointer hierarchies, `void*`. `dynamic_cast` applies to polymorphic types (with virtuals), checks the dynamic type at runtime; for pointers it yields `nullptr` on failure, for references it throws `std::bad_cast`.

## How to answer in an interview

- `static_cast`: compile time, faster, no RTTI; downcast is safe only if you know the dynamic type.
- `dynamic_cast`: runtime RTTI, needs a virtual in the base, safe downcast/cross-cast.
- I show the idiom: `dynamic_cast<Derived*>(base)` plus a `nullptr` check.
- I note `dynamic_cast` on non-polymorphic types fails to compile.
- I add honestly: frequent downcasts hint at weak design; prefer virtual dispatch or `std::variant`.

## How it works

The compiler embeds RTTI for polymorphic classes. `dynamic_cast` consults the type table. `static_cast` only reinterprets the type at compile time.

```cpp
#include <iostream>

struct Base {
    virtual ~Base() = default;
    virtual void ping() const { std::cout << "base\n"; }
};

struct Worker : Base {
    void ping() const override { std::cout << "worker\n"; }
    void task_id() const { std::cout << "job\n"; }
};

int main() {
    Worker concrete;
    Base* base = &concrete;

    auto* fast = static_cast<Worker*>(base);
    fast->task_id();

    Base* errored = new Base();
    auto* safe = dynamic_cast<Worker*>(errored);
    if (!safe) std::cout << "not a worker\n";
    delete errored;
}
```

`static_cast` on `errored` would skip runtime checks and UB on `task_id`.

## Pitfalls

- **Unchecked downcast.** Wrong `static_cast` is UB.
- **References.** `dynamic_cast<Worker&>(base)` throws, no optional form.
- **Multiple inheritance.** `dynamic_cast` may adjust the pointer address.
- **RTTI disabled.** `-fno-rtti` breaks `dynamic_cast`.
- **Cast chains.** Prefer a polymorphic interface over repeated casts.
