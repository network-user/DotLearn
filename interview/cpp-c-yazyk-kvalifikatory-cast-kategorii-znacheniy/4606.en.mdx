# What is static used for in C++?

## In short

`static` means different things by context: a local static variable lives for the whole program and keeps state between calls; namespace static limits linkage to the file; a static class member is one per type, not per object. It controls lifetime and visibility.

## How to answer in an interview

- I list three contexts: local, namespace/file, class member.
- Local static: lazy init (C++11 thread-safe for function-local).
- File static: internal linkage, hide the symbol from other TUs.
- Class static: shared state, factory registries, counters.
- I add honestly: global static objects risk initialization order bugs; function-local static is safer for singletons.

## How it works

Function-local static initializes on first pass (magic statics). Namespace static in a .cpp is not exported by the linker. Class static needs an out-of-class definition when odr-used.

```cpp
#include <iostream>

namespace {
static int file_counter = 0;

static void bump_file() { ++file_counter; }
}

class Session {
    static int active;

public:
    static int count() { return active; }
    Session() { ++active; }
    ~Session() { --active; }
};

int Session::active = 0;

int& call_counter() {
    static int hits = 0;
    return hits;
}

int main() {
    bump_file();
    Session a, b;
    ++call_counter();
    std::cout << Session::count() << ' ' << call_counter() << '\n';
}
```

`call_counter` returns a reference to a single int that lives for the whole runtime.

## Pitfalls

- **Static initialization order fiasco.** Order across TUs is undefined.
- **Non-atomic static++.** Member static increments need atomic or mutex.
- **Missing definition.** `static int Session::active` must live in a .cpp.
- **Hidden global lifetime.** Local static is still a global object by lifetime.
- **static vs constexpr.** Different goals; static does not mean compile time.
