# Зачем ключевое слово inline?

## Суть

`inline` решает две задачи: (1) подсказка оптимизатору встроить тело функции; (2) разрешить определение функции в заголовке без нарушения ODR при включении в несколько translation units. Второй смысл сегодня важнее первого.

## Что ответить на собесе

- Разделяю hint-to-compiler и linkage/ODR роль.
- Объясняю, почему шаблоны и методы класса в .h по сути inline.
- Называю когда уместно: tiny accessors, header-only utilities.
- Предупреждаю про code bloat при больших inline функциях.
- Честно добавляю: в .cpp файле `inline` почти не влияет на perf; LTO решает иначе.

## Как это работает

Без `inline` non-template функция в header даёт multiply defined symbol на линковке. `inline` даёт weak linkage: linker схлопывает копии. Оптимизатор отдельно решает, раскрывать ли тело в call site.

```cpp
// metrics.hpp
#pragma once
#include <iostream>

inline int square(int value) {
    return value * value;
}

inline void report(int value) {
    std::cout << "sq=" << square(value) << '\n';
}
```

Несколько .cpp могут включать `metrics.hpp` без ошибки линковки.

## Подводные камни

- **Ожидание speedup.** Ключевое слово alone не ускоряет.
- **Большие функции в header.** Увеличивают compile time и размер TU.
- **Не путать с static.** `static` в .cpp даёт internal linkage, другая семантика.
- **Виртуальные методы.** Virtual dispatch блокирует обычный inline на вызове через base*.
- **ODR violation.** Inline definition в header должен быть идентичен во всех TU.
