# Зачем в C++ нужен static?

## Суть

`static` меняет смысл по контексту: локальная static переменная живёт до конца программы и сохраняет значение между вызовами; static в namespace ограничивает linkage файлом; static член класса один на весь тип, не на объект. Это управление временем жизни и видимостью.

## Что ответить на собесе

- Перечисляю три контекста: local, namespace/file, class member.
- Local static: lazy init (C++11 thread-safe для function-local).
- File static: internal linkage, скрыть символ от других TU.
- Class static: shared state, factory registries, counters.
- Честно добавляю: глобальные static объекты опасны порядком инициализации; предпочитаю function-local static для singleton.

## Как это работает

Function-local static инициализируется при первом проходе (magic statics). Namespace static в .cpp не экспортируется линкером наружу. Class static требует определения вне класса для odr-used объектов.

```cpp
#include <iostream>

namespace {
static int file_counter = 0;

static void bump_file() { ++file_counter; }
}

class Session {
    static int active;

public:
    static int count() { return active; }
    Session() { ++active; }
    ~Session() { --active; }
};

int Session::active = 0;

int& call_counter() {
    static int hits = 0;
    return hits;
}

int main() {
    bump_file();
    Session a, b;
    ++call_counter();
    std::cout << Session::count() << ' ' << call_counter() << '\n';
}
```

`call_counter` возвращает ссылку на единственный int, живущий весь runtime.

## Подводные камни

- **Static initialization order fiasco.** Между TU порядок не определён.
- **Не thread-safe member static++.** Нужен atomic или mutex.
- **Забытое определение.** `static int Session::active` обязательно в .cpp.
- **Скрытая глобальность.** Local static всё равно глобальный объект по lifetime.
- **static vs constexpr.** Разные цели; static не означает compile-time.
