# Как снизить риск взаимных блокировок (deadlock) в 1С

## Суть

Deadlock возникает при циклическом ожидании блокировок. Снижают риск короткие транзакции, единый порядок захвата регистров, узкие отборы в `БлокировкаДанных`, пакетирование массовых операций и оптимистичная модель там, где нет жёсткой борьбы за остаток.

## Что ответить на собесе

- Я называю «одинаковый порядок блокировок» главным архитектурным правилом для всех процедур проведения.
- На собеседовании привожу пример: сначала товары, потом взаиморасчёты во всех обработках.
- Подчёркиваю сортировку документов по ключам перед массовым проведением.
- Честно говорю: retry с backoff помогает, но не заменяет выравнивание порядка.
- Убираю лишние регистры из движений и сокращаю время транзакции.

## Как это работает

Два регламента блокируют регистры в разном порядке. Сеанс 1: товары → расчёты. Сеанс 2: расчёты → товары. Цикл замкнулся.

```bsl
Процедура ЗаблокироватьРегистрыПроведения(ДокументОбъект) Экспорт
    Блокировка = Новый БлокировкаДанных;
    ЭлементТовары = Блокировка.Добавить("РегистрНакопления.ТоварыНаСкладах");
    ЭлементТовары.Режим = РежимБлокировкиДанных.Исключительный;
    ЭлементТовары.УстановитьЗначение("Склад", ДокументОбъект.Склад);
    ЭлементРасчеты = Блокировка.Добавить("РегистрНакопления.Взаиморасчеты");
    ЭлементРасчеты.Режим = РежимБлокировкиДанных.Исключительный;
    ЭлементРасчеты.УстановитьЗначение("Контрагент", ДокументОбъект.Контрагент);
    Блокировка.Заблокировать();
КонецПроцедуры
```

Запрос с `ДЛЯ ОБНОВЛЕНИЯ` блокирует только выбранные строки на время UPDATE:

```bsl
Запрос.Текст =
"ВЫБРАТЬ
|    Номенклатура.Ссылка
|ИЗ
|    Справочник.Номенклатура КАК Номенклатура
|ГДЕ
|    Номенклатура.Ссылка = &Ссылка
|ДЛЯ ОБНОВЛЕНИЯ";
```

Тяжёлые расчёты выносят до `НачатьТранзакцию()`. Массовые job делят на пакеты по 20–50 документов. Для справочников с редкими коллизиями оставляют оптимистичную запись без предварительной блокировки.

## Подводные камни

- **Разный порядок в типовой и доработанной процедуре.** Deadlock только на кастомном коде.
- **Широкая блокировка «на всякий случай».** Увеличивает пересечения.
- **Параллельные регламенты без расписания.** Два job в одно окно.
- **Retry в бесконечном цикле.** Нагрузка на lock manager.
- **Игнорирование итогов регистра.** Deadlock на агрегатах, не на движениях.
