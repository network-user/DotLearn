# Что такое висящие транзакции и как их находить

## Суть

«Висящая» транзакция остаётся открытой после ошибки, обрыва сеанса или пропущенного `ЗафиксироватьТранзакцию()` / `ОтменитьТранзакцию()`. Она удерживает блокировки и мешает записи других пользователей, пока платформа или СУБД не откатят её по таймауту.

## Что ответить на собесе

- Я называю висящей транзакцию без завершения: ни commit, ни rollback на момент проверки.
- На собеседовании упоминаю монитор блокировок и транзакций в администрировании кластера.
- Подчёркиваю типичную причину: `НачатьТранзакцию()` без `Попытка`/`Исключение` и без отката в каждой ветке.
- Честно говорю: убить сеанс снимает симптом, но без правки кода проблема вернётся.
- Для PostgreSQL и MS SQL дополнительно смотрю представления активных транзакций на стороне СУБД.

## Как это работает

Код открывает транзакцию, записывает документ, затем падает до фиксации. Если исключение не перехвачено или откат пропущен, блок остаётся активным до конца серверного вызова или дольше при ошибке платформы.

```bsl
НачатьТранзакцию();
Попытка
    ДокОбъект = Документы.Реализация.СоздатьДокумент();
    ДокОбъект.Дата = ТекущаяДата();
    ДокОбъект.Записать(РежимЗаписиДокумента.Запись);
    ВыполнитьНекорректныйРасчет();
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ЗаписьЖурналаРегистрации("Проведение", УровеньЖурналаРегистрации.Ошибка,,,
        ПодробноеПредставлениеОшибки(ИнформацияОбОшибке()));
    ВызватьИсключение;
КонецПопытки;
```

Поиск начинают с монитора блокировок: номер сеанса, объект, время удержания. В журнале регистрации и технологическом журнале ищут стек вызова и последний `НачатьТранзакцию()` без парного завершения.

На уровне СУБД для MS SQL смотрят `sys.dm_tran_active_transactions` и блокирующие SPID; для PostgreSQL `pg_stat_activity` с `state = 'idle in transaction'`. Сопоставление SPID сеанса 1С с PID СУБД помогает найти виновный rphost.

Профилактика: шаблон `НачатьТранзакцию()` + `Попытка` + обязательный откат в `Исключение`, короткие транзакции, запрет интерактива внутри блока.

## Подводные камни

- **Отсутствие отката в одной из веток.** Ранний `Возврат` мимо `Исключение` оставляет транзакцию.
- **Повторное НачатьТранзакцию без проверки.** Вложенность не даёт независимой фиксации.
- **Убийство сеанса как единственное лечение.** Данные откатываются, но причина в коде остаётся.
- **Путаница с управляемыми блокировками.** SQL-транзакция закрыта, а прикладная блокировка ещё жива.
- **Ночные регламенты без таймаута.** «Idle in transaction» копится до утреннего пика нагрузки.
