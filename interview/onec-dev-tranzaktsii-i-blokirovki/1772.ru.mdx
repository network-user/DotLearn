# Почему длинные транзакции опасны для 1С

## Суть

Открытая транзакция удерживает блокировки и журнал изменений до фиксации или отката. Чем дольше она живёт, тем выше риск взаимных блокировок, роста tempdb и «зависания» пользователей на записи и проведении.

## Что ответить на собесе

- Я связываю длительность транзакции с удержанием управляемых и SQL-блокировок на затронутых объектах.
- На собеседовании привожу пример: цикл записи внутри одного `НачатьТранзакцию()` без промежуточных фиксаций.
- Подчёркиваю, что пользовательский диалог и HTTP-вызов внутри транзакции недопустимы.
- Честно говорю: разбиение на короткие транзакции снижает изоляцию пакета, зато убирает очереди на регистрах.
- Для диагностики смотрю монитор блокировок, длительность сеанса и активные транзакции в СУБД.

## Как это работает

Типичный антипаттерн: открыли транзакцию, обновили сотни строк, затем выполнили тяжёлый расчёт или ожидание пользователя. Пока транзакция не закрыта, другие сеансы ждут те же ключи регистра или документа.

```bsl
Процедура МассовоеПерераспределение(ТаблицаСтрок) Экспорт
    НачатьТранзакцию();
    Попытка
        Для Каждого Строка Из ТаблицаСтрок Цикл
            ДокОбъект = Строка.Ссылка.ПолучитьОбъект();
            ДокОбъект.Склад = Строка.НовыйСклад;
            ДокОбъект.Записать(РежимЗаписиДокумента.Проведение);
        КонецЦикла;
        ВыполнитьДолгийПересчетСтатистики();
        ЗафиксироватьТранзакцию();
    Исключение
        ОтменитьТранзакцию();
        ВызватьИсключение;
    КонецПопытки;
КонецПроцедуры
```

Правильнее вынести расчёт за пределы транзакции и фиксировать пакетами:

```bsl
РазмерПакета = 50;
Счетчик = 0;
НачатьТранзакцию();
Для Каждого Строка Из ТаблицаСтрок Цикл
    ОбновитьСкладДокумента(Строка);
    Счетчик = Счетчик + 1;
    Если Счетчик >= РазмерПакета Тогда
        ЗафиксироватьТранзакцию();
        НачатьТранзакцию();
        Счетчик = 0;
    КонецЕсли;
КонецЦикла;
Если ТранзакцияАктивна() Тогда
    ЗафиксироватьТранзакцию();
КонецЕсли;
ВыполнитьДолгийПересчетСтатистики();
```

Длинная транзакция раздувает версионный store в MS SQL и удерживает autovacuum-блокировки в PostgreSQL. На регистрах накопления это проявляется как ожидание при проведении и пересчёте итогов.

## Подводные камни

- **Диалог на форме внутри транзакции.** Пользователь думает минутами, блокировки не отпускаются.
- **Внешний HTTP в транзакции.** Таймаут сети держит открытый блок на регистре.
- **Один пакет на весь регламент.** Ночная обработка блокирует дневную работу.
- **Иллюзия «быстрого отката».** Откат длинной транзакции сам по себе дорогой для СУБД.
- **Смешение с фоновыми заданиями.** Параллельные регламенты на одних таблицах усиливают deadlock.
