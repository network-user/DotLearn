# Какие виды блокировок предусмотрены платформой 1С

## Суть

Платформа сочетает управляемые блокировки прикладного уровня, автоматические блокировки при записи объектов и транзакционные блокировки СУБД. Разработчик явно работает с `БлокировкаДанных`, остальное платформа накладывает при `Записать()`, проведении и SQL-запросах с блокировкой.

## Что ответить на собесе

- Я перечисляю три слоя: управляемые, pessimistic при записи, SQL row/page/table.
- На собеседовании называю режимы `Разделяемый` и `Исключительный` в `БлокировкаДанных`.
- Подчёркиваю оптимистичную проверку версии при записи как отдельный механизм от блокировки.
- Честно говорю: монопольный режим конфигурации и сеанса это особый случай эксклюзивного доступа.
- Для диагностики смотрю монитор 1С и блокировки в СУБД одновременно.

## Как это работает

Управляемая блокировка задаётся явно перед изменением регистра:

```bsl
Блокировка = Новый БлокировкаДанных;
Элемент = Блокировка.Добавить("Справочник.Номенклатура");
Элемент.Режим = РежимБлокировкиДанных.Разделяемый;
Элемент.УстановитьЗначение("Ссылка", СсылкаНоменклатуры);
Блокировка.Заблокировать();
```

Автоматическая блокировка включается при записи объекта в транзакции: платформа блокирует строку `_Reference*` до commit. При проведении добавляются блокировки движений регистров.

Транзакционный слой СУБД использует shared/exclusive locks на строках и страницах. Уровень изоляции определяет, видит ли транзакция незафиксированные изменения других.

```bsl
НачатьТранзакцию();
Попытка
    Объект = СсылкаКонтрагента.ПолучитьОбъект();
    Объект.Наименование = НовоеИмя;
    Объект.Записать();
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ВызватьИсключение;
КонецПопытки;
```

Монопольный доступ (обновление конфигурации, тестирование) блокирует запись для всех сеансов. Это не замена управляемым блокировкам в прикладном коде.

## Подводные камни

- **Смешение терминов «жёсткая» и «исключительная».** В документации 1С точнее говорить «исключительная управляемая».
- **Ожидание блокировки при оптимистичной записи.** Конфликт версий не ждёт, а отклоняет.
- **Игнорирование SQL-слоя.** Управляемая блокировка снята, а строка ещё в lock СУБД.
- **Разделяемая там, где идёт запись.** Два сеанса одновременно пишут.
- **Монопольный режим на проде.** Плановое окно забыли закрыть.
