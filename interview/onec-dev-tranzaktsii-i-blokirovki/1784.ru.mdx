# Типы блокировок на уровне СУБД в контексте 1С

## Суть

Под прикладным кодом 1С СУБД использует блокировки строк, страниц, таблиц и метаданных с режимами shared и exclusive. Платформа транслирует запись объектов и движений регистров в INSERT/UPDATE/DELETE, а SQL Server и PostgreSQL решают гранularity и эскалацию самостоятельно.

## Что ответить на собесе

- Я называю row lock как основной при точечной записи и page/table lock при массовых операциях.
- На собеседовании упоминаю `UPDLOCK`, `ROWLOCK`, `TABLOCK` для MS SQL и `FOR UPDATE` для PostgreSQL.
- Подчёркиваю уровни изоляции: read committed у PostgreSQL по умолчанию, snapshot/read committed в SQL Server.
- Честно говорю: монопольный режим 1С это прикладной аналог эксклюзива, не отменяющий SQL-механику.
- Для расследования смотрю DMVs или `pg_locks`, не только монитор 1С.

## Как это работает

Запись одного элемента справочника превращается в UPDATE строки `_Reference12345`:

```sql
UPDATE dbo._Reference12345
SET _Description = @Name
WHERE _IDRRef = @Ref;
```

СУБД ставит exclusive lock на строку. Параллельное чтение с shared lock возможно до момента commit в зависимости от изоляции.

Массовое проведение генерирует пакет INSERT в `_AccumReg*`. При росте числа row locks SQL Server может эскалировать до page или table lock, блокируя конкурентное проведение шире, чем задумал разработчик.

```sql
BEGIN TRANSACTION;
INSERT INTO dbo._AccumReg789 (_Period, _Fld1RRef, _Fld2RRef, _Fld3)
VALUES (@Period, @Item, @Warehouse, @Qty);
COMMIT TRANSACTION;
```

В PostgreSQL `SELECT ... FOR UPDATE` соответствует явному захвату строк перед обновлением. Idle in transaction держит lock даже без активного запроса.

Монопольный режим конфигурации запрещает запись пользователям, но серверные процессы обновления структуры всё равно используют table-level locks на служебных объектах.

## Подводные камни

- **Эскалация при ночном регламенте.** Дневные документы ждут table lock.
- **Длинный SELECT в транзакции.** Shared lock или snapshot блокирует vacuum/rollback.
- **NOLOCK как «лечение».** Грязное чтение ломает учёт.
- **Сравнение только по монитору 1С.** Реальная очередь видна в СУБД.
- **Разные настройки изоляции на разных кластерах.** Один и тот же код ведёт себя по-разному.
