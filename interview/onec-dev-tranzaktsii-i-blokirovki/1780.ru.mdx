# Что делать, если проведение падает из-за взаимной блокировки

## Суть

Deadlock возникает, когда два сеанса блокируют ресурсы в разном порядке. СУБД откатывает одну транзакцию; пользователь видит ошибку проведения. Задача разработчика: зафиксировать пару сеансов, выровнять порядок блокировок и сократить время транзакции.

## Что ответить на собесе

- Я начинаю с монитора блокировок и журнала регистрации в момент ошибки.
- На собеседовании называю типовые меры: единый порядок блокировки регистров, короткие транзакции, повтор с backoff.
- Подчёркиваю `Блокировка.Заблокировать()` с осмысленным отбором вместо блокировки «всего регистра».
- Честно говорю: повтор проведения после deadlock часто успешен, но без правки алгоритма deadlock вернётся.
- Разделяю взаимную блокировку SQL и конфликт управляемых блокировок 1С.

## Как это работает

Сеанс A блокирует регистр товаров, затем взаиморасчётов. Сеанс B делает наоборот. Оба ждут друг друга, СУБД выбирает жертву и откатывает её транзакцию.

```bsl
Процедура ПровестиСБлокировками(ДокументОбъект) Экспорт
    Блокировка = Новый БлокировкаДанных;
    ЭлементТовары = Блокировка.Добавить("РегистрНакопления.ТоварыНаСкладах");
    ЭлементТовары.Режим = РежимБлокировкиДанных.Исключительный;
    ЭлементТовары.УстановитьЗначение("Склад", ДокументОбъект.Склад);
    ЭлементРасчеты = Блокировка.Добавить("РегистрНакопления.Взаиморасчеты");
    ЭлементРасчеты.Режим = РежимБлокировкиДанных.Исключительный;
    ЭлементРасчеты.УстановитьЗначение("Контрагент", ДокументОбъект.Контрагент);
    Блокировка.Заблокировать();
    ДокументОбъект.Движения.Записать();
КонецПроцедуры
```

Порядок элементов в `БлокировкаДанных` должен быть одинаковым во всех процедурах проведения. Массовые обработки сортируют документы по ключам блокировки.

При ошибке deadlock клиент может повторить операцию с паузой:

```bsl
МаксПопыток = 3;
Для НомерПопытки = 1 По МаксПопыток Цикл
    Попытка
        ДокументОбъект.Записать(РежимЗаписиДокумента.Проведение);
        Прервать;
    Исключение
        Если Не ЭтоDeadlock(ИнформацияОбОшибке()) Или НомерПопытки = МаксПопыток Тогда
            ВызватьИсключение;
        КонецЕсли;
        Пауза(НомерПопытки * 200);
    КонецПопытки;
КонецЦикла;
```

Долгосрочно переносят массовое проведение в фон, убирают лишние регистры из движений, проверяют параллельные регламенты на одних таблицах.

## Подводные камни

- **Повтор без изменения порядка блокировок.** Deadlock воспроизводится стабильно под нагрузкой.
- **Расширение отборов «на всякий случай».** Широкие блокировки увеличивают пересечения.
- **Два регламента на одни регистры.** Конкуренция не видна интерактивным пользователям.
- **Лечение только увеличением таймаута.** Система дольше «висит» перед откатом.
- **Путаница с нехваткой остатка.** Это бизнес-ошибка, не deadlock.
