# Чем управляемые блокировки 1С отличаются от блокировок СУБД

## Суть

1С накладывает прикладные управляемые блокировки по пространствам имён и полям метаданных, а СУБД блокирует строки и страницы по физическим таблицам. Оба слоя работают одновременно: платформа решает бизнес-конфликты, SQL обеспечивает изоляцию транзакций.

## Что ответить на собесе

- Я разделяю управляемые блокировки (`БлокировкаДанных`) и неявные блокировки записи при `Записать()` и проведении.
- На собеседовании говорю, что блокировка 1С понимает регистр, измерения и документ, а SQL видит `_AccumReg*` и `_Document*`.
- Подчёркиваю режимы: разделяемая, исключительная, а также автоматические блокировки при записи.
- Честно уточняю: отключить SQL-блокировки нельзя, но можно сузить прикладной набор через точный отбор в `Заблокировать()`.
- Для диагностики смотрю монитор 1С и DMVs/`pg_locks` параллельно.

## Как это работает

Перед изменением остатков разработчик явно блокирует срез регистра. Платформа регистрирует блокировку в своей таблице и при необходимости транслирует её в SQL.

```bsl
Блокировка = Новый БлокировкаДанных;
ЭлементБлокировки = Блокировка.Добавить("РегистрНакопления.ТоварыНаСкладах");
ЭлементБлокировки.Режим = РежимБлокировкиДанных.Исключительный;
ЭлементБлокировки.УстановитьЗначение("Склад", СкладОтгрузки);
ЭлементБлокировки.УстановитьЗначение("Номенклатура", Номенклатура);
Блокировка.Заблокировать();
```

На стороне MS SQL та же операция может выглядеть как блокировка строк по ключу таблицы движений:

```sql
BEGIN TRANSACTION;
SELECT _Period, _Fld123RRef
FROM dbo._AccumReg456
WITH (UPDLOCK, ROWLOCK)
WHERE _Fld100RRef = @WarehouseRef;
COMMIT TRANSACTION;
```

PostgreSQL использует `FOR UPDATE` на выбранных строках. Уровень изоляции (`READ COMMITTED`, `REPEATABLE READ`) определяет, когда SQL блокирует чтение и запись. 1С поверх этого добавляет семантику «заблокировать остаток по складу и номенклатуре», даже если физически затронуто несколько таблиц.

При проведении документа платформа сама ставит набор блокировок на регистратор и затронутые измерения. Ручная блокировка нужна, когда два независимых алгоритма меняют один срез без общего документа.

## Подводные камни

- **Слишком широкий отбор в Заблокировать().** Блокируется весь регистр вместо одного склада.
- **Игнорирование прикладного слоя.** Смотрят только SQL и не видят управляемую блокировку 1С.
- **Разделяемая там, где нужна исключительная.** Два сеанса одновременно списывают один остаток.
- **Блокировка без транзакции.** Снятие зависит от контекста вызова и может прийти не вовремя.
- **Файловый режим.** Механизмы отличаются от клиент-серверной СУБД, перенос опыта один в один не работает.
