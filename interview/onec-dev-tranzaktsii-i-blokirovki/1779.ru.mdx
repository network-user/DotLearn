# Как пакетная обработка проведённых документов влияет на блокировки

## Суть

Пакетное проведение объединяет несколько документов в один или несколько коротких транзакционных блоков. Это снижает накладные расходы на commit, но удлиняет удержание блокировок на регистрах, если весь массив идёт в одной транзакции.

## Что ответить на собесе

- Я разделяю «пакет по количеству документов» и «одна транзакция на весь массив».
- На собеседовании говорю, что оптимальный размер пакета подбирают по времени блокировки и числу deadlock.
- Подчёркиваю сортировку документов по складу и дате для единообразного порядка блокировок.
- Честно уточняю: одна большая транзакция быстрее по commit, но хуже для интерактивных пользователей.
- Использую `РежимПроведения` и фоновые задания для массовых операций вне пиковой нагрузки.

## Как это работает

Регламент перепроводит список документов. Вариант с одной транзакцией держит блокировки на всех затронутых регистрах до конца цикла.

```bsl
Процедура ПерепровестиПакет(СсылкиДокументов, РазмерПакета = 20) Экспорт
    Индекс = 0;
    Пока Индекс < СсылкиДокументов.Количество() Цикл
        НачатьТранзакцию();
        Попытка
            ВерхняяГраница = Мин(Индекс + РазмерПакета, СсылкиДокументов.Количество()) - 1;
            Для Ном = Индекс По ВерхняяГраница Цикл
                ДокОбъект = СсылкиДокументов[Ном].ПолучитьОбъект();
                ДокОбъект.Записать(РежимЗаписиДокумента.Проведение);
            КонецЦикла;
            ЗафиксироватьТранзакцию();
        Исключение
            ОтменитьТранзакцию();
            ВызватьИсключение;
        КонецПопытки;
        Индекс = ВерхняяГраница + 1;
    КонецЦикла;
КонецПроцедуры
```

Перед циклом документы сортируют по складу и дате, чтобы все сеансы блокировали регистры в одном порядке. Между пакетами блокировки снимаются, интерактивное проведение получает окно.

Платформа при пакетной записи переиспользует соединение с СУБД и кэш метаданных. Пересчёт итогов может откладываться до конца пакета, если включён соответствующий режим в регламентных обработках типовых конфигураций.

## Подводные камни

- **Весь массив в одной транзакции.** Один сбой откатывает сотни документов.
- **Случайный порядок документов.** Рост взаимных блокировок между пакетами и интерактивом.
- **Пакет в рабочие часы.** Дневные пользователи ждут регистры.
- **Игнорирование итогов.** Пакет ускоряет проведение, но пересчёт итогов потом блокирует снова.
- **Слишком мелкий пакет.** Commit-доминирует, выигрыша нет.
