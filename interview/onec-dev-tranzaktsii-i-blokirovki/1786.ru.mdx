# Почему массовое проведение падает с ошибкой блокировки

## Суть

Массовая обработка концентрирует проведение на одних регистрах и удерживает блокировки дольше интерактивных операций. Ошибка блокировки появляется из-за конкуренции с пользователями, параллельных регламентов, одной длинной транзакции на весь массив или разного порядка захвата ресурсов.

## Что ответить на собесе

- Я связываю массовое проведение с пересечением по складам, контрагентам и регистрам накопления.
- На собеседовании называю типовые причины: один пакет в одной транзакции, ночной регламент + дневные пользователи, deadlock.
- Подчёркиваю инструменты: монитор блокировок, ТЖ, сортировка документов, разбиение на пакеты.
- Честно говорю: увеличить таймаут без смены алгоритма только откладывает ошибку.
- Рекомендую фоновые задания в окне низкой нагрузки и единый порядок блокировки регистров.

## Как это работает

Обработка перепроводит тысячи реализаций в одном сеансе. Если все документы в одной транзакции, блокировки на `_AccumReg*` живут до конца цикла.

```bsl
Процедура ПерепровестиСписок(МассивСсылок) Экспорт
    МассивСсылок = СортироватьПоСкладуИДате(МассивСсылок);
    РазмерПакета = 25;
    Индекс = 0;
    Пока Индекс < МассивСсылок.Количество() Цикл
        НачатьТранзакцию();
        Попытка
            Лимит = Мин(Индекс + РазмерПакета, МассивСсылок.Количество()) - 1;
            Для Поз = Индекс По Лимит Цикл
                ДокОбъект = МассивСсылок[Поз].ПолучитьОбъект();
                ДокОбъект.Записать(РежимЗаписиДокумента.Проведение);
            КонецЦикла;
            ЗафиксироватьТранзакцию();
        Исключение
            ОтменитьТранзакцию();
            ЗаписьЖурналаРегистрации("МассовоеПроведение", УровеньЖурналаРегистрации.Ошибка,,
                "Сбой на пакете с индекса " + Индекс);
            ВызватьИсключение;
        КонецПопытки;
        Индекс = Лимит + 1;
    КонецЦикла;
КонецПроцедуры
```

Параллельный регламент по тем же регистрам создаёт deadlock: сеанс A держит товары, ждёт взаиморасчёты; сеанс B наоборот. Сортировка документов и одинаковый порядок `БлокировкаДанных` снижают риск.

Перепроведение задним числом тянет пересчёт итогов и расширяет блокировки по периодам. Интерактивный пользователь на том же складе получает timeout.

## Подводные камни

- **Два массовых регламента без координации.** Конкуренция не видна в UI пользователя.
- **Пакет 500+ в одной транзакции.** Commit быстрый, блокировки длинные.
- **Игнорирование итогов регистра.** Ошибка приходит на этапе агрегатов, не движений.
- **Проведение в рабочий день «для скорости».** Лечение хуже болезни.
- **Повтор без паузы после deadlock.** Нагрузка на lock manager растёт.
