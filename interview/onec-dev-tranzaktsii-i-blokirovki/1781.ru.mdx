# Когда применять разделение транзакций в 1С

## Суть

Разделение транзакций (несколько коротких commit вместо одного длинного) нужно при массовых операциях, когда атомарность всего массива не обязательна или её можно восстановить регламентом. Цель: сократить время удержания блокировок и дать другим сеансам работать между пакетами.

## Что ответить на собесе

- Я использую разделение, когда бизнес допускает частичный успех или есть компенсирующий регламент.
- На собеседовании противопоставляю «один документ = одна транзакция» и «регламент на тысячи строк».
- Подчёркиваю, что платформенное проведение документа само по себе атомарно; делить имеет смысл пакеты регламентов.
- Честно говорю: при жёсткой сквозной атомарности разделение нельзя применять без саги или очереди.
- Фиксирую прогресс в регистре сведений или журнале, чтобы продолжить после сбоя.

## Как это работает

Регламент загрузки цен обновляет десятки тысяч записей. Одна транзакция заблокирует регистр на минуты.

```bsl
Процедура ОбновитьЦеныПакетами(ТаблицаЦен, РазмерПакета) Экспорт
    Смещение = 0;
    Пока Смещение < ТаблицаЦен.Количество() Цикл
        НачатьТранзакцию();
        Попытка
            Лимит = Мин(Смещение + РазмерПакета, ТаблицаЦен.Количество());
            Для Инд = Смещение По Лимит - 1 Цикл
                Строка = ТаблицаЦен[Инд];
                Менеджер = РегистрыСведений.ЦеныНоменклатуры.СоздатьМенеджерЗаписи();
                Менеджер.Период = Строка.Период;
                Менеджер.Номенклатура = Строка.Номенклатура;
                Менеджер.Цена = Строка.Цена;
                Менеджер.Записать();
            КонецЦикла;
            ЗафиксироватьТранзакцию();
            ЗафиксироватьПрогресс(Лимит);
        Исключение
            ОтменитьТранзакцию();
            ВызватьИсключение;
        КонецПопытки;
        Смещение = Лимит;
    КонецЦикла;
КонецПроцедуры
```

Разделение уместно для: массовой пометки на удаление, переноса между складами пакетами, синхронизации с внешней системой по порциям. Не уместно, когда документ обязан провести все движения целиком или операция финансовая и неделима.

Между пакетами другие пользователи видят промежуточное состояние. Если это недопустимо, используют очередь заданий или ночное окно.

## Подводные камни

- **Разделение проведения одного документа.** Получаем документ без полного набора движений.
- **Отсутствие журнала прогресса.** После сбоя регламент начинает сначала.
- **Слишком крупный пакет.** Блокировки снова длинные.
- **Слишком мелкий пакет.** Commit съедает выигрыш.
- **Промежуточные данные в отчётах.** Пользователи видят «половину» загрузки.
