# Как распознать и диагностировать конфликт блокировок

## Суть

Конфликт блокировок возникает, когда сеанс ждёт ресурс, удерживаемый другим сеансом, или попадает в deadlock. Симптомы: таймаут ожидания, ошибка проведения, «объект заблокирован». Диагностика идёт через монитор блокировок, технологический журнал и представления СУБД.

## Что ответить на собесе

- Я различаю ожидание блокировки, взаимную блокировку и конфликт версий при записи.
- На собеседовании называю монитор блокировок в консоли кластера и событие `EXCP` в ТЖ с текстом о блокировке.
- Подчёркиваю, что нужно видеть пару «кто блокирует / кто ждёт» и объект (регистр, документ, сеанс).
- Честно говорю: убить блокирующий сеанс снимает симптом, но без анализа стека проблема повторится.
- Для MS SQL смотрю `blocked process report`, для PostgreSQL `pg_blocking_pids`.

## Как это работает

Пользователь проводит реализацию, второй сеанс одновременно перепроводит документ на тот же склад. Первый удерживает исключительную блокировку среза регистра, второй ждёт или получает ошибку по таймауту.

```bsl
Блокировка = Новый БлокировкаДанных;
Элемент = Блокировка.Добавить("РегистрНакопления.ТоварыНаСкладах");
Элемент.Режим = РежимБлокировкиДанных.Исключительный;
Элемент.УстановитьЗначение("Склад", СкладОтгрузки);
Попытка
    Блокировка.Заблокировать();
Исключение
    ЗаписьЖурналаРегистрации("Блокировка", УровеньЖурналаРегистрации.Предупреждение,,
        "Не удалось получить блокировку склада " + СкладОтгрузки);
    ВызватьИсключение;
КонецПопытки;
```

В ТЖ включают события `DBMSLock`, `TTIMEOUT`, `EXCP` с длительностью ожидания. Монитор показывает дерево блокировок по номерам сеансов rphost.

На стороне SQL для PostgreSQL:

```sql
SELECT blocked.pid AS blocked_pid,
       blocking.pid AS blocking_pid,
       blocked.query AS blocked_query
FROM pg_stat_activity blocked
JOIN pg_stat_activity blocking ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))
WHERE cardinality(pg_blocking_pids(blocked.pid)) > 0;
```

После идентификации виновника смотрят стек вызова в отладчике или `CALL`/`Context` в ТЖ. Типичные причины: длинная транзакция, разный порядок блокировки регистров, массовое проведение без пакетирования.

## Подводные камни

- **Путаница deadlock и простого ожидания.** Deadlock разрешает СУБД откатом одной транзакции.
- **Диагностика только в момент ошибки.** Блокировка уже снята, след нужно ловить в ТЖ заранее.
- **Игнорирование фоновых регламентов.** Блокирует не пользователь, а scheduled job.
- **Убийство сеанса без согласования.** Откат чужой незавершённой операции.
- **Лечение таймаутом без смены алгоритма.** Ошибка просто приходит быстрее.
