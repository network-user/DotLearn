# Как сделать Grafana отказоустойчивой?

## Суть

Grafana сама по себе stateless на уровне UI, но хранит дашборды, пользователей и настройки во внешней БД. HA достигается несколькими репликами за балансировщиком, общим Postgres/MySQL и health checks. Без shared DB каждая реплика живёт своим миром и при падении одной pod теряются несинхронизированные правки.

## Что ответить на собесе

- Поднимаю 2+ реплики Grafana в k8s Deployment с `replicas >= 2`.
- Перед ними Ingress или LB с passive/active health check на `/api/health`.
- Внешняя HA БД (Patroni Postgres, RDS Multi-AZ) вместо SQLite.
- Session affinity обычно не нужна; важнее единый `GF_DATABASE_*`.
- Честно: HA Grafana без HA datasource (Prometheus/ES) даёт пустые дашборды при outage backend; лечу оба слоя.

## Как это работает

Минимальный k8s манифест с readiness и внешней БД:

```bash
kubectl apply -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: grafana
  namespace: observability
spec:
  replicas: 3
  selector:
    matchLabels:
      app: grafana
  template:
    metadata:
      labels:
        app: grafana
    spec:
      containers:
      - name: grafana
        image: grafana/grafana:11.2.0
        ports:
        - containerPort: 3000
        env:
        - name: GF_DATABASE_TYPE
          value: postgres
        - name: GF_DATABASE_HOST
          value: patroni.observability.svc:5432
        - name: GF_DATABASE_NAME
          value: grafana
        - name: GF_DATABASE_USER
          valueFrom:
            secretKeyRef:
              name: grafana-db
              key: user
        - name: GF_DATABASE_PASSWORD
          valueFrom:
            secretKeyRef:
              name: grafana-db
              key: password
        readinessProbe:
          httpGet:
            path: /api/health
            port: 3000
          periodSeconds: 10
        livenessProbe:
          httpGet:
            path: /api/health
            port: 3000
          initialDelaySeconds: 30
EOF
```

Проверка, что LB видит только healthy pods:

```bash
for pod in $(kubectl get pods -n observability -l app=grafana -o name); do
  kubectl exec -n observability $pod -- wget -qO- http://127.0.0.1:3000/api/health
done
```

## Подводные камни

- **SQLite в prod.** Default SQLite ломает multi-replica; миграция на Postgres обязательна до масштабирования.
- **Плагины и provisioning.** Разные версии плагинов на репликах дают расхождение UI; фиксирую image tag и init containers.
- **Alerting backend.** Unified alerting хранит state в БД; при медленном Postgres растёт latency notification.
- **Datasource secrets.** Каждая реплика читает одни и те же credentials; ротация через k8s Secret + reload.
- **Single LB SPOF.** HA Grafana бессмысленна, если один nginx без второго AZ; дублирую entry point.
