# GET, POST, and HEAD: where each method has its own area of responsibility

## In short

The three most common HTTP methods solve different problems: GET reads data and puts parameters in the URL, POST sends a payload in the request body, and HEAD returns only headers with no body. The choice of method determines whether the request is idempotent and whether the response can be cached.

## How to answer in an interview

GET fetches a resource from the server: parameters travel in the URL query string, the request does not change server state, so it is idempotent and cacheable. POST sends data in the body, suits large and private payloads, and is non-idempotent and not cacheable by default. HEAD is a GET without a response body: the server returns headers only, which is handy when you need only a resource's metadata (size, date, existence) and do not want to download the content itself.

## How it works

GET looks like this - everything meaningful is in the query string:

```http
GET /index.html?user=123 HTTP/1.1
Host: www.example.com
```

Key properties of GET: parameters are openly visible in the URL, the response can be placed in browser and proxy caches, and repeating the request is safe. GET is not suitable for secrets - the address ends up in history, logs, and the `Referer` header.

POST carries data in the body and explicitly describes its format:

```http
POST /submit-form HTTP/1.1
Host: www.example.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 16

name=John&age=30
```

The body is not limited by the URL length, so it suits forms, uploads, and large volumes. Repeating a POST may create a duplicate record, so the response is usually not cached.

HEAD is identical to GET in form, but the server must return only the headers:

```http
HEAD /index.html HTTP/1.1
Host: www.example.com
```

From the `Content-Length`, `Content-Type`, and `Last-Modified` headers the client learns the size and freshness of the resource without downloading it in full - this saves traffic when checking links and validating the cache.

## Pitfalls

- GET is not meant to change data: methods that the spec calls "safe" (GET, HEAD) can be triggered by crawlers and prefetchers on their own, and a hidden mutation behind a GET will turn into a bug.
- The idempotency of GET/HEAD is a contract, not magic: if your GET handler changes something, you violate the spec, and caches will start serving stale results.
- HEAD must return the same headers as GET for the same URL, but with an empty body. An inconsistent `Content-Length` between HEAD and GET breaks clients.
- The URL length limit (usually a few kilobytes) makes GET unsuitable for large parameters - use POST.
