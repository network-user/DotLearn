# How the TCP/IP stack is built and who is responsible for what in it

## In short

TCP/IP is a multi-layer family of protocols that the internet rests on. Each layer solves its own task: IP delivers packets by address, and TCP on top of it provides reliability: ordering, integrity, and retransmission of what was lost.

## How to answer in an interview

TCP/IP is a stack of four layers: link, internet, transport, and application. The application layer is the protocols of programs (HTTP, SMTP, FTP). The transport layer offers two delivery options: TCP with guaranteed ordering and reliability, or UDP with no guarantees for the sake of speed. The internet layer is IP, which routes packets between nodes by IP address. The link layer handles physical transmission over Ethernet or Wi-Fi. On sending, data is encapsulated top-down; on receiving, it is unwrapped bottom-up.

The layers and their roles:

1. **Application:** protocols of specific programs: HTTP for the web, FTP for files, SMTP for mail.
2. **Transport:** here are TCP and UDP. TCP opens a connection, numbers segments, verifies checksums, and retransmits what was lost. UDP sends datagrams without a connection and without error control: faster, but with no guarantees.
3. **Internet:** the IP protocol routes packets between nodes, and the IP address uniquely identifies a device.
4. **Link:** physical transmission over Ethernet, Wi-Fi, and other interfaces.

The data path when sending over TCP is top-down encapsulation:

- **The application** (the browser) forms a request.
- **TCP** cuts the data into segments, adds numbers and a checksum.
- **IP** appends a header with the sender's and receiver's addresses and routes the packet.
- **The link layer** transmits frames over the physical medium to the next node.

On receiving, everything is unwrapped in reverse: frames move up, TCP reassembles segments by number and hands a coherent stream to the application.

A minimal TCP server and client in Python:

```python
import socket

server_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server_socket.bind(('localhost', 8080))
server_socket.listen()

print("Server is waiting for a connection...")

client_socket, address = server_socket.accept()
print(f"Connection established with {address}")

data = client_socket.recv(1024).decode('utf-8')
print(f"Message received: {data}")

client_socket.send("Hello from the server!".encode('utf-8'))

client_socket.close()
server_socket.close()
```

```python
import socket

client_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
client_socket.connect(('localhost', 8080))

client_socket.send("Hello from the client!".encode('utf-8'))

response = client_socket.recv(1024).decode('utf-8')
print(f"Response from server: {response}")

client_socket.close()
```

Here `AF_INET` means IPv4, and `SOCK_STREAM` is specifically TCP. `bind()` and `listen()` prepare the server to accept, `connect()` opens the connection on the client side, `send()`/`recv()` exchange bytes, and `close()` releases the socket.

## Pitfalls

- **TCP is a byte stream, not a stream of messages.** One `send()` does not equal one `recv()`: data can be glued together or split apart. The application layer is itself responsible for parsing message boundaries.
- **`recv(1024)` returns up to 1024 bytes, not exactly that many.** You cannot rely on everything arriving in a single call, so real code reads in a loop.
- **Confusing the OSI and TCP/IP layers.** Classic OSI has seven layers; TCP/IP collapses them to four. In an interview, clarify which model is meant.
- **UDP lives at the same transport layer as TCP**, not somewhere below it, which is a common mistake in answers.
