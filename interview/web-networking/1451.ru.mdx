# GET, POST и HEAD: где у каждого метода своя зона ответственности

## Суть

Три самых ходовых HTTP-метода решают разные задачи: GET читает данные и кладёт параметры в URL, POST отправляет полезную нагрузку в теле запроса, а HEAD возвращает только заголовки без тела. Выбор метода определяет, идемпотентен ли запрос и можно ли кэшировать ответ.

## Что ответить на собесе

Три метода решают разные задачи, и выбор определяет идемпотентность и кэшируемость:

- GET забирает ресурс: параметры едут в строке запроса URL, состояние сервера не меняется, поэтому он идемпотентен и кэшируется.
- POST посылает данные в теле, годится для крупных и приватных payload'ов, по умолчанию неидемпотентен и не кэшируется.
- HEAD - это GET без тела ответа: сервер отдаёт одни заголовки, удобно, когда нужны только метаданные ресурса (размер, дата, существование), а контент скачивать не хочется.

## Как это работает

GET выглядит так - всё значимое в строке запроса:

```http
GET /index.html?user=123 HTTP/1.1
Host: www.example.com
```

Ключевые свойства GET: параметры открыто видны в URL, ответ можно класть в кэш браузера и прокси, повтор запроса безопасен. Для секретов GET не подходит - адрес попадёт в историю, логи и заголовок `Referer`.

POST переносит данные в тело и явно описывает их формат:

```http
POST /submit-form HTTP/1.1
Host: www.example.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 16

name=John&age=30
```

Тело не ограничено длиной URL, поэтому подходит для форм, загрузок и больших объёмов. Повтор POST может создать дубликат записи, поэтому ответ обычно не кэшируется.

HEAD идентичен GET по форме, но сервер обязан вернуть только заголовки:

```http
HEAD /index.html HTTP/1.1
Host: www.example.com
```

По заголовкам `Content-Length`, `Content-Type`, `Last-Modified` клиент понимает размер и свежесть ресурса, не качая его целиком - это экономит трафик при проверке ссылок и валидации кэша.

## Подводные камни

- GET не предназначен для изменения данных: «безопасные» по спецификации методы (GET, HEAD) краулеры и предзагрузчики могут дёргать сами, и скрытая мутация за GET обернётся багом.
- Идемпотентность GET/HEAD - это контракт, а не магия: если ваш обработчик GET что-то меняет, вы нарушаете спецификацию, и кэши начнут отдавать устаревший результат.
- HEAD обязан отдавать те же заголовки, что и GET для того же URL, но с пустым телом. Несогласованный `Content-Length` между HEAD и GET ломает клиентов.
- Лимит длины URL (обычно несколько килобайт) делает GET непригодным для больших параметров - используйте POST.
