# Какими приёмами ускорить сервис под потоком HTTP-запросов?

## Суть

Производительность под нагрузкой набирают комбинацией приёмов: асинхронная обработка ввода-вывода, HTTP/2 с мультиплексированием, кэширование, сжатие тел ответов и балансировка нагрузки между серверами. Каждый бьёт по своему узкому месту - от блокировки потока до объёма трафика.

## Что ответить на собесе

Я бы шёл по слоям, на каждом закрывая своё узкое место:

- Приложение: для I/O-bound нагрузки беру асинхронность (`asyncio` плюс `aiohttp`) — один поток обслуживает тысячи соединений, не простаивая на ожидании сети.
- Транспорт: HTTP/2 с мультиплексированием шлёт много запросов по одному соединению и убирает head-of-line blocking из HTTP/1.1.
- Кэширование: часто запрашиваемые данные отдаём из кэша (клиент, сервер, CDN/прокси), не дёргая бэкенд.
- Сжатие (Gzip/Brotli) уменьшает объём ответа и время доставки, а балансировщик (Nginx, HAProxy) раскидывает запросы по инстансам ради пропускной способности и отказоустойчивости.

## Как это работает

Асинхронный код не блокирует поток на ожидании сети - пока один запрос ждёт ответа, обрабатываются другие.

```python
import asyncio
import aiohttp

async def fetch(session, url):
    async with session.get(url) as response:
        return await response.text()

async def main():
    urls = ["http://example.com", "http://example.org"]
    async with aiohttp.ClientSession() as session:
        tasks = [fetch(session, url) for url in urls]
        results = await asyncio.gather(*tasks)
    print(len(results))

asyncio.run(main())
```

Здесь одна `ClientSession` переиспользует соединения, `await` отдаёт управление циклу событий на время ожидания, а `asyncio.gather` запускает корутины конкурентно.

Остальные слои:

- **HTTP/2:** мультиплексирование, сжатие заголовков, server push - несколько запросов по одному TCP-соединению с меньшими задержками.
- **Кэш:** на клиенте (статика в браузере), на сервере (результаты вычислений) и на прокси/CDN (повторные ответы ближе к пользователю).
- **Сжатие:** Gzip или Brotli уменьшают тело ответа; сервер жмёт, клиент распаковывает по `Accept-Encoding`.
- **Балансировка:** Nginx или HAProxy распределяют трафик по инстансам, повышая и throughput, и устойчивость к сбоям.

## Подводные камни

- Асинхронность ускоряет I/O-bound задачи, но не CPU-bound. Тяжёлые вычисления заблокируют цикл событий - их выносят в процессы или пул потоков.
- Создавать `ClientSession` на каждый запрос - антипаттерн: теряется переиспользование соединений. Одна сессия на пачку запросов.
- HTTP/2 убирает HOL-blocking на уровне HTTP, но не на уровне TCP при потерях пакетов - это решает уже HTTP/3 (QUIC).
- Кэш без продуманной инвалидации отдаёт устаревшие данные. Согласованность - главная боль кэширования.
- Сжатие тратит CPU. Для крошечных ответов или уже сжатого контента (картинки, видео) оно может только вредить.
- Балансировщик не починит общий бутылочный горлышко: если все инстансы упираются в одну БД, размножение серверов не поможет.
