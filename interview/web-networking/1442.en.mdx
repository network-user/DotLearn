# JWT from the inside: the token's three parts, the signature, and why the server can skip storing sessions

## In short

A JWT (JSON Web Token) is a compact, URL-safe token of three parts (header, payload, signature) encoded with Base64URL and separated by dots. The signature proves the payload was not tampered with, so the server can verify a request without storing session state.

## How to answer in an interview

A JWT is a self-contained token that parties use to exchange information as signed JSON. It consists of three dot-separated parts: header (type and algorithm), payload (claims, i.e. assertions about the user), and signature. The first two parts are just Base64URL, so anyone can read them; only the signature protects them. The server issues the token after login, the client sends it in `Authorization: Bearer <token>`, the server verifies the signature with a secret (HS256) or a public key (RS256) and trusts the contents. No state is kept on the server, which simplifies horizontal scaling and SSO.

The structure is `header.payload.signature`:

1. **Header**: token type and signing algorithm:

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

2. **Payload**: a set of claims. There are registered ones (`iss` for issuer, `sub` for subject, `exp` for expiration, `iat` for issued at) and arbitrary custom ones:

```json
{
  "sub": "1234567890",
  "name": "Ivan Petrov",
  "admin": true,
  "exp": 1735689600
}
```

3. **Signature** takes Base64URL(header) + "." + Base64URL(payload) and signs it with a secret using the chosen algorithm:

```python
import jwt

payload = {"sub": "1234567890", "name": "Ivan", "admin": True}
secret = "your-256-bit-secret"

token = jwt.encode(payload, secret, algorithm="HS256")
decoded = jwt.decode(token, secret, algorithms=["HS256"])
```

Authentication flow: the client logs in → the server issues a JWT → the client sends it with every request → the server verifies the signature and expiry and grants access. Used in REST APIs and in SSO, where a single authentication unlocks multiple applications.

## Pitfalls

- **The payload is encoded, not encrypted**: Base64URL can be decoded by anyone, so do not put secrets in the payload (for encryption you need JWE).
- **`alg: none` and algorithm substitution** (a classic vulnerability): the library must not trust the `alg` field from the token itself; always set `algorithms=[...]` explicitly.
- **Token revocation** is tricky. An issued JWT is valid until `exp` and cannot be "logged out" without a separate blacklist or a short TTL plus refresh tokens.
- **Expiration is mandatory**, since without `exp` a stolen token lives forever; make access tokens short-lived.
- **HS256 vs RS256** differ in trust model: HS256 has a single shared secret (anyone who can verify can also sign), while RS256 splits the private issuing key from the public verification key, which is safer for distributed systems.
