# Where to place the token in an HTTP request, and which way to prefer?

## In short

A token is attached to a request in one of three ways: in an HTTP header, in the URL query string, or in the body. The canonical and most secure option is the `Authorization: Bearer <token>` header.

## How to answer in an interview

I'll name three places and prioritize them right away. The first and preferred one is the `Authorization` header with the `Bearer` scheme: the token doesn't end up in the URL, which means it doesn't settle into server logs, browser history, or the `Referer` header. The second is a query parameter (`?access_token=...`): workable but insecure, because the URL is logged and cached; I use it only when a header isn't available. The third is the request body: the token isn't in the URL, but this way works only for requests with a body (POST and the like) and breaks the idea of stateless authentication, where the token is expected in the header on every request. Bottom line: by default `Authorization: Bearer`, the rest are forced exceptions.

## How it works

**1. The Authorization header.** The standard mechanism. The server reads the header, separates the `Bearer` scheme from the value, and verifies the token.

```http
GET /protected/resource HTTP/1.1
Host: example.com
Authorization: Bearer <token>
```

The token isn't visible in the URL or body, so the risk of leakage through logs and `Referer` is minimal.

**2. URL parameter.** The token is placed in the query string.

```http
GET /protected/resource?access_token=<token> HTTP/1.1
Host: example.com
```

Simple but dangerous: the query string is written into access logs, ends up in browser history, and can leak through `Referer` when navigating to a third-party resource.

**3. Request body.** The token is passed in the JSON body, usually in a POST.

```http
POST /protected/resource HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "access_token": "<token>"
}
```

The token disappears from the URL, but this way applies only to requests with a body and fits poorly with the stateless scheme, where the server expects the token in the header regardless of the method.

## Pitfalls

- **Token in the URL = token in the logs.** Query parameters are written by default into the logs of web servers, proxies, and CDNs. This is the most common channel for accidental leakage.
- **Leakage through Referer.** If a page with a token in the URL has links or external resources, the browser may pass the full URL in the `Referer` header to a foreign domain.
- **GET is not for a body.** Many clients and proxies ignore the body of a GET request, so putting a token in a GET body is unreliable: the header is what remains.
- **Bearer is meaningless without TLS.** A bearer token is a "key to whoever holds it": whoever intercepts it gets access. Any transmission method is secure only over HTTPS.
- **Precision of the scheme.** The server must check exactly `Authorization: Bearer <token>` with a space separator; confusing schemes (`Basic`, `Bearer`) or extra spaces break token parsing.
