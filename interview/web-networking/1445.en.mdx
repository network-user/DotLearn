# What are cookies and what role do they play in how sites work?

## In short

A cookie is a small piece of data that the server asks the browser to store and then attach to every request to the same site. This gives HTTP, which is stateless on its own, a "memory": sessions, authorization, and user settings.

## How to answer in an interview

Cookies solve the stateless nature of HTTP. The server sends a `Set-Cookie` header, the browser remembers it and then automatically sends it back in the `Cookie` header on every request to that domain. The main use is sessions and authentication: after login, a session identifier or token is placed in a cookie, and the server uses it to recognize the user. They are also used for personalization (language, theme) and analytics or advertising. Cookies have security attributes: `HttpOnly` hides the cookie from JavaScript, `Secure` sends it only over HTTPS, and `SameSite` limits cross-domain sending and protects against CSRF.

## How it works

On visiting a site, the browser may receive a `Set-Cookie` from the server. It stores the value and automatically adds it to every subsequent request to that same domain — this is how the server recognizes the client and gives it a personalized response.

```python
from http.cookies import SimpleCookie

cookie = SimpleCookie()
cookie["language"] = "ru"
cookie["language"]["path"] = "/"
cookie["language"]["max-age"] = 3600
cookie["language"]["httponly"] = True

print(cookie.output())
# Set-Cookie: language=ru; HttpOnly; Max-Age=3600; Path=/
```

What the attributes set:

- `path=/` — the cookie is available across the whole site, not just one directory.
- `max-age=3600` — lifetime in seconds (an hour here); without it the cookie is a session cookie and dies when the browser closes.
- `httponly` — the cookie is not visible from JavaScript, which reduces the risk of theft via XSS.

## Pitfalls

- HTTP without cookies does not remember the client between requests. A cookie is exactly the way to give it state, not just a "file with settings".
- Sensitive cookies without `HttpOnly` are vulnerable to theft via XSS, and without `Secure` they can leak over plain HTTP. For sessions these flags are mandatory.
- `SameSite` matters against CSRF. By default in modern browsers it is `Lax`; for cross-domain sending you need `None`, but only together with `Secure`.
- Cookie size is limited (about 4 KB), and they travel with every request. Storing a lot of data in them means bloating the traffic; for that there are server-side sessions and tokens.
- Cookies are bound to a domain and path. A cookie from `client.com` won't travel to `example.com` — a common misconception.
