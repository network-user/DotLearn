# Пара access/refresh: как обновить сессию без повторного логина?

## Суть

Refresh-токен — это долгоживущий секрет, по которому клиент выпрашивает новый короткоживущий access-токен, не вводя логин и пароль заново. Access носит права и живёт минуты, refresh лежит в защищённом хранилище и продлевает сессию.

## Что ответить на собесе

Access-токен делают короткоживущим (например, 15 минут), чтобы при утечке окно злоупотребления было маленьким, а чтобы не логиниться каждые 15 минут, при входе сервер выдаёт пару access + refresh.

- Access ходит в каждый запрос к API; когда он протухает, клиент шлёт refresh на отдельный эндпоинт и получает свежий access.
- Refresh живёт дольше и хранится максимально надёжно — обычно в httpOnly-cookie, недоступной из JavaScript.
- Хороший ответ добавит: refresh стоит делать одноразовым с ротацией (каждое обновление выдаёт новый refresh и инвалидирует старый) и держать на сервере список отзыва, чтобы можно было погасить сессию.

1. **Логин.** Сервер проверяет учётные данные и возвращает два токена. Access уходит в память/заголовок клиента, refresh кладётся в защищённое хранилище.
2. **Обычные запросы.** Клиент шлёт access в заголовке `Authorization`. Пока токен валиден, ресурс отдаётся.
3. **Истечение.** Протухший access приводит к ответу `401`. Это сигнал клиенту обновиться.
4. **Обновление.** Клиент отправляет refresh на эндпоинт обновления; сервер проверяет подпись и срок, сверяется со списком отзыва и выдаёт новый access (а при ротации — и новый refresh).

```python
import requests

refresh_url = "https://example.com/api/token/refresh"
refresh_token = "<refresh_token_из_защищённого_хранилища>"

response = requests.post(
    refresh_url,
    headers={"Content-Type": "application/json"},
    json={"refresh_token": refresh_token},
)

if response.status_code == 200:
    new_access_token = response.json().get("access_token")
    print("Новый access-токен получен")
else:
    print("Не удалось обновить токен:", response.status_code)
```

## Подводные камни

- **Where to store.** Access можно держать в памяти приложения; refresh безопаснее всего в httpOnly + Secure cookie. localStorage уязвим к XSS — украденный оттуда refresh даёт злоумышленнику долгий доступ.
- **Кража refresh.** Долгий срок жизни делает refresh лакомой целью. Защита — ротация при каждом использовании и детект повторного применения уже обменянного токена (признак кражи, на который реагируют отзывом всей цепочки).
- **Отзыв.** JWT по умолчанию самодостаточен и не проверяется по базе, поэтому «выйти со всех устройств» без серверного списка отзыва (или короткого TTL у access) не получится.
- **Гонка обновления.** Если у клиента несколько параллельных запросов словили `401` одновременно, они могут одновременно дёрнуть refresh. Нужно сериализовать обновление, иначе при ротации часть запросов получит уже инвалидированный токен.
- **Состав payload.** В access кладут только то, что не страшно показать: тело JWT лишь base64url-кодировано, а не зашифровано, и легко читается.
