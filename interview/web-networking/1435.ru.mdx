# Какой набор HTTP-методов используют клиенты и за что отвечает каждый

## Суть

HTTP-метод (глагол) сообщает серверу, какое действие клиент хочет выполнить над ресурсом. Базовый набор: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS — от чтения данных до удаления и запроса метаданных.

## Что ответить на собесе

Скажу так: метод запроса задаёт намерение клиента. GET читает ресурс и ничего не меняет, POST создаёт новую сущность или запускает действие, PUT заменяет ресурс целиком, PATCH правит лишь отдельные поля, DELETE удаляет. Особняком стоят HEAD (как GET, но сервер отдаёт только заголовки без тела) и OPTIONS (узнать, какие методы вообще разрешены для ресурса). Важно держать в голове свойства методов: GET, HEAD, PUT и DELETE идемпотентны, GET и HEAD ещё и безопасны (не меняют состояние), а POST и PATCH в общем случае ни тем, ни другим не являются.

## Как это работает

Метод — это первое слово в стартовой строке запроса. Через библиотеку `requests` каждому глаголу соответствует своя функция.

```python
import requests

resp = requests.get("https://api.example.com/items")
print(resp.status_code, resp.json())
```

Создание новой записи через POST: тело уходит на сервер, обычно как JSON.

```python
import requests

payload = {"name": "widget", "price": 10}
resp = requests.post("https://api.example.com/items", json=payload)
print(resp.status_code)
```

Полная замена против частичного обновления — это и есть разница PUT и PATCH:

```python
import requests

requests.put("https://api.example.com/items/1", json={"name": "x", "price": 20})
requests.patch("https://api.example.com/items/1", json={"price": 25})
```

PUT требует, чтобы вы прислали ресурс целиком, иначе пропущенные поля могут затереться. PATCH меняет только то, что передали.

HEAD и OPTIONS — служебные. HEAD проверяет наличие или метаданные ресурса (размер, дату) без скачивания тела, OPTIONS возвращает список разрешённых методов в заголовке `Allow`.

```python
import requests

resp = requests.options("https://api.example.com/items")
print(resp.headers.get("Allow"))
```

## Подводные камни

- GET не должен менять состояние сервера. Если «получение» что-то удаляет или списывает деньги — это ошибка дизайна, такие операции делают через POST.
- Идемпотентность не равна безопасности: DELETE идемпотентен (повтор даёт тот же итог — ресурса нет), но он меняет состояние, поэтому небезопасен.
- POST не идемпотентен: повторная отправка формы создаст дубликат. Для защиты от двойных списаний используют ключи идемпотентности.
- Не путайте PUT и PATCH: ошибочный PUT с неполным телом способен затереть существующие поля значениями по умолчанию.
- GET с телом запроса формально не запрещён, но многие прокси и серверы его игнорируют — параметры передают через query-строку.
