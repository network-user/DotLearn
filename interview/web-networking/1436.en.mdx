# What does the letter S add to HTTPS compared with plain HTTP

## In short

HTTPS is the same HTTP, but over a secure TLS channel. It adds three guarantees: traffic encryption, confirmation of the server's authenticity via a certificate, and integrity checking of data in transit.

## How to answer in an interview

I'd answer like this: HTTP on its own sends everything as plain text over port 80, and anyone along the route can read or tamper with the data. HTTPS wraps HTTP in a TLS layer (formerly called SSL) and works on port 443. TLS gives me three things: confidentiality — the traffic is encrypted; authenticity — the certificate proves I'm talking to exactly the intended server and not a man in the middle; integrity — any byte tampering along the way will be detected. The syntax of requests and methods itself does not change; only the transport underneath them does.

## How it works

At the message level, an HTTP request looks identical in both cases:

```http
GET /index.html HTTP/1.1
Host: www.example.com
```

The difference is that under HTTPS this text does not go onto the network as-is — it is encrypted. Before exchanging data, the client and server perform a TLS handshake:

1. The client connects to port 443 and sends the supported TLS versions and ciphers.
2. The server responds with its certificate, signed by a trusted certificate authority (CA).
3. The parties agree on session keys (modern suites use a Diffie-Hellman exchange for forward secrecy).
4. After that, all application traffic flows symmetrically encrypted.

You can check that the connection is secure from code too — a successful request over `https://` means the certificate passed validation:

```python
import requests

resp = requests.get("https://www.example.com")
print(resp.url.startswith("https://"))
```

## Pitfalls

- HTTPS encrypts the body and headers but does not hide the fact of the connection itself: the host name and IP are visible (the domain leaks via SNI and DNS, unless their encrypted variants are used).
- A valid certificate confirms domain ownership but not the site's "honesty" — a phishing site can also have correct HTTPS.
- Disabling certificate verification (`verify=False`) in production is not acceptable: it removes protection from man-in-the-middle and leaves only encryption "to nowhere".
- HTTPS does not save you from vulnerabilities on the server itself (SQL injection, XSS) — it protects only the transmission channel.
- Mixed content: a page served over HTTPS that loads resources over HTTP loses part of its guarantees, and browsers block such requests.
