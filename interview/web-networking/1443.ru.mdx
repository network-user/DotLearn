# SOAP против REST: где жёсткий протокол, а где архитектурный стиль?

## Суть

SOAP - это формальный протокол с жёсткой структурой сообщений на XML и набором WS-стандартов. REST - не протокол, а архитектурный стиль поверх готового HTTP. Первый выбирают за встроенную безопасность и надёжность, второй - за простоту и масштабируемость.

## Что ответить на собесе

Скажу так: SOAP - это протокол, REST - архитектурный стиль, и это ключевое различие.

- SOAP диктует обязательный конверт из XML (Header плюс Body), не зависит от транспорта (HTTP, SMTP, TCP) и тянет за собой стандарты вроде WS-Security и WS-ReliableMessaging - отсюда его место в банках и enterprise, где нужны гарантии.
- REST оперирует ресурсами через URI и стандартные HTTP-методы (GET, POST, PUT, DELETE), формат тела свободный, обычно JSON.
- Поэтому REST легче, кэшируется средствами HTTP и проще масштабируется - он и стал стандартом для публичных API.

## Как это работает

SOAP оборачивает каждое сообщение в конверт `Envelope`: в `Header` лежат метаданные о маршрутизации и безопасности, в `Body` - сам запрос или ответ. Структура валидируется по схеме, поэтому отклонения не пройдут.

```xml
<soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope">
  <soap:Header>
    <!-- метаданные: безопасность, маршрутизация -->
  </soap:Header>
  <soap:Body>
    <m:GetPrice xmlns:m="http://www.example.org/stock">
      <m:StockName>IBM</m:StockName>
    </m:GetPrice>
  </soap:Body>
</soap:Envelope>
```

REST вместо конверта опирается на сам HTTP: ресурс адресуется через URI, а действие выбирается методом. Тело может быть JSON, XML, HTML - что угодно, но чаще JSON.

```http
GET /stocks/IBM HTTP/1.1
Host: api.example.com
Accept: application/json
```

Разница на пальцах:

- **Природа:** SOAP - протокол с жёсткими правилами; REST - стиль, использующий существующие стандарты.
- **Формат:** SOAP только XML; REST принимает любой формат, по умолчанию JSON.
- **Транспорт:** SOAP не привязан к HTTP; REST живёт в семантике HTTP-методов и статусов.
- **Где уместен:** SOAP - сложные корпоративные системы с требованиями к безопасности; REST - веб и публичные API.

## Подводные камни

- Не путайте уровни сравнения: сравнивать "SOAP и REST" корректно только понимая, что одно - протокол, а другое - стиль. Аналог REST скорее gRPC или GraphQL, а аналог SOAP - другой строгий протокол.
- REST сам по себе не запрещает XML, а SOAP технически умеет ходить не только по HTTP - частая ошибка считать иначе.
- "RESTful" не равно "любой JSON-over-HTTP". Без работы с ресурсами, методами и статус-кодами это просто RPC поверх HTTP.
- Безопасность в REST не встроена как в SOAP - её добавляют отдельно (TLS, OAuth, JWT), и об этом легко забыть.
