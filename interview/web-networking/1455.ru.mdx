# WebSocket против сырого TCP/UDP-сокета: где проходит граница

## Суть

WebSocket — это протокол прикладного уровня, который поверх одного постоянного TCP-соединения даёт полнодуплексный обмен сообщениями с маленьким оверхедом. «Обычный» сокет — это системный примитив транспортного уровня (TCP или UDP), где рамкование, протокол поверх и логику соединения вы пишете руками.

## Что ответить на собесе

Это разные уровни абстракции, а не конкуренты. Сокет — это файловый дескриптор операционной системы, через который вы шлёте байты по TCP или UDP; всё остальное (где кончается одно сообщение и начинается другое, рукопожатие, переподключение) — ваша забота. WebSocket стоит выше: он начинается с обычного HTTP-запроса с `Upgrade: websocket`, после успешного апгрейда то же TCP-соединение переходит в режим обмена дискретными фреймами-сообщениями в обе стороны. То есть WebSocket почти всегда работает поверх TCP-сокета, но добавляет рукопожатие, рамкование сообщений и совместимость с веб-инфраструктурой (порты 80/443, прокси, браузер). Сырой сокет берут, когда нужен полный контроль или UDP; WebSocket — когда нужен дешёвый двусторонний канал из браузера в реальном времени.

## Как это работает

Сокет на транспортном уровне бывает двух видов:

- **TCP (`SOCK_STREAM`)** — соединение с гарантиями: подтверждения, переотправка потерь, порядок байтов. Но это именно поток байтов: границ сообщений в нём нет, и разбивать поток на сообщения нужно самому.
- **UDP (`SOCK_DGRAM`)** — датаграммы без соединения и без гарантий доставки/порядка. Быстрее, годится для стриминга и игр, где опоздавший пакет уже не нужен.

Сырой TCP-сервер на Python:

```python
import socket

server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
server.bind(("localhost", 8080))
server.listen()

conn, addr = server.accept()
data = conn.recv(1024)
conn.sendall(b"Hello, client!")

conn.close()
server.close()
```

Здесь `accept()` отдаёт отдельный сокет на каждого клиента, `recv(1024)` читает до 1024 байт потока (не «одно сообщение»), а `sendall` гарантирует отправку всего буфера.

WebSocket прячет эту возню за рукопожатием и фреймами. Сервер на библиотеке `websockets` уже оперирует целыми сообщениями:

```python
import asyncio
import websockets

async def echo(websocket):
    async for message in websocket:
        await websocket.send(f"Echo: {message}")

async def main():
    async with websockets.serve(echo, "localhost", 8765):
        await asyncio.Future()

asyncio.run(main())
```

`async for message in websocket` выдаёт уже собранные сообщения — рамкование сделала библиотека, а не вы. Схематично разница в стеке:

```text
WebSocket-сообщения  ──┐
HTTP Upgrade-handshake │  прикладной уровень
                       ▼
        TCP-сокет (поток байтов)   транспортный уровень
```

## Подводные камни

- **«WebSocket = сокет» — неверно.** WebSocket почти всегда живёт поверх TCP-сокета; это надстройка с рукопожатием и фреймами, а не альтернатива транспорту.
- **TCP не знает границ сообщений.** `recv()` может вернуть половину сообщения или сразу два склеенных — нужен свой разделитель или префикс длины. WebSocket эту проблему уже решает фреймами.
- **Рукопожатие WebSocket идёт по HTTP.** Поэтому он дружит с портами 80/443 и прокси, в отличие от произвольного TCP-порта, который корпоративный firewall может резать.
- **`recv(1024)` — это лимит чтения, а не размер сообщения.** Частая ошибка новичка — считать, что один `recv` равен одному логическому пакету.
- **UDP ничего не гарантирует.** Если выбрали `SOCK_DGRAM`, доставку, порядок и дедупликацию реализуете сами; WebSocket такой режим в принципе не предоставляет (он привязан к TCP).
- **Старый API event loop устарел.** `asyncio.get_event_loop().run_forever()` из старых примеров заменён на `asyncio.run()`; на свежих версиях прежний код выдаёт DeprecationWarning.
