# Client and server: who calls whom, and why split the roles

## In short

This is a separation of roles: the initiator (client) sends a request, and the resource provider (server) handles that request and returns a response. All of the storage and access-control logic is gathered on one side, which makes the system easier to manage and to grow.

## How to answer in an interview

The client-server model splits responsibilities between two sides. The client is the initiator: it forms a request and waits for a result (for example, a browser opening a page). The server passively listens, accepts incoming requests, does the work, and returns a response. The main benefit of this approach is that the data and access rules are centralized on the server: they are easier to update, secure, and scale under load, because a single server serves many clients at once.

## How it works

The flow is always one-directional by initiative: the request goes from the client, the response comes back. The server never starts anything itself; it reacts.

A minimal example over sockets. The server opens a listening socket, accepts connections in a loop, and replies to each one with a string.

**Server:**

```python
import socket

server_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server_socket.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
server_socket.bind(('localhost', 12345))
server_socket.listen()

print("Server is waiting for connections...")

while True:
    client_socket, address = server_socket.accept()
    print(f"Client connected: {address}")
    client_socket.sendall(b"Privet ot servera")
    client_socket.close()
```

**Client:**

```python
import socket

client_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
client_socket.connect(('localhost', 12345))

message = client_socket.recv(1024)
print(f"Message from server: {message.decode()}")

client_socket.close()
```

The server passively waits in `accept()` until someone connects, then sends the data and closes the connection. The client initiates `connect()` and reads the response via `recv()`.

## Pitfalls

- A single server is a point of failure. If it goes down, every client is cut off; in production you run several instances behind a load balancer.
- `recv(1024)` reads _up to_ 1024 bytes, not exactly that many. For long messages you need to read in a loop until you have received the full expected amount.
- Without `SO_REUSEADDR`, the port may stay occupied after a server restart because of the `TIME_WAIT` state.
- A "thick client" with business logic on its own side breaks centralization: critical checks (access, validation) end up duplicated on the server anyway, because the client cannot be trusted.
