# The access/refresh pair: how to renew a session without logging in again?

## In short

A refresh token is a long-lived secret that the client uses to request a new short-lived access token without re-entering a login and password. The access token carries permissions and lives for minutes, while the refresh token sits in secure storage and extends the session.

## How to answer in an interview

The access token is made short-lived (for example, 15 minutes) so that if it leaks, the window for abuse is small. But forcing the user to log in every 15 minutes is inconvenient, so on sign-in the server issues a pair: access and refresh. The access token rides along in every request to the API; when it expires, the client sends the refresh token to a separate endpoint and gets a fresh access token. The refresh token lives longer and is therefore stored as securely as possible, usually in an httpOnly cookie inaccessible from JavaScript. A good answer adds: the refresh token should be made single-use with rotation (each renewal issues a new refresh token and invalidates the old one) and the server should keep a revocation list so a session can be killed.

1. **Login.** The server checks the credentials and returns two tokens. The access token goes into the client's memory/header, the refresh token is placed in secure storage.
2. **Normal requests.** The client sends the access token in the `Authorization` header. As long as the token is valid, the resource is served.
3. **Expiration.** An expired access token leads to a `401` response. This is a signal for the client to renew.
4. **Renewal.** The client sends the refresh token to the renewal endpoint; the server checks the signature and expiry, consults the revocation list, and issues a new access token (and, with rotation, a new refresh token as well).

```python
import requests

refresh_url = "https://example.com/api/token/refresh"
refresh_token = "<refresh_token_from_secure_storage>"

response = requests.post(
    refresh_url,
    headers={"Content-Type": "application/json"},
    json={"refresh_token": refresh_token},
)

if response.status_code == 200:
    new_access_token = response.json().get("access_token")
    print("New access token obtained")
else:
    print("Failed to refresh token:", response.status_code)
```

## Pitfalls

- **Where to store.** The access token can be kept in the app's memory; the refresh token is safest in an httpOnly + Secure cookie. localStorage is vulnerable to XSS: a refresh token stolen from there gives an attacker long-lived access.
- **Refresh theft.** A long lifetime makes the refresh token a tempting target. The defense is rotation on every use and detection of reuse of an already-exchanged token (a sign of theft, which is met by revoking the whole chain).
- **Revocation.** A JWT is self-contained by default and isn't checked against a database, so "log out from all devices" isn't possible without a server-side revocation list (or a short access TTL).
- **Renewal race.** If a client has several parallel requests catch a `401` at once, they may hit refresh simultaneously. You need to serialize the renewal, otherwise with rotation some requests will get an already-invalidated token.
- **Payload contents.** Put only what's safe to show into the access token: a JWT body is merely base64url-encoded, not encrypted, and is easily read.
