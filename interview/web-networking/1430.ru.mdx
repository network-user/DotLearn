# Где разместить токен в HTTP-запросе и какой способ предпочесть?

## Суть

Токен прикладывают к запросу одним из трёх способов: в HTTP-заголовке, в строке запроса URL или в теле. Канонический и самый безопасный вариант — заголовок `Authorization: Bearer <token>`.

## Что ответить на собесе

Я назову три места и сразу расставлю приоритет: по умолчанию `Authorization: Bearer`, остальное — вынужденные исключения.

- Заголовок `Authorization` со схемой `Bearer` (предпочтительно): токен не попадает в URL, а значит не оседает в логах сервера, истории браузера и заголовке `Referer`.
- Query-параметр (`?access_token=...`): рабочий, но небезопасный, потому что URL логируется и кэшируется; использую только когда заголовок недоступен.
- Тело запроса: токена нет в URL, но способ годится лишь для запросов с телом (POST и подобные) и нарушает идею stateless-аутентификации, где токен ждут в заголовке на каждом запросе.

## Как это работает

**1. Заголовок Authorization.** Стандартный механизм. Сервер читает заголовок, отделяет схему `Bearer` от значения и проверяет токен.

```http
GET /protected/resource HTTP/1.1
Host: example.com
Authorization: Bearer <token>
```

Токен не виден в URL и теле, поэтому риск утечки через логи и `Referer` минимален.

**2. Параметр URL.** Токен кладётся в query-строку.

```http
GET /protected/resource?access_token=<token> HTTP/1.1
Host: example.com
```

Просто, но опасно: query-строка пишется в access-логи, попадает в историю браузера и может утечь через `Referer` при переходе на сторонний ресурс.

**3. Тело запроса.** Токен передаётся в JSON-теле, обычно в POST.

```http
POST /protected/resource HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "access_token": "<token>"
}
```

Из URL токен исчезает, но способ применим только к запросам с телом и плохо ложится на stateless-схему, где сервер ожидает токен в заголовке независимо от метода.

## Подводные камни

- **Токен в URL = токен в логах.** Query-параметры по умолчанию пишутся в журналы веб-серверов, прокси и CDN. Это самый частый канал случайной утечки.
- **Утечка через Referer.** Если на странице с токеном в URL есть ссылки или внешние ресурсы, браузер может передать полный URL в заголовке `Referer` чужому домену.
- **GET не для тела.** Многие клиенты и прокси игнорируют тело у GET-запроса, поэтому класть токен в тело GET ненадёжно — остаётся заголовок.
- **Bearer без TLS бессмысленен.** Bearer-токен — это «ключ на предъявителя»: перехвативший его получает доступ. Любой способ передачи безопасен только поверх HTTPS.
- **Точность схемы.** Сервер должен проверять именно `Authorization: Bearer <token>` с пробелом-разделителем; путаница со схемами (`Basic`, `Bearer`) или лишние пробелы ломают парсинг токена.
