# Ограничение частоты запросов: для чего и какими алгоритмами реализуют?

## Суть

Rate limiting задаёт верхнюю границу на количество обращений к ресурсу в единицу времени. Это удерживает нагрузку в рамках, блокирует злоупотребления и делит пропускную способность между клиентами по-честному.

## Что ответить на собесе

Rate limiting нужен, чтобы сервис не захлёбывался под наплывом запросов и не страдал от перебора со стороны отдельных клиентов. Я бы назвал три причины: защита инфраструктуры от перегрузки, отражение паразитного и атакующего трафика, справедливое распределение квот. Классические алгоритмы — Fixed Window (счётчик в фиксированном интервале), Sliding Window (плавающее окно по меткам времени) и Token Bucket (ведро токенов, пополняемое с постоянной скоростью). Выбор зависит от того, насколько ровно нужно сглаживать всплески: Token Bucket допускает короткие пики, Sliding Window даёт более равномерный поток.

## Как это работает

Идея у всех вариантов одна — отслеживать, сколько запросов уже прошло, и отклонять лишние. Различается лишь способ учёта.

**Fixed Window.** Делим ось времени на равные интервалы; внутри каждого работает свой лимит. Просто, но на стыке двух окон возможен двойной всплеск.

```python
import time

class FixedWindowRateLimiter:
    def __init__(self, max_requests, window_size):
        self.max_requests = max_requests
        self.window_size = window_size
        self.requests = 0
        self.window_start = time.time()

    def allow_request(self):
        now = time.time()
        if now - self.window_start >= self.window_size:
            self.window_start = now
            self.requests = 0
        if self.requests < self.max_requests:
            self.requests += 1
            return True
        return False
```

**Sliding Window.** Храним метки времени запросов в деке и выбрасываем те, что вышли за границу окна. Граница «едет» вместе с текущим моментом, поэтому распределение ровнее.

```python
from collections import deque
import time

class SlidingWindowRateLimiter:
    def __init__(self, max_requests, window_size):
        self.max_requests = max_requests
        self.window_size = window_size
        self.requests = deque()

    def allow_request(self):
        now = time.time()
        while self.requests and self.requests[0] <= now - self.window_size:
            self.requests.popleft()
        if len(self.requests) < self.max_requests:
            self.requests.append(now)
            return True
        return False
```

**Token Bucket.** В ведре лежат токены: каждый разрешённый запрос тратит один. Токены доливаются с постоянной скоростью, а вместимость ведра ограничивает размер допустимого всплеска.

```python
import time

class TokenBucketRateLimiter:
    def __init__(self, max_tokens, refill_rate):
        self.max_tokens = max_tokens
        self.refill_rate = refill_rate
        self.tokens = max_tokens
        self.last_refill = time.time()

    def allow_request(self):
        now = time.time()
        self.tokens = min(
            self.max_tokens,
            self.tokens + (now - self.last_refill) * self.refill_rate,
        )
        self.last_refill = now
        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False
```

## Подводные камни

- **Граница окна в Fixed Window.** На стыке двух соседних интервалов клиент может уложить почти двойной лимит: пачку в конце первого окна и сразу пачку в начале второго.
- **Память Sliding Window.** Дек хранит метку на каждый запрос, так что при больших лимитах расход памяти на клиента растёт линейно.
- **Состояние при горизонтальном масштабировании.** Если инстансов несколько, локальные счётчики в памяти разъезжаются. Нужен общий стор (обычно Redis с атомарными операциями), иначе суммарный лимит превысит задуманный.
- **Гонки.** `allow_request` должен быть атомарным; при конкурентном доступе проверка и инкремент без блокировки пропустят лишнее.
- **Что отдавать клиенту.** Отклонённый запрос — это код `429 Too Many Requests`, желательно с заголовком `Retry-After` и счётчиками `X-RateLimit-*`, иначе клиент не поймёт, когда повторять.
