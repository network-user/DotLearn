# REST как стиль и RESTful как реализация: в чём именно разница

## Суть

REST - это архитектурный стиль с набором ограничений для проектирования сетевых приложений. RESTful API - конкретный API, который этим ограничениям следует: работает с ресурсами через URI и стандартные HTTP-методы. Разница как между «принципами» и «их соблюдением на практике».

## Что ответить на собесе

REST - теория и стиль, RESTful - прилагательное, описывающее реальную реализацию.

- REST (Representational State Transfer) - набор архитектурных ограничений: клиент-сервер, отсутствие состояния, кэшируемость, единообразный интерфейс, слоистость.
- «RESTful» означает, что конкретный API спроектирован по этим ограничениям.
- На практике REST-API адресует ресурсы через URI (`/users`, `/users/42`) и оперирует ими через HTTP-методы: GET читает, POST создаёт, PUT/PATCH обновляет, DELETE удаляет.
- Ключевая идея statelessness: каждый запрос самодостаточен, сервер не помнит клиента между запросами, что упрощает масштабирование.

## Как это работает

Ограничения REST, которые делают API «RESTful»:

- **Клиент-сервер:** интерфейс и хранение данных разделены и развиваются независимо.
- **Stateless:** каждый запрос несёт всё необходимое (токен, параметры); сервер не держит сессионное состояние между обращениями.
- **Кэшируемость:** ответы явно помечаются как кэшируемые или нет, чтобы клиент не дёргал сервер зря.
- **Единообразный интерфейс:** одинаковая адресация ресурсов и предсказуемая семантика методов.
- **Слоистость:** между клиентом и сервером могут стоять прокси, балансировщики и кэши, не ломая контракт.

Для ресурса «пользователи» типичный набор операций выглядит так:

```http
GET    /users        # список пользователей
GET    /users/42     # один пользователь по id
POST   /users        # создать пользователя
PUT    /users/42     # заменить пользователя целиком
DELETE /users/42     # удалить пользователя
```

Один и тот же URI означает один ресурс, а метод задаёт действие над ним. Это и есть единообразный интерфейс: клиенту не нужно учить отдельные «команды» для каждой сущности.

## Подводные камни

- «REST» и «RESTful» в обиходе используют как синонимы, но строго: REST - стиль, RESTful - свойство реализации. На собесе это различие любят проверять.
- Глаголы в URI (`/getUser`, `/createUser`) - антипаттерн: действие должен задавать HTTP-метод, а URI - называть ресурс существительным.
- Statelessness часто нарушают серверными сессиями в памяти - тогда API перестаёт горизонтально масштабироваться без липких сессий.
- PUT обычно заменяет ресурс целиком и идемпотентен, PATCH - частичное обновление; путаница между ними ведёт к потере полей.
- «RESTful» не равно «любой JSON по HTTP»: без соблюдения ограничений это просто HTTP-API, а не REST.
