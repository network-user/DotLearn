# Anatomy of an HTTP request: start line, headers, body

## In short

An HTTP request has three parts: a start line with the method, target, and protocol version; a set of headers with metadata; and an optional body with the payload. An empty line separates the headers from the body.

## How to answer in an interview

An HTTP request is a text message made of three blocks. The first is the start line: the method (GET, POST, PUT, DELETE), the request target (the path to the resource), and the protocol version (HTTP/1.1, HTTP/2). The second is the headers in "key: value" format, each on its own line: Host, User-Agent, Content-Type, and so on. The third is the body, which is not always present: a GET usually has none, while POST and PUT carry the data being sent in it. Between the headers and the body sits an empty separator line.

The request splits into three parts:

1. **Start line (Request Line)**
2. **Headers**
3. **Body**

The **start line** describes the client's intent and contains the method, target, and version:

```
GET /index.html HTTP/1.1
```

Here `GET` requests a resource, `/index.html` is the path to it, and `HTTP/1.1` is the protocol version.

The **headers** carry meta-information, one per line:

```
Host: www.example.com
User-Agent: Mozilla/5.0
Accept-Language: en-US
```

`Host` indicates the server's domain, `User-Agent` describes the client, and `Accept-Language` reports the desired response language.

The **body** sends data to the server and is present in requests like POST or PUT. Note the empty line between the headers and the body:

```
POST /submit-form HTTP/1.1
Host: www.example.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 16

name=John&age=30
```

Here the start line is `POST /submit-form HTTP/1.1`, followed by the headers, then an empty line, and `name=John&age=30` is the body with the form data.

When you type an address and press Enter, the browser assembles such a request, sends it to the server, receives an HTTP response with the resource or an error, and displays the result.

## Pitfalls

- **The empty line is a required separator.** It is exactly what separates the headers from the body; without it the server cannot tell where the headers end.
- **`Content-Length` must match the actual body length in bytes.** A mismatch breaks request parsing. In the example, `name=John&age=30` is 16 bytes.
- **Header names are case-insensitive, values are not.** `Host` and `host` are equivalent, but the value is taken as-is.
- **A GET request can technically have a body, but it is not customary** - many servers and proxies ignore it. GET data is usually passed through the URL query string.
- **HTTP/2 is binary, not textual.** The familiar text form with a start line is about HTTP/1.1; in HTTP/2 the same information is encoded differently.
