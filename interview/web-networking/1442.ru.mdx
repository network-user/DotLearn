# JWT изнутри: три части токена, подпись и почему сервер может не хранить сессии

## Суть

JWT (JSON Web Token) — компактный URL-безопасный токен из трёх частей (заголовок, нагрузка, подпись), закодированных Base64URL и разделённых точками. Подпись доказывает, что нагрузку не подменили, поэтому сервер проверяет запрос, не храня состояние сессии.

## Что ответить на собесе

JWT — это самодостаточный токен, которым стороны обмениваются информацией в виде подписанного JSON. Он состоит из трёх частей через точку: header (тип и алгоритм), payload (claims — утверждения о пользователе) и signature (подпись). Первые две части — это просто Base64URL, их видно всем; защищает именно подпись. Сервер выпускает токен после логина, клиент шлёт его в `Authorization: Bearer <token>`, сервер проверяет подпись секретом (HS256) или публичным ключом (RS256) и доверяет содержимому. Состояние не хранится на сервере — это упрощает горизонтальное масштабирование и SSO.

Структура `header.payload.signature`:

1. **Header** — тип токена и алгоритм подписи:

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

2. **Payload** — набор claims. Есть зарегистрированные (`iss` — издатель, `sub` — субъект, `exp` — срок жизни, `iat` — время выпуска) и произвольные пользовательские:

```json
{
  "sub": "1234567890",
  "name": "Иван Петров",
  "admin": true,
  "exp": 1735689600
}
```

3. **Signature** — берут Base64URL(header) + "." + Base64URL(payload) и подписывают секретом по выбранному алгоритму:

```python
import jwt

payload = {"sub": "1234567890", "name": "Ivan", "admin": True}
secret = "your-256-bit-secret"

token = jwt.encode(payload, secret, algorithm="HS256")
decoded = jwt.decode(token, secret, algorithms=["HS256"])
```

Поток аутентификации: клиент логинится → сервер выпускает JWT → клиент шлёт его с каждым запросом → сервер проверяет подпись и срок и открывает доступ. Применяют в REST API и в SSO, где одна аутентификация открывает несколько приложений.

## Подводные камни

- **Payload не зашифрован, а закодирован** — Base64URL разворачивается обратно любым; секреты в нагрузку не кладут (для шифрования нужен JWE).
- **`alg: none` и подмена алгоритма** — классическая уязвимость: библиотека не должна доверять полю `alg` из самого токена; всегда задавайте `algorithms=[...]` явно.
- **Отзыв токена** — выданный JWT валиден до `exp` и его нельзя «разлогинить» без отдельного blacklist/короткого TTL + refresh-токенов.
- **Срок жизни обязателен** — без `exp` украденный токен живёт вечно; делайте access-токены короткими.
- **HS256 vs RS256** — у HS256 один общий секрет (любой, кто проверяет, может и подписать); RS256 разделяет приватный ключ выпуска и публичный ключ проверки — это безопаснее для распределённых систем.
