# WebSocket: a persistent two-way channel instead of endlessly polling the server

## In short

WebSocket keeps a single persistent full-duplex channel over TCP, through which the client and server send data both ways without repeated handshakes. This removes the overhead of HTTP polling and delivers real time: chats, games, stock quotes.

## How to answer in an interview

WebSocket is a protocol for bidirectional communication over a single TCP connection. The connection starts as an ordinary HTTP request with an `Upgrade: websocket` header; the server replies with `101 Switching Protocols`, after which the channel switches into duplex mode and stays open. Unlike HTTP's request-response model, the server can push data to the client on its own, without waiting for a request. That is why WebSocket is chosen wherever instant two-way exchange is needed and classic polling is inefficient.

Connection lifecycle:

1. **Handshake** — the client sends an HTTP/1.1 request with the `Upgrade: websocket` and `Connection: Upgrade` headers. The server confirms the switch with a `101 Switching Protocols` response.
2. **Duplex exchange** — after the upgrade both sides send messages as frames at any time; the server does not wait for a request to send data.
3. **Persistence** — the channel lives until one side closes it, so there is no need to re-establish the connection for every message.

Client-side code in JavaScript:

```javascript
const socket = new WebSocket('wss://example.com/socket');

socket.onopen = (event) => {
  console.log('Connection established');
  socket.send('Hello, server!');
};

socket.onmessage = (event) => {
  console.log('Message from server:', event.data);
};

socket.onclose = (event) => {
  console.log('Connection closed, code:', event.code);
};

socket.onerror = (error) => {
  console.log('WebSocket error:', error);
};
```

Where it fits: chats and messengers, online games, stock and trading platforms, IoT — anywhere data updates "right now" and timer-based polling would be wasteful.

## Pitfalls

- **`ws://` vs `wss://`** — unencrypted `ws://` sends data in plain text; in production use `wss://` (TLS), otherwise the traffic is readable by a man in the middle.
- **No auto-reconnect** — the browser's `WebSocket` does not recover from a drop on its own; reconnect logic with backoff is written by hand.
- **Proxies and load balancers** — not every intermediate node passes long-lived connections; you need sticky sessions and correct timeout configuration.
- **Heartbeat is mandatory** — without ping/pong a "dead" connection can go unnoticed until you try to send a message.
- **WebSocket does not replace HTTP** — for one-off request-response calls it is overkill; keeping thousands of open channels costs more in resources than serving a short HTTP request.
