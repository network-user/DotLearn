# Клиент и сервер: кто кого вызывает и зачем такое разделение

## Суть

Это разделение ролей: инициатор (клиент) шлёт запрос, а поставщик ресурса (сервер) этот запрос обрабатывает и возвращает ответ. Вся логика хранения и контроля доступа собирается на одной стороне, что упрощает управление и рост системы.

## Что ответить на собесе

Клиент-серверная модель разносит обязанности между двумя сторонами. Клиент - инициатор: он формирует запрос и ждёт результат (например, браузер открывает страницу). Сервер пассивно слушает, принимает входящие запросы, делает работу и отдаёт ответ. Главный выигрыш такого подхода в том, что данные и правила доступа централизованы на сервере: их проще обновлять, защищать и масштабировать под нагрузку, потому что один сервер обслуживает множество клиентов одновременно.

## Как это работает

Минимальный пример на сокетах. Сервер открывает слушающий сокет, принимает подключения в цикле и каждому отвечает строкой.

**Сервер:**

```python
import socket

server_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server_socket.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
server_socket.bind(('localhost', 12345))
server_socket.listen()

print("Сервер ожидает подключения...")

while True:
    client_socket, address = server_socket.accept()
    print(f"Подключен клиент: {address}")
    client_socket.sendall(b"Privet ot servera")
    client_socket.close()
```

**Клиент:**

```python
import socket

client_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
client_socket.connect(('localhost', 12345))

message = client_socket.recv(1024)
print(f"Сообщение от сервера: {message.decode()}")

client_socket.close()
```

Сервер пассивно ждёт в `accept()`, пока кто-то не подключится, затем отдаёт данные и закрывает соединение. Клиент инициирует `connect()` и читает ответ через `recv()`.

## Подводные камни

- Один сервер - точка отказа. Если он упал, недоступны все клиенты; в проде ставят несколько инстансов за балансировщиком.
- `recv(1024)` читает *до* 1024 байт, а не ровно столько. Для длинных сообщений нужно читать в цикле, пока не получишь весь ожидаемый объём.
- Без `SO_REUSEADDR` после перезапуска сервера порт может остаться занятым из-за состояния `TIME_WAIT`.
- "Толстый клиент" с бизнес-логикой на своей стороне ломает централизацию: критичные проверки (доступ, валидацию) всё равно дублируют на сервере, клиенту доверять нельзя.
