# HTTP caching: which headers let the browser decide when not to redownload a file

## In short

Caching at the HTTP level is controlled by response headers: `Cache-Control` and `Expires` set the freshness window, while `ETag` and `Last-Modified` let you cheaply check whether the resource has changed. Together they spare the client from re-downloading unchanged files and take load off the server.

## How to answer in an interview

There are two mechanisms. The first is freshness: `Cache-Control: max-age=N` (or the deprecated `Expires`) says how many seconds the copy may be served from the cache without contacting the server. The second is validation: when the freshness window has expired, the client does not download the file blindly but makes a conditional request with `If-None-Match` (based on `ETag`) or `If-Modified-Since` (based on `Last-Modified`). If the resource has not changed, the server replies `304 Not Modified` with an empty body, and the old copy is used. `Cache-Control` is more flexible than `Expires` because it sets a relative time and the directives `no-cache`, `no-store`, `public`, and `private`.

The main headers and their roles:

- **Cache-Control** - the primary mechanism. `max-age=3600` is freshness in seconds; `no-cache` means it may be cached but must be revalidated with the server before serving; `no-store` means do not store at all; `public` means it may be cached on proxies too; `private` means only in the specific user's browser.
- **Expires** - an absolute expiration date. Less flexible and loses to `Cache-Control` in a conflict.
- **ETag** - a fingerprint of the resource version; next time the client sends it in `If-None-Match`.
- **Last-Modified** - the modification date; the client sends it back in `If-Modified-Since`.

A server response with caching configured:

```http
HTTP/1.1 200 OK
Cache-Control: max-age=3600, public
ETag: "abc123"
Last-Modified: Tue, 20 Oct 2026 07:28:00 GMT
Content-Type: image/png
Content-Length: 12345

[binary file data]
```

Until the hour has passed, the browser serves the image from the cache silently. After that, it sends a conditional request:

```http
GET /logo.png HTTP/1.1
Host: www.example.com
If-None-Match: "abc123"
```

If the file is unchanged, a `304 Not Modified` with no body comes back, and the old copy is deemed current - traffic is nearly zero.

## Pitfalls

- `no-cache` does not forbid caching: it merely requires revalidation before use. Forbidding storage is `no-store`.
- An aggressive `max-age` without versioning file names leads users to see an old version for weeks. The fix is a hash in the name (`app.4f2a.js`) plus a long cache.
- Weak and strong `ETag`s behave differently, and an `ETag` mismatch between nodes behind a load balancer breaks conditional requests.
- `Expires` depends on the client's and server's clocks being in sync; a skew gives incorrect freshness. That is why a relative `max-age` is preferred.
- `Vary` is also part of caching: without it the cache may serve a compressed or wrong-language variant to the wrong client.
