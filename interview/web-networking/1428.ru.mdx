# Версионирование API: какие подходы существуют и где их граница?

## Суть

Версионирование позволяет развивать API и при этом не обрывать старых клиентов. Версию указывают в пути URL, в HTTP-заголовке или в параметре запроса; выбор сводится к компромиссу между наглядностью адреса и гибкостью переговоров о формате.

## Что ответить на собесе

Я выделяю три способа размещения версии, а ключевая идея одна: любую несовместимую правку выносим в новую версию, совместимые меняем без неё.

- Версия в пути (`/v1/resource`): видна сразу, легко кэшируется и маршрутизируется, но при больших расхождениях ведёт к дублированию обработчиков.
- Версия в заголовке (`Accept: application/vnd.example.v1+json`): «правильный» для REST подход через content negotiation, URL чистый, зато версия не видна в адресе и хуже отлаживается из браузера.
- Версия в query-параметре (`?version=1`): гибко, но параметр легко потерять, и он засоряет логику маршрутизации.

На практике для публичных API чаще берут версию в пути ради предсказуемости.

## Как это работает

**Версия в URL.** Каждая версия — отдельный набор маршрутов.

```python
from flask import Flask, jsonify

app = Flask(__name__)

@app.route("/v1/resource")
def resource_v1():
    return jsonify({"message": "resource, схема версии 1"})

@app.route("/v2/resource")
def resource_v2():
    return jsonify({"message": "resource, схема версии 2", "extra": True})

if __name__ == "__main__":
    app.run(debug=True)
```

**Версия в заголовке.** Клиент сообщает желаемую версию через `Accept`, а сервер выбирает обработчик по содержимому заголовка. URL при этом один и тот же (`/resource`), что считается более «честным» REST: меняется представление ресурса, а не его идентификатор.

**Версия в параметре.** Клиент шлёт `/resource?version=2`, сервер читает параметр и ветвит логику. Удобно для быстрых экспериментов, но параметр необязателен и легко выпадает.

Главный критерий выбора — совместимость. Добавление необязательного поля совместимо и новой версии не требует. Удаление поля, переименование, смена типа или изменение семантики — несовместимы, и их выносят в следующую версию.

## Подводные камни

- **Семантика «major», а не «любая правка».** Плодить версии на каждое мелкое изменение вредно. Новая версия нужна только под ломающие изменения; обратносовместимые катятся в текущую.
- **Кэширование и заголовок.** При версии в `Accept` обязательно отдавайте `Vary: Accept`, иначе прокси и CDN отдадут одной версии ответ, закэшированный для другой.
- **Дублирование кода в URL-подходе.** Если v1 и v2 разъезжаются сильно, легко получить два почти одинаковых дерева обработчиков; выносите общую логику в сервисный слой, а в версиях держите только различия.
- **Срок жизни старых версий.** Без явной политики депрекации и дат отключения старые версии живут вечно и копят техдолг. Сообщайте об устаревании заранее, например заголовком `Deprecation`.
- **Версия по умолчанию.** Если клиент не указал версию (особенно в header/query-подходах), нужно заранее решить: подставлять последнюю стабильную или отвечать ошибкой. Молчаливый переход на «свежую» версию ломает интеграции.
