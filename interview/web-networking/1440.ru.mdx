# WebSocket: постоянный двусторонний канал вместо вечного опроса сервера

## Суть

WebSocket держит один постоянный полнодуплексный канал поверх TCP, по которому клиент и сервер шлют данные в обе стороны без повторных рукопожатий. Это убирает накладные расходы HTTP-опроса и даёт реальное время: чаты, игры, биржевые котировки.

## Что ответить на собесе

WebSocket — это протокол двунаправленной связи поверх одного TCP-соединения.

- Соединение начинается как обычный HTTP-запрос с заголовком `Upgrade: websocket`, сервер отвечает `101 Switching Protocols`, после чего канал переключается в дуплексный режим и остаётся открытым.
- В отличие от модели «запрос — ответ» HTTP, сервер может слать данные клиенту сам, не дожидаясь запроса.
- Поэтому WebSocket выбирают там, где нужен мгновенный двусторонний обмен, а классический polling неэффективен.

Жизненный цикл соединения:

1. **Handshake (рукопожатие)** — клиент шлёт HTTP/1.1-запрос с заголовками `Upgrade: websocket` и `Connection: Upgrade`. Сервер подтверждает переход ответом `101 Switching Protocols`.
2. **Дуплексный обмен** — после апгрейда обе стороны передают сообщения в виде фреймов в любой момент; сервер не ждёт запроса, чтобы отправить данные.
3. **Постоянство** — канал живёт, пока одна из сторон его не закроет, поэтому переустанавливать соединение на каждое сообщение не нужно.

Клиентский код на JavaScript:

```javascript
const socket = new WebSocket('wss://example.com/socket');

socket.onopen = (event) => {
  console.log('Соединение установлено');
  socket.send('Привет, сервер!');
};

socket.onmessage = (event) => {
  console.log('Сообщение от сервера:', event.data);
};

socket.onclose = (event) => {
  console.log('Соединение закрыто, код:', event.code);
};

socket.onerror = (error) => {
  console.log('Ошибка WebSocket:', error);
};
```

Где это уместно: чаты и мессенджеры, онлайн-игры, биржевые и торговые платформы, IoT — везде, где данные обновляются «сию секунду» и опрос по таймеру был бы расточителен.

## Подводные камни

- **`ws://` против `wss://`** — незашифрованный `ws://` шлёт данные открытым текстом; в продакшене используют `wss://` (TLS), иначе трафик читается посредником.
- **Нет автопереподключения** — браузерный `WebSocket` не восстанавливает разрыв сам; логику reconnect с backoff пишут руками.
- **Прокси и балансировщики** — не все промежуточные узлы пропускают долгоживущие соединения; нужны sticky-сессии и корректная настройка таймаутов.
- **Heartbeat обязателен** — без ping/pong «мёртвое» соединение можно не заметить, пока не попытаешься отправить сообщение.
- **WebSocket не заменяет HTTP** — для разовых запросов «запрос — ответ» он избыточен; держать тысячи открытых каналов дороже по ресурсам, чем обслужить короткий HTTP-запрос.
