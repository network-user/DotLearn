# Controlling the HTTP cache: which headers decide whether to store a response and how fresh it is

## In short

Four headers govern HTTP cache behavior. `Cache-Control` describes the policy (who may store it and for how long), `Expires` gives an absolute moment of staleness, and `ETag` and `Last-Modified` let you check whether a copy is outdated without re-downloading it in full.

## How to answer in an interview

The HTTP cache is controlled through response headers. The main one is `Cache-Control`, with the directives `max-age`, `no-cache`, `no-store`, `public`, `private`. The older `Expires` sets an absolute expiration date and exists for HTTP/1.0 compatibility. To validate a stale copy there are two validators: `ETag` (a fingerprint of the resource version) and `Last-Modified` (the change date). The client re-checks with conditional requests `If-None-Match` and `If-Modified-Since`, and if nothing has changed, the server replies `304 Not Modified` with no body. This saves bandwidth, time, and server load.

The headers fall into two groups: freshness policy and validators.

**Freshness policy:**

- **`Cache-Control`**: the main mechanism. Directives:
  - `max-age=N`: how many seconds the copy is considered fresh.
  - `no-cache`: caching is allowed, but you must check with the server before using it.
  - `no-store`: do not store the response at all.
  - `public`: any cache may store it, including proxies.
  - `private`: store only in a specific client's cache (the browser).
- **`Expires`** gives an absolute expiration date, a legacy of HTTP/1.0. In HTTP/1.1, on conflict, `Cache-Control: max-age` takes precedence.

```http
Cache-Control: public, max-age=3600
Expires: Wed, 21 Oct 2026 07:28:00 GMT
```

**Validators (freshness check):**

- **`ETag`** (a fingerprint of the resource version) is sent by the server with the response, and the client returns it in `If-None-Match`. A match means `304 Not Modified`.

```http
ETag: "abc123"
```

- **`Last-Modified`** is the date of the last change. The client sends `If-Modified-Since`, and if the resource has not changed since then, the response is `304` again.

```http
Last-Modified: Wed, 21 Oct 2026 07:28:00 GMT
```

A fresh copy is served from the cache instantly; a stale one passes a cheap validation, and on `304` no body is transferred.

## Pitfalls

- **`no-cache` ≠ "do not cache"**: it means "cache, but always revalidate". Forbidding storage is `no-store`.
- **`ETag` is more precise than `Last-Modified`.** The date has one-second granularity and lies on a "change with no content change"; `ETag` catches any byte-level edit.
- **Weak vs strong ETags.** `W/"abc"` (weak) allows semantically equivalent versions; a strong one requires a byte-for-byte match.
- **Conflict between `Expires` and `max-age`.** In HTTP/1.1 `Cache-Control: max-age` wins, so set both at once with care.
- **`private` on a shared proxy**: marking a response `public` by mistake can serve someone else's personal data from the proxy cache to another user.
