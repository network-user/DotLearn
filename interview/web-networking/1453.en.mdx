# REST as a style and RESTful as an implementation: what exactly the difference is

## In short

REST is an architectural style with a set of constraints for designing networked applications. A RESTful API is a concrete API that follows those constraints: it works with resources through URIs and standard HTTP methods. The difference is like the one between "principles" and "following them in practice."

## How to answer in an interview

REST (Representational State Transfer) is a set of architectural constraints: client-server, statelessness, cacheability, a uniform interface, and layering. The word "RESTful" means that a concrete API is designed according to these constraints. So REST is the theory and the style, while RESTful is an adjective describing an actual implementation. In practice a REST API addresses resources through URIs (`/users`, `/users/42`) and operates on them via HTTP methods: GET reads, POST creates, PUT/PATCH updates, DELETE removes. The key idea of statelessness: each request is self-contained, the server does not remember the client between requests, which simplifies scaling.

## How it works

The REST constraints that make an API "RESTful":

- **Client-server:** the interface and data storage are separated and evolve independently.
- **Stateless:** each request carries everything it needs (token, parameters); the server keeps no session state between requests.
- **Cacheability:** responses are explicitly marked as cacheable or not, so the client does not hit the server in vain.
- **Uniform interface:** consistent resource addressing and predictable method semantics.
- **Layering:** proxies, load balancers, and caches can sit between client and server without breaking the contract.

For a "users" resource a typical set of operations looks like this:

```http
GET    /users        # list of users
GET    /users/42     # a single user by id
POST   /users        # create a user
PUT    /users/42     # replace a user entirely
DELETE /users/42     # delete a user
```

The same URI denotes the same resource, and the method specifies the action on it. This is the uniform interface: the client does not need to learn separate "commands" for each entity.

## Pitfalls

- "REST" and "RESTful" are used as synonyms in everyday speech, but strictly: REST is the style, RESTful is a property of the implementation. Interviewers like to test this distinction.
- Verbs in a URI (`/getUser`, `/createUser`) are an antipattern: the action should be specified by the HTTP method, and the URI should name the resource with a noun.
- Statelessness is often violated by in-memory server sessions - then the API stops scaling horizontally without sticky sessions.
- PUT usually replaces a resource entirely and is idempotent, while PATCH is a partial update; confusing the two leads to lost fields.
- "RESTful" is not equal to "any JSON over HTTP": without honoring the constraints it is just an HTTP API, not REST.
