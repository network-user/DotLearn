# REST, GraphQL, gRPC и SOAP: чем стили API отличаются на практике

## Суть

Это четыре подхода к проектированию веб-API с разными приоритетами: REST - простота поверх HTTP, GraphQL - точная выборка данных под клиента, gRPC - производительность через HTTP/2 и бинарный формат, SOAP - строгий XML-контракт с акцентом на безопасность.

## Что ответить на собесе

Четыре подхода с разными приоритетами:

- REST организует API вокруг ресурсов и стандартных HTTP-методов (GET, POST, PUT, DELETE), он stateless и кэшируем - дефолт для большинства публичных API за счёт простоты.
- GraphQL даёт клиенту запросить через одну точку входа ровно те поля, что нужны, что убирает over-fetching и under-fetching, когда разным клиентам нужны разные срезы данных.
- gRPC построен поверх HTTP/2 и Protocol Buffers: бинарный формат и мультиплексирование дают высокую производительность и стриминг, поэтому его берут для связи между микросервисами.
- SOAP - старый XML-протокол со строгой спецификацией и WS-Security, его держат в корпоративных и банковских системах, где важны жёсткий контракт и встроенная безопасность.

## Как это работает

Различия удобно держать в одной таблице.

| Стиль | Транспорт | Формат | Сильная сторона | Где уместен |
| --- | --- | --- | --- | --- |
| REST | HTTP/1.1 | JSON | простота, кэширование | публичные API, CRUD |
| GraphQL | HTTP | JSON | точная выборка полей | разные клиенты, агрегация |
| gRPC | HTTP/2 | Protobuf (бинарь) | скорость, стриминг | межсервисная связь |
| SOAP | HTTP/SMTP | XML | строгий контракт, WS-Security | корпоративные системы |

Один и тот же сценарий "получить книгу" в трёх стилях:

```text
REST:    GET /books/42

GraphQL: query { book(id: 42) { title author } }

gRPC:    rpc GetBook(BookRequest) returns (Book)
```

REST возвращает заранее определённый объект целиком, GraphQL - только запрошенные поля `title` и `author`, gRPC вызывает строго типизированный метод, описанный в `.proto`-схеме.

## Подводные камни

- REST страдает от over-fetching (приходит лишнее) и under-fetching (нужны несколько запросов на один экран) - именно эту боль решает GraphQL.
- GraphQL переносит сложность на сервер: легко сформировать тяжёлый или вложенный запрос, поэтому нужны ограничение глубины, стоимость запросов и защита от N+1.
- gRPC плохо работает напрямую из браузера (нужен gRPC-Web и прокси) и сложнее отлаживается - бинарь не прочитать глазами, в отличие от JSON.
- SOAP многословен и тяжёл: XML-конверты и WSDL увеличивают объём и порог входа, для простого CRUD это избыточно.
