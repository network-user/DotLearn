# How to lock cookies down against theft and forgery: flags and practices

## In short

Cookie security rests on three flags: Secure restricts transmission to HTTPS only, HttpOnly hides the cookie from JavaScript, SameSite cuts cross-site sending. On top of that, add a short lifetime and not storing sensitive data in the clear.

## How to answer in an interview

I'd describe the layers of protection. First, the `HttpOnly` flag: it blocks access to the cookie from `document.cookie`, so even under XSS a script cannot steal the session identifier. Second, `Secure`: the cookie is sent only over HTTPS, so it cannot be intercepted on an open channel. Third, `SameSite` (`Lax` or `Strict`): the browser does not attach the cookie to requests from other sites, which hits CSRF. Additionally, I limit the lifetime via `Max-Age`/`Expires`, do not put raw secrets in the cookie, and instead store an opaque session identifier tied to server-side state. These measures complement each other rather than replace one another.

## How it works

In Flask all flags are set through the `set_cookie` parameters. They can and should be combined in a single call.

```python
response.set_cookie(
    "session_id",
    value="abc123",
    secure=True,
    httponly=True,
    samesite="Lax",
    max_age=3600,
)
```

What each parameter does:

- `secure=True`: the cookie will not be sent over plain HTTP, only over HTTPS.
- `httponly=True`: JavaScript cannot see the cookie, which neutralizes session theft via XSS.
- `samesite="Lax"`: the cookie is attached to navigation within the site, but not to background cross-site requests (protection against CSRF). `Strict` is even tougher, `None` requires a mandatory `Secure`.
- `max_age=3600`: the cookie lives for an hour; the shorter the window, the less use a stolen value has.

If you do need to put data in the cookie rather than a reference to a server-side session, encrypt it:

```python
from cryptography.fernet import Fernet

key = Fernet.generate_key()
cipher = Fernet(key)
token = cipher.encrypt(b"user=42;role=admin")
response.set_cookie("payload", value=token.decode(), secure=True, httponly=True)
```

Without the key the contents cannot be read, and Fernet will detect tampering at decryption.

## Pitfalls

- `HttpOnly` protects against a script reading the cookie, but not against the XSS vulnerability itself and not against CSRF. These are different vectors, each closed by its own flag.
- Modern browsers simply drop `SameSite=None` without `Secure`: these flags work as a pair.
- `Secure` is useless if part of the site is reachable over HTTP, since the cookie will leak in plain text on the first such request.
- Encrypting the contents does not remove the need for the flags: a cookie stolen over HTTP or via JS will be replayed as-is by the attacker, without decrypting it (a replay attack).
- An overly long `Max-Age` turns a stolen session into a long-running problem; for important operations, re-authentication is added.
