# Как закрыть cookie от кражи и подделки: флаги и практики

## Суть

Безопасность cookie держится на трёх флагах: Secure ограничивает передачу только HTTPS, HttpOnly прячет cookie от JavaScript, SameSite режет межсайтовую отправку. Сверху — короткий срок жизни и отказ от хранения чувствительных данных в открытом виде.

## Что ответить на собесе

Расскажу про слои защиты, которые дополняют друг друга, а не заменяют:

- `HttpOnly` закрывает доступ к cookie из `document.cookie`, поэтому даже при XSS скрипт не утащит идентификатор сессии.
- `Secure` — cookie уходит только по HTTPS, и его не перехватить на открытом канале.
- `SameSite` (`Lax` или `Strict`) — браузер не прикладывает cookie к запросам с чужих сайтов, что бьёт по CSRF.
- Дополнительно ограничиваю время жизни через `Max-Age`/`Expires` и храню непрозрачный идентификатор сессии вместо сырых секретов.

## Как это работает

Во Flask все флаги задаются параметрами `set_cookie`. Их можно и нужно комбинировать в одном вызове.

```python
response.set_cookie(
    "session_id",
    value="abc123",
    secure=True,
    httponly=True,
    samesite="Lax",
    max_age=3600,
)
```

Что делает каждый параметр:

- `secure=True` — cookie не отправится по обычному HTTP, только по HTTPS.
- `httponly=True` — JavaScript не видит cookie, что нейтрализует кражу сессии через XSS.
- `samesite="Lax"` — cookie прикладывается к навигации внутри сайта, но не к фоновым межсайтовым запросам (защита от CSRF). `Strict` ещё жёстче, `None` требует обязательного `Secure`.
- `max_age=3600` — cookie живёт час; чем короче окно, тем меньше пользы от украденного значения.

Если в cookie всё же нужно положить данные, а не ссылку на серверную сессию, их шифруют:

```python
from cryptography.fernet import Fernet

key = Fernet.generate_key()
cipher = Fernet(key)
token = cipher.encrypt(b"user=42;role=admin")
response.set_cookie("payload", value=token.decode(), secure=True, httponly=True)
```

Без ключа содержимое прочитать нельзя, а подмену Fernet обнаружит при расшифровке.

## Подводные камни

- `HttpOnly` защищает от чтения cookie скриптом, но не от самой XSS-уязвимости и не от CSRF — это разные векторы, каждый закрывается своим флагом.
- `SameSite=None` без `Secure` современные браузеры просто отбрасывают: эти флаги работают в паре.
- `Secure` бесполезен, если часть сайта доступна по HTTP — cookie успеет уйти открытым текстом при первом таком запросе.
- Шифрование содержимого не отменяет необходимость флагов: украденную по HTTP или через JS cookie злоумышленник переиграет как есть, не расшифровывая (replay-атака).
- Слишком длинный `Max-Age` превращает украденную сессию в долгоиграющую проблему; для важных операций добавляют повторную аутентификацию.
