# RPC и gRPC: чем удалённый вызов от Google сильнее классической схемы

## Суть

RPC прячет сеть за фасадом обычного вызова функции: код «дёргает» процедуру на чужой машине так, будто она лежит рядом. gRPC - реализация этой идеи от Google поверх HTTP/2 с бинарной сериализацией Protocol Buffers и встроенными стримами, отчего работает быстрее и кросс-языково.

## Что ответить на собесе

RPC (Remote Procedure Call) - это стиль межсервисного общения, где вызов кода на удалённом сервере выглядит для разработчика как локальный: аргументы сериализуются, летят по сети, сервер выполняет процедуру и возвращает результат. gRPC - современная реализация RPC, дающая меньшую латентность, меньше трафика и строгий контракт из коробки:

- Транспорт на HTTP/2 даёт мультиплексирование и двунаправленные стримы.
- Сообщения кодируются в компактный бинарный protobuf вместо текстового JSON.
- Контракт описывается в `.proto` и генерирует клиент/сервер сразу под десяток языков.

Классический RPC проходит через стабы (заглушки). На клиенте лежит сгенерированный стаб с теми же сигнатурами, что и на сервере. Шаги одного вызова:

1. Клиент вызывает метод стаба как локальную функцию.
2. Аргументы сериализуются (marshalling) в сетевой формат.
3. Данные уходят на сервер по выбранному транспорту.
4. Серверный стаб десериализует запрос, выполняет реальную процедуру, сериализует ответ.
5. Результат возвращается клиенту и десериализуется обратно в объект.

gRPC надстраивает над этой схемой:

- **HTTP/2:** мультиплексирование запросов в одном соединении и серверный push без блокировок.
- **Protocol Buffers:** бинарный формат, который парсится быстрее и весит меньше JSON/XML.
- **Кодогенерация:** из `.proto` генерируются типизированные клиенты под разные языки.
- **Четыре режима вызова:** unary, server-streaming, client-streaming и bidi-streaming.

Контракт описывают в `.proto`:

```proto
syntax = "proto3";

service ChatService {
  rpc SendMessage (MessageRequest) returns (MessageResponse);
  rpc ChatStream (stream MessageRequest) returns (stream MessageResponse);
}

message MessageRequest {
  string user = 1;
  string text = 2;
}

message MessageResponse {
  string status = 1;
}
```

Здесь `SendMessage` - это разовый (unary) вызов, а `ChatStream` с двумя `stream` - двунаправленный поток для живого чата.

## Подводные камни

- **Браузер напрямую не говорит на gRPC:** нужен gRPC-Web с прокси, потому что обычный JS не управляет фреймами HTTP/2.
- **Бинарный формат непрозрачен:** protobuf не прочитать глазами в логах, отладка сложнее, чем у JSON-REST.
- **Эволюция схемы требует дисциплины:** номера полей в `.proto` нельзя переиспользовать; удалённые поля помечают `reserved`, иначе ломается совместимость.
- **Стрим - не магия:** долгоживущие потоки требуют ручной обработки переподключений, таймаутов и обратного давления.
- **RPC создаёт иллюзию локального вызова**, но сеть всё равно может упасть; ошибки и латентность нельзя игнорировать «потому что выглядит как функция».
