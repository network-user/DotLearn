# How do you catch webhooks on a local machine during development?

## In short

A local server is not reachable from the internet because it has no public IP. A tunnel solves this: a tool like ngrok or localtunnel hands you a temporary public URL and proxies every request arriving at it straight to your `localhost`. That is enough to debug webhooks without deploying to a remote server.

## How to answer in an interview

"A webhook is an outbound HTTP request that a third-party service sends to my URL when an event occurs. For it to reach my machine, I bring up a tunnel: I run ngrok on the same port as the local server, get a public address like `https://abcd.ngrok.io`, and register it as the webhook URL in the external service's settings. All requests flow to my `localhost`, and in the ngrok web panel I see every request and can replay it — debugging takes minutes."

## How it works

A webhook is a push notification over HTTP: instead of the client polling the service, the service itself sends a request to a predefined URL when an event happens (a new payment, a push to a repository, and so on).

The catch is that a home or office machine usually sits behind NAT with a private address — it cannot be reached from outside. A tunnel builds a bridge: it opens an outbound connection to a public relay, and the relay forwards incoming requests back through it to your port.

Running it through ngrok:

1. Start your handler app, for example on `http://localhost:5000`.
2. Open a tunnel to that port:

```bash
ngrok http 5000
```

3. Grab the public URL it issues, something like `https://abcd1234.ngrok.io`.
4. Register that address as the webhook endpoint in the external service's settings.

The ngrok panel at `http://127.0.0.1:4040` shows every incoming request, and any of them can be replayed — there is no need to reproduce the real event.

An alternative is localtunnel:

```bash
npm install -g localtunnel
lt --port 5000
```

The command issues an address like `https://abcd1234.loca.lt`, which is registered in the external service the same way.

## Pitfalls

- The free public URL changes on every restart, and the webhook has to be rewritten each time; for a stable address you need a paid plan or a reserved domain.
- A tunnel exposes your local port to the outside — do not keep it up longer than necessary and do not expose sensitive endpoints.
- Many providers sign webhooks with a secret: verify the signature, otherwise junk from strangers will hit your public URL.
- If the handler responds slowly or with a non-2xx code, the sender will retry — watch your timeouts and idempotency.
- Some services require specifically an HTTPS address for the webhook, so a plain http tunnel may not work.
