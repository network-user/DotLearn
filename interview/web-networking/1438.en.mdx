# Who you are versus what you may do: separating authentication and authorization

## In short

Authentication verifies identity ("who are you?"), authorization determines permissions ("what are you allowed to do?"). These are two sequential steps: first the system makes sure the user is who they claim to be, and only then decides what to let them access.

## How to answer in an interview

I'd put it briefly: authentication answers the question "who are you", authorization — "what may you do". First the user proves their identity, for example with a login and password, a token, or biometrics. That is authentication. Then the system looks at their role or set of permissions and decides which actions and resources are open to them — that is authorization. The order matters: authorization always comes after successful authentication; you cannot check the permissions of someone whose identity we have not yet established. At the HTTP level this shows up in the codes: 401 Unauthorized means "not authenticated", while 403 Forbidden means "identity is clear, but no permission".

## How it works

Authentication is comparing the presented data with what the system stores. A simplified password-check example:

```python
def authenticate(username, password, store):
    return store.get(username) == password

users = {"alice": "pw123", "bob": "secret"}
print(authenticate("alice", "pw123", users))
```

Authorization works with an already established identity and operates on roles or permissions:

```python
def permissions_for(role):
    table = {
        "admin": ["manage_users", "view_reports"],
        "user": ["view_profile"],
    }
    return table.get(role, [])

print(permissions_for("admin"))
```

How they tie together in a real application: middleware first authenticates the request (validating a session or token), then checks whether the action is allowed for the current role. If there is no identity — they respond 401; if there is an identity but no permission — 403.

## Pitfalls

- The main mistake is checking permissions before identity, or substituting one for the other entirely. Authorization without authentication is meaningless.
- 401 and 403 get confused: 401 — "log in / authenticate, we do not know you", 403 — "we know you, but you may not come here". Despite its name, 401 is about authentication.
- Storing passwords in the clear, as in the teaching example above, is not acceptable: in reality they are salted and hashed (bcrypt, argon2).
- Authorization must not be left only on the frontend: a hidden button does not protect the endpoint, the permission check is mandatorily duplicated on the server.
- Roles are not the only model: for fine-grained control, object-level permissions are used (ABAC, checking the resource owner), otherwise a user with a role will see someone else's data.
