# Which set of HTTP methods do clients use and what is each responsible for

## In short

An HTTP method (verb) tells the server which action the client wants to perform on a resource. The basic set is GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS, ranging from reading data to deleting it and requesting metadata.

## How to answer in an interview

I'd put it this way: the request method states the client's intent. GET reads a resource and changes nothing, POST creates a new entity or triggers an action, PUT replaces a resource entirely, PATCH edits only individual fields, DELETE removes. HEAD (like GET, but the server returns only headers, no body) and OPTIONS (to find out which methods are allowed for a resource at all) stand apart. It is important to keep the methods' properties in mind: GET, HEAD, PUT, and DELETE are idempotent, GET and HEAD are also safe (they do not change state), while POST and PATCH are generally neither.

## How it works

The method is the first word in the request's start line. With the `requests` library, each verb has its own function.

```python
import requests

resp = requests.get("https://api.example.com/items")
print(resp.status_code, resp.json())
```

Creating a new record via POST: the body goes to the server, usually as JSON.

```python
import requests

payload = {"name": "widget", "price": 10}
resp = requests.post("https://api.example.com/items", json=payload)
print(resp.status_code)
```

Full replacement versus partial update - that is exactly the difference between PUT and PATCH:

```python
import requests

requests.put("https://api.example.com/items/1", json={"name": "x", "price": 20})
requests.patch("https://api.example.com/items/1", json={"price": 25})
```

PUT requires you to send the resource in full, otherwise the omitted fields may be overwritten. PATCH changes only what you passed.

HEAD and OPTIONS are utility methods. HEAD checks the existence or metadata of a resource (size, date) without downloading the body, and OPTIONS returns the list of allowed methods in the `Allow` header.

```python
import requests

resp = requests.options("https://api.example.com/items")
print(resp.headers.get("Allow"))
```

## Pitfalls

- GET must not change server state. If a "fetch" deletes something or charges money - that is a design flaw, such operations are done via POST.
- Idempotency is not the same as safety: DELETE is idempotent (repeating gives the same outcome: the resource is gone), but it changes state, so it is not safe.
- POST is not idempotent: resubmitting a form will create a duplicate. To guard against double charges, idempotency keys are used.
- Do not confuse PUT and PATCH: an erroneous PUT with an incomplete body can overwrite existing fields with default values.
- A GET with a request body is not formally forbidden, but many proxies and servers ignore it, so parameters are passed via the query string.
