# REST, GraphQL, gRPC, and SOAP: how the API styles differ in practice

## In short

These are four approaches to designing web APIs with different priorities: REST is simplicity on top of HTTP, GraphQL is precise data selection tailored to the client, gRPC is performance through HTTP/2 and a binary format, and SOAP is a strict XML contract with an emphasis on security.

## How to answer in an interview

REST organizes the API around resources and standard HTTP methods (GET, POST, PUT, DELETE); it is stateless and cacheable, which makes it the default for most public APIs thanks to its simplicity. GraphQL lets the client request exactly the fields it needs through a single entry point, eliminating over-fetching and under-fetching when different clients need different slices of data. gRPC is built on top of HTTP/2 and Protocol Buffers: the binary format and multiplexing give high performance and streaming, which is why it is chosen for internal communication between microservices. SOAP is an old XML protocol with a strict specification and WS-Security; it is kept around in enterprise and banking systems where a rigid contract and built-in security matter.

## How it works

The differences are convenient to keep in a single table.

| Style   | Transport | Format            | Strength                     | Where it fits               |
| ------- | --------- | ----------------- | ---------------------------- | --------------------------- |
| REST    | HTTP/1.1  | JSON              | simplicity, caching          | public APIs, CRUD           |
| GraphQL | HTTP      | JSON              | precise field selection      | varied clients, aggregation |
| gRPC    | HTTP/2    | Protobuf (binary) | speed, streaming             | service-to-service          |
| SOAP    | HTTP/SMTP | XML               | strict contract, WS-Security | enterprise systems          |

The same "get a book" scenario in three styles:

```text
REST:    GET /books/42

GraphQL: query { book(id: 42) { title author } }

gRPC:    rpc GetBook(BookRequest) returns (Book)
```

REST returns a predefined object in full, GraphQL returns only the requested `title` and `author` fields, and gRPC calls a strictly typed method described in a `.proto` schema.

## Pitfalls

- REST suffers from over-fetching (you get unnecessary data) and under-fetching (you need several requests for one screen) - this is exactly the pain that GraphQL solves.
- GraphQL shifts the complexity to the server: it is easy to compose a heavy or deeply nested query, so you need depth limiting, query costing, and protection against the N+1 problem.
- gRPC works poorly directly from the browser (you need gRPC-Web and a proxy) and is harder to debug - you can't read binary by eye the way you can JSON.
- SOAP is verbose and heavy: XML envelopes and WSDL increase payload size and the barrier to entry, which is overkill for simple CRUD.
