# What do the groups of HTTP status codes mean and which are the key ones?

## In short

An HTTP status code is a three-digit number with which the server tells the client the outcome of processing a request. The first digit defines the class: 1xx — informational, 2xx — success, 3xx — redirection, 4xx — client error, 5xx — server error. From the code the client decides what to do next.

## How to answer in an interview

"Codes split into five groups by the first digit. 2xx means everything is fine: 200 OK, 201 Created when a resource is created, 204 No Content with no body. 3xx means another step is needed: 301 — moved permanently, 304 — not changed since last time. 4xx means the client is at fault: 400 — malformed request, 401 — authentication needed, 403 — no permission, 404 — resource not found. 5xx means something broke on the server: 500 — generic error, 502 — a bad response from upstream, 503 — service temporarily unavailable. The key distinction is that the client fixes 4xx, while 5xx is fixed on the backend."

## How it works

The code comes with every response and tells you how processing ended and how the client should react.

A breakdown by group:

- **1xx — informational**: the request was accepted, processing continues. Rarely seen in application code (`100 Continue`, `101 Switching Protocols`).
- **2xx — success**: `200 OK` — the request succeeded; `201 Created` — a new resource was created; `204 No Content` — success with no response body.
- **3xx — redirection**: `301 Moved Permanently` — the resource moved for good; `302 Found` — temporarily at a different address; `304 Not Modified` — unchanged since last time, you can take it from cache.
- **4xx — client error**: `400 Bad Request` — malformed request; `401 Unauthorized` — authentication needed; `403 Forbidden` — no permission; `404 Not Found` — the resource is gone.
- **5xx — server error**: `500 Internal Server Error` — a general failure; `502 Bad Gateway` — upstream returned junk; `503 Service Unavailable` — the server temporarily cannot cope.

A simple example exchange:

```http
GET /index.html HTTP/1.1
Host: www.example.com
```

The server's response:

```http
HTTP/1.1 200 OK
Content-Type: text/html

<html>
  <head><title>Example Page</title></head>
  <body><h1>Welcome to Example.com!</h1></body>
</html>
```

The client asked for a page, the server replied `200 OK` and returned the HTML.

## Pitfalls

- 401 and 403 get confused: 401 means "who are you, identify yourself", 403 means "I know who you are, but you may not come here".
- You must not blindly return 200 with `{"error": ...}` in the body for business errors — clients and monitoring rely on the code, so error handling and retries break.
- 5xx is usually safe to retry (especially 503 with `Retry-After`), whereas retrying 4xx is pointless — the request stays malformed.
- 304 works only together with conditional headers (`If-None-Match`/`ETag`); without them the cache will not kick in.
- A 302 for a permanent move hurts SEO and caches — for "forever" you need exactly a 301.
