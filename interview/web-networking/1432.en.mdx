# What HTTP methods exist and what is each responsible for?

## In short

An HTTP method (also called a verb) specifies which action the client wants to perform on a server resource. GET reads, POST creates, PUT replaces the whole thing, PATCH updates partially, DELETE removes. Some methods are safe and idempotent, which directly affects how they can be retried.

## How to answer in an interview

"There are a few core verbs. GET reads data and changes nothing, so it is safe and idempotent. POST sends data and usually creates a resource; calling it again produces a new object, so it is not idempotent. PUT replaces a resource entirely and is idempotent: no matter how many times you repeat it, the state is the same. PATCH changes only the fields you send. DELETE removes a resource and is also idempotent. There is also HEAD (like GET but without a body) and OPTIONS, which tells you which methods the server supports."

## How it works

The key properties of methods are _safety_ (the request does not change state) and _idempotency_ (repeating it produces the same result as a single call).

- **GET**: reads a resource. Safe and idempotent. The browser uses this method to fetch page HTML.
- **POST** sends data, usually creating a new resource. Not idempotent: two POSTs mean two objects. The data lives in the body.
- **PUT** places a resource at a known address, replacing it entirely or creating it. Idempotent.
- **PATCH** is a partial update: it touches only the specified fields and leaves the rest alone.
- **DELETE** removes a resource. Idempotent (deleting the same object again does not change the outcome).
- **HEAD**: like GET, but returns only headers without a body. Handy for checking existence and metadata.
- **OPTIONS** reports the list of supported methods and takes part in the CORS preflight.

Working with methods in Python via `requests`:

```python
import requests

base = "https://api.example.com"

response = requests.get(f"{base}/data")
print(response.status_code, response.json())

created = requests.post(f"{base}/data", json={"key": "value"})
print(created.status_code)

updated = requests.put(f"{base}/data/1", json={"key": "new_value"})
print(updated.status_code)

patched = requests.patch(f"{base}/data/1", json={"key": "patch"})
print(patched.status_code)

deleted = requests.delete(f"{base}/data/1")
print(deleted.status_code)
```

Each call sends the corresponding verb; from it the server understands the client's intent.

## Pitfalls

- GET must not change state: data in the query string is cached and logged, so hiding side effects there is dangerous.
- Confusing PUT and PATCH is a common mistake: PUT overwrites the whole resource, and a forgotten field becomes empty, whereas PATCH makes targeted edits.
- Idempotency is about the final state, not the response code: a second DELETE may return 404, but the server state no longer changes.
- POST is not idempotent, so retries without an idempotency key easily spawn duplicates.
- Not every server actually supports all methods, so before using PATCH/OPTIONS it is worth checking the API documentation.
