# Как устроен стек TCP/IP и кто за что в нём отвечает

## Суть

TCP/IP - это многоуровневое семейство протоколов, на котором держится интернет. Каждый уровень решает свою задачу: IP доставляет пакеты по адресам, а TCP поверх него обеспечивает надёжность - порядок, целостность и повторную отправку потерянного.

## Что ответить на собесе

TCP/IP - это стек из четырёх уровней. Данные на отправке инкапсулируются сверху вниз, на приёме разворачиваются снизу вверх.

- Прикладной — протоколы программ (HTTP, SMTP, FTP).
- Транспортный — два варианта доставки: TCP с гарантией порядка и надёжности или UDP без гарантий ради скорости.
- Сетевой — IP, который маршрутизирует пакеты между узлами по IP-адресам.
- Канальный — физическая передача через Ethernet или Wi-Fi.

## Как это работает

Уровни и их роли:

1. **Прикладной (Application):** протоколы конкретных программ - HTTP для веба, FTP для файлов, SMTP для почты.
2. **Транспортный (Transport):** здесь TCP и UDP. TCP открывает соединение, нумерует сегменты, проверяет контрольные суммы и переотправляет потерянное. UDP шлёт дейтаграммы без соединения и без контроля ошибок - быстрее, но без гарантий.
3. **Сетевой (Internet):** протокол IP маршрутизирует пакеты между узлами, а IP-адрес уникально определяет устройство.
4. **Канальный (Link):** физическая передача через Ethernet, Wi-Fi и прочие интерфейсы.

Путь данных при отправке по TCP - это инкапсуляция сверху вниз:

- **Приложение** (браузер) формирует запрос.
- **TCP** режет данные на сегменты, добавляет номера и контрольную сумму.
- **IP** дописывает заголовок с адресами отправителя и получателя и маршрутизирует пакет.
- **Канальный уровень** передаёт кадры по физической среде к следующему узлу.

На приёме всё разворачивается в обратном порядке: кадры поднимаются вверх, TCP пересобирает сегменты по номерам и отдаёт целостный поток приложению.

Простейшие TCP-сервер и клиент на Python:

```python
import socket

server_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server_socket.bind(('localhost', 8080))
server_socket.listen()

print("Сервер ожидает соединения...")

client_socket, address = server_socket.accept()
print(f"Соединение установлено с {address}")

data = client_socket.recv(1024).decode('utf-8')
print(f"Получено сообщение: {data}")

client_socket.send("Привет от сервера!".encode('utf-8'))

client_socket.close()
server_socket.close()
```

```python
import socket

client_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
client_socket.connect(('localhost', 8080))

client_socket.send("Привет от клиента!".encode('utf-8'))

response = client_socket.recv(1024).decode('utf-8')
print(f"Ответ от сервера: {response}")

client_socket.close()
```

Здесь `AF_INET` означает IPv4, а `SOCK_STREAM` - именно TCP. `bind()` и `listen()` готовят сервер к приёму, `connect()` открывает соединение со стороны клиента, `send()`/`recv()` обмениваются байтами, `close()` освобождает сокет.

## Подводные камни

- **TCP - это поток байтов, а не сообщений.** Один `send()` не равен одному `recv()`: данные могут склеиться или разбиться. Прикладной уровень сам отвечает за разбор границ сообщений.
- **`recv(1024)` возвращает до 1024 байт, а не ровно столько.** Полагаться на то, что всё придёт за один вызов, нельзя - в реальном коде читают в цикле.
- **Путать уровни OSI и TCP/IP.** В классической OSI семь уровней, в TCP/IP их объединяют до четырёх. На собесе уточняйте, о какой модели речь.
- **UDP живёт на том же транспортном уровне, что и TCP**, а не где-то ниже - это частая ошибка в ответах.
