# RPC and gRPC: how Google's remote call beats the classic scheme

## In short

RPC hides the network behind the facade of an ordinary function call: code "invokes" a procedure on someone else's machine as if it were right next door. gRPC is Google's implementation of this idea on top of HTTP/2 with binary Protocol Buffers serialization and built-in streams, which makes it faster and cross-language.

## How to answer in an interview

RPC (Remote Procedure Call) is a style of inter-service communication where invoking code on a remote server looks, to the developer, like a local call: arguments are serialized, fly over the network, the server runs the procedure and returns the result. gRPC is a modern RPC implementation: HTTP/2 transport gives multiplexing and bidirectional streams, messages are encoded into compact binary protobuf instead of text JSON, the contract is described in a `.proto` and generates client/server for a dozen languages at once. The result — lower latency, less traffic, and a strict contract out of the box.

Classic RPC goes through stubs. On the client sits a generated stub with the same signatures as the server's. The steps of a single call:

1. The client calls a stub method as if it were a local function.
2. The arguments are serialized (marshalling) into the network format.
3. The data goes to the server over the chosen transport.
4. The server stub deserializes the request, runs the real procedure, and serializes the response.
5. The result is returned to the client and deserialized back into an object.

gRPC builds on top of this scheme:

- **HTTP/2** — multiplexing requests over a single connection and server push without blocking.
- **Protocol Buffers** — a binary format that parses faster and weighs less than JSON/XML.
- **Code generation** — typed clients for various languages are generated from a `.proto`.
- **Four call modes** — unary, server-streaming, client-streaming, and bidi-streaming.

The contract is described in a `.proto`:

```proto
syntax = "proto3";

service ChatService {
  rpc SendMessage (MessageRequest) returns (MessageResponse);
  rpc ChatStream (stream MessageRequest) returns (stream MessageResponse);
}

message MessageRequest {
  string user = 1;
  string text = 2;
}

message MessageResponse {
  string status = 1;
}
```

Here `SendMessage` is a one-off (unary) call, while `ChatStream` with two `stream`s is a bidirectional stream for a live chat.

## Pitfalls

- **The browser does not speak gRPC directly** — you need gRPC-Web with a proxy, because plain JS does not manage HTTP/2 frames.
- **The binary format is opaque** — protobuf cannot be read by eye in logs, debugging is harder than with JSON-REST.
- **Schema evolution requires discipline** — field numbers in a `.proto` must not be reused; removed fields are marked `reserved`, otherwise compatibility breaks.
- **Streaming is not magic** — long-lived streams require manual handling of reconnects, timeouts, and backpressure.
- **RPC creates the illusion of a local call**, but the network can still fail; errors and latency must not be ignored "because it looks like a function".
