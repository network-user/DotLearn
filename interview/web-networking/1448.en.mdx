# UDP: fast, connectionless delivery with no guarantees

## In short

UDP is a transport protocol that sends data as independent datagrams without establishing a connection and without confirming delivery. It trades reliability for minimal latency, which makes it a good fit where losing a packet is tolerable but extra milliseconds are not.

## How to answer in an interview

UDP is a connectionless transport protocol. Unlike TCP, it does not establish a connection before sending, does not guarantee delivery, ordering, or freedom from duplicates, and does not manage flow or congestion. In exchange it has almost no overhead, so it delivers low latency. It is chosen for streaming video and audio, online games, VoIP, and DNS queries - cases where delivering fresh data quickly matters more than reliably delivering stale data. If an application needs reliability on top of UDP, it implements that itself.

## How it works

UDP is connectionless: the parties do not need to open a connection with a handshake. Data flies as independent datagram packets, each handled separately.

Key properties:

- **Connectionless:** there is no phase for establishing or tearing down the connection.
- **No delivery guarantees:** packets can be lost, duplicated, or arrive out of order.
- **No flow or congestion control:** the sender transmits at any rate and may overload the channel.
- **Low latency and small overhead:** the UDP header is just 8 bytes, against a noticeably larger one in TCP.

Sending a datagram in Python:

```python
import socket

sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
server_address = ('localhost', 10000)
message = b'This is a message.'

try:
    sent = sock.sendto(message, server_address)
finally:
    sock.close()
```

Receiving datagrams:

```python
import socket

sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
server_address = ('localhost', 10000)
sock.bind(server_address)

while True:
    data, address = sock.recvfrom(4096)
    print(f'Received {data} from {address}')
```

Here `SOCK_DGRAM` means a UDP socket. On the sending side, `sendto()` immediately sends a packet to the address without a prior `connect()`. On the receiving side, `recvfrom()` returns both the data and the sender's address - this matters, since there is no connection and the server does not know in advance who will send a packet.

## Pitfalls

- **UDP does not mean "unreliable by the nature of the network."** The protocol simply adds no guarantees - acknowledgements and retries can be implemented manually at the application layer (that is what QUIC and game engines do).
- **Datagram size and fragmentation.** An overly large UDP packet is fragmented at the IP layer, and losing a single fragment loses the whole datagram. The payload is often kept within the MTU.
- **No congestion control.** A UDP flow can crowd TCP traffic out of the channel - in production this is taken into account.
- **`recvfrom()` returns exactly one datagram.** Unlike TCP, message boundaries are preserved, but if the buffer is smaller than the packet, the remainder is truncated.
