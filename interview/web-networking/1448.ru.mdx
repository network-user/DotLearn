# UDP: быстрая передача без соединения и без гарантий

## Суть

UDP - транспортный протокол, который шлёт данные отдельными дейтаграммами без установления соединения и без подтверждения доставки. Он жертвует надёжностью ради минимальной задержки и потому хорош там, где потеря пакета терпима, а лишние миллисекунды - нет.

## Что ответить на собесе

UDP - это транспортный протокол без соединения.

- В отличие от TCP он не устанавливает связь перед отправкой, не гарантирует доставку, порядок и отсутствие дублей, не управляет потоком и перегрузкой.
- Зато у него почти нет накладных расходов, поэтому он даёт низкую задержку.
- Его выбирают для потокового видео и аудио, онлайн-игр, VoIP и DNS-запросов - там, где важнее быстро доставить свежие данные, чем гарантированно доставить устаревшие.
- Если приложению нужна надёжность поверх UDP, оно реализует её само.

## Как это работает

UDP работает без соединения: сторонам не нужно открывать связь рукопожатием. Данные летят независимыми пакетами-дейтаграммами, каждая обрабатывается отдельно.

Ключевые свойства:

- **Без соединения:** нет фазы установления и закрытия связи.
- **Без гарантий доставки:** пакеты могут потеряться, продублироваться или прийти не по порядку.
- **Без управления потоком и перегрузкой:** отправитель шлёт с любой скоростью и может перегрузить канал.
- **Низкая задержка и малый оверхед:** заголовок UDP всего 8 байт против заметно большего у TCP.

Отправка дейтаграммы на Python:

```python
import socket

sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
server_address = ('localhost', 10000)
message = b'This is a message.'

try:
    sent = sock.sendto(message, server_address)
finally:
    sock.close()
```

Приём дейтаграмм:

```python
import socket

sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
server_address = ('localhost', 10000)
sock.bind(server_address)

while True:
    data, address = sock.recvfrom(4096)
    print(f'Received {data} from {address}')
```

Здесь `SOCK_DGRAM` означает UDP-сокет. На отправке `sendto()` сразу шлёт пакет по адресу без предварительного `connect()`. На приёме `recvfrom()` возвращает и данные, и адрес отправителя - это важно, ведь соединения нет и сервер заранее не знает, кто пришлёт пакет.

## Подводные камни

- **UDP не значит "ненадёжно по своей природе сети".** Протокол просто не добавляет гарантий - реализовать подтверждения и повторы можно вручную на прикладном уровне (так делают QUIC и игровые движки).
- **Размер дейтаграммы и фрагментация.** Слишком большой UDP-пакет фрагментируется на IP-уровне, и потеря одного фрагмента теряет всю дейтаграмму. Часто держат полезную нагрузку в пределах MTU.
- **Нет управления перегрузкой.** Поток UDP может вытеснять TCP-трафик из канала - в продакшене это учитывают.
- **`recvfrom()` возвращает ровно одну дейтаграмму.** В отличие от TCP границы сообщений сохраняются, но если буфер меньше пакета, остаток усекается.
