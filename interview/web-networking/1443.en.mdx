# SOAP vs REST: where is the strict protocol, and where is the architectural style?

## In short

SOAP is a formal protocol with a rigid XML message structure and a set of WS-standards. REST is not a protocol but an architectural style built on top of existing HTTP. The first is chosen for built-in security and reliability, the second for simplicity and scalability.

## How to answer in an interview

I'd put it this way: SOAP is a protocol, REST is an architectural style, and that is the key difference. SOAP mandates a required XML envelope (Header plus Body), is transport-independent (HTTP, SMTP, TCP), and brings along standards like WS-Security and WS-ReliableMessaging — hence its place in banking and enterprise, where guarantees matter. REST operates on resources via URIs and standard HTTP methods (GET, POST, PUT, DELETE), the body format is free, usually JSON. That is why REST is lighter, is cacheable with HTTP's own tools, and scales more easily — and it became the standard for public APIs.

## How it works

SOAP wraps every message in an `Envelope`: the `Header` holds routing and security metadata, and the `Body` holds the request or response itself. The structure is validated against a schema, so deviations won't get through.

```xml
<soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope">
  <soap:Header>
    <!-- metadata: security, routing -->
  </soap:Header>
  <soap:Body>
    <m:GetPrice xmlns:m="http://www.example.org/stock">
      <m:StockName>IBM</m:StockName>
    </m:GetPrice>
  </soap:Body>
</soap:Envelope>
```

REST, instead of an envelope, relies on HTTP itself: a resource is addressed via a URI, and the action is chosen by the method. The body can be JSON, XML, HTML — anything, but most often JSON.

```http
GET /stocks/IBM HTTP/1.1
Host: api.example.com
Accept: application/json
```

The difference in plain terms:

- **Nature:** SOAP is a protocol with strict rules; REST is a style that uses existing standards.
- **Format:** SOAP is XML only; REST accepts any format, JSON by default.
- **Transport:** SOAP is not tied to HTTP; REST lives in the semantics of HTTP methods and statuses.
- **Where it fits:** SOAP — complex enterprise systems with security requirements; REST — the web and public APIs.

## Pitfalls

- Don't confuse the levels of comparison: comparing "SOAP and REST" is only correct once you understand that one is a protocol and the other a style. REST's analog is more like gRPC or GraphQL, and SOAP's analog is another strict protocol.
- REST itself doesn't forbid XML, and SOAP technically can travel over more than just HTTP — a common mistake is to think otherwise.
- "RESTful" does not equal "any JSON-over-HTTP". Without working with resources, methods, and status codes, it is just RPC over HTTP.
- Security in REST is not built in the way it is in SOAP — it is added separately (TLS, OAuth, JWT), and that is easy to forget.
