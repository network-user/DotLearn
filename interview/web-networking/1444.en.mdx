# Why does the browser send a preliminary OPTIONS before the main request?

## In short

The browser sends OPTIONS as a CORS preflight check: it asks the other domain whether a request with the given method and headers is allowed there. The OPTIONS itself changes nothing on the server — it is reconnaissance before the main GET or POST.

## How to answer in an interview

An OPTIONS before the main request is a CORS preflight. The browser does it automatically when the request goes to a different origin and is not considered "simple" (for example, a POST with `Content-Type: application/json` or with custom headers). In the preflight the browser includes `Access-Control-Request-Method` and `Access-Control-Request-Headers`, and the server must respond with `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, and `Access-Control-Allow-Headers`. If the response permits it, the browser sends the real request; if not, it blocks it before it is even sent. This is a security mechanism of the browser itself, not the server.

## How it works

The browser sends OPTIONS, describing what it intends to do:

```http
OPTIONS /api/resource HTTP/1.1
Host: example.com
Origin: http://client.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type
```

The server replies with which origins, methods, and headers it allows:

```http
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: http://client.com
Access-Control-Allow-Methods: GET, POST, OPTIONS
Access-Control-Allow-Headers: Content-Type
Access-Control-Max-Age: 600
```

The browser checks the intent against the permission. A match — and the real POST goes through. `Access-Control-Max-Age` tells it how many seconds this response can be cached, so a preflight isn't needed before every request.

## Pitfalls

- A preflight isn't always sent. For "simple" requests (GET/HEAD/POST with standard headers and a `Content-Type` like `text/plain` or `application/x-www-form-urlencoded`) the browser does not send OPTIONS.
- CORS is browser-side protection. curl, Postman, and server-to-server calls ignore the preflight, so "it works for me in Postman" does not mean it works in the browser too.
- The server must explicitly respond to OPTIONS with the right headers. A forgotten `Access-Control-Allow-Headers` for a custom header — and the request fails at the preflight.
- Excessive preflights hurt performance. Use `Access-Control-Max-Age` so the browser caches the permission and doesn't fire OPTIONS before every call.
- `Access-Control-Allow-Origin: *` is incompatible with sending credentials (cookies). For those you need a specific origin plus `Access-Control-Allow-Credentials: true`.
