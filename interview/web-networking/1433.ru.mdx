# Как ловить вебхуки на локальной машине во время разработки?

## Суть

Локальный сервер недоступен из интернета, потому что у него нет публичного IP. Проблему решает туннель: утилита вроде ngrok или localtunnel выдаёт временный публичный URL и проксирует все приходящие на него запросы прямо на ваш `localhost`. Этого достаточно, чтобы отлаживать вебхуки без выкладки на удалённый сервер.

## Что ответить на собесе

Вебхук — это исходящий HTTP-запрос, который чужой сервис шлёт на мой URL при наступлении события. Чтобы он достучался до моей машины, поднимаю туннель:

- Запускаю ngrok на тот же порт, что и локальный сервер, и получаю публичный адрес вроде `https://abcd.ngrok.io`.
- Прописываю этот адрес в настройках внешнего сервиса как webhook URL — все запросы текут на мой `localhost`.
- В веб-панели ngrok вижу каждый запрос и могу его переотправить, поэтому отладка занимает минуты.

## Как это работает

Вебхук — это push-уведомление по HTTP: вместо того чтобы клиент опрашивал сервис, сервис сам отправляет запрос на заранее заданный URL, когда происходит событие (новый платёж, пуш в репозиторий и так далее).

Загвоздка в том, что у домашней или офисной машины обычно серый адрес за NAT — извне до неё не дойти. Туннель строит мост: устанавливает исходящее соединение к публичному релею, а тот переадресует входящие запросы по нему обратно на ваш порт.

Запуск через ngrok:

1. Поднимите приложение-обработчик, например на `http://localhost:5000`.
2. Откройте туннель к этому порту:

```bash
ngrok http 5000
```

3. Заберите выданный публичный URL вида `https://abcd1234.ngrok.io`.
4. Пропишите этот адрес в настройках внешнего сервиса как webhook endpoint.

Через панель ngrok на `http://127.0.0.1:4040` видно все входящие запросы, и любой можно переотправить — повторять реальное событие не нужно.

Альтернатива — localtunnel:

```bash
npm install -g localtunnel
lt --port 5000
```

Команда выдаст адрес вида `https://abcd1234.loca.lt`, который точно так же указывается во внешнем сервисе.

## Подводные камни

- Бесплатный публичный URL меняется при каждом перезапуске, и вебхук придётся переписывать заново; для стабильного адреса нужен платный план или зарезервированный домен.
- Туннель открывает ваш локальный порт наружу — не держите его поднятым дольше нужного и не светите чувствительные эндпоинты.
- Многие провайдеры подписывают вебхуки секретом: проверяйте подпись, иначе на публичный URL прилетит мусор от посторонних.
- Если обработчик отвечает медленно или не 2xx, отправитель будет ретраить — следите за таймаутами и идемпотентностью.
- Некоторые сервисы требуют именно HTTPS-адрес для вебхука, так что обычный http-туннель может не подойти.
