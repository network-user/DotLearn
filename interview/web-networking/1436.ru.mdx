# Что добавляет буква S в HTTPS по сравнению с обычным HTTP

## Суть

HTTPS — это тот же HTTP, но поверх защищённого канала TLS. Он добавляет три гарантии: шифрование трафика, подтверждение подлинности сервера сертификатом и контроль целостности данных в пути.

## Что ответить на собесе

Отвечу так: HTTP передаёт всё открытым текстом по порту 80, любой на маршруте может прочитать или подменить данные. HTTPS оборачивает HTTP в слой TLS (раньше говорили SSL) и работает на порту 443. Синтаксис запросов и методов при этом не меняется, меняется только транспорт под ними. TLS даёт три вещи:

- Конфиденциальность — трафик зашифрован.
- Аутентичность — сертификат удостоверяет, что я говорю именно с нужным сервером, а не с посредником.
- Целостность — любая подмена байтов по дороге будет замечена.

## Как это работает

На уровне сообщения HTTP-запрос выглядит одинаково в обоих случаях:

```http
GET /index.html HTTP/1.1
Host: www.example.com
```

Разница в том, что под HTTPS этот текст не уходит в сеть как есть — он шифруется. Перед обменом данными клиент и сервер проводят TLS-рукопожатие:

1. Клиент подключается к порту 443 и присылает поддерживаемые версии TLS и шифры.
2. Сервер отвечает своим сертификатом, который подписан доверенным центром сертификации (CA).
3. Стороны согласуют сеансовые ключи (современные наборы используют обмен по Диффи-Хеллману для forward secrecy).
4. Дальше весь прикладной трафик идёт симметрично зашифрованным.

Проверить защищённость соединения можно и из кода — успешный запрос по `https://` означает, что сертификат прошёл валидацию:

```python
import requests

resp = requests.get("https://www.example.com")
print(resp.url.startswith("https://"))
```

## Подводные камни

- HTTPS шифрует тело и заголовки, но не скрывает сам факт соединения: имя хоста и IP видны (домен утекает через SNI и DNS, если не используются их зашифрованные варианты).
- Валидный сертификат подтверждает владение доменом, но не «честность» сайта — фишинговый ресурс тоже может иметь корректный HTTPS.
- Отключать проверку сертификата (`verify=False`) в продакшене нельзя: это убирает защиту от man-in-the-middle и оставляет только шифрование «в никуда».
- HTTPS не спасает от уязвимостей на самом сервере (SQL-инъекции, XSS) — он защищает только канал передачи.
- Смешанный контент: страница по HTTPS, подгружающая ресурсы по HTTP, теряет часть гарантий, и браузеры такие запросы блокируют.
