# SSL и TLS: как шифрованный канал защищает трафик и чем они различаются

## Суть

SSL и TLS - криптографические протоколы транспортного уровня, которые создают защищённый канал поверх обычного TCP. TLS - это эволюция SSL: старый SSL давно небезопасен и выведен из эксплуатации, а живой протокол сегодня называется TLS. Оба дают одно и то же трио: шифрование, аутентификацию и проверку целостности.

## Что ответить на собесе

SSL (Secure Sockets Layer) - исторический протокол, его версии скомпрометированы и больше не применяются. TLS (Transport Layer Security) - его преемник, именно он стоит за каждым HTTPS-соединением, хотя по привычке всё ещё говорят «SSL-сертификат». Протокол решает три задачи: конфиденциальность (трафик зашифрован), аутентификация (сертификат подтверждает, что сервер тот, за кого себя выдаёт), целостность (подмену данных в пути видно по MAC/хешу). Соединение начинается с handshake: стороны договариваются о шифрах, сервер предъявляет сертификат, и согласуется сессионный ключ для быстрого симметричного шифрования.

Установка защищённого соединения проходит по шагам:

1. **Handshake (рукопожатие):** клиент шлёт список поддерживаемых шифронаборов, сервер выбирает общий.
2. **Аутентификация и обмен ключами:** сервер присылает сертификат с открытым ключом, клиент проверяет цепочку доверия и согласует сессионный ключ (в современном TLS - через эфемерный Диффи-Хеллман).
3. **Симметричное шифрование:** дальше весь обмен шифруется быстрым симметричным алгоритмом на сессионном ключе.
4. **Контроль целостности:** к каждому сообщению добавляется код аутентичности, так что подмену байтов в канале сразу заметно.

В Python защищённый сокет получают через стандартный модуль `ssl`:

```python
import socket
import ssl

hostname = "example.com"
context = ssl.create_default_context()

with socket.create_connection((hostname, 443)) as raw_sock:
    with context.wrap_socket(raw_sock, server_hostname=hostname) as tls_sock:
        tls_sock.sendall(b"GET / HTTP/1.1\r\nHost: example.com\r\nConnection: close\r\n\r\n")
        chunks = []
        while True:
            data = tls_sock.recv(4096)
            if not data:
                break
            chunks.append(data)

print(b"".join(chunks).decode("utf-8", "replace")[:200])
```

`create_default_context()` сразу включает проверку сертификата и имени хоста, а `wrap_socket` оборачивает обычный TCP-сокет в TLS-слой. Аргумент `server_hostname` нужен и для SNI, и для валидации, что в сертификате указан именно этот домен.

## Подводные камни

- «SSL» в речи обычно значит TLS: старые версии SSL 2.0/3.0 уязвимы (POODLE и др.) и отключены. Не путайте бренд с реально работающим протоколом.
- Не отключайте проверку сертификата (`context.check_hostname = False`, `CERT_NONE`) ради «чтобы заработало» - это убивает аутентификацию и открывает MITM-атаку.
- TLS шифрует канал, но не содержимое после расшифровки и не метаданные вроде имени домена (исторически SNI шёл открытым текстом).
- Просроченный или самоподписанный сертификат рвёт соединение по умолчанию - это фича, а не баг.
