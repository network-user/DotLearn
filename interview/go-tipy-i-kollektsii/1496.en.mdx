# How do int and uint differ?

## In short

`int` is a signed platform-width integer; it holds negative and positive values. `uint` is unsigned, non-negative only, with twice the positive range for the same bit width.

## How to answer in an interview

- On amd64 both are 64-bit; on 32-bit arches they are 32-bit.
- You cannot mix int and uint in arithmetic without explicit conversion.
- Stdlib lengths and indexes almost always use `int`, not uint.
- I honestly add that uint helps with bit masks and hash-like values, but wraparound is silent and underflow through zero is easy to miss.

## How it works

The compiler keeps signed and unsigned separate. Conversion `uint(x)` range-checks constants at compile time only.

```go
package main

import "fmt"

func main() {
	var signed int = -3
	var unsigned uint = 5

	fmt.Println(signed, unsigned)

	// mix := signed + unsigned // compile error

	if signed < 0 {
		unsigned = uint(-signed)
	}
	fmt.Println(unsigned)
}
```

Casting negatives to uint needs care: `-1` becomes the maximum value.

## Pitfalls

- **Negative to uint cast.** `-1` as uint is the top of the range.
- **Printf verbs.** `%d` works on uint but mixed types confuse readers.
- **JSON numbers.** Unmarshaling into uint rejects negatives.
- **32-bit builds.** Code assuming 64-bit int breaks on 32-bit targets.
- **Loop counters.** Idiomatic Go uses int, not uint, for indexes.
