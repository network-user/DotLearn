# Что такое интерфейс в Go и зачем он нужен?

## Суть

Интерфейс в Go это набор методов без реализации. Любой тип, у которого есть эти методы, удовлетворяет интерфейсу неявно. Это даёт полиморфизм без наследования и жёсткой иерархии классов.

## Что ответить на собесе

- Формулирую: интерфейс описывает поведение (контракт), конкретные struct его реализуют методами.
- Привожу stdlib: `io.Reader`, `fmt.Stringer`, `error` как узкие интерфейсы.
- Подчёркиваю implicit satisfaction: нет ключевого слова `implements`.
- Честно добавляю: пустой интерфейс `any` принимает всё, но в новом коде предпочитаю конкретные типы или дженерики вместо безграничного `interface{}`.

## Как это работает

Значение интерфейса в рантайме это пара (tab типа, data pointer). При присваивании concrete value в interface runtime проверяет, есть ли нужный method set. Вызов метода идёт через таблицу, поэтому полиморфный код не знает конкретный struct.

Идиома Go: объявлять маленькие интерфейсы рядом с потребителем («accept interfaces, return structs»). Большие интерфейсы с десятком методов усложняют тестирование и моки.

```go
package main

import "fmt"

type Notifier interface {
	Notify(msg string) error
}

type EmailSender struct {
	Address string
}

func (e EmailSender) Notify(msg string) error {
	fmt.Printf("email to %s: %s\n", e.Address, msg)
	return nil
}

type SMSPinger struct {
	Phone string
}

func (s SMSPinger) Notify(msg string) error {
	fmt.Printf("sms to %s: %s\n", s.Phone, msg)
	return nil
}

func broadcast(channels []Notifier, text string) {
	for _, ch := range channels {
		_ = ch.Notify(text)
	}
}

func main() {
	channels := []Notifier{
		EmailSender{Address: "dev@corp.local"},
		SMSPinger{Phone: "+79001234567"},
	}
	broadcast(channels, "deploy finished")
}
```

`broadcast` работает с любым типом, реализующим `Notify`, без перечисления вариантов. Добавление `SlackWebhook` не меняет сигнатуру `broadcast`.

Проверка на этапе компиляции: `var _ Notifier = (*MyType)(nil)` ловит отсутствие метода до рантайма.

## Подводные камни

- **Nil interface vs typed nil.** `var p *T; var i Notifier = p` даёт non-nil interface с nil data; сравнение с `nil` ломается.
- **Value vs pointer receiver.** Метод с pointer receiver не попадает в interface для value type и наоборот.
- **Слишком широкий any.** Потеря типобезопасности и лишние type assertion в каждом слое.
- **Fat interfaces в пакете models.** Лучше 1–2 метода у потребителя, иначе каждый мок тащит весь контракт.
- **Сравнение интерфейсов.** Два interface equal только если dynamic type и value совпали; сравнивать срезы внутри нельзя без reflect.
