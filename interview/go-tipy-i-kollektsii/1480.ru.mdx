# Какой принцип ООП дают struct в Go?

## Суть

Struct в Go в первую очередь несут инкапсуляцию: данные и методы живут в одном типе, а видимость полей и методов задаётся экспортом (заглавная vs строчная буква). Наследования классов в Java-смысле нет.

## Что ответить на собесе

- Называю инкапсуляцию через unexported поля и публичные методы доступа.
- Уточняю, что Go не классы: composition и embedding вместо наследования.
- Показываю пример `balance` с маленькой буквы и метод `Deposit`.
- Честно добавляю: инкапсуляция пакетная, не объектная: поле с lowercase видно всем файлам того же package.

## Как это работает

Struct группирует состояние, методы добавляют поведение с receiver. Экспорт определяется именем, не модификатором `private`. Валидацию и инварианты держу в конструкторе и сеттерах.

```go
package wallet

type Account struct {
	Owner   string
	balance int
}

func NewAccount(owner string) *Account {
	return &Account{Owner: owner, balance: 0}
}

func (a *Account) Deposit(cents int) {
	if cents <= 0 {
		return
	}
	a.balance += cents
}

func (a *Account) Balance() int {
	return a.balance
}
```

Снаружи package нельзя прочитать `balance` напрямую, только через `Balance()` и изменить через `Deposit`.

## Подводные камни

- **Экспорт поля обходит инкапсуляцию.** Публичное `Balance int` ломает инварианты без setter.
- **Value receiver копирует state.** Метод с `(a Account)` не меняет оригинал; для мутаций нужен pointer receiver.
- **Embedding не private.** Встроенное поле продвигает методы, но не скрывает данные автоматически.
- **JSON tags.** `json:"balance"` может сериализовать unexported поле только через reflection в том же package.
- **Полиморфизм через interface.** Поведение подменяют интерфейсами, а не virtual methods базового класса.
