# Какие свойства нужны hash-функции для map?

## Суть

Хеш для map должен быть детерминированным, быстрым и давать равномерное распределение по bucket index. Коллизии неизбежны, но их должно быть мало; runtime Go сам выбирает hash per type.

## Что ответить на собесе

- Перечисляю: один ключ всегда один hash, разные ключи равномерно размазываются, вычисление дешёвое на hot path.
- Объясняю, что коллизии разрешаются цепочками внутри bucket, поэтому плохой hash деградирует в O(n).
- Уточняю, что пользовательская map не принимает custom hash: алгоритм в runtime.
- Честно добавляю: для своих hash-table структур беру проверенные функции (FNV, xxhash), а для map полагаюсь на stdlib.

## Как это работает

При insert/read ключ хешируется, младшие биты выбирают bucket. Хороший hash снижает длину цепочек. Go для строк и integers использует встроенные быстрые функции с seed, меняющимся между процессами для защиты от HashDoS на string keys.

Пример standalone hash для строк (не map API):

```go
package main

import (
	"fmt"
	"hash/fnv"
)

func digest(label string) uint32 {
	h := fnv.New32a()
	_, _ = h.Write([]byte(label))
	return h.Sum32()
}

func main() {
	for _, word := range []string{"go", "map", "hash"} {
		fmt.Printf("%s -> %d\n", word, digest(word))
	}
}
```

Один и тот же `word` всегда даёт один digest; разные слова с высокой вероятностью разные числа.

## Подводные камни

- **Путать hash и equality.** Равный hash не значит равный ключ; сравнение всё равно через `==`.
- **Float keys.** NaN и -0 vs +0 ломают ожидания равенства и распределения.
- **Атака на string map.** Без randomized seed злоумышленник мог бы подобрать коллизии; Go это учитывает для maps.
- **Слишком широкий hash.** Если брать только младшие биты плохой функции, bucket skew остаётся.
- **Custom map types.** `map` нельзя параметризовать своей hash функцией; нужна отдельная структура.
