# How is a slice structured internally?

## In short

A slice is a descriptor with three fields: a pointer to the underlying array, length, and capacity. Multiple slices can share one array, so mutations through a shared backing store are visible to every slice.

## How to answer in an interview

- I explain the pointer/len/cap triplet and that the slice header does not store elements itself.
- I show creation with `make([]T, len, cap)` and sub-slices like `arr[1:4]`.
- I say `append` allocates a new array and copies when capacity runs out.
- I honestly add that shared backing causes subtle aliasing bugs unless you copy explicitly with `append([]T(nil), src...)`.

## How it works

Arrays are fixed size; a slice defines a window. `len` bounds accessible elements; `cap` is how far you can grow without reallocation from the window start.

```go
package main

import "fmt"

func main() {
	base := []int{10, 20, 30, 40}
	window := base[1:3]
	fmt.Println(window, len(window), cap(window))

	window[0] = 99
	fmt.Println(base)

	grown := append(window, 50)
	fmt.Println(grown, cap(grown))
	base[2] = 777
	fmt.Println(grown)
}
```

After `window[0] = 99`, `base[1]` changes. After `append`, if capacity allowed growth in the same array, changing `base[2]` may affect `grown`.

## Pitfalls

- **Subslice and append.** Appending to a small subslice can overwrite the tail of the shared array.
- **Nil vs empty slice.** `var s []int` and `[]int{}` differ in JSON and some APIs even when len is zero.
- **Copy length.** `copy(dst, src)` moves min(len(dst), len(src)) elements, not capacity.
- **3-index slicing.** `s[low:high:max]` caps capacity and protects against accidental append into shared memory.
- **Range over pointers.** `[]*T` in range copies pointer values, not deep copies of objects.
