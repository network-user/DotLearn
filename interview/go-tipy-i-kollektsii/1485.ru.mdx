# Какие ограничения на ключи map?

## Суть

Ключ map обязан быть comparable: поддерживать `==` и `!=`. Slice, map и func как ключи запрещены; array и struct допустимы, если их элементы/поля comparable.

## Что ответить на собесе

- Формулирую правило comparable из spec, не путаю slice и array.
- Для составных ключей проверяю каждое поле: float/complex/slice внутри struct блокируют ключ.
- Показываю типичный ключ `string` в задаче word frequency.
- Честно добавляю: если нужен mutable ключ, нормализую в immutable string/id вместо обхода ограничения.

## Как это работает

Hash-таблица сравнивает ключи на равенство при lookup и insert. Несравнимый тип не компилируется как key type parameter.

```go
package main

import (
	"fmt"
	"strings"
)

func main() {
	line := "go map go slice"
	freq := make(map[string]int)
	for _, token := range strings.Fields(line) {
		freq[token]++
	}
	for word, count := range freq {
		fmt.Printf("%s: %d\n", word, count)
	}
}
```

`string` comparable, поэтому счётчик частот работает из коробки. Попытка `map[[]byte]int` упадёт на compile.

## Подводные камни

- **Array vs slice.** `[4]byte` может быть ключом, `[]byte` нет; частая путаница на собесах.
- **Interface key.** Dynamic type должен быть comparable; положить slice в interface и использовать как key нельзя.
- **NaN в float полях.** Ломает логику равенства struct-key.
- **Key as mutable buffer.** `[N]byte` comparable, но если менять байты после insert, lookup по старой копии не найдёт запись.
- **Обход через stringify.** `fmt.Sprint(slice)` как ключ рискует коллизиями и дорогим hash.
