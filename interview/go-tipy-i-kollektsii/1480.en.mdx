# Which OOP principle do Go structs provide?

## In short

Go structs mainly deliver encapsulation: data and methods live in one type, and field/method visibility comes from export rules (uppercase vs lowercase). There is no Java-style class inheritance.

## How to answer in an interview

- I name encapsulation via unexported fields and public accessor methods.
- I clarify that Go is not class-based: composition and embedding replace inheritance.
- I show an example with lowercase `balance` and a `Deposit` method.
- I honestly add that encapsulation is package-scoped: a lowercase field is visible to every file in the same package.

## How it works

A struct groups state; methods attach behavior with a receiver. Export is by identifier name, not a `private` keyword. Invariants live in constructors and setters.

```go
package wallet

type Account struct {
	Owner   string
	balance int
}

func NewAccount(owner string) *Account {
	return &Account{Owner: owner, balance: 0}
}

func (a *Account) Deposit(cents int) {
	if cents <= 0 {
		return
	}
	a.balance += cents
}

func (a *Account) Balance() int {
	return a.balance
}
```

Outside the package you cannot read `balance` directly, only through `Balance()` and change it via `Deposit`.

## Pitfalls

- **Exported fields bypass encapsulation.** Public `Balance int` breaks invariants without a setter.
- **Value receivers copy state.** A method on `(a Account)` does not mutate the original; mutations need a pointer receiver.
- **Embedding is not privacy.** An embedded field promotes methods but does not hide data by itself.
- **JSON tags.** `json:"balance"` may serialize unexported fields only via reflection within the same package.
- **Polymorphism via interfaces.** Behavior is swapped with interfaces, not virtual methods on a base class.
