# What data structure backs a Go map?

## In short

The built-in map is a hash table: an array of buckets, a key hash, and overflow chains inside buckets. Insert, lookup, and delete are amortized O(1) on average.

## How to answer in an interview

- I name a hash table with buckets and overflow chains, not a top-level linked list.
- I explain hash, load factor, and growth when the table fills up.
- I demonstrate basic CRUD: make, assign, comma-ok, delete.
- I honestly add that bucket layout changed across Go versions; interviews care about the model, not a runtime hex dump.

## How it works

`make(map[K]V)` allocates an hmap header. The key is hashed, a bucket is chosen, and slots are scanned. Collisions use overflow buckets. High load factor doubles the table and rehashes.

```go
package main

import "fmt"

func main() {
	ages := map[string]int{
		"Ada":   36,
		"Linus": 54,
	}

	ages["Grace"] = 85

	if age, ok := ages["Ada"]; ok {
		fmt.Println("Ada:", age)
	}

	delete(ages, "Linus")

	if _, ok := ages["Linus"]; !ok {
		fmt.Println("Linus removed")
	}
}
```

`delete` removes an entry; access after removal yields the zero value and `ok=false`.

## Pitfalls

- **Nil map write.** Writing to a nil map panics; reads return zero.
- **Not thread-safe.** Concurrent writes without sync are a data race.
- **Iteration order.** Walk follows bucket layout, not sorted keys.
- **Float keys.** NaN breaks deduplication logic.
- **Address of map element.** `&m[k]` is illegal because rehash may move storage.
