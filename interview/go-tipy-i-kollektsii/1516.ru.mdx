# Какие подводные камни у range по map?

## Суть

`for k, v := range m` обходит элементы в псевдослучайном порядке, который меняется между запусками. Порядок не связан с порядком вставки. Изменение map во время итерации имеет ограниченные гарантии: удаление текущего ключа допустимо, произвольные вставки могут пропустить или повторить элементы.

## Что ответить на собесе

- Не полагаюсь на стабильный порядок; для сортировки собираю ключи и сортирую отдельно.
- `range` копирует value в переменную; мутация `v` не меняет map, если value не указатель.
- Для детерминированных тестов фиксирую seed или сравниваю multiset, не порядок.
- Честно добавляю: concurrent `range` и write без sync недопустимы.

## Как это работает

```go
package main

import (
	"fmt"
	"sort"
)

func main() {
	counts := map[string]int{"go": 3, "rust": 1, "zig": 2}

	keys := make([]string, 0, len(counts))
	for k := range counts {
		keys = append(keys, k)
	}
	sort.Strings(keys)

	for _, k := range keys {
		fmt.Println(k, counts[k])
	}
}
```

Runtime перемешивает стартовый bucket для `range`, чтобы порядок не был предсказуемым. Сбор ключей и внешняя сортировка даёт стабильный вывод без изменения сложности обхода O(n).

## Подводные камни

- **Тесты на порядок.** Хрупкие assert'ы ломаются при смене версии Go или hash seed.
- **Копия value.** Большие struct values копируются на каждой итерации; для тяжёлых value храню указатели.
- **Delete during range.** Удаление текущего ключа безопасно; добавление новых ключей во время обхода рискованно.
- **Пустой nil map.** `range` по nil map 0 итераций, без panic.
- **break/continue.** Как в обычном for; нет индекса для partial resume без отдельного state.
