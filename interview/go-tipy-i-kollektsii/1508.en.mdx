# Does capacity change when you shrink a slice?

## In short

A slice expression `s = s[:n]` only shrinks `len`; `cap` stays the same relative to the start of the backing array. Capacity changes on `append` that reallocates, or on a re-slice with a new offset that shifts the window into the array.

## How to answer in an interview

- `full[:3]` yields `len=3` with the same `cap` as the original slice (when slicing from index 0).
- A truncated slice still sees the array tail beyond `len`; hidden elements are not zeroed.
- `append` on a shortened slice can overwrite hidden slots when `cap` allows growth without reallocation.
- I honestly warn: for isolation I copy with `append([]T(nil), s...)` or `slices.Clone`.

## How it works

```go
package main

import "fmt"

func main() {
	base := []int{10, 20, 30, 40, 50}
	fmt.Printf("full: len=%d cap=%d %v\n", len(base), cap(base), base)

	trimmed := base[:3]
	fmt.Printf("trimmed: len=%d cap=%d %v\n", len(trimmed), cap(trimmed), trimmed)

	trimmed = append(trimmed, 99)
	fmt.Println("base after append:", base) // [10 20 30 99 50]
}
```

Slicing only moves the visible length boundary. The slice header still points at the same array with the same offset; `cap` is counted from the current first element to the end of the backing store.

## Pitfalls

- **Hidden tail.** Elements past `len` remain in memory and are reachable through `append` without copying.
- **Leak via subslices.** A short slice of a huge array keeps the whole backing array alive for GC.
- **Offset reslice.** `arr[2:5]` lowers `cap` versus `arr[0:5]` because the start offset changes.
- **Zero length is not free.** `s = s[:0]` does not release the array while other subslices exist.
- **copy vs slice.** `copy(dst, src)` does not change `cap` of `dst`; the destination needs enough `len`.
