# Меняется ли cap при усечении slice?

## Суть

Операция среза `s = s[:n]` уменьшает только `len`; `cap` остаётся прежним относительно начала backing array. Ёмкость меняется при `append` с реаллокацией или при срезе с новым offset, который сдвигает начало окна в массиве.

## Что ответить на собесе

- `full[:3]` даёт `len=3`, `cap` как у исходного slice (если срез с индекса 0).
- Усечённый slice всё ещё видит «хвост» массива за пределами `len`; старые элементы не обнуляются.
- `append` на укороченный slice может перезаписать скрытые элементы, если cap позволяет без реаллокации.
- Честно предупреждаю: для изоляции данных копирую через `append([]T(nil), s...)` или `slices.Clone`.

## Как это работает

```go
package main

import "fmt"

func main() {
	base := []int{10, 20, 30, 40, 50}
	fmt.Printf("full: len=%d cap=%d %v\n", len(base), cap(base), base)

	trimmed := base[:3]
	fmt.Printf("trimmed: len=%d cap=%d %v\n", len(trimmed), cap(trimmed), trimmed)

	trimmed = append(trimmed, 99)
	fmt.Println("base after append:", base) // [10 20 30 99 50]
}
```

Срез сдвигает только границу видимой длины. Указатель slice header по-прежнему ссылается на тот же массив с тем же offset; `cap` считается от текущего первого элемента до конца массива.

## Подводные камни

- **Скрытый хвост.** Элементы за `len` остаются в памяти и доступны через `append` без копии.
- **Утечка через subslices.** Короткий срез большого массива удерживает весь backing array от GC.
- **Срез со смещением.** `arr[2:5]` уменьшает cap относительно `arr[0:5]`, потому что меняется стартовый offset.
- **Полное обнуление.** `s = s[:0]` не освобождает массив, пока живут другие срезы.
- **copy vs slice.** `copy(dst, src)` не меняет cap `dst`; нужен достаточный `len` у приёмника.
