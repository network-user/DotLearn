# What are the pitfalls of ranging over a map?

## In short

`for k, v := range m` walks entries in pseudo-random order that changes between runs. Order is unrelated to insertion order. Mutating a map during iteration has limited guarantees: deleting the current key is allowed; arbitrary inserts may skip or repeat entries.

## How to answer in an interview

- I never rely on stable order; for sorting I collect keys and sort separately.
- `range` copies the value variable; mutating `v` does not update the map unless the value is a pointer.
- For deterministic tests I compare multisets, not iteration order.
- I honestly add: concurrent `range` with unsynchronized writes is invalid.

## How it works

```go
package main

import (
	"fmt"
	"sort"
)

func main() {
	counts := map[string]int{"go": 3, "rust": 1, "zig": 2}

	keys := make([]string, 0, len(counts))
	for k := range counts {
		keys = append(keys, k)
	}
	sort.Strings(keys)

	for _, k := range keys {
		fmt.Println(k, counts[k])
	}
}
```

The runtime randomizes the starting bucket for `range` so order stays unpredictable. Collecting keys and sorting externally yields stable output without changing the O(n) walk cost.

## Pitfalls

- **Order-based tests.** Fragile assertions break across Go versions or hash seeds.
- **Value copy.** Large struct values copy each iteration; heavy values often live behind pointers.
- **Delete during range.** Removing the current key is safe; adding keys mid-walk is risky.
- **Empty nil map.** `range` over nil runs zero iterations without panic.
- **break/continue.** Same as ordinary for; no index for partial resume without extra state.
