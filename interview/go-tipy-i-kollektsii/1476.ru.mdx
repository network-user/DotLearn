# Какие типы могут быть ключами map?

## Суть

Ключ map в Go должен быть comparable: поддерживать оператор `==` и `!=`. Подходят числа, строки, bool, указатели, каналы, интерфейсы и структуры, если все их поля тоже comparable.

## Что ответить на собесе

- Формулирую правило: ключ сравним на равенство, иначе компилятор не примет тип.
- Перечисляю типичные ключи: `string`, целые, struct с comparable полями, указатели на однотипные объекты.
- Называю запреты: slice, map, func и struct с float/complex/slice внутри как ключ не годятся.
- Честно добавляю: для составных ключей иногда проще нормализовать данные в строку или использовать `map[string]V`, чем городить сложный struct-key.

## Как это работает

Hash-таблице нужно быстро понять, тот же это ключ или другой. Go проверяет comparability на этапе компиляции. Структура comparable, если comparable каждое поле. Массив comparable, если comparable его элемент.

Пример со строкой и struct-ключом:

```go
package main

import "fmt"

type Employee struct {
	ID   int
	Dept string
}

func main() {
	byName := map[string]string{
		"Anna": "+7-900-111",
		"Leo":  "+7-900-222",
	}
	fmt.Println(byName["Anna"])

	tenure := map[Employee]int{
		{ID: 10, Dept: "platform"}: 4,
		{ID: 11, Dept: "data"}:     2,
	}
	fmt.Println(tenure[Employee{ID: 10, Dept: "platform"}])
}
```

`Employee` годится как ключ, потому что оба поля comparable. Слайс `[ ]int` как ключ вызвал бы ошибку компиляции.

## Подводные камни

- **Float как поле struct.** NaN != NaN ломает интуицию равенства; float в ключах почти всегда плохая идея.
- **Interface key с nil.** Сравнение typed nil внутри interface может удивить при lookup.
- **Большие struct-ключи.** Копирование крупной struct на каждый доступ дороже, чем string id.
- **Pointer identity.** Два разных указателя на равные struct не равны как ключи, даже если `*a == *b`.
- **Array vs slice.** `[N]T` может быть ключом, `[]T` нет; путают часто на собесах.
