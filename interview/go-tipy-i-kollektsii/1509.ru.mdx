# Куда смотрит указатель внутри slice header?

## Суть

Slice header хранит указатель на первый элемент backing array (или `nil` для `nil` slice), плюс `len` и `cap`. При срезе `arr[low:high]` указатель смещается на `&arr[low]`, а `cap` уменьшается на `low`.

## Что ответить на собесе

- Три поля: data pointer, length, capacity; slice сам по себе не массив, а дескриптор окна.
- Срез массива или другого slice разделяет backing array без копирования.
- `nil` slice: указатель `nil`, `len=cap=0`; отличать от `make([]T, 0)`.
- Честно добавляю: передача slice в функцию копирует header (три слова), не элементы.

## Как это работает

```go
package main

import "fmt"

func main() {
	arr := [5]int{10, 20, 30, 40, 50}
	window := arr[1:4]

	fmt.Println(window)           // [20 30 40]
	fmt.Println(len(window), cap(window)) // 3 4
	fmt.Println(&arr[1] == &window[0])    // true
}
```

`window[0]` это тот же адрес, что `arr[1]`. Указатель slice указывает на начало видимого диапазона. `cap` равен числу элементов от этого адреса до конца исходного массива (здесь 4: позиции 20, 30, 40, 50).

## Подводные камни

- **Алиасинг.** Два slice на один массив видят изменения друг друга.
- **append и указатель.** После реаллокации `append` указатель может смениться; старые subslices остаются на старом массиве.
- **Срез за границей.** `s[low:high]` паникует при `high > cap`, не при `high > len` базового массива в общем случае.
- **Пустой, но не nil.** `make([]int, 0)` имеет ненулевой указатель на нулевой массив в некоторых версиях runtime (деталь), но `len` всё равно 0.
- **unsafe обход.** Через `unsafe` можно достать pointer, но в обычном коде полагаюсь на `len`/`cap`, не на адреса.
