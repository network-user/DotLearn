# In what order does Go iterate over a map?

## In short

The element order in `range` over a map is unspecified and may change between runs. That is an intentional runtime choice: a map is a hash table, not an ordered collection.

## How to answer in an interview

- I explain that `for k, v := range m` does not guarantee alphabetical order or insertion order.
- When I need a stable order, I collect keys into a slice, sort them, and iterate by key.
- I mention that randomized order helps catch hidden dependencies on iteration order in tests.
- I honestly add that within a single `range` call the order is fixed, but you must not rely on it across program runs.

## How it works

A map spreads pairs across buckets using the key hash. During iteration the runtime walks internal table structures, not a logical key order. Two consecutive `range` loops in one program may match, but after map changes or in another process the order differs.

For deterministic output, keys go into a slice and get sorted:

```go
package main

import (
	"fmt"
	"sort"
)

func main() {
	scores := map[string]int{
		"Zara": 88,
		"Alex": 91,
		"Mila": 76,
	}

	names := make([]string, 0, len(scores))
	for name := range scores {
		names = append(names, name)
	}
	sort.Strings(names)

	for _, name := range names {
		fmt.Printf("%s: %d\n", name, scores[name])
	}
}
```

Collect all keys first, then `sort.Strings` defines alphabetical order before reading values.

## Pitfalls

- **Tests with hardcoded order.** A snapshot test tied to map order will be flaky across Go versions or after rehash.
- **JSON serialization.** `encoding/json` sorts object keys, but that does not change the rule for `range` over a map in your code.
- **Sorting large maps on hot paths.** Sorting keys adds O(n log n); sometimes you keep a separate ordered structure.
- **Deleting during range.** Deleting the current key during `range` is safe; adding entries may change order for remaining iterations.
- **Displaying maps.** Two maps with the same pairs are equal regardless of iteration order, but UI output still needs explicit sorting.
