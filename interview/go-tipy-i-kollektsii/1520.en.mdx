# How does polymorphism work in Go?

## In short

Go polymorphism is structural typing through interfaces: a type fits if it implements the required methods, with no `extends` or explicit `implements`. A function taking an interface works with any matching concrete type at runtime via dynamic dispatch.

## How to answer in an interview

- One method is often enough: `io.Writer`, `fmt.Stringer`.
- Different concrete types share a `[]Shape` or function parameter through a common interface type.
- There is no class inheritance hierarchy; struct composition plus interfaces replace subclassing.
- I honestly caveat: polymorphism is not free; interface calls and possible value allocation cost matter.

## How it works

```go
package main

import "fmt"

type Shape interface {
	Area() float64
}

type Rect struct{ W, H float64 }

func (r Rect) Area() float64 { return r.W * r.H }

type Circle struct{ R float64 }

func (c Circle) Area() float64 { return 3.14159 * c.R * c.R }

func totalArea(shapes ...Shape) float64 {
	sum := 0.0
	for _, s := range shapes {
		sum += s.Area()
	}
	return sum
}

func main() {
	fmt.Println(totalArea(Rect{3, 4}, Circle{2}))
}
```

`totalArea` knows no concrete type; the runtime picks the method through an itable. A new shape does not change `totalArea` if it implements `Area() float64`.

## Pitfalls

- **Duck typing surprises.** A type accidentally satisfies an interface because signatures match unrelated intent.
- **Value vs pointer receivers.** `Area` only on `*T` excludes bare value `T` from the interface set.
- **Type assertion without ok.** Panic on the wrong dynamic type.
- **Overload via any.** Expressiveness drops; you fall back to type switches.
- **Test mocks.** Over-wide interfaces make stubs painful; keep interfaces minimal.
