# What algorithmic complexity classes do you know?

## In short

Big-O describes how time or memory grows with n. Core classes: O(1), O(log n), O(n), O(n log n), O(n²), O(2^n), O(n!). I distinguish worst, average, and amortized when it matters.

## How to answer in an interview

- I list classes with examples: map get O(1), binary search O(log n), slice scan O(n), sort O(n log n).
- I note space: recursive DFS O(h) stack, merge sort O(n) extra.
- For Go maps I say amortized O(1) insert, worst O(n) under pathological hashing.
- I honestly add that constants matter in production; Big-O is a model, not a profiling replacement.

## How it works

Pick the dominant term as n grows. Nested loops often yield O(n²); divide-and-conquer sorts O(n log n).

```go
package main

import "fmt"

func findFirst(ids []int, target int) (int, bool) {
	for i, v := range ids {
		if v == target {
			return i, true
		}
	}
	return 0, false
}

func main() {
	data := []int{4, 7, 1, 9}
	fmt.Println(findFirst(data, 7))
}
```

Linear scan is O(n) time, O(1) extra space.

## Pitfalls

- **O vs Θ vs Ω.** Interviews may ask for tight bounds.
- **Amortized vs worst.** append is amortized O(1); one step may O(n) copy.
- **Hidden parameters.** Graph algorithms use V+E, not a single n.
- **Micro benchmarks.** Cache effects dominate at small n.
- **Recursion space.** Deep recursion without tail calls can overflow the stack.
