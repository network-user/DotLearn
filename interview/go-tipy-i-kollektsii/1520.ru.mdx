# Как в Go устроен полиморфизм?

## Суть

Полиморфизм в Go это structural typing через интерфейсы: тип подходит, если реализует нужные методы, без `extends` или явного `implements`. Функция, принимающая интерфейс, работает с любой подходящей concrete type на этапе runtime через dynamic dispatch.

## Что ответить на собесе

- Один метод часто достаточен: `io.Writer`, `fmt.Stringer`.
- Разные concrete types в одном `[]Shape` или аргументе функции через общий interface type.
- Нет классической иерархии наследования; композиция struct + interface заменяет subclassing.
- Честно оговариваю: полиморфизм не бесплатен, interface call и возможная аллокация value.

## Как это работает

```go
package main

import "fmt"

type Shape interface {
	Area() float64
}

type Rect struct{ W, H float64 }

func (r Rect) Area() float64 { return r.W * r.H }

type Circle struct{ R float64 }

func (c Circle) Area() float64 { return 3.14159 * c.R * c.R }

func totalArea(shapes ...Shape) float64 {
	sum := 0.0
	for _, s := range shapes {
		sum += s.Area()
	}
	return sum
}

func main() {
	fmt.Println(totalArea(Rect{3, 4}, Circle{2}))
}
```

`totalArea` не знает concrete type; runtime выбирает метод через itable. Добавление нового shape не меняет `totalArea`, если новый тип реализует `Area() float64`.

## Подводные камни

- **Duck typing сюрпризы.** Тип случайно удовлетворяет interface из-за совпадающей сигнатуры в другом смысле.
- **Value vs pointer receiver.** `Area` только на `*T` исключает value `T` из interface set.
- **Type assertion без ok.** Паника при неверном dynamic type.
- **Перегруз через any.** Потеря выразительности; возврат к switch по типам.
- **Тестовые моки.** Слишком широкий interface усложняет stub; держу interface минимальным.
