# What is the empty interface and when is it not nil?

## In short

`interface{}` (or `any`) declares no methods, so any value type can be stored in it. An interface value is a (type, data) pair, and `== nil` checks are often misleading.

## How to answer in an interview

- I explain that the empty interface is a box for any concrete type, useful for generic-like APIs before generics existed.
- I distinguish a nil interface from an interface holding a typed nil: the second is not equal to nil.
- I show a type switch or type assertion to extract the concrete type safely.
- I honestly add that in modern Go I prefer generics or small interfaces for new APIs; I keep `any` for JSON, logging, and dynamic boundaries.

## How it works

At runtime an interface stores a type tab and a data word. An uninitialized `var box any` equals nil. After assigning a concrete value, box is no longer nil even when the stored value is a nil pointer.

```go
package main

import "fmt"

func describe(v any) {
	switch x := v.(type) {
	case int:
		fmt.Println("int:", x)
	case string:
		fmt.Println("string:", x)
	default:
		fmt.Printf("other %T\n", x)
	}
}

func main() {
	var empty any
	fmt.Println(empty == nil)

	var p *int
	var holder any = p
	fmt.Println(holder == nil)
	fmt.Println(p == nil)

	describe(42)
	describe("go")
}
```

`holder == nil` is false because type `*int` is already stored in the interface although the pointer is nil. This is a classic trap when returning `(any, error)`.

## Pitfalls

- **Typed nil in errors.** `return (*MyErr)(nil), nil` can become a non-nil error interface at the call site.
- **Lost static typing.** Heavy use of `any` moves failures to runtime instead of compile time.
- **JSON into map[string]any.** Numbers often arrive as `float64`, not `int`.
- **Comparing any values.** Two `any` values compare only when the dynamic type is comparable; slices inside an interface cannot be compared.
- **Allocations.** Small value types boxed into an interface may escape to the heap.
