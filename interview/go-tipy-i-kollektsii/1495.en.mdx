# What is special about the built-in copy for slices?

## In short

`copy(dst, src)` moves min(len(dst), len(src)) elements and returns that count. Slice lengths stay the same; overlapping regions copy correctly per spec.

## How to answer in an interview

- The returned int tells how many elements actually copied.
- dst and src element types must match.
- For a full clone I use `append([]T(nil), src...)` or `slices.Clone`.
- I honestly add that copy is shallow; for `[]*T` or structs with pointers I clone the second level separately.

## How it works

The built-in memmoves over underlying arrays. Overlapping dst/src behaves like memmove.

```go
package main

import "fmt"

func main() {
	src := []int{10, 20, 30, 40}
	dst := make([]int, 2)

	n := copy(dst, src)
	fmt.Println(n, dst)

	buf := []int{1, 2, 3, 4, 5}
	copy(buf[1:], buf)
	fmt.Println(buf)
}
```

The first copy moves two elements into short dst. The second shifts an overlapping region inside one slice.

## Pitfalls

- **Short dst truncates silently.** Ignoring the return drops tail elements.
- **Nil dst/src.** copy(nil, src) returns 0 without panic.
- **Same len different cap.** copy never grows dst; resize or append first.
- **Shallow struct copy.** Pointer fields alias after copy.
- **String to bytes.** Use copy on []byte; strings are immutable so convert first.
