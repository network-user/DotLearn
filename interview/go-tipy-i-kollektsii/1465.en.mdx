# What is an interface in Go and why use it?

## In short

A Go interface is a method set without implementation. Any type with those methods satisfies the interface implicitly. That gives polymorphism without inheritance or rigid class hierarchies.

## How to answer in an interview

- I state: an interface describes behavior (a contract); concrete structs implement it with methods.
- I cite stdlib: `io.Reader`, `fmt.Stringer`, `error` as small interfaces.
- I stress implicit satisfaction: there is no `implements` keyword.
- I add honestly: the empty interface `any` accepts everything, but in new code I prefer concrete types or generics over unbounded `interface{}`.

## How it works

An interface value at runtime is a pair (type tab, data pointer). Assigning a concrete value checks the method set. Method calls go through a table, so polymorphic code does not know the concrete struct.

The Go idiom: declare small interfaces next to the consumer ("accept interfaces, return structs"). Wide interfaces with many methods make testing and mocks harder.

```go
package main

import "fmt"

type Notifier interface {
	Notify(msg string) error
}

type EmailSender struct {
	Address string
}

func (e EmailSender) Notify(msg string) error {
	fmt.Printf("email to %s: %s\n", e.Address, msg)
	return nil
}

type SMSPinger struct {
	Phone string
}

func (s SMSPinger) Notify(msg string) error {
	fmt.Printf("sms to %s: %s\n", s.Phone, msg)
	return nil
}

func broadcast(channels []Notifier, text string) {
	for _, ch := range channels {
		_ = ch.Notify(text)
	}
}

func main() {
	channels := []Notifier{
		EmailSender{Address: "dev@corp.local"},
		SMSPinger{Phone: "+79001234567"},
	}
	broadcast(channels, "deploy finished")
}
```

`broadcast` works with any type implementing `Notify` without listing variants. Adding `SlackWebhook` does not change `broadcast`'s signature.

Compile-time check: `var _ Notifier = (*MyType)(nil)` catches a missing method before runtime.

## Pitfalls

- **Nil interface vs typed nil.** `var p *T; var i Notifier = p` yields a non-nil interface with nil data; comparing to `nil` breaks.
- **Value vs pointer receiver.** A pointer-receiver method is not in the interface for a value type and vice versa.
- **Overwide any.** You lose type safety and add type assertions everywhere.
- **Fat interfaces in models package.** Prefer 1–2 methods at the consumer; otherwise every mock carries the whole contract.
- **Comparing interfaces.** Two interfaces are equal only when dynamic type and value match; comparing slices inside needs reflect.
