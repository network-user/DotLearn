# Как сделать составной ключ map без встроенного tuple?

## Суть

Ключ map должен быть сравниваемым типом. Составной ключ оформляю как `struct` с сравниваемыми полями (числа, строки, другие struct без slice/map/func). Так эмулирую пару или кортеж фиксированной формы.

## Что ответить на собесе

- `map[struct{...}]V` работает, если каждое поле поддерживает `==`.
- Литерал `Key{A: 1, B: "x"}` использую как ключ при lookup и insert.
- Для неупорядоченных наборов полей struct с теми же значениями сравнивается поэлементно.
- Честно оговариваю: если нужен ключ с slice или map, делаю каноническую строку или хэш с осторожностью к коллизиям.

## Как это работает

```go
package main

import "fmt"

type Coord struct {
	X int
	Y int
}

func main() {
	grid := make(map[Coord]string)

	grid[Coord{0, 0}] = "origin"
	grid[Coord{3, 4}] = "point"

	lookup := Coord{3, 4}
	fmt.Println(grid[lookup]) // point
}
```

Сравнение struct идёт рекурсивно по полям. Порядок полей в типе фиксирован при объявлении; `Coord{3,4}` и `Coord{X:3,Y:4}` эквивалентны. Хэш map вычисляется от сериализованного представления сравниваемого ключа.

## Подводные камни

- **Несравниваемые поля.** Slice, map, func и array с несравниваемым элементом ломают struct как ключ.
- **Указатели в ключе.** Сравниваются адреса, не содержимое; два разных указателя на равные значения не равны.
- **Плавающая точка.** `float64` сравнивается по IEEE правилам; NaN никогда не равен NaN.
- **Размер ключа.** Большой struct увеличивает стоимость хэширования и копирования при росте map.
- **Строковый surrogate.** Конкатенация полей в ключ требует разделителя, иначе коллизии `"ab"+"c"` vs `"a"+"bc"`.
