# Сколько слотов в bucket map в Go?

## Суть

В текущей реализации runtime каждый bucket map вмещает до 8 пар ключ-значение. При переполнении bucket получает overflow-цепочку; рост map идёт через удвоение bucket-массива и эвакуацию.

## Что ответить на собесе

- Фиксированный размер bucket: 8 элементов (деталь runtime, не гарантия языка навсегда).
- Хэш ключа выбирает bucket; коллизии укладываются в тот же bucket или overflow.
- `make(map[K]V, hint)` задаёт hint по числу элементов, не «число bucket».
- Честно оговариваю: это внутренняя деталь; на собесе важнее амортизированный O(1) и отсутствие упорядоченного обхода.

## Как это работает

```go
package main

import "fmt"

func main() {
	seen := make(map[string]int)

	keys := []string{"a", "b", "c", "d", "e", "f", "g", "h", "i"}
	for i, k := range keys {
		seen[k] = i
	}

	for k, v := range seen {
		fmt.Println(k, v)
	}
}
```

При вставке runtime вычисляет хэш, маскирует его по числу bucket и ищет свободный слот среди восьми. Если bucket заполнен, добавляется overflow bucket. При load factor выше порога таблица расширяется, и элементы постепенно переносятся (incremental evacuation).

## Подводные камни

- **Порог load factor.** Рост map не привязан жёстко к «каждые 8 ключей»; смотрю на hint при `make`.
- **Итерация недетерминирована.** Порядок `range` по map случаен и меняется между запусками.
- **Указатели в значениях.** Изменение полей по указателю в map не требует повторной вставки, но ломает интуицию о копировании.
- **Ключи без сравнения.** Slice, map и func нельзя использовать как ключ; struct сравним только если все поля сравнимы.
- **Concurrent map.** Параллельная запись без `sync.Map` или mutex приводит к fatal error.
