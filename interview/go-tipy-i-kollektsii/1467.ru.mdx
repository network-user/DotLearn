# Чем похожи abstract class и interface (взгляд Go-разработчика)?

## Суть

В Go нет abstract class как в Java. Близкий смысл даёт связка struct (состояние и часть методов) плюс interface (контракт поведения). И abstract class, и interface в классических ООП задают «что должен уметь тип» и включают полиморфизм.

## Что ответить на собесе

- Сходство: оба задают контракт для подтипов и позволяют писать код против абстракции, а не конкретной реализации.
- Различие в Go: interface только методы; «абстрактность» состояния имитирую struct с неэкспортируемыми полями и конструктором.
- Привожу маппинг: Java interface ≈ Go interface; abstract class ≈ struct + interface + embedding, без наследования.
- Честно добавляю: сравнение уместно для собеседующих из Java; в чистом Go говорю про interfaces и composition, не про abstract class буквально.

## Как это работает

**Общее.** И abstract class, и interface в ООП отделяют клиентский код от деталей реализации. Функция принимает абстракцию и работает с любым подходящим типом.

**В Go.** Интерфейс фиксирует method set. Struct хранит данные и может предоставлять общие методы. Embedding передаёт методы вверх без subclassing.

```go
package main

import "fmt"

type Speaker interface {
	Speak() string
}

type BaseCreature struct {
	species string
}

func (b BaseCreature) Species() string {
	return b.species
}

type Dog struct {
	BaseCreature
	nickname string
}

func (d Dog) Speak() string {
	return d.nickname + " says woof"
}

type Cat struct {
	BaseCreature
}

func (c Cat) Speak() string {
	return "meow from " + c.species
}

func greetAll(list []Speaker) {
	for _, s := range list {
		fmt.Println(s.Speak())
	}
}

func main() {
	animals := []Speaker{
		Dog{BaseCreature: BaseCreature{species: "canine"}, nickname: "Rex"},
		Cat{BaseCreature: BaseCreature{species: "feline"}},
	}
	greetAll(animals)
}
```

`BaseCreature` похож на кусок abstract class (общее состояние и helper), `Speaker` на interface. Нельзя инстанцировать «просто Speaker», как нельзя создать abstract class в Java.

Полиморфизм в `greetAll` такой же по идее, как вызов метода по ссылке на interface/abstract type в других языках.

## Подводные камни

- **Говорить, что в Go есть abstract class.** Формально нет; есть идиомы.
- **Множественное наследование через embedding.** Несколько embedded типов с одинаковыми методами дают неоднозначность.
- **Смешивать interface и struct в одном типе.** Interface value не несёт полей BaseCreature без assertion.
- **Толстый interface вместо abstract.** 15 методов в interface хуже, чем маленький контракт у потребителя.
- **Сравнение с C++ pure virtual.** Семантика ближе к interface, но без vtable на уровне языка в том же виде.
