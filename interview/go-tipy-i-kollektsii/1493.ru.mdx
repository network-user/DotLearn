# Какая типизация у Go?

## Суть

Go статически и строго типизирован: типы проверяются компилятором, неявных преобразований между несовместимыми типами нет. При этом есть inference через `:=` и duck typing через interfaces на этапе compile-time check.

## Что ответить на собесе

- Static typing: каждая переменная имеет фиксированный тип, ошибки ловятся до run.
- Strong typing: `int + string` не компилируется без explicit conversion.
- Interfaces дают полиморфизм без inheritance; satisfaction проверяется компилятором.
- Честно добавляю: generics с Go 1.18+ добавили parametric polymorphism, но правило explicit conversion для несвязанных типов осталось.

## Как это работает

Компилятор строит type graph, проверяет assignments, calls, comparisons. Numeric conversions требуют явного cast; interface assignment требует method set compatibility.

```go
package main

import (
	"fmt"
	"strconv"
)

func main() {
	var count int = 10
	text := "20"

	parsed, err := strconv.Atoi(text)
	if err != nil {
		panic(err)
	}

	fmt.Println(count + parsed)

	var anyHolder any = count
	if v, ok := anyHolder.(int); ok {
		fmt.Println(v * 2)
	}
}
```

`strconv.Atoi` явно переводит string в int. Type assertion достаёт int из `any`.

## Подводные камни

- **Untyped constants.** `const x = 1` гибко присваивается, пока не присвоен конкретный тип.
- **Integer overflow.** Conversion между int sizes truncates silently in spec sense.
- **Interface boxing cost.** Маленькие values в `any` могут аллоцироваться в heap.
- **Generics constraints.** Type parameter must satisfy comparable/ordered constraints explicitly.
- **Shadowing with :=.** Новая переменная может скрыть outer name и изменить тип в inner block.
