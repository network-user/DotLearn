# Что делает sort.Slice со slice?

## Суть

`sort.Slice` сортирует элементы in-place по less(i,j bool) comparator. Тот же header slice после вызова указывает на переупорядоченный backing array; новый slice не создаётся.

## Что ответить на собесе

- In-place sort: len/cap не меняются, меняется порядок элементов.
- Comparator получает индексы, не значения; inside читаю slice[i], slice[j].
- Нестабильная сортировка; для stable беру `sort.SliceStable`.
- Честно добавляю: Go 1.21+ `slices.SortFunc` предпочтительнее для нового кода, sort.Slice legacy-friendly.

## Как это работает

```go
package main

import (
	"fmt"
	"sort"
)

func main() {
	people := []struct {
		Name string
		Age  int
	}{
		{"Cy", 30},
		{"Ada", 28},
		{"Leo", 35},
	}

	sort.Slice(people, func(i, j int) bool {
		return people[i].Age < people[j].Age
	})

	for _, p := range people {
		fmt.Printf("%s %d\n", p.Name, p.Age)
	}
}
```

После вызова `people` тот же slice variable, элементы по возрасту.

## Подводные камни

- **Comparator inconsistency.** Нет strict weak ordering → panic или undefined order.
- **Concurrent sort.** Data race если другой goroutine читает slice.
- **Sort subset.** sort.Slice всего slice; subrange нужен offset trick или sort.Slice on slice segment.
- **Pointer slice reorder.** Меняются указатели, не объекты в heap.
- **Performance on tiny n.** Overhead comparator calls dominates.
