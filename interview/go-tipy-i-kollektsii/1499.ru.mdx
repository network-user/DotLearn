# Какова сложность доступа в BST?

## Суть

В сбалансированном BST search/insert/delete в среднем O(log n) по высоте. Вырожденное дерево (sorted insert) даёт высоту n и O(n) операции.

## Что ответить на собесе

- Связываю сложность с height, не с n напрямую.
- Balanced variants (AVL, red-black) держат O(log n) worst case.
- В Go stdlib нет BST; использую map или sorted slice + binary search.
- Честно добавляю: на практике предпочитаю map или B-tree через storage engine, самописный BST редок.

## Как это работает

Каждый шаг сравнения отбрасывает половину subtree. Рекурсия или цикл спускается от root до nil или target.

```go
package main

import "fmt"

type TreeNode struct {
	Val   int
	Left  *TreeNode
	Right *TreeNode
}

func search(root *TreeNode, target int) *TreeNode {
	for root != nil && root.Val != target {
		if target < root.Val {
			root = root.Left
		} else {
			root = root.Right
		}
	}
	return root
}

func main() {
	root := &TreeNode{Val: 10, Left: &TreeNode{Val: 5}, Right: &TreeNode{Val: 15}}
	fmt.Println(search(root, 5) != nil)
}
```

Глубина balanced tree ~ log n; skewed chain ~ n.

## Подводные камни

- **Sorted input.** Последовательная вставка 1..n без rebalance даёт linked list.
- **Pointer chasing.** Cache miss выше чем contiguous slice scan на малых n.
- **Duplicate keys.** BST policy (left/right equal) влияет на height.
- **Concurrent tree.** Нужны locks или immutable structures.
- **Alternative structures.** heap не substitute для ordered search.
