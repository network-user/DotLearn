# How many slots does a Go map bucket hold?

## In short

In the current runtime implementation, each map bucket holds up to 8 key-value pairs. When a bucket overflows, it chains overflow buckets; map growth doubles the bucket array and evacuates entries incrementally.

## How to answer in an interview

- Fixed bucket size: 8 entries (a runtime detail, not a permanent language guarantee).
- The key hash picks a bucket; collisions stay in that bucket or overflow chains.
- `make(map[K]V, hint)` sets an element-count hint, not a literal bucket count.
- I honestly caveat: this is internal; what matters more in interviews is amortized O(1) lookup and unordered iteration.

## How it works

```go
package main

import "fmt"

func main() {
	seen := make(map[string]int)

	keys := []string{"a", "b", "c", "d", "e", "f", "g", "h", "i"}
	for i, k := range keys {
		seen[k] = i
	}

	for k, v := range seen {
		fmt.Println(k, v)
	}
}
```

On insert the runtime hashes the key, masks it to the bucket count, and finds a free slot among eight. A full bucket gets an overflow bucket. When load factor crosses the threshold, the table grows and entries move gradually (incremental evacuation).

## Pitfalls

- **Load factor threshold.** Map growth is not strictly "every 8 keys"; I use `make` hints when size is known.
- **Nondeterministic iteration.** `range` order over a map is random and changes between runs.
- **Pointers as values.** Mutating fields through a stored pointer does not require re-insertion but breaks copy intuition.
- **Non-comparable keys.** Slices, maps, and funcs cannot be keys; a struct key works only if every field is comparable.
- **Concurrent maps.** Parallel writes without `sync.Map` or a mutex trigger a fatal error.
