# Как map в Go разруливает коллизии хешей?

## Суть

Внутри map это хеш-таблица с бакетами: при совпадении хеша ключи попадают в одну ячейку и перебираются последовательно (chaining внутри bucket). При переполнении таблица растёт и ключи перераспределяются.

## Что ответить на собесе

- Объясняю: хеш режет ключ на индекс бакета, внутри бакета до восьми слотов (в современных версиях) с overflow-бакетами при необходимости.
- Уточняю: коллизия это норма, важны равномерный хеш и сравнение ключей на равенство, а не только совпадение хеша.
- Связываю с производительностью: при плохом распределении хеша map деградирует к линейному поиску внутри цепочки бакета.
- Честно добавляю: точную внутреннюю структуру (`hmap`, `bmap`) помню концептуально; на собесе достаточно bucket + overflow + grow.

## Как это работает

Ключ проходит через runtime-хешфункцию (для строк, integer, struct с сравнимыми полями). Младшие биты хеша выбирают bucket. При вставке runtime ищет свободный слот в bucket или цепочке overflow. При lookup сравнивает полный ключ, потому что разные ключи могут дать один хеш.

Когда load factor превышает порог, map удваивает количество бакетов (grow) и постепенно эвакуирует старые записи. Итерация во время grow безопасна, но порядок ключей остаётся недетерминированным.

```go
package main

import "fmt"

func main() {
	stock := make(map[string]int)
	keys := []string{"apple", "apricot", "banana", "cherry", "date"}
	for i, k := range keys {
		stock[k] = (i + 1) * 10
	}

	lookup := "banana"
	if qty, ok := stock[lookup]; ok {
		fmt.Printf("%s -> %d\n", lookup, qty)
	}

	for k, v := range stock {
		fmt.Printf("%s=%d ", k, v)
	}
	fmt.Println()
}
```

Даже если `apple` и `apricot` попали в один bucket из-за похожего хеша, `stock["banana"]` найдёт правильную пару через сравнение строк, а не только по хешу.

Коллизии не «исправляются» выбором другого ключа: они разрешаются структурой таблицы. Пользователь влияет косвенно: избегает огромных map без нужды, не использует мутируемые ключи и следит за ростом через `len`.

## Подводные камни

- **Считать map упорядоченным.** Порядок обхода случаен и меняется между запусками; для сортировки нужен отдельный срез ключей.
- **Мутируемый ключ.** Срез или map как ключ компилируется с ошибкой; struct с указателем внутри может дать сюрпризы при изменении полей после вставки.
- **Гонки без sync.** Параллельная запись в обычный map приводит к fatal error; нужен mutex или `sync.Map`.
- **Путаница с Java HashMap.** В Go нет отдельного linked list на каждый bucket в пользовательском API; детали в runtime.
- **Предварительный make(len).** `make(map[K]V, hint)` снижает число grow при известном размере, но не убирает коллизии как явление.
