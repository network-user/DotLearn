# How do Go interfaces differ from Java and C#?

## In short

A Go interface is a method set, and type satisfaction is implicit: implement the methods and you match, with no `implements` keyword. That is structural typing, not a nominal hierarchy like Java or C#.

## How to answer in an interview

- I stress implicit satisfaction: a type becomes an `io.Reader` when it has `Read([]byte) (int, error)`.
- I say interfaces stay small (accept interfaces, return structs) and are often defined at the consumer, not the type author.
- I compare with Java/C#: explicit `implements`, later default methods, wider hierarchies.
- I honestly add that implicit matching decouples well, but the IDE shows fewer "who implements this" hints, so tests and grep matter.

## How it works

The compiler builds an itable for each (concrete type, interface) pair on first assignment. Methods dispatch through that table. The empty interface `any` holds any type, but domain code uses narrow stdlib interfaces: `fmt.Stringer`, `error`, `context.Context`.

```go
package main

import "fmt"

type Notifier interface {
	Notify(msg string)
}

type EmailGateway struct{}

func (EmailGateway) Notify(msg string) {
	fmt.Println("email:", msg)
}

type SMSGateway struct{}

func (SMSGateway) Notify(msg string) {
	fmt.Println("sms:", msg)
}

func broadcast(channels []Notifier, text string) {
	for _, ch := range channels {
		ch.Notify(text)
	}
}

func main() {
	broadcast([]Notifier{EmailGateway{}, SMSGateway{}}, "deploy done")
}
```

Neither `EmailGateway` nor `SMSGateway` declares `implements Notifier`, yet both sit in `[]Notifier`.

## Pitfalls

- **Pointer vs value receiver.** A pointer-receiver method does not satisfy an interface for the value type and vice versa; a common compile error.
- **Calling methods on nil interface.** A nil interface value panics on method call; a typed nil pointer inside an interface can panic too.
- **Fat interfaces.** A ten-method interface is hard to mock; prefer several small ones.
- **Type assertion without ok.** `v.(Concrete)` panics on mismatch; in hot paths I use `v, ok := x.(T)`.
- **Embedding is not inheritance.** Embedded structs promote methods but do not give classic OOP substitutability.
