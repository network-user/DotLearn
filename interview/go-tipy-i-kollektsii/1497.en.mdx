# What is an ACT tree?

## In short

ACT (Abstract Control Tree) is a control-flow tree in compilers: nodes model seq, if, loop, and call rather than syntactic AST details. It supports control-flow analysis and optimizations.

## How to answer in an interview

- I separate ACT from AST: AST is syntax, ACT is control edges.
- Uses include dataflow, dead code elimination, and loop transforms.
- In the Go toolchain people more often mention SSA (`GOSSAFUNC`), but ACT is the textbook cousin.
- I honestly add that deep ACT rarely appears on typical Go interviews; linking it to CFG and backend opts is enough.

## How it works

Source lowers to a tree where leaves are basic blocks and inner nodes combine control. The compiler walks ACT to build CFG and then SSA.

```go
package main

import "fmt"

func classify(n int) {
	for i := 0; i < n; i++ {
		if i%2 == 0 {
			fmt.Println(i, "even")
		} else {
			fmt.Println(i, "odd")
		}
	}
}

func main() {
	classify(3)
}
```

A loop node wraps an if in the body; ACT captures that control hierarchy, not fmt strings.

## Pitfalls

- **Confusing with AST.** Different IR passes, not interchangeable.
- **SSA dominance.** CFG/ACT groundwork for many optimizations.
- **Go escape analysis.** Lives on another IR layer, not ACT directly.
- **Interview overreach.** Do not claim gcc/llvm internals unless you do.
- **Runtime invisible.** ACT exists only at compile time.
