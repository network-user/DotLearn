# Чем nil map отличается от nil slice?

## Суть

Оба zero value это `nil`, но поведение разное: чтение из nil map безопасно (zero value + `false`), запись паникует. Nil slice читать и `append` можно; индексация `s[i]` при `len=0` паникует.

## Что ответить на собесе

- `var m map[K]V` и `var s []T` оба `== nil`.
- `m[k]` на nil map не паникует; `m[k]=v` паникует.
- `append(nilSlice, x)` легален и аллоцирует backing array.
- Честно сравниваю с JSON: nil map → `null`, пустой slice `[]T{}` → `[]`.

## Как это работает

```go
package main

import "fmt"

func main() {
	var m map[string]int
	var s []int

	v, ok := m["missing"]
	fmt.Println(v, ok) // 0 false

	s = append(s, 7)
	fmt.Println(s) // [7]

	// m["k"] = 1 // panic: assignment to nil map
}
```

Nil map это отсутствие hash table; runtime не создаёт bucket до `make`. Nil slice это header с нулевым указателем, но slice API (`len`, `cap`, `append`, `range`) определён и для nil.

## Подводные камни

- **Ложное ощущение симметрии.** Новички ожидают одинакового поведения nil коллекций.
- **Передача в функции.** Функция, которая пишет в map, должна инициализировать или документировать требование non-nil.
- **len(nil map).** `len(m)` на nil map это 0, но не означает готовности к записи.
- **Сравнение с make.** `make(map[K]V)` и `make([]T, 0)` оба не nil, но только slice сразу готов к `append`.
- **Range по nil.** `for k := range m` безопасен (0 итераций); `for _, v := range s` тоже.
