# What is an uninitialized slice in Go?

## In short

A declared but uninitialized slice (`var s []int`) gets the zero value `nil`: the array pointer is `nil`, and `len` and `cap` are 0. That is valid state, distinct from an empty but initialized slice.

## How to answer in an interview

- A slice zero value is `nil`; `s == nil` is `true`.
- `append` on a `nil` slice works and allocates a backing array when needed.
- An empty literal `[]int{}` is not `nil`, even though `len` is also 0.
- I honestly add: for JSON I distinguish `null` vs `[]` with explicit initialization when the API must return an empty array, not null.

## How it works

```go
package main

import "fmt"

func main() {
	var unset []int
	empty := []int{}

	fmt.Println(unset == nil) // true
	fmt.Println(len(unset), cap(unset)) // 0 0

	unset = append(unset, 42)
	fmt.Println(unset) // [42]

	fmt.Println(empty == nil) // false
}
```

A `nil` slice points at no memory, yet behaves like an empty collection for `len`, `cap`, and `append`. Indexing `s[i]` on a `nil` slice panics the same way as on a zero-length non-nil slice.

## Pitfalls

- **nil vs empty in APIs.** Clients may treat `null` and `[]` differently; I pick serialization behavior deliberately.
- **Passing into functions.** A `nil` slice is safe for `range` and `append`, not for indexing.
- **Comparison with make([]T, 0).** Both have `len` 0, but only `var` yields a `nil` pointer.
- **Reslicing nil.** `nil[:]` panics; I check length before slicing.
- **Leaks via subslices.** Not specific to `nil`, but after `append` from `nil` I avoid keeping stray references to a large backing array.
