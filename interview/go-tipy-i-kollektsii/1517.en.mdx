# What is map lookup time complexity?

## In short

Average and amortized lookup, insert, and delete by key are O(1). The worst case with long collision chains is O(n). Memory is O(n) with bucket-structure overhead.

## How to answer in an interview

- I say "amortized O(1)", not "strictly constant on every call".
- `v, ok := m[k]` shares the same asymptotics as a direct lookup; `ok` does not add an extra pass.
- Map growth on insert can occasionally trigger O(n) rehash spread across many operations.
- I honestly compare: for tiny fixed n a linear slice scan can win on cache locality.

## How it works

```go
package main

import "fmt"

func main() {
	index := map[int]string{
		10: "ten",
		20: "twenty",
	}

	label := index[10]
	fmt.Println(label)

	if _, found := index[99]; !found {
		fmt.Println("missing key")
	}
}
```

The key hash picks a bucket index; key comparison in the slot finishes the search. When load factor exceeds the threshold the table grows and relocation cost amortizes into later O(1) operations.

## Pitfalls

- **"Constant time" confusion.** Interviewers may probe worst case; I mention collision chains.
- **Expensive hashing.** Long string keys add cost before bucket lookup.
- **Iteration vs lookup.** Full `range` is always O(n), not the same as point access.
- **Concurrent counters.** Maps are a poor fit for unsynchronized multi-writer counters.
- **Zero value reads.** Missing keys return zero in O(1) without a separate branch, but that is easy to confuse with a stored zero without `ok`.
