# Может ли map быть упорядоченной?

## Суть

Встроенная map в Go не упорядочена: порядок `range` не фиксирован и может меняться между запусками. Insertion order не сохраняется из-за hash-bucket layout.

## Что ответить на собесе

- Отвечаю «нет» для stdlib map; порядок обхода intentionally unspecified.
- Если нужен sorted или insertion order, комбинирую map с slice ключей или беру отдельную структуру.
- Упоминаю, что random iteration order помогает ловить скрытые зависимости в тестах.
- Честно добавляю: ordered map можно собрать самому или взять пакет, но это уже не встроенный тип.

## Как это работает

Элементы лежат в bucket по hash, не в linked list по времени добавления. Для алфавитного вывода собираю ключи и сортирую:

```go
package main

import (
	"fmt"
	"sort"
)

func main() {
 stock := map[string]int{
		"pear":   4,
		"apple":  9,
		"cherry": 2,
	}

	keys := make([]string, 0, len(stock))
	for k := range stock {
		keys = append(keys, k)
	}
	sort.Strings(keys)

	for _, k := range keys {
		fmt.Printf("%s: %d\n", k, stock[k])
	}
}
```

`range stock` alone мог бы вывести ключи в любом порядке; slice ключей задаёт детерминизм.

## Подводные камни

- **Ожидание insertion order.** Как в Python 3.7+ dict; в Go так не работает.
- **JSON vs range.** `encoding/json` сортирует ключи при marshal, но это не меняет semantics map.
- **LinkedHashMap pattern.** Slice + map дублирует память и требует sync при delete.
- **Concurrent ordered view.** Сортировка snapshot ключей под lock, иначе race.
- **Small map benchmark illusion.** На 3 ключах порядок кажется стабильным, на prod ломается.
