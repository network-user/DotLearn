# Какая асимптотика lookup в map?

## Суть

Средний и амортизированный поиск, вставка и удаление по ключу O(1). Худший случай при длинных коллизионных цепочках O(n). Память O(n) с коэффициентом overhead на bucket структуру.

## Что ответить на собесе

- Говорю «амортизированный O(1)», не «строго константный в каждом вызове».
- `v, ok := m[k]` та же асимптотика, что и прямой lookup; `ok` без лишнего прохода.
- Рост map при insert может иногда триггерить O(n) rehash, размазанный по многим операциям.
- Честно сравниваю: для малого фиксированного n линейный scan slice может быть быстрее из-за cache locality.

## Как это работает

```go
package main

import "fmt"

func main() {
	index := map[int]string{
		10: "ten",
		20: "twenty",
	}

	label := index[10]
	fmt.Println(label)

	if _, found := index[99]; !found {
		fmt.Println("missing key")
	}
}
```

Хэш ключа даёт индекс bucket; сравнение ключей в слоте завершает поиск. При превышении load factor таблица расширяется, стоимость переноса амортизируется на последующие O(1) операции.

## Подводные камни

- **Путаница с «константным временем».** Интервьюер может уточнить worst case; отвечаю про коллизии.
- **Дорогой hash.** Ключи вроде длинных строк добавляют стоимость до bucket lookup.
- **Итерация vs lookup.** Полный `range` всегда O(n), не путать с точечным доступом.
- **False sharing в multi-core.** Редко на собесе, но map не для fine-grained concurrent counters без sync.
- **Zero value read.** Отсутствующий ключ возвращает zero без отдельной ветки O(1), но легко спутать с реальным zero value без `ok`.
