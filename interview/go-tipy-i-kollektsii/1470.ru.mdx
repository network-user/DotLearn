# Как снизить влияние коллизий в map?

## Суть

Коллизии в map неизбежны: runtime разруливает их цепочками в бакетах и ростом таблицы. Разработчик снижает влияние хорошими ключами, предварительным `make` с hint и избеганием pathological hash (одинаковые префиксы, атакующие ключи).

## Что ответить на собесе

- Говорю, что борьба в основном на стороне runtime; прикладной код не выбирает bucket вручную.
- Для плотных map указываю `make(map[K]V, expectedSize)` чтобы реже делать grow и перераскладку.
- Предупреждаю про пользовательские типы-ключи: нужен стабильный `==` и разумный хеш через не экспортируемые поля.
- Честно добавляю: при подозрении на hash flooding в публичном API лучше не принимать произвольные строки как ключи без лимита длины.

## Как это работает

Хеш режет ключ на индекс бакета. Много ключей в одном бакете удлиняет цепочку lookup. Grow увеличивает число бакетов и перераспределяет записи, снова снижая среднюю длину цепочки.

Практики:

```go
package main

import "fmt"

type UserID struct {
	id int64
}

func (u UserID) HashSeed() int {
	return int(u.id ^ (u.id >> 32))
}

func buildIndex(ids []int64) map[UserID]string {
	m := make(map[UserID]string, len(ids))
	for _, raw := range ids {
		key := UserID{id: raw}
		m[key] = fmt.Sprintf("user-%d", raw)
	}
	return m
}

func main() {
	ids := []int64{10, 20, 30, 40, 50}
	idx := buildIndex(ids)
	for k, v := range idx {
		fmt.Println(k.id, v)
	}
}
```

`make` с `len(ids)` уменьшает промежуточные grow. Ключ `UserID` сравнивается по полю `id`; runtime хеширует comparable struct целиком.

Если ключи приходят извне (HTTP query как map key), злоумышленник может подобрать коллизии; mitigation: canonical key type, лимит длины, или map не использовать как primary store.

## Подводные камни

- **Считать, что коллизии это баг.** Это нормальная часть хеш-таблицы.
- **Огромный map без hint.** Частые grow копируют всю таблицу и временно удваивают память.
- **Struct key с mutable slice field.** Не компилируется; с pointer полем мутация после insert ломает lookup.
- **Итерация для «упорядочивания».** Порядок случаен; сортировка ключей отдельно.
- **sync.Map как анти-коллизия.** Он про concurrency, не про качество хеша.
