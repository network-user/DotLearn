# How does Go resolve map collisions?

## In short

Go maps use buckets with a fixed slot count and overflow bucket chains when a bucket fills. Hash collisions are resolved by probing within the bucket and chaining overflow buckets, plus table growth at high load factor.

## How to answer in an interview

- Not a classic linked list per cell; a bucket array with eight key/value slots (runtime detail).
- Overflow buckets attach when eight slots are not enough.
- Incremental evacuation on resize reduces pauses on large maps.
- I honestly caveat: internals can shift between Go versions; interviews care more about amortized O(1) and hash seeding against flooding.

## How it works

```go
package main

import "fmt"

func main() {
	ids := make(map[uint64]string, 16)

	for i := uint64(0); i < 20; i++ {
		ids[i] = fmt.Sprintf("id-%d", i)
	}

	for k, v := range ids {
		fmt.Println(k, v)
	}
}
```

Keys that share a bucket index after masking land in one bucket. Lookup compares keys in bucket slots, then walks overflow chains. On growth the bucket count doubles and entries evacuate gradually.

## Pitfalls

- **"Linked list" shorthand.** Outdated; bucket plus overflow is more accurate.
- **Hash flooding.** Without per-map seeding attackers could force collisions; Go mitigates this.
- **Pointer keys.** Equality is by address, not contents; collision math matters less than key semantics.
- **Iterate during grow.** The runtime coordinates, but unsynchronized concurrent writes remain invalid.
- **Expecting sort order.** Collisions do not affect `range` order; it stays pseudo-random.
