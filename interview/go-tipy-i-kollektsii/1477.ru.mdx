# Что такое пустой интерфейс и где он «не пустой»?

## Суть

`interface{}` (или `any`) не требует методов, поэтому в него можно положить значение любого типа. Но интерфейсное значение состоит из пары (тип, данные), и проверка `== nil` часто вводит в заблуждение.

## Что ответить на собесе

- Объясняю, что пустой интерфейс это контейнер «любой конкретный тип», полезен для generic-подобных API до дженериков.
- Разделяю «nil interface» и «interface с typed nil внутри»: второй случай не равен nil.
- Показываю type switch или type assertion для безопасного извлечения типа.
- Честно добавляю: в современном Go для новых API предпочитаю generics или узкие интерфейсы, `any` оставляю для JSON, логирования и границ с динамикой.

## Как это работает

Интерфейс в рантайме хранит tab типа и указатель на данные. Неинициализированная переменная `var box any` равна nil. После присваивания конкретного значения box уже не nil, даже если внутри лежит nil-указатель.

```go
package main

import "fmt"

func describe(v any) {
	switch x := v.(type) {
	case int:
		fmt.Println("int:", x)
	case string:
		fmt.Println("string:", x)
	default:
		fmt.Printf("other %T\n", x)
	}
}

func main() {
	var empty any
	fmt.Println(empty == nil)

	var p *int
	var holder any = p
	fmt.Println(holder == nil)
	fmt.Println(p == nil)

	describe(42)
	describe("go")
}
```

`holder == nil` ложно, потому что тип `*int` уже записан в интерфейсе, хотя указатель nil. Это классическая ловушка при возврате `(any, error)`.

## Подводные камни

- **Typed nil в error.** `return (*MyErr)(nil), nil` может превратиться в non-nil error interface на вызывающей стороне.
- **Потеря статической типизации.** Чрезмерный `any` переносит ошибки в runtime вместо compile time.
- **JSON decode в map[string]any.** Числа часто приходят как `float64`, не как `int`.
- **Сравнение any.** Два any сравнимы только если dynamic type comparable; slice внутри interface сравнивать нельзя.
- **Аллокации.** Маленькие value types при упаковке в interface могут уйти в heap из-за escape analysis.
