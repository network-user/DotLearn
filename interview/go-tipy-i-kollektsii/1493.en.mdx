# What kind of typing does Go have?

## In short

Go is statically and strongly typed: the compiler checks types and there are no implicit conversions between incompatible types. Inference via `:=` and interface duck typing still compile-time checked.

## How to answer in an interview

- Static typing: every variable has a fixed type; errors surface before run.
- Strong typing: `int + string` does not compile without explicit conversion.
- Interfaces enable polymorphism without inheritance; satisfaction is verified at compile time.
- I honestly add that generics since Go 1.18 add parametric polymorphism, but explicit conversion for unrelated types remains.

## How it works

The compiler builds a type graph and checks assignments, calls, and comparisons. Numeric conversions need explicit casts; interface assignment needs method set compatibility.

```go
package main

import (
	"fmt"
	"strconv"
)

func main() {
	var count int = 10
	text := "20"

	parsed, err := strconv.Atoi(text)
	if err != nil {
		panic(err)
	}

	fmt.Println(count + parsed)

	var anyHolder any = count
	if v, ok := anyHolder.(int); ok {
		fmt.Println(v * 2)
	}
}
```

`strconv.Atoi` explicitly converts string to int. A type assertion pulls int out of `any`.

## Pitfalls

- **Untyped constants.** `const x = 1` assigns flexibly until bound to a concrete type.
- **Integer conversion.** Casting between int sizes truncates per spec rules.
- **Interface boxing cost.** Small values in `any` may heap-allocate.
- **Generic constraints.** Type parameters must satisfy comparable or ordered constraints explicitly.
- **Shadowing with :=.** A new inner variable can hide an outer name and change the type in that block.
