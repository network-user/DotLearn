# Что считает len() для строки в Go?

## Суть

`len(s)` для строки возвращает число байт в UTF-8 представлении, а не число Unicode-символов. Для ASCII один символ равен одному байту; для кириллицы и эмодзи байт на символ больше.

## Что ответить на собесе

- Строка в Go неизменяемая последовательность байт в UTF-8; `len` измеряет байты.
- Для подсчёта символов (code points) конвертирую в `[]rune` или итерирую `for range` по строке.
- Честно уточняю: для user-facing длины (лимит символов в форме) `len` недостаточен без нормализации Unicode.

## Как это работает

```go
package main

import "fmt"

func main() {
	ascii := "Hello"
	cyrillic := "Привет"

	fmt.Println(len(ascii))    // 5 байт, 5 символов
	fmt.Println(len(cyrillic)) // 12 байт, 6 символов

	runes := []rune(cyrillic)
	fmt.Println(len(runes)) // 6 code points
}
```

`for i, r := range cyrillic` даёт индексы в байтах и руны по одной; это безопасный способ обхода по символам без полного копирования в `[]rune`.

## Подводные камни

- **Суррогатные пары и комбинирующие знаки.** `len` и `[]rune` не считают графемы; для «видимых символов» нужна нормализация.
- **Срезы строки по байтовому индексу.** `s[1:3]` может разрезать UTF-8 руну пополам и дать невалидную строку.
- **Сравнение длин локалей.** Один и тот же текст в NFC и NFD даёт разный `len` в байтах.
- **Производительность.** `[]rune(s)` аллоцирует; для длинных строк лучше `range` или `utf8.RuneCountInString`.
- **Путаница с []byte.** `len` одинаков для `string` и соответствующего `[]byte`, но семантика символов та же байтовая.
