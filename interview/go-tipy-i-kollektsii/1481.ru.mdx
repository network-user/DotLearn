# Как устроен slice внутри?

## Суть

Slice это заголовок из трёх полей: указатель на underlying array, length и capacity. Несколько slice могут делить один массив, поэтому мутации через общий backing array видны всем срезам.

## Что ответить на собесе

- Объясняю triplet pointer/len/cap и что slice сам по себе не хранит элементы.
- Показываю создание через `make([]T, len, cap)` и sub-slice `arr[1:4]`.
- Говорю, что `append` при нехватке cap аллоцирует новый массив и копирует данные.
- Честно добавляю: из-за shared backing легко получить subtle bug, если не понимать aliasing; иногда нужна явная копия `append([]T(nil), src...)`.

## Как это работает

Массив фиксирован, slice задаёт окно. `len` ограничивает доступные элементы, `cap` сколько можно расширить без reallocate начиная с первого индекса окна.

```go
package main

import "fmt"

func main() {
	base := []int{10, 20, 30, 40}
	window := base[1:3]
	fmt.Println(window, len(window), cap(window))

	window[0] = 99
	fmt.Println(base)

	grown := append(window, 50)
	fmt.Println(grown, cap(grown))
	base[2] = 777
	fmt.Println(grown)
}
```

После `window[0] = 99` меняется `base[1]`. После `append`, если cap позволил расшириться в том же массиве, изменение `base[2]` может отразиться на `grown`.

## Подводные камни

- **Subslice и append.** Append к маленькому subslicе может перезаписать хвост shared array.
- **Nil slice vs empty.** `var s []int` и `[]int{}` ведут себя по-разному в JSON и некоторых API, хотя len=0 у обоих.
- **Copy length.** `copy(dst, src)` копирует min(len(dst), len(src)), не cap.
- **3-index slice.** `s[low:high:max]` ограничивает cap и защищает от случайного append в shared region.
- **Range по указателям.** `[]*T` в range даёт копию pointer value, не deep copy объектов.
