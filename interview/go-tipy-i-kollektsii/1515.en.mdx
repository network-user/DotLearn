# Is map lookup always O(1)?

## In short

Average and amortized key lookup is O(1), but the worst case with many collisions in a bucket chain grows to O(n) over chain length. Go rebalances the map at high load factor to keep chains short.

## How to answer in an interview

- Hash plus bucket index gives a fast path in the typical case.
- Collisions share a bucket and overflow buckets; long chains slow lookup.
- Hash-flooding attacks can degrade toward O(n); Go mitigates with per-map hash seeding.
- I honestly say: for hard real-time or adversarial input I consider trees or sorted slices, not maps.

## How it works

```go
package main

import "fmt"

func main() {
	cache := make(map[string]int)
	cache["user:42"] = 1

	if score, ok := cache["user:42"]; ok {
		fmt.Println("hit", score)
	}

	delete(cache, "user:42")
	_, still := cache["user:42"]
	fmt.Println("after delete", still)
}
```

The runtime hashes the key, picks a bucket, and compares keys in slots. On growth entries evacuate into a larger bucket array so average chain length stays bounded.

## Pitfalls

- **Heavy keys.** Large struct keys raise comparison cost inside a bucket even with O(1) indexing.
- **Iteration is O(n).** A full walk is always linear in map size.
- **Memory vs speed.** Maps always pay bucket overhead; tiny N sometimes loses to linear slice scan.
- **Concurrent access.** Not about big-O, but lookup during unsynchronized writes races or panics.
- **Deleted keys.** `delete` does not compact immediately; memory may linger until GC and optional shrink.
