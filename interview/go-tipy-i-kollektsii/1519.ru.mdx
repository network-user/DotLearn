# Где объявлять интерфейсы в Go-проекте?

## Суть

Идиома Go: интерфейс объявляю на стороне потребителя (consumer), а не поставщика (producer). Пакет, который вызывает поведение, описывает минимальный контракт; реализации в других пакетах удовлетворяют его неявно.

## Что ответить на собесе

- Маленькие интерфейсы рядом с кодом, который их принимает (`func New(repo Store)` в том же пакете).
- Реализация не импортирует интерфейс ради `implements`; компилятор проверяет method set.
- Стандартная библиотека задаёт эталон: `io.Reader` в `io`, реализации в `os`, `bytes`, `net`.
- Честно добавляю: shared domain interfaces в общем пакете допустимы, если контракт действительно общий, не «на всякий случай».

## Как это работает

```go
package report

type Exporter interface {
	Export(rows [][]string) error
}

type Service struct {
	out Exporter
}

func NewService(out Exporter) *Service {
	return &Service{out: out}
}

func (s *Service) Run(rows [][]string) error {
	return s.out.Export(rows)
}
```

Пакет `csvwriter` реализует `Export` без импорта `report`. В тестах подставляю fake `Exporter` в `NewService`, не трогая production реализацию.

## Подводные камни

- **Интерфейс у реализации.** Тянет лишние зависимости и раздувает API surface пакета.
- **God interface.** Один интерфейс на 15 методов ломает подмену и моки.
- **Циклические импорты.** Общий interfaces-пакет иногда костыль; лучше сузить контракт у consumer.
- **Экспорт интерфейса из internal.** Публичный API пакета не должен утекать через огромные interface types.
- **compile-time assert.** `var _ Exporter = (*CSV)(nil)` в пакете реализации полезен, но не обязателен.
