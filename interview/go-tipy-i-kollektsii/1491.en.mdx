# Is a nil slice equal to an uninitialized one?

## In short

`var s []int` and `var s []int = nil` are the same zero-value slice header: pointer=nil, len=0, cap=0. Both satisfy `s == nil`. The distinction appears only against a non-nil empty slice `[]int{}`.

## How to answer in an interview

- A zero-value slice is already nil; explicit `= nil` changes nothing.
- `append` on a nil slice works and allocates a backing array.
- `len`/`cap` of a nil slice are 0; `range` is safe and runs zero times.
- I honestly add that JSON/API layers treat nil slice and `[]T{}` differently (`null` vs `[]`) even when len is zero.

## How it works

A slice header without an underlying array is a nil slice. After the first successful `append`, the pointer becomes non-nil.

```go
package main

import (
	"encoding/json"
	"fmt"
)

func main() {
	var declared []int
	var explicit []int = nil
	empty := []int{}

	fmt.Println(declared == nil, explicit == nil, empty == nil)

	declared = append(declared, 1)
	fmt.Println(declared, len(declared))

	jNil, _ := json.Marshal([]int(nil))
	jEmpty, _ := json.Marshal([]int{})
	fmt.Println(string(jNil), string(jEmpty))
}
```

`declared` and `explicit` are equivalent. `empty` is a non-nil header; JSON separates null from an empty array.

## Pitfalls

- **nil vs empty in JSON.** Clients break when they always expect an array.
- **reflect.DeepEqual.** `[]int(nil)` is not equal to `[]int{}`.
- **SQL/array drivers.** NULL vs empty array in databases.
- **Passing nil slices.** Fine: `len=0`, safe to range.
- **Pointer to slice.** `var p *[]int` is a different story from a nil slice value.
