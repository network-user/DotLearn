# Is an interface a type in Go?

## In short

Yes: `interface` is its own type that specifies a method set. Any type with those methods satisfies the interface implicitly (structural typing). An interface value stores dynamic type and data (a type + data pair).

## How to answer in an interview

- I declare `var w io.Writer` and assign `bytes.Buffer`, `os.File`, or any type with `Write([]byte)(int, error)`.
- The empty interface `any` holds any value; I recover concrete types with type assertions or switches.
- A nil interface value differs from a typed nil pointer stored inside a non-nil interface.
- I honestly caveat: interfaces on hot paths add indirect calls and can complicate escape analysis.

## How it works

```go
package main

import "fmt"

type Notifier interface {
	Notify(msg string)
}

type SMS struct{}

func (SMS) Notify(msg string) { fmt.Println("sms:", msg) }

type Email struct{}

func (Email) Notify(msg string) { fmt.Println("email:", msg) }

func broadcast(n Notifier, msg string) {
	n.Notify(msg)
}

func main() {
	channels := []Notifier{SMS{}, Email{}}
	for _, ch := range channels {
		broadcast(ch, "deploy done")
	}
}
```

The compiler checks method sets at compile time for known types. At runtime an interface value is two pointers: an itable for methods and a data pointer to the concrete value.

## Pitfalls

- **Typed nil inside interface.** `var p *T = nil; var i interface{} = p` yields a non-nil interface wrapping a nil pointer.
- **Pointer vs value receivers.** Methods only on `*T` do not make bare `T` satisfiable.
- **Large values in interfaces.** Calls may allocate when the value does not fit in the inline data word.
- **any everywhere.** Static checks disappear; I prefer small interfaces at the consumer.
- **Comparing interfaces.** Two interfaces are equal with the same dynamic type and comparable data; slices/maps inside break equality.
