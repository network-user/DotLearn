# Can a map be ordered?

## In short

The built-in Go map is not ordered: `range` order is unspecified and may change between runs. Insertion order is not preserved because of hash-bucket layout.

## How to answer in an interview

- I answer no for the stdlib map; iteration order is intentionally unspecified.
- For sorted or insertion order I combine a map with a key slice or use a dedicated structure.
- I mention randomized iteration helps catch hidden order dependencies in tests.
- I honestly add that you can build or import an ordered map, but that is no longer the built-in type.

## How it works

Entries sit in buckets by hash, not in a time-ordered linked list. For alphabetical output I collect and sort keys:

```go
package main

import (
	"fmt"
	"sort"
)

func main() {
	stock := map[string]int{
		"pear":   4,
		"apple":  9,
		"cherry": 2,
	}

	keys := make([]string, 0, len(stock))
	for k := range stock {
		keys = append(keys, k)
	}
	sort.Strings(keys)

	for _, k := range keys {
		fmt.Printf("%s: %d\n", k, stock[k])
	}
}
```

`range stock` alone could print keys in any order; the key slice makes output deterministic.

## Pitfalls

- **Expecting insertion order.** Unlike Python 3.7+ dicts; Go does not work that way.
- **JSON vs range.** `encoding/json` sorts keys when marshaling, but map semantics unchanged.
- **LinkedHashMap pattern.** Slice plus map doubles memory and needs careful delete sync.
- **Concurrent ordered view.** Sort a key snapshot under lock or you risk races.
- **Tiny map illusion.** Three keys may look stable in dev; production rehash breaks assumptions.
