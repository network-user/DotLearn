# What are the ways to create a map in Go?

## In short

Three main paths: `make(map[K]V)` or with a hint, a composite literal `map[K]V{...}`, and `var m map[K]V` followed by `make`. You cannot write to a nil map before initialization; reads return the zero value.

## How to answer in an interview

- `make(map[string]int, 64)` passes a hint for the expected element count.
- A literal inserts starter pairs and is ready for writes immediately.
- `var m map[K]V` is nil; `m["k"]=v` panics until `make` or a literal runs.
- I honestly separate: nil maps marshal to JSON `null`; an empty initialized map becomes `{}`.

## How it works

```go
package main

import "fmt"

func main() {
	hinted := make(map[string]int, 4)
	hinted["cpu"] = 8

	preloaded := map[string]int{
		"rx": 1,
		"tx": 2,
	}

	var delayed map[string]int
	fmt.Println(delayed == nil) // true
	delayed = make(map[string]int)
	delayed["ok"] = 1

	fmt.Println(hinted, preloaded, delayed)
}
```

`make` allocates the hash table header and bucket array. A literal does the same with an initial key set. Package-level `var` suits delayed init in constructors, but `make` is mandatory before writes.

## Pitfalls

- **Writes to nil maps.** Runtime panic; distinct from an empty but initialized map.
- **Hint is not a cap.** `make(..., n)` does not forbid inserting more than n entries.
- **Map assignment copies header.** Values are not deep-copied; cloning needs an explicit walk.
- **Iterate and delete.** Deleting the current key during `range` is safe; arbitrary inserts during iteration still deserve caution.
- **Concurrent init.** Two goroutines calling `make` into one variable without synchronization is a program-level data race.
