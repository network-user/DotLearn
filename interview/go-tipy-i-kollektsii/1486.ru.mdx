# Как map хранит ключ больше 128 байт?

## Суть

Runtime Go кладёт в bucket маленькие ключи inline, а для ключей крупнее порога (128 байт) сохраняет указатель на аллоцированную копию. Так map не таскает большие struct при каждом сравнении в bucket.

## Что ответить на собесе

- Объясняю оптимизацию runtime: inline vs indirect key storage.
- Порог 128 байт это деталь реализации, не spec; на собесе достаточно идеи «большой ключ по указателю».
- Comparable struct-key по-прежнему сравнивается по содержимому, не по адресу указателя.
- Честно добавляю: для очень больших ключей лучше map[string]V с каноническим string id, чем тяжёлый struct-key.

## Как это работает

При insert ключ копируется в heap если не помещается в slot bucket. Hash и equality работают с фактическим значением ключа. Lookup сравнивает содержимое, поэтому две разные переменные с одинаковым `[200]byte` найдут одну запись.

```go
package main

import "fmt"

type WideKey struct {
	Payload [200]byte
}

func main() {
	store := make(map[WideKey]string)

	k1 := WideKey{}
	k1.Payload[0] = 7

	store[k1] = "cached"

	k2 := WideKey{}
	k2.Payload[0] = 7

	fmt.Println(store[k2])
}
```

`k1` и `k2` разные переменные, но equal как ключи, значение читается через `k2`.

## Подводные камни

- **Копирование при insert.** Большой key value копируется в map; частые insert дороги.
- **Путать с pointer key.** `map[*WideKey]V` сравнивает адреса, не содержимое struct.
- **Изменение key после insert.** Мутировать `[N]byte` внутри локальной переменной не обновляет ключ в map.
- **False sharing порога.** Полагаться на 128 байт в API-дизайне не стоит; версия Go может изменить константу.
- **GC pressure.** Много больших ключей увеличивает heap и время mark phase.
