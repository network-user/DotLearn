# What properties must a hash function have for maps?

## In short

A map hash must be deterministic, fast, and spread keys evenly across bucket indices. Collisions are inevitable but should stay rare; the Go runtime picks the hash per type.

## How to answer in an interview

- I list: same key always same hash, different keys spread uniformly, cheap computation on the hot path.
- I explain collisions are resolved with chains inside buckets, so a bad hash degrades toward O(n).
- I note built-in maps do not accept a custom hash; the algorithm lives in the runtime.
- I honestly add that for my own hash tables I use proven functions (FNV, xxhash), and for maps I rely on stdlib.

## How it works

On insert/read the key is hashed and low bits pick a bucket. A good hash keeps chains short. Go uses fast built-in functions with per-process seed for string keys to mitigate HashDoS.

Standalone string hash example (not the map API):

```go
package main

import (
	"fmt"
	"hash/fnv"
)

func digest(label string) uint32 {
	h := fnv.New32a()
	_, _ = h.Write([]byte(label))
	return h.Sum32()
}

func main() {
	for _, word := range []string{"go", "map", "hash"} {
		fmt.Printf("%s -> %d\n", word, digest(word))
	}
}
```

The same `word` always yields the same digest; different words usually produce different numbers.

## Pitfalls

- **Confusing hash with equality.** Equal hashes do not imply equal keys; comparison still uses `==`.
- **Float keys.** NaN and -0 vs +0 break equality and distribution expectations.
- **String map attacks.** Without randomized seed an attacker could force collisions; Go maps account for this.
- **Weak lower bits.** Taking low bits of a poor hash still skews buckets.
- **Custom map types.** You cannot plug a custom hash into `map`; build a separate structure instead.
