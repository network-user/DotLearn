# Where should you declare interfaces in a Go project?

## In short

Go idiom: declare the interface on the consumer side, not the producer. The package that calls behavior defines the minimal contract; implementations in other packages satisfy it implicitly.

## How to answer in an interview

- Small interfaces live next to code that accepts them (`func New(repo Store)` in the same package).
- Implementations do not import the interface to "implement"; the compiler checks the method set.
- The standard library models this: `io.Reader` lives in `io`, implementations in `os`, `bytes`, `net`.
- I honestly add: shared domain interfaces in a common package are fine when the contract is truly shared, not speculative.

## How it works

```go
package report

type Exporter interface {
	Export(rows [][]string) error
}

type Service struct {
	out Exporter
}

func NewService(out Exporter) *Service {
	return &Service{out: out}
}

func (s *Service) Run(rows [][]string) error {
	return s.out.Export(rows)
}
```

Package `csvwriter` implements `Export` without importing `report`. In tests I inject a fake `Exporter` into `NewService` without touching production code.

## Pitfalls

- **Interface beside implementation.** Pulls extra dependencies and bloats the provider API.
- **God interface.** Fifteen methods on one interface breaks substitution and mocks.
- **Import cycles.** A shared interfaces package is sometimes a band-aid; narrowing at the consumer is better.
- **Leaking internal interfaces.** Public package API should not expose huge interface types.
- **Compile-time assert.** `var _ Exporter = (*CSV)(nil)` in the implementation package helps but is optional.
