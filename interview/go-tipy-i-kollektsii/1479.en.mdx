# Can you store a function in any?

## In short

Yes: functions in Go are first-class values, and `any` (`interface{}`) holds any concrete type, including func. To call it you need a type assertion or type switch to the exact signature.

## How to answer in an interview

- I confirm that a func can be assigned to a variable of type `any`, like int or struct.
- I explain that without an assertion the compiler does not know the signature, so calls go through a type cast.
- I show the pattern `if fn, ok := v.(func(int) int); ok { ... }`.
- I honestly add that storing callbacks in `any` is rarely worth it; a typed func or a small single-method interface is cleaner.

## How it works

`any` stores the function's dynamic type including parameters and results. `func()` and `func(int)` are different types for assertion. Closures pack into `any` as well.

```go
package main

import "fmt"

func runIfCallable(box any) {
	switch fn := box.(type) {
	case func():
		fn()
	case func(string) string:
		fmt.Println(fn("ping"))
	default:
		fmt.Println("unsupported callable")
	}
}

func main() {
	greet := func() { fmt.Println("hi from closure") }
	runIfCallable(greet)

	upper := func(s string) string { return s + "!" }
	runIfCallable(upper)

	runIfCallable(42)
}
```

The first call hits `case func()`, the second another signature, and the integer lands in default.

## Pitfalls

- **Nil func in any.** A typed nil function inside an interface is not a nil interface; calling it may panic.
- **Different signatures.** You cannot assert once to `func()` and pass `func(int)` without another branch.
- **Comparing func values.** Two func values compare only when both are nil; otherwise `==` does not compile.
- **Reflect overhead.** Unknown signatures force reflect, which is slow and brittle.
- **Generics alternative.** For callbacks of one shape, `[]func(Context) error` is safer than `[]any`.
