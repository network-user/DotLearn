# Can an interface add fields to a struct?

## In short

No. A Go interface describes methods only; it holds no data. Fields live in a struct (or another concrete type); the interface only demands behavior those types must provide.

## How to answer in an interview

- I answer "no" and contrast with Java/C# abstract classes that may carry fields and implemented methods.
- I explain composition: shared fields go in an embedded struct, shared behavior in an interface.
- I mention embedding: an embedded type promotes methods outward, but those are not "interface fields".
- I add honestly: when you need both contract and shared state, use struct + interface together instead of expecting magic from the interface.

## How it works

Declaring `type Writer interface { Write([]byte) (int, error) }` does not reserve memory for fields. An interface value stores a pointer to concrete data and a method table. Fields like `Title`, `Body` are declared only on structs.

The "base struct + interface" pattern:

```go
package main

import "fmt"

type Document struct {
	Title string
	Body  string
}

func (d Document) WordCount() int {
	return len(d.Body)
}

type Measurable interface {
	WordCount() int
}

type Report struct {
	Document
	Author string
}

func summarize(m Measurable) {
	fmt.Printf("words=%d\n", m.WordCount())
}

func main() {
	r := Report{
		Document: Document{Title: "Q1", Body: "revenue growth"},
		Author:   "finance",
	}
	summarize(r)
}
```

`Report` embeds `Document` and automatically gets `WordCount`, satisfying `Measurable`. The interface did not add `Title`; embedding supplied it.

For polymorphic field access Go uses type assertion, reflection, or getters on the interface (which bloats the contract and is usually undesirable).

## Pitfalls

- **Expecting field inheritance.** Candidates from OOP languages look for `extends`; Go has composition only.
- **Embedding and shadowing.** An inner field with the same name as an outer one hides access; methods promote but which field is read is not always obvious.
- **Getters on the interface instead of fields.** `Name() string` on the interface works but turns data into API and complicates evolution.
- **Empty interface as "any struct".** `any` does not expose fields without assertion.
- **Nil pointer embedding.** A nil embedded pointer does not promote value-receiver methods; calls panic.
