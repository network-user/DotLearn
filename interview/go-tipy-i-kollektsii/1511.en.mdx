# How does the runtime compute slice capacity?

## In short

`cap(s)` is the maximum number of elements from the slice's first element to the end of the backing array. For `arr[low:high]`, capacity is `len(arr) - low` even when `len(slice) = high - low` is smaller. `append` grows `cap` on reallocation using runtime growth rules.

## How to answer in an interview

- An offset reslice lowers cap: `arr[2:4]` on a length-5 array gives `len=2`, `cap=3`.
- `make([]T, len, cap)` sets cap explicitly; `append` within cap avoids reallocation.
- Doubling cap on growth is typical, but the exact formula is implementation-defined.
- I honestly say: for hot paths I pre-set cap when I know an upper bound on size.

## How it works

```go
package main

import "fmt"

func main() {
	arr := [6]int{1, 2, 3, 4, 5, 6}
	part := arr[2:4]

	fmt.Println("len:", len(part)) // 2  (3,4)
	fmt.Println("cap:", cap(part)) // 4  (3,4,5,6)

	grown := append(part, 7, 8)
	fmt.Println(grown, len(grown), cap(grown))
}
```

Before `append`, `part` starts at `arr[2]` and can grow in place through `arr[5]`. When `append` exceeds `cap`, a new array with larger cap is allocated, the slice pointer updates, and older subslices keep the previous backing store.

## Pitfalls

- **Hidden capacity after truncation.** A short `len` does not mean a small `cap`; `append` may write into an invisible tail.
- **Reallocation breaks aliases.** After growth one slice may point at a new array while a sibling subslice stays on the old one.
- **make(len) without cap.** `cap` defaults to `len`; you cannot `append` past `len` without reallocation until `len < cap` is false.
- **copy and destination cap.** `copy` is limited by `min(len(dst), len(src))`; it does not extend dst cap.
- **Huge prealloc.** An oversized cap reserves memory GC cannot reclaim while any slice references it.
