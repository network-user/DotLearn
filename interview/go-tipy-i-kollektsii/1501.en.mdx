# What is Big O notation?

## In short

Big O gives an upper asymptotic bound on how time or memory grows with input size. Constants and lower-order terms drop out.

## How to answer in an interview

- Focus on the dominant term at large n.
- Examples: index access O(1), linear scan O(n), nested loops O(n²).
- Separate time vs space complexity.
- I honestly add that on a real Go service I use pprof; Big O does not replace measurement.

## How it works

Count operations as f(n), classify growth. One `for range` is linear; stdlib sort is O(n log n).

```go
package main

import "fmt"

func contains(haystack []int, needle int) bool {
	for _, v := range haystack {
		if v == needle {
			return true
		}
	}
	return false
}

func main() {
	data := make([]int, 1000)
	fmt.Println(contains(data, 42))
}
```

`contains` is O(n) time, O(1) extra memory.

## Pitfalls

- **Worst vs average.** Quicksort averages O(n log n), worst O(n²) without care.
- **Multiple inputs.** O(n+m) for two slices, not always one n.
- **Recursion space.** O(1) time can still be O(n) stack.
- **Misleading constants.** O(n) at tiny n can beat O(log n) with huge overhead.
- **Parallelism.** Big O rarely models cores without extra assumptions.
