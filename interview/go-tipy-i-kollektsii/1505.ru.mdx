# Чему равен необъявленный slice?

## Суть

Объявленный без инициализации slice (`var s []int`) получает zero value `nil`: указатель на массив равен `nil`, `len` и `cap` равны 0. Это валидное состояние, отличное от пустого, но инициализированного slice.

## Что ответить на собесе

- Zero value slice это `nil`; сравнение `s == nil` даёт `true`.
- `append` на `nil` slice работает и создаёт backing array при необходимости.
- Пустой литерал `[]int{}` уже не `nil`, хотя `len` тоже 0.
- Честно добавляю: JSON `null` vs `[]` различаю явной инициализацией, если API требует пустой массив, а не null.

## Как это работает

```go
package main

import "fmt"

func main() {
	var unset []int
	empty := []int{}

	fmt.Println(unset == nil) // true
	fmt.Println(len(unset), cap(unset)) // 0 0

	unset = append(unset, 42)
	fmt.Println(unset) // [42]

	fmt.Println(empty == nil) // false
}
```

`nil` slice не указывает на память, но ведёт себя как пустая коллекция для `len`, `cap` и `append`. Операции чтения `s[i]` на `nil` slice паникуют так же, как на пустом с нулевой длиной.

## Подводные камни

- **nil vs empty в API.** Клиенты могут трактовать `null` и `[]` по-разному; для сериализации выбираю осознанно.
- **Передача в функции.** `nil` slice безопасен для `range` и `append`, но не для индексации.
- **Сравнение с make([]T, 0).** Оба имеют `len` 0, но только `var` даёт `nil` указатель.
- **Ре-слайс nil.** `nil[:]` паникует; проверяю длину перед срезом.
- **Утечки через subslices.** Не относится к `nil`, но после `append` из `nil` важно не держать лишние ссылки на большой backing array.
