# Какова сложность доступа к элементу map?

## Суть

Доступ по ключу в Go `map` в среднем выполняется за O(1), потому что под капотом хеш-таблица с корзинами. В худшем случае, когда все ключи попадают в одну корзину, поиск вырождается в O(n).

## Что ответить на собесе

- Говорю, что lookup в map амортизированно константный: хеш ключа выбирает корзину, внутри ищется пара ключ-значение.
- Уточняю, что худший случай теоретически линейный при массовых коллизиях или злонамеренных ключах.
- Показываю идиому `value, ok := m[key]` для проверки наличия ключа без лишних аллокаций.
- Честно добавляю: на практике для обычных ключей и разумных размеров map ведёт себя как O(1), но полагаться на порядок обхода или «магическую» стабильность нельзя.

## Как это работы

Go хранит map как массив корзин (buckets). Ключ прогоняется через хеш-функцию, младшие биты хеша выбирают корзину. Внутри корзины лежит небольшой массив пар и, при переполнении, overflow-цепочки. Чтение `m[key]` вычисляет хеш, находит корзину и сравнивает ключи внутри неё.

При росте map runtime увеличивает число корзин и перераспределяет элементы, чтобы средняя длина цепочки оставалась маленькой. Именно поэтому insert/delete/lookup в типичном случае не зависят линейно от размера всей map.

Пример чтения с проверкой существования:

```go
package main

import "fmt"

func main() {
	inventory := map[string]int{
		"apple":  12,
		"pear":   7,
		"orange": 3,
	}

	qty, found := inventory["pear"]
	if found {
		fmt.Printf("pear stock: %d\n", qty)
	}

	missing, ok := inventory["mango"]
	fmt.Printf("mango ok=%v qty=%d\n", ok, missing)
}
```

Здесь `found`/`ok` отделяет «ключа нет» от «ключ есть и значение ноль», что для счётчиков критично.

## Подводные камни

- **Нулевое значение vs отсутствие.** Без второго bool `ok` нельзя отличить отсутствующий ключ от нулевого значения типа.
- **Коллизии и DoS.** Предсказуемые или подобранные ключи могут раздуть одну корзину; для внешних ключей иногда нужны защищённые map или другая структура.
- **Concurrent access.** Одновременное чтение и запись без синхронизации даёт data race; для shared state использую `sync.Map` или mutex.
- **Итерация во время мутации.** Изменение map в цикле `range` по той же map не определено и может паниковать.
- **Сравнимые ключи.** Ключ должен поддерживать `==`; слайсы и map как ключи не компилируются.
