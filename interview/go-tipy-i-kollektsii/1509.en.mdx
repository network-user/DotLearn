# Where does the slice header pointer point?

## In short

A slice header holds a pointer to the first element of the backing array (or `nil` for a `nil` slice), plus `len` and `cap`. A slice `arr[low:high]` moves the pointer to `&arr[low]` and reduces `cap` by `low`.

## How to answer in an interview

- Three fields: data pointer, length, capacity; a slice is a window descriptor, not the array itself.
- Slicing an array or another slice shares the backing store without copying.
- A `nil` slice has a `nil` pointer and `len=cap=0`; distinguish it from `make([]T, 0)`.
- I honestly add: passing a slice into a function copies the header (three words), not the elements.

## How it works

```go
package main

import "fmt"

func main() {
	arr := [5]int{10, 20, 30, 40, 50}
	window := arr[1:4]

	fmt.Println(window)           // [20 30 40]
	fmt.Println(len(window), cap(window)) // 3 4
	fmt.Println(&arr[1] == &window[0])    // true
}
```

`window[0]` shares the address of `arr[1]`. The slice pointer targets the start of the visible range. `cap` counts elements from that address to the end of the original array (here 4: slots 20, 30, 40, 50).

## Pitfalls

- **Aliasing.** Two slices over one array see each other's mutations.
- **append and pointer.** After a reallocating `append`, the pointer may change; old subslices stay on the old array.
- **Bounds on reslice.** `s[low:high]` panics when `high > cap`, not merely when beyond some logical length.
- **Empty but non-nil.** `make([]int, 0)` may use a non-nil zero-length backing store (runtime detail), but `len` is still 0.
- **unsafe escape hatch.** `unsafe` can expose the pointer; in normal code I rely on `len`/`cap`, not addresses.
