# Can you use an array as a map key?

## In short

A fixed array (`[N]T`) may be a map key when its elements are comparable. A slice (`[]T`) may not, because slices are not comparable. Interviews often confuse the two.

## How to answer in an interview

- I answer: array yes, slice no, because only arrays support whole-value `==`.
- Example: `map[[2]int]string` compiles, `map[[]int]string` does not.
- For dynamic keys I serialize to string or hash canonical bytes.
- I honestly add that array keys are awkward for large N; production code usually uses string or id keys.

## How it works

Arrays compare element-wise when the element type is comparable. Two array variables with the same contents are equal and map to one bucket.

```go
package main

import "fmt"

func main() {
	grid := make(map[[2]int]string)

	pos := [2]int{3, 4}
	grid[pos] = "start"

	lookup := [2]int{3, 4}
	fmt.Println(grid[lookup])

	// grid2 := make(map[[]int]string) // compile error
}
```

`lookup` matches `pos` by value, so the entry is found without pointers.

## Pitfalls

- **Slice vs array wording.** Candidates often say "arrays cannot"; I clarify Go terminology.
- **Array with incomparable elements.** `[3][]int` is not a valid key type.
- **Large array keys.** Insert copies and equality checks get expensive.
- **Mutating a local array.** Changing bytes after insert does not update the key in the map.
- **Slice to array conversion.** Go 1.20+ `array := [N]T(slice)` helps when length is known upfront.
