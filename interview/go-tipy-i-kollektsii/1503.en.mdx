# What does sort.Slice do to a slice?

## In short

`sort.Slice` reorders elements in place using a less(i, j bool) comparator. The same slice header then points to a permuted backing array; no new slice is allocated.

## How to answer in an interview

- In-place sort: len and cap stay the same, order changes.
- The comparator receives indexes; read slice[i] and slice[j] inside.
- It is unstable; use `sort.SliceStable` when equal keys must keep order.
- I honestly add that Go 1.21+ `slices.SortFunc` is preferred for new code; sort.Slice stays for legacy.

## How it works

```go
package main

import (
	"fmt"
	"sort"
)

func main() {
	people := []struct {
		Name string
		Age  int
	}{
		{"Cy", 30},
		{"Ada", 28},
		{"Leo", 35},
	}

	sort.Slice(people, func(i, j int) bool {
		return people[i].Age < people[j].Age
	})

	for _, p := range people {
		fmt.Printf("%s %d\n", p.Name, p.Age)
	}
}
```

After the call `people` is the same variable, sorted by age.

## Pitfalls

- **Inconsistent comparator.** Not a strict weak ordering → panic or garbage order.
- **Concurrent sort.** Other goroutines reading the slice cause a data race.
- **Sorting a subrange.** sort.Slice sorts the whole slice; subranges need tricks or slices.SortFunc on a segment.
- **Pointer slices.** Reorders pointers, not heap objects themselves.
- **Tiny n cost.** Comparator call overhead dominates small arrays.
