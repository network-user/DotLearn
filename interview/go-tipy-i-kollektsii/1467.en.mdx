# How are abstract classes and interfaces alike (from a Go perspective)?

## In short

Go has no Java-style abstract class. The closest idea is a struct (state and some methods) plus an interface (behavior contract). In classic OOP, both abstract classes and interfaces define what a type must do and enable polymorphism.

## How to answer in an interview

- Similarity: both set a contract for subtypes and let you code against abstraction instead of a concrete implementation.
- Go difference: interfaces are methods only; I mimic "abstract" state with structs, unexported fields, and constructors.
- Mapping: Java interface ≈ Go interface; abstract class ≈ struct + interface + embedding, without inheritance.
- I add honestly: the comparison helps interviewers from Java; in pure Go I talk interfaces and composition, not literal abstract classes.

## How it works

**Common ground.** In OOP, abstract classes and interfaces separate client code from implementation details. A function takes the abstraction and works with any matching type.

**In Go.** An interface fixes the method set. A struct holds data and may provide shared methods. Embedding promotes methods upward without subclassing.

```go
package main

import "fmt"

type Speaker interface {
	Speak() string
}

type BaseCreature struct {
	species string
}

func (b BaseCreature) Species() string {
	return b.species
}

type Dog struct {
	BaseCreature
	nickname string
}

func (d Dog) Speak() string {
	return d.nickname + " says woof"
}

type Cat struct {
	BaseCreature
}

func (c Cat) Speak() string {
	return "meow from " + c.species
}

func greetAll(list []Speaker) {
	for _, s := range list {
		fmt.Println(s.Speak())
	}
}

func main() {
	animals := []Speaker{
		Dog{BaseCreature: BaseCreature{species: "canine"}, nickname: "Rex"},
		Cat{BaseCreature: BaseCreature{species: "feline"}},
	}
	greetAll(animals)
}
```

`BaseCreature` acts like part of an abstract class (shared state and helper); `Speaker` acts like an interface. You cannot instantiate "just Speaker", as you cannot instantiate an abstract class in Java.

Polymorphism in `greetAll` matches the idea of calling a method through an interface/abstract reference in other languages.

## Pitfalls

- **Claiming Go has abstract classes.** It does not formally; there are idioms.
- **Multiple inheritance via embedding.** Several embedded types with the same method name cause ambiguity.
- **Mixing interface and struct in one type.** An interface value does not carry `BaseCreature` fields without assertion.
- **Fat interface instead of abstract.** Fifteen methods on an interface is worse than a small consumer-side contract.
- **Comparison with C++ pure virtual.** Semantics are closer to interfaces, but not the same vtable story at language level.
