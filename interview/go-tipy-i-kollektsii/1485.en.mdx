# What restrictions apply to map keys?

## In short

A map key must be comparable: it must support `==` and `!=`. Slices, maps, and funcs are forbidden; arrays and structs work when their elements or fields are comparable.

## How to answer in an interview

- I state the comparable rule from the spec and do not confuse slices with arrays.
- For composite keys I check every field: float, complex, or slice fields block the key type.
- I show a typical `string` key in a word frequency task.
- I honestly add that mutable keys should be normalized to an immutable string or id instead of working around the rule.

## How it works

The hash table compares keys for equality on lookup and insert. A non-comparable type will not compile as the key type.

```go
package main

import (
	"fmt"
	"strings"
)

func main() {
	line := "go map go slice"
	freq := make(map[string]int)
	for _, token := range strings.Fields(line) {
		freq[token]++
	}
	for word, count := range freq {
		fmt.Printf("%s: %d\n", word, count)
	}
}
```

`string` is comparable, so the frequency counter works out of the box. `map[[]byte]int` fails at compile time.

## Pitfalls

- **Array vs slice.** `[4]byte` may be a key, `[]byte` may not; a common interview confusion.
- **Interface keys.** The dynamic type must be comparable; you cannot store a slice in an interface and use it as a key.
- **NaN in float fields.** Breaks struct-key equality logic.
- **Mutable key buffers.** `[N]byte` is comparable, but mutating bytes after insert breaks lookup with the old copy.
- **Stringify workaround.** Using `fmt.Sprint(slice)` as a key risks collisions and expensive hashing.
