# Why did Go add generics?

## In short

Generics (since Go 1.18) let you parameterize types and functions: one algorithm compiles for many concrete types without copy-paste and without losing compile-time type checks.

## How to answer in an interview

- I name typical cases: `slices.Contains`, `maps.Keys`, custom containers and algorithms without `interface{}` and reflection.
- I explain constraints: unions like `~int | ~int64` or `cmp.Ordered`, older `constraints.Ordered` in legacy code.
- I contrast with pre-generics Go: duplicated `MaxInt`, `MaxFloat` or runtime type switches.
- I add honestly: generics do not replace interfaces where runtime polymorphism is needed; sprinkling `<T any>` everywhere hurts readability.

## How it works

A type parameter `T` is filled in at instantiation. The compiler monomorphizes: `[]int` and `[]string` get separate specializations. A constraint describes which operations are legal on `T`.

```go
package main

import (
	"cmp"
	"fmt"
)

func Min[T cmp.Ordered](items []T) T {
	if len(items) == 0 {
		panic("empty slice")
	}
	best := items[0]
	for _, v := range items[1:] {
		if v < best {
			best = v
		}
	}
	return best
}

type Pair[A, B any] struct {
	First  A
	Second B
}

func main() {
	fmt.Println(Min([]int{4, 1, 9}))
	fmt.Println(Min([]string{"pear", "apple", "kiwi"}))
	p := Pair[string, int]{First: "qty", Second: 42}
	fmt.Println(p)
}
```

`cmp.Ordered` from stdlib replaces hand-rolled union types for comparable values. `Pair` shows multiple type parameters on one struct.

Generics cut boilerplate in utilities (the `slices` and `maps` packages are built on them). For DI and mocks, interfaces remain convenient.

## Pitfalls

- **Overwide `any`.** Generics lose their point; reflection stays inside.
- **Constraint explosion.** Ten union arms in one constraint are hard to maintain.
- **Unexported methods in constraints.** A type from another package cannot satisfy a constraint with an unexported method.
- **Binary size.** Monomorphization bloats the binary with many instantiations.
- **Interface constraint confusion.** `interface{ ~[]E; Len() int }` for custom slice-like types needs careful API design.
