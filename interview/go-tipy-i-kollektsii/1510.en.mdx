# How do you build a composite map key without tuples?

## In short

Map keys must be comparable types. I model a composite key as a `struct` with comparable fields (numbers, strings, other structs without slices/maps/funcs). That emulates a fixed-shape pair or tuple.

## How to answer in an interview

- `map[struct{...}]V` works when every field supports `==`.
- I use literals like `Key{A: 1, B: "x"}` for insert and lookup.
- Field order is fixed in the type; structs compare field-wise in declaration order.
- I honestly caveat: for slice or map parts I use a canonical string key or a hash, watching collision risk.

## How it works

```go
package main

import "fmt"

type Coord struct {
	X int
	Y int
}

func main() {
	grid := make(map[Coord]string)

	grid[Coord{0, 0}] = "origin"
	grid[Coord{3, 4}] = "point"

	lookup := Coord{3, 4}
	fmt.Println(grid[lookup]) // point
}
```

Struct comparison walks fields recursively. `Coord{3,4}` and `Coord{X:3,Y:4}` are equivalent. The map hashes the comparable key representation.

## Pitfalls

- **Non-comparable fields.** Slices, maps, funcs, or arrays of non-comparable elements disqualify the struct key.
- **Pointers in keys.** Addresses compare, not pointed values; two pointers to equal data are unequal keys.
- **Floating point.** `float64` follows IEEE rules; NaN is never equal to NaN.
- **Key size.** Large structs raise hashing and copy cost as the map grows.
- **String surrogate.** Concatenating fields needs a delimiter or `"ab"+"c"` collides with `"a"+"bc"`.
