# What is the access complexity of a BST?

## In short

In a balanced BST, search, insert, and delete average O(log n) by height. A skewed tree from sorted inserts has height n and O(n) operations.

## How to answer in an interview

- I tie complexity to height, not n alone.
- Balanced variants (AVL, red-black) keep O(log n) worst case.
- Go stdlib has no BST; I use maps or sorted slices with binary search.
- I honestly add that maps or storage B-trees beat hand-rolled BSTs in most production Go.

## How it works

Each comparison discards half the subtree. Loop or recursion walks from root to nil or target.

```go
package main

import "fmt"

type TreeNode struct {
	Val   int
	Left  *TreeNode
	Right *TreeNode
}

func search(root *TreeNode, target int) *TreeNode {
	for root != nil && root.Val != target {
		if target < root.Val {
			root = root.Left
		} else {
			root = root.Right
		}
	}
	return root
}

func main() {
	root := &TreeNode{Val: 10, Left: &TreeNode{Val: 5}, Right: &TreeNode{Val: 15}}
	fmt.Println(search(root, 5) != nil)
}
```

Balanced depth is ~ log n; a skewed chain is ~ n.

## Pitfalls

- **Sorted inserts.** Inserting 1..n without rebalance builds a linked list.
- **Pointer chasing.** Cache misses can beat slice scan at small n.
- **Duplicate keys.** Equal-key policy affects height.
- **Concurrent trees.** Need locks or immutable structures.
- **Wrong structure.** Heaps are not ordered search trees.
