# How does a nil map differ from a nil slice?

## In short

Both zero values are `nil`, but behavior diverges: reads from a nil map are safe (zero value + `false`), writes panic. A nil slice supports reads via `len`/`range` and `append`; indexing `s[i]` with `len=0` panics.

## How to answer in an interview

- `var m map[K]V` and `var s []T` both satisfy `== nil`.
- `m[k]` on a nil map does not panic; `m[k]=v` panics.
- `append(nilSlice, x)` is legal and allocates a backing array.
- I honestly contrast JSON: nil map → `null`, empty slice `[]T{}` → `[]`.

## How it works

```go
package main

import "fmt"

func main() {
	var m map[string]int
	var s []int

	v, ok := m["missing"]
	fmt.Println(v, ok) // 0 false

	s = append(s, 7)
	fmt.Println(s) // [7]

	// m["k"] = 1 // panic: assignment to nil map
}
```

A nil map means no hash table; the runtime creates buckets only after `make`. A nil slice is a header with a nil pointer, yet slice APIs (`len`, `cap`, `append`, `range`) are defined for nil.

## Pitfalls

- **False symmetry.** Newcomers expect identical nil collection behavior.
- **Passing into functions.** Writers must initialize the map or document a non-nil requirement.
- **len(nil map).** `len(m)` is 0 on nil but does not imply write readiness.
- **Comparison with make.** `make(map[K]V)` and `make([]T, 0)` are non-nil; only the slice is immediately `append`-ready.
- **Range over nil.** `for k := range m` is safe (zero iterations); `for _, v := range s` too.
