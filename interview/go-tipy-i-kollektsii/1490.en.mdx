# Is ranging over a slice faster than over a map?

## In short

A full scan of a slice is often faster than map iteration: elements sit contiguously in memory with less pointer chasing and no hash work. Maps win on key lookup, not on linear scans of all pairs.

## How to answer in an interview

- Slice range is CPU-cache friendly; map range walks buckets and overflow chains.
- When the task is "visit every value", a slice or `[]K` plus map can beat `range m`.
- Benchmarks decide: for small n the gap is tiny; on large hot loops it shows up.
- I honestly add that I pick structures by access pattern (index vs key lookup), not a micro-benchmark in isolation.

## How it works

A slice iterator advances an index over the backing array. A map iterator picks a pseudo-random start and walks bucket bitmaps with extra branches.

```go
package main

import "fmt"

func sumSlice(vals []int) int {
	total := 0
	for _, v := range vals {
		total += v
	}
	return total
}

func sumMap(vals map[string]int) int {
	total := 0
	for _, v := range vals {
		total += v
	}
	return total
}

func main() {
	s := []int{1, 2, 3, 4, 5}
	m := map[string]int{"a": 1, "b": 2, "c": 3}

	fmt.Println(sumSlice(s), sumMap(m))
}
```

Both are O(n), but slice constant factors are usually lower.

## Pitfalls

- **Confusing iteration with lookup.** Map O(1) get beats scanning a slice for one key.
- **Pointer-heavy values.** `[]*T` loses cache wins when objects scatter across the heap.
- **Tiny maps.** Iteration overhead may match; benchmark anyway.
- **Key slice maintenance.** Keeping ordered `[]string` plus map costs memory but speeds ordered passes.
- **Compiler optimizations.** Slice bounds-check elimination does not help map walks.
