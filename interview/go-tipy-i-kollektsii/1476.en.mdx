# Which types can be map keys?

## In short

A map key in Go must be comparable: it must support `==` and `!=`. Numbers, strings, bools, pointers, channels, interfaces, and structs work when every struct field is comparable too.

## How to answer in an interview

- I state the rule: the key must be equality-comparable or the compiler rejects the type.
- I list common keys: `string`, integers, structs with comparable fields, pointers to values of the same type.
- I name forbidden keys: slice, map, func, and structs containing float, complex, or slice fields.
- I honestly add that for composite keys it is often simpler to normalize to a string or use `map[string]V` than to build a heavy struct key.

## How it works

The hash table must quickly tell whether two keys are the same. Go checks comparability at compile time. A struct is comparable when every field is; an array is comparable when its element type is.

Example with string and struct keys:

```go
package main

import "fmt"

type Employee struct {
	ID   int
	Dept string
}

func main() {
	byName := map[string]string{
		"Anna": "+7-900-111",
		"Leo":  "+7-900-222",
	}
	fmt.Println(byName["Anna"])

	tenure := map[Employee]int{
		{ID: 10, Dept: "platform"}: 4,
		{ID: 11, Dept: "data"}:     2,
	}
	fmt.Println(tenure[Employee{ID: 10, Dept: "platform"}])
}
```

`Employee` is a valid key because both fields are comparable. Using `[]int` as a key would not compile.

## Pitfalls

- **Float fields in structs.** NaN != NaN breaks equality intuition; floats in keys are almost always a bad idea.
- **Interface keys with nil.** Typed nil inside an interface can surprise you during lookup.
- **Large struct keys.** Copying a big struct on every access costs more than a string id.
- **Pointer identity.** Two pointers to equal structs are different keys even when `*a == *b`.
- **Array vs slice.** `[N]T` may be a key, `[]T` may not; candidates often confuse the two.
