# Testing 1C HTTP services

## In short

Verify contract: status, headers, JSON schema, negative cases. Use Postman or curl against the published service plus autotests on internal parse functions without full cluster where possible.

## How to answer in an interview

- I keep a request collection with base URL and token variables.
- I run happy path and errors: invalid JSON, no rights, missing id.
- I compare response to golden JSON; fix regressions in CI when available.
- Heavy scenarios use a separate test base with anonymized data.
- I am honest: dev-only tests without prod proxy publication miss CORS and timeouts.

## How it works

```bash
curl -s -o /dev/null -w "%{http_code}" -H "Authorization: Bearer $TOKEN" https://host/base/hs/api/v1/health
```

1C unit test for parser:

```bsl
Данные = РазобратьТелоЗаказа(ЭталонныйJSON);
```

## Pitfalls

**Test without auth.** False 200 on closed API.
**Hardcoded host.** Collection fails on other stands.
**No encoding check.** UTF-8 breaks on proxy.
**Ignoring 500 with error body.** Miss business validation regression.
**Load test on prod.** Blocks live sessions.
