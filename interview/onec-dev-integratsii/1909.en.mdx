# Bulk data exchange: batches and chunked load

## In short

Large volumes load in chunks with fixed batch size, one transaction per batch, and progress log. COM and file exchange are legacy; prefer HTTP, queues, and scheduled jobs.

## How to answer in an interview

- I slice export by 500–5000 objects with last id or date marker.
- Each batch in its own transaction: failure does not roll back whole import.
- Idempotency by external GUID prevents duplicates on retry.
- COM only if the counterparty requires it; otherwise integration service.
- I am honest: one giant XML in session memory often causes OutOfMemory.

## How it works

```bsl
Пока ЕстьДанные Цикл
    Пакет = ВыбратьСледующийПакет(РазмерПакета, Маркер);
    НачатьТранзакцию();
    Попытка
        ЗагрузитьПакет(Пакет);
        ЗафиксироватьТранзакцию();
        СохранитьМаркер(Пакет.ПоследнийМаркер);
    Исключение
        ОтменитьТранзакцию();
        ЗаписатьОшибкуПакета(Пакет);
        Прервать;
    КонецПопытки;
КонецЦикла;
```

## Pitfalls

**One transaction per million rows.** Locks and full rollback.
**No resume marker.** Restart from scratch after failure.
**Duplicates without external key.** Re-loaded batch doubles data.
**COM in background without isolation.** Crash kills session.
**No queue monitoring.** Jobs pile up unnoticed.
