# HTTP-методы для сервисов 1С и REST-практика

## Суть

HTTP-сервис 1С публикует URL-шаблоны с привязкой к методам GET, POST, PUT, DELETE, PATCH, HEAD, OPTIONS. Каждый шаблон вызывает функцию общего модуля с параметрами из пути и тела запроса.

## Что ответить на собесе

- GET для чтения без побочных эффектов, POST для создания и сложных действий.
- PUT/PATCH для обновления; DELETE для удаления по идентификатору.
- OPTIONS/HEAD для CORS и проверки доступности без тела.
- В 1С обработчик получает `HTTPСервисЗапрос` и возвращает `HTTPСервисОтвет` с кодом и телом.
- Честно: идемпотентность PUT не гарантируется без логики на стороне обработчика.

## Как это работает

```bsl
Функция ПолучитьЗаказ(Запрос)
    Ответ = Новый HTTPСервисОтвет(200);
    Ответ.Заголовки.Вставить("Content-Type", "application/json; charset=utf-8");
    Ответ.УстановитьТелоИзСтроки(СериализоватьЗаказ(Идентификатор));
    Возврат Ответ;
КонецФункции
```

Маршрут в публикации: `GET /api/orders/{id}` → `ПолучитьЗаказ`.

## Подводные камни

**POST на всё.** Нарушение семантики кэширования и прокси.
**Нет кода ответа 4xx/5xx.** Клиент не отличает ошибку от успеха.
**Тело без Content-Type.** Клиенты ломают парсинг JSON.
**Отсутствие аутентификации на GET.** Утечка данных по ссылке.
**Смешение логики в одном обработчике.** Сложно сопровождать версии API.
