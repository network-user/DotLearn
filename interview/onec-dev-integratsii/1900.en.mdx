# 1C HTTP services: routes, formats, and responses

## In short

An HTTP service is published on the 1C web server with a root URL and path templates. Handlers return status, headers, and body (JSON, XML, text). Input comes from URL segments, query string, and request body.

## How to answer in an interview

- I configure publication in cluster admin and the service root URL.
- Template `/orders/{id}` passes `id` to the handler.
- I serve JSON with `Content-Type: application/json` and UTF-8.
- Errors return 400/404/500 with structured body.
- I am honest: without HTTPS and auth I do not expose the service to the internet.

## How it works

```bsl
Функция СоздатьЗаказ(Запрос)
    Тело = Запрос.ПолучитьТелоКакСтроку();
    Данные = ПрочитатьJSONВСтруктуру(Тело);
    Ссылка = СоздатьЗаказВБазе(Данные);
    Ответ = Новый HTTPСервисОтвет(201);
    Ответ.УстановитьТелоИзСтроки(ОтветJSON(Ссылка));
    Возврат Ответ;
КонецФункции
```

Versioning: `/api/v1/` prefix in templates.

## Pitfalls

**No web server publication.** External 404 with working code.
**One handler for all methods.** Heavy branching on `Запрос.HTTPМетод`.
**Cyrillic without UTF-8.** Garbled client response.
**Long work in handler.** Reverse proxy timeout.
**Stack trace in 500 body.** Information for attackers.
