# Тестирование HTTP-сервисов 1С

## Суть

Проверяют контракт: код ответа, заголовки, JSON-схема, негативные сценарии. Используют Postman или curl против опубликованного сервиса, плюс автотесты на внутренние функции разбора без поднятия всего кластера где возможно.

## Что ответить на собесе

- Я собираю коллекцию запросов с переменными base URL и токеном.
- Гоняю happy path и ошибки: невалидный JSON, нет прав, не найден id.
- Сверяю ответ с эталонным JSON; фиксирую регрессии в CI при наличии.
- Для тяжёлых сценариев отдельная тестовая база с обезличенными данными.
- Честно: тест только на dev без проверки публикации на prod-прокси не ловит CORS и таймауты.

## Как это работает

```bash
curl -s -o /dev/null -w "%{http_code}" -H "Authorization: Bearer $TOKEN" https://host/base/hs/api/v1/health
```

В 1С юнит-тест парсера:

```bsl
Данные = РазобратьТелоЗаказа(ЭталонныйJSON);
ЮТест.ПроверитьРавенство(Данные.Номер, "Z-100");
```

## Подводные камни

**Тест без аутентификации.** Ложноположительный 200 на закрытом API.
**Жёстко зашитый хост.** Коллекция не работает на стендах.
**Нет проверки кодировки.** UTF-8 ломается на прокси.
**Игнор 500 с телом ошибки.** Пропуск регрессии бизнес-валидации.
**Нагрузочный тест на проде.** Блокировка рабочих сеансов.
