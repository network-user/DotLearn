# HTTP methods for 1C services and REST practice

## In short

A 1C HTTP service publishes URL templates bound to GET, POST, PUT, DELETE, PATCH, HEAD, OPTIONS. Each template calls a common module function with path and body parameters.

## How to answer in an interview

- GET for reads without side effects; POST for create and complex actions.
- PUT/PATCH for update; DELETE by identifier.
- OPTIONS/HEAD for CORS and availability checks without body.
- In 1C the handler gets `HTTPСервисЗапрос` and returns `HTTPСервисОтвет` with status and body.
- I am honest: PUT idempotency is not guaranteed without handler logic.

## How it works

```bsl
Функция ПолучитьЗаказ(Запрос)
    Ответ = Новый HTTPСервисОтвет(200);
    Ответ.Заголовки.Вставить("Content-Type", "application/json; charset=utf-8");
    Ответ.УстановитьТелоИзСтроки(СериализоватьЗаказ(Идентификатор));
    Возврат Ответ;
КонецФункции
```

Published route: `GET /api/orders/{id}` → `ПолучитьЗаказ`.

## Pitfalls

**POST for everything.** Breaks caching and proxy semantics.
**No 4xx/5xx status.** Client cannot tell error from success.
**Body without Content-Type.** Clients break JSON parsing.
**No auth on GET.** Data leak via URL.
**All logic in one handler.** Hard to maintain API versions.
