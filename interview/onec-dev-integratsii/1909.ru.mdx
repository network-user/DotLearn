# Массовый обмен данными: пакеты и порционная загрузка

## Суть

Большие объёмы грузят порциями с фиксированным размером пакета, транзакцией на пакет и журналом прогресса. COM и файловый обмен оставляют для legacy; предпочтение HTTP, очередям и регламентным заданиям.

## Что ответить на собесе

- Я режу выгрузку по 500–5000 объектов с маркером последнего id или даты.
- Каждый пакет в отдельной транзакции: сбой не откатывает весь импорт.
- Идемпотентность по внешнему GUID предотвращает дубли при повторе.
- COM только если контрагент требует; иначе сервис интеграции.
- Честно: один гигантский XML в память сеанса часто даёт OutOfMemory.

## Как это работает

```bsl
Пока ЕстьДанные Цикл
    Пакет = ВыбратьСледующийПакет(РазмерПакета, Маркер);
    НачатьТранзакцию();
    Попытка
        ЗагрузитьПакет(Пакет);
        ЗафиксироватьТранзакцию();
        СохранитьМаркер(Пакет.ПоследнийМаркер);
    Исключение
        ОтменитьТранзакцию();
        ЗаписатьОшибкуПакета(Пакет);
        Прервать;
    КонецПопытки;
КонецЦикла;
```

## Подводные камни

**Одна транзакция на миллион строк.** Блокировки и откат всего.
**Нет маркера продолжения.** Повтор с начала после сбоя.
**Дубли без внешнего ключа.** Повторная загрузка пакета.
**COM в фоне без изоляции.** Падение убивает сеанс.
**Нет мониторинга очереди.** Задания копятся незаметно.
