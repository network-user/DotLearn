# select_related vs prefetch_related: one JOIN or a second query

## In short

Both Django ORM methods fight the N+1 problem, but through different mechanisms. `select_related` pulls in related objects with a single SQL query via a JOIN and suits `ForeignKey` and `OneToOne`. `prefetch_related` issues a separate query for the related rows and stitches them together in Python, so it covers `ManyToMany` and reverse "one-to-many" relations.

## How to answer in an interview

"The difference is who joins the data. `select_related` offloads the work to the database: one JOIN, one query, suitable for relations where exactly one object sits on one side - that is, `ForeignKey` and `OneToOneField`. `prefetch_related` runs two queries and joins the results on the Python side by a key field; you reach for it with `ManyToManyField` and reverse `FK`, where a JOIN would inflate the result set with duplicates. In practice I first catch the N+1, then pick the method by the relation type, and for nested relations I combine both and use `Prefetch` to fine-tune the nested query."

Take authors and books. With a `ForeignKey`, a book references a single author - that's the `select_related` case.

```python
class Author(models.Model):
    name = models.CharField(max_length=100)

class Book(models.Model):
    title = models.CharField(max_length=100)
    author = models.ForeignKey(Author, on_delete=models.CASCADE)

books = Book.objects.select_related("author").all()
for book in books:
    print(book.title, book.author.name)
```

Here the ORM builds a single SQL with a `JOIN` to `Author`, and accessing `book.author` no longer hits the database.

For a "many-to-many" relation a single JOIN won't do - use `prefetch_related`.

```python
class Author(models.Model):
    name = models.CharField(max_length=100)

class Book(models.Model):
    title = models.CharField(max_length=100)
    authors = models.ManyToManyField(Author)

books = Book.objects.prefetch_related("authors").all()
for book in books:
    print(book.title, [a.name for a in book.authors.all()])
```

Django runs a query for the books, then a separate query for all the needed authors, and assembles the relation in memory.

## Pitfalls

- `select_related` on a "many" relation via JOIN multiplies rows and bloats the result set, so it doesn't work for `ManyToMany` - there you use only `prefetch_related`.
- `prefetch_related` is cached at the moment of the call. If after prefetching you apply a new filter to the relation (`book.authors.filter(...)`), Django issues an extra query and the whole point is lost - filter through a `Prefetch` object.
- Both methods help only when you actually access the related objects. If the relation isn't used in the code, prefetching just adds an extra query.
- Excessive `select_related` along a long chain of relations produces a heavy JOIN with many columns; sometimes `only()`/`defer()` or splitting into `prefetch_related` is better.
