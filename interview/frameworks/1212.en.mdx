# When to use APIView vs ViewSet in DRF?

## In short

`APIView` is a low-level class where you write the HTTP methods (`get`, `post`, ...) by hand and get full control. `ViewSet` is a high-level wrapper that maps CRUD actions to HTTP methods through a router and removes the boilerplate for typical resources.

## How to answer in an interview

I choose based on whether the endpoint fits standard CRUD. If the API is resource-oriented (list, create, read, update, delete), I use `ModelViewSet` plus a router: a couple of lines, and routes with all methods are generated automatically. If I need non-standard logic, custom actions, unusual methods, or the endpoint isn't really about a "resource," I use `APIView` and write `get()`/`post()` explicitly. Essentially, a ViewSet is a grouping of several views with automatic routing built on the same mechanism as APIView.

## How it works

`APIView` extends the regular Django `View`, adding authentication, permissions, parsing, and serialization. You write the method for each HTTP verb yourself:

```python
from rest_framework.views import APIView
from rest_framework.response import Response
from rest_framework import status

class MyAPIView(APIView):
    def get(self, request, format=None):
        return Response({"message": "Hello, world!"})

    def post(self, request, format=None):
        return Response(request.data, status=status.HTTP_201_CREATED)
```

`ViewSet` (and especially `ModelViewSet`) maps the `list`/`create`/`retrieve`/`update`/`destroy` actions to HTTP methods. With `ModelViewSet` it's enough to specify a queryset and a serializer:

```python
from rest_framework import viewsets
from .models import MyModel
from .serializers import MyModelSerializer

class MyModelViewSet(viewsets.ModelViewSet):
    queryset = MyModel.objects.all()
    serializer_class = MyModelSerializer
```

Routes for a ViewSet are registered by a router; it expands the actions into real URLs and methods:

```python
from rest_framework.routers import DefaultRouter

router = DefaultRouter()
router.register(r'items', MyModelViewSet)
urlpatterns = router.urls
```

## Pitfalls

- A ViewSet on its own can't be plugged into `urlpatterns` as a regular view: it needs a router or a manual `as_view({'get': 'list', ...})`. This is a common beginner mistake: "I registered the class but get a 404."
- A custom action in a ViewSet is added with the `@action(detail=True, methods=['post'])` decorator, otherwise the router won't know about it.
- `APIView` is more flexible, but you pay for that flexibility with code volume: for plain CRUD it's extra boilerplate. Conversely, pulling in `ModelViewSet` for a single non-standard endpoint is overkill.
- Under the hood a ViewSet uses the same pipeline as APIView (via the generics), so authentication, permissions, and pagination are configured the same way; there's less "magic" than it seems.
