# Signals in Django: how to react to events without tight coupling?

## In short

Signals are a built-in publisher-subscriber mechanism: some parts of the application send events, others subscribe to them and react. This lets you attach side logic (mailings, auditing, cache invalidation) to model events without baking it into the core code.

## How to answer in an interview

Signals provide loose coupling: the sender of an event doesn't know who will react to it or how. For example, on the `post_save` of the `Article` model I attach a receiver that notifies subscribers, and the article-saving code stays clean. The upsides are separation of concerns and testability. But don't overuse them: signals make the control flow implicit, so for logic that should be visible "right there in place" I prefer an explicit call in the `save()` method or in a service layer.

At its core it's the publisher-subscriber pattern. The **sender** emits a signal (a model on save, delete, or migration). The **receiver** is a function subscribed to the signal that performs actions in response.

The subscription is set up with the `@receiver` decorator: it ties the function to a specific signal and, optionally, to a specific sender (`sender`).

```python
from django.db.models.signals import post_save
from django.dispatch import receiver
from .models import Article

@receiver(post_save, sender=Article)
def notify_subscribers(sender, instance, created, **kwargs):
    if created:
        print(f"New article created: {instance.title}")
        # the notification mailing is triggered here
```

The `created` argument distinguishes creation from an update, and `instance` is the saved object itself. For the receiver to get registered, its module must be imported at startup - usually this is done in the `ready()` method of the `AppConfig` class.

Typical uses: auditing and logging changes, cache invalidation when data is edited, user notifications, additional post-processing of records.

## Pitfalls

- **Implicit flow.** A signal fires "off to the side" - which complicates debugging. If the logic clearly belongs to the model, explicit code in `save()` is better.
- **Receiver registration.** If the module with `@receiver` isn't imported anywhere, the signal silently won't fire. Wire it up in `AppConfig.ready()`.
- **`post_save` and transactions.** The signal fires before the transaction commits; if you send emails or call an external service inside it, a rollback leaves you with a phantom action. Use `transaction.on_commit`.
- **Recursion.** Calling `instance.save()` inside a `post_save` handler will raise `post_save` again - it's easy to fall into an infinite loop.
