# When to pull dictionaries from the ORM via values(), and when tuples via values_list()?

## In short

Both methods make the ORM fetch only the columns you need from the table instead of full model objects. The difference is in the shape of the result: `values()` gives dictionaries `{field: value}`, `values_list()` gives tuples of values.

## How to answer in an interview

`values()` and `values_list()` turn a `QuerySet` of model objects into a lightweight dataset. I use `values()` when it's more convenient to access fields by name (for example, to hand the data straight to JSON or a template), and `values_list()` when the names don't matter and I only need the values themselves. To select a single column I add `flat=True` — I get a flat list instead of a list of single-element tuples. The main motivation is to avoid dragging extra fields out of the database and instantiating heavy model objects where they aren't needed.

## How it works

`values()` returns a `QuerySet` in which each element is a dictionary. The keys match the names of the requested fields:

```python
from myapp.models import Book

books = Book.objects.values('title', 'author')
for book in books:
    print(book)  # {'title': 'Clean Code', 'author': 'Martin'}
```

`values_list()` returns the same data as tuples — the order of elements matches the order of the arguments:

```python
books = Book.objects.values_list('title', 'author')
for book in books:
    print(book)  # ('Clean Code', 'Martin')
```

When you need exactly one column, `flat=True` removes the tuple wrapper and returns a flat list of values:

```python
titles = Book.objects.values_list('title', flat=True)
list(titles)  # ['Clean Code', 'Refactoring', ...]
```

A typical scenario is collecting a list of unique ids without pulling whole rows:

```python
from myapp.models import Order

customer_ids = Order.objects.values_list('customer_id', flat=True).distinct()
```

## Pitfalls

- `flat=True` works with one field only. Pass two and you'll get a `TypeError`.
- The result is still a lazy `QuerySet`, not a list. The database query goes out on the first iteration; don't confuse "lightweight" with "already loaded."
- Dictionaries and tuples aren't model objects: they have no model methods, no `@property` properties, and no access to related objects via attributes. If the code later calls `obj.get_absolute_url()`, `values()` will break it.
- To traverse relations, write the path with a double underscore: `values('author__name')`. Dot notation won't work here.
