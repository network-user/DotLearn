# Как в DRF разграничить доступ к эндпоинтам через классы permissions?

## Суть

Permission в DRF — это класс с методом-проверкой, который для каждого запроса возвращает `True` (пропустить) или `False` (отказать). Набор таких классов вешается на представление и решает, кому разрешено выполнять запрашиваемое действие.

## Что ответить на собесе

DRF выносит проверку доступа в отдельные классы-permissions, которые перечисляются в атрибуте `permission_classes` представления. На каждый запрос фреймворк прогоняет их по очереди; если хоть один вернёт `False`, клиент получает 403 (или 401, если он вообще не аутентифицирован). Из коробки есть `AllowAny`, `IsAuthenticated`, `IsAdminUser`, `IsAuthenticatedOrReadOnly`. Для нестандартной логики я наследуюсь от `BasePermission` и переопределяю `has_permission` (доступ к ресурсу в целом) или `has_object_permission` (доступ к конкретному объекту). Так правила доступа лежат отдельно от бизнес-логики вьюхи.

```python
from rest_framework.permissions import IsAuthenticated
from rest_framework.views import APIView
from rest_framework.response import Response

class MyView(APIView):
    permission_classes = [IsAuthenticated]

    def get(self, request):
        return Response({"message": "Доступ только для аутентифицированных"})
```

Готовые классы покрывают типовые случаи:

- `AllowAny` — открыто всем.
- `IsAuthenticated` — только аутентифицированным.
- `IsAdminUser` — только пользователям с `is_staff`.
- `IsAuthenticatedOrReadOnly` — читать может любой, изменять — только вошедшие.

Своё правило — это наследник `BasePermission`. Метод `has_permission` решает доступ к эндпоинту целиком, `has_object_permission` — к отдельному объекту (его вызывают только после `get_object`):

```python
from rest_framework.permissions import BasePermission

class IsEmailDomainAllowed(BasePermission):
    def has_permission(self, request, view):
        user = request.user
        return user.is_authenticated and user.email.endswith('@example.com')
```

Подключается так же, через список:

```python
class MyCustomView(APIView):
    permission_classes = [IsEmailDomainAllowed]

    def get(self, request):
        return Response({"message": "Привет, сотрудник example.com"})
```

## Подводные камни

- Список `permission_classes` объединяется по И: все классы должны вернуть `True`. Логику «или» нужно собирать вручную (в новых версиях DRF — операторами `|`, `&`, `~` над классами).
- `has_object_permission` срабатывает только при доступе к конкретному объекту через `get_object()`. На уровне списка (`list`/`create`) объектная проверка не вызывается — фильтруйте `QuerySet` отдельно.
- Permissions проверяют доступ, но не подменяют аутентификацию: неаутентифицированный пользователь у класса `IsAuthenticated` получит 401, а не 403.
- Пустой `permission_classes = []` снимает все ограничения, включая глобальные из `DEFAULT_PERMISSION_CLASSES`. Это лёгкий способ случайно открыть эндпоинт всем.
