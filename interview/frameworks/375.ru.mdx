# Какой контракт описывает WSGI и как он разделяет веб-сервер и Python-приложение

## Суть

WSGI задаёт единый протокол вызова между HTTP-сервером и Python-приложением: сервер собирает запрос и зовёт приложение, приложение отдаёт ответ. Пока обе стороны держат этот протокол, их можно свободно комбинировать - один и тот же код фреймворка крутится под разными серверами без правок.

## Что ответить на собесе

WSGI (Web Server Gateway Interface) - синхронный интерфейс между веб-сервером и Python-приложением, описанный в PEP 3333.

- Приложение здесь - любой вызываемый объект с двумя параметрами: словарём `environ` (всё про входящий запрос) и колбэком `start_response` (через него задают код статуса и заголовки).
- В ответ приложение отдаёт итерируемый объект с байтами тела.
- Именно из-за этого единого протокола Django или Flask запускаются под Gunicorn, uWSGI или Waitress без изменений в коде.
- Асинхронные сценарии WSGI не покрывает - для них есть ASGI.

## Как это работает

Стандарт делит мир на две роли и проводит между ними жёсткую границу:

- **WSGI-сервер** держит TCP-соединения, разбирает HTTP, наполняет `environ` и вызывает приложение. Сюда относятся Gunicorn, uWSGI, Waitress.
- **WSGI-приложение** читает `environ`, собирает ответ и возвращает байты. Это может быть одна функция или целый фреймворк.

Простейшее приложение - функция с нужной сигнатурой:

```python
def simple_app(environ, start_response):
    status = "200 OK"
    headers = [("Content-Type", "text/plain; charset=utf-8")]
    start_response(status, headers)
    return [b"Hello, World!"]
```

- `environ` - словарь с данными запроса: метод, путь, заголовки, переменные окружения.
- `start_response` - колбэк, который приложение обязано вызвать прежде, чем вернёт тело; ему передают строку статуса и список заголовков.

Чтобы поднять приложение, его передают серверу. У Gunicorn это выглядит так:

```bash
gunicorn myapp:simple_app
```

`myapp` тут - имя модуля, а `simple_app` - вызываемый объект-приложение внутри него.

## Подводные камни

- WSGI синхронен в основе: внутри одного вызова нельзя сделать `await`. Под WebSocket, long-polling и асинхронные вьюхи берут ASGI (Uvicorn, Daphne), а не WSGI.
- Тело ответа - это байты, не строки. Если вернуть `["текст"]` вместо `[b"текст"]`, сервер упадёт с ошибкой.
- `start_response` вызывают ровно один раз и до того, как сервер начнёт читать тело; повторный или запоздалый вызов ломает контракт.
- WSGI - это интерфейс, а не готовый сервер. Встроенный `runserver` для боя не годится: под нагрузку ставят Gunicorn или uWSGI.
