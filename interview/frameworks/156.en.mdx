# The path of an HTTP request inside Django: from URL to response

## In short

When a request arrives, Django matches its URL against the rules in the URLconf, finds the right view, and calls it. The view runs its logic, pulls data through models if needed, and returns an `HttpResponse` - HTML, JSON, or another format that goes back to the client.

## How to answer in an interview

The request lifecycle in Django is as follows: the request first passes through the inbound middleware stack, then the URL resolver picks a view from the URLconf by matching the path against `urlpatterns`. The view is a function or class where the application logic lives: if needed it reaches out to models, which run database queries through the ORM. The view builds an `HttpResponse`, usually by rendering a template with data. On the way back, the response passes through middleware again and is sent to the client. The main point is to understand that the URLconf routes, the view holds the logic, the model is responsible for data, and the template for markup.

**Routing.** Django checks the request path against the list of patterns and picks the first match.

```python
from django.urls import path
from . import views

urlpatterns = [
    path("home/", views.home, name="home"),
    path("articles/", views.article_list, name="article-list"),
]
```

**A simple view** can return a response directly.

```python
from django.http import HttpResponse


def home(request):
    return HttpResponse("Welcome to the home page!")
```

**Data access** goes through models and the ORM, and the result is handed to a template.

```python
from django.shortcuts import render
from .models import Article


def article_list(request):
    articles = Article.objects.all()
    return render(request, "articles/list.html", {"articles": articles})
```

In summary: request -> middleware -> URLconf -> view -> (models via ORM) -> template -> `HttpResponse` -> middleware -> client.

## Pitfalls

- **Order in `urlpatterns` matters.** Django takes the first match top to bottom; a too-general pattern placed higher will "shadow" a more specific one below.
- **Middleware is part of the cycle.** The request doesn't reach the view immediately: the middleware stack runs first, and any of them can return a response earlier.
- **QuerySet laziness.** `Article.objects.all()` doesn't hit the database right away - the query runs when iterated in the template. This affects both when errors surface and the N+1 problem.
- **A view must return an `HttpResponse`.** A forgotten `return`, or returning "bare" data instead of an `HttpResponse`/`render`, will cause an error.
