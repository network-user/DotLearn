# DRF serializers: a two-way bridge between Python objects and JSON

## In short

A serializer in Django REST Framework translates complex objects (models, QuerySets) into Python primitives ready to be rendered as JSON, and performs the reverse operation - parsing incoming JSON into Python data with validation. In essence it is a two-way contract between the internal representation and the external API.

## How to answer in an interview

A serializer solves two tasks. On the way out (serialization) it turns a Django object into a dictionary of primitives, which the DRF renderer hands to the client as JSON. On the way in (deserialization) it accepts raw client data, runs it through `is_valid()`, and on success places the clean data into `validated_data`, from where `create()` or `update()` pick it up. The key point: invalid data never reaches the model - validation is built into the serializer itself. Reading an object goes through `serializer.data`, writing goes through `serializer(data=...)` and `is_valid()`.

## How it works

You declare the fields and their types, and DRF takes the conversion and the checks on itself:

```python
from rest_framework import serializers
from myapp.models import Book

class BookSerializer(serializers.Serializer):
    title = serializers.CharField(max_length=100)
    author = serializers.CharField(max_length=100)
    published_date = serializers.DateField()
```

Serialization - object into primitives (accessed via `.data`):

```python
book_instance = Book.objects.get(id=1)
serializer = BookSerializer(book_instance)
serializer.data
# {'title': 'Django for Beginners', 'author': 'William S. Vincent', 'published_date': '2020-01-01'}
```

Deserialization - incoming dictionary into validated data:

```python
data = {'title': 'New Book', 'author': 'John Doe', 'published_date': '2023-10-10'}
serializer = BookSerializer(data=data)
if serializer.is_valid():
    clean = serializer.validated_data
    # use clean for create()/update()
```

## Pitfalls

- `serializer.data` on a serializer used for writing is available only after a successful `is_valid()` - before that, accessing `.data` on an instance created with `data=...` will throw an error (use `validated_data`, or validate first).
- `is_valid()` itself does not raise an exception by default - it returns `False` and puts the errors into `serializer.errors`. To get a 400 automatically, pass `is_valid(raise_exception=True)`.
- The base `Serializer` cannot save to the database on its own: the `create()` and `update()` methods must be written by hand (unlike `ModelSerializer`).
- For a collection of objects you need `many=True`, otherwise the serializer will try to process the QuerySet as a single object.
- Field-level validation (`validate_<field>`) and object-level validation (`validate`) are different hooks; do not confuse their scope.
