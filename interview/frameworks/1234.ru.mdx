# Почему асинхронные view в Django нельзя считать «бесплатным» ускорением?

## Суть

Async view выигрывает только на операциях ввода-вывода и только если вся цепочка вокруг него тоже не блокирует поток. Синхронный код (включая большую часть ORM), несовместимые middleware и блокирующие библиотеки сводят выигрыш на нет; при неаккуратном вызове они даже ухудшают ситуацию.

## Что ответить на собесе

Django умеет `async def` view, но фреймворк исторически синхронный, и многие его части до сих пор такими и остаются.

- Если из async-вьюхи напрямую дёрнуть блокирующий синхронный код (например, синхронный ORM-вызов), он застопорит event loop и убьёт смысл асинхронности; синхронные участки оборачивают в `sync_to_async` из `asgiref`.
- Не каждый middleware и не каждая сторонняя библиотека асинхронны: Django умеет «адаптировать» их, но через переключение контекста, что добавляет накладные расходы.
- Async-view оправдан на I/O-bound нагрузке (внешние API, файлы), а для CPU-bound задач он бесполезен.

Async view опирается на `async`/`await` и неблокирующие операции ввода-вывода:

```python
from django.http import JsonResponse
from django.views import View
import asyncio

class MyAsyncView(View):
    async def get(self, request):
        await asyncio.sleep(1)
        return JsonResponse({'message': 'Hello, async world!'})
```

Синхронный код из async вызывают через обёртку, чтобы не блокировать цикл событий:

```python
from django.http import JsonResponse
from django.views import View
from asgiref.sync import sync_to_async

class SafeAsyncView(View):
    async def get(self, request):
        result = await sync_to_async(self.some_sync_method)()
        return JsonResponse({'result': result})

    def some_sync_method(self):
        return "Hello, World!"
```

Поток данных: ASGI-сервер держит соединение, пока `await` ждёт I/O, и в это время обслуживает другие запросы.

```text
запрос ─▶ async view ─▶ await (ждём I/O, поток свободен) ─▶ ответ
                          ▲
            синхронный ORM/библиотека → sync_to_async, иначе блокировка
```

## Подводные камни

- Прямой синхронный ORM-вызов из async-view блокирует event loop. Нужен либо `sync_to_async`, либо асинхронный ORM-интерфейс (`aget`, `acreate`, `async for`).
- Async-view ускоряет только I/O-bound задачи. CPU-bound вычисления всё равно держат поток: асинхронность тут не поможет, нужны процессы/воркеры.
- Middleware и сторонние библиотеки бывают только синхронными; Django их адаптирует, но через переключение контекстов, что съедает часть выигрыша.
- Async-view требует ASGI-сервера (uvicorn/daphne). Под классическим WSGI преимущества асинхронности недоступны.
- `sync_to_async` по умолчанию исполняет код в отдельном потоке (thread-sensitive), и неаккуратное смешивание sync/async легко приводит к скрытым блокировкам.
