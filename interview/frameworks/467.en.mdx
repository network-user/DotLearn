# What role does manage.py play in a Django project

## In short

`manage.py` is an executable Python script that Django places at the root of a project when it's created. It acts as a single command-line entry point for the project: you use it to start the development server, apply migrations, scaffold apps, and create a superuser. Its key difference from plain `django-admin` is that it already knows where your project's settings live.

## How to answer in an interview

Put it briefly: `manage.py` is a wrapper around `django-admin` tied to a specific project. It sets the `DJANGO_SETTINGS_MODULE` environment variable pointing at your settings module and then hands off control. That's why commands like `python manage.py <command>` run in the context of your particular project - with the right database, apps, and config. A handy roster of commands to mention: `runserver`, `migrate`, `makemigrations`, `startapp`, `createsuperuser`, `shell`.

## How it works

When you type a command, `manage.py` first sets the path to the settings module in the environment, then calls `execute_from_command_line` from `django.core.management`. Django then looks up the matching command class (its own built-in one or one declared in an app) and runs it. So `manage.py` itself does almost nothing - it just prepares the context and delegates the work to the command dispatcher.

```python
import os
import sys


def main():
    os.environ.setdefault("DJANGO_SETTINGS_MODULE", "myproject.settings")
    from django.core.management import execute_from_command_line
    execute_from_command_line(sys.argv)


if __name__ == "__main__":
    main()
```

Typical invocations:

```bash
python manage.py runserver
python manage.py makemigrations
python manage.py migrate
python manage.py startapp myapp
python manage.py createsuperuser
```

## Pitfalls

- `manage.py` and `django-admin` are not the same thing. `django-admin` is generic and knows nothing about your settings; to make it work with a project you have to set `DJANGO_SETTINGS_MODULE` manually. `manage.py` does this for you.
- Commands must be run from the directory where `manage.py` lives, otherwise Python won't find the project module and its settings.
- The file isn't magic: it's an ordinary script with the settings path hardcoded as a string. If you rename the project package or move `settings.py`, you'll have to fix the `DJANGO_SETTINGS_MODULE` line.
- A custom command is added not by editing `manage.py` but by adding a file under `<app>/management/commands/`. There's no need to touch `manage.py` itself for that.
