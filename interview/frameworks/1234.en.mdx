# Why can't async views in Django be considered a "free" speedup?

## In short

An async view only wins on I/O operations, and only if the entire chain around it doesn't block the thread either. Synchronous code (including most of the ORM), incompatible middleware, and blocking libraries nullify the gain — and with a careless call, they even make things worse.

## How to answer in an interview

Django can do `async def` views, but the framework is historically synchronous, and many of its parts still are. If you directly call blocking synchronous code from an async view — for example, a synchronous ORM call — it will stall the event loop and kill the point of asynchrony. That's why synchronous sections are wrapped in `sync_to_async` from `asgiref`. On top of that, not every middleware and not every third-party library is async: Django can "adapt" them, but through a context switch, which adds overhead. An async view is justified for I/O-bound load (external APIs, files), and for CPU-bound tasks it's useless.

An async view relies on `async`/`await` and non-blocking I/O operations:

```python
from django.http import JsonResponse
from django.views import View
import asyncio

class MyAsyncView(View):
    async def get(self, request):
        await asyncio.sleep(1)
        return JsonResponse({'message': 'Hello, async world!'})
```

Synchronous code is called from async through a wrapper so as not to block the event loop:

```python
from django.http import JsonResponse
from django.views import View
from asgiref.sync import sync_to_async

class SafeAsyncView(View):
    async def get(self, request):
        result = await sync_to_async(self.some_sync_method)()
        return JsonResponse({'result': result})

    def some_sync_method(self):
        return "Hello, World!"
```

The data flow: the ASGI server holds the connection while `await` waits on I/O, and during that time it serves other requests.

```text
request ─▶ async view ─▶ await (waiting on I/O, thread free) ─▶ response
                          ▲
            synchronous ORM/library → sync_to_async, otherwise a block
```

## Pitfalls

- A direct synchronous ORM call from an async view blocks the event loop. You need either `sync_to_async` or the async ORM interface (`aget`, `acreate`, `async for`).
- An async view speeds up only I/O-bound tasks. CPU-bound computation still holds the thread — asynchrony won't help here; you need processes/workers.
- Middleware and third-party libraries are sometimes synchronous-only; Django adapts them, but through context switches, which eats part of the gain.
- An async view requires an ASGI server (uvicorn/daphne). Under classic WSGI the advantages of asynchrony aren't available.
- `sync_to_async` by default runs the code in a separate thread (thread-sensitive), and careless mixing of sync/async easily leads to hidden blocks.
