# Какими инструментами Django ускоряют выдачу за счёт кэша?

## Суть

Django из коробки даёт многоуровневую систему кэша поверх Memcached или Redis: можно положить в хранилище целую страницу, отдельный кусок шаблона или произвольные данные через низкоуровневый API. Идея одна — не пересчитывать дорогой результат повторно.

## Что ответить на собесе

Кэш в Django работает на трёх высотах.

- Декоратор `@cache_page` оборачивает представление и отдаёт готовый HTML, пока не истёк таймаут.
- Шаблонный тег `{% cache %}` кэширует только фрагмент — например, общее меню.
- Когда нужна гибкость, беру низкоуровневый API `django.core.cache.cache` с методами `set`/`get` и сам решаю, что и на сколько класть.
- Бэкенд (Redis, Memcached, локальная память) настраивается одной секцией `CACHES` в `settings.py`, поэтому код кэширования от выбора хранилища не зависит.

## Как это работает

Кэш — это временное хранилище готовых результатов: при повторном обращении мы достаём их оттуда, а не выполняем заново запросы к БД и тяжёлый рендеринг. Это снимает нагрузку с базы и сокращает время отклика.

Бэкенд задаётся секцией `CACHES`. Для Redis:

```python
CACHES = {
    'default': {
        'BACKEND': 'django_redis.cache.RedisCache',
        'LOCATION': 'redis://127.0.0.1:6379/1',
        'OPTIONS': {
            'CLIENT_CLASS': 'django_redis.client.DefaultClient',
        },
    }
}
```

Кэш целого представления — декоратор `@cache_page`, аргумент задаёт время жизни в секундах:

```python
from django.views.decorators.cache import cache_page

@cache_page(60 * 15)  # держим ответ 15 минут
def my_view(request):
    return HttpResponse('Hello, World!')
```

Кэш фрагмента шаблона — теги `load cache` и `cache`, у блока есть таймаут и имя ключа:

```html
{% load cache %} {% cache 500 sidebar %}
<div>
  <!-- тяжёлый для рендеринга контент -->
</div>
{% endcache %}
```

Низкоуровневый API — кладём и читаем произвольные значения:

```python
from django.core.cache import cache

cache.set('my_key', 'my_value', timeout=60 * 15)
value = cache.get('my_key')  # 'my_value', либо None после истечения
```

## Подводные камни

- `@cache_page` кэширует страницу одинаково для всех, поэтому к персональным данным (профиль, корзина) его применять нельзя — иначе один пользователь увидит контент другого. Кэш всей страницы варьируется по URL и заголовкам из `Vary`.
- `cache.get` для отсутствующего или протухшего ключа возвращает `None`, а не бросает исключение. Если валидное значение само может быть `None`, передавайте sentinel-объект по умолчанию: `cache.get(key, MISSING)`.
- Таймаут в секундах. `timeout=None` означает «вечно», `timeout=0` — не кэшировать вовсе; это частая путаница.
- Главная боль кэша — инвалидация. После изменения данных устаревший кэш надо явно сбрасывать (`cache.delete`) либо завязываться на версии ключей, иначе пользователь увидит старое.
