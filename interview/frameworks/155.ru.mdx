# Как Django делит приложение на Model, View и Template

## Суть

MVT (Model-View-Template) - это вариант разделения ответственности, на котором стоит Django: Model хранит данные и логику работы с ними, View обрабатывает запрос и решает, что показать, а Template отвечает за саму вёрстку ответа. Такое разделение упрощает поддержку и позволяет менять отображение независимо от логики.

## Что ответить на собесе

MVT - это Django-вариант MVC, где роль контроллера берёт на себя сам фреймворк (маршрутизатор URL), а слой «View» в терминах Django отвечает за прикладную логику.

- Model описывает структуру данных и инкапсулирует доступ к БД через ORM.
- View принимает `request`, достаёт данные из моделей и выбирает, какой шаблон отрендерить.
- Template - это HTML с языком шаблонов Django, который превращает данные в готовую разметку.
- Ключевая идея - отделить данные, логику и представление, чтобы каждый слой менялся независимо.

## Как это работает

**Model** наследуется от `models.Model` и описывает таблицу с её полями. Через ORM модель скрывает SQL за CRUD-операциями.

```python
from django.db import models


class Book(models.Model):
    title = models.CharField(max_length=100)
    author = models.CharField(max_length=50)
    published_date = models.DateField()
```

**View** принимает `request`, забирает данные у модели и передаёт их в шаблон.

```python
from django.shortcuts import render
from .models import Book


def book_list(request):
    books = Book.objects.all()
    return render(request, "books/book_list.html", {"books": books})
```

**Template** - это HTML с языком шаблонов Django, который подставляет данные.

```html
<h1>Книги</h1>
<ul>
  {% for book in books %}
  <li>{{ book.title }} - {{ book.author }}</li>
  {% endfor %}
</ul>
```

Связка такая: маршрутизатор по URL выбирает view, view берёт данные из model и отдаёт их в template, а собранный HTML уходит клиенту как HTTP-ответ.

## Подводные камни

- **Терминологическая путаница MVC vs MVT.** В Django «View» - это не «представление» из MVC (там это шаблон), а ближе к контроллеру. Это частый источник недопонимания на собесе.
- **Бизнес-логика в шаблонах.** Шаблоны намеренно ограничены; перенос вычислений в them - антипаттерн, логику держим в моделях и view.
- **«Толстые» view.** Логику работы с данными лучше выносить в методы модели или менеджеры, иначе view разрастаются и плохо тестируются.
- **Контроллер «спрятан».** Роль контроллера выполняет URLconf и сам фреймворк, поэтому отдельного слоя «Controller» в проекте искать не нужно.
