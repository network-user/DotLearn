# Как организовать слои бизнес-логики в Django-проекте?

## Суть

Бизнес-логика - это код правил вашего домена: расчёт скидок, оформление заказа, начисление штрафов. В Django её принято распределять по слоям, а не сваливать во вьюхи. Базовый ориентир - "толстые модели, тонкие представления": доменные правила и работа с данными живут в моделях, менеджерах и сервисах, а вьюха только принимает HTTP-запрос и отдаёт ответ. Это упрощает тестирование, повторное использование и сопровождение.

## Что ответить на собесе

Сформулируйте принцип и перечислите слои, куда уходит логика:

1. **Модели** - методы, которые считают и валидируют поверх собственных данных объекта ("толстые модели").
2. **Менеджеры и QuerySet** - инкапсуляция типовых запросов к базе, чтобы не дублировать фильтры по всему коду.
3. **Сервисы и утилиты** - операции, охватывающие несколько моделей или внешние системы; не привязаны к одной модели.
4. **Представления (Views)** - остаются тонкими: разобрать запрос, вызвать сервис/модель, вернуть ответ.
5. **Сигналы** - реакция на события (например, действие после `save`), но без злоупотребления: неявные связи усложняют отладку.

Главный аргумент - разделение ответственности: каждый слой проще тестировать и переиспользовать.

## Как это работает

Метод модели держит правило рядом с данными, над которыми оно работает:

```python
from django.db import models


class Loan(models.Model):
    book = models.ForeignKey("Book", on_delete=models.CASCADE)
    due_date = models.DateField()
    returned_at = models.DateField(null=True, blank=True)

    def is_overdue(self, today):
        return self.returned_at is None and today > self.due_date
```

Операцию, затрагивающую несколько моделей, выносят в сервис - обычную функцию или класс без привязки к одному объекту:

```python
def issue_book(member, book, today, due_in_days=14):
    if book.available_copies <= 0:
        raise NoCopiesAvailable(book.id)
    book.available_copies -= 1
    book.save(update_fields=["available_copies"])
    return Loan.objects.create(
        book=book,
        due_date=today + timedelta(days=due_in_days),
    )
```

## Подводные камни

- **Логика в шаблонах и вьюхах.** Расчёты в шаблоне или раздувшаяся вьюха - первый признак, что правило живёт не на своём слое; такой код тяжело тестировать.
- **Толстые модели превращаются в "божественный объект".** Если модель тянет внешние сервисы, платежи и отправку писем, правило стоит вынести в сервис.
- **Сигналы как скрытая логика.** Они удобны, но создают неявные связи: операцию легко "потерять", потому что она срабатывает где-то в стороне от явного вызова. Для ключевых процессов предпочитайте явный сервис.
- **Дублирование запросов.** Одинаковые `filter(...)` по всему проекту - сигнал завести метод менеджера или кастомный QuerySet.
