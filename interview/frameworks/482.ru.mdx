# Что Django даёт для входа в систему и проверки прав из коробки?

## Суть

Подсистема `django.contrib.auth` приносит готовый набор: модель пользователя, группы и разрешения, формы и вьюхи для регистрации, логина и логаута, плюс хранение залогиненного состояния в сессии. Поэтому контроль доступа собирается за считанные минуты и без самописной криптографии.

## Что ответить на собесе

Django закрывает обе задачи безопасности. Аутентификация (кто ты) держится на модели `User` и бэкендах, проверяющих логин и пароль; пароли хранятся в виде хешей, не открытым текстом. Авторизация (что тебе можно) строится на группах и разрешениях. Готовые `LoginView` и `LogoutView` обрабатывают вход и выход, а декоратор `@login_required` или миксин `LoginRequiredMixin` закрывают вьюхи от анонимов. Состояние пользователя между запросами тащит механизм сессий, поэтому повторно логиниться на каждой странице не нужно. Если стандартных полей мало, заводят кастомную модель через `AUTH_USER_MODEL`.

## Как это работает

Ключевые кирпичики подсистемы:

- **Модель `User`** хранит `username`, `email`, хеш пароля, флаги `is_active`, `is_staff`, `is_superuser`. Расширяется либо профилем через `OneToOneField`, либо собственной моделью на базе `AbstractUser`.
- **Формы** (`UserCreationForm`, `AuthenticationForm`, формы смены пароля) валидируют ввод и берут на себя проверку совпадения паролей.
- **Вьюхи** (`LoginView`, `LogoutView`, `PasswordChangeView`) уже реализуют поток входа-выхода, остаётся подсунуть шаблон.
- **Сессии** после успешного входа кладут id пользователя в сессионное хранилище, а браузеру выдают cookie с идентификатором сессии.
- **Группы и разрешения** позволяют выдавать права пачкой, а не каждому пользователю отдельно.

Регистрация на встроенной форме:

```python
from django.contrib.auth.forms import UserCreationForm
from django.shortcuts import render, redirect


def register(request):
    if request.method == 'POST':
        form = UserCreationForm(request.POST)
        if form.is_valid():
            form.save()
            return redirect('login')
    else:
        form = UserCreationForm()
    return render(request, 'register.html', {'form': form})
```

Вход через готовую вьюху и ограничение доступа декоратором:

```python
from django.contrib.auth.views import LoginView, LogoutView
from django.contrib.auth.decorators import login_required


class MyLoginView(LoginView):
    template_name = 'login.html'


class MyLogoutView(LogoutView):
    next_page = 'home'


@login_required
def create_post(request):
    ...
```

## Подводные камни

- **Кастомную модель пользователя меняют до первой миграции.** Переключить `AUTH_USER_MODEL` на запущенном проекте крайне болезненно, поэтому решение принимают в самом начале.
- **`@login_required` не равно проверке прав.** Декоратор пускает любого залогиненного, но не отделяет автора объекта от чужака — для «редактировать только своё» нужна отдельная проверка во вьюхе.
- **`LogoutView` по GET в новых версиях не работает:** выход требует POST, иначе ссылка-логаут молча ничего не делает.
- **Не путайте `authenticate()` и `login()`:** первый проверяет credentials и возвращает пользователя, второй кладёт его в сессию. Забыть второй вызов — частая ошибка кастомного входа.
- **Пароли никогда не сравнивают вручную:** только через хешер Django, иначе ломается безопасность хранения.
