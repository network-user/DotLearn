# Механизм CSRF-токенов в Django: как фреймворк отсекает поддельные запросы

## Суть

CSRF (Cross-Site Request Forgery) возникает, когда чужой сайт исподтишка инициирует запрос на ваш ресурс, пользуясь тем, что в браузере жертвы уже живёт активная сессия. Браузер послушно прикладывает к такому запросу куки авторизации, и сервер не отличает его от настоящего действия пользователя. Django закрывает эту дыру парным секретом: к каждому изменяющему запросу должен прийти CSRF-токен, который атакующий сайт получить и подставить не может.

## Что ответить на собесе

Ключевая мысль: одних куки для подтверждения действия недостаточно, нужен секрет, недоступный стороннему домену.

- Django требует, чтобы любой небезопасный по HTTP запрос (POST, PUT, PATCH, DELETE) нёс корректный CSRF-токен, иначе `CsrfViewMiddleware` отклонит его с ответом 403.
- Токен связан с конкретным пользователем и проверяется на совпадение.
- Чужой сайт не имеет доступа к содержимому вашей страницы и не может прочитать значение токена, поэтому физически не способен сформировать валидный запрос, даже если куки сессии браузер подставит сам.

## Как это работает

Защиту обеспечивает `CsrfViewMiddleware` плюс шаблонный тег `{% csrf_token %}`. Цикл такой:

1. **Выдача токена.** Когда рендерится страница с формой, Django кладёт CSRF-секрет в cookie (`csrftoken`) и вставляет связанное с ним значение в скрытое поле формы через тег.
2. **Возврат токена.** При отправке формы значение уходит на сервер: в обычной форме в виде скрытого поля `csrfmiddlewaretoken`, в AJAX в виде заголовка `X-CSRFToken`.
3. **Сверка.** На входящем POST middleware сравнивает токен из тела или заголовка с секретом из cookie. Если не совпало или пусто, запрос режется до того, как дойдёт до вью.

Обычная форма закрывается одним тегом:

```html
<form method="post">
  {% csrf_token %}
  <input type="text" name="username" placeholder="Username" />
  <input type="password" name="password" placeholder="Password" />
  <button type="submit">Login</button>
</form>
```

Тег `{% csrf_token %}` разворачивается в скрытый `<input>` с токеном, который автоматически уезжает вместе с POST.

Для AJAX токен приходится подставлять в заголовок руками; фреймворк сам этого не сделает:

```javascript
$.ajax({
  type: 'POST',
  url: '/your-url/',
  data: {
    // ваши данные
  },
  beforeSend: function (xhr, settings) {
    xhr.setRequestHeader('X-CSRFToken', getCookie('csrftoken'));
  },
  success: function (response) {
    // обработка успешного ответа
  },
});
```

Здесь `getCookie` вытаскивает значение токена из cookie `csrftoken`, чтобы положить его в заголовок.

## Подводные камни

- **AJAX без заголовка.** Самая частая ошибка: забыли добавить `X-CSRFToken`, и все POST-запросы летят в 403. Тег `{% csrf_token %}` помогает только обычным формам.
- **Только небезопасные методы.** GET/HEAD/OPTIONS не проверяются по дизайну. Поэтому изменяющие действия нельзя вешать на GET.
- **Куки недостаточно.** Защита держится на том, что чужой домен не прочитает токен из cookie вашего сайта; не отдавайте токен через CORS и не отключайте middleware «чтобы заработало».
- **`@csrf_exempt` точечно.** Снимать проверку можно только осознанно (например, для вебхука с собственной аутентификацией), а не глобально.
