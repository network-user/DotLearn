# What does Django give you out of the box for login and permission checks?

## In short

The `django.contrib.auth` subsystem brings a ready-made set: a user model, groups and permissions, forms and views for registration, login, and logout, plus storing the logged-in state in the session. As a result, access control comes together in minutes and without any home-grown cryptography.

## How to answer in an interview

Django covers both security concerns. Authentication (who you are) rests on the `User` model and backends that verify the username and password; passwords are stored as hashes, not plaintext. Authorization (what you're allowed to do) is built on groups and permissions. The ready-made `LoginView` and `LogoutView` handle login and logout, and the `@login_required` decorator or the `LoginRequiredMixin` mixin lock views away from anonymous users. The session mechanism carries the user's state between requests, so there's no need to log in again on every page. If the standard fields aren't enough, you create a custom model via `AUTH_USER_MODEL`.

## How it works

The subsystem's key building blocks:

- **The `User` model** stores `username`, `email`, the password hash, and the `is_active`, `is_staff`, `is_superuser` flags. It's extended either with a profile via a `OneToOneField` or with your own model based on `AbstractUser`.
- **Forms** (`UserCreationForm`, `AuthenticationForm`, the password-change forms) validate input and take care of checking that passwords match.
- **Views** (`LoginView`, `LogoutView`, `PasswordChangeView`) already implement the login/logout flow; you just slot in a template.
- **Sessions** put the user's id into the session store after a successful login, and hand the browser a cookie with the session identifier.
- **Groups and permissions** let you grant rights in bulk rather than to each user individually.

Registration with the built-in form:

```python
from django.contrib.auth.forms import UserCreationForm
from django.shortcuts import render, redirect


def register(request):
    if request.method == 'POST':
        form = UserCreationForm(request.POST)
        if form.is_valid():
            form.save()
            return redirect('login')
    else:
        form = UserCreationForm()
    return render(request, 'register.html', {'form': form})
```

Logging in through the ready-made view and restricting access with a decorator:

```python
from django.contrib.auth.views import LoginView, LogoutView
from django.contrib.auth.decorators import login_required


class MyLoginView(LoginView):
    template_name = 'login.html'


class MyLogoutView(LogoutView):
    next_page = 'home'


@login_required
def create_post(request):
    ...
```

## Pitfalls

- **The custom user model is changed before the first migration.** Switching `AUTH_USER_MODEL` on a running project is extremely painful, so the decision is made at the very start.
- **`@login_required` is not the same as a permission check.** The decorator lets any logged-in user through, but it doesn't separate an object's author from a stranger - for "edit only your own" you need a separate check in the view.
- **`LogoutView` over GET no longer works in newer versions:** logout requires POST, otherwise a logout link silently does nothing.
- **Don't confuse `authenticate()` and `login()`:** the first checks credentials and returns the user, the second places them into the session. Forgetting the second call is a common mistake in custom login flows.
- **Passwords are never compared by hand:** only through Django's hasher, otherwise the security of password storage breaks down.
