# Путь HTTP-запроса внутри Django: от URL до ответа

## Суть

Получив запрос, Django сопоставляет его URL с правилами в URLconf, находит нужное представление и вызывает его. Представление выполняет логику, при необходимости тянет данные через модели и возвращает `HttpResponse` - HTML, JSON или другой формат, который уходит обратно клиенту.

## Что ответить на собесе

Жизненный цикл запроса в Django такой: запрос сначала проходит стек middleware на входе, затем URL-резолвер по URLconf подбирает view, сверяя путь с `urlpatterns`. View - это функция или класс, где живёт прикладная логика: при необходимости она обращается к моделям, которые через ORM делают запросы к БД. View формирует `HttpResponse`, обычно рендеря шаблон с данными. На обратном пути ответ снова проходит middleware и отправляется клиенту. Главное - понимать, что URLconf маршрутизирует, view содержит логику, модель отвечает за данные, а шаблон за разметку.

**Маршрутизация.** Django сверяет путь запроса со списком шаблонов и выбирает первое совпадение.

```python
from django.urls import path
from . import views

urlpatterns = [
    path("home/", views.home, name="home"),
    path("articles/", views.article_list, name="article-list"),
]
```

**Простое представление** может вернуть ответ напрямую.

```python
from django.http import HttpResponse


def home(request):
    return HttpResponse("Добро пожаловать на главную!")
```

**Обращение к данным** идёт через модели и ORM, а результат отдаётся в шаблон.

```python
from django.shortcuts import render
from .models import Article


def article_list(request):
    articles = Article.objects.all()
    return render(request, "articles/list.html", {"articles": articles})
```

Итог: запрос -> middleware -> URLconf -> view -> (модели через ORM) -> шаблон -> `HttpResponse` -> middleware -> клиент.

## Подводные камни

- **Порядок в `urlpatterns` важен.** Django берёт первое совпадение сверху вниз; слишком общий паттерн выше «затенит» более конкретный ниже.
- **Middleware - часть цикла.** Запрос доходит до view не сразу: сначала отрабатывает стек middleware, и любой из них может вернуть ответ раньше.
- **Ленивость QuerySet.** `Article.objects.all()` не бьёт в базу сразу - запрос выполнится при итерации в шаблоне. Это влияет и на момент возникновения ошибок, и на N+1.
- **View обязан вернуть `HttpResponse`.** Забытый `return` или возврат «голых» данных вместо `HttpResponse`/`render` приведёт к ошибке.
