# Why put DRF on top of Django, and what does it handle for you?

## In short

Django REST Framework (DRF) is a layer on top of Django that handles the boilerplate of building web APIs: serialization, authentication, permissions, and request parsing. Thanks to its ready-made classes, a RESTful service comes together many times faster than building it by hand on plain Django.

## How to answer in an interview

DRF is a toolkit for REST APIs on top of Django. It frees you from boilerplate code: serializers translate models into JSON and back, class-based views handle the HTTP methods, and authentication mechanisms (tokens, JWT, OAuth) and permissions decide who can do what. I reach for DRF when I need an API for an SPA, a mobile client, or an external integration, and I can override almost any of its components for the project when needed.

DRF is built from several layers, each with its own area of responsibility.

**Serializers** turn complex objects (a QuerySet, for example) into simple Python types ready to be returned as JSON, and validate incoming data on write.

```python
from rest_framework import serializers
from .models import Article

class ArticleSerializer(serializers.ModelSerializer):
    class Meta:
        model = Article
        fields = ['id', 'title', 'content', 'author']
```

**Views** take a request and form a response. They're written as functions or classes; the `APIView` class gives you one method per HTTP verb.

```python
from rest_framework.views import APIView
from rest_framework.response import Response
from rest_framework import status
from .models import Article
from .serializers import ArticleSerializer

class ArticleList(APIView):
    def get(self, request):
        articles = Article.objects.all()
        serializer = ArticleSerializer(articles, many=True)
        return Response(serializer.data)

    def post(self, request):
        serializer = ArticleSerializer(data=request.data)
        if serializer.is_valid():
            serializer.save()
            return Response(serializer.data, status=status.HTTP_201_CREATED)
        return Response(serializer.errors, status=status.HTTP_400_BAD_REQUEST)
```

**Authentication and permissions** are wired in declaratively: authentication classes work out who the caller is, permission classes decide what they're allowed to do. **Routing** relies on Django's standard URLs, and for ViewSets there are automatic routers.

```python
from django.urls import path
from .views import ArticleList

urlpatterns = [
    path('articles/', ArticleList.as_view()),
]
```

On top of all this there's the self-documenting browsable API and schema generation, which makes debugging and testing easier.

## Pitfalls

- `ModelSerializer` is convenient, but `fields = '__all__'` is dangerous: it's easy to accidentally expose a password or an internal field. List the fields explicitly.
- A serializer validates data only when you call `is_valid()`; accessing `serializer.data` before that on input data raises an error.
- Without explicit `permission_classes`, the default settings kick in - under which an endpoint may turn out to be open. Check `DEFAULT_PERMISSION_CLASSES`.
- DRF pulls in an extra dependency and a layer of abstraction; for a couple of simple endpoints a plain `JsonResponse` is sometimes cheaper.
