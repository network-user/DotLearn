# Зачем поверх Django ставят DRF и что он берёт на себя?

## Суть

Django REST Framework (DRF) — надстройка над Django, которая закрывает рутину построения веб-API: сериализацию, аутентификацию, права доступа и разбор запросов. Благодаря готовым классам RESTful-сервис собирается в разы быстрее, чем вручную на голом Django.

## Что ответить на собесе

DRF — это инструментарий для REST API поверх Django. Он избавляет от шаблонного кода: сериализаторы переводят модели в JSON и обратно, классовые view обрабатывают HTTP-методы, а механизмы аутентификации (токены, JWT, OAuth) и permissions решают, кому и что можно. Я беру DRF, когда нужен API для SPA, мобильного клиента или внешней интеграции, и почти любой его компонент при необходимости переопределяю под проект.

DRF строится из нескольких слоёв, каждый со своей зоной ответственности.

**Сериализаторы** превращают сложные объекты (например, QuerySet) в простые Python-типы, готовые к отдаче как JSON, и валидируют входящие данные на запись.

```python
from rest_framework import serializers
from .models import Article

class ArticleSerializer(serializers.ModelSerializer):
    class Meta:
        model = Article
        fields = ['id', 'title', 'content', 'author']
```

**Представления** принимают запрос и формируют ответ. Их пишут функциями или классами; класс `APIView` даёт по методу на каждый HTTP-глагол.

```python
from rest_framework.views import APIView
from rest_framework.response import Response
from rest_framework import status
from .models import Article
from .serializers import ArticleSerializer

class ArticleList(APIView):
    def get(self, request):
        articles = Article.objects.all()
        serializer = ArticleSerializer(articles, many=True)
        return Response(serializer.data)

    def post(self, request):
        serializer = ArticleSerializer(data=request.data)
        if serializer.is_valid():
            serializer.save()
            return Response(serializer.data, status=status.HTTP_201_CREATED)
        return Response(serializer.errors, status=status.HTTP_400_BAD_REQUEST)
```

**Аутентификация и права** подключаются декларативно: классы authentication разбирают, кто пришёл, классы permission решают, что ему доступно. **Роутинг** опирается на штатные URL Django, а для ViewSet есть автоматические роутеры.

```python
from django.urls import path
from .views import ArticleList

urlpatterns = [
    path('articles/', ArticleList.as_view()),
]
```

Сверху — самодокументирующийся browsable API и генерация схемы, что упрощает отладку и тестирование.

## Подводные камни

- `ModelSerializer` удобен, но `fields = '__all__'` опасен: легко случайно отдать наружу пароль или служебное поле. Перечисляйте поля явно.
- Сериализатор валидирует данные только если вызвать `is_valid()`; обращение к `serializer.data` до этого на входных данных бросит ошибку.
- Без явных `permission_classes` срабатывают настройки по умолчанию — по ним эндпоинт может оказаться открытым. Проверяйте `DEFAULT_PERMISSION_CLASSES`.
- DRF тянет дополнительную зависимость и слой абстракций; для пары простых эндпоинтов иногда дешевле обычный `JsonResponse`.
