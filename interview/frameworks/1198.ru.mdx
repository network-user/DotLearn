# Транзакции в Django ORM: как обеспечить атомарность операций с БД

## Суть

Транзакция склеивает несколько запросов к базе в одну неделимую операцию: либо применяется всё, либо при ошибке откатывается всё. В Django ядром управления служит `transaction.atomic` - он работает и как декоратор, и как контекстный менеджер.

## Что ответить на собесе

Транзакция гарантирует, что группа изменений в БД либо целиком зафиксируется, либо целиком отменится при ошибке - так данные никогда не остаются в полупримененном состоянии. Django по умолчанию оборачивает каждый HTTP-запрос в транзакцию (режим `ATOMIC_REQUESTS`), но для тонкого контроля я беру `transaction.atomic`: декоратором на функцию или блоком `with`. Внутри блока любое необработанное исключение откатывает все изменения. Вложенные `atomic` создают точки сохранения (savepoints), что позволяет откатывать только часть операций.

## Как это работает

Atomic гарантирует свойство "всё или ничего". Классический пример - перевод денег: списание и зачисление обязаны произойти вместе, иначе баланс разойдется.

Декоратор оборачивает в транзакцию всю функцию:

```python
from django.db import transaction

@transaction.atomic
def transfer_funds(from_account, to_account, amount):
    from_account.balance -= amount
    from_account.save()
    to_account.balance += amount
    to_account.save()
```

Контекстный менеджер ограничивает транзакцию конкретным блоком - удобно, когда атомарным должен быть лишь кусок логики:

```python
from django.db import transaction

def complete_order(order, product):
    with transaction.atomic():
        order.status = 'completed'
        order.save()
        product.stock -= order.quantity
        product.save()
```

Если `product.save()` упадет, статус заказа тоже откатится. Фиксация (COMMIT) происходит только при чистом выходе из блока; исключение приводит к ROLLBACK.

## Подводные камни

- Откат запускает только исключение, выброшенное наружу из блока. Если поймать ошибку внутри `atomic` и проглотить ее, транзакция спокойно закоммитится с битыми данными.
- После того как внутри `atomic` произошла ошибка БД (например, `IntegrityError`), нельзя продолжать выполнять запросы в том же блоке - Django пометит транзакцию как "broken" и выбросит `TransactionManagementError`. Ловите исключение снаружи `atomic` или используйте вложенный блок.
- `transaction.on_commit()` нужен, когда побочный эффект (отправка письма, постановка задачи в очередь) должен сработать только после реального COMMIT, а не до возможного отката.
- Длинные транзакции держат блокировки строк и раздувают конкуренцию - не оборачивайте в `atomic` вызовы внешних API и долгие вычисления.
