# На каких базовых классах держится приложение Django?

## Суть

Django задаёт несколько опорных базовых классов, от которых наследуется ваш код: `Model` — данные и работа с БД, `View` — обработка HTTP-запросов, `Form` — формы и валидация ввода, плюс регистрация в `admin` для готовой админ-панели.

## Что ответить на собесе

Скелет любого Django-приложения держится на четырёх китах, и вместе они покрывают паттерн MTV, по которому устроен фреймворк.

- `Model` описывает структуру данных — каждый класс соответствует таблице, а ORM сам генерирует SQL.
- `View` принимает запрос и отдаёт ответ; пишется функцией или классом от `django.views.View`.
- `Form` валидирует и обрабатывает пользовательский ввод.
- Зарегистрировав модель в админке, получаешь CRUD-интерфейс почти без кода.

## Как это работает

**Model** — наследник `django.db.models.Model`. Атрибуты-поля становятся столбцами таблицы, а Django строит миграции и SQL.

```python
from django.db import models

class Book(models.Model):
    title = models.CharField(max_length=100)
    author = models.CharField(max_length=50)
    published_date = models.DateField()
```

**View** принимает `request` и возвращает `HttpResponse`. Классовый вариант наследуется от `View`, и метод именуется по HTTP-глаголу.

```python
from django.http import HttpResponse
from django.views import View

class HelloWorldView(View):
    def get(self, request):
        return HttpResponse('Hello, World!')
```

**Form** наследуется от `django.forms.Form`, описывает поля и берёт на себя валидацию и рендеринг.

```python
from django import forms

class ContactForm(forms.Form):
    name = forms.CharField(max_length=100)
    email = forms.EmailField()
    message = forms.CharField(widget=forms.Textarea)
```

**Admin** — здесь нет своего базового класса для наследования в простом случае: модель просто регистрируют, и Django генерирует интерфейс. Для тонкой настройки наследуют `admin.ModelAdmin`.

```python
from django.contrib import admin
from .models import Book

admin.site.register(Book)
```

## Подводные камни

- `Model` и `View` решают разные задачи: не тащите бизнес-логику и запросы к БД прямо во view, выносите в модели/менеджеры или сервисный слой.
- `forms.Form` валидирует, только когда вызван `form.is_valid()`; до этого `cleaned_data` недоступен.
- Для CRUD по модели удобнее `forms.ModelForm`, а не голый `Form` — он сам строит поля из модели.
- `admin.site.register(Book)` без `ModelAdmin` даёт интерфейс по умолчанию; для `list_display`, фильтров и поиска нужен класс-наследник `ModelAdmin`.
