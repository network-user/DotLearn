# WebSocket в Django: почему нужен Channels и как собрать ASGI-стек

## Суть

Сам Django заточен под HTTP и WebSocket не умеет; двусторонние соединения добавляет библиотека Django Channels, которая переводит проект на ASGI и обрабатывает постоянные соединения через консьюмеры.

## Что ответить на собесе

Голый Django синхронный и работает по модели запрос-ответ, поэтому держать постоянное WebSocket-соединение он не может. Для real-time я подключаю Django Channels: она переводит приложение на ASGI, поднимается асинхронным сервером вроде Daphne или Uvicorn, а сообщения между процессами гоняет через channel layer (обычно Redis). Логику соединения описываю в консьюмере - классе с методами `connect`, `disconnect`, `receive`. Маршруты WebSocket объявляю отдельно от обычных URL и собираю всё в `ProtocolTypeRouter`, который разводит http и websocket по разным веткам.

## Как это работает

Ключевая разница: HTTP - это разовый цикл запрос-ответ, а WebSocket держит канал открытым, и сервер может слать данные сам, без опроса со стороны клиента. Channels достраивает Django до ASGI и вводит абстракцию консьюмера - аналог view, но для длительных соединений.

Установка:

```bash
pip install channels
```

Регистрация приложения и точки входа ASGI в `settings.py`:

```python
INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    'channels',
]

ASGI_APPLICATION = 'myproject.asgi.application'
```

Сборка ASGI-приложения с разводкой протоколов:

```python
import os
from django.core.asgi import get_asgi_application
from channels.routing import ProtocolTypeRouter, URLRouter
from channels.auth import AuthMiddlewareStack
import myapp.routing

os.environ.setdefault('DJANGO_SETTINGS_MODULE', 'myproject.settings')

application = ProtocolTypeRouter({
    "http": get_asgi_application(),
    "websocket": AuthMiddlewareStack(
        URLRouter(myapp.routing.websocket_urlpatterns)
    ),
})
```

Маршруты WebSocket в `routing.py`:

```python
from django.urls import re_path
from . import consumers

websocket_urlpatterns = [
    re_path(r'ws/somepath/$', consumers.MyConsumer.as_asgi()),
]
```

Консьюмер обрабатывает события соединения:

```python
import json
from channels.generic.websocket import AsyncWebsocketConsumer

class MyConsumer(AsyncWebsocketConsumer):
    async def connect(self):
        await self.accept()

    async def disconnect(self, close_code):
        pass

    async def receive(self, text_data):
        data = json.loads(text_data)
        message = data['message']
        await self.send(text_data=json.dumps({'message': message}))
```

## Подводные камни

- WSGI-сервер (gunicorn без воркеров ASGI) WebSocket не поднимет. Нужен ASGI-сервер: Daphne или Uvicorn.
- Чтобы рассылать сообщение нескольким клиентам (групповой чат), одного консьюмера мало - нужен channel layer (`channels_redis` поверх Redis) и группы; в памяти `InMemoryChannelLayer` годится только для разработки.
- В асинхронном консьюмере нельзя дёргать синхронный Django ORM напрямую - это блокирует event loop. Используйте `database_sync_to_async`.
- `AuthMiddlewareStack` даёт доступ к пользователю в `scope['user']`, но только если сессия/кука пришли в рукопожатии; токены придётся обрабатывать своим middleware.
- WebSocket не проходит обычные middleware и декораторы Django (CSRF, login_required) - их аналоги живут в стеке Channels.
