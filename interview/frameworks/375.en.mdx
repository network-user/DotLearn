# What contract WSGI defines and how it separates the web server from the Python application

## In short

WSGI defines a single calling protocol between an HTTP server and a Python application: the server assembles the request and calls the application, the application returns the response. As long as both sides honor this protocol, they can be freely combined - the same framework code runs under different servers without changes.

## How to answer in an interview

WSGI (Web Server Gateway Interface) is a synchronous interface between a web server and a Python application, described in PEP 3333. The application here is any callable with two parameters: an `environ` dictionary (everything about the incoming request) and a `start_response` callback (used to set the status code and headers). In return, the application yields an iterable of body bytes. It's precisely this single protocol that lets Django or Flask run under Gunicorn, uWSGI, or Waitress with no code changes. WSGI doesn't cover async scenarios - for those there is ASGI.

## How it works

The standard splits the world into two roles and draws a hard boundary between them:

- **The WSGI server** holds the TCP connections, parses HTTP, fills `environ`, and calls the application. This includes Gunicorn, uWSGI, Waitress.
- **The WSGI application** reads `environ`, assembles the response, and returns bytes. This can be a single function or a whole framework.

The simplest application is a function with the required signature:

```python
def simple_app(environ, start_response):
    status = "200 OK"
    headers = [("Content-Type", "text/plain; charset=utf-8")]
    start_response(status, headers)
    return [b"Hello, World!"]
```

- `environ` - a dictionary with request data: method, path, headers, environment variables.
- `start_response` - a callback the application must call before it returns the body; it is passed a status string and a list of headers.

To launch the application, you hand it to the server. With Gunicorn it looks like this:

```bash
gunicorn myapp:simple_app
```

Here `myapp` is the module name, and `simple_app` is the callable application object inside it.

## Pitfalls

- WSGI is synchronous at its core: you can't `await` inside a single call. For WebSockets, long-polling, and async views you use ASGI (Uvicorn, Daphne), not WSGI.
- The response body is bytes, not strings. Returning `["text"]` instead of `[b"text"]` will crash the server with an error.
- `start_response` is called exactly once and before the server begins reading the body; a repeated or late call breaks the contract.
- WSGI is an interface, not a ready-made server. The built-in `runserver` isn't fit for production: under load you deploy Gunicorn or uWSGI.
