# Зачем нужен слой middleware и как он перехватывает запрос

## Суть

Middleware - это перехватчик, встроенный между приёмом HTTP-запроса и формированием ответа. Он позволяет вынести сквозную логику (аутентификацию, логирование, обработку ошибок) из view в один общий слой, чтобы не дублировать её в каждом обработчике.

## Что ответить на собесе

Middleware в Django - это компоненты, через которые проходит каждый запрос по пути к view и каждый ответ на обратном пути.

- Они выстроены в стек (`MIDDLEWARE` в настройках), и порядок важен: на входе запрос идёт сверху вниз, на выходе ответ - снизу вверх.
- Использую middleware для задач, общих для всех представлений: проверка сессии, заголовки безопасности, замер времени, единая обработка исключений.
- Это держит view сфокусированными на бизнес-логике, а кросс-функциональные вещи решаются централизованно.

Современное middleware в Django - это вызываемый объект, который получает `get_response` при инициализации и вызывается на каждый запрос. До `self.get_response(request)` выполняется код «на входе», после - «на выходе».

```python
import time


class TimingMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        started = time.monotonic()

        response = self.get_response(request)

        elapsed_ms = (time.monotonic() - started) * 1000
        response["X-Response-Time-ms"] = f"{elapsed_ms:.1f}"
        return response
```

Здесь код до вызова `get_response` стартует таймер, а код после - дописывает заголовок с длительностью. Каждый слой стека оборачивает следующий, образуя «луковицу»: запрос погружается внутрь до view, ответ всплывает обратно. Дополнительно можно объявить хуки `process_view`, `process_exception` и `process_template_response`.

## Подводные камни

- **Порядок в `MIDDLEWARE` критичен.** Например, `AuthenticationMiddleware` должен идти после `SessionMiddleware`, иначе `request.user` будет недоступен.
- **Раннее завершение цепочки.** Если middleware вернёт ответ, не вызвав `get_response`, нижележащие слои и view не выполнятся - это удобно для редиректов, но легко выстрелить себе в ногу.
- **Исключения и порядок выхода.** Ответ обрабатывается слоями в обратном порядке; `process_exception` срабатывает только если view бросил исключение, и проверяется снизу вверх.
- **Производительность.** Тяжёлая логика в middleware выполняется для каждого запроса, включая статику и health-check, поэтому держите её лёгкой.
