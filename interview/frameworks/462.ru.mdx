# Бизнес-логика против представления: модель или сериализатор DRF

## Суть

Доменные правила и инварианты данных живут в модели (или сервисном слое), а сериализатор отвечает только за перевод объектов между Python и JSON и за валидацию входного формата. Смешивать эти роли - значит привязывать бизнес-логику к конкретному API.

## Что ответить на собесе

Я держу бизнес-логику ближе к данным - в модели или в отдельном сервисном слое, а не в сериализаторе. Причина простая: сериализатор привязан к конкретному API-эндпоинту, и если расчёт скидки или проверка остатка живут в нём, то при вызове из админки, management-команды или фоновой задачи это правило просто не сработает. Сериализатор отвечает за форму данных на проводе: какие поля отдать, как их преобразовать, как провалидировать входящий формат. Модель отвечает за то, что данные корректны независимо от того, кто их трогает.

## Как это работает

Модель описывает данные и доменное поведение над ними - методы, считающие что-то по полям самого объекта:

```python
from django.db import models

class Product(models.Model):
    name = models.CharField(max_length=100)
    price = models.DecimalField(max_digits=10, decimal_places=2)
    stock = models.IntegerField()

    def discounted_price(self, percentage):
        return self.price * (1 - percentage / 100)
```

Сериализатор не дублирует расчёт, а вызывает метод модели, оставаясь тонким слоем представления:

```python
from rest_framework import serializers

class ProductSerializer(serializers.ModelSerializer):
    discounted_price = serializers.SerializerMethodField()

    class Meta:
        model = Product
        fields = ['name', 'price', 'discounted_price']

    def get_discounted_price(self, obj):
        return obj.discounted_price(10)
```

Разделение по ролям:

- Инварианты и доменные правила (можно ли продать товар, как считается цена) - модель или сервис.
- Валидация конкретного запроса и форма ответа (формат даты, склейка полей, какие поля видны) - сериализатор.

## Подводные камни

- Валидация бывает двух сортов. Проверку «email уникален в этом запросе» уместно делать в сериализаторе; инвариант «остаток не уходит в минус» должен жить в модели/сервисе, иначе его обойдёт любой код мимо API.
- `Model.save()` в Django не вызывает `full_clean()` автоматически, а DRF-валидация срабатывает только при сохранении через сериализатор. Если рассчитываете только на сериализатор, прямые `objects.create()` пройдут без проверок.
- Толстые сериализаторы с бизнес-логикой почти невозможно переиспользовать: для второго эндпоинта с другим набором полей логику придётся копировать.
- Когда правило задействует несколько моделей или внешние сервисы, ему тесно и в модели тоже - выносите в отдельный сервисный слой, а не размазывайте по сериализаторам.
- `SerializerMethodField` поля доступны только на чтение; для записи бизнес-значений нужен `create()`/`update()` либо обработка в модели.
