# Which base classes does a Django application rest on?

## In short

Django defines several foundational base classes your code inherits from: `Model` - data and database work, `View` - handling HTTP requests, `Form` - forms and input validation, plus registration in `admin` for a ready-made admin panel.

## How to answer in an interview

The skeleton of any Django application rests on four pillars. `Model` describes the data structure - each class corresponds to a table, and the ORM generates the SQL itself. `View` is the layer that takes a request and returns a response; it's written as a function or a class subclassing `django.views.View`. `Form` validates and processes user input. And by registering a model in the admin, you get a CRUD interface with almost no code. These four cover the MTV pattern the framework is built on.

## How it works

**Model** subclasses `django.db.models.Model`. Field attributes become table columns, and Django builds the migrations and SQL.

```python
from django.db import models

class Book(models.Model):
    title = models.CharField(max_length=100)
    author = models.CharField(max_length=50)
    published_date = models.DateField()
```

**View** takes a `request` and returns an `HttpResponse`. The class-based variant subclasses `View`, and the method is named after the HTTP verb.

```python
from django.http import HttpResponse
from django.views import View

class HelloWorldView(View):
    def get(self, request):
        return HttpResponse('Hello, World!')
```

**Form** subclasses `django.forms.Form`, describes the fields, and takes on validation and rendering.

```python
from django import forms

class ContactForm(forms.Form):
    name = forms.CharField(max_length=100)
    email = forms.EmailField()
    message = forms.CharField(widget=forms.Textarea)
```

**Admin** - in the simple case there's no dedicated base class to inherit from: you just register the model and Django generates the interface. For fine-grained tuning you subclass `admin.ModelAdmin`.

```python
from django.contrib import admin
from .models import Book

admin.site.register(Book)
```

## Pitfalls

- `Model` and `View` solve different problems: don't drag business logic and database queries straight into the view, move them into models/managers or a service layer.
- `forms.Form` validates only once `form.is_valid()` is called; before that, `cleaned_data` is unavailable.
- For CRUD on a model, `forms.ModelForm` is more convenient than a bare `Form` - it builds the fields from the model itself.
- `admin.site.register(Book)` without a `ModelAdmin` gives you the default interface; for `list_display`, filters, and search you need a class subclassing `ModelAdmin`.
