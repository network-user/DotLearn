# Why a middleware layer exists and how it intercepts a request

## In short

Middleware is an interceptor wired in between receiving an HTTP request and producing a response. It lets you pull cross-cutting logic (authentication, logging, error handling) out of views and into a single shared layer, so you don't duplicate it in every handler.

## How to answer in an interview

In Django, middleware are components that every request passes through on the way to a view and every response passes through on the way back. They are arranged in a stack (`MIDDLEWARE` in settings), and order matters: on the way in the request flows top to bottom, on the way out the response flows bottom to top. I use middleware for tasks common to all views: checking the session, adding security headers, measuring timing, unified exception handling. This keeps views focused on business logic, while cross-functional concerns are handled centrally.

Modern Django middleware is a callable that receives `get_response` on initialization and is invoked for each request. The code "on the way in" runs before `self.get_response(request)`, and the code "on the way out" runs after.

```python
import time


class TimingMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        started = time.monotonic()

        response = self.get_response(request)

        elapsed_ms = (time.monotonic() - started) * 1000
        response["X-Response-Time-ms"] = f"{elapsed_ms:.1f}"
        return response
```

Here the code before the `get_response` call starts the timer, and the code after it appends a header with the duration. Each layer of the stack wraps the next, forming an "onion": the request descends inward toward the view, and the response bubbles back out. You can additionally declare the `process_view`, `process_exception`, and `process_template_response` hooks.

## Pitfalls

- **Order in `MIDDLEWARE` is critical.** For example, `AuthenticationMiddleware` must come after `SessionMiddleware`, otherwise `request.user` won't be available.
- **Short-circuiting the chain.** If a middleware returns a response without calling `get_response`, the lower layers and the view won't run - handy for redirects, but easy to shoot yourself in the foot.
- **Exceptions and exit order.** The response is processed by the layers in reverse order; `process_exception` fires only if the view raised an exception, and it is checked bottom to top.
- **Performance.** Heavy logic in middleware runs for every request, including static files and health checks, so keep it lightweight.
