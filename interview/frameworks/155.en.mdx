# How Django splits an application into Model, View, and Template

## In short

MVT (Model-View-Template) is the separation of concerns Django is built on: the Model holds data and the logic for working with it, the View handles the request and decides what to show, and the Template is responsible for the markup of the response itself. This split simplifies maintenance and lets you change presentation independently of logic.

## How to answer in an interview

MVT is Django's flavor of MVC, where the framework itself (the URL router) plays the controller role, and the "View" layer in Django terms handles application logic. The Model describes the data structure and encapsulates database access through the ORM. The View receives the `request`, pulls data from the models, and chooses which template to render. The Template is HTML with the Django template language, which turns data into finished markup. The key idea is to separate data, logic, and presentation so each layer can change independently.

## How it works

**Model** inherits from `models.Model` and describes a table with its fields. Through the ORM, the model hides SQL behind CRUD operations.

```python
from django.db import models


class Book(models.Model):
    title = models.CharField(max_length=100)
    author = models.CharField(max_length=50)
    published_date = models.DateField()
```

**View** receives the `request`, fetches data from the model, and passes it to the template.

```python
from django.shortcuts import render
from .models import Book


def book_list(request):
    books = Book.objects.all()
    return render(request, "books/book_list.html", {"books": books})
```

**Template** is HTML with the Django template language, which substitutes the data.

```html
<h1>Books</h1>
<ul>
{% for book in books %}
    <li>{{ book.title }} - {{ book.author }}</li>
{% endfor %}
</ul>
```

The chain is: the router selects a view by URL, the view pulls data from the model and hands it to the template, and the assembled HTML goes to the client as the HTTP response.

## Pitfalls

- **MVC vs MVT terminology confusion.** In Django, "View" is not the MVC "view" (which there is the template) but closer to the controller. This is a frequent source of misunderstanding in interviews.
- **Business logic in templates.** Templates are intentionally limited; moving computation into them is an anti-pattern - keep logic in models and views.
- **"Fat" views.** Data-handling logic is better extracted into model methods or managers, otherwise views grow bloated and become hard to test.
- **The controller is "hidden."** The controller role is filled by the URLconf and the framework itself, so there's no separate "Controller" layer to look for in the project.
