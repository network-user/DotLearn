# What tools does Django use to speed up responses through caching?

## In short

Out of the box, Django provides a multi-level caching system on top of Memcached or Redis: you can store a whole page, a single fragment of a template, or arbitrary data through the low-level API. The idea is the same - not to recompute an expensive result over and over.

## How to answer in an interview

Caching in Django works at three altitudes. The `@cache_page` decorator wraps a view and serves the ready-made HTML until the timeout expires. The `{% cache %}` template tag caches only a fragment - for example, a shared menu. And when flexibility is needed, I reach for the low-level API `django.core.cache.cache` with its `set`/`get` methods and decide myself what to store and for how long. The backend (Redis, Memcached, local memory) is configured with a single `CACHES` section in `settings.py`, so the caching code does not depend on the choice of store.

## How it works

A cache is a temporary store of ready results: on a repeat request we fetch them from there instead of running database queries and heavy rendering again. This takes the load off the database and shortens the response time.

The backend is set in the `CACHES` section. For Redis:

```python
CACHES = {
    'default': {
        'BACKEND': 'django_redis.cache.RedisCache',
        'LOCATION': 'redis://127.0.0.1:6379/1',
        'OPTIONS': {
            'CLIENT_CLASS': 'django_redis.client.DefaultClient',
        },
    }
}
```

Caching a whole view - the `@cache_page` decorator, whose argument sets the lifetime in seconds:

```python
from django.views.decorators.cache import cache_page

@cache_page(60 * 15)  # keep the response for 15 minutes
def my_view(request):
    return HttpResponse('Hello, World!')
```

Caching a template fragment - the `load cache` and `cache` tags; the block has a timeout and a key name:

```html
{% load cache %} {% cache 500 sidebar %}
<div>
  <!-- content that is heavy to render -->
</div>
{% endcache %}
```

The low-level API - storing and reading arbitrary values:

```python
from django.core.cache import cache

cache.set('my_key', 'my_value', timeout=60 * 15)
value = cache.get('my_key')  # 'my_value', or None after expiry
```

## Pitfalls

- `@cache_page` caches the page identically for everyone, so it cannot be applied to personal data (profile, cart) - otherwise one user would see another user's content. A whole-page cache varies by URL and by the headers in `Vary`.
- `cache.get` for a missing or expired key returns `None`, not an exception. If a valid value can itself be `None`, pass a sentinel object as the default: `cache.get(key, MISSING)`.
- The timeout is in seconds. `timeout=None` means "forever", `timeout=0` means "do not cache at all"; this is a common source of confusion.
- The main pain of caching is invalidation. After the data changes, the stale cache must be explicitly cleared (`cache.delete`) or tied to key versions, otherwise the user will see the old data.
