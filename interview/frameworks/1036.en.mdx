# When to use @task and when @shared_task to declare Celery tasks?

## In short

`@task` binds a task to a specific Celery application instance, while `@shared_task` creates a task without a hard binding, which is why it's safe to declare in reusable modules and pluggable applications.

## How to answer in an interview

Both decorators register a function as an asynchronous task. The difference is in the binding to the `Celery` instance. `@task` is taken straight off a specific app object and registers the task in that one, so it fits when the task lives inside a single project and doesn't need to be reused. `@shared_task` returns a proxy that finds the right application at call time rather than at declaration time. That's critical for libraries and Django apps that get plugged into different projects: at import time a specific app may not exist yet. The practical rule: inside a reusable app or package always use `@shared_task`; for tasks hard-bound to a single app you can use `@task`.

## How it works

`@task` is usually taken off an application instance: `app.task`. The task is registered in that particular application's registry.

```python
from myproject.celery import app

@app.task
def add(x, y):
    return x + y
```

`@shared_task` doesn't require a reference to a specific app. It returns a proxy that resolves the currently active Celery application at call time. That's why such a task can be declared in a module that's imported before the app is created.

```python
from celery import shared_task

@shared_task
def multiply(x, y):
    return x * y
```

Both are launched the same way: the call doesn't run the function immediately but puts a message into the broker; a worker picks it up and executes it.

```python
add.delay(2, 3)
multiply.apply_async(args=[4, 5])
```

## Pitfalls

- In a reusable Django app, `app.task` can break: at the time `tasks.py` is imported, the needed app doesn't exist yet. `@shared_task` solves this with deferred resolution of the application.
- `@shared_task` isn't "magically global": it binds to the active instance via `autodiscover_tasks()` or an explicit import. If no application is configured, the task won't be registered.
- The bare `from celery import task` is deprecated; in modern versions, bound tasks use `app.task` rather than importing `task`.
- The decorator itself doesn't make the function parallel. Parallelism comes from queuing and workers; a direct call `add(2, 3)` runs the function synchronously, bypassing the broker.
