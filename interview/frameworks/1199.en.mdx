# Lazy vs eager loading in Django ORM: select_related and prefetch_related

## In short

A QuerySet in Django is lazy: SQL goes to the database only at the moment the data is actually needed. Eager loading pulls in related objects ahead of time - via `select_related()` (a JOIN for FK and one-to-one) and `prefetch_related()` (separate queries for M2M and reverse FK), which saves you from the N+1 problem.

## How to answer in an interview

By default the ORM is lazy: `Model.objects.all()` does not hit the database immediately but returns a QuerySet that is executed only on iteration, slicing, or `list()`. This is convenient, but when traversing related objects the N+1 problem is born: for each object - a separate query for its relation. The cure is eager loading. `select_related()` does a SQL JOIN and pulls "to-one" relations (ForeignKey, OneToOne) in a single query. `prefetch_related()` fetches "to-many" relations (ManyToMany, reverse FK) with a separate query and stitches the data together on the Python side. The choice depends on the type of relation.

A lazy QuerySet defers the trip to the database:

```python
queryset = MyModel.objects.all()  # SQL is NOT executed yet

for obj in queryset:              # this is where the query goes out
    print(obj.name)
```

Eager loading removes the extra queries when working with relations:

```python
# JOIN: a single SQL statement for FK / OneToOne
books = Book.objects.select_related('author').all()

# Separate query + stitching in Python: for M2M and reverse FK
authors = Author.objects.prefetch_related('book_set').all()
```

Without eager loading, the loop `for book in books: print(book.author.name)` would run one query for the list of books plus one query per author for each book - the very N+1. With `select_related('author')` everything goes out as a single JOIN.

## Pitfalls

- `select_related` cannot be applied to ManyToMany and reverse FK - there is no single row to JOIN against; for those only `prefetch_related` works.
- `prefetch_related` makes at least two queries, but in exchange it does not inflate the result with a Cartesian product the way a deep JOIN does. On large M2M relations this is often more efficient.
- Eager loading is not for every case: if the related objects are not used, it just pulls data for nothing. Profile with `django-debug-toolbar` or `connection.queries`.
- A `.filter()` chain after eager loading may require a `Prefetch()` with its own QuerySet, otherwise the relation filtering will not apply the way you expect.
- A QuerySet caches the result after the first execution, but a new method call (`.filter()`, `.exclude()`) creates a new QuerySet and a new query.
