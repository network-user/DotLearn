# What are the main building blocks Django is divided into?

## In short

Django is a high-level Python framework organized around the MTV (Model-Template-View) pattern, which is close to the classic MVC. Its load-bearing parts are: data models, views that hold business logic, templates for markup, URL routing, the ORM, and an auto-generated admin. Together they remove the boilerplate and leave the developer with the actual logic.

## How to answer in an interview

Django is built on the MTV scheme: the Model is responsible for data, the Template for display, and the View ties them together and holds the business logic. A model describes the structure of the data, and usually one model maps to one table; the ORM handles talking to the database, translating Python objects into SQL. A view receives a `request` and returns a `response`: HTML, JSON, or a redirect. Templates written in DTL assemble HTML and keep the markup separate from the logic. Routing (`urls.py`) maps addresses to views. On top of this, Django automatically builds an admin panel from the models. It is important not to mix up the terminology: what MVC calls the "controller" is called a View in Django, and the Template is the presentation layer.

The **model** defines the data schema, and the ORM frees you from hand-written SQL:

```python
from django.db import models


class Author(models.Model):
    name = models.CharField(max_length=100)
    birth_date = models.DateField()
```

The **view** handles the request and forms the response:

```python
from django.http import HttpResponse


def hello_world(request):
    return HttpResponse("Hello, world!")
```

**Routing** links a URL to a view:

```python
from django.urls import path
from . import views


urlpatterns = [
    path('hello/', views.hello_world, name='hello_world'),
]
```

A **template** in DTL injects data into the markup:

```html
<!DOCTYPE html>
<html>
<head><title>{{ title }}</title></head>
<body><h1>{{ message }}</h1></body>
</html>
```

On top of this work the **ORM** (objects instead of SQL queries) and the **admin**, which is generated from the models and gives you a ready-made web interface for managing data. The request chain is simple: the URL resolver finds a view, the view pulls data from the models through the ORM, passes it to a template, and returns the assembled response.

## Pitfalls

- **MTV vs MVC:** interviewers love to catch you on terminology. A View in Django is the "controller" from MVC, and Django's presentation layer is called the Template.
- **Business logic spreads:** Django does not impose a service layer, so logic often settles into views and bloats them. Fat models or a separate service layer save you from "fat views".
- **The admin is not a replacement for the user interface:** it is meant for administrators and developers, and exposing it to clients is not a good idea.
- **The ORM hides the cost of queries:** the convenience turns into the N+1 problem if you forget about `select_related`/`prefetch_related`.
- **An "app" in Django is a module, not the whole site:** a project consists of several apps, and confusing these terms is a common beginner mistake.
