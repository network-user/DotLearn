# Какими способами одна модель Django может наследовать другую?

## Суть

Django поддерживает три модели наследования, и различаются они тем, сколько таблиц рождается в базе. Абстрактный базовый класс раздаёт поля дочерним моделям, но своей таблицы не имеет. Многотабличное наследование заводит таблицу под каждую модель. Прокси меняет поведение, не трогая ни одну таблицу.

## Что ответить на собесе

Вариантов наследования три, и выбор сводится к вопросу: нужна ли отдельная таблица и нужны ли новые поля.

- Абстрактные модели (`abstract = True` в `Meta`) — чистый шаблон полей: таблицы у базового класса нет, поля копируются в каждую дочернюю модель, своя таблица только у потомков.
- Многотабличное наследование — базовый класс не абстрактный: у него своя таблица, у потомка своя, между ними неявная `OneToOne`-связь, и обращение к унаследованному полю делает JOIN.
- Прокси-модель (`proxy = True`) — та же таблица, но другой Python-класс: можно добавить методы, поменять менеджер или сортировку по умолчанию, без единого нового столбца.

**Абстрактная модель.** Базовый класс помечен `abstract = True`, поэтому таблицы под него не будет, а его поля переедут в каждого потомка:

```python
from django.db import models


class CommonInfo(models.Model):
    name = models.CharField(max_length=100)
    age = models.IntegerField()

    class Meta:
        abstract = True


class Student(CommonInfo):
    school = models.CharField(max_length=100)


class Teacher(CommonInfo):
    subject = models.CharField(max_length=100)
```

В базе появятся таблицы `Student` и `Teacher`, каждая со своими `name` и `age`. Таблицы `CommonInfo` нет.

**Многотабличное наследование.** Базовый класс обычный, не абстрактный:

```python
class BaseModel(models.Model):
    created_at = models.DateTimeField(auto_now_add=True)


class ChildModel(BaseModel):
    name = models.CharField(max_length=100)
```

Здесь две таблицы. У `ChildModel` поля `created_at` нет физически в своей таблице — оно достаётся через автоматическую связь с `BaseModel`, и запрос делает JOIN.

**Прокси-модель.** Та же таблица, но расширенное поведение:

```python
class Person(models.Model):
    name = models.CharField(max_length=100)


class PersonProxy(Person):
    class Meta:
        proxy = True

    def get_uppercase_name(self):
        return self.name.upper()
```

`PersonProxy` читает и пишет ровно в таблицу `Person`, новой таблицы нет, а метод `get_uppercase_name` доступен только через прокси.

## Подводные камни

- **Многотабличное наследование скрывает JOIN-ы:** каждое обращение к полю родителя — это запрос с соединением таблиц, на горячих путях это бьёт по производительности. Часто абстрактная модель предпочтительнее.
- **В прокси нельзя добавлять поля:** только методы, менеджеры и `Meta`-опции вроде `ordering`. Попытка объявить новое поле приведёт к ошибке.
- **`Meta` у абстрактного класса наследуется не полностью:** дочерняя модель получает копию `Meta`, но `abstract` сбрасывается в `False`, иначе потомок тоже стал бы абстрактным.
- **Меняешь схему — пиши миграции:** переход от абстрактной к многотабличной модели меняет структуру таблиц и требует аккуратной миграции данных.
- **У прокси и оригинала общие строки:** объект, сохранённый через `Person`, виден и через `PersonProxy` — это одна и та же запись, не копия.
