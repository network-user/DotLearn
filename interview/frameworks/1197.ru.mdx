# На какие основные строительные блоки разбит Django?

## Суть

Django — высокоуровневый Python-фреймворк, организованный по паттерну MTV (Model-Template-View), близкому к классическому MVC. Его несущие части: модели данных, представления с бизнес-логикой, шаблоны для разметки, маршрутизация URL, ORM и автогенерируемая админка. Вместе они снимают рутину и оставляют разработчику саму логику.

## Что ответить на собесе

Django построен по схеме MTV: Model отвечает за данные, Template за отображение, View связывает их и держит бизнес-логику.

- Модель описывает структуру данных (обычно одна модель — одна таблица); общение с базой берёт на себя ORM, переводя Python-объекты в SQL.
- Представление принимает `request` и возвращает `response`: HTML, JSON или редирект.
- Шаблоны на языке DTL собирают HTML и держат вёрстку отдельно от логики, а маршрутизация (`urls.py`) сопоставляет адреса представлениям; поверх этого Django автоматически строит админ-панель из моделей.
- Важно не путать терминологию: то, что в MVC зовут «контроллером», в Django называют View, а Template — это слой представления.

**Модель** задаёт схему данных, ORM избавляет от ручного SQL:

```python
from django.db import models


class Author(models.Model):
    name = models.CharField(max_length=100)
    birth_date = models.DateField()
```

**Представление** обрабатывает запрос и формирует ответ:

```python
from django.http import HttpResponse


def hello_world(request):
    return HttpResponse("Hello, world!")
```

**Маршрутизация** связывает URL с представлением:

```python
from django.urls import path
from . import views


urlpatterns = [
    path('hello/', views.hello_world, name='hello_world'),
]
```

**Шаблон** на DTL подставляет данные в разметку:

```html
<!DOCTYPE html>
<html>
  <head>
    <title>{{ title }}</title>
  </head>
  <body>
    <h1>{{ message }}</h1>
  </body>
</html>
```

Поверх этого работают **ORM** (объекты вместо SQL-запросов) и **админка**, которая генерируется из моделей и даёт готовый веб-интерфейс управления данными. Цепочка запроса проста: URL-резолвер находит вьюху, вьюха берёт данные из моделей через ORM, передаёт их в шаблон и возвращает собранный ответ.

## Подводные камни

- **MTV против MVC:** на собесе любят ловить на терминологии. View в Django — это «контроллер» из MVC, а слой представления у Django называется Template.
- **Бизнес-логика расползается:** Django не навязывает сервисный слой, поэтому логика часто оседает во вьюхах и раздувает их. Толстые модели или отдельный сервис-слой спасают от «жирных вьюх».
- **Админка — не замена интерфейсу пользователя:** она для администраторов и разработчиков, выставлять её клиентам не стоит.
- **ORM скрывает стоимость запросов:** удобство оборачивается проблемой N+1, если забыть про `select_related`/`prefetch_related`.
- **«Приложение» в Django — это модуль, а не весь сайт:** проект состоит из нескольких приложений, и путаница в этих терминах — частая ошибка новичков.
