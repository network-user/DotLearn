# Когда брать @task, а когда @shared_task при объявлении задач Celery?

## Суть

`@task` привязывает задачу к конкретному экземпляру приложения Celery, а `@shared_task` создаёт задачу без жёсткой привязки, поэтому её безопасно объявлять в переиспользуемых модулях и подключаемых приложениях.

## Что ответить на собесе

Оба декоратора регистрируют функцию как асинхронную задачу, разница — в привязке к экземпляру `Celery`.

- `@task` берётся прямо у конкретного app-объекта и регистрирует задачу именно в нём, поэтому подходит, когда задача живёт внутри одного проекта и переиспользовать её не нужно.
- `@shared_task` отдаёт прокси, который находит нужное приложение в момент вызова, а не объявления.
- Это критично для библиотек и подключаемых Django-приложений: на этапе импорта конкретного app ещё может не быть.
- Практическое правило: внутри reusable-приложения или пакета всегда `@shared_task`, для задач, жёстко привязанных к одному app, можно `@task`.

## Как это работает

`@task` обычно берут у экземпляра приложения: `app.task`. Задача регистрируется в реестре именно этого приложения.

```python
from myproject.celery import app

@app.task
def add(x, y):
    return x + y
```

`@shared_task` не требует ссылки на конкретный app. Он возвращает прокси, который при вызове разрешает текущее активное приложение Celery. Поэтому такую задачу можно объявить в модуле, который импортируется до создания app.

```python
from celery import shared_task

@shared_task
def multiply(x, y):
    return x * y
```

Запускаются обе одинаково: вызов не выполняет функцию сразу, а кладёт сообщение в брокер; воркер забирает его и выполняет.

```python
add.delay(2, 3)
multiply.apply_async(args=[4, 5])
```

## Подводные камни

- В переиспользуемом Django-приложении `app.task` может сломаться: на момент импорта `tasks.py` нужного app ещё нет. `@shared_task` решает это отложенным разрешением приложения.
- `@shared_task` не «магически глобальный»: он привязывается к активному экземпляру через `autodiscover_tasks()` или явный импорт. Если ни одно приложение не сконфигурировано, задача не зарегистрируется.
- Голый `from celery import task` устарел; в современных версиях для привязанных задач используют `app.task`, а не импорт `task`.
- Сам декоратор не делает функцию параллельной. Параллелизм даёт постановка в очередь и воркеры; прямой вызов `add(2, 3)` выполнит функцию синхронно, минуя брокер.
