# When is BST search no longer O(log n)?

## In short

A binary search tree is O(log n) only with bounded height. If inserts arrive sorted or the tree is otherwise unbalanced, it degenerates into a chain of height n and search becomes O(n).

## How to answer in an interview

- I define BST and the logarithmic condition: height h ≈ log n when balanced.
- I give the classic example: inserting 1,2,3,4,5 in order yields a right-skewed chain.
- I name fixes: AVL, red-black trees, STL `std::map`/`std::set`.
- I mention randomized BST (treap) and bulk build via sort plus balanced construction.
- I add honestly: a naive BST is fine to explain in interviews; production rarely uses one without balancing.

## How it works

Search compares the key with a node and moves left or right. Steps equal height. A balanced tree halves the set each level; a degenerate one visits all n nodes.

```cpp
#include <iostream>

struct TreeNode {
    int key;
    TreeNode* left = nullptr;
    TreeNode* right = nullptr;
    explicit TreeNode(int v) : key(v) {}
};

TreeNode* insert_bst(TreeNode* root, int value) {
    if (!root) return new TreeNode(value);
    if (value < root->key) root->left = insert_bst(root->left, value);
    else root->right = insert_bst(root->right, value);
    return root;
}

bool contains(TreeNode* root, int value) {
    while (root) {
        if (value == root->key) return true;
        root = (value < root->key) ? root->left : root->right;
    }
    return false;
}

int height(TreeNode* root) {
    if (!root) return 0;
    int hl = height(root->left);
    int hr = height(root->right);
    return 1 + (hl > hr ? hl : hr);
}

int main() {
    TreeNode* root = nullptr;
    for (int v : {1, 2, 3, 4, 5}) root = insert_bst(root, v);

    std::cout << "found 3: " << contains(root, 3) << '\n';
    std::cout << "height: " << height(root) << " (degenerate)\n";
}
```

After monotone inserts, height equals 5 and search walks every node. AVL and red-black trees rotate on balance violations, keep height O(log n), and restore logarithmic lookup.

## Pitfalls

- **Duplicates.** A "always right" policy worsens balance with many equal keys.
- **Delete without balance.** Even random inserts plus erases can skew the tree.
- **Confusion with heap.** BST and binary heap are different; ordering rules differ.
- **Recursion depth.** A skewed tree risks stack overflow on deep recursive walks.
- **std::map is not universal.** Logarithm loses to hash constants; pick by access pattern.
