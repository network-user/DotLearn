# Является ли std::map хеш-таблицей?

## Суть

Нет. `std::map` это упорядоченный ассоциативный контейнер на базе сбалансированного дерева поиска (обычно red-black tree). Операции O(log n), ключи итерируются по возрастанию. Хеш-таблица в стандарте: `std::unordered_map`.

## Что ответить на собесе

- Чётко отвечаю «нет» и называю внутреннюю структуру map.
- Сравниваю сложности: map O(log n) гарантированно, unordered_map O(1) среднее.
- Подчёркиваю уникальность ключей в `map` и отсутствие hash/equal requirements.
- Упоминаю `std::multimap` и `std::unordered_multimap` для дубликатов.
- Честно добавляю: для малого n разница мала; выбор по нужен ли sorted order и стабильность worst case.

## Как это работает

`std::map` хранит узлы дерева с указателями parent/left/right и цветом (в rb-tree). Вставка идёт по сравнению `Compare`, затем балансировка. Итератор обходит ключи в отсортированном порядке.

```cpp
#include <iostream>
#include <map>
#include <unordered_map>

int main() {
    std::map<int, std::string> ordered;
    ordered[3] = "gamma";
    ordered[1] = "alpha";
    ordered[2] = "beta";

    std::cout << "map order:\n";
    for (const auto& [k, v] : ordered) {
        std::cout << k << ' ' << v << '\n';
    }

    std::unordered_map<int, std::string> hashed;
    hashed[3] = "gamma";
    hashed[1] = "alpha";
    hashed[2] = "beta";

    std::cout << "unordered (no order guarantee):\n";
    for (const auto& [k, v] : hashed) {
        std::cout << k << ' ' << v << '\n';
    }
}
```

`map` всегда печатает 1,2,3; `unordered_map` может в любом порядке.

## Подводные камни

- **Путаница названий.** «Map» в Java/Python часто hash-based; в C++ `map` tree-based.
- **operator[] и default.** Оба контейнера вставляют при отсутствии ключа.
- **Строковые ключи.** `map` сравнивает лексикографически; `unordered_map` требует hash.
- **iterator invalidation.** `map` erase инвалидирует только удалённый; rehash у unordered ломает все итераторы.
- **Performance myth.** «unordered всегда быстрее» неверно при плохом hash или малом n.
