# What is hash table element access time?

## In short

Keyed access in a hash table (`find`, `at`, `count`) is constant on average: amortized O(1) at low load. In the worst case, when collisions cluster, time grows linearly to O(n) versus table size.

## How to answer in an interview

- I use the same frame as lookup/read: average O(1), worst O(n), plus rare rehash spikes on growth.
- I show idiomatic `std::unordered_map` access and the `find` vs `operator[]` split.
- I mention `std::unordered_map::bucket_count` and `load_factor` to spot degradation.
- I contrast with `std::map`: hash gives average constant access, the tree guarantees O(log n).
- I add honestly: the "constant" hides allocation and string hash cost; hot paths sometimes pick flat_hash_map or a sorted vector for small n.

## How it works

The client hashes the key, lands in a bucket, compares keys. The standard container hides rehash: as `size` grows, the table expands to keep chains short.

```cpp
#include <iostream>
#include <string>
#include <unordered_map>

int main() {
    std::unordered_map<std::string, int> stock;
    stock.reserve(32);

    stock.emplace("apple", 120);
    stock.emplace("banana", 95);
    stock["orange"] = 110;

    const std::string key = "banana";
    auto pos = stock.find(key);
    if (pos != stock.end()) {
        std::cout << key << ": " << pos->second << '\n';
    }

    try {
        std::cout << "orange via at: " << stock.at("orange") << '\n';
    } catch (const std::out_of_range&) {
        std::cout << "missing\n";
    }

    std::cout << "buckets=" << stock.bucket_count()
              << " load=" << stock.load_factor() << '\n';
}
```

With even hashing, access to `banana` stays in one bucket and a short string compare. If every key collides, `find` becomes a linear walk. Periodic rehash restores average constant time while the hash stays strong.

## Pitfalls

- **Access vs mutate confusion.** `operator[]` may insert and change `size` even when you "read".
- **Iterator after rehash.** Invalidated; references to elements usually stay valid.
- **Comparison with map.** Ordered traversal needs `map`; hash does not sort keys.
- **Latency stability.** Average O(1) does not cap tail latency; realtime paths watch worst case and chain length.
- **Small n.** For dozens of entries, `std::vector` plus sort can win on locality.
