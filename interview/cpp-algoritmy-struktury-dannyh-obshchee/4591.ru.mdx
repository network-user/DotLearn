# Когда поиск в BST уже не O(log n)?

## Суть

Бинарное дерево поиска даёт O(log n) только при умеренной высоте. Если вставки идут в отсортированном порядке или дерево иначе несбалансировано, оно вырождается в цепочку высоты n, и поиск становится O(n).

## Что ответить на собесе

- Определяю BST и условие логарифма: высота h ≈ log n при сбалансированности.
- Привожу классический пример: вставка 1,2,3,4,5 подряд даёт правое «хлыстовое» дерево.
- Называю решения: AVL, красно-черные деревья, `std::map`/`std::set` в STL.
- Упоминаю рандомизированный BST (treap) и bulk build через сортировку + balanced build.
- Честно добавляю: наивный BST в интервью допустим для объяснения, в продакшене без балансировки почти не используют.

## Как это работает

Поиск сравнивает ключ с узлом и уходит влево или вправо. Число шагов равно высоте. Сбалансированное дерево делит множество пополам на каждом уровне; вырожденное проходит все n узлов.

```cpp
#include <iostream>

struct TreeNode {
    int key;
    TreeNode* left = nullptr;
    TreeNode* right = nullptr;
    explicit TreeNode(int v) : key(v) {}
};

TreeNode* insert_bst(TreeNode* root, int value) {
    if (!root) return new TreeNode(value);
    if (value < root->key) root->left = insert_bst(root->left, value);
    else root->right = insert_bst(root->right, value);
    return root;
}

bool contains(TreeNode* root, int value) {
    while (root) {
        if (value == root->key) return true;
        root = (value < root->key) ? root->left : root->right;
    }
    return false;
}

int height(TreeNode* root) {
    if (!root) return 0;
    int hl = height(root->left);
    int hr = height(root->right);
    return 1 + (hl > hr ? hl : hr);
}

int main() {
    TreeNode* root = nullptr;
    for (int v : {1, 2, 3, 4, 5}) root = insert_bst(root, v);

    std::cout << "found 3: " << contains(root, 3) << '\n';
    std::cout << "height: " << height(root) << " (degenerate)\n";
}
```

После монотонных вставок высота равна 5, поиск идёт через все узлы. AVL и red-black деревья выполняют повороты при нарушении баланса, удерживая высоту O(log n) и возвращая логарифмический поиск.

## Подводные камни

- **Дубликаты.** Политика «всё вправо» ухудшает баланс при множестве равных ключей.
- **Удаление без баланса.** Даже после случайных вставок серия erase может перекосить дерево.
- **Смешение с heap.** BST и binary heap разные структуры; не путать условия упорядочения.
- **Рекурсия глубины.** Вырожденное дерево даёт stack overflow на глубоком рекурсивном обходе.
- **std::map не панацея.** Логарифм хуже константы hash; выбор структуры зависит от паттерна доступа.
