# Что такое коллизия в hash map?

## Суть

Коллизия возникает, когда разные ключи после хеширования попадают в один bucket (один индекс массива корзин). Это нормально: хеш сжимает большое пространство ключей в ограниченное число слотов. Контейнер хранит несколько записей в одной корзине и различает их сравнением ключей.

## Что ответить на собесе

- Даю определение: equal bucket index, not equal keys.
- Объясняю, что коллизии неизбежны по принципу pigeonhole при любом конечном числе bucket.
- Называю стратегии: chaining и open addressing.
- Показываю, как слабая hash (например, `length % size`) создаёт ложные «коллизии» по смыслу.
- Честно добавляю: цель не нулевые коллизии, а короткие цепочки и стабильный load factor.

## Как это работает

Хеш-функция переводит ключ в индекс. Если bucket занят другим ключом, новая запись добавляется в цепочку или ищется следующий слот. Поиск сначала выбирает bucket, затем сравнивает ключи по равенству.

```cpp
#include <iostream>
#include <list>
#include <string>
#include <vector>

class WordMap {
    std::vector<std::list<std::pair<std::string, int>>> slots;
    int modulo;

    int bucket(const std::string& word) const {
        return static_cast<int>(word.size() % modulo);
    }

public:
    explicit WordMap(int buckets) : slots(buckets), modulo(buckets) {}

    void put(const std::string& word, int score) {
        auto& chain = slots[bucket(word)];
        for (auto& item : chain) {
            if (item.first == word) {
                item.second = score;
                return;
            }
        }
        chain.emplace_back(word, score);
    }

    int get(const std::string& word) const {
        for (const auto& item : slots[bucket(word)]) {
            if (item.first == word) return item.second;
        }
        throw std::runtime_error("missing key");
    }
};

int main() {
    WordMap map(8);
    map.put("cat", 3);
    map.put("dog", 5);
    std::cout << map.get("cat") << '\n';
}
```

В учебном примере слова одинаковой длины коллидируют по bucket, но `get` всё равно находит правильный ключ через сравнение строк.

## Подводные камни

- **Путаница hash equality.** Совпадение hash или bucket не означает одинаковые ключи.
- **Слабая hash-функция.** Искусственно раздувает цепочки.
- **Игнор rehash.** Рост без расширения bucket_count усугубляет коллизии.
- **Счётчик коллизий.** Метрика без load factor мало говорит о perf.
- **Security.** Crafted keys могут целенаправленно насыпать один bucket (HashDoS).
