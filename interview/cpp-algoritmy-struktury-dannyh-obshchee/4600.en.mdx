# Is std::map a hash table?

## In short

No. `std::map` is an ordered associative container backed by a balanced search tree (usually a red-black tree). Operations are O(log n); keys iterate in ascending order. The standard hash table is `std::unordered_map`.

## How to answer in an interview

- I answer "no" clearly and name the map's internal structure.
- I compare complexities: map is guaranteed O(log n), unordered_map average O(1).
- I stress unique keys in `map` and no hash/equal requirements.
- I mention `std::multimap` and `std::unordered_multimap` for duplicates.
- I add honestly: for small n the gap is tiny; pick by sorted order needs and worst-case stability.

## How it works

`std::map` stores tree nodes with parent/left/right links and color (in rb-trees). Insert uses `Compare`, then rebalancing. Iterators walk keys in sorted order.

```cpp
#include <iostream>
#include <map>
#include <unordered_map>

int main() {
    std::map<int, std::string> ordered;
    ordered[3] = "gamma";
    ordered[1] = "alpha";
    ordered[2] = "beta";

    std::cout << "map order:\n";
    for (const auto& [k, v] : ordered) {
        std::cout << k << ' ' << v << '\n';
    }

    std::unordered_map<int, std::string> hashed;
    hashed[3] = "gamma";
    hashed[1] = "alpha";
    hashed[2] = "beta";

    std::cout << "unordered (no order guarantee):\n";
    for (const auto& [k, v] : hashed) {
        std::cout << k << ' ' << v << '\n';
    }
}
```

`map` always prints 1,2,3; `unordered_map` may print in any order.

## Pitfalls

- **Name confusion.** "Map" in Java/Python is often hash-based; C++ `map` is tree-based.
- **operator[] defaults.** Both insert on missing keys.
- **String keys.** `map` compares lexicographically; `unordered_map` needs a hash.
- **Iterator invalidation.** Map erase hits only the removed iterator; unordered rehash breaks all iterators.
- **Performance myth.** "unordered always wins" fails with a bad hash or small n.
