# Как запретить выделение объекта в куче?

## Суть

В C++ объект на стеке создаётся объявлением переменной, а в куче через `new`. Чтобы класс жил только на стеке, блокируют оператор `new`: делают его приватным или помечают `= delete`. Тогда внешний вызов `new MyType()` не скомпилируется, а обычное создание на стеке остаётся доступным.

## Что ответить на собесе

- Объясняю разницу: стек освобождается автоматически, куча требует явного `delete` или умного указателя.
- Называю два приёма: приватный `operator new` и удалённый `operator new(size_t) = delete` (C++11+).
- Уточняю, что при приватном `new` дружественный код или методы класса всё ещё могут аллоцировать.
- Показываю, что `delete` для `operator new[]` тоже стоит убрать, иначе обход через массив.
- Честно добавляю: полный запрет кучи не мешает placement new и не защищает от копирования в контейнер с кучевой аллокацией без дополнительных ограничений.

## Как это работает

Когда компилятор видит `new Widget()`, он ищет подходящий `operator new`. Если доступ закрыт, получаем ошибку компиляции ещё до линковки. Стековое `Widget w;` не вызывает `new`, поэтому ограничение не срабатывает.

```cpp
class StackOnly {
public:
    StackOnly() = default;
    ~StackOnly() = default;

    void* operator new(std::size_t) = delete;
    void* operator new[](std::size_t) = delete;
};

class LegacyStackOnly {
public:
    LegacyStackOnly() = default;
    ~LegacyStackOnly() = default;

private:
    static void* operator new(std::size_t sz) {
        return ::operator new(sz);
    }
    static void* operator new[](std::size_t sz) {
        return ::operator new[](sz);
    }
};

int main() {
    StackOnly ok;              // стек
    // StackOnly* bad = new StackOnly();  // ошибка компиляции
}
```

Паттерн применяют для лёгких RAII-обёрток, счётчиков с фиксированным временем жизни, объектов, привязанных к стековому контексту. В стандартной библиотеке похожие идеи встречаются в типах, где владение должно быть явным (`std::scoped_lock` и т.п.).

## Подводные камни

- **Массивная форма.** Без `operator new[] = delete` обходят запрет через `new T[n]`.
- **Placement new.** Запрет глобального `new` не отменяет размещающее выделение в заранее подготовленном буфере.
- **Приватный new.** Внутри класса и у друзей куча остаётся доступной; это не абсолютный барьер.
- **Копирование в vector.** `std::vector` может реаллоцировать элементы в куче; класс «только стек» в таком контейнере ведёт себя неочевидно.
- **Наследование.** Производный класс может объявить свой публичный `operator new`, если базовый не удалён окончательно.
