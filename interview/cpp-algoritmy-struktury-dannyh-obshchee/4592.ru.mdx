# Почему стек быстрее кучи?

## Суть

Стек выделяет память сдвигом указателя в предсказуемой LIFO-зоне: один инкремент при входе в scope, декремент при выходе. Куча ищет подходящий свободный блок, ведёт метаданные и может фрагментироваться, поэтому `new`/`delete` дороже и менее предсказуемы по latency.

## Что ответить на собесе

- Объясняю модель: стек привязан к кадру вызова, куча глобальна для процесса.
- Называю операции: bump pointer на стеке vs allocator search + bookkeeping в куче.
- Упоминаю cache locality: стековые объекты часто лежат рядом в адресном пространстве.
- Сравниваю ограничения: размер стека фиксирован (stack overflow), куча ограничена RAM.
- Честно добавляю: разница в константе; для больших или долгоживущих объектов куча неизбежна, а иногда pool allocator сглаживает разрыв.

## Как это работает

Локальная переменная резервирует место в stack frame текущей функции. При возврате кадр снимается, и вся память освобождается без явного `delete`. Динамический объект живёт, пока не вызовут `delete` или пока не сработает RAII умного указателя.

```cpp
void process_batch() {
    int buffer[64];           // стек, автоматическое освобождение
    // ...
} // buffer исчезает здесь

void process_large() {
    auto* heap_block = new int[10'000];  // куча, ручное управление
    // ...
    delete[] heap_block;
}
```

Аллокатор кучи поддерживает free-list, coalescing, иногда thread-local кэши. Поиск блока нужного размера и обновление структур делают путь длиннее, чем пара арифметических операций над stack pointer.

## Подводные камни

- **Большие массивы на стеке.** Риск переполнения стека и плохой locality при глубокой рекурсии.
- **Время жизни.** Указатель на стековый объект после выхода из функции даёт UB.
- **Фрагментация кучи.** Много мелких alloc/free ухудшают поиск блока.
- **False sense of speed.** Мелкие `new` в hot loop бьют по perf сильнее, чем кажется из теории.
- **Аллокаторы STL.** `std::vector` растёт в куче даже если элементы тривиальны.
