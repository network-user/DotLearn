# Why is the stack faster than the heap?

## In short

The stack allocates by bumping a pointer in a predictable LIFO region: one increment on scope entry, decrement on exit. The heap searches for a free block, keeps metadata, and may fragment, so `new`/`delete` cost more and have less stable latency.

## How to answer in an interview

- I explain the model: the stack belongs to a call frame, the heap is process-wide.
- I name the ops: bump pointer on the stack vs allocator search plus bookkeeping on the heap.
- I mention cache locality: stack objects often sit close in address space.
- I compare limits: stack size is fixed (stack overflow), heap is bounded by RAM.
- I add honestly: it is a constant-factor gap; large or long-lived objects need the heap, and pool allocators sometimes narrow the gap.

## How it works

A local variable reserves space in the current function's stack frame. On return the frame pops and all of it frees without `delete`. A dynamic object lives until `delete` or RAII via a smart pointer.

```cpp
void process_batch() {
    int buffer[64];           // stack, automatic release
    // ...
} // buffer ends here

void process_large() {
    auto* heap_block = new int[10'000];  // heap, manual lifetime
    // ...
    delete[] heap_block;
}
```

The heap allocator maintains free lists, coalescing, sometimes thread-local caches. Finding a suitably sized block and updating structures is longer than a couple of arithmetic ops on the stack pointer.

## Pitfalls

- **Large stack arrays.** Stack overflow risk and poor behavior with deep recursion.
- **Lifetime.** A pointer to a stack object after the function returns is UB.
- **Heap fragmentation.** Many small alloc/free pairs worsen block search.
- **False sense of speed.** Tiny `new` in a hot loop hurts more than theory suggests.
- **STL allocators.** `std::vector` grows on the heap even for trivial elements.
