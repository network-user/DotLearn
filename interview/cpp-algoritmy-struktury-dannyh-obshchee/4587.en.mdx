# What is hash table lookup complexity?

## In short

Key lookup in a hash table hashes first, then enters one bucket. With even spread and moderate load that is O(1) on average. If every key lands in one bucket or chains grow long, lookup degrades to O(n).

## How to answer in an interview

- I state "average O(1), worst O(n)" and tie it to chain length or probe count.
- I walk through hash → bucket → `key_equal` comparisons.
- I contrast chaining (list scan) and open addressing (slot probing).
- I mention `std::unordered_map::find` and that a miss still pays chain walk cost.
- I add honestly: the O(1) constant includes hashing and string compare; large keys matter.

## How it works

After the bucket index, the algorithm finds the node with the target key. Chaining walks a linked list in the bucket; open addressing follows probe steps until it hits the key or an empty tombstone.

```cpp
#include <iostream>
#include <list>
#include <string>
#include <vector>

class PhoneBook {
    static constexpr int kBuckets = 11;
    std::vector<std::list<std::pair<int, std::string>>> slots;

    int bucket_index(int key) const { return key % kBuckets; }

public:
    PhoneBook() : slots(kBuckets) {}

    void upsert(int ext, const std::string& name) {
        auto& chain = slots[bucket_index(ext)];
        for (auto& entry : chain) {
            if (entry.first == ext) {
                entry.second = name;
                return;
            }
        }
        chain.emplace_back(ext, name);
    }

    std::string lookup(int ext) const {
        const auto& chain = slots[bucket_index(ext)];
        for (const auto& entry : chain) {
            if (entry.first == ext) return entry.second;
        }
        return {};
    }
};

int main() {
    PhoneBook book;
    book.upsert(905, "Alex");
    book.upsert(201, "Mia");
    std::cout << book.lookup(201) << '\n';
    std::cout << (book.lookup(999).empty() ? "missing" : "found") << '\n';
}
```

With a good hash and load near 0.7, chains stay short and `lookup` does one compare. Under attack or bad `% table_size`, one bucket becomes length n and search turns linear.

## Pitfalls

- **Average only.** Skipping worst case sounds incomplete at senior level.
- **Equality after hash.** Same bucket does not mean the key matched.
- **Cache misses in open addressing.** Long clusters hurt cache more than short chains.
- **find vs operator[].** `operator[]` inserts by default; pure lookup needs `find`/`contains`.
- **Rehash during lookup.** Safe in single-threaded STL; a homemade table without care breaks iterators.
