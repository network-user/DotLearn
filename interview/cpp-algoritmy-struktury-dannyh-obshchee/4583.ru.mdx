# Как std::unordered_map разрешает коллизии?

## Суть

`std::unordered_map` строится на хеш-таблице с корзинами (buckets). Ключ хешируется, индекс корзины берётся из хеша. При коллизии несколько разных ключей попадают в одну корзину; libstdc++/libc++ хранят их цепочкой (chaining) внутри bucket, сравнивая ключи при поиске.

## Что ответить на собесе

- Описываю схему: hash → bucket index → цепочка узлов с парами ключ/значение.
- Уточняю, что равенство ключей проверяется после совпадения bucket, потому что хеш не уникален.
- Называю амортизированную O(1) при низкой загрузке и деградацию при длинных цепочках.
- Упоминаю rehash при росте `load_factor` и роль `max_load_factor()`.
- Честно добавляю: точная структура узлов зависит от реализации стандартной библиотеки; на собесе важна идея chaining, а не внутренний layout GCC vs MSVC.

## Как это работает

При `emplace` или `operator[]` контейнер вычисляет `std::hash<Key>{}(k)` и сводит значение к диапазону `[0, bucket_count)`. Если в корзине уже есть элементы с тем же bucket (включая разные ключи с совпадающим хешем), новый узел добавляется в цепочку. `find` снова хеширует ключ, заходит в корзину и линейно сравнивает ключи через `key_equal`.

```cpp
#include <iostream>
#include <string>
#include <unordered_map>

int main() {
    std::unordered_map<std::string, int> prices;
    prices.reserve(16);

    prices.emplace("apple", 120);
    prices["banana"] = 95;
    prices.insert({"orange", 110});

    const std::string query = "banana";
    auto it = prices.find(query);
    if (it != prices.end()) {
        std::cout << query << " -> " << it->second << '\n';
    }

    std::cout << "buckets: " << prices.bucket_count()
              << ", load: " << prices.load_factor() << '\n';
}
```

Если коэффициент загрузки превышает порог (часто около 1.0, настраивается), таблица увеличивает число корзин и перераспределяет элементы (rehash), чтобы цепочки оставались короткими. Пользователь может заранее вызвать `reserve(n)`, чтобы снизить число rehash при массовой вставке.

## Подводные камни

- **Плохая хеш-функция.** Слабый `std::hash` для своих типов даёт длинные цепочки даже при большом `bucket_count`.
- **Кастомный hash без equals.** Нужна согласованная пара `Hash` и `KeyEqual`, иначе ложные промахи или дубликаты.
- **Итераторы при rehash.** Rehash инвалидирует итераторы; ссылки на элементы обычно остаются валидными в C++11+ для `unordered_map`.
- **Указатели как ключи.** Хешируется адрес, а не содержимое; коллизии редки, но семантика сравнения часто не та, что ожидают.
- **Атаки на hash.** В сетевых сценариях используют защищённые хеши или `unordered_map` с seed, иначе возможен HashDoS через crafted keys.
