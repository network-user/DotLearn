# Где лежат элементы unordered_map при коллизии?

## Суть

Все элементы `std::unordered_map` живут в одной хеш-таблице: массив корзин плюс узлы с парами ключ/значение. При коллизии разные ключи с одинаковым bucket index связываются в одну цепочку внутри этой корзины, а не в отдельную таблицу.

## Что ответить на собесе

- Рисую модель: `bucket[i]` указывает на голову цепочки узлов `pair<const Key, T>`.
- Подчёркиваю, что коллизия по хешу не означает равенство ключей; сравнение идёт по `operator==`.
- Упоминаю, что узлы обычно аллоцируются из общего pool контейнера, а не копируются в сам массив корзин.
- Показываю API `bucket(key)`, `bucket_size(i)`, `bucket_count()` для диагностики распределения.
- Честно добавляю: внутреннее представление (forward list vs linked nodes) не стандартизовано; на практике смотрю на load factor, а не на байтовый layout.

## Как это работает

Хеш ключа даёт индекс корзины. Если корзина пуста, новый узел становится единственным. Если уже занята, контейнер вставляет узел в цепочку этой корзины. Итераторы обходят все узлы всех корзин, порядок произвольный.

```cpp
#include <iostream>
#include <unordered_map>

struct IdTag {
    int id;
    bool operator==(const IdTag& other) const { return id == other.id; }
};

struct IdHash {
    std::size_t operator()(IdTag t) const {
        return static_cast<std::size_t>(t.id % 4);
    }
};

int main() {
    std::unordered_map<IdTag, std::string, IdHash> registry;

    registry[{1}] = "alpha";
    registry[{5}] = "beta";   // тот же bucket, что у 1 (слабый hash)
    registry[{2}] = "gamma";

    for (const auto& [tag, name] : registry) {
        auto b = registry.bucket(tag);
        std::cout << "id " << tag.id << " -> " << name
                  << " (bucket " << b << ", chain " << registry.bucket_size(b) << ")\n";
    }
}
```

В примере искусственно слабая хеш-функция сталкивает ключи 1 и 5 в одну корзину: оба узла остаются в таблице, но разделяют bucket 1. При росте таблицы rehash меняет число корзин и пересчитывает индексы, цепочки укорачиваются.

## Подводные камни

- **Путаница bucket и hash.** Одинаковый bucket не гарантирует одинаковый полный хеш; проверка ключа обязательна.
- **Ложная демонстрация коллизии.** Соседние int-ключи в учебном примере не обязаны коллидировать с `std::hash<int>`.
- **Производительность обхода.** Длинная цепочка в одной корзине бьёт по `find` сильнее, чем равномерный разброс.
- **Нестабильный порядок итерации.** После rehash порядок обхода меняется; код не должен на него опираться.
- **Размер узла.** Узел хранит ключ, значение и служебные указатели; плотная упаковка хуже, чем у `std::map`, но доступ быстрее при хорошем hash.
