# What is hash table insert complexity?

## In short

With moderate load and a good hash, hash table insert is amortized O(1) on average: one hash, one bucket access, a short chain. In the worst case, when almost every key lands in one bucket or the table never rehashes, cost rises to O(n).

## How to answer in an interview

- I split average, amortized, and worst cases; interviews usually expect "O(1) average, O(n) worst".
- I explain rehash: rare O(n) rebuilds spread over many inserts (amortization).
- I contrast chaining and open addressing: on clash you either scan a chain or probe for a free slot.
- I mention `std::unordered_map::emplace` and hidden node allocation cost.
- I add honestly: without load control, "average O(1)" on paper does not save production latency.

## How it works

Insert hashes the key, maps to a bucket, then places the element or walks a chain / probe sequence. If load exceeds the threshold, the table grows and redistributes everything (rehash), making a single insert expensive but infrequent.

```cpp
#include <iostream>
#include <list>
#include <vector>

class IntSet {
    std::vector<std::list<int>> buckets;
    std::size_t count = 0;

    std::size_t slot(int value) const {
        return static_cast<std::size_t>(value) % buckets.size();
    }

public:
    explicit IntSet(std::size_t bucket_count) : buckets(bucket_count) {}

    void insert(int value) {
        auto& chain = buckets[slot(value)];
        for (int existing : chain) {
            if (existing == value) return;
        }
        chain.push_back(value);
        ++count;
    }

    bool contains(int value) const {
        for (int existing : buckets[slot(value)]) {
            if (existing == value) return true;
        }
        return false;
    }

    void dump() const {
        for (std::size_t i = 0; i < buckets.size(); ++i) {
            std::cout << i << ':';
            for (int v : buckets[i]) std::cout << ' ' << v;
            std::cout << '\n';
        }
    }
};

int main() {
    IntSet set(7);
    for (int v : {15, 22, 8, 35, 15}) set.insert(v);
    set.dump();
}
```

A second insert of 15 is caught by scanning a short chain. With poor key spread one bucket becomes length n and each insert turns O(n). Rehash with more buckets restores amortized constant time under uniform hashing.

## Pitfalls

- **O(1) only.** Omitting the worst case sounds incomplete.
- **Ignoring rehash.** Bulk insert without `reserve` triggers costly rebuild bursts.
- **Node allocation.** In `unordered_map`, `new` on a node can dominate hash work for tiny values.
- **Open-addressing clustering.** Primary and secondary clustering hurt insert even at moderate load.
- **Adaptive attacks.** Crafted keys force worst case; use hardened hashes or a random seed.
