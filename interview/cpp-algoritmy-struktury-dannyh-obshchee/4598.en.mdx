# What is heap fragmentation?

## In short

Heap fragmentation splits free memory into many small non-adjacent blocks. Total free bytes may suffice for a large request, but no contiguous block exists. Internal fragmentation wastes space inside an allocated block; external fragmentation leaves holes between busy regions.

## How to answer in an interview

- I define external vs internal fragmentation.
- I explain the pattern: mixed-size allocs, partial frees, then a large `new` fails or maps fresh memory.
- I name mitigations: pool allocators, slabs, tcmalloc/jemalloc, compacting GC (rare in C++).
- I tie it to latency spikes and RSS that does not return to the OS.
- I add honestly: modern allocators soften the issue; embedded systems feel it more.

## How it works

The allocator hands out blocks from arenas; `delete` returns them to a free list. Adjacent free chunks may coalesce, but alternating sizes leave a Swiss-cheese layout.

```cpp
#include <vector>

void fragment_demo() {
    std::vector<int*> chunks;
    for (int i = 0; i < 8; ++i) {
        chunks.push_back(new int[16]);
    }
    delete[] chunks[2];
    delete[] chunks[5];
    chunks[2] = nullptr;
    chunks[5] = nullptr;
    // many small holes; a large request may miss a contiguous block
    int* big = new int[200];  // may come from a fresh arena
    delete[] big;
    for (auto* p : chunks) delete[] p;
}
```

Fixed-size pools remove external fragmentation inside the pool.

## Pitfalls

- **Counting free bytes only.** Contiguity matters, not just totals.
- **Mixing long-lived and short-lived.** Classic fragmentation recipe.
- **Returning memory to the OS.** glibc often keeps arenas; RSS stays high after free.
- **Alignment overhead.** Internal waste at 16/32 byte boundaries.
- **Over-pooling.** Too many pools raise internal waste.
