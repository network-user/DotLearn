# Какова сложность поиска в хеш-таблице?

## Суть

Поиск по ключу в хеш-таблице сначала вычисляет хеш и заходит в одну корзину. При равномерном распределении и умеренной загрузке это O(1) в среднем. Если все ключи сходятся в одну корзину или цепочка длинная, поиск деградирует до O(n).

## Что ответить на собесе

- Формулирую: «среднее O(1), худшее O(n)» и связываю с длиной цепочки или числом проб.
- Объясняю шаги: hash → bucket → сравнение ключей по `key_equal`.
- Сравниваю chaining (скан списка) и open addressing (пробирование ячеек).
- Упоминаю `std::unordered_map::find` и что отсутствие элемента тоже платит за проход цепочки.
- Честно добавляю: константа в O(1) включает вызов хеша и сравнение строк; для больших ключей это заметно.

## Как это работает

После вычисления индекса корзины алгоритм ищет узел с нужным ключом. При chaining обходят связный список в bucket; при open addressing идут по слотам по формуле пробирования, пока не найдут ключ или пустую метку.

```cpp
#include <iostream>
#include <list>
#include <string>
#include <vector>

class PhoneBook {
    static constexpr int kBuckets = 11;
    std::vector<std::list<std::pair<int, std::string>>> slots;

    int bucket_index(int key) const { return key % kBuckets; }

public:
    PhoneBook() : slots(kBuckets) {}

    void upsert(int ext, const std::string& name) {
        auto& chain = slots[bucket_index(ext)];
        for (auto& entry : chain) {
            if (entry.first == ext) {
                entry.second = name;
                return;
            }
        }
        chain.emplace_back(ext, name);
    }

    std::string lookup(int ext) const {
        const auto& chain = slots[bucket_index(ext)];
        for (const auto& entry : chain) {
            if (entry.first == ext) return entry.second;
        }
        return {};
    }
};

int main() {
    PhoneBook book;
    book.upsert(905, "Alex");
    book.upsert(201, "Mia");
    std::cout << book.lookup(201) << '\n';
    std::cout << (book.lookup(999).empty() ? "missing" : "found") << '\n';
}
```

При хорошем hash и load factor около 0.7 цепочки короткие, и `lookup` делает единичное сравнение. При атаке или плохом `% table_size` одна корзина превращается в список из n элементов, и поиск становится линейным.

## Подводные камни

- **Только средний случай.** Без worst case ответ неполный для senior уровня.
- **Равенство после хеша.** Совпадение bucket не означает найденный ключ.
- **Кэш-промахи в open addressing.** Длинные кластеры бьют по CPU cache сильнее, чем короткие цепочки.
- **find vs operator[].** `operator[]` вставляет по умолчанию; для чистого поиска нужен `find`/`contains`.
- **Rehash во время поиска.** В однопоточном STL безопасно; при своей таблице без синхронизации итераторы ломаются.
