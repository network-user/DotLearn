# Information register subordination to recorder: code check

## In short

An information register metadata property sets whether a record must link to a document. A subordinate register is written only as part of posting; an independent one allows write without a recorder with a full dimension set.

## How to answer in an interview

- I check `Метаданные.РегистрыСведений.<Имя>.РежимЗаписи` and the subordination flag.
- For a subordinate set, `Регистратор` filter is mandatory on read and write.
- In an independent register the key is dimensions plus period (for periodic).
- Writing without a recorder in a subordinate register causes errors or orphan data.
- I am honest: subordination cannot change at runtime; only in the configurator with data migration.

## How it works

```bsl
МетаРегистра = Метаданные.РегистрыСведений.СтатусыЗаказов;
ПодчиненРегистратору = (МетаРегистра.РежимЗаписи = Метаданные.СвойстваОбъектов.РежимЗаписиРегистраСведений.ПодчинениеРегистратору);

Если ПодчиненРегистратору Тогда
    Набор = РегистрыСведений.СтатусыЗаказов.СоздатьНаборЗаписей();
    Набор.Отбор.Регистратор.Установить(ДокументСсылка);
    Набор.Прочитать();
КонецЕсли;
```

On posting the document builds movements in the subordinate register; on unpost recorder rows are removed automatically.

## Pitfalls

**Independent set write to subordinate register.** Missing recorder; reports cannot find the source.
**Read without recorder filter.** Pulls other documents' rows.
**Mode change in production.** Needs export and historical data conversion.
**Periodic subordinate.** Confusion between record period and document date.
**Direct SQL.** Bypassing the recorder mechanism breaks unpost.
