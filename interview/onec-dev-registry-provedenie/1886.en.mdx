# Register virtual tables in the 1C query language

## In short

A virtual table is not a DBMS temp table but a named register slice in a query: `.Остатки()`, `.Обороты()`, `.СрезПоследних()`, and similar. The platform expands them to SQL using totals and period.

## How to answer in an interview

- I do not confuse a register virtual table with a `ПОМЕСТИТЬ` query temp table.
- For balance as of date I use `.Остатки(&Момент, )`; for movements over a period `.Обороты(&Нач, &Кон)`.
- For information registers `.СрезПоследних(&Дата, )` instead of manual max period search.
- Parameters in parentheses set period and dimension filter.
- I am honest: a virtual table without fresh totals on a large register can be slow.

## How it works

```bsl
"ВЫБРАТЬ
|    Цены.Номенклатура,
|    Цены.Цена
|ИЗ
|    РегистрСведений.ЦеныНоменклатуры.СрезПоследних(&Дата, ) КАК Цены"
```

```bsl
"ВЫБРАТЬ
|    Ост.Склад,
|    Ост.КоличествоОстаток
|ИЗ
|    РегистрНакопления.ТоварыНаСкладах.Остатки(&Граница, ) КАК Ост"
```

A temp table via `ПОМЕСТИТЬ` lives only in the query batch and holds an intermediate result.

## Pitfalls

**Confusion with ПОМЕСТИТЬ.** Different mechanisms and scope.
**Slice last without date.** Wrong freshness in reports.
**Balances without КонецДня.** Mid-day slice instead of end of day.
**Ignoring filter in parentheses.** Full register scan.
**Manual aggregation instead of virtual table.** Duplicates platform logic and errors.
