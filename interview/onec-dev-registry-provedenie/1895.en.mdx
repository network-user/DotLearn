# When to use register virtual tables

## In short

Virtual tables are for standard slices: balances, turnovers, slice last. Direct reads of the movements table are only for special admin or debug scenarios.

## How to answer in an interview

- I always start reports with `.Остатки` / `.Обороты` / `.СрезПоследних`.
- Direct `РегистрНакопления.Имя` without a virtual table returns raw movements, not balance.
- A temp table comes after the virtual table when further joins are complex.
- Period parameters in parentheses are required for performance.
- I am honest: on a tiny register the difference is small; in a standard ERP virtual tables are critical.

## How it works

Balance as of date:

```bsl
"РегистрНакопления.Товары.Остатки(&Момент, Склад = &Склад)"
```

Price as of date:

```bsl
"РегистрСведений.Цены.СрезПоследних(&Дата, Номенклатура = &Ном)"
```

## Pitfalls

**Raw movements instead of balance.** Wrong report numbers.
**Virtual table without filter.** Full scan.
**Double aggregation.** Balances are already aggregated; summing again distorts.
**Stale totals.** Slow or wrong on large data.
**Confusion with ПОМЕСТИТЬ.** Different abstraction levels.
