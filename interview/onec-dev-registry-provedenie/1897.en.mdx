# How to tell a register is overloaded

## In short

Overload signs: slow `.Остатки()` and `.Обороты()`, growing movements table, long locks on posting, report timeouts. Diagnose via tech log, DBMS query plans, and comparison to a baseline period.

## How to answer in an interview

- I check query duration to that register virtual tables.
- I count movement volume and dimensions: each dimension multiplies cardinality.
- I verify totals freshness and filters in reports.
- I look for posting without warehouse/company filter on bulk documents.
- I am honest: fixing with movement archival is a project, not a single index.

## How it works

In the tech log I enable `SDBL` and `EXCP` for the report session. On SQL Server I review top waits on `_AccumRg` tables. Metrics: movements per month, average single-document post time, totals hit ratio.

## Pitfalls

**Tuning without period filter.** Bottleneck remains.
**Extra metadata dimensions.** Cannot fix by query alone.
**Totals disabled on a large register.** Every report full scan.
**DBMS index without 1C alignment.** Platform support risk.
**Slow disk confusion.** CPU wait masks register issue.
