# Period parameter in an information register virtual table

## In short

In `.СрезПоследних(&Период, )` and `.СрезПервых()` the period parameter sets the freshness boundary. Allowed values are date, `КонецДня()`, `НачалоМесяца()`, and a point-in-time boundary for consistency with a document.

## How to answer in an interview

- I pass a slice date, not an arbitrary string.
- For end-of-day reports I use `КонецДня(Дата)`.
- The second virtual table parameter filters dimensions.
- A non-periodic register has no date-based `.СрезПоследних()`.
- I am honest: slice without fresh totals on a large register is slow.

## How it works

```bsl
Запрос.УстановитьПараметр("Период", КонецДня(ДатаДокумента));
"РегистрСведений.СтавкиНДС.СрезПоследних(&Период, )"
```

The platform picks the latest row with period not after the parameter for each dimension set.

## Pitfalls

**Date without time.** Slice at day start instead of end.
**Non-periodic register.** Date slice call is wrong.
**Empty filter in parentheses.** Extra data volume.
**Ignoring session time zone.** Mismatch on night documents.
**Manual MAX(Период) instead of slice.** Duplication and grouping errors.
