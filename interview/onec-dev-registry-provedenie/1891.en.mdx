# Virtual and temporary tables in a 1C query

## In short

A virtual table is a built-in register slice (`.Остатки`, `.СрезПоследних`). A temp table is created with `ПОМЕСТИТЬ` and lives only in the query batch for intermediate results.

## How to answer in an interview

- I do not call `ПОМЕСТИТЬ` a register virtual table.
- I do not create a virtual table manually; its name is fixed by register metadata.
- I use `ПОМЕСТИТЬ` for heavy joins and staged aggregation.
- I index temp tables in a follow-up query when needed.
- I am honest: extra `ПОМЕСТИТЬ` without benefit complicates the query plan.

## How it works

Virtual:

```bsl
"ИЗ РегистрНакопления.Остатки.Остатки(&Дата, ) КАК Ост"
```

Temporary:

```bsl
"ВЫБРАТЬ ... ПОМЕСТИТЬ ВТОтбор
|ИЗ ...;
|ВЫБРАТЬ ... ИЗ ВТОтбор КАК ВТ"
```

## Pitfalls

**Aliases across unrelated batch queries.** TT not visible without a temp table manager.
**Huge TT without filter.** tempdb pressure.
**Term confusion in interviews.** Wrong optimization mental model.
**Duplicating register slice in TT.** Two paths with different freshness.
**Missing INDEX on TT.** Slow join on the next step.
