# Физические таблицы регистра накопления: остатки и обороты

## Суть

Регистр накопления вида «Остатки» хранит движения в таблице записей и агрегированные итоги в отдельной таблице итогов (две физические). Регистр вида «Обороты» хранит только движения в одной таблице; виртуальные таблицы «Обороты» и «Остатки» в языке запросов не равны отдельным физическим таблицам для каждого вида.

## Что ответить на собесе

- Я различаю вид регистра «Остатки» и «Обороты» при ответе про число таблиц.
- Для остатков: таблица движений плюс таблица итогов; итоги ускоряют `Остатки()` в запросе.
- Для оборотов: одна таблица движений, остатки не накапливаются платформой.
- Виртуальная таблица `.Обороты(,,)` строится из движений; `.Остатки()` для регистра остатков опирается на итоги.
- Честно: путать виртуальные таблицы запроса с физическими таблицами СУБД на собесе частая ошибка.

## Как это работает

Запрос к регистру остатков с виртуальными таблицами:

```bsl
Запрос = Новый Запрос;
Запрос.Текст =
    "ВЫБРАТЬ
    |    Ост.Номенклатура,
    |    Ост.КоличествоОстаток,
    |    Обор.КоличествоПриход,
    |    Обор.КоличествоРасход
    |ИЗ
    |    РегистрНакопления.ТоварыНаСкладах.Остатки(&Граница, ) КАК Ост
    |        ЛЕВОЕ СОЕДИНЕНИЕ РегистрНакопления.ТоварыНаСкладах.Обороты(&Начало, &Конец) КАК Обор
    |        ПО Ост.Номенклатура = Обор.Номенклатура";
```

При проведении документ пишет набор записей в таблицу движений; для регистра остатков платформа обновляет таблицу итогов. Режим «контроль остатков» и актуальность итогов влияют на момент видимости остатка в параллельных сеансах.

## Подводные камни

**Один ответ для всех видов.** Регистр оборотов не имеет таблицы итогов остатков.
**Прямой SQL к итогам.** Имена служебных таблиц не переносимы между СУБД и версиями.
**Отключённые итоги.** Запросы `Остатки()` без пересчёта итогов становятся тяжёлыми.
**Срез последних вместо остатков.** Разные механизмы; нельзя подставить одно вместо другого.
**Проведение без движений.** Пустой набор записей не меняет ни движений, ни итогов.
