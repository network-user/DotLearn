# Posting chain: movements, totals, and locks

## In short

Document posting in a transaction builds register movements, updates balance register totals, and uses managed locks on dimensions so parallel sessions do not corrupt stock.

## How to answer in an interview

- I describe the order: `ПередЗаписью` → document write → `ОбработкаПроведения` → movements → totals.
- I set data locks before reading the balance I will change.
- Balance control relies on fresh totals and lock mode.
- Unpost removes movements and rolls back totals in the same logic.
- I am honest: a long posting transaction increases mutual locks during bulk import.

## How it works

```bsl
Процедура ОбработкаПроведения(Отказ, РежимПроведения)
    Блокировка = Новый БлокировкаДанных;
    Элемент = Блокировка.Добавить("РегистрНакопления.ТоварыНаСкладах");
    Элемент.Режим = РежимБлокировкиДанных.Исключительный;
    Элемент.УстановитьЗначение("Склад", Склад);
    Блокировка.Заблокировать();

    Движения.ТоварыНаСкладах.Записывать = Истина;
    // build receipt/issue
КонецПроцедуры
```

The platform writes movements and recalculates affected totals within the posting transaction.

## Pitfalls

**Balance read without lock.** Two issues pass with one stock level.
**Posting without movements.** Accounting unchanged while user thinks it posted.
**Ignoring failure in ОбработкаПроведения.** Document marked posted with logic error.
**Bulk posting at one instant.** Lock queue on popular warehouses.
**Manual totals edit.** Mismatch with movements until full rebuild.
