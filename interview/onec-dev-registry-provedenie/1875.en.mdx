# Calculation registers: periods, bases, and dependent accruals

## In short

Calculation registers store results of periodic computations (payroll, leave, deductions) with base period, action period, and recorder binding. The platform supports recalculation, displacement, and dependencies between calculation kinds.

## How to answer in an interview

- I explain: a calculation register does not replace an accumulation register; it is a separate mechanism for HR and payroll algorithms.
- Movements are written by documents such as Payroll accrual on posting.
- I account for base period, action period, and actual registration period.
- Links between calculation kinds are defined in the chart of calculation types (base, leading, displacing).
- I am honest: without understanding recalculation you easily get duplicate accruals for one month.

## How it works

Building a record via a calculation register set:

```bsl
Набор = РегистрыРасчета.Начисления.СоздатьНаборЗаписей();
Набор.Отбор.Регистратор.Установить(ДокументСсылка);

Запись = Набор.Добавить();
Запись.ПериодДействияНачало = НачалоМесяца(Период);
Запись.ПериодДействияКонец = КонецМесяца(Период);
Запись.ВидРасчета = ПланыВидовРасчета.Начисления.Оклад;
Запись.Сотрудник = СотрудникСсылка;
Запись.Результат = СуммаОклада;

Набор.Записать();
```

The chart of calculation types defines calculation order and dependencies. When source data for a past period changes, recalculation reverses old records and adds new ones.

## Pitfalls

**Confusion with accumulation register.** Different metadata objects and physical model.
**Ignoring action period.** Accrual "spills" beyond the month in reports.
**Manual movement edit.** Breaks recorder link and recalculation.
**Wrong calculation kind order.** Percentages run on an unclosed base.
**No recalculation.** Backdated salary change is not reflected without a service document.
