# params у fixture vs parametrize

## Суть

`@pytest.fixture(params=[...])` размножает все тесты, зависящие от fixture, по каждому значению `request.param`. `@pytest.mark.parametrize` размножает конкретную тестовую функцию по таблице аргументов. Fixture params удобен, когда дорогой setup один, а данные меняются; parametrize когда вариации только в assert одного теста.

## Что ответить на собесе

- `params` на `browser` даст три прогона каждого UI теста, который просит `browser`.
- `parametrize` на `test_login(role)` не трогает соседние тесты в файле.
- Комбинация: parametrize + fixture params даёт декартово произведение nodes.
- `request.param` в fixture, не в тесте напрямую.
- Честно: params на session fixture взрывает время suite, лучше function scope.

## Как это работает

Fixture params:

```python
import pytest


@pytest.fixture(params=["chromium", "firefox"])
def browser(request):
    driver = launch(request.param)
    yield driver
    driver.quit()


def test_home(browser):
    browser.get("/")
```

Parametrize на тесте:

```python
@pytest.mark.parametrize("role", ["admin", "guest"])
def test_menu(role):
    assert menu_visible(role)
```

## Подводные камни

**Декартово произведение.** params fixture × parametrize быстро раздувает CI.

**ids по умолчанию.** Отчёт показывает `[chromium]` без человекочитаемого id.

**params + autouse.** Autouse fixture с params запускает setup для всех комбинаций молча.

**Один тест, много браузеров.** Забытый quit в fixture params утекает процессы.

**Дублирование таблицы.** Те же cases в params и parametrize расходятся со временем.
