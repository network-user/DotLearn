# Дебаг падающих тестов

## Суть

Дебаг pytest теста: воспроизвести минимальный case, включить verbose/traceback, остановиться на assert через `pytest.set_trace()` или IDE breakpoint, снять stdout (`-s`), caplog и артефакты (screenshot, HAR). Цель сузить до одного неверного assumption, не перезапускать весь suite вслепую.

## Что ответить на собесе

- `pytest tests/test_x.py::test_y -vv --tb=long -s` локально.
- `--pdb` или `breakpoint()` на падении.
- `--lf` гоняю только last failed после фикса.
- Сравниваю expected/actual из introspection, не переписываю assert сразу.
- Честно: дебаг на shared staging без изоляции данных даёт ложные выводы.

## Как это работает

```bash
pytest tests/test_checkout.py::test_total -vv --tb=long -s
pytest --pdb tests/test_checkout.py::test_total
```

В тесте:

```python
def test_total(cart):
    total = cart.compute()
    if total != 99:
        pytest.set_trace()
    assert total == 99
```

Логи:

```python
def test_flow(caplog):
    with caplog.at_level("DEBUG"):
        run_flow()
    assert "paid" in caplog.text
```

## Подводные камни

**-s в CI всегда.** Захламляет лог, теряется сигнал.

**pdb в headless.** Нужен tty или remote debugger.

**Flaky не воспроизвёлся один раз.** Нужен stress `--count`.

**Правка assert под факт.** Маскирует продуктовый баг.

**Кэш .pytest_cache.** Старый --lf список после rename теста.
