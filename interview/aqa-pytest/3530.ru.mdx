# Как PyTest находит тесты

## Суть

PyTest собирает тесты по naming convention, структуре каталогов и аргументам CLI. По умолчанию ищет файлы `test_*.py` / `*_test.py`, внутри функции и методы `test_*`, плюс классы `Test*` без `__init__`. Маркеры, `pytest.ini` и node id (`file::test`) сужают или расширяют набор запуска.

## Что ответить на собесе

- `pytest tests/api` ограничивает корень collection.
- `pytest tests/ui/test_login.py::test_guest` запускает один node.
- `-k "checkout and not slow"` фильтрует по substring имени.
- `-m smoke` требует зарегистрированный marker в ini.
- Честно: файл `checks_login.py` без префикса `test_` не подхватится без `python_files` в ini.

## Как это работает

Файл с двумя тестами и ложным кандидатом:

```python
def test_addition():
    assert 1 + 1 == 2


def test_subtraction():
    assert 4 - 2 == 2


def helper_only():
    assert True
```

Запуск:

```bash
pytest tests/math_ops.py
pytest tests/math_ops.py::test_addition
pytest -m smoke
```

## Подводные камни

**Класс Test с __init__.** Collection пропустит методы как несовместимые.

**Дубли import path.** Два модуля с одинаковым именем дают phantom duplicates.

**package без __init__.** В старых layout влияет на import mode (`importlib` vs prepend).

**collect ignore.** `collect_ignore` в conftest скрывает файл молча.

**parametrize id.** Один тест превращается в N nodes, отчёт Allure плодит кейсы.
