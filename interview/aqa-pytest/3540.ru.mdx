# Проверка логов через caplog

## Суть

Fixture `caplog` перехватывает записи `logging` во время теста. Через `caplog.records`, `caplog.text` и context `caplog.at_level` проверяю сообщение, level и logger name. Это контракт на observability: тест падает, если prod код перестал логировать ошибку.

## Что ответить на собесе

- `with caplog.at_level(logging.ERROR):` сужает захват.
- `assert "payment failed" in caplog.text` простая smoke проверка.
- `caplog.records[0].levelname == "WARNING"` точнее по структуре.
- Настраиваю `log_cli` в ini для локального debug, caplog для assert.
- Честно: root logger handler вне теста может дублировать записи, фильтрую по logger name.

## Как это работает

```python
import logging


logger = logging.getLogger("billing")


def charge():
    logger.warning("card declined")


def test_charge_logs(caplog):
    with caplog.at_level(logging.WARNING, logger="billing"):
        charge()
    assert "card declined" in caplog.text
    assert caplog.records[0].levelno == logging.WARNING
```

## Подводные камни

**Propagate=False.** Дочерний logger без propagate не попадёт в caplog root.

**Уровень ниже порога.** INFO не виден при caplog на WARNING без at_level.

**Форматтер меняет text.** Сравниваю message, не весь formatted line.

**caplog.clear().** Забытый clear между вызовами смешивает records.

**structlog.** Нужен bridge в stdlib logging, иначе caplog пуст.
