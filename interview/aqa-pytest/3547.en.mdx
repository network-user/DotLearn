# yield vs return in fixtures

## In short

`return` in a fixture gives the test a value and ends the fixture: no teardown code after return (except finalizers via `request.addfinalizer`). `yield` pauses the fixture, supplies the value, then runs code after yield as teardown. For browsers, files, and connections I almost always use `yield`.

## How to answer in an interview

- `return data` for pure immutable factories without cleanup.
- `yield driver` plus `driver.quit()` after yield.
- After `return`, following lines are unreachable; teardown will not run.
- `yield None` is valid for setup/teardown only with no value.
- Honestly: mixing `return` and code below in one fixture is a design mistake.

## How it works

return for data only:

```python
import pytest


@pytest.fixture
def payload():
    return {"role": "qa"}
```

yield with teardown:

```python
@pytest.fixture
def log_file(tmp_path):
    path = tmp_path / "run.log"
    handle = path.open("w", encoding="utf-8")
    yield handle
    handle.close()
```

## Pitfalls

**Exception before yield.** Teardown after yield does not run; resource leak.

**Exception in teardown.** Masks the test's original failure.

**Generator fixture session scope.** A long-lived yield keeps a socket open.

**return None explicitly.** The test gets None when it expected an object.

**Two yields.** A generator fixture with two yields breaks lifecycle.
