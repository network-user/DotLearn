# Ways to attach fixtures

## In short

Attach a fixture three ways: a test function parameter (explicit dependency), `autouse=True` (automatic for the scope), or `@pytest.mark.usefixtures` (side-effect setup without a value in the test). The choice depends on whether the fixture return is needed in the test body or only setup side effects.

## How to answer in an interview

- Parameter `def test_x(db):` when I use `db` in asserts.
- `autouse=True` for logging or env reset on every test.
- `@pytest.mark.usefixtures("seed_db")` when the test does not read the return, only needs data.
- `usefixtures` on a class covers all methods.
- Honestly: autouse on a session fixture with xdist gives unexpected shared state without a lock.

## How it works

Three styles:

```python
import pytest


@pytest.fixture
def token():
    return "secret"


@pytest.fixture(autouse=True)
def trace():
    print("start")


@pytest.mark.usefixtures("token")
def test_health():
    assert True


def test_auth(token):
    assert token == "secret"
```

## Pitfalls

**usefixtures without a signature name.** You cannot access the return inside the test.

**autouse + heavy setup.** Every test pays the cost, even a unit test without UI.

**Fixture order.** Dependencies via parameters, not import order.

**Class usefixtures + unittest.** Inheriting unittest.TestCase breaks discovery.

**String in usefixtures.** A typo in the fixture name is caught only at run time.
