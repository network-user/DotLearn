# Декоратор parametrize

## Суть

`@pytest.mark.parametrize` объявляет имена аргументов и таблицу значений; PyTest создаёт отдельный test item на каждую строку. Это data-driven без копипасты функций. Можно стекировать несколько parametrize или комбинировать с indirect для косвенной передачи в fixture.

## Что ответить на собесе

- `@pytest.mark.parametrize("a,b,expected", [(1,1,2), (2,3,5)])` даёт два node.
- `ids=["add","more"]` улучшает читаемость в CI логе.
- `pytest.param(0, marks=pytest.mark.xfail)` помечает отдельную строку.
- `indirect=True` передаёт значение в fixture с тем же именем.
- Честно: гигантская таблица в декораторе хуже читается, чем `pytest_generate_tests`.

## Как это работает

Проверка чётности:

```python
import pytest


def is_even(value: int) -> bool:
    return value % 2 == 0


@pytest.mark.parametrize(
    "value,expected",
    [(2, True), (3, False), (0, True)],
    ids=["two", "three", "zero"],
)
def test_is_even(value, expected):
    assert is_even(value) is expected
```

## Подводные камни

**Несовпадение arity.** Лишний столбец в tuple ломает collection.

**Порядок стека parametrize.** Внешний декоратор меняет порядок итерации.

**Mutable default в ids.** Не использовать list как shared mutable.

**parametrize на класс.** Нужен pytest 6+ стиль, иначе путаница с unittest.

**Отсутствие expected.** Только input без oracle превращает тест в smoke без assert смысла.
