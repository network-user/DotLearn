# Fighting flaky tests in PyTest

## In short

A flaky test is unstable with unchanged code: timing, shared state, external services. The right path is root cause analysis, data isolation, explicit waits, and a deterministic clock. Rerun (`pytest-rerunfailures`) is acceptable as a temporary cushion for external outages, not to hide bugs in the test.

## How to answer in an interview

- First I reproduce locally with `--count=20` or random order.
- I remove sleep and use poll until condition (tenacity, playwright expect).
- Unique user/email per test, not a shared DB row.
- `@pytest.mark.flaky(reruns=2)` only with a ticket and a fix deadline.
- Honestly: rerun in CI without quarantine turns a red pipeline into "maybe green".

## How it works

Isolated temp resource:

```python
import pytest
import tempfile
import os


@pytest.fixture
def temp_path():
    fd, path = tempfile.mkstemp()
    os.close(fd)
    yield path
    os.remove(path)


def test_write(temp_path):
    with open(temp_path, "w", encoding="utf-8") as handle:
        handle.write("ok")
```

Limited rerun:

```python
import pytest


@pytest.mark.flaky(reruns=2, reruns_delay=1)
def test_external_ping():
    assert ping_gateway()
```

## Pitfalls

**reruns without root cause.** Tech debt grows; trust in the suite drops.

**Parallel + shared file.** Classic order-dependent flake.

**Unstable assert on timestamp.** now() without a frozen clock.

**Network without mock.** External API outage fails code you did not change.

**flaky marker on whole module.** Hides neighboring stable tests in the report.
