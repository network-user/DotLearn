# yield vs return в fixture

## Суть

`return` в fixture отдаёт значение тесту и завершает fixture: teardown кода после return нет (кроме finalizer через `request.addfinalizer`). `yield` приостанавливает fixture, отдаёт значение, затем выполняет код после yield как teardown. Для браузера, файлов и подключений почти всегда `yield`.

## Что ответить на собесе

- `return data` для чистых immutable factory без cleanup.
- `yield driver` + `driver.quit()` после yield.
- После `return` строки недостижимы, teardown не запустится.
- `yield None` валиден для чистого setup/teardown без значения.
- Честно: смешивать `return` и код ниже в одной fixture ошибка проектирования.

## Как это работает

return только данные:

```python
import pytest


@pytest.fixture
def payload():
    return {"role": "qa"}
```

yield с teardown:

```python
@pytest.fixture
def log_file(tmp_path):
    path = tmp_path / "run.log"
    handle = path.open("w", encoding="utf-8")
    yield handle
    handle.close()
```

## Подводные камни

**Исключение до yield.** Teardown после yield не выполнится, resource leak.

**Исключение в teardown.** Маскирует исходный fail теста.

**Generator fixture scope session.** Долгоживущий yield держит socket открытым.

**return None явно.** Тест получает None, ожидал object.

**Два yield.** Generator fixture с двумя yield ломает lifecycle.
