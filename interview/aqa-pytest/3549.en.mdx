# Debugging failing tests

## In short

Debugging a pytest test: reproduce a minimal case, enable verbose/traceback, stop on assert via `pytest.set_trace()` or an IDE breakpoint, capture stdout (`-s`), caplog, and artifacts (screenshot, HAR). The goal is to narrow to one wrong assumption, not rerun the whole suite blindly.

## How to answer in an interview

- `pytest tests/test_x.py::test_y -vv --tb=long -s` locally.
- `--pdb` or `breakpoint()` on failure.
- `--lf` runs only last failed after a fix.
- I compare expected/actual from introspection before rewriting the assert.
- Honestly: debugging on shared staging without data isolation yields false conclusions.

## How it works

```bash
pytest tests/test_checkout.py::test_total -vv --tb=long -s
pytest --pdb tests/test_checkout.py::test_total
```

In the test:

```python
def test_total(cart):
    total = cart.compute()
    if total != 99:
        pytest.set_trace()
    assert total == 99
```

Logs:

```python
def test_flow(caplog):
    with caplog.at_level("DEBUG"):
        run_flow()
    assert "paid" in caplog.text
```

## Pitfalls

**-s always in CI.** Pollutes logs and hides signal.

**pdb headless.** Needs a tty or remote debugger.

**Flaky not reproduced once.** Stress with `--count`.

**Changing assert to match actual.** Masks a product bug.

**.pytest_cache.** Stale --lf list after renaming a test.
