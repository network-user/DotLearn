# The parametrize decorator

## In short

`@pytest.mark.parametrize` declares argument names and a value table; PyTest creates a separate test item per row. This is data-driven testing without function copy-paste. You can stack multiple parametrize decorators or combine with indirect to pass values into a fixture.

## How to answer in an interview

- `@pytest.mark.parametrize("a,b,expected", [(1,1,2), (2,3,5)])` yields two nodes.
- `ids=["add","more"]` improves readability in CI logs.
- `pytest.param(0, marks=pytest.mark.xfail)` marks a single row.
- `indirect=True` forwards the value into a fixture with the same name.
- Honestly: a huge table in the decorator reads worse than `pytest_generate_tests`.

## How it works

Even check:

```python
import pytest


def is_even(value: int) -> bool:
    return value % 2 == 0


@pytest.mark.parametrize(
    "value,expected",
    [(2, True), (3, False), (0, True)],
    ids=["two", "three", "zero"],
)
def test_is_even(value, expected):
    assert is_even(value) is expected
```

## Pitfalls

**Arity mismatch.** An extra column in the tuple breaks collection.

**Stack order.** The outer decorator changes iteration order.

**Mutable default in ids.** Do not use a list as shared mutable state.

**parametrize on a class.** Needs pytest 6+ style; otherwise confusion with unittest.

**Missing expected.** Input-only rows turn the test into a smoke check without a meaningful assert.
