# Способы подключения fixture

## Суть

Fixture подключают тремя путями: параметр тестовой функции (явная зависимость), `autouse=True` (автоматически для scope), `@pytest.mark.usefixtures` (side-effect setup без значения в тесте). Выбор зависит от того, нужен ли return fixture в теле теста или только побочный setup.

## Что ответить на собесе

- Параметр `def test_x(db):` когда использую `db` в assert.
- `autouse=True` для логирования или reset env на каждый test.
- `@pytest.mark.usefixtures("seed_db")` когда тест не читает return, только требует данных.
- `usefixtures` на классе покрывает все методы.
- Честно: autouse на session fixture в xdist даёт неожиданный shared state без lock.

## Как это работает

Три стиля:

```python
import pytest


@pytest.fixture
def token():
    return "secret"


@pytest.fixture(autouse=True)
def trace():
    print("start")


@pytest.mark.usefixtures("token")
def test_health():
    assert True


def test_auth(token):
    assert token == "secret"
```

## Подводные камни

**usefixtures без имени в сигнатуре.** Нельзя обратиться к return внутри теста.

**autouse + heavy setup.** Каждый test платит цену, даже unit без UI.

**Порядок fixture.** Зависимости через параметры, не через import order.

**Классовый usefixtures + unittest.** Наследование unittest.TestCase ломает discovery.

**Строка в usefixtures.** Опечатка имени fixture ловится только на run.
