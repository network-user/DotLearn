# Allure with PyTest

## In short

The `allure-pytest` plugin writes JSON results to `--alluredir`; then `allure generate` or `allure serve` builds HTML. Steps, labels, links, and attachments make the report readable for QA and dev. In UI automation I attach screenshots, console logs, HAR, and traces on failure.

## How to answer in an interview

- `pytest --alluredir=allure-results` in the pipeline.
- `@allure.step` splits the scenario for triage.
- `allure.attach` for PNG, TXT, JSON bodies.
- severity, epic, feature via dynamic API or decorators.
- Honestly: Allure without stable test ids creates a "new" case on every parametrize id shuffle.

## How it works

```bash
pip install allure-pytest
pytest --alluredir=allure-results
allure serve allure-results
```

```python
import allure


@allure.step("Open catalog")
def open_catalog(page):
    page.goto("/catalog")


def test_catalog(page):
    open_catalog(page)
    allure.attach(page.screenshot(), "screen", allure.attachment_type.PNG)
    assert page.title() == "Catalog"
```

## Pitfalls

**Empty alluredir.** A forgotten plugin writes no files; generate fails.

**Large attachments.** Video inflates CI artifact storage.

**Step in helper without context.** Tiny steps add noise.

**history on flaky.** Rerun without quarantine pollutes trends.

**Local serve in CI.** Publish an HTML artifact, not only raw results.
