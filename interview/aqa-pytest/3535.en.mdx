# Markers and selective runs

## In short

Markers are tags on tests (`@pytest.mark.smoke`) for grouping and filtering runs. The CLI `-m` accepts a boolean expression over marker names. In CI a smoke suite runs a fast subset; regression at night takes the full set with `not slow`.

## How to answer in an interview

- `@pytest.mark.api` on contract tests; `pytest -m api`.
- `-m "smoke and not ui"` combines conditions.
- An unregistered marker warns; in strict mode it errors.
- `pytest -m ""` empty expression means all tests.
- Honestly: too many overlapping markers confuse; prefer 5-7 canonical ones.

## How it works

Tagging and runs:

```python
import pytest


@pytest.mark.smoke
def test_health():
    assert ping()


@pytest.mark.regression
@pytest.mark.slow
def test_bulk_import():
    assert import_rows(10_000)
```

```bash
pytest -m smoke
pytest -m "smoke or regression"
pytest -m "regression and not slow"
```

## Pitfalls

**Typo in -m.** `pytest -m smoe` silently selects 0 tests without strict.

**Marker on module.** Use `pytestmark = pytest.mark.api` at the top of the file.

**skip mark vs filter.** `@pytest.mark.skip` is not the same as a custom tag.

**xdist + markers.** Workers still collect the whole tree; filter happens after.

**Duplicate marker names.** `pytest.mark.api` and `pytest.mark.API` are different strings.
