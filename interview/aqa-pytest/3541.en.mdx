# PyTest plugins in practice

## In short

Plugins extend pytest hooks: reports, coverage, parallelism, mocking, asyncio. CI usually runs `pytest-cov`, `pytest-xdist`, `pytest-html` or Allure, and `pytest-mock`. Plugin choice is a trade-off between run speed, report detail, and maintenance cost.

## How to answer in an interview

- `pytest-cov --cov=app` gates on minimum coverage.
- `pytest-xdist -n auto` cuts wall time on a multi-core agent.
- `pytest-mock` provides `mocker.patch` with autouse teardown.
- `pytest-asyncio` for async def tests with an event loop fixture.
- Honestly: ten plugins lengthen collection and raise hook conflict risk.

## How it works

Coverage + parallel:

```bash
pip install pytest-cov pytest-xdist
pytest -n auto --cov=app --cov-report=term-missing
```

Mock:

```python
def test_token(mocker):
    mocker.patch("auth.fetch_token", return_value="fake")
    assert login() == "ok"
```

HTML report:

```bash
pytest --html=report.html --self-contained-html
```

## Pitfalls

**cov + xdist.** Needs a compatible combine mode or percent is understated.

**pytest-randomly without seed.** Flaky order exposes poorly isolated tests.

**allure step without plugin.** Decorator is silently ignored if not installed.

**Plugin vs pytest version.** Major pytest breaks old plugins at collection.

**Duplicate junit.** `pytest-html` and `--junitxml` together in CI without need.
