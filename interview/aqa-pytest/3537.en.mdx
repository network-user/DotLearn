# Key pytest CLI options

## In short

pytest options control verbosity, filtering, fail-stop behavior, tracebacks, stdout capture, and reports. Pipelines usually combine `-q`, `--maxfail`, `--tb=short`, `-m`, `--junitxml`, and `-k`. Understanding capture matters when a test logs to the console and the report looks empty.

## How to answer in an interview

- `-v` verbose list, `-q` quiet for CI.
- `-k expr` filters by node name substring.
- `--maxfail=1` fail-fast on a merge request.
- `--tb=line` short traceback in Jenkins logs.
- Honestly: `--capture=no` is noisy in parallel but saves a single-test debug session.

## How it works

Typical pipeline invocation:

```bash
pytest tests \
  -q \
  -m "smoke and not slow" \
  --maxfail=3 \
  --tb=short \
  --junitxml=reports/junit.xml \
  -k "login"
```

Useful single flags:

```bash
pytest --collect-only
pytest --lf
pytest --disable-warnings
```

## Pitfalls

**-k vs -m.** Substring name filter is confused with marker expressions.

**--maxfail=0.** Zero often means no limit, not stop immediately.

**junit family.** `--junitxml` without `--junit-prefix` collides suite names.

**--lf without cache.** last-failed needs `.pytest_cache` in the workspace.

**-x and --maxfail.** `-x` equals `--maxfail=1`, but mixing both out of habit happens.
