# Multi-arg parametrize and ids

## In short

`@pytest.mark.parametrize("a,b,expected", [(1,2,3), ...])` passes several arguments from one tuple table. The `ids` parameter sets readable node names: a string list, `pytest.param(..., id="...")`, or a callable. Good ids speed triage in Allure and junit.

## How to answer in an interview

- A tuple of length N matches N names in the decorator string.
- `ids=["admin","guest"]` aligns with two table rows.
- `id=lambda val: val[0]` when the id comes from the first field.
- `pytest.param(0, marks=pytest.mark.xfail, id="zero-edge")` for a single row.
- Honestly: auto ids from dict repr are unreadable in reports; always set ids on API matrices.

## How it works

```python
import pytest


@pytest.mark.parametrize(
    "role,status",
    [
        pytest.param("admin", 200, id="admin-ok"),
        pytest.param("guest", 403, id="guest-denied"),
    ],
)
def test_access(role, status, client):
    assert client.get("/panel", role=role).status_code == status
```

Callable ids:

```python
def id_role(row):
    return row[0]


@pytest.mark.parametrize("role,code", [("a", 1), ("b", 2)], ids=id_role)
def test_codes(role, code):
    assert code > 0
```

## Pitfalls

**ids shorter than the table.** PyTest fails at collection.

**Non-unique ids.** Collisions in report history.

**Unicode in id.** Rare issues in older CI parsers.

**Stacked parametrize ids.** Outer ids do not inherit inner auto ids.

**Empty id.** Forbidden; collection error.
