# Fixture params vs parametrize

## In short

`@pytest.fixture(params=[...])` multiplies every test that depends on the fixture for each `request.param` value. `@pytest.mark.parametrize` multiplies a specific test function over an argument table. Fixture params help when setup is expensive and data varies; parametrize when variations live only in one test's asserts.

## How to answer in an interview

- `params` on `browser` gives three runs of each UI test that requests `browser`.
- `parametrize` on `test_login(role)` does not affect neighboring tests in the file.
- Combined: parametrize plus fixture params yields a Cartesian product of nodes.
- `request.param` lives in the fixture, not directly in the test.
- Honestly: params on a session fixture explodes suite time; function scope is safer.

## How it works

Fixture params:

```python
import pytest


@pytest.fixture(params=["chromium", "firefox"])
def browser(request):
    driver = launch(request.param)
    yield driver
    driver.quit()


def test_home(browser):
    browser.get("/")
```

Parametrize on the test:

```python
@pytest.mark.parametrize("role", ["admin", "guest"])
def test_menu(role):
    assert menu_visible(role)
```

## Pitfalls

**Cartesian product.** Fixture params × parametrize quickly inflates CI time.

**Default ids.** The report shows `[chromium]` without a readable id.

**params + autouse.** An autouse fixture with params runs setup for all combinations silently.

**One test, many browsers.** A missed quit in fixture params leaks processes.

**Duplicated table.** The same cases in params and parametrize drift over time.
