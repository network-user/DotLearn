# Мультиаргументный parametrize и ids

## Суть

`@pytest.mark.parametrize("a,b,expected", [(1,2,3), ...])` передаёт несколько аргументов одной таблицей кортежей. Параметр `ids` задаёт человекочитаемые имена node: список строк, `pytest.param(..., id="...")` или callable. Хорошие ids ускоряют triage в Allure и junit.

## Что ответить на собесе

- Кортеж длиной N соответствует N именам в строке декоратора.
- `ids=["admin","guest"]` параллельно двум строкам таблицы.
- `id=lambda val: val[0]` когда id из первого поля.
- `pytest.param(0, marks=pytest.mark.xfail, id="zero-edge")` точечно.
- Честно: auto id из repr dict нечитаем в отчёте, всегда задавай ids на API matrix.

## Как это работает

```python
import pytest


@pytest.mark.parametrize(
    "role,status",
    [
        pytest.param("admin", 200, id="admin-ok"),
        pytest.param("guest", 403, id="guest-denied"),
    ],
)
def test_access(role, status, client):
    assert client.get("/panel", role=role).status_code == status
```

Callable ids:

```python
def id_role(row):
    return row[0]


@pytest.mark.parametrize("role,code", [("a", 1), ("b", 2)], ids=id_role)
def test_codes(role, code):
    assert code > 0
```

## Подводные камни

**ids короче таблицы.** PyTest падает на collection.

**Неуникальные ids.** Коллизии в report history.

**Unicode в id.** Редкие проблемы в старых CI parser.

**Stack parametrize ids.** Внешний ids не наследует inner автоматически.

**Пустой id.** Запрещён, collection error.
