# How PyTest discovers tests

## In short

PyTest collects tests by naming convention, directory layout, and CLI arguments. By default it looks for `test_*.py` / `*_test.py` files, `test_*` functions and methods inside, plus `Test*` classes without `__init__`. Markers, `pytest.ini`, and node ids (`file::test`) narrow or widen the run set.

## How to answer in an interview

- `pytest tests/api` limits the collection root.
- `pytest tests/ui/test_login.py::test_guest` runs a single node.
- `-k "checkout and not slow"` filters by name substring.
- `-m smoke` requires a registered marker in ini.
- Honestly: a file `checks_login.py` without the `test_` prefix is not collected unless `python_files` is set in ini.

## How it works

File with two tests and a false candidate:

```python
def test_addition():
    assert 1 + 1 == 2


def test_subtraction():
    assert 4 - 2 == 2


def helper_only():
    assert True
```

Runs:

```bash
pytest tests/math_ops.py
pytest tests/math_ops.py::test_addition
pytest -m smoke
```

## Pitfalls

**Test class with __init__.** Collection skips methods as incompatible.

**Duplicate import path.** Two modules with the same name create phantom duplicates.

**package without __init__.** In older layouts this affects import mode (`importlib` vs prepend).

**collect ignore.** `collect_ignore` in conftest hides a file silently.

**parametrize id.** One test becomes N nodes; Allure multiplies cases in the report.
