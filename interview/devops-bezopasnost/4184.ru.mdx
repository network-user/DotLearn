# Зачем нужно симметричное шифрование?

## Суть

Один секретный ключ шифрует и расшифровывает данные. Алгоритмы AES-GCM и ChaCha20-Poly1305 дают скорость и authenticated encryption для bulk traffic. Проблема distribution решается asymmetric wrap или pre-shared key из vault.

## Что ответить на собесе

- Использую после TLS handshake, disk encryption (LUKS), secrets at rest в etcd with KMS.
- AES-256-GCM стандарт; уникальный nonce/IV на каждый message обязателен.
- Key rotation через envelope encryption: DEK per object, KEK в KMS.
- Честно добавляю: symmetric без auth (AES-CBC alone) уязвим к padding oracle; только AEAD.

## Как это работает

```bash
KEY=$(openssl rand -hex 32)
IV=$(openssl rand -hex 12)
PLAINTEXT='backup chunk 42'

echo -n "$PLAINTEXT" | openssl enc -aes-256-gcm -K "$KEY" -iv "$IV" -out payload.bin
openssl enc -d -aes-256-gcm -K "$KEY" -iv "$IV" -in payload.bin
```

GCM даёт confidentiality + integrity tag. IV 96 bit уникален per encryption under same key. KEK храню в Vault/KMS; DEK `$KEY` шифрую asymmetric и кладу рядом с blob.

## Подводные камни

- **Nonce reuse.** Два сообщения с одним IV под одним key ломают GCM.
- **Key in env var.** Process dump и `/proc` leak; prefer tmpfs + file mode 0600 or KMS API.
- **ECB mode.** Паттерны в plaintext видны; никогда в prod.
- **Static PSK everywhere.** Компромисс одного сервиса = decrypt all backups.
- **No rotation story.** Old backups unreadable после rotate без re-encrypt plan.
