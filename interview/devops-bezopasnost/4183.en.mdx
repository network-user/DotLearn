# How does public-key encryption work?

## In short

Asymmetric crypto uses a linked key pair: public encrypts (or verifies signatures), private decrypts (or signs). Secrets can be exchanged over open channels; it is slower than symmetric crypto, so TLS uses public-key math only during handshake.

## How to answer in an interview

- Encryption: recipient's public, decrypt with recipient's private.
- Signature: sender's private signs, sender's public verifies (not the same as encryption).
- RSA, ECDSA, Ed25519; I prefer ECC for shorter keys and fast verify.
- I add honestly: raw RSA without OAEP/PSS is weak; prod uses modern padding and key size ≥2048 RSA or 256-bit ECC.

## How it works

```bash
openssl genpkey -algorithm ED25519 -out device.key
openssl pkey -in device.key -pubout -out device.pub

echo 'rotate config v3' | openssl pkeyutl -encrypt -pubin -inkey device.pub -out secret.bin
openssl pkeyutl -decrypt -inkey device.key -in secret.bin

echo 'rotate config v3' | openssl dgst -sha256 -sign device.key -out sig.bin
openssl dgst -sha256 -verify device.pub -signature sig.bin <(echo 'rotate config v3')
```

First block generates Ed25519 keys. `pkeyutl -encrypt` targets only the private key holder. Signing is separate: private sign, public verify without decrypting payload.

## Pitfalls

- **Mixing encrypt and sign.** Private-key signing does not provide confidentiality.
- **Weak RSA 1024.** Factorization is practical; migrate to ECC or RSA 3072+.
- **Public key without trust.** Need CA/WOF or pinned fingerprint or MITM swaps keys.
- **Bulk data performance.** Encrypt with symmetric key; asymmetric only wraps.
- **Key reuse across services.** One private compromise breaks all integrations.
