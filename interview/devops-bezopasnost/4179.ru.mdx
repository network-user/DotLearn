# Из каких шагов состоит TLS handshake?

## Суть

Handshake согласует версию протокола, cipher suite, проверяет сертификат сервера (и клиента при mTLS), выводит общие ключи и переключает канал на шифрованные record. В TLS 1.3 меньше round-trip и нет legacy cipher negotiation.

## Что ответить на собесе

- ClientHello: версии, ciphers, SNI, ALPN (http/1.1 vs h2).
- ServerHello + cert chain + (EC)DHE key share → shared secret.
- Finished messages подтверждают, что MITM не подменил параметры.
- mTLS добавляет CertificateRequest и client cert verify.
- Честно добавляю: на практике смотрю `openssl s_client` и ingress logs, а не рисую все 1.2 сообщения по памяти.

## Как это работает

TLS 1.3 упрощённый поток (1-RTT):

```bash
openssl s_client -connect api.example.com:443 -servername api.example.com -tls1_3 </dev/null \
  2>/dev/null | grep -E 'Protocol|Cipher|Verify return code|ALPN'

# трассировка handshake (если установлен ssldump)
# ssldump -i eth0 port 443
```

Client шлёт supported groups (X25519, P-256). Server выбирает cert matching SNI, отдаёт KeyShare. Обе стороны вычисляют shared secret, derive traffic keys, шлют encrypted Finished.

Для отладки latency:

```bash
curl -w 'connect:%{time_connect} tls:%{time_appconnect}\n' -o /dev/null -s https://api.example.com/
```

`time_appconnect` включает полный handshake; session resumption/tickets снижают на повторных запросах.

## Подводные камни

- **SNI missing в legacy client.** Wrong cert on shared IP.
- **ALPN mismatch.** HTTP/2 client к HTTP/1.1-only backend через misconfigured proxy.
- **Incomplete intermediate на server.** Handshake ok в Chrome, fail в curl/Java.
- **TLS inspection appliances.** Corporate proxy ломает cert pinning в mobile apps.
- **0-RTT replay (TLS 1.3).** Early data не идempotent; отключаю на mutating API.
