# Как устроен OAuth 2.0 для Java-сервисов?

## Суть

OAuth 2.0 делегирует доступ к ресурсам без передачи пароля пользователя клиенту. Клиент получает токен у authorization server и предъявляет его resource server; пароль видит только доверенный IdP.

## Что ответить на собесе

- Роли: resource owner, client, authorization server, resource server.
- Для server-side web беру Authorization Code + PKCE; password grant в prod не использую.
- Access token короткоживущий, refresh храню server-side или в HttpOnly cookie.
- Scope минимальный: `read:orders`, не `admin`.
- Честно: OAuth решает делегирование, не заменяет полноценную модель авторизации внутри API.

## Как это работает

Authorization Code Flow для Spring-приложения:

1. Браузер уходит на `/oauth2/authorize?response_type=code&client_id=...&redirect_uri=...&scope=read`.
2. Пользователь логинится у IdP и подтверждает scope.
3. IdP редиректит на `redirect_uri?code=...`.
4. Backend обменивает code на токены server-to-server:

```java
var form = new LinkedMultiValueMap<String, String>();
form.add("grant_type", "authorization_code");
form.add("code", authCode);
form.add("redirect_uri", redirectUri);
form.add("client_id", clientId);
form.add("client_secret", clientSecret);

var response = restClient.post()
    .uri(tokenEndpoint)
    .contentType(MediaType.APPLICATION_FORM_URLENCODED)
    .body(form)
    .retrieve()
    .body(TokenResponse.class);
```

5. С access token вызываю API:

```java
restClient.get()
    .uri(resourceBase + "/profile")
    .header(HttpHeaders.AUTHORIZATION, "Bearer " + response.accessToken())
    .retrieve()
    .body(Profile.class);
```

PKCE (`code_challenge` / `code_verifier`) защищает public clients от перехвата code. Client Credentials Flow применяю для service-to-service без пользователя.

## Подводные камни

- **Implicit flow в SPA.** Устарел; без PKCE code interception реален.
- **Refresh token в localStorage.** XSS украдёт долгоживущую сессию.
- **Широкий scope.** «Выдать всё» ломает least privilege.
- **Смешение authn и authz.** Токен доказывает личность, права проверяются отдельно.
- **Невалидированный redirect_uri.** Open redirect превращается в утечку code.
