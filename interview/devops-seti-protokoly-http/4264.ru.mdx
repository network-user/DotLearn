# Какие адреса использует канальный уровень OSI?

## Суть

Data Link layer оперирует MAC-адресами (48-bit, EUI-48): уникальный идентификатор NIC в broadcast domain. Switch строит CAM table MAC→port; ARP связывает известный IP с MAC в LAN. MAC не маршрутизируется между subnet за router.

## Что ответить на собесе

- Формат `aa:bb:cc:dd:ee:ff`; первые 3 octets OUI vendor.
- Unicast, multicast (01:00:5e for IPv4 mcast), broadcast ff:ff:ff:ff:ff:ff.
- VLAN 802.1Q добавляет tag на L2; same MAC разные broadcast domains.
- `ip link show`, `bridge fdb show` для inspect.
- Честно: random MAC on Wi-Fi privacy меняет «уникальность» для tracking, не для L2 delivery в frame.

## Как это работает

Show interface MAC:

```bash
ip link show eth0 | grep link/ether
cat /sys/class/net/eth0/address
```

ARP resolve in LAN:

```bash
ip neigh show dev eth0
arping -I eth0 -c 2 192.168.10.1
```

Switch CAM concept: frame dest MAC lookup, forward unicast to one port, flood unknown unicast.

## Подводные камни

- **MAC flap.** Same MAC on two ports; switch toggles, network unstable.
- **Static ARP stale.** Manual entry survives NIC swap; connectivity breaks.
- **VIP MAC move.** Keepalived failover; clients cache old ARP until timeout.
- **Docker veth MAC clutter.** Many interfaces; wrong `-I` in arping.
- **Confuse MAC with IP.** Traceroute L3; arping L2 only same subnet.
