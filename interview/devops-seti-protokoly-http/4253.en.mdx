# How does the TCP three-way handshake work?

## In short

Before data transfer, TCP client and server agree on session parameters with three packets: SYN, SYN-ACK, ACK. The client proposes an ISN, the server confirms and replies with its ISN, the client completes the handshake. Only then does the socket enter ESTABLISHED.

## How to answer in an interview

- Step 1: client sends SYN with seq=N.
- Step 2: server SYN-ACK with seq=M, ack=N+1.
- Step 3: client ACK with seq=N+1, ack=M+1.
- I mitigate half-open connections and SYN flood with SYN cookies and LB rate limits.
- Honestly: TLS handshake runs after TCP; I debug latency for both stages separately.

## How it works

Trace with ss/tcpdump mindset:

```bash
sudo tcpdump -i eth0 'tcp[tcpflags] & tcp-syn != 0' -nn -c 6
```

Logical sequence:

```
Client -> Server: SYN, seq=1000
Server -> Client: SYN-ACK, seq=5000, ack=1001
Client -> Server: ACK, seq=1001, ack=5001
```

Check established sessions:

```bash
ss -tn state established '( dport = :443 or sport = :443 )'
```

## Pitfalls

- **SYN backlog overflow.** New connects drop under burst; tune `somaxconn` and app accept queue.
- **Middlebox RST.** Firewall drops SYN-ACK; client sees timeout, not refused.
- **TIME-WAIT exhaustion.** Many short connections from one IP; reuseport and tw_reuse tuning.
- **MSS black hole.** PMTU not detected; odd hangs on large payload.
- **Load balancer idle.** LB without TCP proxy breaks sticky session expectations.
