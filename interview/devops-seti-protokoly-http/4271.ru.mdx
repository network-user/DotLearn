# На каком транспорте работает HTTP/3?

## Суть

HTTP/3 работает поверх QUIC, а QUIC encapsulates TLS 1.3 и stream multiplexing in UDP datagrams (usually port 443/UDP). Это убирает TCP head-of-line blocking и сокращает handshake latency compared to HTTP/2 over TCP.

## Что ответить на собесе

- Alt-Svc header advertises h3 endpoint; browser upgrades next request.
- Connection migration: QUIC connection survives client IP change (mobile).
- UDP blocked network falls back to HTTP/2 or HTTP/1.1.
- Load balancer must support QUIC termination or pass-through.
- Честно: debugging harder than tcpdump; need qlog or browser netlog.

## Как это работает

Client probe:

```bash
curl -sI --http3-only https://cloudflare.com | head -5
openssl s_client -connect example.com:443 -alpn h3 2>/dev/null | head
```

Nginx QUIC listen (concept):

```bash
grep -E 'listen.*443.*quic|http3' /etc/nginx/nginx.conf
```

Path stack:

```
HTTP/3 -> QUIC streams -> UDP -> IP
```

## Подводные камни

- **UDP 443 blocked.** Silent downgrade increases latency; monitor Alt-Svc success rate.
- **0-RTT replay.** Idempotent GET only with early data; disable for sensitive APIs.
- **Middlebox QUIC-unaware.** Some firewalls drop long UDP flows.
- **Uneven CDN support.** Origin h3 but origin farm HTTP/1.1 only; benefit only edge-client leg.
- **Certificate same as h2.** Forgot UDP listener; cert present but QUIC not bound.
