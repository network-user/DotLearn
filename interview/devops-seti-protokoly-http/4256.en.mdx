# Which HTTP status codes matter for DevOps?

## In short

An HTTP status code is a three-digit request outcome. The first digit is the class: 1xx info, 2xx success, 3xx redirect, 4xx client error, 5xx server error. In monitoring and SLO alerts I watch 2xx/4xx/5xx distribution and latency by path.

## How to answer in an interview

- 200 OK, 201 Created, 204 No Content for happy path API.
- 301/308 permanent redirect, 302/307 temporary; matters for SEO and ingress rules.
- 400 validation, 401 unauthenticated, 403 forbidden, 404 routing, 429 rate limit.
- 500 generic, 502 bad gateway, 503 overload/maintenance, 504 upstream timeout.
- Honestly: alert on 5xx rate; 404 spike may be scanning, not always an incident.

## How it works

Probe and headers:

```bash
curl -sI https://app.example.com/health | head -5
curl -s -o /dev/null -w '%{http_code} %{time_total}\n' https://app.example.com/api/v1/ping
```

Sample response:

```http
HTTP/1.1 503 Service Unavailable
Retry-After: 120
Content-Type: application/json

{"status":"degraded","reason":"database failover"}
```

Ingress/nginx access log parsing:

```bash
awk '{print $9}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head
```

## Pitfalls

- **502 vs 504.** 502 invalid upstream response; 504 upstream timeout; different root cause.
- **Custom 404 page masked.** Health check hits wrong path, always 200 HTML; use explicit endpoint.
- **301 chain.** Redirect loop breaks clients; trace with curl -L.
- **429 without Retry-After.** Clients hammer API; document backoff.
- **Cache 5xx.** CDN caches error page; purge or Cache-Control: no-store on errors.
