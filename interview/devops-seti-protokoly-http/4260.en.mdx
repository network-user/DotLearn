# How does transparent proxy differ from explicit proxy?

## In short

Transparent proxy intercepts traffic at the network layer (WCCP, iptables REDIRECT, policy routing) without client configuration. Explicit (forward) proxy requires `HTTP_PROXY` or browser proxy settings. Reverse proxy sits in front of origin and the client talks to it as the server.

## How to answer in an interview

- Transparent: client thinks it connects directly; used for ISP cache, corporate egress filter.
- Explicit: client knows proxy host:port; auth and PAC file possible.
- Reverse: nginx, HAProxy in front of app; TLS termination, load balance.
- SSL bump on transparent MITM needs corporate CA trust.
- Honestly: transparent proxy conflicts with modern HTTPS and pinning; explicit or DNS filtering is more common.

## How it works

Explicit Squid client env:

```bash
export HTTP_PROXY=http://proxy.corp.local:3128
export HTTPS_PROXY=http://proxy.corp.local:3128
export NO_PROXY=localhost,127.0.0.1,.internal

curl -I https://registry.example.com/v2/
```

Transparent redirect (concept):

```bash
sudo iptables -t nat -A PREROUTING -i eth0 -p tcp --dport 80 \
  -j REDIRECT --to-port 3128
```

Reverse proxy health:

```bash
curl -H 'Host: app.example.com' http://127.0.0.1:8080/health
```

## Pitfalls

- **HTTPS through transparent.** CONNECT or MITM; apps break without CA deploy.
- **X-Forwarded-For trust.** Reverse proxy must strip client spoofed headers at edge.
- **PAC misconfiguration.** Browser bypasses proxy for malware C2 domain.
- **Double proxy timeout.** Chain increases latency multiplicatively.
- **Cache poison on shared proxy.** Multi-tenant cache key must include Host/Vary.
