# What is an SSH key?

## In short

An SSH key is an asymmetric pair: private stays on the client, public goes on the server in `authorized_keys`. On login the client proves private key ownership via challenge signature; no password crosses the wire. It enables passwordless automation and stronger auth than password alone.

## How to answer in an interview

- Generate: `ssh-keygen -t ed25519`; files `id_ed25519` and `.pub`.
- Deploy pubkey: `ssh-copy-id`, Ansible `authorized_key`, cloud-init.
- Permissions: `~/.ssh` 700, private 600, authorized_keys 600.
- Passphrase encrypts private at rest; ssh-agent caches decrypted key in session.
- Honestly: passphrase-less key on shared CI runner is needed, but rotation and scope limits are mandatory.

## How it works

End-to-end setup:

```bash
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519 -C "deploy@ci"
ssh-copy-id -i ~/.ssh/id_ed25519.pub ops@app-server

ssh -i ~/.ssh/id_ed25519 ops@app-server 'whoami'
```

Server side check:

```bash
grep -n ed25519 ~/.ssh/authorized_keys
tail -f /var/log/auth.log
```

Auth flow: server sends challenge, client signs with private, server verifies with listed pubkey.

## Pitfalls

- **Private key leak from backup.** Keys in tarball or AMI snapshot; treat as secret.
- **authorized_keys command= typo.** Typo locks all logins for user.
- **Shared admin key.** Cannot revoke one person; per-user keys with audit.
- **No rotation.** Same key for 5 years; huge compromise window.
- **Agent forwarding.** Laptop agent exposed on remote host compromise.
