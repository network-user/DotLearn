# Как проходит трёхстороннее рукопожатие TCP?

## Суть

Перед передачей данных TCP client и server договариваются о параметрах сессии тремя пакетами: SYN, SYN-ACK, ACK. Клиент предлагает ISN, сервер подтверждает и отвечает своим ISN, клиент завершает handshake. Только после этого socket переходит в ESTABLISHED.

## Что ответить на собесе

- Шаг 1: client отправляет SYN с seq=N.
- Шаг 2: server SYN-ACK с seq=M, ack=N+1.
- Шаг 3: client ACK с seq=N+1, ack=M+1.
- Half-open connections и SYN flood митигирую SYN cookies и rate limit на LB.
- Честно: TLS handshake идёт после TCP; debug latency смотрю оба этапа отдельно.

## Как это работает

Trace через ss/tcpdump mindset:

```bash
sudo tcpdump -i eth0 'tcp[tcpflags] & tcp-syn != 0' -nn -c 6
```

Логическая последовательность:

```
Client -> Server: SYN, seq=1000
Server -> Client: SYN-ACK, seq=5000, ack=1001
Client -> Server: ACK, seq=1001, ack=5001
```

Проверка установленных сессий:

```bash
ss -tn state established '( dport = :443 or sport = :443 )'
```

## Подводные камни

- **SYN backlog overflow.** Новые connect drop при burst; tune `somaxconn` и app accept queue.
- **Middlebox RST.** Firewall режет SYN-ACK; клиент видит timeout, не refused.
- **TIME-WAIT exhaustion.** Много коротких connections с одного IP; reuseport и tuning tw_reuse.
- **MSS black hole.** PMTU не обнаружен; странные hangs на large payload.
- **Load balancer idle.** LB без TCP proxy ломает sticky session expectations.
