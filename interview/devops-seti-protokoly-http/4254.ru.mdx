# Какой уровень OSI отвечает за доставку между хостами?

## Суть

Четвёртый уровень OSI, транспортный, обеспечивает end-to-end доставку между процессами на разных хостах. TCP даёт надёжность, порядок и flow control; UDP минимальные overhead без гарантий. Порт + IP адрес образуют socket endpoint для приложения.

## Что ответить на собесе

- L4 сегментирует поток байт, добавляет port numbers; L3 (IP) только доставляет пакеты между узлами.
- TCP: 3-way handshake, ACK, retransmit, congestion control.
- UDP: datagram, без connection state; DNS, QUIC base, metrics often UDP.
- Multiplexing: один IP, много sockets по разным портам.
- Честно: в реальном troubleshooting чаще говорят L3/L4/L7, OSI модель учебная, но mapping полезен.

## Как это работает

Проверка listening ports (transport layer):

```bash
ss -tulpn | grep -E ':(80|443|8080)\s'
```

TCP connection to service:

```bash
nc -zv api.internal 8443
curl -v --connect-timeout 3 https://api.internal:8443/health
```

Сегментация: HTTP response body идёт как TCP segments с sequence numbers; потерянный segment retransmit по timeout или duplicate ACK.

## Подводные камни

- **Confuse L3 and L4.** Ping OK, но port closed; проблема transport/firewall, не routing.
- **Ephemeral port exhaustion.** Много исходящих connections с одного host; widen ip_local_port_range.
- **NAT timeout.** UDP session без traffic drop state on firewall.
- **TLS is not L4.** Шифрование above transport; `openssl s_client` debug L5+ effectively.
- **QUIC over UDP.** «UDP app» может быть reliable QUIC stream; не путать с raw datagram.
