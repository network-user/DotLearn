# Which addresses does the OSI data link layer use?

## In short

The data link layer uses MAC addresses (48-bit, EUI-48): a unique NIC identifier within a broadcast domain. Switches build a CAM table MAC to port; ARP maps known IP to MAC in LAN. MAC does not route across subnets beyond a router.

## How to answer in an interview

- Format `aa:bb:cc:dd:ee:ff`; first 3 octets are vendor OUI.
- Unicast, multicast (01:00:5e for IPv4 mcast), broadcast ff:ff:ff:ff:ff:ff.
- VLAN 802.1Q adds tag at L2; same MAC in different broadcast domains.
- `ip link show`, `bridge fdb show` for inspection.
- Honestly: random MAC on Wi-Fi privacy changes tracking uniqueness, not L2 frame delivery.

## How it works

Show interface MAC:

```bash
ip link show eth0 | grep link/ether
cat /sys/class/net/eth0/address
```

ARP resolve in LAN:

```bash
ip neigh show dev eth0
arping -I eth0 -c 2 192.168.10.1
```

Switch CAM concept: lookup dest MAC, forward unicast to one port, flood unknown unicast.

## Pitfalls

- **MAC flap.** Same MAC on two ports; switch toggles, network unstable.
- **Static ARP stale.** Manual entry survives NIC swap; connectivity breaks.
- **VIP MAC move.** Keepalived failover; clients cache old ARP until timeout.
- **Docker veth MAC clutter.** Many interfaces; wrong `-I` in arping.
- **Confuse MAC with IP.** Traceroute is L3; arping is L2 same subnet only.
