# Зачем нужен PROXY protocol?

## Суть

PROXY protocol (v1 text или v2 binary) prepend client original IP/port к TCP stream между L4 load balancer и backend. Без него backend видит только LB IP в logs, rate limits и ACL. HAProxy `send-proxy`, nginx `proxy_protocol`, AWS NLB target group support variants.

## Что ответить на собесе

- v1 human-readable `PROXY TCP4 1.2.3.4 10.0.0.5 12345 443\r\n`.
- v2 binary header efficient, supports IPv6 and LOCAL command.
- Backend must explicitly `listen ... proxy_protocol`; иначе header попадёт в HTTP parser as garbage.
- `$proxy_protocol_addr` in nginx for real_ip and logs.
- Честно: alternative is HTTP X-Forwarded-For at L7; PROXY for raw TCP/TLS pass-through.

## Как это работает

HAProxy backend:

```bash
cat <<'EOF'
backend app_nodes
  server web1 10.0.1.10:443 send-proxy-v2 check
EOF
```

Nginx accept:

```bash
cat <<'EOF'
server {
  listen 443 ssl proxy_protocol;
  set_real_ip_from 10.0.0.0/8;
  real_ip_header proxy_protocol;
}
EOF
```

Verify client IP in log after deploy.

## Подводные камни

- **Forgot proxy_protocol on listen.** Garbled TLS handshake, mysterious failures.
- **Trust all real_ip ranges.** Spoofed PROXY header if attacker reaches backend directly.
- **Mixed v1/v2.** HAProxy send-proxy-v2 to server expecting v1 breaks parsing.
- **Health check without PROXY.** Check OK, real traffic fails if check path differs.
- **Double PROXY header.** Chained LB misconfig; backend sees nested garbage.
