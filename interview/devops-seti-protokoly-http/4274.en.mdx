# What is the TCP/IP stack?

## In short

TCP/IP is the practical internet protocol stack: IP (L3) addresses and routes datagrams, TCP (L4) provides a reliable byte stream with ports, UDP is the alternative without guarantees. Application protocols (HTTP, DNS, SSH) sit above. The four-layer model (link, internet, transport, application) is simpler than academic OSI.

## How to answer in an interview

- IP is best-effort; TCP adds ACK, reorder, congestion control.
- Socket equals IP plus port plus protocol (TCP/UDP).
- IPv4/IPv6 at network layer; dual-stack is common.
- ICMP helper for diagnostics (ping, unreachable).
- Honestly: "TCP/IP" colloquially means the whole internet stack including TLS and HTTP, not only two RFCs.

## How it works

Trace path:

```bash
ip route get 1.1.1.1
ss -tn state established | head
curl -4 -v https://example.com 2>&1 | grep -E 'Connected|IPv'
```

Minimal TCP listener test:

```bash
nc -l 127.0.0.1 9876 &
echo probe | nc 127.0.0.1 9876
```

Encapsulation: Ethernet frame, IP packet, TCP segment, HTTP message.

## Pitfalls

- **Confuse TCP with IP.** Routing problem vs port firewalled needs different tools.
- **UDP "unreliable" misuse.** VoIP/QUIC/DNS tuned for UDP; not a "bad protocol."
- **NAT breaks end-to-end.** IP design assumption violated; need STUN/TURN for P2P.
- **Ignore MTU.** TCP/IP works until fragmentation or PMTUD black hole.
- **Memorize layers without tools.** `traceroute`, `ss`, `tcpdump` map theory to prod.
