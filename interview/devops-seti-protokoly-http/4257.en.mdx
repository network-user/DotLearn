# What does the OSI model describe?

## In short

OSI (Open Systems Interconnection) is a seven-layer reference model for network stacks: from physical bit transmission to application protocols. It is not a protocol but a map for diagnosis: which layer has the problem (cable, MAC, IP, TCP, TLS, HTTP).

## How to answer in an interview

- L1 Physical: signal, cable, NIC link up.
- L2 Data Link: MAC, Ethernet, switch, ARP in LAN.
- L3 Network: IP, routing, ICMP ping.
- L4 Transport: TCP/UDP, ports.
- L5 Session, L6 Presentation, L7 Application: often folded into "application layer" (HTTP, DNS, TLS).
- Honestly: TCP/IP stack is more practical than OSI, but OSI helps explain layered troubleshooting.

## How it works

Email request path (simplified):

```
Browser (L7 HTTP) -> TLS (L6) -> TCP (L4) -> IP (L3) -> Ethernet (L2) -> fiber (L1)
```

Layered debug commands:

```bash
ip link show eth0
ip route get 8.8.8.8
ss -tn state established
curl -vI https://example.com 2>&1 | grep -E 'Connected|SSL|HTTP'
```

## Pitfalls

- **Strict layer purity.** Real stacks blur L5-L7; I do not argue in interviews, I explain mapping.
- **Ping is not L7.** Ping is ICMP L3; app down with successful ping.
- **Switch vs router confusion.** L2 vs L3 device in path changes debug tools.
- **Overlay networks.** VXLAN breaks naive L2 assumptions across DC.
- **Memorize without tools.** OSI without tcpdump/ss is useless in practice.
