# Which SSH key type should you choose today?

## In short

For new keys I choose Ed25519: short public key, fast generation, strong crypto on a 256-bit curve. I keep RSA 4096 only for legacy systems without Ed25519 support. I treat DSA and RSA 1024 as obsolete and disable them in sshd.

## How to answer in an interview

- `ssh-keygen -t ed25519 -a 100` with KDF rounds for passphrase.
- RSA fallback: `-t rsa -b 4096` only if an appliance requires it.
- ECDSA nistp256 is acceptable, but Ed25519 is simpler without curve preference drama.
- On server, `PubkeyAcceptedAlgorithms` whitelists modern algos.
- Honestly: "quantum resistance" for Ed25519 is marketing oversell; it is just modern best practice.

## How it works

Generate and audit:

```bash
ssh-keygen -t ed25519 -C "ops@corp" -f ~/.ssh/id_ed25519
ssh-keygen -lf ~/.ssh/id_ed25519.pub

ssh -o PubkeyAuthentication=yes -i ~/.ssh/id_ed25519 admin@host
```

Server hardening snippet:

```bash
grep PubkeyAcceptedAlgorithms /etc/ssh/sshd_config
sshd -T | grep pubkeyacceptedalgorithms
```

## Pitfalls

- **Legacy switch RSA only.** Ed25519 rejected; keep one RSA key for old gear only.
- **Too many key types offered.** ssh client tries all keys; server may rate-limit or lock account.
- **No passphrase on laptop key.** Theft equals full access; use agent plus touch ID or hardware token.
- **Same key everywhere.** Compromise one host leaks all; per-tier deploy keys.
- **Comment PII in pubkey.** Email in `-C` visible in authorized_keys audit export.
