# What happens when a TCP connection breaks?

## In short

On packet loss or peer reset, TCP retransmits with exponential backoff. If no ACK arrives before the retransmit limit, the stack closes the socket and the app gets an error (ECONNRESET, ETIMEDOUT). Graceful close uses FIN four-way handshake through TIME-WAIT.

## How to answer in an interview

- Fast retransmit on three duplicate ACKs.
- RST aborts immediately; FIN is orderly shutdown with buffered data flush.
- Keepalive (`SO_KEEPALIVE`, `TCP_USER_TIMEOUT`) detects dead peer on long idle SSH/DB pool.
- App must retry idempotent ops; use circuit breaker on cascade failure.
- Honestly: "TCP always recovers" is false; partition can outlast app timeout.

## How it works

Observe connections:

```bash
ss -tn state time-wait | head
ss -tn state retrans
```

Simulate reset impact (concept):

```bash
curl -v --max-time 5 http://unreachable.internal:8080/ 2>&1 | tail -3
```

Graceful vs hard close:

```
FIN -> ACK -> FIN -> ACK   (orderly)
RST                        (abort)
```

## Pitfalls

- **TIME-WAIT pile-up.** Many short connections exhaust ephemeral ports on client.
- **Half-open connection.** Remote died without FIN; app hangs until keepalive fires.
- **LB idle timeout < app timeout.** Silent drop; TCP unaware until send fails.
- **Retry storm.** All clients reconnect at once after blip; thundering herd.
- **Ignore retrans metric.** Rising retrans rate is early sign before user-visible outage.
