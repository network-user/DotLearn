# Does DNS use the TCP protocol?

## In short

DNS defaults to query/response on UDP port 53. TCP on the same port is used for responses larger than legacy 512 bytes (truncation TC bit), heavy DNSSEC payloads, and zone transfer AXFR/IXFR between primary and secondary. Resolvers retry with TCP after UDP truncation.

## How to answer in an interview

- `dig +tcp` forces TCP query for testing.
- AXFR is always TCP; restrict `allow-transfer` ACL on authoritative server.
- Firewall must allow TCP/53 outbound from resolvers, not only UDP.
- EDNS0 raises UDP payload size; TCP is rarer but still required.
- Honestly: blocking TCP/53 breaks large DNSSEC chains silently after UDP truncate.

## How it works

Test UDP vs TCP:

```bash
dig @8.8.8.8 example.com A
dig +tcp @8.8.8.8 example.com A
dig +dnssec DNSKEY . | grep -E 'flags:|ANSWER:'
```

BIND allow-transfer snippet:

```bash
grep -E 'allow-transfer|also-notify' /etc/bind/named.conf.local
```

Watch traffic:

```bash
sudo tcpdump -i any port 53 -nn -c 10
```

## Pitfalls

- **Corporate firewall UDP-only 53.** Large responses fail; apps timeout randomly.
- **Open AXFR to world.** Zone enumeration leaks internal hostnames.
- **TCP connection exhaustion.** DNS amplification defense misconfig drops legit TCP.
- **Ignore TC bit.** Stub resolver loops on truncated answer without TCP fallback.
- **DoH/DoT different path.** Application DNS over HTTPS bypasses classic 53/tcp rules.
