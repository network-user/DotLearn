# Which OSI layer handles host-to-host delivery?

## In short

The fourth OSI layer, transport, provides end-to-end delivery between processes on different hosts. TCP adds reliability, ordering, and flow control; UDP offers minimal overhead without guarantees. Port plus IP address form the socket endpoint for an application.

## How to answer in an interview

- L4 segments byte streams and adds port numbers; L3 (IP) only delivers packets between nodes.
- TCP: three-way handshake, ACK, retransmit, congestion control.
- UDP: datagram, no connection state; DNS, QUIC base, metrics often use UDP.
- Multiplexing: one IP, many sockets on different ports.
- Honestly: real troubleshooting usually says L3/L4/L7; OSI is academic but mapping helps.

## How it works

Check listening ports (transport layer):

```bash
ss -tulpn | grep -E ':(80|443|8080)\s'
```

TCP connection to service:

```bash
nc -zv api.internal 8443
curl -v --connect-timeout 3 https://api.internal:8443/health
```

Segmentation: HTTP response body travels as TCP segments with sequence numbers; lost segments retransmit on timeout or duplicate ACK.

## Pitfalls

- **Confuse L3 and L4.** Ping OK but port closed; transport or firewall issue, not routing.
- **Ephemeral port exhaustion.** Many outbound connections from one host; widen ip_local_port_range.
- **NAT timeout.** UDP session without traffic drops state on firewall.
- **TLS is not L4.** Encryption sits above transport; `openssl s_client` debugs L5+ effectively.
- **QUIC over UDP.** A "UDP app" may be reliable QUIC stream; do not confuse with raw datagram.
