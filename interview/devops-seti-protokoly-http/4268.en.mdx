# What encryption underpins HTTPS?

## In short

HTTPS is HTTP over TLS. The handshake uses asymmetric crypto (ECDHE plus RSA or ECDSA certificate) to agree session keys; then symmetric AEAD ciphers (AES-GCM, ChaCha20-Poly1305) encrypt the record layer. Forward secrecy comes from ephemeral DH.

## How to answer in an interview

- TLS 1.2/1.3; 1.3 shortens handshake and removes weak ciphers.
- Cert chain: leaf, intermediate, root in trust store.
- SNI required for shared IP vhost; cert must match hostname.
- HSTS, OCSP stapling, cert rotation via ACME (Let's Encrypt).
- Honestly: "asymmetric for data" is wrong; bulk data is always symmetric after handshake.

## How it works

Inspect live cipher:

```bash
echo | openssl s_client -connect example.com:443 -servername example.com 2>/dev/null \
  | openssl x509 -noout -subject -dates

curl -vI https://example.com 2>&1 | grep -E 'SSL connection|subject:|issuer:'
```

Test TLS 1.3 only:

```bash
openssl s_client -connect example.com:443 -tls1_3 -servername example.com </dev/null
```

Cert renewal (certbot sketch):

```bash
sudo certbot certonly --nginx -d app.example.com --non-interactive --agree-tos -m ops@example.com
```

## Pitfalls

- **Expired cert midnight UTC.** Monitoring must check notAfter daily.
- **Mixed content.** HTTPS page loads HTTP asset; browser blocks or warns.
- **Weak cipher left enabled.** Old client support keeps RSA key exchange without FS.
- **Private key on load balancer only.** Backend HTTP plain inside VPC; document trust boundary.
- **Cert wrong SAN.** Valid cert for `*.example.com` fails on `api.staging.example.com`.
