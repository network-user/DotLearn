# isinstance vs type: which type-check to choose and why

## In short

Besides `type()`, an object's type can be checked with `isinstance()`. The key difference: `isinstance()` honors inheritance (an instance of a subclass counts as an instance of the base class), while `type()` compares the class literally, ignoring the hierarchy.

## How to answer in an interview

`type(x) is int` is true only when `x` is exactly `int`; for any descendant the check fails. `isinstance(x, int)` returns `True` both for `int` and for its subclasses — this aligns with the Liskov substitution principle, which is why code almost always picks it. Bonus: `isinstance` accepts a tuple of types — `isinstance(x, (int, float))`. The case where you deliberately reach for `type()` is when you need to reject subclasses and check for an exact class match.

`type()` returns the object's exact class:

```python
x = 10
print(type(x))  # <class 'int'>
```

`isinstance()` looks at the whole inheritance chain:

```python
class Animal:
    pass

class Dog(Animal):
    pass

dog = Dog()

print(isinstance(dog, Dog))     # True
print(isinstance(dog, Animal))  # True - Dog inherits Animal
print(type(dog) is Animal)      # False - the class is exactly Dog
```

In practice `isinstance` is handy for dispatching by input type:

```python
def process(data):
    if isinstance(data, list):
        return "list"
    if isinstance(data, dict):
        return "dict"
    return "other"

print(process([1, 2, 3]))       # list
print(process({'k': 'v'}))      # dict
```

## Pitfalls

- `bool` is a subclass of `int`, so `isinstance(True, int)` is `True`. If you need to tell a boolean apart from a number, check `type(x) is bool`.
- `type(x) == int` works, but `type(x) is int` is more idiomatic — classes are compared by identity.
- Excessive type checks are a code smell: it's often cleaner to rely on "duck typing" and `try/except` than to check the type up front.
- For abstract checks ("is it iterable") use the ABCs: `isinstance(x, collections.abc.Iterable)`.
