# Why does deep recursion in Python hit a ceiling and slow down?

## In short

CPython limits recursion depth (around 1000 nested calls by default) so the interpreter stack doesn't overflow. On top of that, function calls themselves aren't cheap, so recursion is usually slower than an equivalent loop.

## How to answer in an interview

Python has no tail-call optimization, and every recursive call pushes a new frame onto the stack. To avoid an overflow, the interpreter enforces a depth limit (~1000) and raises `RecursionError` when it's exceeded. You can raise the limit via `sys.setrecursionlimit`, but that only moves the ceiling and risks crashing the process against the real C stack. For deep problems, I rewrite recursion into an iterative form with an explicit stack.

The interpreter keeps active calls on a stack: each call gets a frame with its local variables. When a function calls itself, frames pile up, and once the limit is reached a `RecursionError` is raised.

```python
import sys

print("Current limit:", sys.getrecursionlimit())
sys.setrecursionlimit(1500)
print("New limit:", sys.getrecursionlimit())
```

The overhead matters too: each frame is memory plus time to create and tear it down. The classic example is the factorial:

```python
def factorial(n):
    if n == 0:
        return 1
    return n * factorial(n - 1)

print(factorial(5))  # 120
```

The base case `n == 0` stops the recursion, and the step `n * factorial(n - 1)` reduces the problem to a smaller one. Without a correct base case, the recursion won't stop and will hit the limit.

## Pitfalls

- Exceeding the depth is a `RecursionError`, not a silent hang.
- Setting `sys.setrecursionlimit` too high can crash the interpreter with a segfault: Python's limit doesn't know the real size of the C stack.
- Python doesn't optimize tail recursion - don't count on TCO.
- A missing or unreachable base case is guaranteed to cause a depth error.
- For deep structures (tree or graph traversal), an explicit stack or queue in a loop is more reliable.
