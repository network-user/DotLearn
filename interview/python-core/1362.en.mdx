# How does the interpreter reclaim unused memory on its own?

## In short

CPython manages memory automatically with two mechanisms: reference counting (frees an object as soon as no references to it remain) and the cyclic garbage collector (finds and removes groups of objects that reference each other).

## How to answer in an interview

The base mechanism in CPython is reference counting: every object has a counter, and as soon as it drops to zero, the memory is freed immediately. But reference counting can't handle cycles - two objects that hold references to each other. For that, a cyclic garbage collector from the `gc` module works on top: it periodically walks container objects, finds unreachable cycles, and frees them. It runs on a generational heuristic, not on every operation.

### Reference counting

The counter grows when a new reference appears and falls when a reference is removed or goes out of scope. Once it reaches zero, the object is destroyed right away.

```python
a = []   # reference count for the list = 1
b = a    # one more reference, count = 2
del a    # count = 1
del b    # count = 0, memory is freed immediately
```

### Cyclic garbage collector

If two objects reference each other, their counters won't reach zero even after all external references are lost:

```python
class Node:
    def __init__(self):
        self.reference = None


node1 = Node()
node2 = Node()
node1.reference = node2
node2.reference = node1
```

Reference counting won't break such a cycle. The garbage collector walks reachability from the roots, marks the unreachable objects, and frees them. It works by generations (young objects are checked more often) and can be controlled through `gc`:

```python
import gc

gc.collect()   # run collection immediately
```

## Pitfalls

- There is no "compaction"/heap defragmentation in CPython - objects don't move in memory. Freed blocks are returned to the allocator (pymalloc), not necessarily to the OS.
- The GC tracks only container objects (lists, dicts, instances). Cycles made of immutable scalars are impossible, so they aren't tracked.
- `__del__` on objects in a cycle used to prevent their collection; in modern versions this is fixed, but it's still worth avoiding finalization logic inside cycles.
- Reference counting requires updating the counter on every operation and is tied to the GIL - this is part of the reason why removing the GIL is so hard.
- The behavior is a CPython implementation detail. PyPy has no reference counting; it uses a different collector.
