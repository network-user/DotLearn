# How can you view the full set of members of an arbitrary object at runtime?

## In short

The built-in `dir(obj)` returns a sorted list of names available on the object, including inherited attributes, methods, and dunder members.

## How to answer in an interview

For object introspection I use `dir(obj)` - it returns a list of strings with all available names, including those inherited from parent classes and base types. If I need only the instance's own attributes, I look at `vars(obj)` or `obj.__dict__`; if I need one specific name - `getattr` and `hasattr`. `dir()` with no arguments returns the names from the current local scope.

## How it works

`dir()` collects names across the entire inheritance chain (MRO), not just from the instance itself, so the output includes methods of base classes as well as dunders like `__init__`, `__class__`. The result is always sorted alphabetically and consists of strings.

```python
class MyClass:
    def __init__(self):
        self.attribute1 = "value1"
        self.attribute2 = "value2"

    def method1(self):
        pass

    def method2(self):
        pass

obj = MyClass()

names = dir(obj)
print(names)
```

The output will contain not only `attribute1`, `attribute2`, `method1`, `method2`, but also the entire set of inherited dunder members. If you need exactly the data of a specific instance without inheritance and class methods, `obj.__dict__` works more precisely:

```python
print(obj.__dict__)
```

## Pitfalls

- `dir()` shows names but doesn't distinguish where they are defined - on the instance, on the class, or on an ancestor. For the instance's own fields use `__dict__` or `vars()`.
- An object can define `__dir__`, in which case `dir()`'s output is customized - it's not always an exhaustive list of the real state.
- Don't confuse `dir()` (names as strings) with `getattr(obj, name)` - to get the value, the name must be resolved separately.
- `dir()` won't show attributes added dynamically via `__getattr__` if they don't actually exist in `__dict__`.
