# What is a hash, and where in Python can't you do without it?

## In short

A hash is the result of a hash function: data of any size is turned into a value of fixed length. Hashes underpin dictionaries and sets, file integrity checks, and password storage.

## How to answer in an interview

A hash function maps an input of arbitrary length to a value of fixed size. In Python the built-in `hash()` produces an integer for immutable objects - it is exactly this value that lets `dict` and `set` find an element in almost constant time. For robust cryptographic hashes (SHA-256 and the like) there is the `hashlib` module. The key property: the same input always gives the same hash, and equal objects must have equal hashes - otherwise the dictionary breaks.

Under the hood: a dictionary hashes the key, computes a position in the table from the hash, and stores the value there. That is why only hashable (typically immutable) objects can be keys.

```python
print(hash("abc") == hash("abc"))  # True: same input - same hash
print(hash((1, 2)))                # tuples are hashable
# hash([1, 2]) -> TypeError: a list is not hashable
```

For cryptography use `hashlib`: it works with bytes and is resistant to collisions.

```python
import hashlib

digest = hashlib.sha256("Hello, World!".encode("utf-8")).hexdigest()
print(len(digest))  # 64 hexadecimal characters = 256 bits
```

## Pitfalls

- Hashability and immutability are linked: lists and dictionaries cannot be used as keys, while tuples of immutable elements can.
- The contract: if `a == b`, then `hash(a) == hash(b)`. When you override `__eq__`, you must keep `__hash__` consistent, otherwise objects will get lost in `dict`/`set`.
- The built-in `hash()` is not cryptographically secure and may differ between runs (string hash randomization) - for security use `hashlib`, not `hash()`.
- The same hash for different objects (a collision) is normal; equality is still checked via `__eq__`.
