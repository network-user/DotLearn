# How do you check whether an object can be used as a dictionary key?

## In short

An object is hashable if it has a working `__hash__` method and can be placed in dictionary keys or set elements. The most direct check is to call `hash(obj)`: if it returns a number, the object is hashable; if it raises `TypeError`, it is not.

## How to answer in an interview

Hashability is the property of an object being reducible to an integer hash that stays constant for the object's lifetime. Immutable types are usually hashable: numbers, strings, tuples of hashable elements, `frozenset`. Mutable containers - list, dict, set - are not hashable, because their hash could change. You can check by calling `hash()` inside a `try/except TypeError`, or via `isinstance(obj, collections.abc.Hashable)`.

## How it works

Structures like `dict` and `set` store elements by their hash, so they require keys and elements to be hashable and to keep their hash unchanged.

```python
def is_hashable(obj):
    try:
        hash(obj)
        return True
    except TypeError:
        return False

is_hashable("example")   # True  — a string is immutable
is_hashable((1, 2, 3))   # True  — a tuple of numbers
is_hashable([1, 2, 3])   # False — a list is mutable
is_hashable({1, 2})      # False — a set is mutable
```

An alternative is a check via the abstract base class:

```python
from collections.abc import Hashable
isinstance("example", Hashable)   # True
isinstance([1, 2, 3], Hashable)   # False
```

The link to immutability is direct: if the hash changed, the object would get "lost" in the table, since you would start looking for it by the new hash.

## Pitfalls

- **A tuple's hashability depends on its contents.** `(1, [2])` is not hashable because it contains a mutable list inside; `hash((1, [2]))` will raise `TypeError`.
- **`__hash__ = None` disables hashing.** If a user-defined class defines `__eq__` but not `__hash__`, its instances automatically become non-hashable.
- **`isinstance(obj, Hashable)` can be misleading.** It only checks for the presence of `__hash__`, but the actual call may still fail (as with a tuple containing a list). It is more reliable to actually call `hash()`.
- **Hashability is not the same as uniqueness.** Different objects can have the same hash (a collision) - this is normal and is resolved inside the table.
