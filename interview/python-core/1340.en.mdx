# Identity vs equality: when to use is, and when ==

## In short

`is` asks "is this literally the same object in memory?", while `==` asks "do the contents match?". These are two different questions, and confusing them is dangerous.

## How to answer in an interview

`is` is an identity check: the operator looks at whether two variables refer to the same object (in effect, it compares their `id()`). `==` is a value-equality check, and its behavior is defined by the `__eq__` method. With `None`, `True`, `False` you always use `is`, because these are singletons. To compare data - strings, numbers, collections - you use `==`.

## How it works

A variable in Python is a reference to an object. When you write `b = a`, a new reference points to the same object, and `a is b` is true. A literal with the same content creates a separate object - identity no longer matches, even though the values are equal.

```python
a = [1, 2, 3]
b = a
c = [1, 2, 3]

print(a is b)  # True  — the same reference
print(a is c)  # False — different objects in memory
print(a == c)  # True  — but the contents are identical
```

The `==` operator under the hood calls `a.__eq__(c)`. Any class can override this method and define its own equality logic. `is` cannot be overridden - it is hard-wired to object identity.

## Pitfalls

- Small integers (`-5..256`) and short strings are cached by CPython, so `a is b` for them sometimes happens to give `True`. You cannot rely on this - it is an implementation detail, not a language guarantee.
- Comparison with `None` is always via `is None`, not `== None`: an overridden `__eq__` may return an unexpected result.
- `is` does not make code "faster than ==" for value checks - it answers a different question and is simply wrong for data equality.
