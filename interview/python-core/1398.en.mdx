# A mutable object as a default value: why it's a classic trap

## In short

A default value is evaluated once - when the function is defined, not on every call. If it's a mutable object (a list, a dict), it lives between calls and accumulates changes, breaking expectations.

## How to answer in an interview

Default arguments are created once, at the moment `def` is compiled, and stored in `func.__defaults__`. When the default is a mutable object, all calls without an explicit argument share the same object, and edits to it survive the call. The standard fix: use `None` as the default and create the real object inside the function body when `arg is None`. That way each call gets a fresh copy.

## How it works

The dangerous version - a list as the default:

```python
def append_to_list(value, my_list=[]):
    my_list.append(value)
    return my_list

print(append_to_list(1))   # [1]
print(append_to_list(2))   # [1, 2]   <- not [2]!
print(append_to_list(3))   # [1, 2, 3]
```

The list `[]` is created once and reused across all calls without a second argument - changes leak between them. You can confirm this by peeking at the defaults:

```python
print(append_to_list.__defaults__)   # ([1, 2, 3],) - that same shared list
```

The safe version via `None`:

```python
def append_to_list(value, my_list=None):
    if my_list is None:
        my_list = []
    my_list.append(value)
    return my_list

print(append_to_list(1))   # [1]
print(append_to_list(2))   # [2]
print(append_to_list(3))   # [3]
```

Now a fresh list is born on every call, and no state leaks.

## Pitfalls

- The same trap applies to any mutable default: `dict`, `set`, a class instance. The `None` rule covers all of them.
- Check `if my_list is None`, not `if not my_list`: an empty list passed in is also falsy, and you'd accidentally replace a valid argument.
- Sometimes a shared object is used intentionally - as a primitive cache or counter. It works, but it relies on a side effect and reads poorly; prefer an explicit cache.
- Immutable defaults (`int`, `str`, `tuple`, `None`) are safe: they can't be changed in place, so no state leak occurs.
