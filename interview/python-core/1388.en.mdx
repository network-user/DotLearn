# How do you re-raise a caught exception without swallowing it?

## In short

A bare `raise` with no arguments inside an `except` block re-raises the current exception and hands it to an outer handler, preserving the original traceback.

## How to answer in an interview

If I caught an exception but can't or shouldn't handle it fully, I run a local reaction (for example, logging) and then write `raise` with no arguments. That doesn't create a new exception; it propagates the same one up the stack with its traceback untouched. I won't silence the error with an empty `except` or by returning `None` - that just hides the problem.

## How it works

`raise` inside an `except` uses the "current" active exception, which the interpreter keeps in `sys.exc_info()`. That's why no arguments are needed: Python already knows what to re-raise. The traceback isn't truncated to the current line either - it points back to the original point of failure.

```python
def process_data(data):
    try:
        result = 10 / data
    except ZeroDivisionError:
        print("Recorded the division by zero locally")
        raise

def main():
    try:
        process_data(0)
    except ZeroDivisionError:
        print("Error caught at the top level")

main()
```

Here `process_data` logs the fact of the error, but the decision about what to do is made by `main`. Control rises exactly to where a suitable handler exists.

## Pitfalls

- A bare `raise` outside an `except` block throws `RuntimeError: No active exception to re-raise`.
- Don't confuse a bare `raise` with `raise err`: the latter can overwrite the traceback with the current line. To change the type while preserving the cause, use `raise NewError(...) from err`.
- An empty `except: pass` silently swallows the error - almost always a bug, not a feature.
- Log before `raise`; otherwise, if the exception is propagated early, the local reaction won't have a chance to run.
