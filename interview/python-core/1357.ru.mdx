# Почему is с целыми числами иногда врёт, а иногда нет

## Суть

`is` сравнивает идентичность - один ли это объект в памяти, а не равенство значений. Маленькие целые (в CPython от -5 до 256) закешированы, поэтому для них `is` часто даёт `True`. Для чисел вне этого диапазона результат непредсказуем. Значения всегда сравнивайте через `==`.

## Что ответить на собесе

`is` отвечает «это один и тот же объект?» (сравнивает `id()`), `==` - «значения равны?».

- CPython заранее создаёт пул маленьких целых от -5 до 256, поэтому `a = 100; b = 100` дают `a is b == True`.
- Но `1000` в разных переменных может оказаться `False` - это уже разные объекты.
- Это деталь реализации CPython, а не гарантия языка; опираться нельзя. Надёжный сценарий для `is` - синглтоны: `is None`, `is True`, `is False`.

```python
a = 100
b = 100
print(a is b)    # True: 100 в кеше

x = 1000
y = 1000
print(x is y)    # обычно False: 1000 вне кеша

print(x == y)    # True: значения равны
```

Идиоматичная проверка на `None` использует именно `is`, потому что `None` - единственный экземпляр своего типа:

```python
value = None
if value is None:
    print("пусто")
```

## Подводные камни

- `1000 is 1000` в одной строке/функции может вернуть `True` из-за оптимизации констант компилятором - это сбивает с толку и подтверждает, что полагаться на поведение нельзя.
- Диапазон -5..256 - деталь CPython; в других реализациях (PyPy и т.д.) границы могут отличаться.
- Сравнивать значения через `is` - типичный баг: для больших чисел, строк или объектов он молча даст неверный результат.
- Для `None`, `True`, `False` всегда `is`, а не `==`: это и быстрее, и корректнее по семантике.
