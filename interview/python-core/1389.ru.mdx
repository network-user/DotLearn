# Почему глубокая рекурсия в Python упирается в потолок и тормозит?

## Суть

CPython ограничивает глубину рекурсии (по умолчанию около 1000 вложенных вызовов), чтобы стек интерпретатора не переполнился. Плюс сами вызовы функций недёшевы, поэтому рекурсия обычно медленнее эквивалентного цикла.

## Что ответить на собесе

В Python нет оптимизации хвостовых вызовов, и каждый рекурсивный вызов кладёт в стек новый фрейм. Чтобы не словить переполнение, интерпретатор держит лимит глубины (~1000) и при его превышении бросает `RecursionError`. Лимит можно поднять через `sys.setrecursionlimit`, но это лишь сдвигает потолок и рискует уронить процесс на реальном стеке C. Для глубоких задач я переписываю рекурсию в итеративную форму с явным стеком.

Активные вызовы интерпретатор хранит в стеке: на каждый вызов заводится фрейм с локальными переменными. При самовызове фреймы накапливаются, и при достижении лимита поднимается `RecursionError`.

```python
import sys

print("Текущий лимит:", sys.getrecursionlimit())
sys.setrecursionlimit(1500)
print("Новый лимит:", sys.getrecursionlimit())
```

Накладные расходы тоже важны: каждый фрейм - это память и время на его создание и снятие. Классический пример - факториал:

```python
def factorial(n):
    if n == 0:
        return 1
    return n * factorial(n - 1)

print(factorial(5))  # 120
```

Базовый случай `n == 0` останавливает рекурсию, а шаг `n * factorial(n - 1)` сводит задачу к меньшей. Без корректного базового случая рекурсия не остановится и упрётся в лимит.

## Подводные камни

- Превышение глубины - это `RecursionError`, а не тихое зависание.
- `sys.setrecursionlimit` слишком высоким значением может уронить интерпретатор сегфолтом: лимит Python не знает реального размера стека C.
- Хвостовой рекурсии Python не оптимизирует - не рассчитывайте на TCO.
- Отсутствие или недостижимый базовый случай гарантированно приводит к ошибке глубины.
- Для глубоких структур (обход дерева, графа) надёжнее явный стек или очередь в цикле.
