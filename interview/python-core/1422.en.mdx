# Catch an exception, react to it, and re-raise it up the stack

## In short

Catch the exception in `except`, perform local actions (logging, cleanup), and then a bare `raise` with no arguments sends the same exception further up with its traceback untouched.

## How to answer in an interview

When I need to do something at the point of an error but not swallow it, I use `try/except` and inside the `except` I write a bare `raise` - it re-raises the current exception, preserving the original traceback. This is not the same as `raise e`: a bare `raise` leaves the trace as is, whereas `raise e` or `raise NewError()` may overwrite the original point of failure. If I need to substitute the error type but not lose the cause, I use `raise NewError(...) from e` - this links the exceptions into a chain.

```python
import logging

def process_data(value):
    try:
        return 10 / value
    except ZeroDivisionError:
        logging.warning("Division by zero in process_data")
        raise

try:
    process_data(0)
except ZeroDivisionError:
    print("Error handled at a higher level")
```

What happens step by step:

1. `10 / value` with `value == 0` raises `ZeroDivisionError`.
2. The inner `except` catches it and performs a side action - it writes a log.
3. The bare `raise` re-raises that very exception; the traceback points to the original division line.
4. The outer `try/except` receives the re-raised error and handles it at its level.

This way the local reaction (logging) and the final handling are spread across different levels, and information about the error is not lost.

## Pitfalls

- `raise e` instead of a bare `raise` truncates or distorts the original traceback - making it harder to find the cause.
- To change the exception type and preserve the context, use `raise NewError("...") from e`; without `from`, Python will still show "During handling of the above exception", but an explicit chain reads better.
- A `raise` without an active exception (outside `except`) will raise `RuntimeError: No active exception to re-raise`.
- Resource cleanup logic that must run on any outcome is better placed in `finally` rather than duplicated in `except`.
