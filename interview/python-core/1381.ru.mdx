# isinstance против type: какой способ проверки типа выбрать и почему

## Суть

Помимо `type()`, тип объекта проверяют через `isinstance()`. Ключевое различие: `isinstance()` учитывает наследование (объект подкласса считается экземпляром базового класса), а `type()` сравнивает класс буквально, без учёта иерархии.

## Что ответить на собесе

Почти всегда выбирают `isinstance`, потому что он учитывает наследование.

- `type(x) is int` верно, только когда `x` ровно `int`; для любого наследника проверка провалится.
- `isinstance(x, int)` истинно и для `int`, и для подклассов - это согласуется с принципом подстановки Лисков; плюс принимает кортеж типов: `isinstance(x, (int, float))`.
- `type()` берут сознательно, когда нужно отвергнуть подклассы и проверить точное совпадение класса.

`type()` отдаёт точный класс объекта:

```python
x = 10
print(type(x))  # <class 'int'>
```

`isinstance()` смотрит на всю цепочку наследования:

На практике `isinstance` удобен для диспетчеризации по типу входа:

```python
def process(data):
    if isinstance(data, list):
        return "список"
    if isinstance(data, dict):
        return "словарь"
    return "другое"

print(process([1, 2, 3]))       # список
print(process({'k': 'v'}))      # словарь
```

## Подводные камни

- `bool` — подкласс `int`, поэтому `isinstance(True, int)` это `True`. Если нужно отличить булево от числа, проверяйте `type(x) is bool`.
- `type(x) == int` работает, но идиоматичнее `type(x) is int` — классы сравнивают по идентичности.
- Чрезмерные проверки типов — запах кода: часто чище положиться на «утиную типизацию» и `try/except`, чем заранее проверять тип.
- Для абстрактных проверок («можно ли итерировать») берите ABC: `isinstance(x, collections.abc.Iterable)`.
