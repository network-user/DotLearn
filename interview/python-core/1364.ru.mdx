# Как filter() помогает отбирать элементы по условию?

## Суть

`filter(предикат, коллекция)` лениво проходит по коллекции и оставляет только те элементы, для которых предикат вернул истину. Результат - итератор, а не готовый список.

## Что ответить на собесе

`filter(предикат, iterable)` возвращает ленивый итератор из элементов, на которых предикат истинный.

- Прогоняет функцию-предикат по каждому элементу и оставляет те, где результат истинный.
- Объект ленивый: элементы вычисляются по запросу, поэтому оборачиваю в `list()` или `tuple()`, когда нужен материализованный результат.
- Если вместо функции передать `None`, отсеиваются все ложные по булеву значению элементы.

## Как это работает

Сигнатура - `filter(function, iterable)`. Предикат принимает один элемент и возвращает значение, которое трактуется как истина или ложь. На выходе - объект-итератор, который можно перебрать один раз.

<PyDemo title="filter() с предикатом" autoRun code={`def is_even(number):
    return number % 2 == 0

numbers = [1, 2, 3, 4, 5, 6]

even = filter(is_even, numbers)
print(list(even))  # [2, 4, 6]`} />

Особый случай - первый аргумент `None`: тогда предикатом становится сама истинность элемента.

<PyDemo title="filter(None, ...) отсеивает ложные значения" autoRun code={`mixed = [0, 1, "", "hi", None, [], [1]]
print(list(filter(None, mixed)))  # [1, 'hi', [1]]`} />

Часто `filter()` заменяют генераторным выражением - оно читается естественнее: `(n for n in numbers if is_even(n))`.

## Подводные камни

- Итератор от `filter()` одноразовый: после первого полного прохода он пуст, повторный `list()` вернёт `[]`.
- Сам по себе `filter()` ничего не считает, пока его не начнут перебирать - это лень, а не мгновенный результат.
- В Python 2 `filter()` возвращал список; код, рассчитанный на это, в Python 3 ломается из-за смены на итератор.
- Предикат должен только проверять условие и не иметь побочных эффектов, иначе из-за ленивости поведение станет неочевидным.
