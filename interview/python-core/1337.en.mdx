# The LEGB rule: how does Python resolve a variable name?

## In short

Python has four levels of name visibility, and the interpreter searches them strictly in LEGB order: Local, Enclosing, Global (module level) and Built-in. The first match determines what the name refers to.

## How to answer in an interview

When Python encounters a name, it looks for it first in the local scope of the current function, then in enclosing functions (if this one is nested), then among the module's global names, and finally in the built-in namespace, where `len`, `print`, `Exception` live. That is the LEGB rule. Assignment by default creates a local name; to write to a global or an enclosing variable you need `global` and `nonlocal` respectively.

## How it works

Each level is a separate namespace, and the search proceeds from the inside out.

The local scope is created anew on every function call:

```python
def show():
    x = 10
    print(x)

show()
```

The enclosing scope - variables of an outer function, visible in a nested one:

```python
def outer():
    y = 20
    def inner():
        print(y)
    inner()

outer()
```

The global scope - module-level names, available to all of its functions:

```python
z = 30

def read_global():
    print(z)

read_global()
```

The built-in scope - what Python provides out of the box, without imports:

```python
print(len("Hello"))
```

## Pitfalls

- Assignment inside a function makes the name local for its entire body. If you read that variable before assigning to it, you get `UnboundLocalError`, not the value from the global scope.
- `global x` is needed to reassign a module-level variable; `nonlocal x` - to change a variable of the nearest enclosing function (not the global one).
- Mutation and assignment are different things: `lst.append(1)` for a global list works without `global`, whereas `lst = [...]` creates a local variable.
- Shadowing built-in names (`list = [1, 2]`) breaks access to the `list()` function in that scope - the L name overrides the B one.
