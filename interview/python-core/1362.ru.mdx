# За счёт чего интерпретатор сам возвращает неиспользуемую память?

## Суть

CPython управляет памятью автоматически двумя механизмами: подсчётом ссылок (освобождает объект сразу, как только на него не осталось ссылок) и циклическим сборщиком мусора (находит и убирает группы объектов, ссылающихся друг на друга).

## Что ответить на собесе

Базовый механизм в CPython - подсчёт ссылок: у каждого объекта есть счётчик, и как только он падает до нуля, память освобождается немедленно. Но подсчёт ссылок не справляется с циклами - двумя объектами, которые держат ссылки друг на друга. Для этого поверх работает циклический сборщик мусора из модуля `gc`: он периодически обходит объекты-контейнеры, находит недостижимые циклы и освобождает их. Запускается он по поколенческой эвристике, а не на каждой операции.

### Подсчёт ссылок

Счётчик растёт при появлении новой ссылки и падает, когда ссылка удаляется или выходит из области видимости. Достиг нуля - объект уничтожается тут же.

```python
a = []   # счётчик ссылок на список = 1
b = a    # ещё одна ссылка, счётчик = 2
del a    # счётчик = 1
del b    # счётчик = 0, память освобождается немедленно
```

### Циклический сборщик мусора

Если два объекта ссылаются друг на друга, их счётчики не обнулятся даже после потери всех внешних ссылок:

```python
class Node:
    def __init__(self):
        self.reference = None


node1 = Node()
node2 = Node()
node1.reference = node2
node2.reference = node1
```

Такой цикл подсчёт ссылок не разорвёт. Сборщик мусора обходит достижимость от корней, помечает недостижимые объекты и освобождает их. Он работает по поколениям (молодые объекты проверяются чаще) и его можно контролировать через `gc`:

```python
import gc

gc.collect()   # запустить сборку немедленно
```

## Подводные камни

- Никакого «компактирования»/дефрагментации кучи в CPython нет - объекты не перемещаются в памяти. Освобождённые блоки отдаются аллокатору (pymalloc), а не обязательно ОС.
- GC отслеживает только объекты-контейнеры (списки, словари, экземпляры). Циклы из неизменяемых скаляров невозможны, поэтому они не отслеживаются.
- `__del__` на объектах в цикле раньше мешал их сборке; в современных версиях это исправлено, но логику финализации в циклах всё равно стоит избегать.
- Подсчёт ссылок требует обновления счётчика при каждой операции и завязан на GIL - это часть причины, почему отказаться от GIL так сложно.
- Поведение - деталь реализации CPython. В PyPy подсчёта ссылок нет, там другой сборщик.
