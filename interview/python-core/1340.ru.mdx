# Идентичность против равенства: когда брать is, а когда ==

## Суть

`is` спрашивает «это буквально один и тот же объект в памяти?», а `==` спрашивает «совпадает ли содержимое?». Это два разных вопроса, и путать их опасно.

## Что ответить на собесе

`is` — это проверка идентичности: оператор смотрит, ссылаются ли две переменные на один объект (фактически сравнивает их `id()`). `==` — это проверка равенства значений, и её поведение определяет метод `__eq__`. С `None`, `True`, `False` всегда используют `is`, потому что это синглтоны. Для сравнения данных — строк, чисел, коллекций — берут `==`.

## Как это работает

Переменная в Python — это ссылка на объект. Когда вы пишете `b = a`, новая ссылка указывает на тот же объект, и `a is b` истинно. А литерал с тем же содержимым создаёт отдельный объект — идентичность уже не совпадёт, хотя значения равны.

```python
a = [1, 2, 3]
b = a
c = [1, 2, 3]

print(a is b)  # True  — одна и та же ссылка
print(a is c)  # False — разные объекты в памяти
print(a == c)  # True  — но содержимое одинаковое
```

Оператор `==` под капотом вызывает `a.__eq__(c)`. Любой класс может переопределить этот метод и задать свою логику равенства. А `is` переопределить нельзя — он жёстко завязан на тождество объектов.

## Подводные камни

- Мелкие целые (`-5..256`) и короткие строки CPython кеширует, поэтому `a is b` для них иногда случайно даёт `True`. Полагаться на это нельзя — это деталь реализации, а не гарантия языка.
- Сравнение с `None` всегда через `is None`, а не `== None`: переопределённый `__eq__` может вернуть неожиданный результат.
- `is` не делает код «быстрее ==» для проверки значений — он отвечает на другой вопрос и для равенства данных просто неверен.
