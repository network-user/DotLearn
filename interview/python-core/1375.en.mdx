# Why does a class need **slots**, and at what cost does it save memory

## In short

`__slots__` fixes in advance the list of allowed instance attributes. In return, Python does not create a `__dict__` for each object, which saves memory and slightly speeds up attribute access.

## How to answer in an interview

By default each instance stores its attributes in a `__dict__` dictionary, which is flexible but memory-hungry. If you declare `__slots__` with a list of names, Python allocates compact slots for those attributes instead of a dictionary. This gives a noticeable memory saving with millions of objects and slightly speeds up reads/writes. The price is a loss of dynamism: you can no longer add new, undeclared attributes.

`__slots__` lists the allowed attribute names. An attempt to introduce any other name fails with an error:

```python
class Point:
    __slots__ = ('x', 'y')

    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point(1, 2)
print(p.x, p.y)   # 1 2
# p.z = 3  -> AttributeError: 'Point' object has no attribute 'z'
```

Without `__slots__` an object has a `__dict__`, and you can write anything into it. With `__slots__` there is no such dictionary:

```python
class Free:
    pass

f = Free()
f.anything = 42        # ok, goes into __dict__
print(f.__dict__)      # {'anything': 42}
```

## Pitfalls

- `__slots__` is not inherited automatically: if a subclass does not declare its own `__slots__`, its instances get a `__dict__` again, and the saving is lost.
- With slots you cannot add an attribute that is not in the list.
- If you do need arbitrary attributes, add `'__dict__'` to `__slots__` - but then the memory saving is lost.
- Some tools and libraries (certain serializers, caching via `__dict__`) rely on a dictionary being present and may behave differently with slots.
