# Why is with integers sometimes lies and sometimes doesn't

## In short

`is` compares identity - whether it's the same object in memory, not value equality. Small integers (in CPython from -5 to 256) are cached, so for them `is` often gives `True`. For numbers outside this range the result is unpredictable. Always compare values with `==`.

## How to answer in an interview

`is` answers the question "is this literally the same object?" by comparing `id()`, while `==` answers "are the values equal?". CPython pre-creates a pool of small integers from -5 to 256 and reuses them, so `a = 100; b = 100` gives `a is b == True`. But `1000 is 1000` across different variables may turn out `False`, because these are already different objects. This is a CPython implementation detail, not a language guarantee, so you cannot rely on it. The only reliable scenario for `is` is checking singletons: `is None`, `is True`, `is False`.

## How it works

Numbers from the cached range are the same object; the rest are different.

```python
a = 100
b = 100
print(a is b)    # True: 100 is in the cache

x = 1000
y = 1000
print(x is y)    # usually False: 1000 is outside the cache

print(x == y)    # True: the values are equal
```

The idiomatic check for `None` uses precisely `is`, because `None` is the only instance of its type:

```python
value = None
if value is None:
    print("empty")
```

## Pitfalls

- `1000 is 1000` on a single line/function may return `True` due to the compiler's constant optimization - this is confusing and confirms that you can't rely on the behavior.
- The -5..256 range is a CPython detail; in other implementations (PyPy, etc.) the bounds may differ.
- Comparing values with `is` is a typical bug: for large numbers, strings, or objects it will silently give the wrong result.
- For `None`, `True`, `False` always use `is`, not `==`: it's both faster and semantically correct.
