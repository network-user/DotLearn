# Почему стандартного datetime мало для работы с часовыми поясами и зачем нужен pytz

## Суть

`datetime` из стандартной библиотеки умеет хранить дату и время и считать интервалы, но из коробки ничего не знает о часовых поясах. `pytz` — сторонняя база реальных зон, которая навешивает на объекты `datetime` правильное смещение и логику перехода на летнее/зимнее время.

## Что ответить на собесе

`datetime` сам по себе создаёт «наивные» объекты без привязки к зоне, для зон нужна реализация `tzinfo`.

- «Наивный» `datetime` считает интервалы, форматирует и парсит, но смещение UTC и DST его не интересуют.
- Чтобы получить «осведомлённый» (aware) объект, нужна `tzinfo`; исторически её брали из `pytz` - он тащит полную базу IANA и корректно обрабатывает переходы DST.
- В Python 3.9+ есть встроенный `zoneinfo`: читает ту же базу и в новых проектах предпочтительнее.

## Как это работает

Без зоны `datetime` ничего не знает о смещении:

<PyDemo title="Наивный datetime не знает о часовом поясе" autoRun code={`from datetime import datetime

naive = datetime(2023, 10, 5, 14, 30)
print(naive)          # 2023-10-05 14:30:00
print(naive.tzinfo)   # None`} />

`pytz` даёт объект зоны, который через `astimezone` пересчитывает время и проставляет смещение:

```python
from datetime import datetime, timezone
import pytz

utc_now = datetime(2023, 10, 5, 18, 30, tzinfo=timezone.utc)

ny_tz = pytz.timezone('America/New_York')
ny_time = utc_now.astimezone(ny_tz)

print(ny_time)  # 2023-10-05 14:30:00-04:00
```

Здесь `astimezone` пересчитывает момент времени в зону Нью-Йорка и прикрепляет смещение `-04:00`. Знание зоны позволяет корректно показывать одно и то же мгновение пользователям из разных регионов.

## Подводные камни

- Не вызывайте `pytz.timezone(...)` и сразу не подставляйте её в конструктор `datetime(..., tzinfo=tz)` — у `pytz` это даёт устаревшее смещение (LMT). Правильный путь — `tz.localize(naive)` или `astimezone`.
- `datetime.now()` без аргумента возвращает наивное локальное время; для зон используйте `datetime.now(tz)`.
- В Python 3.9+ предпочитайте `zoneinfo.ZoneInfo('America/New_York')` — он не требует `localize` и работает с конструктором напрямую.
- Наивные и aware-объекты нельзя сравнивать и вычитать между собой — будет `TypeError`.
