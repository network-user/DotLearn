# Namespaces and the LEGB rule: how Python finds a variable

## In short

A namespace is a table mapping names to objects in memory. Python has four levels of them - local, enclosing, global, built-in - and name lookup follows the LEGB rule, from inner to outer.

## How to answer in an interview

A namespace is a "name → object" mapping that keeps names from clashing. There are four levels: local (names inside a function), enclosing (names of an outer function as seen by a nested one), global (module level), and built-in (`len`, `print`, exceptions). When the interpreter encounters a name, it looks it up by the LEGB rule - first in the local scope, then in the enclosing one, then in the global one, and finally in the built-in scope; the first match wins. This is exactly what determines scope: where a name is accessible and which of several same-named bindings shadows the rest.

## How it works

Local - names inside a function body, alive only during its call:

```python
def my_function():
    x = 10          # x in the local namespace
    print(x)

my_function()
```

Enclosing - names of an outer function, visible from a nested one:

```python
def outer():
    y = 20

    def inner():
        print(y)    # y comes from the enclosing namespace
    inner()

outer()
```

Global - module-level names; built-in - what is always available (`print`, `len`). Lookup follows the LEGB chain, from inner to outer:

## Pitfalls

- LEGB is the read order. To assign to a name from an outer scope, you need `global` or `nonlocal`, otherwise the assignment creates a new local variable.
- Assigning to a name anywhere in a function body makes it local for the whole call - reading it before the assignment raises `UnboundLocalError`.
- Classes have their own namespace, but it does not take part in the LEGB of nested functions: methods do not see class names directly, only via `self`/the class name.
- Shadowing a built-in (`list = [...]`, `id = 5`) overrides the built-in name in the current scope - a frequent source of hidden bugs.
