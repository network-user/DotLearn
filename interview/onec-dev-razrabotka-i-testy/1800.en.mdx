# Vendor configuration mismatch: what to do

## In short

The mismatch message means database configuration support diverges from the vendor update file. Compare configurations, merge changes, update the DB on a copy, run regression, then apply to production.

## How to answer in an interview

- I start with a full database copy and vendor cf/cfu for the same release version.
- In an interview I describe the chain: compare → accept/reject marks → merge → update database.
- I stress supported objects with "capture" need release or careful merge.
- I honestly say automatic "accept all from vendor" wipes local customizations.
- After merge I check extensions and patches separately from main configuration.

## How it works

In the designer run "Compare and merge with configuration from file". The report shows object differences: vendor only, local only, changed on both sides.

On a test base after merge run "Update database configuration", then restructure when metadata changed. Critical scenarios: posting key documents, exchanges, print forms, customized reports.

If an object is off vendor support, merge modules manually via comparison. For BSP-based configurations keep a list of captured objects in the update log.

Before production: exclusive access or maintenance window, dt/cf backup, rollback plan to previous cf.

## Pitfalls

- **Merge directly in production.** No test run.
- **Skipping extensions.** Main configuration updated, extension breaks forms.
- **Different platform and cf versions.** Errors when loading the file.
- **Module conflict without manual review.** Silent loss of a BSP call.
- **Database update without restructure.** Metadata and tables diverge.
