# Event log: what it records and how to read it

## In short

The event log is built-in platform audit and diagnostics: logins, errors, scheduled jobs, application events. Entries are filtered by period, user, event, and metadata; developers can write custom events via `ЗаписьЖурналаРегистрации`.

## How to answer in an interview

- I open the log on incidents: first level Error, then context by user and event.
- In an interview I list levels: Information, Warning, Error, Note.
- I stress configuring which events are recorded in administration, or the log grows too large.
- I honestly note the log does not replace the technological log for deep SQL and server call analysis.
- In code I log business events with a clear event name and comment, without secrets in text.

## How it works

The platform automatically records auth failures, job crashes, and BSL exceptions. Developers add entries manually:

```bsl
Процедура ЗафиксироватьОшибкуОбмена(ТекстОшибки)
    ЗаписьЖурналаРегистрации(
        "ОбменСКонтрагентом",
        УровеньЖурналаРегистрации.Ошибка,
        ,
        ,
        ТекстОшибки);
КонецПроцедуры
```

In the Administration UI the log opens with filters. For bulk analysis export to `.lgf` or read via `ВыгрузитьЖурналРегистрации`. The tuple time + user + session helps reconstruct actions before an error.

## Pitfalls

- **Log size.** Without rotation disks fill up; set retention and event filters.
- **Noise.** Excessive logging in loops makes the log unreadable.
- **Personal data.** Passwords and tokens in comments violate security policy.
- **Time zone.** Correlation with external logs needs consistent server time.
- **Access rights.** Regular users do not see others' errors; support needs admin roles.
