# Configuration compare and merge: why and how

## In short

The configuration compare and merge mechanism shows differences between two metadata versions and lets you transfer selected changes. It is used when updating standard solutions, merging development branches, and moving customizations between infobases.

## How to answer in an interview

- I say we first compare supplier and consumer, then accept changes selectively.
- In an interview I separate scenarios: updating a standard release vs merging two custom branches.
- I stress that conflicts in common modules and forms are resolved manually; automation is not enough.
- I honestly note that without file export and Git/repository workflow, team work still relies on this tool.
- Before merging I take a backup and snapshot the current configuration.

## How it works

In the configurator: Configuration → Compare, merge with configuration from file (or from another infobase). The platform builds a diff tree: added objects, changed attributes, modules. The developer marks what to take from the source and what to keep in the target.

Typical UT update: export the new release to CF, compare with the working base, accept platform object changes, manually merge customized modules. EDT and CI automate via XML export and Git, but the logic is the same: diff and merge.

After accepting changes, always update the infobase structure and run regression: posting, reports, exchanges.

## Pitfalls

- **Lost customizations.** Blind "accept all from new" wipes local edits; track changed objects.
- **Module conflicts.** Automatic BSL text merge often produces syntax errors; review by hand.
- **Object order.** Dependencies between subsystems and roles can break after partial merge.
- **Extensions.** Main configuration compare does not replace extension updates; handle those separately.
- **No testing.** Merging without a smoke test on a copy is risky for production.
