# xUnitFor1C: unit testing on the platform

## In short

xUnitFor1C (ADD, Vanessa-ADD family) is a unit test framework for 1C code: annotated test methods, assert helpers, and run reports. It checks server functions, queries, and data processors without UI emulation.

## How to answer in an interview

- I use xUnitFor1C for fast checks of pure functions and posting handlers on an infobase copy.
- In an interview I show structure: test suite, method with a Test prefix, assertions like `ЮТест.ПроверитьРавенство`.
- I stress isolation: create test data in `ПередКаждымТестом`, roll back in `ПослеКаждогоТеста`.
- I honestly note that without discipline tests inflate CI time; I cover critical logic first.
- I do not mix with Vanessa UI: different levels of the test pyramid.

## How it works

The test module is attached as a data processor or extension. The runner finds export procedures marked as tests and runs them on the server.

```bsl
Процедура ТестРасчетСкидки() Экспорт
    ОжидаемаяСумма = 900;
    ФактическаяСумма = РассчитатьСуммуСоСкидкой(1000, 10);
    Если ФактическаяСумма <> ОжидаемаяСумма Тогда
        ВызватьИсключение "Неверная скидка";
    КонецЕсли;
КонецПроцедуры

Функция РассчитатьСуммуСоСкидкой(Сумма, ПроцентСкидки) Экспорт
    Возврат Сумма * (100 - ПроцентСкидки) / 100;
КонецФункции
```

Launch from the `1cv8 ENTERPRISE /Execute ...` command line or from EDT. Output: passed and failed tests with stack traces.

## Pitfalls

- **Data dependency.** Tests reading live catalogs are flaky; build fixtures.
- **Session context.** Calling `&НаКлиенте` from a test fails; test server entry points.
- **Long transactions.** Posting a document in every test slows the run; mock or batch.
- **Framework version.** ADD and xUnitFor1C APIs differ; pin the vendor in the project.
- **False confidence.** High line coverage does not guarantee business invariants are checked.
