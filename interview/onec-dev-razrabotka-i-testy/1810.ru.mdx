# Журнал регистрации: что фиксирует и как читать

## Суть

Журнал регистрации это встроенный аудит и диагностика платформы: входы, ошибки, фоновые задания, события приложения. Записи фильтруют по периоду, пользователю, событию, метаданным; разработчик может писать свои события через `ЗаписьЖурналаРегистрации`.

## Что ответить на собесе

- Я открываю журнал при инциденте: сначала уровень «Ошибка», потом контекст по пользователю и событию.
- На собеседовании перечисляю уровни: Информация, Предупреждение, Ошибка, Примечание.
- Подчёркиваю настройку состава событий в администрировании, иначе журнал раздувается.
- Честно уточняю: журнал не заменяет ТЖ для глубокого разбора SQL и серверных вызовов.
- В коде логирую бизнес-события с понятным событием и комментарием, без секретов в тексте.

## Как это работает

Платформа автоматически пишет провалы авторизации, падения фоновых заданий, исключения в BSL. Разработчик добавляет записи вручную:

```bsl
Процедура ЗафиксироватьОшибкуОбмена(ТекстОшибки)
    ЗаписьЖурналаРегистрации(
        "ОбменСКонтрагентом",
        УровеньЖурналаРегистрации.Ошибка,
        ,
        ,
        ТекстОшибки);
КонецПроцедуры
```

В интерфейсе «Администрирование» журнал открывается с отборами. Для массового анализа выгружают в `.lgf` или читают через `ВыгрузитьЖурналРегистрации`. Связка «время + пользователь + сеанс» помогает восстановить цепочку действий перед ошибкой.

## Подводные камни

- **Размер журнала.** Без ротации диск забивается; настраивайте срок хранения и фильтры событий.
- **Шум.** Избыточное логирование в цикле делает журнал нечитаемым.
- **Персональные данные.** Пароли и токены в комментарии нарушают политику безопасности.
- **Часовой пояс.** Сверка с внешними логами требует единого времени сервера.
- **Права доступа.** Обычный пользователь не видит чужие ошибки; для поддержки нужны роли администратора.
