# Logging in 1C: built-in and external mechanisms

## In short

Logging in 1C records application events for diagnostics and audit. Core channels: the event log, `Сообщить` while debugging, the cluster technological log, plus export to external systems via HTTP or files.

## How to answer in an interview

- I combine the event log for business events and the tech log for performance and SQL.
- In an interview I configure event filters so production disks are not flooded.
- I stress importance levels and meaningful event names instead of "Error1".
- I honestly note `Сообщить` in production for users does not replace centralized logging.
- For ELK/Graylog I send JSON from a processor or parse the tech log with a server agent.

## How it works

Event log from code:

```bsl
Процедура ЛогироватьЭтап(ИмяЭтапа)
    ЗаписьЖурналаРегистрации(
        "ОбработкаЗаказов",
        УровеньЖурналаРегистрации.Информация,
        ,
        ,
        "Этап: " + ИмяЭтапа);
КонецПроцедуры
```

Temporary debug without the log:

```bsl
Процедура ОтладочныйПрогон()
    Сообщить("Старт обработки: " + ТекущаяДата());
КонецПроцедуры
```

Enable the technological log in the cluster console: events `EXCP`, `DBMSSQL`, `SCALL` with duration. For external monitoring, a handler sends a structure to REST or writes CSV for Logstash pickup.

## Pitfalls

- **Duplication.** One event in log, Сообщить, and file triples volume and confuses investigation.
- **Secrets in text.** Tokens and personal data in comments end up in exports.
- **Tech log noise.** Full dump without duration filter fills the disk overnight.
- **Time zones.** Correlation with nginx or postgres logs breaks without UTC.
- **Read permissions.** A developer without admin roles cannot see other sessions in the log.
