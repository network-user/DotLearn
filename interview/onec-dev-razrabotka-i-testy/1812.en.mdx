# Evaluate mode in the debugger

## In short

Evaluate lets you compute an expression or inspect a variable under the cursor while the debugger is paused. It is a fast way to see intermediate results without `Сообщить` or restarting the session.

## How to answer in an interview

- I stop at a breakpoint, enter `Переменная1 + Переменная2` in the evaluate window, and compare with expectation.
- In an interview I show hovering over an identifier in the editor during an active debug session.
- I stress the difference: watch for ongoing monitoring, evaluate for a one-off check.
- I honestly note in a client module you cannot evaluate a server expression without a server call.
- After code changes I restart debugging or I inspect an old module version.

## How it works

```bsl
Процедура ДемоСложения()
    А = 10;
    Б = 20;
    Результат = А + Б;
    Сообщить(Результат);
КонецПроцедуры
```

Breakpoint on the `Результат` assignment line. While paused, hover `А` and `Б` or open Evaluate, enter `А + Б`, get 30 before the line runs. This validates hypotheses about wrong type or empty reference.

In EDT and the configurator the UI is similar: context menu Evaluate expression, history of recent formulas.

## Pitfalls

- **Execution context.** The expression sees only locals and module variables visible in the current stack frame.
- **State mutation.** Calling a function with side effects from Evaluate changes data.
- **Types and Undefined.** Reference comparison without `ЗначениеЗаполнено` leads to wrong conclusions.
- **Optimized client.** In thin client without debug attach, value hints are unavailable.
- **Long expressions.** A heavy query in Evaluate blocks the UI; use a temporary variable instead.
