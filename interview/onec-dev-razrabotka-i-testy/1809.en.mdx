# Profiling code and queries in 1C

## In short

Profiling collects timing statistics for BSL and SQL during a debug session. Enable it in the configurator or client via Debug → Profiling; after running a scenario it reports hot spots.

## How to answer in an interview

- I enable profiling on an infobase copy, reproduce a slow report or posting, and review the time top list.
- In an interview I separate BSL profiling from the technological log/SQL trace for queries.
- I stress that without representative data volume conclusions can be wrong.
- I honestly note profiling slows the session; do not leave it on in production.
- After optimization I remeasure or the effect is invisible.

## How it works

1. Start enterprise with debugging on the target base.
2. Debug → Start profiling.
3. Run the problematic operation (report, processor, posting).
4. Stop profiling and open the report.

The report shows a call tree with accumulated time: which procedure and nested query consumed seconds. For queries I also check the DBMS plan and indexes.

```bsl
Процедура ТяжелаяОбработка()
    Для Индекс = 1 По 1000 Цикл
        ВыполнитьВспомогательныйРасчет(Индекс);
    КонецЦикла;
КонецПроцедуры
```

If the profile shows a loop with server calls inside, move work to one batch query or `&НаСервереБезКонтекста`.

## Pitfalls

- **Debugger noise.** Breakpoints skew timings; use a clean run for measurement.
- **Platform cache.** The second run is faster; compare under equal conditions.
- **Client vs server.** Form time can hide a server issue; inspect both layers.
- **Small data set.** Hot spots may not appear on an empty base.
- **Confusion with tech log.** The technological log does not replace the BSL profiler; use both.
