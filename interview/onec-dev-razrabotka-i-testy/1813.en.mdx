# Call stack when debugging BSL

## In short

The call stack is the chain of active procedures and functions from the current execution point to the entry point. The 1C platform keeps frames with local variables and return addresses; in the debugger the stack shows who called whom on error or stop.

## How to answer in an interview

- I open the stack panel on an exception and walk up from the handler to the caller.
- In an interview I explain LIFO: the last called frame is on top, `Main` or an event handler at the bottom.
- I link it to the stack trace text in errors "{Module...}: line N".
- I honestly note deep recursion without an exit condition overflows the stack.
- For async calls the chain may be unclear; I check the event log and technological log.

## How it works

```bsl
Процедура ТочкаВхода()
    ВыполнитьУровеньА();
КонецПроцедуры

Процедура ВыполнитьУровеньА()
    ВыполнитьУровеньБ();
КонецПроцедуры

Процедура ВыполнитьУровеньБ()
    ВызватьИсключение "Ошибка на нижнем уровне";
КонецПроцедуры
```

When stopped in `ВыполнитьУровеньБ` the stack top to bottom is: `ВыполнитьУровеньБ` → `ВыполнитьУровеньА` → `ТочкаВхода`. Switching frames changes visible locals. This shows with which parameters execution reached the failing line.

## Pitfalls

- **Recursion.** Factorial without a base case exhausts the stack; catch in tests.
- **Mutual calls.** Two modules call each other until overflow.
- **Lost context.** A `Попытка/Исключение` handler without rethrow hides upper frames from the user message.
- **Server and client.** A client frame does not show server locals without switching the debug session.
- **Optimization.** In release without debugging the stack in messages is shorter and lacks variable detail.
