# EDT vs configurator: when to choose which

## In short

The configurator is the classic in-platform 1C IDE for editing metadata and debugging in the familiar tree. EDT (1C:Enterprise Development Tools) is an Eclipse-based IDE with Git, refactoring, formatting, and file sources for large teams and CI.

## How to answer in an interview

- I use the configurator for quick fixes on a stand and EDT for Git branches and code review.
- In an interview I list EDT strengths: Git diff, validators, plugins, XML export.
- I stress both edit the same metadata; mixing CF and git without discipline fails.
- I honestly note EDT has a steeper learning curve; the configurator is faster for small tasks.
- For standard updates compare/merge is still often in the configurator; EDT is catching up.

## How it works

In the configurator open a base, edit an object module, debug in the same session. In EDT the project lives under `src/`, commits go to Git, CF/CFU build runs in a pipeline.

```bsl
&НаСервере
Процедура ЗаполнитьКодПоУмолчанию(ОбъектСправочника)
    Если Не ЗначениеЗаполнено(ОбъектСправочника.Код) Тогда
        ОбъектСправочника.Код = "00001";
    КонецЕсли;
КонецПроцедуры
```

The same BSL is edited in both IDEs; the difference is environment: EDT offers project-wide completion, find references, unified style. The configurator is tighter with a live infobase and structure updates.

## Pitfalls

- **Two sources of truth.** CF-only edits are lost; Git-only edits without base load are untested.
- **EDT version.** Platform version mismatch breaks publish.
- **Performance.** Huge projects in EDT on weak hardware slow indexing.
- **Team habits.** Some developers stay in the configurator and skip Git review.
- **Extensions.** Not all compare workflows in EDT are as smooth for extensions vs main configuration.
