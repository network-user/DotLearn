# What module types exist in a 1C configuration

## In short

1C code is split across modules with different execution contexts: common, object, manager, form, command, session, application, external processors. Module choice defines client/server availability, events, and procedure scope.

## How to answer in an interview

- I list at least seven types and tie each to its call site.
- In an interview I explain object module (instance) vs manager module (type as a whole).
- I stress common module flags: server, client, external connection, privileged.
- I honestly say the form module is split by `&НаКлиенте` / `&НаСервере` directives.
- For reusable logic without metadata binding I pick a common module.

## How it works

Common module `РаботаСоСтроками` with the Server flag:

```bsl
Функция НормализоватьКод(ИсходныйКод) Экспорт
    Возврат ВРег(СокрЛП(ИсходныйКод));
КонецФункции
```

Catalog manager module holds factory methods:

```bsl
Функция НайтиПоАртикулу(Артикул) Экспорт
    Возврат Справочники.Номенклатура.НайтиПоРеквизиту("Артикул", Артикул);
КонецФункции
```

Document object module handles instance events:

```bsl
Процедура ОбработкаПроведения(Отказ, Режим)
    СформироватьДвиженияПоСкладу(Отказ);
КонецПроцедуры
```

Form module handles UI events:

```bsl
&НаКлиенте
Процедура СуммаПриИзменении(Элемент)
    РассчитатьИтогНаСервере();
КонецПроцедуры
```

Session, managed and ordinary application modules initialize session parameters and global startup handlers. Command module links a UI button to a procedure. Information register record set module serves the record set tabular section.

## Pitfalls

- **Common module without the right context.** Calling a server-only procedure from the client.
- **Business logic in the form module.** Duplication and no call from scheduled jobs.
- **Posting in the manager module.** The event lives in the object module.
- **Privileged module without need.** RLS bypass.
- **Confusing object and form modules.** Form `ПриЗаписи` does not replace the object handler.
