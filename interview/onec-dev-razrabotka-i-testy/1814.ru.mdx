# Логирование в 1С: встроенные и внешние механизмы

## Суть

Логирование в 1С это фиксация событий работы приложения для диагностики и аудита. Базовые каналы: журнал регистрации, `Сообщить` при отладке, технологический журнал кластера, плюс выгрузка во внешние системы через HTTP или файлы.

## Что ответить на собесе

- Я комбинирую журнал регистрации для бизнес-событий и ТЖ для производительности и SQL.
- На собеседовании настраиваю фильтры событий, чтобы не забить диск на проде.
- Подчёркиваю уровни важности и осмысленные имена событий вместо «Ошибка1».
- Честно уточняю: `Сообщить` в проде пользователю не заменяет централизованный лог.
- Для ELK/Graylog отправляю JSON из обработки или парсю ТЖ агентом на сервере.

## Как это работает

Журнал регистрации из кода:

```bsl
Процедура ЛогироватьЭтап(ИмяЭтапа)
    ЗаписьЖурналаРегистрации(
        "ОбработкаЗаказов",
        УровеньЖурналаРегистрации.Информация,
        ,
        ,
        "Этап: " + ИмяЭтапа);
КонецПроцедуры
```

Временная отладка без журнала:

```bsl
Процедура ОтладочныйПрогон()
    Сообщить("Старт обработки: " + ТекущаяДата());
КонецПроцедуры
```

Технологический журнал включают в консоли кластера: события `EXCP`, `DBMSSQL`, `SCALL` с длительностью. Для интеграции с внешним мониторингом пишут обработчик, который шлёт структуру в REST или складывает CSV в каталог, откуда забирает Logstash.

## Подводные камни

- **Дублирование.** Одно событие в журнал, Сообщить и файл троит объём и путает расследование.
- **Секреты в тексте.** Токены и персональные данные в комментарии попадают в выгрузку.
- **Шум ТЖ.** Полный дамп без фильтра по длительности убивает диск за ночь.
- **Часовые пояса.** Сопоставление с nginx или postgres логами ломается без UTC.
- **Права на чтение.** Разработчик без админских ролей не увидит чужие сеансы в журнале.
