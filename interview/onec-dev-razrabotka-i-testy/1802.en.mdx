# Managed vs ordinary application: key differences

## In short

Managed application is the main 8.3 mode: thin and web client, managed forms, client-server code split. Ordinary (thick) application uses legacy-style forms and is rare today, mainly for legacy configurations.

## How to answer in an interview

- I say new projects and standard configurations use managed mode only.
- In an interview I list clients: thin, thick (managed), web, mobile vs classic thick UI.
- I stress compilation directives and the ban on server code on the client in managed forms.
- I honestly note ordinary application has no web client and lacks several modern platform features.
- For migration I convert forms to managed and rework modality.

## How it works

In managed application a list form opens via a standard platform command:

```bsl
&НаКлиенте
Процедура ОткрытьСписокНоменклатуры(Команда)
    ОткрытьФорму("Справочник.Номенклатура.ФормаСписка");
КонецПроцедуры
```

Server logic is in a separate procedure:

```bsl
&НаСервереБезКонтекста
Функция ПолучитьКоличествоПозиций()
    Запрос = Новый Запрос;
    Запрос.Текст = "ВЫБРАТЬ КОЛИЧЕСТВО(*) КАК Кт FROM Справочник.Номенклатура";
    Возврат Запрос.Выполнить().Выгрузить()[0].Кт;
КонецФункции
```

In ordinary application forms are often modal; form module code may run on the client without explicit split. Controls bind to the legacy "form = object" model.

Managed mode provides adaptive UI, command interface, standard BSP subsystems, SaaS, and web server publication. Ordinary mode is not evolving and does not fit cloud scenarios.

## Pitfalls

- **Mixing modes in one configuration.** Duplicate forms and inconsistent UX.
- **Modal window in managed client.** UI block and web client issues.
- **Direct SQL/query on the client in managed mode.** Unavailable or unsafe.
- **Porting without reworking events.** `ПриОткрытии` vs `ПриСозданииНаСервере`.
- **Expecting web access in ordinary application.** Architecturally impossible.
