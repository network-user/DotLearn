# Профилирование кода и запросов в 1С

## Суть

Профилирование собирает статистику времени выполнения участков BSL и SQL во время сеанса отладки. Включается в конфигураторе или клиенте через «Отладка» → «Профилирование», после прогона сценария даёт отчёт с горячими точками.

## Что ответить на собесе

- Я включаю профилирование на копии базы, воспроизвожу медленный отчёт или проведение, смотрю топ по времени.
- На собеседовании разделяю профилирование BSL и технологический журнал/SQL trace для запросов.
- Подчёркиваю, что без репрезентативного объёма данных выводы могут быть ложными.
- Честно уточняю: профилирование замедляет сеанс; не оставляю включённым на проде.
- После оптимизации повторяю замер, иначе не видно эффекта.

## Как это работает

1. Запустить предприятие с отладкой на нужной базе.
2. Отладка → Начать профилирование.
3. Выполнить проблемную операцию (отчёт, обработка, проведение).
4. Остановить профилирование, открыть отчёт.

Отчёт показывает дерево вызовов с накопленным временем: какая процедура и какой вложенный запрос съели секунды. Для запросов дополнительно смотрю план в СУБД и индексы.

```bsl
Процедура ТяжелаяОбработка()
    Для Индекс = 1 По 1000 Цикл
        ВыполнитьВспомогательныйРасчет(Индекс);
    КонецЦикла;
КонецПроцедуры
```

Если в профиле доминирует цикл с серверными вызовами внутри, переношу расчёт в один пакетный запрос или `&НаСервереБезКонтекста`.

## Подводные камни

- **Шум отладчика.** Точки останова искажают тайминги; для замера используйте чистый прогон.
- **Кэш платформы.** Второй запуск быстрее первого; сравнивайте в одинаковых условиях.
- **Клиент vs сервер.** Время на форме может скрывать серверную проблему; смотрите оба слоя.
- **Малый набор данных.** На пустой базе узкое место не проявится.
- **Путаница с ТЖ.** Технологический журнал не заменяет профилировщик BSL, нужны оба инструмента.
