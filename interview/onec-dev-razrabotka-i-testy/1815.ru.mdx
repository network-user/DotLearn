# EDT и конфигуратор: когда что выбрать

## Суть

Конфигуратор это классическая среда разработки внутри платформы 1С для правки метаданных и отладки в привычном дереве. EDT (1C:Enterprise Development Tools) это Eclipse-based IDE с Git, рефакторингом, форматированием и работой с исходниками в файлах для крупных команд и CI.

## Что ответить на собесе

- Я использую конфигуратор для быстрых правок на стенде и EDT для веток Git и code review.
- На собеседовании перечисляю плюсы EDT: сравнение с Git, валидаторы, плагины, выгрузка в XML.
- Подчёркиваю, что обе среды правят одни метаданные; смешивать CF и git без дисциплины нельзя.
- Честно уточняю: порог входа в EDT выше, на малых задачах конфигуратор быстрее.
- Для типовых обновлений сравнение/объединение пока чаще в конфигураторе, EDT догоняет.

## Как это работает

В конфигураторе открываем базу, правим модуль объекта, сразу отлаживаем в том же сеансе. В EDT проект хранится в каталоге `src/`, коммиты в Git, сборка CF/CFU через pipeline.

```bsl
&НаСервере
Процедура ЗаполнитьКодПоУмолчанию(ОбъектСправочника)
    Если Не ЗначениеЗаполнено(ОбъектСправочника.Код) Тогда
        ОбъектСправочника.Код = "00001";
    КонецЕсли;
КонецПроцедуры
```

Один и тот же BSL редактируется в обеих IDE; разница в окружении: EDT даёт автодополнение по всему проекту, поиск ссылок, единый стиль. Конфигуратор теснее связан с живой ИБ и обновлением структуры.

## Подводные камни

- **Две правды.** Правки только в CF без Git теряются; правки только в Git без выгрузки в базу не тестируются.
- **Версия EDT.** Несовместимость с версией платформы ломает публикацию.
- **Производительность.** Огромный проект в EDT на слабом ПК тормозит индексацию.
- **Привычки команды.** Часть разработчиков в конфигураторе обходит review в Git.
- **Расширения.** Не все операции сравнения удобны в EDT для расширений vs основной конфигурации.
