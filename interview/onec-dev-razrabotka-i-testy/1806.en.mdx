# TestClient: scenario automation in 1C

## In short

TestClient (Vanessa Automation and related frameworks) runs 1C test scenarios through the managed client: it emulates user actions in forms and checks outcomes. It is a UI and end-to-end tool, not a replacement for modular business-logic tests.

## How to answer in an interview

- I separate TestClient for UI scenarios and xUnit/ADD for modular server code checks.
- In an interview I say scenarios are written in Gherkin or Vanessa step language.
- I stress CI launch via the platform command line and reports in Allure/JUnit.
- I honestly note tests are brittle when forms change; stable element names are required.
- For standard release regression I keep smoke scenarios on critical documents.

## How it works

A scenario describes steps: open a list, create a document, fill attributes, post, verify movements. TestClient attaches to a 1C session and runs UI commands.

```bsl
Процедура ТестПроведенияРеализации() Экспорт
    ОжидаемыйСтатус = Перечисления.СтатусыДокументов.Проведен;
    ФактическийСтатус = ПровестиТестовуюРеализацию();
    Если ФактическийСтатус <> ОжидаемыйСтатус Тогда
        ВызватьИсключение "Документ не проведён";
    КонецЕсли;
КонецПроцедуры
```

In Vanessa, steps bind to handler procedures; TestClient invokes them while parsing the feature file. The report records failed steps with form screenshots.

## Pitfalls

- **Brittle UI.** Renaming a button breaks the scenario; use element `Имя`, not caption.
- **Test base data.** Scenarios depend on opening balances; reset or use a separate environment.
- **Run time.** Full regression via client is slow; combine with modular tests.
- **Platform version.** 8.3.x updates can change thin client behavior; pin version in CI.
- **Parallel runs.** Two TestClients on one base conflict on data; isolate infobase copies.
