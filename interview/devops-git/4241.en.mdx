# How do you create and publish a Git tag?

## In short

Tags attach to a commit: lightweight (`git tag v1.0.0`) or annotated (`git tag -a -m`). Tags do not travel with a normal branch push; use `git push origin v1.0.0` or `--tags`. Delete locally with `-d`, on remote with `git push origin --delete tag`.

## How to answer in an interview

- Releases use annotated tags only: `git tag -a v3.2.1 -m "..."`.
- Tag a specific SHA: `git tag -a v3.2.1 <sha>`.
- After pushing the tag, CI `on: push tags: v*` builds the artifact.
- Semver aligns with the changelog for breaking/minor/patch.
- Honestly: `git push --tags` pushes every local tag including drafts; I push one tag explicitly.

## How it works

```bash
git switch main
git pull --rebase origin main

git tag -a v3.2.1 -m "Release 3.2.1"
git show v3.2.1

git push origin v3.2.1
```

List and delete a mistake:

```bash
git tag -l 'v3.*'
git tag -d v3.2.1-wrong
git push origin --delete v3.2.1-wrong
```

Tag an older commit:

```bash
git log --oneline -10
git tag -a v3.1.2-hotfix abc1234 -m "Hotfix CVE-2024-xxx"
git push origin v3.1.2-hotfix
```

## Pitfalls

- **Tag on wrong commit.** CI deploys the wrong tree; verify SHA in the release checklist.
- **Local tag without push.** «Released» in chat but no artifact on the remote.
- **Rewriting remote tags.** `--force` on tags breaks reproducibility; enforce immutability.
- **Prerelease naming.** `v1.0.0-rc1` vs `v1.0.0` trigger different pipelines; document regex rules.
- **Signed tag without GitHub pubkey.** No verified badge until GPG key is in account settings.
