# Зачем в GitFlow несколько типов веток?

## Суть

GitFlow разводит стабильный релизный код, интеграционную разработку и срочные правки по отдельным веткам. `main` хранит только production-ready теги, `develop` собирает фичи, `feature/*`, `release/*` и `hotfix/*` изолируют работу и снижают риск случайного деплоя сырого кода.

## Что ответить на собесе

- Feature ответвляется от develop, merge обратно после review.
- Release branch стабилизирует версию: только bugfix, без новых фич.
- Hotfix от main попадает в prod и backport в develop.
- Теги на main маркируют semver релизы для changelog и артефактов.
- Честно: для continuous deployment GitFlow часто тяжеловат; trunk-based или GitHub Flow проще.

## Как это работает

Жизненный цикл фичи и hotfix:

```bash
git switch develop
git switch -c feature/rate-limiter
git commit -am "feat(api): add token bucket"
git switch develop
git merge --no-ff feature/rate-limiter
git branch -d feature/rate-limiter

git switch -c release/3.2.0 develop
git commit -am "chore: bump version 3.2.0"
git switch main
git merge --no-ff release/3.2.0
git tag -a v3.2.0 -m "Release 3.2.0"
git switch develop
git merge release/3.2.0

git switch main
git switch -c hotfix/3.2.1
git commit -am "fix(auth): null session edge case"
git switch main && git merge hotfix/3.2.1 && git tag v3.2.1
git switch develop && git merge hotfix/3.2.1
```

Каждый тип ветки ограничивает, какие commits могут попасть в prod.

## Подводные камни

- **Long-lived develop.** Отстаёт от main на месяцы; merge release болезненный.
- **Forgotten backport.** Hotfix только в main, develop снова ломается на следующем релизе.
- **Feature flags vs branches.** Держать feature/* месяцами дешевле через flag в trunk.
- **Naming chaos.** `feature/JIRA-123` без policy; grep и automation страдают.
- **Double merge conflicts.** Release и hotfix одновременно трогают одни файлы.
