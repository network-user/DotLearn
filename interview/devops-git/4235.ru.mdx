# Зачем нужны Git-теги?

## Суть

Тег это именованная ссылка на коммит, обычно для релиза (`v2.4.1`). Lightweight tag это просто pointer; annotated tag хранит message, author, дату и может подписываться GPG. CI собирает артефакты по тегу; semver tag триггерит production deploy.

## Что ответить на собесе

- Annotated tag для релизов: `git tag -a v1.2.0 -m "..."` и `git push origin v1.2.0`.
- Lightweight достаточно для временных меток, но в релизном процессе не использую.
- `git show v1.2.0` показывает коммит и tag message; changelog привязываю к диапазону тегов.
- Signed tags (`-s`) для supply chain в regulated средах.
- Честно: tag на wrong commit в prod дорого откатывать; проверяю SHA в CI перед publish.

## Как это работает

```bash
git switch main
git pull --rebase

git tag -a v2.4.1 -m "Release 2.4.1: payment retry fix"
git push origin v2.4.1

git tag -l 'v2.*'
git show v2.4.1
```

Список коммитов между релизами:

```bash
git log v2.4.0..v2.4.1 --oneline
```

В pipeline checkout по тегу:

```bash
git fetch origin tag v2.4.1 --no-tags
git checkout v2.4.1
```

## Подводные камни

- **Теги не пушатся с branch.** `git push` без `--tags` оставляет тег локальным; релизный job не видит версию.
- **Движение тега.** Перезапись тега на remote (`--force`) ломает воспроизводимость артефактов; immutable tags policy.
- **Tag на merge commit.** Непонятно что вошло в релиз; предпочитаю tag после fast-forward main.
- **Case sensitivity.** `v1.0` и `V1.0` разные refs; фиксирую naming convention.
- **Detached HEAD deploy.** Checkout tag для build ок, но hotfix делаю с branch, не с тега напрямую.
