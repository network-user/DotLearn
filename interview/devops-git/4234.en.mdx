# What does git pull do?

## In short

`git pull` wraps `git fetch` plus integration into the current branch (usually merge, or rebase with `--rebase`). It brings remote commits and embeds them into local history. On CI and feature branches I often run fetch + rebase explicitly for a linear history.

## How to answer in an interview

- `git pull origin main` updates the tracking branch and merges into HEAD.
- `git pull --rebase` replays my commits on top of fresh remote without a merge commit.
- I commit or stash local changes before pulling.
- On conflicts pull stops; I fix markers, `git add`, then `git rebase --continue` or finish the merge commit.
- Honestly: blind pulls on shared main during peak hours create merge commits; I prefer fetch, review, then rebase.

## How it works

```bash
git fetch origin main

git log --oneline HEAD..origin/main

git pull --rebase origin main
```

With upstream configured:

```bash
git branch -vv
git pull --rebase
```

After a successful pull the local branch points at the integrated history; `origin/main` moved during fetch.

## Pitfalls

- **Pull without upstream.** Branches without tracking need explicit `origin name` or pull fails.
- **Merge commit noise.** Default pulls on long-lived branches spawn «Merge branch main»; teams agree on rebase policy.
- **Conflict on dirty tree.** Uncommitted edits in the same lines block rebase; stash or WIP commit first.
- **Force push on remote.** After someone force-pushes, pull may duplicate work; use `git fetch --prune` and coordinated reset/rebase.
- **Submodules.** `git pull` does not update submodule checkouts; run `git submodule update` separately.
