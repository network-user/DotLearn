# Что такое remote в Git?

## Суть

Remote это именованная ссылка на другой Git-репозиторий, обычно на сервере (GitHub, GitLab, self-hosted). Локальный clone хранит commits; remote позволяет fetch/push изменений. По умолчанию после clone создаётся `origin`, указывающий на URL источника.

## Что ответить на собесе

- `git remote -v` показывает fetch/push URL для каждого remote.
- `git remote add upstream <url>` для fork workflow.
- `git fetch origin` обновляет remote-tracking branches без merge в working branch.
- `git push origin feature/x` публикует локальные commits.
- Честно: несколько remotes без naming convention путают; документирую origin/upstream/deploy.

## Как это работает

Настройка и sync:

```bash
git clone git@github.com:team/platform.git
cd platform
git remote -v

git remote add staging git@deploy.internal:platform.git
git fetch staging

git switch main
git pull origin main

git switch -c fix/metrics-label
git commit -am "fix(obs): correct histogram bucket"
git push -u origin fix/metrics-label
```

Inspect remote branches:

```bash
git branch -r
git log origin/main..HEAD --oneline
```

Remote не копирует repo целиком при каждом fetch; обновляются refs under `refs/remotes/<name>/`.

## Подводные камни

- **HTTPS vs SSH URL.** CI с token, dev с SSH; разные remotes или credential helper.
- **Stale remote-tracking.** Fetch забыли; `git pull` surprise merge от старого origin/main.
- **Push default.** `git push` без upstream первый раз fails; `-u` при первом push.
- **Wrong remote delete.** `git push origin :branch` удаляет remote branch без confirm.
- **Mirror vs origin.** Deploy remote с force push ломает, если перепутать с origin.
