# How do you inspect changes in Git?

## In short

`git diff` compares trees: working tree vs index, index vs HEAD, arbitrary commits and branches. `git status` shows what changed; `git log -p` adds patches to history. Before committing I review staged (`--staged`) and unstaged hunks separately so debug code does not ship.

## How to answer in an interview

- `git diff` is unstaged; `git diff --staged` is the next commit.
- `git diff main...feature` shows feature changes from the fork point (three-dot diff for PRs).
- `git diff --stat` for scope; `git log --oneline --graph` for structure.
- Binary files produce terse diffs; I use `git show SHA -- path`.
- Honestly: giant terminal diffs are unreadable; I use platform UI or `delta/difftastic` for review.

## How it works

```bash
git status -sb

git diff
git diff --staged

git diff --stat origin/main...HEAD

git log -5 --oneline --decorate
git show HEAD --stat
```

Compare one file across commits:

```bash
git diff a1b2c3d e4f5g6h -- src/deploy.sh
```

Before commit:

```bash
git add -p
git diff --staged
git commit -m "fix(deploy): tighten health check timeout"
```

## Pitfalls

- **Two-dot vs three-dot diff.** `main..feature` and `main...feature` use different bases; PR reviews confuse them.
- **CRLF line endings.** Diffs noise on Windows; set `.gitattributes` eol=lf for code.
- **Submodules.** Regular diff shows only the submodule SHA, not inner content.
- **Ignored files.** `git diff` skips untracked files; use `git status -u` and explicit adds.
- **Merge commit diff.** `git diff parent1..merge` is subtle; use `git show -m` for merges.
