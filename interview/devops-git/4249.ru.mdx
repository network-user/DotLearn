# Какие стратегии ветвления применяют с Git?

## Суть

Выбор модели ветвления зависит от частоты релизов и размера команды. Git Flow подходит для версионных продуктов с hotfix, GitHub Flow для trunk + PR + CD, GitLab Flow добавляет environment branches, Trunk Based Development минимизирует долгоживущие ветки и требует сильного CI.

## Что ответить на собесе

- Git Flow: main + develop + feature/release/hotfix; релиз раз в спринт или реже.
- GitHub Flow: main всегда deployable, feature branch + PR, merge triggers deploy.
- GitLab Flow: production branch или environment branches (staging, pre-prod).
- Trunk based: короткие ветки (\<2 дней), feature flags, continuous integration.
- Честно: нет универсального best; беру модель под cadence деплоя и размер команды.

## Как это работает

Сравнение на практике:

```bash
git switch main
git pull origin main
git switch -c fix/cache-header
git commit -am "fix(cdn): set Cache-Control on static assets"
git push -u origin fix/cache-header
```

GitHub Flow ends at PR merge to main and pipeline deploy.

Git Flow release cut:

```bash
git switch develop
git switch -c release/2025-q2 develop
git switch main && git merge release/2025-q2 && git tag v2025.2.0
```

Trunk based keeps everyone close to main:

```bash
git switch main
git pull --rebase origin main
git switch -c chore/log-format
git commit -am "chore: structured json logs"
git push && open PR
```

## Подводные камни

- **Cargo cult Git Flow.** CD команда тратит недели на release branches без пользы.
- **Main not green.** GitHub Flow ломается, если merge без required checks.
- **Environment branch drift.** Staging branch живёт своей жизнью, prod merge surprise.
- **Long feature branches.** Любая модель страдает от веток старше недели.
- **No documented choice.** Каждый dev выбирает своё; onboarding хаос.
