# Какие команды Git меняют историю репозитория?

## Суть

Изменения в репозиторий вносят цепочка add → commit → push и операции слияния веток. Отмена возможна через reset (локально), revert (новый commit) или rebase (переписывание). Remote sync: fetch/pull/push. Каждая команда работает на своём слое: working tree, index, commits, remotes.

## Что ответить на собесе

- `git add` / `git restore --staged` управляют index; `git commit` фиксирует snapshot.
- `git push` / `git pull` синхронизируют с remote; pull = fetch + merge/rebase.
- `git merge` и `git rebase` интегрируют ветки по-разному (merge commit vs linear).
- `git reset --soft|--mixed|--hard` откатывает HEAD; `--hard` уничтожает uncommitted work.
- `git revert` безопасен для shared history; честно: reset --hard на pushed branch требует force и согласования.

## Как это работает

Типичный feature cycle:

```bash
git switch -c feat/webhook-signature
echo "verify_hmac()" >> handler.py
git add handler.py
git commit -m "feat(api): verify webhook HMAC"

git fetch origin
git rebase origin/main
git push -u origin feat/webhook-signature
```

Откат опубликованного commit:

```bash
git revert 9f3a2b1 --no-edit
git push origin main
```

Локальная правка последнего commit до push:

```bash
git add tests/test_handler.py
git commit --amend --no-edit
```

## Подводные камни

- **Hard reset after push.** Коллеги с вашим commit получают diverged history.
- **Revert merge commit.** Нужен `-m 1` для указания mainline parent.
- **Add . by mistake.** Секреты в index; использую `.gitignore` и `git add -p`.
- **Pull merge noise.** Ежедневный pull без rebase засоряет log.
- **Amend shared commit.** Меняет SHA уже pulled commit; только до push.
