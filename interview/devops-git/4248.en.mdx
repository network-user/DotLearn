# How do you mark a release version in Git?

## In short

You pin a release version with an annotated tag on the release commit: `git tag -a v2.1.0 -m "..."`. The tag points to an immutable SHA; CI builds artifacts from the tag, and `git checkout v2.1.0` restores exact state. Semver (`vMAJOR.MINOR.PATCH`) aligns with changelog and registry image tags.

## How to answer in an interview

- Lightweight tag (`git tag v1.0`) has no metadata; for releases I prefer annotated.
- Push tags separately: `git push origin v2.1.0` or `git push --tags`.
- Signed tag (`-s`) for supply chain audit when GPG is configured.
- Delete locally with `-d`, remote with `git push origin :refs/tags/v2.1.0`.
- Honestly: a tag on the wrong commit without retag policy confuses consumers; use an immutable release branch or GitHub Release assets.

## How it works

Semver release flow:

```bash
git switch main
git pull origin main

git tag -a v2.1.0 -m "Release 2.1.0: payment retries, metrics dashboard"
git push origin v2.1.0

git show v2.1.0 --no-patch
git checkout v2.1.0
```

CI trigger on tag:

```bash
git tag -l 'v*' --sort=-v:refname | head -5
```

Annotated tags store tagger, date, and message; lightweight tags are just a ref to a commit.

## Pitfalls

- **Unpushed tags.** Local tag without push; CI never sees the release.
- **Moving tags.** Force-updating a tag breaks reproducible builds; I forbid it in policy.
- **Tag on merge commit.** Unclear which parent to deploy; tag the commit after merge to main.
- **v prefix inconsistency.** `1.0.0` vs `v1.0.0`; automation regex breaks.
- **Detached HEAD deploy.** Checkout tag without branch; for hotfix I create a branch from the tag.
