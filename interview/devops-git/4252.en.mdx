# What is a remote in Git?

## In short

A remote is a named reference to another Git repository, usually on a server (GitHub, GitLab, self-hosted). A local clone holds commits; remotes enable fetch and push. After clone, Git creates `origin` pointing at the source URL by default.

## How to answer in an interview

- `git remote -v` shows fetch and push URLs for each remote.
- `git remote add upstream <url>` for fork workflow.
- `git fetch origin` updates remote-tracking branches without merging into the working branch.
- `git push origin feature/x` publishes local commits.
- Honestly: multiple remotes without naming convention confuse people; I document origin/upstream/deploy.

## How it works

Setup and sync:

```bash
git clone git@github.com:team/platform.git
cd platform
git remote -v

git remote add staging git@deploy.internal:platform.git
git fetch staging

git switch main
git pull origin main

git switch -c fix/metrics-label
git commit -am "fix(obs): correct histogram bucket"
git push -u origin fix/metrics-label
```

Inspect remote branches:

```bash
git branch -r
git log origin/main..HEAD --oneline
```

Fetch does not copy the whole repo each time; it updates refs under `refs/remotes/<name>/`.

## Pitfalls

- **HTTPS vs SSH URL.** CI with token, dev with SSH; different remotes or credential helper.
- **Stale remote-tracking.** Forgot fetch; `git pull` surprise merge from old origin/main.
- **Push default.** First `git push` without upstream fails; use `-u` on first push.
- **Wrong remote delete.** `git push origin :branch` deletes remote branch without confirm.
- **Mirror vs origin.** Deploy remote with force push breaks if confused with origin.
