# How do git push and git pull differ?

## In short

`git push` publishes local commits to the remote and moves the remote branch ref. `git pull` fetches remote commits and integrates them into the local branch. Push is outbound sync, pull is inbound. Both use tracking branches and need aligned history (non-fast-forward push is rejected without force).

## How to answer in an interview

- Push after review: `git push origin feature/login-oauth`.
- Pull or rebase before push to avoid rejected non-fast-forward updates.
- `git push -u origin branch` sets upstream for later pull/push without arguments.
- Protected branches on GitHub/GitLab block direct push; land via PR merge only.
- Honestly: `--force-with-lease` only on personal branches; shared main forbids force by policy.

## How it works

Publish a feature:

```bash
git fetch origin
git rebase origin/main
git push -u origin feature/api-cache
```

Refresh local main:

```bash
git switch main
git pull --rebase origin main
```

When push is rejected:

```bash
git pull --rebase origin main
git push origin feature/api-cache
```

## Pitfalls

- **Push without fetch.** Non-fast-forward rejection on busy branches; always rebase onto current remote.
- **Force push.** Rewrites teammates' history; lease only, on owned branches only.
- **Tags push separately.** Release tags will not travel with a normal branch push.
- **Large files without LFS.** Push fails or bloats the repo; pre-receive hooks catch binaries.
- **CI credentials.** Pipeline push uses scoped tokens with minimal permissions.
