# Как разрешить merge-конфликт в Git?

## Суть

Конфликт возникает, когда Git не может автоматически совместить изменения в одних и тех же строках. Процесс: воспроизвести merge/rebase, открыть файлы с маркерами `<<<<<<<`, выбрать итоговый код, `git add`, завершить merge commit или `rebase --continue`. Цель: сохранить намерение обеих веток и прогнать тесты до push.

## Что ответить на собесе

- `git status` показывает unmerged paths; `git diff --merge` помогает увидеть обе стороны.
- Удаляю conflict markers полностью; оставленный `<<<<<<<` ломает сборку.
- Для сложных файлов согласую с автором другой ветки, не гадаю по середине.
- После resolve: `git add` каждый файл, `git merge --continue` или `git commit` при merge.
- Честно: частые конфликты сигнал о слишком долгоживущей ветке; дроблю PR и чаще rebase на main.

## Как это работает

```bash
git switch feature/billing
git fetch origin
git rebase origin/main
```

В файле `src/config.yaml` вручную объединяю блоки, затем:

```bash
git add src/config.yaml
git rebase --continue
```

При merge вместо rebase:

```bash
git merge origin/main
git add src/config.yaml
git commit -m "merge main into feature/billing"
```

Проверка перед push:

```bash
git diff origin/main...HEAD
pnpm test
```

## Подводные камни

- **Abort забыли.** `git merge --abort` / `git rebase --abort` возвращает в чистое состояние; не оставляю полу-мерж.
- **rerere не включён.** Повторяющиеся конфликты решаю заново; `rerere.enabled` экономит время на long rebase.
- **Binary конфликты.** Git не мержит картинки; выбираю одну версию или внешний merge tool.
- **Lockfile конфликты.** `package-lock.json` руками не правлю; пересобираю lock после merge package.json.
- **Тесты после auto-resolve.** IDE merge иногда теряет блок; CI обязателен до merge PR.
