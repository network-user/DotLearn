# Where are SSH and GPG keys stored in GitHub?

## In short

Public SSH and GPG keys live in account Settings → SSH and GPG keys. Private keys never upload to GitHub; they stay on the client (`~/.ssh`) or in CI secret stores. Deploy keys bind to one repository (Settings → Deploy keys). Organization SSH keys gate access for automation bots.

## How to answer in an interview

- User SSH keys for git@github.com clone/push from developers.
- Deploy keys read-only or write for a single repo in CI; no reuse across projects.
- GPG public keys in the profile for verified commits; private keys in the local gpg agent.
- GitHub Actions secrets are separate: Settings → Secrets and variables (not the SSH keys UI).
- Honestly: a write deploy key on a prod repo is risky; prefer OIDC or fine-grained PATs.

## How it works

Generate SSH and register the public key:

```bash
ssh-keygen -t ed25519 -C "alex@company.example" -f ~/.ssh/id_ed25519_github

cat ~/.ssh/id_ed25519_github.pub
ssh -T git@github.com
```

GPG for signed commits:

```bash
gpg --full-generate-key
gpg --list-secret-keys --keyid-format long
gpg --armor --export ABCD1234EFGH5678

git config --global user.signingkey ABCD1234EFGH5678
git config --global commit.gpgsign true
```

A repo deploy key grants access only to that repository; org Action secrets sit under Organization → Settings → Secrets.

## Pitfalls

- **Private key in the repo.** Scanning catches it but rotation hurts; block with pre-commit hooks.
- **One SSH key everywhere.** Compromise exposes all user repos; use per-machine keys.
- **Write deploy key on fork PRs.** CI RCE plus write key equals supply chain risk; default read-only.
- **GPG subkey expiry.** Signing suddenly fails; calendar reminders for rotation.
- **Passphrase-less CI keys.** Store in vault with audit; never plaintext in workflow yaml.
