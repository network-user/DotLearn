# Что делает git cherry-pick?

## Суть

`git cherry-pick <commit>` переносит patch выбранного коммита (или диапазона) на текущую ветку, создавая новый commit с новым SHA. Не тащит всю ветку как merge. Типично: hotfix с main на release branch, выборочный backport, восстановление потерянного коммита после rebase.

## Что ответить на собесе

- Перед pick: `git switch target-branch`, убедиться что working tree clean.
- Несколько коммитов: `git cherry-pick A..B` (exclusive start) или `A^..B`.
- Конфликт: resolve, `git add`, `git cherry-pick --continue`; откат `cherry-pick --abort`.
- `-x` добавляет ссылку на исходный commit в message для аудита backport.
- Честно: серия зависимых коммитов cherry-pick по одному хрупка; иногда merge или rebase всей ветки проще.

## Как это работает

Hotfix backport:

```bash
git switch release/2.4
git pull origin release/2.4

git log origin/main --oneline -5
git cherry-pick -x 8f3a2b1

pnpm test
git push origin release/2.4
```

Диапазон без merge commit:

```bash
git cherry-pick f100000^..f100003
```

При пустом patch (уже применён):

```bash
git cherry-pick --skip
```

## Подводные камни

- **Duplicate commits.** Повторный pick тех же изменений плодит дубли; проверяю `git log --cherry`.
- **Merge commits.** Cherry-pick merge SHA требует `-m 1`; иначе неоднозначность parents.
- **Контекстный конфликт.** Код вокруг изменился; тесты после resolve обязательны.
- **Signed commits.** Новый commit теряет original signature; re-sign если policy требует.
- **Release drift.** Backport без теста на target branch ломает стабильную линию.
