# Как снизить риск merge-конфликтов?

## Суть

Конфликты растут при долгоживущих ветках и правках одних файлов. Профилактика: короткие feature branches, частый rebase на main, мелкие PR, договорённости о code ownership, автоформаттер в CI. Полностью исключить конфликты нельзя, но частоту и blast radius снижаем процессом.

## Что ответить на собесе

- Перед работой `git fetch` и rebase на актуальный main.
- Дроблю задачи: отдельные PR для refactor и для feature.
- CODEOWNERS и слойная архитектура уменьшают правки в одном файле десятью людьми.
- Trunk-based с feature flags вместо месячных release branches.
- Честно: «избежать конфликтов» нереалистично; цель сделать их редкими и дешёвыми в resolve.

## Как это работает

Ежедневный цикл:

```bash
git fetch origin
git rebase origin/main
```

Новая ветка от свежего main:

```bash
git switch main
git pull --rebase origin main
git switch -c feature/metrics-export
```

Перед merge PR:

```bash
git fetch origin
git rebase origin/main
pnpm test
git push --force-with-lease
```

## Подводные камни

- **Гигантские PR.** 3k строк гарантируют конфликт и долгий review; лимит 400 LOC в команде помогает.
- **Reformat всего репо.** Black/prettier одним коммитом ломает все открытые ветки; отдельный chore PR.
- **Generated files в git.** Lockfiles и protobuf генерят конфликты; policy кто регенерирует.
- **Отсутствие integration branch smoke.** Конфликт всплывает только на merge в main; CI на rebase main в PR обязателен.
- **Секретное долгое ответвление.** Release branch живёт недели; merge back в main болезненный.
