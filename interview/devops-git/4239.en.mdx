# How do you change Git global settings?

## In short

`git config --global key value` writes to `~/.gitconfig` for all user repos unless `.git/config` overrides locally. Levels: `--system`, `--global`, `--local` (priority increases toward local). Common keys: `user.name`, `user.email`, `init.defaultBranch`, `pull.rebase`, `core.editor`.

## How to answer in an interview

- Commit identity via global `user.name` and `user.email`; work repos override with `--local` corporate email.
- `git config --global --list` and `git config --show-origin --get user.email` audit the source.
- `includeIf` in gitconfig for path-based profiles (`~/work/` vs `~/oss/`).
- I never store secrets in gitconfig; credential helpers are chosen deliberately.
- Honestly: a personal global `user.email` leaks into company repos; I set local overrides.

## How it works

```bash
git config --global user.name "Alex Devops"
git config --global user.email "alex@company.example"
git config --global init.defaultBranch main
git config --global pull.rebase true

git config --global --list
git config --show-origin --get-regexp '^user\.'
```

Local override in a project:

```bash
cd ~/work/platform
git config --local user.email "alex@company.example"
```

Snippet of `~/.gitconfig`:

```ini
[user]
    name = Alex Devops
    email = alex.personal@example.com

[pull]
    rebase = true
```

## Pitfalls

- **Wrong email in commits.** `git commit --amend` fixes only the tip; history needs filter-repo.
- **credential.helper store.** Plaintext passwords; prefer manager/cache with TTL.
- **Global insteadOf URL.** Breaks clones to other hosts; test on a sandbox remote.
- **core.autocrlf on Linux.** Global true damages shell scripts; use `input` or attributes.
- **includeIf typo.** Silently ignored; verify with `git config --list --show-origin`.
