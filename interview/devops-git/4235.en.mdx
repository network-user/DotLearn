# What are Git tags for?

## In short

A tag is a named reference to a commit, usually for releases (`v2.4.1`). A lightweight tag is a simple pointer; an annotated tag stores message, author, date, and can be GPG-signed. CI builds artifacts from tags; semver tags trigger production deploys.

## How to answer in an interview

- Annotated tags for releases: `git tag -a v1.2.0 -m "..."` and `git push origin v1.2.0`.
- Lightweight tags suffice for temporary markers but not in my release process.
- `git show v1.2.0` shows the commit and tag message; I tie changelogs to tag ranges.
- Signed tags (`-s`) for supply chain in regulated environments.
- Honestly: tagging the wrong commit in prod is costly; CI must verify SHA before publish.

## How it works

```bash
git switch main
git pull --rebase

git tag -a v2.4.1 -m "Release 2.4.1: payment retry fix"
git push origin v2.4.1

git tag -l 'v2.*'
git show v2.4.1
```

Commits between releases:

```bash
git log v2.4.0..v2.4.1 --oneline
```

Pipeline checkout by tag:

```bash
git fetch origin tag v2.4.1 --no-tags
git checkout v2.4.1
```

## Pitfalls

- **Tags do not ride branch push.** `git push` without `--tags` leaves tags local; release jobs miss the version.
- **Moving tags.** Rewriting a remote tag (`--force`) breaks artifact reproducibility; enforce immutable tags.
- **Tag on merge commit.** Unclear release contents; I tag after fast-forwarding main.
- **Case sensitivity.** `v1.0` and `V1.0` are different refs; document naming conventions.
- **Detached HEAD deploy.** Tag checkout is fine for builds; hotfixes branch off, not from the tag directly.
