# Чем git push отличается от git pull?

## Суть

`git push` публикует локальные коммиты на remote и обновляет remote branch ref. `git pull` забирает remote коммиты и интегрирует в локальную ветку. Push это исходящая синхронизация, pull входящая. Оба работают с tracking branches и требуют согласованной истории (non-fast-forward push блокируется без force).

## Что ответить на собесе

- Push после review: `git push origin feature/login-oauth`.
- Pull/rebase перед push, чтобы не упираться в rejected non-fast-forward.
- `git push -u origin branch` задаёт upstream для будущих pull/push без аргументов.
- Protected branches на GitHub/GitLab блокируют прямой push; только через PR merge.
- Честно: `--force-with-lease` только на личных ветках; на shared main force запрещён политикой.

## Как это работает

Публикация feature:

```bash
git fetch origin
git rebase origin/main
git push -u origin feature/api-cache
```

Обновление локальной main:

```bash
git switch main
git pull --rebase origin main
```

Если push отклонён:

```bash
git pull --rebase origin main
git push origin feature/api-cache
```

## Подводные камни

- **Push без fetch.** Отклонение non-fast-forward на busy branch; всегда rebase на актуальный remote.
- **Force push.** Переписывает историю коллег; только lease и только на своих ветках.
- **Push tags отдельно.** Релизный тег не уедет с обычным push branch.
- **Large files без LFS.** Push падает или раздувает repo; pre-receive hook ловит бинарники.
- **Credential helper в CI.** Push из pipeline только через scoped token с минимальными правами.
