# What does git cherry-pick do?

## In short

`git cherry-pick <commit>` applies the chosen commit's patch (or a range) onto the current branch, creating a new commit with a new SHA. It does not pull the whole branch like merge. Typical uses: hotfix from main to a release branch, selective backport, recovering a commit lost after rebase.

## How to answer in an interview

- Before pick: `git switch target-branch` with a clean working tree.
- Multiple commits: `git cherry-pick A..B` (exclusive start) or `A^..B`.
- On conflict: resolve, `git add`, `git cherry-pick --continue`; abort with `cherry-pick --abort`.
- `-x` appends the source commit reference in the message for backport audit.
- Honestly: cherry-picking a chain of dependent commits is fragile; sometimes merging or rebasing the branch is simpler.

## How it works

Hotfix backport:

```bash
git switch release/2.4
git pull origin release/2.4

git log origin/main --oneline -5
git cherry-pick -x 8f3a2b1

pnpm test
git push origin release/2.4
```

Range without a merge commit:

```bash
git cherry-pick f100000^..f100003
```

Empty patch (already applied):

```bash
git cherry-pick --skip
```

## Pitfalls

- **Duplicate commits.** Re-picking the same change creates duplicates; check `git log --cherry`.
- **Merge commits.** Picking a merge SHA needs `-m 1`; otherwise parent ambiguity fails.
- **Context conflicts.** Surrounding code moved; tests after resolve are mandatory.
- **Signed commits.** The new commit drops the original signature; re-sign if policy requires.
- **Release drift.** Backport without tests on the target branch breaks the stable line.
