# Why use git rebase?

## In short

Rebase moves your commits onto the current tip of a base branch by replaying them on top of new changes. History stays linear without "Merge branch main" merge commits. I use it before merging a PR or to refresh a feature branch with latest main.

## How to answer in an interview

- `git switch feature && git rebase main` replays my commits on top of main.
- Conflicts: fix, `git add`, `git rebase --continue`; abort with `git rebase --abort`.
- Interactive: `git rebase -i HEAD~3` to squash or fixup before merge.
- I do not rebase public branches others already pulled; only local branches or after agreeing on force-push.
- Honestly: rebase changes SHAs; without `-x` or reflog, tracing the original in audit is harder.

## How it works

Refreshing feature before PR:

```bash
git switch feature/payment-webhook
git fetch origin
git rebase origin/main

pnpm test

git push --force-with-lease origin feature/payment-webhook
```

Interactive squash:

```bash
git rebase -i origin/main
pick a1b2c3d Add webhook handler
squash d4e5f6a Fix typo in tests
pick g7h8i9j Add metrics
```

After rebase, feature contains all main commits and your changes follow in a straight line.

## Pitfalls

- **Shared branch rewrite.** Force-push after rebase breaks colleagues with local commits on old history.
- **Lost commits.** Mistakes during `--continue` without reflog knowledge; I keep a backup branch.
- **Conflict chains.** A long feature with many commits triggers conflict at each step; squash early.
- **Merge vs rebase policy.** A team without a rule mixes both; I document it in CONTRIBUTING.
- **CI on force-push.** Pipeline may not rerun; I verify webhook after lease push.
