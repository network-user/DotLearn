# How do you reduce merge conflict risk?

## In short

Conflicts grow with long-lived branches and edits to the same files. Prevention: short feature branches, frequent rebase onto main, small PRs, code ownership agreements, formatters in CI. You cannot eliminate conflicts, but process lowers frequency and blast radius.

## How to answer in an interview

- Before work: `git fetch` and rebase onto current main.
- I split tasks: separate PRs for refactors and features.
- CODEOWNERS and layered architecture reduce ten people editing one file.
- Trunk-based development with feature flags beats month-long release branches.
- Honestly: «avoid conflicts» is unrealistic; the goal is rare, cheap resolutions.

## How it works

Daily loop:

```bash
git fetch origin
git rebase origin/main
```

New branch from fresh main:

```bash
git switch main
git pull --rebase origin main
git switch -c feature/metrics-export
```

Before merging a PR:

```bash
git fetch origin
git rebase origin/main
pnpm test
git push --force-with-lease
```

## Pitfalls

- **Huge PRs.** 3k lines guarantee conflict and slow review; a 400 LOC team limit helps.
- **Repo-wide reformat.** One-shot Black/prettier breaks every open branch; ship a dedicated chore PR.
- **Generated files in git.** Lockfiles and protobuf outputs conflict often; define who regenerates.
- **No integration smoke.** Conflict appears only at merge; CI must rebase onto main inside the PR.
- **Long-lived secret branch.** Week-old release branches make merge-back to main painful.
