# How do you resolve a Git merge conflict?

## In short

Conflicts happen when Git cannot auto-merge the same lines. Flow: trigger merge/rebase, edit files with `<<<<<<<` markers, pick the final code, `git add`, then finish with a merge commit or `rebase --continue`. Goal: preserve both branches' intent and run tests before push.

## How to answer in an interview

- `git status` lists unmerged paths; `git diff --merge` shows both sides.
- I remove conflict markers completely; leftover `<<<<<<<` breaks the build.
- On complex files I sync with the other branch author instead of guessing.
- After resolve: `git add` each file, `git merge --continue` or `git commit` for merges.
- Honestly: frequent conflicts mean the branch lived too long; I split PRs and rebase onto main often.

## How it works

```bash
git switch feature/billing
git fetch origin
git rebase origin/main
```

Manually merge blocks in `src/config.yaml`, then:

```bash
git add src/config.yaml
git rebase --continue
```

For merge instead of rebase:

```bash
git merge origin/main
git add src/config.yaml
git commit -m "merge main into feature/billing"
```

Pre-push check:

```bash
git diff origin/main...HEAD
pnpm test
```

## Pitfalls

- **Forgot abort.** `git merge --abort` / `git rebase --abort` restores a clean state; do not leave half-merged trees.
- **rerere off.** Repeated conflicts get solved again; `rerere.enabled` saves time on long rebases.
- **Binary conflicts.** Git cannot merge images; pick one side or use an external merge tool.
- **Lockfile conflicts.** Do not hand-edit `package-lock.json`; regenerate the lock after merging package.json.
- **Tests after IDE merge.** Visual merge tools drop hunks; CI must run before merging the PR.
