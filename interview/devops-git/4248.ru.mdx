# Как в Git помечают версию релиза?

## Суть

Версию фиксируют annotated tag на commit релиза: `git tag -a v2.1.0 -m "..."`. Tag указывает на immutable SHA; CI собирает артефакт по tag, а `git checkout v2.1.0` восстанавливает точное состояние. Semver (`vMAJOR.MINOR.PATCH`) согласуют с changelog и registry image tag.

## Что ответить на собесе

- Lightweight tag (`git tag v1.0`) без метаданных; для релизов предпочитаю annotated.
- Push tags отдельно: `git push origin v2.1.0` или `git push --tags`.
- Signed tag (`-s`) для supply chain audit, если GPG настроен.
- Удаление: локально `-d`, remote `git push origin :refs/tags/v2.1.0`.
- Честно: tag на wrong commit без retag policy путает потребителей; нужен immutable release branch или GitHub Release asset.

## Как это работает

Semver release flow:

```bash
git switch main
git pull origin main

git tag -a v2.1.0 -m "Release 2.1.0: payment retries, metrics dashboard"
git push origin v2.1.0

git show v2.1.0 --no-patch
git checkout v2.1.0
```

CI trigger on tag:

```bash
git tag -l 'v*' --sort=-v:refname | head -5
```

Annotated tag хранит tagger, date и message; lightweight это просто ref на commit.

## Подводные камни

- **Unpushed tags.** Локальный tag без push; CI не видит релиз.
- **Moving tags.** Force-update tag ломает reproducible builds; запрещаю в policy.
- **Tag on merge commit.** Непонятно, какой parent деплоить; tag на commit после merge в main.
- **v prefix inconsistency.** `1.0.0` vs `v1.0.0`; automation regex ломается.
- **Detached HEAD deploy.** Checkout tag без branch; для hotfix создаю branch от tag.
