# Зачем применяют git rebase?

## Суть

Rebase переносит ваши коммиты на актуальный tip базовой ветки, переписывая их поверх новых изменений. История остаётся линейной без merge-коммитов «Merge branch main». Использую перед merge PR или для обновления feature-ветки свежим main.

## Что ответить на собесе

- `git switch feature && git rebase main` ставит мои commits поверх main.
- Конфликты: правлю, `git add`, `git rebase --continue`; отмена `git rebase --abort`.
- Interactive: `git rebase -i HEAD~3` для squash/fixup перед merge.
- Не rebase публичные ветки, которые уже pull'или другие; только локальные или после согласования force-push.
- Честно: rebase меняет SHA; без `-x` или reflog сложнее искать оригинал в audit.

## Как это работает

Обновление feature перед PR:

```bash
git switch feature/payment-webhook
git fetch origin
git rebase origin/main

pnpm test

git push --force-with-lease origin feature/payment-webhook
```

Interactive squash:

```bash
git rebase -i origin/main
pick a1b2c3d Add webhook handler
squash d4e5f6a Fix typo in tests
pick g7h8i9j Add metrics
```

После rebase feature содержит все commits main, ваши изменения идут следом линейно.

## Подводные камни

- **Shared branch rewrite.** Force-push после rebase ломает коллег с локальными commits на старой истории.
- **Lost commits.** Ошибка при `--continue` без понимания reflog; держу backup branch.
- **Conflict chains.** Длинная feature с десятками commits даёт конфликт на каждом шаге; squash заранее.
- **Merge vs rebase policy.** Команда без правила смешивает оба подхода; фиксирую в CONTRIBUTING.
- **CI on force-push.** Pipeline может не перезапуститься; проверяю webhook после lease push.
