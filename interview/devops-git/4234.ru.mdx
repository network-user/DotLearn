# Что делает git pull?

## Суть

`git pull` это сокращение для `git fetch` + интеграция в текущую ветку (обычно merge, с `--rebase` rebase). Команда подтягивает коммиты с remote и встраивает их в локальную историю. На CI и в feature-ветках чаще явно делаю fetch + rebase для линейной истории.

## Что ответить на собесе

- `git pull origin main` обновляет tracking branch и сливает в HEAD.
- `git pull --rebase` ставит мои коммиты поверх свежего remote без merge commit.
- Перед pull коммичу или stash незакоммиченные изменения.
- При конфликте pull останавливается; решаю markers, `git add`, `git rebase --continue` или merge commit.
- Честно: слепой pull на shared main в пиковые часы создаёт merge commits; предпочитаю fetch + review + rebase.

## Как это работает

```bash
git fetch origin main

git log --oneline HEAD..origin/main

git pull --rebase origin main
```

Если настроен upstream:

```bash
git branch -vv
git pull --rebase
```

После успешного pull локальная ветка указывает на объединённую историю; `origin/main` ref обновлён на этапе fetch.

## Подводные камни

- **Pull без upstream.** Ветка без tracking branch требует явного `origin name`; иначе ошибка.
- **Merge commit шум.** Дефолтный pull на long-lived branch плодит «Merge branch main»; команда договаривается о rebase policy.
- **Конфликт при dirty tree.** Незакоммиченные правки в тех же строках блокируют rebase; stash или WIP commit.
- **Force push на remote.** После чужого force push pull может создать дубликаты; нужен `git fetch --prune` и reset/rebase по согласованию.
- **Submodules.** `git pull` не обновляет submodule pointers автоматически; отдельно `git submodule update`.
