# Чем git fetch отличается от git pull?

## Суть

`git fetch` скачивает коммиты и обновляет remote-tracking ветки (`origin/main`), не трогая рабочую директорию и текущую ветку. `git pull` это fetch плюс merge (или rebase с `--rebase`) в текущую ветку, сразу меняя файлы в working tree.

## Что ответить на собесе

- Fetch безопасен для review: смотрю `git log HEAD..origin/main` и diff до интеграции.
- Pull ускоряет sync, но может создать merge commit или конфликт без подготовки.
- В CI/CD и prod deploy обычно явный fetch + checkout sha/tag, не blind pull на сервере.
- Честно добавляю: в команде предпочитаю `fetch` + `rebase`/`merge` осознанно, а не default pull, чтобы контролировать историю.

## Как это работает

Remote refs живут отдельно: `origin/feature` обновляется fetch'ем. Локальная `feature` двигается только merge/rebase/checkout.

```bash
git fetch origin
git status -sb
git log --oneline HEAD..origin/main
git diff HEAD..origin/main --stat

git pull --ff-only origin main

git fetch origin
git rebase origin/main
```

Первый блок только подтягивает объекты. `HEAD..origin/main` показывает, что прилетело с remote. `pull --ff-only` откажется, если нужен merge commit (защита linear history). Явный `fetch` + `rebase` даёт тот же результат, но с паузой на review.

`git pull origin main` эквивалентен `git fetch origin main` затем `git merge FETCH_HEAD` (если не настроен rebase).

## Подводные камни

- **Pull с локальными uncommitted changes.** Merge может встать поверх грязного tree; stash или commit сначала.
- **Merge commit spam.** Частый pull без rebase засоряет историю.
- **Force-push на remote.** Fetch покажет расхождение, pull попытается merge diverged histories.
- **Detached CI clone.** Pull на deploy-хосте без pinned ref опасен; лучше fetch + checkout tag.
- **Submodules.** Обычный pull не обновит submodule pointers; нужен `git submodule update`.
