# Which Git commands modify repository history?

## In short

Repository changes flow through add, commit, push, and branch merge operations. Undo uses reset (local), revert (new commit), or rebase (rewrite). Remote sync uses fetch, pull, and push. Each command targets a layer: working tree, index, commits, or remotes.

## How to answer in an interview

- `git add` and `git restore --staged` manage the index; `git commit` records a snapshot.
- `git push` and `git pull` sync with remote; pull equals fetch plus merge or rebase.
- `git merge` and `git rebase` integrate branches differently (merge commit vs linear).
- `git reset --soft|--mixed|--hard` moves HEAD; `--hard` destroys uncommitted work.
- `git revert` is safe for shared history; honestly: reset --hard on a pushed branch needs force and team agreement.

## How it works

Typical feature cycle:

```bash
git switch -c feat/webhook-signature
echo "verify_hmac()" >> handler.py
git add handler.py
git commit -m "feat(api): verify webhook HMAC"

git fetch origin
git rebase origin/main
git push -u origin feat/webhook-signature
```

Revert a published commit:

```bash
git revert 9f3a2b1 --no-edit
git push origin main
```

Fix last commit locally before push:

```bash
git add tests/test_handler.py
git commit --amend --no-edit
```

## Pitfalls

- **Hard reset after push.** Colleagues with your commit get diverged history.
- **Revert merge commit.** Requires `-m 1` to pick the mainline parent.
- **Add . by mistake.** Secrets in index; I use `.gitignore` and `git add -p`.
- **Pull merge noise.** Daily pull without rebase clutters the log.
- **Amend shared commit.** Changes SHA of an already pulled commit; only before push.
