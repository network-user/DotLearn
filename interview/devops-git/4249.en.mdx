# What branching strategies are used with Git?

## In short

Branching model choice depends on release frequency and team size. Git Flow fits versioned products with hotfixes, GitHub Flow fits trunk plus PR plus CD, GitLab Flow adds environment branches, and Trunk Based Development minimizes long-lived branches and needs strong CI.

## How to answer in an interview

- Git Flow: main plus develop plus feature/release/hotfix; release once per sprint or less.
- GitHub Flow: main always deployable, feature branch plus PR, merge triggers deploy.
- GitLab Flow: production branch or environment branches (staging, pre-prod).
- Trunk based: short branches (under two days), feature flags, continuous integration.
- Honestly: there is no universal best; I pick the model for deploy cadence and team size.

## How it works

Practical comparison:

```bash
git switch main
git pull origin main
git switch -c fix/cache-header
git commit -am "fix(cdn): set Cache-Control on static assets"
git push -u origin fix/cache-header
```

GitHub Flow ends at PR merge to main and pipeline deploy.

Git Flow release cut:

```bash
git switch develop
git switch -c release/2025-q2 develop
git switch main && git merge release/2025-q2 && git tag v2025.2.0
```

Trunk based keeps everyone close to main:

```bash
git switch main
git pull --rebase origin main
git switch -c chore/log-format
git commit -am "chore: structured json logs"
git push && open PR
```

## Pitfalls

- **Cargo cult Git Flow.** A CD team spends weeks on release branches with no benefit.
- **Main not green.** GitHub Flow fails when merge happens without required checks.
- **Environment branch drift.** Staging branch lives its own life; prod merge surprises.
- **Long feature branches.** Every model suffers from branches older than a week.
- **No documented choice.** Each dev picks their own; onboarding becomes chaos.
