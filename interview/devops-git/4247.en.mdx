# Why does GitFlow use multiple branch types?

## In short

GitFlow separates stable release code, integration development, and urgent fixes across dedicated branches. `main` holds only production-ready tags, `develop` integrates features, and `feature/*`, `release/*`, and `hotfix/*` isolate work and reduce the risk of accidentally deploying raw code.

## How to answer in an interview

- Feature branches from develop and merge back after review.
- Release branch stabilizes a version: bugfixes only, no new features.
- Hotfix from main reaches prod and backports to develop.
- Tags on main mark semver releases for changelog and artifacts.
- Honestly: for continuous deployment GitFlow is often heavy; trunk-based or GitHub Flow is simpler.

## How it works

Feature and hotfix lifecycle:

```bash
git switch develop
git switch -c feature/rate-limiter
git commit -am "feat(api): add token bucket"
git switch develop
git merge --no-ff feature/rate-limiter
git branch -d feature/rate-limiter

git switch -c release/3.2.0 develop
git commit -am "chore: bump version 3.2.0"
git switch main
git merge --no-ff release/3.2.0
git tag -a v3.2.0 -m "Release 3.2.0"
git switch develop
git merge release/3.2.0

git switch main
git switch -c hotfix/3.2.1
git commit -am "fix(auth): null session edge case"
git switch main && git merge hotfix/3.2.1 && git tag v3.2.1
git switch develop && git merge hotfix/3.2.1
```

Each branch type limits which commits can reach production.

## Pitfalls

- **Long-lived develop.** Falls behind main for months; release merge becomes painful.
- **Forgotten backport.** Hotfix only on main; develop breaks again on the next release.
- **Feature flags vs branches.** Keeping feature/* for months is cheaper with a flag in trunk.
- **Naming chaos.** `feature/JIRA-123` without policy; grep and automation suffer.
- **Double merge conflicts.** Release and hotfix touch the same files at once.
