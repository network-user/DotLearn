# How does fork differ from git pull?

## In short

Fork creates an independent copy of someone else's repository on the platform (GitHub, GitLab) under your account. `git pull` updates an existing local clone by fetching commits from a remote and merging them into the current branch. These operate at different levels: fork is about hosting and permissions; pull is about synchronizing history.

## How to answer in an interview

- I use fork for open source: work in my own remote, open PR to upstream without write access.
- Pull: `git pull origin main` equals fetch plus merge (or rebase with `--rebase`).
- After fork I clone my fork and add an upstream remote to sync with the original.
- Fork does not copy locally by itself; I still need `git clone`.
- Honestly: in one org with a shared repo, fork is rarely needed; pull/fetch is enough.

## How it works

Fork on GitHub and working with upstream:

```bash
git clone git@github.com:myuser/upstream-project.git
cd upstream-project

git remote add upstream git@github.com:org/upstream-project.git
git fetch upstream
git switch main
git merge upstream/main

git switch -c feature/oauth-callback
git commit -am "fix(auth): handle expired state"
git push origin feature/oauth-callback
```

Pull in daily work:

```bash
git switch main
git pull --rebase origin main
```

Fork lives on the forge side (new bare repo plus PR flow). Pull works only if the remote is configured and you have access.

## Pitfalls

- **Fork divergence.** Without syncing upstream for a long time, I get a huge merge when opening a PR.
- **Confusing remotes.** After fork, `origin` points to my fork, not upstream; I confuse push targets.
- **Pull merge commits.** Frequent `git pull` without rebase clutters history with merge commits.
- **Fork != branch.** Beginners think fork is a branch; I clarify it is a hosting-level concept.
- **Deleted upstream.** If upstream is deleted, the fork remains but PR and sync break.
