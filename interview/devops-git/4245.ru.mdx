# Чем fork отличается от git pull?

## Суть

Fork создаёт независимую копию чужого репозитория на платформе (GitHub, GitLab) под вашим аккаунтом. `git pull` обновляет уже существующий локальный клон, подтягивая коммиты с remote и сливая их с текущей веткой. Это разные уровни: fork про hosting и права, pull про синхронизацию истории.

## Что ответить на собесе

- Fork нужен для open source: работаю в своём remote, PR в upstream без write access.
- Pull: `git pull origin main` = fetch + merge (или rebase с `--rebase`).
- После fork клонирую свой fork, добавляю upstream remote для sync с оригиналом.
- Fork не копирует локально сам по себе; всё равно нужен `git clone`.
- Честно: в одной org с shared repo fork редко нужен; pull/fetch достаточно.

## Как это работает

Fork на GitHub и работа с upstream:

```bash
git clone git@github.com:myuser/upstream-project.git
cd upstream-project

git remote add upstream git@github.com:org/upstream-project.git
git fetch upstream
git switch main
git merge upstream/main

git switch -c feature/oauth-callback
git commit -am "fix(auth): handle expired state"
git push origin feature/oauth-callback
```

Pull в ежедневной работе:

```bash
git switch main
git pull --rebase origin main
```

Fork живёт на стороне forge (новый bare repo + redirect PR). Pull работает только если remote уже настроен и у вас есть доступ.

## Подводные камни

- **Fork divergence.** Долго не синхронизируя upstream, получаю огромный merge при PR.
- **Confusing remotes.** `origin` после fork указывает на свой fork, не на upstream; путаю push target.
- **Pull merge commits.** Частый `git pull` без rebase засоряет историю merge-коммитами.
- **Fork != branch.** Новички думают, что fork это ветка; объясняю уровень hosting.
- **Deleted upstream.** Если upstream удалён, fork остаётся, но PR и sync ломаются.
