# How does git fetch differ from git pull?

## In short

`git fetch` downloads commits and updates remote-tracking branches (`origin/main`) without touching the working directory or current branch. `git pull` is fetch plus merge (or rebase with `--rebase`) into the current branch, immediately changing working tree files.

## How to answer in an interview

- Fetch is safe for review: I inspect `git log HEAD..origin/main` and diff before integrating.
- Pull speeds sync but can create a merge commit or conflicts without preparation.
- In CI/CD and prod deploy I use explicit fetch plus checkout sha/tag, not blind pull on servers.
- I add honestly: on teams I prefer `fetch` plus deliberate `rebase`/`merge` over default pull to control history.

## How it works

Remote refs live separately: `origin/feature` updates on fetch. Local `feature` moves only via merge/rebase/checkout.

```bash
git fetch origin
git status -sb
git log --oneline HEAD..origin/main
git diff HEAD..origin/main --stat

git pull --ff-only origin main

git fetch origin
git rebase origin/main
```

The first block only downloads objects. `HEAD..origin/main` shows incoming commits. `pull --ff-only` fails if a merge commit would be required (linear history guard). Explicit `fetch` + `rebase` achieves the same with a review pause.

`git pull origin main` equals `git fetch origin main` then `git merge FETCH_HEAD` (unless rebase is configured).

## Pitfalls

- **Pull with local uncommitted changes.** Merge may run on a dirty tree; stash or commit first.
- **Merge commit spam.** Frequent pull without rebase clutters history.
- **Force-push on remote.** Fetch reveals divergence; pull tries to merge diverged histories.
- **Detached CI clone.** Pull on deploy hosts without a pinned ref is risky; prefer fetch plus checkout tag.
- **Submodules.** Plain pull does not update submodule pointers; run `git submodule update`.
