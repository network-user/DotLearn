# Dockerfile: declarative image build

## In short

A Dockerfile is a text script the Docker Engine uses to build an image layer by layer. Each instruction records a filesystem change; the result is cached and reused. The `docker build` command reads the Dockerfile and directory context and produces an immutable image with metadata for running containers.

## How to answer in an interview

- A Dockerfile describes the base image, dependencies, artifacts, and entry point.
- `docker build` runs instructions sequentially; each RUN/COPY adds a layer.
- Layer cache speeds up rebuilds until an instruction or context changes.
- Multi-stage Dockerfiles shrink the final image by separating build and runtime.
- Honestly: without `.dockerignore` and pinned base image versions, builds are unstable and heavy.

## How it works

```dockerfile
FROM eclipse-temurin:21-jre-alpine

WORKDIR /opt/billing

COPY build/libs/billing-service.jar app.jar

EXPOSE 8080

ENV SPRING_PROFILES_ACTIVE=prod

USER 10001

ENTRYPOINT ["java", "-jar", "app.jar"]
```

Build:

```
docker build -t billing-service:1.4.0 .
```

Engine flow:

1. **Context.** The daemon receives the directory (usually `.`); `.dockerignore` filters files.
2. **FROM.** The base layer (JRE Alpine) becomes the first layer of the new image.
3. **COPY/RUN.** Each command creates a new layer; a changed line invalidates cache below it.
4. **Metadata.** WORKDIR, ENV, EXPOSE, CMD/ENTRYPOINT are stored in image config without a separate layer (except ENV affecting runtime).

For Java, a two-stage pattern is common: a Gradle/Maven stage builds the JAR; the final stage holds only runtime and the artifact.

## Pitfalls

- **Fat context.** `COPY . .` without ignore ships `.git` and local secrets to the daemon.
- **Root by default.** A container process with uid 0 widens blast radius on escape.
- **Floating tag.** `FROM node:latest` breaks reproducibility between CI and prod.
- **Uncleaned apt layers.** Each RUN with package manager cache bloats the image.
- **CMD vs ENTRYPOINT.** Shell-form CMD may not forward SIGTERM correctly for JVM graceful shutdown.
