# Why an AQA engineer should understand gRPC and how it works

## In short

gRPC is Google's RPC framework for calling remote methods over HTTP/2 with binary serialization via Protocol Buffers. For test automation that means contract-first `.proto` schemas, strict message types, and fast exchange, but a different verification toolkit than REST/JSON.

## How to answer in an interview

- I explain that gRPC describes the service and messages in `.proto`, and the client and server are generated by the protobuf compiler for the target language.
- I highlight the HTTP/2 transport: stream multiplexing, header compression, streaming (unary, server/client/bidi).
- I describe how I test: contract tests against `.proto`, integration calls via grpcio, status codes and metadata checks, mocks via grpc-testing or testcontainers.
- I mention typical use cases: microservices, internal APIs, mobile/IoT with limited bandwidth.
- I add honestly: for public browser APIs gRPC is often awkward without grpc-web and a proxy; REST/OpenAPI still dominates external integrations.

## How it works

The contract is declared upfront. A service defines RPC methods with input and output message types; one `.proto` generates stubs on the client and servicers on the server. The wire carries binary protobuf, not text JSON, so payloads are smaller and parse faster.

Typical flow: define `Greeter` in proto → `python -m grpc_tools.protoc` emits `*_pb2.py` and `*_pb2_grpc.py` → the server implements servicer methods → the client calls the stub like regular functions. A channel (`grpc.insecure_channel` or TLS) holds the HTTP/2 connection; calls are multiplexed on top.

```python
from concurrent import futures
import grpc
import greeter_pb2
import greeter_pb2_grpc


class GreeterService(greeter_pb2_grpc.GreeterServicer):
    def SayHello(self, request, context):
        if not request.name.strip():
            context.set_code(grpc.StatusCode.INVALID_ARGUMENT)
            context.set_details("name must not be empty")
            return greeter_pb2.HelloReply()
        return greeter_pb2.HelloReply(message=f"Hi, {request.name}")


def serve():
    server = grpc.server(futures.ThreadPoolExecutor(max_workers=8))
    greeter_pb2_grpc.add_GreeterServicer_to_server(GreeterService(), server)
    server.add_insecure_port("127.0.0.1:50051")
    server.start()
    server.wait_for_termination()


def call_greeting(name: str) -> str:
    with grpc.insecure_channel("127.0.0.1:50051") as channel:
        stub = greeter_pb2_grpc.GreeterStub(channel)
        reply = stub.SayHello(greeter_pb2.HelloRequest(name=name), timeout=3)
        return reply.message
```

In automation I verify not only happy paths but gRPC statuses (`INVALID_ARGUMENT`, `DEADLINE_EXCEEDED`), call deadlines, and proto version compatibility between services. For isolation I run the server in-process or in a container and swap dependencies with fakes.

## Pitfalls

**Breaking changes in proto.** Adding required fields or renumbering fields breaks compatibility; you need versioning rules and backward-compatible schema evolution.

**Debugging binary traffic.** Without grpcurl/Envoy tap or server-side logging it is harder to spot mismatches than with JSON in a REST client.

**Timeouts and retries.** gRPC does not remove the need for client deadlines; blind retries on non-idempotent methods create duplicates.

**TLS and certificates.** CI often skips mTLS/credentials; tests pass locally on insecure_channel and fail in staging.

**Load testing.** HTTP/2 and streaming need different load profiles and metrics (latency per RPC, backpressure) than classic REST.
