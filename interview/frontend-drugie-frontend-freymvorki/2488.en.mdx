# Angular templates: bindings and directives

## In short

Templates are HTML plus Angular syntax: interpolation, property/event binding, two-way `[(ngModel)]`, structural directives (`*ngIf`, `*ngFor`), and attribute directives (`ngClass`). The compiler generates instructions for change detection updates.

## How to answer in an interview

- `{{ expr }}` displays; `[prop]` goes down; `(event)` goes up; `[(ngModel)]` is two-way for forms.
- `*ngFor trackBy` is required for stable list performance.
- Structural directives create `<ng-template>` under the hood.
- Honestly: prefer `@if` / `@for` control flow (v17+) over legacy `*ngIf`.
- Strict templates (`strictTemplates`) catch null errors at compile time.

## How it works

```ts
import { Component } from '@angular/core';

@Component({
  selector: 'app-todo-panel',
  standalone: true,
  template: `
    <h1>{{ heading }}</h1>
    <button type="button" (click)="toggle()">Toggle list</button>
    @if (open) {
      <ul>
        @for (task of tasks; track task.id) {
          <li [class.done]="task.done">{{ task.label }}</li>
        }
      </ul>
    }
  `,
})
export class TodoPanelComponent {
  heading = 'Tasks';
  open = true;
  tasks = [
    { id: 1, label: 'Ship feature', done: false },
    { id: 2, label: 'Write tests', done: true },
  ];

  toggle() {
    this.open = !this.open;
  }
}
```

Binding grammar keeps views declarative; logic stays in the class. Pipes (`| date`) format display without extra methods. Custom directives encapsulate DOM behavior (tooltip, autofocus).

## Pitfalls

- **Two-way on custom components.** Need `@Input` + `@Output` with the same name for `[(item)]`.
- **ngModel without FormsModule.** Standalone components must import `FormsModule`.
- **Heavy template logic.** Complex expressions rerun each CD cycle.
- **Security.** BypassSecurityTrustHtml only with sanitized sources.
- **i18n.** Mark strings with `@`/`$localize` early or retrofit is costly.
