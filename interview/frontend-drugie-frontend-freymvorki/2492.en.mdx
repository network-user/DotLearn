# Dependency injection in Angular and lower coupling

## In short

Angular DI is built on a hierarchy of injectors: root, module, and component level. A class declares dependencies in its constructor or via `inject()`, and the container creates and supplies instances. The component does not know how to build the service, only the method contract.

## How to answer in an interview

- `@Injectable({ providedIn: 'root' })` gives an app-wide singleton without an explicit providers array.
- Swap implementations: mock `DataGateway` in tests via `TestBed.overrideProvider`.
- Coupling drops because the UI depends on an abstraction (`UserRepository`), not fetch details.
- Honestly: DI without interfaces in TS still binds to a class; tokens + interfaces improve loose coupling.
- Hierarchical injectors allow different scopes: one service per component subtree.

## How it works

```ts
import { Injectable, inject } from '@angular/core';

@Injectable({ providedIn: 'root' })
export class ProfileGateway {
  fetchDisplayName(userId: string) {
    return fetch(`/api/users/${userId}`).then((r) => r.json());
  }
}
```

```ts
import { Component, inject } from '@angular/core';
import { ProfileGateway } from './profile.gateway';

@Component({
  selector: 'app-profile-badge',
  standalone: true,
  template: `<span>{{ displayName }}</span>`,
})
export class ProfileBadgeComponent {
  private readonly gateway = inject(ProfileGateway);
  displayName = '';

  async ngOnInit() {
    const profile = await this.gateway.fetchDisplayName('u-42');
    this.displayName = profile.name;
  }
}
```

Injector tree: platform → root → lazy-loaded module → component. A child injector can shadow a parent provider. `InjectionToken` carries primitives and factories. In tests, `TestBed.configureTestingModule({ providers: [{ provide: ProfileGateway, useValue: stub }] })` swaps the implementation without touching the component.

## Pitfalls

- **Circular deps.** Two services inject each other; fix with `forwardRef` or layer refactor.
- **Transient vs singleton.** `providers: [Svc]` on a component creates a new instance per component instance.
- **Tree-shaking providers.** `providedIn: 'any'` is rarely needed and confuses scope.
- **Implicit any in tests.** A missing provider throws runtime `NullInjectorError`, not a compile error.
- **Over-injection.** A god-service with 15 dependencies signals the module should be split.
