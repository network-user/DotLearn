# Dependency injection in Angular

## In short

The Angular DI container resolves dependencies by token (usually a class) and injects instances into constructors or `inject()`. Providers register factories/singletons at root, module, component, or route level. That decouples components from concrete HTTP/storage implementations and simplifies unit tests with mocks.

## How to answer in an interview

- `@Injectable({ providedIn: 'root' })` for a tree-shakeable singleton service.
- Constructor injection: `constructor(private api: ApiService)`.
- `inject(ApiService)` in functional guards/interceptors.
- Honestly: hierarchical injectors let component-level providers shadow root.
- Testing: `TestBed.configureTestingModule({ providers: [{ provide: ApiService, useValue: mock }] })`.

## How it works

```ts
import { Injectable, inject, Component } from '@angular/core';
import { HttpClient } from '@angular/common/http';

@Injectable({ providedIn: 'root' })
export class ProfileApi {
  private http = inject(HttpClient);

  loadProfile() {
    return this.http.get<{ name: string }>('/api/me');
  }
}

@Component({
  selector: 'app-profile',
  standalone: true,
  template: `<p>{{ name }}</p>`,
})
export class ProfileComponent {
  private profileApi = inject(ProfileApi);
  name = '';

  constructor() {
    this.profileApi.loadProfile().subscribe((p) => {
      this.name = p.name;
    });
  }
}
```

Injector hierarchy: root → route → component. Children can override providers for scoped state (rare). Multi providers back tokens like `HTTP_INTERCEPTORS`.

## Pitfalls

- **Circular deps.** Service A injects B injects A; fix with `forwardRef` or split services.
- **providedIn: 'any'.** Multiple instances per lazy module surprise teams.
- **Manual `new Service()`.** Bypasses DI and interceptors.
- **Environment providers.** Missing `provideHttpClient()` in bootstrap breaks inject.
- **SSR tokens.** PLATFORM_ID checks for browser-only APIs.
