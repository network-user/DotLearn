# Change Detection in Angular

## In short

Change Detection syncs component state with the DOM. The default strategy checks all components on async ticks (Zone.js patches promises/events). OnPush checks a component only on input reference changes, events inside the component, or explicit markForCheck.

## How to answer in an interview

- Zone.js triggers CD after macrotasks/microtasks; `NgZone.runOutsideAngular` for high-frequency handlers.
- OnPush plus immutable inputs reduce work in large tables.
- Signals (v16+) offer fine-grained updates as an alternative to manual OnPush discipline.
- Honestly: async pipe auto-marks dirty; manual subscribe needs `ChangeDetectorRef.markForCheck`.
- Detaching CD for off-screen portals is a rare advanced optimization.

## How it works

```ts
import { ChangeDetectionStrategy, Component, Input } from '@angular/core';

@Component({
  selector: 'app-metric',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<span>{{ value }}</span>`,
})
export class MetricComponent {
  @Input({ required: true }) value = 0;
}
```

```ts
import { Component } from '@angular/core';

@Component({
  selector: 'app-counter',
  standalone: true,
  template: `
    <p>{{ total }}</p>
    <button type="button" (click)="bump()">+1</button>
  `,
})
export class CounterComponent {
  total = 0;

  bump() {
    this.total += 1;
  }
}
```

Click runs in the zone → CD tree walk → text updates. OnPush child `MetricComponent` is skipped until `@Input()` reference changes.

## Pitfalls

- **Mutate input object.** OnPush won't update if the parent mutates fields in place.
- **setTimeout without zone.** UI stays stale until the next external event unless manual detect.
- **Third-party widgets.** Non-zone APIs need `NgZone.run` wrappers.
- **ExpressionChangedAfterItHasBeenCheckedError.** Sync parent/child update order in dev double-check.
- **Over-OnPush.** Premature optimization breaks when teams forget immutability.
