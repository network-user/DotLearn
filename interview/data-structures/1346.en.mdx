# Why is an `in` check faster for a dict than for a list?

## In short

Searching for a value in a list is a linear scan, O(n). In a dict and a set, the search relies on a hash table, so on average it is constant, O(1), and independent of the size of the collection.

## How to answer in an interview

A list stores elements one after another, without any indexing by value. To find out whether an element is present, you have to compare it with each one in turn, which gives O(n) in the worst case. A dict and a set are built as hash tables: a hash is computed from the key and points immediately to the bucket where the element should live. That is why an `in` check and key access are on average O(1). In the worst case (many collisions), degradation to O(n) is theoretically possible, but in practice it almost never happens.

## How it works

In a list, the `in` operator goes through the elements from left to right:

```python
items = [10, 20, 30, 40, 50]
print(30 in items)
```

Here Python compares `30` with `10`, `20`, then `30`: three comparisons. If we were looking for `50` or a missing value, it would scan all five.

In a dict and a set, hashing does the work:

```python
prices = {"apple": 50, "banana": 30, "cherry": 90}
print("banana" in prices)

unique = {1, 2, 3, 4, 5}
print(3 in unique)
```

For a dict, `in` checks the keys: `hash("banana")` is computed, and Python jumps straight to the right bucket, never touching the rest. A set works the same way, only it stores the values themselves. The price for the speed is that elements must be hashable (immutable), and the collection uses more memory than a list.

## Pitfalls

- For a dict, `in` checks the keys, not the values. `value in my_dict` searches among the keys; to search by values you need `value in my_dict.values()`, and that is O(n) again.
- O(1) is an amortized average estimate. You can construct degenerate collisions, but in ordinary code this can be ignored.
- Only hashable objects go into a set or into dict keys. A list or another `dict` cannot be used as a key; you will get a `TypeError`.
- You pay for the speed with memory: a set and a dict take noticeably more space than a dense list of the same data.
