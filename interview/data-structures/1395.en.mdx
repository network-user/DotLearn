# A hashable object by hand: the __hash__ and __eq__ contract

## In short

For a class instance to go into a `set` or serve as a `dict` key, the class needs two consistent methods: `__hash__()` returns a stable integer, and `__eq__()` defines equality. The object itself should also be immutable - a changing hash breaks hash tables.

## How to answer in an interview

Hashability rests on the pair `__hash__` and `__eq__`, bound by a contract: if two objects are equal by `__eq__`, their hashes must match. The reverse is not required - different objects may have the same hash, that is a normal collision. The hash must be constant for the entire lifetime of the object, so the fields that participate in it must not change. A handy trick is to compute the hash from a tuple of the same fields that are compared in `__eq__`. And the iron rule: if you override `__eq__`, override `__hash__` too.

## How it works

Take a point on a plane. In `__eq__` we compare the coordinates, in `__hash__` we hash the tuple of those same coordinates.

```python
class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __eq__(self, other):
        if isinstance(other, Point):
            return self.x == other.x and self.y == other.y
        return NotImplemented

    def __hash__(self):
        return hash((self.x, self.y))
```

Now points with the same coordinates are equal and produce the same hash, so they work correctly as keys and elements:

```python
seen = {Point(1, 2), Point(1, 2)}
print(len(seen))  # 1: the duplicate collapsed
```

A tuple in `__hash__` is convenient because it computes the hash from its own elements and guarantees that the same coordinates produce the same result. The same fields go into both the comparison and the hash - so the contract is upheld automatically.

## Pitfalls

- The main invariant: `a == b` must imply `hash(a) == hash(b)`. Break it and the object gets "lost" in the dict: you stored it under one hash and look it up under another.
- If you override only `__eq__`, Python resets `__hash__` to `None` (making instances unhashable). Always define both.
- The hash must be stable. Compute it from a mutable field, change the field - and the object can no longer be found in the set. That is why hashable objects are made immutable over these fields.
- In `__eq__` it is better to return `NotImplemented` for a foreign type rather than `False`: this gives Python a chance to try the symmetric comparison from the other side.
