# Почему проверка `in` для словаря быстрее, чем для списка?

## Суть

Поиск значения в списке — это линейный перебор, O(n). В словаре и множестве поиск опирается на хеш-таблицу, поэтому в среднем он постоянный, O(1), и не зависит от размера коллекции.

## Что ответить на собесе

Список хранит элементы подряд, без какой-либо индексации по значению. Чтобы узнать, есть ли элемент, приходится сравнивать его с каждым по очереди — отсюда O(n) в худшем случае. Словарь и множество устроены как хеш-таблицы: от ключа считается хеш, который сразу указывает на корзину, где элемент должен лежать. Поэтому проверка `in` и доступ по ключу в среднем O(1). В худшем случае (множество коллизий) деградация до O(n) теоретически возможна, но на практике почти не встречается.

## Как это работает

В списке оператор `in` идёт по элементам слева направо:

```python
items = [10, 20, 30, 40, 50]
print(30 in items)
```

Здесь Python сравнит `30` с `10`, `20`, потом `30` — три сравнения. Если бы искали `50` или отсутствующее значение, перебрал бы все пять.

В словаре и множестве работает хеширование:

```python
prices = {"apple": 50, "banana": 30, "cherry": 90}
print("banana" in prices)

unique = {1, 2, 3, 4, 5}
print(3 in unique)
```

`in` для словаря проверяет ключи: считается `hash("banana")`, и Python сразу прыгает в нужную корзину, не трогая остальные. Множество работает так же, только хранит сами значения. Цена за скорость — элементы должны быть хешируемыми (неизменяемыми), а коллекция тратит больше памяти, чем список.

## Подводные камни

- `in` для словаря проверяет ключи, а не значения. `value in my_dict` ищет среди ключей; чтобы искать по значениям, нужно `value in my_dict.values()`, и это снова O(n).
- O(1) — амортизированная средняя оценка. Конструировать вырожденные коллизии можно, но в обычном коде этим можно пренебречь.
- В множество и в ключи словаря кладутся только хешируемые объекты. Список или другой `dict` ключом не положить — будет `TypeError`.
- За скорость платят памятью: множество и словарь занимают заметно больше, чем плотный список тех же данных.
