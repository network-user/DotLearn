# Does the standard dict guarantee key iteration order?

## In short

Yes. Since Python 3.7, preserving the insertion order of keys in a regular `dict` is a language guarantee (in CPython 3.6 it already worked as an implementation detail). For older versions, `collections.OrderedDict` played the same role.

## How to answer in an interview

Starting with Python 3.7, the order of keys in a `dict` is guaranteed to match the order in which they were added - this is part of the language specification, not an accident of the implementation. Before 3.6 the order was undefined, and `OrderedDict` was used for predictable iteration. Today `OrderedDict` is only needed in special cases: when its specific methods matter (`move_to_end`, `popitem(last=...)`) or when you need order-aware comparison.

## How it works

A regular dictionary remembers the insertion sequence, and iteration proceeds in exactly that order:

```python
my_dict = {}
my_dict['a'] = 1
my_dict['b'] = 2
my_dict['c'] = 3

for key in my_dict:
    print(key, my_dict[key])
```

The output follows the order `a`, `b`, `c` - exactly as the keys were added.

`OrderedDict` is needed when you specifically require ordered behavior as a contract or its additional operations:

```python
from collections import OrderedDict

ordered = OrderedDict()
ordered['a'] = 1
ordered['b'] = 2
ordered.move_to_end('a')   # 'a' moves to the end
print(list(ordered))       # ['b', 'a']
```

## Pitfalls

- The order is **insertion** order, not sorted order. Reassigning an existing key does not change its position; to move a key you have to delete it and add it again.
- Equality comparison: for a regular `dict`, `{'a':1,'b':2} == {'b':2,'a':1}` is true (order is ignored), whereas `OrderedDict` comparison is order-dependent.
- Deleting and re-inserting a key places it at the end - easy to miss during incremental updates.
- The order guarantee does not make a `dict` sorted: for iteration ordered by values/keys you need `sorted(...)`.
