# Two keys landed in the same slot: how does a hash table cope?

## In short

When different keys produce the same index, that's a collision. It's resolved by two families of methods: chaining - stacking everything into a list inside the slot, and open addressing - searching for another free slot within the table itself.

## How to answer in an interview

A collision is the situation where the hash function produces the same index for two different keys. Collisions can't be avoided entirely, because there are infinitely many inputs and only a finite number of slots, so the table must be able to resolve them. There are two classic approaches. The first is chaining: each slot stores a collection (usually a linked list) of all elements with that index, and on a collision we simply append the new pair there. The second is open addressing: all elements live directly in the array, and when a slot is occupied we search for the next free one by a certain rule (linear probing, quadratic probing, or double hashing). CPython, by the way, uses exactly open addressing for its `dict`.

**Chaining.** Each slot is a small collection. Insertion appends the pair to the end, lookup walks the collection and compares keys.

```python
class ChainedHashTable:
    def __init__(self, size):
        self.size = size
        self.table = [[] for _ in range(size)]

    def _index(self, key):
        return hash(key) % self.size

    def insert(self, key, value):
        bucket = self.table[self._index(key)]
        for pair in bucket:
            if pair[0] == key:
                bucket.remove(pair)
                break
        bucket.append((key, value))

    def get(self, key):
        for k, v in self.table[self._index(key)]:
            if k == key:
                return v
        return None
```

**Open addressing.** Elements sit in the array itself. On a collision we step forward linearly until we find an empty slot (that's linear probing).

```python
class OpenAddressingTable:
    def __init__(self, size):
        self.size = size
        self.table = [None] * size

    def _index(self, key):
        return hash(key) % self.size

    def insert(self, key, value):
        index = self._index(key)
        while self.table[index] is not None and self.table[index][0] != key:
            index = (index + 1) % self.size
        self.table[index] = (key, value)

    def get(self, key):
        index = self._index(key)
        while self.table[index] is not None:
            k, v = self.table[index]
            if k == key:
                return v
            index = (index + 1) % self.size
        return None
```

## Pitfalls

- With chaining, under many collisions a slot degenerates into a long list, and lookup degrades from O(1) to O(n). The defense is a good hash function and control of the load factor.
- In open addressing, deleting an element breaks the search chain: you can't just set `None`, otherwise the element following it becomes "invisible". Tombstones are used - special markers for deleted slots.
- Open addressing behaves poorly under a high load factor: the denser the table, the longer the probing. At around 70-75% fill it is usually expanded (rehashed).
- Linear probing suffers from clustering - occupied slots stick together into long groups. Quadratic probing and double hashing break up such clusters.
