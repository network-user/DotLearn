# Why is a tuple more compact in memory than a list of the same length?

## In short

Both a list and a tuple are arrays of pointers to objects. But a list is mutable and keeps a memory reserve for growth, while a tuple is fixed, so it is stored more densely and avoids the overhead of mutation.

## How to answer in an interview

A list is a dynamic array: it is reallocated with spare room (over-allocation) so that `append` is O(1) on average, and it stores length and capacity separately. A tuple is immutable, its size is known at creation, so there is no extra capacity and the header structure is simpler. Hence the lower memory footprint and slightly faster access. The practical criterion for choosing: need mutability - use a list; need fixed data - use a tuple.

When you add elements to a list and the capacity is exhausted, Python allocates a new, larger block and copies the pointers into it:

```python
my_list = [1, 2, 3]
my_list.append(4)   # if capacity runs out, the list is reallocated with reserve
```

A tuple is created at exactly the needed size and never grows:

```python
my_tuple = (1, 2, 3)   # size is fixed, no reserve for growth
```

The difference in memory consumption is directly visible:

```python
import sys

print(sys.getsizeof([1, 2, 3]))   # more
print(sys.getsizeof((1, 2, 3)))   # less
```

## Pitfalls

- A tuple's immutability is shallow: if it contains a list, that list's contents can be changed - `(1, [2, 3])[1].append(4)` works.
- It's the comma, not the parentheses, that makes a tuple: `(1)` is a number, `(1,)` is a single-element tuple.
- "A tuple is faster" applies to access and to creating literals, not to all operations; don't turn this into dogma.
- The specific numbers from `getsizeof` depend on the CPython version and platform - what matters is not the figure but the ratio.
