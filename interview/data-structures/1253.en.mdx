# How a hash table finds a value almost instantly

## In short

A hash table turns a key into a number via a hash function and uses that number to compute the index of a slot in an array. Since the index is known right away, there is no need to reach the value by scanning - so search, insertion, and deletion cost O(1) on average.

## How to answer in an interview

A hash table (hash map) stores key-value pairs. On write, it runs the key through a hash function, takes the remainder of division by the size of the internal array, and puts the value into the resulting bucket. On read, the same computation is performed - and the table goes straight to the right slot without scanning the rest. That is why the average complexity of operations is O(1). A problem arises when two different keys yield the same index (a collision); it is solved with chaining or open addressing. In the worst case, when everything lands in one bucket, operations degrade to O(n).

## How it works

1. **Hash function.** Takes a key and produces an integer. A good function distributes keys across the array evenly - the more even the spread, the fewer collisions.
2. **Buckets.** Inside there is an array; its slots are called buckets. The bucket index is obtained as `hash(key) % len(array)`.
3. **Collisions.** Two different keys can yield the same index. This is normal and unavoidable - the only question is how to resolve it:
   - **Chaining:** the bucket holds a list of all elements that landed in it; a new one is simply appended to the end.
   - **Open addressing:** if the slot is taken, the next free one is found by a rule (linear or quadratic probing).
4. **Rehashing.** When the load factor exceeds a threshold, the array is enlarged and the elements are laid out anew - this keeps the average chain length short.

```python
phone_book = {}
phone_book["Alice"] = "555-1234"
phone_book["Bob"] = "555-5678"

print(phone_book["Alice"])  # 555-1234
```

## Pitfalls

- **O(1) is the average, not a guarantee.** With a bad hash function or artificially chosen keys, everything funnels into one bucket and operations become O(n).
- **A key must be hashable and immutable** with respect to its hash. In Python you cannot put a list into a `dict`/`set` - only a `tuple`, a string, a number, and so on. If you mutate an object in a way that changes its hash, you will not be able to find it afterward.
- **Order is not defined by the logic of the structure.** In CPython since 3.7, `dict` preserves insertion order as an implementation detail, but you cannot rely on hash order as if it were a sort.
- **Memory in exchange for speed:** a low load factor (many empty buckets) speeds up access but spends memory; it is always a trade-off.
