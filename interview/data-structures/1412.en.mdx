# Why can a hash function produce the same hash for different data?

## In short

A collision is the situation where two different input values, run through a hash function, produce the same hash. In hash tables, where a key must point to its own cell, this requires a dedicated resolution mechanism.

## How to answer in an interview

A collision is when the hashes of two distinct inputs coincide. It is inevitable, because the set of possible inputs is practically infinite while the range of hashes is finite (the Dirichlet principle, also known as the "pigeonhole principle"). That is why a hash table does not rely on the uniqueness of the hash: on a match it applies collision resolution - most often separate chaining or open addressing.

## How it works

A hash function maps data of arbitrary size to a fixed-length value. Ideally, different inputs would yield different hashes, but since there are knowingly more inputs than hashes, matches are guaranteed.

Let's build a crude hash function and deliberately trigger a collision:

```python
def simple_hash(key):
    return len(key) % 10

hash_table = {}
keys = ["apple", "grape", "banana", "cherry"]

for key in keys:
    bucket = simple_hash(key)
    hash_table.setdefault(bucket, []).append(key)

print(hash_table)
```

Here the hash depends only on the length of the string. `"apple"` and `"grape"` are both 5 characters long, so they land in the same bucket `5` - that is the collision. The `setdefault` method creates a list under the bucket if it does not exist yet and appends the key to it. This technique is exactly the implementation of chaining: a single cell holds a list of all the elements that collided.

## Pitfalls

- Collisions are not a bug but a mathematical inevitability - you should design their resolution, not their absence.
- With a large number of collisions, lookup in a hash table degrades from O(1) to O(n), because you have to scan the chain linearly.
- A good hash function distributes values evenly; a primitive one (like by length) dumps everything into a few buckets.
- In Python the hash of strings is randomized between runs (`PYTHONHASHSEED`) for protection against attacks, so the built-in `hash()` cannot be cached to disk.
