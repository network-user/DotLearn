# What is under the hood of a Python dict, and why is key lookup so fast?

## In short

A Python dictionary is implemented on top of a hash table. The key is run through a hash function, which points to a slot in the internal array, and thanks to this, insertion, lookup, and deletion by key run in O(1) on average.

## How to answer in an interview

A `dict` is a hash table. When you put in a pair, Python computes `hash(key)`, uses it to determine the index of a slot in the internal array, and stores the entry there. Lookup works the same way: the same hash leads straight to the right slot, with no scanning, hence the average `O(1)`. If two different keys map to the same index, that is a collision, and CPython resolves it with open addressing (probing), finding another free slot. This is why keys must be hashable and stable under hashing - a list cannot be a key, but a tuple of immutable elements can. In the worst case, with massive collisions, operations degrade to `O(n)`, but in practice this is rare. And separately: starting from Python 3.7, a dictionary is guaranteed to preserve insertion order.

## How it works

Inside there is an array of slots. Put in a pair, and the key's hash sets the index:

```python
phone_book = {
    "Alice": "+123456789",
    "Bob": "+987654321",
    "Charlie": "+192837465",
}

print(phone_book["Alice"])

phone_book["David"] = "+564738291"
del phone_book["Bob"]
```

The lookup `phone_book["Alice"]` does not scan the dictionary - it computes the hash of the string `"Alice"`, jumps to its slot, and compares the key for safety. Insertion and deletion work with the same addressing scheme.

```
key "Alice" --hash--> index --> [array slot: "Alice": "+123..."]
```

When the array fills to about two-thirds, the dictionary grows its capacity and re-lays out the entries - this keeps collisions rare and the average operation time constant.

## Pitfalls

- A key can only be a hashable object. `list`, `dict`, and `set` cannot be keys; a `tuple` works only if all its elements are immutable too.
- O(1) is an average, not a guarantee. With massive collisions, operations degrade, so it is important that `__hash__` and `__eq__` are consistent: equal objects must have equal hashes.
- If you mutate an object so that its hash changes after it has been inserted into a dictionary, the key gets "lost" and can no longer be found.
- Insertion order is preserved from Python 3.7 on, but you cannot rely on this as ordering by key - it is exactly the order of addition, not sorting.
- A dictionary pays in memory for its speed: the internal array is kept with spare free slots so that collisions stay rare.
