# Array or linked list: what do we pay for fast access and for fast insertion?

## In short

An array lays elements out one after another in memory and gives instant index access, but pays with expensive insertions into the middle. A linked list connects nodes with references, so insertion and deletion with a ready reference are cheap, but reaching the n-th element is only possible by traversal.

## How to answer in an interview

It's a compromise between access speed and the cost of changes. An array stores data in a contiguous block, the address of any element is computed from the index, so reading is `O(1)`. But insertion or deletion in the middle requires shifting the tail, that's `O(n)`. A linked list is a chain of nodes, each knowing the next (and in a doubly linked one, the previous too); adding or removing a node when you have a reference to a neighbor is `O(1)`, but index access is `O(n)`, because you have to walk from the head. Another nuance: a list spends extra memory on pointers, while an array packs more compactly and is friendlier to the processor cache. By the way, `list` in Python is a dynamic array, not a linked list.

In an array the elements lie tightly packed, and the address of the i-th equals `base + i * element_size`. Hence constant-time access:

```python
array = [10, 20, 30, 40]
print(array[2])
```

The price for this is rebuilding on changes in the middle: to insert an element, everything to the right of it must be shifted.

A linked list doesn't require a contiguous block. Each node is a separate cell with a reference to the next:

```python
class Node:
    def __init__(self, data):
        self.data = data
        self.next = None

class LinkedList:
    def __init__(self):
        self.head = None

    def append(self, data):
        node = Node(data)
        if self.head is None:
            self.head = node
            return
        current = self.head
        while current.next is not None:
            current = current.next
        current.next = node
```

Inserting at the head here is done in `O(1)` - it's enough to relink a reference. But finding the n-th element always goes along the chain from `head`.

```
array:        [10][20][30][40]      <- an index gives the address instantly
linked list:  10 -> 20 -> 30 -> 40  <- to reach 30 we follow references from the head
```

## Pitfalls

- "A linked list gives O(1) insertion" - only if you already have a reference to the right spot. Finding that spot is the same old `O(n)`.
- In practice an array often outruns a list even where theory promises the opposite: contiguous memory befriends the processor cache, while pointer jumps don't.
- In Python `list` is a dynamic array; for a true doubly linked list with cheap operations at the ends, use `collections.deque`.
- An `append` to the end of a Python list is amortized `O(1)`, but occasionally triggers copying the whole array on capacity growth - that's already `O(n)` on that particular insertion.
