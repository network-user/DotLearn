# Why does an engineer need Big O to evaluate an algorithm?

## In short

Big O is the language used to describe how an algorithm's cost (time or memory) grows as the input size increases, usually focusing on the worst case. It drops constants and keeps only the dominant growth factor.

## How to answer in an interview

Big O is there to predict in advance how an algorithm will behave on large data and to compare two solutions without running them. I don't look at absolute speed, but at how many times the cost will grow if the data becomes, say, ten times larger. The key classes I keep in mind are O(1), O(log n), O(n), O(n log n), and O(n^2), which is enough to assess most interview problems.

## How it works

Big O describes asymptotics: the behavior of an algorithm as the input size n goes to infinity. That's why constants and lower-order terms are dropped: 3n + 100 and n give the same class O(n), because for large n it's the linear term that determines the picture.

Classes in order of increasing cost:

1. **O(1)**: the cost is fixed and does not depend on n. Example: accessing an array element by index.
2. **O(log n)**: the cost grows logarithmically; the input is halved at each step. Example: binary search.
3. **O(n)**: the cost is proportional to the number of elements. Example: a single pass over an array.
4. **O(n log n)**: the behavior of efficient sorts (quicksort, merge sort).
5. **O(n^2)**: the cost is the square of the input. Bubble, insertion, and selection sorts fall here.

Let's compare three functions with different asymptotics:

```python
def get_first_element(arr):
    return arr[0]

def find_element(arr, target):
    for element in arr:
        if element == target:
            return True
    return False

def bubble_sort(arr):
    n = len(arr)
    for i in range(n):
        for j in range(0, n - i - 1):
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]
    return arr
```

`get_first_element`: index access in fixed time, O(1). `find_element`: in the worst case checks every element, O(n). `bubble_sort`: nested loops give O(n^2).

## Pitfalls

- Big O is about growth, not absolute speed. An O(n^2) algorithm on a small n can beat O(n log n) thanks to smaller constants.
- Don't confuse worst, average, and amortized cases. A dictionary lookup is O(1) amortized, but O(n) in the worst case with many collisions.
- Time and memory estimates are different things. Sometimes you pay memory for speed and vice versa.
- Big O hides constants, but in practice they matter. A "linear" algorithm with a heavy operation inside can lose.
