# Merge sort: how does sorting via splitting and merging work?

## In short

Merge sort is divide-and-conquer: the array is recursively cut in half down to single elements, and then the halves are merged pairwise into sorted chunks. The algorithm is stable and guarantees O(n log n) in the worst case at the cost of O(n) extra memory.

## How to answer in an interview

Merge sort splits the array in two, recursively sorts each half, and merges them in linear time. The recursion depth is log n levels, and at each level the merge scans all n elements, which gives O(n log n) both on average and in the worst case - unlike quicksort, whose worst case is O(n²). The algorithm is stable: equal elements keep their original order. We pay for this with O(n) memory for a temporary merge buffer. Because of its predictable complexity and stability, merge sort is chosen for external sorting of large data and as the basis for hybrids like Timsort.

## How it works

Three steps: split down to the base case, merge two ordered parts, repeat from the bottom up.

```python
def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])
    right = merge_sort(arr[mid:])
    return merge(left, right)

def merge(left, right):
    result = []
    i = j = 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i])
            i += 1
        else:
            result.append(right[j])
            j += 1
    result.extend(left[i:])
    result.extend(right[j:])
    return result
```

The condition `left[i] <= right[j]` (non-strict) is what provides stability: on a tie, the element from the left part is taken first.

Take `[38, 27, 43, 3, 9, 82, 10]`:

- Split down to singles: `[38] [27] [43] [3] [9] [82] [10]`.
- Merge pairs: `[27, 38]`, `[3, 43]`, `[9, 82]`, `[10]`.
- Merge further: `[3, 27, 38, 43]`, `[9, 10, 82]`.
- Final: `[3, 9, 10, 27, 38, 43, 82]`.

## Pitfalls

- **Extra O(n) memory**. The slices `arr[:mid]` create copies, plus the merge buffer. On very large data this is significant, unlike in-place quicksort or heapsort.
- **Stability is easy to break**. If you change `<=` to `<` in the comparison, equal elements will swap places and stability is lost.
- **Recursion is not free**. For small subarrays the call overhead is large; practical implementations (Timsort) switch to insertion sort for short chunks.
- **O(n log n) is both the ceiling and the floor**. Merge sort does not speed up on already-sorted data, unlike Timsort or insertion sort.
