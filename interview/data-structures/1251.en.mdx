# Why is indexed access O(1) in an array but O(n) in a linked list?

## In short

In an array the elements lie back to back, so the address of any one is computed by arithmetic — that's O(1). In a linked list the nodes are scattered across memory, and to reach the n-th one you have to walk the links, hence O(n).

## How to answer in an interview

The speed of indexed access is determined by the data layout in memory. An array is a contiguous block, and the address of an element is computed as `base + index × element_size`, in constant time. A Python list is internally a dynamic array of pointers, so `lst[i]` is also O(1). A linked list stores its nodes anywhere, each node knows only the next one, and the only way to reach index i is to make i hops along the links, which gives O(n). On the other hand, a linked list wins on insertion/deletion in the middle when you have a reference to the node: O(1) versus O(n) for an array.

## How it works

Access in an array / Python list is direct:

```python
array = [10, 20, 30, 40, 50]
print(array[2])   # 30, address computed directly, O(1)
```

In a linked list you have to walk from the head:

```python
class Node:
    def __init__(self, data):
        self.data = data
        self.next = None

def get(head, index):
    current = head
    count = 0
    while current:
        if count == index:
            return current.data
        current = current.next
        count += 1
    return None

head = Node(10)
head.next = Node(20)
head.next.next = Node(30)
print(get(head, 2))   # 30, but it took 2 hops to get there, O(n)
```

## Pitfalls

- "List" is ambiguous: a list in Python is a dynamic array (O(1) access), whereas a "linked list" structure is something else (O(n) access). In an interview, clarify which one is meant.
- An array has O(1) access, but insertion/deletion at the front or middle is O(n) because of shifting; a linked list is the opposite. The choice of structure depends on which operations dominate.
- O(1) for an array holds specifically for random access by index, not for searching by value — a search is still O(n).
- `collections.deque` is not an array: indexing toward the middle is O(n) for it, but adding/removing at both ends is O(1).
