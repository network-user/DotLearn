# How fast does a hash table find a value by key?

## In short

On average, access by key is amortized O(1): the hash function immediately yields an index, and the value is read directly. In the degenerate case, when everything piles into a single bucket, the time grows to O(n).

## How to answer in an interview

"Lookup by key is on average O(1): the hash function translates the key into an array index, and we land in the right cell in constant time. But that's the average under a uniform distribution and a reasonable load factor. If the hash function is poor or all keys yield the same hash, the bucket turns into a linear structure, and lookup degrades to O(n). That's why in practice both the quality of the hash function and timely table resizing matter."

## How it works

A table stores key-value pairs on top of an ordinary array. The access logic comes down to three steps:

1. **Hashing.** The key is run through a hash function, and the result is folded into an array index (usually via the remainder of division by the size). A good function distributes keys uniformly.
2. **Reading the bucket.** At the index lies either the value itself or a bucket with several elements, if a collision occurred.
3. **Collision resolution.** With the chaining method, a bucket is a list, over which a linear search for the needed key runs. With open addressing, neighboring cells are checked by a given probing strategy.

An example mental model is a phone book: a name is turned into an index, at which the number is stored. If two names yielded the same index, the collision resolution mechanism places both in one bucket.

## Pitfalls

- O(1) is an **average, amortized** value, not a guarantee. In an interview it's easy to slip up and call it the worst case.
- The worst case O(n) is real: deliberately chosen keys with the same hash (hash flooding) intentionally break performance — this is a DoS attack vector.
- At a high load factor the table must grow (rehash). The resize itself is a one-off O(n), but it's what keeps the average access cost constant.
- In Python `dict` and `set` are hash tables, and their `in`/access by key work by the same rules; you can't rely on iteration order.
