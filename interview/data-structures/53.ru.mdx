# Зачем инженеру оценка алгоритма через Big O?

## Суть

Big O — это язык, которым описывают темп роста затрат алгоритма (времени или памяти) по мере увеличения объёма входа, ориентируясь обычно на худший случай. Он отбрасывает константы и оставляет только доминирующий фактор роста.

## Что ответить на собесе

Big O нужен, чтобы заранее предсказать поведение алгоритма на больших данных и сравнить два решения, не запуская их.

- Смотрю не на абсолютную скорость, а на то, во сколько раз вырастут затраты, если данных станет, скажем, в десять раз больше.
- Ключевые классы держу в голове: O(1), O(log n), O(n), O(n log n) и O(n^2).
- Этого хватает, чтобы оценить большинство задач на собеседовании.

## Как это работает

Big O описывает асимптотику — поведение алгоритма, когда размер входа n уходит в бесконечность. Поэтому константы и младшие слагаемые отбрасываются: 3n + 100 и n дают один и тот же класс O(n), потому что на больших n именно линейный член определяет картину.

Классы по нарастанию затрат:

1. **O(1)** — затраты фиксированы и не зависят от n. Пример: обращение к элементу массива по индексу.
2. **O(log n)** — затраты растут логарифмически; вход на каждом шаге делится пополам. Пример: бинарный поиск.
3. **O(n)** — затраты пропорциональны числу элементов. Пример: один проход по массиву.
4. **O(n log n)** — поведение эффективных сортировок (быстрая, слиянием).
5. **O(n^2)** — затраты как квадрат входа. Сюда попадают сортировки пузырьком, вставками, выбором.

Сравним три функции с разной асимптотикой:

```python
def get_first_element(arr):
    return arr[0]

def find_element(arr, target):
    for element in arr:
        if element == target:
            return True
    return False

def bubble_sort(arr):
    n = len(arr)
    for i in range(n):
        for j in range(0, n - i - 1):
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]
    return arr
```

`get_first_element` — индексный доступ за фиксированное время, O(1). `find_element` — в худшем случае проверяет все элементы, O(n). `bubble_sort` — вложенные циклы дают O(n^2).

## Подводные камни

- Big O — про рост, а не про абсолютную скорость. Алгоритм O(n^2) на маленьком n может обогнать O(n log n) из-за меньших констант.
- Не путайте худший, средний и амортизированный случаи. Поиск по словарю — O(1) амортизированно, но O(n) в худшем случае при множестве коллизий.
- Оценка по времени и по памяти — разные вещи. Иногда платят памятью за скорость и наоборот.
- Big O скрывает константы, но на практике они решают. «Линейный» алгоритм с тяжёлой операцией внутри может проиграть.
