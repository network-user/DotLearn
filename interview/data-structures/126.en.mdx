# What do the basic operations on list, dict, set, and deque cost?

## In short

The cost of an operation depends on the structure. For `list`, access by index and appending to the end are O(1), while insertion/removal in the middle is O(n). For `set` and `dict`, addition, removal, and membership check are on average O(1). `collections.deque` adds and removes from both ends in O(1).

## How to answer in an interview

I keep a map of complexities in my head: a list is good for index access and appending to the end (O(1)), but bad for insertions in the middle (O(n)). A set and a dictionary are built on hash tables, so their addition, removal, and membership check are on average O(1). If I need a queue or operations at both ends, I use `deque` instead of a list, because `list.pop(0)` is O(n) while `deque.popleft()` is O(1). I choose the structure based on the dominant operation of the task.

## How it works

**`list`** — a dynamic array with contiguous memory.

- Access by index: O(1), the element's address is computed directly.
- Appending to the end: amortized O(1); when capacity is exhausted, memory is reallocated, and that particular operation costs O(n).
- Insertion and removal not at the end: O(n) because of shifting the subsequent elements.

**`set` and `dict`** — built on hash tables.

- Addition, removal, membership check: on average O(1) by hash. In the worst case, with many collisions, up to O(n).

**`collections.deque`** — a double-ended queue.

- Addition and removal at both ends: O(1), with no shifts, unlike `list.pop(0)`.

```python
from collections import deque

prices = [10, 20, 30]
print(prices[1])          # O(1): access by index
prices.append(40)         # amortized O(1)
prices.insert(0, 5)       # O(n): shifting the tail

seen = {1, 2, 3}
print(2 in seen)          # on average O(1)

queue = deque([1, 2, 3])
queue.appendleft(0)       # O(1)
queue.popleft()           # O(1)
```

## Pitfalls

- `list.pop(0)` and `list.insert(0, x)` are O(n), a common trap when imitating a queue with a list. For a queue, use `deque`.
- O(1) for `set`/`dict` is the average case. With bad hashes or mass collisions it degrades to O(n).
- Amortized O(1) for `append` doesn't mean "always O(1)": a single insertion with memory reallocation costs O(n), it's just that such cases are rare.
- The check `x in my_list` is O(n), while `x in my_set` is O(1). If you need to check membership often, keep the data in a set.
