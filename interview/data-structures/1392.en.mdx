# How to choose a collection for the task: where do you win on speed and memory?

## In short

The four basic collections - list, tuple, set, and dict - have different strengths. The choice comes down to three questions: do you need mutability, how do you reach your data (by index, by key, by presence), and how critical is memory.

## How to answer in an interview

I start from the access pattern. If I need order and access by position - I take a list (O(1) access, but insertion in the middle is O(n)). If the data does not change and compactness matters - a tuple, it does not keep spare capacity for growth. If I often check "is this element present" or remove duplicates - a set, membership check is O(1). If I reach for a value by key - a dict, access is also O(1). For the speed of sets and dicts I pay with the memory for the hash table.

## How it works

**List** - a mutable sequence. Index access is O(1), appending to the end is amortized O(1), but insertion and deletion in the middle is O(n) because of shifting the tail. To prepare for future growth a list keeps spare capacity, so it takes a bit more memory than strictly needed for the current elements.

```python
items = [1, 2, 3, 4]
items.append(5)      # O(1) amortized, at the end
items.insert(2, 10)  # O(n): elements to the right shift
```

**Tuple** - an immutable sequence. Index access is also O(1), but any "change" is a new object. In return it uses less memory: no spare capacity for growth is needed.

```python
point = (10, 20)
# point[0] = 1  # TypeError: tuple is immutable
```

**Set** - an unordered collection of unique hashable elements backed by a hash table. Membership check, addition, and removal are amortized O(1).

```python
seen = {1, 2, 3}
seen.add(4)      # O(1)
print(2 in seen) # O(1), not O(n) like a list
```

**Dict** - key-value pairs backed by a hash table. Access, insertion, and deletion by key are amortized O(1).

```python
prices = {"a": 1, "b": 2}
prices["c"] = 3       # O(1)
value = prices["b"]   # O(1)
```

## Pitfalls

- The check `x in collection` is O(n) for a list, but O(1) for a set and a dict. If you ask about membership often, a list becomes the bottleneck.
- "A set and a dict cost more memory" is true, but in exchange you get constant-time lookup. Do not optimize memory where you are bound by lookup speed.
- A set and a dict require hashable elements/keys: you cannot put a list in there.
- A set does not preserve insertion order; a dict in CPython 3.7+ does preserve insertion order, but that is about iteration, not about "access by position."
