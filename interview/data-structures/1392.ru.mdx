# Как выбрать коллекцию под задачу: где выигрыш по скорости и памяти?

## Суть

У четырёх базовых коллекций - списка, кортежа, множества и словаря - разные сильные стороны. Выбор сводится к трём вопросам: нужна ли изменяемость, по чему вы ходите за данными (индекс, ключ, факт наличия) и насколько критична память.

## Что ответить на собесе

Отталкиваюсь от паттерна доступа, а за скорость множества и словаря плачу памятью на хеш-таблицу.

- Нужен порядок и обращение по позиции - список (доступ O(1), но вставка в середину O(n)).
- Данные не меняются и важна компактность - кортеж, он не держит запас под рост.
- Часто проверяю "есть ли элемент" или убираю дубликаты - множество, проверка наличия O(1).
- Хожу за значением по ключу - словарь, доступ тоже O(1).

## Как это работает

**Список** - изменяемая последовательность. Доступ по индексу O(1), добавление в конец амортизированно O(1), но вставка и удаление в середине O(n) из-за сдвига хвоста. Под будущий рост список держит запас ёмкости, поэтому занимает чуть больше памяти, чем строго нужно под текущие элементы.

```python
items = [1, 2, 3, 4]
items.append(5)      # O(1) амортизированно, в конец
items.insert(2, 10)  # O(n): элементы справа сдвигаются
```

**Кортеж** - неизменяемая последовательность. Доступ по индексу тоже O(1), но любое "изменение" - это новый объект. Зато памяти меньше: запас под рост не нужен.

```python
point = (10, 20)
# point[0] = 1  # TypeError: кортеж неизменяем
```

**Множество** - неупорядоченный набор уникальных хешируемых элементов на хеш-таблице. Проверка наличия, добавление и удаление - амортизированно O(1).

```python
seen = {1, 2, 3}
seen.add(4)      # O(1)
print(2 in seen) # O(1), не O(n) как у списка
```

**Словарь** - пары "ключ-значение" на хеш-таблице. Доступ, вставка и удаление по ключу - амортизированно O(1).

```python
prices = {"a": 1, "b": 2}
prices["c"] = 3       # O(1)
value = prices["b"]   # O(1)
```

## Подводные камни

- Проверка `x in collection` для списка - это O(n), а для множества и словаря - O(1). Если часто спрашиваете о наличии, список превращается в узкое место.
- "Множество и словарь дороже по памяти" - правда, но за это вы получаете константный поиск. Не оптимизируйте память там, где упираетесь в скорость поиска.
- Множество и словарь требуют хешируемых элементов/ключей: положить туда список не получится.
- Множество не хранит порядок вставки; словарь с CPython 3.7+ порядок вставки сохраняет, но это про итерацию, а не про "доступ по позиции".
