# Three ways to reverse a list: which is a copy, which an iterator, which a mutation?

## In short

You can reverse a list with the slice `[::-1]` (a new copy), the `reversed()` function (a lazy iterator), or the `.reverse()` method (changes the list in place). They differ in what they return and whether they touch the original.

## How to answer in an interview

I choose based on whether I need a copy. `lst[::-1]` creates a new reversed list and leaves the original unchanged. `reversed(lst)` returns a lazy iterator - I wrap it in `list()` if I need a list; on the upside it works with any sequence and doesn't copy the data right away. `lst.reverse()` reverses in place, returns nothing (`None`), and saves memory. The main trap is that `.reverse()` mutates the original, while `[::-1]` and `reversed()` don't touch it.

## How it works

The slice - returns a separate new list:

```python
original = [1, 2, 3, 4, 5]
reversed_copy = original[::-1]
# reversed_copy == [5, 4, 3, 2, 1], original is unchanged
```

`reversed()` - returns an iterator that copies nothing until it's materialized:

```python
original = [1, 2, 3, 4, 5]
it = reversed(original)        # this is an iterator, not a list
as_list = list(it)             # [5, 4, 3, 2, 1]
```

`.reverse()` - reverses the object itself and returns `None`:

```python
original = [1, 2, 3, 4, 5]
result = original.reverse()
# original == [5, 4, 3, 2, 1], result is None
```

What to choose:

- **slice** - you need a copy and must not touch the original;
- **`reversed()`** - convenient for iterating from the end or reversing a string/tuple without an extra list;
- **`.reverse()`** - you need to change the order right in the original list without a copy.

## Pitfalls

- `lst.reverse()` returns `None`. Writing `lst = lst.reverse()` will wipe the list to `None` - a classic mistake, just like with `sort()`.
- `reversed()` returns a one-shot iterator: after a full pass it's exhausted and can't be traversed a second time.
- `[::-1]` copies the entire list into memory; on huge data this is more expensive than the lazy `reversed()` or the in-place `.reverse()`.
- `reversed()` requires the object to know its length and support indexing (`__len__` and `__getitem__`) or to implement `__reversed__`; it won't work over an arbitrary generator.
