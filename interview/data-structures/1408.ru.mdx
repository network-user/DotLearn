# За сколько операций dict находит и сохраняет значение по ключу?

## Суть

И запись, и доступ по ключу в `dict` амортизированно стоят O(1), потому что под капотом лежит хеш-таблица. Теоретический худший случай - O(n) при сплошных коллизиях хешей, но в реальном коде он практически не встречается.

## Что ответить на собесе

Словарь - это хеш-таблица, поэтому и вставка, и чтение по ключу в среднем работают за O(1).

- Обе операции не зависят от количества элементов.
- Деградация до O(n) возможна только при массовых коллизиях, когда множество разных ключей попадает в одну цепочку.
- На практике хорошая хеш-функция и пересоздание таблицы при росте этого не допускают.

## Как это работает

Ключевая идея: вместо линейного перебора Python вычисляет позицию элемента арифметически.

При записи интерпретатор берёт хеш ключа (`hash(key)`), по нему определяет индекс ячейки во внутреннем массиве и кладёт туда связку ключ-значение. При чтении хеш считается заново и сразу указывает на ту же ячейку - перебирать ничего не нужно.

```python
my_dict = {}

my_dict['key1'] = 'value1'
# hash('key1') -> индекс ячейки -> туда кладётся пара

value = my_dict['key1']
# hash('key1') считается снова и ведёт прямо к 'value1'
```

Где это особенно ценно: кеши, индексы, хранение сессий по идентификатору - везде, где доступ идёт по уникальному ключу, а коллекция большая.

## Подводные камни

- **O(1) - это амортизированно, а не гарантированно.** При сплошных коллизиях операция деградирует до O(n). Поэтому критично, чтобы у ключей была качественная хеш-функция.
- **Ключ обязан быть хешируемым.** Список или другой изменяемый объект ключом не станет - получите `TypeError`.
- **Изменение ключа после вставки ломает поиск.** Если хеш объекта-ключа поменяется, найти его в таблице больше не выйдет. Поэтому ключи должны быть неизменяемыми.
- **Периодический rehash.** При заполнении таблица пересоздаётся в большем размере; отдельная вставка в этот момент дороже, но в среднем по серии вставок всё равно O(1).
