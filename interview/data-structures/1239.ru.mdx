# Когда связный список выгоднее массива?

## Суть

Связный список берут, когда часто вставляешь и удаляешь элементы в произвольной позиции, а число элементов заранее неизвестно. Он не требует сдвига соседей и растёт динамически - в отличие от массива с его непрерывной памятью и фиксированной раскладкой.

## Что ответить на собесе

"Список выигрывает у массива там, где вставки и удаления в середине частые: имея ссылку на узел, мы переставляем пару указателей за O(1), без сдвига остальных элементов. Ещё он удобен, когда объём данных неизвестен наперёд - размер меняется на лету. Плата за это - доступ по индексу O(n) (нужно пройти от головы) и накладные расходы на хранение указателей плюс плохая локальность кэша. Массив, наоборот, даёт доступ по индексу за O(1) и компактен в памяти, но вставка/удаление в середине - O(n). Выбор - это размен между быстрым доступом и дешёвой модификацией структуры."

## Как это работает

Массив держит элементы в непрерывном блоке памяти: индекс мгновенно переводится в адрес, отсюда доступ O(1). Но вставка или удаление в середине заставляет сдвигать хвост - O(n).

Связный список собран из узлов, каждый хранит данные и ссылку на следующий. Вставка после известного узла - это переустановка двух ссылок, без сдвига:

```python
class Node:
    def __init__(self, data):
        self.data = data
        self.next = None

class LinkedList:
    def __init__(self):
        self.head = None

    def insert_after(self, prev_node, new_data):
        if prev_node is None:
            raise ValueError("Опорный узел не задан")
        new_node = Node(new_data)
        new_node.next = prev_node.next
        prev_node.next = new_node

llist = LinkedList()
llist.head = Node(1)
second = Node(2)
third = Node(3)
llist.head.next = second
second.next = third

llist.insert_after(second, 4)
```

Здесь вставка узла после `second` - это две операции с указателями. В массиве на той же позиции пришлось бы двигать все последующие элементы.

## Подводные камни

- Вставка O(1) у списка - только **при уже известной ссылке на узел**. Если узел сначала надо найти по значению или индексу, добавляется O(n) на поиск.
- Доступ по индексу у списка - O(n), не O(1). Частая ошибка - считать список быстрым во всём.
- Списки **проигрывают по кэшу**: узлы разбросаны по памяти, тогда как массив читается последовательно. На практике массив нередко быстрее даже там, где асимптотика обещает обратное.
- Каждый узел тратит память на указатель(и). На мелких элементах накладные расходы заметны.
- В Python встроенный `list` - это динамический массив, а не связный список; для двусторонней очереди с дешёвыми вставками по краям есть `collections.deque`.
