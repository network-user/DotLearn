# How much does it cost to fetch a value from a dictionary by key?

## In short

Accessing a dictionary value by key is on average O(1): the hash of the key points straight to the right cell. In the pathological case of mass collisions, O(n) is theoretically possible.

## How to answer in an interview

A Python dictionary is built on a hash table. On a `d[key]` lookup the interpreter computes `hash(key)`, uses it to calculate a position in the table, and fetches the value without scanning the other pairs — hence amortized O(1). Collisions (different keys, the same hash) are resolved with open addressing, and as they pile up access degrades, in the worst case to O(n). In practice this almost never happens: the table grows and is reallocated, keeping a low load factor. The same average O(1) holds for insertion and deletion.

```python
prices = {'apple': 1, 'banana': 2, 'cherry': 3}
value = prices['banana']   # hash('banana') -> position -> value
```

Steps for `prices['banana']`:

1. `hash('banana')` is computed;
2. the hash yields the index of a slot in the table;
3. the key in the slot is compared with the target (in case of a collision);
4. the associated value is returned.

The size of the dictionary doesn't affect these steps, so the time is constant. The key must be hashable (immutable): strings, numbers, and tuples qualify, a list does not.

## Pitfalls

- O(1) is an amortized average estimate, not a guarantee for every operation: as the table grows it is rehashed, and an individual insertion may turn out more expensive.
- A poor hash function or artificially chosen keys can provoke mass collisions and a drop to O(n) (the hash-DoS vector); that's why string hashing is randomized.
- Only hashable objects can be keys. `{}[[1, 2]]` will fail with `TypeError: unhashable type`.
- Insertion order is preserved (since 3.7), but that doesn't make access an ordered lookup — it's still by hash, not by position.
