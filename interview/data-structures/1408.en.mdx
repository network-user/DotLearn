# How many operations does a dict take to find and store a value by key?

## In short

Both writing and accessing by key in a `dict` cost an amortized O(1), because under the hood lies a hash table. The theoretical worst case is O(n) under solid hash collisions, but in real code it practically never occurs.

## How to answer in an interview

A dictionary is a hash table, so on average both insertion and reading by key work in constant time O(1) and don't depend on the number of elements. Degradation to O(n) is possible only under mass collisions, when many different keys land in the same chain; in practice a good hash function and recreating the table as it grows prevent that.

## How it works

The key idea: instead of a linear scan, Python computes the element's position arithmetically.

On a write, the interpreter takes the key's hash (`hash(key)`), uses it to determine the slot index in the internal array, and puts the key-value pair there. On a read, the hash is recomputed and immediately points to the same slot - there's nothing to scan through.

```python
my_dict = {}

my_dict['key1'] = 'value1'
# hash('key1') -> slot index -> the pair is put there

value = my_dict['key1']
# hash('key1') is computed again and leads straight to 'value1'
```

Where this is especially valuable: caches, indexes, storing sessions by identifier - anywhere access goes by a unique key and the collection is large.

## Pitfalls

- **O(1) is amortized, not guaranteed.** Under solid collisions the operation degrades to O(n). That's why it's critical for keys to have a high-quality hash function.
- **A key must be hashable.** A list or other mutable object won't become a key - you'll get a `TypeError`.
- **Changing a key after insertion breaks lookup.** If the key object's hash changes, you'll no longer be able to find it in the table. That's why keys must be immutable.
- **Periodic rehash.** When the table fills up, it's recreated at a larger size; an individual insertion at that moment is more expensive, but on average across a series of insertions it's still O(1).
