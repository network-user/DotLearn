# When is a linked list better than an array?

## In short

You reach for a linked list when you frequently insert and delete elements at arbitrary positions and the number of elements isn't known in advance. It doesn't require shifting neighbors and grows dynamically - unlike an array with its contiguous memory and fixed layout.

## How to answer in an interview

"A list beats an array where insertions and deletions in the middle are frequent: with a reference to a node, we relink a couple of pointers in O(1), without shifting the rest of the elements. It's also handy when the volume of data isn't known up front - the size changes on the fly. The price is O(n) index access (you have to walk from the head) and the overhead of storing pointers plus poor cache locality. An array, on the contrary, gives O(1) index access and is compact in memory, but insertion/deletion in the middle is O(n). The choice is a trade-off between fast access and cheap structural modification."

## How it works

An array keeps elements in a contiguous block of memory: an index translates instantly into an address, hence O(1) access. But inserting or deleting in the middle forces the tail to shift - O(n).

A linked list is built from nodes, each storing data and a reference to the next. Inserting after a known node means relinking two references, with no shifting:

```python
class Node:
    def __init__(self, data):
        self.data = data
        self.next = None

class LinkedList:
    def __init__(self):
        self.head = None

    def insert_after(self, prev_node, new_data):
        if prev_node is None:
            raise ValueError("Опорный узел не задан")
        new_node = Node(new_data)
        new_node.next = prev_node.next
        prev_node.next = new_node

llist = LinkedList()
llist.head = Node(1)
second = Node(2)
third = Node(3)
llist.head.next = second
second.next = third

llist.insert_after(second, 4)
```

Here, inserting a node after `second` is two pointer operations. In an array, at the same position you would have to move all the subsequent elements.

## Pitfalls

- O(1) insertion in a list holds **only when you already have a reference to the node**. If the node first has to be found by value or index, an extra O(n) search is added.
- Index access in a list is O(n), not O(1). A common mistake is to consider a list fast at everything.
- Lists **lose on the cache**: nodes are scattered across memory, whereas an array is read sequentially. In practice an array is often faster even where the asymptotics promise the opposite.
- Each node spends memory on its pointer(s). On small elements the overhead is noticeable.
- In Python the built-in `list` is a dynamic array, not a linked list; for a double-ended queue with cheap insertions at the ends there is `collections.deque`.
