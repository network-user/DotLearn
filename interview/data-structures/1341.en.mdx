# Inside a hash table: how a key turns into an address

## In short

A hash table is an array in which a hash function suggests the position for each key-value pair. By computing the index directly, insertion, lookup, and deletion run in amortized O(1).

## How to answer in an interview

A hash table stores key-value pairs on top of an ordinary array. A hash function turns the key into a number, and the remainder of dividing it by the table size gives the bucket index. When two keys land in the same bucket, that is a collision, resolved either by chaining (the bucket holds a list of pairs) or by open addressing (you probe for the next free slot). The average complexity of the operations is O(1), but in the worst case, with a poor hash function, it degrades to O(n).

## How it works

Three building blocks: the hash function, the array of buckets, and the collision-resolution strategy. Below is a teaching implementation with chaining: each bucket is a list into which pairs are placed when their indexes coincide.

```python
class HashTable:
    def __init__(self, size):
        self.size = size
        self.table = [[] for _ in range(size)]

    def _index(self, key):
        return hash(key) % self.size

    def insert(self, key, value):
        bucket = self.table[self._index(key)]
        for i, (k, _) in enumerate(bucket):
            if k == key:
                bucket[i] = (key, value)
                return
        bucket.append((key, value))

    def search(self, key):
        for k, v in self.table[self._index(key)]:
            if k == key:
                return v
        return None
```

On insertion we do not simply add a pair; we first check whether the key already exists, otherwise the same key would be stored twice. The lookup runs only over the relevant bucket, not over the whole table.

## Pitfalls

- A key must be hashable and stable in its hash: a list cannot be a key, but a tuple can. A mutable object as a key breaks the "value → bucket" link.
- A poor hash distribution funnels all pairs into a single bucket, and O(1) turns into O(n); that is why growing the table and rehashing on fill matter.
- `hash()` in CPython is randomized between runs for strings (protection against hash-flooding), so the traversal order and the specific indexes cannot be reproduced.
