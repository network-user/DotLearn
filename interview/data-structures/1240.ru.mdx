# Стек против кучи: где живут переменные и объекты программы?

## Суть

Программа держит данные в двух разных областях памяти. Стек обслуживает вызовы функций и их локальные переменные автоматически по схеме LIFO, а куча выделяет память под объекты, которые должны пережить вызов функции, и управляется куда более гибко.

## Что ответить на собесе

Стек обслуживает кадры вызовов по LIFO, куча хранит динамические и долгоживущие объекты.

- Стек: на каждый вызов кладётся фрейм с локальными переменными и адресом возврата, при выходе снимается; выделение мгновенное, но размер ограничен - отсюда stack overflow при глубокой рекурсии.
- Куча: область для данных, которые должны существовать дольше одного вызова; доступ медленнее из-за работы через указатели, а ошибки с освобождением дают утечки.
- В Python имена-ссылки лежат в кадрах, а сами объекты - в куче.
- Время жизни объектов определяет подсчёт ссылок и сборщик мусора.

## Как это работает

Стек растёт и сжимается синхронно с вложенностью вызовов. Представьте стопку тарелок: новый кадр кладётся сверху при входе в функцию и снимается сверху при выходе. Именно поэтому локальные переменные исчезают сразу после `return` - их кадр уничтожается. Доступ к стеку быстрый, потому что нужно лишь сдвинуть указатель вершины.

Куча устроена сложнее: блоки памяти выделяются и освобождаются в произвольном порядке, по запросу. Это похоже на аренду - вы занимаете место, когда оно нужно, и отдаёте, когда закончили. Подходит для структур, размер которых меняется по ходу работы, и для объектов, которые должны жить после возврата из функции.

```python
def example():
    a = 10
    b = [1, 2, 3, 4, 5]
    return b

result = example()
```

Имена `a` и `b` - это ссылки в кадре функции `example`. Сам кадр исчезает после возврата, поэтому имя `a` пропадает. Но объект-список, на который указывал `b`, остаётся в куче живым, пока на него есть хоть одна ссылка - в данном случае через `result`.

## Подводные камни

- В Python нельзя выбрать вручную, что положить в стек, а что в кучу: почти все объекты живут в куче, а в кадрах хранятся только ссылки на них. Аналогия со стеком и кучей здесь концептуальная, не буквальная.
- Глубокая или бесконечная рекурсия упирается в лимит стека и даёт `RecursionError`, а не молчаливую порчу памяти - Python ставит защитный предел заранее.
- Утечки памяти в Python обычно вызваны не "забытым free", а долгоживущими ссылками: объект не освобождается, пока кто-то на него ссылается.
- Циклические ссылки подсчёт ссылок сам не разорвёт - их вычищает отдельный сборщик мусора, и до его прохода память остаётся занятой.
