# What does a hash table do when different keys produce the same hash?

## In short

When different keys produce the same hash, it is called a collision. It is unavoidable, because there are infinitely many inputs but only a finite number of hashes. A hash table resolves collisions with chaining or open addressing, so elements are still stored and looked up correctly.

## How to answer in an interview

A collision is when the hash function returns the same index for two different keys. Collisions cannot be avoided entirely; they can only be handled gracefully. There are two classic approaches: chaining, where a single cell holds a list of all elements with that hash, and open addressing, where if a cell is occupied the element goes into the next free slot according to a probing rule. On average access stays O(1), but with a poor hash function it degrades to O(n).

## How it works

Hashing turns data of arbitrary size into a fixed-length value. The cell index is usually computed as `hash(key) % size`. When two keys map to the same index, you need a resolution strategy. Chaining stores a list in each cell:

```python
class HashTable:
    def __init__(self, size):
        self.size = size
        self.table = [[] for _ in range(size)]

    def hash_function(self, key):
        return len(key) % self.size

    def insert(self, key, value):
        index = self.hash_function(key)
        self.table[index].append((key, value))

    def search(self, key):
        index = self.hash_function(key)
        for k, v in self.table[index]:
            if k == key:
                return v
        return None
```

A cell is a list of pairs. On insertion an element is appended to the list at the right index; on lookup the list is scanned until the key matches. That way several keys with the same hash coexist peacefully.

## Pitfalls

- The hash function here is a teaching example (`len(key)`): in practice you use functions with good distribution, otherwise all keys collapse into a single chain.
- With poor distribution or a high load factor, the complexity drops from O(1) to O(n).
- A hash collision is not the same as key equality: you always need a final `k == key` check, so an object must correctly implement both `__hash__` and `__eq__`.
- With open addressing deletion is harder: you cannot just "zero out" a cell, or the probing sequence breaks - deletion markers are used instead.
