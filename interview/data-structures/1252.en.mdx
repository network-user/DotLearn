# Why a dynamic array grows on its own and how it differs from a static one

## In short

A dynamic array is a collection in contiguous memory that changes its own capacity. When it runs out of room, it grabs a larger block and moves there; when there are few elements left, it can hand the extra memory back. That is why you can work with it without knowing the final size in advance.

## How to answer in an interview

A dynamic array stores elements contiguously in memory, like a static one, but unlike a static array it can resize on the fly. When the allocated capacity is full, it allocates a new block (usually twice as large), copies the old elements there, and keeps going. Thanks to the capacity-doubling strategy, appending at the end stays amortized O(1): the rare expensive copies are spread out across many cheap additions. In Python, `list` is exactly such a structure.

## How it works

The key difference from a static array is the separation of two quantities: the actual number of elements (size) and the allocated capacity (capacity). Capacity is always at least size, and the difference is the headroom for future insertions.

1. **Start.** The array is created with a small initial capacity.
2. **Insertion.** If there is a free slot, the element goes into it in O(1). If there is no free space, expansion kicks in.
3. **Expansion.** A new block is allocated (usually ×2), all elements are moved there, and the old block is freed. This is a one-off O(n) operation, but it happens rarely.
4. **Shrinking.** On mass deletion, the implementation may switch to a smaller block so as not to hold memory in vain.

Because of the geometric growth of capacity, n additions together perform on the order of n copy operations rather than n², which is where the amortized constant comes from.

```python
dynamic_array = []

dynamic_array.append(1)
dynamic_array.append(2)
dynamic_array.append(3)

dynamic_array.pop()

print(dynamic_array)  # [1, 2]
```

## Pitfalls

- **Insertion at the beginning or middle is O(n)**, not O(1): every element to the right shifts over. The amortized constant applies only to appending at the end.
- **Expansion is not free**: a single `append` that lands on the capacity boundary runs in O(n). If timing predictability matters (real time), this is worth taking into account.
- **Capacity growth costs memory**: after doubling, up to half of the block may sit empty, so under the hood memory is spent with headroom.
- **`pop()` from the end is O(1)**, while `pop(0)` is O(n) due to the shift; for a queue use `collections.deque`, not a list.
