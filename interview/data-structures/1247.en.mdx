# Cost of inserting into a Python list: O(1) or O(n)?

## In short

The cost of insertion depends on the position. `append()` at the end is amortized O(1), because a list is a dynamic array. `insert(i, x)` into the middle or the front is O(n): every element to the right of the insertion point shifts by one position.

## How to answer in an interview

A Python list is a dynamic array of pointers in a contiguous block of memory. That's why indexed access is O(1), but insertion depends on the location. `append()` usually just writes the element into the free tail in O(1); occasionally there isn't enough room, and Python allocates a larger block and copies everything into it in O(n), but these expensive resizes happen so rarely that on average (amortized) append stays O(1). Meanwhile `insert(i, x)` at an arbitrary position is forced to shift right all elements after index i in order to free up a cell — that's O(n). Inserting at the front, `insert(0, x)`, is the worst case: every element shifts. If I need frequent insertions at the front, I reach for `collections.deque`, which is O(1) on both ends.

## How it works

A list is reallocated with spare room: the capacity grows not by one but by a multiple, so the amortized cost of `append` stays constant.

```python
my_list = [1, 2, 3, 4, 5]

my_list.append(6)        # O(1) amortized: write into the tail
my_list.insert(2, 'a')   # O(n): shift elements from index 2 rightward
# -> [1, 2, 'a', 3, 4, 5, 6]
```

With `insert(2, 'a')`, elements `3, 4, 5, 6` physically move one cell to the right to make room for `'a'`. The closer the insertion is to the start, the more elements move.

## Pitfalls

- **`append` is O(1), but not strictly so**. Individual calls during a resize cost O(n); the correct phrasing is "amortized O(1)".
- **`insert(0, x)` is an anti-pattern in a loop**. n insertions at the front give O(n²); for a queue use `deque.appendleft`, which is O(1).
- **`pop(0)` is also O(n)** for the same reason — shifting all elements left, whereas `pop()` from the end is O(1).
- **Memory on resize**. A list keeps spare capacity, so it takes up more than the sum of its elements; that's the price of fast appends.
