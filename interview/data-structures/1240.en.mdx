# Stack vs heap: where do a program's variables and objects live?

## In short

A program keeps data in two different memory regions. The stack serves function calls and their local variables automatically in a LIFO scheme, while the heap allocates memory for objects that must outlive a function call and is managed far more flexibly.

## How to answer in an interview

The stack is memory for call frames: on each function call a frame with local variables and a return address is pushed, and on exit the frame is popped. It works strictly LIFO, allocation and freeing are instant, but the size is limited - hence a stack overflow on overly deep recursion. The heap is the region for dynamic data and long-lived objects; whatever must exist longer than a single call goes there. Access to the heap is slower because it works through pointers, and freeing errors lead to leaks. In Python the name-references live in frames, while the objects themselves live in the heap, and their lifetime is determined by reference counting and the garbage collector.

## How it works

The stack grows and shrinks in sync with call nesting. Picture a stack of plates: a new frame is placed on top on entering a function and removed from the top on exit. This is exactly why local variables disappear right after `return` - their frame is destroyed. Access to the stack is fast, because you only need to move the top pointer.

The heap is arranged in a more complex way: blocks of memory are allocated and freed in arbitrary order, on request. It's like renting - you take space when you need it and give it back when you're done. It suits structures whose size changes during execution, and objects that must live after a function returns.

```python
def example():
    a = 10
    b = [1, 2, 3, 4, 5]
    return b

result = example()
```

The names `a` and `b` are references in the frame of the function `example`. The frame itself disappears after the return, so the name `a` is gone. But the list object that `b` pointed to remains alive in the heap as long as at least one reference to it exists - in this case through `result`.

## Pitfalls

- In Python you can't manually choose what to put on the stack and what on the heap: almost all objects live in the heap, while the frames hold only references to them. The stack-and-heap analogy here is conceptual, not literal.
- Deep or infinite recursion hits the stack limit and gives a `RecursionError`, not a silent memory corruption - Python sets a protective limit in advance.
- Memory leaks in Python are usually caused not by a "forgotten free" but by long-lived references: an object isn't freed while something still references it.
- Reference counting won't break circular references on its own - they are cleaned up by a separate garbage collector, and until its pass the memory stays occupied.
