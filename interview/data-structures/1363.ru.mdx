# Гарантирует ли стандартный dict порядок обхода ключей?

## Суть

Да. С Python 3.7 сохранение порядка вставки ключей в обычном `dict` - это гарантия языка (в CPython 3.6 уже работало как деталь реализации). Для более старых версий ту же роль играл `collections.OrderedDict`.

## Что ответить на собесе

Начиная с Python 3.7 порядок ключей в `dict` гарантированно совпадает с порядком их добавления - это часть спецификации языка, а не случайность реализации. До 3.6 порядок был неопределённым, и для предсказуемого обхода использовали `OrderedDict`. Сегодня `OrderedDict` нужен лишь в особых случаях: когда важны его специфичные методы (`move_to_end`, `popitem(last=...)`) или сравнение с учётом порядка.

## Как это работает

Обычный словарь запоминает последовательность вставки, и итерация идёт ровно в этом порядке:

```python
my_dict = {}
my_dict['a'] = 1
my_dict['b'] = 2
my_dict['c'] = 3

for key in my_dict:
    print(key, my_dict[key])
```

Вывод пойдёт в порядке `a`, `b`, `c` - так, как ключи добавлялись.

`OrderedDict` нужен, когда требуется именно упорядоченное поведение как контракт или его дополнительные операции:

```python
from collections import OrderedDict

ordered = OrderedDict()
ordered['a'] = 1
ordered['b'] = 2
ordered.move_to_end('a')   # 'a' уходит в конец
print(list(ordered))       # ['b', 'a']
```

## Подводные камни

- Порядок - это порядок **вставки**, а не сортировки. Повторное присваивание существующему ключу позицию не меняет; чтобы переместить ключ, его надо удалить и добавить заново.
- Сравнение на равенство: для обычного `dict` `{'a':1,'b':2} == {'b':2,'a':1}` истинно (порядок не учитывается), а у `OrderedDict` сравнение порядко-зависимое.
- Удаление и повторная вставка ключа поместит его в конец - это легко упустить при инкрементальных обновлениях.
- Гарантия порядка не делает `dict` отсортированным: для упорядоченного по значениям/ключам обхода нужен `sorted(...)`.
