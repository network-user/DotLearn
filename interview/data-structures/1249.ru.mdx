# Сколько стоит достать значение из словаря по ключу?

## Суть

Доступ к значению словаря по ключу в среднем O(1): хеш ключа сразу указывает на нужную ячейку. В патологическом случае массовых коллизий теоретически возможно O(n).

## Что ответить на собесе

Словарь Python построен на хеш-таблице: хеш ключа сразу ведёт к значению, отсюда амортизированное O(1).

- При обращении `d[key]` интерпретатор считает `hash(key)`, по нему вычисляет позицию в таблице и достаёт значение, не перебирая остальные пары.
- Коллизии (разные ключи, один хеш) разрешаются открытой адресацией, и при их нагромождении доступ деградирует, в худшем случае до O(n).
- На практике это почти не встречается: таблица растёт и переразмещается, удерживая низкий коэффициент заполнения.
- То же O(1) в среднем верно для вставки и удаления.

```python
prices = {'apple': 1, 'banana': 2, 'cherry': 3}
value = prices['banana']   # hash('banana') -> позиция -> значение
```

Шаги при `prices['banana']`:

1. вычисляется `hash('banana')`;
2. из хеша получается индекс слота в таблице;
3. ключ в слоте сравнивается с искомым (на случай коллизии);
4. возвращается связанное значение.

Размер словаря на эти шаги не влияет, поэтому время константно. Ключ обязан быть хешируемым (неизменяемым): строки, числа, кортежи подходят, список — нет.

## Подводные камни

- O(1) — амортизированная средняя оценка, а не гарантия для каждой операции: при росте таблица перехешируется, и отдельная вставка может оказаться дороже.
- Плохая хеш-функция или искусственно подобранные ключи могут спровоцировать массовые коллизии и просадку до O(n) (вектор hash-DoS); поэтому хеш строк рандомизируется.
- Только хешируемые объекты могут быть ключами. `{}[[1, 2]]` упадёт с `TypeError: unhashable type`.
- Порядок вставки сохраняется (с 3.7), но это не делает доступ упорядоченным поиском — он по-прежнему по хешу, а не по позиции.
