# Что под капотом у dict в Python и почему поиск по ключу такой быстрый?

## Суть

Словарь Python реализован поверх хеш-таблицы. Ключ прогоняется через хеш-функцию, та указывает на ячейку внутреннего массива, и благодаря этому вставка, поиск и удаление по ключу в среднем выполняются за O(1).

## Что ответить на собесе

`dict` - это хеш-таблица: хеш ключа ведёт прямо к нужной ячейке, без перебора, отсюда среднее `O(1)`.

- При записи Python вычисляет `hash(key)`, по нему определяет индекс ячейки во внутреннем массиве и сохраняет запись; поиск работает так же.
- Если два разных ключа дают один индекс - это коллизия, CPython разрешает её открытой адресацией (пробированием).
- Ключи обязаны быть хешируемыми и неизменяемыми по хешу: список ключом быть не может, а кортеж из неизменяемых элементов может.
- В худшем случае при массовых коллизиях операции вырождаются в `O(n)`, но на практике это редкость; и отдельно: с Python 3.7 словарь гарантированно сохраняет порядок вставки.

Когда массив заполняется примерно на две трети, словарь увеличивает ёмкость и перераскладывает записи - это держит коллизии редкими, а среднее время операций константным.

## Подводные камни

- Ключом может быть только хешируемый объект. `list`, `dict` и `set` под ключ не подходят; `tuple` подойдёт, только если все его элементы тоже неизменяемы.
- O(1) - это среднее, а не гарантия. При массовых коллизиях операции деградируют, поэтому важно, чтобы `__hash__` и `__eq__` были согласованы: равные объекты обязаны иметь равный хеш.
- Изменили объект так, что его хеш поменялся после вставки в словарь, - ключ "потеряется", найти его станет нельзя.
- Порядок вставки сохраняется начиная с Python 3.7, но полагаться на это как на упорядоченность по ключам нельзя - это именно порядок добавления, а не сортировка.
- Словарь платит памятью за скорость: внутренний массив держится с запасом свободных ячеек, чтобы коллизии оставались редкими.
