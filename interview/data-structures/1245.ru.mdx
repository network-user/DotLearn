# Merge sort: как работает сортировка через деление и слияние?

## Суть

Сортировка слиянием — это «разделяй и властвуй»: массив рекурсивно режется пополам до одиночных элементов, а затем половины попарно сливаются в отсортированные куски. Алгоритм стабилен и гарантирует O(n log n) в худшем случае ценой O(n) дополнительной памяти.

## Что ответить на собесе

Merge sort делит массив надвое, рекурсивно сортирует каждую половину и сливает их за линейное время.

- Глубина рекурсии — log n уровней, на каждом слияние просматривает все n элементов, отсюда O(n log n) и в среднем, и в худшем случае — в отличие от быстрой сортировки с худшим O(n²).
- Алгоритм стабильный: равные элементы сохраняют исходный порядок.
- Платим за это O(n) памяти под временный буфер слияния.
- Из-за предсказуемой сложности и стабильности его выбирают для внешней сортировки больших данных и как основу для гибридов вроде Timsort.

## Как это работает

Три шага: разделение до базового случая, слияние двух упорядоченных частей, повтор снизу вверх.

```python
def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])
    right = merge_sort(arr[mid:])
    return merge(left, right)

def merge(left, right):
    result = []
    i = j = 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i])
            i += 1
        else:
            result.append(right[j])
            j += 1
    result.extend(left[i:])
    result.extend(right[j:])
    return result
```

Условие `left[i] <= right[j]` (нестрогое) и обеспечивает стабильность: при равенстве первым берётся элемент из левой части.

Возьмём `[38, 27, 43, 3, 9, 82, 10]`:

- Делим до одиночек: `[38] [27] [43] [3] [9] [82] [10]`.
- Сливаем пары: `[27, 38]`, `[3, 43]`, `[9, 82]`, `[10]`.
- Сливаем дальше: `[3, 27, 38, 43]`, `[9, 10, 82]`.
- Финал: `[3, 9, 10, 27, 38, 43, 82]`.

## Подводные камни

- **Лишняя память O(n)**. Срезы `arr[:mid]` создают копии, плюс буфер слияния. На очень больших данных это ощутимо, в отличие от in-place quicksort или heapsort.
- **Стабильность легко сломать**. Если поменять `<=` на `<` в сравнении, равные элементы поменяются местами и стабильность пропадёт.
- **Рекурсия не бесплатна**. Для маленьких подмассивов накладные расходы на вызовы велики; практические реализации (Timsort) переключаются на вставки для коротких кусков.
- **O(n log n) — это и потолок, и пол**. Merge sort не ускоряется на уже отсортированных данных, в отличие от Timsort или insertion sort.
