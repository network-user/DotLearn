# Linear versus quadratic complexity: where the line is drawn

## In short

O(n) beats O(n²) because it handles input growth better. For a linear algorithm, time grows in proportion to the number of elements; for a quadratic one, as the square of that number. On small data the difference is unnoticeable, but as n grows the square blows the linear algorithm to dust.

## How to answer in an interview

Big-O describes how runtime scales as the input size n grows, ignoring constants. O(n) means that doubling the input roughly doubles the time. O(n²) means that doubling the input quadruples the time. So on a thousand elements a linear algorithm does ~1000 steps while a quadratic one does ~1,000,000: a thousandfold difference, and it only grows from there. In an interview it is important to stress: O speaks about asymptotics, not about specific seconds - on tiny n a quadratic algorithm with a small constant can even be faster, and what decides things is the behavior on large volumes.

- **O(n) - linear.** A single pass over the data. Finding the maximum in an array of n elements requires n comparisons: time is proportional to n.
- **O(n²) - quadratic.** Usually this is a nested loop where, for each element, all the others are iterated over. Bubble sort essentially compares each with each, hence n·n.

Why linear is preferable at scale:

1. **Growth.** As n increases, O(n²) time accelerates many times faster than O(n).
2. **Practice.** On large arrays a linear algorithm returns the result noticeably sooner, which directly affects the system's responsiveness.
3. **Resources.** Fewer steps means less load on the CPU; on mobile and embedded devices this is critical.

### Linear search - O(n)

```python
def linear_search(arr, target):
    for i in range(len(arr)):
        if arr[i] == target:
            return i
    return -1
```

In the worst case we check all n elements - O(n).

### Bubble sort - O(n²)

```python
def bubble_sort(arr):
    n = len(arr)
    for i in range(n):
        for j in range(0, n - i - 1):
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]
```

Nested loops give n·n comparisons - O(n²).

## Pitfalls

- **Big-O is about asymptotics, not seconds.** An O(n) algorithm with a huge constant on a small n can lose to an O(n²) one with a tiny constant. Constants are hidden, and for small inputs they sometimes decide the outcome.
- **The worst, average, and best cases differ.** For the same bubble sort, the best case on nearly sorted data can be brought down to O(n) with an early exit; bare "O(n²)" does not reflect this.
- **Space complexity is separate from time.** An algorithm can be fast in time but greedy with memory - evaluate both axes.
- **Not every O(n²) can be mindlessly replaced with O(n).** Sometimes O(n log n) or O(n) require extra memory or preprocessing; "lower complexity" does not always mean "strictly better for the task".
