# What's under the hood of dict and set: why is access almost always O(1)?

## In short

Both dict and set are built on a hash table. A hash function translates a key into an index in an array, so addition, removal, and lookup run in amortized O(1) - without scanning all elements.

## How to answer in an interview

Both types are hash tables. A key is run through `hash()`, the result is reduced to an index in the bucket array, and the element is placed exactly there. That is why lookup does not depend on the collection size and on average costs O(1). When two keys fall into the same bucket - that is a collision; CPython resolves it with open addressing (probing). When the table fills up it grows and elements are relocated - this is amortized, so on average it is still O(1). The only difference between a set and a dict is that a set stores a key without a value.

## How it works

A hash table is an array of cells (buckets). To place an element, Python computes `hash(key)`, trims it to the table size, and gets the bucket index.

```python
d = {}
d["key"] = "value"
```

- `d = {}` allocates the internal array for pairs.
- `d["key"] = "value"` computes `hash("key")`, reduces it to an index, and writes the pair `("key", "value")` there.

Lookup follows the same scheme: compute the key's hash, jump to the right bucket, compare the key with `==`, and return the value. No scanning of all elements.

**Collisions.** If two different keys produce the same index, the bucket is occupied. CPython uses open addressing: it tries the next positions in a deterministic sequence until it finds a free slot or the target key. So in the worst case lookup degrades, but with a reasonable table load it stays close to O(1).

```python
seen = set()
seen.add("element")  # hash("element") -> index -> bucket
print("element" in seen)  # same path, O(1)
```

A set works the same way, but the bucket holds only a key - there is no value. Hence its role: remove duplicates, quickly answer the question "is this element here?".

## Pitfalls

- O(1) is amortized and average, not guaranteed. A bad hash (many collisions) or a rare pathological case give degradation to O(n).
- Keys and elements must be hashable and stable in hash: you cannot put a list there, and a mutable object with a changing hash will break the table.
- A common myth is "dict uses chaining." CPython uses open addressing, not lists in buckets.
- Table growth (resize) and element relocation are a real cost that is only averaged out. For peak insertion it is worth keeping in mind.
