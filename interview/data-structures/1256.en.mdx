# Greedy strategy: when the locally best choice fails at the global level

## In short

A greedy algorithm commits, at each step, to the decision that looks the most beneficial right now, and never goes back to it. This gives simplicity and speed, but the global optimum is guaranteed only when the problem has a special property - the optimality of local choices.

## How to answer in an interview

A greedy algorithm builds the answer step by step: at each stage it takes the locally best available option and moves on without replaying previous steps. It works correctly only if the problem has the greedy-choice property (a local optimum leads to a global one) and optimal substructure. Classic correct examples are Kruskal's and Prim's algorithms, the fractional knapsack, and Huffman coding. But for problems like the 0/1 knapsack or making change with arbitrary denominations, greed breaks down, and there you need dynamic programming. The main thing I would stress in an interview: greed must be proven, not assumed.

## How it works

The principle is simple: at each stage we pick the best of what is available and commit to the choice for good. The algorithm counts on a chain of such steps to add up to an optimal result on its own.

A handy analogy is finding the exit from a maze. At every fork you turn toward what seems closer to the exit, you do not backtrack, and you do not reconsider past turns.

Where greed really does work correctly:

1. **Fractional knapsack.** Items can be split into parts, so it makes sense to sort them by unit value (price per unit of weight) and take from the top down, topping off the remainder with a fractional share. Here greed is provably optimal.
2. **Kruskal's algorithm.** A minimum spanning tree is built by adding edges in increasing order of weight; an edge is skipped if it closes a cycle. Correctness rests on the cut property.
3. **Huffman coding.** At each step the two rarest symbols are merged, which yields an optimal prefix code.

A small example of greedy coverage - how many coins are needed for an amount with standard denominations:

```python
def greedy_coins(amount, coins):
    coins = sorted(coins, reverse=True)
    used = []
    for coin in coins:
        while amount >= coin:
            amount -= coin
            used.append(coin)
    return used

print(greedy_coins(63, [25, 10, 5, 1]))
```

With a standard set of denominations, such greedy change-making gives the minimum number of coins - this is a property of the specific denomination system, not of greed in general.

## Pitfalls

- **Greed cannot be assumed - it is proven.** Without proof of the greedy-choice property, the solution easily turns out to be suboptimal on some test.
- **The 0/1 knapsack.** When items cannot be split, sorting by unit value is deceptive: the algorithm may fill the capacity with cheap items and miss a more profitable combination. Here you need dynamic programming.
- **Change with "inconvenient" denominations.** For a set like `[1, 3, 4]`, greed on the amount 6 gives `4 + 1 + 1` (three coins), although the optimum is `3 + 3` (two). A counterexample is built in a couple of minutes.
- **Irreversibility of decisions.** A greedy algorithm does not roll back choices, so one early mistake carries through to the end. If the problem requires reconsideration, that is a signal toward DP or backtracking search.
- **Greed ≠ guaranteed speed.** Sorting the input often adds O(n log n), and the greedy pass itself is not always the fastest approach; estimate complexity honestly.
