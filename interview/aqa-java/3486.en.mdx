# HashMap for fixture lookup by id

## In short

HashMap stores unique keys with hash bucket lookup: amortized O(1) get/put. Collisions use chains or trees (Java 8+). One null key is allowed; null values are allowed. In tests I map response id to entity for O(1) asserts instead of nested loops.

## How to answer in an interview

- `getOrDefault` for optional fields.
- `computeIfAbsent` for lazy fixture factories.
- Load factor 0.75 resize doubles buckets.
- LinkedHashMap access-order for LRU cache simulation.
- Honestly: hash DoS is not a unit-test topic, but immutable keys are good practice.

## How it works

```
Map<Long, UserDto> byId = new HashMap<>();
for (UserDto u : response) {
  byId.put(u.id(), u);
}
assert byId.get(42L).email().equals("qa@test");
```

Merge:

```
map.merge(key, 1, Integer::sum);
```

Python dict:

```python
def index_by_id(rows: list[dict]) -> dict[int, dict]:
    return {row["id"]: row for row in rows}


def test_hash_lookup():
    data = index_by_id([{"id": 1, "v": "a"}, {"id": 2, "v": "b"}])
    assert data[2]["v"] == "b"
```

## Pitfalls

**Mutable key after put.** Entry lost in wrong bucket.

**equals without hashCode.** Broken contract; get misses.

**ConcurrentHashMap null key.** Not allowed unlike HashMap.

**toMap duplicate in stream.** Exception without merge function.

**Mod count during iteration.** Fail-fast entrySet iterator.
