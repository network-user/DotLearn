# Модификаторы доступа и package-private

## Суть

Java задаёт видимость через `public`, `protected`, `private` и package-private (без модификатора). Package-private ограничивает доступ классами того же пакета. В тестовом коде модификаторы отделяют публичный API фреймворка от внутренних хелперов и защищают инкапсуляцию page object.

## Что ответить на собесе

- `private` для локаторов и wait utils внутри page class.
- `public` только стабильный API шагов сценария.
- `protected` для hooks в базовом `BaseTest` для наследников в том же пакете/иерархии.
- Package-private util классы в `com.app.tests.internal` не торчат наружу.
- Честно: тесты в другом пакете не видят package-private, нужен public API или same package.

## Как это работает

```
package com.acme.tests.ui;

class LocatorKit {
  static By dataTest(String value) {
    return By.cssSelector("[data-test='" + value + "']");
  }
}

public class LoginPage {
  private final By submit = LocatorKit.dataTest("login-submit");

  public void submit(WebDriver driver) {
    driver.findElement(submit).click();
  }
}
```

`LocatorKit` package-private: виден `LoginPage`, не виден из `com.acme.tests.api`.

Python mirror (module convention):

```python
def _locator(value: str) -> str:
    return f"[data-test='{value}']"


class LoginPage:
    def __init__(self) -> None:
        self._submit = _locator("login-submit")
```

## Подводные камни

**Тест в другом пакете.** Не компилируется доступ к package-private helper.

**protected + другой пакет.** Доступ только через наследника, не через композицию.

**public поля.** Ломают инкапсуляцию, ловушка для прямых assert на state.

**module system JPMS.** exports/opens в module-info меняют видимость vs classic classpath.

**Reflection bypass.** Можно, но ломает контракт и скрывает design smell.
