# Intermediate operations in the Stream API

## In short

Intermediate ops return a new Stream and stay lazy: `filter`, `map`, `flatMap`, `sorted`, `distinct`, `limit`, `skip`, `peek`. They compose in a chain and run only at a terminal. In test helpers they filter flaky cases, normalize DTOs, and dedupe keys before asserts.

## How to answer in an interview

- `filter` drops SKIP statuses before aggregation.
- `map` projects entities to ids for set comparison.
- `flatMap` flattens nested lists from API responses.
- `distinct` plus `sorted` stabilizes order-independent asserts.
- Honestly: a long chain without named methods reads worse than a private helper.

## How it works

```
List<String> active = users.stream()
    .filter(u -> u.isActive())
    .map(User::email)
    .distinct()
    .sorted()
    .limit(10)
    .collect(Collectors.toList());
```

skip and flatMap:

```
List<String> tokens = batches.stream()
    .skip(1)
    .flatMap(b -> b.tokens().stream())
    .collect(Collectors.toList());
```

Python pipeline:

```python
def active_emails(users: list[dict]) -> list[str]:
    seen = set()
    result = []
    for user in sorted(u for u in users if u["active"]):
        email = user["email"]
        if email not in seen:
            seen.add(email)
            result.append(email)
        if len(result) == 10:
            break
    return result


def test_intermediate_chain():
    data = [
        {"active": True, "email": "b@test"},
        {"active": False, "email": "x@test"},
        {"active": True, "email": "a@test"},
    ]
    assert active_emails(data) == ["a@test", "b@test"]
```

## Pitfalls

**distinct without equals/hashCode.** Objects will not collapse.

**sorted comparator with nulls.** NPE on nullable fields.

**limit short-circuit.** Downstream filters may never see later elements.

**peek left in prod asserts.** Debug peek floods CI logs.

**flatMap on null collection.** NPE; use an Optional.stream pattern instead.
