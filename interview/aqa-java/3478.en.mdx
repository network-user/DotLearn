# Hierarchy of the Java Collections Framework

## In short

The Java Collections Framework flows from `Iterable` to `Collection` with branches `List`, `Set`, and `Queue`. `Map` stands apart (does not extend Collection) and stores key-value pairs. Implementations include `ArrayList`, `HashSet`, `PriorityQueue`, `HashMap`, and more. For AQA you pick structures for asserts: ordering, uniqueness, lookup by fixture id.

## How to answer in an interview

- List allows duplicates and indexed access; Set is unique; Map maps id to entity.
- I prefer `ArrayDeque` over legacy `Stack` for runner task queues.
- `LinkedHashMap` preserves insertion order in snapshot diffs.
- Generics like `List<String>` catch type errors at compile time, not in runtime asserts.
- Honestly: synchronized `Vector`/`Hashtable` are unnecessary without shared mutable state across test threads.

## How it works

```
List<String> cases = new ArrayList<>();
Set<String> tags = new HashSet<>();
Map<String, Integer> retries = new HashMap<>();
Queue<String> queue = new ArrayDeque<>();
```

Assert helper example:

```
Map<Long, TestCase> byId = results.stream()
    .collect(Collectors.toMap(TestCase::id, Function.identity()));
```

Python mental map:

```python
HIERARCHY = {
    "List": "ordered, duplicates",
    "Set": "unique, no index contract",
    "Map": "key-value, separate interface",
}


def test_collections_hierarchy():
    assert "duplicates" in HIERARCHY["List"]
    assert HIERARCHY["Map"].startswith("key")
```

## Pitfalls

**Raw types in legacy test utils.** `(List) obj` without generics loses safety.

**HashSet order assert.** Order is not guaranteed; compare as a Set.

**Map duplicate key in collect.** Stream toMap without merge throws.

**SubList view mutation.** Changes the parent list and flakes snapshots.

**Queue poll vs remove.** Empty queue: null vs exception.
