# Применение интерфейсов в коде

## Суть

Интерфейс объявляют через `interface`, класс подключает его через `implements` и обязан реализовать abstract методы. Вызов идёт через тип интерфейса, что даёт полиморфизм и подмену реализаций в тестах. Default-методы несут общую логику без базового класса.

## Что ответить на собесе

- `class ChromeDriverAdapter implements WebDriver` для изоляции Selenium API.
- В DI передаю `Clock` interface, в тесте подставляю fixed clock.
- Несколько interfaces на одном классе: `implements Runnable, AutoCloseable`.
- `@Override` на каждом hook для compile-time проверки сигнатуры.
- Честно: default method в interface меняет binary compatibility, осторожно в shared libs.

## Как это работает

Полиморфный вызов:

```
interface StepReporter {
  void onPass(String name);
  default void onSkip(String name) { onPass(name + " (skipped)"); }
}

class JsonReporter implements StepReporter {
  public void onPass(String name) { /* write json */ }
}

class NoOpReporter implements StepReporter {
  public void onPass(String name) { }
}

void execute(StepReporter reporter) {
  reporter.onPass("login");
}
```

Python mirror:

```python
from typing import Protocol


class StepReporter(Protocol):
    def on_pass(self, name: str) -> None: ...


class JsonReporter:
    def on_pass(self, name: str) -> None:
        pass


def execute(reporter: StepReporter) -> None:
    reporter.on_pass("login")
```

## Подводные камни

**public на методах interface.** Забытый modifier в старых стилях ломает compile.

**Diamond default methods.** Два default с одной сигнатурой требуют явного resolve.

**Mock только interface.** Класс без interface сложнее stub без PowerMock.

**Слишком широкий implements.** Тестовый double вынужден реализовать лишние методы.

**Сериализация proxy.** Dynamic proxy по interface не всегда equals-friendly в assert.
