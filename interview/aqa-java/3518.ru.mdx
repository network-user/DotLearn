# Зачем нужны интерфейсы в Java

## Суть

Интерфейс задаёт контракт методов без обязательной реализации в самом типе. Класс может реализовать несколько интерфейсов и подставляться полиморфно там, где ожидается общий тип. В AQA-фреймворках это основа для сменяемых драйверов, репортов и listener hooks.

## Что ответить на собесе

- Интерфейс описывает «что умеет», не «как устроен» внутри.
- Один класс: `implements WebDriver, TakesScreenshot` вместо множественного наследования классов.
- В тестах принимаю `ReportSink`, а не конкретный Allure adapter.
- Default/static методы в интерфейсе дают shared helper без base class.
- Честно: интерфейс не заменяет композицию, если контракт раздувается до god-interface.

## Как это работает

Контракт действия:

```
interface Notifier {
  void send(String message);
  default void ping() { send("ping"); }
}

class SlackNotifier implements Notifier {
  public void send(String message) { /* ... */ }
}

class EmailNotifier implements Notifier {
  public void send(String message) { /* ... */ }
}

void runSuite(Notifier sink) {
  sink.send("suite started");
}
```

Один runner вызывает любую реализацию через общий тип.

Python mirror:

```python
from typing import Protocol


class Notifier(Protocol):
    def send(self, message: str) -> None: ...


class SlackNotifier:
    def send(self, message: str) -> None:
        pass


def run_suite(sink: Notifier) -> None:
    sink.send("suite started")
```

## Подводные камни

**Контракт без семантики.** Метод `close()` в двух интерфейсах с разным смыслом путает реализацию.

**Fat interface.** Десять optional методов ломают моки и stub-классы.

**equals по интерфейсу.** Сравнение реализаций без явного контракта даёт сюрпризы в коллекциях.

**Default method conflict.** Две версии default с одинаковой сигнатурой требуют override в классе.

**Тест double без поведения.** Пустой stub `implements` проходит compile, но падает в runtime на NPE.
