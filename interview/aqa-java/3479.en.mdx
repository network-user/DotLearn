# Comparable and Comparator for sorting in tests

## In short

`Comparable<T>` defines natural order inside the class via `compareTo`. `Comparator<T>` defines external order via `compare(a,b)` without changing the DTO. `Collections.sort(list)` uses comparable; a second argument or `list.sort(comparator)` changes the criterion. In asserts I compare sorted lists regardless of API insertion order.

## How to answer in an interview

- DTO implements Comparable by primary key for a stable default sort.
- `Comparator.comparing(TestRun::startedAt)` for a flaky timeline report.
- Multiple criteria: `thenComparing`.
- Null-safe: `Comparator.nullsLast`.
- Honestly: compareTo inconsistent with equals breaks TreeSet dedup in test fixtures.

## How it works

Comparable:

```
public record CaseResult(int id, int ms) implements Comparable<CaseResult> {
  public int compareTo(CaseResult other) {
    return Integer.compare(this.id, other.id);
  }
}
```

Comparator:

```
Comparator<CaseResult> byDuration = Comparator.comparingInt(CaseResult::ms);
results.sort(byDuration);
```

Python mirror:

```python
from dataclasses import dataclass


@dataclass(order=True)
class CaseResult:
    id: int
    ms: int


def test_sort_natural_order():
    rows = [CaseResult(3, 90), CaseResult(1, 120)]
    assert sorted(rows) == [CaseResult(1, 120), CaseResult(3, 90)]
```

## Pitfalls

**Integer overflow in compareTo.** `(a - b)` is unsafe; use `Integer.compare`.

**Broken Comparator contract.** Non-transitive compare crashes TreeSet mysteriously.

**Reverse order confusion.** sorted() returns new list; sort() mutates in place.

**Locale String compare.** CI locale changes name ordering.

**Parallel sort instability.** Equal elements may swap positions.
