# Array vs collection: choosing test data structures

## In short

A Java array has fixed length, homogeneous type, O(1) index access, minimal overhead. A Collection (List/Set/Map) grows dynamically and offers utility APIs (add, remove, stream). Primitives: `int[]` vs boxed `ArrayList<Integer>`. Parametrized test providers often use `Object[][]` or `List<Arguments>`.

## How to answer in an interview

- Fixed small expected-value matrix → array.
- Unknown API item count → ArrayList.
- Unique tag set → HashSet before assert.
- `@ParameterizedTest` with `MethodSource` is list/stream friendly.
- Honestly: `Arrays.asList` fixed-size lists break add in fixture builders.

## How it works

Array:

```
int[] thresholds = { 100, 500, 1000 };
assert thresholds[1] == 500;
```

Collection:

```
List<String> cases = new ArrayList<>();
cases.add("login");
cases.add("checkout");
```

Conversion:

```
String[] arr = cases.toArray(String[]::new);
List<String> again = Arrays.asList(arr);
```

Python comparison:

```python
def test_array_vs_list():
    fixed = (1, 2, 3)
    dynamic = [1, 2, 3]
    dynamic.append(4)
    assert len(dynamic) == 4
    assert len(fixed) == 3
```

## Pitfalls

**Array covariance trap.** `Object[] o = new String[1]; o[0]=1` fails at runtime.

**Arrays.asList mutation.** set works; add throws.

**toArray zero-size.** JVM allocates correct size; do not blindly use `new String[0]`.

**Primitive array print.** Default toString is useless in log asserts.

**List.of immutable.** add in setup fails; copy to mutable list first.
