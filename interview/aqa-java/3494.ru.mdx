# Четыре столпа ООП в Java тестах

## Суть

ООП в Java строится на инкапсуляции, наследовании, полиморфизме и абстракции. В автотестах это проявляется как скрытие деталей page object, базовые классы фикстур, подменяемые интерфейсы клиентов и абстрактные шаги сценария. Четыре принципа помогают держать тестовый код расширяемым без копипасты.

## Что ответить на собесе

- Инкапсуляция: private локаторы, public методы действий (`clickCheckout()`).
- Наследование: `BasePage` с wait helpers, страницы наследуют.
- Полиморфизм: `AuthProvider` mock/real в одном тесте через интерфейс.
- Абстракция: интерфейс `StoragePort`, детали JDBC скрыты.
- Честно: наследование page object глубже двух уровней чаще вредит, лучше composition.

## Как это работает

Инкапсуляция:

```
class WalletPage {
  private By balance = By.id("balance");

  int readBalance(WebDriver driver) {
    return Integer.parseInt(driver.findElement(balance).getText());
  }
}
```

Абстракция + полиморфизм:

```
interface Clock { long now(); }

class SystemClock implements Clock {
  public long now() { return System.currentTimeMillis(); }
}

void assertFresh(long ts, Clock clock) {
  assertTrue(clock.now() - ts < 5000);
}
```

Python mirror:

```python
class WalletPage:
    def __init__(self, driver) -> None:
        self._balance_locator = "balance"

    def read_balance(self, driver) -> int:
        return int(driver.find_element(self._balance_locator).text)
```

## Подводные камни

**God page object.** Нарушение SRP, инкапсуляция превращается в свалку методов.

**Fragile base class.** Изменение `BaseTest` ломает все наследники.

**Полиморфизм через instanceof.** Антипаттерн вместо интерфейса.

**Абстрактный класс vs interface.** Множественное поведение только через interfaces.

**Тестируемость private.** Чрезмерная инкапсуляция без seam для mock.
