# null in HashMap

## In short

`HashMap` allows one null key and any number of null values. The null key goes in a dedicated bucket cell; `get(null)` distinguishes "key exists, value is null" from a missing key only via `containsKey`. In test maps be careful: a null value breaks assert chains without an explicit check.

## How to answer in an interview

- `map.put(null, "orphan")` and `map.put("k", null)` are both valid.
- `get("missing")` and `get("k")` with a null value can both return null; use `containsKey`.
- `Hashtable` and `ConcurrentHashMap` do not accept null.
- In expected JSON I map null to `Optional` or an explicit sentinel in asserts.
- Honestly: a null key in a shared config map in a parallel suite creates an implicit global slot.

## How it works

Null key and value:

```
Map<String, String> tags = new HashMap<>();
tags.put(null, "default-bucket");
tags.put("owner", null);

boolean hasOwner = tags.containsKey("owner");
String owner = tags.get("owner");
```

Python mirror:

```python
tags: dict[str | None, str | None] = {None: "default-bucket", "owner": None}
assert "owner" in tags
assert tags["owner"] is None
```

## Pitfalls

**containsKey vs get.** Without containsKey, missing and null value both look like null from get.

**Null key in computeIfAbsent.** The lambda still runs when the key was absent.

**JSON round-trip.** A null value becomes an empty string on serialization.

**equals on null key.** Only one null slot; a second put overwrites.

**Stream collect.** `toMap` without a merge function fails on duplicate, not on null value.
