# null в HashMap

## Суть

`HashMap` допускает один null-ключ и любое количество null-значений. null-ключ кладётся в отдельную bucket-ячейку, `get(null)` отличает «ключ есть, value null» от отсутствия ключа только через `containsKey`. В тестовых map осторожно: null value ломает цепочки assert без явной проверки.

## Что ответить на собесе

- `map.put(null, "orphan")` и `map.put("k", null)` оба валидны.
- `get("missing")` и `get("k")` при null value оба могут вернуть null, нужен `containsKey`.
- `Hashtable` и `ConcurrentHashMap` null не принимают.
- В expected JSON null превращаю в `Optional` или явный sentinel в assert.
- Честно: null key в shared config map в parallel suite создаёт неявный global slot.

## Как это работает

Ключ и значение null:

```
Map<String, String> tags = new HashMap<>();
tags.put(null, "default-bucket");
tags.put("owner", null);

boolean hasOwner = tags.containsKey("owner");
String owner = tags.get("owner");
```

Python mirror:

```python
tags: dict[str | None, str | None] = {None: "default-bucket", "owner": None}
assert "owner" in tags
assert tags["owner"] is None
```

## Подводные камни

**containsKey vs get.** Оба null на missing и null value без containsKey неразличимы.

**Null key в computeIfAbsent.** Лямбда всё равно вызовется, если ключа не было.

**JSON round-trip.** null value теряется при сериализации в пустую строку.

**equals на null key.** Только один null slot, второй put перезапишет.

**Stream collect.** `toMap` без merge function падает на duplicate, не на null value.
