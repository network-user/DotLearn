# LinkedList in Java: when it fits

## In short

LinkedList is doubly-linked nodes: O(1) add/remove at head/tail with an iterator, O(n) for get(i). It implements List and Deque. It suits queues/stacks with frequent edge inserts and no random access. In typical AQA code ArrayDeque wins on cache; LinkedList is rare but useful for LRU-like test cache prototypes.

## How to answer in an interview

- Deque ops `addFirst`, `pollLast` are O(1).
- get(5000) walks nodes; avoid it inside assertion loops.
- ListIterator remove during iteration is supported.
- Memory overhead per node vs an ArrayList slot.
- Honestly: choosing "linked" without a profile is often wrong; ArrayList is the default.

## How it works

```
Deque<String> queue = new LinkedList<>();
queue.addLast("case-a");
queue.addLast("case-b");
String next = queue.pollFirst();
```

Iterator remove:

```
ListIterator<String> it = list.listIterator();
while (it.hasNext()) {
  if (it.next().startsWith("skip")) it.remove();
}
```

Python deque:

```python
from collections import deque


def test_linked_like_deque():
    q: deque[str] = deque()
    q.append("case-a")
    q.append("case-b")
    assert q.popleft() == "case-a"
```

## Pitfalls

**Index loop with get(i).** O(n^2) on sort-like passes.

**LinkedList as Queue.** Legacy choice; ArrayDeque is faster.

**Node deep copy.** Clone fixtures get heavy.

**Null elements allowed.** Multiple nulls vs Set semantics.

**Fail-fast iterator.** Concurrent modify without iterator throws CME.
