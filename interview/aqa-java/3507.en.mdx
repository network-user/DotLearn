# Functional interfaces in test code

## In short

A functional interface has exactly one abstract method (SAM) and may include `default`/`static` methods. It accepts lambdas and method references. In tests that is `Predicate` for filtering responses, `Supplier` for lazy fixtures, and `Consumer` for logging hooks.

## How to answer in an interview

- `@FunctionalInterface` on a custom `RetryPolicy` for a compile-time guard.
- `Predicate<String>` in an assert helper: `rows.stream().filter(predicate)`.
- Method reference `String::isBlank` instead of `s -> s.isBlank()`.
- Do not add a second abstract method without knowing it breaks lambdas.
- Honestly: for one or two lines a lambda is fine; for complex logic a named class reads better.

## How it works

```
@FunctionalInterface
interface WaitCondition {
  boolean ready();
}

void waitUntil(WaitCondition condition) {
  while (!condition.ready()) { /* poll */ }
}

waitUntil(() -> driver.getTitle().contains("Dashboard"));
```

Built-in:

```
Predicate<Integer> positive = value -> value > 0;
List<Integer> filtered = ids.stream().filter(positive).toList();
```

Python mirror:

```python
from typing import Callable


def wait_until(condition: Callable[[], bool]) -> None:
    while not condition():
        pass


def test_functional_condition():
    flag = {"ready": False}
    flag["ready"] = True
    wait_until(lambda: flag["ready"])
```

## Pitfalls

**Two abstract methods.** The interface stops being functional; lambdas will not compile.

**Object methods.** equals/hashCode do not count as abstract SAM.

**Serializable lambda.** Serialization pain in distributed test runners.

**this capture.** A lambda holds the outer test instance; GC leak in long suites.

**Exception in SAM.** Checked exceptions in a lambda need a wrapper.
