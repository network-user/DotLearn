# Лямбды и method reference в тестах

## Суть

Лямбда `(args) -> body` реализует SAM функционального интерфейса без boilerplate анонимного класса. Method reference (`Type::method`) сокращает лямбду, когда тело только делегирует существующему методу. В Stream/assert helpers это убирает шум и делает intent явным.

## Что ответить на собесе

- `names.forEach(System.out::println)` вместо `n -> System.out.println(n)`.
- `map(String::toLowerCase)` для нормализации expected.
- Constructor ref `ArrayList::new` в `collect(Collectors.toCollection(ArrayList::new))`.
- Лямбда с несколькими строками выношу в private method, не раздуваю test body.
- Честно: method reference не читабелен, если нужны дополнительные аргументы, оставляю лямбду.

## Как это работает

Лямбда:

```
cases.forEach(item -> log.info("run {}", item));
```

Method references:

```
cases.stream()
  .map(String::trim)
  .filter(Predicate.not(String::isEmpty))
  .forEach(log::info);
```

Статический метод:

```
Function<String, Integer> parser = Integer::parseInt;
```

Python mirror:

```python
def for_each(items: list[str], action) -> None:
    for item in items:
        action(item)


def test_lambda_style():
    seen: list[str] = []
    for_each(["a"], seen.append)
    assert seen == ["a"]
```

## Подводные камни

**Неверный arity.** `Integer::sum` не подходит как `BiFunction` без adapt.

**instance method ref.** `obj::method` привязывает receiver, осторожно при parallel.

**this в лямбде.** Захват mutable test field → race в parallel stream.

**forEach order.** parallelStream + forEach: порядок не для assert.

**Debugging.** Стек лямбды менее очевиден, чем именованный метод.
