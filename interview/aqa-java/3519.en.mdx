# LinkedList insertion complexity

## In short

Inserting into a doubly linked list costs O(1) when you already hold a reference to the node after which you insert. Finding a position by index or value requires traversal and is O(n). Test utilities rarely pick a list for random access, but the trade-off matters when storing a step queue.

## How to answer in an interview

- `add(index, e)` first finds the node: O(n), then relinking: O(1).
- `addFirst` / `addLast` with known head/tail: O(1).
- `ArrayList` is faster for index reads; `LinkedList` for frequent middle inserts with an iterator.
- In a parallel suite I do not share a mutable `LinkedList` without sync.
- Honestly: for 10 elements the difference is invisible; choose by API, not paper Big-O.

## How it works

Insert after a known node:

```
class Node {
  int value;
  Node next;
  Node prev;
}

void insertAfter(Node anchor, int data) {
  Node fresh = new Node();
  fresh.value = data;
  fresh.next = anchor.next;
  fresh.prev = anchor;
  if (anchor.next != null) anchor.next.prev = fresh;
  anchor.next = fresh;
}
```

Index search before insert:

```
void insertAt(LinkedList list, int index, int data) {
  Node cursor = list.head;
  for (int i = 0; i < index; i++) cursor = cursor.next;
  insertAfter(cursor.prev == null ? sentinel : cursor, data);
}
```

Python mirror:

```python
from collections import deque


def append_step(steps: deque[str], step: str) -> None:
    steps.append(step)


def insert_after_known(steps: list[str], pos: int, step: str) -> None:
    steps.insert(pos, step)
```

## Pitfalls

**Iterator + structural modify.** `ConcurrentModificationException` when inserting during foreach.

**list vs deque confusion.** In Python `list.insert(0)` is O(n); do not confuse with Java `LinkedList`.

**get(i) in a loop.** O(n²) when building an expected sequence.

**Null head.** Inserting "after null" without a sentinel breaks the chain.

**Boxed Integer in a node.** Extra allocations on a micro-benchmark hot path, not in a UI test.
