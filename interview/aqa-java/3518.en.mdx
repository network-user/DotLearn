# Why Java interfaces exist

## In short

An interface defines a method contract without requiring implementation in the type itself. A class can implement several interfaces and be used polymorphically where a shared type is expected. In AQA frameworks this is the basis for swappable drivers, reporters, and listener hooks.

## How to answer in an interview

- An interface describes what something can do, not how it is built inside.
- One class: `implements WebDriver, TakesScreenshot` instead of multiple class inheritance.
- In tests I accept `ReportSink`, not a concrete Allure adapter.
- Default/static methods on an interface provide shared helpers without a base class.
- Honestly: an interface does not replace composition when the contract grows into a god interface.

## How it works

Action contract:

```
interface Notifier {
  void send(String message);
  default void ping() { send("ping"); }
}

class SlackNotifier implements Notifier {
  public void send(String message) { /* ... */ }
}

class EmailNotifier implements Notifier {
  public void send(String message) { /* ... */ }
}

void runSuite(Notifier sink) {
  sink.send("suite started");
}
```

One runner invokes any implementation through the shared type.

Python mirror:

```python
from typing import Protocol


class Notifier(Protocol):
    def send(self, message: str) -> None: ...


class SlackNotifier:
    def send(self, message: str) -> None:
        pass


def run_suite(sink: Notifier) -> None:
    sink.send("suite started")
```

## Pitfalls

**Contract without semantics.** A `close()` method on two interfaces with different meaning confuses implementations.

**Fat interface.** Ten optional methods break mocks and stub classes.

**equals via interface.** Comparing implementations without an explicit contract surprises in collections.

**Default method conflict.** Two default versions with the same signature require an override in the class.

**Test double without behavior.** An empty stub `implements` compiles but fails at runtime with NPE.
