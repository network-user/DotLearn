# try-catch-finally edge cases

## In short

`try` runs risky code, `catch` handles specific exceptions, and `finally` almost always runs for cleanup (closing a driver, file, or socket). The `finally` block is skipped on `System.exit`, a fatal JVM error, or process kill. In tests I prefer try-with-resources over manual finally where possible.

## How to answer in an interview

- `finally` runs after try/catch even when try returns.
- `System.exit` in catch stops the JVM before finally.
- try-with-resources compiles to try/finally with autoclose.
- In `@AfterEach` I duplicate cleanup if the test threw before finally.
- Honestly: return in finally masks an exception from try; that is an anti-pattern.

## How it works

Classic chain:

```
WebDriver driver = null;
try {
  driver = new ChromeDriver();
  driver.get("/");
} catch (WebDriverException ex) {
  log.warn("navigation failed", ex);
} finally {
  if (driver != null) driver.quit();
}
```

When finally does not run:

```
try {
  System.exit(1);
} finally {
  System.out.println("never printed");
}
```

Python mirror:

```python
def run_ui() -> None:
    driver = None
    try:
        driver = create_driver()
        driver.get("/")
    except RuntimeError as exc:
        log_warning(exc)
    finally:
        if driver is not None:
            driver.quit()
```

## Pitfalls

**Exception in finally.** Masks the original from try unless addSuppressed is used.

**Double close.** finally plus @AfterEach without a guard causes double quit.

**Swallow in catch.** An empty catch makes the test "green" on a real failure.

**Error vs Exception.** OutOfMemory is not caught by a plain catch Exception.

**Thread death.** `Thread.stop` is obsolete, but kill -9 bypasses finally at the OS level.
