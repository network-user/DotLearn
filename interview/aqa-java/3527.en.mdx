# JVM heap memory

## In short

The heap stores all Java objects created via `new` and arrays. References sit in a stack frame or in fields of other objects; the data lives in the heap. GC frees unreachable objects; heap size is tuned with `-Xmx`. For a Selenium suite it matters not to keep extra objects in static fields between tests.

## How to answer in an interview

- `new ChromeDriver()` allocates on the heap; the driver variable on the stack holds a reference.
- Young/Old generation: short-lived page objects die in Minor GC.
- OutOfMemoryError when the heap is exhausted; do not confuse with stack overflow.
- `-Xmx` caps the process heap upper bound.
- Honestly: a leak through a static `List<WebElement>` survives GC while the reference lives.

## How it works

Object creation:

```
class Session {
  String token;
}

void open() {
  Session session = new Session();
  session.token = "abc";
}
```

## Pitfalls

**Static cache.** The heap will not clear while the classloader lives.

**Huge screenshot byte[].** One frame can consume a lot of Old Gen.

**Premature promotion.** A long `@BeforeAll` keeps objects in Old; expensive Full GC.

**Metaspace vs Heap.** Class metadata is not in the heap; OOM happens separately.

**False leak in profiler.** A live driver session intentionally keeps DOM snapshots.
