# Access modifiers and package-private

## In short

Java visibility uses `public`, `protected`, `private`, and package-private (no modifier). Package-private limits access to classes in the same package. In test code, modifiers separate the public framework API from internal helpers and protect page object encapsulation.

## How to answer in an interview

- `private` for locators and wait utils inside a page class.
- `public` only for stable scenario step APIs.
- `protected` for hooks in `BaseTest` for subclasses in the same package/hierarchy.
- Package-private utils in `com.app.tests.internal` stay internal.
- Honestly: tests in another package cannot see package-private; you need a public API or the same package.

## How it works

```
package com.acme.tests.ui;

class LocatorKit {
  static By dataTest(String value) {
    return By.cssSelector("[data-test='" + value + "']");
  }
}

public class LoginPage {
  private final By submit = LocatorKit.dataTest("login-submit");

  public void submit(WebDriver driver) {
    driver.findElement(submit).click();
  }
}
```

`LocatorKit` is package-private: visible to `LoginPage`, not from `com.acme.tests.api`.

Python mirror (module convention):

```python
def _locator(value: str) -> str:
    return f"[data-test='{value}']"


class LoginPage:
    def __init__(self) -> None:
        self._submit = _locator("login-submit")
```

## Pitfalls

**Test in another package.** Will not compile against a package-private helper.

**protected + different package.** Access only via subclass, not composition.

**public fields.** Break encapsulation; trap for direct state asserts.

**JPMS modules.** exports/opens in module-info change visibility vs classic classpath.

**Reflection bypass.** Possible but breaks the contract and hides a design smell.
