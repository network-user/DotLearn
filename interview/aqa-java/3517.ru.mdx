# Поля класса и локальные переменные

## Суть

Поля (instance/static) объявлены на уровне класса и хранят состояние объекта или shared class state. Локальные переменные и параметры методов живут в stack frame и исчезают после выхода из метода. В page object поля держат driver/locators, локальные используют для временных expected в assert.

## Что ответить на собесе

- Instance field: `WebDriver driver` в base page, не static.
- Static field для config ok, но mutable static в parallel suite опасен.
- Локальная `String expectedTitle` в методе теста, не поле класса.
- Параметры метода как explicit inputs шага сценария.
- Честно: `@TestInstance(PER_CLASS)` делает не-static поля shared между тестами, нужна изоляция.

## Как это работает

```
class CheckoutPage {
  private final By payButton = By.id("pay");

  void pay(WebDriver driver) {
    WebElement button = driver.findElement(payButton);
    button.click();
  }
}

void assertCheckout(WebDriver driver) {
  String expected = "Paid";
  CheckoutPage page = new CheckoutPage();
  page.pay(driver);
  assertEquals(expected, driver.getTitle());
}
```

Python mirror:

```python
class CheckoutPage:
    def __init__(self) -> None:
        self._pay_button = "pay"

    def pay(self, driver) -> None:
        driver.find_element(self._pay_button).click()
```

## Подводные камни

**Static mutable counter.** Flaky в parallel JUnit.

**Поле без final.** Переприсвоение driver mid-test ломает waits.

**Shadowing.** Локальная переменная с именем поля скрывает field, баг в assert.

**Неинициализированное поле.** Default null для reference, NPE в шаге.

**Сериализация полей.** Лишнее state в DTO фикстурах усложняет equals.
