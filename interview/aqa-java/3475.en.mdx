# Terminal operations in the Stream API

## In short

A terminal operation closes the pipeline and materializes a result: a collection (`collect`), an aggregate (`reduce`, `count`, `min`, `max`), or a void side effect (`forEach`). Without a terminal, intermediate steps never run. In tests, the terminal turns a service response stream into an assert-friendly structure.

## How to answer in an interview

- `collect(toList())` for snapshot comparison of JSON lists.
- `reduce` sums latency metrics across cases.
- `count` beats materializing the whole list when you only need size.
- I assert `Optional` from `min/max` via `orElseThrow` in negative tests.
- Honestly: `forEach` without asserts inside is debug logging only, not verification.

## How it works

collect:

```
List<String> names = Stream.of("api", "ui", "db")
    .map(String::toUpperCase)
    .collect(Collectors.toList());
```

reduce and count:

```
int total = Stream.of(120, 80, 200).reduce(0, Integer::sum);
long qty = Stream.of("a", "b", "c").count();
```

min:

```
OptionalInt fastest = IntStream.of(900, 450, 1200).min();
```

pytest mirror:

```python
def terminal_collect(items: list[str]) -> list[str]:
    return [s.upper() for s in items]


def test_terminal_ops():
    assert terminal_collect(["api", "ui"]) == ["API", "UI"]
    assert sum([120, 80, 200]) == 400
    assert min([900, 450, 1200]) == 450
```

## Pitfalls

**Mutating a collected list.** Prefer unmodifiable `toList()` from Java 16+.

**reduce without identity.** Empty stream yields empty `Optional`.

**forEach order in parallel.** Non-deterministic log order.

**count after limit.** Still traverses up to the limit.

**joining without filtering empties.** Extra commas break CSV asserts.
