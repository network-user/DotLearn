# How ArrayList works internally

## In short

`ArrayList` stores elements in a dynamic `Object[]` backing array, gives O(1) indexed access and amortized O(1) append. When capacity is exceeded the array is copied and grown (typically ×1.5). It is not thread-safe without external synchronization.

## How to answer in an interview

- I pick ArrayList for frequent random access and rare middle inserts.
- `ensureCapacity` or a sized constructor reduce reallocations in data-driven tests.
- `remove` and `add(index)` shift the tail, so they are O(n).
- For shared mutable state in a parallel suite I use `CopyOnWriteArrayList` or synchronization.
- Honestly: LinkedList wins only when you often insert at the head and rarely call get.

## How it works

Backing array and size inside:

```
ArrayList<String> tags = new ArrayList<>(4);
tags.add("smoke");
tags.add("regression");
tags.add("api");

String second = tags.get(1);
tags.set(1, "e2e");
tags.remove(0);
```

On growth:

```
// conceptual: capacity 4 -> add 5th -> new array length 6, copy elements
```

Python mirror:

```python
def grow_backing(store: list[str], item: str) -> list[str]:
    store.append(item)
    return store


def test_arraylist_like_access():
    data = grow_backing([], "smoke")
    data = grow_backing(data, "api")
    assert data[0] == "smoke"
    data[1] = "e2e"
    data.pop(0)
    assert data == ["e2e"]
```

## Pitfalls

**Concurrent modification.** Iterator plus structural change without iterator.remove leads to `ConcurrentModificationException`.

**SubList view.** Changes to subList affect the parent; easy to break expected values in asserts.

**Null elements.** Allowed but complicate contains and grouping in tests.

**trimToSize.** Rarely needed; after bulk remove capacity may stay inflated.

**equals on lists.** Order matters; for set-like comparison use another collection type.
