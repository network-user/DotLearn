# Полиморфизм в Java и тестовый дизайн

## Суть

Полиморфизм позволяет вызывать один интерфейсный метод на объектах разных реализаций: runtime выбирает версию по фактическому типу (динамическая диспетчеризация). В Java это override в иерархии классов и реализации интерфейсов. В автотестах это основа подменяемых клиентов, стабов и параметризованных сценариев.

## Что ответить на собесе

- Интерфейс `Notifier`, реализации `EmailNotifier` и `SlackNotifier` в DI контейнере теста.
- Ссылка типа `PaymentGateway` на mock или real impl без смены тестового кода.
- `@Override` + `@Test` на каждой реализации общего контрактного теста.
- Перегрузка (overload) это compile-time, не путать с полиморфизмом override.
- Честно: без явного интерфейса полиморфизм хрупок при package-private override.

## Как это работает

Override в иерархии:

```
interface Reporter {
  String report();
}

class JsonReporter implements Reporter {
  public String report() { return "{\"ok\":true}"; }
}

class PlainReporter implements Reporter {
  public String report() { return "ok"; }
}

void assertAnyReporter(Reporter r) {
  assertTrue(r.report().contains("ok"));
}
```

Вызов `r.report()` резолвится по runtime типу объекта.

Python mirror:

```python
from typing import Protocol


class Reporter(Protocol):
    def report(self) -> str: ...


class JsonReporter:
    def report(self) -> str:
        return '{"ok": true}'


def assert_any_reporter(reporter: Reporter) -> None:
    assert "ok" in reporter.report()


def test_polymorphism():
    assert_any_reporter(JsonReporter())
```

## Подводные камни

**static методы.** Не полиморфны, hide а не override.

**private методы.** Java 9+ private override в subclass не полиморфен для внешнего вызова.

**field hiding.** Поля не виртуальны, shadowing ломает ожидания в assert.

**instanceof цепочки.** Замена полиморфизма if-else антипаттерн в page objects.

**Контрактные тесты.** Одна реализация может пройти mock, но сломать prod adapter.
