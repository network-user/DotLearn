# Polymorphism in Java and test design

## In short

Polymorphism lets you call one interface method on objects of different implementations: the runtime picks the version by actual type (dynamic dispatch). In Java this is override in class hierarchies and interface implementations. In automation it underpins swappable clients, stubs, and parameterized scenarios.

## How to answer in an interview

- Interface `Notifier`; implementations `EmailNotifier` and `SlackNotifier` in the test DI container.
- A `PaymentGateway` reference to a mock or real impl without changing test code.
- `@Override` plus `@Test` on each implementation in a shared contract test.
- Overload is compile-time; do not confuse it with override polymorphism.
- Honestly: without an explicit interface, polymorphism is fragile with package-private overrides.

## How it works

Override in a hierarchy:

```
interface Reporter {
  String report();
}

class JsonReporter implements Reporter {
  public String report() { return "{\"ok\":true}"; }
}

class PlainReporter implements Reporter {
  public String report() { return "ok"; }
}

void assertAnyReporter(Reporter r) {
  assertTrue(r.report().contains("ok"));
}
```

Calling `r.report()` resolves by the object's runtime type.

Python mirror:

```python
from typing import Protocol


class Reporter(Protocol):
    def report(self) -> str: ...


class JsonReporter:
    def report(self) -> str:
        return '{"ok": true}'


def assert_any_reporter(reporter: Reporter) -> None:
    assert "ok" in reporter.report()


def test_polymorphism():
    assert_any_reporter(JsonReporter())
```

## Pitfalls

**static methods.** Not polymorphic; they hide, not override.

**private methods.** Java 9+ private override in a subclass is not polymorphic for external calls.

**field hiding.** Fields are not virtual; shadowing breaks assert expectations.

**instanceof chains.** Replacing polymorphism with if-else is an anti-pattern in page objects.

**Contract tests.** One implementation may pass a mock but break a prod adapter.
