# throw vs throws in test methods

## In short

`throw` creates and raises an exception at a point in execution. `throws` on a method signature declares checked exceptions the caller must handle or propagate. In tests, `throw` validates negative branches; `throws` on `@Test` is acceptable for IO setup.

## How to answer in an interview

- Guard: `if (token == null) throw new IllegalStateException("no token")`.
- `throws IOException` on a helper that reads a disk fixture.
- `assertThrows` replaces manual throw/catch for expected failure.
- Re-throw: `catch (SQLException e) { throw new RuntimeException(e); }` only deliberately.
- Honestly: throws on every test bloats signatures; try-with-resources in `@BeforeEach` is cleaner.

## How it works

throw:

```
void requireAdult(int age) {
  if (age < 18) {
    throw new IllegalArgumentException("too young");
  }
}
```

throws:

```
void readConfig(Path path) throws IOException {
  Files.readString(path);
}
```

JUnit:

```
assertThrows(IllegalArgumentException.class, () -> requireAdult(15));
```

Python mirror:

```python
def require_adult(age: int) -> None:
    if age < 18:
        raise ValueError("too young")


def test_throw_contract():
    import pytest
    with pytest.raises(ValueError):
        require_adult(15)
```

## Pitfalls

**throws without throw.** Empty declaration misleads readers.

**throw checked without throws.** Compile error; wrap or declare throws.

**throws Exception.** Too broad; hides real bugs.

**finally throw.** Masks assertion failure.

**throw Error.** Not for business validation in a test helper.
