# Устройство HashMap и коллизии

## Суть

`HashMap` хранит пары ключ-значение в массиве бакетов; индекс бакета вычисляется из `hashCode` ключа. При коллизии цепочка в бакете (с Java 8 при длине ≥8 превращается в красно-черное дерево). Средняя сложность get/put O(1), порядок итерации не гарантирован.

## Что ответить на собесе

- Ключи с стабильным `equals`/`hashCode`, для DTO в тестах Lombok `@EqualsAndHashCode` или явная реализация.
- `initialCapacity` и `loadFactor` при больших фикстурах снижают resize.
- `ConcurrentHashMap` вместо `Collections.synchronizedMap` для parallel runners.
- Не мутирую ключ после put, иначе entry «теряется».
- Честно: worst case O(n) при плохой hash функции или злонамеренных ключах.

## Как это работает

```
Map<String, Integer> inventory = new HashMap<>();
inventory.put("widget", 10);
inventory.put("gadget", 3);

int qty = inventory.get("widget");
inventory.remove("gadget");
boolean exists = inventory.containsKey("widget");
```

Python mirror:

```python
def bucket_index(key: str, size: int) -> int:
    return hash(key) % size


def test_hash_bucket_placement():
    keys = ["alpha", "beta", "alpha"]
    assert bucket_index(keys[0], 16) == bucket_index(keys[2], 16)
```

## Подводные камни

**Mutable key.** Изменение поля ключа после insert ломает lookup.

**null key.** Один null ключ допустим, в тестовых map часто NPE-ловушка.

**hashCode только по id.** Равные объекты с разным hash дают дубликаты логически, но map их не схлопнет.

**iteration order.** Нельзя assert порядок ключей без `LinkedHashMap`.

**resize rehash.** Массовый put триггерит rebuild, в latency-sensitive load test учитываю.
