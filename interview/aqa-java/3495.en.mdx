# When to pick LinkedList over ArrayList

## In short

`LinkedList` is a doubly linked node list: insert/remove at a known iterator position is O(1), but indexed access is O(n). It fits queues, stacks, and workloads with frequent head/middle edits and rare random access. In typical test collections `ArrayList` wins more often.

## How to answer in an interview

- `Deque` workloads: `addFirst`/`pollLast` for an event buffer in an integration test.
- Do not use it for parameterized data with frequent `get(i)`.
- Implements `Queue` and `Deque`; handy as a work-stealing buffer in a custom runner.
- Memory: each element pays two pointer overheads vs a compact array.
- Honestly: in 95% of test code ArrayList is simpler and faster.

## How it works

```
Deque<String> queue = new LinkedList<>();
queue.addLast("case-a");
queue.addLast("case-b");
String next = queue.pollFirst();

queue.addFirst("priority");
queue.removeLast();
```

Iterator insert without array shifting:

```
ListIterator<String> it = queue.listIterator();
while (it.hasNext()) {
  if ("skip".equals(it.next())) it.remove();
}
```

Python deque mirror:

```python
from collections import deque


def test_linked_queue_behavior():
    q: deque[str] = deque()
    q.append("case-a")
    q.appendleft("priority")
    assert q.popleft() == "priority"
```

## Pitfalls

**get(index).** Linear walk on LinkedList; hidden slowdown in assert loops.

**Not thread-safe.** Parallel poll/add without synchronization races.

**Memory.** Large fixture lists inflate heap vs ArrayList.

**subList.** Costlier and less useful than array-backed lists.

**Name-driven choice.** «Need fast insert» without profiling often wrongly picks LinkedList.
