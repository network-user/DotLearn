# Lambdas and method references in tests

## In short

A lambda `(args) -> body` implements a functional interface SAM without anonymous class boilerplate. A method reference (`Type::method`) shortens a lambda when the body only delegates to an existing method. In Stream/assert helpers this removes noise and makes intent explicit.

## How to answer in an interview

- `names.forEach(System.out::println)` instead of `n -> System.out.println(n)`.
- `map(String::toLowerCase)` to normalize expected values.
- Constructor ref `ArrayList::new` in `collect(Collectors.toCollection(ArrayList::new))`.
- Multi-line lambdas move to a private method; I do not bloat the test body.
- Honestly: a method reference is unreadable when extra arguments are needed; I keep the lambda.

## How it works

Lambda:

```
cases.forEach(item -> log.info("run {}", item));
```

Method references:

```
cases.stream()
  .map(String::trim)
  .filter(Predicate.not(String::isEmpty))
  .forEach(log::info);
```

Static method:

```
Function<String, Integer> parser = Integer::parseInt;
```

Python mirror:

```python
def for_each(items: list[str], action) -> None:
    for item in items:
        action(item)


def test_lambda_style():
    seen: list[str] = []
    for_each(["a"], seen.append)
    assert seen == ["a"]
```

## Pitfalls

**Wrong arity.** `Integer::sum` does not fit as `BiFunction` without adaptation.

**Instance method ref.** `obj::method` binds the receiver; be careful with parallel.

**this in lambda.** Capturing a mutable test field races in a parallel stream.

**forEach order.** parallelStream + forEach: order is not for asserts.

**Debugging.** Lambda stack traces are less obvious than a named method.
