# Фазы жизненного цикла Maven в CI

## Суть

Maven управляет сборкой через жизненные циклы: `clean`, `default` (build) и `site`. Фазы внутри цикла идут строго по порядку: вызов `mvn test` автоматически выполняет validate и compile. В CI pipeline фазы мапятся на compile, unit/integration test, package и deploy артефактов.

## Что ответить на собесе

- `mvn -q test` на каждый PR, `verify` если подключены failsafe и quality gates.
- `clean` перед release build, не обязателен на каждом commit.
- Goals плагинов привязываются к фазам в `pom.xml`, не путаю goal и phase.
- `install` кладёт jar в `~/.m2`, нужен для multi-module зависимостей локально.
- Честно: `deploy` в CI только с credentials и отдельным staging repo, не в dev job.

## Как это работает

Default lifecycle (сокращённо):

```
validate → compile → test → package → verify → install → deploy
```

Clean:

```
pre-clean → clean → post-clean
```

Пример команды в pipeline:

```
mvn -B -DskipTests=false clean verify
```

Python mirror оркестрации стадий:

```python
DEFAULT_PHASES = ["validate", "compile", "test", "package", "verify"]


def run_through(phase: str) -> list[str]:
    stop = DEFAULT_PHASES.index(phase)
    return DEFAULT_PHASES[: stop + 1]


def test_maven_phase_order():
    assert run_through("test") == ["validate", "compile", "test"]
```

## Подводные камни

**Phase vs goal.** `mvn compile` это фаза, `compiler:compile` goal; порядок разный при прямом вызове goal.

**skipTests.** Пропускает surefire, но не компиляцию test sources.

**Reactor order.** Multi-module: upstream module должен `install` до downstream test.

**Кэш CI.** Без кэша `.m2` каждая фаза тянет зависимости заново.

**site lifecycle.** Документация отдельно, не смешиваю с test gate.
