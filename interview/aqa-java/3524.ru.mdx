# try-catch-finally и граничные случаи

## Суть

`try` выполняет рискованный код, `catch` перехватывает конкретные исключения, `finally` почти всегда срабатывает для cleanup (закрытие driver, файла, socket). Блок `finally` пропускается при `System.exit`, фатальной ошибке JVM или kill процесса. В тестах prefer try-with-resources вместо ручного finally где возможно.

## Что ответить на собесе

- `finally` идёт после try/catch даже при return в try.
- `System.exit` в catch обрывает JVM до finally.
- try-with-resources компилируется в try/finally с autoclose.
- В `@AfterEach` дублирую cleanup, если тест бросил до finally.
- Честно: return в finally маскирует исключение из try, антипаттерн.

## Как это работает

Классическая цепочка:

```
WebDriver driver = null;
try {
  driver = new ChromeDriver();
  driver.get("/");
} catch (WebDriverException ex) {
  log.warn("navigation failed", ex);
} finally {
  if (driver != null) driver.quit();
}
```

Когда finally не выполнится:

```
try {
  System.exit(1);
} finally {
  System.out.println("never printed");
}
```

Python mirror:

```python
def run_ui() -> None:
    driver = None
    try:
        driver = create_driver()
        driver.get("/")
    except RuntimeError as exc:
        log_warning(exc)
    finally:
        if driver is not None:
            driver.quit()
```

## Подводные камни

**Exception в finally.** Маскирует исходное из try, если не addSuppressed.

**Double close.** finally + @AfterEach без guard даёт double quit.

**Swallow в catch.** Пустой catch делает тест «зелёным» при реальной поломке.

**Error vs Exception.** OutOfMemory не ловится обычным catch Exception.

**Thread death.** `Thread.stop` устарел, но kill -9 обходит finally на уровне ОС.
