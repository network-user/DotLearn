# Внутреннее устройство ArrayList

## Суть

`ArrayList` хранит элементы в динамическом массиве `Object[]`, даёт O(1) доступ по индексу и амортизированное O(1) добавление в конец. При переполнении capacity массив копируется с ростом (обычно ×1.5). Не thread-safe без внешней синхронизации.

## Что ответить на собесе

- Выбираю ArrayList для частого random access и редких вставок в середину.
- `ensureCapacity` или конструктор с размером снижают realloc в data-driven тестах.
- `remove` и `add(index)` сдвигают хвост, это O(n).
- Для shared mutable state в parallel suite беру `CopyOnWriteArrayList` или синхронизацию.
- Честно: LinkedList выигрывает только при частых вставках в начало при редком get.

## Как это работает

Внутри backing array и size:

```
ArrayList<String> tags = new ArrayList<>(4);
tags.add("smoke");
tags.add("regression");
tags.add("api");

String second = tags.get(1);
tags.set(1, "e2e");
tags.remove(0);
```

При росте:

```
// conceptual: capacity 4 -> add 5th -> new array length 6, copy elements
```

Python mirror:

```python
def grow_backing(store: list[str], item: str) -> list[str]:
    store.append(item)
    return store


def test_arraylist_like_access():
    data = grow_backing([], "smoke")
    data = grow_backing(data, "api")
    assert data[0] == "smoke"
    data[1] = "e2e"
    data.pop(0)
    assert data == ["e2e"]
```

## Подводные камни

**Concurrent modification.** Iterator + structural change без iterator.remove → `ConcurrentModificationException`.

**SubList view.** Изменения subList отражаются в parent, легко сломать expected в assert.

**Null элементы.** Разрешены, но усложняют contains и grouping в тестах.

**trimToSize.** Редко нужен; после массового remove capacity может остаться раздутым.

**equals на списках.** Порядок важен; для set-like сравнения нужен другой тип коллекции.
