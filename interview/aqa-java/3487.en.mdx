# The throw keyword in Java tests

## In short

`throw` immediately creates a Throwable and unwinds the stack to a matching catch or propagates further. Checked exceptions require throws or try/catch on the caller signature. In JUnit 5 `assertThrows` treats a throw as the success path. throw separates error flow from return values.

## How to answer in an interview

- Guard clause: `if (id <= 0) throw new IllegalArgumentException(...)`.
- Custom domain exceptions carry context for assert messages.
- Re-throw with `catch (E e) { throw e; }` or wrap `new ServiceException("ctx", e)`.
- assertThrows verifies type and message.
- Honestly: throw for control flow in happy-path test helpers is worse than Optional.

## How it works

```
void requireNonBlank(String value) {
  if (value == null || value.isBlank()) {
    throw new IllegalArgumentException("blank input");
  }
}
```

JUnit:

```
assertThrows(IllegalArgumentException.class, () -> requireNonBlank(" "));
```

Python raise mirror:

```python
def require_non_blank(value: str) -> None:
    if not value or not value.strip():
        raise ValueError("blank input")


def test_throw_guard():
    import pytest
    with pytest.raises(ValueError, match="blank"):
        require_non_blank(" ")
```

## Pitfalls

**throw null.** Allowed but NPE in handlers.

**Error vs Exception.** catch Exception misses OutOfMemory.

**finally return.** Suppresses throw silently.

**assertThrows lazy lambda.** The method ref must run inside the lambda.

**Checked in lambda.** Unchecked wrap or throws on the test method.
