# Class fields vs local variables

## In short

Fields (instance/static) are declared at class level and hold object or shared class state. Local variables and method parameters live in a stack frame and disappear when the method returns. In page objects, fields hold driver/locators; locals hold temporary expected values in asserts.

## How to answer in an interview

- Instance field: `WebDriver driver` in a base page, not static.
- Static field for config is ok; mutable static in a parallel suite is risky.
- Local `String expectedTitle` inside the test method, not a class field.
- Method parameters as explicit scenario step inputs.
- Honestly: `@TestInstance(PER_CLASS)` shares non-static fields across tests; isolation is required.

## How it works

```
class CheckoutPage {
  private final By payButton = By.id("pay");

  void pay(WebDriver driver) {
    WebElement button = driver.findElement(payButton);
    button.click();
  }
}

void assertCheckout(WebDriver driver) {
  String expected = "Paid";
  CheckoutPage page = new CheckoutPage();
  page.pay(driver);
  assertEquals(expected, driver.getTitle());
}
```

Python mirror:

```python
class CheckoutPage:
    def __init__(self) -> None:
        self._pay_button = "pay"

    def pay(self, driver) -> None:
        driver.find_element(self._pay_button).click()
```

## Pitfalls

**Static mutable counter.** Flaky in parallel JUnit.

**Non-final field.** Reassigning driver mid-test breaks waits.

**Shadowing.** A local with the same name hides the field; assert bugs follow.

**Uninitialized field.** Default null for references leads to NPE in a step.

**Serializing fields.** Extra state in DTO fixtures complicates equals.
