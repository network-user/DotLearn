# Method overloading

## In short

Overloading is several methods with one name and different signatures (count, type, or order of parameters) in one class. Return type alone does not create an overload. The compiler picks the version at compile time from call arguments. In test helper classes overloading simplifies the API: one `click` for By, String, or WebElement.

## How to answer in an interview

- `void type(CharSequence)` and `void type(String)` are different overloads.
- `type(int)` and `type(long)` with `type(5)` picks int (widening comes later).
- Varargs conflicts with an array overload; be explicit.
- Overloading is not runtime polymorphism (that is override).
- Honestly: ten overloads instead of a builder hurts discoverability in the IDE.

## How it works

Retry helper:

```
class Retry {
  int delayMs(int base) { return base; }
  int delayMs(int base, int factor) { return base * factor; }
  double delayMs(double seconds) { return (int)(seconds * 1000); }
}
```

Calls:

```
Retry r = new Retry();
r.delayMs(200);
r.delayMs(200, 3);
r.delayMs(0.5);
```

Python mirror:

```python
def delay_ms(base: int, factor: int = 1) -> int:
    return base * factor
```

## Pitfalls

**Ambiguous call.** `log(null)` between `log(String)` and `log(Throwable)` does not compile.

**Autoboxing overload.** `print(int)` vs `print(Integer)` with null.

**Return type only.** `int parse()` and `String parse()` in one class is an error.

**Bridge methods + overload.** Generics create synthetic methods; rare surprises.

**Override vs overload.** Overload in a child does not override the parent; it hides the name.
