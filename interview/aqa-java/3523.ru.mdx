# Повторный put в HashMap

## Суть

`HashMap.put(key, value)` при совпадающем ключе (equals + hashCode) заменяет старое значение новым и возвращает предыдущее. Размер map не растёт, меняется только value в bucket. В тестовых кэшах и config map это типичный способ обновить expected без удаления ключа.

## Что ответить на собесе

- Второй `put("env", "stage")` перезапишет `"prod"`.
- `putIfAbsent` не трогает value, если ключ уже есть.
- `merge` даёт функцию слияния при коллизии.
- Для assert сравниваю финальный `get`, не историю значений.
- Честно: mutable key после put делает entry «потерянным», это не overwrite, а orphan.

## Как это работает

Замена value:

```
Map<String, String> config = new HashMap<>();
config.put("baseUrl", "https://dev.example");
config.put("baseUrl", "https://stage.example");
String current = config.get("baseUrl");
```

Python mirror:

```python
config: dict[str, str] = {}
config["baseUrl"] = "https://dev.example"
config["baseUrl"] = "https://stage.example"
assert config["baseUrl"] == "https://stage.example"
```

## Подводные камни

**Ключ без стабильного hashCode.** Повторный put с «другим» equal object ведёт себя неожиданно.

**Null key один раз.** Второй `put(null, v2)` снова overwrite, не второй null slot.

**Concurrent put без sync.** Lost update в parallel worker.

**Забытый return старого value.** Нельзя audit что именно заменили.

**Identity vs equals.** `IdentityHashMap` сравнивает по `==`, не по equals.
