# Сложность вставки в LinkedList

## Суть

Вставка в двусвязный список стоит O(1), если уже есть ссылка на узел, после которого вставляем. Поиск позиции по индексу или значению требует обхода и даёт O(n). В тестовых утилитах список редко выбирают ради random access, но важно понимать trade-off при хранении очереди шагов.

## Что ответить на собесе

- `add(index, e)` сначала ищет узел: O(n), потом перелинковка: O(1).
- `addFirst` / `addLast` с известным head/tail: O(1).
- `ArrayList` быстрее на чтение по индексу, `LinkedList` на частые вставки в середину при итераторе.
- В parallel suite не шарю mutable `LinkedList` без sync.
- Честно: для 10 элементов разница незаметна, выбор по API, не по Big-O на бумаге.

## Как это работает

Вставка после известного узла:

```
class Node {
  int value;
  Node next;
  Node prev;
}

void insertAfter(Node anchor, int data) {
  Node fresh = new Node();
  fresh.value = data;
  fresh.next = anchor.next;
  fresh.prev = anchor;
  if (anchor.next != null) anchor.next.prev = fresh;
  anchor.next = fresh;
}
```

Поиск по индексу перед вставкой:

```
void insertAt(LinkedList list, int index, int data) {
  Node cursor = list.head;
  for (int i = 0; i < index; i++) cursor = cursor.next;
  insertAfter(cursor.prev == null ? sentinel : cursor, data);
}
```

Python mirror:

```python
from collections import deque


def append_step(steps: deque[str], step: str) -> None:
    steps.append(step)


def insert_after_known(steps: list[str], pos: int, step: str) -> None:
    steps.insert(pos, step)
```

## Подводные камни

**Итератор + structural modify.** `ConcurrentModificationException` при вставке во время foreach.

**Путаница list vs deque.** В Python `list.insert(0)` O(n), не путать с Java `LinkedList`.

**get(i) в цикле.** O(n²) при построении expected sequence.

**Null head.** Вставка «после null» без sentinel ломает цепочку.

**Boxed Integer в узле.** Лишние аллокации в hot path микробенчмарка, не в UI-тесте.
