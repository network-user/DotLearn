# Applying interfaces in code

## In short

You declare an interface with `interface`, a class attaches it via `implements` and must implement abstract methods. Calls go through the interface type, enabling polymorphism and implementation swaps in tests. Default methods carry shared logic without a base class.

## How to answer in an interview

- `class ChromeDriverAdapter implements WebDriver` to isolate the Selenium API.
- In DI I pass a `Clock` interface; in tests I substitute a fixed clock.
- Multiple interfaces on one class: `implements Runnable, AutoCloseable`.
- `@Override` on every hook for compile-time signature checks.
- Honestly: a default method on an interface changes binary compatibility; be careful in shared libs.

## How it works

Polymorphic call:

```
interface StepReporter {
  void onPass(String name);
  default void onSkip(String name) { onPass(name + " (skipped)"); }
}

class JsonReporter implements StepReporter {
  public void onPass(String name) { /* write json */ }
}

class NoOpReporter implements StepReporter {
  public void onPass(String name) { }
}

void execute(StepReporter reporter) {
  reporter.onPass("login");
}
```

Python mirror:

```python
from typing import Protocol


class StepReporter(Protocol):
    def on_pass(self, name: str) -> None: ...


class JsonReporter:
    def on_pass(self, name: str) -> None:
        pass


def execute(reporter: StepReporter) -> None:
    reporter.on_pass("login")
```

## Pitfalls

**public on interface methods.** A forgotten modifier in older styles breaks compilation.

**Diamond default methods.** Two defaults with the same signature require explicit resolution.

**Mock only the interface.** A class without an interface is harder to stub without PowerMock.

**Too broad implements.** A test double is forced to implement extra methods.

**Serialization proxy.** A dynamic proxy over an interface is not always equals-friendly in asserts.
