# == vs equals() in asserts

## In short

`==` on primitives compares values; on objects it compares references (same instance). `equals()` compares logical content when overridden. In tests you almost always want `assertEquals` (equals), not `==`, except for singleton or enum identity checks.

## How to answer in an interview

- Strings: always `assertEquals(expected, actual)`, never `==`.
- DTOs: overridden equals or AssertJ `usingRecursiveComparison`.
- Primitives: `==` is fine in `assertTrue(a == b)`.
- Enum constants: `==` is safe; equals is also true.
- Honestly: String pool literals may match with `==`; I do not rely on that.

## How it works

```
String a = new String("ok");
String b = new String("ok");
assertFalse(a == b);
assertTrue(a.equals(b));
```

Custom DTO:

```
record CaseId(int value) {}

assertEquals(new CaseId(1), new CaseId(1));
```

Python mirror:

```python
def same_content(a: str, b: str) -> bool:
    return a == b


def test_value_equality():
    assert same_content("ok", "ok")
```

## Pitfalls

**String intern.** `"a" == "a"` is true; `new String("a")` is not.

**equals without hashCode.** HashSet/HashMap asserts break.

**Double equals.** 0.1 + 0.2 needs delta assert.

**Arrays.** Use `assertArrayEquals`, not array `equals`.

**Mockito same().** Checks identity, not equals content.
