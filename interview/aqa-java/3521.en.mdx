# Using abstract classes in practice

## In short

An abstract class cannot be instantiated directly: it provides a shared skeleton with partially implemented methods and required abstract hooks. A subclass inherits shared state and behavior and adds specifics. In a test framework this is a typical base page or base test with common setup and an abstract scenario step.

## How to answer in an interview

- `abstract class BaseTest` with `@BeforeEach` and abstract `void runScenario()`.
- Shared `openApp()` lives in the base; the concrete flow in the subclass.
- You cannot `new BaseTest()`, only a concrete subclass.
- If I only need a contract without state, I prefer an interface.
- Honestly: a deep abstract base hierarchy complicates parallel runs and Mockito spies.

## How it works

Base scenario template:

```
abstract class BaseUiTest {
  WebDriver driver;

  void setUp() { driver = createDriver(); }

  abstract void exerciseFlow();

  void tearDown() { driver.quit(); }
}

class CheckoutTest extends BaseUiTest {
  void exerciseFlow() {
    driver.get("/cart");
    driver.findElement(By.id("pay")).click();
  }
}
```

Python mirror:

```python
from abc import ABC, abstractmethod


class BaseUiTest(ABC):
    def setup(self) -> None:
        self.driver = create_driver()

    @abstractmethod
    def exercise_flow(self) -> None:
        raise NotImplementedError

    def teardown(self) -> None:
        self.driver.quit()
```

## Pitfalls

**Abstract without subclasses.** A class with an abstract method must be abstract.

**Constructor chain.** The super constructor runs before child fields; NPE in setup.

**Template method rigidity.** A fixed step order breaks reuse in data-driven tests.

**Shared mutable driver field.** Flaky with `@TestInstance(PER_CLASS)`.

**Mixing abstract and final.** A final abstract method is meaningless and does not compile.
