# Java collection families

## In short

The Java Collections Framework splits structures into List (ordered access with duplicates), Set (unique elements), and Map (unique keys with values). Concrete implementations (`ArrayList`, `HashSet`, `HashMap`, and others) differ in ordering, speed, and thread safety. In tests the collection choice affects assert determinism and setup cost.

## How to answer in an interview

- List: `ArrayList` for random access, `LinkedList` for frequent inserts with an iterator.
- Set: `HashSet` unordered, `LinkedHashSet` keeps insertion order, `TreeSet` sorts.
- Map: `HashMap` by default, `LinkedHashMap` for LRU-like behavior, `TreeMap` by key.
- For expected values in asserts I use `LinkedHashSet` when order matters without duplicates.
- Honestly: `ConcurrentHashMap` is needed for a shared cache between parallel workers, not "just in case".

## How it works

Three families in one scenario:

```
List<String> steps = new ArrayList<>();
steps.add("login");
steps.add("checkout");

Set<String> roles = new LinkedHashSet<>();
roles.add("admin");
roles.add("admin");

Map<String, Integer> retries = new HashMap<>();
retries.put("api", 3);
retries.put("ui", 1);
```

Python mirror:

```python
steps: list[str] = ["login", "checkout"]
roles: set[str] = {"admin"}
retries: dict[str, int] = {"api": 3, "ui": 1}


def unique_ordered(items: list[str]) -> list[str]:
    return list(dict.fromkeys(items))
```

## Pitfalls

**HashSet without equals/hashCode.** A custom object "duplicates" in the set.

**Mutable key in HashMap.** Changing a key after put loses the entry.

**Arrays.asList fixed size.** `add` throws UnsupportedOperationException.

**SubList structural change.** A detached view breaks after modifying the parent.

**TreeSet without Comparable.** ClassCastException on insert without a comparator.
