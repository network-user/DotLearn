# Java data types in automation tests

## In short

Java splits types into primitives (eight value types on the stack) and reference types (classes, interfaces, arrays, enums on the heap). Wrappers (`Integer`, `Boolean`) are required for generics and nullable test data fields. Type choice affects comparison, fixture serialization, and boxing overhead.

## How to answer in an interview

- `int`/`long` for counters and ids in asserts; `BigDecimal` for money instead of `double`.
- `boolean` for feature flags, not `Boolean` unless you need null semantics.
- `String` is immutable and safe as a key and expected value in JSON comparison.
- Arrays vs `List`: in parameterized tests I prefer `List` and streams.
- Honestly: autoboxing a null `Integer` into arithmetic causes NPE without an explicit guard.

## How it works

Primitives and references:

```
int caseId = 42;
long startedAt = System.currentTimeMillis();
boolean flaky = false;
double ratio = 0.95;

Integer nullableRetries = null;
String suiteName = "checkout-smoke";
int[] priorities = {1, 2, 3};
```

Enum for statuses:

```
enum RunState { QUEUED, RUNNING, PASSED, FAILED }
RunState state = RunState.PASSED;
```

Python mirror for pytest parametrize:

```python
from enum import Enum


class RunState(Enum):
    QUEUED = "queued"
    PASSED = "passed"


def test_typed_fixture(run_state: RunState = RunState.PASSED) -> None:
    assert run_state == RunState.PASSED
```

## Pitfalls

**float comparison.** `assertEquals(0.1 + 0.2, 0.3)` fails; use delta or `BigDecimal`.

**char vs String.** `'A'` is not equal to `"A"` in asserts without conversion.

**Primitive default.** Default `int` 0 hides uninitialized fields in POJO fixtures.

**Raw array covariance.** Assigning `String[]` to `Object[]` leads to `ArrayStoreException` at runtime.

**Optional is not a null replacement.** In test DTOs prefer an explicit nullable wrapper or validation.
