# Heap памяти JVM

## Суть

Heap хранит все Java-объекты, создаваемые через `new` и массивы. Ссылки на объекты лежат в stack frame или в полях других объектов, сами данные в heap. GC освобождает unreachable объекты; размер heap настраивается `-Xmx`. Для Selenium suite важно не держать лишние объекты в static полях между тестами.

## Что ответить на собесе

- `new ChromeDriver()` аллоцирует объект в heap, переменная driver в stack держит reference.
- Young/Old generation: короткоживущие page object умирают в Minor GC.
- OutOfMemoryError при исчерпании heap, не путать с stack overflow.
- `-Xmx` ограничивает верхнюю границу heap процесса.
- Честно: утечка через static `List<WebElement>` переживает GC, пока ссылка жива.

## Как это работает

Создание объекта:

```
class Session {
  String token;
}

void open() {
  Session session = new Session();
  session.token = "abc";
}
```

## Подводные камни

**Static cache.** Heap не очистится, пока classloader жив.

**Huge screenshot byte[].** Один кадр может занять много Old Gen.

**Premature promotion.** Долгий `@BeforeAll` держит объекты в Old, дорогой Full GC.

**Metaspace vs Heap.** Class metadata не в heap, OOM бывает отдельно.

**False leak в profiler.** Живой driver session держит DOM snapshots намеренно.
