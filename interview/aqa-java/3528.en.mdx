# Abstract class as a template

## In short

An abstract class combines partially ready implementation and required abstract methods the subclass must finish. Direct `new` is forbidden; only concrete subclasses work. It templates a family of entities with shared behavior and different details, typical for a base test or base page in automation.

## How to answer in an interview

- `abstract void fillForm()` in the base; `submit()` is already implemented.
- Superclass fields are visible to the child, unlike a pure interface before Java 8 fields.
- One extends, several implements: `class X extends Base implements RetryPolicy`.
- If an abstract method is not implemented, the class stays abstract.
- Honestly: a fat abstract base is worse than composing helper objects in modern test design.

## How it works

Session template (renamed teaching example):

```
abstract class BrowserSession {
  abstract void openHome();

  void logStart() {
    System.out.println("session boot");
  }
}

class ChromeRun extends BrowserSession {
  void openHome() { /* navigate chrome */ }
}

class FirefoxRun extends BrowserSession {
  void openHome() { /* navigate firefox */ }
}
```

Python mirror:

```python
from abc import ABC, abstractmethod


class BrowserSession(ABC):
    @abstractmethod
    def open_home(self) -> None:
        raise NotImplementedError

    def log_start(self) -> None:
        print("session boot")
```

## Pitfalls

**Forgotten abstract on the class.** Compile error on the first abstract method.

**Private abstract.** Not allowed: abstract must be visible to the subclass.

**Constructor in abstract.** Child must call super or the object is incomplete.

**Non-static abstract in static context.** Confusion with static abstract (not allowed).

**Polymorphic array.** `BrowserSession[]` with different children is fine, but equals/hash are not unified.
