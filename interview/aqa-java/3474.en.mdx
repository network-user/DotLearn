# Key features of the Java Stream API

## In short

The Stream API (Java 8+) offers a functional pipeline over data: lazy intermediate ops, eager terminal ops, optional parallel fork. A stream does not store elements; it describes computation on a source. That cuts boilerplate in test utilities but requires knowing streams are single-use and which ops are stateless vs stateful.

## How to answer in an interview

- Lazy: `filter` does not run until `collect` appears.
- `parallelStream()` splits the spliterator; the source must be immutable.
- Stateless ops (`map`, `filter`) compose; `sorted` and `distinct` are stateful and cost memory.
- Primitive streams (`IntStream`) avoid boxing in perf-sensitive asserts.
- Honestly: parallel on 2-vCPU CI can be slower due to fork/join overhead.

## How it works

Lazy chain:

```
long failed = cases.stream()
    .filter(c -> c.durationMs() > 500)
    .peek(c -> log.debug(c.name()))
    .count();
```

Parallel:

```
List<String> urls = pages.parallelStream()
    .map(Page::canonicalUrl)
    .collect(Collectors.toList());
```

Python summary table:

```python
FEATURES = {
    "lazy_intermediate": "runs only after terminal",
    "parallel": "needs thread-safe source",
    "no_storage": "pipeline describes computation",
}


def test_stream_features_doc():
    assert FEATURES["lazy_intermediate"].startswith("runs")
```

## Pitfalls

**sorted on an infinite stream.** Hang or OOM on generators.

**peek for business logic.** Side effect only, not a map replacement.

**parallel with ThreadLocal.** Context is lost across workers.

**toMap duplicate key.** IllegalStateException without a merge function.

**Stream closed twice.** IllegalStateException in nested helpers.
