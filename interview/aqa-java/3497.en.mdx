# Integer overflow in test calculations

## In short

`int`/`Integer` in Java are 32-bit signed: range −2,147,483,648 through 2,147,483,647. Crossing the bound does not throw; it wraps (overflow/underflow). In tests this breaks asserts on counters, timestamp diffs, and aggregates unless bounds are checked explicitly.

## How to answer in an interview

- For large ids and sums I use `long` or `BigInteger` in expected values.
- `Math.addExact`/`multiplyExact` throw `ArithmeticException` instead of silent wrap.
- I test boundaries: `Integer.MAX_VALUE`, `MIN_VALUE` in property-based tests.
- JSON numbers above 2^31 may arrive as long in Jackson; do not mix types.
- Honestly: silent overflow in prod is worse than NPE; in tests I use exact math APIs.

## How it works

```
int peak = Integer.MAX_VALUE;
int wrapped = peak + 1;
// wrapped == Integer.MIN_VALUE

long safe = (long) peak + 1L;
assertEquals(2147483648L, safe);
```

Exact arithmetic:

```
int total = Math.addExact(peak, 1);
```

Python mirror (unbounded int, but contract test):

```python
JAVA_INT_MAX = 2_147_483_647


def java_add_int(a: int, b: int) -> int:
    value = a + b
    if value > JAVA_INT_MAX or value < -JAVA_INT_MAX - 1:
        raise OverflowError("int32 wrap")
    return value


def test_overflow_guard():
    import pytest
    with pytest.raises(OverflowError):
        java_add_int(JAVA_INT_MAX, 1)
```

## Pitfalls

**Counter increment.** After MAX_VALUE a test expecting +1 fails obscurely.

**Multiplication.** `100000 * 100000` wraps without an exception.

**Compare with long.** `int` promotes, but intermediate overflow already happened.

**Integer cache.** `valueOf(127)` is cached; irrelevant to overflow but affects identity asserts.

**Time millis cast.** `(int) System.currentTimeMillis()` is almost always wrong.
