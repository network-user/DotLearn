# Сложность add в ArrayList

## Суть

`add` в конец ArrayList амортизированно O(1): элемент кладётся в internal array по size index. Когда capacity исчерпана, создаётся новый массив (~1.5x), копируются все элементы: O(n) на resize. `add(index, e)` в середину сдвигает хвост: O(n). Для bulk fixture insert в perf test учитываю resize spikes.

## Что ответить на собесе

- Amortized O(1) для append, не worst-case каждый call.
- `ensureCapacity` заранее если знаю N rows.
- Middle insert в большой list дорог, prefer addAll end.
- Microbench в JMH, не один assert time.
- Честно: ten elements resize irrelevant; million rows без presize заметно.

## Как это работает

```
List<Integer> buf = new ArrayList<>(1024);
for (int i = 0; i < 10_000; i++) {
  buf.add(i);
}
```

Resize concept:

```
old [a,b,c,_,_] -> new capacity 6, copy 3 elements, add d
```

Python list growth analogue:

```python
def append_amortized(items: list[int], n: int) -> None:
    for i in range(n):
        items.append(i)


def test_growth():
    data: list[int] = []
    append_amortized(data, 1000)
    assert len(data) == 1000
    assert data[-1] == 999
```

## Подводные камни

**addAll resize once.** Лучше чем loop add без capacity hint.

**SubList add.** Structural mod counter parent.

**Concurrent add.** Not thread-safe без sync.

**Primitive list none.** Autobox Integer overhead в million loop.

**remove from end O(1).** middle remove тоже O(n), симметрично insert.
