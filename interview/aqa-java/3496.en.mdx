# The finally block and resource cleanup

## In short

The `finally` block in `try-catch-finally` runs after `try`/`catch` in almost all cases, even on return or throw. Historically it guaranteed IO close and cleanup. Since Java 7, try-with-resources is preferred, but `finally` remains for non-AutoCloseable cleanup and resetting test state.

## How to answer in an interview

- Close WebDriver, DB connections, and temp files in finally or try-with-resources.
- In finally, do not throw a new exception over the original without addSuppressed.
- For shared static state, `@AfterAll` teardown is cleaner than finally in every test.
- A return in try still runs finally before leaving the method.
- Honestly: try-with-resources covers 90% of cases; I keep finally for edge cleanup.

## How it works

Classic pattern:

```
Connection conn = null;
try {
  conn = dataSource.getConnection();
  runAssertionQuery(conn);
} catch (SQLException ex) {
  fail(ex.getMessage());
} finally {
  if (conn != null) conn.close();
}
```

Try-with-resources (preferred):

```
try (var stream = Files.lines(Path.of("cases.csv"))) {
  assertTrue(stream.count() > 0);
}
```

Python mirror:

```python
def read_cases(path: str) -> list[str]:
    handle = open(path, encoding="utf-8")
    try:
        return [line.strip() for line in handle]
    finally:
        handle.close()
```

## Pitfalls

**finally return.** A return in finally overrides return/throw from try.

**Exception suppression.** An error in finally masks the test's root failure.

**Double close.** Without null-check or try-with-resources you risk NPE.

**System.exit.** finally does not run on a hard JVM kill.

**Slow finally.** Heavy cleanup in finally slows every failing test.
