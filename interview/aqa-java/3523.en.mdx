# Duplicate key put in HashMap

## In short

`HashMap.put(key, value)` with a matching key (equals + hashCode) replaces the old value with the new one and returns the previous entry. The map size does not grow; only the value in the bucket changes. In test caches and config maps this is the usual way to update expected data without removing the key.

## How to answer in an interview

- A second `put("env", "stage")` overwrites `"prod"`.
- `putIfAbsent` does not touch the value if the key already exists.
- `merge` supplies a merge function on collision.
- For asserts I compare the final `get`, not value history.
- Honestly: a mutable key after put makes the entry "lost"; that is not overwrite, it is orphaning.

## How it works

Value replacement:

```
Map<String, String> config = new HashMap<>();
config.put("baseUrl", "https://dev.example");
config.put("baseUrl", "https://stage.example");
String current = config.get("baseUrl");
```

Python mirror:

```python
config: dict[str, str] = {}
config["baseUrl"] = "https://dev.example"
config["baseUrl"] = "https://stage.example"
assert config["baseUrl"] == "https://stage.example"
```

## Pitfalls

**Key without stable hashCode.** A repeat put with another equal object behaves unexpectedly.

**Null key once.** A second `put(null, v2)` overwrites again; there is not a second null slot.

**Concurrent put without sync.** Lost update in a parallel worker.

**Forgotten return of old value.** You cannot audit what was replaced.

**Identity vs equals.** `IdentityHashMap` compares with `==`, not equals.
