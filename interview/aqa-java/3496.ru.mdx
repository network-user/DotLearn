# Блок finally и освобождение ресурсов

## Суть

Блок `finally` в `try-catch-finally` выполняется после `try`/`catch` почти всегда, даже при return или throw. Исторически он гарантировал закрытие IO и cleanup. С Java 7 предпочтительнее try-with-resources, но `finally` остаётся для non-AutoCloseable cleanup и сброса test state.

## Что ответить на собесе

- Закрываю WebDriver, DB connection, temp files в finally или try-with-resources.
- В finally не бросаю новое исключение поверх исходного без addSuppressed.
- Для shared static state в suite teardown `@AfterAll` чище, чем finally в каждом тесте.
- return в try всё равно запускает finally перед выходом из метода.
- Честно: try-with-resources покрывает 90% кейсов, finally оставляю для edge cleanup.

## Как это работает

Классический паттерн:

```
Connection conn = null;
try {
  conn = dataSource.getConnection();
  runAssertionQuery(conn);
} catch (SQLException ex) {
  fail(ex.getMessage());
} finally {
  if (conn != null) conn.close();
}
```

Try-with-resources (предпочтительно):

```
try (var stream = Files.lines(Path.of("cases.csv"))) {
  assertTrue(stream.count() > 0);
}
```

Python mirror:

```python
def read_cases(path: str) -> list[str]:
    handle = open(path, encoding="utf-8")
    try:
        return [line.strip() for line in handle]
    finally:
        handle.close()
```

## Подводные камни

**finally return.** return в finally перетирает return/throw из try.

**Подавление исключения.** Ошибка в finally маскирует root failure теста.

**Двойной close.** Без null-check или try-with-resources риск NPE.

**System.exit.** finally не выполнится при жёстком kill JVM.

**Долгий finally.** Тяжёлый cleanup в finally замедляет каждый failing test.
