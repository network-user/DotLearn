# Checked and unchecked exceptions in tests

## In short

Checked exceptions (`IOException`, `SQLException`) must be caught or declared in `throws` at compile time. Unchecked ones (`RuntimeException` and subclasses) are not compile-time checked. In tests, checked reflect external IO/DB failures; unchecked usually signal a bug or bad input in an assert helper.

## How to answer in an interview

- `assertThrows(IllegalArgumentException.class, ...)` for unchecked contracts.
- Checked in a test: either `throws` on the `@Test` method or try/catch with fail.
- Do not swallow checked with an empty catch or the test goes green falsely.
- Custom unchecked `DomainException` for API client business errors.
- Honestly: wrapping every checked into unchecked is debatable; in tests I prefer explicit throws.

## How it works

Checked:

```
void loadFixture(String path) throws IOException {
  Files.readString(Path.of(path));
}
```

Unchecked:

```
int parsePort(String raw) {
  return Integer.parseInt(raw);
}
```

JUnit 5:

```
assertThrows(NumberFormatException.class, () -> parsePort("abc"));
```

Python mirror:

```python
def parse_port(raw: str) -> int:
    return int(raw)


def test_unchecked_parse():
    import pytest
    with pytest.raises(ValueError):
        parse_port("abc")
```

## Pitfalls

**throws on the test.** Forgotten throws hides unexpected failure as error vs failure.

**catch Exception.** Masks AssertionError in JUnit 3 style tests.

**Error subclasses.** Do not catch OutOfMemory with catch Exception.

**Checked in lambda.** Needs a wrapper or sneaky throw.

**assertDoesNotThrow.** Asserts no exception at all, including unchecked NPE.
