# Stream API: filter-map-reduce chains in tests

## In short

The Stream API processes a sequence with a declarative chain: intermediate ops (`filter`, `map`, `sorted`) are lazy; a terminal op (`collect`, `reduce`, `forEach`) runs the pipeline. Sources include collections, arrays, or `Stream.of`. For AQA it aggregates parallel case results or filters API payloads without imperative loops in assertion helpers.

## How to answer in an interview

- I create streams via `list.stream()` without mutating the source collection.
- I describe intermediate steps as one pipeline with a single terminal.
- For parallel streams I verify thread safety and deterministic asserts.
- In service tests I compare `collect(toList())` against an expected fixture.
- Honestly: streaming a three-item list buys nothing; readability wins.

## How it works

```
List<Integer> scores = List.of(3, 8, 12, 15, 4);

int doubledEvens = scores.stream()
    .filter(n -> n % 2 == 0)
    .map(n -> n * 2)
    .reduce(0, Integer::sum);
```

Collect to list:

```
List<String> tags = results.stream()
    .filter(r -> r.status().equals("FAIL"))
    .map(TestResult::caseName)
    .sorted()
    .collect(Collectors.toList());
```

Python analogue for an assertion helper:

```python
def sum_doubled_evens(values: list[int]) -> int:
    return sum(n * 2 for n in values if n % 2 == 0)


def test_stream_like_pipeline():
    assert sum_doubled_evens([3, 8, 12, 15, 4]) == 40
```

## Pitfalls

**Side effects in forEach.** Mutating an outer list from a stream breaks parallel runs.

**Reusing a stream.** After a terminal op the stream is consumed; a second collect fails.

**Null in stream.** NPE in map; filter with `Objects.nonNull` first.

**boxed vs primitive.** `mapToInt` beats autoboxing `Integer` floods.

**Unstable Comparator.** Equal elements may reorder between JDK runs.
