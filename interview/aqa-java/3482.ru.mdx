# Массив и коллекция: выбор для test data

## Суть

Массив Java фиксированной длины, homogenous by type, O(1) index, minimal overhead. Collection (List/Set/Map) динамически растёт, даёт utility API (add, remove, stream). Примитивы: `int[]` vs boxed `ArrayList<Integer>`. В parametrized test provider часто `Object[][]` или `List<Arguments>`.

## Что ответить на собесе

- Fixed small matrix expected values → array.
- Unknown count API items → ArrayList.
- Unique tag set → HashSet before assert.
- `@ParameterizedTest` с `MethodSource` List stream-friendly.
- Честно: arrays.asList fixed-size list ломает add в fixture builder.

## Как это работает

Array:

```
int[] thresholds = { 100, 500, 1000 };
assert thresholds[1] == 500;
```

Collection:

```
List<String> cases = new ArrayList<>();
cases.add("login");
cases.add("checkout");
```

Conversion:

```
String[] arr = cases.toArray(String[]::new);
List<String> again = Arrays.asList(arr);
```

Python comparison:

```python
def test_array_vs_list():
    fixed = (1, 2, 3)
    dynamic = [1, 2, 3]
    dynamic.append(4)
    assert len(dynamic) == 4
    assert len(fixed) == 3
```

## Подводные камни

**Array covariant trap.** `Object[] o = new String[1]; o[0]=1` runtime error.

**Arrays.asList mutate.** set ok, add throws.

**toArray zero-size.** JVM alloc правильного размера, не `new String[0]` blindly.

**Primitive array print.** default toString useless в log assert.

**List.of immutable.** add в setup падает, нужен mutable copy.
