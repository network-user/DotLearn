# equals/hashCode for HashMap keys

## In short

`HashMap` looks up a key first by `hashCode` (bucket), then by `equals` in the chain. Contract: objects equal by `equals` must share the same `hashCode`. Without a consistent pair, lookup, remove, and dedup in test maps with DTO keys behave unpredictably.

## How to answer in an interview

- For a test data key class I override both methods or use Lombok `@EqualsAndHashCode`.
- In `equals` I check type, use null-safe fields, and avoid relying on a single mutable field without copy.
- `hashCode` combines fields stably (31 * result + fieldHash).
- In asserts, `map.get(new Key(id))` should find an entry stored with another instance.
- Honestly: if the key is mutable, you cannot change fields after put or the entry is lost.

## How it works

```
class CaseKey {
  private final String suite;
  private final int id;

  CaseKey(String suite, int id) {
    this.suite = suite;
    this.id = id;
  }

  @Override
  public boolean equals(Object other) {
    if (this == other) return true;
    if (!(other instanceof CaseKey)) return false;
    CaseKey peer = (CaseKey) other;
    return id == peer.id && suite.equals(peer.suite);
  }

  @Override
  public int hashCode() {
    return 31 * suite.hashCode() + id;
  }
}
```

Python mirror:

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class CaseKey:
    suite: str
    case_id: int


def test_map_key_contract():
    store = {CaseKey("smoke", 1): "ok"}
    assert store[CaseKey("smoke", 1)] == "ok"
```

## Pitfalls

**equals without hashCode.** Classic bug: put works, get does not.

**Inheritance in equals.** Liskov breaks if a subclass adds a field not accounted for in super.

**Identity vs equality.** `==` on keys in map asserts is almost always wrong.

**Enum/String keys.** Already correct; do not override unnecessarily.

**Weak/Soft keys.** Special cache cases; do not confuse with ordinary test HashMaps.
