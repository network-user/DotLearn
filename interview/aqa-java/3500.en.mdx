# Intermediate and terminal Stream operations

## In short

A Stream pipeline uses intermediate operations (`filter`, `map`, `sorted`) that return a new Stream, and terminal ones (`collect`, `count`, `forEach`) that trigger the lazy chain and produce a result. Without a terminal, intermediates do not run. In tests, Stream helps filter API responses and build expected collections.

## How to answer in an interview

- Intermediates are lazy; terminals are eager: `collect` materializes the pipeline.
- You cannot reuse one Stream after a terminal.
- `parallelStream` in tests only for CPU-bound work on isolated data.
- I prefer `toList()` (Java 16+) over mutable `Collectors.toList()`.
- Honestly: for mapping three elements, a loop is more readable in an assert helper.

## How it works

```
List<String> cases = List.of("api-smoke", "ui-reg", "api-load");

List<String> apiOnly = cases.stream()
  .filter(name -> name.startsWith("api"))
  .map(String::toUpperCase)
  .toList();
```

Intermediates do not run until a terminal:

```
Stream<String> pipeline = cases.stream().filter(s -> s.contains("api"));
long count = pipeline.count();
```

Python mirror:

```python
def api_cases(names: list[str]) -> list[str]:
    return [n.upper() for n in names if n.startswith("api")]


def test_stream_like_filter():
    assert api_cases(["api-smoke", "ui-reg"]) == ["API-SMOKE"]
```

## Pitfalls

**Double terminal.** Reusing a stream throws `IllegalStateException`.

**Side effects in forEach.** Mutating external state breaks parallel runs and readability.

**Null in filter/map.** NPE without `filter(Objects::nonNull)`.

**sorted without comparator.** Natural order may not match business order in asserts.

**Primitive streams.** `mapToInt` avoids boxing overhead on large fixtures.
