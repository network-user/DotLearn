# Race conditions and thread-safety in suites

## In short

A race condition happens when multiple threads read/write shared mutable state without synchronization and the outcome depends on scheduling. Thread-safe code keeps invariants under concurrent access via locks, atomics, or immutable data. In parallel JUnit and shared static fixtures this is the main source of flakiness.

## How to answer in an interview

- Do not share a mutable `ArrayList` across `@Test` without sync.
- `AtomicInteger` for a retry counter in a listener, not `int++`.
- Immutable expected values plus locals in each test.
- `ConcurrentHashMap` for a shared metadata cache in the runner.
- Honestly: `@TestInstance(PER_CLASS)` with instance fields needs explicit thread-safety discipline.

## How it works

Unsafe counter:

```
class Hits {
  int value;
  void inc() { value++; }
}
```

Two threads × 1000 `inc()` may yield less than 2000.

Thread-safe:

```
AtomicInteger hits = new AtomicInteger();
hits.incrementAndGet();
```

Python mirror:

```python
import threading


def test_race_without_lock():
    total = {"n": 0}

    def inc() -> None:
        for _ in range(1000):
            total["n"] += 1

    threads = [threading.Thread(target=inc), threading.Thread(target=inc)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()
    assert total["n"] <= 2000
```

## Pitfalls

**check-then-act.** if (!map.contains) put without sync races.

**volatile is not atomicity.** `count++` stays racy with volatile.

**Static SimpleDateFormat.** Classic thread-unsafe singleton in utils.

**Parallel tests + same user.** DB state collision, not JMM, but looks like a race.

**synchronized on the whole suite.** Kills parallel and hides a design bug.
