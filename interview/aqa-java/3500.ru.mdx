# Промежуточные и терминальные операции Stream

## Суть

Stream pipeline строится из intermediate операций (`filter`, `map`, `sorted`), возвращающих новый Stream, и terminal (`collect`, `count`, `forEach`), запускающих ленивую цепочку и выдающих результат. Без terminal intermediate не выполняются. В тестах Stream удобен для фильтрации API ответов и построения expected коллекций.

## Что ответить на собесе

- Intermediate ленивые, terminal eager: `collect` материализует pipeline.
- Один Stream нельзя переиспользовать после terminal.
- `parallelStream` в тестах только при CPU-bound и изолированных данных.
- Предпочитаю `toList()` (Java 16+) вместо mutable `Collectors.toList()`.
- Честно: для простого map из 3 элементов цикл читабельнее stream в assert helper.

## Как это работает

```
List<String> cases = List.of("api-smoke", "ui-reg", "api-load");

List<String> apiOnly = cases.stream()
  .filter(name -> name.startsWith("api"))
  .map(String::toUpperCase)
  .toList();
```

Intermediate не бегут до terminal:

```
Stream<String> pipeline = cases.stream().filter(s -> s.contains("api"));
long count = pipeline.count();
```

Python mirror:

```python
def api_cases(names: list[str]) -> list[str]:
    return [n.upper() for n in names if n.startswith("api")]


def test_stream_like_filter():
    assert api_cases(["api-smoke", "ui-reg"]) == ["API-SMOKE"]
```

## Подводные камни

**Двойной terminal.** Повторное использование stream бросает `IllegalStateException`.

**Side effects в forEach.** Мутация внешнего state ломает parallel и читабельность.

**Null в filter/map.** NPE без `filter(Objects::nonNull)`.

**sorted без comparator.** Natural order может не совпасть с business order в assert.

**Примитивные streams.** `mapToInt` избегает boxing overhead в больших фикстурах.
