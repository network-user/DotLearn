# Семейства коллекций Java

## Суть

Java Collections Framework делит структуры на List (упорядоченный доступ с дубликатами), Set (уникальные элементы) и Map (уникальные ключи со значениями). Конкретные реализации (`ArrayList`, `HashSet`, `HashMap` и др.) отличаются по порядку, скорости и thread-safety. В тестах выбор коллекции влияет на детерминизм assert и стоимость setup.

## Что ответить на собесе

- List: `ArrayList` для random access, `LinkedList` для частых вставок с итератором.
- Set: `HashSet` без порядка, `LinkedHashSet` сохраняет insertion order, `TreeSet` сортирует.
- Map: `HashMap` дефолт, `LinkedHashMap` для LRU-like, `TreeMap` по ключу.
- Для expected в assert беру `LinkedHashSet`, если важен порядок без дублей.
- Честно: `ConcurrentHashMap` нужен при shared cache между parallel workers, не «на всякий случай».

## Как это работает

Три семейства в одном сценарии:

```
List<String> steps = new ArrayList<>();
steps.add("login");
steps.add("checkout");

Set<String> roles = new LinkedHashSet<>();
roles.add("admin");
roles.add("admin");

Map<String, Integer> retries = new HashMap<>();
retries.put("api", 3);
retries.put("ui", 1);
```

Python mirror:

```python
steps: list[str] = ["login", "checkout"]
roles: set[str] = {"admin"}
retries: dict[str, int] = {"api": 3, "ui": 1}


def unique_ordered(items: list[str]) -> list[str]:
    return list(dict.fromkeys(items))
```

## Подводные камни

**HashSet без equals/hashCode.** Кастомный объект «дублируется» в set.

**Mutable key в HashMap.** Изменение ключа после put теряет entry.

**Arrays.asList fixed size.** `add` бросает UnsupportedOperationException.

**SubList structural change.** Detached view ломается после modify parent.

**TreeSet без Comparable.** ClassCastException на insert без comparator.
