# Checked и unchecked исключения в тестах

## Суть

Checked exceptions (`IOException`, `SQLException`) компилятор заставляет ловить или объявлять в `throws`. Unchecked (`RuntimeException` и наследники) не проверяются compile-time. В тестах checked отражают внешние сбои IO/DB, unchecked чаще сигналят баг или неверный input в assert helper.

## Что ответить на собесе

- `assertThrows(IllegalArgumentException.class, ...)` для unchecked контракта.
- Checked в тесте: либо `throws` на `@Test` method, либо try/catch с fail.
- Не глотаю checked пустым catch, тест станет ложноположительным.
- Custom unchecked `DomainException` для бизнес-ошибок API клиента.
- Честно: оборачивать каждый checked в unchecked спорно, в тестах предпочитаю явный throws.

## Как это работает

Checked:

```
void loadFixture(String path) throws IOException {
  Files.readString(Path.of(path));
}
```

Unchecked:

```
int parsePort(String raw) {
  return Integer.parseInt(raw);
}
```

JUnit 5:

```
assertThrows(NumberFormatException.class, () -> parsePort("abc"));
```

Python mirror:

```python
def parse_port(raw: str) -> int:
    return int(raw)


def test_unchecked_parse():
    import pytest
    with pytest.raises(ValueError):
        parse_port("abc")
```

## Подводные камни

**throws на тесте.** Забытый throws скрывает неожиданный failure как error vs failure.

**catch Exception.** Маскирует AssertionError в JUnit 3 style тестах.

**Error subclasses.** OutOfMemory не ловлю catch Exception.

**Checked в lambda.** Нужен wrapper или sneaky throw.

**assertDoesNotThrow.** Проверяет отсутствие любого exception, включая unchecked NPE.
