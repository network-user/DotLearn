# Override и overload в Java page objects

## Суть

Overload: несколько методов с одним именем и разными параметрами в одном классе, compile-time dispatch. Override: подкласс заменяет реализацию метода с той же сигнатурой, runtime dispatch по фактическому типу объекта. В Selenium Java base page часто override `open()`, overload `fillForm` для разных DTO.

## Что ответить на собесе

- `@Override` ловит опечатку в имени при refactor.
- Override требует совместимую сигнатуру и covariant return (Java 5+).
- Overload выбирается статически по compile-time типу reference.
- Полиморфный assert: `WebElement action = page; action.click()` вызывает override.
- Честно: hide overload с varargs легко создаёт ambiguous call в test helper.

## Как это работает

Override:

```
class BasePage {
  void waitReady() { /* spinner */ }
}

class CheckoutPage extends BasePage {
  @Override
  void waitReady() { /* pay widget */ }
}
```

Overload:

```
void type(String value) { type(By.id("field"), value); }
void type(By locator, String value) { driver.findElement(locator).sendKeys(value); }
```

Python analogy:

```python
class BasePage:
    def wait_ready(self) -> None:
        pass


class CheckoutPage(BasePage):
    def wait_ready(self) -> None:
        self._wait_pay_widget()


def fill(field: str, value: str) -> None:
    pass


def fill_locator(locator: str, value: str) -> None:
    pass


def test_override_runtime():
    page: BasePage = CheckoutPage()
    page.wait_ready()
```

## Подводные камни

**private method «override».** Hiding, не polymorphism.

**static method override.** Нет runtime dispatch.

**Overload vs override signature clash.** Компилятор путает при narrow change.

**Bridge methods generics.** Reflection видит synthetic, assert stack странный.

**Lombok @Builder overload.** Generated methods скрывают intent в PO.
