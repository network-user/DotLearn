# Override and overload in Java page objects

## In short

Overload: several methods with the same name and different parameters in one class; compile-time dispatch. Override: a subclass replaces a method with the same signature; runtime dispatch uses the actual object type. In Selenium Java a base page often overrides `open()` and overloads `fillForm` for different DTOs.

## How to answer in an interview

- `@Override` catches name typos during refactors.
- Override needs a compatible signature and covariant return (Java 5+).
- Overload is chosen statically from the compile-time reference type.
- Polymorphic assert: `WebElement action = page; action.click()` calls the override.
- Honestly: varargs overloads in test helpers easily create ambiguous calls.

## How it works

Override:

```
class BasePage {
  void waitReady() { /* spinner */ }
}

class CheckoutPage extends BasePage {
  @Override
  void waitReady() { /* pay widget */ }
}
```

Overload:

```
void type(String value) { type(By.id("field"), value); }
void type(By locator, String value) { driver.findElement(locator).sendKeys(value); }
```

Python analogy:

```python
class BasePage:
    def wait_ready(self) -> None:
        pass


class CheckoutPage(BasePage):
    def wait_ready(self) -> None:
        self._wait_pay_widget()


def fill(field: str, value: str) -> None:
    pass


def fill_locator(locator: str, value: str) -> None:
    pass


def test_override_runtime():
    page: BasePage = CheckoutPage()
    page.wait_ready()
```

## Pitfalls

**«Override» of private method.** Method hiding, not polymorphism.

**static method override.** No runtime dispatch.

**Overload vs override signature clash.** Compiler confusion on narrow changes.

**Bridge methods with generics.** Reflection sees synthetics; assert stacks look odd.

**Lombok @Builder overload.** Generated methods hide intent in POs.
