# Optional: когда уместен в AQA

## Суть

`Optional<T>` явно моделирует «значение может отсутствовать» в return type метода. Он не предназначен для полей, параметров и коллекций. В тестах помогает безопасно читать nullable поля JSON DTO и цепочки `map`/`orElseThrow` вместо вложенных null-check.

## Что ответить на собесе

- Return `Optional<User>` из repository helper, не `null`.
- `orElseThrow` с понятным message для fail fast.
- `ifPresent` для soft assert side effect.
- Не оборачиваю каждый String в Optional в POJO.
- Честно: Optional в поле page object усложняет код без выгоды.

## Как это работает

```
Optional<Order> findById(String id) {
  return repo.load(id);
}

Order order = findById("42")
  .orElseThrow(() -> new AssertionError("order missing"));
```

Цепочка:

```
String city = findById("42")
  .map(Order::address)
  .map(Address::city)
  .orElse("unknown");
```

Python mirror:

```python
from typing import Optional


def find_city(order: Optional[dict]) -> str:
    if not order:
        return "unknown"
    return order.get("city", "unknown")


def test_optional_chain():
    assert find_city({"city": "Berlin"}) == "Berlin"
```

## Подводные камни

**Optional.of(null).** NPE сразу, используй `ofNullable`.

**get() без check.** NoSuchElementException вместо ясного assert.

**Optional в JSON.** Сериализаторы могут ломать schema expected.

**isPresent + get.** Анти-паттерн, используй `ifPresent` или `orElse`.

**Stream of Optionals.** `flatMap(Optional::stream)` (Java 9+) вместо filter+get.
