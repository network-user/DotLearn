# Иерархия Java Collections Framework

## Суть

Java Collections Framework строится от `Iterable` к `Collection` с ветками `List`, `Set`, `Queue`. `Map` стоит отдельно (не extends Collection), хранит пары key-value. Реализации: `ArrayList`, `HashSet`, `PriorityQueue`, `HashMap` и др. Для AQA важно выбирать структуру под assert: порядок, уникальность, lookup по ключу fixture id.

## Что ответить на собесе

- List допускает дубликаты и index access; Set уникален; Map для id → entity.
- `ArrayDeque` предпочитаю `Stack` (legacy) для очереди задач в runner.
- `LinkedHashMap` сохраняет insertion order в snapshot diff.
- Generics `List<String>` ловят type error на compile, не в runtime assert.
- Честно: synchronized `Vector`/`Hashtable` не нужны, если нет shared mutable state между threads test.

## Как это работает

```
List<String> cases = new ArrayList<>();
Set<String> tags = new HashSet<>();
Map<String, Integer> retries = new HashMap<>();
Queue<String> queue = new ArrayDeque<>();
```

Пример assert helper:

```
Map<Long, TestCase> byId = results.stream()
    .collect(Collectors.toMap(TestCase::id, Function.identity()));
```

Python mental map:

```python
HIERARCHY = {
    "List": "ordered, duplicates",
    "Set": "unique, no index contract",
    "Map": "key-value, separate interface",
}


def test_collections_hierarchy():
    assert "duplicates" in HIERARCHY["List"]
    assert HIERARCHY["Map"].startswith("key")
```

## Подводные камни

**Raw types в legacy test util.** `(List) obj` без generic теряет safety.

**HashSet order assert.** Порядок не guaranteed, сравниваю как Set.

**Map duplicate key collect.** Stream toMap без merge падает.

**SubList view mutation.** Меняет parent list, flaky snapshot.

**Queue poll vs remove.** empty queue: null vs exception.
