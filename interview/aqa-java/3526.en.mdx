# Autoboxing and Integer cache

## In short

Autoboxing wraps a primitive in a wrapper (`int` → `Integer`); unboxing reverses it for arithmetic. The JVM caches `Integer` from -128 to 127, so `==` on boxed values in that range may match by reference; outside the range compare only with `equals`. In asserts on counters and retry limits I unbox explicitly or compare primitives.

## How to answer in an interview

- `List<Integer>` requires boxing on `add(5)`.
- `Integer a = 127; Integer b = 127; a == b` may be true from the cache.
- `Integer c = 128; Integer d = 128; c == d` is false, different objects.
- NPE when unboxing a null `Integer` into `int total`.
- Honestly: autoboxing in a hot loop creates extra GC; rarely critical in tests.

## How it works

Boxing in a collection:

```
List<Integer> attempts = new ArrayList<>();
int raw = 3;
attempts.add(raw);
int back = attempts.get(0);
```

`==` trap:

```
Integer inside = 100;
Integer also = 100;
Integer outside = 200;
Integer another = 200;
boolean sameCache = inside == also;
boolean diffHeap = outside == another;
```

Python mirror:

```python
attempts: list[int] = []
raw = 3
attempts.append(raw)
assert attempts[0] == raw
```

## Pitfalls

**Null unboxing.** `Integer n = null; int x = n` throws NPE.

**== in assertEquals autounbox.** Hamcrest sometimes compares not what you expected.

**Mixed int/Integer overload.** The compiler picks another overload unexpectedly.

**Stream mapToInt.** Forgotten unboxing pulls extra objects.

**valueOf vs new Integer.** `new Integer(5)` bypasses the cache; deprecated.
