# Optional: when it fits in AQA

## In short

`Optional<T>` explicitly models «value may be absent» in a method return type. It is not meant for fields, parameters, or collections. In tests it helps safely read nullable JSON DTO fields and use `map`/`orElseThrow` chains instead of nested null checks.

## How to answer in an interview

- Return `Optional<User>` from a repository helper, not `null`.
- `orElseThrow` with a clear message for fail fast.
- `ifPresent` for soft assert side effects.
- Do not wrap every String in Optional inside a POJO.
- Honestly: Optional on a page object field adds complexity without benefit.

## How it works

```
Optional<Order> findById(String id) {
  return repo.load(id);
}

Order order = findById("42")
  .orElseThrow(() -> new AssertionError("order missing"));
```

Chain:

```
String city = findById("42")
  .map(Order::address)
  .map(Address::city)
  .orElse("unknown");
```

Python mirror:

```python
from typing import Optional


def find_city(order: Optional[dict]) -> str:
    if not order:
        return "unknown"
    return order.get("city", "unknown")


def test_optional_chain():
    assert find_city({"city": "Berlin"}) == "Berlin"
```

## Pitfalls

**Optional.of(null).** Immediate NPE; use `ofNullable`.

**get() without check.** NoSuchElementException instead of a clear assert.

**Optional in JSON.** Serializers may break expected schema.

**isPresent + get.** Anti-pattern; use `ifPresent` or `orElse`.

**Stream of Optionals.** `flatMap(Optional::stream)` (Java 9+) instead of filter+get.
