# Stream vs Collection: differences and reuse

## In short

A `Collection` stores elements in memory and supports mutation and repeated traversal. A `Stream` is a one-shot lazy pipeline over a source (collection, array, generator) and does not store data. After a terminal operation the stream is closed; another terminal throws `IllegalStateException`.

## How to answer in an interview

- The collection lives for the whole test; I create a new stream from it for each assert pass.
- Mutating a collection during an active stream causes ConcurrentModification.
- Infinite streams only with limit/skip in pagination logic tests.
- I keep a materialized expected `List`; I use a stream for transforms.
- Honestly: stream reuse is impossible by design; that is not a bug.

## How it works

```
List<Integer> ids = new ArrayList<>(List.of(1, 2, 3));
ids.add(4);

long count = ids.stream().filter(v -> v > 1).count();

Stream<Integer> once = ids.stream();
once.forEach(System.out::println);
// once.count(); IllegalStateException
```

New pipeline:

```
long again = ids.stream().filter(v -> v > 1).count();
```

Python mirror:

```python
def count_gt(values: list[int], bound: int) -> int:
    return len([v for v in values if v > bound])


def test_stream_like_count():
    assert count_gt([1, 2, 3], 1) == 2
```

## Pitfalls

**Closed stream.** Storing a Stream in a page object field is wrong.

**parallel + ordered assert.** forEach order is not guaranteed.

**Modify source.** Filtering a live DB result list while appending.

**Primitive stream close.** Also one-shot.

**Collection.removeIf vs stream filter.** removeIf mutates; stream filter does not.
