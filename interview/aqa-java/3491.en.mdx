# HashMap internals and collisions

## In short

`HashMap` stores key-value pairs in an array of buckets; bucket index comes from the key's `hashCode`. On collision, entries chain in the bucket (from Java 8, chains of length ≥8 become a red-black tree). Average get/put is O(1); iteration order is not guaranteed.

## How to answer in an interview

- Keys need stable `equals`/`hashCode`; for DTOs in tests use Lombok `@EqualsAndHashCode` or explicit implementations.
- `initialCapacity` and `loadFactor` reduce resize churn for large fixtures.
- `ConcurrentHashMap` instead of `Collections.synchronizedMap` for parallel runners.
- Do not mutate a key after put or the entry becomes unreachable.
- Honestly: worst case is O(n) with a poor hash function or adversarial keys.

## How it works

```
Map<String, Integer> inventory = new HashMap<>();
inventory.put("widget", 10);
inventory.put("gadget", 3);

int qty = inventory.get("widget");
inventory.remove("gadget");
boolean exists = inventory.containsKey("widget");
```

Python mirror:

```python
def bucket_index(key: str, size: int) -> int:
    return hash(key) % size


def test_hash_bucket_placement():
    keys = ["alpha", "beta", "alpha"]
    assert bucket_index(keys[0], 16) == bucket_index(keys[2], 16)
```

## Pitfalls

**Mutable key.** Changing a key field after insert breaks lookup.

**null key.** One null key is allowed; in test maps this is often an NPE trap.

**hashCode on id only.** Equal objects with different hashes look distinct to the map.

**iteration order.** Do not assert key order unless you use `LinkedHashMap`.

**resize rehash.** Bulk put triggers rebuild; factor it into latency-sensitive load tests.
