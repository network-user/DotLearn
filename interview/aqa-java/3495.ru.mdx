# Когда выбирать LinkedList вместо ArrayList

## Суть

`LinkedList` реализует двусвязный список узлов: вставка/удаление в известной позиции итератора O(1), но доступ по индексу O(n). Подходит для очередей, стеков и сценариев с частыми операциями в начале/середине при редком random access. В типичных тестовых коллекциях чаще выигрывает `ArrayList`.

## Что ответить на собесе

- `Deque` задачи: `addFirst`/`pollLast` для буфера событий в integration тесте.
- Не использую для параметризованных данных с частым `get(i)`.
- Реализует `Queue` и `Deque`, удобен как work-stealing буфер в custom runner.
- Память: каждый элемент несёт два pointer overhead vs compact array.
- Честно: в 95% тестового кода ArrayList проще и быстрее.

## Как это работает

```
Deque<String> queue = new LinkedList<>();
queue.addLast("case-a");
queue.addLast("case-b");
String next = queue.pollFirst();

queue.addFirst("priority");
queue.removeLast();
```

Итераторная вставка без сдвига массива:

```
ListIterator<String> it = queue.listIterator();
while (it.hasNext()) {
  if ("skip".equals(it.next())) it.remove();
}
```

Python mirror deque:

```python
from collections import deque


def test_linked_queue_behavior():
    q: deque[str] = deque()
    q.append("case-a")
    q.appendleft("priority")
    assert q.popleft() == "priority"
```

## Подводные камни

**get(index).** На LinkedList линейный проход, скрытая деградация в assert loop.

**Не thread-safe.** Параллельный poll/add без синхронизации даёт race.

**Память.** Большие фикстуры списков раздувают heap vs ArrayList.

**subList.** Дороже и реже нужен, чем у array-backed списков.

**Выбор по названию.** «Нужна быстрая вставка» без профиля часто ошибочно ведёт к LinkedList.
