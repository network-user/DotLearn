# Autoboxing и кэш Integer

## Суть

Autoboxing автоматически оборачивает примитив в wrapper (`int` → `Integer`), unboxing делает обратно при арифметике. JVM кэширует `Integer` от -128 до 127, поэтому `==` на boxed в этом диапазоне может совпасть по ссылке, вне диапазона сравнивать только `equals`. В assert счётчиков и retry limit всегда unbox явно или сравниваю примитивы.

## Что ответить на собесе

- `List<Integer>` требует boxing при `add(5)`.
- `Integer a = 127; Integer b = 127; a == b` может быть true из кэша.
- `Integer c = 128; Integer d = 128; c == d` false, разные объекты.
- NPE при unboxing null `Integer` в `int total`.
- Честно: autoboxing в hot loop создаёт лишний GC, в тестах редко критично.

## Как это работает

Boxing в коллекции:

```
List<Integer> attempts = new ArrayList<>();
int raw = 3;
attempts.add(raw);
int back = attempts.get(0);
```

Ловушка `==`:

```
Integer inside = 100;
Integer also = 100;
Integer outside = 200;
Integer another = 200;
boolean sameCache = inside == also;
boolean diffHeap = outside == another;
```

Python mirror:

```python
attempts: list[int] = []
raw = 3
attempts.append(raw)
assert attempts[0] == raw
```

## Подводные камни

**Null unboxing.** `Integer n = null; int x = n` даёт NPE.

**== в assertEquals autounbox.** Hamcrest иногда сравнивает не то, что ожидали.

**Mixed int/Integer overload.** Компилятор выбирает другую перегрузку неожиданно.

**Stream mapToInt.** Забытый unboxing тянет лишние объекты.

**valueOf vs new Integer.** `new Integer(5)` обходит кэш, deprecated.
