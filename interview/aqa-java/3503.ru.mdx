# equals/hashCode для ключей HashMap

## Суть

`HashMap` ищет ключ сначала по `hashCode` (бакет), затем по `equals` внутри цепочки. Контракт: равные по `equals` объекты обязаны иметь одинаковый `hashCode`. Без согласованной пары методов lookup, remove и dedup в тестовых map с DTO ключами работают непредсказуемо.

## Что ответить на собесе

- Для test data key class переопределяю оба метода или Lombok `@EqualsAndHashCode`.
- В `equals` проверяю type, null-safe поля, не использую только mutable поле без copy.
- `hashCode` комбинирую поля (31 * result + fieldHash) стабильно.
- В assert `map.get(new Key(id))` должен находить entry, положенную с другим instance.
- Честно: если ключ mutable, менять поля после put нельзя, иначе entry потеряется.

## Как это работает

```
class CaseKey {
  private final String suite;
  private final int id;

  CaseKey(String suite, int id) {
    this.suite = suite;
    this.id = id;
  }

  @Override
  public boolean equals(Object other) {
    if (this == other) return true;
    if (!(other instanceof CaseKey)) return false;
    CaseKey peer = (CaseKey) other;
    return id == peer.id && suite.equals(peer.suite);
  }

  @Override
  public int hashCode() {
    return 31 * suite.hashCode() + id;
  }
}
```

Python mirror:

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class CaseKey:
    suite: str
    case_id: int


def test_map_key_contract():
    store = {CaseKey("smoke", 1): "ok"}
    assert store[CaseKey("smoke", 1)] == "ok"
```

## Подводные камни

**Только equals без hashCode.** Классический баг: put находит, get нет.

**Наследование equals.** Liskov ломается, если subclass добавляет поле без учёта в super.

**Identity vs equality.** `==` на ключах в map assert почти всегда ошибка.

**Enum/String keys.** Уже корректны, не переопределяю зря.

**Weak/Soft keys.** Спец-кейсы кэшей, не путать с обычным HashMap в тестах.
