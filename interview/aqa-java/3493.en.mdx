# Maven lifecycle phases in CI

## In short

Maven drives builds through lifecycles: `clean`, `default` (build), and `site`. Phases inside a cycle run in strict order: calling `mvn test` also runs validate and compile. In CI pipelines phases map to compile, unit/integration test, package, and deploy artifacts.

## How to answer in an interview

- `mvn -q test` on every PR; `verify` when failsafe and quality gates are wired.
- `clean` before release builds, not required on every commit.
- Plugin goals bind to phases in `pom.xml`; do not confuse goal and phase.
- `install` puts the jar in `~/.m2`, needed for local multi-module dependencies.
- Honestly: `deploy` in CI only with credentials and a staging repo, not in dev jobs.

## How it works

Default lifecycle (short):

```
validate → compile → test → package → verify → install → deploy
```

Clean:

```
pre-clean → clean → post-clean
```

Pipeline command example:

```
mvn -B -DskipTests=false clean verify
```

Python mirror for stage orchestration:

```python
DEFAULT_PHASES = ["validate", "compile", "test", "package", "verify"]


def run_through(phase: str) -> list[str]:
    stop = DEFAULT_PHASES.index(phase)
    return DEFAULT_PHASES[: stop + 1]


def test_maven_phase_order():
    assert run_through("test") == ["validate", "compile", "test"]
```

## Pitfalls

**Phase vs goal.** `mvn compile` is a phase; `compiler:compile` is a goal; order differs on a direct goal call.

**skipTests.** Skips surefire but still compiles test sources.

**Reactor order.** Multi-module: upstream must `install` before downstream tests.

**CI cache.** Without `.m2` cache every phase re-downloads dependencies.

**site lifecycle.** Documentation is separate; do not mix it with the test gate.
