# Практика абстрактных классов

## Суть

Абстрактный класс нельзя инстанцировать напрямую: он задаёт общий каркас с частично реализованными методами и обязательными abstract hooks. Подкласс наследует shared state и поведение, дописывая специфику. В test framework это типичный base page / base test с общим setup и абстрактным шагом сценария.

## Что ответить на собесе

- `abstract class BaseTest` с `@BeforeEach` и abstract `void runScenario()`.
- Общий `openApp()` реализован в base, конкретный flow в наследнике.
- Нельзя `new BaseTest()`, только конкретный подкласс.
- Если нужен только контракт без state, предпочту interface.
- Честно: глубокая иерархия abstract base усложняет parallel и Mockito spy.

## Как это работает

Базовый шаблон сценария:

```
abstract class BaseUiTest {
  WebDriver driver;

  void setUp() { driver = createDriver(); }

  abstract void exerciseFlow();

  void tearDown() { driver.quit(); }
}

class CheckoutTest extends BaseUiTest {
  void exerciseFlow() {
    driver.get("/cart");
    driver.findElement(By.id("pay")).click();
  }
}
```

Python mirror:

```python
from abc import ABC, abstractmethod


class BaseUiTest(ABC):
    def setup(self) -> None:
        self.driver = create_driver()

    @abstractmethod
    def exercise_flow(self) -> None:
        raise NotImplementedError

    def teardown(self) -> None:
        self.driver.quit()
```

## Подводные камни

**Abstract без подклассов.** Класс с abstract методом обязан быть abstract.

**Constructor chain.** Super constructor вызывается до полей child, NPE в setup.

**Template method rigidity.** Жёсткий порядок шагов ломает reuse в data-driven тестах.

**Shared mutable driver field.** Flaky при `@TestInstance(PER_CLASS)`.

**Смешение abstract + final.** final abstract метод бессмысленен и не компилируется.
