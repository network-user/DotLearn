# Time complexity of ArrayList add

## In short

Tail `add` on ArrayList is amortized O(1): the element goes into the internal array at the size index. When capacity is exhausted a new array (~1.5x) is allocated and all elements are copied: O(n) per resize. `add(index, e)` in the middle shifts the tail: O(n). For bulk fixture inserts in perf tests I account for resize spikes.

## How to answer in an interview

- Amortized O(1) for append, not worst-case on every call.
- I call `ensureCapacity` upfront when I know N rows.
- Middle insert on a large list is costly; prefer addAll at the end.
- Microbench with JMH, not a single assert on elapsed time.
- Honestly: resize on ten elements is irrelevant; a million rows without presize hurts.

## How it works

```
List<Integer> buf = new ArrayList<>(1024);
for (int i = 0; i < 10_000; i++) {
  buf.add(i);
}
```

Resize concept:

```
old [a,b,c,_,_] -> new capacity 6, copy 3 elements, add d
```

Python list growth analogue:

```python
def append_amortized(items: list[int], n: int) -> None:
    for i in range(n):
        items.append(i)


def test_growth():
    data: list[int] = []
    append_amortized(data, 1000)
    assert len(data) == 1000
    assert data[-1] == 999
```

## Pitfalls

**addAll triggers one resize.** Better than a loop add without capacity hint.

**SubList add.** Updates parent structural mod counter.

**Concurrent add.** Not thread-safe without synchronization.

**No primitive list.** Autoboxing Integer overhead in million loops.

**remove from end O(1).** Middle remove is also O(n), symmetric to insert.
