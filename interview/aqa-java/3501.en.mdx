# OOP principles with examples for tests

## In short

Encapsulation hides state behind an API, inheritance reuses base behavior, polymorphism swaps implementations through a common type, and abstraction defines a contract without details. Together they structure page objects, API clients, and fixtures so scenarios read as business steps, not field access.

## How to answer in an interview

- Encapsulation: private locators, public `submitOrder()`.
- Inheritance: `AuthenticatedPage` extends `BasePage`.
- Polymorphism: real vs WireMock `PaymentClient` under one type.
- Abstraction: `abstract void open()` on a base page.
- Honestly: if a subclass only adds one method, composition is often cleaner.

## How it works

Encapsulation:

```
class CartPage {
  private final By total = By.cssSelector("[data-test=total]");

  Money readTotal(WebDriver driver) {
    String raw = driver.findElement(total).getText();
    return Money.parse(raw);
  }
}
```

Polymorphism plus abstraction:

```
interface SmsGateway { void send(String phone, String code); }

class FakeSmsGateway implements SmsGateway {
  public void send(String phone, String code) { /* store */ }
}

void verifyOtp(SmsGateway gateway) {
  gateway.send("+1", "1234");
}
```

Python mirror:

```python
class CartPage:
    def __init__(self) -> None:
        self._total_selector = "[data-test=total]"

    def read_total(self, driver) -> str:
        return driver.find_element(self._total_selector).text
```

## Pitfalls

**Anemic model.** PO with only getters and no behavior; OOP in name only.

**Inheritance for field reuse.** Shared protected locators are fragile.

**Polymorphism without contract tests.** New implementation compiles, fails in CI.

**Abstract class with state.** A base holding WebDriver complicates parallel runs.

**Breaking encapsulation in tests.** Reflection into private fields is an anti-pattern.
