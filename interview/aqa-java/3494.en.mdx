# Four OOP pillars in Java tests

## In short

OOP in Java rests on encapsulation, inheritance, polymorphism, and abstraction. In automation that shows up as hidden page object details, base fixture classes, swappable client interfaces, and abstract scenario steps. The four principles keep test code extensible without copy-paste.

## How to answer in an interview

- Encapsulation: private locators, public action methods (`clickCheckout()`).
- Inheritance: `BasePage` with wait helpers; pages extend it.
- Polymorphism: `AuthProvider` mock/real in one test via an interface.
- Abstraction: `StoragePort` interface hides JDBC details.
- Honestly: page object inheritance deeper than two levels usually hurts; prefer composition.

## How it works

Encapsulation:

```
class WalletPage {
  private By balance = By.id("balance");

  int readBalance(WebDriver driver) {
    return Integer.parseInt(driver.findElement(balance).getText());
  }
}
```

Abstraction plus polymorphism:

```
interface Clock { long now(); }

class SystemClock implements Clock {
  public long now() { return System.currentTimeMillis(); }
}

void assertFresh(long ts, Clock clock) {
  assertTrue(clock.now() - ts < 5000);
}
```

Python mirror:

```python
class WalletPage:
    def __init__(self, driver) -> None:
        self._balance_locator = "balance"

    def read_balance(self, driver) -> int:
        return int(driver.find_element(self._balance_locator).text)
```

## Pitfalls

**God page object.** SRP violation; encapsulation becomes a method dump.

**Fragile base class.** Changing `BaseTest` breaks all subclasses.

**Polymorphism via instanceof.** Anti-pattern instead of an interface.

**Abstract class vs interface.** Multiple behaviors only through interfaces.

**Testing private.** Over-encapsulation with no seam for mocks.
