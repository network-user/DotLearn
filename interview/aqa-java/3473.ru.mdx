# Stream API: цепочка filter-map-reduce в тестах

## Суть

Stream API обрабатывает последовательность элементов декларативной цепочкой: промежуточные операции (`filter`, `map`, `sorted`) ленивы, терминальная (`collect`, `reduce`, `forEach`) запускает pipeline. Источником служит Collection, массив или `Stream.of`. Для AQA это способ агрегировать результаты параллельного прогона кейсов или фильтровать ответ API без императивных циклов в assertion helper.

## Что ответить на собесе

- Создаю stream через `list.stream()`, не мутирую исходную коллекцию.
- Промежуточные шаги описываю как pipeline, терминал один.
- Для parallel stream проверяю thread-safety данных и детерминизм assert.
- В тестах сервиса сравниваю `collect(toList())` с expected fixture.
- Честно: stream over tiny list (3 элемента) не даёт выигрыша, читаемость важнее.

## Как это работает

```
List<Integer> scores = List.of(3, 8, 12, 15, 4);

int doubledEvens = scores.stream()
    .filter(n -> n % 2 == 0)
    .map(n -> n * 2)
    .reduce(0, Integer::sum);
```

Сбор в список:

```
List<String> tags = results.stream()
    .filter(r -> r.status().equals("FAIL"))
    .map(TestResult::caseName)
    .sorted()
    .collect(Collectors.toList());
```

Python analogue для assertion helper:

```python
def sum_doubled_evens(values: list[int]) -> int:
    return sum(n * 2 for n in values if n % 2 == 0)


def test_stream_like_pipeline():
    assert sum_doubled_evens([3, 8, 12, 15, 4]) == 40
```

## Подводные камни

**Side effect в forEach.** Мутировать внешний список из stream ломает parallel.

**Reuse stream.** После terminal stream consumed, второй collect упадёт.

**Null в stream.** NPE в map, фильтрую `Objects.nonNull` заранее.

**boxed vs primitive.** `mapToInt` быстрее, чем `Integer` autobox flood.

**Comparator unstable sort.** equal elements меняют порядок между JDK runs.
