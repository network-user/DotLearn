# int и Integer в тестовых данных

## Суть

`int` примитив: значение по умолчанию 0, не null, быстрый arithmetic. `Integer` wrapper: объект в heap, nullable, нужен для generics (`List<Integer>`). Autoboxing/unboxing связывает типы, но добавляет NPE риск на null.

## Что ответить на собесе

- Счётчики и id в assert: `int`/`long`.
- Nullable retry count в DTO: `Integer`.
- `Integer.valueOf(127)` кэшируется, identity assert на малых числах опасен.
- Stream `mapToInt` избегает boxing в больших наборах.
- Честно: смешивать `int` и `Integer` в equals без null guard ломает тест.

## Как это работает

```
int passed = 12;
Integer nullable = null;

List<Integer> ids = new ArrayList<>();
ids.add(42);

int unboxed = ids.get(0);
Integer boxed = Integer.valueOf(unboxed);
```

Autoboxing trap:

```
Integer a = null;
// int x = a; // NPE при unboxing
```

Python mirror (no primitive split):

```python
def total(ids: list[int | None]) -> int:
    return sum(i for i in ids if i is not None)


def test_nullable_filter():
    assert total([1, None, 2]) == 3
```

## Подводные камни

**== на Integer >127.** Разные объекты при equals true.

**parseInt vs valueOf.** parseInt для String, valueOf для int cache.

**null в JSONArray.** Десериализация в int поле без default.

**AtomicInteger vs Integer.** Для concurrent counter нужен atomic, не wrapper.

**Optional\<Integer>.** Допустимо в return, не в поле entity.
