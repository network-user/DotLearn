# int vs Integer in test data

## In short

`int` is a primitive: default 0, not null, fast arithmetic. `Integer` is a wrapper: heap object, nullable, required for generics (`List<Integer>`). Autoboxing/unboxing bridges them but adds NPE risk on null.

## How to answer in an interview

- Counters and ids in asserts: `int`/`long`.
- Nullable retry count in a DTO: `Integer`.
- `Integer.valueOf(127)` is cached; identity asserts on small numbers are risky.
- Stream `mapToInt` avoids boxing on large sets.
- Honestly: mixing `int` and `Integer` in equals without a null guard breaks the test.

## How it works

```
int passed = 12;
Integer nullable = null;

List<Integer> ids = new ArrayList<>();
ids.add(42);

int unboxed = ids.get(0);
Integer boxed = Integer.valueOf(unboxed);
```

Autoboxing trap:

```
Integer a = null;
// int x = a; // NPE on unboxing
```

Python mirror (no primitive split):

```python
def total(ids: list[int | None]) -> int:
    return sum(i for i in ids if i is not None)


def test_nullable_filter():
    assert total([1, None, 2]) == 3
```

## Pitfalls

**== on Integer above 127.** Different objects while equals is true.

**parseInt vs valueOf.** parseInt for String; valueOf for int cache.

**null in JSONArray.** Deserializing into an int field without default.

**AtomicInteger vs Integer.** Concurrent counters need atomic, not wrapper.

**Optional<Integer>.** Fine in returns, not on entity fields.
