# Logs as the source of truth when tests fail

## In short

Logs record application and infrastructure events with timestamp, level, and context. For AQA they link a failed assert to request-id, stack trace, and environment state, especially in CI and nightly runs.

## How to answer in an interview

- I define log levels: DEBUG for development, INFO for business events, WARNING/ERROR for degradation and failures.
- I explain structured logging (JSON) vs plain text and why JSON is easier to filter in ELK/Loki.
- I describe attaching logs to Allure/JUnit: correlation-id from a header, container log tail after failure.
- I mention the observability triad: logs, metrics, traces; logs complement but do not replace latency metrics.
- I add honestly: without retention policy and PII masking logs are risky; I do not enable DEBUG on prod-like data without redaction.

## How it works

The app writes through the logging API; handlers ship to stdout, a file, or an agent. In pytest you can capture logging and assert on messages; on integration failure collect docker logs by service label.

```python
import logging
import json
import pytest


class JsonFormatter(logging.Formatter):
    def format(self, record: logging.LogRecord) -> str:
        payload = {
            "level": record.levelname,
            "message": record.getMessage(),
            "logger": record.name,
            "request_id": getattr(record, "request_id", None),
        }
        return json.dumps(payload, ensure_ascii=False)


def configure_logging():
    handler = logging.StreamHandler()
    handler.setFormatter(JsonFormatter())
    root = logging.getLogger()
    root.handlers.clear()
    root.addHandler(handler)
    root.setLevel(logging.INFO)


def process_payment(order_id: str, request_id: str) -> None:
    log = logging.getLogger("billing")
    extra = {"request_id": request_id}
    log.info("payment started", extra=extra)
    if order_id.endswith("FAIL"):
        log.error("gateway declined", extra=extra)
        raise RuntimeError("payment failed")
    log.info("payment captured", extra=extra)


def test_payment_failure_emits_error_log(caplog):
    configure_logging()
    caplog.set_level(logging.ERROR)
    with pytest.raises(RuntimeError):
        process_payment("ORD-FAIL", request_id="req-42")
    assert any("gateway declined" in r.message for r in caplog.records)
    assert caplog.records[-1].request_id == "req-42"
```

In the pipeline I add "on failure: upload logs artifact" and search by `request_id` from the HTTP response header.

## Pitfalls

**Logs without correlation-id.** You cannot tie UI, API, and worker in one scenario.

**Too verbose DEBUG in CI.** Floods storage and hides signal.

**Secrets in logs.** Password/token in the request body lands in artifacts.

**Timezone without UTC.** Hard to align app log and ingress timestamps.

**Assert only on exit code.** The service fails silently without ERROR level.
