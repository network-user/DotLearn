# Manager module: object factory and shared algorithms

## In short

The manager module belongs to the metadata type, not an instance. It holds exported functions for creating documents, queries, print, and checks used from forms, exchange, and scheduled jobs.

## How to answer in an interview

- I call `Документы.Реализация.СоздатьДокумент()` via the manager, not manual `Новый`.
- Shared queries for the metadata type live in the manager so forms do not duplicate them.
- Exported manager procedures are available as `Справочники.Номенклатура.МояФункция()`.
- I separate code without `ЭтотОбъект` from the object module.
- I am honest: the manager does not replace a common module for cross-metadata logic across several types.

## How it works

Active items query in the catalog manager:

```bsl
Функция АктивныеЭлементы() Экспорт
    Запрос = Новый Запрос;
    Запрос.Текст =
        "ВЫБРАТЬ
        |    Н.Ссылка
        |ИЗ
        |    Справочник.Номенклатура КАК Н
        |ГДЕ
        |    Н.Активна
        |    И НЕ Н.ПометкаУдаления";
    Возврат Запрос.Выполнить().Выгрузить().ВыгрузитьКолонку("Ссылка");
КонецФункции
```

Create and fill a document via the manager API:

```bsl
НовыйДокумент = Документы.ЗаказКлиента.СоздатьДокумент();
НовыйДокумент.Дата = ТекущаяДатаСеанса();
Документы.ЗаказКлиента.ЗаполнитьПоШаблону(НовыйДокумент, ПараметрыЗаполнения);
```

The manager is called from forms, other managers, and server common modules.

## Pitfalls

**Business logic only on the form.** Duplication on programmatic create from exchange.
**Manager as a dump for everything.** Unrelated algorithms mixed in one file.
**Heavy work without cache.** Full selections on every form call.
**Client calls.** Exported manager functions must run on the server.
**Confusion with set module.** For a register the manager and record set module have different roles.
