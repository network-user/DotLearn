# Модуль менеджера: фабрика объектов и общие алгоритмы

## Суть

Модуль менеджера принадлежит типу метаданных, а не экземпляру. В нём размещают экспортные функции создания документов, выборок, печати, проверок перед использованием в формах, обмене и регламентных заданиях.

## Что ответить на собесе

- Я вызываю `Документы.Реализация.СоздатьДокумент()` через менеджер, не `Новый` вручную.
- Общие запросы по типу метаданных держу в менеджере, чтобы не копировать в каждую форму.
- Экспортные процедуры менеджера доступны как `Справочники.Номенклатура.МояФункция()`.
- Отделяю код без `ЭтотОбъект` от модуля объекта.
- Честно: менеджер не заменяет общий модуль для кросс-объектной логики нескольких типов метаданных.

## Как это работает

Выборка активной номенклатуры в менеджере справочника:

```bsl
Функция АктивныеЭлементы() Экспорт
    Запрос = Новый Запрос;
    Запрос.Текст =
        "ВЫБРАТЬ
        |    Н.Ссылка
        |ИЗ
        |    Справочник.Номенклатура КАК Н
        |ГДЕ
        |    Н.Активна
        |    И НЕ Н.ПометкаУдаления";
    Возврат Запрос.Выполнить().Выгрузить().ВыгрузитьКолонку("Ссылка");
КонецФункции
```

Создание и заполнение документа по API менеджера:

```bsl
НовыйДокумент = Документы.ЗаказКлиента.СоздатьДокумент();
НовыйДокумент.Дата = ТекущаяДатаСеанса();
Документы.ЗаказКлиента.ЗаполнитьПоШаблону(НовыйДокумент, ПараметрыЗаполнения);
```

Менеджер вызывают из форм, других менеджеров и серверных общих модулей.

## Подводные камни

**Бизнес-логика только в форме.** Дублирование при программном создании из обмена.
**Менеджер как общий модуль всего.** Смешение несвязанных алгоритмов в одном файле.
**Тяжёлые операции без кэша.** Повторные полные выборки на каждый вызов формы.
**Обращение с клиента.** Экспортные функции менеджера должны вызываться на сервере.
**Путаница с модулем набора.** У регистра менеджер и модуль набора записей разные роли.
