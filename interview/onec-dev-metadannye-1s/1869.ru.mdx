# Как платформа читает и кэширует константы

## Суть

Значения констант лежат в единой таблице СУБД, а при обращении платформа поднимает их в кэш сеанса. API `Константы.<Имя>.Получить()` читает из кэша; `Установить()` пишет в базу и обновляет кэш текущего сеанса.

## Что ответить на собесе

- Я объясняю: одна физическая таблица, много логических констант по метаданным.
- После `Установить()` другие сеансы узнают новое значение при следующем `Получить()`.
- Для массового чтения настроек не делаю десятки `Получить()` в цикле без нужды.
- Разделение: константа для глобального флага, регистр для параметров с периодом.
- Честно: константа не транзакционно версионируется; историю изменений так не восстановить.

## Как это работает

Типичный сценарий в общем модуле:

```bsl
Функция ИспользуетсяУчетПоСериям() Экспорт
    Возврат Константы.УчетПоСериям.Получить();
КонецФункции

Процедура ВключитьУчетПоСериям() Экспорт
    УстановитьПривилегированныйРежим(Истина);
    Константы.УчетПоСериям.Установить(Истина);
КонецПроцедуры
```

При старте фонового задания кэш пуст; первый `Получить()` идёт в СУБД. В интерактивном сеансе повторные вызовы дешевле. Изменение константы через форму «Константы» или программно сразу влияет на новые расчёты в том же сеансе.

## Подводные камни

**Устаревший кэш в долгом сеансе.** Регламентное задание не видит смену ставки без повторного чтения.
**Константа как журнал.** Нет среза «кто и когда менял» без отдельного аудита.
**Частые записи.** `Установить()` в цикле нагружает СУБД и блокировки.
**Тип не совпал.** После смены типа в метаданных старое значение может не читаться.
**Прямой SQL.** Обход платформы ломает кэш и переносимость между СУБД.
