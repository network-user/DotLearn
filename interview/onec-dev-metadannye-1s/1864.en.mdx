# Overriding standard behavior through an extension

## In short

A standard configuration ships default posting, write, and form handlers. In an extension you borrow the object and add procedures with `&После` or `&Вместо` without touching the vendor. That is how you customize ERP while staying on vendor support.

## How to answer in an interview

- I separate editing the standard configuration from borrowing in an extension.
- For posting changes I use `&После("ОбработкаПроведения")` or `&Вместо` with a call to standard code.
- I check compatibility on update: renamed procedures break the extension.
- I put business checks in server handlers, not only on the form.
- I am honest: `&Вместо` without calling base logic disables the standard mechanism and is risky on updates.

## How it works

In the extension I borrow the document and add a handler in the object module:

```bsl
&После("ОбработкаПроведения")
Процедура Расш1_ОбработкаПроведения(Отказ, РежимПроведения)
    Движения.СебестоимостьТоваров.Записывать = Истина;
    НаборЗаписей = РегистрыНакопления.СебестоимостьТоваров.СоздатьНаборЗаписей();
    НаборЗаписей.Отбор.Регистратор.Установить(Ссылка);
    НаборЗаписей.Прочитать();
    Если НаборЗаписей.Количество() = 0 Тогда
        Отказ = Истина;
        ВызватьИсключение "Не сформированы движения себестоимости";
    КонецЕсли;
КонецПроцедуры
```

For a form I borrow the managed form and extend `ПриСозданииНаСервере` with `&После`. The platform runs standard code first, then the extension.

## Pitfalls

**Full replacement without calling standard code.** `&Вместо` without `ПродолжитьВызов` breaks standard posting.
**Edits in the main configuration.** Takes the configuration off vendor support and complicates merge.
**Conflict between extensions.** Connection order affects the result of `&После`.
**Borrowing without update testing.** After a vendor release the extension may fail to build.
**Logic only on the client.** Checks are bypassed on programmatic server write.
