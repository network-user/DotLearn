# Переопределение типового поведения через расширение

## Суть

Типовая конфигурация поставляет стандартные обработчики проведения, записи и форм. В расширении объект заимствуют и добавляют свои процедуры с директивой `&После` или `&Вместо`, не трогая поставщика. Так дорабатывают ERP без снятия с поддержки.

## Что ответить на собесе

- Я различаю изменение типовой конфигурации и заимствование в расширении.
- Для доработки проведения использую `&После("ОбработкаПроведения")` или `&Вместо` с вызовом типового кода.
- Проверяю совместимость при обновлении: переименованные процедуры ломают расширение.
- Вношу бизнес-проверки в серверные обработчики, а не только в форму.
- Честно: `&Вместо` без вызова базовой логики отключает типовой механизм и опасно на обновлениях.

## Как это работает

В расширении заимствую документ и в модуле объекта добавляю обработчик:

```bsl
&После("ОбработкаПроведения")
Процедура Расш1_ОбработкаПроведения(Отказ, РежимПроведения)
    Движения.СебестоимостьТоваров.Записывать = Истина;
    НаборЗаписей = РегистрыНакопления.СебестоимостьТоваров.СоздатьНаборЗаписей();
    НаборЗаписей.Отбор.Регистратор.Установить(Ссылка);
    НаборЗаписей.Прочитать();
    Если НаборЗаписей.Количество() = 0 Тогда
        Отказ = Истина;
        ВызватьИсключение "Не сформированы движения себестоимости";
    КонецЕсли;
КонецПроцедуры
```

Для формы заимствую управляемую форму и расширяю `ПриСозданииНаСервере` с `&После`. Платформа сначала выполняет типовой код, затем расширение.

## Подводные камни

**Полная замена без вызова типового.** `&Вместо` без `ПродолжитьВызов` ломает стандартное проведение.
**Правки в основной конфигурации.** Снимает конфигурацию с поддержки и усложняет merge.
**Конфликт нескольких расширений.** Порядок подключения влияет на результат `&После`.
**Заимствование без теста обновления.** После релиза поставщика расширение может не собраться.
**Логика только на клиенте.** Обход проверок при программной записи на сервере.
