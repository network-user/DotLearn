# Auto-filling Responsible for a new document

## In short

For a new object you fill Responsible before write: in `ПередЗаписью` or on server form create. The value comes from the current infobase user or a users/employees catalog linked to the session.

## How to answer in an interview

- I set Responsible in `ПередЗаписью` when the field is empty and `ЭтоНовый()`.
- On the form I mirror it in `ПриСозданииНаСервере` so the user sees the value immediately.
- I use `Пользователи.ТекущийПользователь()` or a standard SSL mechanism, not a legacy `ТекущийПользователь()` without infobase context.
- I do not overwrite Responsible on rewrite of an existing document.
- I am honest: on exchange or scheduled write I substitute a technical user or keep the passed value.

## How it works

In the object module:

```bsl
Процедура ПередЗаписью(Отказ, РежимЗаписи, РежимПроведения)
    Если ЭтоНовый() И НЕ ЗначениеЗаполнено(Ответственный) Тогда
        Ответственный = Пользователи.ТекущийПользователь();
    КонецЕсли;
КонецПроцедуры
```

On the form when opening a new document:

```bsl
&НаСервере
Процедура ПриСозданииНаСервере(Отказ, СтандартнаяОбработка)
    Если НЕ ЗначениеЗаполнено(Объект.Ответственный) Тогда
        Объект.Ответственный = Пользователи.ТекущийПользователь();
    КонецЕсли;
КонецПроцедуры
```

If the attribute references an employees catalog, I resolve the employee from the user via an information register or a standard configuration function.

## Pitfalls

**ПриЗаписи instead of ПередЗаписью.** Too late to affect posting and locks.
**Overwrite on every write.** You lose the original author when copying a document.
**Type mismatch.** User vs Employee attributes need different functions.
**Background session.** The scheduled job user is not the UI initiator.
**Form only.** Programmatic write without the object module leaves the field empty.
