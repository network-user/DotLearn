# Ссылка, объект и UUID в 1С: что когда применять

## Суть

Ссылка (`СправочникСсылка`, `ДокументСсылка`) хранит идентификатор в базе без полной загрузки данных. Объект в памяти получают через `ПолучитьОбъект()` или `СоздатьЭлемент()` и меняют реквизиты перед записью. `УникальныйИдентификатор` (UUID) однозначно идентифицирует ссылку и используется в обмене, внешних ключах и сопоставлении данных.

## Что ответить на собесе

- Я передаю между модулями ссылки, а не объекты: меньше трафика и памяти.
- Для чтения одного поля беру `ОбщегоНазначения.ЗначениеРеквизитаОбъекта`, без полной загрузки.
- UUID нужен при интеграции, когда внешняя система хранит GUID элемента 1С.
- Объект в памяти использую, когда массово меняю реквизиты и табличные части.
- Честно: сравнение ссылок через строковое представление ненадёжно; сравниваю ссылки как значения типа Ссылка.

## Как это работает

Получение ссылки без чтения всех полей:

```bsl
СсылкаНоменклатуры = Справочники.Номенклатура.НайтиПоРеквизиту("Артикул", "SKU-42");
Уид = СсылкаНоменклатуры.УникальныйИдентификатор();
```

Работа с объектом в памяти:

```bsl
ЭлементОбъект = СсылкаНоменклатуры.ПолучитьОбъект();
ЭлементОбъект.Наименование = "Новое имя";
ЭлементОбъект.Записать();
```

Восстановление ссылки по UUID при загрузке из JSON:

```bsl
СсылкаВосстановленная = Справочники.Номенклатура.ПолучитьСсылку(Уид);
```

Ссылка пустая, если `ЗначениеЗаполнено(Ссылка)` ложно; это не то же самое, что пометка удаления.

## Подводные камни

**Передача объекта на клиент.** Лишняя сериализация и риск устаревших данных.
**Запросы по строке ссылки.** Медленно и ломается при смене представления.
**Путаница UUID и кода.** Код справочника не глобально уникален в распределённой базе.
**Работа с удалёнными.** Ссылка может указывать на помеченный элемент; проверяю `ПометкаУдаления`.
**Создание без записи.** Объект в памяти без `Записать()` не существует для других сеансов.
