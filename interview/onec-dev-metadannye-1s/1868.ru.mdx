# Физическое хранение констант в базе 1С

## Суть

Все константы конфигурации хранятся в одной служебной таблице СУБД. Каждая константа метаданных соответствует строке с идентификатором типа и значением в универсальном хранилище. Отдельная физическая таблица на каждую константу не создаётся.

## Что ответить на собесе

- Я отвечаю: одна таблица на все константы, не по таблице на объект метаданных.
- Чтение через менеджер `Константы.<Имя>.Получить()`, запись через `Установить()` в привилегированном коде.
- Значения кэшируются в сеансе; после смены константы в другом сеансе нужен повторный `Получить()`.
- Для периодических настроек константа не подходит; использую регистр сведений.
- Честно: прямой SQL к таблице констант не делаю; только API платформы.

## Как это работает

Чтение и установка значения:

```bsl
ТекущийПроцент = Константы.СтавкаНДС.Получить();

Процедура УстановитьСтавкуНДС(НоваяСтавка) Экспорт
    Константы.СтавкаНДС.Установить(НоваяСтавка);
КонецПроцедуры
```

При обновлении конфигурации платформа синхронизирует состав констант в метаданных со строками таблицы. Тип значения задаётся в свойствах константы; несовместимое значение вызовет ошибку при `Установить()`.

Константы удобны для редко меняющихся флагов (например, ведётся ли учёт по сериям). Частые изменения лучше хранить в регистре с историей.

## Подводные камни

**Ожидание таблицы на константу.** Неверная модель усложняет администрирование СУБД.
**Запись без прав.** `Установить()` в пользовательском режиме может быть запрещён ролями.
**Кэш в сеансе.** Долгоживущий фоновый сеанс видит устаревшую ставку после смены константы.
**Константа вместо регистра.** Теряется история изменений настроек.
**Тяжёлые объекты в константе.** Хранение больших двоичных данных в константе нецелесообразно.
