# Physical storage of constants in a 1C database

## In short

All configuration constants are stored in one service DBMS table. Each metadata constant maps to a row with a type identifier and value in a universal store. A separate physical table per constant is not created.

## How to answer in an interview

- I answer: one table for all constants, not one table per metadata object.
- Read via `Константы.<Имя>.Получить()`, write via `Установить()` in privileged code.
- Values are cached in the session; after another session changes a constant you need a fresh `Получить()`.
- For time-varying settings a constant is wrong; use an information register.
- I am honest: I do not use direct SQL on the constants table; only the platform API.

## How it works

Read and set a value:

```bsl
ТекущийПроцент = Константы.СтавкаНДС.Получить();

Процедура УстановитьСтавкуНДС(НоваяСтавка) Экспорт
    Константы.СтавкаНДС.Установить(НоваяСтавка);
КонецПроцедуры
```

On configuration update the platform syncs the constant set in metadata with table rows. Value type is set in constant properties; an incompatible value raises an error on `Установить()`.

Constants suit rarely changed flags (for example whether lot accounting is enabled). Frequent changes belong in a register with history.

## Pitfalls

**Expecting a table per constant.** Wrong model complicates DBMS administration.
**Write without rights.** `Установить()` may be denied by roles in user mode.
**Session cache.** A long-lived background session sees a stale rate after a constant change.
**Constant instead of a register.** You lose settings change history.
**Heavy objects in a constant.** Storing large binary data in a constant is a poor fit.
