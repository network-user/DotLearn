# Where the initial session parameter value is set

## In short

Session parameters are declared in configuration ("Session parameters" branch); initial values are set in the session module in procedure `УстановкаПараметровСеанса`. The platform calls it at session start before main application logic.

## How to answer in an interview

- I name two places: parameter metadata (name, type) and session module with `УстановкаПараметровСеанса`.
- I show `ПараметрыСеанса.ParameterName = ...` or `SetSessionParameters()` depending on API version.
- I distinguish from `OnStart` in managed application module: UI and warnings there, not primary parameter init.
- I honestly note session parameters are available on client and server within one session, but init follows platform rules on the server.
- I mention SSL: subsystems like "Current session" fill some parameters.

## How it works

In the designer add session parameter `CurrentOrganization` of type `CatalogRef.Organizations`. In the session module:

```bsl
Процедура УстановкаПараметровСеанса(ИменаПараметровСеанса)

    Если ИменаПараметровСеанса = Неопределено Тогда
        ПараметрыСеанса.ТекущаяОрганизация = ОрганизацияПоУмолчанию();
    Иначе
        Для Каждого Имя Из ИменаПараметровСеанса Цикл
            Если Имя = "ТекущаяОрганизация" Тогда
                ПараметрыСеанса.ТекущаяОрганизация = ОрганизацияПоУмолчанию();
            КонецЕсли;
        КонецЦикла;
    КонецЕсли;

КонецПроцедуры

Функция ОрганизацияПоУмолчанию()

    Возврат Справочники.Организации.ОрганизацияПоУмолчанию();

КонецФункции
```

The platform calls the procedure on session creation and on targeted parameter reset. Application code then reads `ПараметрыСеанса.ТекущаяОрганизация` in queries and forms without repeated infobase lookup on every call.

Session parameters do not persist across sessions and do not replace constants: they are session-scoped cache with metadata typing.

## Pitfalls

**Confusion with application module.** Init in `OnStart` may run later or on client without proper context.

**Heavy setup code.** Long queries at session start slow login for all users.

**Undeclared metadata type.** Access to a missing parameter causes runtime errors.

**Privileged mode.** Reading data during setup may need `SetPrivilegedMode` for system settings.

**Background jobs.** Background sessions have their own parameters; interactive values are not copied automatically.
