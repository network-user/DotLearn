# Reference, object, and UUID in 1C: when to use each

## In short

A reference (`СправочникСсылка`, `ДокументСсылка`) stores a database identifier without loading full data. An in-memory object comes from `ПолучитьОбъект()` or `СоздатьЭлемент()`; you change attributes before write. `УникальныйИдентификатор` (UUID) uniquely identifies a reference for exchange, external keys, and data mapping.

## How to answer in an interview

- I pass references between modules, not objects: less traffic and memory.
- For one field I use `ОбщегоНазначения.ЗначениеРеквизитаОбъекта` without full load.
- UUID is needed in integration when an external system stores the 1C item GUID.
- I use an in-memory object when I change many attributes and tabular sections.
- I am honest: comparing references via string representation is unreliable; I compare reference values directly.

## How it works

Getting a reference without reading all fields:

```bsl
СсылкаНоменклатуры = Справочники.Номенклатура.НайтиПоРеквизиту("Артикул", "SKU-42");
Уид = СсылкаНоменклатуры.УникальныйИдентификатор();
```

Working with an in-memory object:

```bsl
ЭлементОбъект = СсылкаНоменклатуры.ПолучитьОбъект();
ЭлементОбъект.Наименование = "Новое имя";
ЭлементОбъект.Записать();
```

Restoring a reference from UUID when loading from JSON:

```bsl
СсылкаВосстановленная = Справочники.Номенклатура.ПолучитьСсылку(Уид);
```

A reference is empty when `ЗначениеЗаполнено(Ссылка)` is false; that is not the same as a deletion mark.

## Pitfalls

**Passing an object to the client.** Extra serialization and stale data risk.
**Queries on reference strings.** Slow and breaks when representation changes.
**Confusing UUID and code.** Catalog code is not globally unique in a distributed base.
**Working with deleted items.** A reference may point to a marked item; I check `ПометкаУдаления`.
**Create without write.** An in-memory object without `Записать()` does not exist for other sessions.
