# Command in 1C: linking a button to a handler

## In short

A command is a named UI action: a button or menu item calls a form module procedure or a standard platform operation. In managed forms the command is declared in the form tree, gets a handler name, and can be global for the command bar.

## How to answer in an interview

- I separate a form command from the metadata Command object in the command interface.
- On a form the command links a button to an `&НаКлиенте` procedure; heavy work goes to the server.
- Standard commands (Save, Post) are handled by the platform.
- I set command `Доступность` and `Видимость` by roles and object state.
- I am honest: without a server rights check a command only hides UI; it does not protect data.

## How it works

I declare the PickItems command and a handler on a managed form:

```bsl
&НаКлиенте
Процедура ПодобратьТовары(Команда)
    ПараметрыПодбора = Новый Структура("Склад", Объект.Склад);
    Оповещение = Новый ОписаниеОповещения("ПослеПодбора", ЭтотОбъект);
    ОткрытьФорму("Обработка.ПодборТоваровВДокумент.Форма", ПараметрыПодбора, ЭтаФорма, , , , Оповещение);
КонецПроцедуры

&НаКлиенте
Процедура ПослеПодбора(Результат, ДопПараметры) Экспорт
    Если Результат <> Неопределено Тогда
        ЗаполнитьТабличнуюЧастьНаСервере(Результат);
    КонецЕсли;
КонецПроцедуры
```

A command can bind to a hyperlink in a decoration or to a list context menu. The `ИзменяетСохраняемыеДанные` property tells the platform the action needs a write.

## Pitfalls

**Long logic on the client.** Blocks the UI; a server call is required for data.
**Command without a handler.** The form fails to compile or the action is empty.
**Forgotten availability.** The button stays active although the object is posted and cannot change.
**Duplicate command names.** Conflict on one form when copied from a standard config.
**Hiding the button only.** The user can call the same procedure programmatically without a rights check.
