# Metrics and alerts: what does QA verify before release?

## In short

Metrics are numeric time series (latency p95, error rate, RPS). Alerts fire on thresholds (5xx > 1% for 5m) and notify Slack/PagerDuty. QA helps verify dashboards, load smoke, and that alerts actually fire on staging when errors are injected.

## How to answer in an interview

- Golden signals: latency, traffic, errors, saturation.
- SLO example: 99.9% success, p95 < 500ms; burn rate alerts.
- On staging I trigger 503 mock, expect alert in test channel within N minutes.
- Metrics do not replace functional tests; they show prod drift.
- Honestly: alert fatigue if threshold too sensitive; I check runbook link in alert.

## How it works

```gherkin
Feature: Error rate alert

  Scenario: Baseline green
    Given Grafana panel http_5xx_rate below 0.1%
    Then alert CheckoutHighErrors inactive

  Scenario: Inject failures triggers alert
    When chaos test returns 30% 500 on /pay for 10 minutes
    Then Prometheus alert fires CheckoutHighErrors
    And notification delivered to #alerts-staging

  Scenario: Runbook linked
    When alert opens
    Then annotation contains runbook URL and dashboard link
```

Prometheus scrapes `/metrics`; Grafana dashboards; Alertmanager routes by severity.

## Pitfalls

- **Metric without labels.** Aggregate hides one bad tenant.
- **Alert on absolute count not rate.** Peak spike false positive.
- **No staging alert channel.** Test pages on-call at night.
- **Missing alert on queue depth.** API green, jobs stuck.
- **Dashboard timezone.** Incident start misread by 3 hours.
