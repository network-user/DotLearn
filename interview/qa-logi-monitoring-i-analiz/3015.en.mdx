# What is a log file and why does QA need it when investigating bugs?

## In short

A log file is a sequential text record of application or OS events with timestamp, level, and context. QA uses it as evidence of what happened on the server during reproduction when the UI no longer holds details. DEBUG/INFO/WARN/ERROR levels drive triage priority.

## How to answer in an interview

- Logs capture requests, exceptions, business events, and an audit trail.
- On a bug I search ERROR/CRITICAL around the test timestamp and correlation id.
- Structured JSON logs filter more easily in Kibana than plain text.
- Rotation and retention limit history depth; I reproduce quickly.
- Honestly: DEBUG is off in prod; on staging I verify levels match prod policy.

## How it works

```gherkin
Feature: Log file investigation

  Scenario: Trace failed checkout
    Given app writes JSON lines to /var/log/checkout/app.log
    When user completes payment at 2024-03-15T10:05:12Z
    And log line contains level ERROR traceId pay-88
    Then bug report cites log excerpt with traceId

  Scenario: Level filtering
    Given grep pattern '"level":"ERROR"'
    When scan last 500 lines after reproduction
    Then only handled failures visible not noisy INFO

  Scenario: Audit security event
    Given auth.log records failed login
    When brute force test runs
    Then log file shows IP and timestamp for security ticket
```

Sample line (renamed fields):

```json
{"ts":"2024-03-15T10:05:12Z","level":"ERROR","trace":"pay-88","msg":"gateway timeout","userId":"u-104"}
```

QA does not write logs but reads them as a contract between expected and actual backend behavior.

## Pitfalls

- **PII in logs.** Ticket excerpts must be redacted.
- **Unstructured grep.** Free-text regex breaks when format changes.
- **Multiple instances.** One pod log is incomplete without a trace id.
- **Timezone in ts.** Offset breaks HAR correlation.
- **Log without request context.** A stack trace without user id cannot be reproduced.
