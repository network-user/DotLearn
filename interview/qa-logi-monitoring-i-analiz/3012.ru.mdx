# Уровни ошибок в логах: чем ERROR отличается от CRITICAL для QA?

## Суть

DEBUG/INFO нормальный поток; WARN деградация; ERROR handled failure запроса или subsystem; CRITICAL/FATAL процесс или node не может продолжать. QA triage: ERROR → functional bug ticket; CRITICAL → infra/outage with on-call. Mapping varies by framework but severity intent stable.

## Что ответить на собесе

- ERROR: one request failed, others continue.
- CRITICAL: JVM exit, db pool exhausted, k8s CrashLoopBackOff.
- WARN: retry succeeded, deprecated API, disk 85%.
- Alert rules usually ERROR rate and CRITICAL immediate page.
- Честно: teams log everything as ERROR; verify with HTTP status and user impact.

## Как это работает

```gherkin
Feature: Severity triage

  Scenario: ERROR single checkout
    When one POST /pay logs ERROR GatewayTimeout
    Then user sees error toast
    And other users checkout succeeds
    And ticket severity major not outage

  Scenario: CRITICAL service down
    When log CRITICAL Cannot connect database pool
    Then health check fails all instances
    And incident declared P1

  Scenario: WARN before ERROR
    When WARN slow query 2s repeats
    Then proactive ticket before timeout ERROR spike
```

Java: `log.error` vs `log.fatal`; Python `critical()`; map to pager severity in runbook.

## Подводные камни

- **ERROR for 404 scan.** Noise masks real defects.
- **CRITICAL mislabeled handled exception.** False page.
- **No WARN tier.** Miss degradation window.
- **Client-side console.error only.** Server healthy, mis-prioritized.
- **Alert on log text not level.** Regex brittle.
