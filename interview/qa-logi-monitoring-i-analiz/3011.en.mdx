# Log viewing tools: what does QA use by project type?

## In short

Mobile: Logcat, Xcode Console. Web client: DevTools. Backend centralized: ELK, Loki+Grafana, Splunk, Datadog, Graylog. Shipping: Fluent Bit, Filebeat, Vector. QA knows where to click in Kibana and how to adb logcat filter, not required to admin Elasticsearch cluster.

## How to answer in an interview

- ELK: Elasticsearch store, Logstash/Beats ingest, Kibana UI.
- Loki label-based cheaper, Grafana Explore query `{service="api"} |= "error"`.
- Splunk/Datadog enterprise with alerting and APM crossover.
- Mobile adb, desktop Event Viewer rare, journalctl on Linux server.
- Honestly: tool name less important than correlation id and structured fields.

## How it works

| Context | Tool | QA action |
|---------|------|-----------|
| Android | Logcat | filter package ERROR |
| iOS | Xcode | console subsystem filter |
| K8s | kubectl logs / Loki | pod label app=api |
| On-call | Datadog/Splunk | saved search trace_id |

Gherkin:

```gherkin
Feature: Tool selection

  Scenario: Microservice staging
    Given logs in Grafana Loki
    When Explore query {env="staging", service="orders"} |= "ORD-1"
    Then order lifecycle lines visible

  Scenario: Legacy VM
    When SSH and tail /var/log/app.log
    Then grep correlation id works same as Kibana
```

Papertrail, CloudWatch Logs for AWS; know export limit for ticket attachment.

## Pitfalls

- **Kibana index pattern wrong.** Empty results panic.
- **Logstash grok fail.** `_grokparsefailure` tag ignored.
- **Datadog bill sampling.** Missing rare ERROR.
- **adb multiple devices.** Logs from wrong phone.
- **Tool switch per team.** Same field different name breaks search playbook.
