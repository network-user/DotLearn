# How does QA read logs on a remote server over SSH?

## In short

Access to staging or test hosts goes through SSH with a read-only account. Use `tail -f` for live follow-up and `less` or `grep` over the bug time window. In Docker/Kubernetes find the container or pod first, then `docker logs` or `kubectl logs`.

## How to answer in an interview

- I connect with `ssh qa-readonly@staging-host`; keys follow team policy.
- Live: `tail -f /var/log/app/app.log` or `ssh host 'tail -f ...'` in one shot.
- Historical search: `grep 'request-id-7' /var/log/app/*.log` with `--since` in Docker.
- I record UTC timezone and correlation id in the ticket.
- Honestly: prod often has no shell access; then only centralized ELK/Loki.

## How it works

```gherkin
Feature: Remote log access

  Scenario: Follow app log during manual test
    Given SSH session to staging as qa-readonly
    When run tail -f /var/log/checkout/service.log
    And reproduce POST /pay at 14:32 UTC
    Then new ERROR line appears with trace checkout-91

  Scenario: One-shot remote tail
    When ssh staging 'tail -n 200 /var/log/nginx/access.log'
    Then last lines include my test IP and 500 status

  Scenario: Docker service on staging
    Given container name api-checkout-1
    When docker logs api-checkout-1 --since 10m | grep ERROR
    Then output matches HAR timestamp window
```

`less +F` combines follow mode with pause to copy a stack trace. `journalctl -u myapp -f` on systemd hosts.

## Pitfalls

- **Local SSH timezone.** Grep window misses by an hour.
- **logrotate.** Old incident is in `.gz`; use `zgrep`.
- **No rights on /var/log.** Request a role or central logging.
- **tail on a huge file without filter.** Signal lost in noise.
- **Logs on container stdout.** Look via kubectl, not only volume paths.
