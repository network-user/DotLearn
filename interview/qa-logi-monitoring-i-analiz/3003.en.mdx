# Useless logs: what does QA ask to add for observability?

## In short

A log "Processing order" without orderId, userId, correlation id cannot reproduce a bug. QA escalates observability gap: structured JSON, business ids, error codes, duration, sanitized input snapshot. Acceptance criteria include log contract for critical flows.

## How to answer in an interview

- Minimum: timestamp UTC, level, service, trace_id, message, error.code.
- Business context: orderId, paymentId, not password/card PAN.
- Structured JSON for Kibana filters, not free text only.
- Ticket template: "cannot debug without X" with desired line example.
- Honestly: PII in logs is worse than absence; ask for hash/mask.

## How it works

Bad:

```json
{"msg": "failed"}
```

Good:

```json
{
  "level": "ERROR",
  "trace_id": "a1b2",
  "order_id": "ORD-88",
  "step": "payment_capture",
  "error_code": "GATEWAY_TIMEOUT",
  "duration_ms": 3012
}
```

Gherkin:

```gherkin
Feature: Checkout log contract

  Scenario: Failed payment log searchable
    When checkout fails on staging
    Then log line contains trace_id matching response header
    And order_id equals UI displayed order number
    And no raw card number in message

  Scenario: Missing context blocked in review
    Given PR removes order_id from error log
    Then QA blocks until observability checklist met
```

## Pitfalls

- **Log everything DEBUG in prod.** Cost explosion, PII leak.
- **String concat logs.** Cannot query json.order_id.
- **Duplicate logs per retry.** Looks like 5 failures, one timeout.
- **No log on silent catch.** User error, empty logs.
- **Different format per service.** Cross-service search fails.
