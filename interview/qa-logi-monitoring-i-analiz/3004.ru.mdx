# Уровни логирования: как QA выбирает что смотреть при инциденте?

## Суть

Стандартные уровни: TRACE, DEBUG, INFO, WARN, ERROR, FATAL (CRITICAL). Prod обычно INFO+; DEBUG включают временно на staging или sampled tenant. QA фильтрует ERROR/FATAL для triage, WARN для proactive issues, INFO для business audit trail.

## Что ответить на собесе

- ERROR: handled failure, request failed, needs ticket.
- FATAL/CRITICAL: process down, pod crash, page immediately.
- WARN: degraded mode, retry succeeded, tech debt signal.
- INFO: login success, order created for audit.
- Честно: DEBUG в prod без approval = security and cost risk; прошу toggle runbook.

## Как это работает

```gherkin
Feature: Фильтрация по level

  Scenario: Incident triage ERROR only
    Given outage report checkout 14:05
    When Kibana level:ERROR service:payment time range 14:00-14:15
    Then first error line identifies root exception

  Scenario: WARN before outage
    When filter level:WARN same window
    Then connection pool 90% appears before ERROR spike

  Scenario: Staging DEBUG repro
    Given LOG_LEVEL=DEBUG on staging branch
    When reproduce bug steps
    Then DEBUG lines show SQL parameter binding
```

Python logging: `logging.error(...)`; Java SLF4J; mapping consistent across services in platform guide.

## Подводные камни

- **Everything logged as INFO.** Cannot alert on severity.
- **ERROR for expected 404.** Alert noise on scan bots.
- **TRACE left on in prod.** Disk fill in hours.
- **Level per package misconfigured.** Silent module.
- **No structured level field.** Parsed from string " [ERROR] " fragile.
