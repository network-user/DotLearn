# Crash logs: how does QA use them when reproducing failures?

## In short

A crash log captures an unhandled failure moment: exception type, message, stack trace, device/OS, app version, sometimes last breadcrumbs. Mobile (Firebase Crashlytics, Sentry) and desktop collect automatically. QA attaches crash report to ticket and verifies fix on same build matrix.

## How to answer in an interview

- Stack trace top frame is often root cause, not bottom.
- Build version + OS + locale required for repro.
- Symbolicated iOS/Android trace needed for native crash.
- Steps to reproduce from breadcrumbs if user actions logged.
- Honestly: one crash report without steps may be a race; I ask for multiple samples.

## How it works

```
Fatal Exception: java.lang.NullPointerException
  at com.app.checkout.PaymentFragment.onSubmit(PaymentFragment.kt:142)
App version: 3.2.1 (421)
Device: Pixel 6, Android 14
Session breadcrumbs: open_cart → apply_coupon → tap_pay
```

Gherkin:

```gherkin
Feature: Crash report triage

  Scenario: Crash linked to release
    Given Crashlytics spike version 3.2.1 after deploy
    When open top issue NullPointerException PaymentFragment
    Then affected users count documented
    And regression test added for coupon plus pay path

  Scenario: Symbolication required
    Given native crash obfuscated stack
    When dSYM or mapping uploaded
    Then readable frame names for dev fix
```

QA verifies fix in 3.2.2 RC on same device class; monitors crash-free rate metric.

## Pitfalls

- **Obfuscated stack without mapping.** Fix guesswork.
- **Crash on background thread.** User saw ANR not crash dialog.
- **Duplicate issues split.** Same NPE two tickets.
- **PII in crash payload.** Email in custom keys.
- **Flaky race one sample.** Closed cannot reproduce incorrectly.
