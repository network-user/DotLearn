# Log line anatomy: what does QA check in structured logging?

## In short

A typical entry: timestamp (ISO 8601 UTC), severity level, logger/source, message, optional fields (trace_id, user_id hash, http.status, duration_ms). Structured JSON enables Kibana/Loki queries without regex. QA validates completeness on critical paths and absence of secrets.

## How to answer in an interview

- Timestamp UTC for correlation across regions.
- Level for alert routing; source/service name for filter.
- Message human readable; machine fields in JSON keys.
- Process/thread id useful for concurrency bugs.
- Honestly: multiline stack trace must stay one logical event in aggregator.

## How it works

```json
{
  "@timestamp": "2026-07-13T11:23:45.012Z",
  "level": "ERROR",
  "service": "auth-api",
  "logger": "LoginController",
  "trace_id": "c9f1",
  "http.method": "POST",
  "http.path": "/login",
  "http.status": 401,
  "message": "Invalid credentials",
  "duration_ms": 34
}
```

Gherkin:

```gherkin
Feature: Structured log fields

  Scenario: Login failure log complete
    When POST /login wrong password
    Then log contains trace_id http.status 401
    And no plaintext password field

  Scenario: Parseable timestamp
    Given log export
    Then @timestamp sorts chronologically with gateway logs

  Scenario: Source identifies team ownership
    When ERROR in payment-api
    Then on-call routing uses service field
```

Plaintext legacy: `2026-07-13T11:23:45Z ERROR [auth] ...` still parseable if pattern documented.

## Pitfalls

- **Local time without zone.** DST breaks ordering.
- **Message only JSON as string.** Double escape, cannot query.
- **Huge blob in one field.** Index rejected, line dropped.
- **Missing service on worker.** Orphan log in search.
- **Token in message.** Secret scanner alert, log redacted too late.
