# Метрики и алерты: что QA проверяет до релиза?

## Суть

Метрики это числовые time-series (latency p95, error rate, RPS). Алерты срабатывают при пороге (5xx > 1% за 5m) и шлют Slack/PagerDuty. QA участвует в проверке dashboards, smoke под нагрузкой и что алерт реально firing на staging при инжектированной ошибке.

## Что ответить на собесе

- Golden signals: latency, traffic, errors, saturation.
- SLO example: 99.9% success, p95 < 500ms; burn rate alerts.
- На staging вызываю 503 mock, жду alert в test channel за N минут.
- Метрики не заменяют functional tests; показывают drift в prod.
- Честно: alert fatigue если порог слишком чувствительный; проверяю runbook link в alert.

## Как это работает

```gherkin
Feature: Алерт на error rate

  Scenario: Baseline green
    Given Grafana panel http_5xx_rate below 0.1%
    Then alert CheckoutHighErrors inactive

  Scenario: Inject failures triggers alert
    When chaos test returns 30% 500 on /pay 10 minutes
    Then Prometheus alert fires CheckoutHighErrors
    And notification delivered to #alerts-staging

  Scenario: Runbook linked
    When alert opens
    Then annotation contains runbook URL and dashboard link
```

Prometheus scrape `/metrics`; Grafana dashboards; Alertmanager routes by severity.

## Подводные камни

- **Metric without labels.** Aggregate hides one bad tenant.
- **Alert on absolute count not rate.** Spike at peak false positive.
- **No staging alert channel.** Test pages on-call at night.
- **Missing alert on queue depth.** API green, jobs stuck.
- **Dashboard timezone.** Incident start misread by 3 hours.
