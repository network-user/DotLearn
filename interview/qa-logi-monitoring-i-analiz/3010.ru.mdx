# Где QA смотрит логи в web, mobile и backend проекте?

## Суть

Web: DevTools Console/Network, optional server access logs. Mobile: Logcat/Xcode. Backend: docker logs, kubectl, centralized Kibana, local `tail -f` on staging. QA выбирает слой по симптому: UI error → browser console; 500 → api logs + trace id.

## Что ответить на собесе

- Browser Console для frontend JS errors and `console.error`.
- Network tab не заменяет server log, но даёт request id and response body.
- API staging: `docker compose logs -f api` or Grafana Loki explore.
- Document in bug: which layer proved root cause.
- Честно: prod server logs often restricted; staging must mirror prod logging config.

## Как это работает

```gherkin
Feature: Choose log layer

  Scenario: SPA white screen
    When reproduce on Chrome
    Then Console shows Uncaught TypeError stack
    And Network shows no failed API call

  Scenario: API 500 checkout
    When Network shows POST /pay status 500 header X-Request-Id r-8
    Then Kibana search r-8 shows NullPointer in payment service

  Scenario: tail backend on staging
    When docker logs api --since 5m | grep ERROR
    Then lines match timestamp of manual test
```

Makefile `make logs` in monorepos speeds onboarding. Preserve HAR + server log id in ticket.

## Подводные камни

- **Only screenshot Console.** Stack truncated.
- **CORS hides response body.** Misread as frontend bug.
- **Local docker timezone.** Wrong grep window.
- **Verbose frontend logging off in prod build.** Cannot repro minified.
- **Multiple env URLs.** Tested staging UI, grepped prod logs.
