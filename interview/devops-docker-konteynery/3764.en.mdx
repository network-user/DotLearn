# Containers vs VMs: where is the isolation boundary?

## In short

Virtualization emulates hardware and boots a guest OS on a hypervisor: full isolation, higher overhead. Containers share the host kernel, isolated by namespaces and cgroups: fast start and dense packing, but a shared kernel and a weaker hardware boundary.

## How to answer in an interview

- VM: Type-1/Type-2 hypervisor, own kernel, virtio disks/network, typical for multi-tenant legacy and mixed OS.
- Container: image plus runtime (containerd/runc), optional seccomp/AppArmor, one Linux kernel on the host.
- I compare density: tens of VMs vs hundreds of pods on the same hardware.
- I add honestly: for untrusted workloads or Windows+Linux on one machine you still use VMs or separate bare metal; containers are not a universal VM replacement.

## How it works

A hypervisor (KVM, ESXi) assigns vCPU/RAM per VM and boots a guest OS from scratch. Boot takes minutes; snapshots live at the virtual disk layer.

A container starts a process in isolated mount/pid/network namespaces on the host kernel:

```bash
systemd-detect-virt
lscpu | grep -i virtualization

docker run --rm --name bench-alpine alpine:3.20 sleep 1
time docker run --rm alpine:3.20 echo ready

virsh list --all 2>/dev/null || true
```

The first block shows hardware virtualization. `docker run` starts in seconds: no guest kernel boot. VMs via libvirt (`virsh`) live longer and reserve RAM for a full OS.

Kubernetes worker nodes are often VMs (cloud) or bare metal running pods as containers. Nested virtualization (containers in VM) is common in public clouds.

## Pitfalls

- **Kernel CVE.** A host kernel bug hits all containers; VM guests sit behind another kernel.
- **Noisy neighbor without limits.** A container without CPU/memory limits can starve the node.
- **License and OS mix.** Windows vs Linux containers need different host pools; you cannot mix on one kernel.
- **Storage driver overhead.** Overlay on slow disk hurts more than raw block for VM databases.
- **False sense of security.** Default Docker socket mount equals host root; not a "light VM."
