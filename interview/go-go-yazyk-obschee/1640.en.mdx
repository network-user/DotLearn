# What is the order of deferred function calls in Go

## In short

`defer` postpones a call until the current function exits. Execution order is LIFO: the last `defer` runs first. Defer call arguments are evaluated at registration time, not at execution time.

## How to answer in an interview

- Defer stack: Last In, First Out.
- Typical pattern: `defer f.Close()` right after a successful open.
- Defer runs on return, panic, and normal exit.
- A closure in defer sees final variable values if you pass a pointer or use an anonymous func.
- Honestly: defer has a small runtime cost — not in a tight hot loop.

## How it works

```go
func main() {
	fmt.Println("start")
	defer fmt.Println("first")
	defer fmt.Println("second")
	defer fmt.Println("third")
	fmt.Println("end")
}
```

Output: start, end, third, second, first.

```go
func writeReport(path string) error {
	f, err := os.Create(path)
	if err != nil {
		return err
	}
	defer f.Close()

	_, err = f.WriteString("ok")
	return err
}
```

`Close` is guaranteed even on early return. For a mutable outcome:

```go
defer func() {
	if err != nil {
		_ = os.Remove(tmpPath)
	}
}()
```

## Pitfalls

**defer fmt.Println(i) in a loop.** Prints the final `i`, not the current one — use an IIFE with an argument.

**Many defers in a deep stack.** Overhead and non-obvious cleanup order.

**defer Close() after a failed Create.** Nil pointer panic — defer only after successful acquire.

**Forget that the return value is evaluated before defer.** Named return + defer can change the result.

**defer lock.Unlock() without defer on the panic path before lock.** Deadlock if panic happens between lock and defer.
