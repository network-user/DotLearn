# go.mod, go.sum, and module resolution

## In short

`go.mod` records the module name, Go version, and dependency graph (`require`, `replace`, `exclude`). `go.sum` stores cryptographic hashes of modules and their `go.mod` for verification on download. Module resolution picks the minimum compatible version (MVS) and stores artifacts in the module cache.

## How to answer in an interview

- `go mod init`, `go get pkg@v1.2.3`, `go mod tidy` sync the graph.
- `replace` for local development or a fork.
- Commit `go.sum` to VCS together with `go.mod`.
- Proxy `GOPROXY` and checksum DB `GOSUMDB` in CI.
- Honestly: MVS sometimes pulls a higher version than expected; I check `go mod graph` and `go list -m all`.

## How it works

```go
module gitlab.corp/checkout

go 1.22

require (
	github.com/go-chi/chi/v5 v5.0.12
	go.uber.org/zap v1.27.0
)

replace github.com/legacy/lib => ../fork/lib v0.0.0-00010101000000-000000000000
```

On `go build` the toolchain reads `go.mod`, downloads missing modules via proxy, verifies hashes against `go.sum` and `sum.golang.org`. New dependencies add lines to `go.sum` automatically. `go mod vendor` copies sources for reproducible offline builds.

## Pitfalls

**Delete go.sum "for cleanliness".** CI fails or supply-chain check is lost.

**replace in prod without a plan.** Local path dependency breaks deploy.

**Incompatible major path.** A `v2` module requires `/v2` in the import path.

**Private module without GOPRIVATE.** Module name leaks to a public proxy.

**Hand-edit require without tidy.** Stale indirect deps and version conflicts.
