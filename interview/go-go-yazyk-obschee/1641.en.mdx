# What is shadowing, why is it dangerous, and how does Go regulate it

## In short

Shadowing is when an inner block declares an identifier with the same name as an outer scope and "hides" the outer one. In Go this is legal in new blocks (`if`, `for`, nested functions). It is dangerous because we confuse which variable we are changing, especially `err` in `if err := ...`.

## How to answer in an interview

- Shadowing ≠ reassignment: `:=` in an inner block creates a new variable.
- Classic bug: `err` shadow in `if`, the outer scope keeps the old nil/err.
- `go vet` with `-shadow` / staticcheck catches some cases.
- In `for _, item := range` — `item` is reused; closures capture the last value without `item := item`.
- Honestly: shadowing is sometimes intentional (short scope), but avoid shadowing `err`.

## How it works

```go
var x = 10

func main() {
	x := 20
	fmt.Println(x)

	if x := 5; x > 0 {
		fmt.Println(x)
	}
	fmt.Println(x)
}
```

Inside `if` — its own `x`. After the block, `20` again.

```go
func process() error {
	var err error
	if data, err := load(); err != nil {
		return err
	}
	return err
}
```

The inner `err` shadows the outer — `return err` outside may return nil even though load failed. Fix: separate names or `data, loadErr := load()`.

```go
for _, v := range items {
	v := v
	go func() { use(v) }()
}
```

## Pitfalls

**Shadow err in every if.** Silent logic bugs in error handling.

**Shadow an import or package name.** Rare, but hurts readability.

**switch with short variable declaration.** New scope per case in Go 1.22+ — know your version.

**Linter disabled in CI.** Shadowing reaches prod.

**Assume `:=` always creates a new variable.** If at least one name is new — others may be reused.
