# Pros and cons of Go

## In short

Go offers static typing, fast compilation, a single static binary, built-in goroutines and channels, GC, and a pragmatic stdlib for network services. Cons: limited type expressiveness before recent generics, verbose error handling, weaker GUI/DS ecosystem than Python, no classical inheritance.

## How to answer in an interview

- Pros: simple language, predictable latency, cross-compile, strong for cloud backends.
- Cons: `if err != nil` boilerplate, less metaprogramming, historically no generics (since 1.18 they exist).
- Trade-off: I pick Go for IO-bound services, not heavy numeric/UI work.
- I compare with Rust/Java by team and SLA, not dogma.
- Honestly: "simplicity" sometimes means more hand-written code than in Kotlin/Python.

## How it works

```go
func main() {
	jobs := make(chan int, 8)
	var wg sync.WaitGroup

	for w := 0; w < 4; w++ {
		wg.Add(1)
		go func() {
			defer wg.Done()
			for id := range jobs {
				process(id)
			}
		}()
	}

	for id := 1; id <= 20; id++ {
		jobs <- id
	}
	close(jobs)
	wg.Wait()
}
```

One binary without JVM warmup, deploy to a scratch image. GC tuning via `GOGC`, profiling with `pprof`. Generics reduce duplication in container helpers.

## Pitfalls

**Go for CPU-bound work without profiling.** Hit one core or GC pauses.

**Ignore context and cancellation.** Goroutine leaks under load.

**Treat stdlib as a full replacement for mature frameworks.** ORM and observability stacks need extra work.

**Microservices in Go "because trendy".** Ops cost without payoff.

**Expect OOP like Java.** Composition and interfaces, not class hierarchies.
