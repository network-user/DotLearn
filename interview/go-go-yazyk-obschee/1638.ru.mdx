# Плюсы и минусы Go

## Суть

Go даёт статическую типизацию, быструю компиляцию, один статический бинарь, встроенные goroutine и channel, GC и прагматичную stdlib для сетевых сервисов. Минусы: ограниченная выразительность типов до недавних generics, многословность error handling, слабее экосистема GUI/DS чем Python, нет классического наследования.

## Что ответить на собесе

- Плюсы: простой язык, предсказуемая latency, cross-compile, хорош для cloud backend.
- Минусы: boilerplate `if err != nil`, меньше метапрограммирования, исторически без generics (с 1.18 есть).
- Компромисс: выбираю Go для IO-bound сервисов, не для тяжёлого numeric/UI.
- Сравниваю с Rust/Java по команде и SLA, не религиозно.
- Честно: «простота» иногда значит писать больше кода вручную, чем в Kotlin/Python.

## Как это работает

```go
func main() {
	jobs := make(chan int, 8)
	var wg sync.WaitGroup

	for w := 0; w < 4; w++ {
		wg.Add(1)
		go func() {
			defer wg.Done()
			for id := range jobs {
				process(id)
			}
		}()
	}

	for id := 1; id <= 20; id++ {
		jobs <- id
	}
	close(jobs)
	wg.Wait()
}
```

Один бинарь без JVM warmup, деплой в scratch image. GC tuning через `GOGC`, профилирование `pprof`. Generics закрывают часть дублирования container helpers.

## Подводные камни

**Go для CPU-bound без профиля.** Упираемся в одно ядро или GC pause.

**Игнор context и cancellation.** Goroutine leak под нагрузкой.

**Считать stdlib полной заменой зрелым фреймворкам.** ORM, observability стека допиливаем сами.

**Микросервисы на Go «потому что модно».** Ops cost без выигрыша.

**Ожидать ООП как в Java.** Композиция и интерфейсы, не иерархии классов.
