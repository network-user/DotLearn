# Множественные возвращаемые значения в Go

## Суть

Функция в Go может вернуть несколько значений в одном `return`. Идиоматичный паттерн `(T, error)`: результат и ошибка без исключений. Именованные результаты позволяют `return` без аргументов, но злоупотреблять ими не стоит.

## Что ответить на собесе

- `func ReadConfig(path string) (Config, error)` вместо throw.
- Игнорирую лишнее через `_`: `_, err := fmt.Fprintf(w, ...)`.
- Именованные returns удобны в defer для логирования `err`.
- Множественные значения без error для координат, `(value, ok)` из map.
- Честно: больше двух-трёх return снижает читаемость; лучше struct.

## Как это работает

```go
func divide(a, b float64) (float64, error) {
	if b == 0 {
		return 0, errors.New("division by zero")
	}
	return a / b, nil
}

func lookupUser(id int) (User, bool) {
	u, ok := store[id]
	return u, ok
}

func parsePort(raw string) (port int, err error) {
	defer func() {
		if err != nil {
			err = fmt.Errorf("parse port %q: %w", raw, err)
		}
	}()
	port, err = strconv.Atoi(raw)
	return
}
```

Вызов всегда проверяет `err` перед использованием результата. Компилятор требует использовать все объявленные return variables при именованных results.

## Подводные камни

**Проверять err после третьего return value.** Легко перепутать порядок при `(data, cursor, err)`.

**Именованные results + shadowing.** Неявный `return` возвращает не то поле.

**Возврат больших struct по значению.** Лишние копии; иногда указатель или error only.

**Паника вместо error в библиотеке.** Ломает контракт Go API.

**Игнор `_` на error из критичного IO.** Тихая потеря данных.
