# go.mod, go.sum и module resolution

## Суть

`go.mod` фиксирует имя модуля, версию Go и граф зависимостей (`require`, `replace`, `exclude`). `go.sum` хранит криптографические хеши модулей и их `go.mod` для проверки при загрузке. Module resolution выбирает минимальную совместимую версию (MVS) и кладёт артефакты в module cache.

## Что ответить на собесе

- `go mod init`, `go get pkg@v1.2.3`, `go mod tidy` синхронизируют граф.
- `replace` для локальной разработки или форка.
- `go.sum` коммитим в VCS вместе с `go.mod`.
- Proxy `GOPROXY` и checksum DB `GOSUMDB` в CI.
- Честно: MVS иногда тянет версию выше ожидаемой; смотрю `go mod graph` и `go list -m all`.

## Как это работает

```go
module gitlab.corp/checkout

go 1.22

require (
	github.com/go-chi/chi/v5 v5.0.12
	go.uber.org/zap v1.27.0
)

replace github.com/legacy/lib => ../fork/lib v0.0.0-00010101000000-000000000000
```

При `go build` toolchain читает `go.mod`, скачивает недостающие модули через proxy, сверяет хеши с `go.sum` и `sum.golang.org`. Новые зависимости добавляют строки в `go.sum` автоматически. `go mod vendor` копирует исходники для reproducible offline build.

## Подводные камни

**Удалить go.sum «для чистоты».** CI падает или теряется supply-chain check.

**replace в prod без плана.** Зависимость от локального пути ломает деплой.

**Несовместимый major path.** `v2` модуль требует `/v2` в import path.

**Private module без GOPRIVATE.** Утечка имени модуля в публичный proxy.

**Ручное редактирование require без tidy.** Мусорные indirect и конфликт версий.
