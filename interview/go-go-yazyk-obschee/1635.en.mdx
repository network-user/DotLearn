# Multiple return values in Go

## In short

A Go function can return several values in one `return`. The idiomatic pattern is `(T, error)`: result and error without exceptions. Named results allow bare `return`, but should not be overused.

## How to answer in an interview

- `func ReadConfig(path string) (Config, error)` instead of throw.
- Ignore extras with `_`: `_, err := fmt.Fprintf(w, ...)`.
- Named returns help in defer to log `err`.
- Multiple values without error for coordinates, `(value, ok)` from a map.
- Honestly: more than two or three returns hurts readability; prefer a struct.

## How it works

```go
func divide(a, b float64) (float64, error) {
	if b == 0 {
		return 0, errors.New("division by zero")
	}
	return a / b, nil
}

func lookupUser(id int) (User, bool) {
	u, ok := store[id]
	return u, ok
}

func parsePort(raw string) (port int, err error) {
	defer func() {
		if err != nil {
			err = fmt.Errorf("parse port %q: %w", raw, err)
		}
	}()
	port, err = strconv.Atoi(raw)
	return
}
```

The caller always checks `err` before using the result. The compiler requires using all declared return variables with named results.

## Pitfalls

**Check err after the third return value.** Easy to mix order with `(data, cursor, err)`.

**Named results + shadowing.** Bare `return` may return the wrong binding.

**Return large struct by value.** Extra copies; sometimes pointer or error only.

**Panic instead of error in a library.** Breaks the Go API contract.

**Ignore `_` on error from critical IO.** Silent data loss.
