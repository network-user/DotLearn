# Why Domain-Driven Design matters on the frontend

## In short

Domain-Driven Design (DDD) puts the business domain at the center of design: code and UI reflect business rules instead of arbitrary screen structure. On the frontend this keeps data models, state, and components aligned with how the product actually works.

## How to answer in an interview

- I explain that DDD pays off when business logic is heavy: orders, billing, access control, not a three-block landing page.
- I name core ideas: ubiquitous language with the product team, bounded context, entities, value objects, aggregates.
- I tie it to the frontend: the domain model shapes forms, validation, state, and module boundaries (FSD layers, feature modules).
- I clarify that DDD does not replace UX patterns and is not mandatory "clean architecture" on every pet project.
- I add honestly: on small teams full DDD is often overkill; a shared vocabulary and explicit context boundaries are enough.

## How it works

DDD emerged to close the gap between engineers and domain experts. Instead of technical abstractions like `UserDTO`, the team agrees on business terms: "Order", "Line item", "Payment status". That ubiquitous language flows into types, API contracts, and components.

A **bounded context** limits where a term has a precise meaning. "Customer" in CRM and "Customer" in billing can be different models with different fields. On the frontend, contexts often map to routes, micro-frontends, or large feature slices.

Inside a context you define **entities** (identity matters, `OrderId`), **value objects** (compared by value, `Money`, `Address`), **aggregates** (a root entity plus invariants around it), and **repositories** (access to aggregates without leaking SQL into the UI).

Storefront example: a `Cart` aggregate holds line items and recalculates totals; it exposes only `addItem`, `removeItem`, `checkout`, not a raw table row array.

```ts
type ProductId = string & { readonly brand: unique symbol };
type CartId = string & { readonly brand: unique symbol };

interface CartLine {
  productId: ProductId;
  quantity: number;
  unitPrice: number;
}

class ShoppingCart {
  constructor(
    readonly id: CartId,
    private lines: CartLine[] = [],
  ) {}

  addItem(line: CartLine): void {
    const existing = this.lines.find((l) => l.productId === line.productId);
    if (existing) {
      existing.quantity += line.quantity;
      return;
    }
    this.lines.push({ ...line });
  }

  total(): number {
    return this.lines.reduce((sum, l) => sum + l.quantity * l.unitPrice, 0);
  }
}
```

In an interview it helps to show that DDD reduces "telephone game" drift between layers: the same order does not become three incompatible objects from API to checkout.

## Pitfalls

**Over-engineering.** DDD on simple CRUD creates extra entities and folders with no gain in product clarity.

**Anemic model.** Types exist but all logic lives in hooks and services; the domain becomes decoration.

**Ignoring bounded context.** One global store for the whole product mixes terms and breaks aggregate invariants.

**Copying backend DDD one-to-one.** The frontend lives in UI events and cache; not every server aggregate belongs in the browser.

**No living language with the business.** Without regular talks with analysts, ubiquitous language goes stale quickly.
