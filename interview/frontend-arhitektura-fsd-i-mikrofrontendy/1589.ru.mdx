# Зачем фронтенду Domain-Driven Design

## Суть

Domain-Driven Design (DDD) ставит предметную область в центр проектирования: код и интерфейсы отражают бизнес-правила, а не случайную структуру экранов. На фронтенде это помогает держать модель данных, состояние и UI согласованными с тем, как продукт реально работает.

## Что ответить на собесе

- Объясняю, что DDD полезен там, где бизнес-логика сложная: заказы, биллинг, права доступа, а не на лендинге из трёх блоков.
- Называю ключевые идеи: ubiquitous language с продуктом, bounded context, сущности, value objects, агрегаты.
- Показываю связь с фронтом: доменная модель влияет на формы, валидацию, состояние и границы модулей (FSD-слои, feature-модули).
- Уточняю, что DDD не заменяет UX-паттерны и не требует «чистой архитектуры» на каждом pet-проекте.
- Честно добавляю: на маленьких командах полный DDD часто избыточен, достаточно общего словаря и явных границ контекстов.

## Как это работает

DDD родился как ответ на разрыв между разработчиками и экспертами предметной области. Вместо технических абстракций вроде `UserDTO` команда договаривается о терминах бизнеса: «Заказ», «Позиция», «Статус оплаты». Этот ubiquitous language попадает в типы, API-контракты и компоненты.

**Bounded context** ограничивает, где термин имеет конкретный смысл. «Клиент» в CRM и «Клиент» в биллинге могут быть разными моделями с разными полями. На фронте контексты часто совпадают с маршрутами, микрофронтендами или крупными feature-срезами.

Внутри контекста выделяют **сущности** (идентичность важна, `OrderId`), **value objects** (сравниваются по значению, `Money`, `Address`), **агрегаты** (корневая сущность и инварианты вокруг неё) и **репозитории** (доступ к агрегатам без протаскивания SQL в UI).

Пример для витрины: агрегат `Cart` хранит позиции и пересчитывает итог; наружу отдаёт только операции `addItem`, `removeItem`, `checkout`, а не сырой массив строк таблицы.

```ts
type ProductId = string & { readonly brand: unique symbol };
type CartId = string & { readonly brand: unique symbol };

interface CartLine {
  productId: ProductId;
  quantity: number;
  unitPrice: number;
}

class ShoppingCart {
  constructor(
    readonly id: CartId,
    private lines: CartLine[] = [],
  ) {}

  addItem(line: CartLine): void {
    const existing = this.lines.find((l) => l.productId === line.productId);
    if (existing) {
      existing.quantity += line.quantity;
      return;
    }
    this.lines.push({ ...line });
  }

  total(): number {
    return this.lines.reduce((sum, l) => sum + l.quantity * l.unitPrice, 0);
  }
}
```

На собесе важно показать, что DDD снижает «телефонный эффект» между слоями: один и тот же заказ не превращается в три несовместимых объекта на пути от API до чекаута.

## Подводные камни

**Переусложнение.** DDD на простом CRUD даёт лишние сущности и папки без выигрыша в понимании продукта.

**Анемичная модель.** Типы есть, но вся логика в хуках и сервисах; домен превращается в декорацию.

**Игнорирование bounded context.** Один глобальный store на весь продукт смешивает термины и ломает инварианты агрегатов.

**Копирование бэкенд-DDD один в один.** Фронт живёт в UI-событиях и кэше; не каждый серверный агрегат нужен в браузере.

**Отсутствие живого языка с бизнесом.** Без регулярного общения с аналитиками ubiquitous language быстро устаревает.
