# Как определить границы Micro Frontend

## Суть

Micro Frontend это не «много SPA», а bounded context в UI: модуль с own lifecycle, data store, deploy pipeline и team ownership. Граница проходит по business capability (billing, search, profile), не по техническому слою (buttons folder). Критерии: независимая ценность, отдельный SLA, минимум synchronous shared state.

## Что ответить на собесе

- DDD subdomain map: checkout, catalog, loyalty — кандидаты; `Button.tsx` — нет.
- Conway alignment: одна squad owns end-to-end vertical slice.
- Integration async: REST/BFF/events, не shared mutable singleton store между MFE.
- Честно: split по route без org pain даёт distributed monolith; хуже single repo.
- Strangler: выносим самый изолированный domain первым (marketing landing), не core checkout.

## Как это работает

E-commerce decomposition:

| MFE | Owns | Integrates via |
|-----|------|----------------|
| Catalog | PLP, filters | Search API |
| Cart | line items, promos | `cart.updated` custom event |
| Account | profile, orders | SSO cookie from shell |

```ts
// shell router mounts remotes by path prefix
const routes = [
  { path: '/shop/*', remote: 'catalog' },
  { path: '/cart/*', remote: 'cart' },
  { path: '/me/*', remote: 'account' },
];
```

Contract test: catalog publishes `ProductCardProps` schema; cart consumes add-to-cart event payload zod-validated. Design tokens via `@acme/ui-tokens` npm package, не copy-paste CSS.

Decision checklist: отдельный on-call? отдельный error budget? можно ли выключить модуль feature flag без rebuild shell? три «да» → MFE кандидат.

## Подводные камни

- **Shared state trap.** Global Redux across remotes = hidden monolith.
- **Too granular.** Header/Footer как отдельные MFE multiply network + ops cost.
- **Synchronous props drilling.** Host передаёт 40 props в remote → tight coupling.
- **i18n split.** Разные translation bundles ломают consistent tone.
- **Local dev hell.** 5 remotes running + shell без orchestration docker compose.
