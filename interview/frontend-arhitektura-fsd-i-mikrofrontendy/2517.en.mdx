# Why Micro Frontend architecture

## In short

Micro Frontends split UI into autonomous deployable units (catalog, checkout, account), each with its own repo/CI/team. A shell (host) orchestrates composition: runtime integration (Module Federation), iframes, or Web Components. The goal is org parallelism, not only bundle size.

## How to answer in an interview

- Independent release: hotfix payments without redeploying the whole monolith SPA.
- Polyglot teams: checkout on React, marketing CMS widget on Vue, with clear boundaries.
- Blast radius: one remote failing should not take down the shell with graceful fallback.
- Honestly: operational cost is high (version skew, shared design system, e2e); skip for an 8-person startup.
- Alternative: modular monolith plus FSD slices in one repo often suffices until 100+ devs.

## How it works

Host shell (Module Federation concept):

```ts
// host/vite.config.ts — remote declared at build
const remotes = {
  checkout: 'checkout@https://cdn.example.com/checkout/remoteEntry.js',
};

// host bootstrap
const { mountCheckout } = await import('checkout/CheckoutApp');
mountCheckout(document.getElementById('checkout-root')!);
```

Remote exposes:

```ts
// checkout exposes ./CheckoutApp
export function mountCheckout(root: HTMLElement) {
  createRoot(root).render(<CheckoutRoutes />);
}
```

Shared deps: `react`, `react-dom` singleton in federation shared config, or duplicate React hooks crash. Cross-MFE contract: event bus (`window.dispatchEvent`), BFF API, or a shared npm design-system package versioned with semver.

## Pitfalls

- **Version skew.** Host on remote v3 while CDN still serves v2 → runtime `undefined` export.
- **CSS collisions.** Global Tailwind preflight twice breaks layout.
- **Auth/session.** Cookie domain and token refresh are shell concerns, not duplicated.
- **Observability.** Distributed tracing needs a correlation id through shell navigation.
- **Testing matrix.** N remotes → combinatorial e2e; contract tests between teams are mandatory.
