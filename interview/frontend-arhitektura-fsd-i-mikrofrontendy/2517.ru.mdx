# Зачем Micro Frontend архитектура

## Суть

Micro Frontend дробит UI на автономные deployable units (catalog, checkout, account), каждая со своим repo/CI/team. Shell (host) оркестрирует composition: runtime integration (Module Federation), iframe, или Web Components. Цель: scale org parallelism, не только bundle size.

## Что ответить на собесе

- Independent release: hotfix в payments без redeploy всего monolith SPA.
- Polyglot teams: checkout на React, marketing CMS widget на Vue, если boundary чёткий.
- Blast radius: падение одного remote не обязано валить shell, если есть graceful fallback.
- Честно: operational cost высокий (version skew, shared design system, e2e); не брать для стартапа на 8 человек.
- Альтернатива: modular monolith + FSD slices в одном repo часто достаточно до 100+ devs.

## Как это работает

Host shell (Module Federation concept):

```ts
// host/vite.config.ts — remote declared at build
const remotes = {
  checkout: 'checkout@https://cdn.example.com/checkout/remoteEntry.js',
};

// host bootstrap
const { mountCheckout } = await import('checkout/CheckoutApp');
mountCheckout(document.getElementById('checkout-root')!);
```

Remote exposes:

```ts
// checkout exposes ./CheckoutApp
export function mountCheckout(root: HTMLElement) {
  createRoot(root).render(<CheckoutRoutes />);
}
```

Shared deps: `react`, `react-dom` singleton в federation shared config, иначе duplicate React hooks crash. Cross-MFE contract: event bus (`window.dispatchEvent`), BFF API, или shared npm design-system package versioned semver.

## Подводные камни

- **Version skew.** Host на remote v3, CDN ещё v2 → runtime `undefined` export.
- **CSS collisions.** Global Tailwind preflight дважды ломает layout.
- **Auth/session.** Cookie domain и token refresh должны быть shell concern, не дублироваться.
- **Observability.** Distributed tracing нужен correlation id через shell navigation.
- **Testing matrix.** N remotes → combinatorial e2e; contract tests между teams обязательны.
