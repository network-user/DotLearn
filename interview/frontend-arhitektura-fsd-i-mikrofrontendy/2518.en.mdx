# How to define Micro Frontend boundaries

## In short

Micro Frontends are not "many SPAs" but bounded contexts in UI: a module with its own lifecycle, data store, deploy pipeline, and team ownership. Boundaries follow business capabilities (billing, search, profile), not technical layers (a buttons folder). Criteria: independent value, separate SLA, minimal synchronous shared state.

## How to answer in an interview

- DDD subdomain map: checkout, catalog, loyalty are candidates; `Button.tsx` is not.
- Conway alignment: one squad owns an end-to-end vertical slice.
- Async integration: REST/BFF/events, not a shared mutable singleton store across MFEs.
- Honestly: splitting by route without org pain yields a distributed monolith worse than a single repo.
- Strangler: extract the most isolated domain first (marketing landing), not core checkout.

## How it works

E-commerce decomposition:

| MFE | Owns | Integrates via |
|-----|------|----------------|
| Catalog | PLP, filters | Search API |
| Cart | line items, promos | `cart.updated` custom event |
| Account | profile, orders | SSO cookie from shell |

```ts
// shell router mounts remotes by path prefix
const routes = [
  { path: '/shop/*', remote: 'catalog' },
  { path: '/cart/*', remote: 'cart' },
  { path: '/me/*', remote: 'account' },
];
```

Contract test: catalog publishes `ProductCardProps` schema; cart consumes add-to-cart event payload validated with zod. Design tokens via `@acme/ui-tokens` npm package, not copy-paste CSS.

Decision checklist: separate on-call? separate error budget? can the module be disabled with a feature flag without rebuilding the shell? three yes → MFE candidate.

## Pitfalls

- **Shared state trap.** Global Redux across remotes is a hidden monolith.
- **Too granular.** Header/Footer as separate MFEs multiply network and ops cost.
- **Synchronous props drilling.** Host passes 40 props into a remote → tight coupling.
- **i18n split.** Different translation bundles break consistent tone.
- **Local dev hell.** Five remotes plus shell without docker-compose orchestration.
