# Assertions in Java tests: what types exist?

## In short

Assertions capture expectations in tests. JUnit 5 uses `org.junit.jupiter.api.Assertions` and AssertJ fluent API. The Java `assert` keyword with `-ea` suits internal invariants, not production validation or primary test assertions.

## How to answer in an interview

- JUnit: `assertEquals`, `assertThrows`, `assertAll` for grouped checks.
- AssertJ: `assertThat(list).hasSize(2).extracting("id")...` for readable failures.
- Hamcrest matchers for custom predicates; Mockito `verify` does not replace outcome asserts.
- Soft assertions (AssertJ `SoftAssertions`) collect all failures in one test.
- Honestly: `assert` keyword is off by default in prod JVM; tests rely on JUnit.

## How it works

JUnit 5:

```java
@Test
void invoiceTotalsMatch() {
    Invoice invoice = factory.paidInvoice();
    assertAll(
        () -> assertEquals("PAID", invoice.status()),
        () -> assertTrue(invoice.total().signum() > 0),
        () -> assertNotNull(invoice.issuedAt())
    );
}

@Test
void invalidQtyRejected() {
    assertThrows(InvalidQuantityException.class,
        () -> cart.add(line("SKU", -3)));
}
```

AssertJ:

```java
@Test
void sortedByDueDate() {
    assertThat(repository.overdue())
        .hasSize(2)
        .extracting(Task::dueDate)
        .isSorted();
}
```

Internal invariant (dev only):

```java
void applyDiscount(Money base) {
    assert base.signum() >= 0 : "base must be non-negative";
    // business logic
}
```

Run tests with assertions always on; `-ea` is irrelevant for JUnit failures (`AssertionError` from org.junit).

## Pitfalls

- **assert keyword in library code.** Disabled JVM skips silently.
- **assertEquals wrong argument order.** expected vs actual swap confuses diff.
- **Floating compare without delta.** `0.1 + 0.2` fails exact equality.
- **Single assert per heavy setup.** First failure hides rest; use `assertAll`.
- **verify without assert state.** Green interaction, wrong business result.
