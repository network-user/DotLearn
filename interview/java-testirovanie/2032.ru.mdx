# @Test и @ParameterizedTest: в чём разница?

## Суть

`@Test` запускает метод один раз с фиксированным сценарием. `@ParameterizedTest` повторяет тот же метод для каждого набора аргументов из `@CsvSource`, `@MethodSource` и др. Оба в JUnit Jupiter; parameterized экономит дублирование data-driven cases.

## Что ответить на собесе

- `@Test` для уникальных setup/flow; parameterized для табличных inputs.
- Parameterized каждый case отдельная строка в IDE/CI report.
- Нельзя смешать `@Test` и `@ParameterizedTest` на одном method.
- `@NullSource` + `@ValueSource` комбинируют edge inputs.
- Честно: 200-row CSV лучше в file `@CsvFileSource`, не в annotation string.

## Как это работает

Single scenario:

```java
@Test
void archivesCompletedTicket() {
    Ticket ticket = factory.done("T-1");
    service.archive(ticket.id());
    assertTrue(repository.find(ticket.id()).orElseThrow().archived());
}
```

Parameterized:

```java
@ParameterizedTest
@CsvSource({ "2, true", "3, false", "4, true" })
void parityCheck(int value, boolean expectedEven) {
    assertEquals(expectedEven, value % 2 == 0);
}
```

JUnit создаёт три invocations `parityCheck(int, boolean)` с индексами `[1]`, `[2]`, `[3]`. `@MethodSource` передаёт complex objects:

```java
static Stream<Arguments> refundCases() {
    return Stream.of(
        Arguments.of(Money.of("10", "USD"), true),
        Arguments.of(Money.of("0", "USD"), false)
    );
}
```

Failure показывает конкретный argument set, не весь class red.

## Подводные камни

- **Parameterized без descriptive name.** Report `[3]` без контекста.
- **Heavy work in argument provider.** Slow discovery phase.
- **Mutable arguments reused.** Provider returns same list instance.
- **Duplicate coverage with many @Test.** Copy paste вместо one parameterized.
- **JUnit 4 `@RunWith(Parameterized)` mix.** Vintage engine confusion.
