# @Test vs @ParameterizedTest: what's the difference?

## In short

`@Test` runs a method once with a fixed scenario. `@ParameterizedTest` repeats the same method for each argument set from `@CsvSource`, `@MethodSource`, etc. Both are JUnit Jupiter; parameterized saves duplication for data-driven cases.

## How to answer in an interview

- `@Test` for unique setup/flow; parameterized for tabular inputs.
- Each parameterized case is a separate row in IDE/CI report.
- You cannot mix `@Test` and `@ParameterizedTest` on one method.
- `@NullSource` + `@ValueSource` combine edge inputs.
- Honestly: a 200-row CSV belongs in `@CsvFileSource`, not an annotation string.

## How it works

Single scenario:

```java
@Test
void archivesCompletedTicket() {
    Ticket ticket = factory.done("T-1");
    service.archive(ticket.id());
    assertTrue(repository.find(ticket.id()).orElseThrow().archived());
}
```

Parameterized:

```java
@ParameterizedTest
@CsvSource({ "2, true", "3, false", "4, true" })
void parityCheck(int value, boolean expectedEven) {
    assertEquals(expectedEven, value % 2 == 0);
}
```

JUnit creates three invocations `parityCheck(int, boolean)` with indices `[1]`, `[2]`, `[3]`. `@MethodSource` passes complex objects:

```java
static Stream<Arguments> refundCases() {
    return Stream.of(
        Arguments.of(Money.of("10", "USD"), true),
        Arguments.of(Money.of("0", "USD"), false)
    );
}
```

Failure shows the specific argument set, not the whole class red.

## Pitfalls

- **Parameterized without descriptive name.** Report shows `[3]` without context.
- **Heavy work in argument provider.** Slow discovery phase.
- **Mutable arguments reused.** Provider returns same list instance.
- **Duplicate coverage with many @Test.** Copy paste instead of one parameterized.
- **JUnit 4 `@RunWith(Parameterized)` mix.** Vintage engine confusion.
