# Когда использовать parameterized tests в JUnit 5?

## Суть

Parameterized tests запускают один метод с несколькими наборами входных данных. Это убирает copy-paste тестов и расширяет покрытие граничных значений без раздувания класса десятками почти одинаковых `@Test`.

## Что ответить на собесе

- `@ParameterizedTest` + `@CsvSource`, `@MethodSource`, `@ValueSource`.
- Подходит для pure functions, validators, parsers, discount rules.
- Каждая строка CSV = отдельный test case в отчёте CI.
- `@NullAndEmptySource` для строк и коллекций добавляет edge cases одной аннотацией.
- Честно: при сложной setup логике `@MethodSource` с POJO аргументами читабельнее, чем огромный CSV.

## Как это работает

```java
class SlugifierTest {

    @ParameterizedTest
    @CsvSource({
        "Hello World, hello-world",
        "API Gateway, api-gateway",
        "  trim  , trim"
    })
    void buildsSlug(String input, String expected) {
        assertEquals(expected, Slugifier.from(input));
    }

    @ParameterizedTest
    @MethodSource("invalidInputs")
    void rejectsBadInput(String raw) {
        assertThrows(IllegalArgumentException.class, () -> Slugifier.from(raw));
    }

    static Stream<String> invalidInputs() {
        return Stream.of("", "!!!", "a".repeat(300));
    }
}
```

JUnit генерирует display name `buildsSlug(String, String) [1] input=Hello World`. `@EnumSource(DayOfWeek.class)` проверяет все enum constants. Для больших таблиц данные держу в `classpath:cases/slug.csv` через `@CsvFileSource`.

## Подводные камни

- **Один assert скрывает причину.** Именуй параметры `@CsvSource` и включай `name` template.
- **Mutable shared state между iterations.** Поля класса меняются и ломают следующий case.
- **Слишком широкий MethodSource.** 500 кейсов без группировки нечитаемы в IDE.
- **Смешение unit и integration в одном parameterized.** Медленный IO в каждой строке.
- **Забытый `@ParameterizedTest` без provider.** Пустой test class или runtime error.
