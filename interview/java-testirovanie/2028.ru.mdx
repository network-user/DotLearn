# Assertions в Java-тестах: какие бывают?

## Суть

Assertions фиксируют ожидания в тестах. В JUnit 5 используют `org.junit.jupiter.api.Assertions` и AssertJ fluent API. Java keyword `assert` с `-ea` применяют для internal invariants, не для production validation и не как основной test assert.

## Что ответить на собесе

- JUnit: `assertEquals`, `assertThrows`, `assertAll` для grouped checks.
- AssertJ: `assertThat(list).hasSize(2).extracting("id")...` readable failures.
- Hamcrest matchers для custom predicates; Mockito `verify` не заменяет assert outcome.
- Soft assertions (AssertJ `SoftAssertions`) собирают все failures одного test.
- Честно: `assert` keyword off by default in prod JVM; tests rely on JUnit.

## Как это работает

JUnit 5:

```java
@Test
void invoiceTotalsMatch() {
    Invoice invoice = factory.paidInvoice();
    assertAll(
        () -> assertEquals("PAID", invoice.status()),
        () -> assertTrue(invoice.total().signum() > 0),
        () -> assertNotNull(invoice.issuedAt())
    );
}

@Test
void invalidQtyRejected() {
    assertThrows(InvalidQuantityException.class,
        () -> cart.add(line("SKU", -3)));
}
```

AssertJ:

```java
@Test
void sortedByDueDate() {
    assertThat(repository.overdue())
        .hasSize(2)
        .extracting(Task::dueDate)
        .isSorted();
}
```

Internal invariant (dev only):

```java
void applyDiscount(Money base) {
    assert base.signum() >= 0 : "base must be non-negative";
    // business logic
}
```

Run tests with assertions always on; `-ea` irrelevant for JUnit failures (`AssertionError` from org.junit).

## Подводные камни

- **assert keyword in library code.** Disabled JVM silently skips.
- **assertEquals wrong argument order.** expected vs actual swap confuses diff.
- **Floating compare without delta.** `0.1 + 0.2` fails exact equality.
- **Single assert per heavy setup.** First failure hides rest; use `assertAll`.
- **verify without assert state.** Green interaction, wrong business result.
