# What is mutation testing?

## In short

Mutation testing alters production bytecode (mutants: `+` to `-`, `>` to `>=`) and reruns tests. A good suite "kills" the mutant (test fails). A surviving mutant means weak assertions despite green line coverage.

## How to answer in an interview

- Metric: mutation score = killed / total mutants.
- Tools: PIT (pitest-maven-plugin) for Java.
- CPU expensive; I run nightly on critical modules, not every commit.
- I inspect surviving mutants in HTML report and add boundary asserts.
- Honestly: 100% mutation score is impractical; target 70-85% on payment core.

## How it works

Original:

```java
int discount(int total) {
    if (total > 100) return total - 10;
    return total;
}
```

Mutant A: `>` → `>=`; Mutant B: `- 10` → `+ 10`.

Test only `discount(150)==140` may kill B, but Mutant A survives without `discount(100)==100`. PIT config:

```xml
<plugin>
  <groupId>org.pitest</groupId>
  <artifactId>pitest-maven</artifactId>
  <version>1.15.8</version>
  <configuration>
    <targetClasses>
      <param>com.acme.billing.*</param>
    </targetClasses>
    <mutationThreshold>75</mutationThreshold>
  </configuration>
</plugin>
```

`mvn org.pitest:pitest-maven:mutationCoverage` generates report: green killed, red survived with line highlight.

## Pitfalls

- **Equivalent mutants.** `(x * 2)` vs `(x << 1)` both survive legitimately; score noise.
- **Full repo PIT on CI.** Hours runtime; devs disable plugin.
- **Tests with no assertions.** Mutants die by luck; false confidence.
- **Mock-heavy code.** Mutations in unreachable branches inflate score.
- **Chasing score over readability.** Absurd asserts only for PIT.
