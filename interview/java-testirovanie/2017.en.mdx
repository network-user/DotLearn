# Why use JUnit in Java projects?

## In short

JUnit is the standard Java unit test framework: annotations describe tests, the runner discovers and executes them, assertions capture expectations. JUnit 5 (Jupiter) is modular and integrates with Maven Surefire, Gradle, IDEs, and CI.

## How to answer in an interview

- `@Test` method = one scenario; `@BeforeEach` prepares SUT, `@AfterEach` cleans resources.
- Assertions via `org.junit.jupiter.api.Assertions`, not the `assert` keyword (can be disabled with `-disableassertions`).
- `@Nested` groups cases; `@DisplayName` for readable CI output.
- Tests are fast, no network; heavy work goes to integration suite.
- Honestly: JUnit is not a mock framework; I add Mockito separately to isolate dependencies.

## How it works

```java
class PriceCalculatorTest {
    private PriceCalculator calculator;

    @BeforeEach
    void init() {
        calculator = new PriceCalculator(new BigDecimal("0.20"));
    }

    @Test
    void appliesVatToNetPrice() {
        Money total = calculator.withVat(Money.of("100.00", "EUR"));
        assertEquals(new BigDecimal("120.00"), total.amount());
    }

    @Test
    void rejectsNegativeNet() {
        assertThrows(IllegalArgumentException.class,
            () -> calculator.withVat(Money.of("-1", "EUR")));
    }
}
```

Surefire scans `*Test.java`, Jupiter engine runs methods. Failures show expected/actual diff. In Gradle:

```kotlin
tasks.test {
    useJUnitPlatform()
}
```

CI runs `mvn test` on every PR; red build blocks merge. Test pyramid: many unit tests on domain logic, fewer slice/e2e.

## Pitfalls

- **Order-dependent tests.** Shared mutable state between `@Test` causes flakiness.
- **Sleep instead of Awaitility.** `Thread.sleep` hides race conditions.
- **JUnit 4 and 5 in one module.** Vintage engine needed, but `@Before` vs `@BeforeEach` confuses.
- **Testing private methods.** Breaks encapsulation; test behavior via public API.
- **No assertions.** Test passes if no exception, but result is unchecked.
