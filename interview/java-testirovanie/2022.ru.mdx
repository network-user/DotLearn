# Behavior verification vs state verification

## Суть

State verification проверяет итоговое состояние объекта или результата метода (`assertEquals` на balance). Behavior verification проверяет, что произошли нужные взаимодействия (`verify` вызова collaborator). Оба валидны; выбор зависит от того, что является контрактом.

## Что ответить на собесе

- State-first для pure domain и immutable value objects.
- Behavior когда side effect важнее return value (отправка event, audit log).
- Mockito `verify` = behavior; AssertJ on entity = state.
- Комбинирую: assert outcome + verify critical call once.
- Честно: behavior tests хрупче; prefer state if observable result exists.

## Как это работает

Behavior on orchestrator:

```java
@Test
void checkoutChargesPaymentGateway() {
    PaymentGateway gateway = mock(PaymentGateway.class);
    CheckoutService checkout = new CheckoutService(gateway, noopInventory());
    Cart cart = Cart.of(Money.of("42", "USD"));

    checkout.complete(cart);

    verify(gateway).capture("42.00", cart.id());
}
```

State on aggregate:

```java
@Test
void depositIncreasesBalance() {
    Wallet wallet = Wallet.empty();
    wallet.deposit(Money.of("50", "USD"));
    wallet.withdraw(Money.of("15", "USD"));
    assertEquals(Money.of("35", "USD"), wallet.balance());
}
```

Command handler часто тестирую state (rows in fake repo) + one verify на outbox publish. Query side almost always state-only.

## Подводные камни

- **verify internal private calls.** Refactor breaks test without behavior change.
- **State without invariant checks.** Assert field, miss illegal transition.
- **verifyNoInteractions after partial stub.** False green when code path skipped.
- **Behavior on data class.** Getter/setter verify бессмысленен.
- **Order verification overkill.** `InOrder` unless protocol strictly ordered.
