# Line, branch, and condition coverage: differences

## In short

Line coverage counts executed bytecode lines. Branch coverage requires both sides of each jump (if/else, switch, ternary). Condition coverage (MC/DC lite) checks independent effect of each boolean subexpression. Strictness increases: line < branch < condition.

## How to answer in an interview

- 100% line does not imply 100% branch: `if (x) foo()` without else stays red on branch.
- JaCoCo report shows diamond icons on uncovered branches.
- Complex `a && b || c` needs multiple inputs for branch/condition.
- CI gate is usually line + branch on domain packages.
- Honestly: coverage is not correctness; mutation testing complements it.

## How it works

```java
static String classify(int age, boolean vip) {
    if (age >= 18 && vip) {
        return "adult-vip";
    }
    if (age >= 18) {
        return "adult";
    }
    return "minor";
}
```

| Test input | Lines hit | Branches hit |
|------------|-----------|--------------|
| (20, true) | all partial | true path first if |
| (20, false) | needs 2nd if | else implicit vip |
| (16, false) | minor return | underage branch |

One test `(20,true)` yields high line % but misses minor path. Condition coverage adds `(17,false)`, `(20,false)`, `(16,true)` so toggling `vip` changes outcome at fixed age.

JaCoCo HTML: green line, yellow diamond partial branch, red missed.

## Pitfalls

- **Chasing line 100%.** Empty catch swallowed.
- **Ternary and switch without default.** Hidden branch.
- **Short-circuit logic.** `a && b()` second operand never evaluated in test.
- **Generated bytecode inflation.** Lombok builder skews line metric.
- **Condition coverage cost.** Combinatorial explosion on 5+ flags.
