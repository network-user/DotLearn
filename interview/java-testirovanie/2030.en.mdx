# What counts as a unit in unit testing?

## In short

A unit is the smallest piece of behavior you can verify in isolation: a domain service method, pure function, value object invariant. The unit boundary is a team choice; in Java it is often a class or package, not the whole Spring context.

## How to answer in an interview

- Unit = SUT + substituted ports; no real DB or HTTP.
- One failing test points at one class/method.
- I do not call any fast test a unit: `@SpringBootTest` is integration.
- Sociable units allow real value objects; only IO is mocked.
- Honestly: a 2000-line god class is not a unit; decompose first.

## How it works

SUT `TransferPolicy`:

```java
class TransferPolicy {
    private final LimitRepository limits;

    TransferPolicy(LimitRepository limits) {
        this.limits = limits;
    }

    void validate(Money amount, AccountId from) {
        if (amount.signum() <= 0) throw new InvalidAmountException();
        if (amount.compareTo(limits.dailyCap(from)) > 0) throw new LimitExceededException();
    }
}
```

Unit test:

```java
@Test
void blocksAmountAboveDailyCap() {
    LimitRepository limits = mock(LimitRepository.class);
    when(limits.dailyCap(id("a1"))).thenReturn(Money.of("1000", "USD"));
    TransferPolicy policy = new TransferPolicy(limits);

    assertThrows(LimitExceededException.class,
        () -> policy.validate(Money.of("1001", "USD"), id("a1")));
}
```

Collaborators beyond the boundary are mock/fake; cap logic is checked directly. If the test uses `@DataJpaTest`, that is a different unit boundary.

## Pitfalls

- **Unit test entire microservice.** 40s runtime, flaky network.
- **Testing private methods.** Brittle tie to implementation.
- **No clear SUT.** Asserts scattered across helpers.
- **Shared static mutable state.** Order-dependent failures.
- **Unit without isolation proves little.** Green while SQL mapping is broken.
