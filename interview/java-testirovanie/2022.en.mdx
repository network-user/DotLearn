# Behavior verification vs state verification

## In short

State verification checks final object state or method result (`assertEquals` on balance). Behavior verification checks required interactions happened (`verify` on a collaborator). Both are valid; choice depends on what the contract is.

## How to answer in an interview

- State-first for pure domain and immutable value objects.
- Behavior when side effects matter more than return value (event publish, audit log).
- Mockito `verify` = behavior; AssertJ on entity = state.
- I combine: assert outcome plus verify critical call once.
- Honestly: behavior tests are brittle; prefer state if an observable result exists.

## How it works

Behavior on orchestrator:

```java
@Test
void checkoutChargesPaymentGateway() {
    PaymentGateway gateway = mock(PaymentGateway.class);
    CheckoutService checkout = new CheckoutService(gateway, noopInventory());
    Cart cart = Cart.of(Money.of("42", "USD"));

    checkout.complete(cart);

    verify(gateway).capture("42.00", cart.id());
}
```

State on aggregate:

```java
@Test
void depositIncreasesBalance() {
    Wallet wallet = Wallet.empty();
    wallet.deposit(Money.of("50", "USD"));
    wallet.withdraw(Money.of("15", "USD"));
    assertEquals(Money.of("35", "USD"), wallet.balance());
}
```

I often test command handlers for state (rows in fake repo) plus one verify on outbox publish. Query side is almost always state-only.

## Pitfalls

- **verify internal private calls.** Refactor breaks test without behavior change.
- **State without invariant checks.** Assert field, miss illegal transition.
- **verifyNoInteractions after partial stub.** False green when code path skipped.
- **Behavior on data class.** Getter/setter verify is pointless.
- **Order verification overkill.** `InOrder` unless protocol is strictly ordered.
