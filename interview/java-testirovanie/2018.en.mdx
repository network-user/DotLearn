# When should you use parameterized tests in JUnit 5?

## In short

Parameterized tests run one method with multiple input sets. They remove copy-paste tests and broaden edge coverage without bloating the class with dozens of nearly identical `@Test` methods.

## How to answer in an interview

- `@ParameterizedTest` plus `@CsvSource`, `@MethodSource`, `@ValueSource`.
- Fits pure functions, validators, parsers, discount rules.
- Each CSV row is a separate test case in CI reports.
- `@NullAndEmptySource` for strings and collections adds edge cases in one annotation.
- Honestly: with complex setup, `@MethodSource` with POJO arguments reads better than a huge CSV.

## How it works

```java
class SlugifierTest {

    @ParameterizedTest
    @CsvSource({
        "Hello World, hello-world",
        "API Gateway, api-gateway",
        "  trim  , trim"
    })
    void buildsSlug(String input, String expected) {
        assertEquals(expected, Slugifier.from(input));
    }

    @ParameterizedTest
    @MethodSource("invalidInputs")
    void rejectsBadInput(String raw) {
        assertThrows(IllegalArgumentException.class, () -> Slugifier.from(raw));
    }

    static Stream<String> invalidInputs() {
        return Stream.of("", "!!!", "a".repeat(300));
    }
}
```

JUnit generates display name `buildsSlug(String, String) [1] input=Hello World`. `@EnumSource(DayOfWeek.class)` checks all enum constants. For large tables I keep data in `classpath:cases/slug.csv` via `@CsvFileSource`.

## Pitfalls

- **Single assert hides cause.** Name `@CsvSource` parameters and use a `name` template.
- **Mutable shared state between iterations.** Class fields change and break the next case.
- **Overwide MethodSource.** 500 cases without grouping are unreadable in the IDE.
- **Mixing unit and integration in one parameterized test.** Slow IO on every row.
- **Forgotten provider on `@ParameterizedTest`.** Empty class or runtime error.
