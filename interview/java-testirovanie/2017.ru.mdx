# Зачем нужен JUnit в Java-проектах?

## Суть

JUnit стандартный фреймворк unit-тестов в Java: аннотации описывают тесты, runner их находит и выполняет, assertions фиксируют ожидания. JUnit 5 (Jupiter) модульный, интегрируется с Maven Surefire, Gradle, IDE и CI.

## Что ответить на собесе

- `@Test` метод = один сценарий; `@BeforeEach` готовит SUT, `@AfterEach` чистит ресурсы.
- Assertions через `org.junit.jupiter.api.Assertions`, не `assert` keyword (можно отключить `-disableassertions`).
- `@Nested` группирует кейсы; `@DisplayName` читабельный вывод в CI.
- Тесты быстрые, без сети; тяжёлое уходит в integration suite.
- Честно: JUnit не mock framework; Mockito подключаю отдельно для изоляции зависимостей.

## Как это работает

```java
class PriceCalculatorTest {
    private PriceCalculator calculator;

    @BeforeEach
    void init() {
        calculator = new PriceCalculator(new BigDecimal("0.20"));
    }

    @Test
    void appliesVatToNetPrice() {
        Money total = calculator.withVat(Money.of("100.00", "EUR"));
        assertEquals(new BigDecimal("120.00"), total.amount());
    }

    @Test
    void rejectsNegativeNet() {
        assertThrows(IllegalArgumentException.class,
            () -> calculator.withVat(Money.of("-1", "EUR")));
    }
}
```

Surefire сканирует `*Test.java`, Jupiter engine запускает методы. Failure показывает expected/actual diff. В Gradle:

```kotlin
tasks.test {
    useJUnitPlatform()
}
```

CI pipeline гоняет `mvn test` на каждый PR; красный build блокирует merge. Test pyramid: много unit на domain logic, меньше slice/e2e.

## Подводные камни

- **Тесты зависят от порядка.** Shared mutable state между `@Test` даёт flaky.
- **Sleep вместо awaitility.** Race conditions маскируются `Thread.sleep`.
- **JUnit 4 и 5 в одном module.** Vintage engine нужен, но путает аннотации `@Before` vs `@BeforeEach`.
- **Testing private methods.** Ломает encapsulation; тестируй behavior через public API.
- **No assertions.** Test проходит, если exception не летит, но результат не проверен.
