# Mock, stub, spy, and fake: how do they differ?

## In short

Test doubles replace dependencies in unit tests. A stub returns canned answers. A mock verifies collaboration happened as expected. A spy wraps a real object and records calls. A fake provides a working but simplified implementation (in-memory repo).

## How to answer in an interview

- Stub = canned response; mock = verify interaction (Mockito blends both).
- Spy is partially real: `spy(ArrayList)` calls real `add`, but you can stub one method.
- I prefer fakes for repository/cache: real logic without Docker.
- Do not mock value objects or things you do not control (JDK final needs mockito-inline care).
- Honestly: over-mocking tests Mockito, not the domain; fake + slice test is often steadier.

## How it works

```java
interface RateClient { BigDecimal fetch(String pair); }

@Test
void orderUsesLiveRate() {
    RateClient stub = mock(RateClient.class);
    when(stub.fetch("EUR/USD")).thenReturn(new BigDecimal("1.08"));

    OrderPricing pricing = new OrderPricing(stub);
    Money total = pricing.quote(Money.of("100", "EUR"));

    assertEquals(new BigDecimal("108.00"), total.amount());
    verify(stub).fetch("EUR/USD");
}

@Test
void spyKeepsRealListState() {
    List<String> backing = new ArrayList<>();
    List<String> tracked = spy(backing);
    tracked.add("alpha");
    verify(tracked).add("alpha");
    assertEquals(1, tracked.size());
}

class InMemoryOrders implements OrderRepository {
    private final Map<UUID, Order> store = new HashMap<>();
    @Override public void save(Order order) { store.put(order.id(), order); }
    @Override public Optional<Order> find(UUID id) { return Optional.ofNullable(store.get(id)); }
}
```

Mockito `mock()` defaults to stub + verify. Fake `InMemoryOrders` survives refactors better than 15 `when()` calls per DAO method.

## Pitfalls

- **Mock concrete class with final methods.** Needs inline mock maker or extract an interface.
- **Spy on partially constructed object.** Constructor side effects + stub = confusing failures.
- **Verify every call.** Brittle tests break when you add a log call.
- **Fake without thread safety.** Parallel tests corrupt a shared map.
- **Stub void method without doThrow.** `when(voidMethod()).thenThrow` does not compile; use `doThrow`.
