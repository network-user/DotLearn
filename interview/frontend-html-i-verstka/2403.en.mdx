# Semantic markup: why tags carry meaning, not just style

## In short

Semantic markup picks HTML tags by content role, not appearance. The browser, search engine, and screen reader get an explicit document structure: header, main content, article, navigation, footer.

## How to answer in an interview

- I explain that `<div class="header">` draws a box, while `<header>` tells machines "this is the page header."
- I list the wins: a11y (landmarks, narration), SEO, and code readability for the team.
- I name the core tags: `header`, `nav`, `main`, `article`, `section`, `aside`, `footer`.
- I note that semantics do not replace ARIA, but they reduce the need for workarounds.
- I honestly add that without team conventions, semantics quickly devolve into a `<section>` on every block.

## How it works

The parser builds an **accessibility tree** from the DOM. A semantic tag maps to a landmark role: a screen reader lets users jump to `main` or `nav`. Search engines use heading hierarchy and `article` wrappers to identify standalone content.

A typical landing page skeleton:

```html
<body>
  <header>
    <a href="/">DotLearn</a>
    <nav aria-label="Main menu">
      <ul>
        <li><a href="/topics">Topics</a></li>
        <li><a href="/progress">Progress</a></li>
      </ul>
    </nav>
  </header>
  <main>
    <article>
      <h1>How to learn frontend</h1>
      <p>Lesson text...</p>
    </article>
    <aside>
      <h2>Related topics</h2>
    </aside>
  </main>
  <footer>
    <p>© 2026 DotLearn</p>
  </footer>
</body>
```

**`<header>`** can also live inside `article` (post title area). **`<section>`** groups a thematic block with its own heading; an empty `section` without `h2`/`h3` loses meaning. **`<article>`** fits RSS-style items: content you could syndicate elsewhere without losing intent.

CSS still controls visuals: semantics answer **interpretation**, not pixels.

## Pitfalls

**Div soup.** Replacing every tag with `<div>` breaks landmarks; you end up duplicating `role` and `aria-label`.

**Section spam.** A `section` per widget without a heading helps neither users nor assistive tech.

**Wrong main.** Two `main` elements on one page, or `main` inside `aside`, violates the spec and confuses screen readers.

**Article vs section.** A blog post is an `article`; a "Our benefits" block on a landing page is usually a `section`, not an `article`.

**Semantics without heading hierarchy.** Correct tags will not help if `h1`–`h6` levels skip or repeat.
