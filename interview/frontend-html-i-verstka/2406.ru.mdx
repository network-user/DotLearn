# Списки в HTML: ul, ol и dl и когда какой выбирать

## Суть

HTML предлагает три типа списков: маркированный `ul`, нумерованный `ol` и список определений `dl`. Выбор зависит от того, важен ли порядок и нужна ли пара «термин / пояснение».

## Что ответить на собесе

- `ul` для набора равнозначных пунктов без нумерации (меню, теги, ингредиенты).
- `ol` когда последовательность несёт смысл (шаги рецепта, рейтинг, инструкция).
- `dl` для глоссария: `dt` термин, `dd` определение; допускается несколько `dd` на один `dt`.
- Упоминаю вложенные списки и что стили маркера задаются CSS (`list-style`, `::marker`).
- Честно добавляю: для длинных FAQ иногда удобнее `section` + заголовки, но `dl` остаётся семантически точным для пар термин-значение.

## Как это работает

Каждый список рендерится как **группа элементов** с ролью list. Скринридер сообщает число пунктов и уровень вложенности.

Маркированный список:

```html
<ul>
  <li>TypeScript</li>
  <li>React</li>
  <li>Vite</li>
</ul>
```

Нумерованный с шагами:

```html
<ol>
  <li>Установить зависимости: <code>pnpm install</code></li>
  <li>Запустить dev-сервер: <code>pnpm dev:web</code></li>
  <li>Открыть <code>localhost:5173</code></li>
</ol>
```

Атрибут `start` или `reversed` меняет нумерацию; `value` на `li` задаёт номер конкретного пункта.

Список определений:

```html
<dl>
  <dt>CLS</dt>
  <dd>Сдвиг макета при загрузке контента.</dd>
  <dt>LCP</dt>
  <dd>Время появления крупнейшего видимого блока.</dd>
</dl>
```

Навигация часто строится как `nav > ul > li > a`, сохраняя семантику списка ссылок.

## Подводные камни

**br вместо li.** Переносы строк не создают список; AT не видит отдельные пункты.

**ol для декоративной нумерации.** Если порядок не важен, нумерация вводит в заблуждение.

**dl как произвольные колонки.** Пары dt/dd должны быть терминами, не сеткой «лейбл / значение» без смысла глоссария.

**Глубокая вложенность.** Пятый уровень `ul` внутри `ul` ухудшает озвучку; лучше упростить структуру или разбить контент.

**Сброс стилей списка.** `list-style: none` на навигации допустим, но оставляю `role` корректным (нативный `ul`/`li` сохраняет list semantics).
