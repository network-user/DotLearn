# SVG, JPEG, and PNG: choosing the right image format

## In short

SVG stores vector graphics in XML and scales without pixelation. JPEG compresses photos with lossy encoding for small file size. PNG stores lossless raster pixels and supports alpha transparency.

## How to answer in an interview

- I compare nature: vector (SVG) vs raster (JPEG/PNG).
- I say SVG for icons, logos, charts; JPEG for photos; PNG for UI with transparency and sharp edges.
- I mention modern alternatives: WebP/AVIF for photos, but the classic trio still appears in interviews.
- I note performance: a heavy SVG with thousands of paths can beat a light PNG.
- I add honestly: there is no single universal format; content and byte budget decide.

## How it works

**SVG** describes shapes with math. On retina screens a 24px icon stays sharp; color changes via CSS `currentColor` or `fill`.

```html
<svg width="48" height="48" viewBox="0 0 48 48" aria-hidden="true">
  <circle cx="24" cy="24" r="18" fill="currentColor" />
</svg>
```

SVG works inline in React/Vue, in sprites, and as `background-image`. Complex editor exports are sometimes cheaper as raster.

**JPEG** fits photos and smooth gradients. Quality is a compression knob; re-saving accumulates artifacts. No transparency.

**PNG** keeps every pixel; PNG-8 for flat palettes, PNG-24/32 for soft shadows and screenshots. Files are heavier than JPEG at the same dimensions.

`<picture>` lets the browser choose:

```html
<picture>
  <source srcset="/hero.avif" type="image/avif" />
  <source srcset="/hero.webp" type="image/webp" />
  <img src="/hero.jpg" alt="Studio interior" width="1200" height="800" />
</picture>
```

On the frontend set `width`/`height` against CLS, lazy-load below the fold, and compress at the CDN.

## Pitfalls

**JPEG for text and UI.** Blurry edges on buttons and screenshots.

**Huge SVG from design tools.** Tens of kilobytes of paths slower than an optimized PNG.

**PNG for large photos.** Megabytes where JPEG/WebP are far smaller.

**Missing alt on meaningful img.** Format does not fix accessibility.

**Ignoring SVG security.** Inline SVG with user content needs sanitization.
