# HTML5 input types and what they change

## In short

The `type` attribute on `<input>` sets field semantics, built-in validation, and native UI: email, url, number, date, range, color, and more. The browser picks mobile keyboards and basic checks before form submit.

## How to answer in an interview

- I name key types: `email`, `url`, `tel`, `search`, `number`, `range`, `date`, `time`, `datetime-local`, `month`, `week`, `color`, plus classic `text`, `password`, `checkbox`, `radio`, `file`, `hidden`.
- I explain that type drives the constraint validation API and UI hints (calendar, slider, color picker).
- I mention `min`, `max`, `step`, `multiple`, `list` as companions to type.
- I stress server-side validation is still mandatory.
- I add honestly: widgets and rules differ across browsers, especially `date` in Safari.

## How it works

Before HTML5 almost everything was `text` plus manual JS masks. New types move part of the logic into the platform.

**Text and contact:** `email` checks for `@` and a domain; `url` expects a scheme; `tel` does not validate format but opens a numeric keypad; `search` may show a clear button.

**Numeric:** `number` adds a spinner and numeric check; `range` renders a slider within `min`/`max` and `step`.

**Date and time:** `date`, `time`, `datetime-local`, `month`, `week` open native pickers where supported.

**Other:** `color` shows a picker; `checkbox`/`radio` for boolean and exclusive choice; `file` for uploads with `accept` and `multiple`.

Registration form example:

```html
<form>
  <label>
    Email
    <input type="email" name="email" required autocomplete="email" />
  </label>
  <label>
    Age
    <input type="number" name="age" min="18" max="120" />
  </label>
  <label>
    Visit date
    <input type="date" name="visit" />
  </label>
  <label>
    Volume level
    <input type="range" name="volume" min="0" max="100" step="5" />
  </label>
  <label>
    Accent color
    <input type="color" name="accent" value="#3366cc" />
  </label>
</form>
```

In JS you read `input.validity` and `input.validationMessage`; with `type="email"`, `typeMismatch` becomes true for a string without a domain.

On mobile, `inputmode` and type together affect the keyboard: `tel` for phone numbers; sometimes `text` + `inputmode="numeric"` for PIN codes.

## Pitfalls

**Client-only trust.** Any `type` is bypassed via DevTools; the backend must validate.

**Inconsistent date UI.** `date` is a calendar in one browser and almost `text` in another; plan a fallback or library.

**number for codes and phones.** Leading zeros and plus signs are lost in `number`; prefer `text` with a mask.

**Weak email/url validation.** The browser checks shape roughly, not full RFC rules.

**Hidden type via CSS.** The native widget disappears but constraint validation remains: test real UX.
