# Structured data: microdata, schema.org, and JSON-LD

## In short

Structured data describes page entities (product, article, event) in a machine-readable form. Schema.org provides a shared vocabulary of types and properties; microdata embeds it in HTML attributes, JSON-LD places the description in a separate JSON block.

## How to answer in an interview

- I explain the goal: search engines and assistants understand content better and may show rich results.
- I separate formats: microdata (itemscope/itemprop), RDFa, JSON-LD (`application/ld+json`).
- I name common types: `Product`, `Article`, `FAQPage`, `Organization`, `BreadcrumbList`.
- I note JSON-LD is preferred today: it does not disturb markup and is easy to generate at build time.
- I add honestly: markup does not guarantee a snippet; Google validates and may reject bad data.

## How it works

**Schema.org** is an open schema set agreed by major search engines. You pick `@type` and fill required fields for the chosen rich result.

**Microdata** annotates live DOM:

```html
<article itemscope itemtype="https://schema.org/Article">
  <h1 itemprop="headline">Flexbox guide</h1>
  <p itemprop="author" itemscope itemtype="https://schema.org/Person">
    <span itemprop="name">Anna K.</span>
  </p>
  <time itemprop="datePublished" datetime="2026-03-01">March 1, 2026</time>
</article>
```

Pros: data sits next to visible text. Cons: presentation and metadata mix, harder to template.

**JSON-LD** lives in `<script type="application/ld+json">` in `<head>` or `<body>`:

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Product",
  "name": "Aero wireless headphones",
  "sku": "ARO-42",
  "offers": {
    "@type": "Offer",
    "price": "7990",
    "priceCurrency": "RUB",
    "availability": "https://schema.org/InStock"
  }
}
</script>
```

On the frontend JSON-LD is often built during SSR or prerender with SEO meta. In client-only SPAs you can inject on route change, but crawlers must see final HTML.

Rich Results Test and Search Console catch nested object mistakes (`offers`, `aggregateRating`).

Rich snippets (stars, price, FAQ accordions) require parity with on-page content; fake ratings risk penalties.

## Pitfalls

**Mismatch with visible text.** JSON-LD price differs from the UI.

**Legacy http://schema.org samples.** Prefer https and current types.

**Duplicate microdata and JSON-LD.** Conflicting descriptions confuse parsers.

**Missing required fields.** Product without `offers` or `image` won't earn a card.

**Client-only generation without SSR.** Bots see a page without LD-JSON.
