# Доступность в вёрстке: что такое a11y на практике

## Суть

Accessibility (a11y) означает, что интерфейсом могут пользоваться люди с разными возможностями: клавиатура вместо мыши, скринридер вместо экрана, увеличенный контраст. На уровне вёрстки это семантика HTML, фокус, альтернативный текст и ARIA там, где нативных ролей не хватает.

## Что ответить на собесе

- Даю определение a11y и связываю с WCAG: воспринимаемость, управляемость, понятность, надёжность.
- Перечисляю практики: семантические теги, видимый focus, `alt`, контраст, подписи к полям, landmarks.
- Показываю, что доступность улучшает UX для всех: мобильные, медленные сети, голосовое управление.
- Уточняю роль ARIA: дополняет, но не заменяет правильную разметку.
- Честно добавляю: 100% автотестами не закрыть, нужны ручная проверка клавиатурой и скринридером.

## Как это работает

Скринридер строит **accessibility tree** из DOM: заголовки, ссылки, кнопки, landmarks. Если вместо `<button>` стоит `<div onclick>`, элемент не попадает в дерево как кнопка, пока не добавить роль и обработку клавиатуры.

Семантический каркас задаёт структуру без лишних ARIA:

```html
<header>
  <h1>Личный кабинет</h1>
</header>
<nav aria-label="Основное меню">
  <ul>
    <li><a href="/orders">Заказы</a></li>
    <li><a href="/profile">Профиль</a></li>
  </ul>
</nav>
<main>
  <h2>Последние заказы</h2>
  <img src="/chart.png" alt="График заказов за месяц: рост на 12%" />
</main>
```

**Клавиатурная навигация** требует логичного tab-order, видимого `:focus-visible` и отсутствия ловушек фокуса в модалках. Интерактивные элементы достижимы без мыши.

**Контраст** текста и фона проверяют по WCAG (обычно 4.5:1 для обычного текста). Цвет не должен быть единственным носителем смысла: статус «ошибка» дублируется текстом или иконкой с подписью.

**Формы**: `<label for="email">` или вложение связывает подпись с полем; `aria-describedby` подключает текст ошибки.

Для динамических уведомлений используют live regions:

```html
<div role="status" aria-live="polite">
  Корзина обновлена
</div>
```

ARIA (`role`, `aria-expanded`, `aria-controls`) нужна для кастомных виджетов (табы, комбобоксы), когда нативного элемента нет или он скрыт стилями.

## Подводные камни

**ARIA вместо семантики.** `role="button"` на div без Enter/Space хуже, чем настоящий `<button>`.

**Пустой или декоративный alt.** `alt=""` допустим только для чисто декоративных картинок; иначе скринридер пропускает смысл.

**Скрытие outline.** `outline: none` без замены ломает навигацию с клавиатуры.

**Низкий контраст «для красоты».** Светло-серый текст на белом формально читаем, но не проходит WCAG.

**Ловушка фокуса в модалке.** Tab уходит под оверлей, пользователь клавиатуры теряет контекст.
