# Linking label and input: for/id, nesting, and form accessibility

## In short

A `<label>` must be programmatically tied to its control: via matching `for`/`id` or by nesting. Then clicking the text focuses the field, and a screen reader announces the caption with the control.

## How to answer in an interview

- I name two approaches: `for="email"` on the label and `id="email"` on the input, or an input nested inside the label.
- I stress the larger tap target on mobile: tapping text is easier than hitting a tiny checkbox.
- I note that `textarea`, `select`, and custom controls follow the same rules.
- I mention `aria-labelledby` when the visible caption is not a `<label>`, though native association is preferred.
- I honestly add that with complex layout (label and input in separate flex cells), nesting fails and `for`/`id` is required.

## How it works

The browser builds the control's **accessible name** from the associated label. A label click forwards activation to the linked element; for checkboxes and radios it toggles state.

Explicit id association:

```html
<label for="user-email">Work email</label>
<input
  type="email"
  id="user-email"
  name="email"
  autocomplete="email"
  required
/>
```

Nesting without an id:

```html
<label>
  Remember me
  <input type="checkbox" name="remember" />
</label>
```

In the second pattern, the first nested labelable element (`input`, `select`, `textarea`) is linked automatically. Multiple fields inside one label behave unpredictably: use one field or switch to `for`/`id`.

For radio groups, each `input` gets its own `label`; the group title uses `<fieldset>` plus `<legend>`.

## Pitfalls

**Mismatched for and id.** A typo in `id` breaks the link; Accessibility in DevTools will show an empty name.

**Duplicate ids.** Two inputs sharing one `id` violate HTML and break label association.

**Placeholder as label.** Placeholder is not a label: it vanishes on input and is often not exposed as the field name.

**Label wrapping a button.** A nested `<button>` inside a label can steal activation; keep elements separate.

**Hidden input.** A custom-styled checkbox still needs a native input and linked label, or keyboard and AT users lose the control.
