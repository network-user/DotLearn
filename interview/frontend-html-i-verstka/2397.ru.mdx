# Встроенная валидация HTML-форм

## Суть

Браузер проверяет поля формы по constraint validation API ещё до сетевого запроса: обязательность, длина, шаблон, тип, диапазон чисел. Атрибуты `required`, `minlength`, `pattern`, `min`/`max` и сами `type` задают правила на уровне разметки.

## Что ответить на собесе

- Описываю механику: при submit браузер валидирует поля, при ошибке блокирует отправку и показывает подсказку.
- Перечисляю атрибуты: `required`, `minlength`/`maxlength`, `pattern`, `type`, `min`/`max`, `step`.
- Упоминаю API: `checkValidity()`, `reportValidity()`, объект `validity` с флагами `valueMissing`, `patternMismatch` и др.
- Говорю про `novalidate` на форме и кастомизацию через `setCustomValidity()`.
- Честно добавляю: HTML-валидация дополняет, но не заменяет сервер; сложные правила всё равно на бэкенде или в JS.

## Как это работает

Каждый контрол с ограничениями получает **ValidityState**. При клике на submit форма проходит по полям: если хотя бы одно `!field.checkValidity()`, событие `submit` отменяется (если не перехватили в JS).

Типичные атрибуты:

- `required` запрещает пустое значение.
- `minlength` / `maxlength` для текстовых полей.
- `pattern` сопоставляет значение с RegExp (якоря `^$` добавляет браузер).
- `type="email"` / `url` / `number` включают встроенные проверки формата.
- `min`, `max`, `step` для чисел и дат.

```html
<form id="signup" novalidate>
  <label for="mail">Почта</label>
  <input
    id="mail"
    type="email"
    name="mail"
    required
    autocomplete="email"
  />

  <label for="pin">Индекс</label>
  <input
    id="pin"
    type="text"
    name="pin"
    pattern="\d{6}"
    title="Шесть цифр"
    required
  />

  <label for="qty">Количество</label>
  <input id="qty" type="number" name="qty" min="1" max="99" step="1" />

  <button type="submit">Отправить</button>
</form>
```

В TypeScript-обработчике можно унифицировать сообщения:

```ts
const form = document.querySelector<HTMLFormElement>('#signup');

form?.addEventListener('submit', (event) => {
  if (!form.reportValidity()) {
    event.preventDefault();
    return;
  }
  event.preventDefault();
  // дальше fetch / mutation
});
```

`setCustomValidity('Текст')` позволяет добавить бизнес-правило (например, занятый логин после blur). Пустая строка снимает кастомную ошибку.

Атрибут `novalidate` на `<form>` отключает нативный гейт на submit; тогда валидацию полностью берёт на себя JS, но атрибуты всё ещё влияют на `validity` при ручных вызовах.

Стилизация: псевдокласс `:user-invalid` подсвечивает поле после взаимодействия пользователя без дублирования логики.

## Подводные камни

**Доверие только HTML.** Пользователь обходит проверки через консоль или прямой POST.

**pattern без title.** Браузер показывает generic-сообщение, UX страдает.

**Конфликт novalidate и required.** Команда забывает вызвать `reportValidity()` в своём коде.

**Разный текст ошибок по локали.** Нативные сообщения на языке браузера, не приложения; для i18n нужен JS-слой.

**submit без preventDefault на SPA.** Форма перезагружает страницу, хотя валидация сработала.
