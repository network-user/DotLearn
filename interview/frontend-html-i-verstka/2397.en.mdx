# Built-in HTML form validation

## In short

The browser validates form fields through the constraint validation API before any network request: required fields, length, pattern, type, numeric range. Attributes `required`, `minlength`, `pattern`, `min`/`max`, and `type` define rules in markup.

## How to answer in an interview

- I describe the flow: on submit the browser validates fields, blocks send on failure, and shows a hint.
- I list attributes: `required`, `minlength`/`maxlength`, `pattern`, `type`, `min`/`max`, `step`.
- I mention the API: `checkValidity()`, `reportValidity()`, the `validity` object with `valueMissing`, `patternMismatch`, and more.
- I cover `novalidate` on the form and customization via `setCustomValidity()`.
- I add honestly: HTML validation complements but does not replace the server; complex rules still live on the backend or in JS.

## How it works

Each constrained control gets a **ValidityState**. On submit the form walks fields: if any `!field.checkValidity()`, the `submit` event is canceled (unless intercepted in JS).

Common attributes:

- `required` blocks empty values.
- `minlength` / `maxlength` for text fields.
- `pattern` matches a RegExp (the browser adds `^$` anchors).
- `type="email"` / `url` / `number` enable built-in format checks.
- `min`, `max`, `step` for numbers and dates.

```html
<form id="signup" novalidate>
  <label for="mail">Email</label>
  <input
    id="mail"
    type="email"
    name="mail"
    required
    autocomplete="email"
  />

  <label for="pin">Postal code</label>
  <input
    id="pin"
    type="text"
    name="pin"
    pattern="\d{6}"
    title="Six digits"
    required
  />

  <label for="qty">Quantity</label>
  <input id="qty" type="number" name="qty" min="1" max="99" step="1" />

  <button type="submit">Submit</button>
</form>
```

In a TypeScript handler you can unify messages:

```ts
const form = document.querySelector<HTMLFormElement>('#signup');

form?.addEventListener('submit', (event) => {
  if (!form.reportValidity()) {
    event.preventDefault();
    return;
  }
  event.preventDefault();
  // then fetch / mutation
});
```

`setCustomValidity('Message')` adds a business rule (for example a taken username after blur). An empty string clears the custom error.

The `novalidate` attribute on `<form>` disables the native submit gate; then JS owns validation, but attributes still affect `validity` on manual calls.

Styling: the `:user-invalid` pseudo-class highlights a field after user interaction without duplicating logic.

## Pitfalls

**HTML-only trust.** Users bypass checks via the console or a direct POST.

**pattern without title.** The browser shows a generic message and UX suffers.

**novalidate vs required conflict.** The team forgets to call `reportValidity()` in custom code.

**Locale mismatch for errors.** Native messages follow the browser language, not the app; i18n needs a JS layer.

**submit without preventDefault on SPAs.** The form reloads the page even when validation passed.
