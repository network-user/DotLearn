# Accessibility in markup: what a11y means in practice

## In short

Accessibility (a11y) means people with different abilities can use the interface: keyboard instead of mouse, screen reader instead of sight, higher contrast. At the markup level this is semantic HTML, focus management, alternative text, and ARIA where native roles are not enough.

## How to answer in an interview

- I define a11y and tie it to WCAG: perceivable, operable, understandable, robust.
- I list practices: semantic tags, visible focus, `alt`, contrast, field labels, landmarks.
- I show accessibility improves UX for everyone: mobile, slow networks, voice control.
- I clarify ARIA's role: it supplements but does not replace correct markup.
- I add honestly: you cannot cover everything with automation; keyboard and screen reader checks are still required.

## How it works

A screen reader builds an **accessibility tree** from the DOM: headings, links, buttons, landmarks. If a `<div onclick>` replaces `<button>`, the node is not a button until you add role and keyboard handling.

A semantic skeleton sets structure without extra ARIA:

```html
<header>
  <h1>Account dashboard</h1>
</header>
<nav aria-label="Main menu">
  <ul>
    <li><a href="/orders">Orders</a></li>
    <li><a href="/profile">Profile</a></li>
  </ul>
</nav>
<main>
  <h2>Recent orders</h2>
  <img src="/chart.png" alt="Monthly orders chart: 12% growth" />
</main>
```

**Keyboard navigation** needs logical tab order, visible `:focus-visible`, and no focus traps in modals. Interactive controls must work without a mouse.

**Contrast** between text and background is checked against WCAG (typically 4.5:1 for normal text). Color must not be the only signal: an "error" state is duplicated with text or a labeled icon.

**Forms**: `<label for="email">` or nesting links the caption to the control; `aria-describedby` wires error copy.

For dynamic announcements use live regions:

```html
<div role="status" aria-live="polite">
  Cart updated
</div>
```

ARIA (`role`, `aria-expanded`, `aria-controls`) is for custom widgets (tabs, comboboxes) when there is no native element or it is hidden by styles.

## Pitfalls

**ARIA instead of semantics.** `role="button"` on a div without Enter/Space is worse than a real `<button>`.

**Empty or decorative alt.** `alt=""` is fine only for purely decorative images; otherwise screen readers skip meaning.

**Hiding outline.** `outline: none` without a replacement breaks keyboard navigation.

**Low contrast "for aesthetics".** Light gray on white looks readable but fails WCAG.

**Focus trap in a modal.** Tab escapes under the overlay and keyboard users lose context.
